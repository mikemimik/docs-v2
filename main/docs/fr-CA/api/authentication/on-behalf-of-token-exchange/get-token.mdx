---
title: "Obtenir un jeton"
description: "Le On-Behalf-Of (OBO) Token Exchange permet à un service intermédiaire d’échanger un jeton utilisateur entrant contre un nouveau jeton limité au service en aval."
---

`POST /oauth/token`

Le On-Behalf-Of (OBO) Token Exchange ([RFC 8693](https://datatracker.ietf.org/doc/html/rfc8693)) permet aux services intermédiaires d’échanger un jeton utilisateur limité à l’utilisateur contre un nouveau jeton pour appeler des services en aval.

Le nouveau jeton conserve l’identité et les autorisations de l’utilisateur d’origine, tout en étant limité précisément au service en aval, ce qui permet à ce service de prendre des décisions d’autorisation en fonction de l’utilisateur final. La chaîne de délégation est consignée dans la revendication `act` (acteur), chaque niveau représentant un service de la chaîne d’appel. Pour en savoir plus, consultez la [documentation sur le On-Behalf-Of Token Exchange](https://auth0.com/docs/secure/call-apis-on-users-behalf/on-behalf-of-token-exchange).

<div id="remarks">
  ## Remarques
</div>

* Seuls les clients d’API personnalisée associés à un serveur de ressources peuvent utiliser l’échange de jetons OBO. Un client d’API personnalisée doit répondre aux exigences suivantes :
  * Définissez `app_type` sur `resource_server`.
  * Définissez `resource_server_identifier` sur l’identifiant d’un serveur de ressources valide, c.-à-d. `https://my-api.example.com`. Auth0 utilise l’identifiant du serveur de ressources comme paramètre `audience` dans les requêtes d’autorisation.

* Les portées accordées à l’application peuvent différer des portées demandées. Dans ce cas, un paramètre `scope` sera inclus dans le JSON de réponse. Les portées sont fondées sur les [politiques de contrôle d’accès basé sur les rôles (RBAC)](/docs/fr-CA/manage-users/access-control/rbac) de l’utilisateur.

* Les échanges de jetons OBO déclenchent le [déclencheur d’Action `post-login`](/docs/fr-CA/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger), où `event.transaction.protocol` est défini sur `oauth2-token-exchange` et `event.transaction.actor` assure le suivi complet de la chaîne de délégation.

* La chaîne de délégation est limitée à cinq niveaux imbriqués. L’échange de jetons OBO échoue si le jeton du sujet comporte déjà cinq niveaux `act` imbriqués.

* Mettez les jetons d’accès en cache pendant toute leur durée de vie au lieu de demander un nouveau jeton pour chaque appel d’API. Les jetons d’accès peuvent être réutilisés jusqu’à leur expiration; des échanges de jetons répétés gaspillent des ressources, augmentent la latence et peuvent déclencher des limites de débit.

<div id="parameters">
  ## Paramètres
</div>

<ParamField header="DPoP" type="string">
  Une preuve DPoP pour la requête. Ce paramètre est facultatif et n’est requis que si votre application utilise [Demonstrating Proof-of-Possession](/docs/fr-CA/secure/sender-constraining/demonstrating-proof-of-possession-dpop).
</ParamField>

<ParamField header="auth0-forwarded-for" type="string">
  L’adresse IP de l’utilisateur final, sous forme de chaîne de caractères. Définissez ce paramètre si vous voulez que la protection [Suspicious IP Throttling](/docs/fr-CA/secure/attack-protection/suspicious-ip-throttling) fonctionne dans des scénarios côté serveur.
</ParamField>

<div id="request-body">
  ## Corps de la requête
</div>

<ParamField body="grant_type" type="string" required>
  Indique le type d’octroi que vous utilisez. Pour On-Behalf-Of Token Exchange, utilisez `urn:ietf:params:oauth:grant-type:token-exchange`.
</ParamField>

<ParamField body="subject_token_type" type="string" required>
  Le type du jeton du sujet. Pour On-Behalf-Of Token Exchange, utilisez `urn:ietf:params:oauth:token-type:access_token`.
</ParamField>

<ParamField body="subject_token" type="string" required>
  Le jeton d’accès Auth0 entrant provenant de l’utilisateur ou du service en amont que le service intermédiaire détient actuellement.
</ParamField>

<ParamField body="requested_token_type" type="string" required>
  Indique le type de jeton que vous souhaitez obtenir. Pour On-Behalf-Of Token Exchange, utilisez `urn:ietf:params:oauth:token-type:access_token`.
</ParamField>

<ParamField body="client_id" type="string" required>
  Le Client ID de votre client d’API personnalisée. Le client d’API personnalisée doit être associé à un serveur de ressources (même identifiant). Comme pour les autres types d’octroi, vous pouvez aussi transmettre le Client ID dans l’en-tête Authorization au moyen de l’authentification Basic HTTP.
</ParamField>

<ParamField body="client_secret" type="string" required>
  Le Client Secret de votre client d’API personnalisée. Comme pour les autres types d’octroi, vous pouvez aussi transmettre le Client Secret dans l’en-tête Authorization au moyen de l’authentification Basic `HTTP`. Consultez les autres options dans la [documentation de référence de l’API d’authentification Auth0](https://auth0.com/docs/api/authentication#authentication-methods). Notez que vous ne pouvez pas définir `token_endpoint_auth_method` à `none` pour l’OBO token exchange.
</ParamField>

<ParamField body="audience" type="string" required>
  L’identifiant unique de l’API en aval à laquelle vous voulez accéder. Il s’agit de l’identifiant du service en aval qui reçoit et valide le nouveau jeton.
</ParamField>

<ParamField body="scope" type="string">
  (Facultatif) Une liste de permissions précises, séparées par des espaces, demandées pour l’appel en aval. S’il n’est pas précisé, toutes les portées accordées à l’utilisateur pour l’audience cible seront incluses selon les politiques RBAC.
</ParamField>

<div id="response">
  ## Réponse
</div>

| Statut | Description                                                                                                                                                                                   |
| ------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 200    | Réponse réussie. Renvoie un jeton d&#39;accès pour l&#39;API en aval tout en conservant la même identité utilisateur.                                                                         |
| 400    | Requête invalide. Peut se produire si la profondeur de la chaîne de délégation dépasse la limite maximale autorisée de 4 niveaux imbriqués, ou si des paramètres obligatoires sont manquants. |
| 401    | Non autorisé. Le jeton du sujet est invalide, expiré, ou les identifiants du client sont incorrects.                                                                                          |
| 403    | Interdit. Le client n&#39;est pas autorisé à effectuer l&#39;échange de jetons, ou l&#39;utilisateur ne possède pas les portées demandées pour l&#39;audience cible.                          |

<ResponseExample>
  ```json 200 Réponse
  {
    "access_token": "eyJ...",
    "expires_in": 86400,
    "token_type": "Bearer",
    "issued_token_type": "urn:ietf:params:oauth:token-type:access_token"
  }
  ```

  ```json 400 Réponse (Limite de la chaîne de délégation dépassée)
  {
    "error": "invalid_request",
    "error_description": "Delegation chain (`act` claim) depth exceeds the maximum allowed limit of 5"
  }
  ```
</ResponseExample>

<div id="response-fields">
  ## Champs de réponse
</div>

<ResponseField name="access_token" type="string">
  Le nouvel Auth0 jeton d’accès limité à l’API en aval. Ce [JSON Web Token (JWT)](/docs/fr-CA/secure/tokens/json-web-tokens#json-web-tokens) contient la même valeur `sub` (identité de l’utilisateur) que le jeton du sujet, avec `aud` défini sur l’audience de l’API en aval demandée. La revendication `act` assure le suivi de la chaîne de délégation.
</ResponseField>

<ResponseField name="issued_token_type" type="string">
  Confirme le format du jeton renvoyé. Cette valeur correspond au `requested_token_type` de votre requête.

  Valeur : `urn:ietf:params:oauth:token-type:access_token`
</ResponseField>

<ResponseField name="token_type" type="string">
  Indique le schéma d’authentification à utiliser dans l’en-tête Authorization. Pour OBO, il s’agit de `Bearer`, sauf si vous utilisez DPoP, auquel cas `DPoP` est utilisé.
</ResponseField>

<ResponseField name="expires_in" type="number">
  La durée de validité du jeton, en secondes.
</ResponseField>

<ResponseField name="scope" type="string">
  (Facultatif) Inclus uniquement si les portées accordées diffèrent des portées demandées. Liste des portées réellement accordées, délimitées par des espaces.
</ResponseField>
