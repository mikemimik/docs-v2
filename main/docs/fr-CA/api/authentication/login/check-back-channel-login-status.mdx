---

<div id="endpoint">
  ## Point de terminaison
</div>

`POST /oauth/token`

Pour vérifier l’état d’un flux Back-Channel Login, interrogez régulièrement le point de terminaison `/oauth/token` en transmettant les éléments suivants :

* `auth_req_id` renvoyé par l’appel à `/bc-authorize`
* le grant type `urn:openid:params:grant-type:ciba`

<div id="response-body">
  ### Corps de la réponse
</div>

Si l’utilisateur chargé d’autoriser la requête ne l’a pas encore approuvée ni rejetée, vous devriez recevoir une réponse semblable à celle-ci :

```http
{ 
  "error": "authorization_pending", 
  "error_description": "The end-user authorization is pending"
}
```

Si l’utilisateur autorisant rejette la requête, vous devriez recevoir une réponse semblable à la suivante :

```http
{
  "error": "access_denied",
  "error_description": "The end-user denied the authorization request or it has been expired"
}
```

Si vous faites du polling trop rapidement (à une fréquence supérieure à la valeur d’intervalle renvoyée par `/bc-authorize`), vous devriez recevoir une réponse comme celle-ci :

```http
{
  "error": "slow_down",
  "error_description": "You are polling faster than allowed. Try again in 10 seconds."
}
```

De plus, Auth0 ajoutera l’en-tête [Retry-After](https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers/Retry-After) à la réponse pour indiquer le nombre de secondes à attendre avant de tenter d’interroger de nouveau. Si vous interrogez trop souvent, le nombre de secondes que vous devez attendre augmente.

Si l’utilisateur qui autorise a approuvé la push notification, l’appel renvoie l’ID token et l’access token (et potentiellement un refresh token) :

```http
{
  "access_token": "eyJh...",
  "id_token": "eyJh...",
  "expires_in": 86400,
  "scope": "openid"
}
```

Une fois qu’un `auth_req_id` a été échangé contre un ID ou un jeton d’accès, il ne peut plus être utilisé.

<div id="remarks">
  ### Remarques
</div>

Incluez un paramètre facultatif pour l’authentification de l’application dans la requête :

* Secret client avec l’authentification HTTP Basic, auquel cas aucun paramètre n’est requis. Le `client_id` et le `client_secret` sont transmis dans un en-tête.
* Secret client Post, auquel cas le `client_id` et le `client_secret` sont requis.
* Private Key JWT, où le `client_id`, la `client_assertion` et le type de `client_assertion` sont requis.
* mTLS, où le paramètre `client_id` est requis et où les en-têtes `client-certificate` et `client-certificate-ca-verified` sont requis.

<div id="body-parameters">
  ## Paramètres du corps
</div>

<ParamField body="client_id" type="string" required>
  Le `client_id` de votre application.
</ParamField>

<ParamField body="auth_req_id" type="string" required>
  Le `auth_req_id` renvoyé par le point de terminaison `/bc-authorize`.
</ParamField>

<ParamField body="grant_type" type="string" required>
  Doit être défini sur `urn:openid:params:grant-type:ciba`.
</ParamField>

<div id="response-messages">
  ## Messages de réponse
</div>

| Statut | Description                               |
| ------ | ----------------------------------------- |
| 200    | Statut de l’authentification renvoyé.     |
| 400    | Requête incorrecte - paramètres invalides |
| 500    | Erreur interne du serveur                 |
