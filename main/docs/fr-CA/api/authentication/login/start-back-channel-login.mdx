<div id="endpoint">
  ## Point de terminaison
</div>

`POST /bc-authorize`

Le point de terminaison Back-Channel Login permet aux applications d’envoyer une requête d’authentification au téléphone d’un utilisateur, ou à l’appareil d’authentification, à condition qu’une application y soit installée et que l’utilisateur soit inscrit aux [notifications push à l’aide du SDK Guardian](https://auth0.com/docs/secure/multi-factor-authentication/auth0-guardian#enroll-in-push-notifications).

Utilisez le point de terminaison Back-Channel Login pour authentifier les utilisateurs dans les cas d’utilisation suivants :

* Les utilisateurs ne sont pas devant l’application qui exige une authentification, par exemple lorsqu’ils appellent un centre d’appels.
* L’appareil de consommation, c’est-à-dire l’appareil qui permet à l’utilisateur d’utiliser un service, n’est pas sécuritaire pour les opérations sensibles (p. ex., un navigateur Web pour des transactions financières).
* L’appareil de consommation a des capacités d’interaction limitées (p. ex., les vélos ou les trottinettes électriques).

<div id="response-body">
  ### Corps de la réponse
</div>

Si la requête réussit, vous devriez recevoir une réponse semblable à la suivante :

```http
{
  "auth_req_id": "eyJh...",
  "expires_in": 300,
  "interval": 5
}
```

La valeur `auth_req_id` doit être conservée, car elle est utilisée plus tard dans le flux pour identifier la requête d’authentification.

La valeur `expires_in` indique le nombre de secondes dont vous disposez avant l’expiration de la requête d’authentification.

La valeur `interval` indique le nombre de secondes à attendre entre les requêtes de vérification.

La requête doit être approuvée ou refusée sur l’appareil d’authentification de l’utilisateur à l’aide du Guardian SDK.

<div id="remarks">
  ### Remarques
</div>

* La valeur `login_hint` respecte le format `iss_sub`. Par exemple :

```http
{ "format": "iss_sub", "iss": "https://[TENANT_DOMAIN]/", "sub": "auth0|[USER ID]" }
```

* L’authentification peut se faire au moyen de Client Secret (par HTTP avec authentification Basic ou Post), de Private Key JWT ou de mTLS. Veuillez consulter les exemples détaillés ci-dessus.

<div id="body-parameters">
  ## Paramètres du corps
</div>

<ParamField body="client_id" type="string" required>
  Le `client_id` de votre application.
</ParamField>

<ParamField body="binding_message" type="string" required>
  Une chaîne lisible permettant à l’utilisateur de s’assurer qu’il approuve la bonne requête.
</ParamField>

<ParamField body="login_hint" type="string" required>
  Un objet JSON contenant les renseignements de l’utilisateur pour l’authentification au format `iss_sub`.
</ParamField>

<ParamField body="scope" type="string" required>
  Liste de scopes OIDC et de scopes d’API personnalisées séparés par des espaces, incluant au minimum `openid`.
</ParamField>

<ParamField body="audience" type="string">
  Identifiant unique facultatif de l’API cible pour les jetons d’accès.
</ParamField>

<ParamField body="resource" type="string">
  L’identifiant de l’API cible (serveur de ressources) à laquelle vous voulez accéder. Doit correspondre à un identifiant d’API enregistré dans votre tenant Auth0. Utilisé comme solution de rechange à `audience` lorsque le [profil de compatibilité du paramètre Resource](https://auth0.com/docs/get-started/tenant-settings#settings-advanced) du tenant est défini sur `compatibility`.
</ParamField>

<ParamField body="requested_expiry" type="string">
  Durée d’expiration personnalisée de la requête en secondes (1-300). La valeur par défaut est de 300 secondes.
</ParamField>

<ParamField body="authorization_details" type="array">
  Tableau JSON facultatif d’objets décrivant les permissions à autoriser. La valeur `type` de chaque objet doit avoir été enregistrée au préalable sur le serveur de ressources à l’aide du paramètre `authorization_details` du serveur de ressources. Pour en savoir plus, consultez la documentation de la Management API [Update a resource server](https://auth0.com/docs/api/management/v2/resource-servers/patch-resource-servers-by-id#body-parameters).
</ParamField>

<div id="response-messages">
  ## Messages de réponse
</div>

| Statut | Description                                    |
| ------ | ---------------------------------------------- |
| 200    | Requête d’authentification lancée avec succès. |
| 400    | Requête incorrecte - paramètres non valides    |
| 500    | Erreur interne du serveur                      |
