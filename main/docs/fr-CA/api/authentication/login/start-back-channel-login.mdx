<div id="endpoint">
  ## Point de terminaison
</div>

`POST /bc-authorize`

Le point de terminaison Back-Channel Login permet aux applications d’envoyer une demande d’authentification au téléphone d’un utilisateur, ou à son appareil d’authentification, à condition qu’une application y soit installée et que l’utilisateur soit inscrit aux [notifications push avec le Guardian SDK](https://auth0.com/docs/secure/multi-factor-authentication/auth0-guardian#enroll-in-push-notifications).

Utilisez le point de terminaison Back-Channel Login pour authentifier les utilisateurs dans les cas d’utilisation suivants :

* Les utilisateurs ne se trouvent pas devant l’application qui requiert l’authentification, par exemple lorsqu’ils appellent un centre d’appels.
* L’appareil de consommation, c’est-à-dire l’appareil qui permet à l’utilisateur d’accéder à un service, n’est pas sécuritaire pour les opérations sensibles (p. ex. un navigateur Web pour des transactions financières).
* L’appareil de consommation offre des capacités d’interaction limitées (p. ex. les vélos électriques ou les trottinettes électriques).

<div id="response-body">
  ### Corps de la réponse
</div>

Si la requête aboutit, vous devriez recevoir une réponse semblable à celle-ci :

```http
{
  "auth_req_id": "eyJh...",
  "expires_in": 300,
  "interval": 5
}
```

La valeur `auth_req_id` doit être conservée, car elle est utilisée plus tard dans le flux pour identifier la requête d’authentification.

La valeur `expires_in` indique dans combien de secondes la requête d’authentification expirera.

La valeur `interval` indique combien de secondes vous devez attendre entre chaque requête de vérification.

La requête doit être approuvée ou rejetée sur l’appareil d’authentification de l’utilisateur à l’aide du Guardian SDK.

<div id="remarks">
  ### Remarques
</div>

* La valeur `login_hint` respecte le format `iss_sub`. Par exemple :

```http
{ "format": "iss_sub", "iss": "https://[TENANT_DOMAIN]/", "sub": "auth0|[USER ID]" }
```

* L’authentification peut utiliser le Client Secret (au moyen de l’authentification Basic HTTP ou de Post), Private Key JWT ou mTLS. Veuillez consulter les exemples détaillés ci-dessus.

<div id="body-parameters">
  ## Paramètres du corps
</div>

<ParamField body="client_id" type="string" required>
  Le `client_id` de votre application.
</ParamField>

<ParamField body="binding_message" type="string" required>
  Une chaîne lisible pour que l’utilisateur puisse s’assurer qu’il approuve la bonne request.
</ParamField>

<ParamField body="login_hint" type="string" required>
  Un objet JSON contenant les détails de l’utilisateur pour l’authentication au format `iss_sub`.
</ParamField>

<ParamField body="scope" type="string" required>
  Liste de scopes OIDC et de scopes d’API personnalisée séparés par des espaces, incluant au minimum `openid`.
</ParamField>

<ParamField body="audience" type="string">
  Identifiant unique facultatif de l’API cible pour les jetons d’accès.
</ParamField>

<ParamField body="resource" type="string">
  L’identifiant de l’API cible (serveur de ressources) à laquelle vous souhaitez accéder. Il doit correspondre à un identifiant de l’API enregistré dans votre tenant Auth0. Il est utilisé comme solution de rechange à `audience` lorsque le [profil de compatibilité du paramètre resource](https://auth0.com/docs/get-started/tenant-settings#settings-advanced) du tenant est défini à `compatibility`.
</ParamField>

<ParamField body="requested_expiry" type="string">
  Délai d’expiration personnalisé de la request, en secondes (1 à 300). La valeur par défaut est de 300 secondes.
</ParamField>

<ParamField body="authorization_details" type="array">
  Tableau JSON facultatif d’objets décrivant les permissions à autoriser. La valeur `type` de chaque objet doit avoir été enregistrée au préalable sur le serveur de ressources à l’aide du paramètre `authorization_details` du serveur de ressources. Pour en savoir plus, consultez la documentation de la Management API sur [Update a resource server](https://auth0.com/docs/api/management/v2/resource-servers/patch-resource-servers-by-id#body-parameters).
</ParamField>

<div id="response-messages">
  ## Messages de réponse
</div>

| Statut | Description                                    |
| ------ | ---------------------------------------------- |
| 200    | Requête d’authentification lancée avec succès. |
| 400    | Requête incorrecte - paramètres non valides    |
| 500    | Erreur interne du serveur                      |
