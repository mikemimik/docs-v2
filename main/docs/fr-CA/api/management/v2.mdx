<Badge>Version : 2.0 (actuelle)</Badge>

**L’Auth0 Management API** est un ensemble d’endpoints permettant d’effectuer des tâches administratives par programmation et doit être utilisée par des serveurs backend ou des entités de confiance. De façon générale, tout ce qui peut être fait dans l’Auth0 Dashboard peut aussi être fait à l’aide de cette API.

Cette API est distincte de l’[Auth0 Authentication API accessible au public](https://auth0.com/docs/api/authentication), qui est destinée aux frontends et aux entités non fiables.

Lorsque vous utilisez les exemples de code inclus dans cette documentation d’API, les requêtes doivent être envoyées avec un `Content-Type` de `application/json`. Tous les endpoints acceptent un payload d’une taille maximale de 1 mégaoctet.

La documentation de l’Auth0 Management API suit le [schéma OpenAPI v3.1 de l’Auth0 Management API](https://auth0.com/docs/api/management/openapi.json). Veuillez noter que la prise en charge du schéma OpenAPI v3.1 est actuellement en version Beta.

<div id="authentication">
  ## Authentification
</div>

L’utilisation de l’Auth0 Management API nécessite un jeton d’accès à l’API de gestion. Pour savoir comment demander ce jeton, consultez [Jetons d’accès à l’API de gestion](https://auth0.com/docs/secure/tokens/access-tokens/management-api-access-tokens).

L’Auth0 Management API utilise des JSON Web Tokens (JWT) pour authentifier les requêtes. La revendication scopes du jeton d’accès à l’API de gestion indique quelles méthodes de requête peuvent être utilisées lors de l’appel à cette API. L’exemple de jeton désérialisé présenté sur cette page accorde un accès en lecture seule aux utilisateurs et un accès en lecture/écriture aux connexions. Toute tentative d’utiliser une méthode de requête non autorisée par les scopes définis entraînera une réponse **403 Forbidden**.

```bash lines
{
  "aud": "m8DAxghyfE0KdpzogfXgMSxrkCSdKVEF",
  "scopes": {
    "connections": {
      "actions": ["read", "update"]
    }
  },
  "iat": "1446056652",
  "jti": "7e9c6a991f5a227fb7ebaa522536ae4c"
}
```

Pour appeler l’API, envoyez le jeton d’API dans l’en-tête HTTP Authorization à l’aide du [schéma d’authentification Bearer](https://tools.ietf.org/html/draft-ietf-oauth-v2-bearer-20#section-2.1).

```bash lines
curl -H "Authorization: Bearer eyJhb..." https://@@TENANT@@/api/v2/users
```

<div id="request-correlation">
  ## Corrélation des requêtes
</div>

Un Correlation ID est un identifiant unique (jusqu’à 64 caractères) associé à une seule opération de la Management API et permet d’en assurer le suivi dans les logs du tenant. Pour en savoir plus, consultez [Logs](https://auth0.com/docs/deploy-monitor/logs).

L’API accepte un Correlation ID fourni par le client lorsqu’il est envoyé dans l’en-tête HTTP `X-Correlation-ID` avec les méthodes `POST`, `PUT`, `PATCH` et `DELETE`.

```bash lines
curl -H "Authorization: Bearer eyJhb..." -H "x-correlation-id: client1_xyz" https://@@TENANT@@/api/v2/users
```

Si une valeur d’en-tête `X-Correlation-ID` de plus de 64 caractères est fournie, seuls les 64 premiers caractères seront affichés dans les logs.

```bash lines
"references": {
  "correlation_id": "client1_xyz"
}
```

<div id="pagination">
  ## Pagination
</div>

La pagination est une technique utilisée par les API pour diviser de grands ensembles de données en pages faciles à gérer, ce qui réduit la quantité de données renvoyée dans chaque réponse. Deux grands types de pagination sont couramment utilisés dans les API : la pagination **par décalage** et la pagination **par point de contrôle**. Chacune présente des avantages distincts et convient à différents cas d’utilisation selon la taille de l’ensemble de données et les exigences de récupération.

L’Auth0 Management API prend en charge les deux types de pagination sur de nombreux points de terminaison, comme `GET /api/v2/clients` et `GET /api/v2/logs`. Lorsque les deux options sont offertes, la **pagination par point de contrôle** est recommandée en raison de sa plus grande efficacité et de sa stabilité avec les grands ensembles de données.

<div id="offset-based-pagination">
  ### Pagination par décalage
</div>

La pagination par décalage est une méthode simple et couramment utilisée pour paginer des ensembles de données allant jusqu’à environ 1 000 éléments. Cette approche utilise les paramètres `page` et `per_page` pour définir le point de départ et le nombre d’éléments sur chaque page.

* **Paramètres :**
  * `page` : Le numéro de page à récupérer, indexé à partir de zéro. La valeur par défaut est `0` s’il n’est pas précisé.
  * `per_page` : Le nombre d’éléments à renvoyer par page. Pour les tenants Public Cloud, [le maximum est de `50`](https://auth0.com/docs/troubleshoot/product-lifecycle/past-migrations/migrate-to-paginated-queries); pour Private Cloud, le maximum est de `100`. S’il n’est pas précisé, la valeur par défaut correspond à la moitié du maximum.

**Exemple de requête de pagination par décalage :**

```bash lines
curl -L "https://@@TENANT@@/api/v2/clients?per_page=10&page=2" \
-H 'Authorization: Bearer {ACCESS_TOKEN}' \
-H 'Accept: application/json'
```

Dans la pagination par décalage :

* Si `page * per_page` dépasse le nombre total de résultats, un tableau vide est renvoyé.
* Chaque requête de page recalcule le décalage, ce qui peut nuire aux performances lorsque les ensembles de données sont plus volumineux. La pagination par décalage convient généralement mieux aux collections qui risquent peu de dépasser 1 000 éléments.

<div id="checkpoint-based-pagination">
  ### Pagination basée sur un checkpoint
</div>

La pagination basée sur un checkpoint, aussi appelée pagination basée sur un curseur ou un token, est optimisée pour les grands ensembles de données. Cette méthode utilise un checkpoint ID `next` fourni par le serveur pour récupérer les pages suivantes dans une séquence unidirectionnelle. Le checkpoint ID `next` est inclus dans la réponse lorsque d’autres résultats sont disponibles.

Pour poursuivre la pagination, utilisez le checkpoint ID `next` dans le paramètre de requête `from` de la requête suivante. Cet ID est opaque et doit être transmis sans modification.

* **Paramètres :**
  * `from` : Le checkpoint ID `next` de la réponse précédente, utilisé pour récupérer la page de résultats suivante.
  * `take` : Le nombre d’éléments à renvoyer par page. Pour les tenants Public Cloud, [le maximum est de `50`](https://auth0.com/docs/troubleshoot/product-lifecycle/past-migrations/migrate-to-paginated-queries) ; pour Private Cloud, le maximum est de `100`. Par défaut, la valeur correspond à la moitié du maximum si elle n’est pas fournie.

**Exemple de requête de pagination basée sur un checkpoint :**

```bash lines
curl -L "https://@@TENANT@@/api/v2/clients?take=10&from=Cg1HRUY3NEszUERFME40GgAiAQgCEj..." \
-H 'Authorization: Bearer {ACCESS_TOKEN}' \
-H 'Accept: application/json'
```

<div id="checkpoint-id-expiry">
  #### Expiration de l’ID de point de contrôle
</div>

Lorsque vous utilisez la pagination basée sur des points de contrôle, il est important de connaître la durée de validité de chaque ID de point de contrôle `next`. L’ID de point de contrôle est conçu pour être utilisé de façon séquentielle, chaque ID n’étant valide que pendant une période limitée afin d’assurer la cohérence des données.

<Note>
  <p class="uppercase font-bold">Remarque</p>

  L’ID de point de contrôle `next` est valide pendant **24 heures** après son émission. S’il expire, une nouvelle requête est nécessaire pour recommencer au début de l’ensemble de données. Envisagez la mise en cache des résultats si un long délai risque de s’écouler entre les requêtes.
</Note>

<div id="forward-only-constraints">
  #### Contraintes de pagination vers l’avant uniquement
</div>

La pagination par checkpoint se fait uniquement vers l’avant. Évitez d’utiliser l’ID de checkpoint pour revenir en arrière ou effectuer des requêtes hors séquence, car cela peut entraîner des erreurs. Utilisez toujours l’ID de checkpoint `next` de la réponse précédente.

<div id="choosing-between-offset-and-checkpoint-pagination">
  ### Choisir entre la pagination par offset et la pagination par checkpoint
</div>

Lorsque les deux types de pagination sont pris en charge :

* **Utilisez la pagination par checkpoint** pour gérer efficacement de grands ensembles de données.
* Utilisez la pagination par offset pour les ensembles de données plus petits (généralement moins de 1 000 éléments), car elle est plus simple à mettre en œuvre, mais moins efficace pour les grandes collections.

<div id="best-practices-for-handling-pagination">
  ### Best Practices pour gérer la pagination
</div>

* **Cohérence des données :** Chaque request paginée reflète les données au moment de la request. Si des données sont mises à jour ou supprimées, certains éléments peuvent être omis ou apparaître en double. Checkpoint-based pagination peut aider à assurer une pagination plus fluide dans des ensembles de données dynamiques.
* **Stockage des checkpoints :** Pour récupérer de grands volumes de données, envisagez de stocker des checkpoints après chaque page afin de pouvoir reprendre à partir du dernier checkpoint en cas d’interruption.

Cette approche permet une récupération de données efficace et stable pour les grands ensembles de données, en adéquation avec les options de pagination de l’Auth0 Management API.
