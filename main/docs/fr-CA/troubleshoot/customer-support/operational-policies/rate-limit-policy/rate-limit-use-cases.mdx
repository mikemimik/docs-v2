<div id="discover-when-requests-to-a-tenant-are-rate-limited">
  ## Déterminez quand les requêtes vers un tenant font l’objet d’une limitation de débit
</div>

Il existe plusieurs façons de déterminer si le produit d’un client fait l’objet d’une limitation de débit par Auth0. Consultez ci-dessous les causes possibles de cette limitation.

<div id="tenant-logs">
  ### Journaux du tenant
</div>

En vous abonnant à différents journaux du tenant, vous pouvez suivre les problèmes liés au volume de requêtes. Pour comprendre le fonctionnement des journaux d’événements du tenant et des opérations dans Auth0, consultez [Journaux](/docs/fr-CA/deploy-monitor/logs).

<div id="api_limit">
  #### api_limit
</div>

L’événement `api_limit` est déclenché dès qu’une limite de débit est dépassée pour le bucket global de limite de débit de l’authentication ou de la <Tooltip tip="Management API : Un produit permettant aux clients d’effectuer des tâches administratives." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Management+API">Management API</Tooltip>. Si le nombre de jetons de requête utilisés reste supérieur à 80 % après une minute pour ce même bucket de limite de débit, un deuxième log d’avertissement est généré. Si une limite de débit est dépassée pour un autre bucket de limite de débit, un nouvel événement `api_limit` est généré. Cela aide les clients à déterminer quelle configuration de limite de débit est déclenchée par leurs appels d’API, ce qui constitue une première étape cruciale pour diagnostiquer la cause première.

<div id="api_limit_warning">
  #### api_limit_warning
</div>

Le log `api_limit_warning` est déclenché lorsque le volume de requêtes d’un client consomme 80 % des jetons de requête d’un bucket de limitation du débit donné. Si le nombre de jetons de requête utilisés reste au-dessus de 80 % après une minute pour le même bucket de limitation du débit, un deuxième log d’avertissement est généré. Si le seuil de 80 % est dépassé pour un bucket de limitation du débit différent, un nouveau log `api_limit_warning` est créé.

<div id="appi-public-performance-burst-only">
  #### appi (Public Performance Burst seulement)
</div>

Le log `appi` est déclenché lorsqu’un tenant client doté de l’add-on Public Performance Burst dépasse sa limite soutenue de 100 RPS pour les requêtes de l’Authentication API, ce qui consomme un bloc d’une minute de son allocation de burst de 48 heures. Si, après 15 minutes, le taux de requêtes dépasse de nouveau la limite soutenue de 100 RPS, un deuxième log `appi` est déclenché.

<div id="api-responses">
  ### Réponses de l’API
</div>

Les réponses de l’Auth0 API renvoient un code [HTTP 429 (Too Many Requests)](http://tools.ietf.org/html/rfc6585#section-4) lorsqu’une limite de requêtes est dépassée. Cela permet aux clients de constater en temps réel l’application des limites de requêtes. Toutefois, cela n’est utile que pour les applications client développées sur mesure qui interagissent directement avec l’Auth0 API.

<div id="sdk-error-handling">
  ### Gestion des erreurs du SDK
</div>

Si vous utilisez un SDK, consultez les pages d’erreur des [bibliothèques du SDK de la Management API](/docs/fr-CA/libraries#mgmt).

<div id="error-pages">
  ### Pages d’erreur
</div>

Une réponse de page d’erreur est renvoyée par les points de terminaison qui affichent du contenu HTML à l’utilisateur final. Si votre tenant est configuré pour utiliser des pages génériques (hébergées par Auth0), Auth0 affiche la page d’erreur au lieu du contenu attendu lorsque vous dépassez la limite de réponse.  Si votre tenant est configuré pour utiliser des [pages d’erreur personnalisées](/docs/fr-CA/customize/login-pages/custom-error-pages), l’utilisateur est redirigé vers l’URL de la page d’erreur personnalisée avec l’erreur correspondante dans le paramètre de chaîne de requête `error_description`.  Pour en savoir plus, consultez les sections [Points de terminaison concernés](/docs/fr-CA/troubleshoot/customer-support/operational-policies/rate-limit-policy/authentication-api-endpoint-rate-limits#affected-endpoints) et [Erreur JSON](/docs/fr-CA/troubleshoot/customer-support/operational-policies/rate-limit-policy/authentication-api-endpoint-rate-limits#json-error).

<div id="find-out-why-a-tenant-is-being-rate-limited">
  ## Découvrez pourquoi un tenant fait l’objet d’une limitation de débit
</div>

Si vous pensez que les requêtes d’un tenant font l’objet d’une limitation de débit et que vous avez besoin de soutien pour comprendre pourquoi, ouvrez une demande dans le [Support Center](http://support.auth0.com).  Dans votre demande, veuillez inclure le log brut complet dans lequel le problème s’est produit.

<div id="predict-when-requests-to-a-tenant-will-be-rate-limited">
  ## Prédire quand des requêtes adressées à un tenant seront soumises à une limite de débit
</div>

Auth0 fournit des renseignements à jour sur l’état actuel de vos limites de débit au moyen des en-têtes de réponse HTTP provenant des points de terminaison pour lesquels des politiques de limite de débit sont configurées. Cet état est communiqué comme suit :

* `x-ratelimit-limit` : Nombre maximal de requêtes disponibles.
* `x-ratelimit-remaining` : Nombre de requêtes restantes jusqu’à ce que le bucket soit de nouveau rempli avec des requêtes additionnelles.
* `x-ratelimit-reset` : [Horodatage UNIX](https://en.wikipedia.org/wiki/Unix_time), en secondes, correspondant au moment prévu où des requêtes additionnelles seront ajoutées au bucket.

Par exemple :

Une API a la limite de débit suivante :

* Limite de rafale : `1000`
* Sustained Rate Limit : `100` `requêtes par seconde` (sur une fenêtre fixe)

À partir de cette information, vous pouvez déduire ce qui suit :

* La limite de débit soutenue est de `100 requêtes par seconde` sur une fenêtre fixe.
* En raison de la fenêtre fixe, le bucket de requêtes est rempli de nouveau chaque seconde.

Si vous recevez les en-têtes x- suivants dans la réponse de votre API :

* `x-ratelimit-limit: 1000`
* `x-ratelimit-remaining: 50`
* `x-ratelimit-reset: 1675452600`

Vous savez maintenant que :

* Votre tenant a utilisé 950 des 1000 requêtes permises pour cette API, et il ne lui reste que 50 requêtes jusqu’à ce que des requêtes additionnelles soient ajoutées.
* De nouvelles requêtes seront ajoutées à `1675452600`, soit à 19:30:00 UTC le 3 février 2023.
* 1 nouvelle requête sera ajoutée à ce moment-là

Par conséquent, si vous effectuez des requêtes à un rythme supérieur à ce qui est décrit ci-dessus, il faut vous attendre à une limitation de débit. Le délai avant que vos requêtes soient limitées dépend de la limite de rafale et de l’ampleur du dépassement de la limite soutenue.

<div id="examples-of-how-rate-limits-are-enforced">
  ## Exemples d’application des limites de taux
</div>

<div id="requests-per-second-example">
  ### Exemple de requêtes par seconde
</div>

Supposons qu’Auth0 lance une nouvelle API appelée `/ratelimitexample` avec les valeurs de limite de débit suivantes :

* Limite de rafale : cinq (5) requêtes
* Limite de débit soutenue : 10 requêtes par seconde.

Points clés :

* L’API commence avec cinq jetons de requête et n’en dépassera jamais ce nombre, ce qui correspond à la limite de rafale.
* Le réservoir de 10 jetons est rechargé chaque seconde au moyen d’une « fenêtre fixe ». De nouveaux jetons sont ajoutés au réservoir, qui est rempli de nouveau au « début » de chaque seconde.

 Exemple de scénario avec des limites de débit :

<Frame>![](/docs/images/cdy7uua7fh8z/30m5xST81Db6mROVGCV5Bj/34d9638eb7467f02c2d0ecbe87ff0fd8/Examples_of_how_rate_limits_are_enforced_-_Page_1.png)</Frame>

Dans ce scénario :

* T0 - T1sec :  L’utilisateur final effectue six requêtes pendant la première seconde.  Cinq requêtes – soit la limite de rafale – reçoivent une réponse `200`.  La sixième requête reçoit une erreur `429`, car il ne reste plus de jetons de requête dans le réservoir.
* T1sec - T2sec :  Auth0 remplit de nouveau le réservoir de jetons de requête en raison de l’algorithme de fenêtre fixe. Par conséquent, les requêtes 7 à 11 réussissent, puis la 12e épuise le réservoir, ce qui entraîne une erreur `429`.
* T2sec - T3sec :  Auth0 remplit de nouveau le réservoir de jetons, et la requête suivante (13) reçoit une réponse `200`.

<div id="requests-per-minute-example">
  ### Exemple de requêtes par minute
</div>

Supposons qu’Auth0 lance une nouvelle API appelée `/ratelimitexample2` avec les valeurs de limite de débit suivantes :

* Limite de rafale :  Cinq (5) requêtes
* Limite de débit soutenue :  Six (6) requêtes par minute.

Points clés :

* L’API commence avec cinq jetons de requête, ce qui correspond à la limite de rafale.
* Le réservoir de six jetons est rempli de nouveau chaque minute, selon une « fenêtre fixe ». De nouveaux jetons sont ajoutés au réservoir, qui est ainsi rempli de nouveau au « début » de chaque minute.

Exemple de scénario avec limites de débit :

<Frame>![](/docs/images/cdy7uua7fh8z/PhtwGBwS9PfEpNQbPeA1o/dc2b05c72ef960a19958dbafa0295e70/Examples_of_how_rate_limits_are_enforced_-_Page_1__1_.png)</Frame>

Dans ce scénario :

* T0 - T+1min :  L’utilisateur final effectue six requêtes pendant la première minute. Cinq requêtes — soit la limite de rafale — reçoivent une réponse `200`. La sixième requête reçoit une erreur `429` parce qu’il ne reste plus de jetons de requête.
* T+1min - T+2min :  Auth0 remplit de nouveau le réservoir de jetons en raison de l’algorithme de fenêtre fixe. Par conséquent, les 7e à 11e requêtes réussissent, ce qui épuise le réservoir à la 12e requête et entraîne une erreur `429`.
* T+2min : Auth0 remplit de nouveau le réservoir de jetons, et la requête suivante (13) reçoit une réponse `200`.

<div id="other-scenarios">
  ### Autres scénarios
</div>

Il arrive parfois qu’Auth0 attribue deux limites de débit à une seule API.  Cela permet de configurer une limite de rafale et une limite de débit soutenue mieux adaptées aux besoins du service.  En pratique, la première limite de débit devient la limite de rafale effective, et la deuxième limite de débit devient la limite de débit soutenue effective.  Dans ce scénario, Auth0 publie uniquement les limites de rafale et de débit soutenue effectives, plutôt que de communiquer les limites de rafale et de débit soutenue réelles.

<div id="end-user-login-and-signup-api-usage">
  ## Utilisation de l’API pour le Login et le Signup de l’utilisateur final
</div>

Il existe quelques flux d’authentification, notamment Login, Signup et Change Password. Les plus courants sont généralement Login, suivi de Signup.

Le login d’un utilisateur final déclenche plusieurs appels d’API vers des points de terminaison de l’Authentication API afin de déterminer si l’utilisateur final est autorisé à recevoir un jeton d’autorisation et, ainsi, à accéder à l’application demandée.

Le nombre exact d’appels d’API dépend de quelques configurations :

* Expérience d’authentification (p. ex., New <Tooltip tip="Universal Login : votre application redirige vers Universal Login, hébergé sur l’Authorization Server d’Auth0, pour vérifier l’identité d’un utilisateur." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Universal+Login">Universal Login</Tooltip> ou Classic Login)
* Flux d’authentification (p. ex., Login, Signup ou Change Password)
* Type de flux d’authentification (p. ex., Login au moyen d’un Username / Password; Login au moyen de Social Login; Login lorsqu’un jeton d’authentification existe déjà)

Ci-dessous, nous décrivons quelques configurations client courantes et leur incidence sur l’utilisation de l’API.

<div id="universal-login">
  ### Universal Login
</div>

[Auth0 Universal Login](/docs/fr-CA/authenticate/login/auth0-universal-login) offre l’une des fonctionnalités essentielles d’un <Tooltip tip="Serveur d’autorisation : serveur centralisé qui contribue à définir les limites de l’accès d’un utilisateur. Par exemple, votre serveur d’autorisation peut contrôler les données, les tâches et les fonctionnalités offertes à un utilisateur." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=authorization+server">serveur d’autorisation</Tooltip> : le flux de connexion. Lorsqu’un utilisateur doit prouver son identité pour accéder à votre application, vous pouvez le rediriger vers Universal Login et laisser Auth0 gérer le processus d’authentification.

| Flux d’authentification | Type de flux                                                | Requêtes aux points de terminaison de l’Authentication API |
| ----------------------- | ----------------------------------------------------------- | ---------------------------------------------------------- |
| Login                   | Défi par nom d’utilisateur et mot de passe*                 | 5                                                          |
| Login                   | Fournisseur d’identité tiers – p. ex., Social ou Work Login | 6                                                          |
| Login                   | Une session d’authentification Auth0 est déjà active        | 1                                                          |
| Signup                  | par nom d’utilisateur et mot de passe                       | 6                                                          |

<div id="modifiers">
  #### Modificateurs
</div>

Certaines configurations d’authentification modifient le nombre de requêtes de base. Ces ajustements dépendent de mesures de sécurité supplémentaires ou de flux d’authentification :

| Modificateur               | Description                                                                                                          | Requêtes supplémentaires |
| -------------------------- | -------------------------------------------------------------------------------------------------------------------- | ------------------------ |
| **ID First**               | Identifie l’utilisateur avant de demander ses identifiants.                                                          | +2                       |
| **MFA**                    | Ajoute l’authentification multifacteur.                                                                              | +2 par facteur           |
| **OTP**                    | Mot de passe à usage unique pour l’authentification                                                                  | +2                       |
| **Connexion d’entreprise** | Authentification au moyen d’une connexion d’entreprise (p. ex., SAML, OIDC, LDAP).                                   | +1                       |
| **Identifiants du client** | Utilisé pour l’authentification machine à machine. S’applique dans tous les cas, même si des Actions sont utilisées. | +1                       |

*Tout élément utilisé en combinaison augmente le nombre total de requêtes.

<div id="classic-login">
  ### Classic Login
</div>

[Classic Login](/docs/fr-CA/authenticate/login/auth0-universal-login/universal-login-vs-classic-login/classic-experience) est une expérience de connexion hébergée par Auth0 qui s’appuie sur JavaScript pour la personnalisation. La mise en œuvre de Classic Login est moins complexe que l’intégration directe du processus d’authentification dans votre application et peut aider à éviter les risques liés à l’authentification cross-origin.

| Flux d’authentification | Type de flux                                                | Requêtes |
| ----------------------- | ----------------------------------------------------------- | -------- |
| Login                   | Vérification par nom d’utilisateur et mot de passe          | 8        |
| Login                   | Fournisseur d’identité tiers – p. ex., Social ou Work Login | 8        |
| Login                   | Session d’authentification Auth0 existante                  | 2        |
| Signup                  | Nom d’utilisateur/mot de passe                              | 8        |

<Warning>
  Les clients qui configurent Custom Databases doivent ajouter deux (2) appels supplémentaires à l’Authentication API pour chaque flux d’authentification décrit ci-dessus. Pour en savoir plus, consultez [Custom Database Connections](/docs/fr-CA/authenticate/database-connections/custom-db).
</Warning>

<div id="modifiers">
  #### Modificateurs
</div>

Les facteurs suivants augmentent le nombre de requêtes pour Classic Login :

| Modificateur                            | Description                                                                                   | Requêtes supplémentaires |
| --------------------------------------- | --------------------------------------------------------------------------------------------- | ------------------------ |
| **Authentification par SMS uniquement** | Lorsque le SMS est utilisé comme principale méthode d’authentification.                       | +7                       |
| **Native Social Login**                 | Connexion à l’aide d’un fournisseur social natif (p. ex., Google, Facebook).                  | +1                       |
| **Redirections**                        | Des redirections supplémentaires pendant l’authentification augmentent le nombre de requêtes. | +1                       |
