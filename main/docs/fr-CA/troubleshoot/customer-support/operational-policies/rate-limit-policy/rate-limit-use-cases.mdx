<div id="discover-when-requests-to-a-tenant-are-rate-limited">
  ## Découvrez quand les requêtes envoyées à un tenant font l’objet d’une limitation du débit
</div>

Il existe plusieurs façons de déterminer si le produit d’un client fait l’objet d’une limitation du débit par Auth0. Voir ci-dessous les causes possibles de cette limitation du débit.

<div id="tenant-logs">
  ### Journaux du tenant
</div>

En vous abonnant à différents journaux du tenant, vous pouvez repérer les problèmes liés au volume des requêtes. Pour comprendre comment fonctionnent les journaux d’événements du tenant et des opérations dans Auth0, consultez [Logs](/docs/fr-CA/deploy-monitor/logs).

<div id="api_limit">
  #### api_limit
</div>

L’événement `api_limit` est déclenché immédiatement après le dépassement de la limite de débit du bucket global de limite de débit pour l’authentification ou la <Tooltip tip="Management API : Un produit permettant aux clients d’effectuer des tâches administratives." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Management+API">Management API</Tooltip>. Si le nombre de jetons de requête utilisés demeure supérieur à 80 % après une minute pour le même bucket de limite de débit, un deuxième log d’avertissement est généré. Si une limite de débit est dépassée pour un bucket de limite de débit différent, un nouvel événement `api_limit` est généré. Cela aide les clients à déterminer quelle configuration de limite de débit est déclenchée par leurs appels d’API, ce qui constitue une première étape essentielle pour diagnostiquer la cause première.

<div id="api_limit_warning">
  #### api_limit_warning
</div>

L’entrée de journal `api_limit_warning` est déclenchée lorsque le taux de requêtes d’un client consomme 80 % des jetons de requête pour un rate limit bucket donné. Si le nombre de jetons de requête utilisés demeure supérieur à 80 % après une minute pour le même rate limit bucket, une deuxième entrée de journal d’avertissement est générée. Si le seuil de 80 % est dépassé pour un autre rate limit bucket, une nouvelle entrée de journal `api_limit_warning` est créée.

<div id="appi-public-performance-burst-only">
  #### appi (Public Performance Burst uniquement)
</div>

Le log `appi` se déclenche lorsqu’un tenant client doté de l’add-on Public Performance Burst dépasse sa limite soutenue de 100 RPS pour les requêtes de l’Authentication API, ce qui consomme un bloc d’une minute de son allocation de rafale de 48 heures. Si, après 15 minutes, le débit de requêtes dépasse de nouveau la limite soutenue de 100 RPS, un deuxième log `appi` se déclenche.

<div id="api-responses">
  ### Réponses de l’API
</div>

Les réponses de l’Auth0 API renvoient le code [HTTP 429 (Trop de requêtes)](http://tools.ietf.org/html/rfc6585#section-4) lorsque la limite de requêtes est dépassée. Cela permet aux clients de constater en temps réel l’application de cette limite. Toutefois, cela n’est utile que pour les applications clients personnalisées qui interagissent directement avec l’Auth0 API.

<div id="sdk-error-handling">
  ### Gestion des erreurs du SDK
</div>

Si vous utilisez un SDK, consultez les pages sur les erreurs des [bibliothèques SDK de la Management API](/docs/fr-CA/libraries#mgmt).

<div id="error-pages">
  ### Pages d’erreur
</div>

La réponse de page d’erreur est envoyée pour les points de terminaison qui affichent du contenu HTML à l’utilisateur final. Si votre tenant est configuré pour utiliser des pages génériques (hébergées par Auth0), Auth0 affiche la page d’erreur au lieu du contenu attendu lorsque vous dépassez la limite de réponse.  Si votre tenant est configuré pour utiliser des [pages d’erreur personnalisées](/docs/fr-CA/customize/login-pages/custom-error-pages), l’utilisateur est redirigé vers l’URL de la page d’erreur personnalisée avec l’erreur pertinente dans le paramètre de la chaîne de requête `error_description`.  Pour en savoir plus, consultez les sections [Points de terminaison touchés](/docs/fr-CA/troubleshoot/customer-support/operational-policies/rate-limit-policy/authentication-api-endpoint-rate-limits#affected-endpoints) et [Erreur JSON](/docs/fr-CA/troubleshoot/customer-support/operational-policies/rate-limit-policy/authentication-api-endpoint-rate-limits#json-error).

<div id="find-out-why-a-tenant-is-being-rate-limited">
  ## Découvrez pourquoi un tenant est soumis à une limitation de débit
</div>

Si vous pensez que les requêtes du tenant font l’objet d’une limitation de débit et que vous avez besoin de soutien pour en comprendre la raison, ouvrez une requête par l’intermédiaire du [Support Center](http://support.auth0.com).  Dans votre requête, veuillez inclure le journal brut complet où le problème a été constaté.

<div id="predict-when-requests-to-a-tenant-will-be-rate-limited">
  ## Prévoir quand des requêtes vers un tenant feront l’objet d’une limitation de débit
</div>

Auth0 fournit des renseignements à jour sur l’état actuel de vos limites de débit au moyen des en-têtes de réponse HTTP des endpoints pour lesquels des politiques de limite de débit sont configurées. Cet état est communiqué comme suit :

* `x-ratelimit-limit` : Nombre maximal de requêtes disponibles.
* `x-ratelimit-remaining` : Nombre de requêtes restantes jusqu’à ce que le bucket soit réapprovisionné en requêtes additionnelles.
* `x-ratelimit-reset` : [horodatage UNIX](https://en.wikipedia.org/wiki/Unix_time), en secondes, indiquant le moment prévu où des requêtes additionnelles seront ajoutées au bucket.

Par exemple :

Une API a la limite de débit suivante :

* limite en rafale : `1000`
* limite de débit soutenue : `100` `requêtes par seconde` (sur une fenêtre fixe)

À partir de cette information, vous pouvez déduire ce qui suit :

* La limite de débit soutenue est de `100 requêtes par seconde` sur une fenêtre fixe.
* En raison de la fenêtre fixe, le bucket de requêtes est réapprovisionné chaque seconde.

Si vous recevez les x-headers suivants dans votre réponse d’API :

* `x-ratelimit-limit: 1000`
* `x-ratelimit-remaining: 50`
* `x-ratelimit-reset: 1675452600`

Vous savez maintenant que :

* Votre tenant a utilisé 950 des 1000 requêtes autorisées pour cette API, et il ne lui reste que 50 requêtes avant que des requêtes additionnelles soient ajoutées.
* De nouvelles requêtes seront ajoutées à `1675452600`, soit à 19 h 30 min 00 s UTC le 3 février 2023.
* 1 nouvelle requête sera ajoutée à ce moment-là

Par conséquent, si vous effectuez des requêtes à un rythme supérieur à celui décrit ci-dessus, il faut vous attendre à une limitation de débit. Le délai avant qu’une limitation de débit s’applique dépend de la limite en rafale et de l’ampleur du dépassement de la limite de débit soutenue.

<div id="examples-of-how-rate-limits-are-enforced">
  ## Exemples de l’application des limites de débit
</div>

<div id="requests-per-second-example">
  ### Exemple de requêtes par seconde
</div>

Supposons qu’Auth0 lance une nouvelle API appelée `/ratelimitexample` avec les valeurs de limite de débit suivantes :

* Limite en rafale : cinq (5) requêtes
* Limite de débit soutenue : 10 requêtes par seconde.

Points clés :

* L’API démarre avec cinq jetons de requête et n’en comptera jamais plus, ce qui correspond à la limite en rafale.
* La réserve de 10 jetons est reconstituée chaque seconde, selon une « fenêtre fixe ». De nouveaux jetons sont ajoutés à la réserve, qui est remplie de nouveau au « début » de chaque seconde.

 Exemple de scénario avec des limites de débit :

<Frame>![](/docs/images/cdy7uua7fh8z/30m5xST81Db6mROVGCV5Bj/34d9638eb7467f02c2d0ecbe87ff0fd8/Examples_of_how_rate_limits_are_enforced_-_Page_1.png)</Frame>

Dans ce scénario :

* T0 - T1sec :  L’utilisateur final effectue six requêtes pendant la première seconde.  Cinq requêtes – soit la limite en rafale – reçoivent une réponse `200`.  La sixième requête reçoit une erreur `429`, car il ne reste plus de jetons de requête dans la réserve.
* T1sec - T2sec :  Auth0 remplit de nouveau la réserve de jetons de requête en raison de l’algorithme de fenêtre fixe. Par conséquent, les 7e à 11e requêtes réussissent, la 12e épuisant la réserve et entraînant une erreur `429`.
* T2sec - T3sec :  Auth0 remplit de nouveau la réserve de jetons, et la requête suivante (13) reçoit une réponse `200`.

<div id="requests-per-minute-example">
  ### Exemple de requêtes par minute
</div>

Supposons qu’Auth0 lance une nouvelle API appelée `/ratelimitexample2` avec les valeurs de limite de débit suivantes :

* limite en rafale : cinq (5) requêtes
* Limite de débit soutenue : six (6) requêtes par minute.

Points clés :

* L’API commence avec cinq jetons de requête, ce qui correspond à la limite en rafale.
* Le réservoir de six jetons est rempli de nouveau chaque minute à l’aide d’une « fenêtre fixe ». De nouveaux jetons sont ajoutés au réservoir, qui est ainsi rempli de nouveau au « début » de chaque minute.

Exemple de scénario avec des limites de débit :

<Frame>![](/docs/images/cdy7uua7fh8z/PhtwGBwS9PfEpNQbPeA1o/dc2b05c72ef960a19958dbafa0295e70/Examples_of_how_rate_limits_are_enforced_-_Page_1__1_.png)</Frame>

Dans ce scénario :

* T0 - T+1min : L’utilisateur final effectue six requêtes pendant la première minute. Cinq requêtes – soit la limite en rafale – reçoivent une réponse `200`. La sixième requête reçoit une erreur `429` parce qu’il ne reste plus de jetons de requête.
* T+1min - T+2min : Auth0 remplit de nouveau le réservoir de jetons en raison de l’algorithme de fenêtre fixe. Par conséquent, les septième à onzième requêtes réussissent, ce qui épuise le réservoir à la 12e requête et entraîne une erreur `429`.
* T+2min : Auth0 remplit de nouveau le réservoir de jetons, et la requête suivante (13) reçoit une réponse `200`.

<div id="other-scenarios">
  ### Autres scénarios
</div>

Il arrive qu’Auth0 attribue deux limites de débit à une même API.  Cela permet de configurer une limite de rafale et une limite de débit soutenue mieux adaptées aux besoins du service.  En pratique, la première limite de débit devient la limite de rafale effective, et la seconde limite de débit devient la limite de débit soutenue effective.  Dans ce scénario, Auth0 publie uniquement les limites de rafale et de débit soutenues effectives, au lieu de communiquer les limites de rafale et de débit soutenues réelles.

<div id="end-user-login-and-signup-api-usage">
  ## Utilisation de l’API pour le Login et l’inscription des utilisateurs finaux
</div>

Il existe quelques flux d’authentification, notamment Login, inscription et Change Password. Les plus courants sont généralement Login, suivi de l’inscription.

Le login d’un utilisateur final déclenche plusieurs appels d’API vers les points de terminaison de l’Authentication API afin de déterminer si l’utilisateur final est autorisé à recevoir un token d’autorisation et, ainsi, à accéder à l’application demandée.

Le nombre exact d’appels d’API dépend de quelques configurations :

* Expérience d’authentification (p. ex., New <Tooltip tip="Universal Login : votre application redirige vers Universal Login, hébergé sur l’Authorization Server d’Auth0, pour vérifier l’identité d’un utilisateur." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Universal+Login">Universal Login</Tooltip> ou Classic Login)
* Flux d’authentification (p. ex., Login, inscription ou Change Password)
* Type de flux d’authentification (p. ex., Login avec nom d’utilisateur / mot de passe; Login avec Social Login; Login lorsqu’un token d’authentification existe déjà)

Ci-dessous, nous décrivons quelques configurations client courantes, ainsi que leur incidence sur l’utilisation de l’API.

<div id="universal-login">
  ### Universal Login
</div>

[Auth0 Universal Login](/docs/fr-CA/authenticate/login/auth0-universal-login) offre la fonctionnalité essentielle d’un <Tooltip tip="Authorization Server : serveur centralisé qui contribue à définir les limites de l’accès d’un utilisateur. Par exemple, votre serveur d’autorisation peut contrôler les données, les tâches et les fonctionnalités auxquelles un utilisateur a accès." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=authorization+server">serveur d’autorisation</Tooltip> : le flux de connexion. Lorsqu’un utilisateur doit prouver son identité pour accéder à votre application, vous pouvez le rediriger vers Universal Login et laisser Auth0 gérer le processus d’authentification.

| Flux d’authentification | Type de flux                                                | Requêtes vers les points de terminaison de l’Authentication API |
| ----------------------- | ----------------------------------------------------------- | --------------------------------------------------------------- |
| Connexion               | Vérification par nom d’utilisateur et mot de passe*         | 5                                                               |
| Connexion               | Fournisseur d’identité tiers – p. ex., Social ou Work Login | 6                                                               |
| Connexion               | Une session d’authentification Auth0 existe                 | 1                                                               |
| Inscription             | par nom d’utilisateur et mot de passe                       | 6                                                               |

<div id="modifiers">
  #### Modificateurs
</div>

Certaines configurations d’authentification modifient le nombre de requêtes de base. Ces ajustements dépendent de mesures de sécurité supplémentaires ou de flux d’authentification :

| Modificateur                 | Description                                                                                                          | Requêtes supplémentaires |
| ---------------------------- | -------------------------------------------------------------------------------------------------------------------- | ------------------------ |
| **Identification préalable** | Identifie l’utilisateur avant de demander ses identifiants.                                                          | +2                       |
| **MFA**                      | Ajoute l’authentification multifacteur.                                                                              | +2 par facteur           |
| **OTP**                      | Mot de passe à usage unique pour l’authentification                                                                  | +2                       |
| **Connexion d’entreprise**   | Authentification au moyen d’une connexion d’entreprise (p. ex., SAML, OIDC, LDAP).                                   | +1                       |
| **Identifiants du client**   | Utilisé pour l’authentification machine à machine. S’applique dans tous les cas, même si des actions sont utilisées. | +1                       |

*Tout élément utilisé en combinaison s’ajoute au nombre total de requêtes.

<div id="classic-login">
  ### Classic Login
</div>

[Classic Login](/docs/fr-CA/authenticate/login/auth0-universal-login/universal-login-vs-classic-login/classic-experience) est une expérience de connexion hébergée par Auth0 qui s’appuie sur JavaScript pour la personnalisation. La mise en œuvre de Classic Login est moins complexe que l’intégration directe du processus d’authentification dans votre application et peut aider à prévenir les risques liés à l’authentification inter-origines.

| Flux d’authentification | Type de flux                                                | Requêtes |
| ----------------------- | ----------------------------------------------------------- | -------- |
| Connexion               | Nom d’utilisateur/mot de passe                              | 8        |
| Connexion               | Fournisseur d’identité tiers – p. ex., Social ou Work Login | 8        |
| Connexion               | Une session d’authentification Auth0 existe                 | 2        |
| Inscription             | Nom d’utilisateur/mot de passe                              | 8        |

<Warning>
  Les clients qui configurent Custom Databases doivent ajouter deux (2) appels supplémentaires à l’Authentication API à chaque flux d’authentification décrit ci-dessus. Pour en savoir plus, consultez [Custom Database Connections](/docs/fr-CA/authenticate/database-connections/custom-db).
</Warning>

<div id="modifiers">
  #### Modificateurs
</div>

Les facteurs suivants augmentent le nombre de requêtes pour Classic Login :

| Modificateur                            | Description                                                                                   | Requêtes supplémentaires |
| --------------------------------------- | --------------------------------------------------------------------------------------------- | ------------------------ |
| **Authentification par SMS uniquement** | Lorsque le SMS est utilisé comme méthode d’authentification principale.                       | +7                       |
| **Native Social Login**                 | Connexion au moyen d’un fournisseur social natif (p. ex., Google, Facebook).                  | +1                       |
| **Redirections**                        | Des redirections supplémentaires pendant l’authentification augmentent le nombre de requêtes. | +1                       |
