title: Politique relative aux limites d’entités
---

Dans Auth0, les entités sont des éléments de configuration du tenant, comme Applications, Connections, Organizations, les <Tooltip tip="Serveur de ressources : serveur hébergeant des ressources protégées. Les serveurs de ressources acceptent les requêtes visant des ressources protégées et y répondent." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=resource+servers">serveurs de ressources</Tooltip> d’API, Permissions, Roles et Actions.

<Warning>
  Les limites d’entités sont des plafonds stricts appliqués aux entités afin d’assurer la santé de notre plateforme; elles ne font pas partie de la tarification, bien qu’elles puissent varier selon les groupes de forfaits. Les forfaits individuels comportent des limitations distinctes, qui correspondent aux droits comme les MAU et les Enterprise Connections auxquels vous avez droit lorsque vous vous abonnez à un forfait donné. (Pour connaître les limitations de votre forfait, consultez [Auth0 Pricing](https://auth0.com/pricing/).)
</Warning>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Toute utilisation du terme &quot;illimité&quot; signifie qu’il n’y a pas de limites strictes, mais qu’Okta applique tout de même des limites système à ce seuil. Veuillez communiquer avec le soutien si vous avez des questions au sujet de ces limites.
</Callout>

<div id="global-subscription-plan-limits">
  ## Limites globales des forfaits d’abonnement
</div>

Cette section présente les limites applicables à tous les forfaits d’abonnement.

<div id="organizations">
  ### Organizations
</div>

| Entity                                                       | Limit   |
| ------------------------------------------------------------ | ------- |
| Organizations par tenant                                     | 100,000 |
| Membres par Organization                                     | 100,000 |
| Connexions par Organization                                 | 10      |
| Affectations de rôles par membre d’Organization              | 50      |
| Affectations de rôles par groupe Enterprise                   | 10      |
| Rôles uniques pouvant être attribués à des groupes Enterprise | 100     |
| Groupes Enterprise auxquels des rôles peuvent être attribués  | 10,000  |
| Autorisations client M2M par Organization (Public Cloud)            | 100     |
| Autorisations client M2M par Organization (Private Cloud)           | 1000    |
| Profils de Custom Token Exchange                             | 100     |

Les clients ayant un forfait Enterprise peuvent demander une augmentation des limites d’entité pour le nombre d’Organizations par tenant et le nombre de membres par Organization en [communiquant avec le soutien](https://support.auth0.com/).

Ces limites peuvent être portées à 2,000,000 Organizations par tenant et à 2,000,000 membres par Organization sur le Public Cloud. Sur les instances de Private Cloud, ces entités sont illimitées.

<div id="authorization-core-role-based-access-control-rbac">
  ### Contrôle d’accès basé sur les rôles (RBAC) d’Authorization Core
</div>

| Entité                                                       | Limite |
| ------------------------------------------------------------ | ------ |
| Rôles par tenant                                             | 1000   |
| Scopes par API (serveurs de ressources)                      | 1000   |
| Rôles par utilisateur                                        | 50     |
| Rôles par groupe Enterprise                                  | 10     |
| Rôles uniques pouvant être attribués aux groupes Enterprise  | 100    |
| Groupes Enterprise auxquels des rôles peuvent être attribués | 10,000 |
| Permissions par utilisateur                                  | 1000   |
| Permissions par rôle                                         | 1000   |

Les limites ci-dessus pour les groupes Enterprise et les rôles s’appliquent à l’utilisation de RBAC sans la fonctionnalité Organizations. L’utilisation de la fonctionnalité Organizations permet d’appliquer ces limites par Organization plutôt que par tenant.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Notez que les limites du nombre de Permissions par utilisateur s’appliquent aux Permissions attribuées directement.

  Techniquement, un utilisateur peut avoir plus de Permissions que ce qui est indiqué si vous attribuez les Permissions à différents rôles, puis attribuez ces rôles à l’utilisateur.
</Callout>

| Entité                                                | Limite |
| ----------------------------------------------------- | ------ |
| Jeton d’actualisation par utilisateur par application | 200    |

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Si vous atteignez la limite de <Tooltip tip="Jeton d’actualisation : jeton utilisé pour obtenir un nouveau jeton d’accès sans obliger les utilisateurs à se connecter de nouveau." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Refresh+tokens">jetons d’actualisation</Tooltip> et créez un nouveau jeton d’actualisation,
  le système révoque ou supprime le jeton le plus ancien de cet utilisateur dans
  l’Application.

  Les jetons révoqués et les jetons expirés ne comptent pas dans cette limite.
</Callout>

<div id="actions">
  ### Actions
</div>

| Entité                                      | Limite |
| ------------------------------------------- | ------ |
| Actions par tenant                          | 100    |
| Actions par Trigger par tenant              | 20     |
| Actions Modules par tenant                  | 100    |
| Actions Modules par Action                  | 3      |
| Versions par Action                         | 50     |
| Versions par Action Module                  | 50     |
| Secrets par Action                          | 30     |
| Secrets par Action Module                   | 30     |
| Dépendances (modules NPM) par Action        | 10     |
| Dépendances (modules NPM) par Action Module | 10     |

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Les limites des Actions et des Actions Modules incluent à la fois les entités déployées et non déployées. Lorsque vous les supprimez, elles ne sont plus prises en compte dans la limite correspondante.

  Les limites des versions d’Action et des versions d’Action Module incluent à la fois les versions brouillon et les versions actives. Une fois la limite atteinte, chaque version additionnelle ajoutée entraîne la suppression par Auth0 de la plus ancienne version inutilisée.
</Callout>
