title: Politique de limite des entités
---

Dans Auth0, les entités sont des éléments de configuration du tenant, comme les Applications, les Connections, les Organizations, les <Tooltip tip="Serveur de ressources : serveur hébergeant des ressources protégées. Les serveurs de ressources acceptent les requêtes visant des ressources protégées et y répondent." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=resource+servers">serveurs de ressources</Tooltip> d’API, les autorisations, les Roles et les Actions.

<Warning>
  Les limites d’entités sont des limites strictes imposées pour préserver la santé de notre plateforme; elles ne font pas partie de la tarification, même si elles peuvent varier selon les groupes de plans tarifaires. Les plans tarifaires individuels comportent des limites distinctes, qui correspondent aux droits comme les MAU et les Enterprise Connections dont vous bénéficiez lorsque vous vous abonnez à un plan donné. (Pour connaître les limites de votre plan tarifaire, consultez [Auth0 Pricing](https://auth0.com/pricing/).)
</Warning>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Toute utilisation du terme « illimité » signifie qu’il n’y a pas de limites strictes, mais Okta applique des limites système à partir d’un certain seuil. Veuillez communiquer avec le soutien si vous avez des questions au sujet de ces limites.
</Callout>

<div id="global-subscription-plan-limits">
  ## Limites globales des plans d’abonnement
</div>

Cette section présente les limites qui s’appliquent à tous les plans d’abonnement.

<div id="organizations">
  ### Organizations
</div>

| Entité                                                      | Limite  |
| ----------------------------------------------------------- | ------- |
| Organizations par tenant                                    | 100,000 |
| Membres par Organization                                    | 100,000 |
| Connections par Organization                                | 10      |
| Attributions de rôles par membre d’Organization             | 50      |
| Attributions de rôles par Enterprise Group                  | 10      |
| Rôles uniques pouvant être attribués aux Enterprise Groups  | 100     |
| Enterprise Groups auxquels des rôles peuvent être attribués | 10,000  |
| client grants M2M par Organization (Public Cloud)           | 100     |
| client grants M2M par Organization (Private Cloud)          | 1000    |
| Profils de Custom Token Exchange                            | 100     |

Les clients ayant des forfaits Enterprise peuvent demander une augmentation des limites d’entités pour les Organizations par tenant et les membres d’Organization par Organization en [contactant le soutien](https://support.auth0.com/).

Ces limites peuvent être portées à 2,000,000 Organizations par tenant et à 2,000,000 membres d’Organization par Organization dans le Public Cloud. Dans les instances de Private Cloud, ces entités sont illimitées.

<div id="authorization-core-role-based-access-control-rbac">
  ### Contrôle d’accès basé sur les rôles (RBAC) d’Authorization Core
</div>

| Entité                                                      | Limite |
| ----------------------------------------------------------- | ------ |
| Rôles par tenant                                            | 1000   |
| Scopes par API (serveurs de ressources)                     | 1000   |
| Rôles par utilisateur                                       | 50     |
| Rôles par Enterprise Group                                  | 10     |
| Rôles uniques pouvant être attribués aux Enterprise Groups  | 100    |
| Enterprise Groups auxquels des rôles peuvent être attribués | 10,000 |
| Autorisations par utilisateur                               | 1000   |
| Autorisations par rôle                                      | 1000   |

Les limites relatives aux Enterprise Groups et aux rôles ci-dessus s’appliquent à l’utilisation de RBAC sans la fonctionnalité Organizations. L’utilisation de la fonctionnalité Organizations permet d’appliquer ces limites par Organization plutôt que par tenant.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Notez que les limites sur les autorisations par utilisateur touchent les autorisations attribuées directement.

  Techniquement, un utilisateur pourrait avoir plus d’autorisations que ce qui est indiqué si vous attribuez les autorisations à différents rôles, puis attribuez ces rôles à l’utilisateur.
</Callout>

| Entité                                        | Limite |
| --------------------------------------------- | ------ |
| Refresh Token par utilisateur par Application | 200    |

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Si vous atteignez la limite de <Tooltip tip="Refresh Token : jeton utilisé pour obtenir un nouvel Access Token sans obliger les utilisateurs à ouvrir une session de nouveau." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Refresh+tokens">Refresh Tokens</Tooltip> et créez un nouveau Refresh Token,
  le système révoque ou supprime le jeton le plus ancien de cet utilisateur dans
  l’Application.

  Les jetons révoqués et les jetons expirés ne comptent pas dans cette limite.
</Callout>

| Entité                                      | Limite |
| ------------------------------------------- | ------ |

| Actions Modules par Tenant                  | 100    |

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Les limites des Actions et des Actions Modules incluent à la fois les entités déployées et non déployées. Lorsque vous les supprimez, elles ne sont plus comptabilisées dans la limite applicable.

  Les limites des versions d’Action et d’Action Module incluent à la fois les versions en brouillon et actives. Une fois la limite atteinte, chaque nouvelle version ajoutée entraîne la suppression par Auth0 de la plus ancienne version inutilisée.
</Callout>
