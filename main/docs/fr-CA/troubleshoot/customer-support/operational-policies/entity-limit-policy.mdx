title: Politique sur les limites des entités
---

Dans Auth0, les entités sont des éléments de configuration du tenant, comme les Applications, les connexions, les Organizations, les API <Tooltip tip="Serveur de ressources : serveur qui héberge des ressources protégées. Les serveurs de ressources acceptent les requêtes visant des ressources protégées et y répondent." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=resource+servers">serveurs de ressources</Tooltip>, les Permissions, les Roles et les Actions.

<Warning>
  Les limites d’entités sont des plafonds stricts imposés pour assurer la santé de notre plateforme; elles ne font pas partie de la tarification, bien qu’elles puissent varier selon les groupes de plans tarifaires. Les plans tarifaires individuels comportent des limites distinctes, qui correspondent aux droits inclus, comme les MAU et les Enterprise Connections, lorsque vous vous abonnez à un plan donné. (Pour connaître les limites de votre plan tarifaire, consultez [Tarification Auth0](https://auth0.com/pricing/).)
</Warning>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Toute utilisation du terme « illimité » signifie qu’il n’existe pas de plafond strict, mais qu’Okta applique tout de même des limites système au seuil concerné. Veuillez communiquer avec le soutien si vous avez des questions sur ces limites.
</Callout>

<div id="global-subscription-plan-limits">
  ## Limites globales des forfaits d’abonnement
</div>

Cette section présente les limites applicables à tous les forfaits d’abonnement.

<div id="organizations">
  ### Organizations
</div>

| Entité                                                      | Limite  |
| ----------------------------------------------------------- | ------- |
| Organizations par tenant                                    | 100,000 |
| Membres par organization                                    | 100,000 |
| connexions par organization                                 | 10      |
| Attributions de rôles par membre de l’organization          | 50      |
| Attributions de rôles par Enterprise Group                  | 10      |
| Rôles uniques pouvant être attribués aux Enterprise Groups  | 100     |
| Enterprise Groups auxquels des rôles peuvent être attribués | 10,000  |
| autorisations client M2M par organization (Public Cloud)    | 100     |
| autorisations client M2M par organization (Private Cloud)   | 1000    |
| Profils de Custom Token Exchange                            | 100     |

Les clients ayant un forfait Enterprise peuvent demander une augmentation des limites des entités pour les Organizations par tenant et les membres d’organization par organization en [contactant le soutien](https://support.auth0.com/).

Ces limites peuvent être portées à 2,000,000 Organizations par tenant et à 2,000,000 membres d’organization par organization sur le Public Cloud. Sur les instances de Private Cloud, ces entités sont illimitées.

<div id="authorization-core-role-based-access-control-rbac">
  ### Authorization Core : contrôle d’accès basé sur les rôles (RBAC)
</div>

| Entité                                                      | Limite |
| ----------------------------------------------------------- | ------ |
| Rôles par tenant                                            | 1000   |
| Scopes par API (serveurs de ressources)                     | 1000   |
| Rôles par utilisateur                                       | 50     |
| Rôles par Enterprise Group                                  | 10     |
| Rôles uniques pouvant être attribués aux Enterprise Groups  | 100    |
| Enterprise Groups auxquels des rôles peuvent être attribués | 10,000 |
| Permissions par utilisateur                                 | 1000   |
| Permissions par rôle                                        | 1000   |

Les limites ci-dessus pour les Enterprise Groups et les rôles s’appliquent à l’utilisation de RBAC sans la fonctionnalité Organizations. L’utilisation de la fonctionnalité Organizations permet d’appliquer ces limites par Organization plutôt que par tenant.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Notez que les limites de Permissions par utilisateur s’appliquent uniquement aux Permissions attribuées directement.

  Techniquement, un utilisateur pourrait avoir plus de Permissions que ce qui est indiqué si vous attribuez les Permissions à différents rôles, puis attribuez ces rôles à l’utilisateur.
</Callout>

| Entité                                                    | Limite |
| --------------------------------------------------------- | ------ |
| Jetons d’actualisation par utilisateur et par application | 200    |

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Si vous atteignez la limite de <Tooltip tip="Jeton d’actualisation : jeton utilisé pour obtenir un nouveau jeton d’accès sans obliger les utilisateurs à se connecter de nouveau." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Refresh+tokens">jetons d’actualisation</Tooltip> et créez un nouveau jeton d’actualisation,
  le système révoque ou supprime le plus ancien jeton de cet utilisateur dans
  l’Application.

  Les jetons révoqués et les jetons expirés ne comptent pas dans cette limite.
</Callout>

<div id="actions">
  ### Actions
</div>

| Entité                                      | Limite |
| ------------------------------------------- | ------ |
| Actions par tenant                          | 100    |
| Actions par Trigger par tenant              | 20     |
| Actions Modules par tenant                  | 100    |
| Actions Modules par Action                  | 3      |
| Versions par Action                         | 50     |
| Versions par Action Module                  | 50     |
| Secrets par Action                          | 30     |
| Secrets par Action Module                   | 30     |
| Dépendances (modules NPM) par Action        | 10     |
| Dépendances (modules NPM) par Action Module | 10     |

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Les limites des Actions et des Actions Modules incluent à la fois les entités déployées et non déployées. Lorsque vous les supprimez, elles ne comptent plus dans la limite correspondante.

  Les limites des versions d’Action et des versions d’Action Module incluent à la fois les versions brouillon et les versions actives. Une fois la limite atteinte, chaque version supplémentaire ajoutée entraîne la suppression par Auth0 de la plus ancienne version inutilisée.
</Callout>
