title: Exporter les données
---

Toutes les données de votre tenant Auth0 demeurent en tout temps sous votre contrôle et sont accessibles à tout moment au moyen de la <Tooltip tip="Management API : une API qui permet aux clients d’effectuer des tâches administratives." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Management+API">Management API</Tooltip>.

<div id="user-passwords">
  ## Mots de passe des utilisateurs
</div>

Les seules informations qui ne sont pas accessibles par l’API (pour des raisons de sécurité) sont les hachages de mot de passe de vos utilisateurs de base de données hébergée par Auth0 et les clés privées. Vous pouvez tout de même demander ces informations en ouvrant un [ticket de soutien](https://support.auth0.com). Pour en savoir plus, consultez [Exporter les hachages de mot de passe et les secrets MFA](/docs/fr-CA/manage-users/user-migration/export-password-hashes-and-mfa-secrets). Cette opération n’est pas offerte avec notre niveau d’abonnement Free, et nous ne sommes pas en mesure d’accepter ni de garantir les demandes d’exportation à une date et une heure précises.

Si vous stockez les mots de passe des utilisateurs dans votre base de données, configurez une connexion à une base de données personnalisée qu’Auth0 interrogera chaque fois qu’un utilisateur se connecte. Dans ce cas, Auth0 ne stockera jamais de hachages de mot de passe, à moins que vous choisissiez de migrer progressivement les utilisateurs vers Auth0.

<div id="auth0-components">
  ## Composants Auth0
</div>

Auth0 utilise principalement <Tooltip tip="OpenID : norme ouverte d’authentification qui permet aux applications de vérifier l’identité des utilisateurs sans recueillir ni stocker les informations de connexion." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=OpenID">OpenID</Tooltip> Connect (OIDC) comme protocole d’authentification, vous devriez donc pouvoir mettre en place une intégration pour votre application à l’aide de bibliothèques standards. Il en va de même si vous intégrez Auth0 au moyen de [SAML](/docs/fr-CA/authenticate/single-sign-on/outbound-single-sign-on/configure-auth0-saml-identity-provider). Tous les SDKs, bibliothèques et exemples d’Auth0 [sont publiés sur GitHub en tant que logiciels libres](https://github.com/auth0/).

<div id="social-identity-providers">
  ## Fournisseurs d’identité de réseaux sociaux
</div>

Si vous choisissez de ne pas utiliser Auth0, mais souhaitez continuer à utiliser les mêmes <Tooltip tip="OAuth 2.0 : cadre d’autorisation qui définit les protocoles et les workflows d’autorisation." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=OAuth">OAuth</Tooltip> <Tooltip tip="OAuth 2.0 : cadre d’autorisation qui définit les protocoles et les workflows d’autorisation." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=client+IDs">ID client</Tooltip> et secrets pour vos <Tooltip tip="Fournisseur d’identité (IdP) : service qui stocke et gère les identités numériques." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=identity+providers">fournisseurs d’identité</Tooltip> de réseaux sociaux, vous conserverez l’accès aux renseignements sur les utilisateurs sans avoir à afficher de nouvelles boîtes de dialogue de consentement.

<div id="custom-code">
  ## Code personnalisé
</div>

Toutes les fonctionnalités de code personnalisé d’Auth0 (Rules, scripts de base de données personnalisés, connexions OAuth personnalisées, etc.) s’exécutent dans un service sandbox Node.js. Toutes les bibliothèques offertes dans le service sandbox sont aussi offertes sur npm pour être utilisées avec du code Node.js standard.
