---
description: Décrit les fonctionnalités d’Auth0 concernées par la migration de Node.js v12 vers Node.js v16 et fournit des recommandations de migration.
title: Migrer vers Node.js 16
---

Le 30 avr. 2022, le [support à long terme (LTS) de Node.js v12 a pris fin](https://github.com/nodejs/Release#release-schedule), ce qui signifie que l’équipe de développement de Node.js ne rétroporte plus les correctifs de sécurité critiques dans cette version. Cela pourrait exposer votre code d’extensibilité à des failles de sécurité. Par conséquent, Auth0 migre de Node 12 vers Node 16.

<div id="recommendations">
  ## Recommandations
</div>

Bien que la mise à jour vers Node 16 n’introduise aucun changement incompatible dans la bibliothèque standard de Node.js (Rules et Custom Database Action Scripts sont touchés ; consultez la section [Changements incompatibles — Rules et Custom Database Action Scripts uniquement](#breaking-changes-rules-only)), nous encourageons les clients qui utilisent Node 12 à demeurer à jour avec les versions Node en Active Long-Term Support (LTS) pour des raisons de sécurité et de conformité. Les clients qui utilisent encore Node 8 ne respectent plus les exigences de sécurité et doivent migrer vers Node 16 afin d’éliminer les risques de sécurité. Nous avons retiré le runtime Node 8 le 22 févr. 2022 pour les tenants Public Cloud, puis dans la version Private Cloud d’avril 2022. Après ces dates, les tenants toujours configurés pour Node 8 s’exposent à un risque d’interruption de service.

Actions est conçu pour prendre en charge simultanément plusieurs versions de Node et prend actuellement en charge Node 12 et Node 16. De plus, les Actions Integrations sans code suivront un contrôle de version « toujours à jour » conforme aux normes de l’industrie, les versions mineures étant mises à jour automatiquement ; le code sous-jacent restera toujours à jour et sur la version la plus récente de Node.js.

| Clients utilisant                       | Devraient                                                                                                                                                                                                                                                           | Tâche                                                                            |
| --------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------- |
| Actions Node 12                         | Mettre à jour toute version d’Action personnalisée existante vers une nouvelle version, ce qui la fera passer à Node 16.                                                                                                                                            | Mettre à jour toutes les versions d’Action personnalisées dès que possible.      |
| Node 12 hérité (Rules/Hooks/Extensions) | Commencer la migration vers Actions Node 16.                                                                                                                                                                                                                        | [Migrer vers Actions Node 16](/docs/fr-CA/customize/actions/migrate) dès que possible. |
| Node 8 hérité                           | Migrer immédiatement vers Actions partout où c’est possible. Pour les Rules et Hooks qui ne peuvent pas être migrés immédiatement vers Actions, vous devez au minimum mettre votre code à jour vers Node 12 et mettre à jour les paramètres du tenant vers Node 12. | [Migrer vers Actions Node 16](/docs/fr-CA/customize/actions/migrate) immédiatement.    |

<div id="tasks">
  ## Tâches
</div>

<div id="upgrade-actions">
  ### Mettre à niveau les Actions
</div>

Les Actions existantes qui utilisent Node 12 peuvent être mises à niveau vers Node 16, puis, au besoin, revenir par la suite à une version antérieure sous Node 12. Mettez à niveau les Actions vers Node 16 en [créant et en déployant une nouvelle version](/docs/fr-CA/customize/actions/manage-versions) configurée pour utiliser Node 16 comme runtime.

<div id="migrate-rules-and-hooks-to-actions">
  ### Migrer les Rules et Hooks vers Actions
</div>

Dans la mesure du possible, migrez les Rules et Hooks vers Actions. Pour déterminer quels Rules et Hooks peuvent être migrés, consultez [Actions Limitations](/docs/fr-CA/customize/actions/limitations). Pour savoir comment migrer vos Rules et Hooks vers Actions, consultez nos guides disponibles à [Migrate to Actions](/docs/fr-CA/customize/actions/migrate).

<div id="enable-node-16-runtime">
  ### Activer le runtime Node 16
</div>

Si vous utilisez des produits d’Extensibility hérités, comme Rules, Hooks ou Extensions, vous devez activer le runtime Node 16 au niveau du tenant. Nous vous recommandons de faire passer d’abord votre tenant de développement au runtime Node 16, de tester votre configuration, puis de faire passer votre tenant de production.

1. Activez Node 16 sur votre tenant de développement à l’aide du panneau **Extensibility** de la page [Advanced Tenant Settings](https://manage.auth0.com/#/tenant/advanced) du Dashboard. Sélectionnez **Node 16** dans la liste déroulante **Runtime**.
2. Cliquez sur **Save**.
3. Vérifiez que soit vous n’avez pas les éléments indiqués ci-dessous, soit vous avez terminé les étapes de migration pour chacun d’eux.
4. Testez votre configuration. Lorsque vous avez terminé les tests, répétez les étapes 1 et 2 ci-dessus avec votre tenant de production pour activer Node 16 en production.

<div id="breaking-changes-rules-and-custom-database-action-scripts-only">
  ### Changements incompatibles — Rules et Custom Database Action Scripts uniquement
</div>

Les Rules s’exécutent dans un sandbox JavaScript. Le sandbox prend en charge le langage ECMAScript 6 ainsi qu’un grand nombre de modules Node.js (version 16+). Si vous utilisez l’une de ces propriétés globales dans des Rules ou des Custom Database Action Scripts, [cherchez une solution de rechange dans notre liste des modules sandbox pris en charge](https://auth0-extensions.github.io/canirequire/) et importez-la explicitement avec `require`.

* `_`
* `async`
* `Auth0`
* `azure_storage`
* `bcrypt`
* `crypto`
* `couchbase`
* `cql`
* `ip`
* `jwt`
* `Knex`
* `mongo`
* `mysql`
* `mysql_pool`
* `ObjectID`
* `pbkdf2`
* `pg`
* `postgres`
* `Pubnub`
* `q`
* `querystring`
* `request`
* `sqlserver`
* `uuid`
* `xml2js`
* `xmldom`
* `xpath`
* `xtend`
