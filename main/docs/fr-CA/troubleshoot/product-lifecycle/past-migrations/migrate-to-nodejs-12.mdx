---
description: Décrit les fonctionnalités d’Auth0 concernées par la migration de Node.js v8 vers Node.js v12 et fournit des recommandations pour cette migration.
title: Migrer de Node.js 8 vers Node.js 12
---

Depuis le 31 décembre 2019, [Node.js v8 n’est plus pris en charge à long terme (LTS)](https://github.com/nodejs/Release#release-schedule), ce qui signifie que l’équipe de développement de Node.js ne rétroporte plus les correctifs de sécurité critiques vers cette version. Cela pourrait exposer votre code d’extensibilité à des failles de sécurité. Par conséquent, Auth0 a migré de Node.js 8 à Node.js 12.

<div id="features-affected">
  ## Fonctionnalités concernées
</div>

Les fonctionnalités Auth0 suivantes utilisent Node 8 :

* Rules
* Hooks
* connexions de base de données personnalisées
* connexions sociales personnalisées
* Extensions

Si vous n’utilisez aucune des fonctionnalités d’extensibilité mentionnées ci-dessus, vous n’êtes pas concerné par cette migration.

<div id="tasks">
  ## Tâches
</div>

Dans le cadre de l’introduction de Node 12 dans notre runtime Webtask, nous avons effectué plusieurs tests pour déterminer quels modules ne sont pas compatibles avec le passage de Node 8 à Node 12. La plupart des clients devraient pouvoir passer à Node 12 sans problème.

Cela dit, avant de migrer, nous vous recommandons fortement de tester tous vos :

* Rules
* Hooks
* connexions de base de données personnalisées et scripts
* connexions sociales personnalisées
* Extensions

Auth0 recommande de faire passer d’abord votre tenant de développement au runtime Node 12, d’effectuer tous les tests dans ce tenant de développement, puis de migrer votre tenant de production uniquement lorsque vous ne constatez aucun problème en développement.

<div id="enable-node-12-runtime">
  ### Activer le runtime Node 12
</div>

Cette migration peut entraîner d’autres changements de comportement. Nous avons donc ajouté un commutateur de migration qui vous permet de contrôler la migration de votre environnement vers le nouveau runtime Webtask avec Node 12.

Auth0 recommande de faire passer d’abord votre tenant de développement au runtime Node 12, d’effectuer tous les tests dans ce tenant, puis de migrer votre tenant de production seulement une fois que vous n’y constatez aucun problème.

Vous pouvez interroger la [Management API](/docs/fr-CA/api) pour vos rules, hooks, scripts de base de données personnalisée et connexions sociales personnalisées. Cela vous permettra plus facilement de déplacer des éléments de votre tenant de production vers votre tenant de développement à des fins de test.

Lorsque vous utilisez les endpoints de Connections dans la <Tooltip tip="Management API : un produit qui permet aux clients d’effectuer des tâches administratives." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Management+API">Management API</Tooltip>, les scripts de base de données personnalisée peuvent être récupérés ou mis à jour à l’aide de `options.customScripts`. De même, vous trouverez les connexions sociales personnalisées dans `options.scripts.fetchUserProfile`.

1. Activez Node 12 dans votre tenant de développement à l’aide du nouveau panneau **Extensibility** de la page [Advanced Tenant Settings](https://manage.auth0.com/#/tenant/advanced) du Dashboard. Choisissez **Node 12** dans la liste déroulante **Runtime**.
2. Cliquez sur **Save**.
3. Si vous utilisez les éléments ci-dessous, suivez les étapes de migration pour chacun d’eux.
4. Testez votre configuration.
5. Une fois que vous êtes certain que rien n’a été affecté, suivez les étapes 1 et 2 ci-dessus pour activer Node 12 dans votre tenant de production.

<div id="allowlist-new-urls">
  ### Ajouter les nouvelles URL à la liste d’autorisation
</div>

La Delegated Administration Extension et l’extension SSO Dashboard nécessitent l’ajout à la liste d’autorisation des URL utilisées pour accéder aux extensions et aux Webtasks personnalisés. Lorsque vous passez à Node 12, les URL utilisées pour accéder aux extensions et aux Webtasks personnalisés changent. Il s’agit d’un changement incompatible pour ces extensions.

Si vous utilisez l’une de ces extensions, vous devez ajouter les nouvelles URL à la liste d’autorisation à la fois comme Allowed Callback URLs et comme Allowed Logout URLs.

La partie région de l’URL passera de 8 à 12. Si vous accédez à une extension à l’aide de l’URL :

`https://{yourTenant}.us8.webtask.io/dummy-extension-url`

lorsque vous passerez à Node 12, l’URL sera :

`https://{yourTenant}.us12.webtask.io/dummy-extension-url`

1. Accédez à [Dashboard &gt; Applications &gt; Applications &gt; Settings](https://manage.auth0.com/#/applications/\{yourClientId}/settings), puis ajoutez l’URL aux champs **Allowed Callback URLs** et **Allowed Logout URLs**.
2. Les URL d’exécution des Webtasks personnalisés dans votre conteneur Auth0 changeront aussi. Vous devez mettre à jour toutes les applications externes qui appellent ces Webtasks.

<div id="republish-authorization-extension-rule">
  ### Republier la règle Authorization Extension
</div>

Si vous utilisez Authorization Extension, elle génère une règle `auth0-authorization-extension`. Republiez cette règle depuis Authorization Extension afin de mettre à jour automatiquement les URL.

1. Assurez-vous d’avoir mis à niveau Authorization Extension vers sa version la plus récente à partir de l’onglet **Extensions installées**. Si le bouton **Mettre à niveau** est affiché, cliquez dessus pour lancer la mise à niveau. S’il n’est pas affiché, vous utilisez déjà la version la plus récente de l’extension.
2. Ouvrez la page de configuration d’**Authorization Extension**.
3. Pour mettre à jour l’URL dans la règle, republiez-la en cliquant sur le bouton **Publier la règle**.
4. Effectuez un **Test** pour vous assurer que tout fonctionne encore. Si vous voyez une erreur Invalid API Key après la mise à jour, cliquez sur le bouton **Effectuer la rotation** pour générer une nouvelle clé API.

</div>

Par exemple, si vous êtes aux États-Unis et que vous utilisez Delegated Administration, vous devez mettre à jour les champs suivants dans les paramètres de votre application :

* **Allowed Logout URLs** : `https://{yourTenant}.us12.webtask.io/auth0-delegated-admin`

<div id="update-extensions">
  ### Mettre à jour les extensions
</div>

La plupart des extensions utilisent le secret caché `PUBLIC_WT_URL` pour l’autorisation. Ce secret dépend de la version du runtime et ne se met pas à jour automatiquement.

Pour le mettre à jour, vous devez enregistrer les paramètres de l’extension (aucune modification n’est nécessaire). Pour ce faire, après avoir fait passer le runtime à `Node 12`, vous devez ouvrir les paramètres de l’extension dans le dashboard des extensions (icône d’engrenage), puis cliquer sur `Save`. Ensuite, la galerie des extensions mettra à jour le secret `PUBLIC_WT_URL` en fonction du runtime sélectionné.

Si vous ne mettez pas à jour le secret caché `PUBLIC_WT_URL`, vous obtiendrez l’erreur suivante :

<div id="update-pinned-modules">
  ### Mettre à jour les modules verrouillés
</div>

Si vous utilisez les modules intégrés suivants (c’est-à-dire des modules que vous n’avez pas explicitement inclus), sachez que certaines versions ont été mises à jour pour fonctionner avec Node 12. Le tableau suivant résume les changements. Ces nouvelles versions devraient demeurer rétrocompatibles avec les versions précédentes.

| Nom du module | Ancienne version | Nouvelle version |
| ------------- | ---------------- | ---------------- |
| couchbase     | ~2.5.1           | 2.6.10           |
| bcrypt        | 1.0.3            | 3.0.8            |

Si vous avez verrouillé manuellement des modules, vous devrez peut-être aussi les mettre à jour manuellement pour que votre code s’exécute avec Node 12.

Par exemple, vous devez remplacer :

`var bcrypt = require(‘bcrypt@1.0.3’);`

par

`var bcrypt = require(‘bcrypt’);`

ou, si le module doit être verrouillé à une version précise :

`var bcrypt = require(‘bcrypt@3.0.8’);`
