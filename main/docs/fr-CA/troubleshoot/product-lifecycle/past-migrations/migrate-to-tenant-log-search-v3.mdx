description: Décrit comment migrer d’Auth0 Logs Search v2 vers Tenant Log Search v3.
title: Migrer vers Tenant Log Search v3
---

Afin d’offrir à nos clients la solution la plus fiable et la plus évolutive, Auth0 a déprécié Tenant Logs Search Engine v2 au profit de v3.

Auth0 migre de façon proactive les clients non touchés par ce changement, tandis que ceux qui pourraient l’être sont avisés afin d’activer v3 pendant la période de grâce prévue.

<div id="features-affected">
  ## Fonctionnalités concernées
</div>

Vous êtes concerné si vous remplissez tous les critères suivants :

* tenant créés le 21 mai 2019 ou avant
* tenant hébergés dans le Public Cloud d’Auth0 dans les régions AU ou EU
* Vous utilisez l’endpoint GET /api/v2/logs ou l’endpoint GET `/api/v2/users/{user_id}/logs` avec le paramètre `include_totals=true` ou le paramètre `q`.
* Vous parcourez plus de 1000 résultats par pagination.
* Vous utilisez la Delegated Admin Extension. Les anciennes versions de l’extension continueront de fonctionner après la migration de votre tenant vers Search Engine v3, mais il se peut que les totaux de pagination affichés lors de la consultation des logs soient inexacts. La mise à jour vers la version 3.7 de l’extension corrige ce problème.

Les tenant suivants ne sont pas concernés :

* tenant Cloud dans la région US. La région US a été entièrement migrée et utilise déjà Search Engine v3.
* tenant Private Cloud (la migration pour les clients Private Cloud commencera à une date ultérieure.)
* tenant Cloud dans les régions EU et AU qui :

  * n’utilisent pas du tout les endpoints `GET /api/v2/logs` ou `GET /api/v2/users/{user_id}/logs` de la Management API.
  * consultent les logs uniquement dans la section Logs du Dashboard.
  * utilisent le `GET /api/v2/logs endpoint` avec la méthode par checkpoint (au moyen du paramètre `from`).
  * consultent les logs à l’aide de l’une des extensions Dashboard Logs to External Service d’Auth0 (qui utilisent la méthode par checkpoint).

<div id="verify-query-migration">
  ## Migration de la requête Verify
</div>

Auth0 ne génère qu’un seul log du même type et avec la même description toutes les 60 minutes. Peu importe le nombre d’appels que vous effectuez vers les endpoints touchés à l’aide de fonctionnalités dépréciées, vous ne verrez quand même qu’un seul log par fonctionnalité dépréciée chaque heure.

Si vous apportez des modifications à vos requêtes, vous devrez attendre 60 minutes avant de pouvoir conclure avec certitude que l’absence de nouveaux logs `depnote` signifie que le comportement déprécié a été supprimé de votre code.

Vous pouvez rechercher dans les logs de votre tenant ce qui suit afin de repérer les requêtes qui généreraient des erreurs après la migration vers v3 :

`type:depnote AND description:*logs*`

Ces entrées de log comprennent un champ `description` qui précise le comportement déprécié que vous utilisez.

Vous pouvez aussi vérifier les champs `details.request.path` et `client_name` pour voir quelle application appelle `GET /api/v2/logs` ou `GET /api/v2/users/{user_id}/logs`.

<div id="changes">
  ## Changements
</div>

Les changements non rétrocompatibles sont mineurs, mais vous devriez passer vos requêtes en revue pour vous assurer que les résultats obtenus correspondent bien à vos attentes.

Les changements non rétrocompatibles concernent :

<div id="pagination">
  ### Pagination
</div>

* Lorsque votre tenant sera migré vers logs v3, la valeur du champ `total` renvoyée dans le résultat récapitulatif lors de l’appel à `GET /api/v2/logs` ou `GET /api/v2/users/{user_id}/logs` changera. Lorsque vous recherchez des logs à l’aide du moteur de recherche v2, le champ `totals` dans vos résultats indique le nombre de logs correspondant à la requête fournie. Toutefois, dans v3, le champ `totals` indique le nombre de logs renvoyés dans la page (un peu comme le champ `length`). Afin d’éviter toute perturbation potentielle, si votre application s’appuie sur le champ `total` pour la pagination, vous devriez mettre à jour votre logique afin de tenir compte correctement de ce changement.
* Il existe actuellement une limite de 100 logs par requête. Lorsque votre tenant sera migré vers logs v3, vous ne pourrez paginer que dans un maximum de 1 000 résultats de recherche; par conséquent, tout appel dépassant 1 000 résultats renverra une erreur. Afin d’éviter toute perturbation potentielle, vous devriez revoir vos requêtes pour éviter cette limite ou gérer les erreurs en conséquence.

<div id="q-parameter-validation">
  ### validation du paramètre q
</div>

* La syntaxe de requête utilisée avec le paramètre `q` dans `GET /api/v2/logs` comporte des changements mineurs dont il faut tenir compte. Lorsque votre tenant sera migré vers logs v3, cette validation sera appliquée, ce qui fera en sorte que cette requête renverra une erreur. Pour éviter toute interruption potentielle, vous devriez passer en revue vos requêtes pour vous assurer qu’elles respectent la syntaxe de requête prise en charge.
* Le paramètre `q` contient un champ invalide. Lorsque votre tenant sera migré vers logs v3, cette validation sera appliquée, ce qui fera en sorte que cet appel renverra une erreur. Pour éviter toute interruption potentielle, vous devriez passer en revue vos requêtes pour vous assurer que seuls les champs interrogeables sont inclus.

<div id="enable-tenant-log-search-v3">
  ## Activer Tenant Log Search v3
</div>

Après avoir examiné vos requêtes, vous pouvez activer Tenant Logs Search Engine v3 dans le Dashboard.

1. Accédez à [Tenant Settings &gt; Advanced](https://manage.auth0.com/#/tenant/advanced).
2. Faites défiler la page jusqu’à **Migrations**.
3. Désactivez le commutateur **Legacy Logs Search V2**. Le fait de désactiver ce commutateur désactive le moteur de recherche de logs v2 déprécié et force l’utilisation de Search Engine v3. Si vous ne voyez pas la bascule **Legacy Logs Search V2**, c’est que vous avez déjà été migré vers v3. Aucune autre action n’est requise.

Si vous avez besoin d’aide pour la migration, contactez-nous par l’intermédiaire du [Support Center](https://support.auth0.com/).
