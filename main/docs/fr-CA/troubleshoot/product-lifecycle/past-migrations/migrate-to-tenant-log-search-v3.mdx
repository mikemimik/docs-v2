---
description: Décrit comment passer d’Auth0 Logs Search v2 à v3.
title: Migrer vers Tenant Log Search v3
---

Afin d’offrir à nos clients la solution la plus fiable et la plus évolutive, Auth0 a rendu désuet Tenant Logs Search Engine v2 au profit de v3.

Auth0 migre proactivement les clients non touchés par ce changement, tandis que ceux qui pourraient l’être sont avisés qu’ils doivent opter pour v3 pendant la période de grâce prévue.

<div id="features-affected">
  ## Fonctionnalités concernées
</div>

Vous êtes concerné si vous répondez à tous les critères suivants :

* Tenants créés le 21 mai 2019 ou avant
* Tenants hébergés dans le Public Cloud d’Auth0 dans les régions AU ou EU
* Utilisent le endpoint GET /api/v2/logs ou le endpoint GET `/api/v2/users/{user_id}/logs` avec le paramètre `include_totals=true` ou le paramètre `q`.
* Parcourent plus de 1000 résultats avec pagination.
* Utilisent la Delegated Admin Extension. Les anciennes versions de l’extension continueront de fonctionner après la migration de votre tenant vers Logs Search Engine v3, mais il se peut que les totaux de pagination affichés dans les logs soient inexacts. La mise à jour vers la version 3.7 de l’extension règle ce problème.

Les tenants suivants ne sont pas concernés :

* Tenants Cloud dans la région US. La région US a été entièrement migrée et utilise déjà Search Engine v3.
* Tenants Private Cloud (la migration pour les clients de Private Cloud commencera à une date ultérieure.)
* Tenants Cloud dans les régions EU et AU qui :

  * N’utilisent pas du tout les endpoints `GET /api/v2/logs` ou `GET /api/v2/users/{user_id}/logs` de la Management API.
  * Consultent les logs uniquement dans la section Logs du Dashboard.
  * Utilisent le `GET /api/v2/logs endpoint` avec la méthode par checkpoint (au moyen du paramètre `from`).
  * Consultent les logs au moyen de l’une des extensions Dashboard Auth0 Logs to External Service (qui utilisent la méthode par checkpoint).

<div id="verify-query-migration">
  ## Vérifier la migration des requêtes
</div>

Auth0 génère un seul log du même type et avec la même description toutes les 60 minutes. Peu importe le nombre d’appels que vous effectuez vers les endpoints touchés à l’aide de fonctionnalités obsolètes, vous ne verrez quand même qu’un seul log par fonctionnalité obsolète chaque heure.

Si vous modifiez vos requêtes, vous devrez attendre 60 minutes avant de pouvoir conclure avec certitude que l’absence de nouveaux logs `depnote` signifie que le comportement obsolète a été retiré de votre code.

Vous pouvez effectuer la recherche suivante dans les logs de votre tenant pour repérer les requêtes qui généreraient des erreurs après la migration vers v3 :

`type:depnote AND description:*logs*`

Ces entrées de log comprennent un champ `description` qui précise le comportement obsolète que vous utilisez.

Vous pouvez aussi vérifier les champs `details.request.path` et `client_name` pour voir quelle application appelle soit `GET /api/v2/logs`, soit `GET /api/v2/users/{user_id}/logs`.

<div id="changes">
  ## Changements
</div>

Les changements non rétrocompatibles sont mineurs, mais vous devriez passer en revue vos requêtes pour vous assurer que les résultats obtenus correspondent à vos attentes.

Les changements non rétrocompatibles sont liés à :

<div id="pagination">
  ### Pagination
</div>

* Lorsque votre tenant est migré vers logs v3, la valeur du champ `total` renvoyée dans le résultat sommaire lors d’un appel à `GET /api/v2/logs` ou `GET /api/v2/users/{user_id}/logs` change. Lorsque vous recherchez des logs à l’aide du search engine v2, le champ `total` dans vos résultats indique le nombre de logs correspondant à la requête fournie. Cependant, dans v3, le champ `total` indique combien de logs sont renvoyés dans la page (un peu comme le champ `length`). Pour éviter toute perturbation, si votre application s’appuie sur le champ `total` pour la pagination, vous devriez mettre à jour votre logique afin de tenir compte correctement de ce changement.
* Il existe déjà une limite de 100 logs par requête. Lorsque votre tenant est migré vers logs v3, vous ne pouvez paginer que dans un maximum de 1 000 résultats de recherche; par conséquent, les appels portant sur plus de 1 000 résultats renverront une erreur. Pour éviter toute perturbation, vous devriez revoir vos requêtes afin d’éviter cette limite ou de gérer les erreurs en conséquence.

<div id="q-parameter-validation">
  ### validation du paramètre q
</div>

* La syntaxe de requête utilisée avec le paramètre `q` dans `GET /api/v2/logs` comporte des changements mineurs dont il faut tenir compte. Lorsque votre tenant sera migré vers logs v3, cette validation sera appliquée, ce qui fera en sorte que cette requête renverra une erreur. Pour éviter toute perturbation potentielle, vous devriez revoir vos requêtes afin de vous assurer qu’elles respectent la syntaxe de requête prise en charge.
* Le paramètre `q` contient un champ non valide. Lorsque votre tenant sera migré vers logs v3, cette validation sera appliquée, ce qui fera en sorte que cet appel renverra une erreur. Pour éviter toute perturbation potentielle, vous devriez revoir vos requêtes afin de vous assurer que seuls les champs interrogeables sont inclus.

<div id="enable-tenant-log-search-v3">
  ## Activer Tenant Log Search v3
</div>

Après avoir examiné vos requêtes, vous pouvez activer Tenant Logs Search Engine v3 dans le Dashboard.

1. Accédez à [Tenant Settings &gt; Advanced](https://manage.auth0.com/#/tenant/advanced).
2. Faites défiler jusqu’à **Migrations**.
3. Mettez le commutateur **Legacy Logs Search V2** en position désactivée. Cela désactive le moteur de recherche de logs v2 obsolète et force l’utilisation de search engine v3. Si vous ne voyez pas la bascule **Legacy Logs Search V2**, c’est que vous avez déjà été migré vers v3. Aucune autre action n’est requise.

Si vous avez besoin d’aide pour la migration, communiquez avec nous par l’intermédiaire du [Support Center](https://support.auth0.com/).
