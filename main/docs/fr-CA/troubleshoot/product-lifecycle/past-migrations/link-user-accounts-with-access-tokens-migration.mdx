Auparavant, vous pouviez utiliser les <Tooltip tip="Jeton ID : information d’authentification destinée au client lui-même, plutôt qu’à l’accès à une ressource." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=ID+tokens">jetons ID</Tooltip> pour lier et dissocier des comptes d’utilisateur dans certains cas. Auth0 déprécie cette fonctionnalité. Vous devrez désormais utiliser des <Tooltip tip="Jeton ID : information d’authentification destinée au client lui-même, plutôt qu’à l’accès à une ressource." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=access+tokens">jetons d’accès</Tooltip> dans tous les cas.

<Warning>
  Cette dépréciation fait suite à une faille de sécurité potentielle. Auth0 recommande fortement de mettre votre code à jour dès que possible.
</Warning>

<div id="features-affected">
  ## Fonctionnalités touchées
</div>

Les changements concernant la liaison de comptes sont les suivants :

* Vous ne pouvez plus utiliser un jeton d’identification dans l’en-tête `Authorization`; vous devez plutôt utiliser un jeton d’accès.
* Si vous utilisez un jeton d’accès dans l’en-tête `Authorization` avec `update:users` comme permission accordée, vous pouvez envoyer dans le corps de la requête soit le `user_id`, soit le jeton d’identification du compte secondaire.
* Si vous utilisez un jeton d’accès dans l’en-tête `Authorization` avec `update:current_user_metadata` comme permission accordée, vous pouvez seulement envoyer le jeton d’identification du compte secondaire dans le corps de la requête.
* Si vous envoyez le jeton d’identification du compte secondaire dans le corps de la requête (les cas d’utilisation décrits dans les deux puces précédentes), les conditions suivantes doivent être respectées :

  * Le jeton d’identification doit être signé à l’aide de `RS256` (vous pouvez définir cette valeur dans **Dashboard &gt; Clients &gt; Client Settings &gt; Advanced Settings &gt; OAuth**.
  * Le claim `aud` du jeton d’identification doit identifier le client et avoir la même valeur que le claim `azp` du jeton d’accès.
* Pour dissocier des comptes, vous ne pouvez plus utiliser un jeton d’identification dans l’en-tête `Authorization`. Vous devez plutôt utiliser un jeton d’accès.

Il existe plusieurs façons de lier et de dissocier des comptes. Dans la liste suivante, vous pouvez voir les cas d’utilisation et l’incidence de ces changements sur ceux-ci.

| Cas d’utilisation                                                                                                                                                                                                                                        | Statut                    |

<div id="actions">
  ## Actions
</div>

Passez en revue tous vos appels au [point de terminaison Identities](https://auth0.com/docs/api/management/v2/#!/Users/post_identities) pour la liaison de comptes et mettez à jour ceux qui utilisent le flux vulnérable décrit ci-dessus. Vous pouvez mettre à jour vos appels de l’une des façons suivantes :

* **Scénarios de liaison côté client / initiés par l’utilisateur :** Pour les scénarios de liaison côté client, effectuez l’appel au point de terminaison Identities à l’aide d’un jeton d’accès ayant la portée `update:current_user_identities`, et fournissez l’ID token du compte secondaire dans la charge utile (`link_with`). Cet ID token doit être obtenu au moyen d’un flux <Tooltip tip="OAuth 2.0 : cadre d’autorisation qui définit les protocoles et les flux de travail d’autorisation." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=OAuth">OAuth</Tooltip>/OIDC conforme.
* **Scénarios de liaison côté serveur** : Pour les scénarios de liaison côté serveur, effectuez l’appel au point de terminaison Identities à l’aide d’un jeton d’accès ayant la portée `update:users` et fournissez le `user_id` du compte secondaire dans la charge utile.

Consultez [Link User Accounts](/docs/fr-CA/manage-users/user-accounts/user-account-linking/link-user-accounts) pour en savoir plus.

<div id="link-user-accounts">
  ### Lier des comptes d’utilisateur
</div>

Pour lier des comptes d’utilisateur, vous pouvez soit appeler le [Link a User Account endpoint](https://auth0.com/docs/api/management/v2#!/Users/post_identities) de la <Tooltip tip="Management API : un produit permettant aux clients d’effectuer des tâches administratives." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Management+API">Management API</Tooltip>, soit utiliser la [bibliothèque auth0.js](/docs/fr-CA/libraries/auth0js).

<div id="link-current-user-accounts-with-the-management-api">
  #### Lier les comptes d’utilisateur actuels avec la Management API
</div>

Un cas d’utilisation courant consiste à permettre à l’utilisateur connecté de lier ses comptes à l’aide de votre application.

Avant la dépréciation, vous pouviez utiliser l’ID token ou le jeton d’accès de l’utilisateur principal (qui contenait le scope `update:current_user_identities`) pour vous authentifier auprès de la Management API et utiliser le [endpoint Link a User Account](https://auth0.com/docs/api/management/v2#!/Users/post_identities).

Vous devez maintenant obtenir un jeton d’accès (contenant le scope `update:current_user_identities`) et l’utiliser pour vous authentifier auprès de l’API, puis utiliser l’endpoint Link a User Account. La charge utile doit être l’ID token de l’utilisateur secondaire.

1. Obtenez un jeton d’accès avec le scope `update:current_user_identities`, comme illustré dans l’exemple suivant. L’exemple utilise le [flux implicite](/docs/fr-CA/get-started/authentication-and-authorization-flow/implicit-flow-with-form-post). Toutefois, vous pouvez [obtenir des jetons d’accès](/docs/fr-CA/secure/tokens/access-tokens/get-access-tokens) pour tout type d’application.

   1. Définissez `audience` sur `https://{yourDomain}/api/v2/`.
   2. Demandez le `scope` `${scope}`.

<div id="link-any-user-account-with-the-management-api">
  #### Lier n’importe quel compte d’utilisateur avec la Management API
</div>

Si vous obtenez un access token pour la liaison de comptes qui contient la portée `update:users` et que vous envoyez le `user_id` et le `provider` du compte secondaire dans la requête, vous n’avez rien à modifier.

Cependant, cette nouvelle méthode offre une autre possibilité. Vous utilisez toujours un access token qui contient la portée `update:users` pour vous authentifier auprès de l’API, mais dans la charge utile de la requête, vous pouvez envoyer l’ID token du compte secondaire (au lieu du `user_id` et du `provider`).

<div id="security-considerations">
  ## Considérations de sécurité
</div>

Nous avons identifié une faiblesse dans un flux particulier de liaison de comptes qui pourrait être exploitée dans certaines circonstances. Nous n&#39;avons trouvé aucune preuve d&#39;utilisation malveillante, mais nous avons décidé de déprécier ce flux pour éviter que cela ne se produise.

Par conséquent, Auth0 exige que les clients qui utilisent le flux de liaison de comptes touché migrent vers une mise en œuvre plus sécurisée avant le 19 octobre 2018. Des options de migration sont présentées dans ce guide et ne devraient entraîner aucune perte de fonctionnalité.

À compter du 19 octobre 2018, le flux de liaison de comptes touché sera désactivé et vous obtiendrez des erreurs d&#39;exécution.

Vous êtes concerné si vous appelez le [point de terminaison Post Identities](https://auth0.com/docs/api/management/v2#!/Users/post_identities) à l&#39;aide d&#39;un jeton (ID ou jeton d&#39;accès) avec la portée `update:current_user_identities` dans l&#39;en-tête Authorization et que vous incluez le `user_id` du compte secondaire dans la charge utile. Aucun autre cas d&#39;utilisation n&#39;est concerné.
