---
description: Décrit les éléments d’Auth0 Actions touchés par la migration de Node.js v12/v16 vers Node.js v18.
title: Migrer de Node 12 et 16 vers Node 18
---

Le soutien à long terme (LTS) de Node.js 12 et 16 a pris fin en 2023, ce qui signifie que l’équipe de développement de Node.js ne rétroporte plus les correctifs de sécurité critiques pour ces versions. L’utilisation des environnements d’exécution Node 12 ou 16 pourrait exposer votre code d’extensibilité à des vulnérabilités de sécurité.

L’environnement d’exécution d’extensibilité Node 18 est maintenant offert de façon générale (GA) dans l’ensemble de notre offre d’extensibilité. Cela comprend Actions, Rules, Hooks, Database Scripts et Custom Social Connections. Nous vous encourageons fortement à passer à Node 18 dès que possible afin de suivre les meilleures pratiques en matière de sécurité du code.

<div id="general-considerations">
  ## Considérations générales
</div>

<div id="migrate-rules-and-hooks-to-actions">
  ### Migrer les Rules et Hooks vers les Actions
</div>

Si vous utilisez un environnement d’exécution d’extensibilité abandonné, nous vous recommandons de profiter de l’examen de votre mise en œuvre des Rules et Hooks pour les migrer vers les Actions (Node 18). Déterminez quels Rules et Hooks vous pouvez migrer vers les Actions en consultant [Actions Limitations](/docs/fr-CA/customize/actions/limitations). Pour en savoir plus sur la migration de vos Rules et Hooks vers les Actions, consultez [Migrate to Actions](/docs/fr-CA/customize/actions/migrate).

<div id="marketplace-integrations">
  ### Intégrations Marketplace
</div>

<div id="social-connections-integrations">
  #### Intégrations de Social Connections
</div>

Utilisez la [Management API](https://auth0.com/docs/api/management/v2) pour obtenir la liste complète des connexions sociales pouvant être touchées par un changement de version de l’environnement d’exécution Node. Plus précisément, toutes les connexions sociales potentiellement touchées, qu’elles aient été créées explicitement en tant que connexion sociale personnalisée ou ajoutées initialement à partir du Marketplace, ont l’attribut `strategy` dont la valeur est `oauth1` ou `oauth2`.

Vous pouvez ensuite parcourir, par pagination, toutes les connexions sociales personnalisées existantes dans un tenant donné à l’aide du endpoint `GET` connections de la <Tooltip tip="Management API : un produit qui permet aux clients d’effectuer des tâches administratives." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Management+API">Management API</Tooltip>. Par exemple, les options de requête suivantes renvoient les noms et les identificateurs d’un maximum de 100 connexions sociales personnalisées :

```http wrap lines
/api/v2/connections?strategy=oauth1&strategy=oauth2&include_totals=true&fields=name&per_page=100
```

Le <Tooltip tip="Auth0 Dashboard : produit principal d’Auth0 pour configurer vos services." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip> ne permet pas de mettre à jour les scripts des connexions sociales personnalisées ajoutées via le Marketplace. Si une modification du script est nécessaire pour assurer la compatibilité avec Node 18, vous devez utiliser la Management API.

<div id="migration-tasks">
  ## Tâches de migration
</div>

<div id="create-new-custom-actions">
  ### Créer de nouvelles Actions personnalisées
</div>

Pour créer une nouvelle Action personnalisée avec Node 18 à partir de l’Auth0 Dashboard :

1. Accédez à [Auth0 Dashboard &gt; Actions &gt; Library](https://manage.auth0.com/#/dashboard/actions/library).
2. Sélectionnez **Create Action &gt; Build from scratch**.
3. Dans le champ **Runtime***, sélectionnez **Node 18 (Recommended)**.
4. Créez vos Actions personnalisées en Node 18, testez-les et déployez-les une fois qu’elles sont prêtes.

<div id="upgrade-existing-custom-actions">
  ### Mettre à niveau les Actions personnalisées existantes
</div>

Vous pouvez mettre à niveau individuellement les Actions personnalisées existantes basées sur Node 12 ou 16 vers Node 18 et [revenir à la version précédente](/docs/fr-CA/customize/actions/manage-versions#revert-actions) au moyen de l’ancien environnement d’exécution. Pour mettre à niveau les Actions vers Node 18, créez et déployez une nouvelle version de la mise en œuvre existante en y apportant les changements nécessaires, puis configurez-la pour utiliser Node 18 comme environnement d’exécution.

<div id="choose-node-18-for-other-extensibility-products">
  ### Choisissez Node 18 pour les autres produits d’extensibilité
</div>

L’environnement d’exécution utilisé pour les autres offres d’extensibilité (hors Actions) est défini globalement dans les paramètres avancés du tenant. La modification de ce paramètre a une incidence sur les fonctionnalités suivantes en même temps :

* rules
* hooks
* scripts de base de données personnalisés
* scripts de connexion sociale personnalisée

Pour modifier le paramètre de l’environnement d’exécution de l’extensibilité du tenant dans l’Auth0 Dashboard :

1. Accédez à **Dashboard &gt; Settings &gt; Advanced**.
2. Faites défiler jusqu’à la section **Extensibility**.
3. Pour **Runtime**, sélectionnez **Node 18**.

Comme il s’agit d’un paramètre global qui touche plusieurs fonctionnalités d’extensibilité à la fois, nous vous recommandons d’effectuer d’abord cette étape dans votre tenant de développement, de terminer les tests de toutes les fonctionnalités d’extensibilité applicables, puis de passer à votre tenant de production seulement lorsque vous ne constatez aucun problème en développement.

Plus précisément, pour les scripts Custom DB, vous pouvez [suivre les étapes expliquées sur cette page](/docs/fr-CA/authenticate/database-connections/custom-db/custom-database-connections-scripts/environment) afin de vérifier individuellement un script avec une version précise de l’environnement d’exécution avant de modifier la version globale de l’environnement d’exécution.

<div id="known-breaking-changes">
  ## Changements non rétrocompatibles connus
</div>

<div id="magic-npm-modules">
  ### Modules npm magiques
</div>

L’environnement d’exécution d’extensibilité Node 12 permet d’utiliser certains modules `npm` sans avoir à les importer explicitement avec `require` dans le code d’extensibilité. À partir de l’environnement d’exécution Node 16, nous avons supprimé la prise en charge de ce type d’utilisation pour les modules suivants :

* &#95;
* async
* Auth0
* azure&#95;storage
* bcrypt
* crypto
* couchbase
* cql
* ip
* <Tooltip tip="JSON Web Token (JWT) : format standard de ID Token (et souvent de jeton d’accès) utilisé pour représenter des claims de façon sécurisée entre deux parties." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=jwt">jwt</Tooltip>
* Knex
* mongo
* mysql
* mysql&#95;pool
* ObjectID
* pbkdf2
* pg
* postgres
* Pubnub
* q
* querystring
* sqlserver
* uuid
* xml2js
* xmldom
* xpath
* xtend

Si vous exécutez encore du code d’extensibilité sur Node 12, gardez ce qui précède à l’esprit lorsque vous mettez le code à niveau directement vers Node 18. Avant d’utiliser un module, vous devez vous assurer qu’il est importé explicitement.

Dans le contexte des Rules, des Custom Database Connections et des connexions sociales personnalisées, vous devez importer explicitement avec `require` une version du module qui est [indiquée comme disponible pour Node 18](https://auth0-extensions.github.io/canirequire/).

Dans Hooks et Actions, vous devez ajouter la version cible voulue comme dépendance explicite avant d’importer le module avec `require`.

<div id="secure-renegotiation-required-by-default-for-tls-connections">
  ### Renégociation sécurisée requise par défaut pour les connexions TLS
</div>

Node.js 18 exige par défaut la renégociation sécurisée (RFC 5746) pour les connexions TLS, puisque cette exigence a été introduite dans la dépendance OpenSSL sous-jacente.

Si votre code d’extensibilité effectue des appels réseau externes, les serveurs ciblés doivent prendre en charge la renégociation sécurisée; sinon, les requêtes échoueront et vous recevrez une erreur semblable à :

```text wrap lines
Error: write EPROTO C0BAF076:error:0A000152:SSL routines:final_renegotiate:unsafe legacy renegotiation disabled:../deps/openssl/openssl/ssl/statem/extensions.c:922:
```

Compte tenu de ce changement lié à la sécurité, nous vous recommandons de vous assurer que tous les serveurs cibles sont à jour et prennent en charge la renégociation sécurisée. Si les serveurs en question sont des serveurs tiers qui ne sont pas sous votre contrôle, vous pouvez envisager d’activer l’ancien comportement.

Par exemple, pour la bibliothèque `axios`, l’extrait de code suivant montre comment activer le comportement hérité :

```javascript lines
const axios = require('axios');
const https = require('https');
const crypto = require('crypto');

axios.get(
  'https://[LEGACY_SERVER]', 
  {
    httpsAgent: new https.Agent(
      {
        secureOptions: crypto.constants.SSL_OP_LEGACY_SERVER_CONNECT
      }
    )
  })
```
