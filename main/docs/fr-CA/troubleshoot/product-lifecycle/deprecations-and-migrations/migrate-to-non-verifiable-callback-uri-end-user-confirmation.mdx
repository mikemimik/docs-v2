title: Migrate to Non-Verifiable Callback URI End-User Confirmation
validatedOn: 2026-02-10
---

Auth0 recommande à toutes les applications natives qui utilisent le [flux du code d’autorisation](/docs/fr-CA/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce) de passer aux URI de rappel sur HTTPS à l’aide de [Android App Links](https://developer.android.com/training/app-links) et des [liens universels Apple](https://developer.apple.com/documentation/xcode/allowing-apps-and-websites-to-link-to-your-content). Cela renforce la sécurité et réduit le risque d’usurpation d’application et d’attaques d’hameçonnage.

Pour comprendre comment cette approche aide à prévenir les attaques, consultez les [Measures Against Application Impersonation](/docs/fr-CA/secure/security-guidance/measures-against-app-impersonation#prompt-customization).

Les tenants créés avant le 15 octobre 2025 conserveront le comportement précédent par défaut jusqu’au 28 avril 2026. Après la date limite d’octobre, les tenants nouvellement créés pourront, par défaut, afficher la nouvelle invite de confirmation de connexion, avec certaines exceptions selon le calendrier de déploiement de chaque environnement.

Les tenants qui choisissent explicitement de s’en retirer contourneront cette demande indéfiniment. Cela demeure vrai même après le 28 avril 2026, lorsque le service adoptera la invite de confirmation de connexion comme comportement par défaut et supprimera la bascule de migration « Unconfirmed Login with Non-Verifiable Callback URI Redirects ».

<div id="how-are-you-affected">
  ## Comment êtes-vous concerné?
</div>

Pour les applications clientes qui précisent déjà, ou prévoient préciser, un schéma d’URI personnalisé ou une URI de rappel loopback, les utilisateurs finaux pourraient devoir confirmer explicitement la connexion en interagissant avec la nouvelle invite de confirmation de connexion. Vos utilisateurs finaux pourraient percevoir ce changement comme une dégradation de l’expérience utilisateur.

De plus, les requêtes d’authentification comprenant `prompt=none` seront rejetées lorsque des applications utilisent des URI de rappel non vérifiables et sont configurées pour utiliser la nouvelle invite de confirmation de connexion.

<div id="migration-tasks">
  ## Tâches de migration
</div>

**Auth0 recommande fortement de passer aux URI de rappel sur HTTPS** en utilisant Android App Links et Apple Universal Links chaque fois que possible pour toutes les applications natives qui utilisent le flux de code d’autorisation.

De plus, dans les tenants où le comportement par défaut changera après le 28 avril 2026, vous devriez sélectionner explicitement le comportement requis pour les requêtes d’authentification utilisant des schémas d’URI personnalisés ou des URI de rappel de type URI de rappel loopback avant le changement du comportement par défaut du système.

<div id="review-whether-your-applications-are-using-non-verifiable-callback-uris">
  ### Vérifiez si vos applications utilisent des URI de rappel non vérifiables
</div>

Dans les tenants pour lesquels la bascule de migration Unconfirmed Login with Non-Verifiable Callback URI Redirects est disponible et activée, les requêtes d’authentification qui précisent un schéma d’URI personnalisé ou une URI de rappel loopback **généreront un log de tenant contenant un avis de dépréciation** à moins que vous n’ayez explicitement défini l’option suivante au niveau de l’application ou du tenant :

`skip_non_verifiable_callback_uri_confirmation_prompt`

Ces logs de tenant contiennent l’identifiant client de l’application qui effectue la requête. Vous pouvez surveiller ces logs de tenant dans l’Auth0 Dashboard à l’aide de la requête suivante :

```bash
type:depnotetype:depnote AND description:Unconfirmed\ Login\ with\ Non-Verifiable\ Callback\ URI\ Redirects*
```

<div id="opt-in-to-new-login-confirmation-prompt">
  ### Activer la nouvelle invite de confirmation de connexion
</div>

Pour activer à l’avance la nouvelle invite de confirmation de connexion et renforcer la sécurité des flux d’authentification qui utilisent des schémas d’URI personnalisés ou des URI de rappel loopback, suivez les étapes ci-dessous dans votre Auth0 Dashboard :

1. Accédez à [**Auth0 Dashboard &gt; Tenant Settings &gt; Advanced**](https://manage.auth0.com/#/tenant/advanced).
2. Dans la section **Migrations**, désactivez la bascule **Unconfirmed Login with Non-Verifiable Callback URI Redirects**.

<Frame>![Auth0 Dashboard > Tenant Settings > Advanced > bascule désactivée](/docs/images/cdy7uua7fh8z/unconfirmed-login-uri.png)</Frame>

<div id="opt-out-of-new-login-confirmation-prompt">
  ### Désactiver la nouvelle invite de confirmation de connexion
</div>

Si, après avoir évalué les considérations de sécurité, vous décidez de ne pas utiliser la nouvelle invite de confirmation de connexion, vous pouvez configurer des applications précises ou l’ensemble du tenant pour ne pas adopter ce nouveau comportement. Vous pouvez le faire à partir de votre Auth0 Dashboard.

Le paramètre défini au niveau de l’application a préséance sur celui du tenant. Assurez-vous de configurer les paramètres propres à l’application avant de modifier le paramètre au niveau du tenant afin d’éviter des changements de comportement involontaires. Par exemple, vous pourriez vouloir ignorer **Non-Verifiable Callback URI End-User Confirmation** pour certaines applications précises, tout en l’affichant par défaut pour d’autres applications, ou inversement.

Pour désactiver ce comportement pour des applications précises :

1. Accédez à [Auth0 Dashboard &gt; Applications &gt; Settings &gt; Advanced Settings &gt; OAuth](https://manage.auth0.com/applications/settings).
2. Repérez la bascule **Non-Verifiable Callback URI End-User Confirmation**, désactivez-la, puis sélectionnez **Save**. Il se peut que vous deviez sélectionner l’option **Override the tenant setting** pour pouvoir gérer cette configuration de façon permanente.

<Frame>![Auth0 Dashboard > Applications > Settings > Advanced](/docs/images/cdy7uua7fh8z/custom-uri-override.png)</Frame>

Pour désactiver ce comportement pour l’ensemble du tenant :

1. Accédez à [Auth0 Dashboard &gt; Tenant Settings &gt; Advanced](https://manage.auth0.com/tenant/advanced).
2. Repérez la bascule **Non-Verifiable Callback URI End-User Confirmation** dans la section **Login and Logout**, désactivez-la, puis sélectionnez **Save**. Il se peut que vous deviez sélectionner **Turn on** pour pouvoir gérer cette configuration de façon permanente.

<Frame>![Auth0 Dashboard > Tenant Settings > Advanced](/docs/images/cdy7uua7fh8z/migrate-uri-setting.png)</Frame>

Vous pouvez aussi configurer le comportement requis du tenant au moyen de l’Auth0 Management API. Plus précisément, vous pouvez effectuer la configuration à deux niveaux :

* **Configuration au niveau du tenant** : vous pouvez gérer le comportement de la invite de confirmation en définissant la propriété `skip_non_verifiable_callback_uri_confirmation_prompt` au moyen du endpoint [Update Tenant Settings](https://auth0.com/docs/api/management/v2#!/Tenants/patch_settings).​
* **Configuration au niveau de l’application** : pour remplacer le paramètre au niveau du tenant pour des applications précises, définissez la même propriété `skip_non_verifiable_callback_uri_confirmation_prompt` au moyen du endpoint [Update Client](https://auth0.com/docs/api/management/v2#!/Clients/patch_clients_by_id).

Pour obtenir plus d’information et des conseils sur la configuration de vos applications, consultez [Measures Against Application Impersonation](/docs/fr-CA/secure/security-guidance/measures-against-app-impersonation).
