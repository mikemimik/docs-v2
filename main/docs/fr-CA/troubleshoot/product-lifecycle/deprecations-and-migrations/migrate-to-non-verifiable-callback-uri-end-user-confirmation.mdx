Auth0 recommande à toutes les applications natives qui utilisent le [flux de code d’autorisation](/docs/fr-CA/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce) de passer à des URI de rappel sur HTTPS à l’aide de [Android App Links](https://developer.android.com/training/app-links) et des [Apple Universal Links](https://developer.apple.com/documentation/xcode/allowing-apps-and-websites-to-link-to-your-content). Cela renforce la sécurité et réduit les risques d’usurpation d’application et d’hameçonnage.

Pour comprendre comment cela permet de prévenir ces attaques, consultez [Measures Against Application Impersonation](/docs/fr-CA/secure/security-guidance/measures-against-app-impersonation#prompt-customization).

Les tenants créés avant le 15 octobre 2025 conservent l’ancien comportement par défaut jusqu’au 28 avril 2026. Après la date limite d’octobre, les nouveaux tenants peuvent, par défaut, afficher la nouvelle invite de confirmation de connexion, avec certaines exceptions selon le calendrier de déploiement de chaque environnement.

Les tenants qui s’y soustraient explicitement contourneront cette invite indéfiniment. Cela demeure vrai même après le 28 avril 2026, lorsque le service adoptera l’invite de confirmation comme comportement par défaut et supprimera la bascule de migration « Unconfirmed Login with Non-Verifiable Callback URI Redirects ».

<div id="how-are-you-affected">
  ## Quelles sont les répercussions pour vous?
</div>

Pour les applications clientes qui spécifient déjà, ou prévoient de spécifier, un schéma d’URI personnalisé ou une URI de rappel de bouclage, les utilisateurs finaux pourraient devoir confirmer explicitement la connexion en interagissant avec la nouvelle invite de confirmation de connexion. Vos utilisateurs finaux pourraient percevoir ce changement comme une dégradation de l’expérience utilisateur.

De plus, les requêtes d’authentification comprenant `prompt=none` seront rejetées lorsque des applications utilisent des URI de rappel non vérifiables et sont configurées pour utiliser la nouvelle invite de confirmation de connexion.

<div id="migration-tasks">
  ## Tâches de migration
</div>

**Auth0 recommande fortement de passer à des URI de rappel sur HTTPS** en utilisant Android App Links et les liens universels d’Apple chaque fois que possible pour toutes les applications natives qui utilisent le flux de code d’autorisation.

De plus, dans les tenants où le comportement par défaut change après le 28 avril 2026, vous devriez sélectionner explicitement le comportement requis pour les requêtes d’authentification qui utilisent des schémas d’URI personnalisés ou des URI de rappel de bouclage, avant que le comportement par défaut du système ne change.

<div id="review-whether-your-applications-are-using-non-verifiable-callback-uris">
  ### Vérifiez si vos applications utilisent des URI de rappel non vérifiables
</div>

Dans les tenants pour lesquels la bascule de migration Unconfirmed Login with Non-Verifiable Callback URI Redirects est disponible et activée, les requêtes d’authentification qui précisent un schéma d’URI personnalisé ou un URI de bouclage **généreront un journal de tenant contenant un avis de dépréciation**, sauf si vous avez explicitement défini l’option suivante au niveau de l’application ou du tenant :

`skip_non_verifiable_callback_uri_confirmation_prompt`

Ces journaux de tenant contiennent l’identifiant client de l’application qui effectue la requête. Vous pouvez surveiller ces journaux de tenant dans l’Auth0 Dashboard à l’aide de la requête suivante :

```bash
type:depnotetype:depnote AND description:Unconfirmed\ Login\ with\ Non-Verifiable\ Callback\ URI\ Redirects*
```

<div id="opt-in-to-new-login-confirmation-prompt">
  ### Activer la nouvelle invite de confirmation de connexion
</div>

Pour activer à l’avance la nouvelle invite de confirmation de connexion et renforcer la sécurité des flux d’authentification utilisant des schémas d’URI personnalisés ou des URI de bouclage, suivez les étapes ci-dessous dans votre Auth0 Dashboard :

1. Accédez à [**Auth0 Dashboard &gt; Tenant Settings &gt; Advanced**](https://manage.auth0.com/#/tenant/advanced).
2. Dans la section **Migrations**, désactivez la bascule **Unconfirmed Login with Non-Verifiable Callback URI Redirects**.

<Frame>![Auth0 Dashboard > Tenant Settings > Advanced > bascule désactivée](/docs/images/cdy7uua7fh8z/unconfirmed-login-uri.png)</Frame>

<div id="opt-out-of-new-login-confirmation-prompt">
  ### Désactiver la nouvelle invite de confirmation de connexion
</div>

Si, après avoir évalué les considérations de sécurité, vous décidez de ne pas utiliser la nouvelle invite de confirmation de connexion, vous pouvez configurer certaines applications ou l’ensemble du tenant pour exclure ce nouveau comportement. Vous pouvez le faire dans votre Auth0 Dashboard.

Le paramètre au niveau de l’application prévaut sur le paramètre au niveau du tenant. Assurez-vous de configurer les paramètres propres aux applications avant de modifier le paramètre au niveau du tenant afin d’éviter des changements de comportement involontaires. Par exemple, vous pourriez vouloir ne pas afficher le paramètre **Non-Verifiable Callback URI End-User Confirmation** pour certaines applications, tout en l’affichant par défaut pour d’autres applications, ou inversement.

Pour désactiver cette option pour certaines applications :

1. Accédez à [Auth0 Dashboard &gt; Applications &gt; Settings &gt; Advanced Settings &gt; OAuth](https://manage.auth0.com/applications/settings).
2. Repérez et désactivez la bascule **Non-Verifiable Callback URI End-User Confirmation**, puis sélectionnez **Save**. Vous devrez peut-être sélectionner l’option **Override the tenant setting** pour pouvoir gérer cette configuration de façon permanente.

<Frame>![Auth0 Dashboard > Applications > Settings > Advanced](/docs/images/cdy7uua7fh8z/custom-uri-override.png)</Frame>

Pour désactiver cette option pour l’ensemble du tenant :

1. Accédez à [Auth0 Dashboard &gt; Tenant Settings &gt; Advanced](https://manage.auth0.com/tenant/advanced).
2. Repérez et désactivez la bascule **Non-Verifiable Callback URI End-User Confirmation** dans la section **Login and Logout**, puis sélectionnez **Save**. Vous devrez peut-être sélectionner **Turn on** pour pouvoir gérer cette configuration de façon permanente.

<Frame>![Auth0 Dashboard > Tenant Settings > Advanced](/docs/images/cdy7uua7fh8z/migrate-uri-setting.png)</Frame>

Vous pouvez aussi configurer le comportement requis du tenant au moyen de l’Auth0 Management API. Plus précisément, vous pouvez effectuer la configuration à deux niveaux :

* **Configuration au niveau du tenant** : Vous pouvez gérer le comportement de l’invite de confirmation en définissant la propriété `skip_non_verifiable_callback_uri_confirmation_prompt` au moyen du point de terminaison [Update Tenant Settings](https://auth0.com/docs/api/management/v2#!/Tenants/patch_settings).​
* **Configuration au niveau de l’application** : Pour remplacer le paramètre au niveau du tenant pour certaines applications, définissez la même propriété `skip_non_verifiable_callback_uri_confirmation_prompt` au moyen du point de terminaison [Update Client](https://auth0.com/docs/api/management/v2#!/Clients/patch_clients_by_id).

Pour en savoir plus et obtenir des conseils sur la configuration de vos applications, consultez [Measures Against Application Impersonation](/docs/fr-CA/secure/security-guidance/measures-against-app-impersonation).
