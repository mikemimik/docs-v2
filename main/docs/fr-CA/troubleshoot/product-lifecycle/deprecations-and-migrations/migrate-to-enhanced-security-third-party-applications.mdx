À compter du 23 octobre 2026, Auth0 modifiera le mode de sécurité par défaut des applications tierces nouvellement créées par l’intermédiaire de la Management API. Par défaut, les applications tierces utiliseront des contrôles de sécurité renforcés conformes aux [pratiques exemplaires d’OAuth 2.1](https://datatracker.ietf.org/doc/html/draft-ietf-oauth-v2-1).

<Warning>
  Ce changement touche uniquement les tenants qui utilisaient des applications tierces avant le 23 avril 2026, et il n’a d’incidence que sur les applications nouvellement créées. Vos applications tierces existantes continueront de fonctionner comme aujourd’hui, sans qu’aucune modification soit requise.
</Warning>

Auth0 recommande fortement d’adopter des contrôles de sécurité renforcés pour toutes les nouvelles applications tierces. Ces contrôles offrent une autorisation explicite de l’API, l’utilisation obligatoire de PKCE et un ensemble de fonctionnalités ciblé, conforme à OAuth 2.1 et aux pratiques exemplaires en matière de sécurité. Les applications dotées de contrôles renforcés bénéficieront aussi de fonctionnalités à venir, comme des limites de débit au niveau de l’application et des outils de gestion améliorés. Toutefois, vous pouvez conserver le comportement existant au besoin pour certains cas d’utilisation précis.

Pour en savoir plus sur les applications tierces, consultez [Applications tierces](/docs/fr-CA/get-started/applications/third-party-applications). Pour une comparaison détaillée de ce qui est offert dans chaque mode, consultez [Comparaison des fonctionnalités](#feature-comparison). Pour obtenir des conseils sur des schémas d’intégration précis, consultez [Scénarios courants](#common-scenarios).

<div id="how-are-you-affected">
  ## Quelles sont les répercussions pour vous?
</div>

Cette migration a deux répercussions pour vous :

<div id="1-management-api-default-the-deprecation">
  ### 1. Valeur par défaut de Management API (la dépréciation)
</div>

Lorsque vous créez des applications tierces au moyen de `POST /api/v2/clients`, la valeur par défaut de `third_party_security_mode` passera de `permissive` (comportement existant) à `strict` (contrôles de sécurité renforcés) le 23 octobre 2026.

Toutes les applications tierces créées dans le Auth0 Dashboard utilisent déjà des contrôles de sécurité renforcés. Ce paramètre n’est pas configurable dans le Auth0 Dashboard.

<div id="2-dynamic-client-registration-independent-configuration">
  ### 2. Dynamic Client Registration (configuration indépendante)
</div>

Si vous utilisez [Dynamic Client Registration](/docs/fr-CA/get-started/applications/dynamic-client-registration), les clients DCR sont régis par un paramètre de tenant distinct, `dynamic_client_registration_security_mode`. Ce réglage est indépendant de la dépréciation et nécessite sa propre décision de configuration.

<div id="migration-tasks">
  ## Tâches de migration
</div>

<div id="review-your-third-party-applications">
  ### Passez en revue vos applications tierces
</div>

Avant de choisir une voie de migration, examinez le fonctionnement de vos applications tierces afin de bien comprendre leurs exigences actuelles. Tenez compte des éléments suivants :

* Quels types d’octroi utilisent-elles ? (`authorization code`, `implicit`, `client credentials`, etc.)
* Ont-elles besoin de scopes OIDC (`openid`, `profile`, `email`) ou de jetons ID ?
* Utilisent-elles Classic Login ou des points de terminaison hérités ?
* Comment sont-elles créées ? (manuellement dans le Dashboard ou via l’API, ou dynamiquement via DCR)

Si vous avez un petit nombre d’applications tierces, vous pouvez les examiner individuellement dans l’Auth0 Dashboard ou à l’aide de la Management API. Chaque application comprend une propriété `third_party_security_mode` qui indique son mode de sécurité actuel (`strict` ou `permissive`).

**Renforcez les applications permissives existantes** : Envisagez de passer en revue les [politiques d’accès aux API](/docs/fr-CA/get-started/apis/api-access-policies-for-applications) de vos API et de les définir sur **Require Client Grant** lorsque c’est approprié. Cela garantit que les applications tierces permissives doivent disposer d’un octroi explicite pour accéder à ces API. Notez que cette politique s’applique aussi aux applications de première partie; examinez donc vos intégrations existantes avant de la modifier.

Pour comprendre ce que les contrôles de sécurité renforcés signifient pour vos intégrations, consultez la [comparaison des fonctionnalités](#feature-comparison) ci-dessous et vérifiez si votre cas correspond à l’un des [scénarios courants](#common-scenarios). La section [FAQ](#frequently-asked-questions) répond aussi aux questions fréquentes sur la migration.

<div id="step-1-choose-how-to-create-new-third-party-applications">
  ### Étape 1 : Choisissez comment créer de nouvelles applications tierces
</div>

Déterminez si les nouvelles applications tierces créées à l’aide de la Management API doivent être soumises à des contrôles de sécurité renforcés ou conserver leur comportement existant. Ce choix s’applique uniquement à `POST /api/v2/clients`. Les applications créées dans l’Auth0 Dashboard utilisent toujours des contrôles renforcés.

<div id="option-a-complete-the-migration-recommended">
  #### Option A : Terminer la migration (recommandée)
</div>

Terminez la migration avant le 23 octobre 2026 pour que les contrôles de sécurité renforcés deviennent le comportement par défaut. Cette approche harmonise vos applications tierces avec les pratiques exemplaires d’OAuth 2.1 et les prépare aux capacités futures.

<div id="1-test-enhanced-security-controls">
  ##### 1. Tester les contrôles de sécurité renforcés
</div>

Créez une application tierce de test avec des contrôles de sécurité renforcés afin de valider la compatibilité :

<AuthCodeGroup>
  ```bash cURL
  curl --request POST \
    --url 'https://{yourDomain}/api/v2/clients' \
    --header 'Authorization: Bearer {YOUR_MANAGEMENT_API_TOKEN}' \
    --header 'Content-Type: application/json' \
    --data '{
      "name": "Test Third-Party App",
      "is_first_party": false,
      "third_party_security_mode": "strict",
      "app_type": "regular_web",
      "callbacks": ["https://partner.example.com/callback"],
      "grant_types": ["authorization_code", "refresh_token"]
    }'
  ```
</AuthCodeGroup>

La réponse inclura un `client_id` avec le préfixe `tpc_` et `third_party_security_mode: "strict"`.

<div id="2-set-up-default-api-permissions">
  ##### 2. Configurer les permissions d’API par défaut
</div>

Les applications tierces dotées de contrôles de sécurité renforcés nécessitent des attributions d’accès client explicites pour accéder aux API. Les autorisations par défaut définissent un ensemble de base d’API et de scopes auxquels toutes les applications tierces peuvent accéder automatiquement. C’est particulièrement important pour les clients créés dynamiquement, lorsque vous ne pouvez pas configurer les autorisations pour chaque application individuellement.

Vous pouvez aussi définir des autorisations précises pour certaines applications (par `client_id`) afin d’accorder un accès plus étendu ou plus restreint que les valeurs par défaut. Lorsque les deux existent, les autorisations par application ont préséance sur les valeurs par défaut.

<Tabs>
  <Tab title="Auth0 Dashboard">
    1. Accédez à **Applications &gt; APIs**.
    2. Sélectionnez l’API à laquelle vous voulez donner accès aux applications tierces.
    3. Sous l’onglet **Settings**, faites défiler jusqu’à **Autorisations par défaut pour les applications tierces**.
    4. Sélectionnez **Authorized** pour Accès utilisateur et/ou Accès client.
    5. Sélectionnez les scopes à accorder.
    6. Cliquez sur **Save**.
  </Tab>

  <Tab title="Management API">
    <AuthCodeGroup>
      ```bash cURL
      curl --request POST \
        --url 'https://{yourDomain}/api/v2/client-grants' \
        --header 'Authorization: Bearer {YOUR_MANAGEMENT_API_TOKEN}' \
        --header 'Content-Type: application/json' \
        --data '{
          "default_for": "third_party_clients",
          "audience": "https://api.example.com",
          "scope": ["read:items", "write:items"],
          "subject_type": "user"
        }'
      ```
    </AuthCodeGroup>
  </Tab>
</Tabs>

Vous pouvez configurer des permissions d’API par défaut distinctes pour les applications tierces selon l’accès utilisateur (`subject_type: "user"`) et l’accès machine-to-machine (`subject_type: "client"`).

Pour en savoir plus, consultez [Autorisations par défaut pour les applications tierces](/docs/fr-CA/get-started/applications/application-access-to-apis-client-grants#default-permissions-for-third-party-applications).

<div id="3-validate-compatibility">
  ##### 3. Valider la compatibilité
</div>

Testez vos processus de création d’applications tierces avec les contrôles de sécurité renforcés activés. Confirmez que :

* Vos applications peuvent utiliser les types d’octroi `authorization_code`, `refresh_token` et `client_credentials`
* PKCE est implémenté dans vos flux d’autorisation
* Vous n’avez pas besoin de scopes OIDC
* Vous n’avez pas besoin de Classic Login ni de points de terminaison hérités
* Votre tenant n’a pas de [Rules](/docs/fr-CA/customize/rules) actives devant s’exécuter pour les flux de connexion d’applications tierces. Les Rules ne sont pas prises en charge pour les applications tierces `strict` et entraîneront une erreur. Si vous utilisez Rules, envisagez de [migrer vers Actions](/docs/fr-CA/customize/actions/migrate/migrate-from-rules-to-actions) ou d’utiliser le mode permissif.

Si vous constatez des problèmes de compatibilité, consultez [dépannage des applications tierces](/docs/fr-CA/get-started/applications/third-party-applications/troubleshooting).

<div id="4-complete-the-migration">
  ##### 4. Terminer la migration
</div>

Une fois la compatibilité validée, terminez la migration en désactivant la bascule **Create Permissive Third-Party Clients by Default** :

<Frame>![Créer des clients tiers permissifs Bt par défaut](/docs/images/third-party-applications/create_permissive_3p_clients_by_default.png)</Frame>

Dans l’Auth0 Dashboard :

1. Accédez à **Settings &gt; Advanced**.
2. Faites défiler jusqu’à la section **Migrations**.
3. Désactivez **Create Permissive Third-Party Clients by Default**.
4. Sélectionnez **Save**.

Après avoir terminé la migration, lorsque vous créez des applications tierces au moyen de `POST /api/v2/clients`, vous pouvez :

* omettre le paramètre `third_party_security_mode` (les contrôles renforcés s’appliquent par défaut), ou
* définir explicitement `third_party_security_mode: "strict"`

**Après le 23 octobre 2026** : la bascule **Create Permissive Third-Party Clients by Default** sera automatiquement désactivée pour tous les tenants admissibles. Pour créer des applications avec le comportement existant, vous devez préciser explicitement `third_party_security_mode: "permissive"` dans la requête `POST` vers l’endpoint `/api/v2/clients`.

<div id="option-b-preserve-existing-behavior-as-the-default">
  #### Option B : Conserver le comportement actuel par défaut
</div>

Si vous devez continuer à créer des applications tierces en conservant le comportement actuel par défaut, vous pouvez laisser la bascule **Create Permissive Third-Party Clients by Default** activée jusqu’à ce que vous soyez prêt à adopter des contrôles de sécurité renforcés.

<Warning>
  Cette option préserve vos processus actuels, mais n’offre pas les avantages de sécurité des contrôles renforcés. Auth0 recommande fortement d’adopter des contrôles de sécurité renforcés pour toutes les nouvelles applications tierces.
</Warning>

<div id="before-the-deadline">
  ##### Avant l’échéance
</div>

La bascule **Create Permissive Third-Party Clients by Default** reste activée (aucune action n’est requise sur la bascule elle-même). Cependant, vous devez préparer vos flux de travail pour préciser explicitement le mode de sécurité après l’échéance.

Mettez à jour votre code de création d’application pour transmettre explicitement `third_party_security_mode: "permissive"` :

<AuthCodeGroup>
  ```bash cURL
  curl --request POST \
    --url 'https://{yourDomain}/api/v2/clients' \
    --header 'Authorization: Bearer {YOUR_MANAGEMENT_API_TOKEN}' \
    --header 'Content-Type: application/json' \
    --data '{
      "name": "Partner Integration",
      "is_first_party": false,
      "third_party_security_mode": "permissive",
      "app_type": "regular_web",
      "callbacks": ["https://partner.example.com/callback"]
    }'
  ```
</AuthCodeGroup>

Testez cette approche avant l’échéance pour vous assurer que vos flux de travail gèrent correctement ce paramètre explicite.

<div id="after-the-deadline">
  ##### Après l’échéance
</div>

Le 23 octobre 2026, la bascule **Create Permissive Third-Party Clients by Default** sera automatiquement désactivée. Pour continuer à créer des applications avec le comportement existant, vous devez définir explicitement `third_party_security_mode: "permissive"` dans chaque requête `POST /api/v2/clients`.

Si vous omettez le paramètre `third_party_security_mode`, des contrôles de sécurité renforcés s’appliqueront par défaut.

<div id="step-2-choose-how-to-handle-dynamic-client-registration">
  ### Étape 2 : Choisir comment traiter Dynamic Client Registration
</div>

Si vous utilisez [Dynamic Client Registration](/docs/fr-CA/get-started/applications/dynamic-client-registration), configurez séparément le mode de sécurité des clients DCR. Ce réglage est indépendant du [changement de valeur par défaut de la Management API](/docs/fr-CA/troubleshoot/product-lifecycle/deprecations-and-migrations/migrate-to-enhanced-security-third-party-applications#1-management-api-default-the-deprecation) et peut être configuré en tout temps.

<Warning>
  Avant d’activer les contrôles de sécurité renforcés pour DCR, assurez-vous d’avoir configuré les [autorisations par défaut pour les applications tierces](/docs/fr-CA/get-started/applications/application-access-to-apis-client-grants#default-permissions-for-third-party-applications) pour les applications tierces. Sans permissions par défaut, les clients DCR ne pourront accéder à aucune API.
</Warning>

<div id="review-current-dcr-behavior">
  #### Vérifier le comportement DCR actuel
</div>

Vérifiez le réglage actuel du mode de sécurité DCR :

<Tabs>
  <Tab title="Auth0 Dashboard">
    1. Accédez à **Settings &gt; Advanced**. Sous **Dynamic Client Registration (DCR) Security Mode**, vérifiez la valeur actuelle.

    <Frame>![Paramètres avancés du tenant dans le Dashboard avec la liste déroulante du mode de sécurité DCR](/docs/images/third-party-applications/dcr-security-mode.png)</Frame>
  </Tab>

  <Tab title="Management API">
    Effectuez une requête `GET` vers le point de terminaison `/api/v2/tenants/settings` pour récupérer les paramètres de votre tenant :

    <AuthCodeGroup>
      ```bash cURL
      curl --request GET \
        --url 'https://{yourDomain}/api/v2/tenants/settings' \
        --header 'Authorization: Bearer {YOUR_MANAGEMENT_API_TOKEN}'
      ```
    </AuthCodeGroup>

    Recherchez la propriété `dynamic_client_registration_security_mode` dans la réponse. Si elle n’est pas présente, les clients DCR utilisent actuellement le comportement existant par défaut.
  </Tab>
</Tabs>

<div id="configure-dcr-security-mode">
  #### Configurer le mode de sécurité de DCR
</div>

Choisissez le mode de sécurité à appliquer aux clients enregistrés dynamiquement :

**Option A : Contrôles de sécurité renforcés pour les clients DCR** (recommandé)

<Note>
  Avant d’activer le mode `strict` pour DCR, configurez les [autorisations par défaut pour les applications tierces](/docs/fr-CA/get-started/applications/application-access-to-apis-client-grants#default-permissions-for-third-party-applications) pour les applications tierces. Sans autorisations par défaut, les clients DCR ne pourront pas accéder aux API.
</Note>

Définissez `dynamic_client_registration_security_mode` sur `strict` :

<AuthCodeGroup>
  ```bash cURL
  curl --request PATCH \
    --url 'https://{yourDomain}/api/v2/tenants/settings' \
    --header 'Authorization: Bearer {YOUR_MANAGEMENT_API_TOKEN}' \
    --header 'Content-Type: application/json' \
    --data '{
      \"dynamic_client_registration_security_mode\": \"strict\"
    }'
  ```
</AuthCodeGroup>

**Option B : Conserver le comportement existant pour les clients DCR**

Conservez `dynamic_client_registration_security_mode` à `permissive` ou définissez-le sur cette valeur :

<AuthCodeGroup>
  ```bash cURL
  curl --request PATCH \
    --url 'https://{yourDomain}/api/v2/tenants/settings' \
    --header 'Authorization: Bearer {YOUR_MANAGEMENT_API_TOKEN}' \
    --header 'Content-Type: application/json' \
    --data '{
      \"dynamic_client_registration_security_mode\": \"permissive\"
    }'
  ```
</AuthCodeGroup>

Pour en savoir plus, consultez [Dynamic Client Registration](/docs/fr-CA/get-started/applications/dynamic-client-registration).

<div id="feature-comparison">
  ## Comparaison des fonctionnalités
</div>

Le tableau suivant compare les capacités offertes par chaque mode de sécurité :

| Capacité                          | Contrôles de sécurité renforcés                                                                                                               | Comportement existant                      |
| --------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------ |
| **Types d’octroi**                | `authorization_code`, `refresh_token`, `client_credentials`                                                                                   | Tous les types d’octroi sont offerts |
| **PKCE**                          | Obligatoire                                                                                                                                   | Facultatif                                 |
| **OIDC**                          | Non offert. Prévu dans une prochaine version.                                                                                                 | Pris en charge                             |
| **Autorisation de l’API**         | Exige toujours un client grant explicite                                                                                                      | Suit la politique d’accès de l’API         |
| **Classic Login**                 | Non pris en charge                                                                                                                            | Pris en charge                             |
| **Points de terminaison hérités** | Non offerts                                                                                                                                   | Offerts                                    |
| **Format du Client ID**           | Préfixe `tpc_`                                                                                                                                | Format standard                            |
| **Propriétés configurables**      | [Ensemble restreint de propriétés](/docs/fr-CA/get-started/applications/third-party-applications/security-controls#restricted-client-configuration) | Toutes les propriétés                      |
| **Capacités futures**             | Limites de débit et futures capacités améliorées en matière de sécurité et de gestion                                                         | Non offertes                               |
| **Création via Dashboard**        | Utilise toujours des contrôles renforcés                                                                                                      | Non offerte via Dashboard                  |

<div id="common-scenarios">
  ## Cas fréquents
</div>

<div id="scenario-1-partner-integrations-using-modern-oauth">
  ### Scénario 1 : Intégrations partenaires utilisant OAuth moderne
</div>

**Situation** : Vous avez des intégrations partenaires qui utilisent le flux du code d’autorisation avec PKCE et qui accèdent à vos API.

**Recommandation** : Adoptez les contrôles de sécurité renforcés (Option A). Cette option est entièrement compatible avec les mises en œuvre modernes d’OAuth et offre une sécurité accrue.

**Étapes** :

1. Configurez les permissions d’API par défaut pour vos API
2. Testez la création d’une application partenaire avec `third_party_security_mode: "strict"`
3. Terminez la migration en désactivant la bascule de migration

<div id="scenario-2-applications-requiring-oidc">
  ### Scénario 2 : Applications nécessitant OIDC
</div>

**Situation** : Vos applications tierces nécessitent des scopes OIDC (openid, profile, email) ou des ID tokens.

**Recommandation** : La prise en charge d’OIDC pour les applications tierces est prévue dans une prochaine version. D’ici là, conservez le comportement existant (option B) ou migrez vers des jetons d’accès associés à des scopes d’API.

**Étapes** :

* Si vous devez utiliser OIDC, laissez la bascule de migration activée et transmettez explicitement `third_party_security_mode: "permissive"` lors de la création des applications
* Sinon, mettez à jour vos intégrations pour utiliser des scopes d’API au lieu de scopes OIDC

<div id="scenario-3-dynamic-client-registration-mcp-ai-agents">
  ### Scénario 3 : Dynamic Client Registration (MCP, agents d’IA)
</div>

**Situation** : Vous utilisez DCR pour des agents d’IA, des serveurs MCP ou des applications de portail pour développeurs.

**Recommandation** : Configurez `dynamic_client_registration_security_mode: "strict"` et définissez les permissions d’API par défaut. Les clients MCP (Claude Code, VS Code) sont compatibles avec les contrôles de sécurité renforcés.

**Étapes** :

1. Configurer les permissions d’API par défaut
2. Définir `dynamic_client_registration_security_mode: "strict"` au moyen de la Management API
3. Tester un enregistrement DCR
4. Vérifier que les clients DCR peuvent obtenir des jetons d’accès

<div id="scenario-4-applications-using-classic-login">
  ### Scénario 4 : Applications utilisant Classic Login
</div>

**Situation** : Vos applications tierces utilisent Classic Login plutôt que Universal Login.

**Recommandation** : Classic Login n’est pas pris en charge pour les applications tierces avec des contrôles de sécurité renforcés. Migrez vers Universal Login ou conservez le comportement existant.

**Étapes** :

* Recommandé : migrez vers Universal Login avant d’adopter les contrôles renforcés
* Alternative : laissez la bascule de migration activée et utilisez explicitement `third_party_security_mode: "permissive"`

<div id="frequently-asked-questions">
  ## Foire aux questions
</div>

<div id="can-i-change-the-security-mode-on-an-existing-application">
  ### Puis-je modifier le mode de sécurité d’une application existante ?
</div>

Non. Le `third_party_security_mode` est défini au moment de la création de l’application et ne peut pas être modifié par la suite. Pour utiliser un autre mode de sécurité, créez une nouvelle application.

<div id="what-happens-to-my-existing-third-party-applications">
  ### Que deviennent mes applications tierces existantes ?
</div>

Rien. Les applications tierces existantes continuent de fonctionner exactement comme avant. Cette migration n’affecte que le réglage par défaut des applications nouvellement créées.

<div id="can-i-use-both-security-modes-in-the-same-tenant">
  ### Puis-je utiliser les deux modes de sécurité dans le même tenant?
</div>

Oui. Vous pouvez avoir certaines applications tierces avec des contrôles de sécurité renforcés et d’autres avec le comportement existant. Le mode de sécurité se configure pour chaque application.

<div id="what-about-dynamic-client-registration">
  ### Qu’en est-il de Dynamic Client Registration ?
</div>

Le DCR est régi par un paramètre de tenant distinct, `dynamic_client_registration_security_mode`. Cela est indépendant de la mise en désuétude et nécessite une décision de configuration distincte. Pour en savoir plus, consultez [Dynamic Client Registration](/docs/fr-CA/get-started/applications/dynamic-client-registration).

<div id="can-i-create-applications-with-existing-behavior-after-the-deadline">
  ### Puis-je créer des applications avec le comportement existant après la date limite?
</div>

Oui, mais uniquement au moyen de la Management API. Définissez explicitement `third_party_security_mode: "permissive"` lors de la création de chaque application. L’Auth0 Dashboard ne permet pas de créer des applications avec le comportement existant.

<div id="will-future-features-work-with-existing-behavior">
  ### Les futures fonctionnalités seront-elles compatibles avec le comportement existant?
</div>

Certaines fonctionnalités futures (comme les limites de débit au niveau de l’application) ne seront offertes qu’aux applications dotées de contrôles de sécurité renforcés.
