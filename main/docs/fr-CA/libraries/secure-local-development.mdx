Les environnements de développement local utilisent généralement http (c.-à-d. `http://localhost`) par défaut. Ce guide explique à quel moment vous devriez utiliser https pour un serveur local et comment configurer https sur localhost.

<div id="when-to-use-a-local-https-server">
  ## Quand utiliser un serveur https local
</div>

Les navigateurs Web ne se comportent pas de la même façon avec http et https; pour tenir compte de ces différences, il se peut que votre serveur de développement doive servir les pages en https. Par exemple :

* Lorsque votre page est servie en https, les requêtes pour charger des ressources à partir d’une URL http sont bloquées. Vous voudrez peut-être reproduire ce comportement en local pour éviter des bogues qui ne se manifesteraient qu’en environnement de production, ou parce que vous ne voulez pas envoyer par erreur des requêtes http contenant des renseignements sensibles.
* Les cookies qui utilisent l’attribut Secure ou dont l’attribut [SameSite](https://auth0.com/blog/browser-behavior-changes-what-developers-need-to-know/) est défini à None ne sont pas envoyés sur des canaux non sécurisés, ce qui perturbe l’authentification et d’autres fonctionnalités.

<div id="how-to-set-up-a-secure-local-server">
  ## Comment configurer un serveur local sécurisé
</div>

L’idée est d’exécuter un serveur web devant votre serveur de développement local, qui acceptera les requêtes https et les transmettra au serveur web de votre application par http, de sorte que les requêtes de votre application soient servies en HTTPS par l’entremise d’un proxy. C’est ce qu’on appelle un proxy inverse https, et nous pouvons utiliser un produit prêt à l’emploi appelé [Caddy](https://caddyserver.com/v2) pour en créer un.

Le processus est le suivant :

1. Télécharger le binaire Caddy
2. Exécuter le serveur web de votre application sur http comme d’habitude
3. Exécuter le binaire Caddy comme proxy inverse en lui indiquant de transférer le trafic vers le serveur web de votre application
4. Accéder au serveur web du proxy inverse en HTTPS

<div id="step-1-download-the-caddy-binary">
  ### Étape 1. Téléchargez le binaire Caddy
</div>

Rendez-vous à l’adresse https://caddyserver.com/download, sélectionnez votre plateforme et téléchargez le fichier sur votre ordinateur.

<div id="step-2-run-your-applications-web-server-over-http-as-normal">
  ### Étape 2. Exécutez le serveur Web de votre application en HTTP, comme d’habitude
</div>

Démarrez le serveur de développement local de votre application comme d’habitude, par exemple $ PORT=3000 npm start

<div id="step-3-run-the-caddy-binary-as-a-reverse-proxy-telling-it-to-forward-traffic-to-your-applications-web-server">
  ### Étape 3. Exécutez le fichier binaire Caddy en tant que proxy inverse en lui indiquant d’acheminer le trafic vers le serveur Web de votre application
</div>

Repérez l’emplacement du fichier binaire Caddy que vous avez téléchargé et exécutez la commande suivante pour créer un proxy inverse et acheminer les requêtes du port 443 (le port par défaut utilisé pour https) vers le port de votre application (dans ce cas-ci 3000) :

```bash wrap lines
/path/to/caddy reverse-proxy --from localhost:443 --to localhost:3000
```

<div id="step-4-now-visit-your-web-server-over-https">
  ### Étape 4. Accédez maintenant à votre serveur Web en HTTPS
</div>

Accédez à https://localhost et votre serveur traitera les requêtes de votre application en HTTPS

<div id="running-your-application-behind-a-proxy">
  ## Exécuter votre application derrière un proxy
</div>

Il se peut que vous deviez apporter quelques légers ajustements à votre application lorsqu’elle s’exécute derrière un proxy.

Par exemple, dans Express.js, même si l’application continuera de fonctionner, elle enregistrera incorrectement l’adresse IP du proxy comme adresse IP du client, à moins que `trust proxy` ne soit configuré. Consultez &lt;https://expressjs.com/en/guide/behind-proxies.html&gt;
