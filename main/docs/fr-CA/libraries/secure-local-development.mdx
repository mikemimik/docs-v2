Les environnements de développement local s’exécutent généralement en HTTP (c.-à-d. `http://localhost`) par défaut. Ce guide explique quand vous devriez exécuter un serveur local en HTTPS et comment configurer HTTPS sur localhost.

<div id="when-to-use-a-local-https-server">
  ## Quand utiliser un serveur HTTPS local
</div>

Les navigateurs Web ne se comportent pas de la même façon en HTTP et en HTTPS. Pour tenir compte de ces différences, il se peut que votre serveur de développement doive servir les pages en HTTPS. Par exemple :

* Lorsque votre page est servie en HTTPS, les requêtes visant à charger des ressources à partir d’une URL en HTTP sont bloquées. Vous pourriez vouloir reproduire ce comportement en local pour éviter des bogues qui ne se manifesteraient que dans l’environnement Production, ou encore pour éviter d’envoyer par erreur des requêtes HTTP contenant des renseignements sensibles.
* Les cookies qui utilisent l’attribut Secure ou l’attribut [SameSite](https://auth0.com/blog/browser-behavior-changes-what-developers-need-to-know/) réglé sur None ne sont pas envoyés sur des canaux non sécurisés, ce qui perturbe l’authentification et d’autres fonctionnalités.

<div id="how-to-set-up-a-secure-local-server">
  ## Comment configurer un serveur local sécurisé
</div>

L’idée est d’exécuter un serveur web devant votre serveur de développement local pour qu’il accepte les requêtes HTTPS et les transmette au serveur web de votre application en HTTP, de façon à servir les requêtes de votre application en HTTPS par l’entremise d’un proxy. C’est ce qu’on appelle un proxy inverse HTTPS, et nous pouvons utiliser un produit prêt à l’emploi appelé [Caddy](https://caddyserver.com/v2) pour en créer un.

Le processus est le suivant :

1. Télécharger le binaire Caddy
2. Exécuter le serveur web de votre application en HTTP comme d’habitude
3. Exécuter le binaire Caddy comme proxy inverse en lui indiquant de transférer le trafic vers le serveur web de votre application
4. Accéder au serveur web du proxy inverse en HTTPS

<div id="step-1-download-the-caddy-binary">
  ### Étape 1. Téléchargez le fichier binaire Caddy
</div>

Rendez-vous sur https://caddyserver.com/download, sélectionnez votre plateforme et téléchargez le fichier sur votre ordinateur.

<div id="step-2-run-your-applications-web-server-over-http-as-normal">
  ### Étape 2. Démarrez le serveur web de votre application en HTTP, comme d’habitude
</div>

Démarrez le serveur de développement local de votre application comme d’habitude, par exemple $ PORT=3000 npm start

<div id="step-3-run-the-caddy-binary-as-a-reverse-proxy-telling-it-to-forward-traffic-to-your-applications-web-server">
  ### Étape 3. Exécutez le binaire Caddy comme proxy inverse pour transférer le trafic vers le serveur Web de votre application
</div>

Repérez l’emplacement du binaire Caddy que vous avez téléchargé, puis exécutez la commande suivante pour créer un proxy inverse et transférer les requêtes du port 443 (le port par défaut utilisé pour https) vers le port de votre application (dans ce cas-ci, 3000) :

```bash wrap lines
/path/to/caddy reverse-proxy --from localhost:443 --to localhost:3000
```

<div id="step-4-now-visit-your-web-server-over-https">
  ### Étape 4. Accédez maintenant à votre serveur Web en HTTPS
</div>

Accédez à https://localhost et il répondra aux requêtes de votre application en HTTPS

<div id="running-your-application-behind-a-proxy">
  ## Exécuter votre application derrière un proxy
</div>

Il se peut que vous deviez apporter quelques ajustements mineurs à votre application lorsqu’elle s’exécute derrière un proxy.

Par exemple, dans Express.js, même si l’application continuera de s’exécuter, elle enregistrera par erreur l’adresse IP du proxy comme adresse IP du client, à moins que `trust proxy` ne soit configuré. Consultez &lt;https://expressjs.com/en/guide/behind-proxies.html&gt;
