Les environnements de développement local utilisent généralement HTTP (c.-à-d. `http://localhost`) par défaut. Ce guide explique à quel moment il faut exécuter un serveur local en HTTPS et comment configurer HTTPS sur localhost.

<div id="when-to-use-a-local-https-server">
  ## Quand utiliser un serveur HTTPS local
</div>

Les navigateurs Web ne se comportent pas de la même façon en HTTP et en HTTPS. Pour tenir compte de ces différences, il se peut que votre serveur de développement doive servir les pages en HTTPS. Par exemple :

* Lorsque votre page est servie en HTTPS, les requêtes visant à charger des ressources à partir d’une URL HTTP sont bloquées. Vous pourriez vouloir reproduire ce comportement en local pour éviter des bogues qui ne se manifesteraient que dans l’environnement de production, ou parce que vous ne voulez pas envoyer par erreur des requêtes HTTP contenant des renseignements sensibles.
* Les cookies qui utilisent l’attribut Secure ou dont l’attribut [SameSite](https://auth0.com/blog/browser-behavior-changes-what-developers-need-to-know/) est défini à None ne seront pas envoyés sur des canaux non sécurisés, ce qui perturbera l’authentification et d’autres fonctionnalités.

<div id="how-to-set-up-a-secure-local-server">
  ## Comment configurer un serveur local sécurisé
</div>

L’idée est d’exécuter un serveur Web devant votre serveur de développement local afin qu’il accepte les requêtes HTTPS et les transmette au serveur Web de votre application en HTTP, de sorte que les requêtes de votre application soient servies en HTTPS par l’entremise d’un proxy. C’est ce qu’on appelle un proxy inverse HTTPS, et nous pouvons utiliser un produit prêt à l’emploi appelé [Caddy](https://caddyserver.com/v2) pour créer un proxy inverse HTTPS.

Voici les étapes à suivre :

1. Téléchargez le binaire Caddy
2. Exécutez le serveur Web de votre application en HTTP comme d’habitude
3. Exécutez le binaire Caddy comme proxy inverse en lui indiquant de transférer le trafic vers le serveur Web de votre application
4. Accédez au serveur Web du proxy inverse en HTTPS

<div id="step-1-download-the-caddy-binary">
  ### Étape 1. Téléchargez le binaire Caddy
</div>

Rendez-vous sur https://caddyserver.com/download, sélectionnez votre plateforme et téléchargez le fichier sur votre ordinateur.

<div id="step-2-run-your-applications-web-server-over-http-as-normal">
  ### Étape 2. Exécutez le serveur Web de votre application en HTTP, comme d’habitude
</div>

Exécutez le serveur de développement local de votre application normalement, par exemple $ PORT=3000 npm start

<div id="step-3-run-the-caddy-binary-as-a-reverse-proxy-telling-it-to-forward-traffic-to-your-applications-web-server">
  ### Étape 3. Exécutez le binaire Caddy comme proxy inverse en le configurant pour transmettre le trafic vers le serveur Web de votre application
</div>

Repérez l’emplacement du binaire Caddy téléchargé et exécutez la commande suivante pour créer un proxy inverse et transmettre les requêtes du port 443 (le port par défaut utilisé pour https) vers le port de votre application (dans ce cas-ci 3000) :

```bash wrap lines
/path/to/caddy reverse-proxy --from localhost:443 --to localhost:3000
```

<div id="step-4-now-visit-your-web-server-over-https">
  ### Étape 4. Accédez maintenant à votre serveur web en HTTPS
</div>

Accédez à https://localhost et il répondra aux requêtes de votre application en HTTPS

<div id="running-your-application-behind-a-proxy">
  ## Exécuter votre application derrière un proxy
</div>

Il se peut que vous deviez apporter quelques légers ajustements à votre application lorsqu’elle fonctionne derrière un proxy.

Par exemple, dans Express.js, même si l’application peut s’exécuter sans problème, elle enregistrera à tort l’adresse IP du proxy comme étant celle du client, à moins que `trust proxy` ne soit configuré. Consultez &lt;https://expressjs.com/en/guide/behind-proxies.html&gt;
