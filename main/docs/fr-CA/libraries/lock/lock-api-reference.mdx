<div id="auth0lock">
  ## Auth0Lock
</div>

`new Auth0Lock(clientID, domain, options)`

Initialise une nouvelle instance de `Auth0Lock` configurée avec le `clientID` de votre application et le `domain` de votre compte, tirés de votre [tableau de bord de gestion Auth0](https://manage.auth0.com/#/). Le troisième paramètre, facultatif, est un objet `options` servant à configurer Lock selon les besoins de votre application. Vous trouverez cette information dans les [paramètres de l’application](https://manage.auth0.com/#/applications).

* **clientId `{String}`** : Paramètre requis. Le clientId de votre application dans Auth0.
* **domain `{String}`** : Paramètre requis. Votre domaine Auth0. Habituellement, your-account.auth0.com.
* **options `{Object}`** : Paramètre facultatif. Permet de configurer l’apparence et le comportement de Lock. Consultez [la page des options de configuration](/docs/fr-CA/libraries/lock/lock-configuration) pour en savoir plus.

export const codeExample1 = `var Auth = (function() {

  var privateStore = {};

  function Auth() {
    // Instancier Lock - sans options personnalisées
    this.lock = new Auth0Lock(
      '<{yourClientId}>',
      '<{yourDomain}>'
    );
  }

  Auth.prototype.getProfile = function() {
    return privateStore.profile;
  };

  Auth.prototype.authn = function() {
    // Écouter l’événement authenticated et récupérer le profil
    this.lock.on("authenticated", function(authResult) {
      // Utiliser le jeton dans authResult pour getUserInfo() et l’enregistrer au besoin
      this.getUserInfo(authResult.accessToken, function(error, profile) {
        if (error) {
          // Gérer l’erreur
          return;
        }

        // enregistrer le jeton d’accès seulement au besoin
        privateStore.accessToken = accessToken;
        privateStore.profile = profile;

        // Mettre à jour le DOM
      });
    });
  };
  return Auth;
}());`;

<div id="getuserinfo">
  ## getUserInfo()
</div>

`getUserInfo(accessToken, callback)`

Une fois l’utilisateur connecté et en possession d’un jeton, vous pouvez utiliser ce jeton pour obtenir le profil de l’utilisateur avec `getUserInfo`. Cette méthode remplace la méthode obsolète `getProfile()`.

* **accessToken &#123;String&#125;** : Jeton de l’utilisateur.
* **callback &#123;Function&#125;** : Est appelée une fois le profil de l’utilisateur récupéré.

```js lines
lock.getUserInfo(accessToken, function(error, profile) {
  if (!error) {
    alert("hello " + profile.name);
  }
});
```

<div id="show">
  ## show()
</div>

`show(options)`

La méthode `show` affiche le widget. À partir de la version 10.2.0 de Lock, la méthode `show` peut désormais accepter un objet `options` comme paramètre. Notez que ce paramètre sert à remplacer les `options` de votre Lock pour cet affichage précis du widget. Les options doivent être définies lors de l’instanciation de Lock, puis remplacées ici uniquement au besoin, selon votre cas d’utilisation.

Le sous-ensemble suivant de `options` peut remplacer les valeurs attribuées lors de l’instanciation de Lock (ou leurs valeurs par défaut) :

* allowedConnections
* auth.params
* allowLogin
* allowSignUp
* allowForgotPassword
* initialScreen
* rememberLastLogin

Pour en savoir plus sur la liste complète des options configurables pouvant être choisies lors de l’instanciation de Lock, par opposition au sous-ensemble limité ci-dessus qui peut être remplacé dans la méthode `show`, veuillez consulter la [page des options configurables par l’utilisateur](/docs/fr-CA/libraries/lock/lock-configuration).

Exemples de remplacement d’options :

```js lines
// Afficher le widget Lock sans remplacer aucune option
lock.show();

// Afficher le widget Lock en remplaçant certaines options
lock.show({
  allowedConnections: ["twitter", "facebook"],
  allowSignUp: false
});
```

Les options doivent être définies lors de la première initialisation de Lock `var lock = new Auth0Lock(clientId, domain, options);`. Les options ne doivent être passées à `show` que pour remplacer celles que vous avez déjà définies au moment d’afficher le widget dans ce contexte précis.

Il existe une option supplémentaire qui peut être définie dans la méthode `show`, appelée `flashMessage`.

<div id="flashmessage">
  ### message flash
</div>

Cet objet est offert uniquement comme option pour la méthode `show`, et non dans l’objet `options` habituel au moment d’instancier Lock. L’objet `message flash` affiche un message flash d’erreur ou de réussite lorsque Lock s’affiche. Il comporte les paramètres suivants :

* **type** &#123;String&#125;: Le type de message; il doit être `error` ou `success`.
* **text** &#123;String&#125;: Le texte à afficher.

```js lines
lock.show({
  flashMessage:{
    type: 'success',
    text: 'Amazing Success!!'
  }
});
```

Une utilisation pratique de l’option `flashMessage` consiste à gérer les erreurs d’autorisation. Le `flashMessage` peut contenir un texte décrivant l’erreur.

```js lines
lock.on('authorization_error', function(error) {
  lock.show({
    flashMessage: {
      type: 'error',
      text: error.errorDescription
    }
  });
});
```

Ainsi, si `tester@example.com` essayait maintenant de se connecter, puisqu’il s’agit d’un utilisateur bloqué, Lock s’afficherait de nouveau pour lui, avec une barre supérieure indiquant le message d’erreur, au lieu que la connexion échoue simplement et que Lock se ferme.

<div id="hide">
  ## hide()
</div>

`hide()`

La méthode `hide` ferme le widget s’il est ouvert. Comme le widget se ferme de lui-même dans la plupart des cas, cette méthode est surtout utilisée dans des cas d’usage précis. Par exemple, on peut vouloir écouter l’événement `unrecoverable_error`, puis `hide` le Lock et rediriger l’utilisateur vers une page d’erreur personnalisée. Un autre exemple concerne les utilisateurs qui mettent en œuvre le [mode popup](/docs/fr-CA/libraries/lock/lock-authentication-modes) et qui pourraient devoir `hide` manuellement le widget après le déclenchement de l’événement `authenticated`.

Exemple d’utilisation pour masquer (fermer) le widget Lock en mode popup :

```js lines
// Écouter l'événement authenticated et masquer Lock
lock.on("authenticated", function() {
  lock.hide();

  // Tout autre traitement à effectuer lors de l'événement authenticated

});
```

<div id="on">
  ## on()
</div>

Lock émet des événements au cours de son cycle de vie. La méthode `on` peut être utilisée pour écouter certains événements et y réagir.

* `show` : émis lorsque Lock est affiché. Ne prend aucun argument.
* `hide` : émis lorsque Lock est masqué. Ne prend aucun argument.
* `unrecoverable_error` : émis lorsqu’une erreur irrécupérable se produit, par exemple lorsqu’aucune connection n’est disponible. Prend l’erreur comme seul argument.
* `authenticated` : émis après une authentication réussie. Prend le résultat de l’authentication comme seul argument. Le résultat de l’authentication contient le jeton, qui peut être utilisé pour obtenir le profil de l’utilisateur ou être stocké pour ouvrir sa session lors de vérifications ultérieures.
* `authorization_error` : émis lorsque l’autorisation échoue. Prend l’erreur comme seul argument.
* `hash_parsed` : chaque fois qu’un nouvel objet Auth0Lock est initialisé en mode de redirection (par défaut), il tente d’analyser la partie hachage de l’URL pour y chercher le résultat d’une tentative de login. Il s’agit d’un événement de bas niveau destiné à des cas d’utilisation avancés; il est préférable d’utiliser `authenticated` et `authorization_error` lorsque possible. Ensuite, cet événement sera émis avec `null` s’il n’a rien trouvé dans le hachage. Il sera émis avec le même argument que l’événement `authenticated` après un login réussi, ou avec le même argument que `authorization_error` si quelque chose s’est mal passé. Cet événement ne sera pas émis en [mode popup](/docs/fr-CA/libraries/lock/lock-authentication-modes), puisqu’il n’est pas nécessaire d’analyser la partie hachage de l’URL.
* `forgot_password ready` : émis lorsque l’écran « mot de passe oublié » est affiché. (Seulement dans la version &gt;`10.18`)
* `forgot_password submit` : émis lorsque l’utilisateur clique sur le bouton de soumission de l’écran « mot de passe oublié ». (Seulement dans la version &gt;`10.14`)
* `signin ready` : émis lorsque l’écran « Sign in » est affiché.
* `signup ready` : émis lorsque l’écran « Sign up » est affiché.
* `signin submit` : émis lorsque l’utilisateur clique sur le bouton de soumission de l’écran « Login ». (Seulement dans la version &gt;`10.18`)
* `signup submit` : émis lorsque l’utilisateur clique sur le bouton de soumission de l’écran « Sign Up ». (Seulement dans la version &gt;`10.18`)
* `federated login` : émis lorsque l’utilisateur clique sur un bouton de connexion sociale. Prend comme arguments le nom de la connection et la stratégie. (Seulement dans la version &gt;`10.18`)
* `socialOrPhoneNumber ready` : émis lorsque l’écran <Tooltip tip="Passwordless : Forme d’authentication qui ne repose pas sur un mot de passe comme premier facteur." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Passwordless">Passwordless</Tooltip> avec Social + numéro de téléphone est affiché
* `socialOrPhoneNumber submit` : émis lorsque l’écran Passwordless avec Social + numéro de téléphone est envoyé
* `socialOrEmail ready` : émis lorsque l’écran Passwordless avec Social + Email est affiché
* `socialOrEmail submit` : émis lorsque l’écran Passwordless avec Social + Email est envoyé
* `vcode ready` : émis lorsque l’écran Passwordless avec le mot de passe à usage unique est affiché
* `vcode submit` : émis lorsque l’écran Passwordless avec le mot de passe à usage unique est envoyé

L’écouteur de l’événement `authenticated` reçoit un seul argument, un objet `authResult`. Cet objet contient les propriétés suivantes : `accessToken`, `idToken`, `state`, `refreshToken` et `idTokenPayload`.

Voici un exemple d’utilisation de l’événement `authenticated` :

export const codeExample2 = `var Auth = (function() {

  var privateStore = {};

  function Auth() {
    this.lock = new Auth0Lock(
      '<{yourClientId}>',
      '<{yourDomain}>'
    );
  }

  Auth.prototype.getProfile = function() {
    return privateStore.profile;
  };

  Auth.prototype.authn = function() {
    // Écoute de l'événement authenticated
    this.lock.on("authenticated", function(authResult) {
      // Utiliser le token contenu dans authResult pour getUserInfo() et l'enregistrer au besoin
      this.getUserInfo(authResult.accessToken, function(error, profile) {
        if (error) {
          // Gérer l'erreur
          return;
        }

        privateStore.profile = profile;

      });
    });
  };
  return Auth;
}());`;

<div id="resumeauth">
  ## resumeAuth()
</div>

Cette méthode ne peut être utilisée que si vous réglez l’option [auth.autoParseHash](/docs/fr-CA/libraries/lock/lock-configuration) à `false`. Vous devrez appeler `resumeAuth` pour terminer le flux d’authentification. Cette méthode est utile lorsque vous utilisez un routeur client-side qui se sert d’un `#` pour gérer les URL (angular2 avec `useHash` ou react-router avec `hashHistory`).

* **hash** &#123;String&#125;: Le fragment de hachage reçu lors de la redirection.
* **callback** &#123;Function&#125;: Sera appelée une fois l’analyse terminée. Elle reçoit une erreur (s’il y en a une) en premier argument, puis le résultat de l’authentification en second. S’il n’y a aucun hachage disponible, les deux arguments seront `null`.

```js lines
lock.resumeAuth(hash, function(error, authResult) {
  if (error) {
    alert("Could not parse hash");
  }
  //Ceci est un exemple seulement ; vous ne devriez pas journaliser les jetons d'accès en production.
  console.log(authResult.accessToken);
});
```

<div id="checksession">
  ## checkSession()
</div>

La méthode `checkSession` vous permet d’obtenir un nouveau jeton d’Auth0 pour un utilisateur déjà authentifié auprès d’Auth0 pour votre domaine. Elle prend les paramètres suivants :

* **options** &#123;Object&#125;: Facultatif. Accepte n’importe quel paramètre OAuth2 valide qui serait normalement envoyé à `/authorize`. Si vous les omettez, les paramètres fournis lors de l’initialisation d’Auth0 seront utilisés.
* **callback** &#123;Function&#125;: Sera appelée avec le résultat du renouvellement du jeton. Reçoit une erreur (le cas échéant) comme premier argument et le résultat de l’authentification comme deuxième argument.

```js lines
lock.checkSession({}, function(err, authResult) {
  // gérer l'erreur ou les nouveaux jetons
});
```

<div id="logout">
  ## logout()
</div>

Déconnecte l’utilisateur.

* **options** &#123;Objet&#125;: Ce paramètre est facultatif et suit les mêmes règles que logout() d’auth0.js.

```js lines
lock.logout({
  returnTo: 'https://myapp.com/bye-bye'
});
```
