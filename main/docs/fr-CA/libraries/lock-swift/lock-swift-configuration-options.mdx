Vous trouverez ci-dessous de nombreuses options pour configurer le comportement de Lock. De plus, la page [Options de personnalisation du style](/docs/fr-CA/libraries/lock-swift/lock-swift-customization) présente aussi plusieurs options pour modifier l’apparence et le style de Lock.

<div id="configuring-locks-behavior">
  ## Configurer le comportement de Lock
</div>

Vous pouvez ajouter des options de configuration à l’initialisation de votre Lock à l’aide de `withOptions`.

```swift lines
Lock
  .classic()
  .withOptions {
    $0.closable = true
    $0.usernameStyle = [.Username]
    $0.allow = [.Login, .ResetPassword]
  }
  .present(from: self)
```

<div id="behavior-options">
  ## Options de comportement
</div>

<div id="closable">
  ### closable
</div>

Permet à l’utilisateur de fermer Lock. Par défaut, cette option est définie sur `false`.

```swift lines
.withOptions {
  $0.closable = true
}
```

<div id="scope">
  ### scope
</div>

Scope utilisé pour l’authentification. Par défaut, la valeur est `openid`. Il renvoie non seulement le **<Tooltip tip="Jeton d’accès : information d’autorisation, sous la forme d’une chaîne opaque ou d’un JWT, utilisée pour accéder à une API." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Access+Token">jeton d’accès</Tooltip>**, mais aussi un **<Tooltip tip="Jeton d’identité : information d’identification destinée au client lui-même, plutôt qu’à l’accès à une ressource." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=ID+Token">jeton d’identité</Tooltip>**, qui est un <Tooltip tip="Jeton d’identité : information d’identification destinée au client lui-même, plutôt qu’à l’accès à une ressource." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=JSON+Web+Token">JSON Web Token</Tooltip> (JWT) contenant des informations sur l’utilisateur. Consultez la documentation sur les [scopes](/docs/fr-CA/get-started/apis/scopes) pour en savoir plus sur les scopes d’authentification.

```swift lines
.withOptions {
  $0.scope = "openid name email picture"
}
```

<div id="refresh-tokens">
  #### Jetons d’actualisation
</div>

Le fait de préciser la portée `offline_access` dans vos options Lock permet de renvoyer un [jeton d’actualisation](/docs/fr-CA/secure/tokens/refresh-tokens) avec l’access&#95;token et l’id&#95;token. <Tooltip tip="Jeton d’actualisation : jeton utilisé pour obtenir un nouveau jeton d’accès sans obliger les utilisateurs à se reconnecter." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Refresh+Tokens">Jetons d’actualisation</Tooltip> peuvent être enregistrés et utilisés pour obtenir un nouveau jeton d’accès lorsque l’ancien expire. Pour en savoir plus sur l’utilisation des jetons d’actualisation avec l’authentification Auth0, consultez la documentation de référence du [SDK Auth0.Swift](/docs/fr-CA/libraries/auth0-swift), que vous utiliseriez pour mettre en œuvre les jetons d’actualisation, ou le [guide de démarrage rapide Swift](/docs/fr-CA/quickstart/native/ios-swift), qui fournit un exemple complet d’utilisation d’Auth0 dans un développement Swift, y compris la gestion des jetons d’actualisation.

<div id="termsofservice">
  ### termsOfService
</div>

Par défaut, Lock utilise les Conditions d’utilisation et la politique de confidentialité d’Auth0, mais vous pouvez indiquer d’autres URL pour créer des liens vers d’autres conditions d’utilisation et politiques.

```swift lines
.withOptions {
  $0.termsOfService = "https://mycompany.com/terms"
  $0.privacyPolicy = "https://mycompany.com/privacy"
}
```

<div id="show-terms-of-service">
  ### Afficher les Conditions d’utilisation
</div>

Les connexions de base de données affichent la boîte de dialogue des Conditions d’utilisation. La valeur par défaut est `true`. Notez que les Conditions d’utilisation seront toujours affichées si l’indicateur `mustAcceptTerms` est activé.

```swift lines
.withOptions {
    $0.showTerms = true
}
```

<div id="require-users-to-accept-the-terms-of-service">
  ### Exiger que les utilisateurs acceptent les Conditions d’utilisation
</div>

Les connexions de base de données nécessitent l’acceptation explicite des Conditions d’utilisation.

```swift lines
.withOptions {
    $0.mustAcceptTerms = true
}
```

<div id="web-authentication-options">
  ## Options authentification Web
</div>

<div id="leeway">
  ### marge de tolérance
</div>

Décalage d’horloge utilisé pour la validation du jeton d’identité. Il élargit la période pendant laquelle le jeton d’identité est encore considéré comme valide, afin de tenir compte de la différence entre l’heure du serveur et celle du client. Par défaut, elle est de **60000 millisecondes** (60 secondes).

```swift lines
.withOptions {
  $0.leeway = 30000 // 30 secondes
}
```

<div id="maxage">
  ### maxAge
</div>

Temps écoulé maximal autorisé (en millisecondes) depuis la dernière authentification de l’utilisateur. Utilisé pour la validation du jeton d’identité. S’il est défini, le jeton d’identité contiendra une claim `auth_time` avec l’horodatage de l’authentification. La valeur par défaut est `nil`.

```swift lines
.withOptions {
  $0.maxAge = 86400000 // 1 jour
}
```

<div id="database-options">
  ## Options de la connexion par base de données
</div>

<div id="allow">
  ### allow
</div>

Quels écrans base de données seront accessibles; par défaut, tous les écrans sont activés, comme `.Login, .Signup, .ResetPassword`.

```swift lines
.withOptions {
  $0.allow = [.Login, .ResetPassword]
}
```

<div id="initialscreen">
  ### initialScreen
</div>

Le premier écran présenté à l’utilisateur. Par défaut, il s’agit de `.Login`; les autres options sont `.Signup` et `ResetPassword`.

```swift lines
.withOptions {
  $0.initialScreen = .Login
}
```

<div id="usernamestyle">
  ### usernameStyle
</div>

Précisez le type d’identifiant exigé pour la connexion. Par défaut, la valeur est `[.Username, .Email]`, mais elle peut aussi être `[.Username]` ou `[.Email]`. Cependant, il est important de noter que cette option n’est active que si vous avez défini l’indicateur `requires_username` à `true` dans votre [Auth0 Dashboard](https://manage.auth0.com/#/).

```swift lines
.withOptions {
  $0.usernameStyle = [.Username]
}
```

<div id="custom-signup-fields">
  #### Champs d’inscription personnalisés
</div>

Lors de l’inscription, les renseignements requis par défaut sont l’adresse courriel et le mot de passe de l’utilisateur. Vous pouvez élargir la collecte de données au besoin. Les champs d’inscription supplémentaires saisis ici seront stockés dans `user_metadata`; vous trouverez plus d’information à ce sujet dans [Métadonnées](/docs/fr-CA/manage-users/user-accounts/metadata). Notez que vous devez préciser l’icône à utiliser avec votre champ de texte personnalisé.

```swift lines
.withOptions {
  $0.customSignupFields = [
    CustomTextField(name: "first_name", placeholder: "First Name", icon: LazyImage(name: "ic_person", bundle: Lock.bundle)),
    CustomTextField(name: "last_name", placeholder: "Last Name", icon: LazyImage(name: "ic_person", bundle: Lock.bundle))
  ]
}
```

Vous pouvez aussi spécifier des icônes à partir d’autres bundles, comme dans l’exemple suivant :
CustomTextField(name: &quot;slack&#95;handle&quot;, placeholder: &quot;Identifiant Slack&quot;, icon: LazyImage(name: &quot;ic&#95;slack&quot;, bundle: Bundle(identifier: &quot;CustomBundle&quot;)))

<div id="enterprise-options">
  ## Options pour les connexions d’entreprise
</div>

Il existe aussi des options de configuration propres aux connexions d’entreprise :

<div id="enterpriseconnectionusingactiveauth">
  ### enterpriseConnectionUsingActiveAuth
</div>

Par défaut, les connexions d’entreprise utilisent l’authentification Web. Toutefois, vous pouvez préciser les connexions qui utiliseront plutôt l’authentification par identifiants et qui demanderont un nom d’utilisateur et un mot de passe.

```swift lines
.withOptions {
  $0.enterpriseConnectionUsingActiveAuth = ["enterprisedomain.com"]
}
```

<div id="activedirectoryemailasusername">
  ### activeDirectoryEmailAsUsername
</div>

En mode d’authentification par identifiants, l’utilisateur doit-il fournir son courriel comme identifiant ? Par défaut, la valeur est `false` et c’est plutôt un nom d’utilisateur qui est exigé.

```swift lines
.withOptions {
  $0.activeDirectoryEmailAsUsername = true
}
```

<div id="logging-options">
  ## Options de journalisation
</div>

Lock propose des options permettant d’activer ou de désactiver facilement la journalisation, ainsi que de régler d’autres paramètres connexes.

<div id="loglevel">
  ### logLevel
</div>

Par défaut, cette valeur est `.off`; les niveaux de journalisation Syslog sont pris en charge.

```swift lines
.withOptions {
  $0.logLevel = .all
}
```

<div id="loghttprequest">
  ### logHttpRequest
</div>

Indique s’il faut ou non consigner les requêtes de l’API Auth0.swift. Par défaut, cette valeur est `false`.

```swift lines
.withOptions {
  $0.logHttpRequest = true
}
```

<div id="loggeroutput">
  ### loggerOutput
</div>

Indiquez le gestionnaire de sortie du logger. Par défaut, l’instruction `print` est utilisée.

```swift lines
.withOptions {
  $0.loggerOutput = CleanroomLockLogger()
}
```

Dans le code ci-dessus, `loggerOutput` a été configuré pour utiliser [CleanroomLogger](https://github.com/emaloney/CleanroomLogger). En général, il suffit d’implémenter le protocole `loggerOutput` pour y parvenir. Vous pouvez bien sûr utiliser votre bibliothèque de journalisation préférée. Voici un exemple de gestion de la sortie du composant de journalisation avec CleanroomLogger.

```swift lines
class CleanroomLockLogger: LoggerOutput {
  func message(_ message: String, level: LoggerLevel, filename: String, line: Int) {
    let channel: LogChannel?
    switch level {
    case .debug:
        channel = Log.debug
    case .error:
        channel = Log.error
    case .info:
        channel = Log.info
    case .verbose:
        channel = Log.verbose
    case .warn:
        channel = Log.warning
    default:
        channel = nil
    }
    channel?.message(message, filePath: filename, fileLine: line)
  }
}
```
