title: Conseils avant le lancement
---

Voici une liste de conseils que nos clients ont jugés les plus utiles au moment de commencer à utiliser les services Auth0 :

* Vérifiez les URL dans vos callbacks autorisés, le CORS et les URL de redirection autorisées pour Logout afin de vous assurer qu’elles sont exactes, complètes et qu’elles n’utilisent ni `file:///` ni `localhost`. Limitez l’utilisation des wildcards au minimum et évaluez soigneusement leurs répercussions sur la sécurité.
* Séparez l’utilisation en Production du travail de développement en cours [en utilisant des tenants distincts pour prod, test et dev.](/docs/fr-CA/get-started/auth0-overview/create-tenants/set-up-multiple-environments)
* Configurez correctement votre réseau pour autoriser le trafic entrant et sortant vers nos ensembles d’API publiques (les environnements prod peuvent avoir une configuration différente de dev/staging).
* Vérifiez la date d’expiration des certificats que vous avez ajoutés à votre configuration afin de vous assurer que les certificats téléversés pendant les cycles de développement n’expireront pas pendant ou peu après le lancement.
* Assurez-vous que tous les <Tooltip tip="Fournisseur d’identité (IdP) : service qui stocke et gère les identités numériques." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=IDPs">IdP</Tooltip> distants fonctionnent avec NTP afin que l’heure soit correctement synchronisée.
* Le nombre de jours pendant lesquels les données de journal sont accessibles varie selon le forfait, jusqu’à un maximum de 30 jours. Si vous devez conserver des données de journal plus longtemps, vous devriez configurer l’une de nos [solutions de diffusion des journaux](https://marketplace.auth0.com/features/log-streaming). Vous pourrez ainsi conserver ces données plus longtemps. Assurez-vous de mettre cela en place avant la mise en production afin d’avoir les données de journal dont vous aurez besoin au moment voulu.
* Assurez-vous que votre utilisation de nos API demeure dans les [limites autorisées](/docs/fr-CA/troubleshoot/customer-support/operational-policies/rate-limit-policy) et que votre code est conçu pour s’adapter dynamiquement aux informations sur la limite de débit renvoyées dans l’en-tête, ainsi que pour gérer les erreurs. Si vous prévoyez que les limites de débit poseront problème pour votre application, parlez-nous-en à l’avance pour vérifier s’il serait possible de les augmenter temporairement jusqu’à ce que le trafic revienne à la normale. Vous devriez envisager d’utiliser un cache de données utilisateur afin de ne pas avoir à interroger un point de terminaison d’API plus souvent que nécessaire.
* Si vous utilisez des [connexions sociales](/docs/fr-CA/authenticate/identity-providers), assurez-vous d’obtenir vos propres informations d’identification auprès du fournisseur et de les ajouter à la configuration de la connexion sociale.
* Si vous utilisez des [connexions de BD personnalisées](/docs/fr-CA/authenticate/database-connections/custom-db), assurez-vous que tous les scripts de BD personnalisés sont implémentés et renvoient un profil utilisateur cohérent avec un ID utilisateur unique.
* Pour l’envoi de courriels, assurez-vous d’abord de [configurer un fournisseur de courriel personnalisé.](/docs/fr-CA/customize/email/smtp-email-providers)
* Si vous utilisez notre CDN pour le widget Lock, assurez-vous de verrouiller une version précise.
* Assurez-vous que les composants **externes** appelés depuis Actions, Rules, Hooks et les scripts de connexion de base de données personnalisée peuvent gérer la charge prévue.
* Protégez adéquatement toute valeur de <Tooltip tip="Client Secret : Secret utilisé par un client (application) pour s'authentifier auprès de l'Authorization Server ; il doit être connu uniquement du client et de l'Authorization Server et doit être suffisamment aléatoire pour ne pas pouvoir être deviné." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=client+secret">client secret</Tooltip>.
* Vérifiez les [types d’autorisation](/docs/fr-CA/get-started/applications/application-grant-types) activés pour vos applications. Assurez-vous que les bons sont activés et, surtout, désactivez tout type d’autorisation dont vous n’avez pas besoin.
* Si vous utilisez user&#95;metadata, assurez-vous qu’il s’agit de données que les utilisateurs devraient pouvoir modifier eux-mêmes (p. ex., pas le « statut de paiement »).
* Passez en revue vos [paramètres de Protection contre les attaques](https://manage.auth0.com/#/security/attack-protection) et lisez [Protection contre les attaques](/docs/fr-CA/secure/attack-protection) pour comprendre comment débloquer les utilisateurs qui ont été bloqués.
* Vérifiez la [section Admin des paramètres du tenant](https://manage.auth0.com/#/tenant/admins) pour vous assurer que seuls les administrateurs autorisés ont accès au <Tooltip tip="Auth0 Dashboard : le principal produit d’Auth0 pour configurer vos services." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip>. Consultez [Manage Dashboard Access](/docs/fr-CA/get-started/manage-dashboard-access) pour en savoir plus.
* Assurez-vous d’avoir testé tous les cas d’utilisation essentiels de votre application sur tous les appareils susceptibles d’être utilisés par ses utilisateurs finaux. Veillez à tester la connexion, <Tooltip tip="Authentification unique (SSO) : service qui, après qu'un utilisateur se soit connecté à une application, connecte automatiquement cet utilisateur aux autres applications." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=single+sign-on">l’authentification unique</Tooltip> (si elle est prise en charge) et la journalisation, ainsi que le comportement de votre application lorsqu’un utilisateur l’exécute dans plusieurs onglets de navigateur.
* Passez en revue votre [liste de Rules](https://manage.auth0.com/#/rules) et assurez-vous que seules les Rules appropriées sont activées.
* Passez en revue le code de vos Rules, de vos custom DB scripts et de tout code personnalisé dans la page de connexion afin de vous assurer que chaque appel intercepte et traite adéquatement les erreurs. Vérifiez également que les instructions return/callback sont appelées correctement.
* Configurez le nom de votre application, l’URL de soutien et le courriel de soutien dans la section [Tenant Settings General](https://manage.auth0.com/#/tenant) afin que, lorsqu’une erreur se produit, vos utilisateurs finaux soient dirigés vers une page appropriée.
* Assurez-vous que votre application [obtient dynamiquement un jeton pour la Management API](/docs/fr-CA/secure/tokens/access-tokens/management-api-access-tokens).
* Supprimez toute instruction `console.log` de vos Rules ou custom DB scripts, en particulier celles qui pourraient divulguer des renseignements permettant d’identifier un utilisateur, comme l’adresse courriel, le nom d’utilisateur ou le mot de passe.
* N’utilisez pas de secrets en clair dans les Rules ou les connexions de base de données. Ils doivent être ajoutés dans la partie configuration de l’interface. La configuration est chiffrée et fournie Just In Time. Ne consignez pas l’objet de configuration dans le journal.
