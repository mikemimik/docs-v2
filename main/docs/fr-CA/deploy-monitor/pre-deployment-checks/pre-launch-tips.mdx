description: Une liste de conseils utiles pour commencer avec les services Auth0, tirés des commentaires et de l’expérience d’autres utilisateurs
title: Conseils avant le lancement
---

Voici une liste de conseils que nos clients ont trouvés les plus utiles lorsqu’ils font leurs premiers pas avec les services Auth0 :

* Vérifiez les URL de rappel autorisées, les paramètres CORS et les URL de redirection autorisées pour Logout afin de vous assurer qu’ils sont corrects, complets et qu’ils n’utilisent ni `file:///` ni `localhost`. Limitez au minimum l’utilisation de caractères génériques et évaluez attentivement les risques de sécurité qui y sont liés.
* Isolez l’utilisation en production du travail de développement en cours [en utilisant des tenants distincts pour la prod, le test et le dev.](/docs/fr-CA/get-started/auth0-overview/create-tenants/set-up-multiple-environments)
* Configurez correctement votre réseau pour autoriser le trafic vers et depuis nos ensembles d’API publiques (les environnements de prod peuvent avoir une configuration différente de ceux de dev/staging).
* Vérifiez la date d’expiration des certificats que vous avez ajoutés à votre configuration afin de vous assurer que les certificats téléversés pendant les cycles de développement n’expireront pas pendant la mise en production ou peu après.
* Assurez-vous que tous les <Tooltip tip="Fournisseur d’identité (IdP) : service qui stocke et gère les identités numériques." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=IDPs">IdPs</Tooltip> distants utilisent le protocole NTP pour que l’heure soit correctement synchronisée.
* Le nombre de jours pendant lesquels les données de journalisation sont disponibles varie selon le forfait, jusqu’à un maximum de 30 jours. Si vous devez conserver les données de journalisation plus longtemps, vous devriez configurer l’une de nos [solutions de diffusion des journaux](https://marketplace.auth0.com/features/log-streaming). Cela vous permettra de conserver les données de journalisation pendant de plus longues périodes. Assurez-vous de le faire avant la mise en production afin d’avoir accès aux données de journalisation lorsque vous en aurez besoin.
* Assurez-vous que votre utilisation de nos API demeure dans les [limites autorisées](/docs/fr-CA/troubleshoot/customer-support/operational-policies/rate-limit-policy) et que votre code est conçu pour s’ajuster dynamiquement aux informations de limite de débit renvoyées dans l’en-tête et pour gérer les erreurs. Si vous prévoyez que les limites de débit poseront un problème pour votre application, discutez-en avec nous à l’avance pour vérifier s’il serait possible de les augmenter temporairement jusqu’à ce que le trafic revienne à la normale. Vous devriez envisager d’utiliser un cache des données utilisateur afin de ne pas avoir à interroger un point de terminaison d’API plus souvent que nécessaire.
* Si vous utilisez des [connexions sociales](/docs/fr-CA/authenticate/identity-providers), assurez-vous d’obtenir vos propres informations d’identification auprès du fournisseur et de les ajouter à la configuration de la connexion sociale.
* Si vous utilisez des [connexions de BD personnalisées](/docs/fr-CA/authenticate/database-connections/custom-db), assurez-vous que tous les scripts de BD personnalisés sont mis en place et renvoient un profil utilisateur cohérent avec un ID utilisateur unique.
* Pour l’envoi de courriels, assurez-vous d’abord de [configurer un fournisseur de courriel personnalisé.](/docs/fr-CA/customize/email/smtp-email-providers)
* Si vous utilisez notre CDN pour le widget Lock, assurez-vous de vous en tenir à une version précise.
* Assurez-vous que les composants **externes** appelés depuis Actions, Rules, Hooks et les scripts de connexion à une base de données personnalisée peuvent supporter la charge prévue.
* Protégez correctement toute valeur de <Tooltip tip="Secret client : secret qu’un client (application) utilise pour s’authentifier auprès du serveur d’autorisation; il ne devrait être connu que du client et du serveur d’autorisation, et doit être suffisamment aléatoire pour qu’on ne puisse pas le deviner." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=client+secret">secret client</Tooltip>.
* Vérifiez les [types d’autorisation](/docs/fr-CA/get-started/applications/application-grant-types) de vos applications. Assurez-vous que les bons sont activés et, surtout, désactivez tous les types d’autorisation dont vous n’avez pas besoin.
* Si vous utilisez user&#95;metadata, assurez-vous qu’il s’agit bien de données que les utilisateurs devraient pouvoir modifier eux-mêmes (par ex., pas le « statut de paiement »).
* Passez en revue vos [paramètres de Protection contre les attaques](https://manage.auth0.com/#/security/attack-protection) et consultez [Protection contre les attaques](/docs/fr-CA/secure/attack-protection) pour comprendre comment débloquer les utilisateurs qui ont été bloqués.
* Passez en revue votre [section Admin des Tenant Settings](https://manage.auth0.com/#/tenant/admins) pour vous assurer que seuls les administrateurs autorisés ont accès au <Tooltip tip="Auth0 Dashboard : le principal produit d’Auth0 pour configurer vos services." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip>. Consultez [Gérer l’accès au Dashboard](/docs/fr-CA/get-started/manage-dashboard-access) pour en savoir plus.
* Assurez-vous d’avoir testé tous les cas d’utilisation essentiels de votre application sur tous les appareils susceptibles d’être utilisés par les utilisateurs finaux de votre application. N’oubliez pas de tester la connexion, <Tooltip tip="Authentification unique (SSO) : service qui, après qu’un utilisateur se connecte à une application, le connecte automatiquement à d’autres applications." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=single+sign-on">l’authentification unique</Tooltip> (si prise en charge), ainsi que la journalisation, de même que ce qui se passe si un utilisateur exécute votre application dans plusieurs onglets de navigateur.
* Passez en revue votre [liste de Rules](https://manage.auth0.com/#/rules) et assurez-vous que seules les Rules appropriées sont activées.
* Passez en revue le code de votre Rule, tous les custom DB scripts et tout code personnalisé dans la page de connexion afin de vous assurer que chaque appel comprend une gestion des erreurs adéquate. Vérifiez également que les instructions `return` et les callback sont appelés correctement.
* Configurez le nom de votre application, l’URL de soutien et le courriel de soutien dans la section [Tenant Settings General](https://manage.auth0.com/#/tenant) afin que, lorsqu’une erreur survient, vos utilisateurs finaux soient redirigés vers la page appropriée.
* Assurez-vous que votre application [obtient dynamiquement un jeton pour l’API de gestion](/docs/fr-CA/secure/tokens/access-tokens/management-api-access-tokens).
* Supprimez toute instruction `console.log` de vos Rules ou custom DB scripts, en particulier celles qui pourraient divulguer des renseignements permettant d’identifier un utilisateur, comme l’adresse courriel, le nom d’utilisateur ou le mot de passe.
* N’utilisez pas de secrets en texte clair dans les Rules ou les connexions de base de données. Ils doivent être ajoutés dans la section de configuration de l’interface. La configuration est chiffrée et fournie au moment opportun. N’inscrivez pas l’objet de configuration dans le journal.
