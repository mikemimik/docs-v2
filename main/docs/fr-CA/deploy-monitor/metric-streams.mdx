<div id="metric-streams-beta">
  # Metric Streams (bêta)
</div>

<Warning>
  Metric Streams est actuellement offert en bêta. Pour utiliser cette fonctionnalité, vous devez avoir un plan Enterprise. En utilisant cette fonctionnalité, vous acceptez les conditions d’essai gratuit applicables énoncées dans le [Master Subscription Agreement](https://www.okta.com/legal) d’Okta. Pour en savoir plus sur le cycle de publication des produits d’Auth0, consultez [Product Release Stages](https://auth0.com/docs/troubleshoot/product-lifecycle/product-release-stages).
</Warning>

Transmettez en continu les métriques Auth0 en temps réel à votre plateforme de surveillance afin de suivre les performances de l’API, de repérer les problèmes de limite de débit et de résoudre les erreurs plus rapidement. Metric Streams prend en charge à la fois l’intégration native à Datadog et le protocole OpenTelemetry (OTLP) pour des plateformes comme New Relic et Splunk.

<div id="what-you-can-monitor">
  ## Ce que vous pouvez surveiller
</div>

Suivez des métriques clés sur votre utilisation de l’API Auth0 :

* **Volume de requêtes API** : Surveillez le volume total de requêtes aux API Authentication et Management.
* **Taux d’erreur** : Repérez les erreurs côté client et côté serveur qui affectent votre intégration.
* **Atteintes des limites de débit** : Détectez quand votre application atteint les limites de débit.
* **Répartition géographique** : Analysez les tendances du trafic par pays.
* **Flux d’authentification** : Suivez les types d’autorisation et l’utilisation des connexions.

<div id="use-cases">
  ## Cas d’utilisation
</div>

<div id="identify-rate-limit-issues">
  ### Repérer les problèmes liés aux limites de débit
</div>

Surveillez les API et les points de terminaison qui atteignent leurs limites de débit afin d’ajuster votre intégration ou de passer à un niveau supérieur de limite de débit.

<div id="troubleshoot-api-failures">
  ### Résoudre les problèmes d’API
</div>

Repérez rapidement la cause première des erreurs en filtrant les métriques par API, opération et type d’échec (limites de débit, détection d’anomalies ou erreurs générales).

<div id="plan-capacity">
  ### Planification de la capacité
</div>

Analysez l’utilisation de pointe, le nombre moyen de requêtes par seconde (RPS) et les métriques de percentile afin de déterminer le palier de limite de débit approprié pour votre charge de travail.

## Prérequis

Avant de configurer un flux de métriques, vous avez besoin de :

* **accès à l’Auth0 Management API** : créez une application machine-à-machine (M2M) avec les autorisations suivantes :

  * `read:event_streams`
  * `create:event_streams`
  * `update:event_streams`
  * `delete:event_streams`
  * `read:event_deliveries`
  * `update:event_deliveries`

  Pour obtenir des instructions détaillées, consultez [Obtenir des jetons d’accès à l’API de gestion](docs/secure/tokens/access-tokens/management-api-access-tokens/get-management-api-access-tokens-for-production).

* **Auth0 CLI** : installez [Auth0 CLI](https://auth0.github.io/auth0-cli/) pour configurer des flux de métriques à l’aide de commandes d’API.

* **Compte sur une plateforme de surveillance** : un compte actif auprès de l’une des [destinations prises en charge](#supported-destinations).

<div id="metric-reference">
  ## Référence des métriques
</div>

<div id="auth0api_requestcount">
  ### auth0.api_request.count
</div>

Compte les requêtes API envoyées à votre tenant Auth0, avec des balises détaillées de filtrage et d’agrégation. Une nouvelle métrique est émise toutes les 10 secondes et offre une granularité à la seconde.

<div id="available-tags">
  #### Balises disponibles
</div>

| Nom de la balise   | Obligatoire | Description                                                                                                                   |
| ------------------ | ----------- | ----------------------------------------------------------------------------------------------------------------------------- |
| `api`              | Oui         | Type d’API : `AUTHENTICATION` ou `MANAGEMENT`                                                                                 |
| `operation_name`   | Oui         | Méthode HTTP et chemin (p. ex., `POST /oauth/token`)                                                                          |
| `operation_result` | Oui         | Résultat : `SUCCESS` ou `FAILURE`                                                                                             |
| `tenant`           | Oui         | Nom de votre tenant Auth0                                                                                                     |
| `failure_code`     | Non         | Motif de l’échec : `RATE_LIMIT`, `ANOMALY_DETECTION` ou `CLIENT_SERVER_ERROR`                                                 |
| `country_code`     | Non         | Code de pays ISO provenant de l’adresse IP de la requête                                                                      |
| `client_id`        | Non         | ID client de l’application (Authentication API seulement)                                                                     |
| `connection_id`    | Non         | ID de la connexion (Authentication API seulement)                                                                             |
| `grant_type`       | Non         | Type d’octroi OAuth 2.0 (Authentication API seulement)                                                                        |
| `batch_seq`        | Oui         | Numéro de séquence pour les métriques ayant des balises et un horodatage identiques (temporaire; sera supprimé après la bêta) |

<Warning>
  Les balises `client_id` et `connection_id` peuvent générer une cardinalité élevée dans les environnements comportant un grand nombre de clients ou de connexions uniques, ce qui peut augmenter les coûts de surveillance. Utilisez ces balises avec discernement.
</Warning>

<Note>
  Les flux de métriques n’incluent pas les appels d’API mis en cache ou bloqués à la périphérie par le pare-feu d’applications Web (WAF) d’Auth0 afin d’éviter une surcharge du système.
</Note>

<div id="enum-values">
  #### Valeurs d’énumération
</div>

**api**

* `AUTHENTICATION` - Requêtes de l’Authentication API
* `MANAGEMENT` - Requêtes de la Management API

**operation&#95;result**

* `SUCCESS` - Requête traitée avec succès
* `FAILURE` - Échec de la requête en raison d’une erreur client ou serveur

**failure&#95;code** (présent uniquement lorsque `operation_result` est `FAILURE`)

* `RATE_LIMIT` - La requête a dépassé la limite de débit
* `ANOMALY_DETECTION` - Bloquée par la protection contre les attaques d’Auth0
* `CLIENT_SERVER_ERROR` - Erreur HTTP 4xx ou 5xx

<div id="set-up-datadog">
  ## Configurer Datadog
</div>

Transmettez directement les métriques Auth0 à Datadog via l’API de métriques native de Datadog.

<div id="prerequisites">
  ### Prérequis
</div>

* [Compte Datadog](https://www.datadoghq.com/) actif
* Clé API Datadog (générez-la dans **Paramètres de l’organisation &gt; Clés API** dans Datadog)
* Votre site Datadog (identifiant de région, comme `US1` pour app.datadoghq.com, `EU` pour app.datadoghq.eu, `US3` pour us3.datadoghq.com)

<div id="create-metric-stream">
  ### Créer un flux de métriques
</div>

<Tabs>
  <Tab title="Dashboard">
    **Accédez à Metric Streams :**

    1. Dans [Auth0 Dashboard](https://manage.auth0.com), allez à **Monitoring** &gt; **Metric Streams**.
    2. Sélectionnez **+ Create Metric Stream**.
    3. Sélectionnez **Datadog**.

    **Configurer le flux :**

    1. **Name** : Entrez un nom descriptif (par exemple, `datadog-production`).
    2. **Datadog Site** : Sélectionnez votre région Datadog dans la liste déroulante :
       * `US1` pour app.datadoghq.com
       * `US3` pour us3.datadoghq.com
       * `US5` pour us5.datadoghq.com
       * `EU` pour app.datadoghq.eu
       * `AP1` pour ap1.datadoghq.com
    3. **Datadog API Key** : Collez votre clé API Datadog.
    4. Sélectionnez **Save**.

    Cette action crée votre flux de métriques et l’active automatiquement.
  </Tab>

  <Tab title="CLI">
    Utilisez Auth0 CLI pour créer un flux de métriques Datadog :

    ```bash
    auth0 api post metric-streams \
      --data '{
        "name": "datadog-production",
        "subscriptions": [{ "metric_type": "auth0.api_request.count" }],
        "destination": {
          "type": "datadog",
          "configuration": {
            "api_key": "YOUR_DATADOG_API_KEY",
            "site": "app.datadoghq.com"
          }
        }
      }'
    ```

    **Réponse :**

    ```json
    {
      "id": "mst_abc123xyz",
      "status": "enabled",
      "name": "datadog-production",
      "created_at": "2024-02-12T10:30:00Z",
      "updated_at": "2024-02-12T10:30:00Z",
      "destination": {
        "type": "datadog",
        "configuration": {
          "api_key": "***",
          "site": "app.datadoghq.com"
        }
      }
    }
    ```

    Enregistrez la valeur `id` pour pouvoir gérer ce flux plus tard.
  </Tab>
</Tabs>

<div id="verify-in-datadog">
  ### Vérifier dans Datadog
</div>

En quelques minutes, les métriques d’Auth0 s’affichent dans Datadog. Accédez à **Metrics &gt; Explorer** et recherchez `auth0.api_request.count` pour consulter vos données.

<div id="manage-streams">
  ### Gérer les flux
</div>

<Tabs>
  <Tab title="Dashboard">
    **Afficher tous les flux :**

    1. Accédez à [Auth0 Dashboard](https://manage.auth0.com) &gt; **Monitoring** &gt; **Metric Streams**.
    2. Tous les flux configurés sont affichés avec leur Destination, Status, Name et Subscriptions.

    **Modifier un flux :**

    1. Sélectionnez le nom du flux dans la liste.
    2. Mettez la configuration à jour.
    3. Sélectionnez **Save**.

    **Activer/désactiver un flux :**

    1. Utilisez la **bascule** dans la colonne Status pour activer ou désactiver le flux.
    2. Les flux désactivés cessent d’envoyer des métriques, mais conservent leur configuration.

    **Supprimer un flux :**

    1. Sélectionnez le nom du flux dans la liste.
    2. Sélectionnez **Delete** au bas de la page de configuration.
    3. Confirmez la suppression.
  </Tab>

  <Tab title="CLI">
    **Lister tous les flux :**

    ```bash
    auth0 api get /api/v2/metric-streams
    ```

    **Désactiver un flux :**

    ```bash
    auth0 api patch /api/v2/metric-streams/YOUR_STREAM_ID \
      --data '{"status":"disabled"}'
    ```

    **Réactiver un flux :**

    ```bash
    auth0 api patch /api/v2/metric-streams/YOUR_STREAM_ID \
      --data '{"status":"enabled"}'
    ```
  </Tab>
</Tabs>

<div id="set-up-new-relic">
  ## Configurer New Relic
</div>

Acheminez les métriques Auth0 vers New Relic à l’aide du protocole OpenTelemetry (OTLP).

<div id="get-new-relic-credentials">
  ### Obtenir les identifiants New Relic
</div>

<Steps>
  <Step title="Trouver votre point de terminaison OTLP">
    Consultez la [documentation OTLP de New Relic](https://docs.newrelic.com/docs/opentelemetry/best-practices/opentelemetry-otlp/#configure-endpoint-port-protocol) pour trouver le point de terminaison correspondant à votre région :

    * **É.-U. :** `https://otlp.nr-data.net`
    * **UE :** `https://otlp.eu01.nr-data.net`

    Enregistrez l’URL de ce point de terminaison.
  </Step>

  <Step title="Créer une clé de licence d’ingestion">
    1. Accédez à **&#91;votre menu utilisateur&#93;** &gt; **Clés API**.
    2. Sélectionnez **Créer une clé**.
    3. Sélectionnez **Type de clé** = **Ingestion - Licence**.
    4. Saisissez un nom descriptif et, au besoin, une note.
    5. Sélectionnez **Créer une clé**.
    6. Copiez immédiatement la clé API. Vous en aurez besoin pour les étapes suivantes, car elle ne sera plus affichée par la suite.
  </Step>
</Steps>

<div id="create-metric-stream">
  ### Créer un flux de métriques
</div>

<Tabs>
  <Tab title="Dashboard">
    **Accédez à Metric Streams :**

    1. Accédez à [Auth0 Dashboard](https://manage.auth0.com) &gt; **Monitoring** &gt; **Metric Streams**.
    2. Sélectionnez **+ Create Metric Stream**.
    3. Sélectionnez **OpenTelemetry**.

    **Configurer le flux :**

    1. **Name**: Entrez un nom descriptif (p. ex., `newrelic-production`).
    2. **Protocol**: Sélectionnez **HTTP** dans la liste déroulante.
    3. **OTLP Endpoint**: Collez le point de terminaison OTLP de New Relic (p. ex., `https://otlp.nr-data.net`).
    4. **Authentication Method**: Sélectionnez **API Key** dans la liste déroulante.
    5. **Header Name**: Entrez `api-key`.
    6. **API Key**: Collez votre clé de licence d’ingestion New Relic copiée aux étapes précédentes.
    7. Sélectionnez **Save**.

    Cette action crée votre flux de métriques et l’active automatiquement.
  </Tab>

  <Tab title="CLI">
    ```bash
    auth0 api post metric-streams \
      --data '{
        "name": "newrelic-production",
        "subscriptions": [{ "metric_type": "metrics_bridge.api_request" }],
        "destination": {
          "type": "otlp",
          "configuration": {
            "endpoint": "YOUR_NEW_RELIC_ENDPOINT",
            "protocol": "http",
            "auth_type": "api_key",
            "api_key": "YOUR_INGEST_LICENSE_KEY",
            "auth_header_name": "api-key"
          }
        }
      }'
    ```
  </Tab>
</Tabs>

<div id="set-up-splunk">
  ## Configurer Splunk
</div>

Transmettez les métriques Auth0 à Splunk Observability Cloud au moyen du protocole OpenTelemetry (OTLP).

<div id="get-splunk-credentials">
  ### Obtenir les identifiants Splunk
</div>

<Steps>
  <Step title="Trouvez votre point de terminaison OTLP">
    Consultez la [documentation de l’exportateur HTTP OTLP de Splunk](https://help.splunk.com/en/splunk-observability-cloud/manage-data/splunk-distribution-of-the-opentelemetry-collector/get-started-with-the-splunk-distribution-of-the-opentelemetry-collector/collector-components/exporters/otlphttp-exporter) pour trouver votre point de terminaison.

    Le format du point de terminaison est :

    ```
    https://ingest.<REALM>.signalfx.com/v2/datapoint/otlp
    ```

    Remplacez `<REALM>` par votre realm Splunk (par ex. : `us1`, `eu0`, `jp0`).
  </Step>

  <Step title="Créer un jeton d’accès">
    1. Accédez à **Settings** &gt; **Access Tokens** &gt; **New Token**.
    2. Entrez un nom descriptif.
    3. Réglez **Authorization Scope** à **INGEST** (obligatoire).
    4. Définissez l’expiration du jeton.
    5. Sélectionnez **Create**.
    6. Copiez la valeur du jeton et conservez-la en lieu sûr. Vous en aurez besoin pour les étapes suivantes.
  </Step>
</Steps>

<div id="create-metric-stream">
  ### Créer un flux de métriques
</div>

<Tabs>
  <Tab title="Dashboard">
    **Accédez à Metric Streams :**

    1. Accédez à [Auth0 Dashboard](https://manage.auth0.com) &gt; **Monitoring** &gt; **Metric Streams**.
    2. Sélectionnez **+ Create Metric Stream**.
    3. Sélectionnez **OpenTelemetry**.

    **Configurer le flux :**

    1. **Name** : Entrez un nom descriptif (p. ex., `splunk-production`).
    2. **Protocol** : Sélectionnez **HTTP** dans la liste déroulante.
    3. **OTLP Endpoint** : Collez votre point de terminaison OTLP Splunk (p. ex., `https://ingest.us1.signalfx.com/v2/datapoint/otlp`).
    4. **Authentication Method** : Sélectionnez **API Key** dans la liste déroulante.
    5. **Header Name** : Entrez `X-SF-TOKEN`.
    6. **API Key** : Collez votre jeton d’accès Splunk (de l’étape 2 ci-dessus).
    7. Sélectionnez **Save**.

    Le flux de métriques est créé et activé automatiquement.
  </Tab>

  <Tab title="CLI">
    ```bash
    auth0 api post metric-streams \
      --data '{
        "name": "splunk-production",
        "subscriptions": [{ "metric_type": "metrics_bridge.api_request" }],
        "destination": {
          "type": "otlp",
          "configuration": {
            "endpoint": "YOUR_SPLUNK_ENDPOINT",
            "protocol": "http",
            "auth_type": "api_key",
            "api_key": "YOUR_SPLUNK_ACCESS_TOKEN",
            "auth_header_name": "X-SF-TOKEN"
          }
        }
      }'
    ```
  </Tab>
</Tabs>

<div id="understand-metric-cardinality">
  ## Comprendre la cardinalité des métriques
</div>

La cardinalité correspond au nombre de séries chronologiques uniques générées par vos métriques. Plus la cardinalité est élevée, plus les coûts de monitoring augmentent.

| Balise             | Cardinalité      | Impact                                           |
| ------------------ | ---------------- | ------------------------------------------------ |
| `api`              | Faible           | 2 valeurs uniques (Authentification, Management) |
| `operation_name`   | Moyenne          | Varie selon les opérations d’API utilisées       |
| `operation_result` | Faible           | 2 valeurs uniques (Succès, Échec)                |
| `tenant`           | Moyenne          | 1 par tenant Auth0                               |
| `failure_code`     | Faible           | 3 valeurs uniques (uniquement en cas d’échec)    |
| `country_code`     | Moyenne à élevée | Varie selon la répartition géographique          |
| `client_id`        | **Élevée**       | 1 par application cliente                        |
| `connection_id`    | **Élevée**       | 1 par connexion de fournisseur d’identité        |
| `grant_type`       | Faible           | Nombre limité de types d’octroi OAuth 2.0        |
| `batch_seq`        | Faible           | Balise temporaire (retirée après la bêta)        |

<Warning>
  Utilisez les balises `client_id` et `connection_id` avec prudence. Dans les environnements qui comptent de nombreuses applications ou connexions, ces balises peuvent faire augmenter considérablement les coûts. Envisagez de filtrer des clients précis ou d’agréger ces balises si vous n’avez pas besoin d’un niveau de granularité aussi détaillé.
</Warning>

<div id="troubleshooting">
  ## Dépannage
</div>

<div id="metrics-not-appearing">
  ### Les métriques n’apparaissent pas
</div>

**Vérifiez le statut du flux :**

<Tabs>
  <Tab title="Dashboard">
    1. Accédez à [Auth0 Dashboard](https://manage.auth0.com) &gt; **Monitoring** &gt; **Metric Streams**.
    2. Repérez votre flux de métriques dans la liste.
    3. Vérifiez que la colonne **Status** affiche **Activé**.
    4. Sélectionnez le nom du flux pour afficher la configuration détaillée et vérifier les paramètres.
  </Tab>

  <Tab title="CLI">
    ```bash
    auth0 api get /api/v2/metric-streams/YOUR_STREAM_ID
    ```

    Vérifiez que le champ `status` est `enabled`.
  </Tab>
</Tabs>

**Vérifiez les informations d’authentification de la destination :**

* **Datadog**: Assurez-vous que votre clé API est valide et que l’URL du site correspond à votre région Datadog
* **Destinations OTLP**: Vérifiez que l’URL du point de terminaison, le jeton d’authentification et le nom de l’en-tête sont corrects

**Vérifiez la mise en cache à la périphérie :**

N’oubliez pas que les requêtes mises en cache et les requêtes bloquées par le WAF ne sont pas incluses dans les flux de métriques.

<div id="high-monitoring-costs">
  ### Coûts élevés liés à la surveillance
</div>

Si vous constatez des coûts anormalement élevés :

1. Vérifiez quelles balises vous utilisez dans les requêtes et les tableaux de bord.
2. Envisagez de retirer `client_id` et `connection_id` des agrégations.
3. Utilisez l’échantillonnage ou le filtrage dans votre plateforme de surveillance pour réduire le volume de métriques ingérées.
4. Regroupez les données selon des balises de niveau supérieur comme `api` et `operation_result` plutôt que selon des balises trop granulaires.
