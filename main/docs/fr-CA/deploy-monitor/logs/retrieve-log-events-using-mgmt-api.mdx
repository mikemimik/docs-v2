---

Vous pouvez utiliser la <Tooltip tip="Management API : un produit qui permet aux clients d’effectuer des tâches administratives." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Management+API">Management API</Tooltip> pour récupérer jusqu’à 100 journaux par requête à l’aide de l’endpoint [/get&#95;logs](https://auth0.com/docs/api/management/v2/#!/Logs/get_logs), qui prend en charge deux types d’utilisation :

* **Par point de contrôle** : Recommandé si vous souhaitez exporter des journaux vers un service externe d’analyse de données.
* **Par critères de recherche** : Utilisé par le Dashboard.

Les journaux stockés par Auth0 sont assujettis à une période de conservation en fonction de votre Auth0 subscription. Pour en savoir plus, consultez [Log Data Retention](/docs/fr-CA/deploy-monitor/logs/log-data-retention).

Vous pouvez aussi utiliser ces endpoints pour rechercher des journaux :

* `/api/v2/logs/{id}`: Récupère la log entry associée à l’ID de log fourni.
* `/api/v2/users/{user_id}/logs`: Récupère les journaux associés à un user ID précis.

<div id="retrieve-logs-by-checkpoint">
  ## Récupérer les journaux par point de contrôle
</div>

Cette méthode vous permet de récupérer les journaux plus récents qu’un `log_id` donné. Elle
est couramment utilisée pour envoyer vos journaux à un système externe, comme Splunk, Datadog,
Application Insights, Mixpanel, et plus encore.

Pour effectuer une recherche par point de contrôle, utilisez les paramètres suivants :

| Paramètre | Description                                                                                                                                  |
| --------- | -------------------------------------------------------------------------------------------------------------------------------------------- |
| `from`    | ID de l’événement de log à partir duquel commencer à récupérer les journaux. Vous pouvez limiter le nombre de journaux à l’aide du paramètre `take`. |
| `take`    | Le nombre total d’entrées à récupérer avec le paramètre `from`.                                                                              |

Tout paramètre autre que `from` et `take` est ignoré. Les journaux sont triés selon le `log_id`.

<div id="semantics">
  #### Sémantique
</div>

La récupération des journaux à partir d’un point de contrôle a la sémantique suivante :

* Tous les journaux accessibles selon les autorisations associées à votre rôle seront renvoyés
* Les journaux sont triés selon le `log_id`, plutôt qu’en fonction de leur heure de génération, afin d’éviter de sauter des enregistrements en raison des délais du système Auth0
* La pagination n’est pas limitée, ce qui signifie qu’il n’y a aucune limite, contrairement à la limite de `1,000` enregistrements lors de la récupération des journaux par critères de recherche

<div id="pagination">
  #### Pagination
</div>

Lors de la récupération des journaux à l’aide d’un point de contrôle, seule la première requête doit être effectuée
manuellement. Chaque requête subséquente doit être effectuée à l’aide de l’URL de relation `next`
dans l’en-tête `Link`. Cela simplifie l’intégration avec l’API
et permet à Auth0 d’effectuer des optimisations sans nuire à l’expérience
client.

<div id="example">
  #### Exemple
</div>

Supposons que le nom de votre tenant dans `us` est `acme`. Pour commencer à récupérer des enregistrements par
point de contrôle :

1. Récupérez le `log_id` à partir de n’importe quel log de votre système.
   Exemple : `90020211201185132572079310688835925971249535794074878050`.
2. Faites une requête à `https://TENANT_NAME.LOCALITY.auth0.com/api/v2/logs?from=LOG_ID&take=100` pour demander les `100` journaux suivants, plus récents que le `LOG_ID` fourni.
3. La réponse HTTP contiendra

   1. Un tableau JSON de `0` à `100` journaux plus récents que le `LOG_ID` fourni
   2. Un en-tête HTTP `Link` avec une relation `next`
4. Si l’ensemble de résultats est vide, vous recevrez un tableau vide, et la relation `next` correspondra à la même URL que celle utilisée dans la requête la plus récente

<div id="retrieve-logs-by-search-criteria">
  ## Récupérer des journaux selon des critères de recherche
</div>

Cette méthode vous permet de récupérer les journaux qui correspondent aux critères de recherche spécifiés ou, si aucun critère n’est utilisé, de répertorier tous les journaux. Pour effectuer une recherche selon des critères, utilisez les paramètres suivants :

| Paramètre        | Description                                                                                                                                                              |
| ---------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `q`              | Critères de recherche utilisant la syntaxe de chaîne de requête.                                                                                                         |
| `page`           | Le numéro de page commençant à zéro.                                                                                                                                     |
| `per_page`       | Le nombre d’entrées par page.                                                                                                                                            |
| `sort`           | Le champ à utiliser pour le tri. Utilisez `field:order`, où `order` vaut `1` pour l’ordre croissant et `-1` pour l’ordre décroissant. Par exemple, `date:-1`.            |
| `fields`         | Une liste de champs séparés par des virgules à inclure ou à exclure (selon `include_fields`) dans le résultat. Laissez ce paramètre vide pour récupérer tous les champs. |
| `include_fields` | `true` si les champs spécifiés doivent être inclus dans le résultat, sinon `false`. La valeur par défaut est `true`.                                                     |

Pour voir la liste des champs et des paramètres de tri pouvant être utilisés dans la requête de recherche, consultez [Log Search Query Syntax](/docs/fr-CA/deploy-monitor/logs/log-search-query-syntax).

<div id="limitations">
  ## Limitations
</div>

Outre la limite de 100 journaux par requête pour récupérer les journaux, vous ne pouvez parcourir qu’un maximum de 1 000 résultats de recherche par pagination.

Si vous obtenez l’erreur `414 Request-URI Too Large`, cela signifie que votre chaîne de requête dépasse la longueur prise en charge. Dans ce cas, affinez votre recherche.

<Card title="Limites des champs de recherche de Private Cloud">
  Les champs suivants sont interrogeables uniquement dans Private Cloud :

  * `user`
  * `connection`
  * `application`
  * `type`
  * `ip`

  Utilisez des guillemets doubles pour les recherches exactes (par ex., `application:"test"` recherchera tous les événements de journalisation propres à l’application nommée `test`, tandis que `application:test` recherchera les événements de journalisation des applications dont le nom contient « test ».
</Card>
