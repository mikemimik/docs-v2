Vous pouvez utiliser la <Tooltip tip="Management API : un produit qui permet aux clients d’effectuer des tâches administratives." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Management+API">Management API</Tooltip> pour récupérer jusqu’à 100 journaux par requête à l’aide du point de terminaison [/get&#95;logs](https://auth0.com/docs/api/management/v2/#!/Logs/get_logs), qui prend en charge deux modes d’utilisation :

* **Par point de contrôle** : recommandé si vous souhaitez exporter des journaux vers un service externe d’analyse de données.
* **Selon des critères de recherche** : utilisé par le Dashboard.

Les journaux stockés par Auth0 sont assujettis à une période de conservation qui dépend de votre abonnement Auth0. Pour en savoir plus, consultez [Log Data Retention](/docs/fr-CA/deploy-monitor/logs/log-data-retention).

Vous pouvez aussi utiliser ces points de terminaison pour rechercher des journaux :

* `/api/v2/logs/{id}`: récupère l’unique entrée de journal associée à l’ID de journal fourni.
* `/api/v2/users/{user_id}/logs`: récupère les journaux d’un ID utilisateur précis.

<div id="retrieve-logs-by-checkpoint">
  ## Récupérer les journaux par point de contrôle
</div>

Cette méthode vous permet de récupérer les journaux plus récents qu’un `log_id` donné. Elle
est couramment utilisée pour envoyer vos journaux vers un système externe, comme Splunk, Datadog,
Application Insights, Mixpanel, et plus encore.

Pour faire une recherche par point de contrôle, utilisez les paramètres suivants :

| Paramètre | Description                                                                                                                                  |
| --------- | -------------------------------------------------------------------------------------------------------------------------------------------- |
| `from`    | ID de l’événement de journal à partir duquel commencer à récupérer les journaux. Vous pouvez limiter le nombre de journaux à l’aide du paramètre `take`. |
| `take`    | Le nombre total d’entrées à récupérer lorsque vous utilisez le paramètre `from`.                                                             |

Tout paramètre autre que `from` et `take` sera ignoré. Les journaux sont triés selon le `log_id`.

<div id="semantics">
  #### Sémantique
</div>

La récupération des journaux par point de contrôle fonctionne comme suit :

* Tous les journaux auxquels les autorisations de votre rôle vous donnent accès seront renvoyés
* Les journaux sont triés selon le `log_id`, plutôt qu’en fonction de l’heure de génération, afin d’éviter d’omettre des enregistrements en raison des délais du système Auth0
* La pagination est illimitée, c’est-à-dire qu’il n’y a aucune limite, contrairement à la limite de `1,000` enregistrements lors de la récupération des journaux par critères de recherche

<div id="pagination">
  #### Pagination
</div>

Lors de la récupération des journaux par point de contrôle, seule la première requête doit être effectuée
manuellement. Chaque requête suivante doit être effectuée à l’aide de l’URL associée à la relation `next`
dans l’en-tête `Link`. Cela simplifie l’intégration à l’API
et permet à Auth0 d’effectuer des optimisations sans nuire à l’expérience
client.

<div id="example">
  #### Exemple
</div>

Supposons que le nom de votre tenant dans `us` est `acme`. Pour commencer à récupérer des enregistrements à partir d’un point de contrôle :

1. Récupérez le `log_id` de n’importe quel journal de votre système.
   Exemple : `90020211201185132572079310688835925971249535794074878050`.
2. Faites une requête à `https://TENANT_NAME.LOCALITY.auth0.com/api/v2/logs?from=LOG_ID&take=100` pour demander les `100` journaux suivants plus récents que le `LOG_ID` donné.
3. La réponse HTTP contiendra

   1. Un tableau JSON de `0` à `100` journaux plus récents que le `LOG_ID` donné
   2. Un en-tête HTTP `Link` avec une relation `next`
4. Si l’ensemble des résultats est vide, vous recevrez un tableau vide, et la relation `next` correspondra à la même URL que celle utilisée dans la requête la plus récente

<div id="retrieve-logs-by-search-criteria">
  ## Récupérer des journaux selon des critères de recherche
</div>

Cette méthode vous permet de récupérer les journaux qui correspondent aux critères de recherche spécifiés ou, si aucun critère n’est fourni, de lister tous les journaux. Pour effectuer une recherche par critères, utilisez les paramètres suivants :

| Paramètre        | Description                                                                                                                                                     |
| ---------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `q`              | Critères de recherche utilisant la syntaxe de chaîne de requête.                                                                                                |
| `page`           | Le numéro de page commençant à zéro.                                                                                                                            |
| `per_page`       | Le nombre d’entrées par page.                                                                                                                                   |
| `sort`           | Le champ à utiliser pour le tri. Utilisez `field:order`, où `order` vaut `1` pour l’ordre croissant et `-1` pour l’ordre décroissant. Par exemple, `date:-1`.   |
| `fields`         | Une liste de champs séparés par des virgules à inclure ou à exclure du résultat (selon `include_fields`). Laissez ce champ vide pour récupérer tous les champs. |
| `include_fields` | `true` si les champs indiqués doivent être inclus dans le résultat, `false` sinon. La valeur par défaut est `true`.                                             |

Pour consulter la liste des champs et des paramètres de tri pouvant être utilisés dans la requête de recherche, consultez [Log Search Query Syntax](/docs/fr-CA/deploy-monitor/logs/log-search-query-syntax).

<div id="limitations">
  ## Limites
</div>

En plus de la limite de 100 journaux par request pour récupérer les journaux, vous ne pouvez paginer que dans les 1 000 premiers résultats de recherche.

Si vous obtenez l’erreur `414 Request-URI Too Large`, cela signifie que votre query string dépasse la longueur prise en charge. Dans ce cas, affinez votre recherche.

<Card title="Limites des champs de recherche de Private Cloud">
  Les champs suivants sont interrogeables uniquement dans Private Cloud :

  * `user`
  * `connection`
  * `application`
  * `type`
  * `ip`

  Utilisez des guillemets doubles pour les recherches exactes (p. ex., `application:\"test\"` recherchera tous les log events propres à l’application nommée `test`, tandis que `application:test` recherchera les log events des applications dont le nom contient &quot;test&quot;.
</Card>
