Vos applications et services dépendent d’Auth0. Surveiller l’état d’Auth0 vous permet de signaler à vos clients des erreurs précises ou de réduire l’impact sur les utilisateurs en cas de problème avec Auth0.

Il existe de nombreuses façons de surveiller Auth0, et chaque approche complète les autres. Choisissez celle qui convient à vos besoins et à vos capacités d’investissement.

<div id="synthetic-transactions">
  ## Transactions synthétiques
</div>

L&#39;approche la plus simple pour surveiller Auth0 :

* Configurez une request périodique pour effectuer une transaction d&#39;authentication.
* Si la request réussit, Auth0 fonctionne correctement.
* Si la request échoue, cela peut indiquer :

  * un problème avec Auth0
  * un problème propre au tenant utilisé pour la transaction synthétique
  * ou simplement une seule request ayant échoué.

Pour les transactions synthétiques, utilisez un environnement aussi proche que possible de la configuration de votre tenant de production. Vous pouvez même, potentiellement, utiliser ce même tenant de production. Comme la configuration de transactions synthétiques avec des redirect flows et des providers tiers peut être délicate, il est recommandé d&#39;utiliser le [Resource Owner Password Grant](/docs/fr-CA/get-started/authentication-and-authorization-flow/resource-owner-password-flow/call-your-api-using-resource-owner-password-flow). Ce flow n&#39;implique pas de redirections du navigateur et ne nécessite pas de UI.

Si vous utilisez des [rules](/docs/fr-CA/customize/rules) ou des [custom database connection(s)](/docs/fr-CA/authenticate/database-connections/custom-db) ou d&#39;autres points d&#39;extensibilité, la ou les transactions synthétiques doivent être configurées pour utiliser les rules et/ou les custom DB scripts afin de s&#39;assurer que cet aspect du système fonctionne.

Des outils comme [Pingdom](http://pingdom.com/) simplifient la configuration des transactions synthétiques.

<div id="check-period">
  ### Intervalle de vérification
</div>

Nous recommandons d’exécuter des transactions synthétiques toutes les minutes. Cette approche simple consomme peu de votre limitation du débit Auth0, tout en fournissant des réponses rapides.

<div id="limitations-of-synthetic-transactions">
  ### Limites des transactions synthétiques
</div>

Les transactions synthétiques constituent un moyen simple et peu coûteux de surveiller l’état de santé d’un tenant Auth0. Cependant, elles comportent certaines limites :

* Les transactions synthétiques ne représentent pas l’expérience de votre utilisateur final. Elles vous fournissent plutôt un indicateur indirect.
* Les transactions synthétiques pourraient ne pas utiliser les mêmes flux que vos utilisateurs.
* Elles manquent d’atomicité (elles s’exécutent généralement une fois par minute) et ne signalent pas les erreurs que vos utilisateurs finaux ont pu voir.

Si vous souhaitez obtenir des données plus détaillées, consultez [Suivi des erreurs](#error-tracking) et [Métriques et journaux](#metrics-and-logs).

<div id="error-tracking">
  ## Suivi des erreurs
</div>

Cette approche est utile pour faire le suivi des erreurs dans les appels existants à Auth0. Elle consiste à signaler les erreurs chaque fois qu’un appel à Auth0 échoue. [Sentry](https://sentry.io/) est un outil couramment utilisé dans ce type de cas, et il fonctionne autant dans des scénarios frontend que backend.

Cette approche est utile, car elle vous permet de connaître les erreurs réelles que rencontrent vos utilisateurs finaux. Cependant, comme vous ne suivez que les erreurs (et non toutes les requêtes), il n’est pas possible d’avoir une idée précise du nombre d’utilisateurs finaux touchés : est-ce 1 % ou 5 % ? Elle ne vous oblige pas non plus à configurer un « appel synthétique » distinct, qui pourrait consommer une partie de votre quota de limitation du débit, surtout s’il est mal configuré.

<div id="metrics-and-logs">
  ## Métriques et journaux
</div>

Cette approche est utile si les appels à Auth0 sont effectués depuis un backend que vous contrôlez. C’est le cas pour :

* La plupart des appels à l’[API de gestion d’Auth0](https://auth0.com/docs/api/management/v2)
* Les appels à l’[API d’authentification](https://auth0.com/docs/api/authentication) effectués par des Regular Web Applications et des applications machine à machine ([en savoir plus sur les types de client](/docs/fr-CA/get-started/applications)).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Certains produits d’observabilité permettent aussi de transmettre des métriques/journaux à partir d’applications frontend et mobiles.
</Callout>

L’approche consiste à utiliser des métriques et/ou des journaux pour suivre les taux d’erreur des appels à Auth0. Les métriques/journaux fournissent des taux d’erreur qui reflètent fidèlement l’expérience des utilisateurs finaux, sans que vous ayez à configurer un « appel synthétique » distinct, qui pourrait consommer une partie de votre limitation du débit, surtout s’il est mal configuré.

<div id="alerting">
  ## Alertes
</div>

Peu importe les approches de surveillance que vous utilisez, il est courant de recevoir une alerte lorsqu’un certain taux d’erreurs est atteint. Ce taux dépend de votre application.

Lorsque votre équipe reçoit une alerte d’Auth0, nous recommandons d’ajouter un lien vers [Consulter l’état d’Auth0](/docs/fr-CA/deploy-monitor/monitor/check-auth0-status) dans le payload/guide d’intervention de l’alerte. Cela lui permettra de consulter rapidement le canal officiel d’Auth0 pour vérifier si le problème vient d’Auth0 ou de votre application/service.
