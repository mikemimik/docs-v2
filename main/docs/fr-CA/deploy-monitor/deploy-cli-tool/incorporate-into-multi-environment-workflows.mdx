---
description: Découvrez comment prendre en charge des flux de travail multilocataires et multi-environnement lorsque vous utilisez le Deploy CLI d’Auth0.
title: Intégrer aux flux de travail multi-environnement
---

Le Deploy CLI permet de travailler dans un contexte multilocataire et multi-environnement. Lorsqu’il est intégré à vos flux de travail de développement CI/CD, il peut servir à répercuter les changements apportés à Auth0 depuis le développement de fonctionnalités jusqu’à la production.

En général, le flux de travail recommandé est le suivant :

1. Créez un tenant Auth0 distinct pour chaque environnement (développement, préproduction, production).
2. Créez un seul dépôt de fichiers de configuration des ressources pour tous les environnements.
3. Ajoutez une étape à votre pipeline CI/CD qui, lors du déploiement dans les environnements, applique les configurations des ressources Auth0 au tenant Auth0 approprié.

<div id="tenant-to-environment">
  ## Un tenant par environnement
</div>

Il est recommandé d’avoir un tenant ou compte Auth0 distinct pour chaque environnement. Par exemple :

| Environnement | Tenant          |
| ------------- | --------------- |
| Développement | `travel0-dev`   |
| Tests         | `travel0-uat`   |
| Préproduction | `travel0-stage` |
| Production    | `travel0-prod`  |

<div id="resource-configuration-repository">
  ## Dépôt de configuration des ressources
</div>

Lors de l’exportation, l’état de votre tenant Auth0 sera représenté par un ensemble de fichiers de configuration des ressources, au [format YAML ou directory](/docs/fr-CA/deploy-monitor/deploy-cli-tool/available-resource-configuration-formats). Dans un contexte multi-environnement, on s’attend à utiliser un seul dépôt de configurations de ressources pour tous les environnements. En pratique, il peut s’agir d’un répertoire dans la base de code de votre projet ou d’une base de code entièrement distincte.

Vous devriez avoir au moins une branche par tenant dans votre dépôt, ce qui vous permet d’apporter des modifications sans les déployer. Ainsi, les modifications ne seront déployées que lorsque vous fusionnerez la branche de travail dans la branche principale (comme `main` ou `master`). Avec cette configuration, vous pouvez avoir une tâche d’intégration continue pour chaque environnement qui déploie automatiquement les modifications dans l’environnement ciblé chaque fois que la branche principale reçoit des mises à jour.

Votre flux de travail pourrait ressembler à ceci :

1. Apportez des modifications en développement.
2. Fusionnez les modifications vers l’environnement de test (ou UAT).
3. Testez les modifications dans l’UAT. Lorsque vous êtes prêt, transférez-les et fusionnez-les vers la préproduction.
4. Testez la préproduction. Lorsque vous êtes prêt, transférez-les et fusionnez-les vers la production.

Par précaution, vous pourriez vouloir configurer votre environnement de production pour qu’il se déploie uniquement lorsqu’il est déclenché manuellement.

<div id="unidirectional-flow">
  ## Flux unidirectionnel
</div>

Le flux de travail multi-environnement fonctionne mieux lorsque les changements sont propagés « vers le haut » dans une seule direction. Les modifications apportées aux fichiers de configuration des ressources devraient d’abord être appliquées à l’environnement de niveau le plus bas (comme le développement), puis déployées graduellement dans tous les autres environnements jusqu’à la production. Cette approche unidirectionnelle assure des tests et des validations suffisants pour les changements apportés à votre tenant. Une fois cette approche établie, il est recommandé de ne pas appliquer de configurations directement en production par d’autres moyens, comme le <Tooltip tip="Auth0 Dashboard : principal produit d’Auth0 pour configurer vos services." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip> ou la <Tooltip tip="Auth0 Dashboard : principal produit d’Auth0 pour configurer vos services." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Management+API">Management API</Tooltip>, à moins que ces changements ne soient consignés dans une exportation subséquente de Deploy CLI. Sinon, ils risquent d’être écrasés.

<div id="environment-specific-values">
  ## Valeurs propres à l’environnement
</div>

Bien qu’il soit prévu que tous les environnements partagent le même ensemble de fichiers de configuration des ressources, les valeurs propres à l’environnement peuvent être définies au moyen de fichiers de configuration d’outil distincts et du [remplacement de mots-clés](/docs/fr-CA/deploy-monitor/deploy-cli-tool/keyword-replacement) dynamique.

<div id="separate-configuration-files">
  ## Fichiers de configuration distincts
</div>

Le fait de prévoir un fichier de configuration de l’outil distinct pour chaque environnement permet de garder les fichiers de configuration des ressources indépendants de l’environnement, tout en répondant aux besoins de chacun. Au minimum, vous devrez fournir des informations d’authentification distinctes pour chaque environnement, mais il est aussi possible d’exclure certaines ressources, d’activer la suppression et d’effectuer un remplacement dynamique de mots-clés selon l’environnement.

<div id="example-file-structure">
  ### Exemple d’arborescence de fichiers
</div>

```text lines
project-root
│
└───auth0
│   │   config-dev.json   # Fichier de configuration de l'environnement de développement
│   │   config-test.json  # Fichier de configuration de l'environnement de test
│   │   config-prod.json  # Fichier de configuration de l'environnement de production
│   │   ... tous les autres fichiers de configuration des ressources
│
└───src
    │   ... le code de votre projet
```

<div id="dynamic-values-with-keyword-replacement">
  ## Valeurs dynamiques avec remplacement de mots-clés
</div>

Une fois que des fichiers de configuration distincts sont utilisés pour chaque environnement, vous pouvez vous servir du remplacement de mots-clés au moyen de la propriété de configuration `AUTH0_KEYWORD_REPLACE_MAPPINGS` pour définir des valeurs de remplacement dynamiques selon l’environnement. Par exemple, vous pourriez avoir besoin d’un ensemble distinct d’origines autorisées pour vos applications clientes. Pour en savoir plus, consultez [Remplacement de mots-clés](/docs/fr-CA/deploy-monitor/deploy-cli-tool/keyword-replacement).

<div id="example-config-devjson">
  ### Exemple de config-dev.json
</div>

```json lines
{
  "AUTH0_DOMAIN": "travel0-dev.us.auth0.com",
  "AUTH0_CLIENT_ID": "PdwQpGy62sHcsV6ufZNEVrV4GDlDhm74",
  "AUTH0_ALLOW_DELETE": true,
  "AUTH0_KEYWORD_REPLACE_MAPPINGS": {
    "ENV": "dev",
    "ALLOWED_ORIGINS": ["http://localhost:3000", "http://dev.travel0.com"]
  }
}
```

<div id="example-config-prodjson">
  ### Exemple de config-prod.json
</div>

```json lines
{
  "AUTH0_DOMAIN": "travel0.us.auth0.com",
  "AUTH0_CLIENT_ID": "vZCEFsDYzXc1x9IomB8dF185e4cdVah5",
  "AUTH0_ALLOW_DELETE": false,
  "AUTH0_KEYWORD_REPLACE_MAPPINGS": {
    "ENV": "prod",
    "ALLOWED_ORIGINS": ["http://travel0.com"]
  }
}
```
