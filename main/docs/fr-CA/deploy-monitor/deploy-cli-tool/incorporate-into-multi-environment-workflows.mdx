description: Découvrez comment prendre en charge des flux de travail multitenant et multi-environnement lorsque vous utilisez Auth0 Deploy CLI.
title: Intégration aux flux de travail multi-environnement
---

Le Deploy CLI permet de travailler dans un contexte multitenant et multi-environnement. Lorsqu’il est intégré à vos flux de travail de développement CI/CD, il peut servir à propager les modifications apportées à Auth0, du développement de fonctionnalités jusqu’à la production.

De façon générale, le flux de travail recommandé est le suivant :

1. Créez un tenant Auth0 distinct pour chaque environnement (développement, staging, production).
2. Créez un seul dépôt de fichiers de configuration des ressources pour tous les environnements.
3. Ajoutez une étape à votre pipeline CI/CD qui, lors du déploiement dans les environnements, applique les configurations des ressources Auth0 au tenant Auth0 approprié.

<div id="tenant-to-environment">
  ## Tenant par environnement
</div>

Il est recommandé d’avoir un tenant ou un compte Auth0 distinct pour chaque environnement. Par exemple :

| Environnement | Tenant          |
| ------------- | --------------- |
| Développement | `travel0-dev`   |
| Test          | `travel0-uat`   |
| Préproduction | `travel0-stage` |
| Production    | `travel0-prod`  |

<div id="resource-configuration-repository">
  ## Dépôt de configuration des ressources
</div>

Une fois exporté, l’état de votre tenant Auth0 sera représenté sous forme d’un ensemble de fichiers de configuration des ressources, soit en [format YAML ou en format par répertoires](/docs/fr-CA/deploy-monitor/deploy-cli-tool/available-resource-configuration-formats). Dans un contexte multi-environnement, on s’attend à avoir un seul dépôt de configurations de ressources qui s’applique à tous les environnements. En pratique, ce dépôt peut prendre la forme d’un répertoire dans la base de code de votre projet ou d’une base de code distincte.

Vous devriez avoir au moins une branche par tenant dans votre dépôt, ce qui vous permet d’apporter des changements sans les déployer. Ainsi, les changements ne seront déployés que lorsque vous fusionnerez la branche de travail dans la branche principale (comme `main` ou `master`). Avec cette configuration, vous pouvez avoir une tâche d’intégration continue pour chaque environnement qui déploie automatiquement les changements vers l’environnement ciblé chaque fois que la branche principale reçoit des mises à jour.

Votre flux de travail pourrait ressembler à ceci :

1. Apportez des changements à l’environnement de développement.
2. Fusionnez les changements vers l’environnement de test (ou uat).
3. Testez les changements dans uat. Lorsque vous êtes prêt, déplacez-les et fusionnez-les vers l’environnement de préproduction.
4. Testez l’environnement de préproduction. Lorsque vous êtes prêt, déplacez-les et fusionnez-les vers l’environnement de production.

Par mesure de précaution, vous pourriez vouloir configurer votre environnement de production pour qu’il se déploie uniquement lorsqu’il est déclenché manuellement.

<div id="unidirectional-flow">
  ## Flux unidirectionnel
</div>

Le flux de travail multi-environnement fonctionne mieux lorsque les changements se propagent « vers le haut » dans une seule direction. Les modifications aux fichiers de configuration des ressources doivent d’abord être appliquées à l’environnement de niveau inférieur (comme le développement), puis déployées progressivement dans tous les autres environnements jusqu’à la production. Cette approche unidirectionnelle permet d’assurer des tests et des approbations suffisants pour les changements apportés à votre tenant. Une fois ce processus établi, il est recommandé de ne pas appliquer directement des configurations en production par d’autres moyens, comme le <Tooltip tip="Auth0 Dashboard : l’interface principale d’Auth0 pour configurer vos services." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip> ou la <Tooltip tip="Auth0 Dashboard : l’interface principale d’Auth0 pour configurer vos services." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Management+API">Management API</Tooltip>, à moins que ces changements ne soient repris dans une exportation ultérieure avec Deploy CLI. Sinon, ces changements risquent d’être écrasés.

<div id="environment-specific-values">
  ## Valeurs spécifiques à l’environnement
</div>

Bien qu’il soit prévu que tous les environnements partagent le même ensemble de fichiers de configuration des ressources, les valeurs propres à chaque environnement peuvent être définies au moyen de fichiers de configuration d’outil distincts et du [remplacement de mots-clés](/docs/fr-CA/deploy-monitor/deploy-cli-tool/keyword-replacement) dynamique.

<div id="separate-configuration-files">
  ## Fichiers de configuration distincts
</div>

Le fait de prévoir un fichier de configuration de l’outil distinct pour chaque environnement permet de garder les fichiers de configuration des ressources indépendants de l’environnement, tout en répondant aux besoins propres à chacun. Au minimum, vous devrez fournir des informations d’authentification distinctes pour chaque environnement, mais il est aussi possible d’exclure certaines ressources, d’activer la suppression et d’effectuer un remplacement dynamique de mots-clés pour chaque environnement.

<div id="example-file-structure">
  ### Exemple d’arborescence de fichiers
</div>

```text lines
project-root
│
└───auth0
│   │   config-dev.json   # Fichier de configuration de l'environnement de développement
│   │   config-test.json  # Fichier de configuration de l'environnement de test
│   │   config-prod.json  # Fichier de configuration de l'environnement de production
│   │   ... tous les autres fichiers de configuration des ressources
│
└───src
    │   ... le code de votre projet
```

<div id="dynamic-values-with-keyword-replacement">
  ## Valeurs dynamiques avec remplacement de mots-clés
</div>

Une fois des fichiers de configuration distincts adoptés pour chaque environnement, vous pouvez utiliser le remplacement de mots-clés au moyen de la propriété de configuration `AUTH0_KEYWORD_REPLACE_MAPPINGS` pour définir des valeurs de remplacement dyniques selon l’environnement. Par exemple, il peut être nécessaire d’avoir un ensemble distinct d’origines autorisées pour vos clients. Pour en savoir plus, consultez [Remplacement de mots-clés](/docs/fr-CA/deploy-monitor/deploy-cli-tool/keyword-replacement).

<div id="example-config-devjson">
  ### Exemple de config-dev.json
</div>

```json lines
{
  "AUTH0_DOMAIN": "travel0-dev.us.auth0.com",
  "AUTH0_CLIENT_ID": "PdwQpGy62sHcsV6ufZNEVrV4GDlDhm74",
  "AUTH0_ALLOW_DELETE": true,
  "AUTH0_KEYWORD_REPLACE_MAPPINGS": {
    "ENV": "dev",
    "ALLOWED_ORIGINS": ["http://localhost:3000", "http://dev.travel0.com"]
  }
}
```

<div id="example-config-prodjson">
  ### Exemple de config-prod.json
</div>

```json lines
{
  "AUTH0_DOMAIN": "travel0.us.auth0.com",
  "AUTH0_CLIENT_ID": "vZCEFsDYzXc1x9IomB8dF185e4cdVah5",
  "AUTH0_ALLOW_DELETE": false,
  "AUTH0_KEYWORD_REPLACE_MAPPINGS": {
    "ENV": "prod",
    "ALLOWED_ORIGINS": ["http://travel0.com"]
  }
}
```
