description: Découvrez comment gérer un flux de travail multilocataire avec le Deploy CLI d’Auth0 à l’aide du remplacement de mots-clés.
title: Remplacement de mots-clés
---

Le Deploy CLI prend en charge le remplacement dynamique de mots-clés par des valeurs propres à l’environnement. Cela permet un flux de travail multilocataire évolutif dans lequel tous les tenants partagent les mêmes fichiers de configuration des ressources, mais y injectent des valeurs légèrement différentes.

Pour utiliser le remplacement de mots-clés, la propriété de configuration `AUTH0_KEYWORD_REPLACE_MAPPINGS` doit contenir les mappages appropriés. Ensuite, dans les fichiers de configuration des ressources, les mots-clés peuvent être injectés de l’une des deux façons suivantes :

1. `@@EXAMPLE_KEY@@` : L’utilisation des symboles `@` amène l’outil à effectuer un `JSON.stringify` de votre valeur avant de la remplacer. Ainsi, si votre valeur est une chaîne de caractères, l’outil ajoutera des guillemets; si votre valeur est un tableau ou un objet, l’outil ajoutera des accolades.
2. `##EXAMPLE_KEY##` : L’utilisation du symbole `#` amène l’outil à effectuer un remplacement littéral; il n’ajoutera ni guillemets ni accolades.

<div id="example-configjson">
  ## Exemple de fichier config.json
</div>

```json lines
{
  "AUTH0_DOMAIN": "test-tenant.us.auth0.com",
  "AUTH0_CLIENT_ID": "FOO",
  "AUTH0_CLIENT_SECRET": "BAR",
  "AUTH0_KEYWORD_REPLACE_MAPPINGS": {
    "ENVIRONMENT": "dev",
    "ALLOWED_LOGOUT_URLS": ["https://dev-test-site.com/logout", "localhost:3000/logout"],
    "ALLOWED_ORIGINS": ["https://dev-test-site.com", "localhost:3000"]
  }
}
```

## Exemple de tenant.yaml

```yaml lines
tenant:
  friendly_name: "##ENVIRONMENT## tenant"
  allowed_logout_urls: @@ALLOWED_LOGOUT_URLS@@
  enabled_locales:
    - en
clients:
  - name: Test App
    allowed_origins: @@ALLOWED_ORIGINS@@
    allowed_logout_urls: @@ALLOWED_LOGOUT_URLS@@
```

<div id="example-tenantjson">
  ## Exemple de fichier tenant.json
</div>

```json lines
{
  "friendly_name": "##ENVIRONMENT## tenant",
  "allowed_logout_urls": "@@ALLOWED_LOGOUT_URLS@@"
}
```

<div id="array-concatenation">
  ## Concaténation de tableaux
</div>

Il peut arriver que vous ayez à concaténer des valeurs à un tableau statique au moyen du remplacement de mots-clés. Il n’existe aucune syntaxe particulière pour ce cas; toutefois, il est possible d’y parvenir en échappant les guillemets doubles dans une seule chaîne contenant les valeurs appropriées, puis en les injectant à l’aide de la syntaxe de mot-clé `##`. Cette technique fonctionne à la fois avec les [formats YAML et de répertoire](/docs/fr-CA/deploy-monitor/deploy-cli-tool/available-resource-configuration-formats).

### Exemple de config.json

```json lines
{
  "AUTH0_KEYWORD_REPLACE_MAPPINGS": {
    "GLOBAL_WEB_ORIGINS": "\"http://local.me:8080\", \"http://localhost\", \"http://localhost:3000\""
  }
}
```

<div id="example-tenantyaml">
  ### Exemple de tenant.yaml
</div>

```yaml lines
clients:
  - name: Test App
    web_origins: [ "http://production-app.com", "https://production-app.com", ##GLOBAL_WEB_ORIGINS## ]
```

<div id="preserve-keywords-on-export">
  ## Préserver les mots-clés lors de l’exportation
</div>

En général, le Deploy CLI fonctionne mieux lorsqu’il est utilisé dans un flux de travail unidirectionnel, à partir de vos environnements de niveau inférieur (p. ex. : dev, test) jusqu’à vos environnements de production. Il sera toutefois parfois nécessaire d’exporter la configuration d’un environnement de niveau supérieur vers votre répertoire de configuration local. Par défaut, les valeurs distantes écraseront vos valeurs locales, **ce qui supprimera vos marqueurs de mots-clés**. Cependant, la préservation du remplacement de mots-clés peut être activée au moyen de la propriété de configuration booléenne `AUTH0_PRESERVE_KEYWORDS`. En activant cette configuration, le Deploy CLI tentera de préserver, lors de l’exportation, les marqueurs de mots-clés définis dans vos fichiers de configuration locaux.

La fonctionnalité de préservation des mots-clés tentera d’en conserver le plus grand nombre possible tout en maintenant l’exactitude de vos fichiers de configuration de ressources. Dans la majorité des cas, elle fonctionnera sans aucune intervention de l’utilisateur. Cependant, certaines limites importantes s’appliquent :

* Dans le cas d’un champ de configuration avec remplacement de mots-clés dont les valeurs diffèrent entre le local et le distant, la valeur de configuration locale sera toujours privilégiée. Cela fera en sorte que **toute modification hors bande dans l’environnement distant sera effacée** si un marqueur de remplacement de mot-clé existe à quelque endroit que ce soit dans la valeur de ce champ dans le fichier de définition de la ressource; il n’y a aucune réconciliation « intelligente ».
* Les arrays sans identificateurs précis ne peuvent pas être préservés. Ex. : `["http://site.com/logout", "localhost:3000/logout", "##LOGOUT_URL##"]`. Cela s’explique par le fait que l’ordre de ces valeurs n’est pas déterministe. Pour préserver ces valeurs, il est plutôt recommandé d’utiliser la syntaxe de remplacement de mot-clé `@@ARRAY_REPLACE@@` avec la valeur complète.

Pour en savoir plus sur l’historique et les défis techniques liés à la préservation des mots-clés, consultez [RFC: Keyword Preservation During Export](https://github.com/auth0/auth0-deploy-cli/issues/688).
