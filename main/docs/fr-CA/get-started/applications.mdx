Dans Auth0, le terme **application** ou **app** ne renvoie à aucune caractéristique particulière de mise en œuvre. Par exemple, il peut s’agir d’une application native qui s’exécute sur un appareil mobile, d’une application Web monopage qui s’exécute dans un navigateur, ou d’une application Web classique qui s’exécute sur un serveur.

Auth0 classe les apps selon les caractéristiques suivantes :

* **Type d’application** : Pour ajouter l’authentification à votre application, vous devez l’enregistrer dans l’<Tooltip tip="Auth0 Dashboard : le principal produit Auth0 pour configurer vos services." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip> et sélectionner l’un des types d’application suivants :

  * **Application Web classique** : Applications Web traditionnelles qui exécutent la majeure partie de leur logique applicative sur le serveur (comme Express.js ou ASP.NET). Pour savoir comment configurer une application Web classique, consultez [Enregistrer des applications Web classiques](/docs/fr-CA/get-started/auth0-overview/create-applications/regular-web-apps).
  * **Application Web monopage (SPA)** : Applications JavaScript qui exécutent la majeure partie de leur logique d’interface utilisateur dans un navigateur Web et communiquent avec un serveur Web principalement au moyen d’API (comme AngularJS + Node.js ou React). Pour savoir comment configurer une application Web monopage, consultez [Enregistrer des applications Web monopages](/docs/fr-CA/get-started/auth0-overview/create-applications/single-page-web-apps).
  * **Application native** : Applications mobiles ou de bureau qui s’exécutent nativement sur un appareil (comme iOS ou Android). Pour savoir comment configurer une application native, consultez [Enregistrer des applications natives](/docs/fr-CA/get-started/auth0-overview/create-applications/native-apps).
  * **Application machine à machine (M2M)** : Applications non interactives, comme des outils en ligne de commande, des démons, des appareils IoT ou des services exécutés sur votre backend. En général, vous utiliserez cette option si vous avez un service qui a besoin d’accéder à une API. Pour savoir comment configurer une application machine à machine, consultez [Enregistrer des applications machine à machine](/docs/fr-CA/get-started/auth0-overview/create-applications/machine-to-machine-apps).
* **Sécurité des identifiants** : Selon la [spécification OAuth 2.0](https://tools.ietf.org/html/rfc6749#section-2.1), les apps peuvent être classées comme publiques ou confidentielles; les applications confidentielles peuvent conserver leurs identifiants de manière sécurisée, contrairement aux applications publiques. Pour en savoir plus, consultez [applications confidentielles et publiques](/docs/fr-CA/get-started/applications/confidential-and-public-applications).
* **Propriété** : Le fait qu’une app soit considérée comme first-party ou third-party dépend de son propriétaire et de qui en assure le contrôle. Les apps first-party sont contrôlées par la même organisation ou la même personne qui possède le domaine Auth0. Les apps third-party permettent à des parties externes ou à des partenaires d’accéder de façon sécurisée aux ressources protégées derrière votre API. Pour en savoir plus, consultez [applications propriétaires et tierces](/docs/fr-CA/get-started/applications/first-party-and-third-party-applications).

<div id="manage-applications-settings">
  ## Gérer les paramètres des applications
</div>

Vous enregistrez les applications dans [Dashboard &gt; Applications &gt; Applications](https://manage.auth0.com/#/applications/\{yourClientId}/settings). En plus de configurer les applications dans le Dashboard, vous pouvez aussi configurer les applications par programmation, comme décrit dans la spécification [OpenID Connect (OIDC) Dynamic Client Registration 1.0](https://openid.net/specs/openid-connect-registration-1_0.html).

Vous pouvez mettre en place une configuration plus complexe qui permet aux utilisateurs de se connecter différemment selon l’application. Pour en savoir plus, consultez [Bonnes pratiques pour les applications multilocataires](/docs/fr-CA/get-started/auth0-overview/create-tenants/multi-tenant-apps-best-practices) et [Créer plusieurs tenants](/docs/fr-CA/get-started/auth0-overview/create-tenants/create-multiple-tenants).

Par défaut, Auth0 active toutes les connexions associées à votre tenant lorsque vous créez une nouvelle application. Pour modifier ce comportement, [mettez à jour les connexions de l’application](/docs/fr-CA/get-started/applications/update-application-connections) dans les **Application Settings** du Dashboard.

<div id="monitor-applications">
  ## Surveiller les applications
</div>

Vous pouvez [surveiller les applications](/docs/fr-CA/deploy-monitor/monitor/monitor-applications) et effectuer des tests de bout en bout à l’aide de vos propres tests. Auth0 stocke des [données de journal](/docs/fr-CA/deploy-monitor/logs), y compris les actions des administrateurs du Dashboard, les tentatives d’authentification des utilisateurs réussies ou échouées, ainsi que les requêtes de changement de mot de passe. Vous pouvez utiliser la diffusion des journaux dans [Auth0 Marketplace](https://marketplace.auth0.com/features/log-streaming) pour exporter vos données de journal et vous servir d’outils comme Sumo Logic, Splunk ou Mixpanel pour les analyser et les stocker.

<div id="remove-applications">
  ## Supprimer des applications
</div>

Vous pouvez [supprimer une application](/docs/fr-CA/get-started/applications/remove-applications) à l’aide du Dashboard ou de la <Tooltip tip="Management API : un produit qui permet aux clients d’effectuer des tâches administratives." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Management+API">Management API</Tooltip>.

<div id="manage-client-secrets">
  ## Gérer les secrets clients
</div>

Un <Tooltip tip="Serveur d’autorisation : serveur centralisé qui contribue à définir les limites de l’accès d’un utilisateur. Par exemple, votre serveur d’autorisation peut contrôler les données, les tâches et les fonctionnalités accessibles à un utilisateur." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=client+secret">secret client</Tooltip> est un secret connu uniquement de votre application et du <Tooltip tip="Secret client : secret utilisé par un client (application) pour s’authentifier auprès du serveur d’autorisation; il ne doit être connu que du client et du serveur d’autorisation et doit être suffisamment aléatoire pour ne pas pouvoir être deviné." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=authorization+server">serveur d’autorisation</Tooltip>. Il protège vos ressources en n’accordant des [jetons](/docs/fr-CA/secure/tokens) qu’aux entités autorisées à en faire la demande.

Protégez vos secrets clients et ne les incluez **jamais** dans des applications mobiles ou des applications exécutées dans le navigateur. Si votre secret client est compromis, vous devriez [le remplacer par un nouveau](/docs/fr-CA/get-started/applications/rotate-client-secret) et mettre à jour toutes les applications autorisées avec le nouveau secret client.

<div id="grant-types">
  ## Types d’autorisation
</div>

Auth0 offre de nombreux types d’autorisation, ou flux d’authentification et d’autorisation, et vous permet d’indiquer quels types d’autorisation sont appropriés en fonction de la propriété `grant_types` de votre application enregistrée auprès d’Auth0. Pour en savoir plus, consultez [types d’octroi d’application](/docs/fr-CA/get-started/applications/application-grant-types).
