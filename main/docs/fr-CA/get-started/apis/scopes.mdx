Différents types de renseignements sur l’utilisateur sont souvent stockés dans diverses ressources en ligne. Les utilisateurs peuvent téléverser et stocker des photos dans un service comme Flickr, conserver des fichiers numériques sur Dropbox et stocker des contacts et des événements dans Google Calendar ou sur Facebook.

Souvent, de nouvelles applications veulent utiliser l’information qui a déjà été créée dans une ressource en ligne. Pour ce faire, l’application doit demander une autorisation pour accéder à cette information au nom de l’utilisateur. Les scopes définissent les actions précises que les applications peuvent être autorisées à effectuer au nom de l’utilisateur.

<div id="ways-to-use-scopes">
  ## Façons d’utiliser les scopes
</div>

Lorsqu’une application demande la permission d’accéder à une ressource par l’intermédiaire d’un <Tooltip tip="Serveur d’autorisation : serveur centralisé qui contribue à définir les limites de l’accès d’un utilisateur. Par exemple, votre serveur d’autorisation peut contrôler les données, les tâches et les fonctionnalités offertes à un utilisateur." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=authorization+server">serveur d’autorisation</Tooltip>, elle utilise le paramètre `scope` pour préciser l’accès dont elle a besoin, et le serveur d’autorisation utilise le paramètre `scope` pour indiquer l’accès qui a réellement été accordé (si l’accès accordé diffère de celui qui a été demandé).

En général, vous utilisez les scopes de trois façons :

* À partir d’une application, pour vérifier l’identité d’un utilisateur et obtenir des renseignements de base sur son profil, comme son courriel ou sa photo de profil. Dans ce scénario, les scopes à votre disposition comprennent ceux mis en œuvre par le protocole <Tooltip tip="OpenID : norme ouverte d’authentification qui permet aux applications de vérifier l’identité des utilisateurs sans recueillir ni stocker les renseignements de connexion." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=OpenID">OpenID</Tooltip> Connect (OIDC). Pour en savoir plus, consultez [Scopes OpenID Connect](/docs/fr-CA/get-started/apis/scopes/openid-connect-scopes).
* Dans une [API](/docs/fr-CA/api), pour mettre en œuvre le contrôle d’accès. Dans ce cas, vous devez définir des scopes personnalisés pour votre API, puis les identifier afin que les applications appelantes puissent les utiliser. Pour en savoir plus, consultez [Scopes d’API](/docs/fr-CA/get-started/apis/scopes/api-scopes).
* À partir d’une application, pour appeler une API qui a mis en œuvre ses propres scopes personnalisés. Dans ce cas, vous devez savoir quels scopes personnalisés sont définis pour l’API que vous appelez. Pour voir des exemples d’appel d’une API personnalisée à partir d’une application, consultez [Exemples de cas d’utilisation : Scopes et Claims](/docs/fr-CA/get-started/apis/scopes/sample-use-cases-scopes-and-claims)

<div id="best-practices">
  ## Bonnes pratiques
</div>

Comprenez votre cas d’utilisation et choisissez les scopes les plus restrictifs possible.

Si vous demandez des scopes, assurez-vous de demander suffisamment d’accès pour que votre application fonctionne, mais ne demandez que ce dont vous avez absolument besoin. Cherchez-vous à établir l’identité d’un utilisateur ou à obtenir son autorisation pour interagir avec ses données? Il y a une grande différence entre importer les renseignements du profil Facebook d’un utilisateur et publier sur son mur. En ne demandant que ce dont vous avez besoin, vous augmentez vos chances d’obtenir le consentement de l’utilisateur lorsque requis, puisque les utilisateurs sont plus enclins à accorder un accès limité à des scopes clairement définis.

De même, lorsque vous créez des scopes personnalisés pour une API, réfléchissez aux niveaux d’accès granulaires dont les applications pourraient avoir besoin et concevez-les en conséquence.

<div id="requested-scopes-versus-granted-scopes">
  ## Scopes demandés et scopes accordés
</div>

Dans certains cas, les utilisateurs peuvent consentir à l’accès demandé. Même si, habituellement, les scopes renvoyés sont identiques à ceux qui ont été demandés, les utilisateurs peuvent modifier les scopes accordés (au moment du consentement initial et parfois aussi par la suite, selon la ressource), accordant ainsi à une application moins d’accès que ce qu’elle a demandé.

En tant que développeur d’application, vous devez être conscient de cette possibilité et la prendre en compte dans votre application. Par exemple, votre application pourrait avertir l’utilisateur que certaines fonctionnalités seront limitées. Elle pourrait aussi rediriger l’utilisateur dans le <Tooltip tip="Flux d’autorisation : mécanisme d’autorisation (ou workflow) spécifié dans le framework OAuth 2.0." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=authorization+flow">flux d’autorisation</Tooltip> pour demander des permissions supplémentaires. Mais, encore une fois, rappelez-vous que lorsqu’on leur demande leur consentement, les utilisateurs peuvent toujours refuser.

Par défaut, Auth0 ne demande pas le consentement de l’utilisateur pour les applications de première partie, c’est-à-dire les applications enregistrées sous le même domaine Auth0 que l’API qu’elles appellent; toutefois, vous pouvez configurer votre API dans Auth0 pour exiger le consentement de l’utilisateur pour les applications de première partie. Les applications tierces, qui sont des applications externes, exigent le consentement de l’utilisateur.
