Différents types de renseignements sur l’utilisateur sont souvent stockés dans diverses ressources en ligne. Les utilisateurs peuvent téléverser et stocker des photos dans un service comme Flickr, conserver des fichiers numériques dans Dropbox et stocker des contacts et des événements dans Google Calendar ou sur Facebook.

Souvent, de nouvelles applications veulent utiliser les renseignements déjà enregistrés dans une ressource en ligne. Pour ce faire, l’application doit demander l’autorisation d’accéder à ces renseignements au nom de l’utilisateur. Les portées définissent les actions précises que les applications peuvent être autorisées à effectuer au nom d’un utilisateur.

<div id="ways-to-use-scopes">
  ## Façons d’utiliser les portées
</div>

Lorsqu’une application demande l’autorisation d’accéder à une ressource auprès d’un <Tooltip tip="Serveur d’autorisation : serveur centralisé qui contribue à définir les limites de l’accès d’un utilisateur. Par exemple, votre serveur d’autorisation peut contrôler les données, les tâches et les fonctionnalités accessibles à un utilisateur." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=authorization+server">serveur d’autorisation</Tooltip>, elle utilise le paramètre `scope` pour préciser l’accès dont elle a besoin, et le serveur d’autorisation utilise le paramètre `scope` pour renvoyer l’accès qui a réellement été accordé (si l’accès accordé diffère de celui qui a été demandé).

En général, vous utilisez les portées de trois façons :

* À partir d’une application, pour vérifier l’identité d’un utilisateur et obtenir des renseignements de base sur son profil, comme son courriel ou sa photo de profil. Dans ce scénario, les portées offertes comprennent celles mises en œuvre par le protocole <Tooltip tip="OpenID : norme ouverte d’authentification qui permet aux applications de vérifier l’identité des utilisateurs sans recueillir ni stocker leurs renseignements de connexion." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=OpenID">OpenID</Tooltip> Connect (OIDC). Pour en savoir plus, consultez [Portées OpenID Connect](/docs/fr-CA/get-started/apis/scopes/openid-connect-scopes).
* Dans une [API](/docs/fr-CA/api), pour mettre en œuvre le contrôle d’accès. Dans ce cas, vous devez définir des portées personnalisées pour votre API, puis les identifier afin que les applications appelantes puissent les utiliser. Pour en savoir plus, consultez [Portées d’API](/docs/fr-CA/get-started/apis/scopes/api-scopes).
* À partir d’une application, pour appeler une API qui a mis en œuvre ses propres portées personnalisées. Dans ce cas, vous devez savoir quelles portées personnalisées sont définies pour l’API que vous appelez. Pour voir des exemples d’appel d’une API personnalisée à partir d’une application, consultez [Exemples de cas d’utilisation : portées et revendications](/docs/fr-CA/get-started/apis/scopes/sample-use-cases-scopes-and-claims)

<div id="best-practices">
  ## Bonnes pratiques
</div>

Comprenez bien votre cas d’utilisation et choisissez les portées les plus restrictives possible.

Si vous demandez des portées, assurez-vous d’en demander assez pour que votre application fonctionne, mais uniquement ce dont elle a absolument besoin. Cherchez-vous à établir l’identité d’un utilisateur ou à lui demander l’autorisation d’interagir avec ses données? Il y a une grande différence entre importer les renseignements du profil Facebook d’un utilisateur et publier sur son mur. En ne demandant que ce dont vous avez besoin, vous avez davantage de chances d’obtenir le consentement de l’utilisateur lorsque requis, car les utilisateurs sont plus enclins à accorder l’accès à des portées limitées et clairement définies.

De même, lorsque vous créez des portées personnalisées pour une API, réfléchissez aux niveaux d’accès détaillés dont les applications pourraient avoir besoin et concevez-les en conséquence.

<div id="requested-scopes-versus-granted-scopes">
  ## Portées demandées et portées accordées
</div>

Dans certains cas, les utilisateurs doivent consentir à l’accès demandé. Même si, en général, les portées renvoyées sont identiques à celles qui ont été demandées, les utilisateurs peuvent modifier les portées accordées (autant au moment du consentement initial que parfois par la suite, selon la ressource), accordant ainsi à une application moins d’accès que ce qu’elle avait demandé.

En tant que développeur d’application, vous devez tenir compte de cette possibilité et prévoir ces cas dans votre application. Par exemple, votre application pourrait avertir l’utilisateur que certaines fonctionnalités seront limitées. Elle pourrait aussi rediriger l’utilisateur dans le <Tooltip tip="Flux d’autorisation : octroi d’autorisation (ou workflow) précisé dans le framework OAuth 2.0." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=authorization+flow">flux d’autorisation</Tooltip> pour demander des permissions supplémentaires. Mais, encore une fois, rappelez-vous que lorsqu’on leur demande leur consentement, les utilisateurs peuvent toujours refuser.

Par défaut, Auth0 ne demande pas le consentement de l’utilisateur pour les applications de première partie, c’est-à-dire les applications enregistrées sous le même domaine Auth0 que l’API qu’elles appellent; toutefois, vous pouvez configurer votre API dans Auth0 pour exiger le consentement de l’utilisateur pour les applications de première partie. Les applications tierces, c’est-à-dire les applications externes, exigent le consentement de l’utilisateur.
