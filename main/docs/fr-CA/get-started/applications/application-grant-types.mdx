---
description: Décrit les types d’octroi et leur lien avec les applications.
title: Types d’octroi d’application
---

Les types d’octroi d’application (ou flux) sont les mécanismes par lesquels les applications peuvent obtenir des [jetons d’accès](/docs/fr-CA/secure/tokens/access-tokens), et par lesquels vous accordez un accès limité à vos ressources à une autre entité sans exposer d’identifiants. Le [protocole OAuth 2.0](/docs/fr-CA/get-started/authentication-and-authorization-flow/which-oauth-2-0-flow-should-i-use) prend en charge plusieurs types d’octroi, qui permettent différents types d’accès.

Selon les besoins de votre application, certains types d’octroi conviennent mieux que d’autres. Auth0 offre de nombreux types différents d’authentification et de <Tooltip tip="Flux d’autorisation : octroi d’autorisation (ou workflow) défini dans le framework OAuth 2.0." cta="Consulter le glossaire" href="/docs/fr-CA/glossary?term=authorization+flows">flux d’autorisation</Tooltip>, et vous permet d’indiquer quels types d’octroi sont appropriés au moyen de la propriété `grant_types` de votre application.

Par exemple, si vous souhaitez sécuriser une application mobile, le [flux de code d’autorisation avec clé de preuve pour l’échange de code (PKCE)](/docs/fr-CA/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce) est le plus approprié.

À l’inverse, si vous souhaitez sécuriser une application côté client, comme une application monopage (SPA), et que vous ne transmettez pas de jetons entre des serveurs, le [flux implicite avec envoi de formulaire](/docs/fr-CA/get-started/authentication-and-authorization-flow/implicit-flow-with-form-post) est le plus approprié.

Divers types d’octroi sont valides lors de l’enregistrement des applications. Ils peuvent être répartis dans les catégories suivantes :

* **Octrois conformes aux spécifications** : octrois définis par des spécifications externes et conformes à celles-ci, comme <Tooltip tip="OpenID : norme ouverte d’authentification qui permet aux applications de vérifier l’identité des utilisateurs sans recueillir ni stocker leurs renseignements de connexion." cta="Consulter le glossaire" href="/docs/fr-CA/glossary?term=OpenID">OpenID</Tooltip> Connect (OIDC).
* **Octrois d’extension Auth0** : octrois propres à Auth0 qui respectent le [mécanisme d’extension OAuth](https://tools.ietf.org/html/rfc6749#section-4.5) afin de prendre en charge des clients supplémentaires ou de faire le pont entre <Tooltip tip="OAuth 2.0 : framework d’autorisation qui définit les protocoles et workflows d’autorisation." cta="Consulter le glossaire" href="/docs/fr-CA/glossary?term=OAuth">OAuth</Tooltip> et d’autres frameworks de confiance.
* **Octrois hérités Auth0** : types d’octroi traditionnels pris en charge uniquement pour les clients hérités. Si vous êtes un client hérité, nous vous recommandons fortement de passer à une solution de rechange plus sécuritaire.

<div id="specification-conforming-grants">
  ### Types d’octroi conformes à la spécification
</div>

| Type d’octroi                                  | Description                                                                                                                                     |
| ---------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------- |
| `implicit`                                     | [Octroi implicite](/docs/fr-CA/get-started/authentication-and-authorization-flow/implicit-flow-with-form-post)                                        |
| `authorization_code`                           | [Octroi par code d’autorisation](/docs/fr-CA/get-started/authentication-and-authorization-flow/authorization-code-flow)                               |
| `client_credentials`                           | [Octroi par informations d’identification du client](/docs/fr-CA/get-started/authentication-and-authorization-flow/client-credentials-flow)           |
| `password`                                     | [Octroi par mot de passe du propriétaire de la ressource](/docs/fr-CA/get-started/authentication-and-authorization-flow/resource-owner-password-flow) |
| `refresh_token`                                | [Utiliser des jetons d’actualisation](/docs/fr-CA/secure/tokens/refresh-tokens/use-refresh-tokens)                                                    |
| `urn:ietf:params:oauth:grant-type:device_code` | [Octroi d’autorisation par appareil](/docs/fr-CA/get-started/authentication-and-authorization-flow/device-authorization-flow)                         |

<div id="auth0-extension-grants">
  ### Octrois d’extensions Auth0
</div>

| **Type d’octroi**                                         | **Description**                                                                                                 |
| --------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------- |
| **`http://auth0.com/oauth/grant-type/password-realm`**    | Utilisez un octroi d’extension semblable au Resource Owner Password Grant qui permet d’indiquer un realm précis |
| **`http://auth0.com/oauth/grant-type/mfa-oob`**           | Demande d’octroi OOB pour l’authentification multifactorielle                                                   |
| **`http://auth0.com/oauth/grant-type/mfa-otp`**           | Demande d’octroi OTP pour l’authentification multifactorielle                                                   |
| **`http://auth0.com/oauth/grant-type/mfa-recovery-code`** | Demande d’octroi de récupération pour l’authentification multifactorielle                                       |
| **`http://auth0.com/oauth/grant-type/passwordless/otp`**  | Demande d’octroi de connexion Passwordless intégrée                                                             |

<div id="auth0-legacy-grants">
  ### Auth0 legacy grants
</div>

Les grants Legacy comprennent :

* `http://auth0.com/oauth/legacy/grant-type/ro`
* `http://auth0.com/oauth/legacy/grant-type/ro/jwt-bearer`
* `http://auth0.com/oauth/legacy/grant-type/delegation/refresh_token`
* `http://auth0.com/oauth/legacy/grant-type/delegation/id_token`
* `http://auth0.com/oauth/legacy/grant-type/access_token`

Les types de grant Legacy sont des types de grant traditionnels pris en charge uniquement pour les anciens clients. Si vous êtes un ancien client, nous vous recommandons fortement de passer à une solution de rechange plus sécuritaire.

À compter du 8 juin 2017, toutes les applications se sont vu attribuer une propriété `grant_types` qui doit être renseignée. Afin d’éviter tout changement de fonctionnalité pour les clients Auth0 à ce moment-là, nous avons renseigné la propriété `grant_types` de toutes les applications existantes avec tous les types de grant Auth0 legacy, les types de grant d’extension Auth0 et les types de grant conformes à la spécification.

À partir de ce moment, les nouveaux clients Auth0 ne pouvaient plus ajouter de types de grant Legacy à leurs applications. Les types de grant Legacy ne sont offerts qu’aux anciens clients pendant leur migration vers de nouveaux flux, afin d’éviter des changements incompatibles. Si vous étiez client avant le 8 juin 2017, vous pouvez [activer un type de grant Legacy](/docs/fr-CA/get-started/applications/update-grant-types) à l’aide de l’<Tooltip tip="Auth0 Dashboard: principal produit d’Auth0 pour configurer vos services." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip> ou de la <Tooltip tip="Management API: API d’administration d’Auth0 pour gérer par programmation les locataires, les applications, les utilisateurs et d’autres ressources." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Management+API">Management API</Tooltip> d’Auth0.

Si vous utilisez actuellement un type de grant Legacy, consultez le tableau ci-dessous pour voir laquelle des solutions de rechange sécuritaires vous devriez utiliser à la place. Par exemple, si vous mettez en œuvre l’authentification <Tooltip tip="Passwordless: forme d’authentification qui ne repose pas sur un mot de passe comme premier facteur." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Passwordless">Passwordless</Tooltip>,

utilisez [Universal Login](/docs/fr-CA/authenticate/login/auth0-universal-login) au lieu du point de terminaison `oauth/ro`.

<div id="grant-type-mapping">
  ## Correspondance des types d’autorisation
</div>

Une fois enregistrées, les applications ont accès à différents types d’autorisation selon leur type d’application, plus précisément selon qu’elles sont confidentielles ou publiques. De plus, les applications de confiance de première partie ont accès à des types d’autorisation supplémentaires.

<div id="public-applications">
  ### Applications publiques
</div>

Lorsqu’une application native ou une application à page unique (SPA) est enregistrée dans le Dashboard, elle est automatiquement marquée comme publique, ce qui est indiqué par l’indicateur `token_endpoint_auth_method` défini sur `none`. Par défaut, Auth0 crée des applications publiques avec les `grant_types` suivants activés :

* `implicit`
* `authorization_code`
* `refresh_token`

Les applications natives peuvent aussi utiliser le type d’octroi `device_code`.

Les applications publiques ne peuvent pas utiliser le type d’octroi `client_credentials`. Pour utiliser ce type d’octroi, vous devez configurer l’application comme confidentielle plutôt que publique. Utilisez le endpoint [Update a Client](https://auth0.com/docs/api/management/v2#!/Clients/patch_clients_by_id) de l’Auth0 Management API pour définir `token_endpoint_auth_method` sur `client_secret_post,` `client_secret_basic` ou `private_key_jwt`. Pour en savoir plus, consultez [Application Credentials](/docs/fr-CA/secure/application-credentials).

<div id="confidential-applications">
  ### Applications confidentielles
</div>

Lorsqu&#39;une Regular Web Application ou une application machine-to-machine (M2M) est enregistrée dans l’Auth0 Dashboard, elle est automatiquement considérée comme confidentielle, ce qui est indiqué par l’indicateur `token_endpoint_auth_method` défini sur toute valeur autre que `none`. Par défaut, Auth0 crée des applications confidentielles avec les `grant_types` suivants activés :

* `implicit`
* `authorization_code`
* `refresh_token`
* `client_credentials`

<div id="trusted-first-party-applications">
  ### Applications de première partie de confiance
</div>

Les applications de première partie de confiance ont les mêmes `grant_types` activés que les applications confidentielles, ainsi que les suivants :

* `password`
* `http://auth0.com/oauth/grant-type/password-realm`
* `http://auth0.com/oauth/grant-type/mfa-oob`
* `http://auth0.com/oauth/grant-type/mfa-otp`
* `http://auth0.com/oauth/grant-type/mfa-recovery-code`

Si vous utilisez le Dashboard pour activer ou désactiver ces types d’autorisation, sachez que tous les types d’autorisation `Password` et <Tooltip tip="Authentification multifacteur (MFA) : processus d’authentification de l’utilisateur qui utilise un facteur en plus du nom d’utilisateur et du mot de passe, comme un code envoyé par SMS." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=MFA">MFA</Tooltip> sont activés lorsque vous ajoutez le type d’autorisation `Password` ou `MFA` à votre application. Vous ne pouvez pas les sélectionner individuellement.

<div id="third-party-application-restrictions">
  ## Restrictions relatives aux applications tierces
</div>

Les [applications tierces](/docs/fr-CA/get-started/applications/third-party-applications) prennent en charge un nombre limité de types de grant conformes aux [pratiques exemplaires d’OAuth 2.1](https://datatracker.ietf.org/doc/html/draft-ietf-oauth-v2-1) :

| **Type de grant**                              | **Disponible pour les applications tierces** |
| ---------------------------------------------- | -------------------------------------------- |
| `authorization_code`                           | Oui (PKCE obligatoire)                       |
| `refresh_token`                                | Oui                                          |
| `client_credentials`                           | Oui (clients confidentiels uniquement)       |
| `implicit`                                     | Non                                          |
| `password`                                     | Non                                          |
| `urn:ietf:params:oauth:grant-type:device_code` | Non                                          |
| Grants MFA                                     | Non                                          |
| Grants Legacy                                  | Non                                          |

Pour en savoir plus, consultez [Security Controls for Third-Party Applications](/docs/fr-CA/get-started/applications/third-party-applications/security-controls).
