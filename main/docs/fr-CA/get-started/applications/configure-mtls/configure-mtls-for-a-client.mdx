title: Configurer l’authentification mTLS pour un client
---

Apprenez à configurer l’authentification mTLS pour un client avec la [Management API](#configure-mtls-with-management-api) et l’[Auth0 Dashboard](#configure-mtls-with-auth0-dashboard).

<Tabs>
  <Tab title="Auth0 Dashboard">
    Vous pouvez utiliser Auth0 Dashboard pour configurer mTLS pour un client afin d’activer l’authentification client mTLS auprès du serveur d’autorisation.

    1. Accédez à [Auth0 Dashboard &gt; Applications &gt; Applications](https://manage.auth0.com/#/applications).
    2. Sélectionnez l’application que vous souhaitez utiliser avec mTLS ou [créez une nouvelle application](/docs/fr-CA/get-started/auth0-overview/create-applications).
    3. Sélectionnez l’onglet **Informations d’identification**.
    4. Choisissez la **méthode d’authentification** requise, qui peut être l’une des suivantes :

       * mTLS avec un [certificat auto-signé](/docs/fr-CA/customize/custom-domains/self-managed-certificates)
       * mTLS avec un certificat signé par une autorité de certification
    5. Une fois le type de certificat souhaité sélectionné, vous pouvez :

       * Attribuer une information d’identification (certificat) existante à l’application cliente
       * Ajouter une nouvelle information d’identification en téléversant un certificat
  </Tab>

  <Tab title="Management API">
    Utilisez l&#39;[Auth0 Management API](https://auth0.com/docs/api/management/v2) pour configurer mTLS pour un client.

    <Callout icon="file-lines" color="#0EA5E9" iconType="regular">
      Les exemples suivants utilisent **$management&#95;access&#95;token**, c’est-à-dire un [jeton d’accès à la Management API](/docs/fr-CA/secure/tokens/access-tokens/management-api-access-tokens). Il doit être remplacé par un jeton d’accès qui comprend au moins les scopes suivants :

      * `create:custom_domains`
      * `read:custom_domains`
      * `create:clients`
      * `update:clients`
      * `update:client_credentials`
      * `update:client_keys`
      * `update:tenant_settings`

      Pour en savoir plus sur la façon d’obtenir un jeton d’accès avec les scopes requis, consultez [Obtenir des jetons d’accès](/docs/fr-CA/secure/tokens/access-tokens/get-access-tokens).
    </Callout>

    #### Certificats auto-signés

    Utilisez des certificats auto-signés pour vérifier l&#39;identité du client lors de l&#39;authentification mTLS. Cependant, les certificats auto-signés présentent les limites suivantes

    * Les certificats auto-signés ne sont pas acceptés par certains fournisseurs infonuagiques, comme Amazon.
    * Pour assurer la stabilité de notre plateforme, Auth0 n’autorise que deux certificats enregistrés par client.

    ##### Générer un certificat

    Pour vous authentifier à l&#39;aide du mTLS auto-signé, vous devez créer un nouveau certificat client auto-signé.

    L&#39;exemple de code suivant génère un nouveau certificat auto-signé :

    ```bash wrap lines
    openssl req -x509 -newkey rsa:2048 -keyout key.pem -out cert.pem -sha256 -days 365 -nodes -subj "/C=XX/ST=StateName/L=CityName/O=CompanyName/OU=CompanySectionName/CN=CommonNameOrHostname"
    ```

    ##### Créer un nouveau client

    Pour créer un nouveau client, effectuez une requête POST vers l&#39;endpoint [`/clients`](https://auth0.com/docs/api/management/v2/clients/post-clients) avec le payload suivant :

    * `$client_name`**:** le nom attribué au nouveau client
    * `$credential_name`**:** le nom de la clé publique
    * `$credential_certificate`: le contenu de `$certificate_pem`généré à l’étape précédente

    ```bash lines
    curl --location --request POST 'https://$tenant/api/v2/clients' \
      --header 'Authorization: Bearer $management_access_token' \
      --header 'Content-Type: application/json' \
      --data-raw '{
      "name": "$client_name",
      "app_type": "non_interactive",
      "client_authentication_methods": {
        "self_signed_tls_client_auth": {
          "credentials": [{ 
            "name": "$credential_name", 
            "credential_type": "x509_cert", 
            "pem": "$credential_certificate"
          }]
        }
      },
      "jwt_configuration": {
        "alg": "RS256"
      }
    }'
    ```

    Pour en savoir plus, consultez la documentation de l&#39;API [Créer un client](https://auth0.com/docs/api/management/v2/clients/post-clients).

    ##### Modifier un client existant

    Vous pouvez mettre à jour un client existant pour qu&#39;il accepte l&#39;authentification client mTLS en supprimant la valeur du champ `token_endpoint_auth_method` et en ajoutant des valeurs dans le champ `client_authentication_methods`.

    <Warning>
      Une fois que vous avez configuré votre client pour mTLS, vous ne pourrez plus vous authentifier au moyen du Client Secret, à moins de configurer le `token_endpoint_auth_method` pour qu’il n’utilise plus mTLS. Pour en savoir plus, consultez [Rétablir l’utilisation d’un Client Secret pour un client](#revert-a-client-to-use-a-client-secret).
    </Warning>

    ##### Créer la ressource d&#39;identification

    Après avoir généré un certificat, créez la ressource d&#39;identification :

    ```bash lines
    curl --location --request POST 'https://$tenant/api/v2/clients/$client_id/credentials' \
      --header 'Authorization: Bearer $management_access_token' \
      --header 'Content-Type: application/json' \
      --data-raw '{
      "name": "$credential_name", 
      "credential_type": "x509_cert", 
      "pem": "$credential_certificate"
    }'
    ```

    Auth0 renvoie un ID de credential dans la réponse, que vous devrez utiliser pour associer le credential au client.

    Pour en savoir plus, consultez la documentation de l&#39;API [Créer un credential client](https://auth0.com/docs/api/management/v2/clients/post-credentials).

    ##### Associer le credential au client et désactiver `token_endpoint_auth_method`

    Les identifiants téléversés ne sont pas automatiquement activés pour l&#39;authentification du client. Vous devez mettre à jour l&#39;authentification du client afin d&#39;utiliser le nouveau certificat client auto-signé.

    La requête PATCH suivante définit `token_endpoint_auth_method` sur `null`, désactivant ainsi l&#39;authentification par Client Secret. Elle met également à jour `client_authentication_methods` avec l&#39;ID de credential :

    ```bash lines
    curl --location --request PATCH 'https://$tenant/api/v2/clients/$client_id' \
      --header 'Authorization: Bearer $management_access_token' \
      --header 'Content-Type: application/json' \
      --data-raw '{
      "token_endpoint_auth_method": null, 
      "client_authentication_methods": {
        "self_signed_tls_client_auth": {
          "credentials": [{ "id": $credential.id }]
        }
      }
    }'
    ```

    Une fois cette demande traitée, un Client Secret ne sera plus accepté et les clients devront s&#39;authentifier via mTLS.

    Pour en savoir plus, consultez la documentation de l&#39;API [Update a client](https://auth0.com/docs/api/management/v2/clients/patch-clients-by-id).

    #### Certificats signés par une autorité de certification

    Contrairement aux certificats auto-signés, qui sont générés par le client et ne disposent d&#39;aucune chaîne de confiance, les certificats signés par une autorité de certification (CA) sont considérés comme plus fiables puisqu&#39;ils sont émis par un tiers de confiance. Les certificats signés par une CA sont le seul type de certificat accepté par certains fournisseurs infonuagiques, comme Amazon.

    Les certificats signés par une AC intègrent dans leurs informations d&#39;identité la notion de nom distinctif (DN). Bien que chaque certificat individuel créé par une AC donnée soit unique, ils peuvent partager un DN commun. Lors de l&#39;utilisation de certificats signés par une AC, Auth0 stocke le DN et compare les certificats clients transmis avec les DN enregistrés.

    ##### Générer un certificat

    La méthode de génération d&#39;un client certificate signé par une AC dépend fortement de l&#39;Infrastructure à clé publique et ne relève pas du cadre de ce document. Nous recommandons de générer au moins une [paire de clés RSA de 2048 bits](/docs/fr-CA/secure/application-credentials/generate-rsa-key-pair).

    ##### Créer un nouveau client

    Pour créer un client, effectuez une requête POST vers l&#39;endpoint [`/clients`](https://auth0.com/docs/api/management/v2/clients/post-clients) avec le payload suivant :

    * `$client_name`**:** le nom du nouveau client
    * `$credential_name`**:** le nom de la clé publique
    * `$credential_certificate`: le contenu de `$certificate_pem` généré par l’autorité de certification

    ```bash lines
    curl --location --request POST 'https://$tenant/api/v2/clients' \
      --header 'Authorization: Bearer $management_access_token' \
      --header 'Content-Type: application/json' \
      --data-raw '{
      "name": "$client_name",
      "app_type": "non_interactive",
      "client_authentication_methods": {
        "tls_client_auth": {
          "credentials": [{ 
            "name": "$credential_name", 
            "credential_type": "cert_subject_dn", 
            "pem": "$credential_certificate"
          }]
        }
      },
      "jwt_configuration": {
        "alg": "RS256"
      }
    }'
    ```

    Au lieu de transmettre le fichier PEM complet, vous pouvez également transmettre le DN du subject. Le DN du subject doit correspondre au Distinguished Name (DN) extrait des certificats client envoyés lors du handshake mTLS.

    <Callout icon="file-lines" color="#0EA5E9" iconType="regular">
      L’extraction du DN du sujet peut varier d’un écosystème à l’autre. Le moyen le plus fiable de garantir que le serveur d’autorisation correspondra au DN du sujet consiste à téléverser le fichier PEM complet.
    </Callout>

    La requête POST suivante crée un nouveau client avec le DN du sujet :

    ```bash lines
    curl --location --request POST 'https://$tenant/api/v2/clients' \
      --header 'Authorization: Bearer $management_access_token' \
      --header 'Content-Type: application/json' \
      --data-raw '{
      "name": "$client_name",
      "app_type": "non_interactive",
      "client_authentication_methods": {
        "tls_client_auth": {
          "credentials": [{ 
            "name": "$credential_name", 
            "credential_type": "cert_subject_dn", 
            "subject_dn": "C=XX\nST=StateName\nL=CityName\nO=CompanyName\nOU=CompanySectionName\nCN=CommonNameOrHostname"
          }]
        }
      },
      "jwt_configuration": {
        "alg": "RS256"
      }
    }'
    ```

    Pour en savoir plus, consultez la documentation de l&#39;API [Créer un client](https://auth0.com/docs/api/management/v2/clients/post-clients).

    ##### Mettre à jour un client existant

    Si vous ne souhaitez pas créer un nouveau client pour utiliser mTLS, vous pouvez mettre à jour un client existant afin qu&#39;il accepte l&#39;authentification client mTLS. Pour ce faire, supprimez toute valeur dans le champ `token_endpoint_auth_method` et ajoutez des valeurs dans le champ `client_authentication_methods`.

    <Warning>
      Une fois que vous avez configuré votre client pour mTLS, vous ne pourrez plus vous authentifier à l’aide du Client Secret, à moins de configurer `token_endpoint_auth_method` pour ne plus utiliser mTLS. Pour en savoir plus, consultez [Rétablir l’utilisation d’un Client Secret pour un client](#revert-a-client-to-use-a-client-secret).
    </Warning>

    ##### Créer la ressource d&#39;identification

    Une fois que vous avez [généré une paire de clés](/docs/fr-CA/secure/application-credentials/generate-rsa-key-pair) exclusivement pour mTLS, créez la ressource d&#39;identification. Effectuez la requête POST suivante vers le point de terminaison `/clients` :

    ```bash lines
    curl --location --request POST 'https://$tenant/api/v2/clients/$client_id/credentials' \
      --header 'Authorization: Bearer $management_access_token' \
      --header 'Content-Type: application/json' \
      --data-raw '{
      "name": "$credential_name", 
      "credential_type": "cert_subject_dn", 
      "pem": "$credential_certificate"
    }'
    ```

    Au lieu de transmettre le fichier PEM complet, vous pouvez transmettre le DN du subject. Le DN du subject doit correspondre au Distinguished Name (DN) extrait des certificats client envoyés lors du handshake mTLS.

    L&#39;exemple de code suivant crée la ressource d&#39;authentification à l&#39;aide du Subject DN :

    ```bash lines
    curl --location --request POST 'https://$tenant/api/v2/clients/$client_id/credentials' \
      --header 'Authorization: Bearer $management_access_token' \
      --header 'Content-Type: application/json' \
      --data-raw '{
      "name": "$credential_name", 
      "credential_type": "cert_subject_dn", 
      "subject_dn": "C=XX\nST=StateName\nL=CityName\nO=CompanyName\nOU=CompanySectionName\nCN=CommonNameOrHostname"
    }'
    ```

    Quelle que soit la méthode utilisée, n&#39;oubliez pas que l&#39;ID d&#39;identifiant renvoyé dans la réponse est nécessaire pour associer l&#39;identifiant au client.

    Pour en savoir plus, consultez la documentation de l&#39;API [Créer un credential client](https://auth0.com/docs/api/management/v2/clients/post-credentials).

    ##### Associer le credential au client et désactiver `token_endpoint_auth_method`

    Bien que nous ayons créé le credential, nous ne l&#39;avons pas encore associé au client.

    Pour ce faire, mettez à jour `client_authentication_methods` en effectuant la requête PATCH suivante vers le point de terminaison `/clients`. Dans la même requête, définissez `token_endpoint_auth_method` sur `null` :

    ​​

    ```bash lines
    curl --location --request PATCH 'https://$tenant/api/v2/clients/$client_id' \
      --header 'Authorization: Bearer $management_access_token' \
      --header 'Content-Type: application/json' \
      --data-raw '{
      "token_endpoint_auth_method": null, 
      "client_authentication_methods": {
        "tls_client_auth": {
          "credentials": [{ "id": $credential.id }]
        }
      }
    }'
    ```

    Une fois cette requête traitée, le client ne peut être authentifié qu&#39;avec mTLS.

    Pour en savoir plus, consultez la documentation de l&#39;API [Update a client](https://auth0.com/docs/api/management/v2/clients/patch-clients-by-id).

    #### Rétablir un client pour utiliser un secret client

    Pour rétablir la configuration de votre client afin de vous authentifier à l&#39;aide d&#39;un Secret client, désactivez `client_authentication_methods` et réactivez `token_endpoint_auth_method` avec la méthode d&#39;authentification souhaitée.

    Dans la requête PATCH suivante, définissez `token_endpoint_auth_method` sur `client_secret_post` pour réactiver l&#39;authentification par Client Secret :

    ```bash lines
    curl --location --request PATCH 'https://$tenant/api/v2/clients/$client_id' \
      --header 'Authorization: Bearer $management_access_token' \
      --header 'Content-Type: application/json' \
      --data-raw '{
      "token_endpoint_auth_method": "client_secret_post", 
      "client_authentication_methods": null
    }'
    ```

    Pour en savoir plus, consultez la documentation de l&#39;API [Update a client](https://auth0.com/docs/api/management/v2/clients/patch-clients-by-id).
