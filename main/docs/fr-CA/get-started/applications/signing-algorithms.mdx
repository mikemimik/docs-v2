title: Algorithmes de signature
---

Les <Tooltip tip="Signing Algorithm : algorithme utilisé pour signer numériquement des jetons afin de garantir que le jeton n’a pas été altéré." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Signing+algorithms">algorithmes de signature</Tooltip> servent à signer les jetons émis pour votre application ou votre API. Une signature fait partie d’un [JSON Web Token (JWT)](/docs/fr-CA/secure/tokens/json-web-tokens) et sert à vérifier que l’expéditeur du jeton est bien celui qu’il prétend être, et à s’assurer que le message n’a pas été modifié en cours de route.

Vous pouvez choisir parmi les algorithmes de signature suivants :

* **RS256 (signature RSA avec SHA-256)** : un algorithme asymétrique, ce qui signifie qu’il y a deux clés : une clé publique et une clé privée qui doit être gardée secrète. Auth0 possède la clé privée utilisée pour générer la signature, et le consommateur du <Tooltip tip="JSON Web Token (JWT) : format standard de l’ID Token (et souvent du jeton d’accès) utilisé pour représenter des claims de façon sécurisée entre deux parties." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=JWT">JWT</Tooltip> récupère une clé publique à partir des points de terminaison de métadonnées fournis par Auth0 et l’utilise pour valider la signature du JWT.
* **HS256 (HMAC avec SHA-256)** : un algorithme symétrique, ce qui signifie qu’il n’y a qu’une seule clé privée qui doit être gardée secrète, et qu’elle est partagée entre les deux parties. Comme la même clé sert à la fois à générer et à valider la signature, il faut veiller à ce qu’elle ne soit pas compromise. Cette clé privée (ou ce secret) est créée lorsque vous enregistrez votre application (<Tooltip tip="Client Secret : secret utilisé par un client (application) pour s’authentifier auprès de l’Authorization Server; il ne devrait être connu que du client et de l’Authorization Server et doit être suffisamment aléatoire pour ne pas pouvoir être deviné." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=client+secret">secret client</Tooltip>) ou votre API (secret de signature) et que vous choisissez l’algorithme de signature HS256.
* **PS256 (signature RSA avec SHA-256)** : un algorithme asymétrique, ce qui signifie qu’il y a deux clés : une clé publique et une clé privée qui doit être gardée secrète. Auth0 possède la clé privée utilisée pour générer la signature, et le consommateur du JWT récupère une clé publique à partir des points de terminaison de métadonnées fournis par Auth0 et l’utilise pour valider la signature du JWT. Contrairement à RS256, le même en-tête et le même payload JWT généreront une signature différente chaque fois.

La pratique la plus sécuritaire, et celle que nous recommandons, est d’utiliser RS256 parce que :

* Avec RS256, vous avez l’assurance que seul le détenteur de la clé privée (Auth0) peut signer des jetons, tandis que n’importe qui peut vérifier si le jeton est valide à l’aide de la clé publique.
* Avec RS256, si la clé privée est compromise, vous pouvez effectuer une rotation des clés sans avoir à redéployer votre application ou votre API avec le nouveau secret (ce que vous devriez faire avec HS256).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Si vous configurez l’authentification Private Key JWT avec une Enterprise connection Okta ou OpenID Connect (OIDC), vous devrez peut-être choisir des [algorithmes de signature supplémentaires](/docs/fr-CA/authenticate/enterprise-connections/private-key-jwt-client-auth#configure-private-key-jwt-client-authentication).
</Callout>
