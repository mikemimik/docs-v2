<Card title="Avant de commencer">
  * Pour des raisons de sécurité, l’URL d’origine de votre application doit figurer parmi les URL autorisées. Si vous ne l’avez pas déjà ajoutée aux **Allowed Callback URLs** de votre application, vous devrez l’ajouter à la liste des **Origines autorisées (CORS)**.
  * Assurez-vous que **Allowed Web Origins**, dans **Settings** de votre application, est défini sur le domaine qui effectue la requête. Les URL peuvent contenir des caractères génériques pour les sous-domaines, mais ne peuvent pas contenir de chemins relatifs après l’URL du domaine. Pour en savoir plus, consultez [Placeholders for Subdomains](/docs/fr-CA/get-started/applications/wildcards-for-subdomains).
  * Si vous n’activez pas [Domaines personnalisés](/docs/fr-CA/customize/custom-domains), vous devrez créer une page de vérification qui utilise auth0.js comme solution de rechange pour l’authentification inter-origines.
</Card>

Le partage des ressources entre origines multiples (CORS) est un mécanisme qui permet à une application de charger des données (ressources) à partir d’un domaine (origine) différent de celui où elle est hébergée. Une politique CORS constitue une exception explicite à la politique de même origine, courante dans les applications web.

Par exemple, si vous vouliez que votre application (app.mydomain.com) récupère en arrière-plan, à l’aide d’Ajax, une police de Google (fonts.google.com), vous devriez configurer CORS.

<div id="configure-cross-origin-authentication">
  ## Configurer l’authentification inter-origines
</div>

1. Accédez à [Dashboard &gt; Applications &gt; Applications](https://manage.auth0.com/#/applications), puis cliquez sur le nom de l’application pour l’ouvrir.
2. Sous **Authentification inter-origines**, activez **Autoriser l’authentification inter-origines**.
3. Repérez **Origines autorisées (CORS)**, puis saisissez l’URL d’origine de votre application. Pour en savoir plus sur les origines, consultez [Origin dans la documentation Web MDN de Mozilla](https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers/Origin).
4. Cliquez sur **Enregistrer les modifications**.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Si vous n’avez pas besoin d’utiliser CORS pour votre application, assurez-vous que **Autoriser l’authentification inter-origines** est désactivé.
</Callout>

<div id="create-cross-origin-verification-page">
  ## Créer une page de vérification cross-origin
</div>

Il arrive que les cookies tiers ne soient pas disponibles. Certaines versions de navigateurs ne prennent pas en charge les cookies tiers et, même lorsqu’elles le font, ils peuvent être désactivés dans les paramètres de l’utilisateur.

Pour les [navigateurs pris en charge](#browser-testing-support), vous pouvez utiliser la méthode `crossOriginVerification` du [SDK Auth0.js](/docs/fr-CA/libraries/auth0js) dans votre application, sur une page dédiée, pour gérer les cas où les cookies tiers sont désactivés.

Pour les navigateurs qui ne sont pas pris en charge, comme Chrome, Opera et Safari, l’authentification inter-origines ne fonctionnera pas lorsque les cookies tiers sont désactivés, sauf si vous activez [les Domaines personnalisés](/docs/fr-CA/customize/custom-domains).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Dans **Safari**, ce réglage s’appelle &quot;Prevent cross-site tracking&quot; et utilise la [prévention intelligente du suivi](https://webkit.org/blog/7675/intelligent-tracking-prevention/). Malheureusement, cela empêche aussi les cookies tiers d’être utiles dans les scénarios d’authentification. Voici un exemple de son incidence sur le [renouvellement de token dans Safari](https://support.auth0.com/center/s/article/troubleshoot-auth0-token-renewal-issues-in-safari-with-itp-enabled).
</Callout>

1. Créez une page dans votre application qui instancie `WebAuth` à partir d’[Auth0.js](/docs/fr-CA/libraries/auth0js). Appelez immédiatement `crossOriginVerification`. Vous pouvez donner à cette page le nom de votre choix.

export const codeExample = `   <!-- callback-cross-auth.html -->

<head>
  <script src="https://cdn.auth0.com/js/auth0/9.11/auth0.min.js"></script>
  <script type="text/javascript">
    var auth0Client = new auth0.WebAuth({
      clientID: '{yourClientId}',
      domain: '{yourDomain}'
    });
    auth0Client.crossOriginVerification();
  </script>
</head>
`;

<AuthCodeBlock children={codeExample} language="text" />

Lorsque les cookies tiers ne sont pas disponibles, Auth0.js affiche un iframe pour appeler un autre flux de vérification inter-origines.
2. Accédez à [Dashboard &gt; Applications &gt; Applications](https://manage.auth0.com/#/applications), puis sélectionnez l’application à consulter.
3. Sous **Authentification inter-origines**, ajoutez l’URL de la page de rappel que vous avez créée dans le champ **URL de secours pour la vérification inter-origines**.

<Warning>
  Pour les environnements de production, vérifiez que l’URL de la page ne pointe pas vers `localhost`. La page doit se trouver sur le même domaine que celui où le formulaire de connexion intégré est hébergé et doit utiliser le protocole `https`.
</Warning>

4. Cliquez sur **Save Changes**.

Pour en savoir plus, consultez l’[exemple d’authentification inter-origines sur GitHub](https://github.com/auth0/lock/blob/master/support/callback-cross-auth.html).

<div id="error-codes-and-descriptions">
  ## Codes d&#39;erreur et descriptions
</div>

<Warning>
  Les descriptions d&#39;erreur sont destinées à être facilement compréhensibles. Elles peuvent être modifiées à tout moment et ne doivent pas être analysées par du code.
</Warning>

Lorsque Auth0.js (ou Lock) est utilisé pour le login intégré, il appelle l&#39;endpoint `/co/authenticate`, qui peut renvoyer les erreurs suivantes :

| Statut | Code                          | Description                                                                                                                                                                                             |
| ------ | ----------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `400`  | `invalid_request`             | Corps de la requête invalide. Tous les paramètres suivants, et uniquement ceux-ci, sont requis : client&#95;id, credential&#95;type, username, otp, realm.                                              |
| `400`  | `unsupported_credential_type` | Paramètre de type d&#39;identifiant inconnu.                                                                                                                                                            |
| `400`  | `invalid_request`             | Realm inconnu : non-existent-connection.                                                                                                                                                                |
| `401`  | `unauthorized_client`         | Login interorigine non autorisé.                                                                                                                                                                        |
| `401`  | `password_leaked`             | Cette tentative de login a été bloquée parce que le mot de passe que vous utilisez a déjà été divulgué lors d&#39;une fuite de données (pas dans cette application).                                    |
| `403`  | `access_denied`               | Courriel ou mot de passe incorrect.                                                                                                                                                                     |
| `403`  | `access_denied`               | Erreur d&#39;authentification                                                                                                                                                                           |
| `403`  | `blocked_user`                | Utilisateur bloqué                                                                                                                                                                                      |
| `429`  | `too_many_attempts`           | Votre compte a été bloqué après plusieurs tentatives de login consécutives. Nous vous avons envoyé une notification selon votre méthode de contact privilégiée avec des instructions pour le débloquer. |
| `429`  | `too_many_attempts`           | Nous avons détecté un comportement de login suspect et toute nouvelle tentative sera bloquée. Veuillez communiquer avec l&#39;administrateur.                                                           |

De plus, vous pouvez aussi obtenir une erreur générique `403` sans propriété `error` ou `error_description`. Le corps de la réponse inclurait simplement quelque chose comme ceci :

`Origin https://test.app is not allowed.`

<div id="browser-testing-support">
  ## Compatibilité des navigateurs pour les tests
</div>

Les navigateurs suivants peuvent utiliser l’authentification inter-origines lorsque les cookies tiers sont désactivés :

* Microsoft Internet Explorer

<Card title="Attributs des cookies SameSite">
  Auparavant, les options de l’[attribut de cookie `samesite`](/docs/fr-CA/manage-users/cookies/samesite-cookie-attribute-changes) étaient `true`, `false`, `strict` ou `lax`. Si vous ne définissiez pas cet attribut manuellement, Auth0 utilisait la valeur par défaut `false`.

  À partir de février 2020, Google Chrome v80 a modifié la façon dont il gère les cookies, et Auth0 a apporté les changements suivants en conséquence :

  * Les cookies sans attribut `samesite` seront définis sur `lax`.
  * Les cookies avec `sameSite=none` doivent être sécurisés, sinon ils ne peuvent pas être enregistrés dans le cookie jar du navigateur.

  Ces changements visent à améliorer la sécurité et à aider à atténuer les attaques de falsification de requête intersite (CSRF).
</Card>
