Les documents de [découverte OpenID Connect (OIDC)](https://openid.net/specs/openid-connect-discovery-1_0-final.html#RFC5785) contiennent des métadonnées sur le <Tooltip tip="Fournisseur d’identité (IdP) : service qui stocke et gère les identités numériques." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=identity+provider">fournisseur d’identité</Tooltip> (IdP). L’ajout de la découverte à votre SDK pour diriger votre application vers le endpoint `./wellknown` afin d’obtenir des renseignements sur votre IdP peut vous aider à configurer votre intégration avec l’IdP.

L’intégration de la découverte OIDC à votre SDK fournit :

* Les endpoints exposés de l’IdP
* Les [claims et scopes pris en charge par OIDC](/docs/fr-CA/get-started/apis/scopes/openid-connect-scopes) standard (cela exclut les [claims personnalisés](/docs/fr-CA/secure/tokens/json-web-tokens/create-custom-claims) et les scopes définis dans votre tenant)
* Les fonctionnalités prises en charge par l’IdP

Vous pouvez configurer des applications à l’aide des documents de découverte [OpenID Connect (OIDC)](https://openid.net/specs/openid-connect-discovery-1_0.html) qui se trouvent à l’adresse suivante : `https://{yourDomain}/.well-known/openid-configuration`.

  ### Exemple de mise en œuvre
</div>

Par exemple, voici comment configurer le middleware OIDC pour Katana v3 (OWIN) :

1. Installez le package NuGet : **Microsoft.Owin.Security.OpenIdConnect** (v3.x.x)
2. Accédez à `App_Start\Startup.Auth.cs` et remplacez votre mise en œuvre par ce qui suit :

export const codeExample2 = `   app.UseCookieAuthentication(new CookieAuthenticationOptions
{
    AuthenticationType = CookieAuthenticationDefaults.AuthenticationType
});

app.UseOpenIdConnectAuthentication(new OpenIdConnectAuthenticationOptions
{
    Authority = "https://{yourDomain}/",
    ClientId = "{yourClientId}",
    SignInAsAuthenticationType = CookieAuthenticationDefaults.AuthenticationType,
    ResponseType = "token",
    Notifications = new OpenIdConnectAuthenticationNotifications
    {
        // FACULTATIF : vous pouvez lire/modifier les claims ajoutées à partir du JWT
        SecurityTokenValidated = context =>
        {
            // ajouter le jeton d’accès Auth0 en tant que claim
            var accessToken = context.ProtocolMessage.AccessToken;
            if (!string.IsNullOrEmpty(accessToken))
            {
                context.AuthenticationTicket.Identity.AddClaim(new Claim("access_token", accessToken));
            }
            return Task.FromResult(0);
        }
    }
});
`;

<div id="rsa-algorithm-for-jwts">
  ## Algorithme RSA pour les JWTs
</div>

Le middleware OIDC ne prend pas en charge les <Tooltip tip="JSON Web Token (JWT) : format standard de ID Token (et souvent de Access Token) utilisé pour représenter des claims de manière sécurisée entre deux parties." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=JWTs">JWTs</Tooltip> signés avec des clés symétriques. Assurez-vous de configurer votre application pour utiliser l’algorithme RSA avec des clés publique et privée.

1. Accédez à [Dashboard &gt; Settings](https://manage.auth0.com/#/applications/\{YOUR_AUTH0_CLIENT_ID}/settings).
2. Faites défiler jusqu’à **Advanced Settings**.
3. Sous l’onglet **OAuth**, définissez `RS256` comme **Json Web Token(JWT) Signature Algorithm** et cliquez sur **Save**.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Si vous avez une Enterprise Connection OIDC, vous pouvez choisir parmi des [algorithmes de signature supplémentaires](/docs/fr-CA/authenticate/enterprise-connections/private-key-jwt-client-auth#configure-private-key-jwt-client-authentication) pour Private Key JWT.
</Callout>

Avec ce réglage, Auth0 émettra des JWTs signés avec votre clé de signature privée. Votre application les vérifiera à l’aide de votre clé de signature publique.

<div id="configure-applications-with-oauth-20-authorization-server-metadata">
  ## Configurer les applications avec les métadonnées du serveur d’autorisation OAuth 2.0
</div>

Si votre application ou votre SDK fait référence à la spécification de métadonnées [OAuth RFC-8414](https://www.rfc-editor.org/rfc/rfc8414) du <Tooltip tip="Serveur d’autorisation : serveur centralisé qui contribue à définir les limites de l’accès d’un utilisateur. Par exemple, votre serveur d’autorisation peut contrôler les données, les tâches et les fonctionnalités accessibles à un utilisateur." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Authorization+Server">serveur d’autorisation</Tooltip>, vous pouvez utiliser l’alias <Tooltip tip="OAuth 2.0 : framework d’autorisation qui définit les protocoles et les workflows d’autorisation." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=OAuth">OAuth</Tooltip> pour récupérer les métadonnées de l’IdP : `/.well-known/oauth-authorization-server`. Par exemple, le [serveur Auth0 Model Context Protocol](/docs/fr-CA/get-started/auth0-mcp-server) recommande à toutes les applications OAuth de faire référence à la spécification de métadonnées du serveur d’autorisation OAuth.
