Dans Auth0, vous pouvez contrôler la façon dont les applications accèdent à vos API à l’aide des [politiques d’accès des applications aux API](/docs/fr-CA/get-started/apis/api-access-policies-for-applications) et des client grants.

Un client grant fournit un accès granulaire à une API pour une application. Il associe :

* Une API identifiée par son `audience` ou son identificateur unique.
* Une application identifiée par son `client_id`.
* Une liste de permissions, comme des scopes et/ou des `authorization_details_types`, que l’application est autorisée à demander pour l’audience spécifiée.

Pour en savoir plus sur la liste des attributs que vous pouvez définir dans un client grant, consultez [Attributs d’un client grant](#client-grant-attributes). Pour savoir comment définir et gérer des client grants, consultez [Créer un client grant](#create-client-grant).

<div id="application-api-access-policies-and-client-grants">
  ## Politiques d’accès des applications à l’API et client grants
</div>

Lorsque vous configurez la [politique d’accès des applications](/docs/fr-CA/get-started/apis/api-access-policies-for-applications) d’une API à `require_client_grant`, seules les applications pour lesquelles un client grant a été défini peuvent obtenir un jeton d’accès pour l’API. Le client grant établit les permissions maximales qu’une application peut demander à l’API, conformément au principe du moindre privilège. Par conséquent, Auth0 recommande d’utiliser `require_client_grant` lors de la configuration de la politique d’accès des applications d’une API.

<div id="example-social-media-api">
  ### Exemple : API de médias sociaux
</div>

Pour illustrer comment les client grants appliquent le principe du moindre privilège, supposons que vous ayez une API de médias sociaux avec les permissions suivantes : `read:posts`, `write:posts`, `read:friends` et `delete:posts`. Vous créez une application et définissez un client grant avec les permissions `read:posts` et `write:posts`.

Ce client grant constitue maintenant une limite stricte. Même si l’API de médias sociaux comporte d’autres permissions, votre application ne pourra jamais demander ni se voir accorder `read:friends` ou `delete:posts`.

<div id="user-delegated-access-vs-client-access">
  ## Accès délégué par l’utilisateur vs. accès client
</div>

Dans le cas de l’accès utilisateur et de l’accès client, les client grants définissent l’ensemble final des permissions qui contrôlent l’accès d’une application à une API. L’attribut `subject_type` du client grant détermine le type d’accès applicatif autorisé pour une API.

Une application peut avoir jusqu’à deux client grants pour une même API :

* Lorsque vous définissez `subject_type` sur `client`, vous définissez ses permissions machine-à-machine.
* Lorsque vous définissez `subject_type` sur `user`, vous définissez ses permissions pour agir au nom de l’utilisateur.

Le tableau suivant explique comment les client grants contrôlent l’accès des applications aux API en fonction du type de flux d’accès :

| Type d’accès                                                                     | Attribut subject&#95;type               | Description                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                            |
| -------------------------------------------------------------------------------- | --------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Accès avec les informations d’identification du client (accès machine-à-machine) | Définissez `subject_type` sur `client`. | Le client grant autorise directement l’application à accéder à l’API pour son propre compte plutôt qu’au nom de l’utilisateur final. Les permissions que vous définissez dans le client grant sont celles que l’application est autorisée à recevoir dans l’access token.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                              |
| Accès délégué par l’utilisateur                                                  | Définissez `subject_type` sur `user`.   | Le client grant définit les permissions maximales que l’application peut demander à l’API. Les permissions finales dans l’access token émis à l’application au nom de l’utilisateur correspondent à l’intersection des permissions :<br /><ul><li>Demandées par l’application</li><li>Autorisées par le client grant</li><li>Autorisées par les [politiques de contrôle d’accès basé sur les rôles](/docs/fr-CA/manage-users/access-control/rbac) pour l’utilisateur</li><li>[Acceptées par l’utilisateur final](/docs/fr-CA/get-started/applications/third-party-applications/user-consent-and-third-party-applications), s’il y a lieu.</li></ul><br />Pour en savoir plus sur les flux d’accès délégué par l’utilisateur, consultez [Flux d’authentification et d’autorisation](/docs/fr-CA/get-started/authentication-and-authorization-flow). Les flux d’accès délégué par l’utilisateur n’incluent pas le Client Credentials Flow. |

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Vous pouvez modifier les scopes finaux accordés par le serveur d’autorisation à l’application ou à l’utilisateur à l’aide de [Actions](/docs/fr-CA/customize/actions).
</Callout>

<div id="client-grant-attributes">
  ## Attributs du client grant
</div>

Un client grant comporte plusieurs attributs que vous pouvez définir pour configurer l&#39;accès de l&#39;application aux API à l&#39;aide de l&#39;Auth0 Management API :

| Attribut                      | Description                                                                                                                                                                                                                                                                                                                                                                                                                                                                                        |
| ----------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `id`                          | Identifiant unique du client grant.                                                                                                                                                                                                                                                                                                                                                                                                                                                                |
| `audience`                    | Identifiant unique de l&#39;API à laquelle le client grant s&#39;applique.                                                                                                                                                                                                                                                                                                                                                                                                                         |
| `client_id`                   | ID unique de l&#39;application à laquelle l&#39;accès est accordé.                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| `scopes`                      | Tableau de chaînes représentant les permissions que l&#39;application peut demander.                                                                                                                                                                                                                                                                                                                                                                                                               |
| `authorization_details_types` | Tableau de chaînes représentant les types de données d&#39;autorisation enrichies que l&#39;application peut demander. Cet attribut ne peut être spécifié que pour les flux d&#39;accès délégué par l&#39;utilisateur.                                                                                                                                                                                                                                                                             |
| `subject_type`                | Le type d&#39;accès d&#39;application permis par le client grant :<br /><ul><li>`user` : utilisé pour l&#39;accès délégué par l&#39;utilisateur, ce qui correspond à tous les flux qui génèrent un token associé à un utilisateur final.</li><li>`client` : utilisé pour l&#39;accès machine, ce qui correspond au Client Credentials Flow.</li></ul>                                                                                                                                              |
| `allow_all_scopes`            | Booléen. Indique si tous les scopes définis pour l&#39;API sont autorisés pour l&#39;application. Pour cette API, les scopes définis ultérieurement sont automatiquement autorisés.                                                                                                                                                                                                                                                                                                                |
| `organization_usage`          | Détermine comment l&#39;application peut utiliser Organizations lorsqu&#39;elle accède à l&#39;API au moyen du Client Credentials Flow. Les valeurs possibles sont : `deny`, `allow` ou `require`.<br /><br />Pour en savoir plus sur les paramètres d&#39;Organization, consultez [Organizations for M2M Applications: Define Organization Behavior](/docs/fr-CA/manage-users/organizations/organizations-for-m2m-applications/configure-your-application-for-m2m-access#define-organization-behavior). |
| `allow_any_organization`      | Détermine si l&#39;application peut accéder à n&#39;importe quelle Organization lorsqu&#39;elle utilise le Client Credentials Flow.<br /><br />Pour en savoir plus sur les paramètres d&#39;Organization, consultez [Organizations for M2M Applications: Define Organization Behavior](/docs/fr-CA/manage-users/organizations/organizations-for-m2m-applications/configure-your-application-for-m2m-access#define-organization-behavior).                                                                |

<div id="create-client-grant">
  ## Créer une client grant
</div>

Vous pouvez créer :

* [Permissions par application](#per-application-permissions) : attribuez des permissions précises à chaque application de votre tenant.
* [Permissions par défaut pour les applications tierces](#default-permissions-for-third-party-applications) : attribuez des permissions par défaut à toutes les applications tierces de votre tenant.

Lorsque les deux existent pour une même API, les permissions par application priment sur les permissions par défaut des applications tierces.

<div id="per-application-permissions">
  ### Autorisations par application
</div>

<Tabs>
  <Tab title="Auth0 Dashboard">
    Pour configurer les autorisations par application dans l’Auth0 Dashboard :

    1. Accédez à [Dashboard &gt;  Applications &gt; APIs](https://manage.auth0.com/#/apis) et sélectionnez l’API pour laquelle vous souhaitez configurer l’accès des applications.
    2. Ouvrez l’onglet **Settings** et faites défiler jusqu’à **Application Access Policy**.
       * Configurez **User-Delegated Access** à **No apps allowed**, **Per-app authorization** ou **All apps allowed**.
         * **No apps allowed** : aucune application ne peut obtenir de jeton d’accès pour l’API.
         * **Per-app authorization** : seules les applications pour lesquelles un client grant est défini peuvent obtenir un jeton d’accès pour l’API.
         * **All apps allowed** : n’importe quelle application de votre tenant peut obtenir un jeton d’accès pour l’API.
       * Configurez **Client Access** à **Per-app authorization** ou **All apps allowed**.
         * **Per-app authorization** : seules les applications pour lesquelles un client grant est défini peuvent obtenir un jeton d’accès pour l’API.
         * **All apps allowed** : n’importe quelle application de votre tenant peut obtenir un jeton d’accès pour l’API.
    3. Sélectionnez **Save** pour enregistrer les paramètres de **Application Access Policy**.

    <Frame>![Paramètres de l’API dans le Dashboard pour la politique d’accès des applications](/docs/images/third-party-applications/application_access_policy.png)</Frame>

    Pour les autorisations par application, vous devez autoriser individuellement l’accès à l’API pour chaque application.

    1. Accédez à **Applications &gt; APIs** et sélectionnez l’API.
    2. Ouvrez l’onglet **Application Access**.
    3. Faites défiler jusqu’à l’application, sélectionnez **Edit**, puis **Grant Access** pour **User-Delegated Access** et/ou **Client Access**. Sélectionnez ensuite les autorisations voulues.
    4. Sélectionnez **Save**.

    <Frame>![Paramètres de l’API dans le Dashboard pour accorder l’accès à l’API à une application](/docs/images/third-party-applications/grant-api-access.png)</Frame>
  </Tab>

  <Tab title="Management API">
    Envoyez une requête [`POST`](https://auth0.com/docs/api/management/v2/client-grants/post-client-grants) au point de terminaison `/client-grants` avec le corps de la requête suivant :

    ```bash lines
    curl --location 'https://{yourDomain}/api/v2/client-grants' \
    --header 'Content-Type: application/json' \
    --header 'Authorization: Bearer {YOUR_MANAGEMENT_API_TOKEN}' \
    --data '{
        "client_id": "{CLIENT_ID}",
        "audience": "https://api.my-service.com",
        "scope": [
            "read:item"
        ],
        "authorization_details_types":["payment"],
        "subject_type": "user"
    }'
    ```
  </Tab>
</Tabs>

<div id="default-permissions-for-third-party-applications">
  ### Permissions par défaut pour les applications tierces
</div>

Les [applications tierces](/docs/fr-CA/get-started/applications/third-party-applications) exigent toujours une client grant explicite pour accéder à une API, même lorsque la [politique d’accès](/docs/fr-CA/get-started/apis/api-access-policies-for-applications) de l’API est définie sur **Allow All**. Pour simplifier la gestion lorsque vous avez de nombreuses applications tierces ou que vous utilisez [Dynamic Client Registration](/docs/fr-CA/get-started/applications/dynamic-client-registration), configurez des autorisations ou des permissions par défaut qui s’appliquent automatiquement à toutes les applications tierces.

Une client grant tierce par défaut utilise l’attribut `default_for` au lieu d’un `client_id`. Vous pouvez aussi définir des permissions par application en créant une client grant avec un `client_id` précis. Lorsque les deux existent pour la même API, les permissions par application ont préséance.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Les API système (la Management API, la My Account API et d’autres) ne prennent pas en charge les client grants tierces par défaut. Les applications tierces ne peuvent pas se voir accorder l’accès aux API système.
</Callout>

Les attributs `default_for` et `client_id` s’excluent mutuellement. Chaque client grant doit en préciser exactement un.

Pour savoir comment configurer les politiques d’accès aux API pour les applications tierces, consultez [Configure Third-Party Applications](/docs/fr-CA/get-started/applications/third-party-applications/configure-third-party-applications#configure-api-access-policies).

<Tabs>
  <Tab title="Auth0 Dashboard">
    Pour configurer les permissions par défaut pour les applications tierces à l’aide de l’Auth0 Dashboard :

    1. Accédez à [Dashboard &gt;  Applications &gt; APIs](https://manage.auth0.com/#/apis) et sélectionnez l’API pour laquelle vous souhaitez configurer l’accès de l’application.
    2. Allez à l’onglet **Settings** et faites défiler jusqu’à **Default Permissions for Third-Party Applications**.
       * Configurez **User-Delegated Access** et/ou **Client Access** sur **Unauthorized**, **Authorized** ou **All**.
         * **Unauthorized** : aucune permission n’est autorisée.
         * **Authorized** : choisissez les permissions.
         * **All** : inclut les permissions existantes et futures.
    3. Sélectionnez **Save**.

    <Frame>![Paramètres d’API du Dashboard avec les permissions par défaut pour les applications tierces](/docs/images/third-party-applications/default-permissions-settings.png)</Frame>
  </Tab>

  <Tab title="Management API">
    Effectuez une requête `POST` vers le endpoint `/api/v2/client-grants` avec le corps de requête suivant :

    ```bash cURL wrap lines theme={null}
    curl --request POST \
        --url 'https://YOUR_DOMAIN/api/v2/client-grants' \
        --header 'Authorization: Bearer YOUR_MANAGEMENT_API_TOKEN' \
        --header 'Content-Type: application/json' \
        --data '{
            "default_for": "third_party_clients",
            "audience": "https://api.example.com",
            "scope": ["read:items", "write:items"],
            "subject_type": "user"
    }'
    ```

    | **Paramètre**  | **Type** | **Description**                                                                                                                                                                                                                                                                                                           |
    | -------------- | -------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
    | `default_for`  | Chaîne   | Indique si cette autorisation s’applique automatiquement à certains types d’applications. Définissez-la sur `third_party_clients` pour que toutes les applications tierces aient accès à cette API par défaut.                                                                                                            |
    | `audience`     | Chaîne   | L’identifiant unique (URI) de l’API pour laquelle l’autorisation est créée.                                                                                                                                                                                                                                               |
    | `scope`        | Tableau  | Une liste de permissions (`scopes`) autorisées dans le cadre de cette autorisation.                                                                                                                                                                                                                                       |
    | `subject_type` | Chaîne   | Définit le type d’accès d’application autorisé pour l’API :<br /><ul><li>`user` : utilisé pour l’accès délégué par l’utilisateur, ce qui correspond aux flux qui génèrent un jeton associé à un utilisateur final.</li><li>`client` : utilisé pour l’accès machine à machine, comme le Client Credentials Flow.</li></ul> |
  </Tab>
</Tabs>

<div id="update-client-grant">
  ## Mettre à jour la client grant
</div>

Pour mettre à jour une client grant existante, envoyez une requête [`PATCH`](https://auth0.com/docs/api/management/v2/client-grants/patch-client-grants-by-id) à `/client-grants/{id}` :

```bash lines
curl --location --request PATCH 'https://{yourDomain}/api/v2/client-grants/{CLIENT_GRANT_ID}' \
--header 'Content-Type: application/json' \
--header 'Authorization: Bearer {YOUR_MANAGEMENT_API_TOKEN}' \
--data '{
    "scope": [
        "read:item",
        "update:item"
    ],
    "authorization_details_types":["payment", "credits_transfer"]
}'
```

<div id="delete-client-grant">
  ## Supprimer un client grant
</div>

Pour supprimer un client grant, effectuez une requête [`DELETE`](https://auth0.com/docs/api/management/v2/client-grants/delete-client-grants-by-id) vers `/client-grants/{id}` :

```bash lines
curl --location --request DELETE 'https://{yourDomain}/api/v2/client-grants/{CLIENT_GRANT_ID}' \
--header 'Authorization: Bearer {YOUR_MANAGEMENT_API_TOKEN}'
```

<div id="retrieve-client-grants">
  ## Récupérer les client grants
</div>

Vous pouvez aussi interroger les collections `client-grants` et les parcourir avec pagination à l’aide de paramètres comme `client_id`, `audience` ou `subject_type` :

```bash lines
curl --request GET \
--url 'https://{yourDomain}/api/v2/client-grants?subject_type=user&audience=https%3A%2F%2Fapi.my-service.com' \
--header 'Authorization: Bearer {YOUR_MANAGEMENT_API_TOKEN}' \
--header 'Accept: application/json'
```
