Les applications tierces sont des applications détenues et exploitées par une organisation externe — un partenaire, un développeur indépendant ou un agent d’IA — qui authentifient vos utilisateurs et accèdent à vos API.

Contrairement aux [applications de première partie](/docs/fr-CA/get-started/applications/first-party-and-third-party-applications#first-party-applications), que vous contrôlez directement, les applications tierces fonctionnent de manière indépendante : vous leur accordez l’accès à vos ressources, mais vous ne pouvez pas contrôler ce qu’elles en font. Pour en savoir plus sur les différences entre les applications de première partie et les applications tierces, consultez [Applications de première et de tierce partie](/docs/fr-CA/get-started/applications/first-party-and-third-party-applications).

Les applications tierces présentent les caractéristiques distinctives suivantes :

* **Contrôles de sécurité renforcés** : Auth0 applique des [contrôles de sécurité renforcés](/docs/fr-CA/get-started/applications/third-party-applications/security-controls) aux applications tierces, ce qui garantit que les applications externes ne peuvent accéder qu’aux ressources que vous autorisez explicitement. Les fonctionnalités conçues pour les cas d’utilisation de première partie ne sont pas disponibles.
* **Consentement de l’utilisateur** : Auth0 exige toujours le consentement de l’utilisateur lorsqu’une application tierce demande l’accès à des API. Ce consentement ne peut pas être ignoré.
* **Connexions** : Les applications tierces ne peuvent authentifier les utilisateurs qu’au moyen de connexions au niveau du domaine. Pour en savoir plus, consultez [Promouvoir les connexions au niveau du domaine](/docs/fr-CA/authenticate/identity-providers/promote-connections-to-domain-level).

<div id="use-cases">
  ## Cas d’utilisation
</div>

Voici quelques cas d’utilisation courants des applications tierces :

* **Intégrations de partenaires** : Des partenaires externes créent des applications qui appellent vos API au nom de l’utilisateur. Par exemple, un fournisseur de CRM s’intègre à votre plateforme pour permettre à des clients communs de synchroniser des données entre les deux produits.
* **Agents IA et clients MCP** : Des outils propulsés par l’IA, comme Claude Code, VS Code avec Copilot ou des serveurs MCP personnalisés, se connectent à vos API pour effectuer des actions au nom de l’utilisateur. Pour en savoir plus, consultez [Auth for MCP](https://auth0.com/ai/docs/mcp/intro/overview).
* **Écosystèmes de développeurs** : Vous exposez des API pour permettre à des développeurs externes de créer des applications sur votre plateforme, que ce soit au moyen d’un portail pour développeurs, d’une place de marché ou de [Dynamic Client Registration](/docs/fr-CA/get-started/applications/dynamic-client-registration).
* **Cross App Access (XAA)** : Une application de main-d’œuvre dans le tenant d’une autre organisation accède à vos API au moyen d’une relation de confiance, où l’application demandeuse est modélisée comme une application tierce dans votre tenant. Pour en savoir plus, consultez [Cross App Access](/docs/fr-CA/secure/call-apis-on-users-behalf/xaa).

<div id="supported-client-types">
  ## Types de clients pris en charge
</div>

Les applications tierces prennent en charge les types de clients confidentiel et public :

| **Type de client** | **Type d’application**         | **Cas d’utilisation**                       |
| ------------------ | ------------------------------ | ------------------------------------------- |
| Confidentiel       | Application Web traditionnelle | Intégrations partenaires côté serveur       |
| Public             | Application monopage           | Widgets partenaires basés sur le navigateur |
| Public             | Native                         | Applications mobiles partenaires            |

<div id="supported-grant-types">
  ## Types d’octroi pris en charge
</div>

Les applications tierces prennent en charge :

* `authorization_code` avec PKCE obligatoire
* `refresh_token`
* `client_credentials` (clients confidentiels uniquement)

<div id="get-started">
  ## Pour commencer
</div>

Pour configurer une application tierce dans Auth0, suivez les étapes ci-dessous :

<div id="step-1-create-the-application">
  ### Étape 1 : Créer l’application
</div>

[Créer une application tierce](/docs/fr-CA/get-started/applications/third-party-applications/configure-third-party-applications#create-a-third-party-application) dans l’Auth0 Dashboard ou au moyen de la Management API.

<div id="step-2-configure-api-access">
  ### Étape 2 : Configurer l’accès à l’API
</div>

Les applications tierces nécessitent toujours une autorisation explicite pour accéder à vos API, même lorsque la politique d’accès de l’API est définie sur **Allow All.** Vous configurez les politiques d’accès aux API au moyen des [client grants](/docs/fr-CA/get-started/applications/application-access-to-apis-client-grants).

Vous pouvez configurer des permissions par défaut qui s’appliquent automatiquement à toutes les applications tierces. C’est utile si vous gérez un grand nombre d’applications tierces ou si vous utilisez [Dynamic Client Registration](/docs/fr-CA/get-started/applications/dynamic-client-registration). Pour en savoir plus, consultez [Permissions par défaut pour les applications tierces](/docs/fr-CA/get-started/applications/application-access-to-apis-client-grants#default-permissions-for-third-party-applications).

Vous pouvez également définir des permissions précises pour chaque application au moyen des [client grants](/docs/fr-CA/get-started/applications/application-access-to-apis-client-grants). Les permissions par application ont préséance sur les permissions par défaut. Pour en savoir plus, consultez [Create client grant](/docs/fr-CA/get-started/applications/application-access-to-apis-client-grants#create-client-grant).

<div id="step-3-configure-connections">
  ### Étape 3 : Configurer les connexions
</div>

Les applications tierces peuvent authentifier les utilisateurs uniquement au moyen de connexions promues au niveau du domaine. Les connexions au niveau du domaine sont accessibles à toutes les applications tierces du tenant.

Pour promouvoir une connexion au niveau du domaine, consultez [Promouvoir les connexions au niveau du domaine](/docs/fr-CA/authenticate/identity-providers/promote-connections-to-domain-level).

<div id="step-4-users-provide-consent">
  ### Étape 4 : Les utilisateurs donnent leur consentement
</div>

Lorsqu’un utilisateur s’authentifie par l’intermédiaire d’une application tierce, Auth0 affiche une boîte de dialogue de consentement qui lui demande d’approuver les permissions demandées. Le consentement est toujours requis pour les applications tierces et ne peut pas être passé outre.

Pour en savoir plus, consultez [Consentement de l’utilisateur et applications tierces](/docs/fr-CA/get-started/applications/third-party-applications/user-consent-and-third-party-applications).

<div id="dynamic-client-registration">
  ## Dynamic Client Registration
</div>

[Dynamic Client Registration](/docs/fr-CA/get-started/applications/dynamic-client-registration) crée par défaut des applications tierces avec des contrôle de sécurité renforcé par défaut. Avant d’activer DCR pour les applications tierces, vous devez [configurer les permissions d’API par défaut](/docs/fr-CA/get-started/applications/dynamic-client-registration#configure-api-access-for-dcr-clients) afin que les clients enregistrés dynamiquement puissent accéder à vos API.
