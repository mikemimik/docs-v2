Sur la page [Applications](https://manage.auth0.com/#/applications) du Dashboard, repérez votre application dans la liste, puis cliquez sur son nom pour afficher les paramètres disponibles.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Les [applications tierces](/docs/fr-CA/get-started/applications/third-party-applications) disposent d’un ensemble restreint de propriétés configurables. Les propriétés qui ne font pas partie de l’ensemble pris en charge ne peuvent pas être configurées dans l’Auth0 Dashboard ni au moyen de la Management API. Pour en savoir plus, consultez [Contrôles de sécurité pour les applications tierces](/docs/fr-CA/get-started/applications/third-party-applications/security-controls#restricted-client-configuration).
</Callout>

<div id="basic-settings">
  ## Paramètres de base
</div>

Lorsque vous modifiez les paramètres d’une application existante ou créez une nouvelle application, vous entrez des renseignements sur l’application dans la vue **Paramètres**.

<div id="basic-information">
  ### Informations de base
</div>

<Frame>![Dashboard Applications Onglet Paramètres de l’application Informations de base](/docs/images/cdy7uua7fh8z/2GPUw7BODYuYYH3658Upz3/92a49ec57e6b4d07be96093989baac03/2023-04-11_15-34-58.png)</Frame>

* **Nom** : Le nom de votre application. Il peut être modifié et sera visible dans le portail, les courriels, les journaux, etc.
* **Domain** : Le nom de votre tenant Auth0. Vous le choisissez lorsque vous créez un nouveau tenant Auth0 et il ne peut pas être modifié. Si vous avez besoin d’un domaine différent, vous devez créer un nouveau tenant en sélectionnant **+ Create Tenant** dans le menu en haut à droite.
* **<Tooltip tip="Client ID : Valeur d’identification attribuée à votre ressource enregistrée par Auth0." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Client+ID">Client ID</Tooltip>** : L’identifiant unique de votre application. Vous l’utiliserez lors de la configuration de l’authentification avec Auth0. Il est généré par le système lorsque vous créez une nouvelle application et ne peut pas être modifié.
* **<Tooltip tip="Client Secret : Secret utilisé par un client (application) pour s’authentifier auprès du serveur d’autorisation; il ne devrait être connu que du client et du serveur d’autorisation et doit être suffisamment aléatoire pour ne pas pouvoir être deviné." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Client+Secret">Client Secret</Tooltip>** : Une chaîne de caractères utilisée pour signer et valider les <Tooltip tip="Client Secret : Secret utilisé par un client (application) pour s’authentifier auprès du serveur d’autorisation; il ne devrait être connu que du client et du serveur d’autorisation et doit être suffisamment aléatoire pour ne pas pouvoir être deviné." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=ID+Tokens">ID Tokens</Tooltip> dans les flux d’authentification et pour accéder à certains endpoints de l’API Auth0. Par défaut, la valeur est masquée; cochez donc la case **Reveal Client Secret** pour l’afficher. Bien que le Client ID soit considéré comme une information publique, le Client Secret **doit rester confidentiel**. Si quelqu’un peut accéder à votre Client Secret, cette personne peut émettre des jetons et accéder à des ressources auxquelles elle ne devrait pas avoir accès.
* **Description** : Une description libre de l’objectif de l’application. Maximum de 140 caractères.

<div id="application-properties">
  ### Propriétés de l’application
</div>

<Frame>![Dashboard Applications Onglet paramètre de l’application Propriétés de l’application](/docs/images/cdy7uua7fh8z/4Z66jnQOVFwNo5iuHtdlxo/153fabafccac939aa15c23a65ba974cb/Application_Properties_-_English.png)</Frame>

* **Logo de l’application** : URL du logo (taille recommandée : 150 x 150 pixels) à afficher pour l’application. Elle apparaît à plusieurs endroits, notamment dans la liste des applications du Dashboard et dans les formulaires de consentement personnalisés. Si aucun logo n’est défini, l’insigne par défaut pour ce type d’application s’affichera.
* **Application Ownership** : Indique si l’application est first-party ou third-party. Les applications tierces sont soumises à des [contrôles de sécurité renforcés](/docs/fr-CA/get-started/applications/third-party-applications/security-controls). Le type de propriété de l’application est défini à la création et ne peut pas être modifié. Pour en savoir plus, consultez [Applications first-party et third-party](/docs/fr-CA/get-started/applications/first-party-and-third-party-applications).
* **Application Type** : Le type d’application Auth0 détermine les paramètres que vous pouvez configurer dans le Dashboard. (Non modifiable pour les applications M2M. Peut parfois être désactivé pour d’autres types d’applications Auth0 si les types d’octroi sélectionnés ne sont autorisés que pour le type d’application actuellement choisi.) Utilisez la liste déroulante pour sélectionner l’un des types suivants :

  * [Machine to Machine](/docs/fr-CA/get-started/auth0-overview/create-applications/machine-to-machine-apps) : Applications non interactives, comme des outils en ligne de commande, des démons, des appareils IoT ou des services exécutés sur votre backend. En général, vous utiliserez cette option si vous avez un service qui doit accéder à une API.
  * [Native App](/docs/fr-CA/get-started/auth0-overview/create-applications/native-apps) : Applications mobiles ou de bureau qui s’exécutent nativement sur un appareil (comme iOS ou Android).
  * [Regular Web App](/docs/fr-CA/get-started/auth0-overview/create-applications/regular-web-apps) : Applications web traditionnelles qui exécutent la majeure partie de leur logique côté serveur (comme Express.js ou ASP.NET).
  * [Single Page App](/docs/fr-CA/get-started/auth0-overview/create-applications/single-page-web-apps) : Applications JavaScript qui exécutent la majeure partie de leur logique d’interface utilisateur dans un navigateur web et communiquent avec un serveur web principalement au moyen d’API (comme AngularJS + Node.js ou React).

<div id="application-uris">
  ### URI de l’application
</div>

<Frame>![Dashboard Applications Paramètres de l’application URI de l’application](/docs/images/cdy7uua7fh8z/1QhW2i4fTCCp8owey2tMPI/a19f7fc3f84118920d0e5cb2357144da/Application_URIs.png)</Frame>

* **URI de connexion de l’application** : Dans certains cas, Auth0 aura besoin que votre application redirige vers sa page de connexion. Cette URI doit pointer vers une route de votre application qui redirige vers le endpoint `/authorize` de votre tenant [endpoint](https://auth0.com/docs/api/authentication#authorize-application). Elle prend habituellement la forme `https://myapp.org/login`. Vous pouvez utiliser les espaces réservés suivants dans ce champ :

  * [**Espaces réservés des métadonnées de l’organisation**](/docs/fr-CA/get-started/applications/wildcards-for-subdomains#organization-url-placeholders) : Utilisez `{organization.metadata.KEY}` pour renseigner dynamiquement l’URL en fonction des métadonnées de l’Auth0 Organization associée à la requête (par exemple, `https://{organization.metadata.public_login_host}/login`).
  * [**Espaces réservés du domaine personnalisé**](/docs/fr-CA/get-started/applications/wildcards-for-subdomains#custom-domain-url-placeholders) : Utilisez `{custom_domain.metadata.KEY}` pour renseigner dynamiquement l’URL en fonction des métadonnées du domaine personnalisé utilisé dans la requête (par exemple, `https://{custom_domain.metadata.public_app_host}/login`).

  Pour en savoir plus, consultez [Configurer les routes de connexion par défaut](/docs/fr-CA/authenticate/login/auth0-universal-login/configure-default-login-routes#dynamic-login-uris-with-metadata-placeholders).
* **URL de rappel autorisées** : Ensemble d’URL vers lesquelles Auth0 est autorisé à rediriger les utilisateurs après leur authentification. Vous pouvez indiquer plusieurs URL valides en les séparant par des virgules (généralement pour gérer différents environnements, comme l’AQ ou les tests). Pour les environnements de production, vérifiez que les URL ne pointent pas vers localhost. Vous pouvez utiliser les espaces réservés suivants dans ce champ :
  * [**Caractères génériques**](/docs/fr-CA/get-started/applications/wildcards-for-subdomains#wildcard-url-placeholders) : Utilisez `*` pour les sous-domaines (`*.google.com`) *Non recommandé pour les environnements de production.*
  * [**Espaces réservés de l’organisation**](/docs/fr-CA/get-started/applications/wildcards-for-subdomains#organization-url-placeholders) : Utilisez `{organization_name}` pour spécifier dynamiquement le nom d’une organisation enregistrée (par exemple, `https://{organization_name}.example.com`).
  * [**Espaces réservés du domaine personnalisé**](/docs/fr-CA/get-started/applications/wildcards-for-subdomains#custom-domain-url-placeholders) : Utilisez `{custom.domain.metadata.KEY}` pour renseigner dynamiquement l’URL en fonction des métadonnées du domaine personnalisé utilisé dans la requête (par exemple, `https://{custom_domain.metadata.public_app_url}/callback`).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  La première URL indiquée dans ce champ est utilisée comme URL de rappel par défaut lorsque le flux de protocole correspondant n’en précise pas explicitement une. Cela s’applique plus précisément aux flux SAML, WS-Fed et SSO initiés par le SAML IdP.
</Callout>

N’utilisez pas d’espaces réservés génériques ni d’URL localhost dans les champs callbacks ou origines autorisées de votre application. L’utilisation d’URL de redirection avec des espaces réservés génériques peut rendre votre application vulnérable aux attaques. Pour en savoir plus, consultez [Unvalidated Redirects and Forwards Cheat Sheet on owasp.org](https://www.owasp.org/index.php/Unvalidated_Redirects_and_Forwards_Cheat_Sheet). Privilégiez plutôt, lorsque pertinent, les URL comportant l’espace réservé `{organization_name}`. Pour en savoir plus, consultez [Espaces réservés d’URL de sous-domaine](/docs/fr-CA/get-started/applications/wildcards-for-subdomains).

* **Allowed Logout URLs** : Après qu’un utilisateur s’est déconnecté d’Auth0, vous pouvez le rediriger à l’aide du paramètre de requête `returnTo`. L’URL utilisée dans `returnTo` doit être indiquée ici. Vous pouvez spécifier plusieurs URL valides en les séparant par des virgules. Pour les environnements de production, vérifiez que les URL ne pointent pas vers localhost.
  * [**Wildcards**](/docs/fr-CA/get-started/applications/wildcards-for-subdomains#wildcard-url-placeholders) : Utilisez `*` pour les sous-domaines (`*.google.com`) *Non recommandé pour les environnements de production.*
  * [**espaces réservés du domaine personnalisé**](/docs/fr-CA/get-started/applications/wildcards-for-subdomains#custom-domain-url-placeholders) : Utilisez `{custom.domain.metadata.KEY}` pour remplir l’URL dynamiquement à partir des métadonnées du domaine personnalisé utilisé dans la requête (par exemple, `https://{custom_domain.metadata.public_app_url}/callback`).
* **Allowed Web Origins** : Liste des URL à partir desquelles peut provenir une authorization request utilisant [Cross-Origin Authentication](/docs/fr-CA/authenticate/login/cross-origin-authentication), [Device Flow](/docs/fr-CA/get-started/authentication-and-authorization-flow/device-authorization-flow) et `web_message` comme mode de réponse. Vous pouvez spécifier plusieurs URL valides en les séparant par des virgules. Pour les environnements de production, vérifiez que les URL ne pointent pas vers localhost. Les chemins, les query strings et les informations de hachage ne sont pas pris en compte lors de la validation de ces URL (et peuvent même faire échouer la correspondance). Vous pouvez fournir jusqu’à 100 URL dans le champ **Allowed Web Origins**.
  * [**Wildcards**](/docs/fr-CA/get-started/applications/wildcards-for-subdomains#wildcard-url-placeholders) : Utilisez `*` pour les sous-domaines (`*.google.com`) *Non recommandé pour les environnements de production.*
  * [**espaces réservés du domaine personnalisé**](/docs/fr-CA/get-started/applications/wildcards-for-subdomains#custom-domain-url-placeholders) : Utilisez `{custom.domain.metadata.KEY` pour remplir l’URL dynamiquement à partir des métadonnées du domaine personnalisé utilisé dans la requête (par exemple, `https://{custom_domain.metadata.public_app_url}/callback`).
* **Allowed Origins (CORS)** : Liste des URL autorisées à envoyer des requêtes Cross-Origin Resource Sharing (CORS) à Auth0.
  * [**espaces réservés du domaine personnalisé**](/docs/fr-CA/get-started/applications/wildcards-for-subdomains#custom-domain-url-placeholders) : Utilisez `{custom.domain.metadata.KEY}` pour remplir l’URL dynamiquement à partir des métadonnées du domaine personnalisé utilisé dans la requête (par exemple, `https://{custom_domain.metadata.public_app_url}/callback`).

<Warning>
  Si vous configurez les URL de votre Application exclusivement à l’aide d’espaces réservés du domaine personnalisé, les authentication requests effectuées au moyen du canonical domain de votre tenant (par exemple, https://your-tenant.us.auth0.com) échoueront.

  Cela se produit parce que le canonical domain ne possède pas les métadonnées personnalisées nécessaires pour résoudre l’espace réservé. Assurez-vous que votre application utilise le domaine personnalisé précis pour l’authentication, ou fournissez une URL statique de solution de secours si l’utilisation du canonical domain est requise.
</Warning>

<div id="id-token">
  ### ID Token
</div>

Dans la section **ID Token**, saisissez l’**expiration du ID Token** (en secondes), c’est-à-dire le délai avant l’expiration du `id_token` d’Auth0. La valeur par défaut est de 36000 secondes, soit 10 heures.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  **Utiliser Auth0 au lieu de l’IdP pour le Single Sign-on** : si ce paramètre est activé, Auth0 empêche la redirection des utilisateurs authentifiés ayant une session valide vers le fournisseur d’identité (comme Facebook ou ADFS). **Tenants Legacy uniquement.**
</Callout>

<div id="refresh-token-rotation">
  ### Rotation des Refresh Token
</div>

Dans la section **<Tooltip tip="Refresh Token : jeton utilisé pour obtenir un Access Token renouvelé sans que les utilisateurs aient à se reconnecter." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Refresh+Token">Refresh Token</Tooltip> Rotation**, activez ou désactivez la rotation. Lorsqu’elle est activée, chaque échange d’un refresh token entraîne l’émission d’un nouveau refresh token et l’invalidation du jeton existant. Cela permet de détecter automatiquement la réutilisation d’un jeton en cas de fuite. De plus, saisissez la **période de chevauchement de la rotation** (en secondes). Cet intervalle correspond au délai de tolérance pendant lequel le même `refresh_token` peut être utilisé pour demander un `access_token` sans déclencher la détection automatique de la réutilisation. Pour en savoir plus, consultez [Refresh Token Rotation](/docs/fr-CA/secure/tokens/refresh-tokens/refresh-token-rotation).

<Frame>![Dashboard Applications Applications onglet Settings Refresh Token Rotation](/docs/images/cdy7uua7fh8z/Rk6i8MUVWZG34P7K6aurb/9457a6ea7ca5942522778c7ed7925023/Screenshot_2024-11-05_at_18.14.22.png)</Frame>

<div id="refresh-token-expiration">
  ### Expiration du Refresh Token
</div>

Dans la section **Expiration du Refresh Token**, activez ou désactivez l’expiration absolue et l’expiration pour inactivité, puis définissez la durée de vie (en secondes) de chacune. <div id="open-redirect-protection">
  ### Protection contre les redirections ouvertes
</div>

Détermine comment Auth0 gère les redirections pour les applications tierces. Ce paramètre est offert uniquement pour les applications tierces dotées de contrôles de sécurité renforcés.

<Frame>![bascule de protection contre les redirections ouvertes dans le Dashboard](/docs/images/third-party-applications/open-redirect-protection.png)</Frame>

Lorsqu’elle est activée (par défaut pour les applications tierces), Auth0 ne redirige pas vers l’URL de rappel de l’application en cas d’erreurs d’authentification et n’expose pas `application.callback_domain` dans les modèles de courriel. Cela permet d’éviter les attaques par redirection ouverte lorsque l’URI de redirection est contrôlée par une partie non fiable.

Ne désactivez la protection contre les redirections ouvertes que pour les applications tierces dont les URI de rappel configurées sont fiables.

Pour en savoir plus, consultez [Protection contre les redirections](/docs/fr-CA/get-started/applications/third-party-applications/security-controls#redirect-protection).

<div id="application-metadata">
  ### Métadonnées de l’application
</div>

Les métadonnées de l’application sont des clés et des valeurs de chaîne personnalisées (chacune étant limitée à 255 caractères), définies pour chaque application. Les métadonnées sont exposées dans l’objet application sous la forme de `client_metadata`, et dans les règles sous la forme de `context.clientMetadata`. Vous pouvez créer jusqu’à 10 ensembles de métadonnées.

<div id="oauth">
  ### OAuth
</div>

<Frame>![Dashboard Applications Application Settings paramètre avancé OAuth Tab](/docs/images/cdy7uua7fh8z/OAuth-tab.png)</Frame>

* Par défaut, toutes les apps/API peuvent effectuer une requête de délégation, mais si vous souhaitez accorder explicitement des permissions à certaines apps/API, vous pouvez le faire dans **Allowed Apps/APIs**.
* Pour les clients qui utilisent le module complémentaire Highly Regulated Identity, utilisez le paramètre **Compliance Enforcement Level** pour définir votre niveau de conformité. Pour en savoir plus, consultez [Configure FAPI Compliance](/docs/fr-CA/get-started/applications/configure-fapi-compliance).
* **Non-Verifiable Callback URI End-User Confirmation** : utilisez ce paramètre pour déterminer si l’utilisateur doit confirmer la connexion lorsqu’un URI non vérifiable est utilisé comme callback. Auth0 recommande de ne pas ignorer la confirmation de l’utilisateur final dans ces cas. Ce paramètre a préséance sur le paramètre du tenant portant le même nom. Pour en savoir plus, consultez [Measures Against Application Impersonation](/docs/fr-CA/secure/security-guidance/measures-against-app-impersonation.mdx).
* Définissez l’algorithme utilisé ([**HS256**](https://en.wikipedia.org/wiki/Symmetric-key_algorithm) ou [**RS256**](https://en.wikipedia.org/wiki/Public-key_cryptography)) pour signer vos <Tooltip tip="JSON Web Token (JWT) : format standard de ID Token (et souvent de jeton d’accès) utilisé pour représenter des claims de façon sécurisée entre deux parties." cta="View Glossary" href="/docs/fr-CA/glossary?term=JSON+web+tokens">JSON Web Tokens</Tooltip>. Pour en savoir plus, consultez [JSON Web Token Signing Algorithms](/docs/fr-CA/get-started/applications/signing-algorithms). Lorsque vous sélectionnez `RS256` (recommandé), le token sera signé à l’aide de la clé privée de votre tenant.
* Activez ou désactivez le paramètre **Trust Token Endpoint IP Header**; s’il est activé, `auth0-forwarded-for` est considéré comme fiable et utilisé comme source de l’adresse IP de l’utilisateur final pour la protection contre les attaques par force brute sur le Token endpoint. Ce paramètre est offert uniquement pour les Regular Web Apps et les M2M Apps.
* Activez ou désactivez le commutateur pour indiquer si votre application est **OIDC Conformant** ou non. Les applications marquées comme OIDC Conformant respecteront strictement la spécification OIDC.
