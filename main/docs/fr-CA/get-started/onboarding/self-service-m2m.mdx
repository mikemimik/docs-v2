Si votre cas d’utilisation concerne une <Tooltip tip="Audience : Identifiant unique de l’audience d’un jeton émis. Nommée aud dans un jeton, sa valeur contient l’ID d’une application (Client ID) pour un ID Token ou d’une API (identifiant d’API) pour un jeton d’accès." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=audience">audience</Tooltip> non interactive, comme des API ou des serveurs backend, vous utiliserez une configuration machine-to-machine (M2M).

<div id="use-cases">
  ## Cas d’utilisation
</div>

Utilisez le parcours d’onboarding M2M si vous :

* prenez en charge les communications de service à service
* avez des jobs planifiés ou des tâches cron exécutées sur des serveurs qui doivent accéder à des ressources protégées ou à des API
* permettez à des appareils IoT de communiquer avec des services backend ou des API
* avez une couche API qui doit communiquer avec d’autres couches API sans intervention de l’utilisateur ou après l’expiration d’un jeton utilisateur
* avez une API privilégiée qui peut devoir être appelée avant qu’un utilisateur soit authentifié (c.-à-d. à partir d’une Action ou d’un script de base de données personnalisé dans votre tenant Auth0)
* utilisez API Gateway pour gérer des services backend
* utilisez ou prenez en charge des applications non interactives ou d’autres outils n’impliquant aucune interaction humaine, comme des démons ou des services backend

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Ces services auront quand même besoin d’un jeton d’accès M2M pour l’authentification.
</Callout>

<div id="how-to-use-this-guide">
  ## Comment utiliser ce guide
</div>

Ce guide vous accompagne dans la création de votre mise en œuvre M2M dans Auth0. Nous y présentons des éléments à considérer, des pratiques exemplaires et des concepts à passer en revue.

* Dans Architecture, nous vous conseillons de configurer Auth0 de façon à prendre en charge votre cycle de vie du développement logiciel et votre infrastructure existante.
* Dans Créer un compte, nous vous fournissons des instructions pour créer votre instance d’API dans Auth0 ainsi qu’une application pour prendre en charge le flux d’authentification (ou `grant`) nécessaire à l’authentification machine à machine.
* Dans Authentification, nous vous expliquons le `grant` à utiliser pour l’authentification, ainsi que les <Tooltip tip="Jeton d’accès : information d’autorisation sous forme de chaîne opaque ou de JWT, utilisée pour accéder à une API." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=access+tokens">jetons d’accès</Tooltip> et les permissions (ou scopes) que vous pouvez définir.
* Dans Image de marque, nous vous indiquons où trouver l’information sur la façon de configurer les <Tooltip tip="Domaine personnalisé : domaine tiers ayant un nom spécialisé ou personnalisé." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Custom+Domains">domaines personnalisés</Tooltip> selon la façon dont vous prévoyez gérer les certificats.
* Dans Automatisations de déploiement, vous pouvez en apprendre davantage sur nos outils pour vous aider avec le déploiement.
* Dans Assurance qualité, vous pouvez en apprendre davantage sur les tests unitaires et les vérifications de préparation que nous offrons dans le <Tooltip tip="Auth0 Dashboard : principal produit d’Auth0 pour configurer vos services." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip>.

<div id="architecture">
  ## Architecture
</div>

Avant de configurer votre compte Auth0 et votre tenant, ou les groupes et structures de vos services Auth0, dressez d’abord une cartographie de votre infrastructure existante afin de tirer le meilleur parti des capacités d’Auth0 dans votre écosystème actuel.

Comme mentionné dans les scénarios courants, vous pourriez avoir d’autres technologies non interactives dans votre domaine d’application, votre domaine réseau ou votre domaine d’appareils M2M à prendre en compte avant de configurer Auth0. Pour consulter un exemple de scénario M2M, lisez [Server + API](/docs/fr-CA/get-started/architecture-scenarios/server-application-api/part-1). Pour essayer un atelier pratique avec Node et tester le déploiement d’API, consultez notre [dépôt GitHub](https://www.google.com/url?q=https://github.com/auth0-training/labs-node-working-with-apis\&sa=D\&source=docs\&ust=1723578190947730\&usg=AOvVaw3IXRQG9ogH4pW6LEY047T6).

Vous pourriez aussi créer une visualisation de votre pile technologique actuelle et planifier la façon dont Auth0 s’intégrera à votre cycle de développement logiciel (SDLC) actuel. Cela peut vous aider à déterminer de combien de tenants vous pourriez avoir besoin.

<div id="considerations">
  ### Considérations
</div>

Avant de créer un nouveau compte ou de configurer votre premier tenant, vous devriez tenir compte des éléments suivants :

* La façon dont vous partitionnez ou regroupez vos API pour appeler des points de terminaison précis.

  * Cela peut déterminer l’audience et d’autres claims dans les jetons d’accès.
* Les consommateurs tiers de votre ressource peuvent demander un jeton d’accès pour chaque appel. Un nombre excessif d’appels pourrait avoir une incidence sur votre [limite de débit](/docs/fr-CA/troubleshoot/customer-support/operational-policies/rate-limit-policy/rate-limit-configurations/essentials-professional-b2b).

  * Vous pouvez utiliser API Gateway pour limiter le nombre de jetons d’accès qu’un tiers peut demander. Pour en savoir plus sur API Gateway et Auth0, consultez [Configurer un fournisseur d’identité dans Access Gateway](/docs/fr-CA/get-started/auth0-overview/create-applications/configure-an-identity-provider-in-access-gateway).

<div id="create-an-account">
  ## Créer un compte
</div>

Maintenant que vous avez défini votre architecture, vous allez configurer votre compte Auth0 et vos tenants. Lorsque vous vous inscrivez aux services Auth0, vous créez votre premier [tenant](/docs/fr-CA/get-started/auth0-overview/create-tenants). C’est dans ce tenant que vous configurez les assets, les services et les ressources Auth0. [Inscrivez-vous](https://auth0.com/signup) pour commencer.

<Card title="Avant de commencer">
  Dans l’Auth0 Dashboard ou avec l’Auth0 Management API, créez :

  * Une API pour représenter votre API
  * Une application M2M pour utiliser le flux Client Credentials

  Vous voudrez peut-être planifier certains détails de configuration avant de créer un compte.

  * Le nom de votre tenant joue un rôle dans votre domaine Auth0. Avant d’en choisir un, vous devriez consulter les [caractéristiques d’un tenant](/docs/fr-CA/get-started/auth0-overview/create-tenants#create-a-tenant-and-domain).
  * Déterminez quelles fonctionnalités Auth0 sont nécessaires pour votre cas d’utilisation. Certaines fonctionnalités sont offertes uniquement avec les forfaits Professional et Enterprise.
  * Déterminez si vous devez prendre en charge plusieurs environnements, comme le développement, la préproduction et la production. Pour en savoir plus, consultez [Set Up Multiple Environments](/docs/fr-CA/get-started/auth0-overview/create-tenants/set-up-multiple-environments).
  * Si votre cas d’utilisation implique des applications tierces que vous souhaitez enregistrer dans un tenant, vous pouvez utiliser [Dynamic Application Registration](/docs/fr-CA/get-started/applications/dynamic-client-registration) selon la spécification [OIDC Client Registration](https://openid.net/specs/openid-connect-registration-1_0.html).
</Card>

<div id="authentication">
  ## Authentification
</div>

Lorsque vous appelez une API à partir d’une autre API, ou dans toute situation où il n’y a pas de contexte d’utilisateur authentifié, vous devez pouvoir autoriser l’application plutôt qu’un utilisateur. Il s’agit d’un processus en une seule étape où l’application est authentifiée (à l’aide d’un `client_id` et d’un `client_secret`), puis autorisée au moyen d’un seul appel.

Pour l’authentification des applications ou services non interactifs, vous devez sélectionner un client grant, ou un flux d’authentification. Le <Tooltip tip="OAuth 2.0 : cadre d’autorisation qui définit les protocoles et les workflows d’autorisation." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=OAuth+2.0">OAuth 2.0</Tooltip> [flux Client Credentials](https://tools.ietf.org/html/rfc6749#section-4.4) ne nécessite aucune intervention humaine et convient le mieux aux applications M2M.

<Card title="Avant de commencer">
  Dans Auth0 Dashboard ou la Management API, vous allez :

  * Configurer votre application pour utiliser le flux Client Credentials
  * Mettre à jour les scopes de vos jetons d’accès M2M

  Avant de configurer votre méthode d’authentification :

  * Consultez le [flux Client Credentials](/docs/fr-CA/get-started/authentication-and-authorization-flow/client-credentials-flow) pour l’authentification machine à machine. Il s’agit du workflow d’authentification et d’autorisation non interactives.
  * Déterminez le niveau d’accès de vos API. Cela vous aidera à déterminer les [scopes](/docs/fr-CA/get-started/apis/scopes/api-scopes), ou permissions, que vous configurerez lorsque vous créerez votre API.
</Card>

<div id="configure-the-client-credential-flow">
  ### Configurer le Client Credential Flow
</div>

Vous pouvez utiliser l’Auth0 Dashboard ou <Tooltip tip="Management API : un produit qui permet aux clients d’effectuer des tâches administratives." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Management+API">Management API</Tooltip> pour configurer le flux d’authentification afin de fournir des identifiants client en échange d’un jeton d’accès.

Suivez les instructions de [Mettre à jour les types d’autorisation](/docs/fr-CA/get-started/applications/update-grant-types) pour utiliser l’Auth0 Dashboard ou la Management API.

<div id="m2m-access-tokens">
  ### Jetons d’accès M2M
</div>

Dans l’authentification fondée sur les jetons, les clients non interactifs fournissent `client_id` et `client_secret` dans un appel au [point de terminaison de jeton de l’Authentication API](/docs/fr-CA/customize/actions/explore-triggers/machine-to-machine-trigger) pour obtenir un [jeton d’accès](/docs/fr-CA/secure/tokens/access-tokens). Ce jeton d’accès permet d’accéder à votre API protégée.

Le profil, ou format, par défaut est le profil de jeton Auth0, qui fait partie des deux profils de jeton offerts. Vous pouvez choisir de passer au profil de jeton RFC 9068. Pour en savoir plus, consultez [Profils de jeton d’accès](/docs/fr-CA/secure/tokens/access-tokens/access-token-profiles). Pour vérifier que le jeton est valide, votre API vérifiera les [algorithmes de signature](/docs/fr-CA/get-started/applications/signing-algorithms). L’<Tooltip tip="Algorithme de signature : algorithme utilisé pour signer numériquement les jetons afin de garantir que le jeton n’a pas été altéré." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=signing+algorithm">algorithme de signature</Tooltip> par défaut est RSA256, un algorithme à clé.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Auth0 prend en charge d’autres méthodes d’authentification du client que la simple fourniture du Client ID et du Client Secret comme identifiants. Ces méthodes, y compris notre recommandation d’utiliser [Private Key JWT](/docs/fr-CA/get-started/authentication-and-authorization-flow/authenticate-with-private-key-jwt) pour les configurations M2M, sont offertes avec un plan Enterprise. Pour en savoir plus, consultez [Informations d’identification de l’application](/docs/fr-CA/secure/application-credentials).
</Callout>

  ```

  package main

  	"fmt"
  	"strings"
  	"net/http"
  	"io/ioutil"

  }
  ```

<div id="token-expiration">
  #### Expiration des jetons
</div>

Vos jetons d’accès ont une durée de validité limitée. Comme vos communications se font par le canal arrière, vous ne pouvez pas utiliser de <Tooltip tip="Refresh Token : jeton utilisé pour obtenir un nouveau jeton d’accès sans obliger les utilisateurs à se reconnecter." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=refresh+tokens">jetons d’actualisation</Tooltip> pour prolonger les sessions; vous devriez donc envisager de configurer vos jetons d’accès avec une durée d’expiration d’une heure. Vous devrez peut-être trouver le bon équilibre entre la sécurité et les performances pour votre environnement. Pour en savoir plus, consultez [Mettre à jour la durée de vie du jeton d’accès](/docs/fr-CA/secure/tokens/access-tokens/update-access-token-lifetime).

<div id="scopes">
  ### Scopes
</div>

Avant que des clients ou des services non interactifs appellent votre API, vous devez définir les permissions, ou scopes, qu’elle autorise. Vous pouvez configurer dans Auth0 Dashboard les scopes à inclure dans votre requête d’authentification à l’Authentication API. Pour voir d’autres exemples d’utilisation des scopes d’API, consultez [Scopes d’API](/docs/fr-CA/get-started/apis/scopes/api-scopes).

Pour configurer les scopes, suivez les instructions de [Ajouter des permissions d’API](/docs/fr-CA/get-started/apis/add-api-permissions) dans Auth0 Dashboard ou utilisez l’exemple fourni pour la Management API.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Pour ajouter des claims personnalisés à un jeton d’accès, vous pouvez utiliser le flux Machine to Machine d’Actions. Pour en savoir plus, consultez Machine to Machine Flow.
</Callout>

<div id="branding">
  ## Image de marque
</div>

Même si vous prenez en charge des clients non interactifs ou des services fonctionnant sur le canal arrière, vous pouvez quand même personnaliser votre expérience pour qu’elle corresponde à votre image de marque actuelle.

<div id="custom-domains">
  ### Domaines personnalisés
</div>

Auth0 prend en charge l’utilisation de domaines personnalisés lorsque vous appelez le point de terminaison `/authorize` pour demander des jetons d’accès.

<Card title="Onboarding M2M - domaines personnalisés">
  Dans Auth0 Dashboard, vous devez :

  * Enregistrer et vérifier votre domaine avant de pouvoir l’utiliser avec vos services Auth0.
  * Déterminer si vous souhaitez gérer vos propres certificats ou utiliser un certificat géré par Auth0. Pour en savoir plus sur les certificats, consultez [Options de gestion des certificats](/docs/fr-CA/customize/custom-domains#certificate-management-options).
  * Vérifier que la version TLS (SSL) et l’algorithme de chiffrement que vous souhaitez utiliser pour les certificats autogérés sont pris en charge par Auth0. Pour en savoir plus, consultez [Versions et algorithmes de chiffrement TLS (SSL)](/docs/fr-CA/customize/custom-domains/self-managed-certificates/tls-ssl).
</Card>

1. Pour configurer votre domaine personnalisé avec des certificats gérés par Auth0, suivez les instructions de [Configurer des domaines personnalisés avec des certificats gérés par Auth0](/docs/fr-CA/customize/custom-domains/auth0-managed-certificates).

   1. Si vous souhaitez gérer vos propres certificats, suivez les instructions de [Configurer des domaines personnalisés avec des certificats autogérés](/docs/fr-CA/customize/custom-domains/self-managed-certificates).

      <Callout icon="file-lines" color="#0EA5E9" iconType="regular">
        Vous devez avoir un abonnement Enterprise pour gérer les certificats de votre domaine personnalisé. Pour en savoir plus, consultez [Auth0 Pricing](https://auth0.com/pricing/) et [Login](/docs/fr-CA/authenticate/login).
      </Callout>
2. Consultez [Configuration de l’API avec des domaines personnalisés](/docs/fr-CA/customize/custom-domains/configure-features-to-use-custom-domains#apis). Vous devrez peut-être ajuster les paramètres de votre API pour y intégrer un domaine personnalisé.

<div id="deployment-automation">
  ## Automatisation du déploiement
</div>

Auth0 offre du soutien pour différentes options d’automatisation du déploiement, et elles peuvent aussi être utilisées ensemble.

<Card title="Bonne pratique">
  Quelle que soit la façon dont vous configurez l’automatisation du déploiement, nous vous recommandons d’effectuer des tests unitaires de votre code personnalisé et de vos Actions avant le déploiement, puis d’exécuter également des tests d’intégration sur votre tenant après le déploiement.
</Card>

<div id="deploy-cli-tool">
  ### Deploy CLI Tool
</div>

Comme recommandé dans la section Architecture, vous devriez avoir des tenants Auth0 pour le développement, les tests et la production. Ces tenants devraient avoir des configurations identiques pour les vérifications de qualité et les tests; toutefois, des erreurs peuvent survenir si les configurations ne correspondent pas d’un environnement à l’autre. Par exemple, chaque environnement aura des <Tooltip tip="Client ID : valeur d’identification attribuée à votre ressource enregistrée par Auth0." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Client+IDs">ID client</Tooltip> et des <Tooltip tip="Client ID : valeur d’identification attribuée à votre ressource enregistrée par Auth0." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Client+Secrets">secrets client</Tooltip> différents.

Pour réduire ces erreurs de correspondance, vous pouvez utiliser le Deploy CLI Tool pour intégrer votre instance Auth0 à votre pipeline CI/CD existant. Grâce au remplacement dynamique de mots-clés, vous pouvez remplacer des variables d’environnement dans des tenants qui partagent des configurations semblables. Pour en savoir plus, consultez [Deploy CLI Tool](/docs/fr-CA/deploy-monitor/deploy-cli-tool) et [Keyword Replacement](/docs/fr-CA/deploy-monitor/deploy-cli-tool/keyword-replacement).

<div id="actions-real-time-logs">
  ### Logs en temps réel d’Actions
</div>

Les logs en temps réel d’Actions affichent tous les logs du code personnalisé en temps réel, y compris la sortie de `console.log` et les autres exceptions. Si vous utilisez Auth0 Actions ou une autre logique personnalisée, vous pouvez utiliser cette extension pour déboguer et résoudre les problèmes. Pour en savoir plus sur l’installation et la configuration, consultez [Logs en temps réel d’Actions](/docs/fr-CA/customize/actions/actions-real-time-logs).

<div id="quality-assurance">
  ## Assurance qualité
</div>

L’assurance qualité est essentielle pour repérer les problèmes avant la mise en ligne. Selon la nature de votre projet, il existe plusieurs types de tests d’assurance qualité à envisager dans le cadre de votre intégration avec Auth0 :

* Comment vos API se comporteront-elles lorsqu’elles seront soumises à des pointes de charge imprévues en production ?
* Quel effet les applications tierces auront-elles sur vos limites de débit ?

Si vous n’utilisez pas de widgets ou de fonctionnalités d’Auth0, comme <Tooltip tip="Universal Login : votre application redirige vers Universal Login, hébergé sur le serveur d’autorisation d’Auth0, pour vérifier l’identité d’un utilisateur." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Universal+Login">Universal Login</Tooltip>, vous ne bénéficierez pas d’emblée des pratiques exemplaires intégrées en matière de convivialité et d’accessibilité sur un large éventail de navigateurs et d’appareils. Pour vous assurer que les exigences fonctionnelles sont respectées et que les événements inattendus sont gérés correctement, des directives sont fournies pour tester l’intégration entre vos application(s) et Auth0, ainsi que pour effectuer des tests unitaires sur des modules d’extensibilité individuels, comme Auth0 Actions. Nous vous recommandons également de consulter la [politique relative aux tests d’intrusion](/docs/fr-CA/troubleshoot/customer-support/operational-policies/penetration-testing-policy) d’Auth0 et d’effectuer des tests simulés, que vous pouvez mener conjointement avec notre [politique relative aux tests de charge](/docs/fr-CA/troubleshoot/customer-support/operational-policies/load-testing-policy), afin de contribuer à garantir le bon fonctionnement de votre ou vos application(s) en cas de charge imprévue.

<div id="unit-testing">
  ### Tests unitaires
</div>

Les tests unitaires servent à vérifier des unités d’extensibilité, comme les Auth0 Actions. Si vous utilisez du code personnalisé, nous vous recommandons d’utiliser un framework de test (comme [Mocha](https://mochajs.org/)) pour tester ce code avant le déploiement.

<div id="mock-testing">
  ### Tests simulés
</div>

Pour concilier la [load testing policy](/docs/fr-CA/troubleshoot/customer-support/operational-policies/load-testing-policy) d’Auth0 et le besoin d’effectuer des tests de charge, il est courant de mettre en place un test simulé des points de terminaison d’Auth0. C’est une pratique utile pour s’assurer que votre configuration fonctionne avec les interfaces prévues, sans avoir à limiter vos tests, et des outils comme [MockServer](http://www.mock-server.com/), [JSON Server](https://github.com/typicode/json-server), ou même [Postman](https://learning.getpostman.com/docs/postman/mock_servers/setting_up_mock/) peuvent vous y aider.

<div id="deployment">
  ## Déploiement
</div>

Notre section [Déploiement et surveillance](/docs/fr-CA/deploy-monitor) fournit des conseils sur les meilleures pratiques de déploiement. Nous vous recommandons de consulter les [Vérifications préalables au déploiem](/docs/fr-CA/deploy-monitor/pre-deployment-checks)[ent](/docs/fr-CA/deploy-monitor/pre-deployment-checks), en particulier les [Readiness Checks intégrés de l’Auth0 Dashboard](/docs/fr-CA/deploy-monitor/pre-deployment-checks/how-to-run-production-checks).

Pour consulter la vérification de préparation, sélectionnez le menu déroulant sous le nom de votre tenant et l’étiquette d’environnement dans [Auth0 Dashboard &gt; Run Readiness Checks](https://manage.auth0.com/#/production-checks).

<Frame>![Auth0 Dashboard > Liste de vérification de préparation](/docs/images/cdy7uua7fh8z/7iu1CzH0NgaXJbLBNWXZli/b598b8cdaad0c676e83a0fb0595d4603/Readiness_Checks_-_English.png)</Frame>

Vous pouvez utiliser le filtre pour appliquer les vérifications de préparation aux applications sélectionnées. **Ces vérifications ne s’appliquent pas aux API que vous avez configurées**.

Pour les vérifications qui ne s’appliquent pas à votre configuration, vous pouvez sélectionner **Supprimer** pour les retirer des résultats finaux.

Nous vous recommandons de consulter les [meilleures pratiques de déploiement](/docs/fr-CA/deploy-monitor/deployment-best-practices) pour effectuer les dernières vérifications avant la mise en production, et de tirer parti des [Logs](/docs/fr-CA/deploy-monitor/logs) pour surveiller vos services.
