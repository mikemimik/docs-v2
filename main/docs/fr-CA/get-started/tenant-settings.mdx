Utilisez la page **Paramètres du tenant** dans le <Tooltip tip="Auth0 Dashboard : le principal produit d’Auth0 pour configurer vos services." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip>, à [Dashboard &gt; Paramètres](https://manage.auth0.com/#/tenant), pour configurer différents paramètres liés à votre tenant Auth0.

<div id="recommended-settings">
  ## Paramètres recommandés
</div>

Lorsque vous configurez votre tenant, définissez les éléments suivants :

* **Spécifiez l’étiquette d’environnement.** Les tenants identifiés comme **Production** bénéficient de limites de débit plus élevées que les tenants identifiés comme **Development** ou **Staging**. Avec les forfaits non Enterprise, un seul tenant par abonnement peut être identifié comme **Production**. Pour en savoir plus, consultez [Set Up Multiple Environments](/docs/fr-CA/get-started/auth0-overview/create-tenants/set-up-multiple-environments).
* **Définissez le Courriel de soutien et l’URL de soutien.** Si un utilisateur rencontre un problème au moment de se connecter, il voudra obtenir de l’aide. Définissez ces valeurs pour le diriger vers une adresse courriel ou une page de destination où il pourra obtenir de l’assistance.
* **Configurez une page d’erreur personnalisée.** Si possible, vous devriez héberger votre propre page d’erreur personnalisée et configurer Auth0 pour l’utiliser à la place de la page par défaut. Cela vous permet de fournir aux utilisateurs des explications plus complètes et adaptées sur ce qu’ils doivent faire en cas d’erreur.
* **Configurez un <Tooltip tip="Custom Domain : domaine tiers avec un nom spécialisé ou personnalisé." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=custom+domain">domaine personnalisé</Tooltip>.** Si vous avez un forfait payant, vous pouvez configurer un domaine personnalisé pour votre tenant Auth0. Un domaine personnalisé harmonise l’expérience de connexion avec votre marque et offre des avantages supplémentaires. Pour en savoir plus, consultez [Domaines personnalisés](/docs/fr-CA/customize/custom-domains).
* **Définissez le délai d’expiration de la session <Tooltip tip="Single Sign-On (SSO) : service qui, après qu’un utilisateur s’est connecté à une application, le connecte automatiquement à d’autres applications." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Single+Sign-On">Single Sign-On</Tooltip> (SSO).** La valeur du délai d’expiration de la session SSO indique après combien de temps la session d’un utilisateur expire. Par défaut, cette valeur est de 7 jours, soit la période pendant laquelle les utilisateurs peuvent accéder à vos applications intégrées à Auth0 sans saisir de nouveau leurs identifiants. Pour en savoir plus, consultez [Sessions](/docs/fr-CA/manage-users/sessions).
* **Configurez des membres du tenant.** Configurez d’autres utilisateurs de l’Auth0 Dashboard et activez l’<Tooltip tip="authentification multifacteur (MFA) : processus d’authentification de l’utilisateur qui utilise un facteur en plus du nom d’utilisateur et du mot de passe, comme un code envoyé par SMS." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=multi-factor+authentication">authentification multifacteur</Tooltip> (MFA). Pour en savoir plus, consultez [Gérer l’accès au Dashboard](/docs/fr-CA/get-started/manage-dashboard-access) et [Manage Dashboard Access with Multi-Factor Authentication](/docs/fr-CA/get-started/manage-dashboard-access/add-change-remove-mfa).
* **Désactivez le paramètre Enable Application Connections.** Si ce paramètre est activé, toutes les connexions configurées seront automatiquement activées pour toute nouvelle application que vous créerez. Par conséquent, les utilisateurs pourraient être en mesure de se connecter à l’application au moyen de connexions que vous ne vouliez pas rendre disponibles. Désactivez ce paramètre afin de pouvoir activer explicitement les connexions appropriées pour chaque application.
* **Activez la <Tooltip tip="Attack Protection : fonctionnalités qu’Auth0 offre pour détecter et atténuer les attaques, y compris la protection contre les attaques par force brute, la limitation des IP suspectes, la détection des mots de passe compromis, la détection des robots et l’authentification multifacteur adaptative." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Attack+Protection">protection contre les attaques</Tooltip>.** Protégez vos utilisateurs contre les attaques par force brute et les mots de passe compromis. Pour en savoir plus, consultez [Attack Protection](/docs/fr-CA/secure/attack-protection).

* **Nom affiché** : Nom que vous souhaitez afficher à vos utilisateurs sur la page <Tooltip tip="Universal Login : votre application redirige vers Universal Login, hébergé sur le serveur d’autorisation d’Auth0, pour vérifier l’identité d’un utilisateur." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Universal+Login">Universal Login</Tooltip>. * **Courriel de soutien** : Adresse courriel utilisée pour communiquer avec votre équipe de soutien.
* **URL de soutien** : Lien vers la page de soutien de votre entreprise ou de votre organisation.

<div id="api-authorization-settings">
  ### Paramètres d’autorisation de l’API
</div>

<Frame>![Paramètres d’autorisation de l’API dans l’onglet General de Tenant Settings du Dashboard](/docs/images/cdy7uua7fh8z/17yNCRrB6dWcHp6zLFb74j/d1f7c516bce3bf66e9f4bfbd201411a3/Auth_Settings_-_EN.png)</Frame>

* **Audience par défaut <Tooltip tip="Audience : identificateur unique de l’audience d’un jeton émis. Appelée aud dans un jeton, sa valeur contient l’ID d’une application (Client ID) pour un ID Token ou d’une API (API Identifier) pour un jeton d’accès." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Audience">Audience</Tooltip>** : identificateur d’API à utiliser pour les [flux d’autorisation](/docs/fr-CA/get-started/authentication-and-authorization-flow). Si vous entrez une valeur, tous les <Tooltip tip="Jeton d’accès : information d’autorisation, sous la forme d’une chaîne opaque ou d’un JWT, utilisée pour accéder à une API." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=access+tokens">jetons d’accès</Tooltip> émis par Auth0 indiqueront cet identificateur d’API comme audience. Définir l’**Audience par défaut** revient à ajouter cette audience à chaque demande d’autorisation envoyée à votre tenant pour chaque application. Cela entraînera un nouveau comportement qui pourrait causer des changements non rétrocompatibles pour certaines de vos applications. Veuillez [contacter le soutien](/docs/fr-CA/troubleshoot/customer-support/open-and-manage-support-tickets) si vous avez besoin d’aide.
* **Répertoire par défaut** : nom de la connection par défaut à utiliser à la fois pour le [flux Resource Owner Password](/docs/fr-CA/get-started/authentication-and-authorization-flow/resource-owner-password-flow) et l’[expérience Universal Login](/docs/fr-CA/authenticate/login/auth0-universal-login/universal-login-vs-classic-login/universal-experience). Sa valeur doit correspondre exactement au nom d’une connection existante pour l’une des stratégies suivantes : `auth0-adldap`, `ad`, `auth0`, `email`, `sms`, `waad` ou `adfs`.

<div id="signing-keys">
  ## Clés de signature
</div>

Dans l’onglet **Clés de signature**, vous pouvez gérer de façon sécuritaire la clé de signature et le certificat utilisés pour signer les <Tooltip tip="ID Token : jeton destiné au client lui-même, plutôt qu’à l’accès à une ressource." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=ID+tokens">jetons ID</Tooltip>, les jetons d’accès, les assertions <Tooltip tip="Security Assertion Markup Language (SAML) : protocole normalisé permettant à deux parties d’échanger des renseignements d’authentification sans mot de passe." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=SAML">SAML</Tooltip> et les assertions <Tooltip tip="Security Assertion Markup Language (SAML) : protocole normalisé permettant à deux parties d’échanger des renseignements d’authentification sans mot de passe." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=WS-Fed">WS-Fed</Tooltip> envoyées à vos applications.

<Frame>![Onglet Clés de signature des Tenant Settings du Dashboard](/docs/images/cdy7uua7fh8z/7r8t3EGctFmvkCgPrU0i2R/f79ac74dac5cded37b39bd8a8b80951d/dashboard-tenant-settings-signing-keys.png)</Frame>

* **Paramètres de rotation** : paramètres qui vous permettent d’effectuer la rotation de la clé de signature et du certificat de l’application. Vous pouvez choisir de révoquer ou non la clé de signature au moment de la rotation. Pour en savoir plus, consultez [Clés de signature](/docs/fr-CA/get-started/tenant-settings/signing-keys).

  * **Faire la rotation de la clé de signature** : effectue la rotation de la clé de signature sans la révoquer; en pratique, la clé actuelle devient la clé précédente. Tous les jetons signés avec la clé précédente resteront valides jusqu’à sa révocation.
  * **Faire la rotation et révoquer la clé de signature** : effectue la rotation de la clé de signature, puis la révoque; en pratique, la clé actuelle devient la clé précédente, puis la clé précédente est invalidée. Assurez-vous d’avoir mis à jour votre application avec la prochaine clé dans la file d’attente avant d’effectuer la rotation et de révoquer la clé actuelle.
* **Liste des clés valides** : liste des clés de signature d’application valides pour votre tenant, qui sont aussi disponibles au point de terminaison Metadata de votre application. Les clés valides comprennent :

  * **Prochaine dans la file d’attente** : clé qui sera utilisée lors de la prochaine rotation de la clé de signature.
  * **Actuellement utilisée** : clé actuellement utilisée.
  * **Précédemment utilisée** : clé qui a été utilisée auparavant. Sa présence indique que la clé de signature a fait l’objet d’une rotation, mais que la clé utilisée précédemment n’a pas encore été révoquée.
* **Liste des clés révoquées** : liste des trois dernières clés révoquées pour votre tenant. Plus de renseignements sur les clés révoquées sont disponibles dans les journaux du tenant.

<div id="login-and-logout">
  ### Connexion et déconnexion
</div>

<Frame>![Onglet Advanced des Tenant Settings du Dashboard : Login and Logout](/docs/images/cdy7uua7fh8z/Login_and_Logout_-_EN.png)</Frame>

* **URI de connexion du tenant** : URI qui pointe vers une route de votre application lançant le flux de connexion OIDC en redirigeant vers le point de terminaison `/authorize`; elle doit prendre la forme `https://mytenant.org/login`. Elle n’est utilisée que dans les cas où Auth0 a besoin que votre tenant lance le flux de connexion OIDC. Pour en savoir plus, consultez [Configurer les routes de connexion par défaut](/docs/fr-CA/authenticate/login/auth0-universal-login/configure-default-login-routes).
* **Allowed Logout URLs** : URL vers lesquelles Auth0 peut rediriger après la déconnexion lorsqu’aucun client&#95;id n’est précisé lors de l’appel au point de terminaison Logout. Utile comme liste globale lorsque le Single Sign-On (SSO) est activé. Pour en savoir plus, consultez [Logout](docs/authenticate/login/logout).
* **Valeurs ACR autorisées** : Liste des Authentication Context Class Reference (ACR) autorisées. Ces valeurs sont incluses dans le document OpenID Configuration. Lorsqu’elle est remplie, toute valeur qui n’y figure pas sera rejetée si elle est utilisée dans des flux d’authentification.
* **Découverte du point de terminaison de fin de session RP-Initiated Logout** : Détermine si le point de terminaison de logout est annoncé dans les réponses OpenID Configuration sous la forme `end_session_endpoint`.
* **Confirmation de l’utilisateur final pour RP-Initiated Logout** : Détermine si l’utilisateur doit confirmer la connexion si la requête de logout RP-Initiated n’inclut pas les bons paramètres d’indication.
* **Non-Verifiable Callback URI End-User Confirmation** : Détermine si l’utilisateur est invité à confirmer la connexion lorsqu’un custom URI scheme est utilisé comme callback. Auth0 recommande de **ne pas** ignorer la confirmation de l’utilisateur final dans ces cas. Pour en savoir plus, consultez [Measures Against Application Impersonation](docs/secure/security-guidance/measures-against-app-impersonation.mdx).

<div id="login-session-management">
  ### Gestion de la session de connexion
</div>

Les paramètres de **Gestion de la session de connexion** configurent la durée de la session de connexion, qui correspond à la couche de session du <Tooltip tip="Authorization Server : serveur centralisé qui contribue à définir les limites de l’accès d’un utilisateur. Par exemple, votre serveur d’autorisation peut contrôler les données, les tâches et les fonctionnalités accessibles à un utilisateur." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Authorization+Server">serveur d’autorisation</Tooltip> d’Auth0. La couche de session du serveur d’autorisation gère l’authentification unique (SSO). Pour en savoir plus, consultez [Authentification unique](/docs/fr-CA/authenticate/single-sign-on).

Les délais d’expiration des jetons émis par Auth0 peuvent être configurés ailleurs. Ils servent souvent à gérer la couche de session de l’application et apparaissent dans les claims du jeton, par exemple dans le claim d’expiration des jetons ID <Tooltip tip="OpenID : norme ouverte d’authentification qui permet aux applications de vérifier l’identité des utilisateurs sans recueillir ni stocker les renseignements de connexion." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=OpenID">OpenID</Tooltip> Connect (OIDC) ou dans l’assertion de durée de vie pour SAML.

<Frame>![Dashboard Tenant Settings Advanced Login Session Management](/docs/images/cdy7uua7fh8z/q2fGPzoUqCcXj7OxcHhjy/de15afcd26e8ea80937733034ffbd387/Screenshot_2024-11-06_at_11.02.16.png)</Frame>

* **Délai d’inactivité** : période (en minutes) après laquelle la session d’un utilisateur expire s’il n’a pas interagi avec le serveur d’autorisation. La valeur sera remplacée par les limites du système si elle dépasse 4 320 minutes (3 jours) pour les forfaits non Enterprise ou 144 000 minutes (100 jours) pour les forfaits Enterprise.
* **Exiger une nouvelle connexion après** : période (en minutes) après laquelle un utilisateur devra se reconnecter, peu importe son activité. La valeur sera remplacée par les limites du système si elle dépasse 43 200 minutes (30 jours) pour les forfaits non Enterprise ou 525 600 minutes (365 jours) pour les forfaits Enterprise.

<div id="settings-advanced">
  ### Paramètres (avancés)
</div>

* **Change Password Flow v2** : Lorsqu’elle est activée, la plus récente version du Change Password Flow sera utilisée. La version précédente est obsolète, et nous recommandons fortement d’activer v2. Cette option est offerte uniquement pour assurer la rétrocompatibilité et, une fois activée, vous ne pourrez plus la désactiver. Vous pouvez personnaliser l’interface utilisateur du Change Password widget dans l’onglet [Universal Login &gt; Password Reset](https://manage.auth0.com/#/password_reset) de l’Auth0 Dashboard.
* **Dynamic Client Registration (DCR)** : Lorsqu’elle est activée, les développeurs tiers peuvent enregistrer dynamiquement des applications pour vos API. Vous pouvez également mettre à jour cet indicateur au moyen du point de terminaison [`/tenant/patch_settings`](https://auth0.com/docs/api/management/v2#!/Tenants/patch_settings) de la <Tooltip tip="Management API : Un produit qui permet aux clients d’effectuer des tâches administratives." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Management+API">Management API</Tooltip> d’Auth0. Par défaut, cette fonctionnalité est désactivée. Pour en savoir plus, consultez [Dynamic Client Registration](/docs/fr-CA/get-started/applications/dynamic-client-registration).
* **Profil de compatibilité du paramètre de ressource** : le profil de compatibilité du paramètre de ressource définit la manière dont Auth0 traite le paramètre `resource` dans les requêtes d’autorisation.
  * Lorsqu’il est activé :
    * Auth0 utilise le paramètre `resource` pour indiquer à quel serveur de ressources (API) l’application cliente souhaite accéder
    * Auth0 vérifie d’abord le paramètre `audience` ; s’il n’est pas fourni, il utilise le paramètre `resource` à défaut
    * Le paramètre `resource` est utilisé par Auth0 et n’est pas transmis à l’IdP en amont
  * Lorsqu’il est désactivé (par défaut) :
    * Auth0 utilise uniquement le paramètre `audience` pour indiquer le serveur de ressources
    * Le paramètre `resource` est traité comme un paramètre de l’IdP en amont et sera transmis à l’IdP
* **Enregistrement du Client ID Metadata Document (CIMD)** : Activez l’enregistrement manuel du CIMD en important depuis une URL un Client ID Metadata Document (CIMD) hébergé à l’externe. Le CIMD est un fichier JSON qui contient les métadonnées du client et qui est hébergé sur un domaine HTTPS sécurisé contrôlé par l’application. Pour en savoir plus, consultez [Enregistrer des applications avec CIMD](/docs/fr-CA/get-started/auth0-overview/create-applications/register-applications-with-cimd).
* **Activer les connexions de l’application** : Lorsque cette option est activée, toutes les connexions actuelles seront activées pour toute nouvelle application créée.
* **Utiliser une réponse générique dans le message d’erreur de l’API publique d’inscription** : Lorsqu’elle est activée, les erreurs générées lors de l’utilisation de l’API publique d’inscription renverront une réponse générique. Cela contribue à protéger contre l’énumération des inscriptions d’utilisateurs en empêchant les <Tooltip tip="Bad Actors : entité (une personne ou un groupe) qui représente une menace pour l’entreprise ou l’environnement avec l’intention de causer du tort." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=bad+actors">acteurs malveillants</Tooltip> de deviner des identifiants déjà enregistrés (nom d’utilisateur, courriel ou numéro de téléphone) en lisant les codes de réponse d’erreur, comme `user_exists`.
* **Activer la publication des renseignements sur les connexions d’entreprise avec les domaines d’<Tooltip tip="Identity Provider (IdP) : Service qui stocke et gère les identités numériques." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=IdP">IdP</Tooltip>** : Lorsqu’elle est activée, cette option prend en charge [Home Realm Discovery](/docs/fr-CA/authenticate/login/auth0-universal-login/identifier-first), et [Auth0 Lock](/docs/fr-CA/libraries/lock) utilise un fichier public vérifié qui contient des renseignements sur les connexions d’entreprise. Si vous n’avez pas besoin de cette fonctionnalité, vous pouvez la désactiver.
* **Activer le flux de vérification du courriel à la connexion pour les connexions Azure AD et ADFS** : Lorsque cette option est activée, les utilisateurs verront l’écran de vérification du courriel lors de leur première connexion au moyen d’une connexion Azure AD ou ADFS.
* **<Tooltip tip="Refresh Token : token utilisé pour obtenir un nouvel Access Token sans obliger les utilisateurs à se reconnecter." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Refresh+Token">Refresh Token</Tooltip> La révocation supprime l’autorisation** : Lorsqu’elle est activée, elle supprime l’autorisation sous-jacente lorsque vous révoquez un Refresh Token à l’aide du point de terminaison `/oauth/revoke` de l’Authentication API.

  <Callout icon="file-lines" color="#0EA5E9" iconType="regular">
    Pour les tenants existants, cette fonctionnalité est activée par défaut afin de préserver le comportement existant. Pour les nouveaux tenants (à partir du 13 janvier 2021), cette fonctionnalité est désactivée par défaut afin de garantir que la révocation d’un jeton d’actualisation n’entraînera pas la révocation du grant. Si la révocation d’un grant est nécessaire, une requête distincte doit être envoyée au moyen d’un endpoint de révocation de grant.
  </Callout>
* **Autoriser les noms d’organisation dans l’Authentication API** : Lorsqu’elle est activée, les endpoints [/authorize](https://auth0.com/docs/api/authentication#authorize-application) et [SAML](https://auth0.com/docs/api/authentication#saml) peuvent accepter à la fois les ID et les noms d’organisation. De plus, les jetons d’ID et d’accès incluront les claims `org_id` et `org_name`. Avant d’activer ce paramètre, consultez [Use Organization Names in Authentication API](/docs/fr-CA/manage-users/organizations/configure-organizations/use-org-name-authentication-api) pour prendre connaissance des considérations importantes et des répercussions possibles.
* **Autoriser les Pushed Authorization Requests (PAR) :** Lorsqu’elle est activée, l’endpoint `/par` peut accepter les requêtes d’autorisation qui lui sont envoyées par une application cliente. Cela empêche l’application cliente d’envoyer des requêtes par le front channel non sécurisé (c.-à-d. le navigateur).
