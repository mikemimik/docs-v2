Utilisez la page **Paramètres du tenant** dans l’<Tooltip tip="Auth0 Dashboard : le principal produit d’Auth0 pour configurer vos services." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip>, sous [Dashboard &gt; Settings](https://manage.auth0.com/#/tenant), pour configurer divers paramètres liés à votre tenant Auth0.

<div id="recommended-settings">
  ## Paramètres recommandés
</div>

Lorsque vous configurez votre tenant, définissez les éléments suivants :

* **Spécifiez la balise d’environnement.** Les tenants marqués **Production** bénéficient de limites de taux plus élevées que les tenants marqués **Development** ou **Staging**. Avec les forfaits non Enterprise, un seul tenant par abonnement peut être marqué **Production**. Pour en savoir plus, consultez [Configurer plusieurs environnements](/docs/fr-CA/get-started/auth0-overview/create-tenants/set-up-multiple-environments).
* **Définissez le courriel de soutien et l’URL de soutien.** Si un utilisateur rencontre un problème lors de la connexion, il voudra obtenir de l’aide. Définissez ces valeurs pour le diriger vers une adresse courriel ou une page de destination où obtenir du soutien.
* **Configurez une page d’erreur personnalisée.** Si possible, vous devriez héberger votre propre page d’erreur personnalisée et configurer Auth0 pour l’utiliser à la place de la page par défaut. Cela vous permet de fournir aux utilisateurs des explications plus complètes et personnalisées sur la marche à suivre en cas d’erreur.
* **Configurez un <Tooltip tip="Domaine personnalisé : domaine tiers avec un nom spécialisé ou personnalisé." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=custom+domain">domaine personnalisé</Tooltip>.** Si vous avez un forfait payant, vous pouvez configurer un domaine personnalisé pour votre tenant Auth0. Un domaine personnalisé harmonise l’expérience de connexion avec votre marque et offre des avantages supplémentaires. Pour en savoir plus, consultez [Domaines personnalisés](/docs/fr-CA/customize/custom-domains).
* **Définissez le délai d’expiration de la session <Tooltip tip="Single Sign-On (SSO) : service qui, après qu’un utilisateur s’est connecté à une application, connecte automatiquement cet utilisateur à d’autres applications." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Single+Sign-On">Single Sign-On</Tooltip> (SSO).** Le délai d’expiration de la session SSO indique après combien de temps la session d’un utilisateur expire. Par défaut, cette valeur est de 7 jours, soit la durée pendant laquelle les utilisateurs peuvent accéder à vos applications intégrées à Auth0 sans saisir de nouveau leurs données d’identification. Pour en savoir plus, consultez [Sessions](/docs/fr-CA/manage-users/sessions).
* **Configurez les membres du tenant.** Configurez des utilisateurs supplémentaires de l’Auth0 Dashboard et activez l’<Tooltip tip="Authentification multifacteur (MFA) : processus d’authentification de l’utilisateur qui utilise un facteur en plus du nom d’utilisateur et du mot de passe, comme un code envoyé par SMS." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=multi-factor+authentication">authentification multifacteur</Tooltip> (MFA). Pour en savoir plus, consultez [Gérer l’accès au Dashboard](/docs/fr-CA/get-started/manage-dashboard-access) et [Gérer l’accès au Dashboard avec l’authentification multifacteur](/docs/fr-CA/get-started/manage-dashboard-access/add-change-remove-mfa).
* **Désactivez le paramètre Enable Application Connections.** Si ce paramètre est activé, toutes les connexions configurées seront automatiquement activées pour toute nouvelle application que vous créez. Par conséquent, les utilisateurs pourraient être en mesure de se connecter à l’application par l’intermédiaire de connexions que vous n’aviez pas l’intention de rendre disponibles. Désactivez ce paramètre afin de pouvoir activer explicitement les connexions appropriées pour chaque application.
* **Activez la <Tooltip tip="Attack Protection : fonctionnalités qu’Auth0 fournit pour détecter et atténuer les attaques, y compris la protection contre les attaques par force brute, le throttling des adresses IP suspectes, la détection de mots de passe compromis, la détection de robots et l’authentification multifacteur adaptative." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Attack+Protection">protection contre les attaques</Tooltip>.** Protégez vos utilisateurs contre les attaques par force brute et les mots de passe compromis. Pour en savoir plus, consultez [Protection contre les attaques](/docs/fr-CA/secure/attack-protection).

  ### Paramètres
</div>

<Frame>![Onglet General Settings de Tenant Settings dans le Dashboard](/docs/images/cdy7uua7fh8z/4okToiwlkNQBxwr8QGE3Rs/7e96db0efab4082ebb9f78cb43ee302c/Basic_Settings_-_EN.png)</Frame>

* **Nom affiché** : Nom à afficher aux utilisateurs sur la page <Tooltip tip="Universal Login : votre application redirige les utilisateurs vers Universal Login, hébergé sur le serveur d’autorisation d’Auth0, pour vérifier leur identité." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Universal+Login">Universal Login</Tooltip>. * **URL de soutien** : Lien vers la page de soutien de votre entreprise ou de votre organisation.

<div id="api-authorization-settings">
  ### Paramètres d’autorisation de l’API
</div>

<Frame>![Paramètres d’autorisation de l’API dans l’onglet Général des paramètres du tenant du Dashboard](/docs/images/cdy7uua7fh8z/17yNCRrB6dWcHp6zLFb74j/d1f7c516bce3bf66e9f4bfbd201411a3/Auth_Settings_-_EN.png)</Frame>

* **Audience par défaut <Tooltip tip="Audience : identifiant unique de l’audience d’un jeton émis. Appelée aud dans un jeton, sa valeur contient l’ID d’une application (Client ID) pour un ID Token ou d’une API (API Identifier) pour un jeton d’accès." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Audience">Audience</Tooltip>** : identifiant d’API à utiliser pour les [flux d’autorisation](/docs/fr-CA/get-started/authentication-and-authorization-flow). Si vous entrez une valeur, tous les <Tooltip tip="Access Token : jeton d’autorisation, sous forme de chaîne opaque ou de JWT, utilisé pour accéder à une API." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=access+tokens">jetons d’accès</Tooltip> émis par Auth0 préciseront cet identifiant d’API comme audience. Définir l’**Audience par défaut** revient à ajouter cette audience à chaque demande d’autorisation envoyée à votre tenant pour chaque application. Cela entraînera un nouveau comportement qui pourrait causer des changements incompatibles pour certaines de vos applications. Veuillez [contacter le soutien](/docs/fr-CA/troubleshoot/customer-support/open-and-manage-support-tickets) si vous avez besoin d’aide.
* **Répertoire par défaut** : nom de la connexion par défaut à utiliser à la fois pour le [Resource Owner Password Flow](/docs/fr-CA/get-started/authentication-and-authorization-flow/resource-owner-password-flow) et l’[expérience Universal Login](/docs/fr-CA/authenticate/login/auth0-universal-login/universal-login-vs-classic-login/universal-experience). Sa valeur doit correspondre exactement au nom d’une connexion existante pour l’une des stratégies suivantes : `auth0-adldap`, `ad`, `auth0`, `email`, `sms`, `waad` ou `adfs`.

<div id="signing-keys">
  ## Clés de signature
</div>

Dans l’onglet **Clés de signature**, vous pouvez gérer en toute sécurité la clé de signature et le certificat utilisés pour signer les <Tooltip tip="ID Token : Jeton destiné au client lui-même, plutôt qu’à l’accès à une ressource." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=ID+tokens">jetons d’identification</Tooltip>, les jetons d’accès, les assertions <Tooltip tip="Security Assertion Markup Language (SAML) : Protocole normalisé permettant à deux parties d’échanger des renseignements d’authentification sans mot de passe." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=SAML">SAML</Tooltip> et les assertions <Tooltip tip="Security Assertion Markup Language (SAML) : Protocole normalisé permettant à deux parties d’échanger des renseignements d’authentification sans mot de passe." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=WS-Fed">WS-Fed</Tooltip> envoyées à vos applications.

<Frame>![Onglet Clés de signature des paramètres du tenant dans le Dashboard](/docs/images/cdy7uua7fh8z/7r8t3EGctFmvkCgPrU0i2R/f79ac74dac5cded37b39bd8a8b80951d/dashboard-tenant-settings-signing-keys.png)</Frame>

* **Paramètres de rotation** : Paramètres qui vous permettent d’effectuer la rotation de la clé de signature et du certificat de l’application. Vous pouvez choisir de révoquer ou non la clé de signature lors de la rotation. Pour en savoir plus, consultez [Clés de signature](/docs/fr-CA/get-started/tenant-settings/signing-keys).

  * **Effectuer la rotation de la clé de signature** : Effectue la rotation de la clé de signature sans la révoquer; en pratique, la clé actuelle devient la clé précédente. Tous les jetons signés avec la clé précédente demeureront valides jusqu’à sa révocation.
  * **Effectuer la rotation et révoquer la clé de signature** : Effectue la rotation de la clé de signature, puis la révoque; en pratique, la clé actuelle devient la clé précédente, puis la clé précédente est invalidée. Assurez-vous d’avoir mis à jour votre application avec la prochaine clé dans la file d’attente avant d’effectuer la rotation et de révoquer la clé actuelle.
* **Liste des clés valides** : Liste des clés de signature d’application valides pour votre tenant, qui sont aussi accessibles au point de terminaison Metadata de votre application. Les clés valides comprennent :

  * **Prochaine dans la file d’attente** : Clé qui sera utilisée lors de la prochaine rotation de la clé de signature.
  * **Actuellement utilisée** : Clé actuellement utilisée.
  * **Précédemment utilisée** : Clé qui était utilisée auparavant. Sa présence indique que la clé de signature a fait l’objet d’une rotation, mais que la clé utilisée précédemment n’a pas encore été révoquée.
* **Liste des clés révoquées** : Liste des trois dernières clés révoquées pour votre tenant. Plus de renseignements sur les clés révoquées sont disponibles dans les journaux du tenant.

  ## Advanced

<div id="login-and-logout">
  ### Connexion et déconnexion
</div>

<Frame>![Dashboard Paramètres du tenant Onglet Avancé Connexion et déconnexion](/docs/images/cdy7uua7fh8z/Login_and_Logout_-_EN.png)</Frame>

* **URI de connexion du tenant** : URI qui pointe vers une route de votre application lançant le flux de connexion OIDC en redirigeant vers le point de terminaison `/authorize` ; elle doit avoir la forme `https://mytenant.org/login`. Elle n’est utilisée que dans les cas où Auth0 a besoin que votre tenant lance le flux de connexion OIDC. Pour en savoir plus, consultez [Configurer les routes de connexion par défaut](/docs/fr-CA/authenticate/login/auth0-universal-login/configure-default-login-routes).
* **URL de déconnexion autorisées** : URL vers lesquelles Auth0 peut rediriger après la déconnexion lorsqu’aucun client&#95;id n’est précisé lors de l’appel au point de terminaison Logout. Utile comme liste globale lorsque l’authentification unique (SSO) est activée. Pour en savoir plus, consultez [Déconnexion](docs/authenticate/login/logout).
* **Valeurs ACR autorisées** : Liste des valeurs de Authentication Context Class Reference (ACR) autorisées. Ces valeurs sont incluses dans le document de configuration OpenID. Lorsqu’elle est renseignée, toute valeur qui n’y figure pas sera rejetée si elle est utilisée dans des flux d’authentification.
* **RP-Initiated Logout End Session Endpoint Discovery** : Contrôle si le point de terminaison de déconnexion est annoncé dans les réponses de configuration OpenID sous la forme `end_session_endpoint`.
* **RP-Initiated Logout End-User Confirmation** : Contrôle si l’utilisateur doit confirmer la connexion si la requête de logout RP-Initiated n’inclut pas les bons indices.
* **Non-Verifiable Callback URI End-User Confirmation** : Contrôle si l’utilisateur est invité à confirmer la connexion lorsqu’un schéma d’URI personnalisé est utilisé comme URI de rappel. Auth0 recommande de **ne pas** ignorer la confirmation de l’utilisateur final dans ces cas. Pour en savoir plus, consultez [Measures Against Application Impersonation](docs/secure/security-guidance/measures-against-app-impersonation.mdx).

<div id="login-session-management">
  ### Gestion de la session de connexion
</div>

Les paramètres de **Gestion de la session de connexion** configurent la durée de la session de connexion, qui correspond à la couche de session de l’<Tooltip tip="Authorization Server : serveur centralisé qui contribue à définir les limites de l’accès d’un utilisateur. Par exemple, votre serveur d’autorisation peut contrôler les données, les tâches et les fonctionnalités auxquelles un utilisateur a accès." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Authorization+Server">Authorization Server</Tooltip> d’Auth0. La couche de session de l’authorization server pilote l’authentification unique (SSO). Pour en savoir plus, consultez [Authentification unique](/docs/fr-CA/authenticate/single-sign-on).

Les délais d’expiration des jetons émis par Auth0 peuvent être configurés ailleurs. Ils servent souvent à gérer la couche de session de l’application et figurent dans les claims du jeton, par exemple dans le claim d’expiration des <Tooltip tip="OpenID : norme ouverte d’authentification qui permet aux applications de vérifier l’identité des utilisateurs sans collecter ni stocker d’informations de connexion." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=OpenID">OpenID</Tooltip> Connect (OIDC) ID tokens ou dans l’assertion de durée de vie pour SAML.

<Frame>![Dashboard Tenant Settings Advanced Login Session Management](/docs/images/cdy7uua7fh8z/q2fGPzoUqCcXj7OxcHhjy/de15afcd26e8ea80937733034ffbd387/Screenshot_2024-11-06_at_11.02.16.png)</Frame>

* **Délai d’inactivité** : Période (en minutes) après laquelle la session d’un utilisateur expire s’il n’a pas interagi avec l’Authorization Server. Les limites du système s’appliquent si cette durée dépasse 4 320 minutes (3 jours) pour les forfaits non Enterprise ou 144 000 minutes (100 jours) pour les forfaits Enterprise.
* **Exiger une connexion après** : Période (en minutes) après laquelle un utilisateur devra se connecter de nouveau, peu importe son activité. Les limites du système s’appliquent si cette durée dépasse 43 200 minutes (30 jours) pour les forfaits non Enterprise ou 525 600 minutes (365 jours) pour les forfaits Enterprise.

<div id="settings-advanced">
  ### Paramètres (avancés)
</div>

* **Change Password Flow v2** : Lorsqu’elle est activée, la version la plus récente du Change Password Flow sera utilisée. La version précédente est obsolète, et nous vous recommandons fortement d’activer la v2. Cette option est fournie uniquement pour assurer la rétrocompatibilité et, une fois activée, vous ne pourrez plus la désactiver. Vous pouvez personnaliser l’interface utilisateur du widget Change Password dans l’onglet [Universal Login &gt; Password Reset](https://manage.auth0.com/#/password_reset) de l’Auth0 Dashboard.
* **Dynamic Client Registration (DCR)** : Lorsqu’elle est activée, les développeurs tiers peuvent enregistrer dynamiquement des applications pour vos API. Vous pouvez aussi mettre à jour ce paramètre à l’aide du point de terminaison [`/tenant/patch_settings`](https://auth0.com/docs/api/management/v2#!/Tenants/patch_settings) du <Tooltip tip="Management API : un produit qui permet aux clients d’effectuer des tâches administratives." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Management+API">Management API</Tooltip> d’Auth0. Par défaut, cette fonctionnalité est désactivée. Pour en savoir plus, consultez [Dynamic Client Registration](/docs/fr-CA/get-started/applications/dynamic-client-registration).
* **Profil de compatibilité du paramètre resource** : le profil de compatibilité du paramètre resource définit la manière dont Auth0 traite le paramètre `resource` dans les requêtes d’autorisation.
  * Lorsqu’il est activé :
    * Auth0 utilise le paramètre `resource` pour préciser à quel serveur de ressources (API) l’application cliente souhaite accéder
    * Auth0 vérifie d’abord le paramètre `audience` ; s’il n’est pas fourni, il utilise le paramètre `resource`
    * Le paramètre `resource` est utilisé par Auth0 et n’est pas transmis à l’IdP en amont
  * Lorsqu’il est désactivé (par défaut) :
    * Auth0 utilise uniquement le paramètre `audience` pour préciser le serveur de ressources
    * Le paramètre `resource` est traité comme un paramètre d’IdP en amont et sera transmis à l’IdP
* **Enregistrement du Client ID Metadata Document (CIMD)** : Activez l’enregistrement manuel par CIMD en important, depuis une URL, un Client ID Metadata Document (CIMD) hébergé à l’externe. Un CIMD est un fichier JSON contenant les métadonnées du client et hébergé sur un domaine HTTPS sécurisé contrôlé par l’application. Pour en savoir plus, consultez [Enregistrer des applications avec CIMD](/docs/fr-CA/get-started/auth0-overview/create-applications/register-applications-with-cimd).
* **Activer les connexions de l’application** : Lorsque cette option est activée, toutes les connexions actuelles seront activées pour toute nouvelle application créée.
* **Utiliser une réponse générique dans le message d’erreur de l’API publique d’inscription** : Lorsqu’elle est activée, les erreurs générées lors de l’utilisation de l’API publique d’inscription renvoient une réponse générique. Cela aide à prévenir l’énumération des comptes à l’inscription en empêchant les <Tooltip tip="Acteurs malveillants : entité (une personne ou un groupe) qui représente une menace pour l’entreprise ou l’environnement avec l’intention de causer du tort." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=bad+actors">acteurs malveillants</Tooltip> de deviner des identifiants déjà enregistrés (nom d’utilisateur, courriel ou numéro de téléphone) en lisant les codes de réponse d’erreur, comme `user_exists`.
* **Activer la publication des informations sur les connexions d’entreprise avec les domaines d’<Tooltip tip="Fournisseur d’identité (IdP) : Service qui stocke et gère les identités numériques." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=IdP">IdP</Tooltip>** : Lorsqu’elle est activée, cette option prend en charge la [Home Realm Discovery](/docs/fr-CA/authenticate/login/auth0-universal-login/identifier-first), et [Auth0 Lock](/docs/fr-CA/libraries/lock) s’appuie sur un fichier public validé qui contient des informations sur les connexions d’entreprise. Si vous n’avez pas besoin de cette fonctionnalité, vous pouvez la désactiver.
* **Activer le flux de vérification du courriel à la connexion pour les connexions Azure AD et ADFS** : Lorsque cette option est activée, les utilisateurs verront un écran de vérification du courriel lors de leur première connexion avec des connexions Azure AD ou ADFS.
* **<Tooltip tip="Refresh Token : jeton utilisé pour obtenir un Access Token renouvelé sans obliger les utilisateurs à se connecter de nouveau." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Refresh+Token">Refresh Token</Tooltip> La révocation supprime l’autorisation** : Lorsqu’elle est activée, cette option supprime l’autorisation sous-jacente lorsque vous révoquez un refresh token à l’aide du point de terminaison `/oauth/revoke` de l’Authentication API.

  <Callout icon="file-lines" color="#0EA5E9" iconType="regular">
    Pour les tenants existants, cette fonctionnalité est activée par défaut afin de préserver le comportement existant. Pour les nouveaux tenants (à compter du 13 janvier 2021), cette fonctionnalité est désactivée par défaut afin de garantir que la révocation d’un jeton d’actualisation n’entraînera pas celle de l’autorisation. Si la révocation d’une autorisation est nécessaire, une requête distincte doit être envoyée au moyen d’un point de terminaison de révocation d’autorisation.
  </Callout>
* **Autoriser les noms d’organisation dans l’Authentication API** : Lorsque cette option est activée, les points de terminaison [/authorize](https://auth0.com/docs/api/authentication#authorize-application) et [SAML](https://auth0.com/docs/api/authentication#saml) peuvent accepter à la fois les ID d’organisation et les noms d’organisation. De plus, les jetons d’ID et d’accès incluront les claims `org_id` et `org_name`. Avant d’activer ce paramètre, consultez [Use Organization Names in Authentication API](/docs/fr-CA/manage-users/organizations/configure-organizations/use-org-name-authentication-api) pour prendre connaissance des considérations importantes et des répercussions possibles.
* **Autoriser les Pushed Authorization Requests (PAR)** : Lorsque cette option est activée, le point de terminaison `/par` peut accepter les requêtes d’autorisation qui lui sont transmises par une application cliente. Cela empêche l’application cliente d’envoyer des requêtes par le front channel non sécurisé (c.-à-d. le navigateur).
