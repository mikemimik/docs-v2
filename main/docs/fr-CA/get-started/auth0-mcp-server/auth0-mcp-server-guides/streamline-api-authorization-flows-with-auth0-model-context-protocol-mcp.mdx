<Card title="Avant de commencer">
  1. Créez un compte Auth0 avec des droits d’administration
  2. Installez Auth0 MCP Server et intégrez-le à Claude Desktop
</Card>

Utilisez des interactions en langage naturel avec [Auth0 MCP Server](/docs/fr-CA/get-started/auth0-mcp-server) pour configurer et gérer rapidement un système complet d’autorisation d’API. En tirant parti de la capacité de [Claude](https://claude.ai/) à interagir avec Auth0 au moyen du Model Context Protocol, vous pouvez mettre en place des modèles d’autorisation complexes en quelques minutes plutôt qu’en quelques heures.

<div id="step-1-explore-current-api-configuration">
  ### Étape 1 : Explorer la configuration actuelle des API
</div>

Avant de créer de nouvelles API, il est important de comprendre ce qui est déjà configuré dans votre tenant. Avec Auth0 MCP Server, vous pouvez obtenir rapidement ces renseignements en langage naturel.

Demandez à Claude :

```bash wrap lines
Montre-moi tous les serveurs de ressources (API) actuellement configurés dans mon tenant Auth0.
```

Claude utilisera l’outil `auth0_list_resource_servers` pour récupérer et afficher vos API existantes. Vous obtenez ainsi une vue d’ensemble rapide de vos API, sans avoir à naviguer dans l’<Tooltip tip="Auth0 Dashboard : le principal produit d’Auth0 pour configurer vos services." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Auth0+dashboard">Auth0 Dashboard</Tooltip>.

<div id="step-2-create-new-api-definitions">
  ### Étape 2 : Créer de nouvelles définitions d’API
</div>

Maintenant que vous comprenez votre configuration actuelle, vous pouvez créer de nouvelles API pour votre architecture de microservices. Au lieu de configurer chaque API individuellement dans le dashboard, vous pouvez les définir toutes d’un seul coup.

Demandez à Claude :

```bash lines
Je dois créer trois nouvelles API pour nos microservices :
1. Une API d'inventaire avec des scopes de lecture et d'écriture
2. Une API de commandes avec des scopes de lecture et d'écriture
3. Une API de clients avec des scopes de lecture, d'écriture et admin

Chaque API doit imposer la validation des scopes.
```

Claude utilisera l’outil `auth0_create_resource_server` à plusieurs reprises pour configurer ces API.

<div id="step-3-implement-authorization-logic">
  ### Étape 3 : Mettre en place la logique d’autorisation
</div>

Une fois vos API définies, vous devrez mettre en place une logique d’autorisation personnalisée. Avec Auth0 MCP Server, vous pouvez décrire vos règles d’affaires en langage naturel, et Claude générera le code approprié pour les [Actions](/docs/fr-CA/customize/actions) Auth0.

Demandez à Claude :

```bash lines
Créez une Action qui applique les règles d'autorisation suivantes :
1. Vérifiez que l'application qui fait la requête dispose des scopes appropriés pour l'API à laquelle on accède.
2. Pour l'API customers, les requêtes utilisant le scope admin doivent provenir uniquement de nos plages d'adresses IP internes (10.0.0.0/8 et 172.16.0.0/12).
3. Consignez toutes les décisions d'autorisation à des fins d'audit.
```

Claude utilisera l’outil `auth0_create_action` pour générer puis créer une Action avec le code approprié.

<div id="step-4-deploy-the-authorization-action">
  ### Étape 4 : Déployer l’Action d’autorisation
</div>

Une fois satisfait du code de l’Action, vous pouvez la déployer dans votre environnement de production à l’aide d’une simple instruction.

Demandez à Claude :

```bash lines
Deploy this action to run during the token issuance process.
```

Claude utilisera l’outil `auth0_deploy_action` pour déployer l’Action.

<div id="step-5-configure-client-applications">
  ### Étape 5 : Configurer les applications clientes
</div>

Ensuite, vous devrez configurer vos applications clientes pour accéder aux API appropriées avec les bons scopes.

Demandez à Claude :

```bash wrap lines
Create an 'inventory-manager' application on my Auth0 tenant and update the app to access the inventory and orders APIs with read and write permissions, but not the customers API.
```

Claude utilisera d’abord `auth0_get_application` pour récupérer la configuration actuelle, puis `auth0_update_application` pour la mettre à jour.

<div id="step-6-verify-configuration">
  ### Étape 6 : Vérifiez la configuration
</div>

Pour vous assurer que votre système d’autorisation fonctionne correctement, vous pouvez consulter les logs afin de repérer tout problème d’autorisation.

Demandez à Claude :

```bash wrap lines
Check the logs for any failed API access attempts in the last hour. Focus on authorization failures.
```

Claude utilisera `auth0_list_logs` pour trouver les entrées de journal pertinentes.

<div id="step-7-refine-the-authorization-rules">
  ### Étape 7 : Affinez les règles d’autorisation
</div>

Selon l’analyse des journaux, vous devrez peut-être affiner vos règles d’autorisation. Par exemple, vous devrez peut-être autoriser l’accès depuis des plages d’adresses IP supplémentaires.

Demandez à Claude :

```bash wrap lines
Mettez à jour l'action d'autorisation pour également autoriser les requêtes provenant de notre plage VPN de développement 192.168.100.0/24.
```

Claude utilisera `auth0_get_action` pour récupérer le code actuel de l’Action, puis `auth0_update_action` pour le modifier.

Enfin, Claude utilisera `auth0_deploy_action` pour déployer l’Action mise à jour.

<div id="conclusion">
  ### Conclusion
</div>

En utilisant Auth0 MCP Server avec Claude, vous avez mis en place un système complet d&#39;autorisation d&#39;API en :

* Créant plusieurs définitions d&#39;API avec les scopes appropriés
* Mettant en place une logique d&#39;autorisation personnalisée avec des restrictions basées sur l&#39;adresse IP
* Configurant des applications clientes pour accéder à des API précises
* Surveillant et affinant vos règles d&#39;autorisation

Cette approche en langage naturel vous fait non seulement gagner du temps, mais vous permet aussi d&#39;appliquer les pratiques exemplaires en matière de sécurité sans avoir une connaissance approfondie du système Actions d&#39;Auth0 ni de la syntaxe des requêtes de journaux.
