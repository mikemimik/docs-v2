---

L’[Agent Experience Score](https://auth0.com/agent-experience) mesure dans quelle mesure les agents d’IA intègrent Auth0 dans différents modèles et frameworks. Il vous permet de comparer les scores actuels d’agents qui mettent en œuvre des services et des fonctionnalités Auth0 — comme l’authentification multifacteur (AMF) ou Auth0 Actions — dans des environnements de développement et de test, afin d’évaluer comment les outils Auth0 améliorent les performances des agents.

Utilisez cette ressource pour en savoir plus sur la méthodologie de notation, notamment sur la façon dont les scores sont calculés, les dimensions mesurées et l’attribution des notes.

<div id="test-specifications">
  ## Spécifications des tests
</div>

Les agents d’IA — Claude Code, GitHub Copilot, Gemini CLI — exécutent des tâches d’intégration à Auth0 dans des environnements de développement isolés. Chaque agent utilise les mêmes outils qu’un développeur utiliserait dans un environnement réaliste : un espace de travail, un shell et des outils comme Auth0 CLI. Les demandes sont courtes et réalistes : « ajoute l’authentification à mon application Next.js », et non des procédures détaillées étape par étape.

Chaque modèle est testé avec et sans les outils d’Auth0 ([MCP Server](/docs/fr-CA/get-started/build-with-ai-tools#auth0-docs-mcp-server) et [Agent Skills](/docs/fr-CA/quickstart/agent-skills)). L’écart entre ces scores mesure l’impact des outils d’IA d’Auth0 sur l’expérience des développeurs.

<div id="score-dimensions">
  ## Dimensions du score
</div>

Chaque exécution est évaluée selon 7 dimensions réparties en deux catégories. Quatre dimensions portent sur le processus de l’agent, de bout en bout, avec les outils Auth0. Trois dimensions évaluent le résultat final. Chaque dimension reçoit une note individuelle de 0 à 100, puis ces notes sont pondérées et combinées pour produire le score global.

| Dimension                     | Catégorie | Description                                                                                                                                                                           |
| ----------------------------- | --------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Friction de configuration** | Processus | Score déterminé par la capacité de l’agent à accomplir la tâche de façon autonome. Si l’agent s’est interrompu pour poser des questions ou a rencontré des erreurs, le score diminue. |
| **Vitesse de configuration**  | Processus | Score déterminé par le temps d’exécution actif de l’agent. Les résultats sont comparables d’un environnement à l’autre.                                                               |
| **Efficacité**                | Processus | Score déterminé par le nombre d’appels à des outils requis pour accomplir la tâche. Moins il faut d’appels à des outils, moins il y a de coûts et de complexité.                      |
| **Récupération après erreur** | Processus | Score déterminé par les erreurs d’infrastructure (limites de débit, délais d’expiration) qui ont perturbé l’exécution.                                                                |
| **Exactitude**                | Résultat  | Score déterminé selon que le code généré importe de vrais packages, appelle de vraies méthodes et relie correctement les composants.                                                  |
| **Hallucination**             | Résultat  | Score déterminé selon que l’agent a inventé des packages inexistants ou utilisé de mauvaises variantes du SDK.                                                                        |
| **Sécurité**                  | Résultat  | Score déterminé selon que l’agent a codé en dur des secrets, stocké des jetons de façon non sécurisée ou versé des identifiants d’authentification dans le code source.               |

<div id="grades">
  ## Notes
</div>

Les scores globaux correspondent aux notes alphabétiques suivantes :

| Note | Score min. | Description                                        |
| ---- | ---------- | -------------------------------------------------- |
| A    | 90         | Prêt pour la production. Très peu de problèmes.    |
| B    | 75         | Solide, mais comporte quelques lacunes à corriger. |
| C    | 60         | Utilisable, mais demande du nettoyage.             |
| D    | 40         | Problèmes importants.                              |
| F    | &lt; 40    | Peu utile — mieux vaut repartir de zéro.           |

Les notes sont calibrées pour correspondre à l’intuition des développeurs. Un score de 91 devrait donner l’impression d’un code que vous accepteriez avec très peu de révision. Un score de 55 devrait donner l’impression de quelque chose qui demande un vrai travail de correction.

<div id="result-validation">
  ## Validation des résultats
</div>

Chaque évaluateur vérifie le code généré — pas le texte ni les explications. Les évaluateurs s’assurent que le code compile, importe de vrais packages, appelle de vraies méthodes du SDK et n’introduit pas de vulnérabilités de sécurité.

Les résultats sont validés à plusieurs niveaux :

* **Vérifications de présence** : les symboles SDK requis, les importations et les clés de configuration sont présents dans le résultat.
* **Détection des hallucinations** : les packages inventés, les mauvaises variantes du SDK et les méthodes d’API fabriquées sont détectés.
* **Vérifications de sécurité** : les identifiants codés en dur, les jetons stockés dans un espace de stockage non sécurisé et les secrets dans le code source sont signalés.
* **Validation structurelle** : le code est correctement assemblé — les bons composants dans les bons fichiers, les hooks de cycle de vie sont bien gérés et le middleware est dans le bon ordre.
* **Exactitude des versions** : l’agent utilise les API actuelles, et non des modèles dépréciés (vérifié uniquement lorsque l’agent a accès à la documentation à jour).
* **Révision globale** : un juge LLM évalue l’exactitude globale de la mise en œuvre.

<div id="estimated-cost-and-time">
  ## Coût et temps estimés
</div>

La page des résultats affiche le coût estimé et le temps estimé pour chaque configuration. Ces valeurs correspondent à une seule exécution d’évaluation avec Auth0 MCP + Skills activés.

<div id="estimated-cost">
  ### Coût estimé
</div>

Le coût est calculé en fonction du nombre total de jetons consommés pendant l’exécution de l’évaluation (jetons d’entrée + jetons de sortie), multiplié par le tarif par jeton publié par le fournisseur du modèle. Auth0 ne facture pas l’exécution des évaluations — ce coût correspond à ce que vous paieriez à votre fournisseur de modèle pour une utilisation équivalente de jetons.

Le tarif des jetons varie selon le modèle et le fournisseur. Pour connaître les tarifs en vigueur, consultez la page de tarification de votre fournisseur :

* [Tarification d’Anthropic (Claude)](https://docs.anthropic.com/en/docs/about-claude/models#model-comparison-table)
* [Tarification d’OpenAI (GPT)](https://openai.com/api/pricing/)
* [Tarification de Google (Gemini)](https://ai.google.dev/gemini-api/docs/pricing/)

<div id="estimated-time">
  ### Temps estimé
</div>

Le temps correspond à la durée réelle d’une exécution d’évaluation, de la soumission du prompt jusqu’au résultat final. Il comprend toute l’activité de l’agent : lecture de fichiers, appels d’outils, attente des réponses de l’API et écriture de code.

Le temps peut varier selon :

* La latence de l’API du fournisseur de modèle et les limitations de débit
* Le nombre d’appels d’outils requis (varie selon la complexité de la tâche)
* Les conditions réseau entre l’environnement d’évaluation et le fournisseur de modèle
* La profondeur de la file d’attente et la charge du côté du fournisseur

Le temps n’est pas normalisé d’un fournisseur à l’autre. Un temps plus court reflète à la fois l’efficacité du modèle et les performances de l’infrastructure du fournisseur.
