Dans ce tutoriel, nous avons présenté un scénario simple : une application web traditionnelle, hébergée dans le cloud, qui utilise Auth0 pour l’authentification tout en s’appuyant sur le répertoire d’utilisateurs Active Directory existant. Nous avons vu ce qu’offre <Tooltip tip="OpenID : norme ouverte d’authentification qui permet aux applications de vérifier l’identité des utilisateurs sans recueillir ni stocker leurs identifiants de connexion." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=OpenID">OpenID</Tooltip> Connect (OIDC) et pourquoi il convenait mieux à ce cas d’utilisation, comment fonctionne le flux d’authentification, ce qu’est un <Tooltip tip="OpenID : norme ouverte d’authentification qui permet aux applications de vérifier l’identité des utilisateurs sans recueillir ni stocker leurs identifiants de connexion." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=ID+Token">ID Token</Tooltip> et comment le valider et le manipuler, comment configurer les applications et les connexions dans le <Tooltip tip="Auth0 Dashboard : principal produit d’Auth0 pour configurer vos services." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Auth0+dashboard">Auth0 Dashboard</Tooltip>, comment implémenter la connexion et la déconnexion des utilisateurs à l’aide de Lock, ainsi que le fonctionnement de la gestion des sessions et du contrôle d’accès.

Nous avons commencé par décrire le cas d’utilisation et les exigences, puis nous avons expliqué comment chacune de ces exigences peut être satisfaite, ainsi que le raisonnement qui a mené à chacun de ces choix.

Nous avons utilisé ASP .NET Core pour la mise en œuvre de l’exemple d’application web, mais nous espérons qu’après avoir parcouru ce document, vous serez en mesure de créer une telle application web à l’aide du framework de votre choix.
