Dans ce scénario, nous allons créer une API de feuilles de temps pour une entreprise fictive nommée ExampleCo. L’API permettra d’ajouter des entrées de feuille de temps pour un employé ou un sous-traitant.

Nous allons également créer une tâche cron qui traitera des entrées de feuille de temps provenant d’un système externe et les acheminera vers la base de données centralisée des feuilles de temps au moyen de l’API.

<Info>
  ### En bref

  * Auth0 fournit l’authentification et l’autorisation des API comme moyen de sécuriser l’accès aux points de terminaison de l’API (lisez [Authentification et autorisation des API](/docs/fr-CA/get-started/architecture-scenarios/server-application-api/part-1#api-authentication-and-authorization))
  * Pour autoriser une Machine-to-Machine Application (un CLI, un service ou un démon sans interaction utilisateur), Auth0 prend en charge le Client Credentials Grant (lisez [Client Credentials Grant](/docs/fr-CA/get-started/architecture-scenarios/server-application-api/part-1#client-credentials-grant))
  * La Machine-to-Machine Application et l’API doivent toutes deux être configurées dans l’Auth0 Dashboard (lisez [Auth0 Configuration](/docs/fr-CA/get-started/architecture-scenarios/server-application-api/part-2))
  * L’API sera sécurisée en veillant à ce qu’un access token valide (implémenté sous la forme d’un JSON Web Token) soit transmis dans l’en-tête HTTP Authorization lors des appels à l’API (lisez [Implémenter l’API](/docs/fr-CA/get-started/architecture-scenarios/server-application-api/part-3))
  * Une fois l’autorisation accordée, un access token est émis à la Machine-to-Machine Application (lisez [Obtenir un access token](/docs/fr-CA/get-started/architecture-scenarios/server-application-api/part-3#get-an-access-token))
  * La Machine-to-Machine Application peut ensuite utiliser cet access token et le transmettre dans l’en-tête HTTP Authorization pour authentifier les appels aux points de terminaison de l’API (lisez [Appeler l’API](/docs/fr-CA/get-started/architecture-scenarios/server-application-api/part-3#invoke-the-api))
</Info>

<div id="the-premise">
  ## Le principe
</div>

ExampleCo est une jeune société de consultation. Elle compte actuellement environ 100 employés et fait aussi appel à des sous-traitants externes. Tous les employés et sous-traitants externes doivent remplir des feuilles de temps chaque semaine. À cette fin, elle a créé une application de feuilles de temps, un scénario que nous avons présenté dans [Single Sign-On for Regular Web Apps](/docs/fr-CA/get-started/architecture-scenarios/sso-for-regular-web-apps). Les employés à l’interne utilisent cette application web pour remplir leurs feuilles de temps, mais certains sous-traitants externes utilisent déjà un autre outil pour assurer le suivi de leurs feuilles de temps. ExampleCo a besoin d’une solution pour éviter le travail en double et a donc décidé de créer une tâche cron afin de lire les entrées de feuilles de temps de ce système externe et de les téléverser automatiquement vers le backend d’ExampleCo au moyen d’une API.

<div id="goals-requirements">
  ### Objectifs et exigences
</div>

ExampleCo veut bâtir une solution flexible qui :

* Permet des processus automatisés s&#39;exécutant dans le backend
* Permet d&#39;intégrer à l&#39;architecture de futurs lancements d&#39;applications, comme une application mobile
* Permet à la Timesheets API d&#39;être sécurisée et accessible aux utilisateurs et aux applications autorisés
* Permet de partager une grande partie du code et de la logique d&#39;affaires de l&#39;application avec d&#39;autres applications
