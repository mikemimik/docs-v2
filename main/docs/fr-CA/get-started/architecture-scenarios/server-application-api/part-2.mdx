<div id="configure-the-api">
  ## Configurer l’API
</div>

Pour protéger une API, vous devez en enregistrer une dans le <Tooltip tip="Auth0 Dashboard : l’interface principale d’Auth0 pour configurer vos services." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip>. Suivez les instructions de [Register APIs](/docs/fr-CA/get-started/auth0-overview/set-up-apis) pour configurer l’API Timesheets.

<div id="json-web-token-profile">
  ### Profil de JSON Web Token
</div>

<Tooltip tip="Jeton d’accès : information d’autorisation, sous la forme d’une chaîne opaque ou d’un JWT, utilisée pour accéder à une API." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=OAuth+2.0">OAuth 2.0</Tooltip> définit les types de profils, ou dialectes, d’un <Tooltip tip="OAuth 2.0 : cadre d’autorisation qui définit les protocoles et les workflows d’autorisation." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=access+token">jeton d’accès</Tooltip>. Auth0 prend en charge deux profils :

* Profil de jeton Auth0
* Profil RFC 9068

Le profil par défaut est le profil Auth0. La principale différence entre ces deux profils réside dans les noms des claims. Le profil Auth0 utilise `azp` plutôt que `client_id`. Vous pouvez en apprendre davantage sur ces différences en consultant [Profils de jetons d’accès](/docs/fr-CA/secure/tokens/access-tokens/access-token-profiles).

<div id="signing-algorithms">
  ### Algorithmes de signature
</div>

Lorsque vous créez une API, vous devez sélectionner l’algorithme qui servira à signer vos jetons. La signature permet de vérifier que l’expéditeur du <Tooltip tip="JSON Web Token (JWT) : format standard de jeton ID (et souvent de jeton d’accès) utilisé pour représenter de façon sécurisée des claims entre deux parties." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=JWT">JWT</Tooltip> est bien celui qu’il prétend être et de s’assurer que le message n’a pas été modifié en cours de route.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  La signature fait partie d’un JWT. Si vous ne connaissez pas bien la structure d’un JWT, consultez [Structure du JSON Web Token](/docs/fr-CA/secure/tokens/json-web-tokens/json-web-token-structure).
</Callout>

Pour créer la signature, vous devez prendre l’en-tête encodé, le payload encodé, un secret, l’algorithme indiqué dans l’en-tête, puis signer l’ensemble. Cet algorithme, qui fait partie de l’en-tête du JWT, est celui que vous sélectionnez pour votre API : `HS256,` `RS256,` ou `PS256`.

La pratique la plus sûre, et celle que nous recommandons, consiste à utiliser **RS256**. Voici quelques raisons :

* Avec RS256, vous avez l’assurance que seul le détenteur de la clé privée (Auth0) peut signer des jetons, tandis que n’importe qui peut vérifier si le jeton est valide à l’aide de la clé publique.
* Avec HS256, si la clé privée est compromise, vous devrez redéployer l’API avec le nouveau secret. Avec RS256, vous pouvez demander un jeton valide pour plusieurs audiences.
* Avec RS256, vous pouvez effectuer une rotation des clés sans avoir à redéployer l’API avec le nouveau secret.

<div id="configure-the-scopes">
  ## Configurer les scopes
</div>

Dans les paramètres de votre API, vous devrez configurer les scopes que les applications peuvent demander lors de l’autorisation. Accédez à l’onglet Permissions.

Dans cette section, vous pouvez ajouter les quatre scopes mentionnés précédemment : `batch:upload`, `read:timesheets`, `create:timesheets`, `delete:timesheets` et `approve:timesheets`. Ajoutez aussi un scope supplémentaire : `batch:upload`.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Dans le cadre de ce document, nous nous limiterons au scope `batch:upload`, puisqu’il s’agit du seul requis par le cron job. Toutefois, par souci d’exhaustivité, nous ajoutons les scopes nécessaires dont auront besoin de futures applications.
</Callout>

<div id="create-the-application">
  ## Créer l’application
</div>

Lors de la création d’une API dans l’Auth0 Dashboard, une application de test pour l’API est générée automatiquement. Dans l’Auth0 Dashboard, accédez à la [section Applications](https://manage.auth0.com/#/applications) pour voir l’application de test de l’API Timesheets.

<Frame>![undefined](/docs/images/cdy7uua7fh8z/7lnuzFgs3IuWyL4bK5aVZV/05abca8035c011bf018b27e31f77a055/2024-06-28_17-13-34.png)</Frame>

Accédez aux paramètres de l’application en sélectionnant l’option de menu à l’extrémité droite. Renommez l’application en `Timesheets Import Job`.

Pour le cron job, vous aurez besoin d’une application machine-to-machine. L’application de test générée lors de la création de l’API a été automatiquement configurée comme application machine-to-machine.

<div id="configure-applications-access-to-the-api">
  ## Configurer l&#39;accès de l&#39;application à l&#39;API
</div>

Enfin, vous devez autoriser l&#39;application à accéder à l&#39;API Timesheets. Revenez à la configuration de l&#39;API, puis sélectionnez l&#39;onglet Machine to Machine Application.

Vous verrez l&#39;application **Timesheets Import Job** dans la liste, et elle devrait avoir accès à l&#39;API, comme l&#39;indique le commutateur à droite du nom de l&#39;application, qui affiche la valeur `Authorized`. Si ce n&#39;est pas le cas, faites simplement basculer la valeur du commutateur de `Unauthorized` à `Authorized`.

Vous devrez également préciser quels scopes seront inclus dans les jetons d&#39;accès émis à l&#39;application lorsque celle-ci s&#39;authentifie auprès d&#39;Auth0.

Développez les paramètres de l&#39;application en cliquant sur la flèche vers le bas tout à droite, et vous verrez la liste des scopes disponibles. Le cron job n&#39;aura besoin que du scope `batch:upload`, puisqu&#39;il créera simplement de nouvelles feuilles de temps à partir des entrées de feuille de temps du système externe.

Une fois le scope `batch:upload` sélectionné, enregistrez les paramètres en cliquant sur le bouton **Update**.
