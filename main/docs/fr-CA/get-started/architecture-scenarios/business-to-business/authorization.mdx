Commençons par prendre un peu de recul pour parler du contrôle d’accès. Il n’existe pas, dans l’industrie, de définition unique et parfaitement tranchée du contrôle d’accès, mais si vous prenez le temps de faire quelques recherches, vous verrez que la plupart des sources faisant autorité s’entendent pour dire qu’il s’agit du concept global qui regroupe l’authentification, l’autorisation, le consentement et l’application des politiques afin de veiller à ce que seules les bonnes personnes et les bons services aient accès à vos applications et à vos API.

Examinons maintenant de plus près les distinctions entre l’authentification, l’autorisation, le consentement et l’application des politiques. Votre tenant Auth0 (votre <Tooltip tip="Serveur d’autorisation : serveur centralisé qui contribue à définir les limites de l’accès d’un utilisateur. Par exemple, votre serveur d’autorisation peut contrôler les données, les tâches et les fonctionnalités accessibles à un utilisateur." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Authorization+Server">serveur d’autorisation</Tooltip>) est généralement responsable de l’authentification et du consentement, ainsi que d’une partie ou de la totalité de l’autorisation et de l’application des politiques. De plus, l’application ou l’API elle-même est presque toujours le principal mécanisme d’application des politiques, surtout lorsqu’un accès contextuel est requis :

* **Authentification**: processus consistant à déterminer si un principal (un utilisateur ou une application) est bien la personne ou l’entité qu’il prétend être.
* **Autorisation**: processus consistant à déterminer ce qui est autorisé, selon le principal, les permissions qui lui ont été accordées et/ou l’ensemble des critères d’accès propres au contexte.
* **Consentement**: permissions que l’utilisateur (<Tooltip tip="Propriétaire de la ressource : entité (comme un utilisateur ou une application) capable d’accorder l’accès à une ressource protégée." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Resource+Owner">Propriétaire de la ressource</Tooltip>) a accordées à une application pour agir en son nom. Il s’agit généralement d’une exigence de l’autorisation déléguée. L’utilisateur doit accorder au client la permission d’accéder à ses données dans un autre système.
* **Application des politiques**: application des politiques de l’application ou de l’API, en refusant ou en autorisant l’accès selon les renseignements d’authentification et/ou d’autorisation d’un utilisateur.

En général, nous regroupons les différents types de contrôle d’accès en trois catégories distinctes afin qu’il soit plus facile de comprendre a) quel acteur est responsable du stockage de l’information, b) quel acteur est responsable de la prise de décision, et c) quel acteur est responsable de l’application des restrictions.

* La première catégorie correspond au cas où l’accès à une application ou à une API est accordé ou refusé dans son ensemble. Les données nécessaires à cette application ainsi que le processus d’application lui-même sont généralement définis dans le contexte du serveur d’autorisation. Par exemple, en utilisant [`app_metadata`](/docs/fr-CA/manage-users/user-accounts/metadata) associé à un utilisateur et une [Action](/docs/fr-CA/customize/actions) définie dans votre tenant Auth0.
* La deuxième catégorie correspond au cas où l’accès à un sous-ensemble précis des fonctionnalités d’une application ou d’une API est accordé ou refusé. Les données nécessaires à cette application sont généralement stockées dans le serveur d’autorisation. Par exemple, en utilisant `app_metadata` pour un utilisateur dans votre tenant Auth0, tandis que le processus d’application est effectué dans l’application ou l’API elle-même. Dans ce scénario, les données sont généralement transmises sous forme d’un ou de plusieurs claims personnalisés dans un jeton [`id`](/docs/fr-CA/secure/tokens/id-tokens) ou [`access`](/docs/fr-CA/secure/tokens/access-tokens).
* La troisième catégorie correspond au cas où l’accès est accordé ou refusé selon ce sur quoi le principal (sujet) peut agir dans le contexte d’une application ou d’une API. Les données nécessaires à cette application, ainsi que le processus d’application lui-même, sont généralement définis dans le contexte de l’application ou de l’API. Dans ce scénario, les données transmises sous forme d’un ou de plusieurs claims personnalisés dans un jeton `id` ou `access` peuvent être utilisées avec ou sans données provenant d’une source externe autre qu’Auth0.

En outre, les mécanismes de contrôle d’accès basé sur les rôles (RBAC) et de contrôle d’accès basé sur les attributs (ABAC) peuvent être appliqués dans n’importe laquelle des catégories de contrôle d’accès décrites ci-dessus. Quel que soit votre cas d’utilisation, il y a donc un certain nombre d’éléments à prendre en compte lorsque vous évaluez les fonctionnalités et le workflow dont vous avez besoin :

* Existe-t-il des scénarios où l’accès à l’ensemble d’une application ou à une API devrait être rejeté ?
* Allez-vous fournir des API auxquelles des applications tierces pourront accéder ?
* Vos API seront-elles également accessibles à vos propres applications (de première partie) ?
* Votre application appellera-t-elle une API tierce ?
* Vos applications et/ou API devraient-elles appliquer le contrôle d’accès en fonction des claims de l’utilisateur ?
* Et si j’ai besoin de savoir à quelle organisation un <Tooltip tip="Jeton d’accès : justificatif d’autorisation, sous la forme d’une chaîne opaque ou d’un JWT, utilisé pour accéder à une API." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=access+token">jeton d’accès</Tooltip> ou un <Tooltip tip="Access Token: Authorization credential, in the form of an opaque string or JWT, used to access an API." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=ID+token">jeton d’identification</Tooltip> est associé ?

Auth0 prend en charge la restriction d’accès aux applications ou aux API en fonction de certaines conditions. Dans certains cas, vous pourriez vouloir créer une Action qui renvoie une `UnauthorizedError` lorsque, par exemple, un utilisateur tente d’accéder à une application ou à une API au mauvais moment (comme indiqué dans cet [exemple](/docs/fr-CA/manage-users/access-control/sample-use-cases-rules-with-authorization#allow-access-only-on-weekdays-for-a-specific-application)), ou si l’utilisateur ne possède pas le ou les claims nécessaires dans son `app_metadata`. Pour une application qui utilise [OpenID Connect (OIDC)](/docs/fr-CA/authenticate/protocols/openid-connect-protocol), cela empêcherait l’émission du [ID Token](/docs/fr-CA/secure/tokens/id-tokens) servant à autoriser l’accès. De même, pour une API, l’émission de tout [Access Token](/docs/fr-CA/secure/tokens/access-tokens) OAuth2 (utilisé lors de l’appel à l’API) pourrait être empêchée, comme indiqué dans cet [exemple](/docs/fr-CA/manage-users/access-control/sample-use-cases-rules-with-authorization#deny-access-to-anyone-calling-an-api).

<Info>
  ### Meilleure pratique

  De façon générale, nous avons constaté que [OIDC](/docs/fr-CA/authenticate/protocols/openid-connect-protocol) est le protocole standard de l’industrie le plus souvent utilisé par les clients d’Auth0 pour l’authentification dans leurs applications. Nous avons également constaté que, même si [OAuth2](/docs/fr-CA/authenticate/protocols/oauth) a été créé comme protocole de délégation, il est couramment utilisé dans les applications de première partie lorsqu’il y a une API qui ne partage pas de session avec l’application.
</Info>

Auth0 peut également fournir l’information nécessaire pour qu’une application puisse appliquer des restrictions. Pour [l’intégration au niveau de l’application](#application-integration), Auth0 vous permet d’ajouter des [claims personnalisés](#id-token-claims) à un ID Token, que votre application peut ensuite vérifier et utiliser pour faire l’application des politiques. Dans ce cas, vous devrez déterminer quelles informations sont nécessaires pour que votre application puisse prendre des décisions d’application. Si vous devez prendre des décisions au niveau d’une API plutôt que dans votre application, vous devrez probablement utiliser un jeton d’accès plutôt qu’un ID Token. Poursuivez votre lecture pour en savoir plus.

<Warning>
  Au moment de décider quelles données inclure dans votre ID Token et/ou votre jeton d’accès, tenez compte de la taille du token, surtout si vous transmettez le token dans l’URL. Même si vous ne transmettez pas les tokens dans l’URL, vous devrez aussi tenir compte du risque d’exposer des PII (Personally Identifiable Information) sensibles. Les informations contenues dans le token ne sont pas chiffrées ; ainsi, même si la divulgation d’un ID Token ne pose généralement pas un problème de sécurité, elle peut devenir un enjeu de confidentialité selon les données incluses dans le token.
</Warning>

Pour [l’intégration au niveau de l’API](#api-integration), Auth0 prend en charge à la fois les [claims personnalisés](#access-token-claims) et la reconfiguration des [scopes](#access-token-scopes), les deux dans le contexte d’un Access Token. Encore une fois, vous devrez déterminer quelles informations sont nécessaires pour que votre API puisse prendre des décisions d’accès, et votre API devra les appliquer en validant le contenu de l’Access Token.

<Info>
  ### Meilleure pratique

  Au moment de décider si vous devriez utiliser des permissions au moyen de claims personnalisés ou de scopes, assurez-vous de bien comprendre la nature et l’objectif des scopes. Il existe un excellent [article de blogue](https://auth0.com/blog/on-the-nature-of-oauth2-scopes/) à ce sujet, facile à lire et utile pour clarifier la question.
</Info>

Dans les scénarios multiorganisations, il est souvent important de savoir à quelle organisation un jeton d’accès (ou même un ID Token) s’applique. Le fait de suivre les [meilleures pratiques](#organization-data-in-an-access-token) peut vous faire gagner du temps et des efforts.

<div id="application-integration">
  ## Intégration de l’application
</div>

Dans ce scénario, votre tenant Auth0 fournit un jeton comme preuve d’accès autorisé à une application. Pour les applications qui utilisent [OpenID Connect (OIDC)](/docs/fr-CA/authenticate/protocols/openid-connect-protocol), le protocole standard de l’industrie généralement le plus utilisé pour les applications destinées aux clients, il s’agirait d’un ID Token présenté sous forme de [JWT](/docs/fr-CA/secure/tokens/json-web-tokens).

<div id="id-token-claims">
  ### Claims de l’ID Token
</div>

Grâce à l’extensibilité Actions, Auth0 vous permet d’[ajouter facilement des claims personnalisées à un ID Token](/docs/fr-CA/get-started/apis/scopes/sample-use-cases-scopes-and-claims#add-custom-claims-to-a-token) en fonction, par exemple, du contenu des [métadonnées](/docs/fr-CA/manage-users/user-accounts/metadata) d’un utilisateur. Votre application peut ensuite vérifier que l’ID Token contient les claims nécessaires, puis autoriser ou refuser l’accès à certaines fonctionnalités, selon le besoin. Notez que, même si le processus d’ajout de claims personnalisées au moyen d’Actions est simplifié, Actions offre une grande flexibilité et vous permet d’écrire du code personnalisé qui pourrait avoir des effets négatifs.

<Info>
  ### Bonne pratique

  Lorsque vous envisagez d’ajouter des claims personnalisées, nous vous recommandons de stocker dans le [`app_metadata`](/docs/fr-CA/manage-users/user-accounts/metadata) de l’utilisateur toutes les données de contrôle d’accès que vous pourriez devoir inclure dans des claims. D’abord, cela vous évite d’avoir à appeler une API externe pour récupérer ces données, ce qui peut nuire aux performances et à l’évolutivité du processus de connexion. Ensuite, `app_metadata` **ne peut pas** être modifié par un utilisateur — celui-ci ne peut donc pas contourner directement les restrictions de contrôle d’accès en modifiant ses propres métadonnées. N’oubliez pas non plus de consulter nos [bonnes pratiques relatives aux métadonnées](/docs/fr-CA/get-started/architecture-scenarios/business-to-business/profile-management#metadata).
</Info>

Si vous créez différentes instances de votre application pour les organizations de vos clients, il est courant de créer une claim personnalisée dans votre ID token pour représenter l’organisation de l’utilisateur. Par exemple :

```js wrap lines
context.idToken["http://yourdomain.com/claims/organization"] = "organization A";
```

<div id="id-token-scopes">
  ### Scopes de l’ID Token
</div>

Les [scopes OIDC](/docs/fr-CA/get-started/apis/scopes/openid-connect-scopes) sont généralement utilisés par une application pour obtenir le consentement nécessaire à l’accès autorisé aux renseignements d’un utilisateur pendant l’authentification. Chacun des scopes prédéfinis renvoie l’ensemble des claims standard lorsqu’ils sont définis, comme indiqué dans la [spécification OIDC](https://openid.net/specs/openid-connect-core-1_0.html#StandardClaims). Les scopes qu’une application demande dépendent des attributs utilisateur dont elle a besoin. Une fois les scopes demandés autorisés par l’utilisateur, les claims sont renvoyés dans l’ID Token et sont également accessibles à partir du point de terminaison [/userinfo](https://auth0.com/docs/api/authentication/user-profile/get-user-info).

<div id="api-integration">
  ## Intégration d’API
</div>

Dans ce scénario, votre tenant Auth0 peut fournir un [jeton d’accès](/docs/fr-CA/secure/tokens/access-tokens) OAuth2, généralement présenté sous la forme d’un [JWT](/docs/fr-CA/secure/tokens/json-web-tokens), que votre API peut utiliser pour restreindre l’accès à certaines entités. De plus, Auth0 prend en charge ce qu’on appelle les [applications de première partie et applications tierces](/docs/fr-CA/get-started/applications/first-party-and-third-party-applications).

En agissant comme serveur d’autorisation, et avec le consentement de l’utilisateur (le propriétaire de la ressource), votre tenant Auth0 peut fournir un jeton d’accès — généralement présenté sous la forme d’un [JWT](/docs/fr-CA/secure/tokens/json-web-tokens) — à une application (client) afin qu’elle puisse accéder, au nom du propriétaire de la ressource, à des ressources protégées hébergées par un <Tooltip tip="Serveur de ressources : serveur hébergeant des ressources protégées. Les serveurs de ressources acceptent les requêtes vers des ressources protégées et y répondent." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=resource+server">serveur de ressources</Tooltip>. Le jeton d’accès émis est généralement transmis sous forme de jeton Bearer dans l’en-tête HTTP Authorization envoyé à une API.

Que vous ayez une seule API ou une suite d’[API de microservices](/docs/fr-CA/get-started/apis/set-logical-api) logiquement liées, vous pouvez tirer parti des jetons d’accès fournis par Auth0 afin de sécuriser l’accès à votre ou vos services. Bien qu’il soit relativement facile de configurer cela dans le [Auth0 Dashboard](https://manage.auth0.com/) ou au moyen de l’[Auth0 Management API](https://auth0.com/docs/api/management/v2#!/Resource_Servers/post_resource_servers), il est important d’examiner les différents scénarios d’application et les architectures d’API afin de déterminer la meilleure solution pour votre système.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Les jetons d’accès OAuth2 sont principalement conçus pour sécuriser des API exposées publiquement; lorsqu’il est présenté sous forme de JWT, un jeton d’accès est une entité autonome qui peut être vérifiée sans devoir effectuer d’appel supplémentaire à une API tierce. Si vos API n’entrent pas dans cette catégorie — c.-à-d. qu’elles font partie de l’application elle-même (autrement dit, qu’elles sont appelées uniquement par cette application) ou qu’elles se trouvent derrière votre pare-feu — les protéger à l’aide de jetons pourrait être excessif, et votre flux de travail actuel fondé sur des témoins (et autres mécanismes semblables) pourrait très bien suffire.
</Callout>

OAuth2 a été conçu précisément en fonction de l’accès par des tiers. Par exemple, un utilisateur (propriétaire de la ressource) შეიძლება vouloir utiliser une application (un client) qui n’appartient pas à la même organisation que le service qui fournit ses données (le serveur de ressources). Dans ce cas, lorsque l’application doit accéder à des données appartenant à l’utilisateur, elle le redirige vers l’organisation où se trouvent ses données; cette organisation authentifie alors l’utilisateur, puis l’invite à autoriser l’application à accéder à ses données. Cette demande d’autorisation est appelée [consentement](/docs/fr-CA/get-started/applications/third-party-applications/user-consent-and-third-party-applications) et constitue une part importante de la prise en charge des [applications tierces](/docs/fr-CA/get-started/apis/scopes/api-scopes#example-an-api-called-by-a-third-party-application). Si vous prévoyez intégrer des applications tierces, il est important de les définir comme tierces dès le départ afin qu’Auth0 gère la demande de consentement de l’utilisateur.

À l’inverse, si votre organisation possède l’application ou les applications, les données utilisateur elles-mêmes et l’API ou les API par lesquelles ces données sont accessibles, le consentement n’est généralement pas requis, puisque ces interactions sont toutes de [première partie](/docs/fr-CA/get-started/apis/scopes/api-scopes#example-an-api-called-by-a-first-party-application). Si vous ne créez que des applications de première partie, vous pouvez éviter de présenter inutilement des écrans de consentement à vos utilisateurs en [autorisant l’omission du consentement de l’utilisateur](/docs/fr-CA/get-started/apis) dans toute définition de service de ressource.

<Warning>
  Bien que vous puissiez configurer vos applications comme étant de première partie, puis configurer vos API pour permettre aux clients de première partie d’ignorer le consentement, si vous utilisez `localhost`, Auth0 ne peut pas vérifier que l’application est réellement une application de première partie; vos utilisateurs seront donc quand même invités à donner leur consentement. Pour contourner cette contrainte, lors des tests sur votre machine locale pendant le développement, créez un [faux nom d’hôte local et utilisez-le à la place](https://community.auth0.com/t/how-do-i-skip-the-consent-page-for-my-api-authorization-flow/6035).
</Warning>

Autrement, il se peut que vous ayez des données liées à un utilisateur pour lesquelles des [fonctionnalités supplémentaires sont offertes](/docs/fr-CA/get-started/apis/scopes/api-scopes#example-an-api-called-by-a-back-end-service) et pour lesquelles il est impossible d’obtenir le consentement explicite de l’utilisateur (c.-à-d. qu’il n’existe aucun utilisateur authentifié en mesure de le fournir). Dans ce cas, il est possible de définir une [liste d’applications pour lesquelles le Client Credentials grant est activé](/docs/fr-CA/get-started/authentication-and-authorization-flow/client-credentials-flow).

<div id="access-token-claims">
  ### Claims du jeton d’accès
</div>

Comme pour les jetons d’identité, vous pouvez [ajouter des claims personnalisés aux jetons d’accès](/docs/fr-CA/get-started/apis/scopes/sample-use-cases-scopes-and-claims#add-custom-claims-to-a-token) à l’aide de la fonctionnalité d’extensibilité Actions d’Auth0. Une fois ajoutés, votre API peut vérifier qu’un jeton d’accès contient les claims requis et autoriser ou refuser l’accès à certaines fonctionnalités, selon le besoin.

<Info>
  ### Pratique exemplaire

  Si vous envisagez d’ajouter des claims personnalisés, nous vous recommandons de stocker dans le [`app_metadata`](/docs/fr-CA/manage-users/user-accounts/metadata) de l’utilisateur toute donnée de contrôle d’accès que vous pourriez devoir inclure dans des claims. D’abord, cela vous évite d’avoir à appeler une API externe pour récupérer ces données, ce qui peut nuire aux performances et à l’évolutivité. Ensuite, `app_metadata` **ne peut pas** être modifié par un utilisateur — il ne peut donc pas contourner directement les restrictions de contrôle d’accès en modifiant ses propres métadonnées. N’oubliez pas non plus de consulter nos [pratiques exemplaires relatives aux métadonnées](/docs/fr-CA/get-started/architecture-scenarios/business-to-business/profile-management#metadata).
</Info>

<div id="access-token-scopes">
  ### Scopes du jeton d’accès
</div>

Les [scopes OAuth2](/docs/fr-CA/get-started/apis/scopes/api-scopes) sont généralement utilisées comme mécanisme permettant à une API de déterminer quelles actions peuvent être effectuées au nom d’un utilisateur. Des scopes peuvent être ajoutées pour chaque API afin de [définir des permissions d’accès précises](/docs/fr-CA/get-started/apis/add-api-permissions) dans le <Tooltip tip="Auth0 Dashboard : produit principal d’Auth0 pour configurer vos services." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip> ou au moyen de l’Auth0 <Tooltip tip="Auth0 Dashboard : produit principal d’Auth0 pour configurer vos services." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Management+API">Management API</Tooltip>). Les scopes peuvent aussi être modifiées au moyen de l’extensibilité d’Auth0 (p. ex. au moyen d’une Action, comme dans cet [exemple](/docs/fr-CA/get-started/architecture-scenarios/spa-api/part-2#create-a-rule-to-validate-token-scopes)). Les scopes qu’une application demande pour accéder à une API doivent dépendre des fonctionnalités pour lesquelles l’utilisateur doit autoriser l’application. Une fois les scopes demandés autorisés, elles seront renvoyées dans le jeton d’accès, qui pourra ensuite être [validé par l’API en question](/docs/fr-CA/secure/tokens/access-tokens/validate-access-tokens). Un bon exemple est celui où vous ouvrez une session dans une application qui utilise un fournisseur social pour la connexion : l’API du fournisseur social exige que l’application précise si l’utilisateur veut lui permettre de publier du contenu en son nom. Cela permet à l’utilisateur d’accepter ou de rejeter cette requête. Cet exemple montre comment l’utilisateur délègue une permission à l’application — ce qui est différent du cas où une API restreint l’accès selon le rôle d’un utilisateur et doit donc être géré différemment.

<Info>
  ### Bonne pratique

  Même si vous avez la possibilité de modifier entièrement les scopes du jeton d’accès au moyen de l’extensibilité d’Auth0, par bonne pratique de sécurité, vous devriez uniquement supprimer les scopes non autorisés et vous abstenir d’ajouter des scopes qui n’ont pas été demandées.
</Info>

Bien que les scopes soient souvent utilisées comme moyen d’appliquer les permissions d’accès d’un utilisateur, il existe des situations où cela peut devenir [complexe lorsqu’on les utilise ainsi](https://auth0.com/blog/on-the-nature-of-oauth2-scopes/). Nous recommandons donc d’utiliser les scopes pour l’usage auquel elles sont destinées (c.-à-d. déléguer une permission à une application) et d’utiliser des [claims personnalisées](#access-token-claims) pour vos scénarios fondés sur les rôles ou d’autres scénarios de contrôle d’accès.

<div id="fine-grained-authorization-fga">
  ## Autorisation à granularité fine (FGA)
</div>

L’[autorisation à granularité fine](https://docs.fga.dev/fga-concepts) vous permet d’accorder à des utilisateurs un accès à une ressource ou à un objet précis en fonction de :

* du rôle d’un utilisateur au sein d’une organisation, comme `editor` ou `admin`
* d’un attribut de l’utilisateur ou de l’objet, comme `manager` pour un utilisateur ou `marketing` pour un objet
* d’une relation entre un utilisateur et un objet; par exemple, un utilisateur ayant un accès en consultation à un dossier parent a aussi un accès en consultation au dossier enfant

Avec <Tooltip tip="Autorisation à granularité fine (FGA) : produit Auth0 permettant à des utilisateurs d’accéder à des objets ou à des ressources précis." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=FGA">FGA</Tooltip>, vous pouvez créer un modèle d’autorisation pour définir la relation à utiliser afin de déterminer l’accès d’un utilisateur.

<div id="role-based-access-control-rbac">
  ## Contrôle d’accès basé sur les rôles (RBAC)
</div>

Auth0 offre une prise en charge intégrée du contrôle d’accès basé sur les rôles ([RBAC](/docs/fr-CA/manage-users/access-control/rbac)). Le RBAC consiste à attribuer des permissions aux utilisateurs en fonction de leur rôle au sein d’une organisation, ce qui simplifie le contrôle d’accès grâce à une approche plus facile à gérer et moins sujette aux erreurs.

La fonctionnalité RBAC de base peut être utilisée dans de nombreux scénarios à organisations multiples. Consultez [Données d’organisation dans un jeton d’accès](#organization-data-in-an-access-token) pour en savoir plus sur la façon de vous assurer que votre configuration peut répondre à vos besoins en matière de RBAC.

<div id="machine-to-machine-m2m-authorization">
  ## Autorisation machine à machine (M2M)
</div>

Il existe de nombreux scénarios où une application sans aucune session utilisateur interactive doit obtenir un jeton d’accès afin d’appeler une API. Dans ce type de scénario, vous devez authentifier le client plutôt que l’utilisateur, et <Tooltip tip="OAuth 2.0 : framework d'autorisation qui définit les protocoles et les workflows d'autorisation." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=OAuth">OAuth</Tooltip> 2 fournit le type d’octroi [client credentials](/docs/fr-CA/get-started/authentication-and-authorization-flow/client-credentials-flow), ce qui facilite les choses. Voici quelques exemples courants où cela est nécessaire :

* Un cron job ou un autre service qui doit communiquer avec votre API (par exemple, lorsqu’un rapport quotidien doit être généré et envoyé par courriel à un administrateur).
* Une API distincte qui prend en charge un accès privilégié (par exemple, l’API n’est pas exposée directement aux utilisateurs, mais seulement à un backend).
* Dans certaines architectures de microservices, où certaines couches d’API doivent communiquer avec d’autres couches d’API sans intervention d’un utilisateur, ou après l’expiration du jeton d’un utilisateur.
* Une API privilégiée qui peut devoir être appelée avant qu’un utilisateur soit authentifié (c.-à-d. à partir d’une Action ou d’un script DB personnalisé dans votre tenant Auth0)

<Info>
  ### bonne pratique

  Traditionnellement, on créait un « compte de service » spécial pour répondre à ces scénarios : un utilisateur avec un nom d’utilisateur et un mot de passe configurés pour des services prenant en charge des cas d’utilisation non interactifs. Ce n’est plus une approche recommandée pour plusieurs raisons, et la bonne pratique actuelle consiste à utiliser [OAuth 2.0 Client Credentials Grant](/docs/fr-CA/get-started/authentication-and-authorization-flow/client-credentials-flow) dans ces situations.
</Info>

<div id="organization-data-in-an-access-token">
  ## Données d’organisation dans un jeton d’accès
</div>

Si, dans votre système, vous avez une API distincte de votre application qui prend en charge votre application multi-organisation, il est important de limiter les opérations uniquement à l’organisation pour laquelle le jeton a été généré. Il faut donc qu’une certaine information figure dans le jeton d’accès pour indiquer à l’API pour quelle organisation il a été émis. Cela peut se faire de différentes façons, selon les réponses à quelques questions simples :

1. Les utilisateurs finaux de cette organisation pourraient-ils appartenir à plus d’une organisation, ou chaque utilisateur final est-il associé à une organisation précise ?
2. Allez-vous autoriser l’accès machine à machine (M2M) à votre API ?
3. Si vous autorisez l’accès machine à machine (M2M) à votre API, aurez-vous des développeurs qui ont besoin d’un seul ID client et secret pour accéder à plusieurs organisations (mais pas à toutes les organisations) ?
4. Allez-vous autoriser la création d’applications tierces qui exigent un consentement ?

Si les utilisateurs finaux sont associés à une seule organisation **et** que vous n’autoriserez pas l’accès M2M à votre API, ou que vous aurez un <Tooltip tip="ID client : valeur d’identification attribuée à votre ressource enregistrée par Auth0." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=client+ID">ID client</Tooltip>/secret distinct pour chaque organisation qui a besoin d’un accès **et** que vous n’autoriserez pas les applications tierces qui exigent un consentement, l’approche la plus simple consiste alors à créer une claim personnalisée dans le jeton d’accès [à l’aide d’Actions pour les jetons liés à l’utilisateur](#access-token-claims) et [à l’aide du hook client credentials pour les appels M2M](#machine-to-machine-m2m-authorization). Vous pouvez stocker le nom de l’organisation dans les métadonnées du client et l’extraire depuis Actions ou les hooks pour l’inclure dans `access_token` comme claim personnalisée. La fonctionnalité RBAC fonctionnera également telle quelle avec cette approche, tant que chaque utilisateur final ne peut appartenir qu’à une seule organisation.

Si les utilisateurs finaux peuvent appartenir à plus d’une organisation, ou si vous pourriez donner à un même développeur un ID client et un secret pour des appels M2M vers plus d’une organisation, la meilleure approche consiste à créer une <Tooltip tip="Audience : identificateur unique de l’audience d’un jeton émis. Nommée aud dans un jeton, sa valeur contient l’ID soit d’une application (ID client) pour un ID Token, soit d’une API (API Identifier) pour un jeton d’accès." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=audience">audience</Tooltip> distincte (une instance d’API distincte dans votre tenant Auth0) pour chaque organisation. Cela vous offre quelques avantages intéressants :

1. D’abord, cela vous permet de transmettre l’audience à Auth0 comme paramètre natif, sans avoir à créer un paramètre personnalisé. L’avantage, c’est qu’Auth0 contribuera à en faire respecter la présence et la transmettra à vos Actions. Cela garantira aussi qu’un refresh token émis ne fonctionnera que pour l’audience précise pour laquelle il a été initialement émis.
2. Cela vous permet de restreindre les client grants à des organisations précises, sans configuration supplémentaire. L’autre option consiste à créer un hook client credentials plus complexe pour tenter de récupérer ces restrictions ailleurs, tout en exigeant aussi une méthode beaucoup plus complexe et potentiellement problématique pour indiquer à l’appel client credentials pour quelle organisation émettre le jeton d’accès.
3. Cela vous permet aussi d’utiliser la fonctionnalité RBAC de base d’Auth0 et de faire en sorte que les utilisateurs finaux qui ont accès à plus d’une organisation puissent avoir un rôle différent pour chaque organisation.
