---

La gestion du profil dans les scénarios basés sur les organisations est généralement la même que dans les autres scénarios d’architecture. Dans nos scénarios d’architecture, nous fournissons des conseils d’ordre général sur la [gestion des profils B2B](/docs/fr-CA/get-started/architecture-scenarios/business-to-business/profile-management), que nous vous recommandons de consulter en complément des indications fournies ici.

<Warning>
  Avant de pouvoir gérer les rôles associés à un utilisateur, un compte d’utilisateur doit exister dans Auth0. Cela demeure vrai même si vous utilisez la [fonctionnalité de rôles d’Auth0 Organisations associée à l’appartenance](/docs/fr-CA/manage-users/organizations/configure-organizations/add-member-roles). Toutefois, les comptes d’utilisateur provisionnés au moyen du flux de [l’invitation d’utilisateur](/docs/fr-CA/get-started/architecture-scenarios/multiple-organization-architecture/single-identity-provider-organizations/provisioning) peuvent se voir attribuer automatiquement un rôle dans le cadre du processus d’invitation. Pour en savoir plus, consultez [Inviter des membres d’une organisation](/docs/fr-CA/manage-users/organizations/configure-organizations/invite-members). Si vous préinscrivez un utilisateur par un autre mécanisme, vous devez stocker les renseignements sur le rôle à l’extérieur d’Auth0 et y accéder ou les copier au moyen de l’extensibilité (par exemple, dans le cadre d’une [Rule](/docs/fr-CA/customize/rules)) lors de la première authentification.
</Warning>

Votre application peut comporter un ensemble précis d’attributs utilisateur associés (par exemple, des préférences utilisateur ou des renseignements d’identification que vous utilisez pour mieux servir le client) pour lesquels vous offrez une certaine forme de gestion en libre-service. De plus, ou à la place, vous pouvez offrir une gestion du profil en libre-service pour les attributs habituellement maintenus par le <Tooltip tip="Fournisseur d’identité (IdP) : service qui stocke et gère les identités numériques." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Identity+Provider">fournisseur d’identité</Tooltip> (IdP).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  **Bonne pratique**

  Dans les scénarios liés aux organisations, les adresses courriel devraient toujours être vérifiées. Par conséquent, vous devriez offrir une fonctionnalité de vérification d’adresse courriel en libre-service lorsque l’adresse courriel d’un utilisateur ne peut pas être vérifiée autrement. Pour en savoir plus, consultez [Vérification des adresses courriel](/docs/fr-CA/manage-users/user-accounts/verify-emails).
</Callout>

<div id="database-connection">
  ## Connexion de base de données
</div>

Auth0 vous permet de mettre en œuvre la gestion libre-service du profil au moyen de l’Auth0 <Tooltip tip="Management API : un produit permettant aux clients d’effectuer des tâches administratives." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Management+API">Management API</Tooltip>. Si vous utilisez Auth0 Organizations pour offrir le [provisionnement des utilisateurs par invitation](/docs/fr-CA/manage-users/organizations/configure-organizations/invite-members), vous devrez probablement restreindre les modifications apportées aux champs généralement gérés par votre tenant Auth0 en tant que fournisseur d’identité (IdP). Par exemple, vous voudrez restreindre les modifications à l’adresse courriel, afin qu’un utilisateur ne puisse pas utiliser une adresse courriel autre que celle à laquelle son invitation a été envoyée. Restreindre les modifications au champ de l’adresse courriel empêcherait que des courriels propres à l’entreprise soient envoyés à des adresses courriel personnelles saisies.

Autrement, vous pourriez envisager d’offrir quelques options en libre-service aux utilisateurs qui s’authentifient au moyen d’une connexion de base de données dans Auth0. Vous pourriez permettre aux utilisateurs de :

* modifier leur adresse courriel
* modifier les numéros de téléphone associés
* modifier leur nom d’utilisateur
* retirer leurs comptes du système dans le cadre de la conformité réglementaire (comme le GDPR)
* effectuer un changement de mot de passe, que nous recommandons habituellement de mettre en œuvre au moyen de la [réinitialisation du mot de passe](/docs/fr-CA/authenticate/database-connections/password-change#trigger-an-interactive-password-reset-flow) et qui s’appuiera généralement sur l’image de marque propre à l’organisation décrite dans [Image de marque : page de réinitialisation du mot de passe](/docs/fr-CA/get-started/architecture-scenarios/multiple-organization-architecture/single-identity-provider-organizations/branding).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  La fonctionnalité libre-service doit généralement être mise en œuvre et hébergée à l’extérieur d’Auth0, et doit être très sécurisée.
</Callout>

<div id="enterprise-connection">
  ## Connexion d’entreprise
</div>

Comme le fournisseur d’identité en amont (IdP) gère habituellement les attributs du profil utilisateur administrés par l’IdP, la gestion du profil peut être presque inexistante dans ce cas d’utilisation. Toutefois, si vous utilisez des attributs utilisateur propres à l’application, vous pourriez quand même vouloir offrir une option de libre-service.

De plus, vous voudrez presque certainement donner à une organisation un moyen de déprovisionner des utilisateurs de votre tenant Auth0. Auth0 ne communique pas avec un IdP en amont, sauf lorsque la session <Tooltip tip="Authentification unique (SSO) : service qui, après qu’un utilisateur ouvre une session dans une application, connecte automatiquement cet utilisateur à d’autres applications." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=SSO">SSO</Tooltip> d’Auth0 expire. Comme le délai d’expiration d’une session SSO sera probablement trop long dans la plupart des cas où un utilisateur est supprimé, un administrateur d’organisation devra disposer d’un moyen de bloquer ou de supprimer un utilisateur de manière autonome.

<div id="social-connection">
  ## Connexion sociale
</div>

Dans le contexte des connexions sociales, la gestion du profil suit un modèle semblable à celui associé à une [Connexion d’entreprise](#enterprise-connection), mais l’IdP en amont est associé au fournisseur d’identité sociale plutôt qu’à une organisation particulière.

<div id="administration">
  ## Administration
</div>

Dans certaines situations, vous voudrez donner à vos clients l’accès nécessaire pour gérer les comptes d’utilisateur associés à leur organisation. C’est souvent le cas dans des scénarios de type centre d’assistance, où un agent peut mettre à jour les renseignements de profil au nom d’un utilisateur ou aider un utilisateur à débloquer son compte.

Par défaut, Auth0 fournit le <Tooltip tip="Auth0 Dashboard : le principal produit d’Auth0 pour configurer vos services." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip>, qui sert à l’administration générale d’un tenant Auth0. Toutefois, vous ne voudriez pas donner à un client l’accès au Dashboard de votre tenant Auth0, car il pourrait alors gérer tous les utilisateurs de toutes les organisations, ce qui ne serait pas souhaitable.

<Warning>
  Bien que le Dashboard du tenant Auth0 puisse être utilisé pour gérer des comptes d’utilisateur, il n’offre pas d’isolation au niveau des organisations. Donner à un client l’accès à votre Auth0 Dashboard lui permet de modifier des utilisateurs dans toutes les organisations, ce qui n’est pas souhaitable. Le tableau de bord Auth0 Delegated Admin peut être configuré pour permettre une administration des comptes d’utilisateur adaptée aux Organizations d’Auth0; toutefois, il ne peut pas être utilisé pour gérer l’[appartenance](/docs/fr-CA/manage-users/organizations/create-first-organization) d’un utilisateur ni les [invitations](/docs/fr-CA/manage-users/organizations/configure-organizations/invite-members).
</Warning>

Si vous offrez déjà à vos clients des fonctionnalités de type centre d’assistance, vous pouvez utiliser l’Auth0 Management API pour gérer les comptes d’utilisateur dans Auth0. Par exemple, la Management API peut servir à [récupérer les membres d’une organisation](/docs/fr-CA/manage-users/organizations/configure-organizations/retrieve-members) et les [organisations auxquelles un utilisateur appartient](/docs/fr-CA/manage-users/organizations/configure-organizations/retrieve-user-membership). Si vous n’offrez pas déjà ce type de soutien, vous devrez développer cette fonctionnalité si vous en avez besoin.
