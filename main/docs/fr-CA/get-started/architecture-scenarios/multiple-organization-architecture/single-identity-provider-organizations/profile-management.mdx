title: 'Fournisseur d’identité unique : gestion du profil'
---

La gestion du profil dans les scénarios basés sur les Organizations est généralement la même que dans les autres scénarios d’architecture. Dans nos scénarios d’architecture, nous proposons des conseils généraux sur la [gestion du profil B2B](/docs/fr-CA/get-started/architecture-scenarios/business-to-business/profile-management), que nous vous recommandons de consulter en complément des conseils présentés ici.

<Warning>
  Avant de pouvoir gérer les rôles associés à un utilisateur, un compte utilisateur doit exister dans Auth0. Cela s’applique même si vous utilisez la [fonctionnalité de rôle d’Auth0 Organizations associée à l’appartenance](/docs/fr-CA/manage-users/organizations/configure-organizations/add-member-roles). Toutefois, les comptes utilisateur approvisionnés au moyen du processus de [l’invitation d’utilisateur](/docs/fr-CA/get-started/architecture-scenarios/multiple-organization-architecture/single-identity-provider-organizations/provisioning) peuvent se voir attribuer automatiquement un rôle dans le cadre du processus d’invitation. Pour en savoir plus, consultez [Inviter des membres d’une organisation](/docs/fr-CA/manage-users/organizations/configure-organizations/invite-members). Si vous préinscrivez un utilisateur à l’aide d’un autre mécanisme, vous devez alors stocker l’information sur le rôle à l’extérieur d’Auth0 et y accéder ou la copier au moyen de l’extensibilité (par exemple, dans le cadre d’une [Rule](/docs/fr-CA/customize/rules)) lors de la première authentification.
</Warning>

Votre application peut avoir un ensemble précis d’attributs utilisateur associés (par exemple, des préférences utilisateur ou des renseignements d’identification que vous utilisez pour mieux servir le client) pour lesquels vous offrez une certaine forme de gestion libre-service. De plus, vous pouvez aussi offrir une gestion du profil en libre-service pour des attributs qui sont généralement gérés par le <Tooltip tip="Identity Provider (IdP) : service qui stocke et gère les identités numériques." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Identity+Provider">fournisseur d’identité</Tooltip> (IdP).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  **Pratique exemplaire**

  Dans les scénarios liés aux organisations, les adresses courriel devraient toujours être vérifiées. Par conséquent, vous devriez offrir une fonctionnalité libre-service de vérification de l’adresse courriel lorsque l’adresse courriel d’un utilisateur ne peut pas être vérifiée autrement. Pour en savoir plus, consultez [Vérification de l’adresse courriel](/docs/fr-CA/manage-users/user-accounts/verify-emails).
</Callout>

<div id="database-connection">
  ## Connexion de base de données
</div>

Auth0 vous permet de prendre en charge la gestion libre-service du profil au moyen de l’Auth0 <Tooltip tip="Management API : un produit qui permet aux clients d’effectuer des tâches administratives." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Management+API">Management API</Tooltip>. Si vous utilisez Auth0 Organizations pour offrir le [Provisioning des utilisateurs par invitation](/docs/fr-CA/manage-users/organizations/configure-organizations/invite-members), vous devrez probablement limiter les modifications aux champs qui relèvent habituellement de votre tenant Auth0 en tant que fournisseur d’identité (IdP). Par exemple, vous voudrez limiter les modifications à l’adresse courriel, puisqu’un utilisateur ne devrait pas pouvoir utiliser une adresse courriel autre que celle à laquelle son invitation a été envoyée. Le fait de limiter les modifications au champ de l’adresse courriel empêcherait l’envoi de courriels propres à l’entreprise à des adresses courriel personnelles saisies.

Vous pourriez aussi envisager d’offrir certaines options libre-service aux utilisateurs qui s’authentifient au moyen d’une connexion de base de données dans Auth0. Vous pourriez vouloir permettre aux utilisateurs de :

* changer leur adresse courriel
* changer tout numéro de téléphone associé
* changer leur nom d’utilisateur
* déprovisionner leur compte pour respecter des exigences réglementaires (comme le GDPR)
* effectuer un changement de mot de passe, ce que nous recommandons généralement de mettre en œuvre au moyen de la [réinitialisation du mot de passe](/docs/fr-CA/authenticate/database-connections/password-change#trigger-an-interactive-password-reset-flow), laquelle tirera généralement parti de l’image de marque propre à l’organisation décrite dans [Image de marque : page de réinitialisation du mot de passe](/docs/fr-CA/get-started/architecture-scenarios/multiple-organization-architecture/single-identity-provider-organizations/branding).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Les fonctionnalités libre-service doivent généralement être mises en œuvre et hébergées à l’extérieur d’Auth0, et elles doivent être hautement sécurisées.
</Callout>

<div id="enterprise-connection">
  ## Connexion d’entreprise
</div>

Comme le fournisseur d’identité en amont (IdP) gère généralement les attributs du profil utilisateur qui relèvent de lui, la gestion du profil peut être presque inexistante dans ce cas d’utilisation. Cependant, si vous utilisez des attributs utilisateur propres à l’application, vous pourriez quand même vouloir offrir une option de libre-service.

De plus, vous voudrez presque certainement offrir à une organisation un moyen de déprovisionner des utilisateurs de votre tenant Auth0. Auth0 ne communique avec un IdP en amont que lorsque la session <Tooltip tip="Authentification unique (SSO) : service qui, après qu’un utilisateur se connecte à une application, connecte automatiquement cet utilisateur à d’autres applications." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=SSO">SSO</Tooltip> d’Auth0 expire. Comme le délai avant l’expiration d’une session SSO sera probablement trop long dans la plupart des scénarios où un utilisateur est supprimé, un administrateur d’organisation devra pouvoir bloquer ou supprimer un utilisateur de façon indépendante.

<div id="social-connection">
  ## Connexion sociale
</div>

Dans le contexte des connexions sociales, la gestion du profil suit un modèle semblable à celui d’une [Connexion d’entreprise](#enterprise-connection), mais l’IdP en amont est associé au fournisseur d’identité sociale plutôt qu’à une organisation donnée.

<div id="administration">
  ## Administration
</div>

Dans certaines situations, vous voudrez permettre à vos clients de gérer les comptes d’utilisateur associés à leur organisation. C’est souvent le cas dans des scénarios de type centre d’assistance, où un agent du soutien peut mettre à jour les renseignements du profil au nom d’un utilisateur ou aider un utilisateur à débloquer son compte.

Par défaut, Auth0 fournit le <Tooltip tip="Auth0 Dashboard : le produit principal d’Auth0 pour configurer vos services." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip>, qui sert à la gestion générale d’un tenant Auth0. Cependant, vous ne voudriez pas donner à un client l’accès à l’Auth0 Dashboard de votre tenant Auth0, car il pourrait alors gérer tous les utilisateurs de toutes les organisations, ce qui ne serait pas souhaitable.

<Warning>
  Bien que l’Auth0 Dashboard du tenant puisse être utilisé pour gérer des comptes d’utilisateur, il n’offre pas d’isolation au niveau de l’organisation. Donner à un client l’accès à votre Auth0 Dashboard lui permet de modifier des utilisateurs dans toutes les organisations, ce qui n’est pas souhaitable. Le tableau de bord Auth0 Delegated Admin peut être configuré pour offrir une administration des comptes d’utilisateur adaptée aux organisations Auth0; toutefois, il ne peut pas être utilisé pour gérer l’[appartenance](/docs/fr-CA/manage-users/organizations/create-first-organization) ou l’[invitation](/docs/fr-CA/manage-users/organizations/configure-organizations/invite-members) d’un utilisateur.
</Warning>

Si vous offrez déjà à vos clients des fonctions de type centre d’assistance, vous pouvez utiliser l’Auth0 Management API pour gérer les comptes d’utilisateur dans Auth0. Par exemple, la Management API peut être utilisée pour [récupérer les membres d’une organisation](/docs/fr-CA/manage-users/organizations/configure-organizations/retrieve-members) et [les organisations auxquelles un utilisateur appartient](/docs/fr-CA/manage-users/organizations/configure-organizations/retrieve-user-membership). Si vous n’offrez pas déjà ce type de soutien, vous devrez développer cette fonctionnalité si vous en avez besoin.
