Quand vous réfléchissez à l’autorisation, vous devez généralement déterminer comment établir ce qu’une personne a le droit de faire et comment communiquer cette information à vos applications et/ou API. Selon les applications dont vous disposez, un seul ou les deux aspects peuvent vous concerner. Dans nos scénarios d’architecture, nous fournissons des conseils d’ordre général sur [l’autorisation B2B](/docs/fr-CA/get-started/architecture-scenarios/business-to-business/authorization), que nous vous recommandons de consulter en complément des conseils fournis ici.

* Les [jetons d’identité](/docs/fr-CA/secure/tokens/id-tokens) servent souvent à transmettre aux applications des informations d’autorisation sur l’utilisateur au moyen de revendications personnalisées, qui peuvent être ajoutées à l’aide de l’extensibilité [Rules](/docs/fr-CA/customize/rules). Les revendications ajoutées peuvent vous permettre de présenter une interface utilisateur dans laquelle les utilisateurs ne peuvent pas essayer d’effectuer une action qu’ils n’ont pas le droit d’exécuter. Les informations d’autorisation contenues dans un <Tooltip tip="Jeton d’identité : identifiant destiné au client lui-même, plutôt qu’à l’accès à une ressource." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=ID+Token">jeton d’identité</Tooltip> permettent aussi à tout backend applicatif d’empêcher les utilisateurs de contourner les contrôles du frontend dans les applications web traditionnelles.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  **Pratique exemplaire**

  Vous pouvez tirer parti de [Auth0 Role-Based Access Control](/docs/fr-CA/manage-users/access-control/rbac) (RBAC) grâce à la fonctionnalité [Auth0 Authorization Core](/docs/fr-CA/manage-users/access-control/configure-core-rbac) pour définir des permissions d’accès, qui peuvent être automatiquement appliquées aux jetons d’accès. Pour en savoir plus, consultez [activer le RBAC d’Auth0 Authorization Core pour une API](/docs/fr-CA/get-started/apis/enable-role-based-access-control-for-apis).

  La fonctionnalité RBAC d’Auth0 peut aussi fournir des informations que vous pouvez ajouter sous forme de revendications personnalisées aux jetons d’identité (et aux jetons d’accès, si vous préférez les appliquer manuellement). Auth0 Organizations peut aussi tirer parti des capacités RBAC d’Auth0 au moyen d’un ou de plusieurs rôles attribués aux membres. Pour en savoir plus, consultez [Add Roles to Organization Members](/docs/fr-CA/manage-users/organizations/configure-organizations/add-member-roles).
</Callout>

* Les API qui offrent un accès public à des services de ressources partagées sont généralement protégées par des mécanismes de contrôle d’accès. À cette fin, Auth0 permet de créer un bearer token d’autorisation, c’est-à-dire un <Tooltip tip="OAuth 2.0 : cadre d’autorisation qui définit les protocoles et les workflows d’autorisation." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=OAuth">OAuth</Tooltip> 2 [jeton d’accès](/docs/fr-CA/secure/tokens/access-tokens), qui peut transmettre à une API des informations d’autorisation sur l’utilisateur, généralement en utilisant [Auth0 Role-Based Access Control (RBAC)](/docs/fr-CA/manage-users/access-control/configure-core-rbac) d’Auth0 pour appliquer un ou plusieurs [rôles attribués aux membres](/docs/fr-CA/manage-users/organizations/configure-organizations/add-member-roles), ou en ajoutant des revendications personnalisées via l’extensibilité [Rules](/docs/fr-CA/customize/rules). Vous pouvez aussi tirer parti des capacités RBAC d’Auth0 pour ajuster automatiquement la revendication `scope` d’un <Tooltip tip="Jeton d’accès : identifiant d’autorisation, sous la forme d’une chaîne opaque ou d’un JWT, utilisé pour accéder à une API." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=access+token">jeton d’accès</Tooltip>. Les API peuvent ensuite utiliser cette information pour appliquer le niveau de contrôle d’accès approprié, ce qui permet à votre API d’appliquer des règles de politique sans devoir effectuer une recherche supplémentaire pour obtenir des renseignements sur l’utilisateur.
* Dans certains cas, vous pourriez vouloir mettre en place des politiques au niveau de l’application dans le tenant Auth0; cela vous permet d’appliquer des politiques à tout un éventail d’applications et de services de ressources (API) sans avoir à modifier chacun séparément. En général, cela se fait au moyen de l’extensibilité [Rules](/docs/fr-CA/customize/rules).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  **Pratique exemplaire**

  Auth0 Organizations permet à Auth0 Rules d’accéder à des informations centrées sur l’organisation lors de l’authentification d’un utilisateur. Ces informations sont accessibles au moyen de l’objet `organization` contenu dans l’objet `context` de Rules. L’objet `organization` donne aussi accès aux métadonnées associées à une définition d’Organization dans Auth0. Pour en savoir plus, consultez [Custom Development for Organizations](/docs/fr-CA/manage-users/organizations/custom-development).
</Callout>

<div id="id-token-claims">
  ## Revendications du jeton d’identité
</div>

En règle générale, des revendications peuvent être ajoutées aux jetons d’identité, comme l’explique notre guide de bonnes pratiques sur les [revendications du jeton d’identité](/docs/fr-CA/get-started/architecture-scenarios/business-to-consumer/authorization). Lorsque vous utilisez la fonctionnalité Auth0 Organizations, une revendication `org_id` est automatiquement ajoutée à tout jeton d’identité (pour un exemple, consultez [Utiliser les jetons et les organisations](/docs/fr-CA/manage-users/organizations/using-tokens)) émis pour les utilisateurs membres d’une organisation. Ce paramètre est validé par les SDK Auth0. Vous pouvez aussi ajouter des informations supplémentaires associées à une organisation Auth0 en ajoutant une revendication personnalisée au jeton d’identité :

```js wrap lines
context.idToken['http://travel0.net/multifactor'] = context.multifactor;
```

**Remarque** : Si vous avez configuré votre tenant pour prendre en charge les noms d’organisation dans l’Authentication API, le claim `org_name` est automatiquement inclus dans les ID tokens. Pour en savoir plus, consultez [Use Organization Names in Authentication API](/docs/fr-CA/manage-users/organizations/configure-organizations/use-org-name-authentication-api).

<div id="saml-assertion">
  ### Assertion SAML
</div>

Bien que la fonctionnalité Auth0 Organization ne prenne pas en charge les applications compatibles avec <Tooltip tip="Security Assertion Markup Language (SAML) : protocole normalisé permettant à deux parties d’échanger des informations d’authentification sans mot de passe." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=SAML">SAML</Tooltip>, une assertion SAML générée par un <Tooltip tip="Identity Provider (IdP) : service qui stocke et gère les identités numériques." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Identity+Provider">fournisseur d’identité</Tooltip> (IdP) en amont peut être configurée pour alimenter des revendications standard ou personnalisées dans un jeton d’identité utilisé en aval. Par exemple, vous pouvez définir la section des mappings d’une connexion d’entreprise SAML :

```json lines
{ 
  "user_id": [
    "http://schemas.xmlsoap.org/ws/2005/05/identity/claims/nameidentifier",
    "http://schemas.xmlsoap.org/ws/2005/05/identity/claims/upn",
    "http://schemas.xmlsoap.org/ws/2005/05/identity/claims/name"
  ],
  "email": "http://schemas.xmlsoap.org/ws/2005/05/identity/claims/emailaddress",
  "name": "http://schemas.xmlsoap.org/ws/2005/05/identity/claims/name",
  "given_name": [
    "http://schemas.xmlsoap.org/ws/2005/05/identity/claims/givenname",
    "http://schemas.xmlsoap.org/ws/2005/05/identity/claims/name"
  ], 
  "family_name": "http://schemas.xmlsoap.org/ws/2005/05/identity/claims/surname",
  "groups": "http://schemas.xmlsoap.org/claims/Group"
}
```

Dans cet exemple, chaque champ se verra attribuer une valeur dans l’objet `user` d’une Rule, qui correspondra soit à des revendications standard dans un jeton d’identité, soit pourra être mappée au moyen de revendications personnalisées. Pour en savoir plus sur la personnalisation des mappings SAML, consultez [Connecter votre application aux fournisseurs d’identité SAML : Configurer les mappings](/docs/fr-CA/authenticate/identity-providers/enterprise-identity-providers/saml#set-up-mappings).

<div id="access-token-claims">
  ## Revendications du jeton d’accès
</div>

En plus des autres revendications que vous ajoutez à votre jeton d’accès pour prendre des décisions de contrôle d’accès (consultez nos recommandations générales de [Revendications du jeton d’accès](/docs/fr-CA/get-started/architecture-scenarios/business-to-consumer/authorization)), vous voudrez généralement indiquer l’organisation à laquelle l’utilisateur appartient.

Comme pour le jeton d’identité, lorsque vous utilisez la fonctionnalité Auth0 Organizations, la revendication `org_id` est automatiquement ajoutée à tout jeton d’accès (pour un exemple, consultez [Utiliser les jetons et les organisations](/docs/fr-CA/manage-users/organizations/using-tokens)) émis pour des utilisateurs membres d’une organisation. Vous pouvez aussi ajouter des renseignements supplémentaires associés à une organisation Auth0 en ajoutant une revendication personnalisée au jeton d’accès :

```js wrap lines
context.accessToken['http://travel0.net/multifactor'] = context.multifactor;
```

**Remarque** : Si vous avez configuré votre tenant pour prendre en charge les noms d’organisation dans l’Authentication API, la revendication `org_name` est automatiquement incluse dans les jetons d’accès. Pour en savoir plus, consultez [Use Organization Names in Authentication API](/docs/fr-CA/manage-users/organizations/configure-organizations/use-org-name-authentication-api).

Vous pouvez aussi créer une <Tooltip tip="Audience : identifiant unique de l’audience d’un jeton émis. Appelée aud dans un jeton, sa valeur contient l’ID soit d’une application (Client ID) pour un ID Token, soit d’une API (API Identifier) pour un jeton d’accès." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=audience">audience</Tooltip> d’API distincte pour chaque organisation, ce qui créerait une définition d’API distincte dans Auth0. Bien que ce mécanisme puisse réduire le recours à l’extensibilité personnalisée par Rule, la complexité qu’il ajoute peut être difficile à gérer. Voici une comparaison simple :

| Approach            | Pros                                                                                                                                                                                                                                                                                    | Cons                                                                                                                                                                                                                                                                                                               |
| ------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Unique API Audience | <ul><li>Prise en charge prête à l’emploi de l’accès machine-à-machine pour une seule organisation.</li><li>L’audience est une revendication standard dans un jeton d’accès.</li><li>Le traitement du Refresh Token ne nécessite aucune logique d’organisation supplémentaire.</li></ul> | <ul><li>Il faut automatiser la création d’une API pour chaque organisation.</li><li>Il peut être nécessaire de créer des rôles distincts si vous utilisez RBAC.</li>Il faut automatiser l’attribution des Roles aux Membership.<li>La mise en œuvre de l’API doit prendre en charge plusieurs audiences.</li></ul> |
| Custom Claim        | Simplifie la configuration du tenant Auth0.                                                                                                                                                                                                                                             | Du code personnalisé est nécessaire dans une rule pour ajouter l’organisation au jeton d’accès.                                                                                                                                                                                                                    |

<div id="roles">
  ### Rôles
</div>

La fonctionnalité Auth0 Organizations prend également en charge le [contrôle d’accès basé sur les rôles (RBAC)](/docs/fr-CA/manage-users/access-control/rbac) au moyen de la fonctionnalité [Authorization Core](/docs/fr-CA/manage-users/access-control/configure-core-rbac) associée à un tenant Auth0. Le RBAC est [appliqué au niveau de l’appartenance à une organisation Auth0](/docs/fr-CA/manage-users/organizations/configure-organizations/add-member-roles).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Vous pouvez [activer le RBAC d’Auth0 Authorization Core pour une API](/docs/fr-CA/get-started/apis/enable-role-based-access-control-for-apis), ce qui modifiera automatiquement la revendication `scope` par défaut dans les jetons d’accès et ajoutera, par défaut, une revendication `permission` (pour un exemple, consultez [Utiliser les jetons et les organisations](/docs/fr-CA/manage-users/organizations/using-tokens)). Vous pouvez aussi ajouter des renseignements sur les rôles aux jetons d’identité sous forme de revendications personnalisées en accédant à l’objet `authorization` disponible dans l’objet `context` de Rules. Pour en savoir plus, consultez [Exemples de cas d’utilisation de Rules avec Authorization : ajouter les rôles d’utilisateur aux jetons](/docs/fr-CA/manage-users/access-control/sample-use-cases-rules-with-authorization).
</Callout>

<div id="access-control">
  ## Contrôle d&#39;accès
</div>

L’application des politiques au niveau des ressources relève des applications et/ou des API de votre système. Si vous tentez d’appliquer des politiques dans un <Tooltip tip="Authorization Server : serveur centralisé qui contribue à définir les limites de l’accès d’un utilisateur. Par exemple, votre serveur d’autorisation peut contrôler les données, les tâches et les fonctionnalités offertes à un utilisateur." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Authorization+Server">Authorization Server</Tooltip> centralisé, comme votre tenant Auth0, vous vous retrouverez rapidement avec un système de contrôle complexe, difficile à maintenir et à comprendre. Votre Authorization Server centralisé peut plutôt s’assurer que les renseignements appropriés sur un utilisateur sont inclus dans les jetons, afin que vos applications et API disposent des renseignements nécessaires pour prendre des décisions d’application des politiques. À tout le moins, dans les situations où il y a trop de renseignements pour les inclure dans un seul jeton (par exemple, des permissions au niveau des ressources) ou lorsque ces renseignements changent assez souvent pour être périmés s’ils sont consultés directement dans le jeton, vos applications et API devraient être en mesure d’aller chercher les bons renseignements.

En revanche, une certaine application des politiques à haut niveau peut être gérée de façon centralisée. Par exemple, lorsque vous utilisez le contexte d’un tenant Auth0, il existe certaines situations où vous pourriez implémenter une Rule qui évite que chaque application et/ou API doive appliquer les mêmes restrictions. En voici quelques-unes :

* bloquer l’accès des utilisateurs provenant d’une adresse IP particulière
* implémenter des exigences précises liées à la MFA contextuelle ou à la <Tooltip tip="Adaptive Multi-factor Authentication : authentification multifacteur (MFA) qui n’est déclenchée pour les utilisateurs que lorsqu’une tentative de connexion est considérée comme ayant un faible niveau de confiance." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Adaptive+MFA">Adaptive MFA</Tooltip>
* restreindre la connexion aux seuls utilisateurs qui ont vérifié leur adresse courriel
* restreindre l’accès à une audience d’API donnée, afin qu’un utilisateur ne puisse pas obtenir de jeton d’accès pour une autre audience d’API, ou ne puisse pas obtenir de jeton d’accès pour cette audience dans certaines circonstances. Dans ce cas, si vous créez une audience d’API personnalisée pour chaque organization, votre Rule doit aussi s’assurer que l’utilisateur qui s’authentifie appartient à l’organization correspondant à l’audience d’API.

<Warning>
  L’accès à l’Auth0 Management API n’est pas restreint par organization; l’accès à l’Auth0 Management API se fait au moyen d’un jeton d’accès attribué à une utilisation dans un contexte machine-to-machine et ne peut pas être contraint par une Auth0 Organization. Par conséquent, ne donnez pas à vos clients un accès direct à votre instance de l’Auth0 Management API. Si vos clients doivent gérer certains aspects de leur organization, comme les comptes d’utilisateurs (voir [Gestion du profil](/docs/fr-CA/get-started/architecture-scenarios/multiple-organization-architecture/single-identity-provider-organizations/profile-management)), vous devriez plutôt créer votre propre application et/ou API indépendante à cette fin.
</Warning>
