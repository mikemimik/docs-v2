title: 'Fournisseur d’identité unique : autorisation'
---

Lorsque vous réfléchissez à l’autorisation, vous devez généralement vous demander comment déterminer ce qu’une personne a le droit de faire et comment communiquer cette information à vos applications et/ou API. Selon les applications dont vous disposez, l’un ou l’autre de ces aspects — ou les deux — peut vous concerner. Dans nos scénarios d’architecture, nous fournissons des conseils d’ordre général sur [l’autorisation B2B](/docs/fr-CA/get-started/architecture-scenarios/business-to-business/authorization), que nous vous recommandons de consulter en complément des indications fournies ici.

* Les [ID Tokens](/docs/fr-CA/secure/tokens/id-tokens) servent souvent à transmettre aux applications des informations d’autorisation sur l’utilisateur au moyen de claims personnalisés, qui peuvent être ajoutés à l’aide de la fonctionnalité d’extensibilité [Rules](/docs/fr-CA/customize/rules). Les claims ajoutés peuvent vous permettre de présenter une interface utilisateur dans laquelle les utilisateurs ne peuvent même pas tenter d’effectuer une action pour laquelle ils n’ont pas de permission. Les informations d’autorisation contenues dans un <Tooltip tip="ID Token : justificatif destiné au client lui-même, plutôt qu’à l’accès à une ressource." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=ID+Token">ID Token</Tooltip> permettent aussi au backend d’une application d’empêcher les utilisateurs de contourner les contrôles du frontend dans les applications web traditionnelles.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  **Bonne pratique**

  Vous pouvez tirer parti de [Auth0 Role-Based Access Control](/docs/fr-CA/manage-users/access-control/rbac) (RBAC) au moyen de la fonctionnalité [Auth0 Authorization Core](/docs/fr-CA/manage-users/access-control/configure-core-rbac) pour définir des permissions d’accès, qui peuvent être automatiquement appliquées aux jetons d’accès. Pour plus de détails, consultez [Enable Role-Based Access Control for APIs](/docs/fr-CA/get-started/apis/enable-role-based-access-control-for-apis).

  La fonctionnalité RBAC d’Auth0 peut aussi fournir des informations que vous pouvez ajouter sous forme de claims personnalisés aux jetons d’identité (et aux jetons d’accès, si vous préférez les appliquer manuellement). Auth0 Organizations peut aussi tirer parti des capacités RBAC d’Auth0 au moyen d’un ou de plusieurs rôles attribués à l’adhésion. Pour en savoir plus, consultez [Add Roles to Organization Members](/docs/fr-CA/manage-users/organizations/configure-organizations/add-member-roles).
</Callout>

* Les API qui offrent un accès public à des services de ressources partagées sont généralement protégées par des mécanismes de contrôle d’accès. À cette fin, Auth0 permet de créer un jeton d’autorisation de type bearer, ou <Tooltip tip="OAuth 2.0 : cadre d’autorisation qui définit les protocoles et workflows d’autorisation." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=OAuth">OAuth</Tooltip> 2 [jeton d’accès](/docs/fr-CA/secure/tokens/access-tokens), qui peut transmettre à une API des informations d’autorisation sur l’utilisateur, habituellement en utilisant [Role-Based Access Control (RBAC)](/docs/fr-CA/manage-users/access-control/configure-core-rbac) d’Auth0 pour appliquer un ou plusieurs [rôles attribués à l’adhésion](/docs/fr-CA/manage-users/organizations/configure-organizations/add-member-roles), ou en ajoutant des claims personnalisés au moyen de la fonctionnalité d’extensibilité [Rules](/docs/fr-CA/customize/rules). Vous pouvez également tirer parti des capacités RBAC d’Auth0 pour ajuster automatiquement le claim `scope` d’un <Tooltip tip="Access Token : justificatif d’autorisation, sous la forme d’une chaîne opaque ou d’un JWT, utilisé pour accéder à une API." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=access+token">jeton d’accès</Tooltip>. Les API peuvent ensuite utiliser cette information pour appliquer le niveau de contrôle d’accès approprié, ce qui permet à votre API d’appliquer des règles de politique sans devoir effectuer une recherche supplémentaire pour obtenir de l’information sur l’utilisateur.
* Dans certains cas, vous pourriez vouloir mettre en œuvre des politiques au niveau de l’application dans le locataire Auth0; cela vous permet d’appliquer des politiques à tout un ensemble d’applications et de services de ressources (API) sans avoir à les modifier individuellement. En général, cela se fait au moyen de la fonctionnalité d’extensibilité [Rules](/docs/fr-CA/customize/rules).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  **Bonne pratique**

  Auth0 Organizations donne à Auth0 Rules accès à des informations propres à l’organisation, accessibles lors de l’authentification d’un utilisateur. Cette information est disponible au moyen de l’objet `organization` contenu dans l’objet `context` de Rules. L’objet `organization` donne aussi accès à toutes les métadonnées provisionnées dans une définition d’Organization dans Auth0. Pour en savoir plus, consultez [Custom Development for Organizations](/docs/fr-CA/manage-users/organizations/custom-development).
</Callout>

<div id="id-token-claims">
  ## Revendications du jeton d’identité
</div>

En général, des revendications peuvent être ajoutées aux jetons d’identité, comme l’explique notre guide de pratique exemplaire [Revendications du jeton d’identification](/docs/fr-CA/get-started/architecture-scenarios/business-to-consumer/authorization). Lorsque vous utilisez la fonctionnalité Auth0 Organizations, une claim `org_id` est automatiquement ajoutée à tout jeton d’identité (pour un exemple, consultez [Work with Tokens and Organizations](/docs/fr-CA/manage-users/organizations/using-tokens)) émis pour les utilisateurs ayant une appartenance à une organisation. Ce paramètre est validé par les Auth0 SDKs. Vous pouvez également ajouter des renseignements supplémentaires associés à une Auth0 Organization en ajoutant une claim personnalisée au jeton d’identité :

```js wrap lines
context.idToken['http://travel0.net/multifactor'] = context.multifactor;
```

**Remarque** : Si vous avez configuré votre tenant pour prendre en charge les noms d’organisation dans l’Authentication API, la claim `org_name` est automatiquement incluse dans les ID tokens. Pour en savoir plus, consultez [Use Organization Names in Authentication API](/docs/fr-CA/manage-users/organizations/configure-organizations/use-org-name-authentication-api).

<div id="saml-assertion">
  ### Assertion SAML
</div>

Bien que la fonctionnalité Auth0 Organization ne prenne pas en charge les applications compatibles avec <Tooltip tip="Security Assertion Markup Language (SAML) : protocole normalisé permettant à deux parties d’échanger de l’information d’authentification sans mot de passe." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=SAML">SAML</Tooltip>, une assertion SAML générée par un <Tooltip tip="Fournisseur d’identité (IdP) : service qui stocke et gère les identités numériques." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Identity+Provider">fournisseur d’identité</Tooltip> (IdP) en amont peut être configurée pour renseigner des claims standard ou personnalisées dans un jeton d’identité utilisé en aval. Par exemple, vous pourriez définir la section des mappages d’une connexion d’entreprise SAML :

```json lines
{ 
  "user_id": [
    "http://schemas.xmlsoap.org/ws/2005/05/identity/claims/nameidentifier",
    "http://schemas.xmlsoap.org/ws/2005/05/identity/claims/upn",
    "http://schemas.xmlsoap.org/ws/2005/05/identity/claims/name"
  ],
  "email": "http://schemas.xmlsoap.org/ws/2005/05/identity/claims/emailaddress",
  "name": "http://schemas.xmlsoap.org/ws/2005/05/identity/claims/name",
  "given_name": [
    "http://schemas.xmlsoap.org/ws/2005/05/identity/claims/givenname",
    "http://schemas.xmlsoap.org/ws/2005/05/identity/claims/name"
  ], 
  "family_name": "http://schemas.xmlsoap.org/ws/2005/05/identity/claims/surname",
  "groups": "http://schemas.xmlsoap.org/claims/Group"
}
```

Dans cet exemple, chaque champ se verra attribuer une valeur dans l’objet `user` d’une Rule; cette valeur sera ensuite soit mappée à des claims standard dans un jeton d’identité, soit mappée au moyen de claims personnalisés. Pour en savoir plus sur la personnalisation des mappages SAML, consultez [Connectez votre application aux fournisseurs d’identité SAML : Configurer les mappages](/docs/fr-CA/authenticate/identity-providers/enterprise-identity-providers/saml#set-up-mappings).

<div id="access-token-claims">
  ## Claims du jeton d’accès
</div>

En plus de tout autre claim que vous ajoutez à votre jeton d’accès pour prendre des décisions de contrôle d’accès (consultez nos conseils généraux sur les pratiques exemplaires concernant les [claims du jeton d’accès](/docs/fr-CA/get-started/architecture-scenarios/business-to-consumer/authorization)), vous voudrez généralement indiquer l’organisation à laquelle appartient l’utilisateur.

Comme pour le jeton d’identité, lorsque vous utilisez la fonctionnalité Auth0 Organizations, le claim `org_id` est automatiquement ajouté à tout jeton d’accès (pour un exemple, consultez [Work with Tokens and Organizations](/docs/fr-CA/manage-users/organizations/using-tokens)) émis pour les utilisateurs membres d’une organisation. Vous pouvez aussi ajouter des renseignements complémentaires associés à une Auth0 Organization en ajoutant un claim personnalisé au jeton d’accès :

```js wrap lines
context.accessToken['http://travel0.net/multifactor'] = context.multifactor;
```

**Remarque** : Si vous avez configuré votre tenant pour prendre en charge les noms d’organisation dans l’Authentication API, la claim `org_name` est automatiquement incluse dans les jetons d’accès. Pour en savoir plus, consultez [Use Organization Names in Authentication API](/docs/fr-CA/manage-users/organizations/configure-organizations/use-org-name-authentication-api).

Vous pouvez aussi créer une <Tooltip tip="Audience : Identifiant unique de l’audience d’un jeton émis. Nommée aud dans un jeton, sa valeur contient l’ID d’une application (Client ID) pour un ID Token ou d’une API (API Identifier) pour un Access Token." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=audience">audience</Tooltip> d’API distincte pour chaque organisation, ce qui créerait une définition d’API distincte dans Auth0. Bien que ce mécanisme puisse réduire le recours à l’extensibilité personnalisée au moyen de Rule, la complexité qu’il ajoute peut être difficile à gérer. Voici une comparaison simple :

| Approach                    | Pros                                                                                                                                                                                                                                                                                        | Cons                                                                                                                                                                                                                                                                                                             |
| --------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Audience d’API unique       | <ul><li>Prise en charge prête à l’emploi de l’accès machine-to-machine pour une seule organisation.</li><li>L’audience est une revendication standard dans un jeton d’accès.</li><li>Le traitement du Refresh Token ne nécessite aucune logique organisationnelle supplémentaire.</li></ul> | <ul><li>Il faut automatiser la création d’une API pour chaque organisation.</li><li>Il peut être nécessaire de créer des rôles distincts si vous utilisez RBAC.</li>Il faut automatiser l’attribution des Roles à Membership.<li>La mise en œuvre de l’API doit prendre en charge plusieurs audiences.</li></ul> |
| Revendication personnalisée | Simplifie la configuration du tenant Auth0.                                                                                                                                                                                                                                                 | Du code personnalisé est nécessaire dans une Rule pour ajouter l’organisation au jeton d’accès.                                                                                                                                                                                                                  |

<div id="roles">
  ### Rôles
</div>

La fonctionnalité Auth0 Organizations prend également en charge le [contrôle d’accès basé sur les rôles (RBAC)](/docs/fr-CA/manage-users/access-control/rbac) au moyen de la fonctionnalité [Authorization Core](/docs/fr-CA/manage-users/access-control/configure-core-rbac) associée à un tenant Auth0. Le RBAC est [appliqué au niveau de l’appartenance à une Auth0 Organization](/docs/fr-CA/manage-users/organizations/configure-organizations/add-member-roles).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Vous pouvez [activer Auth0 Authorization Core RBAC pour une API](/docs/fr-CA/get-started/apis/enable-role-based-access-control-for-apis), ce qui modifiera automatiquement la claim `scope` par défaut dans les jetons d’accès et ajoutera également, par défaut, une claim `permission` (pour un exemple, consultez [Work with Tokens and Organizations](/docs/fr-CA/manage-users/organizations/using-tokens)). Vous pouvez aussi ajouter des informations sur les rôles aux jetons d’identité sous forme de claims personnalisées en accédant à l’objet `authorization` disponible dans l’objet `context` de Rules. Pour en savoir plus, consultez [Rules with Authorization Sample Use Cases: Add User Roles to Tokens](/docs/fr-CA/manage-users/access-control/sample-use-cases-rules-with-authorization).
</Callout>

<div id="access-control">
  ## Contrôle d’accès
</div>

L’application des politiques au niveau des ressources relève des applications et/ou des API de votre système. Si vous tentez d’appliquer des politiques dans un <Tooltip tip="Serveur d’autorisation : serveur centralisé qui contribue à définir les limites de l’accès d’un utilisateur. Par exemple, votre serveur d’autorisation peut contrôler les données, les tâches et les fonctionnalités offertes à un utilisateur." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Authorization+Server">serveur d’autorisation</Tooltip> centralisé, comme votre tenant Auth0, vous vous retrouverez rapidement avec un système de contrôle complexe, difficile à maintenir et à comprendre. Votre serveur d’autorisation centralisé peut plutôt veiller à ce que les renseignements pertinents sur un utilisateur soient inclus dans les jetons, afin que vos applications et vos API disposent des renseignements nécessaires pour prendre des décisions relatives à l’application des politiques. À tout le moins, dans les situations où il y a trop de renseignements pour tenir dans un seul jeton (par exemple, des permissions au niveau des ressources) ou lorsque les renseignements changent assez souvent pour devenir périmés s’ils sont consultés directement dans le jeton, vos applications et vos API devraient pouvoir aller chercher les bons renseignements.

Certaines applications de politiques de haut niveau, en revanche, peuvent être gérées de façon centralisée. Par exemple, lorsque vous utilisez le contexte d’un tenant Auth0, il existe des situations dans lesquelles vous pourriez implémenter une Rule, ce qui évite à chaque application et/ou API d’appliquer les mêmes restrictions. Celles-ci comprennent :

* bloquer l’accès aux utilisateurs provenant d’une adresse IP particulière
* implémenter des exigences précises liées au Contextual ou à <Tooltip tip="Adaptive Multi-factor Authentication : authentification multifacteur (MFA) qui n’est déclenchée pour les utilisateurs que lorsqu’une tentative de connexion est jugée comme étant une connexion à faible niveau de confiance." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Adaptive+MFA">Adaptive MFA</Tooltip>
* limiter la connexion aux seuls utilisateurs qui ont vérifié leur adresse courriel
* limiter l’accès à une audience d’API donnée, de sorte qu’un utilisateur ne puisse pas obtenir de jeton d’accès pour une autre audience d’API ou ne puisse pas obtenir de jeton d’accès pour cette audience dans certaines circonstances. Dans ce cas, si vous créez une audience d’API personnalisée pour chaque organisation, votre Rule doit aussi s’assurer que l’utilisateur qui s’authentifie appartient à l’organisation correspondant à l’audience d’API.

<Warning>
  L’accès à l’Auth0 Management API n’est pas limité par organisation; l’accès à l’Auth0 Management API se fait au moyen d’un jeton d’accès attribué à une utilisation dans un contexte machine-to-machine et ne peut pas être limité par une organisation Auth0. Par conséquent, n’accordez pas à vos clients un accès direct à votre instance de l’Auth0 Management API. Si vos clients doivent gérer certains aspects de leur organisation, comme les comptes d’utilisateur (consultez [Gestion du profil](/docs/fr-CA/get-started/architecture-scenarios/multiple-organization-architecture/single-identity-provider-organizations/profile-management)), vous devriez alors créer votre propre application et/ou API indépendante à cette fin.
</Warning>
