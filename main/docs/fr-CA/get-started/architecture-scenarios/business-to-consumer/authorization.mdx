Commençons par prendre un peu de recul et par parler du contrôle d’accès. Il n’existe pas de définition unique et tranchée du contrôle d’accès dans l’industrie, mais si vous prenez le temps de faire quelques recherches et lectures, vous verrez que la plupart des sources faisant autorité s’entendent pour dire qu’il s’agit du concept englobant qui réunit l’authentification, l’autorisation, le consentement et l’application des politiques afin de garantir que seules les bonnes personnes et les bons services ont accès à vos applications et à vos API. Examinons maintenant de plus près les distinctions entre l’authentification, l’autorisation, le consentement et l’application des politiques. Votre tenant <Tooltip tip="Groupe d’utilisateurs logiquement isolé qui partage un accès commun avec des privilèges précis à une seule instance logicielle." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Tenant">tenant</Tooltip> Auth0 (votre <Tooltip tip="Authorization Server : serveur centralisé qui contribue à définir les limites de l’accès d’un utilisateur. Par exemple, votre serveur d’autorisation peut contrôler les données, les tâches et les fonctionnalités accessibles à un utilisateur." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Authorization+Server">Authorization Server</Tooltip>) est généralement responsable de l’authentification et du consentement, ainsi que d’une partie ou de la totalité de l’autorisation et de l’application des politiques. De plus, une <Tooltip tip="Votre logiciel qui s’appuie sur Auth0 pour l’authentification et la gestion des identités." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Application">Application</Tooltip> ou l’API elle-même constitue presque toujours le principal point d’application des politiques, surtout lorsqu’un accès contextuel est requis :

* **Authentification** : le processus qui consiste à déterminer si un principal (un utilisateur ou une application) est bien celui ou celle qu’il prétend être.
* **Autorisation** : le processus qui consiste à déterminer ce qui est autorisé, en fonction du principal, des permissions qui lui ont été accordées et/ou de l’ensemble des critères d’accès contextuels applicables.
* **Consentement** : les permissions que l’utilisateur (<Tooltip tip="Resource Owner : entité (comme un utilisateur ou une application) capable d’accorder l’accès à une ressource protégée." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Resource+Owner">Resource Owner</Tooltip>) a accordées à une application pour agir en son nom. Il s’agit généralement d’une exigence liée à l’autorisation déléguée. L’utilisateur doit autoriser le client à accéder à ses données dans un autre système.
* **Application des politiques** : le fait d’appliquer les politiques de l’application ou de l’API, en refusant ou en autorisant l’accès selon les renseignements d’authentification et/ou d’autorisation d’un utilisateur.

En général, nous regroupons les différents types de contrôle d’accès en trois catégories distinctes afin de mieux comprendre a) quel acteur est responsable du stockage de l’information, b) quel acteur est responsable de la prise de décision, et c) quel acteur est responsable de l’application des restrictions.

* La première catégorie est celle où l’accès à une application ou à une API dans son ensemble est accordé ou refusé. Les données nécessaires à cette application et le processus d’application lui-même sont généralement définis dans le contexte de l’Authorization Server. Par exemple, en utilisant [`app_metadata`](/docs/fr-CA/manage-users/user-accounts/metadata) associé à un utilisateur et une [Action](/docs/fr-CA/customize/actions) définie dans votre tenant Auth0.
* La deuxième catégorie est celle où l’accès à un sous-ensemble précis des fonctionnalités d’une application ou d’une API est accordé ou refusé. Les données nécessaires à cette application sont généralement stockées dans l’Authorization Server. Par exemple, en utilisant `app_metadata` pour un utilisateur dans votre tenant Auth0, tandis que le processus d’application est exécuté dans l’application ou l’API elle-même. Dans ce scénario, les données sont généralement transmises sous la forme d’un ou de plusieurs claims personnalisés dans un jeton [`id`](/docs/fr-CA/secure/tokens/id-tokens) ou [`access`](/docs/fr-CA/secure/tokens/access-tokens).
* La troisième catégorie est celle où l’accès est accordé ou refusé selon ce sur quoi le principal (subject) peut agir dans le contexte d’une application ou d’une API. Les données nécessaires à cette application et le processus d’application lui-même sont généralement définis dans le contexte de l’application ou de l’API. Dans ce scénario, les données transmises sous la forme d’un ou de plusieurs claims personnalisés dans un jeton `id` ou `access` peuvent être utilisées avec ou sans données provenant d’une source externe autre qu’Auth0.

De plus, les mécanismes de contrôle d’accès basé sur les rôles (RBAC) et de contrôle d’accès basé sur les attributs (ABAC) peuvent être appliqués à n’importe laquelle des catégories de contrôle d’accès décrites ci-dessus. Quel que soit votre cas d’utilisation, il y a donc un certain nombre d’éléments à prendre en compte lorsque vous évaluez les fonctionnalités et le workflow dont vous avez besoin :

* Existe-t-il des scénarios dans lesquels l’accès à une application ou à une API entière devrait être refusé ?
* Offrirez-vous des API auxquelles des applications tierces pourront accéder ?
* Vos API seront-elles aussi accessibles à vos propres applications (first-party) ?
* Votre application appellera-t-elle une API tierce ?
* Vos applications et/ou API devraient-elles appliquer le contrôle d’accès en fonction des claims utilisateur ?

Auth0 prend en charge la restriction d’accès aux applications ou aux API selon certaines conditions. Dans certains cas, vous voudrez peut-être créer une Action qui refuse l’accès à l’aide de `api.access.deny()` lorsque, par exemple, un utilisateur tente d’accéder à une application ou à une API à un moment non autorisé (comme décrit dans cet [exemple](/docs/fr-CA/manage-users/access-control/sample-use-cases-actions-with-authorization#allow-access-only-on-weekdays-for-a-specific-application)) - ou si l’utilisateur n’a pas le ou les bons <Tooltip tip="Attribut encapsulé dans un jeton de sécurité qui représente une revendication formulée par le fournisseur du jeton à propos d’une entité." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Claim">claim</Tooltip> dans son `app_metadata`. Pour une application qui utilise [OpenID Connect (OIDC)](/docs/fr-CA/authenticate/protocols/openid-connect-protocol), cela empêcherait l’émission du [ID Token](/docs/fr-CA/secure/tokens/id-tokens) utilisé pour autoriser l’accès. De même, pour une API, l’émission de tout jeton OAuth2 [jeton d’accès](/docs/fr-CA/secure/tokens/access-tokens) (utilisé lors de l’appel de l’API) pourrait être bloquée, comme décrit dans cet [exemple](/docs/fr-CA/manage-users/access-control/sample-use-cases-actions-with-authorization#deny-access-to-anyone-calling-an-api).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  De manière générale, nous avons constaté que [OIDC](/docs/fr-CA/authenticate/protocols/openid-connect-protocol) est le protocole normalisé de l’industrie le plus souvent utilisé par les clients d’Auth0 pour l’authentication dans leurs applications. Nous avons aussi constaté que, même si [OAuth2](/docs/fr-CA/authenticate/protocols/oauth) a été créé comme protocole de délégation, il est souvent utilisé dans des applications de première partie lorsqu’une API ne partage pas de session avec l’application.
</Callout>

Auth0 peut également fournir les renseignements nécessaires pour qu’une application puisse appliquer des restrictions. Pour [l’intégration au niveau de l’application](#application-integration), Auth0 vous permet d’ajouter des [custom claims](#id-token-claims) à un <Tooltip tip="ID Token : information d’identification destinée au client lui-même, plutôt qu’à l’accès à une ressource." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=ID+Token">ID Token</Tooltip>, que votre application peut ensuite vérifier et utiliser pour appliquer ses politiques. Dans ce cas, vous devrez déterminer de quels renseignements votre application a besoin pour prendre des décisions d’application. Si vous devez prendre des décisions au niveau d’une API plutôt que dans votre application, vous devrez probablement utiliser un <Tooltip tip="Access Token : information d’autorisation, sous la forme d’une chaîne opaque ou d’un JWT, utilisée pour accéder à une API." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Access+Token">jeton d’accès</Tooltip> plutôt qu’un ID token. Poursuivez votre lecture pour en savoir plus.

<Warning>
  Lorsque vous décidez quelles données inclure dans votre ID token et/ou votre access token, tenez compte de la taille du token, surtout si vous le transmettez dans l’URL. Même si vous ne transmettez pas de tokens dans l’URL, vous devrez aussi tenir compte du risque d’exposer des PII (renseignements personnels identifiables) sensibles. Les renseignements contenus dans le token ne sont pas chiffrés; ainsi, même si la divulgation d’un ID token ne pose généralement pas de problème de sécurité, elle peut devenir un enjeu de confidentialité selon les données incluses dans le token.
</Warning>

Pour [l’intégration au niveau de l’API](#api-integration), Auth0 prend en charge à la fois les [custom claims](#access-token-claims) et la reconfiguration des [portée](#access-token-scopes), les deux dans le contexte d’un jeton d’accès. Encore une fois, vous devrez déterminer quels renseignements seront nécessaires pour que votre API puisse prendre des décisions d’accès, et votre API devra les appliquer en validant le contenu du jeton d’accès.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Lorsque vous décidez si vous devez utiliser des permissions au moyen de custom claims ou de portées, assurez-vous de bien comprendre la nature et l’objectif des portées. Il existe un excellent [article de blogue](https://auth0.com/blog/on-the-nature-of-oauth2-scopes/) à ce sujet, facile à lire et utile pour clarifier la question.
</Callout>

<div id="application-integration">
  ## Intégration de l’application
</div>

Dans ce scénario, votre tenant Auth0 fournit un jeton servant d’indicateur d’accès autorisé à une application. Pour les applications qui utilisent [OpenID Connect (OIDC)](/docs/fr-CA/authenticate/protocols/openid-connect-protocol), le protocole normalisé de l’industrie généralement le plus utilisé pour les applications destinées aux clients, il s’agirait d’un ID Token présenté sous la forme d’un [JWT](/docs/fr-CA/secure/tokens/json-web-tokens).

<div id="id-token-claims">
  ### Claims de l’ID Token
</div>

Grâce à l’extensibilité Action, Auth0 vous permet facilement [d’ajouter des claims personnalisées à un ID Token](/docs/fr-CA/get-started/apis/scopes/sample-use-cases-scopes-and-claims#add-custom-claims-to-a-token) en fonction, par exemple, du contenu des [métadonnées](/docs/fr-CA/manage-users/user-accounts/metadata) d’un utilisateur. Votre application peut ensuite vérifier que l’ID Token contient les claims nécessaires, puis autoriser ou refuser l’accès à certaines fonctionnalités, selon le cas. Notez que, même si le processus d’ajout de claims personnalisées au moyen d’Action est simplifié, l’environnement d’exécution d’Action est flexible et vous permet d’écrire du code personnalisé qui peut avoir des effets négatifs.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Si vous envisagez d’ajouter des claims personnalisées, nous vous recommandons de stocker dans le [`app_metadata`](/docs/fr-CA/manage-users/user-accounts/metadata) de l’utilisateur toute donnée de contrôle d’accès que vous pourriez devoir inclure dans les claims. D’abord, cela vous évite d’avoir à appeler une API externe pour récupérer ces données, ce qui peut nuire aux performances et à l’évolutivité du processus de connexion. Ensuite, `app_metadata` **ne peut pas** être modifié par un utilisateur; celui-ci ne peut donc pas contourner directement les restrictions de contrôle d’accès en modifiant ses propres <Tooltip tip="Renseignements que les utilisateurs peuvent mettre à jour, comme leurs préférences ou les paramètres de leur profil." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Metadata">métadonnées</Tooltip>. N’oubliez pas non plus de consulter nos [pratiques exemplaires relatives aux métadonnées](/docs/fr-CA/get-started/architecture-scenarios/business-to-consumer/profile-management#metadata).
</Callout>

<div id="id-token-scopes">
  ### Portées du ID Token
</div>

Les [portées OIDC](/docs/fr-CA/get-started/apis/scopes/openid-connect-scopes) sont généralement utilisées par une application pour obtenir le consentement permettant d’accéder aux renseignements d’un utilisateur pendant l’authentification. Chacune des portées prédéfinies renvoie l’ensemble des claims standard lorsqu’elles sont définies, comme indiqué dans la [spécification OIDC](https://openid.net/specs/openid-connect-core-1_0.html#StandardClaims). Les portées qu’une application demande dépendent des attributs utilisateur dont elle a besoin. Une fois que l’utilisateur a autorisé les portées demandées, les claims sont renvoyés dans le ID Token et sont également accessibles par l’entremise du point de terminaison [/userinfo](https://auth0.com/docs/api/authentication#get-user-info).

<div id="api-integration">
  ## Intégration d’API
</div>

Dans ce scénario, votre tenant Auth0 peut fournir un jeton d’accès OAuth2 ([Access Token](/docs/fr-CA/secure/tokens/access-tokens)), généralement présenté sous la forme d’un [JWT](/docs/fr-CA/secure/tokens/json-web-tokens), que votre API peut utiliser pour restreindre l’accès à certaines entités. De plus, Auth0 prend en charge ce qu’on appelle généralement les [applications de première partie et applications tierces](/docs/fr-CA/get-started/applications/first-party-and-third-party-applications).

En agissant comme serveur d’autorisation, et avec le consentement de l’utilisateur (le propriétaire de la ressource), votre tenant Auth0 peut fournir un jeton d’accès — généralement présenté sous la forme d’un [JWT](/docs/fr-CA/secure/tokens/json-web-tokens) — à une application (client) afin qu’elle puisse accéder à une ressource protégée hébergée par un <Tooltip tip="Serveur de ressources : serveur qui héberge des ressources protégées. Les serveurs de ressources acceptent les requêtes de ressources protégées et y répondent." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=resource+server">serveur de ressources</Tooltip> au nom du propriétaire de la ressource. Le jeton d’accès émis est généralement transmis comme jeton Bearer dans l’en-tête HTTP Authorization envoyé à une API.

Que vous ayez une seule API ou peut-être un ensemble d’[API de microservices](/docs/fr-CA/get-started/apis/set-logical-api) liées logiquement, vous pouvez tirer parti des jetons d’accès fournis par Auth0 pour sécuriser l’accès à vos services. Bien qu’il soit relativement simple de configurer cela dans l’[Auth0 Dashboard](https://manage.auth0.com/) ou au moyen de l’[Auth0 Management API](https://auth0.com/docs/api/management/v2#!/Resource_Servers/post_resource_servers), il est important d’examiner les différents scénarios d’application et les structures d’API afin de déterminer l’architecture la mieux adaptée à votre système.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Les jetons d’accès OAuth2 sont principalement conçus pour sécuriser des API exposées au public; lorsqu’il est présenté sous la forme d’un JWT, un jeton d’accès est une entité autonome qui peut être vérifiée sans avoir à effectuer d’appel API supplémentaire à un tiers. Si vos API n’entrent pas dans cette catégorie — c.-à-d. si elles font partie de l’application elle-même (autrement dit, si elles ne sont appelées que par cette application) ou si elles se trouvent derrière votre pare-feu — alors les protéger au moyen de jetons pourrait être excessif, et votre flux de travail actuel fondé sur des cookies (entre autres) pourrait très bien suffire.
</Callout>

OAuth2 a été conçu précisément pour l’accès par des tiers; par exemple, un utilisateur (propriétaire de la ressource) peut vouloir utiliser une application (un client) qui n’appartient pas à la même organisation que le service qui fournit ses données (le serveur de ressources). Dans ce cas, lorsque l’application doit accéder à des données appartenant à l’utilisateur, elle redirige vers l’organisation où se trouvent les données de l’utilisateur; cette organisation authentifie alors l’utilisateur, puis lui demande d’autoriser l’application à accéder à ses données. Cette demande d’autorisation est appelée [consentement](/docs/fr-CA/get-started/applications/third-party-applications/user-consent-and-third-party-applications) et constitue une grande part de ce qu’implique la prise en charge des [applications tierces](/docs/fr-CA/get-started/apis/scopes/api-scopes#example-an-api-called-by-a-third-party-application). Si vous prévoyez intégrer des applications tierces, il est important de les marquer comme telles dès le départ afin qu’Auth0 gère la demande de consentement de l’utilisateur.

D’un autre côté, si votre organisation possède les applications, les données utilisateur elles-mêmes et les API par lesquelles ces données sont accessibles, le consentement n’est généralement pas requis puisque les interactions sont toutes de [première partie](/docs/fr-CA/get-started/apis/scopes/api-scopes#example-an-api-called-by-a-first-party-application). Si vous créez uniquement des applications de première partie, vous pouvez vous assurer de ne présenter à vos utilisateurs aucun écran de consentement inutile en [permettant d’ignorer le consentement de l’utilisateur](/docs/fr-CA/get-started/apis) dans le cadre de toute définition de serveur de ressources.

<Warning>
  Bien que vous puissiez configurer vos applications comme étant de première partie, puis configurer vos API pour permettre aux clients de première partie d’ignorer le consentement, si vous utilisez `localhost`, Auth0 ne peut pas vérifier que l’application est réellement une application de première partie; vos utilisateurs seront donc quand même invités à donner leur consentement. Pour contourner cette contrainte, lors des tests sur votre machine locale pendant le développement, créez un [faux nom d’hôte local et utilisez-le à la place](https://community.auth0.com/t/how-do-i-skip-the-consent-page-for-my-api-authorization-flow/6035).
</Warning>

Sinon, il se peut que vous ayez des données concernant un utilisateur pour lesquelles des [fonctionnalités supplémentaires sont offertes](/docs/fr-CA/get-started/apis/scopes/api-scopes#example-an-api-called-by-a-back-end-service) et pour lesquelles il est impossible d’obtenir le consentement explicite de l’utilisateur (c.-à-d. qu’aucun utilisateur authentifié ne peut le fournir). Dans ce scénario, une [liste d’applications pour lesquelles le Client Credentials grant est activé](/docs/fr-CA/get-started/authentication-and-authorization-flow/client-credentials-flow) peut être définie.

<div id="access-token-claims">
  ### Claims des jetons d’accès
</div>

Comme pour les ID tokens, vous pouvez [ajouter des claims personnalisées aux jetons d’accès](/docs/fr-CA/get-started/apis/scopes/sample-use-cases-scopes-and-claims#add-custom-claims-to-a-token) à l’aide de l’extensibilité Auth0 Action. Une fois ajoutées, votre API peut vérifier qu’un jeton d’accès contient les claims requises, puis autoriser ou refuser l’accès à certaines fonctionnalités, selon les besoins.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Si vous envisagez d’ajouter des claims personnalisées, nous vous recommandons de stocker dans le [`app_metadata`](/docs/fr-CA/manage-users/user-accounts/metadata) de l’utilisateur toutes les données de contrôle d’accès que vous pourriez devoir inclure dans des claims. D’abord, cela vous évite d’avoir à appeler une API externe pour récupérer ces données, ce qui peut nuire aux performances et à l’évolutivité. Ensuite, `app_metadata` **ne peut pas** être modifié par un utilisateur; celui-ci ne peut donc pas contourner directement les restrictions de contrôle d’accès en modifiant ses propres métadonnées. N’oubliez pas non plus de consulter nos [pratiques exemplaires relatives aux métadonnées](/docs/fr-CA/get-started/architecture-scenarios/business-to-consumer/profile-management#metadata).
</Callout>

<div id="access-token-scopes">
  ### Portées du jeton d’accès
</div>

Les [portées OAuth2](/docs/fr-CA/get-started/apis/scopes/api-scopes) servent généralement de mécanisme permettant à une API de déterminer quelles <Tooltip tip="Fonctions sécurisées, propres au tenant et versionnées, écrites en Node." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Actions">actions</Tooltip> peuvent être exécutées au nom d’un utilisateur. Des portées peuvent être ajoutées pour chaque API afin de [définir des permissions d’accès précises](/docs/fr-CA/get-started/apis/add-api-permissions) dans le <Tooltip tip="Auth0 Dashboard : le produit principal d’Auth0 pour configurer vos services." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip> ou au moyen de l’Auth0 <Tooltip tip="Auth0 Dashboard : le produit principal d’Auth0 pour configurer vos services." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Management+API">Management API</Tooltip>. Les portées peuvent aussi être modifiées au moyen de l’extensibilité d’Auth0 (par ex., via une Action, comme dans cet [exemple](/docs/fr-CA/get-started/architecture-scenarios/spa-api/part-2#create-an-action-to-validate-token-scopes)). Les portées qu’une application demande pour accéder à une API devraient dépendre des fonctionnalités pour lesquelles l’utilisateur doit autoriser l’application à les utiliser. Une fois les portées demandées autorisées, elles sont renvoyées dans le jeton d’accès, qui peut ensuite être [validé par l’API en question](/docs/fr-CA/secure/tokens/access-tokens/validate-access-tokens). Un bon exemple est celui où vous ouvrez une session dans une application qui utilise un fournisseur social pour la connexion : l’API du fournisseur social exige que l’application précise si l’utilisateur veut lui permettre de publier du contenu en son nom. L’utilisateur peut ainsi accepter ou rejeter cette requête. Cet exemple montre comment l’utilisateur délègue une permission à l’application — ce qui diffère d’une API qui restreint l’accès en fonction du <Tooltip tip="Aspect de l’identité d’un utilisateur qui lui est attribué pour indiquer le niveau d’accès qu’il devrait avoir au système." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Role">rôle</Tooltip> d’un utilisateur — et cela doit être géré différemment.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  **Bonne pratique**

  Même si vous pouvez entièrement manipuler les portées du jeton d’accès au moyen de l’extensibilité d’Auth0, par bonne pratique de sécurité, vous devriez uniquement supprimer les portées qui ne sont pas autorisées et éviter d’ajouter des portées qui n’ont pas été demandées.
</Callout>

Bien que les portées soient souvent utilisées pour appliquer des permissions d’accès à un utilisateur, il existe des situations où cela peut devenir [délicat lorsqu’on les utilise de cette façon](https://auth0.com/blog/on-the-nature-of-oauth2-scopes/). Nous vous recommandons donc d’utiliser les portées pour l’usage auquel elles sont destinées (c.-à-d. déléguer une permission à une application) et d’utiliser des [claims personnalisées](#access-token-claims) pour vos scénarios de contrôle d’accès fondé sur les rôles ou autres.

<div id="fine-grained-authorization-fga">
  ## Autorisation granulaire (FGA)
</div>

[L’autorisation granulaire](https://docs.fga.dev/fga-concepts) vous permet d’accorder à des utilisateurs individuels l’accès à une ressource ou à un objet précis en fonction de :

* du rôle d’un utilisateur au sein d’une organisation, comme `editor` ou `admin`
* d’un attribut de l’utilisateur ou de l’objet, comme `manager` pour un utilisateur ou `marketing` pour un objet
* d’une relation entre un utilisateur et un objet; par exemple, un utilisateur qui a un accès de consultation à un dossier parent a aussi un accès de consultation au dossier enfant

Avec <Tooltip tip="Autorisation granulaire (FGA) : produit Auth0 permettant à des utilisateurs individuels d’accéder à des objets ou à des ressources précis." cta="Consulter le glossaire" href="/docs/fr-CA/glossary?term=FGA">FGA</Tooltip>, vous pouvez créer un modèle d’autorisation pour définir les relations à utiliser afin de déterminer l’accès des utilisateurs.

<div id="role-based-access-control-rbac">
  ## Contrôle d’accès basé sur les rôles (RBAC)
</div>

Auth0 prend en charge nativement le contrôle d’accès basé sur les rôles ([RBAC](/docs/fr-CA/manage-users/access-control/rbac)). Le RBAC consiste à attribuer des permissions aux utilisateurs en fonction de leur rôle au sein d’une organisation et simplifie le contrôle d’accès grâce à une approche plus facile à gérer et moins sujette aux erreurs.

<div id="machine-to-machine-m2m-authorization">
  ## Autorisation machine-à-machine (M2M)
</div>

De nombreux scénarios exigent qu’une application, sans session utilisateur interactive, obtienne un jeton d’accès afin d’appeler une API. Dans de tels scénarios, vous devez authentifier le client plutôt que l’utilisateur, et <Tooltip tip="OAuth 2.0 : cadre d’autorisation qui définit les protocoles et les flux d’autorisation." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=OAuth">OAuth</Tooltip> 2 fournit le type d’octroi [client credentials](/docs/fr-CA/get-started/authentication-and-authorization-flow/client-credentials-flow), ce qui facilite les choses. Voici quelques exemples courants où cela est nécessaire :

* Une tâche cron ou un autre service qui doit communiquer avec votre API (p. ex. lorsqu’un rapport quotidien doit être généré et envoyé par courriel à un administrateur).
* Une API distincte qui prend en charge un accès privilégié (p. ex. l’API n’est pas exposée directement aux utilisateurs, mais uniquement au backend).
* Dans certaines architectures de microservices, où certaines couches d’API doivent communiquer avec d’autres couches d’API sans intervention d’un utilisateur, ou après l’expiration d’un jeton utilisateur.
* Une API privilégiée qui peut devoir être appelée avant qu’un utilisateur soit authentifié (c.-à-d. à partir d’une action ou d’un script de base de données personnalisé dans votre tenant Auth0)

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Traditionnellement, un « compte de service » spécial était créé pour répondre à ces scénarios : un utilisateur avec un nom d’utilisateur et un mot de passe configurés pour des services prenant en charge des cas d’utilisation non interactifs. Cette approche n’est plus recommandée pour de nombreuses raisons, et la pratique exemplaire actuelle consiste à utiliser [OAuth 2.0 Client Credentials Grant](/docs/fr-CA/get-started/authentication-and-authorization-flow/client-credentials-flow) dans ces situations.
</Callout>
