Commençons par prendre un peu de recul et par parler du contrôle d’accès. Il n’existe pas de définition unique et universelle du contrôle d’accès dans l’industrie, mais si vous prenez le temps de faire quelques recherches et lectures, vous verrez que la plupart des sources faisant autorité s’entendent pour dire qu’il s’agit du concept global qui regroupe l’authentification, l’autorisation, le consentement et l’application des politiques, afin de garantir que seules les bonnes personnes et les bons services ont accès à vos applications et API. Examinons maintenant de plus près les distinctions entre l’authentification, l’autorisation, le consentement et l’application des politiques. Votre tenant Auth0 (votre <Tooltip tip="Groupe d’utilisateurs logiquement isolé qui partagent un accès commun avec des privilèges précis à une seule instance logicielle." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Tenant">tenant</Tooltip> <Tooltip tip="Authorization Server : serveur centralisé qui contribue à définir les limites de l’accès d’un utilisateur. Par exemple, votre serveur d’autorisation peut contrôler les données, les tâches et les fonctionnalités accessibles à un utilisateur." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Authorization+Server">Authorization Server</Tooltip>) est généralement responsable de l’authentification et du consentement, ainsi que d’une partie ou de la totalité de l’autorisation et de l’application des politiques. De plus, une <Tooltip tip="Votre logiciel qui s’appuie sur Auth0 pour l’authentification et la gestion des identités." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Application">application</Tooltip> ou une API est presque toujours le principal mécanisme d’application des politiques, surtout lorsqu’un accès contextuel est requis :

* **Authentification** : le processus qui consiste à déterminer si un principal (un utilisateur ou une application) est bien qui ou ce qu’il prétend être.
* **Autorisation** : le processus qui consiste à déterminer ce qui est autorisé, en fonction du principal, des permissions qui lui ont été accordées et/ou d’un ensemble de critères d’accès contextuels précis.
* **Consentement** : les permissions que l’utilisateur (<Tooltip tip="Resource Owner : entité (comme un utilisateur ou une application) capable d’accorder l’accès à une ressource protégée." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Resource+Owner">Resource Owner</Tooltip>) a accordées à une application pour agir en son nom. Il s’agit généralement d’une exigence de l’autorisation déléguée. L’utilisateur doit donner au client la permission d’accéder à ses données dans un autre système.
* **Application des politiques** : l’acte d’appliquer les politiques de l’application ou de l’API, en rejetant ou en autorisant l’accès selon les renseignements d’authentification et/ou d’autorisation d’un utilisateur.

De façon générale, nous regroupons les différents types de contrôle d’accès en trois catégories distinctes afin de mieux comprendre a) quel acteur est responsable de stocker l’information, b) quel acteur est responsable de prendre les décisions, et c) quel acteur est responsable d’appliquer les restrictions.

* La première catégorie correspond à la situation où l’accès à une application ou à une API est accordé ou refusé dans son ensemble. Les données requises pour appliquer ce contrôle, de même que le processus d’application, sont généralement définis dans le contexte du Authorization Server. Par exemple, en utilisant [`app_metadata`](/docs/fr-CA/manage-users/user-accounts/metadata) associé à un utilisateur et une [Action](/docs/fr-CA/customize/actions) définie dans votre tenant Auth0.
* La deuxième catégorie correspond à la situation où l’accès à un sous-ensemble précis des fonctionnalités d’une application ou d’une API est accordé ou refusé. Les données requises pour appliquer ce contrôle sont généralement stockées dans le Authorization Server. Par exemple, en utilisant `app_metadata` sur un utilisateur dans votre tenant Auth0, tandis que le processus d’application est effectué dans l’application ou l’API elle-même. Dans ce scénario, les données sont généralement transmises sous forme d’un ou de plusieurs claims personnalisés dans un jeton [`id`](/docs/fr-CA/secure/tokens/id-tokens) ou [`access`](/docs/fr-CA/secure/tokens/access-tokens).
* La troisième catégorie correspond à la situation où l’accès est accordé ou refusé selon ce sur quoi le principal (subject) peut agir dans le contexte d’une application ou d’une API. Les données requises pour appliquer ce contrôle, de même que le processus d’application, sont généralement définis dans le contexte de l’application ou de l’API. Dans ce scénario, les données transmises sous forme d’un ou de plusieurs claims personnalisés dans un jeton `id` ou `access` peuvent être utilisées avec ou sans données provenant d’une source externe autre qu’Auth0.

De plus, les mécanismes de contrôle d’accès basé sur les rôles (RBAC) et de contrôle d’accès basé sur les attributs (ABAC) peuvent être appliqués dans n’importe laquelle des catégories de contrôle d’accès décrites ci-dessus. Quel que soit votre cas d’utilisation, il y a donc un certain nombre d’éléments à prendre en compte lorsque vous évaluez les fonctionnalités et le flux de travail dont vous avez besoin :

* Existe-t-il des scénarios où l’accès à une application entière ou à une API devrait être rejeté ?
* Allez-vous fournir des API auxquelles des applications tierces pourront accéder ?
* Vos API seront-elles aussi accessibles par vos propres applications (propriétaires) ?
* Votre application appellera-t-elle une API tierce ?
* Vos applications et/ou API devraient-elles appliquer le contrôle d’accès en fonction des claims des utilisateurs ?

Auth0 prend en charge la restriction de l’accès aux applications ou aux API selon certaines conditions. Dans certains scénarios, vous pourriez vouloir créer une Action qui refuse l’accès au moyen de `api.access.deny()` lorsque, par exemple, un utilisateur tente d’accéder à une application ou à une API au mauvais moment (comme décrit dans cet [example](/docs/fr-CA/manage-users/access-control/sample-use-cases-actions-with-authorization#allow-access-only-on-weekdays-for-a-specific-application)) - ou si l’utilisateur n’a pas le ou les bons <Tooltip tip="Attribut inclus dans un jeton de sécurité qui représente un claim formulé par le fournisseur du jeton au sujet d’une entité." cta="View Glossary" href="/docs/fr-CA/glossary?term=Claim">claim</Tooltip> dans son `app_metadata`. Pour une application qui utilise [OpenID Connect (OIDC)](/docs/fr-CA/authenticate/protocols/openid-connect-protocol), cela empêcherait l’émission du [ID Token](/docs/fr-CA/secure/tokens/id-tokens) utilisé pour autoriser l’accès. De même, pour une API, l’émission de tout OAuth2 [jeton d’accès](/docs/fr-CA/secure/tokens/access-tokens) (utilisé lors de l’appel de l’API) pourrait être empêchée, comme décrit dans cet [example](/docs/fr-CA/manage-users/access-control/sample-use-cases-actions-with-authorization#deny-access-to-anyone-calling-an-api).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  De façon générale, nous avons constaté que [OIDC](/docs/fr-CA/authenticate/protocols/openid-connect-protocol) est le protocole standard de l’industrie le plus couramment utilisé par les clients d’Auth0 en matière d’authentification dans leurs applications. Nous avons aussi constaté que, même si [OAuth2](/docs/fr-CA/authenticate/protocols/oauth) a été créé comme protocole de Delegation, il est couramment utilisé dans des applications propriétaires lorsqu’il existe une API qui ne partage pas de session avec l’application.
</Callout>

Auth0 peut également fournir l’information nécessaire pour qu’une application puisse appliquer des restrictions. Pour l’[intégration au niveau de l’application](#application-integration), Auth0 vous permet d’ajouter des [custom claims](#id-token-claims) à un <Tooltip tip="ID Token : justificatif destiné au client lui-même, plutôt qu’à l’accès à une ressource." cta="View Glossary" href="/docs/fr-CA/glossary?term=ID+Token">ID Token</Tooltip>, que votre application peut ensuite vérifier et utiliser pour appliquer des politiques. Dans ce cas, vous devrez déterminer de quelle information votre application a besoin pour prendre des décisions d’application. Si vous devez prendre des décisions au niveau d’une API plutôt que dans votre application, vous devrez probablement utiliser un <Tooltip tip="jeton d’accès : justificatif d’autorisation, sous la forme d’une chaîne opaque ou d’un JWT, utilisé pour accéder à une API." cta="View Glossary" href="/docs/fr-CA/glossary?term=Access+Token">jeton d’accès</Tooltip> au lieu d’un ID token. Poursuivez votre lecture pour en savoir plus.

<Warning>
  Lorsque vous décidez quelles données inclure dans votre ID token et/ou access token, tenez compte de la taille du token, surtout si vous le transmettez dans l’URL. Même si vous ne transmettez pas les tokens dans l’URL, vous devrez aussi tenir compte du risque d’exposer des PII sensibles (renseignements personnels identifiables). L’information du token n’est pas encrypted, donc même si ce n’est généralement pas un enjeu de sécurité qu’un ID token soit divulgué, cela peut devenir un enjeu de confidentialité selon les données incluses dans le token.
</Warning>

Pour l’[intégration au niveau de l’API](#api-integration), Auth0 prend en charge à la fois les [custom claims](#access-token-claims) et la reconfiguration des [scope](#access-token-scopes), les deux dans le contexte d’un jeton d’accès. Encore une fois, vous devrez déterminer quelle information est requise pour que votre API puisse prendre des décisions d’accès, et votre API devra les appliquer en validant le contenu de l’jeton d’accès.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Lorsque vous décidez si vous devriez utiliser des permissions au moyen de custom claims ou de scopes, vous devez vous assurer de bien comprendre la nature et l’objectif des scopes. Il y a un excellent [article de blogue](https://auth0.com/blog/on-the-nature-of-oauth2-scopes/) à ce sujet, facile à lire et qui aide à clarifier la question.
</Callout>

<div id="application-integration">
  ## Intégration de l’application
</div>

Dans ce scénario, votre tenant Auth0 fournit un jeton servant d’indicateur d’accès autorisé à une application. Pour les applications qui utilisent [OpenID Connect (OIDC)](/docs/fr-CA/authenticate/protocols/openid-connect-protocol), le protocole standard de l’industrie que l’on retrouve le plus souvent dans les applications destinées aux clients, il s’agirait d’un ID Token présenté sous forme de [JWT](/docs/fr-CA/secure/tokens/json-web-tokens).

<div id="id-token-claims">
  ### Claims de l’ID Token
</div>

Grâce à l’extensibilité Action, Auth0 vous permet d’[ajouter facilement des claims personnalisées à un ID Token](/docs/fr-CA/get-started/apis/scopes/sample-use-cases-scopes-and-claims#add-custom-claims-to-a-token) en fonction, par exemple, du contenu des [métadonnées](/docs/fr-CA/manage-users/user-accounts/metadata) d’un utilisateur. Votre application peut ensuite vérifier que l’ID Token contient les claims requises, puis autoriser ou refuser l’accès à certaines fonctionnalités, selon les besoins. Notez que, même si le processus d’ajout de claims personnalisées au moyen d’une Action est simplifié, le runtime d’Action est flexible et vous permet d’écrire du code personnalisé qui peut avoir des effets négatifs.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Si vous envisagez d’ajouter des claims personnalisées, nous vous recommandons de stocker dans l’[`app_metadata`](/docs/fr-CA/manage-users/user-accounts/metadata) de l’utilisateur toutes les données de contrôle d’accès que vous pourriez devoir inclure dans les claims. D’abord, cela vous évite d’avoir à appeler une API externe pour récupérer ces données, ce qui peut nuire aux performances et à l’évolutivité de la séquence de connexion. Ensuite, `app_metadata` **ne peut pas** être modifié par un utilisateur — celui-ci ne peut donc pas contourner directement les restrictions de contrôle d’accès en modifiant ses propres <Tooltip tip="Renseignements que les utilisateurs peuvent mettre à jour, comme les préférences ou les paramètres de profil." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Metadata">métadonnées</Tooltip>. N’oubliez pas non plus de consulter nos recommandations sur les [pratiques exemplaires relatives aux métadonnées](/docs/fr-CA/get-started/architecture-scenarios/business-to-consumer/profile-management#metadata).
</Callout>

<div id="id-token-scopes">
  ### Scopes de l’ID Token
</div>

Les [scopes OIDC](/docs/fr-CA/get-started/apis/scopes/openid-connect-scopes) sont généralement utilisées par une application pour obtenir le consentement nécessaire à l’accès autorisé aux renseignements d’un utilisateur pendant l’authentification. Chacune des portées prédéfinies renvoie l’ensemble des standard claims, lorsqu’elles sont définies, comme le décrit la [spécification OIDC](https://openid.net/specs/openid-connect-core-1_0.html#StandardClaims). Les scopes qu’une application demande dépendent des attributs utilisateur dont elle a besoin. Une fois les scopes demandés autorisées par l’utilisateur, les claims sont renvoyés dans l’ID Token et sont aussi accessibles par l’entremise du point de terminaison [/userinfo](https://auth0.com/docs/api/authentication#get-user-info).

<div id="api-integration">
  ## Intégration d’API
</div>

Dans ce scénario, votre tenant Auth0 peut fournir un jeton d’accès OAuth2 ([Access Token](/docs/fr-CA/secure/tokens/access-tokens)), généralement présenté sous la forme d’un [JWT](/docs/fr-CA/secure/tokens/json-web-tokens), que votre API peut utiliser pour restreindre l’accès à certaines entités. De plus, Auth0 offre du soutien pour ce qu’on appelle généralement les [applications propriétaires et les applications tierces](/docs/fr-CA/get-started/applications/first-party-and-third-party-applications).

En agissant comme serveur d’autorisation, et avec le consentement de l’utilisateur (le propriétaire de la ressource), votre tenant Auth0 peut être utilisé pour fournir un jeton d’accès — généralement présenté sous la forme d’un [JWT](/docs/fr-CA/secure/tokens/json-web-tokens) — à une application (client) afin qu’elle puisse accéder à une ressource protégée hébergée par un <Tooltip tip="Serveur de ressources : serveur hébergeant des ressources protégées. Les serveurs de ressources acceptent les requêtes vers des ressources protégées et y répondent." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=resource+server">serveur de ressources</Tooltip> au nom du propriétaire de la ressource. Le jeton d’accès émis est généralement transmis comme jeton Bearer dans l’en-tête HTTP Authorization envoyé à une API.

Que vous ayez une seule API, ou encore une suite d’[API de microservices](/docs/fr-CA/get-started/apis/set-logical-api) logiquement liées, vous pouvez tirer parti des jetons d’accès fournis par Auth0 afin de sécuriser l’accès à vos services. Bien qu’il soit relativement simple de configurer cela dans le [Auth0 Dashboard](https://manage.auth0.com/) ou par l’entremise de l’[Auth0 Management API](https://auth0.com/docs/api/management/v2#!/Resource_Servers/post_resource_servers), il est important d’examiner les différents scénarios d’application et les différentes configurations d’API afin de déterminer l’architecture la mieux adaptée à votre système.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Les jetons d’accès OAuth2 sont principalement conçus pour sécuriser des API exposées publiquement; lorsqu’il est présenté sous la forme d’un JWT, un jeton d’accès est une entité autonome qui peut être vérifiée sans qu’il soit nécessaire d’effectuer un appel d’API tiers supplémentaire. Si vos API n’entrent pas dans cette catégorie — c.-à-d. qu’elles font partie d’une application elle-même (autrement dit, qu’elles ne sont appelées que par cette application) ou qu’elles se trouvent derrière votre pare-feu — alors les protéger au moyen de jetons pourrait très bien être excessif, et votre mécanisme existant fondé sur des cookies (ou autre) pourrait très bien suffire.
</Callout>

OAuth2 a été conçu précisément en tenant compte de l’accès tiers; par exemple, un scénario possible serait qu’un utilisateur (propriétaire de la ressource) souhaite utiliser une application (un client) qui n’appartient pas à la même organisation que le service qui fournit les données de l’utilisateur (le serveur de ressources). Dans ce cas, lorsque l’application doit accéder à des données appartenant à l’utilisateur, elle redirige vers l’organisation où résident les données de l’utilisateur, laquelle authentifie alors l’utilisateur, puis lui demande d’autoriser l’application à accéder à ses données. Cette demande d’autorisation est appelée [consentement](/docs/fr-CA/get-started/applications/third-party-applications/user-consent-and-third-party-applications) et constitue une grande partie de ce qu’implique la prise en charge des [applications tierces](/docs/fr-CA/get-started/apis/scopes/api-scopes#example-an-api-called-by-a-third-party-application). Si vous prévoyez intégrer des applications tierces, il est important de les marquer comme tierces dès le départ afin qu’Auth0 gère la demande de consentement de l’utilisateur.

D’autre part, si votre organisation est propriétaire des applications, des données utilisateur elles-mêmes et des API par lesquelles ces données sont consultées, alors le consentement n’est généralement pas requis puisque les interactions sont toutes [propriétaires](/docs/fr-CA/get-started/apis/scopes/api-scopes#example-an-api-called-by-a-first-party-application). Si vous ne créez que des applications propriétaires, vous pouvez alors vous assurer de ne présenter aucun écran de consentement inutile à vos utilisateurs [en permettant d’ignorer le consentement de l’utilisateur](/docs/fr-CA/get-started/apis) dans le cadre de toute définition de serveur de ressources.

<Warning>
  Bien que vous puissiez configurer vos applications comme étant propriétaires et ensuite configurer vos API pour permettre aux clients propriétaires d’ignorer le consentement, si vous utilisez `localhost`, Auth0 ne peut pas vérifier que l’application est réellement une application propriétaire; vos utilisateurs seront donc quand même invités à donner leur consentement. Pour contourner cette contrainte, lorsque vous faites des tests sur votre machine locale pendant le développement, créez un [faux nom d’hôte local et utilisez-le à la place](https://community.auth0.com/t/how-do-i-skip-the-consent-page-for-my-api-authorization-flow/6035).
</Warning>

Sinon, vous pouvez avoir des données liées à un utilisateur pour lesquelles des [fonctionnalités supplémentaires sont offertes](/docs/fr-CA/get-started/apis/scopes/api-scopes#example-an-api-called-by-a-back-end-service) et pour lesquelles il est impossible d’obtenir le consentement explicite de l’utilisateur (c.-à-d. qu’il n’y a aucun utilisateur authentifié pouvant le donner). Dans ce cas, une [liste d’applications pour lesquelles le Client Credentials Grant est activé](/docs/fr-CA/get-started/authentication-and-authorization-flow/client-credentials-flow) peut être définie.

<div id="access-token-claims">
  ### Claims du jeton d’accès
</div>

Comme pour les ID Tokens, vous pouvez [ajouter des claims personnalisés aux jetons d’accès](/docs/fr-CA/get-started/apis/scopes/sample-use-cases-scopes-and-claims#add-custom-claims-to-a-token) à l’aide de l’extensibilité Auth0 Action. Une fois ajoutés, votre API peut vérifier qu’un jeton d’accès contient les claims requis et autoriser ou refuser l’accès à certaines fonctionnalités, selon le cas.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Si vous envisagez d’ajouter des claims personnalisés, nous vous recommandons de stocker dans le [`app_metadata`](/docs/fr-CA/manage-users/user-accounts/metadata) de l’utilisateur toute donnée de contrôle d’accès que vous pourriez devoir inclure dans des claims. D’abord, cela vous évite d’avoir à appeler une API externe pour récupérer ces données, ce qui peut nuire aux performances et à l’évolutivité. Ensuite, `app_metadata` **ne peut pas** être modifié par l’utilisateur — il ne peut donc pas contourner directement les restrictions de contrôle d’accès en modifiant ses propres métadonnées. N’oubliez pas non plus de consulter nos [pratiques exemplaires sur les métadonnées](/docs/fr-CA/get-started/architecture-scenarios/business-to-consumer/profile-management#metadata).
</Callout>

<div id="access-token-scopes">
  ### Scopes du jeton d’accès
</div>

Les [scopes OAuth2](/docs/fr-CA/get-started/apis/scopes/api-scopes) sont généralement utilisés comme mécanisme permettant à une API de déterminer quelles <Tooltip tip="Fonctions sécurisées, propres au tenant et versionnées écrites en Node." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Actions">Actions</Tooltip> peuvent être exécutées au nom d’un utilisateur. Des scopes peuvent être ajoutés pour chaque API afin de [définir des permissions d’accès précises](/docs/fr-CA/get-started/apis/add-api-permissions) dans le <Tooltip tip="Auth0 Dashboard : le principal produit d’Auth0 pour configurer vos services." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip> ou au moyen de l’Auth0 <Tooltip tip="Auth0 Dashboard : le principal produit d’Auth0 pour configurer vos services." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Management+API">Management API</Tooltip>. Les scopes peuvent aussi être modifiés au moyen de l’extensibilité d’Auth0 (par ex. au moyen d’une Action, comme dans cet [exemple](/docs/fr-CA/get-started/architecture-scenarios/spa-api/part-2#create-an-action-to-validate-token-scopes)). Les scopes qu’une application demande pour accéder à une API devraient dépendre des fonctionnalités pour lesquelles l’utilisateur doit autoriser l’application à les utiliser. Une fois les scopes demandés autorisés, ils seront renvoyés dans le jeton d’accès, qui pourra ensuite être [validé par l’API en question](/docs/fr-CA/secure/tokens/access-tokens/validate-access-tokens). Un bon exemple est lorsque vous vous connectez à une application qui utilise un fournisseur social pour la connexion : l’API du fournisseur social exige que l’application précise si l’utilisateur souhaite qu’elle publie du contenu en son nom. Cela permet à l’utilisateur d’accepter ou de rejeter cette requête. Cet exemple montre comment l’utilisateur délègue une permission à l’application, ce qui diffère d’une API qui restreint l’accès en fonction du <Tooltip tip="Aspect de l’identité d’un utilisateur attribué à l’utilisateur pour indiquer le niveau d’accès qu’il devrait avoir au système." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Role">rôle</Tooltip> de l’utilisateur, et cela devrait être géré autrement.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  **Bonne pratique**

  Même si vous pouvez manipuler entièrement les scopes du jeton d’accès au moyen de l’extensibilité d’Auth0, par bonne pratique de sécurité, vous devriez uniquement supprimer les scopes qui ne sont pas autorisés et vous abstenir d’en ajouter qui n’ont pas été demandés.
</Callout>

Bien que les scopes soient souvent utilisés comme moyen d’appliquer des permissions d’accès à un utilisateur, il existe des situations où cela peut devenir [complexe lorsqu’ils sont utilisés de cette façon](https://auth0.com/blog/on-the-nature-of-oauth2-scopes/). Nous vous recommandons donc d’utiliser les scopes selon leur objectif prévu (c.-à-d. déléguer une permission à une application) et d’utiliser des [claims personnalisés](#access-token-claims) pour vos scénarios de contrôle d’accès fondé sur les rôles ou autres.

<div id="fine-grained-authorization-fga">
  ## Autorisation fine (FGA)
</div>

[L’autorisation fine](https://docs.fga.dev/fga-concepts) vous permet d’accorder à des utilisateurs un accès à une ressource ou à un objet précis en fonction de :

* du rôle d’un utilisateur au sein d’une organisation, comme `editor` ou `admin`
* d’un attribut de l’utilisateur ou de l’objet, comme `manager` pour un utilisateur ou `marketing` pour un objet
* d’une relation entre un utilisateur et un objet; par exemple, un utilisateur qui a un droit de consultation sur un dossier parent a aussi un droit de consultation sur le dossier enfant

Avec <Tooltip tip="Autorisation fine (FGA) : produit Auth0 permettant aux utilisateurs d’accéder à des objets ou à des ressources précis." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=FGA">FGA</Tooltip>, vous pouvez créer un modèle d’autorisation pour définir la relation à utiliser afin de déterminer l’accès des utilisateurs.

<div id="role-based-access-control-rbac">
  ## Contrôle d’accès basé sur les rôles (RBAC)
</div>

Auth0 prend en charge nativement le contrôle d’accès basé sur les rôles ([RBAC](/docs/fr-CA/manage-users/access-control/rbac)). Le modèle RBAC consiste à attribuer des permissions aux utilisateurs en fonction de leur rôle au sein d’une organisation et permet de simplifier le contrôle d’accès grâce à une approche plus facile à gérer et moins sujette aux erreurs.

<div id="machine-to-machine-m2m-authorization">
  ## Autorisation machine à machine (M2M)
</div>

De nombreux scénarios exigent qu’une application sans session utilisateur interactive obtienne un jeton d’accès afin d’appeler une API. Dans ce type de scénario, vous devez authentifier le client plutôt que l’utilisateur, et <Tooltip tip="OAuth 2.0 : framework d’autorisation qui définit les protocoles et workflows d’autorisation." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=OAuth">OAuth</Tooltip> 2 fournit le type d’octroi [client credentials](/docs/fr-CA/get-started/authentication-and-authorization-flow/client-credentials-flow), ce qui facilite les choses. Voici quelques exemples courants où cela est nécessaire :

* Un cron job ou un autre service qui doit communiquer avec votre API (p. ex. lorsqu’un rapport quotidien doit être généré et envoyé par courriel à un administrateur).
* Une API distincte qui prend en charge un accès privilégié (p. ex. l’API n’est pas exposée directement aux utilisateurs, mais uniquement à un backend).
* Dans certaines architectures de microservices, où certaines couches d’API doivent communiquer avec d’autres couches d’API sans intervention d’un utilisateur, ou après l’expiration du jeton d’un utilisateur.
* Une API privilégiée qui შეიძლება devoir être appelée avant qu’un utilisateur ne se soit authentifié (c.-à-d. à partir d’une action ou d’un script DB personnalisé dans votre Auth0 tenant)

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Traditionnellement, on créait un « compte de service » spécial pour ce type de scénario : un utilisateur avec un nom d’utilisateur et mot de passe configuré pour des services prenant en charge des cas d’utilisation non interactifs. Cette approche n’est plus recommandée pour de nombreuses raisons, et la meilleure pratique actuelle consiste à utiliser [OAuth 2.0 Client Credentials Grant](/docs/fr-CA/get-started/authentication-and-authorization-flow/client-credentials-flow) dans ces situations.
</Callout>
