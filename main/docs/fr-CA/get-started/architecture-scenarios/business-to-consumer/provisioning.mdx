Il est important de déterminer tôt comment les utilisateurs s’inscriront, car les décisions prises ici influenceront bon nombre de celles que vous devrez prendre par la suite. Nous avons constaté qu’il existe un ensemble de modèles courants pour la façon dont les utilisateurs sont ajoutés à votre système, ainsi que certains éléments à prendre en compte dans la conception du workflow.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Bien qu’Auth0 prenne en charge de nombreux workflows, les workflows Web qui utilisent [Universal Login](/docs/fr-CA/authenticate/login/auth0-universal-login) d’Auth0 pour l’inscription sont considérés à la fois comme une bonne pratique du secteur et comme une bonne pratique d’Auth0, puisqu’ils offrent une fonctionnalité optimale et le meilleur niveau de sécurité.
</Callout>

Auth0 prend en charge l’inscription des utilisateurs au moyen de différents [fournisseurs d’identité](/docs/fr-CA/authenticate/identity-providers). Lors de l’inscription, Auth0 provisionne le profil utilisateur afin qu’il contienne les renseignements du compte de l’utilisateur. Il y a plusieurs éléments à considérer en ce qui concerne les fonctionnalités et le workflow :

* Devriez-vous utiliser Auth0 comme référentiel d’identités ?
* Pouvez-vous utiliser votre propre magasin d’identités hérité avec Auth0 ?
* Comment migrez-vous les identités utilisateur de votre référentiel d’identités vers Auth0 ?
* Vos utilisateurs peuvent-ils s’inscrire à l’aide de leurs comptes sociaux existants, comme Google et Facebook ?

Auth0 fournit un stockage d’identités prêt à l’emploi qui peut être mis à profit pour stocker les identifiants des utilisateurs de façon sûre et sécurisée. Consultez [Self Sign Up](#self-sign-up) pour en savoir plus. Si vous avez déjà un magasin d’identités hérité et souhaitez en déléguer la gestion, les capacités de [User Migration](#user-migration) vous offrent plusieurs options pour le faire.

Autrement, si vous devez conserver votre magasin d’identités hérité — par exemple parce que vous avez des applications que vous n’êtes pas prêt à migrer ou qui ne peuvent pas l’être — vous pouvez utiliser la capacité de [proxy de magasin d’identités](#identity-store-proxy). Permettre à vos clients d’apporter leur propre identité est aussi une option intéressante et, même si nous constatons que nos clients ne le font pas d’emblée, vous pouvez utiliser la capacité [Social Sign Up](#social-sign-up) pour l’offrir.

<div id="user-migration">
  ## Migration des utilisateurs
</div>

En plus d’héberger le [profil utilisateur](/docs/fr-CA/get-started/architecture-scenarios/business-to-consumer/profile-management), Auth0 peut aussi [agir comme proxy](#identity-store-proxy) pour votre magasin d’identités hérité et offrir une solution de remplacement sécurisée hébergée par Auth0. Ces deux capacités sont prises en charge grâce aux [Connexions de base de données](/docs/fr-CA/authenticate/database-connections) d’Auth0. Si vous décidez d’utiliser Auth0 pour remplacer votre magasin d’identités hérité, vous pouvez [migrer les utilisateurs](/docs/fr-CA/manage-users/user-migration) soit en une seule fois au moyen d’une migration en bloc, soit progressivement avec la migration automatique.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Les clients optent souvent pour une approche de migration des utilisateurs en deux étapes : d’abord la migration automatique pour migrer le plus grand nombre d’utilisateurs possible, puis la migration en bloc pour les utilisateurs restants. Consultez [Scénarios de migration des utilisateurs](/docs/fr-CA/manage-users/user-migration/user-migration-scenarios) pour en savoir plus.
</Callout>

La migration automatique est à privilégier, car elle permet de migrer les utilisateurs individuellement et de leur faire conserver leur mot de passe existant dans presque tous les cas. Pour la migration en bloc, nous recommandons d’utiliser la [Management API](https://auth0.com/docs/api/management/v2#!/Jobs/post_users_imports) plutôt que l’[extension User Import/Export](/docs/fr-CA/manage-users/user-migration/user-import-export-extension) dans tous les cas sauf les plus simples, puisque la <Tooltip tip="Management API : un produit permettant aux clients d’effectuer des tâches administratives." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Management+API">Management API</Tooltip> offre plus de souplesse et un meilleur contrôle.

Avec la migration en bloc, les utilisateurs doivent généralement **réinitialiser leur mot de passe une fois la migration terminée**, à moins que les mots de passe soient stockés sous forme hachée dans votre magasin d’identités hérité à l’aide de bcrypt (ou que vous puissiez les générer au format bcrypt). Dans ce cas, vous pourriez être en mesure d’utiliser la migration en bloc tout en **préservant les mots de passe des utilisateurs** dans le cadre du processus, selon l’algorithme bcrypt et le nombre d’itérations de salt utilisées. Consultez les [exemples de schéma d’importation en bloc pour les bases de données](/docs/fr-CA/manage-users/user-migration/bulk-user-import-schema) pour en savoir plus.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Les appels à la Management API sont assujettis à la [politique de limitation du débit d’Auth0](/docs/fr-CA/troubleshoot/customer-support/operational-policies/rate-limit-policy). Vous devez en tenir compte et, pour vous aider, Auth0 recommande généralement d’utiliser le [Auth0 SDK](/docs/fr-CA/libraries) adapté à votre environnement de développement plutôt que d’appeler nos API directement.
</Callout>

<div id="identity-store-proxy">
  ### Proxy de magasin d’identités
</div>

Les types de connexion de base de données d’Auth0 peuvent aussi être configurés pour agir comme proxy d’un magasin d’identités existant (hérité). Si vous devez conserver des identités utilisateur définies dans votre propre magasin hérité — par exemple, si vous avez une ou plusieurs applications essentielles à l’entreprise que vous ne pouvez pas migrer vers Auth0, mais qui ont quand même besoin d’accéder à ces identités — vous pouvez facilement intégrer Auth0. Consultez [Authentifier les utilisateurs à l’aide de votre base de données](/docs/fr-CA/authenticate/database-connections/custom-db) pour en savoir plus.

<div id="self-sign-up">
  ## Inscription libre-service
</div>

L’inscription libre-service s’appuie sur les [connexions de base de données](/docs/fr-CA/authenticate/database-connections) d’Auth0 pour stocker l’ID utilisateur, le mot de passe et, s’il y a lieu, le nom d’utilisateur recueillis auprès des nouveaux utilisateurs pendant le processus d’inscription. Les politiques des connexions de base de données qui régissent, par exemple, la [longueur minimale du nom d’utilisateur](/docs/fr-CA/authenticate/database-connections/require-username) ou la [force et la complexité du mot de passe](/docs/fr-CA/authenticate/database-connections/password-options) peuvent être configurées dans le <Tooltip tip="Auth0 Dashboard : l’interface principale d’Auth0 pour configurer vos services." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip>.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Le [Universal Login](/docs/fr-CA/authenticate/login/auth0-universal-login) d’Auth0, ainsi que des widgets Auth0 comme [Lock](https://auth0.com/lock), s’intègrent aux connexions de base de données afin d’offrir, dès le départ, des fonctionnalités d’interface utilisateur complètes pour l’inscription. Ces éléments d’interface sont entièrement réactifs et, grâce à une configuration riche et à de nombreuses options de personnalisation, vous pouvez déployer des fonctionnalités d’inscription libre-service ainsi que de connexion.
</Callout>

<div id="social-sign-up">
  ## Inscription sociale
</div>

L’inscription sociale est synonyme de la connexion par [authentification sociale](/docs/fr-CA/get-started/architecture-scenarios/business-to-consumer/authentication) — il n’y a pas de distinction à faire ici *à proprement parler*, puisque la création du [profil](/docs/fr-CA/get-started/architecture-scenarios/business-to-consumer/profile-management) utilisateur se fait automatiquement lors de la première connexion sociale.
