En plus d’adopter les meilleures pratiques en matière de gestion des changements et d’[AQ](/docs/fr-CA/get-started/architecture-scenarios/business-to-consumer/quality-assurance), les clients qui réussissent intègrent également la gestion des artéfacts Auth0 à un processus de déploiement automatisé. Comme il est indiqué dans la section Architecture, sous [SDLC support](/docs/fr-CA/get-started/architecture-scenarios/business-to-consumer/architecture#sdlc-support), vous devez configurer des tenants Auth0 distincts pour les environnements de développement, de test et de production, et veiller à ce que cette configuration soit presque identique d’un environnement à l’autre. L’automatisation du déploiement permet d’y parvenir en garantissant une configuration uniforme pour chaque tenant d’environnement, ce qui réduit les risques de bogues causés par des écarts de configuration entre les environnements.

<Info>
  ### Bonne pratique

  Peu importe la façon dont vous configurez l’automatisation du déploiement, nous vous recommandons d’effectuer des tests unitaires sur vos rules, custom DB scripts et hooks avant le déploiement, puis d’exécuter aussi des tests d’intégration sur votre tenant après le déploiement. Pour en savoir plus, consultez les directives sur l’[Assurance qualité](/docs/fr-CA/get-started/architecture-scenarios/business-to-consumer/quality-assurance).
</Info>

Auth0 offre du soutien pour quelques approches différentes d’automatisation du déploiement, et elles peuvent être utilisées ensemble au besoin :

* L’[outil Auth0 Deploy CLI](/docs/fr-CA/deploy-monitor/deploy-cli-tool) fournit un script facile à utiliser qui peut vous aider à l’intégrer à votre pipeline existant d’intégration continue et de déploiement continu (CI/CD).
* Si vous ne pouvez pas l’intégrer directement à un pipeline CI/CD, ou si vous n’en avez pas, les [Source Control Extensions](/docs/fr-CA/customize/extensions) d’Auth0 peuvent fournir un processus d’automatisation de base simple à configurer et nécessitant très peu de maintenance.

<Warning>
  Notez que l’outil Deploy CLI et les source control extensions peuvent tous deux entraîner des changements destructifs : les changements manuels apportés directement dans le dashboard entre les déploiements automatisés pourraient être perdus ! C’est pourquoi, si l’un ou l’autre est utilisé, **tous** les changements doivent être déployés à partir du sous-système de contrôle de code source utilisé par l’outillage, et non effectués manuellement.
</Warning>

Chaque environnement peut aussi nécessiter une configuration propre à l’environnement — par exemple, les <Tooltip tip="Client ID : valeur d’identification attribuée à votre ressource enregistrée par Auth0." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Client+ID">Client ID</Tooltip> et les <Tooltip tip="Client Secret : secret utilisé par un client (application) pour s’authentifier auprès de l’Authorization Server; il ne devrait être connu que du client et de l’Authorization Server et doit être suffisamment aléatoire pour ne pas pouvoir être deviné." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Client+Secrets">Client Secrets</Tooltip> seront différents d’un tenant Auth0 à l’autre — vous aurez donc besoin d’un moyen d’y faire référence de façon dynamique plutôt que d’utiliser des valeurs codées en dur. Auth0 offre du soutien pour gérer l’information de configuration propre à l’environnement au moyen de l’une des deux approches suivantes :

* Utiliser les [Variables propres au tenant](#tenant-specific-variables)
* Utiliser le keyword replacement si vous utilisez l’outil Auth0 Deploy CLI

<div id="tenant-specific-variables">
  ## Variables propres au tenant
</div>

Auth0 vous permet de configurer des variables accessibles depuis une [extensibilité](/docs/fr-CA/customize/extensions) personnalisée; on peut les considérer comme des variables d’environnement pour votre tenant Auth0. Au lieu d’intégrer en dur des références qui changent lorsque le code passe d’un environnement de développement, de test ou de production à un autre, vous pouvez utiliser un nom de variable configuré dans le tenant et référencé par le code d’extensibilité personnalisée. Il devient ainsi plus facile d’utiliser le même code personnalisé, sans modification, dans différents tenants, puisque le code peut référencer des variables qui seront renseignées avec des valeurs propres au tenant au moment de l’exécution :

* Pour utiliser des variables dans Actions, consultez [Write Your First Action](/docs/fr-CA/customize/actions/write-your-first-action) pour apprendre à configurer des secrets dans l’éditeur
* Pour utiliser des variables dans Rules, voyez comment [configurer des valeurs](/docs/fr-CA/customize/rules/configuration)
* Pour utiliser des variables dans Hooks, voyez comment configurer des [secrets](/docs/fr-CA/customize/hooks/hook-secrets) dans l’éditeur
* Pour utiliser des variables dans Custom DB Scripts, consultez les [paramètres de configuration](/docs/fr-CA/authenticate/database-connections/custom-db/create-db-connection#step-3-add-configuration-parameters)

<Info>
  ### Bonne pratique

  Il est recommandé d’utiliser des variables pour stocker des valeurs propres au tenant, ainsi que tout secret sensible qui ne devrait pas être exposé dans votre code personnalisé. Si votre code personnalisé est déployé dans GitHub/GitLab/Bitbucket/VSTS, l’utilisation d’une variable propre au tenant évite d’exposer des valeurs sensibles dans votre dépôt.
</Info>
