Dans ce scénario, nous créerons une API de feuilles de temps pour une entreprise fictive nommée ExampleCo. L’API permettra de gérer les entrées de feuilles de temps d’un employé ou d’un sous-traitant.

Nous créerons également une application mobile qui servira à consulter et à consigner des entrées de feuilles de temps dans la base de données centralisée des feuilles de temps au moyen de l’API.

<Info>
  ### En bref

  * Auth0 fournit l’authentication and authorization des API pour sécuriser l’accès aux points de terminaison de l’API (voir [Authentification et autorisation d’API](/docs/fr-CA/get-started/architecture-scenarios/mobile-api/part-1#api-authentication-and-authorization))
  * Pour autoriser l’utilisateur d’une application mobile et lui accorder l’accès à l’API, Auth0 prend en charge le [flux de code d’autorisation avec clé de preuve pour l’échange de code (PKCE)](/docs/fr-CA/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce) (voir [clé de preuve pour l’échange de code](/docs/fr-CA/get-started/architecture-scenarios/mobile-api/part-1#proof-key-for-code-exchange-pkce-))
  * L’application mobile et l’API doivent toutes deux être configurées dans l’Auth0 Dashboard (voir [Auth0 Configuration](/docs/fr-CA/get-started/architecture-scenarios/mobile-api/part-2))
  * Les Permissions de l’utilisateur peuvent être appliquées à l’aide de l’Authorization Extension (voir [Configurer l’Authorization Extension](/docs/fr-CA/get-started/architecture-scenarios/mobile-api/part-2#configure-the-authorization-extension))
  * L’API est sécurisée en veillant à ce qu’un <Tooltip tip="" cta="Consulter le glossaire" href="/docs/fr-CA/glossary?term=Access+Token">Access Token</Tooltip> valide soit transmis dans l’en-tête HTTP Authorization lors des appels à l’API (voir [Implémenter l’API](/docs/fr-CA/get-started/architecture-scenarios/mobile-api/part-3#secure-the-endpoints))
  * Le SDK Auth0.Android peut être utilisé pour autoriser l’utilisateur de l’application mobile et obtenir un Access Token valide qui servira à appeler l’API (voir [Autoriser l’utilisateur](/docs/fr-CA/get-started/architecture-scenarios/mobile-api/part-3#authorize-the-user))
  * L’application mobile peut récupérer les renseignements du profil de l’utilisateur en décodant l’ID Token (voir [Obtenir le profil de l’utilisateur](/docs/fr-CA/get-started/architecture-scenarios/mobile-api/part-3#get-the-user-profile))
  * Des éléments de l’interface utilisateur peuvent être affichés de façon conditionnelle selon le scope accordé à l’utilisateur (voir [Afficher des éléments de l’interface utilisateur de façon conditionnelle selon le scope](/docs/fr-CA/get-started/architecture-scenarios/mobile-api/part-3#display-ui-elements-conditionally-based-on-scope))
  * L’application mobile fournit l’Access Token dans l’en-tête HTTP Authorization lorsqu’elle effectue des appels à l’API (voir [Appeler l’API](/docs/fr-CA/get-started/architecture-scenarios/mobile-api/part-3#call-the-api))
  * L’Access Token de l’utilisateur de l’application mobile peut être renouvelé pour éviter qu’il ait à se reconnecter pendant une session (voir [Renouveler le jeton](/docs/fr-CA/get-started/architecture-scenarios/mobile-api/part-3#renew-the-token))
</Info>

<div id="the-premise">
  ## Le contexte
</div>

ExampleCo est une jeune entreprise de services-conseils. À l’heure actuelle, elle compte environ 100 employés et confie aussi plusieurs activités à des sous-traitants externes. Tous les employés et les sous-traitants externes doivent remplir leurs feuilles de temps chaque semaine.

L’entreprise a créé une application de feuilles de temps, un scénario que nous avons abordé dans [Authentification unique pour les applications Web classiques](/docs/fr-CA/get-started/architecture-scenarios/sso-for-regular-web-apps). Les employés internes utilisent cette application Web pour remplir leurs feuilles de temps, mais l’entreprise veut une application mobile que les employés et les sous-traitants pourront utiliser lorsqu’ils ne sont pas sur place. L’application servira à saisir des entrées de feuilles de temps et à envoyer les données à la base de données centralisée des feuilles de temps au moyen de l’API. L’application permettra aussi aux gestionnaires d’approuver les entrées de feuilles de temps.

<div id="goals-requirements">
  ### Objectifs et exigences
</div>

ExampleCo souhaite mettre en place une solution flexible. Il pourrait y avoir plusieurs employés et sous-traitants qui doivent pouvoir consigner des entrées de feuilles de temps, ainsi que des processus par lots pouvant téléverser des entrées de feuilles de temps à partir d’autres systèmes externes.

L’entreprise a donc décidé de développer une seule Timesheets API, qui servira à consigner le temps non seulement pour cette application mobile, mais aussi pour toutes les autres applications. Elle souhaite mettre en place une architecture de sécurité suffisamment flexible pour répondre à ce besoin. ExampleCo veut s’assurer qu’une grande partie du code et de la logique opérationnelle de l’application puisse être partagée entre les différentes applications.

Seuls les utilisateurs et les applications autorisés doivent avoir accès à la Timesheets API.
