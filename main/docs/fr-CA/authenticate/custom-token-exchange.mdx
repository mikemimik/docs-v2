L’échange de jeton personnalisé permet aux applications d’échanger leurs jetons existants contre des jetons Auth0 lors de l’appel du point de terminaison `/oauth/token`, tel que défini dans la [RFC 8693](https://datatracker.ietf.org/doc/html/rfc8693). Les cas d’utilisation courants de l’échange de jeton personnalisé incluent :

* Obtenir des jetons Auth0 pour une autre <Tooltip tip="Audience : Identifiant unique de l’audience pour un jeton émis. Nommée aud dans un jeton, sa valeur contient l’ID d’une application (Client ID) pour un ID Token ou d’une API (API Identifier) pour un jeton d’accès." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=audience">audience</Tooltip>
* S’intégrer à un <Tooltip tip="Identity Provider (IdP) : Service qui stocke et gère les identités numériques." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=identity+provider">fournisseur d’identité</Tooltip> externe
* Migrer vers Auth0
* Autorisation déléguée : un principal (comme un service, un agent IA ou un agent du soutien) agit au nom d’un utilisateur

Pour en savoir plus, consultez [Exemples de cas d’utilisation et échantillons de code](/docs/fr-CA/authenticate/custom-token-exchange/cte-example-use-cases).

Chaque requête d’échange de jeton personnalisé correspond à un [profil d’échange de jeton personnalisé](/docs/fr-CA/authenticate/custom-token-exchange/configure-custom-token-exchange#create-custom-token-exchange-profile), régi par une [Action](/docs/fr-CA/customize/actions), dans laquelle vous pouvez :

* Écrire du code personnalisé pour décoder et valider les `subject_tokens` transmis au point de terminaison `/oauth/token`
* Autoriser l’accès et définir l’utilisateur afin de poursuivre la transaction.

Vous pouvez [configurer plusieurs profils d’échange de jeton personnalisés](/docs/fr-CA/authenticate/custom-token-exchange/configure-custom-token-exchange#create-custom-token-exchange-profile) pour une application. Une fois que le serveur d’autorisation Auth0 a validé que la requête d’échange de jeton personnalisé est valide et correspond à un profil d’échange de jeton personnalisé existant, le [déclencheur d’échange de jeton personnalisé](/docs/fr-CA/customize/actions/explore-triggers/signup-and-login-triggers/custom-token-exchange-trigger) exécute l’unique Action associée à ce profil. L’application peut ensuite tirer parti de l’échange de jeton personnalisé pour authentifier l’utilisateur et obtenir pour lui des jetons d’accès, des jetons ID et des jetons d’actualisation Auth0.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  L’échange de jeton personnalisé vous offre la flexibilité supplémentaire de définir l’utilisateur pour la transaction, en assumant aussi la responsabilité additionnelle de valider de façon sécuritaire le jeton subject correspondant qui identifie l’utilisateur pour la transaction.

  N’oubliez pas que les jetons subject et actor utilisés avec l’échange de jeton personnalisé peuvent être de n’importe quel format ou type de jeton, pourvu que le code de votre Action puisse les interpréter. **Vous devez mettre en œuvre une validation rigoureuse des jetons que vous recevez et acceptez.** Si vous ne le faites pas, vous vous exposez à différents vecteurs d’attaque, comme l’usurpation d’identité ou les attaques par rejeu, ce qui pourrait permettre à des acteurs malveillants de s’authentifier avec l’ID utilisateur de quelqu’un d’autre ou d’agir en son nom de manière non autorisée.

  Pour en savoir plus sur les différentes options pour mettre en œuvre une validation sécuritaire de vos jetons subject, lisez et appliquez les recommandations incluses dans [Exemples de cas d’utilisation et échantillons de code](/docs/fr-CA/authenticate/custom-token-exchange/cte-example-use-cases). Assurez-vous aussi de prendre en compte et d’appliquer les capacités de [Protection contre les attaques](/docs/fr-CA/authenticate/custom-token-exchange/cte-attack-protection).
</Callout>

<div id="tenant-logs">
  ## Logs du tenant
</div>

Chaque transaction d’échange de jeton personnalisé génère un log d’événement du tenant :

* Transactions réussies : logs `secte`
* Transactions échouées : logs `fecte`

Lorsqu’un acteur est défini au moyen de `setActor()`, la propriété `actor` (contenant `sub` et tout `actor` imbriqué) est incluse dans les entrées de log `secte` à des fins d’audit.

Utilisez les logs du tenant pour vous aider à résoudre les problèmes que vous pourriez rencontrer lors de votre échange de jetons.

<div id="limitations">
  ## Limites
</div>

L’échange de jeton personnalisé ne prend pas en charge les éléments suivants :

* les méthodes de la MFA API `api.authentication.challengeWith()` et `api.authentication.EnrollWith()`
* les connexions DB personnalisées avec le mode d’importation `ON` ne sont pas prises en charge pour les opérations `setUserByConnection()`
* les clients tiers et non conformes à OIDC
* l’API cible doit avoir l’option **Autoriser l’omission du consentement de l’utilisateur** activée, puisque le consentement ne peut pas être recueilli dans un flux non interactif.
