<div id="use-sms-and-email-as-authentication-factors">
  ## Utiliser les SMS et le courriel comme facteurs d&#39;authentification
</div>

La mise en œuvre de l&#39;authentification Passwordless d&#39;Auth0 permet d&#39;authentifier les utilisateurs au moyen d&#39;un seul facteur. Ce facteur unique peut être un code à usage unique envoyé par courriel ou par SMS, ou encore un lien magique envoyé par courriel.

Même si l&#39;utilisation du courriel ou des SMS peut être plus sécuritaire qu&#39;un mot de passe faible, les deux présentent des faiblesses connues :

* Les numéros de téléphone ne suffisent pas à l&#39;authentification des utilisateurs. Le [système d&#39;acheminement téléphonique SS7 utilisé par les réseaux cellulaires présente des faiblesses avérées](https://thehackernews.com/2017/05/ss7-vulnerability-bank-hacking.html), si bien qu&#39;il n&#39;est pas recommandé comme facteur d&#39;authentification. Il existe de nombreux vecteurs d&#39;attaque, allant de l&#39;ingénierie sociale au remplacement de cartes SIM et à l&#39;achat d&#39;accès au réseau SS7.
* Le simple fait d&#39;avoir accès à une adresse courriel ne suffit pas à l&#39;authentification des utilisateurs (par exemple les alias, le transfert de courriels ou les comptes partagés entre plusieurs utilisateurs). Les fournisseurs de courriel ont des pratiques de sécurité variables, et certains n&#39;exigent aucune vérification de l&#39;identité de l&#39;utilisateur. SMTP est un protocole très ancien, et de nombreux fournisseurs acheminent encore le trafic SMTP sans chiffrement, ce qui accroît le risque d&#39;interception.

Pour ces raisons, nous vous recommandons, si vous utilisez l&#39;authentification Passwordless, de mettre également en œuvre [l&#39;authentification multifacteur (MFA)](/docs/fr-CA/secure/multi-factor-authentication) au moyen d&#39;un facteur différent lorsque l&#39;utilisateur effectue une opération sensible sur le plan de la sécurité.

<div id="prevent-phishing-attacks">
  ## Prévenir les attaques d’hameçonnage
</div>

Voici à quoi pourrait ressembler une attaque d’hameçonnage :

1. L’utilisateur clique sur un lien dans un courriel ou sur un site Web malveillant.
2. L’utilisateur arrive sur le faux site de l’attaquant, où on lui demande d’entrer son numéro de téléphone pour s’authentifier.
3. L’utilisateur entre son numéro de téléphone, puis l’attaquant saisit le même numéro dans l’application légitime.
4. L’application légitime envoie un SMS à l’utilisateur.
5. L’utilisateur saisit le code à usage unique sur le site Web de l’attaquant.
6. L’attaquant peut maintenant se connecter au site Web légitime.

Pour réduire les chances de réussite de cette attaque, l’utilisateur devrait s’attendre à ce que le SMS indique clairement l’application. Vous devriez configurer le modèle de SMS pour qu’il mentionne le nom du tenant et/ou le nom de l’application :

```http wrap lines
Your verification code for accessing's Acme @@application.name@@ is @@code@@
```

<div id="prevent-brute-force-attacks">
  ## Prévenir les attaques par force brute
</div>

Auth0 offre les protections suivantes contre les attaques par force brute :

* Seul le code à usage unique (ou le lien) le plus récent émis sera accepté. Dès qu’un nouveau code ou lien est émis, tous les précédents sont invalidés. Une fois utilisé, le plus récent est lui aussi invalidé.
* Seules trois tentatives infructueuses de saisie d’un même code à usage unique sont autorisées. Après cela, il faudra demander un nouveau code.
* Le code à usage unique émis sera valide pendant trois minutes (par défaut) avant d’expirer.
* Si un utilisateur Passwordless a été [bloqué administrativement](/docs/fr-CA/manage-users/user-accounts/block-and-unblock-users), Auth0 n’enverra aucun code OTP par SMS ou par courriel tant que l’utilisateur n’aura pas été débloqué. Ce comportement évite que vos fournisseurs de SMS et de courriel reçoivent des requêtes inutiles.

Le délai d’expiration du code à usage unique peut être modifié dans [Auth0 Dashboard &gt; Authentication &gt; Passwordless](https://manage.auth0.com/#/connections/passwordless).

<div id="prevent-user-enumeration-attacks">
  ## Prévenir les attaques par énumération d’utilisateurs
</div>

L’énumération d’utilisateurs se produit lorsqu’une personne malveillante utilise des techniques de force brute pour deviner ou confirmer la présence d’utilisateurs valides dans un système.

Lorsque **Disable Sign Ups** est activé, votre application peut devenir vulnérable aux attaques par énumération d’utilisateurs. Auth0 recommande de ne pas activer ce paramètre afin d’assurer la sécurité maximale de votre application et de ses utilisateurs.

Si vous activez **Disable Signs Ups**, la [protection contre les attaques par force brute](/docs/fr-CA/secure/attack-protection/brute-force-protection) peut atténuer la menace associée à certaines de ces attaques, mais elle ne protège pas entièrement votre application.

<div id="link-accounts">
  ## Lier des comptes
</div>

Les utilisateurs peuvent vouloir s’authentifier au moyen de différents facteurs Passwordless au fil du temps. Par exemple, ils peuvent d’abord s’inscrire par SMS, puis plus tard commencer à s’authentifier par courriel. Vous pouvez le faire en leur permettant de lier leurs différents profils au moyen de la [liaison de comptes](/docs/fr-CA/manage-users/user-accounts/user-account-linking/link-user-accounts).

<div id="set-the-auth0-forwarded-for-header-for-rate-limit-purposes">
  ## Définir l’en-tête auth0-forwarded-for aux fins de limitation du débit
</div>

Le point de terminaison `/passwordless/start` a une [limite de débit](/docs/fr-CA/troubleshoot/customer-support/operational-policies/rate-limit-policy) de 50 requêtes par heure par adresse IP. Si vous appelez l’API à partir du serveur, l’adresse IP de votre backend peut facilement atteindre ces limites de débit. Pour savoir comment résoudre ce problème, consultez la section Limitation du débit dans les points de terminaison Passwordless de [Using Passwordless APIs](/docs/fr-CA/authenticate/passwordless/implement-login/embedded-login/relevant-api-endpoints).
