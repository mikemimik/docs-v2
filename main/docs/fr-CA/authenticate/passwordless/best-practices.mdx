<div id="use-sms-and-email-as-authentication-factors">
  ## Utiliser les SMS et les courriels comme facteurs d’authentification
</div>

La mise en œuvre de l’authentification Passwordless d’Auth0 permet d’authentifier les utilisateurs à l’aide d’un seul facteur. Ce facteur peut être un code à usage unique envoyé par courriel ou par SMS, ou encore un Magic Link envoyé par courriel.

Même si le courriel ou les SMS peuvent être plus sécuritaires qu’un mot de passe faible, les deux présentent des lacunes connues :

* Les numéros de téléphone ne suffisent pas pour l’authentification des utilisateurs. Le [système de routage téléphonique SS7 utilisé par les réseaux cellulaires présente des faiblesses connues](https://thehackernews.com/2017/05/ss7-vulnerability-bank-hacking.html), si bien qu’il n’est pas recommandé comme facteur d’authentification. Il existe de nombreux vecteurs d’attaque, allant de l’ingénierie sociale à l’échange de cartes SIM et à l’achat d’un accès au réseau SS7.
* La possession d’une adresse courriel ne suffit pas pour l’authentification des utilisateurs (les alias, le transfert et les comptes partagés par plusieurs utilisateurs en sont des exemples). Les fournisseurs de courriel n’ont pas tous les mêmes pratiques de sécurité, et certains n’exigent aucune vérification de l’identité de l’utilisateur. SMTP est un protocole très ancien, et de nombreux fournisseurs acheminent encore le trafic SMTP sans chiffrement, ce qui augmente le risque d’interception.

Pour ces raisons, nous recommandons que, si vous utilisez l’authentification Passwordless, vous mettiez aussi en œuvre [l’authentification multifacteur (MFA)](/docs/fr-CA/secure/multi-factor-authentication) avec un facteur différent lorsque l’utilisateur effectue une opération sensible sur le plan de la sécurité.

<div id="prevent-phishing-attacks">
  ## Prévenir les attaques d’hameçonnage
</div>

Voici un exemple de ce à quoi pourrait ressembler une attaque d’hameçonnage :

1. L’utilisateur clique sur un lien dans un courriel ou sur un site Web malveillant.
2. L’utilisateur arrive sur le faux site de l’attaquant, où on lui demande d’entrer son numéro de téléphone pour s’authentifier.
3. L’utilisateur entre son numéro de téléphone, et l’attaquant saisit le même numéro dans l’application légitime.
4. L’application légitime envoie un SMS à l’utilisateur.
5. L’utilisateur saisit le code à usage unique sur le site Web de l’attaquant.
6. L’attaquant peut alors se connecter au site Web légitime.

Pour réduire les chances de réussite de cette attaque, l’utilisateur devrait s’attendre à ce que le SMS identifie clairement l’application. Vous devriez configurer le modèle SMS pour qu’il mentionne le nom du tenant et/ou le nom de l’application :

```http wrap lines
Your verification code for accessing's Acme @@application.name@@ is @@code@@
```

<div id="prevent-brute-force-attacks">
  ## Prévenir les attaques par force brute
</div>

Auth0 offre les protections suivantes contre les attaques par force brute :

* Seul le code à usage unique (ou lien) émis le plus récemment sera accepté. Dès qu’un nouveau code est émis, tous les autres sont invalidés. Une fois utilisé, le plus récent est lui aussi invalidé.
* Seulement trois tentatives infructueuses de saisie d’un même code à usage unique sont autorisées. Après cela, il faudra demander un nouveau code.
* Le code à usage unique émis sera valide pendant trois minutes (par défaut) avant d’expirer.
* Si un utilisateur Passwordless a été [bloqué par un administrateur](/docs/fr-CA/manage-users/user-accounts/block-and-unblock-users), Auth0 n’enverra aucun code OTP par SMS ou courriel tant que l’utilisateur n’aura pas été débloqué. Ce comportement évite que vos fournisseurs de SMS et de courriel reçoivent des requêtes inutiles.

Le délai d’expiration du code à usage unique peut être modifié dans [Auth0 Dashboard &gt; Authentication &gt; Passwordless](https://manage.auth0.com/#/connections/passwordless).

<div id="prevent-user-enumeration-attacks">
  ## Prévenir les attaques d’énumération d’utilisateurs
</div>

L’énumération d’utilisateurs se produit lorsqu’un acteur malveillant utilise des techniques de force brute pour deviner ou confirmer l’existence d’utilisateurs valides dans un système.

Lorsque **Disable Sign Ups** est activé, votre application peut devenir vulnérable aux attaques d’énumération d’utilisateurs. Auth0 recommande de ne pas activer ce paramètre afin d’assurer la sécurité maximale de votre application et de ses utilisateurs.

Si vous activez tout de même **Disable Signs Ups**, la [protection contre les attaques par force brute](/docs/fr-CA/secure/attack-protection/brute-force-protection) peut atténuer la menace de certaines de ces attaques, mais elle ne protège pas complètement votre application.

<div id="link-accounts">
  ## Lier des comptes
</div>

Les utilisateurs peuvent vouloir s’authentifier à l’aide de différents facteurs sans mot de passe au fil du temps. Par exemple, ils peuvent d’abord s’inscrire par SMS, puis commencer plus tard à s’authentifier par courriel. Vous pouvez le permettre en leur donnant la possibilité de lier leurs différents profils au moyen de la [liaison de comptes](/docs/fr-CA/manage-users/user-accounts/user-account-linking/link-user-accounts).

<div id="set-the-auth0-forwarded-for-header-for-rate-limit-purposes">
  ## Définir l’en-tête auth0-forwarded-for pour la limitation du débit
</div>

Le point de terminaison `/passwordless/start` a une [limite de débit](/docs/fr-CA/troubleshoot/customer-support/operational-policies/rate-limit-policy) de 50 requêtes par heure, par adresse IP. Si vous appelez l’API côté serveur, l’adresse IP de votre backend peut facilement atteindre ces limites de débit. Pour savoir comment corriger ce problème, consultez la section Limitation du débit dans les points de terminaison Passwordless de [Using Passwordless APIs](/docs/fr-CA/authenticate/passwordless/implement-login/embedded-login/relevant-api-endpoints).
