<ReleaseStageNotice feature="Clés d’accès natives" stage="ea" contact="soutien Auth0" />

Les clés d’accès sont une alternative résistante à l’hameçonnage aux méthodes d’authentification traditionnelles (comme le nom d’utilisateur et mot de passe) et offrent une expérience utilisateur plus simple et plus sécurisée. Pour obtenir tous les détails de la mise en œuvre, consultez [Native Passkeys for Mobile Applications](/docs/fr-CA/authenticate/database-connections/passkeys/native-passkeys-for-mobile-applications).

Les clés d’accès natives utilisent une combinaison d’Auth0 et d’API iOS ou Android natives pour intégrer directement les flux de défi à votre application mobile. Les points de terminaison ci-dessous constituent un sous-ensemble de l’Auth0 Authentication API. Pour en savoir plus sur l’utilisation de cette API, consultez [Introduction à l’Authentication API](https://auth0.com/docs/api/authentication#introduction).

Les clés d’accès comportent trois flux connexes :

* **Signup** : crée un nouveau compte d’utilisateur avec une clé d’accès comme principale méthode d’authentification.
* **enrôlement** : ajoute une clé d’accès comme méthode d’authentification au compte existant d’un utilisateur.
* **Login** : demande à un utilisateur existant de s’authentifier à l’aide d’une clé d’accès associée à son compte.

<div id="signup-flow">
  ## Flux d’inscription
</div>

<div id="request-signup-challenge">
  ### Demander le défi d’inscription
</div>

`POST /passkey/register`

Lance le flux d’inscription avec clé d’accès pour un nouvel utilisateur.

En réponse, Auth0 renvoie [PublicKeyCredentialCreationOptions](https://www.w3.org/TR/webauthn-3/#dictdef-publickeycredentialcreationoptions) et un ID de session. Vérifiez la valeur `timeout` dans `authn_params_public_key` de la réponse pour connaître le délai d’expiration de la session.

Le flux d’inscription avec clé d’accès prend en charge les Organizations au moyen du paramètre `organization`, conformément au comportement décrit dans [Login Flows for Organizations](/docs/fr-CA/manage-users/organizations/login-flows-for-organizations). Si votre application est configurée pour les **Business Users**, vous devez fournir le paramètre `organization` ainsi qu’un nom d’Organization valide ou une valeur d’identifiant valide. Une fois que l’utilisateur a enregistré une clé d’accès, Auth0 l’inscrit dans l’Organization fournie.

<div id="request-parameters">
  #### Paramètres de la requête
</div>

| Paramètre      | Description                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                   |
| -------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `client_id`    | **Obligatoire**. Le `client_id` de votre application.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                         |
| `realm`        | **Facultatif**. Le nom de la connexion à associer à cet utilisateur.<br /><br />Si aucune connexion n&#39;est précisée, l&#39;annuaire par défaut de votre tenant est utilisé.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                |
| `user_profile` | **Obligatoire**. Objet contenant les renseignements d&#39;identification de l&#39;utilisateur. Par défaut, il comprend une adresse `email` valide et un `name` d&#39;affichage facultatif.<br /><br />Si vous avez activé [Flexible Identifiers](/docs/fr-CA/authenticate/database-connections/activate-and-configure-attributes-for-flexible-identifiers) pour votre connexion de base de données, vous pouvez utiliser une combinaison de `email`, `phone_number` ou `username` comme identifiants. Ces options peuvent être obligatoires ou facultatives et doivent correspondre à votre configuration de Flexible Identifiers.<br /><br />Si l&#39;identifiant transmis (comme `email`) existe déjà dans l&#39;annuaire, l&#39;utilisateur devrait plutôt être invité à terminer le Login flow. |

<div id="remarks">
  #### Remarques
</div>

* Une fois la demande de défi terminée, votre application peut poursuivre le processus d’inscription de l’utilisateur à l’aide des API natives [Android](https://developer.android.com/identity/passkeys/create-passkeys) ou [iOS](https://developer.apple.com/documentation/authenticationservices/supporting-passkeys#Register-a-new-account-on-a-service).
* Vous devez ensuite [authentifier le nouvel utilisateur](#authenticate-new-user) à l’aide des renseignements obtenus par les API natives pour terminer le processus.

<div id="authenticate-new-user">
  ### Authentifier un nouvel utilisateur
</div>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  L’enregistrement Native Clé d’accès n’est pas pris en charge actuellement lorsqu’une vérification par SMS ou Email OTP est requise sur la même connexion au moment de l’inscription.
</Callout>

`POST /oauth/token`

Utilise le <Tooltip tip="point de terminaison de jeton : point de terminaison sur le serveur d’autorisation utilisé pour demander des jetons par programmation." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Token+endpoint">point de terminaison de jeton</Tooltip> pour authentifier l’utilisateur à l’aide des informations d’identification fournies, créer son compte et renvoyer les jetons demandés.

Le paramètre `authn_response` repose sur la [spécification de la Web Authentication API](https://www.w3.org/TR/webauthn-3/). Dans le flux Native Clé d’accès, les renseignements transmis à ce point de terminaison peuvent être récupérés au moyen des API natives de votre application mobile :

* [documentation d’enregistrement Android](https://developer.android.com/identity/passkeys/create-passkeys)
* [documentation d’enregistrement iOS](https://developer.apple.com/documentation/authenticationservices/supporting-passkeys#Register-a-new-account-on-a-service)

<div id="enrollment-flow">
  ## Enrôlement
</div>

L’enrôlement d’une nouvelle clé d’accès pour un utilisateur se fait en deux étapes à l’aide de la My Account API. Ce processus garantit que l’enrôlement de la clé d’accès est lancé de façon sécuritaire, puis vérifié.

Avant de lancer l’enrôlement, assurez-vous d’avoir un <Tooltip tip="Jeton d’accès : identifiant d’autorisation, sous la forme d’une chaîne opaque ou d’un JWT, utilisé pour accéder à une API." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=access+token">jeton d’accès</Tooltip> avec la portée `create:me:authentication_methods` pour le point de terminaison `/me`.

<div id="initiate-passkey-enrollment">
  ### Commencer l’enrôlement de la clé d’accès
</div>

`POST /me/v1/authentication-methods`

La première étape consiste à démarrer le processus d’enrôlement. Pour ce faire, envoyez une requête POST au point de terminaison `/me/v1/authentication-methods`.

#### Paramètres de requête

| Paramètre    | Description                                                                  |
| ------------ | ---------------------------------------------------------------------------- |
| `type`       | **Obligatoire**. Inclure la valeur : `public-key`.                           |
| `connection` | **Facultatif**. Le nom de la connexion dans laquelle créer la clé d’accès.   |
| `identity`   | **Facultatif**. L’identité de l’utilisateur. Utilisée avec les comptes liés. |

<div id="remarks">
  #### Remarques
</div>

* La propriété `auth_session` dans le corps de la réponse correspond à l’identifiant de la session d’authentification en cours. Elle doit être transmise à l’endpoint `/verify`.
* Une fois la requête de défi terminée, votre application peut poursuivre le processus d’enrôlement de l’utilisateur à l’aide des API natives [Android](https://developer.android.com/identity/passkeys/create-passkeys) ou [iOS](https://developer.apple.com/documentation/authenticationservices/supporting-passkeys#Register-a-new-account-on-a-service). L’utilisateur sera alors invité à créer une clé d’accès avec son authentificateur (par exemple, un lecteur d’empreintes digitales, une clé de sécurité ou un téléphone).

<div id="verify-passkey-enrollment">
  ### Vérifier l’enrôlement de la clé d’accès
</div>

`POST /me/v1/authentication-methods/passkey|new/verify`

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  L’ID dans le chemin est toujours `passkey|new` pour les nouveaux enrôlements.
</Callout>

Une fois que l’utilisateur a créé sa clé d’accès avec succès à l’aide de son authentificateur, l’application cliente recevra une `AuthenticatorAttestationResponse` de l’API WebAuthn. Cette réponse doit ensuite être renvoyée au service Auth0 pour terminer et valider l’enrôlement.

* [Documentation sur l’enregistrement Android](https://developer.android.com/identity/passkeys/create-passkeys)
* [Documentation sur l’enregistrement iOS](https://developer.apple.com/documentation/authenticationservices/supporting-passkeys#Register-a-new-account-on-a-service)

ID d’identification Base64URL.                                                                                                                                                                                                                                                                                               |
| `authn_response.rawId`                   | **Obligatoire**. <div id="login-flow">
  ## Flux de connexion
</div>

<div id="request-login-challenge">
  ### Request Login Défi
</div>

`POST /passkey/challenge`

Lance le flux de connexion par clé d’accès pour un utilisateur existant qui a enregistré une clé d’accès dans son compte lors de son inscription initiale.

En réponse, Auth0 renvoie [PublicKeyCredentialRequestOptions](https://www.w3.org/TR/webauthn-3/#dictdef-publickeycredentialrequestoptions), un ID de session et un identifiant de Relying Party `rpId`. Pour connaître le délai d’expiration de la session, vérifiez `timeout` sous `authn_params_public_key` dans la réponse. Le `rpId` renvoyé dans la réponse est l’identifiant que l’appareil natif (iOS/Android) utilise pour retrouver les credentials enregistrés associés à un domaine. Pour qu’une clé d’accès créée dans une application web (par ex. `example.com`) soit accessible dans le flux natif, le `rpId` renvoyé doit correspondre au `rpId` de l’application web. Pour savoir comment personnaliser le RP ID pour votre tenant, consultez [Configure Clé d’accès Policy](/docs/fr-CA/authenticate/database-connections/passkeys/configure-passkey-policy#configure-relying-party-id-rp-id).

Le flux de connexion par clé d’accès prend en charge Organizations au moyen du paramètre `organization`, selon le comportement décrit dans [Login Flows for Organizations](/docs/fr-CA/manage-users/organizations/login-flows-for-organizations). Si votre application est configurée pour les **Business Users**, vous devez fournir le paramètre `organization` ainsi qu’un nom d’Organization valide ou une valeur d’identifiant valide.

Tous les jetons émis s’inscrivent dans le contexte de l’Organization fournie. Si vous avez activé [Auto-Membership](/docs/fr-CA/manage-users/organizations/login-flows-for-organizations#auto-membership) pour votre Organization, l’utilisateur est automatiquement inscrit à l’Organization après s’être authentifié avec succès.

<div id="authenticate-existing-user">
  ### Authentifier un utilisateur existant
</div>

`POST /oauth/token`

Utilise le point de terminaison de jeton pour authentifier l’utilisateur à l’aide des identifiants fournis et renvoyer les jetons demandés.

Le paramètre `authn_response` est basé sur la spécification [Web Authentication API](https://www.w3.org/TR/webauthn-3/). Dans le flux natif de clé d’accès, les informations transmises à ce point de terminaison peuvent être récupérées au moyen des API natives de votre application mobile :

* [Documentation Android sur la connexion](https://developer.android.com/identity/passkeys/sign-in-with-passkeys)
* [Documentation iOS sur la connexion](https://developer.apple.com/documentation/authenticationservices/supporting-passkeys#Connect-to-a-service-with-an-existing-account)

| `authn_response.type`                    | **Obligatoire**. Incluez la valeur : `public-key`                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                         |
| `authn_response.authenticatorAttachment` | **Facultatif**. Incluez les valeurs :<ul><li>`platform`</li><li>`cross-platform`</li></ul>                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                |
| `authn_response.response`                | **Obligatoire**. Un objet contenant les éléments suivants : <ul><li>`authenticatorData` : Contient les [données de l’authentificateur](https://www.w3.org/TR/webauthn-3/#authenticator-data) renvoyées par l’authentificateur.</li><li>`clientDataJSON` : Contient une sérialisation JSON-compatible des données du client; héritée de [AuthenticatorResponse](https://www.w3.org/TR/webauthn-3/#authenticatorresponse).</li><li>`signature` : Signature Base64URL renvoyée par l’authentificateur.</li><li>`userHandle` : identifiant Base64URL du compte d’utilisateur, renvoyé sous la forme de `user.id` à l’étape d’enregistrement.</li></ul> |
| `authn_response.clientExtensionResults`  | **Facultatif**. Contient les résultats du traitement des extensions client demandées par la relying party.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                |
