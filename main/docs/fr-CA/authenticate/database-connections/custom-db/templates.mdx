Si vous avez votre propre base de données (appelée stockage de données hérité dans Auth0) qui contient des données d’identité utilisateur, vous pouvez l’utiliser comme <Tooltip tip="Fournisseur d’identité (IdP) : service qui stocke et gère les identités numériques." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=identity+provider">fournisseur d’identité</Tooltip> pour authentifier les utilisateurs.
Vous créez et configurez la connexion à votre stockage de données hérité comme une base de données personnalisée dans Auth0. Vous pouvez choisir de migrer progressivement les données de votre base de données héritée vers le stockage de données d’Auth0 au fil du temps, ou continuer à l’utiliser sans migrer les données. Nous fournissons des modèles de scripts pour exécuter des opérations sur la base de données personnalisée, que vous pouvez utiliser et personnaliser.

Il existe deux types de scripts pour base de données personnalisée :

* **Migration automatique** : Chaque fois qu’un utilisateur se connecte à Auth0, s’il n’existe pas encore dans Auth0, le script vérifie s’il existe dans la base de données héritée. Si c’est le cas et que l’indicateur **Import Users to Auth0** est activé, les données utilisateur sont migrées vers le stockage de données d’Auth0. Cette capacité est parfois appelée **migration progressive** ou **migration paresseuse**.
* **Base de données héritée** : Auth0 interroge toujours la base de données sous-jacente lorsqu’un utilisateur tente de se connecter, est créé, change son mot de passe, vérifie son adresse courriel ou est supprimé. Si l’utilisateur est trouvé et que l’indicateur **Import Users to Auth0** n’est **pas** activé, les données utilisateur restent dans la base de données héritée et ne sont **pas** migrées vers Auth0.

* [Delete User](/docs/fr-CA/authenticate/database-connections/custom-db/templates/delete)
* [Get User](/docs/fr-CA/authenticate/database-connections/custom-db/templates/get-user)
* [Login](/docs/fr-CA/authenticate/database-connections/custom-db/templates/login)
* [Verify User](/docs/fr-CA/authenticate/database-connections/custom-db/templates/verify)
* [Change Email](/docs/fr-CA/authenticate/database-connections/custom-db/templates/change-email)

<div id="script-execution">
  ## Exécution du script
</div>

Comme décrit dans [Connexions de base de données personnalisées](/docs/fr-CA/authenticate/database-connections/custom-db), un type de connexion à une base de données personnalisée vous permet de configurer des scripts d&#39;action : du code personnalisé utilisé pour interagir avec votre répertoire d’identités existant. Chaque script d&#39;action est essentiellement une fonction Javascript nommée à laquelle un certain nombre de paramètres sont transmis; le nom de la fonction et les paramètres transmis varient selon le script.

<div id="limits">
  ### Limites
</div>

L’exécution du script d’action prend en charge la nature asynchrone de Javascript, et des constructions comme les objets [Promise](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/Promise) peuvent être utilisées. Le traitement asynchrone entraîne essentiellement une suspension jusqu’à ce qu’une opération soit terminée, et un conteneur Webtask serverless d’Auth0 a généralement une limite d’exécution de 20 secondes, après quoi il peut être recyclé. Le recyclage d’un conteneur en raison de cette limite met prématurément fin à l’opération, ce qui entraîne le renvoi d’une erreur (et peut aussi réinitialiser l’objet `global`).

<div id="completion-and-the-callback-function">
  ### Fin de l’exécution et fonction `callback`
</div>

La fonction `callback` fournie à chaque script d&#39;action agit comme un signal pour indiquer la fin de l’opération. Un script d&#39;action doit se terminer immédiatement après un appel à la fonction `callback` (implicitement ou en exécutant explicitement une instruction `return` en Javascript) et doit s’abstenir de toute autre opération.

<Warning>
  La fonction `callback` fournie par Auth0 doit être appelée exactement **une** fois; l’appeler plus d’une fois dans un script d&#39;action entraînera des résultats imprévisibles et/ou des erreurs.
</Warning>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Lorsque `callback` est exécutée sans paramètre, comme dans `callback()`, cela signifie que la fonction a été appelée comme si `callback(null)` avait été exécutée.
</Callout>

Si un script d&#39;action utilise un traitement asynchrone, l’appel à la fonction `callback` doit être reporté jusqu’au moment où ce traitement se termine, et il doit s’agir du tout dernier appel effectué. L’exécution asynchrone fait en sorte qu’une fonction Javascript `callback` soit exécutée une fois l’opération asynchrone terminée; cette fonction est généralement déclenchée après l’exécution du corps principal (synchrone) d’une fonction Javascript.

<Warning>
  Le fait de ne pas exécuter la fonction `callback` entraînera un blocage de l’exécution et, au bout du compte, le renvoi d’une erreur. Le script d&#39;action doit appeler la fonction `callback` exactement une fois : elle doit être appelée au moins une fois pour éviter le blocage de l’exécution, mais pas plus d’une fois, sinon des résultats imprévisibles et/ou des erreurs se produiront.
</Warning>
