Si vous avez votre propre base de données (appelée magasin de données hérité dans Auth0) qui contient des données d’identité utilisateur, vous pouvez l’utiliser comme <Tooltip tip="Fournisseur d’identité (IdP) : service qui stocke et gère les identités numériques." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=identity+provider">fournisseur d’identité</Tooltip> pour authentifier les utilisateurs.
Vous créez et configurez la connexion à votre magasin de données hérité en tant que base de données personnalisée dans Auth0. Vous pouvez choisir de migrer progressivement les données de votre ancienne base de données vers le magasin de données Auth0, ou continuer à l’utiliser sans migrer les données. Nous fournissons des modèles de scripts pour exécuter des opérations sur la base de données personnalisée, que vous pouvez utiliser et personnaliser.

Il existe deux types de scripts de base de données personnalisée :

* **Migration automatique** : Chaque fois qu’un utilisateur se connecte à Auth0, s’il n’est pas encore dans Auth0, le script vérifie dans la base de données héritée si cet utilisateur s’y trouve. Si c’est le cas et que l’indicateur **Import users to Auth0** est activé, les données utilisateur sont migrées vers le magasin de données Auth0. Cette fonctionnalité est parfois appelée **migration progressive** ou **migration paresseuse**.
* **Base de données héritée** : Auth0 interroge toujours la base de données sous-jacente lorsqu’un utilisateur tente de se connecter, est créé, change son mot de passe, vérifie son adresse courriel ou est supprimé. Si l’utilisateur est trouvé et que l’indicateur **Import users to Auth0** n’est **pas** activé, ses données restent dans la base de données héritée et **ne** sont **pas** migrées vers Auth0.

<div id="script-execution">
  ## Exécution du script
</div>

Comme décrit dans [Custom Database Connections](/docs/fr-CA/authenticate/database-connections/custom-db), un type de connexion à une base de données personnalisée vous permet de configurer des scripts d’action : du code personnalisé utilisé pour interagir avec votre référentiel d’identités existant. Chaque script d’action est essentiellement une fonction JavaScript nommée à laquelle un certain nombre de paramètres sont transmis; le nom de la fonction et les paramètres transmis varient selon le script.

<div id="limits">
  ### Limites
</div>

L’exécution du script d’action prend en charge la nature asynchrone de Javascript, et des éléments comme les objets [Promise](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/Promise) peuvent être utilisés. Le traitement asynchrone entraîne effectivement une suspension en attendant qu’une opération se termine, et un conteneur Webtask serverless d’Auth0 a généralement une limite d’exécution de 20 secondes, après quoi il peut être recyclé. Le recyclage d’un conteneur en raison de cette limite met fin prématurément à l’opération, ce qui finit par entraîner le renvoi d’une erreur (et peut aussi provoquer une réinitialisation potentielle de l’objet `global`).

<div id="completion-and-the-callback-function">
  ### Fin d’exécution et fonction `callback`
</div>

La fonction `callback` fournie à chaque script d’action sert essentiellement de signal pour indiquer la fin de l’opération. Un script d’action doit se terminer immédiatement après l’appel de la fonction `callback` (soit implicitement, soit au moyen d’une instruction `return` explicite en JavaScript) et ne doit effectuer aucune autre opération.

<Warning>
  La fonction `callback` fournie par Auth0 doit être appelée exactement **une** fois ; l’appeler plus d’une fois dans un script d’action entraînera des résultats imprévisibles et/ou des erreurs.
</Warning>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Lorsque `callback` est exécutée sans paramètre, comme dans `callback()`, cela signifie que la fonction a été appelée comme si `callback(null)` avait été exécutée.
</Callout>

Si un script d’action utilise un traitement asynchrone, l’appel à la fonction `callback` doit alors être reporté jusqu’au moment où ce traitement se termine, et il doit s’agir du tout dernier appel effectué. L’exécution asynchrone fait en sorte qu’une fonction `callback` JavaScript soit exécutée une fois l’opération asynchrone terminée ; cette fonction de rappel est généralement déclenchée après l’exécution du corps principal (synchrone) d’une fonction JavaScript.

<Warning>
  Si la fonction `callback` n’est pas exécutée, l’exécution se bloquera et une erreur finira par être renvoyée. Le script d’action doit appeler la fonction `callback` exactement une fois : la fonction `callback` doit être appelée au moins une fois afin d’éviter le blocage de l’exécution, mais elle ne doit pas l’être plus d’une fois, sinon des résultats imprévisibles et/ou des erreurs se produiront.
</Warning>
