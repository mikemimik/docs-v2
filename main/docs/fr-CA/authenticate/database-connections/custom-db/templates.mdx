Si vous avez votre propre base de données (appelée ancien stockage de données dans Auth0) contenant des données d’identité utilisateur, vous pouvez l’utiliser comme <Tooltip tip="Fournisseur d’identité (IdP) : service qui stocke et gère les identités numériques." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=identity+provider">fournisseur d’identité</Tooltip> pour authentifier les utilisateurs.
Vous créez et configurez la connexion à votre ancien stockage de données sous forme de base de données personnalisée dans Auth0. Vous pouvez choisir de migrer progressivement les données de votre ancienne base de données vers le stockage de données d’Auth0 au fil du temps, ou continuer à l’utiliser sans migrer les données. Nous fournissons des modèles de scripts pour exécuter des fonctions sur la base de données personnalisée, que vous pouvez utiliser et personnaliser.

Il existe deux types de scripts de base de données personnalisée :

* **Migration automatique** : chaque fois qu’un utilisateur se connecte à Auth0, si cet utilisateur ne s’y trouve pas encore, le script vérifie s’il existe dans l’ancienne base de données. Si c’est le cas et que l’indicateur **Import users to Auth0** est activé, les données de l’utilisateur sont migrées vers le stockage de données d’Auth0. Cette fonctionnalité est parfois appelée **migration progressive** ou **migration à la demande**.
* **Ancienne base de données** : Auth0 interroge toujours la base de données sous-jacente lorsqu’un utilisateur tente de se connecter, est créé, change son mot de passe, vérifie son courriel ou est supprimé. Si l’utilisateur existe et que l’indicateur **Import users to Auth0** n’est **pas** activé, ses données restent dans l’ancienne base de données et ne sont **pas** migrées vers Auth0.

* [Supprimer un utilisateur](/docs/fr-CA/authenticate/database-connections/custom-db/templates/delete)
* [Obtenir un utilisateur](/docs/fr-CA/authenticate/database-connections/custom-db/templates/get-user)
* [Connexion](/docs/fr-CA/authenticate/database-connections/custom-db/templates/login)

* [Changer l’adresse courriel](/docs/fr-CA/authenticate/database-connections/custom-db/templates/change-email)

<div id="script-execution">
  ## Exécution de scripts
</div>

Comme décrit dans [les connexions de base de données personnalisées](/docs/fr-CA/authenticate/database-connections/custom-db), un type de connexion de base de données personnalisée vous permet de configurer des scripts d’action : du code personnalisé servant à interagir avec votre ancien référentiel d’identité. Chaque script d’action correspond essentiellement à une fonction JavaScript nommée à laquelle un certain nombre de paramètres sont transmis, le nom de la fonction et les paramètres transmis variant selon le script.

<div id="limits">
  ### Limites
</div>

L’exécution du script d’action prend en charge la nature asynchrone de JavaScript, et des constructions comme les objets [Promise](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/Promise) peuvent être utilisées. Le traitement asynchrone entraîne en pratique une suspension jusqu’à ce qu’une opération soit terminée, et un conteneur Webtask serverless d’Auth0 est généralement assujetti à une limite d’exécution de 20 secondes, après quoi le conteneur peut être recyclé. Le recyclage d’un conteneur en raison de cette limite met fin prématurément à l’opération, ce qui finit par renvoyer une erreur (et peut aussi entraîner la réinitialisation de l’objet `global`).

<div id="completion-and-the-callback-function">
  ### Fin et fonction `callback`
</div>

La fonction `callback` fournie à chaque script d’action sert essentiellement de signal pour indiquer que l’opération est terminée. Un script d’action doit se terminer immédiatement après l’appel de la fonction `callback` (soit implicitement, soit au moyen d’une instruction `return` explicite en Javascript) et ne doit effectuer aucune autre opération.

<Warning>
  La fonction `callback` fournie par Auth0 doit être appelée exactement **une** fois; si elle est appelée plus d’une fois dans un script d’action, cela entraînera des résultats imprévisibles et/ou des erreurs.
</Warning>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Lorsque `callback` est exécutée sans paramètre, comme dans `callback()`, cela signifie que la fonction a été appelée comme si `callback(null)` avait été exécutée.
</Callout>

Si un script d’action utilise un traitement asynchrone, l’appel à la fonction `callback` doit être reporté jusqu’au moment où ce traitement est terminé, et il doit s’agir du tout dernier appel effectué. L’exécution asynchrone fait en sorte qu’une fonction Javascript `callback` soit exécutée une fois l’opération asynchrone terminée; cette fonction de rappel est généralement déclenchée après l’exécution du corps principal (synchrone) d’une fonction Javascript.

<Warning>
  Si la fonction `callback` n’est pas exécutée, cela entraînera un blocage de l’exécution et, au bout du compte, le retour d’une erreur. Le script d’action doit appeler la fonction `callback` exactement une fois : elle doit être appelée au moins une fois pour éviter le blocage de l’exécution, mais pas plus d’une fois, faute de quoi des résultats imprévisibles et/ou des erreurs se produiront.
</Warning>
