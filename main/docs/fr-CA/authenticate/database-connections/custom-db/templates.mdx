Si vous avez votre propre base de données (appelée magasin de données hérité dans Auth0) qui contient des données d’identité utilisateur, vous pouvez l’utiliser comme <Tooltip tip="Fournisseur d’identité (IdP) : service qui stocke et gère les identités numériques." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=identity+provider">fournisseur d’identité</Tooltip> pour authentifier les utilisateurs.
Vous créez et configurez la connexion à votre magasin de données hérité en tant que base de données personnalisée dans Auth0. Vous pouvez choisir de migrer progressivement les données de votre base de données héritée vers le magasin de données d’Auth0 au fil du temps, ou continuer à l’utiliser sans migrer les données. Nous fournissons des modèles de script pour exécuter des opérations sur la base de données personnalisée, que vous pouvez utiliser et personnaliser.

Il existe deux types de scripts de base de données personnalisée :

* **Migration automatique** : chaque fois qu’un utilisateur se connecte à Auth0, s’il n’existe pas encore dans Auth0, le script vérifie s’il existe dans la base de données héritée. Si c’est le cas et que l’indicateur **Importer les utilisateurs vers Auth0** est activé, les données utilisateur sont migrées vers le magasin de données d’Auth0. Cette fonctionnalité est parfois appelée **migration progressive** ou **migration paresseuse**.
* **Base de données héritée** : Auth0 interroge toujours la base de données sous-jacente lorsqu’un utilisateur tente de se connecter, est créé, change son mot de passe, vérifie son adresse courriel ou est supprimé. Si l’utilisateur y est trouvé et que l’indicateur **Importer les utilisateurs vers Auth0** n’est **pas** activé, les données utilisateur restent dans la base de données héritée et ne sont **pas** migrées vers Auth0.

* [Supprimer un utilisateur](/docs/fr-CA/authenticate/database-connections/custom-db/templates/delete)

* [Changer l’adresse courriel](/docs/fr-CA/authenticate/database-connections/custom-db/templates/change-email)

<div id="script-execution">
  ## Exécution des scripts
</div>

Comme décrit dans [les connexions de base de données personnalisées](/docs/fr-CA/authenticate/database-connections/custom-db), un type de connexion à une base de données personnalisée vous permet de configurer des scripts d’action : du code personnalisé utilisé pour interagir avec votre ancien répertoire d’identités. Chaque script d’action est essentiellement une fonction JavaScript nommée à laquelle un certain nombre de paramètres sont transmis; le nom de la fonction et les paramètres transmis varient selon le script.

<div id="limits">
  ### Limites
</div>

L’exécution du script d’action prend en charge la nature asynchrone de JavaScript, et des constructions comme les objets [Promise](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/Promise) peuvent être utilisées. Le traitement asynchrone a pour effet de suspendre l’exécution jusqu’à ce qu’une opération soit terminée, et un conteneur Webtask serverless d’Auth0 a généralement une limite d’exécution de 20 secondes, après quoi le conteneur peut être recyclé. Le recyclage d’un conteneur en raison de cette limite met fin prématurément à l’opération, ce qui finit par renvoyer une erreur (et peut aussi entraîner la réinitialisation de l’objet `global`).

<div id="completion-and-the-callback-function">
  ### Fin et fonction `callback`
</div>

La fonction `callback` fournie à chaque script d’action sert essentiellement de signal pour indiquer la fin de l’opération. Un script d’action doit se terminer immédiatement après un appel à la fonction `callback` (soit implicitement, soit au moyen d’une instruction `return` explicite en Javascript) et ne doit effectuer aucune autre opération.

<Warning>
  La fonction `callback` fournie par Auth0 doit être appelée exactement **une** fois; si elle est appelée plus d’une fois dans un script d’action, cela entraînera des résultats imprévisibles et/ou des erreurs.
</Warning>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Lorsque `callback` est exécutée sans paramètre, comme dans `callback()`, cela signifie que la fonction a été appelée comme si `callback(null)` avait été exécutée.
</Callout>

Si un script d’action utilise un traitement asynchrone, l’appel à la fonction `callback` doit alors être reporté jusqu’à la fin de ce traitement, et il doit s’agir de la toute dernière chose appelée. En exécution asynchrone, une fonction `callback` Javascript est exécutée une fois l’opération asynchrone terminée; cette fonction est généralement déclenchée après l’exécution du corps principal (synchrone) d’une fonction Javascript.

<Warning>
  Si la fonction `callback` n’est pas exécutée, l’exécution se bloquera et une erreur finira par être renvoyée. Le script d’action doit appeler la fonction `callback` exactement une fois : elle doit être appelée au moins une fois pour éviter le blocage de l’exécution, mais jamais plus d’une fois, sinon des résultats imprévisibles et/ou des erreurs se produiront.
</Warning>
