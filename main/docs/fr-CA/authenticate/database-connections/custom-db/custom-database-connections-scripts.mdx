  La disponibilité de cette fonctionnalité dépend à la fois de votre implémentation de connexion et de votre plan Auth0 ou de votre entente personnalisée. Pour en savoir plus, consultez la [tarification](https://auth0.com/pricing).

L’extensibilité permet aux administrateurs d’ajouter une connexion personnalisée dans Auth0 comme mécanisme pour mettre en place des solutions de dernière étape en gestion des identités et des accès (IdAM). Chaque type d’extensibilité utilise Node.js et s’exécute sur la plateforme Auth0 dans un tenant Auth0.

L’extensibilité Auth0 se présente sous plusieurs formes :

* [Actions](/docs/fr-CA/customize/actions) : Les Actions sont des fonctions sécurisées, propres au tenant et versionnées, écrites en Node.js, qui s’exécutent à certains points de la plateforme Auth0. Utilisez les Actions pour personnaliser et étendre les capacités d’Auth0 avec une connexion personnalisée.
* [Rules](/docs/fr-CA/customize/rules) : S’exécutent lorsque des artefacts d’authentification de l’utilisateur sont générés. Par exemple :

  * ID Token dans OpenID Connect (OIDC)
  * Access Token dans OAuth 2.0
  * Une assertion dans SAML
* [Hooks](/docs/fr-CA/customize/hooks) : Offrent une extensibilité supplémentaire lorsqu’il y a échange d’artefacts non liés à l’utilisateur et lors de la création d’identités utilisateur, comme pre-user registration et post-user registration.

Vous trouverez ci-dessous quelques pratiques exemplaires pour l’utilisation des produits d’extensibilité Auth0 :

* Les scripts pour les connexions de base de données personnalisées et les migrations servent à intégrer un référentiel d’identités utilisateur existant ou lorsqu’une migration automatique d’utilisateurs à partir d’un référentiel d’identités distinct ou hérité est requise. Pour en savoir plus sur la configuration, consultez [Create Custom Database Connections.](/docs/fr-CA/authenticate/database-connections/custom-db/create-db-connection)
* Pour éviter les doublons d’utilisateurs, définissez un `user_id` cohérent dans chaque profil utilisateur renvoyé. De plus, assurez-vous de ne pas renvoyer la même adresse courriel pour deux utilisateurs différents dans le script `get_user` ou `login`, et veillez à ce que les deux scripts renvoient le même utilisateur.
* Pour assurer la rétrocompatibilité, `app_metadata` est appelé `metadata` dans les scripts de base de données personnalisés. Comme le transfert de données peut être interrompu, utilisez une Action pour récupérer les métadonnées qui devraient se trouver dans `app_metadata` ou `user_metadata` si elles sont absentes.
* Assurez-vous de restreindre l’accès au <Tooltip tip="Audience : Identifiant unique du destinataire d’un jeton émis. Nommée aud dans un jeton, sa valeur contient l’ID d’une application (Client ID) pour un ID Token ou d’une API (API Identifier) pour un Access Token." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=audience">audience</Tooltip> avec une [Action](/docs/fr-CA/customize/actions).

L’extensibilité Auth0 est une capacité complète et sophistiquée pour adapter les opérations d’IdAM, mais elle peut aussi exposer votre base de données à une utilisation inappropriée ou non intentionnelle. Les documents de cette section présentent les pratiques exemplaires à l’intention des concepteurs et des responsables de la mise en œuvre; nous vous recommandons de les lire en entier au moins une fois, même si vous avez déjà amorcé votre parcours avec Auth0.
