L’extensibilité permet aux administrateurs d’ajouter une connexion personnalisée dans Auth0 afin de mettre en place des solutions sur mesure de gestion des identités et des accès (IdAM). Chaque type d’extensibilité utilise Node.js et s’exécute sur la plateforme Auth0, dans un tenant Auth0.

Auth0 Extensibility se présente sous plusieurs formes :

* [Actions](/docs/fr-CA/customize/actions) : Les Actions sont des fonctions sécurisées, propres au tenant et versionnées, écrites en Node.js, qui s’exécutent à certains moments sur la plateforme Auth0. Utilisez les Actions pour personnaliser et étendre les capacités d’Auth0 avec une connexion personnalisée.
* [Rules](/docs/fr-CA/customize/rules) : S’exécutent lorsque des artefacts attestant l’authenticité de l’utilisateur sont générés. Par exemple :

  * ID Token dans OpenID Connect (OIDC)
  * Access Token dans OAuth 2.0
  * Une assertion dans SAML
* [Hooks](/docs/fr-CA/customize/hooks) : Offrent une extensibilité supplémentaire lorsqu’il y a échange d’artefacts non liés à l’utilisateur et lors de la création d’identités utilisateur, comme avant l’inscription d’un utilisateur et après son inscription.

Voici quelques meilleures pratiques pour utiliser les produits Auth0 Extensibility :

* Les scripts des connexions de base de données personnalisées et des migrations servent à intégrer un magasin d’identités utilisateur existant ou à prendre en charge la migration automatique d’utilisateurs à partir d’un magasin d’identités indépendant ou existant. Pour en savoir plus sur la configuration, consultez [Create Custom Database Connections.](/docs/fr-CA/authenticate/database-connections/custom-db/create-db-connection)
* Pour éviter les doublons d’utilisateurs, définissez un `user_id` cohérent pour chaque profil utilisateur renvoyé. De plus, assurez-vous de ne pas renvoyer la même adresse courriel pour deux utilisateurs différents dans le script `get_user` ou `login`, et veillez à ce que les deux scripts renvoient le même utilisateur.
* Pour assurer la rétrocompatibilité, `app_metadata` s’appelle `metadata` dans les scripts Custom DB. Comme le transfert de données peut être interrompu, utilisez une Action pour récupérer les métadonnées qui devraient se trouver dans `app_metadata` ou `user_metadata` si elles sont absentes.
* Veillez à restreindre l’accès à l’<Tooltip tip="Audience : Identifiant unique de l’audience pour un jeton émis. Nommée aud dans un jeton, sa valeur contient l’ID soit d’une application (Client ID) pour un ID Token, soit d’une API (API Identifier) pour un Access Token." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=audience">audience</Tooltip> au moyen d’une [Action](/docs/fr-CA/customize/actions).

Auth0 Extensibility est une capacité complète et sophistiquée pour adapter les opérations d’IdAM, mais elle peut aussi exposer votre base de données à une utilisation inadéquate ou non intentionnelle. Les documents de cette section présentent les meilleures pratiques à l’intention des concepteurs comme des responsables de la mise en œuvre; nous vous recommandons de les lire en entier au moins une fois, même si vous avez déjà commencé votre parcours avec Auth0.
