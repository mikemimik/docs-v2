L’Extensibilité permet aux administrateurs d’ajouter une authentification personnalisée dans Auth0 afin de créer des solutions adaptées aux derniers besoins en gestion des identités et des accès (IdAM). Chaque type d’extensibilité utilise Node.js et s’exécute sur la plateforme Auth0 dans un tenant Auth0.

L’Extensibilité Auth0 se présente sous plusieurs formes :

* [Actions](/docs/fr-CA/customize/actions) : Les Actions sont des fonctions sécurisées, propres au tenant et versionnées, écrites en Node.js, qui s’exécutent à certains moments sur la plateforme Auth0. Utilisez les Actions pour personnaliser et étendre les capacités d’Auth0 avec une authentification personnalisée.
* [Rules](/docs/fr-CA/customize/rules) : S’exécutent lorsque des artéfacts liés à l’authentification de l’utilisateur sont générés. Par exemple :

  * ID Token dans OpenID Connect (OIDC)
  * Access Token dans OAuth 2.0
  * Une assertion dans SAML
* [Hooks](/docs/fr-CA/customize/hooks) : Offrent une extensibilité supplémentaire lors de l’échange d’artéfacts non liés à l’utilisateur et lors de la création d’identités utilisateur, comme Pre-User Registration et Post-User Registration.

Voici quelques pratiques exemplaires pour l’utilisation des produits d’Extensibilité Auth0 :

* Les scripts pour les connexions de base de données personnalisées et les migrations servent à intégrer un référentiel d’identités utilisateur existant, ou lorsqu’une migration automatique d’utilisateurs à partir d’un référentiel d’identités indépendant ou existant est requise. Pour en savoir plus sur la configuration, consultez [Créer des connexions de base de données personnalisées.](/docs/fr-CA/authenticate/database-connections/custom-db/create-db-connection)
* Pour éviter les doublons d’utilisateurs, définissez un `user_id` cohérent sur chaque profil utilisateur renvoyé. De plus, assurez-vous de ne pas renvoyer la même adresse courriel pour deux utilisateurs différents dans le script `get_user` ou `login`, et veillez à ce que les deux scripts renvoient le même utilisateur.
* Pour assurer la rétrocompatibilité, `app_metadata` s’appelle `metadata` dans les scripts DB personnalisés. Comme le transfert de données peut être interrompu, utilisez une Action pour récupérer les métadonnées qui devraient se trouver dans `app_metadata` ou `user_metadata` si elles sont absentes.
* Assurez-vous de restreindre l’accès à la <Tooltip tip="Audience : identifiant unique de l’audience d’un jeton émis. Nommée aud dans un jeton, sa valeur contient l’ID d’une application (Client ID) pour un ID Token ou d’une API (API Identifier) pour un Access Token." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=audience">audience</Tooltip> avec une [Action](/docs/fr-CA/customize/actions).

L’Extensibilité Auth0 est une capacité complète et sophistiquée pour adapter les opérations d’IdAM, mais elle peut aussi exposer votre base de données à une utilisation inappropriée ou non intentionnelle. Les documents de cette section présentent des pratiques exemplaires à l’intention des concepteurs comme des équipes de mise en œuvre; nous vous recommandons de les lire en entier au moins une fois, même si vous avez déjà commencé votre parcours avec Auth0.
