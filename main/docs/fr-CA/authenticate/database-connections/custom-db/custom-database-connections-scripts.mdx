L’Extensibilité permet aux administrateurs d’ajouter une connexion personnalisée dans Auth0 afin de créer des solutions de dernier kilomètre pour la gestion des identités et des accès (IdAM). Chaque type d’extensibilité utilise Node.js et s’exécute sur la plateforme Auth0 dans un tenant Auth0.

L’Extensibilité Auth0 se décline sous plusieurs formes :

* [Actions](/docs/fr-CA/customize/actions) : Les Actions sont des fonctions sécurisées, propres au tenant et versionnées, écrites en Node.js, qui s’exécutent à certains moments sur la plateforme Auth0. Utilisez les actions pour personnaliser et étendre les capacités d’Auth0 avec une connexion personnalisée.
* [Rules](/docs/fr-CA/customize/rules) : S’exécutent lorsque des artefacts liés à l’authenticité de l’utilisateur sont générés. Par exemple :

  * jeton d’identité dans OpenID Connect (OIDC)
  * jeton d’accès dans OAuth 2.0
  * une assertion dans SAML
* [Hooks](/docs/fr-CA/customize/hooks) : Offrent une extensibilité supplémentaire lorsqu’il y a échange d’artefacts non liés à l’utilisateur et lors de la création d’identités utilisateur, par exemple avant et après l’inscription d’un utilisateur.

Voici quelques pratiques exemplaires pour utiliser les produits d’Extensibilité Auth0 :

* Les scripts des connexions de base de données personnalisées et des migrations servent à intégrer un magasin d’identités utilisateur existant, ou lorsqu’une migration automatique des utilisateurs à partir d’un magasin d’identités indépendant ou hérité est nécessaire. Pour en savoir plus sur la configuration, consultez [Create Custom Database Connections.](/docs/fr-CA/authenticate/database-connections/custom-db/create-db-connection)
* Pour éviter les doublons d’utilisateurs, définissez un `user_id` cohérent dans chaque profil utilisateur renvoyé. De plus, assurez-vous de ne pas renvoyer la même adresse courriel pour deux utilisateurs différents dans le script `get_user` ou `login`, et veillez à ce que les deux scripts renvoient le même utilisateur.
* Pour assurer la rétrocompatibilité, `app_metadata` s’appelle `metadata` dans les scripts de base de données personnalisés. Comme le transfert de données peut être interrompu, utilisez une Action pour récupérer les métadonnées qui devraient se trouver dans `app_metadata` ou `user_metadata` si elles sont absentes.
* Veillez à restreindre l’accès à l’<Tooltip tip="Audience : identifiant unique de l'audience d'un jeton émis. Nommée aud dans un jeton, sa valeur contient l'ID soit d'une application (ID client) pour un jeton d'identité, soit d'une API (identifiant d'API) pour un jeton d'accès." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=audience">audience</Tooltip> au moyen d’une [Action](/docs/fr-CA/customize/actions).

L’Extensibilité Auth0 est une fonctionnalité complète et sophistiquée qui permet d’adapter les opérations d’IdAM, mais elle peut aussi exposer votre base de données à une utilisation inappropriée ou involontaire. Les documents de cette section présentent des pratiques exemplaires à l’intention des concepteurs comme des responsables de la mise en œuvre; nous vous recommandons de les lire en entier au moins une fois, même si vous avez déjà commencé votre parcours avec Auth0.
