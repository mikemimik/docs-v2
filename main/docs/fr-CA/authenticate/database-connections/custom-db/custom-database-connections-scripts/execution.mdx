title: Meilleures pratiques pour l’exécution des scripts d’action de connexion à une base de données personnalisée
---

Un type de connexion à une base de données personnalisée vous permet de configurer des scripts d’action, qui contiennent du code personnalisé qu’Auth0 utilise pour interagir avec votre référentiel d’identités existant. Les scripts d’action prennent la forme d’une fonction JavaScript nommée qui accepte un ensemble prédéfini de paramètres.

<div id="webtask-containers">
  ## Conteneurs Webtask
</div>

Les scripts d’action s’exécutent dans un conteneur Webtask individuel et sont limités à environ 20 secondes d’exécution. Une fois l’exécution des fonctions terminée, le conteneur est recyclé.

Lorsqu’un conteneur est recyclé, il interrompt les opérations du script d’action qui sont encore en cours. Cela peut entraîner le renvoi d’une erreur et possiblement la réinitialisation de l’objet `global`. Pour en savoir plus sur l’objet `global`, consultez [Meilleures pratiques pour l’environnement des scripts d’action de base de données personnalisée](/docs/fr-CA/authenticate/database-connections/custom-db/custom-database-connections-scripts/environment).

<div id="asynchronous-features">
  ## Fonctionnalités asynchrones
</div>

Les fonctionnalités asynchrones de Javascript, y compris les objets [Promise](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/Promise) et les [async functions](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Statements/async_function), sont prises en charge dans les scripts d’action.

Vous pouvez utiliser des fonctionnalités asynchrones pour exécuter des opérations non bloquantes dans un script d’action, mais assurez-vous qu&#39;elles ne dépassent pas la limite d&#39;exécution du conteneur Webtask. Lorsque le conteneur Webtask est recyclé, il met fin à toute opération en attente, ce qui peut entraîner un comportement inattendu ou une erreur.

Si vous effectuez un appel à un service externe ou à une API dans votre script d’action, configurez la fonction de façon à ce qu&#39;elle expire après un délai raisonnable, puis [retournez une erreur](#error-handling) si le service externe ou l&#39;API est inaccessible.

<div id="examples">
  ### Exemples
</div>

Auth0 prend en charge les objets [Promise](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/Promise) et les [fonctions async](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Statements/async_function) dans les scripts d’action.

<div id="promise-object">
  #### Objet Promise
</div>

Cet exemple utilise la méthode `fetch` intégrée à JavaScript, qui récupère une ressource sur un réseau puis renvoie un objet Promise. Il est rédigé à l’aide du [chaînage de promesses](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Guide/Using_promises#chaining).

```javascript lines expandable
function login(userNameOrEmail, password, callback) {
	const hashedPassword = hash(password);
	const apiEndpoint = 'https://example.com/api/authenticate';
	const options = {
		method: 'POST',
		body: {
			email: userNameOrEmail,
			password: hashedPassword
		}
	};

	fetch(apiEndpoint, options) 
		.then((response) => {
			if (!response.ok) {
				return callback(new Error(`HTTP error! Status: ${response.status}`));
			}

			return response.json();
		})
		.then((response) => {
			if (response.err) {
				return callback(new Error(`Error authenticating user: ${err}`));
			}

			let profile = {
				email: response.profileData.email,
				username: response.profileData.username
			};

			return callback(null, profile);
		})
		.catch((err) => {
			return callback(new Error(`An error occurred: ${err}`));
		});
  }
```

<div id="async-function">
  #### Fonction async
</div>

Cet exemple, rédigé à l’aide de [fonctions async](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Statements/async_function), utilise la méthode `fetch` intégrée à JavaScript, qui récupère une ressource sur le réseau puis renvoie un objet Promise.

```js lines expandable
async function login(userNameOrEmail, password, callback) {
	const hashedPassword = hash(password);
	const apiEndpoint = 'https://example.com/api/authenticate';
	const options = {
		method: 'POST',
		body: {
			email: userNameOrEmail,
			password: hashedPassword
		}
	};

	const response = await fetch(apiEndpoint, options);

	if (!response.ok) {
		return callback(new Error(`HTTP error! Status: ${response.status}`));
	}

	const result = response.json();

	if (result.err) {
		return callback(new Error(`Error authenticating user: ${err}`));
	}

	let profile = {
		email: response.profileData.email,
		username: response.profileData.username
	};

	return callback(null, profile);
}
```

<div id="callback-function">
  ## Fonction `callback`
</div>

La fonction `callback` indique que l’opération du script d’action est terminée et doit être appelée exactement une fois. Un script d’action doit se terminer immédiatement après un appel à la fonction `callback`, de préférence en utilisant explicitement l’instruction `return`.

<Warning>
  Si la fonction `callback` est appelée plus d’une fois, cela peut entraîner des résultats imprévus ou des erreurs.

  Si la fonction `callback` n’est jamais appelée, l’exécution du script d’action restera bloquée, et une erreur sera renvoyée lorsque le conteneur Webtask sera recyclé.
</Warning>

<div id="asynchronous-processing">
  ### Traitement asynchrone
</div>

Si un script d’action effectue un traitement asynchrone, la fonction `callback` doit être appelée une fois que toutes les opérations asynchrones sont terminées.

<div id="parameters">
  ### Paramètres
</div>

Si la fonction `callback` est appelée sans paramètre, elle s’exécute comme si un paramètre `null` avait été fourni.

<div id="size">
  ## Taille
</div>

La taille totale de l’implémentation de tout script d’action ne doit pas dépasser 100 kB. Cette limite exclut les modules `npm` importés. Pour en savoir plus sur les modules `npm`, consultez [Meilleures pratiques pour l’environnement des scripts d’action de base de données personnalisée](/docs/fr-CA/authenticate/database-connections/custom-db/custom-database-connections-scripts/environment).

Plus un script est volumineux, plus la latence augmente en raison du processus d’empaquetage et de transport utilisé par la plateforme Webtask. La taille a une incidence sur les performances du système.

<div id="anonymous-functions">
  ## Fonctions anonymes
</div>

Les scripts d’action peuvent être implémentés sous forme de fonctions anonymes, mais ce n’est pas recommandé. Les fonctions anonymes compliquent le débogage du script d’action et l’interprétation de la pile d’appels générée lorsqu’une erreur exceptionnelle se produit. Pour en savoir plus sur les fonctions anonymes, consultez [IIFE on MDN Web Docs](https://developer.mozilla.org/en-US/docs/Glossary/IIFE).

<div id="error-handling">
  ## Gestion des erreurs
</div>

Transmettez un objet `Error` à la fonction `callback` avec un message d’erreur descriptif :

```js lines
return callback(new Error('My custom error message'));
```

<div id="security">
  ## Sécurité
</div>

<div id="database-interface-vs-api">
  ### Interface de base de données ou API
</div>

Assurez-vous de sécuriser toutes les communications entre Auth0 et votre référentiel d’identités existant. Si votre référentiel d’identités existant n’a pas déjà d’API, il est fortement recommandé d’en mettre une en place.

Si votre référentiel d’identités existant dispose d’une API, vous pouvez [enregistrer l’API](/docs/fr-CA/get-started/auth0-overview/set-up-apis) dans Auth0 et [créer une Action](/docs/fr-CA/manage-users/access-control/sample-use-cases-actions-with-authorization#deny-access-to-anyone-calling-an-api) afin de restreindre l’accès des utilisateurs finaux.

Si votre référentiel d’identités existant ne dispose pas d’API — et qu’il n’est pas possible d’en mettre une en place — vous pouvez quand même communiquer directement avec votre base de données. Assurez-vous [d’ajouter les adresses IP d’Auth0 à la liste d’autorisation de votre pare-feu](/docs/fr-CA/secure/security-guidance/data-security/allowlist) afin d’autoriser le trafic entrant provenant d’Auth0.

<div id="identity-provider-tokens">
  ## Jetons du fournisseur d’identité
</div>

Si l’objet `user` renvoie les propriétés `access_token` et `refresh_token`, Auth0 les traite différemment des autres types d’informations sur l’utilisateur. Auth0 les stocke dans la propriété `identities` de l’objet `user` :

```json lines
{
	"email": "you@example.com",
	"updated_at": "2019-03-15T15:56:44.577Z",
	"user_id": "auth0|some_unique_id",
	"nickname": "a_nick_name",
	"identities": [ 
		{
			"user_id": "some_unique_id",
			"access_token": "e1b5.................92ba",
			"refresh_token": "a90c.................620b",
			"provider": "auth0", 
			"connection": "custom_db_name",
			"isSocial": false 
		}
  ], 
  "created_at": "2019-03-15T15:56:44.577Z",
  "last_ip": "192.168.1.1",
  "last_login": "2019-03-15T15:56:44.576Z",
  "logins_count": 3
}
```

Si vous souhaitez récupérer l’une ou l’autre de ces propriétés au moyen de l’Auth0 <Tooltip tip="Management API : Un produit permettant aux clients d’effectuer des tâches administratives." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Management+API">Management API</Tooltip>, incluez la portée `read:user_idp_tokens` lorsque vous [demandez un jeton d’accès](/docs/fr-CA/secure/tokens/access-tokens/management-api-access-tokens/get-management-api-access-tokens-for-production).
