---

Vous utilisez généralement une connexion de base de données personnalisée pour donner accès à votre propre magasin d’identités hérité aux fins de l’authentification (parfois appelée **authentification héritée**) ou pour [effectuer l’importation d’utilisateurs par migration automatique](/docs/fr-CA/manage-users/user-migration/configure-automatic-migration-from-your-database) (souvent appelée migration **progressive** ou **paresseuse**). Vous pouvez aussi utiliser des connexions de base de données personnalisées pour servir de proxy d’accès à un tenant Auth0 dans les scénarios où vous utilisez une architecture Auth0 multilocataire. Pour en savoir plus, consultez [Pratiques exemplaires pour les applications multilocataires](/docs/fr-CA/get-started/auth0-overview/create-tenants/multi-tenant-apps-best-practices).

Vous créez et configurez généralement des connexions de base de données personnalisées à l’aide du <Tooltip tip="Auth0 Dashboard : le principal produit d’Auth0 pour configurer vos services." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip>. Vous créez une connexion de base de données, puis activez la bascule **Use my own database** pour permettre la modification des scripts d’action de base de données. Une connexion de base de données personnalisée peut aussi être créée et configurée à l’aide du <Tooltip tip="Management API : un produit qui permet aux clients d’effectuer des tâches administratives." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Management+API">Management API</Tooltip> d’Auth0, du [endpoint Create a connection](https://auth0.com/docs/api/management/v2#!/Connections/post_connections) et de la stratégie `auth0`.

En plus des éléments communs à tous les types de connexion de base de données, une connexion de base de données personnalisée vous permet de configurer des scripts d’action — du code personnalisé utilisé pour interagir avec des magasins d’identités hérités. Les scripts que vous choisissez de configurer dépendent de l’objectif de la connexion : authentification héritée ou migration automatique.

Les scripts d’action peuvent être implémentés sous forme de fonctions anonymes; toutefois, les fonctions anonymes rendent le débogage difficile lorsqu’il faut interpréter la pile d’appels générée à la suite d’erreurs exceptionnelles. Par souci de simplicité, nous vous recommandons d’attribuer un nom de fonction à chaque script d’action. Pour voir quelques noms recommandés, consultez [Pratiques exemplaires relatives à l’exécution des scripts d’action de base de données personnalisée](/docs/fr-CA/authenticate/database-connections/custom-db/custom-database-connections-scripts/execution).

Dans un scénario d’authentification héritée, aucun nouvel enregistrement d’utilisateur n’est créé; l’utilisateur demeure dans le magasin d’identités hérité et Auth0 utilise l’identité qui s’y trouve lorsqu’il authentifie l’utilisateur. Les connexions de base de données personnalisées sont aussi utilisées en dehors du flux de <Tooltip tip="Universal Login : votre application redirige vers Universal Login, hébergé sur l’Authorization Server d’Auth0, afin de vérifier l’identité d’un utilisateur." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Universal+Login">Universal Login</Tooltip>. Par exemple, le script d’action `changePassword` d’une connexion est appelé lorsqu’une opération de changement de mot de passe survient pour un utilisateur qui se trouve dans un magasin d’identités hérité.

<div id="automatic-migration">
  ## Migration automatique
</div>

Pendant la migration automatique, Auth0 crée un nouvel utilisateur dans un magasin d’identités (base de données) géré par Auth0. Auth0 utilise l’identité du magasin d’identités géré par Auth0 pour authentifier l’utilisateur. Pour que cela se produise, Auth0 exige d’abord que l’utilisateur soit authentifié auprès du magasin d’identités hérité, et ce n’est qu’en cas de réussite que le nouvel utilisateur sera créé dans la base de données gérée par Auth0. Auth0 crée le nouvel utilisateur en utilisant le même ID et le même mot de passe que ceux fournis pendant l’authentification.

Dans un scénario de migration automatique, la création de l’utilisateur a généralement lieu après l’exécution du script d’action Login. Nous vous recommandons de ne pas tenter de supprimer des utilisateurs du magasin d’identités hérité au moyen d’une opération en ligne dans le script Login, mais plutôt dans le cadre d’un processus distinct. Cela permet d’éviter la suppression accidentelle d’utilisateurs si une erreur survient pendant la migration.

Avec la migration automatique, les utilisateurs restent dans le magasin d’identités hérité et peuvent être supprimés ou archivés au besoin. Un effet secondaire peut se produire lorsqu’un utilisateur est supprimé d’Auth0, mais demeure dans l’ancien magasin de données. Dans ce cas, une connexion effectuée par l’utilisateur supprimé pourrait entraîner l’exécution du script Login ou Get User, et l’utilisateur serait de nouveau migré à partir du magasin d’identités hérité.

Nous vous recommandons de marquer les identités des utilisateurs de l’ancien magasin comme migrées avant la fin de l’exécution des scripts Login ou Get User, et avant toute suppression éventuelle dans l’ancien magasin, afin d’empêcher la recréation involontaire d’utilisateurs supprimés intentionnellement.

<div id="size">
  ## Taille
</div>

Nous recommandons que la taille totale d’un script d’action ne dépasse pas 100 KB. Plus cette taille est importante, plus la latence causée par le processus d’empaquetage et de transport utilisé par la plateforme Auth0 augmente, ce qui aura une incidence sur les performances de votre système. Notez que la limite de 100 KB n’inclut pas les modules `npm` qui peuvent être appelés dans des instructions `require`.
