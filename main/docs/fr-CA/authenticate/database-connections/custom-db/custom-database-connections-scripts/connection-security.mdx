<div id="access-legacy-identity-storage-via-custom-api">
  ## Accéder au stockage d’identités hérité au moyen d’une API personnalisée
</div>

Nous vous recommandons de mettre en place une API afin d’appliquer le principe du moindre privilège à votre stockage d’identités hérité, plutôt que d’ouvrir simplement un accès général par Internet.

Protéger le stockage d’identités hérité contre tout accès général est une pratique exemplaire recommandée. Par exemple, exposer directement à Internet une base de données (d’identités héritées) peut être extrêmement problématique : les interfaces de base de données pour SQL et autres technologies du genre offrent des fonctionnalités très étendues, ce qui va à l’encontre du principe du moindre privilège en matière de sécurité.

La solution consiste à créer une API (personnalisée) simple — protégée au moyen d’un <Tooltip tip="Jeton d’accès : justificatif d’autorisation, sous la forme d’une chaîne opaque ou d’un JWT, utilisé pour accéder à une API." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=access+token">jeton d’accès</Tooltip> — que chaque script d’action peut appeler. Cette API servirait alors d’interface vers le magasin d’identités hérité. Le flux Client Credentials Grant peut ensuite être utilisé pour obtenir un jeton d’accès à partir d’un script; ce jeton peut ensuite être mis en cache dans l’objet `global` afin d’être réutilisé et d’améliorer les performances. L’API peut alors fournir un nombre limité de points de terminaison protégés qui n’exécutent que les fonctions de gestion (des identités) héritées requises (p. ex., lire un utilisateur, changer le mot de passe).

Par défaut, Auth0 vous remettra un jeton pour n’importe quelle API si vous vous authentifiez avec succès et incluez l’<Tooltip tip="Audience : identifiant unique de l’audience d’un jeton émis. Nommée aud dans un jeton, sa valeur contient l’ID soit d’une application (Client ID) pour un ID Token, soit d’une API (API Identifier) pour un Access Token." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=audience">audience</Tooltip> appropriée. Restreindre l’accès à l’API du magasin d’identités hérité en limitant l’attribution des jetons d’accès au moyen d’une rule empêchera toute utilisation non autorisée et atténuera plusieurs scénarios de vecteurs d’attaque, notamment lorsqu’une redirection vers `/authorize` est interceptée et que l’audience de l’API est ajoutée.

<Warning>
  Restreindre l’accès à l’API au moyen d’une Rule atténuera les scénarios de vecteurs d’attaque, par exemple lorsqu’une redirection vers `/authorize` est interceptée et que l’audience de l’API est ajoutée, et garantira que seul l’accès au moyen de justificatifs du client précis est accordé.
</Warning>

<div id="access-to-legacy-identity-storage">
  ## Accès au stockage d’identités hérité
</div>

Que vous gériez l’accès au stockage d’identités hérité au moyen d’une API personnalisée ou de l’interface native fournie, vous devriez limiter cet accès à la liste d’adresses IP associées à votre tenant Auth0. Pour consulter la liste des adresses IP, reportez-vous à [Adresses IP Auth0 pour les AllowLists](/docs/fr-CA/secure/security-guidance/data-security/allowlist). L’ajout d’URL à l’AllowList limite l’accès au stockage d’identités hérité et garantit que seuls les scripts d’actions de base de données personnalisée définis dans Auth0 sont autorisés.

<Warning>
  L’AllowList d’adresses IP Auth0 est partagée entre tous les tenants Auth0 définis dans une région. N’utilisez jamais la liste d’autorisation comme seule méthode pour sécuriser l’accès à votre stockage d’identités hérité; cela pourrait entraîner des vulnérabilités de sécurité permettant un accès non autorisé à vos utilisateurs.
</Warning>
