<div id="access-legacy-identity-storage-via-custom-api">
  ## Accéder au stockage d’identités hérité au moyen d’une API personnalisée
</div>

Nous vous recommandons de mettre en place une API afin d’appliquer le principe du moindre privilège à votre stockage d’identités hérité, plutôt que d’ouvrir simplement un accès général par Internet.

Protéger le stockage d’identités hérité contre tout accès général est une bonne pratique recommandée. Par exemple, exposer directement à Internet une base de données (d’identités héritée) peut être extrêmement problématique : les interfaces de bases de données SQL et autres offrent des fonctionnalités très étendues, ce qui va à l’encontre du principe du moindre privilège en matière de sécurité.

L’autre solution consiste à créer une API simple (personnalisée) — protégée par l’utilisation d’un <Tooltip tip="Jeton d’accès : information d’autorisation, sous la forme d’une chaîne opaque ou d’un JWT, utilisée pour accéder à une API." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=access+token">jeton d’accès</Tooltip> — que chaque script d’action peut appeler. Cette API servirait d’interface avec le stockage d’identités hérité. Vous pouvez ensuite utiliser le flux Client Credentials Grant Flow pour obtenir un jeton d’accès à partir d’un script, puis le mettre en cache dans l’objet `global` afin de le réutiliser et d’améliorer les performances. L’API peut alors fournir un nombre limité de points de terminaison protégés qui n’exécutent que les fonctions de gestion des identités héritées requises (p. ex., lire un utilisateur, changer le mot de passe).

Par défaut, Auth0 vous donnera un jeton pour n’importe quelle API si vous vous authentifiez avec succès et incluez l’<Tooltip tip="Audience : identifiant unique de l’audience d’un jeton émis. Nommée aud dans un jeton, sa valeur contient l’ID d’une application (Client ID) pour un ID Token ou d’une API (API Identifier) pour un Access Token." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=audience">audience</Tooltip> appropriée. Restreindre l’accès à l’API du stockage d’identités hérité en limitant l’attribution des jetons d’accès au moyen d’une Rule empêchera toute utilisation non autorisée et atténuera plusieurs scénarios de vecteurs d’attaque, par exemple lorsqu’une redirection vers `/authorize` est interceptée et que l’audience de l’API est ajoutée.

<Warning>
  Restreindre l’accès à l’API au moyen d’une Rule atténuera les scénarios de vecteurs d’attaque, par exemple lorsqu’une redirection vers `/authorize` est interceptée et que l’audience de l’API est ajoutée, et garantira que seul l’accès au moyen d’informations d’identification client précises est accordé.
</Warning>

<div id="access-to-legacy-identity-storage">
  ## Accès au stockage d’identités hérité
</div>

Que vous gériez l’accès au stockage d’identités hérité au moyen d’une API personnalisée ou de l’interface native fournie, vous devriez restreindre l’accès à la liste des adresses IP associées à votre tenant Auth0. Pour consulter la liste des adresses IP, voyez [Adresses IP d’Auth0 pour les AllowLists](/docs/fr-CA/secure/security-guidance/data-security/allowlist). L’ajout d’URL à l’AllowList limite l’accès au stockage d’identités hérité et garantit que seuls les scripts d’action de base de données personnalisée définis dans Auth0 sont autorisés.

<Warning>
  L’AllowList d’adresses IP d’Auth0 est partagée entre tous les tenants Auth0 définis dans une région. N’utilisez jamais l’allow list comme seule méthode pour sécuriser l’accès à votre stockage d’identités hérité; cela pourrait créer des failles de sécurité permettant un accès non autorisé à vos utilisateurs.
</Warning>
