Les scripts d’action s’exécutent comme une série de fonctions Javascript appelées dans une instance d’un conteneur Webtask serverless. Dans ce contexte, un environnement précis est fourni, ainsi qu’un certain nombre d’artefacts provenant à la fois du conteneur Webtask et du serveur <Tooltip tip="Serveur d’authentification : serveur qui confirme ou refuse l’identité d’un utilisateur." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=authentication+server">d’authentification</Tooltip> Auth0 (votre tenant Auth0) lui-même.

<div id="npm-modules">
  ## Modules npm
</div>

Les conteneurs Webtask serverless d’Auth0 peuvent utiliser une vaste gamme de modules [`npm`](https://www.npmjs.com/) ; les modules `npm` réduisent non seulement la taille globale du code des `scripts d’action`, mais donnent aussi accès à un large éventail de fonctionnalités préconstruites.

Par défaut, un grand nombre de modules `npm` accessibles au public sont pris en charge d’emblée. Cette liste a été compilée et vérifiée afin de repérer tout risque de sécurité potentiel. Pour voir quels modules `npm` sont pris en charge, consultez [Can I require: Auth0 Extensibility](https://auth0-extensions.github.io/canirequire/).

Si vous avez besoin d’un module `npm` qui n’est pas pris en charge d’emblée, vous pouvez faire une demande au moyen du [portail Auth0 Extensibility](https://auth0-extensions.github.io/canirequire/) ou par l’entremise de votre représentant Auth0. Auth0 évaluera votre demande afin d’en déterminer la pertinence; si vous devez faire suivre une demande non résolue, ouvrez un ticket de soutien au moyen du [portail de soutien Auth0](https://support.auth0.com/). À l’heure actuelle, Auth0 ne prend pas en charge l’utilisation de modules `npm` provenant de dépôts privés.

<div id="variables">
  ## Variables
</div>

Les scripts d’action Auth0 prennent en charge les variables d’environnement, auxquelles on accède au moyen de l’objet `configuration`, disponible globalement. L’objet `configuration` doit être traité comme étant en lecture seule et devrait servir à stocker des renseignements sensibles, comme des données d’authentification ou des clés API permettant d’accéder à des référentiels d’identité externes. Cela réduit le risque d’avoir des valeurs sensibles codées en dur dans un script d’action.

L’objet `configuration` peut aussi servir à appliquer les stratégies de bonnes pratiques du cycle de vie du développement logiciel (SDLC) que vous utilisez, comme la [configuration de plusieurs environnements](/docs/fr-CA/get-started/auth0-overview/create-tenants/set-up-multiple-environments), en vous permettant de définir des variables dont les valeurs sont propres au tenant. Cela réduit le recours à des valeurs codées en dur dans un script d’action, lesquelles peuvent varier selon le tenant qui l’exécute.

<div id="global-object">
  ## objet global
</div>

Les conteneurs Webtask serverless d’Auth0 sont provisionnés à partir d’un pool associé à chaque tenant Auth0. Chaque instance de conteneur met à disposition un objet global, accessible par tous les scripts d’action qui s’y exécutent (dans cette instance de conteneur). L’objet global agit comme une variable globale propre au conteneur et peut servir à définir des informations — ou même des fonctions — pouvant être utilisées par tous les scripts d’action qui s’y exécutent (dans cette instance de conteneur).

Cela signifie que l’objet global peut servir de cache pour des ressources coûteuses, à condition que ces ressources ne soient pas propres à un utilisateur. Par exemple, un <Tooltip tip="Access Token : information d’autorisation d’accès, sous la forme d’une chaîne opaque ou d’un JWT, utilisée pour accéder à une API." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Access+Token">Access Token</Tooltip> pour une API tierce (p. ex. une API de logging) offrant des fonctionnalités non propres à l’utilisateur pourrait y être stocké. Il peut aussi servir à stocker un Access Token pour votre propre API non propre à l’utilisateur, définie dans Auth0 et obtenue au moyen du Client Credentials flow.

<Warning>
  Un script d’action peut s’exécuter dans n’importe quelle instance de conteneur déjà en cours d’exécution ou dans une nouvelle instance de conteneur (qui pourra ensuite être ajoutée au pool). Il n’existe aucune affinité de conteneur pour l’exécution des scripts d’action dans Auth0. Cela signifie que vous devriez éviter de stocker dans l’objet global toute information propre à un utilisateur et toujours vous assurer que toute déclaration faite dans l’objet global prévoit également une initialisation.
</Warning>

Chaque fois qu’un conteneur Webtask est recyclé, ou lors de l’instanciation d’un nouveau conteneur Webtask, l’objet global qu’il définit est réinitialisé. Ainsi, toute déclaration d’assignation dans l’objet global associé à un conteneur devrait aussi inclure une initialisation. Pour offrir une certaine souplesse sur le plan des performances, les conteneurs Webtask serverless sont provisionnés dans Auth0 de façon ponctuelle et sont aussi assujettis à diverses politiques de recyclage. En général, nous recommandons de ne pas considérer qu’un objet global a une durée de vie de plus de 20 minutes.

<div id="custom-database-connection-environment-checklist">
  ## Liste de vérification de l’environnement d’une connexion de base de données personnalisée
</div>

* Assurez-vous que votre base de données contient les champs appropriés pour stocker les attributs du profil utilisateur, comme **id**, **nickname**, **email** et **password**. Pour en savoir plus sur le schéma de profil utilisateur d’Auth0 et les champs attendus, consultez [Profils d’utilisateur normalisés](/docs/fr-CA/manage-users/user-accounts/user-profiles/normalized-user-profiles). Pour savoir comment mettre à jour les profils utilisateur, consultez [Mettre à jour des profils utilisateur à l’aide de votre base de données](/docs/fr-CA/manage-users/user-accounts/user-profiles/update-user-profiles-using-your-database).
* Vous pouvez utiliser les erreurs renvoyées par votre connexion de base de données personnalisée à des fins de dépannage.
* La propriété `id` (ou `user_id`) du profil utilisateur renvoyé sera utilisée par Auth0 pour identifier l’utilisateur. Si vous utilisez plusieurs connexions de base de données personnalisées, la valeur **id** **doit être unique dans l’ensemble des connexions de base de données personnalisées** afin d’éviter les collisions d’**ID utilisateur**. Nous vous recommandons de faire précéder la valeur de **id** du nom de la connexion (sans espaces). Pour en savoir plus sur les ID utilisateur, consultez [Identifier les utilisateurs](/docs/fr-CA/manage-users/user-accounts/identify-users).
* La latence sera plus élevée que pour les magasins d’utilisateurs hébergés par Auth0.
* La base de données ou le service doit être accessible depuis les serveurs Auth0. Vous devrez configurer des connexions entrantes si votre magasin se trouve derrière un pare-feu.

<div id="testing-a-specific-runtime-version">
  ## Tester une version précise du runtime
</div>

La version du runtime des scripts Database personnalisés est définie dans **Dashboard &gt; Settings &gt; Advanced**, dans la section **Extensibility**.

Suivez ces étapes pour tester un script de base de données personnalisé précis avec une version particulière du runtime :

1. Accédez à **Authentication &gt; Database.**
2. Sélectionnez la connexion Database dans laquelle vous avez défini un script personnalisé.
3. Dans la page de la connexion de base de données sélectionnée, sélectionnez l’onglet **Custom Database**.
4. Faites défiler la page jusqu’à la section **Database Action Scripts**.
5. Sélectionnez le script précis que vous voulez vérifier (p. ex. Login, Create, etc.) dans l’onglet correspondant.
6. Sélectionnez **Save and Try**. Une fenêtre modale s’ouvre avec des paramètres de test précis et des détails contextuels d’exemple. Mettez-les à jour au besoin.
7. Sélectionnez **Try** dans la fenêtre modale pour ouvrir une liste déroulante permettant de choisir une version précise de Node.
8. Le test se déclenche lorsque vous sélectionnez la version de Node souhaitée, et les résultats s’affichent dans un message sur le même écran.
