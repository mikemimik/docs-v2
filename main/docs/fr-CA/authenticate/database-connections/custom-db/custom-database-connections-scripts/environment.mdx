Les scripts d’action s’exécutent comme une série de fonctions JavaScript appelées dans une instance d’un conteneur Webtask serverless. Dans ce cadre, un environnement précis est fourni, ainsi qu’un certain nombre d’artefacts provenant à la fois du conteneur Webtask et du <Tooltip tip="Serveur d’authentification : serveur qui confirme ou refuse l’identité d’un utilisateur." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=authentication+server">serveur d’authentification</Tooltip> Auth0 lui-même (votre tenant Auth0).

<div id="npm-modules">
  ## modules npm
</div>

Les conteneurs Webtask serverless Auth0 peuvent utiliser un large éventail de modules [`npm`](https://www.npmjs.com/) ; les modules `npm` réduisent non seulement la taille globale du code des scripts d’action, mais donnent aussi accès à un large éventail de fonctionnalités prêtes à l’emploi.

Par défaut, une longue liste de modules `npm` accessibles au public sont pris en charge d’emblée. Cette liste a été compilée et examinée afin de repérer tout risque potentiel en matière de sécurité. Pour voir quels modules `npm` sont pris en charge, consultez [Can I require: Auth0 Extensibility](https://auth0-extensions.github.io/canirequire/).

Si vous avez besoin d’un module `npm` qui n’est pas pris en charge d’emblée, vous pouvez faire une demande au moyen du [portail Auth0 Extensibility](https://auth0-extensions.github.io/canirequire/) ou par l’intermédiaire de votre représentant Auth0. Auth0 évaluera votre demande afin d’en déterminer la pertinence ; si vous devez faire remonter une demande non résolue, ouvrez un ticket de soutien au moyen du [portail de soutien Auth0](https://support.auth0.com/). À l’heure actuelle, Auth0 n’offre aucun soutien pour l’utilisation de modules `npm` provenant de dépôts privés.

<div id="variables">
  ## Variables
</div>

Les scripts d’action Auth0 prennent en charge les variables d’environnement, auxquelles on accède au moyen de l’objet `configuration`, accessible globalement. L’objet `configuration` doit être considéré comme étant en lecture seule et utilisé pour stocker des renseignements sensibles, comme des identifiants ou des clés API permettant d’accéder à des répertoires d’identités externes. Cela évite d’avoir des valeurs sensibles pour la sécurité codées en dur dans un script d’action.

L’objet `configuration` peut aussi servir à appliquer les stratégies de bonnes pratiques du cycle de vie du développement logiciel (SDLC) que vous utilisez, comme [configurer plusieurs environnements](/docs/fr-CA/get-started/auth0-overview/create-tenants/set-up-multiple-environments), en vous permettant de définir des variables dont les valeurs sont propres à chaque tenant. Cela évite d’avoir des valeurs codées en dur dans un script d’action, lesquelles peuvent changer selon le tenant qui l’exécute.

<div id="global-object">
  ## objet global
</div>

Les conteneurs Webtask serverless d’Auth0 sont provisionnés à partir d’un pool associé à chaque tenant Auth0. Chaque instance de conteneur met à disposition un objet global, accessible depuis tous les scripts d’action qui s’y exécutent. L’objet global agit comme une variable globale propre au conteneur et peut servir à définir des informations — ou même des fonctions — utilisables dans tous les scripts d’action qui s’y exécutent.

Cela signifie que l’objet global peut servir à mettre en cache des ressources coûteuses, tant que ces ressources ne sont pas propres à un utilisateur. Par exemple, un <Tooltip tip="Jeton d’accès : information d’autorisation, sous la forme d’une chaîne opaque ou d’un JWT, utilisée pour accéder à une API." cta="Consulter le glossaire" href="/docs/fr-CA/glossary?term=Access+Token">jeton d’accès</Tooltip> pour une API tierce (p. ex., de journalisation) qui fournit des fonctionnalités non propres à un utilisateur pourrait y être stocké. Il pourrait aussi servir à stocker un jeton d’accès pour votre propre API, non propre à un utilisateur, définie dans Auth0 et obtenu au moyen du flux Client Credentials Flow.

<Warning>
  Un script d’action peut s’exécuter dans n’importe laquelle des instances de conteneur déjà en cours d’exécution ou dans une nouvelle instance de conteneur (qui pourra ensuite être ajoutée au pool). Dans Auth0, l’exécution des scripts d’action n’est liée à aucun conteneur en particulier. Cela signifie que vous devriez éviter de stocker dans l’objet global toute information propre à un utilisateur et toujours vous assurer que toute déclaration faite dans l’objet global prévoit aussi une initialisation.
</Warning>

Chaque fois qu’un conteneur Webtask est recyclé, ou qu’un nouveau conteneur Webtask est créé, l’objet global qu’il définit est réinitialisé. Ainsi, toute affectation dans l’objet global associé à un conteneur devrait aussi prévoir une initialisation. Pour offrir une plus grande flexibilité de performance, les conteneurs Webtask serverless sont provisionnés dans Auth0 de façon ponctuelle et sont aussi assujettis à diverses politiques de recyclage. En général, nous recommandons de ne pas considérer que la durée de vie d’un objet global dépasse 20 minutes.

<div id="custom-database-connection-environment-checklist">
  ## Liste de vérification de l&#39;environnement d&#39;une connexion de base de données personnalisée
</div>

* Assurez-vous que votre base de données comporte les champs appropriés pour stocker les attributs du profil utilisateur, comme **id**, **nickname**, **email** et **password**. Pour en savoir plus sur le schéma de profil utilisateur d&#39;Auth0 et les champs attendus, consultez [Profils utilisateur normalisés](/docs/fr-CA/manage-users/user-accounts/user-profiles/normalized-user-profiles). Pour savoir comment mettre à jour les profils utilisateur, consultez [Mettre à jour les profils utilisateur à l&#39;aide de votre base de données](/docs/fr-CA/manage-users/user-accounts/user-profiles/update-user-profiles-using-your-database).
* Vous pouvez utiliser les erreurs renvoyées par votre connexion de base de données personnalisée à des fins de dépannage.
* La propriété `id` (ou, à défaut, `user_id`) dans le profil utilisateur renvoyé sera utilisée par Auth0 pour identifier l&#39;utilisateur. Si vous utilisez plusieurs connexions de base de données personnalisées, la valeur **id** **doit être unique dans l&#39;ensemble de toutes les connexions de base de données personnalisées** afin d&#39;éviter les collisions d&#39;**ID utilisateur**. Nous vous recommandons de faire précéder la valeur de **id** du nom de la connexion (sans espaces). Pour en savoir plus sur les ID utilisateur, consultez [Identifier les utilisateurs](/docs/fr-CA/manage-users/user-accounts/identify-users).
* La latence sera plus élevée que pour les magasins d&#39;utilisateurs hébergés par Auth0.
* La base de données ou le service doit être accessible depuis les serveurs Auth0. Vous devrez configurer les connexions entrantes si votre magasin d&#39;utilisateurs se trouve derrière un pare-feu.

<div id="testing-a-specific-runtime-version">
  ## Tester une version précise du runtime
</div>

La version du runtime des scripts **Database** personnalisés est définie dans **Dashboard &gt; Settings &gt; Advanced**, dans la section **Extensibility**.

Suivez ces étapes pour tester un script précis de base de données personnalisée avec une version précise du runtime :

1. Accédez à **Authentication &gt; Database.**
2. Sélectionnez la connexion **Database** dans laquelle vous avez défini un script personnalisé.
3. Dans la page de la connexion de base de données sélectionnée, sélectionnez l’onglet **Custom Database**.
4. Faites défiler la page jusqu’à la section **Database Action Scripts**.
5. Sélectionnez le script précis que vous voulez vérifier (p. ex. **Login**, **Create**, etc.) dans l’onglet correspondant.
6. Sélectionnez **Enregistrer et essayer**. Une fenêtre modale s’ouvre avec des paramètres de test précis et des exemples de détails de contexte. Mettez-les à jour au besoin.
7. Sélectionnez **Try** dans la fenêtre modale pour ouvrir une liste déroulante permettant de choisir une version précise de Node.
8. Le test se déclenchera lorsque vous sélectionnerez la version de Node voulue, et les résultats s’afficheront dans un message sur le même écran.
