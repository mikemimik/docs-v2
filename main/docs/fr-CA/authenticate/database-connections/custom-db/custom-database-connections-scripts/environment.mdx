Les scripts d’action s’exécutent comme une série de fonctions Javascript invoquées dans une instance d’un conteneur Webtask serverless. Dans ce contexte, un environnement précis est fourni, ainsi qu’un certain nombre d’artefacts fournis à la fois par le conteneur Webtask et par le <Tooltip tip="Serveur d’authentification : serveur qui confirme ou refuse l’identité d’un utilisateur." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=authentication+server">serveur d’authentification</Tooltip> d’Auth0 lui-même (votre tenant Auth0).

<div id="npm-modules">
  ## modules npm
</div>

Les conteneurs Webtask serverless d’Auth0 peuvent utiliser une vaste gamme de modules [`npm`](https://www.npmjs.com/) ; les modules `npm` réduisent non seulement la taille globale du code des scripts d’action, mais donnent aussi accès à un large éventail de fonctionnalités prêtes à l’emploi.

Par défaut, une longue liste de modules `npm` accessibles au public est prise en charge nativement. Cette liste a été compilée et examinée afin de déceler tout risque de sécurité potentiel. Pour voir quels modules `npm` sont pris en charge, consultez [Can I require: Auth0 Extensibility](https://auth0-extensions.github.io/canirequire/).

Si vous avez besoin d’un module `npm` qui n’est pas pris en charge nativement, vous pouvez faire une demande par l’entremise du [portail Auth0 Extensibility](https://auth0-extensions.github.io/canirequire/) ou de votre représentant Auth0. Auth0 évaluera votre demande pour en déterminer la pertinence; si vous devez faire remonter une demande non résolue, ouvrez un ticket de soutien au moyen du [portail de soutien d’Auth0](https://support.auth0.com/). À l’heure actuelle, Auth0 n’offre aucun soutien pour l’utilisation de modules `npm` provenant de dépôts privés.

<div id="variables">
  ## Variables
</div>

Les scripts d’action Auth0 prennent en charge les variables d’environnement, accessibles au moyen de l’objet `configuration` disponible globalement. L’objet `configuration` doit être considéré comme étant en lecture seule et servir à stocker des renseignements sensibles, comme des identifiants ou des clés API permettant d’accéder à des magasins d’identités externes. Cela permet d’éviter d’avoir des valeurs sensibles sur le plan de la sécurité codées en dur dans un script d’action.

L’objet `configuration` peut aussi servir à mettre en œuvre les stratégies de pratiques exemplaires du cycle de vie du développement logiciel (SDLC) que vous utilisez, comme [la configuration de plusieurs environnements](/docs/fr-CA/get-started/auth0-overview/create-tenants/set-up-multiple-environments), en vous permettant de définir des variables ayant des valeurs propres au tenant. Cela évite les valeurs codées en dur dans un script d’action, qui peuvent varier selon le tenant qui l’exécute.

<div id="global-object">
  ## objet global
</div>

Les conteneurs Webtask serverless d’Auth0 sont provisionnés à partir d’un pool associé à chaque tenant Auth0. Chaque instance de conteneur met à disposition un objet global, accessible par tous les scripts d’action qui s’y exécutent (c’est-à-dire dans cette instance de conteneur). L’objet global agit comme une variable globale propre au conteneur et peut servir à définir des informations — ou même des fonctions — réutilisables par tous les scripts d’action qui s’y exécutent (dans cette instance de conteneur).

Cela signifie que l’objet global peut servir à mettre en cache des ressources coûteuses, à condition qu’elles ne soient pas propres à un utilisateur. Par exemple, un <Tooltip tip="Jeton d’accès : information d’identification d’autorisation, sous la forme d’une chaîne opaque ou d’un JWT, utilisée pour accéder à une API." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Access+Token">jeton d’accès</Tooltip> pour une API tierce (p. ex. de journalisation) offrant des fonctionnalités non propres à un utilisateur pourrait y être stocké. Il peut aussi servir à stocker un jeton d’accès pour votre propre API non propre à un utilisateur, définie dans Auth0 et obtenue au moyen du flux Client Credentials.

<Warning>
  Un script d’action peut s’exécuter dans n’importe quelle instance de conteneur déjà en cours d’exécution ou dans une nouvelle instance de conteneur (qui pourra ensuite être ajoutée au pool). Dans Auth0, il n’existe aucune affinité de conteneur pour l’exécution des scripts d’action. Cela signifie que vous devez éviter de stocker dans l’objet global toute information propre à un utilisateur et toujours vous assurer que toute déclaration qui y est faite prévoit aussi son initialisation.
</Warning>

Chaque fois qu’un conteneur Webtask est recyclé, ou qu’un nouveau conteneur Webtask est instancié, l’objet global qu’il définit est réinitialisé. Ainsi, toute déclaration d’assignation dans l’objet global associé à un conteneur devrait aussi prévoir l’initialisation. Afin d’offrir une certaine flexibilité sur le plan des performances, les conteneurs Webtask serverless sont provisionnés dans Auth0 de manière ponctuelle et sont également assujettis à diverses politiques de recyclage. En règle générale, nous recommandons de ne pas considérer qu’un objet global a une durée de vie de plus de 20 minutes.

<div id="custom-database-connection-environment-checklist">
  ## Liste de vérification de l’environnement d’une connexion de base de données personnalisée
</div>

* Assurez-vous que votre base de données contient les champs appropriés pour stocker les attributs des profils utilisateur, comme **id**, **nickname**, **email** et **password**. Pour en savoir plus sur le schéma des profils utilisateur d’Auth0 et les champs attendus, consultez [Profils utilisateur normalisés](/docs/fr-CA/manage-users/user-accounts/user-profiles/normalized-user-profiles). Pour savoir comment mettre à jour les profils utilisateur, consultez [Mettre à jour les profils utilisateur à l’aide de votre base de données](/docs/fr-CA/manage-users/user-accounts/user-profiles/update-user-profiles-using-your-database).
* Vous pouvez utiliser les erreurs renvoyées par votre connexion de base de données personnalisée à des fins de dépannage.
* La propriété `id` (ou `user_id`) du profil utilisateur renvoyé sera utilisée par Auth0 pour identifier l’utilisateur. Si vous utilisez plusieurs connexions de base de données personnalisées, la valeur **id** **doit être unique dans l’ensemble des connexions de base de données personnalisées** afin d’éviter les collisions d’**ID utilisateur**. Nous vous recommandons de préfixer la valeur de **id** par le nom de la connexion (sans espaces). Pour en savoir plus sur les ID utilisateur, consultez [Identifier les utilisateurs](/docs/fr-CA/manage-users/user-accounts/identify-users).
* La latence sera plus élevée que dans le cas des magasins d’utilisateurs hébergés par Auth0.
* La base de données ou le service doit être accessible depuis les serveurs Auth0. Vous devrez configurer les connexions entrantes si votre magasin est derrière un pare-feu.

<div id="testing-a-specific-runtime-version">
  ## Tester une version précise du runtime
</div>

La version du runtime des scripts de base de données personnalisés est définie dans **Dashboard &gt; Paramètres &gt; Avancé**, dans la section **Extensibilité**.

Suivez ces étapes pour tester un script de base de données personnalisé précis avec une version de runtime donnée :

1. Accédez à **Authentification &gt; Base de données.**
2. Sélectionnez la connexion de base de données dans laquelle vous avez défini un script personnalisé.
3. Dans la page de la connexion de base de données sélectionnée, sélectionnez l’onglet **Base de données personnalisée**.
4. Faites défiler la page jusqu’à la section **Scripts d’action de base de données**.
5. Sélectionnez le script précis que vous voulez tester (p. ex. Login, Create, etc.) dans l’onglet correspondant.
6. Sélectionnez **Enregistrer et essayer**. Une fenêtre modale s’ouvre avec des paramètres de test précis et un exemple de contexte. Mettez-les à jour au besoin.
7. Sélectionnez **Essayer** dans la fenêtre modale pour ouvrir une liste déroulante permettant de choisir une version précise de Node.
8. Le test se déclenchera lorsque vous sélectionnerez la version de Node souhaitée, et les résultats s’afficheront dans un message sur le même écran.
