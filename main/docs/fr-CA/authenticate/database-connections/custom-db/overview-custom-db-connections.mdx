<Card title="La disponibilité varie selon le forfait Auth0">
  La disponibilité de cette fonctionnalité dépend de votre forfait Auth0 ou de votre entente personnalisée. Pour en savoir plus, consultez [Pricing](https://auth0.com/pricing).
</Card>

Utilisez une connexion de base de données personnalisée lorsque vous souhaitez donner accès à votre propre magasin de données d’identité indépendant (hérité) aux fins suivantes :

* **Authentification** : utilisez votre base de données comme <Tooltip tip="Fournisseur d’identité (IdP) : service qui stocke et gère les identités numériques." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=identity+provider">fournisseur d’identité</Tooltip> dans Auth0 pour authentifier les utilisateurs. (On parle alors d’authentification héritée.)
* **Importer des utilisateurs** : utilisez la migration automatique (migration progressive ou paresseuse)
* **Accès par proxy à un tenant Auth0** : utilisez l’architecture multilocataire d’Auth0.

Vous pouvez créer et configurer une connexion de base de données personnalisée en effectuant l’une des tâches suivantes :

1. Utilisez le endpoint [Create connections](https://auth0.com/docs/api/management/v2#!/Connections/post_connections) avec la stratégie `auth0`.
2. Accédez à [Auth0 Dashboard &gt; Authentication &gt; Database](https://manage.auth0.com/#/connections/database), créez la connexion et activez l’option **Use my own database** pour permettre la modification du script d’action de base de données.

   <Frame>![Activer l’option Use My Own Database de Custom Database](/docs/images/cdy7uua7fh8z/3kgHDpBFdVWNq9XOfhsTXI/2efa43e44793b8dd69b9f9e8b54bf752/2025-02-25_10-20-20.png)</Frame>

<div id="how-it-works">
  ## Fonctionnement
</div>

Comme l’illustre le diagramme ci-dessous, vous utilisez des connexions de base de données personnalisées dans le flux de <Tooltip tip="Universal Login : votre application redirige vers Universal Login, hébergé sur le serveur d’autorisation d’Auth0, afin de vérifier l’identité d’un utilisateur." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Universal+Login">Universal Login</Tooltip> pour obtenir des renseignements sur l’identité des utilisateurs à partir de votre propre magasin d’identités hérité.

<Frame>![Anatomie des connexions de base de données personnalisées](/docs/images/cdy7uua7fh8z/2lHqvZKFiEbAXURU2gmchc/626cac94211c266f2135a41456b2e49d/custom-database-connections.png)</Frame>

En plus des éléments communs à tous les types de [connexion de base de données](/docs/fr-CA/authenticate/database-connections), une connexion de base de données personnalisée vous permet de configurer des scripts d’action : du code personnalisé utilisé pour interagir avec votre magasin d’identités hérité. Auth0 fournit des [modèles de scripts d’action de base de données personnalisée](/docs/fr-CA/authenticate/database-connections/custom-db/templates) pour la configuration, et ceux que vous utilisez dépendent du fait que vous créiez une connexion de base de données personnalisée pour l’authentification héritée ou pour la migration automatique.

<Card title="Pratique exemplaire">
  Les scripts d’action peuvent être implémentés sous forme de fonctions anonymes. Toutefois, les fonctions anonymes compliquent le débogage lorsqu’il faut interpréter la pile d’appels générée par une condition d’erreur exceptionnelle. Pour plus de commodité, nous vous recommandons d’attribuer un nom de fonction à chaque script d’action.
</Card>

<div id="legacy-authentication-scenario">
  ### Scénario d’authentification héritée
</div>

Dans un scénario d’authentification héritée, un nouveau dossier utilisateur est créé dans Auth0 lors de la première connexion de l’utilisateur, mais Auth0 ne stocke pas le hachage du mot de passe de l’utilisateur. Auth0 utilise toujours le magasin d’identités hérité et l’identité qu’il contient pour authentifier l’utilisateur.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Les connexions de base de données personnalisées sont aussi utilisées en dehors du flux de travail Universal Login. Par exemple, le script d’action `changePassword` d’une connexion est appelé lorsqu’un changement de mot de passe est effectué pour un utilisateur dont le compte se trouve dans un magasin d’identités hérité.
</Callout>

<div id="automatic-migration-scenario">
  ### Scénario de migration automatique
</div>

Lors d’une migration automatique ou progressive, Auth0 crée un nouvel utilisateur dans un référentiel d’identités (base de données) géré par Auth0. Par la suite, Auth0 utilise toujours l’identité du référentiel d’identités géré par Auth0 pour authentifier l’utilisateur. Pour que cela se produise, Auth0 exige d’abord que l’utilisateur soit authentifié auprès du magasin d’identités hérité; ce n’est qu’en cas de succès que le nouvel utilisateur est créé. Le nouvel utilisateur est créé avec le même ID et le même mot de passe que ceux fournis lors de l’authentification.

<Card title="Bonne pratique">
  La création d’un utilisateur dans un scénario de migration automatique se produit généralement après l’exécution du script d’action `login`. Nous recommandons donc de ne pas tenter de supprimer un utilisateur d’un magasin d’identités hérité au moyen d’une opération inline (c.-à-d. dans le script `login`), mais plutôt d’effectuer cette suppression dans le cadre d’un processus distinct. Cela évitera la suppression accidentelle d’un utilisateur si une erreur survient pendant le processus de migration.
</Card>

Dans un scénario de migration automatique, les utilisateurs demeurent dans le magasin d’identités hérité et peuvent être supprimés ou archivés au besoin. Un effet secondaire peut toutefois se produire si un utilisateur est supprimé d’Auth0, mais demeure présent dans le magasin d’identités hérité. Dans ce cas, une connexion effectuée par l’utilisateur supprimé pourrait entraîner l’exécution du script `login` et/ou `getUser`, puis la migration de l’utilisateur à partir du magasin d’identités hérité une fois de plus.

<Card title="Bonne pratique">
  Nous recommandons d’indiquer que toute identité utilisateur héritée a été migrée avant la fin de `login` ou de `getUser`, et avant toute suppression éventuelle du magasin d’identités hérité.
</Card>

<div id="size">
  ## Taille
</div>

La taille totale de la mise en œuvre de tout script d’action ne doit pas dépasser 100 kB. Plus cette taille est élevée, plus la latence causée par le processus d’empaquetage et de transport utilisé par la plateforme serverless Webtask d’Auth0 augmente, ce qui aura une incidence sur les performances de votre système.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  La limite de 100 kB n’inclut pas les modules `npm` référencés dans des instructions require.
</Callout>

<div id="environment">
  ## Environnement
</div>

Les scripts d’action s’exécutent comme une série de fonctions JavaScript appelées dans une instance d’un conteneur Webtask serverless. Dans ce contexte, un environnement précis est fourni, ainsi qu’un certain nombre d’artefacts fournis à la fois par le Webtask container et par le <Tooltip tip="Serveur d’authentification : serveur qui confirme ou refuse l’identité d’un utilisateur." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=authentication+server">serveur d’authentification</Tooltip> Auth0 lui-même (votre Auth0 tenant).

<div id="npm-modules">
  ### modules npm
</div>

Les conteneurs Webtask serverless d’Auth0 peuvent utiliser une [vaste gamme de modules npm](https://auth0-extensions.github.io/canirequire/) ; les modules `npm` réduisent non seulement la taille globale du code des scripts d’action, mais donnent aussi accès à un large éventail de fonctionnalités prêtes à l’emploi.

De nombreux modules `npm` accessibles publiquement sont pris en charge d’emblée. La liste a été compilée et vérifiée afin de repérer tout risque potentiel en matière de sécurité. Si vous avez besoin d’un module `npm` qui n’est pas pris en charge d’emblée, vous pouvez soumettre une demande par l’entremise du [portail de soutien Auth0](https://support.auth0.com) ou de votre représentant Auth0. Auth0 évaluera votre demande afin d’en déterminer la pertinence. À l’heure actuelle, Auth0 ne prend pas en charge l’utilisation de modules `npm` provenant de dépôts privés.

<div id="variables">
  ### Variables
</div>

Les scripts d&#39;action Auth0 prennent en charge les variables d&#39;environnement, auxquelles on accède au moyen de l&#39;objet `configuration`, accessible globalement. Pour en savoir plus, consultez la section Ajouter des paramètres de configuration dans [Créer des connexions de base de données personnalisées](/docs/fr-CA/authenticate/database-connections/custom-db/create-db-connection).

<Card title="Pratique exemplaire">
  L&#39;objet `configuration` doit être traité comme étant en lecture seule et servir à stocker des renseignements sensibles, comme des identifiants ou des clés API, permettant d&#39;accéder à des référentiels d&#39;identité externes. Cela évite d&#39;inscrire en dur, dans un script d&#39;action, des valeurs sensibles sur le plan de la sécurité.
</Card>

L&#39;objet de configuration peut aussi servir à prendre en charge les stratégies de [cycle de vie du développement logiciel (SDLC)](/docs/fr-CA/get-started/auth0-overview/create-tenants/set-up-multiple-environments) que vous utilisez, en vous permettant de définir des variables ayant des valeurs propres au tenant. Cela évite d&#39;inscrire en dur, dans un script d&#39;action, des valeurs qui peuvent changer selon le tenant qui l&#39;exécute.

<div id="global-object">
  ### objet global
</div>

Les conteneurs Webtask serverless d’Auth0 sont provisionnés à partir d’un pool associé à chaque tenant Auth0. Chaque instance de conteneur met l’objet `global` à disposition; on peut y accéder à partir de tous les scripts d’action exécutés dans cette instance. L’objet `global` agit comme une variable globale propre au conteneur et peut servir à définir des informations ou des fonctions utilisées par tous les scripts d’action qui s’exécutent dans l’instance du conteneur.

Cela signifie que l’objet `global` peut servir à mettre en cache des ressources coûteuses, tant que ces ressources ne sont pas propres à un utilisateur. Par exemple, vous pourriez l’utiliser pour stocker un <Tooltip tip="jeton d’accès : information d’autorisation, sous la forme d’une chaîne opaque ou d’un JWT, utilisée pour accéder à une API." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Access+Token">jeton d’accès</Tooltip> pour une API de journalisation tierce qui offre des fonctionnalités non propres à un utilisateur. Vous pourriez aussi stocker un jeton d’accès pour votre propre API non propre à un utilisateur, définie dans Auth0 et obtenue au moyen du [Client Credentials Flow](/docs/fr-CA/get-started/authentication-and-authorization-flow/client-credentials-flow).

<Warning>
  Un script d’action peut s’exécuter dans n’importe quelle instance de conteneur déjà en cours d’exécution, ou dans une nouvelle instance de conteneur (qui pourra ensuite être ajoutée au pool). Il n’y a aucune affinité de conteneur pour l’exécution des scripts d’action dans Auth0. Cela signifie que vous devriez éviter de stocker des renseignements propres à un utilisateur dans l’objet `global`, et toujours vous assurer que toute déclaration effectuée dans l’objet `global` prévoit aussi une initialisation.
</Warning>

Chaque fois qu’un conteneur Webtask est recyclé, ou lors de l’instanciation d’un nouveau conteneur Webtask, l’objet `global` qu’il définit est réinitialisé. Ainsi, toute déclaration d’assignation dans l’objet `global` associé à un conteneur devrait aussi inclure une initialisation.

<Card title="Bonne pratique">
  Pour offrir une certaine souplesse sur le plan des performances, les conteneurs Webtask serverless sont provisionnés dans Auth0 au besoin et sont aussi assujettis à diverses politiques de recyclage. En général, nous recommandons de ne pas considérer qu’un objet `global` a une durée de vie de plus de 20 minutes.
</Card>

<div id="security">
  ## Sécurité
</div>

<div id="access-legacy-identity-storage-via-custom-api">
  ### Accéder au magasin d’identités hérité au moyen d’une API personnalisée
</div>

Protéger le magasin d’identités hérité contre tout accès général constitue une bonne pratique recommandée. Exposer directement une base de données à Internet, par exemple, peut être extrêmement problématique : les interfaces de base de données pour SQL et autres technologies du genre offrent un éventail de fonctionnalités beaucoup trop large, ce qui va à l’encontre du principe du moindre privilège en matière de sécurité.

<Card title="Bonne pratique">
  Nous vous recommandons de mettre en place une API afin d’appliquer le principe du moindre privilège à votre magasin d’identités hérité (base de données), plutôt que d’ouvrir simplement un accès général par Internet.
</Card>

L’autre option consiste à créer une API simple (personnalisée), protégée par l’utilisation d’un jeton d’accès, que chaque script d’action peut appeler. Cette API servirait alors d’interface avec le magasin d’identités hérité. Le Client Credentials Grant Flow peut ensuite être utilisé pour obtenir un jeton d’accès à partir d’un script, puis ce jeton peut être mis en cache dans l’objet `global` pour être réutilisé afin d’améliorer les performances. L’API peut ensuite exposer un nombre limité d’endpoints protégés qui n’exécutent que les fonctions de gestion héritées requises (par exemple, `read user`, `change password`).

<Card title="Bonne pratique">
  Par défaut, Auth0 vous remettra un jeton pour n’importe quelle API si vous vous authentifiez avec succès et incluez l’audience appropriée. Restreindre l’accès à l’API du magasin d’identités hérité en limitant l’allocation des jetons d’accès au moyen d’une Rule empêchera toute utilisation non autorisée et atténuera plusieurs scénarios de vecteurs d’attaque, par exemple lorsqu’une redirection vers `/authorize` est interceptée et que l’audience de l’API y est ajoutée.
</Card>

<div id="allowlist-access-to-legacy-identity-storage">
  ### Autoriser l’accès au stockage d’identités hérité
</div>

Que vous gériez l’accès à un magasin d’identités hérité au moyen d’une API personnalisée ou que vous utilisiez l’interface native fournie, limitez l’accès à la liste des adresses IP associées à votre tenant Auth0. La mise en liste d’autorisation restreint l’accès au magasin d’identités hérité, afin que seuls les scripts d’action de base de données personnalisée définis dans Auth0 soient autorisés.

<Warning>
  La liste d’autorisation d’adresses IP d’Auth0 est partagée entre tous les tenants Auth0 définis dans une région. N’utilisez jamais la liste d’autorisation comme seule méthode pour sécuriser l’accès à votre magasin d’identités hérité ; cela pourrait entraîner des vulnérabilités de sécurité et permettre un accès non autorisé à vos utilisateurs.
</Warning>
