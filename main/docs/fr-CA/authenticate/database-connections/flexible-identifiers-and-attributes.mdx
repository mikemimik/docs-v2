<Warning>
  L’activation des Flexible Identifiers sur votre tenant risque d’introduire des changements non rétrocompatibles dans votre environnement de production. Testez cette fonctionnalité de façon approfondie dans un environnement de développement et prenez note de vos paramètres de connexion actuels avant de la déployer à grande échelle.
</Warning>

Un **Identifiant flexible** est l’attribut qu’un utilisateur saisit sur un écran de connexion pour s’authentifier. Vous pouvez choisir parmi l’adresse courriel, le username, le numéro de téléphone ou une combinaison de deux attributs ou plus.

<div id="attribute-and-identifier-definitions">
  ### Définitions d&#39;attribut et d&#39;identifiant
</div>

Pour ce produit, un **attribut** est une donnée utilisateur qui peut être stockée, comme une adresse courriel, un numéro de téléphone et un username. Tous les identifiants sont des attributs, mais seuls certains attributs sont des identifiants.

Un **identifiant** est un attribut unique qui permet de reconnaître un utilisateur distinct dans une connection donnée. Une adresse courriel, phone et un username peuvent identifier une personne de façon unique et servir d&#39;identifiants, tandis que d&#39;autres attributs enrichissent le profil de l&#39;utilisateur sans permettre de l&#39;identifier de façon unique.

<div id="use-flexible-identifiers">
  ### Utiliser Flexible Identifiers
</div>

Flexible Identifiers est destiné à un usage général, avec les limites suivantes :

* Flexible Identifiers, y compris l’attribut phone, n’est offert qu’avec [Universal Login](/docs/fr-CA/authenticate/login/auth0-universal-login/universal-login-vs-classic-login/universal-experience), et vous devez [configurer un phone provider](/docs/fr-CA/customize/phone-messages/configure-phone-messaging-providers).
* Vous devez configurer [Identifier First](/docs/fr-CA/authenticate/login/auth0-universal-login/identifier-first) pour utiliser la vérification par téléphone lors du Signup.
* L’attribut d’adresse courriel doit être activé pour utiliser [Adaptive MFA](/docs/fr-CA/secure/multi-factor-authentication/adaptive-mfa).
* Vous devez avoir une adresse courriel dans le profil utilisateur pour utiliser les invites de Signup pour [Organizations](/docs/fr-CA/manage-users/organizations/organizations-overview).
* Les utilisateurs finaux bloqués par <Tooltip tip="Brute Force Protection : forme de protection contre les attaques qui protège contre les attaques par force brute provenant d’une seule adresse IP et visant un seul compte utilisateur." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Brute+Force+Protection">Brute Force Protection</Tooltip> ne peuvent pas se débloquer eux-mêmes au moyen d’un message SMS. D’autres méthodes sont offertes; pour en savoir plus, [consultez Brute Force Protection](/docs/fr-CA/secure/attack-protection/brute-force-protection).
* Flexible Identifiers déplace le champ d’identifiant vers le premier écran de connexion et remplace l’adresse courriel par le username dans l’invite de réinitialisation du mot de passe.
* Les jetons OTP pour la vérification des identifiants par téléphone et par courriel ont une durée de validité de 900 secondes.

<div id="issues-using-flexible-identifiers">
  ### Problèmes liés à l’utilisation de Flexible Identifiers
</div>

Voici une liste de problèmes potentiels que vous pourriez rencontrer lors de la configuration et de la gestion de Flexible Identifiers :

* Si le scope `phone` n’est pas indiqué dans la authorization request de votre application, vous ne recevrez pas le claim `phone_number`. Pour en savoir plus sur les scopes, [consultez Scopes](/docs/fr-CA/get-started/apis/scopes).
* Votre action script Get User de Custom Database doit être valide lorsque **Import Users to Auth0** est activé. Pour en savoir plus, [consultez Configure Automatic Migration from Your Database](/docs/fr-CA/manage-users/user-migration/configure-automatic-migration-from-your-database).
* Chaque utilisateur doit avoir un username, une adresse courriel et un numéro de téléphone uniques, quel que soit le connection type. Le numéro de téléphone doit être unique même s’il n’est pas ajouté comme attribut.
* Si vous utilisez l’action script Change Password de Custom Database et souhaitez définir `email` et `email_verified` à `True`, vous devez renvoyer la valeur `email_verified` souhaitée dans l’objet. Pour en savoir plus, [consultez Change Password](/docs/fr-CA/authenticate/database-connections/custom-db/templates/change-password).
* Si vous utilisez une custom database connection avec Import Users to Auth0 désactivé, vous devez harmoniser les properties de votre profil utilisateur avec le Normalized User Profile d’Auth0. Pour en savoir plus, [consultez Normalized User Profile](/docs/fr-CA/manage-users/user-accounts/user-profiles/normalized-user-profiles).
* Si vous utilisez une custom database connection avec Import Users to Auth0 activé, Auth0 vérifiera l’unicité de `phone_number` et de `phone_verified`.
* Les prompts Identifier First affichent tous les identifiants sur le premier screen et suppriment vos paramètres précédents, et le reset password prompt affichera le input field pour Username au lieu de Email.
* Familiarisez-vous avec les pratiques exemplaires pour éviter les attaques de SMS Pumping. Pour en savoir plus, consultez notre [livre blanc sur le SMS Pumping](https://www.okta.com/resources/whitepaper-handling-toll-fraud-and-sms-pumping-with-twilio-in-the-okta-customer-identity/).
