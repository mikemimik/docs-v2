---
description: Comment configurer les courriels non uniques et les identifiants qui y sont associés
title: Non-Unique Emails
---

Les Non-Unique Emails permettent à plusieurs comptes d’utilisateur partageant une connexion de base de données d’utiliser une même adresse courriel, tandis qu’un autre attribut (comme le nom d’utilisateur ou le numéro de téléphone) sert d’identifiant principal. Que vous soyez un parent qui gère plusieurs comptes d’enfants avec une seule adresse courriel ou une petite entreprise avec une adresse courriel par emplacement, les Non-Unique Emails vous permettent d’offrir une expérience de connexion et de réinitialisation du mot de passe sécurisée et propre à chaque compte.

<div id="considerations">
  ### Points à considérer
</div>

Consultez les points suivants afin de déterminer si Non-Unique Emails convient à votre cas d’utilisation.

<div id="primary-identifier-requirements">
  #### Exigences relatives à l’identifiant principal
</div>

Le courriel ne peut pas être utilisé comme identifiant principal lorsque vous utilisez Non-Unique Emails. Vous devez configurer un autre attribut comme identifiant principal, qui servira à l’authentification, à la réinitialisation du mot de passe et à la gestion du compte.

Pour en savoir plus sur les identifiants et les attributs, consultez [Identifiants flexibles](/docs/fr-CA/authenticate/database-connections/flexible-identifiers-and-attributes).

<div id="password-resets">
  #### Réinitialisation des mots de passe
</div>

Les utilisateurs finaux doivent fournir leur nom d’utilisateur, leur numéro de téléphone ou tout autre attribut que leur administrateur a configuré comme attribut principal lorsqu’ils réinitialisent leur mot de passe. Auth0 utilise cet identifiant principal pour retrouver et réinitialiser le compte associé à une adresse courriel partagée.

<div id="irreversible-settings">
  #### Paramètres irréversibles
</div>

Une fois que l’attribut de courriel d’une connexion est défini comme non unique, il ne peut plus redevenir unique. De plus, seules de nouvelles connexions de base de données peuvent être créées avec la prise en charge de courriels non uniques; vous ne pouvez pas modifier une connexion existante et vous devez mettre à jour votre application pour utiliser l’identifiant principal choisi.

<div id="flexible-identifiers">
  #### Identifiants flexibles
</div>

Les identifiants flexibles doivent être activés dans la connexion de base de données pour pouvoir utiliser les courriels non uniques, et ils ne peuvent plus être désactivés une fois la connexion créée. Lorsque vous activez les courriels non uniques à l’aide de l’<Tooltip tip="" cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Management+API">Management API</Tooltip>, les identifiants flexibles sont configurés automatiquement.

<div id="api-behavior-changes">
  ### Changements au comportement de l’API
</div>

`GET /api/v2/users-by-email` renvoie tous les utilisateurs qui partagent la même adresse courriel.

`DELETE /api/v2/connections/{id}/users` n’est pas compatible avec les connexions avec courriel non unique.

`POST /dbconnections/change_password` ne fonctionne pas avec les connexions avec courriel non unique, car il nécessite une adresse courriel unique pour trouver le compte d’utilisateur. Les utilisateurs doivent utiliser des flux qui s’appuient sur l’identifiant principal pour réinitialiser leur mot de passe.

<div id="enable-non-unique-emails-in-the-auth0-dashboard">
  ### Activer Non-Unique Emails dans l’Auth0 Dashboard
</div>

1. Allez à **Authentication &gt; Database** et créez une nouvelle connexion.
2. Accédez à la section **Choose one or more attributes as user identifiers**, puis mettez la bascule **adresse courriel** à **On,** et activez ensuite la bascule **Allow non-unique email addresses** qui s’affiche.
3. Activez également **username** ou le numéro de téléphone afin qu’il serve d’identifiant principal pour la connexion et les processus de réinitialisation du mot de passe.
4. Après avoir confirmé que l’adresse courriel ne sera pas utilisée comme identifiant, sélectionnez **Create** pour enregistrer la connexion.

<div id="enable-non-unique-emails-via-the-management-api">
  ### Activer les Non-Unique Emails au moyen de la Management API
</div>

Utilisez le endpoint `POST /api/v2/connections` de la Management API pour créer une connexion de base de données qui prend en charge les Non-Unique Emails.

Lors de la création de la connexion :

* Définissez **unique: false** dans l’objet `options.attributes.email` pour permettre à plusieurs comptes d’avoir la même adresse courriel. Définissez **identifier.active: false** pour vous assurer que le courriel ne peut pas servir d’identifiant principal lorsqu’il n’est pas unique.
* Choisissez un autre attribut comme identifiant principal et définissez **identifier.active: true** pour l’attribut choisi.

Sans identifiant principal autre que le courriel, les flux d’authentification et de réinitialisation du mot de passe ne fonctionnent pas correctement. Assurez-vous qu’au moins un attribut est défini comme identifiant actif.

<div id="example-request">
  #### Exemple de requête
</div>

Vous trouverez ci-dessous un exemple de corps de la requête pour créer une connexion de base de données qui utilise un `username` comme identifiant principal et permet les adresses courriel non uniques :

```json lines expandable
{
 "name": "new-non-unique",
 "strategy": "authO",
 "options": {
  "attributes": {
   "email": {
     "unique": false,
     "signup" : {
       "status": "required"
     },
     "identifier": {
      "active": false
     },
     "profile_required": true
   ｝，
   "username": {
    "signup": {
     "status": "required"
   },
   "identifier": {
    "active": true
   ｝，
   "profile_required": true
   }
  }
 }
}
```

<div id="shared-email-risk-disclaimer">
  ### Avis concernant les risques liés aux courriels partagés
</div>

Bien que la fonctionnalité **Non-Unique Emails** comporte des mesures de protection, comme l’interdiction d’utiliser le courriel comme identifiant principal et l’obligation d’effectuer les réinitialisations de mot de passe à l’aide d’un nom d’utilisateur ou d’un numéro de téléphone**,** il subsiste un risque inhérent lorsque plusieurs comptes d’utilisateur partagent la même adresse courriel. Par exemple :

* Toutes les communications par courriel (p. ex., les liens de réinitialisation de mot de passe, les notifications) seront livrées à la même boîte de réception, peu importe l’utilisateur qui a lancé l’action.
* Cela pourrait entraîner de la confusion chez les utilisateurs ou donner accès involontairement à des liens envoyés par courriel si la boîte de réception est partagée.

En activant cette fonctionnalité, vous reconnaissez et acceptez qu’il vous incombe de vous assurer que :

* Les situations de courriel partagé conviennent à votre cas d’utilisation.
* Les utilisateurs finaux sont informés et formés en conséquence.
* La conception de votre application tient compte des chevauchements possibles dans les flux de travail reposant sur le courriel.

Ce compromis offre de la souplesse, mais exige une mise en œuvre réfléchie et une communication claire avec vos utilisateurs.
