description: Comment configurer les courriels non uniques et les identifiants qui y sont associés
title: Non-Unique Emails
---

Les Non-Unique Emails permettent à plusieurs comptes d’utilisateur partageant une même connexion de base de données d’utiliser une seule adresse courriel, tandis qu’un autre attribut (comme le nom d’utilisateur ou le numéro de téléphone) sert d’identifiant principal. Que vous soyez un parent qui gère plusieurs comptes d’enfant à l’aide d’une seule adresse courriel ou une petite entreprise qui utilise une adresse courriel par emplacement, les Non-Unique Emails vous permettent d’offrir une expérience de connexion et de réinitialisation du mot de passe sécurisée et propre à chaque compte.

<div id="considerations">
  ### Points à considérer
</div>

Consultez les éléments suivants pour vérifier si Non-Unique Emails convient à votre cas d’utilisation.

<div id="primary-identifier-requirements">
  #### Exigences relatives à l’identifiant principal
</div>

Le courriel ne peut pas être utilisé comme identifiant principal lorsque vous utilisez Non-Unique Emails. Vous devez configurer un autre attribut comme identifiant principal; il servira à l’authentification, à la réinitialisation du mot de passe et à la gestion du compte.

Pour en savoir plus sur les identifiants et les attributs, consultez [Identifiants flexibles](/docs/fr-CA/authenticate/database-connections/flexible-identifiers-and-attributes).

<div id="password-resets">
  #### Réinitialisation du mot de passe
</div>

Les utilisateurs finaux doivent fournir leur nom d’utilisateur, leur numéro de téléphone ou l’attribut que leur administrateur a configuré comme attribut principal lorsqu’ils réinitialisent leur mot de passe. Auth0 utilise cet identifiant principal pour retrouver le compte associé à une adresse courriel partagée et en réinitialiser le mot de passe.

<div id="irreversible-settings">
  #### Paramètres irréversibles
</div>

Une fois que l’attribut de courriel est défini comme non unique dans une connexion, il ne peut plus redevenir unique. De plus, seules les nouvelles connexions de base de données peuvent être créées avec la prise en charge des courriels non uniques; vous ne pouvez pas modifier une connexion existante et vous devez mettre à jour votre application pour utiliser l’identifiant principal que vous choisissez.

<div id="flexible-identifiers">
  #### Identifiants flexibles
</div>

Les Identifiants flexibles doivent être activés dans la connexion de base de données pour utiliser les courriels non uniques, et ils ne peuvent plus être désactivés une fois la connexion créée. Lorsque vous activez les courriels non uniques à l’aide de l’<Tooltip tip="" cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Management+API">Management API</Tooltip>, les Identifiants flexibles sont automatiquement configurés pour vous.

<div id="api-behavior-changes">
  ### Changements au comportement de l’API
</div>

`GET /api/v2/users-by-email` renvoie tous les utilisateurs qui partagent la même adresse courriel.

`DELETE /api/v2/connections/{id}/users` n’est pas compatible avec les connexions utilisant des adresses courriel non uniques.

`POST /dbconnections/change_password` ne fonctionne pas avec les connexions utilisant des adresses courriel non uniques, car il nécessite une adresse courriel unique pour trouver le compte utilisateur. Les utilisateurs doivent utiliser des flux qui s’appuient sur l’identifiant principal pour réinitialiser leur mot de passe.

<div id="enable-non-unique-emails-in-the-auth0-dashboard">
  ### Activer les adresses courriel non uniques dans Auth0 Dashboard
</div>

1. Accédez à **Authentification &gt; Database** et créez une nouvelle connexion.
2. Allez à la section **Choisissez un ou plusieurs attributs comme identifiants d’utilisateur** et mettez la bascule **Email Address** à **Activé,** puis activez la bascule **Autoriser les adresses courriel non uniques** qui s’affiche.
3. Choisissez soit **username**, soit le numéro de téléphone, et mettez-le aussi à **Activé** pour qu’il serve d’identifiant principal pour le login et les flux de réinitialisation du mot de passe.
4. Après avoir confirmé que le courriel ne sera pas utilisé comme identifiant, choisissez **Create** pour enregistrer la connexion.

<div id="enable-non-unique-emails-via-the-management-api">
  ### Activer Non-Unique Emails au moyen de la Management API
</div>

Utilisez le point de terminaison `POST /api/v2/connections` de la Management API pour créer une connexion de base de données qui prend en charge Non-Unique Emails.

Lors de la création de la connexion :

* Définissez **unique: false** dans l’objet `options.attributes.email` pour permettre à plusieurs comptes d’utiliser la même adresse courriel. Définissez **identifier.active: false** afin que le courriel ne puisse pas servir d’identifiant principal lorsqu’il n’est pas unique.
* Choisissez un autre attribut comme identifiant principal et définissez **identifier.active: true** pour l’attribut choisi.

Sans identifiant principal autre que le courriel, les flux d’authentification et de réinitialisation du mot de passe ne fonctionnent pas correctement. Assurez-vous qu’au moins un attribut est défini comme identifiant actif.

<div id="example-request">
  #### Exemple de requête
</div>

Vous trouverez ci-dessous un exemple de corps de la requête pour créer une connexion de base de données qui utilise un nom d’utilisateur comme identifiant principal et permet les adresses courriel non uniques :

```json lines expandable
{
 "name": "new-non-unique",
 "strategy": "authO",
 "options": {
  "attributes": {
   "email": {
     "unique": false,
     "signup" : {
       "status": "required"
     },
     "identifier": {
      "active": false
     },
     "profile_required": true
   ｝，
   "username": {
    "signup": {
     "status": "required"
   },
   "identifier": {
    "active": true
   ｝，
   "profile_required": true
   }
  }
 }
}
```

<div id="shared-email-risk-disclaimer">
  ### Avis de non-responsabilité concernant les risques liés aux courriels partagés
</div>

Bien que la fonctionnalité **Non-Unique Emails** comprenne des mesures de protection, comme l’interdiction d’utiliser le courriel comme identifiant principal et l’obligation d’effectuer les réinitialisations de mot de passe à l’aide d’un nom d’utilisateur ou d’un numéro de téléphone**,** il demeure tout de même un risque inhérent lorsque plusieurs comptes d’utilisateur partagent la même adresse courriel. Par exemple :

* Toutes les communications par courriel (p. ex., les liens de réinitialisation de mot de passe, les notifications) seront envoyées à la même boîte de réception, peu importe l’utilisateur qui a lancé l’action.
* Cela pourrait entraîner de la confusion chez les utilisateurs ou un accès involontaire à des liens envoyés par courriel si la boîte de réception est partagée.

En activant cette fonctionnalité, vous reconnaissez et acceptez la responsabilité de veiller à ce que :

* Les scénarios de courriel partagé conviennent à votre cas d’utilisation.
* Les utilisateurs finaux en soient informés et reçoivent la formation appropriée.
* La conception de votre application tienne compte des chevauchements possibles dans les flux de travail liés aux courriels.

Ce compromis offre plus de souplesse, mais exige une mise en œuvre réfléchie et une communication claire avec vos utilisateurs.
