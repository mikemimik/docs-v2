Vous pouvez transmettre à un <Tooltip tip="Fournisseur d’identité (IdP) : service qui stocke et gère les identités numériques." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Identity+Provider">fournisseur d’identité</Tooltip> (IdP), lors de l’authentification, des paramètres qui lui sont spécifiques. Les valeurs peuvent être statiques pour une connexion donnée ou dynamiques pour chaque utilisateur.

<div id="limitations">
  ## Limites
</div>

Pour cette configuration, tenez compte des restrictions suivantes :

* Seuls les [paramètres OAuth 2.0/OIDC valides](http://openid.net/specs/openid-connect-core-1_0.html#AuthorizationEndpoint) sont acceptés.
* Les IdPs ne prennent pas tous en charge les paramètres en amont. Vérifiez auprès de l’IdP concerné avant de poursuivre votre implémentation.
* Les IdPs <Tooltip tip="Security Assertion Markup Language (SAML) : protocole normalisé permettant à deux parties d’échanger des renseignements d’authentification sans mot de passe." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=SAML">SAML</Tooltip> ne prennent pas en charge les paramètres en amont.

<div id="static-parameters">
  ## Paramètres statiques
</div>

Utilisez des paramètres statiques pour configurer votre connexion de façon à envoyer un ensemble standard de paramètres à l’IdP lorsqu’un utilisateur se connecte.

Pour configurer des paramètres statiques, appelez le point de terminaison Auth0 <Tooltip tip="Management API : un produit qui permet aux clients d’effectuer des tâches administratives." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Management+API">Management API</Tooltip> [Create a connection](https://auth0.com/docs/api/management/v2/#!/Connections/post_connections) ou [Update a connection](https://auth0.com/docs/api/management/v2/#!/Connections/patch_connections_by_id), puis transmettez l’objet `upstream_params` dans l’objet `options`, avec les paramètres que vous souhaitez envoyer à l’IdP.

<div id="example-wordpress">
  ### Exemple : WordPress
</div>

WordPress vous permet de transmettre un paramètre `blog` facultatif à son point de terminaison d’autorisation <Tooltip tip="OAuth 2.0 : cadre d’autorisation qui définit les protocoles et les flux d’autorisation." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=OAuth+2.0">OAuth 2.0</Tooltip> et de demander automatiquement l’accès à un blogue précis pour les utilisateurs au moment de leur connexion. Pour en savoir plus, consultez la [documentation OAuth 2.0 de WordPress](https://developer.wordpress.com/docs/oauth2/).

Pour suivre cet exemple, vous aurez besoin d’une [connexion sociale WordPress](https://marketplace.auth0.com/integrations/wordpress-social-connection) fonctionnelle.

<div id="dynamic-parameters">
  ## Paramètres dynamiques
</div>

Utilisez des paramètres dynamiques pour configurer votre connexion afin d’envoyer à l’IdP, au moment de la connexion, un ensemble de paramètres dont les valeurs sont propres à l’utilisateur.

Pour configurer des paramètres dynamiques, appelez le point de terminaison [Create a connection](https://auth0.com/docs/api/management/v2/#!/Connections/post_connections) ou [Update a connection](https://auth0.com/docs/api/management/v2/#!/Connections/patch_connections_by_id) de l’Auth0 Management API, incluez l’objet `upstream_params` dans l’objet `options` avec les paramètres que vous souhaitez envoyer à l’IdP, et indiquez le champ auquel le paramètre est associé à l’aide de l’attribut `alias`.

Voici un exemple d’objet `options` que nous reprendrons plus loin dans l’exemple X :

```json lines
{
  "options": {
    "upstream_params": {
      "screen_name": {
        "alias": "login_hint"
      }
    }
  }
}
```

<div id="available-fields">
  ### Champs disponibles
</div>

Voici les champs disponibles pour l’attribut `alias` :

* `acr_values`
* `audience`
* `client_id`
* `display`
* `id_token_hint`
* `login_hint`
* `max_age`
* `prompt`
* `resource`
* `response_mode`
* `response_type`
* `ui_locales`

export const codeExample41 = `https://{YOUR_AUTH0_DOMAIN}/authorize

  &login_hint=user@domain.com`;
