Les [fournisseurs d&#39;identité (IdP)](/docs/fr-CA/authenticate/identity-providers) les plus courants sont offerts dans le [Auth0 Dashboard](https://manage.auth0.com/#) et dans le [Auth0 Marketplace](https://marketplace.auth0.com/features/social-connections). Vous pouvez toutefois ajouter n&#39;importe quel fournisseur <Tooltip tip="OAuth 2.0 : framework d'autorisation qui définit les protocoles et les workflows d'autorisation." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=OAuth+2.0">OAuth 2.0</Tooltip> comme **Custom Social Connection** dans le <Tooltip tip="Auth0 Dashboard : principal produit d'Auth0 pour configurer vos services." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip>.

1. Dans le Dashboard, accédez à [Authentication &gt; Social](https://manage.auth0.com/#/connections/social).
2. Sélectionnez **Create Connection**, allez au bas de la liste, puis sélectionnez **Create Custom**.

Le formulaire qui s&#39;affiche contient plusieurs champs que vous devez utiliser pour configurer la connexion personnalisée :

* **Connection Name** : Identifiant logique de la connexion que vous créez. Ce nom ne peut pas être modifié, doit commencer et se terminer par un caractère alphanumérique et ne peut contenir que des caractères alphanumériques et des tirets.
* **Authorization URL** : URL vers laquelle les utilisateurs sont redirigés pour se connecter.

  <Callout icon="file-lines" color="#0EA5E9" iconType="regular">
    N&#39;essayez pas de définir le paramètre OAuth2 `response_mode` dans l&#39;URL d&#39;autorisation. Cette connexion prend uniquement en charge la valeur `response_mode` par défaut (`query`).
  </Callout>
* **Token URL** : URL utilisée pour échanger le code d&#39;autorisation reçu contre des <Tooltip tip="Access Token : information d'identification d'autorisation, sous la forme d'une chaîne opaque ou d'un JWT, utilisée pour accéder à une API." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=access+tokens">jetons d&#39;accès</Tooltip> et, s&#39;il y a lieu, des <Tooltip tip="ID Token : information d'identification destinée au client lui-même plutôt qu'à l'accès à une ressource." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=ID+tokens">jetons ID</Tooltip>.
* **Scope** : Paramètres `scope` à envoyer avec la demande d&#39;autorisation. Séparez plusieurs scopes par des espaces.
* **Separate scopes using a space** : Bascule qui détermine comment les scopes sont délimités si le paramètre `connection_scope` est inclus lors de [l&#39;appel de l&#39;API de l&#39;IdP](/docs/fr-CA/authenticate/identity-providers/calling-an-external-idp-api). Par défaut, les scopes sont délimités par une virgule. Si la bascule est activée, les scopes sont délimités par une espace. Pour en savoir plus, consultez [Add Scopes/Permissions to Call Identity Provider APIs](/docs/fr-CA/authenticate/identity-providers/adding-scopes-for-an-external-idp).
* **<Tooltip tip="Client ID : valeur d'identification attribuée à votre ressource enregistrée par Auth0." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Client+ID">Client ID</Tooltip>** : Client ID d&#39;Auth0 en tant qu&#39;application utilisée pour demander l&#39;autorisation et échanger le code d&#39;autorisation. Pour obtenir un Client ID, vous devrez vous inscrire auprès du <Tooltip tip="Identity Provider (IdP) : service qui stocke et gère les identités numériques." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=identity+provider">fournisseur d&#39;identité</Tooltip>.
* **<Tooltip tip="Client Secret : secret utilisé par un client (application) pour s'authentifier auprès du Authorization Server; il ne doit être connu que du client et du Authorization Server et doit être suffisamment aléatoire pour ne pas pouvoir être deviné." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Client+Secret">Client Secret</Tooltip>** : Client Secret d&#39;Auth0 en tant qu&#39;application utilisée pour échanger le code d&#39;autorisation. Pour obtenir un Client Secret, vous devrez vous inscrire auprès du fournisseur d&#39;identité.
* **Fetch User Profile Script** : Script Node.js utilisé pour appeler une URL userinfo avec le jeton d&#39;accès fourni. Pour en savoir plus sur ce script, consultez [Fetch User Profile Script](#fetch-user-profile-script).
* **Purpose** : Active la connexion sociale pour l&#39;authentication, Connected Accounts for Token Vault, ou les deux. Pour en savoir plus, consultez [User authentication vs Connected Accounts](/docs/fr-CA/secure/tokens/token-vault/connected-accounts-for-token-vault#user-authentication-vs-connected-accounts).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Lorsque vous configurez le fournisseur d’identité personnalisé, utilisez l’URL de rappel `https://{yourDomain}/login/callback`.
</Callout>

Une fois la connexion personnalisée créée, la vue **Applications** s’affichera, et votre connexion sera assujettie à la [Rate Limit Policy](/docs/fr-CA/troubleshoot/customer-support/operational-policies/rate-limit-policy) d’Auth0. Vous pouvez y activer ou désactiver les applications pour lesquelles vous souhaitez que la connexion apparaisse.

<div id="update-authentication-flow">
  ## Mettre à jour le flux d’authentification
</div>

Lorsque vous créez une connexion, le type d’octroi OAuth 2.0 attribué par défaut à la connexion est le flux de code d’autorisation. Si vous avez une application publique incapable de stocker un secret client, comme une application monopage ou une application native, vous pouvez utiliser la <Tooltip tip="Management API : un produit qui permet aux clients d’effectuer des tâches administratives." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Management+API">Management API</Tooltip> pour mettre à jour la connexion afin qu’elle utilise le [flux de code d’autorisation + PKCE](/docs/fr-CA/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce). Pour en savoir plus sur les <Tooltip tip="Flux d’autorisation : mécanisme d’octroi d’autorisation (ou processus) défini dans le framework OAuth 2.0." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=authorization+flows">flux d’autorisation</Tooltip>, consultez [Quel flux OAuth 2.0 devrais-je utiliser ?](/docs/fr-CA/get-started/authentication-and-authorization-flow/which-oauth-2-0-flow-should-i-use)

1. Effectuez une requête `GET` vers le point de terminaison [`/get-connections-by-id`](https://auth0.com/docs/api/management/v2/connections/get-connections-by-id). La réponse ressemblera à ceci :

   ```json lines
   {
     "id": "[connectionID]",
     "options": {
       "email": true,
       "scope": [
         "email",
         "profile"
       ],
       "profile": true
     },
     "strategy": "google-oauth2",
     "name": "google-oauth2",
     "is_domain_connection": false,
     "realms": [
       "google-oauth2"
     ]
   }
   ```

2. Copiez l’objet `options` au complet.

3. Effectuez une requête [`PATCH`](https://auth0.com/docs/api/management/v2/connections/patch-connections-by-id) avec l’objet `options`, puis ajoutez `"pkce_enabled":` `true`.

<Warning>
  Si vous n’incluez pas l’objet `options` au complet, des informations seront perdues et la connexion cessera de fonctionner.
</Warning>

<div id="fetch-user-profile-script">
  ## Script Fetch User Profile
</div>

Le script Fetch User Profile est appelé après que l’utilisateur s’est connecté avec le fournisseur OAuth2. Auth0 exécute ce script pour appeler l’API du fournisseur OAuth2 et obtenir le profil de l’utilisateur :

```javascript lines expandable
function fetchUserProfile(accessToken, context, callback) {
  request.get(
    {
      url: 'https://auth.example.com/userinfo',
      headers: {
        'Authorization': 'Bearer ' + accessToken,
      }
    },
    (err, resp, body) => {
      if (err) {
        return callback(err);
      }
      if (resp.statusCode !== 200) {
        return callback(new Error(body));
      }
      let bodyParsed;
      try {
        bodyParsed = JSON.parse(body);
      } catch (jsonError) {
        return callback(new Error(body));
      }
      const profile = {
        user_id: bodyParsed.account.uuid,
        email: bodyParsed.account.email
      };
      callback(null, profile);
    }
  );
}
```

La propriété `user_id` du profil renvoyé est requise, et la propriété `email` est facultative, mais fortement recommandée. Pour en savoir plus sur les attributs pouvant être renvoyés, consultez [Attributs root du profil utilisateur](/docs/fr-CA/manage-users/user-accounts/user-profiles/root-attributes/update-root-attributes-for-users).

Vous pouvez filtrer, ajouter ou supprimer n’importe quel élément du profil renvoyé par le fournisseur. Cependant, il est recommandé de garder ce script aussi simple que possible. Une gestion plus poussée des renseignements utilisateur peut se faire au moyen des [Rules](/docs/fr-CA/customize/rules). L’un des avantages des Rules est qu’elles s’appliquent à toute connexion.

<div id="pass-dynamic-parameters">
  ### Transmettre des paramètres dynamiques
</div>

Dans certaines circonstances, vous pourriez vouloir transmettre une valeur dynamique à un fournisseur d’identité OAuth 2.0. Dans ce cas, vous pouvez utiliser l’élément `authParamsMap` dans `options` pour définir un mapping entre l’un des paramètres supplémentaires existants acceptés par le [endpoint Auth0 `/authorize`](https://auth0.com/docs/api/authentication#social) et le paramètre accepté par le fournisseur d’identité.

En reprenant l’exemple ci-dessus, supposons que vous vouliez transmettre le paramètre `custom_param` à l’authorization endpoint, mais que vous vouliez définir la valeur réelle du paramètre au moment d’appeler l’endpoint Auth0 `/authorize`.

Dans ce cas, vous pouvez utiliser l’un des paramètres supplémentaires existants acceptés par l’endpoint `/authorize`, comme `access_type`, et le mapper au paramètre `custom_param` :

<div id="pass-extra-headers">
  ## Transmettre des en-têtes supplémentaires
</div>

Dans certains cas, vous devrez transmettre des en-têtes supplémentaires au <Tooltip tip="Token Endpoint : endpoint sur le serveur d’autorisation utilisé pour demander des jetons par programmation." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Token+endpoint">Token endpoint</Tooltip> d’un fournisseur OAuth 2.0. Pour configurer des en-têtes supplémentaires, ouvrez les Settings de la connexion, puis, dans le champ **Custom Headers**, indiquez un objet JSON contenant les en-têtes personnalisés sous forme de paires clé-valeur :

```json lines
{
    "Header1" : "Value",
    "Header2" : "Value"
}
```

Prenons l’exemple d’un fournisseur d’identité qui peut exiger que vous transmettiez un en-tête `Authorization` avec des identifiants d’[authentification Basic access](https://en.wikipedia.org/wiki/Basic_access_authentication). Dans ce cas, vous pouvez indiquer l’objet JSON suivant dans le champ **En-têtes personnalisés** :

```json lines
{
  "Authorization": "Basic [your credentials]"
}
```

Où `[your credentials]` correspond aux véritables informations d’authentification à transmettre au fournisseur d’identité.
