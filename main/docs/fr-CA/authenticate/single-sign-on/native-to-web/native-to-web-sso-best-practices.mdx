Pour tirer le meilleur parti de Native to Web <Tooltip tip="Authentification unique (SSO) : service qui, après qu’un utilisateur se connecte à une application, le connecte automatiquement à d’autres applications." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=SSO">SSO</Tooltip>, Auth0 recommande de suivre les meilleures pratiques ci-dessous. Ces recommandations peuvent contribuer à assurer l’intégrité de la session :

<div id="use-post-login-actions-to-limit-session-lifetime">
  ## Utilisez les Actions Post Login pour limiter la durée de vie de la session
</div>

Utilisez les déclencheurs d’Action [`post-login`](/docs/fr-CA/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger) pour vous assurer que les sessions web créées au moyen de Native to Web SSO sont limitées dans le temps de façon appropriée et expirent rapidement lorsqu’elles sont inactives. 
Vous pouvez utiliser les Actions `post-login` pour détecter lorsqu’une session est établie au moyen d’un `session_transfer_token` et appliquer des délais d’inactivité et d’expiration absolue plus courts :

```swift lines
exports.onExecutePostLogin = async (event, api) => {
  if (event.session_transfer_token) {
    const now = Date.now();

    // Limiter la session à 30 minutes au total
    api.session.setExpiresAt(now + 30 * 60 * 1000);

    // Définir le délai d'inactivité à 15 minutes
    api.session.setIdleExpiresAt(now + 15 * 60 * 1000);
  }
};
```

<div id="bind-session_transfer_token-to-the-device-or-ip-address">
  ## Associer session_transfer_token à l’appareil ou à l’adresse IP
</div>

Pour réduire le risque de rejeu de token si un token est divulgué, journalisé ou intercepté, associez toujours le [`session_transfer_token`](https://auth0.com/docs/api/management/v2/clients/patch-clients-by-id)à l’environnement d’origine à l’aide de `enforce_device_binding`.

<div id="use-secure-cookies-over-query-parameters">
  ## Privilégiez les cookies sécurisés aux paramètres de requête
</div>

Envoyez le `session_transfer_token` à l’application web au moyen d’un cookie sécurisé et HTTPOnly limité à votre domaine Auth0 afin d’éviter la consignation accidentelle du jeton ou son partage dans les URL, et de réduire la surface d’attaque liée à son interception. Si vous devez utiliser un paramètre de requête (par exemple, pour Chrome Custom Tabs), assurez-vous que l’URL utilise HTTPS et retirez le jeton de l’URL après utilisation.

<div id="avoid-issuing-refresh-tokens-to-web-apps-unless-necessary">
  ## Évitez d’émettre des jetons d’actualisation aux applications Web, sauf si nécessaire
</div>

N’activez [`allow_refresh_token`](https://auth0.com/docs/api/management/v2/clients/patch-clients-by-id) que pour les applications Web qui ont réellement besoin de jetons de longue durée. Dans la plupart des cas, des <Tooltip tip="Jeton d’accès : justificatif d’autorisation, sous la forme d’une chaîne opaque ou d’un JWT, utilisé pour accéder à une API." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=access+tokens">jetons d’accès</Tooltip> de courte durée, combinés à l’authentification silencieuse, suffisent et sont plus sécuritaires dans un navigateur.

<div id="enable-allow-refresh-tokens-when-appropriate-to-set-the-refresh-tokens-as-online">
  ## Activez Allow Refresh Tokens au besoin pour définir les jetons d’actualisation comme « online »
</div>

Pour éviter les données d’authentification orphelines et empêcher les <Tooltip tip="Refresh Token : jeton utilisé pour obtenir un nouveau jeton d’accès sans obliger les utilisateurs à se connecter de nouveau." cta="Consulter le glossaire" href="/docs/fr-CA/glossary?term=refresh+tokens">jetons d’actualisation</Tooltip> de rester actifs après la déconnexion, utilisez le paramètre [`allow_refresh_token`](https://auth0.com/docs/api/management/v2/clients/patch-clients-by-id) pour vous assurer que les jetons d’actualisation émis au moyen de Native to Web SSO sont liés à la session dont ils proviennent. Si la session est révoquée ou expire, le jeton d’actualisation est automatiquement invalidé.

<div id="enable-enforce-cascade-revocation">
  ## Activer la révocation en cascade forcée
</div>

Pour que toutes les sessions web et tous les jetons d’actualisation associés à un `session_transfer_token` soient révoqués, activez [`enforce_cascade_revocation`](https://auth0.com/docs/api/management/v2/clients/patch-clients-by-id) dans l’application native. Cela est essentiel pour garantir l’invalidation sécurisée des sessions d’une application à l’autre.
