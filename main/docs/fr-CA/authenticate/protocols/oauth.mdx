<Card title="Overview">
  Concepts clés

  * Auth0 prend en charge le protocole OAuth 2.0 élaboré par l&#39;Internet Engineering Task Force (IETF).
  * Découvrez les rôles, les types d&#39;octroi (ou workflows) et les points de terminaison de la spécification OAuth 2.0.
</Card>

Le [cadre d&#39;autorisation OAuth 2.0](https://tools.ietf.org/html/rfc6749) est un protocole qui permet à un utilisateur d&#39;accorder à un site Web ou à une application tierce l&#39;accès à ses ressources protégées, sans nécessairement divulguer ses identifiants à long terme ni même son identité.

<Tooltip tip="OAuth 2.0: Authorization framework that defines authorization protocols and workflows." cta="View Glossary" href="/docs/fr-CA/glossary?term=OAuth">
  OAuth
</Tooltip>

introduit une couche d&#39;autorisation et sépare le rôle du client de celui du

<Tooltip tip="propriétaire de la ressource: Entity (such as a user or application) capable of granting access to a protected resource." cta="View Glossary" href="/docs/fr-CA/glossary?term=resource+owner">
  resource owner
</Tooltip>

. Dans OAuth, le client demande l&#39;accès aux ressources contrôlées par le resource owner et hébergées par le

<Tooltip tip="serveur de ressources: Server hosting protected resources. Resource servers accept and respond to protected resource requests." cta="View Glossary" href="/docs/fr-CA/glossary?term=resource+server">
  resource server
</Tooltip>

et se voit remettre un ensemble d&#39;identifiants différent de celui du resource owner. Plutôt que d&#39;utiliser les identifiants du resource owner pour accéder aux ressources protégées, le client obtient un

<Tooltip tip="Access Token: Authorization credential, in the form of an opaque string or JWT, used to access an API." cta="View Glossary" href="/docs/fr-CA/glossary?term=access+token">
  jeton d’accès
</Tooltip>

— une chaîne de caractères représentant un scope particulier, une durée de vie et d&#39;autres attributs d&#39;accès. Les jetons d’accès sont émis aux clients tiers par un

<Tooltip tip="serveur d’autorisation: Centralized server that contributes to defining the boundaries of a user’s access. For example, your authorization server can control the data, tasks, and features available to a user." cta="View Glossary" href="/docs/fr-CA/glossary?term=authorization+server">
  authorization server
</Tooltip>

avec l&#39;approbation du resource owner. Le client utilise ensuite le jeton d’accès pour accéder aux ressources protégées hébergées par le resource server.

Auth0 génère des jetons d’accès pour les scénarios d&#39;autorisation d&#39;API, au [format JSON Web Token (JWT)](/docs/fr-CA/secure/tokens/json-web-tokens/json-web-token-structure). Les permissions représentées par le jeton d’accès, en termes OAuth, sont appelées [scopes](/docs/fr-CA/get-started/apis/scopes). Lorsqu&#39;une application s&#39;authentifie auprès d&#39;Auth0, elle précise les scopes qu&#39;elle souhaite obtenir. Si ces scopes sont autorisés par l&#39;utilisateur, le jeton d’accès représentera alors ces scopes autorisés.

<div id="roles">
  ## Rôles
</div>

Un flux OAuth 2.0 comprend les rôles suivants :

* **Propriétaire de la ressource** : entité qui peut accorder l&#39;accès à une ressource protégée. Il s&#39;agit généralement de l&#39;utilisateur final.
* **Serveur de ressources** : serveur qui héberge les ressources protégées. Il s&#39;agit de l&#39;API à laquelle vous voulez accéder.
* **Client** : application qui demande l&#39;accès à une ressource protégée au nom du propriétaire de la ressource.
* **Serveur d&#39;autorisation** : serveur qui authentifie le propriétaire de la ressource et émet des jetons d&#39;accès après avoir obtenu l&#39;autorisation nécessaire. Dans ce cas-ci, il s&#39;agit d&#39;Auth0.

<div id="grant-types">
  ## Types d’autorisation
</div>

OAuth 2.0 définit quatre flux pour obtenir un jeton d’accès. Ces flux sont appelés des types d’octroi. [Le choix de celui qui convient à votre cas](/docs/fr-CA/get-started/authentication-and-authorization-flow/which-oauth-2-0-flow-should-i-use) dépend surtout de votre type d’application.

* [Flux du code d’autorisation](/docs/fr-CA/get-started/authentication-and-authorization-flow/authorization-code-flow) : utilisé par les applications Web exécutées sur un serveur. Il est aussi utilisé par les applications mobiles, au moyen de la technique [clé de preuve pour l’échange de code (PKCE)](/docs/fr-CA/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce).
* [Flux implicite avec envoi de formulaire](/docs/fr-CA/get-started/authentication-and-authorization-flow/implicit-flow-with-form-post) : utilisé par les applications axées sur JavaScript (applications monopages) exécutées dans le navigateur de l’utilisateur.
* [Flux du mot de passe du propriétaire de la ressource](/docs/fr-CA/get-started/authentication-and-authorization-flow/resource-owner-password-flow) : utilisé par les applications de grande confiance.
* [Flux des informations d’identification du client](/docs/fr-CA/get-started/authentication-and-authorization-flow/client-credentials-flow) : utilisé pour la communication machine à machine.

La spécification prévoit aussi un mécanisme d’extensibilité permettant de définir des types d’octroi supplémentaires. Pour en savoir plus sur le fonctionnement de chaque type d’octroi et sur les situations où il doit être utilisé, consultez [Flux d’authentification et d’autorisation](/docs/fr-CA/get-started/authentication-and-authorization-flow).

<div id="endpoints">
  ## Points de terminaison
</div>

OAuth 2.0 utilise deux points de terminaison : `/authorize` et `/oauth/token`.

<div id="authorization-endpoint">
  ### Point de terminaison d’autorisation
</div>

Le point de terminaison `/authorize` sert à interagir avec le propriétaire de la ressource et à obtenir l’autorisation d’accéder à la ressource protégée. Pour mieux comprendre, imaginez que vous voulez vous connecter à un service à l’aide de votre compte Google. D’abord, le service vous redirige vers Google pour vous authentifier (si vous n’êtes pas déjà connecté), puis un écran de consentement s’affiche, où l’on vous demande d’autoriser le service à accéder à certaines de vos données (ressources protégées), par exemple votre adresse courriel et votre liste de contacts.

Les paramètres de requête du point de terminaison `/authorize` sont les suivants :

| Paramètre       | Description                                                                                                                                                                                                                                                                                                                                                                                                                                                                              |
| --------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `response_type` | Indique au serveur d’autorisation quel type d’octroi exécuter.                                                                                                                                                                                                                                                                                                                                                                                                                          |
| `response_mode` | (Facultatif) Indique comment le résultat de la demande d’autorisation est formaté. Valeurs :<br />- `query` : pour le type d’octroi Authorization Code. `302 Found` déclenche une redirection.<br />- `fragment` : pour le type d’octroi Implicit. `302 Found` déclenche une redirection.<br />- `form_post` : `200 OK` avec les paramètres de réponse intégrés dans un formulaire HTML comme paramètres masqués.<br />- `web_message` : pour la Silent Authentication. Utilise la messagerie web HTML5. |
| `client_id`     | L’ID de l’application qui demande l’autorisation.                                                                                                                                                                                                                                                                                                                                                                                                                                        |
| `redirect_uri`  | Contient une URL. Une réponse réussie de ce point de terminaison entraîne une redirection vers cette URL.                                                                                                                                                                                                                                                                                                                                                                                |
| `scope`         | Une liste de permissions séparées par des espaces dont l’application a besoin.                                                                                                                                                                                                                                                                                                                                                                                                           |
| `state`         | Une valeur opaque, utilisée à des fins de sécurité. Si ce paramètre de requête est défini dans la demande, il est renvoyé à l’application dans le cadre du `redirect_uri`.                                                                                                                                                                                                                                                                                                               |
| `connection`    | Spécifie le type de connection pour les connections Passwordless                                                                                                                                                                                                                                                                                                                                                                                                                         |

Vous pouvez configurer des paramètres de requête personnalisés lorsque votre application effectue l’appel initial au point de terminaison `/authorize` pour authentifier un utilisateur. Vous pouvez utiliser des paramètres de requête personnalisés pour fournir un contexte supplémentaire au gabarit de page de l’expérience <Tooltip tip="Universal Login : votre application redirige vers Universal Login, hébergé sur le serveur d’autorisation d’Auth0, afin de vérifier l’identité d’un utilisateur." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Universal+Login">Universal Login</Tooltip>.

Vous devez activer ID First pour utiliser le paramètre `connection`. Pour en savoir plus sur le paramètre `connection` et l’expérience Universal Login, consultez [Passwordless for Universal Login](/docs/fr-CA/authenticate/passwordless/passwordless-with-universal-login).

Les paramètres de requête préfixés par `ext-` apparaissent automatiquement dans le [contexte du gabarit de page](/docs/fr-CA/customize/login-pages/universal-login/customize-templates#custom-query-parameters).

Ce point de terminaison est utilisé par les types d’octroi Authorization Code et Implicit. Le serveur d’autorisation doit savoir quel type d’octroi l’application veut utiliser, puisque cela a une incidence sur le type d’identifiant qu’il émettra :

* Pour le type d’octroi Authorization Code, il émet un authorization code (qui peut ensuite être échangé contre un jeton d’accès au point de terminaison `/oauth/token`).
* Pour le type d’octroi Implicit, il émet un jeton d’accès, qui est une chaîne opaque (ou un <Tooltip tip="JSON Web Token (JWT) : format standard de ID Token (et souvent du jeton d’accès) utilisé pour représenter des revendications de manière sécurisée entre deux parties." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=JWT">JWT</Tooltip> dans une mise en œuvre Auth0) qui indique qui a autorisé quelles permissions (scopes) à quelle application.

Pour indiquer au serveur d’autorisation quel type d’octroi utiliser, le paramètre de requête `response_type` est utilisé comme suit :

* Pour le type d’octroi Authorization Code, utilisez `response_type=code` pour inclure le authorization code.
* Pour le type d’octroi Implicit, utilisez `response_type=token` pour inclure un jeton d’accès. Une autre possibilité consiste à utiliser `response_type=id_token token` pour inclure à la fois un jeton d’accès et un <Tooltip tip="ID Token : identifiant destiné au client lui-même, plutôt qu’à l’accès à une ressource." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=ID+token">ID token</Tooltip>.

Un ID token est un JWT qui contient des renseignements sur l’utilisateur connecté. Il a été introduit par <Tooltip tip="OpenID : norme ouverte d’authentification qui permet aux applications de vérifier l’identité des utilisateurs sans recueillir ni stocker d’information de connexion." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=OpenID">OpenID</Tooltip> Connect (OIDC).

La [spécification OAuth 2.0 Multiple Response Type Encoding Practices](https://openid.net/specs/oauth-v2-multiple-response-types-1_0.html) a ajouté un paramètre qui précise comment le résultat de la demande d’autorisation est formaté. Ce paramètre s’appelle `response_mode`. Il est facultatif et peut prendre les valeurs suivantes :

| Valeur        | Description                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| ------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `query`       | Il s’agit du mode par défaut pour le type d’octroi Authorization Code. Une réponse réussie est `302 Found`, ce qui déclenche une redirection vers le `redirect_uri`. Les paramètres de la réponse sont inclus dans la composante de requête (la partie après `?`) du `redirect_uri`, dans l’en-tête `Location`.<br />Par exemple :<br />`HTTP/1.1 302 Found`<br />`Location: https://my-redirect-uri.callback?code=js89p2x1` où le code d’autorisation est `js89p21`.                                                                                                                                                    |
| `fragment`    | Il s’agit du mode par défaut pour le type d’octroi Implicit. Une réponse réussie est `302 Found`, ce qui déclenche une redirection vers le `redirect_uri` (qui est un paramètre de requête). Les paramètres de la réponse sont inclus dans la composante fragment (la partie après `#`) du `redirect_uri`, dans l’en-tête `Location`.<br />Par exemple :<br />`HTTP/1.1 302 Found`<br />`Location: https://my-redirect-uri/callback#access_token=eyB...78f&token_type=Bearer&expires_in=3600`.                                                                                                                           |
| `form_post`   | Le mode de réponse est défini par la [spécification OAuth 2.0 Form Post Response Mode](https://openid.net/specs/oauth-v2-form-post-response-mode-1_0.html). Une réponse réussie est `200 OK`, et les paramètres sont inclus dans un formulaire HTML en tant que params masqués. L’`action` du formulaire correspond au `redirect_uri`, et l’attribut `onload` est configuré pour soumettre le formulaire. Une fois le HTML chargé par le navigateur, une redirection vers le `redirect_uri` est effectuée.                                                                                                       |
| `web_message` | Ce mode de réponse est défini dans la [spécification OAuth 2.0 Web Message Response Mode](https://tools.ietf.org/html/draft-sakimura-oauth-wmrm-00). Il utilise la messagerie Web HTML5 au lieu de la redirection pour la réponse d’autorisation provenant du point de terminaison `/authorization`. C’est particulièrement utile lorsque vous utilisez l’authentification silencieuse. Pour utiliser ce mode de réponse, vous devez enregistrer l’URL de votre application dans le champ **Allowed Web Origins** des [paramètres de l’application](https://manage.auth0.com/#/applications/\{yourClientId}/settings) Auth0. |

<div id="token-endpoint">
  ### Point de terminaison des jetons
</div>

Le point de terminaison `/oauth/token` est utilisé par l’application pour obtenir un jeton d’accès ou un <Tooltip tip="Refresh Token : jeton utilisé pour obtenir un nouveau jeton d’accès sans obliger les utilisateurs à se reconnecter." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=refresh+token">jeton d’actualisation</Tooltip>. Il est utilisé par tous les flux, sauf le flux implicite, car dans ce cas, un jeton d’accès est émis directement.

* Dans le flux du code d’autorisation, l’application échange le code d’autorisation qu’elle a obtenu du point de terminaison d’autorisation contre un jeton d’accès.
* Dans le flux des informations d’identification du client et l’échange d’octroi par mot de passe du propriétaire de la ressource, l’application s’authentifie à l’aide d’un ensemble d’informations d’identification, puis obtient un jeton d’accès.

<div id="state-parameters">
  ## Paramètres d’état
</div>

Les protocoles d’autorisation fournissent un paramètre `state` qui vous permet de rétablir l’état précédent de votre application. Le paramètre `state` conserve un objet d’état défini par le client dans la requête d’autorisation et le met à la disposition du client dans la réponse. La principale raison d’utiliser le paramètre `state` est de prévenir les attaques CSRF. Consultez [Utiliser les paramètres d’état OAuth 2.0](/docs/fr-CA/secure/attack-protection/state-parameters) pour en savoir plus.
