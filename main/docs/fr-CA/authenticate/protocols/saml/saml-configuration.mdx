<div id="saml-service-providers">
  ## Fournisseurs de services SAML
</div>

Les applications, surtout les applications personnalisées, peuvent authentifier les utilisateurs auprès d’un <Tooltip tip="Fournisseur d’identité (IdP) : service qui stocke et gère les identités numériques." cta="Consulter le glossaire" href="/docs/fr-CA/glossary?term=IdP">IdP</Tooltip> externe au moyen de protocoles comme <Tooltip tip="OpenID : norme ouverte d’authentification qui permet aux applications de vérifier l’identité des utilisateurs sans recueillir ni stocker les renseignements de connexion." cta="Consulter le glossaire" href="/docs/fr-CA/glossary?term=OpenID">OpenID</Tooltip> Connect (OIDC) ou <Tooltip tip="OpenID : norme ouverte d’authentification qui permet aux applications de vérifier l’identité des utilisateurs sans recueillir ni stocker les renseignements de connexion." cta="Consulter le glossaire" href="/docs/fr-CA/glossary?term=OAuth+2.0">OAuth 2.0</Tooltip>. Toutefois, il se peut que vous souhaitiez utiliser un fournisseur <Tooltip tip="Security Assertion Markup Language (SAML) : protocole normalisé permettant à deux parties d’échanger des renseignements d’authentification sans mot de passe." cta="Consulter le glossaire" href="/docs/fr-CA/glossary?term=SAML">SAML</Tooltip> d’entreprise pour l’authentification, même si votre application a été conçue pour utiliser l’un ou l’autre de ces protocoles.

<div id="saml-identity-providers">
  ## Fournisseurs d&#39;identité SAML
</div>

Certaines applications (comme Salesforce, Box et Workday) permettent aux utilisateurs de s&#39;authentifier auprès d&#39;un IdP externe au moyen du protocole SAML. Vous pouvez ensuite intégrer l&#39;application à Auth0, qui agit comme l&#39;IdP SAML de l&#39;application. Les utilisateurs de l&#39;application seront redirigés vers Auth0 pour se connecter, et Auth0 peut les authentifier à l&#39;aide de n&#39;importe quelle connexion d&#39;authentification backend, comme un annuaire LDAP, une base de données ou un autre IdP SAML ou fournisseur social. Une fois l&#39;utilisateur authentifié, Auth0 renvoie à l&#39;application une assertion SAML l&#39;indiquant.

<div id="auth0-as-service-provider">
  ## Auth0 comme fournisseur de services
</div>

Si Auth0 agit comme fournisseur de services dans une fédération SAML, Auth0 peut acheminer les requêtes d’authentification vers un fournisseur d’identité sans qu’un compte ait déjà été créé pour un utilisateur précis. À l’aide de l’assertion renvoyée par le fournisseur d’identité, Auth0 peut recueillir les renseignements nécessaires pour créer un profil utilisateur (ce processus est parfois appelé provisionnement juste à temps). Pour en savoir plus, consultez [Sélectionner parmi plusieurs options de connexion](/docs/fr-CA/libraries/lock/selecting-from-multiple-connection-options).

Même si Auth0 n’exige pas que des comptes d’utilisateur soient créés d’avance avant le processus d’authentification, l’application intégrée à Auth0 pourrait l’exiger. Si c’est le cas, vous avez plusieurs façons de gérer cette situation :

* Une fois que le fournisseur d’identité a créé l’utilisateur, vous pouvez recourir à un processus hors bande pour créer le compte utilisateur correspondant dans l’application (ou dans Auth0) et y ajouter tous les attributs du profil utilisateur exigés par l’application. Si, après l’authentification, certains attributs sont absents du profil, l’application peut les obtenir de la source appropriée et les enregistrer dans le profil utilisateur Auth0. Ces attributs supplémentaires seront ensuite transmis à l’application (en plus de ceux ajoutés par le fournisseur d’identité) lors de la prochaine connexion de l’utilisateur.
* Vous pouvez utiliser une règle Auth0 pour appeler une API afin de récupérer toute information manquante et l’ajouter dynamiquement au profil Auth0 (qui est ensuite renvoyé à l’application). Les règles s’exécutent après une authentification réussie, et votre application peut récupérer les attributs du profil à chaque fois, ou vous pouvez les enregistrer dans le profil Auth0.
* Auth0 peut transmettre à l’application les renseignements de base du profil fournis par le fournisseur d’identité, puis l’application récupère toute information manquante à partir d’une autre source. À partir de ces deux ensembles de renseignements, l’application crée un profil utilisateur local.

Vous pouvez préciser des domaines de courriel dans la configuration de la connexion Auth0 SAMLP pour contrôler l’IdP qui gère un groupe précis d’utilisateurs. Par exemple, si vous ajoutez le domaine de courriel `example.com` à la configuration de la connexion Auth0 SAMLP de Company X, tous les utilisateurs dont l’adresse courriel appartient au domaine `example.com` seront pris en charge par l’IdP propre à Company X.

<div id="auth0-as-identity-provider">
  ## Auth0 comme fournisseur d’identité
</div>

Si Auth0 agit comme fournisseur d’identité dans une fédération SAML, les comptes d’utilisateur peuvent être créés de plusieurs façons :

* À l’aide d’un système d’authentification back-end, comme un annuaire LDAP, une base de données ou un autre fournisseur d’identité SAML.
* À l’aide du <Tooltip tip="Auth0 Dashboard : produit principal d’Auth0 pour configurer vos services." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip>.
* En appelant la <Tooltip tip="Management API : un produit permettant aux clients d’effectuer des tâches administratives." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Management+API">Management API</Tooltip> d’Auth0.
* En mettant en place l’inscription libre-service des utilisateurs.

Si votre application est conçue pour récupérer les renseignements du profil utilisateur à partir d’un stockage local, vous devrez créer le profil local une fois les comptes créés dans Auth0. Voici quelques façons de procéder :

* Un processus hors bande qui crée des profils utilisateur dans l’application ;
* Une règle Auth0 qui s’exécute lors de la première connexion et appelle une API de l’application pour y créer le profil utilisateur ;
* Modifier l’application afin de créer des profils utilisateur de façon dynamique, à partir de l’information contenue dans l’assertion SAML.
