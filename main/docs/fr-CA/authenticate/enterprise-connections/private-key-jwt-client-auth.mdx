L’authentification client Private Key <Tooltip tip="JSON Web Token (JWT) : format standard d’ID Token (et souvent format d’Access Token) utilisé pour représenter des claims de façon sécurisée entre deux parties." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=JWT">JWT</Tooltip> est une autre méthode d’authentification du client pour les connexions d’entreprise <Tooltip tip="JSON Web Token (JWT) : format standard d’ID Token (et souvent format d’Access Token) utilisé pour représenter des claims de façon sécurisée entre deux parties." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=OpenID">OpenID</Tooltip> Connect (OIDC) et Okta Workforce. Alors que l’authentification du client se fait le plus souvent en transmettant un <Tooltip tip="Client Secret : secret utilisé par un client (application) pour s’authentifier auprès du Authorization Server; il ne doit être connu que du client et du Authorization Server et doit être suffisamment aléatoire pour ne pas pouvoir être deviné." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=client+secret">secret client</Tooltip> partagé, l’authentification client Private Key JWT utilise plutôt un JWT signé pour renforcer la sécurité de l’application.

En utilisant cette fonctionnalité, vous pouvez éviter certaines lacunes de sécurité courantes souvent observées avec l’authentification standard par secret client, notamment :

* Un risque accru d’interception et de réutilisation, puisque les secrets clients doivent être transmis entre les parties pour chaque requête.
* Des mécanismes limités pour faire respecter l’expiration et empêcher la réutilisation par des acteurs malveillants.
* Un risque accru de fuites ou d’exposition, puisque les deux parties détiennent le secret client.

Vous pouvez configurer l’authentification client Private Key JWT pour vos connexions d’entreprise OIDC et Okta Workforce au moyen du <Tooltip tip="Auth0 Dashboard : principal produit d’Auth0 pour configurer vos services." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip> ou de la <Tooltip tip="Auth0 Dashboard : principal produit d’Auth0 pour configurer vos services." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Management+API">Management API</Tooltip>.

<div id="how-it-works">
  ## Comment ça fonctionne
</div>

Le flux de connexion OIDC utilise des endpoints authentifiés comme `/oauth/token` ou `/oauth/par` pour vérifier l’identité d’un client auprès du <Tooltip tip="Serveur d’autorisation : serveur centralisé qui contribue à définir les limites de l’accès d’un utilisateur. Par exemple, votre serveur d’autorisation peut contrôler les données, les tâches et les fonctionnalités offertes à un utilisateur." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=authorization+server">serveur d’autorisation</Tooltip> ou du fournisseur OpenID. Avec Private Key JWT Client Authentication, une assertion JWT client signée est transmise au fournisseur OpenID au lieu d’un secret client.

L’assertion JWT client contient les claims suivantes :

* Un `aud` (<Tooltip tip="Audience : identifiant unique de l’audience d’un token émis. Nommée aud dans un token, sa valeur contient l’ID d’une application (Client ID) pour un ID Token ou d’une API (API Identifier) pour un Access Token." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=audience">audience</Tooltip>) qui identifie le <Tooltip tip="Audience : identifiant unique de l’audience d’un token émis. Nommée aud dans un token, sa valeur contient l’ID d’une application (Client ID) pour un ID Token ou d’une API (API Identifier) pour un Access Token." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=token+endpoint">point de terminaison de jeton</Tooltip> du fournisseur OpenID.
* Un `jti` (ID JWT) pour permettre un usage unique ou assurer une protection contre la réutilisation.
* Un `exp` (heure d’expiration) qui limite la période de validité du token.
* Un `sub` et un `iss` qui identifient le <Tooltip tip="Client ID : valeur d’identification attribuée à votre ressource enregistrée par Auth0." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=client+ID">client ID</Tooltip>.

Private Key JWT Client Authentication offre une méthode d’authentification plus sécuritaire en éliminant l’utilisation de secrets client partagés. Le JWT est plutôt signé à l’aide de la clé privée du client, et le fournisseur OpenID n’a accès qu’à la clé publique.

<div id="private-key-jwt-client-authentication-flow">
  ### Flux d’authentification du client avec Private Key JWT
</div>

Une fois l’authentification d’un utilisateur auprès d’un <Tooltip tip="Fournisseur d’identité (IdP) : service qui stocke et gère les identités numériques." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=identity+provider">fournisseur d’identité</Tooltip> (IdP) en amont terminée, l’utilisateur est redirigé vers Auth0 avec un code d’autorisation, qui est échangé contre des jetons au point de terminaison de jeton du fournisseur OpenID. Lorsque Private Key JWT est activé pour une connexion, l’appel au point de terminaison de jeton du fournisseur OpenID utilise une assertion client plutôt qu’un secret client, pour une authentification plus sécurisée.

Les étapes suivantes présentent un flux d’authentification du client avec Private Key JWT typique.

<Frame>![Schéma illustrant le flux d’authentification du client avec Private Key JWT. ](/docs/images/cdy7uua7fh8z/4JOxbdG7aweqpDHCJ8VOZq/aea18936c0cbdbfbb39d0cd79af3987e/Client_Assertion_JWT_-_Diagram.png)</Frame>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Pour effectuer ce flux, vous devez d’abord configurer une connexion OIDC ou Okta Workforce, nouvelle ou existante, avec `token_endpoint_auth_method=private_key_jwt`, à partir de votre Auth0 Dashboard ou de la Management API. Pour en savoir plus, consultez la section [Configurer l’authentification du client avec Private Key JWT](#configure-private-key-jwt-client-authentication).
</Callout>

1. Après avoir configuré votre connexion, Auth0 génère et stocke automatiquement deux paires de clés publiques et privées.

   * Une paire de clés correspond à l’ensemble actif `current`, tandis que l’autre est libellée `next` pour prendre en charge la [rotation des clés](#rotate-signing-keys).
2. Selon votre IdP, vous devez ensuite soit :

   * télécharger la clé publique `current` et téléverser le fichier sur le serveur d’autorisation, soit :
   * copier et coller le [`jwks_uri`](#retrieve-signing-keys) dans le serveur d’autorisation.
3. Un utilisateur effectue une action qui nécessite une authentification, comme ouvrir une session dans votre application.
4. Auth0 envoie une requête au serveur d’autorisation pour lancer l’authentification.
5. Le serveur d’autorisation affiche à l’utilisateur les écrans d’authentification et de consentement.
6. L’utilisateur s’authentifie et donne son consentement au serveur d’autorisation.
7. Le serveur d’autorisation envoie un code d’autorisation à Auth0.
8. Auth0 génère une assertion JWT client et la signe à l’aide de la clé privée `current`.
9. Auth0 transmet l’assertion JWT client au serveur d’autorisation.
10. Le serveur d’autorisation recherche le client à partir du `client_id` fourni.
11. Le serveur d’autorisation récupère les clés publiques auprès d’Auth0 si un `jwks_uri` a été fourni; sinon, il repère la clé publique enregistrée à l’étape 2.
12. Si le `jwks_uri` a été demandé, Auth0 renvoie les clés publiques au format JWKS.
13. Le serveur d’autorisation valide le JWT en vérifiant la signature à l’aide de la clé publique `current`, identifiée par `kid` dans l’en-tête du JWT `client_assertion`.
14. Le serveur d’autorisation génère un jeton d’accès.
15. Le serveur d’autorisation transmet le jeton d’accès à Auth0.
16. À l’aide du jeton d’accès, Auth0 demande une ressource au serveur de ressources.
17. Le serveur de ressources fournit la ressource pour terminer le flux.

<div id="configure-private-key-jwt-client-authentication">
  ## Configurer Private Key JWT Client Authentication
</div>

Vous pouvez configurer des connexions d’entreprise OIDC et Okta Workforce pour utiliser Private Key JWT Client Authentication à partir de l’Auth0 Dashboard ou de la Management API. Les étapes pour chaque méthode sont décrites ci-dessous.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  * Auth0 génère automatiquement une paire de clés de signature privée et publique pour chaque connexion.
  * Vous pouvez utiliser les algorithmes suivants pour signer les JWT d’assertion du client : `RS256`, `RS384`, `RS512`, `PS256`, `PS384`, `ES256` et `ES384` pour les connexions d’entreprise Okta et OIDC. La valeur par défaut est `RS256` si elle n’est pas précisée.
  * Les JWT signés expirent automatiquement après 60 secondes.
</Callout>

<div id="auth0-dashboard">
  #### Auth0 Dashboard
</div>

Vous pouvez utiliser Auth0 Dashboard pour configurer Private Key JWT Client Authentication pour des connexions OIDC et Okta Workforce, nouvelles ou existantes.

<Tabs>
  <Tab title="Nouvelle connexion">
    1. Dans Auth0 Dashboard, accédez à [Authentication &gt; Enterprise](https://manage.auth0.com/#/connections/enterprise).
    2. À côté de **OpenID Connect** ou **Okta Workforce**, sélectionnez **Create**.
    3. Dans la section **General**, entrez les détails de votre nouvelle connexion, y compris son nom et l’URL de découverte.
    4. Configurez les champs suivants pour activer Private Key JWT :

       * Réglez **Communication Channel** sur **Back Channel**.
       * Réglez **Authentication Method** sur **Private Key JWT**.
    5. Sélectionnez **Create** pour enregistrer votre nouvelle connexion.
  </Tab>

  <Tab title="Connexion existante">
    1. Dans Auth0 Dashboard, accédez à [Authentication &gt; Enterprise](https://manage.auth0.com/#/connections/enterprise).
    2. À côté de **OpenID Connect** ou **Okta Workforce**, sélectionnez **Browse**.
    3. Choisissez la connexion appropriée, puis accédez à son onglet **Credentials**.
    4. Sous **Authentication Settings**, configurez les éléments suivants :

       * Réglez **Communication Channel** sur **Back Channel**.
       * Réglez **Authentication Method** sur **Private Key JWT**.
    5. Sélectionnez **Save**.
    6. Dans la fenêtre contextuelle de confirmation, sélectionnez **Change** pour appliquer vos modifications.
  </Tab>
</Tabs>

  #### Management API
</div>

Vous pouvez utiliser la Management API pour configurer Private Key JWT Client Authentication pour les connexions OIDC nouvelles et existantes.

<Tabs>
  <Tab title="Nouvelle connexion">
    Pour créer une nouvelle connexion OIDC qui utilise Private Key JWT Client Authentication, appelez le point de terminaison [Create a Connection](https://auth0.com/docs/api/management/v2/connections/post-connections) en définissant correctement les propriétés `connection.options` suivantes :

    | `token_endpoint_auth_signing_alg` | Facultatif. Algorithme utilisé pour signer les assertions du client. Valeurs acceptées : `RS256`, `RS384`, `RS512`, `PS256`, `PS384`, `ES256`, `ES384`. Lorsqu’elle est définie, Auth0 rejette les ID tokens signés avec un algorithme qui ne figure pas dans cette liste. Valeurs acceptées : `RS256`, `RS384`, `RS512`, `PS256`, `PS384`, `ES256`, `ES384`. Si elle n’est pas définie, Auth0 accepte les ID tokens signés avec n’importe quel algorithme pris en charge. S’applique uniquement aux stratégies `oidc` et `okta`. |

La valeur par défaut est `token_endpoint`.                                                                                                                                           |

<div id="rotate-signing-keys">
  ## Rotation des clés de signature
</div>

Private Key JWT Client Authentication prend en charge la rotation des clés de signature, ce qui offre une sécurité accrue par rapport au caractère statique et durable des secrets client partagés. La rotation des clés de signature améliore la sécurité en limitant la durée d’exposition d’une même clé, ce qui réduit la possibilité qu’un attaquant la compromette. Elle permet aussi de réagir rapidement à la suite d’un incident de sécurité.

Pour éviter toute interruption, Auth0 recommande d’effectuer une rotation des clés de signature après un an. Vous pouvez utiliser soit l’Auth0 Dashboard, soit la Management API pour effectuer une rotation des clés de signature :

<AccordionGroup>
  <Accordion title="Auth0 Dashboard">
    Pour effectuer une rotation de vos clés de signature dans l’Auth0 Dashboard :

    1. Accédez à [Authentication &gt; Enterprise](https://manage.auth0.com/#/connections/enterprise).
    2. À côté de **OpenID Connect** ou de **Okta Workforce**, sélectionnez **Browse**.
    3. Choisissez la connexion appropriée, puis ouvrez son onglet **Credentials**.
    4. Dans la section **Credentials**, sélectionnez **Rotate Keys**.
    5. Dans la fenêtre contextuelle, sélectionnez **Save** pour confirmer la rotation.

    Après la rotation, tous les JWT en cours signés avec la clé précédente deviennent immédiatement inactifs et pourraient échouer à la vérification auprès de votre IdP.
  </Accordion>

  <Accordion title="Management API">
    Pour afficher les clés publiques au moyen de la Management API, appelez l’endpoint Rotate Connection Signing Keys à l’aide de l’ID de votre connexion.

    <Warning>
      L’utilisation de cet endpoint exige les scopes `create:connections_keys` et `update:connections_keys`.
    </Warning>

    ```js lines
    POST /v2/connections/{id}/keys/rotate
    ```

    Après la rotation, tous les JWT en cours signés avec la clé précédente deviennent immédiatement inactifs et pourraient échouer à la vérification auprès de votre IdP.
  </Accordion>
</AccordionGroup>

<Card title="Comprendre la rotation des clés">
  Dans votre connexion OIDC ou Okta Workforce, vos clés de signature reçoivent l’un des statuts suivants :

  * **Current** : La clé de signature actuellement utilisée pour l’application.
  * **Next** : La prochaine clé de signature qui sera utilisée pour l’application une fois la clé actuelle révoquée.
  * **Previous** : Une clé de signature expirée ou autrement révoquée qui n’est plus utilisée.

  Lorsque Private Key JWT Client Authentication est activé pour la première fois pour une connexion, seule une paire de clés `current` et `next` est générée. Une clé n’est marquée comme `previous` qu’après une rotation.

  Lors de la rotation des clés de signature, les changements suivants se produisent :

  1. La clé `current` est supprimée et révoquée, et tous les JWT signés avec cette clé échoueront à la vérification auprès de l’IdP si l’IdP a été configuré avec le `jwks_uri`.
  2. La clé `current` reçoit le statut `previous`.
  3. La clé `next` devient la clé active et reçoit le statut `current`. Dorénavant, les JWT d’assertion client seront signés avec cette clé.
  4. Une nouvelle clé de signature est automatiquement générée pour remplacer la clé ayant fait l’objet d’une rotation. Cette nouvelle clé de signature reçoit le statut `next`.
</Card>
