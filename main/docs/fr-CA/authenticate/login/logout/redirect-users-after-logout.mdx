Vous pouvez rediriger les utilisateurs vers une URL précise après leur déconnexion. Vous devrez enregistrer l’URL de redirection dans les paramètres de votre tenant ou de votre application. Auth0 ne redirige vers des URL de la liste d’autorisation qu’après la déconnexion. Si vous avez besoin de redirections différentes pour chaque application, vous pouvez ajouter les URL à la liste d’autorisation dans les paramètres de votre application.

1. Ajoutez un paramètre de chaîne de requête `returnTo` avec l’URL cible comme valeur. Encodez l’URL cible transmise. Par exemple, pour rediriger l’utilisateur vers `https://www.example.com` après la déconnexion, effectuez la requête suivante :
   `https://{yourDomain}/v2/logout?returnTo=https%3A%2F%2Fwww.example.com`.
2. Ajoutez l’URL `returnTo` non encodée (par exemple, `https://www.example.com`) comme **Allowed Logout URLs** à l’un des deux endroits suivants :

   * **Tenant Settings** : Pour les requêtes de logout qui n’incluent pas le paramètre `client_id`, vous devez ajouter l’URL `returnTo` à la liste **Allowed Logout URLs** dans l’[onglet Advanced de vos Tenant Settings](https://manage.auth0.com/#/tenant/advanced).

     Pour ajouter une liste d’URL vers lesquelles l’utilisateur peut être redirigé après s’être déconnecté au niveau du tenant, accédez à [Tenant Settings &gt; Advanced](https://manage.auth0.com/#/tenant/advanced) dans l’Auth0 Dashboard.

     <Frame>![Paramètres de l’Auth0 Dashboard, onglet Advanced, Login and Logout](/docs/images/cdy7uua7fh8z/77s6AWMuzH78ABObR3lg0o/e71244706641f43e292d719056f3e651/2025-02-27_11-14-31.png)</Frame>
   * **Auth0 Application Settings** : Pour les requêtes de logout qui incluent le paramètre `client_id`, vous devez ajouter l’URL `returnTo` à la liste **Allowed Logout URLs** dans l’onglet **Settings** de la ou des applications Auth0 associées.

     <Frame>![Dashboard Applications Application Settings URI de l’application](/docs/images/cdy7uua7fh8z/1QhW2i4fTCCp8owey2tMPI/a19f7fc3f84118920d0e5cb2357144da/Application_URIs.png)</Frame>

Lorsque vous fournissez la liste d’URL, vous pouvez :

1. Spécifier plusieurs URL valides séparées par des virgules.
2. Utiliser `*` comme [caractère générique pour les sous-domaines](/docs/fr-CA/get-started/applications/wildcards-for-subdomains) (par exemple, `http://*.example.com`).

Si le paramètre `client_id` est inclus et que l’URL `returnTo` n’est pas définie, le serveur renvoie l’utilisateur vers la première valeur de **Allowed Logout URLs** définie dans le Dashboard.

Afin d’éviter les erreurs de validation, assurez-vous d’inclure le protocole dans l’URL. Par exemple, si vous définissez la valeur sur `*.example.com`, cela entraînera une erreur de validation; vous devriez donc utiliser `http://*.example.com` à la place.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Pour rediriger à l’aide de l’endpoint `oidc/logout` d’Auth0, consultez [Use OIDC Endpoint to Log Users Out of Auth0](/docs/fr-CA/authenticate/login/logout/log-users-out-of-auth0).
</Callout>

<div id="federated-logouts">
  ## Logouts fédérés
</div>

[Plusieurs fournisseurs](/docs/fr-CA/authenticate/login/logout/log-users-out-of-idps#federated-logout-support) prennent en charge la déconnexion fédérée.

Pour rediriger les utilisateurs à partir d’applications qui utilisent la déconnexion fédérée, lancez une déconnexion fédérée à l’aide de l’endpoint suivant :

`https://{yourDomain}/v2/logout?federated`

Il vous revient de vous assurer que votre application met fin à la session de l’utilisateur avant la redirection.

Lorsque l’utilisateur atteint l’endpoint `/logout`, Auth0 met fin à la session Auth0, redirige l’utilisateur vers l’endpoint de logout des <Tooltip tip="Fournisseur d’identité (IdP) : service qui stocke et gère les identités numériques." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Identity+Providers">Fournisseurs d’identité</Tooltip>, puis met fin à la session auprès du fournisseur d’identité.

<div id="limitations">
  ## Limites
</div>

* La validation des URL fournies comme valeurs du paramètre `returnTo`, de la chaîne de requête et des informations de hachage incluses dans l’URL n’est pas prise en compte.
* Le comportement des déconnexions fédérées avec les fournisseurs d’identité sociale n’est pas uniforme. Chaque fournisseur traitera le paramètre `returnTo` différemment et, dans certains cas, il ne fonctionnera pas. Veuillez vérifier les paramètres de votre fournisseur d’identité sociale pour vous assurer qu’il accepte le paramètre `returnTo` et comprendre comment il sera traité.
* Les URL fournies dans la liste **Allowed Logout URLs** sont sensibles à la casse. L’URL utilisée pour les déconnexions doit donc respecter exactement la casse de l’URL de déconnexion configurée dans le Dashboard. Notez toutefois que les parties schéma et hôte ne sont pas sensibles à la casse. Par exemple, si votre URL est `http://www.Example.Com/FooHoo.html`, la portion `http://www.Example.Com` n’est pas sensible à la casse, tandis que la portion `FooHoo.html` l’est.

Si vous utilisez des fournisseurs d’identité sociale comme Google ou Facebook, vous devez définir votre `Client ID` et votre `Secret` pour ces fournisseurs dans le [Dashboard](https://manage.auth0.com/#) afin que la déconnexion fonctionne correctement.

<div id="additional-requirements-for-facebook">
  ## Exigences supplémentaires pour Facebook
</div>

Utilisez le paramètre `returnTo` pour indiquer où rediriger l’utilisateur après la déconnexion.

Les IdPs ne prennent pas tous en charge `returnTo`.

Pour Facebook, l’exemple de code ci-dessous utilise le paramètre `returnTo` et définit un site Web comme destination de redirection.

export const codeExample = `https://{yourDomain}/v2/logout?federated&
      returnTo=https://{yourDomain}/logout?returnTo=http://www.example.com
      &access_token={facebookAccessToken}`;
