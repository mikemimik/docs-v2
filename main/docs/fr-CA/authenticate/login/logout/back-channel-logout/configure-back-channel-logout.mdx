<Card title="Avant de commencer">
  Pour utiliser OIDC Back-Channel Logout, votre application doit répondre aux exigences suivantes :

  * Le point de terminaison URI de votre mécanisme OIDC Back-Channel Logout doit être accessible sur Internet pour que votre tenant Auth0 puisse y accéder.
  * Les applications de production doivent utiliser TLS. Pour en savoir plus, consultez [Versions TLS (SSL) et algorithmes de chiffrement](/docs/fr-CA/customize/custom-domains/self-managed-certificates/tls-ssl).
  * L&#39;application doit pouvoir stocker l&#39;identifiant de session (`sid`) fourni et l&#39;associer aux sessions des utilisateurs qu&#39;elle crée. Nous recommandons aux applications de production d&#39;utiliser un stockage de session persistant.
  * L&#39;application doit pouvoir récupérer une session existante à l&#39;aide du `sid` pendant le processus de déconnexion sans utiliser de cookie côté client. Les cookies existent dans le navigateur et sont inaccessibles au point de terminaison de rappel de déconnexion.
</Card>

<div id="availability">
  ## Disponibilité
</div>

OIDC Back-Channel Logout est offert à tous les clients du forfait Enterprise. Vous devriez vérifier le point de terminaison de métadonnées OIDC standard `/.well-known/*` pour déterminer si votre application répond aux exigences.

```json lines
GET https://acme.eu.auth0.com/.well-known/openid-configuration

HTTP 200
{ ..., "backchannel_logout_supported": true, 
  "backchannel_logout_session_supported": true }
```

<div id="back-channel-logout-restrictions">
  ## Restrictions du Back-Channel Logout
</div>

Les URL de Back-Channel Logout sont appelées sur des points de terminaison exposés publiquement et doivent respecter certaines restrictions :

1. Vous devez utiliser le protocole HTTPS. Le protocole HTTP non chiffré ou tout autre protocole n&#39;est pas autorisé.
2. Vous ne devez pas utiliser de sous-domaines Auth0. Voici quelques sous-domaines Auth0 :

   * auth0.com
   * auth0app.com
   * webtask.io
   * webtask.run
3. Nous ne recommandons pas d&#39;utiliser des adresses IP sans nom de domaine. Pour utiliser le Back-Channel Logout, les adresses IP doivent être publiques. Les adresses IP provenant de plages internes, réservées ou de bouclage ne sont pas autorisées.

<div id="configure-auth0">
  ## Configurer Auth0
</div>

Enregistrez votre application pour recevoir des jetons de déconnexion via <Tooltip tip="Auth0 Dashboard : le produit principal d’Auth0 pour configurer vos services." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip> ou <Tooltip tip="Auth0 Dashboard : le produit principal d’Auth0 pour configurer vos services." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Management+API">Management API</Tooltip>.

<Tabs>
  <Tab title="Auth0 Dashboard">
    #### Inscrire des applications

    1. Accédez à [**Auth0 Dashboard &gt; Applications**](https://manage.auth0.com/#/applications).
    2. Choisissez l’application que vous voulez enregistrer.
    3. Sélectionnez l’onglet **Settings**.
    4. Sous **OpenID Connect Back-Channel Logout &gt; Back-Channel Logout URI**, ajoutez l’URI de déconnexion de l’application qui recevra les `logout_tokens`
    5. Une fois terminé, sélectionnez **Save Changes**.

       <Frame>![Auth0 Dashboard > Applications > Settings](/docs/images/cdy7uua7fh8z/19uQRkVswdmaO8gHTVTXCc/0564bfeda05bc7df90af9e8ee852047f/Screenshot_2024-11-05_at_13.31.54.png)</Frame>

    #### Désinscrire des applications

    La désinscription de votre application empêche le service de suivre les nouvelles connexions et d’envoyer des événements de déconnexion. Le service supprime les événements de déconnexion en attente une fois votre application désinscrite.

    Pour désinscrire votre application, supprimez l’URL de déconnexion back-channel.

    1. Accédez à [Auth0 Dashboard &gt; Applications](http://manage.auth0.com/#/applications).
    2. Choisissez l’application que vous voulez enregistrer.
    3. Sélectionnez l’onglet **Settings**.
    4. Sous **OpenID Connect Back-Channel Logout &gt; Back-Channel Logout URI**
    5. Supprimez l’URL back-channel.
    6. Une fois terminé, sélectionnez **Save Changes**.

       <Frame>![Auth0 Dashboard > Applications > Settings](/docs/images/cdy7uua7fh8z/5SlrBhwrNbJx0Wdw2kQ19t/9b69eb926e44c35b426daa76a851a5db/Screenshot_2024-11-05_at_13.31.54.png)</Frame>

    À des fins d’audit, le service consigne toujours l’inscription ou la désinscription des URL de déconnexion back-channel comme `API Operation Event` dans les journaux du tenant Auth0. Pour en savoir plus, consultez [Journaux](/docs/fr-CA/deploy-monitor/logs).
  </Tab>

  <Tab title="Management API">
    #### Inscrire une application

    Ajoutez la propriété `oidc_logout` pour effectuer un `PATCH` sur un client via la Management API :

    ```bash cURL lines
    PATCH /api/v2/clients/{id}
    "oidc_logout": {
      "backchannel_logout_urls": ["http://acme.eu.auth0.com/events"]
    }
    ```

    <Callout icon="file-lines" color="#0EA5E9" iconType="regular">
      La requête peut inclure les autres paramètres de l’application.
    </Callout>

    #### Désinscrire une application

    Supprimez le paramètre `oidc_logout` :

    ```bash cURL lines
    PATCH https://acme.eu.auth0.com/api/v2/clients/{id}
    {
      "oidc_logout": {
        "backchannel_logout_urls": []
      }
    }
    ```
  </Tab>
</Tabs>

<div id="configure-your-application">
  ## Configurez votre application
</div>

Une fois que vous avez configuré Back-Channel Logout dans l’Auth0 Dashboard ou au moyen de la Management API, configurez votre application pour utiliser le service en fonction de la technologie ou du framework.

1. Implémentez l’authentification des utilisateurs finaux selon le type d’application.

   1. Les utilisateurs finaux doivent pouvoir se connecter à l’application, et une session doit être créée.
   2. Un ID jeton doit être émis par Auth0 et être accessible dans le backend de l’application pour un traitement ultérieur.
2. Étendez le processus de connexion pour enregistrer les claims `sid` et, au besoin, `sub` après avoir validé l’ID jeton.

   1. Ces claims doivent être enregistrés avec la session d’application en cours.
   2. Les fonctions de gestion de session doivent pouvoir récupérer une session précise à partir de la valeur `sid`.
3. Configurez le point de terminaison de Back-Channel Logout :

   1. Le point de terminaison ne doit traiter que les requêtes `HTTP POST`.
   2. Extrayez le paramètre `logout_token` et validez-le comme un JWT ordinaire conformément à la spécification.
   3. Vérifiez que le jeton contient un claim `events` avec une valeur d’objet JSON et un membre nommé `http://schemas.openid.net/event/backchannel-logout`.
   4. Vérifiez que le jeton contient les claims `sid` et/ou `sub`.
   5. Vérifiez que le jeton ne contient PAS le claim `nonce`. Cela est requis afin d’éviter les abus en distinguant le Logout Token de l’ID jeton.
   6. Une fois le jeton validé, récupérez la session correspondant à la valeur `sid` et/ou `sub` reçue et terminez-la. Le processus exact de terminaison de la session d’application dépend des détails de la mise en œuvre. Par exemple, il peut être nécessaire de communiquer cet événement au front-end.

<div id="expected-responses">
  ## Réponses attendues
</div>

* **HTTP 200**:  Confirme la déconnexion de l’utilisateur de l’application concernée.
* **HTTP 400**: Indique une requête incorrecte. La requête n’est pas comprise ou le jeton a échoué à la validation. Auth0 consigne le problème dans les journaux du tenant, mais ne tente pas de requêtes supplémentaires pour cette session précise.

<div id="troubleshoot-issues">
  ## Résoudre les problèmes
</div>

<div id="application-did-not-receive-the-logout-events">
  ### L’application n’a pas reçu les événements de déconnexion
</div>

Si votre application n’a pas reçu de requête de déconnexion à la suite d’un événement de déconnexion.

1. Assurez-vous que votre application a une URL de Back-Channel Logout enregistrée dans Auth0 Dashboard.
2. Assurez-vous que l’URL de Back-Channel Logout est accessible à partir du tenant Auth0.
3. Assurez-vous qu’une session valide est établie. L’utilisateur final doit se connecter à cette application précise au moyen d’Auth0.

   <Callout icon="file-lines" color="#0EA5E9" iconType="regular">
     Les applications reçoivent des événements de déconnexion uniquement si les utilisateurs finaux se connectent à cette application précise avec Auth0. Si l’utilisateur final se connecte à d’autres applications, les événements de déconnexion ne sont pas déclenchés.
   </Callout>
4. Vérifiez les journaux du tenant Auth0 pour repérer des messages indiquant un échec de livraison des messages de déconnexion.
5. Assurez-vous que la déconnexion est déclenchée au moyen du point de terminaison de déconnexion standard. Les autres événements ne déclenchent pas les événements de déconnexion.
6. Si possible, vérifiez s’il y a des requêtes bloquées dans les journaux du serveur Web et/ou du pare-feu de l’application.

<div id="i-cant-find-oidc-back-channel-logout-tenant-logs">
  #### Je ne trouve pas les journaux du tenant liés à OIDC Back-Channel Logout
</div>

Cette fonctionnalité sera déployée graduellement auprès de tous les tenants à compter du 3 octobre 2023. Il se peut que vous voyiez encore les [codes d’événement du journal du tenant](/docs/fr-CA/deploy-monitor/logs/log-event-type-codes) `sslo` pour `oidc_backchannel_logout_succeeded` ou `fslo` pour `oidc_backchannel_logout_failed`.

<div id="client-app-cannot-verify-the-received-logout-token">
  ### L’application cliente ne peut pas vérifier le Logout Token reçu
</div>

Si la transaction échoue toujours avec une erreur 400 :

1. Vérifiez si le jeton est un JWT standard encodé en base64. Certains serveurs Web peuvent tronquer les paramètres trop longs. Pour en savoir plus, consultez [Algorithmes de signature](/docs/fr-CA/get-started/applications/signing-algorithms).
2. Si possible, capturez un jeton et vérifiez qu’il s’agit bien d’un JWT. Utilisez une source fiable, comme [JWT.IO](http://jwt.io).
3. Assurez-vous que la fonction de vérification récupère dynamiquement la clé de signature du tenant au moyen des [JSON Web Key Sets](/docs/fr-CA/secure/tokens/json-web-tokens/json-web-key-sets) (JWKS).

   <Callout icon="file-lines" color="#0EA5E9" iconType="regular">
     Nous ne recommandons pas de coder en dur une clé statique. Si votre configuration exige qu’une clé statique soit codée en dur, assurez-vous qu’elle contient la clé publique la plus récente du tenant Auth0.
   </Callout>

<div id="response-timeout-errors">
  ### Erreurs de délai d’attente de réponse
</div>

Auth0 attendra jusqu’à cinq secondes pour que l’URL de déconnexion OIDC Back-Channel de l’application réponde. Passé ce délai, il consignera « Failed OIDC Back-Channel Logout request » dans les journaux du tenant, avec une description de réponse vide.
