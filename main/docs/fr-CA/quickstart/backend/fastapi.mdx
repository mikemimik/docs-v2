<Steps>
  <Step title="Créer un nouveau projet FastAPI" stepNumber={1}>
    Créez un nouveau répertoire pour votre projet FastAPI et mettez en place un environnement virtuel.

    ```bash
    mkdir my-fastapi-api
    cd my-fastapi-api
    python3 -m venv venv
    source venv/bin/activate  # Windows : venv\Scripts\activate
    ```
  </Step>

  <Step title="Installer les dépendances" stepNumber={2}>
    Créez un fichier `requirements.txt` contenant les dépendances suivantes :

    ```txt requirements.txt
    fastapi>=0.115.0
    uvicorn[standard]>=0.34.0
    auth0-fastapi-api>=1.0.0b5
    python-dotenv>=1.0.0
    ```

    Installez les dépendances :

    ```bash
    pip install -r requirements.txt
    ```
  </Step>

  <Step title="Configurez votre API Auth0" stepNumber={3}>
    Vous devrez créer une API Auth0 pour représenter votre application FastAPI.

    <Tabs>
      <Tab title="Dashboard">
        1. Accédez à [Applications &gt; APIs](https://manage.auth0.com/#/apis) dans le Auth0 Dashboard
        2. Cliquez sur **Create API**
        3. Saisissez un **Name** pour votre API (p. ex., &quot;My FastAPI API&quot;)
        4. Définissez l’**Identifier** sur l’identifiant de votre API (p. ex., `https://my-fastapi-api`)
        5. Laissez le **Signing Algorithm** à RS256
        6. Cliquez sur **Create**
      </Tab>

      <Tab title="CLI">
        Si vous avez installé l’[Auth0 CLI](https://github.com/auth0/auth0-cli), vous pouvez créer une API et configurer votre environnement à partir de votre terminal :

        <CodeGroup>
          ```shellscript Mac
          AUTH0_API_NAME="My FastAPI API" && AUTH0_API_IDENTIFIER="https://my-fastapi-api" && brew tap auth0/auth0-cli && brew install auth0 && auth0 login --no-input && auth0 apis create --name "${AUTH0_API_NAME}" --identifier "${AUTH0_API_IDENTIFIER}" --signing-alg RS256 --no-input && echo "AUTH0_DOMAIN=$(auth0 tenants list --json | jq -r '.[] | select(.active == true) | .name')\nAUTH0_AUDIENCE=${AUTH0_API_IDENTIFIER}" > .env
          ```

          ```powershell Windows
          $ApiName = "My FastAPI API"; $ApiIdentifier = "https://my-fastapi-api"; auth0 login --no-input; auth0 apis create -n $ApiName -i $ApiIdentifier --signing-alg RS256 --no-input; $ActiveTenant = (auth0 tenants list --json | ConvertFrom-Json | Where-Object { $_.active -eq $true }).name; "AUTH0_DOMAIN=$ActiveTenant`nAUTH0_AUDIENCE=$ApiIdentifier" | Out-File -FilePath .env -Encoding utf8
          ```
        </CodeGroup>

        Cette opération créera votre API et générera automatiquement un fichier `.env` avec votre configuration Auth0.
      </Tab>
    </Tabs>

    <Info>
      L’**Identifier** est un identifiant unique pour votre API. Nous vous recommandons d’utiliser une URL, mais elle n’a pas à être accessible publiquement — Auth0 n’y enverra aucune requête. Cette valeur ne peut pas être modifiée par la suite.
    </Info>

    Prenez en note les valeurs **Domain** et **Identifier** (Audience). Vous en aurez besoin à l’étape suivante.
  </Step>

  <Step title="Définir les permissions d’API" stepNumber={4}>
    Les permissions (aussi appelées scopes) vous permettent de définir comment il est possible d’accéder à votre API. Vous pouvez créer des permissions pour votre API dans l’Auth0 Dashboard.

    1. Dans l’Auth0 Dashboard, accédez à l’onglet **Permissions** de votre API
    2. Ajoutez les permissions suivantes :
       * `read:messages` avec la description &quot;Lire les messages&quot;
       * `write:messages` avec la description &quot;Écrire les messages&quot;

    Ces permissions serviront à contrôler l’accès à des points de terminaison précis de votre API.
  </Step>

  <Step title="Configurer le client Auth0" stepNumber={5}>
    Créez un fichier `.env` à la racine de votre projet pour y stocker votre configuration Auth0 :

    ```txt .env
    AUTH0_DOMAIN=YOUR_AUTH0_DOMAIN
    AUTH0_AUDIENCE=YOUR_API_IDENTIFIER
    ```

    Remplacez `YOUR_AUTH0_DOMAIN` par votre domaine Auth0 (p. ex. `dev-abc123.us.auth0.com`) et `YOUR_API_IDENTIFIER` par l’identifiant que vous avez défini lors de la création de votre API.

    <Warning>
      Ne validez jamais votre fichier `.env` dans le contrôle de version. Ajoutez-le à votre fichier `.gitignore` afin de protéger vos informations d’identification.
    </Warning>

    Créez maintenant un fichier `app.py` et initialisez votre application FastAPI avec Auth0 :

    ```python app.py
    from fastapi import FastAPI, Depends
    from fastapi_plugin.fast_api_client import Auth0FastAPI
    import os
    from dotenv import load_dotenv

    # Charger les variables d'environnement
    load_dotenv()

    app = FastAPI()

    # Initialiser Auth0
    auth0 = Auth0FastAPI(
        domain=os.environ.get("AUTH0_DOMAIN"),
        audience=os.environ.get("AUTH0_AUDIENCE")
    )
    ```
  </Step>

  <Step title="Créer des routes protégées" stepNumber={6}>
    Ajoutez les routes suivantes à votre fichier `app.py`. Ces routes illustrent différents niveaux de contrôle d’accès :

    ```python app.py
    from fastapi import FastAPI, Depends
    from fastapi_plugin.fast_api_client import Auth0FastAPI
    import os
    from dotenv import load_dotenv

    load_dotenv()

    app = FastAPI()

    auth0 = Auth0FastAPI(
        domain=os.environ.get("AUTH0_DOMAIN"),
        audience=os.environ.get("AUTH0_AUDIENCE")
    )

    # Route publique - aucune authentification requise
    @app.get("/api/public")
    async def public():
        return {
            "message": "Hello from a public endpoint! You don't need to be authenticated to see this."
        }

    # Route protégée - authentification requise
    @app.get("/api/private")
    async def private(claims: dict = Depends(auth0.require_auth())):
        return {
            "message": "Hello from a private endpoint! You need to be authenticated to see this.",
            "user_id": claims.get("sub")
        }

    # Route avec portée - permission spécifique requise
    @app.get("/api/private-scoped")
    async def private_scoped(claims: dict = Depends(auth0.require_auth(scopes="read:messages"))):
        return {
            "message": "Hello from a private endpoint! You need to be authenticated and have a scope of read:messages to see this.",
            "user_id": claims.get("sub")
        }
    ```

    La méthode `require_auth()` valide le jeton d’accès envoyé dans l’en-tête Authorization. Lorsqu’elle est appelée avec le paramètre `scopes`, elle vérifie également que le jeton contient la permission spécifiée.
  </Step>

  <Step title="Exécutez votre API" stepNumber={7}>
    Démarrez votre application FastAPI :

    ```bash
    uvicorn app:app --reload
    ```

    Votre API est maintenant accessible à l’adresse `http://localhost:8000`.

    <Check>
      Ouvrez `http://localhost:8000/api/public` dans votre navigateur. Vous devriez voir le message public sans devoir vous authentifier.
    </Check>
  </Step>
</Steps>

<div id="test-your-api">
  ## Testez votre API
</div>

Pour tester les points de terminaison protégés, vous devrez obtenir un jeton d’accès auprès d’Auth0.

<div id="get-an-access-token">
  ### Obtenir un jeton d’accès
</div>

La façon la plus simple d’obtenir un jeton d’accès pour effectuer des tests est de passer par l’Auth0 Dashboard :

1. Accédez à [Applications &gt; APIs](https://manage.auth0.com/#/apis) dans l’Auth0 Dashboard
2. Sélectionnez votre API
3. Cliquez sur l’onglet **Test**
4. Cliquez sur **Copy Token** dans la section **Asking Auth0 for tokens from my application**

<div id="call-your-api">
  ### Interrogez votre API
</div>

Utilisez le jeton d’accès pour effectuer une requête à votre point de terminaison protégé :

```bash
curl -X GET http://localhost:8000/api/private -H 'authorization: Bearer YOUR_ACCESS_TOKEN'
```

Vous devriez recevoir une réponse contenant le message privé et votre ID d’utilisateur.

Pour tester ce point de terminaison à portée limitée, assurez-vous que votre jeton inclut la portée `read:messages` :

```bash
curl -X GET http://localhost:8000/api/private-scoped -H 'authorization: Bearer YOUR_ACCESS_TOKEN'
```

Si votre token ne comporte pas le scope requis, vous recevrez une réponse 403 Forbidden.

<Accordion title="Valider les claims personnalisés">
  Vous pouvez accéder aux claims personnalisés ajoutés au jeton d’accès au moyen d’[Auth0 Actions](https://auth0.com/docs/customize/actions).

  Accédez aux claims personnalisés dans votre handler de route :

  ```python
  @app.get("/api/profile")
  async def profile(claims: dict = Depends(auth0.require_auth())):
      return {
          "user_id": claims.get("sub"),
          "email": claims.get("email"),
          "permissions": claims.get("permissions", []),
          "custom_claim": claims.get("https://myapp.example.com/custom_claim")
      }
  ```

  Pour ajouter des claims personnalisés à vos jetons d’accès, créez une Auth0 Action :

  1. Accédez à **Actions &gt; Library** dans l’Auth0 Dashboard
  2. Cliquez sur **Create Action**
  3. Sélectionnez **Build from scratch**
  4. Donnez un nom à votre action et sélectionnez le Trigger **Login / Post Login**
  5. Ajoutez vos claims personnalisés :

  ```javascript
  exports.onExecutePostLogin = async (event, api) => {
    const namespace = 'https://myapp.example.com';

    if (event.authorization) {
      // Ajouter un claim personnalisé au jeton d'accès
      api.accessToken.setCustomClaim(`${namespace}/roles`, event.user.app_metadata.roles || []);
    }
  };
  ```

  6. Cliquez sur **Deploy** et ajoutez l’action à votre Login flow

  <Info>
    Les claims personnalisés doivent utiliser un format avec espace de noms (p. ex. `https://myapp.example.com/claim_name`) afin d’éviter les conflits avec les claims standard.
  </Info>
</Accordion>

<Accordion title="Protéger des endpoints sans utiliser de claims">
  Si vous devez protéger un endpoint sans avoir besoin d’accéder aux claims, vous pouvez utiliser le paramètre `dependencies` :

  ```python
  @app.get("/api/protected", dependencies=[Depends(auth0.require_auth())])
  async def protected():
      return {"message": "This endpoint is protected"}
  ```

  Cela valide le jeton d’accès, sans injecter les claims dans votre fonction.
</Accordion>

<Accordion title="Soutien pour DPoP">
  <Info>
    DPoP (Demonstrating Proof-of-Possession) est actuellement en [Early Access](https://auth0.com/docs/troubleshoot/product-lifecycle/product-release-stages). Communiquez avec [Auth0 soutien](https://support.auth0.com/) pour l’activer pour votre tenant.
  </Info>

  DPoP offre une sécurité renforcée en liant cryptographiquement les jetons d’accès au client qui les a demandés. Cela aide à prévenir le vol de jetons et les replay attacks.

  Le SDK active le soutien pour DPoP par défaut. Vous pouvez configurer le comportement de DPoP :

  ```python
  auth0 = Auth0FastAPI(
      domain=os.environ.get("AUTH0_DOMAIN"),
      audience=os.environ.get("AUTH0_AUDIENCE"),
      dpop_enabled=True,      # Activer DPoP (par défaut : True)
      dpop_required=False     # Exiger DPoP (par défaut : False)
  )
  ```

  **Mode mixte** (par défaut) accepte à la fois les jetons Bearer et DPoP :

  ```python
  auth0 = Auth0FastAPI(
      domain=os.environ.get("AUTH0_DOMAIN"),
      audience=os.environ.get("AUTH0_AUDIENCE"),
      dpop_enabled=True,
      dpop_required=False
  )
  ```

  **Mode DPoP בלבד** rejette les jetons Bearer :

  ```python
  auth0 = Auth0FastAPI(
      domain=os.environ.get("AUTH0_DOMAIN"),
      audience=os.environ.get("AUTH0_AUDIENCE"),
      dpop_required=True
  )
  ```

  Lorsque vous utilisez DPoP, les clients doivent inclure à la fois les en-têtes `Authorization: DPoP <token>` et `DPoP: <proof>`. Le SDK valide automatiquement la preuve DPoP et l’associe au jeton d’accès.
</Accordion>

<Accordion title="Configurer pour un proxy inverse">
  <Warning>
    N’activez `trust_proxy` que lorsque votre application se trouve derrière un proxy inverse de confiance. Ne l’activez jamais pour des applications exposées directement à Internet.
  </Warning>

  Si votre application s’exécute derrière un proxy inverse (nginx, AWS ALB, etc.), vous devez activer la confiance envers le proxy pour que la validation DPoP fonctionne correctement :

  ```python
  from fastapi import FastAPI

  app = FastAPI()

  # Activer la confiance envers le proxy
  app.state.trust_proxy = True

  auth0 = Auth0FastAPI(
      domain=os.environ.get("AUTH0_DOMAIN"),
      audience=os.environ.get("AUTH0_AUDIENCE")
  )
  ```

  Configurez votre proxy inverse pour transmettre les en-têtes nécessaires :

  ```nginx
  location /api {
      proxy_pass http://localhost:8000;
      proxy_set_header X-Forwarded-Proto $scheme;
      proxy_set_header X-Forwarded-Host $host;
      proxy_set_header X-Forwarded-Prefix /api;
  }
  ```

  C’est essentiel pour la validation DPoP, car le SDK doit faire correspondre exactement l’URL utilisée par le client. Sans confiance envers le proxy, votre application voit des URL internes, tandis que les preuves DPoP font référence à des URL externes, ce qui entraîne des échecs de validation.
</Accordion>

<Accordion title="Gestion des erreurs">
  Le SDK soulève une exception `HTTPException` en cas d’erreurs d’authentification. FastAPI les gère automatiquement et renvoie au client les réponses HTTP appropriées.

  Vous pouvez mettre en place une gestion personnalisée des erreurs au besoin :

  ```python
  from fastapi import Request
  from fastapi.responses import JSONResponse
  from fastapi.exceptions import HTTPException

  @app.exception_handler(HTTPException)
  async def custom_http_exception_handler(request: Request, exc: HTTPException):
      if exc.status_code in [401, 403]:
          return JSONResponse(
              status_code=exc.status_code,
              content={
                  "error": "authentication_failed",
                  "message": "You must be authenticated to access this resource",
                  "details": exc.detail
              }
          )
      return JSONResponse(
          status_code=exc.status_code,
          content={"detail": exc.detail}
      )
  ```

  Les erreurs d’authentification comprennent :

  * **401 Unauthorized** : jeton d’accès manquant, invalide ou expiré
  * **403 Forbidden** : jeton valide, mais permissions insuffisantes (scopes)
</Accordion>

<div id="common-issues">
  ## Problèmes courants
</div>

<AccordionGroup>
  <Accordion title="401 Unauthorized - Invalid audience">
    **Problème** : la validation du jeton échoue avec l’erreur &quot;Invalid audience&quot;.

    **Solution** : vérifiez que `AUTH0_AUDIENCE` dans votre fichier `.env` correspond exactement à l’**Identifier** que vous avez configuré pour votre API dans l’Auth0 Dashboard.

    1. Ouvrez l’Auth0 Dashboard et accédez à **Applications &gt; APIs**
    2. Sélectionnez votre API
    3. Vérifiez la valeur **Identifier** dans l’onglet **Settings**
    4. Mettez à jour votre fichier `.env` :
       ```txt
       AUTH0_AUDIENCE=https://your-exact-api-identifier
       ```
    5. Redémarrez votre application
  </Accordion>

  <Accordion title="401 Unauthorized - Invalid issuer">
    **Problème** : la validation du jeton échoue avec l’erreur &quot;Invalid issuer&quot;.

    **Solution** : vérifiez que votre `AUTH0_DOMAIN` est correct et n’inclut pas le protocole `https://`.

    Votre domaine doit ressembler à `dev-abc123.us.auth0.com`, et non à `https://dev-abc123.us.auth0.com`.

    Mettez à jour votre fichier `.env` :

    ```txt
    AUTH0_DOMAIN=dev-abc123.us.auth0.com
    ```
  </Accordion>

  <Accordion title="403 Forbidden - Insufficient scope">
    **Problème** : le point de terminaison protégé retourne un code 403 même avec un jeton d’accès valide.

    **Solution** : le jeton d’accès n’inclut pas le scope requis.

    1. Vérifiez quels scopes votre point de terminaison requiert
    2. Lorsque vous demandez un jeton, assurez-vous d’inclure les scopes requis
    3. Vérifiez que le scope existe dans l’onglet **Permissions** de votre API dans l’Auth0 Dashboard
    4. Décodez votre jeton sur [jwt.io](https://jwt.io) pour vérifier qu’il contient le claim `scope` avec les valeurs requises
  </Accordion>

  <Accordion title="ModuleNotFoundError: No module named 'fastapi_plugin'">
    **Problème** : Python ne trouve pas l’Auth0 FastAPI SDK.

    **Solution** : assurez-vous que le SDK est installé dans votre environnement virtuel actif.

    ```bash
    # Activez votre environnement virtuel
    source venv/bin/activate  # Windows: venv\Scripts\activate

    # Installez le SDK
    pip install auth0-fastapi-api

    # Vérifiez l’installation
    pip show auth0-fastapi-api
    ```
  </Accordion>

  <Accordion title="Impossible de se connecter à Auth0 (échec de récupération des JWKS)">
    **Problème** : l’application ne peut pas récupérer les clés de signature depuis Auth0.

    **Solution** : vérifiez votre connectivité réseau et la configuration du domaine.

    1. Vérifiez que votre domaine est accessible :
       ```bash
       curl https://YOUR_DOMAIN/.well-known/openid-configuration
       ```

    2. Vérifiez que votre pare-feu autorise les connexions HTTPS sortantes (port 443) vers `*.auth0.com`

    3. Si vous êtes derrière un proxy d’entreprise, configurez les variables d’environnement `HTTP_PROXY` et `HTTPS_PROXY`
  </Accordion>

  <Accordion title="La validation DPoP échoue">
    **Problème** : l’authentification DPoP retourne des erreurs liées à l’URL ou à la validation de la proof.

    **Solution** :

    1. Si vous êtes derrière un proxy inverse, activez la confiance du proxy :
       ```python
       app.state.trust_proxy = True
       ```

    2. Vérifiez que votre proxy transmet ces en-têtes :
       * `X-Forwarded-Proto`
       * `X-Forwarded-Host`
       * `X-Forwarded-Prefix`

    3. Assurez-vous que DPoP est activé pour votre tenant (contactez le soutien Auth0)

    4. Vérifiez que le claim `htu` de la DPoP proof correspond exactement à l’URL de votre requête
  </Accordion>
</AccordionGroup>
