    <Section id={sections[2].id} title={sections[2].title} stepNumber="3" isSingleColumn>
      Vous utiliserez une bibliothèque appelée [Authlib](https://github.com/lepture/authlib) pour créer un [ResourceProtector](https://docs.authlib.org/en/latest/flask/1/resource-server.html), c’est-à-dire un type de [décorateur de vue Django](https://docs.djangoproject.com/en/4.0/topics/http/decorators/) qui protège vos ressources (vues d’API) à l’aide d’un
      validateur donné.

      Le validateur vérifiera le jeton d’accès que vous transmettez à la ressource en s’assurant que sa
      signature et ses claims sont valides.

      Vous pouvez utiliser le validateur `JWTBearerTokenValidator` d’AuthLib en y apportant quelques ajustements pour vous assurer qu’il respecte
      nos exigences de [validation des jetons d’accès](https://auth0.com/docs/secure/tokens/access-tokens/validate-access-tokens).

      Pour créer votre `Auth0JWTBearerTokenValidator`, vous devez lui transmettre votre `domain` et votre
      `audience` (identificateur d’API). Il récupérera ensuite la clé publique nécessaire pour vérifier la signature du jeton
      et la transmettra à la classe `JWTBearerTokenValidator`.

      Vous remplacerez ensuite le `claims_options` de la classe pour vous assurer que les claims `expiry`,
      `audience` et `issue` du jeton sont validés conformément à nos exigences.

      Créez le fichier `apiexample/validator.py` à l’aide du code dans le panneau interactif.

    <Section id={sections[3].id} title={sections[3].title} stepNumber="4" isSingleColumn>
      Ensuite, vous créerez trois vues d’API dans `apiexample/views.py`:

      * `/api/public` : un point de terminaison public qui ne nécessite aucune authentification.
      * `/api/private` : un point de terminaison privé qui nécessite un JWT de jeton d’accès valide.
      * `/api/private-scoped` : un point de terminaison privé qui nécessite un JWT de jeton d’accès valide contenant la
        `scope` indiquée.

      Les routes protégées auront un décorateur `require_auth`, qui est un `ResourceProtector`
      utilisant le `Auth0JWTBearerTokenValidator` que vous avez créé précédemment.

      Pour créer le `Auth0JWTBearerTokenValidator`, vous lui transmettrez le domaine de votre tenant et l’API
      Identifier de l’API que vous avez créée précédemment.

      Le décorateur `require_auth` sur la route `private_scoped` accepte un argument supplémentaire,
      `"read:messages"`, qui vérifie dans le jeton d’accès la permission (scope) que vous avez créée précédemment.
