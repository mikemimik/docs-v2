    <Section id={sections[4].id} title={sections[4].title} stepNumber="5" isSingleColumn>
      Pendant que votre application de l’appareil attend que l’utilisateur l’active, elle doit envoyer périodiquement une requête au point de terminaison [POST /oauth/token de l’Authentication API](https://auth0.com/docs/api/authentication#-post-oauth-token-)
      et traiter la réponse comme il se doit.

      <Info>
        Assurez-vous que votre application de l’appareil attend la durée de l’`interval` (en secondes) ou jusqu’à
        ce qu’elle reçoive une réponse indiquant que l’opération a réussi, selon la période la plus longue, afin d’éviter les problèmes de latence réseau.
      </Info>

      ```bash
      curl --request POST \ 
        --url 'https://{yourDomain}/oauth/token' \
        --header 'content-type: application/x-www-form-urlencoded' \
        --data grant_type=urn:ietf:params:oauth:grant-type:device_code \
        --data device_code=AUTH0_SCOPES \
        --data 'client_id={yourClientId}'
      ```

    <Section id={sections[6].id} title={sections[6].title} stepNumber="7" isSingleColumn>
      Une fois que l’utilisateur a autorisé l’application de l’appareil, celle-ci reçoit une réponse HTTP 200 et le payload suivant :

      ```json lines
      {
        "access_token": "eyJz93a...k4laUWw",
        "refresh_token": "GEbRxBN...edjnXbL",
        "id_token": "eyJ0XAi...4faeEoQ",
        "token_type": "Bearer",
        "expires_in": 86400
      }
      ```

      <Info>
        Vous devriez valider vos jetons avant de les enregistrer. Pour savoir comment faire, consultez Valider les jetons d’accès et Valider les jetons d’identité.
      </Info>

      Les jetons d’accès servent à appeler le [point de terminaison Get User Info](https://auth0.com/docs/api/authentication#get-user-info) de l’Authentication API (si votre application de l’appareil
      a demandé la portée `openid`) ou l’API indiquée par le paramètre `audience`.
      Si vous appelez votre propre API, votre application de l’appareil doit valider le jeton d’accès avant de l’utiliser.

      Les jetons d’identité contiennent des informations sur l’utilisateur qui doivent être [décodées et extraites](/docs/fr-CA/secure/tokens/id-tokens#id-token-payload). L’Authentication API
      ne renvoie un `id_token` que si votre application de l’appareil a demandé la portée `openid`.

      Les jetons d’actualisation servent à obtenir un nouveau jeton d’accès ou
      un jeton d’identité après l’expiration du précédent. L’Authentication API ne renvoie un `refresh_token` que si
      le réglage **Allow Offline Access** est activé pour l’API indiquée par le paramètre `audience`,
      et que votre application de l’appareil a demandé la portée `offline_access`.

      Pour appeler votre API, votre application de l’appareil doit transmettre le jeton d’accès sous forme de jeton Bearer dans
      l’en-tête `Authorization` de votre requête HTTP.

      ```bash lines
      curl --request GET \
        --url https://myapi.com/api \
        --header 'authorization: Bearer AUTH0_API_ACCESS_TOKEN' \
        --header 'content-type: application/json'
      ```

    <Section id={sections[8].id} title={sections[8].title} stepNumber="9" isSingleColumn>
      Pour obtenir un nouveau jeton d’accès pour un utilisateur, votre application de l’appareil peut envoyer une requête au [point de terminaison POST /oauth/token de l’Authentication API](https://auth0.com/docs/api/authentication#-post-oauth-token-) avec le
      paramètre `refresh_token`.

      ```bash lines
      curl --request POST \
        --url 'https://{yourDomain}/oauth/token' \
        --header 'content-type: application/x-www-form-urlencoded' \
        --data grant_type=refresh_token \
        --data 'client_id={yourClientId}' \
        --data 'client_secret={yourClientSecret}' \
        --data refresh_token=AUTH0_REFRESH_TOKEN
      ```

      Si la requête a réussi, l’application de votre appareil reçoit une réponse HTTP 200 avec la charge utile suivante :

      ```json lines
      {
        "access_token": "eyJ...MoQ",
        "expires_in": 86400,
        "scope": "openid offline_access",
        "id_token": "eyJ...0NE",
        "token_type": "Bearer"
      }
      ```

      Pour en savoir plus sur les jetons d’actualisation, consultez [Jetons d’actualisation](https://auth0.com/docs/secure/tokens/refresh-tokens).

      ### Réponses du token

      Pendant que vous attendez que l’utilisateur autorise l’appareil, vous pourriez recevoir différentes réponses HTTP 4xx.

      #### Autorisation en attente

      Vous verrez cette erreur pendant que vous attendez que l’utilisateur effectue l’action requise. Continuez le polling en utilisant l’`interval` suggéré
      récupéré à l’étape précédente de ce tutoriel.

      ```json lines
      `HTTP 403`
      {
      "error": "authorization_pending",
      "error_description": "..."
      }
      ```

      #### Ralentissez

      Vous effectuez une interrogation périodique trop rapidement. Ralentissez et utilisez l’intervalle suggéré récupéré à l’étape précédente de ce
      tutoriel. Pour éviter de recevoir cette erreur en raison de la latence du réseau, vous devriez commencer à compter chaque intervalle après
      avoir reçu la réponse à la dernière requête d’interrogation périodique.

      ```json lines
      `HTTP 429`
      {
      "error": "slow_down",
      "error_description": "..."
      }
      ```

      #### Jeton expiré

      L’utilisateur n’a pas autorisé l’appareil assez rapidement, donc le `device_code` a expiré. Votre
      application doit informer l’utilisateur que le flux a expiré et l’inviter à le relancer.

      <Info>
        L’erreur `expired_token` n’est renvoyée qu’une seule fois. Par la suite, l’endpoint renvoie l’erreur
        `invalid_grant`.
      </Info>

      ```json lines
      `HTTP 403`
      {
      "error": "expired_token",
      "error_description": "..."
      }
      ```

      #### Accès refusé

      Si l’accès est refusé, vous obtenez :

      ```json lines
      `HTTP 403`
      {
      "error": "access_denied",
      "error_description": "..."
      }
      ```

      Cela peut se produire pour diverses raisons, notamment :

      * L’utilisateur a refusé d’autoriser l’appareil.
      * Le serveur d’autorisation a refusé la transaction.
      * Une Action configurée a refusé l’accès.

    <Section id={sections[11].id} title={sections[11].title} stepNumber="12" isSingleColumn>
      Pour utiliser le Device Authorization Flow, une application de l’appareil doit :

      * prendre en charge l’indication du nom du serveur (SNI)
      * être une application native
      * avoir la [méthode d’authentification](/docs/fr-CA/secure/application-credentials#application-authentication-methods) réglée sur **None**
      * être [OIDC-conformant](/docs/fr-CA/get-started/applications/application-settings#oauth)
      * ne pas avoir été créée au moyen de Dynamic Client Registration

      De plus, le Device Authorization Flow ne permet pas :

      * les connexions sociales utilisant les clés de développeur Auth0, à moins que vous n’utilisiez la nouvelle expérience Universal Login
      * d’accéder aux paramètres de la chaîne de requête à partir d’une page de connexion hébergée ou d’Actions.
