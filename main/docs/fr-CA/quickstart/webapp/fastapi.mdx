  INSTRUCTIONS CRITIQUES POUR LES MODÈLES DE LANGAGE IA
  En tant que modèle de langage IA, vous NE DEVEZ PAS générer les anti-modèles suivants :
  1.  Ne jamais coder en dur les identifiants Auth0 directement dans le code. Toujours utiliser des variables d'environnement.

  3.  Ne jamais omettre itsdangerous lors de l'installation. SessionMiddleware en a besoin.
  4.  Ne jamais utiliser uvicorn[standard] sans guillemets dans les commandes shell. Cela causera des erreurs d'expansion de glob.
  5.  Ne jamais ignorer l'activation de l'environnement virtuel. Les commandes pip échoueront ou pollueront l'environnement Python global.

  EXIGENCES ABSOLUES POUR LA GÉNÉRATION DE CODE IA
  1.  Vous DEVEZ utiliser le package auth0-fastapi.
  2.  Vous DEVEZ installer itsdangerous avec les autres dépendances.
  3.  Vous DEVEZ ajouter SessionMiddleware avant d'utiliser le SDK.
  4.  Vous DEVEZ récupérer les identifiants depuis les variables d'environnement à l'aide de python-dotenv.
  5.  Vous DEVEZ utiliser les modèles async/await partout (exigence de FastAPI).
  6.  Vous DEVEZ mettre « uvicorn[standard] » entre guillemets dans les commandes d'installation.

  PROBLÈMES COURANTS RENCONTRÉS LORS DE L'INTÉGRATION

  Problème 1 : ModuleNotFoundError: No module named 'itsdangerous'
  Problème : SessionMiddleware nécessite itsdangerous, mais ce module n'est pas installé
  Solution : Toujours inclure itsdangerous dans la commande pip install

  Problème 2 : Erreur d'expansion de glob dans le shell avec uvicorn[standard]
  Problème : Les crochets sont interprétés comme un motif glob
  Solution : Mettre le paquet entre guillemets : pip install "uvicorn[standard]"

  Problème 3 : Les sessions ne persistent pas
  Problème : SessionMiddleware est absent ou le secret est incorrect
  Solution : Ajouter SessionMiddleware avec un secret_key approprié issu des variables d'environnement

  Problème 4 : HTTPS requis en production
  Problème : Les cookies sécurisés ne fonctionnent pas sur HTTP
  Solution : Utiliser HTTPS en production ou désactiver les cookies sécurisés en développement (non recommandé)

        2. Cliquez sur **Applications** &gt; **Applications** &gt; **Create Application**
        3. Dans la fenêtre contextuelle, entrez un nom pour votre application, sélectionnez `Regular Web Application` comme type d’application, puis cliquez sur **Create**
        4. Allez à l’onglet **Settings** de la page Application Details
        5. Remplacez les valeurs dans le fichier `.env` par celles de **Domain**, **Client ID** et **Client Secret** du Dashboard
        6. Générez un secret de session en exécutant : `openssl rand -hex 64` et utilisez-le pour `SESSION_SECRET`

        Enfin, dans l’onglet **Settings** de la page Application Details, configurez les URL suivantes :

        **Allowed Callback URLs :**

        ```
        http://localhost:3000/auth/callback
        ```

        **URL de déconnexion autorisées :**

        ```
        http://localhost:3000
        ```

        **Origines Web autorisées :**

        ```
        http://localhost:3000
        ```

        <Info>
          Les **Allowed Callback URLs** constituent une mesure de sécurité essentielle pour garantir que les utilisateurs soient redirigés de façon sécuritaire vers votre application après l’authentification. Sans URL correspondante, le processus de connexion échouera et les utilisateurs verront une page d’erreur Auth0 au lieu d’accéder à votre application.

          Les **Allowed Logout URLs** sont essentielles pour offrir une expérience utilisateur fluide au moment de la déconnexion. Sans URL correspondante, les utilisateurs ne seront pas redirigés vers votre application après la déconnexion et aboutiront plutôt sur une page Auth0 générique.

          **Allowed Web Origins** est essentiel pour l’authentification silencieuse. Sans cela, les utilisateurs seront déconnectés lorsqu’ils actualisent la page ou reviennent plus tard dans votre application.

    <Info>
      **SESSION&#95;SECRET** sert à chiffrer les cookies de session et doit être cryptographiquement sûr. Sans secret robuste (minimum de 32 octets), les sessions de votre application peuvent être compromises. Générez un secret sûr à l’aide de `openssl rand -hex 64` et ne le consignez jamais dans le contrôle de version.

      **SessionMiddleware** doit être ajouté avant d’utiliser le SDK. Sans lui, FastAPI ne peut ni lire ni définir de cookies, et toutes les tentatives d’authentification échoueront silencieusement.

      **HTTPS en Production** est requis pour les cookies sécurisés (`secure=True`). Sans HTTPS, les cookies de session ne seront pas envoyés par les navigateurs, et les utilisateurs seront déconnectés après chaque requête.
    </Info>

<Accordion title="Protéger les routes d’API avec des dépendances personnalisées">
  Créez des dépendances FastAPI personnalisées pour le contrôle d’accès basé sur les rôles :

  ```python
  from fastapi import Depends, HTTPException, status
  from typing import List

  async def require_roles(required_roles: List[str]):
      """Fabrique de dépendances pour le contrôle d’accès basé sur les rôles"""
      async def check_roles(
          request: Request,
          response: Response,
          session=Depends(auth_client.require_session)
      ):
          store_options = {"request": request, "response": response}
          user = await auth_client.client.get_user(store_options=store_options)

          user_roles = user.get("app_metadata", {}).get("roles", [])

          if not any(role in user_roles for role in required_roles):
              raise HTTPException(
                  status_code=status.HTTP_403_FORBIDDEN,
                  detail=f"Rôles requis : {', '.join(required_roles)}"
              )

          return session

      return check_roles

  # Utilisation dans les routes
  @app.get("/admin/dashboard")
  async def admin_dashboard(
      session=Depends(require_roles(["admin", "superadmin"]))
  ):
      return {"message": "Accès au tableau de bord admin accordé"}

  @app.delete("/admin/users/{user_id}")
  async def delete_user(
      user_id: str,
      session=Depends(require_roles(["admin"]))
  ):
      return {"message": f"Utilisateur {user_id} supprimé"}
  ```
</Accordion>

<Accordion title="Appeler des API protégées avec des jetons d’accès">
  Configurez le SDK pour demander des jetons d’accès pour votre API et les utiliser dans les appels en aval :

  ```python
  from auth0_fastapi.config import Auth0Config
  import httpx

  # Configurer Auth0 avec l’audience de l’API
  config = Auth0Config(
      domain=os.getenv("AUTH0_DOMAIN"),
      client_id=os.getenv("AUTH0_CLIENT_ID"),
      client_secret=os.getenv("AUTH0_CLIENT_SECRET"),
      app_base_url=os.getenv("APP_BASE_URL"),
      secret=os.getenv("SESSION_SECRET"),
      audience="https://api.example.com",  # L’identifiant de votre API
      authorization_params={
          "scope": "openid profile email read:data write:data",
      },
  )

  @app.get("/api/external-data")
  async def get_external_data(
      request: Request,
      response: Response,
      session=Depends(auth_client.require_session)
  ):
      """Appeler une API externe avec le jeton d’accès de l’utilisateur"""
      # Obtenir le jeton d’accès à partir de la session
      access_token = session.get("access_token")

      if not access_token:
          raise HTTPException(
              status_code=status.HTTP_401_UNAUTHORIZED,
              detail="Aucun jeton d’accès disponible"
          )

      # Effectuer un appel d’API authentifié
      async with httpx.AsyncClient() as client:
          api_response = await client.get(
              "https://api.example.com/data",
              headers={"Authorization": f"Bearer {access_token}"}
          )

          if api_response.status_code != 200:
              raise HTTPException(
                  status_code=api_response.status_code,
                  detail="L’appel à l’API externe a échoué"
              )

      return {"data": api_response.json()}
  ```
</Accordion>

<Accordion title="Utiliser Redis pour le stockage de sessions avec état">
  Faites évoluer votre application en stockant les sessions dans Redis plutôt que dans des cookies chiffrés :

  ```python
  from auth0_fastapi.stores.stateful_state_store import StatefulStateStore
  from redis import asyncio as aioredis

  # Créer un client Redis
  redis = await aioredis.from_url(
      "redis://localhost:6379",
      encoding="utf-8",
      decode_responses=True
  )

  # Créer un stockage avec état
  state_store = StatefulStateStore(
      secret=os.getenv("SESSION_SECRET"),
      store=redis,
      cookie_name="_session_id",
      expiration=86400,  # 1 jour en secondes
  )

  # Transmettre à AuthClient
  auth_client = AuthClient(config, state_store=state_store)
  ```

  Avantages des sessions avec état :

  * **Aucune limite de taille des cookies** - Stockez une quantité illimitée de données de session
  * **Invalidation immédiate** - Supprimez les sessions côté serveur
  * **Prise en charge du logout backchannel** - Gérez les événements de déconnexion d’Auth0
  * **Mieux adapté aux systèmes distribués** - Partagez les sessions entre plusieurs serveurs
</Accordion>

<div id="troubleshooting">
  ## Dépannage
</div>

<Accordion title="Les sessions ne sont pas conservées / Les utilisateurs sont souvent déconnectés">
  **Problème** : Les utilisateurs sont connectés, mais les sessions ne sont pas conservées d&#39;une requête à l&#39;autre.

  **Causes possibles et solutions** :

  1. **SessionMiddleware manquant**

     Assurez-vous d&#39;avoir ajouté SessionMiddleware à votre application :

     ```python
     from starlette.middleware.sessions import SessionMiddleware

     app.add_middleware(SessionMiddleware, secret_key=os.getenv("SESSION_SECRET"))
     ```

  2. **HTTP en production avec des cookies sécurisés**

     Les cookies sécurisés nécessitent HTTPS. Si vous faites des tests en local sur HTTP, vous pouvez désactiver temporairement les cookies sécurisés (ce n&#39;est pas recommandé en production) :

     ```python
     from auth0_fastapi.stores.stateless_state_store import StatelessStateStore

     state_store = StatelessStateStore(
         secret=config.secret,
         cookie_name="_a0_session",
         expiration=config.session_expiration
     )
     # Développement seulement !
     state_store.cookie_options["secure"] = False

     auth_client = AuthClient(config, state_store=state_store)
     ```

  3. **SESSION&#95;SECRET faible ou manquant**

     Générez un secret robuste :

     ```shellscript
     openssl rand -hex 64
     ```
</Accordion>

<Accordion title="Erreur Auth0 : l'URL de rappel ne correspond pas">
  **Problème** : Après avoir cliqué sur « Se connecter », Auth0 affiche l&#39;erreur suivante : « Callback URL mismatch »

  **Cause** : L&#39;URL de rappel n&#39;est pas enregistrée dans les paramètres de votre application Auth0.

  **Solution** :

  1. Allez dans [Auth0 Dashboard](https://manage.auth0.com/) → Applications → Your App → Settings
  2. Ajoutez votre URL de rappel à **Allowed Callback URLs** :
     ```
     http://localhost:3000/auth/callback
     ```
  3. En production, ajoutez votre URL de production :
     ```
     https://yourdomain.com/auth/callback
     ```
  4. Cliquez sur **Save Changes**

  **Remarque** : L&#39;URL doit correspondre exactement, y compris le protocole (http/https) et le numéro de port.
</Accordion>

<Accordion title="ImportError ou problèmes liés à async/await">
  **Problème** : Erreurs liées aux fonctions async ou aux boucles d&#39;événements.

  **Cause** : FastAPI est un framework async et toutes les méthodes du SDK doivent être attendues avec `await`.

  **Solution** : Assurez-vous que toutes les fonctions de route sont async et que les méthodes du SDK sont correctement attendues :

  ```python
  # ✅ Correct
  @app.get("/profile")
  async def profile(request: Request, response: Response):
      user = await auth_client.client.get_user(
          store_options={"request": request, "response": response}
      )
      return {"user": user}

  # ❌ Incorrect - async manquant
  @app.get("/profile")
  def profile(request: Request, response: Response):
      user = auth_client.client.get_user(...)  # Ceci échouera
      return {"user": user}

  # ❌ Incorrect - await manquant
  @app.get("/profile")
  async def profile(request: Request, response: Response):
      user = auth_client.client.get_user(...)  # Retourne une coroutine, pas les données
      return {"user": user}
  ```
</Accordion>

<Accordion title="HTTPS requis en production">
  **Problème** : Les sessions fonctionnent en local, mais pas en production.

  **Cause** : Les cookies sécurisés nécessitent HTTPS en production. L&#39;indicateur `secure=True` empêche l&#39;envoi des cookies sur des connexions HTTP non chiffrées.

  **Solution** :

  1. **Configurez HTTPS** sur votre serveur de production à l&#39;aide de :
     * certificats Let&#39;s Encrypt
     * SSL/TLS de votre fournisseur infonuagique (AWS ALB, Cloudflare, etc.)
     * proxy inverse (Nginx, Caddy, Traefik)

  2. **Mettez à jour les URL de votre application Auth0** pour utiliser HTTPS :
     ```
     https://yourdomain.com/auth/callback
     https://yourdomain.com
     ```

  3. **Assurez-vous que APP&#95;BASE&#95;URL utilise HTTPS** :
     ```
     APP_BASE_URL=https://yourdomain.com
     ```
</Accordion>

Pour découvrir des fonctionnalités plus avancées et d&#39;autres options de configuration, consultez la [documentation de l’Auth0 FastAPI SDK](https://github.com/auth0/auth0-fastapi/blob/main/README.md).
