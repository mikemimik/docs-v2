---
description: Récupérez les événements d’Auth0 selon votre propre cadence grâce à l’Events API basée sur les événements envoyés par le serveur (SSE).
title: Consommer des événements avec l’Events API
---

L’Events API offre une solution de rechange par extraction à Event Streams. Au lieu qu’Auth0 envoie les événements vers une destination, votre application ouvre une connexion persistante à `GET /api/v2/events` et reçoit les événements sous forme de flux Server-Sent Events (SSE). Vous contrôlez quand vous connecter, comment reprendre après une déconnexion et à quelle vitesse consommer les événements.

Cette approche est utile lorsque vous devez :

* Traiter les événements à votre propre rythme sans mettre en place de webhook endpoint.
* Relire les événements à partir d’un moment précis pour le remplissage rétroactif ou la récupération.
* Intégrer des systèmes qui privilégient l’interrogation périodique plutôt qu’une livraison de type push.

<div id="how-the-events-api-works">
  ## Fonctionnement de l’Events API
</div>

Lorsque votre application se connecte à l’Events API, elle reçoit un flux de messages SSE. Chaque message comprend un champ `id` qui sert d’**offset**. Si la connexion est interrompue, votre application se reconnecte et fournit le dernier offset reçu. Auth0 reprend l’envoi à partir de ce point, de sorte qu’aucun événement n’est perdu.

Le flux SSE comprend les types de messages suivants :

| Type de message                                 | Rôle                                                                                                                                                            |
| ----------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `:connected`                                    | Confirme que la connexion est établie. Il s’agit d’un commentaire SSE, et non d’un événement de données.                                                        |
| `retry: <ms>`                                   | Indique au client SSE combien de temps attendre avant de se reconnecter après une déconnexion.                                                                  |
| `event: <type>` (par exemple, `user.created`)   | Un événement réel dont le payload complet se trouve dans le champ `data`.                                                                                       |
| `event: offset-only`                            | Un marqueur de progression envoyé à intervalles réguliers (selon la fréquence du heartbeat). Met à jour l’offset sans transmettre de données d’événement.       |
| `:` suivi de texte (par exemple, `: heartbeat`) | Un commentaire de maintien en vie qui empêche les serveurs proxy et les répartiteurs de charge de fermer les connexions inactives. Aucune action n’est requise. |
| `event: error`                                  | Une erreur terminale. Le flux se ferme après ce message.                                                                                                        |

<div id="example-sse-stream">
  ### Exemple de flux SSE
</div>

```text wrap lines
:connected

retry: 2000

event: user.created
id: MTIzNDIzNDEzCg==
data: {"offset":"MTIzNDIzNDEzCg==","event":{"id":"evt_abc123","type":"user.created","time":"2025-06-01T12:00:00Z","data":{"object":{"user_id":"auth0|123","email":"jane@example.com"}}}}

event: offset-only
id: 4LcuTXmVDASuNRQt
data: {"offset":"4LcuTXmVDASuNRQt"}

: heartbeat

event: user.updated
id: NTY3ODkwMTIzCg==
data: {"offset":"NTY3ODkwMTIzCg==","event":{"id":"evt_def456","type":"user.updated","time":"2025-06-01T12:05:00Z","data":{"object":{"user_id":"auth0|123","email":"jane.doe@example.com"}}}}
```

<div id="prerequisites">
  ## Prérequis
</div>

Avant de commencer, assurez-vous d’avoir :

* Un tenant Auth0 avec Events activé. Le nombre de connexions Event Stream offertes dépend de votre forfait :

  | Forfait      | Limite de connexions |
  | ------------ | -------------------- |
  | Free         | 1                    |
  | Self-service | 4                    |
  | Enterprise   | 8                    |

* Un jeton d’accès à l’API de gestion avec la portée `read:events`. Pour en savoir plus, consultez [Jetons d’accès à l’API de gestion](/docs/fr-CA/secure/tokens/access-tokens/management-api-access-tokens).

<div id="connect-to-the-events-api">
  ## Se connecter à l’Events API
</div>

Ouvrez une connexion SSE vers l’endpoint d’événements de votre tenant. L’exemple suivant utilise `curl` :

```bash wrap lines
curl -N --http2 \
    -H "Authorization: Bearer YOUR_MANAGEMENT_API_TOKEN" \
    -H "Accept: text/event-stream" \
    "https://YOUR_DOMAIN/api/v2/events"
```

<div id="query-parameters">
  ### Paramètres de requête
</div>

Utilisez les paramètres de requête pour filtrer le flux ou le reprendre :

| Paramètre        | Type   | Description                                                                                                                                                                                                                                                                                                                                  |
| ---------------- | ------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `from`           | string | Un offset renvoyé dans un champ `id` précédent. La livraison reprend après cet offset.                                                                                                                                                                                                                                                       |
| `from_timestamp` | string | Un horodatage ISO 8601. Renvoie les événements qui se sont produits à ce moment ou après. Ne peut pas être utilisé avec `from`. À privilégier pour la configuration initiale ou pour rejouer des événements à partir d’un moment précis; pour une consommation en continu, préférez reprendre avec `from`, car les offsets sont plus précis. |
| `event_type`     | string | Le type d’événement à inclure. Répétez le paramètre pour chaque type (par exemple, `event_type=user.created&event_type=user.updated`).                                                                                                                                                                                                       |

```bash wrap lines
curl -N --http2 \
    -H "Authorization: Bearer YOUR_MANAGEMENT_API_TOKEN" \
    -H "Accept: text/event-stream" \
    "https://YOUR_DOMAIN/api/v2/events?event_type=user.created&event_type=user.updated&from_timestamp=2025-06-01T00:00:00Z"
```

<div id="resume-after-a-disconnection">
  ## Reprendre après une déconnexion
</div>

Les connexions SSE peuvent être interrompues pour de nombreuses raisons : problèmes de réseau, expiration du token ou renouvellement des connexions côté serveur (Auth0 ferme périodiquement les connexions pour faire de l’équilibrage de charge — généralement toutes les quelques minutes). Les bibliothèques clientes SSE standard gèrent cela de manière transparente en se reconnectant et en envoyant le dernier offset.

Il existe deux façons de fournir l’offset lors de la reconnexion :

* **`Last-Event-ID` header** — le mécanisme standard de reconnexion SSE. La plupart des bibliothèques clientes SSE définissent automatiquement cet en-tête lors de la reconnexion.
* **`from` query parameter** — utilisez-le lorsque votre client ne prend pas en charge l’en-tête `Last-Event-ID`.

Si les deux sont fournis, l’en-tête `Last-Event-ID` est prioritaire.

```bash wrap lines
curl -N --http2 \
    -H "Authorization: Bearer YOUR_MANAGEMENT_API_TOKEN" \
    -H "Accept: text/event-stream" \
    -H "Last-Event-ID: MTIzNDIzNDEzCg==" \
    "https://YOUR_DOMAIN/api/v2/events"
```

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Enregistrez la valeur `id` la plus récente de chaque message (y compris les messages `offset-only`) dans un stockage persistant. Si votre application redémarre, utilisez l’offset enregistré pour reprendre la réception là où elle s’était arrêtée.
</Callout>

<div id="handle-message-types">
  ## Traiter les types de messages
</div>

<div id="real-events">
  ### Événements complets
</div>

Les messages dont le champ `event` correspond à un type d’événement connu (par exemple, `user.created`) contiennent le payload complet de l’événement dans le champ `data`. Analysez le JSON et traitez l’événement selon votre logique métier.

<div id="offset-only-messages">
  ### Messages contenant uniquement un offset
</div>

Auth0 envoie des messages `offset-only` à intervalles réguliers (à la fréquence du heartbeat) pour faire avancer votre position dans le flux. Ces messages ne contiennent aucun payload d’événement. Mettez à jour l’offset que vous avez enregistré lorsque vous les recevez, afin qu’une reconnexion ultérieure ne rejoue pas des événements que vous avez déjà traités.

<div id="error-messages">
  ### Messages d’erreur
</div>

Un message `event: error` signale une erreur fatale, comme un offset expiré ou un problème côté serveur. Après avoir reçu ce message, le flux se ferme. Votre application doit consigner l’erreur, puis se reconnecter avec l’offset approprié ou un nouveau `from_timestamp`.

<div id="heartbeats">
  ### Heartbeats
</div>

Les lignes qui commencent par `:` sont des commentaires SSE utilisés comme signaux de heartbeat. Ils maintiennent la connexion active à travers les serveurs mandataires et les répartiteurs de charge. Aucun traitement n’est nécessaire.

<div id="server-side-connection-cycling">
  ## Rotation des connexions côté serveur
</div>

Auth0 ferme périodiquement les connexions SSE pour répartir la charge (généralement toutes les quelques minutes). Il s’agit d’un comportement normal, et non d’une erreur. Les bibliothèques clientes SSE standard (y compris le package npm `eventsource`) se reconnectent automatiquement à l’aide de l’en-tête `Last-Event-ID`, ce qui permet à votre application de reprendre au bon offset sans perdre d’événements.

Si vous créez un client SSE personnalisé, assurez-vous qu’il gère bien les interruptions de connexion en enregistrant le dernier offset et en s’y reconnectant.

<div id="implement-a-consumer">
  ## Mettre en place un consommateur
</div>

L’exemple Node.js suivant montre un consommateur minimal de l’Events API qui traite les événements et enregistre l’offset dans un fichier.

```javascript wrap lines
const EventSource = require("eventsource");
const fs = require("fs");

const OFFSET_FILE = "./offset.txt";
const AUTH0_DOMAIN = "YOUR_DOMAIN";
const TOKEN = "YOUR_MANAGEMENT_API_TOKEN";

function loadOffset() {
    try {
        return fs.readFileSync(OFFSET_FILE, "utf8").trim();
    } catch {
        return null;
    }
}

function saveOffset(offset) {
    fs.writeFileSync(OFFSET_FILE, offset);
}

function connect() {
    const offset = loadOffset();
    const params = new URLSearchParams();
    params.append("event_type", "user.created");
    params.append("event_type", "user.updated");
    params.append("event_type", "user.deleted");

    if (offset) {
        params.append("from", offset);
    }

    const url = `https://${AUTH0_DOMAIN}/api/v2/events?${params.toString()}`;

    const es = new EventSource(url, {
        headers: {
            "Authorization": `Bearer ${TOKEN}`
        }
    });

    // Gérer les événements réels
    for (const type of ["user.created", "user.updated", "user.deleted"]) {
        es.addEventListener(type, (msg) => {
            const payload = JSON.parse(msg.data);
            console.log(`Received ${type}:`, payload.event.id);

            // Traiter l'événement ici

            saveOffset(msg.lastEventId);
        });
    }

    // Gérer les marqueurs de progression sans données
    es.addEventListener("offset-only", (msg) => {
        saveOffset(msg.lastEventId);
    });

    // Gérer les erreurs terminales
    es.addEventListener("error", (msg) => {
        if (msg.data) {
            const errorPayload = JSON.parse(msg.data);
            console.error("Stream error:", errorPayload.error);
        }
        es.close();

        // Se reconnecter après un délai
        setTimeout(connect, 5000);
    });
}

connect();
```

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Le package npm `eventsource` implémente le protocole SSE et gère automatiquement la reconnexion à l’aide de l’en-tête `Last-Event-ID`. Si vous utilisez une autre bibliothèque SSE, vérifiez qu’elle prend en charge la reconnexion automatique et la transmission de l’offset.
</Callout>

<div id="error-responses">
  ## Réponses d’erreur
</div>

L’Events API renvoie les codes d’état HTTP standard lorsque la connexion ne peut pas être établie :

| Code d’état | Signification                                                                                                     |
| ----------- | ----------------------------------------------------------------------------------------------------------------- |
| `200`       | Connexion établie. La diffusion des Events commence.                                                              |
| `400`       | Requête non valide. La valeur de l’offset est mal formée ou le type d’événement demandé n’est pas pris en charge. |
| `401`       | Jeton d’accès manquant ou non valide.                                                                             |
| `403`       | Le jeton d’accès n’inclut pas la portée `read:events`.                                                             |
| `410`       | L’offset a expiré. Utilisez une valeur `from_timestamp` pour reprendre à un moment précis.                        |
| `429`       | Rate limit dépassée. Attendez, puis réessayez en utilisant l’intervalle indiqué dans l’en-tête `Retry-After`.     |
