Les sections suivantes présentent de l’information sur les tests des flux d’événements et la gestion des événements.

<div id="test-the-event-stream">
  ## Tester le flux d’événements
</div>

Après avoir créé et activé un flux d’événements, vous pouvez tester ce nouveau flux. À l’heure actuelle, Auth0 ne permet pas d’envoyer des événements vers des flux d’événements désactivés.

Pour tester votre flux d’événements, commencez par créer un utilisateur pour votre application en exécutant la commande suivante avec les identifiants de votre choix :

<Tabs>
  <Tab title="Auth0 CLI">
    ```bash lines
    auth0 users create # suivez les instructions pour créer un utilisateur de test
    ```
  </Tab>

  <Tab title="cURL">
    ```bash lines
    curl -s \
      -H "Authorization: Bearer $TOKEN" \
      -H "Content-Type: application/json" \
      -X POST https://$TENANT/api/v2/users \
      -d '{
            "email": "nick-testing-21@example.com",
            "password": "Password123!",
            "connection": "Username-Password-Authentication"
          }'
    ```
  </Tab>
</Tabs>

Une fois l’utilisateur créé, cela devrait déclencher l’événement `user.created` dans votre flux d’événements et renvoyer une réponse. Dans cet exemple, des webhooks sont utilisés pour acheminer le flux d’événements.

```json lines expandable
"Webhook received": {
  "id": "evt_6SfwXrtm8nVcjzzNCFFH1L",
  "source": "urn:auth0:",
  "specversion": "1.0",
  "type": "user.created",
  "time": "2025-02-02T17:56:01.573Z",
  "data": {
    "object": {
      "created_at": "2025-02-02T17:56:01.566Z",
      "email": "nick-testing-21@example.com",
      "email_verified": false,
      "identities": [
        {
          "connection": "Username-Password-Authentication",
          "isSocial": false,
          "provider": "auth0",
          "user_id": "679fb1b1e809445f129bc18d"
        }
      ],
      "name": "nick-testing-21@example.com",
      "nickname": "nick-testing-21",
      "picture": "test.png",
      "updated_at": "2025-02-02T17:56:01.566Z",
      "user_id": "auth0|679fb1b1e809445f129bc18d"
    }
  },
  "a0tenant": "acme",
  "a0stream": "est_38ANnv8pQhqsZrX2VFbftv",
  "a0purpose": "test"
}
```

Vous pouvez afficher la liste des échecs de livraison pour un flux d’événements précis en exécutant :

<Tabs>
  <Tab title="Auth0 CLI">
    ```bash lines
    auth0 api get event-streams/$EVENT_STREAM_ID/deliveries
    ```
  </Tab>

  <Tab title="cURL">
    ```bash lines
    curl -s -H "Authorization: Bearer $TOKEN" \
      "https://$TENANT/api/v2/event-streams/${EVENT_STREAM_ID}/deliveries" | jq .
    ```
  </Tab>
</Tabs>

Si le tableau des échecs de livraison renvoyé est vide, votre flux d’événements fonctionne correctement.

<div id="observability">
  ## Observabilité
</div>

Pour assurer la fiabilité des flux d’événements, mettez en place un processus de surveillance qui contrôle périodiquement les échecs de livraison et déclenche des alertes au besoin.

Les échecs de livraison peuvent être causés par ce qui suit :

* Flux d’événements désactivés.
* Flux d’événements non disponibles.
* Erreurs de connexion.

<div id="polling-deliveries-endpoint">
  ### Point de terminaison Deliveries pour le polling
</div>

Interrogez le point de terminaison Deliveries toutes les **5 minutes** pour vérifier s’il y a des événements en échec. Utilisez l’appel d’API suivant pour récupérer les livraisons d’événements récentes :

<Tabs>
  <Tab title="Auth0 CLI">
    ```bash lines
    auth0 api get event-streams/$EVENT_STREAM_ID/deliveries
    ```
  </Tab>

  <Tab title="cURL">
    ```bash lines
    curl -s -H "Authorization: Bearer $TOKEN" \
      "https://$TENANT/api/v2/event-streams/${EVENT_STREAM_ID}/deliveries" | jq .
    ```
  </Tab>
</Tabs>

Si des échecs sont détectés :

* Consignez les détails des événements en échec.
* Alertez l’équipe ou le système approprié pour examiner et diagnostiquer le problème.

<div id="recovery">
  ## Reprise
</div>

La livraison des événements fera automatiquement l’objet de nouvelles tentatives. Pour en savoir plus, consultez les [Bonnes pratiques](/docs/fr-CA/customize/events/event-testing-observability-and-failure-recovery#batch-redelivery-of-failed-events).

En cas d’échec, utilisez la Redelivery API pour tenter de livrer l’événement de nouveau.

<div id="redeliver-a-single-failed-event">
  ### Réacheminer un seul événement en échec
</div>

Pour réessayer la livraison d’un événement en échec précis, utilisez :

<Tabs>
  <Tab title="Auth0 CLI">
    ```bash lines
    EVENT_ID=<your event id>
    auth0 api POST event-streams/$EVENT_STREAM_ID/redeliver/$EVENT_ID --data '{}'
    ```
  </Tab>

  <Tab title="cURL">
    ```bash lines
    curl -s -X POST -H "Authorization: Bearer $TOKEN" \
      "https://<your-domain>/api/v2/event-streams/$EVENT_STREAM_ID/redeliver/$EVENT_ID"
    ```
  </Tab>
</Tabs>

<div id="batch-redelivery-of-failed-events">
  ### Renvoi par lot des événements en échec
</div>

Pour réessayer tous les événements en échec dans le flux d’événements :

<Tabs>
  <Tab title="Auth0 CLI">
    ```bash wrap lines
    auth0 api POST event-streams/$EVENT_STREAM_ID/redeliver --data '{}'
    ```
  </Tab>

  <Tab title="cURL">
    ```bash lines
    curl -s -X POST -H "Authorization: Bearer $TOKEN" \
      "https://<your-tenant>.auth0.com/api/v2/event-streams/<event_stream_id>/redeliver"
    ```
  </Tab>
</Tabs>

<div id="auto-disable-mechanism">
  ### Mécanisme de désactivation automatique
</div>

Auth0 désactive automatiquement un flux d’événements en cas de trop grand nombre d’échecs.

| Condition              | Mesure prise                                        |
| ---------------------- | --------------------------------------------------- |
| 500 échecs consécutifs | Le flux d’événements est automatiquement désactivé. |
| 5000 échecs au total   | Le flux d’événements est automatiquement désactivé. |

* Si un flux d’événements est désactivé automatiquement, vous **devez** utiliser la Redelivery API pour traiter tous les événements en échec.
* Le flux d’événements **ne peut pas** être réactivé manuellement tant que le nombre total d’échecs ne redescend pas sous 5000.
* Une fois le nombre d’échecs réduit, vous pouvez réactiver le flux par API.
