title: Synchroniser les données entre les systèmes
---

Avec flux d’événements, vous pouvez conserver une copie de vos données d’identité d’Auth0 dans un système externe, comme une base de données relationnelle, un entrepôt de données ou un index de recherche. Lorsqu’un profil d’utilisateur est créé, mis à jour ou supprimé dans Auth0, un événement est envoyé vers la destination de votre stream pour que votre système externe puisse y appliquer la même modification.

<div id="why-synchronize-identity-data">
  ## Pourquoi synchroniser les données d’identité
</div>

Il est utile de conserver une copie locale des données d’identité lorsque vous devez :

* Effectuer des analyses, produire des rapports ou exécuter des requêtes de conformité sans appeler la Management API.
* Offrir des fonctions de recherche qui nécessitent des recherches à faible latence parmi les attributs des utilisateurs.
* Alimenter des pipelines de données qui combinent des enregistrements d’identité avec d’autres données d’entreprise.
* Conserver une copie de sauvegarde de l’état du profil utilisateur pour la reprise après sinistre.

<div id="how-it-works">
  ## Fonctionnement
</div>

1. Auth0 publie un événement chaque fois qu’un profil utilisateur est modifié.
2. Votre flux d’événements transmet cet événement à une destination (webhook, AWS EventBridge ou Auth0 Action).
3. Votre gestionnaire examine le type d’événement et applique l’opération d’écriture correspondante dans le système externe.

Les types d’événements suivants sont pertinents pour la synchronisation des données :

| Type d’événement | Quand il est déclenché                             |
| ---------------- | -------------------------------------------------- |
| `user.created`   | Un nouveau profil utilisateur est créé dans Auth0. |
| `user.updated`   | Un profil utilisateur existant est modifié.        |
| `user.deleted`   | Un profil utilisateur est supprimé d’Auth0.        |

<div id="prerequisites">
  ## Prérequis
</div>

Avant de commencer, assurez-vous d’avoir :

* Un tenant Auth0 avec Events activé. Pour en savoir plus sur la disponibilité selon votre plan, consultez [Create an Event Stream](/docs/fr-CA/customize/events/create-an-event-stream).
* Un flux d’événements actif abonné à `user.created`, `user.updated` et `user.deleted`. Pour en savoir plus, consultez [Create an Event Stream](/docs/fr-CA/customize/events/create-an-event-stream).
* Une banque de données externe (par exemple, PostgreSQL, MySQL ou un entrepôt de données) dans laquelle votre gestionnaire peut écrire.

<div id="implement-data-synchronization">
  ## Mettre en œuvre la synchronisation des données
</div>

Les sections ci-dessous montrent comment gérer chaque type d’événement. Les fonctions de gestionnaire sont les mêmes, quelle que soit la destination de votre flux d’événements. La section [Acheminer les événements par type](#route-events-by-type) montre comment acheminer les événements vers des destinations webhook et Auth0 Action.

<div id="handle-usercreated">
  ### Traiter user.created
</div>

Lorsqu’Auth0 publie un événement `user.created`, insérez une nouvelle entrée dans votre base de données.

```javascript wrap lines
async function handleUserCreated(user, time) {
    const { user_id, email, name, nickname, created_at, updated_at } = user;

    const query = `
        INSERT INTO users (user_id, email, name, nickname, created_at, updated_at, last_event_processed)
        VALUES ($1, $2, $3, $4, $5, $6, $7)
    `;
    const values = [user_id, email, name, nickname, created_at, updated_at, time];

    await pool.query(query, values);
}
```

<div id="handle-userupdated">
  ### Traiter user.updated
</div>

Quand Auth0 publie un événement `user.updated`, mettez à jour la ligne correspondante. Comparez l’horodatage de l’événement à la colonne `last_event_processed` pour éviter de remplacer des données plus récentes par des données périmées.

```javascript wrap lines
async function handleUserUpdated(user, time) {
    const { user_id, email, name, nickname, updated_at } = user;

    const query = `
        UPDATE users
        SET email = $1, name = $2, nickname = $3, updated_at = $4, last_event_processed = $5
        WHERE user_id = $6 AND last_event_processed < $5
    `;
    const values = [email, name, nickname, updated_at, time, user_id];

    await pool.query(query, values);
}
```

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Les événements peuvent arriver hors séquence. Comparez toujours les horodatages avant d’appliquer les mises à jour pour éviter que des données périmées n’écrasent des enregistrements plus récents. Pour en savoir plus, consultez [Events Best Practices](/docs/fr-CA/customize/events/events-best-practices).
</Callout>

<div id="handle-userdeleted">
  ### Traiter l’événement user.deleted
</div>

Lorsqu’Auth0 publie un événement `user.deleted`, supprimez l’enregistrement correspondant ou marquez-le comme supprimé.

```javascript wrap lines
async function handleUserDeleted(user) {
    const { user_id } = user;

    const query = `DELETE FROM users WHERE user_id = $1`;

    await pool.query(query, [user_id]);
}
```

<div id="route-events-by-type">
  ### Acheminer les événements par type
</div>

Utilisez un routeur principal pour acheminer chaque événement vers le bon gestionnaire. Les exemples ci-dessous montrent comment acheminer les événements vers des destinations webhook et Auth0 Action.

<Tabs>
  <Tab title="Webhook">
    ```javascript wrap lines
    app.post("/webhook", async (req, res) => {
        const { type, time, data } = req.body;
        const user = data.object;

        try {
            switch (type) {
                case "user.created":
                    await handleUserCreated(user, time);
                    break;
                case "user.updated":
                    await handleUserUpdated(user, time);
                    break;
                case "user.deleted":
                    await handleUserDeleted(user);
                    break;
                default:
                    console.log(`Unhandled event type: ${type}`);
            }

            res.sendStatus(204);
        } catch (err) {
            console.error("Error processing event:", err);
            res.status(500).json({ error: "Internal server error" });
        }
    });
    ```

    <Callout icon="file-lines" color="#0EA5E9" iconType="regular">
      Retournez une réponse HTTP `2XX` le plus rapidement possible. Si votre gestionnaire doit exécuter des opérations longues, mettez l’événement dans une file d’attente interne et traitez-le de façon asynchrone. Pour en savoir plus, consultez [Events Best Practices](/docs/fr-CA/customize/events/events-best-practices).
    </Callout>
  </Tab>

  <Tab title="Auth0 Action">
    ```javascript wrap lines
    exports.onExecuteEventStream = async (event, api) => {
        const eventType = event.message.type;
        const user = event.message.data.object;
        const time = event.message.time;

        switch (eventType) {
            case "user.created":
                await handleUserCreated(user, time);
                break;
            case "user.updated":
                await handleUserUpdated(user, time);
                break;
            case "user.deleted":
                await handleUserDeleted(user);
                break;
            default:
                console.log(`Unhandled event type: ${eventType}`);
        }
    };
    ```
  </Tab>
</Tabs>

<div id="guard-against-duplicates-and-ordering-issues">
  ## Se prémunir contre les doublons et les problèmes d’ordre
</div>

flux d’événements assure une livraison au moins une fois, ce qui signifie que votre gestionnaire peut recevoir le même événement plus d’une fois. Pour gérer cela de façon sécuritaire :

* **Suivez les ID d’événement.** Enregistrez chaque `id` d’événement traité et ignorez tout événement que vous avez déjà pris en charge.
* **Comparez les horodatages.** Le payload de chaque événement comprend les champs `created_at` et `updated_at` dans `data.object`. Utilisez ces champs pour déterminer si un événement entrant est plus récent que celui que votre système a déjà enregistré.
* **Utilisez des écritures idempotentes.** Structurez vos opérations de base de données de manière à ce que l’application du même événement deux fois produise le même résultat. Par exemple, utilisez `INSERT ... ON CONFLICT DO UPDATE` dans PostgreSQL.

```sql wrap lines
INSERT INTO users (user_id, email, name, nickname, created_at, updated_at, last_event_processed)
VALUES ($1, $2, $3, $4, $5, $6, $7)
ON CONFLICT (user_id) DO UPDATE
SET email = EXCLUDED.email,
    name = EXCLUDED.name,
    nickname = EXCLUDED.nickname,
    updated_at = EXCLUDED.updated_at,
    last_event_processed = EXCLUDED.last_event_processed
WHERE users.last_event_processed < EXCLUDED.last_event_processed;
```

<div id="verify-synchronization">
  ## Vérifiez la synchronisation
</div>

Après avoir déployé votre gestionnaire, créez un utilisateur de test dans Auth0 et confirmez ce qui suit :

1. Une nouvelle ligne apparaît dans votre base de données externe avec les bonnes données de profil.
2. Mettez à jour le nom ou l’adresse courriel de l’utilisateur dans Auth0. Confirmez que la ligne de la base de données reflète ce changement.
3. Supprimez l’utilisateur dans Auth0. Confirmez que la ligne est supprimée (ou marquée comme supprimée) dans votre base de données.

Pour en savoir plus sur les tests des flux d’événements, consultez [Event Testing, Observability, and Failure Recovery](/docs/fr-CA/customize/events/event-testing-observability-and-failure-recovery).
