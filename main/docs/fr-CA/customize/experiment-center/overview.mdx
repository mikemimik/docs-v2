Auth0 Experiment Center est un moteur d’expérimentation intégré d’Auth0. Il vous permet d’effectuer des tests A/B sur les modifications apportées à votre expérience d’[authentification](/docs/fr-CA/authenticate/login/auth0-universal-login) et d’en voir l’incidence dans les [journaux](/docs/fr-CA/deploy-monitor/logs) de vos événements d’authentification.

Avec Experiment Center, vous définissez votre test A/B dans Auth0, le trafic est réparti de façon déterministe, et vos événements d’authentification existants sont enrichis de métadonnées d’expérience afin que vous puissiez analyser les résultats dans vos propres outils.

<div id="how-it-works">
  ## Fonctionnement
</div>

Experiment Center vous permet d’exécuter des tests A/B contrôlés sur votre pipeline d’authentification Auth0. Au lieu de déployer un changement pour tous les utilisateurs en même temps, vous exposez un nouveau comportement à un pourcentage contrôlé du trafic, mesurez les résultats à l’aide d’auth events enrichis, puis déployez la variation gagnante lorsque vous êtes prêt.

Experiment Center repose sur trois entités :

* **Experiment** : définit comment le trafic est réparti et à quel moment le test s’exécute.
* **Feature flag** : définit ce que vous testez et les variations possibles.
* **Segment** : définit un ensemble de règles pour orienter les expériences vers des variations précises.

<Warning>
  Pendant la phase bêta, Experiment Center fonctionne uniquement sur les tenants de développement. Les tenants de production ne sont pas pris en charge.
</Warning>

<div id="experiment">
  ## Expérience
</div>

Une expérience sert de couche de mesure autour d’un Feature flag. Elle définit :

* Quel Feature flag est testé
* Comment le trafic est réparti entre les variations
* Quand le test se déroule

Chaque expérience est associée à un seul Feature flag.

<div id="experiment-lifecycle">
  ### Cycle de vie de l’expérience
</div>

Les expériences ont cinq états :

| Statut      | Signification                                                                                                                              |
| ----------- | ------------------------------------------------------------------------------------------------------------------------------------------ |
| `draft`     | Créé, mais pas en cours d’exécution. Peut être testé sans risque à l’aide des remplacements de paramètres de requête.                      |
| `active`    | En cours d’exécution. L’attribution des variantes et l’injection de contexte sont actives pour toutes les transactions d’authentification. |
| `paused`    | Temporairement suspendu. Aucune nouvelle attribution. Les sessions en cours conservent la variation qui leur a été attribuée.              |
| `completed` | Terminé. Aucune nouvelle attribution. La configuration est conservée à des fins de référence pendant la promotion manuelle.                |
| `archived`  | Supprimé de façon logique. Masqué dans les vues de liste par défaut. Les données analytiques sont conservées.                              |

<div id="allocation-strategies">
  ### Stratégies d’allocation
</div>

Une expérience utilise l’une des deux stratégies d’allocation suivantes :

**Basée sur un pourcentage :** Le trafic est réparti entre les variations selon leur pondération. Le total des pondérations doit être de 100. Une pondération de 0 est valide (la variation figure dans la définition de l’expérience, mais ne reçoit aucun trafic).

**Basée sur des segments (ciblée) :** Le trafic est acheminé vers les variations selon l’appartenance à un segment. Les segments sont évalués par ordre de priorité. Le premier segment correspondant est retenu. Si aucun segment ne correspond, l’allocation `is_fallback` reçoit la requête.

Pour en savoir plus sur l’entité d’expérience, consultez [Détails des entités](/docs/fr-CA/customize/experiment-center/entities#experiment).

<div id="experiment-context">
  ### Contexte de l’expérience
</div>

Lorsqu’une expérience est active et qu’une variation est attribuée, Experiment Center injecte un objet `ExperimentContext` dans vos surfaces d’exécution.

Les écrans [ACUL](/docs/fr-CA/customize/login-pages/advanced-customizations#advanced-customizations-for-universal-login-acul) ne le reçoivent que si vous les activez au moyen de `context_configuration`.
Pour en savoir plus, consultez le [guide d’intégration ACUL](/docs/fr-CA/customize/experiment-center/integrations/acul-integration-guide).

Les [Actions](/docs/fr-CA/customize/actions/actions-overview) et les [gabarits de page](/docs/fr-CA/customize/login-pages/universal-login/customize-templates#customize-universal-login-page-templates) le reçoivent automatiquement dès qu’une expérience est active. Structure de l’objet :

```typescript
{
  experiment_id: string;   // L'ID d'expérience actif
  variation_id: string;    // L'ID de variation assigné
  config: {                // Configuration fusionnée (configuration de base + remplacements)
    [paramName]: { value: unknown }
  };
  is_control: boolean;     // Vrai si c'est la variation de contrôle
}
```

Le champ `config` contient la configuration complète issue de la fusion pour la variation attribuée. Chaque paramètre défini pour le feature flag a toujours une valeur. Vous n’avez pas besoin d’écrire de logique de secours.

<div id="assignment">
  ### Attribution
</div>

Une attribution correspond à la façon dont un utilisateur est dirigé vers une variation précise pendant une transaction d’authentification. Les attributions sont déterministes et persistantes : le même utilisateur voit toujours la même variation pour la même expérience sur le même appareil, de sorte que son expérience demeure stable d’une connexion à l’autre.

* **Attribution en pourcentage** répartit le trafic entre les variations selon leur pondération. Un sujet donné est toujours attribué à la même variation.
* **Attribution par segment** évalue les propriétés de la requête en fonction des règles de votre segment, par ordre de priorité. Le premier segment correspondant détermine la variation, et une requête qui correspond au même segment aboutit toujours à la même variation.

Les résultats d’attribution sont consignés dans les journaux de votre tenant sous `details.experiment`; ils ne sont pas exposés par une API distincte.

Pour en savoir plus sur l’entité d’attribution, consultez [Détails des entités](/docs/fr-CA/customize/experiment-center/entities#assignment).

<div id="feature-flag">
  ## Feature flag
</div>

Un Feature flag est l’unité de contrôle de ce qui est testé. Il contient :

* Une **configuration de base** : des paramètres typés et leurs valeurs par défaut
* Une ou plusieurs **variations** : des configurations de rechange qui diffèrent de la configuration de base

Les Feature flags sont propres au tenant et réutilisables. Le même flag peut être utilisé par plusieurs expériences au fil du temps (par exemple, un test du T1 et un ajustement du T2 sur la même fonctionnalité).

<div id="feature-flag-lifecycle">
  ### Cycle de vie d’un Feature flag
</div>

Les Feature flags ont un cycle de vie enregistré qui comporte trois états :

| Statut     | Signification                                                                                                  |
| ---------- | -------------------------------------------------------------------------------------------------------------- |
| `draft`    | Créé, mais pas encore actif. Ne peut pas être référencé par une expérience active.                             |
| `active`   | Prêt à être utilisé. Requis avant qu’une expérience qui référence ce Feature flag puisse être activée.         |
| `archived` | État terminal. Aucune nouvelle expérience ne peut référencer ce Feature flag. Créez un nouveau flag au besoin. |

Pour en savoir plus sur l’entité Feature flag, consultez [Détails des entités](/docs/fr-CA/customize/experiment-center/entities#feature-flag).

<div id="variation">
  ### Variation
</div>

Une variation est une version de l’expérience définie dans un feature flag. Elle précise quels paramètres de configuration diffèrent de la configuration de base, et dans quelle mesure.

* La **variation de contrôle** correspond à la configuration de base; ses remplacements sont vides (aucun paramètre n’est modifié par rapport aux valeurs par défaut du flag).
* Les **variations de traitement** définissent chacune un ou plusieurs remplacements de paramètres.

Les variations n’ont pas de marqueur `is_control`. Le fait qu’une variation soit le contrôle statistique d’une expérience donnée est défini au niveau de l’allocation, et non de la variation. Une même variation peut servir de contrôle dans une expérience et de traitement dans une autre.

Pour en savoir plus sur l’entité variation, consultez [Détails des entités](/docs/fr-CA/customize/experiment-center/entities#variation).

<div id="segment">
  ## Segment
</div>

Un segment est un groupe nommé de requêtes d’authentification qui correspondent à un ensemble de règles. Vous utilisez les segments dans des expériences d’allocation ciblée pour diriger des cohortes de trafic précises vers des variations précises.

Les segments sont limités à un tenant et réutilisables d’une expérience à l’autre.

Pour en savoir plus sur l’entité Segment, consultez [Détails des entités](/docs/fr-CA/customize/experiment-center/entities#segment).

<div id="limitations">
  ## Limites
</div>

Les limites suivantes s’appliquent pendant la bêta :

* **Une expérience active par tenant.** Vous pouvez avoir plusieurs expériences dans les états `draft`, `paused` ou `completed`, mais une seule peut être `active` à la fois.
* **Paramètres structurés uniquement.** Les paramètres des feature flags utilisent une structure clé/type/valeur.
* **Trois déclencheurs d’Actions.** Le contexte de l’expérience est disponible dans [`post_login`](/docs/fr-CA/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger), [`pre_user_registration`](/docs/fr-CA/customize/actions/explore-triggers/signup-and-login-triggers/pre-user-registration-trigger) et [`post_user_registration`](/docs/fr-CA/customize/actions/explore-triggers/signup-and-login-triggers/post-user-registration-trigger).
* **Trafic de test uniquement.** La bêta fonctionne sur des tenants de développement avec le trafic que vous générez. Les données des utilisateurs finaux de Production ne passent pas par la bêta.
* **Promotion manuelle.** Lorsqu’une expérience se termine, vous appliquez manuellement à votre tenant la configuration de la variation gagnante. La promotion automatisée sera offerte après la disponibilité générale.
