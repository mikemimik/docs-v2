Auth0 Experiment Center est le moteur d’expérimentation natif d’Auth0. Il vous permet d’effectuer des tests A/B sur votre expérience d’[authentification](/docs/fr-CA/authenticate/login/auth0-universal-login) et d’en voir l’impact dans les [journaux](/docs/fr-CA/deploy-monitor/logs) de vos événements d’authentification.

Avec Experiment Center, vous définissez votre test A/B dans Auth0, le trafic est réparti de façon déterministe et vos événements d’authentification existants sont enrichis de métadonnées d’expérience, afin que vous puissiez analyser les résultats dans vos propres outils.

<div id="how-it-works">
  ## Fonctionnement
</div>

Experiment Center vous permet d’exécuter des tests A/B contrôlés sur votre pipeline d’authentification Auth0. Au lieu de déployer un changement pour tous les utilisateurs en même temps, vous exposez un nouveau comportement à un pourcentage contrôlé du trafic, mesurez les résultats au moyen d’événements d’authentification enrichis et déployez la variante gagnante lorsque vous êtes prêt.

Experiment Center s’articule autour de trois entités :

* **Experiment** : définit comment le trafic est réparti et à quel moment le test s’exécute.
* **Feature flag** : définit ce que vous testez et les variations possibles.
* **Segment** : définit un ensemble de règles pour diriger les expériences vers des variations précises.

<Warning>
  Pendant la phase Beta, Experiment Center fonctionne uniquement sur les tenants de développement. Les tenants de production ne sont pas pris en charge.
</Warning>

<div id="experiment">
  ## Expérience
</div>

Une expérience est une couche de mesure appliquée à un feature flag. Elle définit :

* Quel feature flag est testé
* Comment le trafic est réparti entre les variations
* Quand le test est en cours

Chaque expérience renvoie à un seul feature flag.

<div id="experiment-lifecycle">
  ### Cycle de vie de l’expérience
</div>

Les expériences ont cinq états :

| Statut      | Signification                                                                                                                              |
| ----------- | ------------------------------------------------------------------------------------------------------------------------------------------ |
| `draft`     | Créé, mais non actif. Peut être testé sans risque à l’aide des surcharges de paramètres de requête.                                        |
| `active`    | En cours d’exécution. L’affectation des variantes et l’injection du contexte sont actives pour toutes les transactions d’authentification. |
| `paused`    | Temporairement suspendu. Aucune nouvelle attribution. Les sessions déjà en cours conservent la variation qui leur a été attribuée.         |
| `completed` | Terminé. Aucune nouvelle attribution. La configuration est conservée à titre de référence lors de la promotion manuelle.                   |
| `archived`  | Supprimé de façon réversible. Masqué des vues de liste par défaut. Les données analytiques sont conservées.                                |

<div id="allocation-strategies">
  ### Stratégies d’allocation
</div>

Une expérience utilise l’une des deux stratégies d’allocation suivantes :

**Basée sur un pourcentage :** le trafic est réparti entre les variations selon leur poids. Le total des poids doit être de 100. Un poids de 0 est valide (la variation figure dans la définition de l’expérience, mais ne reçoit aucun trafic).

**Basée sur des segments (ciblée) :** le trafic est acheminé vers les variations selon l’appartenance à un segment. Les segments sont évalués par ordre de priorité. Le premier segment correspondant est retenu. Si aucun segment ne correspond, l’allocation `is_fallback` reçoit la requête.

Pour en savoir plus sur l’entité d’expérience, consultez [Détails des entités](/docs/fr-CA/customize/experiment-center/entities#experiment).

<div id="experiment-context">
  ### Contexte de l’expérience
</div>

Lorsqu’une expérience est active et qu’une variation est attribuée, Experiment Center injecte un objet `ExperimentContext` dans vos surfaces d’exécution.

Les écrans [ACUL](/docs/fr-CA/customize/login-pages/advanced-customizations#advanced-customizations-for-universal-login-acul) ne le reçoivent que si vous l’activez pour l’écran au moyen de `context_configuration`.
Pour en savoir plus, consultez le [guide d’intégration ACUL](/docs/fr-CA/customize/experiment-center/integrations/acul-integration-guide).

Les [Actions](/docs/fr-CA/customize/actions/actions-overview) et les [gabarits de page](/docs/fr-CA/customize/login-pages/universal-login/customize-templates#customize-universal-login-page-templates) le reçoivent automatiquement dès qu’une expérience est active. Structure de l’objet :

```typescript
{
  experiment_id: string;   // L'ID de l'expérience active
  variation_id: string;    // L'ID de la variation assignée
  config: {                // Configuration fusionnée (référence de base + substitutions)
    [paramName]: { value: unknown }
  };
  is_control: boolean;     // True si c'est la variation de contrôle
}
```

Le champ `config` contient la configuration complète fusionnée pour la variation attribuée. Chaque paramètre défini dans le feature flag a toujours une valeur. Vous n’avez pas à écrire de logique de secours.

<div id="assignment">
  ### Affectation
</div>

Une affectation correspond à la façon dont un utilisateur est dirigé vers une variation précise pendant une transaction d’authentification. Les affectations sont déterministes et persistantes : le même utilisateur voit toujours la même variation pour la même expérience sur le même appareil, de sorte que son expérience demeure stable d’une connexion à l’autre.

* **Allocation en pourcentage** répartit le trafic entre les variations selon leur pondération. Un utilisateur donné est toujours affecté à la même variation.
* **Allocation par segment** évalue les propriétés de la requête selon les règles de votre segment, par ordre de priorité. Le premier segment correspondant détermine la variation, et une requête qui correspond au même segment est toujours associée à la même variation.

Les résultats d’affectation sont consignés dans les journaux de votre tenant sous `details.experiment`; ils ne sont pas exposés par une API distincte.

Pour en savoir plus sur l’entité d’affectation, consultez [Détails des entités](/docs/fr-CA/customize/experiment-center/entities#assignment).

<div id="feature-flag">
  ## Feature flag
</div>

Un feature flag est l’unité de contrôle de ce qui est testé. Il contient :

* Une **configuration de référence** : des paramètres typés et leurs valeurs par défaut
* Une ou plusieurs **variations** : des configurations de rechange qui diffèrent de la configuration de référence

Les feature flags sont limités à un tenant et réutilisables. Le même flag peut être utilisé par plusieurs experiments au fil du temps (par exemple, un test du T1 et un ajustement du T2 sur la même fonctionnalité).

<div id="feature-flag-lifecycle">
  ### Cycle de vie du feature flag
</div>

Les feature flags suivent un cycle de vie enregistré comportant trois états :

| Statut     | Signification                                                                                   |
| ---------- | ----------------------------------------------------------------------------------------------- |
| `draft`    | Créé, mais pas encore actif. Ne peut pas être utilisé par une expérience active.                |
| `active`   | Prêt à être utilisé. Requis pour qu’une expérience qui utilise ce flag puisse être activée.     |
| `archived` | État final. Aucune nouvelle expérience ne peut utiliser ce flag. Créez-en un nouveau au besoin. |

Pour en savoir plus sur l’entité feature flag, consultez [Détails des entités](/docs/fr-CA/customize/experiment-center/entities#feature-flag).

<div id="variation">
  ### Variation
</div>

Une variation est une version de l’expérience définie dans un feature flag. Elle précise quels paramètres de configuration diffèrent de la référence et dans quelle mesure.

* La **variation de contrôle** correspond à la référence; ses substitutions sont vides (aucun paramètre n’est modifié par rapport aux valeurs par défaut du flag).
* Les **variations de traitement** précisent chacune une ou plusieurs substitutions de paramètres.

Les variations n’ont pas de marqueur `is_control`. Le fait qu’une variation soit le contrôle statistique d’une expérience donnée est défini dans l’allocation, et non dans la variation. Une même variation peut être le contrôle dans une expérience et un traitement dans une autre.

Pour en savoir plus sur l’entité Variation, consultez [Détails des entités](/docs/fr-CA/customize/experiment-center/entities#variation).

<div id="segment">
  ## Segment
</div>

Un segment est un groupe nommé de requêtes d’authentification qui correspondent à un ensemble de règles. Les segments sont utilisés dans les expérimentations d’allocation ciblée pour acheminer des cohortes de trafic précises vers des variations précises.

Les segments sont propres à un tenant et réutilisables d’une expérience à l’autre.

Pour en savoir plus sur l’entité Segment, consultez [Détails des entités](/docs/fr-CA/customize/experiment-center/entities#segment).

<div id="limitations">
  ## Limites
</div>

Les limites suivantes s’appliquent pendant la bêta :

* **Une expérience active par tenant.** Vous pouvez avoir de nombreuses expériences dans les états `draft`, `paused` ou `completed`, mais une seule peut être `active` à la fois.
* **Paramètres structurés uniquement.** Les paramètres des feature flags utilisent la structure clé/type/valeur.
* **Trois déclencheurs d’Actions.** Le contexte de l’expérience est disponible dans [`post_login`](/docs/fr-CA/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger), [`pre_user_registration`](/docs/fr-CA/customize/actions/explore-triggers/signup-and-login-triggers/pre-user-registration-trigger) et [`post_user_registration`](/docs/fr-CA/customize/actions/explore-triggers/signup-and-login-triggers/post-user-registration-trigger).
* **Trafic de test uniquement.** La bêta fonctionne sur des tenants de développement avec le trafic que vous générez. Les données des utilisateurs finaux de Production ne transitent pas par la bêta.
* **Promotion manuelle.** Lorsqu’une expérience se termine, vous devez appliquer manuellement à votre tenant la configuration de la variation gagnante. La promotion automatisée est prévue après la disponibilité générale.
