Avec la [Management API](/docs/fr-CA/api/management/v2) d’Auth0, vous pouvez définir les détails des entités d’Experiment Center, y compris les propriétés, les états du cycle de vie et les règles de validation.

<Warning>
  Durant la période bêta, Experiment Center est offert uniquement pour les tenants de développement. Les tenants de production ne sont pas pris en charge.
</Warning>

<div id="experiment">
  ## Expérience
</div>

<div id="experiment-lifecycle">
  ### Cycle de vie de l’expérience
</div>

Les expériences ont cinq états :

| **Statut**  | **Signification**                                                                                                                       |
| ----------- | --------------------------------------------------------------------------------------------------------------------------------------- |
| `draft`     | Créé, mais pas en cours d’exécution. Peut être testé sans risque à l’aide des remplacements de paramètres de requête.                   |
| `active`    | En cours d’exécution. L’assignation des variantes et l’injection de contexte s’appliquent à toutes les transactions d’authentification. |
| `paused`    | Temporairement suspendu. Aucune nouvelle assignation. Les sessions en cours conservent la variation qui leur a été attribuée.           |
| `completed` | Terminé. Aucune nouvelle assignation. La configuration est conservée à titre de référence lors de la promotion manuelle.                |
| `archived`  | Supprimé de manière réversible. Masqué dans les vues de liste par défaut. Les données analytiques sont conservées.                      |

Pour changer d’état, utilisez le point de terminaison de la Management API [`/api/v2/experimentation/experiments/{id}/status`](/docs/fr-CA/api/management/v2/experimentation/update-experiment-status).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  La transition de `completed → active` n’est pas permise. Si vous devez exécuter de nouveau le même test, créez une nouvelle expérience.
</Callout>

<div id="the-is_valid-gate">
  ### Le contrôle `is_valid`
</div>

Avant qu’une expérience puisse être activée, elle doit réussir une vérification de préparation.

Experiment Center stocke un booléen `is_valid` pour chaque expérience et le recalcule à chaque modification. Cela vous donne une rétroaction immédiate pendant la configuration.

Vous pouvez aussi déterminer explicitement si une expérience est prête à l’aide du point de terminaison de la Management API [`/api/v2/experimentation/experiments/{id}/validate`](/docs/fr-CA/api/management/v2/experimentation/validate-experiment).

Cette opération renvoie `is_valid: true/false` ainsi qu’un tableau `errors` qui énumère tous les éléments bloquants. La même vérification s’exécute automatiquement lorsque vous tentez d’activer l’expérience via le point de terminaison de statut.

**Règles de validation vérifiées lors de l’activation :**

* Le drapeau de fonctionnalité référencé a le statut `active`
* Au moins une allocation existe
* Toutes les valeurs `variation_id` des allocations appartiennent au drapeau de fonctionnalité de l’expérience
* Exactement une allocation a `is_control: true`
* La somme des poids d’allocation est de 100 (stratégie en pourcentage)
* Exactement une allocation a `is_fallback: true` (stratégie par segment)
* Aucune autre expérience n’est actuellement active pour le tenant (Beta seulement)

<div id="assignment">
  ### Assignation
</div>

**Substitutions de paramètres de requête**

Vous pouvez forcer une assignation précise pour toute requête `/authorize` en transmettant des paramètres de requête :

| Paramètre       | Description                                              |
| --------------- | -------------------------------------------------------- |
| `experiment_id` | Inscrire la requête à une expérience précise             |
| `variation_id`  | Forcer une variation précise (nécessite `experiment_id`) |
| `segment_id`    | Forcer un segment précis (nécessite `experiment_id`)     |

Cela fonctionne pour les expériences, peu importe leur statut, y compris `draft`. Utilisez cette option pendant le développement pour vérifier que les deux variations s’affichent correctement avant l’activation.

<div id="feature-flag">
  ## Drapeau de fonctionnalité
</div>

<div id="feature-flag-lifecycle">
  ### Cycle de vie des drapeaux de fonctionnalité
</div>

Les drapeaux de fonctionnalité suivent un cycle de vie comportant trois états :

| **Statut** | **Signification**                                                                               |
| ---------- | ----------------------------------------------------------------------------------------------- |
| `draft`    | Créé, mais pas encore actif. Ne peut pas être utilisé par une expérience active.                |
| `active`   | Prêt à être utilisé. Requis avant qu’une expérience qui utilise ce flag puisse être activée.    |
| `archived` | État final. Aucune nouvelle expérience ne peut utiliser ce flag. Créez-en un nouveau au besoin. |

Pour passer d’un statut à un autre, utilisez le endpoint de l’Auth0 Management API [`/api/v2/experimentation/feature-flags/{id}/status`](/docs/fr-CA/api/management/v2/experimentation/update-feature-flag-status).

**Condition d’activation :** Un drapeau de fonctionnalité ne peut pas passer à `active` tant qu’il ne comporte pas au moins deux variations. Cela garantit que chaque flag actif comprend au minimum un contrôle et un traitement avant sa mise en ligne.

```text wrap lines
draft → active   (nécessite ≥2 variantes)
draft → archived
active → archived (terminal ; irréversible)
```

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Il n’est pas possible de passer de `archived → active`. Si vous archivez un drapeau de fonctionnalité et devez faire un autre test, créez un nouveau drapeau de fonctionnalité.
</Callout>

<div id="variation">
  ### Variation
</div>

**Paramètres et overrides**

Les paramètres du drapeau de fonctionnalité définissent la configuration de votre expérience. Chaque paramètre comprend :

* Un nom (par exemple, `show_passkey_prompt`)
* Un type : `string`, `boolean`, `number`, `array` ou `object`
* Une valeur par défaut (la configuration de référence)

Le champ `overrides` d’une variation indique uniquement les paramètres qui diffèrent. Au moment de l’exécution, Experiment Center fusionne la configuration de référence du flag avec les `overrides` de la variation et transmet l’objet `config` complet à ACUL, Actions et aux modèles de page. Chaque paramètre a toujours une valeur dans `config` ; vous n’avez jamais besoin d’écrire une logique de secours.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Les paramètres utilisent **uniquement le mode structuré**. Chaque paramètre a une clé nommée et une valeur typée.
</Callout>

<div id="segment">
  ## Segment
</div>

<div id="segment-rules">
  ### Règles de segment
</div>

La propriété `rules` d’un segment est un tableau d’objets de règle. Une requête correspond au segment si **au moins une** règle du tableau correspond. Une règle correspond si ses conditions respectent le `match_type` :

* `match_type: "all"` : toutes les conditions doivent être vraies (logique ET)
* `match_type: "any"` : au moins une condition doit être vraie (logique OU)

Chaque condition compare un attribut à une valeur au moyen d’un opérateur : `equals`, `not_equals`, `in`, `not_in`, `exists`, `not_exists`.

<div id="available-condition-attributes">
  ### Attributs de condition disponibles
</div>

Les Segments ne peuvent utiliser que les attributs disponibles au début de la transaction `/authorize` :

| **Catégorie**                 | **Attributs**                                      |
| ----------------------------- | -------------------------------------------------- |
| Client                        | `client_id`                                        |
| Connection                    | `connection`, `connection_type`                    |
| Organization                  | `organization_id`                                  |
| Domain                        | `domain`                                           |
| Appareil et navigateur        | `device_type`, `browser`, `platform`, `user_agent` |
| Géographique (dérivé de l’IP) | `country`, `region`                                |
