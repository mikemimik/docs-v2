Avec l’[Management API](/docs/fr-CA/api/management/v2) d’Auth0, vous pouvez définir les détails des entités d’Experiment Center, y compris les propriétés, les états du cycle de vie et les règles de validation.

<Warning>
  En version bêta, Experiment Center fonctionne uniquement avec les tenants de développement. Les tenants de production ne sont pas pris en charge.
</Warning>

<div id="experiment">
  ## Expérience
</div>

<div id="experiment-lifecycle">
  ### Cycle de vie de l’expérience
</div>

Les expériences ont cinq états :

| **Statut**  | **Signification**                                                                                                              |
| ----------- | ------------------------------------------------------------------------------------------------------------------------------ |
| `draft`     | Créé, mais non lancé. Peut être testé en toute sécurité à l’aide des remplacements de paramètres de requête.                   |
| `active`    | En cours. L’assignation des variantes et l’injection de contexte sont actives pour toutes les transactions d’authentification. |
| `paused`    | Temporairement suspendu. Aucune nouvelle assignation. Les sessions en cours conservent la variation qui leur a été assignée.   |
| `completed` | Terminé. Aucune nouvelle assignation. La configuration est conservée à titre de référence lors d’une promotion manuelle.       |
| `archived`  | Supprimé de façon logique. Masqué dans les vues de liste par défaut. Les données analytiques sont conservées.                  |

Pour changer d’état dans le cycle de vie, utilisez le point de terminaison de la Management API [`/api/v2/experimentation/experiments/{id}/status`](/docs/fr-CA/api/management/v2/experimentation/update-experiment-status).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  La transition de `completed → active` n’est pas autorisée. Si vous devez exécuter de nouveau le même test, créez une nouvelle expérience.
</Callout>

<div id="the-is_valid-gate">
  ### Le contrôle `is_valid`
</div>

Avant qu’une expérience puisse être activée, elle doit réussir une vérification de l’état de préparation.

Experiment Center stocke un booléen `is_valid` pour chaque expérience et le recalcule à chaque modification. Vous obtenez ainsi une rétroaction immédiate pendant la configuration.

Vous pouvez aussi vérifier explicitement l’état de préparation à l’aide du endpoint de la Management API [`/api/v2/experimentation/experiments/{id}/validate`](/docs/fr-CA/api/management/v2/experimentation/validate-experiment).

Cette opération renvoie `is_valid: true/false` et un tableau `errors` qui énumère tous les éléments bloquants. La même vérification s’exécute automatiquement lorsque vous tentez d’activer l’expérience au moyen du endpoint de status.

**Règles de validation vérifiées à l’activation :**

* Le feature flag référencé a le status `active`
* Au moins une allocation existe
* Toutes les valeurs `variation_id` d’allocation appartiennent au feature flag de l’expérience
* Exactement une allocation a `is_control: true`
* Le total des pondérations d’allocation est de 100 (stratégie en pourcentage)
* Exactement une allocation a `is_fallback: true` (stratégie de Segment)
* Aucune autre expérience n’est actuellement active pour le tenant (Beta seulement)

<div id="assignment">
  ### Affectation
</div>

**Remplacements par paramètres de requête**

Vous pouvez forcer une affectation précise pour toute requête `/authorize` en transmettant des paramètres de requête :

| Paramètre       | Description                                              |
| --------------- | -------------------------------------------------------- |
| `experiment_id` | Inscrire dans une expérience précise                     |
| `variation_id`  | Forcer une variation précise (nécessite `experiment_id`) |
| `segment_id`    | Forcer un segment précis (nécessite `experiment_id`)     |

Cela fonctionne pour les expériences, quel que soit leur statut, y compris `draft`. Utilisez cette option pendant le développement pour vérifier que les deux variations s’affichent correctement avant l’activation.

<div id="feature-flag">
  ## Feature flag
</div>

<div id="feature-flag-lifecycle">
  ### Cycle de vie des feature flags
</div>

Les feature flags suivent un cycle de vie comportant trois états :

| **Statut** | **Signification**                                                                                        |
| ---------- | -------------------------------------------------------------------------------------------------------- |
| `draft`    | Créé, mais pas encore actif. Ne peut pas être utilisé par une expérience active.                         |
| `active`   | Prêt à être utilisé. Obligatoire avant qu’une expérience qui utilise ce feature flag puisse être activée. |
| `archived` | État final. Aucune nouvelle expérience ne peut utiliser ce feature flag. Créez-en un nouveau au besoin.   |

Pour passer d’un statut à un autre, utilisez le point de terminaison de l’Auth0 Management API [`/api/v2/experimentation/feature-flags/{id}/status`](/docs/fr-CA/api/management/v2/experimentation/update-feature-flag-status).

**Condition d’activation :** un feature flag ne peut pas passer à l’état `active` tant qu’il ne comporte pas au moins deux variations. Cela garantit que chaque feature flag actif comporte au minimum un groupe témoin et un traitement avant sa mise en service.

```text wrap lines
draft → active   (requires ≥2 variations)
draft → archived
active → archived (terminal; no way back)
```

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Il n’est pas possible de passer de `archived → active`. Si vous archivez un feature flag et devez effectuer un autre test, créez un nouveau feature flag.
</Callout>

<div id="variation">
  ### Variante
</div>

**Paramètres et remplacements**

Les paramètres du feature flag définissent la configuration de votre expérience. Chaque paramètre comprend :

* Un nom (par exemple, `show_passkey_prompt`)
* Un type : `string`, `boolean`, `number`, `array` ou `object`
* Une valeur par défaut (la configuration de base)

Les `overrides` d’une variante indiquent uniquement les paramètres qui changent. À l’exécution, Experiment Center fusionne la configuration de base du flag avec les `overrides` de la variante et transmet l’objet `config` complet ainsi fusionné à ACUL, à Actions et aux gabarits de page. Chaque paramètre a toujours une valeur dans `config` ; vous n’avez donc jamais à écrire de logique de repli.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Les paramètres utilisent uniquement le **mode structuré**. Chaque paramètre est composé d’une clé nommée et d’une valeur typée.
</Callout>

<div id="segment">
  ## Segment
</div>

<div id="segment-rules">
  ### Règles du segment
</div>

Le `rules` d’un segment est un tableau d’objets règle. Une requête correspond au segment si **au moins une** règle du tableau correspond. Une règle correspond si ses conditions respectent le `match_type` :

* `match_type: "all"`: toutes les conditions doivent être vraies (logique ET)
* `match_type: "any"`: au moins une condition doit être vraie (logique OU)

Chaque condition compare un attribut à une valeur au moyen d’un opérateur : `equals`, `not_equals`, `in`, `not_in`, `exists`, `not_exists`.

<div id="available-condition-attributes">
  ### Attributs de condition disponibles
</div>

Les segments peuvent uniquement utiliser les attributs disponibles au démarrage de la transaction `/authorize` :

| **Catégorie**                 | **Attributs**                                      |
| ----------------------------- | -------------------------------------------------- |
| Client                        | `client_id`                                        |
| Connexion                     | `connection`, `connection_type`                    |
| Organisation                  | `organization_id`                                  |
| Domaine                       | `domain`                                           |
| Appareil et navigateur        | `device_type`, `browser`, `platform`, `user_agent` |
| Géographique (dérivé de l’IP) | `country`, `region`                                |
