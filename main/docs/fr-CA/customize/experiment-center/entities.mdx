Avec la [Management API](/docs/fr-CA/api/management/v2) d’Auth0, vous pouvez définir les détails de l’entité Experiment Center, y compris ses propriétés, ses états du cycle de vie et ses règles de validation.

<Warning>
  Durant la période bêta, Experiment Center fonctionne uniquement sur les tenants de développement. Les tenants de production ne sont pas pris en charge.
</Warning>

<div id="experiment">
  ## Expérience
</div>

<div id="experiment-lifecycle">
  ### Cycle de vie des expériences
</div>

Les expériences ont cinq états :

| **Statut**  | **Signification**                                                                                                                          |
| ----------- | ------------------------------------------------------------------------------------------------------------------------------------------ |
| `draft`     | Créé, mais non en cours d’exécution. Peut être testé sans risque à l’aide de remplacements de paramètres de requête.                       |
| `active`    | En cours d’exécution. L’attribution des variantes et l’injection du contexte sont actives pour toutes les transactions d’authentification. |
| `paused`    | Temporairement suspendu. Aucune nouvelle attribution. Les sessions en cours conservent leur variation attribuée.                           |
| `completed` | Terminé. Aucune nouvelle attribution. La configuration est conservée à des fins de référence pendant la promotion manuelle.                |
| `archived`  | Supprimé de façon réversible. Masqué dans les vues de liste par défaut. Les données analytiques sont conservées.                           |

Pour faire passer le cycle de vie d’un état à l’autre, utilisez le point de terminaison de la Management API [`/api/v2/experimentation/experiments/{id}/status`](/docs/fr-CA/api/management/v2/experimentation/update-experiment-status).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  La transition de `completed → active` n’est pas autorisée. Si vous devez relancer le même test, créez une nouvelle expérience.
</Callout>

<div id="the-is_valid-gate">
  ### Le contrôle `is_valid`
</div>

Avant qu’une expérience puisse être activée, elle doit passer une vérification de l’état de préparation.

Experiment Center stocke un booléen `is_valid` pour chaque expérience et le recalcule à chaque modification. Cela vous donne une rétroaction immédiate pendant la configuration.

Vous pouvez aussi déterminer explicitement l’état de préparation à l’aide du point de terminaison [`/api/v2/experimentation/experiments/{id}/validate`](/docs/fr-CA/api/management/v2/experimentation/validate-experiment) de la Management API.

Cela renvoie `is_valid: true/false` et un tableau `errors` qui répertorie tous les éléments bloquants. La même vérification s’exécute automatiquement lorsque vous tentez d’activer une expérience au moyen du point de terminaison de status.

**Règles de validation vérifiées lors de l’activation :**

* Le feature flag référencé a le status `active`
* Au moins une allocation existe
* Toutes les valeurs `variation_id` des allocations appartiennent au feature flag de l’expérience
* Exactement une allocation a `is_control: true`
* La somme des pondérations des allocations est de 100 (stratégie en pourcentage)
* Exactement une allocation a `is_fallback: true` (stratégie par segment)
* Aucune autre expérience n’est actuellement active pour le tenant (Beta seulement)

<div id="assignment">
  ### Attribution
</div>

**Remplacements par paramètres de requête**

Vous pouvez forcer une attribution précise pour toute requête `/authorize` en transmettant des paramètres de requête :

| Paramètre       | Description                                              |
| --------------- | -------------------------------------------------------- |
| `experiment_id` | Inscrire à une expérience précise                        |
| `variation_id`  | Forcer une variation précise (nécessite `experiment_id`) |
| `segment_id`    | Forcer un segment précis (nécessite `experiment_id`)     |

Cela fonctionne pour les expériences, peu importe leur état, y compris `draft`. Utilisez cette option pendant le développement pour vérifier que les deux variations s’affichent correctement avant l’activation.

<div id="feature-flag">
  ## Drapeau de fonctionnalité
</div>

<div id="feature-flag-lifecycle">
  ### Cycle de vie du Feature flag
</div>

Les Feature flags ont un cycle de vie enregistré comportant trois états :

| **Statut** | **Signification**                                                                                      |
| ---------- | ------------------------------------------------------------------------------------------------------ |
| `draft`    | Créé, mais pas encore actif. Ne peut pas être référencé par une expérience active.                     |
| `active`   | Prêt à être utilisé. Obligatoire avant qu’une expérience qui référence ce flag puisse être activée.    |
| `archived` | État terminal. Aucune nouvelle expérience ne peut référencer ce flag. Créez un nouveau flag au besoin. |

Pour passer d’un statut à l’autre, utilisez le point de terminaison [`/api/v2/experimentation/feature-flags/{id}/status`](/docs/fr-CA/api/management/v2/experimentation/update-feature-flag-status) de l’Auth0 Management API.

**Condition d’activation :** un Feature flag ne peut pas passer à `active` tant qu’il ne comporte pas au moins deux variations. Cela garantit que chaque flag actif possède au minimum un control et un treatment avant sa mise en service.

```text wrap lines
draft → active   (requires ≥2 variations)
draft → archived
active → archived (terminal; no way back)
```

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Il n’est pas possible de passer de `archived → active`. Si vous archivez un flag et que vous devez lancer un autre test, créez un nouveau feature flag.
</Callout>

<div id="variation">
  ### Variante
</div>

**Paramètres et remplacements**

Les paramètres du feature flag définissent les éléments de configuration de votre expérience. Chaque paramètre comprend :

* Un nom (par exemple, `show_passkey_prompt`)
* Un type : `string`, `boolean`, `number`, `array` ou `object`
* Une valeur par défaut (la configuration de base)

Les `overrides` d’une variante indiquent uniquement les paramètres qui diffèrent. À l’exécution, Experiment Center fusionne la configuration de base du flag avec les remplacements de la variante et transmet l’objet `config` complet ainsi fusionné à ACUL, Actions et aux gabarits de page. Chaque paramètre a toujours une valeur dans `config`; vous n’avez donc jamais besoin d’écrire de logique de secours.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Les paramètres utilisent **uniquement le mode structuré**. Chaque paramètre comporte une clé nommée et une valeur typée.
</Callout>

<div id="segment">
  ## Segment
</div>

<div id="segment-rules">
  ### Règles de segment
</div>

Le `rules` d’un segment est un array d’objets rule. Une request correspond au segment si **au moins une** rule du array correspond. Une rule correspond si ses conditions respectent le `match_type` :

* `match_type: "all"` : toutes les conditions doivent être vraies (logique ET)
* `match_type: "any"` : au moins une condition doit être vraie (logique OU)

Chaque condition compare un attribut à une valeur à l’aide d’un opérateur : `equals`, `not_equals`, `in`, `not_in`, `exists`, `not_exists`.

<div id="available-condition-attributes">
  ### Attributs de condition disponibles
</div>

Les segments peuvent uniquement utiliser les attributs disponibles au début de la transaction `/authorize` :

| **Catégorie**                       | **Attributs**                                      |
| ----------------------------------- | -------------------------------------------------- |
| Client                              | `client_id`                                        |
| Connexion                           | `connection`, `connection_type`                    |
| Organisation                        | `organization_id`                                  |
| Domaine                             | `domain`                                           |
| Appareil et navigateur              | `device_type`, `browser`, `platform`, `user_agent` |
| Géographique (tiré de l’adresse IP) | `country`, `region`                                |
