Lorsqu’une expérience est active, Auth0 Experiment Center injecte l’objet `ExperimentContext` dans les déclencheurs d’Action pris en charge.

<Warning>
  Pendant la bêta, Auth0 Experiment Center s’exécute uniquement sur les tenants de développement. Les tenants de production ne sont pas pris en charge.
</Warning>

<div id="supported-triggers">
  ## Déclencheurs pris en charge
</div>

Le contexte de l’expérience est disponible pour les [Auth0 Actions](/docs/fr-CA/customize/actions/actions-overview) suivantes :

* le déclencheur [`post_login`](/docs/fr-CA/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger).
* le déclencheur [`pre_user_registration`](/docs/fr-CA/customize/actions/explore-triggers/signup-and-login-triggers/pre-user-registration-trigger).
* le déclencheur [`post_user_registration`](/docs/fr-CA/customize/actions/explore-triggers/signup-and-login-triggers/post-user-registration-trigger).

<div id="the-eventexperiment-object">
  ## L’objet event.experiment
</div>

Dans les déclencheurs pris en charge, Experiment Center ajoute un champ `experiment` à l’objet event :

```typescript
interface ExperimentContext {
  experiment_id: string;   // L'ID de l'expérience active
  variation_id: string;    // L'ID de la variation attribuée
  config: {                // Configuration fusionnée : configuration de référence + substitutions
    [paramName: string]: { value: unknown };
  };
  is_control: boolean;     // True lorsqu'il s'agit de la variation de contrôle
}
```

Lorsqu’aucune expérience n’est active (ou lorsque la fonctionnalité n’est pas activée pour le tenant), `event.experiment` vaut `null` (et non `undefined`).

<div id="config-object-contains-the-full-merged-configuration">
  ### L&#39;objet `config` contient l&#39;intégralité de la configuration fusionnée
</div>

L&#39;objet `config` contient tous les paramètres définis dans le feature flag, fusionnés avec les remplacements de la variation attribuée. Vous n&#39;avez jamais besoin de rechercher les valeurs de la configuration de référence ni d&#39;écrire une logique de solution de secours. Si le paramètre existe dans le feature flag, il existe dans `config`.

<div id="null-safety-pattern">
  ## Patron de protection contre les valeurs nulles
</div>

Vérifiez qu’une expérience est active avant de lire ses propriétés :

```javascript
exports.onExecutePostLogin = async (event, api) => {
  const ec = event.experiment;
  if (!ec) return; // Aucune expérience active ; rien à faire

  // Accès sécurisé à ec.config, ec.variation_id, ec.is_control ici
};
```

Utilisez ce modèle dans les trois déclencheurs pris en charge. Le retour anticipé permet de garder votre Action claire et garantit qu’elle se comporte correctement pendant les périodes où aucune expérimentation n’est en cours.

<div id="example-post_login-conditional-mfa-policy">
  ## Exemple : post_login — politique MFA conditionnelle
</div>

Cet exemple lit un paramètre booléen et applique une politique MFA différente en fonction de la variation.

```javascript
// Action post_login
exports.onExecutePostLogin = async (event, api) => {
  const ec = event.experiment;
  if (!ec) return;

  const enforceMfa = ec.config?.require_mfa?.value;

  if (enforceMfa === true) {
    // Variation de traitement : appliquer l'AMF pour cet utilisateur
    api.multifactor.enable("any", { allowRememberBrowser: false });
  }
  // Variation de contrôle : aucun changement d'application de l'AMF (comportement de la configuration de référence)
};
```

`ec.config.require_mfa.value` est `true` pour les utilisateurs de la variante de traitement et `false` (la configuration de référence) pour les utilisateurs de la variante témoin. Aucune logique de solution de secours n’est nécessaire.

<div id="example-post_login-set-a-custom-claim-based-on-variation">
  ## Exemple : post_login — définir un claim personnalisé en fonction de la variation
</div>

Cet exemple ajoute au jeton d’ID de l’utilisateur un claim personnalisé qui indique son attribution dans l’expérience. Certains pipelines d’analyse lisent les claims du token plutôt que les journaux du tenant.

```javascript
// Action post_login
exports.onExecutePostLogin = async (event, api) => {
  const ec = event.experiment;
  if (!ec) return;

  // Ajouter l'assignation de l'expérience au jeton d'ID
  api.idToken.setCustomClaim("https://example.com/experiment", {
    experiment_id: ec.experiment_id,
    variation_id: ec.variation_id,
  });
};
```

Utilisez une URL de claim avec espace de noms, conformément à la convention des claims personnalisés d’OIDC. Le destinataire (votre application) peut ensuite lire le claim dans le token sans interroger les journaux du tenant.

<div id="example-pre_user_registration-variation-based-metadata">
  ## Exemple pre_user_registration — métadonnées selon la variation
</div>

Cet exemple utilise une expérience sur le flux d’inscription pour définir `user_metadata` en fonction de la variante dans laquelle tombe l’utilisateur qui s’inscrit.

```javascript
// Action pre_user_registration
exports.onExecutePreUserRegistration = async (event, api) => {
  const ec = event.experiment;
  if (!ec) return;

  const onboardingVariant = ec.config?.onboarding_variant?.value;

  if (onboardingVariant) {
    api.user.setUserMetadata("onboarding_variant", onboardingVariant);
    api.user.setUserMetadata("onboarding_experiment_id", ec.experiment_id);
  }
};
```

Les métadonnées sont enregistrées lors de l’inscription, de sorte qu’elles suivent l’utilisateur d’une session à l’autre. Vous pouvez ensuite les interroger pour savoir à quelle cohorte d’inscription appartient l’utilisateur.

<div id="example-post_user_registration-trigger-downstream-enrollment">
  ## Exemple : post_user_registration — déclencher l’enrôlement en aval
</div>

Cet exemple déclenche un webhook après l’inscription en fonction de la variation attribuée au nouvel utilisateur.

```javascript
// Action post_user_registration
exports.onExecutePostUserRegistration = async (event, api) => {
  const ec = event.experiment;
  if (!ec) return;

  const enrollInNewProgram = ec.config?.enroll_welcome_program?.value;

  if (enrollInNewProgram === true) {
    await fetch("https://your-backend.example.com/api/enrollment", {
      method: "POST",
      headers: { "Content-Type": "application/json" },
      body: JSON.stringify({
        user_id: event.user.user_id,
        email: event.user.email,
        experiment_id: ec.experiment_id,
        variation_id: ec.variation_id,
      }),
    });
  }
};
```

Seuls les utilisateurs de la variante testée déclenchent le webhook d’enrôlement. Les utilisateurs témoins suivent le parcours standard après l’inscription.

<div id="use-is_control">
  ## Utiliser is_control
</div>

Le paramètre `is_control` vaut `true` lorsque l’utilisateur appartient au groupe témoin (il a reçu la configuration de référence, sans aucun remplacement appliqué). Utilisez-le lorsque vous devez savoir quels utilisateurs ont vu l’expérience non modifiée, ou lorsque vous voulez omettre un traitement facultatif pour les utilisateurs du groupe témoin.

```javascript
exports.onExecutePostLogin = async (event, api) => {
  const ec = event.experiment;
  if (!ec) return;

  if (!ec.is_control) {
    // Suivre uniquement les utilisateurs du traitement dans votre système d'analytique
    // (les utilisateurs du contrôle sont suivis séparément, en dehors de l'Action)
    await logTreatmentEvent(ec.experiment_id, ec.variation_id, event.user.user_id);
  }
};
```

Pour moduler le comportement (et non l’analytique), vérifiez directement les valeurs du paramètre `config`. Elles sont plus explicites et plus faciles à comprendre.
