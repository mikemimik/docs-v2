Lorsqu’une expérience est active, Experiment Center injecte l’objet `ExperimentContext` dans les déclencheurs d’Action pris en charge.

<Warning>
  Pendant la bêta, Experiment Center fonctionne uniquement sur les tenants de développement. Les tenants de production ne sont pas pris en charge.
</Warning>

<div id="supported-triggers">
  ## Déclencheurs pris en charge
</div>

Le contexte d’Experiment est disponible pour les [Actions](/docs/fr-CA/customize/actions/actions-overview) Auth0 suivantes :

* le déclencheur [`post_login`](/docs/fr-CA/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger).
* le déclencheur [`pre_user_registration`](/docs/fr-CA/customize/actions/explore-triggers/signup-and-login-triggers/pre-user-registration-trigger).
* le déclencheur [`post_user_registration`](/docs/fr-CA/customize/actions/explore-triggers/signup-and-login-triggers/post-user-registration-trigger).

<div id="the-eventexperiment-object">
  ## L’objet event.experiment
</div>

Pour les déclencheurs pris en charge, Experiment Center ajoute un champ `experiment` à l’objet d’événement :

```typescript
interface ExperimentContext {
  experiment_id: string;   // L'ID de l'expérience active
  variation_id: string;    // L'ID de variation assignée
  config: {                // Configuration fusionnée : référence + overrides
    [paramName: string]: { value: unknown };
  };
  is_control: boolean;     // Vrai lorsqu'il s'agit de la variation de contrôle
}
```

Lorsqu’aucune expérience n’est active (ou que la fonctionnalité n’est pas activée pour le tenant), `event.experiment` vaut `null` (et non `undefined`).

<div id="config-object-contains-the-full-merged-configuration">
  ### l’objet config contient l’ensemble de la configuration fusionnée
</div>

L’objet `config` contient tous les paramètres définis dans le feature flag, fusionnés avec les overrides de la variation attribuée. Vous n’avez jamais besoin de rechercher les valeurs de référence ni d’écrire une logique de repli. Si le paramètre existe dans le feature flag, il existe dans `config`.

<div id="null-safety-pattern">
  ## Modèle de sécurité des valeurs nulles
</div>

Vérifiez qu’une expérience est active avant de lire ses propriétés :

```javascript
exports.onExecutePostLogin = async (event, api) => {
  const ec = event.experiment;
  if (!ec) return; // Aucune expérience active ; rien à faire

  // Accès sûr à ec.config, ec.variation_id, ec.is_control ici
};
```

Utilisez ce modèle dans les trois déclencheurs pris en charge. Le retour anticipé permet de garder votre Action claire et garantit qu’elle se comporte correctement pendant les périodes où aucune expérience n’est en cours.

<div id="example-post_login-conditional-mfa-policy">
  ## Exemple : post_login — politique MFA conditionnelle
</div>

Cet exemple lit un paramètre booléen et applique une politique MFA différente en fonction de la variation.

```javascript
// Action post_login
exports.onExecutePostLogin = async (event, api) => {
  const ec = event.experiment;
  if (!ec) return;

  const enforceMfa = ec.config?.require_mfa?.value;

  if (enforceMfa === true) {
    // Variation de traitement : appliquer le MFA pour cet utilisateur
    api.multifactor.enable("any", { allowRememberBrowser: false });
  }
  // Variation de contrôle : aucun changement d'application du MFA (comportement de référence)
};
```

`ec.config.require_mfa.value` est `true` pour les utilisateurs de la variation de traitement et `false` (la référence) pour les utilisateurs de la variation témoin. Aucune logique de repli n’est nécessaire.

<div id="example-post_login-set-a-custom-claim-based-on-variation">
  ## Exemple : post_login — définir un claim personnalisé selon la variation
</div>

Cet exemple inscrit l’assignation à l’expérience dans le jeton d’identité de l’utilisateur sous forme de claim personnalisé. Certains pipelines d’analyse lisent les claims du token plutôt que les logs du tenant.

```javascript
// Action post_login
exports.onExecutePostLogin = async (event, api) => {
  const ec = event.experiment;
  if (!ec) return;

  // Ajouter l'assignation de l'expérience au jeton d’identité
  api.idToken.setCustomClaim("https://example.com/experiment", {
    experiment_id: ec.experiment_id,
    variation_id: ec.variation_id,
  });
};
```

Utilisez une URL de claim avec espace de noms, conformément à la convention OIDC relative aux claims personnalisés. Le destinataire (votre application) peut ensuite lire le claim dans le token sans avoir à consulter les journaux du tenant.

<div id="example-pre_user_registration-variation-based-metadata">
  ## Exemple pre_user_registration — métadonnées fondées sur une variation
</div>

Cet exemple utilise une expérience sur le flux d’inscription pour définir `user_metadata` en fonction de la variation dans laquelle tombe l’utilisateur au moment de son inscription.

```javascript
// Action pre_user_registration
exports.onExecutePreUserRegistration = async (event, api) => {
  const ec = event.experiment;
  if (!ec) return;

  const onboardingVariant = ec.config?.onboarding_variant?.value;

  if (onboardingVariant) {
    api.user.setUserMetadata("onboarding_variant", onboardingVariant);
    api.user.setUserMetadata("onboarding_experiment_id", ec.experiment_id);
  }
};
```

Les métadonnées sont enregistrées au moment de l’inscription; elles suivent donc l’utilisateur d’une session à l’autre. Vous pourrez ensuite les interroger pour déterminer à quelle cohorte d’inscription appartient un utilisateur.

<div id="example-post_user_registration-trigger-downstream-enrollment">
  ## Exemple : post_user_registration — déclencher l’inscription en aval
</div>

Selon la variation attribuée au nouvel utilisateur, cet exemple déclenche un webhook après l’inscription.

```javascript
// Action post_user_registration
exports.onExecutePostUserRegistration = async (event, api) => {
  const ec = event.experiment;
  if (!ec) return;

  const enrollInNewProgram = ec.config?.enroll_welcome_program?.value;

  if (enrollInNewProgram === true) {
    await fetch("https://your-backend.example.com/api/enrollment", {
      method: "POST",
      headers: { "Content-Type": "application/json" },
      body: JSON.stringify({
        user_id: event.user.user_id,
        email: event.user.email,
        experiment_id: ec.experiment_id,
        variation_id: ec.variation_id,
      }),
    });
  }
};
```

Seuls les utilisateurs de la variation de test déclenchent le webhook d’inscription. Les utilisateurs du groupe témoin suivent le parcours standard après l’inscription.

<div id="use-is_control">
  ## Utiliser is_control
</div>

Le paramètre `is_control` vaut `true` lorsque l’utilisateur appartient au groupe témoin (il a reçu la configuration de référence, sans aucune substitution appliquée). Utilisez-le lorsque vous devez savoir quels utilisateurs ont vu l’expérience non modifiée, ou lorsque vous voulez ignorer un traitement facultatif pour les utilisateurs du groupe témoin.

```javascript
exports.onExecutePostLogin = async (event, api) => {
  const ec = event.experiment;
  if (!ec) return;

  if (!ec.is_control) {
    // Suivre uniquement les utilisateurs du traitement dans votre système d'analytique
    // (les utilisateurs du contrôle sont suivis séparément, en dehors de l'Action)
    await logTreatmentEvent(ec.experiment_id, ec.variation_id, event.user.user_id);
  }
};
```

Pour gérer le comportement de façon conditionnelle (et non pour l’analyse), vérifiez directement les valeurs du paramètre `config`. Elles sont plus explicites et plus lisibles.
