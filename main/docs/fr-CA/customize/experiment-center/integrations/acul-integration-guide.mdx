Lorsqu’une expérience est active et qu’une variation est attribuée, Experiment Center injecte un objet `ExperimentContext` dans votre composant [ACUL](/docs/fr-CA/customize/login-pages/advanced-customizations) sous la forme d’un `experiment`.

<Warning>
  Pendant la bêta, Experiment Center fonctionne uniquement sur les tenants de développement. Les tenants de production ne sont pas pris en charge.
</Warning>

L’injection se produit uniquement sur les écrans pour lesquels vous avez explicitement activé cette option. Vous l’activez écran par écran en ajoutant `"experiment"` au tableau `context_configuration` de cet écran.

Pour activer un écran afin qu’il reçoive le contexte de l’expérience, effectuez un appel `PATCH` au point de terminaison [`/api/v2/prompts/{prompt}/screen/{screen}/rendering`](/docs/fr-CA/api/management/v2/prompts/patch-rendering).

```json Example
    "context_configuration": ["experiment"]
```

Remplacez `{prompt}` par le nom du prompt (par exemple, `login`) et `{screen}` par le nom de l’écran (par exemple, `login`).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  La résolution de l’expérience et l’enrichissement du journal du tenant s’exécutent toujours, qu’un écran soit activé explicitement ou non. L’activation explicite sert uniquement à déterminer si les propriétés `experiment` sont transmises à votre composant ACUL. Il s’agit d’une mesure de minimisation des données : n’activez pas explicitement les écrans qui n’ont pas besoin du contexte de l’expérience.
</Callout>

<div id="the-experiment-context-shape">
  ## La structure du contexte de l’expérience
</div>

Lorsqu’un écran a été explicitement activé et qu’une expérience est en cours, vous pouvez accéder au contexte de l’expérience par l’intermédiaire de `window.universal_login_context.experiment`.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Durant Experiment Center (Beta), le SDK ACUL n’ajoute pas automatiquement la propriété `experiment`; elle doit être définie au moyen de `window.universal_login_context.experiment`.
</Callout>

```typescript
const experiment = window.universal_login_context.experiment;
```

Le composant ACUL reçoit un contexte `experiment` :

```typescript
interface ExperimentContext {
  experiment_id: string;   // L'ID d'expérience actif
  variation_id: string;    // L'ID de variation attribué
  config: {                // Configuration fusionnée : référence + overrides
    [paramName: string]: { value: unknown };
  };
  is_control: boolean;     // True lorsqu'il s'agit de la variation témoin
}
```

Lorsqu’aucune expérience n’est active (ou lorsque la fonctionnalité n’est pas activée pour le tenant), `experiment` vaut `null`.

Le paramètre `config` contient la configuration complète fusionnée pour la variation attribuée. Experiment Center prend les paramètres de base de l’indicateur de fonctionnalité et y applique les remplacements de la variation attribuée. Chaque paramètre défini dans l’indicateur de fonctionnalité a toujours une valeur dans `config`.

Par exemple, si votre indicateur de fonctionnalité comprend un paramètre `button_label` dont la valeur de référence est `"Sign in"` et que la variation attribuée la remplace par `"Continue"`, alors `config.button_label.value` vaut `"Continue"`.
Pour la variation témoin (sans remplacement), `config.button_label.value` vaut `"Sign in"`.

<div id="read-a-parameter-value">
  ## Lire la valeur d’un paramètre
</div>

Vous pouvez accéder à la valeur d’un paramètre au moyen de `config[paramName].value` :

```typescript
const experiment = window.universal_login_context.experiment;
const label = experiment?.config?.button_label?.value;
```

Utilisez l’enchaînement optionnel (`?.`) partout. La prop `experiment` est `undefined` lorsqu’aucune expérience n’est en cours.

<div id="use-is_control">
  ## Utiliser is_control
</div>

Le paramètre `is_control` vaut `true` lorsque l’utilisateur appartient au groupe témoin (il a reçu la configuration de référence, sans aucun remplacement appliqué). Utilisez-le lorsque vous devez repérer quels utilisateurs ont vu l’expérience non modifiée, ou lorsque vous voulez éviter le traitement facultatif pour les utilisateurs du groupe témoin.

```typescript
if (!experiment?.is_control) {
  // S'exécute uniquement pour les utilisateurs du traitement
  trackExperimentImpression(experiment.experiment_id, experiment.variation_id);
}
```

Pour le rendu par embranchements, vérifiez directement la valeur du paramètre (`config.my_param.value`) plutôt que `is_control`.
Les vérifications fondées sur les paramètres sont plus lisibles et continuent de fonctionner correctement même si vous changez, dans une expérience future, la variation servant de témoin statistique.

<div id="example-copy-variant-experiment">
  ## Exemple : expérience sur des variantes de libellé
</div>

Cet exemple montre un indicateur de fonctionnalité avec deux variations qui modifie le libellé d’un bouton. Le témoin utilise le libellé standard; le traitement utilise une variante.

**Paramètres du indicateur de fonctionnalité :**

```json
{
  "button_label": {
    "type": "string",
    "value": "Sign in",
    "description": "Label for the primary login button"
  }
}
```

**Variante témoin :** aucune surcharge (hérite de `button_label: "Sign in"`)

**Variante de traitement :**

```json
{
  "overrides": {
    "button_label": { "value": "Continue to your account" }
  }
}
```

**Composant ACUL qui lit le paramètre :**

```typescript
// LoginScreen.tsx
export default function LoginScreen({...props }) {
  // config a toujours une valeur — aucune solution de secours nécessaire
  const experiment = window.universal_login_context.experiment;
  const buttonLabel = experiment?.config?.button_label?.value ?? "Sign in";

  return (
    <div>
      <h1>Welcome back</h1>
      <form onSubmit={props.onSubmit}>
        <input type="email" name="email" placeholder="Email" />
        <input type="password" name="password" placeholder="Password" />
        <button type="submit">{buttonLabel}</button>
      </form>
    </div>
  );
}
```

La solution de secours `?? "Sign in"` à la dernière ligne couvre le cas où aucune expérience n’est active (auquel cas `experiment` vaut `undefined` et `config?.button_label?.value` donne aussi `undefined`). Si vous préférez, vous pouvez utiliser une vérification distincte de nullité :

```typescript
const buttonLabel = experiment
  ? experiment.config.button_label.value
  : "Sign in"; // Aucune expérience active ; utiliser la valeur par défaut
```

<div id="example-boolean-feature-rollout">
  ## Exemple : déploiement d’une fonctionnalité contrôlé par un booléen
</div>

Cet exemple utilise un paramètre booléen pour afficher un nouvel élément de l’interface utilisateur de façon conditionnelle.

```typescript
// PostLoginScreen.tsx
export default function PostLoginScreen({...props }) {
  const experiment = window.universal_login_context.experiment;
  const showPasskeyBanner = experiment?.config?.show_passkey_banner?.value === true;

  return (
    <div>
      <p>You're signed in.</p>
      {showPasskeyBanner && (
        <div className="banner">
          <p>Set up a passkey for faster sign-in next time.</p>
          <button onClick={props.onEnrollPasskey}>Set up passkey</button>
          <button onClick={props.onDismiss}>Not now</button>
        </div>
      )}
    </div>
  );
}
```

La vérification `=== true` (plutôt qu’un test « truthy ») est intentionnelle : elle garantit que la bannière s’affiche uniquement lorsque le paramètre vaut explicitement `true`, et non lorsque `experiment` est `undefined` ou que `config` est absent.

<div id="troubleshoot">
  ## Dépannage
</div>

La propriété `experiment` est `undefined` dans trois situations :

1. Aucune expérience n’est actuellement active pour le tenant
2. L’écran n’a pas été explicitement activé via `context_configuration`
3. Experiment Center n’est pas activé pour le tenant

Traitez toujours `experiment` comme une valeur potentiellement `undefined`. La méthode la plus sûre :

```typescript
const experiment = window.universal_login_context.experiment;
const myParam = experiment?.config?.my_param?.value;
// myParam est indéfini lorsqu'aucune expérience n'est active
// Utilisez une valeur par défaut : myParam ?? "your-default"
```

Ne supposez jamais que la propriété `experiment` est présente. Le code qui suppose que `experiment` est définie générera des erreurs lorsqu’aucune expérience n’est en cours, ce qui est le cas la plupart du temps.
