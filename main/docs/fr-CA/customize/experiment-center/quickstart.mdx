  Durant la bêta, Experiment Center fonctionne uniquement sur des tenants de développement. Les tenants de production ne sont pas pris en charge.
</Warning>

<div id="prerequisites">
  ## Prérequis
</div>

Pour commencer à utiliser Experiment Center, vous avez besoin de :

* **D’un tenant Auth0 [de développement](/docs/fr-CA/get-started/tenant-settings#environment-tag)**

* **D’une application [Machine-to-Machine](/docs/fr-CA/get-started/apis/create-m2m-app-test#create-machine-to-machine-applications-for-testing)** dotée des scopes Management API suivants :

  ```text wrap lines
  read:experimentation
  create:experimentation
  update:experimentation
  delete:experimentation
  ```

<div id="1-create-and-activate-the-feature-flag">
  ## 1. Créer et activer l’indicateur de fonctionnalité
</div>

Un indicateur de fonctionnalité définit ce que vous testez et les variations possibles.

Pour créer un indicateur de fonctionnalité, effectuez un appel `POST` vers le point de terminaison [`/api/v2/experimentation/feature-flags`](/docs/fr-CA/api/management/v2/experimentation/create-feature-flag).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  * La réponse comprend une valeur `feature_flag_id`; vous en aurez besoin dans les étapes suivantes.
  * L’indicateur de fonctionnalité est initialement au statut `draft`.
</Callout>

<div id="add-two-variations">
  ### Ajouter deux variations
</div>

Le indicateur de fonctionnalité doit comporter au moins deux variations avant de pouvoir être activé : une variation témoin et une variation de traitement.

<div id="create-the-control-variation">
  #### Créer la variation témoin
</div>

La variation témoin ne contient aucun remplacement. Elle représente la référence inchangée.
Pour ajouter une variation témoin, effectuez un appel `POST` au point de terminaison [`/api/v2/experimentation/feature-flags/{feature_flag_id}/variations`](/docs/fr-CA/api/management/v2/experimentation/create-variation).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  La réponse comprend une valeur `variation_id` pour le témoin. Vous en aurez besoin pour configurer les allocations.
</Callout>

<div id="create-the-treatment-variation">
  #### Créez la variation de traitement
</div>

La variation de traitement remplace les paramètres que vous voulez modifier.

Pour ajouter la variation de traitement, effectuez un appel `POST` au point de terminaison [`/api/v2/experimentation/feature-flags/{feature_flag_id}/variations`](/docs/fr-CA/api/management/v2/experimentation/create-variation) et configurez l’objet `overrides`.

```json Example
    "overrides": {
      "show_passkey_prompt": { "value": true },
      "prompt_style": { "value": "modal" }
    }
```

<div id="activate-the-feature-flag">
  ### Activer le indicateur de fonctionnalité
</div>

Maintenant que vous avez deux variations, activez le indicateur de fonctionnalité. Une expérience ne peut pas être activée à moins que le indicateur de fonctionnalité auquel elle se rattache ait le statut `active`.

Pour faire passer le statut du indicateur de fonctionnalité à `active`, envoyez une requête `POST` au point de terminaison [`/api/v2/experimentation/feature-flags/{feature_flag_id}/status`](/docs/fr-CA/api/management/v2/experimentation/update-feature-flag-status).

<div id="2-create-a-segment-optional">
  ## 2. Créer un segment (facultatif)
</div>

Si vous voulez cibler un trafic spécifique, créez un segment. Ignorez cette étape si vous voulez simplement répartir l’ensemble du trafic selon un pourcentage.

Pour créer un segment, effectuez une requête `POST` vers le point de terminaison [`/api/v2/experimentation/segments`](https://auth0.com/docs/api/management/v2/experimentation/create-segment).

Cet exemple crée un segment correspondant aux utilisateurs mobiles aux États-Unis :

```json Example
    "name": "mobile-us-users",
    "description": "Mobile users from the United States",
    "rules": [
      {
        "match_type": "all",
        "conditions": [
          { "attribute": "device_type", "operator": "equals", "value": "mobile" },
          { "attribute": "country", "operator": "equals", "value": "US" }
        ]
      }
    ]
```

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  La réponse contient une valeur `segment_id`. Les segments s’appliquent uniquement lorsque vous utilisez la [stratégie d’allocation](/docs/fr-CA/api/management/v2/experimentation/create-experiment#body-allocation-strategy) `segment` dans l’expérience.
</Callout>

<div id="3-create-and-activate-an-experiment">
  ## 3. Créer et activer une expérience
</div>

Créez une expérience qui s’appuie sur votre indicateur de fonctionnalité et définit comment répartir le trafic.

Pour créer une expérience, envoyez une requête `POST` au point de terminaison [`/api/v2/experimentation/experiments`](/docs/fr-CA/api/management/v2/experimentation/create-experiment#body-allocations).

Cet exemple utilise une répartition 90/10 au moyen de l’objet `allocations` : 90 % des utilisateurs sont affectés au groupe témoin, et 10 % au traitement. Il s’agit d’un point de départ typique pour un déploiement prudent.

```json Example
    "allocations": [
      {
        "variation_id": "var_Id",
        "weight": 90,
        "is_control": true
      },
      {
        "variation_id": "var_Id",
        "weight": 10,
        "is_control": false
      }
    ]

```

L’expérience débute avec le statut `draft` et `is_valid: false`. C’est normal : la vérification complète de l’état de préparation s’effectue à l’étape suivante.

<div id="validate-the-experiment">
  ### Valider l’expérience
</div>

Avant de l’activer, assurez-vous que l’expérience est prête. Le point de terminaison `/validate` renvoie les mêmes vérifications que celles effectuées à l’activation.

Pour confirmer l’expérience, effectuez un appel `POST` vers le point de terminaison [`/api/v2/experimentation/experiments/{experiment_id}/validate`](/docs/fr-CA/api/management/v2/experimentation/validate-experiment).

La réponse inclut la valeur `is_valid`, qui indique si vous pouvez l’activer.

<div id="activate-the-experiment">
  ### Activer l’expérience
</div>

Lorsque l’expérience est valide et que vous avez testé les deux variations, activez-la.

Pour faire passer le statut de l’expérience à `active`, envoyez une requête `POST` au point de terminaison [`/api/v2/experimentation/experiments/{experiment_id}/status`](/docs/fr-CA/api/management/v2/experimentation/update-experiment-status).

La valeur `started_at` de l’expérience est définie lors de la première activation et ne change pas si vous la mettez en pause puis l’activez de nouveau.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  **Une seule expérience active par tenant.** Si une autre expérience est déjà active, l’activation renvoie `400 experiment_active_limit_exceeded`. Mettez d’abord l’autre expérience en pause ou terminez-la.
</Callout>

<div id="4-trigger-an-auth-event-and-observe-results">
  ## 4. Déclenchez un événement d’authentification et observez les résultats
</div>

Déclenchez une ouverture de session de test dans votre tenant de test. Par exemple, ouvrez un navigateur et accédez à votre URL `/authorize` sans paramètre de surcharge.

Terminez le flow de connexion. Experiment Center détermine l’expérience active, attribue une variation au moyen d’un hachage déterministe, injecte le contexte de l’expérience et enrichit l’événement d’authentification qui en résulte.

<div id="verify-enriched-logs">
  ### Vérifier les journaux enrichis
</div>

Vous pouvez consulter les journaux Auth0 pour les événements d’expérience :

* Accédez à Auth0 [**Dashboard &gt; Monitoring &gt; Logs**](https://manage.auth0.com/dashboard/#/logs).

* Utilisez la diffusion des journaux : si vous diffusez déjà les journaux du tenant vers un outil d’analytique (Datadog, Splunk, Segment, etc.), les métadonnées d’expérience enrichies transitent automatiquement par le même flux.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  L’événement comprend des métadonnées d’expérience dans l’objet `details.experiment`. Le champ `variation_id` indique à quelle variation l’utilisateur a été attribué.
</Callout>

<Check>
  Vous avez exécuté votre première expérience avec succès lorsque :

  1. Vos événements d’authentification dans les journaux du tenant incluent `details.experiment` avec `experiment_id` et `variation_id`
  2. Les deux variations produisent le comportement attendu lorsqu’elles sont testées avec des substitutions de paramètres de requête
  3. Le même appareil ou utilisateur obtient systématiquement le même `variation_id` dans plusieurs événements d’authentification (attribution déterministe)
</Check>
