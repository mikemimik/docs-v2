<Warning>
  La date de fin de vie (EOL) de règles et Hooks est fixée au **18 novembre 2026**, et ils ne sont plus offerts aux nouveaux tenants créés à compter du **16 octobre 2023**. Les tenants existants ayant des Hooks actifs conserveront l’accès au produit Hooks jusqu’à la fin de vie.

  Nous vous recommandons fortement d’utiliser Actions pour étendre Auth0. Avec Actions, vous avez accès à des informations de type détaillées, à de la documentation intégrée et à des packages `npm` publics, et vous pouvez connecter des intégrations externes qui enrichissent votre expérience globale en matière d’extensibilité. Pour en savoir plus sur ce qu’offre Actions, consultez [Comprendre le fonctionnement d’Auth0 Actions](/docs/fr-CA/customize/actions/actions-overview).

  Pour vous aider dans votre migration, nous mettons à votre disposition des guides qui vous aideront à [migrer de règles vers Actions](/docs/fr-CA/customize/actions/migrate/migrate-from-rules-to-actions) et à [migrer de Hooks vers Actions](/docs/fr-CA/customize/actions/migrate/migrate-from-hooks-to-actions). Nous avons également une page dédiée, [Passer à Actions](https://auth0.com/platform/extensibility/movetoactions), qui présente des comparaisons de fonctionnalités, [une démonstration d’Actions](https://www.youtube.com/watch?v=UesFSY1klrI) et d’autres ressources pour vous accompagner dans votre migration.

  Pour en savoir plus sur la dépréciation de règles et Hooks, consultez notre article de blogue : [Preparing for Rules and Hooks End of Life](https://auth0.com/blog/preparing-for-rules-and-hooks-end-of-life/).
</Warning>

<Warning>
  Puisque nous prévoyons retirer les fonctions règles et Hooks en 2026, vous ne devriez créer de nouvelles règles ou de nouveaux Hooks que dans votre environnement de développement, et uniquement pour tester la migration vers Actions.

  Pour savoir comment migrer vos règles vers Actions, consultez [Migrer de règles vers Actions](/docs/fr-CA/customize/actions/migrate/migrate-from-rules-to-actions). Pour savoir comment migrer vos Hooks vers Actions, consultez [Migrer de Hooks vers Actions](/docs/fr-CA/customize/actions/migrate/migrate-from-hooks-to-actions).

Vous pouvez créer vos propres règles pour répondre à vos besoins fonctionnels précis. Vous pouvez modifier un modèle de règle existant ou choisir de partir de zéro à l’aide de l’un de nos exemples. Auth0 fournit plusieurs règles et modèles de règle pour vous aider à atteindre vos objectifs. Pour en voir la liste, consultez notre [dépôt de règles sur GitHub](https://github.com/auth0/rules).

<div id="how-rules-work">
  ## Comment fonctionnent les Rules
</div>

Les Rules sont des fonctions JavaScript qui s’exécutent lorsqu’un utilisateur s’authentifie auprès de votre application. Elles s’exécutent une fois le processus d’authentification terminé, et vous pouvez les utiliser pour personnaliser et étendre les fonctionnalités d’Auth0. Pour des raisons de sécurité, le code de vos Rules s’exécute de manière isolée du code des autres tenants Auth0 dans un sandbox. Les Rules s’exécutent aussi pendant le flux d’actualisation du jeton. Pour en savoir plus, consultez [Refresh Tokens](/docs/fr-CA/secure/tokens/refresh-tokens).

Dans Auth0, le flux de transaction d’authentification fonctionne comme suit lorsque vous utilisez des Rules :

<Frame>![Schéma des Rules dans le flux d’authentification](/docs/images/cdy7uua7fh8z/2gtBtkPChIyguA24x6enx2/ffbb8e21e86920ef9914f6fc126dc1df/flow.png)</Frame>

1. Une application lance une demande d’authentification à Auth0.
2. Auth0 achemine la demande vers un fournisseur d’identité au moyen d’une connexion configurée.
3. L’utilisateur s’authentifie avec succès.
4. Le jeton d’ID et/ou le jeton d’accès passe par le pipeline des Rules, puis est envoyé à l’application.

<div id="use-the-management-api">
  ## Utiliser la Management API
</div>

Faites un appel `POST` au [point de terminaison Create Rule](https://auth0.com/docs/api/management/v2#!/Rules/post_rules). Assurez-vous de remplacer les valeurs de substitution `MGMT_API_ACCESS_TOKEN`, `RULE_NAME`, `RULE_SCRIPT`, `RULE_ORDER` et `RULE_ENABLED` par votre <Tooltip tip="Management API : un produit qui permet aux clients d’effectuer des tâches administratives." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Management+API">Management API</Tooltip> <Tooltip tip="Management API : un produit qui permet aux clients d’effectuer des tâches administratives." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Access+Token">jeton d’accès</Tooltip>, le nom de la règle, le script de la règle, le numéro d’ordre de la règle et la valeur d’activation de la règle, respectivement.

| Valeur                      | Description                                                                                                                                                                                                                                                             |
| --------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `MGMT_API_ACCESS_TOKEN`     | Jeton d’accès pour la Management API avec la portée `create:rules`.                                                                                                                                                                                                     |
| `RULE_NAME`                 | Nom de la règle que vous souhaitez créer. Le nom de la règle ne peut contenir que des caractères alphanumériques, des espaces et des traits d’union; il ne peut pas commencer ni se terminer par des espaces ou des traits d’union.                                     |
| `RULE_SCRIPT`               | Script contenant le code de la règle. Il doit correspondre à ce que vous saisiriez si vous créiez une nouvelle règle dans le Dashboard.                                                                                                                                 |
| `RULE_ORDER` (facultatif)   | Entier représentant l’ordre dans lequel la règle doit être exécutée par rapport aux autres règles. Les règles ayant un numéro inférieur sont exécutées avant celles ayant un numéro supérieur. Si aucun numéro d’ordre n’est fourni, la règle sera exécutée en dernier. |
| `RULE_ENABLED` (facultatif) | Booléen indiquant si la règle est activée (`true`) ou désactivée (`false`).                                                                                                                                                                                             |

<Warning>
  Nous exposons des adresses IPv6 dans nos points de terminaison publics (p. ex., `travel0.us.auth0.com`). Si une requête provient d’une machine compatible avec IPv6, la propriété `context.request.ip` contiendra une adresse IPv6. Si vous effectuez une manipulation manuelle des adresses IP, nous vous suggérons d’utiliser la [bibliothèque ipaddr.js@1.9.0](https://www.npmjs.com/package/ipaddr.js/v/1.9.0).
</Warning>

<div id="manage-rate-limits">
  ## Gérer les limites de débit
</div>

Pour les règles qui appellent des API Auth0, vous devriez toujours gérer la limitation du débit en vérifiant l’en-tête `X-RateLimit-Remaining` et en réagissant de façon appropriée lorsque le nombre renvoyé approche de 0. Vous devriez aussi ajouter une logique pour gérer les cas où vous dépassez les limites de débit fournies et recevez le code d’état HTTP 429 (Trop de requêtes) ; dans ce cas, si une nouvelle tentative est nécessaire, il est préférable de prévoir un délai d’attente progressif afin d’éviter une boucle infinie de nouvelles tentatives. Pour en savoir plus sur les limites de débit, consultez [la politique de limite de débit pour les API Auth0](/docs/fr-CA/troubleshoot/customer-support/operational-policies/rate-limit-policy).

<div id="available-modules">
  ## Modules disponibles
</div>

Rules s’exécutent dans un sandbox JavaScript configuré pour une version précise de Node.js.

Le sandbox prend en charge toutes les versions du langage JavaScript (et la syntaxe associée) compatibles avec la version de Node.js configurée, ainsi qu’un grand nombre de modules Node.js. Pour obtenir la liste des modules sandbox pris en charge, consultez [Can I require: Auth0 Extensibility](https://auth0-extensions.github.io/canirequire/).
