La page de connexion Auth0 ne peut pas afficher un message d’erreur provenant d’une Rule. Si vous voulez que les utilisateurs voient un message personnalisé généré par une Rule, votre application doit l’afficher.

Toute erreur provenant d’une Rule est envoyée à votre URL de rappel dans le mode de réponse standard du protocole. Par exemple, si votre application demande une authentification au moyen d’OIDC/OAuth2, elle reçoit une erreur OAuth2 standard avec les valeurs de résultat `error` et `error_description`.

Supposons que votre URL de rappel soit https://travel0.com/callback et que vous utilisiez le mode de réponse par requête standard. Vous recevrez cette réponse :

```http wrap lines
https://travel0.com/callback?error=access_denied&error_description=[The message you wrote in the rule]
```

C’est à votre application d’afficher ce message à l’utilisateur, par exemple :

<div id="retry-login-after-a-rule-error">
  ## Réessayer la connexion après une erreur de rule
</div>

L’étape suivante la plus probable consiste à envoyer une nouvelle request d’authorization. Cependant, dans la plupart des cas, Auth0 sait déjà qui est l’utilisateur et ne lui redemande donc pas ses informations de login. Les rules s’exécutent alors immédiatement, et la rule refuse de nouveau l’authorization. Cela peut créer une boucle sans fin.

L’utilisateur doit avoir la possibilité de fournir de nouveaux credentials. Votre application peut faire réafficher la login page de deux façons :

* Ajoutez une directive à la request pour forcer l’affichage de la login page :

  * Utilisez un paramètre `prompt=login` dans une request OIDC/OAuth2.
  * Utilisez un attribut `ForceAuthn="true"` dans une SAML request.

—or—

* Déconnectez l’utilisateur d’Auth0 avant d’envoyer une nouvelle request.
