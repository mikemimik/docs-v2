La page de connexion Auth0 ne peut pas afficher un message d’erreur provenant d’une Rule. Si vous voulez que les utilisateurs voient un message personnalisé généré par une Rule, votre application doit afficher ce message.

Toute erreur produite par une Rule est envoyée à votre URL de rappel dans le mode de réponse standard du protocole. Par exemple, si votre application demande une authentification au moyen d’OIDC/OAuth2, elle reçoit une erreur OAuth2 standard avec les valeurs de résultat `error` et `error_description`.

Supposons que votre URL de rappel soit https://travel0.com/callback et que vous utilisiez le mode de réponse standard par chaîne de requête. Vous recevriez cette réponse :

```http wrap lines
https://travel0.com/callback?error=access_denied&error_description=[The message you wrote in the rule]
```

Il revient à votre application d’afficher le message à l’utilisateur, par exemple :

<div id="retry-login-after-a-rule-error">
  ## Réessayer la connexion après une erreur de Rule
</div>

L’étape suivante la plus probable consiste à demander une nouvelle autorisation. Cependant, dans la plupart des cas, Auth0 sait déjà qui est l’utilisateur et ne lui redemande donc pas ses identifiants de connexion. Au lieu de cela, les rules s’exécutent immédiatement et la rule refuse de nouveau l’autorisation. Cela peut créer une boucle sans fin.

L’utilisateur doit avoir la possibilité de fournir de nouveaux identifiants. Votre application peut forcer de nouveau l’affichage de la page de connexion de deux façons :

* Ajoutez une directive dans la requête pour forcer l’affichage de la page de connexion :

  * Utilisez un paramètre `prompt=login` dans une requête OIDC/OAuth2.
  * Utilisez un attribut `ForceAuthn="true"` dans une requête SAML.

—ou—

* Déconnectez l’utilisateur d’Auth0 avant d’envoyer une nouvelle requête.
