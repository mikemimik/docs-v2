description: Comment utiliser Auth0 pour sécuriser un CLI.
title: Sécuriser un CLI avec Auth0
---

Les trois façons de sécuriser un CLI avec Auth0, de la plus sécurisée à la moins sécurisée, sont :

* [flux d’autorisation de l’appareil](#device-authorization-flow) lorsque l&#39;utilisateur ne peut pas ouvrir un navigateur
* [Client Credentials Grant Flow](#client-credentials-grant-flow) pour les applications qui agissent en leur propre nom et ne sont pas attribuables à un utilisateur
* [flux Resource Owner Password Grant](#resource-owner-password-grant-flow) seulement lorsque vous essayez d&#39;authentifier le client CLI lui-même, ce qui est une situation très rare (autrement, ce n&#39;est pas recommandé)

<div id="device-authorization-flow">
  ## Flux d’autorisation de l’appareil
</div>

Avec les appareils à capacité de saisie limitée qui se connectent à Internet, au lieu d’authentifier directement l’utilisateur, l’appareil lui demande d’ouvrir un lien sur son ordinateur ou son téléphone intelligent et d’autoriser l’appareil. Cela évite une mauvaise expérience utilisateur sur les appareils qui ne permettent pas d’entrer du texte facilement. Pour ce faire, les applications de l’appareil utilisent le <Tooltip tip="Flux d’autorisation : mécanisme d’autorisation (ou workflow) spécifié dans le framework OAuth 2.0." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Authorization+Flow">flux d’autorisation</Tooltip> de l’appareil (défini dans OAuth 2.0), dans lequel elles transmettent leur <Tooltip tip="Client ID : valeur d’identification attribuée à votre ressource enregistrée par Auth0." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Client+ID">Client ID</Tooltip> pour lancer le processus d’autorisation et obtenir un jeton.

La façon la plus simple de mettre en œuvre le flux d’autorisation de l’appareil est de suivre les étapes de [Call API Using Device Authorization Flow](/docs/fr-CA/get-started/authentication-and-authorization-flow/device-authorization-flow/call-your-api-using-the-device-authorization-flow).

Pour en savoir plus sur le flux d’autorisation de l’appareil dans <Tooltip tip="OAuth 2.0 : framework d’autorisation qui définit les protocoles et workflows d’autorisation." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=OAuth+2.0">OAuth 2.0</Tooltip>, vous pouvez consulter le document provisoire de l’Internet Engineering Task Force (IETF) [OAuth 2.0 Authorization Grant](https://tools.ietf.org/html/draft-ietf-oauth-device-flow-15). Vous pouvez aussi consulter notre article [Device Authorization Flow](/docs/fr-CA/get-started/authentication-and-authorization-flow/device-authorization-flow).

<div id="client-credentials-grant-flow">
  ## Client Credentials Grant Flow
</div>

Utilisez le flux Client Credentials Grant (CCG) lorsque ni les utilisateurs ni les <Tooltip tip="Identity Provider (IdP) : service qui stocke et gère les identités numériques." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=identity+providers">fournisseurs d’identité</Tooltip> en aval ne sont impliqués et que vous souhaitez authentifier des machines ou des appareils distincts.

Si votre fournisseur d’identité prend en charge l’envoi de credentials, nous vous recommandons de consulter notre article [Client Credentials Flow](/docs/fr-CA/get-started/authentication-and-authorization-flow/client-credentials-flow). Pour savoir comment mettre en œuvre ce flux, consultez [Call API Using the Client Credentials Flow](/docs/fr-CA/get-started/authentication-and-authorization-flow/client-credentials-flow/call-your-api-using-the-client-credentials-flow).

<div id="resource-owner-password-grant-flow">
  ## Flux Resource Owner Password Grant
</div>

Nous ne recommandons pas d’utiliser le flux <Tooltip tip="Resource Owner : entité (comme un utilisateur ou une application) capable d’accorder l’accès à une ressource protégée." cta="Consulter le glossaire" href="/docs/fr-CA/glossary?term=Resource+Owner">Resource Owner</Tooltip> Password Grant (ROPG) pour les applications natives. Dans l’article de l’IETF, [RFC 8252 OAuth 2.0 for Native Apps](https://tools.ietf.org/html/rfc8252), il est recommandé que « les demandes d’autorisation OAuth 2.0 provenant d’applications natives soient UNIQUEMENT effectuées au moyen d’agents utilisateurs externes, principalement le navigateur de l’utilisateur ». Pour en savoir plus, consultez [RFC 8252 Embedded User-Agents](https://tools.ietf.org/html/rfc8252#section-8.12).

L’utilisation de Resource Owner Password Grant (ROPG) est moins sécuritaire que les options basées sur la redirection décrites ci-dessus. ROPG est réservé aux anciens systèmes. Dans le contexte des CLI, cela n’a de sens que pour des éléments comme les chaînes de connexion, lorsque vous devez prendre en charge d’anciens programmes.

Si vous devez utiliser ROPG dans votre application native plutôt que le Device Flow, comme nous le recommandons, vous pouvez utiliser notre [endpoint ROPG conforme à OIDC](https://auth0.com/docs/api/authentication#resource-owner-password).
