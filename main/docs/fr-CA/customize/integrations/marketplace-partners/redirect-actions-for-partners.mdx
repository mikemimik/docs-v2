title: Actions de redirection pour les partenaires
---

Lorsqu’un utilisateur se connecte ou s’inscrit, une Action de redirection vous permet de le rediriger vers une page externe (comme un formulaire de consentement), puis de le renvoyer vers Auth0 pour terminer sa connexion ou son inscription. Vous pouvez rediriger les utilisateurs vers une application externe qui leur demande de :

* Effectuer une action, comme une vérification d’identité
* Fournir des renseignements, comme dans le cadre du profilage progressif
* Accepter quelque chose, comme un consentement ou des conditions d’utilisation

Ce processus se déroule dans une Action de redirection :

1. L’**application du client** redirige l’utilisateur vers Auth0 pour qu’il se connecte.
2. Si la connexion réussit, toutes les Actions du [déclencheur Post Login](/docs/fr-CA/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger) s’exécutent (cela se produit avant la MFA, si elle est activée).
3. Si une Action déclenche une redirection, l’utilisateur est envoyé vers l’URL désignée avec un paramètre state. Cette URL doit être hébergée par votre service ou par le client.
4. L’utilisateur est redirigé ou renvoyé par POST vers Auth0 à un chemin précis de son domaine, avec la valeur state d’origine, et l’Action exécute tout code présent dans `onContinuePostLogin`.
5. L’utilisateur est renvoyé vers l’application avec son identité ou, si quelque chose a échoué, un message d’erreur.

Lorsque vous êtes prêt à intégrer votre service à ce processus, il y a des éléments importants à prendre en compte :

* Comment déterminez-vous quand rediriger l’utilisateur hors d’Auth0 ?

  * Un indicateur dans l’[app&#95;metadata de l’utilisateur](/docs/fr-CA/manage-users/user-accounts/metadata/manage-metadata-rules) ?
  * En fonction d’un champ de métadonnées particulier sur le client ?
* Comment gérez-vous les données de profil utilisateur existantes qui doivent être vérifiées ? (Ces données peuvent être fournies par l’utilisateur ou provenir d’une source d’identité fédérée, comme Google, Facebook ou Azure AD.)
* De quelles données d’Auth0 avez-vous besoin dans votre service et comment les y acheminer de manière sécurisée ?
* Comment enregistrez-vous la valeur state provenant d’Auth0 dans votre service ?
* Comment obtenez-vous et enregistrez-vous l’URL `/continue` vers laquelle vous voulez faire un POST ou une redirection ?
* Que renvoyez-vous à Auth0 et comment le faites-vous de façon sécurisée ?
* Comment indiquez-vous que la vérification d’identité est terminée et en transmettez-vous le statut ?
* Comment stockez-vous l’information nécessaire dans l’[app&#95;metadata de l’utilisateur](/docs/fr-CA/manage-users/user-accounts/metadata/manage-metadata-rules) ou le [profil utilisateur normalisé](/docs/fr-CA/manage-users/user-accounts/user-profiles/user-profile-structure)
* Gardez à l’esprit les [limites de débit](/docs/fr-CA/troubleshoot/customer-support/operational-policies/rate-limit-policy/management-api-endpoint-rate-limits) et mettez à jour seulement lorsque nécessaire
* Comment renvoyez-vous l’information à l’application demandeuse à l’aide de [claims personnalisées du jeton](/docs/fr-CA/get-started/apis/scopes/sample-use-cases-scopes-and-claims) ?

Pour vous aider à répondre à toutes ces questions et plus encore, lisez [Redirection avec Actions](/docs/fr-CA/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger/redirect-with-actions). Lorsque vous êtes prêt à soumettre votre intégration d’Action, suivez les étapes 4 à 6 dans [Intégrations d’Action pour les partenaires.](/docs/fr-CA/customize/integrations/marketplace-partners/actions-integrations-for-partners)
