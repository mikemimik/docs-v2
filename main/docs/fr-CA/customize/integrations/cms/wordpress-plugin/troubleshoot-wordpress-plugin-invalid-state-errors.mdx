title: Dépannage des erreurs de state invalide du plugin WordPress
---

Nous avons ajouté la validation du state à la version 3.6.0 du plugin WordPress, que vous trouverez dans le [`auth0/wordpress` GitHub repository](https://github.com/auth0/wordpress/releases/tag/3.6.0). Cette mesure de sécurité aide à atténuer les attaques CSRF en s’assurant que la réponse correspond à une requête lancée par le même utilisateur. Pour en savoir plus, consultez [Prévenir les attaques et rediriger les utilisateurs avec le paramètre de state OAuth 2.0](/docs/fr-CA/secure/attack-protection/state-parameters).

<div id="how-state-validation-works">
  ## Comment fonctionne la validation du state
</div>

Le plugin effectue la validation du state de la façon suivante :

1. il définit un cookie `auth0_state` dans le navigateur au moyen de Javascript lorsque le formulaire de connexion Lock s’affiche (sur `wp-login.php` ou sur toute autre page si vous utilisez un shortcode ou un widget) ;
2. il transmet cette valeur au formulaire de connexion intégré de Lock afin qu’elle soit envoyée avec la requête d’authentification ;
3. il reçoit cette valeur en retour d’Auth0, sans modification, dans un paramètre d’URL `state` si la connexion Auth0 a réussi ;
4. il valide que la valeur reçue correspond à celle qui a été envoyée et stockée dans le cookie `auth0_state`. Si elle est valide, le processus de connexion se poursuit. Sinon, le processus s’arrête et le message d’erreur « Invalid state » s’affiche ;
5. il supprime le cookie, qu’il soit valide ou non ;
6. il utilise les valeurs de l’objet décodé en base64 pour rediriger l’utilisateur ou effectuer d’autres actions de connexion, si elles sont valides.

Ce processus devrait être complètement transparent, autant pour l’utilisateur qui se connecte que pour l’administrateur du site. Le serveur Auth0 ne valide pas la valeur state, ne l’exige pas, et la renvoie telle quelle à l’URL de rappel. Si le message « Invalid state » s’affiche, c’est qu’une des 4 premières étapes ci-dessus ne se produit pas.

<div id="common-causes-of-the-invalid-state-error">
  ## Causes courantes de l’erreur « invalid state »
</div>

Voici quelques causes courantes de l’erreur « invalid state », ainsi que des étapes de dépannage que vous pouvez suivre.

<div id="cached-callback-urls">
  ### URL de rappel mises en cache
</div>

La cause la plus fréquente de l’erreur `invalid state` est que l’URL de rappel est mise en cache sur le serveur.

Excluez la mise en cache sur votre serveur pour toutes les URL indiquées dans le champ **Allowed Callback URLs** de [Auth0 Dashboard &gt; Applications &gt; Applications &gt; Settings](https://manage.auth0.com/#/applications/\{yourClientId}/settings), puis refaites le test. De plus, excluez aussi de la mise en cache l’URL du site (`/index.php` dans une installation standard) si elle contient un paramètre d’URL Auth0.

Vérifiez également que l’heure de votre serveur est bien réglée. L’erreur `BeforeValidException` peut se produire lorsque le token est perçu comme ayant été généré avant l’heure actuelle, ce qui peut arriver si l’heure du serveur n’est pas correcte. Vous pouvez vérifier l’heure du serveur en utilisant `echo current_time( 'c' )`. Comme solution temporaire, vous pouvez aussi modifier le plugin pour ajouter un décalage horaire si vous ne pouvez pas changer l’heure du serveur, mais le problème devrait être corrigé en production.

Si cela ne règle pas le problème, poursuivez avec les étapes de dépannage ci-dessous.

<div id="cached-cookies-and-url-parameters">
  ### Cookies et paramètres d’URL mis en cache
</div>

Si vous utilisez un hébergeur géré comme WP-Engine, vous devrez peut-être communiquer avec son équipe de soutien pour obtenir de l’aide supplémentaire. Nous avons reçu des signalements de problèmes d’accès aux cookies requis sur l’URL de rappel, ainsi que de problèmes au moment de vérifier l’authentification sur la page finale que les utilisateurs voient après s’être connectés. Plus précisément, demandez l’ajout d’exclusions de cache pour :

* **Cookie :** `auth0_state`
* **Cookie :** `auth0_nonce`
* **Arg/paramètre d’URL :** `auth0`
* **Arg/paramètre d’URL :** `code`
* **Arg/paramètre d’URL :** `state`
* **Arg/paramètre d’URL :** `id_token`

<div id="page-refresh-after-error-message">
  ### Actualisation de la page après un message d’erreur
</div>

Si vous actualisez la page après avoir vu un autre message d’erreur (vérification du courriel, etc.), le message `invalid state` s’affichera, car le système tente de valider de nouveau une valeur déjà utilisée. C’est normal.

<div id="cookie-names-requirement">
  ### Exigence relative aux noms des cookies
</div>

Certains hébergeurs, comme Pantheon, exigent l’utilisation de noms de cookies précis. Vous pouvez modifier le nom du cookie à l’aide du filtre `auth0_state_cookie_name` dans votre thème ou votre plugin personnalisé. Pour en savoir plus sur le filtre `auth0_state_cookie_name`, consultez [Extend Login by Auth0 WordPress Plugin](/docs/fr-CA/customize/integrations/cms/wordpress-plugin/extend-login-by-auth0). Pour plus d’information, [consultez l’issue GitHub connexe](https://github.com/auth0/wordpress/issues/494) et [le correctif correspondant](https://github.com/auth0/wordpress/pull/495).

<div id="universal-login-page-and-link-building">
  ### Universal Login Page et création du lien
</div>

Si votre site utilise la page <Tooltip tip="Universal Login : votre application redirige vers Universal Login, hébergé sur l’Authorization Server d’Auth0, pour vérifier l’identité d’un utilisateur." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Universal+Login">Universal Login</Tooltip> et que vous créez vous-même le lien dans un thème ou un plugin, vous devez :

* Définir un cookie nommé `auth0_state` avec une valeur générée aléatoirement
* Envoyer cette valeur dans un paramètre d’URL `state`.

Vous pouvez aussi aller dans Settings &gt; onglet Features &gt; Universal Login Page et rediriger les requêtes de connexion vers la page `wp-login.php`, où ce cookie et ce paramètre d’URL seront définis automatiquement.

<div id="visiting-the-callback-url-directly">
  ### Accéder directement à l’URL de rappel
</div>

Si vous accédez directement à votre URL de rappel (généralement `yourdomain.com/index.php?auth0=1`) ou si vous y retournez une deuxième fois après l’échange du code d’autorisation, l’erreur invalid state peut s’afficher. Cela indique que la valeur state a déjà été vérifiée et supprimée.

<div id="troubleshoot-invalid-state-errors">
  ## Dépannage des erreurs « invalid state »
</div>

Notez que certaines des étapes ci-dessous exigeront d’interrompre le processus de connexion en cours de route (cela sera indiqué) :

1. Pendant que vous êtes déconnecté de WordPress et d’Auth0, accédez à la page de connexion à tester.
2. Vérifiez si le cookie `auth0_state` est défini (dans Chrome, View &gt; Developer &gt; JavaScript Console &gt; onglet Application &gt; Storage à gauche &gt; Cookies &gt; domaine testé; recherchez un cookie `auth0_state` avec une valeur non vide).

   * Si cette valeur n’est pas définie, vérifiez s’il y a des erreurs dans la console JS et assurez-vous que votre navigateur accepte les cookies (la connexion ne fonctionnera pas sans cookies).
   * Si la valeur est définie, copiez-la et affichez le code source de la page (dans Chrome, **View** &gt; **Developer** &gt; **View Source**). Recherchez cette valeur; elle devrait apparaître comme valeur associée au paramètre `wpAuth0LockGlobal.settings.auth.params.state` ([voir un exemple de JSON](https://gist.github.com/joshcanhelp/1b8bb990048325eb7214e2b3d7136b78)). Prenez cette valeur en note (vous en aurez besoin à une étape ultérieure).
3. Si la valeur apparaît à cet endroit et que le formulaire Lock se charge normalement, cela signifie que les étapes 1 et 2 de la première liste ci-dessus fonctionnent correctement.
4. Avant de vous connecter, [ajoutez cet extrait de code au début de votre fichier `wp-config.php`](https://gist.github.com/joshcanhelp/ba98f748747c7fd2ecdf54e73c6110f3), afin de pouvoir faire une installation de test. **AVERTISSEMENT** : cela interrompra la connexion au site WordPress testé; utilisez-le donc uniquement sur une installation hors production.
5. Connectez-vous normalement.
6. Une fois redirigé vers l’URL de rappel de votre site, le processus s’arrêtera. Vous devriez voir une sortie semblable à celle montrée dans le Gist lié à l’étape #4 ci-dessus. Si vous voyez quelque chose comme `Array()` sans autre valeur, l’une des deux situations suivantes est probablement en cause :

   * L’URL de rappel de WordPress est mise en cache. La mise en cache des pages peut se produire de nombreuses façons; nous ne pouvons donc pas fournir d’étapes précises ici. Vérifiez les plugins de mise en cache que vous avez peut-être installés; elles offrent généralement un mécanisme d’exclusion d’URL. Vérifiez aussi auprès de votre hébergeur, car la mise en cache peut être automatique et nécessiter l’intervention du soutien.
   * Le serveur ne lit pas le cookie Auth0. Pour une solution possible, [consultez l’issue GitHub connexe](https://github.com/auth0/wordpress/issues/494) et [explorez son correctif](https://github.com/auth0/wordpress/pull/495).
7. Si les valeurs sont présentes, vérifiez les en-têtes de réponse de l’URL de rappel en cours de chargement (dans Chrome, View &gt; Developer &gt; JavaScript Console &gt; onglet Network, cliquez sur le premier « document » affiché avec un code d’état 500, puis recherchez « Response Headers »). Cherchez tout indice de mise en cache, comme un `Cache-Control` avec un `max-age` différent de zéro, un `x-cache` autre que `MISS`, ou tout autre signe que cette page est servie à partir d’un cache.
8. Toujours dans les en-têtes de réponse, vérifiez que `set-cookie` inclut une directive comme `auth0_state=deleted` pour confirmer que le processus de validation a bien lieu.
9. Assurez-vous que le paramètre `state` dans l’URL correspond à celui consigné à partir du cookie défini à l’étape #3 ci-dessus.
10. S’il n’y a aucun signe de mise en cache, supprimez l’extrait de débogage de `wp-config.php` et actualisez l’URL de rappel. Vous devriez voir de nouveau le message « Invalid state ». Si des changements liés à la mise en cache ont été apportés, exécutez le processus de connexion jusqu’au bout (assurez-vous d’effacer vos cookies et le cache du navigateur pour ce site avant de tester).

Les étapes de dépannage suivantes nécessitent des modifications au plugin qui interrompront le processus de connexion et devront être annulées une fois l’opération terminée. Ces étapes devraient être effectuées sur un serveur de test ou de staging.

11. Nous devons ensuite vérifier pourquoi la valeur state est bien reçue, mais ne correspond pas à la valeur stockée.

12. Dans `lib/WP_Auth0_LoginManager.php`, affichez les valeurs du state stocké et du state renvoyé, puis arrêtez le processus. Juste avant [la ligne 148](https://github.com/auth0/wp-auth0/blob/master/lib/WP_Auth0_LoginManager.php#L148), ajoutez :

```bash wrap lines
echo '<h1>$_REQUEST</h1>'; var_dump($_REQUEST); echo '<h1>$_COOKIE</h1>'; var_dump($_COOKIE); die('<h1>Done</h1>');
```

13. Encore une fois, assurez-vous d&#39;être déconnecté, puis terminez le processus de connexion.

14. Vous devriez voir des valeurs s&#39;afficher lorsque vous êtes redirigé vers l&#39;URL de rappel de WordPress.

15. Vérifiez si la valeur `state` dans `$_REQUEST` existe et correspond à la valeur `auth0_state` dans `$_COOKIE`.

* Si elle est différente, elle devrait correspondre à la valeur d’origine consignée à l’étape 3 ci-dessus. Cela signifie que la valeur `state` de `$_COOKIE` a changé à un moment du processus.

Si aucune des étapes ci-dessus ne résout le problème, veuillez recueillir les résultats des étapes ci-dessus et [contacter le soutien](https://support.auth0.com/center/s/) ou [publier sur Community](https://community.auth0.com/tag/wordpress) avec l’étiquette `wordpress`. Veuillez aussi inclure :

* Version de PHP
* Version de WordPress
* Version du plugin Auth0
* Navigateur et système d’exploitation utilisés pour le test
* Un [fichier HAR](https://support.auth0.com/center/s/article/generate-and-analyze-har-files-to-troubleshoot-auth0-issues) enregistrant l’ensemble du processus, du chargement de la page avec le formulaire de connexion jusqu’au message « invalid state ».

  <Warning>
    Avant de partager un fichier HAR avec qui que ce soit (y compris Auth0), assurez-vous de supprimer ou d’obfusquer toutes les données sensibles, comme :

    * Renseignements confidentiels sur l’utilisateur
    * Renseignements personnels identifiables (PII)
    * Renseignements confidentiels sur l’application

  </Warning>
