description: Décrit la fonctionnalité de migration des utilisateurs du plugin Login by Auth0 WordPress
title: Migration des utilisateurs dans le plugin Login by Auth0 WordPress
---

La fonctionnalité de migration des utilisateurs s’appuie sur une fonctionnalité centrale d’Auth0 appelée Custom Databases, combinée aux points de terminaison URL du plugin Login by Auth0, pour permettre aux utilisateurs de s’authentifier avec des comptes WordPress existants. Pour en savoir plus sur les bases de données personnalisées, consultez [les connexions de base de données personnalisées](/docs/fr-CA/authenticate/database-connections/custom-db).

<div id="how-it-works">
  ## Fonctionnement
</div>

Lorsque vous activez la migration des données, le plugin expose deux points de terminaison sécurisés qui permettent à Auth0 d’authentifier les utilisateurs à l’aide de comptes WordPress. Ces points de terminaison sont protégés par un jeton secret et peuvent être configurés pour n’autoriser que les adresses IP utilisées par Auth0.

Le flux de connexion se déroule comme suit :

1. Un utilisateur tente de se connecter à l’aide d’un formulaire de connexion Auth0 (intégré à votre site ou hébergé par Auth0).
2. Si Auth0 ne trouve pas d’utilisateur associé aux informations d’identification fournies dans votre connexion de base de données, il appelle alors le point de terminaison de migration sur votre site WordPress avec les informations d’identification de l’utilisateur et le jeton de migration.
3. Le plugin trouve un utilisateur dans votre base de données WordPress à partir du nom d’utilisateur ou de l’adresse courriel fournis, puis vérifie le mot de passe.
4. Si l’utilisateur peut être authentifié avec succès, Auth0 crée l’utilisateur dans la connexion de base de données de votre site, l’authentifie et ouvre sa session.
5. La prochaine fois que l’utilisateur ouvrira une session, il utilisera son compte Auth0 et le point de terminaison de migration sera ignoré.

La migration des utilisateurs doit être configurée lorsque le site est connecté à Auth0 pour la première fois. Toute tentative d’activer ou de désactiver des scripts de base de données personnalisés pour une connexion de base de données qui contient déjà des utilisateurs échouera. Consultez la section **Dépannage** pour en savoir plus sur le passage d’un mode à l’autre.

<div id="set-up-and-configuration">
  ## Mise en place et configuration
</div>

La façon la plus simple de configurer la migration des utilisateurs consiste à utiliser l’assistant de configuration lors de l’installation initiale de l’extension. Pour en savoir plus sur le processus, consultez [Installer Login by Auth0](/docs/fr-CA/customize/integrations/cms/wordpress-plugin/install-login-by-auth0).

Si l’assistant de configuration de la migration des utilisateurs n’a pas pu être mené à terme, ou si vous voulez voir le processus plus en détail, suivez les étapes ci-dessous. Encore une fois, il s’agit de repartir de zéro avec une connexion de base de données qui ne contient aucun utilisateur. Le processus suivant doit être effectué sur un site sans trafic ou avec le mode maintenance activé.

1. [Créez et configurez correctement une Application](/docs/fr-CA/customize/integrations/cms/wordpress-plugin/configure-login-by-auth0), puis créez et activez une Database Connection vide pour l’Application. Il peut s’agir des mêmes éléments que ceux créés dans le processus standard du Setup Wizard, ou vous pouvez les créer à partir de zéro. Pour en savoir plus sur le Setup Wizard, consultez [Install Login by Auth0](/docs/fr-CA/customize/integrations/cms/wordpress-plugin/install-login-by-auth0).
2. Dans l’écran **Auth0 &gt; Settings** de WordPress, assurez-vous que le Domain, le Client ID et le Client Secret de l’Application sont enregistrés dans les champs appropriés de l’onglet **Basic**.
3. Dans la vue **Advanced**, activez le paramètre **User Migration Endpoints**, puis sélectionnez **Save Changes**. Si vous utilisez des paramètres définis par des constantes, réglez `AUTH0_ENV_MIGRATION_WS` à `true` et `AUTH0_ENV_MIGRATION_TOKEN` à une chaîne aléatoire sécurisée d’au moins 16 caractères, sans guillemets simples ni barres obliques inverses.
4. Sous les paramètres, vous devriez maintenant voir un **Security Token**. Gardez cette page ouverte, car vous aurez besoin de cette valeur plus tard dans le processus.
5. Dans l’Auth0 Dashboard, accédez à la Database Connection que vous voulez utiliser et activez **Requires Username** et **Import Users to Auth0**.
6. Sélectionnez la vue **Custom Database**, puis activez **Use my own database**.
7. Vous devriez voir deux onglets sous ce paramètre dans **Database Action Scripts** : un pour **Login** et un pour **Get User**.
8. Sélectionnez la vue **Login**, supprimez le code existant, [copiez le code db-login.js du GitHub repository](https://raw.githubusercontent.com/auth0/wp-auth0/4.4.0/lib/scripts-js/db-login.js), puis collez-le dans l’éditeur de code.
9. **Cette étape s’applique aux versions 3.10.0 et antérieures :** Recherchez `{THE_WS_URL}` et remplacez-le par l’URL du site de votre instance WordPress, suivie de `/index.php?a0_action=migration-ws-login`. L’URL du site se trouve dans l’écran **Settings &gt; General** de wp-admin. Vous pouvez la tester en collant l’URL complète dans votre navigateur. Vous devriez voir `{"status":401,"error":"Unauthorized"}`.
10. **Cette étape s’applique aux versions 3.10.0 et antérieures :** Recherchez `{THE_WS_TOKEN}` et remplacez-le par le jeton qui apparaît sous le paramètre **User Migration Endpoints**.
11. Il ne devrait y avoir aucune erreur dans l’éditeur. Si tout semble correct, cliquez sur **Save** en haut.
12. **Cette étape s’applique à la version 3.11.0 et aux versions ultérieures :** descendez jusqu’à **Paramètres** et ajoutez les variables de configuration suivantes :

    * `endpointUrl` défini comme l’URL du site de votre instance WordPress (**wp-admin &gt; Settings &gt; General &gt; champ &quot;URL du site&quot;**), suivie de `/index.php?a0_action=`.
    * `migrationToken` défini comme la valeur du token de sécurité indiquée à l’étape 4 ci-dessus.
    * `userNamespace` défini comme le nom de votre Application dans Auth0, ou toute autre valeur contenant uniquement des lettres, des chiffres et des traits d’union.

    <Frame>![Paramètres de la base de données personnalisée du plugin WordPress](/docs/images/cdy7uua7fh8z/1icr0yX47nODUrzP7B6Xl2/b43621c3b6641f68bcb14ac38b237c11/auth0-custom-database-config.png)</Frame>
13. Cliquez sur le bouton **Try** en haut et utilisez un compte d’utilisateur WordPress valide dans le formulaire qui s’affiche. Vous devriez voir &quot;The profile is&quot;, suivi des données de l’utilisateur. Sinon, veuillez consulter la section **Dépannage** ci-dessous.
14. Sélectionnez la vue **Get User**, supprimez le code existant, [copiez le code db-get-user.js du dépôt GitHub](https://raw.githubusercontent.com/auth0/wp-auth0/4.4.0/lib/scripts-js/db-get-user.js), puis collez-le dans l’éditeur de code.
15. **Cette étape s’applique à la version 3.10.0 et aux versions antérieures :** Repérez `{THE_WS_URL}` et remplacez-le par l’URL du site de votre instance WordPress, suivie de /index.php?a0&#95;action=`migration-ws-get-user`. L’URL du site se trouve dans l’écran **Settings &gt; General** de wp-admin. Vous pouvez la tester en collant l’URL complète dans votre navigateur. Vous devriez voir `{"status":401,"error":"Unauthorized"}`.
16. **Cette étape s’applique à la version 3.10.0 et aux versions antérieures :** Repérez `{THE_WS_TOKEN}` et remplacez-le par le jeton affiché sous le paramètre **User Migration Endpoints**.
17. Il ne devrait y avoir aucune erreur dans l’éditeur. Si tout semble bon, cliquez sur **Save**.
18. Cliquez sur le bouton **Try** en haut et utilisez une adresse courriel associée à un compte d’utilisateur WordPress valide dans le formulaire qui s’affiche. Vous devriez voir &quot;The profile is&quot;, suivi des données de l’utilisateur. Sinon, veuillez consulter la section **Dépannage** ci-dessous.
19. Dans une nouvelle session de navigateur, accédez à une page de connexion sur le site WordPress et essayez de vous connecter (l’utilisateur ne devrait pas déjà exister dans la base de données). Vous remarquerez que le processus de connexion prend un peu plus de temps que d’habitude au début, mais il devrait réussir. Les connexions suivantes seront plus rapides.
20. (FACULTATIF) Pour activer une sécurité supplémentaire pour les points de terminaison de migration, accédez à l’écran **Auth0 &gt; Settings** dans WordPress, activez-la, puis cliquez sur **Save Changes**. Essayez de vous connecter avec un autre utilisateur pour vérifier qu’Auth0 peut toujours joindre les points de terminaison.

À ce stade, la configuration de la migration des utilisateurs est terminée, et les utilisateurs WordPress existants seront migrés graduellement vers la connexion de base de données Auth0.

<div id="troubleshooting">
  ## Dépannage
</div>

Les problèmes liés à la migration des utilisateurs proviennent généralement de quelques sources :

* URL ou jeton incorrect dans les scripts de base de données personnalisée.
* Liste d’autorisation IP activée, mais avec des adresses IP incorrectes.
* Points de terminaison restreints ou mis en cache sur votre instance WordPress.

La meilleure façon de commencer le dépannage consiste à utiliser le bouton **Try** du script **Login**, dans la vue Custom Database de la connexion de base de données utilisée dans [Auth0 Dashboard &gt; Authentication &gt; Database](https://manage.auth0.com/#/connections/database). Voici les messages d’erreur que vous pourriez voir et les étapes à suivre pour corriger le problème.

<div id="unexpected-token-in-json-at-position-0">
  ### Jeton inattendu &lt; dans JSON à la position 0
</div>

Cela signifie que le script personnalisé ne reçoit pas les données dans un format qu’il peut utiliser. Dans le script de base de données, la cause la plus probable est une URL de point de terminaison incorrecte.

Commencez par copier l’URL à la ligne 10 du script et la coller dans votre navigateur. Si le point de terminaison est correct, l’un des deux messages ci-dessous devrait s’afficher :

`{"status":401,"error":"Unauthorized"}`

`// or`

`{"status":403,"error":"Forbidden"}`

Si vous voyez plutôt la page d’accueil ou une erreur 404, l’URL est incorrecte. Cherchez l’URL de votre site sous **Settings &gt; General &gt; Site URL** dans l’interface d’administration de WordPress. Ajoutez `/index.php?a0_action=migration-ws-login` à la fin pour le script Login, et `/index.php?a0_action=migration-ws-get-user` à la fin pour le script Get User.

* **Pour les versions 3.10.0 et antérieures** : la valeur de l’URL devrait figurer directement dans le script, comme premier paramètre de l’appel `request.post`.
* **Pour les versions 3.11.0 et ultérieures** : la valeur du jeton devrait être enregistrée dans une variable de configuration. Ajoutez ce qui suit à la première ligne de la fonction et utilisez le bouton **Try** pour voir ce qui est enregistré pour `endpointUrl` :

`callback(null, configuration);`

Si vous êtes certain que les URL sont correctes et que le problème persiste, vérifiez auprès de votre hébergeur pour vous assurer que ces URL ne sont ni mises en cache ni restreintes de quelque façon que ce soit.

<div id="wrong-email-or-password">
  ### Courriel ou mot de passe incorrect
</div>

Il s’agit de l’erreur affichée par défaut quand un autre problème survient. Le moyen le plus simple de déterminer ce qui se passe consiste à afficher temporairement l’erreur renvoyée (elles sont opaques par défaut afin d’éviter d’afficher quoi que ce soit qui pourrait être utile à des attaquants).

À la ligne 30 du script Login, remplacez :

`callback(null);`

par :

`callback(wpUser.error);`

Enregistrez le script, puis réessayez la connection. Vous devriez voir l’un des messages suivants et pouvoir cerner le problème à l’aide des étapes ci-dessous. Une fois le problème résolu, remettez le script comme avant.

<div id="forbidden">
  ### Interdit
</div>

Cela signifie que les points de terminaison de migration des utilisateurs sont désactivés dans votre installation WordPress. Dans WordPress, allez à **Auth0 &gt; Settings &gt; Advanced** et activez **User Migration Endpoints**. Assurez-vous que le jeton qui s’y affiche est le même que celui utilisé dans les deux scripts de base de données personnalisée :

* **Pour les versions 3.10.0 et antérieures** : la valeur du jeton doit apparaître dans le script lui-même après `access_token:`
* **Pour les versions 3.11.0 et ultérieures** : la valeur du jeton doit être enregistrée dans une variable de configuration. Ajoutez ce qui suit à la première ligne de la fonction et utilisez le bouton **Try** pour voir ce qui est stocké dans `migrationToken` :

`callback(null, configuration);`

<div id="unauthorized">
  ### Non autorisé
</div>

Cela signifie que la liste d’autorisation IP pour la migration est activée, mais que l’adresse IP entrante ne s’y trouve pas. Juste sous le script Login, vous devriez voir une liste d’adresses IP :

<Frame>![WordPress User Migration - Auth0 IP Addresses](/docs/images/cdy7uua7fh8z/3RnxoRlvXm8FmNfu02fXVn/3672806c9cbf2bfe28503be8b8a63eda/auth0-incoming-ip-addresses.png)</Frame>

Assurez-vous que toutes ces adresses IP figurent dans WordPress sous **Auth0 &gt; Settings &gt; Advanced** dans le plugin :

<Frame>![WordPress User Migration - IP Whitelist](/docs/images/cdy7uua7fh8z/3w6C1IrsSC9hoivpSY122l/d117857524ef8d7872f364d3a7dbabda/migration-ip-whitelist-setting-field.png)</Frame>

Si une ou plusieurs des adresses IP listées dans Auth0 n’apparaissent pas dans WordPress, ajoutez celles qui manquent dans le champ et enregistrez la page Settings. De plus, [créez une publication dans l’Auth0 Community](https://community.auth0.com/tags/wordpress) (et ajoutez-y la balise &quot;wordpress&quot;) contenant la ou les adresses IP manquantes afin que nous puissions régler le problème.

<div id="unauthorized-missing-authorization-header">
  ### Unauthorized : en-tête `Authorization` manquant
</div>

Le <Tooltip tip="Jeton de sécurité : artefact signé numériquement utilisé pour prouver que l’utilisateur a été authentifié avec succès." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=security+token">jeton de sécurité</Tooltip> est soit absent du script de base de données (ligne 16), soit votre serveur ne traite pas correctement les en-têtes. Vérifiez le script Login et assurez-vous que le jeton existe et correspond à celui dans WordPress. Si le jeton est présent et correct, vous devrez communiquer avec votre hébergeur pour activer l’analyse de l’en-tête `Authorization`. Pour obtenir de l’aide pour le dépannage du serveur, consultez [Apache 2.4 + PHP-FPM and Authorization headers on stackoverflow.com](https://stackoverflow.com/questions/17018586/apache-2-4-php-fpm-and-authorization-headers). Pour voir comment le jeton est récupéré, consultez le [code du Plugin dans le GitHub repository.](https://github.com/auth0/wp-auth0/blob/master/lib/WP_Auth0_Routes.php#L138)

<div id="invalid-token">
  ### Jeton invalide
</div>

Le jeton de sécurité du script de base de données est incorrect. Vérifiez la ligne 16 du script Login et assurez-vous que le jeton correspond à celui de WordPress.

<div id="invalid-credentials">
  ### Identifiants invalides
</div>

L’adresse courriel et/ou le mot de passe utilisés sont incorrects. Vérifiez que vous avez saisi la bonne adresse courriel et le bon mot de passe. Vous pouvez réinitialiser le mot de passe de l’utilisateur afin de vous assurer d’utiliser le bon.

<div id="cannot-change-email-or-incorrect-user-data">
  ### Impossible de modifier l’adresse courriel ou données utilisateur erronées
</div>

Si vous utilisez plus d’une connexion de base de données personnalisée dans votre tenant Auth0 et que vous ne parvenez pas à modifier l’adresse courriel, ou que des données utilisateur sont enregistrées pour le mauvais utilisateur, il est probable que des ID utilisateur se chevauchent dans Auth0. Ce problème a été corrigé pour les nouveaux sites qui installent la version 3.11.0, mais pour les connexions créées auparavant, vous devrez le corriger manuellement en procédant de l’une des façons suivantes :

* Si vous n’avez aucune donnée utilisateur enregistrée à conserver (si vous utilisez seulement la connexion pour prendre en charge le login et sans stocker de métadonnées), vous pouvez créer une nouvelle connexion de base de données personnalisée en suivant les étapes ci-dessus (à l’aide des notes de la version 3.11.0) et faire passer l’Application à cette nouvelle connexion (assurez-vous de désactiver l’ancienne connexion). La migration redémarrera et il n’y aura aucune incidence sur l’expérience utilisateur.
* Si vous avez des données dans Auth0 qui doivent être conservées, vous pouvez [utiliser notre extension User Import/Export](/docs/fr-CA/manage-users/user-migration/user-import-export-extension) pour ajuster les données utilisateur.
  1. Créez une nouvelle connexion de base de données personnalisée en suivant les étapes ci-dessus (à l’aide des notes de la version 3.11.0).
  2. Exportez tous les utilisateurs de la connexion existante (nous vous recommandons de mettre votre site en mode maintenance pendant la transition afin qu’aucun utilisateur ne soit omis).
  3. Modifiez tous les ID utilisateur pour ajouter l’espace de noms utilisé lors de la création de la nouvelle connexion. Les ID utilisateur devraient passer de `auth0|123` à `auth0|Your-WP-Site-Name|123`, par exemple. Ajustez tous les autres champs nécessaires pour respecter le schéma d’importation. Pour en savoir plus, consultez [Schéma et exemples d’importation en bloc d’utilisateurs de base de données](/docs/fr-CA/manage-users/user-migration/bulk-user-import-database-schema-and-examples).
  4. Activez la nouvelle connexion et désactivez l’ancienne pour votre application.
  5. Importez les nouvelles données utilisateur dans la nouvelle connexion et effectuez un Test.
* Si vous avez un compte payant, vous pouvez communiquer avec notre équipe de soutien afin d’exécuter un script de mise à jour de la base de données pour convertir les ID utilisateur en version avec espace de noms et ajouter en même temps l’espace de noms à votre script de base de données (étape 12 dans **Configuration et paramétrage** ci-dessus).
