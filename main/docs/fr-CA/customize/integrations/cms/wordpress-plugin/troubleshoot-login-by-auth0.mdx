Voici quelques questions de dépannage fréquentes. Si les éléments ci-dessous ne résolvent pas votre problème, envisagez les solutions suivantes :

* Si vous configurez l’extension pour la première fois ou si des utilisateurs ont de la difficulté à se connecter, veuillez lire [Configurer Login by Auth0](/docs/fr-CA/customize/integrations/cms/wordpress-plugin/configure-login-by-auth0).
* Si vous avez trouvé un bogue dans le code de l’extension, [soumettez une issue](https://github.com/auth0/wordpress/issues) ou [créez une pull request](https://github.com/auth0/wordpress/pulls) dans Github.
* Si vous avez des questions sur l’utilisation d’Auth0 ou de l’extension, veuillez [publier dans notre site Community](https://community.auth0.com/) ou [créer une demande dans le forum de soutien WordPress de l’extension](https://wordpress.org/support/plugin/auth0/).
* Pour consulter de la documentation supplémentaire et trouver plus de réponses, [visitez notre site de soutien](https://support.auth0.com/center/s/). Les clients ayant un plan Auth0 payant peuvent soumettre un ticket d’assistance pour obtenir une réponse rapide.

<div id="im-seeing-the-error-message-invalid-state-that-prevents-me-from-logging-in">
  ## Je vois le message d’erreur « État invalide » qui m’empêche de me connecter
</div>

La validation de l’état est une fonctionnalité de sécurité ajoutée dans la [version 3.6.0](https://github.com/auth0/wordpress/releases/tag/3.6.0). Une URL de rappel mise en cache est généralement à l’origine de ce message d’erreur (consultez les paramètres de votre application pour voir les URL de rappel qui ne doivent pas être mises en cache). Pour en savoir plus, consultez [Résoudre les erreurs « État invalide » du WordPress Extension](/docs/fr-CA/customize/integrations/cms/wordpress-plugin/troubleshoot-wordpress-plugin-invalid-state-errors).

<div id="im-having-an-issue-logging-in-or-changing-emailpassword-using-a-custom-database">
  ## J’ai un problème lorsque j’essaie de me connecter ou de modifier mon adresse courriel ou mon mot de passe à l’aide d’une base de données personnalisée
</div>

Pour en savoir plus sur les solutions possibles, consultez [la Migration des utilisateurs dans plugin WordPress Login by Auth0](/docs/fr-CA/customize/integrations/cms/wordpress-plugin/user-migration-in-login-by-auth0).

<div id="im-seeing-the-error-message-invalid-id-token-or-expired-id-token-that-prevents-me-from-logging-in">
  ## Je vois le message d’erreur &quot;Invalid ID token&quot; ou &quot;Expired ID token&quot; qui m’empêche de me connecter
</div>

Cela est généralement dû à un serveur dont l’heure est mal réglée. Si le message d’erreur contient &quot;used too early,&quot;, c’est que l’heure de votre serveur est réglée dans le futur. S’il indique que le token a expiré, c’est que l’heure du serveur est réglée trop loin dans le passé. Un écart d’heure entre deux serveurs est courant. Exécutez `echo date(DateTime::ISO8601)` en PHP sur votre serveur et comparez le résultat, secondes comprises, à l’heure UTC actuelle. Si l’heure de votre serveur diffère de plus de 60 secondes (la marge par défaut) de l’heure UTC, vous devrez définir une marge plus longue pour tenir compte du décalage de l’horloge de votre serveur. Vous pouvez coller le code ci-dessous dans le fichier `functions.php` de votre thème ou ailleurs, à condition qu’il s’exécute après le chargement du plugin et avant l’exécution du hook de connexion :

```js wrap lines
add_filter( 'auth0_jwt_leeway', function( $default_leeway ) { return 120; } );
```

Cela accorderait une marge de 120 secondes. Vous devrez peut-être l’ajuster selon le décalage de l’horloge de votre serveur.

<div id="i-see-the-error-message-this-account-does-not-have-an-email-associated-that-prevents-me-from-logging-in">
  ## Je vois le message d’erreur &quot;Ce compte n’a pas d’adresse courriel associée...&quot; qui m’empêche de me connecter
</div>

Si vous obtenez cette erreur, assurez-vous de demander une adresse courriel à chaque fournisseur dans le <Tooltip tip="Auth0 Dashboard : le principal produit d’Auth0 pour configurer vos services." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip>, sous **connexions &gt; Social** (développez chaque fournisseur). Gardez à l’esprit que certains fournisseurs ne renvoient pas d’adresse courriel pour leurs utilisateurs (p. ex., X). Dans ce cas, vous pouvez toujours ajouter une adresse courriel à tout utilisateur connecté à partir de l’Auth0 Dashboard (ou de l’API). Consultez **Users &gt; Edit**.

Pour les connexions qui ne fournissent pas d’indicateur `email_verified` (certaines connexions Enterprise ne l’incluent pas) ou pour ignorer cette validation pour certaines Social connexions, ajoutez la stratégie de cette connexion dans le champ &quot;Skip Strategies&quot;. Ce champ se trouve sous le commutateur **Require Verified Email**, accessible via **wp-admin** &gt; **Auth0** &gt; **Settings** &gt; **Advanced**.

**Ce champ ne doit être utilisé qu’au besoin, car il contourne les mesures de sécurité recommandées par Auth0.**

<div id="i-see-the-error-message-there-is-a-user-with-the-same-email-that-prevents-me-from-logging-in">
  ## Je vois le message d’erreur « There is a user with the same email » qui m’empêche de me connecter
</div>

Cela signifie qu’un utilisateur WordPress a la même adresse courriel que celle utilisée pour se connecter, mais qu’elle est associée à un autre utilisateur Auth0. Si vous testez le plugin ou souhaitez plutôt associer l’utilisateur existant au nouvel utilisateur Auth0 :

1. Connectez-vous en tant qu’administrateur
2. Accédez à **wp-admin &gt; Utilisateurs** et recherchez l’adresse courriel utilisée
3. Ouvrez le profil de l’utilisateur et faites défiler jusqu’en bas
4. Cliquez sur **Delete Auth0 Data** et confirmez

Si vous avez 2 comptes d’utilisateur dans Auth0 avec la même adresse courriel, ce message d’erreur persistera. Nous vous recommandons de [lier des comptes d’utilisateur](/docs/fr-CA/manage-users/user-accounts/user-account-linking/link-user-accounts).

<div id="i-see-the-error-message-failed-cross-origin-authentication-or-no-verifier-returned-from-client-in-my-browsers-console-logs-when-trying-to-log-in">
  ## Je vois le message d’erreur &quot;Failed cross origin authentication&quot; ou &quot;No verifier returned from client&quot; dans les journaux de la console de mon navigateur lorsque j’essaie de me connecter
</div>

Accédez à [Auth0 Dashboard &gt; Applications &gt; Applications](https://manage.auth0.com/#/applications), sélectionnez votre application WordPress pour afficher ses paramètres, puis vérifiez que les **Allowed Callback URLs** et les **Allowed Origins (CORS)** correspondent à votre site WordPress.

Si vous utilisez un navigateur basé sur Chromium, assurez-vous que les cookies tiers ne sont pas désactivés. Pour en savoir plus, consultez [Authentification inter-origines](/docs/fr-CA/authenticate/login/cross-origin-authentication).

<div id="i-need-to-rerun-the-setup-wizard-but-i-dont-see-that-menu-option-anymore">
  ## J’ai besoin de relancer le Setup Wizard, mais je ne vois plus cette option de menu.
</div>

Cela signifie que l’extension est déjà configurée avec un Domain, <Tooltip tip="Client ID : valeur d’identification attribuée à votre ressource enregistrée par Auth0." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Client+ID">Client ID</Tooltip> et un <Tooltip tip="Client ID : valeur d’identification attribuée à votre ressource enregistrée par Auth0." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Client+Secret">Client Secret</Tooltip>. Exécuter le Setup Wizard une deuxième fois peut produire des résultats imprévisibles. Si vous configurez WordPress pour la première fois et souhaitez repartir à zéro avant toute connexion :

1. Allez dans **wp-admin** &gt; **Auth0** &gt; **Settings** &gt; **Basic**.
2. Supprimez le Domain et le Client ID. Faites défiler la page vers le bas, puis cliquez sur **Save Changes**.
3. Allez dans [Auth0 Dashboard &gt; Applications &gt; Applications](https://manage.auth0.com/#/applications).
4. Repérez l’Application créée par WordPress (son nom devrait correspondre à celui de votre site WordPress).
5. Sélectionnez l’Application pour afficher ses paramètres. Faites défiler la page vers le bas, puis sélectionnez **Delete Application**.
6. Allez dans [Auth0 Dashboard &gt; Authentication &gt; Database](https://manage.auth0.com/#/connections/database).
7. Repérez la connexion créée par WordPress (son nom devrait être précédé de « DB- » et correspondre à celui de votre site WordPress).
8. Sélectionnez la connexion pour afficher ses paramètres. Faites défiler la page vers le bas, puis sélectionnez **I Want To Delete This Connection.** Veuillez noter que cela supprimera la connexion ainsi que tous les utilisateurs qui y ont été créés.
9. Retournez dans WordPress. Vous verrez maintenant l’option Setup Wizard sous Auth0 dans le menu d’administration.

<div id="how-do-i-setup-passwordless-login">
  ## Comment configurer la connexion Passwordless ?
</div>

La connexion <Tooltip tip="Passwordless : forme d’authentification qui n’utilise pas de mot de passe comme premier facteur." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Passwordless">Passwordless</Tooltip> est possible sur tout site Web compatible avec Auth0 à l’aide du courriel ou des SMS. Pour que cela fonctionne sur votre site WordPress :

1. Activez **Passwordless Login** dans la section **Features** des paramètres du plugin, puis enregistrez.
2. Accédez à [Auth0 Dashboard &gt; Authentication &gt; Passwordless](https://manage.auth0.com/#/connections/passwordless).

* Pour utiliser le courriel, activez la connexion **Email** et modifiez les paramètres au besoin. Cela activera la connexion par code envoyé par courriel (les utilisateurs reçoivent un code par courriel qu’ils saisissent ensuite dans le formulaire de connexion de votre site).
* Pour utiliser un « Magic Link » (un lien envoyé par courriel qui connecte automatiquement les utilisateurs), ajoutez `{passwordlessMethod: 'code'}` au champ « Extra Settings » dans l’onglet **Advanced** des paramètres du plugin.
* Pour utiliser la connexion par SMS, activez la connexion **SMS** et suivez les étapes pour configurer un compte développeur Twilio (cela peut nécessiter un compte Twilio payant, selon l’utilisation).

Le formulaire de connexion Auth0 choisira une méthode Passwordless selon la connexion activée ci-dessus. Si les deux connexions sont actives, le courriel sera utilisé par défaut. Dans ce cas, désactivez la connexion **Email** pour afficher SMS, ou ajoutez `sms` au champ **Connections** dans l’onglet **Advanced** des paramètres du plugin.

<div id="i-have-two-accounts-for-the-same-user-in-wordpress">
  ## J’ai deux comptes pour le même utilisateur dans WordPress
</div>

Dans certaines situations, un même utilisateur peut se retrouver avec deux comptes. WordPress vous permet de fusionner des utilisateurs en supprimant l’un des comptes et en attribuant le contenu de ce compte à un autre utilisateur. Allez dans wp-admin &gt; Users, sélectionnez le compte que vous voulez supprimer, puis, dans la boîte de dialogue de confirmation, sélectionnez un autre utilisateur auquel transférer le contenu.

<div id="my-configuration-is-wrong-and-i-cant-authenticate-using-auth0-is-there-another-way-to-access-the-plugin">
  ## Ma configuration est incorrecte et je ne peux pas m’authentifier avec Auth0. Y a-t-il un autre moyen d’accéder à l’extension?
</div>

On peut accéder à l’extension à l’aide d’identifiants WordPress valides, en passant par la page de connexion WordPress habituelle et en ajoutant `?wle` à l’URL de connexion. Par exemple : `http://yourdomain.com/wp-login.php?wle`.

<div id="i-am-having-problems-when-a-user-logs-in-where-can-i-find-a-log-of-what-is-happening">
  ## J’ai des problèmes lorsqu’un utilisateur se connecte. Où puis-je trouver un log de ce qui se passe?
</div>

L’extension fournit un journal d’erreurs qui vous permet de vérifier ce qui s’est passé. Accédez-y à partir du sous-élément **Error Log** du menu de l’extension **Auth0**. Vous trouverez plus d’information dans [Auth0 Dashboard &gt; Monitoring &gt; Logs](https://manage.auth0.com/#/logs).

<div id="how-can-i-show-the-widget-or-shortcode-in-signup-mode-by-default">
  ## Comment puis-je afficher le widget ou le shortcode en mode inscription par défaut?
</div>

Vous pouvez utiliser le paramètre `Extra configuration` du widget (ou l’attribut `extra_conf` dans le shortcode) et ajouter ce JSON `{"mode":"signup" }` pour forcer l’affichage du plugin dans ce mode.

<div id="when-using-a-plugin-to-force-the-login-the-user-is-not-logged-in">
  ## Lorsqu’une extension force la connexion, l’utilisateur n’est pas connecté
</div>

Cela est généralement causé par une page mise en cache après la connexion. Vérifiez auprès de votre hébergeur quelles stratégies peuvent atténuer ce problème, ou essayez d’ajouter un paramètre d’invalidation du cache à l’URL. Pour en savoir plus, [consultez ce gist sur GitHub](https://gist.github.com/joshcanhelp/e3eb693749f0fe66aad097c3bbb3b415).

<div id="the-user-is-not-logged-in-when-using-the-force-login-plugin">
  ### L’utilisateur n’est pas connecté lorsqu’il utilise le plugin &quot;Force Login&quot;
</div>

C’est parce que l’URL de rappel n’a pas été ajoutée à l’AllowList. Essayez d’ajouter ce code au filtre `my_forcelogin_whitelist` :

```javascript lines
function wp_auth0_forcelogin_whitelist( $whitelist ) {
  if ( ! empty( $_GET['auth0'] ) ) {
    $whitelist[] = site_url($_SERVER['REQUEST_URI']);
  }
  return $whitelist;
}
add_filter('v_forcelogin_whitelist', 'wp_auth0_forcelogin_whitelist', 10, 1);
```

<div id="how-can-i-redirect-the-users-to-a-specific-url-after-login">
  ## Comment puis-je rediriger les utilisateurs vers une URL précise après la connexion?
</div>

Toutes les redirections sont vérifiées à l’aide de `wp_safe_redirect()` avant d’être effectuées. Si vous essayez de rediriger vers un domaine autre que votre domaine principal, ajoutez ce domaine à la vérification à l’aide du filtre `allowed_redirect_hosts` de WordPress. Pour en savoir plus, consultez [apply&#95;filters( &#39;allowed&#95;redirect&#95;hosts&#39;, string[] $hosts, string $host ) sur wordpress.org](https://developer.wordpress.org/reference/hooks/allowed_redirect_hosts/).

<div id="on-the-login-page">
  ### Sur la page de connexion
</div>

Cette extension tire parti des fonctionnalités de WordPress pour fonctionner harmonieusement avec les paramètres par défaut. Pour ajouter une redirection, vous pouvez ajouter le paramètre de requête `redirect_to` à l’URL lorsque vous dirigez l’utilisateur vers la page de connexion. L’extension redirigera l’utilisateur vers cette URL après une connexion réussie.

Vous pouvez aussi utiliser le paramètre **URL de redirection après connexion** dans la page des paramètres de l’extension Auth0. Cette URL sera utilisée pour rediriger l’utilisateur lorsque le paramètre `redirect_to` n’est pas fourni.

<div id="using-the-widget">
  ### Utilisation du widget
</div>

Le widget redirigera automatiquement l’utilisateur vers la même page où il se trouvait avant l’authentification. Vous pouvez modifier ce comportement à l’aide du paramètre `Redirect after login:`.

<div id="using-the-shortcode">
  ### Utilisation du shortcode
</div>

Le shortcode redirige automatiquement vers la page où se trouvait l’utilisateur avant l’authentification. Vous pouvez modifier ce comportement à l’aide de l’attribut `redirect_to`.

<div id="how-can-i-migrate-my-wordpress-users-to-auth0">
  ## Comment puis-je migrer mes utilisateurs WordPress vers Auth0 ?
</div>

La version actuelle de l’extension n’offre aucun moyen de migrer automatiquement les utilisateurs vers Auth0, mais quelques options s’offrent à vous :

* L’extension expose deux endpoints permettant de mettre votre connexion de base de données personnalisée en mode **importation vers Auth0**. Pour en savoir plus sur ces endpoints, consultez [Configurer la migration automatique à partir de votre base de données](/docs/fr-CA/manage-users/user-migration/configure-automatic-migration-from-your-database).
* Exportez vos données utilisateur dans un fichier JSON et téléversez-le pour une importation en bloc dans Auth0. Au départ, vos utilisateurs devront réinitialiser leur mot de passe lorsqu’ils se connecteront avec Auth0, car Auth0 ne peut pas déchiffrer les mots de passe WordPress pendant la migration. Pour savoir comment générer le fichier JSON, consultez [Importation en bloc d’utilisateurs](/docs/fr-CA/manage-users/user-migration/bulk-user-imports). Vous devrez ensuite téléverser le fichier à l’aide de l’<Tooltip tip="Management API : un produit permettant aux clients d’effectuer des tâches administratives." cta="Consulter le glossaire" href="/docs/fr-CA/glossary?term=Management+API">Management API</Tooltip> d’Auth0 et de son [endpoint Import Users](https://auth0.com/docs/api/management/v2/jobs/post-users-imports).
* [Utilisez ce script](https://gist.github.com/glena/b31716e3c8fe48927be2) et l’endpoint XML-RPC de WordPress pour configurer le flux de migration à l’aide d’une connexion de base de données personnalisée. Pour en savoir plus sur la configuration du flux de migration, consultez [Configurer la migration automatique à partir de votre base de données](/docs/fr-CA/manage-users/user-migration/configure-automatic-migration-from-your-database). Pour en savoir plus sur l’endpoint XML-RPC de WordPress, consultez [la prise en charge de XML-RPC sur wordpress.org](https://codex.wordpress.org/XML-RPC_Support).

<div id="the-form_title-setting-is-ignored-when-i-set-up-the-dict-setting">
  ## Le paramètre form_title est ignoré lorsque je configure le paramètre dict
</div>

En interne, le plugin utilise le paramètre dict pour modifier le titre du widget Auth0. Lorsque vous configurez le champ dict, celui-ci remplace form&#95;title.

Pour modifier form&#95;title dans ce cas, vous devez ajouter l’attribut suivant au JSON dict :

```json lines
{
    "signin": {
        "title": "The desired form title"
    }
}
```

<div id="how-can-i-modify-the-embedded-auth0-login-form">
  ## Comment puis-je modifier le formulaire de connexion Auth0 intégré ?
</div>

De nombreuses options dans l’onglet **Apparence** de la page des paramètres de l’extension permettent de modifier l’apparence et le fonctionnement du formulaire de connexion intégré à votre site (page `wp-login/php`, `shortcodes` ou `widgets`). Pour en savoir plus sur ces options, consultez [Configure Login by Auth0](/docs/fr-CA/customize/integrations/cms/wordpress-plugin/configure-login-by-auth0). Cela n’affectera pas le formulaire de connexion sur la page <Tooltip tip="Universal Login : votre application redirige vers Universal Login, hébergé sur le Authorization Server d’Auth0, afin de vérifier l’identité d’un utilisateur." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Universal+Login">Universal Login</Tooltip> hébergée par Auth0.

Il y a aussi un champ appelé « Paramètres supplémentaires » dans l’onglet **Advanced** qui accepte un objet JSON valide contenant tous les paramètres que vous souhaitez configurer. Cela remplacera toutes les modifications apportées dans l’onglet **Apparence**. Pour voir toutes les options de configuration possibles, consultez [Lock Configuration Options](/docs/fr-CA/libraries/lock/lock-configuration).

Les feuilles de style externes et les fichiers JS doivent être chargés dans votre thème à l’aide du hook `wp_enqueue_scripts` pour les `shortcodes`/`widgets` et du hook `login_enqueue_scripts` pour `wp-login.php`. Pour en savoir plus, consultez [`do_action( 'wp_enqueue_scripts' )` on wordpress.org](https://developer.wordpress.org/reference/hooks/wp_enqueue_scripts/) et [`do_action( 'login_enqueue_scripts' )` on wordpress.org](https://developer.wordpress.org/reference/hooks/login_enqueue_scripts/).

<div id="the-session-expires-too-soon">
  ## La session expire trop tôt
</div>

Le plugin Auth0 ne gère pas les sessions ; il s’appuie sur les réglages de WordPress. Par défaut, les sessions utilisateur restent actives pendant deux jours. Vous pouvez activer le paramètre `Remember users session` dans l’onglet **Avancé** des paramètres du plugin pour que les sessions puissent rester actives jusqu’à 14 heures.

<div id="how-do-i-implement-a-refresh-token">
  ## Comment puis-je mettre en place un Refresh Token ?
</div>

Nous avons ajouté des paramètres supplémentaires aux méthodes de connexion utilisées par l’extension pour permettre l’utilisation de <Tooltip tip="Refresh Token : jeton utilisé pour obtenir un nouvel Access Token sans obliger les utilisateurs à se connecter de nouveau." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Refresh+Tokens">Refresh Tokens</Tooltip>. Utilisez le filtre `auth0_auth_scope` combiné à l’action `auth0_user_login` pour ce faire.

<div id="profile-data-saved-in-wordpress-is-not-being-synced-to-the-auth0-user-account">
  ## Les données de profil enregistrées dans WordPress ne sont pas synchronisées avec le compte d’utilisateur Auth0.
</div>

Il s’agit d’une limitation actuelle de l’extension, mais nous comptons y remédier dans une prochaine version. La seule exception concerne le mot de passe de l’utilisateur. Si le mot de passe est modifié dans WordPress et qu’il respecte la politique de sécurité définie pour la connexion de base de données utilisée, il sera également mis à jour pour l’utilisateur Auth0. Nous ajouterons un message d’erreur dans une prochaine version pour interrompre le processus si le mot de passe n’est pas suffisamment fort.

<div id="how-do-i-migrate-from-social-login-with-auth0-to-login-by-auth0">
  ## Comment migrer de &quot;Social Login with Auth0&quot; vers &quot;Login by Auth0&quot; ?
</div>

Historiquement, Auth0 a maintenu deux extensions WordPress :

* [Login by Auth0](https://wordpress.org/plugins/auth0/)
* [Social Login with Auth0](https://wordpress.org/plugins/social-login-with-auth0/)

Ces deux extensions sont essentiellement identiques, mais **Social Login with Auth0** ne recevra plus aucune mise à jour après la version 3.7.0 (publiée le 13 août 2018). Migrer de **Social Login with Auth0** vers **Login by Auth0** est simple et n&#39;entraînera aucune perte de données Auth0 ou WordPress.

Passer de **Social Login with Auth0** à **Login by Auth0** mettra à jour le numéro de version affiché. Assurez-vous donc de tester ce changement d&#39;abord sur un serveur de staging ou de développement (comme vous le feriez si vous mettiez l’extension à jour dans wp-admin). De plus, il se peut que les connexions ne fonctionnent pas pendant le processus de migration. Veillez donc à utiliser une extension de mode maintenance ou à effectuer la migration en dehors des heures de pointe.

La façon la plus simple de migrer est via (S)FTP :

1. Connectez-vous au site WordPress en tant qu&#39;administrateur.
2. [Téléchargez Login by Auth0](https://downloads.wordpress.org/plugin/auth0.zip) et décompressez-le localement.
3. Désactivez l’extension **Social Login with Auth0** depuis l&#39;écran WordPress admin &gt; Plugins.
4. Connectez-vous au serveur vers lequel vous voulez migrer et accédez à `wp-content/plugins`.
5. Déplacez le dossier `social-login-with-auth0` hors du dossier des extensions afin d&#39;en sauvegarder le contenu.
6. Téléversez le nouveau dossier d’extension `auth0` dans le répertoire des extensions.
7. Activez la nouvelle extension &quot;Login by Auth0&quot; depuis l&#39;écran WordPress **Admin** &gt; **Plugins**.

Si vous ne pouvez pas accéder au site par FTP, vous pouvez aussi effectuer le processus directement depuis l&#39;administration WordPress :

1. Connectez-vous au site WordPress en tant qu&#39;administrateur.
2. Allez à **Auth0** &gt; **Import-Export Settings**.
3. Cliquez sur **Export Settings**, puis sur **Export**.
4. Désactivez l’extension **Social Login with Auth0** depuis l&#39;écran WordPress **Admin** &gt; **Plugins**.
5. Supprimez l’extension **Social Login with Auth0** et confirmez.
6. Allez à **Plugins** &gt; **Add New** et recherchez &quot;Auth0&quot;.
7. Pour l’extension **Login by Auth0** (assurez-vous de bien vérifier le nom), cliquez sur **Install Now**.
8. Lorsque l&#39;installation est terminée, cliquez sur **Activate**.
9. Vérifiez dans **Auth0** &gt; **Settings** que les paramètres précédents sont toujours présents. Sinon :

   1. Allez à **Auth0** &gt; **Import-Export Settings**.
   2. Collez le JSON des paramètres exporté précédemment et cliquez sur **Import**.
10. Supprimez complètement le fichier JSON exporté des paramètres (il contient des renseignements sensibles).

Tout devrait maintenant fonctionner comme prévu avec la nouvelle extension, et les mises à jour reprendront normalement.
