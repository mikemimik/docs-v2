Voici quelques questions courantes de dépannage. Si les éléments ci-dessous ne résolvent pas votre problème, envisagez les solutions suivantes :

* Si vous configurez le plugin pour la première fois ou si des utilisateurs ont de la difficulté à se connecter, veuillez lire [Configurer Login by Auth0](/docs/fr-CA/customize/integrations/cms/wordpress-plugin/configure-login-by-auth0).
* Si vous avez trouvé un bogue dans le code du plugin, [soumettez un issue](https://github.com/auth0/wordpress/issues) ou [créez une pull request](https://github.com/auth0/wordpress/pulls) dans GitHub.
* Si vous avez des questions sur l’utilisation d’Auth0 ou du plugin, veuillez [publier sur notre site Community](https://community.auth0.com/) ou [créer une demande dans le forum de soutien WordPress du plugin](https://wordpress.org/support/plugin/auth0/).
* Pour consulter de la documentation supplémentaire et trouver plus de réponses, [visitez notre site de soutien](https://support.auth0.com/center/s/). Les clients ayant un forfait Auth0 payant peuvent soumettre une demande de soutien pour obtenir une réponse rapide.

<div id="im-seeing-the-error-message-invalid-state-that-prevents-me-from-logging-in">
  ## Je vois le message d’erreur « Invalid state », ce qui m’empêche de me connecter
</div>

La validation de l’état est une fonctionnalité de sécurité ajoutée dans la [version 3.6.0](https://github.com/auth0/wordpress/releases/tag/3.6.0). Ce message d’erreur est généralement causé par une URL de rappel mise en cache (consultez les paramètres de votre application pour voir quelles URL de rappel ne doivent pas être mises en cache). Pour en savoir plus, consultez [Résoudre les erreurs « Invalid state » du plugin WordPress](/docs/fr-CA/customize/integrations/cms/wordpress-plugin/troubleshoot-wordpress-plugin-invalid-state-errors).

<div id="im-having-an-issue-logging-in-or-changing-emailpassword-using-a-custom-database">
  ## J’ai de la difficulté à me connecter ou à modifier l’adresse courriel/le mot de passe avec une base de données personnalisée
</div>

Pour en savoir plus sur les solutions possibles, consultez [Migration des utilisateurs dans le plugin WordPress Login by Auth0](/docs/fr-CA/customize/integrations/cms/wordpress-plugin/user-migration-in-login-by-auth0).

<div id="im-seeing-the-error-message-invalid-id-token-or-expired-id-token-that-prevents-me-from-logging-in">
  ## Je vois le message d&#39;erreur &quot;Invalid ID token&quot; ou &quot;Expired ID token&quot;, ce qui m&#39;empêche de me connecter
</div>

Cela est généralement causé par un serveur dont l&#39;heure est mal réglée. Si le message d&#39;erreur comprend &quot;used too early,&quot; cela signifie que l&#39;heure de votre serveur est réglée dans le futur. S&#39;il indique que le jeton est expiré, cela signifie que l&#39;heure du serveur est réglée trop loin dans le passé. Il est courant qu&#39;il y ait un décalage horaire entre deux serveurs. Exécutez `echo date(DateTime::ISO8601)` en PHP sur votre serveur et comparez le résultat, secondes comprises, à l&#39;heure UTC actuelle. Si l&#39;heure de votre serveur diffère de l&#39;heure UTC de plus de 60 secondes (la marge par défaut), vous devrez définir une marge plus longue pour tenir compte du décalage de l&#39;horloge de votre serveur. Vous pouvez coller le code ci-dessous dans le fichier `functions.php` de votre thème ou à tout autre endroit où il sera exécuté après le chargement du plugin et avant l&#39;exécution du hook de connexion :

```js wrap lines
add_filter( 'auth0_jwt_leeway', function( $default_leeway ) { return 120; } );
```

Cela offrirait une marge de 120 secondes. Vous devrez peut-être l’ajuster selon le décalage de l’horloge de votre serveur.

<div id="i-see-the-error-message-this-account-does-not-have-an-email-associated-that-prevents-me-from-logging-in">
  ## Je vois le message d’erreur &quot;This account does not have an email associated...&quot; qui m’empêche de me connecter
</div>

Si vous obtenez cette erreur, assurez-vous de demander une adresse courriel à chaque fournisseur dans le <Tooltip tip="Auth0 Dashboard : principal produit d’Auth0 pour configurer vos services." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip>, sous **Connections &gt; Social** (développez chaque fournisseur). Gardez à l’esprit que certains fournisseurs ne renvoient pas d’adresse courriel pour les utilisateurs (p. ex. X). Si c’est le cas, vous pouvez toujours ajouter une adresse courriel à n’importe quel utilisateur connecté dans l’Auth0 Dashboard (ou l’API). Consultez **Users &gt; Edit**.

Pour les connexions qui ne fournissent pas l’indicateur `email_verified` (certaines connexions Enterprise ne l’incluent pas) ou pour ignorer cette validation pour certaines connexions Social, ajoutez la stratégie de cette connexion dans le champ &quot;Skip Strategies&quot;. Ce champ se trouve sous le commutateur **Require Verified Email**, accessible par **wp-admin** &gt; **Auth0** &gt; **Settings** &gt; **Advanced**.

**Ce champ ne doit être utilisé qu’en cas de nécessité, car il contourne les mesures de sécurité recommandées par Auth0.**

<div id="i-see-the-error-message-there-is-a-user-with-the-same-email-that-prevents-me-from-logging-in">
  ## Je vois le message d&#39;erreur « There is a user with the same email » qui m&#39;empêche de me connecter
</div>

Cela signifie qu&#39;il y a dans WordPress un utilisateur qui a la même adresse courriel que celle utilisée pour la connexion, mais qui est associé à un autre utilisateur Auth0. Si vous êtes en train de tester l’extension ou si vous voulez plutôt associer l’utilisateur existant au nouvel utilisateur Auth0 :

1. Connectez-vous en tant qu’administrateur
2. Allez dans **wp-admin &gt; Users** et recherchez l’adresse courriel utilisée
3. Ouvrez le profil de l’utilisateur et faites défiler jusqu’en bas
4. Cliquez sur **Delete Auth0 Data** et confirmez

Si vous avez 2 comptes d’utilisateur dans Auth0 avec la même adresse courriel, ce message d’erreur continuera de s’afficher. Nous vous recommandons de [lier des comptes d’utilisateur](/docs/fr-CA/manage-users/user-accounts/user-account-linking/link-user-accounts).

<div id="i-see-the-error-message-failed-cross-origin-authentication-or-no-verifier-returned-from-client-in-my-browsers-console-logs-when-trying-to-log-in">
  ## Je vois le message d’erreur &quot;Failed cross origin authentication&quot; ou &quot;No verifier returned from client&quot; dans les logs de la console de mon navigateur lorsque j’essaie de me connecter
</div>

Dans [Auth0 Dashboard &gt; Applications &gt; Applications](https://manage.auth0.com/#/applications), sélectionnez votre application WordPress pour afficher ses paramètres, puis vérifiez que les **Allowed Callback URLs** et **Allowed Origins (CORS)** correspondent à votre site WordPress.

Si vous utilisez un navigateur basé sur Chromium, assurez-vous que les cookies tiers ne sont pas désactivés. Pour en savoir plus, consultez [Authentification inter-origine](/docs/fr-CA/authenticate/login/cross-origin-authentication).

<div id="i-need-to-rerun-the-setup-wizard-but-i-dont-see-that-menu-option-anymore">
  ## J’ai besoin de relancer le Setup Wizard, mais je ne vois plus cette option de menu.
</div>

Cela signifie que le Plugin est déjà configuré avec un Domain, un <Tooltip tip="Client ID : valeur d’identification attribuée à votre ressource enregistrée par Auth0." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Client+ID">Client ID</Tooltip> et un <Tooltip tip="Client ID : valeur d’identification attribuée à votre ressource enregistrée par Auth0." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Client+Secret">Client Secret</Tooltip>. Exécuter le Setup Wizard une deuxième fois peut produire des résultats imprévisibles. Si vous configurez WordPress pour la première fois et souhaitez repartir à zéro avant toute login :

1. Allez dans **wp-admin** &gt; **Auth0** &gt; **Settings** &gt; **Basic**.
2. Supprimez le Domain et le Client ID. Faites défiler la page vers le bas et cliquez sur **Save Changes**.
3. Allez à [Auth0 Dashboard &gt; Applications &gt; Applications](https://manage.auth0.com/#/applications).
4. Trouvez l’Application créée par WordPress (son nom devrait être celui de votre site WordPress).
5. Sélectionnez l’Application pour afficher ses paramètres. Faites défiler la page vers le bas, puis sélectionnez **Delete Application**.
6. Allez à [Auth0 Dashboard &gt; Authentication &gt; Database](https://manage.auth0.com/#/connections/database).
7. Trouvez la connexion créée par WordPress (son nom devrait être celui de votre site WordPress, précédé de « DB- »).
8. Sélectionnez la connexion pour afficher ses paramètres. Faites défiler la page vers le bas, puis sélectionnez **I Want To Delete This Connection.** Veuillez noter que cela supprimera la connexion et tous les utilisateurs qui y ont été créés.
9. Retournez dans WordPress. Vous verrez maintenant l’option Setup Wizard sous Auth0 dans le menu d’administration.

<div id="how-do-i-setup-passwordless-login">
  ## Comment configurer la connexion Passwordless?
</div>

<Tooltip tip="Passwordless : forme d’authentification qui ne repose pas sur un mot de passe comme premier facteur." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Passwordless">Passwordless</Tooltip> permet la connexion sur n’importe quel site Web compatible avec Auth0 au moyen du courriel ou des SMS. Pour que cela fonctionne sur votre site WordPress :

1. Activez **Passwordless Login** dans la vue **Features** des paramètres du Plugin, puis enregistrez.
2. Accédez à [Auth0 Dashboard &gt; Authentication &gt; Passwordless](https://manage.auth0.com/#/connections/passwordless).

* Pour utiliser le courriel, activez la connexion **Email** et modifiez les paramètres au besoin. Cela activera la connexion par code envoyé par courriel (les utilisateurs recevront un code par courriel, qu’ils devront ensuite saisir dans le formulaire de connexion de votre site).
* Pour utiliser un « magic link » (un lien envoyé par courriel qui connectera automatiquement les utilisateurs), ajoutez `{passwordlessMethod: 'code'}` au champ « Extra Settings » dans l’onglet **Advanced** des paramètres du Plugin.
* Pour utiliser la connexion par SMS, activez la connexion **SMS** et suivez les étapes pour configurer un compte développeur Twilio (cela peut nécessiter un compte Twilio payant, selon l’utilisation).

Le formulaire de connexion Auth0 sélectionnera une méthode Passwordless selon la connexion activée ci-dessus. Si les deux connexions sont actives, le courriel sera utilisé par défaut. Dans ce cas, désactivez la connexion courriel pour afficher SMS, ou ajoutez `sms` au champ **Connections** dans l’onglet **Advanced** des paramètres du Plugin.

<div id="i-have-two-accounts-for-the-same-user-in-wordpress">
  ## J’ai deux comptes pour le même utilisateur dans WordPress
</div>

Dans certaines situations, un utilisateur peut se retrouver avec deux comptes. WordPress vous permet de fusionner des utilisateurs en supprimant l’un des comptes et en attribuant le contenu de ce compte à un autre utilisateur. Allez dans wp-admin &gt; Users, sélectionnez le compte que vous voulez supprimer, puis, dans la boîte de dialogue de confirmation, sélectionnez un autre utilisateur à qui transférer le contenu.

<div id="my-configuration-is-wrong-and-i-cant-authenticate-using-auth0-is-there-another-way-to-access-the-plugin">
  ## Ma configuration est incorrecte et je ne peux pas m’authentifier avec Auth0. Y a-t-il une autre façon d’accéder au Plugin?
</div>

On peut accéder au Plugin en utilisant des identifiants WordPress valides par la page de connexion habituelle de WordPress, en ajoutant `?wle` à l’URL de connexion. Par exemple : `http://yourdomain.com/wp-login.php?wle`.

<div id="i-am-having-problems-when-a-user-logs-in-where-can-i-find-a-log-of-what-is-happening">
  ## J’ai des problèmes lorsqu’un utilisateur se connecte. Où puis-je trouver un journal de ce qui se passe?
</div>

Le plugin fournit un journal des erreurs qui vous permet de voir ce qui s’est passé. Accédez-y à partir du sous-élément **Journal des erreurs** du menu du plugin **Auth0**. Vous trouverez des renseignements supplémentaires dans [Auth0 Dashboard &gt; Monitoring &gt; Logs](https://manage.auth0.com/#/logs).

<div id="how-can-i-show-the-widget-or-shortcode-in-signup-mode-by-default">
  ## Comment puis-je afficher le widget ou le shortcode en mode d’inscription par défaut ?
</div>

Vous pouvez utiliser le paramètre `Extra configuration` du widget (ou l’attribut `extra_conf` dans le shortcode) et ajouter ce JSON `{"mode":"signup" }` pour que le plugin s’affiche dans ce mode par défaut.

<div id="when-using-a-plugin-to-force-the-login-the-user-is-not-logged-in">
  ## Lorsqu’un plugin est utilisé pour forcer la connexion, l’utilisateur n’est pas connecté
</div>

Cela est généralement dû à une page mise en cache après la connexion. Vérifiez auprès de votre hébergeur quelles stratégies peuvent atténuer ce problème, ou essayez d’ajouter un paramètre anti-cache à l’URL. Pour en savoir plus, [consultez ce gist sur GitHub](https://gist.github.com/joshcanhelp/e3eb693749f0fe66aad097c3bbb3b415).

<div id="the-user-is-not-logged-in-when-using-the-force-login-plugin">
  ### L’utilisateur n’est pas connecté lorsqu’il utilise l’extension &quot;Force Login&quot;
</div>

Cela s’explique par le fait que l’URL de rappel n’a pas été ajoutée à l’AllowList. Essayez d’ajouter ce code au filtre `my_forcelogin_whitelist` :

```javascript lines
function wp_auth0_forcelogin_whitelist( $whitelist ) {
  if ( ! empty( $_GET['auth0'] ) ) {
    $whitelist[] = site_url($_SERVER['REQUEST_URI']);
  }
  return $whitelist;
}
add_filter('v_forcelogin_whitelist', 'wp_auth0_forcelogin_whitelist', 10, 1);
```

<div id="how-can-i-redirect-the-users-to-a-specific-url-after-login">
  ## Comment puis-je rediriger les utilisateurs vers une URL précise après la connexion?
</div>

Toutes les redirections sont vérifiées à l’aide de `wp_safe_redirect()` avant d’être exécutées. Si vous essayez de rediriger vers un domaine autre que votre domaine principal, ajoutez ce domaine à la vérification à l’aide du filtre WordPress principal `allowed_redirect_hosts`. Pour en savoir plus, consultez [apply&#95;filters( &#39;allowed&#95;redirect&#95;hosts&#39;, string[] $hosts, string $host ) sur wordpress.org](https://developer.wordpress.org/reference/hooks/allowed_redirect_hosts/).

<div id="on-the-login-page">
  ### Sur la page de connexion
</div>

Cette extension tire parti des fonctionnalités de WordPress pour fonctionner sans problème avec les paramètres par défaut. Pour ajouter une redirection, vous pouvez ajouter le paramètre de requête `redirect_to` à l’URL lorsque vous dirigez l’utilisateur vers la page de connexion. L’extension redirigera l’utilisateur vers cette URL après une connexion réussie.

Vous pouvez aussi utiliser le paramètre **URL de redirection de connexion** dans la page des paramètres de l’extension Auth0. Cette URL sera utilisée pour rediriger l’utilisateur lorsque le paramètre `redirect_to` n’est pas fourni.

<div id="using-the-widget">
  ### Utilisation du widget
</div>

Le widget redirigera automatiquement l’utilisateur vers la même page où il se trouvait avant l’authentification. Vous pouvez modifier ce comportement à l’aide du paramètre `Redirect after login:`.

<div id="using-the-shortcode">
  ### Utiliser le shortcode
</div>

Le shortcode redirigera automatiquement vers la page où l’utilisateur se trouvait avant l’authentification. Vous pouvez remplacer ce comportement à l’aide de l’attribut `redirect_to`.

<div id="how-can-i-migrate-my-wordpress-users-to-auth0">
  ## Comment puis-je migrer mes utilisateurs WordPress vers Auth0 ?
</div>

La version actuelle de l’extension ne permet pas de migrer automatiquement les utilisateurs vers Auth0, mais vous avez quelques options :

* L’extension expose deux points de terminaison pour mettre votre connexion de base de données personnalisée en mode **importation vers Auth0**. Pour en savoir plus sur ces points de terminaison, consultez [Configurer la migration automatique à partir de votre base de données](/docs/fr-CA/manage-users/user-migration/configure-automatic-migration-from-your-database).
* Exportez vos données utilisateur dans un fichier JSON et téléversez-le pour une importation en bloc dans Auth0. Au départ, vos utilisateurs devront réinitialiser leur mot de passe lorsqu’ils se connecteront avec Auth0, car Auth0 ne peut pas déchiffrer les mots de passe WordPress pendant la migration. Pour savoir comment générer le fichier JSON, consultez [Importation en bloc d’utilisateurs](/docs/fr-CA/manage-users/user-migration/bulk-user-imports). Vous devrez ensuite téléverser le fichier à l’aide de la <Tooltip tip="Management API : Un produit qui permet aux clients d’effectuer des tâches administratives." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Management+API">Management API</Tooltip> d’Auth0 et de son [point de terminaison Import Users](https://auth0.com/docs/api/management/v2/jobs/post-users-imports).
* [Utilisez ce script](https://gist.github.com/glena/b31716e3c8fe48927be2) et le point de terminaison XML-RPC de WordPress pour configurer le flux de migration à l’aide d’une connexion de base de données personnalisée. Pour en savoir plus sur la configuration du flux de migration, consultez [Configurer la migration automatique à partir de votre base de données](/docs/fr-CA/manage-users/user-migration/configure-automatic-migration-from-your-database). Pour en savoir plus sur le point de terminaison XML-RPC de WordPress, consultez [la prise en charge de XML-RPC sur wordpress.org](https://codex.wordpress.org/XML-RPC_Support).

<div id="the-form_title-setting-is-ignored-when-i-set-up-the-dict-setting">
  ## Le paramètre form_title est ignoré lorsque je définis le paramètre dict
</div>

À l’interne, le plugin utilise le paramètre dict pour modifier le titre du widget Auth0. Lorsque vous définissez le champ dict, celui-ci remplace form&#95;title.

Pour modifier form&#95;title dans ce cas, vous devez ajouter l’attribut suivant au JSON dict :

```json lines
{
    "signin": {
        "title": "The desired form title"
    }
}
```

<div id="how-can-i-modify-the-embedded-auth0-login-form">
  ## Comment puis-je modifier le formulaire de connexion Auth0 intégré?
</div>

De nombreuses options dans l’onglet **Apparence** de la page des paramètres du Plugin peuvent modifier l’apparence du formulaire de connexion intégré à votre site (page `wp-login/php`, shortcodes ou widgets). Pour en savoir plus sur ces options, consultez [Configurer Login by Auth0](/docs/fr-CA/customize/integrations/cms/wordpress-plugin/configure-login-by-auth0). Cela n’affectera pas le formulaire de connexion de la page <Tooltip tip="Universal Login : votre application redirige vers Universal Login, hébergé sur l’Authorization Server d’Auth0, pour vérifier l’identité d’un utilisateur." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Universal+Login">Universal Login</Tooltip> hébergée par Auth0.

Il y a aussi un champ appelé &quot;Paramètres supplémentaires&quot; dans l’onglet **Avancé** qui accepte un objet JSON valide contenant tous les paramètres que vous souhaitez configurer. Cela remplacera toute modification effectuée dans l’onglet **Apparence**. Pour voir toutes les options de configuration possibles, consultez [Options de configuration de Lock](/docs/fr-CA/libraries/lock/lock-configuration).

Les feuilles de style externes et les fichiers JS doivent être chargés dans votre thème à l’aide du hook `wp_enqueue_scripts` pour les shortcodes/widgets et du hook `login_enqueue_scripts` pour `wp-login.php`. Pour en savoir plus, consultez [`do_action( 'wp_enqueue_scripts' )` sur wordpress.org](https://developer.wordpress.org/reference/hooks/wp_enqueue_scripts/) et [`do_action( 'login_enqueue_scripts' )` sur wordpress.org](https://developer.wordpress.org/reference/hooks/login_enqueue_scripts/).

<div id="the-session-expires-too-soon">
  ## La session expire trop tôt
</div>

Le plugin Auth0 ne gère pas les sessions ; il utilise les réglages de WordPress. Par défaut, les sessions des utilisateurs restent actives pendant deux jours. Vous pouvez activer le paramètre `Remember users session` dans l’onglet **Advanced** des réglages du plugin pour permettre aux sessions de rester actives jusqu’à 14 heures.

<div id="how-do-i-implement-a-refresh-token">
  ## Comment puis-je implémenter un Refresh Token?
</div>

Nous avons ajouté des paramètres supplémentaires aux méthodes de connexion utilisées par le plugin afin de prendre en charge les <Tooltip tip="Refresh Token : jeton utilisé pour obtenir un nouvel Access Token sans obliger les utilisateurs à se reconnecter." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Refresh+Tokens">Refresh Tokens</Tooltip>. Utilisez le filtre `auth0_auth_scope` combiné à l’action `auth0_user_login` pour ce faire.

<div id="profile-data-saved-in-wordpress-is-not-being-synced-to-the-auth0-user-account">
  ## Les données de profil enregistrées dans WordPress ne sont pas synchronisées avec le compte utilisateur Auth0.
</div>

Il s’agit d’une limitation actuelle de l’extension, mais c’est un point que nous prévoyons corriger dans une prochaine version. La seule exception concerne le mot de passe de l’utilisateur. Si le mot de passe est modifié dans WordPress et qu’il respecte la politique de sécurité définie pour la connexion de base de données utilisée, il sera aussi mis à jour pour l’utilisateur Auth0. Nous ajouterons un message d’erreur dans une prochaine version afin d’interrompre le processus si le mot de passe n’est pas assez fort.

<div id="how-do-i-migrate-from-social-login-with-auth0-to-login-by-auth0">
  ## Comment migrer de &quot;Social Login with Auth0&quot; vers &quot;Login by Auth0&quot; ?
</div>

Historiquement, Auth0 a maintenu deux extensions WordPress :

* [Login by Auth0](https://wordpress.org/plugins/auth0/)
* [Social Login with Auth0](https://wordpress.org/plugins/social-login-with-auth0/)

Ces deux extensions sont essentiellement identiques, mais **Social Login with Auth** ne recevra plus de mises à jour après la version 3.7.0 (publiée le 13 août 2018). La migration de **Social Login with Auth** vers **Login by Auth0** est simple et n’entraînera aucune perte de données dans Auth0 ou WordPress.

Le passage de **Social Login with Auth** à **Login by Auth0** mettra à jour le numéro de version affiché. Assurez-vous donc de tester ce changement d’abord sur un serveur de préproduction ou de développement (comme vous le feriez pour toute mise à jour de l’extension dans wp-admin). De plus, les connexions pourraient ne pas fonctionner pendant le processus de migration; veillez donc à utiliser une extension de mode maintenance ou à effectuer la migration pendant les heures creuses.

La façon la plus simple de migrer est d’utiliser (S)FTP :

1. Connectez-vous au site WordPress en tant qu’administrateur.
2. [Téléchargez Login by Auth0](https://downloads.wordpress.org/plugin/auth0.zip) et décompressez-le localement.
3. Désactivez l’extension **Social Login with Auth0** depuis l’écran WordPress admin &gt; Plugins.
4. Connectez-vous au serveur vers lequel vous voulez migrer et accédez à `wp-content/plugins`.
5. Déplacez le dossier `social-login-with-auth0` à l’extérieur du dossier plugins afin d’en sauvegarder le contenu.
6. Téléversez le nouveau dossier d’extension `auth0` dans le répertoire plugins.
7. Activez la nouvelle extension &quot;Login by Auth0&quot; depuis l’écran WordPress **Admin** &gt; **Plugins**.

Si vous ne pouvez pas accéder au site par FTP, vous pouvez aussi exécuter le processus directement depuis l’interface d’administration WordPress :

1. Connectez-vous au site WordPress en tant qu’administrateur.
2. Allez à **Auth0** &gt; **Paramètres d’importation-exportation**.
3. Cliquez sur **Exporter les paramètres**, puis sur **Exporter**.
4. Désactivez l’extension **Social Login with Auth0** depuis l’écran WordPress **Admin** &gt; **Plugins**.
5. Supprimez l’extension **Social Login with Auth0** et confirmez.
6. Allez à **Plugins** &gt; **Ajouter** et recherchez &quot;Auth0&quot;.
7. Pour l’extension **Login by Auth0** (assurez-vous de bien vérifier le nom), cliquez sur **Installer maintenant**.
8. Une fois l’opération terminée, cliquez sur **Activer**.
9. Vérifiez **Auth0** &gt; **Paramètres** et assurez-vous que les paramètres précédents sont toujours présents. Sinon :

   1. Allez à **Auth0** &gt; **Paramètres d’importation-exportation**.
   2. Collez le JSON des paramètres exporté précédemment et cliquez sur **Importer**.
10. Supprimez complètement le fichier JSON d’exportation des paramètres (il contient des renseignements sensibles).

Tout devrait maintenant fonctionner comme prévu avec la nouvelle extension, et les mises à jour reprendront normalement.
