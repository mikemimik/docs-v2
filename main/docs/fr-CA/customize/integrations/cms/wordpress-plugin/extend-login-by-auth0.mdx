---
description: Découvrez comment étendre le Login by Auth0 WordPress Extension avec des hooks, des filtres et des fonctions.
title: Étendre le Login by Auth0 WordPress Extension
---

Les extensions WordPress peuvent être étendues pour répondre à vos besoins particuliers à l’aide d’actions et de filtres, afin d’exécuter du code personnalisé à des points précis de l’exécution. Ce document présente les hooks existants dans l’extension Login by Auth0. Nous serons heureux d’examiner et d’approuver de nouveaux filtres et de nouvelles actions pour vous permettre d’intégrer encore davantage ce plugin. Pour en savoir plus, consultez la section Contributing dans le [`README` du dépôt GitHub de l’extension](https://github.com/auth0/wordpress?tab=readme-ov-file#contributing).

<div id="wordpress-actions">
  ## Actions de WordPress
</div>

Les actions de WordPress exécutent du code personnalisé à des étapes précises du traitement. Pour en savoir plus sur les actions de WordPress, consultez [WordPress Plugin Handbook: Actions on wordpress.org](https://developer.wordpress.org/plugins/hooks/actions/).

<div id="auth0_before_login">
  ### auth0_before_login
</div>

Cette action s’exécute dans `WP_Auth0_LoginManager` après qu’un utilisateur a été authentifié avec succès, mais avant qu’il ne soit connecté à WordPress. Elle peut servir à interrompre le processus de connexion, au besoin, à l’aide de `wp_die()` ou en lançant une exception.

<div id="auth0_user_login">
  ### auth0_user_login
</div>

Cette action s’exécute dans `WP_Auth0_LoginManager` après qu’un utilisateur a été authentifié avec succès et connecté à WordPress. Elle peut servir à définir des métadonnées précises, à envoyer des notifications ou à envoyer une requête ping à d’autres services.

<div id="wpa0_user_created">
  ### wpa0_user_created
</div>

Cette action s’exécute dans `WP_Auth0_Users`, juste après qu’un utilisateur WordPress a été créé avec succès. Elle peut servir à modifier les valeurs de l’utilisateur, à définir des métadonnées utilisateur supplémentaires ou à déclencher d’autres actions liées aux nouveaux utilisateurs.

<div id="filters">
  ## Filtres
</div>

Les filtres de WordPress exécutent eux aussi du code personnalisé à des étapes précises du traitement, mais ils renvoient toujours une valeur modifiée du même type que celle qui leur a été transmise. Pour en savoir plus sur les filtres, consultez le [WordPress Plugin Handbook: Filters sur wordpress.org](https://developer.wordpress.org/plugins/hooks/filters/).

<div id="auth0_create_user_data">
  ### auth0_create_user_data
</div>

Ce filtre peut être utilisé pour modifier le tableau de données utilisateur lors de la création d’un nouvel utilisateur dans WordPress à partir d’un profil utilisateur Auth0 reçu. Il reçoit 2 paramètres :

* `$user_data` correspond aux données utilisateur extraites du profil Auth0.
* `$userinfo` correspond au profil utilisateur Auth0.

Ce filtre doit toujours renvoyer un tableau de données qui sera transmis à la fonction principale `wp_insert_user()`.

<div id="auth0_get_wp_user">
  ### auth0_get_wp_user
</div>

Ce filtre est appelé après que l’extension a trouvé l’utilisateur associé à la connexion (en fonction du `user_id` d’Auth0) et sert à remplacer le comportement par défaut par des règles de correspondance personnalisées (par exemple, toujours faire la correspondance par courriel).

Si le filtre renvoie null, la recherche se fera par courriel. Pour en savoir plus, consultez [Intégrer à WordPress](/docs/fr-CA/customize/integrations/cms/wordpress-plugin/integrate-with-wordpress).

<div id="auth0_verify_email_page">
  ### auth0_verify_email_page
</div>

Ce filtre s’exécute dans `WP_Auth0_Email_Verification` pour modifier le code HTML affiché lorsqu’un utilisateur qui tente de se connecter doit vérifier son adresse courriel avant de pouvoir accéder au site. Notez que ce code HTML est transmis à `wp_die()`, où il est modifié avant d’être affiché (consultez la définition de `_default_wp_die_handler()` dans le noyau pour en savoir plus).

<div id="auth0_get_auto_login_connection">
  ### auth0_get_auto_login_connection
</div>

Ce filtre est utilisé dans `WP_Auth0_LoginManager` pour modifier la connexion utilisée pour le processus de connexion automatique. Le paramètre de wp-admin est récupéré, puis transmis à ce filtre.

<div id="wp_auth0_get_option">
  ### wp_auth0_get_option
</div>

Ce filtre est utilisé par les fonctions et les méthodes qui récupèrent des options pour modifier la valeur renvoyée.

<div id="auth0_migration_ws_authenticated">
  ### auth0_migration_ws_authenticated
</div>

Ce filtre est utilisé dans `WP_Auth0_Routes` pour modifier l’objet `WP_User`, qui est converti en JSON et renvoyé à Auth0 dans le cadre d’une migration d’utilisateurs.

<div id="wpa0_should_create_user">
  ### wpa0_should_create_user
</div>

Ce filtre est utilisé dans `WP_Auth0_Users` pour déterminer si un utilisateur doit être créé. La valeur initiale transmise est `TRUE`. Si `FALSE` est renvoyé pour quelque raison que ce soit, l’inscription sera refusée et l’utilisateur qui tente de s’inscrire verra un message d’erreur (`WP_Auth0_UsersRepo::create()`).

<div id="auth0_login_css">
  ### auth0_login_css
</div>

Ce filtre sert à modifier le CSS de la page de connexion, y compris le Login Widget lui-même. Ce filtre s’exécute avant que le CSS ne soit récupéré à partir de la page Settings de wp-admin.

<div id="auth0_login_form_tpl">
  ### auth0_login_form_tpl
</div>

Filtre le gabarit utilisé pour le formulaire de connexion Auth0. Cela doit renvoyer le chemin vers un fichier contenant du HTML qui remplace le contenu de `wp-content/plugins/auth0/templates/auth0-login-form.php`. Le script JS standard d’initialisation de Lock recherche un attribut ID `auth0-login-form` pour instancier le formulaire de connexion; assurez-vous donc qu’il est présent, sinon remplacez le fichier `wp-content/plugins/auth0/assets/js/lock-init.js` par le vôtre.

<div id="auth0_settings_fields">
  ### auth0_settings_fields
</div>

Ce filtre sert à modifier un champ de formulaire existant ou à en ajouter un nouveau. Il doit renvoyer un tableau `$options` modifié contenant vos changements ou ajouts. Les nouveaux champs doivent avoir un callback de champ, comme indiqué ci-dessous.

<div id="auth0_auth_scope">
  ### auth0_auth_scope
</div>

Ce filtre permet aux développeurs d’ajouter ou de modifier la portée demandée au moment de la connexion. Il peut servir à ajouter des [claims personnalisées](/docs/fr-CA/secure/tokens/json-web-tokens/create-custom-claims) ou à demander un <Tooltip tip="Jeton d’actualisation : jeton utilisé pour obtenir un nouveau jeton d’accès sans obliger les utilisateurs à se reconnecter." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Refresh+Token">jeton d’actualisation</Tooltip>.

<div id="auth0_nonce_cookie_name">
  ### auth0_nonce_cookie_name
</div>

Utilisez ce filtre pour modifier le nom du témoin servant à la validation du <Tooltip tip="Nonce : Nombre arbitraire émis une seule fois dans un protocole d’authentification pour détecter et prévenir les attaques par rejeu." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=nonce">nonce</Tooltip>. Consultez le filtre `auth0_state_cookie_name` ci-dessous pour voir un exemple.

<div id="auth0_state_cookie_name">
  ### auth0_state_cookie_name
</div>

Utilisez ce filtre pour modifier le nom du témoin utilisé pour la valeur du paramètre state. Vous pouvez ajouter un préfixe ou un suffixe, ou remplacer entièrement la chaîne. Pour en savoir plus sur le paramètre state, consultez [Prévenir les attaques et rediriger les utilisateurs à l’aide des paramètres state d’OAuth 2.0](/docs/fr-CA/secure/attack-protection/state-parameters).
Assurez-vous d’utiliser uniquement des caractères valides dans les modifications apportées :

`A <cookie-name> can be any US-ASCII characters except control characters (CTLs), spaces, or tabs. It also must not contain a separator character like the following: ( ) < > @ , ; : \ " / ? = { }.`

Pour en savoir plus sur l’en-tête de réponse HTTP `Set-Cookie`, consultez [Set-Cookie dans MDN Web Docs](https://developer.mozilla.org/en-US/docs/Web/HTTP/Reference/Headers/Set-Cookie).

<div id="auth0_settings_constant_prefix">
  ### auth0_settings_constant_prefix
</div>

Utilisez ce filtre pour modifier le préfixe de la constante utilisée pour surcharger les paramètres de l’extension. Veuillez noter que ce filtre **doit** être exécuté avant le chargement de l’extension Auth0; il doit donc se trouver dans une extension MU. Pour en savoir plus sur les extensions MU, consultez [Must Use Plugins on wordpress.org](https://developer.wordpress.org/advanced-administration/plugins/mu-plugins/).

<div id="auth0_authorize_url_params">
  ### auth0_authorize_url_params
</div>

Ce filtre permet aux développeurs d’ajuster au besoin les paramètres du point de terminaison `/authorize`. La fonction doit retourner un tableau de type dictionnaire de paramètres d’URL. Pour en savoir plus sur l’utilisation de ces paramètres, consultez [Authentication API Explorer: Login](https://auth0.com/docs/api/authentication#login).

<div id="auth0_authorize_url">
  ### auth0_authorize_url
</div>

Ce filtre permet aux développeurs de modifier l’URL complète `/authorize` avant son utilisation. La fonction doit renvoyer une URL valide sous forme de String. Pour en savoir plus sur l’utilisation de cette URL, consultez [Authentication API Explorer: Login](https://auth0.com/docs/api/authentication#login).

<div id="auth0_die_on_login_output">
  ### auth0_die_on_login_output
</div>

Ce filtre vous permet de modifier ou de remplacer le contenu HTML transmis à `wp_die()` lorsqu’une erreur survient lors de la connexion. Ce filtre n’affecte pas le contenu de la vérification du courriel (voir auth0&#95;verify&#95;email&#95;page).

<div id="auth0_coo_auth0js_url">
  ### auth0_coo_auth0js_url
</div>

Ce filtre vous permet de redéfinir l’URL CDN par défaut d’Auth0.js lors du chargement de la page de repli COO.

<div id="auth0_slo_return_to">
  ### auth0_slo_return_to
</div>

Ce filtre vous permet de redéfinir l’URL `returnTo` par défaut lors de votre déconnexion d’Auth0.

<div id="auth0_logout_url">
  ### auth0_logout_url
</div>

Ce filtre vous permet de remplacer l’URL de déconnexion d’Auth0. Pour en savoir plus sur la façon dont elle est utilisée, consultez [Déconnexion](/docs/fr-CA/authenticate/login/logout).

<div id="auth0_use_management_api_for_userinfo">
  ### auth0_use_management_api_for_userinfo
</div>

Ce filtre détermine s’il faut utiliser les données du profil utilisateur récupérées depuis la <Tooltip tip="Management API : Un produit qui permet aux clients d’effectuer des tâches administratives." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Management+API">Management API</Tooltip> lorsque vous n’utilisez pas le flux de connexion implicite. Renvoyez la valeur booléenne `true` (par défaut) pour utiliser l’API, ou `false` pour utiliser le <Tooltip tip="ID Token : Informations d’identification destinées au client lui-même, plutôt qu’à l’accès à une ressource." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=ID+token">ID token</Tooltip>.

<div id="auth0_lock_options">
  ### auth0_lock_options
</div>

Ce filtre permet de modifier les options du formulaire de connexion Lock intégré utilisé dans les shortcodes, les widgets et sur la page wp-login.php lorsque **Fonctionnalités &gt; <Tooltip tip="Universal Login : votre application est redirigée vers Universal Login, hébergé sur le Authorization Server d'Auth0, afin de vérifier l'identité de l'utilisateur." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Universal+Login">page Universal Login</Tooltip>** est désactivée.

<div id="auth0_jwt_leeway">
  ### auth0_jwt_leeway
</div>

Ce filtre vous permet d’ajuster le délai de tolérance utilisé pour valider les jetons d’identité et doit renvoyer un nombre entier de secondes.

<div id="auth0_jwt_max_age">
  ### auth0_jwt_max_age
</div>

Ce filtre vous permet d’ajuster le paramètre d’URL `max_age` envoyé dans l’URL d’autorisation.

<div id="auth0_authorize_state">
  ### auth0_authorize_state
</div>

Ce filtre vous permet de filtrer les données de state avant qu’elles ne soient encodées et utilisées pour la connexion. Ces données seront vérifiées après une connexion réussie, puis fournies telles quelles pour utilisation.
