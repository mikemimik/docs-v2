<div id="step-2-secure-and-deploy-the-amazon-api-gateway">
  ## Étape 2 - Sécuriser et déployer Amazon API Gateway
</div>

Maintenant que votre API fonctionne, vous devez la sécuriser. Dans cette étape, vous allez :

* Sécuriser l’API de mise à jour afin de limiter l’accès aux utilisateurs authentifiés qui ont un rôle AWS IAM précis ;
* Configurer la délégation Auth0 pour utiliser les capacités de fédération AWS IAM ;
* Obtenir un <Tooltip tip="Jeton d’accès : information d’autorisation, sous la forme d’une chaîne opaque ou d’un JWT, utilisée pour accéder à une API." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Access+Token">jeton d’accès</Tooltip> AWS qui utilise le rôle AWS IAM.

Une fois votre API sécurisée, vous créerez une application monopage (SPA) sans serveur. La SPA s’appuiera sur la fédération d’identité pour déterminer quels utilisateurs sont autorisés à y accéder. En combinant AWS IAM Integration for AWS Gateway API, AWS IAM Identity Federation for <Tooltip tip="Security Assertion Markup Language (SAML) : protocole normalisé permettant à deux parties d’échanger des renseignements d’authentification sans mot de passe." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=SAML">SAML</Tooltip> et la délégation Auth0 for AWS, vous pouvez permettre à des utilisateurs issus de nombreuses sources différentes, y compris des fournisseurs d’identité sociale ou des connexions d’entreprise, d’accéder à vos API. Le schéma suivant illustre un exemple de flux utilisant un <Tooltip tip="Fournisseur d’identité (IdP) : service qui stocke et gère des identités numériques." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Identity+Provider">fournisseur d’identité</Tooltip> basé sur SAML ainsi qu’Auth0 SAML Federation and Delegation for AWS.

Vous verrez deux façons de mettre en œuvre ce flux :

1. Utiliser la délégation Auth0 avec AWS IAM ;
2. Ajouter un jeton d’identité pour transmettre l’identité à la fonction Lambda.

La délégation vous permet d’obtenir facilement des jetons d’AWS pour accéder aux services AWS dans votre application.

<div id="ways-to-secure-the-amazon-api-gateway">
  ### Façons de sécuriser Amazon API Gateway
</div>

AWS API Gateway offre plusieurs méthodes pour sécuriser vos API :

1. des clés API;
2. IAM;
3. [Amazon Cognito](/docs/fr-CA/customize/integrations/aws/aws-api-gateway-cognito).

L’utilisation de clés API convient généralement aux interactions de service à service, comme illustré ci-dessous. Cependant, cette approche présente plusieurs inconvénients :

* Il est risqué de stocker dans l’application un secret à longue durée de vie (les applications sont plus faciles à compromettre) ;
* La création d’un framework pour émettre et gérer des clés API exige une mise en œuvre sécurisée, qui peut être difficile à développer.

Cette section du tutoriel utilisera des rôles IAM et des politiques pour sécuriser votre API dans API Gateway, mais vous pouvez aussi choisir de le faire à l’aide de User Pools dans Amazon Cognito. Pour consulter des instructions détaillées sur la sécurisation de votre API AWS, lisez [Secure AWS API Gateway Using Cognito](/docs/fr-CA/customize/integrations/aws/aws-api-gateway-cognito). Pour plus d’information sur l’utilisation des rôles IAM et des politiques, consultez l’article d’Amazon, [Control access to an API with IAM permissions](http://docs.aws.amazon.com/apigateway/latest/developerguide/permissions.html). Pour en savoir plus sur les User Pools dans Cognito, consultez [Amazon Cognit user pool](http://docs.aws.amazon.com/cognito/latest/developerguide/cognito-user-identity-pools.html).

<div id="step-1-configure-iam-and-auth0-for-saml-integration-with-the-api-gateway">
  ### Étape 1. Configurer IAM et Auth0 pour l’intégration SAML avec l’API Gateway
</div>

Vous pouvez spécifier un rôle AWS IAM pour le jeton SAML que vous échangez contre un jeton AWS. Selon les permissions accordées à ce rôle IAM (définies au moyen de votre fournisseur d’identité), le jeton reçu aura ces mêmes permissions. En émettant différents jetons SAML, chacun associé à son propre rôle AWS IAM, vous pouvez contrôler les niveaux d’accès de vos utilisateurs.

Par exemple, l’IdP pourrait spécifier le rôle IAM en fonction de l’appartenance à un groupe (par exemple, un administrateur dans Active Directory) ou de la source d’authentification (par exemple, une connexion de base de données ou un fournisseur social comme Facebook). Cette approche vous permet de différencier l’accès de vos utilisateurs aux méthodes de votre Amazon API Gateway lorsqu’elles sont sécurisées à l’aide d’AWS IAM.

<div id="configure-auth0">
  #### Configurer Auth0
</div>

Connectez-vous à votre compte Auth0. Vous serez redirigé vers le Management Dashboard. Cliquez sur **+ New Application**, situé dans le coin supérieur droit de la page.

Nommez votre nouvelle application AWS API Gateway et indiquez que cette Application sera une application monopage. Cliquez sur **Create**.

Accédez à l’onglet Addons de votre nouvelle Application. À l’aide du bouton bascule approprié, activez Amazon Web Services. Cela active AWS Delegation.

<div id="configure-aws">
  #### Configurer AWS
</div>

Suivez le tutoriel [Comment configurer AWS pour l’authentification déléguée](/docs/fr-CA/customize/integrations/aws/how-to-set-up-aws-for-delegated-authentication) pour configurer AWS pour l’accès délégué, qui utilise SAML. Quelques points à noter :

* Suivez les [instructions ci-dessous](#setting-the-permissions-policy-on-your-iws-iam-role) pour associer la politique d’autorisations à votre rôle, plutôt que celles du tutoriel correspondant ;
* Nommez le fournisseur SAML que vous créez `auth0` ;
* Nommez le rôle AWS IAM `auth0-api-role`.

<div id="set-the-permissions-policy-on-your-iws-iam-role">
  ##### Définissez la politique d’autorisations de votre rôle IAM IWS
</div>

Une fois le rôle AWS IAM configuré, vous ajouterez une politique à `auth0-api-role` qui vous permettra d’exécuter les méthodes de votre API Gateway. Pour en savoir plus sur ce processus, consultez [User Access Permissions for Amazon API Gateway](http://docs.aws.amazon.com/apigateway/latest/developerguide/permissions.html).

<div id="getting-the-gateway-api-arn">
  #### Récupération de l’ARN de Gateway API
</div>

Avant de commencer, vous aurez besoin de l’ARN de votre Gateway API :

1. Accédez à [Amazon API Gateway Console](https://console.aws.amazon.com/apigateway) et ouvrez une session.
2. Sélectionnez l’API appropriée.
3. Cliquez sur l’une des Methods associées à l’API pour afficher la page Method Execution.
4. Sur la page Method Execution, la zone Method Request, dans le coin supérieur gauche, affiche l’**ARN** de l’API, mais il inclut le nom de la Method :

`arn:aws:execute-api:us-east-2:484857107747:97i1dwv0j4/*/POST/`

Vous devrez retirer le nom de la méthode pour obtenir l’ARN de base de l’API :

`arn:aws:execute-api:us-east-2:484857107747:97i1dwv0j4/*`

Le wildcard (`*`) dans l’ARN ci-dessus accorde les permissions à votre API pour toutes les stages, mais vous pouvez déployer différentes stages individuellement (par exemple, développement, puis test, puis production).

Sélectionnez le rôle `auth0-api-role` que vous venez de créer pour ouvrir sa page Summary.

Développez **Inline Policies**, puis cliquez sur **click here**.

Sélectionnez **Custom Policy**, puis cliquez sur **Select**.

Modifiez votre document de policy. Vous pouvez donner au **Policy Name** le nom de votre choix, mais nous vous suggérons quelque chose comme `api-gateway-policy`. Pour activer l’accès aux méthodes de l’API pour ce rôle, appliquez la policy suivante après avoir remplacé l’ARN par celui de votre API.

```json lines
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Effect": "Allow",
            "Action": [
                "execute-api:*"
            ],
            "Resource": [
                "arn:[{yourArn}]"
            ]
        }
    ]
}
```

Cliquez sur **Appliquer la politique**.

Puisque API Gateway assumera ce rôle au nom de l’utilisateur, la politique d’approbation doit autoriser cette action. Pour ce faire, modifiez les relations d’approbation du rôle en accédant à cet onglet sur la page Summary du rôle.

La relation d’approbation finale devrait ressembler à ce qui suit :

export const codeExample = `{

À cette étape, vous devrez configurer les paramètres d’autorisation dans [API Gateway](https://console.aws.amazon.com/apigateway).

Dans la vue **Resources**, sélectionnez la méthode POST sous `/pets`.

Cliquez sur le lien **Method Request**.

Cliquez sur l’icône de modification à côté de **Authorization Type**, puis sélectionnez AWS&#95;IAM. Cliquez ensuite sur le **Check Button** à côté du champ pour enregistrer ce paramètre.

<div id="step-2-set-up-cors-and-deploy-the-api">
  ### Étape 2. Configurer CORS et déployer l’API
</div>

Notre application monopage (SPA) accédera à des méthodes de Web API à partir d’un domaine différent de celui de la page. Le paramètre Cross-Origin Resource Sharing doit autoriser explicitement cette action pour que le navigateur permette l’accès à AWS API Gateway. En général, le navigateur enverra d’abord une requête `OPTIONS` pour vérifier quelles actions le site autorise.

Sélectionnez `/pets` sous Resources, puis cliquez sur **Create Method**. Dans la liste déroulante, sélectionnez **OPTIONS**, puis cliquez sur la **coche** pour enregistrer le paramètre.

La méthode `OPTIONS` est utilisée par le navigateur pour obtenir les en-têtes HTTP nécessaires, mais la fonction doit recevoir des instructions supplémentaires. Sous l’écran `OPTIONS` Setup, définissez les variables/paramètres suivants :

* **Integration Type**: Lambda Function;
* **Use Lambda Proxy Integration**: laissez décoché;
* **Lambda Region**: sélectionnez votre région;
* **Lambda Function**: NoOp.

Cliquez sur **Save**. Dans la fenêtre contextuelle suivante, accordez à votre fonction Lambda les autorisations dont elle a besoin.

Vous serez ensuite automatiquement redirigé vers la page `OPTIONS` Method Execution. Ouvrez la page Method Response.

Développez la section **200** située sous la barre HTTP Status et ajoutez les en-têtes de réponse suivants :

* Access-Control-Allow-Headers;
* Access-Control-Allow-Methods;
* Access-Control-Allow-Origin.

Ensuite, mappez les valeurs appropriées à chacun des en-têtes de réponse. Après être revenu à la page Method Execution, cliquez sur **Integration Response**. Après avoir développé la ligne associée à la réponse de méthode **200**, développez **Header Mappings**, puis appliquez les mappages suivants :

* Access-Control-Allow-Headers: `'Content-Type,X-Amz-Date,Authorization,x-api-key,x-amz-security-token'`;
* Access-Control-Allow-Origin: `'*'`
* Access-Control-Allow-Methods: `'POST, GET, OPTIONS'`

Enfin, répétez les étapes ci-dessus pour activer CORS pour les méthodes POST et GET. Toutefois, pour ces deux méthodes, vous ajouterez un en-tête, Access-Control-Allow-Origin, et sa valeur devra être définie à `'*'`.

<div id="deploy-the-api">
  ### Déployer l’API
</div>

Retournez à la vue **Resources** de votre API. Cliquez sur **Actions**, puis sélectionnez **DEPLOY API**.

Sélectionnez **New Stage** pour **Deploy State**, puis nommez l’étape `Test`. Cliquez sur le bouton **Deploy**.

Sur la page de résultat, accédez à **SDK Generation**. Sélectionnez JavaScript comme **Platform**. Cliquez sur le bouton **Generate SDK**.

Enregistrez le fichier ZIP téléchargé pour l’utiliser plus tard.
