<div id="step-5-use-identity-tokens-to-flow-identity">
  ## Étape 5 - Utiliser des jetons d’identité pour transmettre l’identité
</div>

Dans cette dernière étape, vous allez :

* transmettre l’identité au service en lui envoyant votre <Tooltip tip="OpenID : norme ouverte d’authentification qui permet aux applications de vérifier l’identité des utilisateurs sans recueillir ni stocker leurs renseignements de connexion." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=OpenID">OpenID</Tooltip> <Tooltip tip="OpenID : norme ouverte d’authentification qui permet aux applications de vérifier l’identité des utilisateurs sans recueillir ni stocker leurs renseignements de connexion." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=JSON+Web+Token">JSON Web Token</Tooltip> (JWT) ;
* valider le jeton ;
* extraire les renseignements du profil pour associer un acheteur à un animal de compagnie.

<div id="use-an-identity-token">
  ## Utiliser un jeton d’identité
</div>

Vous pouvez utiliser votre fonction Lambda pour traiter et obtenir des renseignements sur l’utilisateur. Par exemple, pendant une transaction d’achat, vous avez récupéré le nom d’utilisateur dans le profil renvoyé avec le <Tooltip tip="ID Token : informations d’identification destinées au client lui-même, plutôt qu’à l’accès à une ressource." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=identity+token">jeton d’identité</Tooltip>. Cependant, vous pouvez aussi choisir d’intégrer directement les renseignements de l’utilisateur au jeton lui-même, qui est un JSON Web Token (JWT).

Les avantages de l’utilisation des JWT sont les suivants :

1. Vérifier l’authenticité du JWT;
2. Avoir la certitude que l’utilisateur appelant est authentifié (au lieu de s’appuyer sur un paramètre en texte brut qui pourrait avoir été falsifié).

De plus, vous pouvez utiliser le JWT pour l’autorisation, ce qui vous permet de contourner l’intégration IAM avec Amazon API Gateway. Veuillez toutefois noter que l’utilisation d’API Gateway pour l’autorisation vous permet d’interrompre l’appel d’API avant l’invocation de votre fonction Lambda.

<div id="add-information-to-the-jwt">
  ### Ajouter de l’information au JWT
</div>

Il existe plusieurs façons d’ajouter des renseignements sur l’utilisateur au JWT. L’exemple suivant ajoute l’adresse courriel de l’utilisateur au JWT, mais les mêmes principes s’appliquent à d’autres données utilisateur.

<div id="use-rules">
  #### Utiliser les Rules
</div>

Une façon d’ajouter l’adresse courriel d’un utilisateur au JWT est d’utiliser une [Rule](/docs/fr-CA/customize/rules). C’est une bonne façon de s’assurer que cette valeur est toujours présente dans le JWT lorsqu’un utilisateur s’authentifie.

Dans `login.js`, vous pouvez voir ce scope parmi les paramètres transmis à `auth.signin` :

```js lines
$scope.login = function() {
    var params = {
        authParams: {
          scope: 'openid email'
        }
      };

    auth.signin(params, function(profile, token) {
      ...
    }
  }
```

Bien que vous puissiez inclure le profil complet de l’utilisateur dans le JWT, il est préférable de n’y inclure que ce qui est nécessaire, puisque le JWT est généralement transmis avec chaque request.

<div id="validate-the-jwt-token">
  ## Valider le token JWT
</div>

Comme la console AWS Lambda n’a accès qu’à un nombre limité de modules Node pouvant être utilisés lorsque vous saisissez votre code Node.js dans la console du navigateur, vous devrez inclure des modules supplémentaires et téléverser la fonction Lambda sous forme de package pour traiter le jeton d’identité.

Pour plus de détails, consultez [Création de packages de déploiement avec Node.js](http://docs.aws.amazon.com/lambda/latest/dg/nodejs-create-deployment-pkg.html) et [Téléversement de packages de déploiement et tests](http://docs.aws.amazon.com/lambda/latest/dg/walkthrough-s3-events-adminuser-create-test-function-upload-zip-test.html).

Le projet de départ suivant contient le code dont vous aurez besoin pour votre fonction AWS Lambda mise à jour.

```js wrap lines
<%= include('../../../_includes/_package', { org: 'auth0', repo: 'auth0-aws', path: 'examples/api-gateway/lambda' }) %>
```

Vous verrez deux fichiers JavaScript personnalisés dans le projet de départ :

* `index.js` : contient votre code principal ;
* `auth0-variables` : contient le code que vous devez mettre à jour.

En plus des fichiers personnalisés, il y a un fichier Node.js `package.json` standard.

Le code ajoute la fonctionnalité nécessaire pour extraire des informations du JWT et le valider. Par défaut, Auth0 utilise une clé symétrique pour signer le JWT, mais vous pouvez aussi choisir d’utiliser des clés asymétriques (si vous devez permettre à un tiers de valider votre token, vous devriez utiliser une clé asymétrique et ne partager que votre clé publique).

Pour en savoir plus sur la vérification des tokens, consultez [Identity Protocols Supported by Auth0](/docs/fr-CA/authenticate/protocols).

Mettez à jour `auth0-variables.js` avec votre clé secrète, que vous trouverez dans l’onglet Settings de votre Application dans l’<Tooltip tip="Auth0 Dashboard : principal produit d’Auth0 pour configurer vos services." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip> :

```javascript lines
var env={};
env.AUTH0_SECRET='{yourAuth0Secret}';
module.exports = env;
```

Exécutez **npm install** dans le répertoire où se trouvent vos fichiers, compressez le contenu dans un fichier ZIP (`index.js` doit se trouver à la racine du fichier ZIP), puis téléversez-le pour qu’il soit utilisé par la fonction Lambda `PurchasePet`. Si vous faites le test, vous devriez voir une erreur d’autorisation, puisque le JWT ne se trouve pas dans le corps du message.

Jetez un coup d’œil à la logique dans `index.js`. Vers la ligne 60, vous verrez le code qui valide le token et extrait les informations décodées contenant les renseignements d’identité utilisés par la logique d’achat :

```js lines
if(event.authToken) {
     jwt.verify(event.authToken, secret, function(err, decoded) {
         if(err) {

         } else if(!decoded.email)

           dynamo.getItem({TableName:"Pets", Key:{username:"default"}}, readcb);
         }
     });
  } else {
     console.log('invalid authorization token', event.authToken);
     context.done('authorization failure', null);
  }
    ...
```

<div id="extract-profile-information-to-assign-a-buyer">
  ### Extraire les informations du profil pour attribuer un acheteur
</div>

La dernière étape consiste à transmettre le JWT à la méthode utilisée par le client du navigateur.

La méthode standard utilise un en-tête `Authorization` avec un bearer token, et vous pouvez utiliser cette méthode en désactivant l’autorisation IAM et en vous fiant uniquement au jeton OpenID pour l’autorisation (vous devrez aussi faire le mapping de l’en-tête `Authorization` dans les données d’événement transmises à la fonction AWS Lambda).

Toutefois, si vous utilisez IAM, API Gateway utilise l’en-tête `Authorization` pour contenir la signature du message, et vous compromettrez l’authentification en insérant le JWT dans cet en-tête. Pour ce faire, vous pouvez soit :

* Ajouter un en-tête personnalisé pour le JWT ;
* Mettre l’en-tête personnalisé dans le corps du message.

Si vous choisissez d’utiliser un en-tête personnalisé, vous devrez aussi effectuer du mapping pour la requête d’intégration de la méthode POST pour `pets/purchase`.

Pour garder le processus de validation simple, transmettez le JWT dans le corps de la requête POST à la fonction AWS Lambda. Pour ce faire, mettez à jour la méthode `buyPet` dans `home.js` en retirant le `userName` du corps et en ajoutant `authToken` comme suit :

```javascript lines
function buyPet(user, id) {
    var apigClient = getSecureApiClient();
    var body = {
      petId:id,
      authToken: store.get('token')
    };

    apigClient.petsPurchasePost({}, body)
      .then(function(response) {
        console.log(response);
        $scope.pets = response.data;
        $scope.$apply();
      }).catch(function (response) {
        alert('buy pets failed');
        showError(response);
    });
}
```

Téléversez votre code dans votre bucket S3 et essayez d’acheter un animal de compagnie. Vous verrez le courriel de l’acheteur dans le message qui s’affiche.

Si vous obtenez des erreurs, vérifiez de nouveau que vous avez correctement configuré votre clé secrète. Un outil utile pour vérifier les problèmes de décodage de votre token est [jwt.io](http://jwt.io/).

  ## Résumé

Dans ce tutoriel, vous avez :

* Utilisé un JWT pour fournir un contexte d’autorisation supplémentaire et transmettre les renseignements d’identité à la fonction Lambda appropriée.
