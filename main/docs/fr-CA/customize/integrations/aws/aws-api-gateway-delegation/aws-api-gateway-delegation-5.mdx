<div id="step-5-use-identity-tokens-to-flow-identity">
  ## Étape 5 - Utiliser des jetons d’identité pour transmettre l’identité
</div>

Dans cette dernière étape, vous allez :

* Transmettre l’identité au service en lui transmettant votre <Tooltip tip="OpenID: Norme ouverte d’authentification qui permet aux applications de vérifier l’identité des utilisateurs sans recueillir ni stocker les renseignements de connexion." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=OpenID">OpenID</Tooltip> <Tooltip tip="OpenID: Norme ouverte d’authentification qui permet aux applications de vérifier l’identité des utilisateurs sans recueillir ni stocker les renseignements de connexion." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=JSON+Web+Token">JSON Web Token</Tooltip> (JWT) ;
* Valider le jeton ;
* Extraire les renseignements du profil afin d’attribuer un acheteur à un animal de compagnie.

<div id="use-an-identity-token">
  ## Utiliser un jeton d’identité
</div>

Vous pouvez utiliser votre fonction Lambda pour traiter des renseignements sur l’utilisateur et en obtenir. Par exemple, au cours d’une transaction d’achat, vous avez récupéré le username à partir du profil renvoyé avec le <Tooltip tip="ID Token : informations d’identification destinées au client lui-même, plutôt qu’à l’accès à une ressource." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=identity+token">jeton d’identité</Tooltip>. Cependant, vous pouvez aussi choisir d’intégrer les renseignements de l’utilisateur directement dans l’identité elle-même, c’est-à-dire un JSON Web Token (JWT).

Les avantages de l’utilisation des JWT sont les suivants :

1. Vérifier l’authenticité du JWT;
2. Vous assurer que l’utilisateur appelant est authentifié (au lieu de vous fier à un paramètre en texte brut qui pourrait avoir été falsifié).

De plus, vous pouvez utiliser le JWT pour l’autorisation, ce qui vous permet de contourner l’intégration IAM avec Amazon API Gateway. Veuillez toutefois noter que l’utilisation d’API Gateway pour l’autorisation vous permet d’interrompre l’appel d’API avant l’invocation de votre fonction Lambda.

<div id="add-information-to-the-jwt">
  ### Ajouter de l’information au JWT
</div>

Il existe plusieurs façons d’ajouter des renseignements sur un utilisateur au JWT. L’exemple suivant ajoute l’adresse courriel de l’utilisateur au JWT, mais les mêmes principes s’appliquent à d’autres données utilisateur.

<div id="use-rules">
  #### Utiliser Rules
</div>

Une façon d’ajouter l’adresse courriel d’un utilisateur au JWT consiste à utiliser une [rule](/docs/fr-CA/customize/rules). C’est une bonne approche si vous voulez vous assurer que cette valeur sera toujours disponible dans le JWT pour un utilisateur en cours d’authentification.

Dans `login.js`, vous pouvez voir ce scope indiqué dans les paramètres transmis à `auth.signin` :

```js lines
$scope.login = function() {
    var params = {
        authParams: {
          scope: 'openid email'
        }
      };

    auth.signin(params, function(profile, token) {
      ...
    }
  }
```

Bien que vous puissiez inclure le profil complet de l’utilisateur dans le JWT, il est préférable d’y inclure seulement ce qui est nécessaire, puisque le JWT est généralement transmis avec chaque request.

<div id="validate-the-jwt-token">
  ## Valider le jeton JWT
</div>

Comme la console AWS Lambda ne donne accès qu’à un nombre limité de modules Node.js lorsque vous entrez votre code Node.js dans la console du navigateur, vous devrez inclure des modules supplémentaires et téléverser la fonction Lambda sous forme de package afin de traiter le jeton d’identité.

Pour plus de détails, consultez [Créer des packages de déploiement à l’aide de Node.js](http://docs.aws.amazon.com/lambda/latest/dg/nodejs-create-deployment-pkg.html) et [Téléverser des packages de déploiement et effectuer des tests](http://docs.aws.amazon.com/lambda/latest/dg/walkthrough-s3-events-adminuser-create-test-function-upload-zip-test.html).

Le projet de départ suivant contient le code dont vous aurez besoin pour mettre à jour votre fonction AWS Lambda.

```js wrap lines
<%= include('../../../_includes/_package', { org: 'auth0', repo: 'auth0-aws', path: 'examples/api-gateway/lambda' }) %>
```

Vous verrez deux fichiers JavaScript personnalisés dans le projet de démarrage :

* `index.js` : contient votre code principal;
* `auth0-variables` : contient le code que vous devez mettre à jour.

En plus des fichiers personnalisés, il y a un fichier `package.json` Node.js standard.

Le code ajoute des fonctionnalités permettant d’extraire des renseignements du JWT et de le valider. Par défaut, Auth0 utilise une clé symétrique pour signer le JWT, mais vous pouvez choisir d’utiliser des clés asymétriques (si vous devez permettre la validation de votre token par une tierce partie, vous devriez utiliser une clé asymétrique et ne partager que votre clé publique).

Pour en savoir plus sur la vérification des tokens, consultez [Protocoles d’identité pris en charge par Auth0](/docs/fr-CA/authenticate/protocols).

Mettez à jour `auth0-variables.js` avec votre clé secrète, que vous trouverez dans l’onglet Settings de votre Application dans le <Tooltip tip="Auth0 Dashboard : le principal produit d’Auth0 pour configurer vos services." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip> :

```javascript lines
var env={};
env.AUTH0_SECRET='{yourAuth0Secret}';
module.exports = env;
```

Exécutez **npm install** dans le répertoire où se trouvent vos fichiers, compressez le contenu en fichier zip (`index.js` doit se trouver à la racine du zip), puis téléversez-le pour qu’il soit utilisé par la fonction Lambda `PurchasePet`. Si vous faites le test, vous devriez voir une erreur d’autorisation, puisque le JWT ne se trouve pas dans le corps du message.

Jetez un coup d’œil à la logique dans `index.js`. Vous verrez, vers la ligne 60, du code qui valide le token et extrait les informations décodées contenant les renseignements d’identité utilisés par la logique d’achat :

```js lines
if(event.authToken) {
     jwt.verify(event.authToken, secret, function(err, decoded) {
         if(err) {
           console.log('failed jwt verify: ', err, 'auth: ', event.authToken);
           context.done('authorization failure', null);
         } else if(!decoded.email)
         {
           console.log('err, email missing in jwt', 'jwt: ', decoded);
           context.done('authorization failure', null);
         } else {
           userEmail = decoded.email;
           console.log('authorized, petId', petId, 'userEmail:', userEmail);
           dynamo.getItem({TableName:"Pets", Key:{username:"default"}}, readcb);
         }
     });
  } else {
     console.log('invalid authorization token', event.authToken);
     context.done('authorization failure', null);
  }
    ...
```

<div id="extract-profile-information-to-assign-a-buyer">
  ### Extraire les renseignements du profil pour attribuer un acheteur
</div>

La dernière étape consiste à transmettre le JWT à la méthode utilisée par le client de navigateur.

La méthode standard comprend un en-tête `Authorization` sous forme de bearer token, et vous pouvez utiliser cette méthode en désactivant l’autorisation IAM et en vous fiant uniquement au jeton OpenID pour l’autorisation (vous devrez aussi mapper l’en-tête `Authorization` dans les données d’événement transmises à la fonction AWS Lambda).

Si, toutefois, vous utilisez IAM, AWS API Gateway utilise l’en-tête `Authorization` pour contenir la signature du message, et vous compromettrez l’authentication en insérant le JWT dans cet en-tête. Pour ce faire, vous pouvez soit :

* Ajouter un en-tête personnalisé pour le JWT;
* Mettre l’en-tête personnalisé dans le corps du message.

Si vous choisissez d’utiliser un en-tête personnalisé, vous devrez aussi effectuer du mappage pour l’Integration Request de la méthode POST pour `pets/purchase`.

Pour simplifier le processus de validation, transmettez le JWT dans le corps du POST à la fonction AWS Lambda. Pour ce faire, mettez à jour la méthode `buyPet` dans `home.js` en retirant `userName` du corps et en ajoutant `authToken` comme suit :

```javascript lines
function buyPet(user, id) {
    var apigClient = getSecureApiClient();
    var body = {
      petId:id,
      authToken: store.get('token')
    };

    apigClient.petsPurchasePost({}, body)
      .then(function(response) {
        console.log(response);
        $scope.pets = response.data;
        $scope.$apply();
      }).catch(function (response) {
        alert('buy pets failed');
        showError(response);
    });
}
```

Téléversez votre code dans votre bucket S3 et essayez d’acheter un animal de compagnie. Vous verrez le courriel de l’acheteur dans le message qui s’affiche.

Si vous obtenez des erreurs, assurez-vous d’avoir bien configuré votre clé secrète. Un outil utile pour vérifier les problèmes liés au décodage de votre jeton est [jwt.io](http://jwt.io/).

* Utilisé un JWT pour fournir un contexte d’autorisation supplémentaire et transmettre les renseignements d’identité à la fonction Lambda appropriée.
