<div id="step-5-use-identity-tokens-to-flow-identity">
  ## Étape 5 - Utiliser des jetons d’identité pour transmettre l’identité
</div>

Dans cette dernière étape, vous allez :

* Transmettre l’identité au service en lui transmettant votre <Tooltip tip="OpenID : norme ouverte d’authentification qui permet aux applications de vérifier l’identité des utilisateurs sans recueillir ni stocker les renseignements de connexion." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=OpenID">OpenID</Tooltip> <Tooltip tip="OpenID : norme ouverte d’authentification qui permet aux applications de vérifier l’identité des utilisateurs sans recueillir ni stocker les renseignements de connexion." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=JSON+Web+Token">JSON Web Token</Tooltip> (JWT) ;
* Valider le jeton ;
* Extraire les renseignements du profil pour associer un acheteur à un animal de compagnie.

<div id="use-an-identity-token">
  ## Utiliser un jeton d’identité
</div>

Vous pouvez utiliser votre fonction Lambda pour traiter des renseignements sur l’utilisateur et en obtenir. Par exemple, pendant une transaction d’achat, vous avez récupéré le nom d’utilisateur dans le profil renvoyé avec le <Tooltip tip="ID Token : informations d’identification destinées au client lui-même, plutôt qu’à l’accès à une ressource." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=identity+token">jeton d’identité</Tooltip>. Toutefois, vous pouvez aussi choisir d’intégrer les renseignements de l’utilisateur directement à l’identité elle-même, qui est un JSON Web Token (JWT).

L’utilisation de JWT présente les avantages suivants :

1. Vérifier l’authenticité du JWT;
2. Avoir la certitude que l’utilisateur appelant est authentifié (au lieu de s’appuyer sur un paramètre en texte brut qui pourrait avoir été altéré).

De plus, vous pouvez utiliser le JWT pour l’autorisation, ce qui vous permet de contourner l’intégration IAM avec Amazon API Gateway. Veuillez noter toutefois que l’utilisation d’Amazon API Gateway pour l’autorisation vous permet d’interrompre l’appel d’API avant l’invocation de votre fonction Lambda.

<div id="add-information-to-the-jwt">
  ### Ajouter des informations au JWT
</div>

Il existe plusieurs façons d’ajouter des informations sur l’utilisateur au JWT. L’exemple suivant ajoute l’adresse courriel de l’utilisateur au JWT, mais les mêmes principes s’appliquent à d’autres données utilisateur.

<div id="use-rules">
  #### Utiliser les Rules
</div>

Une façon d’ajouter l’adresse courriel d’un utilisateur au JWT consiste à utiliser une [règle](/docs/fr-CA/customize/rules). C’est une bonne approche si vous voulez vous assurer que cette valeur est toujours présente dans le JWT pour un utilisateur en cours d’authentification.

Dans `login.js`, vous pouvez voir ce scope indiqué dans les paramètres transmis à `auth.signin` :

```js lines
$scope.login = function() {
    var params = {
        authParams: {
          scope: 'openid email'
        }
      };

    auth.signin(params, function(profile, token) {
      ...
    }
  }
```

Bien que vous puissiez inclure le profil complet de l’utilisateur dans le JWT, il est préférable de n’y inclure que le nécessaire, puisque le JWT est généralement transmis avec chaque requête.

<div id="validate-the-jwt-token">
  ## Valider le jeton JWT
</div>

Comme la console AWS Lambda n’a accès qu’à un nombre limité de modules Node.js lorsque vous saisissez votre code Node.js dans la console du navigateur, vous devrez inclure des modules supplémentaires et téléverser la fonction Lambda sous forme de package afin de traiter le jeton d’identité.

Pour plus de détails, consultez [Création de packages de déploiement avec Node.js](http://docs.aws.amazon.com/lambda/latest/dg/nodejs-create-deployment-pkg.html) et [Téléversement de packages de déploiement et test](http://docs.aws.amazon.com/lambda/latest/dg/walkthrough-s3-events-adminuser-create-test-function-upload-zip-test.html).

Le projet de départ suivant contient le code nécessaire pour mettre à jour votre fonction AWS Lambda.

```js wrap lines
<%= include('../../../_includes/_package', { org: 'auth0', repo: 'auth0-aws', path: 'examples/api-gateway/lambda' }) %>
```

Vous verrez deux fichiers JavaScript personnalisés dans le projet de départ :

* `index.js` : contient votre code principal;
* `auth0-variables` : contient le code à mettre à jour.

En plus de ces fichiers personnalisés, il y a un fichier Node.js `package.json` standard.

Le code ajoute la fonctionnalité nécessaire pour extraire des renseignements du JWT et le valider. Par défaut, Auth0 utilise une clé symétrique pour signer le JWT, mais vous pouvez choisir d’utiliser des clés asymétriques (si vous devez permettre à une tierce partie de valider votre jeton, vous devriez utiliser une clé asymétrique et ne partager que votre clé publique).

Pour en savoir plus sur la vérification des jetons, consultez [Identity Protocols Supported by Auth0](/docs/fr-CA/authenticate/protocols).

Mettez à jour `auth0-variables.js` avec votre clé secrète, que vous trouverez dans l’onglet Settings de votre Application dans le <Tooltip tip="Auth0 Dashboard : principal produit d’Auth0 pour configurer vos services." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip> :

```javascript lines
var env={};
env.AUTH0_SECRET='{yourAuth0Secret}';
module.exports = env;
```

Exécutez **npm install** à partir du répertoire où se trouvent vos fichiers, compressez le tout dans un fichier zip (`index.js` doit se trouver à la racine du zip), puis téléversez-le pour l’utiliser avec la Lambda Function `PurchasePet`. Si vous faites le Test, vous devriez voir un échec d’autorisation, puisque le JWT ne se trouve pas dans le corps du message.

Jetez un coup d’œil au code dans `index.js`. Vous verrez, vers la ligne 60, du code qui valide le jeton et en extrait les informations décodées contenant les renseignements d’identité utilisés par la logique d’achat :

```js lines
if(event.authToken) {
     jwt.verify(event.authToken, secret, function(err, decoded) {
         if(err) {
           console.log('failed jwt verify: ', err, 'auth: ', event.authToken);
           context.done('authorization failure', null);
         } else if(!decoded.email)
         {
           console.log('err, email missing in jwt', 'jwt: ', decoded);
           context.done('authorization failure', null);
         } else {
           userEmail = decoded.email;
           console.log('authorized, petId', petId, 'userEmail:', userEmail);
           dynamo.getItem({TableName:"Pets", Key:{username:"default"}}, readcb);
         }
     });
  } else {
     console.log('invalid authorization token', event.authToken);
     context.done('authorization failure', null);
  }
    ...
```

<div id="extract-profile-information-to-assign-a-buyer">
  ### Extraire les renseignements du profil pour attribuer un acheteur
</div>

La dernière étape consiste à transmettre le JWT à la méthode utilisée par le client du navigateur.

La méthode standard comprend un en-tête `Authorization` sous forme de bearer jeton, et vous pouvez utiliser cette méthode en désactivant l’autorisation IAM et en vous appuyant uniquement sur le jeton OpenID pour l’autorisation (vous devrez aussi mapper l’en-tête `Authorization` aux données d’événement transmises à la fonction AWS Lambda).

Cependant, si vous utilisez IAM, AWS API Gateway se sert de l’en-tête `Authorization` pour contenir la signature du message, et l’insertion du JWT dans cet en-tête interrompra l’authentication. Pour ce faire, vous pouvez soit :

* ajouter un en-tête personnalisé pour le JWT;
* placer l’en-tête personnalisé dans le corps du message.

Si vous choisissez d’utiliser un en-tête personnalisé, vous devrez aussi effectuer un mapping pour la requête d’intégration de la méthode POST de `pets/purchase`.

Pour simplifier le processus de validation, transmettez le JWT dans le corps de la requête POST à la fonction AWS Lambda. Pour ce faire, mettez à jour la méthode `buyPet` dans `home.js` en supprimant `userName` du corps et en ajoutant `authToken` comme suit :

```javascript lines
function buyPet(user, id) {
    var apigClient = getSecureApiClient();
    var body = {
      petId:id,
      authToken: store.get('token')
    };

    apigClient.petsPurchasePost({}, body)
      .then(function(response) {
        console.log(response);
        $scope.pets = response.data;
        $scope.$apply();
      }).catch(function (response) {
        alert('buy pets failed');
        showError(response);
    });
}
```

Téléversez votre code dans votre S3 bucket et essayez d’acheter un animal de compagnie. Vous verrez le courriel de l’acheteur dans le message qui en résulte.

Si vous obtenez des erreurs, vérifiez bien que vous avez correctement défini votre clé secrète. Un outil utile pour vérifier les problèmes liés au décodage de votre jeton est [jwt.io](http://jwt.io/).
