<div id="step-4-use-multiple-roles-with-amazon-api-gateway">
  ## Étape 4 - Utiliser plusieurs rôles avec Amazon API Gateway
</div>

Dans cette étape, vous attribuerez différents rôles AWS IAM aux utilisateurs selon les informations d’authentification :

* Les utilisateurs qui s’authentifient avec des Social Connections seront traités comme des acheteurs;
* Les utilisateurs qui s’authentifient avec des Database Connections seront traités comme des administrateurs.

Vous effectuerez cette logique d’attribution de rôles de deux façons différentes :

* JavaScript;
* Auth0 Rules.

Pour de nombreuses Applications Auth0, vous voudrez que différents utilisateurs aient différents niveaux d’accès, et utiliser des renseignements supplémentaires sur une identité donnée dans la logique de votre service. Lorsqu’il suffit de restreindre l’accès au niveau de l’API, vous pouvez utiliser différents rôles AWS IAM (par exemple, les administrateurs peuvent utiliser la fonction de mise à jour pour ajouter et supprimer des animaux, tandis que les utilisateurs d’un fournisseur d’identité sociale peuvent seulement acheter des animaux).

Le diagramme suivant illustre l’attribution des rôles AWS IAM à deux catégories d’utilisateurs : les utilisateurs authentifiés au moyen de Social Connections et les utilisateurs authentifiés au moyen de Database Connections. Il montre également que des rôles AWS IAM peuvent être attribués à d’autres entités, comme des fonctions AWS Lambda, afin de contrôler les permissions attribuées à ces entités pour un compte. Bref, un rôle IAM est un groupe de permissions liées aux capacités AWS, défini par une ou plusieurs politiques, puis attribué à une entité.

Dans les cas où vous souhaitez prendre des décisions dans votre code (par exemple, si vous voulez effectuer une vérification de solvabilité pour un utilisateur qui achète un animal), vous voudrez aussi transmettre l’identité. Cela sera démontré ci-dessous dans [Étape 5 - Utiliser des jetons d’identité pour transmettre l’identité](/docs/fr-CA/customize/integrations/aws/aws-api-gateway-delegation/aws-api-gateway-delegation-5).

<div id="step-1-create-the-petpurchase-api-resource">
  ### Étape 1. Créer la ressource d’API PetPurchase
</div>

Dans la [Amazon API Gateway Console](https://console.aws.amazon.com/apigateway), sélectionnez votre API Pets. Vous serez redirigé vers sa page Resources.

Cliquez sur **Actions**, puis sur **Create Resource**. Nommez la New Child Resource `Purchase`. Cliquez sur **Create Resource**.

Ajoutez une méthode OPTIONS pour la ressource `purchase`, comme indiqué précédemment pour `pets` dans la [section Configurer CORS et déployer l’API de l’étape 2 - Sécuriser et déployer Amazon API Gateway](/docs/fr-CA/customize/integrations/aws/aws-api-gateway-delegation/aws-api-gateway-delegation-2#set-up-cors-and-deploy-the-api).

Créez une nouvelle fonction AWS Lambda pour l’achat d’un animal appelée `PetPurchase`, qui ajoute les attributs `isSold` et `soldTo` à un animal comme suit :

```javascript lines expandable
var AWS = require('aws-sdk');
var DOC = require('dynamodb-doc');
var dynamo = new DOC.DynamoDB();

exports.handler = function(event, context) {
   var petId = event.petId;
   var user = event.userName;
   var pets = {};
   console.log('start PetsPurchase, petId', petId, ' userName', user);

   var writecb = function(err, data) {
      if(!err) {
          context.done(null, pets);
      } else {
          console.log('error on GetPetsInfo: ',err);
          context.done('failed on update', null);
      }
   };

   var readcb = function(err, data) {
      if(err) {
          console.log('error on GetPetsInfo: ',err);
          context.done('failed to retrieve pet information', null);
      } else {
          // vérifier que nous avons des animaux
          if(data.Item && data.Item.pets) {
              pets = data.Item.pets;
              var found = false;

              for(var i = 0; i < pets.length && !found; i++) {
                  if(pets[i].id === petId) {
                     if(!pets[i].isSold) {
                        pets[i].isSold = true;
                        pets[i].soldTo = user;
                        var item = { username:"default",pets: pets};
                        dynamo.putItem({TableName:"Pets", Item:item}, writecb);
                        found = true;
                     }
                  }
               }
               if(!found) {
                 console.log('pet not found');
                 context.done('That pet is not available.', null);
               }
           } else {
              console.log('pet already sold');
              context.done('That pet is not available.', null);
           }
       }
   };

   dynamo.getItem({TableName:"Pets", Key:{username:"default"}}, readcb);
};
```

Une fois que vous avez défini la fonction Lambda, [ajoutez une méthode POST à la ressource `purchase`](/docs/fr-CA/customize/integrations/aws/aws-api-gateway-delegation/aws-api-gateway-delegation-1#method-post-pet-information) qui appelle la fonction Lambda `PetPurchase`. Assurez-vous également d’ajouter l’en-tête `Access-Control-Allow-Origin` avec la valeur `*` à la méthode POST à l’aide de la configuration de réponse de méthode/réponse d’intégration décrite dans la section [Set Up Cors and Deploy the API de l’étape 2 - Securing and Deploying the Amazon API Gateway](/docs/fr-CA/customize/integrations/aws/aws-api-gateway-delegation/aws-api-gateway-delegation-2#set-up-cors-and-deploy-the-api).

Testez la méthode d’API Gateway en fournissant le message d’entrée suivant :

```json lines
{
    "petId": 1,
    "userName": "fred flintstone"
 }
```

Dans la réponse du test, vous devriez voir que l’animal avec l’ID 1 a maintenant été vendu à Fred Flintstone :

```json lines
[
  {
    "id": 1,
    "price": 249.99,
    "type": "dog",
    "isSold": true,
    "soldTo": "fred flintstone"
  },

  ...
```

<div id="update-iam">
  #### Mettre à jour IAM
</div>

Pour sécuriser votre API, suivez le même processus pour ajouter un nouveau rôle que celui que vous avez [suivi dans la partie 2](/docs/fr-CA/customize/integrations/aws/aws-api-gateway-delegation/aws-api-gateway-delegation-2) de ce tutoriel. Nommez ce nouveau rôle `auth0-api-social-role`.

L’ARN de la méthode que vous allez sécuriser dans la stratégie IAM devrait ressembler à ceci :

```bash wrap lines
arn:aws:execute-api:us-east-1:your-accountid:your-api-id/*/pets/purchase
```

Assurez-vous également de mettre à jour la politique d’approbation.

Accédez à la [Amazon API Gateway Console](https://console.aws.amazon.com/apigateway) et sélectionnez la méthode POST pour la ressource `/pets/purchase`. Sélectionnez **Method Request** et remplacez **Authorization Type** par AWS&#95;IAM. Cliquez sur la coche pour enregistrer ce paramètre.

À ce stade, vous avez défini deux rôles que vous pouvez utiliser avec Amazon API Gateway :

* `auth0-api-role` : permet de mettre à jour les animaux de compagnie
* `auth0-api-social-role` : permet d’acheter un animal de compagnie

<div id="configure-login-with-amazon-and-update-auth0">
  #### Configurer Login with Amazon et mettre à jour Auth0
</div>

Vous pouvez créer un rôle social à l’aide de Login with Amazon (LWA).

Bien que ce tutoriel comprenne des instructions pour utiliser Login with Amazon, notez que vous pouvez aussi utiliser d’autres fournisseurs d’identité sociale.

1. Accédez à [Auth0 Dashboard &gt; Authentication &gt; Social](https://manage.auth0.com/#/connections/social), puis sélectionnez **Create Connection**.
2. Choisissez la connexion que vous voulez configurer, puis donnez votre consentement.
3. Copiez et collez le `Client ID` et le `Client Secret` de votre fournisseur d’identité sociale, sélectionnez les **Attributes** (et les **Permissions**, s’il y a lieu), puis cliquez sur **Save**.
4. Sélectionnez la vue **Applications**, activez le commutateur pour chacune de vos applications Auth0 devant pouvoir utiliser cette connexion, puis sélectionnez **Save**.

Une fois les renseignements appropriés saisis, sélectionnez **Try Connection** pour vous assurer que tout est correctement configuré.

Lorsque vous configurez LWA dans la console Amazon, veillez à saisir dans Allowed Return URLs l’URL de rappel de votre application Auth0, qui devrait ressembler à `https://johndoe.auth0.com/login/callback`. La page d’aide d’Auth0 vous indiquera exactement ce que vous devez saisir.

Accédez à [Auth0 Dashboard &gt; Applications &gt; Applications](https://manage.auth0.com/#/applications), puis sélectionnez votre Application pour en afficher les paramètres. Sélectionnez la vue **Connections**, repérez la section **Social** et assurez-vous que **Amazon** est activé.

<div id="enforce-role-assignment-with-auth0-rules">
  ### Imposer l’attribution des rôles avec les Auth0 Rules
</div>

Dans certains cas, vous pourriez déterminer le rôle approprié à partir de l’Application (comme illustré ici), mais pour des raisons de sécurité (par exemple, pour empêcher l’utilisateur d’assumer un rôle plus privilégié que nécessaire), il შეიძლება être préférable de déterminer les privilèges de l’utilisateur côté serveur.

Avec Auth0, cela se fait au moyen de Rules, c’est-à-dire une logique de service que vous définissez et qui s’exécute ensuite pendant le processus d’authentification Auth0. Par exemple, vous pourriez créer des Rules pour :

* Éliminer la transmission des informations sur le rôle du navigateur vers l’Application;
* Insérer les informations sur le rôle dans la requête de délégation en fonction de la source d’authentification.

<div id="enforce-role-assignment">
  #### Appliquer l’attribution des rôles
</div>

Vous allez ajouter une rule qui vérifiera si le rôle demandé par l’utilisateur est autorisé, selon son association avec une connexion Social ou Database Connection.

1. Accédez à [Auth0 Dashboard &gt; Auth Pipeline &gt; Rules](https://manage.auth0.com/#/rules), puis sélectionnez **Create Rule**.
2. Choisissez le modèle **Empty rule**
3. Nommez la rule **AWS Pets** (ou un nom semblable), puis ajoutez le code JavaScript suivant dans le corps de la rule :

export const codeExample = `   function (user, context, callback) {
  if(context.clientID === '{yourClientId}') {
    var socialRoleInfo = {
      role:"arn:aws:iam::<your account>:role/auth0-api-social-role",
      principal: "arn:aws:iam::your account>:saml-provider/auth0"
    };

    var adminRoleInfo = {
      role:"arn:aws:iam::<your account>:role/auth0-api-role",
      principal: "arn:aws:iam::<your account>:saml-provider/auth0"
    };

    var requestRole = context.request.body.role;
    var requestPrincipal = context.request.body.principal;
    var allowedRole = null;

    if(user.identities[0].isSocial === false) {
      allowedRole = adminRoleInfo;
    } else {
      allowedRole = socialRoleInfo;
    }

    if((requestRole && requestRole !== allowedRole.role) ||
      (requestPrincipal && requestPrincipal !== allowedRole.principal)) {
        console.log('le rôle demandé ne correspond pas :',requestRole, ':', requestPrincipal);
        console.log('remplacement');
    } else {
      console.log('rôle valide ou aucun rôle demandé pour la délégation');
    }

    context.addonConfiguration = context.addonConfiguration || {};
    context.addonConfiguration.aws = context.addonConfiguration.aws || {};
    context.addonConfiguration.aws.role = allowedRole.role;
    context.addonConfiguration.aws.principal = allowedRole.principal;
    callback(null, user, context);

  } else {
    callback(null, user, context);
  }
}
`;

Assurez-vous d’ajuster le code ci-dessus avec les valeurs appropriées pour votre intégration. Les champs sont **Princial ARN**, **Role ARN** et **Client Secret**.
4. Cliquez sur **Save** pour enregistrer vos modifications.

<div id="caveats">
  #### Mises en garde
</div>

* Les Rules s’exécutent à portée globale pour chaque authentification. Vous ne devriez exécuter cette logique que pour les requêtes d’authentification associées à une application donnée (c’est pourquoi le script utilisé demande le clientID. Sans cette information, la logique s’exécute pour chaque requête d’authentification associée à votre compte Auth0.
* L’information est transmise à la Rule avec le contexte et l’utilisateur.
* Vous pouvez étendre les objets transmis à la Rule. Dans le code ci-dessus, la Rule vérifie le corps de la requête pour y trouver l’information sur le rôle. Le rôle est défini dans le context addonConfiguration du rôle autorisé, ce qui remplace toujours les paramètres du corps de la requête.
