Suivez les lignes directrices ci-dessous pour rédiger du code Actions performant, sécurisé et clair, afin d’assurer un environnement de production fluide.

<div id="actions-basics">
  ## Notions de base des Actions
</div>

* Utilisez le moins de requêtes HTTP possible et définissez un délai d’expiration raisonnable (moins de 10 secondes) pour éviter l’accumulation de requêtes lors de la connexion.
* Utilisez les [métadonnées d’application](/docs/fr-CA/get-started/applications/configure-application-metadata) pour filtrer des applications précises afin de déterminer si une Action doit être exécutée.

  * Exemple : des valeurs propres à l’application ou de regroupement d’applications servant à faire varier partiellement ou entièrement l’exécution de l’Action.
* Utilisez `user_metadata`/`app_metadata` pour enregistrer des données personnalisées dans le profil utilisateur.

  * Exemple : les préférences de l’utilisateur ou des valeurs d’état.
* Utilisez les [métadonnées de transaction](/docs/fr-CA/customize/actions/transaction-metadata) pour partager des données entre les [déclencheurs post-login](/docs/fr-CA/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger) au cours d’une même exécution.

  * Exemple : des valeurs récupérées de services externes ou calculées qui doivent être réutilisées dans des Actions subséquentes et qui sont liées à l’exécution en cours.
* Utilisez le cache pour les données à partager entre des Actions appartenant à différentes exécutions.

  * Exemple : des valeurs récupérées de services externes ou calculées qui doivent être réutilisées par des Actions et qui ne sont liées ni à l’utilisateur ni à l’exécution en cours.
* Assurez-vous que les Actions qui servent à la vérification ou qui déclenchent la <Tooltip tip="Authentification multifacteur (MFA) : processus d’authentification de l’utilisateur qui utilise un facteur en plus du nom d’utilisateur et du mot de passe, comme un code envoyé par SMS." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=MFA">MFA</Tooltip> ne peuvent pas être contournées, intentionnellement ou de façon malveillante.
* Les Actions ne devraient jamais lancer intentionnellement une erreur; si un processus s’arrête en raison d’une erreur ou d’une condition, utilisez la méthode `api` appropriée, comme `api.access.deny()`.
* Utilisez `event.request.hostname` pour le domaine utilisé dans les appels à l’Authentication API; il peut s’agir du domaine par défaut du tenant Auth0 ou d’un [domaine personnalisé](/docs/fr-CA/customize/custom-domains).

<div id="actions-modules">
  ## Modules Actions
</div>

* Utilisez les Modules Actions lorsqu’il faut réutiliser des fonctions dans différentes Actions, afin d’éviter la duplication du code.
* Réduisez le temps d’instanciation des objets lorsque c’est possible en appliquant le [patron Singleton](https://en.wikipedia.org/wiki/Singleton_pattern) pour réutiliser des instances d’objet dans le module Action.
* Suivez les lignes directrices pertinentes présentées dans ce document concernant le code, la sécurité, la journalisation, les dépendances et les données utilisateur.
* Lorsque vous utilisez un module Action à partir d’une Action, portez toujours attention à la version du module utilisée.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Lorsque vous publiez une nouvelle version d’un module Action, Auth0 ne met pas automatiquement à jour la version à laquelle les Actions font référence, ce qui évite les problèmes potentiels si la mise à jour n’est pas compatible avec l’Action.
</Callout>

<div id="coding-basics">
  ## Notions de base du codage
</div>

* Vérifiez l’égalité stricte `===` pour toutes les données reçues ou stockées.
* Utilisez une instruction `return` lorsque le processus d’Action doit s’arrêter.
* Exécutez un outil d’analyse de code, comme [ESLint](https://eslint.org), et un analyseur, comme [Semgrep](https://semgrep.dev/docs/getting-started/), afin d’améliorer la qualité du code et de repérer automatiquement les problèmes.

<div id="security-basics">
  ## Notions de base sur la sécurité
</div>

* N’écrivez pas de secrets ni d’artefacts de code sensibles en texte clair dans votre code Actions. Utilisez plutôt le [Secrets Manager](/docs/fr-CA/customize/actions/write-your-first-action#add-a-secret), ou intégrez votre propre gestionnaire à votre code Actions.
* Ne transmettez pas de renseignements personnels identifiables (PII) non chiffrés en clair, par exemple dans des URL ou des messages d’erreur.
* Utilisez toujours des URL HTTPS pour les redirections et les appels d’API.
* Utilisez une AllowList d’adresses IP lorsque possible.
* Portez attention aux données entrantes qui pourraient être falsifiées (paramètres d’URL, agent utilisateur, etc.).

<div id="defensive-coding">
  ## Programmation défensive
</div>

* [Interceptez les erreurs](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Guide/Control_flow_and_error_handling#exception_handling_statements) et gérez-les au besoin.
* Utilisez des clauses de garde et quittez rapidement si le traitement de l’Action ne doit pas se poursuivre.

<div id="logging">
  ## Journalisation
</div>

* Ne consignez jamais de données sensibles, de secrets ou de PII.
* Limitez-vous à un maximum de 256 caractères consignés par Action.

<div id="dependencies">
  ## Dépendances
</div>

* Utilisez des paquets fiables et maintenus.
* Vérifiez les CVE non corrigées à l&#39;aide de la [fonctionnalité d&#39;audit de `npm`](https://docs.npmjs.com/cli/v6/commands/npm-audit) ou d&#39;un vérificateur automatisé de dépendances relié à un dépôt.
* Utilisez la version la plus récente d&#39;un paquet lorsque possible.

<div id="user-data">
  ## Données utilisateur
</div>

* Vérifiez qu’un courriel est validé à l’aide de `event.user.email_verified` s’il est utilisé dans un contexte sensible ou à sécurité élevée.
* Les différentes Connections fournissent des données de profil utilisateur différentes; le seul [champ du profil utilisateur](/docs/fr-CA/manage-users/user-accounts/user-profiles/user-profile-structure) dont la présence est garantie est `user_id`.

<div id="redirect-actions-in-the-login-flow">
  ## Redirect Actions dans le flux de connexion
</div>

* Le jeton renvoyé par `api.redirect.encodeToken` est signé, mais non chiffré; les données sensibles ou les PII ne devraient donc pas être incluses dans la charge utile.
* Le flux de connexion s’exécute après une connexion réussie, ce qui comprend :

  * SSO (aucun formulaire de connexion affiché)
  * authentification silencieuse (vérification d’une session à l’aide de `prompt=none` dans l’URL d’autorisation)
  * échange de jeton d’actualisation (aucune interaction de l’utilisateur)
  * grants de mot de passe RO (identifiants recueillis auprès d’une application et échangés avec le point de terminaison de jeton)
* Les Actions qui effectuent une redirection doivent tenir compte des cas ci-dessus et soit rejeter l’accès si une interaction est requise, soit autoriser délibérément le contournement, ce qui reporte la responsabilité sur l’application qui demande la connexion.
