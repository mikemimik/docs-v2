Suivez les lignes directrices ci-dessous pour rédiger du code Actions performant, sécuritaire et clair dans un environnement de production optimisé.

<div id="actions-basics">
  ## Principes de base d’Actions
</div>

* Utilisez le moins de requêtes HTTP possible et définissez un délai d’expiration raisonnable (moins de 10 secondes) afin d’éviter l’accumulation de requêtes lors de la connexion.
* Utilisez les [métadonnées d’application](/docs/fr-CA/get-started/applications/configure-application-metadata) pour filtrer des applications précises afin de déterminer si une Action doit être exécutée.

  * Exemple : des valeurs propres à une application ou des valeurs de regroupement d’applications servant à faire varier partiellement ou entièrement l’exécution de l’Action.
* Utilisez `user_metadata`/`app_metadata` pour enregistrer des données personnalisées dans le profil de l’utilisateur.

  * Exemple : les préférences de l’utilisateur ou des valeurs d’état.
* Utilisez les [métadonnées de transaction](/docs/fr-CA/customize/actions/transaction-metadata) pour partager des données entre les [déclencheurs post-login](/docs/fr-CA/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger) au cours d’une même exécution.

  * Exemple : des valeurs récupérées à partir de services externes ou calculées qui doivent être réutilisées dans des Actions subséquentes et qui sont liées à l’exécution en cours.
* Utilisez le cache pour les données à partager entre des Actions appartenant à différentes exécutions.

  * Exemple : des valeurs récupérées à partir de services externes ou calculées qui doivent être réutilisées par des Actions et qui ne sont liées ni à l’utilisateur ni à l’exécution en cours.
* Assurez-vous que les Actions qui servent à la vérification ou qui déclenchent la <Tooltip tip="Authentification multifacteur (MFA) : processus d’authentification de l’utilisateur qui utilise un facteur en plus du nom d’utilisateur et du mot de passe, comme un code envoyé par SMS." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=MFA">MFA</Tooltip> ne peuvent pas être contournées, intentionnellement ou non.
* Les Actions ne devraient jamais générer intentionnellement une erreur; si un processus doit s’arrêter en raison d’une erreur ou d’une condition, utilisez la méthode `api` appropriée, comme `api.access.deny()`.
* Utilisez `event.request.hostname` pour le domaine utilisé dans les appels à l’Authentication API; il peut s’agir du domaine de tenant Auth0 par défaut ou d’un [domaine personnalisé](/docs/fr-CA/customize/custom-domains).

<div id="actions-modules">
  ## Modules Actions
</div>

* Utilisez les Modules Actions lorsqu’il faut réutiliser des fonctions dans différentes Actions, afin d’éviter la redondance du code.
* Réduisez, lorsque possible, le temps d’instanciation des objets en implémentant le [patron Singleton](https://en.wikipedia.org/wiki/Singleton_pattern) afin de réutiliser des instances d’objet dans le module Action.
* Suivez les directives applicables présentées dans ce document concernant la programmation, la sécurité, la journalisation, les dépendances et les données utilisateur.
* Lorsque vous utilisez un module Action à partir d’une Action, tenez toujours compte de la version du module utilisée.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Lorsque vous publiez une nouvelle version d’un module Action, Auth0 ne met pas automatiquement à niveau la version référencée par les Actions, ce qui évite d’éventuels problèmes si la mise à niveau n’est pas compatible avec l’Action.
</Callout>

<div id="coding-basics">
  ## Notions de base de la programmation
</div>

* Vérifiez l’égalité stricte `===` pour toutes les données reçues ou stockées.
* Utilisez une instruction `return` lorsque le processus Action doit s’interrompre.
* Exécutez un linter de code, comme [ESLint](https://eslint.org), et un analyseur, comme [Semgrep](https://semgrep.dev/docs/getting-started/), pour améliorer la qualité du code et détecter automatiquement les problèmes.

<div id="security-basics">
  ## Principes de base de la sécurité
</div>

* N’inscrivez pas de secrets ni d’artéfacts de code sensibles en clair dans votre code Actions. Utilisez plutôt le [gestionnaire de secrets](/docs/fr-CA/customize/actions/write-your-first-action#add-a-secret), ou intégrez votre propre gestionnaire au code Actions.
* Ne transmettez pas de renseignements personnels identifiables (PII) non chiffrés en évidence, par exemple dans des URL ou des messages d’erreur.
* Utilisez toujours des URL HTTPS pour les redirections et les appels d’API.
* Utilisez une AllowList d’adresses IP lorsque possible.
* Soyez attentif aux données entrantes qui peuvent être altérées (paramètres d’URL, agent utilisateur, etc.).

<div id="defensive-coding">
  ## Programmation défensive
</div>

* [Interceptez les erreurs](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Guide/Control_flow_and_error_handling#exception_handling_statements) et gérez-les au besoin.
* Utilisez des clauses de garde et retournez immédiatement si le traitement de l’Action ne doit pas se poursuivre.

<div id="logging">
  ## Journalisation
</div>

* Ne consignez jamais de données sensibles, de secrets ni de PII.
* Ne dépassez pas le maximum de 256 caractères consignés par Action.

<div id="dependencies">
  ## Dépendances
</div>

* Utilisez des packages fiables et bien maintenus.
* Vérifiez s’il existe des CVE non corrigées à l’aide de la [fonction d’audit de `npm`](https://docs.npmjs.com/cli/v6/commands/npm-audit) ou d’un vérificateur automatisé de dépendances connecté à un dépôt.
* Utilisez la version la plus récente d’un package lorsque c’est possible.

<div id="user-data">
  ## Données utilisateur
</div>

* Vérifiez si une adresse courriel a été vérifiée à l’aide de `event.user.email_verified` lorsqu’elle est utilisée dans un contexte sensible ou de haute sécurité.
* Différentes connexions fournissent différentes données de profil utilisateur; le seul [champ du profil utilisateur](/docs/fr-CA/manage-users/user-accounts/user-profiles/user-profile-structure) garanti est `user_id`.

<div id="redirect-actions-in-the-login-flow">
  ## Actions de redirection dans le flux de connexion
</div>

* Le jeton renvoyé par `api.redirect.encodeToken` est signé, mais non chiffré; les données sensibles ou les PII ne doivent donc pas être incluses dans la charge utile.
* Le flux de connexion s’exécute après une connexion réussie, ce qui comprend :

  * SSO (aucun formulaire de connexion n’est affiché)
  * authentification silencieuse (vérification d’une session à l’aide de `prompt=none` dans l’URL d’autorisation)
  * échange de jeton d’actualisation (aucune interaction utilisateur)
  * octrois RO par mot de passe (identifiants recueillis à partir d’une application et échangés avec le point de terminaison de jeton)
* Les Actions qui effectuent une redirection doivent tenir compte des cas ci-dessus et soit rejeter l’accès si une interaction est requise, soit permettre intentionnellement le contournement, ce qui transfère la responsabilité à l’application qui demande la connexion.
