Selon l’étape de votre mise en œuvre, plusieurs options s’offrent à vous pour tester et déboguer Auth0 Actions.

Nous vous recommandons de tester des actions individuelles à partir du <Tooltip tip="Auth0 Dashboard: le principal produit d'Auth0 pour configurer vos services." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip> au fur et à mesure de leur mise en œuvre. Une fois prêt, vous devriez vérifier que le flux de bout en bout fonctionne comme prévu à l’aide des journaux du tenant dans Auth0 Dashboard.

<div id="test-individual-actions">
  ## Tester des Actions individuelles
</div>

Vous pouvez tester des Actions individuelles à l’aide de l’Actions Code Editor. La fonctionnalité de test de l’éditeur simule un appel à l’Action à l’aide d’un payload d’exemple basé sur le flux auquel l’Action est associée. Pour tester une Action individuelle :

1. Accédez à [Auth0 Dashboard &gt; Actions](https://manage.auth0.com/#/actions), puis choisissez le flux et l’Action que vous souhaitez modifier.
2. Repérez l’Actions Code Editor, puis sélectionnez **Test** (l’icône de lecture) dans sa barre latérale. Modifiez le payload dans la barre latiale Test pour analyser le résultat de différents payloads.
3. Sélectionnez **Run**.

Les résultats du test affichent les étapes exécutées par l’Action, la sortie de la console, les erreurs survenues et des statistiques utiles sur l’exécution de l’Action.

<Frame>![Capture d’écran du module de test dans l’Actions Code Editor](/docs/images/cdy7uua7fh8z/7I4CRM8SmRFiibjuk55L86/794042f525cddd55a8005cc646b3c9c2/test-action-code-editor.png)</Frame>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Vous pouvez aussi tester une Action à l’aide du endpoint [**Test an Action** du Management API](https://auth0.com/docs/api/management/v2#!/Actions/post_test_action), en fournissant un payload correspondant au flux auquel l’Action est associée. Pour obtenir un payload d’exemple, testez l’Action à l’aide de la fonctionnalité Test de l’Actions Code Editor.
</Callout>

<div id="debug-deployed-actions">
  ## Déboguer les Actions déployées
</div>

Maintenant que vous avez testé votre Action, il est temps de la déployer et d’en observer l’exécution. Si vous avez déjà déployé votre Action et l’avez ajoutée à un flux (pour savoir comment attacher des Actions à des flux, consultez la section « Attacher l’Action à un flux » dans [Rédiger votre première Action](/docs/fr-CA/customize/actions/write-your-first-action)), vous pouvez déboguer votre Action en production en quelques étapes.

<div id="verify-end-to-end-login-flow">
  ### Vérifier le Login flux de bout en bout
</div>

Pour une Action `post-login`, vous pouvez vérifier le Login flux de bout en bout en exécutant une tentative de connexion pour votre tenant :

1. Accédez à [Auth0 Dashboard &gt; Authentication &gt; Authentication Profile](https://manage.auth0.com/#/authentication-profiles), puis sélectionnez **Try**. Une fenêtre contenant un exemple de connexion s’ouvrira.
2. Suivez le Login flux. Toutes les Actions configurées seront exécutées.

Une fois l’opération terminée, vous serez redirigé vers une page qui affiche soit les attributs du profil utilisateur que vos applications recevront, soit un message d’erreur expliquant ce qui s’est mal passé.

<div id="analyze-tenant-logs">
  ### Analyser les journaux du tenant
</div>

Chaque fois qu’un déclencheur s’exécute et qu’une Action associée est liée à son flux, vos [journaux du tenant](/docs/fr-CA/deploy-monitor/logs/view-log-events) incluront les détails d’exécution de l’Action.

Dans la liste des journaux du tenant, vous pouvez voir les journaux associés aux Actions et choisir de les consulter. Dans l’exemple ci-dessous, le type de journal `Success Login` est associé à une Action sur le déclencheur `post-login` de ce tenant.

<div id="unit-test-actions">
  ## Tester les Actions avec des tests unitaires
</div>

​Vous pouvez tester vos Auth0 Actions avec des tests unitaires en simulant l’événement et l’API passés à votre fonction Action. Vos Actions peuvent se trouver dans le contrôle de version ou dans un répertoire local, et vous pouvez utiliser un outil comme Jest pour tester automatiquement vos Actions. Dans l’exemple suivant, nous utilisons le déclencheur [Login / Post Login](/docs/fr-CA/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger) et [Jest](https://jestjs.io/) pour simuler et tester.

**Code de l’Action :**

```javascript lines
// action.js

exports.onExecutePostLogin = async (event, api) => {
  const namespace = "https://my-app.example.com";
  if (event.authorization) {
    api.idToken.setCustomClaim(`${namespace}/favorite_color`, "pink");
  }
};
```

**Code du fichier de test :**

```javascript lines
// action.spec.js

const { onExecutePostLogin } = require("./action.js");
test("sets favorite_color ID token claim to pink", async () => {
  const namespace = "https://my-app.example.com";
  const mockFunction = jest.fn();
  const mockApi = { idToken: { setCustomClaim: mockFunction } };
  const mockEvent = {authorization: {}};

  await onExecutePostLogin(mockEvent, mockApi);
  expect(mockFunction).toBeCalledWith(
    `${namespace}/favorite_color`,
    "pink"
  );
});
```

<div id="best-practices">
  ## Bonnes pratiques
</div>

Lorsque vous développez des Actions dans Auth0, [nous vous recommandons de créer un tenant Auth0 distinct pour chaque environnement](/docs/fr-CA/get-started/auth0-overview/create-tenants/set-up-multiple-environments). Vous pourrez ainsi isoler vos utilisateurs de production de ceux de vos autres environnements, et attribuer des administrateurs différents à chaque environnement (par exemple, les ingénieurs pourraient ne pas avoir accès à la configuration de production).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Si vous souhaitez gérer, tester et déployer entièrement vos Actions au moyen d’un CLI ou d’un pipeline CI/CD, vous pouvez utiliser le [Deploy CLI](/docs/fr-CA/deploy-monitor/deploy-cli-tool).
</Callout>
