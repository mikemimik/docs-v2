Le déclencheur de connexion s’exécute lorsqu’un utilisateur s’authentifie avec succès auprès d’une application dans un tenant Auth0. Cela inclut également l’authentification après l’inscription.

<Frame>![Schéma du flux de connexion dans Actions.](/docs/images/cdy7uua7fh8z/2SkfIOm4fFOJ8N0GNJwam8/77230cdb4411b2383751b037fb56fe29/2024-09-30_10-01-42.png)</Frame>

Les Actions de ce déclencheur sont bloquantes (synchrones), ce qui signifie qu’elles s’exécutent dans le cadre du processus du déclencheur et empêchent le reste du pipeline Auth0 de s’exécuter tant que l’Action n’est pas terminée.

<div id="triggers">
  ## Déclencheurs
</div>

<div id="login-post-login">
  ### Login / Post Login
</div>

Le déclencheur `post-login` est une fonction exécutée après qu’un utilisateur a ouvert une session et lorsqu’un <Tooltip tip="jeton d’actualisation : jeton utilisé pour obtenir un nouveau jeton d’accès sans obliger les utilisateurs à se reconnecter." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Refresh+Token">jeton d’actualisation</Tooltip> est échangé.

<div id="references">
  #### Références
</div>

* [objet Event](/docs/fr-CA/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger/post-login-event-object) : fournit des renseignements contextuels sur un utilisateur qui se connecte au moyen d’Auth0.
* [objet API](/docs/fr-CA/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger/post-login-api-object) : fournit des méthodes permettant de modifier le comportement du flux.

<div id="common-use-cases">
  ## Cas d’utilisation courants
</div>

<div id="access-control">
  ### Contrôle d’accès
</div>

Une Action post-login peut être utilisée pour exécuter une logique personnalisée afin de rejeter l’accès d’un utilisateur qui tente d’accéder à une application :

```javascript lines
/**
 * @param {Event} event - Détails sur l'utilisateur et le contexte dans lequel il se connecte.
 * @param {PostLoginAPI} api - Interface dont les méthodes peuvent être utilisées pour modifier le comportement de la connexion.
 */
exports.onExecutePostLogin = async (event, api) => {
  if (event.user.email && event.user.email.endsWith("@example.com") && event.client.name === "My SPA") {
    api.access.deny(`Access to ${event.client.name} is not allowed.`);
  }
};
```

<div id="allow-access-only-on-weekdays-for-a-specific-application">
  ### Autoriser l’accès à une application précise uniquement les jours de semaine
</div>

Si vous avez une application et que vous voulez vous assurer qu’elle n’est accessible que les jours de semaine, vous pouvez créer l’Action suivante :

```javascript lines
/**
 * @param {Event} event - Détails sur l'utilisateur et le contexte dans lequel il se connecte.
 * @param {PostLoginAPI} api - Interface dont les méthodes peuvent être utilisées pour modifier le comportement de la connexion.
 */
 exports.onExecutePostLogin = async (event, api) => {
  if (event.client.name === "APP_NAME") {
    const d = new Date().getDay();

    if (d === 0 || d === 6) {
      api.access.deny("This app is only available during the week.");
    }
  }
}
```

<div id="deny-access-to-anyone-calling-an-api">
  ### Rejeter l’accès à quiconque appelle une API
</div>

Supposons que vous vouliez rejeter l’accès à tous les utilisateurs qui appellent une API. Cela signifie que vous devez rejeter l’accès en fonction de la valeur d’<Tooltip tip="Audience : identificateur unique de l’audience d’un token émis. Appelée aud dans un token, sa valeur contient l’ID d’une application (Client ID) pour un ID Token ou d’une API (API Identifier) pour un Access Token." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=audience">audience</Tooltip> de votre API, que vous trouverez dans le champ **API Audience** de votre API dans [Dashboard &gt; Applications &gt; APIs](https://manage.auth0.com/#/apis). Pour ce faire, vous créeriez l’Action suivante :

```javascript lines
/**
 * @param {Event} event - Détails sur l'utilisateur et le contexte dans lequel il se connecte.
 * @param {PostLoginAPI} api - Interface dont les méthodes peuvent être utilisées pour modifier le comportement de la connexion.
 */
 exports.onExecutePostLogin = async (event, api) => {
  // Dans Actions, une API est désignée comme un Resource Server.
  if (event.resource_server && event.resource_server.identifier === "http://todoapi2.api") {
    api.access.deny("end_users_not_allowed");
  }
}
```

<div id="add-user-roles-to-id-and-access-tokens">
  ### Ajouter les rôles utilisateur aux jetons d’ID et d’accès
</div>

Pour ajouter les rôles utilisateur aux jetons émis par Auth0, utilisez l’objet `event.authorization` ainsi que les méthodes `api.idToken.setCustomClaim` et `api.accessToken.setCustomClaim` :

```javascript lines
/**
 * @param {Event} event - Détails sur l'utilisateur et le contexte dans lequel il se connecte.
 * @param {PostLoginAPI} api - Interface dont les méthodes peuvent être utilisées pour modifier le comportement de la connexion.
 */
exports.onExecutePostLogin = async (event, api) => {
  const namespace = 'https://my-app.example.com';
  if (event.authorization) {
    api.idToken.setCustomClaim(`${namespace}/roles`, event.authorization.roles);
    api.accessToken.setCustomClaim(`${namespace}/roles`, event.authorization.roles);
  }
}
```

* Un claim personnalisé ne peut pas inclure certains termes, et nous recommandons fortement d’utiliser un claim avec espace de noms sous la forme d’un URI. Consultez [notre documentation sur les claims personnalisés](/docs/fr-CA/secure/tokens/json-web-tokens/create-custom-claims) pour en savoir plus.
* Le <Tooltip tip="JSON Web Token (JWT) : format standard de ID Token (et souvent de access token) utilisé pour représenter des claims de façon sécurisée entre deux parties." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=JWT">JWT</Tooltip> renvoyé à l’application demandeuse est généré et signé à la fin du traitement du Trigger. Le JWT final signé n’est pas accessible dans une Action.

<div id="enrich-the-user-profile">
  ### Enrichir le profil utilisateur
</div>

Auth0 offre un système de stockage des métadonnées dans un [profil utilisateur](/docs/fr-CA/manage-users/user-accounts/user-profiles/normalized-user-profiles). Pour définir `user_metadata` ou `app_metadata` dans le profil d’un utilisateur au moment de sa connexion, utilisez les fonctions `api.user.setUserMetadata` ou `api.user.setAppMetadata`.

```javascript lines
/**
 * @param {Event} event - Détails sur l'utilisateur et le contexte dans lequel il se connecte.
 * @param {PostLoginAPI} api - Interface dont les méthodes peuvent être utilisées pour modifier le comportement de la connexion.
 */
exports.onExecutePostLogin = async (event, api) => {
  api.user.setUserMetadata("favorite_color", "blue");
};
```

Une fois toutes les actions post-login exécutées, Actions met à jour le profil utilisateur en une seule opération. Cette opération est soumise à la [limite de taux](/docs/fr-CA/troubleshoot/customer-support/operational-policies/rate-limit-policy/management-api-endpoint-rate-limits) « Write Users ».

<div id="enforce-custom-mfa-policy">
  ### Appliquer une politique d’AMF personnalisée
</div>

Une Action post-login peut servir à exiger dynamiquement l’<Tooltip tip="Authentification multifacteur (AMF) : processus d’authentification de l’utilisateur qui utilise un facteur supplémentaire, en plus du nom d’utilisateur et du mot de passe, comme un code envoyé par SMS." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=MFA">AMF</Tooltip> pour un utilisateur, selon les besoins de votre application.

```javascript lines
/**
 * @param {Event} event - Détails sur l'utilisateur et le contexte dans lequel il effectue sa connexion.
 * @param {PostLoginAPI} api - Interface dont les méthodes peuvent être utilisées pour modifier le comportement de la connexion.
 */
exports.onExecutePostLogin = async (event, api) => {
  // Exiger l'AMF pour toute personne se connectant depuis l'Amérique du Nord.
  if (event.request.geoip.continentCode === "NA") {
    api.multifactor.enable("any");
  };
};
```

Un fournisseur MFA doit être configuré pour activer l’authentification multifacteur lors d’une connexion. Pour en savoir plus, consultez [Authentification multifacteur](/docs/fr-CA/secure/multi-factor-authentication).

<div id="reduce-friction-with-passkeys">
  ### Réduire les obstacles avec les clés d’accès
</div>

Une post-login Action peut être utilisée pour réduire les obstacles en ignorant dynamiquement l’AMF pour un utilisateur qui s’est authentifié avec une clé d’accès.

```javascript lines
/**
* Gestionnaire qui sera appelé lors de l'exécution d'un flux PostLogin.
*
* @param {Event} event - Détails sur l'utilisateur et le contexte dans lequel il se connecte.
* @param {PostLoginAPI} api - Interface dont les méthodes peuvent être utilisées pour modifier le comportement de la connexion.
*/
exports.onExecutePostLogin = async (event, api) => {
 // Vérifier si une clé d'accès a été utilisée pour s'authentifier
 const skipMFA = event.authentication?.methods.some(
   (method) => method.name === "passkey"
 );

 // Si une clé d'accès a été utilisée, ignorer l'AMF
 if (skipMFA) {
   api.multifactor.enable("none");
 }
};
```

Les clés d’accès et l’AMF doivent être activées pour la connexion. Pour en savoir plus, consultez [Clés d’accès](/docs/fr-CA/authenticate/database-connections/passkeys) et [Authentification multifacteur](/docs/fr-CA/secure/multi-factor-authentication).

<div id="redirect-the-user-to-an-external-site">
  ### Rediriger l’utilisateur vers un site externe
</div>

Comme avec les [Redirect Rules](/docs/fr-CA/customize/rules/redirect-users), une Action post-login peut servir à rediriger l’utilisateur vers un site externe. Une fois l’opération terminée, l’utilisateur peut être redirigé vers Auth0 pour poursuivre son flux de connexion. Dans l’exemple ci-dessous, une Redirect Action est utilisée pour demander à l’utilisateur d’indiquer sa couleur préférée.

```javascript lines expandable
/**
 * @param {Event} event - Détails sur l'utilisateur et le contexte dans lequel il se connecte.
 * @param {PostLoginAPI} api - Interface dont les méthodes peuvent être utilisées pour modifier le comportement de la connexion.
 */
exports.onExecutePostLogin = async (event, api) => {
  // Ignorer la redirection si l'utilisateur a déjà choisi une couleur préférée.
  if (event.user.user_metadata.favorite_color) {
    return;
  }

  const token = api.redirect.encodeToken({
    secret: event.secrets.MY_SHARED_SECRET,
    payload: {
      email: event.user.email,
    },
  });

  // Envoyer l'utilisateur vers https://my-app.example.com
  // avec un paramètre de chaîne de requête `session_token`.
  api.redirect.sendUserTo("https://my-app.example.com", { 
    query: { session_token: token }
  });
};

/**
 * @param {Event} event - Détails sur l'utilisateur et le contexte dans lequel il se connecte.
 * @param {PostLoginAPI} api - Interface dont les méthodes peuvent être utilisées pour modifier le comportement de la connexion.
 */
exports.onContinuePostLogin = async (event, api) => {
  // Une fois que l'endpoint /continue a été appelé, extraire le jeton signé
  // et stocker la couleur préférée dans les métadonnées de l'utilisateur.

  const payload = api.redirect.validateToken({
    secret: event.secrets.MY_SHARED_SECRET,
  });

  api.user.setUserMetadata("favorite_color", payload.favorite_color);
};
```

Le pipeline d’Actions sera suspendu pendant la redirection de l’utilisateur. Une fois que l’utilisateur reprendra son processus de connexion avec Auth0, le pipeline d’Actions redémarrera à l’endroit où il avait été suspendu. Les actions exécutées avant la Redirect ne seront pas exécutées à nouveau.

Pour en savoir plus sur les Redirect Actions, consultez [Redirect with Actions](/docs/fr-CA/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger/redirect-with-actions).

<div id="modify-scopes-on-an-access-token">
  ### Modifier les scopes d’un jeton d’accès
</div>

Lorsque vous modifiez les scopes associés à un <Tooltip tip="Jeton d’accès : information d’autorisation, sous la forme d’une chaîne opaque ou d’un JWT, utilisée pour accéder à une API." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=access+token">jeton d’accès</Tooltip>, veillez à suivre les pratiques exemplaires relatives à la définition de l’audience.

* Vérifiez toujours l’audience attendue avant d’ajouter des scopes.
* Évitez d’utiliser des entrées non fiables lors de l’ajout de scopes.

```javascript lines
/**
 * @param {Event} event - Détails sur l'utilisateur et le contexte dans lequel il se connecte.
 * @param {PostLoginAPI} api - Interface dont les méthodes peuvent être utilisées pour modifier le comportement de la connexion.
 */
exports.onExecutePostLogin = async (event, api) => {
  if (event?.resource_server?.identifier === 'https://example.com/api') {
    api.accessToken.addScope("read:xyz");
  }
};
```

<div id="deny-access-to-specific-ja3ja4-fingerprints">
  ### Rejeter l’accès pour des empreintes JA3/JA4 précises
</div>

L’objet `event.security_context` contient les valeurs d’empreinte JA3/JA4 pour la transaction en cours.

```js lines
exports.onExecutePreUserRegistration = async (event, api) => {
  const clientJa4 = event?.security_context?.ja4;
  console.log('[ACTION]', {clientJa4});
  const badFingerprints = ['t13d1517h2_8daaf6152771_b6f405a00624','t13d1516h2_8daaf6152771_d8a2da3f94cd'];
  if (clientJa4 && badFingerprints.includes(clientJa4)){
    api.access.deny('suspicious_tls_fingerprint', 'Your TLS fingerprint has been flagged as suspicious');
  }
};
```
