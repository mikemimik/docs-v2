Une fois que l’Auth0 Authorization Server a validé que la requête échange de jeton personnalisé est valide et qu’elle correspond à un [échange de jeton personnalisé Profile](/docs/fr-CA/authenticate/custom-token-exchange/configure-custom-token-exchange#create-custom-token-exchange-profile) existant, le déclencheur `custom-token-exchange` exécute l’unique Action associée à ce profil. Si l’Action se termine avec succès et définit un utilisateur pour la transaction, les Actions `post-login` ainsi que le reste du pipeline Auth0 sont ensuite exécutés pour cet utilisateur connecté.

<Frame>![](/docs/images/custom-token-exchange/cte_trigger_flow.png)</Frame>

Les Actions de ce flux sont bloquantes (synchrones), ce qui signifie qu’elles s’exécutent dans le cadre du processus du déclencheur et empêchent le reste du pipeline Auth0 de s’exécuter tant que l’Action n’est pas terminée.

<div id="custom-token-exchange-vs-post-login-actions">
  ## Échange de jetons personnalisé vs Actions post-login
</div>

Le déclencheur des Actions post-login s’exécute après l’Action d’échange de jetons personnalisé, pour l’utilisateur que vous avez défini pour la transaction. Pour d’autres cas d’utilisation, comme l’ajout de claims personnalisés aux jetons d’accès, utilisez le [déclencheur d’Action post-login](/docs/fr-CA/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger).

L’échange de jetons personnalisé utilise le type d’autorisation d’échange de jeton; dans le déclencheur d’Action post-login, `event.transaction.protocol` est alors défini sur `oauth2-token-exchange`. L’échange de jetons personnalisé et Native Social Login utilisent tous deux ce type d’autorisation. Pour les distinguer, utilisez la valeur de `subject_token_type`, qui doit correspondre à l’un de vos profils d’échange de jetons personnalisés.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  L’accès anticipé de l’Échange de jetons personnalisé ne prend pas en charge `api.authentication.challengeWith()` ni `api.authentication.enrollWith()`. Si vous utilisez ces méthodes avec vos Actions Post-Login, la transaction échouera avec une erreur irrécupérable. Assurez-vous d’ignorer ces deux méthodes lorsque `event.transaction.protocol==oauth2-token-exchange`, selon la valeur de `subject_token_type`.
</Callout>

<div id="actor-support-for-delegation">
  ## Prise en charge de l’acteur pour la délégation
</div>

Le déclencheur `custom-token-exchange` prend en charge les scénarios d’autorisation déléguée. L’Action peut appeler [`api.authentication.setActor()`](/docs/fr-CA/customize/actions/explore-triggers/signup-and-login-triggers/custom-token-exchange-trigger/custom-token-exchange-api-object#api-authentication-setactor-actor) pour inclure une revendication `act` dans les jetons émis, afin d’identifier l’entité qui agit au nom de l’utilisateur sujet.

`setActor()` peut être appelé, que la requête inclue ou non un `actor_token`. Le paramètre `actor_token` est l’un des moyens permettant à l’application appelante de transmettre à l’Action des renseignements sur l’acteur, mais l’identité de l’acteur peut aussi être intégrée au `subject_token` lui-même ou obtenue par d’autres moyens personnalisés dans le code de l’Action.

Lorsque la requête inclut `actor_token` et `actor_token_type`, ces valeurs sont accessibles dans [`event.transaction`](/docs/fr-CA/customize/actions/explore-triggers/signup-and-login-triggers/custom-token-exchange-trigger/custom-token-exchange-event-object#event-transaction). Si `actor_token_type` est `urn:ietf:params:oauth:token-type:id_token` et que le jeton est un ID token Auth0 valide provenant du même tenant, le serveur valide automatiquement le jeton et remplit `event.transaction.actor_token_user` avec le profil utilisateur de l’acteur.

La revendication `act` définie via `setActor()` est aussi accessible dans les Actions post-login par l’intermédiaire de [`event.transaction.actor`](/docs/fr-CA/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger/post-login-event-object#event-transaction).

  ## Déclencheurs
</div>

<div id="custom-token-exchange">
  ### Échange de jeton personnalisé
</div>

Le déclencheur `custom-token-exchange` est la première étape du pipeline Auth0. Cette fonction est exécutée après validation de la requête d’échange de jeton personnalisé reçue et son mappage réussi à un profil existant.
