Une fois que l’Auth0 Authorization Server a validé la requête échange de jeton personnalisé et confirmé qu’elle correspond à un [profil d’échange de jeton personnalisé](/docs/fr-CA/authenticate/custom-token-exchange/configure-custom-token-exchange#create-custom-token-exchange-profile) existant, le déclencheur `custom-token-exchange` exécute l’unique Action associée à ce profil. Si l’Action se termine avec succès et définit un utilisateur pour la transaction, les Actions `post-login` et le reste du pipeline Auth0 s’exécutent ensuite pour cet utilisateur connecté.

<Frame>![](/docs/images/custom-token-exchange/cte_trigger_flow.png)</Frame>

Les Actions de ce flux sont bloquantes (synchrones), ce qui signifie qu’elles s’exécutent dans le cadre du processus d’un déclencheur et empêchent le reste du pipeline Auth0 de s’exécuter tant que l’Action n’est pas terminée.

<div id="custom-token-exchange-vs-post-login-actions">
  ## Échange de jeton personnalisé vs Actions Post Login
</div>

Le déclencheur des Actions Post Login s’exécute après l’Action d’échange de jeton personnalisé, pour l’utilisateur que vous avez défini pour la transaction. Pour d’autres cas d’usage, par exemple l’ajout de claims personnalisées aux jetons d’accès, utilisez le [déclencheur d’Action Post Login](/docs/fr-CA/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger).

L’échange de jeton personnalisé utilise le type d’octroi d’échange de jeton, où `event.transaction.protocol` prend la valeur `oauth2-token-exchange` dans le déclencheur d’Action Post Login. L’échange de jeton personnalisé et la connexion sociale native utilisent tous deux le type d’octroi d’échange de jeton. Pour les distinguer, utilisez la valeur de `subject_token_type`, qui doit correspondre à l’un de vos profils d’échange de jeton personnalisé.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  L’Early Access à l’échange de jeton personnalisé ne prend pas en charge `api.authentication.challengeWith()` ni `api.authentication.enrollWith()`. Si vous utilisez ces méthodes avec vos Actions Post Login, la transaction échouera avec une erreur non récupérable. Assurez-vous d’ignorer ces deux méthodes lorsque `event.transaction.protocol==oauth2-token-exchange`, selon la valeur de `subject_token_type`.
</Callout>

<div id="actor-support-for-delegation">
  ## Prise en charge de l’acteur dans la délégation
</div>

Le déclencheur `custom-token-exchange` prend en charge les scénarios d’autorisation déléguée. L’Action peut appeler [`api.authentication.setActor()`](/docs/fr-CA/customize/actions/explore-triggers/signup-and-login-triggers/custom-token-exchange-trigger/custom-token-exchange-api-object#api-authentication-setactor-actor) pour inclure un claim `act` dans les jetons émis afin d’identifier le principal qui agit au nom de l’utilisateur concerné.

`setActor()` peut être appelé que la request inclue ou non un `actor_token`. Le paramètre `actor_token` est une façon pour l’application appelante de transmettre des renseignements sur l’acteur à l’Action, mais l’identité de l’acteur peut aussi être intégrée au `subject_token` lui-même ou obtenue par d’autres moyens personnalisés dans le code de l’Action.

Lorsque la request inclut `actor_token` et `actor_token_type`, ces valeurs sont disponibles dans [`event.transaction`](/docs/fr-CA/customize/actions/explore-triggers/signup-and-login-triggers/custom-token-exchange-trigger/custom-token-exchange-event-object#event-transaction). Si `actor_token_type` est `urn:ietf:params:oauth:token-type:id_token` et que le jeton est un ID token Auth0 valide du même tenant, le serveur valide automatiquement le jeton et remplit `event.transaction.actor_token_user` avec le profil utilisateur de l’acteur.

Le claim `act` défini au moyen de `setActor()` est aussi disponible dans les Actions Post-Login par l’intermédiaire de [`event.transaction.actor`](/docs/fr-CA/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger/post-login-event-object#event-transaction).

<div id="triggers">
  ## Déclencheurs
</div>

<div id="custom-token-exchange">
  ### Échange de jeton personnalisé
</div>

Le déclencheur `custom-token-exchange` est la première étape du pipeline Auth0. Cette fonction s’exécute une fois la requête d’échange de jeton personnalisé reçue validée et correctement associée à un profil existant.
