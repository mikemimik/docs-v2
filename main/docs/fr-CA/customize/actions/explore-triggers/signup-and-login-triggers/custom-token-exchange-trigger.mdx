Après que le serveur d’autorisation Auth0 a validé la requête d’échange de jeton personnalisé et établi sa correspondance avec un [profil d’échange de jeton personnalisé](/docs/fr-CA/authenticate/custom-token-exchange/configure-custom-token-exchange#create-custom-token-exchange-profile) existant, le déclencheur `custom-token-exchange` exécute l’unique Action associée à ce profil. Si l’Action s’exécute correctement et définit un utilisateur pour la transaction, les Actions `post-login`, puis le reste du pipeline Auth0, sont exécutés pour cet utilisateur connecté.

<Frame>![](/docs/images/custom-token-exchange/cte_trigger_flow.png)</Frame>

Les Actions de ce flux sont bloquantes (synchrones), ce qui signifie qu’elles s’exécutent dans le cadre du processus du déclencheur et empêchent le reste du pipeline Auth0 de s’exécuter tant que l’Action n’est pas terminée.

<div id="custom-token-exchange-vs-post-login-actions">
  ## échange de jeton personnalisé et les Actions Post Login
</div>

Le déclencheur des Actions Post Login s’exécute une fois que l’Action d’échange de jeton personnalisé a été exécutée pour l’utilisateur que vous avez défini pour la transaction. Pour d’autres cas d’utilisation, comme l’ajout de claims personnalisées aux jetons d’accès, utilisez le [déclencheur Post Login Action](/docs/fr-CA/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger).

L’échange de jeton personnalisé utilise le grant type de token exchange, où `event.transaction.protocol` est défini sur `oauth2-token-exchange` dans le déclencheur Post Login Action. L’échange de jeton personnalisé et Native Social Login utilisent tous deux le grant type de token exchange. Pour distinguer les deux, utilisez la valeur de `subject_token_type`, qui doit correspondre à l’un de vos profils d’échange de jeton personnalisé.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  La version Early Access de l’échange de jeton personnalisé ne prend pas en charge `api.authentication.challengeWith()` ou `api.authentication.enrollWith()`. Si vous utilisez ces méthodes dans vos Actions Post Login, la transaction échouera avec une erreur non récupérable. Veillez à ignorer ces deux méthodes lorsque `event.transaction.protocol==oauth2-token-exchange`, selon la valeur de `subject_token_type`.
</Callout>

<div id="actor-support-for-delegation">
  ## Prise en charge de l’acteur pour la délégation
</div>

Le déclencheur `custom-token-exchange` prend en charge les scénarios d’autorisation déléguée. L’Action peut appeler [`api.authentication.setActor()`](/docs/fr-CA/customize/actions/explore-triggers/signup-and-login-triggers/custom-token-exchange-trigger/custom-token-exchange-api-object#api-authentication-setactor-actor) pour inclure une claim `act` dans les jetons émis, afin d’identifier le principal qui agit au nom de l’utilisateur subject.

`setActor()` peut être appelé, que la requête inclue ou non un `actor_token`. Le paramètre `actor_token` est l’un des moyens permettant à l’application appelante de transmettre des renseignements sur l’acteur à l’Action, mais l’identité de l’acteur peut aussi être intégrée au `subject_token` lui-même ou obtenue par d’autres moyens personnalisés dans le code de l’Action.

Lorsque la requête inclut `actor_token` et `actor_token_type`, ces valeurs sont disponibles dans [`event.transaction`](/docs/fr-CA/customize/actions/explore-triggers/signup-and-login-triggers/custom-token-exchange-trigger/custom-token-exchange-event-object#event-transaction). Si `actor_token_type` est `urn:ietf:params:oauth:token-type:id_token` et que le jeton est un ID token Auth0 valide provenant du même tenant, le serveur valide automatiquement le jeton et renseigne `event.transaction.actor_token_user` avec le profil utilisateur de l’acteur.

La claim `act` définie au moyen de `setActor()` est aussi disponible dans les Actions post-login via [`event.transaction.actor`](/docs/fr-CA/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger/post-login-event-object#event-transaction).

<div id="triggers">
  ## Déclencheurs
</div>

<div id="custom-token-exchange">
  ### Échange de jeton personnalisé
</div>

Le déclencheur `custom-token-exchange` est la première étape du pipeline d’Auth0. Cette fonction s’exécute une fois la requête d’échange de jeton personnalisé reçue validée et mappée avec succès à un profil existant.
