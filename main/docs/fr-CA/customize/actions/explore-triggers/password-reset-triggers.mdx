Les déclencheurs de réinitialisation du mot de passe s’exécutent pendant le processus de réinitialisation du mot de passe lorsqu’un utilisateur franchit le premier défi, généralement en cliquant sur un lien envoyé à l’[adresse courriel de l’utilisateur](/docs/fr-CA/secure/multi-factor-authentication/authenticate-using-ropg-flow-with-mfa/enroll-and-challenge-email-authenticators), mais avant qu’un nouveau mot de passe soit défini. Vous pouvez les utiliser pour soumettre l’utilisateur à un facteur supplémentaire d’<Tooltip tip="Authentification multifacteur (MFA) : processus d’authentification de l’utilisateur qui utilise un facteur en plus du nom d’utilisateur et du mot de passe, comme un code envoyé par SMS." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=multi-factor+authentication">authentification multifacteur</Tooltip> (MFA) ou pour rediriger l’utilisateur vers un site externe, comme un service de vérification tiers.

Après la vérification, les utilisateurs peuvent saisir le nouveau mot de passe de leur compte.

<Frame>![Pour utiliser le flux de réinitialisation du mot de passe, accédez à Dashboard > Actions > Flows ](/docs/images/cdy7uua7fh8z/1Pugm9fFYCmdhh7QO5Klzm/7f7eefbf810fd4ebeb22d6393151ffa2/2023-08-14_15-54-03.png)</Frame>

Les Actions de ce flux sont bloquantes (synchrones), ce qui signifie qu’elles s’exécutent dans le cadre du processus d’un déclencheur et empêchent le reste du pipeline Auth0 de s’exécuter tant que l’Action n’est pas terminée.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Pour que ces Actions s’exécutent correctement, vous devez avoir Universal Login activé. Ces Actions ne peuvent pas être déclenchées lorsque vous utilisez Classic Login.
</Callout>

<div id="triggers">
  ## Déclencheurs
</div>

<div id="postchallenge">
  ### PostChallenge
</div>

Le trigger `post-challenge` est une fonction qui s’exécute après qu’un utilisateur a terminé le premier défi de réinitialisation du mot de passe, généralement au moyen d’un lien magique envoyé par courriel. Vous pouvez créer jusqu’à quatre Actions dans votre tenant qui utilisent le trigger `post-challenge`.

<div id="references">
  #### Références
</div>

* [Objet Event](/docs/fr-CA/customize/actions/explore-triggers/password-reset-triggers/post-challenge-trigger/post-challenge-event-object) : Fournit des informations contextuelles sur un utilisateur qui se connecte via Auth0.
* [Objet API](/docs/fr-CA/customize/actions/explore-triggers/password-reset-triggers/post-challenge-trigger/post-challenge-api-object) : Fournit des méthodes permettant de modifier le comportement du flux.

<div id="limitations">
  ## Limites
</div>

Les déclencheurs de réinitialisation du mot de passe ne sont pas compatibles avec les [connexions Active Directory/LDAP](/docs/fr-CA/authenticate/identity-providers/enterprise-identity-providers/active-directory-ldap).

<div id="common-use-cases">
  ## Cas d’utilisation courants
</div>

<div id="secure-password-reset-with-additional-mfa-factors">
  ### Réinitialisation sécurisée du mot de passe avec des facteurs MFA supplémentaires
</div>

Une Action `password-reset` / `post-challenge` peut déclencher un défi MFA une fois que l&#39;utilisateur a terminé le premier défi. Par exemple, vous pouvez déclencher un défi fondé sur WebAuthn comme facteur secondaire si WebAuthN est activé comme facteur dans votre tenant.

```javascript lines
/**@type {PasswordResetPostChallengeAction}**/
module.exports.onExecutePostChallenge = async (event, api) => {
  const enrolledFactors = event.user.enrolledFactors.map((x) => ({
    type: x.type
  }));
  api.authentication.challengeWith({ type: 'webauthn-roaming' }, { additionalFactors: enrolledFactors });
};
```

<div id="redirect-users-to-a-third-party-application">
  ### Rediriger les utilisateurs vers une application tierce
</div>

En plus d’un défi MFA, vous pouvez aussi essayer d’ajouter une redirection dans l’Action personnalisée, par exemple vers un service tiers de vérification ou d’évaluation des risques.

```javascript lines
/** @type {PasswordResetPostChallengeAction}
 * Cet exemple d'action redirige l'utilisateur vers une application exemple
 * et reprend l'action après la redirection pour soumettre
 * l'utilisateur à un facteur d'AMF
 */

module.exports.onExecutePostChallenge = async (event, api) => {
  // Envoyer l'utilisateur vers https://my-app.example.com
  api.redirect.sendUserTo('https://my-app.example.com');
};

module.exports.onContinuePostChallenge = async (event, api) => {
  const enrolledFactors = event.user.enrolledFactors.map((x) => ({
    type: x.type
  }));

  // Soumettre l'utilisateur à un défi par OTP par courriel OU par un autre facteur inscrit
  api.authentication.challengeWith({ type: 'email' }, { additionalFactors: enrolledFactors });

  // Exemple de défi de l'utilisateur avec plusieurs options
  // dans ce cas, OTP par courriel OU OTP par SMS
  // api.authentication.challengeWithAny([{ type: 'email' }, { type: 'sms' }]);
};
```

Le pipeline des Actions n’est pas actif pendant qu’Auth0 redirige l’utilisateur. Une fois que l’utilisateur reprend le processus de connexion avec Auth0, le pipeline des Actions reprend aussi. Les Actions exécutées avant la redirection ne sont pas exécutées de nouveau. Pour en savoir plus, consultez [Redirection avec Actions](/docs/fr-CA/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger/redirect-with-actions).
