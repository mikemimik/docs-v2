description: Description de l’utilisation de l’outil de migration des Rules dans l’Auth0 Dashboard
title: Outil de migration
---

Simplifiez le processus de migration des Rules vers les Actions grâce à l’outil de migration des Rules dans l’[Auth0 Dashboard](https://manage.auth0.com/). En plus de faciliter le passage d’une Rule à une Action, ce nouvel outil offre aussi des conseils intégrés et des correctifs pour les modèles de Rules courants.

<div id="before-migration">
  ## Avant la migration
</div>

Avant de commencer la migration, consultez [Migrer des Rules vers les Actions](/docs/fr-CA/customize/actions/migrate/migrate-from-rules-to-actions). Ce guide de migration est une excellente ressource qui explique les différences entre les Rules et les Actions et qui vous sera utile tout au long du processus de migration.

<div id="rule-migration-order">
  ### Ordre de migration des Rules
</div>

Les Rules s’exécutent avant les Actions, ce qui signifie qu’elles doivent être migrées dans l’ordre inverse de leur environnement d’exécution. Migrez d’abord la dernière Rule à s’exécuter dans votre flux actuel, puis remontez jusqu’à la première Rule, dans l’ordre chronologique inverse.

Le non-respect de cet ordre de migration peut entraîner des erreurs en raison d’effets secondaires imprévus causés par des dépendances dans l’ordre des Rules et des Actions.

<div id="safely-rolling-back">
  ### Revenir en arrière en toute sécurité
</div>

L’outil de migration vous permet de revenir en arrière facilement et en toute sécurité si quelque chose se passe mal. Réactivez votre Rule et ignorez la logique Action correspondante.

<div id="migrating-a-rule">
  ## Migration d’une Rule
</div>

Pour commencer à migrer votre Rule, sélectionnez **Auth Pipeline &gt; Rules**. Une fois cette option sélectionnée, vos Rules s’affichent avec l’option de migrer chacune d’elles individuellement.

<Frame>![](/docs/images/cdy7uua7fh8z/79DmJp8C2vYZpdos51Z0nM/5a665d04c66dd9f0b8f0e25198cb9e4e/SS1_Dashboard_migration_screenshot.png)</Frame>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Le processus de migration commence seulement lorsque vous choisissez **Migrate to Action**. En tant que développeur, vous pouvez ensuite choisir quand activer la nouvelle Action.
</Callout>

<div id="migrate-to-action">
  ### Migrate to Action
</div>

Après avoir sélectionné **Migrate to Action**, l’outil vous invite à créer une nouvelle Action.

<Frame>![](/docs/images/cdy7uua7fh8z/1A8FnVyvqAwW8hbrkOQDoG/f83ba9d5157db14ac2a3a666b2280574/SS2_Migrate_to_Action_screenshot.png)</Frame>

Après avoir sélectionné **Create,** la nouvelle Action est créée et l’Actions Editor s’affiche.

<Frame>![](/docs/images/cdy7uua7fh8z/2A10fE69aVngEcL2ZQCqWX/720dca2a8e6005c9378efad3bc140af0/Screenshot_2024-06-06_at_11.55.38_AM.png)</Frame>

Si la Rule est migrée vers Actions et qu’elle est toujours active, la Rule s’arrête avant que la logique ne s’exécute.

Cela se fait en appelant `api.rules.wasExecuted()`. Si la fonction `wasExecuted` reçoit une chaîne contenant l’ID de la Rule, elle renvoie `true` si la Rule a déjà été exécutée (sinon, elle renvoie `false`).

C’est un outil utile pour éviter que votre logique ne s’exécute deux fois. Dès que vous désactivez la Rule référencée, la fonction renvoie `false` et la logique métier s’exécute alors dans l’Action.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  La valeur de l’ID de la Rule est renseignée automatiquement pour toute rule migrée à l’aide de la fonctionnalité **Migrate to Action**.
</Callout>

Pour en savoir plus sur la fonction `api.rules.wasExecuted()`, consultez [Actions Triggers: post-login - API Object](/docs/fr-CA/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger/post-login-api-object).

<div id="rewrite-rule-code">
  ## Réécrire le code de la Rule
</div>

Sélectionnez le bouton **Rules** dans l’Actions Editor pour charger l’outil de migration des Rules. Vous pourrez alors consulter rapidement le code de la Rule migrée. Vous devriez vérifier le code de la Rule sélectionnée dans la barre latérale.

Copiez le contenu de la fonction Rule et collez-le sous le commentaire `// YOUR CODE HERE` dans l’Action. Il est probable que ce code ne soit pas adapté au modèle de programmation des Actions.

<Frame>![](/docs/images/cdy7uua7fh8z/2jWCZFnfERuN7NzBPrpCDn/80ba271699893a007dac7baaf2dd44cb/Screenshot_2024-06-06_at_11.57.03_AM.png)</Frame>

L’Actions Editor indique les endroits où le code doit être remanié. Il propose aussi des suggestions pour appliquer automatiquement un correctif dans la plupart des cas — pour consulter la suggestion ou le correctif, survolez le code souligné :

<Frame>![](/docs/images/cdy7uua7fh8z/5jRGI4iDX9ZgwcPRADTbZ1/caa90923a76800526871b137806e2af3/Screenshot_2024-06-06_at_11.58.07_AM.png)</Frame>

Vous pourrez peut-être appliquer des mises à jour propres aux Actions en sélectionnant une option dans la fenêtre modale **Quick Fix**.

<Frame>![](/docs/images/cdy7uua7fh8z/qTS4RD6cjWFVfI0T9Uph3/45776c464f7247ddd8e91860f4646f2e/Screenshot_2024-06-06_at_11.58.48_AM.png)</Frame>

Lorsque vous sélectionnez le correctif recommandé, votre code est automatiquement mis à jour pour devenir compatible.

<Frame>![](/docs/images/cdy7uua7fh8z/1cAq5A1MNVLPAsinHjtROh/c71276afd1f6df9e01d71cc869778907/Screenshot_2024-06-06_at_11.59.37_AM.png)</Frame>

Dans certaines situations, les correctifs ne peuvent pas être appliqués automatiquement, mais des indications sont tout de même fournies :

<Frame>![](/docs/images/cdy7uua7fh8z/4Lx3RxJh8v4tj1J2SpPVvU/0ff35b0f40bb24d195ab2f4b06817d86/Screenshot_2024-06-06_at_12.00.22_PM.png)</Frame>

Par exemple : dans le cas de la fonction `callback` de la Rule, il est recommandé d’utiliser [`api.access.deny`](/docs/fr-CA/customize/actions/explore-triggers/signup-and-login-triggers/pre-user-registration-trigger/pre-user-registration-api-object) pour les erreurs, ou simplement `return` lorsque tout se passe bien.

<div id="test-the-new-action">
  ### Tester la nouvelle Action
</div>

Les Actions peuvent être testées directement dans l’éditeur, puisqu’il s’agit d’une étape essentielle de la migration. Pour en savoir plus, consultez [Tester les Actions](/docs/fr-CA/customize/actions/test-actions).

<Warning>
  Nous vous recommandons fortement de valider toute la logique de migration dans un tenant de développement avant de l’activer en production.
</Warning>

<div id="automatically-enabling-the-action">
  ### Activation automatique de l’Action
</div>

Lorsque le code de l’Action est mis à jour et que vous êtes prêt à faire le changement :

1. Sélectionnez **Deploy**.
2. Dans l’alerte qui confirme votre déploiement, sélectionnez **Add to flow**.
3. Ajoutez l’Action que vous venez de créer au flux, puis sélectionnez **Apply.**
4. Sélectionnez **Continue**, **puis désactivez toutes les Rules sélectionnées** pour activer simultanément la nouvelle Action et désactiver la Rule d’origine. Si vous n’êtes pas prêt, sélectionnez **Cancel**; vous pourrez désactiver la Rule plus tard pour activer la logique de l’Action migrée.

Pour en savoir plus sur l’ajout de votre Action au flux, consultez [Write Your First Action](/docs/fr-CA/customize/actions/write-your-first-action).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Si vous réactivez une Rule, la logique de l’Action correspondante est ignorée.
</Callout>

<div id="migrate-remaining-rules">
  ## Migrer les Rules restantes
</div>

Cette procédure peut être répétée pour migrer chaque Rule de votre flux de connexion. N’oubliez pas que les Rules doivent être migrées dans l’ordre inverse de leur exécution dans le flux de connexion.
