Les limitations suivantes s’appliquent à l’utilisation des Actions :

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Pour en savoir plus sur les limites des entités, consultez : [Limites des entités des Actions](/docs/fr-CA/troubleshoot/customer-support/operational-policies/entity-limit-policy#actions).
</Callout>

<div id="actions">
  ## Actions
</div>

* Chaque Action ne doit pas dépasser 100 kB. Plus sa taille est grande, plus elle ajoute de latence, ce qui peut nuire aux performances de votre système. Cette limite de taille n&#39;inclut aucun module `npm` susceptible d&#39;être référencé dans une instruction `require`.

<div id="actions-modules">
  ## Actions Modules
</div>

* Chaque Action Module peut avoir des dépendances sur des modules NPM, mais pas sur d’autres Actions Modules.
* **Les Actions Modules n’ont pas leur propre runtime.** Ils s’exécutent dans le runtime de l’Action qui les inclut.

<Warning>
  **Lorsque vous créez un Action Module, Auth0 le génère avec le runtime du builder. Toutefois, lorsque le module est exécuté, il utilise le runtime de l’Action qui l’importe.**

  *Exemple :*

  *Si votre Action Module utilise des appels d’API Node.js obsolètes dans Node.js 22, les Actions qui utilisent le runtime Node.js 18 fonctionneront très bien, mais celles qui utilisent le runtime Node.js 22 échoueront.*
</Warning>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Rédigez vos Actions Modules à l’aide d’API Node.js compatibles avec toutes les versions de runtime que vous prévoyez prendre en charge.
</Callout>

<div id="account-linking-setprimaryuser">
  ## Liaison de comptes (`setPrimaryUser`)
</div>

* `primary_user_id` est limité à 128 caractères
* `setPrimaryUser` ne peut être appelé qu’une fois par transaction
* Toute valeur de userMetadata définie dans la même Action que `setPrimaryUser` sera ignorée et perdue. Les Actions suivantes de la même transaction conserveront userMetadata pour le nouvel utilisateur principal.
* `setPrimaryUser` ne peut pas être utilisé dans la même transaction où une Rule définit `context.primaryUser.`

<div id="cached-data">
  ## Données en cache
</div>

* Les éléments mis en cache sont enregistrés pendant un maximum de 24 heures.
* Un maximum de 20 entrées peuvent être mises en cache par déclencheur.
* Les clés de cache ont une taille maximale de 64 octets et les valeurs ont une taille maximale de 4 kB.
* La taille cumulée des clés mises en cache et de leurs valeurs ne doit pas dépasser 8 kB.
* Le cache devrait être disponible de façon fiable pour toutes les Actions d’un même déclencheur au cours d’une seule exécution. Toutefois, cela n’est pas garanti pour les exécutions subséquentes (comme un flow différent, la connexion d’un autre utilisateur ou le retour d’un utilisateur à partir d’une action de redirection).
* Les Actions qui effectuent une exécution qui rend la main (comme une redirection) peuvent faire en sorte que les actions subséquentes soient planifiées sur une instance distincte avec un état du cache différent. Les données mises en cache peuvent être incohérentes d’une Action à l’autre, même s’il s’agit de la même exécution.

<div id="executions">
  ## Exécutions
</div>

* Chaque exécution d’un déclencheur doit se terminer en 20 secondes ou moins, faute de quoi le traitement se terminera par une erreur. Limiter les requêtes HTTP est le meilleur moyen de respecter cette limite de temps.
* Chaque exécution d’un déclencheur doit se terminer en 20 secondes ou moins, faute de quoi le traitement se terminera par une erreur. Il est nécessaire de limiter les processus trop longs, comme les requêtes HTTP sortantes sans délai d’expiration, pour respecter cette limite de temps. Une Action qui [redirige les utilisateurs vers une page externe](/docs/fr-CA/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger/redirect-with-actions) est soumise à un délai d’expiration distinct avant la redirection, puis à un autre après.
* Un nouvel objet `event.request` est créé chaque fois qu’un déclencheur d’Action est suspendu, puis repris par la suite (par exemple, en raison d’une redirection ou d’une demande de vérification <Tooltip tip="Management API : un produit permettant aux clients d’effectuer des tâches administratives." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=MFA">MFA</Tooltip>).

<div id="logs">
  ## Logs
</div>

* Un maximum de 256 caractères peut être enregistré de façon persistante pour les sorties `console.log()` de chaque Action.
* Les logs d’exécution sont conservés pendant 10 jours.

<div id="programming-languages">
  ## Langages de programmation
</div>

* Nous ne prenons pas en charge TypeScript dans les Actions. Les fichiers source doivent être écrits en JavaScript avant d’être déployés.

<div id="secrets">
  ## Secrets
</div>

* La clé de chaque secret peut comporter au maximum 128 caractères.
* La valeur de chaque secret peut comporter au maximum 4096 caractères.

<div id="saml-attributes">
  ## Attributs SAML
</div>

* Il est possible de modifier ou d’ajouter un maximum de 100 attributs <Tooltip tip="Authentification multifacteur (MFA) : processus d’authentification de l’utilisateur faisant appel à un facteur en plus du nom d’utilisateur et du mot de passe, comme un code envoyé par SMS." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=SAML">SAML</Tooltip> au moyen d’Actions.
* Les noms d’attribut SAML ont une taille maximale de 1 kB.
* Les valeurs SAML ont une taille maximale de 2 kB.
* La taille totale des assertions SAML ne peut pas dépasser 10 kB.

<div id="saml-configuration">
  ## Configuration de SAML
</div>

* `audience` a une taille maximale de 2kB
* `recipient` a une taille maximale de 2kB
* `destination` a une taille maximale de 2kB
* `nameIdentifierFormat` a une taille maximale de 0.5kB
* `nameIdentifierProbes` a un maximum de 10 probes, chacune d’une taille maximale de 0.5kB
* `authnContextClassRef` a une taille maximale de 0.5kB
* `signingCert` a une taille maximale de 4kB
* `encryptionCert` a une taille maximale de 4kB
* `encryptionPublicKey` a une taille maximale de 4kB
* `cert` a une taille maximale de 4kB
* `key` a une taille maximale de 4kB

<div id="service-requests">
  ## Requêtes de service
</div>

* Les appels effectués à l’API Auth0 <Tooltip tip="Security Assertion Markup Language (SAML) : protocole normalisé permettant à deux parties d’échanger des renseignements d’authentification sans mot de passe." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Management+API">Management API</Tooltip> ainsi que les mises à jour des User Metadata sont [assujettis à des limites de débit](/docs/fr-CA/troubleshoot/customer-support/operational-policies/rate-limit-policy/management-api-endpoint-rate-limits).
* Les appels HTTP vers des services externes proviennent de [ces adresses IP](/docs/fr-CA/secure/security-guidance/data-security/allowlist).

<div id="transaction-metadata">
  ## Métadonnées de transaction
</div>

* Sont disponibles uniquement dans les [Actions post-login](/docs/fr-CA/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger).
* Ne sont pas enregistrées au-delà de la fin d’un déclencheur d’authentification.
* Ne sont pas accessibles à l’extérieur des Actions d’une même transaction.
* Les clés sont limitées à 64 caractères.
* Les valeurs sont limitées à 8 Ko.
* Les valeurs prennent uniquement en charge les types `string`, `number` et `boolean`.
* La taille totale maximale des métadonnées est de 16 Ko dans une même transaction.
* N’acceptent pas comme valeurs valides les nombres qui échouent à une vérification de sécurité. Les entiers non sécuritaires doivent être sérialisés de façon sécuritaire par le développeur. Pour en savoir plus, consultez la documentation sur les [entiers sécuritaires](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/Number/isSafeInteger).

<div id="upstream-tokens-from-external-idps">
  ## Jetons provenant d’IdP externes
</div>

* Récupération des jetons d’<Tooltip tip="Fournisseur d’identité (IdP) : service qui stocke et gère les identités numériques." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=IdP">IdP</Tooltip> externes dans le tableau Identities

<div id="user-metadata-and-app-metadata">
  ## Métadonnées utilisateur et métadonnées de l’application
</div>

* Chaque session peut comporter au maximum 32 kB de métadonnées utilisateur enregistrées et 32 kB de métadonnées de l’application enregistrées.
