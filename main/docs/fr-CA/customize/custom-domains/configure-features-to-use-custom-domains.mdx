Pour configurer les fonctionnalités d’Auth0 afin qu’elles utilisent votre <Tooltip tip="Domaine personnalisé : domaine tiers avec un nom spécialisé ou personnalisé." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=custom+domain">domaine personnalisé</Tooltip>, il se peut que vous deviez suivre des étapes supplémentaires selon les fonctionnalités que vous utilisez. Par exemple, vous devrez peut-être apporter des modifications avant de pouvoir utiliser votre domaine personnalisé sur votre page de connexion ou pour appeler vos API.

Si vous utilisez Auth0 depuis un certain temps et que vous décidez d’activer un domaine personnalisé, vous devrez migrer vos applications existantes et mettre à jour les paramètres comme décrit ci-dessous, y compris ceux de tout VPN ou pare-feu que vous utilisez. Notez que les sessions existantes créées à `{yourDomain}` ne seront plus valides une fois que vous commencerez à utiliser votre domaine personnalisé; les utilisateurs devront donc se connecter de nouveau.

  ## Prérequis
</div>

Vous devriez avoir déjà configuré et vérifié votre domaine personnalisé.

<div id="universal-login">
  ## Universal Login
</div>

Si vous utilisez [Auth0 Universal Login](/docs/fr-CA/authenticate/login/auth0-universal-login) et que vous avez personnalisé la page de connexion, vous devez mettre à jour le code pour utiliser votre domaine personnalisé. Si vous utilisez la page de connexion **par défaut** sans la personnaliser, vous n’avez rien à modifier. Pour en savoir plus, consultez <Tooltip tip="Universal Login : votre application est redirigée vers Universal Login, hébergé sur le serveur d’autorisation d’Auth0, afin de vérifier l’identité d’un utilisateur." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Universal+Login">Universal Login</Tooltip> d’Auth0.

Si vous utilisez [Lock for Web](/docs/fr-CA/libraries/lock), vous devez définir les options `configurationBaseUrl` et `overrides`, comme dans l’exemple de script suivant :

```javascript lines
var lock = new Auth0Lock(config.clientID, config.auth0Domain, {
  //code omis par souci de concision
  configurationBaseUrl: config.clientConfigurationBaseUrl,
  overrides: {
  	__tenant: config.auth0Tenant,
  	__token_issuer: config.authorizationServer.issuer
  },
  //code omis par souci de concision
});
```

Si vous utilisez [Auth0.js](/docs/fr-CA/libraries/auth0js) sur la page Universal Login, vous devez définir l’option `overrides`.

```javascript lines
var webAuth = new auth0.WebAuth({
  clientID: config.clientID,
  domain: config.auth0Domain,
  //code omitted par souci de concision
  overrides: {
  	__tenant: config.auth0Tenant,
  	__token_issuer: config.authorizationServer.issuer
  },
  //code omitted par souci de concision
});
```

Dans la plupart des cas, les bibliothèques Auth0.js et Lock récupèrent le nom du tenant (requis pour `/usernamepassword/login`) ainsi que l’émetteur (requis pour la validation de `id_token`) à partir du domaine. Toutefois, si vous êtes un client Private Cloud et que vous utilisez un proxy ou un nom de domaine personnalisé dont le nom de domaine diffère du tenant ou de l’émetteur, vous pouvez utiliser `__tenant` et `__token_issuer` pour fournir vos propres valeurs.

<div id="embedded-lock">
  ## Embedded Lock
</div>

Si vous utilisez [Lock for Web](/docs/fr-CA/libraries/lock) en mode intégré dans votre application, vous devez mettre à jour le code pour qu’il utilise votre domaine personnalisé lors de l’initialisation de Lock. Vous devrez aussi définir `configurationBaseUrl` avec l’URL CDN appropriée.

{/* codeblockOld.header.login.logInButton codeblockOld.header.login.configureSnippet */}

export const codeExample1 = `var lock = new Auth0Lock('{yourClientId}', '{yourCustomDomain}', {
  //code omis par souci de brièveté
  configurationBaseUrl: 'https://cdn.us.auth0.com'
  //code omis par souci de brièveté
});`;

<AuthCodeBlock children={codeExample1} language="javascript" />

L’URL du CDN varie selon la région. Utilisez `https://cdn.[us|eu|au|jp].auth0.com` (`us` pour les États-Unis, `eu` pour l’Europe, `au` pour l’Australie ou `jp` pour le Japon).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  L’URL du CDN varie selon la région. Les tenants créés avant le 11 juin 2020 doivent utiliser `https://cdn.auth0.com` si la région est celle des États-Unis, ou ajouter `eu`, `au` ou `jp` pour l’Europe, l’Australie ou le Japon. Si votre tenant a été créé après le 11 juin 2020, utilisez `https://cdn.us.auth0.com` si la région est celle des États-Unis.
</Callout>

<div id="use-custom-domains-in-emails-and-phone-notifications">
  ## Utiliser des domaines personnalisés dans les courriels et les notifications téléphoniques
</div>

Si vous souhaitez utiliser votre domaine personnalisé avec vos courriels Auth0 ou vos notifications téléphoniques, vous devez activer cette fonctionnalité.

1. Accédez à [Auth0 Dashboard &gt; Branding &gt; Domaines personnalisés](https://manage.auth0.com/#/branding/custom_domains).
2. Activez la bascule **Utiliser un domaine personnalisé dans les courriels**.

<div id="configure-social-identity-providers">
  ## Configurer les fournisseurs d’identité sociaux
</div>

Si vous souhaitez utiliser votre domaine personnalisé avec des <Tooltip tip="Fournisseur d’identité (IdP) : service qui stocke et gère les identités numériques." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=identity+providers">fournisseurs d’identité</Tooltip> sociaux (IdP), vous devez mettre à jour la liste des URI de redirection autorisées de votre IdP pour y inclure votre domaine personnalisé (par exemple, `https://login.northwind.com/login/callback`).

Vous ne pouvez pas utiliser les [clés de développeur Auth0](/docs/fr-CA/authenticate/identity-providers/social-identity-providers/devkeys) avec des domaines personnalisés.

<div id="configure-google-workspace-connections">
  ## Configurer les connexions Google Workspace
</div>

Si vous voulez utiliser votre domaine personnalisé avec les connexions Google Workspace, vous devez mettre à jour l’URI de redirection autorisée dans les paramètres de votre client <Tooltip tip="OAuth 2.0 : cadre d’autorisation qui définit les protocoles et les flux de travail d’autorisation." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=OAuth">OAuth</Tooltip>. Dans Google Cloud Console, accédez à **Credentials**, choisissez votre client OAuth dans la liste, puis vous verrez une page de paramètres contenant le <Tooltip tip="Client ID : valeur d’identification attribuée par Auth0 à votre ressource enregistrée." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Client+ID">Client ID</Tooltip> de l’application, le secret et d’autres champs. Dans le champ **Authorized Redirect URIs**, ajoutez une URL au format `https://<YOUR-CUSTOM-DOMAIN>/login/callback` qui inclut votre domaine personnalisé (par exemple, `https://login.northwind.com/login/callback`).

<div id="apis">
  ## APIs
</div>

Les identifiants d’API (c.-à-d. `audience`) ne changeront pas. Il s’agit d’une valeur constante pour chaque API et, même s’il est courant d’utiliser un URI, elle est entièrement indépendante du domaine utilisé pour obtenir le jeton.

Auth0 émet des jetons dont la revendication `iss` correspond au domaine que vous avez utilisé pour obtenir le jeton.

<div id="auth0-apis">
  ##### API d’Auth0
</div>

Continuez à utiliser le nom de domaine par défaut de votre tenant (comme `https://{yourDomain}/userinfo` et `https://{yourDomain}/api/v2/`) au lieu de votre domaine personnalisé lorsque vous indiquez une `audience`. C’est le seul endroit où vous devez utiliser le nom de domaine par défaut de votre tenant.

Toutes les requêtes (c.-à-d. l’obtention du jeton et l’appel à l’API proprement dit) doivent utiliser le même domaine. Les jetons obtenus au moyen d’un domaine personnalisé doivent être utilisés avec une API d’Auth0 en utilisant ce même domaine personnalisé.

Si vous utilisez un flux d’authentification avec votre domaine personnalisé pour demander des <Tooltip tip="Jetons d’accès : informations d’autorisation d’accès, sous la forme d’une chaîne opaque ou d’un JWT, utilisées pour accéder à une API." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Access+Tokens">jetons d’accès</Tooltip> afin d’accéder à la <Tooltip tip="Management API : un produit qui permet aux clients d’effectuer des tâches administratives." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Management+API">Management API</Tooltip>, vous devez aussi appeler le point de terminaison de la Management API avec votre domaine personnalisé.

```json lines
POST https://mycustomdomain.com/oauth/token
... // autres paramètres 
...
audience:https://defaulttenant.eu.auth0.com/api/v2/
```

Votre requête de jeton d’accès devrait ressembler à ceci

```json lines
GET https://mycustomdomain.com/api/v2/clients

Headers:
Authorization: Bearer <access_token>
```

<div id="custom-apis">
  ##### API personnalisées
</div>

Si vous utilisez Auth0 avec un domaine personnalisé pour émettre des jetons d’accès pour vos API, vous devez valider le ou les émetteurs du <Tooltip tip="JSON Web Token (JWT) : format standard de ID Token (et souvent de Access Token) servant à représenter de façon sécurisée des claims entre deux parties." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=JWT">JWT</Tooltip> en fonction de votre domaine personnalisé. Par exemple, si vous utilisez le middleware [express-jwt](https://github.com/auth0/express-jwt), vous devez apporter la modification suivante :

```js lines
app.use(jwt({
  issuer: 'https://<YOUR-CUSTOM-DOMAIN>/',
  //code omis par souci de concision
}));
```

<div id="configure-saml-identity-providers">
  ## Configurer les fournisseurs d’identité SAML
</div>

Pour utiliser votre domaine personnalisé avec les fournisseurs d’identité <Tooltip tip="Security Assertion Markup Language (SAML) : protocole normalisé permettant à deux parties d’échanger des renseignements d’authentification sans mot de passe." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=SAML">SAML</Tooltip> (IdPs), vous devez mettre à jour vos **URL Assertion Consumer Service (ACS)** auprès du ou des fournisseurs d’identité. Selon ce que l’IdP prend en charge, vous pouvez le faire de l’une des deux façons suivantes :

1. Vous pouvez obtenir les métadonnées du fournisseur de services à partir d’Auth0 à l’adresse `https://<YOUR-CUSTOM-DOMAIN>/samlp/metadata?connection=<YOUR-CONNECTION-NAME>`. Elles incluront l’URL ACS mise à jour. Vous devrez ensuite mettre à jour manuellement cette valeur dans les paramètres de votre ou vos IdPs. Cette modification de votre ou vos IdPs doit être effectuée au même moment où vous commencerez à utiliser votre domaine personnalisé dans vos applications. Cela peut poser problème s’il y a plusieurs IdPs à configurer.
2. Si l’IdP le prend en charge, vous pouvez utiliser des requêtes signées pour répondre à cette exigence :

* Téléchargez le certificat de signature à partir de `https://<TENANT>.auth0.com/pem`. Notez que `https://<YOUR-CUSTOM-DOMAIN>.com/pem` renverra le même certificat
* Remettez le certificat au ou aux IdPs afin qu’ils puissent le téléverser. Cela permet à l’IdP de valider la signature du message `AuthnRequest` qu’Auth0 envoie à l’IdP
* L’IdP importera le certificat et, au besoin, la vérification de la signature devra être activée (les étapes exactes varient selon l’IdP)
* Activez la **bascule Sign Request** dans le Dashboard sous **Connections &gt; Enterprise &gt; SAML &gt; CONNECTION**. Cela fera en sorte qu’Auth0 signe les messages SAML `AuthnRequest` qu’il envoie à l’IdP.

Une fois cela fait, lorsque vous commencerez à utiliser votre domaine personnalisé pour lancer une requête d’authentification dans votre application, l’IdP recevra ce domaine personnalisé dans votre requête signée. Comme la requête signée de votre application est fiable, l’IdP devrait automatiquement remplacer toute URL ACS configurée par la valeur envoyée dans la requête signée. Toutefois, certains IdPs n’acceptent **pas** l’URL ACS dans la requête signée, vous devez donc d’abord vérifier auprès du vôtre si cette fonction est prise en charge.

Si c’est pris en charge, cela vous évitera d’avoir à modifier les paramètres d’un ou de plusieurs IdPs en même temps et vous permettra de les préparer à accepter vos requêtes signées à l’avance. Vous pourrez aussi modifier plus tard l’URL ACS configurée statiquement dans les paramètres de votre IdP.

Notez que si votre fournisseur d’identité SAML est configuré pour utiliser votre domaine personnalisé, le test de la connexion au moyen du bouton **Try** dans le Dashboard ne fonctionnera **pas** et les liens par défaut pour télécharger les métadonnées depuis Auth0 afficheront toujours le domaine par défaut, et non le domaine personnalisé.

Si vous avez un flux d’authentification initié par l’IdP, vous devrez mettre à jour le ou les IdPs ainsi que votre ou vos applications en même temps pour utiliser le domaine personnalisé.

<div id="configure-saml-applications">
  ## Configurer les applications SAML
</div>

Si vous souhaitez utiliser votre domaine personnalisé avec des applications SAML (lorsqu’Auth0 est l’IdP), vous devez mettre à jour votre fournisseur de services avec les nouvelles métadonnées du fournisseur d’identité d’Auth0. Vous pouvez obtenir les métadonnées à jour correspondant au domaine personnalisé à l’adresse `https://<YOUR-CUSTOM-DOMAIN>/samlp/metadata/<YOUR-CLIENT-ID>`. Notez que l’ID d’entité de l’émetteur de l’assertion renvoyée par Auth0 changera lorsque vous utiliserez un domaine personnalisé (par exemple, de `urn:northwind.auth0.com` à une valeur utilisant le domaine personnalisé, comme `urn:login.northwind.com`).

Si vous avez un flux d’authentification initié par l’IdP, vous devrez mettre à jour l’URL utilisée pour le déclencher afin qu’elle corresponde au domaine personnalisé. Au lieu de `https://<TENANT>.auth0.com/samlp/<YOUR-CLIENT-ID>`, vous devriez utiliser `https://<YOUR-CUSTOM-DOMAIN>/samlp/<YOUR-CLIENT-ID>`.

<div id="configure-ws-fed-applications">
  ## Configurer les applications WS-Fed
</div>

Si vous souhaitez utiliser votre domaine personnalisé avec des applications <Tooltip tip="Web Service Federation (WS-Fed) : protocole de gestion des identités utilisateur entre domaines." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=WS-Fed">WS-Fed</Tooltip> avec Auth0 comme IdP, vous devez mettre à jour votre fournisseur de services avec les nouvelles métadonnées du fournisseur d&#39;identité d’Auth0. Vous pouvez obtenir les métadonnées correspondant à ce domaine personnalisé à l’adresse suivante : `https://<YOUR-CUSTOM-DOMAIN>/wsfed/FederationMetadata/2007-06/FederationMetadata.xml`.

<div id="configure-azure-ad-connections">
  ## Configurer les connexions Azure AD
</div>

Si vous souhaitez utiliser votre domaine personnalisé avec les connexions Azure AD, vous devez mettre à jour l’URL de réponse autorisée dans les paramètres de votre Azure AD. Dans votre Azure Active Directory, allez dans **Apps registrations** et sélectionnez votre application. Cliquez ensuite sur **Settings -&gt; Reply URLs** et ajoutez une URL avec votre domaine personnalisé au format `https://<YOUR-CUSTOM-DOMAIN>/login/callback` (par exemple, `https://login.northwind.com/login/callback`).

<div id="configure-adfs-connections">
  ## Configurer les connexions ADFS
</div>

Si vous souhaitez utiliser votre domaine personnalisé avec des connexions ADFS, vous devez mettre à jour le point de terminaison dans votre configuration ADFS. Vous devrez le modifier pour que l’URL de rappel utilise votre domaine personnalisé, au format `https://<YOUR-CUSTOM-DOMAIN>/login/callback` (par exemple, `https://login.northwind.com/login/callback`).

<div id="configure-adldap-connections">
  ## Configurer les connexions AD/LDAP
</div>

Si vous n’avez pas besoin de Kerberos, les connexions AD/LDAP ne nécessitent aucune configuration supplémentaire.

Pour utiliser des connexions AD/LDAP avec la prise en charge de Kerberos, vous devrez mettre à jour le point de terminaison Ticket afin qu’il fonctionne avec le domaine personnalisé. Comme il est indiqué dans la [documentation du Connector AD/LDAP Auth0](/docs/fr-CA/authenticate/identity-providers/enterprise-identity-providers/active-directory-ldap/ad-ldap-connector/ad-ldap-connector-to-auth0#point-an-ad-ldap-connector-to-a-new-connection), le fichier `config.json` doit être modifié en remplaçant la valeur `PROVISIONING_TICKET` afin d’utiliser votre domaine personnalisé au format `https://<YOUR-CUSTOM-DOMAIN>/p/ad/jUG0dN0R`.

Une fois cette modification enregistrée, vous devez redémarrer le service AD/LDAP Connector pour qu’elle prenne effet.
