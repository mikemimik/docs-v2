[Les clés d’accès](/docs/fr-CA/secure/multi-factor-authentication/fido-authentication-with-webauthn) offrent une expérience d’authentification sans mot de passe résistante à l’hameçonnage grâce à WebAuthn. Lorsqu’on utilise [plusieurs domaines personnalisés](/docs/fr-CA/customize/custom-domains/multiple-custom-domains), les clés d’accès sont inscrites pour chaque domaine en raison du modèle de sécurité de WebAuthn.

<div id="how-passkeys-work-with-custom-domains">
  ## Comment les clés d’accès fonctionnent avec les domaines personnalisés
</div>

<div id="webauthn-relying-party-id-rp-id">
  ### ID de la partie de confiance WebAuthn (RP ID)
</div>

WebAuthn utilise un identifiant de partie de confiance (RP ID) pour définir la portée des informations d’authentification de clé d’accès. Le RP ID détermine :

* **Où les clés d’accès peuvent être utilisées** : les clés d’accès sont associées au domaine où elles ont été créées
* **Limites de sécurité** : empêche l’utilisation de clés d’accès sur des domaines non autorisés
* **Expérience utilisateur** : les utilisateurs doivent inscrire leurs clés d’accès séparément pour chaque domaine personnalisé

<div id="per-domain-enrollment">
  ### Inscription par domaine
</div>

Avec plusieurs domaines personnalisés, chaque domaine a son propre ID RP, ce qui signifie :

* Une clé d’accès inscrite sur `login.brand1.com` **ne peut pas** être utilisée sur `login.brand2.com`
* Les utilisateurs qui s’authentifient au moyen de différents domaines personnalisés doivent inscrire une clé d’accès pour chaque domaine
* Les clés d’accès de chaque domaine sont gérées indépendamment

<div id="understanding-the-passkey-user-experience">
  ## Comprendre l’expérience utilisateur des clés d’accès
</div>

<div id="single-brand-single-domain">
  ### Une seule marque, un seul domaine
</div>

**Configuration**: Un domaine personnalisé pour une seule marque

**Expérience utilisateur**:

1. L’utilisateur visite `login.example.com`
2. L’utilisateur inscrit une clé d’accès
3. L’utilisateur peut utiliser la clé d’accès pour toutes ses connexions futures sur `login.example.com`

**Complexité**: Faible - expérience de clé d’accès simple

<div id="multi-brand-separate-domains">
  ### Plusieurs marques, domaines distincts
</div>

**Configuration**: Plusieurs marques, chacune ayant son propre domaine personnalisé

**Expérience utilisateur**:

1. L’utilisateur visite `login.brand1.com` et inscrit une clé d’accès
2. Le même utilisateur visite plus tard `login.brand2.com` (une autre marque)
3. La clé d’accès enregistrée précédemment n’est pas disponible
4. L’utilisateur doit inscrire une nouvelle clé d’accès pour `login.brand2.com`

**Complexité**: Moyenne - les utilisateurs ont besoin d’une clé d’accès distincte pour chaque marque

**Bonne pratique**: Indiquez aux utilisateurs que chaque marque nécessite l’inscription d’une clé d’accès distincte

<div id="multi-tenant-with-common-domain">
  ### Multilocataire avec domaine commun
</div>

**Configuration**: Plusieurs clients avec un domaine personnalisé commun pour les services partagés.

**Expérience utilisateur**:

1. La plupart des utilisateurs s’authentifient au moyen du domaine commun
2. Les utilisateurs inscrivent leurs clés d’accès une seule fois pour le domaine commun
3. Les clés d’accès fonctionnent de façon uniforme dans la plupart des scénarios d’authentification
4. Les cas particuliers (domaines propres à certains clients) nécessitent une inscription distincte

**Complexité**: Faible à moyenne - la plupart des utilisateurs bénéficient d’une expérience uniforme

<div id="configuration">
  ## Configuration
</div>

<div id="enable-passkeys-for-your-tenant">
  ### Activer les clés d’accès pour votre tenant
</div>

Avant d’utiliser les clés d’accès avec des domaines personnalisés, assurez-vous qu’elles sont activées :

1. Accédez à **Auth0 Dashboard** &gt; **Security** &gt; **Multi-factor Auth**
2. Activez **WebAuthn with FIDO Security Keys**
3. Configurez les paramètres des clés d’accès

<div id="configure-custom-domains-for-passkeys">
  ### Configurer des domaines personnalisés pour les clés d’accès
</div>

Chaque domaine personnalisé se voit automatiquement attribuer son propre ID RP :

* **Format de l’ID RP** : le domaine personnalisé lui-même (p. ex., `login.example.com`)
* **Aucune configuration additionnelle n’est requise** : Auth0 configure automatiquement l’ID RP pour chaque domaine personnalisé vérifié

<div id="verify-rp-id-configuration">
  ### Vérifier la configuration de l’ID RP
</div>

Pour vérifier l’ID RP d’un domaine personnalisé :

1. Accédez à **Auth0 Dashboard** &gt; **Branding** &gt; **Custom Domains**
2. Sélectionnez votre domaine personnalisé
3. Dans les détails du domaine, l’ID RP s’affiche

<div id="implementation-patterns">
  ## Modèles de mise en œuvre
</div>

<div id="prompt-for-passkey-enrollment-per-domain">
  ### Inviter les utilisateurs à inscrire des clés d’accès pour chaque domaine
</div>

Guidez les utilisateurs pour qu’ils inscrivent des clés d’accès pour chaque domaine personnalisé qu’ils utilisent :

```javascript
import { createAuth0Client } from '@auth0/auth0-spa-js';

async function setupPasskeyEnrollment() {
  const auth0 = await createAuth0Client({
    domain: 'login.example.com',
    clientId: 'YOUR_CLIENT_ID'
  });

  // Vérifier si l'utilisateur est authentifié
  const isAuthenticated = await auth0.isAuthenticated();

  if (isAuthenticated) {
    // Vérifier si la clé d'accès est inscrite pour ce domaine
    const user = await auth0.getUser();

    if (!user.passkey_enrolled) {
      // Inviter l'utilisateur à inscrire une clé d'accès
      showPasskeyEnrollmentPrompt();
    }
  }
}

function showPasskeyEnrollmentPrompt() {
  // Afficher l'interface pour encourager l'inscription d'une clé d'accès
  const banner = document.createElement('div');
  banner.innerHTML = `
    <div class="passkey-prompt">
      <p>Set up passkey for faster, more secure login on this site</p>
      <button onclick="enrollPasskey()">Set Up Passkey</button>
    </div>
  `;
  document.body.prepend(banner);
}
```

<div id="track-passkey-enrollment-by-domain">
  ### Suivre l’inscription des clés d’accès par domaine
</div>

Stockez les domaines pour lesquels un utilisateur a inscrit des clés d’accès :

```javascript
// Dans votre Auth0 Action (Post-Login)
exports.onExecutePostLogin = async (event, api) => {
  const domain = event.custom_domain?.domain;
  const authMethods = event.authentication?.methods || [];

  // Vérifier si l'utilisateur s'est authentifié avec une clé d'accès
  const usedPasskey = authMethods.some(method =>
    method.name === 'webauthn' || method.name === 'passkey'
  );

  if (usedPasskey) {
    // Suivre les domaines pour lesquels l'utilisateur a enregistré des clés d'accès
    const enrolledDomains = event.user.app_metadata?.passkey_domains || [];

    if (domain && !enrolledDomains.includes(domain)) {
      enrolledDomains.push(domain);
      api.user.setAppMetadata('passkey_domains', enrolledDomains);
    }

    // Ajouter un claim au jeton
    api.idToken.setCustomClaim('passkey_enrolled', true);
    api.idToken.setCustomClaim('passkey_domain', domain);
  } else {
    // L'utilisateur n'a pas utilisé de clé d'accès
    api.idToken.setCustomClaim('passkey_enrolled', false);
  }
};
```

Ensuite, dans votre application :

```javascript
async function checkPasskeyEnrollment() {
  const auth0 = await createAuth0Client({
    domain: window.CUSTOM_DOMAIN,
    clientId: 'YOUR_CLIENT_ID'
  });

  const isAuthenticated = await auth0.isAuthenticated();

  if (isAuthenticated) {
    const user = await auth0.getUser();
    const claims = await auth0.getIdTokenClaims();

    // Vérifier si la clé d'accès est enregistrée pour le domaine actuel
    const passkeyEnrolledHere = claims.passkey_enrolled &&
                                 claims.passkey_domain === window.CUSTOM_DOMAIN;

    if (!passkeyEnrolledHere) {
      // Inviter l'utilisateur à enregistrer une clé d'accès pour ce domaine
      promptPasskeyEnrollment();
    }
  }
}
```

<div id="domain-specific-enrollment-pages">
  ### Pages d’inscription propres à chaque domaine
</div>

Créez des pages d’inscription distinctes pour chaque domaine personnalisé :

```javascript
// Page d'inscription pour Brand 1
// URL: https://login.brand1.com/enroll-passkey

import { createAuth0Client } from '@auth0/auth0-spa-js';

async function enrollPasskeyForBrand1() {
  const auth0 = await createAuth0Client({
    domain: 'login.brand1.com',
    clientId: 'YOUR_CLIENT_ID'
  });

  try {
    // Déclencher l'inscription de la clé d'accès
    await auth0.loginWithPopup({
      authorizationParams: {
        acr_values: 'http://schemas.openid.net/pape/policies/2007/06/multi-factor',
        prompt: 'login'
      }
    });

    alert('Passkey enrolled successfully for Brand 1!');
  } catch (error) {
    console.error('Passkey enrollment failed:', error);
  }
}
```

<div id="contextual-enrollment-prompts">
  ### Invites contextuelles d’inscription
</div>

Affichez des invites d’inscription aux clés d’accès en fonction du comportement de l’utilisateur. Points à considérer :

* Suivez quand les utilisateurs suppriment les invites d’inscription (stockez l’information dans `localStorage`)
* Vérifiez `logins_count` dans les métadonnées utilisateur pour afficher les invites après plusieurs visites
* Vérifiez qu’une clé d’accès n’est pas déjà inscrite pour le domaine actuel

```javascript
async function shouldShowEnrollmentPrompt(auth0, customDomain) {
  const storageKey = `passkey_prompt_dismissed_${customDomain}`;

  // Ne pas afficher si l'utilisateur l'a supprimé
  if (localStorage.getItem(storageKey)) return false;

  const claims = await auth0.getIdTokenClaims();
  const passkeyEnrolled = claims.passkey_enrolled &&
                          claims.passkey_domain === customDomain;

  if (passkeyEnrolled) return false;

  // Afficher l'invite après la 3e connexion
  const user = await auth0.getUser();
  return (user.logins_count || 0) >= 3;
}
```

<div id="help-documentation">
  ### Documentation d’aide
</div>

Fournissez une documentation d’aide claire :

**Exemple d’entrée de FAQ** :

**Q : Pourquoi dois-je configurer une clé d’accès à nouveau ?**

R : Les clés d’accès sont liées à des domaines précis pour des raisons de sécurité. Si vous vous connectez à partir d’un portail différent (p. ex., Brand A ou Brand B), vous devrez configurer une clé d’accès pour chacun. Cela protège vos comptes en veillant à ce que les clés d’accès ne fonctionnent que là où elles le doivent.

**Q : Ai-je besoin d’un appareil différent pour chaque clé d’accès ?**

R : Non ! Vous pouvez utiliser le même appareil (téléphone, ordinateur ou clé matérielle) pour des clés d’accès associées à différents domaines. Chaque clé d’accès est simplement un identifiant distinct enregistré sur votre appareil.

<div id="related-origins-future-feature">
  ### Related origins (fonctionnalité à venir)
</div>

Auth0 prévoit prendre en charge les related origins de WebAuthn, ce qui permettra le partage de clés d’accès entre les domaines spécifiés. Cette fonctionnalité :

* Vous permettra de configurer des domaines comme « liés » pour les clés d’accès
* Permettra aux utilisateurs d’utiliser une clé d’accès inscrite sur `login.brand1.com` sur `login.brand2.com` si ces domaines sont configurés comme liés
* Offrira plus de souplesse pour les mises en œuvre multimarques

**Statut** : Prévu après la disponibilité générale (GA)

<div id="migration-scenarios">
  ## Scénarios de migration
</div>

<div id="migrating-from-single-custom-domain-to-multiple">
  ### Migration d’un domaine personnalisé unique vers plusieurs domaines personnalisés
</div>

**Avant**: Un domaine personnalisé unique avec des clés d’accès déjà inscrites

**Après**: Plusieurs domaines personnalisés pour différentes marques

**Défi**: Les clés d’accès existantes ne fonctionnent que sur le domaine d’origine

**Approche de migration**:

1. **Conserver le domaine d’origine actif**: Garder le domaine personnalisé d’origine comme domaine commun
2. **Déploiement progressif**: Ajouter progressivement de nouveaux domaines personnalisés
3. **Avis aux utilisateurs**: Informer les utilisateurs qu’ils devront inscrire des clés d’accès sur les nouveaux domaines
4. **Prévoir un parcours de réinscription**: Permettre aux utilisateurs d’inscrire facilement des clés d’accès sur les nouveaux domaines
5. **Suivre l’adoption**: Suivre les taux d’inscription des clés d’accès par domaine

**Modèle de communication**:

> &quot;Nous lançons des pages de connexion propres à chaque marque! Votre clé d’accès actuelle continuera de fonctionner sur [domaine d’origine]. Lorsque vous visiterez nos nouvelles pages de connexion, on vous demandera aussi de configurer une clé d’accès pour vous y connecter plus rapidement.&quot;

<div id="migrating-between-custom-domains">
  ### Migration d’un domaine personnalisé à un autre
</div>

**Scénario** : Passer de `old-domain.com` à `new-domain.com`

**Défi** : Les clés d’accès ne peuvent pas être transférées

**Étapes de la migration** :

1. **Fonctionnement en parallèle** : Faites fonctionner les deux domaines en même temps pendant la transition
2. **Détecter les clés d’accès déjà configurées** : Repérez les utilisateurs qui ont des clés d’accès sur l’ancien domaine
3. **Inviter à se réinscrire** : Lorsque les utilisateurs ouvrent une session avec le nouveau domaine, invitez-les à inscrire une clé d’accès
4. **Période de grâce** : Gardez l’ancien domaine actif pendant une période de transition
5. **Retirer l’ancien domaine** : Une fois l’adoption faite, mettez l’ancien domaine hors service

```javascript
// Dans votre Auth0 Action
exports.onExecutePostLogin = async (event, api) => {
  const domain = event.custom_domain?.domain;
  const oldDomain = 'old-domain.com';
  const newDomain = 'new-domain.com';

  // Vérifier si l'utilisateur avait une clé d'accès sur l'ancien domaine
  const hadOldPasskey = event.user.app_metadata?.passkey_domains?.includes(oldDomain);

  // L'utilisateur est sur le nouveau domaine mais n'a pas encore de clé d'accès inscrite
  if (domain === newDomain && hadOldPasskey) {
    const newDomainPasskeys = event.user.app_metadata?.passkey_domains?.includes(newDomain);

    if (!newDomainPasskeys) {
      // Définir un indicateur pour demander la réinscription
      api.idToken.setCustomClaim('should_enroll_passkey', true);
      api.idToken.setCustomClaim('migrated_from', oldDomain);
    }
  }
};
```

<div id="best-practices">
  ## Bonnes pratiques
</div>

1. **Utilisez un domaine commun** : Utilisez un domaine personnalisé commun pour réduire au minimum le nombre de domaines nécessitant l’inscription d’une clé d’accès
2. **Communication claire** : Informez les utilisateurs des exigences d’inscription propres à chaque domaine
3. **Invitez les utilisateurs de façon stratégique** : Affichez les invites d’inscription après que les utilisateurs ont démontré un certain engagement (p. ex., 3 connexions ou plus)
4. **Suivez les inscriptions** : Surveillez quels utilisateurs ont inscrit des clés d’accès sur quels domaines
5. **Offrez de l’aide** : Proposez une documentation claire et du soutien pour la gestion des clés d’accès
6. **Testez minutieusement** : Testez les flux de clés d’accès sur tous les domaines personnalisés avant le déploiement en production
7. **Planifiez les migrations** : Lorsque vous changez de domaine personnalisé, prévoyez la réinscription des utilisateurs
8. **Surveillez l’adoption** : Suivez les taux d’inscription et d’utilisation des clés d’accès par domaine

<div id="troubleshooting">
  ## Dépannage
</div>

<div id="passkey-not-working-on-custom-domain">
  ### Clé d’accès ne fonctionne pas sur le domaine personnalisé
</div>

**Symptômes** : L’utilisateur a configuré une clé d’accès, mais ne peut pas l’utiliser

**Causes possibles** :

* L’utilisateur se trouve sur un domaine personnalisé différent de celui où il a configuré sa clé d’accès
* Problèmes de compatibilité du navigateur
* La clé d’accès a été supprimée de l’appareil

**Résolution** :

1. Confirmez que l’utilisateur se trouve sur le bon domaine personnalisé
2. Vérifiez si le navigateur prend en charge WebAuthn
3. Au besoin, demandez à l’utilisateur d’inscrire de nouveau sa clé d’accès

<div id="user-confused-about-multiple-enrollments">
  ### L’utilisateur ne comprend pas pourquoi il y a plusieurs inscriptions
</div>

**Symptômes** : L’utilisateur signale que la « clé d’accès ne fonctionne pas » lorsqu’il change de domaine

**Cause** : L’utilisateur ne comprend pas que l’inscription se fait pour chaque domaine

**Résolution** :

1. Fournir des messages clairs sur les clés d’accès propres à chaque domaine
2. Indiquer pour quels domaines l’utilisateur a inscrit des clés d’accès
3. Proposer l’inscription lorsqu’un utilisateur visite un nouveau domaine
