Les [clés d’accès](/docs/fr-CA/secure/multi-factor-authentication/fido-authentication-with-webauthn) offrent une expérience d’authentification sans mot de passe résistante à l’hameçonnage grâce à WebAuthn. Lorsqu’on utilise [plusieurs domaines personnalisés](/docs/fr-CA/customize/custom-domains/multiple-custom-domains), les clés d’accès doivent être enregistrées séparément pour chaque domaine en raison du modèle de sécurité de WebAuthn.

<div id="how-passkeys-work-with-custom-domains">
  ## Fonctionnement des clés d’accès avec les domaines personnalisés
</div>

<div id="webauthn-relying-party-id-rp-id">
  ### ID de partie de confiance WebAuthn (RP ID)
</div>

WebAuthn utilise un identifiant de partie de confiance (RP ID) pour définir la portée des informations d’identification des clés d’accès. Le RP ID détermine :

* **Où les clés d’accès peuvent être utilisées** : les clés d’accès sont associées au domaine où elles ont été créées
* **Limites de sécurité** : empêche l’utilisation des clés d’accès sur des domaines non autorisés
* **Expérience utilisateur** : les utilisateurs doivent inscrire des clés d’accès séparément pour chaque domaine personnalisé

<div id="per-domain-enrollment">
  ### Inscription propre à chaque domaine
</div>

Avec plusieurs domaines personnalisés, chaque domaine a son propre RP ID, ce qui signifie :

* Une clé d’accès inscrite sur `login.brand1.com` **ne peut pas** être utilisée sur `login.brand2.com`
* Les utilisateurs qui s’authentifient à partir de différents domaines personnalisés doivent inscrire des clés d’accès pour chaque domaine
* Les clés d’accès de chaque domaine sont gérées indépendamment

<div id="understanding-the-passkey-user-experience">
  ## Comprendre l’expérience utilisateur des clés d’accès
</div>

<div id="single-brand-single-domain">
  ### Une seule marque, un seul domaine
</div>

**Configuration** : Un domaine personnalisé pour une seule marque

**Expérience utilisateur** :

1. L’utilisateur accède à `login.example.com`
2. L’utilisateur configure une clé d’accès
3. L’utilisateur peut utiliser cette clé d’accès pour toutes les connexions futures via `login.example.com`

**Complexité** : Faible - expérience de clé d’accès simple et directe

<div id="multi-brand-separate-domains">
  ### Multimarque, domaines distincts
</div>

**Configuration** : Plusieurs marques, chacune avec son propre domaine personnalisé

**Expérience utilisateur** :

1. L’utilisateur se rend sur `login.brand1.com` et y enregistre une clé d’accès
2. Plus tard, ce même utilisateur se rend sur `login.brand2.com` (une autre marque)
3. La clé d’accès enregistrée précédemment n’est pas disponible
4. L’utilisateur doit enregistrer une nouvelle clé d’accès pour `login.brand2.com`

**Complexité** : Moyenne - les utilisateurs ont besoin de clés d’accès distinctes pour chaque marque

**Bonne pratique** : Indiquez aux utilisateurs que chaque marque nécessite l’inscription distincte d’une clé d’accès

<div id="multi-tenant-with-common-domain">
  ### Multilocataire avec domaine commun
</div>

**Configuration**: Plusieurs clients, avec un domaine personnalisé commun pour les services partagés.

**Expérience utilisateur**:

1. La plupart des utilisateurs s’authentifient via le domaine commun
2. Les utilisateurs n’inscrivent leurs clés d’accès qu’une seule fois pour le domaine commun
3. Les clés d’accès fonctionnent de manière cohérente dans la plupart des scénarios d’authentification
4. Les cas particuliers (domaines propres à un client) nécessitent une inscription distincte

**Complexité**: Faible à moyenne - la plupart des utilisateurs bénéficient d’une expérience cohérente

<div id="configuration">
  ## Configuration
</div>

<div id="enable-passkeys-for-your-tenant">
  ### Activez les clés d’accès dans votre tenant
</div>

Avant d’utiliser les clés d’accès avec des domaines personnalisés, assurez-vous qu’elles sont activées :

1. Dans le **Auth0 Dashboard**, accédez à **Sécurité** &gt; **Multi-factor Auth**
2. Activez **WebAuthn avec des clés de sécurité FIDO**
3. Configurez les paramètres des clés d’accès

<div id="configure-custom-domains-for-passkeys">
  ### Configurer les domaines personnalisés pour les clés d’accès
</div>

Chaque domaine personnalisé reçoit automatiquement son propre RP ID :

* **Format du RP ID** : Le domaine personnalisé lui-même (p. ex., `login.example.com`)
* **Aucune configuration supplémentaire n’est nécessaire** : Auth0 configure automatiquement le RP ID pour chaque domaine personnalisé vérifié

<div id="verify-rp-id-configuration">
  ### Vérification de la configuration du RP ID
</div>

Pour vérifier le RP ID d’un domaine personnalisé :

1. Accédez à **Auth0 Dashboard** &gt; **Branding** &gt; **Custom Domains**
2. Sélectionnez votre domaine personnalisé
3. Dans les détails du domaine, le RP ID s’affiche

<div id="implementation-patterns">
  ## Schémas de mise en œuvre
</div>

<div id="prompt-for-passkey-enrollment-per-domain">
  ### Demande d’inscription de clés d’accès par domaine
</div>

Incitez les utilisateurs à inscrire des clés d’accès pour chaque domaine personnalisé qu’ils utilisent :

```javascript
import { createAuth0Client } from '@auth0/auth0-spa-js';

async function setupPasskeyEnrollment() {
  const auth0 = await createAuth0Client({
    domain: 'login.example.com',
    clientId: 'YOUR_CLIENT_ID'
  });

  // Vérifier si l'utilisateur est authentifié
  const isAuthenticated = await auth0.isAuthenticated();

  if (isAuthenticated) {
    // Vérifier si la clé d'accès est inscrite pour ce domaine
    const user = await auth0.getUser();

    if (!user.passkey_enrolled) {
      // Inviter l'utilisateur à inscrire une clé d'accès
      showPasskeyEnrollmentPrompt();
    }
  }
}

function showPasskeyEnrollmentPrompt() {
  // Afficher l'interface pour encourager l'inscription d'une clé d'accès
  const banner = document.createElement('div');
  banner.innerHTML = `
    <div class="passkey-prompt">
      <p>Set up passkey for faster, more secure login on this site</p>
      <button onclick="enrollPasskey()">Set Up Passkey</button>
    </div>
  `;
  document.body.prepend(banner);
}
```

<div id="track-passkey-enrollment-by-domain">
  ### Suivre l’inscription des clés d’accès par domaine
</div>

Stockez les domaines pour lesquels un utilisateur a inscrit des clés d’accès :

```javascript
// Dans votre Auth0 Action (Post-Login)
exports.onExecutePostLogin = async (event, api) => {
  const domain = event.custom_domain?.domain;
  const authMethods = event.authentication?.methods || [];

  // Vérifier si l'utilisateur s'est authentifié avec une clé d'accès
  const usedPasskey = authMethods.some(method =>
    method.name === 'webauthn' || method.name === 'passkey'
  );

  if (usedPasskey) {
    // Suivre les domaines pour lesquels l'utilisateur a enregistré des clés d'accès
    const enrolledDomains = event.user.app_metadata?.passkey_domains || [];

    if (domain && !enrolledDomains.includes(domain)) {
      enrolledDomains.push(domain);
      api.user.setAppMetadata('passkey_domains', enrolledDomains);
    }

    // Ajouter un claim au jeton
    api.idToken.setCustomClaim('passkey_enrolled', true);
    api.idToken.setCustomClaim('passkey_domain', domain);
  } else {
    // L'utilisateur n'a pas utilisé de clé d'accès
    api.idToken.setCustomClaim('passkey_enrolled', false);
  }
};
```

Ensuite, dans votre application :

```javascript
async function checkPasskeyEnrollment() {
  const auth0 = await createAuth0Client({
    domain: window.CUSTOM_DOMAIN,
    clientId: 'YOUR_CLIENT_ID'
  });

  const isAuthenticated = await auth0.isAuthenticated();

  if (isAuthenticated) {
    const user = await auth0.getUser();
    const claims = await auth0.getIdTokenClaims();

    // Vérifier si la clé d'accès est inscrite pour le domaine actuel
    const passkeyEnrolledHere = claims.passkey_enrolled &&
                                 claims.passkey_domain === window.CUSTOM_DOMAIN;

    if (!passkeyEnrolledHere) {
      // Inviter l'utilisateur à inscrire une clé d'accès pour ce domaine
      promptPasskeyEnrollment();
    }
  }
}
```

<div id="domain-specific-enrollment-pages">
  ### Pages d’inscription propres à chaque domaine
</div>

Créez des pages d’inscription dédiées pour chaque domaine personnalisé :

```javascript
// Page d'inscription pour Brand 1
// URL: https://login.brand1.com/enroll-passkey

import { createAuth0Client } from '@auth0/auth0-spa-js';

async function enrollPasskeyForBrand1() {
  const auth0 = await createAuth0Client({
    domain: 'login.brand1.com',
    clientId: 'YOUR_CLIENT_ID'
  });

  try {
    // Déclencher l'inscription de la clé d'accès
    await auth0.loginWithPopup({
      authorizationParams: {
        acr_values: 'http://schemas.openid.net/pape/policies/2007/06/multi-factor',
        prompt: 'login'
      }
    });

    alert('Passkey enrolled successfully for Brand 1!');
  } catch (error) {
    console.error('Passkey enrollment failed:', error);
  }
}
```

<div id="contextual-enrollment-prompts">
  ### Invites contextuelles d’inscription de clé d’accès
</div>

Affichez des invites d’inscription de clé d’accès selon le comportement de l’utilisateur. Points à considérer :

* Suivez quand les utilisateurs suppriment les invites d’inscription (stockez l’information dans `localStorage`)
* Vérifiez `logins_count` dans les métadonnées de l’utilisateur pour afficher les invites après plusieurs visites
* Vérifiez qu’une clé d’accès n’est pas déjà inscrite pour le domaine actuel

```javascript
async function shouldShowEnrollmentPrompt(auth0, customDomain) {
  const storageKey = `passkey_prompt_dismissed_${customDomain}`;

  // Ne pas afficher si l'utilisateur l'a supprimée
  if (localStorage.getItem(storageKey)) return false;

  const claims = await auth0.getIdTokenClaims();
  const passkeyEnrolled = claims.passkey_enrolled &&
                          claims.passkey_domain === customDomain;

  if (passkeyEnrolled) return false;

  // Afficher l'invite après la 3e connexion
  const user = await auth0.getUser();
  return (user.logins_count || 0) >= 3;
}
```

<div id="help-documentation">
  ### Documentation d’aide
</div>

Fournissez une documentation d’aide claire :

**Exemple d’entrée de FAQ** :

**Q : Pourquoi dois-je configurer une clé d’accès à nouveau ?**

R : Les clés d’accès sont associées à des domaines précis pour des raisons de sécurité. Si vous ouvrez une session à partir d’un portail différent (p. ex., Brand A vs Brand B), vous devrez configurer une clé d’accès pour chacun d’eux. Cela contribue à protéger vos comptes en garantissant que les clés d’accès ne fonctionnent que là où elles le doivent.

**Q : Ai-je besoin d’un appareil différent pour chaque clé d’accès ?**

R : Non ! Vous pouvez utiliser le même appareil (téléphone, ordinateur ou clé matérielle) pour des clés d’accès sur différents domaines. Chaque clé d’accès est simplement un identifiant distinct stocké sur votre appareil.

<div id="related-origins-future-feature">
  ### Origines associées (fonctionnalité à venir)
</div>

Auth0 prévoit prendre en charge les origines associées de WebAuthn, ce qui permettra de partager des clés d’accès entre les domaines précisés. Cette fonctionnalité :

* Vous permettra de configurer des domaines comme « associés » pour l’utilisation des clés d’accès
* Permettra aux utilisateurs d’utiliser sur `login.brand2.com` une clé d’accès enregistrée sur `login.brand1.com` si ces domaines sont configurés comme associés
* Offrira plus de souplesse pour les mises en œuvre multimarques

**Statut** : Prévu pour une version ultérieure à la GA

<div id="migration-scenarios">
  ## Scénarios de migration
</div>

<div id="migrating-from-single-custom-domain-to-multiple">
  ### Passer d’un seul domaine personnalisé à plusieurs
</div>

**Avant** : Un seul domaine personnalisé avec des clés d’accès déjà configurées

**Après** : Plusieurs domaines personnalisés pour différentes marques

**Défi** : Les clés d’accès existantes ne fonctionnent que sur le domaine d’origine

**Approche de migration** :

1. **Conserver le domaine d’origine actif** : Garder le domaine personnalisé d’origine comme domaine commun
2. **Déploiement graduel** : Introduire graduellement de nouveaux domaines personnalisés
3. **Informer les utilisateurs** : Aviser les utilisateurs qu’ils devront configurer des clés d’accès sur les nouveaux domaines
4. **Prévoir un parcours de réinscription** : Faciliter la configuration de clés d’accès sur les nouveaux domaines
5. **Suivre l’adoption** : Suivre les taux d’inscription des clés d’accès par domaine

**Modèle de communication** :

> &quot;Nous lançons des pages de connexion propres à chaque marque ! Votre clé d’accès actuelle continuera de fonctionner sur [original domain]. Lorsque vous visiterez nos nouvelles pages de connexion, on vous demandera aussi de configurer une clé d’accès pour vous y connecter plus rapidement.&quot;

<div id="migrating-between-custom-domains">
  ### Migration entre des domaines personnalisés
</div>

**Scénario** : Passer de `old-domain.com` à `new-domain.com`

**Défi** : Les clés d’accès ne peuvent pas être transférées

**Étapes de migration** :

1. **Fonctionnement en parallèle** : Utiliser les deux domaines simultanément pendant la transition
2. **Détecter les clés d’accès déjà configurées** : Repérer les utilisateurs qui ont des clés d’accès sur l’ancien domaine
3. **Inviter à se réinscrire** : Lorsque les utilisateurs ouvrent une session au moyen du nouveau domaine, leur demander d’inscrire une clé d’accès
4. **Période de grâce** : Garder l’ancien domaine actif pendant une période de transition
5. **Retirer progressivement l’ancien domaine** : Une fois l’adoption faite, mettre l’ancien domaine hors service

```javascript
// Dans votre Auth0 Action
exports.onExecutePostLogin = async (event, api) => {
  const domain = event.custom_domain?.domain;
  const oldDomain = 'old-domain.com';
  const newDomain = 'new-domain.com';

  // Vérifier si l'utilisateur avait une clé d'accès sur l'ancien domaine
  const hadOldPasskey = event.user.app_metadata?.passkey_domains?.includes(oldDomain);

  // L'utilisateur est sur le nouveau domaine mais n'a pas encore de clé d'accès inscrite
  if (domain === newDomain && hadOldPasskey) {
    const newDomainPasskeys = event.user.app_metadata?.passkey_domains?.includes(newDomain);

    if (!newDomainPasskeys) {
      // Définir un indicateur pour demander la réinscription
      api.idToken.setCustomClaim('should_enroll_passkey', true);
      api.idToken.setCustomClaim('migrated_from', oldDomain);
    }
  }
};
```

<div id="best-practices">
  ## Meilleures pratiques
</div>

1. **Utilisez un domaine commun** : Utilisez un domaine personnalisé commun pour réduire le nombre de domaines nécessitant l’inscription d’une clé d’accès
2. **Communication claire** : Informez les utilisateurs des exigences d’inscription propres à chaque domaine
3. **Invitez au bon moment** : Affichez les invites d’inscription après que les utilisateurs ont démontré leur engagement (p. ex., 3 connexions ou plus)
4. **Suivez les inscriptions** : Surveillez quels utilisateurs ont inscrit des clés d’accès pour quels domaines
5. **Offrez de l’aide** : Fournissez une documentation claire et du soutien pour la gestion des clés d’accès
6. **Testez minutieusement** : Testez les flux de clés d’accès sur tous les domaines personnalisés avant la mise en production
7. **Planifiez les migrations** : Si vous changez de domaine personnalisé, prévoyez la réinscription des utilisateurs
8. **Surveillez l’adoption** : Suivez les taux d’inscription et d’utilisation des clés d’accès par domaine

<div id="troubleshooting">
  ## Dépannage
</div>

<div id="passkey-not-working-on-custom-domain">
  ### La clé d’accès ne fonctionne pas avec un domaine personnalisé
</div>

**Symptômes** : l’utilisateur a configuré une clé d’accès, mais ne peut pas l’utiliser

**Causes possibles** :

* L’utilisateur se trouve sur un domaine personnalisé différent de celui où il a configuré sa clé d’accès
* Problèmes de compatibilité du navigateur
* La clé d’accès a été supprimée de l’appareil

**Résolution** :

1. Confirmez que l’utilisateur est sur le bon domaine personnalisé
2. Vérifiez si le navigateur prend en charge WebAuthn
3. Au besoin, demandez à l’utilisateur de configurer de nouveau sa clé d’accès

<div id="user-confused-about-multiple-enrollments">
  ### Utilisateur confus face à plusieurs inscriptions
</div>

**Symptômes** : L’utilisateur signale que la « clé d’accès » ne fonctionne pas lorsqu’il passe d’un domaine à l’autre

**Cause** : L’utilisateur ne comprend pas que l’inscription est propre à chaque domaine

**Résolution** :

1. Fournir des messages clairs indiquant que les clés d’accès sont propres à chaque domaine
2. Montrer pour quels domaines l’utilisateur a inscrit des clés d’accès
3. Inviter l’utilisateur à s’inscrire lorsqu’il accède à un nouveau domaine
