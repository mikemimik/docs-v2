Lorsque vous avez configuré [plusieurs domaines personnalisés](/docs/fr-CA/customize/custom-domains/multiple-custom-domains) pour votre tenant Auth0, vous pouvez en désigner un comme **domaine personnalisé par défaut**. Le domaine par défaut simplifie la configuration et garantit une expérience utilisateur uniforme lorsqu’aucun domaine personnalisé n’est explicitement précisé.

<div id="what-is-a-default-custom-domain">
  ## Qu’est-ce qu’un domaine personnalisé par défaut?
</div>

Le domaine personnalisé par défaut est le domaine personnalisé qu’Auth0 utilise automatiquement pour :

* **Communications par courriel et téléphone** : l’envoi de courriels de réinitialisation du mot de passe, de liens de vérification du courriel et d’autres notifications générées par Auth0 lorsqu’aucun domaine personnalisé précis n’est indiqué. Cela comprend les cas où des notifications sont envoyées pendant l’authentification.
* **Appels à la Management API** : le traitement des requêtes d’API qui déclenchent des notifications sans l’en-tête `auth0-custom-domain`

Vous pouvez configurer votre domaine canonique de tenant (`YOUR_TENANT.auth0.com` ou `YOUR_TENANT.REGION.auth0.com`) en tant que domaine personnalisé par défaut. Un domaine personnalisé par défaut sera toujours configuré pour votre tenant.

<div id="benefits-of-setting-a-default-domain">
  ## Avantages de définir un domaine par défaut
</div>

Définir un domaine personnalisé par défaut offre plusieurs avantages :

* **Configuration simplifiée** : Réduit la nécessité de préciser des domaines personnalisés dans chaque appel à la Management API ou dans la configuration
* **Image de marque cohérente** : Garantit que les utilisateurs voient toujours votre image de marque, même lorsqu’aucun domaine précis n’est indiqué
* **En-tête `auth0-custom-domain` facultatif** : Rend l’en-tête de domaine personnalisé facultatif dans les requêtes de la Management API
* **Comportement de secours** : Fournit des valeurs par défaut appropriées pour les mises en œuvre multimarques ou multitenantes
* **Migration simplifiée** : Facilite la transition d’un seul domaine personnalisé à plusieurs domaines personnalisés

<div id="configure-a-default-domain">
  ## Configurer un domaine par défaut
</div>

Vous pouvez définir un domaine personnalisé par défaut dans l’Auth0 Dashboard ou à l’aide de la Management API.

<div id="using-the-auth0-dashboard">
  ### Utilisation d’Auth0 Dashboard
</div>

Pour définir un domaine personnalisé par défaut dans Auth0 Dashboard :

1. Accédez à **Auth0 Dashboard** &gt; **Branding &gt; Custom Domains**
2. Repérez dans la liste le domaine personnalisé vérifié que vous voulez définir comme domaine par défaut
3. Cliquez sur le bouton **Set as Default** pour ce domaine
4. Confirmez l’action dans la boîte de dialogue qui s’affiche

Le domaine affichera maintenant l’étiquette « Default » dans la liste des domaines personnalisés.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Seuls les domaines personnalisés vérifiés peuvent être définis comme domaine par défaut. Assurez-vous que votre domaine personnalisé est entièrement vérifié et actif avant de le désigner comme domaine par défaut.
</Callout>

<div id="using-the-management-api">
  ### Utilisation de la Management API
</div>

Pour définir un domaine personnalisé par défaut à l’aide de la Management API, utilisez le point de terminaison `PATCH /api/v2/custom-domains/default` :

<AuthCodeGroup>
  ```bash cURL
  curl --request PATCH \
    --url 'https://{yourDomain}/api/v2/custom-domains/default' \
    --header 'authorization: Bearer {yourMgmtApiAccessToken}' \
    --header 'content-type: application/json' \
    --data '{
      "domain": "my-custom-domain.com"
    }'
  ```

  ```javascript Node.js
  import { ManagementClient } from "auth0";

  const management = new ManagementClient({
    domain: '{yourDomain}',
    clientId: '{yourClientId}',
    clientSecret: '{yourClientSecret}',
  });

  await management.customDomains.setDefault({
    domain: 'my-custom-domain.com',
  });
  ```

  ```python Python
  from auth0.management import ManagementClient

  client = ManagementClient(
      domain='{yourDomain}',
      client_id='{yourClientId}',
      client_secret='{yourClientSecret}',
  )

  client.custom_domains.set_default({
      'domain': 'my-custom-domain.com',
  })
  ```

  ```go Go
  import (
      "context"
      "github.com/auth0/go-auth0/v2/management"
      management_client "github.com/auth0/go-auth0/v2/management/client"
      "github.com/auth0/go-auth0/v2/management/option"
  )

  mgmt, err := management_client.New(
      "{yourDomain}",
      option.WithClientCredentials("{yourClientId}", "{yourClientSecret}"),
  )
  if err != nil {
      // Gérer l’erreur
  }

  err = mgmt.CustomDomains.SetDefault(context.Background(), &management.SetDefaultCustomDomainRequestContent{
      Domain: "my-custom-domain.com",
  })
  ```

  ```java Java
  import com.auth0.client.mgmt.ManagementAPI;
  import com.auth0.json.mgmt.customdomains.SetDefaultCustomDomainRequest;

  ManagementAPI mgmt = ManagementAPI.newBuilder(
      "{yourDomain}",
      "{yourMgmtApiAccessToken}"
  ).build();

  SetDefaultCustomDomainRequest request = new SetDefaultCustomDomainRequest("my-custom-domain.com");

  mgmt.customDomains()
      .setDefault(request)
      .execute();
  ```
</AuthCodeGroup>

<div id="remove-default-domain-designation">
  ## Retirer le statut de domaine par défaut
</div>

Pour retirer le statut par défaut d’un domaine personnalisé :

1. Définissez un autre domaine personnalisé comme domaine par défaut (un seul domaine peut être défini par défaut à la fois), ou
2. Utilisez la Management API pour définir `is_default: false` pour le domaine par défaut actuel

Lorsqu’aucun domaine personnalisé n’est défini par défaut, Auth0 utilisera le domaine canonique de votre tenant.

<div id="how-the-default-domain-is-used">
  ## Utilisation du domaine par défaut
</div>

<div id="email-notifications">
  ### Notifications par courriel
</div>

Lorsque Auth0 envoie des notifications par courriel (réinitialisation de mot de passe, vérification du courriel, courriels de bienvenue), les liens et les éléments de personnalisation de ces notifications utilisent le domaine par défaut. Vous pouvez personnaliser les modèles de courriel afin d’utiliser les renseignements du domaine personnalisé dans l’adresse « De », l’objet et le corps du courriel.

<div id="management-api-endpoints-that-trigger-notifications">
  ### Points de terminaison de la Management API qui déclenchent des notifications
</div>

Le domaine personnalisé par défaut est utilisé précisément pour les points de terminaison de la Management API qui déclenchent des notifications par courriel ou par téléphone. L’en-tête `auth0-custom-domain` devient facultatif pour ces points de terminaison lorsqu’un domaine par défaut est configuré.

**Points de terminaison qui déclenchent des notifications par courriel :**

* `POST /api/v2/tickets/password-change` - Envoie le courriel de réinitialisation du mot de passe
* `POST /api/v2/tickets/email-verification` - Envoie la vérification du courriel
* `POST /api/v2/jobs/verification-email` - Envoie le courriel de vérification à un ou plusieurs utilisateurs
* `POST /api/v2/users` - Peut déclencher un courriel de bienvenue si cette option est configurée
* `PATCH /api/v2/users/{id}` - Peut déclencher un courriel de vérification lorsque l’adresse courriel est mise à jour

**Points de terminaison qui déclenchent des notifications téléphoniques :**

* `POST /api/v2/users/{id}/phone/verification` - Envoie un code de vérification par SMS

**Exemple :**

<AuthCodeGroup>
  ```bash Sans en-tête de domaine personnalisé (utilise la valeur par défaut)
  # Le ticket de réinitialisation du mot de passe utilise le domaine par défaut dans les liens du courriel
  curl --request POST \
    --url 'https://{yourDomain}/api/v2/tickets/password-change' \
    --header 'authorization: Bearer {yourMgmtApiAccessToken}' \
    --header 'content-type: application/json' \
    --data '{
      "user_id": "auth0|123456",
      "result_url": "https://example.com/password-changed"
    }'
  ```

  ```bash Avec l’en-tête de domaine personnalisé (remplace la valeur par défaut)
  # Indique explicitement un domaine personnalisé différent pour cette notification
  curl --request POST \
    --url 'https://{yourDomain}/api/v2/tickets/password-change' \
    --header 'authorization: Bearer {yourMgmtApiAccessToken}' \
    --header 'auth0-custom-domain: brand-specific.com' \
    --header 'content-type: application/json' \
    --data '{
      "user_id": "auth0|123456",
      "result_url": "https://example.com/password-changed"
    }'
  ```
</AuthCodeGroup>

Lorsque vous ne fournissez pas l’en-tête `auth0-custom-domain` pour ces points de terminaison de notification, Auth0 utilise automatiquement le domaine personnalisé par défaut pour les liens et les personnalisations dans le courriel ou le SMS. Vous pouvez le remplacer pour chaque requête en incluant l’en-tête `auth0-custom-domain`.

<div id="using-the-default-domain-with-actions">
  ## Utiliser le domaine par défaut avec Actions
</div>

Utilisez [Actions](/docs/fr-CA/customize/actions) pour exécuter une logique en fonction du domaine personnalisé :

```javascript
exports.onExecutePostLogin = async (event, api) => {
  const domain = event.custom_domain?.domain;

  // Stocker le domaine de connexion dans les métadonnées de l'utilisateur
  if (domain) {
    api.user.setAppMetadata('login_domain', domain);
  }
};
```

Pour en savoir plus sur l’utilisation des renseignements sur les domaines personnalisés dans Actions, consultez [Intégration d’Actions à plusieurs domaines personnalisés](/docs/fr-CA/customize/custom-domains/multiple-custom-domains/actions-integration).

<div id="best-practices">
  ## Bonnes pratiques
</div>

Lorsque vous configurez un domaine personnalisé par défaut, tenez compte de ces bonnes pratiques :

* **Choisissez un domaine stable** : Sélectionnez un domaine qui ne changera pas souvent comme domaine personnalisé par défaut
* **Vérifiez-le avant de le définir** : Assurez-vous que le domaine est entièrement vérifié et opérationnel avant de le définir par défaut
* **Documentez la décision** : Notez pourquoi un domaine en particulier a été choisi comme domaine par défaut pour référence future
* **Testez les flux de courriel** : Après avoir défini un domaine par défaut, testez les flux de réinitialisation du mot de passe et de vérification du courriel
* **Surveillez les émetteurs de jetons** : Vérifiez que les jetons contiennent la revendication `iss` attendue après avoir défini un domaine par défaut
* **Prévoyez les scénarios multimarques** : Si vous prenez en charge plusieurs marques, choisissez un domaine générique ou administratif comme domaine par défaut
* **Mettez à jour les intégrations** : Informez votre équipe du domaine par défaut afin d’assurer une configuration adéquate des intégrations
