<Card title="Avant de commencer">
  1. Assurez-vous qu’un [**Custom Domain**](/docs/fr-CA/customize/custom-domains) est configuré pour votre tenant.
  2. Confirmez que vous utilisez [**Universal Login**](/docs/fr-CA/authenticate/login/auth0-universal-login/universal-login-vs-classic-login/universal-experience) pour tous les prompts d&#39;inscription et de connexion, et que la bascule **Customize Login Page** est désactivée pour les prompts de connexion.
  3. Vérifiez qu’un [modèle de page personnalisé](/docs/fr-CA/customize/login-pages/universal-login/customize-templates#custom-sign-up-prompts) est configuré.
</Card>

**Personnaliser les prompts d&#39;inscription et de connexion** est une fonctionnalité qui permet aux clients qui ont activé <Tooltip tip="Custom Domain : domaine tiers avec un nom spécialisé ou personnalisé." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Custom+Domain">Custom Domain</Tooltip> ainsi qu’un modèle de page personnalisé d’ajouter des champs personnalisés et du contenu aux prompts d&#39;inscription et de connexion de leur application.

<div id="use-cases">
  ## Cas d’utilisation
</div>

Personnaliser les prompts d&#39;inscription et de connexion couvre deux cas d’utilisation : **contenu personnalisé** et **collecte de données.**

Le **contenu personnalisé** correspond à du contenu statique, comme du texte, des liens ou des images, placé directement dans les prompts d&#39;inscription et de connexion.

La **collecte de données** s’appuie sur des éléments de formulaire ajoutés dynamiquement aux prompts d&#39;inscription et de connexion, ce qui est utile pour recueillir et valider le consentement de l’utilisateur ou des données fournies par l’utilisateur, comme le nom de famille.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  N’utilisez les personnalisations des prompts d&#39;inscription et de connexion pour transmettre ou recueillir des données sensibles ou réglementées que dans la mesure permise par votre entente avec Okta.
</Callout>

<div id="terminology">
  ## Terminologie
</div>

Un **prompt** est une étape précise dans un flux d’authentification donné. Chaque prompt comprend au moins un **écran** et, selon la configuration du tenant, chaque écran pris en charge comporte soit quatre, soit six **points d’entrée**, c’est-à-dire des emplacements dans l’écran où du code personnalisé (**partials**) peut être inséré.

Les prompts suivants peuvent être personnalisés :

**Écrans de connexion**

* `login`
* `login-id`
* `login-password`
* `login-passwordless-sms-otp`
* `login-passwordless-email-code`
* `passkey-enrollment`
* `passkey-enrollment-local`

**Écrans d’inscription**

* `signup`
* `signup-id`
* `signup-password`
* `passkey-enrollment`
* `passkey-enrollment-local`

Les partials prennent en charge le HTML, le CSS, le JavaScript et la [syntaxe Liquid](https://github.com/Shopify/liquid/wiki/Liquid-for-Designers) pour gérer la logique conditionnelle et les variables dynamiques. De plus, toute variable Liquid offerte dans le [Page Template](/docs/fr-CA/customize/login-pages/universal-login/customize-templates) est également prise en charge.

Ces points d’entrée sont disponibles lorsqu’une connexion de base de données ou sans mot de passe est activée :

* `form-content-start`
* `form-content-end`
* `form-footer-start`
* `form-footer-end`

Les points d’entrée suivants sont disponibles lorsqu’au moins une connexion sociale ou Enterprise est activée :

* `secondary-actions-start`
* `secondary-actions-end`

<div id="use-the-auth0-dashboard-to-manage-partials">
  ## Utilisez l’Auth0 Dashboard pour gérer les Partials
</div>

Utilisez le Dashboard pour insérer des champs personnalisés et du contenu dans les écrans Login et Signup à l’aide de partials.

1. Accédez à [Auth0 Dashboard &gt; Branding &gt; Universal Login](https://manage.auth0.com/#/universal-login/customize-partials), puis sélectionnez **Enhance screens with partials**.
2. Sélectionnez le **écran** à personnaliser dans l’éditeur de partials.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  **Les écrans** peuvent avoir les **modes de rendu** suivants :

  * **STANDARD** : le `screen` est affiché à l’aide de l’interface utilisateur [Universal Login](/docs/fr-CA/customize/login-pages/universal-login) par défaut; vous pouvez utiliser des partials pour insérer des extraits de code et des [variables de gabarit](/docs/fr-CA/customize/login-pages/universal-login/customize-templates#page-template-variables).
  * **ADVANCED** : le `screen` est affiché à l’aide d’[ACUL](/docs/fr-CA/customize/login-pages/advanced-customizations) et les partials ne s’appliquent pas.
  * **ADVANCED (FILTERED)** : le `screen` est affiché avec ACUL appliqué à certaines applications et organizations; les partials s’appliquent uniquement aux `screens` exclus des filtres ACUL.
</Callout>

<div id="manage-partials-programmatically">
  ## Gérer les Partials par programmation
</div>

Vous pouvez gérer les Partials à l’aide de l’[Auth0 Management API](https://auth0.com/docs/api/management/v2/prompts/put-partials) (`GET /prompts/{prompts_name}/partials`). Chaque prompt doit préciser le `screen` lors de l’ajout, de la mise à jour ou de la suppression d’un partial. Les Partials peuvent comporter au maximum 10 000 caractères.

Vous pouvez aussi gérer les Partials à l’aide de l’interface Customize de <Tooltip tip="Universal Login : votre application redirige l’utilisateur vers Universal Login, hébergé sur l’Authorization Server d’Auth0, afin de vérifier son identité." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Universal+Login">Universal Login</Tooltip> dans Auth0 CLI, en exécutant `auth0 ul customize` dans votre terminal.

<Frame>![Une image montrant l’interface de ligne de commande pour les Partials. ](/docs/images/cdy7uua7fh8z/KxWfVAMkumyI2jrUqkEp8/b9315709899864a2f8758ad7f5a4f5b2/Partials_CLI_Image.jpg)</Frame>

<div id="style-and-validate-form-inputs">
  ## Styliser et valider les champs de saisie de Form
</div>

Personnaliser les prompts d&#39;inscription et de connexion offre des styles prédéfinis ainsi que la prise en charge de la validation côté client pour certains éléments de formulaire HTML. Les éléments suivants sont pris en charge :

* `<input type="text">`
* `<input type="number">`
* `<input type="checkbox">`
* `<input type="password">`
* `<input type="email">`
* `<input type="tel">`
* `<input type="url">`
* `<select>`
* `<textarea>`

Pour utiliser les styles de saisie prédéfinis, placez l&#39;élément de formulaire de votre choix dans un `<div>` doté de la classe `ulp-field`. De même, ajoutez la classe `ulp-error` au même `<div>` pour utiliser les styles d&#39;erreur prédéfinis. Si l&#39;élément `ulp-error-info` est présent, un message d&#39;erreur mis en forme s&#39;affichera également.

Vous pouvez trouver des extraits de code prédéfinis dans l&#39;éditeur de [partials](#use-the-auth0-dashboard-to-manage-partials).

<div id="client-side-validation">
  ### Validation côté client
</div>

Le framework de validation côté client de cette fonctionnalité permet aux clients de valider les données saisies par l’utilisateur à l’aide d’attributs HTML afin d’exécuter une ou plusieurs fonctions de validation personnalisées. Les fonctions de validation peuvent être incluses directement dans le Partial ou dans le `<head>` du gabarit de page.

Pour ajouter une validation côté client à un élément de formulaire :

* Référencez la fonction de validation à l’aide de l’attribut `data-ulp-validation-function` sur l’élément `<div class="ulp-error-info">`.
* Indiquez les événements DOM qui doivent déclencher la fonction de validation à l’aide de l’attribut `data-ulp-validation-event-listeners` sur l’élément `<div class="ulp-error-info">`, en notant que les validations s’exécutent automatiquement lors de la soumission.
* Pour assurer la conformité aux WCAG, les champs de saisie doivent être liés par programmation à leurs messages d’erreur — par exemple, à l’aide de `aria-describedby="error-id"` et de `aria-invalid="true"` — afin que les lecteurs d’écran annoncent les erreurs de validation.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Faites preuve de prudence lorsque vous utilisez du Javascript tiers sur votre page d’inscription. Des renseignements sensibles liés à la sécurité transitent souvent par la page d’inscription, ce qui la rend vulnérable au script intersites.

  Dans la mesure du possible, Auth0 recommande de valider les données fournies par l’utilisateur avant de les soumettre.
</Callout>

<div id="localize-content">
  ## Localiser le contenu
</div>

Il est possible de localiser une partie du contenu en définissant de nouvelles variables de texte personnalisées à l’aide de la [Custom Text API](/docs/fr-CA/api/management/v2/prompts/put-custom-text-by-language). Il est possible de définir jusqu’à trente variables de texte personnalisées par combinaison écran-langue.

<div id="create-or-update-a-custom-text-variable">
  ### Créer ou mettre à jour une variable de texte personnalisée
</div>

Gérez les variables de texte personnalisées avec la [Custom Text API](/docs/fr-CA/api/management/v2/prompts/get-custom-text-by-language). Les appels doivent préciser le `screen` lors de l’ajout, de la mise à jour ou de la suppression d’une variable de texte personnalisée. Les variables de texte personnalisées suivent la convention de nommage `var-<name>`. Les liens Markdown sont pris en charge et convertis en éléments HTML `<a>` avant d’être affichés aux utilisateurs.

Vous trouverez ci-dessous un exemple d’appel pour ajouter une variable pour le texte du libellé d’une case à cocher relative aux Conditions d’utilisation, en anglais et en espagnol. Consultez la <Tooltip tip="Management API : Un produit permettant aux clients d’effectuer des tâches administratives." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Management+API">Management API</Tooltip> pour en savoir plus.

```json lines
# PUT /api/v2/prompts/signup-id/custom-text/fr
{
  "signup": {
    "var-tos": "I agree with the [Terms of Service](https://en.example.com/tos)"
  }
}

# PUT /api/v2/prompts/signup-id/custom-text/es
{
  "signup": {
    "var-tos": "Estoy de acuerdo con los [Términos de Servicio](https://es.example.com/tos)"
  }
}
```

<div id="use-a-custom-text-variable-in-a-partial">
  ### Utiliser une variable de texte personnalisée dans un Partial
</div>

Les variables de texte personnalisées sont référencées dans les partials à l’aide de l’objet `prompts.screen.text`; pour l’exemple `var-tos` de la section précédente, la référence est `prompt.screen.text.varTos`. Voir l’exemple ci-dessous pour savoir comment utiliser, dans un partial du Signup ID Prompt, une variable créée précédemment. Notez que la variable `var-tos` de la Management API est référencée sous la forme `varTos` dans le partial.

```html
# PUT api/v2/prompts/signup/partials
<div class='ulp-field'>
  <input type='checkbox' name='ulp-terms-of-service' id='terms-of-service'>
  <label for='terms-of-service'>{{ prompt.screen.text.varTos }}</label>
</div>
```

<div id="validate-and-save-captured-data">
  ## Valider et enregistrer les données capturées
</div>

Les données capturées par les éléments de formulaire personnalisés sont disponibles dans Actions, et Auth0 recommande de valider les données recueillies avant de les enregistrer ou de les soumettre.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Lorsque vous utilisez des éléments de formulaire personnalisés, vous devez inclure le préfixe `ulp-` dans tous les noms de champ afin de vous assurer que les données sont disponibles dans [Actions](/docs/fr-CA/customize/actions/actions-overview).
</Callout>

Chaque Action reçoit les données capturées sous forme d’objet dans `event.request.body`. Les clients peuvent renvoyer une erreur de validation en utilisant la fonction `api.validation.error`.

Lorsque vous utilisez une [connexion de base de données](/docs/fr-CA/authenticate/database-connections#use-auth0-user-store) :

* Les données des prompts d’inscription sont accessibles dans le trigger [`pre-user-registration`](/docs/fr-CA/customize/actions/explore-triggers/signup-and-login-triggers/pre-user-registration-trigger). Une erreur de validation provenant du trigger empêche l’utilisateur de s’inscrire.
* Les données des prompts de connexion sont accessibles dans le trigger [`post-login`](/docs/fr-CA/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger). Les erreurs de validation sont transmises à la page d’erreur de l’application du client.

Lorsque vous utilisez une [connexion de base de données personnalisée](/docs/fr-CA/authenticate/database-connections/custom-db) :

* Les données des prompts d’inscription sont accessibles dans le trigger [`pre-user-registration`](/docs/fr-CA/customize/actions/explore-triggers/signup-and-login-triggers/pre-user-registration-trigger) avec les scripts d’action de base de données personnalisée suivants : [Create User](/docs/fr-CA/authenticate/database-connections/custom-db/templates/create) et [Login](/docs/fr-CA/authenticate/database-connections/custom-db/templates/login).

* Les données des prompts de connexion sont accessibles dans le trigger [`post-login`](/docs/fr-CA/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger) avec les scripts d’action de base de données personnalisée suivants : [Login](/docs/fr-CA/authenticate/database-connections/custom-db/templates/login) et [Change Password](/docs/fr-CA/authenticate/database-connections/custom-db/templates/change-password).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Pour utiliser les scripts d’action de base de données personnalisée, activez **Context objects in database scripts** dans votre connexion de base de données personnalisée. Pour en savoir plus, consultez [Enable context object](https://auth0.com/docs/authenticate/database-connections/custom-db/create-db-connection#enable-context-object).
</Callout>

Lorsque vous utilisez une [connexion Passwordless](/docs/fr-CA/authenticate/passwordless/passwordless-with-universal-login) :

* Les données des prompts d’inscription et de connexion sont accessibles dans le trigger [`post-login`](/docs/fr-CA/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger). Les erreurs de validation sont transmises à la page d’erreur de l’application du client.

Lorsque vous utilisez les prompts [`passkey`](/docs/fr-CA/authenticate/database-connections/passkeys) :

* Les données du prompt `passkey-enrollment` sont accessibles dans le trigger [`pre-user-registration`](/docs/fr-CA/customize/actions/explore-triggers/signup-and-login-triggers/pre-user-registration-trigger). Une erreur de validation provenant du trigger empêche l’utilisateur de s’inscrire.
* Les prompts `passkey-enrollment` et `passkey-enrollment-local` ne capturent jamais de données dans le trigger Post Login.
* Les données du prompt `passkey-enrollment-local` ne sont pas accessibles, puisqu’elle est toujours affichée après l’exécution du trigger [`post-login`](/docs/fr-CA/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger).

Lorsque vous utilisez des connexions sociales ou d’entreprise :

* Les données ne sont *pas* capturées lorsqu’un utilisateur se connecte avec une connexion [sociale](/docs/fr-CA/authenticate/identity-providers/social-identity-providers) ou une connexion [d’entreprise](/docs/fr-CA/authenticate/enterprise-connections).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Nettoyez toutes les données que vous recueillez dans le formulaire avant de les enregistrer ou de les afficher.

  * Assurez-vous que toutes les données enregistrées ont été traitées avec la fonction d’aide [Liquid `{{escape}}`](https://shopify.github.io/liquid/filters/escape/)
  * Si vous affichez des données dans un gabarit de courriel, supprimez la syntaxe Liquid
  * Si vous affichez des données sur une page Web, échappez les entités HTML
  * Si vous enregistrez des données dans une base de données, utilisez des requêtes paramétrées
  * Si vous transmettez des données dans la chaîne de requête, encodez-les avec, par exemple : `{{encodeURI}}` ou `{{encodeURIParam}}`

  Pour plus d’informations sur l’atténuation des risques et les pratiques exemplaires pour stocker les données de façon sécuritaire, consultez [cet aide-mémoire](https://cheatsheetseries.owasp.org/cheatsheets/Input_Validation_Cheat_Sheet.html).
</Callout>

<div id="save-to-user-metadata">
  ### Enregistrer dans les métadonnées utilisateur
</div>

À partir de l’Action, les données recueillies peuvent être envoyées à une API externe à des fins de validation et de stockage, ou enregistrées dans le `user_metadata` de l’utilisateur au moyen de `api.user.setUserMetadata`.

```js lines
// Avec ce code dans le formulaire d'inscription
// <div class="ulp-field">
//   <label for="full-name">Full Name</label>
//   <input type="text" name="ulp-full-name" id="full-name">
// </div>

exports.onExecutePreUserRegistration = async (event, api) => {
  const fullName = event.request.body['ulp-full-name'];
  if(!fullName) {
    api.validation.error("invalid_payload", "Missing Name");
    return;
  }

  api.user.setUserMetadata("fullName", fullName);
};
```
