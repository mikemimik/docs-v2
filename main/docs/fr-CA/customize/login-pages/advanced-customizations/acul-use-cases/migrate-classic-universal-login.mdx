<Warning>
  À l’heure actuelle, les efforts de développement d’Auth0 se concentrent sur Universal Login, et Classic Login n’est plus mis à jour. À moins que votre cas d’utilisation n’exige précisément l’expérience Classic, nous vous recommandons d’adopter Universal Login.
  Pour comparer les fonctionnalités des différentes expériences de connexion, consultez [Universal Login vs. Classic Login](/docs/fr-CA/authenticate/login/auth0-universal-login/universal-login-vs-classic-login).
</Warning>

Nous vous recommandons de migrer l’implémentation de votre écran [Classic Login](/docs/fr-CA/authenticate/login/auth0-universal-login/universal-login-vs-classic-login/classic-experience#classic-login-experience) vers [Universal Login](/docs/fr-CA/authenticate/login/auth0-universal-login/universal-login-vs-classic-login/universal-experience) à l’aide de [Advanced Customizations for Universal Login (ACUL)](/docs/fr-CA/customize/login-pages/advanced-customizations#advanced-customizations-for-universal-login-acul).

<Card title="Avant de commencer">
  Vous aurez besoin de ce qui suit :

  * Un tenant de développement Auth0 configuré avec [Universal Login](/docs/fr-CA/authenticate/login/auth0-universal-login) et un [domaine personnalisé](/docs/fr-CA/customize/custom-domains).
  * [Auth0 CLI](https://auth0.github.io/auth0-cli/) installé et configuré avec votre tenant de développement.
  * Un dépôt [GitHub](https://github.com/) pour y stocker vos écrans de connexion ACUL personnalisés.
  * Un inventaire des ressources de votre expérience Classic Login :
    * Composants UI.
    * Jetons CSS/de thème et ressources de marque.
    * Règles de validation (format du courriel, champs obligatoires).
    * Comportement de gestion des erreurs.
</Card>

<div id="initialize-an-acul-project">
  ## Initialiser un projet ACUL
</div>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Si vous n’avez pas configuré votre tenant de développement à l’aide d’Auth0 CLI, utilisez la commande [auth0 login](https://auth0.github.io/auth0-cli/auth0_login.html).
</Callout>

Placez-vous à l’endroit où vous souhaitez créer votre projet ACUL, puis exécutez la commande suivante :

```bash
auth0 acul init acul-login --screens login-id,login-password
```

Auth0 CLI initialise votre projet avec les écrans d’authentification prédéfinis `login-id` et `login-password`.

Pour en savoir plus, consultez [auth0 aclu sur GitHub](https://auth0.github.io/auth0-cli/auth0_acul_init.html).

<div id="customize-acul-screens">
  ## Personnaliser les écrans ACUL
</div>

Personnalisez les écrans `login-id` et `login-password` pour qu’ils correspondent aux normes de votre organisation.

Vous devriez tenir compte des éléments suivants :

* Réutiliser les composants React ainsi que le CSS/thème existants.
  * Appliquer l’image de marque (ressources, typographie, mise en page).
  * Ajouter une validation de base côté client (courriel requis/format ; mot de passe requis).
  * Uniformiser la présentation des erreurs (de façon cohérente et accessible).

Vous pouvez développer vos écrans ACUL localement et les personnaliser à l’aide de la commande :

```bash
auth0 acul dev
```

Pour en savoir plus, consultez le [workflow de développement d’ACUL](/docs/fr-CA/customize/login-pages/advanced-customizations/development-workflow).

<div id="validate-the-acul-migration">
  ## Valider la migration ACUL
</div>

Pour valider l’expérience de connexion de l’utilisateur, de bout en bout, vous pouvez utiliser la commande [auth0 acul](https://auth0.github.io/auth0-cli/auth0_acul_dev.html) :

```bash
auth0 acul dev --connected
```

Vous devez tester et vous assurer des points suivants :

* Le flux d’authentification utilise votre UI personnalisée pour les écrans `login-id` et `login-password`, et ceux-ci s’affichent correctement.
  * Les messages d’erreur, comme celui indiquant un mot de passe incorrect, ainsi que les autres messages d’erreur côté serveur, s’affichent clairement.
  * Les utilisateurs sont redirigés correctement vers votre application.
  * La validation de l’adresse courriel, si elle est activée, fonctionne comme prévu.

Pour en savoir plus, consultez [les tests d’intégration avec un tenant réel d’ACUL](/docs/fr-CA/customize/login-pages/advanced-customizations/development-workflow#live-tenant-integration-testing).

<div id="deploy-acul-screens-to-production-tenant">
  ## Déployer des écrans ACUL dans le tenant de production
</div>

Le projet d’écrans ACUL comprend une configuration de déploiement que vous pouvez utiliser comme point de départ pour déployer vos écrans dans votre tenant de production.

Les étapes de déploiement d’ACUL comprennent :

* La création des bundles de ressources de vos écrans.
  * Leur publication sur votre CDN.
  * L’activation d’ACUL dans votre tenant de production Auth0.

Pour en savoir plus, consultez le [workflow de déploiement d’ACUL](/docs/fr-CA/customize/login-pages/advanced-customizations/deployment-workflow).
