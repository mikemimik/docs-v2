<Warning>
  À l’heure actuelle, les efforts de développement d’Auth0 sont axés sur Universal Login, et Classic Login ne reçoit plus de mises à jour. Pour personnaliser vos pages de connexion, Auth0 recommande [Advanced Customizations for Universal Login](/docs/fr-CA/customize/login-pages/advanced-customizations).
</Warning>

Vous pouvez personnaliser les messages d’erreur affichés dans certaines situations en fournissant l’option `languageDictionary`. La liste complète des champs `languageDictionary` que vous pouvez personnaliser se trouve dans le dépôt GitHub, dans le [fichier de dictionnaire anglais pour Lock](https://github.com/auth0/lock/blob/master/src/i18n/en.js). Voici un exemple de messages d’erreur personnalisés :

{/* codeblockOld.header.login.logInButton codeblockOld.header.login.configureSnippet */}

export const codeExample = `// Exemples de messages d’erreur personnalisés dans l’option languageDictionary
var options = {
  languageDictionary: {
    error: {
      login: {
        "lock.invalid_email_password": "Message personnalisé concernant des identifiants invalides",
        "lock.network": "Message personnalisé indiquant une erreur réseau et suggérant à l’utilisateur de vérifier sa connexion",
        "lock.unauthorized": "Message personnalisé concernant un échec d’autorisation",
        "too_many_attempts": "Message personnalisé indiquant que l’utilisateur a échoué trop souvent à se connecter."
      },
      signUp: {
        "invalid_password": "Message personnalisé indiquant qu’un mot de passe était invalide",
        "user_exists": "Message personnalisé indiquant qu’un utilisateur existe déjà"
      }
    }
  }
};

// Initialisation de notre Auth0Lock
var lock = new Auth0Lock(
  '{yourClientId}',
  '{yourDomain}',
  options
);`;

<div id="custom-errors-in-rules">
  ## Erreurs personnalisées dans Rules
</div>

Si vous renvoyez des codes d’erreur personnalisés à partir d’une [Rule](/docs/fr-CA/customize/rules) ou d’un script de base de données personnalisé, vous pouvez gérer ces erreurs personnalisées :

* Dans l’URL de redirection de votre application, en lisant les paramètres de chaîne de requête `error` et `error_mesage`.
* En redirigeant l’utilisateur vers vos pages hébergées avec un message d’erreur personnalisé, puis en l’affichant au moyen d’un [message flash](/docs/fr-CA/libraries/lock/lock-api-reference).
