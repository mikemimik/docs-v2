Vous pouvez importer en bloc des données utilisateur dans Auth0 à l’aide du point de terminaison [Create Import Users Job](https://auth0.com/docs/api/management/v2#!/Jobs/post_users_imports). Les importations en bloc sont utiles pour migrer des utilisateurs d’une base de données ou d’un service existant vers Auth0.

<Warning>
  Si vous essayez d’utiliser plus d’une méthode de migration (par exemple, la migration automatique, puis l’importation en bloc d’utilisateurs), vous pourriez obtenir une erreur `DUPLICATED_USER`. Cette erreur indique que l’utilisateur existe dans le stockage interne des utilisateurs d’Auth0, mais pas dans votre tenant. Pour corriger cette erreur, supprimez l’utilisateur à l’aide du point de terminaison [Delete a Connection User](https://auth0.com/docs/api/management/v2#!/Connections/delete_users_by_email) de l’Auth0 Management API, puis relancez l’importation.
</Warning>

<div id="prerequisites">
  ## Prérequis
</div>

Avant de lancer la tâche d’importation des utilisateurs :

* Configurez une connexion de base de données pour y importer les utilisateurs et activez-la pour au moins une application.
* Si vous importez des mots de passe, assurez-vous qu’ils sont hachés à l’aide de l’un des algorithmes pris en charge. Les utilisateurs dont les mots de passe sont hachés avec des algorithmes non pris en charge devront réinitialiser leur mot de passe lorsqu’ils se connecteront pour la première fois après l’importation en bloc.
* Si vous importez des inscriptions <Tooltip tip="Authentification multifacteur (AMF) : processus d’authentification de l’utilisateur qui utilise un facteur en plus du nom d’utilisateur et du mot de passe, comme un code envoyé par SMS." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=MFA">AMF</Tooltip>, assurez-vous qu’elles sont d’un type pris en charge : `email`, `phone` ou `totp`.
* [Obtenez un jeton de la Management API](/docs/fr-CA/secure/tokens/access-tokens/management-api-access-tokens) pour les requêtes au point de terminaison des tâches.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Si vous utilisez un fichier d’exportation provenant d’un tenant Auth0, vous devez convertir le fichier exporté de `ndjson` en [JSON](/docs/fr-CA/manage-users/user-migration/bulk-user-exports#json-compatible-format). Pour conserver les mêmes ID d’utilisateur, vous devez supprimer le `préfixe auth0|` de tous les ID d’utilisateur importés.

  Le processus d’importation ajoute automatiquement le `préfixe auth0|` aux ID d’utilisateur importés. Si vous ne supprimez pas le préfixe `auth0|` avant l’importation, les ID d’utilisateur seront renvoyés sous la forme `auth0|auth0|...`
</Callout>

<div id="create-users-json-file">
  ## Créer le fichier JSON des utilisateurs
</div>

Créez un fichier JSON contenant les données utilisateur que vous souhaitez importer dans Auth0. La façon d’exporter ces données vers un fichier JSON varie selon votre base de données utilisateur actuelle. Le point de terminaison <Tooltip tip="Management API : Un produit qui permet aux clients d’effectuer des tâches administratives." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Management+API">Management API</Tooltip> s’attend à recevoir des sections du fichier JSON. Ainsi, au lieu d’utiliser `fs.readFileSync`, il faut utiliser `fs.createReadStream`. Le point de terminaison attend un flux de lecture transmis par pipeline plutôt que le fichier JSON au complet.

Pour en savoir plus sur le schéma du fichier JSON et voir des exemples, consultez [Bulk Import Database Schema and Examples](/docs/fr-CA/manage-users/user-migration/bulk-user-import-database-schema-and-examples).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  La taille maximale d’un fichier pour une importation en bloc est de 500KB. Vous devrez lancer plusieurs importations si vos données dépassent cette taille.
</Callout>

<div id="request-bulk-user-import">
  ## Lancer une importation en bloc d’utilisateurs
</div>

Pour lancer une tâche d’importation en bloc d’utilisateurs, envoyez une requête `POST` au point de terminaison [Create Import Users Job](https://auth0.com/docs/api/management/v2#!/Jobs/post_users_imports). Assurez-vous de remplacer les valeurs de l’espace réservé `MGMT_API_ACCESS_TOKEN`, `USERS_IMPORT_FILE.json`, `CONNECTION_ID` et `EXTERNAL_ID` par votre <Tooltip tip="Jeton d’accès : information d’autorisation, sous la forme d’une chaîne opaque ou d’un JWT, utilisée pour accéder à une API." cta="Consulter le glossaire" href="/docs/fr-CA/glossary?term=Access+Token">jeton d’accès</Tooltip> de la Management API, votre fichier JSON d’utilisateurs, l’[ID de connexion](/docs/fr-CA/authenticate/identity-providers/locate-the-connection-id) de la base de données et votre ID externe, respectivement.

| Paramètre               | Description                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                           |
| ----------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `users`                 | Fichier au format JSON contenant les utilisateurs à importer.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                         |
| `connection_id`         | ID de la connexion dans laquelle les utilisateurs seront insérés. Vous pouvez récupérer cet ID à l’aide du point de terminaison [GET /api/v2/connections](https://auth0.com/docs/api/management/v2#!/Connections/get_connections).                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                    |
| `upsert`                | Valeur booléenne ; `false` par défaut. Lorsqu’elle est définie sur `false`, les utilisateurs existants correspondant à l’adresse courriel, à l’ID utilisateur, à `phone` ou au nom d’utilisateur échoueront. Lorsqu’elle est définie sur `true`, les utilisateurs existants correspondant à l’adresse courriel seront mis à jour, mais uniquement avec les attributs pouvant être mis à jour par upsert. Pour obtenir la liste des champs du profil utilisateur pouvant être mis à jour par upsert lors de l’importation, consultez [User Profile Structure: User profile attributes](/docs/fr-CA/manage-users/user-accounts/user-profiles/user-profile-structure#user-profile-attributes). Remarque : si vous fournissez une entrée utilisateur en double dans le fichier d’importation, une erreur se produira. Dans ce cas, Auth0 n’effectuera pas d’insertion suivie d’une mise à jour. |
| `external_id`           | Chaîne facultative définie par l’utilisateur qui peut servir à corréler plusieurs tâches. Elle est renvoyée dans la réponse de statut de la tâche.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                    |
| `send_completion_email` | Valeur booléenne ; `true` par défaut. Lorsqu’elle est définie sur `true`, un courriel de fin d’exécution est envoyé à tous les propriétaires du locataire une fois la tâche d’importation terminée. Si vous ne voulez *pas* que des courriels soient envoyés, vous devez définir explicitement ce paramètre sur `false`.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                              |

Si la requête réussit, vous recevrez une réponse semblable à celle-ci :

```json lines
{
  "status": "pending",
  "type": "users_import",
  "created_at": "",
  "id": "job_abc123",
  "connection_id": "CONNECTION_ID",
  "upsert": false,
  "external_id": "EXTERNAL_ID",
  "send_completion_email": true
}
```

L’entité renvoyée représente la tâche d’importation.

Lorsque la tâche d’importation des utilisateurs se termine et que `send_completion_email` a été défini à `true`, le ou les administrateurs de tenant recevront un courriel les informant que la tâche a échoué ou réussi. Un courriel concernant une tâche ayant échoué peut informer le ou les administrateurs qu’elle n’a pas pu analyser le fichier JSON des utilisateurs lors de l’importation des utilisateurs.

<div id="concurrent-import-jobs">
  ### Tâches d’importation simultanées
</div>

L’point de terminaison [Create Import Users Job](https://auth0.com/docs/api/management/v2#!/Jobs/post_users_imports) est limité à deux tâches d’importation simultanées. Si vous demandez des tâches supplémentaires alors que deux sont déjà en attente, une réponse `429 Too Many Requests` est renvoyée :

```json lines
{
  "statusCode": 429,
  "error": "Too Many Requests",
  "message": "There are 2 active import users jobs, please wait until some of them are finished and try again
}
```

  "status": "pending",

Si une tâche est Completed, la réponse sur le statut de la tâche inclura le nombre total d’enregistrements réussis, échoués, insérés et mis à jour.

<div id="job-timeouts">
  ## Délais d’expiration des tâches
</div>

Toutes les tâches d’importation d’utilisateurs expirent après **deux (2) heures**. Si votre tâche n’est pas terminée dans ce délai, elle est marquée comme **Failed**.

De plus, toutes les données liées à votre tâche sont automatiquement supprimées après 24 heures et ne sont plus accessibles par la suite. Par conséquent, **nous vous recommandons fortement d’enregistrer les résultats des tâches à l’aide du mécanisme de stockage de votre choix**.

<div id="retrieve-failed-entries">
  ## Récupérer les entrées ayant échoué
</div>

<Warning>
  Toutes les données liées à la tâche sont automatiquement supprimées après 24 heures et ne peuvent plus être consultées ensuite. C’est pourquoi nous vous recommandons fortement de stocker les résultats de la tâche à l’aide du mécanisme de stockage de votre choix.
</Warning>

<div id="best-practices-for-large-scale-migrations">
  ## Bonnes pratiques pour les migrations à grande échelle
</div>

Lors de l’importation d’un grand nombre d’utilisateurs (nécessitant 10 tâches ou plus), suivez ces recommandations pour garantir une migration réussie :

<div id="file-size-and-chunking">
  ### Taille des fichiers et segmentation
</div>

Divisez les enregistrements d’utilisateurs en petits blocs de moins de 400 KB par fichier. Cela permet de traiter environ 1 000 utilisateurs par tâche, avec moins de 10 champs de métadonnées par utilisateur, tout en restant bien en deçà de la limite de 500 KB par fichier.

<div id="job-scheduling-and-concurrency">
  ### Planification des tâches et gestion de la simultanéité
</div>

Comme Auth0 limite les importations d’utilisateurs à deux tâches simultanées par tenant, utilisez un framework de planification des tâches (comme [Bull](https://github.com/OptimalBits/bull) ou [Agenda](https://github.com/agenda/agenda)) pour :

* Planifier systématiquement les tâches d’importation
* Faire respecter la limite de simultanéité requise
* Suivre la progression de l’importation pour reprendre là où elle a été interrompue
* Conserver l’historique des tâches et leurs états actuels

<div id="error-handling-strategy">
  ### Stratégie de gestion des erreurs
</div>

* N’interrompez pas le processus d’importation en cas d’échec pour un seul utilisateur. Importez tous les utilisateurs, puis traitez les problèmes par la suite.
* Mettez en place une tâche de « finalisation » qui passe en revue toutes les tâches générées et regroupe les enregistrements en échec dans un nouveau fichier.
* Corrigez les erreurs dans les enregistrements en échec recueillis et importez-les dans une nouvelle tâche.
* Prévoyez une stratégie de nouvelle tentative pour gérer les pertes de connexion réseau et les échecs temporaires.

<div id="upsert-mode-considerations">
  ### Considérations concernant le mode upsert
</div>

Soyez prudent lorsque vous activez le mode `upsert` :

* Les importations avec upsert sont plus lentes que les importations standard.
* Le mode upsert ne prend pas en charge la fusion de `user_metadata` ou de `app_metadata`. Les métadonnées existantes seront entièrement écrasées par les nouvelles valeurs.
* N’utilisez le mode upsert que si vous devez mettre à jour des utilisateurs existants et que vous comprenez bien ces limites.

<div id="data-retention">
  ### Conservation des données
</div>

Notez que toutes les données liées aux tâches sont automatiquement supprimées après 24 heures. Conservez les résultats des tâches et les détails sur les erreurs dans votre propre mécanisme de stockage afin d’en assurer le suivi et l’analyse à long terme.
