<div id="short-lived-sessions">
  ## Sessions de courte durée
</div>

Ce flux de travail montre comment le SDK auth0-spa-js doit être mis en œuvre pour prendre en charge la gestion des sessions multisites. Dans ce scénario, on suppose que le délai d’expiration d’inactivité <Tooltip tip="Single Sign-On (SSO) : service qui, après qu’un utilisateur se connecte à une application, le connecte automatiquement aux autres applications." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=SSO">SSO</Tooltip> du tenant est défini à 300 secondes, et que l’expiration du <Tooltip tip="ID Token : jeton destiné au client lui-même, plutôt qu’à l’accès à une ressource." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=ID+Token">ID Token</Tooltip> de chaque application SPA est définie à 150 secondes. Il s’agit d’une session « de courte durée ».

<div id="sdk-features">
  ### Fonctionnalités du SDK
</div>

<div id="pkce-flow">
  #### Flux PKCE
</div>

Pour toutes les méthodes permettant d’obtenir un ID Token ou un <Tooltip tip="Jeton d’accès : donnée d’autorisation, sous la forme d’une chaîne opaque ou d’un JWT, utilisée pour accéder à une API." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Access+Token">jeton d’accès</Tooltip>, le SDK s’occupe de toute la complexité liée à la clé de preuve pour l’échange de code. Aucun effort ni aucune configuration supplémentaire ne sont nécessaires pour que cela fonctionne.

<div id="deep-linking">
  #### Liens profonds
</div>

Pour améliorer l’expérience utilisateur, le SDK comprend un paramètre `appState` pour la méthode `loginWithRedirect()`. Des détails sur l’application en cours sont inclus dans la requête envoyée au serveur Auth et sont renvoyés une fois l’authentification réussie, assurant ainsi une continuation harmonieuse du parcours utilisateur.

Dans le Quickstart, le composant `PrivateRoute` définit un paramètre state `targetUrl`, et la fonction `onRedirectCallback` de `index.js` récupère cette valeur pour rediriger l’utilisateur lorsque l’authentification est terminée.

<div id="token-storage">
  #### Stockage des jetons
</div>

Pour stocker les jetons renvoyés de la façon la plus sécuritaire possible, tous les jetons sont placés dans un cache local. Les ID Token et jetons d’accès sont stockés par paires, et les valeurs d’<Tooltip tip="Audience : identifiant unique de l’audience d’un jeton émis. Nommée aud dans un jeton, sa valeur contient l’ID soit d’une application (Client ID) pour un ID Token, soit d’une API (identifiant d’API) pour un jeton d’accès." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=audience">audience</Tooltip> et de portée servent à récupérer les jetons au besoin.

De plus, les jetons en cache sont supprimés dès que l’ID Token ou le jeton d’accès expire, de sorte que si un jeton se trouve dans le cache, on peut présumer qu’il est toujours valide.

<div id="call-apis">
  #### Appeler les API
</div>

La méthode `getTokenSilently()` sert d’abord à utiliser le cache de jetons et, s’il n’y en a pas, lance un iframe invisible pour récupérer un nouveau jeton. À cette fin, toutes les requêtes aux API peuvent utiliser cette méthode pour construire l’en-tête du jeton Bearer, sans logique supplémentaire pour gérer les jetons expirés.

Dans le Quickstart, la vue `ExternalService` envoie une requête à l’API Express à l’aide de cette fonctionnalité.

<div id="warn-users-to-continue-their-sessions">
  #### Avertissez les utilisateurs pour qu’ils poursuivent leur session
</div>

Si un utilisateur n’a effectué aucune action qui entraînerait la mise à jour de la session Auth0, Auth0 recommande d’afficher un avertissement invitant l’utilisateur à choisir explicitement de poursuivre sa session.

Cette approche permet à la session de devenir inactive si l’utilisateur n’est plus présent, tout en offrant un moyen de déclencher l’actualisation silencieuse du jeton pour qu’il puisse poursuivre sa session sans avoir à fournir de nouveau ses identifiants.

Pour en savoir plus sur les minuteurs d’inactivité et les boîtes de dialogue d’expiration, consultez [les URL de déconnexion propres à l’application](/docs/fr-CA/manage-users/sessions/configure-session-lifetime-settings).

<div id="example-workflow">
  ### Exemple de flux de travail
</div>

1. Authentification initiale
2. Maintien de la session Auth0
3. Seamless SSO
4. Demander à l&#39;utilisateur de prolonger la session
5. L&#39;utilisateur se déconnecte explicitement de l&#39;application
6. L&#39;utilisateur retourne à l&#39;application initiale après s&#39;être déconnecté

<div id="initial-authentication">
  #### Authentification initiale
</div>

1. Ouverture d’un nouvel onglet
2. Requête de connexion
3. L’utilisateur saisit ses identifiants
4. Le cookie SSO (avec expiration) est créé
5. Échange de jetons effectué

<div id="maintain-auth0-session">
  #### Maintien de la session Auth0
</div>

1. L’utilisateur demande des données à une ressource protégée
2. `getTokenSilently()` est appelé
3. La ressource est récupérée
4. L’utilisateur met à jour des données dans la ressource protégée
5. `getTokenSilently()` est appelé

   1. Une iframe s’ouvre
   2. Échange de jeton effectué
6. La ressource est mise à jour

<div id="seamless-sso">
  #### Seamless SSO
</div>

1. L’utilisateur accède à une route privée
2. Vérifiez avec `isAuthenticated()`
3. Si ce n’est pas le cas, `loginWithRedirect()`

<div id="prompt-user-to-extend-session">
  #### Demander à l’utilisateur de prolonger la session
</div>

1. Après 240 secondes, demander à l’utilisateur s’il veut garder la session active au moyen d’une fenêtre modale affichée pendant 60 secondes
2. S’il choisit de maintenir la session, `getTokenSilently()`

  #### L&#39;utilisateur se déconnecte explicitement de l&#39;application
</div>

1. L&#39;utilisateur choisit de se déconnecter
2. `logout()` est appelé

   1. Vider le cache de jetons
   2. Appeler `/oidc/logout`
   3. Effacer le cookie SSO et supprimer les données de session
   4. Rediriger l&#39;utilisateur vers la page de déconnexion

<div id="long-lived-sessions">
  ## Sessions de longue durée
</div>

Auth0 prend en charge les sessions de longue durée pour les forfaits Enterprise. Avec les sessions de longue durée, vous pouvez configurer des limites de session allant jusqu’à 100 jours d’inactivité (délai d’inactivité) et jusqu’à un an au total (délai d’expiration absolu). Si vous fonctionnez selon des échéanciers trimestriels, mensuels ou autres, cela vous permet de réduire la friction pour les utilisateurs finaux et de donner accès à du contenu et à des fonctionnalités à faible risque. De plus, les entreprises de médias peuvent tirer parti des sessions de longue durée pour améliorer l’expérience utilisateur grâce à un accès transparent au contenu. Vous pouvez également choisir entre les sessions de longue durée et la validation du mot de passe selon vos exigences en matière d’expérience utilisateur et de sécurité.

Les détails du flux de travail changeraient dans le cas d’une session de longue durée, où la session d’application serait très probablement plus courte que la session de [l’authentification unique (SSO)](/docs/fr-CA/authenticate/single-sign-on).

Pour en savoir plus, consultez [Configurer les limites de durée de session](/docs/fr-CA/manage-users/sessions/configure-session-lifetime-settings) et [Mettre à jour la durée de vie du jeton d’accès](/docs/fr-CA/secure/tokens/access-tokens/update-access-token-lifetime).
