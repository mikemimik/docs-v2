Auth0 distingue trois types de métadonnées servant à stocker des catégories précises d’information.

<Warning>
  Les métadonnées Auth0 ne constituent pas un stockage sécurisé et ne doivent pas être utilisées pour stocker des renseignements sensibles, comme des secrets à haut risque et des renseignements personnels identifiables (PII), tels que des numéros d’assurance sociale ou de carte de crédit. Auth0 recommande fortement à ses clients d’évaluer les données stockées dans les métadonnées et de n’y conserver que celles qui sont nécessaires à la gestion des identités et des accès.
</Warning>

| Type de métadonnées                | Nom du champ                                                                                                                    | Description                                                                                                                                                                                                                                                                                                                        |
| ---------------------------------- | ------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Informations utilisateur**       | `user_metadata`                                                                                                                 | Stocke des attributs utilisateur, comme des préférences, qui n’ont pas d’incidence sur les fonctionnalités de base de l’utilisateur. Ces données **peuvent** être modifiées par des utilisateurs connectés si vous créez un formulaire à l’aide de la Management API et ne doivent **pas** être utilisées comme stockage sécurisé. |
| **Informations d’accès**           | `app_metadata`                                                                                                                  | Stocke des renseignements comme les permissions, le plan Auth0 et les ID externes qui peuvent avoir une incidence sur l’accès de l’utilisateur à certaines fonctionnalités. Ces données **ne peuvent pas** être modifiées par les utilisateurs, et il existe des restrictions quant à ce qui peut être stocké dans ce champ.       |
| **Informations sur l’application** | `client_metadata` in the `Client` object, `context.clientMetadata` in Rules, and `event.client.metadata` in post-login Actions. | Stocke des renseignements sur une application (ou un *client* dans la terminologie OIDC OAuth2). Par exemple, l’URL de la page d’accueil de l’application (toute valeur qu’Auth0 ne définit pas dans les paramètres de l’application).                                                                                             |

<div id="metadata-field-names">
  ## Noms de champ des métadonnées
</div>

<div id="accepted-characters">
  ### Caractères autorisés
</div>

Les noms de champ ne doivent pas contenir les caractères `.` (point) ni `$` (signe du dollar).

Par exemple, ceci n’est pas permis :

```json lines
{
  "preference.color": "pink"
}
```

Mais vous pouvez le développer ainsi :

```json lines
{
    "preference": { 
        "color": "pink" 
    }
}
```

<div id="dynamic-field-names">
  ### Noms de champs dynamiques
</div>

Les noms de champs doivent être statiques. Les noms de champs dynamiques réduisent l’efficacité de l’indexation et nuisent aux performances des requêtes de recherche. Un schéma statique est plus facile à parcourir, à manipuler et à utiliser.

Au lieu de faire ceci :

```json lines
{
    "participants": [
        "Alice": {
            "role": "sender"
         },
        "Bob": {
            "role": "receiver"
        }
    ]
}
```

Faites ceci :

```json lines
{
    "participants": [
        {
            "name": "Alice",
            "role": "sender"
        },
        {
            "name" : "Bob",
            "role": "receiver"
        }
    ]
}
```

<div id="name-collision">
  ### Collision de noms
</div>

Évitez d’utiliser le même nom pour les champs `app_metadata` et les champs à la racine du profil. Le champ `app_metadata` est fusionné avec le profil racine dans les Rules et les Actions, ce qui peut écraser des champs à la racine du profil.

Par exemple, si un utilisateur a un champ `groups` à la racine de son profil (renvoyé par un <Tooltip tip="Security Assertion Markup Language (SAML) : protocole normalisé permettant à deux parties d’échanger des renseignements d’authentification sans mot de passe." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=SAML">SAML</Tooltip> <Tooltip tip="Identity Provider (IdP) : service qui stocke et gère les identités numériques." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=identity+provider">fournisseur d’identité</Tooltip>) et un champ `groups` dans `app_metadata`, son profil pourrait ressembler à ceci :

```json lines
{
    "user_id": "samlp|example-samlp-connection|username@domain.com",
    "groups": [
        "external-group-1",
        "external-group-2"
    ],
    "app_metadata": {
        "groups": [
            "internal-group-1",
            "internal-group-2"
        ]
    }
}
```

Lorsque vous lisez le champ `groups` de l’[objet `user`](/docs/fr-CA/customize/rules/user-object-in-rules) dans une [Rule](/docs/fr-CA/manage-users/user-accounts/metadata/manage-metadata-rules), la valeur renvoyée est : `["internal-group-1", "internal-group-2"]`.

<div id="metadata-data-types">
  ## Types de données des métadonnées
</div>

Les champs de métadonnées prennent en charge tous les [types de données compatibles avec JSON](https://datatracker.ietf.org/doc/html/rfc7159) :

* String
* Number
* Array
* Object

Veillez à conserver des types de données uniformes d’un utilisateur à l’autre. Par exemple, si vous stockez une valeur sous forme de chaîne pour un utilisateur (`user.user_metadata.age = "23"`) et sous forme de nombre pour un autre (`user.user_metadata.age = 23`), vous pourriez rencontrer des problèmes au moment de récupérer les données.

<div id="limitations-and-restrictions">
  ## Limitations et restrictions
</div>

<div id="rate-limits">
  ### Limites de débit
</div>

Lorsque vous mettez à jour des métadonnées pendant la connexion avec [Rules](/docs/fr-CA/manage-users/user-accounts/metadata/manage-metadata-rules) ou [Actions](/docs/fr-CA/manage-users/user-accounts/metadata/manage-user-metadata), vous êtes assujetti aux limites de débit de votre tenant. Pour en savoir plus, consultez [Limites de débit des points de terminaison de la Management API](/docs/fr-CA/troubleshoot/customer-support/operational-policies/rate-limit-policy/management-api-endpoint-rate-limits).

<Warning>
  Vous devriez uniquement stocker dans les métadonnées des données liées à l’authentification des utilisateurs. Les capacités de stockage et de recherche d’Auth0 sont conçues pour des cas d’utilisation qui n’exigent pas des recherches intensives ou des mises à jour fréquentes.

  Si vous devez conserver des données détaillées sur le profil des utilisateurs, vous devriez le faire dans un système externe. Vous pouvez stocker dans Auth0 l’identifiant de l’utilisateur dans ce système à titre de champ de métadonnées.
</Warning>

<div id="size-limits-and-storage">
  ### Limites de taille et stockage
</div>

* Il existe une limite de 1 Mo par utilisateur pour les données utilisateur qui peuvent être indexées, interrogées et renvoyées par le [point de terminaison de recherche des utilisateurs](https://auth0.com/docs/api/management/v2/users/get-users). Si un profil utilisateur dépasse 1 Mo, les valeurs d’attribut de plus de 256 caractères dans `app_metadata` et `user_metadata` ne seront ni interrogeables ni renvoyées dans les résultats de recherche. Si le profil utilisateur dépasse encore 1 Mo après l’omission de ces valeurs volumineuses, aucun des attributs `app_metadata` et `user_metadata` ne sera alors interrogeable ni renvoyable pour cet utilisateur. Auth0 enregistre et consigne les cas où un profil utilisateur dépasse toujours 1 Mo après ces omissions sous le [code d’événement](/docs/fr-CA/deploy-monitor/logs/log-event-type-codes) `wum`. Le [point de terminaison Get User](https://auth0.com/docs/api/management/v2/users/get-users-by-id) doit être utilisé pour récupérer tous les attributs de métadonnées des profils utilisateur trop volumineux.
* Lorsque vous définissez le champ `user_metadata` à l’aide du [point de terminaison Signup](https://auth0.com/docs/api/authentication?javascript#signup) de l’Auth0 Authentication API, vous pouvez inclure au maximum 10 champs de type chaîne dont les valeurs ne dépassent pas 500 caractères chacune. Pour voir un exemple d’utilisation des métadonnées dans un processus d’inscription personnalisé, consultez [Custom Signup](/docs/fr-CA/libraries/custom-signup).
* Le champ `client_metadata` peut contenir au maximum 10 clés. Ses clés et ses valeurs ont chacune une longueur maximale de 255 caractères et ne peuvent pas contenir de caractères spéciaux UTF-8.

<div id="restrictions">
  ### Restrictions
</div>

Le champ `app_metadata` ne doit contenir aucune des propriétés suivantes :

* `__tenant`
* `_id`
* `blocked`
* `clientID`
* `created_at`
* `email_verified`
* `email`
* `globalClientID`
* `global_client_id`
* `identities`
* `lastIP`
* `lastLogin`
* `loginsCount`
* `metadata`
* `multifactor_last_modified`
* `multifactor`
* `updated_at`
* `user_id`
