title: 'Exemples de cas d’utilisation : Role-Based Access Control'
---

Voyons un exemple de la raison pour laquelle vous pourriez avoir besoin du [contrôle d’accès basé sur les rôles (RBAC)](/docs/fr-CA/manage-users/access-control/rbac) et de la façon dont vous pourriez l’utiliser dans votre <Tooltip tip="Flux d’autorisation : mécanisme d’autorisation (ou workflow) spécifié dans le framework OAuth 2.0." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=authorization+flow">flux d’autorisation</Tooltip>.

Supposons que vous êtes une entreprise qui offre un logiciel-service interentreprises à des organismes sans but lucratif. Votre produit permet à ces organismes de créer, de gérer et de commercialiser des produits auprès de donateurs potentiels. Votre application comprend plusieurs modules, dont les deux suivants :

* un module de point de vente (POS) de boutique-cadeaux qui permet aux organismes sans but lucratif de créer efficacement des boutiques éphémères de t-shirts et d’en gérer les ventes;
* un module de marketing qui permet aux organismes sans but lucratif de créer et de distribuer des infolettres à leurs donateurs.

Vous voulez utiliser Auth0 pour contrôler l’accès de vos clients sans but lucratif aux différentes parties de votre application. Sans le RBAC, tous les employés et bénévoles de ces organismes auraient accès à toutes les fonctionnalités de votre application, ce qui n’est pas idéal, surtout si l’un d’eux est un refuge pour animaux qui compte des bénévoles ayant seulement des connaissances dans le domaine où ils font du bénévolat.

Vous mettez donc en œuvre le RBAC en [créant certaines permissions](/docs/fr-CA/get-started/apis/add-api-permissions) dont les utilisateurs de votre module POS de boutique-cadeaux auraient besoin :

* `read:catalog-item`
* `read:customer-profile`
* `create:invoice`

Et, pour en faciliter la gestion, vous [créez un rôle](/docs/fr-CA/manage-users/access-control/configure-core-rbac/roles/create-roles) appelé `Gift Shop Manager` et [ajoutez ces permissions à ce rôle](/docs/fr-CA/manage-users/access-control/configure-core-rbac/roles/add-permissions-to-roles).

De même, vous créez des permissions pour les utilisateurs de votre module de marketing, notamment :

* `create:newsletter`
* `edit:newsletter`
* `delete:newsletter`
* `send:newsletter`
* `edit:distribution-list`

Et vous créez un rôle appelé `Newsletter Admin` et ajoutez ces permissions à ce rôle.

Ainsi, lorsque votre refuge pour animaux fait appel à sa bénévole, Astrid, pour gérer sa boutique éphémère de t-shirts, Astrid peut se voir [attribuer le rôle](/docs/fr-CA/manage-users/access-control/configure-core-rbac/rbac-users/assign-roles-to-users) `Gift Shop Manager`. Lorsque vous attribuez ce rôle à Astrid, elle obtient toutes les permissions associées à ce rôle. Comme Astrid n’y connaît rien à la publication d’infolettres (et qu’elle n’est pas très à l’aise avec le courriel), vous ne lui avez jamais attribué le rôle `Newsletter Admin`, donc elle n’a jamais accès au module de marketing.

D’un point de vue plus technique, lorsqu’Astrid ouvre une session dans votre produit, Auth0 l’authentifie et l’autorise, puis inclut les permissions dans le <Tooltip tip="Jeton d’accès : information d’identification d’autorisation, sous la forme d’une chaîne opaque ou d’un JWT, utilisée pour accéder à une API." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Access+Token">jeton d’accès</Tooltip> renvoyé. Ensuite, votre produit inspecte le token pour déterminer quel module afficher à Astrid.

En utilisant le RBAC d’Auth0, vous évitez d’avoir à créer et à maintenir des systèmes d’autorisation distincts; vous utilisez plutôt le token que vous recevez déjà pendant l’autorisation. Et lorsqu’Astrid déménage ou décide qu’elle est fatiguée de gérer la boutique-cadeaux et préférerait coordonner le programme de familles d’accueil, vous pouvez facilement [lui retirer le rôle Gift Shop Manager](/docs/fr-CA/manage-users/access-control/configure-core-rbac/rbac-users/remove-roles-from-users) et lui attribuer un nouveau rôle.

Et si la gestion des rôles et des permissions pour l’ensemble de vos clients devient trop lourde, vous pouvez aussi utiliser l’Auth0 API pour créer un module dans votre produit qui permet aux clients de gérer leur propre RBAC, réduisant ainsi la responsabilité et les coûts en personnel.
