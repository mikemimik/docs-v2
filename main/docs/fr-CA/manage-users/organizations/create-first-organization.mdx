Pour commencer à utiliser la fonctionnalité [Organizations](/docs/fr-CA/manage-users/organizations/organizations-overview), vous devez créer et configurer vos organisations.

<Card title="La disponibilité varie selon le forfait Auth0">
  Votre forfait Auth0 ou votre entente personnalisée détermine si cette fonctionnalité est offerte. Pour en savoir plus, consultez [Pricing](https://auth0.com/pricing).
</Card>

<div id="create-an-organization">
  ## Créer une organisation
</div>

Lorsque vous [créez une organisation](/docs/fr-CA/manage-users/organizations/configure-organizations/create-organizations), les paramètres que vous définissez servent à personnaliser la page de connexion et les modèles de courriel que les utilisateurs finaux voient lorsqu’ils s’authentifient dans le contexte de l’organisation.

Si vous utilisez Organizations tel quel, ces paramètres remplaceront ceux des pages <Tooltip tip="Universal Login : votre application redirige vers Universal Login, hébergé sur le serveur d’autorisation d’Auth0, pour vérifier l’identité d’un utilisateur." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Universal+Login">Universal Login</Tooltip>. Pour en savoir plus sur la personnalisation plus poussée de la page de connexion ou des modèles de courriel, consultez [Personnaliser les prompts et les modèles de courriel](#customize-prompts-and-email-templates).

<div id="define-organization-behavior">
  ## Définir le comportement de l’organisation
</div>

Vous connaissez peut-être des applications comme Heroku, dont le comportement varie selon qu’un utilisateur se connecte avec un compte personnel ou choisit une organisation associée à son compte d’entreprise. Auth0 vous permet de définir un comportement semblable pour une organisation au niveau de l’application.

Pour chaque application que vous créez dans Auth0, vous pouvez décider si les utilisateurs finaux doivent se connecter directement ou s’ils doivent s’authentifier dans le contexte d’une organisation. Pour les applications qui exigent que les utilisateurs se connectent par l’intermédiaire d’une organisation, vous pouvez aussi préciser ce qui se passe si aucune organisation n’est fournie au flux de connexion.

Vous voudrez peut-être aussi mettre à jour les **Allowed Callback URLs** et les **Allowed Origins (CORs)** de votre Application dans les [paramètres de l’application](/docs/fr-CA/get-started/applications/application-settings) afin d’y inclure des [espaces réservés Organization pour les sous-domaines](/docs/fr-CA/get-started/applications/wildcards-for-subdomains).

<div id="customize-prompts-and-email-templates">
  ## Personnaliser les prompts et les modèles de courriel
</div>

Lorsque vous utilisez Organizations avec les prompts prêts à l’emploi de Universal Login, les paramètres d’image de marque que vous configurez lorsque vous [créez des organisations](/docs/fr-CA/manage-users/organizations/configure-organizations/create-organizations) remplacent l’image de marque des pages de Universal Login et des modèles de courriel. Si vous souhaitez personnaliser davantage les pages de Universal Login et les courriels reçus par l’utilisateur final, vous pouvez personnaliser les modèles de page et de courriel.

<div id="page-templates">
  ### Modèles de page
</div>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Pour accéder à la personnalisation des modèles de page, vous devez utiliser des domaines personnalisés, une fonctionnalité incluse dans tout forfait payant.
</Callout>

Pour modifier la page Universal Login, personnalisez un modèle de page à l’aide du [langage de modèles Liquid](https://shopify.github.io/liquid/) et de variables de modèle, puis appliquez ce modèle à l’aide de l’API Universal Login Page Templates. Pour en savoir plus, consultez [Universal Login Page Templates](/docs/fr-CA/customize/login-pages/universal-login/customize-templates).

Comme le même modèle est utilisé pour toutes les pages, vous pouvez créer des pages de connexion uniformes avec un minimum d’effort.

Le modèle le plus simple que vous pouvez écrire est :

```html lines
<!DOCTYPE html>
<html>
  <head>
    {%- auth0:head -%}
  </head>
  <body>
    {%- auth0:widget -%}
  </body>
</html>
```

Les balises suivantes doivent être présentes dans le gabarit :

* `auth0:widget` : Contient le code HTML qui définit la structure du widget affiché sur chaque type de page (p. ex., Login, Reset Password).
* **`auth0:head`** : Contient les balises requises pour afficher le widget.

Pour centrer le widget sur la page, remplacez la balise `<body>` par `<body class="_widget-auto-layout">`.

<div id="email-templates">
  ### Modèles de courriel
</div>

Pour modifier les courriels que reçoit l’utilisateur final, personnalisez un modèle de courriel à l’aide du [langage de modèles Liquid](https://shopify.github.io/liquid/) et des variables de modèle. Pour en savoir plus, consultez [Personnaliser les modèles de courriel](/docs/fr-CA/customize/email/email-templates).

Les modèles de courriel suivants peuvent aussi être personnalisés davantage pour les organisations :

* **Bienvenue** : Reçu par l’utilisateur final après qu’il a vérifié son adresse courriel ou, si la vérification de l’adresse courriel est désactivée, au moment de son inscription (ou de sa première connexion).
* **Changement de mot de passe** : Reçu par l’utilisateur final lorsqu’il demande un changement de mot de passe. Contient un lien qui le redirige vers la page de réinitialisation du mot de passe.
* **Inviter un utilisateur** : Reçu par l’utilisateur final lorsqu’il est invité dans une organisation. Contient un lien qui le redirige vers la route de connexion par défaut que vous avez configurée. Pour en savoir plus, consultez [Inviter des membres d’une organisation](/docs/fr-CA/manage-users/organizations/configure-organizations/invite-members).

<div id="template-variables">
  ### Variables de modèle
</div>

Les modèles de page et de courriel peuvent avoir accès à un ensemble de variables de contexte que vous pouvez utiliser pour influer sur la façon dont le modèle est rendu.

Pour consulter la liste des variables disponibles, voyez [Universal Login Page Templates](/docs/fr-CA/customize/login-pages/universal-login/customize-templates) pour les variables des modèles de page, et [Personnaliser les modèles de courriel](/docs/fr-CA/customize/email/email-templates) pour les variables communes des modèles de courriel.

Le modèle de page dispose de variables supplémentaires lorsqu’un utilisateur se connecte par l’intermédiaire d’une organisation :

* `organization.id`
* `organization.display_name`
* `organization.name`
* `organization.metadata`
* `organization.branding.logo_url`
* `organization.branding.colors.primary`
* `organization.branding.colors.page_background`

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Les paramètres d’organisation ne sont exposés à un modèle de courriel que lorsque le courriel associé est déclenché dans le contexte d’une organisation. Par exemple, le courriel d’invitation d’utilisateur est toujours déclenché dans le contexte d’une organisation; les paramètres d’organisation sont donc toujours exposés à son modèle. En revanche, les courriels de bienvenue peuvent être déclenchés ou non dans le contexte d’une organisation; les paramètres d’organisation ne seront donc pas toujours disponibles dans leur modèle.
</Callout>

<div id="configure-organization-domains">
  ## Configurer les domaines d’Organization
</div>

<Warning>
  Organization Domains for Discovery est actuellement en Early Access. En utilisant cette fonctionnalité, vous acceptez les conditions applicables de l’essai gratuit figurant dans le [Contrat-cadre d’abonnement d’Okta](https://www.okta.com/legal). Pour en savoir plus sur les étapes de lancement du produit, consultez [Product Release Stages](/docs/fr-CA/troubleshoot/product-lifecycle/product-release-stages).
</Warning>

Pendant l’étape pre-login, Auth0 peut détecter les Organizations d’un utilisateur soit à partir du nom de l’Organization, soit à partir du domaine de courriel qu’il saisit. Pour associer une Organization à un domaine, vous pouvez créer des domaines d’Organization. Auth0 les achemine ensuite automatiquement vers le bon Identity Provider (IdP).

<Note>
  Remarque : les utilisateurs doivent tout de même saisir leur adresse courriel complète ou le nom de l’Organization pendant le login. Auth0 utilise cette information pour déterminer l’Organization associée.
</Note>

<div id="add-a-domain-to-an-organization">
  ### Ajouter un domaine à une Organization
</div>

Les administrateurs de tenant peuvent ajouter et gérer des domaines dans la section Domains d’une Organization de l’Auth0 Dashboard ou au moyen de la Management API.

Pour ajouter un domaine dans l’Auth0 Dashboard :

1. Accédez à [Auth0 Dashboard &gt; Organizations](https://manage.auth0.com/#/organizations).
2. Sélectionnez le nom de l’Organization. Sélectionnez **Domains**.
3. Choisissez **Add Domain**.
4. Saisissez le domaine (par exemple, `travel0.com`)
5. Une fois vérifié, le domaine s’affichera comme Verified et sera utilisé pour la découverte d’Organization avant le login.

<Note>
  Seuls les domaines vérifiés sont utilisés dans le cadre de la découverte de domaine. Les domaines en attente ou non vérifiés ne déclencheront pas le routage.
</Note>

Pour en savoir plus sur la façon dont la découverte de domaine influence le comportement de login, consultez [Login Flows for Organizations](https://auth0.com/docs/manage-users/organizations/login-flows-for-organizations).

<div id="configure-organization-membership">
  ## Configurer l’appartenance à une organisation
</div>

Chaque organisation a ses propres membres, c’est-à-dire les utilisateurs qui peuvent accéder à vos applications en se connectant par la page de connexion configurée de l’organisation.

Les applications SaaS et B2B sont généralement fournies à des équipes plutôt qu’à des utilisateurs individuels. Dans ce contexte, une équipe peut correspondre à une entreprise entière, à un petit groupe d’employés (par exemple, le département du marketing) ou même à un groupe temporaire d’utilisateurs réunis autour d’un objectif commun (par exemple, un groupe de voisins qui se sont inscrits pour collaborer avec un organisme de bienfaisance et qui rivalisent en équipe pour amasser le plus d’argent possible pour une cause).

Les Organizations dans Auth0 sont assez flexibles pour répondre à tous ces cas d’utilisation, mais la création d’une application qui prend en charge des équipes exige une certaine planification. L’un des premiers éléments à considérer est la façon de gérer l’appartenance à l’équipe, ce que vous pouvez faire en :

* [invitant des utilisateurs par courriel](/docs/fr-CA/manage-users/organizations/configure-organizations/invite-members)
* [accordant l’appartenance juste à temps](/docs/fr-CA/manage-users/organizations/configure-organizations/grant-just-in-time-membership) aux utilisateurs qui se connectent au moyen d’une connexion activée
* gérant directement l’appartenance au moyen de la <Tooltip tip="Management API : Un produit permettant aux clients d’effectuer des tâches administratives." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Management+API">Management API</Tooltip> ou du <Tooltip tip="Management API : Un produit permettant aux clients d’effectuer des tâches administratives." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip> en [assignant des membres](/docs/fr-CA/manage-users/organizations/configure-organizations/assign-members) ou en [supprimant des membres](/docs/fr-CA/manage-users/organizations/configure-organizations/remove-members)

Une fois l’appartenance établie, vous pouvez [récupérer les membres des organisations](/docs/fr-CA/manage-users/organizations/configure-organizations/retrieve-members) ou [récupérer l’appartenance d’un utilisateur à une organisation](/docs/fr-CA/manage-users/organizations/configure-organizations/retrieve-user-membership).

<div id="assign-roles-to-members">
  ## Attribuer des rôles aux membres
</div>

Chaque membre de l’organisation peut se voir attribuer un ou plusieurs rôles, qui s’appliquent lorsque les utilisateurs ouvrent une session par l’intermédiaire de l’organisation. Plus précisément, vous pouvez [ajouter des rôles aux membres de l’organisation](/docs/fr-CA/manage-users/organizations/configure-organizations/add-member-roles), [supprimer des rôles de membres de l’organisation](/docs/fr-CA/manage-users/organizations/configure-organizations/remove-member-roles) ou [récupérer les rôles des membres d’une organisation](/docs/fr-CA/manage-users/organizations/configure-organizations/retrieve-member-roles). Vous pouvez utiliser les rôles et les permissions qui y sont associées pour contrôler l’accès aux [applications que vous créez](/docs/fr-CA/manage-users/organizations/custom-development). Pour en savoir plus sur les rôles, consultez [le contrôle d’accès basé sur les rôles](/docs/fr-CA/manage-users/access-control/rbac).
