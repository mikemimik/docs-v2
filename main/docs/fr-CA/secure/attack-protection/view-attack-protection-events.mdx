---
title: Afficher les événements de journal de protection contre les attaques
description: Décrit comment utiliser les données de journal du trafic du tenant pour afficher les événements de protection contre les attaques.
---

Les journaux de votre tenant contiennent des données utiles que vous pouvez utiliser pour créer des graphiques afin d’analyser le profil du trafic qui transite par votre tenant. Cela est utile pour évaluer l’activité liée à la <Tooltip tip="Protection contre les attaques : fonctionnalités qu’Auth0 fournit pour détecter et atténuer les attaques, notamment la protection contre les attaques par force brute, la limitation des IP suspectes, la détection des mots de passe compromis, la détection des robots et l’authentification multifacteur adaptative." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=attack+protection">protection contre les attaques</Tooltip>. Par exemple, vous pouvez rechercher les événements suivants pour déterminer si vous êtes la cible d’une attaque :

* Des pointes anormales de trafic vers le flux de connexion qui entraînent des erreurs (par exemple, des erreurs de nom d’utilisateur ou de mot de passe incorrect).
* Des pointes anormales de trafic provenant de zones géographiques d’adresses IP inattendues.

Ces événements ont tendance à se produire sans changement important du taux de connexions réussies.

Vous pouvez utiliser le champ `event` des données de journal de votre tenant pour afficher les données de trafic du tenant. Nous recommandons de créer un histogramme quotidien des événements d’échec des types suivants :

| Code d’événement | Événement                                                |
| ---------------- | -------------------------------------------------------- |
| `f`              | Échec de connexion                                       |
| `fcoa`           | Échec de l’authentification inter-origine                |
| `feccft`         | Échec de l’échange                                       |
| `fepft`          | Échec de l’échange                                       |
| `fsa`            | Échec de l’authentification silencieuse                  |
| `fu`             | Échec de connexion (courriel/nom d’utilisateur invalide) |
| `pla`            | Évaluation pré-connexion                                 |
| `sepft`          | Échange réussi                                           |

Ces événements d’échec dépendent du flow que vous avez configuré avec Auth0.

L’exemple suivant montre une attaque par bourrage d’identifiants le 13/02, avec une forte hausse des événements de type `fu`, ce qui correspond à un échec lié au nom d’utilisateur (typique d’une attaque par bourrage d’identifiants).

<div id="rate-of-errors-in-login-flow">
  ## Taux d’erreurs dans le flux de connexion
</div>

Surveillez toute hausse soudaine ou tout nombre anormal d’erreurs liées à un nom d’utilisateur ou à un mot de passe incorrect. Par exemple : vous attendez-vous à plus de 30 000 erreurs par heure ?

| Code d’événement | Événement                                               |
| ---------------- | ------------------------------------------------------- |
| `s`              | Connexion réussie                                       |
| `fu`             | Échec de connexion, courriel/nom d’utilisateur invalide |
| `fp`             | Échec de connexion, mot de passe incorrect              |

Voici à quoi les données pourraient ressembler.

<Frame>![Exemple de graphique montrant une hausse des échecs de connexion par rapport au trafic normal](/docs/images/cdy7uua7fh8z/5PQBhfw1B1yx8S2BnHSDTk/ab753016110fbf957540aa5a534fea38/Login_events_by_type_-_English.png)</Frame>

<div id="rate-of-attack-protection-events">
  ## Fréquence des événements de protection contre les attaques
</div>

Recherchez un volume anormalement élevé de trafic lié à des événements de protection contre les attaques, comme la <Tooltip tip="Breached Password Detection : forme de protection contre les attaques dans laquelle Auth0 avise vos utilisateurs s'ils utilisent une combinaison nom d'utilisateur/mot de passe qui a été compromise dans une fuite de données sur un site Web ou une application tiers." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=breached+password+detection">détection de mots de passe compromis</Tooltip> ou des attaques par force brute visant plusieurs comptes.

| Code d’événement  | Événement                                                      |
| ----------------- | -------------------------------------------------------------- |
| `limit_mu`        | Adresse IP bloquée                                             |
| `limit_wc`        | Compte bloqué                                                  |
| `pwd_leak`        | Mot de passe compromis à la connexion                          |
| `signup_pwd_leak` | Mot de passe compromis à l’inscription                         |
| `reset_pwd_leak`  | Mot de passe compromis lors de la récupération du mot de passe |

Voici un exemple de l’apparence possible des données.

<Frame>![Exemple de graphique des événements de détection d’anomalies](/docs/images/cdy7uua7fh8z/4hCSxSwOq5jmiwChEKnSuQ/731ba44b3be3fcb3f5a8e0cfac9ee89b/Types_-_English.png)</Frame>

<div id="number-of-ips-producing-errors-and-their-locations">
  ## Nombre d’adresses IP générant des erreurs et leur provenance
</div>

Recherchez un nombre élevé d’adresses IP provenant d’endroits peu plausibles. Par exemple : vous attendez-vous à voir du trafic provenant chaque jour de 10 000 adresses IP en Russie ? Examinez les données d’adresse `ip` en parallèle avec le trafic des événements `fu` pour déterminer d’où provient le trafic en échec.

Les données de géolocalisation IP ne sont pas disponibles dans les journaux du tenant, à moins de pouvoir les enrichir à partir d’une autre source.
