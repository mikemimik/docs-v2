---
title: Afficher les événements de journal de la protection contre les attaques
description: Décrit comment utiliser les données de journal du trafic du tenant pour afficher les événements de protection contre les attaques.
---

Les journaux du tenant contiennent des données utiles que vous pouvez exploiter pour créer des graphiques afin d’examiner le profil du trafic qui transite par votre tenant. Cela est utile pour évaluer l’activité de <Tooltip tip="Protection contre les attaques : fonctionnalités qu’Auth0 fournit pour détecter et atténuer les attaques, y compris la protection contre la force brute, la limitation des IP suspectes, la détection des mots de passe compromis, la détection de robots et l’authentification multifacteur adaptative." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=attack+protection">protection contre les attaques</Tooltip>. Par exemple, vous pouvez rechercher les événements suivants pour déterminer si vous êtes visé par une attaque :

* Des pointes anormales de trafic dans le flux de connexion qui entraînent des erreurs (par exemple, des erreurs de nom d’utilisateur ou de mot de passe incorrect).
* Des pointes anormales de trafic provenant de régions géographiques IP inhabituelles.

Ces événements ont tendance à se produire sans changement important du taux de connexions réussies.

Vous pouvez utiliser le champ `event` des données de journal de votre tenant pour afficher les données de trafic du tenant. Nous vous recommandons de créer un histogramme quotidien des événements d’échec des types suivants :

| Code d’événement | Événement                                               |
| ---------------- | ------------------------------------------------------- |
| `f`              | Échec de connexion                                      |
| `fcoa`           | Échec de l’authentification cross-origin                |
| `feccft`         | Échange échoué                                          |
| `fepft`          | Échange échoué                                          |
| `fsa`            | Échec de l’authentification silencieuse                 |
| `fu`             | Échec de connexion (courriel/nom d’utilisateur invalide) |
| `pla`            | Évaluation pré-login                                    |
| `sepft`          | Échange réussi                                          |

Ces événements d’échec dépendent du flow que vous avez configuré avec Auth0.

L’exemple suivant montre une attaque par bourrage d’identifiants le 13/02, avec une forte hausse des événements de type `fu`, ce qui correspond à un échec de nom d’utilisateur (typique d’une attaque par bourrage d’identifiants).

<div id="rate-of-errors-in-login-flow">
  ## Taux d’erreurs dans le flux de connexion
</div>

Recherchez une hausse soudaine ou un nombre anormal d’erreurs dues à un nom d’utilisateur ou à un mot de passe incorrect. Par exemple : vous attendez-vous à &gt;30 000 erreurs par heure ?

| Code d’événement | Événement                                               |
| ---------------- | ------------------------------------------------------- |
| `s`              | Connexion réussie                                       |
| `fu`             | Échec de connexion, courriel/nom d’utilisateur invalide |
| `fp`             | Échec de connexion, mot de passe incorrect              |

Voici un exemple de ce à quoi les données pourraient ressembler.

<Frame>![Exemple de graphique montrant une hausse des échecs de connexion par rapport au trafic normal](/docs/images/cdy7uua7fh8z/5PQBhfw1B1yx8S2BnHSDTk/ab753016110fbf957540aa5a534fea38/Login_events_by_type_-_English.png)</Frame>

<div id="rate-of-attack-protection-events">
  ## Taux des événements de protection contre les attaques
</div>

Surveillez un volume anormalement élevé d’événements de protection contre les attaques, comme <Tooltip tip="Détection des mots de passe compromis : forme de protection contre les attaques dans laquelle Auth0 avertit vos utilisateurs s’ils utilisent une combinaison nom d’utilisateur/mot de passe qui a été compromise dans une fuite de données sur un site Web ou une application d’un tiers." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=breached+password+detection">la détection des mots de passe compromis</Tooltip> ou des attaques par force brute visant plusieurs comptes.

| Code d’événement  | Événement                                                          |
| ----------------- | ------------------------------------------------------------------ |
| `limit_mu`        | Adresse IP bloquée                                                 |
| `limit_wc`        | Compte bloqué                                                      |
| `pwd_leak`        | Mot de passe compromis lors de la connexion                        |
| `signup_pwd_leak` | Mot de passe compromis lors de l’inscription                       |
| `reset_pwd_leak`  | Mot de passe compromis lors de la réinitialisation du mot de passe |

Voici un exemple de ce à quoi les données pourraient ressembler.

<div id="number-of-ips-producing-errors-and-their-locations">
  ## Nombre d’adresses IP générant des erreurs et leur provenance
</div>

Repérez un nombre élevé d’adresses IP provenant de régions inhabituelles. Par exemple : vous attendez-vous à recevoir chaque jour du trafic provenant de 10 000 adresses IP situées en Russie ? Examinez les données d’adresse `ip` en les croisant avec le trafic des événements `fu` pour déterminer d’où provient le trafic à l’origine des échecs.

Les données de géolocalisation IP ne sont pas disponibles dans les journaux du tenant, sauf si vous pouvez les enrichir à partir d’une autre source.
