Voici quelques menaces courantes en cybersécurité :

* Attaques Bucket brigade
* Attaques par rejeu
* Attaques par falsification de requête intersites (CSRF ou XSRF)
* Attaques par force brute
* <Tooltip tip="Suspicious IP Throttling : forme de protection contre les attaques qui protège votre tenant contre les connexions suspectes visant un trop grand nombre de comptes à partir d’une seule adresse IP." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Suspicious+IP+throttling">Suspicious IP Throttling</Tooltip>
* Attaques de type credential stuffing
* Attaques de validation de listes

Auth0 offre des [fonctionnalités de protection contre les attaques](/docs/fr-CA/secure/attack-protection) qui peuvent aider à contrer certains types d’attaques.

<div id="bucket-brigade-attacks">
  ## Attaques Bucket brigade
</div>

Un type de menace est l’attaque Bucket brigade, parfois appelée attaque « de l’homme du milieu », où l’attaquant s’interpose entre deux parties. Chaque partie croit interagir par l’intermédiaire d’une connexion privée, alors que l’échange est en réalité contrôlé par un attaquant tiers. Pour que ce type d’attaque réussisse, l’attaquant doit établir une authentification mutuelle entre les deux parties.

Habituellement, les attaques Bucket brigade impliquent qu’un attaquant utilise un routeur WiFi pour intercepter les communications d’un utilisateur. L’utilisateur se connecte au routeur de l’attaquant, visite ensuite un site Web et ouvre une session avec ses identifiants confidentiels. L’attaquant enregistre alors les identifiants de connexion de cet utilisateur, qu’il peut ensuite utiliser pour se faire passer pour lui.

Certaines attaques Bucket brigade peuvent aussi modifier les données transmises entre l’application et le serveur. Des attaques peuvent provenir de nouveaux PC d’essai, de logiciels préinstallés, d’outils de mise à jour logicielle et d’autres vulnérabilités logicielles. Dans le pire des cas, cela pourrait entraîner l’exécution de code à distance, l’ajout d’une porte dérobée au système, l’installation de logiciels malveillants, etc. Cela pourrait aussi mener à la compromission d’autres ressources réseau.

Pour aider à se protéger contre ce type d’attaque, il est important d’utiliser un chiffrement fort et une authentification entre l’application et le serveur. Grâce au chiffrement, le serveur authentifie la requête de l’application en présentant un certificat numérique, et ce n’est qu’ensuite que la connexion peut être établie. Par exemple, HTTPS utilise la fonctionnalité Secure Sockets Layer (SSL) du navigateur pour masquer le trafic Web. Pour déchiffrer HTTPS, un attaquant devrait obtenir les clés utilisées pour chiffrer le trafic réseau.

Pour configurer TLS sur vos serveurs, nous vous suggérons d’utiliser les [recommandations OpSec de Mozilla](https://wiki.mozilla.org/Security/Server_Side_TLS), qui utilisent TLSv1.2. Mozilla fournit aussi un [générateur de configuration SSL](https://mozilla.github.io/server-side-tls/ssl-config-generator/) ; pour l’utiliser, choisissez d’abord la plateforme sur laquelle votre serveur fonctionne, puis la configuration la plus moderne selon ce que votre parc d’applications peut prendre en charge. Les configurations les plus modernes offrent des suites de chiffrement plus robustes afin d’aider à prévenir les attaques.

<div id="replay-attacks">
  ## Attaques par rejeu
</div>

Les attaques par rejeu (ou par relecture) permettent aux attaquants de :

* Obtenir l’accès à un réseau et à des informations qui n’auraient pas été facilement accessibles
* Exécuter une transaction en double

Il s’agit d’attaques contre le protocole de sécurité qui consistent à rejouer des transmissions de données provenant d’un autre expéditeur vers le système destinataire visé. Ces attaques trompent les participants en leur faisant croire que la transmission des données s’est déroulée correctement.

On peut éviter les attaques par rejeu en utilisant des jetons de session. Toutefois, si ces identifiants sont volés à partir du stockage local (comme lors d’une attaque XSS), il existe des moyens d’empêcher quelqu’un de conserver un jeton valide indéfiniment :

* Définir une courte durée d’expiration pour les jetons
* Prévoir un moyen d’ajouter à la DenyList les jetons qui ont été utilisés (et possiblement même les utilisateurs)
* Utiliser des mots de passe à usage unique

La spécification [JWT](/docs/fr-CA/secure/tokens/json-web-tokens) fournit le champ `jti` comme moyen de prévenir les attaques par rejeu. Bien que les jetons Auth0 ne renvoient pas actuellement le `jti`, vous pouvez ajouter des jetons à la DenyList à l’aide du `jti` pour empêcher qu’un jeton soit utilisé plus d’un certain nombre de fois. De cette façon, vous mettez en œuvre quelque chose de semblable à un <Tooltip tip="Nonce : Nombre arbitraire émis une seule fois dans un protocole d’authentification pour détecter et prévenir les attaques par rejeu." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=nonce">nonce</Tooltip> (considérez la signature du jeton comme le nonce). Si un jeton est volé ou s’il est utilisé plus souvent que le nombre de fois autorisé, il doit être ajouté à la DenyList. Cela empêche qu’un jeton valide soit utilisé de façon malveillante. Une fois le jeton expiré, l’attaquant ne pourra plus usurper l’identité de l’utilisateur.

Vous pouvez aussi éviter les attaques par rejeu en utilisant des mots de passe à usage unique. Avec Auth0, vous pouvez utiliser l’authentification <Tooltip tip="Passwordless : Forme d’authentification qui ne repose pas sur un mot de passe comme premier facteur." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Passwordless">Passwordless</Tooltip>, qui repose sur des codes à usage unique et des liens envoyés par courriel plutôt que sur des mots de passe traditionnels. Auth0 offre aussi l’<Tooltip tip="Authentification multifacteur (MFA) : Processus d’authentification de l’utilisateur qui utilise un facteur en plus du nom d’utilisateur et du mot de passe, comme un code envoyé par SMS." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=multi-factor+authentication">authentification multifacteur</Tooltip> (MFA), qui utilise des mots de passe à usage unique comme deuxième facteur d’authentification et peut les envoyer par notifications push et par SMS.

<div id="cross-site-request-forgery">
  ## Falsification de requête intersites
</div>

Une attaque de falsification de requête intersites (CSRF ou XSRF) se produit lorsqu’un programme malveillant amène le navigateur Web d’un utilisateur à exécuter une action indésirable sur un site de confiance où l’utilisateur est déjà authentifié. Ce type d’attaque cible précisément les requêtes qui modifient l’état afin de déclencher une action, plutôt que d’obtenir des données utilisateur, puisque l’attaquant n’a aucun moyen de voir la réponse à la requête falsifiée.

Une façon de vérifier les requêtes envoyées consiste à utiliser le [paramètre state](/docs/fr-CA/secure/attack-protection/state-parameters) du protocole <Tooltip tip="OAuth 2.0 : framework d’autorisation qui définit les protocoles et les processus d’autorisation." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=OAuth+2.0">OAuth 2.0</Tooltip> pour valider la réponse.
