**Publié**: 10 janvier 2019

<div id="overview">
  ## Aperçu
</div>

Ce bulletin de sécurité présente plusieurs scénarios que les clients devraient éviter dans le code de règle personnalisé, puisqu’ils peuvent créer des vulnérabilités dans le flux d’authentification. Vous trouverez ci-dessous une description de chaque problème et, lorsque possible, une autre façon d’écrire la même règle.

* Le code de règle personnalisé qui utilise une logique conditionnelle pour imposer l’<Tooltip tip="Authentification multifacteur (MFA) : processus d’authentification de l’utilisateur qui exige un facteur supplémentaire en plus du nom d’utilisateur et du mot de passe, par exemple un code envoyé par SMS." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=multi-factor+authentication">authentification multifacteur</Tooltip> (MFA) peut permettre de contourner complètement la MFA, surtout lors de l’utilisation de l’authentification silencieuse.
* Le code de règle personnalisé qui met en œuvre des contrôles d’autorisation fondés sur une sous-chaîne précise sans logique appropriée peut entraîner une élévation des privilèges.
* Le code de règle personnalisé qui envoie des renseignements de diagnostic à des services tiers, plutôt que d’utiliser les fonctions de débogage intégrées d’Auth0, peut entraîner la divulgation de renseignements sensibles.
* Le code de règle personnalisé qui déclenche des services payants peut permettre à un adversaire de générer des frais non désirés.
* Le code de règle personnalisé qui accorde une autorisation en fonction d’adresses courriel non vérifiées peut permettre à des adversaires d’obtenir ces privilèges au moyen d’un type de connexion secondaire.
* Le code de règle personnalisé contenant des secrets codés en dur, comme des clés API, plutôt que d’utiliser l’objet de configuration global, augmente le risque d’exposer ces secrets.

<div id="improper-mfa-rules">
  ## Règles MFA mal configurées
</div>

<div id="silent-authentication">
  ### Authentification silencieuse
</div>

Dans une application monopage (SPA), l’authentification silencieuse permet d’émettre de nouveaux <Tooltip tip="Jeton d’accès : justificatif d’autorisation, sous la forme d’une chaîne opaque ou d’un JWT, utilisé pour accéder à une API." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=access+tokens">jetons d’accès</Tooltip> sans interaction de l’utilisateur, tant que la session en cours est valide auprès du <Tooltip tip="Serveur d’autorisation : serveur centralisé qui aide à définir les limites de l’accès d’un utilisateur. Par exemple, votre serveur d’autorisation peut contrôler les données, les tâches et les fonctionnalités offertes à un utilisateur." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=authorization+server">serveur d’autorisation</Tooltip>.

Auth0 offre l’authentification silencieuse au moyen d’un paramètre spécial ajouté au point de terminaison d’autorisation, comme suit :
`/authorize?prompt=none`

Toutefois, si la MFA est ajoutée au processus d’authentification silencieuse, une interaction de l’utilisateur devient nécessaire.

Il ne faut pas utiliser de logique conditionnelle fondée sur l’authentification silencieuse pour contourner la MFA.

Des Rules comme celle-ci permettent de contourner complètement la MFA et ne doivent pas être utilisées :

```javascript lines
function (user, context, callback) {
  if (context.request.query && context.request.query.prompt === 'none') {
  // ignorer l'AMF pour les requêtes de jeton silencieuses
  return callback(null, user, context);
  }
...
}
```

<div id="silent-authentication-with-redirection-to-custom-mfa-provider">
  ### Authentification silencieuse avec redirection vers un fournisseur MFA personnalisé
</div>

Les Rules qui déterminent s’il faut rediriger l’utilisateur vers une authentification multifacteur personnalisée pendant l’authentification silencieuse peuvent, dans certaines circonstances inhabituelles, permettre de contourner la MFA. Par exemple, évitez ce qui suit :

```javascript lines
function (user, context, callback) {
  if (context.request.query && context.request.query.prompt === 'none') {
  //rediriger vers l'AMF personnalisé
  context.redirect = {
    url: "https://example.com/"
  };
...
}
```

<div id="improper-verification">
  ### Vérification inadéquate
</div>

L’utilisation de règles pour contourner l’authentification multifacteur dans certaines conditions peut entraîner un contournement de la MFA lorsque les vérifications en place ne permettent pas de confirmer de façon fiable l’état souhaité.

Dans ces cas, le fond du problème est que les règles ne peuvent jamais déterminer si la MFA a été exécutée avec succès, puisque la MFA a lieu après l’exécution d’une règle. Les règles configurées de cette manière (voir les exemples ci-dessous) mettent à jour prématurément l’enregistrement utilisateur en supposant à tort que la MFA sera exécutée avec succès, tout en s’appuyant sur ce même enregistrement utilisateur pour ignorer la MFA dans certaines conditions.

Les exemples ci-dessous illustrent des vérifications inadéquates pouvant entraîner un contournement complet de l’authentification multifacteur.

<div id="device-fingerprint">
  #### Empreinte de l’appareil
</div>

```javascript lines
function (user, context, callback) {

  var crypto = require('crypto');

  var deviceFingerPrint = getDeviceFingerPrint();
  user.app_metadata = user.app_metadata || {};

  // Vérification inadéquate
  if (user.app_metadata.lastLoginDeviceFingerPrint !== deviceFingerPrint) {
    user.app_metadata.lastLoginDeviceFingerPrint = deviceFingerPrint;
    context.multi-factor = {
      ...
    };
    ...
  }
  function getDeviceFingerPrint() {
    var shasum = crypto.createHash('sha1');
    shasum.update(context.request.userAgent);
    shasum.update(context.request.ip);
    return shasum.digest('hex');
  }
}
```

<div id="country-code">
  #### Indicatif de pays
</div>

```javascript lines
function (user, context, callback) {
  user.app_metadata = user.app_metadata || {};

  // Vérification inadéquate
  if (user.app_metadata.last_location !== context.request.geoip.country_code) {
    user.app_metadata.last_location = context.request.geoip.country_code;
    context.multi-factor = {
    ...
    };
  }
```

<div id="am-i-affected">
  ### Suis-je concerné ?
</div>

Si vous utilisez l’une des logiques de règle présentées précédemment, un adversaire ayant réussi à s’authentifier avec le premier facteur d’authentification pourrait contourner la MFA dans votre application.

<div id="mitigation-steps">
  ### Étapes d’atténuation
</div>

**Si vous êtes concerné par le scénario d’authentification silencieuse**, supprimez la logique conditionnelle basée sur `prompt === 'none'`. Cela déclenchera l’authentification multifacteur à chaque appel d’authentification silencieuse afin de vérifier l’état de la session.

**Si vous êtes concerné par le scénario d’authentification silencieuse avec redirection**, supprimez la logique conditionnelle basée sur `prompt === 'none'` et utilisez un fournisseur d’authentification multifacteur pris en charge par Auth0.

Pour éviter d’inviter l’utilisateur à effectuer une MFA trop souvent, vous pouvez définir le paramètre `allowRememberBrowser` à true, ce qui permettra aux utilisateurs finaux de cocher une case pour n’être invités à effectuer l’authentification multifacteur que tous les 30 jours. Par exemple :

```js lines
context.multi-factor = {
    provider: 'guardian',
    allowRememberBrowser: true
  };
```

Il peut être utile d’aviser vos utilisateurs finaux de cocher la case afin d’éviter qu’on le leur demande trop souvent.

**Si vous êtes concerné par des règles qui utilisent une vérification inadéquate pour contourner la MFA**, assurez-vous d’utiliser des méthodes de vérification fiables, d’éliminer complètement les exceptions de MFA ou d’utiliser l’option de configuration `allowRememberBrowser` décrite ci-dessus.

### Cette mise à jour aura-t-elle une incidence sur mes utilisateurs?

Selon la façon dont vous ajustez vos règles ou votre configuration, cette mise à jour pourrait avoir une incidence sur vos utilisateurs en leur demandant d’effectuer la MFA plus souvent qu’à l’habitude.

<div id="improper-substring-verification">
  ## Vérification inadéquate d’une sous-chaîne
</div>

Si la logique de votre règle exige un contrôle d’accès fondé sur la correspondance exacte d’une chaîne donnée, comme un domaine de courriel, mais que vous vérifiez seulement la présence d’une sous-chaîne au lieu d’une correspondance exacte, il se peut que cette logique ne fonctionne pas comme prévu. Par exemple :

`if( _.findIndex(connection.options.domain_aliases,function(d){ return user.email.indexOf(d) >= 0;`

La logique ci-dessus retournerait `true` pour des courriels comme ceux-ci :

* `user.domain.com@not-domain.com`
* `"user@domain.com"@not-domain.com` (guillemets inclus)

<div id="am-i-affected">
  ### Suis-je concerné ?
</div>

Seuls les clients qui utilisent une logique conditionnelle dans les règles, comme décrit ci-dessus, sont concernés.

<div id="mitigation-steps">
  ### Étapes d’atténuation
</div>

Utilisez une logique du type :

`const emailSplit = user.email.split('@'); const userEmailDomain = emailSplit[emailSplit.length - 1].toLowerCase();`

Pour en savoir plus, consultez le modèle de règle [Check Domains Against Connection Aliases](https://github.com/auth0/rules/blob/master/src/rules/check-domains-against-connection-aliases.js). Sinon, dans la section **Rules** du <Tooltip tip="Auth0 Dashboard : le principal produit d’Auth0 pour configurer vos services." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip>, consultez le modèle de règle nommé **Check if user email domain matches configured domain**.

### Cette mise à jour aura-t-elle une incidence sur mes utilisateurs ?

Possiblement. La modification recommandée aux règles dont la logique repose sur des vérifications de sous-chaînes peut avoir une incidence sur les utilisateurs finaux, mais vous êtes le mieux placé pour juger de l’intention de vos règles et déterminer si les changements recommandés risquent d’avoir une incidence sur vos utilisateurs finaux.

<div id="debugging-using-external-services">
  ## Débogage à l’aide de services externes
</div>

Si votre logique de règle envoie l’objet de contexte d’Auth0 à un service externe, vous exposez des éléments comme `id_token` ou `access_token` associés à la requête. Par exemple :

```js lines
request.post({
    url: 'http://requestbin.fullcontact.com/YourBinUrl',
    json: {
      user: user,
      context: context,
    },
    timeout: 15000
  }, function(err, response, body){
    if (err) return callback(err);
    return callback(null, user, context);
  });
```

### Suis-je concerné?

Seuls les clients qui utilisent une logique de règle, comme décrit ci-dessus, sont concernés.

<div id="mitigation-steps">
  ### Étapes d’atténuation
</div>

Vous devriez modifier votre règle pour qu’elle n’envoie plus l’intégralité de l’objet de contexte à requestbin, car cela pourrait exposer tout `id_token` ou `access_token` associé à chaque requête. Envoyez plutôt un sous-ensemble d’attributs de l’objet de contexte moins sensibles.

Veuillez consulter le [modèle de règle Requestbin](https://github.com/auth0/rules/blob/master/src/rules/requestbin.js) pour en savoir plus. Sinon, dans la section **Rules** de l’Auth0 Dashboard, affichez le modèle de règle nommé **Dump rule variables to RequestBin**.

Auth0 offre également des méthodes intégrées pour déboguer les règles sans envoyer d’information à des services externes.

### Cette mise à jour aura-t-elle une incidence sur mes utilisateurs?

Non. La règle touchée était généralement utilisée à des fins de débogage. Cette modification ne devrait pas avoir d’incidence sur les utilisateurs finaux.

<div id="rules-using-a-paid-service">
  ## Règles utilisant un service payant
</div>

Si la logique de vos règles fait appel à un service payant, comme l’envoi de messages SMS via Twillio dans le cadre du processus d’authentification, vous pourriez être exposé à une hausse des coûts, puisqu’un adversaire disposant d’un nom d’utilisateur et mot de passe valides pourrait déclencher un appel au service et entraîner des frais sans avoir à mener à terme l’ensemble du processus d’authentification défini par les règles. Par exemple :

```javascript lines
//Envoie un SMS à l'utilisateur via Twilio
  function notifyUser(done) {
    const request = require('request');
    const twilioAccount = '{yourTwilioAccount}';
    const twilioAuthToken = '{yourTwilioAuthToken}';

    request.post({
      url: 'https://api.twilio.com/2010-04-01/Accounts/' + twilioAccount + '/Messages.json',
      auth: {
        'user': twilioAccount,
        'pass': twilioAuthToken,
      },
      form: {
        body: 'You\'ve logged in from a different device or location.',
        to: user.phone,
        from: '+18668888888'
      }
    }, function (error, response, body) {
      if (error) return done(error);
      if (response.statusCode !== 201) return done(new Error(response.statusCode));
      return done(null);
    });
  }
```

<div id="am-i-affected">
  ### Suis-je concerné ?
</div>

Seuls les clients qui ont une logique de règle, telle que décrite ci-dessus, et dont cette logique de règle peut être déclenchée par n’importe quel utilisateur non digne de confiance sont concernés.

<div id="mitigation-steps">
  ### Étapes d’atténuation
</div>

Pour atténuer ce risque, envisagez de prendre une ou plusieurs des mesures suivantes :

* Désactivez les inscriptions publiques, si elles ne sont pas nécessaires, afin de réduire le nombre d’utilisateurs pouvant s’inscrire et déclencher des appels vers des services payants.
* Réduisez le risque de vol d’identifiants afin d’éviter la prise de contrôle de comptes par des pirates qui pourraient utiliser des comptes compromis pour déclencher des appels vers des services payants.
* Assurez-vous que vos utilisateurs ont des mots de passe robustes lorsqu’ils utilisent des connexions de base de données.
* Assurez-vous que vos utilisateurs utilisent l’authentification multifacteur.
* Assurez-vous que la règle ne se déclenche que pour un sous-ensemble autorisé d’utilisateurs, ou dans d’autres conditions appropriées. Par exemple, vous pourriez ajouter une logique qui vérifie si un utilisateur possède un domaine de courriel, un rôle/groupe ou un niveau d’abonnement particulier avant de déclencher l’appel au service payant.

### Cette mise à jour aura-t-elle une incidence sur mes utilisateurs?

L’incidence sur les utilisateurs finaux dépend de l’option que vous choisissez parmi celles décrites ci-dessus pour atténuer le problème.

<div id="authorization-based-on-unverified-email-address">
  ## Autorisation fondée sur une adresse courriel non vérifiée
</div>

Si votre logique de règle exige un contrôle d&#39;accès en fonction d&#39;une adresse courriel ou d&#39;un domaine de courriel précis, mais que vous ne vérifiez pas si cet utilisateur a bien vérifié son adresse courriel, un acteur malveillant pourrait obtenir des privilèges supplémentaires en utilisant un deuxième type de connexion. Par exemple :

```javascript lines
var userHasAccess = allowList.some(function (email) {
    return email === user.email;
  });
```

La logique ci-dessus renverrait `true` si un adversaire pouvait créer un compte à l’aide d’un type de connexion différent (par exemple, une connexion sociale) avec une adresse courriel figurant dans la liste des adresses autorisées. Cela s’explique par le fait qu’une même adresse courriel peut exister dans différents types de connexion.

<div id="am-i-affected">
  ### Suis-je concerné ?
</div>

Seuls les clients qui utilisent une logique conditionnelle dans les rules, comme décrit ci-dessus, et qui prennent en charge plusieurs types de connexion sont concernés.

<div id="mitigation-steps">
  ### Étapes d’atténuation
</div>

Lorsque vous accordez une autorisation en fonction d’adresses courriel, commencez toujours votre règle par la logique suivante :

```javascript lines
function (user, context, callback) {
  // L'accès ne doit être accordé qu'aux utilisateurs vérifiés.
  if (!user.email || !user.email_verified) {
    return callback(new UnauthorizedError('Access denied.'));
  }
```

Parfois, même une adresse courriel vérifiée peut ne pas suffire comme vérification pour l’autorisation que vous souhaitez accorder. Pour en savoir plus sur ce type de cas, veuillez consulter la section sur l’utilisation de l’adresse courriel vérifiée.

### Cette mise à jour aura-t-elle une incidence sur mes utilisateurs?

Elle n’aura d’effet sur les utilisateurs existants que s’ils n’ont pas vérifié leur adresse courriel.

<div id="plaintext-secrets-within-rule-code">
  ## Secrets en clair dans le code de la règle
</div>

Si vous incluez des renseignements sensibles, comme des clés API, dans le code de la règle, vous devriez plutôt configurer ces valeurs dans les paramètres de la règle. Par exemple :

`const myApiKey = 'abcdefghijklmnopqrstuvwxyz';`

De telles valeurs sensibles, puisqu’elles se trouvent dans le code de la règle, resteront non chiffrées dans nos systèmes et risqueront donc d’être exposées.

<div id="am-i-affected">
  ### Suis-je concerné ?
</div>

Si vos règles contiennent des valeurs sensibles directement dans le code même de la règle, vous êtes concerné.

<div id="mitigation-steps">
  ### Étapes d’atténuation
</div>

Placez les valeurs sensibles dans la zone **Settings** de la section principale **Rules** de l’Auth0 Dashboard, puis récupérez-les dans votre règle à l’aide du code suivant :

`const myApiKey = configuration.myApiKey;`

Cela garantit que toutes les valeurs sensibles sont chiffrées dans les systèmes d’Auth0, ce qui réduit le risque d’exposition.

<div id="will-this-update-impact-my-users">
  ### Cette mise à jour aura-t-elle des répercussions sur vos utilisateurs?
</div>

Cela n’aura aucune répercussion sur vos utilisateurs.
