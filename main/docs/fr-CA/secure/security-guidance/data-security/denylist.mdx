S&#39;il y a des champs utilisateur qui ne devraient pas être stockés dans les bases de données Auth0 pour des raisons de confidentialité, vous pouvez les ajouter à la liste de rejet. Pour ajouter des attributs à la liste de rejet, envoyez une requête `PATCH` au point de terminaison Update Connection de la <Tooltip tip="Management API : un produit qui permet aux clients d'effectuer des tâches administratives." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Management+API">Management API</Tooltip>.

1. Obtenez un jeton d&#39;accès valide pour accéder au [point de terminaison](https://auth0.com/docs/api/management/v2#!/Connections/patch_connections_by_id) `/patch_connections_by_id`. Le jeton doit inclure la portée `update:connections`. Consultez [Management API Access Tokens](/docs/fr-CA/secure/tokens/access-tokens/management-api-access-tokens) pour en savoir plus.
2. À l&#39;aide du jeton d&#39;accès et de la liste des attributs à rejeter, appelez l&#39;API. Voici un exemple de requête HTTP qui rejette deux attributs : l&#39;origine ethnique et le genre. Gardez à l&#39;esprit que vous devez récupérer l&#39;objet options et envoyer l&#39;objet au complet dans votre requête PATCH, car aucune « fusion » n&#39;est effectuée lorsque vous ne mettez à jour qu&#39;une ou deux valeurs.

Où :

1. `{yourConnectionId}` est l’[ID de la connexion](/docs/fr-CA/authenticate/identity-providers/locate-the-connection-id) pour laquelle ces attributs seront rejetés.
   2. `{yourToken}` est le jeton d’accès que vous avez reçu à l’étape précédente.
   3. L’objet `options.non_persistent_attrs` contient un tableau des attributs qui seront rejetés. Si la revendication que vous souhaitez rejeter est envoyée par un fournisseur d’identité en amont (IdP), vous devez la définir exactement telle qu’elle est envoyée par l’IdP en amont. Par exemple, pour une revendication reçue sous la forme `https://acme.com/temporary_idtoken`, l’objet `non_persistent_attrs` de l’exemple ci-dessus se lirait ainsi :

   ```json wrap lines
   {"non_persistent_attrs": ["ethnicity", "gender", "https://acme.com/temporary_idtoken"]}
   ```

<div id="limitations">
  ## Limitations
</div>

* Seuls les [champs racine](/docs/fr-CA/manage-users/user-accounts/user-profiles/user-profile-structure) (comme `user.name` ou `user.phone_number`) peuvent être rejetés.

  * Si `user.name` ou `user.nickname` sont rejetés, ils ne seront pas inclus dans les jetons.
  * Si `user.email` est rejeté, la valeur ne peut pas être mappée à un custom claim. Par exemple, dans une rule, `context.idToken[namespace + 'work_email'] = user.email` ne fonctionnerait pas.
* Lorsque vous rejetez des attributs, ils restent accessibles via les Rules et les jetons sortants. Toutefois, si l&#39;un des cas suivants s&#39;applique, les attributs de la liste de rejet ne seront **pas** inclus dans les jetons :

  * Vous avez activé l&#39;authentification multifacteur (MFA)
  * Vous avez effectué une redirection via les Rules
  * Votre appli utilise Delegation (et vous n&#39;avez pas défini `scope = passthrough`)
  * Votre appli utilise l&#39;impersonation
  * Vous avez activé le paramètre **Use Auth0 instead of the IdP to do Single Sign-On** (tenants Legacy seulement)
* Pour les connexions SAMLP, si vous activez le mode Debug, vos logs contiendront des renseignements sur les attributs de la liste de rejet

Si l&#39;une ou l&#39;autre de ces limitations est inacceptable, vous pouvez écrire une [rule](/docs/fr-CA/customize/rules) pour chiffrer les données et faire en sorte qu&#39;elles soient enregistrées dans l&#39;objet `user.app_metadata`.
