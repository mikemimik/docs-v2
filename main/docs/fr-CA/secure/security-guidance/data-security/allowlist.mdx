Si du code personnalisé exécuté dans Auth0 appelle un service au sein de votre réseau, ou si vous avez configuré dans Auth0 un fournisseur SMTP local, vous devrez peut-être configurer votre pare-feu pour autoriser le trafic entrant en provenance d’Auth0.

Parmi les fonctionnalités pouvant nécessiter que vous autorisiez le trafic entrant en provenance d’Auth0, on retrouve :

* [Actions](/docs/fr-CA/customize/actions)
* [Rules](/docs/fr-CA/customize/rules)
* [Hooks](/docs/fr-CA/customize/hooks)
* [Custom Database Action Scripts](/docs/fr-CA/authenticate/database-connections/custom-db/templates)
* [Log Streams](/docs/fr-CA/customize/log-streams)
* [Fournisseurs de courriel SMTP externes](/docs/fr-CA/customize/email/smtp-email-providers)

<div id="outbound-calls">
  ## Appels sortants
</div>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Les adresses IP peuvent changer. En cas de changement d’adresse IP, Auth0 envoie des avis plusieurs mois avant l’entrée en vigueur des changements. Les listes sont à jour au moment de la rédaction.
</Callout>

Lorsque Auth0 effectue des appels sortants, les adresses IP sont statiques. Auth0 fait correspondre les adresses IP internes à l’une des options affichées lorsqu’il communique au moyen de la NAT.

<div id="public-cloud">
  ### Public Cloud
</div>

Pour les tenants Public Cloud, les adresses IP que vous devez autoriser dans votre pare-feu varient selon la région du tenant.

<div id="machine-readable-list">
  #### Liste exploitable par machine
</div>

Vous pouvez accéder à une liste exploitable par machine des plages d’adresses IP CIDR du CDN Auth0 à l’adresse suivante : https://cdn.auth0.com/ip-ranges.json.

<div id="schema">
  #### Schéma
</div>

La liste exploitable par machine contient les propriétés suivantes :

| **Propriété**     | **Type de données** | **Description**                                                                                                                                                                                                                                                                                                                                                        |
| :---------------- | :------------------ | :--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `last_updated_at` | timestamp (ISO8601) | Date et heure de la mise à jour la plus récente de la liste.                                                                                                                                                                                                                                                                                                           |
| `regions`         | objet               | Ensemble des [régions de tenant Auth0](/docs/fr-CA/get-started/auth0-overview/create-tenants#region-locality-and-sub-locality). Chaque objet de région de tenant contient la propriété suivante : * `ipv4_cidrs`: tableau de chaînes. Liste des plages CIDR IPv4 pour la région donnée.                                                                                      |
| `changelog`       | tableau d’objets    | Historique de tous les ajouts et retraits de plages CIDR IP. Chaque objet contient les propriétés suivantes : * `date`: chaîne. Horodatage au format YYYY-MM-DD. * `region`: chaîne. Région Auth0 concernée. * `action`: chaîne. Action effectuée. Les valeurs possibles sont : `add`, `remove`. * `ipv4_cidrs`: tableau de chaînes. Liste des plages CIDR concernées. |

<div id="detecting-changes">
  #### Détection des changements
</div>

Vous pouvez détecter par programmation les changements apportés à la liste exploitable par machine en vous basant sur la propriété last&#95;updated&#95;at ou sur la réponse d’Auth0 à une requête HTTP conditionnelle. Auth0 met à jour cette liste avant toute incidence fonctionnelle sur les tenants existants. Planifiez une vérification hebdomadaire automatisée pour surveiller les changements et maintenir des configurations de pare-feu adéquates.

Si vous envoyez une requête HTTP conditionnelle, vous pouvez utiliser l’en-tête If-None-Match ou If-Modified-Since. Si la liste a été modifiée, Auth0 renvoie une réponse HTTP 200 (OK) avec la liste mise à jour dans le corps. Sinon, Auth0 renvoie une réponse 304 (Not Modified) sans corps.
Pour en savoir plus sur les requêtes HTTP conditionnelles, consultez [RFC 9110: HTTP Semantics on RFC Editor](https://www.rfc-editor.org/rfc/rfc9110.html#name-conditional-requests).

<div id="example-of-conditional-http-request-if-none-match-header">
  #### Exemple de requête HTTP conditionnelle (en-tête If-None-Match)
</div>

```curl
# La première requête.
$ curl https://cdn.auth0.com/ip-ranges.json
HTTP/2 200
last-modified: Tue, 22 Apr 2025 07:50:08 GMT
etag: "1652cbe64a22099d62424fbec1022da5"
...

# Le contenu n'a pas changé, la réponse est `HTTP 304`
$ curl https://cdn.auth0.com/ip-ranges.json \
       -H "If-None-Match: 1652cbe64a22099d62424fbec1022da5" 
HTTP/2 304
last-modified: Tue, 22 Apr 2025 07:50:08 GMT
etag: "1652cbe64a22099d62424fbec1022da5"
...

# Le contenu a changé, la réponse est `HTTP 200` avec un nouvel `etag` :
$ curl https://cdn.auth0.com/ip-ranges.json \
       -H "If-None-Match: 1652cbe64a22099d62424fbec1022da5" 
HTTP/2 200
last-modified: Mon, 07 Jul 2025 01:04:18 GMT
etag: "8e58c97d39a67706c644cbe3b5c682d3"
...
```

<div id="private-cloud">
  ### Private Cloud
</div>

Pour les tenants Private Cloud, les adresses IP que vous devez autoriser dans votre pare-feu sont propres à l’environnement de votre tenant. Auth0 peut recevoir les adresses IP privées de votre tenant si vous activez des fonctionnalités comme les journaux du tenant, <Tooltip tip="Suspicious IP Throttling : forme de protection contre les attaques qui protège votre tenant contre les connexions suspectes visant trop de comptes à partir d’une seule adresse IP." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Suspicious+IP+throttling">Suspicious IP throttling</Tooltip>, Custom Databases et les Actions qui en dépendent.

Ces adresses IP sont appelées **adresses IP de sortie primaires** et figurent dans les données de configuration de l’environnement accessibles dans le [Auth0 Support Center](https://support.auth0.com/tenants/private).

<div id="inbound-calls">
  ## Appels entrants
</div>

Les adresses IP associées aux appels entrants vers Auth0 peuvent varier en l’absence d’adresses IP fixes sur les équilibreurs de charge. Dans ce cas, les règles de pare-feu doivent être basées sur le nom du service (par exemple : `<YOUR_TENANT>.<YOUR_REGION>.auth0.com`).

Si votre abonnement Auth0 vous permet de configurer un <Tooltip tip="Domaine personnalisé : domaine tiers portant un nom spécialisé ou personnalisé." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=custom+domain">domaine personnalisé</Tooltip> autogéré, vous pouvez le configurer avec une adresse IP statique. Les domaines personnalisés autogérés vous donnent le contrôle sur le point d’entrée réseau et vous permettent de vous assurer que l’adresse IP est fixe. Pour en savoir plus sur les forfaits, consultez [Auth0 Pricing](https://auth0.com/pricing).
