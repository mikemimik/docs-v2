***

Le Tenant Access Control List (ACL) gère le trafic vers les ressources de votre tenant en fonction des règles que vous définissez.

L’utilisation du Tenant ACL peut vous aider à préserver vos limites de débit contre les menaces potentielles (comme les attaques par déni de service) et à faire en sorte que seules les personnes autorisées puissent accéder à vos applications.

<div id="overview">
  ## Aperçu
</div>

Chaque Tenant ACL se compose d’un ensemble de règles. Lorsqu’une requête est reçue par votre tenant ou par l’<Tooltip tip="Management API : un produit permettant aux clients d’effectuer des tâches administratives." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Management+API">Management API</Tooltip>, ce sont les règles de votre Tenant ACL qui déterminent la réponse.

Les options de configuration granulaires du Tenant ACL vous permettent d’appliquer des règles à une variété de signaux et de portées. Par exemple :

* Vous pouvez utiliser la portée [`hostnames`](./tenant-access-control-list/reference#param-hostnames) pour désactiver ou restreindre l’accès à votre domaine canonique, tout en autorisant le trafic via des domaines personnalisés.

* Si vous adoptez les Model Context Protocols (MCP) dans votre tenant, vous pouvez utiliser la portée [Dynamic Client Registration (`dcr`)](./tenant-access-control-list/reference#param-dcr) pour réduire des risques comme l’enregistrement non autorisé d’applications ou les tentatives d’hameçonnage au moyen de noms d’application trompeurs.

À des fins de surveillance et d’audit, un [événement du journal du tenant](./tenant-access-control-list/tenant-log-reference) (`acls_summary`) est créé toutes les 10 minutes pour chaque règle de Tenant ACL, avec des détails sur la façon dont cette règle affecte le trafic. Vous pouvez activer le mode de surveillance pour les règles afin de tester et de consigner leur comportement sans appliquer réellement la règle au trafic.
