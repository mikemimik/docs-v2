La Web Authentication API (aussi appelée WebAuthn) est une [spécification](https://www.w3.org/TR/webauthn/) rédigée par le W3C et FIDO, avec la participation de Google, Mozilla, Microsoft, Yubico et d’autres. Cette API permet d’authentifier les utilisateurs au moyen de la cryptographie à clé publique.

<Warning>
  Le RP ID de WebAuthn est actuellement limité à une configuration à domaine unique et n’est pas compatible avec les clients qui utilisent plusieurs domaines personnalisés.
</Warning>

WebAuthn est la méthode d’authentification la plus sécuritaire et la plus conviviale sur le Web. Voici quelques-unes des principales raisons :

* Elle réduit au minimum les frictions à la connexion. Un geste simple et familier suffit pour authentifier les utilisateurs.
* C’est la seule méthode d’authentification Web résistante à l’hameçonnage.
* Elle repose sur des normes et est prise en charge par les navigateurs et les systèmes d’exploitation.

WebAuthn permet aux utilisateurs de s’authentifier à l’aide de deux types d’authentificateurs :

* Les **authentificateurs itinérants** sont amovibles et multiplateformes, comme une YubiKey, et peuvent être utilisés sur plusieurs appareils. Pour vous authentifier avec un authentificateur itinérant, vous devez le connecter à l’appareil (par USB, NFC ou Bluetooth), fournir une preuve de présence (en le touchant, par exemple) et, au besoin, effectuer la vérification de l’utilisateur, par exemple en entrant un NIP.
* Les **authentificateurs de plateforme** sont intégrés à un appareil et ne fonctionnent que sur celui-ci. Par exemple, la Touch Bar des MacBook, Windows Hello, Touch ID/Face ID sur iOS, ainsi que la reconnaissance faciale ou par empreinte digitale sur Android. Les données biométriques sont stockées sur l’appareil et ne sont jamais envoyées au serveur. Lorsqu’il n’est pas possible d’utiliser la biométrie, d’autres méthodes d’authentification sont généralement offertes. Par exemple, si vous portez un masque, vous pouvez entrer votre code d’accès au lieu d’utiliser Face ID.

<div id="webauthn-and-web-origins">
  ## WebAuthn et les origines Web
</div>

WebAuthn fonctionne en générant une paire de clés privée/publique pour chaque [origine Web](https://en.wikipedia.org/wiki/Same-origin_policy), enregistrée dans l’appareil ou la clé de sécurité.

Comme la paire de clés est liée au domaine, les utilisateurs sont protégés contre les attaques d’hameçonnage. Si un attaquant les incite à utiliser WebAuthn dans un autre domaine, l’authentificateur WebAuthn n’aura pas de paire de clés pour ce domaine et l’authentification échouera. L’attaquant n’obtiendra aucune donnée permettant d’identifier l’utilisateur.

  ## Authentificateurs itinérants

Pour savoir comment activer les authentificateurs itinérants dans Auth0, consultez la documentation [WebAuthn with Security Keys](/docs/fr-CA/secure/multi-factor-authentication/fido-authentication-with-webauthn/configure-webauthn-security-keys-for-mfa).

Le flux d’inscription par défaut des clés de sécurité dans Android illustré ci-dessous comprend les étapes suivantes :

* Les utilisateurs s’authentifient avec leur nom d’utilisateur et leur mot de passe.
* Ils sont invités à sélectionner la méthode d’authentification qu’ils veulent inscrire.
* S’ils choisissent les clés de sécurité, une liste d’instructions s’affiche.
* Ils sont ensuite invités à utiliser leur clé de sécurité.
* L’appareil affiche son interface utilisateur native pour terminer la vérification par clé de sécurité.
* Les utilisateurs peuvent nommer la clé pour l’identifier plus tard, au cas où ils en inscriraient plusieurs.

<div id="platform-authenticators">
  ## Authentificateurs de plateforme
</div>

Pour savoir comment activer les authentificateurs de plateforme, consultez la documentation [Webauthn with biométrie de l’appareil](/docs/fr-CA/secure/multi-factor-authentication/fido-authentication-with-webauthn/configure-webauthn-device-biometrics-for-mfa).

Comme les authentificateurs de plateforme ne peuvent être utilisés que sur un seul appareil, **ils ne devraient pas être les seuls facteurs enregistrés par les utilisateurs.** Pour éviter que les utilisateurs se retrouvent exclus de leur compte, Auth0 les invitera à enregistrer des authentificateurs de plateforme après qu’ils se seront authentifiés avec succès à l’aide d’une autre méthode d’authentification.

Auth0 essaiera **d’enregistrer progressivement** les appareils de tous les utilisateurs. Les utilisateurs seront invités à enregistrer les authentificateurs de plateforme sur chaque appareil qu’ils utilisent.

Le flux d’inscription par défaut pour Face ID sur iOS est illustré ci-dessous et comprend les étapes suivantes :

* Les utilisateurs s’authentifient avec leur nom d’utilisateur et leur mot de passe.
* Ils enregistrent une autre méthode d’authentification <Tooltip tip="Authentification multifacteur (MFA) : processus d’authentification de l’utilisateur qui utilise un facteur en plus du nom d’utilisateur et du mot de passe, comme un code par SMS." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=MFA">MFA</Tooltip>, comme [SMS](/docs/fr-CA/secure/multi-factor-authentication/multi-factor-authentication-factors/configure-sms-voice-notifications-mfa), [Push](/docs/fr-CA/secure/multi-factor-authentication/multi-factor-authentication-factors/configure-push-notifications-for-mfa) ou [OTP temporel](/docs/fr-CA/secure/multi-factor-authentication/multi-factor-authentication-factors/configure-otp-notifications-for-mfa).
* Ils nomment leur appareil afin de pouvoir l’identifier plus tard.

La prochaine fois qu’ils se connecteront à partir de cet appareil, les utilisateurs entreront leur nom d’utilisateur et leur mot de passe, puis termineront l’authentification MFA à l’aide de l’authentificateur biométrique de leur appareil.

<div id="webauthn-as-multi-factor-authentication">
  ## WebAuthn comme authentification multifacteur
</div>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Lorsque les utilisateurs s’authentifient avec WebAuthn, ils utilisent un élément **qu’ils possèdent** comme facteur d’authentification : une clé de sécurité ou un appareil.

  Les clés de sécurité et la biométrie de l’appareil prennent toutes deux en charge la **vérification de l’utilisateur**, qui oblige les utilisateurs à fournir un élément **qu’ils connaissent** (un NIP ou un code d’accès) et un élément **qu’ils sont** (comme des caractéristiques biométriques).

  Avec la biométrie de l’appareil, la vérification de l’utilisateur est toujours effectuée. Pour effectuer cette vérification avec des clés de sécurité, vous [devez configurer Auth0](/docs/fr-CA/secure/multi-factor-authentication/fido-authentication-with-webauthn/configure-webauthn-security-keys-for-mfa) pour exiger un NIP. Les utilisateurs devront alors entrer un NIP, stocké uniquement dans la clé de sécurité, pour terminer l’authentification. Une fois la vérification de l’utilisateur effectuée, les utilisateurs peuvent ouvrir une session avec WebAuthn comme seule méthode d’authentification tout en satisfaisant aux exigences de l’authentification multifacteur.

  En utilisant WebAuthn pour l’authentification avec la vérification de l’utilisateur, vous remplacez non seulement le mot de passe par une solution beaucoup plus simple à utiliser, mais vous éliminez aussi la nécessité d’ajouter une autre étape d’authentification lorsque MFA est requise.
</Callout>
