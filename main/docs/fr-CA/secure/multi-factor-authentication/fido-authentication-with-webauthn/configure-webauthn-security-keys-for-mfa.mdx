<Warning>
  Si vous devez configurer un domaine personnalisé, faites-le **avant** de déployer WebAuthn dans des environnements de production. Si vous configurez ou modifiez un domaine personnalisé, les utilisateurs déjà inscrits ne pourront pas s’authentifier.

  Vous pouvez utiliser le paramètre [Relying Party](#configure-relying-party) pour préciser le domaine utilisé pour authentifier les utilisateurs.

<div id="use-the-dashboard">
  ## Utiliser le Dashboard
</div>

1. Activez **WebAuthn avec des clés de sécurité** dans [Dashboard &gt; Sécurité &gt; Multi-factor Auth](https://manage.auth0.com/#/security/mfa).
2. Configurez la façon dont vous voulez gérer la vérification de l’utilisateur. Pour les clés de sécurité, la vérification de l’utilisateur consiste généralement à demander aux utilisateurs de saisir un NIP pour répondre au challenge WebAuthn.

   1. Never : les utilisateurs ne seront jamais invités à saisir un NIP. Il s’agit de la valeur par défaut, et elle est généralement suffisante lorsque vous utilisez des clés de sécurité pour la MFA. Les utilisateurs ont déjà saisi leur mot de passe; ils ont donc déjà fourni un certain niveau de vérification.
   2. If supported : les utilisateurs seront invités à saisir un NIP s’ils en ont déjà configuré un sur la clé.
   3. Required : les utilisateurs devront définir un NIP s’il n’est pas déjà défini, puis le saisir chaque fois. C’est l’option qui offre le niveau de sécurité le plus élevé. Certains navigateurs ne l’implémentent pas correctement (par exemple, [Brave](https://brave.com/) sur iOS), donc l’authentification échouera et Auth0 demandera aux utilisateurs d’utiliser un autre navigateur.

Notez que seules les clés de sécurité conformes à FIDO-2 prennent en charge la vérification de l’utilisateur. Les clés FIDO-1 peuvent être utilisées avec WebAuthn, mais elles ne peuvent pas être utilisées si vous définissez User Verification sur Required.

<div id="configure-relying-party">
  ## Configurer Relying Party
</div>

WebAuthn rend l’hameçonnage impossible en liant les informations d’identification à l’origine du navigateur. Les utilisateurs ne peuvent donc pas utiliser WebAuthn pour un site auquel ils ne se sont pas inscrits.

Comme les informations d’identification sont liées à l’origine, si vous configurez un <Tooltip tip="Custom Domain : domaine tiers avec un nom spécialisé ou personnalisé." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=custom+domain">domaine personnalisé</Tooltip> ou que vous le modifiez, les utilisateurs inscrits avant ce changement ne pourront pas s’authentifier.

WebAuthn définit un [attribut d’ID Relying Party](https://www.w3.org/TR/webauthn-2/#relying-party-identifier), qui vous permet de préciser le domaine utilisé pour authentifier les utilisateurs. Vous pouvez le définir sur tout suffixe de domaine enregistrable de l’origine du navigateur. Par exemple, si le domaine personnalisé est `login.example.com`, vous pouvez configurer l’ID de la <Tooltip tip="Relying Party : entité (comme un service ou une application) qui dépend d’un fournisseur d’identité tiers pour authentifier un utilisateur." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Relying+Party">Relying Party</Tooltip> à `example.com`. Les utilisateurs pourront ainsi s’authentifier sur n’importe quel domaine relevant de `example.com` avec leurs informations d’identification WebAuthn.

Auth0 vous permet de préciser l’ID Relying Party seulement si vous avez configuré un domaine personnalisé. Si le domaine personnalisé change, vous devez mettre à jour l’ID Relying Party.

<div id="device-support">
  ## Compatibilité des appareils
</div>

Pour utiliser des clés de sécurité, le navigateur doit avoir JavaScript activé et prendre en charge WebAuthn. Si ces conditions ne sont pas remplies, Auth0 ne proposera pas l’option d’inscrire des clés de sécurité ni de s’authentifier avec celles-ci. Auth0 demandera alors un autre facteur ou un code de récupération (si aucun autre facteur n’est inscrit).

Les versions les plus récentes des navigateurs et des systèmes d’exploitation courants prennent en charge WebAuthn avec des clés de sécurité. Pour plus de détails, consultez la [section sur la compatibilité des navigateurs dans webauthn.me](https://webauthn.me/browser-support).

<div id="limitations">
  ## Limitations
</div>

* Lorsque vous utilisez la [MFA API](/docs/fr-CA/secure/multi-factor-authentication/multi-factor-authentication-developer-resources/mfa-api), vous pouvez lister et supprimer les inscriptions WebAuthn, mais vous ne pouvez pas en créer.
