<Warning>
  Si vous devez définir un domaine personnalisé, faites-le **avant** de déployer WebAuthn dans des environnements de production. Si vous définissez ou modifiez un domaine personnalisé, les utilisateurs qui ont déjà enregistré ce facteur ne pourront pas s’authentifier.

  Vous pouvez utiliser le paramètre [partie de confiance](#configure-relying-party) pour préciser le domaine utilisé pour authentifier les utilisateurs.
</Warning>

<div id="use-the-dashboard">
  ## Utiliser le Dashboard
</div>

1. Activez **WebAuthn with clé de sécurité** dans [Dashboard &gt; Sécurité &gt; Multi-factor Auth](https://manage.auth0.com/#/security/mfa).
2. Configurez la façon dont vous souhaitez gérer la vérification de l’utilisateur. Pour les clés de sécurité, la vérification de l’utilisateur demande généralement aux utilisateurs de saisir un NIP pour compléter le challenge WebAuthn.

   1. Jamais : les utilisateurs ne seront jamais invités à saisir un NIP. Il s’agit de la valeur par défaut, et elle est généralement suffisante lorsque vous utilisez des clés de sécurité pour la AMF. Les utilisateurs ont déjà saisi leur mot de passe et ont donc déjà fourni une certaine vérification.
   2. Si pris en charge : les utilisateurs seront invités à saisir un NIP s’ils en ont déjà configuré un sur la clé.
   3. Requis : les utilisateurs devront définir un NIP s’il n’est pas déjà configuré, et ils devront le saisir chaque fois. C’est l’option qui offre le niveau de sécurité le plus élevé. Certains navigateurs ne prennent pas correctement cela en charge (par exemple, [Brave](https://brave.com/) sur iOS), donc l’authentification échouera et Auth0 demandera aux utilisateurs d’utiliser un autre navigateur.

Notez que seules les clés de sécurité conformes à FIDO-2 prennent en charge la vérification de l’utilisateur. Les clés FIDO-1 peuvent être utilisées avec WebAuthn, mais elles ne peuvent pas être utilisées si vous définissez la vérification de l’utilisateur sur Requis.

<div id="configure-relying-party">
  ## Configurer la partie de confiance
</div>

WebAuthn rend l’hameçonnage impossible en liant les informations d’identification à l’origine du navigateur. Les utilisateurs ne peuvent pas non plus utiliser WebAuthn sur un site pour lequel ils ne se sont pas inscrits.

Le fait de lier les informations d’identification à l’origine signifie que si vous configurez un <Tooltip tip="Domaine personnalisé : domaine tiers avec un nom spécialisé ou personnalisé." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=custom+domain">domaine personnalisé</Tooltip> ou le modifiez, les utilisateurs inscrits avant ce changement ne pourront pas s’authentifier.

WebAuthn définit un [attribut d’ID de partie de confiance](https://www.w3.org/TR/webauthn-2/#relying-party-identifier), qui vous permet de préciser le domaine utilisé pour authentifier les utilisateurs. Vous pouvez le définir sur n’importe quel suffixe de domaine enregistrable de l’origine du navigateur. Par exemple, si le domaine personnalisé est `login.example.com`, vous pouvez configurer l’ID de la <Tooltip tip="Partie de confiance : entité (comme un service ou une application) qui s’appuie sur un fournisseur d’identité tiers pour authentifier un utilisateur." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Relying+Party">partie de confiance</Tooltip> à `example.com`. Cela permet aux utilisateurs de s’authentifier sur n’importe quel domaine `example.com` avec leurs informations d’identification WebAuthn.

Auth0 vous permet de préciser l’ID de partie de confiance seulement si vous avez configuré un domaine personnalisé. Si le domaine personnalisé change, vous devez mettre à jour l’ID de partie de confiance.

<div id="device-support">
  ## Prise en charge des appareils
</div>

Pour utiliser des clés de sécurité, un navigateur doit avoir Javascript activé et prendre en charge WebAuthn. Si ces conditions ne sont pas remplies, Auth0 n’offrira pas l’option d’inscrire des clés de sécurité ni de s’authentifier avec celles-ci. Auth0 proposera alors une vérification avec un autre facteur ou un code de récupération (si aucun autre facteur n’est déjà inscrit).

Les versions les plus récentes des navigateurs et des systèmes d’exploitation courants prennent en charge WebAuthn with clé de sécurité. Pour en savoir plus, consultez la [section sur la compatibilité des navigateurs sur webauthn.me](https://webauthn.me/browser-support).

<div id="limitations">
  ## Limites
</div>

* Lorsque vous utilisez l’[API AMF](/docs/fr-CA/secure/multi-factor-authentication/multi-factor-authentication-developer-resources/mfa-api), vous pouvez consulter et supprimer les inscriptions WebAuthn, mais vous ne pouvez pas en créer.
