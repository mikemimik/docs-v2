Les Online Refresh Tokens (ORT) sont un type de [jeton d’actualisation](/docs/fr-CA/secure/tokens/refresh-tokens) conçu spécialement pour les applications Web basées sur le navigateur, comme les [applications monopages](/docs/fr-CA/get-started/auth0-overview/create-applications/single-page-web-apps) (SPA).

Les ORT peuvent suivre la [session](/docs/fr-CA/manage-users/sessions) d’un utilisateur dans les SPA lorsque des fonctions de protection de la vie privée du navigateur, comme Intelligent Tracking Prevention (ITP), bloquent l’accès aux cookies tiers, ce qui rend l’[authentification silencieuse](/docs/fr-CA/authenticate/login/configure-silent-authentication#configure-silent-authentication) traditionnelle peu fiable.

Les ORT sont liés à la [session](/docs/fr-CA/manage-users/sessions) Auth0 de l’utilisateur et ne peuvent être utilisés que tant que cette session est active, ce qui garantit que lorsqu’un utilisateur se déconnecte, toutes les applications qui utilisent cette session sont déconnectées en même temps.

<div id="how-it-works">
  ## Fonctionnement
</div>

Lorsque vous demandez un ORT :

1. Votre application inclut la portée `online_access` dans la demande d&#39;autorisation.
2. Auth0 renvoie un ORT lié à la session en cours de l&#39;utilisateur.
3. Chaque fois que vous échangez l&#39;ORT contre un nouveau jeton d&#39;accès, Auth0 prolonge le délai d&#39;inactivité de la session.
4. Le jeton demeure valide tant que la session Auth0 correspondante est active.

<div id="comparison-of-orts-versus-refresh-tokens">
  ## Comparaison entre les ORT et les jetons d’actualisation
</div>

| **Caractéristique**                                                                                                                        | **ORT**                                                                                                      | **jeton d’actualisation**                                                                                        |
| ------------------------------------------------------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------ | ---------------------------------------------------------------------------------------------------------------- |
| **Portée demandée**                                                                                                                        | `online_access`                                                                                              | `offline_access`                                                                                                 |
| **Liaison à la session**                                                                                                                   | Lié à la session Auth0                                                                                       | Indépendant de la session                                                                                        |
| **Durée de vie**                                                                                                                           | Déterminée par la [durée de vie de la session](/docs/fr-CA/manage-users/sessions/configure-session-lifetime)       | Configurée par la [durée de vie du jeton](/docs/fr-CA/secure/tokens/refresh-tokens/configure-refresh-token-expiration) |
| **Rotation du jeton**                                                                                                                      | Ne fait pas l’objet d’une rotation                                                                           | Prend en charge la [rotation](/docs/fr-CA/secure/tokens/refresh-tokens/configure-refresh-token-rotation)               |
| **Exigence de [démonstration de la preuve de possession](/docs/fr-CA/secure/sender-constraining/demonstrating-proof-of-possession-dpop) (DPoP)** | Requise pour les clients publics (SPA, Native)                                                               | Facultative (configuration du serveur de ressources)                                                             |
| **Révocation**                                                                                                                             | [Révoque toute la session](/docs/fr-CA/manage-users/sessions/manage-sessions-actions#revoke-sessions-with-actions) | [Révoque seulement le jeton](/docs/fr-CA/secure/tokens/refresh-tokens/revoke-refresh-tokens)                           |
| **Management API**                                                                                                                         | Non visible                                                                                                  | Visible dans la Management API                                                                                   |
| **Délai d’inactivité de la session**                                                                                                       | Prolongé à chaque échange                                                                                    | Aucun effet sur la session                                                                                       |

<div id="key-behavioral-differences">
  ### Principales différences de comportement
</div>

**Prolongation de la session :** Lorsque vous échangez un ORT, Auth0 réinitialise le délai d’inactivité de la session, ce qui maintient la session de l’utilisateur active. Auth0 ne prolonge pas la [durée de vie absolue](/docs/fr-CA/manage-users/sessions/session-lifecycle#absolute-timeout-maximum) de la session.

**Application de DPoP :** Pour les clients publics (comme les SPA), Auth0 exige DPoP lors de l’utilisation d’ORT. Comme les ORT offrent le même niveau de sécurité qu’un cookie de session donnant accès à la session authentifiée de l’utilisateur, ils doivent être protégés avec la même rigueur. DPoP empêche le vol de jetons et les attaques par rejeu.

**Sans état :** Les ORT sont liés à une session; ils ne peuvent donc pas être consultés ni gérés indépendamment. Vous ne pouvez pas les voir dans la Management API ni les révoquer individuellement.

<div id="best-practices">
  ## Pratiques exemplaires
</div>

1. **Conservez les jetons en mémoire :** Évitez de stocker les ORT dans `localStorage` ou `sessionStorage`. Gardez-les dans la mémoire Javascript pour une meilleure sécurité. Comme les ORT sont liés à la session Auth0, le cookie de session rétablira le contexte lors de la prochaine visite de l’utilisateur, ce qui permettra à votre application de demander un nouvel ORT.
2. **Gérez l’expiration de la session adéquatement :** Prévoyez une gestion appropriée des erreurs lorsque la session a expiré. Redirigez les utilisateurs pour qu’ils s’authentifient de nouveau sans perdre leur travail.
3. **Utilisez DPoP pour les clients publics :** Les ORT exigent DPoP pour les clients publics. Assurez-vous que votre mise en œuvre génère et utilise correctement les preuves DPoP.
4. **Ne vous fiez pas à la structure des jetons :** Traitez les ORT comme des chaînes opaques. N’essayez pas d’en analyser le contenu et ne vous fiez à aucune structure interne — cela pourrait changer.
5. **Surveillez la durée de vie de la session :** Tenez compte à la fois du délai d’inactivité et du délai d’expiration absolu de la session. Le délai d’expiration absolu ne peut pas être prolongé au moyen d’un échange de jeton d’actualisation.

<div id="limitations">
  ## Limites
</div>

<div id="unsupported-flows">
  ### Flux non pris en charge
</div>

Les ORT sont offerts uniquement pour les flux qui créent une session interactive dans le navigateur. Les flux suivants **ne peuvent pas** émettre d’ORT :

| **Flux**                                                                                                                                        | **Raison**                                                                        |
| ----------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------- |
| [Flux d’autorisation de l’appareil](/docs/fr-CA/get-started/authentication-and-authorization-flow/device-authorization-flow)                          | Session créée dans le navigateur, mais jeton utilisé sur un autre appareil        |
| [Flux implicite](/docs/fr-CA/get-started/authentication-and-authorization-flow/implicit-flow-with-form-post)                                          | La spécification OAuth interdit les jetons d’actualisation dans le flux implicite |
| [Octroi par mot de passe du propriétaire de la ressource](/docs/fr-CA/get-started/authentication-and-authorization-flow/resource-owner-password-flow) | Aucune session de navigateur n’est créée                                          |
| [Flux d’identifiants client](/docs/fr-CA/get-started/authentication-and-authorization-flow/client-credentials-flow)                                   | Aucun utilisateur ni aucune session ne sont impliqués                             |
| [Échange de jetons personnalisé](/docs/fr-CA/get-started/authentication-and-authorization-flow/token-exchange-flow)                                   | Aucune session à laquelle rattacher le jeton                                      |

Si vous demandez la portée `online_access` dans l’un de ces flux, elle est ignorée sans avertissement et aucun jeton d’actualisation n’est émis.

<div id="scope-conflicts">
  ### Conflits de portées
</div>

Vous ne pouvez pas demander `online_access` et `offline_access` dans la même requête d’autorisation. Si les deux portées sont incluses, Auth0 renvoie une erreur :

```json
{
  "error": "invalid_request",
  "error_description": "Cannot request both online_access and offline_access scopes"
}
```

Sélectionnez la portée appropriée selon votre cas d’utilisation :

* Utilisez `online_access` pour les applications Web qui bénéficient de la continuité du SSO.
* Utilisez `offline_access` pour les applications qui ont besoin de jetons pour fonctionner indépendamment de la session du navigateur.
