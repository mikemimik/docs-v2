---

Pour lire les claims personnalisés dans les jetons d’accès et les <Tooltip tip="ID Token : information d’identification destinée au client lui-même, plutôt qu’à l’accès à une ressource." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=ID+tokens">jetons d’identité</Tooltip>, vous devez utiliser des <Tooltip tip="ID Token : information d’identification destinée au client lui-même, plutôt qu’à l’accès à une ressource." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=JSON+Web+Tokens">JSON Web Tokens</Tooltip> (JWT) et inclure une <Tooltip tip="JSON Web Token (JWT) : format de jeton d’identification standard (et souvent format de jeton d’accès) utilisé pour représenter des claims de façon sécurisée entre deux parties." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=audience">audience</Tooltip> (`aud`) dans un flux de connexion OIDC. Pour en savoir plus, consultez [Access Tokens](/docs/fr-CA/secure/tokens/access-tokens).

Lorsque vous configurez des claims personnalisés sur des JWT, vous devez éviter les collisions. Pour éviter que vos claims personnalisés entrent en conflit avec des claims réservés ou des claims provenant d’autres ressources, donnez-leur un nom résistant aux collisions. Auth0 recommande d’utiliser un format avec espace de noms.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Auth0 autorise les claims avec et sans espace de noms, mais certaines restrictions s’appliquent (consultez [Restrictions générales](/docs/fr-CA/secure/tokens/json-web-tokens/create-custom-claims#general-restrictions)). Pour éviter les collisions de noms, nous recommandons d’utiliser des claims avec espace de noms. En cas de collision, la transaction n’échouera pas, mais votre claim personnalisé ne sera pas ajouté à vos jetons.
</Callout>

<div id="general-restrictions">
  ## Restrictions générales
</div>

Auth0 applique les restrictions suivantes aux claims personnalisés :

* La taille de la charge utile des claims personnalisés est limitée à 100 KB maximum
* Les [claims standard OPENID](https://openid.net/specs/openid-connect-core-1_0.html#IDToken) et les claims utilisés à l’interne par Auth0 ne peuvent être ni personnalisés ni modifiés
* Les <Tooltip tip="Jeton d’accès : justificatif d’autorisation, sous la forme d’une chaîne opaque ou d’un JWT, utilisé pour accéder à une API." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Access+tokens">jetons d’accès</Tooltip> ayant une audience d’API Auth0, à l’exception du point de terminaison `/userinfo`, ne peuvent pas contenir de claims personnalisés privés sans espace de noms
* Seuls les claims de profil utilisateur OIDC indiqués peuvent être ajoutés aux jetons d’accès

<div id="non-restricted-claims">
  ### Claims non restreints
</div>

Vous pouvez créer des claims contenant des renseignements sensibles sur l’utilisateur afin d’enrichir le profil utilisateur et d’améliorer l’expérience utilisateur. Votre application consomme ces claims à partir des jetons d’identité. Pour en savoir plus sur l’utilisation des claims non restreints, consultez [jeton d’identité](/docs/fr-CA/secure/tokens/id-tokens) et gardez à l’esprit [Token Best Practices](/docs/fr-CA/secure/tokens/token-best-practices) si vous les utilisez.

Les claims suivants sont uniquement soumis à des restrictions générales :

* `address`
* `birthdate`
* `email`
* `email_verified`
* `family_name`
* `gender`
* `given_name`
* `locale`
* `middle_name`
* `name`
* `nickname`
* `phone_number`
* `phone_number_verified`
* `picture`
* `preferred_username`
* `profile`
* `updated_at`
* `website`
* `zoneinfo`

<div id="namespaced-guidelines">
  ## Directives relatives aux espaces de noms
</div>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  L’URN Auth0 `urn:auth0` ne peut pas servir d’identifiant d’espace de noms.
</Callout>

Utilisez les directives suivantes pour les identifiants d’espace de noms :

* Utilisez comme identifiant d’espace de noms n’importe quelle URL HTTP ou HTTPS autre qu’une URL Auth0. Les domaines Auth0 ne peuvent pas servir d’identifiants d’espace de noms, notamment :

  * auth0.com
  * webtask.io
  * webtask.run
* Utilisez comme identifiant d’espace de noms une URL que vous contrôlez; vous réduirez ainsi le risque qu’une autre personne utilise le même espace de noms. L’URL de l’espace de noms n’a pas à pointer vers une ressource réelle. Elle sert uniquement d’identifiant; elle ne sera pas appelée.
* Commencez l’URL par `http://` ou `https://`.
* Vous pouvez aussi utiliser des identifiants d’espace de noms fondés sur des URN si vous utilisez une API personnalisée. Dans ce cas, l’URN Auth0 `urn:auth0` est réservé et ne peut pas servir d’identifiant d’espace de noms.
* Créez plusieurs espaces de noms, au besoin.

Une fois votre espace de noms choisi, ajoutez-y le claim pour créer un claim associé à un espace de noms, qui peut être ajouté à un jeton. Par exemple :

`http://www.example.com/favorite_color`

<div id="non-namespaced-guidelines">
  ## Directives sur les claims personnalisés sans espace de noms
</div>

Utilisez les directives suivantes pour les claims personnalisés sans espace de noms :

* Sauf en cas de nécessité absolue pour votre application, utilisez des claims personnalisés publics avec espace de noms afin d’éviter les collisions.
* Créez des claims avec des noms explicites qui évitent les collisions. Par exemple, utilisez `employee_id` au lieu de `e_id`.
* Gardez les noms et les valeurs des claims aussi légers que possible en ne transmettant que les données strictement nécessaires à votre application.
* Évitez d’attribuer des payloads volumineux aux claims personnalisés.

Pour voir d’autres exemples de claims personnalisés ajoutés à un jeton, consultez [Exemples de cas d’utilisation : portées et claims](/docs/fr-CA/get-started/apis/scopes/sample-use-cases-scopes-and-claims).

<div id="create-custom-claims">
  ## Créer des claims personnalisés
</div>

Utilisez Auth0 Actions pour créer des claims personnalisés. L’objet `api` vous permet d’utiliser la méthode `setCustomClaim` sur les jetons d’accès ou les jetons d’identité.

<div id="example">
  ### Exemple
</div>

```js lines
exports.onExecuteCredentialsExchange = async (event, api) => {
  api.accessToken.setCustomClaim('myClaim', 'this is a private, non namespaced claim');
};
```
