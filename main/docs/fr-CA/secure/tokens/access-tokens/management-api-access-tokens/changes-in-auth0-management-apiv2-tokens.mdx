---

Il y a quelque temps, nous avons modifié le processus d’obtention d’un jeton Management APIv2. Cet article explique ce qui a changé, pourquoi ce changement a été apporté et comment le contourner (ce qui n’est pas recommandé).

<div id="what-changed-and-why">
  ## Ce qui a changé et pourquoi
</div>

<div id="the-user-experience">
  ### L’expérience utilisateur
</div>

Jusqu’à tout récemment, vous pouviez générer un jeton Management APIv2 directement depuis l’explorateur <Tooltip tip="Management API : un produit qui permet aux clients d’effectuer des tâches administratives." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Management+API">Management API</Tooltip>. Vous sélectionniez les scopes selon le endpoint que vous vouliez appeler, puis obteniez un jeton à partir de cette même page.

Cette méthode était très simple, mais elle était aussi peu sécurisée. Nous l’avons donc modifiée.

La nouvelle méthode utilise le [Client Credentials Flow](/docs/fr-CA/get-started/authentication-and-authorization-flow/client-credentials-flow). Pour savoir comment utiliser le nouveau processus, consultez [jeton d’accès de la Management API](/docs/fr-CA/secure/tokens/access-tokens/management-api-access-tokens).

<div id="why-this-changed">
  #### Pourquoi cela a changé
</div>

Pour générer le token, la Management API devait avoir accès à votre <Tooltip tip="Client Secret : secret utilisé par un client (application) pour s'authentifier auprès de l'Authorization Server; il ne doit être connu que du client et de l'Authorization Server et doit être suffisamment aléatoire pour ne pas pouvoir être deviné." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Client+Secret">Client Secret</Tooltip> global (utilisé pour signer le token). Or, il s&#39;agit d&#39;une information qui ne devrait pas être exposée aux navigateurs Web.

De plus, l&#39;API Explorer n&#39;a aucun moyen de gérer l&#39;autorisation. Cela signifie que si un utilisateur pouvait ouvrir une session et accéder à l&#39;API Explorer, il pourrait générer un token avec n&#39;importe quel scope, même s&#39;il n&#39;était pas autorisé à avoir ce scope.

La nouvelle mise en œuvre ne présente pas ce genre de risques. Une fois la configuration initiale terminée, vous pouvez obtenir un token soit en accédant à l&#39;<Tooltip tip="Auth0 Dashboard : le principal produit d'Auth0 pour configurer vos services." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip>, soit en envoyant une simple requête `POST` au [`endpoint /oauth/token` de notre Authentication API](https://auth0.com/docs/api/authentication#client-credentials).

Cependant, en ce qui concerne le processus manuel, nous comprenons bien que passer d&#39;un écran à l&#39;autre n&#39;offre pas toujours la meilleure expérience utilisateur; nous cherchons donc des moyens de rendre le nouveau flux plus intuitif.

<div id="the-validity-period">
  ### La durée de validité
</div>

Dans l’ancien flux, les jetons n’expiraient jamais. Dans le nouveau flux, tous les jetons Management APIv2 expirent par défaut après 24 heures.

#### Pourquoi ce changement

Avoir un jeton qui n’expire jamais peut être très risqué si un attaquant met la main dessus. Si le jeton expire au bout de quelques heures, l’attaquant ne dispose que d’une courte période pour accéder à vos ressources protégées.

Pour obtenir un jeton, vous ne devez suivre que la procédure décrite dans [Jetons d’accès de la Management API](/docs/fr-CA/secure/tokens/access-tokens/management-api-access-tokens).
