description: Décrit comment valider un jeton d’accès.
title: Valider les jetons d’accès
---

Un <Tooltip tip="Jeton d’accès : information d’autorisation, sous la forme d’une chaîne opaque ou d’un JWT, utilisée pour accéder à une API." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=access+token">jeton d’accès</Tooltip> est destiné à une API et ne doit être validé que par l’API à laquelle il est destiné.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Les jetons d’accès du fournisseur d’identité (IdP) ne nécessitent pas de validation. Transmettez le jeton d’accès de l’IdP à l’IdP émetteur pour qu’il effectue la validation. Pour en savoir plus, consultez [Jetons d’accès du fournisseur d’identité](/docs/fr-CA/secure/tokens/access-tokens/identity-provider-access-tokens).
</Callout>

Si l’une de ces vérifications échoue, le jeton est considéré comme invalide et la requête doit être rejetée avec un code `401 Unauthorized`.

1. **Effectuez la validation JWT standard.** Puisque le jeton d’accès est un JWT, vous devez suivre les étapes standard de validation JWT. Consultez [Valider les JSON Web Tokens](/docs/fr-CA/secure/tokens/json-web-tokens/validate-json-web-tokens) pour en savoir plus.
2. **Vérifiez les claims d’audience du jeton.** Si vous avez effectué la validation JWT standard, vous avez déjà décodé la charge utile du JWT et examiné ses claims standard. Le claim d’audience du jeton (`aud`, tableau de chaînes) dépend de la requête de jeton initiale. Le champ `aud` peut contenir à la fois une audience correspondant à votre API personnalisée et une audience correspondant au point de terminaison `/userinfo`. Au moins une des valeurs d’audience du jeton doit correspondre à l’identificateur unique de l’API cible, tel qu’il est défini dans les [Settings de votre API](https://manage.auth0.com/#/apis), dans le champ **Identifier**. Consultez [obtenir des jetons d’accès](/docs/fr-CA/secure/tokens/access-tokens/get-access-tokens) pour en savoir plus.
3. **Vérifiez les permissions (scopes).** Vérifiez que l’application a reçu les permissions requises pour accéder à votre API. Pour ce faire, vous devrez vérifier le claim `scope` (`scope`, liste de chaînes séparées par des espaces) dans la charge utile du JWT décodé. Il doit correspondre aux permissions requises pour le point de terminaison consulté. Par exemple, si votre API personnalisée fournit trois points de terminaison pour lire, créer ou supprimer un enregistrement d’utilisateur, lorsque vous avez enregistré votre API avec Auth0, vous avez créé trois permissions correspondantes :

   1. `create:users` donne accès au point de terminaison `/create`
   2. `read:users` donne accès au point de terminaison `/read`
   3. `delete:users` donne accès au point de terminaison `/delete`

   Dans ce cas, si une application demande l’accès au point de terminaison `/create`, mais que le claim `scope` du jeton d’accès n’inclut pas la valeur `create:users`, l’API doit rejeter la requête.
