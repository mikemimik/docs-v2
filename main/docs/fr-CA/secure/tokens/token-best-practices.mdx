title: Pratiques exemplaires relatives aux jetons
---

Voici quelques points de base à garder à l’esprit lorsque vous utilisez des jetons :

* **Gardez-la secrète. Gardez-la en lieu sûr** : la clé de signature doit être traitée comme tout autre justificatif d’authentification et ne doit être divulguée qu’aux services qui en ont besoin.
* **N’ajoutez pas de données sensibles au payload** : les jetons sont signés pour éviter toute falsification, mais ils se décodent facilement. Ajoutez le nombre minimal de claims au payload afin d’optimiser les performances et la sécurité.
* **Définissez une date d’expiration pour les jetons** : techniquement, une fois qu’un jeton est signé, il reste valide indéfiniment, à moins que la clé de signature soit modifiée ou qu’une expiration soit explicitement définie. Cela peut poser problème; prévoyez donc une stratégie pour faire expirer et/ou révoquer les jetons.
* **Privilégiez HTTPS** : n’envoyez pas de jetons sur des connexions non HTTPS, car ces requêtes peuvent être interceptées et les jetons compromis.
* **Tenez compte de tous vos cas d’utilisation en matière d’autorisation** : l’ajout d’un système secondaire de vérification des jetons, qui garantit qu’ils ont bien été générés par votre serveur, peut être nécessaire pour répondre à vos exigences.
* **Stockez et réutilisez :** réduisez les allers-retours inutiles qui augmentent la surface d’attaque de votre application, et optimisez les limites de jetons prévues par votre forfait (le cas échéant) en stockant les <Tooltip tip="Jeton d’accès : justificatif d’autorisation, sous la forme d’une chaîne opaque ou d’un JWT, utilisé pour accéder à une API." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=access+tokens">jetons d’accès</Tooltip> obtenus auprès du <Tooltip tip="Serveur d’autorisation : serveur centralisé qui contribue à définir les limites de l’accès d’un utilisateur. Par exemple, votre serveur d’autorisation peut contrôler les données, les tâches et les fonctionnalités accessibles à un utilisateur." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=authorization+server">serveur d’autorisation</Tooltip>. Au lieu de demander un nouveau jeton, utilisez le jeton stocké lors des appels ultérieurs jusqu’à son expiration. La façon de stocker les jetons dépend des caractéristiques de votre application : les solutions courantes comprennent les bases de données (pour les applications qui doivent effectuer des appels d’API même en l’absence de session) et les sessions HTTP (pour les applications dont la fenêtre d’activité est limitée à une session interactive). Pour un exemple de stockage côté serveur et de réutilisation des jetons, consultez [Stockage des jetons](/docs/fr-CA/secure/security-guidance/data-security/token-storage).

<div id="tokens-vs-cookies">
  ## Jetons vs. cookies
</div>

En général, les applications à page unique (comme React, Vue et AngularJS + Node), les applications mobiles natives (comme iOS et Android) et les API Web (écrites en Node, Ruby, ASP.NET, ou dans une combinaison de ces technologies) profitent le plus de l’authentification fondée sur des jetons. Les applications Web traditionnelles côté serveur ont, quant à elles, traditionnellement utilisé l’authentification fondée sur des cookies.

L’authentification fondée sur des jetons consiste à générer un jeton lorsque l’utilisateur s’authentifie, puis à placer ce jeton dans l’en-tête `Authorization` de chaque requête subséquente vers votre API. Idéalement, ce jeton devrait être dans un format standard, comme <Tooltip tip="JSON Web Token (JWT) : format standard d’ID Token (et souvent d’Access Token) utilisé pour représenter de façon sécuritaire des claims entre deux parties." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=JSON+web+tokens">les JSON Web Tokens</Tooltip>, puisque vous trouverez des bibliothèques sur la plupart des plateformes et que vous n’aurez pas à implémenter votre propre cryptographie.

Avec les deux approches, vous pouvez obtenir la même quantité d’information de l’utilisateur. Cela est contrôlé par le paramètre `scope` envoyé dans la requête de login (soit en utilisant Lock, notre bibliothèque JavaScript, soit un simple lien). Le `scope` est un paramètre de la méthode `.signin({scope: 'openid name email'})` qui se retrouve ensuite dans la chaîne de requête de la requête de login.

Par défaut, nous utilisons `scope=openid` dans l’authentification fondée sur des jetons pour éviter d’avoir un jeton trop volumineux. Vous pouvez contrôler les claims standard de <Tooltip tip="OpenID : norme ouverte d’authentification qui permet aux applications de vérifier l’identité des utilisateurs sans recueillir ni stocker leurs renseignements de login." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=OpenID">OpenID</Tooltip> Connect (OIDC) que vous voulez obtenir dans le jeton en les ajoutant comme valeurs de scope. Par exemple, `scope=openid name email family_name address phone_number`. Pour en savoir plus, consultez [Standard Claims on openid.net](https://openid.net/specs/openid-connect-core-1_0.html#StandardClaims).

Vous pouvez combiner l’authentification fondée sur des jetons avec l’authentification fondée sur des cookies. Gardez à l’esprit que les cookies fonctionneront très bien si l’application Web et l’API sont servies à partir du même domaine; vous n’aurez donc peut-être pas besoin d’une authentification fondée sur des jetons. Si nécessaire, nous renvoyons aussi un JWT dans le flux de l’application Web. Chacun de nos SDKs le fait différemment. Si vous voulez appeler vos API à partir de JavaScript (au lieu d’utiliser le cookie existant), vous devez alors gérer les jetons d’accès à l’aide de Web Workers ou de closures JavaScript pour assurer leur transmission et leur stockage. Pour en savoir plus, consultez la section Browser in-memory scenarios de notre page [Token Storage](/docs/fr-CA/secure/security-guidance/data-security/token-storage).

<div id="refresh-token-usage">
  ## Utilisation des jetons d’actualisation
</div>

Vous ne pouvez obtenir un <Tooltip tip="Jeton d’actualisation : jeton utilisé pour obtenir un nouveau jeton d’accès sans obliger les utilisateurs à se reconnecter." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Refresh+token">jeton d’actualisation</Tooltip> que si vous mettez en œuvre les flux suivants :

* [Flux de code d’autorisation](/docs/fr-CA/get-started/authentication-and-authorization-flow/authorization-code-flow/add-login-auth-code-flow)
* [Flux de code d’autorisation avec clé de preuve pour l’échange de code (PKCE)](/docs/fr-CA/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce/add-login-using-the-authorization-code-flow-with-pkce)
* [Flux de mot de passe du propriétaire de la ressource](/docs/fr-CA/get-started/authentication-and-authorization-flow/resource-owner-password-flow)
* [Flux d’autorisation de l’appareil](/docs/fr-CA/get-started/authentication-and-authorization-flow/device-authorization-flow)

Si vous limitez l’accès hors ligne à votre API au moyen du commutateur **Allow Offline Access** dans [Auth0 Dashboard &gt; Applications &gt; APIs &gt; Settings](https://manage.auth0.com/#/apis), Auth0 ne renverra pas de jeton d’actualisation pour l’API (même si vous incluez le scope `offline_access` dans votre requête).

Les Rules s’exécutent lors de l’échange du jeton d’actualisation. Pour exécuter une logique particulière, vous pouvez consulter la propriété `context.protocol` dans votre rule. Si la valeur est `oauth2-refresh-token`, la rule s’exécute alors pendant l’échange.

Lorsque vous tentez d’obtenir un jeton d’actualisation, le paramètre <Tooltip tip="Audience : identifiant unique de l’audience d’un jeton émis. Nommée aud dans un jeton, sa valeur contient l’ID d’une application (Client ID) pour un ID Token ou d’une API (API Identifier) pour un jeton d’accès." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=audience">audience</Tooltip> n’est pas disponible dans l’objet `context` des Rules. Si vous recevez une erreur lorsque vous tentez d’ajouter le paramètre audience, vérifiez qu’il n’est pas défini dans le jeton.

Si vous essayez d’effectuer une redirection avec `context.redirect`, le flux d’authentification renverra une erreur.

Si vous avez ajouté des custom claims à vos jetons à l’aide d’une rule, elles apparaîtront dans les nouveaux jetons émis lors de l’utilisation d’un jeton d’actualisation tant que votre rule sera en place. Bien que les nouveaux jetons n’héritent pas automatiquement des custom claims, les Rules s’exécutent pendant le flux de jeton d’actualisation, de sorte que le même code sera exécuté. Cela vous permet d’ajouter ou de modifier des custom claims dans les jetons nouvellement émis sans obliger les applications déjà autorisées à obtenir un nouveau jeton d’actualisation.

<div id="refresh-token-limits">
  ### Limites des jetons d’actualisation
</div>

Auth0 limite à 200 le nombre de jetons d’actualisation actifs par utilisateur et par application. Cette limite s’applique uniquement aux jetons actifs. Si la limite est atteinte et qu’un nouveau jeton d’actualisation est créé, le système révoque et supprime le jeton le plus ancien pour cet utilisateur et cette application. Les jetons révoqués et expirés ne comptent pas dans cette limite.

<div id="automated-tests">
  #### Tests automatisés
</div>

Les jetons d’actualisation s’accumulent à cause des tests automatisés et sont généralement utilisés pendant toute la durée des tests. Pour éviter d’accumuler des jetons et de vous heurter aux limites des jetons d’actualisation, vous pouvez utiliser l’Auth0 <Tooltip tip="Management API : un produit qui permet aux clients d’effectuer des tâches administratives." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Management+API">Management API</Tooltip> pour supprimer les jetons d’actualisation inutiles.

1. [Créer un utilisateur](https://auth0.com/docs/api/management/v2#!/Users/post_users) avec Management API. Vous utiliserez cet utilisateur pour les tests.
2. La réponse renvoie un `user_id` que vous devez enregistrer pendant les tests afin de pouvoir l’utiliser plus tard.
3. Une fois les tests terminés, [supprimez l’utilisateur](https://auth0.com/docs/api/management/v2#!/Users/delete_users_by_id) au moyen de Management API. Lorsque l’utilisateur de test est supprimé, les artefacts associés le sont aussi, y compris les jetons d’actualisation.

<Warning>
  Pour ce cas d’utilisation, nous ne recommandons pas d’utiliser un ID utilisateur statique. Nous ne recommandons pas non plus de conserver les utilisateurs de test et les artefacts, ni de nettoyer les jetons d’actualisation à l’aide des points de terminaison des informations d’identification d’appareil, car vous pourriez atteindre les limites de débit de la Management API. Pour en savoir plus, consultez [les limites de débit des points de terminaison de la Management API.](/docs/fr-CA/troubleshoot/customer-support/operational-policies/rate-limit-policy/management-api-endpoint-rate-limits)
</Warning>

Si vous souhaitez conserver l’utilisateur de test pour de futurs tests :

1. Dressez la liste des jetons d’actualisation de l’utilisateur à l’aide du [point de terminaison des informations d’identification d’appareil de la Management API](https://auth0.com/docs/api/management/v2#!/Device_Credentials/get_device_credentials). Le point de terminaison renverra un maximum de 1000 jetons, sans ordre particulier, peu importe le nombre de jetons accumulés ou l’utilisation de la pagination.
2. Supprimez ces informations d’identification [à l’aide de la méthode DELETE](https://auth0.com/docs/api/management/v2#!/Device_Credentials/delete_device_credentials_by_id).
3. Si l’utilisateur a plus de 1k jetons, répétez l’opération de listage et de suppression des jetons jusqu’à ce qu’il n’en reste plus pour cet utilisateur.

<div id="configure-expiring-refresh-tokens">
  #### Configurer l’expiration des jetons d’actualisation
</div>

Lorsque les utilisateurs se connectent à votre application avec Auth0 et que `offline_access` est demandé dans la requête d’autorisation, un nouveau jeton d’actualisation est émis pour l’utilisateur. Si les utilisateurs se déconnectent, puis se reconnectent avec le même appareil, un nouveau jeton d’actualisation est émis. Selon la manière dont votre application stocke et utilise les jetons d’actualisation, l’ancien jeton d’actualisation issu de la première connexion peut devenir obsolète, et votre application utilisera très probablement les nouveaux jetons d’actualisation si les deux jetons sont émis avec la même audience. Pour en savoir plus, consultez [Stockage des jetons](/docs/fr-CA/secure/security-guidance/data-security/token-storage).

Pour éviter l’accumulation de jetons d’actualisation obsolètes, même si la limite de jetons supprime d’abord le plus ancien, nous vous recommandons de configurer l’expiration des jetons d’actualisation. Les jetons d’actualisation rotatifs et non rotatifs (ou réutilisables) peuvent être configurés pour expirer après une période d’inactivité ou selon une expiration absolue. Ces deux types d’expiration aident à supprimer les jetons qui ne sont pas activement utilisés et à éviter leur accumulation pour l’utilisateur. Pour en savoir plus, consultez [Configurer l’expiration des jetons d’actualisation](/docs/fr-CA/secure/tokens/refresh-tokens/configure-refresh-token-expiration).

  ## Validation des JWT
</div>

Nous vous recommandons fortement d’utiliser un middleware ou l’une des bibliothèques tierces open source existantes pour analyser et valider les JWT. Sur [JWT.io](https://jwt.io/#libraries-io), vous trouverez des bibliothèques pour diverses plateformes et divers langages de programmation, comme .NET, Python, Java, Ruby, Objective-C, Swift et PHP.

<div id="signing-algorithms">
  ## Algorithmes de signature
</div>

L’algorithme utilisé pour signer les jetons émis pour votre application ou votre API. Une signature fait partie d’un JWT et sert à vérifier que l’expéditeur du jeton est bien celui qu’il prétend être, et à s’assurer que le message n’a pas été modifié en cours de route. Pour en savoir plus sur les JWT, consultez [JSON Web Tokens](/docs/fr-CA/secure/tokens/json-web-tokens). Pour en savoir plus sur les signatures, consultez [JSON Web Token Structure](/docs/fr-CA/secure/tokens/json-web-tokens/json-web-token-structure).

Vous pouvez choisir parmi les <Tooltip tip="Algorithme de signature : algorithme utilisé pour signer numériquement des jetons afin de garantir que le jeton n’a pas été altéré." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=signing+algorithms">algorithmes de signature</Tooltip> suivants :

* **RS256** (signature RSA avec SHA-256) : algorithme asymétrique, ce qui signifie qu’il y a deux clés : une clé publique et une clé privée qui doit être gardée secrète. Auth0 détient la clé privée utilisée pour générer la signature, et le consommateur du JWT récupère une clé publique à partir des endpoints Metadata fournis par Auth0 et l’utilise pour [valider la signature du JWT](/docs/fr-CA/secure/tokens/json-web-tokens/validate-json-web-tokens).
* **HS256** (HMAC avec SHA-256) : algorithme symétrique, ce qui signifie qu’il n’y a qu’une seule clé privée qui doit être gardée secrète, et qu’elle est partagée entre les deux parties. Comme la même clé sert à la fois à générer la signature et à la valider, il faut veiller à ce qu’elle ne soit pas compromise. Cette clé privée (ou ce secret) est créée lorsque vous enregistrez votre Application (**<Tooltip tip="Secret client : secret utilisé par un client (application) pour s'authentifier auprès du serveur d'autorisation; il ne doit être connu que du client et du serveur d'autorisation et doit être suffisamment aléatoire pour ne pas pouvoir être deviné." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Client+Secret">Secret client</Tooltip>**) ou votre API (**secret de signature**) et que vous choisissez l’algorithme de signature HS256.

La pratique la plus sécuritaire, et celle que nous recommandons, consiste à utiliser **RS256** pour les raisons suivantes :

* Avec RS256, vous avez l’assurance que seul le détenteur de la clé privée (Auth0) peut signer des jetons, tandis que n’importe qui peut vérifier la validité du jeton à l’aide de la clé publique.
* Avec RS256, vous pouvez demander un jeton valide pour plusieurs audiences.
* Avec RS256, si la clé privée est compromise, vous pouvez effectuer une rotation de clé sans avoir à redéployer votre application ou votre API avec le nouveau secret (ce que vous devriez faire si vous utilisiez HS256).
* Avec HS256, si la clé secrète est compromise, vous devriez redéployer l’API avec le nouveau secret.

<div id="signing-keys">
  ## Clés de signature
</div>

Il est recommandé de partir du principe que plusieurs clés de signature peuvent être présentes dans votre JWKS. Cela peut sembler inutile, puisque le point de terminaison JWKS d’Auth0 contient généralement une seule clé de signature; toutefois, plusieurs clés peuvent s’y trouver lors de la rotation des certificats de signature.

Nous vous recommandons de mettre en cache vos clés de signature pour améliorer les performances de l’application et éviter de vous heurter aux limites de débit. Assurez-vous toutefois que si le décodage d’un jeton échoue, vous invalidez le cache et récupérez de nouvelles clés de signature avant de réessayer **une seule** autre fois.
