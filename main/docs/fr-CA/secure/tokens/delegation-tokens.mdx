description: Décrit le fonctionnement des jetons de délégation dans Auth0.
title: Jetons de délégation
---

<Warning>
  Par défaut, la délégation est désactivée pour les tenants qui n’utilisaient aucun add-on en date du 8 juin 2017. Les tenants existants qui utilisent actuellement un add-on nécessitant la délégation peuvent continuer à utiliser cette fonctionnalité. Si la fonctionnalité de délégation devait être modifiée ou retirée du service un jour, les clients qui l’utilisent actuellement en seraient avisés à l’avance et disposeraient de suffisamment de temps pour migrer.
</Warning>

Un jeton de délégation doit être obtenu et utilisé lorsqu’une application doit appeler l’API d’un add-on d’application, comme Firebase ou SAP, enregistré et configuré dans Auth0, dans le même tenant que le programme appelant.

À partir d’un jeton existant, ce point de terminaison génère un nouveau jeton signé avec le secret de l’application `target`. Cela sert à transmettre l’identité de l’utilisateur de l’application à une API.

Le type du jeton de délégation varie selon le fournisseur. Par exemple, s’il est émis pour Azure Blob Storage, il s’agira d’une SAS (Shared Access Signature). S’il est destiné à l’add-on Firebase, il s’agira d’un <Tooltip tip="JSON Web Token (JWT) : format standard de jeton ID (et souvent de jeton d’accès) utilisé pour représenter des claims de façon sécurisée entre deux parties." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=JWT">JWT</Tooltip>.

Le <Tooltip tip="Jeton ID : information d’authentification destinée au client lui-même, plutôt qu’à l’accès à une ressource." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=ID+token">jeton ID</Tooltip> d’un utilisateur authentifié peut être utilisé avec le point de terminaison `/delegation` pour demander un jeton de délégation pour une cible précise. La cible peut être un add-on d’application configuré dans Auth0. Les add-ons pour lesquels cela est possible sont ceux qui ne sont pas des add-ons <Tooltip tip="Security Assertion Markup Language (SAML) : protocole normalisé permettant à deux parties d’échanger des informations d’authentification sans mot de passe." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=SAML">SAML</Tooltip> ou <Tooltip tip="Security Assertion Markup Language (SAML) : protocole normalisé permettant à deux parties d’échanger des informations d’authentification sans mot de passe." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=WS-Fed">WS-Fed</Tooltip>. L’add-on doit être configuré dans Auth0 avec des secrets obtenus auprès du service d’add-on, comme Firebase. Des instructions pour configurer les secrets sont offertes sur la page de configuration de chaque add-on. Les secrets servent à signer le jeton de délégation afin que l’API de l’add-on puisse valider le jeton et s’y fier.

Le point de terminaison de délégation vous permet de définir plusieurs paramètres qui régissent le contenu du jeton de délégation, notamment `target`, `scope`, l’API à appeler (`api_type`), ainsi qu’une zone de forme libre pour des paramètres additionnels.

</div>

Pour voir comment obtenir un nouveau jeton pour un add-on que vous avez activé à l’aide d’**Auth0.js**, consultez Delegation Token Request. Notez que cet exemple concerne la **version 7** de la bibliothèque **Auth0.js**; la délégation n’est **pas prise en charge** dans la version 8 d’**Auth0.js**.

<div id="validity-period-and-termination">
  ## Période de validité et révocation
</div>

La période de validité et la possibilité de révoquer un jeton de délégation varient selon chaque add-on. Pour en savoir plus, consultez la documentation fournie par le fournisseur de l’API de l’add-on concerné.

<div id="use-delegation-tokens-with-public-applications">
  ## Utiliser des jetons de délégation avec des applications publiques
</div>

Il y a une mise en garde importante à prendre en compte lorsque vous utilisez le point de terminaison de délégation avec des applications publiques.

Si vous appelez le endpoint `/token` à partir d’une application publique, le jeton ID sera obligatoirement signé avec `RS256`, même si l’algorithme de signature JsonWebToken dans les paramètres de l’application est configuré sur `HS256`.

Si vous appelez ensuite le point de terminaison de délégation avec ce jeton ID, l’appel échouera si l’algorithme de signature JsonWebToken de l’application a été configuré sur `HS256`. Cela s’explique par le fait que Delegation effectue la validation selon les paramètres de l’application, mais que le jeton ID a été émis avec un algorithme différent en raison de ce changement d’algorithme imposé.

Il est donc important, si vous prévoyez utiliser Delegation avec une application publique, de configurer l’algorithme de signature JsonWebToken de votre application sur `RS256`.
