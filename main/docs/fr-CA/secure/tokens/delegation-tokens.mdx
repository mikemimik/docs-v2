title: Jetons de délégation
---

<Warning>
  Par défaut, la délégation est désactivée pour les tenants qui n’utilisaient pas de module complémentaire en date du 8 juin 2017. Les tenants Legacy qui utilisent actuellement un module complémentaire nécessitant la délégation peuvent continuer à utiliser cette fonctionnalité. Si la fonctionnalité de délégation est modifiée ou retirée du service un jour, les clients qui l’utilisent actuellement en seront avisés à l’avance et disposeront de suffisamment de temps pour migrer.
</Warning>

Un jeton de délégation doit être obtenu et utilisé lorsqu’une application doit appeler l’API d’un module complémentaire d’application, comme Firebase ou SAP, enregistré et configuré dans Auth0, dans le même tenant que le programme appelant.

À partir d’un jeton existant, ce endpoint génère un nouveau jeton signé avec le secret de l’application `target`. Il sert à transmettre l’identité de l’utilisateur de l’application à une API.

Le type de jeton de délégation varie selon le fournisseur. Par exemple, s’il est émis pour Azure Blob Storage, il s’agira d’une SAS (Shared Access Signature). S’il est destiné au module complémentaire Firebase, il s’agira d’un <Tooltip tip="JSON Web Token (JWT) : format standard de jeton d’identification (et souvent de jeton d’accès) utilisé pour représenter des claims de façon sécuritaire entre deux parties." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=JWT">JWT</Tooltip>.

Le <Tooltip tip="ID Token : information d’identification destinée au client lui-même, plutôt qu’à l’accès à une ressource." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=ID+token">ID token</Tooltip> d’un utilisateur authentifié peut être utilisé avec le endpoint `/delegation` pour demander un jeton de délégation pour une cible donnée. La cible peut être un module complémentaire d’application configuré dans Auth0. Les modules complémentaires pour lesquels cela est possible sont ceux qui ne sont pas des modules complémentaires <Tooltip tip="Security Assertion Markup Language (SAML) : protocole normalisé permettant à deux parties d’échanger des informations d’authentification sans mot de passe." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=SAML">SAML</Tooltip> ou <Tooltip tip="Security Assertion Markup Language (SAML) : protocole normalisé permettant à deux parties d’échanger des informations d’authentification sans mot de passe." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=WS-Fed">WS-Fed</Tooltip>. Le module complémentaire doit être configuré dans Auth0 avec des secrets obtenus auprès du service du module complémentaire, comme Firebase. Les instructions pour configurer les secrets sont disponibles sur la page de configuration de chaque module complémentaire. Les secrets servent à signer le jeton de délégation afin que l’API du module complémentaire puisse valider le jeton et s’y fier.

Le endpoint de délégation vous permet de définir plusieurs paramètres qui régissent le contenu du jeton de délégation, notamment `target`, `scope`, l’API à appeler (`api_type`), ainsi qu’une zone de texte libre pour des paramètres additionnels.

<div id="auth0js-example">
  ## Exemple d’Auth0.js
</div>

Pour obtenir un exemple montrant comment obtenir un nouveau jeton pour un module complémentaire que vous avez activé à l’aide d’**Auth0.js**, consultez Demande de jeton de délégation. Notez que cet exemple concerne la **version 7** de la bibliothèque **Auth0.js**; la délégation n’est **pas prise en charge** dans la version 8 d’**Auth0.js**.

<div id="validity-period-and-termination">
  ## Période de validité et révocation
</div>

La période de validité et la possibilité de révoquer un jeton de délégation varient d’un module complémentaire à l’autre. Consultez la documentation fournie par le fournisseur de l’API du module complémentaire concerné pour en savoir plus.

<div id="use-delegation-tokens-with-public-applications">
  ## Utiliser les jetons de délégation avec les applications publiques
</div>

Il y a une mise en garde importante à prendre en compte lorsque vous utilisez l&#39;endpoint de délégation avec des Applications publiques.

Si vous appelez l&#39;endpoint `/token` à partir d&#39;une Application publique, le ID Token sera obligatoirement signé avec `RS256`, même si l&#39;algorithme de signature JsonWebToken dans les paramètres de l&#39;application est configuré sur `HS256`.

Si vous appelez ensuite l&#39;endpoint de délégation avec ce ID Token, l&#39;opération échouera si l&#39;algorithme de signature JsonWebToken de l&#39;Application a été configuré sur `HS256`. En effet, délégation effectue la validation en fonction des paramètres de l&#39;Application, mais le ID Token a été émis avec un algorithme différent en raison de ce changement forcé.

Il est donc important que, si vous prévoyez utiliser délégation avec une Application publique, vous configuriez l&#39;algorithme de signature JsonWebToken de votre application sur `RS256`.
