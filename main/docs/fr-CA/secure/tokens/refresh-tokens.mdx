Auth0 émet un <Tooltip tip="Jeton d’accès : identifiant d’autorisation, sous la forme d’une chaîne opaque ou d’un JWT, utilisé pour accéder à une API." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=access+token">jeton d’accès</Tooltip> ou un <Tooltip tip="Jeton d’identité : identifiant destiné au client lui-même, plutôt qu’à l’accès à une ressource." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=ID+token">jeton d’identité</Tooltip> en réponse à une demande d’authentification. Vous pouvez utiliser des jetons d’accès pour effectuer des appels authentifiés à une API sécurisée, tandis que le jeton d’identité contient des attributs du profil utilisateur représentés sous forme de claims. Les deux sont des <Tooltip tip="Jeton d’identité : identifiant destiné au client lui-même, plutôt qu’à l’accès à une ressource." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=JSON+web+tokens">jetons Web JSON</Tooltip> (JWT) et ont donc une date d’expiration indiquée à l’aide du claim `exp`, ainsi que des mesures de sécurité comme des signatures. En général, un utilisateur a besoin d’un nouveau jeton d’accès lorsqu’il accède à une ressource pour la première fois, ou après l’expiration du jeton d’accès précédent qui lui avait été accordé.

Un <Tooltip tip="OAuth 2.0 : framework d’autorisation qui définit les protocoles et workflows d’autorisation." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=OAuth">OAuth</Tooltip> <Tooltip tip="OAuth 2.0 : framework d’autorisation qui définit les protocoles et workflows d’autorisation." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Refresh+Token">Refresh Token</Tooltip> est un artefact d’identification qu’OAuth peut utiliser pour obtenir un nouveau jeton d’accès sans interaction de l’utilisateur. Cela permet à l’<Tooltip tip="Authorization Server : serveur centralisé qui contribue à définir les limites de l’accès d’un utilisateur. Par exemple, votre serveur d’autorisation peut contrôler les données, les tâches et les fonctionnalités accessibles à un utilisateur." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Authorization+Server">Authorization Server</Tooltip> de raccourcir la durée de vie du jeton d’accès pour des raisons de sécurité, sans faire intervenir l’utilisateur lorsque le jeton d’accès expire. Vous pouvez demander de nouveaux jetons d’accès jusqu’à ce que le jeton d’actualisation soit placé dans la DenyList.

Il est important de maintenir le nombre de jetons d’actualisation à un niveau raisonnable pour que la gestion de ces identifiants demeure simple et sécuritaire. Les applications doivent stocker les jetons d’actualisation de façon sécurisée, car ils permettent essentiellement à un utilisateur de rester authentifié indéfiniment.

<Card title="Accès hors ligne">
  Si vous voulez permettre aux utilisateurs d’obtenir des jetons d’actualisation en mode hors ligne, vous pouvez activer le commutateur **Autoriser l’accès hors ligne** dans les [paramètres de l’API](/docs/fr-CA/get-started/apis/api-settings).
</Card>

Vous pouvez accroître la sécurité en utilisant la [rotation des jetons d’actualisation](/docs/fr-CA/secure/tokens/refresh-tokens/refresh-token-rotation), qui émet un nouveau jeton d’actualisation et invalide le jeton précédent à chaque requête envoyée à Auth0 pour obtenir un nouveau jeton d’accès. La rotation des jetons d’actualisation réduit le risque qu’un jeton d’actualisation soit compromis.

<div id="limitations">
  ### Limitations
</div>

* Auth0 limite le nombre de jetons d’actualisation actifs à 200 par utilisateur et par application. Cette limite s’applique uniquement aux jetons actifs. Si cette limite est atteinte et qu’un nouveau jeton d’actualisation est créé, le système révoque et supprime le plus ancien jeton de cet utilisateur pour cette application. Les jetons révoqués et expirés ne sont pas pris en compte dans cette limite. Pour consulter nos recommandations et nos pratiques exemplaires afin d’éviter un nombre excessif de jetons, lisez [bonne pratique relative au jeton](/docs/fr-CA/secure/tokens/token-best-practices).

* Après un [échange de jeton d’actualisation](/docs/fr-CA/secure/tokens/refresh-tokens/use-refresh-tokens), Auth0 ne met pas à jour les propriétés `event.refresh_token.device.last*` pour les jetons qui n’expirent pas. Pour savoir comment activer l’expiration des jetons d’actualisation, lisez [Configurer l’expiration des jetons d’actualisation](/docs/fr-CA/secure/tokens/refresh-tokens/configure-refresh-token-expiration).

<div id="enable-oidc-flag">
  ### Activer l’indicateur OIDC
</div>

Le comportement du jeton d’actualisation s’applique aux applications conformes à OIDC. Vous pouvez configurer une application pour qu’elle soit conforme à OIDC de l’une des façons suivantes :

1. En activant l’indicateur **OIDC Conformant** pour une application.
2. En transmettant la revendication `audience` au point de terminaison `/authorize` de l’Authentication API.

  ## Soutien pour le SDK
</div>

<div id="for-web-apps">
  ### Pour les applications Web
</div>

Les SDKs Auth0 prennent en charge les jetons d’actualisation, notamment :

* Node.js
* ASP.NET Core
* PHP
* Java

Pour obtenir la liste complète, consultez les [Quickstarts](/docs/fr-CA/quickstart/webapp).

<div id="for-single-page-apps">
  ### Pour les applications monopages
</div>

Assurer une authentification sécurisée dans les SPA présente un certain nombre de défis selon le cas d’utilisation de votre application. De nouveaux mécanismes de protection de la vie privée dans les navigateurs, comme Intelligent Tracking Prevention (ITP), nuisent à l’expérience utilisateur dans les SPA en bloquant l’accès aux cookies tiers.

Auth0 recommande d’utiliser la <Tooltip tip="Rotation des jetons d’actualisation : stratégie consistant à remplacer fréquemment les jetons d’actualisation afin de minimiser les vulnérabilités. Avec la rotation des jetons d’actualisation, chaque fois que votre application échange un jeton d’actualisation pour obtenir un nouveau jeton d’accès, Auth0 renvoie aussi un nouveau jeton d’actualisation." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=refresh+token+rotation">rotation des jetons d’actualisation</Tooltip>, qui offre une méthode sécurisée pour utiliser des jetons d’actualisation dans les SPA tout en donnant aux utilisateurs finaux un accès fluide aux ressources, sans les perturbations de l’expérience utilisateur causées par des technologies de protection de la vie privée des navigateurs comme ITP.

<div id="for-nativemobile-apps">
  ### Pour les applications natives/mobiles
</div>

Pour les applications natives, les jetons d’actualisation améliorent considérablement l’expérience d’authentification. L’utilisateur n’a besoin de s’authentifier qu’une seule fois, au moyen du processus d’authentification Web. Les réauthentifications suivantes peuvent ensuite se faire sans intervention de l’utilisateur, à l’aide du jeton d’actualisation.

Pour en savoir plus sur l’utilisation des jetons d’actualisation avec nos SDK mobiles, consultez :

* [Quickstarts mobiles/natifs](/docs/fr-CA/quickstart/native)
* [Lock Android: Actualiser les jetons JWT](/docs/fr-CA/libraries/lock-android/lock-android-refresh-jwt)
* [Lock iOS: Enregistrer et renouveler les jetons JWT](/docs/fr-CA/libraries/auth0-swift/auth0-swift-save-and-renew-tokens)
