L’utilisation de Customer Managed Keys d’Auth0 vous permet de configurer le cycle de vie de votre Tenant Master Key et de fournir votre propre Customer Provided Root Key pour remplacer l’Environment Root Key de votre tenant Auth0.

Customer Managed Keys offre deux méthodes pour gérer vos clés :

* **Control Your Own Key :** permet aux utilisateurs ayant le [Key Management Editor role](/docs/fr-CA/get-started/manage-dashboard-access/add-dashboard-users) de personnaliser le cycle de vie de la Tenant Master Key dans Auth0 KMS.
* **Bring Your Own Key :** permet aux utilisateurs ayant le [Key Management Editor role](/docs/fr-CA/get-started/manage-dashboard-access/add-dashboard-users) de remplacer l’Auth0 Environment Root Key et d’importer leur propre Wrapped Encryption Key dans le Hardware Security Module (HSM) d’Auth0 Cloud correspondant.

Pour en savoir plus sur les clés d’Auth0, [consultez la hiérarchie des clés d’Auth0](/docs/fr-CA/secure/highly-regulated-identity/customer-managed-keys#auth0-key-hierarchy).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  La fonctionnalité Customer Managed Keys est offerte aux utilisateurs ayant le Key Management Editor role. Ce rôle n’est pas attribué aux utilisateurs par défaut et doit être explicitement attribué à un Tenant Member. Pour en savoir plus, consultez [Add Tenant Members](/docs/fr-CA/get-started/manage-dashboard-access/add-dashboard-users).
</Callout>

<div id="control-your-own-key">
  ## Control Your Own Key
</div>

Vous pouvez contrôler le cycle de vie de votre Tenant Master Key au moyen du point de terminaison [Rekey](https://auth0.com/docs/api/management/v2/keys/post-encryption-rekey) de l’Auth0 <Tooltip tip="Management API : un produit qui permet aux clients d’effectuer des tâches administratives." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Management+API">Management API</Tooltip> pour :

* Effectuer la rotation de l’ancienne Tenant Master Key en la remplaçant par une nouvelle Tenant Master Key.
* Effectuer la rotation des Namespace Keys et les rechiffrer à l’aide de la nouvelle Tenant Master Key.

<div id="permissions">
  ### Permissions
</div>

Utilisez les permissions suivantes pour fournir l’accès aux endpoints Rekey :

| Permissions                                       | Description                                                                                                                                             |
| ------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `create:encryption_keys` `update:encryption_keys` | Les utilisateurs peuvent effectuer la rotation et rechiffrer la hiérarchie de clés d’un tenant Auth0 à l’aide de l’endpoint Rekey de la Management API. |

<div id="endpoint">
  ### Point de terminaison
</div>

Utilisez le point de terminaison [Rekey](https://auth0.com/docs/api/management/v2/keys/post-encryption-rekey) de la Management API pour faire pivoter la Tenant Master Key, ainsi que pour faire pivoter et rechiffrer les Namespace Keys.

* Faire pivoter la Tenant Master Key : désactive la Tenant Master Key actuellement active et crée une nouvelle Tenant Master Key.
* Faire pivoter toutes les Namespace Keys du tenant : désactive les clés actuellement actives et active de nouvelles clés.

  * Les nouvelles Namespace Keys sont utilisées pour les nouvelles opérations de chiffrement.
  * Les Namespace Keys désactivées sont utilisées pour déchiffrer les données chiffrées précédemment.
* Rechiffre toutes les Namespace Keys existantes avec la nouvelle Tenant Master Key.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Le point de terminaison est offert uniquement aux membres du tenant disposant de scopes d’autorisation précis dans Auth0. Par défaut, ces scopes sont accordés aux utilisateurs ayant le [Key Management Editor role](/docs/fr-CA/get-started/manage-dashboard-access/add-dashboard-users).
</Callout>

<div id="bring-your-own-key">
  ## Bring Your Own Key
</div>

<Warning>
  En important votre propre Customer Provided Root Key avec Bring Your Own Key, vous retirez implicitement à Auth0 l’autorisation de gérer le cycle de vie de la Customer Provided Root Key, à l’exception de sa suppression.
</Warning>

Bring Your Own Key permet aux Key Management Editors d’importer de façon sécurisée une Wrapped Encryption Key (Customer Provided Root Key) dans un Hardware Security Module (HSM) FIPS 140-2 L3 dans l’Auth0 Cloud correspondant.

Avec Bring Your Own Key, vous pouvez :

* Remplacer l’Environment Root Key générée par défaut par Auth0 par une nouvelle Customer Provided Root Key.
* Effectuer une rotation de la hiérarchie des clés et la rechiffrer avec la Customer Provided Root Key. Par exemple : créer et rechiffrer une nouvelle Tenant Master Key et une nouvelle Namespace Key.

Vous pouvez utiliser le [Dashboard](/docs/fr-CA/secure/highly-regulated-identity/customer-managed-keys/customer-managed-keys-dashboard) ou la [Management API](/docs/fr-CA/secure/highly-regulated-identity/customer-managed-keys/customer-managed-keys-management-api) pour activer Bring Your Own Key.

<div id="monitor-customer-managed-keys-log-events">
  ## Surveiller les événements de journalisation de Customer Managed Keys
</div>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Auth0 effectue automatiquement la rotation des clés de chiffrement du tenant une fois par année, ce qui ajoute les événements suivants aux journaux de votre tenant :

  * `kms_key_state_changed`
  * `kms_key_management_success`
</Callout>

Les opérations Customer Managed Keys ajoutent les événements de journalisation suivants dans vos [journaux du tenant](/docs/fr-CA/deploy-monitor/logs) :

Un code d’événement `sapi` indiquant :

* Créer la nouvelle clé de chiffrement
* Créer la clé d’enveloppement publique
* Importer la clé de chiffrement
* Supprimer la clé de chiffrement à l’aide de son ID de clé
* Rekey de la hiérarchie des clés

Un code d’événement `kms_key_management_success` indiquant une opération KMS réussie.

Un code d’événement `kms_key_management_failure` indiquant une opération KMS ayant échoué.

Un code d’événement `kms_key_state_changed` indiquant un changement d’état d’une clé KMS.

<div id="auth0-key-hierarchy">
  ## Hiérarchie des clés d’Auth0
</div>

Dans la couche applicative d’Auth0, Auth0 protège les secrets et les données des clients au moyen du chiffrement par enveloppe.

La hiérarchie de chiffrement par enveloppe d’Auth0 comprend les clés suivantes, chacune étant chiffrée à l’aide de la clé qui se trouve au-dessus d’elle. Le tableau ci-dessous résume la hiérarchie des clés :

| Clé                            | Algorithme                                                       | Stockage                                  |
| ------------------------------ | ---------------------------------------------------------------- | ----------------------------------------- |
| Environment Root Key           | RSA 2048 OAEP (Auth0 sur Azure)<br />AES-256-GCM (Auth0 sur AWS) | Module de sécurité matériel FIPS 140-2 L3 |
| Tenant Master Key              | AES-256-GCM                                                      | Base de données KMS d’Auth0               |
| Namespace Key                  | AES-256-GCM                                                      | Base de données KMS d’Auth0               |
| Clé de chiffrement des données | AES-256-GCM                                                      | Stockée à côté des données                |

<div id="environment-root-key">
  ### Environment Root Key
</div>

L’Environment Root Key se trouve au sommet de la hiérarchie et encapsule la Tenant Master Key afin d’empêcher qu’elle soit divulguée ou altérée à l’extérieur d’Auth0.

Une Auth0 Environment Root Key distincte est générée pour chaque environnement Auth0 et stockée dans un HSM adjacent. Les HSM sont déployés dans une configuration hautement disponible, répartie sur plusieurs zones géographiques. Cela signifie qu’ils basculeront vers une autre région en cas d’incident grave touchant une région entière.

L’Auth0 Environment Root Key est partagée par tous les tenants. Les clients peuvent utiliser la fonctionnalité Bring Your Own Key afin d’avoir une Environment Root Key dédiée à leur tenant.

Auth0 utilise les algorithms suivants pour encapsuler la Tenant Master Key avec l’Environment Root Key, selon votre Auth0 Cloud Service Provider :

* Auth0 sur Azure : RSA 2048 OAEP
* Auth0 sur AWS : AES 256 GCM

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  À partir de l’Auth0 Dashboard ou de la Management API, les Admins du tenant peuvent remplacer l’Auth0 Environment Root Key par leur propre Customer Provided Root Key.
</Callout>

<div id="tenant-master-keys">
  ### Tenant Master Keys
</div>

Chaque tenant possède une Tenant Master Key chiffrée, stockée dans l’Auth0 Key Management Service, qui sert à chiffrer les Namespace Keys.

L’algorithme utilisé pour chiffrer la Tenant Master Key est AES256 GCM.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Lorsqu’un Admin de tenant utilise l’Auth0 Dashboard ou la Management API pour fournir sa propre Customer Provided Root Key, une nouvelle Tenant Master Key est créée.
</Callout>

<div id="namespace-keys">
  ### Namespace Keys
</div>

Les Namespace Keys servent à séparer les clés de chiffrement utilisées à différentes fins au sein du tenant. Le nombre de Namespace Keys et leur utilisation sont configurés à l’interne par Auth0 et ne peuvent pas être personnalisés.

Le chiffrement et le déchiffrement des Namespace Keys nécessitent l’accès à la Tenant Master Key. Les Namespace Keys ne quittent jamais l’Auth0 Key Management Service et ne sont accessibles ni aux développeurs ni aux administrateurs.

Les Namespace Keys se trouvent dans l’Auth0 Key Management Service et sont chiffrées avec l’algorithme AES256 GCM.

<div id="data-encryption-keys">
  ### Clés de chiffrement des données
</div>

Auth0 Key Management Service génère de façon sécuritaire différentes clés de chiffrement des données pour les requêtes ultérieures de chiffrement des données. Auth0 Key Management Service optimise la sécurité et la performance en émettant régulièrement de nouvelles clés de chiffrement des données.

Le chiffrement et le déchiffrement des clés de chiffrement des données nécessitent l’accès à la Namespace Key qui leur est attribuée. Les clés de chiffrement des données ne peuvent pas être déchiffrées à l’extérieur d’Auth0 Key Management Service ni par les Key Management Editors.

Les clés de chiffrement des données sont stockées avec les données et sont chiffrées à l’aide de l’algorithme AES256 GCM.
