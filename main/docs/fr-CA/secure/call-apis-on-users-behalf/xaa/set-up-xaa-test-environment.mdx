Cette section explique comment configurer l’environnement de test de bout en bout pour la Resource App. En configurant votre Auth0 tenant comme Authorization Server de la Resource App, votre application SaaS peut commencer à accepter les requêtes ID-JAG entrantes sans nécessiter de modification du code. Votre API SaaS pourra ainsi générer des jetons d’accès en réponse à ces requêtes, ce qui permettra aux agents IA et à d’autres applications de consommer votre API en toute transparence.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Ce guide suppose que vous utilisez Okta comme fournisseur d’identité d’entreprise (IdP) et que vous avez un accès administratif à un Okta tenant que vous pouvez utiliser à des fins de test. Si ce n’est pas le cas, consultez [Créer et configurer votre Okta tenant](#create-and-configure-your-okta-tenant).
</Callout>

Pour configurer votre environnement de test de bout en bout pour la Resource App :

* Configurez et enregistrez votre Resource App : cela comprend la configuration de votre Auth0 tenant et l’enregistrement de votre application SaaS comme Resource App dans Okta. Pour en savoir plus, consultez [Configuration de la Resource App](#resource-app-setup).
* Configurez la Requesting App pour tester le flux de bout en bout : cela comprend l’enregistrement d’une Requesting App de test dans votre Auth0 tenant et la mise à jour d’Okta pour la lier à votre Resource App. Pour en savoir plus, consultez [Configuration de la Requesting App](#requesting-app-setup).
* Configurez la façon dont votre Auth0 tenant se fédère avec l’IdP d’entreprise de votre client : dans notre environnement de test, l’IdP d’entreprise sera votre Okta tenant de test, représentant l’un de vos clients d’entreprise. Pour en savoir plus, consultez [Fédération avec l’IdP d’entreprise et configuration de l’Organization](#federate-with-the-enterprise-idp-and-organization-configuration).
* Gérez Cross App Access dans Okta : configurez les connexions agent-application et application-application dans la Okta Admin Console. Pour en savoir plus, consultez [Gérer Cross App Access dans Okta](/docs/fr-CA/secure/call-apis-on-users-behalf/xaa/manage-xaa-in-okta).

<div id="create-and-configure-your-okta-tenant">
  ## Créez et configurez votre tenant Okta
</div>

Pour configurer votre environnement de test de bout en bout pour la Resource App, vous devez créer et configurer votre tenant Okta pour Cross App Access.

* Sur le [site Web des développeurs Okta](https://developer.okta.com/signup/), inscrivez-vous au plan Okta Integrator Free. Une fois l’inscription terminée, vous devriez être redirigé vers votre nouveau tenant Okta.
* Dans la console d’administration Okta, accédez à **Paramètres &gt; Fonctionnalités**. Sous **Fonctionnalités en accès anticipé**, activez **Cross App Access**.

<Frame>![](/docs/images/xaa/okta_enable_xaa.png)</Frame>

Pour configurer votre Resource App, vous devez :

<div id="create-the-api-in-auth0">
  ### Créer l’API dans Auth0
</div>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Si vous avez déjà créé une API personnalisée dans votre tenant Auth0, vous pouvez passer cette section.
</Callout>

Dans l’Auth0 Dashboard, [enregistrez une API personnalisée](/docs/fr-CA/get-started/auth0-overview/set-up-apis) qui représente votre API SaaS dans votre tenant Auth0.

<Frame>![](/docs/images/xaa/xaa_register_api.png)</Frame>

Après avoir créé l’API, vous pouvez aussi définir son audience comme **Default Audience** pour votre tenant Auth0 dans les [Tenant Settings](/docs/fr-CA/get-started/tenant-settings).

Vous pouvez également utiliser les [API Access Policies for Applications](/docs/fr-CA/get-started/apis/api-access-policies-for-applications) pour contrôler de façon granulaire quelles applications ont accès à votre API et pour quels scopes.

<div id="create-the-resource-app-in-auth0">
  ### Créer la Resource App dans Auth0
</div>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Si votre tenant Auth0 comporte déjà une ou plusieurs applications prêtes à se connecter à votre application SaaS, vous pouvez passer cette section.
</Callout>

Dans l’Auth0 Dashboard, [créez une application](/docs/fr-CA/get-started/auth0-overview/create-applications), comme une application Web traditionnelle, une SPA ou une application native, qui sert d’interface principale pour permettre aux utilisateurs finaux d’accéder aux fonctionnalités de votre application SaaS.

<div id="register-the-resource-app-in-okta">
  ### Enregistrer la Resource App dans Okta
</div>

Vous devez enregistrer votre application SaaS dans l’Okta Integration Network (OIN) pour qu’elle soit considérée comme une Resource App valide.

Pour enregistrer votre application SaaS en tant que Resource App dans Okta, vous avez deux options :

* Pour configurer rapidement un test, nous vous recommandons d’utiliser l’application Todo0, déjà enregistrée dans l’OIN. Dans l’Okta Admin Console, accédez à **Applications &gt; Applications &gt; Browse App Catalog** et recherchez `Todo0`. Sélectionnez-la, puis ajoutez l’intégration.

<Frame>![](/docs/images/xaa/xaa_browse_todo0_in_oin.png)</Frame>

* Vous pouvez aussi demander l’enregistrement d’une nouvelle application dans l’OIN depuis votre tenant Okta. Pour en savoir plus, consultez le [processus de soumission pour les intégrations SSO et SCIM](https://developer.okta.com/docs/guides/submit-app-overview/#submission-process-for-sso-and-scim-integrations). Pour accélérer le processus d’enregistrement, communiquez avec votre représentant Auth0 ou Okta.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Dans un environnement de production, vos clients d’entreprise installeront votre application SaaS à partir du catalogue OIN pendant la configuration de leur IdP.
</Callout>

De plus, vous devez fournir à Okta l’URL d’émetteur de votre tenant Auth0, qui est associée à votre Resource App. Les Requesting Apps utilisent l’URL d’émetteur pour demander à se connecter à votre Resource App. Pour en savoir plus, consultez [Tester le flux XAA de bout en bout](/docs/fr-CA/secure/call-apis-on-users-behalf/xaa/test-xaa-flow).

<div id="requesting-app-setup">
  ## Configuration de Requesting App
</div>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  En environnement de production, vous configurez chaque Requesting App une seule fois pour activer sa connexion à votre Resource App.
</Callout>

Pour configurer votre Requesting App, vous devez :

* [Créer la Requesting App dans Auth0](#create-the-requesting-app-in-auth0)
* [Enregistrer la Requesting App dans Okta](#register-the-requesting-app-in-okta)

<div id="create-the-requesting-app-in-auth0">
  ### Créer la Requesting App dans Auth0
</div>

Pour tester l’environnement de bout en bout, créez et enregistrez une application qui agit comme la Requesting App. L’application doit être un client confidentiel capable de stocker des secrets client, par exemple une application web.

Pour [créer une application](/docs/fr-CA/get-started/auth0-overview/create-applications) représentant la Requesting App dans votre tenant Auth0 :

* Accédez à **Applications &gt; Applications** et sélectionnez **Create Application**.
* Saisissez un nom et sélectionnez **Regular Web Application**.

<Frame>![](/docs/images/xaa/xaa_create_regular_web_app.png)</Frame>

* Une fois l’application créée, faites défiler jusqu’à **Settings** et activez la bascule **Cross App Access**.

<Frame>![](/docs/images/xaa/allow_xaa_auth0_app.png)</Frame>

Une fois votre application créée et configurée, vous devez fournir à Okta le `client_id` de l’application ainsi que l’URL de l’émetteur de votre tenant Auth0. Cela établit la connexion entre la Requesting App, identifiée par le `client_id`, et la Resource App, identifiée par l’URL de l’émetteur. Pour en savoir plus, consultez [Tester le flux XAA de bout en bout](/docs/fr-CA/secure/call-apis-on-users-behalf/xaa/test-xaa-flow).

<div id="register-the-requesting-app-in-okta">
  ### Enregistrer la Requesting App dans Okta
</div>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Dans un environnement de Production, le développeur de la Requesting App enregistre celle-ci dans l’Okta Integration Network (OIN). Les clients d’entreprise installeront la Requesting App à partir du catalogue OIN pendant la configuration de leur IdP.
</Callout>

Vous devez enregistrer l’application dans l’Okta Integration Network (OIN) pour qu’elle soit considérée comme une XAA Requesting App valide lorsque vous utilisez Okta comme IdP d’entreprise.

Pour enregistrer la Requesting App dans Okta, vous avez deux options :

* Pour effectuer rapidement une configuration de test, nous vous recommandons d’utiliser l’application Agent0, qui est déjà enregistrée dans l’OIN. Dans l’Okta Admin Console, accédez à **Applications &gt; Applications &gt; Browse App Catalog** et recherchez `Agent0`. Sélectionnez-la, puis ajoutez l’intégration.

<Frame>![](/docs/images/xaa/xaa_select_agent0_in_oin.png)</Frame>

* Vous pouvez aussi demander l’enregistrement d’une nouvelle application dans l’OIN. Pour en savoir plus, consultez le [processus de soumission pour les intégrations SSO et SCIM](https://developer.okta.com/docs/guides/submit-app-overview/#submission-process-for-sso-and-scim-integrations). Pour accélérer le processus d’enregistrement, communiquez avec votre représentant Auth0 ou Okta.

Comme la Requesting App authentifie les employés d’entreprise avec Okta, vous devez configurer la [sign-on policy](https://help.okta.com/en-us/content/topics/security/policies/policies-home.htm) de l’application dans Okta.

1. Accédez à **Applications &gt; Applications** et sélectionnez l’application (par ex. Agent0).
2. Sous **Sign On**, sélectionnez **Edit** et ajoutez l’URL de rappel de la Requesting App dans le champ **Redirect URI**. Ajustez la valeur de **Redirect URI** selon l’application de test que vous souhaitez utiliser. Pour en savoir plus, consultez [Test the end-to-end XAA flow](/docs/fr-CA/secure/call-apis-on-users-behalf/xaa/test-xaa-flow).
3. Sélectionnez **Save**.

<Frame>![](/docs/images/xaa/agent0_sign_on_policy.png)</Frame>

Enfin, autorisez votre utilisateur de test à se connecter à la Requesting App dans Okta.

Dans l’Okta Admin Console :

1. Accédez à **Applications** et sélectionnez l’application (par ex. Agent0).
2. Sélectionnez **Assign &gt; Assign to People** et choisissez votre utilisateur de test.
3. Sélectionnez **Save**.

<div id="federate-with-the-enterprise-idp-and-organization-configuration">
  ## Fédérer avec l’IdP d’entreprise et la configuration de l’organisation
</div>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Dans un environnement de production, vous configurez chacun de vos clients d’entreprise une seule fois pour le fédérer à votre tenant Auth0. Auth0 ajoutera la prise en charge du [SSO en libre-service](/docs/fr-CA/authenticate/enterprise-connections/self-service-enterprise-configuration) dans de futures versions, ce qui vous permettra de déléguer la configuration XAA à vos clients d’entreprise dans le cadre de la configuration du SSO.
</Callout>

Vous devez fédérer votre tenant Auth0, qui agit comme serveur d’autorisation pour votre Resource App, avec le tenant Okta de votre client d’entreprise. Cette fédération établit une relation de confiance cryptographique, ce qui permet à votre application de valider et d’accepter les assertions signées (ID-JAG) émises par l’IdP du client.

Pour tester le flux XAA de bout en bout pour plusieurs clients d’entreprise connectés à votre Resource App, vous pouvez répéter les étapes de cette section pour plusieurs connexions d’entreprise Okta Workforce dans votre tenant Auth0. Chaque connexion correspond à un tenant de test Okta distinct, et chaque tenant représente un client d’entreprise différent.

<div id="configure-an-okta-workforce-enterprise-connection">
  ### Configurer une connexion Okta Workforce Enterprise
</div>

Utilisez le `client_id` et le `client_secret` de votre Resource App pour [créer une connexion Okta Workforce Enterprise](/docs/fr-CA/authenticate/identity-providers/enterprise-identity-providers/okta) dans votre tenant Auth0.

Lorsque vous créez la connexion Okta Workforce Enterprise, activez le rôle **Cross App Access - Resource Application**. Cela permet à votre Resource App d’accepter les ID-JAG émis par l’IdP d’entreprise associé à cette connexion, dans ce cas-ci, votre tenant Okta.

<Frame>![](/docs/images/xaa/xaa_new_okta_workforce_connection.png)</Frame>

Après avoir créé la connexion Okta Workforce Enterprise, copiez l’URL de rappel fournie par Auth0 dans les paramètres de la connexion. Vous aurez besoin de cette URL de rappel pour configurer les politiques de connexion de la Resource App dans votre tenant Okta.

Dans l’Okta Admin Console :

1. Accédez à **Applications &gt; Applications** et sélectionnez l’application (par ex. Todo0).
2. Dans les paramètres **Sign On**, sélectionnez **Edit** et ajoutez l’URL de rappel dans le champ **Redirect URI**.
3. Sélectionnez **Save**.

<Frame>![](/docs/images/xaa/xaa_advanced_sign_on_settings.png)</Frame>

Pour tester la connexion Okta Workforce Enterprise, créez un utilisateur de test et donnez-lui la permission de se connecter à la Requesting App.

Dans l’Okta Admin Console :

* Accédez à **Applications** et sélectionnez l’application (par ex. Agent0).
* Sélectionnez **Assign &gt; Assign to People** et choisissez votre utilisateur de test.
* Sélectionnez **Save**.

Dans l’Auth0 Dashboard :

* Accédez à **Authentication &gt; Enterprise &gt; Okta Workforce**:
  * Ouvrez la connexion Okta Workforce Enterprise que vous avez créée et sélectionnez l’onglet **Applications**. Ensuite, activez la Requesting App que vous avez créée pour cette connexion.
  * Revenez à la liste des connexions Okta Workforce. Sélectionnez les trois points à droite de votre connexion, puis sélectionnez **Try**. Vous serez redirigé vers votre tenant Okta pour vous authentifier et terminer la connexion avec votre utilisateur de test.

<Frame>![](/docs/images/xaa/xaa_create_okta_workforce_connection.png)</Frame>

<div id="configure-an-organization">
  ### Configurer une organisation
</div>

Au besoin, si vous souhaitez qu’un client d’entreprise utilise Organizations, [créez une organisation](/docs/fr-CA/manage-users/organizations/configure-organizations/create-organizations) et [activez la connexion Enterprise Okta Workforce](/docs/fr-CA/manage-users/organizations/configure-organizations/enable-connections) pour cette organisation. Cela associe automatiquement les jetons d’accès générés à l’aide de XAA, pour cette connexion, au `org_id` correspondant si l’utilisateur cible est membre de l’organisation.

<Frame>![](/docs/images/xaa/xaa_enable_connection.png)</Frame>

Vous pouvez aussi configurer le [comportement d’organisation](/docs/fr-CA/manage-users/organizations/configure-organizations/define-organization-behavior) de la Requesting App pour définir s’il est obligatoire ou autorisé d’utiliser Organizations. Nous vous recommandons de commencer vos tests avec **Both**, ce qui permet aux utilisateurs de se connecter comme membre d’une organisation ou de s’inscrire avec un compte personnel.

<Frame>![](/docs/images/xaa/xaa_organizations_both.png)</Frame>
