Le [sender constraining](/docs/fr-CA/secure/sender-constraining) est un mécanisme de sécurité d’<Tooltip tip="OAuth 2.0 : framework d'autorisation qui définit les protocoles et les workflows d'autorisation." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=OAuth+2.0">OAuth 2.0</Tooltip> et d’<Tooltip tip="OAuth 2.0 : framework d'autorisation qui définit les protocoles et les workflows d'autorisation." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=OpenID">OpenID</Tooltip> Connect (OIDC) qui lie de manière cryptographique les jetons d’accès et les <Tooltip tip="Refresh Token : jeton utilisé pour obtenir un jeton d’accès renouvelé sans obliger les utilisateurs à se reconnecter." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=refresh+tokens">jetons d’actualisation</Tooltip> à l’application cliente qui les a obtenus, empêchant ainsi le vol et l’utilisation abusive des jetons.

Auth0 prend en charge le mTLS sender constraining et Demonstrating Proof-of-Possession (DPoP). Si vous activez le sender constraining pour une application cliente, vous devez aussi l’imposer pour le <Tooltip tip="Resource Server : serveur hébergeant des ressources protégées. Les serveurs de ressources acceptent les requêtes visant des ressources protégées et y répondent." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=resource+server">serveur de ressources</Tooltip> auquel vous faites des appels d’API.

Pour configurer le sender constraining dans Auth0, vous devez :

* [Configurer le sender constraining pour une application cliente](#configure-sender-constraining-for-a-client-application)
* [Configurer le sender constraining pour un serveur de ressources](#configure-sender-constraining-for-a-resource-server)

<div id="how-it-works">
  ## Fonctionnement
</div>

<Tooltip tip="Jeton d’accès : justificatif d’autorisation, sous la forme d’une chaîne opaque ou d’un JWT, utilisé pour accéder à une API." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Access+tokens">Jetons d’accès</Tooltip> sont liés à l’expéditeur dans Auth0 selon la façon dont vous configurez le Sender Constraining pour votre application cliente et votre serveur de ressources :

1. **Audience demandée :** Dans une demande de jeton, le fait que l’audience demandée soit uniquement `/userinfo`, qu’elle soit destinée à être utilisée seulement avec le point de terminaison [`/userinfo`](https://auth0.com/docs/api/authentication/user-profile/get-user-info), ou qu’il s’agisse d’une API personnalisée pouvant inclure `/userinfo` lorsque vous demandez aussi le scope `openid`, a une incidence sur le fait que les jetons d’accès soient liés à l’expéditeur.
2. **Application cliente :** Le fait que vous définissiez le Sender Constraining comme `required` pour l’application cliente.
3. **Serveur de ressources :** Le fait que vous configuriez le Sender Constraining pour le serveur de ressources :

   1. `none` : Vous n’avez pas configuré le Sender Constraining pour le serveur de ressources.
   2. `allowed` : Vous avez configuré le Sender Constraining pour le serveur de ressources en définissant une méthode de Sender Constraining.
   3. `required` : Vous avez configuré le Sender Constraining comme obligatoire pour le serveur de ressources, ce qui signifie que les jetons d’accès doivent être liés à l’expéditeur pour une application. Vous pouvez exiger le Sender Constraining pour toutes les applications ou seulement pour les applications publiques. Une méthode de Sender Constraining est requise.

<Callout icon="lightbulb" color="#0EA5E9" iconType="regular">
  Lorsque vous utilisez mTLS comme méthode de Sender Constraining, le Sender Constraining est toujours obligatoire pour toutes les applications. Les applications publiques ne sont pas prises en charge avec mTLS; vous ne pouvez donc pas limiter le Sender Constraining par mTLS aux seuls clients publics.
</Callout>

* **Preuve de possession :** Le fait que l’application cliente ait envoyé une assertion de preuve de possession dans la demande de jeton :

  * liaison à l’expéditeur mTLS : La preuve de possession est démontrée par la présentation réussie par le client d’une clé privée précise (associée à un certificat client) lors de l’établissement de la connexion TLS.
  * DPoP : La preuve de possession est établie lorsque le client crée un DPoP Proof JWT en le signant cryptographiquement avec sa clé privée et en incluant le DPoP Proof JWT `DPoP` dans l’en-tête HTTP DPoP de chaque demande où le jeton d’accès associé est utilisé.

Le tableau suivant décrit comment les jetons d’accès sont émis et s’ils sont liés à l’expéditeur en fonction de divers paramètres de demande du client et des configurations du serveur de ressources Auth0 :

| Type d’audience demandée                        | Le client exige une PoP ? | Preuve de possession (PoP) envoyée par le client ? | Politique du serveur de ressources Auth0 : None | Politique du serveur de ressources Auth0 : Allowed (Not Required) | Politique du serveur de ressources Auth0 : Required |
| ----------------------------------------------- | ------------------------- | -------------------------------------------------- | ----------------------------------------------- | ----------------------------------------------------------------- | --------------------------------------------------- |
| Userinfo seulement                              | Non                       | Non                                                | Émis, non lié à l’expéditeur                    | S.O.                                                              | S.O.                                                |
| Userinfo seulement                              | Non                       | Oui                                                | Émis, lié à l’expéditeur                        | S.O.                                                              | S.O.                                                |
| Userinfo seulement                              | Oui                       | Non                                                | Non émis                                        | S.O.                                                              | S.O.                                                |
| Userinfo seulement                              | Oui                       | Oui                                                | Émis, lié à l’expéditeur                        | S.O.                                                              | S.O.                                                |
| Audience personnalisée (peut contenir Userinfo) | Non                       | Non                                                | Émis, non lié à l’expéditeur                    | Émis, non lié à l’expéditeur                                      | Non émis                                            |
| Audience personnalisée (peut contenir Userinfo) | Non                       | Oui                                                | Émis, non lié à l’expéditeur                    | Émis, lié à l’expéditeur                                          | Émis, lié à l’expéditeur                            |

<div id="configure-sender-constraining-for-a-client-application">
  ## Configurer Sender Constraining pour une application cliente
</div>

Lorsque vous exigez Sender Constraining pour une application cliente, les jetons d’accès sont liés à cette application. Auth0 vérifie les requêtes pour s’assurer que seule l’application qui a demandé le jeton peut l’utiliser pour accéder à la ressource associée.

Une fois qu’une application cliente est configurée pour exiger Sender Constraining, vous pouvez définir la méthode à utiliser, soit mTLS ou DPoP, lors de la configuration de votre serveur de ressources.

Vous pouvez configurer Sender Constraining pour une application cliente à l’aide du <Tooltip tip="Auth0 Dashboard : le principal produit d’Auth0 pour configurer vos services." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip> ou de l’<Tooltip tip="Management API : un produit qui permet aux clients d’effectuer des tâches administratives." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Management+API">Management API</Tooltip>.

<Tabs>
  <Tab title="Auth0 Dashboard">
    1. Accédez à [Dashboard &gt; Applications &gt; Applications](https://manage.auth0.com/#/applications). Sélectionnez l’application que vous souhaitez configurer.
    2. Sous **Paramètres**, faites défiler jusqu’à **Token Sender-Constraining**.
    3. Activez l’option **Require Sender Constraining**. Désactivez-la pour retirer l’exigence de Sender Constraining pour l’application.

    <Frame>![](/docs/images/cdy7uua7fh8z/3BoGW9NEYKPBkfk2tPBL7l/0f884ac3f64f17b23fd6a0e9436bb472/Screenshot_2024-07-23_at_1.30.19_PM.png)</Frame>
  </Tab>

  <Tab title="Management API">
    Pour configurer Sender Constraining pour un client, utilisez la [Management API](https://auth0.com/docs/api/management/v2).

    Pour exiger Sender Constraining pour un client, envoyez une requête PATCH pour [mettre à jour les paramètres du client](https://auth0.com/docs/api/management/v2/clients/patch-clients-by-id). Définissez le paramètre `require_proof_of_possession` à `true`.

    ```bash lines
    curl -L -X PATCH 'https://{YOUR_DOMAIN}/api/v2/clients/{YOUR_CLIENT_ID}' \
    -H 'Authorization: Bearer {YOUR_MANAGEMENT_API_TOKEN}' \
    -H 'Content-Type: application/json' \
    -d '{"require_proof_of_possession": true}'
    ```

    Pour retirer l’exigence de Sender Constraining, définissez le paramètre `require_proof_of_possession` à `false`.

    ```bash lines
    curl -L -X PATCH 'https://{YOUR_DOMAIN}/api/v2/clients/{YOUR_CLIENT_ID}' \
    -H 'Authorization: Bearer {YOUR_MANAGEMENT_API_TOKEN}' \
    -H 'Content-Type: application/json' \
    -d '{"require_proof_of_possession": false}'
    ```
  </Tab>
</Tabs>

<div id="configure-sender-constraining-for-a-resource-server">
  ## Configurer Sender Constraining pour un serveur de ressources
</div>

Les jetons d’accès émis par Auth0 peuvent être liés à l’expéditeur (c.-à-d. l’application cliente) qui doit accéder aux API hébergées sur un serveur de ressources.

Vous pouvez configurer Sender Constraining pour un serveur de ressources à l’aide de l’Auth0 Dashboard ou de la Management API.

<Tabs>
  <Tab title="Auth0 Dashboard">
    Pour activer Token Binding ou sender constraining, configurez les **API Settings** de votre API.

    1. Accédez à [Auth0 Dashboard &gt; Applications &gt; APIs](https://manage.auth0.com/#/apis).
    2. Sélectionnez l’API que vous souhaitez configurer.
    3. Sous l’onglet **Settings**, repérez la section **Token Sender-Constraining**.
    4. Configurez les éléments suivants :

       1. Méthode de sender constraining :

          1. **None:** N’activez aucune méthode de sender constraining pour votre serveur de ressources.
          2. **mTLS**: Activez mTLS comme méthode de sender constraining pour votre serveur de ressources.
          3. **DPoP:** Activez DPoP comme méthode de sender constraining pour votre serveur de ressources.

       2. Exiger Token Sender Constraining : sélectionnez la policy de sender constraining qui s’applique à cette API :

          1. **Always:** Exigez sender constraining pour toutes les applications. C’est la seule option offerte avec mTLS.
          2. **For Public Applications:** Exigez sender constraining pour les applications publiques seulement. Cette option n’est pas offerte avec mTLS.
          3. **Never:** Sender constraining n’est pas requis.

    <Frame>![Auth0 Dashboard > APIs > Settings > Token binding](/docs/images/sender-constraining/method_is_dpop.png)</Frame>
  </Tab>

  <Tab title="Management API">
    Pour activer Sender Constraining avec la Management API, envoyez une requête PATCH pour [mettre à jour le serveur de ressources](https://auth0.com/docs/api/management/v2/resource-servers/patch-resource-servers-by-id). Définissez les paramètres de l’objet `proof_of_possession` comme suit :

    | Parameter      | Description                                                                                                                                                                                                                                                                                                                                               |
    | -------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
    | `mechanism`    | Définit la méthode de sender constraining : `none`, `mtls` ou `dpop`.                                                                                                                                                                                                                                                                                     |
    | `required`     | Obligatoire. Lorsqu’il est défini à `true`, les jetons d’accès émis à une application pour cette API seront liés à cette application. Utilisez `required_for` pour préciser à quels types d’application cette exigence s’applique. Lorsqu’il est défini à `false`, sender constraining n’est pas requis.                                                   |
    | `required_for` | Filtre les types d’application pour lesquels sender constraining est requis lorsque `required` est défini à `true` : `all_clients` (toutes les applications) ou `public_clients` (applications publiques seulement). La valeur par défaut est `all_clients` s’il n’est pas précisé. Lorsque vous utilisez mTLS, seule la valeur `all_clients` est valide. |

    L’exemple de code suivant montre un request body qui configure un serveur de ressources pour mTLS Sender Constraining. Comme mTLS prend seulement en charge `all_clients` pour `required_for` (qui est la valeur par défaut), il n’est pas nécessaire de le préciser :

    ```bash lines
    curl -L -X PATCH 'https://{YOUR_DOMAIN}/api/v2/resource-servers/{YOUR_RESOURCE_SERVER_ID}' \
    -H 'Authorization: Bearer {YOUR_MANAGEMENT_API_TOKEN}' \
    -H 'Content-Type: application/json' \
    -H 'Accept: application/json' \
    -d '{
      "proof_of_possession": {
        "mechanism": "mtls",
        "required": true
      }
    }'
    ```

    L’exemple de code suivant montre un request body qui configure un serveur de ressources pour DPoP Sender Constraining, requis pour les applications publiques seulement :

    ```bash lines
    curl -L -X PATCH 'https://{YOUR_DOMAIN}/api/v2/resource-servers/{YOUR_RESOURCE_SERVER_ID}' \
    -H 'Authorization: Bearer {YOUR_MANAGEMENT_API_TOKEN}' \
    -H 'Content-Type: application/json' \
    -H 'Accept: application/json' \
    -d '{
      "proof_of_possession": {
        "mechanism": "dpop",
        "required": true,
        "required_for": "public_clients"
      }
    }'
    ```

    L’exemple de code suivant configure DPoP Sender Constraining comme étant requis pour toutes les applications. Comme `all_clients` est la valeur par défaut, `required_for` peut être omis :

    ```bash lines
    curl -L -X PATCH 'https://{YOUR_DOMAIN}/api/v2/resource-servers/{YOUR_RESOURCE_SERVER_ID}' \
    -H 'Authorization: Bearer {YOUR_MANAGEMENT_API_TOKEN}' \
    -H 'Content-Type: application/json' \
    -H 'Accept: application/json' \
    -H 'Accept: application/json' \
    -d '{
      "proof_of_possession": {
        "mechanism": "dpop",
        "required": true
      }
    }'
    ```

    L’exemple de code suivant désactive sender constraining pour un serveur de ressources (l’équivalent de sélectionner &quot;Never&quot; dans le Dashboard) :

    ```bash lines
    curl -L -X PATCH 'https://{YOUR_DOMAIN}/api/v2/resource-servers/{YOUR_RESOURCE_SERVER_ID}' \
    -H 'Authorization: Bearer {YOUR_MANAGEMENT_API_TOKEN}' \
    -H 'Content-Type: application/json' \
    -H 'Accept: application/json' \
    -d '{
      "proof_of_possession": null 
    }'
    ```
  </Tab>
</Tabs>
