[Sender constraining](/docs/fr-CA/secure/sender-constraining) est un mécanisme de sécurité d’<Tooltip tip="OAuth 2.0 : cadre d’autorisation qui définit les protocoles et les workflows d’autorisation." cta="Consulter le glossaire" href="/docs/fr-CA/glossary?term=OAuth+2.0">OAuth 2.0</Tooltip> et d’<Tooltip tip="OAuth 2.0 : cadre d’autorisation qui définit les protocoles et les workflows d’autorisation." cta="Consulter le glossaire" href="/docs/fr-CA/glossary?term=OpenID">OpenID</Tooltip> Connect (OIDC) qui lie de façon cryptographique les jetons d’accès et les <Tooltip tip="Refresh Token : jeton utilisé pour obtenir un jeton d’accès renouvelé sans obliger les utilisateurs à se reconnecter." cta="Consulter le glossaire" href="/docs/fr-CA/glossary?term=refresh+tokens">jetons d’actualisation</Tooltip> à l’application cliente précise qui les a obtenus, ce qui empêche le vol et leur utilisation abusive.

Auth0 prend en charge mTLS sender constraining et Demonstrating Proof-of-Possession (DPoP). Si vous activez sender constraining pour une application cliente, vous devez aussi l’imposer pour le <Tooltip tip="Resource Server : serveur hébergeant des ressources protégées. Les serveurs de ressources acceptent les requêtes visant des ressources protégées et y répondent." cta="Consulter le glossaire" href="/docs/fr-CA/glossary?term=resource+server">serveur de ressources</Tooltip> auquel vous envoyez des appels d’API.

Pour configurer sender constraining dans Auth0, vous devez :

* [Configurer sender constraining pour une application cliente](#configure-sender-constraining-for-a-client-application)
* [Configurer sender constraining pour un serveur de ressources](#configure-sender-constraining-for-a-resource-server)

<div id="how-it-works">
  ## Fonctionnement
</div>

<Tooltip tip="Jeton d’accès : justificatif d’autorisation, sous la forme d’une chaîne opaque ou d’un JWT, utilisé pour accéder à une API." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Access+tokens">Les jetons d’accès</Tooltip> sont liés à l’expéditeur dans Auth0 selon la façon dont vous configurez le sender constraining pour votre application cliente et votre serveur de ressources :

1. **Audience demandée :** Dans une demande de jeton, le fait que l’audience demandée soit uniquement `/userinfo`, qu’elle soit destinée à être utilisée seulement avec le endpoint [`/userinfo`](https://auth0.com/docs/api/authentication/user-profile/get-user-info), ou qu’il s’agisse d’une API personnalisée pouvant inclure `/userinfo` lorsque vous demandez aussi la portée openid, a une incidence sur le fait que les jetons d’accès soient liés à l’expéditeur.
2. **Application cliente :** Selon que vous définissiez ou non le sender constraining sur `required` pour l’application cliente.
3. **Serveur de ressources :** Selon que vous configuriez ou non le sender constraining pour le serveur de ressources :

   1. `none`: Vous n’avez pas configuré le sender constraining pour le serveur de ressources.
   2. `allowed`: Vous avez configuré le sender constraining pour le serveur de ressources en définissant une méthode de sender constraining.
   3. `required`: Vous avez rendu le sender constraining obligatoire pour le serveur de ressources, ce qui signifie que les jetons d’accès doivent être liés à l’expéditeur pour une application. Vous pouvez exiger le sender constraining pour toutes les applications ou seulement pour les applications publiques. Nécessite une méthode de sender constraining.

<Callout icon="lightbulb" color="#0EA5E9" iconType="regular">
  Lorsque vous utilisez mTLS comme méthode de sender constraining, le sender constraining est toujours obligatoire pour toutes les applications. Les applications publiques ne sont pas prises en charge avec mTLS; vous ne pouvez donc pas limiter le mTLS sender constraining aux seuls clients publics.
</Callout>

* **Proof-of-Possession :** Selon que l’application cliente a envoyé ou non une assertion de preuve de possession dans la demande de jeton :

  * mTLS sender constraining: La preuve de possession est démontrée lorsque le client présente avec succès une clé privée précise (associée à un certificat client) pendant le handshake TLS.
  * DPoP: La preuve de possession est établie lorsque le client crée un DPoP Proof JWT en le signant cryptographiquement avec sa clé privée et en incluant le DPoP Proof JWT `DPoP` dans l’en-tête HTTP DPoP de chaque request où le jeton d’accès associé est utilisé.

Le tableau suivant décrit comment les jetons d’accès sont émis et s’ils sont liés à l’expéditeur selon différents paramètres de demande du client et différentes configurations du serveur de ressources Auth0 :

| Type d’audience demandée                        | Le client exige une PoP ? | Proof-of-Possession (PoP) envoyée par le client ? | Policy du serveur de ressources Auth0 : None | Policy du serveur de ressources Auth0 : Allowed (Not Required) | Policy du serveur de ressources Auth0 : Required |
| ----------------------------------------------- | ------------------------- | ------------------------------------------------- | -------------------------------------------- | -------------------------------------------------------------- | ------------------------------------------------ |
| Userinfo seulement                              | Non                       | Non                                               | Émis, non lié à l’expéditeur                 | S/O                                                            | S/O                                              |
| Userinfo seulement                              | Non                       | Oui                                               | Émis, lié à l’expéditeur                     | S/O                                                            | S/O                                              |
| Userinfo seulement                              | Oui                       | Non                                               | Non émis                                     | S/O                                                            | S/O                                              |
| Userinfo seulement                              | Oui                       | Oui                                               | Émis, lié à l’expéditeur                     | S/O                                                            | S/O                                              |
| Audience personnalisée (peut contenir Userinfo) | Non                       | Non                                               | Émis, non lié à l’expéditeur                 | Émis, non lié à l’expéditeur                                   | Non émis                                         |
| Audience personnalisée (peut contenir Userinfo) | Non                       | Oui                                               | Émis, non lié à l’expéditeur                 | Émis, lié à l’expéditeur                                       | Émis, lié à l’expéditeur                         |

<div id="configure-sender-constraining-for-a-client-application">
  ## Configurer sender constraining pour une application cliente
</div>

Lorsque vous exigez sender constraining pour une application cliente, les jetons d’accès sont liés à cette application. Auth0 vérifie les requêtes afin de s’assurer que seule l’application qui a demandé le jeton peut l’utiliser pour accéder à la ressource associée.

Une fois que vous avez configuré une application cliente pour exiger sender constraining, vous pouvez définir la méthode de sender constraining, soit mTLS, soit DPoP, lors de la configuration de votre serveur de ressources.

Vous pouvez configurer sender constraining pour une application cliente à l’aide du <Tooltip tip="Auth0 Dashboard : le produit principal d’Auth0 pour configurer vos services." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip> ou de la <Tooltip tip="Management API : un produit qui permet aux clients d’effectuer des tâches administratives." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Management+API">Management API</Tooltip>.

<Tabs>
  <Tab title="Auth0 Dashboard">
    1. Accédez à [Dashboard &gt; Applications &gt; Applications](https://manage.auth0.com/#/applications). Sélectionnez l’application que vous souhaitez configurer.
    2. Sous **Settings**, faites défiler jusqu’à **Token Sender-Constraining**.
    3. Activez la bascule pour **Require Sender Constraining**. Désactivez-la pour supprimer l’exigence de Sender Constraining pour l’application.

    <Frame>![](/docs/images/cdy7uua7fh8z/3BoGW9NEYKPBkfk2tPBL7l/0f884ac3f64f17b23fd6a0e9436bb472/Screenshot_2024-07-23_at_1.30.19_PM.png)</Frame>
  </Tab>

  <Tab title="Management API">
    Pour configurer un client pour Sender Constraining, utilisez la [Management API](https://auth0.com/docs/api/management/v2).

    Pour exiger Sender Constraining pour un client, envoyez une requête PATCH pour [mettre à jour les paramètres du client](https://auth0.com/docs/api/management/v2/clients/patch-clients-by-id). Définissez le paramètre `require_proof_of_possession` sur `true`.

    ```bash lines
    curl -L -X PATCH 'https://{YOUR_DOMAIN}/api/v2/clients/{YOUR_CLIENT_ID}' \
    -H 'Authorization: Bearer {YOUR_MANAGEMENT_API_TOKEN}' \
    -H 'Content-Type: application/json' \
    -d '{"require_proof_of_possession": true}'
    ```

    Pour supprimer l’exigence de Sender Constraining, définissez le paramètre `require_proof_of_possession` sur `false`.

    ```bash lines
    curl -L -X PATCH 'https://{YOUR_DOMAIN}/api/v2/clients/{YOUR_CLIENT_ID}' \
    -H 'Authorization: Bearer {YOUR_MANAGEMENT_API_TOKEN}' \
    -H 'Content-Type: application/json' \
    -d '{"require_proof_of_possession": false}'
    ```
  </Tab>
</Tabs>

<div id="configure-sender-constraining-for-a-resource-server">
  ## Configurer Sender Constraining pour un serveur de ressources
</div>

Les jetons d’accès émis par Auth0 peuvent être liés à l’expéditeur (c.-à-d. l’application cliente) qui doit accéder aux API d’un serveur de ressources.

Vous pouvez configurer Sender Constraining pour un serveur de ressources au moyen de l’Auth0 Dashboard ou de la Management API.

<Tabs>
  <Tab title="Auth0 Dashboard">
    Pour activer Token Binding ou le sender constraining, configurez les **API Settings** de votre API.

    1. Accédez à [Auth0 Dashboard &gt; Applications &gt; APIs](https://manage.auth0.com/#/apis).
    2. Sélectionnez l’API que vous souhaitez configurer.
    3. Dans l’onglet **Settings**, repérez la section **Token Sender-Constraining**.
    4. Configurez les éléments suivants :

       1. Méthode de Sender Constraining :

          1. **None:** N’activez aucune méthode de sender constraining pour votre serveur de ressources.
          2. **mTLS**: Activez mTLS comme méthode de sender constraining pour votre serveur de ressources.
          3. **DPoP:** Activez DPoP comme méthode de sender constraining pour votre serveur de ressources.

       2. Exiger le Token Sender Constraining : Sélectionnez la policy de sender constraining à appliquer à cette API :

          1. **Always:** Exigez le sender constraining pour toutes les applications. Il s’agit de la seule option offerte lorsque vous utilisez mTLS.
          2. **For Public Applications:** Exigez le sender constraining uniquement pour les applications publiques. Cette option n’est pas offerte lorsque vous utilisez mTLS.
          3. **Never:** Le sender constraining n’est pas requis.

    <Frame>![Auth0 Dashboard > APIs > Settings > Token binding](/docs/images/sender-constraining/method_is_dpop.png)</Frame>
  </Tab>

  <Tab title="Management API">
    Pour activer Sender Constraining avec la Management API, envoyez une requête PATCH pour [mettre à jour le serveur de ressources](https://auth0.com/docs/api/management/v2/resource-servers/patch-resource-servers-by-id). Définissez les paramètres de l’objet `proof_of_possession` comme suit :

    | Paramètre      | Description                                                                                                                                                                                                                                                                                                                                       |
    | -------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
    | `mechanism`    | Définit la méthode de sender constraining : `none`, `mtls` ou `dpop`.                                                                                                                                                                                                                                                                             |
    | `required`     | Obligatoire. Lorsque ce paramètre est défini sur `true`, les jetons d’accès émis à une application pour cette API seront liés à cette application. Utilisez `required_for` pour préciser à quels types d’application cette exigence s’applique. Lorsque ce paramètre est défini sur `false`, le sender constraining n’est pas requis.             |
    | `required_for` | Filtre les types d’application pour lesquels le sender constraining est requis lorsque `required` est défini sur `true` : `all_clients` (toutes les applications) ou `public_clients` (applications publiques seulement). La valeur par défaut est `all_clients` si elle n’est pas précisée. Avec mTLS, seule la valeur `all_clients` est valide. |

    L’exemple de code suivant montre un corps de requête qui configure un serveur de ressources pour mTLS Sender Constraining. Puisque mTLS ne prend en charge que `all_clients` pour `required_for` (qui est la valeur par défaut), il n’est pas nécessaire de le préciser :

    ```bash lines
    curl -L -X PATCH 'https://{YOUR_DOMAIN}/api/v2/resource-servers/{YOUR_RESOURCE_SERVER_ID}' \
    -H 'Authorization: Bearer {YOUR_MANAGEMENT_API_TOKEN}' \
    -H 'Content-Type: application/json' \
    -H 'Accept: application/json' \
    -d '{
      "proof_of_possession": {
        "mechanism": "mtls",
        "required": true
      }
    }'
    ```

    L’exemple de code suivant montre un corps de requête qui configure un serveur de ressources pour DPoP Sender Constraining, requis uniquement pour les applications publiques :

    ```bash lines
    curl -L -X PATCH 'https://{YOUR_DOMAIN}/api/v2/resource-servers/{YOUR_RESOURCE_SERVER_ID}' \
    -H 'Authorization: Bearer {YOUR_MANAGEMENT_API_TOKEN}' \
    -H 'Content-Type: application/json' \
    -H 'Accept: application/json' \
    -d '{
      "proof_of_possession": {
        "mechanism": "dpop",
        "required": true,
        "required_for": "public_clients"
      }
    }'
    ```

    L’exemple de code suivant configure DPoP Sender Constraining comme requis pour toutes les applications. Puisque `all_clients` est la valeur par défaut, `required_for` peut être omis :

    ```bash lines
    curl -L -X PATCH 'https://{YOUR_DOMAIN}/api/v2/resource-servers/{YOUR_RESOURCE_SERVER_ID}' \
    -H 'Authorization: Bearer {YOUR_MANAGEMENT_API_TOKEN}' \
    -H 'Content-Type: application/json' \
    -H 'Accept: application/json' \
    -d '{
      "proof_of_possession": {
        "mechanism": "dpop",
        "required": true
      }
    }'
    ```

    L’exemple de code suivant désactive le sender constraining pour un serveur de ressources (l’équivalent de sélectionner « Never » dans le Dashboard) :

    ```bash lines
    curl -L -X PATCH 'https://{YOUR_DOMAIN}/api/v2/resource-servers/{YOUR_RESOURCE_SERVER_ID}' \
    -H 'Authorization: Bearer {YOUR_MANAGEMENT_API_TOKEN}' \
    -H 'Content-Type: application/json' \
    -H 'Accept: application/json' \
    -d '{
      "proof_of_possession": null 
    }'
    ```
  </Tab>
</Tabs>
