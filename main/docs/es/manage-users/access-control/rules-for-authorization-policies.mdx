Puede agregar [Rules](/docs/es/customize/rules) a la política de autorización preconfigurada para ejercer un mayor control sobre si se permite o se deniega el acceso del usuario. Una Rule contiene código personalizado que toma una decisión de autorización según la lógica correspondiente. Cuando se combina con otras Rules, ayuda a definir lo que sucede en distintos contextos.

Las Rules pueden restringir el acceso en función de cualquier combinación de atributos que almacene para los usuarios, como el departamento del usuario, la hora del día, la ubicación o cualquier otro atributo del usuario o de la API (como el username, el nivel de autorización de seguridad o el nombre de la API).

Por ejemplo, si usara Rules para proporcionar un control de acceso detallado en una organización sin fines de lucro, podría dar acceso a una aplicación solo a empleados W2 que trabajen en el departamento de Investigación y Desarrollo de la oficina de Nueva Delhi.

Para ver ejemplos de implementaciones de Rules con políticas de autorización, lea [Casos de uso de ejemplo: Rules con autorización](/docs/es/manage-users/access-control/sample-use-cases-rules-with-authorization).

<div id="rules-in-the-authorization-process">
  ## Rules en el proceso de autorización
</div>

Según el orden en que se ejecutan, las Rules pueden cambiar el resultado de la decisión de autorización antes de que se agreguen los permisos al <Tooltip tip="Token de acceso: credencial de autorización, en forma de una cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=Access+Token">token de acceso</Tooltip>. El proceso básico con Rules inyectadas es el siguiente:

1. El usuario intenta autenticarse en la aplicación.
2. Auth0 envía la request al proveedor de identidad seleccionado.
3. Una vez que el proveedor de identidad confirma que las credenciales del usuario son válidas, todas las Rules creadas se ejecutan en el orden en que están configuradas en el Dashboard.
4. Siempre que ninguna rule haya restringido el acceso del usuario, este queda autorizado para acceder a la aplicación.
