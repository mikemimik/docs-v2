Con [reglas](/docs/es/customize/rules), puedes modificar o complementar el resultado de la decisión tomada por la [política de autorización](/docs/es/manage-users/access-control/authorization-policies) preconfigurada para gestionar casos más complejos de lo que es posible solo con [control de acceso basado en roles (RBAC)](/docs/es/manage-users/access-control/rbac). Según el orden en que se ejecuten, las reglas pueden cambiar el resultado de la decisión de autorización antes de que los permisos se agreguen al <Tooltip tip="Token de acceso: credencial de autorización, en forma de una cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=Access+Token">token de acceso</Tooltip>. También pueden permitirte personalizar el contenido de tus tokens.

<div id="allow-access-only-on-weekdays-for-a-specific-application">
  ## Permitir el acceso solo entre semana para una aplicación específica
</div>

Supongamos que tienes una aplicación y quieres asegurarte de que solo se pueda acceder a ella durante los días laborables. Para ello, debes [crear la siguiente rule](/docs/es/customize/rules/create-rules):

```js lines
function (user, context, callback) {

  if (context.clientName === 'APP_NAME') {
    const d = Date.getDay();

    if (d === 0 || d === 6) {
      return callback(new UnauthorizedError('This app is only available during the week.'));
    }
  }

  callback(null, user, context);
}
```

Si un usuario intenta acceder a la aplicación durante el fin de semana, se le denegará el acceso, incluso si se autentica y tiene los privilegios adecuados.

<div id="allow-access-only-to-users-who-are-inside-the-corporate-network">
  ## Permitir acceso solo a los usuarios que estén dentro de la red corporativa
</div>

Supongamos que quieres permitir el acceso a una aplicación, pero solo a los usuarios que accedan a la aplicación desde dentro de tu red corporativa. Para ello, crearías la siguiente rule:

```js lines
function (user, context, callback) {
  const ipaddr = require('ipaddr.js@1.9.0');
  const corp_network = "192.168.1.134/26";
  const current_ip = ipaddr.parse(context.request.ip);

  if (!current_ip.match(ipaddr.parseCIDR(corp_network))) {
    return callback(new UnauthorizedError('This app is only available from inside the corporate network.'));
  };

  callback(null, user, context);
}
```

Si el usuario está fuera de la red corporativa, se le denegará el acceso aunque se autentique correctamente y tenga los privilegios adecuados.

<div id="deny-access-to-anyone-calling-an-api">
  ## Denegar el acceso a cualquier usuario que llame a una API
</div>

Supongamos que quieres denegar el acceso a todos los usuarios que llaman a una API. Esto significa que debes denegar el acceso en función del valor de `audience` de tu API, que puedes encontrar en el campo **API <Tooltip tip="Audience: Identificador único de la audiencia de un token emitido. Se denomina aud en un token; su valor contiene el ID de una aplicación (ID de cliente) para un token de ID o de una API (identificador de API) para un token de acceso." cta="Ver glosario" href="/docs/es/glossary?term=Audience">Audience</Tooltip>** de tu API en [Dashboard &gt; Applications &gt; APIs](https://manage.auth0.com/#/apis). Para ello, crearías la siguiente rule:

```js lines
function (user, context, callback) {
  /*
   *  Deniega el acceso a flujos basados en usuario según el audience
   */
  var audience = '';
  audience = audience
              || (context.request && context.request.query && context.request.query.audience)
              || (context.request && context.request.body && context.request.body.audience);
  if (audience === 'http://todoapi2.api' || !audience) {
    return callback(new UnauthorizedError('end_users_not_allowed'));
  }
  return callback(null, user, context);
}
```

En este caso, el valor de `audience` para la API es `http:://todoapi2.api`, por lo que este es el valor de `audience` que rechazaremos. Si alguien intenta acceder a la API con este valor de `audience`, se le denegará el acceso y recibirá una respuesta `HTTP 401`.

<div id="add-user-roles-to-tokens">
  ## Agregar roles de usuario a los tokens
</div>

Si [habilita RBAC para las API](/docs/es/get-started/apis/enable-role-based-access-control-for-apis) junto con &quot;Agregar permisos en el token de acceso&quot; (o habilita RBAC mediante la <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip> y establece el **Dialecto del token** en `access_token_authz`), recibirá los permisos del usuario en sus tokens de acceso. Para agregar roles de usuario a los tokens, puede usar el objeto `context.authorization` al crear la siguiente rule:

```js lines
function (user, context, callback) {
  const namespace = 'http://demozero.net';
  const assignedRoles = (context.authorization || {}).roles;

  let idTokenClaims = context.idToken || {};
  let accessTokenClaims = context.accessToken || {};

  idTokenClaims[`${namespace}/roles`] = assignedRoles;
  accessTokenClaims[`${namespace}/roles`] = assignedRoles;

  context.idToken = idTokenClaims;
  context.accessToken = accessTokenClaims;

  callback(null, user, context);
}
```

<div id="manage-delegated-administration-extension-roles-using-the-authorization-core-feature-set">
  ## Administrar roles de Delegated Administration Extension con el conjunto de funcionalidades de Authorization Core
</div>

Aunque [Delegated Administration Extension (DAE)](/docs/es/customize/extensions/delegated-administration-extension) y el conjunto de funcionalidades de Authorization Core son funcionalidades completamente independientes, puedes usar el conjunto de funcionalidades de Authorization Core para crear y administrar roles para DAE si usas una rule.

1. [Crea roles de DAE](/docs/es/manage-users/access-control/configure-core-rbac/roles/create-roles) con el conjunto de funcionalidades de Authorization Core. Los nombres de los roles que crees deben coincidir con los nombres de los [roles de DAE predefinidos](/docs/es/customize/extensions/delegated-administration-extension#assign-roles-to-users).
2. [Asigna los roles de DAE que creaste a los usuarios adecuados](/docs/es/manage-users/access-control/configure-core-rbac/rbac-users/assign-roles-to-users) con el conjunto de funcionalidades de Authorization Core.
3. Agrega roles de usuario al espacio de nombres de DAE en el token de ID. Para hacerlo, [crea la siguiente rule](/docs/es/customize/rules/create-rules), recordando reemplazar el valor de marcador de posición `CLIENT_ID` por el ID de cliente de tu aplicación:

```js lines
function (user, context, callback) {
    if (context.clientID === 'CLIENT_ID') {
        const namespace = 'https://example.com/auth0-delegated-admin';
        context.idToken[namespace] = {
            roles: (context.authorization || {}).roles
        };
    }
    callback(null, user, context);
}
```

<Warning>
  Auth0 devuelve información del perfil en un formato estructurado de claims, tal como lo define la [especificación de OpenID Connect (OIDC)](https://openid.net/specs/openid-connect-core-1_0.html#StandardClaims). Esto significa que los claims personalizados agregados a los tokens de ID o a los tokens de acceso deben [cumplir con las directrices y restricciones](/docs/es/secure/tokens/json-web-tokens/create-custom-claims) para evitar posibles conflictos.
</Warning>
