title: Control de acceso basado en roles
---

El control de acceso basado en roles (RBAC) se refiere a la práctica de asignar permisos a los usuarios en función de su rol dentro de una organización. Ofrece un enfoque sencillo y fácil de gestionar para la gestión del acceso, menos propenso a errores que asignar permisos a los usuarios de forma individual.

Al usar RBAC para la gestión de roles, analizas las necesidades de tus usuarios y los agrupas en roles según sus responsabilidades comunes. Luego asignas uno o más roles a cada usuario y uno o más permisos a cada rol. Las relaciones entre usuario y rol, y entre rol y permisos, simplifican la asignación de usuarios, ya que estos dejan de administrarse individualmente y, en su lugar, obtienen privilegios conforme a los permisos asignados a sus roles.

Por ejemplo, si usaras RBAC para controlar el acceso a una aplicación de RR. HH., podrías asignar a los gerentes de RR. HH. un rol que les permita actualizar los datos de los empleados, mientras que otros empleados solo podrían ver sus propios datos.

Al planificar tu estrategia de control de acceso, es una práctica recomendada asignar a los usuarios la menor cantidad de permisos posible para que puedan realizar su trabajo.

  Para un modelo de autorización más sólido, consulta [Autorización detallada (FGA)](https://docs.fga.dev/authorization-concepts) para la autorización basada en atributos y en relaciones.
</Callout>

<div id="benefits-of-rbac">
  ## Beneficios de RBAC
</div>

Con RBAC, la gestión del acceso es más sencilla siempre que respetes estrictamente los requisitos de cada rol. RBAC te ayuda a:

* crear una asignación de permisos sistemática y repetible
* auditar fácilmente los privilegios de los usuarios y corregir los problemas detectados
* agregar y modificar roles rápidamente, así como implementarlos en todas las API
* reducir la posibilidad de errores al asignar permisos a los usuarios
* integrar usuarios externos mediante roles predefinidos
* cumplir de forma más eficaz con los requisitos normativos y legales en materia de confidencialidad y privacidad

<div id="rbac-model">
  ## Modelo de RBAC
</div>

<div id="roles">
  ### Roles
</div>

En esencia, un rol es un conjunto de permisos que puedes aplicar a los usuarios. Usar roles facilita agregar, quitar y ajustar permisos, en lugar de asignarlos individualmente a cada usuario. A medida que tu base de usuarios crece en escala y complejidad, los roles se vuelven especialmente útiles.

También puedes usar roles para agrupar permisos definidos para varias API. Por ejemplo, supongamos que tienes un módulo de marketing que permite a los usuarios crear y distribuir boletines para los clientes. Tu especialista en contenido de marketing crea todos los boletines y los prepara para su distribución. Del mismo modo, tienes un módulo de eventos que permite a los usuarios crear, publicar y gestionar el registro de eventos. Tu coordinador de eventos crea los eventos. Una vez que el vicepresidente de Marketing aprueba los boletines y los eventos, su asistente publica los eventos y distribuye los boletines. En este caso, tu API de boletines podría tener un permiso `distribute:newsletters` y tu API de eventos podría tener un permiso `publish:events`. Luego, estos permisos podrían agruparse en un rol llamado `Marketing Publisher` y asignarse al asistente del vicepresidente de Marketing.

Además, se pueden [agregar roles específicos de una organización a los miembros de la organización](/docs/es/manage-users/organizations/configure-organizations/add-member-roles) y usarlos para permitir el acceso en tu aplicación según las organizaciones con las que un usuario final inicia sesión. Esto resulta especialmente útil en productos multi-tenant y SaaS, donde un mismo usuario puede tener un rol privilegiado en una organización, pero no en otras.

<div id="overlapping-role-assignments">
  ## Asignaciones de roles superpuestas
</div>

RBAC es un modelo aditivo, por lo que, si tienes asignaciones de roles superpuestas, tus permisos efectivos son la unión de esas asignaciones.

Por ejemplo, supongamos que tienes una API que proporciona datos para una aplicación de eventos. Creas un rol de `Organizer` y le asignas permisos que le permiten ver, crear y editar eventos. También creas un rol de `Registrant` y le asignas permisos que le permiten ver eventos y registrarse en ellos. Cualquier usuario que tenga ambos roles, `Organizer` y `Registrant`, podrá ver, crear, editar y registrarse en eventos.

<div id="role-based-access-control-in-auth0">
  ## Control de acceso basado en roles en Auth0
</div>

Actualmente, ofrecemos dos formas de implementar el control de acceso basado en roles (RBAC), que puede usar en lugar del sistema interno de control de acceso de su API o en combinación con este:

* [Authorization Core](/docs/es/manage-users/access-control/configure-core-rbac)
* [Authorization Extension](/docs/es/customize/extensions/authorization-extension)

Estamos ampliando el conjunto de funcionalidades de Authorization Core para equipararlo con la funcionalidad de Authorization Extension. Nuestra nueva implementación de RBAC en el núcleo mejora el rendimiento y la escalabilidad, y con el tiempo ofrecerá un sistema RBAC más flexible que Authorization Extension.

Por ahora, ambas implementan las funcionalidades clave de RBAC y le permiten restringir los scopes personalizados definidos para una API a aquellos que se hayan asignado al usuario como permisos. Para ver una comparación, consulte [Authorization Core vs. Authorization Extension](/docs/es/manage-users/access-control/authorization-core-vs-authorization-extension).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  El conjunto de funcionalidades de [Authorization Core](/docs/es/manage-users/access-control/rbac) y [Authorization Extension](/docs/es/customize/extensions/authorization-extension) son funcionalidades completamente independientes. Para administrar grupos, roles o permisos, debe usar la funcionalidad en la que se crearon originalmente.
</Callout>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Aunque [Delegated Administration Extension (DAE)](/docs/es/customize/extensions/delegated-administration-extension) y el conjunto de funcionalidades de [Authorization Core](/docs/es/manage-users/access-control/rbac) son funcionalidades completamente independientes, puede usar el conjunto de funcionalidades de Authorization Core para crear y administrar roles para la DAE mediante Actions. Para saber cómo hacerlo, consulte [Casos de uso de ejemplo: Actions con autorización](/docs/es/manage-users/access-control/sample-use-cases-actions-with-authorization).
</Callout>

<div id="extending-rbac">
  ### Extender RBAC
</div>

Puedes tener un mayor control usando [Actions](/docs/es/customize/actions) para restringir el acceso en función de una combinación de atributos, como el departamento del usuario, la hora del día, la ubicación desde la que se accede o cualquier otro atributo del usuario o de la API (por ejemplo, username, nivel de autorización o nombre de la API).

Para obtener más información sobre cómo usar Actions con políticas de autorización, consulta [Casos de uso de ejemplo: Actions con autorización](/docs/es/manage-users/access-control/sample-use-cases-actions-with-authorization).
