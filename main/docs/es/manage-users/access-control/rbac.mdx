description: Comprende el concepto de control de acceso basado en roles y cómo se aplica en Auth0.
title: Control de acceso basado en roles
---

El control de acceso basado en roles (RBAC) consiste en asignar permisos a los usuarios según su rol dentro de una organización. Ofrece un enfoque simple y fácil de administrar para la gestión del acceso, y es menos propenso a errores que asignar permisos a cada usuario de forma individual.

Al usar RBAC para la gestión de roles, se analizan las necesidades de los usuarios y se los agrupa en roles según responsabilidades comunes. Luego, se asignan uno o más roles a cada usuario y uno o más permisos a cada rol. Las relaciones entre usuarios y roles, y entre roles y permisos, simplifican la asignación de usuarios, ya que estos ya no tienen que administrarse de forma individual, sino que reciben privilegios acordes con los permisos asignados a sus roles.

Por ejemplo, si usara RBAC para controlar el acceso a una aplicación de RR. HH., podría asignar a los gerentes de RR. HH. un rol que les permita actualizar la información de los empleados, mientras que otros empleados solo podrían ver su propia información.

Al planificar su estrategia de control de acceso, la práctica recomendada es asignar a los usuarios la menor cantidad de permisos necesaria para que puedan realizar su trabajo.

<div id="benefits-of-rbac">
  ## Beneficios de RBAC
</div>

Con RBAC, la gestión del acceso es más sencilla siempre que sigas estrictamente los requisitos de los roles. RBAC te ayuda a:

* crear asignaciones de permisos sistemáticas y repetibles
* auditar fácilmente los privilegios de los usuarios y corregir los problemas detectados
* agregar y cambiar roles rápidamente, así como implementarlos en todas las API
* reducir la posibilidad de errores al asignar permisos a los usuarios
* integrar usuarios de terceros asignándoles roles predefinidos
* cumplir de forma más eficaz con los requisitos normativos y legales de confidencialidad y privacidad

<div id="rbac-model">
  ## Modelo RBAC
</div>

<div id="roles">
  ### Roles
</div>

En esencia, un rol es una colección de permisos que puedes aplicar a los usuarios. Usar roles facilita agregar, quitar y ajustar permisos, en lugar de asignarlos a los usuarios individualmente. A medida que tu base de usuarios aumenta en tamaño y complejidad, los roles se vuelven especialmente útiles.

También puedes usar roles para agrupar permisos definidos para varias API. Por ejemplo, supongamos que tienes un módulo de marketing que permite a los usuarios crear y distribuir boletines entre los clientes. Tu especialista en contenido de marketing crea todos los boletines y los prepara para su distribución. Del mismo modo, tienes un módulo de eventos que permite a los usuarios crear, publicar y gestionar el registro de eventos. Tu coordinador de eventos crea los eventos. Una vez que el vicepresidente de marketing aprueba los boletines y los eventos, su asistente publica los eventos y distribuye los boletines. En este caso, tu API de boletines podría tener un permiso `distribute:newsletters` y tu API de eventos podría tener un permiso `publish:events`. Estos permisos podrían luego agruparse en un rol llamado `Marketing Publisher` y asignarse al asistente del vicepresidente de marketing.

Además, los roles específicos de la organización se pueden [agregar a los miembros de la organización](/docs/es/manage-users/organizations/configure-organizations/add-member-roles) y usar para permitir el acceso en tu aplicación según las organizaciones con las que un usuario final inicia sesión. Esto resulta especialmente útil para productos SaaS y multi-tenant, donde un usuario en particular podría tener un rol con privilegios en una organización, pero no en otras.

<div id="overlapping-role-assignments">
  ## Asignaciones de roles solapadas
</div>

RBAC es un modelo aditivo, por lo que, si tienes asignaciones de roles solapadas, tus permisos efectivos son la unión de esas asignaciones.

Por ejemplo, supongamos que tienes una API que proporciona datos para una aplicación de eventos. Creas un rol de `Organizer` y le asignas permisos que le permiten ver, crear y editar eventos. También creas un rol de `Registrant` y le asignas permisos que le permiten ver eventos e inscribirse en ellos. Cualquier usuario que tenga ambos roles, `Organizer` y `Registrant`, podrá ver, crear, editar e inscribirse en eventos.

<div id="role-based-access-control-in-auth0">
  ## Control de acceso basado en roles en Auth0
</div>

Actualmente, ofrecemos dos formas de implementar el control de acceso basado en roles (RBAC), que puede usar en lugar del sistema interno de control de acceso de su API o en combinación con este:

* [Authorization Core](/docs/es/manage-users/access-control/configure-core-rbac)
* [Authorization Extension](/docs/es/customize/extensions/authorization-extension)

Estamos ampliando el conjunto de funcionalidades de Authorization Core para igualar la funcionalidad de Authorization Extension. Nuestra nueva implementación central de RBAC mejora el rendimiento y la escalabilidad, y con el tiempo ofrecerá un sistema de RBAC más flexible que Authorization Extension.

Por ahora, ambas implementan las funcionalidades clave de RBAC y le permiten restringir los `scopes` personalizados definidos para una API a aquellos que se hayan asignado al usuario como permisos. Para ver una comparación, consulte [Authorization Core vs. Authorization Extension](/docs/es/manage-users/access-control/authorization-core-vs-authorization-extension).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  El conjunto de funcionalidades [Authorization Core](/docs/es/manage-users/access-control/rbac) y [Authorization Extension](/docs/es/customize/extensions/authorization-extension) son funcionalidades completamente independientes. Para administrar grupos, roles o permisos, debe usar la funcionalidad en la que se crearon originalmente.
</Callout>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Aunque [Delegated Administration Extension (DAE)](/docs/es/customize/extensions/delegated-administration-extension) y el conjunto de funcionalidades [Authorization Core](/docs/es/manage-users/access-control/rbac) son funcionalidades completamente independientes, puede usar el conjunto de funcionalidades de Authorization Core para crear y administrar roles para DAE mediante Actions. Para saber cómo hacerlo, consulte [Casos de uso de muestra: Actions con autorización](/docs/es/manage-users/access-control/sample-use-cases-actions-with-authorization).
</Callout>

<div id="extending-rbac">
  ### Ampliar RBAC
</div>

Puede lograr un control más preciso usando [Actions](/docs/es/customize/actions) para restringir el acceso en función de una combinación de atributos, como el departamento del usuario, la hora del día, la ubicación de acceso o cualquier otro atributo del usuario o de la API (por ejemplo, username, nivel de seguridad o nombre de la API).

Para obtener más información sobre cómo usar Actions con políticas de autorización, consulte [Casos de uso de muestra: Actions con autorización](/docs/es/manage-users/access-control/sample-use-cases-actions-with-authorization).
