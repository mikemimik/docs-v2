title: Control de acceso basado en roles
---

El control de acceso basado en roles (RBAC) se refiere a la asignación de permisos a los usuarios según su rol dentro de una organización. Ofrece un enfoque sencillo y fácil de administrar para la gestión del acceso, y es menos propenso a errores que asignar permisos a cada usuario de forma individual.

Al usar RBAC para la administración de roles, se analizan las necesidades de los usuarios y se los agrupa en roles según responsabilidades comunes. Después, se asignan uno o más roles a cada usuario y uno o más permisos a cada rol. Las relaciones entre usuario y rol, y entre rol y permisos, simplifican la asignación de usuarios, ya que estos ya no necesitan administrarse de forma individual, sino que obtienen privilegios acordes con los permisos asignados a sus roles.

Por ejemplo, si usara RBAC para controlar el acceso a una aplicación de RR. HH., podría asignar a los responsables de RR. HH. un rol que les permita actualizar los datos de los empleados, mientras que otros empleados solo podrían ver sus propios datos.

Al planificar su estrategia de control de acceso, una práctica recomendada es asignar a los usuarios la menor cantidad de permisos posible para que puedan realizar su trabajo.

<div id="benefits-of-rbac">
  ## Beneficios de RBAC
</div>

Con RBAC, la gestión del acceso es más sencilla siempre que respetes estrictamente los requisitos de cada rol. RBAC te ayuda a:

* crear asignaciones de permisos sistemáticas y repetibles
* auditar fácilmente los privilegios de los usuarios y corregir las incidencias identificadas
* agregar y cambiar roles rápidamente, así como aplicarlos en todas las API
* reducir la posibilidad de errores al asignar permisos a los usuarios
* integrar usuarios externos dándoles roles predefinidos
* cumplir con mayor eficacia los requisitos normativos y legales de confidencialidad y privacidad

<div id="rbac-model">
  ## Modelo de RBAC
</div>

<div id="roles">
  ### Roles
</div>

Básicamente, un rol es un conjunto de permisos que puedes aplicar a los usuarios. Usar roles facilita agregar, quitar y ajustar permisos, en lugar de asignarlos individualmente a cada usuario. A medida que tu base de usuarios crece en tamaño y complejidad, los roles se vuelven especialmente útiles.

También puedes usar roles para agrupar permisos definidos para varias API. Por ejemplo, supongamos que tienes un módulo de marketing que permite a los usuarios crear y distribuir boletines informativos a los clientes. Tu especialista en contenido de marketing crea todos los boletines y los prepara para su distribución. Del mismo modo, tienes un módulo de eventos que permite a los usuarios crear, publicar y gestionar el registro de eventos. Tu coordinador de eventos crea los eventos. Una vez que el vicepresidente de Marketing aprueba los boletines y los eventos, su asistente publica los eventos y distribuye los boletines. En este caso, tu Newsletter API podría tener un permiso `distribute:newsletters` y tu Event API podría tener un permiso `publish:events`. Estos permisos podrían agruparse en un rol llamado `Marketing Publisher` y asignarse al asistente del vicepresidente de Marketing.

Además, se pueden [agregar roles específicos de la organización a los miembros de la organización](/docs/es/manage-users/organizations/configure-organizations/add-member-roles) y usarlos para permitir el acceso en tu aplicación según las organizaciones con las que un usuario final inicia sesión. Esto resulta especialmente útil para productos multi-tenant y SaaS, en los que un usuario determinado puede tener un rol con privilegios en una organización, pero no en otras.

<div id="overlapping-role-assignments">
  ## Asignaciones de roles superpuestas
</div>

RBAC es un modelo aditivo, por lo que, si tienes asignaciones de roles superpuestas, tus permisos efectivos serán la suma de los permisos de esos roles.

Por ejemplo, supongamos que tienes una API que proporciona datos para una aplicación de eventos. Creas un rol de `Organizer` y le asignas permisos que le permiten ver, crear y editar eventos. También creas un rol de `Registrant` y le asignas permisos que le permiten ver eventos e inscribirse en ellos. Cualquier usuario que tenga tanto el rol `Organizer` como el rol `Registrant` podrá ver, crear, editar e inscribirse en eventos.

<div id="role-based-access-control-in-auth0">
  ## Control de acceso basado en roles en Auth0
</div>

Actualmente, ofrecemos dos formas de implementar el control de acceso basado en roles (RBAC), que puedes usar en lugar del sistema interno de control de acceso de tu API o junto con él:

* [Authorization Core](/docs/es/manage-users/access-control/configure-core-rbac)
* [Authorization Extension](/docs/es/customize/extensions/authorization-extension)

Estamos ampliando nuestro conjunto de funcionalidades de Authorization Core para igualar la funcionalidad de Authorization Extension. Nuestra nueva implementación principal de RBAC mejora el rendimiento y la escalabilidad y, con el tiempo, ofrecerá un sistema RBAC más flexible que Authorization Extension.

Por ahora, ambas implementan las funcionalidades clave de RBAC y te permiten restringir los scopes personalizados definidos para una API a aquellos que se hayan asignado al usuario como permisos. Para ver una comparación, consulta [Authorization Core vs. Authorization Extension](/docs/es/manage-users/access-control/authorization-core-vs-authorization-extension).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  El conjunto de funcionalidades de [Authorization Core](/docs/es/manage-users/access-control/rbac) y [Authorization Extension](/docs/es/customize/extensions/authorization-extension) son funcionalidades completamente independientes. Para administrar grupos, roles o permisos, debes usar la funcionalidad en la que se crearon originalmente.
</Callout>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Aunque [Delegated Administration Extension (DAE)](/docs/es/customize/extensions/delegated-administration-extension) y el conjunto de funcionalidades de [Authorization Core](/docs/es/manage-users/access-control/rbac) son funcionalidades completamente independientes, puedes usar el conjunto de funcionalidades de Authorization Core para crear y administrar roles para DAE mediante Actions. Para saber cómo hacerlo, consulta [Sample Use Cases: Actions with Authorization](/docs/es/manage-users/access-control/sample-use-cases-actions-with-authorization).
</Callout>

<div id="extending-rbac">
  ### Ampliación de RBAC
</div>

Puedes obtener un mayor control usando [Actions](/docs/es/customize/actions) para restringir el acceso en función de una combinación de atributos, como el departamento del usuario, la hora del día, la ubicación desde la que se accede o cualquier otro atributo del usuario o de la API (por ejemplo, username, nivel de autorización o nombre de la API).

Para obtener más información sobre el uso de Actions con políticas de autorización, consulta [Casos de uso de ejemplo: Actions con autorización](/docs/es/manage-users/access-control/sample-use-cases-actions-with-authorization).
