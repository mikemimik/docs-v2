Las cookies, que se utilizan para la autenticación y el mantenimiento de sesiones, pueden protegerse mediante atributos. Auth0 usa cookies para lo siguiente:

* OIDC empresarial con `form_post`
* <Tooltip tip="Security Assertion Markup Language (SAML): Protocolo estandarizado que permite a dos partes intercambiar información de autenticación sin una contraseña." cta="Ver glosario" href="/docs/es/glossary?term=SAML">SAML</Tooltip> con Binding HTTP-POST
* Mensaje web (también conocido como `checkSession`)

<div id="samesite-attributes">
  ## Atributos SameSite
</div>

Puedes agregar atributos de [cookie SameSite](https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers/Set-Cookie/SameSite) en el encabezado de respuesta HTTP `set-cookie` para restringir el comportamiento del navegador. Esto puede impedir que el navegador envíe el par `key=value` de la cookie según el tipo de interacción que desencadenó la solicitud HTTP.

Los valores de atributo aceptados son los siguientes:

| Atributo | Descripción                                                                                                                                                                                           |
| -------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `strict` | Envía la cookie si el usuario navega dentro de los límites de origen del sitio web                                                                                                                    |
| `lax`    | Envía la cookie si el usuario navega entre dominios, pero no en contextos de terceros (iframes o posts)                                                                                               |
| `none`   | Envía la cookie con solicitudes que cruzan los límites de origen del sitio web. A menos que se den otras condiciones (por ejemplo, que las cookies de terceros estén bloqueadas), no envía la cookie. |

Algunos de los atributos de cookie que quizá ya conozcas son:

| Atributo            | Descripción                                                                                                                                                                                                |
| ------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `httpOnly`          | Permite que una cookie se envíe solo con solicitudes HTTP; no se puede leer con `document.cookie` de Javascript                                                                                            |
| `secure`            | Permite que el navegador envíe la cookie solo a un contexto seguro; que el contexto se considere seguro o no depende del navegador, pero normalmente esto requiere usar HTTPS                              |
| `max-age / expires` | Controla si la cookie es una cookie de **sesión** (por ejemplo, se elimina cuando el navegador termina su sesión) o **persistente** (por ejemplo, la cookie permanece más allá de la sesión del navegador) |

Cuando los recibe, el navegador analiza los headers y actualiza su almacén de cookies en consecuencia.

<div id="browser-cookie-changes">
  ## Cambios en las cookies del navegador
</div>

Desde febrero de 2020, Google Chrome v80 cambió la forma en que gestiona las cookies. Auth0 implementó los siguientes cambios en la forma en que gestiona las cookies:

* Las cookies sin el atributo `SameSite` configurado pasarán a configurarse como `lax`
* Las cookies con `SameSite=none` deben ser seguras; de lo contrario, no pueden guardarse en el almacén de cookies del navegador

El objetivo de estos cambios es mejorar la seguridad y ayudar a mitigar los ataques CSRF.

Estos cambios afectan a las siguientes cookies:

* `auth0` (gestiona las sesiones de usuario)
* `auth0-mf` (gestiona información relevante para la <Tooltip tip="Autenticación multifactor (MFA): Proceso de autenticación de usuario que usa un factor además de nombre de usuario y contraseña, como un código por SMS." cta="Ver glosario" href="/docs/es/glossary?term=multi-factor+authentication">autenticación multifactor</Tooltip>)
* `did` (el identificador de un dispositivo/agente de usuario)

Para estas cookies, Auth0:

* Configurará el atributo `SameSite` como `none`, por lo que la cookie requerirá el uso de HTTPS (independientemente del entorno)
* Configurará cookies de respaldo en caso de que un navegador Legacy no admita que `SameSite` se establezca en `None`. Estas cookies de respaldo son `auth0_compat`, `auth0-mf_compat` y `did_compat`

El servidor cambia la forma en que renderiza cuando el visitante regresa y establece una cookie de visto. <div id="features-affected">
  ## Funcionalidades afectadas
</div>

La siguiente tabla muestra cómo los cambios en el atributo `SameSite` pueden afectar a sus aplicaciones.

| Comportamiento de la aplicación                                                                                                                                                        | Afectado por el cambio |
| -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------- |
| Cookies configuradas como `sameSite=none` cuando el sitio web no es `https://`                                                                                                         | Sí                     |
| Las cookies no tienen un valor explícito configurado para el atributo `sameSite` y se requieren en un contexto de origen cruzado (como HTTP form&#95;post o la inserción de un iframe) | Sí                     |
| Aplicaciones nativas (todo lo que no sea cookies + basado en web)                                                                                                                      | No (M2M)               |
| Ya se está configurando un valor explícito para el atributo de cookie `sameSite`                                                                                                       | No                     |
| Subdominio diferente dentro del mismo eTLD+1 (la aplicación está en el mismo eTLD+1 que el tenant de Auth0 con dominio personalizado)                                                  | Potencialmente         |

Si utiliza una aplicación web con sesiones (por ejemplo, para guardar preferencias del usuario, carritos de compra, etc.) y permite que los usuarios inicien sesión mediante <Tooltip tip="Proveedor de identidad (IdP): servicio que almacena y gestiona identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=identity+providers">proveedores de identidad</Tooltip> como Google, Github o Auth0, entonces depende de las cookies para ofrecer esa funcionalidad. Hay cambios en el comportamiento de las cookies del navegador que pueden afectar a la experiencia del usuario. Google Chrome, por ejemplo, es el primer proveedor de navegadores en implementar un cambio que podría no ser compatible con su aplicación web.

Puede que note que las especificaciones de Google Chrome y Microsoft Edge para establecer `SameSite` como indefinido han cambiado: `SameSite` ya no toma `none` como valor predeterminado, sino `lax`.

Por ejemplo, supongamos que crea una nueva UI y tiene varios servicios a los que accede mediante un proxy a través de un gateway de Auth0. En este gateway, crea una sesión de cookie. Si hace una solicitud de origen cruzado, puede ver esta advertencia en la consola de Javascript:

`A cookie associated with a cross-site resource (URL) was set without the SameSite attribute. A future release of Chrome will only deliver cookies with cross-site requests if they are set with SameSite=None and Secure. You can review cookies in developer tools under Application>Storage>Cookies and see more details at https://www.chromestatus.com/feature/5088147346030592 and https://www.chromestatus.com/feature/5633521622188032`

<div id="actions-you-need-to-take">
  ## Acciones que debe tomar
</div>

Para prepararse para este cambio, debe:

* Revisar la lista de [navegadores no compatibles](https://www.chromium.org/updates/same-site/incompatible-clients).
* Configurar su aplicación para que use `SameSite=none` si utiliza `response_mode=form_post` al interactuar con Auth0 (tenga en cuenta que Chrome no hace excepciones, ni siquiera para `localhost`)
* Marcar su cookie como segura si su atributo `SameSite` es `None.` De lo contrario, el navegador la rechazará. Si usa HTTP para sus URL de devolución de llamada, estas dejarán de funcionar si utiliza esas cookies para vincular el state de la solicitud de autorización/<Tooltip tip="Nonce: Número arbitrario emitido una sola vez en un protocolo de autenticación para detectar y prevenir ataques de repetición." cta="Ver glosario" href="/docs/es/glossary?term=nonce">nonce</Tooltip>. Por lo tanto, debe usar HTTPS o establecer `SameSite=lax`
