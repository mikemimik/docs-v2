Las cookies, que se utilizan para la autenticación y el mantenimiento de sesiones, se pueden proteger configurando atributos. Auth0 usa cookies para lo siguiente:

* OIDC empresarial con `form_post`
* <Tooltip tip="Security Assertion Markup Language (SAML): Protocolo estandarizado que permite a dos partes intercambiar información de autenticación sin una contraseña." cta="Ver glosario" href="/docs/es/glossary?term=SAML">SAML</Tooltip> binding HTTP-POST
* Mensaje web (también conocido como `checkSession`)

<div id="samesite-attributes">
  ## Atributos SameSite
</div>

Puede agregar atributos de [cookie SameSite](https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers/Set-Cookie/SameSite) en el header de respuesta HTTP `set-cookie` para restringir el comportamiento del navegador. Esto puede impedir que el navegador envíe el par `key=value` de la cookie según el tipo de interacción que activó la solicitud HTTP.

Los valores de atributo aceptados son los siguientes:

| Atributo | Descripción                                                                                                                                                                                         |
| -------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `strict` | Envía la cookie si el usuario navega dentro de los límites del origen del sitio web                                                                                                                 |
| `lax`    | Envía la cookie si el usuario navega entre dominios, pero no en contextos de terceros (iframes o posts)                                                                                             |
| `none`   | Envía la cookie con solicitudes que cruzan los límites del origen del sitio web. A menos que se den otras condiciones (es decir, que las cookies de terceros estén bloqueadas), no envíe la cookie. |

Algunos de los atributos de cookie que quizá ya conozca incluyen:

| Atributo            | Descripción                                                                                                                                                                                                 |
| ------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `httpOnly`          | Permite que una cookie se envíe solo con solicitudes HTTP; no puede leerse con `document.cookie` de Javascript                                                                                              |
| `secure`            | Permite que el navegador envíe la cookie solo a un contexto seguro; que el contexto se considere seguro o no depende del navegador, pero esto normalmente requiere el uso de HTTPS                          |
| `max-age / expires` | Controla si la cookie es una cookie de **sesión** (por ejemplo, se descarta cuando el navegador finaliza su sesión) o **persistente** (por ejemplo, la cookie permanece después de la sesión del navegador) |

Al recibirlos, el navegador analiza los headers y actualiza su Cookie jar en consecuencia.

<div id="browser-cookie-changes">
  ## Cambios en las cookies del navegador
</div>

A partir de febrero de 2020, Google Chrome v80 cambió la forma en que gestiona las cookies. Auth0 implementó los siguientes cambios en la forma en que gestiona las cookies:

* Las cookies sin el atributo `SameSite` se configurarán como `lax`
* Las cookies con `SameSite=none` deben ser seguras; de lo contrario, no se pueden guardar en el cookie jar del navegador

El objetivo de estos cambios es mejorar la seguridad y ayudar a mitigar los ataques CSRF.

Estos cambios afectan a las siguientes cookies:

* `auth0` (gestiona las sesiones de usuario)
* `auth0-mf` (gestiona información relevante para la <Tooltip tip="Autenticación multifactor (MFA): Proceso de autenticación de usuario que utiliza un factor además de nombre de usuario y contraseña, como un código por SMS." cta="Ver glosario" href="/docs/es/glossary?term=multi-factor+authentication">autenticación multifactor</Tooltip>)
* `did` (el identificador de un dispositivo/agente de usuario)

Para estas cookies, Auth0 hará lo siguiente:

* Configurará el atributo `SameSite` como `none`, y la cookie requerirá el uso de HTTPS (independientemente del entorno)
* Configurará cookies de respaldo en caso de que un navegador Legacy no admita que `SameSite` se establezca en `None`. Estas cookies de respaldo son `auth0_compat`, `auth0-mf_compat` y `did_compat`

El servidor cambia la forma en que se renderiza cuando el visitante regresa y establece una cookie de visita. La parte gris del encabezado `set-cookie` es la cookie real `key=value.` La parte roja son los atributos de la cookie que el navegador almacena en el cookie jar para decidir más adelante si debe incluir el par de cookie `key+value` en las solicitudes.

<div id="features-affected">
  ## Funcionalidades afectadas
</div>

La siguiente tabla muestra cómo los cambios en el atributo `SameSite` pueden afectar a sus aplicaciones.

| Comportamiento de la aplicación                                                                                                                                                          | Afectado por el cambio |
| ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------- |
| Cookies configuradas como `sameSite=none` cuando el sitio web no es `https://`                                                                                                           | Sí                     |
| Las cookies no tienen establecido un valor explícito para el atributo `sameSite` y son necesarias en un contexto de origen cruzado (como HTTP form&#95;post o la inserción de un iframe) | Sí                     |
| Aplicaciones nativas (todo lo que no sea cookies + basado en la web)                                                                                                                     | No (M2M)               |
| Ya se está estableciendo un valor explícito para el atributo `sameSite` de la cookie                                                                                                     | No                     |
| Subdominio diferente en el mismo eTLD+1 (la aplicación está en el mismo eTLD+1 que el tenant de Auth0 con dominio personalizado)                                                         | Potencialmente         |

Si usa una aplicación web con sesiones (p. ej., para guardar las preferencias del usuario, carritos de compra, etc.) y permite que los usuarios inicien sesión mediante <Tooltip tip="Proveedor de identidad (IdP): servicio que almacena y administra identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=identity+providers">proveedores de identidad</Tooltip> como Google, Github o Auth0, entonces depende de las cookies para que esa funcionalidad sea posible. Hay cambios en el comportamiento de las cookies del navegador que pueden afectar la experiencia del usuario. Google Chrome, por ejemplo, es el primer proveedor de navegadores en implementar un cambio que podría no ser compatible con su aplicación web.

Puede notar que las especificaciones de Google Chrome y Microsoft Edge para establecer `SameSite` como indefinido han cambiado: antes `SameSite` usaba `none` de forma predeterminada y ahora usa `lax`.

Por ejemplo, supongamos que crea una nueva UI y tiene varios servicios a los que accede mediante proxy a través de un gateway de Auth0. En este gateway, crea una sesión con cookies. Si realiza una solicitud de origen cruzado, puede ver esta advertencia en la consola de Javascript:

`A cookie associated with a cross-site resource (URL) was set without the SameSite attribute. A future release of Chrome will only deliver cookies with cross-site requests if they are set with SameSite=None and Secure. You can review cookies in developer tools under Application>Storage>Cookies and see more details at https://www.chromestatus.com/feature/5088147346030592 and https://www.chromestatus.com/feature/5633521622188032`

<div id="actions-you-need-to-take">
  ## Acciones que debe realizar
</div>

Para prepararse para este cambio, debe:

* Revisar la lista de [navegadores no compatibles](https://www.chromium.org/updates/same-site/incompatible-clients).
* Configurar su aplicación para usar `SameSite=none` si utiliza `response_mode=form_post` al interactuar con Auth0 (tenga en cuenta que Chrome no hace excepciones, ni siquiera para `localhost`)
* Configurar la cookie como segura si su atributo `SameSite` es igual a `None.` De lo contrario, el navegador la rechazará. Si usa HTTP para sus URL de devolución de llamada, estas dejarán de funcionar si utiliza esas cookies para asociar el `state`/ <Tooltip tip="Nonce: Número arbitrario emitido una sola vez en un protocolo de autenticación para detectar y prevenir ataques de repetición." cta="Ver glosario" href="/docs/es/glossary?term=nonce">nonce</Tooltip> de la solicitud de autorización. Por lo tanto, debe usar HTTPS o configurar `SameSite=lax`
