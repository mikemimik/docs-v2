Proteger una aplicación web de página única (SPA) puede ser un desafío. Sin embargo, si tu SPA cumple con los siguientes criterios, puedes simplificar tu implementación usando cookies para autenticar.

* Se entrega al cliente desde tu propio backend.
* Tiene el mismo dominio que tu backend.
* Hace llamadas a la API de tu backend que requieren autenticación.

Aquí tienes una descripción de este enfoque, así como una implementación de ejemplo con [Node.js](https://nodejs.org/en/).

<div id="how-it-works">
  ## Cómo funciona
</div>

Los pasos a continuación muestran cómo se obtienen y utilizan los tokens. En este enfoque, se usa el [flujo implícito con Form Post](/docs/es/get-started/authentication-and-authorization-flow/implicit-flow-with-form-post) en lugar de un [flujo de código de autorización con Proof Key for Code Exchange](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce) tradicional. Esto se debe a que el modo de respuesta Form Post es una forma más sencilla de implementar el inicio de sesión cuando el recurso al que se solicita acceso es propio.

<Frame>![Autenticar aplicaciones web de página única usando cookies: diagrama de cómo funciona, parte 1](/docs/images/cdy7uua7fh8z/7suPpG6SuHuCOFHeFmE21T/1bf0a56fd21f65460cff7fd01b4c93f4/2023-09-22_13-21-34.png)</Frame>

1. El usuario accede a una ruta protegida mediante el navegador, o realiza alguna acción que requiere iniciar un paso de autenticación (como hacer clic en un botón de Login)
2. El navegador del cliente redirige a una ruta `/login` en el backend, o a la ruta protegida, según lo que haya hecho el usuario
3. El backend construye una solicitud al endpoint `/authorize` del servidor de autorización y redirige allí al navegador del cliente
4. Se le solicita al usuario que se autentique con el método que presente el servidor de autorización
5. El servidor de autorización envía los tokens mediante POST al Redirect URI como un form post codificado en URL. El backend puede recuperar esos tokens analizando los datos del body.

En este punto, el usuario ya está autenticado y el backend tiene los tokens necesarios. Ahora se puede crear una cookie para representar este estado en el cliente. Luego, se redirige el navegador del cliente a una ruta que sirve la SPA y que también recibe la cookie de autenticación.

A partir de ese momento, esta cookie se intercambia entre el cliente y el backend cuando se realizan llamadas a la API mediante AJAX. En cada solicitud, el backend verifica si la cookie sigue siendo válida y, de ser así, permite que la solicitud continúe.

<Frame>![Autenticar aplicaciones web de página única usando cookies: diagrama de cómo funciona, parte 2](/docs/images/cdy7uua7fh8z/6SCp9KwQkwJV7L26TOuxIO/d625575a43d00a00fcf17c7310ac2bc8/2023-09-22_13-21-50.png)</Frame>

<div id="invalid-or-missing-cookies">
  ### Cookies no válidas o faltantes
</div>

Al implementar este enfoque, tendrás que gestionar los casos en los que la cookie de autenticación no sea válida o falte. La llamada a la API desde el cliente al backend se realiza en segundo plano, por lo que el cliente tiene que encargarse de cualquier respuesta del servidor que indique que el usuario debe volver a autenticarse.

En la siguiente aplicación de ejemplo, este caso se maneja de forma simple, pidiendo al usuario que vuelva a autenticarse si la llamada a la API devuelve un 302 Redirect. El 302 se produce porque, cuando la validación de la cookie falla, el servidor intenta redirigir al endpoint de autorización del <Tooltip tip="Servidor de autorización: servidor centralizado que contribuye a definir los límites del acceso de un usuario. Por ejemplo, tu servidor de autorización puede controlar los datos, las tareas y las funcionalidades disponibles para un usuario." cta="Ver glosario" href="/docs/es/glossary?term=authorization+server">servidor de autorización</Tooltip> y envía esta respuesta al cliente.

<div id="example">
  ## Ejemplo
</div>

La aplicación de ejemplo usa Node.js y Express para demostrar los conceptos explicados anteriormente.

<div id="prerequisites">
  ### Requisitos previos
</div>

* Para seguir este tutorial, asegúrate de tener instalada la [última versión de Node](https://nodejs.org/en/download/).
* Una vez instalado Node, [descarga o clona el código fuente](https://github.com/auth0-blog/spa-cookie-demo/) y abre la carpeta del proyecto en una terminal.

  ```bash lines
  # Clona el repositorio del tutorial usando SSH
  git clone git@github.com:auth0-blog/spa-cookie-demo
  # ... o, si usas HTTPS:
  git clone https://github.com/auth0-blog/spa-cookie-demo.git
  # Ve al directorio del proyecto
  cd spa-cookie-demo
  ```
* La rama `master` representa el estado de la aplicación antes de añadir la autenticación. Si quieres consultar la versión final de la aplicación, cambia a la rama `with-oidc`:
  `git checkout with-oidc`

<div id="initialize-the-nodejs-app">
  ### Inicializa la aplicación de Node.js
</div>

Instala las dependencias de la aplicación ejecutando `npm install` en la terminal. Para ejecutar la aplicación, usa `npm run dev`. Esto inicia el servidor de Express. Ve a [http://localhost:3000](http://localhost:3000/) en tu navegador para ver la aplicación.

El servidor de desarrollo usa `nodemon`, que se reinicia automáticamente cada vez que detecta cambios en algún archivo.

<div id="explore-the-application">
  ### Explora la aplicación
</div>

Con la aplicación abierta en [http://localhost:3000](http://localhost:3000/), haz clic en el botón **Llamar a la API**. Deberías ver un mensaje en pantalla.

<Frame>![Pantalla Explora la aplicación de autenticación SPA con cookies](/docs/images/cdy7uua7fh8z/7JhdgzFLig2xcDle7ziiGp/c1db6272d20d4a291ba90a6968460109/image3.png)</Frame>

Ten en cuenta que pudiste llamar a la API sin haber iniciado sesión. Vamos a solucionarlo agregando middleware que exija que el usuario se autentique antes de poder llamar a la API.

<div id="set-up-environment">
  ### Configurar el entorno
</div>

Para que la aplicación funcione con la autenticación, `express-openid-connect` requiere que haya definidas algunas variables de entorno. En esta aplicación, estas variables pueden especificarse en un archivo `.env`. Crea un archivo `.env` en la raíz del directorio del proyecto y complétalo con lo siguiente:

```shell .env lines
ISSUER_BASE_URL="<YOUR OIDC URL>"
CLIENT_ID="<YOUR OIDC CLIENT ID>"
BASE_URL="http://localhost:3000"
APP_SESSION_SECRET="<YOUR SECRET VALUE>"
```

<div id="set-up-app-in-auth0">
  ### Configura la aplicación en Auth0
</div>

1. Ve a [Dashboard &gt; Applications &gt; Applications](https://manage.auth0.com/#/applications) y haz clic en **Create Application**.
2. Asigna un nombre a tu nueva aplicación, selecciona **aplicación web clásica** y haz clic en **Create**.
3. En **Settings** de tu nueva aplicación, agrega `http:/localhost:3000/callback` a **Allowed Callback URLs**.
4. Agrega `http:/localhost:3000` a **Allowed Logout URLs**.
5. Haz clic en **Save Changes**.
6. Ve a [Dashboard &gt; Authentication &gt; Social](https://manage.auth0.com/#/connections/social) y configura algunas conexiones sociales. Habilítalas para tu aplicación en las opciones de **aplicación** de la pestaña **Connections**. El ejemplo usa una base de datos de nombre de usuario y contraseña, Facebook, Google y X.
7. En la pantalla **Settings**, toma nota de la configuración de domain y del ID de cliente en la parte superior.
8. Hay dos valores que debes configurar como parte de la aplicación. Vuelve a abrir el archivo `.env` y establece estos valores:

<div id="run-the-app">
  ### Ejecuta la aplicación
</div>

1. Con la configuración del servidor y del entorno completada, busca la ventana del navegador donde está abierta la aplicación. Si cerraste el navegador y detuviste el servidor, ejecuta lo siguiente en la terminal para reiniciar la aplicación:
   `npm run dev`
2. Abre `http://localhost:3000` en el navegador. Desde la perspectiva de la interfaz de usuario, la aplicación debería verse igual; sin embargo, esta vez, al hacer clic en el botón **Call API**, deberías ver una advertencia de que el usuario no ha iniciado sesión. Ten en cuenta también que no verás el mensaje &quot;Hello, World&quot; como antes, ya que la llamada a la API fue rechazada.
3. Haz clic en **Log in now** para iniciar sesión. Una vez que te hayas autenticado, volverás a la aplicación y verás una UI actualizada que refleja tu nuevo estado de sesión. Deberías poder pulsar el botón **Call API** una vez más para hacer una llamada a la API al servidor, y ahora sí funcionará.
4. Haz clic en el enlace **Perfil** en la parte superior de la página para ver la información del usuario recuperada del token de identidad.
