De forma predeterminada, la [Authentication API](https://auth0.com/docs/api/authentication) usa los IDs de las organizaciones para identificar organizaciones específicas. Si es necesario, puedes configurar tu tenant para que también use los nombres de organización como identificador. Sin embargo, hay implicaciones de usabilidad y seguridad que debes tener en cuenta antes de habilitar esta funcionalidad. Para comprender mejor el posible impacto, revisa la sección [Consideraciones y recomendaciones](#considerations-and-recommendations).

<div id="how-it-works">
  ## Cómo funciona
</div>

Configurar su tenant para admitir nombres de organización en la Authentication API tiene los siguientes resultados:

* El parámetro `organization` de los endpoints [/authorize](https://auth0.com/docs/api/authentication#authorize-application) y [SAML](https://auth0.com/docs/api/authentication#saml) puede aceptar tanto nombres de organización como IDs.
* Los tokens de acceso y los <Tooltip tip="Token de ID: credencial destinada al propio cliente, no para acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=ID+tokens">tokens de ID</Tooltip> incluyen automáticamente las claims `org_name` y `org_id`.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  - [Organizations](/docs/es/manage-users/organizations/configure-organizations/create-organizations) puede configurarse con dos nombres: un valor Name obligatorio, que sirve como identificador lógico único, y un Display Name opcional y más fácil de usar. El parámetro `org_name` solo acepta el valor Name obligatorio y no admite valores de Display Name.
  - Esta funcionalidad se gestiona a nivel del tenant. No puede habilitarse de forma individual para organizaciones específicas.
</Callout>

Puede habilitar esta funcionalidad desde su <Tooltip tip="Auth0 Dashboard: el producto principal de Auth0 para configurar sus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip> o mediante la <Tooltip tip="Auth0 Dashboard: el producto principal de Auth0 para configurar sus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip>:

* **Auth0 Dashboard**: Seleccione [Settings](/docs/es/get-started/tenant-settings) en el menú lateral izquierdo y elija la pestaña **Advanced**. En la sección **Settings**, habilite el interruptor **Allow Organization Names in Authentication API**.
* **Management API**: Use el endpoint `PATCH /api/v2/tenants/settings` para establecer `allow_organization_name_in_authentication_api` en `true`. Para obtener más información, consulte la documentación de la [Management API](https://auth0.com/docs/api/management/v2).

<div id="example-flow">
  ### Flujo de ejemplo
</div>

El siguiente ejemplo muestra un flujo de código de autorización que utiliza nombres de organización.

1. Llama al endpoint `/authorize` y pasa el nombre de tu organización en el parámetro `organization`:

export const codeExample1 = `https://{yourDomain}/authorize?
    response_type=code&
    client_id={yourClientId}&
    redirect_uri={https://yourApp/callback}&
    scope={scope}&
    state={state}&
    organization={yourOrganizationName}`;

<AuthCodeBlock children={codeExample1} language="http" />

2. Después de obtener el código de autorización, llama al endpoint `POST /oauth/token` para obtener tokens de acceso y de ID:

export const codeExample2 = `curl --request POST \
  --url 'https://{yourDomain}/oauth/token' \
  --header 'content-type: application/x-www-form-urlencoded' \
  --data grant_type=authorization_code \
  --data 'client_id={yourClientId}' \
  --data 'client_secret={yourClientSecret}' \
  --data 'code=yourAuthorizationCode}' \
  --data 'redirect_uri={https://yourApp/callback}'`;

<AuthCodeBlock children={codeExample2} language="bash" />

3. Los tokens decodificados que se devuelven contienen tanto los claims `org_id` como `org_name`:

```json lines
{
    "sub": "google-oauth2|10...17",
    "aud": [
        "https://yourApp"
    ],
    "iat": 1686840988,
    "exp": 1686927388,
    "azp": "Suo...qTd",
    "scope": "openid profile",
    "org_id": "{yourOrganizationId}",
    "org_name": "{yourOrganizationName}"
}
```

<div id="considerations-and-recommendations">
  ## Consideraciones y recomendaciones
</div>

Antes de usar nombres de organización en la Authentication API, es importante comprender las principales diferencias entre los nombres de organización y los IDs.

A diferencia de los IDs de organización (que permanecen estáticos), puedes cambiar el nombre de una organización en cualquier momento después de crearla. Además, puedes reutilizar nombres de organización dentro de un mismo tenant, siempre que solo esté asignado a **exactamente una** organización a la vez. En la práctica, esto significa que puedes cambiar el nombre de una de tus organizaciones y reutilizar su nombre original para otra organización de tu tenant. Los nombres de organización solo son únicos dentro de un mismo tenant; el mismo nombre puede usarse para dos o más organizaciones en distintos tenants.

En general, se recomienda usar IDs de organización para validar tokens. Sin embargo, si usar nombres de organización resulta más adecuado para tu caso de uso, considera las implicaciones que se indican a continuación al implementar la funcionalidad.

<div id="usability-and-security-considerations">
  ### Consideraciones de usabilidad y seguridad
</div>

Ten en cuenta los posibles impactos que se indican a continuación al usar nombres de organización para solicitar y validar tokens:

* **Los nombres de organización pueden reutilizarse**: Los tokens de larga duración no caducan cuando una organización cambia su nombre, y los claims de `org_name` en esos tokens conservan su valor original. Si más adelante otra organización reutiliza el nombre original, esos tokens podrían conceder a los usuarios acceso no autorizado a datos y recursos gestionados por la nueva organización.
* **Los nombres de organización solo son únicos dentro de un mismo tenant**: Si tu API no verifica los claims de `iss` (issuer) en los tokens, una organización con el mismo nombre en otro tenant podría generar tokens que tu API aceptaría por error.
* **Los nombres de organización pueden cambiarse**: Si cambias el nombre de una organización, tus aplicaciones deben proporcionar el nuevo nombre de la organización en las requests a Authentication API. Dado que los tokens pueden ser de larga duración, es posible que el claim `org_name` de un token ya no coincida con el nombre actual de la organización, lo que podría impedir que las aplicaciones concedan acceso a la organización correcta.

<div id="recommended-best-practices">
  ### Prácticas recomendadas
</div>

Debido a los posibles efectos en la seguridad y la usabilidad, se recomienda usar IDs en lugar de nombres para validar tokens al trabajar con organizaciones. Si decides usar nombres de organización, sigue las prácticas recomendadas a continuación para obtener una experiencia óptima:

* Valida siempre el claim `iss` para asegurarte de que un token haya sido emitido por tu tenant de Auth0.
* Evita reutilizar nombres de organización que hayan existido anteriormente en tu tenant. Para evitar la reutilización y garantizar que los tokens emitidos previamente no puedan usarse para acceder a otras organizaciones, mantén un registro preciso y actualizado de los nombres históricos de las organizaciones.
* Evita cambiar el nombre de las organizaciones una vez que estén en uso, salvo que sea absolutamente necesario. Si decides cambiar el nombre de una organización, ten en cuenta que los tokens de acceso y de ID existentes no incluyen automáticamente el nuevo nombre de la organización. Después de cambiar el nombre de una organización, asegúrate de pedir a los usuarios que vuelvan a iniciar sesión.
