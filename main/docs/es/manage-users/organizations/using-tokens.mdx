La mayoría de los tokens de identidad (ID) y <Tooltip tip="Token de acceso: credencial de autorización, en forma de una cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=access+tokens">tokens de acceso</Tooltip> que devuelve Auth0 son <Tooltip tip="Token de acceso: credencial de autorización, en forma de una cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=JSON+Web+Tokens">JSON Web Tokens</Tooltip> (JWT) que contienen una variedad de claims, es decir, fragmentos de información que se afirman sobre un subject. Por ejemplo, un [token de identidad](/docs/es/secure/tokens/id-tokens) (que siempre es un JWT) puede contener un claim llamado `name` que afirma que el nombre del usuario que se autentica es &quot;John Doe&quot;.

Hay dos tipos de claims en los JWT:

* **Registered**: Claims definidas por la [especificación JWT](https://tools.ietf.org/html/rfc7519) para garantizar la interoperabilidad con applications de terceros o externas. Las claims estándar de [OpenID Connect (OIDC)](/docs/es/authenticate/protocols/openid-connect-protocol) son claims reservadas.
* **Custom**: Claims que defines tú mismo. Estas claims pueden ser claims públicas no registradas y resistentes a colisiones, o claims privadas no registradas y no públicas sujetas a colisión. Asigna nombres a estas claims con cuidado, por ejemplo, mediante el [uso de espacios de nombres](/docs/es/secure/tokens/json-web-tokens/create-custom-claims), para evitar colisiones con claims reservadas u otras claims personalizadas. Puede resultar complicado manejar dos claims con el mismo nombre que contienen información diferente.

Para obtener más información sobre las claims, consulta [JSON Web Token Claims](/docs/es/secure/tokens/json-web-tokens/json-web-token-claims).

<div id="authenticate-users-through-an-organization">
  ## Autenticar usuarios a través de una organización
</div>

Para autenticar a un usuario a través de una organización, se agrega un parámetro `organization` a una solicitud al endpoint `/authorize`. A continuación, se muestran ejemplos de tokens devueltos cuando un usuario inicia sesión a través de Organizations.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  De forma predeterminada, los tokens de ID y de acceso solo incluyen IDs de organización. Sin embargo, puedes configurar tu tenant para permitir el uso de nombres de organización en la Authentication API. Cuando se configura así, los tokens de ID y de acceso contienen las claims `org_id` y `org_name`. Para obtener más información, consulta [Use Organization Names in Authentication API](/docs/es/manage-users/organizations/configure-organizations/use-org-name-authentication-api).
</Callout>

<div id="id-token">
  ### Token de identidad
</div>

En el siguiente ejemplo, observa que `https://marketplace/roles` y `https://namespace.exampleco.com/` son claims personalizadas añadidas al token, mientras que las demás claims incluidas son estándar.

```json lines
{
  "https://marketplace/roles": [
    "marketplace-administrator"
  ],
  "https://namespace.exampleco.com": "my custom claim",
  "nickname": "firstName.lastName",
  "name": "firstName.lastName@email.com",
  "picture": "https://s.gravatar.com/avatar/638",
  "updated_at": "2021-03-23T11:34:14.566z",
  "email": "username@exampleco.com",
  "email_verified": true,
  "sub": "auth0|602c0dcab993d10073daf680",
  "org_id": "org_9ybsU1dN2dKfDkBi"
}
```

<div id="access-token">
  ### Token de acceso
</div>

```json lines
{
  "iss": "https://exampleco.auth0.com/",
  "sub": "auth0|602c0dcab993d10073daf680",
  "aud": [
    "https://example-api/",
    "https://exampleco.auth0.com/userinfo"  
  ],
  "iat": 1616499255,
  "exp": 1616585655,
  "azp": "ENDmmAJsbwI1hOG1KPJddQ8LHjV6kLkV",
  "scope": "openid profile email",
  "org_id": "org_9ybsU1dN2dKfDkBi",
  "permissions": [
    "delete:stuff",
    "read:stuff",
    "write:stuff"  
  ]
}
```

<div id="machine-to-machine-access-to-an-organization">
  ## Acceso de máquina a máquina a una organización
</div>

En los casos de uso de máquina a máquina, se agrega un parámetro `organization` a la solicitud de credenciales de cliente al endpoint `/oauth/token` para que una aplicación pueda obtener un token de acceso para sí misma en lugar de para un usuario.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  De forma predeterminada, los tokens de ID y de acceso solo incluyen IDs de organización. Sin embargo, puedes configurar tu tenant para permitir el uso de nombres de organización en la Authentication API. Una vez configurado, los tokens de ID y de acceso contienen tanto los claims `org_id` como `org_name`. Para obtener más información, consulta [Use Organization Names in the Authentication API](/docs/es/manage-users/organizations/configure-organizations/use-org-name-authentication-api).
</Callout>

La siguiente muestra de código es un ejemplo de token de acceso devuelto en casos de uso de máquina a máquina:

```json lines
{
  "iss": "https://exampleco.auth0.com/",
  "sub": "CS2MNgcX1VZFCJaEzfKw2VPAAS0gzhqP@clients",
  "aud": "https://example-api",
  "iat": 1727782196,
  "exp": 1727868596,
  "scope": "scope1 scope2",
  "org_id": "org_vIK75NKFvaozQsFy",
  "org_name": "acme",
  "gty": "client-credentials",
  "azp": "CS2MNgcX1VZFCJaEzfKw2VPAAS0gzhqP"
}
```

<div id="validate-tokens">
  ## Validar tokens
</div>

Cuando se añade el parámetro `organization` a una llamada al endpoint `/authorize` o al endpoint `/oauth/token`, los SDKs de Auth0 validan automáticamente el claim `org_id`, que se devuelve como parte de cualquier token generado. Sin embargo, por motivos de seguridad, se debe realizar una validación adicional al recibir tokens.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Si has configurado tu tenant para permitir el uso de nombres de organización en la Authentication API, los tokens de ID y de acceso contienen tanto el claim `org_id` como el claim `org_name`. Si está presente, valida el claim `org_name` además de `org_id` para asegurarte de que los valores recibidos correspondan a una entidad de confianza.

  En general, usar ID de organización es el método preferido para validar tokens. Sin embargo, los nombres de organización pueden usarse si se ajustan mejor a tu caso de uso. Para comprender las posibles implicaciones de usar nombres de organización para validar tokens, consulta [Use Organization Names in Authentication API](/docs/es/manage-users/organizations/configure-organizations/use-org-name-authentication-api).
</Callout>

**Para aplicaciones web:**

Si no se pasó ningún parámetro `organization` al endpoint `/authorize`, pero hay un claim `org_id` presente en el <Tooltip tip="token de ID: Credencial destinada al propio cliente, en lugar de para acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=ID+token">token de identidad</Tooltip>, tu aplicación debe validar el claim para asegurarse de que el valor recibido sea el esperado o conocido y de que corresponda a una entidad en la que tu aplicación confía, como un cliente de pago. Si el claim no puede validarse, la aplicación debe considerar inválido el token.

**Para APIs:**

Si hay un claim `org_id` presente en el token de acceso, tu API debe validar el claim para asegurarse de que el valor recibido sea el esperado o conocido y de que corresponda a una entidad en la que tu aplicación confía, como un cliente de pago. Si el claim no puede validarse, la API debe considerar inválido el token.

En particular:

* Se debe comprobar el claim `iss` (issuer) para garantizar que el token fue emitido por Auth0.
* Se debe comprobar el claim `org_id` para garantizar que sea un valor que la aplicación ya conoce. Esto puede validarse con una lista conocida de ID de organización, o comprobarse junto con la URL actual de la solicitud. Por ejemplo, el subdominio puede indicar qué organización debe usarse al validar el token de ID.

Normalmente, validar solo el issuer bastaría para garantizar que el token fue emitido por Auth0. Sin embargo, en el caso de las organizaciones, se deben realizar comprobaciones adicionales para asegurarse de que la organización dentro de tu tenant de Auth0 sea la esperada.

También es muy importante que tus servidores de la API segmenten el acceso a los datos y recursos en función de `org_id`. Esto garantiza que solo se pueda acceder a la información de una organización determinada, o modificarla, cuando en el token de acceso se reciba el valor `org_id` correspondiente a esa organización.
