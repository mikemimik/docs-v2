<div id="short-lived-sessions">
  ## Sesiones de corta duración
</div>

Este flujo de trabajo muestra cómo debe implementarse el SDK auth0-spa-js para admitir la gestión de sesiones en múltiples sitios. En este escenario, se asume que el tiempo de espera por inactividad del <Tooltip tip="Inicio de sesión único (inicio de sesión único): servicio que, después de que un usuario inicia sesión en una aplicación, inicia automáticamente la sesión de ese usuario en otras aplicaciones." cta="Ver glosario" href="/docs/es/glossary?term=SSO">inicio de sesión único</Tooltip> del tenant está configurado en 300 segundos y que la expiración del <Tooltip tip="token de identidad: credencial destinada al propio cliente, en lugar de acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=ID+Token">token de identidad</Tooltip> de cada aplicación SPA está configurada en 150 segundos. Esto se considera una sesión de &quot;corta duración&quot;.

<div id="sdk-features">
  ### Funcionalidades del SDK
</div>

<div id="pkce-flow">
  #### Flujo de PKCE
</div>

Para todos los métodos de obtención de un token de identidad o un <Tooltip tip="Token de acceso: credencial de autorización, en forma de una cadena opaca o JWT, usada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=Access+Token">Token de acceso</Tooltip>, el SDK se encarga de toda la complejidad de Proof Key for Code Exchange. No se necesita ninguna configuración ni esfuerzo adicional para que funcione.

<div id="deep-linking">
  #### Enlaces profundos
</div>

Para mejorar la experiencia del usuario, el SDK incluye un parámetro `appState` para el método `loginWithRedirect()`. La información sobre la aplicación actual se incluye como parte de la solicitud al servidor de Auth y se devuelve tras una autenticación correcta. Esto permite que el usuario continúe su recorrido sin interrupciones.

En el Quickstart, el componente `PrivateRoute` establece un parámetro state con el valor `targetUrl`, y la función de devolución de llamada `onRedirectCallback` de `index.js` extrae este valor para redirigir al usuario cuando se completa la autenticación.

<div id="token-storage">
  #### almacenamiento de tokens
</div>

Para almacenar los tokens devueltos de la forma más segura posible, todos los tokens se guardan en una caché local. El token de identidad y el token de acceso se almacenan en pares, y los valores de <Tooltip tip="Audience: Identificador único del destinatario de un token emitido. En un token, se denomina aud; su valor contiene el ID de una aplicación (ID de cliente) para un token de identidad o de una API (identificador de API) para un token de acceso." cta="Ver glosario" href="/docs/es/glossary?term=audience">audience</Tooltip> y scope se usan para recuperar los tokens cuando sea necesario.

Además, los tokens de la caché se eliminan en cuanto expire el token de identidad o el token de acceso, de modo que, si un token está en la caché, puede asumirse que sigue siendo válido.

<div id="call-apis">
  #### Llamar a las API
</div>

El método `getTokenSilently()` se usa para aprovechar primero la caché de tokens y, si no hay ninguna, iniciar un iframe invisible para obtener un token nuevo. Para ello, todas las solicitudes a las API pueden usar este método para crear el encabezado de bearer token sin necesidad de lógica adicional para gestionar tokens vencidos.

En el Quickstart, la vista `ExternalService` hace una solicitud a la API de express mediante esta funcionalidad.

<div id="warn-users-to-continue-their-sessions">
  #### Advertir a los usuarios para que continúen sus sesiones
</div>

En caso de que un usuario no haya realizado ninguna acción que haga que se actualice la sesión de Auth0, Auth0 recomienda mostrarle una advertencia para que decida explícitamente si desea continuar su sesión.

El objetivo de este enfoque es permitir que la sesión quede inactiva si el usuario ya no está presente, pero, en caso contrario, ofrecer una forma de activar la renovación silenciosa del token para que pueda continuar su sesión sin que se le vuelvan a solicitar credenciales.

Para obtener más información sobre los temporizadores de inactividad y los modales de tiempo de espera, lea [URL de cierre de sesión específicas de la aplicación](/docs/es/manage-users/sessions/configure-session-lifetime-settings).

<div id="example-workflow">
  ### Ejemplo de flujo de trabajo
</div>

1. Autenticación inicial
2. Mantenimiento de la sesión de Auth0
3. Seamless inicio de sesión único
4. Se le solicita al usuario que extienda la sesión
5. El usuario cierra sesión explícitamente en la aplicación
6. El usuario vuelve a la aplicación inicial tras cerrar sesión

<div id="initial-authentication">
  #### Autenticación inicial
</div>

1. Se abre una nueva pestaña
2. Se solicita iniciar sesión
3. El usuario introduce sus credenciales
4. Se establece una cookie de inicio de sesión único (con expiración)
5. Se realiza el intercambio de tokens

<div id="maintain-auth0-session">
  #### Mantener la sesión de Auth0
</div>

1. El usuario solicita datos de un recurso protegido
2. Se llama a `getTokenSilently()`
3. Se recupera el recurso
4. El usuario actualiza los datos del recurso protegido
5. Se llama a `getTokenSilently()`

   1. Se abre un iframe
   2. Se realiza el intercambio de tokens
6. Se actualiza el recurso

<Frame>![Diagrama del mantenimiento de la sesión](/docs/images/cdy7uua7fh8z/7CI29ouvhuJ4TFC1G2C5n8/4d2bd0c9372a6cc16e75fc76f438a0bd/maintain-auth0-session.png)</Frame>

<div id="seamless-sso">
  #### Seamless inicio de sesión único
</div>

1. El usuario navega a una ruta privada
2. Compruébalo con `isAuthenticated()`
3. Si es `false`, `loginWithRedirect()`

<Frame>![Diagrama del flujo de Seamless inicio de sesión único para sesiones](/docs/images/cdy7uua7fh8z/38heJlnN0letWMVhpTtXad/4d4127150fb818522ea29eea28ccc00e/seamless-sso.png)</Frame>

<div id="prompt-user-to-extend-session">
  #### Solicitar al usuario que extienda la sesión
</div>

1. A los 240 segundos, mostrar al usuario un modal para mantener la sesión activa durante 60 segundos
2. Si elige mantener la sesión, `getTokenSilently()`

<Frame>![Diagrama de cómo solicitar a un usuario que mantenga una sesión activa](/docs/images/cdy7uua7fh8z/6kH2G3oclCfSudwd0hJEUM/70651da37771c9945c5a34dcad409d9f/prompt-user-extend-session.png)</Frame>

<div id="user-explicitly-logs-out-of-application">
  #### El usuario cierra sesión explícitamente en la aplicación
</div>

1. El usuario decide cerrar sesión
2. Se llama a `logout()`

   1. Se borra la caché de tokens
   2. Se llama a `/oidc/logout`
   3. Se borra la cookie de inicio de sesión único y se eliminan los datos de la sesión
   4. Se redirige al usuario a la página de cierre de sesión

<Frame>![Diagrama de un usuario cerrando sesión](/docs/images/cdy7uua7fh8z/4b55FKM2vZBoskXTPY4nxs/1a9e69437677f2cfac9a3d8e6bff7ffe/user-explicitly-logs-out-of-app.png)</Frame>

<div id="user-returns-to-initial-application-after-logging-out">
  #### El usuario vuelve a la aplicación inicial tras cerrar sesión
</div>

1. El usuario solicita datos de un recurso protegido
2. Se llama a `getTokenSilently()`
3. Comportamiento según la aplicación

<Frame>![Diagrama del usuario que vuelve a la aplicación tras cerrar sesión](/docs/images/cdy7uua7fh8z/3geXGwsXkIgFkwopoGoLMx/01685b6b8e8473c6ce6f26e59b402cf3/user-returns-to-initial-app.png)</Frame>

<div id="long-lived-sessions">
  ## Sesiones de larga duración
</div>

Auth0 admite sesiones de larga duración para planes empresariales. Con las sesiones de larga duración, puedes configurar límites de sesión de hasta 100 días de inactividad (`idle timeout`) y hasta un año de duración total (`absolute timeout`). Si trabajas con plazos trimestrales, mensuales o de otro tipo, esto te permite reducir la fricción para los usuarios finales y proporcionar acceso a contenido y capacidades de bajo riesgo. Además, las empresas de medios pueden aprovechar las sesiones de larga duración para mejorar la experiencia de usuario mediante un acceso al contenido sin interrupciones. También puedes optar entre sesiones de larga duración y la validación de contraseñas según tus requisitos de experiencia de usuario y seguridad.

Los detalles del flujo de trabajo cambiarían en el caso de una sesión de larga duración, en la que la sesión de la aplicación probablemente sería más corta que la sesión de [inicio de sesión único (SSO)](/docs/es/authenticate/single-sign-on).

Para obtener más información, consulta [Configurar los límites de duración de la sesión](/docs/es/manage-users/sessions/configure-session-lifetime-settings) y [Actualizar la duración del token de acceso](/docs/es/secure/tokens/access-tokens/update-access-token-lifetime).
