<div id="short-lived-sessions">
  ## Sesiones de corta duración
</div>

Este flujo de trabajo muestra cómo debe implementarse el SDK auth0-spa-js para admitir la gestión de sesiones multisitio. En este escenario, se asume que el tiempo de espera por inactividad del <Tooltip tip="inicio de sesión único (SSO): Servicio que, después de que un usuario inicia sesión en una aplicación, lo inicia automáticamente en otras aplicaciones." cta="Ver glosario" href="/docs/es/glossary?term=SSO">SSO</Tooltip> del tenant está configurado en 300 segundos y que la expiración del <Tooltip tip="token de ID: Credencial destinada al propio cliente, en lugar de para acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=ID+Token">token de ID</Tooltip> de cada aplicación SPA está configurada en 150 segundos. Esto se considera una sesión &quot;de corta duración&quot;.

<div id="sdk-features">
  ### Funcionalidades del SDK
</div>

<div id="pkce-flow">
  #### Flujo PKCE
</div>

Para todos los métodos para obtener un token de ID o <Tooltip tip="token de acceso: Credencial de Authorization, en forma de una cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=Access+Token">token de acceso</Tooltip>, el SDK se encarga de toda la complejidad del flujo de trabajo de Proof Key for Code Exchange. No se necesita ningún esfuerzo ni configuración adicional para que funcione.

<div id="deep-linking">
  #### Enlaces profundos
</div>

Para mejorar la experiencia del usuario, el SDK incluye un parámetro `appState` para el método `loginWithRedirect()`. Los detalles de la aplicación actual se incluyen como parte de la solicitud al servidor de Auth y se devuelven cuando la autenticación se completa correctamente. Esto permite que el recorrido del usuario continúe sin interrupciones.

En el Quickstart, el componente `PrivateRoute` establece un parámetro `state` con el valor `targetUrl`, y la función `onRedirectCallback` de `index.js` extrae este valor para redirigir al usuario cuando se completa la autenticación.

<div id="token-storage">
  #### Almacenamiento de tokens
</div>

Para almacenar los tokens devueltos de la forma más segura posible, todos se guardan en una caché local. Los tokens de ID y de acceso se almacenan en pares, y los valores de <Tooltip tip="Audiencia: identificador único de la audiencia de un token emitido. Se denomina aud en un token y su valor contiene el ID de una aplicación (ID de cliente) para un token de ID o de una API (identificador de API) para un token de acceso." cta="Ver glosario" href="/docs/es/glossary?term=audience">audiencia</Tooltip> y scope se usan para recuperar los tokens cuando sea necesario.

Además, los tokens almacenados en caché se eliminan en cuanto vence el token de ID o el token de acceso, de modo que, si un token está en la caché, puede asumirse que sigue siendo válido.

<div id="call-apis">
  #### Hacer llamadas a las API
</div>

El método `getTokenSilently()` se usa para aprovechar primero la caché de tokens y, si no hay ninguna, inicia un iframe invisible para obtener un token nuevo. Para ello, todas las solicitudes a las API pueden usar este método para construir el encabezado del bearer token sin necesidad de lógica adicional para gestionar tokens vencidos.

En el Quickstart, la vista `ExternalService` hace una solicitud a la API de express mediante esta funcionalidad.

<div id="warn-users-to-continue-their-sessions">
  #### Advierta a los usuarios para que continúen sus sesiones
</div>

Si un usuario no ha realizado ninguna acción que actualice la sesión de Auth0, Auth0 recomienda mostrarle una advertencia para que elija continuar explícitamente con su sesión.

El objetivo de este enfoque es permitir que la sesión quede inactiva si el usuario ya no está presente, pero, de lo contrario, ofrece una forma de activar la actualización silenciosa del token para que pueda continuar su sesión sin que se le vuelvan a solicitar las credenciales.

Para obtener más información sobre los temporizadores de inactividad y los cuadros de diálogo de timeout, lea [URL de cierre de sesión específicas de la aplicación](/docs/es/manage-users/sessions/configure-session-lifetime-settings).

<div id="example-workflow">
  ### Ejemplo de flujo de trabajo
</div>

1. Autenticación inicial
2. Mantenimiento de la sesión de Auth0
3. Seamless SSO
4. Se le pide al usuario que extienda la sesión
5. El usuario cierra sesión explícitamente en la aplicación
6. El usuario vuelve a la aplicación inicial después de cerrar sesión

<div id="initial-authentication">
  #### Autenticación inicial
</div>

1. Se abre una nueva pestaña
2. Se solicita el inicio de sesión
3. El usuario introduce sus credenciales
4. Se establece la cookie de SSO (con vencimiento)
5. Se realiza el intercambio de tokens

<div id="maintain-auth0-session">
  #### Mantener la sesión de Auth0
</div>

1. El usuario solicita datos de un recurso protegido
2. Se llama a `getTokenSilently()`
3. Se obtiene el recurso
4. El usuario actualiza los datos del recurso protegido
5. Se llama a `getTokenSilently()`

   1. Se abre un iframe
   2. Se realiza el intercambio de tokens
6. Se actualiza el recurso

<Frame>![Diagrama de mantenimiento de la sesión](/docs/images/cdy7uua7fh8z/7CI29ouvhuJ4TFC1G2C5n8/4d2bd0c9372a6cc16e75fc76f438a0bd/maintain-auth0-session.png)</Frame>

<div id="seamless-sso">
  #### Seamless SSO
</div>

1. El usuario navega a una ruta privada
2. Comprueba con `isAuthenticated()`
3. Si devuelve `false`, `loginWithRedirect()`

<Frame>![Diagrama del flujo de Seamless SSO para sesiones](/docs/images/cdy7uua7fh8z/38heJlnN0letWMVhpTtXad/4d4127150fb818522ea29eea28ccc00e/seamless-sso.png)</Frame>

<div id="prompt-user-to-extend-session">
  #### Solicitar al usuario que extienda la sesión
</div>

1. A los 240 segundos, solicitar al usuario que mantenga la sesión activa con una ventana modal que dure 60 segundos
2. Si elige mantener la sesión, `getTokenSilently()`

<Frame>![Diagrama de cómo se solicita a un usuario que mantenga una sesión](/docs/images/cdy7uua7fh8z/6kH2G3oclCfSudwd0hJEUM/70651da37771c9945c5a34dcad409d9f/prompt-user-extend-session.png)</Frame>

<div id="user-explicitly-logs-out-of-application">
  #### El usuario cierra sesión explícitamente en la aplicación
</div>

1. El usuario decide cerrar sesión
2. Se llama a `logout()`

   1. Se vacía la caché de tokens
   2. Se llama a `/oidc/logout`
   3. Se borra la cookie de SSO y se eliminan los datos de la sesión
   4. Se redirige al usuario a la página de cierre de sesión

  #### El usuario vuelve a la aplicación inicial después de cerrar sesión
</div>

1. El usuario solicita datos de un recurso protegido
2. Se llama a `getTokenSilently()`
3. El comportamiento depende de la aplicación

<div id="long-lived-sessions">
  ## Sesiones de larga duración
</div>

Auth0 admite sesiones de larga duración en los planes empresariales. Con las sesiones de larga duración, puedes configurar límites de sesión de hasta 100 días de inactividad (`idle timeout`) y hasta un año de duración total (`absolute timeout`). Si manejas plazos trimestrales, mensuales o de otro tipo, esto te permite reducir la fricción para los usuarios finales y dar acceso a contenido y funcionalidades de bajo riesgo. Además, las empresas de medios pueden aprovechar las sesiones de larga duración para mejorar la experiencia de usuario mediante un acceso fluido al contenido. También puedes elegir entre sesiones de larga duración y validación de contraseñas en función de tus requisitos de experiencia de usuario y seguridad.

Los detalles del flujo de trabajo cambiarían en el caso de una sesión de larga duración, en la que la sesión de la aplicación probablemente sería más corta que la sesión de [inicio de sesión único (SSO)](/docs/es/authenticate/single-sign-on).

Para obtener más información, consulta [Configurar los límites de duración de la sesión](/docs/es/manage-users/sessions/configure-session-lifetime-settings) y [Actualizar la duración del token de acceso](/docs/es/secure/tokens/access-tokens/update-access-token-lifetime).
