La My Account API de Auth0 proporciona un conjunto específico de endpoints para que los usuarios administren la información de su propia cuenta. Los clientes pueden usar estas API para crear experiencias de autoservicio en sus aplicaciones o ir agregando datos a una cuenta de usuario de forma progresiva.

La My Account API funciona en el contexto del usuario que ha iniciado sesión actualmente y puede usarse directamente en aplicaciones orientadas al usuario.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  **Uso del dominio de Auth0 frente a un dominio personalizado**

  La My Account API admite usar tu dominio canónico de Auth0 o tu dominio personalizado, pero debes usar el mismo en todo el proceso, lo que incluye:

  * Obtener un token de acceso
  * Establecer el valor de `audience`
  * Llamar al endpoint de la My Account API

  Para obtener más información, lee [Custom Domains](/docs/es/customize/custom-domains).
</Callout>

<div id="activate-the-my-account-api">
  ## Activar My Account API
</div>

Puedes activar My Account API para tu tenant en el <Tooltip tip="Auth0 Dashboard: el producto principal de Auth0 para configurar tus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip>:

1. Ve a **Applications &gt; APIs**.
2. Busca el banner de **MyAccount API**.
3. Selecciona **Activar**.

<Frame>![](/docs/images/cdy7uua7fh8z/DGkDQNIHcegN8vtQyEvF3/a2e98b06867701a66fa1f1b3df1c996b/My_Account_API_-_Activate.png)</Frame>

De forma predeterminada, My Account API se crea con las siguientes políticas de acceso de las aplicaciones a la API:

* `require_client_grant` para los flujos de usuario
* `deny_all` para los flujos de cliente (máquina a máquina)

Para que una aplicación acceda a My Account API en nombre del usuario, debes crear explícitamente una autorización de cliente para esa aplicación, lo que te permite definir los scopes máximos que la aplicación puede solicitar. Como alternativa, puedes cambiar la política de los flujos de acceso de usuario a `allow_all`, lo que permite que cualquier aplicación de tu tenant solicite cualquier scope de My Account API.

Auth0 no recomienda usar `allow_all` para los flujos de acceso de usuario porque My Account API expone información y operaciones sensibles. Debes aplicar el principio de privilegio mínimo con My Account API para asegurarte de que las aplicaciones solo obtengan acceso a lo que realmente necesitan, minimizando así los posibles riesgos de seguridad.

Los permisos finales concedidos a la aplicación estarán determinados por la intersección entre los scopes permitidos por la política de acceso de la aplicación a la API, los permisos de control de acceso basado en roles (RBAC) asignados al usuario final y el consentimiento otorgado por el usuario (si corresponde).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  No puedes actualizar la política de acceso de cliente a My Account API para la aplicación, lo que significa que no puedes acceder a My Account API mediante el Client Credentials Flow.
</Callout>

Para obtener más información sobre cómo administrar las políticas de acceso de las aplicaciones a la API y sus autorizaciones de cliente asociadas, consulta [Application Access to APIs: Client Grants](/docs/es/get-started/applications/application-access-to-apis-client-grants).

<div id="default-policy-setting">
  ## Configuración de la política predeterminada
</div>

<Warning>
  La garantía de autenticación para la **My Account API** se encuentra actualmente en acceso anticipado con una política de opción única. Al usar esta funcionalidad, aceptas los términos aplicables de la prueba gratuita del [Master Subscription Agreement](https://www.okta.com/legal) de Okta. Para obtener más información sobre el ciclo de lanzamientos de productos de Auth0, consulta [Product Release Stages](/docs/es/troubleshoot/product-lifecycle/product-release-stages). Para participar en el programa, ponte en contacto con el [equipo de soporte de Auth0](https://support.auth0.com).
</Warning>

La política predeterminada proporciona garantía de autenticación integrada para la My Account API al requerir [autenticación escalonada](/docs/es/secure/multi-factor-authentication/step-up-authentication). Cuando está habilitada, Auth0 exige automáticamente que los usuarios se hayan autenticado recientemente y con un segundo factor.

La política exige 2FA en un plazo de 15 minutos. Auth0 aplica esta regla al iniciar sesión y en cada intercambio de token de actualización:

* Si un usuario tiene un factor de MFA ya configurado, debe completar la 2FA al iniciar sesión y de nuevo cuando sus tokens tengan más de 15 minutos.
* Si un usuario no tiene ningún factor que pueda registrar, Auth0 permite el acceso inicial, pero devuelve un error `unmet_authentication_requirements` en los intercambios de token de actualización después de 15 minutos.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  La política predeterminada no es compatible con Classic Login. Habilita esta funcionalidad si tu tenant usa Universal Login o un flujo integrado compatible ([flujo de contraseña del propietario del recurso](/docs/es/get-started/authentication-and-authorization-flow/resource-owner-password-flow) o [claves de acceso nativas](/docs/es/authenticate/database-connections/passkeys/native-passkeys-api)).
</Callout>

<div id="enable-default-policy">
  ### Habilitar la política predeterminada
</div>

Para habilitar la política predeterminada de la **My Account API**:

1. Ve a [**Applications &gt; APIs**](https://manage.auth0.com/#/apis) y selecciona la **My Account API**.
2. Selecciona la pestaña **Configuración**.
3. En **Política predeterminada**, activa **Requerir 2FA**.
4. Selecciona **Guardar**.

Cuando la política predeterminada está habilitada en tu tenant, se adjunta automáticamente cada vez que se crea una nueva **My Account API**.

<div id="authentication-requirements-hierarchy">
  ### Jerarquía de los requisitos de autenticación
</div>

La política predeterminada se sitúa entre la política de MFA a nivel del tenant y cualquier lógica de MFA que definas en Actions:

1. **Política de MFA del tenant** — la configuración predeterminada base que se aplica a toda la autenticación de tu tenant
2. **Política predeterminada** — sobrescribe la del tenant específicamente para My Account API
3. **Actions** — cualquier comando de MFA en Actions siempre tiene prioridad sobre ambas

<div id="default-policy-behavior">
  ### Comportamiento de la política predeterminada
</div>

El comportamiento depende de si el usuario tiene un segundo factor que puede configurar.

**Usuarios con un factor de MFA configurado**

Para usuarios que hayan configurado TOTP, correo electrónico u otro factor compatible:

1. Al iniciar sesión, Auth0 verifica al usuario con su factor configurado antes de emitir los tokens.
2. El token de actualización registra el método de autenticación y la marca de tiempo (AMR).
3. En el intercambio de token de actualización dentro de los 15 minutos posteriores al último desafío, Auth0 emite un nuevo token de acceso sin volver a solicitar la verificación.
4. En el intercambio de token de actualización después de 15 minutos, Auth0 vuelve a verificar al usuario antes de emitir los tokens.

**Usuarios sin un factor de MFA configurado**

Para usuarios sin correo electrónico verificado y sin ningún factor configurado:

1. Al iniciar sesión, Auth0 permite el acceso sin un segundo factor.
2. En el intercambio de token de actualización dentro de los 15 minutos, Auth0 emite un nuevo token de acceso sin desafío.
3. En el intercambio de token de actualización después de 15 minutos, Auth0 devuelve un error `unmet_authentication_requirements`.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Cuando se devuelve `unmet_authentication_requirements` en un intercambio de token de actualización, el token no se puede renovar. Tu aplicación debe reiniciar el flujo de autenticación completo para obtener nuevos tokens.

  Se devuelve el mismo error en un inicio de sesión silencioso (`prompt=none`) cuando el usuario no puede cumplir la política después de 15 minutos.
</Callout>

<div id="audience">
  ### Audiencia
</div>

La <Tooltip tip="Audiencia: identificador único de la audiencia de un token emitido. Se denomina aud en un token; su valor contiene el ID de una aplicación (ID de cliente) para un token de ID o de una API (identificador de API) para un token de acceso." cta="Ver glosario" href="/docs/es/glossary?term=audience">audiencia</Tooltip> de My Account API es `https://{yourDomain}/me/`.

<div id="scope">
  ### Scope
</div>

My Account API admite los siguientes scopes:

| Scope                              | Descripción                                                           |
| ---------------------------------- | --------------------------------------------------------------------- |
| `create:me:authentication_methods` | Permite al usuario registrar un nuevo método de autenticación.        |
| `read:me:authentication_methods`   | Permite al usuario ver los métodos de autenticación existentes.       |
| `update:me:authentication_methods` | Permite al usuario modificar los métodos de autenticación existentes. |
| `delete:me:authentication_methods` | Permite al usuario modificar los métodos de autenticación existentes. |
| `read:me:factors`                  | Permite al usuario ver los factores que puede registrar.              |

Para [Connected Accounts with Token Vault](/docs/es/secure/call-apis-on-users-behalf/token-vault/connected-accounts-for-token-vault), My Account API admite los siguientes scopes:

| Scope                          | Descripción                                                                                 |
| ------------------------------ | ------------------------------------------------------------------------------------------- |
| `create:me:connected_accounts` | Permite al usuario conectar una cuenta nueva a su perfil de usuario.                        |
| `read:me:connected_accounts`   | Permite al usuario ver las cuentas conectadas existentes vinculadas a su perfil de usuario. |
| `delete:me:connected_accounts` | Permite al usuario eliminar una cuenta conectada de su perfil de usuario.                   |

<div id="cross-origin-requests">
  ## Solicitudes de origen cruzado
</div>

Si tiene pensado llamar a My Account API directamente desde una aplicación basada en navegador (como una aplicación web de página única) que se ejecuta en un dominio distinto al de su tenant de Auth0, se encontrará con políticas de seguridad del navegador conocidas como Cross-Origin Resource Sharing (CORS). De forma predeterminada, los navegadores bloquean estas solicitudes de origen cruzado.

Para permitir que su aplicación haga solicitudes a la API correctamente, debe agregar el dominio de su aplicación (su &quot;origen&quot;) a la configuración de su cliente:

1. Vaya a [Dashboard &gt; Applications](https://manage.auth0.com/#/applications). Seleccione la aplicación para verla.
2. En Cross-Origin Authentication, active Allow Cross-Origin Authentication.
3. Busque **Allowed Origins (CORS)** e introduzca la URL de origen de su aplicación.
4. Seleccione **Save**.

Para obtener más información, lea [Configurar Cross-Origin Resource Sharing](/docs/es/get-started/applications/set-up-cors).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Si no necesita usar CORS para su aplicación, asegúrese de que Allow Cross-Origin Authentication esté desactivado. Agregar la URL de su aplicación a esta lista le indica a Auth0 que confíe en las solicitudes procedentes de ese origen, lo que permite que su aplicación del lado del cliente acceda a la API.
</Callout>
