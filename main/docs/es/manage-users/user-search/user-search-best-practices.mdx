title: Prácticas recomendadas de User Search
---

Estas son algunas prácticas recomendadas para la búsqueda de usuarios:

* Necesita un token para hacer solicitudes a la <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip>. Para obtener más información, consulte [Management API Access Tokens](/docs/es/secure/tokens/access-tokens/management-api-access-tokens).
* Para realizar solicitudes de búsqueda de usuarios, se requiere el scope `read:users`.
* Aproveche la búsqueda no determinista: si su aplicación no requiere un orden coherente de los resultados de búsqueda y sus queries son lo suficientemente cortas como para no depender de la paginación, establezca `primary_order=false` para mejorar el rendimiento de la query.
* Para obtener los resultados de búsqueda más recientes, use un endpoint **de consistencia inmediata** durante los procesos de autenticación, como [Get Users by ID](/docs/es/manage-users/user-search/retrieve-users-with-get-users-by-id-endpoint) y [Get Users by Email](/docs/es/manage-users/user-search/retrieve-users-with-get-users-by-email-endpoint). Las búsquedas que usan estos endpoints reflejan los resultados de todas las operaciones de escritura exitosas, incluidas las que ocurrieron poco antes de su solicitud.
* Use un schema conocido para metadata:

  * Use tipos de datos coherentes para las propiedades.
  * Evite los nombres de propiedades dinámicos.
  * Evite schemas grandes y estructuras profundas.
  * Evite almacenar datos que no necesite para fines de autenticación y autorización.
* Las queries de búsqueda agotan el tiempo de espera (código de estado HTTP 503) si no se completan en dos segundos o menos. Las queries que tardan más indican que se trata de una query costosa o que contiene un error que impide que se complete rápidamente.
* Las queries de búsqueda con atributos definidos por el usuario en `app_metadata` y `user_metadata` pueden agotar el tiempo de espera. Le recomendamos no usar [List or search users](https://auth0.com/docs/api/management/v2/users/get-users) para operaciones críticas.
* No use un criterio de búsqueda que devuelva un conjunto de datos grande (más de 1000 resultados).
* No use queries de existencia (por ejemplo, &quot;deme todos los usuarios con una propiedad independientemente de su valor&quot;).
* No haga sondeos a las API de búsqueda.
* No realice solicitudes de búsqueda de usuarios dentro de puntos de extensión del flujo de inicio de sesión, como Rules o Actions de `post-login`.
* No use campos de metadata grandes (intente mantener los campos de metadata en 2 KB o menos).
* El uso de wildcard en las búsquedas puede afectar el rendimiento. En algunos casos, las búsquedas con wildcard en conjuntos de datos grandes pueden provocar errores de tiempo de espera. También recomendamos evitar wildcards al principio del término de búsqueda, mientras que usarlos como sufijos ofrece un mejor rendimiento.
* Escape el carácter de espacio para mejorar el rendimiento (por ejemplo, `q=name:John Doe` debe escribirse como `q=name:John\ Doe`).
