description: Aprenda a ofrecer una interfaz del lado del cliente que permita a los usuarios autenticarse en sus otras cuentas y vincularlas a su cuenta principal mediante una SPA.
title: 'Vinculación de cuentas iniciada por el usuario: implementación del lado del cliente'
---

Auth0 admite la vinculación de cuentas de usuario de varios <Tooltip tip="Proveedor de identidad (IdP): servicio que almacena y gestiona identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=identity+providers">proveedores de identidad</Tooltip>. Una forma de implementar esta funcionalidad es permitir que el usuario vincule cuentas de forma explícita. En este escenario, el usuario se autentica a través de la interfaz de usuario de su aplicación de una sola página (SPA) y, más tarde, puede usar un enlace o botón para vincular otra cuenta con la primera. Cuando el usuario hace clic en este enlace o botón, su aplicación realiza una solicitud para que, cuando el usuario inicie sesión con el segundo proveedor, la segunda cuenta quede vinculada con la primera.

Cuando inicia la vinculación de cuentas, puede seleccionar qué identidad usar como cuenta principal y cuál como secundaria. Esta elección depende del conjunto de atributos que desee conservar en el perfil principal, ya que solo se conservarán los atributos de la cuenta principal.

Cree una aplicación de ejemplo de React de página única con este proyecto [en GitHub](https://github.com/auth0-samples/auth0-link-accounts-sample/tree/master/SPA).

En este proyecto, abra el archivo `auth_config.json` en su editor de texto y actualice el dominio y `client_id` con sus datos.

Agregue `http://localhost:3000` en los campos “Allowed Callback URLs”, “Allowed Logout URLs” y “Allowed Web Origins”.

1. Haga que el usuario inicie sesión en su aplicación.<br /><br />

   El usuario se autentica en su SPA mediante [Universal Login](/docs/es/authenticate/login/auth0-universal-login), solicitando un [token de acceso para la Management API](https://auth0.com/docs/api/management/v2/get-access-tokens-for-spas).<br /><br />

   En un login típico de una SPA, la devolución de llamada se gestiona del lado del cliente en la misma página, y se recibe un JWT tras una autenticación correcta. Para obtener más información, consulte el [Quickstart para aplicaciones web de página única](/docs/es/quickstart/spa).
2. El usuario inicia la vinculación de cuentas. Su SPA debe proporcionar una interfaz de usuario para que el usuario pueda iniciar la vinculación con sus otras cuentas. Por ejemplo, su SPA podría incluir una página de configuración del usuario:<br /><br />

   Cuando el usuario hace clic en el botón **Vincular cuenta**, su aplicación lo redirige a la página de Universal Login, donde inicia sesión con la conexión que desea vincular. Después de una autenticación correcta, use el token obtenido para vincular las cuentas.<br /><br />

   También puede agregar un botón para cada conexión (p. ej., &#39;Vincular cuenta de Facebook&#39;, &#39;Vincular cuenta de Google&#39;) y redirigir al usuario a `/authorize` con el parámetro `connection` establecido (p. ej., `/authorize?connection=facebook`).

   <Frame>![Ejemplo de página de perfil con vinculación de cuentas de usuario](/docs/images/cdy7uua7fh8z/4qxdXGBnlVRDaTkw2wv9WP/8792cff7c5fc21910ec12495828539da/account-linking-spa.png)</Frame>
3. Vincule las cuentas llamando al [endpoint Link a User Account](https://auth0.com/docs/api/v2#!/Users/post_identities) de Auth0 Management API.

   <Warning>
     Para conservar y fusionar la `user_metadata` de la cuenta secundaria, debe recuperarla y fusionarla con los metadatos de la cuenta principal antes de llamar al endpoint de la API. Después de vincular las cuentas, los metadatos de la cuenta secundaria se descartan.

     Cuando inicia la vinculación de cuentas, puede seleccionar qué identidad se usará como cuenta principal y cuál como cuenta secundaria. Esta elección dependerá del conjunto de atributos que desee conservar en el perfil principal.
   </Warning>
4. En la función `linkAccount`, llame a la Management API. Autentíquese con la API usando el JWT principal, que es el token de acceso, y realice la vinculación usando el ID del usuario principal y el JWT secundario, que es el token de ID del usuario secundario.

   ```javascript lines expandable
   const linkAccount = async () => {
     const accessToken = await auth0.getTokenSilently();
     const { sub } = await auth0.getUser();
     // authenticateUser debe autenticar al usuario con la cuenta que se va a vincular
     // Consulte el ejemplo de Github para obtener más detalles
     const {
       __raw: targetUserIdToken,
       email_verified,
       email,
     } = await authenticateUser();
     if (!email_verified) {
       throw new Error(
         `Account linking is only allowed to a verified account. Please verify your email ${email}.`
       );
     }
     await fetch(`https://${config.domain}/api/v2/users/${sub}/identities`, {
       method: "POST",
       headers: {
         "Content-Type": "application/json",
         Authorization: `Bearer ${accessToken}`,
       },
       body: JSON.stringify({
         link_with: targetUserIdToken,
       }),
     });
   };
   ```

<Card title="Vinculación de cuentas: tokens de acceso vs. tokens de ID">
  Anteriormente, en algunos casos, podía usar tokens de ID para vincular y desvincular cuentas de usuario. Esta funcionalidad está quedando obsoleta. Tendrá que usar tokens de acceso en todos los casos. El cambio en la desvinculación de cuentas es que ya no puede usar un token de ID en el encabezado Authorization. En su lugar, debe usar un token de acceso. La funcionalidad sigue estando disponible, pero no se recomienda, y se anima a los usuarios afectados a migrar. Consulte la [Guía de migración: Vinculación de cuentas con tokens de acceso vs. tokens de ID](/docs/es/troubleshoot/product-lifecycle/past-migrations/link-user-accounts-with-access-tokens-migration) para obtener más detalles.
</Card>
