title: 'Vinculación de cuentas iniciada por el usuario: implementación del lado del cliente'
---

Auth0 admite la vinculación de cuentas de usuario de varios <Tooltip tip="Proveedor de identidad (IdP): Servicio que almacena y administra identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=identity+providers">proveedores de identidad</Tooltip>. Una forma de implementar esta funcionalidad es permitir que el usuario vincule cuentas explícitamente. En este escenario, el usuario se autentica mediante la interfaz de tu aplicación web de página única (SPA) y, más adelante, puede usar un enlace o botón para vincular otra cuenta con la primera. Cuando el usuario hace clic en este enlace o botón, tu aplicación realiza una llamada para que, cuando el usuario inicie sesión con el segundo proveedor, la segunda cuenta se vincule con la primera.

Cuando inicias la vinculación de cuentas, puedes seleccionar qué identidad usar como cuenta principal y cuál como secundaria. Esta elección depende del conjunto de atributos que quieras conservar en el perfil principal, ya que solo se conservarán los atributos de la cuenta principal.

Crea una aplicación web de página única de ejemplo en React con este proyecto [en GitHub](https://github.com/auth0-samples/auth0-link-accounts-sample/tree/master/SPA).

En este proyecto, abre el archivo `auth_config.json` en tu editor de texto y actualiza el dominio y `client_id` con tus datos.

Agrega `http://localhost:3000` en los campos “Allowed Callback URLs”, “Allowed Logout URLs” y “Allowed Web Origins”.

1. Haz que el usuario inicie sesión en tu aplicación.<br /><br />

   El usuario se autentica en tu SPA mediante [Universal Login](/docs/es/authenticate/login/auth0-universal-login) y solicita un [Access Token for the Management API](https://auth0.com/docs/api/management/v2/get-access-tokens-for-spas).<br /><br />

   En el flujo de login típico de una SPA, la devolución de llamada se maneja del lado del cliente en la misma página, y se recibe un JWT tras una autenticación correcta. Para obtener más información, consulta el [Quickstart de aplicación web de página única](/docs/es/quickstart/spa).
2. El usuario inicia la vinculación de cuentas. Tu SPA debe proporcionar una interfaz de usuario para que el usuario inicie la vinculación con sus otras cuentas. Por ejemplo, tu SPA podría incluir la página de configuración del usuario:<br /><br />

   Cuando el usuario hace clic en el botón **Vincular cuenta**, tu aplicación lo redirige a la página de Universal Login, donde inicia sesión con la conexión que quiere vincular. Después de una autenticación correcta, usa el token obtenido para vincular las cuentas.<br /><br />

   También podrías agregar un botón para cada conexión (p. ej., &#39;Vincular cuenta de Facebook&#39;, &#39;Vincular cuenta de Google&#39;) y redirigir al usuario a `/authorize` con el parámetro `connection` establecido (p. ej., `/authorize?connection=facebook`).

   <Frame>![Ejemplo de página de perfil con vinculación de cuentas de usuario](/docs/images/cdy7uua7fh8z/4qxdXGBnlVRDaTkw2wv9WP/8792cff7c5fc21910ec12495828539da/account-linking-spa.png)</Frame>
3. Vincula las cuentas llamando al [endpoint Link a User Account](https://auth0.com/docs/api/v2#!/Users/post_identities) de Auth0 Management API.

   <Warning>
     Para conservar y fusionar el `user_metadata` de la cuenta secundaria, debes recuperarlo y fusionarlo con los metadatos de la cuenta principal antes de llamar al endpoint de API. Después de vincular las cuentas, los metadatos de la cuenta secundaria se descartan.

     Cuando inicias la vinculación de cuentas, puedes seleccionar qué identidad se usará como cuenta principal y cuál como secundaria. Esta elección dependerá del conjunto de atributos que quieras conservar en el perfil principal.
   </Warning>
4. En la función `linkAccount`, llama a la Management API. Autentícate en la API con el JWT principal, que es el Access Token, y realiza la vinculación usando el ID del usuario principal y el JWT secundario, que es el token de ID del usuario secundario.

   ```javascript lines expandable
   const linkAccount = async () => {
     const accessToken = await auth0.getTokenSilently();
     const { sub } = await auth0.getUser();
     // authenticateUser debe autenticar al usuario con la cuenta que se va a vincular
     // Consulta el ejemplo de Github para obtener más detalles
     const {
       __raw: targetUserIdToken,
       email_verified,
       email,
     } = await authenticateUser();
     if (!email_verified) {
       throw new Error(
         `La vinculación de cuentas solo se permite con una cuenta verificada. Verifica tu correo electrónico ${email}.`
       );
     }
     await fetch(`https://${config.domain}/api/v2/users/${sub}/identities`, {
       method: "POST",
       headers: {
         "Content-Type": "application/json",
         Authorization: `Bearer ${accessToken}`,
       },
       body: JSON.stringify({
         link_with: targetUserIdToken,
       }),
     });
   };
   ```

<Card title="Vinculación de cuentas: Access Tokens vs. tokens de ID">
  Anteriormente, en algunos casos, podías usar tokens de ID para vincular y desvincular cuentas de usuario. Esta funcionalidad está quedando obsoleta. Tendrás que usar Access Tokens en todos los casos. El cambio en la desvinculación de cuentas es que ya no puedes usar un token de ID en el encabezado Authorization. En su lugar, debes usar un Access Token. La funcionalidad sigue estando disponible, pero no se recomienda y se anima a los usuarios afectados a migrar. Consulta la [Guía de migración: Vinculación de cuentas con Access Tokens vs. tokens de ID](/docs/es/troubleshoot/product-lifecycle/past-migrations/link-user-accounts-with-access-tokens-migration) para obtener más información.
</Card>
