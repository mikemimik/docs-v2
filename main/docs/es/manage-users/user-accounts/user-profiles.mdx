Para acceder a cualquiera de las aplicaciones de tu tenant, cada usuario debe tener un perfil en este. Los perfiles de usuario contienen información sobre tus usuarios, como su nombre y sus datos de contacto. Puedes gestionar los perfiles de usuario desde el [Auth0 Dashboard](/docs/es/get-started/auth0-overview/dashboard) y la [Auth0 Management API](https://auth0.com/docs/api/management/v2).

<div id="user-data-sources">
  ## Fuentes de datos de usuario
</div>

Los datos de usuario pueden provenir de muchas fuentes, incluidas tus propias bases de datos, así como <Tooltip tip="Proveedor de identidad (IdP): Servicio que almacena y gestiona identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=identity+providers">proveedores de identidad</Tooltip> sociales, legales y empresariales. Algunos ejemplos son Google, Facebook, Active Directory o <Tooltip tip="Proveedor de identidad (IdP): Servicio que almacena y gestiona identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=SAML">SAML</Tooltip>. Puedes normalizar los datos de usuario procedentes de cualquier fuente de datos compatible.

Los atributos del perfil de usuario pueden incluir información del proveedor de identidad. Por ejemplo:

| Datos de usuario de... | Podrían incluir...                        |
| ---------------------- | ----------------------------------------- |
| LinkedIn               | Empleador actual o títulos obtenidos      |
| Facebook               | Foto de perfil, cumpleaños o estado civil |
| Active Directory       | Número de empleado, cargo o departamento  |

Auth0 se refiere a todas las fuentes de datos de usuario como **conexiones** porque se conecta a ellas para autenticar al usuario.

<div id="user-data-normalization">
  ### Normalización de datos de usuario
</div>

Auth0 admite una variedad de [proveedores de identidad](/docs/es/authenticate/identity-providers) y [Database Connections](/docs/es/authenticate/database-connections). Cada conexión puede devolver un conjunto distinto de atributos de usuario. A veces, distintas conexiones usan nombres diferentes para el mismo atributo. Por ejemplo, el campo `surname` en una conexión podría aparecer como `last_name` o `family_name` en otras fuentes de datos de usuario.

Para gestionar esta complejidad, Auth0 proporciona un [perfil de usuario normalizado](/docs/es/manage-users/user-accounts/user-profiles/normalized-user-profiles), un estándar propio de Auth0 para almacenar datos de usuario.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Para ver la [estructura de datos del perfil de usuario](/docs/es/manage-users/user-accounts/user-profiles/user-profile-structure) de Auth0, ve a [Dashboard &gt; User Management &gt; Users](https://manage.auth0.com/#/users) y, después, haz clic en un usuario.
</Callout>

<div id="user-profile-attribute-mapping">
  ### Mapeo de atributos del perfil de usuario
</div>

<div id="adldap-connector">
  #### Conector AD/LDAP
</div>

Para Active Directory o cualquier otra conexión LDAP que utilice el Conector AD/LDAP de Auth0, existe un mecanismo para mapear los atributos del perfil de usuario del servicio de directorio al perfil de usuario normalizado de Auth0. El archivo `profileMapper.js`, ubicado en el directorio de instalación del Conector AD/LDAP, realiza el mapeo de los atributos cuando un usuario se autentica.

<div id="saml-assertions">
  #### Aserciones SAML
</div>

Si tu aplicación usa el protocolo SAML para comunicarse con Auth0, uno de estos dos mecanismos hace coincidir los atributos del usuario con el perfil de usuario normalizado de Auth0:

| Si Auth0 es un...           | Entonces...                                                                                                                                                                                                                                                                                                                                                                                          |
| --------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Proveedor de servicio SAML  | Usa la pestaña Mappings de la conexión SAML para mapear los atributos que provienen de un IdP con los atributos del perfil de usuario de Auth0: Ve a [Dashboard &gt; Authentication &gt; empresarial &gt; SAMLP](https://manage.auth0.com/#/connections/database/enterprise/samlp). Haz clic en el nombre de la conexión SAML y luego en **Mappings**.                                               |
| Proveedor de identidad SAML | Usa la pestaña Settings de los Addons de la aplicación para mapear los atributos del perfil de usuario de Auth0 con los atributos de la aserción SAML que se envía de vuelta al proveedor de servicio: Ve a [Dashboard &gt; Aplicaciones](https://manage.auth0.com/#/connections/database/applications). Haz clic en el nombre de tu aplicación, luego en **Addons** y después en **SAML2 Web App**. |

<div id="account-linking">
  ### Vinculación de cuentas
</div>

Los usuarios pueden iniciar sesión en una aplicación inicialmente con una conexión (como una base de datos personalizada) y luego, más adelante, iniciar sesión con una conexión diferente (como Facebook). En este caso, el `user_id` de la segunda autenticación es diferente del `user_id` de la primera autenticación.

Auth0 proporciona un mecanismo para vincular las dos cuentas. Cuando Auth0 vincula las dos cuentas, almacena dos elementos en la parte del arreglo `identities` del perfil de usuario, uno por cada conexión.

Auth0 no fusiona los atributos del perfil de usuario de varios proveedores. Auth0 obtiene los atributos **principales** del perfil de usuario del primer proveedor utilizado.

Para obtener más información, consulta [Vinculación de cuentas de usuario](/docs/es/manage-users/user-accounts/user-account-linking).

<div id="caching-user-profiles">
  ### Caché de perfiles de usuario
</div>

Auth0 almacena en caché el perfil de usuario recibido de una conexión antes de enviarlo a la aplicación que realiza la solicitud. Esta caché se encuentra en la base de datos de Auth0. Cada vez que el usuario se autentica, Auth0 actualiza la caché. Para obtener más información, consulta [Actualizar el perfil de usuario con tu base de datos](/docs/es/manage-users/user-accounts/user-profiles/update-user-profiles-using-your-database).

<div id="custom-user-profile-data">
  ## Datos personalizados del perfil de usuario
</div>

Auth0 te permite almacenar [metadatos](/docs/es/manage-users/user-accounts/metadata), es decir, datos relacionados con cada usuario que no provienen del proveedor de identidad. Puedes usar `user_metadata` para almacenar atributos personalizados, como el color favorito o el pasatiempo del usuario.

<div id="change-user-profile-data">
  ## Cambiar los datos del perfil del usuario
</div>

Puedes modificar la información del perfil de un usuario de varias formas.

* [Scopes](/docs/es/get-started/apis/scopes): Los flujos de autenticación compatibles con Auth0 incluyen un parámetro opcional que te permite especificar un scope. Esto controla la información del perfil del usuario (claims) incluida en el <Tooltip tip="JSON Web Token (JWT): formato estándar del token de identidad (y, a menudo, también del token de acceso) que se utiliza para representar claims de forma segura entre dos partes." cta="Ver glosario" href="/docs/es/glossary?term=ID+token">token de identidad</Tooltip> (<Tooltip tip="token de identidad: credencial destinada al propio cliente, en lugar de usarse para acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=JWT">JWT</Tooltip>).
* [Auth0 Dashboard](/docs/es/manage-users/user-accounts/manage-users-using-the-dashboard): El Dashboard te permite editar manualmente las secciones `user_metadata` y `app_metadata` del perfil de cualquier usuario.
* [Management API](/docs/es/manage-users/user-accounts/manage-users-using-the-management-api): Puedes leer, actualizar y eliminar perfiles de usuario almacenados en la base de datos de Auth0.
* [Scripts de base de datos personalizados](/docs/es/authenticate/database-connections/custom-db/templates): Si utilizas una base de datos personalizada como conexión, puedes escribir scripts para implementar eventos del ciclo de vida, como crear, iniciar sesión, verificar, eliminar y cambiar la contraseña. Auth0 proporciona plantillas para estos scripts que puedes modificar según la base de datos y el esquema concretos.
* [Rules](/docs/es/manage-users/user-accounts/metadata/manage-metadata-rules): Usa Rules para ampliar el perfil del usuario durante la transacción de autenticación y, de forma opcional, guardar esos cambios en Auth0.

<div id="access-user-profiles">
  ## Acceder a los perfiles de usuario
</div>

Tanto las aplicaciones como la <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip> de Auth0 pueden acceder a la información del perfil de un usuario.

<div id="access-user-profiles-from-apps">
  ### Acceder a perfiles de usuario desde aplicaciones
</div>

Una vez que Auth0 completa la autenticación y devuelve el control a su aplicación, proporciona el perfil de usuario a la aplicación. A bajo nivel, puede hacerlo mediante uno de los [protocolos](/docs/es/authenticate/protocols) de aplicación compatibles con Auth0. Sin embargo, la mayoría de los desarrolladores prefieren usar los SDKs de Auth0. Para obtener más información, consulte [Quickstarts](/docs/es/quickstarts).

Uno de esos SDKs es el widget de Lock de Auth0, que proporciona una interfaz de inicio de sesión para el usuario. Para obtener más información, consulte:

* [Lock for Web](/docs/es/libraries/lock)
* [Lock Swift](/docs/es/libraries/lock-swift)
* [Lock.Android](/docs/es/libraries/lock-android)

Si quiere que su aplicación web tenga una interfaz de inicio de sesión personalizada, puede usar Auth0.js. Esta biblioteca de JavaScript sin interfaz de Auth0 invoca el flujo de autenticación (y otras tareas) y, a cambio, recibe un objeto User Profile. Para obtener más información, consulte la [referencia de Auth0.js](/docs/es/libraries/auth0js).

<div id="access-user-profiles-from-the-management-api">
  ### Accede a los perfiles de usuario desde Management API
</div>

Auth0 proporciona una API REST que permite a las aplicaciones y los servicios acceder al objeto User Profile y manipularlo.

El API Explorer permite a los usuarios explorar de forma interactiva la Management API. Proporciona:

* Llamadas a la API disponibles
* Información necesaria para cada llamada
* Información devuelta por cada llamada

Con el explorador, los usuarios pueden probar cada endpoint en la UI del explorador o mediante un comando de CuRL en la línea de comandos. Para probar uno de los comandos de Management API, ve al [API Explorer](https://auth0.com/docs/api/management/v2). Selecciona el acceso requerido en **Scopes** dentro del comando que elijas, como `update:users`, y luego haz clic en **TRY**.

La Auth0 Authentication API es específica para los flujos de autenticación. Para obtener más información, consulta [Explorador de Authentication API](https://auth0.com/docs/api/authentication). Normalmente, la mayoría de estos endpoints los utilizan los distintos Auth0 SDKs, no tu propio código.

<div id="the-difference-between-user-profiles-and-tokens">
  ## La diferencia entre los perfiles de usuario y los tokens
</div>

En los flujos de autenticación descritos anteriormente, Auth0 devuelve un conjunto de tokens en lugar de un perfil de usuario completo.

Uno de los tokens devueltos es el token de identidad, que es un JSON Web Token (JWT) que contiene atributos del perfil de usuario representados en forma de claims. Estos claims son afirmaciones sobre el usuario. Se puede confiar en un claim si quien consume el token puede verificar su firma, que (en el caso de HS256) se genera con el <Tooltip tip="secreto de cliente: secreto que usa un cliente (aplicación) para autenticarse con el servidor de autorización; solo deberían conocerlo el cliente y el servidor de autorización, y debe ser lo suficientemente aleatorio como para que no pueda adivinarse." cta="Ver glosario" href="/docs/es/glossary?term=client+secret">secreto de cliente</Tooltip> de la aplicación de Auth0. Si tu app usa cifrado RS256, el token de identidad se firmará con una clave privada y se verificará con una clave pública. Tu app puede entonces decodificar el JWT y obtener la información del usuario contenida en su payload. Algunos ejemplos de esta información son el nombre del usuario, el correo electrónico y otros datos que normalmente forman parte de la experiencia del usuario.

Los claims dentro de un JWT por lo general contienen solo un subconjunto de la información disponible en el perfil de usuario. El objetivo es minimizar el tamaño del token. Para obtener más información, lee [JSON Web Tokens](/docs/es/secure/tokens/json-web-tokens).

Hay otros tres tipos de tokens que pueden devolverse durante la autenticación:

* Auth0 <Tooltip tip="token de acceso: credencial de autorización, en forma de una cadena opaca o JWT, que se usa para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=access+token">token de acceso</Tooltip>
* Token de acceso de proveedor externo
* <Tooltip tip="token de actualización: token que se usa para obtener un token de acceso renovado sin obligar a los usuarios a volver a iniciar sesión." cta="Ver glosario" href="/docs/es/glossary?term=Refresh+token">token de actualización</Tooltip>

Para obtener más información sobre tokens y claims, lee [Tokens](/docs/es/secure/tokens).
