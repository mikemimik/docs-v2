Para acceder a cualquiera de las aplicaciones de tu tenant, cada usuario debe tener un perfil en él. Los perfiles de usuario contienen información sobre tus usuarios, como su nombre y sus datos de contacto. Puedes gestionar los perfiles de usuario desde el [Auth0 Dashboard](/docs/es/get-started/auth0-overview/dashboard) y la [Auth0 Management API](https://auth0.com/docs/api/management/v2).

<div id="user-data-sources">
  ## Fuentes de datos de usuario
</div>

Los datos de usuario pueden provenir de muchas fuentes, incluidas sus propias bases de datos, así como <Tooltip tip="Proveedor de identidad (IdP): servicio que almacena y gestiona identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=identity+providers">proveedores de identidad</Tooltip> sociales, jurídicos y empresariales. Algunos ejemplos son Google, Facebook, Active Directory o <Tooltip tip="Proveedor de identidad (IdP): servicio que almacena y gestiona identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=SAML">SAML</Tooltip>. Puede normalizar los datos de usuario procedentes de cualquier fuente de datos compatible.

Los atributos del perfil de usuario pueden incluir información del proveedor de identidad. Estos son algunos ejemplos:

| Datos de usuario de... | Podrían incluir...                                       |
| ---------------------- | -------------------------------------------------------- |
| LinkedIn               | Empresa actual o títulos obtenidos                       |
| Facebook               | Foto de perfil, fecha de nacimiento o estado sentimental |
| Active Directory       | Número de empleado, cargo o departamento                 |

Auth0 llama **conexiones** a todas las fuentes de datos de usuario porque se conecta a ellas para autenticar al usuario.

<div id="user-data-normalization">
  ### Normalización de datos de usuario
</div>

Auth0 admite una variedad de [proveedores de identidad](/docs/es/authenticate/identity-providers) y [Database Connections](/docs/es/authenticate/database-connections). Cada conexión puede devolver un conjunto distinto de atributos de usuario. A veces, distintas conexiones usan nombres diferentes para el mismo atributo. Por ejemplo, `surname` en una conexión podría aparecer como `last_name` o `family_name` en otras fuentes de datos de usuario.

Para gestionar esta complejidad, Auth0 proporciona un [perfil de usuario normalizado](/docs/es/manage-users/user-accounts/user-profiles/normalized-user-profiles), un estándar propio de Auth0 para almacenar datos de usuario.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Para ver la [estructura de datos del perfil de usuario](/docs/es/manage-users/user-accounts/user-profiles/user-profile-structure) de Auth0, ve a [Dashboard &gt; User Management &gt; Users](https://manage.auth0.com/#/users) y haz clic en un usuario.
</Callout>

<div id="user-profile-attribute-mapping">
  ### Mapeo de atributos del perfil de usuario
</div>

<div id="adldap-connector">
  #### Conector AD/LDAP
</div>

En Active Directory o en cualquier otra conexión LDAP que utilice el Conector AD/LDAP de Auth0, existe un mecanismo para mapear los atributos del perfil de usuario del servicio de directorio al perfil de usuario normalizado de Auth0. Un archivo `profileMapper.js`, ubicado en el directorio de instalación del Conector AD/LDAP, mapea los atributos cuando un usuario se autentica.

<div id="saml-assertions">
  #### Aserciones SAML
</div>

Si su aplicación usa el protocolo SAML para comunicarse con Auth0, se utiliza uno de estos dos mecanismos para hacer coincidir los atributos del usuario con el perfil de usuario normalizado de Auth0:

| Si Auth0 es un...           | Entonces...                                                                                                                                                                                                                                                                                                                                                                                           |
| --------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| proveedor de servicios SAML | Use la pestaña **Mappings** de la conexión SAML para mapear los atributos que provienen de un IdP a los atributos del perfil de usuario de Auth0: vaya a [Dashboard &gt; Authentication &gt; Enterprise &gt; SAMLP](https://manage.auth0.com/#/connections/database/enterprise/samlp). Haga clic en el nombre de la conexión SAML y, luego, en **Mappings**.                                          |
| proveedor de identidad SAML | Use la pestaña **Settings** de los **Addons** de la aplicación para mapear atributos del perfil de usuario de Auth0 a los atributos de la aserción SAML que se devuelve al proveedor de servicios: vaya a [Dashboard &gt; Applications](https://manage.auth0.com/#/connections/database/applications). Haga clic en el nombre de su aplicación, luego en **Addons** y, después, en **SAML2 Web App**. |

<div id="account-linking">
  ### Vinculación de cuentas
</div>

Los usuarios pueden iniciar sesión en una aplicación inicialmente con una conexión (como una base de datos personalizada) y, más adelante, iniciar sesión con una conexión distinta (como Facebook). En este caso, el `user_id` de la segunda autenticación es diferente del `user_id` de la primera autenticación.

Auth0 proporciona un mecanismo para vincular ambas cuentas. Cuando Auth0 vincula las dos cuentas, almacena dos elementos en la parte del perfil de usuario correspondiente al arreglo `identities`, uno por cada conexión.

Auth0 no fusiona los atributos del perfil de usuario de varios proveedores. Auth0 toma los atributos **principales** del perfil de usuario del primer proveedor utilizado.

Para obtener más información, consulta [Vinculación de cuentas de usuario](/docs/es/manage-users/user-accounts/user-account-linking).

<div id="caching-user-profiles">
  ### Almacenamiento en caché de perfiles de usuario
</div>

Auth0 almacena en caché el perfil de usuario recibido de una conexión antes de enviarlo a la aplicación que hace la llamada. Esta caché se encuentra en la base de datos de Auth0. Cada vez que el usuario se autentica, Auth0 actualiza la caché. Para obtener más información, consulta [Actualizar el perfil de usuario con tu base de datos](/docs/es/manage-users/user-accounts/user-profiles/update-user-profiles-using-your-database).

<div id="custom-user-profile-data">
  ## Datos personalizados del perfil de usuario
</div>

Auth0 te permite almacenar [metadatos](/docs/es/manage-users/user-accounts/metadata), es decir, datos relacionados con cada usuario que no provienen del proveedor de identidad. Puedes usar `user_metadata` para almacenar atributos personalizados, como el color favorito del usuario o sus pasatiempos.

<div id="change-user-profile-data">
  ## Cambiar los datos del perfil del usuario
</div>

Puede modificar la información del perfil de un usuario de varias maneras.

* [Scopes](/docs/es/get-started/apis/scopes): Los flows de autenticación compatibles con Auth0 incluyen un parámetro opcional que le permite especificar un scope. Esto controla la información del perfil del usuario (claims) incluida en el <Tooltip tip="JSON Web Token (JWT): formato estándar del token de identidad (y, a menudo, también del Access Token) utilizado para representar claims de forma segura entre dos partes." cta="Ver glosario" href="/docs/es/glossary?term=ID+token">token de identidad</Tooltip> (<Tooltip tip="Token de identidad: credencial destinada al propio cliente, en lugar de usarse para acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=JWT">JWT</Tooltip>).
* [Auth0 Dashboard](/docs/es/manage-users/user-accounts/manage-users-using-the-dashboard): El Dashboard le permite editar manualmente las secciones `user_metadata` y `app_metadata` del perfil de cualquier usuario.
* [Management API](/docs/es/manage-users/user-accounts/manage-users-using-the-management-api): Puede leer, actualizar y eliminar perfiles de usuario almacenados en la base de datos de Auth0.
* [Scripts de base de datos personalizada](/docs/es/authenticate/database-connections/custom-db/templates): Si utiliza una base de datos personalizada como conexión, puede escribir scripts para implementar eventos del ciclo de vida como create, login, verify, delete y cambiar la contraseña. Auth0 proporciona plantillas para estos scripts que puede modificar según la base de datos y el esquema concretos.
* [Rules](/docs/es/manage-users/user-accounts/metadata/manage-metadata-rules): Use Rules para ampliar el perfil del usuario durante la transacción de autenticación y, opcionalmente, persistir esos cambios en Auth0.

<div id="access-user-profiles">
  ## Acceso a los perfiles de usuario
</div>

Tanto las aplicaciones como la <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip> de Auth0 pueden acceder a la información del perfil de un usuario.

<div id="access-user-profiles-from-apps">
  ### Acceder a perfiles de usuario desde aplicaciones
</div>

Una vez que Auth0 completa la autenticación y devuelve el control a tu aplicación, proporciona el perfil del usuario a la aplicación. A bajo nivel, puedes lograrlo usando uno de los [protocolos](/docs/es/authenticate/protocols) de aplicación compatibles con Auth0. Sin embargo, la mayoría de los desarrolladores prefieren usar los SDKs de Auth0. Para obtener más información, consulta [Quickstarts](/docs/es/quickstarts).

Uno de esos SDKs es el widget de Auth0 Lock, que proporciona una interfaz de inicio de sesión para usuarios. Para obtener más información, consulta:

* [Lock for Web](/docs/es/libraries/lock)
* [Lock Swift](/docs/es/libraries/lock-swift)
* [Lock.Android](/docs/es/libraries/lock-android)

Si quieres que tu aplicación web tenga una interfaz de inicio de sesión personalizada, puedes usar Auth0.js. Esta biblioteca de JavaScript sin interfaz de Auth0 invoca el flujo de autenticación (y otras tareas) y, a cambio, recibe un objeto User Profile. Para obtener más información, consulta [Auth0.js Reference](/docs/es/libraries/auth0js).

<div id="access-user-profiles-from-the-management-api">
  ### Accede a los perfiles de usuario desde la Management API
</div>

Auth0 proporciona una API REST que permite a las applications y a los servicios acceder al objeto User Profile y manipularlo.

API Explorer permite a los usuarios explorar interactivamente la Management API. Proporciona:

* Llamadas a la API disponibles
* Información necesaria para cada llamada
* Información devuelta por cada llamada

Con el explorador, los usuarios pueden probar cada endpoint en la UI del explorador o mediante un comando de CuRL en la línea de comandos. Para probar uno de los comandos de la Management API, ve a [API Explorer](https://auth0.com/docs/api/management/v2). Selecciona el acceso necesario en **Scopes** dentro del comando que elijas, como `update:users`, y luego haz clic en **TRY**.

La Auth0 Authentication API es específica para los flujos de autenticación. Para obtener más información, consulta [Authentication API Explorer](https://auth0.com/docs/api/authentication). Normalmente, la mayoría de estos endpoints los utilizan los distintos Auth0 SDKs, no tu propio código.

<div id="the-difference-between-user-profiles-and-tokens">
  ## La diferencia entre perfiles de usuario y tokens
</div>

En los flujos de autenticación descritos anteriormente, Auth0 devuelve un conjunto de tokens en lugar de un perfil de usuario completo.

Uno de los tokens devueltos es el token de identidad, que es un JSON Web Token (JWT) que contiene atributos del perfil de usuario representados en forma de claims. Estos claims son afirmaciones sobre el usuario. Se puede confiar en un claim si quien consume el token puede verificar su firma, que (en el caso de HS256) se genera con el <Tooltip tip="secreto de cliente: Secret usado por un cliente (aplicación) para autenticarse con el servidor de autorización; solo el cliente y el servidor de autorización deben conocerlo, y debe ser lo suficientemente aleatorio como para que no pueda adivinarse." cta="Ver glosario" href="/docs/es/glossary?term=client+secret">secreto de cliente</Tooltip> de la aplicación de Auth0. Si tu app usa cifrado RS256, el token de identidad se firmará con una clave privada y se verificará con una clave pública. Tu app puede entonces decodificar el JWT y obtener la información del usuario contenida en su payload. Algunos ejemplos de esta información son el nombre del usuario, el correo electrónico y otros datos que normalmente forman parte de la experiencia del usuario.

Los claims dentro de un JWT por lo general contienen solo un subconjunto de la información disponible en el perfil de usuario. El objetivo es reducir al mínimo el tamaño del token. Para obtener más información, lee [JSON Web Tokens](/docs/es/secure/tokens/json-web-tokens).

Hay otros tres tipos de tokens que pueden devolverse durante la autenticación:

* Auth0 <Tooltip tip="access token: Credencial de autorización, en forma de una cadena opaca o JWT, usada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=access+token">access token</Tooltip>
* token de acceso de proveedor externo
* <Tooltip tip="token de actualización: Token usado para obtener un Access Token renovado sin obligar a los usuarios a volver a iniciar sesión." cta="Ver glosario" href="/docs/es/glossary?term=Refresh+token">Token de actualización</Tooltip>

Para obtener más información sobre los tokens y los claims, lee [Tokens](/docs/es/secure/tokens).
