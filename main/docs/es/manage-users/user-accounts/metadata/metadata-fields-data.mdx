title: Nombres de campos de metadatos y tipos de datos
---

Auth0 distingue entre tres tipos de metadatos que se utilizan para almacenar tipos específicos de información.

<Warning>
  Los metadatos de Auth0 no son un almacén de datos seguro y no deben usarse para almacenar información sensible, como secretos de alto riesgo e información de identificación personal (PII), como números de seguridad social o de tarjetas de crédito. Se recomienda encarecidamente a los clientes de Auth0 que evalúen los datos almacenados en los metadatos y almacenen únicamente los que sean necesarios para fines de gestión de identidades y accesos.
</Warning>

| Tipo de metadatos                | Nombre del campo                                                                                                                   | Descripción                                                                                                                                                                                                                                                                                  |
| -------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Información del usuario**      | `user_metadata`                                                                                                                    | Almacena atributos del usuario, como preferencias, que no afectan la funcionalidad principal del usuario. Estos datos **pueden** ser editados por usuarios que hayan iniciado sesión si se crea un formulario mediante la Management API y **no** deben usarse como almacén de datos seguro. |
| **Información de acceso**        | `app_metadata`                                                                                                                     | Almacena información como permisos, el plan de Auth0 e IDs externos que pueden afectar el acceso del usuario a las funcionalidades. Estos datos **no pueden** ser editados por los usuarios y existen restricciones sobre lo que se puede almacenar en este campo.                           |
| **Información de la aplicación** | `client_metadata` en el objeto `Client`, `context.clientMetadata` en Rules y `event.client.metadata` en las Actions de Post Login. | Almacena información sobre una aplicación (o *cliente* en la terminología de OIDC y OAuth2). Por ejemplo, la URL de la página principal de la aplicación (cualquier valor que Auth0 no establezca en la configuración de la aplicación).                                                     |

<div id="metadata-field-names">
  ## Nombres de campo de metadato
</div>

<div id="accepted-characters">
  ### Caracteres permitidos
</div>

Los nombres de campo no deben contener los caracteres `.` (punto) ni `$` (signo de dólar).

Por ejemplo, esto no está permitido:

```json lines
{
  "preference.color": "pink"
}
```

Pero puedes ampliarlo así:

```json lines
{
    "preference": { 
        "color": "pink" 
    }
}
```

<div id="dynamic-field-names">
  ### Nombres de campo dinámicos
</div>

Los nombres de campo deben ser estáticos. Los nombres de campo dinámicos reducen la eficiencia de la indexación y perjudican el rendimiento de las consultas de búsqueda. Un esquema estático es más fácil de buscar, manipular y usar.

En lugar de hacer esto:

```json lines
{
    "participants": [
        "Alice": {
            "role": "sender"
         },
        "Bob": {
            "role": "receiver"
        }
    ]
}
```

Haz lo siguiente:

```json lines
{
    "participants": [
        {
            "name": "Alice",
            "role": "sender"
        },
        {
            "name" : "Bob",
            "role": "receiver"
        }
    ]
}
```

<div id="name-collision">
  ### Conflicto de nombres
</div>

Evita usar el mismo nombre para los campos de `app_metadata` y los campos de la raíz del perfil. El campo `app_metadata` se fusiona con la raíz del perfil tanto en Rules como en Actions, lo que puede sobrescribir campos de la raíz del perfil.

Por ejemplo, si un usuario tiene un campo `groups` en la raíz de su perfil (devuelto por un <Tooltip tip="Security Assertion Markup Language (SAML): Protocolo estandarizado que permite a dos partes intercambiar información de autenticación sin necesidad de contraseña." cta="Ver glosario" href="/docs/es/glossary?term=SAML">SAML</Tooltip> <Tooltip tip="Identity Provider (IdP): Servicio que almacena y administra identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=identity+provider">proveedor de identidad</Tooltip>) y un campo `groups` dentro de `app_metadata`, su perfil podría verse así:

```json lines
{
    "user_id": "samlp|example-samlp-connection|username@domain.com",
    "groups": [
        "external-group-1",
        "external-group-2"
    ],
    "app_metadata": {
        "groups": [
            "internal-group-1",
            "internal-group-2"
        ]
    }
}
```

Cuando leas el campo `groups` del [objeto de usuario](/docs/es/customize/rules/user-object-in-rules) en una [Rule](/docs/es/manage-users/user-accounts/metadata/manage-metadata-rules), obtendrás: `["internal-group-1", "internal-group-2"]`.

<div id="metadata-data-types">
  ## Tipos de datos de metadato
</div>

Los campos de metadato admiten todos los [tipos de datos compatibles con JSON](https://datatracker.ietf.org/doc/html/rfc7159):

* String
* Number
* Arreglo
* Object

Asegúrate de mantener la coherencia de los tipos de datos entre usuarios. Por ejemplo, si almacenas un valor como una cadena para un usuario (`user.user_metadata.age = "23"`) y como un número para otro usuario (`user.user_metadata.age = 23`), es posible que tengas problemas al recuperar los datos.

<div id="limitations-and-restrictions">
  ## Limitaciones y restricciones
</div>

<div id="rate-limits">
  ### Límites de frecuencia
</div>

Cuando actualizas metadatos durante el inicio de sesión con [Rules](/docs/es/manage-users/user-accounts/metadata/manage-metadata-rules) o [Actions](/docs/es/manage-users/user-accounts/metadata/manage-user-metadata), se aplican los límites de frecuencia de tu tenant. Para obtener más información, consulta [Límites de frecuencia de los endpoints de la Management API](/docs/es/troubleshoot/customer-support/operational-policies/rate-limit-policy/management-api-endpoint-rate-limits).

<Warning>
  Solo debes almacenar en los metadatos datos relacionados con la autenticación de usuario. Las capacidades de almacenamiento y búsqueda de Auth0 están diseñadas para casos de uso que no requieren una alta frecuencia de búsqueda y/o actualización.

  Si necesitas mantener datos detallados relacionados con el perfil de los usuarios, debes hacerlo en un sistema externo. Puedes almacenar el identificador del usuario de ese sistema como un campo de metadatos en Auth0.
</Warning>

<div id="size-limits-and-storage">
  ### Límites de tamaño y almacenamiento
</div>

* Hay un límite de 1 MB por usuario para los datos de usuario que se pueden indexar, consultar y devolver mediante el [endpoint de búsqueda de usuarios](https://auth0.com/docs/api/management/v2/users/get-users). Si un perfil de usuario supera 1 MB, los valores de atributo de más de 256 caracteres dentro de `app_metadata` y `user_metadata` no se podrán buscar ni se devolverán en los resultados de búsqueda. Si el perfil de usuario sigue superando 1 MB después de omitir esos valores grandes, ninguno de los atributos de `app_metadata` y `user_metadata` se podrá buscar ni devolver para ese usuario. Auth0 captura y registra los casos en los que un perfil de usuario sigue superando 1 MB después de esas omisiones con el `wum` [código de evento](/docs/es/deploy-monitor/logs/log-event-type-codes). Se debe usar el [endpoint para obtener un usuario](https://auth0.com/docs/api/management/v2/users/get-users-by-id) para recuperar todos los atributos de metadatos de los perfiles de usuario que superan ese tamaño.
* Cuando estableces el campo `user_metadata` mediante el [Signup endpoint](https://auth0.com/docs/api/authentication?javascript#signup) de Auth0 Authentication API, puedes incluir un máximo de 10 campos String cuyos valores no superen los 500 caracteres cada uno. Para ver un ejemplo de cómo trabajar con metadatos durante un proceso de registro personalizado, consulta [Custom Signup](/docs/es/libraries/custom-signup).
* El campo `client_metadata` puede tener un máximo de 10 claves. Sus claves y valores pueden tener una longitud máxima de 255 caracteres cada uno y no pueden contener caracteres especiales UTF-8.

<div id="restrictions">
  ### Restricciones
</div>

El campo `app_metadata` no debe contener ninguna de estas propiedades:

* `__tenant`
* `_id`
* `blocked`
* `clientID`
* `created_at`
* `email_verified`
* `email`
* `globalClientID`
* `global_client_id`
* `identities`
* `lastIP`
* `lastLogin`
* `loginsCount`
* `metadata`
* `multifactor_last_modified`
* `multifactor`
* `updated_at`
* `user_id`
