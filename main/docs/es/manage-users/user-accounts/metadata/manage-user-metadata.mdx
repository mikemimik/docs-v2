---
description: Aprende a usar el trigger de Action Post Login para modificar user_metadata y app_metadata como parte del flujo de login de un usuario.
title: Gestiona los metadatos del usuario con el trigger de Action Post Login
---

Auth0 proporciona un sistema sólido para almacenar [metadatos](/docs/es/manage-users/user-accounts/metadata) en el perfil de usuario de Auth0. Puedes configurar un trigger `post-login` para modificar `user_metadata` y `app_metadata` como parte del flujo de login de un usuario. Los triggers Post Login son útiles para tareas como almacenar datos específicos de la aplicación en el perfil del usuario, registrar logs de operaciones del usuario, mapear atributos de <Tooltip tip="Security Assertion Markup Language (SAML): Protocolo estandarizado que permite a dos partes intercambiar información de autenticación sin una contraseña." cta="Ver glosario" href="/docs/es/glossary?term=SAML">SAML</Tooltip> al campo de metadatos, o almacenar en caché valores de operaciones costosas en el perfil del usuario para reutilizarlos en futuros inicios de sesión.

El objeto `api` de `post-login` proporciona operaciones comunes que pueden realizarse en este trigger. Para gestionar los metadatos del usuario, usaremos los métodos `api.user.setAppMetadata` y `api.user.setUserMetadata`. Por ejemplo, para evitar que un comportamiento se ejecute más de una vez para un usuario concreto, considera una Action como esta:

```js lines
exports.onExecutePostLogin = async (event, api) => {
  if (event.user.app_metadata.didAnExpensiveTask) {
    console.log(`Skipping the expensive task because it already occurred for ${event.user.email}.`);
    return;
  }
  // realizar una tarea costosa
  api.user.setAppMetadata("didAnExpensiveTask", true);
};
```

Aquí, agregamos una comprobación al inicio de la Action para ver si ya hemos realizado la tarea costosa para este usuario. Si el campo de metadatos existe, salimos de la función.

Al final de la Action, llamamos a `api.user.setAppMetadata` para indicar que queremos almacenar metadatos en el objeto del usuario. Al final de la ejecución de cada trigger, Actions actualizará el perfil del usuario en una sola operación. Si se hacen varias llamadas a `setUserMetadata`, incluso si se hacen en distintas Actions como parte del mismo flujo, Actions solo actualizará el perfil del usuario una sola vez: al final de la ejecución del trigger.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Varias llamadas a `setUserMetadata` o `setAppMetadata` se agruparán en una sola actualización del perfil del usuario al final de la ejecución del trigger, incluso si las realizan distintas Actions.
</Callout>

<div id="best-practices">
  ## Prácticas recomendadas
</div>

Evita almacenar demasiados datos en el perfil de Auth0. Estos datos están pensados para usarse con fines de autenticación y autorización, y los usuarios pueden editar su propio campo `user_metadata`, así que no almacenes datos confidenciales en él. Los metadatos y las capacidades de búsqueda de Auth0 no están diseñados para estudios de mercado ni para otros casos que requieran búsquedas o actualizaciones muy frecuentes. Si usas Auth0 con ese propósito, es probable que tu sistema tenga problemas de escalabilidad y rendimiento. Una mejor opción es almacenar los datos en un sistema externo y guardar un puntero (el ID de usuario) en Auth0 para que los sistemas de backend puedan recuperar los datos si es necesario.

<div id="rate-limits">
  ## Límites de frecuencia
</div>

<Warning>
  La actualización de los metadatos del usuario y de la aplicación está sujeta a los límites de frecuencia de tu tenant y puede afectar al rendimiento del inicio de sesión.
</Warning>

Aunque solo se realiza una llamada para actualizar el perfil del usuario, esa operación sigue estando sujeta a los [límites de frecuencia](/docs/es/troubleshoot/customer-support/operational-policies/rate-limit-policy/management-api-endpoint-rate-limits) de &quot;Write User&quot; de tu tenant. Si se alcanza el límite de frecuencia al intentar actualizar los metadatos, Actions reintentará la solicitud mientras se siga devolviendo un código de estado HTTP `429`. El intervalo entre reintentos viene determinado por el valor del encabezado `X-RateLimit-Reset` devuelto como parte de la respuesta `429`.

<div id="redirects">
  ## Redirecciones
</div>

Si se invoca un Redirect con `api.redirect.sendUserTo()`, cualquier actualización pendiente de metadatos del usuario o de metadatos de la aplicación se aplicará al perfil del usuario antes de redirigirlo al sitio externo. Para obtener más información, consulta [Redirigir con Actions](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger/redirect-with-actions).
