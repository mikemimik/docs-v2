description: Aprende a usar el trigger de Action de Post Login para modificar `user_metadata` y `app_metadata` como parte del flujo de inicio de sesión de un usuario.
title: Administrar los metadatos del usuario con el trigger de Action de Post Login
---

Auth0 proporciona un sistema completo para almacenar [metadatos](/docs/es/manage-users/user-accounts/metadata) en el perfil de usuario de Auth0. Puedes configurar un trigger `post-login` para modificar `user_metadata` y `app_metadata` como parte del flujo de inicio de sesión de un usuario. Los triggers post-login son útiles para tareas como almacenar datos específicos de la aplicación en el perfil del usuario, capturar registros de actividad del usuario, mapear atributos de <Tooltip tip="Security Assertion Markup Language (SAML): Protocolo estandarizado que permite a dos partes intercambiar información de autenticación sin una contraseña." cta="Ver glosario" href="/docs/es/glossary?term=SAML">SAML</Tooltip> al campo de metadatos o almacenar en caché valores de operaciones costosas en el perfil del usuario para reutilizarlos en futuros inicios de sesión.

El objeto `api` de `post-login` proporciona operaciones comunes que se pueden realizar en este trigger. Para administrar los metadatos del usuario, queremos usar los métodos `api.user.setAppMetadata` y `api.user.setUserMetadata`. Por ejemplo, para evitar que cierto comportamiento se ejecute más de una vez para un usuario específico, considera una Action similar a esta:

```js lines
exports.onExecutePostLogin = async (event, api) => {
  if (event.user.app_metadata.didAnExpensiveTask) {
    console.log(`Skipping the expensive task because it already occurred for ${event.user.email}.`);
    return;
  }
  // realizar una tarea costosa
  api.user.setAppMetadata("didAnExpensiveTask", true);
};
```

Aquí, agregamos una verificación al inicio de la Action para ver si ya realizamos la tarea costosa para este usuario. Si el campo de metadato existe, salimos de la función.

Al final de la Action, llamamos a `api.user.setAppMetadata` para indicar que queremos almacenar metadato en el objeto de usuario. Al final de la ejecución de cada trigger, Actions actualizará el perfil del usuario en una sola operación. Si se realizan varias llamadas a `setUserMetadata`, incluso si se hacen en diferentes Actions como parte del mismo flujo, Actions solo actualizará el perfil del usuario una vez: al final de la ejecución del trigger.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Varias llamadas a `setUserMetadata` o `setAppMetadata` se agruparán en una sola actualización del perfil del usuario al final de la ejecución del trigger, incluso si las realizan diferentes Actions.
</Callout>

<div id="best-practices">
  ## Prácticas recomendadas
</div>

Evita almacenar demasiados datos en el perfil de Auth0. Estos datos están pensados para fines de autenticación y autorización, y los usuarios pueden editar su propio campo `user_metadata`, así que no guardes datos sensibles en él. Los metadatos y las capacidades de búsqueda de Auth0 no están diseñados para la investigación de marketing ni para otros casos que requieran búsquedas intensivas o una alta frecuencia de actualizaciones. Si usas Auth0 con este fin, es probable que tu sistema tenga problemas de escalabilidad y rendimiento. Es mejor almacenar los datos en un sistema externo y guardar un puntero (el ID de usuario) en Auth0 para que los sistemas backend puedan recuperar los datos si es necesario.

<div id="rate-limits">
  ## Límites de frecuencia
</div>

<Warning>
  La configuración de metadato de usuario y de la aplicación está sujeta a los límites de frecuencia de tu tenant y puede afectar el rendimiento del inicio de sesión.
</Warning>

Aunque se hace una sola llamada para actualizar el perfil de usuario, esa operación sigue estando sujeta a los [límites de frecuencia](/docs/es/troubleshoot/customer-support/operational-policies/rate-limit-policy/management-api-endpoint-rate-limits) de &quot;Write User&quot; de tu tenant. Si se alcanza el límite de frecuencia al intentar actualizar el metadato, Actions reintentará la solicitud siempre que se devuelva un código de estado HTTP `429`. El tiempo de espera entre reintentos se rige por el valor del encabezado `X-RateLimit-Reset` devuelto como parte de la respuesta `429`.

<div id="redirects">
  ## Redirecciones
</div>

Si se invoca una redirección con `api.redirect.sendUserTo()`, cualquier actualización pendiente de los metadatos del usuario o de la aplicación se aplicará al perfil del usuario antes de redirigirlo al sitio externo. Para obtener más información, consulta [Redirección con Actions](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger/redirect-with-actions).
