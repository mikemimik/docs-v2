* Authentication API: Sigue leyendo para aprender a llamar directamente a nuestra API. Si prefieres una experiencia interactiva, consulta nuestro [Device Flow Playground](https://auth0.github.io/device-flow-playground/).

<div id="prerequisites">
  ## Requisitos previos
</div>

**Antes de comenzar este tutorial:**

* Revisa las [limitaciones](#limitations) para asegurarte de que el flujo de autorización de dispositivos sea adecuado para tu implementación.
* [Registra la aplicación con Auth0](/docs/es/get-started/auth0-overview/create-applications/native-apps).

  * Selecciona **Native** como **Tipo de aplicación**.
  * Si es necesario, configura **Allowed Web Origins**. Puedes usar esta opción para permitir localhost como origen para el desarrollo local, o para establecer un origen permitido para determinado software de TV con una arquitectura sujeta a CORS (p. ej., HTML5 + JS). La mayoría de las aplicaciones no usarán esta configuración.
  * Asegúrate de que la opción **OIDC Conformant** esté habilitada. Esta configuración se encuentra en el [Dashboard](https://manage.auth0.com/#), en **Application Settings &gt; Advanced &gt; OAuth**.
  * Asegúrate de que los [**Grant Types**](/docs/es/get-started/applications/update-grant-types) de la aplicación incluyan **Device Code**. También encontrarás esta opción en el [Dashboard](https://manage.auth0.com/#), en **Application Settings &gt; Advanced &gt; Grant Types**.
  * Si quieres que tu aplicación pueda usar [tokens de actualización](/docs/es/secure/tokens/refresh-tokens), asegúrate de que los [**Grant Types**](/docs/es/get-started/applications/update-grant-types) de la aplicación incluyan **token de actualización**.
* Configura y habilita al menos una conexión para la aplicación: [conexiones de base de datos](/docs/es/get-started/applications/set-up-database-connections), [conexiones sociales](https://marketplace.auth0.com/features/social-connections)
* [Registra tu API con Auth0](/docs/es/get-started/architecture-scenarios/mobile-api/part-2#create-the-api)

  * Si quieres que tu API reciba [tokens de actualización](/docs/es/secure/tokens/refresh-tokens) para poder obtener tokens nuevos cuando caduquen los anteriores, habilita **Allow Offline Access**.
* [Configura los ajustes del código de usuario del dispositivo](/docs/es/get-started/tenant-settings/configure-device-user-code-settings) para definir el conjunto de caracteres, el formato y la longitud del código de usuario generado aleatoriamente.

<div id="steps">
  ## Pasos
</div>

1. [Solicitar código de dispositivo](#request-device-code) (Flujo del dispositivo): Solicita un código de dispositivo que el usuario pueda usar para autorizar el dispositivo.
2. [Solicitar activación del dispositivo](#request-device-activation) (Flujo del dispositivo): Solicita que el usuario autorice el dispositivo usando su laptop o smartphone.
3. [Solicitar tokens](#request-tokens) (Flujo del dispositivo): Consulta el endpoint de token para solicitar un token.
4. [Autorización del usuario](#user-authorization) (Flujo del navegador): El usuario autoriza el dispositivo para que este pueda recibir tokens.
5. [Recibir tokens](#receive-tokens) (Flujo del dispositivo): Después de que el usuario autorice correctamente el dispositivo, recibe los tokens.
6. [Llamar a tu API](#call-your-api) (Flujo del dispositivo): Usa el token de acceso obtenido para llamar a tu API.
7. [Actualizar tokens](#refresh-tokens) (Flujo del dispositivo): Usa un token de actualización para solicitar nuevos tokens cuando los actuales caduquen.

Opcional: [Explorar ejemplos de casos de uso](#sample-use-cases)

Opcional: [Solución de problemas](#troubleshooting)

<div id="request-device-code">
  ## Solicitar un código de dispositivo
</div>

Una vez que el usuario haya iniciado su app del dispositivo y quiera autorizar el dispositivo, tendrás que obtener un código de dispositivo. Cuando el usuario inicie su sesión en su dispositivo con navegador, este código quedará vinculado a esa sesión.

Para obtener el código de dispositivo, tu aplicación debe solicitar un código desde la [URL del código de dispositivo](https://auth0.com/docs/api/authentication#get-device-code), incluido el ID de cliente.

<div id="example-post-to-device-code-url">
  ### Ejemplo de solicitud POST a la URL del código de dispositivo
</div>

<AuthCodeGroup>
  ```bash cURL
  curl --request POST \
    --url 'https://{yourDomain}/oauth/device/code' \
    --header 'content-type: application/x-www-form-urlencoded' \
    --data 'client_id={yourClientId}' \
    --data 'scope=SCOPE' \
    --data 'audience=AUDIENCE'
  ```

  ```csharp C#
  var client = new RestClient("https://{yourDomain}/oauth/device/code");
  var request = new RestRequest(Method.POST);
  request.AddHeader("content-type", "application/x-www-form-urlencoded");
  request.AddParameter("application/x-www-form-urlencoded", "client_id={yourClientId}&scope=SCOPE&audience=AUDIENCE", ParameterType.RequestBody);
  IRestResponse response = client.Execute(request);
  ```

  ```go Go
  package main

  import (
  	"fmt"
  	"strings"
  	"net/http"
  	"io/ioutil"
  )

  func main() {

  	url := "https://{yourDomain}/oauth/device/code"

  	payload := strings.NewReader("client_id=%24%7Baccount.clientId%7D&scope=SCOPE&audience=AUDIENCE")

  	req, _ := http.NewRequest("POST", url, payload)

  	req.Header.Add("content-type", "application/x-www-form-urlencoded")

  	res, _ := http.DefaultClient.Do(req)

  	defer res.Body.Close()
  	body, _ := ioutil.ReadAll(res.Body)

  	fmt.Println(res)
  	fmt.Println(string(body))

  }
  ```

  ```java Java
  HttpResponse<String> response = Unirest.post("https://{yourDomain}/oauth/device/code")
    .header("content-type", "application/x-www-form-urlencoded")
    .body("client_id=%24%7Baccount.clientId%7D&scope=SCOPE&audience=AUDIENCE")
    .asString();
  ```

  ```javascript Node.JS
  var axios = require("axios").default;

  var options = {
    method: 'POST',
    url: 'https://{yourDomain}/oauth/device/code',
    headers: {'content-type': 'application/x-www-form-urlencoded'},
    data: new URLSearchParams({
      client_id: '{yourClientId}',
      scope: 'SCOPE',
      audience: 'AUDIENCE'
    })
  };

  axios.request(options).then(function (response) {
    console.log(response.data);
  }).catch(function (error) {
    console.error(error);
  });
  ```

  ```php PHP
  $curl = curl_init();

  curl_setopt_array($curl, [
    CURLOPT_URL => "https://{yourDomain}/oauth/device/code",
    CURLOPT_RETURNTRANSFER => true,
    CURLOPT_ENCODING => "",
    CURLOPT_MAXREDIRS => 10,
    CURLOPT_TIMEOUT => 30,
    CURLOPT_HTTP_VERSION => CURL_HTTP_VERSION_1_1,
    CURLOPT_CUSTOMREQUEST => "POST",
    CURLOPT_POSTFIELDS => "client_id=%24%7Baccount.clientId%7D&scope=SCOPE&audience=AUDIENCE",
    CURLOPT_HTTPHEADER => [
      "content-type: application/x-www-form-urlencoded"
    ],
  ]);

  $response = curl_exec($curl);
  $err = curl_error($curl);

  curl_close($curl);

  if ($err) {
    echo "cURL Error #:" . $err;
  } else {
    echo $response;
  }
  ```

  ```python Python
  import http.client

  conn = http.client.HTTPSConnection("")

  payload = "client_id=%24%7Baccount.clientId%7D&scope=SCOPE&audience=AUDIENCE"

  headers = { 'content-type': "application/x-www-form-urlencoded" }

  conn.request("POST", "/{yourDomain}/oauth/device/code", payload, headers)

  res = conn.getresponse()
  data = res.read()

  print(data.decode("utf-8"))
  ```

  response = http.request(request)
  puts response.read_body

<div id="device-code-parameters">
  #### Parámetros del código de dispositivo
</div>

Ten en cuenta que, al solicitar un código de dispositivo para llamar a una API personalizada:

* debes incluir un parámetro `audience`
* puedes incluir scopes adicionales compatibles con la API de destino

<Info>
  Si tu aplicación quiere un token de acceso solo para obtener información sobre el usuario autenticado, no se requiere ningún parámetro `audience`.
</Info>

| Nombre del parámetro | Descripción                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                         |
| -------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `client_id`          | El ID de cliente de tu aplicación. Puedes encontrar este valor en [Application Settings](https://manage.auth0.com/#/Applications/\{yourClientId}/settings).                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                         |
| `scope`              | Los [scopes](/docs/es/get-started/apis/scopes) para los que quieres solicitar autorización. Deben estar separados por un espacio. Puedes solicitar cualquiera de los [scopes estándar de OIDC](https://openid.net/specs/openid-connect-core-1_0.html#StandardClaims) sobre los usuarios, como `profile` y `email`, [custom claims](/docs/es/secure/tokens/json-web-tokens/json-web-token-claims#custom-claims) que sigan un [formato con espacio de nombres](/docs/es/secure/tokens/json-web-tokens/create-custom-claims), o cualquier [scope compatible con la API de destino](/docs/es/get-started/apis/scopes/api-scopes) (por ejemplo, `read:contacts`). Incluye `openid` para obtener un token de ID o para poder usar el endpoint [/userinfo](https://auth0.com/docs/api/authentication#user-profile) y recuperar información del perfil del usuario. Incluye `offline_access` para obtener un token de actualización (asegúrate de que el campo **Allow Offline Access** esté habilitado en [API Settings](https://manage.auth0.com/#/apis)). Ten en cuenta que este valor debe estar codificado en URL. |
| `audience`           | El identificador único de la API a la que tu aplicación quiere acceder. Usa el valor **Identifier** en la pestaña [Settings](https://manage.auth0.com/#/apis) de la API que creaste como parte de los requisitos previos de este tutorial. Ten en cuenta que este valor debe estar codificado en URL.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                               |

<div id="device-code-response">
  ### Respuesta de código de dispositivo
</div>

Si todo sale bien, recibirás una respuesta HTTP 200 con un payload que contiene los valores `device_code`, `user_code`, `verification_uri`, `expires_in`, `interval` y `verification_uri_complete`:

```json lines
{
  "device_code": "Ag_EE...ko1p",
  "user_code": "QTZL-MCBW",
  "verification_uri": "https://accounts.acmetest.org/activate",
  "verification_uri_complete": "https://accounts.acmetest.org/activate?user_code=QTZL-MCBW",
  "expires_in": 900,
  "interval": 5
}
```

* `device_code` es el código único del dispositivo. Cuando el usuario vaya a `verification_uri` en su device con navegador, este código quedará vinculado a su sesión.
* `user_code` contiene el código que debe introducirse en `verification_uri` para autorizar el dispositivo.
* `verification_uri` contiene la URL que el usuario debe visitar para autorizar el dispositivo.
* `verification_uri_complete` contiene la URL completa que el usuario debe visitar para autorizar el dispositivo. Esto permite que tu aplicación incluya el `user_code` en la URL, si así lo deseas.
* `expires_in` indica el tiempo de validez (en segundos) de `device_code` y `user_code`.
* `interval` indica el intervalo (en segundos) con el que la aplicación debe consultar la URL del token para solicitar un token.

<Info>
  Puedes [configurar el conjunto de caracteres, el formato y la longitud de tu código de usuario generado aleatoriamente](/docs/es/get-started/tenant-settings/configure-device-user-code-settings) en la Configuración del tenant.

  Para evitar ataques de fuerza bruta, aplicamos los siguientes límites a `user_code`:

  **Longitud mínima**:

  * Letras BASE20: 8 caracteres
  * Números: 9 caracteres

  **Longitud máxima**:

  * 20 caracteres (incluidos los guiones y espacios, que pueden añadirse como separadores para facilitar la lectura)

  **Tiempo de expiración**:

  * 15 minutos
</Info>

<div id="request-device-activation">
  ## Solicitar la activación del dispositivo
</div>

Una vez que hayas recibido un `device_code` y un `user_code`, debes pedirle al usuario que acceda a la `verification_uri` desde su portátil o smartphone e introduzca el `user_code`:

<Frame>![Solicitar la activación del dispositivo](https://cdn2.auth0.com/docs/1.14550.0/media/articles/flows/guides/device-auth/request-device-activation.png)</Frame>

El `device_code` no está destinado directamente al usuario y no debe mostrarse durante la interacción para evitar confusiones.

<Info>
  Al crear una CLI, podrías omitir este paso y abrir inmediatamente el navegador con `verification_uri_complete`.
</Info>

<div id="request-tokens">
  ## Solicitar tokens
</div>

Mientras esperas a que el usuario active el dispositivo, empieza a consultar la URL del token para solicitar un token de acceso. Usando el intervalo de consulta (`interval`) extraído del paso anterior, tendrás que hacer `POST` a la [URL del token](https://auth0.com/docs/api/authentication#device-auth) enviando el `device_code`.

Para evitar errores debidos a la latencia de red, debes empezar a contar cada intervalo después de recibir la respuesta a la última solicitud de consulta.

<div id="token-request-parameters">
  #### Parámetros de la solicitud de token
</div>

| Nombre del parámetro | Descripción                                                                                                                                                                                                                                                                   |
| -------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `grant_type`         | Configúralo como &quot;urn:ietf:params:oauth:grant-type:device&#95;code&quot;. Este es un tipo de concesión de extensión (según se define en la sección 4.5 de [RFC6749](https://tools.ietf.org/html/rfc6749#section-4.5)). Ten en cuenta que debe estar codificado para URL. |
| `device_code`        | El `device_code` obtenido en el paso anterior de este tutorial.                                                                                                                                                                                                               |
| `client_id`          | El ID de cliente de tu aplicación. Puedes encontrar este valor en la [Application Settings](https://manage.auth0.com/#/Applications/\{yourClientId}/settings).                                                                                                      |

<div id="token-responses">
  ### Respuestas de token
</div>

Mientras espera a que el usuario autorice el dispositivo, es posible que reciba distintas respuestas `HTTP 4xx`:

<div id="authorization-pending">
  #### Autorización pendiente
</div>

Verás este error mientras esperas a que el usuario actúe. Sigue consultando con el intervalo sugerido obtenido en el paso anterior de este tutorial.

```
`HTTP 403`

{
  "error": "authorization_pending",
  "error_description": "..."
}
```

<div id="slow-down">
  #### Ve más despacio
</div>

Estás consultando demasiado rápido. Ve más despacio y usa el intervalo sugerido obtenido en el paso anterior de este tutorial. Para evitar recibir este error debido a la latencia de la red, debes empezar a contar cada intervalo una vez recibida la respuesta a la última solicitud de consulta.

```
`HTTP 429`

{
  "error": "slow_down",
  "error_description": "..."
}
```

<div id="expired-token">
  #### Token expirado
</div>

El usuario no ha autorizado el dispositivo con la suficiente rapidez, por lo que el `device_code` ha expirado. Su aplicación debe notificar al usuario que el flujo ha expirado e indicarle que lo reinicie.

<Info>
  El error `expired_token` se devolverá exactamente una vez; después, se devolverá el temido `invalid_grant`. Su dispositivo *debe* dejar de consultar.
</Info>

```
`HTTP 403`

{ 
  "error": "expired_token",
  "error_description": "..."
}
```

<div id="access-denied">
  #### Acceso denegado
</div>

Por último, si se te deniega el acceso, recibirás:

```
`HTTP 403`

{
  "error": "access_denied",
  "error_description": "..."
}
```

Esto puede ocurrir por diversos motivos, entre ellos:

* el usuario se negó a autorizar el dispositivo
* el servidor de autorización rechazó la transacción
* una [Rule](/docs/es/customize/rules) configurada denegó el acceso

<div id="user-authorization">
  ## Autorización del usuario
</div>

El usuario escaneará el código QR o, en su defecto, abrirá la página de activación e introducirá el código de usuario:

<Frame>![Introducir código de usuario](https://cdn2.auth0.com/docs/1.14550.0/media/articles/flows/guides/device-auth/enter-user-code.png)</Frame>

Se mostrará una página de confirmación para que el usuario confirme que este es el dispositivo correcto:

<Frame>![Confirmar dispositivo](https://cdn2.auth0.com/docs/1.14550.0/media/articles/flows/guides/device-auth/confirm-device.png)</Frame>

El usuario completará la transacción iniciando sesión. Este paso puede incluir uno o más de los siguientes procesos:

* Autenticar al usuario;
* Redirigir al usuario a un proveedor de identidad para gestionar la autenticación;
* Comprobar si hay sesiones de SSO activas;
* Obtener el consentimiento del usuario para el dispositivo, a menos que ya se haya otorgado previamente.

<Frame>![Autenticar al usuario](https://cdn2.auth0.com/docs/1.14550.0/media/articles/flows/guides/device-auth/user-auth.png)</Frame>

Tras completar correctamente la autenticación y el consentimiento, se mostrará el mensaje de confirmación:

<Frame>![Confirmación del usuario](https://cdn2.auth0.com/docs/1.14550.0/media/articles/flows/guides/device-auth/user-confirmation.png)</Frame>

En este punto, el usuario se ha autenticado y el dispositivo ha sido autorizado.

<div id="receive-tokens">
  ## Recibir tokens
</div>

Mientras el usuario se autentica y autoriza el dispositivo, la aplicación del dispositivo sigue consultando la URL del token para solicitar un token de acceso.

Una vez que el usuario haya autorizado correctamente el dispositivo, recibirás una respuesta `HTTP 200` con un payload que contiene los valores `access_token`, `refresh_token` (opcional), `id_token` (opcional), `token_type` y `expires_in`:

```json lines
{
  "access_token":"eyJz93a...k4laUWw",
  "refresh_token":"GEbRxBN...edjnXbL",
  "id_token": "eyJ0XAi...4faeEoQ",
  "token_type":"Bearer",
  "expires_in":86400
}
```

<Warning>
  Debes validar tus tokens antes de guardarlos. Para saber cómo, consulta [Validar tokens de acceso](/docs/es/secure/tokens/access-tokens/validate-access-tokens) y [Validar tokens de ID](/docs/es/secure/tokens/id-tokens/validate-id-tokens).
</Warning>

Los [tokens de acceso](/docs/es/secure/tokens/access-tokens) se usan para llamar al [endpoint `/userinfo` de la Auth0 Authentication API](https://auth0.com/docs/api/authentication#get-user-info) o a otra API. Solo podrás usar el token de acceso para llamar a `/userinfo` si incluiste el scope `openid`. Si estás llamando a tu propia API, lo primero que deberá hacer tu API es [verificar el token de acceso](/docs/es/secure/tokens/access-tokens/validate-access-tokens).

Los [tokens de ID](/docs/es/secure/tokens/id-tokens) contienen información del usuario que debe [decodificarse y extraerse](/docs/es/secure/tokens/id-tokens#id-token-payload). El `id_token` solo estará presente en la respuesta si incluiste el scope `openid`.

Los [tokens de actualización](/docs/es/secure/tokens/refresh-tokens) se usan para obtener un nuevo token de acceso o token de ID después de que el anterior haya expirado. El `refresh_token` solo estará presente en la respuesta si incluiste el scope `offline_access` y habilitaste **Allow Offline Access** para tu API en el Dashboard.

<Warning>
  Los tokens de actualización deben almacenarse de forma segura, ya que permiten que un usuario permanezca autenticado prácticamente para siempre.
</Warning>

<div id="refresh-tokens">
  ## Tokens de actualización
</div>

Ya has recibido un [token de actualización](/docs/es/secure/tokens/refresh-tokens) si has seguido este tutorial y completaste lo siguiente:

* configuraste tu API para permitir el acceso sin conexión
* incluiste el scope `offline_access` cuando iniciaste la solicitud de autenticación a través del endpoint [authorize](https://auth0.com/docs/api/authentication#authorize-application)

Puedes usar el token de actualización para obtener un nuevo token de acceso. Por lo general, un usuario solo necesitará un nuevo token de acceso después de que el anterior caduque o cuando obtenga acceso a un nuevo recurso por primera vez. No es una buena práctica hacer una solicitud al endpoint para obtener un nuevo token de acceso cada vez que llamas a una API, y Auth0 aplica límites de frecuencia que restringen la cantidad de solicitudes al endpoint que pueden ejecutarse con el mismo token desde la misma IP.

Para actualizar tu token, haz una solicitud `POST` al endpoint `/oauth/token` de la Authentication API, usando `grant_type=refresh_token`.

<div id="refresh-token-request-parameters">
  #### Parámetros de la solicitud del token de actualización
</div>

| Nombre del parámetro | Descripción                                                                                                                                                                                                                             |
| -------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `grant_type`         | Establécelo como &quot;refresh&#95;token&quot;.                                                                                                                                                                                         |
| `client_id`          | El ID de cliente de tu aplicación. Puedes encontrar este valor en la [Application Settings](https://manage.auth0.com/#/Applications/\{yourClientId}/settings).                                                                |
| `client_secret`      | El secreto de cliente de tu aplicación. Puedes encontrar este valor en la [Application Settings](https://manage.auth0.com/#/Applications/\{yourClientSecret}/settings).                                                       |
| `refresh_token`      | El token de actualización que se usará.                                                                                                                                                                                                 |
| `scope`              | (Opcional) Una lista de scopes solicitados, separada por espacios. Si no se envía, se usarán los scopes originales; de lo contrario, puedes solicitar un conjunto reducido de scopes. Ten en cuenta que debe estar codificado como URL. |

<div id="refresh-token-response">
  ### Respuesta del token de actualización
</div>

Si todo sale bien, recibirás una respuesta `HTTP 200` con un payload que incluye un nuevo `access_token`, `id_token` (opcional), la duración del token de acceso en segundos (`expires_in`), los valores de `scope` concedidos y `token_type`:

```json lines
{
  "access_token": "eyJ...MoQ",
  "expires_in": 86400,
  "scope": "openid offline_access",
  "id_token": "eyJ...0NE",
  "token_type": "Bearer"
}
```

<Warning>
  Debes validar tus tokens antes de guardarlos. Para saber cómo hacerlo, consulta [Validar tokens de acceso](/docs/es/secure/tokens/access-tokens/validate-access-tokens) y [Validar tokens de ID](/docs/es/secure/tokens/id-tokens/validate-id-tokens).
</Warning>

<div id="limitations">
  ## Limitaciones
</div>

Para usar el flujo de autorización de dispositivos, los dispositivos deben:

* Admitir Server Name Indication (SNI)
* Tener un [tipo de aplicación de Auth0](/docs/es/get-started/applications) **Native**
* Tener el [**Token Endpoint Authentication Method**](/docs/es/get-started/applications/application-settings) configurado como **None**
* Ser [OIDC-conformant](/docs/es/get-started/applications/application-settings#oauth)
* No haber sido creados mediante [registro dinámico de clientes](/docs/es/get-started/applications/dynamic-client-registration)

Además, el flujo de autorización de dispositivos no permite:

* [Social Connections](/docs/es/authenticate/identity-providers) que usen [Auth0 developer keys](/docs/es/authenticate/identity-providers/social-identity-providers/devkeys), a menos que use la nueva [experiencia de Universal Login](/docs/es/authenticate/login/auth0-universal-login/universal-login-vs-classic-login/universal-experience).
* Que se acceda a los parámetros de la query string desde la página de inicio de sesión alojada o las reglas

Admitimos el [Draft 15](https://tools.ietf.org/html/draft-ietf-oauth-device-flow-15) completo, excepto para clientes confidenciales.
