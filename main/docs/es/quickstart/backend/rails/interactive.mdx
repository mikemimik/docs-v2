  <Content>
    Este tutorial realiza la validación del token de acceso usando la gema [jwt](https://github.com/jwt/ruby-jwt) dentro de una clase `Auth0Client` personalizada. Se usa un Concern llamado
    `Secured` para autorizar los endpoints que requieren autenticación a través de un token de acceso entrante.

    Si aún no has creado una API en tu Auth0 Dashboard, usa el selector interactivo para crear una nueva API de Auth0
    o selecciona una API existente para tu proyecto.

    Para configurar tu primera API a través del Auth0 Dashboard, consulta [nuestra guía de introducción](https://auth0.com/docs/get-started/auth0-overview/set-up-apis).

    Cada Auth0 API utiliza el identificador de API, que su aplicación necesita para validar el token de acceso.

    <Section id={sections[2].id} title={sections[2].title} stepNumber="3">
      Crea una clase llamada `Auth0Client`. Esta clase decodifica y verifica el token de acceso entrante tomado
      del encabezado `Authorization` de la solicitud.

      La clase `Auth0Client` recupera la clave pública de tu tenant de Auth0 y luego la usa para verificar la
      firma del token de acceso. La estructura `Token` define un método `validate_permissions` para
      buscar un `scope` específico en un token de acceso, proporcionando un arreglo de scopes requeridos y comprobando si
      están presentes en la carga útil del token.
    </Section>

    <Section id={sections[3].id} title={sections[3].title} stepNumber="4">
      Cree un Concern llamado `Secured` que busque el token de acceso en el encabezado `Authorization`
      de una request entrante.

      Si el token está presente, `Auth0Client.validate_token` usará la gema `jwt` para verificar
      la firma del token y validar los claims del token.

      Además de verificar que el token de acceso sea válido, el Concern también incluye un mecanismo para confirmar que el
      token tiene el **scope** suficiente para acceder a los recursos solicitados. En este ejemplo definimos un
      método `validate_permissions` que recibe un bloque y comprueba los permisos llamando al
      método `Token.validate_permissions` de la clase `Auth0Client`.

      Para la ruta `/private-scoped`, los scopes definidos se cruzarán con los scopes que llegan en el
      payload para determinar si contiene uno o más elementos del otro arreglo.
    </Section>

    <Section id={sections[4].id} title={sections[4].title} stepNumber="5">
      Al añadir el concern `Secure` a tu controlador de aplicación, solo tendrás que usar un
      filtro `before_action` en el controlador que requiera autorización.
    </Section>

    <Section id={sections[5].id} title={sections[5].title} stepNumber="6">
      Crea un controlador para gestionar el endpoint público `/api/public`.

      El endpoint `/public` no requiere autorización, por lo que no hace falta ningún `before_action`.
    </Section>

    <Section id={sections[6].id} title={sections[6].title} stepNumber="7">
      Crea un controlador para gestionar los endpoints privados: `/api/private` y
      `/api/private-scoped`.

      `/api/private` está disponible para solicitudes autenticadas que contengan un access token sin
      ámbitos adicionales.

      `/api/private-scoped` está disponible para solicitudes autenticadas que contengan un token de acceso con el
      scope `read:messages` concedido

      Los endpoints protegidos necesitan llamar al método `authorize` del concern `Secured`. Para ello, se usa `before_action :authorize`, lo que garantiza que el método `Secured.authorize` se ejecute antes de cada acción en el `PrivateController`.
