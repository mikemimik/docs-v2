<div id="get-started">
  ## Primeros pasos
</div>

Auth0 te permite agregar autorización a cualquier tipo de aplicación. Esta guía muestra cómo integrar Auth0 con cualquier aplicación ASP.NET OWIN Web API, ya sea nueva o existente, mediante el paquete `Microsoft.Owin.Security.Jwt`. Cada API de Auth0 usa el identificador de API, que tu aplicación necesita para validar el token de acceso.

Este ejemplo muestra:

* Cómo comprobar si hay un JSON Web Token (JWT) en el encabezado `Authorization` de una solicitud HTTP entrante.
* Cómo comprobar si el token es válido usando el [JSON Web Key Set (JWKS)](/docs/es/secure/tokens/json-web-tokens/json-web-key-sets) de tu cuenta de Auth0. Para obtener más información sobre cómo validar tokens de acceso, consulta [Validar tokens de acceso](/docs/es/secure/tokens/access-tokens/validate-access-tokens).

<Steps>
  <Step title="Crear una API" stepNumber={1}>
    En la sección [APIs](https://manage.auth0.com/#/apis) del Auth0 Dashboard, haz clic en **Crear API**. Proporciona un nombre y un identificador para tu API, por ejemplo, `https://quickstarts/api`. Más adelante usarás el identificador como `audience` al configurar la verificación del access token. Deja el **Algoritmo de firma** en **RS256**.

    <Frame>![Crear API](https://cdn2.auth0.com/docs/1.14550.0/media/articles/server-apis/create-api.png)</Frame>

    De forma predeterminada, tu API usa RS256 como algoritmo para firmar tokens. Como RS256 usa un par de claves privada/pública, valida los tokens con la clave pública de tu cuenta de Auth0. La clave pública está en formato [JSON Web Key Set (JWKS)](/docs/es/secure/tokens/json-web-tokens/json-web-key-sets) y puedes acceder a ella [aquí](https://\{yourDomain}/.well-known/jwks.json).
  </Step>

  <Step title="Definir permisos" stepNumber={2}>
    Los permisos te permiten definir cómo se puede acceder a los recursos en nombre del usuario con un determinado token de acceso. Por ejemplo, puedes optar por conceder acceso de lectura al recurso `messages` si los usuarios tienen el nivel de acceso de gerente, y acceso de escritura a ese recurso si tienen el nivel de acceso de administrador.

    Puedes definir los permisos permitidos en la vista **Permissions** de la sección [APIs](https://manage.auth0.com/#/apis) de Auth0 Dashboard.

    <Frame>![Configurar permisos](https://cdn2.auth0.com/docs/1.14550.0/media/articles/server-apis/configure-permissions.png)</Frame>

    <Info>
      Este ejemplo usa el scope `read:messages`.
    </Info>
  </Step>

  <Step title="Configura el proyecto de ejemplo" stepNumber={3}>
    El código de ejemplo incluye una sección `appsettings` en `Web.config` que lo configura para usar el **Dominio** y el **identificador de API** correctos para su API. Si descarga el código de esta página, se completará automáticamente. Si usa el ejemplo de Github, tendrá que completarlo usted mismo.

    ```xml web.config lines
    <appSettings>
      <add key="Auth0Domain" value="{yourDomain}" />
      <add key="Auth0ApiIdentifier" value="{yourApiIdentifier}" />
    </appSettings>
    ```
  </Step>

  <Step title="Instala las dependencias" stepNumber={4}>
    Para usar los tokens de acceso de Auth0 con ASP.NET, utilizará el middleware JWT de OWIN, que está disponible en el paquete NuGet `Microsoft.Owin.Security.Jwt`.

    ```bash lines
    Install-Package Microsoft.Owin.Security.Jwt
    ```
  </Step>

  <Step title="Verificar la firma del token" stepNumber={5}>
    Como el middleware JWT de OWIN no usa el descubrimiento de OpenID Connect de forma predeterminada, deberá proporcionar un `IssuerSigningKeyResolver` personalizado. Para ello, agregue lo siguiente al archivo `Support/OpenIdConnectSigningKeyResolver.cs`:

    <Info>
      Anteriormente, este resolvedor personalizado se publicaba como parte del paquete `Auth0.OpenIdConnectSigningKeyResolver` a través de NuGet. Como [este paquete ya no está disponible](https://github.com/auth0/auth0-aspnet-owin/blob/master/SECURITY-NOTICE.md), deberá proporcionarlo usted mismo.
    </Info>

    ```cs OpenIdConnectSigningKeyResolver.cs lines
    public class OpenIdConnectSigningKeyResolver
    {
        private readonly OpenIdConnectConfiguration openIdConfig;

        public OpenIdConnectSigningKeyResolver(string authority)
        {
            var cm = new ConfigurationManager<OpenIdConnectConfiguration>($"{authority.TrimEnd('/')}/.well-known/openid-configuration", new OpenIdConnectConfigurationRetriever());
            openIdConfig = AsyncHelper.RunSync(async () => await cm.GetConfigurationAsync());
        }

        public SecurityKey[] GetSigningKey(string kid)
        {
            return new[] { openIdConfig.JsonWebKeySet.GetSigningKeys().FirstOrDefault(t => t.KeyId == kid) };
        }
    }
    ```

    `OpenIdConnectSigningKeyResolver` descargará automáticamente el JSON Web Key Set que se utiliza para firmar los tokens RS256 desde el endpoint de configuración de OpenID Connect (en `/.well-known/openid-configuration`). Después, podrás usarlo para resolver la clave de firma del issuer, como se mostrará en el código de registro de JWT que aparece a continuación.
  </Step>

  <Step title="Configurar la autenticación JWT" stepNumber={6}>
    Ve al método `Configuration` de tu clase `Startup` y agrega una llamada a `UseJwtBearerAuthentication`, pasando las `JwtBearerAuthenticationOptions` configuradas.

    `JwtBearerAuthenticationOptions` debe especificar tu identificador de API de Auth0 en la propiedad `ValidAudience`, y la ruta completa de tu dominio de Auth0 como `ValidIssuer`. Tendrás que configurar `IssuerSigningKeyResolver` para que use la instancia de `OpenIdConnectSigningKeyResolver` para resolver la clave de firma:

    ```cs Startup.cs lines
    public void Configuration(IAppBuilder app)
    {
        var domain = $"https://{ConfigurationManager.AppSettings["Auth0Domain"]}/";
        var apiIdentifier = ConfigurationManager.AppSettings["Auth0ApiIdentifier"];
        var keyResolver = new OpenIdConnectSigningKeyResolver(domain);

        app.UseJwtBearerAuthentication(
            new JwtBearerAuthenticationOptions
            {
                AuthenticationMode = AuthenticationMode.Active,
                TokenValidationParameters = new TokenValidationParameters()
                {
                    ValidAudience = apiIdentifier,
                    ValidIssuer = domain,
                    IssuerSigningKeyResolver = (token, securityToken, kid, parameters) => keyResolver.GetSigningKey(kid)
                }
            });

        // Configurar Web API
        WebApiConfig.Configure(app);
    }
    ```

    <Warning>
      ### No olvides la barra invertida final

      Asegúrate de que la URL especificada para `ValidIssuer` incluya una barra invertida al final, ya que debe coincidir exactamente con el claim `issuer` del JWT. Este es un error de configuración frecuente que hará que tus llamadas a la API no se autentiquen correctamente.
    </Warning>
  </Step>

  <Step title="Validar permisos de acceso" stepNumber={7}>
    El `middleware` de JWT anterior verifica que el token de acceso incluido en la solicitud sea válido; sin embargo, todavía no incluye ningún mecanismo para comprobar que el token tenga el **scope** suficiente para acceder a los recursos solicitados.

    Cree una clase llamada `ScopeAuthorizeAttribute` que herede de `System.Web.Http.AuthorizeAttribute`. Este atributo de autorización comprobará que el claim `scope` emitido por su tenant de Auth0 esté presente y, de ser así, se asegurará de que el claim `scope` contenga el scope solicitado.

    ```cs ScopeAuthorizeAttribute.cs lines
    public class ScopeAuthorizeAttribute : AuthorizeAttribute
    {
        private readonly string scope;

        public ScopeAuthorizeAttribute(string scope)
        {
            this.scope = scope;
        }

        public override void OnAuthorization(HttpActionContext actionContext)
        {
            base.OnAuthorization(actionContext);

            // Obtener el Auth0 domain para validar el issuer
            var domain = $"https://{ConfigurationManager.AppSettings["Auth0Domain"]}/";

            // Obtener el principal del claim
            ClaimsPrincipal principal = actionContext.ControllerContext.RequestContext.Principal as ClaimsPrincipal;

            // Obtener el claim de scope. Verificar que el issuer corresponde al Auth0 domain correcto
            var scopeClaim = principal?.Claims.FirstOrDefault(c => c.Type == "scope" && c.Issuer == domain);
            if (scopeClaim != null)
            {
                // Separar los scopes
                var scopes = scopeClaim.Value.Split(' ');

                // Retornar si el arreglo de scopes contiene el scope requerido
                if (scopes.Any(s => s == scope))
                    return;
            }

            HandleUnauthorizedRequest(actionContext);
        }
    }
    ```
  </Step>

  <Step title="Protege los endpoints de la API" stepNumber={8}>
    Las rutas que se muestran a continuación están disponibles para las siguientes solicitudes:

    * `GET /api/public`: disponible para solicitudes no autenticadas
    * `GET /api/private`: disponible para solicitudes autenticadas que contengan un token de acceso sin `scopes` adicionales
    * `GET /api/private-scoped`: disponible para solicitudes autenticadas que contengan un token de acceso con el `scope` `read:messages` concedido

    El middleware JWT se integra con los mecanismos estándar de ASP.NET Authentication and Authorization, por lo que solo necesita decorar la acción del controlador con el atributo `[Authorize]` para proteger un endpoint. Para asegurarse de que se incluya un `scope` y así poder llamar a un endpoint de API en particular, decore la acción con el atributo `ScopeAuthorize` y pase el nombre del `scope` requerido en el parámetro `scope`.

    ```cs ApiController.cs lines
    [RoutePrefix("api")]
    public class ApiController : ApiController
    {
        [HttpGet]
        [Route("public")]
        public IHttpActionResult Public()
        {
            return Json(new
            {
                Message = "Hello from a public endpoint!"
            });
        }

        [HttpGet]
        [Route("private")]
        [Authorize]
        public IHttpActionResult Private()
        {
            return Json(new
            {
                Message = "Hello from a private endpoint! You need to be authenticated to see this."
            });
        }

        [HttpGet]
        [Route("private-scoped")]
        [ScopeAuthorize("read:messages")]
        public IHttpActionResult Scoped()
        {
            return Json(new
            {
                Message = "Hello from a private endpoint! You need to be authenticated and have a scope of read:messages to see this."
            });
        }
    }
    ```
  </Step>
</Steps>

<Check>
  **Verificación**

  Ahora que has configurado tu aplicación, ejecútala y verifica que:

  * `GET /api/public` esté disponible para solicitudes no autenticadas.
  * `GET /api/private` esté disponible para solicitudes autenticadas.
  * `GET /api/private-scoped` esté disponible para solicitudes autenticadas que contengan un token de acceso con el scope `read:messages`.
</Check>
