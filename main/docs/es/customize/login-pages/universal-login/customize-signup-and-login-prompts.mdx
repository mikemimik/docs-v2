<Card title="Antes de empezar">
  1. Asegúrate de que tu tenant tenga un [**Custom Domain**](/docs/es/customize/custom-domains) configurado.
  2. Confirma que estás usando [**Universal Login**](/docs/es/authenticate/login/auth0-universal-login/universal-login-vs-classic-login/universal-experience) para todos los prompts de registro e inicio de sesión y asegúrate de que el toggle **Customize Login Page** esté desactivado para los prompts de inicio de sesión.
  3. Comprueba que tengas configurada una [plantilla de página personalizada](/docs/es/customize/login-pages/universal-login/customize-templates#custom-sign-up-prompts).
</Card>

**Personalizar los prompts de registro e inicio de sesión** es una funcionalidad que permite a los clientes que tengan habilitados <Tooltip tip="Custom Domain: Dominio de terceros con un nombre especializado o personalizado." cta="Ver glosario" href="/docs/es/glossary?term=Custom+Domain">Custom Domain</Tooltip> y una plantilla de página personalizada agregar campos personalizados y contenido a los prompts de registro e inicio de sesión de su aplicación.

<div id="use-cases">
  ## Casos de uso
</div>

Personalizar los prompts de registro e inicio de sesión admite dos casos de uso: **contenido personalizado** y **captura de datos**.

El **contenido personalizado** es contenido estático, como texto, enlaces o imágenes, que se coloca directamente en los prompts de registro e inicio de sesión.

La **captura de datos** utiliza elementos de formulario que se agregan dinámicamente a los prompts de registro e inicio de sesión, lo cual resulta útil para recopilar y validar el consentimiento del usuario o datos proporcionados por el usuario, como el apellido.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Usa Signup and Login Prompt Customizations únicamente para comunicar o recopilar datos sensibles o regulados, según lo permita tu acuerdo con Okta.
</Callout>

<div id="terminology">
  ## Terminología
</div>

Un **prompt** es un paso específico de un flujo de autenticación. Cada prompt tiene al menos una **pantalla** y, según la configuración del tenant, cada pantalla compatible tiene cuatro o seis **puntos de entrada**, es decir, ubicaciones dentro de la pantalla donde se puede insertar código personalizado (**partials**).

Se pueden personalizar los siguientes prompts:

**Pantallas de inicio de sesión**

* `login`
* `login-id`
* `login-password`
* `login-passwordless-sms-otp`
* `login-passwordless-email-code`
* `passkey-enrollment`
* `passkey-enrollment-local`

**Pantallas de registro**

* `signup`
* `signup-id`
* `signup-password`
* `passkey-enrollment`
* `passkey-enrollment-local`

Los partials admiten HTML, CSS, JavaScript y la [sintaxis Liquid](https://github.com/Shopify/liquid/wiki/Liquid-for-Designers) para habilitar lógica condicional y variables dinámicas. Además, también se admite cualquier variable de Liquid disponible para la [plantilla de página](/docs/es/customize/login-pages/universal-login/customize-templates).

Estos puntos de entrada están disponibles cuando está habilitada una conexión de base de datos o una conexión passwordless:

* `form-content-start`
* `form-content-end`
* `form-footer-start`
* `form-footer-end`

Los siguientes puntos de entrada están disponibles cuando está habilitada al menos una conexión social o empresarial:

* `secondary-actions-start`
* `secondary-actions-end`

<div id="use-the-auth0-dashboard-to-manage-partials">
  ## Usa el Auth0 Dashboard para gestionar Partials
</div>

Usa el Dashboard para insertar campos personalizados y contenido en las pantallas de Login y Signup mediante partials.

1. Ve a [Auth0 Dashboard &gt; Branding &gt; Universal Login](https://manage.auth0.com/#/universal-login/customize-partials) y selecciona **Enhance screens with partials**.
2. Selecciona la **Screen** que quieres personalizar con el editor de partials.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  **Las pantallas** pueden tener los siguientes **modos de renderización**:

  * **STANDARD**: La `screen` se renderiza con la UI predeterminada de [Universal Login](/docs/es/customize/login-pages/universal-login); puedes usar partials para insertar fragmentos de código y [template variables](/docs/es/customize/login-pages/universal-login/customize-templates#page-template-variables).
  * **ADVANCED**: La `screen` se renderiza con [ACUL](/docs/es/customize/login-pages/advanced-customizations) y los partials no se aplican.
  * **ADVANCED (FILTERED)**: La `screen` se renderiza con ACUL aplicado a aplicaciones y organizaciones específicas; los partials solo se aplican a las `screens` excluidas por los filtros de ACUL.
</Callout>

<div id="manage-partials-programmatically">
  ## Gestiona Partials mediante programación
</div>

Puedes gestionar Partials con la [Auth0 Management API](https://auth0.com/docs/api/management/v2/prompts/put-partials) (`GET /prompts/{prompts_name}/partials`). Cada prompt debe especificar la `screen` al agregar, actualizar o eliminar un partial. Los Partials pueden tener un máximo de 10,000 caracteres.

Los Partials también se pueden gestionar con la interfaz Customize de <Tooltip tip="Universal Login: tu aplicación redirige a Universal Login, alojado en el servidor de autorización de Auth0, para verificar la identidad de un usuario." cta="Ver glosario" href="/docs/es/glossary?term=Universal+Login">Universal Login</Tooltip> de Auth0 CLI ejecutando `auth0 ul customize` en tu terminal.

<div id="style-and-validate-form-inputs">
  ## Dar estilo y validar los campos de entrada del formulario
</div>

Customize Signup and Login Prompts ofrece estilos prediseñados y compatibilidad con validación del lado del cliente para ciertos elementos HTML de `form`. Se admiten los siguientes elementos:

* `<input type="text">`
* `<input type="number">`
* `<input type="checkbox">`
* `<input type="password">`
* `<input type="email">`
* `<input type="tel">`
* `<input type="url">`
* `<select>`
* `<textarea>`

Para usar los estilos prediseñados de los campos de entrada, envuelve el elemento de `form` que elijas en un `<div>` con la clase `ulp-field`. Del mismo modo, agrega la clase `ulp-error` al mismo `<div>` para usar los estilos de error prediseñados. Si el elemento `ulp-error-info` está presente, también se mostrará un mensaje de error con estilo.

Puedes encontrar fragmentos de código prediseñados al usar el editor de [partials](#use-the-auth0-dashboard-to-manage-partials).

<div id="client-side-validation">
  ### Validación del lado del cliente
</div>

El marco de validación del lado del cliente de esta funcionalidad permite a los clientes validar los datos introducidos por el usuario mediante atributos HTML para ejecutar una o varias funciones de validación personalizadas. Las funciones de validación pueden incluirse directamente en el Partial o en el `<head>` de la plantilla de página.

Para agregar validación del lado del cliente a un elemento de formulario:

* Haga referencia a la función de validación mediante el atributo `data-ulp-validation-function` en el elemento `<div class="ulp-error-info">`.
* Indique en qué eventos del DOM debe ejecutarse la función de validación mediante el atributo `data-ulp-validation-event-listeners` en el elemento `<div class="ulp-error-info">`, teniendo en cuenta que las validaciones se ejecutan automáticamente al enviar el formulario.
* Para cumplir con las WCAG, los campos de entrada deben estar vinculados mediante programación a sus mensajes de error; por ejemplo, con `aria-describedby="error-id"` y `aria-invalid="true"` para garantizar que los lectores de pantalla anuncien los errores de validación.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Tenga cuidado al usar JavaScript de terceros en su página de registro. La información confidencial relacionada con la seguridad suele pasar por la página de registro, lo que la hace vulnerable al cross-site scripting.

  Siempre que sea posible, Auth0 recomienda validar los datos proporcionados por el usuario antes de enviarlos.
</Callout>

<div id="localize-content">
  ## Localizar contenido
</div>

Es posible localizar contenido parcial definiendo nuevas variables de texto personalizadas mediante la [Custom Text API](/docs/es/api/management/v2/prompts/put-custom-text-by-language). Se pueden definir hasta treinta variables de texto personalizadas por combinación de screen/idioma.

<div id="create-or-update-a-custom-text-variable">
  ### Crear o actualizar una variable de texto personalizada
</div>

Gestiona las variables de texto personalizadas con la [Custom Text API](/docs/es/api/management/v2/prompts/get-custom-text-by-language). Las llamadas deben especificar la `screen` al agregar, actualizar o eliminar una variable de texto personalizada. Las variables de texto personalizadas siguen la convención de nomenclatura `var-<name>`. Se admiten enlaces de Markdown y se convierten en elementos HTML `<a>` antes de mostrarse a los usuarios.

A continuación, se muestra una llamada de ejemplo para agregar una variable para el texto de la etiqueta de una casilla de verificación de Términos de servicio en inglés y en español. Consulta la <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip> para obtener más información.

```json lines
# PUT /api/v2/prompts/signup-id/custom-text/en
{
  "signup": {
    "var-tos": "I agree with the [Terms of Service](https://en.example.com/tos)"
  }
}

# PUT /api/v2/prompts/signup-id/custom-text/es
{
  "signup": {
    "var-tos": "Estoy de acuerdo con los [Términos de Servicio](https://es.example.com/tos)"
  }
}
```

<div id="use-a-custom-text-variable-in-a-partial">
  ### Usa una variable de texto personalizada en un Partial
</div>

Las variables de texto personalizadas se referencian en partials mediante el objeto `prompts.screen.text`; la referencia para el ejemplo de `var-tos` de la sección anterior es `prompt.screen.text.varTos`. A continuación, se muestra un ejemplo de cómo usar una variable creada previamente en un partial del prompt de ID de Signup, teniendo en cuenta que la variable `var-tos` de la Management API se referencia como `varTos` en el partial.

```html
# PUT api/v2/prompts/signup/partials
<div class='ulp-field'>
  <input type='checkbox' name='ulp-terms-of-service' id='terms-of-service'>
  <label for='terms-of-service'>{{ prompt.screen.text.varTos }}</label>
</div>
```

<div id="validate-and-save-captured-data">
  ## Validar y guardar los datos capturados
</div>

Los datos capturados por los elementos de formulario personalizados están disponibles en Actions, y Auth0 recomienda validar los datos recopilados antes de guardarlos o enviarlos.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Al usar elementos de formulario personalizados, debes incluir el prefijo `ulp-` en cualquier nombre de campo para asegurarte de que los datos estén disponibles en [Actions](/docs/es/customize/actions/actions-overview).
</Callout>

Cada Action recibe los datos capturados como un objeto en `event.request.body`. Los clientes pueden devolver un error de validación mediante la función `api.validation.error`.

Al usar una [conexión de base de datos](/docs/es/authenticate/database-connections#use-auth0-user-store):

* Se puede acceder a los datos de los prompts de registro en el trigger [`pre-user-registration`](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/pre-user-registration-trigger). Un error de validación en el trigger impide que el usuario se registre.
* Se puede acceder a los datos de los prompts de inicio de sesión en el trigger [`post-login`](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger). Los errores de validación se reenvían a la página de error de la aplicación del cliente.

Al usar una [conexión de base de datos personalizada](/docs/es/authenticate/database-connections/custom-db):

* Se puede acceder a los datos de los prompts de registro en el trigger [`pre-user-registration`](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/pre-user-registration-trigger) con los siguientes scripts de acción de base de datos personalizados: [Crear usuario](/docs/es/authenticate/database-connections/custom-db/templates/create) y [Login](/docs/es/authenticate/database-connections/custom-db/templates/login).

* Se puede acceder a los datos de los prompts de inicio de sesión en el trigger [`post-login`](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger) con los siguientes scripts de acción de base de datos personalizados: [Login](/docs/es/authenticate/database-connections/custom-db/templates/login) y [Change Password](/docs/es/authenticate/database-connections/custom-db/templates/change-password).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Para usar scripts de acción de base de datos personalizados, habilita **Objetos `context` en scripts de base de datos** en tu conexión de base de datos personalizada. Para obtener más información, consulta [Habilitar objeto `context`](https://auth0.com/docs/authenticate/database-connections/custom-db/create-db-connection#enable-context-object).
</Callout>

Al usar una [conexión passwordless](/docs/es/authenticate/passwordless/passwordless-with-universal-login):

* Se puede acceder a los datos tanto de los prompts de registro como de los prompts de inicio de sesión en el trigger [`post-login`](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger). Los errores de validación se reenvían a la página de error de la aplicación del cliente.

Al usar prompts de [`clave de acceso`](/docs/es/authenticate/database-connections/passkeys):

* Se puede acceder a los datos del prompt `passkey-enrollment` en el trigger [`pre-user-registration`](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/pre-user-registration-trigger). Un error de validación en el trigger impide que el usuario se registre.
* Los prompts `passkey-enrollment` y `passkey-enrollment-local` nunca capturan datos en el trigger Post Login.
* No se puede acceder a los datos del prompt `passkey-enrollment-local`, ya que siempre se muestra después de que se ejecuta el trigger [`post-login`](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger).

Al usar conexiones sociales o empresariales:

* Los datos *no* se capturan cuando un usuario inicia sesión con una conexión [social](/docs/es/authenticate/identity-providers/social-identity-providers) o [empresarial](/docs/es/authenticate/enterprise-connections).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Sanea cualquier dato que recopiles en el formulario antes de guardarlo o renderizarlo.

  * Asegúrate de que cualquier dato que se guarde haya pasado por la [función auxiliar `{{escape}}` de Liquid](https://shopify.github.io/liquid/filters/escape/)
  * Si renderizas datos en una plantilla de correo electrónico, elimina la sintaxis de Liquid
  * Si renderizas datos en una página web, escapa las entidades HTML
  * Si guardas datos en una base de datos, usa consultas parametrizadas
  * Si pasas datos en una cadena de consulta, codifícalos con, por ejemplo: `{{encodeURI}}` o `{{encodeURIParam}}`

  Para obtener más información sobre la mitigación de riesgos y las prácticas recomendadas para almacenar datos de forma segura, consulta [esta hoja de referencia](https://cheatsheetseries.owasp.org/cheatsheets/Input_Validation_Cheat_Sheet.html).
</Callout>

<div id="save-to-user-metadata">
  ### Guardar en los metadatos de usuario
</div>

Desde la Action, los datos recopilados pueden enviarse a una API externa para su validación y almacenamiento, o bien guardarse en `user_metadata` del usuario mediante `api.user.setUserMetadata`.

```js lines
// Dado este código en el formulario de registro
// <div class="ulp-field">
//   <label for="full-name">Full Name</label>
//   <input type="text" name="ulp-full-name" id="full-name">
// </div>

exports.onExecutePreUserRegistration = async (event, api) => {
  const fullName = event.request.body['ulp-full-name'];
  if(!fullName) {
    api.validation.error("invalid_payload", "Missing Name");
    return;
  }

  api.user.setUserMetadata("fullName", fullName);
};
```
