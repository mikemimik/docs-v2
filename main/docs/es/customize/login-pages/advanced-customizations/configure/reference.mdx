---

Cada pantalla de Universal Login tiene su propia configuración de renderización, un objeto JSON que controla cómo Auth0 renderiza esa pantalla. La configuración de renderización define el modo de renderización, los recursos de JavaScript y CSS que componen tu interfaz personalizada y los datos de contexto disponibles para tu aplicación en el runtime.

Puedes administrar la configuración de renderización a través del Auth0 Dashboard, herramientas de infrastructure-as-code o la Auth0 Management API. Para saber cómo hacerlo, consulta [Configurar ACUL](/docs/es/customize/login-pages/advanced-customizations/configure/overview). Para ver el esquema completo de la API, revisa el endpoint [`PATCH /v2/prompts/{prompt}/screen/{screen}/rendering`](/docs/es/api/management/v2/prompts/patch-rendering).

La configuración de renderización incluye las siguientes propiedades:

| **Propiedad**                | **Tipo** | **Descripción**                                                                                                                                                         |
| ---------------------------- | -------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `rendering_mode`             | string   | Controla cómo se renderiza la pantalla. Acepta `standard` o `advanced`.                                                                                                 |
| `head_tags`                  | array    | Un arreglo de elementos HTML insertados en el `<head>` de la página. Es obligatorio cuando `rendering_mode` es `advanced`; debe incluir al menos una etiqueta `script`. |
| `default_head_tags_disabled` | boolean  | Cuando es `true`, elimina las head tags predeterminadas de Universal Login (como el favicon). El valor predeterminado es `false`.                                       |
| `context_configuration`      | array    | Una lista de valores de contexto para enviar al navegador en el runtime. Es obligatoria cuando `rendering_mode` es `advanced`.                                          |
| `use_page_template`          | boolean  | Cuando es `true`, renderiza la pantalla de ACUL dentro de la page template personalizada del tenant, si hay una definida. El valor predeterminado es `false`.           |
| `filters`                    | object   | Restringe la renderización de ACUL a aplicaciones, organizaciones o dominios personalizados específicos. Cuando se omite, ACUL se aplica a todo el tenant.              |

<div id="rendering-modes">
  ## Modos de renderización
</div>

La propiedad `rendering_mode` determina si una pantalla usa la UI predeterminada de Universal Login o tu implementación personalizada de ACUL.

* **Standard**: La pantalla se renderiza con la UI predeterminada de [Universal Login](/docs/es/customize/login-pages/universal-login). Puedes usar [partials](/docs/es/customize/login-pages/universal-login/customize-signup-and-login-prompts) y [template variables](/docs/es/customize/login-pages/universal-login/customize-templates#page-template-variables) para personalizar la pantalla.
* **Advanced**: La pantalla se renderiza con tus recursos personalizados de JavaScript y CSS cargados mediante `head_tags`. Los partials no se aplican. De forma predeterminada, el modo avanzado se aplica a todas las aplicaciones y organizaciones del tenant.
* **Advanced (filtered)**: La pantalla se renderiza con ACUL para las solicitudes que coinciden con los criterios definidos en `filters` (aplicaciones, organizaciones o dominios personalizados específicos). Las solicitudes que no coinciden con los filtros vuelven a la renderización estándar, donde sí se aplican los partials.

Para usar el modo filtrado, establece `rendering_mode` en `advanced` y proporciona un objeto `filters` con un `match_type` (`includes_any` o `excludes_any`) y al menos un arreglo de entidades (`clients`, `organizations` o `domains`). Cada entidad puede identificarse por `id` o mediante un par clave/valor de `metadata`.

<div id="head-tags">
  ## Head tags
</div>

Head tags son el mecanismo principal para cargar tu interfaz personalizada en la página host de ACUL. Cuando Auth0 renderiza una pantalla en modo avanzado, entrega una página HTML mínima e inyecta en el `<head>` de la página los elementos definidos en `head_tags`. Estas etiquetas cargan los paquetes de recursos de JavaScript y CSS que conforman tu pantalla de autenticación personalizada.

Cada entrada del arreglo `head_tags` es un objeto con las siguientes propiedades:

| **Propiedad** | **Tipo** | **Descripción**                                                                                                        |
| ------------- | -------- | ---------------------------------------------------------------------------------------------------------------------- |
| `tag`         | string   | El tipo de elemento HTML. Valores válidos: `script`, `link`, `style`, `meta`, `title`, `base`, `noscript`, `template`. |
| `attributes`  | object   | Atributos del elemento HTML (por ejemplo, `src`, `href`, `defer`, `integrity`). Máximo de 10 atributos por etiqueta.   |
| `content`     | string   | Texto o marcado entre las etiquetas de apertura y cierre del elemento. Máximo de 250 caracteres.                       |

El atributo `integrity` es obligatorio para las etiquetas `script` y las hojas de estilo. Auth0 usa [Subresource Integrity (SRI)](https://developer.mozilla.org/en-US/docs/Web/Security/Subresource_Integrity) para verificar que los recursos no hayan sido modificados.

<div id="dynamic-url-segments">
  ### Segmentos dinámicos de URL
</div>

Las URL de la etiqueta `<head>` admiten segmentos dinámicos mediante la sintaxis de plantillas Liquid, lo que te permite servir distintos paquetes de recursos según el contexto de autenticación actual. Por ejemplo, puedes cargar un paquete de JavaScript distinto por cliente o por organización para habilitar el multi-branding.

Las variables disponibles para los segmentos dinámicos incluyen:

* **Prompt y screen**: `{{ screen.name }}`, `{{ prompt.name }}`, `{{ locale }}`
* **Cliente**: `{{ client.id }}`, `{{ client.name }}`, `{{ client.metadata.KEY_NAME }}`
* **Organización**: `{{ organization.id }}`, `{{ organization.name }}`, `{{ organization.metadata.KEY_NAME }}`

Al usar segmentos dinámicos de URL, proporciona un hash SRI para cada posible variante de recurso. El atributo `integrity` acepta varios hashes separados por espacios en blanco; el navegador carga el recurso si alguno de los hashes coincide.

```json
{
  "head_tags": [
    {
      "tag": "script",
      "attributes": {
        "src": "https://cdn.example.com/assets/{{ client.name }}/bundle.js",
        "defer": true,
        "integrity": "sha256-hashForClientA sha256-hashForClientB"
      }
    },
    {
      "tag": "link",
      "attributes": {
        "rel": "stylesheet",
        "href": "https://cdn.example.com/assets/{{ client.name }}/styles.css",
        "integrity": "sha256-cssHashForClientA sha256-cssHashForClientB"
      }
    }
  ]
}
```

También puedes usar head tags para cargar scripts de analítica, fuentes personalizadas o meta tags.

Para reemplazar los head tags predeterminados de Universal Login (como el favicon) por los tuyos, establece `default_head_tags_disabled` en `true`.

<div id="context-data">
  ## Datos de contexto
</div>

Los datos de contexto determinan qué información de tenant, cliente, organización y branding envía Auth0 a tu pantalla personalizada en runtime. Auth0 entrega estos datos a través del objeto `universal_login_context` en la página host. En el código de tu aplicación, usa los hooks de [ACUL React SDK](/docs/es/libraries/acul/react-sdk) o los métodos de [ACUL JS SDK](/docs/es/libraries/acul/js-sdk) para acceder a los datos de contexto.

Cada valor que quieras tener disponible en tu pantalla debe incluirse explícitamente en el arreglo `context_configuration`. Auth0 elimina del payload las claves que se resuelven en valores vacíos o nulos.

Los siguientes valores de contexto estáticos están disponibles:

| **Valor**                                         | **Descripción**                                                |
| ------------------------------------------------- | -------------------------------------------------------------- |
| `branding.settings`                               | Configuración de branding del tenant (colores, logotipo)       |
| `branding.themes.default`                         | Tema predeterminado de Universal Login                         |
| `client.logo_uri`                                 | URL del logotipo de la aplicación                              |
| `client.description`                              | Descripción de la aplicación                                   |
| `organization.display_name`                       | Nombre para mostrar de la organización                         |
| `organization.branding`                           | Configuración de branding de la organización                   |
| `screen.texts`                                    | Cadenas de texto localizadas para la pantalla actual           |
| `tenant.name`                                     | Nombre del tenant                                              |
| `tenant.friendly_name`                            | Nombre descriptivo del tenant                                  |
| `tenant.logo_url`                                 | URL del logotipo del tenant                                    |
| `tenant.enabled_locales`                          | Locales habilitados en el tenant                               |
| `untrusted_data.submitted_form_data`              | Datos del formulario enviados previamente                      |
| `untrusted_data.authorization_params.login_hint`  | Sugerencia de inicio de sesión de la solicitud de autorización |
| `untrusted_data.authorization_params.screen_hint` | Sugerencia de pantalla de la solicitud de autorización         |
| `untrusted_data.authorization_params.ui_locales`  | Locales de la UI solicitados                                   |
| `user.organizations`                              | Organizaciones a las que pertenece el usuario                  |
| `transaction.custom_domain.domain`                | Dominio personalizado para la transaction actual               |

También puedes agregar claves dinámicas usando notación de puntos:

* `client.metadata.YOUR_KEY` para metadata de la aplicación
* `organization.metadata.YOUR_KEY` para metadata de la organización
* `transaction.custom_domain.domain_metadata.YOUR_KEY` para metadata del dominio personalizado
* `untrusted_data.authorization_params.ext-YOUR_KEY` para parámetros de consulta personalizados pasados al endpoint `/authorize` con el prefijo `ext-`
