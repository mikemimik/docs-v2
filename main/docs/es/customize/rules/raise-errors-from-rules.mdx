La página de inicio de sesión de Auth0 no puede mostrar un mensaje de error de una Rule. Si quieres que los usuarios vean un mensaje personalizado generado desde una Rule, tu aplicación debe mostrar ese mensaje.

Cualquier error que se produzca en una Rule se envía a tu URL de callback en el modo de respuesta estándar del protocolo. Por ejemplo, si tu aplicación solicita autenticación mediante OIDC/OAuth2, recibe un error estándar de OAuth2 con los valores de resultado `error` y `error_description`.

Supongamos que tu URL de callback es https://travel0.com/callback y que estás usando el modo de respuesta estándar por consulta. Recibirías esta respuesta:

```http wrap lines
https://travel0.com/callback?error=access_denied&error_description=[The message you wrote in the rule]
```

Tu aplicación es la encargada de mostrar el mensaje al usuario, por ejemplo:

<div id="retry-login-after-a-rule-error">
  ## Reintentar el inicio de sesión después de un error en una Rule
</div>

Un siguiente paso probable es solicitar una nueva autorización. Sin embargo, en la mayoría de los casos, Auth0 ya sabe quién es el usuario y, por lo tanto, no vuelve a pedir la información de inicio de sesión. En su lugar, las Rules se ejecutan de inmediato y la Rule vuelve a denegar la autorización. Esto puede convertirse en un bucle infinito.

El usuario necesita tener la oportunidad de proporcionar credenciales nuevas. Hay dos formas de hacer que tu aplicación vuelva a mostrar la página de inicio de sesión:

* Agrega una directiva en la solicitud para forzar que se muestre la página de inicio de sesión:

  * Usa un parámetro `prompt=login` en una solicitud OIDC/OAuth2.
  * Usa un atributo `ForceAuthn="true"` en una solicitud SAML.

—or—

* Cierra la sesión del usuario en Auth0 antes de enviar una nueva solicitud.
