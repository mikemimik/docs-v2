La página de inicio de sesión de Auth0 no puede mostrar un mensaje de error de una rule. Si quieres que los usuarios vean un mensaje personalizado generado desde una rule, tu aplicación debe mostrar ese mensaje.

Cualquier error que resulte de una rule se envía a tu URL de devolución de llamada en el modo de respuesta estándar del protocolo. Por ejemplo, si tu aplicación solicita autenticación mediante OIDC/OAuth2, recibe un error estándar de OAuth2 con los valores de resultado `error` y `error_description`.

Supongamos que tu URL de devolución de llamada es https://travel0.com/callback y que estás usando el modo de respuesta de consulta estándar. Recibirías esta respuesta:

```http wrap lines
https://travel0.com/callback?error=access_denied&error_description=[The message you wrote in the rule]
```

Es responsabilidad de tu aplicación mostrar el mensaje al usuario, por ejemplo:

<div id="retry-login-after-a-rule-error">
  ## Reintentar el login después de un error en una rule
</div>

Un posible siguiente paso es solicitar una nueva autorización. Sin embargo, en la mayoría de los casos, Auth0 ya sabe quién es el usuario y, por lo tanto, no le vuelve a pedir sus credenciales de login. En cambio, las Rules se ejecutan de inmediato y la rule vuelve a denegar la autorización. Esto puede convertirse en un bucle sin fin.

El usuario necesita tener la oportunidad de proporcionar nuevas credenciales. Hay dos formas de hacer que tu aplicación vuelva a mostrar la página de login:

* Agrega una directiva en la solicitud para forzar que se muestre la página de login:

  * Usa un parámetro `prompt=login` en una solicitud OIDC/OAuth2.
  * Usa un atributo `ForceAuthn="true"` en una solicitud SAML.

—o bien—

* Cierra la sesión del usuario en Auth0 antes de enviar una nueva solicitud.
