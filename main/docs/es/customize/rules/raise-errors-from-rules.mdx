La página de inicio de sesión de Auth0 no puede mostrar un mensaje de error de una rule. Si quieres que los usuarios vean un mensaje personalizado generado desde una rule, tu aplicación debe mostrar ese mensaje.

Cualquier error que se produzca en una rule se envía a tu URL de devolución de llamada en el modo de respuesta estándar del protocolo. Por ejemplo, si tu aplicación solicita autenticación mediante OIDC/OAuth2, recibe un error estándar de OAuth2 con los valores `error` y `error_description`.

Supón que tu URL de devolución de llamada es https://travel0.com/callback y que estás usando el modo de respuesta estándar mediante consulta. Recibirías esta respuesta:

```http wrap lines
https://travel0.com/callback?error=access_denied&error_description=[The message you wrote in the rule]
```

Tu aplicación es la encargada de mostrarle el mensaje al usuario, por ejemplo:

<div id="retry-login-after-a-rule-error">
  ## Reintentar el inicio de sesión después de un error de rule
</div>

Un paso probable a continuación es solicitar una nueva autorización. Sin embargo, en la mayoría de los casos, Auth0 ya sabe quién es el usuario, así que no le vuelve a pedir la información de inicio de sesión. En su lugar, las reglas se ejecutan de inmediato y la rule vuelve a denegar la autorización. Esto puede convertirse en un bucle infinito.

El usuario necesita tener la oportunidad de proporcionar nuevas credenciales. Hay dos formas en que tu aplicación puede hacer que la página de inicio de sesión vuelva a mostrarse:

* Agrega una directiva en la solicitud para forzar que se muestre la página de inicio de sesión:

  * Usa un parámetro `prompt=login` en una solicitud OIDC/OAuth2.
  * Usa un atributo `ForceAuthn="true"` en una solicitud SAML.

—or—

* Cierra la sesión del usuario en Auth0 antes de enviar una nueva solicitud.
