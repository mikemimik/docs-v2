Puedes usar [Reglas de Auth0](/docs/es/customize/rules) para redirigir a los usuarios antes de que se complete una transacción de autenticación. Esto te permite implementar flujos de autenticación personalizados que requieren interacción adicional del usuario además del formulario de inicio de sesión estándar. Las reglas de redirección se usan habitualmente para realizar <Tooltip tip="Autenticación multifactor (MFA): Proceso de autenticación de usuario que utiliza un factor además del nombre de usuario y la contraseña, como un código por SMS." cta="Ver glosario" href="/docs/es/glossary?term=Multi-factor+Authentication">Autenticación multifactor</Tooltip> (MFA) personalizada en Auth0, pero también se pueden usar para:

* La aceptación personalizada de la política de privacidad, los términos de servicio y los formularios de divulgación de datos.
* Recopilar de forma segura, una sola vez, datos de perfil adicionales obligatorios.
* Permitir que los usuarios remotos de Active Directory cambien su contraseña.
* Exigir que los usuarios proporcionen una verificación adicional al iniciar sesión desde ubicaciones desconocidas.
* Recopilar más información sobre tus usuarios de la que proporcionaron durante el registro inicial.

Puedes redirigir a un usuario **una vez** por flujo de autenticación. Si tienes una regla que redirige a un usuario, **no puedes** invocar una segunda regla para redirigirlo más adelante.

Para obtener más información, consulta [Autenticación multifactor en Auth0](/docs/es/secure/multi-factor-authentication).

<div id="start-redirect-and-resume-authentication">
  ## Redirigir desde Start y reanudar la autenticación
</div>

Configura la propiedad `context.redirect` de la siguiente manera:

```javascript lines
function (user, context, callback) {
  context.redirect = {
    url: "https://example.com/foo"
  };
  return callback(null, user, context);
}
```

Una vez que todas las reglas hayan terminado de ejecutarse, Auth0 redirige al usuario a la URL especificada en la propiedad `context.redirect.url`. Auth0 también incluye un parámetro `state` en esa URL. Por ejemplo:

```http lines
https://example.com/foo?state=abc123
```

Tu URL de redirección debe extraer el parámetro `state` y enviarlo de vuelta a Auth0 para reanudar la transacción de autenticación. `state` es un valor opaco que se usa para prevenir [ataques de falsificación de solicitud entre sitio (CSRF)](/docs/es/secure/security-guidance/prevent-threats#cross-site-request-forgery).

Después de la redirección, reanuda la autenticación redirigiendo al usuario al endpoint `/continue` e incluyendo en la URL el parámetro `state` que recibiste. Si no devuelves el valor original de `state` al endpoint `/continue`, Auth0 perderá el contexto de la transacción de inicio de sesión y el usuario no podrá iniciar sesión debido a un error `invalid_request`.

Por ejemplo:

export const codeExample1 = `https://{yourDomain}/continue?state={originalState}`;

<AuthCodeBlock children={codeExample1} language="http" />

Si usas un <Tooltip tip="Dominio personalizado: dominio de terceros con un nombre especializado o personalizado." cta="Ver glosario" href="/docs/es/glossary?term=custom+domain">dominio personalizado</Tooltip>:

```http lines
https://{yourAuth0CustomDomain}/continue?state={originalState}
```

`THE_ORIGINAL_STATE` es el valor que Auth0 generó y envió a la URL de redirección. Por ejemplo, si tu rule redirigió a `https://example.com/foo`, Auth0 usaría una URL de redirección similar a `https://example.com/foo?state=abc123`. Así, `abc123` sería `THE_ORIGINAL_STATE`. Para reanudar la transacción de autenticación, deberías redirigir a:

export const codeExample2 = `https://{yourDomain}/continue?state=abc123`;

<AuthCodeBlock children={codeExample2} language="http" />

Cuando se haya redirigido a un usuario al endpoint `/continue`:

* **todas las reglas se volverán a ejecutar**, pero `context.redirect` se ignorará para permitir que la autenticación continúe.
* cualquier cambio en el objeto de usuario se realiza durante la redirección, antes de llamar al endpoint `/continue`. Por ejemplo, las actualizaciones realizadas a través de la Auth0 <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip> estarán disponibles después de continuar la transacción.

<div id="validate-resumed-login">
  ## Validar el inicio de sesión reanudado
</div>

Para distinguir entre los inicios de sesión iniciados por el usuario y los flujos de inicio de sesión reanudados, verifique la propiedad `context.protocol`:

```javascript lines
function (user, context, callback) {
    if (context.protocol === "redirect-callback") {
        // El usuario fue redirigido al endpoint /continue
    } else {
        // El usuario está iniciando sesión directamente
    }
}
```

<div id="force-password-change-example">
  ## Ejemplo para forzar el cambio de contraseña
</div>

En algunos casos, puede que quieras obligar a los usuarios a cambiar sus contraseñas en determinadas circunstancias. Puedes escribir una rule con el siguiente comportamiento:

1. El usuario intenta iniciar sesión y necesita cambiar su contraseña.
2. Se redirige al usuario a una página específica de la aplicación con un JWT en la cadena de consulta. Este JWT garantiza que solo pueda cambiarse la contraseña de este usuario y **debe ser validado** por la aplicación.
3. El usuario cambia su contraseña en la página específica de la aplicación haciendo que la aplicación llame a la [Auth0 Management API](https://auth0.com/docs/api/management/v2/users/patch-users-by-id)
4. Una vez que el usuario haya cambiado correctamente su contraseña, la aplicación extrae el claim `authorize_again` del JWT verificado y decodificado, y luego lo redirige a esa URL para que pueda iniciar sesión con su nueva contraseña.

```javascript lines expandable
function(user, context, callback) {
   /*
   * Requisitos previos:
   * 1. Implementar una función `mustChangePassword`
   * 2. Establecer variables de configuración para lo siguiente:
   *    - CLIENT_ID
   *    - CLIENT_SECRET
   *    - ISSUER
   */

  const url = require('url@0.10.3');
  const req = context.request;

  function mustChangePassword() {
    // TODO: implementar la función
    return true;
  }

  if (mustChangePassword()) {
    // El usuario ha iniciado sesión y se le obliga a cambiar su contraseña
    // Enviar la información del usuario y los parámetros de consulta en un JWT para evitar manipulaciones
    function createToken(clientId, clientSecret, issuer, user) {
      const options = {
        expiresInMinutes: 5,
        audience: clientId,
        issuer: issuer
      };
      return jwt.sign(user, clientSecret, options);
    }

    const token = createToken(
      configuration.CLIENT_ID,
      configuration.CLIENT_SECRET,
      configuration.ISSUER,
      {
        sub: user.user_id,
        email: user.email,
        authorize_again: url.format({
          protocol: 'https',
          hostname: auth0.com,
          pathname: '/authorize',
          query: req.query
        })
      }
    );

    context.redirect = {
      url: `https://example.com/change-pw?token=${token}`
    };
  }

  return callback(null, user, context);
}
```

<div id="where-to-store-data">
  ## Dónde almacenar datos
</div>

Evita almacenar demasiados datos en el perfil de Auth0. Estos datos están pensados para fines de autenticación y autorización. Los metadatos y las capacidades de búsqueda de Auth0 no están diseñados para investigación de marketing ni para ningún otro caso que requiera búsquedas intensivas o una alta frecuencia de actualizaciones. Si usas Auth0 con ese fin, es probable que tu sistema tenga problemas de escalabilidad y rendimiento. Una mejor opción es almacenar los datos en un sistema externo y guardar una referencia (el ID de usuario) en Auth0 para que los sistemas backend puedan recuperar los datos si es necesario. Una regla sencilla es almacenar solo los elementos que piensas usar en reglas para añadirlos a tokens o tomar decisiones.

<div id="security-considerations">
  ## Consideraciones de seguridad
</div>

Pasar información de un lado a otro por el front channel amplía la superficie de ataque disponible para que <Tooltip tip="Actores malintencionados: entidad (una persona o grupo) que representa una amenaza para el negocio o el entorno con la intención de causar daño." cta="Ver glosario" href="/docs/es/glossary?term=bad+actors">actores malintencionados</Tooltip> ataquen. Sin duda, esto solo debe hacerse en situaciones en las que debas tomar medidas en la rule (como rechazar el intento de autorización con `UnauthorizedError`).

Sin embargo, si necesitas comunicarte directamente con Auth0 y darle instrucciones para restringir el acceso (si estás implementando comprobaciones de CAPTCHA o MFA personalizados), debes contar con una forma segura de indicarle a Auth0 que se cumplieron los requisitos de esa operación. Del mismo modo, si necesitas pasar información a la aplicación a la que estás redirigiendo, también debes contar con una forma segura de garantizar que la información transferida no haya sido manipulada.

<div id="ensure-app-is-logging-into-the-same-user">
  ### Asegúrese de que la aplicación inicie sesión con el mismo usuario
</div>

La aplicación va a redirigir al usuario de vuelta al tenant de Auth0, por lo que cualquier dato relacionado con el usuario puede recopilarse a través del <Tooltip tip="token de ID: Credencial destinada al propio cliente, en lugar de para acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=ID+token">token de identidad</Tooltip> que se devuelve a la aplicación. Sin embargo, quizá quiera asegurarse de que la aplicación esté iniciando sesión con el mismo usuario que está siendo redirigido, para garantizar que no haya ningún tipo de manipulación durante el proceso. Por lo tanto, probablemente querrá enviar un token junto con la solicitud.

El token enviado a la aplicación debe cumplir los siguientes requisitos:

| Elemento del token | Descripción                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                   |
| ------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `sub`              | El `user_id` de Auth0 del usuario.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                            |
| `iss`              | Un identificador que identifica a la propia regla.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                            |
| `aud`              | La aplicación a la que va dirigida la redirección.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                            |
| `jti`              | Una cadena generada aleatoriamente que se almacena para su confirmación en el user object (en el código de la regla, establezca user.jti = uuid.v4(); y luego agréguelo como `jti` al token que cree). user.jti seguirá establecido cuando las reglas se vuelvan a ejecutar al llamar a /continue. Esto está en línea con las especificaciones.                                                                                                                                                                                                                                                                                                                                                                   |
| `exp`              | Debe ser lo más breve posible para evitar la reutilización del token.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                         |
| `other`            | Cualquier otra información de custom claims que necesite pasar.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                               |
| `signature`        | Suponiendo que la aplicación tenga un lugar seguro para almacenar un secreto, puede usar firmas con HS256. Esto reduce en gran medida la complejidad de la solución y, dado que el token que se devuelve también tendrá que estar firmado, es un requisito de esta solución. Puede usar RS256, pero requiere crear un certificado y actualizarlo cuando expire. Si no está pasando ninguna información directamente de vuelta a las reglas, podría usar una SPA para esta aplicación intermedia y quizá prefiera RS256 para que la aplicación no tenga que almacenar la información. Esto requeriría una forma de validar el token, ya sea a través de un endpoint de introspección o de un endpoint JWKS público. |

<Warning>
  ¡Este token **no** debe tratarse como un token Bearer! Es una pieza de información firmada para usarla en la aplicación. La aplicación debe seguir redirigiendo de vuelta a Auth0 para autenticar al usuario.
</Warning>

<div id="pass-information-back-to-the-rule">
  ### Pasar información de vuelta a la rule
</div>

En la mayoría de los casos, aunque quiera pasar información de la rule a la aplicación, lo ideal es que la aplicación pueda almacenar esa información de forma segura en el almacenamiento que corresponda. Incluso si la idea es actualizar los metadatos de la aplicación o del usuario en Auth0, eso puede hacerse mediante la Management API y la información del usuario se actualizará siempre que el proceso se complete antes de redirigir al usuario de vuelta al endpoint `/continue`. Solo debería pasar información de vuelta a la rule si la propia rule necesita obtener esa información y si esta solo es relevante para esta sesión de inicio de sesión en particular.

Al pasar información de vuelta al endpoint `/continue`, el token que se envíe debe cumplir los siguientes requisitos:

| Elemento del token | Descripción                                                                                                                                                                                                                                                                                                                                                         |
| ------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `sub`              | El `user_id` del usuario en Auth0.                                                                                                                                                                                                                                                                                                                                  |
| `iss`              | La aplicación a la que va dirigida la redirección.                                                                                                                                                                                                                                                                                                                  |
| `aud`              | Algún identificador que identifique a la propia rule.                                                                                                                                                                                                                                                                                                               |
| `jti`              | El mismo JTI que se almacenó en el token enviado a la aplicación (NOTA: debe coincidir con user.jti o fallará).                                                                                                                                                                                                                                                     |
| `exp`              | Debe ser lo más corto posible para evitar la reutilización del token.                                                                                                                                                                                                                                                                                               |
| `other`            | Cualquier otra información de custom claims que necesite pasar.                                                                                                                                                                                                                                                                                                     |
| `signature`        | Suponiendo que la aplicación tenga un lugar seguro donde almacenar un secreto, puede usar firmas con HS256. Esto reduce enormemente la complejidad de la solución y, dado que el token que se devuelve también tendrá que estar firmado, este es un requisito de esta solución. Puede usar RS256, pero requiere crear un certificado y actualizarlo cuando caduque. |

Debe enviarse mediante POST y luego recuperarse en `context.request.body.token` (o algo similar), en lugar de pasarlo como parámetro de consulta. Esto es similar al método form-post para la autenticación.

Si no está pasando información de vuelta al endpoint `/continue`, quizá le convenga incluir el JTI en una lista de bloqueo, a menos que los tiempos de expiración sean lo bastante cortos como para que los ataques de repetición sean casi imposibles.

<div id="restrictions-and-limitations">
  ## Restricciones y limitaciones
</div>

Las Redirect Rules no funcionan con:

* [endpoint del propietario del recurso](https://auth0.com/docs/api/authentication/reference#resource-owner)
* [intercambio de contraseña](/docs/es/get-started/authentication-and-authorization-flow/resource-owner-password-flow)
* [intercambio de token de actualización](/docs/es/secure/tokens/refresh-tokens)

Puedes detectar los casos anteriores comprobando `context.protocol`:

* Para el intercambio de contraseña: `context.protocol === 'oauth2-password'`
* Para el intercambio de <Tooltip tip="Refresh Token: Token utilizado para obtener un token de acceso renovado sin obligar a los usuarios a volver a iniciar sesión." cta="Ver glosario" href="/docs/es/glossary?term=Refresh+Token">token de actualización</Tooltip>: `context.protocol === 'oauth2-refresh-token'`
* Para los inicios de sesión con <Tooltip tip="Resource Owner: Entidad (como un usuario o una aplicación) capaz de otorgar acceso a un recurso protegido." cta="Ver glosario" href="/docs/es/glossary?term=Resource+Owner">Resource Owner</Tooltip>: `context.protocol === 'oauth2-resource-owner'`

<div id="session-timeout">
  ### Tiempo de espera de la sesión
</div>

Las sesiones de `redirect rule` suelen ser válidas durante 3 días, a menos que hayas configurado un tiempo de espera menor en la configuración de **Login Session Management**. Puedes encontrar esta configuración en la [configuración avanzada de tu tenant](https://manage.auth0.com/#/tenant/advanced).

<div id="resource-owner-endpoint">
  ### endpoint del propietario del recurso
</div>

No es posible usar reglas de redirección en el contexto en el que llamas directamente a `/oauth/token` para la concesión de contraseña del propietario del recurso. Como el usuario no forma parte de un flujo de redirección desde el principio, no puedes redirigirlo en una rule. Si intentas establecer `context.redirect`, obtendrás un intento fallido de inicio de sesión con el error `interaction_required`.

<div id="flows-where-promptnone">
  ### Flujos donde prompt=none
</div>

Dado que el objetivo de `prompt=none` es evitar cualquier situación en la que el usuario tenga que introducir datos, cualquier redirección dará lugar a un `error=interaction_required`.

Como las reglas se ejecutan después de que se crea una sesión de autenticación, no puedes usar `prompt=none` si tienes una redirect regla que intenta bloquear el acceso a los tokens en determinadas condiciones (MFA personalizada, CAPTCHA en el login, etc.).

No puedes crear un flujo de redirección que bloquee el acceso a los tokens y omita la redirect regla con `prompt=none`, porque, después de un intento fallido, un usuario puede simplemente volver a llamar con `prompt=none` y obtener tokens, ya que su sesión de autenticación ya se habrá creado aunque las reglas hayan fallado la primera vez.

<div id="refresh-tokens">
  ### Tokens de actualización
</div>

Dado que usar un `token de actualización` requiere una llamada por canal secundario a `/oauth/token`, esto también fallará si estableces `context.redirect`.

Es difícil verificar de forma segura que se hayan aplicado las restricciones de login. No hay un ID de sesión coherente en el context que pueda usarse para recopilar información asociada con la sesión, como si este usuario superó desafíos MFA. Por lo tanto, no puedes usar `prompt=none` en absoluto.

Siempre que se establezca `context.redirect` en una rule, si se pasó `prompt=none`, la autorización falla con `error=interaction_required`, pero como la sesión del usuario se crea incluso si las reglas fallan, no podemos confiar en que un usuario haya superado todos los desafíos de `context.redirect` y, por lo tanto, no podemos usar `prompt=none` como forma de obtener tokens.

En este caso concreto, te recomendamos usar exclusivamente tokens de actualización, porque así puedes asegurarte de que el usuario superó los desafíos si esos desafíos son necesarios para generar un token de actualización.
