  Para obtener más información sobre la obsolescencia de Reglas y Hooks, consulta nuestra entrada del blog: [Preparing for Rules and Hooks End of Life](https://auth0.com/blog/preparing-for-rules-and-hooks-end-of-life/).

Puedes crear tu propia rule o reglas para cubrir tus necesidades específicas de funcionalidad. Puedes modificar una plantilla de rule existente o empezar desde cero con uno de nuestros ejemplos. Auth0 proporciona varias reglas y plantillas de rule predefinidas para ayudarte a alcanzar tus objetivos. Para ver la lista, visita nuestro [repositorio de reglas en GitHub](https://github.com/auth0/rules).

<div id="how-rules-work">
  ## Cómo funcionan las Reglas
</div>

Las Reglas son funciones de JavaScript que se ejecutan cuando un usuario se autentica en tu aplicación. Se ejecutan una vez completado el proceso de autenticación, y puedes usarlas para personalizar y ampliar las capacidades de Auth0. Por motivos de seguridad, el código de tus Reglas se ejecuta de forma aislada del código de otros tenants de Auth0 en un sandbox. Las Reglas también se ejecutan durante el flujo de actualización de tokens. Para obtener más información, consulta [Tokens de actualización](/docs/es/secure/tokens/refresh-tokens).

En Auth0, el flujo de la transacción de autenticación funciona de la siguiente manera cuando usas Reglas:

<Frame>![Diagrama de las Reglas en el flujo de autenticación](/docs/images/cdy7uua7fh8z/2gtBtkPChIyguA24x6enx2/ffbb8e21e86920ef9914f6fc126dc1df/flow.png)</Frame>

1. Una aplicación inicia una solicitud de autenticación a Auth0.
2. Auth0 dirige la solicitud a un proveedor de identidad mediante una conexión configurada.
3. El usuario se autentica correctamente.
4. El token de identidad y/o el token de acceso pasa por el pipeline de reglas y luego se envía a la aplicación.

<div id="use-the-management-api">
  ## Usa la Management API
</div>

Haz una llamada `POST` al [endpoint Create Rule](https://auth0.com/docs/api/management/v2#!/Rules/post_rules). Asegúrate de reemplazar los valores de marcador de posición `MGMT_API_ACCESS_TOKEN`, `RULE_NAME`, `RULE_SCRIPT`, `RULE_ORDER` y `RULE_ENABLED` por tu <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip> <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Access+Token">token de acceso</Tooltip>, el nombre de la rule, el script de la rule, el número de orden de la rule y el valor que indica si la rule está habilitada, respectivamente.

| Valor                     | Descripción                                                                                                                                                                                                                                              |
| ------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `MGMT_API_ACCESS_TOKEN`   | Token de acceso para la Management API con el `scope` `create:rules`.                                                                                                                                                                                    |
| `RULE_NAME`               | Nombre de la rule que desea crear. El nombre de la rule solo puede contener caracteres alfanuméricos, espacios y guiones; no puede comenzar ni terminar con espacios ni guiones.                                                                         |
| `RULE_SCRIPT`             | Script que contiene el código de la rule. Debe coincidir con lo que introduciría si estuviera creando una nueva rule en el Dashboard.                                                                                                                    |
| `RULE_ORDER` (opcional)   | Entero que representa el orden en que debe ejecutarse la rule con respecto a otras reglas. Las Reglas con números más bajos se ejecutan antes que las Reglas con números más altos. Si no se proporciona un número de orden, la rule se ejecutará al final. |
| `RULE_ENABLED` (opcional) | Boolean que representa si la rule está habilitada (`true`) o deshabilitada (`false`).                                                                                                                                                                    |

<Warning>
  Exponemos direcciones IPv6 en nuestros endpoints públicos (p. ej., `travel0.us.auth0.com`). Si una request llega desde una máquina compatible con IPv6, la propiedad `context.request.ip` contendrá una dirección IPv6. Si manipula direcciones IP manualmente, le sugerimos usar la [biblioteca ipaddr.js@1.9.0](https://www.npmjs.com/package/ipaddr.js/v/1.9.0).
</Warning>

<div id="manage-rate-limits">
  ## Gestionar los límites de frecuencia
</div>

En las reglas que llaman a las API de Auth0, siempre debe gestionar los límites de frecuencia comprobando la cabecera `X-RateLimit-Remaining` y actuando adecuadamente cuando el número devuelto se acerque a 0. También debe añadir lógica para gestionar los casos en que supere los límites de frecuencia establecidos y reciba el código de estado HTTP 429 (Too Many Requests); en ese caso, si es necesario reintentar, lo mejor es aplicar un tiempo de espera progresivo para evitar entrar en un bucle infinito de reintentos. Para obtener más información sobre los límites de frecuencia, consulte la [Política de límite de tasa para las API de Auth0](/docs/es/troubleshoot/customer-support/operational-policies/rate-limit-policy).

<div id="available-modules">
  ## Módulos disponibles
</div>

Rules se ejecuta en un sandbox configurado para una versión específica de Node.js.

El sandbox admite todas las versiones del lenguaje JavaScript (y su sintaxis asociada) para la versión configurada de Node.js, así como una gran cantidad de módulos de Node.js. Para consultar una lista de los módulos compatibles con el sandbox, visita [Can I require: Auth0 Extensibility](https://auth0-extensions.github.io/canirequire/).
