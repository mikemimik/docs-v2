description: Aprende a usar Auth0 para proteger el acceso a los proxies de API desarrollados y gestionados por Apigee Edge.
title: Protege Apigee con Auth0
---

Si usas Apigee Edge para desarrollar y gestionar las API de tu backend, puedes usar Auth0 para proteger el acceso a tus proxies de API.

<div id="prerequisites">
  ## Requisitos previos
</div>

Antes de comenzar, necesitarás:

1. Tener un proxy de API de Apigee Edge que deba protegerse. Para obtener más información, consulta el artículo [Get Started](https://docs.apigee.com/api-platform/get-started/get-started) de Apigee.
2. [Regístrate](https://auth0.com/signup) para crear una cuenta en Auth0.

El proceso de crear tu proxy de API queda fuera del alcance de este artículo. En su lugar, nos centraremos en proteger con Auth0 un proxy de API que ya tienes.

<div id="create-a-custom-api">
  ## Crear una API personalizada
</div>

Primero, [registre su proxy de API de Apigee Edge en el Dashboard](/docs/es/get-started/auth0-overview/set-up-apis). Auth0 necesita reconocer a Apigee como una <Tooltip tip="Audiencia: identificador único de la audiencia de un token emitido. Se denomina aud en un token; su valor contiene el ID de una aplicación (ID de cliente) para un token de ID o de una API (identificador de API) para un token de acceso." cta="Ver glosario" href="/docs/es/glossary?term=audience">audience</Tooltip> para garantizar que todos los <Tooltip tip="Audiencia: identificador único de la audiencia de un token emitido. Se denomina aud en un token; su valor contiene el ID de una aplicación (ID de cliente) para un token de ID o de una API (identificador de API) para un token de acceso." cta="Ver glosario" href="/docs/es/glossary?term=Access+Tokens">tokens de acceso</Tooltip> que se emitan tengan la audiencia correcta. El usuario se autentica con Auth0 a través de la aplicación, y esta especifica ese valor de audiencia para garantizar que el token de acceso tenga los `scopes` adecuados para la audiencia proporcionada.

Necesitará hacer lo siguiente:

1. Proporcione un nombre para su API (por ejemplo, `apigee`).
2. Proporcione un identificador para su API: `urn:apigee:target:api`
3. Elija un algoritmo de firma: `RS256` (predeterminado)

Cuando registra su proxy de API de Apigee Edge, Auth0 también crea una aplicación de **máquina a máquina (M2M)** en su nombre y le asigna un nombre que coincide con la API que registró. Puede usar esta aplicación para hacer pruebas; se configura automáticamente con autorización para llamar a su API.

<div id="note-variables-from-the-test-application">
  ## Anota las variables de la aplicación de prueba
</div>

Ve a la aplicación de prueba creada al registrar tu API y anota las variables que se establecieron durante el proceso de registro de tu API y de creación de la aplicación de máquina a máquina asociada. Las necesitarás en los siguientes pasos de este tutorial.

1. Ve a [Auth0 Dashboard &gt; Applications](https://manage.auth0.com/#/applications)], y haz clic en el nombre de tu aplicación de prueba.
2. Desplázate hacia abajo y anota las siguientes variables:

   * audiencia de la API
   * dominio de Auth0
   * ID de cliente
   * Allowed callback URL(s): Las URL a las que se puede redirigir al usuario después de la autenticación. Puedes especificar varias URL separándolas por comas. (Esto suele hacerse para gestionar distintos entornos, cada uno con sus propias redirecciones).

<div id="implement-the-client-credentials-flow">
  ## Implementa el flujo Client Credentials
</div>

Ahora ya puedes implementar el Client Credentials Flow para solicitar los tokens de acceso que luego puedes proporcionar a Apigee Edge. Ten en cuenta que, en este escenario, usarás el Client Credentials Flow porque estás usando Apigee con las API del backend de tu servicio, lo que representa una aplicación de máquina a máquina (M2M); en otros escenarios, puede ser necesario usar flujos diferentes.

Para obtener información sobre cómo iniciar sesión y obtener un token de acceso que pueda usarse para llamar a Apigee Edge, consulta [Llamar a la API usando el Client Credentials Flow](/docs/es/get-started/authentication-and-authorization-flow/client-credentials-flow/call-your-api-using-the-client-credentials-flow).

<div id="validate-the-access-token">
  ## Validar el token de acceso
</div>

Una vez que hayas solicitado y recibido un token de acceso de Auth0, podrás usarlo para llamar al proxy de API que configuraste en Edge.

Cuando [uses tu token de acceso](https://docs.apigee.com/api-platform/security/oauth/using-access-tokens.html), lo primero que hará Apigee Edge será [verificar el token](https://docs.apigee.com/api-platform/security/oauth/using-access-tokens.html#addingaverifyaccesstokenpolicy).

Los tokens de acceso de Auth0 son <Tooltip tip="JSON Web Token (JWT): formato estándar de token de ID (y, con frecuencia, también de token de acceso) que se utiliza para representar claims de forma segura entre dos partes." cta="Ver glosario" href="/docs/es/glossary?term=JSON+Web+Tokens">JSON Web Tokens</Tooltip> (JWT), por lo que puedes aprovechar la policy Verify JWT de Apigee Edge para hacerlo. Para obtener más información, consulta el artículo de Apigee [Verify JWT policy signed with the RS256 algorithm](https://docs.apigee.com/api-platform/reference/policies/verify-jwt-policy#verify-a-jwt-signed-with-the-rs256-algorithm).

Apigee Edge verificará el token antes de cualquier otra cosa; si el token se rechaza, se detendrá todo el procesamiento y Edge devolverá un error al cliente.

<div id="testing">
  ## Pruebas
</div>

Para probar tu implementación, haz una solicitud HTTP POST a Apigee Edge con el token de acceso emitido por Auth0 incluido en el encabezado de la solicitud. Puedes ver cómo hacerlo en el artículo de Apigee [Calling the protected API](https://docs.apigee.com/api-platform/security/oauth/oauth-20-client-credentials-grant-type.html#callingtheprotectedapi).

Cuando recibas una respuesta exitosa, extrae el token y revísalo para verificar que incluya los claims de usuario requeridos o solicitados.
