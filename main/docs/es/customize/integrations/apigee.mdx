description: Aprende a usar Auth0 para proteger el acceso a los proxies de API desarrollados y administrados con Apigee Edge.
title: Protege Apigee con Auth0
---

Si usas Apigee Edge para desarrollar y administrar las API de tus servicios de backend, puedes usar Auth0 para proteger el acceso a tus proxies de API.

<div id="prerequisites">
  ## Requisitos previos
</div>

Antes de comenzar, necesitarás:

1. Tener un proxy de API de Apigee Edge que necesites proteger. Para obtener más información, lee el artículo [Primeros pasos](https://docs.apigee.com/api-platform/get-started/get-started) de Apigee.
2. [Regístrate](https://auth0.com/signup) para crear una cuenta en Auth0.

El proceso de crear tu proxy de API queda fuera del alcance de este artículo. En su lugar, nos centraremos en proteger con Auth0 un proxy de API que ya tienes.

<div id="create-a-custom-api">
  ## Crear una API personalizada
</div>

Primero, [registra tu proxy de API de Apigee Edge en el Dashboard](/docs/es/get-started/auth0-overview/set-up-apis). Auth0 necesita reconocer a Apigee como una <Tooltip tip="Audience: Identificador único de la audiencia de un token emitido. Denominado aud en un token, su valor contiene el ID de una aplicación (ID de cliente) para un ID Token o de una API (API Identifier) para un token de acceso." cta="Ver glosario" href="/docs/es/glossary?term=audience">audiencia</Tooltip> para asegurarse de que cualquier <Tooltip tip="Audience: Identificador único de la audiencia de un token emitido. Denominado aud en un token, su valor contiene el ID de una aplicación (ID de cliente) para un ID Token o de una API (API Identifier) para un token de acceso." cta="Ver glosario" href="/docs/es/glossary?term=Access+Tokens">token de acceso</Tooltip> que se emita tenga la audiencia correcta. El usuario se autentica con Auth0 a través de la aplicación, y la aplicación especifica este valor de audiencia para asegurarse de que el token de acceso tenga los scopes correctos para la audiencia proporcionada.

Tendrás que hacer lo siguiente:

1. Proporciona un nombre para tu API (p. ej., `apigee`).
2. Proporciona un identificador para tu API: `urn:apigee:target:api`
3. Elige un algoritmo de firma: `RS256` (predeterminado)

Cuando registras tu proxy de API de Apigee Edge, Auth0 también crea una aplicación de **Machine to Machine (M2M)** en tu nombre y le asigna un nombre que coincide con la API que registraste. Puedes usar esta aplicación para hacer pruebas; se configura automáticamente para que esté autorizada a llamar a tu API.

<div id="note-variables-from-the-test-application">
  ## Anota las variables de la aplicación de prueba
</div>

Ve a la aplicación de prueba que creaste al registrar tu API y toma nota de las variables que se configuraron durante el proceso de registro de la API y de creación de la aplicación M2M asociada. Las necesitarás en los siguientes pasos de este tutorial.

1. Ve a [Auth0 Dashboard &gt; Aplicaciones](https://manage.auth0.com/#/applications)], y haz clic en el nombre de tu aplicación de prueba.
2. Desplázate hacia abajo y toma nota de las siguientes variables:

   * Audiencia de la API
   * Dominio de Auth0
   * ID de cliente
   * URL de callback permitidas: las URL a las que se puede redirigir al usuario después de la autenticación. Puedes especificar varias URL separándolas por comas. (Esto suele hacerse para gestionar distintos entornos, cada uno con sus propias redirecciones.)

<div id="implement-the-client-credentials-flow">
  ## Implementa el flujo Client Credentials
</div>

Ahora ya estás listo para implementar el flujo Client Credentials para solicitar los tokens de acceso que puedes proporcionar a Apigee Edge. Ten en cuenta que, en este escenario, usarás el flujo Client Credentials porque estás usando Apigee con las API de tu servicio de backend, lo que corresponde a una aplicación de máquina a máquina (M2M); en otros escenarios, puede que necesites usar flujos diferentes.

Para obtener información sobre cómo iniciar sesión y obtener un token de acceso que pueda usarse para hacer llamadas a Apigee Edge, consulta [Llamar a la API con el flujo Client Credentials](/docs/es/get-started/authentication-and-authorization-flow/client-credentials-flow/call-your-api-using-the-client-credentials-flow).

<div id="validate-the-access-token">
  ## Validar el token de acceso
</div>

Una vez que hayas solicitado y recibido un token de acceso de Auth0, podrás usarlo para llamar al proxy de API que configuraste con Edge.

Cuando [uses tu token de acceso](https://docs.apigee.com/api-platform/security/oauth/using-access-tokens.html), lo primero que hará Apigee Edge será [verificar el token](https://docs.apigee.com/api-platform/security/oauth/using-access-tokens.html#addingaverifyaccesstokenpolicy).

Los tokens de acceso de Auth0 son <Tooltip tip="JSON Web Token (JWT): formato estándar de ID Token (y, a menudo, también de token de acceso) que se utiliza para representar claims de forma segura entre dos partes." cta="Ver glosario" href="/docs/es/glossary?term=JSON+Web+Tokens">JSON Web Tokens</Tooltip> (JWT), así que puedes aprovechar la política Verify JWT de Apigee Edge para hacerlo. Para obtener más información, consulta el artículo de Apigee [Verify JWT policy signed with the RS256 algorithm](https://docs.apigee.com/api-platform/reference/policies/verify-jwt-policy#verify-a-jwt-signed-with-the-rs256-algorithm).

Apigee Edge verificará el token antes que nada; si el token se rechaza, se detendrá todo el procesamiento y Edge devolverá un error al cliente.

<div id="testing">
  ## Pruebas
</div>

Para probar tu implementación, realiza una solicitud HTTP POST a Apigee Edge con el token de acceso emitido por Auth0 incluido en el encabezado de la solicitud. Puedes ver cómo hacerlo en el artículo de Apigee [Calling the protected API](https://docs.apigee.com/api-platform/security/oauth/oauth-20-client-credentials-grant-type.html#callingtheprotectedapi).

Cuando recibas una respuesta satisfactoria, extrae el token y revísalo para verificar que incluya los claims de usuario requeridos o solicitados.
