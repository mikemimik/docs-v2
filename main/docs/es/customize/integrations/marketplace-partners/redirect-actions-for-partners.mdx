title: Redirect Actions para partners
---

Cuando un usuario inicia sesión o se registra, una Redirect Action te permite redirigirlo a una página externa (como un formulario de consentimiento) y después devolverlo a Auth0 para completar el inicio de sesión o el registro. Puedes redirigir a los usuarios para que accedan a una aplicación externa que les solicite lo siguiente:

* Realizar una acción, como una verificación de identidad
* Proporcionar información, como perfilado progresivo
* Aceptar algo, como el consentimiento o los Términos de servicio

Este proceso ocurre en una Redirect Action:

1. La **aplicación del cliente** redirige al usuario a Auth0 para iniciar sesión.
2. Si el inicio de sesión se realiza correctamente, se ejecutan todas las Actions del [trigger Post Login](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger) (esto ocurre antes de MFA, si está activa).
3. Si una Action desencadena una redirección, el usuario es enviado a la URL designada junto con un parámetro state. Esta URL debe estar alojada en tu servicio o en el del cliente.
4. Luego, el usuario es redirigido o enviado mediante POST de vuelta a Auth0 en una ruta específica de su dominio, junto con el valor state original, y la Action ejecuta cualquier código que exista en `onContinuePostLogin`.
5. Por último, el usuario vuelve a la aplicación con su identidad o, si algo falló, con un mensaje de error.

Una vez que estés listo para integrar tu servicio en el proceso, hay varios elementos importantes que debes tener en cuenta:

* ¿Cómo determinas cuándo redirigir fuera de Auth0?

  * ¿Con una marca en el [app&#95;metadata del usuario](/docs/es/manage-users/user-accounts/metadata/manage-metadata-rules)?
  * ¿En función de un campo de metadata específico en el cliente?
* ¿Cómo gestionas los datos existentes del perfil del usuario que deben verificarse? (Estos datos pueden haber sido proporcionados por el usuario o provenir de una fuente de identidad federada, como Google, Facebook o Azure AD).
* ¿Qué datos necesitas de Auth0 en tu servicio y cómo los haces llegar allí de forma segura?
* ¿Cómo conservas el valor state de Auth0 en tu servicio?
* ¿Cómo obtienes y conservas la URL `/continue` a la que quieres hacer POST o redirigir?
* ¿Qué devuelves a Auth0 y cómo lo haces de forma segura?
* ¿Cómo indicas que la verificación de identidad se completó y transmites el estado?
* ¿Cómo almacenas la información necesaria en el [app&#95;metadata del usuario](/docs/es/manage-users/user-accounts/metadata/manage-metadata-rules) o en el [perfil de usuario normalizado](/docs/es/manage-users/user-accounts/user-profiles/user-profile-structure)
* Ten en cuenta los [límites de frecuencia](/docs/es/troubleshoot/customer-support/operational-policies/rate-limit-policy/management-api-endpoint-rate-limits) y actualiza solo cuando sea necesario
* ¿Cómo envías información de vuelta a la aplicación solicitante mediante [claims personalizados del token](/docs/es/get-started/apis/scopes/sample-use-cases-scopes-and-claims)?

Para responder a todas estas preguntas y más, consulta [Redirect with Actions](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger/redirect-with-actions). Cuando estés listo para enviar tu integración de Action, sigue los pasos 4 a 6 de [Action Integrations for Partners.](/docs/es/customize/integrations/marketplace-partners/actions-integrations-for-partners)
