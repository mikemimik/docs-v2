title: Actions Integrations para Partners
---

Partners como tú usan Actions Integrations para crear soluciones sin código que integran sus productos en los flujos de trabajo de Auth0. Actions Integrations permiten a los clientes de Auth0 implementar soluciones como la verificación de identidad o la gestión del consentimiento sin necesidad de escribir código. Tú desarrollas la Actions Integration; los clientes la arrastran y la sueltan en su flujo de trabajo.

Las Actions Integrations son funciones autocontenidas que se ejecutan en puntos específicos de la plataforma de Auth0. Están escritas en JavaScript y son de código cerrado; los clientes no pueden modificar el código.

<div id="create-and-publish-an-actions-integration">
  ## Crear y publicar una Actions Integration
</div>

Crear una Actions Integration implica planificar, desarrollar y hacer pruebas. Antes de ponerla a disposición de los clientes en el [Auth0 Marketplace](https://marketplace.auth0.com/), Auth0 revisará la Actions Integration. Para obtener información sobre el Auth0 Marketplace, consulta [Marketplace Partners](/docs/es/customize/integrations/marketplace-partners).

<div id="step-1-determine-your-use-case">
  ### Paso 1. Define tu caso de uso
</div>

Identifica qué caso de uso abordará tu Actions Integration. ¿Qué problema resolverá para los clientes? ¿Cómo lo resolverá? Para obtener más información, consulta [Definir un caso de uso para la integración](/docs/es/customize/integrations/marketplace-partners/defining-an-integration-use-case).

<div id="step-2-choose-a-flow">
  ### Paso 2. Elige un flujo
</div>

Las Actions Integrations constan de una sola [Action](/docs/es/customize/actions). Lo que una Action puede hacer está determinado por el lugar donde se ejecuta dentro del entorno de ejecución de Auth0, su Action  [flujo](/docs/es/customize/actions/explore-triggers). Los flujos de Action disponibles son:

* **Login flujo** se ejecuta de forma síncrona después de que un usuario se registre o inicie sesión correctamente. Los eventos de login incluyen flujos interactivos (como username/password o inicio de sesión federado) y flujos no interactivos (como verificaciones de sesión e intercambios de <Tooltip tip="Refresh Token: Token used to obtain a renewed Access Token without forcing users to log in again." cta="View Glossary" href="/docs/es/glossary?term=refresh+token">refresh token</Tooltip> exchanges). Para obtener más información sobre este flujo, lee [Login Flow](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger).
* **Machine to Machine flujo** se ejecuta de forma síncrona cuando una aplicación solicita un <Tooltip tip="Access Token: Authorization credential, in the form of an opaque string or JWT, used to access an API." cta="View Glossary" href="/docs/es/glossary?term=access+token">access token</Tooltip> usando la client credentials grant. Para obtener más información sobre este flujo, lee [Machine to Machine Flow](/docs/es/customize/actions/explore-triggers/machine-to-machine-trigger).
* **Pre User Registration flujo** se ejecuta de forma síncrona antes de que Auth0 agregue un usuario a una conexión de base de datos o <Tooltip tip="Passwordless: Form of authentication that does not rely on a password as the first factor." cta="View Glossary" href="/docs/es/glossary?term=passwordless">passwordless</Tooltip> connection. Este flujo no se ejecuta para inicios de sesión federados, como las conexiones empresariales o sociales. Para obtener más información sobre este flujo, lee [Pre User Registration Flow](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/pre-user-registration-trigger).
* **Post User Registration flujo** se ejecuta de forma asíncrona después de que Auth0 agregue un usuario a una conexión de base de datos o passwordless connection. Este flujo no se ejecuta para inicios de sesión federados, como las conexiones empresariales o sociales. Para obtener más información sobre este flujo, lee [Post User Registration Flow](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/post-user-registration-trigger).
* **Post Change Password flujo** se ejecuta de forma asíncrona después de que el cliente cambie la contraseña de un usuario de una conexión de base de datos. Este flujo no se ejecuta para cambios de contraseña en otros tipos de conexiones. Para obtener más información sobre este flujo, lee [Post Change Password Flow](/docs/es/customize/actions/explore-triggers/password-reset-triggers/post-change-password-trigger).
* **Enviar mensaje telefónico flujo** entrega un mensaje telefónico o SMS como parte de un proveedor personalizado de <Tooltip tip="Multi-factor authentication (MFA): User authentication process that uses a factor in addition to username and password such as a code via SMS." cta="View Glossary" href="/docs/es/glossary?term=MFA">MFA</Tooltip> provider. Para obtener información sobre proveedores MFA personalizados, lee [Customize Multi-factor Authentication SMS and Voice Messages](/docs/es/customize/customize-sms-or-voice-messages). Para obtener más información sobre este flujo, lee [Send Phone Message Flow](/docs/es/customize/actions/explore-triggers/mfa-notifications-trigger).

Para obtener más información sobre los flujos de Action, lee [Explore Flows and Triggers](/docs/es/customize/actions/explore-triggers). Si necesitas redirigir a los usuarios a un endpoint durante el login, lee [Redirect Actions for Partners](/docs/es/customize/integrations/marketplace-partners/redirect-actions-for-partners).

<div id="step-3-build-an-actions-integration">
  ## Paso 3. Crear una Actions Integration
</div>

Una vez que hayas decidido qué flujo se ajusta al caso de uso de tu Actions Integration, crea tu Action en un tenant de prueba. Para obtener instrucciones, consulta [Write Your First Action](/docs/es/customize/actions/write-your-first-action). Esta Action será la base del código de tu Actions Integration.

Además, para ayudarte a crear tu integración, puedes usar nuestras plantillas de integración de Action:

* [Plantilla de integración de Action de Post Login](https://github.com/auth0-marketplace/TEMPLATE-action-post-login)
* [Plantilla de integración de Action de máquina a máquina](https://github.com/auth0-marketplace/TEMPLATE-action-client-credentials)
* [Plantilla de integración de Action de Registro previo de usuario](https://github.com/auth0-marketplace/TEMPLATE-action-pre-user-registration)
* [Plantilla de integración de Action de Post User Registration](https://github.com/auth0-marketplace/TEMPLATE-action-post-user-registration)
* [Plantilla de integración de Action de Post Change Password](https://github.com/auth0-marketplace/TEMPLATE-action-post-password-change)
* [Plantilla de integración de Action de Enviar mensaje telefónico](https://github.com/auth0-marketplace/TEMPLATE-action-phone-message)

Para aprender a hacer que tu Actions Integration sea segura y estable, consulta [Actions Coding Guidelines](/docs/es/customize/actions/action-coding-guidelines). Además, sigue estas pautas específicas para las Actions Integrations:

* Al crear tu Action en un tenant de prueba, usa secrets para todos los valores que el cliente deba proporcionar (como identificadores, nombres de tenant y umbrales). Cuando envías tu Actions Integration, Auth0 te pide que indiques qué campos deben estar cifrados en reposo (seguir siendo secrets) y cuáles pueden ser campos de texto editables.
* Usa un indicador de configuración para el modo debug/test de logging que ayude con la configuración o la resolución de problemas. Las Actions Integrations rara vez deberían generar logs en todas las ejecuciones.
* Escribe los nombres de Secret en `ALL_CAPS_SNAKE_CASE` para facilitar la revisión.
* Las Actions Integrations son de código cerrado; los clientes no pueden ver ni modificar el código. Cualquier dato específico del cliente necesario para controlar la Actions Integration debe provenir de secrets o de la configuración.

<div id="step-4-test-your-actions-integrations">
  ### Paso 4. Prueba tus Actions Integrations
</div>

Como los revisores de Auth0 no tienen acceso a tu servicio, debes probar tu Action antes de enviarla para su revisión. Haz pruebas manuales tanto de las ejecuciones correctas como de los casos límite. Incluye estas comprobaciones:

* ¿La Action falla de forma silenciosa o bloquea la ejecución posterior?
* ¿Cómo funciona la Action si falla una solicitud HTTP?
* ¿Qué mensajes de error, si los hay, quedan expuestos a los usuarios?
* ¿Qué información se registra?
* ¿Se puede eludir o suplantar la Action?

Durante el proceso de envío, Auth0 solicita cobertura de pruebas unitarias para la ruta correcta. Auth0 proporciona todas las herramientas y ejemplos necesarios para que esto sea lo más sencillo posible.

<div id="step-5-submit-your-actions-integration">
  ### Paso 5. Envía tu Actions Integration
</div>

Una vez que hayas creado y probado exhaustivamente tu Actions Integration, envíala a Auth0 para su revisión: presenta una [solicitud de servicio de Auth0 Marketplace](https://autheco.atlassian.net/servicedesk/customer/portal/1/group/1/create/43). Tras una revisión inicial satisfactoria, la integración se evalúa para verificar su cumplimiento de los requisitos de seguridad.

<div id="step-6-publish-your-actions-integration">
  ### Paso 6. Publica tu Actions Integration
</div>

Una vez que Auth0 complete las revisiones de tu Actions Integration, te notificaremos si la integración las ha superado y cuáles son los plazos de publicación.
