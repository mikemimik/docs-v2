<div id="step-5-use-identity-tokens-to-flow-identity">
  ## Paso 5 - Usa tokens de identidad para transmitir la identidad
</div>

En este último paso, harás lo siguiente:

* Transmitir la identidad al servicio pasando tu <Tooltip tip="OpenID: Estándar abierto para la autenticación que permite a las aplicaciones verificar las identidades de los usuarios sin recopilar ni almacenar información de inicio de sesión." cta="Ver glosario" href="/docs/es/glossary?term=OpenID">OpenID</Tooltip> <Tooltip tip="OpenID: Estándar abierto para la autenticación que permite a las aplicaciones verificar las identidades de los usuarios sin recopilar ni almacenar información de inicio de sesión." cta="Ver glosario" href="/docs/es/glossary?term=JSON+Web+Token">JSON Web Token</Tooltip> (JWT);
* Validar el token;
* Extraer información del perfil para asignar un comprador a una mascota.

<div id="use-an-identity-token">
  ## Usa un token de identidad
</div>

Puedes usar tu función Lambda para procesar y obtener información sobre el usuario. Por ejemplo, durante una transacción de compra, recuperaste el nombre de usuario del perfil devuelto con el <Tooltip tip="ID Token: Credencial destinada al propio cliente, en lugar de para acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=identity+token">token de identidad</Tooltip>. Sin embargo, también puedes optar por incluir la información del usuario en el propio token de identidad, que es un JSON Web Token (JWT).

Las ventajas de usar JWT son las siguientes:

1. Verificar la autenticidad del JWT;
2. Asegurarte de que el usuario que realiza la llamada está autenticado (en lugar de depender de un parámetro de texto sin formato que podría haber sido manipulado).

Además, puedes usar el JWT para la autorización, lo que te permite omitir la integración de IAM con Amazon API Gateway. Ten en cuenta, sin embargo, que usar Amazon API Gateway para la autorización te permite detener la llamada a la API antes de invocar tu función Lambda.

<div id="add-information-to-the-jwt">
  ### Agregar información al JWT
</div>

Hay varias formas de agregar información de un usuario al JWT. El siguiente ejemplo agrega la dirección de correo electrónico del usuario al JWT, pero los conceptos son los mismos para otros datos del usuario.

<div id="use-rules">
  #### Usar Rules
</div>

Una forma de agregar la dirección de correo electrónico de un usuario al JWT es usar una [rule](/docs/es/customize/rules). Este es un buen enfoque si quieres asegurarte de que este valor esté siempre disponible en el JWT para el usuario que se está autenticando.

En `login.js`, puedes ver este scope especificado en los parámetros que se pasan a `auth.signin`:

```js lines
$scope.login = function() {
    var params = {
        authParams: {
          scope: 'openid email'
        }
      };

    auth.signin(params, function(profile, token) {
      ...
    }
  }
```

Aunque puedes incluir el perfil completo del usuario en el JWT, conviene incluir solo lo necesario, ya que el JWT normalmente se envía con cada solicitud.

<div id="validate-the-jwt-token">
  ## Validar el token JWT
</div>

Como la consola de AWS Lambda solo tiene acceso a un número limitado de módulos de Node.js que pueden usarse cuando introduces tu código de Node.js desde la consola del navegador, tendrás que incluir módulos adicionales y cargar la función Lambda como un paquete para procesar el token de identidad.

Para obtener más detalles, consulta [Creación de paquetes de implementación con Node.js](http://docs.aws.amazon.com/lambda/latest/dg/nodejs-create-deployment-pkg.html) y [Carga y prueba de paquetes de implementación](http://docs.aws.amazon.com/lambda/latest/dg/walkthrough-s3-events-adminuser-create-test-function-upload-zip-test.html).

El siguiente proyecto base contiene el código que necesitarás para tu función de AWS Lambda actualizada.

```js wrap lines
<%= include('../../../_includes/_package', { org: 'auth0', repo: 'auth0-aws', path: 'examples/api-gateway/lambda' }) %>
```

Verás dos archivos JavaScript personalizados dentro del proyecto de ejemplo:

* `index.js`: contiene tu código principal;
* `auth0-variables`: contiene el código que debes actualizar.

Además de los archivos personalizados, hay un archivo estándar de Node.js: `package.json`.

El código añade la funcionalidad necesaria para extraer información del JWT y validarlo. De forma predeterminada, Auth0 usa una clave simétrica para firmar el JWT, aunque también puedes optar por usar claves asimétricas (si necesitas permitir que terceros validen tu token, debes usar una clave asimétrica y compartir solo tu clave pública).

Para obtener más información sobre la verificación de tokens, consulta [Protocolos de identidad compatibles con Auth0](/docs/es/authenticate/protocols).

Actualiza `auth0-variables.js` con tu clave secreta, que puedes encontrar en la pestaña Settings de tu aplicación en el <Tooltip tip="Auth0 Dashboard: el producto principal de Auth0 para configurar tus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip>:

```javascript lines
var env={};
env.AUTH0_SECRET='{yourAuth0Secret}';
module.exports = env;
```

Ejecuta **npm install** desde el directorio donde están tus archivos, comprime el contenido en un archivo zip (`index.js` debe estar en la raíz del zip) y súbelo para que lo use la función Lambda `PurchasePet`. Si haces esta prueba, deberías ver un error de autorización, ya que el JWT no está en el cuerpo del mensaje.

Echa un vistazo a la lógica de `index.js`. Verás que, alrededor de la línea 60, hay lógica que valida el token y extrae la información decodificada, que contiene la información de identidad utilizada en la lógica de compra:

```js lines
if(event.authToken) {

  } else {
     console.log('invalid authorization token', event.authToken);
     context.done('authorization failure', null);
  }
    ...
```

<div id="extract-profile-information-to-assign-a-buyer">
  ### Extraer información del perfil para asignar un comprador
</div>

El paso final es pasar el JWT al método que usa el cliente del navegador.

El método estándar incluye un encabezado `Authorization` con un bearer token, y puede usar este método desactivando la autorización de IAM y basándose únicamente en el token de OpenID para la autorización (también tendrá que mapear el encabezado `Authorization` en los datos del evento que se pasan a la función de AWS Lambda).

Si, por el contrario, está usando IAM, AWS API Gateway usa el encabezado `Authorization` para contener la firma del mensaje, y romperá la autenticación si inserta el JWT en este encabezado. Para hacerlo, puede:

* Agregar un encabezado personalizado para el JWT;
* Poner el encabezado personalizado en el cuerpo del mensaje.

Si elige usar un encabezado personalizado, también tendrá que hacer algo de mapeo para el Integration Request del método POST para `pets/purchase`.

Para simplificar el proceso de validación, pase el JWT en el cuerpo del post a la función de AWS Lambda. Para hacerlo, actualice el método `buyPet` en `home.js` eliminando `userName` del cuerpo y agregando `authToken` de la siguiente manera:

```javascript lines
function buyPet(user, id) {
    var apigClient = getSecureApiClient();
    var body = {
      petId:id,
      authToken: store.get('token')
    };

    apigClient.petsPurchasePost({}, body)
      .then(function(response) {
        console.log(response);
        $scope.pets = response.data;
        $scope.$apply();
      }).catch(function (response) {
        alert('buy pets failed');
        showError(response);
    });
}
```

Sube tu código a tu bucket de S3 e intenta comprar una mascota. Verás el correo electrónico del comprador en el mensaje resultante.

Si se produce algún error, vuelve a comprobar que hayas configurado correctamente tu clave secreta. Una herramienta útil para revisar problemas con la decodificación de tu token es [jwt.io](http://jwt.io/).

<div id="summary">
  ## Resumen
</div>

En este tutorial, has hecho lo siguiente:

* Creado una API con AWS API Gateway que incluye métodos que usan funciones AWS Lambda;
* Protegido el acceso a tu API mediante roles de IAM;
* Integrado un <Tooltip tip="Proveedor de identidad (IdP): servicio que almacena y administra identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=SAML">SAML</Tooltip> <Tooltip tip="Security Assertion Markup Language (SAML): protocolo estandarizado que permite a dos partes intercambiar información de autenticación sin una contraseña." cta="Ver glosario" href="/docs/es/glossary?term=identity+provider">proveedor de identidad</Tooltip> con IAM para vincular el acceso a la API con tu base de usuarios;
* Proporcionado distintos niveles de acceso según si un usuario se autenticó a través de Database o Social Connection;
* Usado una rule de Auth0 para hacer cumplir la asignación de roles;
* Usado un JWT para proporcionar contexto adicional de autorización y pasar información de identidad a la función Lambda correspondiente.
