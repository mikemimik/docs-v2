<div id="step-5-use-identity-tokens-to-flow-identity">
  ## Paso 5: usa tokens de identidad para transmitir la identidad
</div>

En este paso final, harás lo siguiente:

* Transmitir la identidad al servicio pasando tu <Tooltip tip="OpenID: estándar abierto de autenticación que permite que las aplicaciones verifiquen la identidad de los usuarios sin recopilar ni almacenar información de inicio de sesión." cta="Ver glosario" href="/docs/es/glossary?term=OpenID">OpenID</Tooltip> <Tooltip tip="OpenID: estándar abierto de autenticación que permite que las aplicaciones verifiquen la identidad de los usuarios sin recopilar ni almacenar información de inicio de sesión." cta="Ver glosario" href="/docs/es/glossary?term=JSON+Web+Token">JSON Web Token</Tooltip> (JWT);
* Validar el token;
* Extraer información del perfil para asignar un comprador a una mascota.

<div id="use-an-identity-token">
  ## Usa un token de identidad
</div>

Puedes usar tu función Lambda para procesar y obtener información sobre el usuario. Por ejemplo, durante una transacción de compra, recuperaste el nombre de usuario del perfil devuelto con el <Tooltip tip="token de identidad: Credencial destinada al propio cliente, en lugar de para acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=identity+token">token de identidad</Tooltip>. Sin embargo, también puedes optar por incluir la información del usuario en el propio token de identidad, que es un JSON Web Token (JWT).

Las ventajas de usar JWT son que puedes:

1. Verificar la autenticidad del JWT;
2. Asegurarte de que el usuario que hace la llamada esté autenticado (en lugar de depender de un parámetro de texto sin formato que podría haber sido manipulado).

Además, puedes usar el JWT para la autorización, lo que te permite omitir la integración de IAM con Amazon API Gateway. Ten en cuenta, sin embargo, que usar API Gateway para la autorización te permite detener la llamada a la API antes de invocar tu función Lambda.

<div id="add-information-to-the-jwt">
  ### Agrega información al JWT
</div>

Hay varias formas de agregar información de un usuario al JWT. En el siguiente ejemplo se agrega la dirección de correo electrónico del usuario al JWT, pero los mismos conceptos se aplican a otros datos de usuario.

<div id="use-rules">
  #### Usar Rules
</div>

Una forma de agregar la dirección de correo electrónico de un usuario al JWT es usar una [rule](/docs/es/customize/rules). Este es un buen enfoque si quieres asegurarte de que este valor esté siempre disponible en el JWT para un usuario que se autentica.

En `login.js`, puedes ver este scope especificado en los parámetros que se pasan a `auth.signin`:

```js lines
$scope.login = function() {
    var params = {
        authParams: {
          scope: 'openid email'
        }
      };

    auth.signin(params, function(profile, token) {
      ...
    }
  }
```

Aunque puedes incluir el perfil completo del usuario en el JWT, te conviene incluir solo lo necesario, ya que el JWT suele enviarse con cada solicitud.

<div id="validate-the-jwt-token">
  ## Validar el token JWT
</div>

Como la consola de AWS Lambda solo tiene acceso a un número limitado de módulos de Node.js cuando introduces tu código de Node.js desde la consola del navegador, tendrás que incluir módulos adicionales y subir la función Lambda como un paquete para procesar el token de identidad.

Para obtener más detalles, consulta [Creación de paquetes de implementación con Node.js](http://docs.aws.amazon.com/lambda/latest/dg/nodejs-create-deployment-pkg.html) y [Carga de paquetes de implementación y pruebas](http://docs.aws.amazon.com/lambda/latest/dg/walkthrough-s3-events-adminuser-create-test-function-upload-zip-test.html).

El siguiente proyecto base contiene el código que necesitarás para tu función de AWS Lambda actualizada.

```js wrap lines
<%= include('../../../_includes/_package', { org: 'auth0', repo: 'auth0-aws', path: 'examples/api-gateway/lambda' }) %>
```

Verás dos archivos Javascript personalizados en el proyecto base:

* `index.js`: contiene tu código principal;
* `auth0-variables`: contiene el código que debes actualizar.

Además de los archivos personalizados, hay un archivo estándar de Node.js llamado `package.json`.

El código agrega la funcionalidad necesaria para extraer información del JWT y validarlo. De forma predeterminada, Auth0 usa una clave simétrica para firmar el JWT, aunque también puedes optar por usar claves asimétricas (si necesitas permitir la validación de tu token por parte de terceros, debes usar una clave asimétrica y compartir únicamente tu clave pública).

Para obtener más información sobre la verificación de tokens, consulta [Protocolos de identidad compatibles con Auth0](/docs/es/authenticate/protocols).

Actualiza `auth0-variables.js` con tu clave secreta, que encontrarás en la pestaña Settings de tu aplicación en el <Tooltip tip="Auth0 Dashboard: el producto principal de Auth0 para configurar tus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip>:

```javascript lines
var env={};
env.AUTH0_SECRET='{yourAuth0Secret}';
module.exports = env;
```

Ejecuta **npm install** desde el directorio donde están tus archivos, comprime el contenido en un archivo zip (`index.js` debe estar en la raíz del zip) y súbelo para que lo use la función Lambda `PurchasePet`. Si haces esta prueba, deberías ver un error de autorización, ya que el JWT no está en el cuerpo del mensaje.

Echa un vistazo a la lógica de `index.js`. Verás, cerca de la línea 60, la lógica que valida el token y extrae la información decodificada que contiene los datos de identidad usados en la lógica de compra:

```js lines
if(event.authToken) {
     jwt.verify(event.authToken, secret, function(err, decoded) {

         } else if(!decoded.email)

         }
     });
  } else {
     console.log('invalid authorization token', event.authToken);
     context.done('authorization failure', null);
  }
    ...
```

<div id="extract-profile-information-to-assign-a-buyer">
  ### Extraer información del perfil para asignar un comprador
</div>

El paso final es pasar el JWT al método que utiliza el cliente del navegador.

El método estándar incluye un encabezado `Authorization` con un bearer token, y puede usar este método desactivando la autorización de IAM y basándose únicamente en el token de OpenID para la autorización (también tendrá que mapear el encabezado `Authorization` en los datos del evento que se pasan a la función AWS Lambda).

Sin embargo, si está usando IAM, AWS API Gateway utiliza el encabezado `Authorization` para contener la firma del mensaje, y la autenticación se interrumpirá si inserta el JWT en este encabezado. Para hacerlo, puede:

* Agregar un encabezado personalizado para el JWT;
* Colocar el encabezado personalizado en el cuerpo del mensaje.

Si decide usar un encabezado personalizado, también tendrá que hacer algo de mapeo para la Integration Request del método POST para `pets/purchase`.

Para simplificar el proceso de validación, pase el JWT en el cuerpo del POST a la función AWS Lambda. Para ello, actualice el método `buyPet` en `home.js` eliminando `userName` del cuerpo y agregando `authToken` de la siguiente manera:

```javascript lines
function buyPet(user, id) {
    var apigClient = getSecureApiClient();
    var body = {
      petId:id,
      authToken: store.get('token')
    };

    apigClient.petsPurchasePost({}, body)
      .then(function(response) {
        console.log(response);
        $scope.pets = response.data;
        $scope.$apply();
      }).catch(function (response) {
        alert('buy pets failed');
        showError(response);
    });
}
```

Sube tu código a tu bucket de S3 e intenta comprar una mascota. Verás el correo electrónico del comprador en el mensaje que se genera.

Si tienes algún error, vuelve a comprobar que hayas configurado correctamente tu clave secreta. Una herramienta útil para revisar problemas con la decodificación de tu token es [jwt.io](http://jwt.io/).

<div id="summary">
  ## Resumen
</div>

En este tutorial, has hecho lo siguiente:

* Creado una API con AWS API Gateway que incluye métodos que usan funciones de AWS Lambda;
* Protegido el acceso a tu API mediante roles de IAM;
* Integrado un <Tooltip tip="Proveedor de identidad (IdP): Servicio que almacena y gestiona identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=SAML">SAML</Tooltip> <Tooltip tip="Security Assertion Markup Language (SAML): Protocolo estandarizado que permite a dos partes intercambiar información de autenticación sin una contraseña." cta="Ver glosario" href="/docs/es/glossary?term=identity+provider">proveedor de identidad</Tooltip> con IAM para vincular el acceso a la API con tu base de usuarios;
* Proporcionado distintos niveles de acceso según si un usuario se autenticó mediante Database o Social Connection;
* Usado una Rule de Auth0 para aplicar la asignación de roles;
* Usado un JWT para proporcionar más contexto de autorización y pasar información de identidad a la función Lambda adecuada.
