<div id="step-2-secure-and-deploy-the-amazon-api-gateway">
  ## Paso 2: proteger e implementar Amazon API Gateway
</div>

Ahora que tu API ya está funcionando, debes añadir seguridad. En este paso, harás lo siguiente:

* Proteger la API de actualización para limitar el acceso a usuarios autenticados con un rol de AWS IAM específico;
* Configurar Auth0 Delegación para usar las capacidades de federación de AWS IAM;
* Obtener un AWS <Tooltip tip="Access Token: Credencial de autorización, en forma de cadena opaca o JWT, que se utiliza para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=Access+Token">Access Token</Tooltip> que use el rol de AWS IAM.

Una vez que tu API esté protegida, crearás una aplicación de página única (SPA) sin servidor. La SPA dependerá de la federación de identidad para determinar qué usuarios tienen acceso. Al combinar AWS IAM Integration for AWS Gateway API, AWS IAM Identity Federation for <Tooltip tip="Security Assertion Markup Language (SAML): Protocolo estandarizado que permite a dos partes intercambiar información de autenticación sin una contraseña." cta="Ver glosario" href="/docs/es/glossary?term=SAML">SAML</Tooltip> y Auth0 Delegación for AWS, puedes permitir que usuarios de muchas fuentes distintas, incluidos proveedores sociales o conexiones empresariales, accedan a tus APIs. El siguiente diagrama ilustra un flujo de ejemplo que usa un <Tooltip tip="Identity Provider (IdP): Servicio que almacena y administra identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=Identity+Provider">proveedor de identidad</Tooltip> basado en SAML y Auth0 SAML Federation and Delegación for AWS.

Verás dos formas de implementar este flujo:

1. Usar Auth0 Delegación con AWS IAM;
2. Añadir un token de identidad para transmitir la identidad a la función Lambda.

Delegación te permite obtener fácilmente tokens de AWS para acceder a los servicios de AWS desde tu aplicación.

<div id="ways-to-secure-the-amazon-api-gateway">
  ### Formas de proteger Amazon API Gateway
</div>

AWS API Gateway ofrece varios métodos para proteger tus API:

1. claves API;
2. IAM;
3. [Amazon Cognito](/docs/es/customize/integrations/aws/aws-api-gateway-cognito).

Usar claves API suele ser apropiado para interacciones de servicio a servicio, como se muestra a continuación. Sin embargo, este enfoque tiene varias desventajas:

* Incluir un secreto de larga duración en la aplicación es arriesgado (las aplicaciones son más fáciles de vulnerar);
* Crear un framework para emitir y administrar claves API requiere una implementación segura que puede ser difícil de desarrollar.

En esta sección del tutorial se utilizarán roles y políticas de IAM para proteger tu API en API Gateway, aunque también puedes optar por hacerlo con User Pools de Amazon Cognito. Para consultar instrucciones detalladas sobre cómo proteger tu API de AWS, lee [Protege AWS API Gateway con Cognito](/docs/es/customize/integrations/aws/aws-api-gateway-cognito). Para obtener más información sobre el uso de roles y políticas de IAM, consulta el artículo de Amazon [Controla el acceso a una API con permisos de IAM](http://docs.aws.amazon.com/apigateway/latest/developerguide/permissions.html). Para obtener más información sobre User Pools en Cognito, visita [User Pool de Amazon Cognit](http://docs.aws.amazon.com/cognito/latest/developerguide/cognito-user-identity-pools.html).

<div id="step-1-configure-iam-and-auth0-for-saml-integration-with-the-api-gateway">
  ### Paso 1. Configura IAM y Auth0 para la integración de SAML con Amazon API Gateway
</div>

Puedes especificar un rol de AWS IAM para el token SAML que intercambias por un AWS token. Según los permisos concedidos a ese rol de AWS IAM (que se establecen mediante tu proveedor de identidad), el token recibido tendrá esos mismos permisos. Al emitir distintos tokens SAML, cada uno con su propio rol de AWS IAM, puedes controlar los niveles de acceso de tus usuarios.

Por ejemplo, el IdP podría especificar el rol de AWS IAM en función de la pertenencia a un grupo (por ejemplo, un administrador en Active Directory) o de la fuente de authentication (por ejemplo, una conexión de base de datos o un proveedor social como Facebook). Este enfoque te permite diferenciar el acceso de los usuarios a los métodos de Amazon API Gateway cuando están protegidos con AWS IAM.

<div id="configure-auth0">
  #### Configurar Auth0
</div>

Inicia sesión en tu cuenta de Auth0. Accederás al Dashboard de administración. Haz clic en **+ New Application**, que se encuentra en la esquina superior derecha de la página.

Asigna a tu nueva aplicación el nombre AWS API Gateway e indica que esta aplicación será una aplicación de página única. Haz clic en **Create**.

Ve a la pestaña Addons de la aplicación que acabas de crear. Con el interruptor correspondiente, habilita Amazon Web Services. Esto activa AWS Delegation.

<div id="configure-aws">
  #### Configurar AWS
</div>

Sigue el tutorial [How to Set Up AWS for Delegated Authentication](/docs/es/customize/integrations/aws/how-to-set-up-aws-for-delegated-authentication) para configurar AWS para el acceso delegado, que utiliza SAML. Ten en cuenta lo siguiente:

* Sigue las [instrucciones a continuación](#setting-the-permissions-policy-on-your-iws-iam-role) para adjuntar la política de permisos a tu rol, en lugar de las del tutorial enlazado;
* Nombra `auth0` al proveedor SAML que crees;
* Nombra `auth0-api-role` al rol de AWS IAM.

<div id="set-the-permissions-policy-on-your-iws-iam-role">
  ##### Configura la política de permisos en tu rol de IWS en IAM
</div>

Una vez que hayas configurado el rol de AWS IAM, añadirás una política a `auth0-api-role` que te permitirá ejecutar los métodos de tu Amazon API Gateway. Para obtener más información sobre este proceso, consulta [Permisos de acceso de usuario para Amazon API Gateway](http://docs.aws.amazon.com/apigateway/latest/developerguide/permissions.html).

<div id="getting-the-gateway-api-arn">
  #### Obtención del ARN de Gateway API
</div>

Antes de comenzar, necesitará el ARN de su Gateway API:

1. Vaya a [Amazon API Gateway Console](https://console.aws.amazon.com/apigateway) e inicie sesión.
2. Seleccione la API correspondiente.
3. Haga clic en cualquiera de los métodos asociados con la API para abrir la página Method Execution.
4. En la página Method Execution, el cuadro Method Request de la esquina superior izquierda muestra el **ARN** de la API, aunque incluye el nombre del método:

`arn:aws:execute-api:us-east-2:484857107747:97i1dwv0j4/*/POST/`

Deberá quitar el nombre del método para obtener el ARN base de la API:

`arn:aws:execute-api:us-east-2:484857107747:97i1dwv0j4/*`

El wildcard (`*`) del ARN anterior habilita permisos para su API en todas las etapas, pero puede implementar distintas etapas por separado (por ejemplo, desarrollo, luego pruebas y luego producción).

Seleccione el role `auth0-api-role` que acaba de crear para abrir su página Summary.

Expanda **Inline Policies** y haga clic en **click here**.

Seleccione **Custom Policy** y haga clic en **Select**.

Edite su documento de política. Puede establecer **Policy Name** con el nombre que desee, pero le sugerimos algo como `api-gateway-policy`. Para habilitar el acceso a los métodos de la API para este role, aplique la siguiente política después de actualizar el ARN con el correspondiente a su API.

```json lines
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Effect": "Allow",
            "Action": [
                "execute-api:*"
            ],
            "Resource": [
                "arn:[{yourArn}]"
            ]
        }
    ]
}
```

Haz clic en **Aplicar política**.

Dado que API Gateway asumirá este rol en nombre del usuario, la política de confianza debe permitir esta acción. Para ello, edita las relaciones de confianza del rol y ve a esta pestaña en la página de resumen del rol.

La relación de confianza final debería verse así:

<div id="step-2-set-up-cors-and-deploy-the-api">
  ### Paso 2. Configura CORS e implementa la API
</div>

Nuestra aplicación de página única (SPA) accederá a métodos de Web API desde un dominio distinto al de la página. La configuración de Cross-Origin Resource Sharing debe permitir explícitamente esta acción para que el navegador autorice el acceso a AWS API Gateway. Normalmente, el navegador primero enviará una solicitud `OPTIONS` para ver qué acciones permite el sitio.

Selecciona `/pets` en Resources y haz clic en **Create Method**. En el menú desplegable, selecciona **OPTIONS** y haz clic en la **marca de verificación** para guardar la configuración.

El navegador usa el método Options para obtener los encabezados HTTP necesarios, pero la función necesita más instrucciones sobre qué hacer. En la pantalla de Setup de `OPTIONS`, configura las siguientes variables/parámetros:

* **Integration Type**: Lambda Function;
* **Use Lambda Proxy Integration**: déjalo sin marcar;
* **Lambda Region**: selecciona tu región;
* **Lambda Function**: NoOp.

Haz clic en **Save**. En la siguiente ventana emergente, concede a tu función Lambda los permisos que necesita.

Luego se te redirigirá automáticamente a la página `OPTIONS` Method Execution. Abre la página Method Response.

Expande la sección **200** ubicada debajo de la barra HTTP Status y agrega los siguientes encabezados de respuesta:

* Access-Control-Allow-Headers;
* Access-Control-Allow-Methods;
* Access-Control-Allow-Origin.

A continuación, asigna los valores adecuados a cada uno de los encabezados de respuesta. Después de volver a la página Method Execution, haz clic en **Integration Response**. Después de expandir la fila asociada con el estado de respuesta del método **200**, expande **Header Mappings** y aplica los siguientes mapeos:

* Access-Control-Allow-Headers: `'Content-Type,X-Amz-Date,Authorization,x-api-key,x-amz-security-token'`;
* Access-Control-Allow-Origin: `'*'`
* Access-Control-Allow-Methods: `'POST, GET, OPTIONS'`

Por último, repite los pasos anteriores para habilitar CORS para los métodos POST y GET. Sin embargo, para estos dos métodos, agregarás un encabezado, Access-Control-Allow-Origin, y su valor debe establecerse en `'*'`.

<div id="deploy-the-api">
  ### Implementar la API
</div>

Vuelve a la vista **Resources** de tu API. Haz clic en **Actions** y selecciona **DEPLOY API**.

Selecciona **New Stage** en Deploy State y asigna a la etapa el nombre `Test`. Haz clic en el botón **Deploy**.

En la página de resultados, ve a **SDK Generation**. Selecciona JavaScript como **Platform**. Haz clic en el botón **Generate SDK**.

Guarda el archivo zip descargado para usarlo más adelante.
