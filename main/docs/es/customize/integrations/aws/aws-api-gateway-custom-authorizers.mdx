Proteja los endpoints de AWS API Gateway con autorizadores personalizados que acepten <Tooltip tip="Token de acceso: Credencial de autorización, en forma de una cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=access+tokens">tokens de acceso</Tooltip> emitidos por Auth0. Para ello, configure su API en API Gateway, cree y configure sus funciones de AWS Lambda (incluidos los autorizadores personalizados) para proteger los endpoints de su API e implemente el <Tooltip tip="Flujo de autorización: Concesión de autorización (o flujo de trabajo) especificado en el marco de OAuth 2.0." cta="Ver glosario" href="/docs/es/glossary?term=authorization+flow">flujo de autorización</Tooltip> para que sus usuarios puedan obtener de Auth0 los tokens de acceso necesarios para acceder a su API.

Para obtener más información, visite [AWS Lambda Overview](https://aws.amazon.com/lambda/) después de iniciar sesión en [AWS API Gateway](https://aws.amazon.com/api-gateway/).

API Gateway amplía las capacidades de Lambda al añadir una capa de servicio delante de sus funciones de Lambda para reforzar la seguridad, gestionar las transformaciones de los mensajes de entrada y salida, y ofrecer capacidades como limitación de velocidad y auditoría. Un enfoque sin servidor simplifica sus necesidades operativas, ya que aspectos como el escalado y la tolerancia a fallos pasan a ser responsabilidad del servicio de cómputo que ejecuta su código.

Los autorizadores personalizados:

* Confirmarán que el token de acceso se haya enviado mediante el encabezado `authorization` de la solicitud para acceder a la API.
* Verificarán la firma RS256 del token de acceso mediante una clave pública obtenida a través de un endpoint JWKS.
* Garantizarán que el token de acceso incluya las reclamaciones requeridas de emisor `iss` y <Tooltip tip="Audiencia: Identificador único de la audiencia de un token emitido. Llamado aud en un token, su valor contiene el ID de una aplicación (Client ID) para un ID Token o de una API (API Identifier) para un Access Token." cta="Ver glosario" href="/docs/es/glossary?term=audience">audiencia</Tooltip> `aud`.

<div id="how-api-gateway-custom-authorizers-work">
  ## Cómo funcionan los autorizadores personalizados de API Gateway
</div>

[Según Amazon](http://docs.aws.amazon.com/apigateway/latest/developerguide/use-custom-authorizer.html), un autorizador personalizado de API Gateway es una &quot;función Lambda que proporcionas para controlar el acceso a tu API mediante estrategias de autenticación con tokens Bearer, como <Tooltip tip="OAuth 2.0: Marco de autorización que define protocolos y flujos de trabajo de autorización." cta="Ver glosario" href="/docs/es/glossary?term=OAuth">OAuth</Tooltip> o <Tooltip tip="OAuth 2.0: Marco de autorización que define protocolos y flujos de trabajo de autorización." cta="Ver glosario" href="/docs/es/glossary?term=SAML">SAML</Tooltip>.&quot;

Cada vez que alguien (o algún programa) intenta llamar a tu API, API Gateway comprueba si hay un autorizador personalizado configurado para la API.

Si **hay un autorizador personalizado para la API**, API Gateway llama al autorizador personalizado y le proporciona el token de autorización extraído de la cabecera de la solicitud.

Puedes usar el autorizador personalizado para implementar distintos tipos de estrategias de autorización, incluida la verificación de JWT, y devolver políticas de IAM que autoricen la solicitud. Si la política devuelta no es válida o si se deniegan los permisos, la llamada a la API falla.

Si la política es válida, API Gateway almacena en caché la política devuelta, la asocia con el token recibido y la usa para la solicitud actual y las posteriores. Puedes configurar durante cuánto tiempo se almacena en caché la política. El valor predeterminado es `300` segundos y la duración máxima del almacenamiento en caché es de `3600` segundos (también puedes establecer el valor en 0 para desactivar el almacenamiento en caché).

Para obtener más información, visita [¿Qué es Amazon API Gateway?](http://docs.aws.amazon.com/apigateway/latest/developerguide/use-custom-authorizer.html) en la guía para desarrolladores de Amazon. Para obtener más información sobre la verificación de <Tooltip tip="JSON Web Token (JWT): Formato estándar de token de ID (y, a menudo, formato de token de acceso) usado para representar claims de forma segura entre dos partes." cta="Ver glosario" href="/docs/es/glossary?term=JWT">JWT</Tooltip>, consulta nuestro artículo sobre [JSON Web Token](/docs/es/secure/tokens/json-web-tokens).

<div id="create-the-custom-authorizers">
  ## Crear los autorizadores personalizados
</div>

Ahora que tenemos una API totalmente funcional administrada por API Gateway, protege esta API para que solo quienes tengan la autorización adecuada puedan acceder al back-end que está detrás de ella.

Usa los autorizadores de solicitud personalizados de API Gateway para autorizar tus API mediante estrategias de autorización con tokens de token Bearer, como OAuth 2.0 o SAML. Para cada solicitud entrante, ocurre lo siguiente:

1. API Gateway comprueba que haya un autorizador personalizado configurado correctamente.
2. API Gateway invoca al autorizador personalizado (que es una función de Lambda) con el token de autorización.
3. Si el token de autorización es válido, el autorizador personalizado devuelve las políticas correspondientes de AWS Identity and Access Management (IAM).
4. API Gateway usa las políticas devueltas en el paso 3 para autorizar la solicitud.

<div id="prepare-the-custom-authorizer">
  ### Preparar el autorizador personalizado
</div>

Puedes [descargar un ejemplo de autorizador personalizado](https://github.com/auth0-samples/jwt-rsa-aws-custom-authorizer) compatible con tokens emitidos por Auth0. Después, tendrás que personalizar los archivos para que el autorizador personalizado funcione en tu entorno.

1. Descomprime en la ubicación que prefieras la carpeta que contiene los archivos de ejemplo que descargaste anteriormente y ve a esa carpeta desde la línea de comandos.
2. Dentro de la carpeta de ejemplo, ejecuta `npm install` para instalar los paquetes de Node.js necesarios para la implementación; AWS exige que estos archivos se incluyan en el paquete que subirás a AWS en un paso posterior.
3. Configura tu entorno local con un archivo `.env`. Puedes copiar el archivo `.env.sample` (y renombrarlo al mismo tiempo como `.env`) con `cp .env.sample .env`. Realiza los siguientes cambios:

   | Parámetro          | Valor                                                                                                                            |
   | ------------------ | -------------------------------------------------------------------------------------------------------------------------------- |
   | **`TOKEN_ISSUER`** | El emisor del token. Si Auth0 es el emisor del token, usa `https://{yourDomain}/`. Asegúrate de incluir la barra diagonal final. |
   | **`JWKS_URI`**     | La URL del endpoint de JWKS. Si Auth0 es el emisor del token, usa `https://{yourDomain}/.well-known/jwks.json`                   |
   | **`AUDIENCE`**     | El valor del **identificador** de la API que creaste en la sección anterior, Crear una API de Auth0.                             |

<div id="test-the-custom-authorizer-locally">
  ### Probar el autorizador personalizado de forma local
</div>

Obtén un token de acceso JWT válido. Hay varias formas de obtenerlo, y el método que elijas dependerá del tipo de aplicación, el nivel de confianza o la experiencia general del usuario final. 1. Puedes obtener un token de prueba para tu API yendo a [Auth0 Dashboard &gt; Applications &gt; APIs](https://manage.auth0.com/#/apis), seleccionando tu API y haciendo clic en **Test**.
2. Crea un archivo local `event.json` que contenga el token. Puedes copiar el archivo de ejemplo (ejecuta `cp event.json.sample event.json`). Reemplaza `ACCESS_TOKEN` por tu token JWT y `methodArn` por el valor de ARN correspondiente al método `GET` de tu API.

1. En la consola de API Gateway, abre la API de **PetStore**.
2. En el panel de navegación izquierdo, selecciona **Resources**.
3. En el panel central **Resources**, expande el árbol de recursos. Debajo de `/pets`, selecciona **GET**.
4. En el cuadro **Method Request**, verás el **ARN**.
5. Ejecuta la prueba con `npm test`.

<div id="create-the-iam-role">
  ### Crear el rol de IAM
</div>

El rol de IAM tiene los permisos necesarios para invocar funciones Lambda; antes de continuar con nuestro autorizador personalizado, tendremos que crear un rol de IAM que pueda invocarlo cada vez que API Gateway reciba una solicitud de acceso.

1. Inicia sesión en AWS y ve a la [Consola de IAM](https://console.aws.amazon.com/iam). En el panel de navegación de la izquierda, selecciona **Roles**.

2. Selecciona **Create new role**.

3. En **AWS service**, selecciona la fila **AWS Lambda** y, a continuación, **Next: Permissions**.

4. En la pantalla **Attach permissions policy**, selecciona **AWSLambdaRole**. Puedes usar el filtro disponible para acotar la lista de opciones. Selecciona **Next: Tags** y, después, **Next: Review** para continuar.

5. En la pantalla **Review**, indica un **Role name**, como `Auth0Integration`. Deja el resto de los campos tal como están. Selecciona **Create role**.

6. Cuando AWS haya creado el rol, volverás a la página **Roles** de IAM. Selecciona tu nuevo rol.

7. En la página **Summary** del rol que acabas de crear, selecciona la vista **Trust relationships**.

8. Selecciona **Edit trust relationship** y completa el campo **Policy Document** con el siguiente fragmento JSON:

   ```json lines
   {
      "Version": "2012-10-17",
      "Statement": [
         {
            "Effect": "Allow",
            "Principal": {
               "Service": [
                  "apigateway.amazonaws.com",
                  "lambda.amazonaws.com"
               ]
            },
            "Action": "sts:AssumeRole"
         }
      ]
   }
   ```

9. Haz clic en **Update Trust Policy**.

10. Se te redirigirá de nuevo a la página **Summary**. Copia el valor de **Role ARN** para usarlo más adelante.

<div id="create-the-lambda-function-and-deploy-the-custom-authorizer">
  ### Cree la función Lambda y despliegue el autorizador personalizado
</div>

Ahora que ha configurado su autorizador personalizado para su entorno y ha comprobado que funciona, desplieguelo en AWS.

1. Cree un paquete que pueda cargar en AWS ejecutando `npm run bundle`. Esto genera un paquete `custom-authorizer.zip` que contiene el código fuente, la configuración y los módulos de Node requeridos por AWS Lambda.
2. Vaya a la [consola de Lambda](https://console.aws.amazon.com/lambda) y haga clic en **Create function**.
3. En la página **Select blueprint**, haga clic en **Author from scratch** para crear una función en blanco. En **Basic information**, proporcione valores para los siguientes parámetros:

   | Parámetro       | Valor                                                           |
   | --------------- | --------------------------------------------------------------- |
   | **Name**        | Un nombre para su función Lambda, como `jwtRsaCustomAuthorizer` |
   | **Description** | Una descripción para su función Lambda (opcional)               |
   | **Runtime**     | Seleccione `Node.js 10.x`                                       |
4. Haga clic en **Create Function** para continuar.
5. En la página **Configuration** de su función, desplácese hacia abajo hasta la sección **Function Code**.
6. Seleccione **Upload a .ZIP file** como **Code entry type**.
7. Haga clic en **Upload** y seleccione el paquete `custom-authorizer.zip` que creó anteriormente.
8. Luego, cree las siguientes tres **Environment variables**. Tenga en cuenta que esta información es idéntica a la del archivo `.env`.

   | Parámetro          | Valor                                                                                                       |
   | ------------------ | ----------------------------------------------------------------------------------------------------------- |
   | **`TOKEN_ISSUER`** | El emisor del token. Si Auth0 es el emisor del token, use `https://{yourDomain}/`                           |
   | **`JWKS_URI`**     | La URL del endpoint JWKS. Si Auth0 es el emisor del token, use `https://{yourDomain}/.well-known/jwks.json` |
   | **`AUDIENCE`**     | El valor de **identificador** de la API que creó en el paso 1.                                                 |
9. En la sección **Execution role**, seleccione **Use an existing role** y, a continuación, seleccione como **Existing role** el rol de IAM que creó anteriormente.
10. En **Basic settings**, establezca **Timeout** en **30** segundos.
11. Haga clic en **Save**.
12. Para probar la función Lambda que acaba de crear, haga clic en **Test** en la esquina superior derecha.
13. Copie el contenido de su archivo `event.json` en el formulario **Configure test event**. Puede usar la plantilla de evento predeterminada &quot;Hello World&quot;.
14. Haga clic en **Create**.
15. Ejecute la prueba seleccionándola y haciendo clic en **Test**. Si la prueba se ejecutó correctamente, verá: &quot;Execution result: succeeded&quot;. Al expandir la ventana de salida, debería ver un mensaje similar al que recibió tras completar correctamente la prueba local.

<div id="configure-api-gateway-custom-authorizer">
  ### Configurar el autorizador personalizado de API Gateway
</div>

1. Vuelve a la consola de API Gateway y abre la API **PetStore** que creamos antes.
2. En el panel de navegación de la izquierda, abre **Authorizers** y selecciona **Create New Authorizer**. Luego, configura los siguientes parámetros y haz clic en **Create**.

   | Parámetro                | Valor                                                        |
   | ------------------------ | ------------------------------------------------------------ |
   | **Name**                 | `jwt-rsa-custom-authorizer`                                  |
   | **Type**                 | Selecciona **Lambda**                                        |
   | **Lambda Region**        | Usa la región de la función Lambda que creaste anteriormente |
   | **Lambda Function**      | `jwtRsaCustomAuthorizer`                                     |
   | **Lambda Invoke Role**   | El ARN del rol de IAM que copiaste antes                     |
   | **Lambda Event Payload** | Selecciona **Token**                                         |
   | **Token Source**         | `Authorization`                                              |
   | **Token Validation**     | `^Bearer [-0-9a-zA-z\.]*$`                                   |
   | **TTL (seconds)**        | `3600`                                                       |
3. Una vez que AWS cree el autorizador y se actualice la página, pruébalo haciendo clic en **Test** e introduciendo el token de Auth0 (`Bearer ey...`) que usaste anteriormente.

<div id="configure-api-gateway-resources-to-use-the-custom-authorizer">
  ### Configura los recursos de API Gateway para usar el autorizador personalizado
</div>

1. Inicia sesión en AWS y ve a la [consola de API Gateway](http://console.aws.amazon.com/apigateway).

   <Callout icon="file-lines" color="#0EA5E9" iconType="regular">
     Los autorizadores personalizados se configuran método por método; si quieres proteger varios métodos con un solo autorizador, tendrás que repetir las siguientes instrucciones para cada uno.
   </Callout>
2. Abre la API de **PetStore** que creamos en el paso 2 de este tutorial. En el árbol de **Resource** del panel central, selecciona el método **GET** del recurso `/pets`.

   <Frame>![undefined](/docs/images/cdy7uua7fh8z/3Z9Gmy20NDr9Hb9Yk0piXF/0c7572fc188e13106d9340562580e5e0/pt3-2.png)</Frame>
3. Selecciona **Method Request**.
4. En **Settings**, haz clic en el icono de **lápiz** a la derecha de **Authorization** y elige el autorizador personalizado `jwt-rsa-custom-authorizer` que creaste en el paso 3.
5. Haz clic en el icono de **marca de verificación** para guardar la selección del autorizador personalizado. Asegúrate de que el campo **API Key Required** esté establecido en `false`.

### Despliega la API

Para hacer públicos tus cambios, despliega la API.

1. En el menú **Actions**, selecciona **Deploy API**.
2. Introduce los siguientes valores y haz clic en **Deploy**:

   | Parámetro                  | Valor                                                  |
   | -------------------------- | ------------------------------------------------------ |
   | **Deployment stage**       | Elige `[New Stage]`                                    |
   | **Stage name**             | Introduce un nombre para la etapa                      |
   | **Stage description**      | Introduce una descripción para la etapa                |
   | **Deployment description** | Introduce una descripción para el despliegue de la API |

Si el despliegue se realiza correctamente, se te redirigirá al **Test Stage Editor**. Anota la **Invoke URL** que aparece en la franja azul de la parte superior, ya que la necesitarás para probar el despliegue.
