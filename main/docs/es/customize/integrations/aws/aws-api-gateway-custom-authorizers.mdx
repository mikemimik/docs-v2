Protege los endpoints de AWS API Gateway con autorizadores personalizados que aceptan <Tooltip tip="Token de acceso: credencial de autorización, en forma de una cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=access+tokens">tokens de acceso</Tooltip> emitidos por Auth0. Para ello, configuras tu API con API Gateway, creas y configuras tus funciones de AWS Lambda (incluidos los autorizadores personalizados) para proteger los endpoints de tu API, e implementas el <Tooltip tip="Flujo de autorización: concesión de autorización (o flujo de trabajo) especificada en el framework OAuth 2.0." cta="Ver glosario" href="/docs/es/glossary?term=authorization+flow">flujo de autorización</Tooltip> para que tus usuarios puedan obtener de Auth0 los tokens de acceso necesarios para acceder a tu API.

Para obtener más información, visita [Información general de AWS Lambda](https://aws.amazon.com/lambda/) después de iniciar sesión en tu [AWS API Gateway](https://aws.amazon.com/api-gateway/).

API Gateway amplía las capacidades de Lambda al añadir una capa de servicio delante de tus funciones de Lambda para reforzar la seguridad, gestionar las transformaciones de los mensajes de entrada y salida, y ofrecer capacidades como limitación y auditoría. Un enfoque sin servidor simplifica tus necesidades operativas, ya que aspectos como el escalado y la tolerancia a fallos pasan a ser responsabilidad del servicio de cómputo que ejecuta tu código.

Los autorizadores personalizados:

* Confirmarán que el token de acceso se haya enviado mediante el encabezado `authorization` de la solicitud para acceder a la API.
* Verificarán la firma RS256 del token de acceso mediante una clave pública obtenida a través de un endpoint de JWKS.
* Se asegurarán de que el token de acceso incluya las claims `iss` de emisor y `aud` de <Tooltip tip="Audiencia: identificador único de la audiencia de un token emitido. Llamado aud en un token, su valor contiene el ID de una aplicación (ID de cliente) para un token de ID o de una API (identificador de API) para un token de acceso." cta="Ver glosario" href="/docs/es/glossary?term=audience">audiencia</Tooltip> requeridas.

<div id="how-api-gateway-custom-authorizers-work">
  ## Cómo funcionan los autorizadores personalizados de API Gateway
</div>

[Según Amazon](http://docs.aws.amazon.com/apigateway/latest/developerguide/use-custom-authorizer.html), un autorizador personalizado de API Gateway es una &quot;función de Lambda que usted proporciona para controlar el acceso a su API mediante estrategias de autenticación con bearer token, como <Tooltip tip="OAuth 2.0: Marco de autorización que define protocolos y flujos de trabajo de autorización." cta="Ver glosario" href="/docs/es/glossary?term=OAuth">OAuth</Tooltip> o <Tooltip tip="OAuth 2.0: Marco de autorización que define protocolos y flujos de trabajo de autorización." cta="Ver glosario" href="/docs/es/glossary?term=SAML">SAML</Tooltip>.&quot;

Siempre que alguien (o algún programa) intenta llamar a su API, API Gateway comprueba si hay un autorizador personalizado configurado para la API.

Si **hay un autorizador personalizado para la API**, API Gateway llama al autorizador personalizado y proporciona el token de autorización extraído del encabezado de la solicitud recibida.

Puede usar el autorizador personalizado para implementar distintos tipos de estrategias de autorización, incluida la verificación de JWT, y devolver políticas de IAM que autoricen la solicitud. Si la política devuelta no es válida o si se deniegan los permisos, la llamada a la API falla.

En el caso de una política válida, la API almacena en caché la política devuelta, la asocia con el token entrante y la usa para la solicitud actual y las posteriores. Puede configurar durante cuánto tiempo se almacena en caché la política. El valor predeterminado es `300` segundos y la duración máxima del almacenamiento en caché es de `3600` segundos (también puede establecer el valor en 0 para deshabilitar el almacenamiento en caché).

Para obtener más información, visite [¿Qué es Amazon API Gateway?](http://docs.aws.amazon.com/apigateway/latest/developerguide/use-custom-authorizer.html) en la Guía para desarrolladores de Amazon. Para obtener más información sobre la verificación de <Tooltip tip="JSON Web Token (JWT): Formato estándar de token de ID (y, a menudo, formato de token de acceso) utilizado para representar claims de forma segura entre dos partes." cta="Ver glosario" href="/docs/es/glossary?term=JWT">JWT</Tooltip>, consulte nuestro artículo sobre [JSON Web Token](/docs/es/secure/tokens/json-web-tokens).

<div id="create-the-custom-authorizers">
  ## Cree los autorizadores personalizados
</div>

Ahora que tenemos una API totalmente funcional administrada por API Gateway, proteja esta API para que solo quienes cuenten con la autorización adecuada puedan acceder al backend que hay detrás de la API.

Use los custom request authorizers de API Gateway para autorizar sus API mediante estrategias de autorización con bearer token, como OAuth 2.0 o SAML. Para cada solicitud entrante, ocurre lo siguiente:

1. API Gateway comprueba que haya un autorizador personalizado configurado correctamente.
2. API Gateway llama al autorizador personalizado (que es una Lambda Function) con el token de autorización.
3. Si el token de autorización es válido, el autorizador personalizado devuelve las políticas de AWS Identity and Access Management (IAM) correspondientes.
4. API Gateway usa las políticas devueltas en el paso 3 para autorizar la solicitud.

<div id="prepare-the-custom-authorizer">
  ### Prepara el autorizador personalizado
</div>

Puedes [descargar un autorizador personalizado de ejemplo](https://github.com/auth0-samples/jwt-rsa-aws-custom-authorizer) compatible con tokens emitidos por Auth0. Después, tendrás que personalizar los archivos para que el autorizador personalizado funcione en tu entorno.

1. Descomprime en la ubicación que prefieras la carpeta que contiene los archivos de ejemplo que descargaste antes y, desde la línea de comandos, ve a esa carpeta.
2. Dentro de la carpeta de ejemplo, ejecuta `npm install` para instalar los paquetes de Node.js necesarios para el despliegue; AWS requiere que estos archivos se incluyan en el paquete que subirás a AWS en un paso posterior.
3. Configura tu entorno local con un archivo `.env`. Puedes copiar el archivo `.env.sample` (y renombrarlo a `.env` al mismo tiempo) usando `cp .env.sample .env`. Realiza los siguientes cambios:

   | Parámetro          | Valor                                                                                                                            |
   | ------------------ | -------------------------------------------------------------------------------------------------------------------------------- |
   | **`TOKEN_ISSUER`** | El emisor del token. Si Auth0 es el emisor del token, usa `https://{yourDomain}/`. Asegúrate de incluir la barra diagonal final. |
   | **`JWKS_URI`**     | La URL del JWKS endpoint. Si Auth0 es el emisor del token, usa `https://{yourDomain}/.well-known/jwks.json`                      |
   | **`AUDIENCE`**     | El valor de **`identifier`** de la API que creaste en la sección Crear una API de Auth0 anterior.                                |

<div id="test-the-custom-authorizer-locally">
  ### Probar el autorizador personalizado en local
</div>

Obtén un JWT de token de acceso válido. Hay varias formas de conseguir uno, y el método que elijas dependerá del tipo de tu aplicación, del nivel de confianza o de la experiencia general del usuario final. 5. Ejecuta la prueba con `npm test`.

<div id="create-the-iam-role">
  ### Crear el rol de IAM
</div>

El rol de IAM tiene los permisos necesarios para invocar funciones Lambda; antes de poder continuar con nuestro custom authorizer, tendremos que crear un rol de IAM que pueda invocar nuestro custom authorizer cada vez que API Gateway reciba una solicitud de acceso.

1. Inicia sesión en AWS y ve a la [IAM Console](https://console.aws.amazon.com/iam). En el menú de navegación de la izquierda, selecciona **Roles**.

2. Selecciona **Create new role**.

3. En **AWS service**, selecciona la fila **AWS Lambda** y luego **Next: Permissions**.

4. En la pantalla **Attach permissions policy**, selecciona **AWSLambdaRole**. Puedes usar el filtro disponible para reducir la lista de opciones. Selecciona **Next: Tags** y luego **Next: Review** para continuar.

5. En la pantalla **Review**, escribe un **Role name**, como `Auth0Integration`. Deja el resto de los campos como están. Selecciona **Create role**.

6. Cuando AWS haya creado tu rol, se te redirigirá de nuevo a la página **Roles** de IAM. Selecciona tu nuevo rol.

7. En la página **Summary** del rol que acabas de crear, selecciona la vista **Trust relationships**.

8. Selecciona **Edit trust relationship** y completa el campo **Policy Document** con el siguiente fragmento JSON:

   ```json lines
   {
      "Version": "2012-10-17",
      "Statement": [
         {
            "Effect": "Allow",
            "Principal": {
               "Service": [
                  "apigateway.amazonaws.com",
                  "lambda.amazonaws.com"
               ]
            },
            "Action": "sts:AssumeRole"
         }
      ]
   }
   ```

9. Haz clic en **Update Trust Policy**.

10. Se te redirigirá de nuevo a la página **Summary**. Copia el valor de **Role ARN** para usarlo más adelante.

<div id="create-the-lambda-function-and-deploy-the-custom-authorizer">
  ### Crea la función Lambda y despliega el autorizador personalizado
</div>

Ahora que has configurado tu autorizador personalizado para tu entorno y lo has probado para verificar que funciona, desplégalo en AWS.

1. Crea un paquete que puedas cargar en AWS ejecutando `npm run bundle`. Esto genera un archivo `custom-authorizer.zip` que contiene el código fuente, la configuración y los módulos de Node.js que requiere AWS Lambda.
2. Ve a la [consola de Lambda](https://console.aws.amazon.com/lambda) y haz clic en **Create function**.
3. En la página **Select blueprint**, haz clic en **Author from scratch** para crear una función vacía. En **Basic information**, proporciona valores para los siguientes parámetros:

   | Parámetro       | Valor                                                           |
   | --------------- | --------------------------------------------------------------- |
   | **Name**        | Un nombre para tu función Lambda, como `jwtRsaCustomAuthorizer` |
   | **Description** | Una descripción para tu función Lambda (opcional)               |
   | **Runtime**     | Selecciona `Node.js 10.x`                                       |
4. Haz clic en **Create Function** para continuar.
5. En la página **Configuration** de tu función, desplázate hacia abajo hasta la sección **Function Code**.
6. Selecciona **Upload a .ZIP file** como **Code entry type**.
7. Haz clic en **Upload** y selecciona el archivo `custom-authorizer.zip` que creaste antes.
8. Luego, crea las siguientes tres **Environment variables**. Ten en cuenta que esta información es idéntica a la del archivo `.env`.

   | Parámetro          | Valor                                                                                                       |
   | ------------------ | ----------------------------------------------------------------------------------------------------------- |
   | **`TOKEN_ISSUER`** | El emisor del token. Si Auth0 es el emisor del token, usa `https://{yourDomain}/`                           |
   | **`JWKS_URI`**     | La URL del endpoint JWKS. Si Auth0 es el emisor del token, usa `https://{yourDomain}/.well-known/jwks.json` |
   | **`AUDIENCE`**     | El valor de **identifier** de la API que creaste en el paso 1.                                              |
9. En la sección **Execution role**, selecciona **Use an existing role** y luego selecciona el rol de IAM que creaste anteriormente como **Existing role**.
10. En **Basic settings**, configura **Timeout** en **30** segundos.
11. Haz clic en **Save**.
12. Para probar la función Lambda que acabas de crear, haz clic en **Test** en la esquina superior derecha.
13. Copia el contenido de tu archivo `event.json` en el formulario **Configure test event**. Puedes usar la plantilla de evento predeterminada &quot;Hello World&quot;.
14. Haz clic en **Create**.
15. Ejecuta la prueba seleccionándola y haciendo clic en **Test**. Si la prueba se ejecutó correctamente, verás: &quot;Execution result: succeeded&quot;. Al expandir la ventana de salida, deberías ver un mensaje similar al que recibiste después de ejecutar correctamente la prueba local.

<div id="configure-api-gateway-custom-authorizer">
  ### Configurar el autorizador personalizado de API Gateway
</div>

1. Regresa a API Gateway Console y abre la API de **PetStore** que creamos antes.
2. En el menú de navegación de la izquierda, abre **Authorizers** y selecciona **Create New Authorizer**. Luego, establece los siguientes parámetros y haz clic en **Create**.

   | Parámetro                | Valor                                                        |
   | ------------------------ | ------------------------------------------------------------ |
   | **Name**                 | `jwt-rsa-custom-authorizer`                                  |
   | **Type**                 | Selecciona **Lambda**                                        |
   | **Lambda Region**        | Usa la región de la función Lambda que creaste anteriormente |
   | **Lambda Function**      | `jwtRsaCustomAuthorizer`                                     |
   | **Lambda Invoke Role**   | El Role ARN del rol de IAM que copiaste arriba               |
   | **Lambda Event Payload** | Selecciona **Token**                                         |
   | **Token Source**         | `Authorization`                                              |
   | **Token Validation**     | `^Bearer [-0-9a-zA-z\.]*$`                                   |
   | **TTL (seconds)**        | `3600`                                                       |
3. Después de que AWS cree el autorizador personalizado y se actualice la página, pruébalo haciendo clic en **Test** e introduciendo el token de Auth0 (`Bearer ey...`) que usaste anteriormente.
