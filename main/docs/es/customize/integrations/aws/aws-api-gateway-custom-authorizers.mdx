Protege los endpoints de AWS API Gateway con autorizadores personalizados que aceptan <Tooltip tip="Token de acceso: credencial de autorización, en forma de una cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=access+tokens">tokens de acceso</Tooltip> emitidos por Auth0. Para ello, configuras tu API con API Gateway, creas y configuras tus funciones de AWS Lambda (incluidos los autorizadores personalizados) para proteger los endpoints de tu API, e implementas el <Tooltip tip="Flujo de autorización: concesión de autorización (o workflow) especificada en el framework OAuth 2.0." cta="Ver glosario" href="/docs/es/glossary?term=authorization+flow">flujo de autorización</Tooltip> para que tus usuarios puedan obtener de Auth0 los tokens de acceso necesarios para acceder a tu API.

Para obtener más información, visita [Información general de AWS Lambda](https://aws.amazon.com/lambda/) después de iniciar sesión en [AWS API Gateway](https://aws.amazon.com/api-gateway/).

API Gateway amplía las capacidades de Lambda al agregar una capa de servicio delante de tus funciones de Lambda para reforzar la seguridad, gestionar las transformaciones de mensajes de entrada y salida, y ofrecer funciones como limitación y auditoría. Un enfoque sin servidor simplifica tus necesidades operativas, ya que aspectos como el escalado y la tolerancia a fallos pasan a ser responsabilidad del servicio de cómputo que ejecuta tu código.

Los autorizadores personalizados harán lo siguiente:

* Confirmarán que el token de acceso se haya enviado a través del encabezado `authorization` de la solicitud para acceder a la API.
* Verificarán la firma RS256 del token de acceso usando una clave pública obtenida a través de un endpoint de JWKS.
* Se asegurarán de que el token de acceso tenga las claims requeridas: Issuer `iss` y <Tooltip tip="Audiencia: identificador único de la audiencia de un token emitido. Llamado aud en un token, su valor contiene el ID de una aplicación (ID de cliente) para un token de ID o de una API (identificador de API) para un token de acceso." cta="Ver glosario" href="/docs/es/glossary?term=audience">audiencia</Tooltip> `aud`.

<div id="how-api-gateway-custom-authorizers-work">
  ## Cómo funcionan los autorizadores personalizados de API Gateway
</div>

[Según Amazon](http://docs.aws.amazon.com/apigateway/latest/developerguide/use-custom-authorizer.html), un autorizador personalizado de API Gateway es una &quot;función Lambda que proporcionas para controlar el acceso a tu API mediante estrategias de autenticación con bearer token, como <Tooltip tip="OAuth 2.0: Marco de autorización que define protocolos y flujos de trabajo de autorización." cta="Ver glosario" href="/docs/es/glossary?term=OAuth">OAuth</Tooltip> o <Tooltip tip="OAuth 2.0: Marco de autorización que define protocolos y flujos de trabajo de autorización." cta="Ver glosario" href="/docs/es/glossary?term=SAML">SAML</Tooltip>.&quot;

Cada vez que alguien (o algún programa) intenta llamar a tu API, API Gateway comprueba si hay un autorizador personalizado configurado para la API.

Si **hay un autorizador personalizado para la API**, API Gateway llama al autorizador personalizado y le proporciona el token de autorización extraído del encabezado de la solicitud recibida.

Puedes usar el autorizador personalizado para implementar distintos tipos de estrategias de autorización, incluida la verificación de JWT, y devolver políticas de IAM que autoricen la solicitud. Si la política devuelta no es válida o si se deniegan los permisos, la llamada a la API falla.

Para una política válida, la API almacena en caché la política devuelta, la asocia con el token entrante y la usa para las solicitudes actuales y posteriores. Puedes configurar durante cuánto tiempo se almacena en caché la política. El valor predeterminado es `300` segundos y la duración máxima del almacenamiento en caché es `3600` segundos (también puedes establecer el valor en 0 para desactivar el almacenamiento en caché).

Para leer más, visita [What is Amazon API Gateway?](http://docs.aws.amazon.com/apigateway/latest/developerguide/use-custom-authorizer.html) en la Guía para desarrolladores de Amazon. Para obtener más información sobre la verificación de <Tooltip tip="JSON Web Token (JWT): Formato estándar de token de ID (y, con frecuencia, formato de Access Token) utilizado para representar claims de forma segura entre dos partes." cta="Ver glosario" href="/docs/es/glossary?term=JWT">JWT</Tooltip>, consulta nuestro artículo sobre [JSON Web Token](/docs/es/secure/tokens/json-web-tokens).

<div id="create-the-custom-authorizers">
  ## Crear los autorizadores personalizados
</div>

Ahora que tenemos una API completamente funcional administrada por API Gateway, protégela para que solo quienes cuenten con la autorización adecuada puedan acceder al back-end de la API.

Usa los autorizadores personalizados de solicitud de API Gateway para autorizar tus API mediante estrategias de autorización con bearer token, como OAuth 2.0 o SAML. Para cada solicitud entrante, ocurre lo siguiente:

1. API Gateway comprueba que haya un autorizador personalizado configurado correctamente.
2. API Gateway llama al autorizador personalizado (que es una función Lambda) con el token de autorización.
3. Si el token de autorización es válido, el autorizador personalizado devuelve las políticas adecuadas de AWS Identity and Access Management (IAM).
4. API Gateway usa las políticas devueltas en el paso 3 para autorizar la solicitud.

<div id="prepare-the-custom-authorizer">
  ### Prepare el autorizador personalizado
</div>

Puede [descargar un autorizador personalizado de ejemplo](https://github.com/auth0-samples/jwt-rsa-aws-custom-authorizer) compatible con tokens emitidos por Auth0. Después, deberá personalizar los archivos para que el autorizador personalizado funcione en su entorno.

1. Descomprima la carpeta que contiene los archivos de ejemplo que descargó anteriormente en la ubicación que prefiera y vaya a esa carpeta desde la línea de comandos.
2. Dentro de la carpeta de ejemplo, ejecute `npm install` para instalar los paquetes de Node.js necesarios para el despliegue; AWS exige que estos archivos se incluyan en el paquete que cargará en AWS en un paso posterior.
3. Configure su entorno local con un archivo `.env`. Puede copiar el archivo `.env.sample` (renombrándolo al mismo tiempo como `.env`) con `cp .env.sample .env`. Realice los siguientes cambios:

   | Parámetro          | Valor                                                                                                                            |
   | ------------------ | -------------------------------------------------------------------------------------------------------------------------------- |
   | **`TOKEN_ISSUER`** | El emisor del token. Si Auth0 es el emisor del token, use `https://{yourDomain}/`. Asegúrese de incluir la barra diagonal final. |
   | **`JWKS_URI`**     | La URL del endpoint de JWKS. Si Auth0 es el emisor del token, use `https://{yourDomain}/.well-known/jwks.json`                  |
   | **`AUDIENCE`**     | El valor de **`identifier`** de la API que creó en la sección Create an Auth0 API anterior.                                      |

<div id="test-the-custom-authorizer-locally">
  ### Prueba el autorizador personalizado localmente
</div>

Obtén un token de acceso JWT válido. Hay varias formas de obtener uno, y el método que elijas dependerá del tipo de tu aplicación, el nivel de confianza o la experiencia general del usuario final. 1. Puedes obtener un token de prueba para tu API en [Auth0 Dashboard &gt; Applications &gt; APIs](https://manage.auth0.com/#/apis): selecciona tu API y luego **Test**.
2. Crea un archivo local `event.json` que contenga el token. Puedes copiar el archivo de ejemplo (ejecuta `cp event.json.sample event.json`). Sustituye `ACCESS_TOKEN` por tu token JWT y `methodArn` por el valor ARN correspondiente al método `GET` de tu API.

Para obtener el `methodArn`:

1. En la consola de API Gateway, abre la API **PetStore**.
2. En el panel de navegación izquierdo, selecciona **Resources**.
3. En el panel central **Resources**, expande el árbol de recursos. Debajo de `/pets`, selecciona **GET**.
4. En el cuadro **Method Request**, verás el **ARN**.
5. Ejecuta la prueba con `npm test`.

<div id="create-the-iam-role">
  ### Cree el rol de IAM
</div>

El rol de IAM tiene los permisos necesarios para invocar funciones Lambda; antes de continuar con nuestro autorizador personalizado, tendremos que crear un rol de IAM que pueda llamar a nuestro autorizador personalizado cada vez que API Gateway reciba una solicitud de acceso.

1. Inicie sesión en AWS y vaya a la [IAM Console](https://console.aws.amazon.com/iam). En el panel de navegación de la izquierda, seleccione **Roles**.

2. Seleccione **Create new role**.

3. En **AWS service**, seleccione la fila **AWS Lambda** y, a continuación, **Next: Permissions**.

4. En la pantalla **Attach permissions policy**, seleccione **AWSLambdaRole**. Puede usar el filtro disponible para reducir la lista de opciones. Seleccione **Next: Tags** y, después, **Next: Review** para continuar.

5. En la pantalla **Review**, indique un **Role name**, como `Auth0Integration`. Deje el resto de los campos tal como están. Seleccione **Create role**.

6. Una vez que AWS haya creado el rol, se le redirigirá a la página **Roles** de IAM. Seleccione su nuevo rol.

7. En la página **Summary** del rol que acaba de crear, seleccione la vista **Trust relationships**.

8. Seleccione **Edit trust relationship** y rellene el campo **Policy Document** con el siguiente fragmento de JSON:

   ```json lines
   {
      "Version": "2012-10-17",
      "Statement": [
         {
            "Effect": "Allow",
            "Principal": {
               "Service": [
                  "apigateway.amazonaws.com",
                  "lambda.amazonaws.com"
               ]
            },
            "Action": "sts:AssumeRole"
         }
      ]
   }
   ```

9. Haga clic en **Update Trust Policy**.

10. Se le redirigirá de nuevo a la página **Summary**. Copie el valor de **Role ARN** para usarlo más adelante.

<div id="create-the-lambda-function-and-deploy-the-custom-authorizer">
  ### Crea la función Lambda e implementa el autorizador personalizado
</div>

Ahora que has configurado tu autorizador personalizado para tu entorno y lo has probado para comprobar que funciona, implémentalo en AWS.

1. Crea un paquete que puedas cargar en AWS ejecutando `npm run bundle`. Esto genera un paquete `custom-authorizer.zip` que contiene el código fuente, la configuración y los módulos de Node.js que requiere AWS Lambda.
2. Ve a la [consola de Lambda](https://console.aws.amazon.com/lambda) y haz clic en **Create function**.
3. En la página **Select blueprint**, haz clic en **Author from scratch** para crear una función vacía. En **Basic information**, proporciona valores para los siguientes parámetros:

   | Parámetro       | Valor                                                           |
   | --------------- | --------------------------------------------------------------- |
   | **Name**        | Un nombre para tu función Lambda, como `jwtRsaCustomAuthorizer` |
   | **Description** | Una descripción para tu función Lambda (opcional)               |
   | **Runtime**     | Selecciona `Node.js 10.x`                                       |
4. Haz clic en **Create Function** para continuar.
5. En la página **Configuration** de tu función, desplázate hacia abajo hasta la sección **Function Code**.
6. Selecciona **Upload a .ZIP file** como **Code entry type**.
7. Haz clic en **Upload** y selecciona el paquete `custom-authorizer.zip` que creaste antes.
8. Luego, crea las siguientes tres **variables de entorno**. Ten en cuenta que esta información es idéntica a la del archivo `.env`.

   | Parámetro          | Valor                                                                                                          |
   | ------------------ | -------------------------------------------------------------------------------------------------------------- |
   | **`TOKEN_ISSUER`** | El emisor del token. Si Auth0 es el emisor del token, usa `https://{yourDomain}/`                              |
   | **`JWKS_URI`**     | La URL del endpoint de JWKS. Si Auth0 es el emisor del token, usa `https://{yourDomain}/.well-known/jwks.json` |
   | **`AUDIENCE`**     | El valor de **identifier** de la API que creaste en el paso 1.                                                 |
9. En la sección **Execution role**, selecciona **Use an existing role** y luego selecciona el rol de IAM que creaste anteriormente como **Existing role**.
10. En **Basic settings**, establece **Timeout** en **30** segundos.
11. Haz clic en **Save**.
12. Para probar la función Lambda que acabas de crear, haz clic en **Test** en la esquina superior derecha.
13. Copia el contenido de tu archivo `event.json` en el formulario **Configure test event**. Puedes usar la plantilla de evento predeterminada &quot;Hello World&quot;.
14. Haz clic en **Create**.
15. Ejecuta la prueba seleccionándola y haciendo clic en **Test**. Si la prueba se ejecutó correctamente, verás: &quot;Execution result: succeeded&quot;. Al expandir la ventana de salida, debería mostrarse un mensaje similar al que recibiste después de tu prueba local exitosa.

<div id="configure-api-gateway-custom-authorizer">
  ### Configurar el autorizador personalizado de API Gateway
</div>

1. Vuelve a consola de API Gateway y abre la API de **PetStore** que creamos antes.
2. En el panel de navegación izquierdo, abre **Authorizers** y selecciona **Create New Authorizer**. Luego, establece los siguientes parámetros y haz clic en **Create**.

   | Parámetro                | Valor                                                        |
   | ------------------------ | ------------------------------------------------------------ |
   | **Name**                 | `jwt-rsa-custom-authorizer`                                  |
   | **Type**                 | Selecciona **Lambda**                                        |
   | **Lambda Region**        | Usa la región de la función Lambda que creaste anteriormente |
   | **Lambda Function**      | `jwtRsaCustomAuthorizer`                                     |
   | **Lambda Invoke Role**   | El Role ARN del rol de IAM que copiaste antes                |
   | **Lambda Event Payload** | Selecciona **Token**                                         |
   | **Token Source**         | `Authorization`                                              |
   | **Token Validation**     | `^Bearer [-0-9a-zA-z\.]*$`                                   |
   | **TTL (seconds)**        | `3600`                                                       |
3. Después de que AWS cree el autorizador y la página se actualice, prueba el autorizador haciendo clic en **Test** e introduciendo el token de Auth0 (`Bearer ey...`) que usaste anteriormente.
   Si la prueba se completó correctamente, verás una respuesta similar a la siguiente.

  ### Configura los recursos de API Gateway para usar el autorizador personalizado
</div>

1. Inicia sesión en AWS y ve a la [consola de API Gateway](http://console.aws.amazon.com/apigateway).

   <Callout icon="file-lines" color="#0EA5E9" iconType="regular">
     Los autorizadores personalizados se configuran método por método; si quieres proteger varios métodos con un solo autorizador personalizado, tendrás que repetir las siguientes instrucciones para cada método.
   </Callout>
2. Abre la API **PetStore** que creamos en el paso 2 de este tutorial. En el árbol de **Resource** del panel central, selecciona el método **GET** del recurso `/pets`.

3. Selecciona **Method Request**.
4. En **Settings**, haz clic en el icono de **lápiz** a la derecha de **Authorization** y elige el autorizador personalizado `jwt-rsa-custom-authorizer` que creaste en el paso 3.
5. Haz clic en el icono de **marca de verificación** para guardar el autorizador personalizado seleccionado. Asegúrate de que el campo **API Key Required** esté configurado como `false`.

   | Parámetro                            | Valor                                                   |
   | ------------------------------------ | ------------------------------------------------------- |
   | **Etapa de implementación**          | Elige `[New Stage]`                                     |

   | **Descripción de la etapa**          | Indica una descripción para tu etapa                    |
   | **Descripción de la implementación** | Indica una descripción para la implementación de tu API |

Si todo sale bien, se te redirigirá a **Test Stage Editor**. Toma nota de la **Invoke URL** que aparece en la franja azul de la parte superior, ya que la necesitarás para probar tu implementación.
