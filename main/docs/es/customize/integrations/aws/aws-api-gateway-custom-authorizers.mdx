Protege los endpoints de API de AWS API Gateway con autorizadores personalizados que aceptan <Tooltip tip="Token de acceso: Credencial de autorización, en forma de una cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=access+tokens">tokens de acceso</Tooltip> emitidos por Auth0. Para ello, configuras tu API con API Gateway, creas y configuras tus funciones de AWS Lambda (incluidos los autorizadores personalizados) para proteger los endpoints de tu API, e implementas el <Tooltip tip="Flujo de autorización: Concesión de autorización (o workflow) especificada en el framework OAuth 2.0." cta="Ver glosario" href="/docs/es/glossary?term=authorization+flow">flujo de autorización</Tooltip> para que tus usuarios puedan obtener de Auth0 los tokens de acceso necesarios para acceder a tu API.

Para obtener más información, visita [Información general de AWS Lambda](https://aws.amazon.com/lambda/) una vez que inicies sesión en [AWS API Gateway](https://aws.amazon.com/api-gateway/).

API Gateway amplía las capacidades de Lambda al añadir una capa de servicio delante de tus funciones de Lambda para reforzar la seguridad, gestionar las transformaciones de los mensajes de entrada y salida, y ofrecer capacidades como limitación y auditoría. Un enfoque sin servidor simplifica tus necesidades operativas, ya que aspectos como el escalado horizontal y la tolerancia a fallos pasan a ser responsabilidad del servicio de computación que ejecuta tu código.

Los autorizadores personalizados:

* Confirmarán que el token de acceso se haya enviado a través del encabezado `authorization` de la solicitud para acceder a la API.
* Verificarán la firma RS256 del token de acceso mediante una clave pública obtenida a través de un endpoint de JWKS.
* Garantizarán que el token de acceso tenga los claims obligatorios `iss` del emisor y `aud` de <Tooltip tip="Audience: Identificador único de la audiencia de un token emitido. Llamado aud en un token, su valor contiene el ID de una aplicación (ID de cliente) para un token de ID o de una API (identificador de API) para un token de acceso." cta="Ver glosario" href="/docs/es/glossary?term=audience">audiencia</Tooltip>.

<div id="how-api-gateway-custom-authorizers-work">
  ## Cómo funcionan los autorizadores personalizados de API Gateway
</div>

[Según Amazon](http://docs.aws.amazon.com/apigateway/latest/developerguide/use-custom-authorizer.html), un autorizador personalizado de API Gateway es una &quot;función Lambda que usted proporciona para controlar el acceso a su API mediante estrategias de autenticación con bearer token, como <Tooltip tip="OAuth 2.0: Framework de autorización que define protocolos y flujos de trabajo de autorización." cta="Ver glosario" href="/docs/es/glossary?term=OAuth">OAuth</Tooltip> o <Tooltip tip="OAuth 2.0: Framework de autorización que define protocolos y flujos de trabajo de autorización." cta="Ver glosario" href="/docs/es/glossary?term=SAML">SAML</Tooltip>.&quot;

Siempre que alguien (o algún programa) intenta llamar a su API, API Gateway comprueba si hay un autorizador personalizado configurado para la API.

Si **hay un autorizador personalizado para la API**, API Gateway llama al autorizador personalizado y proporciona el token de autorización extraído del encabezado de la solicitud recibida.

Puede usar el autorizador personalizado para implementar distintos tipos de estrategias de autorización, incluida la verificación de JWT, para devolver políticas de IAM que autoricen la solicitud. Si la política devuelta no es válida o si se deniegan los permisos, la llamada a la API falla.

En el caso de una política válida, la API almacena en caché la política devuelta, la asocia con el token entrante y la usa para la solicitud actual y las posteriores. Puede configurar durante cuánto tiempo se almacena en caché la política. El valor predeterminado es `300` segundos y la duración máxima del almacenamiento en caché es `3600` segundos (también puede establecer el valor en 0 para deshabilitar el almacenamiento en caché).

Para leer más, visite [¿Qué es Amazon API Gateway?](http://docs.aws.amazon.com/apigateway/latest/developerguide/use-custom-authorizer.html) en la Guía para desarrolladores de Amazon. Para obtener más información sobre la verificación de <Tooltip tip="JSON Web Token (JWT): Formato estándar de token de ID (y a menudo formato de Access Token) utilizado para representar claims de forma segura entre dos partes." cta="Ver glosario" href="/docs/es/glossary?term=JWT">JWT</Tooltip>, revise nuestro artículo sobre [JSON Web Token](/docs/es/secure/tokens/json-web-tokens).

   | Algoritmo de firma | El algoritmo que desea que Auth0 use para firmar el [token de acceso](/docs/es/glossary?term=access+token) emitido. Para obtener más información, consulte [Algoritmos de firma](/docs/es/get-started/applications/signing-algorithms). |

<div id="create-the-custom-authorizers">
  ## Cree los autorizadores personalizados
</div>

Ahora que tenemos una API totalmente funcional administrada por API Gateway, proteja esta API para que solo quienes tengan la autorización adecuada puedan acceder al back-end que hay detrás de la API.

Use los custom request autorizadores personalizados de API Gateway para autorizar sus API mediante estrategias de autorización con bearer token, como OAuth 2.0 o SAML. Para cada solicitud entrante, ocurre lo siguiente:

1. API Gateway comprueba que haya un autorizador personalizado configurado correctamente.
2. API Gateway llama al autorizador personalizado (que es una función Lambda) con el token de autorización.
3. Si el token de autorización es válido, el autorizador personalizado devuelve las políticas adecuadas de AWS Identity and Access Management (IAM).
4. API Gateway usa las políticas devueltas en el paso 3 para autorizar la solicitud.

<div id="prepare-the-custom-authorizer">
  ### Preparar el autorizador personalizado
</div>

Puede [descargar un autorizador personalizado de ejemplo](https://github.com/auth0-samples/jwt-rsa-aws-custom-authorizer) compatible con tokens emitidos por Auth0. Después, deberá personalizar los archivos para que el autorizador personalizado funcione en su entorno.

1. Descomprima la carpeta que contiene los archivos de ejemplo que descargó anteriormente en la ubicación que prefiera y vaya a esa carpeta desde la línea de comandos.
2. Dentro de la carpeta de ejemplo, ejecute `npm install` para instalar los paquetes de Node.js necesarios para el despliegue; AWS requiere que estos archivos se incluyan en el paquete que cargará en AWS en un paso posterior.
3. Configure su entorno local con un archivo `.env`. Puede copiar el archivo `.env.sample` (y al mismo tiempo cambiarle el nombre a `.env`) con `cp .env.sample .env`. Realice los siguientes cambios:

   | Parámetro          | Valor                                                                                                                            |
   | ------------------ | -------------------------------------------------------------------------------------------------------------------------------- |
   | **`TOKEN_ISSUER`** | El issuer del token. Si Auth0 es el issuer del token, use `https://{yourDomain}/`. Asegúrese de incluir la barra diagonal final. |
   | **`JWKS_URI`**     | La URL del endpoint de JWKS. Si Auth0 es el issuer del token, use `https://{yourDomain}/.well-known/jwks.json`                      |
   | **`AUDIENCE`**     | El valor de **identifier** de la API que creó en la sección Create an Auth0 API anterior.                                        |

<div id="test-the-custom-authorizer-locally">
  ### Probar el autorizador personalizado localmente
</div>

Obtén un token de acceso JWT válido. Hay varias formas de conseguirlo, y el método que elijas dependerá del tipo de tu aplicación, del nivel de confianza o de la experiencia general del usuario final. Para obtener más información, consulta [Get Access Tokens](/docs/es/secure/tokens/access-tokens/get-access-tokens).

1. Puedes obtener un token de prueba para tu API yendo a [Auth0 Dashboard &gt; Applications &gt; APIs](https://manage.auth0.com/#/apis), seleccionando tu API y haciendo clic en **Test**.
2. Crea un archivo local `event.json` que contenga el token. Puedes copiar el archivo de ejemplo (ejecuta `cp event.json.sample event.json`). Reemplaza `ACCESS_TOKEN` por tu token JWT y `methodArn` por el valor de ARN correspondiente al método `GET` de tu API.

Para obtener el `methodArn`:

1. En la API Gateway Console, abre la API **PetStore**.
2. En el panel de navegación de la izquierda, selecciona **Resources**.
3. En el panel central **Resources**, expande el árbol de recursos. Debajo de `/pets`, selecciona **GET**.
4. En el cuadro **Method Request**, verás el **ARN**.
5. Ejecuta la prueba con `npm test`.

La prueba usa el paquete lambda-local para probar el autorizador personalizado con tu token. Si la prueba se realizó correctamente, verás una salida similar a la siguiente:

        "Version": "2012-10-17",

<div id="create-the-iam-role">
  ### Crear el rol de IAM
</div>

El rol de IAM tiene los permisos necesarios para invocar funciones de Lambda; antes de poder continuar con nuestro custom authorizer, tendremos que crear un rol de IAM que pueda llamar a nuestro custom authorizer cada vez que API Gateway reciba una solicitud de acceso.

1. Inicia sesión en AWS y ve a la [IAM Console](https://console.aws.amazon.com/iam). En la navegación de la izquierda, selecciona **Roles**.

2. Selecciona **Create new role**.

3. En **AWS service**, selecciona la fila **AWS Lambda** y, a continuación, **Next: Permissions**.

4. En la pantalla **Attach permissions policy**, selecciona **AWSLambdaRole**. Puedes usar el filtro proporcionado para acotar la lista de opciones. Selecciona **Next: Tags** y luego **Next: Review** para continuar.

5. En la pantalla **Review**, proporciona un **Role name**, como `Auth0Integration`. Deja el resto de los campos como están. Selecciona **Create role**.

6. Una vez que AWS haya creado tu rol, volverás a la página **Roles** de IAM. Selecciona tu nuevo rol.

7. En la página **Summary** del rol que acabas de crear, selecciona la vista **Trust relationships**.

8. Selecciona **Edit trust relationship** y completa el campo **Policy Document** con el siguiente fragmento de JSON:

   ```json lines
   {
      "Version": "2012-10-17",
      "Statement": [
         {
            "Effect": "Allow",
            "Principal": {
               "Service": [
                  "apigateway.amazonaws.com",
                  "lambda.amazonaws.com"
               ]
            },
            "Action": "sts:AssumeRole"
         }
      ]
   }
   ```

9. Haz clic en **Update Trust Policy**.

10. Se te redirigirá de nuevo a la página **Summary**. Copia el valor de **Role ARN** para usarlo más adelante.

<div id="create-the-lambda-function-and-deploy-the-custom-authorizer">
  ### Crea la función Lambda y despliega el autorizador personalizado
</div>

Ahora que has configurado tu autorizador personalizado para tu entorno y lo has probado para comprobar que funciona, desplíegalo en AWS.

1. Crea un paquete que puedas subir a AWS ejecutando `npm run bundle`. Esto genera un paquete `custom-authorizer.zip` que contiene el código fuente, la configuración y los módulos de Node.js que requiere AWS Lambda.
2. Ve a la [consola de Lambda](https://console.aws.amazon.com/lambda) y haz clic en **Crear función**.
3. En la página **Seleccionar blueprint**, haz clic en **Crear desde cero** para crear una función vacía. En **Información básica**, proporciona valores para los siguientes parámetros:

   | Parámetro       | Valor                                                           |
   | --------------- | --------------------------------------------------------------- |
   | **Nombre**      | Un nombre para tu función Lambda, como `jwtRsaCustomAuthorizer` |
   | **Descripción** | Una descripción para tu función Lambda (opcional)               |
   | **Runtime**     | Selecciona `Node.js 10.x`                                       |
4. Haz clic en **Crear función** para continuar.
5. En la página **Configuración** de tu función, desplázate hacia abajo hasta la sección **Código de la función**.
6. Selecciona **Subir un archivo .ZIP** como **Tipo de entrada de código**.
7. Haz clic en **Subir** y selecciona el paquete `custom-authorizer.zip` que creaste antes.
8. Luego, crea las siguientes tres **Variables de entorno**. Ten en cuenta que esta información es idéntica a la del archivo `.env`.

   | Parámetro          | Valor                                                                                                       |
   | ------------------ | ----------------------------------------------------------------------------------------------------------- |
   | **`TOKEN_ISSUER`** | El issuer del token. Si Auth0 es el issuer del token, usa `https://{yourDomain}/`                           |
   | **`JWKS_URI`**     | La URL del endpoint de JWKS. Si Auth0 es el issuer del token, usa `https://{yourDomain}/.well-known/jwks.json` |
   | **`AUDIENCE`**     | El valor de **identifier** de la API que creaste en el paso 1.                                              |
9. En la sección **Rol de ejecución**, selecciona **Usar un rol existente** y luego selecciona el rol de IAM que creaste anteriormente como **Rol existente**.
10. En **Configuración básica**, establece **Timeout** en **30** segundos.
11. Haz clic en **Guardar**.
12. Para probar la función Lambda que acabas de crear, haz clic en **Test** en la esquina superior derecha.
13. Copia el contenido de tu archivo `event.json` en el formulario **Configurar evento de prueba**. Puedes usar la plantilla de evento predeterminada &quot;Hello World&quot;.
14. Haz clic en **Crear**.
15. Ejecuta la prueba seleccionándola y haciendo clic en **Test**. Si la prueba se realizó correctamente, verás: &quot;Execution result: succeeded&quot;. Al expandir la ventana de salida, debería mostrarse un mensaje similar al que recibiste después de que la prueba local se ejecutara correctamente.

<div id="configure-api-gateway-custom-authorizer">
  ### Configurar el autorizador personalizado de API Gateway
</div>

1. Vuelve a la API Gateway Console y abre la API **PetStore** que creamos antes.
2. En la navegación de la izquierda, abre **Authorizers** y selecciona **Create New Authorizer**. Luego, configura los siguientes parámetros y haz clic en **Create**.

   | Parámetro                        | Valor                                                        |
   | -------------------------------- | ------------------------------------------------------------ |
   | **Nombre**                       | `jwt-rsa-custom-authorizer`                                  |
   | **Tipo**                         | Selecciona **Lambda**                                        |
   | **Región de Lambda**             | Usa la región de la función Lambda que creaste anteriormente |
   | **Función Lambda**               | `jwtRsaCustomAuthorizer`                                     |
   | **Rol de invocación de Lambda**  | El ARN del rol de IAM que copiaste arriba                    |
   | **Payload del evento de Lambda** | Selecciona **Token**                                         |
   | **Origen del token**             | `Authorization`                                              |
   | **Validación de token**          | `^Bearer [-0-9a-zA-z\.]*$`                                   |
   | **TTL (segundos)**               | `3600`                                                       |
3. Cuando AWS cree el autorizador y la página se actualice, prueba el autorizador haciendo clic en **Test** e introduciendo el token de Auth0 (`Bearer ey...`) que usaste antes.

<div id="configure-api-gateway-resources-to-use-the-custom-authorizer">
  ### Configurar recursos de API Gateway para usar el autorizador personalizado
</div>

1. Inicia sesión en AWS y ve a la [API Gateway Console](http://console.aws.amazon.com/apigateway).

   <Callout icon="file-lines" color="#0EA5E9" iconType="regular">
     Los autorizadores personalizados se configuran método por método; si quieres proteger varios métodos con un solo authorizer, tendrás que repetir las siguientes instrucciones para cada método.
   </Callout>
2. Abre la API **PetStore** que creamos en el paso 2 de este tutorial. En el árbol de **Resource** del panel central, selecciona el método **GET** del recurso `/pets`.

   <Frame>![undefined](/docs/images/cdy7uua7fh8z/3Z9Gmy20NDr9Hb9Yk0piXF/0c7572fc188e13106d9340562580e5e0/pt3-2.png)</Frame>
3. Selecciona **Method Request**.
4. En **Settings**, haz clic en el icono de **lápiz** a la derecha de **Authorization** y elige el autorizador personalizado `jwt-rsa-custom-authorizer` que creaste en el paso 3.
5. Haz clic en el icono de **marca de verificación** para guardar tu autorizador personalizado seleccionado. Asegúrate de que el campo **API Key Required** esté configurado como `false`.

   | Parámetro                  | Valor                                                   |
   | -------------------------- | ------------------------------------------------------- |
   | **Deployment stage**       | Elige `[New Stage]`                                     |
   | **Stage name**             | Indica un nombre para la etapa                          |

   | **Deployment description** | Indica una descripción para la implementación de la API |

Si todo sale bien, se te redirigirá a **Test Stage Editor**. Toma nota de la **Invoke URL** que aparece en la franja azul de la parte superior, ya que la necesitarás para probar la implementación.
