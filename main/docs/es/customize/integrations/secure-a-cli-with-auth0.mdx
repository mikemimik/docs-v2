---
description: Cómo usar Auth0 para proteger una CLI.
title: Protege una CLI con Auth0
---

Las tres formas de proteger una CLI con Auth0, ordenadas de la más segura a la menos segura, son:

* [flujo de autorización de dispositivos](#device-authorization-flow) para cuando el usuario no puede abrir un navegador
* [flujo de concesión de credenciales de cliente](#client-credentials-grant-flow) para aplicaciones que actúan por cuenta propia y no pueden atribuirse a un usuario
* [flujo de Resource Owner Password Grant](#resource-owner-password-grant-flow) solo cuando intentas autenticar al propio cliente de la CLI, una situación muy poco frecuente (por lo demás, no se recomienda)

<div id="device-authorization-flow">
  ## Flujo de autorización de dispositivos
</div>

En los dispositivos con limitaciones de entrada que se conectan a internet, en lugar de autenticar directamente al usuario, el dispositivo le pide que vaya a un enlace desde su computadora o teléfono inteligente y autorice el dispositivo. Esto evita una mala experiencia de usuario en dispositivos que no cuentan con una forma sencilla de introducir texto. Para ello, las aplicaciones del dispositivo usan el <Tooltip tip="Flujo de autorización: concesión de autorización (o flujo de trabajo) especificada en el marco de OAuth 2.0." cta="Ver glosario" href="/docs/es/glossary?term=Authorization+Flow">Flujo de autorización</Tooltip> de dispositivos (definido en OAuth 2.0), en el que envían su <Tooltip tip="ID de cliente: valor de identificación asignado a su recurso registrado por Auth0." cta="Ver glosario" href="/docs/es/glossary?term=Client+ID">ID de cliente</Tooltip> para iniciar el proceso de autorización y obtener un token.

La forma más sencilla de implementar el Flujo de autorización de dispositivos es seguir los pasos de [Llamar a la API usando el Flujo de autorización de dispositivos](/docs/es/get-started/authentication-and-authorization-flow/device-authorization-flow/call-your-api-using-the-device-authorization-flow).

Para obtener más información sobre el Flujo de autorización de dispositivos en <Tooltip tip="OAuth 2.0: marco de autorización que define protocolos y flujos de trabajo de autorización." cta="Ver glosario" href="/docs/es/glossary?term=OAuth+2.0">OAuth 2.0</Tooltip>, puede consultar el borrador de Internet Engineering Task Force (IEFT) [OAuth 2.0 Authorization Grant](https://tools.ietf.org/html/draft-ietf-oauth-device-flow-15). También puede consultar nuestro artículo [Flujo de autorización de dispositivos](/docs/es/get-started/authentication-and-authorization-flow/device-authorization-flow).

<div id="client-credentials-grant-flow">
  ## Flujo de concesión de credenciales de cliente
</div>

Usa el flujo de concesión de credenciales de cliente (CCG) cuando no intervienen ni los usuarios ni los <Tooltip tip="Proveedor de identidad (IdP): servicio que almacena y gestiona identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=identity+providers">proveedores de identidad</Tooltip> posteriores, y quieres autenticarte en función de máquinas o dispositivos específicos.

Si tu proveedor de identidad admite el envío de credenciales, te recomendamos revisar nuestro artículo [Flujo de credenciales de cliente](/docs/es/get-started/authentication-and-authorization-flow/client-credentials-flow). Para obtener más información sobre cómo implementar este flujo, consulta [Llamar a la API usando el flujo de credenciales de cliente](/docs/es/get-started/authentication-and-authorization-flow/client-credentials-flow/call-your-api-using-the-client-credentials-flow).

<div id="resource-owner-password-grant-flow">
  ## Flujo de Resource Owner Password Grant
</div>

No recomendamos usar el flujo de <Tooltip tip="Resource Owner: Entidad (como un usuario o una aplicación) capaz de otorgar acceso a un recurso protegido." cta="Ver glosario" href="/docs/es/glossary?term=Resource+Owner">Resource Owner</Tooltip> Password Grant (ROPG) para aplicaciones nativas. En el artículo de IEFT, [RFC 8252 OAuth 2.0 para aplicaciones nativas](https://tools.ietf.org/html/rfc8252), se recomienda que “las solicitudes de autorización de OAuth 2.0 desde aplicaciones nativas SOLO se realicen a través de agentes de usuario externos, principalmente el navegador del usuario”. Para más detalles, consulta [RFC 8252 Embedded User-Agents](https://tools.ietf.org/html/rfc8252#section-8.12).

Usar Resource Owner Password Grant (ROPG) es menos seguro que las opciones basadas en redirección descritas anteriormente. ROPG es solo para escenarios heredados. En el contexto de las CLI, solo tiene sentido en casos como las cadenas de conexión, donde necesitas admitir programas heredados.

Si debes usar ROPG en tu aplicación nativa en lugar de Device Flow, como recomendamos, entonces puedes usar nuestro [endpoint de ROPG compatible con OIDC](https://auth0.com/docs/api/authentication#resource-owner-password).
