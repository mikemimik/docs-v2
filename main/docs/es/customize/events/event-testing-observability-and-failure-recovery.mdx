En las siguientes secciones se explica cómo probar los Event Streams y gestionar los eventos.

<div id="test-the-event-stream">
  ## Probar el Event Stream
</div>

Después de crear y habilitar un Event Stream, puedes probarlo. Actualmente, Auth0 no admite enviar eventos a Event Streams deshabilitados.

Para probar tu Event Stream, primero crea un usuario para tu aplicación ejecutando lo siguiente con las credenciales que prefieras:

<Tabs>
  <Tab title="Auth0 CLI">
    ```bash lines
    auth0 users create # sigue las instrucciones para crear un usuario de prueba
    ```
  </Tab>

  <Tab title="cURL">
    ```bash lines
    curl -s \
      -H "Authorization: Bearer $TOKEN" \
      -H "Content-Type: application/json" \
      -X POST https://$TENANT/api/v2/users \
      -d '{
            "email": "nick-testing-21@example.com",
            "password": "Password123!",
            "connection": "Username-Password-Authentication"
          }'
    ```
  </Tab>
</Tabs>

Una vez creado el usuario, debería activar el evento `user.created` en tu Event Stream y devolver una respuesta. En este ejemplo, se usan webhooks para enviar el Event Stream.

```json lines expandable

  "id": "evt_6SfwXrtm8nVcjzzNCFFH1L",
  "source": "urn:auth0:",
  "specversion": "1.0",
  "type": "user.created",
  "time": "2025-02-02T17:56:01.573Z",
  "data": {
    "object": {
      "created_at": "2025-02-02T17:56:01.566Z",
      "email": "nick-testing-21@example.com",
      "email_verified": false,
      "identities": [
        {
          "connection": "Username-Password-Authentication",
          "isSocial": false,
          "provider": "auth0",
          "user_id": "679fb1b1e809445f129bc18d"
        }
      ],
      "name": "nick-testing-21@example.com",
      "nickname": "nick-testing-21",
      "picture": "test.png",
      "updated_at": "2025-02-02T17:56:01.566Z",
      "user_id": "auth0|679fb1b1e809445f129bc18d"
    }
  },
  "a0tenant": "acme",
  "a0stream": "est_38ANnv8pQhqsZrX2VFbftv",
  "a0purpose": "test"
}
```

Puedes consultar los errores de entrega de un Event Stream específico ejecutando:

<Tabs>
  <Tab title="Auth0 CLI">
    ```bash lines
    auth0 api get event-streams/$EVENT_STREAM_ID/deliveries
    ```
  </Tab>

  <Tab title="cURL">
    ```bash lines
    curl -s -H "Authorization: Bearer $TOKEN" \
      "https://$TENANT/api/v2/event-streams/${EVENT_STREAM_ID}/deliveries" | jq .
    ```
  </Tab>
</Tabs>

Si se devuelve un arreglo vacío de errores de entrega, tu Event Stream funciona correctamente.

<div id="observability">
  ## observabilidad
</div>

Para garantizar la fiabilidad de los Event Streams, establezca un proceso de monitoreo que compruebe periódicamente si hay errores de entrega y active alertas cuando sea necesario.

Los errores de entrega pueden deberse a lo siguiente:

* Event Streams deshabilitados.
* Event Streams no disponibles.
* Errores de conexión.

<div id="polling-deliveries-endpoint">
  ### Consulta periódica del endpoint Deliveries
</div>

Consulta el endpoint Deliveries cada **5 minutos** para comprobar si hay eventos fallidos. Usa la siguiente llamada a la API para obtener las entregas de eventos recientes:

<Tabs>
  <Tab title="Auth0 CLI">
    ```bash lines
    auth0 api get event-streams/$EVENT_STREAM_ID/deliveries
    ```
  </Tab>

  <Tab title="cURL">
    ```bash lines
    curl -s -H "Authorization: Bearer $TOKEN" \
      "https://$TENANT/api/v2/event-streams/${EVENT_STREAM_ID}/deliveries" | jq .
    ```
  </Tab>
</Tabs>

Si se detectan fallos:

* Registra los detalles del evento fallido.
* Avisa al equipo o sistema correspondiente para revisar y diagnosticar el problema.

<div id="recovery">
  ## Recuperación
</div>

La entrega de eventos se reintentará automáticamente. Para obtener más información, consulta [Prácticas recomendadas](/docs/es/customize/events/event-testing-observability-and-failure-recovery#batch-redelivery-of-failed-events).

Si se producen fallos, usa la Redelivery API para intentar entregar los eventos de nuevo.

<div id="redeliver-a-single-failed-event">
  ### Volver a entregar un único evento fallido
</div>

Para volver a intentar la entrega de un evento fallido específico, usa:

<Tabs>
  <Tab title="Auth0 CLI">
    ```bash lines
    EVENT_ID=<your event id>
    auth0 api POST event-streams/$EVENT_STREAM_ID/redeliver/$EVENT_ID --data '{}'
    ```
  </Tab>

  <Tab title="cURL">
    ```bash lines
    curl -s -X POST -H "Authorization: Bearer $TOKEN" \
      "https://<your-domain>/api/v2/event-streams/$EVENT_STREAM_ID/redeliver/$EVENT_ID"
    ```
  </Tab>
</Tabs>

<div id="batch-redelivery-of-failed-events">
  ### Reenvío por lotes de eventos fallidos
</div>

Para volver a intentar todos los eventos fallidos del Event Stream:

<Tabs>
  <Tab title="Auth0 CLI">
    ```bash wrap lines
    auth0 api POST event-streams/$EVENT_STREAM_ID/redeliver --data '{}'
    ```
  </Tab>

  <Tab title="cURL">
    ```bash lines
    curl -s -X POST -H "Authorization: Bearer $TOKEN" \
      "https://<your-tenant>.auth0.com/api/v2/event-streams/<event_stream_id>/redeliver"
    ```
  </Tab>
</Tabs>

<div id="auto-disable-mechanism">
  ### Mecanismo de desactivación automática
</div>

Auth0 desactiva automáticamente un Event Stream si se produce un número excesivo de fallos.

| Condición               | Acción realizada                                  |
| ----------------------- | ------------------------------------------------- |
| 500 fallos consecutivos | El Event Stream se desactiva automáticamente. |
| 5000 fallos en total    | El Event Stream se desactiva automáticamente. |

* Si un Event Stream se desactiva automáticamente, **debes** usar la Redelivery API para procesar cualquier evento fallido.
* El Event Stream **no puede** volver a habilitarse manualmente hasta que el número total de fallos sea inferior a 5000.
* Una vez reducido el número de fallos, puedes volver a habilitar el flujo mediante la API.
