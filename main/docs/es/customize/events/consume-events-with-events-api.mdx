---
description: Extrae eventos de Auth0 cuando tú decidas con la Events API basada en Server-Sent Events (SSE).
title: Consumir eventos con la Events API
---

La Events API ofrece una alternativa basada en extracción a Event Streams. En lugar de que Auth0 envíe eventos a un destino, tu aplicación abre una conexión de larga duración a `GET /api/v2/events` y recibe eventos como un flujo de Server-Sent Events (SSE). Tú decides cuándo conectarte, cómo reanudar la conexión tras una desconexión y a qué velocidad consumir los eventos.

Este enfoque resulta útil cuando necesitas:

* Procesar eventos a tu propio ritmo sin tener que implementar un endpoint de webhook.
* Reproducir eventos desde un momento específico para relleno histórico o recuperación.
* Integrarte con sistemas que prefieren el sondeo en lugar de la entrega mediante push.

<div id="how-the-events-api-works">
  ## Cómo funciona la Events API
</div>

Cuando tu aplicación se conecta a la Events API, recibe un flujo de mensajes SSE. Cada mensaje incluye un campo `id` que actúa como **offset**. Si la conexión se interrumpe, tu aplicación se vuelve a conectar y proporciona el último offset que recibió. Auth0 reanuda la entrega desde ese punto, por lo que no se pierde ningún evento.

El flujo SSE incluye los siguientes tipos de mensajes:

| Tipo de mensaje                                   | Finalidad                                                                                                                                     |
| ------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------- |
| `:connected`                                      | Confirma que la conexión se ha establecido. Es un comentario de SSE, no un evento de datos.                                                   |
| `retry: <ms>`                                     | Indica al cliente SSE cuánto tiempo debe esperar antes de volver a conectarse tras una desconexión.                                           |
| `event: <type>` (por ejemplo, `user.created`)     | Un evento real con el payload completo en el campo `data`.                                                                                    |
| `event: offset-only`                              | Un marcador de progreso enviado a intervalos regulares (con la frecuencia de heartbeat). Actualiza el offset sin entregar datos del evento.   |
| `:` seguido de texto (por ejemplo, `: heartbeat`) | Un comentario de keep-alive que evita que los proxies y los equilibradores de carga cierren conexiones inactivas. No requiere ninguna acción. |
| `event: error`                                    | Un error terminal. El flujo se cierra después de este mensaje.                                                                                |

<div id="example-sse-stream">
  ### Ejemplo de flujo SSE
</div>

```text wrap lines
:connected

retry: 2000

event: user.created
id: MTIzNDIzNDEzCg==
data: {"offset":"MTIzNDIzNDEzCg==","event":{"id":"evt_abc123","type":"user.created","time":"2025-06-01T12:00:00Z","data":{"object":{"user_id":"auth0|123","email":"jane@example.com"}}}}

event: offset-only
id: 4LcuTXmVDASuNRQt
data: {"offset":"4LcuTXmVDASuNRQt"}

: heartbeat

event: user.updated
id: NTY3ODkwMTIzCg==
data: {"offset":"NTY3ODkwMTIzCg==","event":{"id":"evt_def456","type":"user.updated","time":"2025-06-01T12:05:00Z","data":{"object":{"user_id":"auth0|123","email":"jane.doe@example.com"}}}}
```

<div id="prerequisites">
  ## Requisitos previos
</div>

Antes de comenzar, asegúrate de tener lo siguiente:

* Un tenant de Auth0 con Events habilitado. La cantidad de conexiones de Event Stream disponibles depende de tu plan:

  | Plan         | Límite de conexiones |
  | ------------ | -------------------- |
  | Free         | 1                    |
  | Self-service | 4                    |
  | Empresarial  | 8                    |

* Un token de acceso de la Management API con el scope `read:events`. Para obtener más información, consulta [Tokens de acceso de la Management API](/docs/es/secure/tokens/access-tokens/management-api-access-tokens).

<div id="connect-to-the-events-api">
  ## Conectarse a la Events API
</div>

Abra una conexión SSE con el endpoint de eventos de su tenant. El siguiente ejemplo usa `curl`:

```bash wrap lines
curl -N --http2 \
    -H "Authorization: Bearer YOUR_MANAGEMENT_API_TOKEN" \
    -H "Accept: text/event-stream" \
    "https://YOUR_DOMAIN/api/v2/events"
```

<div id="query-parameters">
  ### Parámetros de consulta
</div>

Usa parámetros de consulta para filtrar o reanudar el flujo:

| Parámetro        | Tipo   | Descripción                                                                                                                                                                                                                                                                                                                                            |
| ---------------- | ------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `from`           | string | Un offset devuelto en un campo `id` anterior. La entrega se reanuda después de este offset.                                                                                                                                                                                                                                                            |
| `from_timestamp` | string | Una marca de tiempo en formato ISO 8601. Devuelve los eventos ocurridos en ese momento o después. Es mutuamente excluyente con `from`. Se usa sobre todo para la configuración inicial o para volver a reproducir eventos desde un momento conocido; para el consumo continuo, es preferible reanudar con `from`, porque los offsets son más precisos. |
| `event_type`     | string | El tipo de evento que se debe incluir. Repite el parámetro para cada tipo (por ejemplo, `event_type=user.created&event_type=user.updated`).                                                                                                                                                                                                            |

```bash wrap lines
curl -N --http2 \
    -H "Authorization: Bearer YOUR_MANAGEMENT_API_TOKEN" \
    -H "Accept: text/event-stream" \
    "https://YOUR_DOMAIN/api/v2/events?event_type=user.created&event_type=user.updated&from_timestamp=2025-06-01T00:00:00Z"
```

<div id="resume-after-a-disconnection">
  ## Reanudar después de una desconexión
</div>

Las conexiones SSE pueden interrumpirse por muchos motivos: problemas de red, expiración del token o rotación de conexiones del lado del servidor (Auth0 cierra periódicamente las conexiones para distribuir la carga, normalmente cada pocos minutos). Las bibliotecas cliente estándar de SSE gestionan esto de forma transparente, ya que se reconectan y envían el último offset.

Hay dos formas de proporcionar el offset al volver a conectarse:

* **`Last-Event-ID` header** — el mecanismo estándar de reconexión de SSE. La mayoría de las bibliotecas cliente de SSE establecen este encabezado automáticamente al reconectarse.
* **`from` query parameter** — úsalo cuando tu cliente no admita el encabezado `Last-Event-ID`.

Si se proporcionan ambos, el encabezado `Last-Event-ID` tiene prioridad.

```bash wrap lines
curl -N --http2 \
    -H "Authorization: Bearer YOUR_MANAGEMENT_API_TOKEN" \
    -H "Accept: text/event-stream" \
    -H "Last-Event-ID: MTIzNDIzNDEzCg==" \
    "https://YOUR_DOMAIN/api/v2/events"
```

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Guarda de forma persistente el valor de `id` más reciente de cada mensaje (incluidos los mensajes `offset-only`) en un almacenamiento duradero. Si tu aplicación se reinicia, usa el offset guardado para reanudar la entrega desde donde la dejaste.
</Callout>

<div id="handle-message-types">
  ## Administrar tipos de mensajes
</div>

<div id="real-events">
  ### Eventos reales
</div>

Los mensajes cuyo campo `event` coincide con un tipo de evento conocido (por ejemplo, `user.created`) contienen el payload completo del evento en el campo `data`. Analiza el JSON y procesa el evento según tu lógica de negocio.

<div id="offset-only-messages">
  ### Mensajes solo de offset
</div>

Auth0 envía mensajes `offset-only` a intervalos regulares (según la frecuencia de heartbeat) para hacer avanzar tu posición en el flujo. Estos mensajes no contienen un payload de evento. Actualiza el offset que tienes guardado cuando los recibas para que una reconexión futura no vuelva a reproducir eventos que ya has dejado atrás.

<div id="error-messages">
  ### Mensajes de error
</div>

Un mensaje `event: error` indica un problema definitivo, como un offset vencido o un problema del servidor. Después de recibir este mensaje, el flujo se cierra. Tu aplicación debe registrar el error y volver a conectarse con el offset adecuado o con un `from_timestamp` nuevo.

<div id="heartbeats">
  ### Heartbeats
</div>

Las líneas que comienzan con `:` son comentarios de SSE que se usan como heartbeats. Mantienen la conexión activa a través de proxies y equilibradores de carga. No requieren procesamiento.

<div id="server-side-connection-cycling">
  ## Rotación de conexiones del lado del servidor
</div>

Auth0 cierra periódicamente las conexiones SSE por motivos de balanceo de carga (normalmente, cada pocos minutos). Este es un comportamiento esperado, no un error. Las bibliotecas cliente SSE estándar (incluido el paquete de npm `eventsource`) se reconectan automáticamente usando el encabezado `Last-Event-ID`, por lo que tu aplicación retoma el procesamiento desde el offset correcto sin perder eventos.

Si creas un cliente SSE personalizado, asegúrate de que gestione las caídas de conexión correctamente, guardando el último offset y volviéndose a conectar con él.

<div id="implement-a-consumer">
  ## Implementar un consumidor
</div>

El siguiente ejemplo de Node.js muestra un consumidor mínimo de Events API que procesa eventos y guarda el offset en un archivo.

```javascript wrap lines
const EventSource = require("eventsource");
const fs = require("fs");

const OFFSET_FILE = "./offset.txt";
const AUTH0_DOMAIN = "YOUR_DOMAIN";
const TOKEN = "YOUR_MANAGEMENT_API_TOKEN";

function loadOffset() {
    try {
        return fs.readFileSync(OFFSET_FILE, "utf8").trim();
    } catch {
        return null;
    }
}

function saveOffset(offset) {
    fs.writeFileSync(OFFSET_FILE, offset);
}

function connect() {
    const offset = loadOffset();
    const params = new URLSearchParams();
    params.append("event_type", "user.created");
    params.append("event_type", "user.updated");
    params.append("event_type", "user.deleted");

    if (offset) {
        params.append("from", offset);
    }

    const url = `https://${AUTH0_DOMAIN}/api/v2/events?${params.toString()}`;

    const es = new EventSource(url, {
        headers: {
            "Authorization": `Bearer ${TOKEN}`
        }
    });

    // Procesar eventos reales
    for (const type of ["user.created", "user.updated", "user.deleted"]) {
        es.addEventListener(type, (msg) => {
            const payload = JSON.parse(msg.data);
            console.log(`Received ${type}:`, payload.event.id);

            // Procesar el evento aquí

            saveOffset(msg.lastEventId);
        });
    }

    // Procesar marcadores de progreso de solo offset
    es.addEventListener("offset-only", (msg) => {
        saveOffset(msg.lastEventId);
    });

    // Gestionar errores terminales
    es.addEventListener("error", (msg) => {
        if (msg.data) {
            const errorPayload = JSON.parse(msg.data);
            console.error("Stream error:", errorPayload.error);
        }
        es.close();

        // Reconectar tras un retraso
        setTimeout(connect, 5000);
    });
}

connect();
```

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  El paquete de npm `eventsource` implementa el protocolo SSE y gestiona la reconexión automáticamente mediante el encabezado `Last-Event-ID`. Si usa una biblioteca SSE diferente, verifique que admita la reconexión automática y el reenvío del offset.
</Callout>

<div id="error-responses">
  ## Respuestas de error
</div>

La Events API devuelve códigos de estado HTTP estándar cuando no se puede establecer la conexión:

| Código de estado | Significado                                                                                                            |
| ---------------- | ---------------------------------------------------------------------------------------------------------------------- |
| `200`            | Conexión establecida. Comienza la transmisión de eventos.                                                              |
| `400`            | Solicitud no válida. El valor de offset tiene un formato incorrecto o el tipo de evento solicitado no se admite.       |
| `401`            | Falta el token de acceso o no es válido.                                                                               |
| `403`            | El token de acceso no incluye el `read:events` scope.                                                                  |
| `410`            | El offset ha expirado. Usa un valor `from_timestamp` para reanudar desde un punto específico en el tiempo.             |
| `429`            | Se ha superado el límite de frecuencia. Espera y vuelve a intentarlo usando el intervalo de la cabecera `Retry-After`. |
