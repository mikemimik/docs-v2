title: Prácticas recomendadas para Events
---

Para obtener los mejores resultados al desarrollar flujos de eventos, Auth0 recomienda seguir las prácticas recomendadas que se indican a continuación. Estas pautas pueden ayudar a reducir la fricción, aumentar la eficiencia y ofrecer una experiencia de eventos de primer nivel.

<div id="use-asynchronous-queues-to-process-events">
  ### Usa colas asíncronas para procesar eventos
</div>

Al recibir un evento, responde con un HTTP 2XX OK (p. ej., 200OK, 202 Accepted, 204 No Content) para indicar que el evento se entregó correctamente. Sin esta respuesta, Auth0 considerará la entrega del evento como un fallo y volverá a intentarlo de acuerdo con el comportamiento descrito en [Comportamiento del sistema durante las entregas fallidas de eventos](#system-behavior-during-failed-event-deliveries).

Para evitar problemas de escalabilidad, configura tu controlador para que procese los eventos entrantes con una cola asíncrona. Si procesas los eventos de forma síncrona, cualquier aumento repentino en las entregas de webhook (por ejemplo, a principios de mes, cuando se renuevan todas las suscripciones) puede saturar los servidores que alojan tu endpoint. Las colas asíncronas te permiten procesar eventos simultáneos a un ritmo que tu sistema pueda soportar.

<div id="ignore-duplicate-events">
  ### Ignora los eventos duplicados
</div>

Los endpoints de webhook pueden recibir en ocasiones el mismo evento más de una vez. Para evitar procesar eventos duplicados, registra los IDs de los eventos que ya hayas procesado.

<div id="handle-out-of-sequence-events">
  ### Gestionar eventos fuera de orden
</div>

Auth0 no garantiza que los eventos se entreguen en el mismo orden en que se generan. Por ejemplo, cuando se crea un usuario y luego se actualiza de inmediato para vincular otro usuario con él, es posible que reciba:

* `user.updated (for User 1)`
* `user.created (for User 1)`
* `user.deleted (for User 2)`
* `user.updated (for User 2)`

Para evitar sobrescribir sus datos con información desactualizada, compare la marca de tiempo de los datos entrantes del webhook con la marca de tiempo de los datos de su sistema. Cada `data.object` del payload incluye un campo `created_at` y un campo `updated_at`.

<div id="listen-for-specific-event-types">
  ### Escucha tipos de eventos específicos
</div>

Configura tus endpoints de webhook para que reciban solo los tipos de eventos que requiere tu integración. Escuchar eventos adicionales (o todos los eventos) supone una carga innecesaria para tu servidor y no se recomienda.

Puedes cambiar los eventos que recibe un endpoint de webhook [actualizando los Event Streams](/docs/es/customize/events/create-an-event-stream).

<div id="system-behavior-during-failed-event-deliveries">
  ### Comportamiento del sistema durante las entregas fallidas de eventos
</div>

Actualmente, Auth0 solo permite cuatro intentos en total, incluido el intento inicial y tres reintentos con retroceso exponencial (1s, 2s, 4s).  Si se recibe un 4XX, el evento falla de inmediato y no se vuelve a intentar, ya que es poco probable que reintentarlo cambie la respuesta.

Si su endpoint sigue sin responder satisfactoriamente después de este tiempo, el evento se marca como fallido y Auth0 dejará de intentar enviarlo. Para obtener más información sobre los eventos fallidos, consulte [Pruebas, observabilidad y recuperación ante fallos de flujos de eventos](/docs/es/customize/events/event-testing-observability-and-failure-recovery).

<div id="disable-failing-event-streams">
  ### Deshabilitar los Event Streams con errores
</div>

Los Event Streams se deshabilitan automáticamente en los siguientes casos:

* Auth0 recibe 500 errores seguidos (es decir, cualquier respuesta distinta de `2XX`).
* El Event Stream alcanza un total de 5000 entregas fallidas.
* Auth0 recibe tres fallos consecutivos en los intentos de entrega (es decir, cualquier respuesta distinta de `2XX`).

  * Si desea permitir más fallos en los intentos de entrega, envíe un ticket de soporte en el que indique la cantidad de fallos que desea permitir, y Auth0 revisará su solicitud.
