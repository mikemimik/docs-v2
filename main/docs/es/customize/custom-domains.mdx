Puede usar su propio nombre de dominio (también conocido como CNAME o vanity URL) en las páginas de autenticación. Un <Tooltip tip="Dominio personalizado: dominio de terceros con un nombre especializado o vanity." cta="Ver glosario" href="/docs/es/glossary?term=custom+domain">dominio personalizado</Tooltip> le permite unificar la experiencia de inicio de sesión con su propia marca y productos. Sus usuarios verán una URL que refleja su marca, como `login.YOUR_DOMAIN.com`, en lugar de `YOUR_DOMAIN.auth0.com.` El dominio personalizado en Auth0 actúa como una &quot;máscara&quot; para la URL del dominio de su tenant.

Puede configurar su dominio personalizado al crear su tenant o agregarlo a una implementación existente con cambios menores en el código y la configuración.

<div id="benefits-of-using-a-custom-domain">
  ## Beneficios de usar un dominio personalizado
</div>

Con un dominio personalizado, sus usuarios tienen la tranquilidad de que están proporcionando sus credenciales a la parte correcta. La autenticación se realiza dentro del contexto de su marca, lo que le ayuda a fortalecer la fidelidad a la marca. Los usuarios no son redirigidos a un sitio de terceros que rompe ese contexto de marca. Esto evita que se confundan sobre si todavía están realizando una transacción u operación con usted.

Concentrar sus servicios de autenticación en un solo lugar hace que la arquitectura de su aplicación sea más fácil de mantener. Las aplicaciones obtienen solo el acceso que necesitan y los servicios de autenticación escalan fácilmente. Otros beneficios de seguridad de usar un dominio personalizado incluyen:

* Algunos navegadores, de forma predeterminada, dificultan la comunicación dentro de un iFrame si no tiene un dominio compartido.
* Es más difícil hacer phishing de su dominio si tiene una URL personalizada, porque el atacante tendría que crear una URL personalizada para imitar la suya. Por ejemplo, con un dominio personalizado, puede usar su propio certificado para obtener una Validación Extendida, lo que dificulta el phishing.
* Las [claves de acceso](/docs/es/authenticate/database-connections/passkeys), una alternativa resistente al phishing a los factores de autenticación tradicionales, están vinculadas a su dominio personalizado mediante el atributo de ID de la parte solicitante `rpId`.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Antes de configurar su dominio personalizado, le recomendamos usar el dominio principal o raíz (por ejemplo, `yourDomain.com`) en lugar de un subdominio (`login.yourDomain.com`). Si su dominio personalizado en Auth0 está configurado como el dominio raíz, los usuarios finales pueden usar una sola clave de acceso para autenticarse en aplicaciones nativas o aplicaciones web móviles. Para obtener más información, consulte [Configurar la política de claves de acceso](/docs/es/authenticate/database-connections/passkeys/configure-passkey-policy#configure-relying-party-id-rpid).
</Callout>

<div id="how-it-works">
  ## Cómo funciona
</div>

Configura un dominio personalizado en la pestaña [Auth0 Dashboard &gt; Branding &gt; Custom Domains](https://manage.auth0.com/#/custom_domains) del <Tooltip tip="Auth0 Dashboard: el producto principal de Auth0 para configurar tus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip>. Añade tu dominio personalizado, elige el tipo de certificado y sigue las instrucciones. Completarás un proceso de verificación de tu dominio que varía según uses un certificado gestionado por Auth0 o autogestionado. Cuando crees un CNAME, debes declararlo en Auth0 para que Auth0 pueda verificarlo y usar el dominio personalizado. Después de configurar y verificar el dominio personalizado, debes [configurar las funciones de Auth0](/docs/es/customize/custom-domains/configure-features-to-use-custom-domains) para usar el nuevo dominio personalizado.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Para configurar un dominio personalizado gratuito, los tenants de Auth0 deben tener registrada una tarjeta de crédito válida con fines de verificación y prevención del fraude. No se realizará ningún cargo en la tarjeta.
</Callout>

Auth0 recomienda que crees tu dominio personalizado durante la fase de desarrollo (antes de pasar a producción) para asegurarte de haber configurado correctamente el CNAME. Por ejemplo, puedes crear un CNAME que apunte `login.YOUR_DOMAIN.com` a `YOUR_DOMAIN.auth0.com`.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Auth0 no proporciona una lista estática de direcciones IP porque están sujetas a cambios. En su lugar, te recomendamos que añadas tu dominio personalizado a la [lista de permitidos](/docs/es/secure/security-guidance/data-security/allowlist).
</Callout>

Puedes actualizar un tenant existente para usar un dominio personalizado. Tus integraciones existentes que usan `YOUR_DOMAIN.auth0.com` seguirán funcionando. Después del cambio, tus usuarios deberán iniciar sesión de nuevo porque las sesiones existentes dejarán de ser válidas. Además, es posible que los usuarios deban eliminar la cookie del navegador asociada a tu dominio personalizado si se producen errores durante el inicio de sesión. Si usas Lock integrado o un SDK, puedes optar por usar la configuración de dominio estándar o un dominio personalizado.

<Warning>
  Tu dominio personalizado debe ajustarse a las prácticas recomendadas de HTTP. Si el orden de los campos no es correcto, podrías enviar encabezados duplicados. Consulta [RFC 7230 HTTP/1.1 Message Syntax Routing - Field Order](https://tools.ietf.org/html/rfc7230#section-3.2.2) para más información.
</Warning>

| ------------------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Universal Login                                  | Para una experiencia de usuario fluida y segura                                                                                                                                                                              |
| MFA                                              | Todos los factores                                                                                                                                                                                                           |
| Guardian                                         | Android SDK/Swift SDK/MFA Widget versión 1.3.3/Guardian.js versión 1.3.0 o posterior                                                                                                                                         |
| Correos electrónicos                             | Los enlaces incluidos en los correos electrónicos usan tu dominio personalizado                                                                                                                                              |
| Conexiones                                       | Database, social, Google Workspace, Azure AD, ADFS, AD/LDAP                                                                                                                                                                  |
| [Lock](/docs/es/glossary?term=Lock)                 | Versión 11 con autenticación de origen cruzado                                                                                                                                                                               |
| [Passwordless](/docs/es/glossary?term=Passwordless) | Con Universal Login (El enlace del correo electrónico se envía con el dominio personalizado si la opción está habilitada en **Dashboard &gt; Configuración del tenant &gt; Dominios personalizados**.)                       |
| [SAML](/docs/es/glossary?term=SAML)                 | Conexiones y aplicaciones                                                                                                                                                                                                    |
| WS-Federation                                    | Auth0 como proveedor de identidad mediante el complemento WS-Fed                                                                                                                                                             |
| OAuth 2.0/OIDC-Compliant flows                   | Usando los endpoints [`/authorize`](https://auth0.com/docs/api/authentication/authorization-code-flow/authorize-application) y [`/oauth/token`](https://auth0.com/docs/api/authentication/authorization-code-flow/get-token) |

<div id="custom-domains-and-uris">
  ### Dominios personalizados y URI
</div>

Auth0 usa ciertos endpoints de metadatos para la interoperabilidad y la configuración de <Tooltip tip="Proveedor de identidad (IdP): servicio que almacena y administra identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=identity+providers">proveedores de identidad</Tooltip> y aplicaciones de terceros. Cuando los metadatos contienen URI que apuntan a Auth0, la URL puede ser el subdominio de Auth0 o tu dominio personalizado, según el nombre de host que hayas usado para solicitar los metadatos. Por ejemplo:

| Si usas                                        | Referencia dentro de los metadatos |
| ---------------------------------------------- | ---------------------------------- |
| `https://travel0.auth0.com/.well-known/...`    | `https://travel0.auth0.com/...`    |
| `https://travel0.auth0.com/samlp/metadata/...` | `https://travel0.auth0.com/...`    |
| `https://login.travel0.com/samlp/metadata/...` | `https://login.travel0.com/...`    |

Para obtener más información, consulta [Redirigir a los usuarios después de iniciar sesión](/docs/es/authenticate/login/redirect-users-after-login).

Esta flexibilidad se aplica a los siguientes escenarios de autenticación:

* [Configurar aplicaciones con OpenID Connect Discovery](/docs/es/get-started/applications/configure-applications-with-oidc-discovery)
* [Configurar Auth0 como proveedor de servicios SAML](/docs/es/authenticate/protocols/saml/saml-sso-integrations/configure-auth0-saml-service-provider)
* [Configurar Auth0 como proveedor de identidad SAML](/docs/es/authenticate/single-sign-on/outbound-single-sign-on/configure-auth0-saml-identity-provider)

<div id="custom-domains-and-token-requests">
  ### Dominios personalizados y solicitudes de token
</div>

Auth0 emite tokens con el claim `iss` correspondiente al dominio que usaste en la solicitud de token. Por ejemplo:

| Si usas                                                                                  | Valor del claim `iss`        |
| ---------------------------------------------------------------------------------------- | ---------------------------- |
| `https://travel0.auth0.com/authorize...`<br />`https://travel0.auth0.com/oauth/token...` | `https://travel0.auth0.com/` |
| `https://login.travel0.com/authorize...`<br />`https://login.travel0.com/oauth/token...` | `https://login.travel0.com/` |

Si obtienes un <Tooltip tip="Token de acceso: credencial de autorización, en forma de una cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=access+token">token de acceso</Tooltip> para la <Tooltip tip="Management API: un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip> mediante un <Tooltip tip="Management API: un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=authorization+flow">flujo de autorización</Tooltip> con tu dominio personalizado, debes llamar a la Management API usando ese dominio personalizado; de lo contrario, tu token se considerará inválido. El claim `iss` del token es independiente de la <Tooltip tip="Audience: identificador único de la audiencia de un token emitido. Se llama aud en un token; su valor contiene el ID de una aplicación (ID de cliente) para un token de ID o de una API (identificador de API) para un token de acceso." cta="Ver glosario" href="/docs/es/glossary?term=audience">audiencia</Tooltip>. Los valores de audiencia siguen siendo los mismos para los tokens obtenidos con un dominio personalizado. Para obtener más información sobre los tokens, consulta [Tokens de acceso de la Management API](/docs/es/secure/tokens/access-tokens/management-api-access-tokens).

<div id="certificate-management-options">
  ## Opciones de gestión de certificados
</div>

<div id="auth0-managed-certificates">
  ### Certificados gestionados por Auth0
</div>

Auth0 puede administrar los certificados de tu dominio personalizado y gestionar directamente el handshake de SSL. Añades un registro CNAME al dominio, Auth0 valida el registro y genera el certificado en los servidores de Auth0. El certificado se renueva automáticamente cada tres meses. Una vez verificado, [configura tus funcionalidades de Auth0](/docs/es/customize/custom-domains/configure-features-to-use-custom-domains) para empezar a usar tu dominio personalizado. Para obtener más información, consulta [Configurar dominios personalizados con certificados gestionados por Auth0](/docs/es/customize/custom-domains/auth0-managed-certificates).

<div id="self-managed-certificates">
  ### Certificados autogestionados
</div>

Puede obtener y [administrar sus propios certificados](/docs/es/customize/custom-domains/self-managed-certificates) en dominios personalizados. En este caso, usted es responsable de gestionar los certificados SSL, así como de configurar y administrar un proxy inverso para enviar contenido a Auth0. Auth0 negocia SSL con el proxy, no directamente con el cliente del usuario final. A su vez, el proxy negocia SSL con el usuario final. Para evitar que alguien intente usar su cuenta de Auth0 desde un dominio que no le pertenece, Auth0 necesita validar que el dominio le pertenece: debe proporcionar a Auth0 un encabezado (`cname-api-key`) para realizar la validación. Debe ser suscriptor de Auth0 Enterprise para usar esta opción.

Auth0 proporciona instrucciones para configurar un proxy inverso para los siguientes proveedores:

* [Google Cloud Platform con Load Balancing](/docs/es/customize/custom-domains/self-managed-certificates/configure-gcp-as-reverse-proxy)
* [Cloudflare](/docs/es/customize/custom-domains/self-managed-certificates/configure-cloudflare-for-use-as-reverse-proxy)
* [AWS CloudFront](/docs/es/customize/custom-domains/self-managed-certificates/configure-aws-cloudfront-for-use-as-reverse-proxy)
* [Azure CDN](/docs/es/customize/custom-domains/self-managed-certificates/configure-azure-cdn-for-use-as-reverse-proxy)
