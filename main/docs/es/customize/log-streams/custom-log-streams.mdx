description: Aprende a configurar un flujo de logs con un webhook personalizado para enviar eventos de log a la URL que elijas.
title: Crear flujos de logs personalizados con webhooks
---

Los webhooks permiten enviar eventos a un servidor web externo. Auth0 ofrece varias integraciones que envían automáticamente eventos a sistemas de gestión y análisis de logs de terceros. Sin embargo, si tu sistema de terceros no es compatible, puedes configurar una integración de webhook personalizada.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Solo puedes suscribir una URL del payload por cada configuración de webhook, pero puedes usar la misma URL para varios flujos. El payload se enviará en formato JSON lines. Tenlo en cuenta al consumir los logs en la configuración de tu webhook.
</Callout>

<div id="configure-api-endpoint">
  ## Configurar endpoint de API
</div>

Crearás una API que exponga una sola ruta (como `/api/logs`) y que acepte solicitudes `HTTP POST`.

<Warning>
  No se admiten certificados HTTP autofirmados.
</Warning>

<div id="configure-log-stream">
  ## Configurar flujo de logs
</div>

1. Ve a [Dashboard &gt; monitoreo &gt; Streams](https://manage.auth0.com/#/log-streams) y haz clic en **Crear flujo**.

   <Frame>![Dashboard Monitoring Streams](/docs/images/cdy7uua7fh8z/5ZY6sT68DabXNL9Dr4bhss/ff0f7306fcf9d64247a61a48b12a39a9/log_stream.png)</Frame>
2. Selecciona **Custom Webhook** e introduce un nombre único para tu nuevo flujo.
3. Configura los siguientes ajustes:

   <Frame>![Dashboard Monitoring Streams Custom Webhook Settings Tab](/docs/images/cdy7uua7fh8z/5vJVD09aqSF0xEZZFk6wh2/e6469041b2dd1e4f1b142961ca0a12ab/custom-webhook-settings.png)</Frame>

   | Ajuste                              | Descripción                                                                           |
   | ----------------------------------- | ------------------------------------------------------------------------------------- |
   | **Nombre**                          | Un nombre para mostrar único para distinguir esta integración de otras integraciones. |
   | **URL del payload**                 | La URL a la que se envían los payloads de eventos como solicitudes HTTP POST.         |
   | **Token de autorización**           | (Opcional) El valor del encabezado Authorization de la solicitud.                     |
   | **Tipo de contenido**               | El tipo de medio del payload que se enviará al webhook.                               |
   | **Formato del contenido**           | Recibe datos en JSON lines, arreglos u objetos.                                       |
   | **Filtrar por categoría de evento** | Lista de filtros del flujo de logs.                                                   |
   | **Cursor inicial**                  | (Opcional) Día y hora específicos a partir de los cuales iniciar el flujo.            |
4. Haz clic en **Guardar**.

<div id="test-configuration">
  ## Probar la configuración
</div>

1. Cambia a la vista **Health** y confirma que el **Estado del flujo** sea **Activo.**
2. Realiza una acción que cree un evento de log del tenant, como un inicio de sesión, un intercambio de tokens o una llamada a la Auth0 Management API.
3. Confirma que el evento se haya creado en los registros de tu tenant.
4. Confirma que tu API recibió una solicitud `HTTP POST` con los datos del evento.
5. Ve a la vista **Health** y confirma que no se haya producido ningún error.

<div id="troubleshoot-webhooks">
  ## Solucionar problemas con webhooks
</div>

Si tu webhook no funciona, puede ser difícil diagnosticarlo y determinar qué está causando el problema. Los webhooks son asíncronos, así que probarlos puede implicar activar el webhook, esperar y luego revisar la respuesta (suponiendo que hayas recibido una).

Sin embargo, hay alternativas al proceso ineficiente que describimos antes. Aunque explicar en detalle cómo solucionar problemas de un webhook en particular queda fuera del alcance de este artículo, aquí tienes algunos pasos que puedes seguir para depurarlo:

1. Revisa [Auth0 Dashboard &gt; Monitoreo &gt; Logs](/docs/es/deploy-monitor/logs) para encontrar mensajes útiles.
2. Analiza las solicitudes que hace tu webhook con una herramienta como [Mockbin](http://mockbin.io/), [Beeceptor](https://beeceptor.com/) o [RequestBin](https://github.com/RonTt/Requestbin) (autoalojado).
3. Simula solicitudes con cURL o [Postman](https://www.getpostman.com/).
4. Prueba el código de tu webhook localmente con herramientas como [localtunnel](https://github.com/localtunnel/localtunnel) o [ngrok](https://ngrok.com/).
5. Usa una herramienta como [Runscope](https://www.runscope.com/) o [Assertible](https://assertible.com/) para supervisar todo el flujo de ejecución.
