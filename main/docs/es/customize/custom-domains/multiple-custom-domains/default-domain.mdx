Cuando tienes [múltiples dominios personalizados](/docs/es/customize/custom-domains/multiple-custom-domains) configurados en tu tenant de Auth0, puedes designar uno como el **dominio personalizado predeterminado**. El dominio predeterminado simplifica la configuración y garantiza una experiencia de usuario consistente cuando no se especifica explícitamente un dominio personalizado.

<div id="what-is-a-default-custom-domain">
  ## ¿Qué es un dominio personalizado predeterminado?
</div>

El dominio personalizado predeterminado es el dominio personalizado que Auth0 usa automáticamente para:

* **Comunicaciones por correo electrónico y teléfono**: enviar correos electrónicos de restablecimiento de contraseña, enlaces de verificación de correo electrónico y otras notificaciones generadas por Auth0 cuando no se especifica un dominio personalizado concreto. Esto incluye los casos en que se envían notificaciones durante la autenticación.
* **Llamadas a la Management API**: procesar solicitudes a la API que activan notificaciones sin el encabezado `auth0-custom-domain`

Puedes configurar tu dominio canónico del tenant (`YOUR_TENANT.auth0.com` o `YOUR_TENANT.REGION.auth0.com`) como dominio personalizado predeterminado. Siempre habrá un dominio personalizado predeterminado configurado para tu tenant.

<div id="benefits-of-setting-a-default-domain">
  ## Beneficios de establecer un dominio predeterminado
</div>

Establecer un dominio personalizado predeterminado ofrece varias ventajas:

* **Configuración simplificada**: Reduce la necesidad de especificar dominios personalizados en cada llamada a la Management API o en cada configuración
* **Branding coherente**: Garantiza que los usuarios siempre vean tu marca, incluso cuando no se especifica un dominio concreto
* **Encabezado `auth0-custom-domain` opcional**: Hace que el encabezado de dominio personalizado sea opcional en las solicitudes a la Management API
* **Comportamiento de respaldo**: Proporciona valores predeterminados adecuados para implementaciones multimarca o multi-tenant
* **Migración más sencilla**: Simplifica la transición de un único dominio personalizado a múltiples dominios personalizados

<div id="configure-a-default-domain">
  ## Configurar un dominio predeterminado
</div>

Puedes establecer un dominio personalizado predeterminado desde el Auth0 Dashboard o mediante la Management API.

<div id="using-the-auth0-dashboard">
  ### Uso del Auth0 Dashboard
</div>

Para establecer un dominio personalizado predeterminado en el Auth0 Dashboard:

1. Ve a **Auth0 Dashboard** &gt; **Branding &gt; Custom Domains**
2. Busca en la lista el dominio personalizado verificado que quieras establecer como predeterminado
3. Haz clic en el botón **Set as Default** de ese dominio
4. Confirma la acción en el cuadro de diálogo que aparece

El dominio ahora mostrará una etiqueta &quot;Default&quot; en la lista de dominios personalizados.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Solo los dominios personalizados verificados pueden establecerse como predeterminados. Asegúrate de que tu dominio personalizado esté completamente verificado y activo antes de designarlo como predeterminado.
</Callout>

<div id="using-the-management-api">
  ### Uso de Management API
</div>

Para establecer un dominio personalizado predeterminado mediante Management API, usa el endpoint `PATCH /api/v2/custom-domains/default`:

<AuthCodeGroup>
  ```bash cURL
  curl --request PATCH \
    --url 'https://{yourDomain}/api/v2/custom-domains/default' \
    --header 'authorization: Bearer {yourMgmtApiAccessToken}' \
    --header 'content-type: application/json' \
    --data '{
      "domain": "my-custom-domain.com"
    }'
  ```

  ```javascript Node.js
  import { ManagementClient } from "auth0";

  const management = new ManagementClient({
    domain: '{yourDomain}',
    clientId: '{yourClientId}',
    clientSecret: '{yourClientSecret}',
  });

  await management.customDomains.setDefault({
    domain: 'my-custom-domain.com',
  });
  ```

  ```python Python
  from auth0.management import ManagementClient

  client = ManagementClient(
      domain='{yourDomain}',
      client_id='{yourClientId}',
      client_secret='{yourClientSecret}',
  )

  client.custom_domains.set_default({
      'domain': 'my-custom-domain.com',
  })
  ```

  ```go Go
  import (
      "context"
      "github.com/auth0/go-auth0/v2/management"
      management_client "github.com/auth0/go-auth0/v2/management/client"
      "github.com/auth0/go-auth0/v2/management/option"
  )

  mgmt, err := management_client.New(
      "{yourDomain}",
      option.WithClientCredentials("{yourClientId}", "{yourClientSecret}"),
  )
  if err != nil {
      // Gestionar error
  }

  err = mgmt.CustomDomains.SetDefault(context.Background(), &management.SetDefaultCustomDomainRequestContent{
      Domain: "my-custom-domain.com",
  })
  ```

  ```java Java
  import com.auth0.client.mgmt.ManagementAPI;
  import com.auth0.json.mgmt.customdomains.SetDefaultCustomDomainRequest;

  ManagementAPI mgmt = ManagementAPI.newBuilder(
      "{yourDomain}",
      "{yourMgmtApiAccessToken}"
  ).build();

  SetDefaultCustomDomainRequest request = new SetDefaultCustomDomainRequest("my-custom-domain.com");

  mgmt.customDomains()
      .setDefault(request)
      .execute();
  ```
</AuthCodeGroup>

<div id="remove-default-domain-designation">
  ## Quitar la designación de dominio predeterminado
</div>

Para quitar la designación predeterminada de un dominio personalizado:

1. Establece otro dominio personalizado como predeterminado (solo un dominio puede ser el predeterminado a la vez), o
2. Usa la Management API para establecer `is_default: false` en el dominio predeterminado actual

Si no hay ningún dominio personalizado configurado como predeterminado, Auth0 usará el dominio canónico de tu tenant.

<div id="how-the-default-domain-is-used">
  ## Cómo se utiliza el dominio predeterminado
</div>

<div id="email-notifications">
  ### Notificaciones por correo electrónico
</div>

Cuando Auth0 envía notificaciones por correo electrónico (restablecimientos de contraseña, verificaciones de correo electrónico y correos de bienvenida), se utiliza el dominio predeterminado para los enlaces y las personalizaciones de estas notificaciones. Puedes personalizar las plantillas de correo electrónico para usar información del dominio personalizado en la dirección &quot;De&quot;, el asunto y el cuerpo del correo electrónico.

<div id="management-api-endpoints-that-trigger-notifications">
  ### Endpoints de Management API que desencadenan notificaciones
</div>

El dominio personalizado predeterminado se usa específicamente para los endpoints de Management API que desencadenan notificaciones por correo electrónico o teléfono. El encabezado `auth0-custom-domain` pasa a ser opcional para estos endpoints cuando tienes un dominio predeterminado configurado.

**Endpoints que desencadenan notificaciones por correo electrónico:**

* `POST /api/v2/tickets/password-change` - Envía un correo electrónico de restablecimiento de contraseña
* `POST /api/v2/tickets/email-verification` - Envía una verificación de correo electrónico
* `POST /api/v2/jobs/verification-email` - Envía un correo de verificación al usuario o a los usuarios
* `POST /api/v2/users` - Puede desencadenar un correo electrónico de bienvenida si está configurado
* `PATCH /api/v2/users/{id}` - Puede desencadenar un correo de verificación cuando se actualiza el correo electrónico

**Endpoints que desencadenan notificaciones telefónicas:**

* `POST /api/v2/users/{id}/phone/verification` - Envía un código de verificación por SMS

**Ejemplo:**

<AuthCodeGroup>
  ```bash Sin el encabezado de dominio personalizado (usa el predeterminado)
  # El ticket de restablecimiento de contraseña usa el dominio predeterminado en los enlaces del correo electrónico
  curl --request POST \
    --url 'https://{yourDomain}/api/v2/tickets/password-change' \
    --header 'authorization: Bearer {yourMgmtApiAccessToken}' \
    --header 'content-type: application/json' \
    --data '{
      "user_id": "auth0|123456",
      "result_url": "https://example.com/password-changed"
    }'
  ```

  ```bash Con el encabezado de dominio personalizado (anula el predeterminado)
  # Especifica explícitamente un dominio personalizado distinto para esta notificación
  curl --request POST \
    --url 'https://{yourDomain}/api/v2/tickets/password-change' \
    --header 'authorization: Bearer {yourMgmtApiAccessToken}' \
    --header 'auth0-custom-domain: brand-specific.com' \
    --header 'content-type: application/json' \
    --data '{
      "user_id": "auth0|123456",
      "result_url": "https://example.com/password-changed"
    }'
  ```
</AuthCodeGroup>

Cuando no proporcionas el encabezado `auth0-custom-domain` para estos endpoints de notificación, Auth0 usa automáticamente el dominio personalizado predeterminado para los enlaces y las personalizaciones del correo electrónico o SMS. Puedes anularlo para una solicitud concreta incluyendo el encabezado `auth0-custom-domain`.

<div id="using-the-default-domain-with-actions">
  ## Cómo usar el dominio predeterminado con Actions
</div>

Usa [Actions](/docs/es/customize/actions) para implementar lógica según el dominio personalizado:

```javascript
exports.onExecutePostLogin = async (event, api) => {
  const domain = event.custom_domain?.domain;

  // Guardar el dominio de inicio de sesión en los metadatos del usuario
  if (domain) {
    api.user.setAppMetadata('login_domain', domain);
  }
};
```

Para obtener más información sobre cómo usar información sobre dominios personalizados en Actions, consulta [Integración de Actions con múltiples dominios personalizados](/docs/es/customize/custom-domains/multiple-custom-domains/actions-integration).

<div id="best-practices">
  ## Prácticas recomendadas
</div>

Al configurar un dominio personalizado predeterminado, ten en cuenta estas prácticas recomendadas:

* **Elige un dominio estable**: Selecciona como predeterminado un dominio que no vaya a cambiar con frecuencia
* **Verifica antes de establecerlo**: Asegúrate de que el dominio esté completamente verificado y operativo antes de convertirlo en el predeterminado
* **Documenta la decisión**: Registra por qué se eligió un dominio concreto como predeterminado para futuras consultas
* **Prueba los flujos de correo electrónico**: Después de establecer un dominio predeterminado, prueba los flujos de Restablecimiento de contraseña y verificación de correo electrónico
* **Supervisa los emisores de los tokens**: Verifica que los tokens incluyan la claim `iss` esperada después de establecer un valor predeterminado
* **Planifica escenarios con múltiples marcas**: Si trabajas con varias marcas, elige un dominio genérico o de administrador como predeterminado
* **Actualiza las integraciones**: Informa a tu equipo sobre el dominio predeterminado para garantizar una configuración correcta de las integraciones
