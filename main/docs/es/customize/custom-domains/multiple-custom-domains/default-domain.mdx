Si tienes [múltiples dominios personalizados](/docs/es/customize/custom-domains/multiple-custom-domains) configurados para tu tenant de Auth0, puedes designar uno como el **dominio personalizado predeterminado**. El dominio predeterminado simplifica la configuración y garantiza una experiencia de usuario coherente cuando no se especifica explícitamente un dominio personalizado.

<div id="what-is-a-default-custom-domain">
  ## ¿Qué es un dominio personalizado predeterminado?
</div>

El dominio personalizado predeterminado es el dominio personalizado que Auth0 usa automáticamente para:

* **Comunicaciones por correo electrónico y teléfono**: enviar correos electrónicos de restablecimiento de contraseña, enlaces de verificación de correo electrónico y otras notificaciones generadas por Auth0 cuando no se especifica ningún dominio personalizado concreto. Esto incluye los casos en los que se envían notificaciones durante la autenticación.
* **Llamadas a la Management API**: procesar solicitudes a la API que activan notificaciones sin el encabezado `auth0-custom-domain`

Puedes configurar tu dominio canónico del tenant (`YOUR_TENANT.auth0.com` o `YOUR_TENANT.REGION.auth0.com`) como el dominio personalizado predeterminado. Siempre habrá un dominio personalizado predeterminado configurado para tu tenant.

<div id="benefits-of-setting-a-default-domain">
  ## Beneficios de establecer un dominio predeterminado
</div>

Establecer un dominio personalizado predeterminado ofrece varias ventajas:

* **Configuración simplificada**: Reduce la necesidad de especificar dominios personalizados en cada llamada a la Management API o en la configuración
* **Branding consistente**: Garantiza que los usuarios siempre vean tu marca, incluso cuando no se especifica un dominio concreto
* **Encabezado `auth0-custom-domain` opcional**: Hace que el encabezado de dominio personalizado sea opcional en las requests a la Management API
* **Comportamiento fallback**: Proporciona valores predeterminados adecuados para implementaciones multimarca o multi-tenant
* **Migración más sencilla**: Simplifica la transición de un único dominio personalizado a múltiples dominios personalizados

<div id="configure-a-default-domain">
  ## Configurar un dominio predeterminado
</div>

Puede establecer un dominio personalizado predeterminado desde el Auth0 Dashboard o la Management API.

<div id="using-the-auth0-dashboard">
  ### Uso del Auth0 Dashboard
</div>

Para establecer un dominio personalizado predeterminado en el Auth0 Dashboard:

1. Ve a **Auth0 Dashboard** &gt; **Branding &gt; Custom Domains**
2. Busca en la lista el dominio personalizado verificado que quieras establecer como predeterminado
3. Haz clic en el botón **Set as Default** de ese dominio
4. Confirma la acción en el cuadro de diálogo que aparece

El dominio ahora mostrará la etiqueta &quot;Default&quot; en la lista de dominios personalizados.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Solo se pueden establecer como predeterminados los dominios personalizados verificados. Asegúrate de que tu dominio personalizado esté completamente verificado y activo antes de designarlo como predeterminado.
</Callout>

<div id="using-the-management-api">
  ### Uso de la Management API
</div>

Para configurar un dominio personalizado predeterminado mediante la Management API, utiliza el endpoint `PATCH /api/v2/custom-domains/default`:

<AuthCodeGroup>
  ```bash cURL
  curl --request PATCH \
    --url 'https://{yourDomain}/api/v2/custom-domains/default' \
    --header 'authorization: Bearer {yourMgmtApiAccessToken}' \
    --header 'content-type: application/json' \
    --data '{
      "domain": "my-custom-domain.com"
    }'
  ```

  ```javascript Node.js
  import { ManagementClient } from "auth0";

  const management = new ManagementClient({
    domain: '{yourDomain}',
    clientId: '{yourClientId}',
    clientSecret: '{yourClientSecret}',
  });

  await management.customDomains.setDefault({
    domain: 'my-custom-domain.com',
  });
  ```

  ```python Python
  from auth0.management import ManagementClient

  client = ManagementClient(
      domain='{yourDomain}',
      client_id='{yourClientId}',
      client_secret='{yourClientSecret}',
  )

  client.custom_domains.set_default({
      'domain': 'my-custom-domain.com',
  })
  ```

  ```go Go
  import (
      "context"
      "github.com/auth0/go-auth0/v2/management"
      management_client "github.com/auth0/go-auth0/v2/management/client"
      "github.com/auth0/go-auth0/v2/management/option"
  )

  mgmt, err := management_client.New(
      "{yourDomain}",
      option.WithClientCredentials("{yourClientId}", "{yourClientSecret}"),
  )
  if err != nil {
      // Gestiona el error
  }

  err = mgmt.CustomDomains.SetDefault(context.Background(), &management.SetDefaultCustomDomainRequestContent{
      Domain: "my-custom-domain.com",
  })
  ```

  ```java Java
  import com.auth0.client.mgmt.ManagementAPI;
  import com.auth0.json.mgmt.customdomains.SetDefaultCustomDomainRequest;

  ManagementAPI mgmt = ManagementAPI.newBuilder(
      "{yourDomain}",
      "{yourMgmtApiAccessToken}"
  ).build();

  SetDefaultCustomDomainRequest request = new SetDefaultCustomDomainRequest("my-custom-domain.com");

  mgmt.customDomains()
      .setDefault(request)
      .execute();
  ```
</AuthCodeGroup>

<div id="remove-default-domain-designation">
  ## Eliminar la condición de dominio predeterminado
</div>

Para quitar la condición de predeterminado de un dominio personalizado:

1. Establece otro dominio personalizado como predeterminado (solo un dominio puede ser el predeterminado a la vez), o
2. Usa la Management API para establecer `is_default: false` en el dominio predeterminado actual

Si no hay ningún dominio personalizado configurado como predeterminado, Auth0 usará tu dominio canónico del tenant.

<div id="how-the-default-domain-is-used">
  ## Cómo se utiliza el dominio predeterminado
</div>

<div id="email-notifications">
  ### Notificaciones por correo electrónico
</div>

Cuando Auth0 envía notificaciones por correo electrónico (restablecimiento de contraseña, verificación de correo electrónico y correos de bienvenida), se usa el dominio predeterminado para los enlaces y las personalizaciones de estas notificaciones. Puede personalizar las plantillas de correo electrónico para usar información del dominio personalizado en la dirección &quot;De&quot;, el asunto y el cuerpo del correo electrónico.

<div id="management-api-endpoints-that-trigger-notifications">
  ### Endpoints de la Management API que activan notificaciones
</div>

El dominio personalizado predeterminado se usa específicamente para los endpoints de la Management API que activan notificaciones por correo electrónico o por teléfono. El encabezado `auth0-custom-domain` pasa a ser opcional para estos endpoints cuando tienes configurado un dominio predeterminado.

**Endpoints que activan notificaciones por correo electrónico:**

* `POST /api/v2/tickets/password-change` - Envía un correo electrónico de restablecimiento de contraseña
* `POST /api/v2/tickets/email-verification` - Envía una verificación por correo electrónico
* `POST /api/v2/jobs/verification-email` - Envía un correo electrónico de verificación al usuario o a los usuarios
* `POST /api/v2/users` - Puede activar un correo electrónico de bienvenida si está configurado
* `PATCH /api/v2/users/{id}` - Puede activar un correo electrónico de verificación cuando se actualiza el correo electrónico

**Endpoints que activan notificaciones por teléfono:**

* `POST /api/v2/users/{id}/phone/verification` - Envía un código de verificación por SMS

**Ejemplo:**

<AuthCodeGroup>
  ```bash Without custom domain header (uses default)
  # Sin encabezado de dominio personalizado (usa el predeterminado)
  # El ticket de restablecimiento de contraseña usa el dominio predeterminado en los enlaces del correo electrónico
  curl --request POST \
    --url 'https://{yourDomain}/api/v2/tickets/password-change' \
    --header 'authorization: Bearer {yourMgmtApiAccessToken}' \
    --header 'content-type: application/json' \
    --data '{
      "user_id": "auth0|123456",
      "result_url": "https://example.com/password-changed"
    }'
  ```

  ```bash With custom domain header (overrides default)
  # Con encabezado de dominio personalizado (anula el predeterminado)
  # Especifica explícitamente un dominio personalizado diferente para esta notificación
  curl --request POST \
    --url 'https://{yourDomain}/api/v2/tickets/password-change' \
    --header 'authorization: Bearer {yourMgmtApiAccessToken}' \
    --header 'auth0-custom-domain: brand-specific.com' \
    --header 'content-type: application/json' \
    --data '{
      "user_id": "auth0|123456",
      "result_url": "https://example.com/password-changed"
    }'
  ```
</AuthCodeGroup>

Cuando no proporcionas el encabezado `auth0-custom-domain` para estos endpoints de notificación, Auth0 usa automáticamente el dominio personalizado predeterminado para los enlaces y las personalizaciones del correo electrónico o SMS. Puedes anularlo en cada solicitud incluyendo el encabezado `auth0-custom-domain`.

<div id="using-the-default-domain-with-actions">
  ## Cómo usar el dominio predeterminado con Actions
</div>

Usa [Actions](/docs/es/customize/actions) para implementar lógica en función del dominio personalizado:

```javascript
exports.onExecutePostLogin = async (event, api) => {
  const domain = event.custom_domain?.domain;

  // Guardar el dominio de inicio de sesión en los metadatos del usuario
  if (domain) {
    api.user.setAppMetadata('login_domain', domain);
  }
};
```

Para obtener más información sobre cómo usar la información sobre el dominio personalizado en Actions, consulta [Integración de Actions con múltiples dominios personalizados](/docs/es/customize/custom-domains/multiple-custom-domains/actions-integration).

<div id="best-practices">
  ## Prácticas recomendadas
</div>

Al configurar un dominio personalizado predeterminado, tenga en cuenta estas prácticas recomendadas:

* **Elija un dominio estable**: Seleccione un dominio que no cambie con frecuencia como predeterminado
* **Verifique antes de definirlo**: Asegúrese de que el dominio esté totalmente verificado y operativo antes de convertirlo en el predeterminado
* **Documente la decisión**: Registre por qué se eligió un dominio concreto como predeterminado para futuras consultas
* **Pruebe los flujos de correo electrónico**: Después de definir un dominio predeterminado, pruebe los flujos de restablecimiento de contraseña y verificación del correo electrónico
* **Supervise los emisores de tokens**: Verifique que los tokens tengan el claim `iss` esperado después de definir un valor predeterminado
* **Planifique para escenarios multimarca**: Si trabaja con varias marcas, elija un dominio genérico o de administración como predeterminado
* **Actualice las integraciones**: Informe a su equipo sobre el dominio predeterminado para garantizar una configuración correcta de las integraciones
