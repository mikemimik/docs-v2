Esta guía le ayuda a migrar de una configuración con un solo dominio personalizado a [múltiples dominios personalizados](/docs/es/customize/custom-domains/multiple-custom-domains). Ya sea que agregue dominios para distintas marcas, regiones o segmentos de clientes, esta guía ofrece instrucciones paso a paso para que la transición sea fluida.

<div id="migration-scenarios">
  ## Escenarios de migración
</div>

Elige el escenario que mejor se adapte a tu situación:

| Escenario                            | Descripción                                                                                                                | Caso de uso                                                           | Complejidad  | Tiempo de inactividad              |
| ------------------------------------ | -------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------- | ------------ | ---------------------------------- |
| **Añadir dominios adicionales**      | Actualmente tienes un dominio personalizado y quieres añadir más sin dejar de usar el dominio actual                       | Ampliarte para dar servicio a varias marcas o regiones                | Low          | None                               |
| **Reemplazar un dominio existente**  | Quieres sustituir tu dominio personalizado actual por uno nuevo                                                            | Cambio de marca o cambio en la titularidad del dominio                | Media        | Mínimo (durante el cutover de DNS) |
| **Migrar desde el dominio canónico** | Actualmente usas tu dominio canónico de Auth0 (por ejemplo, `tenant.auth0.com`) y quieres migrar a dominios personalizados | Implementación inicial de dominios personalizados con varios dominios | Media a alta | None (funcionamiento en paralelo)  |

<div id="pre-migration-checklist">
  ## Lista de verificación previa a la migración
</div>

Antes de comenzar la migración, asegúrese de haber completado lo siguiente:

* Verificado la titularidad del dominio de todos los nuevos dominios personalizados
* Revisado los procesos actuales de autenticación y las integraciones con API
* Identificado todas las aplicaciones que utilizan el dominio actual
* Documentado las plantillas de correo electrónico y los enlaces actuales
* Obtenido los certificados SSL/TLS (si utiliza certificados autogestionados)
* Probado la configuración del nuevo dominio personalizado en un entorno de desarrollo o de staging
* Preparado un plan de reversión
* Programado la migración durante un período de baja actividad (si corresponde)
* Notificado a las partes interesadas y a los usuarios (si es necesario)

<div id="migration-steps">
  ## Pasos para la migración
</div>

<div id="add-your-new-custom-domains">
  ### Añade tus nuevos dominios personalizados
</div>

Añade tus nuevos dominios personalizados mediante el [Auth0 Dashboard](/docs/es/customize/custom-domains/multiple-custom-domains#configure-multiple-custom-domains) o la [Management API](https://auth0.com/docs/api/management/v2).

<AuthCodeGroup>
  ```bash cURL
  curl --request POST \
    --url 'https://{yourDomain}/api/v2/custom-domains' \
    --header 'authorization: Bearer {yourMgmtApiAccessToken}' \
    --header 'content-type: application/json' \
    --data '{
      "domain": "new-domain.example.com",
      "type": "auth0_managed_certs",
      "domain_metadata": {
        "purpose": "new-brand"
      }
    }'
  ```

  ```javascript Node.js
  const newDomain = await management.customDomains.create({
    domain: 'new-domain.example.com',
    type: 'auth0_managed_certs',
    domain_metadata: {
      purpose: 'new-brand'
    }
  });

  console.log('Domain added:', newDomain.custom_domain_id);
  ```
</AuthCodeGroup>

<div id="verify-domain-ownership">
  ### Verificar la titularidad del dominio
</div>

Completa la verificación del dominio de cada nuevo dominio personalizado:

<div id="for-auth0-managed-certificates">
  #### Para certificados gestionados por Auth0
</div>

1. Toma nota del registro CNAME proporcionado por Auth0
2. Añade el registro CNAME en tu proveedor de DNS
3. Verifica el dominio a través del Dashboard o de la API

```bash
# Obtener detalles de verificación
curl --request GET \
  --url 'https://{yourDomain}/api/v2/custom-domains/{customDomainId}' \
  --header 'authorization: Bearer {yourMgmtApiAccessToken}'

# Verificar después de agregar el registro DNS
curl --request POST \
  --url 'https://{yourDomain}/api/v2/custom-domains/{customDomainId}/verify' \
  --header 'authorization: Bearer {yourMgmtApiAccessToken}'
```

<div id="for-self-managed-certificates">
  #### Para certificados autogestionados
</div>

1. Añade el registro TXT necesario en tu DNS
2. Configura tu proxy inverso o CDN
3. Sube tu certificado SSL
4. Verifica el dominio

<div id="configure-default-domain-optional">
  ### Configurar dominio predeterminado (opcional)
</div>

Si quieres usar un dominio de forma predeterminada para los correos electrónicos y las llamadas a la API, [configúralo como dominio predeterminado](/docs/es/customize/custom-domains/multiple-custom-domains/default-domain):

```bash
curl --request PATCH \
  --url 'https://{yourDomain}/api/v2/custom-domains/{customDomainId}' \
  --header 'authorization: Bearer {yourMgmtApiAccessToken}' \
  --header 'content-type: application/json' \
  --data '{
    "is_default": true
  }'
```

<div id="update-application-configurations">
  ### Actualiza la configuración de las aplicaciones
</div>

Actualiza tus aplicaciones para usar el dominio personalizado correspondiente:

<div id="sdk-configuration">
  #### Configuración del SDK
</div>

Actualiza la inicialización del SDK de Auth0:

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  El único cambio necesario es actualizar el parámetro `domain`: sustituye el dominio canónico de Auth0 por tu nuevo dominio personalizado.
</Callout>

<AuthCodeGroup>
  ```javascript Auth0 SPA SDK
  // Antes
  const auth0 = await createAuth0Client({
    domain: 'tenant.auth0.com',
    client_id: '{yourClientId}'
  });

  // Después
  const auth0 = await createAuth0Client({
    domain: 'new-domain.example.com', // Usa tu nuevo dominio personalizado
    client_id: '{yourClientId}'
  });
  ```

  ```javascript Auth0.js
  // Antes
  const webAuth = new auth0.WebAuth({
    domain: 'tenant.auth0.com',
    clientID: '{yourClientId}'
  });

  // Después
  const webAuth = new auth0.WebAuth({
    domain: 'new-domain.example.com',
    clientID: '{yourClientId}'
  });
  ```

  ```javascript Node.js
  import { ManagementClient } from "auth0";

  // Antes
  const auth0 = new ManagementClient({
    domain: 'tenant.auth0.com',
    clientId: '{yourClientId}',
    clientSecret: '{yourClientSecret}',
  });

  // Después
  const auth0 = new ManagementClient({
    domain: 'new-domain.example.com',
    clientId: '{yourClientId}',
    clientSecret: '{yourClientSecret}',
  });
  ```
</AuthCodeGroup>

<div id="callback-urls">
  #### URLs de devolución de llamada
</div>

Actualiza las URL de devolución de llamada de tu aplicación en Auth0 Dashboard:

1. Ve a [**Auth0 Dashboard** &gt; **Applications**](https://manage.auth0.com/#/applications). Elige la aplicación que quieres configurar y selecciona la pestaña **Settings**.
2. Actualiza **Allowed Callback URLs** para incluir el nuevo dominio:
   ```
   https://new-domain.example.com/callback
   ```
3. Actualiza **Allowed Logout URLs**:
   ```
   https://new-domain.example.com/logout
   ```
4. Actualiza **Allowed Web Origins**:
   ```
   https://new-domain.example.com
   ```

<div id="update-email-templates">
  ### Actualiza las plantillas de correo electrónico
</div>

Para que la información del dominio personalizado esté disponible en tus plantillas de correo electrónico:

1. Ve a **Branding** &gt; **Custom Domains**
2. Establece como predeterminado el dominio que quieras
3. Personaliza tus plantillas de correo electrónico para usar la información del dominio personalizado en la dirección &quot;De&quot;, el asunto y el cuerpo, según sea necesario

<Note>
  Establecer un dominio como predeterminado no cambia automáticamente tus correos electrónicos. Lo que hace es poner el contexto del dominio predeterminado a disposición de las plantillas de correo electrónico. Debes personalizar tus plantillas para usar esa información. El contexto del dominio predeterminado estará disponible cuando no se proporcione un dominio específico mediante el encabezado `auth0-custom-domain`.
</Note>

<div id="update-social-identity-providers-idp">
  ### Actualiza los proveedores de identidad social (IdP)
</div>

Actualiza las URI de redirección de tus proveedores de identidad social:

<div id="google">
  #### Google
</div>

1. Ve a [Google Cloud Console](https://console.cloud.google.com)
2. Ve a **APIs y servicios** &gt; **Credenciales**
3. Agrega `https://new-domain.example.com/login/callback` a **URI de redirección autorizadas**

<div id="facebook">
  #### Facebook
</div>

1. Ve a [Facebook Developers](https://developers.facebook.com)
2. Ve a tu aplicación &gt; **Facebook Login** &gt; **Configuración**
3. Agrega `https://new-domain.example.com/login/callback` a **URI de redirección de OAuth válidos**

<div id="other-providers">
  #### Otros proveedores
</div>

Actualiza las URI de redirección de todos los proveedores sociales configurados según la documentación de cada proveedor.

<div id="update-enterprise-connections-if-applicable">
  ### Actualiza las conexiones empresariales (si aplica)
</div>

Si usas SAML, WS-Fed, Azure AD u otras conexiones empresariales, actualiza su configuración:

<div id="saml-connections">
  #### Conexiones SAML
</div>

Actualiza la URL del Servicio de Consumo de Aserciones (ACS):

```
https://new-domain.example.com/login/callback?connection={connectionName}
```

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  No es necesario que actualice manualmente la URL de ACS en su IdP si usa solicitudes SAML iniciadas por el SP y su IdP acepta ACS dinámico.
</Callout>

<div id="azure-ad-connections">
  #### Conexiones de Azure AD
</div>

1. Ve a Azure Active Directory &gt; **App registrations**
2. Selecciona tu aplicación &gt; **Authentication**
3. Agrega `https://new-domain.example.com/login/callback` a **Redirect URIs**

<div id="adfs-connections">
  #### Conexiones de ADFS
</div>

Actualiza el endpoint en la configuración de ADFS para usar el nuevo dominio personalizado.

<div id="test-authentication">
  ### Probar la autenticación
</div>

Antes de completar la migración, prueba exhaustivamente lo siguiente:

1. **Inicio de sesión de usuario**: Verifica que los usuarios puedan autenticarse a través del nuevo dominio
2. **Restablecimiento de contraseña**: Comprueba que los correos electrónicos de restablecimiento de contraseña usen el dominio correcto
3. **Verificación del correo electrónico**: Verifica que los enlaces de verificación del correo electrónico funcionen
4. **Inicios de sesión con proveedores sociales**: Prueba cada proveedor social configurado
5. **Llamadas a la API**: Verifica que las llamadas a la API funcionen con el nuevo dominio
6. **Validación de tokens**: Asegúrate de que los JWT tengan el claim `iss` correcto

<div id="monitor-and-verify">
  ### Supervisar y verificar
</div>

Después de la migración:

1. Supervise los registros de autenticación para detectar cualquier error
2. Compruebe la entrega de correos electrónicos y el funcionamiento de los enlaces
3. Verifique la validez del certificado SSL y las fechas de expiración
4. Pruebe desde distintas regiones geográficas (si corresponde)
5. Confirme que todas las aplicaciones estén usando los dominios personalizados previstos

<div id="decommission-old-domain-if-applicable">
  ### Desactivar el dominio antiguo (si corresponde)
</div>

Si estás reemplazando un dominio personalizado existente:

1. Asegúrate de que todas las aplicaciones se hayan migrado al nuevo dominio.
2. Supervisa el tráfico del dominio antiguo para confirmar que ya no se esté usando.
3. Considera mantener el dominio antiguo activo durante un período de transición.
4. Elimina el dominio personalizado antiguo cuando estés listo:

```bash
curl --request DELETE \
  --url 'https://{yourDomain}/api/v2/custom-domains/{oldCustomDomainId}' \
  --header 'authorization: Bearer {yourMgmtApiAccessToken}'
```

<div id="migration-patterns">
  ## Patrones de migración
</div>

<AccordionGroup>
  <Accordion title="Operación en paralelo (sin tiempo de inactividad)">
    Usa simultáneamente el dominio personalizado antiguo y el nuevo:

    1. Agrega un nuevo dominio personalizado
    2. Actualiza las aplicaciones nuevas para que usen el nuevo dominio
    3. Mantén las aplicaciones existentes en el dominio antiguo
    4. Migra las aplicaciones de forma gradual
    5. Retira el dominio antiguo cuando la migración se haya completado

    **Ventajas**: Sin tiempo de inactividad, despliegue gradual, reversión sencilla

    **Desventajas**: Sobrecarga temporal de gestión
  </Accordion>

  <Accordion title="Migración por fases por aplicación">
    Migra las aplicaciones una por una:

    1. Identifica las aplicaciones según la prioridad o el riesgo
    2. Migra primero las aplicaciones de bajo riesgo
    3. Supervisa posibles incidencias antes de continuar
    4. Migra las aplicaciones restantes
    5. Limpia las configuraciones antiguas

    **Ventajas**: Despliegue controlado, detección temprana de incidencias

    **Desventajas**: Proceso de migración más largo
  </Accordion>

  <Accordion title="Despliegue blue-green">
    Usa entornos separados para las pruebas:

    1. Configura un nuevo dominio personalizado en el entorno de staging
    2. Prueba exhaustivamente toda la funcionalidad
    3. Cambia a Production en una sola operación
    4. Mantén el dominio antiguo como respaldo
    5. Retíralo después del período de verificación

    **Ventajas**: Pruebas exhaustivas, reversión rápida

    **Desventajas**: Requiere un entorno independiente
  </Accordion>
</AccordionGroup>

<div id="handling-existing-user-sessions">
  ## Cómo gestionar las sesiones de usuario existentes
</div>

Al migrar dominios personalizados, las sesiones de usuario existentes pueden verse afectadas:

<div id="session-considerations">
  ### Consideraciones sobre las sesiones
</div>

* Las sesiones creadas con el dominio anterior seguirán siendo válidas hasta su expiración
* Los nuevos inicios de sesión generarán sesiones con el nuevo dominio
* Las sesiones entre dominios requieren una planificación cuidadosa

<div id="recommended-approach">
  ### Enfoque recomendado
</div>

1. **Notifique a los usuarios**: Infórmeles de que es posible que tengan que volver a iniciar sesión
2. **Período de gracia**: Mantenga activo el dominio anterior durante la transición
3. **Transferencia de sesión**: Use Universal Login para gestionar la migración de la sesión
4. **Instrucciones claras**: Proporcione instrucciones claras si los usuarios tienen problemas

<div id="rollback-plan">
  ## Plan de reversión
</div>

Si encuentras problemas durante la migración:

<div id="immediate-rollback">
  ### Reversión inmediata
</div>

1. Revertir las configuraciones de la aplicación para usar el dominio anterior
2. Mantener configurado el nuevo dominio personalizado para futuros intentos
3. Documentar la incidencia para facilitar la solución de problemas

<div id="partial-rollback">
  ### Reversión parcial
</div>

1. Identifica las aplicaciones afectadas
2. Revierte solo esas aplicaciones al dominio anterior
3. Investiga y soluciona los problemas
4. Vuelve a migrar cuando estés listo

<div id="complete-rollback">
  ### Reversión completa
</div>

1. Actualiza todas las configuraciones de la aplicación para usar el dominio anterior
2. Si configuraste un dominio predeterminado, vuelve a cambiarlo al dominio anterior
3. Informa a los usuarios de cualquier acción necesaria
4. Programa otro intento de migración

<div id="troubleshooting">
  ## Solución de problemas
</div>

<div id="common-issues-and-solutions">
  ### Problemas comunes y soluciones
</div>

| Problema                                                     | Causa                                             | Solución                                                                                                       |
| ------------------------------------------------------------ | ------------------------------------------------- | -------------------------------------------------------------------------------------------------------------- |
| La verificación del dominio falla                            | Los registros DNS no se han propagado             | Espere a que se propaguen los DNS (puede tardar hasta 48 horas) y verifique que el registro CNAME sea correcto |
| El inicio de sesión redirige al dominio anterior             | La configuración de la aplicación no se actualizó | Actualice la inicialización del SDK y las URL de devolución de llamada                                         |
| Los enlaces de correo electrónico usan el dominio incorrecto | El dominio predeterminado no está configurado     | Establezca el dominio predeterminado o configure el enrutamiento del correo electrónico                        |
| El inicio de sesión social falla                             | Redirect URIs no se actualizaron                  | Agregue el nuevo dominio personalizado a las Redirect URIs permitidas del proveedor social                     |
| Emisor del token no válido                                   | La validación del JWT espera el dominio anterior  | Actualice la validación de tokens para aceptar el nuevo dominio como emisor                                    |
| Errores de aserción SAML                                     | La URL de ACS no se actualizó                     | Actualice la configuración de SAML con la nueva URL del dominio personalizado                                  |
| Errores de certificado                                       | El certificado no se aprovisionó o venció         | Verifique el dominio, espere al aprovisionamiento del certificado o actualice el certificado                   |

<div id="getting-help">
  ### Obtener ayuda
</div>

Si tienes problemas durante la migración:

1. Consulta [Auth0 Community](https://community.auth0.com/) para ver si hay problemas similares
2. Revisa la [documentación de solución de problemas](https://support.auth0.com/center/s/knowledge?selectedTopics=Custom%20Domains\&isTopicFilter=true)
3. Ponte en contacto con el [equipo de soporte de Auth0](https://support.auth0.com/) e incluye:
   * El nombre de tu tenant
   * Los ID de dominio personalizado
   * Una descripción detallada del problema
   * Los pasos para reproducirlo
   * Mensajes de error o registros

<div id="post-migration-best-practices">
  ## Prácticas recomendadas posteriores a la migración
</div>

Después de completar la migración:

1. **Documenta la configuración**: Registra qué aplicaciones usan qué dominios personalizados
2. **Supervisa la expiración del certificado**: Configura alertas para su renovación
3. **Revisa periódicamente**: Asegúrate de que la configuración del dominio personalizado se ajuste a las necesidades del negocio
4. **Actualiza los procedimientos operativos**: Actualiza la documentación operativa con la nueva información del dominio personalizado
5. **Capacita a los miembros del equipo**: Asegúrate de que tu equipo entienda la nueva configuración multidominio
6. **Planifica el crecimiento**: Considera cómo administrarás dominios adicionales en el futuro
