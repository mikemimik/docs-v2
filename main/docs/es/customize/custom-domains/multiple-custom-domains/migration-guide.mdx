Esta guía le ayuda a migrar de una configuración con un solo dominio personalizado a [múltiples dominios personalizados](/docs/es/customize/custom-domains/multiple-custom-domains). Tanto si va a añadir dominios para distintas marcas, regiones o segmentos de clientes, esta guía ofrece instrucciones paso a paso para que la transición sea fluida.

<div id="migration-scenarios">
  ## Escenarios de migración
</div>

Elige el escenario que mejor se adapte a tu situación:

| Escenario                                       | Descripción                                                                                                           | Caso de uso                                                           | Complejidad  | Tiempo de inactividad             |
| ----------------------------------------------- | --------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------- | ------------ | --------------------------------- |
| **Agregar dominios personalizados adicionales** | Actualmente tienes un dominio personalizado y quieres agregar más sin dejar de mantener operativo el dominio actual   | Expansión para dar servicio a varias marcas o regiones                | Low          | None                              |
| **Reemplazar un dominio existente**             | Quieres reemplazar tu dominio personalizado actual por uno nuevo                                                      | Cambio de marca o cambio en la titularidad del dominio                | Media        | Mínimo (durante el cambio de DNS) |
| **Migrar desde el dominio canónico**            | Actualmente usas tu dominio canónico de Auth0 (p. ej., `tenant.auth0.com`) y quieres migrar a dominios personalizados | Implementación inicial de dominios personalizados con varios dominios | Media a alta | None (operación en paralelo)      |

<div id="pre-migration-checklist">
  ## Lista de verificación previa a la migración
</div>

Antes de iniciar la migración, asegúrese de haber completado lo siguiente:

* Verificado la titularidad del dominio de todos los nuevos dominios personalizados
* Revisado los procesos actuales de autenticación y las integraciones de API
* Identificado todas las aplicaciones que usan el dominio actual
* Documentado las plantillas de correo electrónico y los enlaces actuales
* Obtenido los certificados SSL/TLS (si usa certificados autogestionados)
* Probado la nueva configuración del dominio personalizado en un entorno de desarrollo o de staging
* Preparado un plan de reversión
* Programado la migración durante un período de baja actividad (si corresponde)
* Notificado a las partes interesadas y a los usuarios (si es necesario)

<div id="migration-steps">
  ## Pasos para la migración
</div>

<div id="add-your-new-custom-domains">
  ### Agrega tus nuevos dominios personalizados
</div>

Agrega tus nuevos dominios personalizados con el [Auth0 Dashboard](/docs/es/customize/custom-domains/multiple-custom-domains#configure-multiple-custom-domains) o la [Management API](https://auth0.com/docs/api/management/v2).

<AuthCodeGroup>
  ```bash cURL
  curl --request POST \
    --url 'https://{yourDomain}/api/v2/custom-domains' \
    --header 'authorization: Bearer {yourMgmtApiAccessToken}' \
    --header 'content-type: application/json' \
    --data '{
      "domain": "new-domain.example.com",
      "type": "auth0_managed_certs",
      "domain_metadata": {
        "purpose": "new-brand"
      }
    }'
  ```

  ```javascript Node.js
  const newDomain = await management.customDomains.create({
    domain: 'new-domain.example.com',
    type: 'auth0_managed_certs',
    domain_metadata: {
      purpose: 'new-brand'
    }
  });

  console.log('Domain added:', newDomain.custom_domain_id);
  ```
</AuthCodeGroup>

<div id="verify-domain-ownership">
  ### Verificar la titularidad del dominio
</div>

Completa el proceso de verificación del dominio para cada nuevo dominio personalizado:

<div id="for-auth0-managed-certificates">
  #### Si usas certificados administrados por Auth0
</div>

1. Anota el registro CNAME proporcionado por Auth0
2. Añade el registro CNAME a tu proveedor de DNS
3. Verifica el dominio a través del Dashboard o de la API

```bash
# Obtener detalles de verificación
curl --request GET \
  --url 'https://{yourDomain}/api/v2/custom-domains/{customDomainId}' \
  --header 'authorization: Bearer {yourMgmtApiAccessToken}'

# Verificar después de agregar el registro DNS
curl --request POST \
  --url 'https://{yourDomain}/api/v2/custom-domains/{customDomainId}/verify' \
  --header 'authorization: Bearer {yourMgmtApiAccessToken}'
```

<div id="for-self-managed-certificates">
  #### Para certificados autogestionados
</div>

1. Añade el registro TXT necesario en tu DNS
2. Configura tu proxy inverso o CDN
3. Sube tu certificado SSL
4. Verifica el dominio

<div id="configure-default-domain-optional">
  ### Configurar el dominio predeterminado (opcional)
</div>

Si quieres que un dominio se use de forma predeterminada para los correos electrónicos y las llamadas a la API, [configúralo como el dominio predeterminado](/docs/es/customize/custom-domains/multiple-custom-domains/default-domain):

```bash
curl --request PATCH \
  --url 'https://{yourDomain}/api/v2/custom-domains/{customDomainId}' \
  --header 'authorization: Bearer {yourMgmtApiAccessToken}' \
  --header 'content-type: application/json' \
  --data '{
    "is_default": true
  }'
```

<div id="update-application-configurations">
  ### Actualizar las configuraciones de las aplicaciones
</div>

Actualiza tus aplicaciones para usar el dominio personalizado adecuado:

<div id="sdk-configuration">
  #### Configuración del SDK
</div>

Actualiza la inicialización de tu SDK de Auth0:

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  El único cambio necesario es actualizar el parámetro `domain`: sustituye tu dominio canónico de Auth0 por tu nuevo dominio personalizado.
</Callout>

<AuthCodeGroup>
  ```javascript Auth0 SPA SDK
  // Antes
  const auth0 = await createAuth0Client({
    domain: 'tenant.auth0.com',
    client_id: '{yourClientId}'
  });

  // Después
  const auth0 = await createAuth0Client({
    domain: 'new-domain.example.com', // Usa tu nuevo dominio personalizado
    client_id: '{yourClientId}'
  });
  ```

  ```javascript Auth0.js
  // Antes
  const webAuth = new auth0.WebAuth({
    domain: 'tenant.auth0.com',
    clientID: '{yourClientId}'
  });

  // Después
  const webAuth = new auth0.WebAuth({
    domain: 'new-domain.example.com',
    clientID: '{yourClientId}'
  });
  ```

  ```javascript Node.js
  import { ManagementClient } from "auth0";

  // Antes
  const auth0 = new ManagementClient({
    domain: 'tenant.auth0.com',
    clientId: '{yourClientId}',
    clientSecret: '{yourClientSecret}',
  });

  // Después
  const auth0 = new ManagementClient({
    domain: 'new-domain.example.com',
    clientId: '{yourClientId}',
    clientSecret: '{yourClientSecret}',
  });
  ```
</AuthCodeGroup>

<div id="callback-urls">
  #### URL de devolución de llamada
</div>

Actualiza las URL de devolución de llamada de tu aplicación en Auth0 Dashboard:

1. Ve a [**Auth0 Dashboard** &gt; **Applications**](https://manage.auth0.com/#/applications). Elige la aplicación que quieres configurar y selecciona la pestaña **Settings**.
2. Actualiza **Allowed Callback URLs** para incluir el nuevo dominio:
   ```
   https://new-domain.example.com/callback
   ```
3. Actualiza **Allowed Logout URLs**:
   ```
   https://new-domain.example.com/logout
   ```
4. Actualiza **Allowed Web Origins**:
   ```
   https://new-domain.example.com
   ```

<div id="update-email-templates">
  ### Actualizar plantillas de correo electrónico
</div>

Para que la información del dominio personalizado esté disponible en tus plantillas de correo electrónico:

1. Ve a **Branding** &gt; **Custom Domains**
2. Establece como predeterminado el dominio que desees
3. Personaliza tus plantillas de correo electrónico para usar la información del dominio personalizado en la dirección &quot;De&quot;, el asunto y el cuerpo, según sea necesario

<Note>
  Establecer un dominio como predeterminado no cambia automáticamente tus correos electrónicos. Hace que el contexto del dominio predeterminado esté disponible en las plantillas de correo electrónico. Debes personalizar tus plantillas para usar esa información. El contexto del dominio predeterminado estará disponible cuando no se proporcione ningún dominio específico mediante el encabezado `auth0-custom-domain`.
</Note>

<div id="update-social-identity-providers-idp">
  ### Actualizar los proveedores de identidad social (IdP)
</div>

Actualiza las URI de redirección de tus proveedores de identidad social:

<div id="google">
  #### Google
</div>

1. Ve a [Google Cloud Console](https://console.cloud.google.com)
2. Ve a **APIs y servicios** &gt; **Credenciales**
3. Agrega `https://new-domain.example.com/login/callback` a **URI de redirección autorizados**

<div id="facebook">
  #### Facebook
</div>

1. Ve a [Facebook Developers](https://developers.facebook.com)
2. Ve a tu aplicación &gt; **Facebook Login** &gt; **Configuración**
3. Agrega `https://new-domain.example.com/login/callback` a **URI de redirección de OAuth válidas**

<div id="other-providers">
  #### Otros proveedores
</div>

Actualiza los URI de redirección de todos los proveedores sociales configurados siguiendo la documentación de cada proveedor.

<div id="update-enterprise-connections-if-applicable">
  ### Actualiza las conexiones empresariales (si aplica)
</div>

Si usas SAML, WS-Fed, Azure AD u otras conexiones empresariales, actualiza su configuración:

<div id="saml-connections">
  #### Conexiones SAML
</div>

Actualiza la URL del Servicio de Consumo de Aserciones (ACS):

```
https://new-domain.example.com/login/callback?connection={connectionName}
```

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  No necesita actualizar manualmente la URL de ACS en su IdP si usa solicitudes SAML iniciadas por el SP y su IdP acepta ACS dinámico.
</Callout>

<div id="azure-ad-connections">
  #### Conexiones de Azure AD
</div>

1. Ve a Azure Active Directory &gt; **App registrations**
2. Selecciona tu aplicación &gt; **Authentication**
3. Agrega `https://new-domain.example.com/login/callback` a **Redirect URIs**

<div id="adfs-connections">
  #### Conexiones ADFS
</div>

Actualiza el endpoint en la configuración de ADFS para que use el nuevo dominio personalizado.

<div id="test-authentication">
  ### Probar la autenticación
</div>

Antes de completar la migración, prueba exhaustivamente lo siguiente:

1. **Inicio de sesión de usuarios**: Verifica que los usuarios puedan autenticarse a través del nuevo dominio
2. **Restablecimiento de contraseña**: Comprueba que los correos electrónicos de restablecimiento de contraseña usen el dominio correcto
3. **Verificación del correo electrónico**: Verifica que los enlaces de verificación del correo electrónico funcionen
4. **Inicios de sesión sociales**: Prueba cada proveedor social configurado
5. **Llamadas a la API**: Verifica que las llamadas a la API funcionen con el nuevo dominio
6. **Validación de tokens**: Asegúrate de que los JWT tengan el claim `iss` correcto

<div id="monitor-and-verify">
  ### Supervisar y verificar
</div>

Después de la migración:

1. Revise los registros de autenticación por si hubiera errores
2. Compruebe la entrega de correos electrónicos y el funcionamiento de los enlaces
3. Verifique la validez del certificado SSL y las fechas de vencimiento
4. Haga pruebas desde distintas regiones geográficas (si corresponde)
5. Confirme que todas las aplicaciones usen los dominios personalizados previstos

<div id="decommission-old-domain-if-applicable">
  ### Retirar el dominio antiguo (si corresponde)
</div>

Si vas a reemplazar un dominio personalizado existente:

1. Asegúrate de que todas las aplicaciones se hayan migrado al nuevo dominio
2. Supervisa el tráfico al dominio antiguo para confirmar que ya no está en uso
3. Considera mantener activo el dominio antiguo durante un período de gracia
4. Elimina el dominio personalizado antiguo cuando estés listo:

```bash
curl --request DELETE \
  --url 'https://{yourDomain}/api/v2/custom-domains/{oldCustomDomainId}' \
  --header 'authorization: Bearer {yourMgmtApiAccessToken}'
```

<div id="migration-patterns">
  ## Patrones de migración
</div>

<AccordionGroup>
  <Accordion title="Operación en paralelo (sin tiempo de inactividad)">
    Mantén el dominio personalizado antiguo y el nuevo funcionando simultáneamente:

    1. Agrega un nuevo dominio personalizado
    2. Actualiza las aplicaciones nuevas para que usen el nuevo dominio
    3. Mantén las aplicaciones existentes en el dominio antiguo
    4. Migra las aplicaciones gradualmente
    5. Retira el dominio antiguo cuando la migración se haya completado por completo

    **Ventajas**: Sin tiempo de inactividad, despliegue gradual, reversión sencilla

    **Desventajas**: Sobrecarga temporal de gestión
  </Accordion>

  <Accordion title="Migración por fases por aplicación">
    Migra las aplicaciones una a una:

    1. Identifica las aplicaciones según su prioridad o riesgo
    2. Migra primero las aplicaciones de bajo riesgo
    3. Supervisa si surgen problemas antes de continuar
    4. Migra las aplicaciones restantes
    5. Elimina las configuraciones antiguas

    **Ventajas**: Despliegue controlado, detección temprana de problemas

    **Desventajas**: Proceso de migración más largo
  </Accordion>

  <Accordion title="Despliegue blue-green">
    Usa entornos separados para las pruebas:

    1. Configura el nuevo dominio personalizado en el entorno de staging
    2. Prueba exhaustivamente toda la funcionalidad
    3. Pasa a producción en una sola operación
    4. Mantén el dominio antiguo como respaldo
    5. Retíralo después del período de verificación

    **Ventajas**: Pruebas exhaustivas, reversión rápida

    **Desventajas**: Requiere un entorno separado
  </Accordion>
</AccordionGroup>

<div id="handling-existing-user-sessions">
  ## Gestión de las sesiones de usuario existentes
</div>

Al migrar dominios personalizados, las sesiones de usuario existentes pueden verse afectadas:

<div id="session-considerations">
  ### Consideraciones sobre la sesión
</div>

* Las sesiones creadas con el dominio anterior seguirán siendo válidas hasta que caduquen
* Los nuevos inicios de sesión crearán sesiones con el nuevo dominio
* Las sesiones entre dominios requieren una planificación cuidadosa

<div id="recommended-approach">
  ### Enfoque recomendado
</div>

1. **Notificar a los usuarios**: Informe a los usuarios de que quizá deban volver a iniciar sesión
2. **Período de gracia**: Mantenga activo el dominio anterior durante la transición
3. **Transferencia de sesión**: Use Universal Login para gestionar la migración de la sesión
4. **Orientación clara**: Proporcione instrucciones claras si los usuarios tienen problemas

<div id="rollback-plan">
  ## Plan de reversión
</div>

Si tienes problemas durante la migración:

<div id="immediate-rollback">
  ### Reversión inmediata
</div>

1. Revertir la configuración de la aplicación para usar el dominio anterior
2. Mantener configurado el nuevo dominio personalizado para futuros intentos
3. Documentar la incidencia para facilitar el diagnóstico y la resolución de problemas

<div id="partial-rollback">
  ### Reversión parcial
</div>

1. Identifica las aplicaciones afectadas
2. Revierte solo esas aplicaciones al dominio anterior
3. Investiga y corrige los problemas
4. Vuelve a migrar cuando todo esté listo

<div id="complete-rollback">
  ### Reversión completa
</div>

1. Actualiza la configuración de todas las aplicaciones para que usen el dominio antiguo
2. Si configuraste un dominio predeterminado, vuelve a cambiarlo al dominio antiguo
3. Informa a los usuarios de cualquier acción necesaria
4. Programa otro intento de migración

<div id="troubleshooting">
  ## Solución de problemas
</div>

<div id="common-issues-and-solutions">
  ### Problemas comunes y soluciones
</div>

| Problema                                                     | Causa                                                  | Solución                                                                                                       |
| ------------------------------------------------------------ | ------------------------------------------------------ | -------------------------------------------------------------------------------------------------------------- |
| La verificación del dominio falla                            | Los registros DNS no se han propagado                  | Espere a que se propaguen los DNS (puede tardar hasta 48 horas) y verifique que el registro CNAME sea correcto |
| El login redirige al dominio anterior                        | La configuración de la aplicación no se ha actualizado | Actualice la inicialización del SDK y las URLs de devolución de llamada                                        |
| Los enlaces de correo electrónico usan el dominio incorrecto | El dominio predeterminado no está configurado          | Establezca el dominio predeterminado o configure el enrutamiento del correo electrónico                        |
| El inicio de sesión social falla                             | Las URI de redirección no se han actualizado           | Agregue el nuevo dominio personalizado a las URI de redirección permitidas del proveedor social                |
| Emisor de token no válido                                    | La validación de JWT espera el dominio anterior        | Actualice la validación de tokens para aceptar el nuevo dominio como emisor                                    |
| Errores de aserción SAML                                     | La URL de ACS no se ha actualizado                     | Actualice la configuración de SAML con la nueva URL del dominio personalizado                                  |
| Errores de certificado                                       | El certificado no se aprovisionó o ha caducado         | Verifique el dominio, espere al aprovisionamiento del certificado o actualice el certificado                   |

<div id="getting-help">
  ### Obtener ayuda
</div>

Si tiene problemas durante la migración:

1. Consulte [Auth0 Community](https://community.auth0.com/) para ver si hay problemas similares
2. Revise la [documentación de solución de problemas](https://support.auth0.com/center/s/knowledge?selectedTopics=Custom%20Domains\&isTopicFilter=true)
3. Póngase en contacto con el [equipo de soporte de Auth0](https://support.auth0.com/) e incluya:
   * El nombre de su tenant
   * Los IDs del dominio personalizado
   * Una descripción detallada del problema
   * Los pasos para reproducirlo
   * Mensajes de error o registros

<div id="post-migration-best-practices">
  ## Prácticas recomendadas posteriores a la migración
</div>

Después de completar la migración:

1. **Documenta la configuración**: Registra qué aplicaciones usan qué dominios personalizados
2. **Supervisa el vencimiento del certificado**: Configura alertas para la renovación del certificado
3. **Revisa periódicamente**: Asegúrate de que la configuración del dominio personalizado se ajuste a las necesidades del negocio
4. **Actualiza los procedimientos operativos**: Actualiza la documentación operativa con la nueva información del dominio personalizado
5. **Capacita a los miembros del equipo**: Asegúrate de que tu equipo entienda la nueva configuración multidominio
6. **Planifica el escalado**: Considera cómo gestionarás dominios adicionales en el futuro
