Para configurar Cloudflare como proxy inverso con el enfoque recomendado, se requiere un plan Enterprise de Cloudflare con las siguientes funciones:

| Función                       | Descripción                                                                                                                                                                                                                                                                                                                                      |
| ----------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Anulación del encabezado Host | Permite reescribir los encabezados Host mediante distintas reglas de Cloudflare. Para obtener más información, consulta [Reescribir encabezados Host en la documentación de Cloudflare](https://developers.cloudflare.com/rules/origin-rules/tutorials/change-uri-path-and-host-header/).                                                        |
| Encabezado True-Client-IP     | Al habilitar el encabezado True-Client-IP, se agrega el encabezado True-Client-IP a todas las solicitudes a tu servidor de origen, incluida la dirección IP del usuario final. Para obtener más información, consulta [Comprender el encabezado True-Client-IP en Cloudflare](https://developers.cloudflare.com/network/true-client-ip-header/). |

<div id="configure-cloudflare">
  ## Configurar Cloudflare
</div>

<Warning>
  Como requisito previo, el dominio principal del dominio personalizado que elijas debe estar [agregado y activado en el Dashboard de Cloudflare](https://developers.cloudflare.com/learning-paths/get-started/add-domain-to-cf/minimize-downtime/#activate-your-domain). Además, comprueba que el dominio personalizado que quieres usar no exista ya en tu zona de Cloudflare. Si ya existe, la verificación de Cloudflare fallará.
</Warning>

Para configurar Cloudflare como proxy inverso, deberás crear un registro CNAME, una Page Rule y una Transform Rule en Cloudflare.

1. [Configura y verifica un dominio personalizado con certificados autogestionados](/docs/es/customize/custom-domains/self-managed-certificates) si aún no lo has hecho. Toma nota de los valores **Origin Domain Name** y **cname-api-key**, ya que los necesitarás más adelante.
2. En el Dashboard de Cloudflare de la zona de destino, [crea un registro CNAME](https://developers.cloudflare.com/dns/manage-dns-records/how-to/create-dns-records/#create-dns-records) con la siguiente configuración:

   | Configuración    | Valor                                                  |
   | ---------------- | ------------------------------------------------------ |
   | Nombre           | El nombre del dominio personalizado.                   |
   | Destino          | El valor de **Origin Domain Name** que anotaste antes. |
   | Estado del proxy | `Proxied`                                              |
3. [Crea una Page Rule](https://developers.cloudflare.com/rules/page-rules/) que se aplique a todas las URL del dominio personalizado elegido y con la siguiente configuración:

   | Configuración        | Valor                                                  |
   | -------------------- | ------------------------------------------------------ |
   | Host Header Override | El valor de **Origin Domain Name** que anotaste antes. |
   | True-Client-IP       | `Enable`                                               |
4. [Crea una Transform Rule](https://developers.cloudflare.com/rules/transform/request-header-modification/create-dashboard/):

   <Callout icon="file-lines" color="#0EA5E9" iconType="regular">
     Aunque es posible usar Cloudflare Workers en lugar de reglas Page y Transform para configurar un proxy inverso que cumpla los requisitos de un dominio personalizado con certificados autogestionados, te recomendamos usar el enfoque basado en reglas porque evita la necesidad de escribir código personalizado.
   </Callout>

   1. Cambia a la vista **Modify Request Header**.
   2. Selecciona **Create Rule** y asígnale el nombre que prefieras.
   3. En **When incoming requests match**, selecciona **Custom filter expression** y configura una expresión que limite la Rule a las solicitudes asociadas al dominio personalizado elegido. Por ejemplo, usa una coincidencia exacta en el campo **Hostname**.
   4. En **Modify request header**, selecciona **Set static** y luego configura los siguientes campos:

      | Campo       | Valor                                             |
      | ----------- | ------------------------------------------------- |
      | Header name | `cname-api-key`                                   |
      | Valor       | El valor de **cname-api-key** que anotaste antes. |
5. Asegúrate de que [Always Use HTTPS](https://developers.cloudflare.com/ssl/edge-certificates/additional-options/always-use-https/) esté habilitado y de que el [modo de cifrado](https://developers.cloudflare.com/ssl/origin-configuration/ssl-modes/) esté configurado, como mínimo, en **Full** para el dominio personalizado elegido.

<div id="use-managed-challenges">
  ## Usar Managed Challenges
</div>

Los [Managed Challenges](https://developers.cloudflare.com/cloudflare-challenges/challenge-types/challenge-pages/#managed-challenges) de Cloudflare te permiten filtrar el tráfico de bots antes de que las solicitudes lleguen a Auth0 Universal Login. Cuando una solicitud coincide con tu regla, Cloudflare la intercepta y presenta un desafío de verificación. Como las páginas de desafío devuelven HTML, los Managed Challenges solo son compatibles con flujos basados en navegador; aplicarlos a endpoints de API o flujos sin interfaz hará que esos flujos fallen porque el cliente recibe una página HTML de desafío en lugar de la respuesta esperada.

<div id="endpoints-to-exclude">
  ### Endpoints que se deben excluir
</div>

No apliques un Managed Challenge a los siguientes endpoints. Estos son llamados por servidores, SDKs o servidores de recursos y no pueden resolver un desafío interactivo:

| Endpoint                            | Descripción                                                                           |
| ----------------------------------- | ------------------------------------------------------------------------------------- |
| `/oauth/token`                      | endpoint de token                                                                     |
| `/oauth/revoke`                     | endpoint de revocación de tokens                                                      |
| `/userinfo`                         | endpoint de UserInfo                                                                  |
| `/.well-known/openid-configuration` | documento de descubrimiento de OIDC                                                   |
| `/.well-known/jwks.json`            | JSON Web Key Set; lo obtienen los servidores de recursos para la validación de tokens |
| `/api/v2/*`                         | Management API                                                                        |
| `/co/authenticate`                  | autenticación de origen cruzado                                                       |
| `/dbconnections/signup`             | conexiones de base de datos: registro                                                 |
| `/dbconnections/change_password`    | conexiones de base de datos: cambiar la contraseña                                    |
| `/usernamepassword/login`           | envío del formulario de Classic Universal Login                                       |
| `/mfa/challenge`                    | solicitud de desafío                                                                  |
| `/mfa/associate`                    | asociación de autenticador                                                            |
| `/passwordless/start`               | Passwordless: solicitud de inicio                                                     |
| `/samlp/*`                          | endpoints del protocolo SAML                                                          |
| `/wsfed/*`                          | endpoints de WS-Federation                                                            |
| `/v2/logout`                        | Puede invocarse del lado del servidor en flujos de cierre de sesión por back-channel  |

<div id="example-rule">
  ### Regla de ejemplo
</div>

Para aplicar Managed Challenges solo a los flujos de Universal Login basados en navegador, cree una [regla personalizada de WAF](https://developers.cloudflare.com/waf/custom-rules/) en Cloudflare. Configure la acción de la regla como **Managed Challenge** y use la siguiente expresión, sustituyendo `YOUR_CUSTOM_DOMAIN` por su dominio personalizado (por ejemplo, `login.example.com`):

```text
(http.host eq "YOUR_CUSTOM_DOMAIN" and (
  http.request.uri.path eq "/authorize" or
  starts_with(http.request.uri.path, "/u/") or
  http.request.uri.path eq "/login"
))
```

Esto limita el desafío únicamente a los endpoints de Universal Login con interfaz de usuario y evita interrumpir el tráfico de la API y de máquina a máquina.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Algunos casos de uso pueden comportarse de forma diferente:

  * **Persistencia de la cookie de validación**: Una vez que un navegador supera un Managed Challenge, Cloudflare emite una cookie de validación que normalmente persiste durante la sesión. Según tu configuración, limitar la regla solo a `/authorize` puede ser suficiente para cubrir todo el flujo de Universal Login sin aplicarla a cada ruta `/u/*`.
  * **Puntos de entrada no OAuth**: Los flujos que comienzan en puntos de entrada iniciados por SAML SP o WS-Federation usan `/samlp/*` o `/wsfed/*` en lugar de `/authorize`. Estas rutas están en la lista de exclusión y no se les debe aplicar un Managed Challenge.
</Callout>

  ## Configurar Auth0
</div>

Llama al <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip> de Auth0 [Update custom domain configuration](https://auth0.com/docs/api/management/v2#!/Custom_Domains/patch_custom_domains_by_id) endpoint con el siguiente payload en el cuerpo:

```json lines
{
  "custom_client_ip_header": "true-client-ip"
}
```
