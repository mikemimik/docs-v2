Para configurar las funcionalidades de Auth0 para que usen tu <Tooltip tip="Custom Domain: Dominio de terceros con un nombre especializado o personalizado." cta="Ver glosario" href="/docs/es/glossary?term=custom+domain">dominio personalizado</Tooltip>, es posible que debas completar pasos adicionales según las funcionalidades que estés usando. Por ejemplo, puede que necesites hacer cambios antes de poder usar tu dominio personalizado en tu página de inicio de sesión o para hacer llamadas a tus API.

Si has estado usando Auth0 durante algún tiempo y decides habilitar un dominio personalizado, tendrás que migrar tus aplicaciones existentes y actualizar la configuración como se describe a continuación, incluida la de cualquier VPN o firewall que uses. Ten en cuenta que las sesiones existentes creadas en `{yourDomain}` dejarán de ser válidas una vez que comiences a usar tu dominio personalizado, por lo que los usuarios tendrán que volver a iniciar sesión.

  ## Requisitos previos
</div>

Deberías haber configurado y verificado previamente tu dominio personalizado.

<div id="universal-login">
  ## Universal Login
</div>

Si usas [Auth0 Universal Login](/docs/es/authenticate/login/auth0-universal-login) y has personalizado la página de inicio de sesión, debes actualizar el código para que use tu dominio personalizado. Si usas la página de inicio de sesión **predeterminada** sin personalizar, no necesitas hacer ningún cambio. Para obtener más información, consulta Auth0 <Tooltip tip="Universal Login: Tu aplicación redirige a Universal Login, alojado en el servidor de autorización de Auth0, para verificar la identidad de un usuario." cta="Ver glosario" href="/docs/es/glossary?term=Universal+Login">Universal Login</Tooltip>.

Si usas [Lock for Web](/docs/es/libraries/lock), debes configurar las opciones `configurationBaseUrl` y `overrides`, como se muestra en el siguiente script de ejemplo:

```javascript lines
var lock = new Auth0Lock(config.clientID, config.auth0Domain, {
  //código omitido por brevedad
  configurationBaseUrl: config.clientConfigurationBaseUrl,
  overrides: {
  	__tenant: config.auth0Tenant,
  	__token_issuer: config.authorizationServer.issuer
  },
  //código omitido por brevedad
});
```

Si utiliza [Auth0.js](/docs/es/libraries/auth0js) en la página de Universal Login, debe configurar la opción `overrides`.

```javascript lines
var webAuth = new auth0.WebAuth({
  clientID: config.clientID,
  domain: config.auth0Domain,
  //código omitido por brevedad
  overrides: {
  	__tenant: config.auth0Tenant,
  	__token_issuer: config.authorizationServer.issuer
  },
  //código omitido por brevedad
});
```

En la mayoría de los casos, las bibliotecas Auth0.js y Lock obtienen el nombre del tenant (necesario para `/usernamepassword/login`) y el emisor (necesario para la validación de `id_token`) a partir del dominio. Sin embargo, si eres cliente de Private Cloud y usas un proxy o un dominio personalizado cuyo nombre de dominio es distinto del tenant/emisor, puedes usar `__tenant` y `__token_issuer` para proporcionar tus valores específicos.

<div id="embedded-lock">
  ## Embedded Lock
</div>

Si usas [Lock for Web](/docs/es/libraries/lock) integrado en tu aplicación, debes actualizar el código para que use tu dominio personalizado al inicializar Lock. También tendrás que configurar `configurationBaseUrl` con la URL de CDN adecuada.

});`;

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  La URL de la CDN varía según la región. Los tenant creados antes del 11 de junio de 2020 deben usar `https://cdn.auth0.com` si la región es Estados Unidos, o agregar `eu`, `au` o `jp` para Europa, Australia o Japón. Si tu tenant se creó después del 11 de junio de 2020, usa `https://cdn.us.auth0.com` si la región es Estados Unidos.
</Callout>

<div id="use-custom-domains-in-emails-and-phone-notifications">
  ## Usa dominios personalizados en correos electrónicos y notificaciones al teléfono
</div>

Si quieres usar tu dominio personalizado con los correos electrónicos o las notificaciones al teléfono de Auth0, debes habilitar esta funcionalidad.

1. Ve a [Auth0 Dashboard &gt; Branding &gt; Custom Domains](https://manage.auth0.com/#/branding/custom_domains).
2. Activa el interruptor **Use Custom Domain in Emails**.

<div id="configure-social-identity-providers">
  ## Configurar proveedores de identidad social
</div>

Si quieres usar tu dominio personalizado con <Tooltip tip="Proveedor de identidad (IdP): servicio que almacena y administra identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=identity+providers">proveedores de identidad</Tooltip> sociales (IdP), debes actualizar la lista de URI de redirección autorizadas de tu IdP para que incluya tu dominio personalizado (como `https://login.northwind.com/login/callback`).

No puedes usar [Auth0 developer keys](/docs/es/authenticate/identity-providers/social-identity-providers/devkeys) con dominios personalizados.

<div id="configure-google-workspace-connections">
  ## Configurar conexiones de Google Workspace
</div>

Si quieres usar tu dominio personalizado con conexiones de Google Workspace, debes actualizar el URI de redirección autorizado en la configuración del cliente de <Tooltip tip="OAuth 2.0: Marco de autorización que define protocolos y flujos de trabajo de autorización." cta="Ver glosario" href="/docs/es/glossary?term=OAuth">OAuth</Tooltip>. En Google Cloud Console, ve a **Credenciales**, elige tu cliente de OAuth en la lista y verás una página de configuración con el <Tooltip tip="Client ID: Valor de identificación asignado a tu recurso registrado por Auth0." cta="Ver glosario" href="/docs/es/glossary?term=Client+ID">ID de cliente</Tooltip> de la aplicación, el secreto y otros campos. En el campo **URI de redirección autorizados**, agrega una URL con el formato `https://<YOUR-CUSTOM-DOMAIN>/login/callback` que incluya tu dominio personalizado (como `https://login.northwind.com/login/callback`).

<div id="apis">
  ## APIs
</div>

Los identificadores de API (es decir, `audience`) no cambian. Es un valor constante para cada API y, aunque por convención se suele usar un URI, es completamente independiente del dominio que se utilice para obtener el token.

Auth0 emite tokens con el claim `iss` del dominio que se haya utilizado para obtener el token.

<div id="auth0-apis">
  ##### APIs de Auth0
</div>

Siga usando el nombre de dominio predeterminado de su tenant (como `https://{yourDomain}/userinfo` y `https://{yourDomain}/api/v2/`) en lugar de su dominio personalizado al especificar una `audience`. Este es el único caso en el que debe usar el dominio predeterminado de su tenant.

Todas las solicitudes (es decir, obtener el token y llamar a la API) deben usar el mismo dominio. Los tokens obtenidos mediante un dominio personalizado deben usarse en una API de Auth0 con ese mismo dominio personalizado.

Si usa un flujo de autenticación con su dominio personalizado para solicitar <Tooltip tip="Token de acceso: credencial de autorización, en forma de una cadena opaca o JWT, usada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=Access+Tokens">tokens de acceso</Tooltip> para acceder a la <Tooltip tip="Management API: un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip>, también debe llamar al endpoint de la Management API con su dominio personalizado.

```json lines
POST https://mycustomdomain.com/oauth/token
... // otros parámetros 
...
audience:https://defaulttenant.eu.auth0.com/api/v2/
```

Tu solicitud de token de acceso debería ser similar a

```json lines
GET https://mycustomdomain.com/api/v2/clients

Headers:
Authorization: Bearer <access_token>
```

<div id="custom-apis">
  ##### APIs personalizadas
</div>

Si usas Auth0 con un dominio personalizado para emitir tokens de acceso para tus APIs, debes validar el emisor o los emisores del <Tooltip tip="JSON Web Token (JWT): formato estándar de token de ID (y, a menudo, de Access Token) que se utiliza para representar claims de forma segura entre dos partes." cta="Ver glosario" href="/docs/es/glossary?term=JWT">JWT</Tooltip> en tu dominio personalizado. Por ejemplo, si usas el middleware [express-jwt](https://github.com/auth0/express-jwt), debes hacer el siguiente cambio:

```js lines
app.use(jwt({
  issuer: 'https://<YOUR-CUSTOM-DOMAIN>/',
  //código omitido por simplicidad
}));
```

<div id="configure-saml-identity-providers">
  ## Configurar proveedores de identidad SAML
</div>

Para usar tu dominio personalizado con <Tooltip tip="Security Assertion Markup Language (SAML): Protocolo estandarizado que permite a dos partes intercambiar información de autenticación sin una contraseña." cta="Ver glosario" href="/docs/es/glossary?term=SAML">SAML</Tooltip> proveedores de identidad (IdP), debes actualizar tus **URL del Servicio de Consumo de Aserciones (ACS)** en los proveedores de identidad. Según lo que admita el IdP, puedes hacerlo de una de estas dos maneras:

1. Puedes obtener los metadatos del proveedor de servicios de Auth0 en `https://<YOUR-CUSTOM-DOMAIN>/samlp/metadata?connection=<YOUR-CONNECTION-NAME>`. Esto incluirá la URL ACS actualizada. Luego, debes actualizar manualmente este valor en la configuración de tus IdP. Este cambio en tus IdP debe hacerse al mismo tiempo que empiezas a usar tu dominio personalizado en tus aplicaciones. Esto puede suponer un problema si hay varios IdP que configurar.
2. Si el IdP lo admite, puedes usar solicitudes firmadas para cumplir este requisito:

* Descarga el certificado de firma desde `https://<TENANT>.auth0.com/pem`. Ten en cuenta que `https://<YOUR-CUSTOM-DOMAIN>.com/pem` devolverá el mismo certificado
* Entrega el certificado a los IdP para que lo carguen. Esto permite que el IdP valide la firma del mensaje `AuthnRequest` que Auth0 envía al IdP
* El IdP importará el certificado y, si es necesario, deberá habilitarse la verificación de firma (los pasos exactos varían según el IdP)
* Activa el interruptor **Sign Request** en el Dashboard, en **Connections &gt; Enterprise &gt; SAML &gt; CONNECTION**. Esto hará que Auth0 firme los mensajes SAML `AuthnRequest` que envía al IdP.

Una vez hecho esto, cuando empieces a usar tu dominio personalizado al iniciar una solicitud de autenticación en tu aplicación, el IdP recibirá ese dominio personalizado en tu solicitud firmada. Como la solicitud firmada de tu aplicación es de confianza, el IdP debería sustituir automáticamente cualquier valor configurado como URL ACS por el valor enviado en la solicitud firmada. Sin embargo, hay IdP que **no** aceptan la URL ACS en la solicitud firmada, así que primero debes confirmar con tu proveedor si esto es compatible o no.

Si esto es compatible, evitarás tener que cambiar una o varias configuraciones de IdP todas al mismo tiempo y podrás prepararlas de antemano para que acepten tus solicitudes firmadas. Después, también podrás cambiar más adelante la URL ACS configurada estáticamente en la configuración de tu IdP.

Ten en cuenta que, si tu proveedor de identidad SAML está configurado para usar tu dominio personalizado, probar la conexión mediante el botón **Try** en el Dashboard **no** funcionará, y los enlaces predeterminados para descargar metadatos desde Auth0 siempre mostrarán el dominio predeterminado, no el dominio personalizado.

Si tienes un flujo de autenticación iniciado por el IdP, tendrás que actualizar los IdP y tus aplicaciones al mismo tiempo para usar el dominio personalizado.

<div id="configure-saml-applications">
  ## Configurar aplicaciones SAML
</div>

Si quieres usar tu dominio personalizado con aplicaciones SAML (cuando Auth0 es el IdP), debes actualizar tu proveedor de servicios con nuevos metadatos del proveedor de identidad de Auth0. Puedes obtener los metadatos actualizados que reflejan el dominio personalizado en `https://<YOUR-CUSTOM-DOMAIN>/samlp/metadata/<YOUR-CLIENT-ID>`. Ten en cuenta que el ID de entidad del emisor de la aserción devuelta por Auth0 cambiará al usar un dominio personalizado (de algo como `urn:northwind.auth0.com` a otro con el dominio personalizado, como `urn:login.northwind.com`).

Si tienes un flujo de autenticación iniciado por IdP, tendrás que actualizar la URL que se utiliza para invocarlo de modo que refleje el dominio personalizado. En lugar de `https://<TENANT>.auth0.com/samlp/<YOUR-CLIENT-ID>`, debes usar `https://<YOUR-CUSTOM-DOMAIN>/samlp/<YOUR-CLIENT-ID>`.

<div id="configure-ws-fed-applications">
  ## Configurar aplicaciones WS-Fed
</div>

Si quieres usar tu dominio personalizado con aplicaciones <Tooltip tip="Web Service Federation (WS-Fed): Protocolo para administrar identidades de usuario entre dominios." cta="Ver glosario" href="/docs/es/glossary?term=WS-Fed">WS-Fed</Tooltip> con Auth0 como IdP, debes actualizar tu proveedor de servicios con los nuevos metadatos del proveedor de identidad de Auth0. Puedes obtener los metadatos correspondientes al dominio personalizado en `https://<YOUR-CUSTOM-DOMAIN>/wsfed/FederationMetadata/2007-06/FederationMetadata.xml`.

<div id="configure-azure-ad-connections">
  ## Configurar conexiones de Azure AD
</div>

Si quieres usar tu dominio personalizado con conexiones de Azure AD, debes actualizar la Allowed Reply URL en la configuración de Azure AD. En Azure Active Directory, ve a **Apps registrations** y selecciona tu aplicación. Después, haz clic en **Settings -&gt; Reply URLs** y añade una URL con tu dominio personalizado con el formato `https://<YOUR-CUSTOM-DOMAIN>/login/callback` (por ejemplo, `https://login.northwind.com/login/callback`).

<div id="configure-adfs-connections">
  ## Configurar conexiones de ADFS
</div>

Si quieres usar tu dominio personalizado con conexiones de ADFS, debes actualizar el endpoint en la configuración de ADFS. Debes actualizarlo para que use tu dominio personalizado en la URL de devolución de llamada con el formato `https://<YOUR-CUSTOM-DOMAIN>/login/callback` (por ejemplo, `https://login.northwind.com/login/callback`).

<div id="configure-adldap-connections">
  ## Configurar conexiones AD/LDAP
</div>

Si no necesita compatibilidad con Kerberos, las conexiones AD/LDAP no requieren ninguna configuración adicional.

Para usar conexiones AD/LDAP con compatibilidad con Kerberos, deberá actualizar el endpoint de Ticket para que funcione con el dominio personalizado. Como se menciona en la [documentación del Conector AD/LDAP de Auth0](/docs/es/authenticate/identity-providers/enterprise-identity-providers/active-directory-ldap/ad-ldap-connector/ad-ldap-connector-to-auth0#point-an-ad-ldap-connector-to-a-new-connection), es necesario modificar el archivo `config.json` y cambiar el valor de `PROVISIONING_TICKET` para que use su dominio personalizado con el formato `https://<YOUR-CUSTOM-DOMAIN>/p/ad/jUG0dN0R`.

Una vez guardado este cambio, debe reiniciar el servicio del Conector AD/LDAP para que surta efecto.
