Para configurar las funciones de Auth0 para que usen tu <Tooltip tip="Dominio personalizado: dominio de terceros con un nombre especializado o de vanidad." cta="Ver glosario" href="/docs/es/glossary?term=custom+domain">dominio personalizado</Tooltip>, es posible que tengas que completar pasos adicionales según las funciones que estés usando. Por ejemplo, puede que necesites hacer cambios antes de poder usar tu dominio personalizado en la página de inicio de sesión o para llamar a tus API.

Si llevas un tiempo usando Auth0 y decides habilitar un dominio personalizado, tendrás que migrar tus aplicaciones existentes y actualizar la configuración como se describe a continuación, incluida la de cualquier VPN o firewall que uses. Ten en cuenta que las sesiones existentes creadas en `{yourDomain}` dejarán de ser válidas una vez que empieces a usar tu dominio personalizado, por lo que los usuarios tendrán que volver a iniciar sesión.

<div id="universal-login">
  ## Universal Login
</div>

Si usas [Auth0 Universal Login](/docs/es/authenticate/login/auth0-universal-login) y has personalizado la página de inicio de sesión, debes actualizar el código para que use tu dominio personalizado. Si usas la página de inicio de sesión **predeterminada** sin personalizar, no necesitas hacer ningún cambio. Para obtener más información, consulta el <Tooltip tip="Universal Login: Tu aplicación redirige a Universal Login, alojado en el servidor de autorización de Auth0, para verificar la identidad de un usuario." cta="Ver glosario" href="/docs/es/glossary?term=Universal+Login">Universal Login</Tooltip> de Auth0.

Si usas [Lock for Web](/docs/es/libraries/lock), debes configurar las opciones `configurationBaseUrl` y `overrides`, como se muestra en el siguiente script de ejemplo:

```javascript lines
var lock = new Auth0Lock(config.clientID, config.auth0Domain, {
  //código omitido por brevedad
  configurationBaseUrl: config.clientConfigurationBaseUrl,
  overrides: {
  	__tenant: config.auth0Tenant,
  	__token_issuer: config.authorizationServer.issuer
  },
  //código omitido por brevedad
});
```

Si usas [Auth0.js](/docs/es/libraries/auth0js) en la página de Universal Login, debes configurar la opción `overrides`.

```javascript lines
var webAuth = new auth0.WebAuth({
  clientID: config.clientID,
  domain: config.auth0Domain,
  //código omitido por brevedad
  overrides: {
  	__tenant: config.auth0Tenant,
  	__token_issuer: config.authorizationServer.issuer
  },
  //código omitido por brevedad
});
```

En la mayoría de los casos, las bibliotecas Auth0.js y Lock obtienen el nombre del tenant (necesario para `/usernamepassword/login`) y el emisor (necesario para validar `id_token`) a partir del dominio. Sin embargo, si eres cliente de Private Cloud y usas un proxy o un nombre de dominio personalizado en el que el nombre de dominio es distinto del tenant/emisor, puedes usar `__tenant` y `__token_issuer` para proporcionar tus propios valores.

<div id="use-custom-domains-in-emails-and-phone-notifications">
  ## Usa dominios personalizados en correos electrónicos y notificaciones por teléfono
</div>

Si quieres usar tu dominio personalizado con tus correos electrónicos o notificaciones por teléfono de Auth0, debes habilitar esta función.

1. Ve a [Auth0 Dashboard &gt; Branding &gt; Custom Domains](https://manage.auth0.com/#/branding/custom_domains).
2. Activa el interruptor **Use Custom Domain in Emails**.

<div id="configure-social-identity-providers">
  ## Configurar proveedores de identidad sociales
</div>

Si quieres usar tu dominio personalizado con proveedores de <Tooltip tip="Proveedor de identidad (IdP): servicio que almacena y administra identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=identity+providers">identidad</Tooltip> sociales (IdP), debes actualizar la lista de URI de redirección autorizadas de tu IdP para incluir tu dominio personalizado (por ejemplo, `https://login.northwind.com/login/callback`).

No puedes usar las [claves de desarrollador de Auth0](/docs/es/authenticate/identity-providers/social-identity-providers/devkeys) con dominios personalizados.

<div id="configure-google-workspace-connections">
  ## Configurar conexiones de Google Workspace
</div>

Si quieres usar tu dominio personalizado con conexiones de Google Workspace, debes actualizar el URI de redirección autorizada en la configuración de tu cliente <Tooltip tip="OAuth 2.0: Marco de autorización que define protocolos y flujos de trabajo de autorización." cta="Ver glosario" href="/docs/es/glossary?term=OAuth">OAuth</Tooltip>. En Google Cloud Console, ve a **Credentials**, elige tu cliente OAuth en la lista y verás una página de configuración con el <Tooltip tip="Client ID: Valor de identificación asignado a tu recurso registrado en Auth0." cta="Ver glosario" href="/docs/es/glossary?term=Client+ID">Client ID</Tooltip> de la aplicación, el secreto y otros campos. En el campo **Authorized redirect URIs**, agrega una URL con el formato `https://<YOUR-CUSTOM-DOMAIN>/login/callback` que incluya tu dominio personalizado (por ejemplo, `https://login.northwind.com/login/callback`).

<div id="apis">
  ## APIs
</div>

Los identificadores de API (es decir, `audience`) no cambiarán. Este es un valor constante para cada API y, aunque es habitual usar un URI, es completamente independiente del dominio utilizado para obtener el token.

Auth0 emite tokens con el `claim` `iss` del dominio que hayas utilizado para obtener el token.

<div id="auth0-apis">
  ##### APIs de Auth0
</div>

Sigue usando el nombre de dominio predeterminado de tu tenant (como `https://{yourDomain}/userinfo` y `https://{yourDomain}/api/v2/`) en lugar de tu dominio personalizado al especificar un `audience`. Este es el único caso en el que debes usar el dominio predeterminado de tu tenant.

Todas las solicitudes (es decir, la obtención del token y la llamada a la API en sí) deben usar el mismo dominio. Los tokens obtenidos a través de un dominio personalizado deben usarse en una API de Auth0 que utilice ese mismo dominio personalizado.

Si usas un flujo de autenticación con tu dominio personalizado para solicitar <Tooltip tip="Token de acceso: credencial de autorización, en forma de una cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=Access+Tokens">tokens de acceso</Tooltip> para acceder a la <Tooltip tip="Management API: un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip>, también debes llamar al endpoint de la Management API con tu dominio personalizado.

```json lines
POST https://mycustomdomain.com/oauth/token
... // otros parámetros 
...
audience:https://defaulttenant.eu.auth0.com/api/v2/
```

Tu solicitud de token de acceso debería ser similar a

```json lines
GET https://mycustomdomain.com/api/v2/clients

Headers:
Authorization: Bearer <access_token>
```

<div id="custom-apis">
  ##### APIs personalizadas
</div>

Si usas Auth0 con un dominio personalizado para emitir tokens de acceso para tus APIs, debes validar que el emisor o los emisores del <Tooltip tip="JSON Web Token (JWT): formato estándar de token de ID (y, a menudo, de token de acceso) utilizado para representar claims de forma segura entre dos partes." cta="Ver glosario" href="/docs/es/glossary?term=JWT">JWT</Tooltip> coincidan con tu dominio personalizado. Por ejemplo, si usas el middleware [express-jwt](https://github.com/auth0/express-jwt), debes realizar el siguiente cambio:

```js lines
app.use(jwt({
  issuer: 'https://<YOUR-CUSTOM-DOMAIN>/',
  //código omitido por simplicidad
}));
```

<div id="configure-saml-identity-providers">
  ## Configurar proveedores de identidad SAML
</div>

Para usar tu dominio personalizado con proveedores de identidad (IdP) de <Tooltip tip="Security Assertion Markup Language (SAML): Protocolo estandarizado que permite a dos partes intercambiar información de autenticación sin una contraseña." cta="Ver glosario" href="/docs/es/glossary?term=SAML">SAML</Tooltip>, debes actualizar tus **URL de Assertion Consumer Service (ACS)** en los proveedores de identidad. Según lo que admita el IdP, puedes hacerlo de una de estas dos maneras:

1. Puedes obtener los metadatos del proveedor de servicios desde Auth0 en `https://<YOUR-CUSTOM-DOMAIN>/samlp/metadata?connection=<YOUR-CONNECTION-NAME>`. Esto incluirá la URL de ACS actualizada. Luego, debes actualizar manualmente este valor en la configuración de tus IdP. Este cambio en tus IdP debe hacerse al mismo tiempo que empieces a usar tu dominio personalizado en tus aplicaciones. Esto puede suponer un problema si hay varios IdP que configurar.
2. Si el IdP lo admite, puedes usar solicitudes firmadas para cumplir este requisito:

* Descarga el certificado de firma desde `https://<TENANT>.auth0.com/pem`. Ten en cuenta que `https://<YOUR-CUSTOM-DOMAIN>.com/pem` devolverá el mismo certificado
* Entrega el certificado a los IdP para que lo suban. Esto permite que el IdP valide la firma del mensaje `AuthnRequest` que Auth0 envía al IdP
* El IdP importará el certificado y, si es necesario, se deberá habilitar la verificación de firma (los pasos exactos varían según el IdP)
* Activa el interruptor **Sign Request** en el Dashboard, en **Connections &gt; Enterprise &gt; SAML &gt; CONNECTION**. Esto hará que Auth0 firme los mensajes SAML `AuthnRequest` que envía al IdP.

Una vez hecho esto, cuando empieces a usar tu dominio personalizado al iniciar una solicitud de autenticación en tu aplicación, el IdP recibirá ese dominio personalizado en tu solicitud firmada. Como la solicitud firmada de tu aplicación es de confianza, el IdP debería reemplazar automáticamente cualquier valor configurado como tu URL de ACS por el valor enviado en la solicitud firmada. Sin embargo, hay IdP que **no** aceptan la URL de ACS en la solicitud firmada, por lo que primero debes consultar con el tuyo para confirmar si lo admite o no.

Si esto es compatible, evitará que tengas que cambiar una o varias configuraciones de IdP exactamente al mismo tiempo y te permitirá prepararlas con antelación para que acepten tus solicitudes firmadas. Después, también podrás cambiar más adelante la URL de ACS configurada estáticamente en la configuración de tu IdP.

Ten en cuenta que, si tu proveedor de identidad SAML está configurado para usar tu dominio personalizado, probar la conexión mediante el botón **Try** en el Dashboard **no** funcionará y los enlaces predeterminados para descargar metadatos desde Auth0 siempre mostrarán el dominio predeterminado, no el dominio personalizado.

Si tienes un flujo de autenticación iniciado por IdP, tendrás que actualizar los IdP y tus aplicaciones al mismo tiempo para usar el dominio personalizado.

<div id="configure-saml-applications">
  ## Configurar aplicaciones SAML
</div>

Si desea usar su dominio personalizado con aplicaciones SAML (cuando Auth0 es el IdP), debe actualizar su proveedor de servicios con nuevos metadatos del proveedor de identidad de Auth0. Puede obtener los metadatos actualizados que reflejan el dominio personalizado en `https://<YOUR-CUSTOM-DOMAIN>/samlp/metadata/<YOUR-CLIENT-ID>`. Tenga en cuenta que el ID de entidad del emisor de la aserción devuelta por Auth0 cambiará al usar un dominio personalizado (de algo como `urn:northwind.auth0.com` a otro con el dominio personalizado, como `urn:login.northwind.com`).

Si tiene un flujo de autenticación iniciado por el IdP, deberá actualizar la URL que se usa para invocar ese flujo de autenticación, de modo que refleje el dominio personalizado. En lugar de `https://<TENANT>.auth0.com/samlp/<YOUR-CLIENT-ID>`, debe usar `https://<YOUR-CUSTOM-DOMAIN>/samlp/<YOUR-CLIENT-ID>`.

<div id="configure-ws-fed-applications">
  ## Configurar aplicaciones WS-Fed
</div>

Si quieres usar tu dominio personalizado con las aplicaciones <Tooltip tip="Web Service Federation (WS-Fed): Protocolo para gestionar identidades de usuario entre dominios." cta="Ver glosario" href="/docs/es/glossary?term=WS-Fed">WS-Fed</Tooltip> con Auth0 como IdP, debes actualizar tu proveedor de servicios con los nuevos metadatos del proveedor de identidad de Auth0. Puedes obtener los metadatos correspondientes al dominio personalizado en `https://<YOUR-CUSTOM-DOMAIN>/wsfed/FederationMetadata/2007-06/FederationMetadata.xml`.

<div id="configure-azure-ad-connections">
  ## Configurar conexiones de Azure AD
</div>

Si quieres usar tu dominio personalizado con conexiones de Azure AD, debes actualizar la URL de respuesta permitida en la configuración de Azure AD. En Azure Active Directory, ve a **Apps registrations** y selecciona tu aplicación. Luego haz clic en **Settings -&gt; Reply URLs** y añade una URL con tu dominio personalizado con el formato `https://<YOUR-CUSTOM-DOMAIN>/login/callback` (por ejemplo, `https://login.northwind.com/login/callback`).

<div id="configure-adfs-connections">
  ## Configurar conexiones ADFS
</div>

Si quieres usar tu dominio personalizado con conexiones ADFS, debes actualizar el endpoint en la configuración de ADFS. Deberás actualizarlo para que use tu dominio personalizado en la URL de callback, con el formato `https://<YOUR-CUSTOM-DOMAIN>/login/callback` (por ejemplo, `https://login.northwind.com/login/callback`).

<div id="configure-adldap-connections">
  ## Configurar conexiones AD/LDAP
</div>

Si no necesita compatibilidad con Kerberos, las conexiones AD/LDAP no requieren ninguna configuración adicional.

Para usar conexiones AD/LDAP con compatibilidad con Kerberos, deberá actualizar el endpoint de Ticket para que funcione con el dominio personalizado. Como se menciona en la [documentación del conector AD/LDAP de Auth0](/docs/es/authenticate/identity-providers/enterprise-identity-providers/active-directory-ldap/ad-ldap-connector/ad-ldap-connector-to-auth0#point-an-ad-ldap-connector-to-a-new-connection), es necesario modificar el archivo `config.json` y cambiar el valor de `PROVISIONING_TICKET` para que use su dominio personalizado con el formato `https://<YOUR-CUSTOM-DOMAIN>/p/ad/jUG0dN0R`.

Una vez guardado este cambio, debe reiniciar el servicio del conector AD/LDAP para que surta efecto.
