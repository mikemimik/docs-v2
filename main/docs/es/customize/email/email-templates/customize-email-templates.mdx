<Warning>
  Para personalizar las plantillas de correo electrónico, debes [configurar un proveedor externo de correo electrónico SMTP](/docs/es/customize/email/smtp-email-providers). Las plantillas de correo electrónico no están disponibles cuando se usa el proveedor de correo electrónico integrado de Auth0.
</Warning>

<div id="customize-email-templates">
  ## Personaliza las plantillas de correo electrónico
</div>

Para personalizar una plantilla de correo electrónico:

1. Ve a [Dashboard &gt; Branding &gt; Plantillas de correo electrónico](https://manage.auth0.com/#/templates).

2. En el menú desplegable **Template**, selecciona la plantilla de correo electrónico que quieres actualizar.

3. En la página de la plantilla de correo electrónico, actualiza los campos que quieras personalizar.

   Los campos **Dirección del remitente**, **Asunto**, **Redirect To** y **Message** admiten Liquid. Para obtener más información, consulta [Sintaxis de Liquid admitida](/docs/es/customize/email/email-templates/supported-liquid-syntax).

4. Haz clic en **Save** para guardar los cambios, en **Try** para probarlos o en **Reset** para revertirlos.

<div id="from-address">
  ### Dirección del remitente
</div>

El campo **Dirección del remitente** establece la dirección de correo electrónico que los usuarios ven como remitente cuando reciben un correo de Auth0. Si se deja sin configurar, los correos usarán la dirección de correo electrónico del campo **From** configurado para tu proveedor de correo electrónico.

Al configurar el campo **Dirección del remitente**, para permitir que Auth0 envíe correos firmados digitalmente en tu nombre, debes configurar dos formas de autenticación de correo electrónico:

* [Sender Policy Framework (SPF)](https://en.wikipedia.org/wiki/Sender_Policy_Framework), que autoriza direcciones IP específicas para enviar correo desde un dominio

* [DomainKeys Identified Mail (DKIM)](https://en.wikipedia.org/wiki/DKIM), que firma criptográficamente los correos para que los servidores de correo puedan verificar que fueron enviados desde el dominio indicado

Sin SPF y DKIM configurados, es posible que los correos no se entreguen, que los filtros de spam los bloqueen o que muestren &quot;On Behalf Of&quot; y revelen el remitente (tu proveedor de correo electrónico) además de la dirección del remitente.

Para configurar SPF y DKIM, debes crear registros TXT para tu dominio. Los valores de los registros TXT y otros detalles del proceso varían, así que sigue las instrucciones de tu proveedor de correo electrónico para esta configuración.

En general, el registro TXT para SPF debe tener el nombre de host establecido en `@` o vacío, y el valor en `v=spf1 include:<YOUR_PROVIDER_SPF_DOMAIN> -all`. El registro TXT para DKIM debe tener el nombre de host establecido en el dominio que usas para enviar correos y el valor en la firma DKIM que generes con tu proveedor.

<div id="subject">
  ### Asunto
</div>

El campo **Asunto** define el asunto del correo electrónico. Si no se configura, Auth0 completa automáticamente el asunto según el tipo de correo electrónico.

<div id="message">
  ### Message
</div>

El campo **Message** define el contenido HTML del cuerpo del mensaje. Cada plantilla incluye un cuerpo de mensaje predeterminado que puedes modificar o descartar por completo para escribir el tuyo propio.

<div id="url-lifetime-and-redirect-to">
  ### URL lifetime y Redirect To
</div>

Las plantillas de correo electrónico que incluyen un enlace (**Verification Email (Link)**, **Change Password (Link)** y **Blocked Account Email**) tienen dos campos adicionales para gestionar esos enlaces:

* El campo **URL lifetime** establece cuánto tiempo sigue siendo válido un enlace antes de expirar. De forma predeterminada, la duración es de 432.000 segundos (cinco días).

* El campo **Redirect To** establece la URL a la que se redirige al usuario después de completar la acción desde el enlace incluido.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Actualmente, Universal Login ignora el valor del campo **Redirect To** en la plantilla **Restablecimiento de contraseña** y, en su lugar, redirige a la [ruta de inicio de sesión predeterminada](/docs/es/authenticate/login/auth0-universal-login/configure-default-login-routes) o a una [página de error](/docs/es/authenticate/login/auth0-universal-login/error-pages).

  Para personalizar la URL de **Redirect To** del **Restablecimiento de contraseña** cuando uses Universal Login, utiliza [`api.transaction.setResultURL()` del trigger `post-challenge` de Actions](/docs/es/customize/actions/explore-triggers/password-reset-triggers/post-challenge-trigger/post-challenge-api-object#api-transaction-setresulturl-url,-options)
</Callout>

Se agregan dos parámetros de consulta a la URL de **Redirect To**:

* `success`, con el valor `true` o `false`, que indica si la acción se completó correctamente
* `message`, con una descripción adicional del resultado, como &quot;El acceso expiró.&quot; o &quot;Tu correo electrónico fue verificado. Puedes seguir usando la aplicación.&quot;

Si tienes problemas con la ubicación de los parámetros de consulta en las URL de **Redirect To** para aplicaciones de página única (SPA), la siguiente solución alternativa puede ayudarte.

<Accordion title="Solución alternativa para los parámetros de consulta de la URL Redirect To en SPFs">
  [RFC 3986](https://tools.ietf.org/html/rfc3986#section-3) define el orden esperado de una URL como `scheme|authority|path|query|fragment`. Sin embargo, los frameworks de SPA (como Angular) normalmente esperan las URL en el formato `scheme|authority|path|fragment|query`, donde la consulta va después del fragmento.

  Esto puede causar un problema con la ubicación de los parámetros de consulta en las URL de **Redirect To**. Si la URL de **Redirect To** de tu SPA es `http://localhost:3000/#/register`, se redirige al usuario a `http://localhost:3000/?exampleParameter=exampleValue#/register` en lugar de a `http://localhost:3000/#/register?exampleParameter=exampleValue`.

  Para solucionar esta limitación de los frameworks de SPA, puedes hacer lo siguiente:

  1. Agrega una URL del lado del servidor como URL de **Redirect To** con un parámetro `route` que registre la ruta de la SPA para la redirección.

     Por ejemplo, `http://localhost:3000/register?route=register`.

  2. Crea un controlador de ruta del lado del servidor que lea `route` y los demás parámetros de la URL, redirija a la ruta de la SPA especificada en el parámetro `route` y agregue los demás parámetros recibidos de Auth0.

     Por ejemplo:

     ```javascript lines wrap
     var express = require('express');
     var router = express.Router();
     // Para leer los parámetros de la cadena de consulta y convertirlos en cadena:
     var qs = require('qs');

     router.get('/register', function(req, res, next) {
         // Recupera el parámetro route que contiene la ruta
         // del lado del cliente de la SPA a la que debe redirigirse al usuario.
         var route = req.query.route;

         // Elimina route de los parámetros de consulta.
         delete req.query.route;

         // Envía una redirección 302 para la ruta esperada.
         res.redirect('http://localhost:3000/#/' + route + '?' +  qs.stringify(req.query));
     });

     module.exports = router;
     ```
</Accordion>

<div id="test-updated-templates">
  ## Probar las plantillas actualizadas
</div>

Para probarlas, haz clic en **Try**, introduce una dirección de correo electrónico válida a la que tengas acceso y elige el tipo de conexión correcto. Auth0 envía el correo para una aplicación predeterminada que comparte el nombre de tu tenant (no el nombre descriptivo de tu tenant).

Para probar plantillas de distintas aplicaciones, crea un usuario de prueba para que recorra los flujos correspondientes.

Puedes activar manualmente correos de verificación para aplicaciones y usuarios concretos mediante el endpoint [Send an email address verification email](https://auth0.com/docs/api/management/v2#!/Jobs/post_verification_email) de la Management API.

<div id="example-customization-use-cases">
  ## Casos de uso de personalización
</div>

Personalizar las plantillas de correo electrónico permite muchos casos de uso. Por ejemplo:

<AccordionGroup>
  <Accordion title="URL dinámica de Redirect To">
    Puedes configurar distintas URL de **Redirect To** según el nombre de tu aplicación. Por ejemplo:

    ```liquid
    {% if application.name == 'JWT.io' %}
        https://jwt.io
    {% else %}
        https://auth0.com
    {% endif %}
    ```

    Como el nombre de la aplicación se codifica por seguridad, usa un valor codificado (especialmente si el nombre de tu aplicación contiene algún carácter que cambia al codificarse). Por ejemplo, usa `My%20App` en lugar de `My App`.
  </Accordion>

  <Accordion title="Asunto y Message multilingües">
    Con Liquid, puedes usar el parámetro `request_language` para obtener la configuración de idioma del valor de la cabecera o, de forma predeterminada, la configuración de idioma del navegador del usuario.

    Por ejemplo:

    ```liquid lines
    {% assign language = request_language %}
      {% if language %}
        {% assign language = request_language | slice: 0,2 %}
      {% endif %}
    {% if language == 'es' %}
      Cuenta de Example: bloqueada
    {% elsif language == 'de' %}
      Ihr Example wurde gesperrt
    {% elsif language == 'fr' %}
      Compte Example bloqué
    {% elsif language == 'ja' %}
      Example アカウントがブロックされました
    {% elsif language == 'pt' %}
      Conta da Example bloqueada
    {% elsif language == 'zh' %}
      Example　帐户被阻止
    {% else %}
      Example account blocked
    {% endif %}
    ```

    También puedes usar la propiedad `user_metadata.lang` para adaptar el contenido según el idioma preferido del usuario. Por ejemplo, puedes [usar una Action](/docs/es/customize/actions/actions-overview) para establecer la propiedad `user_metadata.lang` y luego leer el parámetro `user_metadata.lang` en tus plantillas de correo electrónico para enviar correos en el idioma adecuado.
  </Accordion>
</AccordionGroup>

Para un control más detallado, puedes enviar correos electrónicos fuera de sus flujos de trabajo estándar y crear tickets (URL generadas para acciones del flujo de trabajo de correo electrónico, como restablecimientos de contraseña) con la Management API. Los endpoints de correo electrónico y tickets de la Management API tienen parámetros adicionales que te permiten personalizar su comportamiento.

Para obtener más información, consulta [Personaliza el manejo de correo electrónico y tickets con la Management API](/docs/es/customize/email/manage-email-flow).
