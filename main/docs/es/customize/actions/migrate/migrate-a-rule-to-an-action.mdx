description: Descripción de cómo usar la herramienta de migración de Rule en el Auth0 Dashboard
title: Herramienta de migración
---

Simplifica el proceso de migración de reglas a Actions con la herramienta de migración de Rule del [Auth0 Dashboard](https://manage.auth0.com/). Además de facilitar el paso de una Rule a una Action, la nueva herramienta también ofrece orientación integrada y correcciones para patrones comunes de Rule.

<div id="before-migration">
  ## Antes de la migración
</div>

Antes de comenzar la migración, visita [Migrate from Rules to Actions](/docs/es/customize/actions/migrate/migrate-from-rules-to-actions). Esta guía de migración es un excelente recurso para entender las diferencias entre reglas y Actions, y resulta muy útil durante el proceso de migración.

<div id="rule-migration-order">
  ### Orden de migración de Rule
</div>

Las reglas se ejecutan antes que Actions, lo que implica que las reglas deben migrarse en orden inverso al del runtime. Migra la última Rule que se ejecuta en tu flujo actual y continúa en orden cronológico inverso hasta la primera Rule.

No migrar en este orden puede provocar errores debido a efectos secundarios imprevistos causados por dependencias en el orden de reglas y Actions.

<div id="safely-rolling-back">
  ### Cómo revertir de forma segura
</div>

La herramienta de migración te permite revertir los cambios de forma fácil y segura si algo sale mal. Vuelve a habilitar tu Rule y omite la lógica de la Action correspondiente.

<div id="migrating-a-rule">
  ## Migración de una Rule
</div>

Para empezar a migrar su Rule, seleccione **pipeline de autenticación &gt; reglas**. Una vez hecho esto, sus reglas aparecerán con una opción para migrar cada una de ellas individualmente.

<Frame>![](/docs/images/cdy7uua7fh8z/79DmJp8C2vYZpdos51Z0nM/5a665d04c66dd9f0b8f0e25198cb9e4e/SS1_Dashboard_migration_screenshot.png)</Frame>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  La migración solo comienza cuando elige **Migrate to Action**. Como desarrollador, puede elegir cuándo activar la nueva Action.
</Callout>

<div id="migrate-to-action">
  ### Migrate to Action
</div>

Después de seleccionar **Migrate to Action**, la herramienta le solicita que cree una nueva Action.

<Frame>![](/docs/images/cdy7uua7fh8z/1A8FnVyvqAwW8hbrkOQDoG/f83ba9d5157db14ac2a3a666b2280574/SS2_Migrate_to_Action_screenshot.png)</Frame>

Después de seleccionar **Create,** se crea la nueva Action y se muestra el Actions Editor.

<Frame>![](/docs/images/cdy7uua7fh8z/2A10fE69aVngEcL2ZQCqWX/720dca2a8e6005c9378efad3bc140af0/Screenshot_2024-06-06_at_11.55.38_AM.png)</Frame>

Si la Rule se migra a Actions y sigue activa, la Rule finaliza antes de que se ejecute la lógica.

Esto funciona llamando a `api.rules.wasExecuted()`. Si la función `wasExecuted` recibe una cadena que contiene el ID de la Rule, devuelve `true` si la Rule ya se ejecutó (y, en caso contrario, devuelve `false`).

Esta es una herramienta útil para evitar que la lógica se ejecute dos veces. En cuanto deshabilite la Rule a la que se hace referencia, la función devuelve `false` y la lógica de negocio pasa a ejecutarse en la Action.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  El valor del ID de la Rule se completa automáticamente para cualquier rule que se migre mediante la funcionalidad **Migrate to Action**.
</Callout>

Para obtener más información sobre la función `api.rules.wasExecuted()`, consulte [Actions Triggers: Post Login - objeto de API](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger/post-login-api-object).

<div id="rewrite-rule-code">
  ## Reescribir el código de Rule
</div>

Selecciona el botón **reglas** en el Actions Editor para cargar la herramienta de migración de reglas. Allí podrás consultar rápidamente el código de la Rule migrada. Debes revisar el código de la Rule seleccionada en la barra lateral.

Copia el contenido de la función de Rule y pégalo debajo del comentario `// YOUR CODE HERE` en la Action. Es probable que este código no esté adaptado al modelo de programación de Actions.

<Frame>![](/docs/images/cdy7uua7fh8z/2jWCZFnfERuN7NzBPrpCDn/80ba271699893a007dac7baaf2dd44cb/Screenshot_2024-06-06_at_11.57.03_AM.png)</Frame>

El Actions Editor indica dónde debe refactorizarse el código. El editor también ofrece sugerencias para corregir automáticamente la mayoría de los casos; para revisar la sugerencia o la corrección, pasa el cursor sobre el código subrayado:

<Frame>![](/docs/images/cdy7uua7fh8z/5jRGI4iDX9ZgwcPRADTbZ1/caa90923a76800526871b137806e2af3/Screenshot_2024-06-06_at_11.58.07_AM.png)</Frame>

Es posible que puedas aplicar actualizaciones específicas de Actions seleccionando una opción en el modal **Quick Fix**.

<Frame>![](/docs/images/cdy7uua7fh8z/qTS4RD6cjWFVfI0T9Uph3/45776c464f7247ddd8e91860f4646f2e/Screenshot_2024-06-06_at_11.58.48_AM.png)</Frame>

Cuando seleccionas la corrección recomendada, el código se actualiza automáticamente para que sea compatible.

<Frame>![](/docs/images/cdy7uua7fh8z/1cAq5A1MNVLPAsinHjtROh/c71276afd1f6df9e01d71cc869778907/Screenshot_2024-06-06_at_11.59.37_AM.png)</Frame>

Hay algunas situaciones en las que las correcciones no pueden aplicarse automáticamente, pero aun así se proporcionan indicaciones:

<Frame>![](/docs/images/cdy7uua7fh8z/4Lx3RxJh8v4tj1J2SpPVvU/0ff35b0f40bb24d195ab2f4b06817d86/Screenshot_2024-06-06_at_12.00.22_PM.png)</Frame>

Por ejemplo: en el caso de la función `callback` de la Rule, se recomienda usar [`api.access.deny`](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/pre-user-registration-trigger/pre-user-registration-api-object) para los errores, o simplemente `return` en los casos exitosos.

<div id="test-the-new-action">
  ### Prueba la nueva Action
</div>

Las Actions pueden probarse directamente en el editor, ya que esto forma parte esencial del proceso de migración. Para obtener más información, consulta [Test Actions](/docs/es/customize/actions/test-actions).

<Warning>
  Te recomendamos encarecidamente que valides toda la lógica de migración en un tenant de desarrollo antes de habilitarla en producción.
</Warning>

<div id="automatically-enabling-the-action">
  ### Habilitación automática de la Action
</div>

Cuando se actualice el código de la Action y estés listo para hacer el cambio:

1. Selecciona **Deploy**.
2. En la alerta que confirma la implementación, selecciona **Add to flow**.
3. Agrega la Action que acabas de crear al flujo y luego selecciona **Apply.**
4. Selecciona **Continuar** **y deshabilita cualquier regla seleccionada** para habilitar la nueva Action y deshabilitar la Rule original al mismo tiempo. Si aún no estás listo, selecciona **Cancel** y podrás deshabilitar la Rule más adelante para activar la lógica de la Action migrada.

Para obtener más información sobre cómo agregar tu Action al flujo, consulta [Write your first Action](/docs/es/customize/actions/write-your-first-action).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Si vuelves a habilitar una Rule, se omitirá la lógica de la Action correspondiente.
</Callout>

<div id="migrate-remaining-rules">
  ## Migrar las reglas restantes
</div>

Este proceso puede repetirse para migrar cada Rule de tu flujo de inicio de sesión. Recuerda que las reglas deben migrarse en el orden inverso a aquel en que se ejecutan durante el flujo de inicio de sesión.
