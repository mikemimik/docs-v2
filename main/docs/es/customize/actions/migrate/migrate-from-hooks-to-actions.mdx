description: Aprende a migrar tu código existente de Auth0 Hooks al código de Auth0 Actions.
title: Migrar de Hooks a Actions
---

Al convertir Hooks existentes en Actions, debes asociar la nueva Action con el trigger correspondiente al tipo de Hook. Si sigues los pasos a continuación y usas los mapeos que identificamos en ellos, la funcionalidad debería ser idéntica.

<div id="plan-your-migration">
  ## Planifica tu migración
</div>

Las Actions desplegadas se ejecutan después de los Hooks activos, por lo que puedes convertir los Hooks uno por uno en el Dashboard o todos a la vez mediante la <Tooltip tip="Management API: un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip>.

Tendrás que convertir el código y, después, activar la Action y desactivar el Hook. Activar la Action y desactivar el Hook puede hacerse rápidamente y en sucesión, pero, según el orden, podría haber un breve período en el que se ejecuten ambos o ninguno.

Por eso, te recomendamos migrar tu pipeline paso a paso: convierte partes del código de tus Hooks en código de Action, haz pruebas en un entorno de staging y luego pasa a producción una parte cada vez. Como los Hooks activos se ejecutan antes que las Actions desplegadas, puedes mantener parte de la lógica en Hooks mientras desarrollas y pruebas otra lógica en Actions.

<Card title="Consejos para planificar tu migración">
  * Usa indicadores para evitar duplicar operaciones costosas o que solo deben ejecutarse una vez.
  * Asegúrate de realizar los cambios en un momento en que el impacto y el tráfico sean mínimos.
  * Considera usar [Auth0 Deploy CLI](/docs/es/deploy-monitor/deploy-cli-tool) para crear scripts, hacer pruebas e implementar rápidamente toda la migración de una vez o de forma iterativa.
</Card>

<div id="understand-limitations">
  ## Entienda las limitaciones
</div>

Aunque Actions puede encargarse de la gran mayoría de las tareas que Hooks puede realizar, debe tener en cuenta algunas limitaciones antes de comenzar la migración. (Recuerde: puede tener Hooks y Actions ejecutándose al mismo tiempo mientras migra).

* Actions no puede persistir datos, como <Tooltip tip="Token de acceso: credencial de autorización, en forma de cadena opaca o JWT, que se utiliza para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=access+tokens">tokens de acceso</Tooltip> o respuestas de API, entre ejecuciones.
* Actions no proporciona [un token de acceso para la Management API](/docs/es/customize/rules/use-management-api) ni [acceso al objeto global `auth0`](/docs/es/customize/actions/action-coding-guidelines#global-object), como sí ocurre en Hooks.

Para ver la lista completa de limitaciones, consulte [Actions Limitations](/docs/es/customize/actions/limitations).

<div id="convert-code">
  ## Convertir código
</div>

Para convertir un Hook en una Action, debes reemplazar el código específico de Hooks por código de Actions. En esta sección se describen las tareas que debes realizar para convertir un Hook funcional en su Action correspondiente.

<Card title="Consejos para convertir código">
  * En general, busca en el objeto `event` de Actions las propiedades de solo lectura de los objetos que se pasan a la función de Hooks.
  * Usa el Actions Code Editor en el Auth0 Dashboard para escribir tu código; te ayudará resaltando errores y ofreciéndote sugerencias de autocompletado.
  * Antes de pasar a producción, [prueba exhaustivamente tus nuevas Actions](/docs/es/customize/actions/test-actions) en un [entorno de staging o de prueba](/docs/es/get-started/auth0-overview/create-tenants/set-up-multiple-environments).
</Card>

<div id="copy-hook-code-to-a-new-action">
  ### Copia el código del Hook en una Action nueva
</div>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Te recomendamos copiar el código de tu Hook en una Action nueva y usar el Actions Code Editor en el Auth0 Dashboard; te ayudará a identificar los problemas pendientes en tu código.
</Callout>

1. Inicia sesión en tu tenant de producción y copia el código del Hook que quieras convertir.
2. Cambia a un tenant de no producción y ve a [Auth0 Dashboard &gt; Actions &gt; Library](https://manage.auth0.com/#/select-tenant?path=/actions/library).
3. Selecciona **Build Custom** y, a continuación:

   * Introduce un **Name** para tu Action que coincida con el nombre del Hook que estás convirtiendo.

   * Busca **Trigger** y selecciona el trigger adecuado**:**

     | Tipo de Hook                           | Actions Trigger        |
     | -------------------------------------- | ---------------------- |
     | Intercambio de credenciales de cliente | M2M/Client-Credentials |
     | Registro previo de usuario             | Pre User Registration  |
     | Registro posterior de usuario          | Post User Registration |
     | Posterior al cambio de contraseña      | Post Change Password   |
     | Enviar mensaje telefónico              | Enviar mensaje telefónico     |

   * Busca **Runtime** y selecciona **Node 18.**

   * Selecciona **Create**.
4. En el bloque de código del Actions Code Editor, pega el código del Hook que quieras convertir debajo de la función exportada.
5. Realiza los cambios detallados en el resto de este artículo a medida que integras el código en la función.
   También deberías leer sobre el objeto `event` asociado al nuevo Actions Trigger; verás enlaces a la documentación correspondiente cuando llegues a la sección [Cambiar la forma de acceder a los datos](#change-how-data-is-accessed) más adelante en esta guía.

<div id="change-the-function-declaration">
  ### Cambiar la declaración de la función
</div>

Las funciones de Hooks se exportan mediante una exportación por defecto, mientras que las funciones de Actions usan exportaciones con nombre. Según el tipo de Hook que esté convirtiendo, la exportación con nombre cambiará. Los mapeos incluyen:

| Tipo de Hook                           | Exportación con nombre          |
| -------------------------------------- | ------------------------------- |
| Intercambio de credenciales de cliente | `onExecuteCredentialsExchange`  |
| Pre-User Registration                  | `onExecutePreUserRegistration`  |
| Post-User Registration                 | `onExecutePostUserRegistration` |
| Posterior al cambio de contraseña      | `onExecutePostChangePassword`   |
| Enviar mensaje telefónico              | `onExecuteSendPhoneMessage`     |

**Antes**

```javascript lines
module.exports = async function myHooksFunction(){}
```

**Después**

```javascript lines
// Intercambio de credenciales de cliente
exports.onExecuteCredentialsExchange = async (event, api) => {}

// Registro previo del usuario
exports.onExecutePreUserRegistration = async (event, api) => {}

// Registro posterior del usuario
exports.onExecutePostUserRegistration = async (event) => {}

// Posterior al cambio de contraseña
exports.onExecutePostChangePassword = async (event) => {}

// Enviar mensaje telefónico
exports.onExecuteSendPhoneMessage = async (event) => {}
```

<div id="convert-dependencies">
  ### Convertir las dependencias
</div>

Hooks y Actions manejan las dependencias de forma similar. En ambos casos, las dependencias se agregan por separado a través de la UI o la Management API y se incluyen en el código. Además, en ambos casos, puedes usar `require` con cualquier paquete disponible en el registro de `npm`.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Si tus módulos de `npm` no están en la versión más reciente, ¡este es un buen momento para actualizarlos!
</Callout>

1. Busca declaraciones `require` dentro del código de tu Hook.
2. Elimina los números de versión, pero toma nota de ellos.
3. Agrega la dependencia siguiendo los pasos de la sección &quot;Add a Dependency&quot; de [Write Your First Action](/docs/es/customize/actions/write-your-first-action) (si la dependencia no es un [módulo nativo de NodeJS](https://github.com/nodejs/node/tree/master/lib); si la dependencia es un módulo nativo de NodeJS, no necesitas incluirla).
4. Mueve las declaraciones `require` que encontraste fuera de la declaración `function`.

<div id="convert-secrets">
  ### Convertir secretos
</div>

Hooks y Actions gestionan los secretos de forma similar. En ambos casos, los Secrets se agregan a cada Hook/Action a través de la UI o de la Management API y se incluyen en el código.

Para convertir secretos de Hooks a Actions:

1. Guarda los valores necesarios para la Action específica en la que estás trabajando.
2. Agrega un Secret por cada valor al que necesites acceder desde la Action. Para saber cómo hacerlo, consulta la sección **Add a Secret** en [Write Your First Action](/docs/es/customize/actions/write-your-first-action).
3. Convierte tu código:

**Antes**

```javascript lines
async function (user, context, cb) {
    const { SECRET_NAME } = context.webtask.secrets;

    // ... código adicional
}
```

**Después**

```javascript lines
async (event, api) => {
    const { SECRET_NAME } = event.secrets;

	// ... código adicional
};
```

Al igual que con Hooks, Auth0 cifra todos los valores secretos almacenados.

<div id="change-how-data-is-accessed">
  ### Cambiar la forma de acceder a los datos
</div>

Con Hooks, los datos del usuario, el cliente, la solicitud y otros datos contextuales se almacenan en varios argumentos que se pasan a la función Hook. En Actions, estos datos se han reorganizado y trasladado al objeto `event`. Muchas de las propiedades se trasladaron tal cual, pero algunas se han combinado para aportar mayor claridad.

Según el tipo de Hook que estés convirtiendo, el objeto `event` cambiará:

* [Intercambio de credenciales de cliente - objeto event de Actions](/docs/es/customize/actions/explore-triggers/machine-to-machine-trigger/credentials-exchange-event-object)
* [Posterior al cambio de contraseña - objeto event de Actions](/docs/es/customize/actions/explore-triggers/password-reset-triggers/post-change-password-trigger/post-change-password-event-object)
* [Post-User Registration - objeto event de Actions](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/post-user-registration-trigger/post-user-registration-event-object)
* [Registro previo de usuario - objeto event de Actions](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/pre-user-registration-trigger/pre-user-registration-event-object)
* [Enviar mensaje telefónico - objeto event de Actions](/docs/es/customize/actions/explore-triggers/mfa-notifications-trigger/send-phone-message-event-object)

**Antes**

```javascript lines
async function (user, context, cb) {
	const clientId = context.clientID;
	const tenant = context.connection.tenant

	// ... código adicional
}
```

**Después**

```javascript lines
async (event, api) => {
	const clientId = event.client.client_id;
	const tenant = event.tenant.id;

	// ... código adicional
};
```

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  A diferencia del objeto `context` de Hooks, los datos almacenados o modificados en las propiedades del objeto `event` no persisten en Actions posteriores. Si tu Hook está asignando datos a estas propiedades para activar la funcionalidad principal, tendrás que usar la interfaz `api`, disponible en los Flows de Actions de [Machine to Machine](/docs/es/customize/actions/explore-triggers/machine-to-machine-trigger/credentials-exchange-api-object) y [Registro previo de usuario](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/pre-user-registration-trigger/pre-user-registration-api-object), para conservar los datos entre Actions.
</Callout>

<div id="convert-callbacks">
  ### Convertir callbacks
</div>

Cuando un Hook termina de procesarse, debe llamar a la función `callback()` para completar su ejecución. En cambio, Actions no usan un mecanismo de callback; por lo tanto, deberá eliminar todas las instancias de `callback()` de su función de Actions.

Si antes usaba la función `callback()` en un Hook de Intercambio de credenciales de cliente o de Registro previo de usuario para rechazar la solicitud o actualizar un usuario, podrá seguir haciéndolo en Actions a través de una nueva interfaz `api`.

<div id="client-credentials-exchange">
  #### Intercambio de credenciales de cliente
</div>

Si agregabas claims adicionales al token de acceso en un Hook de Intercambio de credenciales de cliente:

```javascript lines
// Intercambio de credenciales de cliente Hook
module.exports = function(client, scope, audience, context, cb) {
  var access_token = {};
  access_token.scope = scope;

  access_token['https://example.com/claim'] = 'bar';
  cb(null, access_token);
};
```

Ahora puede usar el [objeto de API de Actions para el Intercambio de credenciales de cliente](/docs/es/customize/actions/explore-triggers/machine-to-machine-trigger/credentials-exchange-api-object):

```javascript lines
// Action de Intercambio de credenciales de cliente
exports.onExecuteCredentialsExchange = async (event, api) => {
  api.accessToken.setCustomClaim("https://example.com/claim", 'bar');  
};
```

<div id="pre-user-registration">
  #### Registro previo de usuario
</div>

Si estabas agregando claims adicionales al token de acceso en un Hook de Registro previo de usuario:

```javascript lines
// Hook de Registro previo de usuario
module.exports = function (user, context, cb) {
	if (user.app_metadata.condition === "success") {
      var response = {};
      response.user = { user_metadata: { favorite_color: "purple" } };
      // Este Hook se ejecutó correctamente, continúa con el siguiente Hook.
	  return callback(null, response);
	}

	if (user.app_metadata.condition === "failure") {
		// Este Hook falló, detén el inicio de sesión con una respuesta de error.
		return callback(new Error("Failure message"));
	}

	// ... código adicional
};
```

Ya puede usar el [objeto de API de Registro previo de usuario](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/pre-user-registration-trigger/pre-user-registration-api-object):

```javascript lines
// Action de Registro previo de usuario
exports.onExecutePreUserRegistration = async (event, api) => {
	if (event.user.app_metadata.condition === "success") {
		// Este Action se completó correctamente, continúa con el siguiente Action.
		api.user.setUserMetadata("favorite_color", "purple");
		return;
	}

	if (event.user.app_metadata.condition === "failure") {
		// Este Action falló, detén la llamada con una respuesta de error.
		return api.access.deny("Failure message");
	}

	// ... código adicional
};
```

<div id="complete-the-migration">
  ## Completa la migración
</div>

Una vez que hayas escrito y probado el nuevo código de Actions, debes activar la Action y desactivar el Hook. Estas dos tareas pueden hacerse rápidamente una tras otra, pero, según el orden, puede haber un breve período de tiempo en el que se estén ejecutando ambas o ninguna. Como los Hooks activos se ejecutan antes que las Actions desplegadas, puedes mantener parte de la lógica en Rules mientras desarrollas y pruebas otra lógica en Actions.
