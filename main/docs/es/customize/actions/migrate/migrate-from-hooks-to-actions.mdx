description: Aprende a migrar tu código existente de Auth0 Hooks a Auth0 Actions.
title: Migrar de Hooks a Actions
---

Al convertir Hooks existentes en Actions, debes asociar la nueva Action con el trigger correspondiente al tipo de Hook. Si sigues los pasos que se indican a continuación y utilizas los mapeos que identificamos en ellos, la funcionalidad debería ser idéntica.

<div id="plan-your-migration">
  ## Planifica tu migración
</div>

Las Actions desplegadas se ejecutan después de los Hooks activos, así que puedes convertir los Hooks uno por uno en el Dashboard o todos a la vez mediante la <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip>.

Tendrás que convertir el código y luego activar la Action y desactivar el Hook. Activar la Action y desactivar el Hook puede hacerse rápidamente de forma consecutiva, pero, según el orden, podría haber un breve período en el que se estén ejecutando ambos o ninguno.

Por eso, te recomendamos migrar tu pipeline paso a paso: convierte partes del código de tus Hooks en código de Action, haz pruebas en un entorno de preproducción y luego pon en producción una parte cada vez. Como los Hooks activos se ejecutan antes que las Actions desplegadas, puedes mantener parte de la lógica en Hooks mientras desarrollas y pruebas otra lógica en Actions.

<Card title="Consejos para planificar tu migración">
  * Usa indicadores para evitar duplicar operaciones costosas o que solo deben ejecutarse una vez.
  * Asegúrate de aplicar los cambios en un momento en que el impacto y el tráfico sean mínimos.
  * Considera usar [Auth0 Deploy CLI](/docs/es/deploy-monitor/deploy-cli-tool) para crear scripts, probar e implementar rápidamente la migración de una sola vez o de forma iterativa.
</Card>

<div id="understand-limitations">
  ## Comprende las limitaciones
</div>

Aunque Actions puede encargarse de casi todo lo que Hooks puede hacer, debes tener en cuenta algunas limitaciones antes de comenzar la migración. (Recuerda: durante la migración, puedes tener Hooks y Actions ejecutándose al mismo tiempo.)

* Actions no puede conservar datos, como <Tooltip tip="Token de acceso: Credencial de Authorization, en forma de cadena opaca o JWT, que se usa para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=access+tokens">tokens de acceso</Tooltip> o respuestas de la API, entre ejecuciones.
* Actions no proporciona [un token de acceso para la Management API](/docs/es/customize/rules/use-management-api) ni [acceso al objeto global `auth0`](/docs/es/customize/actions/action-coding-guidelines#global-object), como sí ocurre en Hooks.

Para ver la lista completa de limitaciones, consulta [Actions Limitations](/docs/es/customize/actions/limitations).

<div id="convert-code">
  ## Convertir código
</div>

Para convertir un Hook en una Action, debe sustituir el código específico de Hooks por código de Actions. En esta sección se describen las tareas que deberá realizar para convertir un Hook funcional en su Action equivalente.

<Card title="Consejos para convertir código">
  * En general, busque las propiedades de solo lectura de los objetos que se pasan a la función de Hooks en el objeto `event` de Actions.
  * Use el Actions Code Editor en el Auth0 Dashboard para escribir su código; le ayudará resaltando errores y ofreciendo sugerencias de autocompletado.
  * Antes de pasar a producción, [pruebe exhaustivamente sus nuevas Actions](/docs/es/customize/actions/test-actions) en un [entorno de preproducción o de pruebas](/docs/es/get-started/auth0-overview/create-tenants/set-up-multiple-environments).
</Card>

<div id="copy-hook-code-to-a-new-action">
  ### Copie el código del Hook en una Action nueva
</div>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Le recomendamos copiar el código de su Hook en una Action nueva y usar el Actions Code Editor en el Auth0 Dashboard; esto le ayudará a identificar los problemas pendientes en su código.
</Callout>

1. Inicie sesión en su tenant de producción y copie el código del Hook que desea convertir.
2. Cambie a un tenant no productivo y vaya a [Auth0 Dashboard &gt; Actions &gt; Library](https://manage.auth0.com/#/select-tenant?path=/actions/library).
3. Seleccione **Build Custom** y, a continuación:

   * Introduzca un **Name** para su Action que coincida con el nombre del Hook que está convirtiendo.

   * Busque **Trigger** y seleccione el trigger adecuado**:**

     | Tipo de Hook                           | Trigger de Actions     |
     | -------------------------------------- | ---------------------- |
     | Intercambio de credenciales de cliente | M2M/Client-Credentials |
     | Registro previo de usuario             | Pre User Registration  |
     | Registro posterior de usuario          | Post User Registration |
     | Posterior al cambio de contraseña      | Post Change Password   |
     | Enviar mensaje telefónico              | Send Phone Message     |

   * Busque **Runtime** y seleccione **Node 18.**

   * Seleccione **Create**.
4. En el bloque de código del Actions Code Editor, pegue debajo de la función exportada el código del Hook que desea convertir.
5. Realice los cambios detallados en el resto de este artículo a medida que traslada el código a la función.
   También debería leer sobre el objeto `event` asociado al nuevo trigger de Actions; verá enlaces a la documentación correspondiente cuando llegue más adelante, en esta guía, a la sección [Cambie la forma de acceder a los datos](#change-how-data-is-accessed).

<div id="change-the-function-declaration">
  ### Cambiar la declaración de la función
</div>

Las funciones de Hooks se exportan mediante una exportación por defecto, mientras que las funciones de Actions usan exportaciones con nombre. Según el tipo de Hook que esté convirtiendo, la exportación con nombre cambiará. Los mapeos incluyen:

| Tipo de Hook                           | Exportación con nombre          |
| -------------------------------------- | ------------------------------- |
| Intercambio de credenciales de cliente | `onExecuteCredentialsExchange`  |
| Pre-User Registration                  | `onExecutePreUserRegistration`  |
| Post-User Registration                 | `onExecutePostUserRegistration` |
| Posterior al cambio de contraseña      | `onExecutePostChangePassword`   |
| Enviar mensaje telefónico              | `onExecuteSendPhoneMessage`     |

**Antes**

```javascript lines
module.exports = async function myHooksFunction(){}
```

**Después**

```javascript lines
// Intercambio de credenciales de cliente
exports.onExecuteCredentialsExchange = async (event, api) => {}

// Registro previo de usuario
exports.onExecutePreUserRegistration = async (event, api) => {}

// Registro posterior de usuario
exports.onExecutePostUserRegistration = async (event) => {}

// Posterior al cambio de contraseña
exports.onExecutePostChangePassword = async (event) => {}

// Enviar mensaje telefónico
exports.onExecuteSendPhoneMessage = async (event) => {}
```

<div id="convert-dependencies">
  ### Convertir las dependencias
</div>

Hooks y Actions gestionan las dependencias de forma similar. En ambos casos, las dependencias se agregan por separado mediante la UI o la Management API y se incluyen en el código. Además, en ambos casos, puedes usar cualquier paquete disponible en el `npm` registry.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Si tus módulos de `npm` no están en la versión más reciente, ¡este es un buen momento para actualizarlos!
</Callout>

1. Busca instrucciones `require` dentro del código de tu Hook.
2. Elimina los números de versión, pero anótalos.
3. Agrega la dependencia siguiendo los pasos de la sección &quot;Add a Dependency&quot; de [Write Your First Action](/docs/es/customize/actions/write-your-first-action) (si la dependencia no es un [módulo principal de NodeJS](https://github.com/nodejs/node/tree/master/lib); si la dependencia es un módulo principal de NodeJS, no necesitas incluirla).
4. Mueve las instrucciones `require` que encontraste fuera de la declaración de `function`.

<div id="convert-secrets">
  ### Convertir secrets
</div>

Hooks y Actions gestionan los secrets de forma similar. En ambos casos, los Secrets se agregan por Hook/Action a través de la UI o de Management API y se incluyen en el código.

Para convertir secrets de Hooks a Actions:

1. Guarda los valores necesarios para la Action específica en la que estás trabajando.
2. Agrega un Secret para cada valor al que necesites acceder desde la Action. Para saber cómo hacerlo, consulta la sección **Add a Secret** en [Write Your First Action](/docs/es/customize/actions/write-your-first-action).
3. Convierte tu código:

**Antes**

```javascript lines
async function (user, context, cb) {
    const { SECRET_NAME } = context.webtask.secrets;

    // ... código adicional
}
```

**Después**

```javascript lines
async (event, api) => {
    const { SECRET_NAME } = event.secrets;

	// ... código adicional
};
```

Al igual que con Hooks, Auth0 cifra todos los valores secretos en reposo.

<div id="change-how-data-is-accessed">
  ### Cambiar cómo se accede a los datos
</div>

Con Hooks, la información del usuario, el cliente, la solicitud y otros datos de contexto se almacena en varios argumentos que se pasan a la función del Hook. En Actions, estos datos se han reorganizado y trasladado al objeto `event`. Muchas de las propiedades se trasladaron sin cambios, pero algunas se han combinado para aportar mayor claridad.

Según el tipo de Hook que esté convirtiendo, el objeto `event` cambiará:

* [Intercambio de credenciales de cliente - Objeto `event` de Actions](/docs/es/customize/actions/explore-triggers/machine-to-machine-trigger/credentials-exchange-event-object)
* [Posterior al cambio de contraseña - Objeto `event` de Actions](/docs/es/customize/actions/explore-triggers/password-reset-triggers/post-change-password-trigger/post-change-password-event-object)
* [Post-User Registration - Objeto `event` de Actions](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/post-user-registration-trigger/post-user-registration-event-object)
* [Registro previo de usuario - Objeto `event` de Actions](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/pre-user-registration-trigger/pre-user-registration-event-object)
* [Enviar mensaje telefónico - Objeto `event` de Actions](/docs/es/customize/actions/explore-triggers/mfa-notifications-trigger/send-phone-message-event-object)

**Antes**

```javascript lines
async function (user, context, cb) {
	const clientId = context.clientID;
	const tenant = context.connection.tenant

	// ... código adicional
}
```

**Después**

```javascript lines
async (event, api) => {
	const clientId = event.client.client_id;
	const tenant = event.tenant.id;

	// ... código adicional
};
```

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  A diferencia del objeto `context` de Hooks, los datos almacenados o modificados en las propiedades del objeto `event` no persisten en Actions posteriores. Si tu Hook asigna datos a estas propiedades para activar funcionalidades principales, tendrás que usar la interfaz `api` disponible en los Flows de Actions de [máquina a máquina](/docs/es/customize/actions/explore-triggers/machine-to-machine-trigger/credentials-exchange-api-object) y [Registro previo de usuario](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/pre-user-registration-trigger/pre-user-registration-api-object) para conservar los datos entre Actions.
</Callout>

<div id="convert-callbacks">
  ### Convertir callbacks
</div>

Cuando un Hook termina de procesarse, debe llamar a la función `callback()` para completar su ejecución. En cambio, Actions no usa un mecanismo de callback; por lo tanto, deberá eliminar todas las instancias de `callback()` de su función de Actions.

Si antes usaba la función `callback()` en un Hook de Intercambio de credenciales de cliente o de Registro previo de usuario para hacer fallar la solicitud o actualizar un usuario, podrá seguir haciéndolo en Actions mediante una nueva interfaz `api`.

<div id="client-credentials-exchange">
  #### Intercambio de credenciales de cliente
</div>

Si añadías claims adicionales al token de acceso en un Hook de Intercambio de credenciales de cliente:

```javascript lines
// Intercambio de credenciales de cliente Hook
module.exports = function(client, scope, audience, context, cb) {
  var access_token = {};
  access_token.scope = scope;

  access_token['https://example.com/claim'] = 'bar';
  cb(null, access_token);
};
```

Ahora puedes usar el [objeto de API de Intercambio de credenciales de cliente de Actions](/docs/es/customize/actions/explore-triggers/machine-to-machine-trigger/credentials-exchange-api-object):

```javascript lines
// Action de Intercambio de credenciales de cliente
exports.onExecuteCredentialsExchange = async (event, api) => {
  api.accessToken.setCustomClaim("https://example.com/claim", 'bar');  
};
```

<div id="pre-user-registration">
  #### Registro previo de usuario
</div>

Si agregabas claims adicionales al token de acceso en un Hook de Registro previo de usuario:

```javascript lines
// Hook de registro previo de usuario
module.exports = function (user, context, cb) {
	if (user.app_metadata.condition === "success") {
      var response = {};
      response.user = { user_metadata: { favorite_color: "purple" } };
      // Este Hook se ejecutó correctamente, continuar con el siguiente Hook.
	  return callback(null, response);
	}

	if (user.app_metadata.condition === "failure") {
		// Este Hook falló, detener el inicio de sesión con una respuesta de error.
		return callback(new Error("Failure message"));
	}

	// ... código adicional
};
```

Ahora ya puede usar el [objeto de API de Registro previo de usuario](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/pre-user-registration-trigger/pre-user-registration-api-object):

```javascript lines
// Action de Registro previo de usuario
exports.onExecutePreUserRegistration = async (event, api) => {
	if (event.user.app_metadata.condition === "success") {
		// Este Action se completó correctamente, continuar con el siguiente Action.
		api.user.setUserMetadata("favorite_color", "purple");
		return;
	}

	if (event.user.app_metadata.condition === "failure") {
		// Este Action falló, detener la llamada con una respuesta de error.
		return api.access.deny("Failure message");
	}

	// ... código adicional
};
```

<div id="complete-the-migration">
  ## Completa la migración
</div>

Una vez que hayas escrito y probado el nuevo código de Actions, debes activar la Action y desactivar el Hook. Estas dos tareas pueden realizarse rápidamente una tras otra, pero, según el orden, podría haber un breve período de tiempo en el que estén en ejecución ambas o ninguna de las dos. Como los Hooks activos se ejecutan antes que las Actions desplegadas, puedes mantener parte de la lógica en Rules mientras desarrollas y pruebas otra lógica en Actions.
