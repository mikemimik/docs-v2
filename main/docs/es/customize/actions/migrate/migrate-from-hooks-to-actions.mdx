description: Aprende a migrar tu código existente de Auth0 Hooks a Auth0 Actions.
title: Migrar de Hooks a Actions
---

Al convertir Hooks existentes en Actions, debes asociar la nueva Action al trigger correspondiente al tipo de Hook. Si sigues los pasos a continuación y usas los mapeos que indicamos en ellos, la funcionalidad debería ser idéntica.

<div id="plan-your-migration">
  ## Planifica tu migración
</div>

Las Actions desplegadas se ejecutan después de los Hooks activos, así que puedes convertir los Hooks uno por uno en el Dashboard o todos de una vez con la <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip>.

Tendrás que convertir el código y luego activar la Action y desactivar el Hook. Activar la Action y desactivar el Hook puede hacerse rápidamente de forma consecutiva, pero, según el orden, podría haber un breve período en el que se estén ejecutando ambos o ninguno.

Por eso, te recomendamos migrar tu pipeline paso a paso: convierte partes del código de tus Hooks en código de Action, haz pruebas en un entorno de staging y luego pasa a producción una parte a la vez. Como los Hooks activos se ejecutan antes que las Actions desplegadas, puedes mantener parte de la lógica en Hooks mientras desarrollas y pruebas otra lógica en Actions.

<Card title="Consejos al planificar tu migración">
  * Usa indicadores para evitar duplicar operaciones costosas o que solo deben ejecutarse una vez.
  * Asegúrate de aplicar los cambios en un momento en que el impacto y el tráfico sean mínimos.
  * Considera usar [Auth0 Deploy CLI](/docs/es/deploy-monitor/deploy-cli-tool) para crear scripts, hacer pruebas e implementar rápidamente la migración de una sola vez o de forma iterativa.
</Card>

<div id="understand-limitations">
  ## Conozca las limitaciones
</div>

Aunque Actions puede hacer casi todo lo que Hooks, conviene tener en cuenta algunas limitaciones antes de empezar la migración. (Recuerde: durante la migración, puede tener Hooks y Actions ejecutándose al mismo tiempo).

* Actions no puede almacenar datos, como <Tooltip tip="Token de acceso: credencial de autorización, en forma de una cadena opaca o JWT, que se utiliza para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=access+tokens">tokens de acceso</Tooltip> o respuestas de la API, entre ejecuciones.
* Actions no proporciona [un token de acceso para la Management API](/docs/es/customize/rules/use-management-api) ni [acceso al objeto global `auth0`](/docs/es/customize/actions/action-coding-guidelines#global-object), como sí ocurre en Hooks.

Para ver la lista completa de limitaciones, consulte [Actions Limitations](/docs/es/customize/actions/limitations).

<div id="convert-code">
  ## Convertir código
</div>

Para convertir un Hook en una Action, debes reemplazar el código específico de Hook por código de Actions. En esta sección se explican las tareas que deberás realizar para convertir un Hook funcional en su Action equivalente.

<Card title="Consejos para convertir código">
  * En general, busca las propiedades de solo lectura de los objetos que se pasan a la función de Hooks en el objeto `event` de Actions.
  * Usa el Actions Code Editor en el Auth0 Dashboard para escribir tu código; te ayudará resaltando errores y ofreciendo sugerencias de autocompletado.
  * Antes de pasar a producción, [prueba a fondo tus nuevas Actions](/docs/es/customize/actions/test-actions) en un [entorno de staging o de prueba](/docs/es/get-started/auth0-overview/create-tenants/set-up-multiple-environments).
</Card>

<div id="copy-hook-code-to-a-new-action">
  ### Copiar el código de Hook en una nueva Action
</div>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Te recomendamos copiar el código de tu Hook en una nueva Action y usar el Actions Code Editor en el Auth0 Dashboard; esto te ayudará a identificar los problemas pendientes en tu código.
</Callout>

1. Inicia sesión en tu tenant de producción y copia el código del Hook que quieras convertir.
2. Cambia a un tenant no productivo y ve a [Auth0 Dashboard &gt; Actions &gt; Biblioteca](https://manage.auth0.com/#/select-tenant?path=/actions/library).
3. Selecciona **Build Custom** y luego:

   * Introduce un **Nombre** para tu Action que coincida con el nombre del Hook que estás convirtiendo.

   * Busca **Trigger** y selecciona el trigger adecuado**:**

     | Tipo de Hook                           | Trigger de Actions         |
     | -------------------------------------- | -------------------------- |
     | Intercambio de credenciales de cliente | M2M/Client-Credentials     |
     | Pre-User-Registration                  | Registro previo de usuario |
     | Post-User-Registration                 | Post User Registration     |
     | Post-Change-Password                   | Post Change Password       |
     | Send Phone Message                     | Enviar mensaje telefónico  |

   * Busca **Runtime** y selecciona **Node 18.**

   * Selecciona **Create**.
4. En el bloque de código del Actions Code Editor, pega el código del Hook que quieras convertir debajo de la función exportada.
5. Realiza los cambios detallados en el resto de este artículo a medida que mueves el código a la función.
   También deberías leer sobre el objeto `event` asociado al nuevo trigger de Actions; verás enlaces a la documentación correspondiente cuando llegues más adelante, en esta guía, a la sección [Cambiar cómo se accede a los datos](#change-how-data-is-accessed).

<div id="change-the-function-declaration">
  ### Cambiar la declaración de la función
</div>

Las funciones de Hooks se exportan mediante una exportación predeterminada, mientras que las funciones de Actions usan exportaciones con nombre. Según el tipo de Hook que estés convirtiendo, cambiará la exportación con nombre. Los mapeos incluyen:

| Tipo de Hook                           | Exportación con nombre          |
| -------------------------------------- | ------------------------------- |
| Intercambio de credenciales de cliente | `onExecuteCredentialsExchange`  |
| registro previo de usuario             | `onExecutePreUserRegistration`  |
| registro posterior de usuario          | `onExecutePostUserRegistration` |
| Posterior al cambio de contraseña      | `onExecutePostChangePassword`   |
| Enviar mensaje telefónico              | `onExecuteSendPhoneMessage`     |

**Antes**

```javascript lines
module.exports = async function myHooksFunction(){}
```

**Después**

```javascript lines
// Intercambio de credenciales de cliente
exports.onExecuteCredentialsExchange = async (event, api) => {}

// Registro previo de usuario
exports.onExecutePreUserRegistration = async (event, api) => {}

// Registro posterior de usuario
exports.onExecutePostUserRegistration = async (event) => {}

// Posterior al cambio de contraseña
exports.onExecutePostChangePassword = async (event) => {}

// Enviar mensaje telefónico
exports.onExecuteSendPhoneMessage = async (event) => {}
```

<div id="convert-dependencies">
  ### Convertir dependencias
</div>

Hooks y Actions gestionan las dependencias de forma similar. En ambos casos, las dependencias se agregan por separado a través de la UI o la Management API y se incluyen en el código. Además, en ambos casos, puedes usar cualquier paquete que esté disponible en el registro de `npm`.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Si tus módulos de `npm` no están en la versión más reciente, ¡este es un excelente momento para actualizarlos!
</Callout>

1. Busca sentencias `require` dentro del código de tu Hook.
2. Elimina los números de versión, pero toma nota de ellos.
3. Agrega la dependencia siguiendo los pasos de la sección &quot;Add a Dependency&quot; de [Write Your First Action](/docs/es/customize/actions/write-your-first-action) (si la dependencia no es un [módulo principal de NodeJS](https://github.com/nodejs/node/tree/master/lib); si la dependencia es un módulo principal de NodeJS, no necesitas incluirla).
4. Mueve las sentencias `require` encontradas fuera de la declaración `function`.

<div id="convert-secrets">
  ### Convertir secretos
</div>

Hooks y Actions gestionan los secretos de forma similar. En ambos casos, los Secret se agregan para cada Hook/Action a través de la UI o la Management API y se incluyen en el código.

Para convertir secretos de Hooks a Actions:

1. Guarda los valores necesarios para la Action específica en la que estás trabajando.
2. Agrega un Secret por cada valor al que necesites acceder desde la Action. Para saber cómo hacerlo, consulta la sección **Agregar un Secret** en [Write Your First Action](/docs/es/customize/actions/write-your-first-action).
3. Convierte tu código:

**Antes**

```javascript lines
async function (user, context, cb) {
    const { SECRET_NAME } = context.webtask.secrets;

    // ... código adicional
}
```

**Después**

```javascript lines
async (event, api) => {
    const { SECRET_NAME } = event.secrets;

	// ... código adicional
};
```

Al igual que con Hooks, Auth0 cifra todos los valores secretos cuando están almacenados.

<div id="change-how-data-is-accessed">
  ### Cambie la forma de acceder a los datos
</div>

Con Hooks, los datos sobre el usuario, el cliente, la solicitud y otros datos de contexto se almacenan en varios argumentos que se pasan a la función Hook. En Actions, estos datos se han reorganizado y trasladado al objeto `event`. Muchas de las propiedades se trasladaron sin cambios, pero algunas se han combinado para aportar mayor claridad.

Según el tipo de Hook que convierta, el objeto `event` variará:

* [Intercambio de credenciales de cliente - objeto `event` de Actions](/docs/es/customize/actions/explore-triggers/machine-to-machine-trigger/credentials-exchange-event-object)
* [Posterior al cambio de contraseña - objeto `event` de Actions](/docs/es/customize/actions/explore-triggers/password-reset-triggers/post-change-password-trigger/post-change-password-event-object)
* [registro posterior de usuario - objeto `event` de Actions](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/post-user-registration-trigger/post-user-registration-event-object)
* [registro previo de usuario - objeto `event` de Actions](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/pre-user-registration-trigger/pre-user-registration-event-object)
* [Enviar mensaje telefónico - objeto `event` de Actions](/docs/es/customize/actions/explore-triggers/mfa-notifications-trigger/send-phone-message-event-object)

**Antes**

```javascript lines
async function (user, context, cb) {
	const clientId = context.clientID;
	const tenant = context.connection.tenant

	// ... código adicional
}
```

**Después**

```javascript lines
async (event, api) => {
	const clientId = event.client.client_id;
	const tenant = event.tenant.id;

	// ... código adicional
};
```

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  A diferencia del objeto `context` de Hooks, los datos almacenados o modificados en las propiedades del objeto `event` no se conservan en las Actions posteriores. Si tu Hook establece datos en estas propiedades para activar funcionalidades principales, tendrás que usar la interfaz `api` disponible en los Flows de Actions [Machine to Machine](/docs/es/customize/actions/explore-triggers/machine-to-machine-trigger/credentials-exchange-api-object) y [Registro previo de usuario](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/pre-user-registration-trigger/pre-user-registration-api-object) para conservar los datos entre Actions.
</Callout>

<div id="convert-callbacks">
  ### Convertir las devoluciones de llamada
</div>

Cuando un Hook termina de procesar, debe llamar a la función `callback()` para completar su ejecución. Por el contrario, Actions no utilizan un mecanismo de devolución de llamada; por lo tanto, tendrás que eliminar todas las instancias de `callback()` de tu función de Actions.

Si antes usabas la función `callback()` en un Hook de Intercambio de credenciales de cliente o de Registro previo de usuario para rechazar la solicitud o actualizar un usuario, aún podrás hacerlo en Actions a través de una nueva interfaz `api`.

<div id="client-credentials-exchange">
  #### Intercambio de credenciales de cliente
</div>

Si añadías claims adicionales al token de acceso en un Hook de Intercambio de credenciales de cliente:

```javascript lines
// Hook de intercambio de credenciales de cliente
module.exports = function(client, scope, audience, context, cb) {
  var access_token = {};
  access_token.scope = scope;

  access_token['https://example.com/claim'] = 'bar';
  cb(null, access_token);
};
```

Ahora puede usar el [objeto de API de Actions para Intercambio de credenciales de cliente](/docs/es/customize/actions/explore-triggers/machine-to-machine-trigger/credentials-exchange-api-object):

```javascript lines
// Action de Intercambio de credenciales de cliente
exports.onExecuteCredentialsExchange = async (event, api) => {
  api.accessToken.setCustomClaim("https://example.com/claim", 'bar');  
};
```

<div id="pre-user-registration">
  #### Registro previo de usuario
</div>

Si estabas añadiendo claims adicionales al access token en un Hook de Registro previo de usuario:

```javascript lines
// Hook de Registro previo de usuario
module.exports = function (user, context, cb) {
	if (user.app_metadata.condition === "success") {
      var response = {};
      response.user = { user_metadata: { favorite_color: "purple" } };
      // Este Hook se ejecutó correctamente, continúa con el siguiente Hook.
	  return callback(null, response);
	}

	if (user.app_metadata.condition === "failure") {
		// Este Hook falló, detén el inicio de sesión con una respuesta de error.
		return callback(new Error("Failure message"));
	}

	// ... código adicional
};
```

Ahora puedes usar el [objeto de API de Registro previo de usuario](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/pre-user-registration-trigger/pre-user-registration-api-object):

```javascript lines
// Action de Registro previo de usuario
exports.onExecutePreUserRegistration = async (event, api) => {
	if (event.user.app_metadata.condition === "success") {
		// Este Action se ejecutó correctamente, continuar con el siguiente Action.
		api.user.setUserMetadata("favorite_color", "purple");
		return;
	}

	if (event.user.app_metadata.condition === "failure") {
		// Este Action falló, detener la llamada con una respuesta de error.
		return api.access.deny("Failure message");
	}

	// ... código adicional
};
```

<div id="complete-the-migration">
  ## Completa la migración
</div>

Una vez que hayas escrito y probado el código de tus nuevas Actions, debes activar la Action y desactivar el Hook. Estas dos tareas pueden realizarse rápidamente una tras otra, pero, según el orden, podría haber un breve periodo en el que ambas o ninguna se estén ejecutando. Como los Hooks activos se ejecutan antes que las Actions desplegadas, puedes mantener parte de la lógica en Rules mientras desarrollas y pruebas otra lógica en Actions.
