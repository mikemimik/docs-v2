description: Aprende a migrar tu código actual de Auth0 Rules a código de Auth0 Actions.
title: Migrar de Rules a Actions
---

Al convertir tus Rules existentes en Actions, debes asociar la nueva Action al trigger Post Login (`post-login`) del Login Flow. Si sigues los pasos que se indican a continuación y mantienes tus Actions en el mismo orden que tus Rules originales, la funcionalidad debería ser idéntica.

<div id="plan-your-migration">
  ## Planifica tu migración
</div>

Las Actions de Post Login se ejecutan después de las Rules existentes, por lo que puedes convertir las Rules una por una en el Dashboard o todas de una sola vez mediante la <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip>.

Tendrás que convertir el código y, después, activar la Action y desactivar la Rule. Esto puede hacerse rápidamente y de forma consecutiva, pero, según el orden, podría haber un breve período en el que se ejecuten ambas o ninguna.

Por ello, te recomendamos migrar tu pipeline paso a paso: convierte partes del código de tus Rules en código de Action, pruébalas en un entorno de staging y luego pásalas a producción una por una. Como las Rules activas se ejecutan antes que las Actions desplegadas, si empiezas por el final de tu pipeline de Rules y avanzas hacia atrás, podrás mantener parte de la lógica en Rules mientras desarrollas y pruebas otra lógica en Actions.

<Card title="Consejos para planificar tu migración">
  * Mantén una relación 1:1 entre tus Actions y Rules, para que la funcionalidad pueda activarse y desactivarse por bloques, y probarse.
  * Usa indicadores en los metadatos del usuario para evitar duplicar operaciones costosas o que solo deban ejecutarse una vez.
  * Empieza por el final de tu pipeline de Rules y avanza hacia atrás; como las Rules activas se ejecutan antes que las Actions desplegadas, puedes mantener parte de la lógica en Rules mientras desarrollas y pruebas otra lógica en Actions.
  * Asegúrate de aplicar los cambios en un momento en que el impacto y el tráfico sean mínimos.
  * Considera [personalizar temporalmente tu página de inicio de sesión](/docs/es/customize) para detener los inicios de sesión si el cambio pudiera provocar inicios de sesión no válidos o brechas de protección.
  * Considera usar [Auth0 Deploy CLI](/docs/es/deploy-monitor/deploy-cli-tool) para crear scripts, probar e implementar rápidamente la migración de una sola vez o de forma iterativa.
</Card>

<div id="understand-limitations">
  ## Comprende las limitaciones
</div>

Aunque Actions puede encargarse de la gran mayoría de las tareas que puede realizar Rules, debes tener en cuenta algunas limitaciones antes de comenzar la migración. (Recuerda: puedes tener tanto Rules como Actions ejecutándose mientras migras).

* Actions no incluye [un token de acceso para la Management API](/docs/es/customize/rules/use-management-api) ni acceso al objeto global `auth0`, como sucede con Rules. Para saber cómo se pueden seguir haciendo llamadas a la Management API, consulta la sección [Convertir código](#convert-code).

Para ver la lista completa de limitaciones, consulta [Actions Limitations](/docs/es/customize/actions/limitations).

<div id="convert-code">
  ## Convertir código
</div>

Para convertir una Rule en una Action, debes reemplazar el código específico de Rule por código de Actions. En esta sección se describen las tareas que debes realizar para convertir una Rule funcional en su Action equivalente.

<Card title="Consejos al convertir código">
  * En general, busca en el objeto `event` de Actions las propiedades de solo lectura de los objetos `user` y `context` de Rules. Busca en las funciones del objeto `api` cualquier efecto secundario que tus Actions tengan en el sistema, como interrumpir un inicio de sesión o actualizar los metadatos del usuario.
  * Usa el Actions Code Editor en el Auth0 Dashboard para escribir tu código; te ayudará resaltando errores y ofreciendo sugerencias de autocompletado.
  * Antes de ponerlo en producción, [prueba exhaustivamente tus nuevas Actions](/docs/es/customize/actions/test-actions) en un [entorno de staging o de prueba](/docs/es/get-started/auth0-overview/create-tenants/set-up-multiple-environments).
</Card>

<div id="copy-rule-code-to-a-new-action">
  ### Copia el código de la Rule en una nueva Action
</div>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Te recomendamos copiar el código de tu Rule en una nueva Action y usar el Actions Code Editor en el Auth0 Dashboard; te ayudará a identificar los problemas que aún tenga tu código.
</Callout>

1. Inicia sesión en tu tenant de producción y copia el código de la Rule que quieres convertir.
2. Cambia a un tenant de no producción y ve a [Auth0 Dashboard &gt; Actions &gt; Library](https://manage.auth0.com/#/select-tenant?path=/actions/library).
3. Selecciona **Build Custom** y luego:

   * Introduce un **Name** para tu Action que coincida con el nombre de la Rule que estás convirtiendo.
   * Busca **Trigger** y selecciona **Login / Post Login**.
   * Busca **Runtime** y selecciona **Node 16.**
   * Selecciona **Create**.
4. En el bloque de código del Actions Code Editor, pega el código de la Rule que quieres convertir debajo de la función exportada `onExecutePostLogin`.
5. Haz los cambios detallados en el resto de este artículo a medida que mueves el código a la función.

<div id="change-the-function-declaration">
  ### Cambia la declaración de la función
</div>

Rules usa una función simple declarada con los parámetros `user`, `context` y `callback`, mientras que Actions usa una función exportada con un nombre específico. Haz el siguiente cambio; por ahora, ignora cualquier error que aparezca.

**Antes**

```js lines
async function myRulesFunction(user, context, callback) {
    // ... código adicional
}
```

**Después**

```js lines
exports.onExecutePostLogin = async (event, api) => {
	// ... código adicional
};
```

<div id="change-how-user-data-is-accessed">
  ### Cambiar cómo se accede a los datos de usuario
</div>

En Rules, los datos del usuario que inicia sesión se almacenan en el [objeto `user`](/docs/es/customize/rules/user-object-in-rules). En Actions, estos datos se encuentran en la propiedad `user` del [objeto `event`](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger/post-login-event-object). Se puede acceder a la mayoría de las propiedades existentes desde esta nueva ubicación.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Los datos almacenados o modificados en las propiedades del objeto `event` no son accesibles desde otras Actions.
</Callout>

**Antes**

```javascript lines
function myRulesFunction(user, context, callback) {
	const userEmail = user.email;
	const userId = user.user_id;

	// Esta propiedad podría ser undefined en Rules.
	const userAppMetadata = user.app_metadata || {};

	// ... código adicional
}
```

**Después**

```js lines
exports.onExecutePostLogin = async (event, api) => {
	const userEmail = event.user.email;
	const userId = event.user.user_id;

	// Esta propiedad nunca será undefined en Actions.
	const userAppMetadata = event.user.app_metadata;

	// ... código adicional
};
```

<div id="change-how-context-data-is-accessed">
  ### Cambiar cómo se accede a los datos de contexto
</div>

En Rules, los datos de la sesión de inicio de sesión actual se almacenan en el [objeto `context`](/docs/es/customize/rules/context-object). En Actions, estos datos se han reorganizado y trasladado al [objeto `event`](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger/post-login-event-object). Muchas de las propiedades se trasladaron sin cambios, pero algunas se han combinado para aportar mayor claridad.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  No se puede acceder desde otras Actions a los datos almacenados o modificados en propiedades del objeto `event`. Si tu Rule activa alguna funcionalidad principal al asignar datos a estas propiedades, como `context.idToken` o `context.multifactor`, lee una de las secciones siguientes que corresponda a tu caso de uso.
</Callout>

**Antes**

```javascript lines
function myRulesFunction(user, context, callback) {
	const clientId = context.clientID;
	const clientMetadata = context.clientMetadata || {};

	const connectionId = context.connectionID;
	const connectionMetadata = context.connectionMetadata || {};

	const protocol = context.protocol;

	const tenant = context.tenant;

	// ... código adicional
}
```

**Después**

```js lines
exports.onExecutePostLogin = async (event, api) => {
	const clientId = event.client.client_id;
	const clientMetadata = event.client.metadata;

	const connectionId = event.connection.id;
	const connectionMetadata = event.connection.metadata;

	const protocol = event.transaction.protocol;

	const tenant = event.tenant.id;

	// ... código adicional
};
```

<div id="convert-dependencies">
  ### Convertir dependencias
</div>

Rules incluyen dependencias de una forma que exige incluir el número de versión en una instrucción `require`. Actions usan una sintaxis CommonJS más estándar y requieren que las versiones se indiquen fuera del editor de código.

En Rules, solo se permiten versiones concretas de paquetes concretos, y para agregar paquetes o versiones nuevos hay que hacer una solicitud a Auth0. En Actions, puedes usar cualquier paquete disponible en el registro de `npm`.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Si tus módulos de `npm` no están en la versión más reciente, ¡este es un buen momento para actualizarlos!
</Callout>

1. Busca instrucciones `require` dentro del código de tu Rule.
2. Elimina los números de versión, pero anótalos.
3. Agrega la dependencia siguiendo los pasos de la sección &quot;Add a Dependency&quot; de [Write Your First Action](/docs/es/customize/actions/write-your-first-action) (si la dependencia no es un [módulo principal de NodeJS](https://github.com/nodejs/node/tree/main/lib); si la dependencia es un módulo principal de NodeJS, no necesitas incluirla).
4. Mueve las instrucciones `require` encontradas fuera de la declaración `function`:

**Antes**

```javascript lines
function myRulesFunction(user, context, callback) {
	const dependency = require("dependency@1.2.3");

	// ... código adicional
}
```

**Después**

```javascript lines
const dependency = require("dependency"); // v1.2.3
exports.onExecutePostLogin = async (event, api) => {
	// ... código adicional
};
```

<div id="convert-callbacks">
  ### Convertir devoluciones de llamada
</div>

Cuando una Rule termina de ejecutarse, debe llamar a la función `callback()` y pasarle un error si el inicio de sesión falla. En cambio, las Actions pueden hacer `return` si se completan correctamente, o llamar a un método de `api` con un mensaje si el inicio de sesión falla. Todas las instancias de `callback()` en una Rule deben eliminarse o reemplazarse por `api.access.deny()` en caso de fallo. Tanto en Rules como en Actions, si el procesamiento debe detenerse por una condición específica, use una instrucción `return`.

**Antes**

```javascript lines
function myRulesFunction(user, context, callback) {
	const userAppMetadata = user.app_metadata || {};
	if (userAppMetadata.condition === "success") {
		// Esta Rule tuvo éxito, continuar con la siguiente Rule.
		return callback(null, user, context);
	}

	if (userAppMetadata.condition === "failure") {
		// Esta Rule falló, detener el inicio de sesión con una respuesta de error.
		return callback(new Error("Failure message"));
	}

	// ... código adicional
}
```

**Después**

```js lines
exports.onExecutePostLogin = async (event, api) => {
	if (event.user.app_metadata.condition === "success") {
		// Esta Action se realizó correctamente, continuar con la siguiente Action.
		return;
	}

	if (event.user.app_metadata.condition === "failure") {
		// Esta Action falló, detener el inicio de sesión con una respuesta de error.
		return api.access.deny("Failure message");
	}

	// ... código adicional
};
```

<div id="change-handling-of-secrets">
  ### Cambiar la gestión de secretos
</div>

En Rules, defines los valores de configuración de forma global, lo que significa que todas las Rules tienen acceso a todos los secretos. (Para obtener más información, consulta [Store Rule Configurations](/docs/es/customize/rules/configuration).) En Actions, defines los valores de configuración para cada Action de forma individual. No puedes acceder al valor secreto de una Action desde fuera del contexto de la Action.

Para convertir secretos de Rules a Actions:

1. Guarda los valores necesarios para la Action específica en la que estás trabajando.
2. Agrega un Secret para cada valor al que necesites acceder desde la Action. Para saber cómo hacerlo, consulta la sección **Add a Secret** de [Write Your First Action](/docs/es/customize/actions/write-your-first-action).
3. Convierte tu código:

**Antes**

```javascript lines
function myRulesFunction (user, context, callback) {
  const { CLIENT_ID, CLIENT_SECRET } = configuration;

  // ... código adicional
}
```

**Después**

```js lines
exports.onExecutePostLogin = async (event, api) => {
  const { CLIENT_ID, CLIENT_SECRET } = event.secrets;

  // ... código adicional
}
```

Al igual que con Rules, Auth0 cifra en reposo todos los valores secretos.

<div id="convert-custom-claims-in-tokens">
  ### Convertir claims personalizados en tokens
</div>

Tanto Rules como Actions pueden agregar claims personalizados a los tokens de ID y a los <Tooltip tip="Token de acceso: credencial de autorización, en forma de cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=access+tokens">tokens de acceso</Tooltip>. En Rules, esto es una propiedad del objeto `context`, mientras que en Actions se usa un método del [`objeto api`](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger/post-login-api-object).

**Antes**

```javascript lines
function myRulesFunction(user, context, callback) {
	const userAppMetadata = user.app_metadata || {};
	const namespace = "https://namespace/";

	context.idToken[`${namespace}/emp_id`] = userAppMetadata.emp_id;
	context.accessToken[`${namespace}/emp_id`] = userAppMetadata.emp_id;

	// ... código adicional
}
```

**Después**

```js lines
exports.onExecutePostLogin = async (event, api) => {
	const namespace = "https://namespace/";

	api.idToken.setCustomClaim(
		`${namespace}/emp_id`, 
		event.user.app_metadata.emp_id
	); 		   

	api.accessToken.setCustomClaim(
		`${namespace}/emp_id`, 
		event.user.app_metadata.emp_id
	);

	// ... código adicional
};
```

<div id="convert-multi-factor-triggering">
  ### Convertir la activación de la autenticación multifactor
</div>

En Rules, la <Tooltip tip="Autenticación multifactor (MFA): proceso de autenticación de usuario que utiliza un factor adicional además del nombre de usuario y la contraseña, como un código por SMS." cta="Ver glosario" href="/docs/es/glossary?term=multi-factor+authentication">autenticación multifactor</Tooltip> se puede activar modificando la propiedad `multifactor` del [objeto `context`](/docs/es/customize/rules/context-object). En Actions, esto se hace con un [método del objeto `api`](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger/post-login-api-object).

**Antes**

```javascript lines
function myRulesFunction(user, context, callback) {
	if (user.app_metadata.needs_mfa === true) {
		context.multifactor = { 
			provider: "any", 
			allowRememberBrowser: false,
		};
	}

	// ... código adicional
}
```

**Después**

```js lines
exports.onExecutePostLogin = async (event, api) => {
	if (event.user.app_metadata.needs_mfa === true) {
		api.multifactor.enable("any", { allowRememberBrowser: false });
	}

	// ... código adicional
};
```

<div id="convert-user-metadata-updates">
  ### Convertir actualizaciones de metadatos de usuario
</div>

Actualizar las propiedades `user_metadata` y `app_metadata` en Rules requiere llamar a la Management API, lo que puede provocar errores de [límite de frecuencia](/docs/es/troubleshoot/customer-support/operational-policies/rate-limit-policy/management-api-endpoint-rate-limits). Sin embargo, Actions permite indicar varios cambios en los metadatos del usuario y llamar a la Management API una sola vez.

**Antes**

```javascript lines
function myRulesFunction(user, context, callback) {
	user.app_metadata = user.app_metadata || {}; 
	user.app_metadata.roles = user.app_metadata.roles || [];
	user.app_metadata.roles.push("administrator"); 

	auth0.users
		.updateAppMetadata(user.user_id, user.app_metadata) 
		.then(() => callback(null, user, context))
		.catch((err) => callback(err));

	// ... código adicional
}
```

Si las Rules posteriores necesitan actualizar los metadatos del usuario, tendrían que llamar a la Management API por separado, lo que aumenta la probabilidad de que se alcance el [límite de frecuencia](/docs/es/troubleshoot/customer-support/operational-policies/rate-limit-policy/management-api-endpoint-rate-limits).

**Después**

```js lines
exports.onExecutePostLogin = async (event, api) => {
	const userRolesUpdated = event.user.app_metadata.roles || [];
	userRolesUpdated.push("administrator"); 

	// Observa los dos métodos diferentes aquí. 
	api.user.setAppMetadata("roles", userRolesUpdated);
	api.user.setUserMetadata("hasRoles", true);

	// ... código adicional
};
```

Si las Actions posteriores necesitaran actualizar los metadatos del usuario, tendrían que llamar a `api.user.setUserMetadata` o `api.user.setAppMetadata`. En Actions, varias llamadas a estas funciones en una o varias Actions darán lugar a una sola llamada a la Management API una vez que se complete el flujo.

<div id="convert-other-management-api-calls">
  ### Convertir otras llamadas a la Management API
</div>

En general, no recomendamos llamar a la Management API desde un flujo crítico de alto tráfico, como Rules o Actions. Las solicitudes a todas las API de Auth0 están sujetas a [límites de frecuencia](/docs/es/troubleshoot/customer-support/operational-policies/rate-limit-policy), incluidas las llamadas desde puntos de extensibilidad, y hacer una llamada a la API en cada inicio de sesión podría provocar fácilmente inicios de sesión fallidos en momentos de mucho tráfico.

Sin embargo, si esas llamadas son necesarias y están configuradas para evitar los límites de frecuencia, es posible llamar a la Management API desde Actions. Como se mencionó antes en la sección &quot;Comprender las limitaciones&quot; de este artículo, Actions no incluyen un token de acceso para la Management API, por lo que tendrás que obtener un token de acceso antes de activar tu Action:

1. [Registra una aplicación de máquina a máquina y autorízala para la Management API](/docs/es/get-started/auth0-overview/create-applications/machine-to-machine-apps).
2. Guarda el **ID de cliente** y el **secreto de cliente** en la Action.
3. [Obtén un token de acceso para la Management API](/docs/es/secure/tokens/access-tokens/management-api-access-tokens/get-management-api-access-tokens-for-production).
4. Llama a la Management API:

   <Warning>
     Las Actions no pueden conservar datos entre ejecuciones, por lo que no es posible almacenar en caché el token de acceso durante ningún período de tiempo; además, como cada llamada a la Management API también requiere una llamada a la Authentication API, llamar a la Management API es una operación muy costosa.
   </Warning>

**Antes**

```javascript lines
function myRulesFunction(user, context, callback) {
	const ManagementClient = require("auth0@2.9.1").ManagementClient; 
	const managementClientInstance = new ManagementClient({
		// Estos provienen de las variables globales integradas de Rules
		token: auth0.accessToken, 
		domain: auth0.domain,
	}); 

	managementClientInstance.users.assignRoles(
		{ id: user.user_id }, 
		{ roles: ["ROLE_ID_TO_ADD"] }, 
		(error, user) => {
			if (error) {
				return callback(error);
			}

			// ... código adicional
		}
	);
}
```

**Después**

```javascript lines
const auth0Sdk = require("auth0");
exports.onExecutePostLogin = async (event, api) => {
	const ManagementClient = auth0Sdk.ManagementClient;

	// Esto realizará una llamada a la Authentication API
	const managementClientInstance = new ManagementClient({
		// Estos provienen de una aplicación máquina a máquina
		domain: event.secrets.M2M_DOMAIN,
		clientId: event.secrets.M2M_CLIENT_ID,
		clientSecret: event.secrets.M2M_CLIENT_SECRET,
		scope: "update:users"
	});

	managementClientInstance.users.assignRoles(
		{ id: event.user.user_id }, 
		{ roles: ["ROLE_ID_TO_ADD"]}, 
		(error, user) => {
			if (error) {
				return api.access.deny(error.message);
			}

			// ... código adicional
		}
	);
};
```

<div id="convert-redirects">
  ### Convertir redirecciones
</div>

Rules puede redirigir a un usuario que está iniciando sesión a una página externa y luego esperar una respuesta. En este caso, todas las Rules anteriores a la redirección se ejecutarán dos veces: una antes de la redirección y otra al recibir la respuesta. La lógica de la redirección y de la respuesta suele estar contenida en la misma Rule.

En Actions, el pipeline de Action se pausa cuando se produce la redirección y se reanuda cuando el usuario regresa. Además, la función exportada que desencadena la redirección está separada de la devolución de llamada de la redirección.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Explicar cómo implementar correctamente todas las redirecciones en Actions va más allá del alcance de esta guía. Para obtener información más detallada, consulta [Redirect with Actions](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger/redirect-with-actions).
</Callout>

**Antes**

```javascript lines
function myRulesFunction(user, context, callback) {
    if (context.protocol === "redirect-callback") {
        // El usuario fue redirigido al endpoint /continue
        user.app_metadata.wasRedirected = true;
        return callback(null, user, context);
    } else if (
        context.protocol === "oauth2-password" ||
        context.protocol === "oauth2-refresh-token" ||
        context.protocol === "oauth2-resource-owner"
    ) {
        // El usuario no puede ser redirigido
        return callback(null, user, context);
    }
    // El usuario está iniciando sesión directamente
    if (!user.app_metadata.wasRedirected) {
        context.redirect = {
            url: "https://example.com",
        };
        callback(null, user, context);
    }
}
```

**Después**

```js lines
exports.onExecutePostLogin = async (event, api) => {
    api.accessToken.setClaim("https://dev.TLD/wasRedirected", true)
```

<div id="convert-current-sso-clients-references">
  ### Convertir las referencias actuales a clientes SSO
</div>

El objeto `context.sso` de Rules proporciona detalles sobre la sesión actual y los clientes que la utilizan. Para obtener más información, consulta la entrada `context.sso` en [Propiedades del objeto `context` en Rules](/docs/es/customize/rules/context-object). Encontrarás información similar en el objeto `event.session` de Actions.

**Antes**

```javascript lines
function (user, context, callback) {

  const clients = context.sso?.current_clients ?? []; 

  if (clients.length > 0) { 
	context.idToken.clients = clients.join(" "); 
  }

  return callback(null, user, context);
}
```

**Después**

```js lines
exports.onExecutePostLogin = async (event, api) => {
  const clients = event?.session?.clients ?? []; 

  if (clients.length > 0) { 
    api.idToken.setCustomClaim('clients', clients.map(c=> c?.client_id).join(" ")); 
  }
};
```

<div id="complete-the-migration">
  ## Completa la migración
</div>

Una vez que hayas escrito y probado tu nuevo código de Actions, debes activar la Action y desactivar la Rule. Estas dos tareas pueden realizarse rápidamente una después de la otra, pero, según el orden, podría haber un breve período en el que se estén ejecutando ambas o ninguna. Como las Rules activas se ejecutan antes que las Actions desplegadas, si empiezas al final del pipeline de Rules y avanzas hacia atrás, puedes mantener parte de la lógica en Rules mientras desarrollas y pruebas otra lógica en Actions.
