Sigue las directrices a continuación para escribir código de Actions claro, seguro y de alto rendimiento para un entorno de producción optimizado.

<div id="actions-basics">
  ## Conceptos básicos de Actions
</div>

* Usa la menor cantidad posible de solicitudes HTTP y establece un tiempo de espera razonable (menos de 10 segundos) para evitar la acumulación de solicitudes durante el login.
* Usa [metadatos de la aplicación](/docs/es/get-started/applications/configure-application-metadata) para filtrar aplicaciones específicas y determinar si debe ejecutarse una Action.

  * Ejemplo: valores específicos de la aplicación o valores de agrupación de aplicaciones que se usan para variar parcial o totalmente la ejecución de la Action.
* Usa `user_metadata`/`app_metadata` para persistir datos personalizados en el perfil de usuario.

  * Ejemplo: preferencias del usuario o valores de estado.
* Usa [metadatos de transacción](/docs/es/customize/actions/transaction-metadata) para compartir datos entre [triggers de Post Login](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger) durante la misma ejecución.

  * Ejemplo: valores recuperados de servicios externos o calculados que deben reutilizarse en Actions posteriores y que están relacionados con la ejecución actual.
* Usa caché para los datos que deban compartirse entre Actions que pertenezcan a distintas ejecuciones.

  * Ejemplo: valores recuperados de servicios externos o calculados que las Actions deben reutilizar y que no estén relacionados con el usuario ni con la ejecución actual específica.
* Asegúrate de que las Actions que proporcionan verificación o activan <Tooltip tip="Autenticación multifactor (MFA): Proceso de autenticación de usuario que utiliza un factor adicional al nombre de usuario y la contraseña, como un código por SMS." cta="Ver glosario" href="/docs/es/glossary?term=MFA">MFA</Tooltip> no puedan omitirse de forma no intencionada o maliciosa.
* Las Actions nunca deben generar intencionadamente un error; si los procesos se detienen debido a un error o a una condición, usa el método `api` adecuado, como `api.access.deny()`.
* Usa `event.request.hostname` para el dominio que se usa en las llamadas a Authentication API; este puede ser el dominio predeterminado del tenant de Auth0 o un [dominio personalizado](/docs/es/customize/custom-domains).

<div id="actions-modules">
  ## Módulos de Actions
</div>

* Usa Módulos de Actions cuando necesites reutilizar funciones en distintas Actions y evitar la redundancia de código.
* Reduce el tiempo de creación de objetos siempre que sea posible implementando el [patrón Singleton](https://en.wikipedia.org/wiki/Singleton_pattern) para reutilizar instancias de objetos en el módulo de Action.
* Sigue las directrices aplicables que se comparten en este documento sobre programación, seguridad, registro, dependencias y datos de usuario.
* Al usar un módulo de Action desde una Action, ten siempre en cuenta la versión del módulo que se está utilizando.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Cuando publicas una nueva versión de un módulo de Action, Auth0 no actualiza automáticamente la versión a la que hacen referencia las Actions, lo que evita posibles fallos cuando la actualización no es compatible con la Action.
</Callout>

<div id="coding-basics">
  ## Conceptos básicos de programación
</div>

* Usa la comparación estricta `===` con cualquier dato recibido o almacenado.
* Usa una instrucción `return` cuando el proceso de la Action deba detenerse.
* Ejecuta un linter de código, como [ESLint](https://eslint.org), y un analizador, como [Semgrep](https://semgrep.dev/docs/getting-started/), para mejorar la calidad del código y detectar problemas automáticamente.

<div id="security-basics">
  ## Conceptos básicos de seguridad
</div>

* No escriba secretos ni artefactos de código sensibles en texto sin formato como parte de su código de Actions. En su lugar, use [Secrets Manager](/docs/es/customize/actions/write-your-first-action#add-a-secret) o integre su propio gestor en el código de Actions.
* No transmita información de identificación personal (PII) sin cifrar de forma visible, por ejemplo en URL o mensajes de error.
* Use siempre URL HTTPS para redirecciones y llamadas a la API.
* Incluya las direcciones IP en la lista de permitidos cuando sea posible.
* Esté atento a los datos entrantes que puedan manipularse (parámetros de URL, agente de usuario, etc.).

<div id="defensive-coding">
  ## Programación defensiva
</div>

* [Captura los errores](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Guide/Control_flow_and_error_handling#exception_handling_statements) y gestiónalos según sea necesario.
* Usa cláusulas de guarda y finaliza antes si el procesamiento de la Action no debe continuar.

<div id="logging">
  ## Registro
</div>

* Nunca registres datos sensibles, secrets ni PII.
* No superes el máximo de 256 caracteres registrados por Action.

<div id="dependencies">
  ## Dependencias
</div>

* Usa paquetes fiables y con mantenimiento activo.
* Comprueba si hay CVE sin resolver con la [funcionalidad de auditoría de `npm`](https://docs.npmjs.com/cli/v6/commands/npm-audit) o con un verificador automático de dependencias vinculado a un repositorio.
* Usa la versión más reciente de un paquete siempre que sea posible.

<div id="user-data">
  ## Datos de usuario
</div>

* Verifica si un correo electrónico está verificado con `event.user.email_verified` si se usa en un contexto sensible o de alta seguridad.
* Las distintas conexiones proporcionan diferentes datos del perfil de usuario; el único [campo del perfil de usuario](/docs/es/manage-users/user-accounts/user-profiles/user-profile-structure) garantizado es `user_id`.

<div id="redirect-actions-in-the-login-flow">
  ## Redirect Actions en el flujo de Login
</div>

* El token que devuelve `api.redirect.encodeToken` está firmado, pero no cifrado, por lo que no debe incluirse información sensible ni PII en el payload.
* El flujo de Login se ejecuta después de que el inicio de sesión se completa correctamente, lo que incluye:

  * SSO (no se muestra ningún formulario de inicio de sesión)
  * autenticación silenciosa (comprobación de una sesión con `prompt=none` en la URL de autorización)
  * intercambio de token de actualización (sin interacción del usuario)
  * concesiones de contraseña RO (credenciales recopiladas de una aplicación e intercambiadas con el endpoint de token)
* Las Actions que redirigen deben tener en cuenta los casos anteriores y, si se requiere interacción, denegar el acceso o permitir intencionalmente la omisión, lo que traslada la responsabilidad a la aplicación que solicita el inicio de sesión.
