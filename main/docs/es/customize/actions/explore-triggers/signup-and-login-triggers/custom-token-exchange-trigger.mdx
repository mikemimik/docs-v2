  Custom Token Exchange (CTE) está disponible actualmente en acceso anticipado para todos los clientes de Auth0 B2C Professional, B2B Professional y Enterprise. Al usar esta funcionalidad, aceptas los términos aplicables de la prueba gratuita del [Master Subscription Agreement de Okta](https://www.okta.com/legal/). Para obtener más información sobre el ciclo de lanzamiento de productos de Auth0, consulta [Product Release Stages](/docs/es/troubleshoot/product-lifecycle/product-release-stages). Para obtener más información sobre los tipos de suscripción, revisa la página de [precios](https://auth0.com/pricing) de Auth0.
</Warning>

Después de que el servidor de autorización de Auth0 valida que la solicitud de Custom Token Exchange es válida y se asigna a un [Custom Token Exchange Profile](/docs/es/authenticate/custom-token-exchange/configure-custom-token-exchange#create-custom-token-exchange-profile) existente, el trigger `custom-token-exchange` ejecuta la única Action asociada a ese perfil. Si la Action se completa correctamente y establece un usuario para la transacción, se ejecutan las Actions de `post-login` y el resto del pipeline de Auth0 para ese usuario que ha iniciado sesión.

<Frame>![](/docs/images/custom-token-exchange/cte_trigger_flow.png)</Frame>

Las Actions de este flujo son bloqueantes (sincrónicas), lo que significa que se ejecutan como parte del proceso de un trigger y evitan que el resto del pipeline de Auth0 se ejecute hasta que la Action se complete.

<div id="custom-token-exchange-vs-post-login-actions">
  ## Custom Token Exchange vs Actions de Post Login
</div>

El trigger de Actions de Post Login se ejecuta después de que se ejecute la Action de Custom Token Exchange para el usuario que configures para la transacción. Para otros casos de uso, como agregar claims personalizados a los tokens de acceso, usa el [trigger de Action de Post Login](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger).

Custom Token Exchange usa el tipo de concesión de intercambio de tokens, en el que `event.transaction.protocol` se establece en `oauth2-token-exchange` en el trigger de Action de Post Login. Tanto Custom Token Exchange como el inicio de sesión social nativo usan el tipo de concesión de intercambio de tokens. Para distinguir entre ambos, usa el valor de `subject_token_type`, que debe corresponder a uno de tus perfiles de Custom Token Exchange.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Custom Token Exchange en acceso anticipado no admite `api.authentication.challengeWith()` ni `api.authentication.enrollWith()`. Si usas esos métodos con tus Actions de Post Login, la transacción fallará con un error no recuperable. Asegúrate de omitir estos dos métodos cuando `event.transaction.protocol==oauth2-token-exchange`, según el valor de `subject_token_type`.
</Callout>

<div id="actor-support-for-delegation">
  ## Compatibilidad de actor para delegación
</div>

El trigger `custom-token-exchange` admite escenarios de autorización delegada. La Action puede llamar a [`api.authentication.setActor()`](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/custom-token-exchange-trigger/custom-token-exchange-api-object#api-authentication-setactor-actor) para incluir un claim `act` en los tokens emitidos, que identifica a la entidad que actúa en nombre del usuario.

Se puede llamar a `setActor()` independientemente de si la request incluye un `actor_token`. El parámetro `actor_token` es una forma de que la aplicación que realiza la llamada pase información del actor a la Action, pero la identidad del actor también puede estar integrada en el propio `subject_token` u obtenerse por otros medios personalizados dentro del código de la Action.

Cuando la request sí incluye `actor_token` y `actor_token_type`, estos valores están disponibles en [`event.transaction`](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/custom-token-exchange-trigger/custom-token-exchange-event-object#event-transaction). Si `actor_token_type` es `urn:ietf:params:oauth:token-type:id_token` y el token es un token de ID de Auth0 válido del mismo tenant, el servidor valida automáticamente el token y completa `event.transaction.actor_token_user` con el perfil de usuario del actor.

El claim `act` establecido mediante `setActor()` también está disponible en Actions de Post Login a través de [`event.transaction.actor`](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger/post-login-event-object#event-transaction).

<div id="triggers">
  ## Triggers
</div>

<div id="custom-token-exchange">
  ### Custom Token Exchange
</div>

El trigger `custom-token-exchange` es el primer paso del pipeline de Auth0. Esta función se ejecuta después de que la solicitud de Custom Token Exchange recibida se haya validado y asociado correctamente a un perfil existente.

<div id="references">
  ### Referencias
</div>

* [Objeto de evento](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/custom-token-exchange-trigger/custom-token-exchange-event-object): Proporciona información contextual sobre una solicitud de intercambio de tokens, incluido el `subject_token`, la dirección IP, el cliente y más.
* [Objeto de API](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/custom-token-exchange-trigger/custom-token-exchange-api-object): Proporciona métodos para usar con tu Action de intercambio de tokens que te permiten establecer el usuario, definir el actor para la delegación, rechazar la transacción, configurar los metadatos del usuario y acceder a la caché de Actions, entre otras cosas.
