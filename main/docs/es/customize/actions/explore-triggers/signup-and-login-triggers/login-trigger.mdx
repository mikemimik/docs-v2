El trigger de Login se ejecuta cuando un usuario se autentica correctamente en una aplicación de un tenant de Auth0. Esto también incluye la autenticación posterior al registro.

<Frame>![Diagrama que muestra el flujo de Login de Actions.](/docs/images/cdy7uua7fh8z/2SkfIOm4fFOJ8N0GNJwam8/77230cdb4411b2383751b037fb56fe29/2024-09-30_10-01-42.png)</Frame>

Las Actions de este trigger son bloqueantes (sincrónicas), lo que significa que se ejecutan como parte del proceso del trigger e impiden que el resto del pipeline de Auth0 se ejecute hasta que la Action se complete.

<div id="triggers">
  ## Triggers
</div>

<div id="login-post-login">
  ### Login / Post Login
</div>

El trigger `post-login` es una función que se ejecuta después de que un usuario inicia sesión y cuando se intercambia un <Tooltip tip="Token de actualización: token que se usa para obtener un nuevo token de acceso sin obligar a los usuarios a iniciar sesión de nuevo." cta="Ver glosario" href="/docs/es/glossary?term=Refresh+Token">token de actualización</Tooltip>.

* [Objeto de API](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger/post-login-api-object): Proporciona métodos para cambiar el comportamiento del flujo.

<div id="common-use-cases">
  ## Casos de uso comunes
</div>

<div id="access-control">
  ### Control de acceso
</div>

Se puede usar una Action de Post Login para implementar lógica personalizada que permita denegar el acceso a un usuario que intenta acceder a una aplicación:

```javascript lines
/**
 * @param {Event} event - Detalles sobre el usuario y el contexto en el que está iniciando sesión.
 * @param {PostLoginAPI} api - Interfaz cuyos métodos pueden usarse para cambiar el comportamiento del inicio de sesión.
 */
exports.onExecutePostLogin = async (event, api) => {
  if (event.user.email && event.user.email.endsWith("@example.com") && event.client.name === "My SPA") {
    api.access.deny(`Access to ${event.client.name} is not allowed.`);
  }
};
```

<div id="allow-access-only-on-weekdays-for-a-specific-application">
  ### Permitir el acceso solo entre semana para una aplicación específica
</div>

Si tienes una aplicación y quieres asegurarte de que solo se pueda acceder a ella entre semana, puedes crear la siguiente Action:

```javascript lines
/**
 * @param {Event} event - Detalles sobre el usuario y el contexto en el que está iniciando sesión.
 * @param {PostLoginAPI} api - Interfaz cuyos métodos pueden usarse para cambiar el comportamiento del inicio de sesión.
 */
 exports.onExecutePostLogin = async (event, api) => {
  if (event.client.name === "APP_NAME") {
    const d = new Date().getDay();

    if (d === 0 || d === 6) {
      api.access.deny("This app is only available during the week.");
    }
  }
}
```

<div id="deny-access-to-anyone-calling-an-api">
  ### Denegar el acceso a quienes llamen a una API
</div>

Supongamos que quieres denegar el acceso a todos los usuarios que llaman a una API. Esto significa que debes denegar el acceso en función del valor de la <Tooltip tip="Audiencia: identificador único de la audiencia de un token emitido. Se llama aud en un token; su valor contiene el ID de una aplicación (ID de cliente) para un token de ID o de una API (identificador de API) para un token de acceso." cta="Ver glosario" href="/docs/es/glossary?term=audience">audiencia</Tooltip> de tu API, que puedes encontrar en el campo **API Audience** de tu API en [Dashboard &gt; Applications &gt; APIs](https://manage.auth0.com/#/apis). Para ello, crearías la siguiente Action:

```javascript lines
/**
 * @param {Event} event - Detalles sobre el usuario y el contexto en el que está iniciando sesión.
 * @param {PostLoginAPI} api - Interfaz cuyos métodos pueden usarse para cambiar el comportamiento del inicio de sesión.
 */
 exports.onExecutePostLogin = async (event, api) => {
  // En Actions, una API se denomina Resource Server.
  if (event.resource_server && event.resource_server.identifier === "http://todoapi2.api") {
    api.access.deny("end_users_not_allowed");
  }
}
```

<div id="add-user-roles-to-id-and-access-tokens">
  ### Agrega roles de usuario a los tokens de ID y de acceso
</div>

Para agregar roles de usuario a los tokens emitidos por Auth0, usa el objeto `event.authorization` junto con los métodos `api.idToken.setCustomClaim` y `api.accessToken.setCustomClaim`:

```javascript lines
/**
 * @param {Event} event - Detalles sobre el usuario y el contexto en el que está iniciando sesión.
 * @param {PostLoginAPI} api - Interfaz cuyos métodos pueden usarse para cambiar el comportamiento del inicio de sesión.
 */
exports.onExecutePostLogin = async (event, api) => {
  const namespace = 'https://my-app.example.com';
  if (event.authorization) {
    api.idToken.setCustomClaim(`${namespace}/roles`, event.authorization.roles);
    api.accessToken.setCustomClaim(`${namespace}/roles`, event.authorization.roles);
  }
}
```

* Un claim personalizado no puede incluir ciertos términos, y recomendamos encarecidamente usar un namespaced claim con forma de URI. Consulta [nuestra documentación sobre custom claims](/docs/es/secure/tokens/json-web-tokens/create-custom-claims) para obtener más información.
* El <Tooltip tip="JSON Web Token (JWT): formato estándar de token de ID (y, a menudo, de token de acceso) utilizado para representar claims de forma segura entre dos partes." cta="Ver glosario" href="/docs/es/glossary?term=JWT">JWT</Tooltip> que se devuelve a la aplicación solicitante se crea y firma al final del procesamiento del trigger. No se puede acceder al JWT final firmado desde una Action.

<div id="enrich-the-user-profile">
  ### Enriquece el perfil de usuario
</div>

Auth0 proporciona un sistema para almacenar metadatos en un [perfil de usuario](/docs/es/manage-users/user-accounts/user-profiles/normalized-user-profiles). Para establecer `user_metadata` o `app_metadata` en el perfil de un usuario durante el inicio de sesión, usa las funciones `api.user.setUserMetadata` o `api.user.setAppMetadata`.

```javascript lines
/**
 * @param {Event} event - Detalles sobre el usuario y el contexto en el que está iniciando sesión.
 * @param {PostLoginAPI} api - Interfaz cuyos métodos pueden usarse para cambiar el comportamiento del inicio de sesión.
 */
exports.onExecutePostLogin = async (event, api) => {
  api.user.setUserMetadata("favorite_color", "blue");
};
```

Una vez ejecutadas todas las Actions de Post Login, Actions actualizará el perfil del usuario en una sola operación. Esta operación está sujeta al [límite de frecuencia](/docs/es/troubleshoot/customer-support/operational-policies/rate-limit-policy/management-api-endpoint-rate-limits) de &quot;Write Users&quot;.

<div id="enforce-custom-mfa-policy">
  ### Aplicar una política personalizada de MFA
</div>

Se puede usar una Action de Post Login para exigir dinámicamente <Tooltip tip="Autenticación multifactor (MFA): Proceso de autenticación de usuario que usa un factor adicional al nombre de usuario y la contraseña, como un código por SMS." cta="Ver glosario" href="/docs/es/glossary?term=MFA">MFA</Tooltip> a un usuario según las necesidades de la aplicación.

```javascript lines
/**
 * @param {Event} event - Detalles sobre el usuario y el contexto en el que está iniciando sesión.
 * @param {PostLoginAPI} api - Interfaz cuyos métodos pueden usarse para cambiar el comportamiento del inicio de sesión.
 */
exports.onExecutePostLogin = async (event, api) => {
  // Requerir MFA para cualquier usuario que inicie sesión desde América del Norte.
  if (event.request.geoip.continentCode === "NA") {
    api.multifactor.enable("any");
  };
};
```

Se debe configurar un Provider de MFA para habilitar MFA durante el inicio de sesión. Para obtener más información, consulta [Autenticación multifactor](/docs/es/secure/multi-factor-authentication).

<div id="reduce-friction-with-passkeys">
  ### Reduce la fricción con claves de acceso
</div>

Se puede usar una Action de Post Login para reducir la fricción al omitir dinámicamente la MFA en el caso de un usuario que se haya autenticado con una clave de acceso.

```javascript lines
/**
* Controlador que se llamará durante la ejecución de un flujo Post Login.
*
* @param {Event} event - Detalles sobre el usuario y el contexto en el que está iniciando sesión.
* @param {PostLoginAPI} api - Interfaz cuyos métodos pueden usarse para cambiar el comportamiento del inicio de sesión.
*/
exports.onExecutePostLogin = async (event, api) => {
 // Verificar si se usó una clave de acceso para autenticarse
 const skipMFA = event.authentication?.methods.some(
   (method) => method.name === "passkey"
 );

 // Si se usó una clave de acceso, omitir MFA
 if (skipMFA) {
   api.multifactor.enable("none");
 }
};
```

La conexión debe tener habilitadas las claves de acceso y MFA. Para obtener más información, consulte [Claves de acceso](/docs/es/authenticate/database-connections/passkeys) y [Autenticación multifactor](/docs/es/secure/multi-factor-authentication).

<div id="redirect-the-user-to-an-external-site">
  ### Redirigir al usuario a un sitio externo
</div>

Al igual que con [Redirect Rules](/docs/es/customize/rules/redirect-users), se puede usar una Action de Post Login para enviar al usuario a un sitio externo. Una vez completado, se puede redirigir al usuario de vuelta a Auth0 para que continúe su flujo de inicio de sesión. En el ejemplo siguiente, se usa una Action de Redirect para pedirle al usuario que indique su color favorito.

```javascript lines expandable
/**
 * @param {Event} event - Detalles sobre el usuario y el contexto en el que está iniciando sesión.
 * @param {PostLoginAPI} api - Interfaz cuyos métodos pueden usarse para cambiar el comportamiento del inicio de sesión.
 */
exports.onExecutePostLogin = async (event, api) => {
  // Omitir la redirección si el usuario ya eligió un color favorito.
  if (event.user.user_metadata.favorite_color) {
    return;
  }

  const token = api.redirect.encodeToken({
    secret: event.secrets.MY_SHARED_SECRET,
    payload: {
      email: event.user.email,
    },
  });

  // Enviar al usuario a https://my-app.example.com junto
  // con un parámetro de query string `session_token`.
  api.redirect.sendUserTo("https://my-app.example.com", { 
    query: { session_token: token }
  });
};

/**
 * @param {Event} event - Detalles sobre el usuario y el contexto en el que está iniciando sesión.
 * @param {PostLoginAPI} api - Interfaz cuyos métodos pueden usarse para cambiar el comportamiento del inicio de sesión.
 */
exports.onContinuePostLogin = async (event, api) => {
  // Una vez que se haya llamado al endpoint /continue, desempaquetar el token firmado
  // y almacenar el color favorito como metadatos del usuario.

  const payload = api.redirect.validateToken({
    secret: event.secrets.MY_SHARED_SECRET,
  });

  api.user.setUserMetadata("favorite_color", payload.favorite_color);
};
```

El pipeline de Actions se suspenderá mientras el usuario es redirigido. Una vez que el usuario continúe con su proceso de inicio de sesión de Auth0, el pipeline de Actions se reanudará en el punto en que se suspendió. Las Actions que se ejecutaron antes de la redirección no volverán a ejecutarse.

Para obtener más información sobre Redirect Actions, lee [Redirect with Actions](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger/redirect-with-actions).

<div id="modify-scopes-on-an-access-token">
  ### Modificar scopes de un access token
</div>

Al modificar los scopes asociados a un <Tooltip tip="Access Token: Credencial de autorización, en forma de una cadena opaque o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=access+token">access token</Tooltip>, asegúrese de seguir las prácticas recomendadas sobre la especificación de audience.

* Verifique siempre la audience esperada antes de agregar scopes.
* Evite usar entradas no confiables al agregar scopes.

```javascript lines
/**
 * @param {Event} event - Detalles sobre el usuario y el contexto en el que está iniciando sesión.
 * @param {PostLoginAPI} api - Interfaz cuyos métodos pueden usarse para cambiar el comportamiento del inicio de sesión.
 */
exports.onExecutePostLogin = async (event, api) => {
  if (event?.resource_server?.identifier === 'https://example.com/api') {
    api.accessToken.addScope("read:xyz");
  }
};
```

<div id="deny-access-to-specific-ja3ja4-fingerprints">
  ### Denegar el acceso a huellas digitales JA3/JA4 específicas
</div>

El objeto `event.security_context` contiene los valores de huella digital JA3/JA4 de la transacción actual.

```js lines
exports.onExecutePreUserRegistration = async (event, api) => {
  const clientJa4 = event?.security_context?.ja4;
  console.log('[ACTION]', {clientJa4});
  const badFingerprints = ['t13d1517h2_8daaf6152771_b6f405a00624','t13d1516h2_8daaf6152771_d8a2da3f94cd'];
  if (clientJa4 && badFingerprints.includes(clientJa4)){
    api.access.deny('suspicious_tls_fingerprint', 'Your TLS fingerprint has been flagged as suspicious');
  }
};
```
