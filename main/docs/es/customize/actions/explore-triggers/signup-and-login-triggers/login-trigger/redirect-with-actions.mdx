title: Redirigir con Actions
---

Puedes usar Actions de Post Login para redirigir a los usuarios antes de que se complete una transacción de autenticación. Esto te permite implementar flujos de autenticación personalizados que requieren interacción adicional del usuario más allá del formulario de inicio de sesión estándar.

Las redirecciones se usan habitualmente para realizar [autenticación multifactor (MFA)](/docs/es/secure/multi-factor-authentication) personalizada en Auth0, pero también pueden usarse para:

* Permitir experiencias personalizadas para aceptar la política de privacidad, los Términos de servicio y formularios de divulgación de datos.
* Recopilar de forma segura, y una sola vez, datos de perfil adicionales necesarios.
* Permitir que los usuarios remotos de Active Directory cambien su contraseña.
* Exigir que los usuarios completen una verificación adicional al iniciar sesión desde ubicaciones desconocidas.
* Recopilar más información sobre tus usuarios de la que proporcionaron al registrarse inicialmente.

<div id="overview">
  ## Información general
</div>

A grandes rasgos, una Action de redirección funciona de la siguiente manera:

1. Una Action emite una redirección a una URL.
2. La pipeline de Actions se suspende después de que esa Action complete su ejecución.
3. Se redirige al usuario a la URL junto con un parámetro `state`.
4. Cuando el flujo externo concluye, el sitio externo redirige al usuario a un endpoint `/continue` junto con el parámetro `state`.
5. La pipeline de Actions se reanuda en la misma Action que invocó la redirección.

<div id="start-a-redirect">
  ## Inicia una redirección
</div>

Llama a la función `api.redirect.sendUserTo()` de la siguiente manera:

```javascript lines
/**
* @param {Event} event - Detalles sobre el usuario y el contexto en el que está iniciando sesión.
* @param {PostLoginAPI} api - Interfaz cuyos métodos pueden usarse para cambiar el comportamiento del inicio de sesión.
*/
exports.onExecutePostLogin = async (event, api) => {
  api.redirect.sendUserTo("https://my-app.exampleco.com");
};
```

Actions finalizará la ejecución de esta Action y luego suspenderá el pipeline de Actions para enviar al usuario a `https://my-app.exampleco.com`. En otras palabras, las Actions vinculadas a los triggers de Post Login que se ejecuten después de la Action que invoca la redirección no se ejecutarán hasta que se haya reanudado el flujo de autenticación. Si está familiarizado con Redirect Rules, tenga en cuenta que esta es una diferencia clave entre Redirect Actions y Redirect Rules.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  A diferencia de Redirect Rules, Redirect Actions suspenderá el pipeline de Actions cuando se emita una redirección y se reanudará en la misma Action que emitió la redirección cuando se reanude el flujo de autenticación.
</Callout>

Después de que la Action termine de ejecutarse, Auth0 redirige al usuario a la URL especificada en la función `api.redirect.sendUserTo()`. Auth0 también incluye un parámetro `state` en esa URL. Por ejemplo:

`https://my-app.exampleco.com/?state=abc123`

La URL de redirección deberá extraer el parámetro `state` y enviarlo de vuelta a Auth0 para reanudar la transacción de autenticación. `state` es un valor opaco que se usa para prevenir ataques de [falsificación de solicitud entre sitio (CSRF)](/docs/es/secure/security-guidance/prevent-threats).

<div id="resume-the-authentication-flow">
  ## Reanudar el flujo de autenticación
</div>

Después de la redirección, reanude la autenticación redirigiendo al usuario al endpoint `/continue` e incluyendo el parámetro `state` que recibió en la URL. Si no devuelve el valor original de `state` al endpoint `/continue`, Auth0 perderá el contexto de la transacción de inicio de sesión y el usuario no podrá iniciar sesión debido a un error `invalid_request`.

Por ejemplo:

`https://{yourAuth0Domain}/continue?state=THE_ORIGINAL_STATE`

En este ejemplo, `THE_ORIGINAL_STATE` es el valor que Auth0 generó y envió a la URL de redirección. Por ejemplo, si su Action redirige a `https://my-app.exampleco.com/`, Auth0 usaría una URL de redirección similar a `https://my-app.exampleco.com/?state=abc123`, por lo que `abc123` sería `THE_ORIGINAL_STATE`. Para reanudar la transacción de autenticación, redirigiría a:

`https://{yourAuth0Domain}/continue?state=abc123`

Cuando se haya redirigido a un usuario al endpoint `/continue`, el pipeline de Actions se reanudará en la misma Action que invocó la redirección mediante una llamada a la función `onContinuePostLogin`. Para que las redirecciones funcionen correctamente, debe tener una función con la siguiente firma en la misma Action que invocó la redirección:

```javascript lines
/**
* @param {Event} event - Detalles sobre el usuario y el contexto en el que está iniciando sesión.
* @param {PostLoginAPI} api - Interfaz cuyos métodos pueden usarse para cambiar el comportamiento del inicio de sesión.
*/
exports.onExecutePostLogin = async (event, api) => {
  api.redirect.sendUserTo("https://my-app.exampleco.com");
};

/**
* @param {Event} event - Detalles sobre el usuario y el contexto en el que está iniciando sesión.
* @param {PostLoginAPI} api - Interfaz cuyos métodos pueden usarse para cambiar el comportamiento del inicio de sesión.
*/

exports.onContinuePostLogin = async (event, api) => {
}
```

<div id="pass-data-to-the-external-site">
  ## Pasar datos al sitio externo
</div>

Para pasar datos al sitio externo, recomendamos codificar esos datos en un [JWT](/docs/es/secure/tokens/json-web-tokens) firmado para que su aplicación tenga la certeza de que no se hayan manipulado durante el envío. Con Actions, esto puede hacerse con las funciones `api.redirect.encodeToken` y `api.redirect.sendUserTo`:

```javascript lines expandable
/**
* @param {Event} event - Detalles sobre el usuario y el contexto en el que está iniciando sesión.
* @param {PostLoginAPI} api - Interfaz cuyos métodos pueden usarse para cambiar el comportamiento del inicio de sesión.
*/
exports.onExecutePostLogin = async (event, api) => {
  const yourDomain = event.secrets.YOUR_AUTH0_DOMAIN || event.request.hostname

  // Crear un token de sesión firmado
  const token = api.redirect.encodeToken({
    secret: event.secrets.MY_REDIRECT_SECRET,
    expiresInSeconds: 60, 
    payload: {
      // Claims personalizados que se agregarán al token
      email: event.user.email,
      externalUserId: 1234,
      continue_uri: `https://${yourDomain}/continue`
    },
  });

  // Enviar al usuario a https://my-app.exampleco.com junto
  // con un parámetro de query string `session_token` que incluye
  // el correo electrónico.
  api.redirect.sendUserTo("https://my-app.exampleco.com", {
    query: { session_token: token }
  });
}
```

El código anterior agrega un parámetro de cadena de consulta `session_token` a la URL usada en la redirección (además del parámetro `state` que Auth0 agrega automáticamente). Este token contendrá lo siguiente:

| Elemento del token | Descripción                                                                                                                                                                                                      |
| ------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `sub`              | `user_id` de Auth0 del usuario.                                                                                                                                                                                  |
| `iss`              | Nombre de host del dominio de tu tenant de Auth0 (por ejemplo, `example.auth0.com`).                                                                                                                             |
| `exp`              | Hora de expiración (en segundos) especificada con el parámetro `expiresInSeconds`. Debe ser lo más breve posible para evitar que el token se reutilice. El valor predeterminado es de 900 segundos (15 minutos). |
| `ip`               | Dirección IP de la solicitud de autenticación original.                                                                                                                                                          |
| `email`            | Claim personalizado con el valor especificado en el parámetro `payload.email`.                                                                                                                                   |
| `externalUserId`   | Claim personalizado con el valor especificado en el parámetro `payload.externalUserId`.                                                                                                                          |
| `signature`        | El token se firmará con el algoritmo HS256 usando el secreto especificado anteriormente.                                                                                                                         |

<div id="ensure-the-token-has-not-been-tampered-with">
  ### Verifica que no se haya manipulado el token
</div>

El sistema externo debe verificar que este token no se haya manipulado durante la transmisión. Para ello, el sistema remoto debe asegurarse de que la firma del token sea válida y, si corresponde, de que la sesión en el sistema externo pertenezca al mismo usuario de Auth0 indicado en el claim `sub` del token.

<div id="pass-data-back-to-auth0">
  ## Enviar datos de vuelta a Auth0
</div>

Después de que el usuario complete el flujo personalizado en el sitio externo, debe ser redirigido al endpoint `/continue`. En algunas situaciones, puede que quieras enviar datos de vuelta a Auth0 para influir en la autenticación o en el <Tooltip tip="Flujo de autorización: Concesión de autorización (o workflow) especificada en el framework OAuth 2.0." cta="Ver glosario" href="/docs/es/glossary?term=authorization+flow">flujo de autorización</Tooltip> de ese usuario (por ejemplo, si estás implementando comprobaciones de CAPTCHA o <Tooltip tip="Autenticación multifactor (MFA): Proceso de autenticación de usuario que utiliza un factor además del nombre de usuario y la contraseña, como un código por SMS." cta="Ver glosario" href="/docs/es/glossary?term=MFA">MFA</Tooltip> personalizada).

<div id="use-app-metadata-where-possible">
  ### Usa metadatos de aplicación siempre que sea posible
</div>

Si es posible, el sistema remoto debe usar la [Auth0 Management API](https://auth0.com/docs/api/management/v2/) para almacenar información personalizada como metadatos de aplicación en el perfil de usuario de Auth0. Cuando se reanude el flujo de Auth0 Action, esta información estará disponible en el objeto `event.user.app_metadata`. Este enfoque evita pasar información confidencial a Auth0 a través del front channel.

<div id="be-selective-when-storing-data-on-the-auth0-user-profile">
  ### Sé selectivo al almacenar datos en el perfil de usuario de Auth0
</div>

Evita almacenar demasiados datos en el perfil de usuario de Auth0. Estos datos están pensados para usarse con fines de autenticación y autorización. Las capacidades de metadatos y búsqueda de Auth0 no están diseñadas para casos de uso que requieran una alta frecuencia de búsquedas o actualizaciones, como la investigación de mercados. Es probable que tu sistema tenga problemas de escalabilidad y rendimiento si usas Auth0 con este tipo de fines.

Si tu aplicación requiere acceso a una cantidad considerable de datos de usuario, lo recomendable es almacenarlos en un sistema externo y guardar una clave foránea (el ID de usuario) en Auth0 para que los sistemas de back-end puedan recuperar esos datos si es necesario.

<div id="pass-information-back-to-the-action">
  ### Enviar información de vuelta a la Action
</div>

Se debe usar un token de sesión firmado para enviar información confidencial de vuelta a Auth0. Este token se puede validar fácilmente dentro de una Action con el siguiente código:

```javascript lines
/**
 * @param {Event} event - Detalles sobre el usuario y el contexto en el que está iniciando sesión.
 * @param {PostLoginAPI} api - Interfaz cuyos métodos pueden usarse para cambiar el comportamiento del inicio de sesión.
 */
exports.onContinuePostLogin = async (event, api) => {
  const payload = api.redirect.validateToken({
    secret: event.secrets.PRECONFIGURED_SECRET,
    tokenParameterName: 'my_token',
  });

  // usar los datos codificados en el token, como por ejemplo: 
  api.idToken.setCustomClaim('color', payload.favorite_color);
}
```

El token se validará para garantizar que:

* La firma sea válida.
* El token no haya expirado.
* El claim `state` dentro del token coincida con el parámetro `state` usado como parte de la redirección.

| Token Element | Description                                                                                                                           |
| ------------- | ------------------------------------------------------------------------------------------------------------------------------------- |
| `sub`         | `user_id` de Auth0 del usuario.                                                                                                       |
| `iss`         | Aplicación a la que va dirigida la redirección.                                                                                       |
| `exp`         | Debe ser lo más corto posible para evitar la reutilización del token.                                                                 |
| `state`       | Parámetro `state` enviado al sitio remoto como parte de la redirección. Debe incluirse en el token para evitar ataques de repetición. |
| `other`       | Cualquier otro claim personalizado se expondrá como el `payload` en el código anterior.                                                      |
| `signature`   | El token debe estar firmado con el algoritmo HS256.                                                                                   |

Para evitar ataques de repetición, el token debe enviarse de vuelta a Auth0 mediante una solicitud POST al endpoint `/continue`. La opción `tokenParameterName` en el código te permite especificar el nombre del campo que contiene tu token.

<div id="custom-authentication-methods">
  ## Métodos de autenticación personalizados
</div>

Después de una redirección exitosa en el pipeline de login, Actions puede registrar eventos de métodos de autenticación personalizados en la sesión del usuario. El arreglo `event.authentication.methods` contendrá una entrada para el método personalizado durante toda la sesión del navegador del usuario. Cada entrada de este arreglo tiene una marca de tiempo que indica cuándo se registró el método de autenticación.

Una Action personalizada puede activar una redirección si el método personalizado requerido no está en el arreglo `event.authentication.methods` o si la entrada es demasiado antigua.

Puede usar `api.redirect.sendUserTo()` para enviar al usuario a una página que implementa un método de autenticación personalizado. Puede usar `api.authentication.recordMethod()` en el controlador `exports.onContinuePostLogin` para almacenar un registro del método completado en la sesión del usuario.

El registro almacenado en el arreglo `event.authentication.methods` tendrá una propiedad `name` que coincidirá con la URL elegida en `api.authentication.recordMethod()`. La URL capturada aquí le permite buscar entre los métodos de autenticación completados de la transacción actual para determinar si su método personalizado ya se completó.

Su flujo de trabajo puede requerir que el método personalizado vuelva a realizarse periódicamente durante la vigencia de la sesión del usuario. Por ejemplo, los escenarios de MFA personalizados pueden requerir una nueva verificación del usuario después de un período determinado.

En el siguiente ejemplo, se compara la marca de tiempo de un registro existente para determinar cuándo volver a ejecutar el método personalizado:

```javascript lines expandable
const CUSTOM_METHOD_URL = "https://path.to.prompt";
const PROMPT_TTL = 1000 * 60 * 60 * 24; // 24h

/**
 * Handler that will be called during the execution of a PostLogin flow.
 *
 * @param {Event} event - Details about the user and the context in which
 * they are logging in.
 * @param {PostLoginAPI} api - Interface whose methods can be used to
 * change the behavior of the login.
 */
exports.onExecutePostLogin = async (event, api) => {
  // Search authentication method records for an entry representing our
  // custom method.
  const methodRecord = event.authentication?.methods.find((record) =>
    validateCustomRecord(record, CUSTOM_METHOD_URL, PROMPT_TTL)
  );

  if (!methodRecord) {
    const sessionToken = api.redirect.encodeToken({
      payload: {
        user_id: event.user.user_id,
      },
      secret: event.secrets.SESSION_TOKEN_SECRET,
    });

    // We didn't find a valid record, so we send the user to the
    // URL that implements the custom method with the signed
    // data we encoded in `sessionToken`.
    api.redirect.sendUserTo(CUSTOM_METHOD_URL, {
      query: { session_token: sessionToken },
    });
  }
};

/**
 * Controlador que se invocará cuando esta Action se reanude tras una
 * redirección externa. Si su función onExecutePostLogin no realiza
 * una redirección, esta función puede ignorarse sin problema.
 *
 * @param {Event} event - Detalles sobre el usuario y el contexto en el que
 * está iniciando sesión.
 * @param {PostLoginAPI} api - Interfaz cuyos métodos pueden usarse para
 * cambiar el comportamiento del inicio de sesión.
 */
exports.onContinuePostLogin = async (event, api) => {
  const payload = api.redirect.validateToken({
    secret: event.secrets.SESSION_TOKEN_SECRET,
    tokenParameterName: "session_token",
  });

  if (!validateSessionToken(payload)) {
    return api.access.deny("Unauthorized");
  }

  // Record the completion of our custom authentication method.
  // THIS NEW API IS ONLY AVAILABLE IN `onContinuePostLogin`.
  api.authentication.recordMethod(CUSTOM_METHOD_URL);
};

function validateCustomRecord(record, url, ttl) {
  if (!record) {
    // No record means it isn't valid.
    return false;
  }

  if (record.url !== url) {
    // This isn't a record of our custom method.
    return false;
  }

  // Timestamps are rendered as ISO8601 strings.
  const timestamp = new Date(record.timestamp);

  // The record is valid if it was recorded recently enough.
  return timestamp.valueOf() >= Date.now() - ttl;
}

function validateSessionToken(payload) {
  // Custom validation logic for the data returned by the
  // custom method goes here.
  return true;
}
```

La API `api.authentication.recordMethod()` solo está disponible en el controlador `exports.onContinuePostLogin`. Esto evita posibles ataques durante el inicio de sesión al registrar el método personalizado después de completar la redirección.

<div id="restrictions-and-limitations">
  ## Restricciones y limitaciones
</div>

Redirect Actions no funcionan con:

* [endpoint del propietario del recurso](https://auth0.com/docs/api/authentication/reference#resource-owner)
* [intercambio de contraseña](/docs/es/get-started/authentication-and-authorization-flow/resource-owner-password-flow)
* [intercambio de token de actualización](/docs/es/secure/tokens/refresh-tokens)

<div id="resource-owner-endpoint">
  ### endpoint del propietario del recurso
</div>

No es posible usar Redirect Actions cuando llamas al endpoint de la Authentication API [Get Token](https://auth0.com/docs/api/authentication#resource-owner-password) para el flujo Password de <Tooltip tip="Resource Owner: Entidad (como un usuario o una aplicación) capaz de otorgar acceso a un recurso protegido." cta="Ver glosario" href="/docs/es/glossary?term=Resource+Owner">Resource Owner</Tooltip>. Como el usuario no forma parte de un flujo de redirección desde el inicio, no puedes redirigirlo en una Action.

<div id="flows-where-promptnone">
  ### Flujos donde prompt=none
</div>

Dado que el objetivo de `prompt=none` es evitar cualquier situación en la que el usuario tenga que introducir información, cualquier redirección dará lugar a un `error=interaction_required`.

Como Actions se ejecuta después de que se crea una sesión de autenticación, no se puede usar `prompt=none` si tiene una redirect rule que intenta bloquear el acceso a tokens en determinadas condiciones (por ejemplo, MFA personalizada, captcha en el login, etc.).

No se puede crear un flujo de redirección que bloquee el acceso a tokens y omita la Action de redirección con `prompt=none` porque, después de un intento fallido, un usuario puede simplemente volver a llamar con `prompt=none` y obtener tokens, ya que su sesión de autenticación ya se habrá creado aunque Actions haya fallado la primera vez.

<div id="refresh-tokens">
  ### Tokens de actualización
</div>

Dado que usar un <Tooltip tip="Token de actualización: token usado para obtener un token de acceso renovado sin obligar a los usuarios a volver a iniciar sesión." cta="Ver glosario" href="/docs/es/glossary?term=refresh+token">token de actualización</Tooltip> requiere una llamada back-channel al endpoint [Get Token](https://auth0.com/docs/api/authentication#refresh-token) de la Authentication API, esto también fallará si se intenta redirigir.

Es difícil verificar de forma segura que se hayan aplicado restricciones al inicio de sesión. No hay un ID de sesión coherente en el contexto que pueda usarse para recopilar información asociada con la sesión, como si este usuario superó desafíos MFA. Por lo tanto, no puedes usar `prompt=none` en absoluto.

Cada vez que se llama a `api.redirect.sendUserTo()` en una Action, si se pasó `prompt=none`, la autorización falla con `error=interaction_required`; pero, como la sesión del usuario se crea incluso si las Actions fallan, no podemos confiar en que un usuario haya superado desafíos de redirección y, por lo tanto, no podemos usar `prompt=none` como forma de obtener tokens.

En este caso concreto, te recomendamos usar exclusivamente tokens de actualización, porque puedes asegurarte de que un usuario haya superado los desafíos si estos son necesarios para generar un token de actualización.
