Según la etapa de tu implementación en la que te encuentres, tienes varias opciones para probar y depurar Auth0 Actions.

Te recomendamos probar las Actions individuales desde el <Tooltip tip="Auth0 Dashboard: el producto principal de Auth0 para configurar tus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip> a medida que las implementas. Cuando todo esté listo, debes verificar que el flujo de extremo a extremo funcione como se espera usando los registros del tenant a través del Auth0 Dashboard.

<div id="test-individual-actions">
  ## Probar Actions individuales
</div>

Puede probar Actions individuales con el Actions Code Editor. La función de prueba del editor simula una llamada a la Action con un payload de ejemplo basado en el flujo al que está asociada la Action. Para probar una Action individual:

1. Vaya a [Auth0 Dashboard &gt; Actions](https://manage.auth0.com/#/actions) y elija el Flow y la Action que desea editar.
2. Busque el Actions Code Editor y seleccione **Test** (el ícono de reproducción) en su barra lateral. Edite el payload en la barra lateral de Test para analizar el resultado de distintos payloads.
3. Seleccione **Run**.

Los resultados de la prueba muestran los pasos que siguió la Action, la salida de la consola, los errores que se hayan producido y estadísticas útiles sobre la ejecución de la Action.

<Frame>![Captura de pantalla del módulo de prueba dentro del Actions Code Editor](/docs/images/cdy7uua7fh8z/7I4CRM8SmRFiibjuk55L86/794042f525cddd55a8005cc646b3c9c2/test-action-code-editor.png)</Frame>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  También puede probar una Action mediante el endpoint [**Test an Action**](https://auth0.com/docs/api/management/v2#!/Actions/post_test_action) de la Management API, proporcionando un payload correspondiente al flujo con el que está asociada la Action. Para obtener un payload de ejemplo, pruebe la Action con la función Test del Actions Code Editor.
</Callout>

<div id="debug-deployed-actions">
  ## Depura las Actions desplegadas
</div>

Ahora que ya probaste tu Action, es momento de desplegarla y observar su ejecución. Suponiendo que ya desplegaste tu Action y la agregaste a un flujo (para saber cómo adjuntar Actions a Flows, consulta la sección &quot;Attach the Action to a flow&quot; en [Write Your First Action](/docs/es/customize/actions/write-your-first-action)), puedes depurar tu Action activa en unos pocos pasos.

<div id="verify-end-to-end-login-flow">
  ### Verificar el flujo de Login de extremo a extremo
</div>

Para una Action `post-login`, puedes verificar el flujo de Login de extremo a extremo ejecutando un intento de login en tu tenant:

1. Ve a [Auth0 Dashboard &gt; Authentication &gt; Authentication Profile](https://manage.auth0.com/#/authentication-profiles) y selecciona **Try**. Se abrirá una ventana con un ejemplo de login.
2. Completa el flujo de login. El flujo de Login ejecutará todas las Actions configuradas.

Una vez completado, se te redirigirá a una página que mostrará los atributos del perfil de usuario que recibirán tus aplicaciones o un mensaje de error que explica qué salió mal.

<div id="analyze-tenant-logs">
  ### Analizar los registros del tenant

En la lista de registros del tenant, puedes ver los registros asociados con Actions y elegir consultarlos. En el ejemplo de abajo, el tipo de registro `Success Login` está asociado con una Action en el trigger `post-login` de este tenant.

<div id="unit-test-actions">
  ## Pruebas unitarias de Actions
</div>

​Puedes hacer pruebas unitarias de tus Auth0 Actions simulando el event y la api que se pasan a tu función de Action. Tus Actions pueden estar en un sistema de control de versiones o en un directorio local, y puedes usar una herramienta como Jest para probarlas automáticamente. En el siguiente ejemplo, usamos el trigger [Login / Post Login](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger) y [Jest](https://jestjs.io/) para simular y probar.

**Código de la Action:**

```javascript lines
// action.js

exports.onExecutePostLogin = async (event, api) => {
  const namespace = "https://my-app.example.com";
  if (event.authorization) {
    api.idToken.setCustomClaim(`${namespace}/favorite_color`, "pink");
  }
};
```

**Código del archivo de prueba:**

```javascript lines
// action.spec.js

const { onExecutePostLogin } = require("./action.js");
test("sets favorite_color ID token claim to pink", async () => {
  const namespace = "https://my-app.example.com";
  const mockFunction = jest.fn();
  const mockApi = { idToken: { setCustomClaim: mockFunction } };
  const mockEvent = {authorization: {}};

  await onExecutePostLogin(mockEvent, mockApi);
  expect(mockFunction).toBeCalledWith(
    `${namespace}/favorite_color`,
    "pink"
  );
});
```

<div id="best-practices">
  ## Prácticas recomendadas
</div>

Al crear Actions en Auth0, [te recomendamos crear un tenant de Auth0 específico para cada entorno](/docs/es/get-started/auth0-overview/create-tenants/set-up-multiple-environments). Esto te permite aislar tu base de usuarios de producción de otros entornos, así como configurar distintos administradores para cada entorno (por ejemplo, es posible que los ingenieros no tengan acceso a la configuración de producción).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Si quieres administrar, probar y desplegar por completo tus Actions mediante una CLI o un pipeline de CI/CD, puedes usar la [herramienta Deploy CLI](/docs/es/deploy-monitor/deploy-cli-tool).
</Callout>
