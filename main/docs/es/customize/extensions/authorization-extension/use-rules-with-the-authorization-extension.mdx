<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Auth0 ofrece dos formas de implementar el control de acceso basado en roles (RBAC), que puedes usar en lugar del sistema interno de control de acceso de tu API o en combinación con él:

  * [Authorization Core](/docs/es/manage-users/access-control/rbac)
  * [Authorization Extension](/docs/es/customize/extensions/authorization-extension)

  El conjunto de funcionalidades de Authorization Core equivale a la funcionalidad de Authorization Extension, mejora el rendimiento y la escalabilidad, y ofrece un sistema RBAC más flexible que Authorization Extension.

  Actualmente, ambas implementan las funcionalidades clave de RBAC y te permiten restringir los scopes personalizados definidos para una API a aquellos que se hayan asignado al usuario como permisos.
</Callout>

Puedes usar reglas de Auth0 con Authorization Extension para hacer cosas como las siguientes:

* Agregar claims personalizadas al token emitido.
* Determinar la pertenencia del usuario a grupos, así como sus roles y permisos.
* Almacenar la información de grupos, roles y permisos del usuario como parte de `app_metadata`.
* Agregar los grupos, roles y permisos del usuario al token de salida (que se puede solicitar mediante el scope `openid groups permissions roles`).

Como la lógica anterior forma parte de una regla, solo se ejecutará en el contexto de un inicio de sesión. Si se agrega a los usuarios a un grupo o se los quita de él, este cambio solo se reflejará en Auth0 después del siguiente inicio de sesión del usuario.

Para obtener más información, consulta [reglas de Auth0](/docs/es/customize/rules).

<div id="add-custom-claims-to-the-issued-token">
  ## Añade claims personalizadas al token emitido
</div>

Para agregar claims personalizadas a tus tokens, puedes hacerlo creando una rule adicional que permita a Authorization Extension hacerlo. Las claims personalizadas pueden tener espacio de nombres o no tenerlo.

Para obtener más información, lee [Crear claims personalizadas.](/docs/es/secure/tokens/json-web-tokens/create-custom-claims)

Debes limitar la cantidad de claims que agregas al token.

```javascript lines
function (user, context, callback) {
  var namespace = 'http://yourdomain/claims/'; // Puedes definir tu propio namespace, pero no uses un Auth0 domain

  // Agrega los tokens con namespace. Elimina los que no sean necesarios para tu escenario
  context.idToken[namespace + "permissions"] = user.permissions;
  context.idToken[namespace + "groups"] = user.groups;
  context.idToken[namespace + "roles"] = user.roles;
  
  callback(null, user, context);
}
```

Esta rule debe ejecutarse **después** de la Authorization Extension rule. Para asegurarte de que esto suceda, colócala debajo de la Authorization Extension rule.

Al llamar al endpoint `/authorize` o al configurar Lock, tendrás que especificar la información que quieres incluir en el `scope` indicando `groups`, `permissions` y/o `roles`.

<div id="control-app-access">
  ## Controlar el acceso a la aplicación
</div>

También puede escribir reglas que se ejecuten después de la Authorization Extension rule para, por ejemplo, controlar el acceso a su aplicación. Una forma de hacerlo es especificar los roles necesarios para cada aplicación mediante los metadatos de la aplicación.

Para más detalles, consulte [Administrar metadato con Rules](/docs/es/manage-users/user-accounts/metadata/manage-metadata-rules).

<div id="set-app-metadata-required-roles">
  ### Establecer los roles requeridos en app metadata
</div>

Puedes establecer el metadata de la aplicación con roles, que son grupos de permisos que se agrupan para crear un conjunto específico de funcionalidades. Puedes pensar en este paso como “etiquetar” la aplicación para que las reglas que configures sepan sobre qué aplicación deben actuar.

1. ⁠⁠⁠⁠Para establecer el campo `context.clientMetadata` con `required_roles`, selecciona la aplicación con la que quieres trabajar en [Auth0 Dashboard &gt; Applications &gt; Applications](https://manage.auth0.com/#/applications).
   Esto te llevará a **Settings** de la aplicación. Desplázate hacia abajo y selecciona **Show Advanced Settings** al final de la página.
2. En **Application Metadata**, agrega un elemento estableciendo **Key** en `required_roles` y, en el campo **Value**, enumera tus roles separados por comas. Selecciona **+ Add** para agregar el campo.
3. Cuando termines, selecciona **Save Changes**. Ahora, cuando inicies sesión desde esta aplicación, en `context.clientMetadata` tendrás `required_roles` con la cadena de roles que ingresaste.

<div id="create-rule-enforcing-app-roles">
  ### Crear una Rule que aplique los roles de la aplicación
</div>

Ahora que cada aplicación tiene un rol asociado, puede crear la Rule para que se ejecute con esta información de la aplicación en el contexto.

1. Antes de crear esta Rule, habilite **Roles** en **Token Contents** y publique la Authorization Extension rule.
2. Agregue esta Rule y asegúrese de que aparezca después de la Rule generada &quot;auth0-authorization-extension&quot;.
3. Después de configurar `required_roles`, cree una nueva [rule](https://manage.auth0.com/#/rules) con el siguiente cuerpo:

   ```javascript lines
   function (user, context, callback) {
     context.clientMetadata = context.clientMetadata || {};
     if (context.clientMetadata.required_roles && context.clientMetadata.required_roles.length){
       if (user.roles) {
         var _ = require('lodash');
         var roles = context.clientMetadata.required_roles.split(',');
         var matchingRoles =_.filter(user.roles, function(roleName) {
           return _.includes(roles, roleName);
         });

         if (matchingRoles && matchingRoles.length) {
           return callback(null, user, context);
         }
       }

       return callback(new UnauthorizedError('You do not have the required role to access ' + context.clientName));
     }

    callback(null, user, context);
   }
   ```
