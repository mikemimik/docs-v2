<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Auth0 ofrece dos formas de implementar el control de acceso basado en roles (RBAC), que puedes usar en lugar del sistema interno de control de acceso de tu API o en combinación con él:

  * [Authorization Core](/docs/es/manage-users/access-control/rbac)
  * [Authorization Extension](/docs/es/customize/extensions/authorization-extension)

  El conjunto de funcionalidades de Authorization Core ofrece la misma funcionalidad que la Authorization Extension, mejora el rendimiento y la escalabilidad, y proporciona un sistema RBAC más flexible que la Authorization Extension.

  Actualmente, ambas implementan las funcionalidades clave de RBAC y te permiten restringir los scopes personalizados definidos para una API a aquellos que se hayan asignado al usuario como permisos.
</Callout>

Puedes usar Auth0 reglas con la Authorization Extension para hacer cosas como:

* Agregar claims personalizadas al token emitido.
* Determinar la pertenencia del usuario a grupos, sus roles y sus permisos.
* Almacenar la información de grupos, roles y permisos del usuario como parte de `app_metadata`.
* Agregar los grupos, roles y permisos del usuario al token de salida (que puede solicitarse mediante el scope `openid groups permissions roles`).

Debido a que la lógica anterior forma parte de una rule, solo se ejecutará en el contexto de un inicio de sesión. Si se agrega o elimina a un usuario de un grupo, este cambio solo se reflejará en Auth0 después del siguiente inicio de sesión del usuario.

Para obtener más información, consulta [Auth0 reglas](/docs/es/customize/rules).

<div id="add-custom-claims-to-the-issued-token">
  ## Agrega claims personalizados al token emitido
</div>

Para agregar claims personalizados a tus tokens, puedes hacerlo creando una rule adicional que permita a Authorization Extension hacerlo. Los claims personalizados pueden tener espacio de nombres o no tenerlo.

Para obtener más información, lee [Crear claims personalizados.](/docs/es/secure/tokens/json-web-tokens/create-custom-claims)

Debes limitar la cantidad de claims que agregas al token.

```javascript lines
function (user, context, callback) {
  var namespace = 'http://yourdomain/claims/'; // Puedes definir tu propio namespace, pero no uses un Auth0 domain

  // Agrega los tokens con namespace. Elimina los que no sean necesarios para tu escenario
  context.idToken[namespace + "permissions"] = user.permissions;
  context.idToken[namespace + "groups"] = user.groups;
  context.idToken[namespace + "roles"] = user.roles;
  
  callback(null, user, context);
}
```

Esta rule debe ejecutarse **después** de la rule de Authorization Extension. Para asegurarte de que esto suceda, colócala debajo de la rule de Authorization Extension.

Al llamar al endpoint `/authorize` o al configurar Lock, deberás especificar en el `scope` la información que quieres incluir, indicando `groups`, `permissions` y/o `roles`.

<div id="control-app-access">
  ## Controla el acceso a la aplicación
</div>

También puedes escribir reglas que se ejecuten después de la regla de Authorization Extension para hacer cosas como controlar el acceso a tu aplicación. Una forma de hacerlo es especificar los roles necesarios para cada aplicación mediante los metadatos de la aplicación.

Para más detalles, consulta [Administrar metadatos con reglas](/docs/es/manage-users/user-accounts/metadata/manage-metadata-rules).

<div id="set-app-metadata-required-roles">
  ### Establecer los roles requeridos en los metadatos de la aplicación
</div>

Puedes configurar los metadatos de la aplicación con roles, que son grupos de permisos que se agrupan para crear un conjunto específico de funcionalidades. Puedes pensar en este paso como una forma de &quot;etiquetar&quot; la aplicación para que las reglas que configures sepan sobre qué aplicación actuar.

1. ⁠⁠⁠⁠Para establecer el campo `context.clientMetadata` con `required_roles`, selecciona la aplicación con la que quieres trabajar en [Auth0 Dashboard &gt; Applications &gt; Applications](https://manage.auth0.com/#/applications).
   Esto te llevará a **Settings** de la aplicación. Desplázate hacia abajo y selecciona **Show Advanced Settings** al final de la página.
2. En **Application Metadata**, agrega un elemento estableciendo **Key** en `required_roles` y, en el campo **Value**, indica tus roles separados por comas. Selecciona **+ Add** para añadir el campo.
3. Cuando termines, selecciona **Save Changes**. Ahora, cuando inicies sesión desde esta aplicación, en `context.clientMetadata` tendrás `required_roles` con la cadena de texto de roles que introdujiste.

<div id="create-rule-enforcing-app-roles">
  ### Crear una rule que aplique los roles de la aplicación
</div>

Ahora que cada aplicación tiene un role asociado, puedes crear la rule para que se ejecute con esta información de la aplicación en el context.

1. Antes de crear esta rule, habilita **Roles** en **Token Contents** y publica la rule de Authorization Extension.
2. Agrega esta rule y asegúrate de que aparezca después de la rule generada &quot;auth0-authorization-extension&quot;.
3. Después de configurar `required_roles`, crea una nueva [rule](https://manage.auth0.com/#/rules) con el siguiente cuerpo:

   ```javascript lines
   function (user, context, callback) {
     context.clientMetadata = context.clientMetadata || {};
     if (context.clientMetadata.required_roles && context.clientMetadata.required_roles.length){
       if (user.roles) {
         var _ = require('lodash');
         var roles = context.clientMetadata.required_roles.split(',');
         var matchingRoles =_.filter(user.roles, function(roleName) {
           return _.includes(roles, roleName);
         });

         if (matchingRoles && matchingRoles.length) {
           return callback(null, user, context);
         }
       }

       return callback(new UnauthorizedError('You do not have the required role to access ' + context.clientName));
     }

    callback(null, user, context);
   }
   ```
