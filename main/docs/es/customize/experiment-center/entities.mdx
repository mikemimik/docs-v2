Con la [Management API](/docs/es/api/management/v2) de Auth0, puedes definir los detalles de las entidades de Experiment Center, incluidas las propiedades, los estados del ciclo de vida y las reglas de validación.

<Warning>
  Durante la beta, Experiment Center solo funciona en tenants de desarrollo. No es compatible con tenants de producción.
</Warning>

<div id="experiment">
  ## Experimento
</div>

<div id="experiment-lifecycle">
  ### Ciclo de vida del experimento
</div>

Los experimentos tienen cinco estados:

| **Estado**  | **Significado**                                                                                                                |
| ----------- | ------------------------------------------------------------------------------------------------------------------------------ |
| `draft`     | Creado, pero no está en ejecución. Se puede probar de forma segura usando anulaciones mediante parámetros de consulta.         |
| `active`    | En ejecución. La asignación de variantes y la inyección de contexto están activas en todas las transacciones de autenticación. |
| `paused`    | Suspendido temporalmente. No se realizan nuevas asignaciones. Las sesiones en curso conservan la variante asignada.            |
| `completed` | Finalizado. No se realizan nuevas asignaciones. La configuración se conserva como referencia durante la promoción manual.      |
| `archived`  | Eliminado de forma no permanente. Oculto en las vistas de lista predeterminadas. Se conservan los datos analíticos.            |

Para cambiar de estado en el ciclo de vida, usa el endpoint de la Management API [`/api/v2/experimentation/experiments/{id}/status`](/docs/es/api/management/v2/experimentation/update-experiment-status).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  No se permite la transición de `completed → active`. Si necesitas ejecutar la misma prueba otra vez, crea un experimento nuevo.
</Callout>

<div id="the-is_valid-gate">
  ### La validación `is_valid`
</div>

Antes de que un experimento pueda activarse, debe pasar una verificación de preparación.

Experiment Center almacena un booleano `is_valid` en cada experimento y lo recalcula cada vez que se escribe. Esto te da información inmediata durante la configuración.

También puedes comprobar explícitamente si está listo usando el endpoint de la Management API [`/api/v2/experimentation/experiments/{id}/validate`](/docs/es/api/management/v2/experimentation/validate-experiment).

Esto devuelve `is_valid: true/false` y una matriz `errors` con cada bloqueo. La misma comprobación se ejecuta automáticamente cuando intentas activar mediante el endpoint de estado.

**Reglas de validación que se comprueban durante la activación:**

* La feature flag referenciada está en estado `active`
* Existe al menos una asignación
* Todos los valores `variation_id` de las asignaciones pertenecen a la feature flag del experimento
* Exactamente una asignación tiene `is_control: true`
* Los pesos de las asignaciones suman 100 (estrategia de porcentaje)
* Exactamente una asignación tiene `is_fallback: true` (estrategia de segmento)
* Ningún otro experimento está activo actualmente para el tenant (solo en Beta)

<div id="assignment">
  ### Asignación
</div>

**Anulación mediante parámetros de consulta**

Puedes forzar una asignación concreta en cualquier solicitud a `/authorize` pasando parámetros de consulta:

| Parámetro       | Descripción                                                |
| --------------- | ---------------------------------------------------------- |
| `experiment_id` | Inscribe en un experimento específico                      |
| `variation_id`  | Fuerza una variación específica (requiere `experiment_id`) |
| `segment_id`    | Fuerza un segmento específico (requiere `experiment_id`)   |

Esto funciona con experimentos en cualquier estado, incluido `draft`. Úsalo durante el desarrollo para comprobar que ambas variaciones se muestran correctamente antes de activarlo.

<div id="feature-flag">
  ## Feature flag
</div>

<div id="feature-flag-lifecycle">
  ### Ciclo de vida de la feature flag
</div>

Las feature flags tienen un ciclo de vida almacenado con tres estados:

| **Estado** | **Significado**                                                                                                    |
| ---------- | ------------------------------------------------------------------------------------------------------------------ |
| `draft`    | Creada, pero aún no está activa. No puede ser referenciada por un experimento activo.                              |
| `active`   | Lista para usarse. Es un requisito para que pueda activarse cualquier experimento que haga referencia a esta flag. |
| `archived` | Terminal. Ningún experimento nuevo puede hacer referencia a esta flag. Crea una nueva flag si es necesario.        |

Para pasar de un estado a otro, usa el endpoint de Auth0 Management API [`/api/v2/experimentation/feature-flags/{id}/status`](/docs/es/api/management/v2/experimentation/update-feature-flag-status).

**Condición de activación:** Una feature flag no puede pasar a `active` hasta que tenga al menos dos variaciones. Esto garantiza que toda flag activa tenga, como mínimo, un control y un tratamiento antes de activarse.

```text wrap lines
draft → active   (requires ≥2 variations)
draft → archived
active → archived (terminal; no way back)
```

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  No se puede pasar de `archived → active`. Si archivas una feature flag y necesitas ejecutar otra prueba, crea una nueva feature flag.
</Callout>

<div id="variation">
  ### Variación
</div>

**Parámetros y anulaciones**

Los parámetros de las feature flags definen las opciones de configuración de tu experimento. Cada parámetro tiene:

* Un nombre (por ejemplo, `show_passkey_prompt`)
* Un tipo: `string`, `boolean`, `number`, `array` u `object`
* Un valor predeterminado (el valor base)

La propiedad `overrides` de una variación especifica solo los parámetros que cambian. En tiempo de ejecución, Experiment Center combina el valor base del flag con las `overrides` de la variación y entrega el objeto `config` completo a ACUL, Actions y las plantillas de página. Cada parámetro siempre tiene un valor en `config`; no necesitas implementar lógica alternativa.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Los parámetros usan **solo el modo estructurado**. Cada parámetro tiene una clave con nombre y un valor tipado.
</Callout>

<div id="segment">
  ## Segmento
</div>

<div id="segment-rules">
  ### Reglas del segmento
</div>

Las `rules` de un segmento son un arreglo de objetos de regla. Una solicitud coincide con el segmento si **alguna** regla del arreglo coincide. Una regla coincide si sus condiciones cumplen con el `match_type`:

* `match_type: "all"`: todas las condiciones deben ser verdaderas (lógica AND)
* `match_type: "any"`: al menos una condición debe ser verdadera (lógica OR)

Cada condición compara un atributo con un valor mediante un operador: `equals`, `not_equals`, `in`, `not_in`, `exists`, `not_exists`.

<div id="available-condition-attributes">
  ### Atributos de condición disponibles
</div>

Los segmentos solo pueden usar atributos disponibles al iniciar la transacción de `/authorize`:

| **Categoría**                  | **Atributos**                                      |
| ------------------------------ | -------------------------------------------------- |
| Cliente                        | `client_id`                                        |
| Conexión                       | `connection`, `connection_type`                    |
| Organización                   | `organization_id`                                  |
| Dominio                        | `domain`                                           |
| Dispositivo y navegador        | `device_type`, `browser`, `platform`, `user_agent` |
| Geográfico (derivado de la IP) | `country`, `region`                                |
