Con la [Management API](/docs/es/api/management/v2) de Auth0, puedes definir los detalles de la entidad de Experiment Center, incluidas las propiedades, los estados del ciclo de vida y las reglas de validación.

<Warning>
  Durante la Beta, Experiment Center solo está disponible en tenants de desarrollo. Los tenants de producción no son compatibles.
</Warning>

<div id="experiment">
  ## Experimento
</div>

<div id="experiment-lifecycle">
  ### Ciclo de vida del experimento
</div>

Los experimentos tienen cinco estados:

| **Estado**  | **Significado**                                                                                                                  |
| ----------- | -------------------------------------------------------------------------------------------------------------------------------- |
| `draft`     | Creado, pero sin ejecutar. Se puede probar de forma segura mediante anulaciones de parámetros de consulta.                       |
| `active`    | En ejecución. La asignación de variantes y la inyección de contexto están activas para todas las transacciones de autenticación. |
| `paused`    | Suspendido temporalmente. No se realizan nuevas asignaciones. Las sesiones en curso mantienen la variación asignada.             |
| `completed` | Finalizado. No se realizan nuevas asignaciones. La configuración se conserva como referencia durante la promoción manual.        |
| `archived`  | Eliminado de forma lógica. Oculto en las vistas de lista predeterminadas. Se conservan los datos analíticos.                     |

Para cambiar el estado del ciclo de vida, usa el endpoint de la Management API [`/api/v2/experimentation/experiments/{id}/status`](/docs/es/api/management/v2/experimentation/update-experiment-status).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  No se permite pasar de `completed → active`. Si necesitas volver a ejecutar la misma prueba, crea un nuevo experimento.
</Callout>

<div id="the-is_valid-gate">
  ### La compuerta `is_valid`
</div>

Antes de que un experimento pueda activarse, debe pasar una verificación de preparación.

Experiment Center almacena un valor booleano `is_valid` en cada experimento y lo recalcula con cada escritura. Esto te da retroalimentación inmediata durante el Setup.

También puedes determinar explícitamente si está listo mediante el endpoint de la Management API [`/api/v2/experimentation/experiments/{id}/validate`](/docs/es/api/management/v2/experimentation/validate-experiment).

Esto devuelve `is_valid: true/false` y un arreglo `errors` que enumera todos los bloqueadores. La misma verificación se ejecuta automáticamente cuando intentas activar a través del endpoint de estado.

**Reglas de validación comprobadas durante la activación:**

* El flag de funcionalidad referenciado está en estado `active`
* Existe al menos una asignación
* Todos los valores `variation_id` de las asignaciones pertenecen al flag de funcionalidad del experimento
* Exactamente una asignación tiene `is_control: true`
* Los pesos de las asignaciones suman 100 (estrategia de porcentaje)
* Exactamente una asignación tiene `is_fallback: true` (estrategia de segmento)
* Ningún otro experimento está activo actualmente para el tenant (solo en Beta)

<div id="assignment">
  ### Asignación
</div>

**Anulación de parámetros de consulta**

Puedes forzar una asignación específica en cualquier solicitud a `/authorize` pasando parámetros de consulta:

| Parámetro       | Descripción                                                |
| --------------- | ---------------------------------------------------------- |
| `experiment_id` | Inscribir en un experimento específico                     |
| `variation_id`  | Forzar una variación específica (requiere `experiment_id`) |
| `segment_id`    | Forzar un segmento específico (requiere `experiment_id`)   |

Esto funciona para experimentos en cualquier estado, incluido `draft`. Úsalo durante el desarrollo para verificar que ambas variaciones se muestran correctamente antes de activarlo.

<div id="feature-flag">
  ## Flag de funcionalidad
</div>

<div id="feature-flag-lifecycle">
  ### Ciclo de vida del flag de funcionalidad
</div>

Los flags de funcionalidad tienen un ciclo de vida definido con tres estados:

| **Estado** | **Significado**                                                                                                |
| ---------- | -------------------------------------------------------------------------------------------------------------- |
| `draft`    | Creado, pero todavía no está activo. No puede ser referenciado por un experimento activo.                      |
| `active`   | Listo para usarse. Es un requisito para activar cualquier experimento que haga referencia a este flag.         |
| `archived` | Estado final. Ningún experimento nuevo puede hacer referencia a este flag. Crea un nuevo flag si es necesario. |

Para cambiar de un estado a otro, usa el endpoint de Auth0 Management API [`/api/v2/experimentation/feature-flags/{id}/status`](/docs/es/api/management/v2/experimentation/update-feature-flag-status).

**Requisito de activación:** Un flag de funcionalidad no puede pasar a `active` hasta que tenga al menos dos variaciones. Esto garantiza que cada flag activo tenga, como mínimo, un control y un tratamiento antes de entrar en funcionamiento.

```text wrap lines
draft → active   (requires ≥2 variations)
draft → archived
active → archived (terminal; sin vuelta atrás)
```

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  No se permite pasar de `archived → active`. Si archivas un flag de funcionalidad y necesitas realizar otra prueba, crea un nuevo flag de funcionalidad.
</Callout>

<div id="variation">
  ### Variación
</div>

**Parámetros y sobrescrituras**

Los parámetros del flag de funcionalidad definen la configuración de tu experimento. Cada parámetro tiene:

* Un nombre (por ejemplo, `show_passkey_prompt`)
* Un tipo: `string`, `boolean`, `number`, `array` u `object`
* Un valor predeterminado (la línea base)

La propiedad `overrides` de una variación especifica solo los parámetros que difieren. En el runtime, Experiment Center combina la línea base del flag con las sobrescrituras de la variación y entrega el objeto `config` completo resultante a ACUL, Actions y las plantillas de página. Cada parámetro siempre tiene un valor en `config`; nunca necesitas escribir lógica de respaldo.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Los parámetros usan **solo el modo estructurado**. Cada parámetro tiene una clave con nombre y un valor tipado.
</Callout>

<div id="segment">
  ## Segmento
</div>

<div id="segment-rules">
  ### Reglas del segmento
</div>

La propiedad `rules` de un segmento es un arreglo de objetos de regla. Una solicitud coincide con el segmento si **cualquiera** de las reglas del arreglo coincide. Una regla coincide si sus condiciones cumplen el `match_type`:

* `match_type: "all"`: todas las condiciones deben cumplirse (lógica AND)
* `match_type: "any"`: al menos una condición debe cumplirse (lógica OR)

Cada condición compara un atributo con un valor usando un operador: `equals`, `not_equals`, `in`, `not_in`, `exists`, `not_exists`.

<div id="available-condition-attributes">
  ### Atributos de condición disponibles
</div>

Los segmentos solo pueden usar los atributos disponibles al iniciar la transacción de `/authorize`:

| **Categoría**                    | **Atributos**                                      |
| -------------------------------- | -------------------------------------------------- |
| Cliente                          | `client_id`                                        |
| Conexión                         | `connection`, `connection_type`                    |
| Organización                     | `organization_id`                                  |
| Dominio                          | `domain`                                           |
| Dispositivo y navegador          | `device_type`, `browser`, `platform`, `user_agent` |
| Geográficos (derivados de la IP) | `country`, `region`                                |
