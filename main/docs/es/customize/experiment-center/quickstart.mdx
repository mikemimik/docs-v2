<div id="prerequisites">
  ## Prerrequisitos
</div>

Para empezar a usar Experiment Center, necesitas:

* **Un [tenant](/docs/es/get-started/tenant-settings#environment-tag) de desarrollo de Auth0**

* **Una [aplicación máquina a máquina](/docs/es/get-started/apis/create-m2m-app-test#create-machine-to-machine-applications-for-testing)** con los siguientes scopes de la Management API:

  ```text wrap lines
  read:experimentation
  create:experimentation
  update:experimentation
  delete:experimentation
  ```

<div id="1-create-and-activate-the-feature-flag">
  ## 1. Crear y activar la flag de funcionalidad
</div>

Una flag de funcionalidad define lo que se está probando y las posibles variaciones.

Para crear una flag de funcionalidad, haz una llamada `POST` al endpoint [`/api/v2/experimentation/feature-flags`](/docs/es/api/management/v2/experimentation/create-feature-flag).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  * La respuesta incluye un valor `feature_flag_id`; lo necesitarás en los pasos siguientes.
  * La flag de funcionalidad comienza con el estado `draft`.
</Callout>

<div id="add-two-variations">
  ### Añade dos variaciones
</div>

El flag de funcionalidad necesita al menos dos variaciones antes de poder activarse: un control y un tratamiento.

<div id="create-the-control-variation">
  #### Crear la variación de control
</div>

La variación de control no tiene sobrescrituras. Representa la línea base sin cambios.
Para añadir una variación de control, haz una llamada `POST` al endpoint [`/api/v2/experimentation/feature-flags/{feature_flag_id}/variations`](/docs/es/api/management/v2/experimentation/create-variation).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  La respuesta incluye un valor `variation_id` para el control. Lo necesitarás al configurar las asignaciones.
</Callout>

<div id="create-the-treatment-variation">
  #### Crear la variación de tratamiento
</div>

La variación de tratamiento sobrescribe los parámetros que quieras cambiar.

Para añadir la variación de tratamiento, haz una solicitud `POST` al endpoint [`/api/v2/experimentation/feature-flags/{feature_flag_id}/variations`](/docs/es/api/management/v2/experimentation/create-variation) y configura el objeto `overrides`.

```json Example
    "overrides": {
      "show_passkey_prompt": { "value": true },
      "prompt_style": { "value": "modal" }
    }
```

<div id="activate-the-feature-flag">
  ### Activa el flag de funcionalidad
</div>

Ahora que tienes dos variaciones, activa el flag de funcionalidad. Un experimento no puede activarse a menos que el flag de funcionalidad al que hace referencia tenga el estado `active`.

Para cambiar el estado del flag de funcionalidad a `active`, haz una llamada `POST` al endpoint [`/api/v2/experimentation/feature-flags/{feature_flag_id}/status`](/docs/es/api/management/v2/experimentation/update-feature-flag-status).

<div id="2-create-a-segment-optional">
  ## 2. Crea un segmento (opcional)
</div>

Si quieres dirigir tráfico específico, crea un segmento. Omite este paso si quieres una división porcentual simple de todo el tráfico.

Para crear un segmento, haz una llamada `POST` al endpoint [`/api/v2/experimentation/segments`](https://auth0.com/docs/api/management/v2/experimentation/create-segment).

Este ejemplo crea un segmento que incluye a usuarios móviles en Estados Unidos:

```json Example
    "name": "mobile-us-users",
    "description": "Mobile users from the United States",
    "rules": [
      {
        "match_type": "all",
        "conditions": [
          { "attribute": "device_type", "operator": "equals", "value": "mobile" },
          { "attribute": "country", "operator": "equals", "value": "US" }
        ]
      }
    ]
```

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  La respuesta incluye un valor `segment_id`. Los segmentos solo se aplican cuando usas la [estrategia de asignación](/docs/es/api/management/v2/experimentation/create-experiment#body-allocation-strategy) `segment` en el experimento.
</Callout>

<div id="3-create-and-activate-an-experiment">
  ## 3. Crear y activar un experimento
</div>

Crea un experimento que haga referencia a tu flag de funcionalidad y defina cómo se divide el tráfico.

Para crear un experimento, haz una llamada `POST` al endpoint [`/api/v2/experimentation/experiments`](/docs/es/api/management/v2/experimentation/create-experiment#body-allocations).

Este ejemplo usa una división porcentual de 90/10 mediante el objeto `allocations`: al 90 % de los usuarios se les asigna el control y al 10 % el tratamiento. Este suele ser un punto de partida prudente para un despliegue gradual.

```json Example
    "allocations": [
      {
        "variation_id": "var_Id",
        "weight": 90,
        "is_control": true
      },
      {
        "variation_id": "var_Id",
        "weight": 10,
        "is_control": false
      }
    ]

```

El experimento comienza en estado `draft` con `is_valid: false`. Esto es normal; la comprobación completa de preparación se realiza en el siguiente paso.

<div id="validate-the-experiment">
  ### Validar el experimento
</div>

Antes de activarlo, confirma que el experimento esté listo. El endpoint `/validate` devuelve las mismas comprobaciones que se ejecutan durante la activación.

Para verificar el experimento, realiza una llamada `POST` al endpoint [`/api/v2/experimentation/experiments/{experiment_id}/validate`](/docs/es/api/management/v2/experimentation/validate-experiment).

La respuesta incluye el valor `is_valid` para indicar si está listo para activarse.

<div id="activate-the-experiment">
  ### Activa el experimento
</div>

Cuando el experimento sea válido y hayas probado ambas variaciones, actívalo.

Para cambiar el estado del experimento a `active`, haz una solicitud `POST` al endpoint [`/api/v2/experimentation/experiments/{experiment_id}/status`](/docs/es/api/management/v2/experimentation/update-experiment-status).

El valor `started_at` del experimento se establece en la primera activación y no cambia si lo pausas y lo vuelves a activar.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  **Un experimento activo por tenant.** Si ya hay otro experimento activo, la activación devuelve `400 experiment_active_limit_exceeded`. Primero pausa o completa el otro experimento.
</Callout>

<div id="4-trigger-an-auth-event-and-observe-results">
  ## 4. Genera un evento de autenticación y observa los resultados
</div>

Genera un inicio de sesión de prueba en tu tenant de prueba. Por ejemplo, abre un navegador y ve a tu URL de `/authorize` sin ningún parámetro de override.

Completa el flujo de inicio de sesión. Experiment Center resuelve el experimento activo, asigna una variación mediante hashing determinista, inyecta el contexto del experimento y enriquece el evento de autenticación resultante.

<div id="verify-enriched-logs">
  ### Verifica los logs enriquecidos
</div>

Puedes revisar los logs de Auth0 de los eventos del experimento:

* Ve a [**Dashboard &gt; Monitoreo &gt; Logs**](https://manage.auth0.com/dashboard/#/logs) en Auth0.

* Usa la transmisión de logs: si ya transmites los registros del tenant a una herramienta de analítica (Datadog, Splunk, Segment, etc.), los metadatos enriquecidos del experimento se envían automáticamente a través del mismo flujo.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  El evento incluye metadatos del experimento en el objeto `details.experiment`. El campo `variation_id` te indica qué variación se le asignó al usuario.
</Callout>

<Check>
  Has ejecutado correctamente tu primer experimento cuando:

  1. Tus eventos de autenticación en los registros del tenant incluyen `details.experiment` con `experiment_id` y `variation_id`
  2. Ambas variaciones producen el comportamiento correcto cuando se prueban con anulaciones mediante parámetros de consulta
  3. El mismo dispositivo o usuario recibe de forma consistente el mismo `variation_id` en varios eventos de autenticación (asignación determinista)
</Check>
