<ReleaseStageNotice feature="Auth0 Experiment Center" stage="beta" terms="true" contact="equipo de soporte de Auth0" />

Cuando un experimento está activo, Experiment Center inyecta el objeto `ExperimentContext` en los triggers de Action compatibles.

<Warning>
  Durante la beta, Experiment Center solo funciona en tenants de desarrollo. Los tenants de producción no son compatibles.
</Warning>

<div id="supported-triggers">
  ## Triggers admitidos
</div>

El contexto del experimento está disponible para las siguientes [Actions](/docs/es/customize/actions/actions-overview) de Auth0:

* [`post_login`](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger) trigger.
* [`pre_user_registration`](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/pre-user-registration-trigger) trigger.
* [`post_user_registration`](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/post-user-registration-trigger) trigger.

<div id="the-eventexperiment-object">
  ## El objeto event.experiment
</div>

En los triggers compatibles, Experiment Center añade un campo `experiment` al objeto `event`:

```typescript
interface ExperimentContext {
  experiment_id: string;   // El ID del experimento activo
  variation_id: string;    // El ID de la variación asignada
  config: {                // Configuración combinada: línea base + sobrescritura
    [paramName: string]: { value: unknown };
  };
  is_control: boolean;     // True cuando esta es la variación de control
}
```

Cuando no hay ningún experimento activo (o cuando la funcionalidad no está habilitada para el tenant), `event.experiment` es `null` (no `undefined`).

<div id="config-object-contains-the-full-merged-configuration">
  ### el objeto `config` contiene toda la configuración combinada
</div>

El objeto `config` contiene todos los parámetros definidos en el flag de funcionalidad, combinados con las sobrescrituras de la variación asignada. Nunca tendrás que buscar valores de la línea base ni escribir lógica de respaldo. Si el parámetro existe en el flag de funcionalidad, existe en `config`.

<div id="null-safety-pattern">
  ## Patrón de seguridad frente a `null`
</div>

Comprueba si hay un experimento activo antes de leer ninguna propiedad:

```javascript
exports.onExecutePostLogin = async (event, api) => {
  const ec = event.experiment;
  if (!ec) return; // Sin experimento activo; nada por hacer

  // Aquí es seguro acceder a ec.config, ec.variation_id, ec.is_control
};
```

Usa este patrón en los tres triggers compatibles. La salida anticipada mantiene tu Action limpia y garantiza que se comporte correctamente durante los períodos en los que no se está ejecutando ningún experimento.

<div id="example-post_login-conditional-mfa-policy">
  ## Ejemplo: post_login — política condicional de MFA
</div>

Este ejemplo lee un parámetro booleano y aplica una política de MFA distinta según la variación.

```javascript
// Action post_login
exports.onExecutePostLogin = async (event, api) => {
  const ec = event.experiment;
  if (!ec) return;

  const enforceMfa = ec.config?.require_mfa?.value;

  if (enforceMfa === true) {
    // Variación de tratamiento: aplicar MFA a este usuario
    api.multifactor.enable("any", { allowRememberBrowser: false });
  }
  // Variación de control: sin cambios en la aplicación de MFA (comportamiento de línea base)
};
```

`ec.config.require_mfa.value` es `true` para los usuarios de la variación de tratamiento y `false` (la línea base) para los usuarios de la variación de control. No hace falta lógica de respaldo.

<div id="example-post_login-set-a-custom-claim-based-on-variation">
  ## Ejemplo: post_login — establecer un claim personalizado según la variación
</div>

Este ejemplo incorpora la asignación del experimento en el token de identidad del usuario como un claim personalizado. Algunos flujos de analítica leen los claims del token en lugar de los registros del tenant.

```javascript
// Action post_login
exports.onExecutePostLogin = async (event, api) => {
  const ec = event.experiment;
  if (!ec) return;

  // Agregar la asignación del experimento al token de ID
  api.idToken.setCustomClaim("https://example.com/experiment", {
    experiment_id: ec.experiment_id,
    variation_id: ec.variation_id,
  });
};
```

Usa una URL de `claim` con espacio de nombres, según la convención de `claim` personalizado de OIDC. El destinatario (tu aplicación) puede leer después el `claim` del token sin consultar los logs del tenant.

<div id="example-pre_user_registration-variation-based-metadata">
  ## Ejemplo pre_user_registration — metadatos basados en la variación
</div>

Este ejemplo usa un experimento del flujo de registro para establecer `user_metadata` según la variante en la que quede el usuario al registrarse.

```javascript
// Action pre_user_registration
exports.onExecutePreUserRegistration = async (event, api) => {
  const ec = event.experiment;
  if (!ec) return;

  const onboardingVariant = ec.config?.onboarding_variant?.value;

  if (onboardingVariant) {
    api.user.setUserMetadata("onboarding_variant", onboardingVariant);
    api.user.setUserMetadata("onboarding_experiment_id", ec.experiment_id);
  }
};
```

Los metadatos se guardan durante el registro, por lo que acompañan al usuario de una sesión a otra. Más tarde, puedes consultarlos para saber a qué cohorte de registro pertenece un usuario.

<div id="example-post_user_registration-trigger-downstream-enrollment">
  ## Ejemplo: post_user_registration — iniciar la inscripción posterior
</div>

Este ejemplo envía un webhook después del registro según la variación asignada al nuevo usuario.

```javascript
// Action post_user_registration
exports.onExecutePostUserRegistration = async (event, api) => {
  const ec = event.experiment;
  if (!ec) return;

  const enrollInNewProgram = ec.config?.enroll_welcome_program?.value;

  if (enrollInNewProgram === true) {
    await fetch("https://your-backend.example.com/api/enrollment", {
      method: "POST",
      headers: { "Content-Type": "application/json" },
      body: JSON.stringify({
        user_id: event.user.user_id,
        email: event.user.email,
        experiment_id: ec.experiment_id,
        variation_id: ec.variation_id,
      }),
    });
  }
};
```

Solo los usuarios de la variación de tratamiento desencadenan el webhook de inscripción. Los usuarios de control siguen la ruta estándar posterior al registro.

<div id="use-is_control">
  ## Usa is_control
</div>

El parámetro `is_control` es `true` cuando el usuario está en el grupo de control (recibió la línea base, sin sobrescrituras aplicadas). Úsalo cuando necesites identificar qué usuarios vieron la experiencia sin modificar o cuando quieras omitir el procesamiento opcional para los usuarios del grupo de control.

```javascript
exports.onExecutePostLogin = async (event, api) => {
  const ec = event.experiment;
  if (!ec) return;

  if (!ec.is_control) {
    // Solo registra usuarios de tratamiento en tu sistema de analítica
    // (los usuarios de control se registran por separado, fuera del Action)
    await logTreatmentEvent(ec.experiment_id, ec.variation_id, event.user.user_id);
  }
};
```

Para condicionar el comportamiento (no el análisis), verifica directamente los valores del parámetro `config`. Son más explícitos y fáciles de leer.
