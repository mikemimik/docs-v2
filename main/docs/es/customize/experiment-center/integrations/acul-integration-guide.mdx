Cuando un experimento está activo y se asigna una variación, Experiment Center inyecta un objeto `ExperimentContext` en tu componente [ACUL](/docs/es/customize/login-pages/advanced-customizations) como `experiment`.

<Warning>
  Durante la beta, Experiment Center funciona solo en tenants de desarrollo. Los tenants de producción no son compatibles.
</Warning>

La inyección solo se produce en las pantallas que has habilitado explícitamente. Debes habilitar explícitamente cada pantalla agregando `"experiment"` al arreglo `context_configuration` de esa pantalla.

Para habilitar explícitamente una pantalla para que reciba contexto del experimento, haz una llamada `PATCH` al endpoint [`/api/v2/prompts/{prompt}/screen/{screen}/rendering`](/docs/es/api/management/v2/prompts/patch-rendering).

```json Example
    "context_configuration": ["experiment"]
```

Reemplaza `{prompt}` por el nombre del prompt (por ejemplo, `login`) y `{screen}` por el nombre de la pantalla (por ejemplo, `login`).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  La resolución del experimento y el enriquecimiento del registro del tenant siempre se ejecutan, independientemente de si una pantalla se ha habilitado explícitamente o no. El opt-in solo controla si las propiedades de `experiment` se pasan a tu componente ACUL. Esta es una medida de minimización de datos: no habilites explícitamente las pantallas que no necesiten el contexto del experimento.
</Callout>

<div id="the-experiment-context-shape">
  ## La estructura del contexto del experimento
</div>

Cuando una pantalla se ha habilitado explícitamente y hay un experimento activo, puedes acceder al contexto del experimento a través de `window.universal_login_context.experiment`.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Durante Experiment Center (Beta), el SDK de ACUL no añade automáticamente la propiedad `experiment`, por lo que debe definirse mediante `window.universal_login_context.experiment`.
</Callout>

```typescript
const experiment = window.universal_login_context.experiment;
```

El componente ACUL recibe un contexto `experiment`:

```typescript
interface ExperimentContext {
  experiment_id: string;   // El ID del experimento activo
  variation_id: string;    // El ID de la variación asignada
  config: {                // Configuración combinada: línea base + sobrescritura
    [paramName: string]: { value: unknown };
  };
  is_control: boolean;     // True cuando se trata de la variación de control
}
```

Cuando no hay ningún experimento activo (o cuando la funcionalidad no está habilitada para el tenant), `experiment` es `null`.

El parámetro `config` contiene la configuración completa resultante de la combinación para la variación asignada. Experiment Center toma los parámetros de línea base del flag de funcionalidad y les aplica las sobrescrituras de la variación asignada. Todos los parámetros definidos en el flag de funcionalidad siempre tienen un valor en `config`.

Por ejemplo, si tu flag de funcionalidad tiene un parámetro `button_label` con un valor de línea base de `"Sign in"` y la variación asignada lo sobrescribe por `"Continue"`, entonces `config.button_label.value` es `"Continue"`.
Para la variación de control (sin sobrescrituras), `config.button_label.value` es `"Sign in"`.

<div id="read-a-parameter-value">
  ## Leer el valor de un parámetro
</div>

Puedes acceder al valor de un parámetro mediante `config[paramName].value`:

```typescript
const experiment = window.universal_login_context.experiment;
const label = experiment?.config?.button_label?.value;
```

Usa el encadenamiento opcional (`?.`) en todos los casos. La prop `experiment` es `undefined` cuando no hay ningún experimento activo.

<div id="use-is_control">
  ## Uso de is_control
</div>

El parámetro `is_control` es `true` cuando el usuario está en el grupo de control (recibió la línea base, sin sobrescrituras aplicadas). Úsalo cuando necesites identificar qué usuarios vieron la experiencia original o cuando quieras omitir el procesamiento opcional para los usuarios del grupo de control.

```typescript
if (!experiment?.is_control) {
  // Solo se ejecuta para usuarios en treatment
  trackExperimentImpression(experiment.experiment_id, experiment.variation_id);
}
```

Para la renderización por ramas, comprueba directamente el valor del parámetro (`config.my_param.value`) en lugar de `is_control`.
Las comprobaciones basadas en parámetros son más legibles y siguen funcionando correctamente incluso si cambias qué variación es el control estadístico en un experimento futuro.

<div id="example-copy-variant-experiment">
  ## Ejemplo: experimento de variantes de texto
</div>

Este ejemplo muestra un flag de funcionalidad con dos variantes que cambia el texto de un botón. El control usa la etiqueta estándar; el tratamiento usa una alternativa.

**Parámetros del flag de funcionalidad:**

```json
{
  "button_label": {
    "type": "string",
    "value": "Sign in",
    "description": "Label for the primary login button"
  }
}
```

**Variación de control:** sin sobrescrituras (hereda `button_label: "Sign in"`)

**Variación de tratamiento:**

```json
{
  "overrides": {
    "button_label": { "value": "Continue to your account" }
  }
}
```

**Componente ACUL que lee el parámetro:**

```typescript
// LoginScreen.tsx
export default function LoginScreen({...props }) {
  // config siempre tiene un valor — no se necesita fallback
  const experiment = window.universal_login_context.experiment;
  const buttonLabel = experiment?.config?.button_label?.value ?? "Sign in";

  return (
    <div>
      <h1>Welcome back</h1>
      <form onSubmit={props.onSubmit}>
        <input type="email" name="email" placeholder="Email" />
        <input type="password" name="password" placeholder="Password" />
        <button type="submit">{buttonLabel}</button>
      </form>
    </div>
  );
}
```

El fallback `?? "Sign in"` de la última línea cubre el caso en que no hay ningún experimento activo (en ese caso, `experiment` es `undefined` y `config?.button_label?.value` se evalúa como `undefined`). Si lo prefieres, puedes usar una comprobación de null por separado:

```typescript
const buttonLabel = experiment
  ? experiment.config.button_label.value
  : "Sign in"; // Sin experimento activo; usar el valor predeterminado
```

<div id="example-boolean-feature-rollout">
  ## Ejemplo: despliegue de una funcionalidad booleana
</div>

Este ejemplo usa un parámetro booleano para mostrar de forma condicional un nuevo elemento de la interfaz de usuario.

```typescript
// PostLoginScreen.tsx
export default function PostLoginScreen({...props }) {
  const experiment = window.universal_login_context.experiment;
  const showPasskeyBanner = experiment?.config?.show_passkey_banner?.value === true;

  return (
    <div>
      <p>You're signed in.</p>
      {showPasskeyBanner && (
        <div className="banner">
          <p>Set up a passkey for faster sign-in next time.</p>
          <button onClick={props.onEnrollPasskey}>Set up passkey</button>
          <button onClick={props.onDismiss}>Not now</button>
        </div>
      )}
    </div>
  );
}
```

La comprobación `=== true` (en lugar de una comprobación *truthy*) es intencional: garantiza que el banner solo se muestre cuando el parámetro sea explícitamente `true`, no cuando `experiment` sea `undefined` o falte `config`.

<div id="troubleshoot">
  ## Solución de problemas
</div>

La propiedad `experiment` es `undefined` en tres situaciones:

1. No hay ningún experimento activo actualmente en el tenant
2. La pantalla no se ha habilitado explícitamente mediante `context_configuration`
3. Experiment Center no está habilitado en el tenant

Trata siempre `experiment` como si pudiera ser `undefined`. El patrón más seguro:

```typescript
const experiment = window.universal_login_context.experiment;
const myParam = experiment?.config?.my_param?.value;
// myParam es undefined cuando no hay ningún experimento activo
// Usa un valor por defecto: myParam ?? "your-default"
```

No des por hecho que la propiedad `experiment` esté presente. El código que depende de que `experiment` esté definida generará errores cuando no haya ningún experimento en ejecución, que es lo más habitual.
