Hay dos tipos de tokens relacionados con la identidad: <Tooltip tip="token de identidad: credencial destinada al propio cliente, no al acceso a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=ID+tokens">tokens de identidad</Tooltip> y <Tooltip tip="token de identidad: credencial destinada al propio cliente, no al acceso a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=access+tokens">tokens de acceso</Tooltip>.

<div id="id-tokens">
  ## Tokens de identidad
</div>

Los [tokens de identidad](/docs/es/secure/tokens/id-tokens) son [JSON Web Tokens (JWT)](/docs/es/secure/tokens/json-web-tokens) pensados para que los use únicamente la aplicación. Por ejemplo, si hay una aplicación que usa Google para iniciar sesión de los usuarios y sincronizar sus calendarios, Google envía un token de identidad a la aplicación que incluye información sobre el usuario. Luego, la aplicación analiza el [contenido del token](https://openid.net/specs/openid-connect-core-1_0.html#StandardClaims) y usa la información (incluidos detalles como el nombre y la foto de perfil) para personalizar la experiencia del usuario.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Asegúrate de [validar los tokens de identidad](/docs/es/secure/tokens/id-tokens/validate-id-tokens) antes de usar la información que contienen. Puedes usar una [biblioteca](https://jwt.io/#libraries-io) para ayudarte con esta tarea.
</Callout>

**No** uses tokens de identidad para obtener acceso a una API. Cada token contiene información para la <Tooltip tip="Audiencia: Identificador único de la audiencia de un token emitido. Se denomina aud en un token; su valor contiene el ID de una aplicación (ID de cliente) para un token de identidad o de una API (identificador de API) para un token de acceso." cta="Ver glosario" href="/docs/es/glossary?term=audience">audiencia</Tooltip> prevista (que suele ser el destinatario). Según la especificación de <Tooltip tip="Audiencia: Identificador único de la audiencia de un token emitido. Se denomina aud en un token; su valor contiene el ID de una aplicación (ID de cliente) para un token de identidad o de una API (identificador de API) para un token de acceso." cta="Ver glosario" href="/docs/es/glossary?term=OpenID">OpenID</Tooltip> Connect, la audiencia del token de identidad (indicada por el claim **aud**) debe ser el <Tooltip tip="ID de cliente: Valor de identificación asignado por Auth0 a tu recurso registrado." cta="Ver glosario" href="/docs/es/glossary?term=client+ID">ID de cliente</Tooltip> de la aplicación que realiza la solicitud de autenticación. Si no es así, no debes confiar en el token.

El contenido decodificado de un token de identidad es similar al siguiente:

export const codeExample1 = `{
  "iss": "http://{yourDomain}/",
  "sub": "auth0|123456",
  "aud": "{yourClientId}",
  "exp": 1311281970,
  "iat": 1311280970,
  "name": "Jane Doe",
  "given_name": "Jane",
  "family_name": "Doe",
  "gender": "female",
  "birthdate": "0000-10-31",
  "email": "janedoe@example.com",
  "picture": "http://example.com/janedoe/me.jpg"
}`;

<AuthCodeBlock children={codeExample1} language="json" />

Este token autentica al usuario ante la aplicación. La audience (el **claim** **aud**) del token se establece en el identificador de la aplicación, lo que significa que solo esta aplicación en particular debe consumir este token.

Por el contrario, una API espera un token cuyo valor de **aud** sea igual al identificador único de la API. Por lo tanto, a menos que mantengas el control tanto de la aplicación como de la API, por lo general enviar un token de identidad a una API no funcionará. Como el token de identidad no está firmado por la API, la API no tendría forma de saber si la aplicación hubiera modificado el token (por ejemplo, añadiendo más scopes) si aceptara el token de identidad. Consulta el [JWT Handbook](https://auth0.com/resources/ebooks/jwt-handbook) para obtener más información.

<div id="access-tokens">
  ## Tokens de acceso
</div>

Los [tokens de acceso](/docs/es/secure/tokens/access-tokens) (que no siempre son <Tooltip tip="JSON Web Token (JWT): formato estándar del token de ID (y, a menudo, del token de acceso) usado para representar claims de forma segura entre dos partes." cta="Ver glosario" href="/docs/es/glossary?term=JWTs">JWTs</Tooltip>) se usan para informar a una API de que el portador del token ha sido autorizado para acceder a la API y realizar un conjunto predeterminado de acciones (especificadas por los **scopes** concedidos).

En el ejemplo de Google anterior, Google envía un token de acceso a la aplicación después de que el usuario inicia sesión y da su consentimiento para que la aplicación lea o escriba en Google Calendar. Siempre que la aplicación quiere escribir en Google Calendar, envía una solicitud a la Google Calendar API, incluyendo el token de acceso en el **header** HTTP **Authorization**.

Los tokens de acceso **nunca** deben usarse para la [autenticación](/docs/es/authenticate). Los tokens de acceso no pueden indicar si el usuario se ha autenticado. La única información del usuario que contiene el token de acceso es el ID de usuario, ubicado en el claim **sub**. En sus aplicaciones, trate los tokens de acceso como cadenas opacas, ya que están destinados a las API. Su aplicación no debe intentar decodificarlos ni esperar recibir tokens en un formato determinado.

Aquí tiene un ejemplo de un token de acceso:

export const codeExample2 = `{
  "iss": "https://{yourDomain}/",
  "sub": "auth0|123456",
  "aud": [
    "my-api-identifier",
    "https://{yourDomain}/userinfo"
  ],
  "azp": "{yourClientId}",
  "exp": 1489179954,
  "iat": 1489143954,
  "scope": "openid profile email address phone read:appointments"
}`;

<AuthCodeBlock children={codeExample2} language="json" />

Ten en cuenta que el token no contiene ninguna información sobre el usuario aparte de su ID (claim **sub**). Solo contiene información de autorización sobre qué acciones puede realizar la aplicación en la API (claim **scope**). Esto es lo que lo hace útil para proteger una API, pero no para autenticar a un usuario.

En algunas situaciones, puede ser conveniente incluir información adicional sobre el usuario u otros custom claims, además de su claim **sub**, en el token de acceso para evitar que la API tenga que hacer trabajo adicional para obtener detalles sobre el usuario. Si decides hacerlo, ten presente que estos claims adicionales podrán leerse en el token de acceso. Para obtener más información, consulta [Create Custom Claims](/docs/es/secure/tokens/json-web-tokens/create-custom-claims).

<div id="specialized-tokens">
  ## Tokens especializados
</div>

Hay tres tokens especializados que se utilizan en los escenarios de autenticación basada en tokens de Auth0:

* **<Tooltip tip="Token de actualización: Token usado para obtener un token de acceso renovado sin obligar a los usuarios a volver a iniciar sesión." cta="Ver glosario" href="/docs/es/glossary?term=Refresh+tokens">Tokens de actualización</Tooltip>**: Un token que se utiliza para obtener un token de acceso renovado sin tener que volver a autenticar al usuario.
* **<Tooltip tip="Proveedor de identidad (IdP): Servicio que almacena y administra identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=IDP">IDP</Tooltip> tokens de acceso**: Tokens de acceso emitidos por proveedores de identidad después de la autenticación del usuario que puedes usar para llamar a las API de terceros.
* **Auth0 <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip> tokens de acceso**: Tokens de corta duración que contienen claims específicas (scopes) y te permiten llamar a los endpoints de Management API.
