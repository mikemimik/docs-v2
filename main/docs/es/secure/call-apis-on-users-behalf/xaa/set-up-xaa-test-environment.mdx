Esta sección explica cómo configurar el entorno de prueba de extremo a extremo para la aplicación de recursos. Al configurar tu tenant de Auth0 como el servidor de autorización de la aplicación de recursos, tu aplicación SaaS puede empezar a aceptar solicitudes entrantes de ID-JAG sin necesidad de cambiar el código. Esto permite que tu API SaaS genere tokens de acceso en respuesta a estas solicitudes, para que los agentes de IA y otras aplicaciones puedan consumir tu API sin problemas.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Esta guía asume que usas Okta como tu proveedor de identidad empresarial (IdP) y que tienes acceso administrativo a un tenant de Okta que puedes usar para hacer pruebas. Si no tienes uno, consulta [Crea y configura tu tenant de Okta](#create-and-configure-your-okta-tenant).
</Callout>

Para configurar tu entorno de prueba de extremo a extremo para la aplicación de recursos:

* Configura y registra tu aplicación de recursos: esto incluye configurar tu tenant de Auth0 y registrar tu aplicación SaaS como aplicación de recursos en Okta. Para obtener más información, consulta [Configuración de la aplicación de recursos](#resource-app-setup).
* Configura la aplicación solicitante para probar el flujo de extremo a extremo: esto incluye registrar una aplicación solicitante de prueba en tu tenant de Auth0 y actualizar Okta para vincularla con tu aplicación de recursos. Para obtener más información, consulta [Configuración de la aplicación solicitante](#requesting-app-setup).
* Configura cómo tu tenant de Auth0 se federa con el proveedor de identidad empresarial de tu cliente: en nuestro entorno de prueba, el proveedor de identidad empresarial será tu tenant de prueba de Okta, que representará a uno de tus clientes empresariales. Para obtener más información, consulta [Federación con el proveedor de identidad empresarial y configuración de la organización](#federate-with-the-enterprise-idp-and-organization-configuration).
* Administra Cross App Access en Okta: configura conexiones de agente a aplicación y de aplicación a aplicación en la Consola de administración de Okta. Para obtener más información, consulta [Administra Cross App Access en Okta](/docs/es/secure/call-apis-on-users-behalf/xaa/manage-xaa-in-okta).

<div id="create-and-configure-your-okta-tenant">
  ## Crea y configura tu tenant de Okta
</div>

Para preparar tu entorno de prueba de extremo a extremo para la aplicación de recursos, necesitas crear y configurar tu tenant de Okta para Cross App Access.

* En el [sitio web para desarrolladores de Okta](https://developer.okta.com/signup/), regístrate para obtener un plan Okta Integrator Free. Una vez que completes el registro, se te redirigirá a tu nuevo tenant de Okta.
* En la Consola de administración de Okta, ve a **Configuración &gt; Funcionalidades**. En **Funcionalidades de acceso anticipado**, habilita **Cross App Access**.

<Frame>![](/docs/images/xaa/okta_enable_xaa.png)</Frame>

<div id="resource-app-setup">
  ## Configuración de la aplicación de recursos
</div>

Para configurar tu aplicación de recursos, debes:

* [Crear la API en Auth0](#create-the-api-in-auth0)
* [Crear la aplicación de recursos en Auth0](#create-the-resource-app-in-auth0)
* [Registrar la aplicación de recursos en Okta](#register-the-resource-app-in-okta)

<div id="create-the-api-in-auth0">
  ### Crear la API en Auth0
</div>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Si ya has creado una API personalizada en tu tenant de Auth0, puedes omitir esta sección.
</Callout>

En el Auth0 Dashboard, [registra una API personalizada](/docs/es/get-started/auth0-overview/set-up-apis) que represente la API de tu SaaS en tu tenant de Auth0.

<Frame>![](/docs/images/xaa/xaa_register_api.png)</Frame>

Después de crear la API, también puedes configurar opcionalmente su audiencia como **Default Audience** para tu tenant de Auth0 en [Configuración del tenant](/docs/es/get-started/tenant-settings).

También puedes usar [Políticas de acceso a la API para aplicaciones](/docs/es/get-started/apis/api-access-policies-for-applications) para controlar de forma granular qué aplicaciones obtienen acceso a tu API y para qué scopes.

<div id="create-the-resource-app-in-auth0">
  ### Crear la aplicación de recursos en Auth0
</div>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Si su tenant de Auth0 ya tiene una o varias aplicaciones listas para iniciar sesión en su aplicación SaaS, puede omitir esta sección.
</Callout>

En el Auth0 Dashboard, [cree una aplicación](/docs/es/get-started/auth0-overview/create-applications), como una aplicación web tradicional, una SPA o una aplicación nativa, que sirva como interfaz principal para que los usuarios finales accedan a las funciones de su aplicación SaaS.

<div id="register-the-resource-app-in-okta">
  ### Registrar la aplicación de recursos en Okta
</div>

Debes registrar tu aplicación SaaS en Okta Integration Network (OIN) para que se considere una aplicación de recursos válida.

Para registrar tu aplicación SaaS como una aplicación de recursos en Okta, tienes dos opciones:

* Para una configuración de prueba rápida, te recomendamos usar la aplicación Todo0, que ya está registrada en la OIN. En la Consola de administración de Okta, ve a **Applications &gt; Applications &gt; Browse App Catalog** y busca `Todo0`. Selecciónala y agrega la integración.

<Frame>![](/docs/images/xaa/xaa_browse_todo0_in_oin.png)</Frame>

* También puedes solicitar el registro de una aplicación nueva en la OIN desde tu tenant de Okta. Para obtener más información, consulta [Submission process for SSO and SCIM integrations](https://developer.okta.com/docs/guides/submit-app-overview/#submission-process-for-sso-and-scim-integrations). Para agilizar el proceso de registro, ponte en contacto con tu representante de Auth0 u Okta.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  En un entorno de producción, tus clientes empresariales instalarán tu aplicación SaaS desde el catálogo de OIN durante la configuración de su IdP.
</Callout>

Además, debes proporcionar a Okta la URL del emisor de tu tenant de Auth0 asociada a tu aplicación de recursos. Las aplicaciones solicitantes usan la URL del emisor para solicitar la conexión con tu aplicación de recursos. Para obtener más información, consulta [Test the end-to-end XAA flow](/docs/es/secure/call-apis-on-users-behalf/xaa/test-xaa-flow).

<div id="requesting-app-setup">
  ## Configuración de la aplicación solicitante
</div>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  En un entorno de producción, cada aplicación solicitante se configura una sola vez para habilitar su conexión con tu aplicación de recursos.
</Callout>

Para configurar tu aplicación solicitante, debes:

* [Crear la aplicación solicitante en Auth0](#create-the-requesting-app-in-auth0)
* [Registrar la aplicación solicitante en Okta](#register-the-requesting-app-in-okta)

<div id="create-the-requesting-app-in-auth0">
  ### Crear la aplicación solicitante en Auth0
</div>

Para probar el entorno de extremo a extremo, cree y registre una aplicación que actúe como la aplicación solicitante. La aplicación debe ser un cliente confidencial que pueda almacenar secretos de cliente, como una aplicación web.

Para [crear una aplicación](/docs/es/get-started/auth0-overview/create-applications) que represente a la aplicación solicitante en su tenant de Auth0:

* Vaya a **Applications &gt; Applications** y seleccione **Create Application**.
* Introduzca un nombre y seleccione **Regular Web Application**.

<Frame>![](/docs/images/xaa/xaa_create_regular_web_app.png)</Frame>

* Una vez creada la aplicación, desplácese hasta **Settings** y active el interruptor **Cross App Access**.

<Frame>![](/docs/images/xaa/allow_xaa_auth0_app.png)</Frame>

Una vez creada y configurada la aplicación, debe proporcionar a Okta el `client_id` de la aplicación y la URL del emisor de su tenant de Auth0. Esto habilita la conexión entre la aplicación solicitante, identificada por el `client_id`, y la aplicación de recursos, identificada por la URL del emisor. Para obtener más información, consulte [Test the end-to-end XAA flow](/docs/es/secure/call-apis-on-users-behalf/xaa/test-xaa-flow).

<div id="register-the-requesting-app-in-okta">
  ### Registrar la aplicación solicitante en Okta
</div>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  En un entorno de producción, el desarrollador de la aplicación solicitante registra la aplicación solicitante en la Okta Integration Network (OIN). Los clientes empresariales instalarán la aplicación solicitante desde el catálogo de la OIN durante la configuración de su IdP.
</Callout>

Debes registrar la aplicación en la Okta Integration Network (OIN) para que se considere una aplicación solicitante XAA válida al usar Okta como IdP empresarial.

Para registrar la aplicación solicitante en Okta, tienes dos opciones:

* Para una configuración de prueba rápida, te recomendamos usar la aplicación Agent0, que ya está registrada en la OIN. En la Consola de administración de Okta, ve a **Applications &gt; Applications &gt; Browse App Catalog** y busca `Agent0`. Selecciónala y agrega la integración.

<Frame>![](/docs/images/xaa/xaa_select_agent0_in_oin.png)</Frame>

* También puedes solicitar el registro de una aplicación nueva en la OIN. Para obtener más información, consulta el [proceso de envío para integraciones de SSO y SCIM](https://developer.okta.com/docs/guides/submit-app-overview/#submission-process-for-sso-and-scim-integrations). Para agilizar el proceso de registro, ponte en contacto con tu representante de Auth0 o de Okta.

Dado que la aplicación solicitante autentica a empleados empresariales con Okta, debes configurar la [sign-on policy](https://help.okta.com/en-us/content/topics/security/policies/policies-home.htm) de la aplicación en Okta.

1. Ve a **Applications &gt; Applications** y selecciona la aplicación (por ejemplo, Agent0).
2. En **Sign On**, selecciona **Edit** y agrega la URL de devolución de llamada de la aplicación solicitante en el campo **Redirect URI**. Ajusta el valor de **Redirect URI** según la aplicación de prueba que quieras usar. Para obtener más información, consulta [Probar el flujo XAA de extremo a extremo](/docs/es/secure/call-apis-on-users-behalf/xaa/test-xaa-flow).
3. Selecciona **Save**.

<Frame>![](/docs/images/xaa/agent0_sign_on_policy.png)</Frame>

Por último, permite que tu usuario de prueba inicie sesión en la aplicación solicitante en Okta.

En la Consola de administración de Okta:

1. Ve a **Applications** y selecciona la aplicación (por ejemplo, Agent0).
2. Selecciona **Assign &gt; Assign to People** y luego a tu usuario de prueba.
3. Selecciona **Save**.

<div id="federate-with-the-enterprise-idp-and-organization-configuration">
  ## Federación con el IdP empresarial y la configuración de la organización
</div>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  En un entorno de producción, configuras cada uno de tus clientes empresariales una sola vez para federarlo con tu tenant de Auth0. Auth0 incorporará compatibilidad con [SSO de autoservicio](/docs/es/authenticate/enterprise-connections/self-service-enterprise-configuration) en versiones posteriores, lo que te permitirá delegar la configuración de XAA a tus clientes empresariales como parte de la configuración de SSO.
</Callout>

Debes federar tu tenant de Auth0, que actúa como servidor de autorización para tu aplicación de recursos, con el tenant de Okta de tu cliente empresarial. Esta federación establece confianza criptográfica y permite que tu aplicación valide y acepte aserciones firmadas (ID-JAG) emitidas por el IdP del cliente.

Para probar el flujo XAA de extremo a extremo para varios clientes empresariales conectados a tu aplicación de recursos, puedes repetir los pasos de esta sección para varias conexiones empresariales de Okta Workforce en tu tenant de Auth0. Cada conexión corresponde a un tenant de prueba de Okta distinto, y cada tenant representa a un cliente empresarial diferente.

<div id="configure-an-okta-workforce-enterprise-connection">
  ### Configurar una conexión empresarial de Okta Workforce
</div>

Usa el `client_id` y el `client_secret` de tu aplicación de recursos para [crear una conexión empresarial de Okta Workforce](/docs/es/authenticate/identity-providers/enterprise-identity-providers/okta) en tu tenant de Auth0.

Al crear la conexión empresarial de Okta Workforce, activa el rol **Cross App Access - Resource Application**. Esto permite que tu aplicación de recursos acepte ID-JAG emitidos por el IdP empresarial asociado a esa conexión; en este caso, tu tenant de Okta.

<Frame>![](/docs/images/xaa/xaa_new_okta_workforce_connection.png)</Frame>

Después de crear la conexión empresarial de Okta Workforce, copia la URL de devolución de llamada que Auth0 proporciona en la configuración de la conexión. Necesitarás la URL de devolución de llamada para configurar las políticas de inicio de sesión de la aplicación de recursos en tu tenant de Okta.

En la Consola de administración de Okta:

1. Ve a **Applications &gt; Applications** y selecciona la aplicación (por ejemplo, Todo0).
2. En la configuración de **Sign On**, selecciona **Edit** y agrega la URL de devolución de llamada en el campo **Redirect URI**.
3. Selecciona **Save**.

<Frame>![](/docs/images/xaa/xaa_advanced_sign_on_settings.png)</Frame>

Para probar la conexión empresarial de Okta Workforce, crea un usuario de prueba y dale permiso para iniciar sesión en la aplicación solicitante.

En la Consola de administración de Okta:

* Ve a **Applications** y selecciona la aplicación (por ejemplo, Agent0).
* Selecciona **Assign &gt; Assign to People** y elige tu usuario de prueba.
* Selecciona **Save**.

En el Auth0 Dashboard:

* Ve a **Authentication &gt; Enterprise &gt; Okta Workforce**:
  * Abre la conexión empresarial de Okta Workforce que creaste y selecciona la pestaña **Applications**. Luego, habilita para la conexión la aplicación solicitante que creaste.
  * Vuelve a la lista de conexiones de Okta Workforce. Selecciona los tres puntos a la derecha de tu conexión y elige **Try**. Se te redirigirá para autenticarte en tu tenant de Okta y completar el inicio de sesión con tu usuario de prueba.

<Frame>![](/docs/images/xaa/xaa_create_okta_workforce_connection.png)</Frame>

<div id="configure-an-organization">
  ### Configurar una organización
</div>

De forma opcional, si quiere que un cliente empresarial use Organizations, [cree una organización](/docs/es/manage-users/organizations/configure-organizations/create-organizations) y [habilite la conexión empresarial de Okta Workforce](/docs/es/manage-users/organizations/configure-organizations/enable-connections) para esa organización. Esto asocia automáticamente al `org_id` correspondiente los tokens de acceso generados con XAA dentro del scope de esta conexión, si el usuario de destino es miembro de la organización.

<Frame>![](/docs/images/xaa/xaa_enable_connection.png)</Frame>

También puede configurar el [comportamiento de la organización](/docs/es/manage-users/organizations/configure-organizations/define-organization-behavior) de la aplicación solicitante para definir si el uso de Organizations es obligatorio o está permitido. Le recomendamos empezar las pruebas con **Both**, ya que permite a los usuarios iniciar sesión como miembros de una organización o registrarse con una cuenta personal.

<Frame>![](/docs/images/xaa/xaa_organizations_both.png)</Frame>
