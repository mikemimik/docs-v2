Token Vault admite Privileged Worker Token Exchange, que permite a una aplicación cliente intercambiar un JWT firmado (`subject token`) por el token de acceso o el token de actualización de un proveedor externo (`requested token`).

Después de que la autenticación de usuario y la autorización se completen correctamente, una aplicación cliente normalmente pasa el contexto del usuario, que contiene su identidad, permisos y estado de sesión, como un token de acceso o un token de actualización para realizar el intercambio de tokens con Token Vault. En los flujos de servicio a servicio, una aplicación cliente, como una aplicación de backend o un service worker, puede necesitar acceder a recursos en nombre del usuario, pero como el “usuario no está presente” en una sesión interactiva, la aplicación cliente no tiene acceso al contexto del usuario.

En estos escenarios de servicio a servicio, la aplicación cliente puede generar un token JWT bearer firmado y usarlo como token de sujeto para realizar el intercambio de tokens y recibir los tokens necesarios para llamar a APIs externas. Esto significa que la aplicación cliente puede realizar acciones en nombre del usuario sin una interacción activa del usuario ni una sesión activa.

Para usar Privileged Worker Token Exchange with Token Vault, la aplicación cliente debe ser un cliente con altos privilegios que también pueda solicitar tokens de actualización de proveedores externos a través de Token Vault. Debe autenticarse con Token Vault mediante métodos criptográficos asimétricos, como una aserción de [clave privada JWT](/docs/es/get-started/authentication-and-authorization-flow/authenticate-with-private-key-jwt) o [autenticación mutua TLS](/docs/es/get-started/authentication-and-authorization-flow/authenticate-with-mtls).

<div id="prerequisites">
  ## Requisitos previos
</div>

Solo ciertos tipos de clientes pueden usar Privileged Worker Token Exchange with Token Vault:

* El cliente debe ser un cliente de primera parte; es decir, la propiedad `is_first_party` debe ser `true`.
* El cliente debe ser un cliente confidencial con un mecanismo de autenticación válido; es decir, la propiedad `token_endpoint_auth_method` no debe estar configurada como `none`.
* El cliente debe ser compatible con OIDC; es decir, `oidc_conformant` debe ser `true`.

Antes de configurar Privileged Worker Token Exchange para tu aplicación cliente:

1. [Habilita el tipo de concesión de Token Vault](/docs/es/secure/tokens/token-vault/configure-token-vault#configure-application) para tu aplicación cliente.
2. Configura [Clave privada JWT](/docs/es/get-started/authentication-and-authorization-flow/authenticate-with-private-key-jwt) o [autenticación mutua TLS](/docs/es/get-started/authentication-and-authorization-flow/authenticate-with-mtls) para tu aplicación cliente.

<div id="configure-client-application">
  ## Configurar la aplicación cliente
</div>

Para configurar el acceso privilegiado de la aplicación cliente a Token Vault, debes proporcionar una clave pública que se utilizará para verificar un JWT firmado usado como token de sujeto.

Al igual que al [configurar la solicitud de autorización protegida con JWT](/docs/es/get-started/applications/configure-jar#configure-jwt-secured-authorization-requests-jar), puedes establecer la clave pública para el acceso privilegiado a Token Vault al crear un nuevo cliente:

```bash lines
POST https://{yourDomain}.auth0.com/api/v2/clients
Authorization: Bearer <YOUR_MANAGEMENT_API_ACCESS_TOKEN>
Content-Type: application/json
{
  "name": "My App using JAR",
   “grant_types”: [“urn:auth0:params:oauth:grant-type:token-exchange:federated-connection-access-token”],
     “oidc_conformant”: true,
           “is_first_party”: true,
           “jwt_configuration”: {
             “alg”: 'RS256',
           },

  "token_vault_privileged_access": {
"credentials": [{
        "name": "My credential for Token Vault Privileged Access",
        "credential_type": "public_key",
        "pem": "<YOUR PEM FILE CONTENT>",
        "alg": "RS256"
}]
  },
}
```

También puedes actualizar un cliente existente con la clave pública de acceso privilegiado de Token Vault:

```bash lines
PATCH https://{yourDomain}.auth0.com/api/v2/clients/{yourClientId}
Authorization: Bearer <YOUR_MANAGEMENT_API_ACCESS_TOKEN>
Content-Type: application/json
{
  "token_vault_privileged_access": {
    "credentials": [{"id": "<YOUR CREDENTIAL ID>"}]
  }
}
```

<div id="create-signed-jwt-subject-token">
  ## Crear token de sujeto JWT firmado
</div>

Después de [configurar tu aplicación cliente con la clave pública](#configure-client-application), debes crear el token de sujeto que se intercambiará por un token de acceso para una API externa. El token de sujeto es un JSON Web Token (JWT) con los claims necesarios. Está firmado con la clave privada.

El JWT tiene un formato estándar y claims estándar, donde:

* El `typ` del header es `token-vault-req+jwt`
* El `kid` del header es opcional si solo tienes una clave pública configurada
* El `sub` del payload es el ID de usuario para el que quieres obtener el token
* El `aud` del payload es el host de tu tenant
* El `iss` del payload es el ID de cliente desde el que haces la solicitud

A continuación, se muestra un ejemplo de JWT:

```json lines
{
    alg: "RS256"  
    typ: "token-vault-req+jwt"
}
.
{
    sub: "auth0|000012030101231",
    aud: "https://{yourDomain}.auth0.com/",
    iss: "<YOUR_CLIENT_ID>",
    iat: 1758799540,
    exp: 1758800540,
    nbf: 1758799540
}
```

El siguiente ejemplo de código es un script que genera un token de sujeto JWT firmado:

```tsx lines
import * as jwt from 'jsonwebtoken';
   const privateKey = ‘-----BEGIN RSA PRIVATE KEY-----........’;
   const subjectToken = jwt.sign(
     {
       iss: CLIENT_ID,
       aud: 'https://' + TENANT_DOMAIN + '/',
       sub: USER_ID,
     },
     privateKey,
     {
       algorithm: 'RS256',
       header: {
         typ: 'token-vault-req+jwt',
       },
     }
   );
```

<div id="request-token-for-external-api">
  ## Solicitar un token para la API externa
</div>

Una vez que tengas el JWT firmado, puedes hacer una solicitud para obtener el token de acceso para la API externa:

```bash lines
curl --request POST 'https://{yourDomain}/oauth/token' \
--header 'Content-Type: application/json' \
--data '{
  "client_id": "<YOUR_CLIENT_ID>",
  "client_secret": "<YOUR_CLIENT_SECRET>",
  "subject_token": "<YOUR_SIGNED_JWT_BEARER>",
  "grant_type": "urn:auth0:params:oauth:grant-type:token-exchange:federated-connection-access-token",
  "subject_token_type": "urn:ietf:params:oauth:token-type:jwt",
  "requested_token_type": "http://auth0.com/oauth/token-type/token-vault-access-token",
  "connection": "google-oauth2"
}'
```

| Parámetro              | Descripción                                                                                                                                 |
| ---------------------- | ------------------------------------------------------------------------------------------------------------------------------------------- |
| `grant_type`           | El tipo de concesión. Para Token Vault, establécelo en `urn:auth0:params:oauth:grant-type:token-exchange:federated-connection-access-token` |
| `client_id`            | ID de la aplicación cliente                                                                                                                 |
| `client_secret`        | Secreto de cliente. **Nota:** Para Privileged Worker Token Exchange, recomendamos usar clave privada JWT o autenticación mTLS.              |
| `subject_token_type`   | Tipo de token de sujeto. Para Privileged Worker Token Exchange, establécelo en JWT: `urn:ietf:params:oauth:token-type:jwt`                  |
| `subject_token`        | El token JWT bearer firmado que valida el servidor de autorización de Auth0 para identificar al usuario.                                    |
| `requested_token_type` | El tipo de token solicitado. Para Privileged Worker Token Exchange, puedes solicitar un token de acceso o un token de actualización.        |
| `connection`           | El nombre de la conexión, en este caso, `google-oauth2`.                                                                                    |

Deberías recibir el token de acceso almacenado en Token Vault. Del mismo modo, puedes solicitar el token de actualización para la API externa:

```bash lines
curl --request POST 'https://{yourDomain}/oauth/token' \
--header 'Content-Type: application/json' \
--data '{
  "client_id": "<YOUR_CLIENT_ID>",
  "client_secret": "<YOUR_CLIENT_SECRET>",
  "subject_token": "<YOUR_SIGNED_JWT_BEARER>",
  "grant_type": "urn:auth0:params:oauth:grant-type:token-exchange:federated-connection-access-token",
  "subject_token_type": "urn:ietf:params:oauth:token-type:jwt",
  "requested_token_type": "http://auth0.com/oauth/token-type/token-vault-refresh-token",
  "connection": "google-oauth2"
}'
```
