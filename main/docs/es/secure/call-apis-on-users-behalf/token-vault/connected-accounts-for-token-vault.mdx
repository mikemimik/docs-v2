Connected Accounts for Token Vault permite a las aplicaciones acceder de forma segura a APIs externas en nombre del usuario mediante [Token Vault](/docs/es/secure/tokens/token-vault). Mientras que la autenticación de usuario estándar gestiona el inicio de sesión del usuario a través de un proveedor de identidad social o empresarial, Connected Accounts vincula un perfil de usuario con servicios externos como Google, GitHub, Slack y otros, lo que facilita el acceso delegado a APIs externas en nombre del usuario.

Una vez que un usuario conecta correctamente su cuenta y autoriza el acceso a un [proveedor externo compatible](/docs/es/secure/tokens/token-vault#supported-external-providers), Auth0:

* Asocia la cuenta con el usuario como una cuenta conectada.
* Almacena en Token Vault los tokens de acceso y los tokens de actualización del proveedor externo para la cuenta conectada.

Connected Accounts for Token Vault crea y gestiona un perfil de usuario de Auth0 unificado vinculado a varias cuentas externas, lo que permite una autorización fluida. A continuación, tu aplicación obtiene las credenciales almacenadas en Token Vault para interactuar con APIs externas en nombre del usuario.

<div id="user-authentication-vs-connected-accounts">
  ## autenticación de usuario vs Connected Accounts
</div>

Cuando [configuras Connected Accounts](#configure-connected-accounts) para una conexión social o empresarial compatible, Auth0 usa el flujo de Connected Accounts (endpoint `/me/v1/connected-accounts`) para recuperar y almacenar tokens de acceso y tokens de actualización en Token Vault, en lugar de usar el flujo de inicio de sesión social o empresarial (endpoint `/authorize`). Después de completar correctamente el flujo de Connected Accounts, Auth0 agrega la cuenta de usuario al arreglo `connected_accounts` del perfil de usuario. En cambio, en el flujo de inicio de sesión social o empresarial, Auth0 agrega la cuenta de usuario al arreglo `identities` del perfil de usuario.

La siguiente tabla explica las diferencias entre los flujos de autenticación de usuario y Connected Accounts:

|         | autenticación de usuario                                                    | Connected Accounts                                                                                                                                                                                           |
| ------- | --------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Flujo   | Flujo de inicio de sesión que usa el endpoint `/authorize`                  | Flujo de Connected Accounts que usa el endpoint `/me/v1/connected-accounts` de My Account API                                                                                                                |
| Purpose | Autentica a los usuarios con un proveedor de identidad social o empresarial | Almacena los tokens de acceso y tokens de actualización de la cuenta conectada en Token Vault cuando un usuario inicia sesión a través de un proveedor externo compatible, se conecta y autoriza la conexión |

Puedes habilitar la autenticación de usuario, Connected Accounts o ambos para conexiones sociales o empresariales compatibles. La siguiente tabla explica el comportamiento de las distintas configuraciones de Purpose, incluida la forma de pasar scopes a la conexión:

| Authentication | Connected Accounts | Behavior                                                                                                                                                                                                                                                                    | Scopes                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                              |
| -------------- | ------------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Habilitado     | Deshabilitado      | La conexión usa el flujo de inicio de sesión `/authorize` para autenticar usuarios como un proveedor de identidad válido.                                                                                                                                                   | Usa el Auth0 Dashboard o Management API para pasar los scopes que quieras a tu conexión. En runtime, esta lista se completa automáticamente con cualquier scope adicional incluido en el parámetro `connection_scope` de la authorization request.                                                                                                                                                                                                                                                                                                                                                                  |
| Deshabilitado  | Habilitado         | La conexión usa el flujo de Connected Accounts para recuperar y almacenar los tokens de la conexión en Token Vault. La conexión no usa el flujo de inicio de sesión `/authorize` para autenticar usuarios y queda excluida de la lista de proveedores de identidad válidos. | Usa el Auth0 Dashboard o Management API para pasar los scopes que quieras a tu conexión. En runtime, cualquier scope incluido en el parámetro `scopes` de la authorization request tiene prioridad sobre los scopes seleccionados en el Auth0 Dashboard, excepto `offline_access`, si la conexión lo requiere y está habilitado en el Auth0 Dashboard.<br /><br />**Nota:** Si la conexión lo requiere, Auth0 te pedirá que habilites `offline_access`, lo que permite a la aplicación cliente obtener un token de actualización de Auth0. Debes habilitar `offline_access` para la conexión en el Auth0 Dashboard. |
| Habilitado     | Habilitado         | La conexión usa el flujo de inicio de sesión `/authorize` para autenticar usuarios como un proveedor de identidad válido. También usa el flujo de Connected Accounts para recuperar y almacenar tokens de acceso de la conexión en Token Vault.                             | Usa el Auth0 Dashboard y Management API para pasar los scopes que quieras a la conexión. En runtime, cualquier scope incluido en el parámetro `scopes` tiene prioridad sobre los scopes seleccionados en el Auth0 Dashboard, excepto `offline_access`, si la conexión lo requiere y está habilitado en el Auth0 Dashboard.<br /><br />**Nota:** Si la conexión lo requiere, Auth0 te pedirá que habilites `offline_access`, lo que permite a la aplicación cliente obtener un token de actualización de Auth0. Debes habilitar `offline_access` para la conexión en el Auth0 Dashboard.                             |

<div id="how-it-works">
  ## Cómo funciona
</div>

El flujo de Connected Accounts usa la [My Account API](/docs/es/manage-users/my-account-api) para crear y gestionar cuentas conectadas de un usuario en distintos proveedores externos compatibles.

Antes de que el usuario pueda iniciar una solicitud de Connected Accounts desde la aplicación cliente, la aplicación cliente debe [obtener un access token](/docs/es/manage-users/my-account-api#get-an-access-token) con los scopes de Connected Accounts para acceder a la My Account API.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Si tu aplicación usa [Organizations](/docs/es/manage-users/organizations), autentica al usuario con la organización de destino antes de iniciar el flujo de Connected Accounts. Token Vault almacena la cuenta conectada en el perfil de Auth0 del usuario, por lo que cada miembro de la organización conecta y autoriza su propia cuenta externa.
</Callout>

El siguiente diagrama de secuencia ilustra el flujo de Connected Accounts de extremo a extremo:

<Frame>![](/docs/images/token-vault/connected_accounts_flow_diagram.png)</Frame>

Cuando un usuario inicia sesión a través de un proveedor externo compatible mediante Auth0, inicia una solicitud de Connected Accounts desde la aplicación cliente:

1. La aplicación cliente realiza una solicitud `POST` al endpoint `/me/v1/connected-accounts/connect` de la My Account API, pasando scopes y otros parámetros que se enviarán al proveedor externo. Para obtener más información, consulta [Iniciar solicitud de Connected Accounts](#initiate-connected-accounts-request).
2. La My Account API crea un `auth_session` único y un `connect_uri` que contiene un `ticket` y redirige al usuario a un navegador web. La aplicación cliente guarda el `auth_session` para verificarlo más adelante. Si [DPoP](/docs/es/secure/sender-constraining/demonstrating-proof-of-possession-dpop) está configurado, la My Account API valida el JWT de prueba DPoP.
3. La aplicación cliente redirige al usuario al `connect_uri` con el `ticket` como parámetro de consulta para la autenticación y autorización del usuario en el navegador. La aplicación cliente también puede pasar un `code_challenge` o `code_challenge_method` a la URL, como en el [flujo de código de autorización con PKCE](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce).
4. El usuario conecta y autoriza los permisos de la conexión en la pantalla de consentimiento.
5. Después de que el usuario autoriza correctamente la conexión, el proveedor externo redirige al usuario a la My Account API, que a su vez lo redirige a la aplicación cliente mediante el `redirect_uri` con un `connect_code` de un solo uso.
6. La aplicación cliente presenta el `connect_code`, el `code_verifier` (si corresponde) y el `auth_session` original a la My Account API mediante una solicitud `POST` al endpoint `/me/v1/connected-accounts/complete`. Para obtener más información, consulta [Completar solicitud de Connected Accounts](#complete-connected-accounts-request).
7. La My Account API valida la solicitud confirmando:
   * Que el `auth_session` coincide con el ID emitido originalmente para el usuario
   * Que la solicitud proviene del mismo dispositivo que inició el flujo de Connected Accounts
   * El JWT de prueba DPoP (si está configurado)
   * El `connect_code` de un solo uso
   * El `code_verifier` (si se usa el flujo de PKCE)
8. Después de validarla correctamente, el servidor de autorización de Auth0 agrega la cuenta al array `connected_accounts` del perfil del usuario y almacena los tokens de acceso y de actualización de la cuenta conectada en Token Vault.
9. La My Account API completa el flujo enviando un código de estado `200` de vuelta a la aplicación cliente, lo que indica que la cuenta se conectó correctamente.

<div id="prerequisites">
  ## Requisitos previos
</div>

Antes de configurar Connected Accounts, asegúrate de hacer lo siguiente:

* [Configurar Token Vault](/docs/es/secure/tokens/token-vault/configure-token-vault) para tu aplicación cliente, a fin de almacenar de forma segura en Token Vault los tokens de acceso y los tokens de actualización asociados a cada cuenta conectada.
* [Configurar la My Account API](#configure-my-account-api), que los usuarios autenticados utilizan para conectar y administrar cuentas.
* [Configurar el token de actualización de múltiples recursos (MRRT)](#configure-multi-resource-refresh-token) para obtener un token de acceso para la My Account API.
* (Opcional) [Configurar DPoP](/docs/es/secure/sender-constraining/configure-sender-constraining) para la My Account API y tu aplicación cliente, a fin de aplicar Sender Constraining a los tokens de acceso y evitar el robo de tokens. De forma predeterminada, la My Account API puede aceptar tokens de acceso vinculados a DPoP.

<div id="configure-my-account-api">
  ### Configurar My Account API
</div>

Para usar Connected Accounts, configura la My Account API en el Auth0 Dashboard:

1. Ve a **Applications &gt; APIs** y [activa la My Account API](/docs/es/manage-users/my-account-api#activate-the-my-account-api).
2. Una vez activada, selecciona **Auth0 My Account API** y luego la pestaña **Application Access**.
3. Busca tu aplicación cliente y selecciona **Edit** para configurar sus [políticas de acceso de la aplicación](/docs/es/get-started/apis/api-access-policies-for-applications).
4. Selecciona **User Access** y, en **Authorization**, selecciona **Authorized**.
5. En los permisos, selecciona **All** para todos los [Connected Accounts scopes](/docs/es/manage-users/my-account-api#scope) de la aplicación.
6. Selecciona **Save**. Esto crea una [autorización de cliente](/docs/es/get-started/applications/application-access-to-apis-client-grants) que permite que tu aplicación cliente acceda a la My Account API con los Connected Accounts scopes en nombre del usuario.
7. Si estás usando [token de actualización de múltiples recursos](/docs/es/secure/tokens/refresh-tokens/multi-resource-refresh-token#multi-resource-refresh-token), ve a la pestaña **Settings**. En **Access Settings**, selecciona **Allow Skipping User Consent**.

<div id="configure-multi-resource-refresh-token">
  ### Configurar el token de actualización de múltiples recursos
</div>

Configura el token de actualización de múltiples recursos (MRRT) para obtener un único token de actualización de larga duración que pueda intercambiarse por nuevos tokens de acceso de My Account API y otras API, sin necesidad de que el usuario vuelva a autenticarse.

Puedes configurar MRRT con el [Auth0 Dashboard](https://manage.auth0.com) o la [Management API](https://auth0.com/docs/api/management/v2).

<Tabs>
  <Tab title="Auth0 Dashboard">
    Para configurar MRRT con Auth0 Dashboard:

    1. Ve a **Applications &gt; Applications** y selecciona tu aplicación.
    2. En **Multi-Resource Refresh Token**, selecciona **Edit Configuration**.
    3. Para habilitar MRRT con My Account API, activa **My Account API**.
  </Tab>

  <Tab title="Management API">
    Para configurar MRRT para una aplicación cliente, realiza una solicitud `PATCH` al endpoint `/api/v2/clients/{clientId}` y agrega el identificador de My Account API y los scopes de Connected Accounts a las policies del token de actualización:

    ```bash lines
    curl -X PATCH --location "https://${account.namespace}/api/v2/clients/{clientId}" \
        -H "Authorization: Bearer <YOUR_MANAGEMENT_API_TOKEN>" \
        -H "Content-Type: application/json" \
        -d '{
              "is_first_party": true,
              "refresh_token": {
                "expiration_type": "non-expiring",
                "leeway": 0,
                "infinite_token_lifetime": true,
                "infinite_idle_token_lifetime": true,
                "token_lifetime": 31557600,
                "idle_token_lifetime": 2592000,
                "rotation_type": "non-rotating",
                "policies": [
                  {
                    "audience": "https://'"$DOMAIN"'/me/",
                    "scope": [
                      "create:me:connected_accounts",
                      "read:me:connected_accounts",
                      "delete:me:connected_accounts"
                    ]
                  }
                ]
              }
            }'
    ```
  </Tab>
</Tabs>

<div id="configure-connected-accounts">
  ## Configurar Connected Accounts
</div>

Antes de configurar Connected Accounts para una conexión, asegúrate de haber autorizado la conexión para tu aplicación cliente.

En el Auth0 Dashboard:

1. Ve a **Authentication &gt; Social Connections** o **Enterprise Connections** y selecciona la conexión.
2. Selecciona **Applications** y, a continuación, activa la conexión para tu aplicación cliente.

Puedes configurar Connected Accounts con el [Auth0 Dashboard](https://manage.auth0.com) o la [Management API](https://auth0.com/docs/api/management/v2).

<Tabs>
  <Tab title="Auth0 Dashboard">
    Para configurar Connected Accounts con el Auth0 Dashboard:

    1. Ve a **Authentication &gt; Social Connections** o **Enterprise Connections**.
    2. Selecciona **Create Connection** o una conexión existente.
    3. En **Purpose**, activa **Connected Accounts for Token Vault**. Según tu configuración de **Purpose**, es posible que debas habilitar `offline_access` en el Auth0 Dashboard para que la aplicación cliente pueda obtener un token de actualización del proveedor externo durante el flujo de Connected Accounts. Para obtener más información, consulta [User authentication vs Connected Accounts](#user-authentication-vs-connected-accounts).
    4. Haz clic en **Save**.
  </Tab>

  <Tab title="Management API">
    Para configurar Connected Accounts con la Management API, realiza una solicitud `PATCH` al endpoint `/connections/{connectionId}` para establecer `connected_accounts` en `true`:

    ```bash lines
    curl -L -X PATCH "https://{yourDomain}/api/v2/connections/{connectionId}" \
    -H 'Content-Type: application/json' \
    -H 'Accept: application/json' \
    -H "Authorization: Bearer <YOUR_MANAGEMENT_API_TOKEN>" \
    -d '{"connected_accounts":{"active":true}}'
    ```
  </Tab>
</Tabs>

<div id="get-access-token-for-connected-accounts">
  ## Obtener un token de acceso para Connected Accounts
</div>

Antes de iniciar una solicitud de Connected Accounts, [obtén un token de acceso](/docs/es/manage-users/my-account-api#scope) para la My Account API con los scopes de Connected Accounts.

Las siguientes secciones explican cómo usar [el token de actualización de múltiples recursos (MRRT)](/docs/es/secure/tokens/refresh-tokens/multi-resource-refresh-token) para obtener un token de acceso para la My Account API.

<div id="fetch-a-refresh-token">
  ### Obtener un token de actualización
</div>

Después de [configurar el token de actualización de múltiples recursos (MRRT)](#configure-multi-resource-refresh-token) para la aplicación cliente, inicia el flujo de código de autorización e intercambia el código de autorización resultante por un token de actualización.

A continuación se muestra una solicitud de flujo de código de autorización para un cliente confidencial que incluye `offline_scope` para devolver un token de actualización y un código de autorización de un solo uso para el identificador de la API My Account `https://{yourDomain}/me/`:

```bash lines
open "https://{yourDomain}/authorize?client_id=<CLIENT_ID>&response_type=code&prompt=login&scope=openid%20profile%20offline_access&redirect_uri=<REDIRECT_URI>&state=<STATE>&audience=https://my-example-api.com"
```

Canjea el código de autorización de un solo uso por un token de actualización en el endpoint `/token`:

```bash lines
curl -s --request POST \
  --url "https://{yourDomain}/oauth/token" \
  --header 'Content-Type: application/json' \
  --data-binary @- <<EOF | jq -r '.refresh_token'
{
  "grant_type": "authorization_code",
  "code": "<CONNECT_CODE>",
  "client_id": "<CLIENT_ID>",
  "client_secret": "<CLIENT_SECRET>",
  "redirect_uri": "<REDIRECT_URI>"
}
EOF
```

<div id="exchange-refresh-token-for-my-account-api-access-token">
  ### Canjear un token de actualización por un token de acceso a My Account API
</div>

Una vez que hayas obtenido un token de actualización, cámbialo por un token de acceso a My Account API con los scopes de Connected Accounts mediante el tipo de concesión de token de actualización:

```bash lines
curl -s -X POST "https://{yourDomain}/oauth/token" \
  -H "Content-Type: application/x-www-form-urlencoded" \
  --data-urlencode "grant_type=refresh_token" \
  --data-urlencode "client_id=<CLIENT_ID>" \
  --data-urlencode "client_secret=<CLIENT_SECRET>" \
  --data-urlencode "refresh_token=<REFRESH_TOKEN>" \
  --data-urlencode "audience=https://{yourDomain}/me/" \
  --data-urlencode "scope=openid profile offline_access create:me:connected_accounts read:me:connected_accounts delete:me:connected_accounts"
```

<div id="initiate-connected-accounts-request">
  ## Iniciar una solicitud de Connected Accounts
</div>

Para iniciar una solicitud de Connected Accounts, haz una solicitud `POST` al endpoint `/me/v1/connected-accounts/connect` de My Account API con los siguientes parámetros:

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Para una conexión social de Google, asegúrate de seleccionar `offline_access` en el Auth0 Dashboard al configurar la conexión. Esto es necesario para que tu aplicación cliente obtenga un token de actualización del servidor de autorización de Auth0.
</Callout>

| Parámetro      | Descripción                                                                                                                                                                                                                                                                                                                                                                                                                             |
| -------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `connection`   | Nombre de la conexión. Para una conexión social de Google, usa `google-oauth2`.                                                                                                                                                                                                                                                                                                                                                         |
| `redirect_uri` | La URL de devolución de llamada de tu aplicación cliente.                                                                                                                                                                                                                                                                                                                                                                               |
| `state`        | Una cadena única y aleatoria asociada con la solicitud para evitar ataques.                                                                                                                                                                                                                                                                                                                                                             |
| `scopes`       | (Opcional) Los scopes que se pasan al proveedor externo como un arreglo de cadenas.<br /><br />Si se usa para pasar scopes para una conexión social de Google, incluye como mínimo `openid` y `profile`. En el runtime, cualquier scope incluido en el parámetro `scopes` prevalece sobre los scopes seleccionados en el Auth0 Dashboard, excepto `offline_access`, si la conexión lo requiere y está habilitado en el Auth0 Dashboard. |

```bash lines
curl --request POST "https://{yourDomain}/me/v1/connected-accounts/connect" \
--header 'Content-Type: application/json' \
--header "Authorization: Bearer <MY_ACCOUNT_API_TOKEN>" \
--data '{
    "connection": "google-oauth2",
    "redirect_uri": "<REDIRECT_URI>",
    "state": "<STATE>",
    "scopes": ["openid","profile"] // los scopes que se pasen sobrescriben los scopes seleccionados en el Auth0 Dashboard
}'
```

Si la operación se realiza correctamente, My Account API devuelve una respuesta como la siguiente:

| Parámetro        | Descripción                                                                                                                                                 |
| ---------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `auth_session`   | ID de sesión que representa la sesión autenticada actual del usuario principal. La aplicación cliente guarda el ID de sesión para verificarlo más adelante. |
| `connect_uri`    | URL a la que la aplicación cliente redirige al usuario y que abre un navegador web para gestionar la autorización con el proveedor externo.                 |
| `connect_params` | Parámetros adicionales necesarios para el URI de conexión. Incluye un ticket temporal que My Account API usa para verificar la solicitud.                   |
| `expires_in`     | Tiempo de expiración de la sesión, en segundos.                                                                                                             |

```json
{
  "auth_session": "PKM-CYkdx2FyLb4Oob4ED91cSE7i_XJ4SVJByik0xKQxz9CgUZ5JlYr-aMPty0Xr",
  "connect_uri": "https://{yourDomain}.us.auth0.com/connect",
  "connect_params": {
    "ticket": "9375f326-5846-4b57-ae8b-8042573f7c1f"
  },
  "expires_in": 300
}
```

En el navegador web, ve a `connect_uri` con el `ticket` como parámetro de consulta. Autoriza la lista de scopes en la pantalla de consentimiento y luego extrae y guarda el `connect_code` del fragmento de la URL.

```bash lines
open https://{yourDomain}.us.auth0.com/connected-accounts/connect?ticket={tickedId}
```

<div id="complete-connected-accounts-request">
  ## Completar la solicitud de Connected Accounts
</div>

Para completar una solicitud de Connected Accounts, realiza una solicitud `POST` al endpoint `/me/v1/connected-accounts/complete` con los siguientes parámetros:

| Parámetro      | Descripción                                                                                                                                                                                                                                |
| -------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `auth_session` | ID de sesión que representa la sesión autenticada actual del usuario principal. La aplicación cliente guarda el ID de sesión para su posterior verificación.                                                                               |
| `connect_code` | Código de un solo uso y de corta duración recibido del proceso de autorización del proveedor externo. Este código se intercambia de forma segura del lado del servidor para recuperar los tokens de acceso finales de la API externa.      |
| `redirect_uri` | La URL de devolución de llamada exacta de tu aplicación a la que se envió al usuario después de autorizar correctamente la conexión con el proveedor externo. Este valor debe coincidir con el `redirect_uri` usado para iniciar el flujo. |

```bash lines
curl --location "https://{yourDomain}/me/v1/connected-accounts/complete" \
--header 'Content-Type: application/json' \
--header "Authorization: Bearer <MY_ACCOUNT_API_TOKEN>" \
--data '{
    "auth_session": "<AUTH_SESSION>",
    "connect_code": "<CONNECT_CODE>",
    "redirect_uri": "<REDIRECT_URI>"
}'
```

Si la operación se realiza correctamente, My Account API devuelve una respuesta como la siguiente:

| Parameter     | Description                                                                                                                                                                                                                                           |
| ------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `id`          | Identificador único de la cuenta conectada.                                                                                                                                                                                                           |
| `connection`  | Nombre de la conexión.                                                                                                                                                                                                                                |
| `created_at`  | Marca de tiempo de cuándo se creó la cuenta conectada y se vinculó al perfil del usuario.                                                                                                                                                             |
| `scopes`      | Los `scopes` específicos de OAuth (permisos) a los que el usuario concedió acceso a tu aplicación al conectarse con el proveedor externo. Estos `scopes` determinan qué acciones puede realizar tu aplicación en la API externa.                      |
| `access_type` | Indica el tipo de acceso concedido. Un valor habitual es `offline`, lo que significa que se obtuvo y almacenó correctamente un token de actualización, lo que permite a tu aplicación mantener el acceso incluso cuando el usuario no está conectado. |

```json
{
  "id": "cac_6ZqSK7Kj1R8LDZJvSb1tAn",
  "connection": "google-oauth2",
  "created_at": "2025-10-13T21:09:04.126Z",
  "scopes": [
    "https://www.googleapis.com/auth/calendar",
    "https://www.googleapis.com/auth/calendar.addons.execute",
    "https://www.googleapis.com/auth/calendar.events",
    "https://www.googleapis.com/auth/calendar.events.readonly",
    "https://www.googleapis.com/auth/calendar.settings.readonly",
    "https://www.googleapis.com/auth/userinfo.profile",
    "openid"
  ],
  "access_type": "offline"
}
```

<div id="manage-connected-accounts">
  ## Gestionar Connected Accounts
</div>

Para gestionar las Connected Accounts de un usuario, utilice la colección `/me/v1/connected-accounts`.

Antes de usar la colección `/connected-accounts`, [obtenga un token de acceso para Connected Accounts](#get-access-token-for-connected-accounts).

<div id="query-connected-accounts-connections">
  ### Consultar conexiones de Connected Accounts
</div>

<div id="delete-connected-account">
  ### Eliminar cuenta conectada
</div>

Haz una solicitud `DELETE` al endpoint `/me/v1/connected-accounts/accounts/{connectedAccountId}` para eliminar una cuenta conectada con un ID determinado:

```bash lines
curl -X DELETE --location "https://{yourDomain}/me/v1/connected-accounts/accounts/{connectedAccountId}" \
--header 'Content-Type: application/json' \
--header "Authorization: Bearer <MY_ACCOUNT_API_TOKEN>"
```

Cuando eliminas una cuenta conectada, Auth0 elimina del Token Vault los tokens de acceso y de actualización del proveedor externo. Esto no revoca automáticamente los tokens del proveedor externo, y el token de actualización todavía podría usarse para obtener nuevos tokens de acceso. Debes revocar manualmente los tokens del proveedor externo si se han compartido o copiado en otro lugar.

Si la operación se realiza correctamente, la API My Accounts devuelve una respuesta como la siguiente:

```
HTTP/1.1 204 No Content
```
