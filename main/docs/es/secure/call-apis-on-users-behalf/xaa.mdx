Conectar aplicaciones de terceros y agentes de IA en una empresa genera dos problemas clave: poca visibilidad por parte de TI sobre el intercambio de datos y flujos de consentimiento repetitivos para los usuarios.

Cross App Access (XAA) resuelve estos desafíos al permitir que los administradores de TI definan de forma centralizada controles de acceso sobre cómo las aplicaciones SaaS, como los agentes de IA, se conectan en nombre de un usuario. Los administradores gestionan estas conexiones desde un panel central, como la Consola de administración de Okta, lo que elimina las molestas solicitudes de consentimiento de OAuth para los usuarios finales. El resultado es una mejora en la seguridad, la gobernanza y la experiencia de usuario de la organización.

XAA implementa el [Identity Assertion Authorization Grant](https://datatracker.ietf.org/doc/draft-ietf-oauth-identity-assertion-authz-grant/), una extensión de OAuth en desarrollo que permite que un agente de IA o una aplicación (aplicación solicitante) obtenga un token seguro a través del IdP empresarial. Este token permite que la aplicación solicitante llame a las API de otra aplicación (aplicación de recursos) en nombre del usuario final. Para obtener más información, consulta [Cómo funciona](#how-it-works).

<div id="key-benefits">
  ## Beneficios clave
</div>

XAA ofrece beneficios clave para cada rol de tu ecosistema empresarial:

* Para los administradores de TI de la empresa: control centralizado, visibilidad y aplicación de políticas sobre el acceso de las aplicaciones a los datos de la empresa y de los usuarios.
* Para los proveedores de SaaS y los desarrolladores: integración estandarizada y segura para la IA empresarial que impulsa el crecimiento del ecosistema.
* Para los usuarios finales: conexiones ágiles y sin fricciones entre aplicaciones, lo que elimina los complejos flujos de consentimiento de OAuth.

<div id="use-cases">
  ## Casos de uso
</div>

Entre los casos de uso más habituales de XAA se incluyen:

* Conectar agentes de IA a aplicaciones empresariales: un empleado usa un agente de IA para consultar su aplicación de calendario y publicar una actualización sobre su disponibilidad en la aplicación empresarial de mensajería. En lugar de exigir que el empleado pase por flujos de redirección y solicitudes de consentimiento, el agente de IA usa XAA para obtener un token de acceso del IdP empresarial y llamar de forma segura a las API tanto de la aplicación de calendario como de la de mensajería, si la política de acceso empresarial lo permite.
* Conectar aplicaciones SaaS: en el ejemplo anterior, tanto la aplicación empresarial de calendario como la de mensajería son compatibles con XAA. Los empleados pueden conectar sin problemas la aplicación de mensajería para acceder a la API de la aplicación de calendario sin redirección del usuario ni consentimiento, y a la vez cumplir con las políticas de acceso empresariales.

<div id="how-it-works">
  ## Cómo funciona
</div>

El flujo XAA involucra a los siguientes actores:

* Aplicación solicitante: La aplicación o agente de IA que necesita acceder a un recurso.
* Aplicación de recursos: La aplicación que posee el recurso protegido y lo expone mediante una API
* IdP empresarial: El IdP, como Okta, que autentica a los empleados.

Después de que el usuario final se autentique con el IdP empresarial, la aplicación solicitante contacta con el IdP empresarial para solicitar acceso a la aplicación de recursos en nombre del usuario. Tras aplicar su política de acceso para comprobar si esta conexión entre aplicaciones está permitida, el IdP empresarial genera una aserción llamada ID-JAG, que la aplicación solicitante presenta después a la aplicación de recursos para obtener un token de acceso y consumir la API. 

En el siguiente diagrama, Acme es el cliente empresarial cuyos empleados se autentican con su IdP empresarial, como Okta, para acceder a la aplicación solicitante (Agent0) y a la aplicación de recursos (Todo0):

<Frame>![](/docs/images/xaa/xaa_high_level_diagram.png)</Frame>

* El servidor de autorización de la aplicación de recursos (Todo0) está federado con el IdP empresarial a través de OIDC para que pueda generar tokens de acceso para usuarios finales autenticados por ese IdP.
* La aplicación solicitante (Agent0) está registrada en el servidor de autorización de la aplicación de recursos como un cliente de OAuth 2.0 con un `client_id` válido y credenciales para solicitar tokens de acceso al servidor de autorización de la aplicación de recursos.
* El administrador de TI de Acme ha definido controles de acceso XAA entre Agent0 y Todo0.

<div id="end-to-end-xaa-flow">
  ## Flujo XAA de extremo a extremo
</div>

Teniendo en cuenta nuestro ejemplo de Acme, el flujo XAA de extremo a extremo consta de los siguientes pasos:

1. El empleado de Acme inicia sesión en la aplicación solicitante (Agent0) mediante SSO con el IdP empresarial. La aplicación solicitante obtiene un token de identidad para verificar la identidad del empleado de Acme.
2. La aplicación solicitante realiza una solicitud de intercambio de tokens al IdP para intercambiar el token de identidad por una concesión de autorización JWT de aserción de identidad entre dominios, también conocida como ID-JAG. El IdP valida la solicitud y comprueba la política XAA definida por el administrador de TI de Acme.
3. Si la política XAA lo permite, el IdP devuelve el ID-JAG a la aplicación solicitante.
4. La aplicación solicitante realiza una solicitud de token al servidor de autorización de la aplicación de recursos (Todo0) mediante el ID-JAG.
5. El servidor de autorización de la aplicación de recursos valida el ID-JAG mediante la clave pública que también utiliza para su flujo de OpenID Connect con el IdP. Si es válido, el servidor de autorización devuelve un token de acceso.
6. La aplicación solicitante realiza una solicitud a la API de la aplicación de recursos con el token de acceso.

Al aprovechar el flujo XAA, las políticas del administrador de TI de Acme rigen el acceso de Agent0 a Todo0, sin requerir redirección ni interacción del usuario final.

<div id="beta-limitations">
  ## Limitaciones de la Beta
</div>

XAA Beta tiene las siguientes limitaciones:

* La aplicación solicitante debe ser un cliente confidencial y una aplicación de primera parte en tu tenant de Auth0. Los clientes públicos, como las SPA y las aplicaciones nativas, no se admiten.
* La Administración delegada no es compatible. El cliente empresarial no puede configurar directamente conexiones de SSO en tu tenant de Auth0. La compatibilidad con [SSO de autoservicio](/docs/es/authenticate/enterprise-connections/self-service-enterprise-configuration) está prevista para una versión posterior.
* Solo puede haber una conexión con XAA habilitado por cada emisor del IdP de origen. El mismo tenant de Okta no puede usarse para más de una conexión empresarial con XAA habilitado.
* La compatibilidad con Organizations es limitada:
  * Una conexión tiene una asignación 1:1 con una organización. No se pueden asignar varias Organizations a la misma conexión para el acceso a XAA.
  * Cuando la aplicación solicitante está configurada para requerir el uso de Organizations, los usuarios ya deben ser miembros de la organización de destino.
* Si no se especifica el parámetro `resource` en la solicitud ID-JAG, la API de destino se determina mediante `tenant.default_audience`.
* No hay creación dinámica de usuarios: el usuario debe haber iniciado sesión previamente en tu aplicación de recursos con la conexión de Okta configurada. De lo contrario, la solicitud para intercambiar la aserción ID-JAG por un token de acceso fallará con un `User not found error`.

<div id="rate-limits">
  ## Límites de frecuencia
</div>

En XAA Beta, los intercambios ID-JAG en el endpoint `/token` de tu tenant de Auth0 estarán limitados a 5 RPS.
