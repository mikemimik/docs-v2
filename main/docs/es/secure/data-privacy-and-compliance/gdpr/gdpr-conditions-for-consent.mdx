De conformidad con el artículo 7 del RGPD, debe solicitar a los usuarios su consentimiento para el tratamiento de sus datos personales de forma clara y fácilmente accesible. También debe poder demostrar que el usuario ha dado su consentimiento y ofrecer un mecanismo sencillo para retirarlo en cualquier momento.

En este artículo se explica cómo puede utilizar las funcionalidades de Auth0 para cumplir estos requisitos.

  El contenido de estos documentos no pretende ser asesoramiento jurídico ni debe considerarse un sustituto del asesoramiento legal. La responsabilidad final de entender y cumplir el RGPD recae en usted, aunque Auth0 le ayudará, en la medida de lo posible, a cumplir los requisitos del RGPD.
</Callout>

<div id="ask-for-consent">
  ## Solicita el consentimiento
</div>

Al registrarse, debes pedirles a tus usuarios su consentimiento. Con Auth0, puedes guardar esta información en los metadatos del usuario. Hay varias opciones disponibles según cómo uses Auth0 para autenticar a tus usuarios. Antes de diseñar tu solución con metadatos, asegúrate de conocer las restricciones. Auth0 limita el tamaño total de `user_metadata` a **16 MB**. Para obtener más información, consulta [Nombres de campos de metadatos y tipos de datos](/docs/es/manage-users/user-accounts/metadata/metadata-fields-data).

<Warning>
  Los metadatos de Auth0 no son un almacenamiento de datos seguro y no deben usarse para guardar información confidencial, como secretos de alto riesgo e información de identificación personal (PII), por ejemplo, números de seguridad social o de tarjetas de crédito. Se recomienda encarecidamente a los clientes de Auth0 que evalúen los datos almacenados en los metadatos y que solo guarden los que sean necesarios para fines de gestión de identidades y accesos.
</Warning>

<div id="use-lock">
  ### Usar Lock
</div>

Puedes personalizar la interfaz de usuario de Lock para mostrar enlaces a tus páginas de términos y condiciones o de declaración de privacidad, así como una casilla de consentimiento que el usuario debe marcar para poder registrarse. Esto se puede hacer con la opción de Lock `mustAcceptTerm`s. Esta propiedad, cuando se establece en `true`, muestra una casilla junto a los términos y condiciones que debe marcarse antes de registrarse. Los términos y condiciones se pueden especificar mediante la opción `languageDictionary`. Para obtener más información, consulta [Lock Configuration Options](/docs/es/libraries/lock/lock-configuration).

Una vez que el usuario haya aceptado y se haya registrado, guarda la información de consentimiento en `user_metadata` mediante una `rule` que se ejecutará en el primer inicio de sesión. Para obtener más información sobre Rules, consulta [Auth0 Rules](/docs/es/customize/rules).

Si quieres obtener más información de los usuarios durante el registro y autenticas a los usuarios con una conexión de base de datos, puedes añadir campos personalizados a la interfaz de usuario de Lock. Esto se puede hacer con la opción de Lock `additionalSignUpFields`. Todos los campos personalizados se añaden automáticamente a `user_metadata`.

Si utilizas social logins, añadir campos personalizados no es una opción, pero puedes redirigir al usuario a otra página para solicitarle el consentimiento y cualquier información adicional, y luego redirigirlo de vuelta para completar la transaction de autenticación. Esto se puede hacer con redirect rules. Para obtener más información, consulta [Redirect Users from Within Rules](/docs/es/customize/rules/redirect-users). Una vez completado el proceso de registro, guarda la información de consentimiento en `user_metadata` llamando al <Tooltip tip="Management API: Un producto para permitir a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip> [**Update User** endpoint](https://auth0.com/docs/api/management/v2#!/Users/patch_users_by_id).

Para aprender a implementar cualquiera de estos escenarios, consulta [GDPR: Track Consent with Lock](/docs/es/secure/data-privacy-and-compliance/gdpr/gdpr-track-consent-with-lock).

<div id="use-custom-ui">
  ### Usa una UI personalizada
</div>

Si usas un signup form personalizado con una conexión de base de datos, tienes que añadir un campo adicional a la pantalla de Signup para recopilar el consentimiento del usuario. Después, llama al [**endpoint de Signup**](https://auth0.com/docs/api/authentication#signup) de la Authentication API para crear el usuario en Auth0. En este punto, puedes establecer la información de consentimiento como parte de `user_metadata`.

Como alternativa, si usas Auth0.js desde una SPA, puedes [usar el método `signup`](/docs/es/libraries/auth0js) para crear el usuario en Auth0 y establecer la información de consentimiento como parte de `user_metadata`.

Si usas un signup form personalizado con proveedores sociales, no puedes establecer la información de consentimiento del usuario durante el Signup, pero puedes actualizarla en cuanto se cree el usuario. Guarda la información de consentimiento en `user_metadata` llamando al [**endpoint de Update User**](https://auth0.com/docs/api/management/v2#!/Users/patch_users_by_id) de la Management API.

Para aprender a implementar cualquiera de estos escenarios, consulta [RGPD: seguimiento del consentimiento con UI personalizada](/docs/es/secure/data-privacy-and-compliance/gdpr/gdpr-track-consent-with-custom-ui).

<div id="re-consent-and-user-migration">
  ### Renovación del consentimiento y migración de usuarios
</div>

Si necesitas solicitar el consentimiento de usuarios existentes y decides migrarlos desde una base de datos existente a Auth0, puedes [usar nuestra funcionalidad de migración automática de usuarios](/docs/es/manage-users/user-migration/configure-automatic-migration-from-your-database). Al activarla, cada vez que un usuario inicie sesión por primera vez (desde su activación), se creará en Auth0 sin necesidad de restablecer su contraseña. Para ello, debes:

* Redactar la notificación que verán los usuarios sobre cómo se usarán sus datos, durante cuánto tiempo se utilizarán, cuáles son sus derechos, etc., así como personalizar el cuadro de registro de la interfaz de usuario.
* Determinar si es necesario volver a solicitar el consentimiento de tus usuarios, en función de tus términos y condiciones anteriores y de las certificaciones de privacidad previas.

Ten en cuenta que cada vez que cambien tus Términos y Condiciones, **debes** volver a solicitar el consentimiento de los usuarios.

<div id="track-consent">
  ## Seguimiento del consentimiento
</div>

Según el RGPD, deberías poder demostrar que el usuario ha dado su consentimiento para el tratamiento de sus datos personales.

Con Auth0, puedes guardar la información sobre el consentimiento del usuario como parte de `user_metadata`. Puedes guardar solo un indicador que muestre si el usuario ha dado su consentimiento o no, o bien un conjunto de datos y preferencias de consentimiento (incluidos, por ejemplo, el día en que lo otorgó, los términos que aceptó, etc.). Después, puedes acceder a esta información y gestionarla mediante la Management API.

La Management API también ofrece varias opciones para la búsqueda de usuarios, así como endpoints para actualizar los metadatos del usuario o exportar usuarios por lotes.

Para acceder a la Management API, necesitarás un <Tooltip tip="Token de acceso: credencial de autorización, en forma de una cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=access+token">token de acceso</Tooltip>. Para aprender a obtener un token de acceso para la Management API, consulta [Tokens de acceso de la Management API](/docs/es/secure/tokens/access-tokens/management-api-access-tokens).

<div id="search-for-users-by-email-address">
  ### Buscar usuarios por dirección de correo electrónico
</div>

Para buscar un usuario por su dirección de correo electrónico, utiliza el [endpoint **Search User by Email**](/docs/es/manage-users/user-search/user-search-best-practices).

Establece el parámetro de solicitud **fields** en `user_metadata` para limitar los campos que se devuelven. De esta manera, solo se devolverá user&#95;metadata en lugar del perfil de usuario completo.

Solicitud de ejemplo:

<AuthCodeGroup>
  ```bash cURL
  curl --request GET \
    --url 'https://{yourDomain}/api/v2/users-by-email?email=USER_EMAIL_ADDRESS&fields=user_metadata' \
    --header 'authorization: Bearer YOUR_MGMT_API_ACCESS_TOKEN'
  ```

  ```csharp C#
  var client = new RestClient("https://{yourDomain}/api/v2/users-by-email?email=USER_EMAIL_ADDRESS&fields=user_metadata");
  var request = new RestRequest(Method.GET);
  request.AddHeader("authorization", "Bearer YOUR_MGMT_API_ACCESS_TOKEN");
  IRestResponse response = client.Execute(request);
  ```

  ```go Go
  package main

  import (
  	"fmt"
  	"net/http"
  	"io/ioutil"
  )

  func main() {

  	url := "https://{yourDomain}/api/v2/users-by-email?email=USER_EMAIL_ADDRESS&fields=user_metadata"

  	req, _ := http.NewRequest("GET", url, nil)

  	req.Header.Add("authorization", "Bearer YOUR_MGMT_API_ACCESS_TOKEN")

  	res, _ := http.DefaultClient.Do(req)

  	defer res.Body.Close()
  	body, _ := ioutil.ReadAll(res.Body)

  	fmt.Println(res)
  	fmt.Println(string(body))

  }
  ```

  ```java Java
  HttpResponse response = Unirest.get("https://{yourDomain}/api/v2/users-by-email?email=USER_EMAIL_ADDRESS&fields=user_metadata")
    .header("authorization", "Bearer YOUR_MGMT_API_ACCESS_TOKEN")
    .asString();
  ```

  ```javascript Node.JS
  var axios = require("axios").default;

  var options = {
    method: 'GET',
    url: 'https://{yourDomain}/api/v2/users-by-email',
    params: {email: 'USER_EMAIL_ADDRESS', fields: 'user_metadata'},
    headers: {authorization: 'Bearer YOUR_MGMT_API_ACCESS_TOKEN'}
  };

  axios.request(options).then(function (response) {
    console.log(response.data);
  }).catch(function (error) {
    console.error(error);
  });
  ```

  ```php PHP
  $curl = curl_init();

  curl_setopt_array($curl, [
    CURLOPT_URL => "https://{yourDomain}/api/v2/users-by-email?email=USER_EMAIL_ADDRESS&fields=user_metadata",
    CURLOPT_RETURNTRANSFER => true,
    CURLOPT_ENCODING => "",
    CURLOPT_MAXREDIRS => 10,
    CURLOPT_TIMEOUT => 30,
    CURLOPT_HTTP_VERSION => CURL_HTTP_VERSION_1_1,
    CURLOPT_CUSTOMREQUEST => "GET",
    CURLOPT_HTTPHEADER => [
      "authorization: Bearer YOUR_MGMT_API_ACCESS_TOKEN"
    ],

  $response = curl_exec($curl);
  $err = curl_error($curl);

  curl_close($curl);

  if ($err) {
    echo "cURL Error #:" . $err;
  } else {
    echo $response;
  }
  ```

  ```python Python

  conn = http.client.HTTPSConnection("")

  headers = { 'authorization': "Bearer YOUR_MGMT_API_ACCESS_TOKEN" }

  conn.request("GET", "/{yourDomain}/api/v2/users-by-email?email=USER_EMAIL_ADDRESS&fields=user_metadata", headers=headers)

  res = conn.getresponse()
  data = res.read()

  print(data.decode("utf-8"))
  ```

  ```ruby Ruby

  response = http.request(request)
  puts response.read_body
  ```
</AuthCodeGroup>

<div id="update-consent-information">
  ### Actualizar la información de consentimiento
</div>

Para actualizar el `user_metadata` de un usuario, usa el endpoint [**Update a User**](https://auth0.com/docs/api/management/v2#!/Users/patch_users_by_id).

La forma de estructurar tu solicitud depende de cómo hayas estructurado tus metadatos: como propiedades de raíz o como propiedades internas.

Si tus metadatos se almacenan como propiedades de raíz:

```json lines
{
  "consentGiven": true,
  "consentDetails": "some-url"
}
```

Si tus metadatos se almacenan en propiedades internas:

```json lines
{
  "consent": {
    "given": true,
    "text_details": "some-url"
  }
}
```

<div id="update-root-property">
  #### Actualizar propiedad raíz
</div>

Las actualizaciones de las propiedades de nivel raíz se fusionan, por lo que solo debes enviar el valor del campo que quieres actualizar. Por ejemplo, supongamos que queremos agregar una fecha de consentimiento y establecerla en `01/23/2018`.

<AuthCodeGroup>
  ```bash cURL
  curl --request PATCH \
    --url 'https://{yourDomain}/api/v2/users/USER_ID' \
    --header 'authorization: Bearer YOUR_MGMT_API_ACCESS_TOKEN' \
    --header 'content-type: application/json' \
    --data '{"user_metadata":{"consentDate":"01/24/2018"}}'
  ```

  ```csharp C#
  var client = new RestClient("https://{yourDomain}/api/v2/users/USER_ID");
  var request = new RestRequest(Method.PATCH);
  request.AddHeader("authorization", "Bearer YOUR_MGMT_API_ACCESS_TOKEN");
  request.AddHeader("content-type", "application/json");
  request.AddParameter("application/json", "{"user_metadata":{"consentDate":"01/24/2018"}}", ParameterType.RequestBody);
  IRestResponse response = client.Execute(request);
  ```

  ```go Go
  package main

  import (
  	"fmt"
  	"strings"
  	"net/http"
  	"io/ioutil"
  )

  func main() {

  	url := "https://{yourDomain}/api/v2/users/USER_ID"

  	payload := strings.NewReader("{"user_metadata":{"consentDate":"01/24/2018"}}")

  	req, _ := http.NewRequest("PATCH", url, payload)

  	req.Header.Add("authorization", "Bearer YOUR_MGMT_API_ACCESS_TOKEN")
  	req.Header.Add("content-type", "application/json")

  	res, _ := http.DefaultClient.Do(req)

  	defer res.Body.Close()
  	body, _ := ioutil.ReadAll(res.Body)

  	fmt.Println(res)
  	fmt.Println(string(body))

  }
  ```

  ```java Java
  HttpResponse response = Unirest.patch("https://{yourDomain}/api/v2/users/USER_ID")
    .header("authorization", "Bearer YOUR_MGMT_API_ACCESS_TOKEN")
    .header("content-type", "application/json")
    .body("{"user_metadata":{"consentDate":"01/24/2018"}}")
    .asString();
  ```

  ```javascript Node.JS

    method: 'PATCH',

    console.log(response.data);
  }).catch(function (error) {
    console.error(error);
  });
  ```

  ```php PHP

  ```

  ```python Python

      'authorization': "Bearer YOUR_MGMT_API_ACCESS_TOKEN",
      'content-type': "application/json"
      }

  conn.request("PATCH", "/{yourDomain}/api/v2/users/USER_ID", payload, headers)

  puts response.read_body
  ```
</AuthCodeGroup>

Esto añadirá una nueva propiedad al perfil del usuario, **user&#95;metadata.consentDate**, que contendrá la fecha en que el cliente dio su consentimiento. La respuesta será el perfil del usuario completo. Los metadatos actualizados se verán así:

```json lines
{
  "consentGiven": true,
  "consentDate": "01/23/2018",
  "consentDetails": "some-url"
}
```

  #### Actualizar una propiedad interna
</div>

<div id="export-consent-information">
  ### Exportar información sobre el consentimiento
</div>

Para exportar una lista de sus usuarios mediante la Management API, use el [**User Export** endpoint](/docs/es/manage-users/user-search/user-search-best-practices).

Este endpoint crea una tarea que exporta todos los usuarios asociados a una conexión. Necesitará el ID de la conexión. Para encontrar este ID, use el [**Get Connections** endpoint](https://auth0.com/docs/api/management/v2#!/Connections/get_connections) (puede establecer el parámetro **name** con el nombre de la conexión para recuperar solo esa).

Una vez que tenga el ID de la conexión y un token de acceso de la Management API, estará listo para comenzar a exportar usuarios. Para ver una solicitud y una respuesta de ejemplo, lea [Import and Export Users](/docs/es/manage-users/user-migration). Para saber cómo obtener un token de acceso de la Management API, lea [Management API Access Tokens](/docs/es/secure/tokens/access-tokens/management-api-access-tokens).

También debe:

* Determinar cómo desea hacer el seguimiento del consentimiento. Recomendamos incluir información no solo sobre la fecha en que el usuario dio su consentimiento, sino también sobre la versión de los términos y condiciones que aceptó. También recomendamos incluir un arreglo para almacenar información sobre los usuarios que retiran su permiso (recuerde que el usuario puede dar y retirar su consentimiento varias veces).
* Elegir dónde desea almacenar el consentimiento: en la base de datos de Auth0 o en otro lugar.

<div id="withdraw-consent">
  ## Retirar el consentimiento
</div>

El usuario debe tener la opción de retirar el consentimiento desde tu aplicación. Esta opción debe ser fácilmente accesible y fácil de identificar. Una vez que el usuario decida retirar su consentimiento, debes tomar medidas.

Primero, tienes que decidir cómo vas a gestionar la retirada del consentimiento: ¿eliminarás a los usuarios o los marcarás como eliminados?

<div id="flag-user-as-deleted">
  ### Marcar al usuario como eliminado
</div>

Si no quieres eliminar al usuario, marca su perfil como eliminado mediante el endpoint de [**app&#95;metadata**](https://auth0.com/docs/api/management/v2#!/Users/patch_users_by_id). Luego, agrega código para hacer que falle el proceso de autenticación de cualquier usuario cuyo perfil esté marcado de ese modo.

Esto te permite conservar un registro de los usuarios eliminados para usarlos en el futuro.

<div id="disable-login-for-flagged-users">
  #### Deshabilita el inicio de sesión para los usuarios marcados
</div>

A continuación, debes deshabilitar el inicio de sesión de los usuarios marcados como eliminados. Para ello, agregarás una rule (un fragmento de JavaScript que se ejecuta como parte del pipeline de autenticación).

1. Ve a [Auth0 Dashboard &gt; pipeline de autenticación &gt; Rules](https://manage.auth0.com/#/rules) y crea una rule.
2. Copia el siguiente script:

   ```javascript lines
   function (user, context, callback) {
     user.app_metadata = user.app_metadata || {};
     if (user.app_metadata.deleted){
     	return callback(new UnauthorizedError('Access denied (deleted user)'));
     }
     callback(null, user, context);
   }
   ```

   El script hace lo siguiente:

   1. Comprueba el valor de la propiedad de metadato **deleted** (`user.app_metadata.deleted`).
   2. Devuelve un error `Access denied (deleted user)` a tu aplicación si `user.app_metadata.deleted = true`
3. Asigna un nombre a tu rule y guarda los cambios.

También debes:

* Asegurarte de que la parte de revocación del consentimiento sea lo suficientemente granular.
* Configurar en la aplicación el área donde los clientes retirarán el consentimiento.
