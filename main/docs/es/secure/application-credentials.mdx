Las aplicaciones confidenciales, a diferencia de las aplicaciones públicas, pueden almacenar credenciales de forma segura. Cuando las aplicaciones confidenciales solicitan acceso o <Tooltip tip="Token de ID: credencial destinada al propio cliente, en lugar de para acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=ID+tokens">token de ID</Tooltip> en el [endpoint de token](https://auth0.com/docs/api/authentication#get-token), la aplicación debe autenticarse con el <Tooltip tip="Token de ID: credencial destinada al propio cliente, en lugar de para acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=authorization+server">servidor de autorización</Tooltip>. Durante esta solicitud de tokens, la aplicación proporciona sus propias credenciales. Además, las credenciales de la aplicación también pueden proporcionar protección de autenticidad e integridad para los parámetros de la solicitud enviados al endpoint [`/authorize`](https://auth0.com/docs/api/authentication#authorize-application).

Para obtener más información sobre las aplicaciones confidenciales y las aplicaciones públicas, lea [Aplicaciones confidenciales y públicas](/docs/es/get-started/applications/confidential-and-public-applications).

<div id="application-authentication-methods">
  ## Métodos de autenticación de aplicaciones
</div>

Para obtener tokens de Auth0, tu aplicación debe autenticarse a través de la [Authentication API](https://auth0.com/docs/api/authentication). Auth0 admite las siguientes formas de autenticación para tu aplicación:

* **<Tooltip tip="Secreto de cliente: Secreto que utiliza un cliente (aplicación) para autenticarse con el servidor de autorización; solo deben conocerlo el cliente y el servidor de autorización, y debe ser lo suficientemente aleatorio como para que no se pueda adivinar." cta="Ver glosario" href="/docs/es/glossary?term=Client+Secret">Secreto de cliente</Tooltip>:** Un método de autenticación simétrico. En la autenticación con secreto de cliente, utilizas el secreto de cliente que Auth0 generó cuando creaste la aplicación.
* **Clave privada <Tooltip tip="JSON Web Token (JWT): Formato estándar de token de ID (y, a menudo, de token de acceso) que se utiliza para representar claims de forma segura entre dos partes." cta="Ver glosario" href="/docs/es/glossary?term=JWT">JWT</Tooltip>:** Un método de autenticación asimétrico. En clave privada JWT, generas un par de claves, pública y privada, para usarlas como credenciales. Proporcionas la clave pública y almacenas de forma segura la clave privada en tus propios sistemas sin compartirla con Auth0.
* **mTLS para <Tooltip tip="OAuth 2.0: Marco de autorización que define protocolos y flujos de trabajo de autorización." cta="Ver glosario" href="/docs/es/glossary?term=OAuth">OAuth</Tooltip>:** Un método de autenticación asimétrico. En mTLS para OAuth, registras un certificado de cliente X.509 estándar en Auth0. Luego, utilizas la clave privada correspondiente para establecer de forma segura el túnel mTLS y enviar solicitudes a los endpoints de tu tenant de Auth0.

<div id="client-secret-authentication">
  ### Autenticación con secreto de cliente
</div>

La autenticación con secreto de cliente es un método de autenticación simétrica incluido en la [especificación OAuth 2.0](https://www.rfc-editor.org/rfc/rfc6749#section-2.3). La autenticación con secreto de cliente es el método de autenticación predeterminado en Auth0.

Este método de autenticación es compatible con todas las aplicaciones y herramientas existentes. El secreto de cliente es un valor de alta entropía que Auth0 genera al crear una aplicación y que conocen tanto tu aplicación como Auth0. Tu aplicación se autentica incluyendo el secreto de cliente en la solicitud al servidor de autorización.

El uso del secreto de cliente como credencial conlleva algunos riesgos de seguridad, especialmente en escenarios con requisitos de seguridad más exigentes:

* El secreto que usa la aplicación se comparte con Auth0.
* El secreto se envía por la red y podría ser interceptado en caso de un ataque de intermediario.

<Warning>
  Para mejorar tu postura de seguridad, te recomendamos usar el método de autenticación con clave privada JWT.
</Warning>

Una aplicación solo puede tener un secreto de cliente. No es posible rotar el secreto mientras actualizas tu implementación con el nuevo secreto. Para obtener más información, consulta [Rotate Client Secrets](/docs/es/get-started/applications/rotate-client-secret).

<div id="private-key-jwt-authentication">
  ### Autenticación con clave privada JWT
</div>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  La clave privada JWT está disponible para clientes con el plan Enterprise. Para actualizar, contacta a [Auth0 pricing](https://auth0.com/pricing/).
</Callout>

La autenticación con clave privada JWT es un método de autenticación asimétrico que se basa en pares de claves públicas y privadas. Para obtener más información, consulta [JSON Web Token (JWT) Profile for OAuth 2.0 Client Authentication and Authorization Grants](http://tools.ietf.org/html/draft-ietf-oauth-jwt-bearer).

Puedes usar el <Tooltip tip="Auth0 Dashboard: El producto principal de Auth0 para configurar tus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip> o la <Tooltip tip="Auth0 Dashboard: El producto principal de Auth0 para configurar tus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip> de Auth0 para configurar un tenant para usar clave privada JWT. Para obtener más información, consulta [Configurar la autenticación con clave privada JWT.](/docs/es/get-started/applications/configure-private-key-jwt)

En la clave privada JWT, una solicitud al servidor de autorización consta de dos pasos principales:

1. Configura las claves pública y privada:

   1. [Genera un par de claves](/docs/es/secure/application-credentials/generate-rsa-key-pair) (una clave pública y una clave privada).
   2. Registra la clave privada en la aplicación que realiza la solicitud de autenticación y registra la clave pública en el proveedor de identidad (IdP).
2. Crea aserciones para solicitudes al servidor de autorización:

   1. Crea una nueva aserción con claims específicos en formato JWT y fírmala con la clave privada. Incluye esta aserción como parte de la solicitud al IdP.
   2. El IdP valida la aserción con la clave pública.

Para configurar la clave privada JWT para Auth0, consulta [Configurar la autenticación con clave privada JWT](/docs/es/get-started/applications/configure-private-key-jwt). Para obtener más información sobre cómo crear una aserción para clave privada JWT, consulta [autenticarse con clave privada JWT](/docs/es/get-started/authentication-and-authorization-flow/authenticate-with-private-key-jwt).

Estas son algunas de las ventajas de seguridad de usar clave privada JWT:

* La clave privada no se transmite por la red, lo que reduce el riesgo de exposición de las credenciales de tu aplicación. Los <Tooltip tip="Identity Provider (IdP): Servicio que almacena y gestiona identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=Identity+providers">proveedores de identidad</Tooltip> como Auth0 no conocen la clave privada, y solo las aplicaciones que tienen acceso a ella pueden crear solicitudes de autenticación.
* Las aserciones firmadas tienen un tiempo de expiración corto, lo que limita la ventana de oportunidad para ataques de repetición.

<div id="mtls-for-oauth">
  ### mTLS for OAuth
</div>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Para usar las funcionalidades de Identidad altamente regulada, debes tener un plan Enterprise con el add-on Highly Regulated Identity. Consulta [Auth0 Pricing](https://auth0.com/pricing/) para obtener más información.
</Callout>

[mTLS for OAuth](https://www.rfc-editor.org/rfc/rfc8705) autentica las solicitudes al servidor de autorización mediante TLS mutuo basado en certificados autofirmados o en una infraestructura de clave pública (PKI). Lee [Autenticar con mTLS](/docs/es/get-started/authentication-and-authorization-flow/authenticate-with-mtls) para obtener más información sobre cómo funciona la autenticación mTLS en Auth0.

La funcionalidad mTLS for OAuth de Auth0 está dirigida inicialmente a clientes de sectores altamente regulados, como finanzas y salud, que probablemente ya cuenten con implementaciones de mTLS. Para facilitar su adopción, la funcionalidad de mTLS se basa en [dominios personalizados](/docs/es/customize/custom-domains) y aprovecha la infraestructura de mTLS existente del cliente para realizar el aprovisionamiento y la verificación de certificados. Para obtener más información sobre cómo autenticar con mTLS y configurar tu red perimetral, lee [Autenticar con mTLS](/docs/es/get-started/authentication-and-authorization-flow/authenticate-with-mtls) y [Configura tu customer edge](/docs/es/get-started/applications/configure-mtls/set-up-the-customer-edge).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Puedes configurar [alias de endpoint mTLS](/docs/es/get-started/applications/configure-mtls/configure-mtls-for-a-tenant#enable-mtls-endpoint-aliases) para usar un subdominio específico con mTLS for OAuth.
</Callout>

Para aprender a configurar mTLS, lee [Configurar la autenticación mTLS](/docs/es/get-started/applications/configure-mtls). Una vez que hayas configurado tu red perimetral y mTLS, tu aplicación debe establecer el túnel mTLS para enviar solicitudes a Auth0, como se explica en [Llamar al servidor de autorización](/docs/es/get-started/authentication-and-authorization-flow/authenticate-with-mtls#call-the-authorization-server).

En mTLS, la clave privada del certificado del cliente no se transmite por la red, lo que reduce el riesgo de exponer las credenciales de tu aplicación. Los proveedores de identidad como Auth0 no tienen acceso a la clave privada. Solo las aplicaciones que tienen acceso a la clave privada pueden autenticarse.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  mTLS también admite Sender Constraining o Token Binding para proteger los tokens de acceso frente a atacantes. Para obtener más información, lee [Configurar Sender Constraining](/docs/es/secure/sender-constraining/configure-sender-constraining). Token Binding no requiere credenciales de aplicación registradas previamente, como un certificado de cliente, para usarse con mTLS.
</Callout>

<div id="jwt-secured-authorization-request-jar">
  ### Solicitud de autorización protegida con JWT (JAR)
</div>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Para usar las funcionalidades de Identidad altamente regulada, debe tener un plan Enterprise con el add-on Highly Regulated Identity. Consulte [Auth0 Pricing](https://auth0.com/pricing/) para obtener más información.
</Callout>

[Solicitud de autorización protegida con JWT (JAR)](https://datatracker.ietf.org/doc/rfc9101/) es una extensión del protocolo OAuth2 que refuerza la seguridad de las solicitudes de autorización. Para ello, utiliza un parámetro de solicitud de JSON Web Token (JWT) para proteger la integridad y la confidencialidad de los parámetros de la solicitud de autorización.

Puede usar la [Auth0 Management API](https://auth0.com/docs/api/management/v2) para configurar JAR en su aplicación. La implementación de Auth0 para JAR usa criptografía asimétrica: usted registra la clave pública y almacena de forma segura la clave privada en su entorno. Para obtener más información, consulte [Configure JWT-Secured Authorization Requests](/docs/es/get-started/applications/configure-jar).

Al usar JAR, el cliente crea un JWT que incluye los parámetros de la solicitud de autorización, lo firma con su clave privada y lo envía al servidor de autorización. Luego, el servidor de autorización verifica la firma con la clave pública del cliente y, si la firma es válida, extrae del JWT los parámetros de la solicitud de autorización y procesa la solicitud como de costumbre. Para obtener más información sobre cómo usar JAR, consulte [Authorization Code Flow with JWT-Secured Authorization Requests (JAR)](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow/authorization-code-flow-with-jar).

<div id="key-and-certificate-registration">
  ### Registro de claves y certificados
</div>

<Warning>
  Debe generar un par de claves distinto para cada tipo de uso de credenciales. Por ejemplo, no reutilice los mismos pares de claves para JAR y para la autenticación con clave privada JWT.
</Warning>

Puede registrar dos claves públicas para una aplicación al mismo tiempo. Auth0 realiza la verificación con la clave correcta y le permite rotarlas sin interrupciones. Una vez que la clave anterior se elimine o desactive, todas las solicitudes firmadas con la clave privada correspondiente se invalidarán.

**Nota:** Auth0 admite los siguientes algoritmos para la autenticación de aplicaciones y la firma de solicitudes de autorización: RS256, RS384 y PS256. Asegúrese de proporcionar las claves adecuadas para cada uno. Para obtener más información, consulte [Configurar la autenticación con clave privada JWT](/docs/es/get-started/applications/configure-private-key-jwt) y [Configurar solicitudes de autorización protegidas con JWT](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow/authorization-code-flow-with-jar).

Del mismo modo, para los certificados de cliente mTLS, puede registrar dos certificados X.509 de cliente (autofirmados o con el Subject DN del certificado de la CA) al mismo tiempo para una aplicación. Auth0 verifica ambos certificados de cliente, lo que le permite rotar certificados sin interrupciones.
