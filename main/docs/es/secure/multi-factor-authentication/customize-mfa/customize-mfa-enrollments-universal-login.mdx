title: Personaliza las inscripciones de MFA para Universal Login
---

Auth0 admite una variedad de factores para proteger el acceso de los usuarios con [autenticación multifactor (MFA](/docs/es/secure/multi-factor-authentication/multi-factor-authentication-factors)). Con las Actions de `Post Login`, puedes personalizar tus flujos de <Tooltip tip="Autenticación multifactor (MFA): proceso de autenticación de usuario que utiliza un factor además de nombre de usuario y contraseña, como un código por SMS." cta="Ver glosario" href="/docs/es/glossary?term=MFA">MFA</Tooltip> para pedir a los usuarios que registren factores específicos. Después de que un usuario registre un factor, podrá usarlo como método secundario de autenticación en futuros inicios de sesión.

También puedes usar información contextual para personalizar aún más tus flujos de inscripción de MFA. Por ejemplo, puedes pedir a los usuarios que se inscriban en SMS para una aplicación, mientras que para otra aplicación puedes pedirles que se inscriban en notificaciones push o WebAuthN.

Esta funcionalidad te permite personalizar tus flujos de inscripción de MFA. Si quieres personalizar los flujos de MFA para usuarios que ya están inscritos, consulta [Customize MFA Selection for Universal Login](/docs/es/secure/multi-factor-authentication/customize-mfa/customize-mfa-selection-universal-login).

<div id="how-it-works">
  ## Cómo funciona
</div>

Puedes usar [Actions](/docs/es/customize/actions) para personalizar tus flujos de inscripción de MFA. En concreto, puedes modificar el trigger `Post Login` del [Login Flow](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger) con los siguientes métodos de Authentication API:

* `enrollWith`: Especifica el factor predeterminado que se muestra a los usuarios durante la inscripción. De forma opcional, puedes proporcionar una lista alternativa de factores para que los usuarios elijan. Si se proporciona, se muestra un enlace Try Another Method en el prompt de inscripción.
* `enrollWithAny`: Especifica un conjunto de factores entre los que los usuarios pueden elegir durante la inscripción. De forma predeterminada, este método muestra un prompt de selección que permite a los usuarios elegir el factor que prefieran. En algunos casos, la experiencia de usuario puede variar:

  * Si se han especificado dos o más factores, se muestra al usuario el prompt de selección.
  * Si el usuario ya se ha inscrito en todos los factores especificados excepto uno, se omite el prompt de selección y se le pide que se inscriba en el factor restante.
  * Si el usuario ya se ha inscrito en todos los factores especificados, el comando falla y la secuencia de login continúa.

Puedes usar una combinación de estos métodos para personalizar tus flujos de inscripción de MFA. También puedes incorporar metadatos del usuario, como roles o la fecha del último login, para crear experiencias más personalizadas.

Los flujos de inscripción personalizados admiten los siguientes factores:

* `otp`
* `recovery-code`
* `push-notification`
* `phone`

  * `preferredMethod: voice`
  * `preferredMethod: sms`
  * `preferredMethod: both`
* `webauthn-platform`
* `webauthn-roaming`

Después de que un usuario se inscribe en un factor, su valor se añade a `enrolledFactors`. Esta propiedad representa la lista de factores activos asociados a su cuenta de usuario.

El arreglo `event.authentication.methods` incluye un campo `type` cuando el nombre del método se establece en `mfa`. Este campo contiene valores de factor (string) que coinciden con los que usa el campo `type` de `enrolledFactors`.

Cuando se produce una inscripción de MFA, `methods` contiene el objeto `name:mfa` con `type` establecido en el factor usado para ese evento. `methods` y `enrolledFactors` solo se actualizan cuando una Action comienza por primera vez. Puedes acceder a los resultados de un evento de inscripción en la siguiente Action del flujo.

Para obtener más información, consulta los siguientes recursos:

* [Actions Triggers: Post Login - Event Object](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger/post-login-event-object)
* [Actions Triggers: Post Login - objeto de API](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger/post-login-api-object)

<div id="sequenced-and-contextual-flows">
  ### Flujos secuenciados y contextuales
</div>

Con los comandos `enrollWith` o `enrollWithAny`, puedes usar información contextual para determinar la mejor inscripción, o serie de inscripciones, que presentar a los usuarios.

* El comando `enrollWith` admite un factor inicial o predeterminado y una lista de alternativas. Los usuarios solo pueden inscribirse en un factor por comando.
* El comando `enrollWithAny` admite una lista de factores. El orden especificado de los factores determina cómo se muestra la lista a los usuarios. Los usuarios solo pueden inscribirse en un factor por comando.

Con estos comandos, puedes aprovechar lo siguiente:

* **Flujos secuenciados**: Inscribe a los usuarios en una serie de factores en un orden específico.
* **Flujos contextuales**: Determina qué factor mostrar al usuario en función de los metadatos o de los comandos anteriores en el flujo.

Para ilustrar mejor estos flujos, considera el siguiente ejemplo:

```javascript lines expandable
// Action 1

exports.onExecutePostLogin = async (event, api) => {
  if (event.user.enrolledFactors.length) {
    // ya registrado, desafiar
    api.authentication.challengeWithAny(event.user.enrolledFactors.map(m => ({type: m.type})));
    if (event.user.app_metadata.isAdmin &&
        !event.user.enrolledFactors.some(m => m.type === 'webauthn-roaming')) {
          // si es administrador y no tiene una clave de seguridad, es decir, se usó un factor diferente, registrar ahora
          api.authentication.enrollWith({type: 'webauthn-roaming'})
        }
  }
  else {
    // no registrado; elegir un factor para registrar ahora
    api.authentication.enrollWithAny([{type: 'webauthn-roaming'}, {type: 'otp'}]);
    if (event.user.app_metadata.isAdmin) {
      // un factor más para administradores
      api.authentication.enrollWithAny([{type: 'webauthn-roaming'}, {type: 'otp'}]);
    }
  }
};

// Action 2

exports.onExecutePostLogin = async (event, api) => {
  function performed(type) {
    return event.authentication.methods.some(m => m.name === 'mfa' &&
           m.type === type &&
           Date.now() - new Date(m.timestamp).getTime() < 5000)
  }
  if (event.user.app_metadata.isAdmin) {
      // forzar el uso de ambos factores desafiando el que aún no se ha utilizado
      if (!performed('webauthn-roaming')) {
        api.authentication.challengeWith({type: 'webauthn-roaming'})
      }
      else if (!performed('otp')) {
        api.authentication.challengeWith({type: 'otp'})
      }
  }
};
```

Estas dos Actions se combinan para crear un escenario en el que los usuarios **sin** el rol de administrador deben inscribirse con una contraseña de un solo uso (OTP) o una clave de seguridad. Por el contrario, los usuarios **con** el rol de administrador deben inscribirse en ambos factores.

La Action 1 revisa `app_metadata` para determinar si el usuario es administrador y luego le solicita que se inscriba en factores específicos. Si un usuario administrador solo se ha inscrito en OTP, primero se le presenta un desafío con OTP para completar la autenticación. Después, se le solicita que se inscriba con claves de seguridad (`webauthn-roaming`).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Para mantener seguras las cuentas de usuario, los usuarios deben completar un desafío MFA usando uno de sus `enrolledFactors` ya inscritos antes de poder inscribirse en factores adicionales. Esta condición garantiza que puedas implementar de forma segura una política personalizada de inscripción en MFA después de que una aplicación con distintos factores y configuraciones ya se haya utilizado.
</Callout>

El flujo se pausa después de que se ejecuta la Action 1, y tanto `event.user.enrolledFactors` como `event.authentication.methods` se actualizarán cuando se ejecute la Action 2. Esto permite que el código de la Action tome decisiones basadas en datos de usuario reales cuando se ofrece a los usuarios la opción de completar un desafío o inscribirse en distintos factores.

**Nota**: Este método de ejecución de Actions solo se aplica a aquellas que contienen los comandos `enrollWith` o `enrollWithAny`. Las Actions que cumplen otros propósitos no se ven afectadas.

<div id="before-you-begin">
  ## Antes de comenzar
</div>

Antes de poder personalizar tus flujos de MFA, debes configurar MFA en tu tenant y habilitar la opción Customize MFA Factors using Actions. Puedes habilitar uno o varios factores y definir tus políticas de MFA en el <Tooltip tip="Auth0 Dashboard: el principal producto de Auth0 para configurar tus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip>, en [Security &gt; Multi-factor Auth](https://manage.auth0.com/#/security/mfa).

* Para obtener más información sobre el proceso de configuración, consulta [Enable Multi-Factor Authentication](/docs/es/secure/multi-factor-authentication/enable-mfa).
* Para obtener información sobre cómo configurar factores específicos, consulta [Multi-factor Authentication Factors](/docs/es/secure/multi-factor-authentication/multi-factor-authentication-factors).

Para personalizar tus flujos, debes habilitar el interruptor Customize MFA Factors using Actions en la sección Additional Settings. Tus flujos personalizados no funcionarán correctamente si esta configuración no está habilitada.

<Frame>![Auth0 Dashboard > Security > Multi-factor Auth > Additional Settings](/docs/images/cdy7uua7fh8z/2hv0ELTkkka3t230SXfxw/46def5395652b2451cfc9e0ad01a371a/MFA_actions.png)</Frame>

**Nota**: Las Actions con los comandos `enrollWith` o `enrollWithAny` anulan cualquier política o regla existentes que habiliten o deshabiliten MFA en un tenant.

<div id="customize-mfa-enrollment-flows">
  ## Personaliza los flujos de inscripción en MFA
</div>

Después de configurar MFA para tu tenant, puedes crear Actions de `Post Login` para personalizar tus flujos de inscripción en MFA.

<Warning>
  Las Actions (o series de Actions) en un tenant solo pueden ejecutar **cuatro** de los siguientes comandos por flujo de usuario:

  * `enrollWith`
  * `enrollWithAny`
  * `challengeWith`
  * `challengeWithAny`

  Si se supera este límite (es decir, si se intenta ejecutar un quinto comando de este tipo), se producirá un error de autenticación.
</Warning>

<div id="create-your-post-login-action">
  ### Crea tu Action de Post Login
</div>

Puedes crear Actions desde el Auth0 Dashboard:

1. Ve a [Actions &gt; Flows](https://manage.auth0.com/#/actions/flows) y selecciona **Login**.
2. En el panel Add Action, selecciona el icono del **signo más (+)** y elige **Build from scratch**.
3. En la ventana emergente Create Action:

   * Ingresa un nombre para tu Action.
   * Selecciona **Login / Post-Login** como trigger.
   * Usa **Node 22 (Recommended)** como runtime.
4. Revisa la ventana emergente para asegurarte de que todo sea correcto. Luego, selecciona **Create**.
5. En el editor de código, agrega tu código personalizado al comando `onPostExecute`.
6. Cuando tu comando esté listo, selecciona **Deploy**.
7. Selecciona **Add to Flow** en la notificación de implementación exitosa.

   * **Nota**: Si la notificación se cierra, elige **Back to Flow** encima del editor de código.
8. Arrastra y suelta tu nuevo comando desde el panel Add Action a tu login flow. Luego, selecciona **Apply**.

Para hacer cambios adicionales después de guardar, ve a [Actions &gt; Library &gt; Custom](https://manage.auth0.com/#/actions/library) y selecciona tu Action. Después, puedes actualizar y volver a desplegar tu código según sea necesario.

<div id="test-your-post-login-action">
  ### Prueba tu Action de Post Login
</div>

Para asegurarte de que tus comandos funcionen correctamente, puedes probar tu Action desde el Auth0 Dashboard:

1. Ve a [Authentication &gt; Authentication Profile](https://manage.auth0.com/#/authentication-profiles).
2. Selecciona **Try** para abrir un prompt de inicio de sesión de ejemplo en una pestaña nueva.
3. Introduce tus credenciales y prueba tu nuevo flujo de MFA.

Si el flujo se completa correctamente, se mostrará una pantalla de confirmación. Si tienes algún problema, puedes actualizar tu código yendo a [Actions &gt; Library &gt; Custom](https://manage.auth0.com/#/actions/library) en tu Auth0 Dashboard.

<div id="troubleshooting-checklist">
  ### Lista de comprobación para la resolución de problemas
</div>

La siguiente lista de comprobación ofrece sugerencias adicionales para identificar y resolver problemas comunes con flujos de MFA personalizados.

1. El interruptor **Customize MFA factors with Actions** debe estar habilitado.

   * Ve a [Auth0 Dashboard &gt; Security &gt; Multi-factor Auth](https://manage.auth0.com/#/security/mfa) y asegúrate de que el interruptor de la sección Additional Settings esté habilitado.
2. Los factores a los que hacen referencia tus Actions deben estar habilitados en tu tenant.

   * **Revisa tu código**: Ve a [Auth0 Dashboard &gt; Actions &gt; Library &gt; Custom](https://manage.auth0.com/#/actions/library) y revisa tu código de Actions. Asegúrate de que todos los factores a los que se hace referencia sean aplicables a tus casos de uso.
   * **Revisa tus factores**: Ve a [Auth0 Dashboard &gt; Security &gt; Multi-factor Auth](https://manage.auth0.com/#/security/mfa) y asegúrate de que todos los factores a los que se hace referencia en tus Actions estén habilitados.
3. Asegúrate de que tus Actions se hayan desplegado y guardado en tu pipeline.

   1. Ve a [Auth0 Dashboard &gt; Actions &gt; Library &gt; Custom](https://manage.auth0.com/#/actions/library). Busca tu Action en la lista y asegúrate de que su estado sea **Deployed**. Si aparece un estado diferente, entra en tu Action, revisa tu código y haz clic en **Deploy** en la esquina superior derecha.
   2. Ve a [Auth0 Dashboard &gt; Actions &gt; Library &gt; Flows](https://manage.auth0.com/#/actions/flows) y selecciona **Login**. Asegúrate de que tu Action aparezca en el flujo. Si no es así, abre la **pestaña Custom** del panel Add Action y arrastra y suelta tu Action en tu Login flow. Luego, selecciona **Apply**.
4. Asegúrate de haber actualizado a la versión más reciente de las Actions `Post Login`.

   * Ve a [Auth0 Dashboard &gt; Actions &gt; Library &gt; Custom](https://manage.auth0.com/#/actions/library) y selecciona tu Action. Si tu Action está desactualizada, verás un banner amarillo que te indicará que debes actualizarla. Si aparece el banner, selecciona **Update**.
   * También puedes especificar la versión más reciente de las Actions `Post Login` para el despliegue cuando uses Deploy CLI. Para obtener más información, consulta [Configurar Deploy CLI](/docs/es/deploy-monitor/deploy-cli-tool/configure-the-deploy-cli).
