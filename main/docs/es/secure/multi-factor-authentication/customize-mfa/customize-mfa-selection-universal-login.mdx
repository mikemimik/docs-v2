title: Personaliza la selección de MFA en Universal Login
---

Auth0 admite diversos factores para proteger el acceso de los usuarios con [autenticación multifactor (MFA)](/docs/es/secure/multi-factor-authentication/multi-factor-authentication-factors). Con las Actions de `Post Login`, puedes personalizar tus flujos de <Tooltip tip="Autenticación multifactor (MFA): proceso de autenticación de usuario que usa un factor además de nombre de usuario y contraseña, como un código por SMS." cta="Ver glosario" href="/docs/es/glossary?term=MFA">MFA</Tooltip> para solicitar a los usuarios un factor específico o una secuencia de factores. También puedes usar información contextual sobre los usuarios y sus organizaciones para crear experiencias más personalizadas. Por ejemplo, puedes personalizar tus flujos para solicitar a los usuarios factores específicos según su pertenencia a determinadas Organizations o los roles que tengan asignados.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  * Esta funcionalidad te permite personalizar tus flujos de selección de MFA. Si quieres personalizar tus flujos de inscripción en MFA, consulta [Personaliza la inscripción en MFA en Universal Login](/docs/es/secure/multi-factor-authentication/customize-mfa/customize-mfa-enrollments-universal-login).
  * Solo puedes personalizar flujos de MFA con Actions si usas [Universal Login](/docs/es/authenticate/login/auth0-universal-login/universal-login-vs-classic-login/universal-experience).
  * Algunos de los métodos descritos solo pueden usarse en Universal Login y no en Classic Login. Para obtener más información, consulta [Personaliza MFA para Classic Login](/docs/es/secure/multi-factor-authentication/customize-mfa/customize-mfa-classic-login).
</Callout>

<div id="how-it-works">
  ## Cómo funciona
</div>

Puedes usar [Actions](/docs/es/customize/actions) para personalizar tus flujos de MFA. En concreto, puedes modificar el trigger `Post Login` del [flujo de Login](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger) con los siguientes métodos de Authentication API:

* [`challengeWith`](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger/post-login-api-object#api-authentication-challengewith-factor-options): Especifica el factor o los factores que los usuarios deben usar para autenticarse, como una contraseña de un solo uso (OTP). Este método presenta un desafío predeterminado a los usuarios y, de forma opcional, puede darles acceso a un selector de factores que les permite elegir un método de autenticación diferente.
* [`challengeWithAny`](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger/post-login-api-object#api-authentication-challengewithany-factors): Define un grupo de factores entre los que los usuarios pueden elegir al autenticarse, como correo electrónico y OTP. De forma predeterminada, este método muestra un selector de factores a los usuarios en lugar de un desafío específico, de acuerdo con las siguientes condiciones:

  * Si se especifican dos o más factores, se muestra un selector de factores al usuario.
  * Si el usuario solo tiene configurado uno de los factores especificados (o solo se proporciona un factor), se omite el selector de factores.
  * Si el usuario no tiene configurado ninguno de los factores especificados, el comando falla.

Puedes usar una combinación de estos métodos para adaptar tus flujos de MFA según sea necesario. También puedes incorporar metadatos del usuario, como roles o factores usados anteriormente, en estos métodos para crear flujos más personalizados.

Al elegir desafíos MFA para tus comandos, puedes usar los factores que se indican a continuación o el valor `enrolledFactors`. `enrolledFactors` representa la lista de factores activos asociados a la cuenta de un usuario.

* `otp`
* `email`
* `push-notification`

  * `otpFallback`
* `phone`

  * `preferredMethod: voice`
  * `preferredMethod: sms`
  * `preferredMethod: both`
* `webauthn-platform`
* `webauthn-roaming`

El arreglo [`event.authentication.methods`](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger/post-login-event-object#multi-factor-authentication) incluye un campo `type` cuando el nombre del método se establece en `mfa`. `type` es una cadena que contiene valores de factores que coinciden con los usados por el campo `type` de `enrolledFactors` (indicados arriba). Cuando se realiza un desafío MFA, `methods` contiene un objeto `name:mfa` con `type` establecido en el factor usado para ese desafío. `methods` solo se actualiza cuando comienza una Action. Para ver los resultados de un desafío, debes acceder a `methods` en la siguiente Action del flujo.

Para obtener más información, revisa los siguientes recursos:

* [Actions Triggers: Post Login - Objeto Event](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger/post-login-event-object)
* [Actions Triggers: Post Login - objeto de API](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger/post-login-api-object)

<div id="sequenced-and-contextual-flows">
  ### Flujos secuenciados y contextuales
</div>

Con los comandos `challengeWith` o `challengeWithAny`, puedes usar información contextual para determinar cuál es el mejor desafío o la mejor secuencia de desafíos para presentar a los usuarios. En concreto, puedes aprovechar lo siguiente:

* **Flujos secuenciados**: Presenta a los usuarios una serie de factores distintos en un orden específico.
* **Flujos contextuales**: Determina con qué factor desafiar al usuario a continuación en función de los desafíos anteriores del flujo.

Para ilustrar mejor estos flujos, considera el siguiente ejemplo:

```javascript lines
// ACTION 1 

exports.onExecutePostLogin = async (event, api) => {

   api.authentication.challengeWithAny([{ type: 'phone'}, { type: 'push-notification' }]);

} 

// ============================================ 

// ACTION 2 

// Decidir según lo que hizo el usuario en la Action anterior 

exports.onExecutePostLogin = async (event, api) => { 

    if(event.authentication.methods.find(m => m.type === 'phone') && event.authorization?.roles.includes('admin')) { 

        api.authentication.challengeWith({ type: 'push-notification' }); 

    }

}
```

En este escenario, primero se presenta al usuario un desafío por SMS mediante el comando `challengeWithAny` en Action 1. Luego, Action 2 presenta al usuario una notificación push como desafío porque tiene el rol de usuario Admin y, además, completó el desafío por SMS.

En este flujo, puedes decidir con qué factor presentar el desafío al usuario por lo siguiente:

1. El flujo se pausa después de ejecutar Action 1.
2. El usuario completa el flujo de MFA solicitado por Action 1.
3. `event.authentication.methods.type` en Action 2 se completa con información del desafío MFA anterior.
4. El flujo se reanuda para ejecutar Action 2 usando la información contextual de Action 1.

Aunque este ejemplo ofrece una experiencia similar a usar redirecciones en tus Actions, los comandos `challengeWith` y `challengeWithAny` ofrecen las siguientes ventajas específicas:

* Los Flows se pausan después de cada comando, lo que te permite recopilar información del usuario que puede usarse en Actions posteriores. En cambio, las redirecciones solo se producen una vez, como comando final de un flujo.
* MFA se activa después de ejecutar cada Action que contiene comandos `challengeWith` o `challengeWithAny`. Con las redirecciones, MFA se ejecuta como la Action final del pipeline.

**Nota**: Este método de ejecución de Actions solo se aplica a aquellas que contienen comandos `challengeWith` o `challengeWithAny`. Las Actions que cumplen otros propósitos no se ven afectadas.

<div id="before-you-begin">
  ## Antes de empezar
</div>

Antes de poder personalizar sus flujos de MFA, primero debe habilitar MFA en su tenant y pedir a sus usuarios que registren los factores adecuados.

<div id="prepare-your-tenant">
  ### Prepara tu tenant
</div>

Para empezar, configura MFA en tu tenant y habilita la opción Customize MFA Factors using Actions. Puedes configurar uno o más factores y definir tus políticas de MFA en el <Tooltip tip="Auth0 Dashboard: el producto principal de Auth0 para configurar tus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip>, en [Security &gt; Multifactor Auth](https://manage.auth0.com/#/security/mfa).

* Para obtener más información sobre el proceso de configuración, consulta [Enable Multi-Factor Authentication](/docs/es/secure/multi-factor-authentication/enable-mfa).
* Para obtener información sobre cómo configurar factores específicos, consulta [Multi-factor Authentication Factors](/docs/es/secure/multi-factor-authentication/multi-factor-authentication-factors).

Para personalizar tus flujos, **debes habilitar** el interruptor Customize MFA Factors using Actions en la sección Additional Settings. Tus flujos personalizados no funcionarán correctamente si esta opción no está habilitada.

<Frame>![Auth0 Dashboard > Security > Multi-factor Auth > Additional Settings](/docs/images/cdy7uua7fh8z/2hv0ELTkkka3t230SXfxw/46def5395652b2451cfc9e0ad01a371a/MFA_actions.png)</Frame>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  * Las Actions que invocan los comandos `challengeWith` y `challengeWithAny` reemplazan cualquier desafío habilitado mediante [api.multifactor.enable](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger/post-login-api-object). También tienen prioridad sobre la configuración de MFA disponible en **Define Policies**.
  * Para garantizar que los usuarios completen MFA al intentar acceder a tu aplicación, establece la opción **Require Multi-factor Auth** en **Use Adaptive MFA** o **Always**. Si tu código de Actions no se ejecuta, esta opción sirve como respaldo y evita que los usuarios omitan MFA.
  * Si deseas usar evaluaciones de riesgo en tus comandos, habilita el interruptor **Adaptive MFA Risk Assessment** y usa [event.authentication.riskAssessment](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger/post-login-api-object) en tu código de Actions de `Post Login`.
</Callout>

<div id="enroll-users-in-factors">
  ### Inscribir a los usuarios en factores
</div>

Una vez configurada MFA, asegúrate de que tus usuarios se inscriban en uno o más de los factores que habilitaste. Los usuarios deben inscribirse en autenticadores antes de que los comandos de la Action `Post Login` puedan presentarles una verificación.

Después de que un usuario se registre o se cree en tu tenant, puedes crear inscripciones con el <Tooltip tip="Management API: Un producto para permitir que los clientes realicen tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip> endpoint [authentication-methods](https://auth0.com/docs/api/management/v2/users/post-authentication-methods), o bien administrar directamente las inscripciones de los usuarios desde sus páginas de perfil en el Auth0 Dashboard.

<div id="customize-your-mfa-flows">
  ## Personaliza tus flujos de MFA
</div>

Una vez que tu tenant esté listo, puedes crear Actions de `Post Login` para personalizar tus flujos de MFA. A continuación, se incluyen los pasos y algunos casos de uso de ejemplo.

<Warning>
  Las Actions (o series de Actions) de un tenant solo pueden ejecutar **cuatro** de los siguientes comandos por flujo de usuario:

  * `enrollWith`
  * `enrollWithAny`
  * `challengeWith`
  * `challengeWithAny`

  Si se supera este límite (es decir, si se intenta ejecutar un quinto comando de este tipo), se producirá un error de autenticación.
</Warning>

<div id="create-your-post-login-action">
  ### Crea tu Action Post Login
</div>

1. En tu Auth0 Dashboard, ve a [Actions &gt; Flows](https://manage.auth0.com/#/actions/flows) y selecciona **Login**.
2. En Add Action, selecciona **Custom** y elige **Create Action**.
3. En la ventana emergente Create Action:

   * Ingresa un nombre para tu Action.
   * Selecciona **Login / Post-Login** como trigger.
   * Usa **Node 22 (Recommended)** como runtime.
4. Revisa la ventana emergente para asegurarte de que todo sea correcto. Luego, selecciona **Create**.
5. Después de crearla, el editor de código mostrará el comando `onPostExecute`. Agrega tu código personalizado o el código de ejemplo al comando.
6. Cuando el comando esté listo, selecciona **Deploy**.
7. Selecciona **Add to Flow** en la notificación de despliegue correcto.

   * **Nota**: Si la notificación ya se cerró, elige **Back to Flow** encima del editor de código.
8. Arrastra y suelta tu nuevo comando desde el panel Add Action a tu flujo de Login. Luego, selecciona **Apply**.

Para hacer más cambios en tu Action, ve a [Actions &gt; Library &gt; Custom](https://manage.auth0.com/#/actions/library) y selecciona tu comando. Luego, puedes actualizar y volver a desplegar tu código según sea necesario.

<Warning>
  Al agregar un Redirect a tu flujo de MFA, asegúrate de que tu Action cumpla las siguientes condiciones para evitar que los usuarios omitan o eludan MFA:

  * El comando de Redirect (`sendUserTo`) debe estar en una Action separada de tus comandos de MFA.
  * La Action de Redirect debe ser la última Action que se ejecute en tu flujo.

  Para obtener más información sobre Redirects, consulta [Redirect with Actions](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger/redirect-with-actions).
</Warning>

<div id="test-your-post-login-action">
  ### Prueba tu Action de Post Login
</div>

Para asegurarte de que tus comandos funcionen correctamente, puedes probar tu Action desde el Auth0 Dashboard:

* Ve a [Authentication &gt; Authentication Profile](https://manage.auth0.com/#/authentication-profiles).
* Selecciona **Try** para abrir un prompt de inicio de sesión de muestra en una pestaña nueva.
* Introduce tus credenciales y prueba tu nuevo flujo de MFA.

Si el flujo se completa correctamente, se mostrará una pantalla de confirmación. Si tienes algún problema, puedes actualizar tu código yendo a [Actions &gt; Biblioteca &gt; Custom](https://manage.auth0.com/#/actions/library) en tu Auth0 Dashboard.

<div id="use-current-enrollments-to-determine-challenge-method">
  ### Usa los registros actuales para determinar el método de desafío
</div>

El siguiente ejemplo presenta un desafío de MFA a un usuario si tiene inscritos los siguientes factores:

* Contraseña de un solo uso (OTP)
* Teléfono

```js lines
exports.onExecutePostLogin = async (event, api) => {

 api.authentication.challengeWithAny([{type: 'otp'}, {type: 'phone'}]);

}
```

<div id="use-roles-to-determine-challenge-method">
  ### Usa roles para determinar el método de desafío
</div>

El siguiente ejemplo aplica un desafío con OTP a todos los usuarios. Si un usuario tiene el rol Admin y necesita un nivel de acceso más alto a tu aplicación, se le solicita un factor adicional como forma de [autenticación escalonada](/docs/es/secure/multi-factor-authentication/step-up-authentication).

```js lines
exports.onExecutePostLogin = async (event, api) => {
    api.authentication.challengeWith({type: 'otp'});

    const isAdmin = event.authorization?.roles.includes('admin');
    if(isAdmin) {
        api.authentication.challengeWith({type: 'phone'});
    }
}
```

<div id="use-metadata-to-determine-challenge-method">
  ### Usa metadatos para determinar el método de verificación
</div>

En este ejemplo, los factores de MFA están habilitados a nivel de la [organización](/docs/es/manage-users/organizations). Este ejemplo usa distintas categorías de metadatos para determinar el método de verificación adecuado para cada usuario:

* **Metadatos de la organización**: Datos a nivel de organización, como los factores específicos habilitados para una organización.
* **Metadatos de usuario**: Datos a nivel de usuario, como si un usuario tiene un número de teléfono asociado a su perfil.

```js lines expandable
exports.onExecutePostLogin = async (event, api) => {
  const orgFactors = event.organization?.metadata.factors.split(',') ?? [];

  // Obtener la intersección de los factores disponibles para el usuario y los habilitados para la organización
  const availableFactors = orgFactors.filter(f => event.user?.enrolledFactors?.some(ef => ef.type === f));

  // Dar preferencia a push si está disponible
  if(availableFactors.includes('push-notification')) {
    api.authentication.challengeWith({ type: 'push-notification' });
    return;
  }

  // Si el usuario tiene un número de teléfono verificado y la organización
  // permite SMS y correo electrónico, dar preferencia a SMS y permitir el correo electrónico como alternativa
  // si está disponible
  if(event.user.phone_number && 
     event.user.phone_verified && 
     availableFactors.includes('phone')) {
    if(availableFactors.includes('email')) {
      api.authentication.challengeWith({ type: 'phone' }, {
        additionalFactors: [{
          type: 'email'
        }]
      });
    } else {
      api.authentication.challengeWith({ type: 'phone' });
    }

    return;
  }

  // Si no fue posible priorizar las notificaciones push y/o el teléfono, recurrir al correo electrónico si
  // está habilitado para la organización; de lo contrario, fallar.
  if(availableFactors.includes('email')) {
    api.authentication.challengeWith({ type: "email" });
    return;
  }

  api.access.deny("No MFA factors available for this org + user");
};
```

<div id="allow-users-to-select-an-alternate-method-of-authentication">
  ### Permitir que los usuarios seleccionen un método de autenticación alternativo
</div>

Para ofrecer una experiencia más flexible, puedes mostrar a los usuarios un enlace **Probar otro método** como parte de su desafío MFA. Este enlace les permite seleccionar un método de autenticación distinto del desafío predeterminado.

Para ello, incluye el parámetro [`additionalFactors`](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger/post-login-api-object#param-additional-factors) en tu código de Actions. Puedes configurar este parámetro con un factor específico para todos los usuarios o usar `enrolledFactors` para que elijan el factor que prefieran.

**Factor específico**

En el siguiente ejemplo, se presenta a los usuarios un desafío con OTP de forma predeterminada. Si lo desean, pueden acceder al enlace Probar otro método para autenticarse por email en su lugar.

```js lines
exports.onExecutePostLogin = async (event, api) => {
  api.authentication.challengeWith({ type: 'otp' }, 
    { additionalFactors: [{type: 'email'}] })
};
```

**Factores registrados**

En el siguiente ejemplo, a los usuarios se les solicita un OTP de forma predeterminada. Si lo desean, pueden usar el enlace Probar otro método para autenticarse con alguno de sus otros factores registrados.

```js lines
exports.onExecutePostLogin = async (event, api) => {
  const enrolledFactors = event.user.enrolledFactors.map((f) => ({type: f.type}));

  api.authentication.challengeWith({ type: 'otp' }, 
    { additionalFactors: enrolledFactors })
};
```

<div id="use-adaptive-mfa-to-determine-when-to-challenge-users">
  ### Usa MFA adaptativo para determinar cuándo solicitar verificación adicional a los usuarios
</div>

El siguiente ejemplo usa [MFA adaptativo](/docs/es/get-started/auth0-teams/tenant-management) para determinar si se debe solicitar verificación adicional a los usuarios.

<Tooltip tip="Autenticación multifactor adaptativa: autenticación multifactor (MFA) que solo se activa para los usuarios cuando se determina que un intento de inicio de sesión es de baja confianza." cta="Ver glosario" href="/docs/es/glossary?term=Adaptive+MFA">MFA adaptativo</Tooltip> es una política de MFA flexible que protege a tu tenant de <Tooltip tip="Autenticación multifactor adaptativa: autenticación multifactor (MFA) que solo se activa para los usuarios cuando se determina que un intento de inicio de sesión es de baja confianza." cta="Ver glosario" href="/docs/es/glossary?term=bad+actors">actores maliciosos</Tooltip> al evaluar el riesgo potencial durante las transacciones de inicio de sesión y solicitar a los usuarios una verificación adicional cuando corresponda.

En este caso, se solicita MFA a los usuarios si inician sesión desde un dispositivo no reconocido y su puntuación de confianza general es baja o media.

```javascript lines expandable
/**
* Controlador que será invocado durante la ejecución de un flujo Post Login.
*
* @param {Event} event - Detalles sobre el usuario y el contexto en el que está iniciando sesión.
* @param {PostLoginAPI} api - Interfaz cuyos métodos pueden usarse para cambiar el comportamiento del inicio de sesión.
*/
exports.onExecutePostLogin = async (event, api) => {
  if (event.authentication?.riskAssessment?.assessments.NewDevice) {

  // Condición de ejemplo: solicitar MFA únicamente en función del nivel de confianza 
    // de NewDevice; esto solicitará MFA cuando un usuario inicie sesión 
    // desde un dispositivo desconocido.
    let shouldPromptMfa;

    switch (event.authentication.riskAssessment.assessments.NewDevice.confidence) {
      case 'low':
      case 'medium':
        shouldPromptMfa = true;
        break;
      case 'high':
        shouldPromptMfa = false;
        break;
      case 'neutral':
        // Cuando este evaluador no dispone de información útil sobre la confianza, 
        // no solicitar MFA.
        shouldPromptMfa = false;
        break;
    }

      // Solo tiene sentido solicitar MFA cuando el usuario tiene al menos un 
      // factor MFA registrado.
    const canPromptMfa = event.user.enrolledFactors?.length > 0;

    if (shouldPromptMfa && canPromptMfa) {
      const enrolledFactors = event.user.enrolledFactors.map((f) => ({type: f.type}));
      api.authentication.challengeWithAny(enrolledFactors);
    }

  }

};
```

<div id="use-actions-to-challenge-users">
  ### Usa Actions para solicitar una verificación a los usuarios
</div>

Puedes usar Actions para personalizar los flujos de MFA modificando el trigger `Post Login` del flujo de Login. Este ejemplo usa el método `phone` de autenticación y `preferredMethod: 'both'`, que hace referencia a los factores de MFA activos asociados a la cuenta de un usuario. Para más información, consulta [Actions Triggers: Post Login - Objeto Event](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger/post-login-event-object).

```js lines
api.authentication.challengeWith({ 
  type: 'phone', 
  options: { preferredMethod: 'both'} 
});
```

<div id="troubleshooting-checklist">
  ### Lista de comprobación para la resolución de problemas
</div>

La siguiente lista de comprobación ofrece sugerencias adicionales para identificar y resolver problemas habituales con flujos de MFA personalizados.

1. El interruptor **Customize MFA factors with Actions** debe estar habilitado.

   * Vaya a [Auth0 Dashboard &gt; Security &gt; Multi-factor Auth](https://manage.auth0.com/#/security/mfa) y asegúrese de que el interruptor de la sección Additional Settings esté habilitado.
2. Los factores a los que hacen referencia sus Actions deben estar habilitados en su tenant.

   * **Revise su código**: Vaya a [Auth0 Dashboard &gt; Actions &gt; Library &gt; Custom](https://manage.auth0.com/#/actions/library) y revise el código de sus Actions. Asegúrese de que todos los factores a los que se hace referencia sean aplicables a sus casos de uso.
   * **Revise sus factores**: Vaya a [Auth0 Dashboard &gt; Security &gt; Multi-factor Auth](https://manage.auth0.com/#/security/mfa) y asegúrese de que todos los factores a los que hacen referencia sus Actions estén habilitados.
3. Los usuarios deben estar registrados en los factores a los que hacen referencia sus Actions.

   * Si una persona recibe un error, revise los detalles de su usuario para asegurarse de que esté registrada en los factores correctos. Vaya a [**Auth0 Dashboard &gt; User Management &gt; Users**](https://manage.auth0.com/#/users) y seleccione su nombre de la lista.

     * Revise la sección Multi-factor Authentication en la **pestaña Detail** para verificar su inscripción. Si el usuario no está registrado, puede usar el enlace **Send an enrollment invitation** disponible en esta sección.
     * Como alternativa, verifique la inscripción del usuario en la **pestaña Raw JSON**. Esta información también se puede recuperar mediante el endpoint de la [Management API](https://auth0.com/docs/api/management/v2/users/get-authentication-methods). Sin embargo, es importante tener en cuenta que la API no enumera los autenticadores registrados automáticamente, como los factores Email configurados mediante un enlace de verificación por correo electrónico.
   * Si los usuarios no están registrados en los factores adecuados, puede crear inscripciones con el endpoint [authentication-methods](https://auth0.com/docs/api/management/v2/users/post-authentication-methods) de la Management API. También puede administrar la inscripción de los usuarios directamente desde sus páginas de perfil en el Auth0 Dashboard.
4. Asegúrese de que sus Actions se hayan desplegado y guardado en su pipeline.

   * Vaya a [Auth0 Dashboard &gt; Actions &gt; Library &gt; Custom](https://manage.auth0.com/#/actions/library). Busque su Action en la lista y asegúrese de que su estado sea **Deployed**. Si aparece un estado distinto, acceda a su Action, revise su código y haga clic en **Deploy** en la parte superior derecha.
   * Vaya a [Auth0 Dashboard &gt; Actions &gt; Library &gt; Flows](https://manage.auth0.com/#/actions/flows) y seleccione **Login**. Asegúrese de que su Action aparezca en el flujo. Si no es así, acceda a la **pestaña Custom** del panel Add Action y arrastre y suelte su Action en su Login flow. Luego, seleccione **Apply**.
5. Asegúrese de haber actualizado a la versión más reciente de las Actions de `Post Login`.

   * Vaya a [Auth0 Dashboard &gt; Actions &gt; Library &gt; Custom](https://manage.auth0.com/#/actions/library) y seleccione su Action. Si su Action está desactualizada, verá un aviso amarillo que le pedirá actualizar la Action. Si aparece el aviso, seleccione **Update**.
   * También puede especificar la versión más reciente de las Actions de `Post Login` para el despliegue al usar Deploy CLI. Para obtener más información, consulte [Configurar Deploy CLI](/docs/es/deploy-monitor/deploy-cli-tool/configure-the-deploy-cli).
