<Warning>
  Si necesitas configurar un dominio personalizado, hazlo **antes** de implementar WebAuthn en entornos de producción. Si configuras o cambias un dominio personalizado, los usuarios que ya se hayan inscrito no podrán autenticarse.

  Puedes usar la configuración [parte de confianza](#configure-relying-party) para especificar el dominio que se usa para autenticar a los usuarios.
</Warning>

<div id="use-the-dashboard">
  ## Usa el Dashboard
</div>

1. Habilita **WebAuthn with claves de seguridad** en [Dashboard &gt; Security &gt; Multi-factor Auth](https://manage.auth0.com/#/security/mfa).
2. Configura cómo quieres gestionar la verificación del usuario. En el caso de las claves de seguridad, lo habitual es que se pida a los usuarios que introduzcan un PIN para completar el challenge de WebAuthn.

   1. Never: Nunca se pedirá a los usuarios que introduzcan un PIN. Este es el valor predeterminado y, por lo general, suele ser suficiente cuando se usan claves de seguridad para MFA. Los usuarios ya introdujeron su Password, así que ya aportaron cierta verificación.
   2. If supported: Se pedirá a los usuarios que introduzcan un PIN si ya han configurado uno en la clave.
   3. Required: Se pedirá a los usuarios que establezcan un PIN si aún no lo han hecho, y tendrán que introducirlo cada vez. Esta es la opción que ofrece el mayor nivel de seguridad. Algunos navegadores no implementan esto correctamente (por ejemplo, [Brave](https://brave.com/) en iOS), por lo que la authentication fallará y Auth0 pedirá a los usuarios que utilicen otro navegador.

Ten en cuenta que solo las claves de seguridad compatibles con FIDO-2 admiten la verificación del usuario. Las claves FIDO-1 pueden usarse con WebAuthn, pero no sirven si configuras la verificación del usuario como Required.

<div id="configure-relying-party">
  ## Configurar parte de confianza
</div>

WebAuthn hace que el phishing sea imposible al vincular las credenciales con el origen del navegador. Los usuarios tampoco pueden usar WebAuthn en un sitio en el que no se registraron.

Vincular las credenciales al origen significa que, si configuras un <Tooltip tip="Dominio personalizado: dominio de terceros con un nombre especializado o de vanidad." cta="Ver glosario" href="/docs/es/glossary?term=custom+domain">dominio personalizado</Tooltip> o lo cambias, los usuarios inscritos antes del cambio no podrán autenticarse.

WebAuthn define un [atributo de ID de parte de confianza](https://www.w3.org/TR/webauthn-2/#relying-party-identifier), que te permite especificar el dominio que se usa para autenticar a los usuarios. Puedes establecerlo en cualquier sufijo de dominio registrable del origen del navegador. Por ejemplo, si el dominio personalizado es `login.example.com`, puedes configurar el ID de la <Tooltip tip="Parte de confianza: entidad (como un servicio o una aplicación) que depende de un proveedor de identidad de terceros para autenticar a un usuario." cta="Ver glosario" href="/docs/es/glossary?term=Relying+Party">parte de confianza</Tooltip> como `example.com`. Esto permite a los usuarios autenticarse en cualquier dominio `example.com` con sus credenciales de WebAuthn.

Auth0 te permite especificar el ID de la parte de confianza solo si tienes configurado un dominio personalizado. Si el dominio personalizado cambia, debes actualizar el ID de la parte de confianza.

<div id="device-support">
  ## Compatibilidad de dispositivos
</div>

Para usar claves de seguridad, un navegador debe tener JavaScript habilitado y ser compatible con WebAuthn. Si no se cumplen esas condiciones, Auth0 no ofrecerá la opción de registrar ni de autenticarse con claves de seguridad. Auth0 solicitará la verificación con otro factor o con un código de recuperación (si no tienen registrado otro factor).

Las versiones más recientes de los navegadores y sistemas operativos más populares son compatibles con WebAuthn with claves de seguridad. Para obtener más información, consulta la [sección de compatibilidad de navegadores en webauthn.me](https://webauthn.me/browser-support).

<div id="limitations">
  ## Limitaciones
</div>

* Al usar la [API de MFA](/docs/es/secure/multi-factor-authentication/multi-factor-authentication-developer-resources/mfa-api), puedes listar y eliminar registros WebAuthn, pero no puedes registrarlos.
