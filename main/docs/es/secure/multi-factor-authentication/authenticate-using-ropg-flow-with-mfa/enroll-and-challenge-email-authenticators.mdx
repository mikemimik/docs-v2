Auth0 proporciona un flujo integrado de inscripción y autenticación de <Tooltip tip="Autenticación multifactor (MFA): proceso de autenticación de usuario que utiliza un factor adicional además de username y password, como un código por SMS." cta="Ver glosario" href="/docs/es/glossary?term=MFA">MFA</Tooltip> mediante [Universal Login](/docs/es/authenticate/login/auth0-universal-login). Sin embargo, si quieres crear tu propia interfaz de usuario, puedes usar la [API de MFA](/docs/es/secure/multi-factor-authentication/multi-factor-authentication-developer-resources/mfa-api) para hacerlo.

Cuando el correo electrónico está habilitado como factor, todos los usuarios con direcciones de correo electrónico verificadas podrán usarlas para completar MFA.

</Card>

<div id="prerequisites">
  ## Requisitos previos
</div>

Antes de poder usar las API de MFA, debes habilitar el tipo de concesión de MFA para tu aplicación. Ve a [Auth0 Dashboard &gt; Applications &gt; Advanced Settings &gt; Grant Types](https://manage.auth0.com/#/applications) y selecciona **MFA**.

* [Configurar Email](/docs/es/secure/multi-factor-authentication/multi-factor-authentication-factors/configure-email-notifications-for-mfa) como factor en el Dashboard o mediante la [Management API](https://auth0.com/docs/api/management/v2#!/Guardian/put_factors_by_name).

<div id="enroll-with-email">
  ## Registrar con Email
</div>

Para permitir que los usuarios registren direcciones de correo electrónico además del correo electrónico verificado de su identidad principal, debe completar los siguientes pasos.

### Obtener token de MFA

Según cuándo inicies la inscripción, puedes obtener un <Tooltip tip="Token de acceso: credencial de autorización, en forma de una cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=access+token">token de acceso</Tooltip> para usar la API de MFA de diferentes maneras:

* Si realizas la inscripción durante la autenticación, consulta [Autenticar con la concesión de contraseña del propietario del recurso y MFA](/docs/es/secure/multi-factor-authentication/authenticate-using-ropg-flow-with-mfa).
* Si quieres permitir que el usuario registre un factor en cualquier momento, consulta [Gestionar inscripciones de factores de MFA](/docs/es/secure/multi-factor-authentication/manage-mfa-auth0-apis/manage-authenticator-factors-mfa-api).

<div id="enroll-authenticator">
  ### Registrar autenticador
</div>

Haz una solicitud `POST` al endpoint de MFA Associate para registrar el autenticador del usuario. El bearer token que requiere este endpoint es el MFA token obtenido en el paso anterior.

Usa los siguientes parámetros:

| Parámetro              | Valor                                                                |
| ---------------------- | -------------------------------------------------------------------- |
| `authentication_types` | `[oob]`                                                              |
| `oob_channels`         | `[email]`                                                            |
| `email`                | `email@address.com`, la dirección de correo electrónico del usuario. |

<AuthCodeGroup>
  ```bash cURL
  curl --request POST \
    --url 'https://{yourDomain}/mfa/associate' \
    --header 'authorization: Bearer MFA_TOKEN' \
    --header 'content-type: application/json' \
    --data '{ "authenticator_types": ["oob"], "oob_channels": ["email"], "email" : "email@address.com" }'
  ```

  ```csharp C#
  var client = new RestClient("https://{yourDomain}/mfa/associate");
  var request = new RestRequest(Method.POST);
  request.AddHeader("authorization", "Bearer MFA_TOKEN");
  request.AddHeader("content-type", "application/json");
  request.AddParameter("application/json", "{ "authenticator_types": ["oob"], "oob_channels": ["email"], "email" : "email@address.com" }", ParameterType.RequestBody);
  IRestResponse response = client.Execute(request);
  ```

  ```go Go
  package main

  import (
  	"fmt"
  	"strings"
  	"net/http"
  	"io/ioutil"
  )

  func main() {

  	url := "https://{yourDomain}/mfa/associate"

  	payload := strings.NewReader("{ "authenticator_types": ["oob"], "oob_channels": ["email"], "email" : "email@address.com" }")

  	req, _ := http.NewRequest("POST", url, payload)

  	req.Header.Add("authorization", "Bearer MFA_TOKEN")
  	req.Header.Add("content-type", "application/json")

  	res, _ := http.DefaultClient.Do(req)

  	defer res.Body.Close()
  	body, _ := ioutil.ReadAll(res.Body)

  	fmt.Println(res)
  	fmt.Println(string(body))

  }
  ```

  ```java Java

    .header("content-type", "application/json")

    .asString();
  ```

  ```javascript Node.JS
    console.error(error);
  ```

  ```python Python

  conn = http.client.HTTPSConnection("")

  payload = "{ "authenticator_types": ["oob"], "oob_channels": ["email"], "email" : "email@address.com" }"

  headers = {
      'authorization': "Bearer MFA_TOKEN",
      'content-type': "application/json"
      }

  res = conn.getresponse()
  data = res.read()

  print(data.decode("utf-8"))
  ```

  ```ruby Ruby

  require 'net/http'
  require 'openssl'

  http.use_ssl = true
  http.verify_mode = OpenSSL::SSL::VERIFY_NONE

  request = Net::HTTP::Post.new(url)
  request["authorization"] = 'Bearer MFA_TOKEN'
  request["content-type"] = 'application/json'
  request.body = "{ "authenticator_types": ["oob"], "oob_channels": ["email"], "email" : "email@address.com" }"

  response = http.request(request)
  puts response.read_body
  ```
</AuthCodeGroup>

Si la solicitud se realiza correctamente, recibirás una respuesta como esta:

```json lines
{
    "authenticator_type": "oob",
    "binding_method": "prompt",
    "oob_code" : "Fe26..nWE",
    "oob_channel": "email",
    "recovery_codes": [ "N3BGPZZWJ85JLCNPZBDW6QXC" ]
  }
```

Si recibes un error `User is already enrolled`, el usuario ya tiene inscrito un factor de MFA. Antes de asociar otro factor con el usuario, debes desafiarlo con el factor existente.

Si es la primera vez que el usuario asocia un autenticador, verás que la respuesta incluye `recovery_codes`. Los códigos de recuperación se usan para acceder a la cuenta del usuario en caso de que pierda acceso a la cuenta o al dispositivo utilizado para la autenticación de segundo factor. Estos códigos son de un solo uso y se generan otros nuevos cuando es necesario.

<div id="confirm-email-enrollment">
  ### Confirmar la inscripción por correo electrónico
</div>

El usuario debería recibir un correo electrónico que contenga el código de 6 dígitos que luego podrá proporcionar a la aplicación.

Para completar la inscripción, haz una solicitud `POST` al endpoint [**`/oath/token`**](https://auth0.com/docs/api/authentication#get-token). Incluye el `oob_code` devuelto en la respuesta anterior y el `binding_code` con el valor incluido en el mensaje de correo electrónico.

<AuthCodeGroup>
  ```bash cURL
  curl --request POST \
    --url 'https://{yourDomain}/oauth/token' \
    --data grant_type=http://auth0.com/oauth/grant-type/mfa-oob \
    --data 'mfa_token={mfaToken}' \
    --data 'oob_code={oobCode}' \
    --data 'binding_code={userEmailOtpCode}' \
    --data 'client_id={yourClientId}' \
    --data 'client_secret={yourClientSecret}'
  ```


<div id="customize-mfa">
  ## Personaliza MFA
</div>

<Warning>
  La MFA personalizable con los flujos de contraseña del propietario del recurso, Embedded o token de actualización está en acceso anticipado. Al usar esta funcionalidad, aceptas los términos aplicables de la prueba gratuita del [Master Subscription Agreement](https://www.okta.com/legal/) de Okta. Para obtener más información sobre las etapas de lanzamiento de Auth0, consulta [Product Release Stages](/docs/es/troubleshoot/product-lifecycle/product-release-stages). Para participar en el acceso anticipado, contacta al [equipo de soporte de Auth0](https://support.auth0.com/).
</Warning>

La MFA personalizable permite a los usuarios registrar y usar en desafíos los factores compatibles con tu aplicación que prefieran.

Durante la autenticación en el endpoint `oauth/token`, la respuesta devuelve el error `mfa_required`, que incluye `mfa_token` para usar la API de MFA y el parámetro `mfa_requirements` con una lista de autenticadores:

```json
{
  "error": "mfa_required",
  "error_description": "Multifactor authentication required",
  "mfa_token": "Fe26...Ha",
  "mfa_requirements": {
    "challenge": [
      { "type": "otp" },
      { "type": "push-notification" },
      { "type": "phone" },
      { "type": "recovery-code" }
      { "type": "email"} //solo puede funcionar con challenge
    ]
  }
}
```

Usa el `mfa_token` para llamar al endpoint [`mfa/authenticator`](/docs/es/api/authentication/muti-factor-authentication/list-authenticators) y obtener una lista de todos los factores que el usuario tiene registrados, y compárala con el mismo tipo que admite tu aplicación.  También debes obtener el `authenticator_type` correspondiente para emitir desafíos:

```json
[
  {
    "type": "recovery-code",
    "id": "recovery-code|dev_qpOkGUOxBpw6R16t",
    "authenticator_type": "recovery-code",
    "active": true
  },
  {
    "type": "otp",
    "id": "totp|dev_6NWz8awwC8brh2dN",
    "authenticator_type": "otp",
    "active": true
  }
]
```

Exija el desafío MFA llamando al endpoint [`request/mfa/challenge`](/docs/es/api/authentication/muti-factor-authentication/request-mfa-challenge).

Personalice aún más su flujo de MFA con Auth0 Actions. Para obtener más información, consulte [Actions Triggers: post-challenge - objeto de API](/docs/es/customize/actions/explore-triggers/password-reset-triggers/post-challenge-trigger/post-challenge-api-object).
