Auth0 proporciona un flujo integrado de inscripción y autenticación de <Tooltip tip="Autenticación multifactor (MFA): proceso de autenticación de usuario que usa un factor además de nombre de usuario y contraseña, como un código enviado por SMS." cta="Ver glosario" href="/docs/es/glossary?term=MFA">MFA</Tooltip> con [Universal Login](/docs/es/authenticate/login/auth0-universal-login). Sin embargo, si quieres crear tu propia interfaz de usuario, puedes usar la [API de MFA](/docs/es/secure/multi-factor-authentication/multi-factor-authentication-developer-resources/mfa-api) para hacerlo.

Puedes inscribir a los usuarios y enviarles solicitudes de desafío mediante notificaciones push con la aplicación Guardian o el SDK usando la API de MFA.

<div id="prerequisites">
  ## Requisitos previos
</div>

Antes de usar las API de MFA, debes habilitar el tipo de concesión de MFA para tu aplicación. Ve a [Auth0 Dashboard &gt; Applications &gt; Advanced Settings &gt; Grant Types](https://manage.auth0.com/#/applications) y selecciona **MFA**.

* [Configurar Push](/docs/es/secure/multi-factor-authentication/multi-factor-authentication-factors/configure-push-notifications-for-mfa) como factor en el Dashboard o mediante la [Management API](https://auth0.com/docs/api/management/v2#!/Guardian/put_factors_by_name).

<div id="enroll-with-push">
  ## Inscribirse con push
</div>

<div id="get-mfa-token">
  ### Obtener un token de MFA
</div>

Según en qué momento inicies la inscripción, puedes obtener un <Tooltip tip="Token de acceso: Credencial de autorización, en forma de una cadena opaca o JWT, usada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=access+token">token de acceso</Tooltip> para usar la API de MFA de distintas maneras:

* Si realizas la inscripción durante la autenticación, consulta [Autenticarse con Resource Owner Password Grant y MFA](/docs/es/secure/multi-factor-authentication/authenticate-using-ropg-flow-with-mfa).
* Si quieres permitir que el usuario inscriba un factor en cualquier momento, consulta [Administrar inscripciones de factores de MFA](/docs/es/secure/multi-factor-authentication/manage-mfa-auth0-apis/manage-authenticator-factors-mfa-api).

<div id="enroll-authenticator">
  ### Inscribir autenticador
</div>

Haz una solicitud `POST` al endpoint MFA Associate para inscribir el autenticador del usuario. El bearer token que requiere este endpoint es el MFA token obtenido en el paso anterior.

Para inscribir el autenticador con push, establece el parámetro `authenticator_types` en `[oob]` y el parámetro `oob_channels` en `[auth0]`.

<AuthCodeGroup>
  ```bash cURL
  curl --request POST \
    --url 'https://{yourDomain}/mfa/associate' \
    --header 'authorization: Bearer MFA_TOKEN' \
    --header 'content-type: application/json' \
    --data '{ "authenticator_types": ["oob"], "oob_channels": ["auth0"] }'
  ```

  ```csharp C#
  var client = new RestClient("https://{yourDomain}/mfa/associate");
  var request = new RestRequest(Method.POST);
  request.AddHeader("authorization", "Bearer MFA_TOKEN");
  request.AddHeader("content-type", "application/json");
  request.AddParameter("application/json", "{ "authenticator_types": ["oob"], "oob_channels": ["auth0"] }", ParameterType.RequestBody);
  IRestResponse response = client.Execute(request);
  ```

  ```go Go
  package main

  import (
  	"fmt"
  	"strings"
  	"net/http"

  func main() {

  	url := "https://{yourDomain}/mfa/associate"

  	payload := strings.NewReader("{ "authenticator_types": ["oob"], "oob_channels": ["auth0"] }")

  	req, _ := http.NewRequest("POST", url, payload)

  	req.Header.Add("authorization", "Bearer MFA_TOKEN")
  	req.Header.Add("content-type", "application/json")

  	res, _ := http.DefaultClient.Do(req)

  	defer res.Body.Close()
  	body, _ := ioutil.ReadAll(res.Body)

  	fmt.Println(res)
  	fmt.Println(string(body))

  }
  ```

</AuthCodeGroup>

Si la solicitud se realiza correctamente, recibirás una respuesta como esta:

```json lines
{
    "authenticator_type": "oob",
    "barcode_uri": "otpauth://totp/tenant:user?enrollment_tx_id=qfjn2eiNYSjU3xID7dBYeCBSrdREWJPY&base_url=tenan",
    "recovery_codes": [
        "ALKE6EJZ4853BJYLM2DM2WU7"
    ],
    "oob_channel": "auth0",
    "oob_code": "Fe26.2...SYAg"
}
```

Si obtienes un error `User is already enrolled`, el usuario ya tiene inscrito un factor de MFA. Antes de asociar otro factor con el usuario, debes pedirle que complete un desafío con el factor existente.

Si esta es la primera vez que el usuario asocia un autenticador, notarás que la respuesta incluye `recovery_codes`. Los códigos de recuperación se usan para acceder a la cuenta del usuario en caso de que pierda el acceso a ella o al dispositivo utilizado para su autenticación de segundo factor. Estos códigos son de un solo uso y se generan nuevos según sea necesario.

<div id="confirm-push-enrollment">
  ### Confirmar la inscripción con push
</div>

Para confirmar la inscripción, el usuario final deberá escanear un código QR con el `barcode_uri` en la aplicación Guardian, en los próximos 5 minutos.

Una vez hecho esto, la aplicación Guardian notificará a Auth0 que el usuario completó la inscripción correctamente. Para saber si eso ocurrió, consulta el <Tooltip tip="Endpoint de token: endpoint en el servidor de autorización que se usa para solicitar tokens de forma programática." cta="Ver glosario" href="/docs/es/glossary?term=Token+endpoint">endpoint de token</Tooltip> de Auth0 con el `oob_code` devuelto por la llamada al endpoint MFA Associate.

Si el usuario no ha escaneado el código, se devolverá una respuesta `authorization_pending`, lo que indica que debes volver a llamar a `oauth_token` en unos segundos:

```json lines
{
    "error": "authorization_pending",
    "error_description": "Authorization pending: please repeat the request in a few seconds."
}
```

Si la llamada se realizó correctamente, recibirás una respuesta con el siguiente formato, que contiene el token de acceso:

```json lines
{
  "id_token": "eyJ...i",
  "access_token": "eyJ...i",
  "expires_in": 600,
  "scope": "openid profile",
  "token_type": "Bearer"
}
```

En este punto, el autenticador ya está completamente asociado y listo para usarse, y ya tienes los tokens de autenticación del usuario.

Puedes comprobar en cualquier momento si se ha confirmado un autenticador consultando el endpoint de autenticadores de MFA. Si el autenticador está confirmado, el valor que se devuelve para `active` es `true`.

La llamada puede devolver uno de los siguientes resultados:

| Resultado                        | Descripción                                                                                   |
| -------------------------------- | --------------------------------------------------------------------------------------------- |
| `authorization_pending`          | Error: Si el desafío aún no se ha aceptado ni rechazado.                                     |
| `slow_down`                      | Error: Si la consulta periódica es demasiado frecuente.                                      |
| `access_token` y `refresh_token` | Si el desafío ha sido aceptado; la consulta periódica debe detenerse en este punto.          |
| `invalid_grant`                  | Error: Si el desafío ha sido rechazado; la consulta periódica debe detenerse en este punto. |

<div id="customize-mfa">
  ## Personalizar MFA
</div>

<Warning>
  La MFA personalizable con los flujos Resource Owner Password Grant, Embedded o de token de actualización está en acceso anticipado. Al usar esta funcionalidad, aceptas los términos aplicables de la prueba gratuita del [Master Subscription Agreement](https://www.okta.com/legal/) de Okta. Para obtener más información sobre las etapas de lanzamiento de Auth0, consulta [Product Release Stages](/docs/es/troubleshoot/product-lifecycle/product-release-stages). Para participar en el acceso anticipado, ponte en contacto con el [equipo de soporte de Auth0](https://support.auth0.com/).
</Warning>

La MFA personalizable permite a los usuarios inscribir y desafiar los factores compatibles con su aplicación que prefieran.

Durante la autenticación en el endpoint `oauth/token`, la respuesta devuelve el error `mfa_required`, que incluye el `mfa_token` para usar la API de MFA y el parámetro `mfa_requirements` con una lista de autenticadores:

```json
{
  "error": "mfa_required",
  "error_description": "Multifactor authentication required",
  "mfa_token": "Fe26...Ha",
  "mfa_requirements": {
    "challenge": [
      { "type": "otp" },
      { "type": "push-notification" },
      { "type": "phone" },
      { "type": "recovery-code" }
      { "type": "email"} //solo puede funcionar con challenge
    ]
  }
}
```

Usa el `mfa_token` para llamar al endpoint [`mfa/authenticator`](/docs/es/api/authentication/muti-factor-authentication/list-authenticators) y obtener una lista de todos los factores que el usuario tiene inscritos, y busca el que sea del mismo tipo que admite tu aplicación. También necesitas obtener el `authenticator_type` correspondiente para emitir desafíos:

```json
[
  {
    "type": "recovery-code",
    "id": "recovery-code|dev_qpOkGUOxBpw6R16t",
    "authenticator_type": "recovery-code",
    "active": true
  },
  {
    "type": "otp",
    "id": "totp|dev_6NWz8awwC8brh2dN",
    "authenticator_type": "otp",
    "active": true
  }
]
```

Aplique el desafío MFA llamando al endpoint [`request/mfa/challenge`](/docs/es/api/authentication/muti-factor-authentication/request-mfa-challenge).

Personalice aún más su flujo MFA con Auth0 Actions. Para obtener más información, lea [Triggers de Actions: post-challenge - objeto de API](/docs/es/customize/actions/explore-triggers/password-reset-triggers/post-challenge-trigger/post-challenge-api-object).
