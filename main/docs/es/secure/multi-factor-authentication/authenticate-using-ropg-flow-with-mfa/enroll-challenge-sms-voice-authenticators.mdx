Auth0 proporciona un flujo integrado de inscripción y autenticación de <Tooltip tip="Autenticación multifactor (MFA): Proceso de autenticación de usuarios que utiliza un factor además del nombre de usuario y la contraseña, como un código por SMS." cta="Ver glosario" href="/docs/es/glossary?term=MFA">MFA</Tooltip> mediante [Universal Login](/docs/es/authenticate/login/auth0-universal-login). Sin embargo, si quieres crear tu propia interfaz de usuario, puedes usar la [API de MFA](/docs/es/secure/multi-factor-authentication/multi-factor-authentication-developer-resources/mfa-api) para hacerlo.

<div id="prerequisites">
  ## Requisitos previos
</div>

Antes de usar las API de MFA, debes habilitar el tipo de concesión de MFA para tu aplicación. Ve a [Auth0 Dashboard &gt; Applications &gt; Advanced Settings &gt; Grant Types](https://manage.auth0.com/#/applications) y selecciona **MFA**.

* [Configura phone como factor](/docs/es/secure/multi-factor-authentication/multi-factor-authentication-factors/configure-sms-voice-notifications-mfa) en el Dashboard o con la [Management API](https://auth0.com/docs/api/management/v2#!/Guardian/put_factors_by_name).

<div id="enroll-with-sms-or-voice">
  ## Registrarse con SMS o voz
</div>

<div id="get-mfa-token">
  ### Obtener un token de MFA
</div>

Según el momento en que inicies la inscripción, puedes obtener un <Tooltip tip="Token de acceso: credencial de autorización, en forma de una cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=access+token">token de acceso</Tooltip> para usar la API de MFA de distintas maneras:

* Si realizas la inscripción durante la autenticación, consulta [Autenticarse con la concesión de contraseña del propietario del recurso y MFA](/docs/es/secure/multi-factor-authentication/authenticate-using-ropg-flow-with-mfa).
* Si quieres permitir que el usuario inscriba un factor en cualquier momento, consulta [Administrar las inscripciones de factores de MFA](/docs/es/secure/multi-factor-authentication/manage-mfa-auth0-apis/manage-authenticator-factors-mfa-api).

<div id="enroll-authenticator">
  ### Registrar autenticador
</div>

Realiza una solicitud `POST` al endpoint MFA Associate para registrar el autenticador del usuario. El token bearer que requiere este endpoint es el token de MFA obtenido en el paso anterior.

Para registrarse con SMS o voz, se registra un número de teléfono que puede desafiarse por SMS o por voz. Especifica los siguientes parámetros para llamar al endpoint. El parámetro `oob_channels` indica cómo quieres enviarle el código al usuario (SMS o voz).

<AuthCodeGroup>
  ```bash cURL
  curl --request POST \
    --url 'https://{yourDomain}/mfa/associate' \
    --header 'authorization: Bearer {mfaToken}' \
    --header 'content-type: application/json' \
    --data '{ "authenticator_types": ["oob"], "oob_channels": ["sms"], "phone_number": "+11...9" }'
  ```

  ```csharp C#
  var client = new RestClient("https://{yourDomain}/mfa/associate");
  var request = new RestRequest(Method.POST);
  request.AddHeader("authorization", "Bearer {mfaToken}");
  request.AddHeader("content-type", "application/json");
  request.AddParameter("application/json", "{ "authenticator_types": ["oob"], "oob_channels": ["sms"], "phone_number": "+11...9" }", ParameterType.RequestBody);
  IRestResponse response = client.Execute(request);
  ```

  ```go Go
  package main

  import (
  	"fmt"
  	"strings"
  	"net/http"
  	"io/ioutil"
  )

  func main() {

  	url := "https://{yourDomain}/mfa/associate"

  	payload := strings.NewReader("{ "authenticator_types": ["oob"], "oob_channels": ["sms"], "phone_number": "+11...9" }")

  	req, _ := http.NewRequest("POST", url, payload)

  	req.Header.Add("authorization", "Bearer {mfaToken}")
  	req.Header.Add("content-type", "application/json")

  	res, _ := http.DefaultClient.Do(req)

  	defer res.Body.Close()
  	body, _ := ioutil.ReadAll(res.Body)

  	fmt.Println(res)
  	fmt.Println(string(body))

  }
  ```

</AuthCodeGroup>

| Parámetro              | Valor                                                                                    |
| ---------------------- | ---------------------------------------------------------------------------------------- |
| `authentication_types` | `[oob]`                                                                                  |
| `oob_channels`         | `[sms]` o `[voice]`                                                                      |
| `phone_number`         | `+11...9`, el número de teléfono en [formato E.164](https://en.wikipedia.org/wiki/e.164) |

Si todo sale bien, recibirá una respuesta como esta:

```json lines
{
  "authenticator_type": "oob",
  "binding_method": "prompt",
  "recovery_codes": [ "N3BGPZZWJ85JLCNPZBDW6QXC" ],
  "oob_channel": "sms",
  "oob_code": "ata6daXAiOi..."
}
`
```

Si recibes un error `User is already enrolled`, el usuario ya tiene un factor de MFA registrado. Antes de asociar otro factor con el usuario, debes desafiar al usuario con el factor existente.

Si esta es la primera vez que el usuario asocia un autenticador, notarás que la respuesta incluye `recovery_codes`. Los códigos de recuperación se usan para acceder a la cuenta del usuario en caso de que pierda el acceso a la cuenta o al dispositivo utilizado para la autenticación de segundo factor. Son códigos de un solo uso y se generan nuevos según sea necesario.

<div id="confirm-sms-or-voice-enrollment">
  ### Confirmar la inscripción por SMS o voz
</div>

Los usuarios deberían recibir un mensaje con un código de 6 dígitos que deberán proporcionar a la aplicación.

Para completar la inscripción, realiza una solicitud `POST` al <Tooltip tip="Endpoint de token: endpoint del servidor de autorización que se usa para solicitar tokens mediante programación." cta="Ver glosario" href="/docs/es/glossary?term=OAuth">OAuth</Tooltip> <Tooltip tip="OAuth 2.0: marco de autorización que define protocolos y flujos de trabajo de autorización." cta="Ver glosario" href="/docs/es/glossary?term=Token+endpoint">endpoint de token</Tooltip>. Debes incluir el `oob_code` devuelto en la respuesta anterior y el `binding_code` con el valor recibido en el mensaje.

<AuthCodeGroup>
  ```bash cURL
  curl --request POST \
    --url 'https://{yourDomain}/oauth/token' \
    --header 'authorization: Bearer {mfaToken}' \
    --header 'content-type: application/x-www-form-urlencoded' \
    --data grant_type=http://auth0.com/oauth/grant-type/mfa-oob \
    --data 'client_id={yourClientId}' \
    --data 'client_secret={yourClientSecret}' \
    --data 'mfa_token={mfaToken}' \
    --data 'oob_code={oobCode}' \
    --data 'binding_code={userOtpCode}'
  ```

  ```csharp C#
  var client = new RestClient("https://{yourDomain}/oauth/token");
  var request = new RestRequest(Method.POST);
  request.AddHeader("authorization", "Bearer {mfaToken}");
  request.AddHeader("content-type", "application/x-www-form-urlencoded");
  request.AddParameter("application/x-www-form-urlencoded", "grant_type=http%3A%2F%2Fauth0.com%2Foauth%2Fgrant-type%2Fmfa-oob&client_id={yourClientId}&client_secret=%7ByourClientSecret%7D&mfa_token=%7BmfaToken%7D&oob_code=%7BoobCode%7D&binding_code=%7BuserOtpCode%7D", ParameterType.RequestBody);

  ```

  ### Solicitar desafío del usuario con OTP

  "challenge_type": "oob",

<div id="customize-mfa">
  ## Personaliza MFA
</div>

<Warning>
  La MFA personalizable con los flujos de concesión de contraseña del propietario del recurso, Embedded o token de actualización está en acceso anticipado. Al usar esta función, aceptas los términos aplicables de la prueba gratuita del [Master Subscription Agreement](https://www.okta.com/legal/) de Okta. Para obtener más información sobre las fases de lanzamiento de Auth0, consulta [Product Release Stages](/docs/es/troubleshoot/product-lifecycle/product-release-stages). Para participar en el acceso anticipado, contacta al [equipo de soporte de Auth0](https://support.auth0.com/).
</Warning>

La MFA personalizable permite a los usuarios registrar y desafiar factores de su elección compatibles con tu aplicación.

Durante la autenticación en el endpoint `oauth/token`, la respuesta devuelve el error `mfa_required`, que incluye el `mfa_token` para usar la API de MFA y el parámetro `mfa_requirements` con una lista de autenticadores:

```json
{
  "error": "mfa_required",
  "error_description": "Multifactor authentication required",
  "mfa_token": "Fe26...Ha",
  "mfa_requirements": {
    "challenge": [
      { "type": "otp" },
      { "type": "push-notification" },
      { "type": "phone" },
      { "type": "recovery-code" }
      { "type": "email"} //solo puede funcionar con challenge
    ]
  }
}
```

Usa el `mfa_token` para llamar al endpoint [`mfa/authenticator`](/docs/es/api/authentication/muti-factor-authentication/list-authenticators) y obtener una lista de todos los factores que el usuario haya registrado, y encontrar uno del mismo tipo que admite tu aplicación.  También debes obtener el `authenticator_type` correspondiente para emitir desafíos:

```json
[
  {
    "type": "recovery-code",
    "id": "recovery-code|dev_qpOkGUOxBpw6R16t",
    "authenticator_type": "recovery-code",
    "active": true
  },
  {
    "type": "otp",
    "id": "totp|dev_6NWz8awwC8brh2dN",
    "authenticator_type": "otp",
    "active": true
  }
]
```

Exija el desafío de MFA llamando al endpoint [`request/mfa/challenge`](/docs/es/api/authentication/muti-factor-authentication/request-mfa-challenge).

Personalice aún más el flujo de MFA con Auth0 Actions. Para obtener más información, lea [Triggers de Actions: post-challenge - objeto de API](/docs/es/customize/actions/explore-triggers/password-reset-triggers/post-challenge-trigger/post-challenge-api-object).
