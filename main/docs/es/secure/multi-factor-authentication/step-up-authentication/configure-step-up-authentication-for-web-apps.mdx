Con la autenticación escalonada, las aplicaciones que permiten acceder a distintos tipos de recursos pueden exigir que los usuarios se autentiquen con un mecanismo más sólido para acceder a información confidencial o realizar determinadas transacciones.

Por ejemplo, se puede permitir que un usuario acceda a vistas con datos confidenciales o restablezca su contraseña solo después de confirmar su identidad mediante <Tooltip tip="Autenticación multifactor (MFA): Proceso de autenticación de usuario que utiliza un factor además de nombre de usuario y contraseña, como un código por SMS." cta="Ver glosario" href="/docs/es/glossary?term=multi-factor+authentication">autenticación multifactor</Tooltip> (MFA).

Para implementar la autenticación escalonada en tu web app, crearás una Action que solicite al usuario autenticarse con MFA cuando la web app lo pida, comprobarás las claims del <Tooltip tip="Token de identidad: Credencial destinada al propio cliente, en lugar de usarse para acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=ID+Token">token de identidad</Tooltip> relacionadas con MFA si el usuario intenta acceder a una página restringida y, si MFA no está incluida en la claim, volverás a solicitar la autenticación del usuario.

<div id="validate-id-tokens-for-mfa">
  ## Validar tokens de ID para MFA
</div>

Cuando un usuario inicia sesión, recibes un [token de identidad](/docs/es/secure/tokens/id-tokens/get-id-tokens) que contiene información relevante sobre la sesión del usuario en forma de claims. El claim relevante es `amr` (referencia de métodos de autenticación), que es un arreglo JSON de strings que indica el método de autenticación utilizado durante el inicio de sesión. Debe estar presente en el payload del token de identidad y debe contener el valor `mfa`.

Sus valores pueden incluir cualquiera de los [Authentication Method Reference Values](https://tools.ietf.org/html/rfc8176) predefinidos. Como puede contener claims además de `mfa`, al validar debes comprobar tanto su existencia como revisar su contenido para verificar que incluya el valor `mfa`.

Si un usuario intenta acceder a una página restringida y el token muestra que no se autenticó con MFA, puedes volver a activar la autenticación, que has configurado para activar MFA mediante una Action. Una vez que el usuario proporcione el segundo factor, se generará un nuevo token de identidad que contiene el claim `amr` y se enviará a la aplicación.

1. [Obtén el token de identidad](/docs/es/secure/tokens/id-tokens/get-id-tokens).
2. Verifica la firma del token, que se utiliza para comprobar que el emisor del token es quien dice ser y para garantizar que el mensaje no se haya modificado en el camino.
3. Valida los siguientes claims:

| Claim | Descripción                                                                                                                                                                                                |
| ----- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `exp` | Expiración del token                                                                                                                                                                                       |
| `iss` | Emisor del token                                                                                                                                                                                           |
| `aud` | Destinatario previsto del token                                                                                                                                                                            |
| `amr` | Si `amr` no existe en el payload o no contiene el valor `mfa`, el usuario no inició sesión con MFA. Si `amr` existe en el payload y contiene el valor `mfa`, entonces el usuario sí inició sesión con MFA. |

<div id="amr-claim-exceptions">
  #### Excepciones del claim `amr`
</div>

El claim `amr` es obligatorio, excepto en los siguientes casos de uso:

1. En los flujos de inicio de sesión alojados, el claim `amr` solo se inyecta en el token de identidad después de que el usuario supere correctamente un desafío MFA. Si la aplicación usa autenticación silenciosa o tokens de actualización para emitir nuevos tokens de identidad, el claim `amr` no estará presente porque el usuario ya completó previamente el inicio de sesión con MFA.
2. Los tokens emitidos por la API de MFA no contienen el claim `amr`. El claim `amr` indica los métodos de autenticación utilizados cuando el usuario recibe el token de identidad. En el proceso de autenticación de la API de MFA, la aplicación controla el flujo de autenticación y puede exigir MFA según sea necesario.

En los ejemplos a continuación, puede comparar los posibles valores incluidos en el payload de un token de identidad cuando un usuario se ha autenticado con MFA frente a cuando no lo ha hecho.

<div id="scenario-salary-data-with-push-notifications">
  ## Escenario: datos salariales con notificaciones push
</div>

En el siguiente escenario, una aplicación web autentica a un usuario mediante nombre de usuario y contraseña. Algunos usuarios quieren acceder a una pantalla específica que muestra datos salariales, por lo que deben autenticarse con el factor push de Guardian.

<div id="prerequisites">
  ### Requisitos previos
</div>

Para este escenario, debe configurar los siguientes elementos en el Dashboard:

* [Registrar una aplicación web](/docs/es/get-started/auth0-overview/create-applications/regular-web-apps).
* [Crear una conexión de base de datos](https://manage.auth0.com/#/connections/database).
* [Habilitar MFA](/docs/es/secure/multi-factor-authentication/enable-mfa) para usar notificaciones push.

<div id="create-an-action">
  ### Crear una Action
</div>

Cree una Action que solicite al usuario autenticarse con MFA cuando la aplicación web lo requiera. Vaya a [Dashboard &gt; Actions &gt; Flows](https://manage.auth0.com/#/actions/flows) y cree una Action que contenga el siguiente contenido:

```js lines
exports.onExecutePostLogin = async (event, api) => {
 const CLIENTS_WITH_MFA = ['REPLACE_WITH_{yourClientId}'];
 // ejecutar solo para los clientes especificados
 if (CLIENTS_WITH_MFA.includes(event.client.client_id)) {
 // solicitar MFA solo si la aplicación web lo indicó en la solicitud de autenticación
 if (event.transaction?.acr_values.includes('http://schemas.openid.net/pape/policies/2007/06/multi-factor')) {
 api.multifactor.enable('any', { allowRememberBrowser: false });
 }
 }
}
```

* La variable `CLIENTS_WITH_MFA` contiene los <Tooltip tip="ID de cliente: valor de identificación asignado por Auth0 a tu recurso registrado." cta="Ver glosario" href="/docs/es/glossary?term=client+IDs">IDs de cliente</Tooltip> de las aplicaciones a las que quieres aplicar esta Action. Puedes eliminarla (y el condicional `if` que aparece a continuación) si no la necesitas.
* La propiedad `event.transaction.acr_values` es un arreglo de cadenas que contiene las referencias de clase del contexto de autenticación (`acr`). Es una propiedad opcional que solo existe cuando la aplicación la incluye en la solicitud de autenticación al <Tooltip tip="Servidor de autorización: servidor centralizado que ayuda a definir los límites del acceso de un usuario. Por ejemplo, tu servidor de autorización puede controlar los datos, las tareas y las funcionalidades disponibles para un usuario." cta="Ver glosario" href="/docs/es/glossary?term=Authorization+Server">servidor de autorización</Tooltip>. En este ejemplo, nuestra aplicación web la incluirá en la solicitud de autenticación, pero solo cuando un usuario que aún no se haya autenticado con MFA intente acceder a información salarial. Cuando nuestra aplicación web la incluya, establecerá el valor `http://schemas.openid.net/pape/policies/2007/06/multi-factor`, lo que indica que queremos que el servidor de autorización exija MFA, y el valor de la propiedad `api.multifactor` que configuramos en nuestro código hará que se solicite MFA al usuario mediante cualquiera de los métodos disponibles que se hayan configurado en el tenant. Para obtener más información sobre el método `api.multifactor.enable()`, consulta [Action Triggers: objeto de API de Post Login](/docs/es/customize/actions/explore-triggers/password-reset-triggers/post-challenge-trigger/post-challenge-api-object).
* La política `http://schemas.openid.net/pape/policies/2007/06/multi-factor` define un mecanismo de autenticación en el que el usuario final se autentica ante el <Tooltip tip="OpenID: estándar abierto de autenticación que permite a las aplicaciones verificar la identidad de los usuarios sin recopilar ni almacenar información de inicio de sesión." cta="Ver glosario" href="/docs/es/glossary?term=OpenID">OpenID</Tooltip> Provider proporcionando más de un factor de autenticación, o MFA. Para obtener más información, consulta [OpenID Provider Authentication Policy Extension 1.0](https://openid.net/specs/openid-provider-authentication-policy-extension-1_0.html).

<div id="configure-app">
  ### Configurar la app
</div>

Configura la app para comprobar que el usuario se haya autenticado con MFA cuando intente acceder a la página restringida de información salarial. (Cuando un usuario se ha autenticado con MFA, los claims del token de identidad contienen el claim `amr` con un valor de `mfa`.) Si el usuario ya se ha autenticado con MFA, la aplicación web mostrará la página restringida; de lo contrario, la aplicación web enviará una nueva solicitud de autenticación que incluye el parámetro `acr_values` con un valor de:
`http://schemas.openid.net/pape/policies/2007/06/multi-factor`lo que activará la Action.

La aplicación web en este escenario usa el [flujo de código de autorización](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow) para autenticarse, por lo que la solicitud es la siguiente:

export const codeExample3 = `https://{yourDomain}/authorize?
        audience=https://{yourDomain}/userinfo&
        scope=openid&
        response_type=code&
        client_id={yourClientId}&
        redirect_uri={https://yourApp/callback}&
        state={yourOpaqueValue}&
        acr_values=http://schemas.openid.net/pape/policies/2007/06/multi-factor`;

<AuthCodeBlock children={codeExample3} language="http" />

Una vez que el usuario se autentica con MFA, la aplicación web recibe el código de autorización, que debe intercambiarse por el nuevo token de identidad, que ahora debería contener el claim `amr` con un valor de `mfa`. Para obtener más información sobre cómo intercambiar el código por un token de identidad, consulta [Agregar inicio de sesión con el flujo de código de autorización](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow/add-login-auth-code-flow).

<div id="validate-id-token">
  ### Validar token de identidad
</div>

En este escenario, realiza las validaciones con el [código de ejemplo de JSON Web Token](https://github.com/auth0/node-jsonwebtoken), que verifica la firma del token (`jwt.verify`), decodifica el token, comprueba si el payload contiene `amr` y, si es así, registra los resultados en la consola.

export const codeExample4 = `const AUTH0_CLIENT_SECRET = '{yourClientSecret}';
const jwt = require('jsonwebtoken');

jwt.verify(id_token, AUTH0_CLIENT_SECRET, { algorithms: ['HS256'] }, function(err, decoded) {
  if (err) {
    console.log('token no válido');
    return;
  }

  if (Array.isArray(decoded.amr) && decoded.amr.indexOf('mfa') >= 0) {
    console.log('Has usado MFA');
    return;
  }

  console.log('No estás usando MFA');
});`;
