<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Estas personalizaciones no se aplican a Duo, que tiene su propia interfaz de usuario.
</Callout>

Puede personalizar las páginas de <Tooltip tip="Autenticación multifactor (MFA): proceso de autenticación de usuario que usa un factor además del nombre de usuario y la contraseña, como un código por SMS." cta="Ver glosario" href="/docs/es/glossary?term=MFA">MFA</Tooltip> que aparecen a sus usuarios ajustando las opciones de marca de <Tooltip tip="Autenticación multifactor (MFA): proceso de autenticación de usuario que usa un factor además del nombre de usuario y la contraseña, como un código por SMS." cta="Ver glosario" href="/docs/es/glossary?term=Universal+Login">Universal Login</Tooltip> en la sección [Universal Login Settings](https://manage.auth0.com/#/login_settings) del <Tooltip tip="Auth0 Dashboard: el producto principal de Auth0 para configurar sus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip>.

Si utiliza la experiencia de Universal Login, puede personalizar la página con sus opciones de personalización del flujo.

También puede personalizar todo el contenido HTML para reflejar los requisitos específicos de experiencia de usuario de su organización. Para obtener más información, consulte [opción de tema del MFA Widget](/docs/es/secure/multi-factor-authentication/customize-mfa/mfa-widget-theme-options).

Con la experiencia de Universal Login, MFA se presenta al usuario de forma aún más sencilla. Una vez que ingresan sus credenciales para iniciar sesión, se les muestra la pantalla de MFA. Si no tienen ningún factor de MFA registrado, se les pedirá que lo registren y, si ya tienen uno, se les pedirá que presenten su credencial de MFA.

<div id="use-the-mfa-api">
  ## Usa la API de MFA
</div>

Auth0 proporciona una [API de MFA](/docs/es/secure/multi-factor-authentication/multi-factor-authentication-developer-resources/mfa-api) para los siguientes casos:

* [Autenticar a los usuarios con la concesión de contraseña del propietario del recurso](/docs/es/secure/multi-factor-authentication/authenticate-using-ropg-flow-with-mfa).
* Crear una interfaz para [permitir que los usuarios administren sus propios factores de autenticación](/docs/es/secure/multi-factor-authentication/manage-mfa-auth0-apis/manage-authenticator-factors-mfa-api).
* Invitar a los usuarios a inscribirse en MFA mediante [tickets de inscripción](/docs/es/secure/multi-factor-authentication/multi-factor-authentication-developer-resources/create-custom-enrollment-tickets).

Para usar la API de MFA, debes habilitar el tipo de concesión MFA para tu aplicación. Para habilitar la concesión MFA en el Auth0 Dashboard:

1. Ve a [Dashboard &gt; Applications &gt; Applications](https://manage.auth0.com/#/applications) y selecciona tu aplicación.
2. Al final de la pestaña **Settings**, haz clic en **Advanced Settings**.
3. Haz clic en la pestaña **Grant Types** y selecciona **MFA**.

   <Frame>![Dashboard Applications Applications Advanced Settings pestaña Grant Types](/docs/images/cdy7uua7fh8z/3knG7flXhvUi0b2yhH0JEp/007e84fe56a9828ea59e9d06e0516d34/Screen_Shot_2021-05-18_at_2.24.26_PM.png)</Frame>

   Para obtener más información, consulta [Enable Multi-Factor Authentication](/docs/es/secure/multi-factor-authentication/enable-mfa).
4. Haz clic en **Save Changes**.

<div id="limitations">
  ### Limitaciones
</div>

La API de MFA está diseñada para funcionar con los factores SMS, notificaciones push dentro de la aplicación Guardian, correo electrónico y OTP. No permite la inscripción con Duo.

<div id="customize-mfa-policy-with-actions">
  ## Personaliza la política de MFA con Actions
</div>

Puedes usar Actions para personalizar la política de MFA de tu tenant, lo que incluye exigir MFA para usuarios de aplicaciones específicas, usuarios con metadatos de usuario o rangos de IP específicos, entre otros casos.

Si no estás familiarizado con la creación de Actions, lee [Write Your First Action](/docs/es/customize/actions/write-your-first-action).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Cualquier comportamiento relacionado con MFA definido en Actions tiene prioridad sobre la configuración de **Define policy** establecida en la sección **Multi-Factor Auth** del Dashboard.

  Para obtener más información, lee [Enable Multi-Factor Authentication](/docs/es/secure/multi-factor-authentication/enable-mfa).
</Callout>

Al escribir Actions, puedes usar la configuración `provider` para especificar si se debe forzar MFA y qué factor usar. El comportamiento varía según uses la experiencia Classic o Universal Login:

| Provider             | Experiencia Classic  | Nueva experiencia                                                                    |
| -------------------- | -------------------- | ------------------------------------------------------------------------------------ |
| any                  | Push, SMS u OTP      | Push, SMS, voz, OTP, Email o clave de seguridad de WebAuthn (cuando esté habilitada) |
| guardian             | Push, SMS u OTP      | Push, SMS, OTP o Email                                                               |
| google-authenticator | Google Authenticator | Push, SMS, OTP o Email                                                               |
| duo                  | Duo                  | Duo                                                                                  |

Si usas Universal Login, puedes obtener el comportamiento de la experiencia de Classic Login si habilitas la personalización de la página de inicio de sesión de MFA.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Las opciones `guardian` y `google-authenticator` son configuraciones heredadas que se mantienen por motivos de compatibilidad retroactiva y no deben usarse de ahora en adelante. Recomendamos usar `any` en su lugar. La opción `google-authenticator` no permite que los usuarios registren un recovery code.
</Callout>

Establecer `provider` en una opción específica anula manualmente los controles de activado/desactivado del Dashboard. Puedes usar la siguiente Action para pedirle al usuario que se registre en Duo incluso si hay otros factores habilitados en el Dashboard:

```js lines
exports.onExecutePostLogin = async (event, api) => {
    // Forzar el proveedor a Duo de forma programática
    api.multifactor.enable('duo')
}
```

<div id="require-mfa-for-specific-applications">
  ### Requerir MFA para aplicaciones específicas
</div>

Puedes personalizar la MFA para que solo se active cuando los usuarios inicien sesión en aplicaciones específicas.

```js lines
exports.onExecutePostLogin = async (event, api) => {
	const CLIENTS_WITH_MFA = [
		'{yourClientId1}', 
		'{yourClientId2}',
		'{yourClientId3}'
	];

	if (CLIENTS_WITH_MFA.includes(event.client.client_id)) {
		api.multifactor.enable('duo', { allowRememberBrowser: false })
	}
}
```

<div id="require-mfa-for-specific-users">
  ### Requerir MFA para usuarios específicos
</div>

Puedes personalizar MFA para que solo se active en los usuarios marcados para usar MFA.

```js lines
exports.onExecutePostLogin = async (event, api) => {
  if (event.user.user_metadata && event.user.user_metadata.use_mfa) {
    api.multifactor.enable('duo', { allowRememberBrowser: false });
  }
}
```

<div id="change-frequency-of-mfa-prompts">
  ### Cambiar la frecuencia de los prompts de MFA
</div>

En algunos casos, quizá no quieras pedirle al usuario MFA cada vez que inicia sesión desde el mismo navegador. Puedes cambiar ese comportamiento con la propiedad `allowRememberBrowser`:

```js lines
exports.onExecutePostLogin = async (event, api) => {  
  api.multifactor.enable('any', { allowRememberBrowser: false });
}
```

Según el valor de la propiedad, el comportamiento será el siguiente:

| Value   | Description                                                                                                                                                                                                                                                                                                                                                                                                                                                                |
| ------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `true`  | Cuando `provider` está configurado como `google-authenticator` o `duo`, se solicita MFA al usuario una vez cada 30 días. Cuando `provider` está configurado como `guardian`, el prompt de MFA muestra la casilla de inscripción para que los usuarios elijan si desean inscribirse o no. El valor predeterminado es `false`. Para obtener más información, consulta [Customize Multi-Factor Authentication Pages ](/docs/es/secure/multi-factor-authentication/customize-mfa) |
| `false` | Se solicita MFA al usuario cada vez que se autentica.                                                                                                                                                                                                                                                                                                                                                                                                                      |

Los valores de tiempo se aplican a los usuarios activos. Si un usuario permanece inactivo durante un período de siete días o más, su cookie caducará y se le solicitará MFA en su siguiente intento de inicio de sesión (independientemente de si `allowRememberBrowser` está configurado como `true` y de que no hayan pasado 30 días desde su último prompt de MFA).

Cuando permites que el usuario omita MFA, se almacena una cookie en el navegador del usuario. Si el usuario tiene la cookie establecida pero aun así quieres exigir MFA, tienes dos opciones:

* Configura `allowRememberBrowser` como `false`
* Configura `acr_values` como `http://schemas.openid.net/pape/policies/2007/06/multi-factor` al llamar al endpoint `/authorize`.

En situaciones en las que un usuario pierde un dispositivo de confianza, puedes solicitarle MFA a ese usuario específico durante su próximo inicio de sesión llamando al endpoint [Invalidate Remember Browser](https://auth0.com/docs/api/management/v2#!/Users/post_invalidate_remember_browser) de Auth0 <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip>.

<div id="require-mfa-for-users-outside-of-the-network">
  ### Exigir MFA a los usuarios fuera de la red
</div>

Si el acceso a la red especificada de direcciones IP internas está bien controlado, también puede hacer que Auth0 solicite MFA solo a los usuarios cuyas solicitudes provengan de fuera de la red corporativa:

```js lines
exports.onExecutePostLogin = async (event, api) => {
  const ipaddr = require('ipaddr.js@1.9.0');
  const corp_network = '192.168.1.134/26';
  const current_ip = ipaddr.parse(event.request.ip);

  if (!current_ip.match(ipaddr.parseCIDR(corp_network))) {
    api.multifactor.enable('any', {allowRememberBrowser: false});
  }
}
```

<div id="bypass-mfa-for-refresh-token-requests">
  ### Omitir MFA en las solicitudes de token de actualización
</div>

Si tus usuarios informan cierres de sesión frecuentes, es posible que debas omitir las solicitudes de <Tooltip tip="token de actualización: Token usado para obtener un token de acceso renovado sin obligar a los usuarios a volver a iniciar sesión." cta="Ver glosario" href="/docs/es/glossary?term=refresh+token">token de actualización</Tooltip> cuando usan MFA. Puedes modificar el comportamiento predeterminado con una Action de filtro en la propiedad `event.transaction.protocol` del objeto.

```js lines
exports.onExecutePostLogin = async (event, api) => {
  if (event.transaction.protocol !== 'oauth2-refresh-token') {
    // Comandos relacionados con MFA aquí
    api.multifactor.enable('any');
  }
};
```

Para obtener más información sobre el objeto `event` de Action, lea [Actions Triggers: Post Login - Objeto de evento](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger/post-login-event-object).
