<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Estas personalizaciones no se aplican a Duo, que tiene su propia interfaz de usuario.
</Callout>

Puedes personalizar las páginas de <Tooltip tip="Autenticación multifactor (MFA): proceso de autenticación del usuario que utiliza un factor además del nombre de usuario y la contraseña, como un código por SMS." cta="Ver glosario" href="/docs/es/glossary?term=MFA">MFA</Tooltip> que ven tus usuarios ajustando las opciones de marca de <Tooltip tip="Autenticación multifactor (MFA): proceso de autenticación del usuario que utiliza un factor además del nombre de usuario y la contraseña, como un código por SMS." cta="Ver glosario" href="/docs/es/glossary?term=Universal+Login">Universal Login</Tooltip> en la sección [Universal Login Settings](https://manage.auth0.com/#/login_settings) del <Tooltip tip="Auth0 Dashboard: el producto principal de Auth0 para configurar tus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip>.

Si usas la experiencia de Universal Login, puedes personalizar la página con sus opciones de personalización del flujo.

También puedes personalizar todo el contenido HTML para adaptarlo a los requisitos específicos de experiencia de usuario de tu organización. Para obtener más información, consulta [MFA Widget Theme Options](/docs/es/secure/multi-factor-authentication/customize-mfa/mfa-widget-theme-options).

Con la experiencia de Universal Login, MFA se presenta al usuario de una forma aún más sencilla. Una vez que haya introducido sus credenciales para iniciar sesión, se le mostrará la pantalla de MFA. Si no tiene ningún factor de MFA inscrito, se le pedirá que lo inscriba; si ya tiene uno, se le pedirá que presente su credencial de MFA.

<div id="use-the-mfa-api">
  ## Usa la API de MFA
</div>

Auth0 proporciona una [API de MFA](/docs/es/secure/multi-factor-authentication/multi-factor-authentication-developer-resources/mfa-api) para los siguientes casos:

* [Autenticar usuarios con la concesión de contraseña del propietario del recurso](/docs/es/secure/multi-factor-authentication/authenticate-using-ropg-flow-with-mfa).
* Crear una interfaz para [permitir que los usuarios administren sus propios factores de autenticación](/docs/es/secure/multi-factor-authentication/manage-mfa-auth0-apis/manage-authenticator-factors-mfa-api).
* Invitar a los usuarios a inscribirse en MFA mediante [tickets de inscripción](/docs/es/secure/multi-factor-authentication/multi-factor-authentication-developer-resources/create-custom-enrollment-tickets).

Para usar la API de MFA, debes habilitar el tipo de concesión MFA para tu aplicación. Para habilitar la concesión MFA en el Auth0 Dashboard:

1. Ve a [Dashboard &gt; Applications &gt; Applications](https://manage.auth0.com/#/applications) y selecciona tu aplicación.
2. En la parte inferior de la pestaña **Settings**, haz clic en **Advanced Settings**.
3. Haz clic en la pestaña **Grant Types** y selecciona **MFA**.

   <Frame>![Pestaña Grant Types de Advanced Settings en Dashboard Applications Applications](/docs/images/cdy7uua7fh8z/3knG7flXhvUi0b2yhH0JEp/007e84fe56a9828ea59e9d06e0516d34/Screen_Shot_2021-05-18_at_2.24.26_PM.png)</Frame>

   Para obtener más información, consulta [Habilitar la autenticación multifactor](/docs/es/secure/multi-factor-authentication/enable-mfa).
4. Haz clic en **Save Changes**.

<div id="limitations">
  ### Limitaciones
</div>

La API de MFA está diseñada para funcionar con SMS, notificaciones push en la aplicación Guardian, correo electrónico y factores OTP. No admite la inscripción con Duo.

<div id="customize-mfa-policy-with-actions">
  ## Personaliza la política de MFA con Actions
</div>

Puedes usar Actions para personalizar la política de MFA de tu tenant, por ejemplo, para exigir MFA a usuarios de aplicaciones específicas, a usuarios con determinada metadata de usuario o de rangos de IP concretos, entre otros casos.

Si no estás familiarizado con la creación de Actions, lee [Write Your First Action](/docs/es/customize/actions/write-your-first-action).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Cualquier comportamiento relacionado con MFA definido en Actions prevalece sobre la configuración de **Define policy** establecida en la sección **Multi-Factor Auth** del Dashboard.

  Para obtener más información, lee [Enable Multi-Factor Authentication](/docs/es/secure/multi-factor-authentication/enable-mfa).
</Callout>

Al escribir Actions, puedes usar la configuración `provider` para especificar si se debe forzar MFA y qué factor usar. El comportamiento varía según uses la experiencia Classic o la experiencia Universal Login:

| Provider             | Classic Experience   | New Experience                                                                      |
| -------------------- | -------------------- | ----------------------------------------------------------------------------------- |
| any                  | Push, SMS u OTP      | Push, SMS, Voice, OTP, Email o clave de seguridad WebAuthN (cuando está habilitada) |
| guardian             | Push, SMS u OTP      | Push, SMS, OTP o Email                                                              |
| google-authenticator | Google Authenticator | Push, SMS, OTP o Email                                                              |
| duo                  | Duo                  | Duo                                                                                 |

Si usas Universal Login, puedes obtener el comportamiento de la experiencia Classic Login si habilitas la personalización de la página de login de MFA.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Las opciones `guardian` y `google-authenticator` son configuraciones heredadas que se mantienen por motivos de compatibilidad con versiones anteriores y no deben seguir usándose. Te recomendamos usar `any` en su lugar. La opción `google-authenticator` no permite que los usuarios inscriban un código de recuperación.
</Callout>

Configurar `provider` con una opción específica anula manualmente los controles de habilitación del Dashboard. Puedes usar la siguiente Action para pedir al usuario que se inscriba en Duo incluso si hay otros factores habilitados en el Dashboard:

```js lines
exports.onExecutePostLogin = async (event, api) => {
    // Forzar el proveedor a Duo de forma programática
    api.multifactor.enable('duo')
}
```

  ## Ejemplos de casos de uso comunes
</div>

Aquí tienes algunos escenarios comunes de MFA contextual que pueden gestionarse con Actions.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Para asegurarte de que tu tenant active MFA solo cuando lo exijas explícitamente en tus Actions, establece la política de MFA predeterminada de tu tenant en **Never**.

  Para obtener más información, consulta la sección &quot;Políticas&quot; en [Factores de autenticación multifactor](/docs/es/secure/multi-factor-authentication/multi-factor-authentication-factors#policies).
</Callout>

<div id="require-mfa-for-specific-applications">
  ### Requerir MFA para aplicaciones específicas
</div>

Puedes configurar MFA para que solo se active cuando los usuarios inicien sesión en aplicaciones específicas.

```js lines
exports.onExecutePostLogin = async (event, api) => {
	const CLIENTS_WITH_MFA = [
		'{yourClientId1}', 
		'{yourClientId2}',
		'{yourClientId3}'
	];

	if (CLIENTS_WITH_MFA.includes(event.client.client_id)) {
		api.multifactor.enable('duo', { allowRememberBrowser: false })
	}
}
```

<div id="require-mfa-for-specific-users">
  ### Requerir MFA para usuarios específicos
</div>

Puede configurar MFA para que solo se active en los usuarios marcados para usar MFA.

```js lines
exports.onExecutePostLogin = async (event, api) => {
  if (event.user.user_metadata && event.user.user_metadata.use_mfa) {
    api.multifactor.enable('duo', { allowRememberBrowser: false });
  }
}
```

<div id="change-frequency-of-mfa-prompts">
  ### Cambiar la frecuencia de los prompts de MFA
</div>

En algunos casos, quizá no quieras solicitar MFA al usuario cada vez que inicie sesión desde el mismo navegador. Puedes modificar ese comportamiento con la propiedad `allowRememberBrowser`:

```js lines
exports.onExecutePostLogin = async (event, api) => {  
  api.multifactor.enable('any', { allowRememberBrowser: false });
}
```

Según el valor de la propiedad, el comportamiento será el siguiente:

| Valor   | Descripción                                                                                                                                                                                                                                                                                                                                                                                                                                                    |
| ------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `true`  | Cuando `provider` se establece en `google-authenticator` o `duo`, se solicita MFA al usuario una vez cada 30 días. Cuando `provider` se establece en `guardian`, el prompt de MFA muestra la casilla de inscripción para que los usuarios elijan si desean inscribirse o no. El valor predeterminado es `false`. Para obtener más información, consulta [Customize Multi-Factor Authentication Pages ](/docs/es/secure/multi-factor-authentication/customize-mfa) |
| `false` | Se solicita MFA al usuario cada vez que se autentica.                                                                                                                                                                                                                                                                                                                                                                                                          |

Estos valores de tiempo se aplican a usuarios activos. Si un usuario permanece inactivo durante un período de siete días o más, su cookie expirará y se le solicitará MFA en su siguiente intento de inicio de sesión (independientemente de si `allowRememberBrowser` está establecido en `true` y de que no hayan pasado 30 días desde su último prompt de MFA).

Cuando permites que el usuario omita MFA, se almacena una cookie en el navegador del usuario. Si el usuario tiene esa cookie, pero aun así quieres requerir MFA, tienes dos opciones:

* Establecer `allowRememberBrowser` en `false`
* Establecer `acr_values` en `http://schemas.openid.net/pape/policies/2007/06/multi-factor` al llamar al endpoint `/authorize`.

En las situaciones en que un usuario pierde un dispositivo de confianza, puedes solicitar MFA a ese usuario específico durante su próximo inicio de sesión llamando al endpoint [Invalidate Remember Browser](https://auth0.com/docs/api/management/v2#!/Users/post_invalidate_remember_browser) de Auth0 <Tooltip tip="Management API: Un producto para permitir que los clientes realicen tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip>.

<div id="require-mfa-for-users-outside-of-the-network">
  ### Exigir MFA a los usuarios que estén fuera de la red
</div>

Suponiendo que el acceso al conjunto especificado de direcciones IP internas esté bien controlado, también puede hacer que Auth0 solicite MFA solo a los usuarios cuyas solicitudes provengan de fuera de la red corporativa:

```js lines
exports.onExecutePostLogin = async (event, api) => {
  const ipaddr = require('ipaddr.js@1.9.0');
  const corp_network = '192.168.1.134/26';
  const current_ip = ipaddr.parse(event.request.ip);

  if (!current_ip.match(ipaddr.parseCIDR(corp_network))) {
    api.multifactor.enable('any', {allowRememberBrowser: false});
  }
}
```

<div id="bypass-mfa-for-refresh-token-requests">
  ### Omitir MFA para solicitudes de token de actualización
</div>

Si tus usuarios reportan cierres de sesión frecuentes, es posible que tengas que excluir de MFA las solicitudes de <Tooltip tip="Token de actualización: token usado para obtener un token de acceso renovado sin obligar a los usuarios a iniciar sesión de nuevo." cta="Ver glosario" href="/docs/es/glossary?term=refresh+token">token de actualización</Tooltip>. Puedes modificar el comportamiento predeterminado con una Action de filtro en la propiedad de objeto `event.transaction.protocol`.

```js lines
exports.onExecutePostLogin = async (event, api) => {
  if (event.transaction.protocol !== 'oauth2-refresh-token') {
    // Comandos relacionados con MFA aquí
    api.multifactor.enable('any');
  }
};
```

Para obtener más información sobre el objeto `event` de la Action, consulta [Triggers de Actions: Post Login - Objeto Event](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger/post-login-event-object).
