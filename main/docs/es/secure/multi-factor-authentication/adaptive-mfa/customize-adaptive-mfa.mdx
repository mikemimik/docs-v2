Puedes personalizar <Tooltip tip="Autenticación multifactor adaptativa: autenticación multifactor (MFA) que solo se activa para los usuarios cuando se determina que un intento de inicio de sesión es de baja confianza." cta="Ver glosario" href="/docs/es/glossary?term=Adaptive+MFA">MFA adaptativo</Tooltip> para distintos escenarios con [Auth0 Actions](/docs/es/customize/actions).

<div id="when-to-customize-adaptive-mfa">
  ## Cuándo personalizar MFA adaptativo
</div>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Solo deberías considerar personalizar MFA adaptativo si tus usuarios están inscritos en MFA y deben usar una dirección de correo electrónico como identificador.
</Callout>

Si tus usuarios no están inscritos en <Tooltip tip="Autenticación multifactor (MFA): proceso de autenticación de usuario que utiliza un factor adicional al nombre de usuario y la contraseña, como un código por SMS." cta="Ver glosario" href="/docs/es/glossary?term=MFA">MFA</Tooltip>, deberías usar la política predeterminada de MFA adaptativo. Si un usuario no está inscrito en MFA y tu Action evalúa un riesgo alto, tendrás opciones limitadas para detener a un <Tooltip tip="Actores maliciosos: entidad (una persona o grupo) que representa una amenaza para la empresa o el entorno con la intención de causar daño." cta="Ver glosario" href="/docs/es/glossary?term=bad+actor">actor malicioso</Tooltip>.

Antes de empezar a personalizar MFA adaptativo, hazte algunas preguntas:

* ¿Con qué nivel de confianza quieres activar MFA?
* ¿Cómo quieres medir el riesgo?
* ¿Quieres que Auth0 mida la confianza o prefieres una medición personalizada?
* ¿Cómo gestionarás a los usuarios que no están inscritos en MFA?

<div id="confidence-scores">
  ## Puntuaciones de confianza
</div>

MFA adaptativo calcula una puntuación de confianza general a partir del análisis de tres evaluaciones: `NewDevice`, `ImpossibleTravel` y `UntrustedIP`. Para obtener más información, consulta [MFA adaptativo: cómo funciona](/docs/es/secure/multi-factor-authentication/adaptive-mfa).

Cada evaluación tiene su propia puntuación de confianza, y cada puntuación de confianza tiene una acción asociada:

| Puntuación de confianza | Descripción                                                                                                                          | Acción                          |
| ----------------------- | ------------------------------------------------------------------------------------------------------------------------------------ | ------------------------------- |
| `low`                   | La transacción de inicio de sesión no coincide con patrones observados anteriormente en el comportamiento del usuario.               | Exigir MFA.                     |
| `medium`                | La transacción de inicio de sesión coincide en cierta medida con patrones observados anteriormente en el comportamiento del usuario. | No exigir MFA.                  |
| `high`                  | La transacción de inicio de sesión coincide estrechamente con patrones observados anteriormente en el comportamiento del usuario.    | No exigir MFA.                  |
| `neutral`               | N/A. Reservado para uso futuro.                                                                                                      | N/A. Reservado para uso futuro. |

<Accordion title="Ver ejemplos de escenarios de alto y bajo riesgo con diferentes puntuaciones de confianza.">
  La siguiente tabla describe escenarios de alto riesgo que dan como resultado una puntuación de confianza `low`:

  | Estado del usuario | Fricción de inicio de sesión deseada      | Política de inscripción deseada                                 | Implementación                                                           |
  | ------------------ | ----------------------------------------- | --------------------------------------------------------------- | ------------------------------------------------------------------------ |
  | Inscrito en MFA    | No exigir MFA                             | N/A (el usuario ya está inscrito)                               | Usar una Action para omitir MFA                                          |
  | No inscrito en MFA | Exigir verificación de correo electrónico | Omitir la inscripción (no recopilar autenticadores adicionales) | Comportamiento predeterminado (sin ninguna Action relacionada con MFA)   |
  | No inscrito en MFA | Exigir verificación de correo electrónico | Exigir inscripción en MFA (recopilar un autenticador adicional) | Usar una Action para forzar la inscripción en MFA (plantilla disponible) |

  La siguiente tabla describe escenarios de bajo riesgo que dan como resultado una puntuación de confianza `high`:

  | Estado del usuario | Fricción de inicio de sesión deseada | Política de inscripción deseada                                 | Implementación                                                           |
  | ------------------ | ------------------------------------ | --------------------------------------------------------------- | ------------------------------------------------------------------------ |
  | Inscrito en MFA    | Sin fricción                         | N/A (el usuario ya está inscrito)                               | Comportamiento predeterminado (sin ninguna Action relacionada con MFA)   |
  | No inscrito en MFA | Sin fricción                         | Omitir la inscripción (no recopilar autenticadores adicionales) | Comportamiento predeterminado (sin ninguna Action relacionada con MFA)   |
  | No inscrito en MFA | Sin fricción                         | Exigir inscripción en MFA (recopilar un autenticador adicional) | Usar una Action para forzar la inscripción en MFA (plantilla disponible) |
</Accordion>

Si quieres implementar tu propio método para evaluar la puntuación de confianza general en distintos escenarios, puedes usar los datos disponibles en el objeto `riskAssessment`, que contiene la puntuación de confianza general, información de control de versiones y detalles de las evaluaciones individuales.

Puedes ver la descripción completa, las Propiedades y los valores del objeto `riskAssessment` en la [referencia de `riskAssessment` del trigger de Actions Post Login](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger/post-login-event-object#param-risk-assessment).

<div id="action-result-outcomes">
  ## Posibles resultados de la Action
</div>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Las Actions que activan MFA tienen prioridad sobre el comportamiento predeterminado de MFA adaptativo.
</Callout>

Si alguna de tus Actions activa MFA en función de la puntuación de confianza, la política predeterminada de MFA adaptativo activa MFA cuando la puntuación de confianza es `low`.

La siguiente tabla muestra los posibles resultados según la combinación de las Actions y las acciones de la política predeterminada de MFA adaptativo.

| Resultado de la Action | Acción de MFA adaptativo | Resultado          |
| ---------------------- | ------------------------ | ------------------ |
| Unauthorized           | Activar MFA              | Unauthorized       |
| Unauthorized           | No se requiere MFA       | Unauthorized       |
| Activar MFA            | Activar MFA              | Activar MFA        |
| Activar MFA            | No se requiere MFA       | Activar MFA        |
| No se requiere MFA     | Activar MFA              | Activar MFA        |
| No se requiere MFA     | No se requiere MFA       | No se requiere MFA |

<div id="action-templates">
  ## Plantillas de Action
</div>

Auth0 proporciona dos plantillas de Action basadas en MFA adaptativo para que puedas personalizarlas: [MFA adaptativo](#adaptive-mfa-template) y [Requerir inscripción en MFA](#require-mfa-enrollment-template).

<div id="adaptive-mfa-template">
  ### Plantilla de MFA adaptativo
</div>

Esta plantilla ofrece un ejemplo y un punto de partida para crear un flujo de negocio personalizado mediante evaluaciones de riesgo individuales. Este ejemplo usa:

* El trigger de Action [`api.multifactor.enable`](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger/post-login-api-object#api-multifactor) para gestionar tanto la inscripción como el envío de los desafíos MFA configurados al final del flujo de inicio de sesión.
* El trigger de Actions [`event.user.multifactor`](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger/post-login-event-object) con los factores inscritos del usuario.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Como las notificaciones de `email` no son un factor independiente, la condición `event.user.multifactor && event.user.multifactor.length > 0` devolverá `false` si el usuario solo tiene `email` como factor. Para obtener más información, consulta [Configure Email Notifications for MFA](/docs/es/secure/multi-factor-authentication/multi-factor-authentication-factors/configure-email-notifications-for-mfa).
</Callout>

```javascript lines expandable
/**
* Manejador que será invocado durante la ejecución de un flujo Post Login.
*
* @param {Event} event - Detalles sobre el usuario y el contexto en el que está iniciando sesión.
* @param {PostLoginAPI} api - Interfaz cuyos métodos pueden usarse para modificar el comportamiento del login.
*/
exports.onExecutePostLogin = async (event, api) => {
    // Define qué niveles de puntuación de confianza deben activar MFA; para más información, consulta
    const promptConfidences = ['low', 'medium'];

    // Condición de ejemplo: solicitar MFA únicamente según el nivel de confidence
    // de NewDevice; esto solicitará MFA cuando un usuario inicie sesión
    // desde un dispositivo desconocido.
    const confidence =
        event.authentication?.riskAssessment?.assessments?.NewDevice
            ?.confidence;
    const shouldPromptMfa =
        confidence && promptConfidences.includes(confidence);

    // Solo tiene sentido solicitar MFA cuando el usuario tiene al menos un
    // factor de MFA registrado.
    const canPromptMfa =
        event.user.multifactor && event.user.multifactor.length > 0;
    if (shouldPromptMfa && canPromptMfa) {
        api.multifactor.enable('any', { allowRememberBrowser: true });
    }
};
```

Para solicitar la verificación a los usuarios, reemplaza `api.multifactor.enable` por `api.authentication.challengeWithAny()` para forzar un desafío MFA con un factor existente que el usuario ya haya registrado. Para revisar los factores compatibles con Actions, consulta el [parámetro `factors`](https://auth0.com/docs/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger/post-login-api-object#api-authentication-challengewithany-factors). Por ejemplo:

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Este ejemplo usa `event.user.enrolledFactors` en lugar de `event.user.multifactor` para comprobar el registro. A diferencia de `event.user.multifactor`, `event.user.enrolledFactors` incluye `email` como factor, por lo que devuelve correctamente los factores registrados de los usuarios que solo tienen configurado el correo electrónico.
</Callout>

```javascript
// Solo tiene sentido solicitar MFA cuando el usuario tiene al menos un
// factor de MFA registrado.
const canPromptMfa = event.user.enrolledFactors && event.user.enrolledFactors.length > 0;

if (shouldPromptMfa && canPromptMfa) {
    api.authentication.challengeWithAny([ {type: "email"}, {type:"phone"} ]);
}
```

<div id="require-mfa-enrollment-template">
  ### Plantilla Requerir inscripción en MFA
</div>

Esta plantilla muestra cómo puedes exigir la inscripción en MFA al usar una política estándar o de MFA adaptativo. Usa `event.user.multifactor` para comprobar si el usuario está inscrito en MFA y, si no lo está, le solicita que se inscriba.

```javascript lines
/**
* Manejador que será invocado durante la ejecución de un flujo Post Login.
*
* @param {Event} event - Detalles sobre el usuario y el contexto en el que está iniciando sesión.
* @param {PostLoginAPI} api - Interfaz cuyos métodos pueden usarse para modificar el comportamiento del inicio de sesión.
*/
exports.onExecutePostLogin = async (event, api) => {
    if (!event.user.multifactor?.length) {
        api.multifactor.enable('any', { allowRememberBrowser: false });
    }
};
```

<div id="action-use-cases">
  ## Casos de uso de Action
</div>

Aquí tienes algunas sugerencias para crear Actions personalizadas según tu caso de uso.

<AccordionGroup>
  <Accordion title="Realizar una acción si la puntuación de confianza general es X">
    Evalúa la propiedad `riskAssessment.confidence` y luego compárala con las constantes `high`, `medium` o `low`:

    ```js lines
    exports.onExecutePostLogin = async (event, api) => {
      const { riskAssessment } = event.authentication || {};
      const riskIsMedium = riskAssessment && riskAssessment.confidence === 'medium';

      if (riskIsMedium) {
        // ....
      }
    }
    ```
  </Accordion>

  <Accordion title="Realizar una acción si la puntuación de confianza es superior o inferior a X">
    Las puntuaciones de confianza son valores discretos, no un rango, por lo que no puedes usar operadores de comparación (como `<` o `>`) para evaluar varios valores en una sola condición.

    Usa varias condiciones para combinar lógicamente todas las puntuaciones de confianza que quieras contemplar. Por ejemplo, si quieres saber cuándo la puntuación de confianza es mayor que `low`, comprueba si es igual a `medium` o `high`:

    ```js lines
    exports.onExecutePostLogin = async (event, api) => {
      const { riskAssessment } = event.authentication || {};
      const riskIsMediumOrHigh = riskAssessment && 
                                      (riskAssessment.confidence === 'high' || 
                                       riskAssessment.confidence === 'medium');

      if (riskIsMediumOrHigh) {
        // ...
      }
    }
    ```
  </Accordion>

  <Accordion title="Obtener detalles adicionales si la puntuación de confianza general es X">
    El objeto `riskAssessment` se guarda en los registros del tenant. Puedes consultar las entradas de registro para ver la puntuación de la evaluación de riesgo y los factores determinantes (motivos).

    También puedes ver el objeto `riskAssessment` y notificar los resultados en otro lugar. Por ejemplo, puedes enviar un correo electrónico o guardar un registro en una base de datos externa.

    ```js lines
    exports.onExecutePostLogin = async (event, api) => {
      const { riskAssessment } = event.authentication || {};
      const riskIsLow = riskAssessment && riskAssessment.confidence === 'low';

      if (riskIsLow) {
        // log(externalDatabase, riskAssessment);
      }
    }
    ```
  </Accordion>

  <Accordion title="Realizar una acción si una evaluación específica tiene un resultado específico">
    Usa el objeto [`assessments`](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger/post-login-event-object#param-assessments) para acceder a los detalles de evaluaciones individuales, incluida la propiedad `code`:

    ```js lines
    exports.onExecutePostLogin = async (event, api) => {
      const { riskAssessment } = event.authentication || {};
      const { ImpossibleTravel } = riskAssessment && riskAssessment.assessments;

      if (ImpossibleTravel.code === 'impossible_travel_from_last_login') {
        // ...
      }
    }
    ```
  </Accordion>

  <Accordion title="Agregar evaluaciones para obtener una puntuación de confianza general personalizada.">
    Usa el objeto [`assessments`](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger/post-login-event-object#param-assessments) para acceder a los detalles de evaluaciones individuales y luego usa la propiedad `confidence`, la propiedad `code` o ambas.
  </Accordion>

  <Accordion title="Bloquear la transacción actual y devolver un error y un mensaje si una evaluación específica tiene un resultado específico">
    Usa el objeto [`assessments`](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger/post-login-event-object#param-assessments) para acceder a los detalles de evaluaciones individuales, incluida la propiedad `code`.

    Impide que se complete la transacción de login devolviendo la función callback con un objeto `UnauthorizedError` como parámetro de error. El objeto `UnauthorizedError` siempre establece `error` en `unauthorized`, pero puedes personalizar `error_message`:

    ```js lines
    exports.onExecutePostLogin = async (event, api) => {
      const { riskAssessment } = event.authentication || {};
      const { ImpossibleTravel } = riskAssessment && riskAssessment.assessments;

      if (ImpossibleTravel.code === 'impossible_travel_from_last_login') {
        return api.access.deny('Login blocked due to impossible travel detected.')
      }
    }
    ```

    Esto redirige al usuario de vuelta a la URL de devolución de llamada de la aplicación con los parámetros `error` y `error_message` incluidos.
  </Accordion>

  <Accordion title="Gestionar de forma segura los casos en los que Auth0 no consigue ejecutar evaluaciones">
    Auth0 asigna automáticamente una puntuación de confianza `low` si se produce algún tipo de fallo al realizar la evaluación de riesgo.

    Para mitigar este caso, usa el objeto [`assessments`](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger/post-login-event-object#param-assessments) para inspeccionar la propiedad `code` de cada evaluación individual y comprobar si el valor está establecido en `assessment_not_available`.
  </Accordion>
</AccordionGroup>
