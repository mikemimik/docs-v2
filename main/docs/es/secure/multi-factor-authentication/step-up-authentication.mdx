title: Agregar autenticación escalonada
---

Con la autenticación escalonada, las aplicaciones que dan acceso a distintos tipos de recursos pueden exigir a los usuarios que se autentiquen con un mecanismo de autenticación más robusto para acceder a recursos sensibles.

Puedes agregar autenticación escalonada a tu aplicación con la compatibilidad extensible de Auth0 con la <Tooltip tip="Autenticación multifactor (MFA): proceso de autenticación de usuario que utiliza un factor además de nombre de usuario y contraseña, como un código por SMS." cta="Ver glosario" href="/docs/es/glossary?term=multi-factor+authentication">autenticación multifactor</Tooltip> (MFA). Tu aplicación puede verificar que el usuario haya iniciado sesión con MFA y, de no ser así, exigirle una autenticación reforzada para acceder a determinados recursos.

<div id="how-it-works">
  ## Cómo funciona
</div>

Por ejemplo, la intranet de Fabrikam exige que los usuarios se autentiquen con su nombre de usuario y contraseña para acceder a los datos de clientes. Sin embargo, una solicitud de acceso a datos de empleados (que pueden contener información salarial confidencial) activa un mecanismo de autenticación más robusto, como MFA.

<div id="step-up-authentication-for-apis">
  ## Autenticación escalonada para APIs
</div>

Cuando tu <Tooltip tip="Audiencia: Identificador único de la audiencia de un token emitido. Llamado aud en un token, su valor contiene el ID de una aplicación (ID de cliente) para un token de ID o de una API (identificador de API) para un token de acceso." cta="Ver glosario" href="/docs/es/glossary?term=audience">audiencia</Tooltip> es una API, puedes implementar autenticación escalonada con Auth0 usando [scopes](/docs/es/get-started/apis/scopes), [tokens de acceso](/docs/es/secure/tokens/access-tokens) y [Actions](/docs/es/customize/actions). Puedes usar una Action para activar el mecanismo de autenticación escalonada (por ejemplo, solicitar MFA) cada vez que el usuario solicite scopes asociados a recursos sensibles.

En nuestro ejemplo, un usuario inicia sesión en la aplicación web de Fabrikam. El inicio de sesión estándar le permite interactuar con su API y obtener la lista de cuentas del usuario. Esto significa que el <Tooltip tip="Token de acceso: Credencial de autorización, en forma de una cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=access+token">token de acceso</Tooltip> que recibe la aplicación después de la autenticación de usuario contiene un scope como `read:accounts`.

Ahora el usuario quiere transferir fondos de una cuenta a otra, lo que se considera una transacción de alto valor. Para realizar esta acción, la API requiere el scope `transfer:funds`.

El token de acceso actual del usuario no incluye este scope y la aplicación lo sabe (porque conoce el conjunto de scopes que solicitó en la llamada de autenticación inicial). La aplicación realiza otra llamada de autenticación, pero esta vez solicita el scope `transfer:funds`. El navegador redirige al usuario a Auth0. De acuerdo con la Action de Fabrikam, Auth0 le solicita al usuario que se autentique con MFA porque se pidió un scope de alto valor. Una vez que el usuario se autentica correctamente con MFA, Auth0 genera y envía un nuevo token de acceso que incluye ese scope de alto valor. La aplicación pasa el token de acceso a la API, que lo descarta después de verificarlo y, de ese modo, lo trata como un token de un solo uso.

Para obtener más información, lee [Configurar la autenticación escalonada para APIs](/docs/es/secure/multi-factor-authentication/step-up-authentication/configure-step-up-authentication-for-apis).

<div id="step-up-authentication-for-web-apps">
  ## Autenticación escalonada para aplicaciones web
</div>

Si es una aplicación web la que verifica el nivel de autenticación, y no una API, entonces no tienes un token de acceso. En este caso, puedes comprobar si un usuario ha iniciado sesión con MFA revisando el contenido de su <Tooltip tip="token de ID: Credencial destinada al propio cliente, en lugar de usarse para acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=ID+token">token de identidad</Tooltip>. Luego, puedes configurar tu aplicación para denegar el acceso a páginas con información sensible si el token de identidad indica que el usuario no inició sesión con MFA, y usar una Action para activar el mecanismo de autenticación escalonada (por ejemplo, el prompt de MFA). Por ejemplo, podrías tener una aplicación para empleados que autentica a los usuarios con nombres de usuario y contraseñas, pero si un usuario quiere acceder a información salarial, tiene que proporcionar un segundo factor, como una notificación push en el móvil.

Puedes implementar esto comprobando el token de identidad cuando el usuario intenta acceder a esa página. Si las claims muestran que el usuario ya se autenticó con MFA, entonces muestra la información sensible. De lo contrario, vuelve a activar la autenticación y, usando una Action, solicita al usuario que se autentique con MFA.

Para obtener más información, lee [Configurar la autenticación escalonada para aplicaciones web](/docs/es/secure/multi-factor-authentication/step-up-authentication/configure-step-up-authentication-for-web-apps).
