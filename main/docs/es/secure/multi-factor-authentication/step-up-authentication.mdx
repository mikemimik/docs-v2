title: Agregar autenticación escalonada
---

Con la autenticación escalonada, las aplicaciones que permiten acceder a distintos tipos de recursos pueden exigir a los usuarios que se autentiquen con un mecanismo de autenticación más sólido para acceder a recursos sensibles.

Puedes agregar autenticación escalonada a tu aplicación con el soporte extensible de <Tooltip tip="Autenticación multifactor (MFA): Proceso de autenticación de usuario que utiliza un factor además de nombre de usuario y contraseña, como un código por SMS." cta="Ver glosario" href="/docs/es/glossary?term=multi-factor+authentication">autenticación multifactor</Tooltip> (MFA) de Auth0. Tu aplicación puede verificar que el usuario haya iniciado sesión con MFA y, de no ser así, exigirle una autenticación de mayor nivel para acceder a determinados recursos.

<div id="how-it-works">
  ## Cómo funciona
</div>

Por ejemplo, la intranet de Fabrikam exige que los usuarios se autentiquen con su nombre de usuario y contraseña para acceder a los datos de clientes. Sin embargo, una solicitud de acceso a datos de empleados (que pueden incluir información salarial confidencial) activa un mecanismo de autenticación más sólido, como MFA.

<div id="step-up-authentication-for-apis">
  ## Autenticación escalonada para APIs
</div>

Cuando tu <Tooltip tip="Audience: Identificador único de la audiencia de un token emitido. Se denomina aud en un token, y su valor contiene el ID de una aplicación (ID de cliente) para un token de ID o de una API (identificador de API) para un token de acceso." cta="Ver glosario" href="/docs/es/glossary?term=audience">audiencia</Tooltip> es una API, puedes implementar autenticación escalonada con Auth0 mediante [scopes](/docs/es/get-started/apis/scopes), [tokens de acceso](/docs/es/secure/tokens/access-tokens) y [Actions](/docs/es/customize/actions). Puedes usar una Action para activar el mecanismo de autenticación escalonada (por ejemplo, solicitar MFA) cada vez que el usuario solicite scopes que correspondan a recursos sensibles.

En nuestro ejemplo, un usuario inicia sesión en la aplicación web de Fabrikam. El inicio de sesión estándar permite a este usuario interactuar con su API y recuperar la lista de cuentas. Esto significa que el <Tooltip tip="Access Token: Credencial de autorización, en forma de una cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=access+token">token de acceso</Tooltip> que la aplicación recibe después de la autenticación del usuario contiene un scope como `read:accounts`.

Ahora el usuario quiere transferir fondos de una cuenta a otra, lo que se considera una transacción de alto valor. Para realizar esta acción, la API requiere el scope `transfer:funds`.

El token de acceso actual del usuario no incluye este scope y la aplicación lo sabe (porque conoce el conjunto de scopes que solicitó en la llamada de autenticación inicial). La aplicación realiza otra llamada de autenticación, pero esta vez solicita el scope `transfer:funds`. El navegador se redirige a Auth0. Según la Action de Fabrikam, Auth0 le pide al usuario que se autentique con MFA porque se solicitó un scope de alto valor. Una vez que el usuario se autentica correctamente con MFA, Auth0 genera y envía un nuevo token de acceso que incluye ese scope de alto valor. La aplicación pasa el token de acceso a la API, que lo descarta después de verificarlo, tratándolo así como un token de un solo uso.

Para obtener más información, consulta [Configurar la autenticación escalonada para APIs](/docs/es/secure/multi-factor-authentication/step-up-authentication/configure-step-up-authentication-for-apis).

<div id="step-up-authentication-for-web-apps">
  ## Autenticación escalonada para aplicaciones web
</div>

Si se trata de una aplicación web que verifica el nivel de autenticación, y no de una API, entonces no dispone de un token de acceso. En este caso, puede comprobar si un usuario ha iniciado sesión con MFA revisando el contenido de su <Tooltip tip="token de ID: Credencial destinada al propio cliente, en lugar de utilizarse para acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=ID+token">token de identidad</Tooltip>. A continuación, puede configurar su aplicación para denegar el acceso a páginas con información confidencial si el token de identidad indica que el usuario no inició sesión con MFA, y usar una Action para activar el mecanismo de autenticación escalonada (por ejemplo, prompt MFA). Por ejemplo, podría tener una aplicación para empleados que autentica a los usuarios con nombres de usuario y contraseñas, pero, si un usuario quiere acceder a información salarial, debe proporcionar un segundo factor, como una notificación push en el móvil.

Puede implementar esto comprobando el token de identidad cuando el usuario intente acceder a esa página. Si los claims muestran que el usuario ya se ha autenticado con MFA, muestre la información confidencial. De lo contrario, vuelva a activar la autenticación y, mediante una Action, solicite al usuario que se autentique con MFA.

Para obtener más información, lea [Configurar la autenticación escalonada para aplicaciones web](/docs/es/secure/multi-factor-authentication/step-up-authentication/configure-step-up-authentication-for-web-apps).
