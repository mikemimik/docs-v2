<Card title="Antes de empezar">
  * Habilita el **tipo de concesión** MFA para tu aplicación. Para obtener más información, consulta [Update Grant Types](/docs/es/get-started/applications/update-grant-types).
</Card>

Auth0 proporciona varios [endpoints de la API](https://auth0.com/docs/api/authentication#multi-factor-authentication) para ayudarte a gestionar los autenticadores que utilizas con una aplicación para la <Tooltip tip="Autenticación multifactor (MFA): Proceso de autenticación de usuario que utiliza un factor adicional al nombre de usuario y la contraseña, como un código por SMS." cta="Ver glosario" href="/docs/es/glossary?term=multi-factor+authentication">autenticación multifactor</Tooltip> (MFA). Puedes usar estos endpoints para crear una interfaz de usuario completa que permita a los usuarios gestionar sus factores de autenticación.

<div id="get-mfa-api-access-tokens">
  ## Obtener tokens de acceso de la API de MFA
</div>

Para llamar a la API de MFA y gestionar inscripciones, primero debes obtener un <Tooltip tip="token de acceso: Credencial de autorización, en forma de una cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=access+token">token de acceso</Tooltip> para la API de MFA.

Para usar la API de MFA como parte de un flujo de autenticación, puedes seguir los pasos detallados en [Autenticar con contraseña del propietario del recurso y MFA](/docs/es/secure/multi-factor-authentication/authenticate-using-ropg-flow-with-mfa). Si estás creando una interfaz de usuario para gestionar factores de autenticación, necesitarás obtener un token que puedas usar con la API de MFA en cualquier momento, no solo durante la autenticación.

Un token de acceso de MFA, o un token de acceso con la <Tooltip tip="Audience: Identificador único del destinatario de un token emitido. Se denomina aud en un token; su valor contiene el ID de una aplicación (ID de cliente) para un token de ID o de una API (API Identifier) para un token de acceso." cta="Ver glosario" href="/docs/es/glossary?term=audience">audience</Tooltip> `https://{yourDomain}/mfa/`, autentica al mismo tiempo que inicia una [solicitud de desafío](https://auth0.com/docs/api/authentication/muti-factor-authentication/request-mfa-challenge) MFA. La habilitación de factores en tu tenant y la inscripción de tus usuarios determinan si se inicia el desafío MFA:

* Si tu tenant permite la inscripción en varios factores y tus usuarios se inscriben en un factor válido (excepto email), como una contraseña de un solo uso (OTP), el usuario recibe el desafío MFA al iniciar sesión.
* Si tus usuarios no están inscritos en ningún factor de autenticación, o solo están inscritos con el factor `email`, se emite el token de MFA y no se envía ningún desafío al usuario.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  El tiempo de expiración predeterminado de los tokens de acceso con la audience `https://{yourDomain}/mfa/*` es de 10 minutos. Este valor no se puede configurar.
</Callout>

<div id="universal-login">
  ### Universal Login
</div>

Si usas [Universal Login](/docs/es/authenticate/login/auth0-universal-login), redirige al endpoint Authorize y especifica la audience `https://{yourDomain}/mfa/`.

<Warning>
  Cuando se especifica `https://{yourDomain}/mfa/` como audience, se exige MFA. Si los usuarios finales activan **Remember this browser** mientras `.../mfa/` se especifica como audience, esta opción no surtirá efecto.

  Auth0 recomienda que los administradores del tenant [creen una Action](/docs/es/customize/actions/write-your-first-action) que establezca `allowRememberBrowser` en false. Esto ocultará **Remember this browser** en la experiencia del usuario final.
</Warning>

<div id="resource-owner-password-grant">
  ### Concesión de contraseña del propietario del recurso
</div>

Si estás usando la <Tooltip tip="Propietario del recurso: entidad (como un usuario o una aplicación) capaz de conceder acceso a un recurso protegido." cta="Ver glosario" href="/docs/es/glossary?term=Resource+Owner">contraseña del propietario del recurso</Tooltip> Grant (ROPG), tienes 3 opciones:

* Solicita la `audience` `https://{yourDomain}/mfa/` al iniciar sesión y usa un [token de actualización](/docs/es/secure/tokens/refresh-tokens) para renovarlo más adelante.
* Si necesitas listar y eliminar autenticadores, pídele al usuario que [se autentique de nuevo](/docs/es/secure/multi-factor-authentication/authenticate-using-ropg-flow-with-mfa) con `/oauth/token`, especificando la `audience` `https://{yourDomain}/mfa/`. Los usuarios deberán completar MFA antes de poder listar o eliminar factores de autenticación.
* Si solo necesitas listar autenticadores, pídele al usuario que [se autentique de nuevo](/docs/es/secure/multi-factor-authentication/authenticate-using-ropg-flow-with-mfa) mediante `/oauth/token` con nombre de usuario y contraseña. El endpoint devolverá un error `mfa_required` y un `mfa_token` que puedes usar para listar autenticadores. Los usuarios deberán proporcionar su contraseña para ver sus autenticadores.

<div id="scopes">
  ### Scopes
</div>

Al solicitar un token para la audience de MFA, puedes solicitar los siguientes scopes:

| Scope                   | Descripción                                |
| ----------------------- | ------------------------------------------ |
| `enroll`                | Para registrar un nuevo autenticador.      |
| `read:authenticators`   | Para listar los autenticadores existentes. |
| `remove:authenticators` | Para eliminar un autenticador.             |

<div id="list-authenticators">
  ## Listar autenticadores
</div>

Para obtener una lista de los autenticadores de un usuario, puedes llamar al endpoint de autenticadores de MFA:

<AuthCodeGroup>
  ```bash cURL
  curl --request GET \
    --url 'https://{yourDomain}/mfa/authenticators' \
    --header 'authorization: Bearer MFA_TOKEN'
  ```

  ```csharp C#
  var client = new RestClient("https://{yourDomain}/mfa/authenticators");
  var request = new RestRequest(Method.GET);
  request.AddHeader("authorization", "Bearer MFA_TOKEN");
  IRestResponse response = client.Execute(request);
  ```

  ```go Go
  package main

  import (
  	"fmt"
  	"net/http"
  	"io/ioutil"
  )

  func main() {

  	url := "https://{yourDomain}/mfa/authenticators"

  	req, _ := http.NewRequest("GET", url, nil)

  	req.Header.Add("authorization", "Bearer MFA_TOKEN")

  	res, _ := http.DefaultClient.Do(req)

  	defer res.Body.Close()
  	body, _ := ioutil.ReadAll(res.Body)

  	fmt.Println(res)
  	fmt.Println(string(body))

  }
  ```

  ```java Java
  HttpResponse<String> response = Unirest.get("https://{yourDomain}/mfa/authenticators")
    .header("authorization", "Bearer MFA_TOKEN")
    .asString();
  ```

  ```javascript Node.JS
  var axios = require("axios").default;

  var options = {
    method: 'GET',
    url: 'https://{yourDomain}/mfa/authenticators',
    headers: {authorization: 'Bearer MFA_TOKEN'}
  };

  axios.request(options).then(function (response) {
    console.log(response.data);
  }).catch(function (error) {
    console.error(error);
  });
  ```

  ```php PHP
  $curl = curl_init();

  curl_setopt_array($curl, [
    CURLOPT_URL => "https://{yourDomain}/mfa/authenticators",
    CURLOPT_RETURNTRANSFER => true,
    CURLOPT_ENCODING => "",
    CURLOPT_MAXREDIRS => 10,
    CURLOPT_TIMEOUT => 30,
    CURLOPT_HTTP_VERSION => CURL_HTTP_VERSION_1_1,
    CURLOPT_CUSTOMREQUEST => "GET",
    CURLOPT_HTTPHEADER => [
      "authorization: Bearer MFA_TOKEN"
    ],
  ]);

  $response = curl_exec($curl);
  $err = curl_error($curl);

  curl_close($curl);

  if ($err) {
    echo "cURL Error #:" . $err;
  } else {
    echo $response;
  }
  ```

  ```python Python
  import http.client

  conn = http.client.HTTPSConnection("")

  headers = { 'authorization': "Bearer MFA_TOKEN" }

  conn.request("GET", "/{yourDomain}/mfa/authenticators", headers=headers)

  res = conn.getresponse()
  data = res.read()

  print(data.decode("utf-8"))
  ```

  ```ruby Ruby
  require 'uri'
  require 'net/http'
  require 'openssl'

  url = URI("https://{yourDomain}/mfa/authenticators")

  http = Net::HTTP.new(url.host, url.port)
  http.use_ssl = true
  http.verify_mode = OpenSSL::SSL::VERIFY_NONE

  request = Net::HTTP::Get.new(url)
  request["authorization"] = 'Bearer MFA_TOKEN'

  response = http.request(request)
  puts response.read_body
  ```
</AuthCodeGroup>

En la respuesta deberías recibir información sobre el tipo o los tipos de autenticador:

```json lines
[
  {
    "authenticator_type": "recovery-code",
    "id": "recovery-code|dev_IsBj5j3H12VAdOIj",
    "active": true
  },
  {
    "authenticator_type": "otp",
    "id": "totp|dev_nELLU4PFUiTW6iWs",
    "active": true,
  },
  {
    "authenticator_type": "oob",
    "oob_channel": "sms",
    "id": "sms|dev_sEe99pcpN0xp0yOO",
    "name": "+1123XXXXX",
    "active": true
  }
]
```

A la hora de crear una interfaz de usuario para que los usuarios finales administren sus factores, debes ignorar los autenticadores cuyo valor de `active` sea `false`. Esos autenticadores no han sido confirmados por los usuarios, por lo que no pueden usarse para el desafío de MFA.

La API de MFA mostrará los siguientes registros de inscripción según el tipo de autenticador:

| Autenticador   | Acciones                                                                                                                                                                                      |
| -------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Push y OTP** | Si push está habilitado, Auth0 también crea una inscripción de OTP. Verás ambas al listar las inscripciones.                                                                                  |
| **SMS y voz**  | Si tanto SMS como voz están habilitados, cuando un usuario se inscribe con SMS o voz, Auth0 crea automáticamente dos autenticadores para el número de teléfono: uno para SMS y otro para voz. |
| **Email**      | Todos los correos electrónicos verificados aparecerán como autenticadores.                                                                                                                    |

<div id="delete-authenticators">
  ## Eliminar autenticadores
</div>

Para eliminar un autenticador asociado, envía una solicitud `DELETE` al endpoint de autenticadores de MFA y reemplaza `AUTHENTICATOR_ID` por el ID del autenticador correspondiente. Puedes obtener el ID al listar los autenticadores.

Si se usó un `mfa_token` para listar autenticadores, los usuarios deberán [completar MFA para obtener un token de acceso](https://auth0.com/docs/api/authentication/muti-factor-authentication/verify-mfa-with-otp) con una audience de `https://{yourDomain}/mfa/` para poder eliminar un autenticador.

<AuthCodeGroup>
  ```bash cURL
  curl --request DELETE \
    --url 'https://{yourDomain}/mfa/authenticators/AUTHENTICATOR_ID' \
    --header 'authorization: Bearer ACCESS_TOKEN'
  ```

  ```csharp C#
  var client = new RestClient("https://{yourDomain}/mfa/authenticators/AUTHENTICATOR_ID");
  var request = new RestRequest(Method.DELETE);
  request.AddHeader("authorization", "Bearer ACCESS_TOKEN");
  IRestResponse response = client.Execute(request);
  ```

</AuthCodeGroup>

Si el autenticador se eliminó, se devuelve una respuesta 204.

Cuando eliminas un autenticador, se producen las siguientes acciones según el tipo de autenticador:

| Autenticador   | Acción                                                                                                                                                                                               |
| -------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Push y OTP** | Cuando un usuario inscribe un autenticador push, Auth0 también inscribe OTP. Si eliminas cualquiera de los dos, el otro también se eliminará.                                                        |
| **SMS y voz**  | Cuando un usuario se inscribe en SMS o voz, Auth0 crea dos autenticadores: SMS y voz. Si eliminas cualquiera de los dos, el otro también se eliminará.                                               |
| **Email**      | Todos los correos electrónicos verificados aparecen como autenticadores, pero no puedes eliminarlos. Solo puedes eliminar autenticadores de correo electrónico que se hayan inscrito explícitamente. |

<div id="regenerate-recovery-codes">
  ## Regenerar códigos de recuperación
</div>

Para eliminar un código de recuperación y generar otro nuevo, obtén un [token de acceso de la Management API](https://auth0.com/docs/api/management/v2/tokens) de Auth0 y usa el <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip> [endpoint de regeneración de códigos de recuperación](https://auth0.com/docs/api/management/v2/#!/Users/post_recovery_code_regeneration).

<AuthCodeGroup>
  ```bash cURL
  curl --request POST \
    --url 'https://{yourDomain}/api/v2/users/USER_ID/recovery-code-regeneration' \
    --header 'authorization: Bearer MANAGEMENT_API_TOKEN'
  ```
