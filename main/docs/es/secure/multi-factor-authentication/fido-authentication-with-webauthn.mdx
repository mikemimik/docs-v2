La API de autenticación web (también conocida como WebAuthn) es una [especificación](https://www.w3.org/TR/webauthn/) creada por el W3C y FIDO, con la participación de Google, Mozilla, Microsoft, Yubico y otras organizaciones. Esta API permite autenticar a los usuarios mediante criptografía de clave pública.

<Warning>
  Actualmente, el RP ID de WebAuthn está limitado a una configuración de un solo dominio y no es compatible con clientes que usan múltiples dominios personalizados.
</Warning>

WebAuthn es el método de autenticación más seguro y fácil de usar en la web. Algunas de las principales razones son:

* Minimiza la fricción durante el inicio de sesión. Un gesto simple y familiar permite a los usuarios autenticarse.
* Es el único método de autenticación web resistente al phishing.
* Se basa en estándares y está implementado en navegadores y sistemas operativos.

WebAuthn permite a los usuarios autenticarse con dos tipos de autenticadores:

* **Los autenticadores itinerantes** son extraíbles y multiplataforma, como una YubiKey, y pueden usarse en varios dispositivos. Para autenticarse con un autenticador itinerante, es necesario conectarlo al dispositivo (mediante USB, NFC o Bluetooth), demostrar presencia (por ejemplo, tocándolo) y, opcionalmente, realizar verificación de usuario, por ejemplo, introduciendo un PIN.
* **Los autenticadores de plataforma** están integrados en un dispositivo y solo funcionan en ese dispositivo. Algunos ejemplos son Touch Bar de MacBook, Windows Hello, Touch ID/Face ID de iOS y el reconocimiento facial o de huella dactilar de Android. Los datos biométricos se almacenan en el dispositivo y nunca se envían al servidor. Cuando no se pueden usar datos biométricos, normalmente se ofrecen métodos de autenticación alternativos. Por ejemplo, si llevas una mascarilla, en lugar de usar Face ID puedes introducir tu código de acceso.

<div id="webauthn-and-web-origins">
  ## WebAuthn y orígenes web
</div>

WebAuthn funciona generando un par de claves privada/pública para cada [origen web](https://en.wikipedia.org/wiki/Same-origin_policy), que se registra en el dispositivo o la clave de seguridad.

Dado que el par de claves está vinculado al dominio, los usuarios están protegidos contra ataques de phishing. Si el atacante los engaña para que usen WebAuthn en un dominio diferente, el autenticador de WebAuthn no tendrá un par de claves para ese dominio y la autenticación fallará. El atacante no obtendrá ningún dato que pueda identificar al usuario.

<div id="roaming-authenticators">
  ## Autenticadores itinerantes
</div>

Para saber cómo habilitar los autenticadores itinerantes en Auth0, consulta la documentación de [WebAuthn with Security Keys](/docs/es/secure/multi-factor-authentication/fido-authentication-with-webauthn/configure-webauthn-security-keys-for-mfa).

El flujo de registro predeterminado para claves de seguridad en Android que se muestra a continuación tiene los siguientes pasos:

* Los usuarios se autentican con nombre de usuario y contraseña.
* Se les pide que seleccionen el método de autenticación que quieren registrar.
* Si eligen claves de seguridad, se les muestra una lista de instrucciones.
* Después, se les pide que usen su clave de seguridad.
* El dispositivo muestra su UI nativa para completar el desafío de clave de seguridad.
* Los usuarios pueden asignarle un nombre a la clave para identificarla más adelante, en caso de que registren varias.

<div id="platform-authenticators">
  ## Autenticadores de plataforma
</div>

Para saber cómo habilitar los autenticadores de plataforma, consulta la documentación de [WebAuthn con biometría del dispositivo](/docs/es/secure/multi-factor-authentication/fido-authentication-with-webauthn/configure-webauthn-device-biometrics-for-mfa).

Dado que los autenticadores de plataforma solo se pueden usar en un único dispositivo, **no deberían ser los únicos factores que registren los usuarios.** Para evitar que los usuarios pierdan el acceso a sus cuentas, Auth0 les pedirá que registren autenticadores de plataforma después de autenticarse correctamente con otro método de autenticación.

Auth0 intentará **registrar progresivamente** todos los dispositivos de los usuarios. Se pedirá a los usuarios que registren los autenticadores de plataforma de sus dispositivos en cada dispositivo que utilicen.

El flujo de registro predeterminado para Face ID en iOS se muestra a continuación y tiene los siguientes pasos:

* Los usuarios se autentican con nombre de usuario y contraseña.
* Registran otro método de autenticación <Tooltip tip="Autenticación multifactor (MFA): Proceso de autenticación de usuario que utiliza un factor adicional al nombre de usuario y la contraseña, como un código por SMS." cta="Ver glosario" href="/docs/es/glossary?term=MFA">MFA</Tooltip>, como [SMS](/docs/es/secure/multi-factor-authentication/multi-factor-authentication-factors/configure-sms-voice-notifications-mfa), [Push](/docs/es/secure/multi-factor-authentication/multi-factor-authentication-factors/configure-push-notifications-for-mfa) o [OTP basado en tiempo](/docs/es/secure/multi-factor-authentication/multi-factor-authentication-factors/configure-otp-notifications-for-mfa).
* Asignan un nombre a su dispositivo para identificarlo más adelante.

<Frame>![Registro de MFA con biometría del dispositivo](/docs/images/cdy7uua7fh8z/1lGCq8BMA0rdkbiG3WZXWr/db44bb26054e6fadc540140f65bbc026/image.png)</Frame>

La próxima vez que inicien sesión desde ese dispositivo, los usuarios introducen su nombre de usuario y contraseña y completan la autenticación MFA con el autenticador biométrico del dispositivo.

<div id="webauthn-as-multi-factor-authentication">
  ## WebAuthn como autenticación multifactor
</div>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Cuando los usuarios se autentican con WebAuthn, usan algo **que tienen** como factor de autenticación: una clave de seguridad o un dispositivo.

  Tanto las claves de seguridad como la biometría del dispositivo admiten la **verificación de usuario**, que requiere que los usuarios proporcionen algo **que saben** (un PIN o un código de acceso) y algo **que son** (como rasgos biométricos).

  Al usar biometría del dispositivo, la verificación de usuario siempre se realiza. Para realizar la verificación con claves de seguridad, [debes configurar Auth0](/docs/es/secure/multi-factor-authentication/fido-authentication-with-webauthn/configure-webauthn-security-keys-for-mfa) para que requiera un PIN. Entonces, se pedirá a los usuarios que introduzcan un PIN, que solo se almacena en la clave de seguridad, para completar la autenticación. Cuando se realiza la verificación de usuario, los usuarios pueden iniciar sesión con WebAuthn como único método de autenticación y así lograr autenticación multifactor.

  Al usar WebAuthn para la autenticación junto con la verificación de usuario, no solo reemplazas la contraseña por algo mucho más sencillo de usar, sino que también eliminas la necesidad de otro paso de autenticación cuando se requiere MFA.
</Callout>
