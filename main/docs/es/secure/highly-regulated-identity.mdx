Identidad altamente regulada (HRI) es la solución Financial-Grade Identity™ de Auth0 para proteger operaciones con datos sensibles y servicios críticos para su negocio. Diseñada inicialmente para sectores altamente regulados, como el financiero y el sanitario, Identidad altamente regulada eleva el nivel de seguridad para proteger una amplia variedad de casos de uso de los clientes, entre ellos transferencias de dinero, pagos digitales y acceso a historiales médicos. También puede usar Identidad altamente regulada para otras operaciones sensibles que requieren seguridad reforzada, como aprobar cambios en credenciales administrativas, proteger el acceso privilegiado a un portal web, entre otras.

Para proteger sus operaciones empresariales sensibles, Identidad altamente regulada ofrece:

* [Seguridad avanzada con OpenID Connect (FAPI)](#advanced-security-with-openid-connect-fapi-)
* [Autenticación reforzada del cliente (SCA)](#strong-customer-authentication-sca-) y [vinculación dinámica](#dynamic-linking)
* [Protección de la confidencialidad y la integridad](#confidentiality-and-integrity-protection)
* [Autenticación reforzada de las aplicaciones](#stronger-application-authentication)
* [Protección de tokens de acceso con Token Binding](#protect-access-tokens-with-token-binding)
* [Flujos de aprobación personalizables para una mejor experiencia de usuario](#customizable-approval-flows-for-better-user-experience)

<div id="advanced-security-with-openid-connect-fapi">
  ## Seguridad avanzada con OpenID Connect (FAPI)
</div>

[OpenID FAPI](https://openid.net/wg/fapi/specifications/) es un conjunto de especificaciones de seguridad y privacidad desarrollado por la <Tooltip tip="OpenID: Estándar abierto de autenticación que permite a las aplicaciones verificar la identidad de los usuarios sin recopilar ni almacenar información de inicio de sesión." cta="Ver glosario" href="/docs/es/glossary?term=OpenID">OpenID</Tooltip> Foundation. Las API que cumplen las normas FAPI se clasifican como «de nivel financiero», lo que significa que ofrecen mecanismos sólidos de autenticación y autorización que ayudan a proteger el acceso a datos y servicios financieros, así como a otros datos y servicios sensibles.

Auth0 es un proveedor certificado de FAPI. Para obtener más información sobre las mejoras de seguridad que introdujimos para cumplir las normas FAPI, consulta las siguientes secciones:

* [Protección de la confidencialidad y la integridad](#confidentiality-and-integrity-protection)
* [Autenticación reforzada de las aplicaciones](#stronger-application-authentication)
* [Protección de tokens de acceso con Token Binding](#protect-access-tokens-with-token-binding)

Para obtener más información sobre FAPI, consulta el informe técnico de OpenID [Open Banking, Open Data, and Financial-grade APIs](https://openid.net/wordpress-content/uploads/2022/03/OIDF-Whitepaper_Open-Banking-Open-Data-and-Financial-Grade-APIs_2022-03-16.pdf) y las [especificaciones del grupo de trabajo de FAPI](https://openid.net/wg/fapi/specifications/).

<div id="strong-customer-authentication-sca">
  ## autenticación reforzada del cliente (SCA)
</div>

<Frame>![](/docs/images/cdy7uua7fh8z/3JDIerJcevImoIDRd7OfIy/98597a9164b66cde9ec7ddc23f3e849b/image14.png)</Frame>

Introducida por la [Directiva de Servicios de Pago (PSD2)](https://www.europeanpaymentscouncil.eu/sites/default/files/infographic/2018-04/EPC_Infographic_PSD2_April%202018.pdf) de Europa, la autenticación reforzada del cliente (SCA) exige el uso de al menos dos factores de autenticación distintos de los tres siguientes:

* Algo que el usuario sabe (por ejemplo, una contraseña)
* Algo que el usuario posee (por ejemplo, un dispositivo)
* Algo inherente al usuario (por ejemplo, una huella dactilar)

Los factores de autenticación deben ser independientes, de modo que si uno se ve comprometido, no ponga en riesgo a los demás. La SCA se está convirtiendo rápidamente en el estándar mundial para proteger datos y servicios sensibles.

Para facilitar el cumplimiento de la SCA, Auth0 ofrece varios factores de autenticación que permiten registrar a los usuarios y presentarles verificaciones durante una transacción de inicio de sesión. Identidad altamente regulada usa los siguientes factores de autenticación para proteger sus transacciones:

* Notificaciones push en dispositivo móvil
* SMS
* Email
* WebAuthn

Con [Actions](/docs/es/customize/actions), puede determinar dinámicamente qué factores de autenticación usar. Esto le da la flexibilidad de personalizar la lógica de su código. Por ejemplo, puede agregar un segundo factor de autenticación para pagos superiores a 10 USD. Para obtener más información, consulte [Aplicar una política dinámica](/docs/es/secure/highly-regulated-identity/transactional-authorization-with-authorization-code-flow#apply-dynamic-policy).

<div id="dynamic-linking">
  ## Vinculación dinámica
</div>

PSD2 exige que los proveedores de servicios de pago implementen la vinculación dinámica junto con la autenticación reforzada del cliente. La vinculación dinámica presenta al usuario los detalles de la transacción para su validación y aprobación explícitas, y vincula de forma única la autorización con los detalles de la transacción. Esto garantiza una buena experiencia de usuario y contribuye al cumplimiento normativo.

Para habilitar la vinculación dinámica, puede usar Rich Authorization Requests (RAR) para enviar datos detallados de autorización de transacciones al endpoint de autorización de <Tooltip tip="OAuth 2.0: Marco de autorización que define protocolos y flujos de trabajo de autorización." cta="Ver glosario" href="/docs/es/glossary?term=OAuth">OAuth</Tooltip>. El siguiente ejemplo de código muestra un objeto JSON `authorization_details`, que contiene información como el tipo de pago, el importe, la moneda y el destinatario:

```json lines
"authorization_details": [
 {
   "type": "one_time_payment",
   "amount": {
     "amount": 2460.46,
     "currency": "USD"
   },
   "sourceAccount": "xxxxxxxxxxx4567",
   "recipient": "Acme Travel, Inc.",
   "concept": "All Inclusive Resort Package for Two",
 }
]
```

A `authorization_details` se le asigna una referencia única de transacción, que Auth0 usa para pedirle al usuario que realice una autenticación escalonada:

* Usa notificaciones push para mostrar los detalles de la transacción y obtener la aprobación en un dispositivo distinto, como una aplicación móvil.
* Usa SMS, correo electrónico o WebAuthn para confirmar los detalles en el dispositivo que originó la transacción después de que el usuario complete el segundo factor de autenticación.

<Warning>
  No envíes datos de autorización detallados de la transacción ni otros datos sensibles o regulados fuera de `authorization_details`.
</Warning>

Si el usuario confirma los detalles, la transacción continúa y Auth0 emite un <Tooltip tip="Token de acceso: Credencial de autorización, en forma de una cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=access+token">token de acceso</Tooltip> asociado con authorization&#95;details, ya aprobado. Los desarrolladores también pueden agregar la referencia única de la transacción al token de acceso. Como resultado, tus servidores de la API pueden validar más adelante los detalles aprobados de la transacción al recibir y procesar solicitudes de API.

Para obtener más información sobre RAR, consulta [Flujo de código de autorización con Rich Authorization Requests](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow/authorization-code-flow-with-rar).

<div id="confidentiality-and-integrity-protection">
  ## Protección de la confidencialidad y la integridad
</div>

Los detalles de autorización pueden incluir números de cuenta, importes monetarios, nombres de comercios y otra información sumamente sensible que se transmite a través de URL o tokens de acceso no seguros. Para proteger los datos sensibles frente al acceso no autorizado y la manipulación, Identidad altamente regulada ofrece una protección integral de la confidencialidad y la integridad.

<div id="protect-sensitive-data-in-the-front-channel">
  ### Proteja los datos sensibles en el front channel
</div>

Para proteger los datos sensibles en el front channel, como un navegador web, Identidad altamente regulada ofrece las siguientes soluciones como parte del perfil de seguridad FAPI 1 Advanced.

<div id="pushed-authorization-requests-par">
  #### Solicitudes de autorización push (PAR)
</div>

[PAR](https://datatracker.ietf.org/doc/rfc9126/) introduce un nuevo endpoint que permite a los clientes enviar directamente el payload de una solicitud de autorización de OAuth 2.0 al <Tooltip tip="Servidor de autorización: servidor centralizado que contribuye a definir los límites del acceso de un usuario. Por ejemplo, tu servidor de autorización puede controlar los datos, las tareas y las funcionalidades disponibles para un usuario." cta="Ver glosario" href="/docs/es/glossary?term=authorization+server">servidor de autorización</Tooltip> (es decir, Auth0 en este caso). De este modo, se evita pasar los parámetros de autorización a través del front channel no seguro (es decir, el navegador), lo que reduce el riesgo de que un intermediario acceda sin autorización a esos parámetros.

Para obtener más información sobre PAR, consulta [Flujo de código de autorización con solicitudes de autorización push (PAR)](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow/authorization-code-flow-with-par) y [Configurar solicitudes de autorización push (PAR)](/docs/es/get-started/applications/configure-par).

<div id="jwt-secured-authorization-request-jar">
  #### Solicitud de autorización protegida con JWT (JAR)
</div>

[JAR](https://datatracker.ietf.org/doc/rfc9101/) es una extensión del protocolo OAuth2 que refuerza la seguridad de las solicitudes de autorización. Lo hace mediante un parámetro de solicitud <Tooltip tip="JSON Web Token (JWT): formato estándar del token de ID (y, a menudo, también del token de acceso) que se utiliza para representar claims de forma segura entre dos partes." cta="Ver glosario" href="/docs/es/glossary?term=JSON+Web+Token">JSON Web Token</Tooltip> (JWT) para proteger la integridad y, opcionalmente, la confidencialidad de los parámetros de la solicitud de autorización.

Para obtener más información sobre JAR, consulta [Authorization Code Flow with JWT-Secured Authorization Requests (JAR)](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow/authorization-code-flow-with-jar) y [Configure JWT-Secured Authorization Requests (JAR)](/docs/es/get-started/applications/configure-jar).

<div id="protect-sensitive-data-in-access-tokens">
  #### Protege los datos sensibles en los tokens de acceso
</div>

Para proteger los datos de autorización incluidos en los tokens de acceso, Identidad altamente regulada admite el uso de [JSON Web Encryption (JWE)](https://datatracker.ietf.org/doc/html/rfc7516) para cifrar el payload de los tokens de acceso. Esto protege los tokens de acceso frente a violaciones de la seguridad de los datos en la aplicación y frente a la inspección no autorizada de las llamadas a la API por parte de intermediarios.

Para obtener más información sobre JWE, consulta [JSON Web Encryption](/docs/es/secure/tokens/access-tokens/json-web-encryption) y [Configure JSON Web Encryption](/docs/es/get-started/apis/configure-json-web-encryption).

<div id="stronger-application-authentication">
  ## Autenticación de aplicaciones más sólida
</div>

Para mejorar la seguridad de la autenticación de aplicaciones, Identidad altamente regulada ofrece dos alternativas como parte del perfil de seguridad FAPI 1 Advanced:

* [clave privada JWT](http://tools.ietf.org/html/draft-ietf-oauth-jwt-bearer): implica generar un par de claves pública y privada para usarlo como credenciales para autenticar una aplicación. Ya está disponible para los clientes del plan Enterprise. Para obtener más información, consulta [autenticación con clave privada JWT](/docs/es/secure/application-credentials#private-key-jwt-authentication).
* [mTLS for OAuth](https://datatracker.ietf.org/doc/html/rfc8705): implica registrar un certificado X.509 estándar vinculado a una aplicación en tu tenant. El certificado puede estar emitido por una CA o ser autofirmado. Siguiendo los procedimientos estándar de mTLS, la clave privada correspondiente al certificado se usa del lado del cliente para establecer el túnel mTLS al enviar solicitudes a los endpoints de tu tenant de Auth0. Como resultado, Auth0 puede autenticar la aplicación sin transmitir secretos a través de la red. Para obtener más información, consulta [mTLS for OAuth](/docs/es/get-started/authentication-and-authorization-flow/authenticate-with-mtls).

Con la clave privada JWT y OAuth 2.0 mTLS, puedes rotar credenciales sin tiempo de inactividad manteniendo temporalmente dos claves y/o certificados activos al mismo tiempo para una aplicación determinada.

<div id="protect-access-tokens-with-token-binding">
  ## Protección de tokens de acceso con Token Binding
</div>

La compatibilidad con mTLS también permite usar Token Binding o Sender Constraining. Token Binding asocia la huella digital del certificado de cliente utilizado para establecer el túnel mTLS con un token de acceso. Cuando el cliente llama a una API con el token de acceso vinculado al certificado, el servidor de la API puede verificar si el cliente también está usando el certificado de cliente asociado. Como resultado, incluso si el token de acceso se ve comprometido, las personas malintencionadas que no conocen el certificado de cliente no podrán acceder a los recursos protegidos.

**Nota:** Token Binding funciona de forma independiente del método de autenticación de la aplicación y no requiere el registro previo del certificado de cliente. Para obtener más información, consulta [Configurar Sender Constraining](/docs/es/secure/sender-constraining/configure-sender-constraining).

<div id="customizable-approval-flows-for-better-user-experience">
  ## Flujos de aprobación personalizables para una mejor experiencia de usuario
</div>

Al diseñar soluciones del mundo real para una seguridad de grado financiero, es importante tener en cuenta la experiencia del usuario. Aplicar el mismo flujo de autenticación a todas las transacciones no es tan eficaz como ajustarlo dinámicamente según los detalles de la transacción y los casos de uso.

Puedes personalizar tu flujo de autenticación con [Actions](/docs/es/customize/actions). Por ejemplo, después de que el usuario inicie sesión, puedes revisar los detalles de la transacción recibidos mediante RAR, listar los factores de autenticación del usuario que ya están registrados y validados, y usar servicios externos, como motores de evaluación de riesgos, para determinar cuál será el siguiente factor de autenticación. Para obtener más información, consulta [Apply dynamic policy](/docs/es/secure/highly-regulated-identity/transactional-authorization-with-authorization-code-flow#apply-dynamic-policy).

Las nuevas plantillas de <Tooltip tip="Universal Login: Tu aplicación redirige a Universal Login, alojado en el servidor de autorización de Auth0, para verificar la identidad de un usuario." cta="Ver glosario" href="/docs/es/glossary?term=Universal+Login">Universal Login</Tooltip> también te permiten personalizar los atributos que se muestran en la pantalla de aprobación de la transacción según el tipo de transacción y otros detalles de autorización. Para obtener más información, consulta [Configure Rich Authorization Requests (RAR)](/docs/es/get-started/apis/configure-rich-authorization-requests).
