[Sender Constraining](/docs/es/secure/sender-constraining) es un mecanismo de seguridad de <Tooltip tip="OAuth 2.0: Marco de autorización que define protocolos y flujos de trabajo de autorización." cta="Ver glosario" href="/docs/es/glossary?term=OAuth+2.0">OAuth 2.0</Tooltip> y <Tooltip tip="OAuth 2.0: Marco de autorización que define protocolos y flujos de trabajo de autorización." cta="Ver glosario" href="/docs/es/glossary?term=OpenID">OpenID</Tooltip> Connect (OIDC) que vincula criptográficamente los tokens de acceso y los <Tooltip tip="Refresh Token: Token que se usa para obtener un token de acceso renovado sin obligar a los usuarios a volver a iniciar sesión." cta="Ver glosario" href="/docs/es/glossary?term=refresh+tokens">tokens de actualización</Tooltip> a la aplicación cliente específica que los obtuvo, lo que evita el robo y el uso indebido de tokens.

Auth0 admite Sender Constraining con mTLS y Demonstrating Proof-of-Possession (DPoP). Si habilitas Sender Constraining para una aplicación cliente, también debes aplicarlo al <Tooltip tip="Resource Server: Servidor que aloja recursos protegidos. Los servidores de recursos aceptan y responden a solicitudes de recursos protegidos." cta="Ver glosario" href="/docs/es/glossary?term=resource+server">servidor de recursos</Tooltip> al que haces llamadas a la API.

Para configurar Sender Constraining en Auth0, debes:

* [Configurar Sender Constraining para una aplicación cliente](#configure-sender-constraining-for-a-client-application)
* [Configurar Sender Constraining para un servidor de recursos](#configure-sender-constraining-for-a-resource-server)

<div id="how-it-works">
  ## Cómo funciona
</div>

<Tooltip tip="Token de acceso: credencial de autorización, en forma de cadena opaca o JWT, que se utiliza para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=Access+tokens">Tokens de acceso</Tooltip> se vinculan en Auth0 según cómo configures Sender Constraining para tu aplicación cliente y servidor de recursos:

1. **Audiencia solicitada:** En una solicitud de token, si la audiencia solicitada es únicamente `/userinfo`, o si solo está pensada para usarse con el endpoint [`/userinfo`](https://auth0.com/docs/api/authentication/user-profile/get-user-info), o si es una API personalizada que puede incluir `/userinfo` cuando también solicitas el scope openid, influye en si los tokens de acceso quedan vinculados.
2. **Aplicación cliente:** Si configuras Sender Constraining como `required` para la aplicación cliente.
3. **Servidor de recursos:** Si configuras Sender Constraining para el servidor de recursos:

   1. `none`: No has configurado Sender Constraining para el servidor de recursos.
   2. `allowed`: Has configurado Sender Constraining para el servidor de recursos estableciendo un método de Sender Constraining.
   3. `required`: Has configurado Sender Constraining como obligatorio para el servidor de recursos, lo que significa que los tokens de acceso deben estar vinculados a una aplicación. Puedes exigir Sender Constraining para todas las aplicaciones o solo para las aplicaciones públicas. Requiere un método de Sender Constraining.

<Callout icon="lightbulb" color="#0EA5E9" iconType="regular">
  Cuando se usa mTLS como método de Sender Constraining, Sender Constraining siempre es obligatorio para todas las aplicaciones. Las aplicaciones públicas no son compatibles con mTLS, por lo que no puedes limitar mTLS sender constraining solo a clientes públicos.
</Callout>

* **Prueba de posesión:** Si la aplicación cliente envió una aserción de prueba de posesión en la solicitud de token:

  * mTLS sender constraining: La prueba de posesión se demuestra mediante la presentación satisfactoria por parte del cliente de una clave privada específica (asociada a un certificado de cliente) durante el handshake TLS.
  * DPoP: La prueba de posesión se logra cuando el cliente crea un JWT de prueba DPoP firmándolo criptográficamente con su clave privada e incluye el JWT de prueba `DPoP` en el encabezado HTTP DPoP de cada solicitud en la que se utiliza el token de acceso asociado.

La siguiente tabla describe cómo se emiten los tokens de acceso y si quedan vinculados en función de varios parámetros de solicitud del cliente y de las configuraciones del servidor de recursos de Auth0:

| Tipo de audiencia solicitada                      | ¿El cliente requiere PoP? | ¿El cliente envió una prueba de posesión (PoP)? | Política del servidor de recursos de Auth0: None | Política del servidor de recursos de Auth0: Allowed (No obligatorio) | Política del servidor de recursos de Auth0: Required |
| ------------------------------------------------- | ------------------------- | ----------------------------------------------- | ------------------------------------------------ | -------------------------------------------------------------------- | ---------------------------------------------------- |
| Solo Userinfo                                     | No                        | No                                              | Emitido, no vinculado                            | N/A                                                                  | N/A                                                  |
| Solo Userinfo                                     | No                        | Sí                                              | Emitido, vinculado                               | N/A                                                                  | N/A                                                  |
| Solo Userinfo                                     | Sí                        | No                                              | No emitido                                       | N/A                                                                  | N/A                                                  |
| Solo Userinfo                                     | Sí                        | Sí                                              | Emitido, vinculado                               | N/A                                                                  | N/A                                                  |
| Audiencia personalizada (puede contener Userinfo) | No                        | No                                              | Emitido, no vinculado                            | Emitido, no vinculado                                                | No emitido                                           |
| Audiencia personalizada (puede contener Userinfo) | No                        | Sí                                              | Emitido, no vinculado                            | Emitido, vinculado                                                   | Emitido, vinculado                                   |

<div id="configure-sender-constraining-for-a-client-application">
  ## Configurar Sender Constraining para una aplicación cliente
</div>

Cuando exige Sender Constraining para una aplicación cliente, los tokens de acceso quedan vinculados a esa aplicación. Auth0 verifica las solicitudes para garantizar que solo la aplicación que solicitó el token pueda usarlo para acceder al recurso asociado.

Una vez que configure una aplicación cliente para que requiera Sender Constraining, puede establecer el método de Sender Constraining, ya sea mTLS o DPoP, al configurar su servidor de recursos.

Puede configurar Sender Constraining para una aplicación cliente desde el <Tooltip tip="Auth0 Dashboard: el producto principal de Auth0 para configurar sus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip> o la <Tooltip tip="Management API: un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip>.

<Tabs>
  <Tab title="Auth0 Dashboard">
    1. Vaya a [Dashboard &gt; Applications &gt; Applications](https://manage.auth0.com/#/applications). Seleccione la aplicación que desea configurar.
    2. En **Settings**, desplácese hasta **Token Sender-Constraining**.
    3. Active **Require Sender Constraining**. Desactívelo para quitar el requisito de Sender Constraining de la aplicación.

    <Frame>![](/docs/images/cdy7uua7fh8z/3BoGW9NEYKPBkfk2tPBL7l/0f884ac3f64f17b23fd6a0e9436bb472/Screenshot_2024-07-23_at_1.30.19_PM.png)</Frame>
  </Tab>

  <Tab title="Management API">
    Para configurar un cliente para Sender Constraining, use la [Management API](https://auth0.com/docs/api/management/v2).

    Para exigir Sender Constraining para un cliente, envíe una solicitud PATCH para [actualizar la configuración del cliente](https://auth0.com/docs/api/management/v2/clients/patch-clients-by-id). Establezca el parámetro `require_proof_of_possession` en `true`.

    ```bash lines
    curl -L -X PATCH 'https://{YOUR_DOMAIN}/api/v2/clients/{YOUR_CLIENT_ID}' \
    -H 'Authorization: Bearer {YOUR_MANAGEMENT_API_TOKEN}' \
    -H 'Content-Type: application/json' \
    -d '{"require_proof_of_possession": true}'
    ```

    Para quitar el requisito de Sender Constraining, establezca el parámetro `require_proof_of_possession` en `false`.

    ```bash lines
    curl -L -X PATCH 'https://{YOUR_DOMAIN}/api/v2/clients/{YOUR_CLIENT_ID}' \
    -H 'Authorization: Bearer {YOUR_MANAGEMENT_API_TOKEN}' \
    -H 'Content-Type: application/json' \
    -d '{"require_proof_of_possession": false}'
    ```
  </Tab>
</Tabs>

<div id="configure-sender-constraining-for-a-resource-server">
  ## Configurar Sender Constraining para un servidor de recursos
</div>

Los tokens de acceso emitidos por Auth0 pueden estar vinculados al remitente (es decir, la aplicación cliente) que necesita acceder a las API de un servidor de recursos.

Puede configurar Sender Constraining para un servidor de recursos desde el Auth0 Dashboard o la Management API.

<Tabs>
  <Tab title="Auth0 Dashboard">
    Para habilitar Token Binding o Sender Constraining, configura los **API Settings** de tu API.

    1. Ve a [Auth0 Dashboard &gt; Applications &gt; APIs](https://manage.auth0.com/#/apis).
    2. Selecciona la API que quieras configurar.
    3. En la pestaña **Settings**, busca la sección **Token Sender-Constraining**.
    4. Configura lo siguiente:

       1. Método de Sender Constraining:

          1. **None:** No habilites ningún método de Sender Constraining para tu servidor de recursos.
          2. **mTLS**: Habilita mTLS como método de Sender Constraining para tu servidor de recursos.
          3. **DPoP:** Habilita DPoP como método de Sender Constraining para tu servidor de recursos.

       2. Requerir Token Sender Constraining: Selecciona qué política de Sender Constraining se aplica a esta API:

          1. **Always:** Requiere Sender Constraining para todas las aplicaciones. Esta es la única opción disponible cuando se usa mTLS.
          2. **For Public Applications:** Requiere Sender Constraining solo para aplicaciones públicas. Esta opción no está disponible cuando se usa mTLS.
          3. **Never:** No se requiere Sender Constraining.

    <Frame>![Auth0 Dashboard > APIs > Settings > Token binding](/docs/images/sender-constraining/method_is_dpop.png)</Frame>
  </Tab>

  <Tab title="Management API">
    Para habilitar Sender Constraining con la Management API, envía una solicitud PATCH para [actualizar el servidor de recursos](https://auth0.com/docs/api/management/v2/resource-servers/patch-resource-servers-by-id). Establece los parámetros del objeto `proof_of_possession` de la siguiente manera:

    | Parámetro      | Descripción                                                                                                                                                                                                                                                                                             |
    | -------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
    | `mechanism`    | Establece el método de sender constraining: `none`, `mtls` o `dpop`.                                                                                                                                                                                                                                    |
    | `required`     | Obligatorio. Cuando se establece en `true`, los tokens de acceso emitidos a una aplicación para esta API quedarán vinculados a esa aplicación. Usa `required_for` para especificar a qué tipos de aplicación se aplica este requisito. Cuando se establece en `false`, no se requiere Sender Constraining. |
    | `required_for` | Filtra qué tipos de aplicación requieren Sender Constraining cuando `required` es `true`: `all_clients` (todas las aplicaciones) o `public_clients` (solo aplicaciones públicas). De forma predeterminada, es `all_clients` si no se especifica. Al usar mTLS, solo `all_clients` es válido.            |

    La siguiente muestra de código es un ejemplo de cuerpo de solicitud que configura un servidor de recursos para mTLS Sender Constraining. Como mTLS solo admite `all_clients` para `required_for` (que es el valor predeterminado), no es necesario especificarlo:

    ```bash lines
    curl -L -X PATCH 'https://{YOUR_DOMAIN}/api/v2/resource-servers/{YOUR_RESOURCE_SERVER_ID}' \
    -H 'Authorization: Bearer {YOUR_MANAGEMENT_API_TOKEN}' \
    -H 'Content-Type: application/json' \
    -H 'Accept: application/json' \
    -d '{
      "proof_of_possession": {
        "mechanism": "mtls",
        "required": true
      }
    }'
    ```

    La siguiente muestra de código es un ejemplo de cuerpo de solicitud que configura un servidor de recursos para DPoP Sender Constraining, obligatorio solo para aplicaciones públicas:

    ```bash lines
    curl -L -X PATCH 'https://{YOUR_DOMAIN}/api/v2/resource-servers/{YOUR_RESOURCE_SERVER_ID}' \
    -H 'Authorization: Bearer {YOUR_MANAGEMENT_API_TOKEN}' \
    -H 'Content-Type: application/json' \
    -H 'Accept: application/json' \
    -d '{
      "proof_of_possession": {
        "mechanism": "dpop",
        "required": true,
        "required_for": "public_clients"
      }
    }'
    ```

    La siguiente muestra de código configura DPoP Sender Constraining como obligatorio para todas las aplicaciones. Como `all_clients` es el valor predeterminado, `required_for` puede omitirse:

    ```bash lines
    curl -L -X PATCH 'https://{YOUR_DOMAIN}/api/v2/resource-servers/{YOUR_RESOURCE_SERVER_ID}' \
    -H 'Authorization: Bearer {YOUR_MANAGEMENT_API_TOKEN}' \
    -H 'Content-Type: application/json' \
    -H 'Accept: application/json' \
    -d '{
      "proof_of_possession": {
        "mechanism": "dpop",
        "required": true
      }
    }'
    ```

    La siguiente muestra de código deshabilita Sender Constraining para un servidor de recursos (equivale a seleccionar &quot;Never&quot; en el Dashboard):

    ```bash lines
    curl -L -X PATCH 'https://{YOUR_DOMAIN}/api/v2/resource-servers/{YOUR_RESOURCE_SERVER_ID}' \
    -H 'Authorization: Bearer {YOUR_MANAGEMENT_API_TOKEN}' \
    -H 'Content-Type: application/json' \
    -H 'Accept: application/json' \
    -d '{
      "proof_of_possession": null 
    }'
    ```
  </Tab>
</Tabs>
