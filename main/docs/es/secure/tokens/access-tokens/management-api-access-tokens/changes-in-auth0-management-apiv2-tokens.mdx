title: Cambios en los tokens de Auth0 Management APIv2
---

Hace algún tiempo, cambiamos el proceso para obtener un token de Management APIv2. En este artículo se explica qué cambió, por qué se hizo y cómo puede sortearlo temporalmente (no recomendado).

<div id="what-changed-and-why">
  ## Qué cambió y por qué
</div>

<div id="the-user-experience">
  ### La experiencia de usuario
</div>

Hasta hace poco, podías generar un token de Management APIv2 directamente desde el Explorer de la <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip>. Seleccionabas los scopes según el endpoint que querías invocar y obtenías un token desde esa misma página.

Ese método era muy sencillo, pero también muy inseguro. Por eso lo cambiamos.

El nuevo método utiliza el [Client Credentials Flow](/docs/es/get-started/authentication-and-authorization-flow/client-credentials-flow). Para aprender a usar el nuevo proceso, consulta [Management API Access Tokens](/docs/es/secure/tokens/access-tokens/management-api-access-tokens).

<div id="why-this-changed">
  #### Por qué cambió esto
</div>

Para generar el token, la Management API requería acceso a tu <Tooltip tip="Client Secret: Secret usado por un cliente (aplicación) para autenticarse con el servidor de autorización; solo deberían conocerlo el cliente y el servidor de autorización, y debe ser lo suficientemente aleatorio como para que no sea adivinable." cta="Ver glosario" href="/docs/es/glossary?term=Client+Secret">secreto de cliente</Tooltip> global (utilizado para firmar el token). Esta es información que no debería exponerse en los navegadores web.

Además, el API Explorer no tiene forma de gestionar la autorización. Esto significa que, si un usuario pudiera iniciar sesión y acceder al API Explorer, podría generar un token con cualquier scope, incluso si no tuviera permiso para contar con ese scope.

La nueva implementación no plantea esos riesgos. Una vez que completes la configuración inicial, puedes obtener un token ya sea visitando el <Tooltip tip="Auth0 Dashboard: El producto principal de Auth0 para configurar tus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip> o haciendo una sencilla solicitud `POST` al [endpoint `/oauth/token` de nuestra Authentication API](https://auth0.com/docs/api/authentication#client-credentials).

Sin embargo, en lo que respecta al proceso manual, entendemos que tener que cambiar de pantalla no siempre ofrece la mejor experiencia de usuario, así que estamos buscando formas de hacer que el nuevo flujo sea más intuitivo.

<div id="the-validity-period">
  ### El período de validez
</div>

Con el flujo anterior, los tokens nunca caducaban. Con el nuevo flujo, todos los tokens de Management APIv2 caducan por defecto a las 24 horas.

<div id="why-this-changed">
  #### Por qué cambió esto
</div>

Tener un token que nunca caduca puede ser muy arriesgado si un atacante se hace con él. Si el token caduca al cabo de unas horas, el atacante solo dispone de un breve período de tiempo para acceder a tus recursos protegidos.

Para obtener un token, debes seguir únicamente el proceso descrito en [Management API Access Tokens](/docs/es/secure/tokens/access-tokens/management-api-access-tokens).
