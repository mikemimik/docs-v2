title: Cambios en los tokens de Auth0 Management APIv2
---

Hace algún tiempo, cambiamos el proceso para obtener un token de Auth0 Management APIv2. En este artículo se explica qué cambió, por qué se hizo y cómo puede eludirse (no recomendado).

<div id="what-changed-and-why">
  ## Qué ha cambiado y por qué
</div>

<div id="the-user-experience">
  ### La experiencia de usuario
</div>

Hasta hace poco, se podía generar un token de Auth0 Management APIv2 directamente desde el Explorador de la <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip>. Seleccionaba los scopes según el endpoint que quisiera invocar y obtenía un token desde esa misma página.

Ese método era muy sencillo, pero también muy inseguro. Por eso lo cambiamos.

El nuevo método usa el [Client Credentials Flow](/docs/es/get-started/authentication-and-authorization-flow/client-credentials-flow). Para obtener más información sobre cómo usar el nuevo proceso, consulte [Management API Access Tokens](/docs/es/secure/tokens/access-tokens/management-api-access-tokens).

#### Por qué cambió esto

Para generar el token, la Management API requería acceso a tu <Tooltip tip="Secreto de cliente: secreto que usa un cliente (aplicación) para autenticarse ante el servidor de autorización; solo el cliente y el servidor de autorización deberían conocerlo, y debe ser lo suficientemente aleatorio como para que no sea fácil de adivinar." cta="Ver glosario" href="/docs/es/glossary?term=Client+Secret">Secreto de cliente</Tooltip> global (utilizado para firmar el token). Esta es información que no debería exponerse a los navegadores web.

Además, el API Explorer no tiene ningún mecanismo de autorización. Esto significa que, si un usuario pudiera iniciar sesión y acceder al API Explorer, podría generar un token con cualquier scope, incluso si no tuviera permiso para tener ese scope.

La nueva implementación no plantea esos riesgos. Una vez que realices la configuración inicial, puedes obtener un token visitando el <Tooltip tip="Auth0 Dashboard: el producto principal de Auth0 para configurar tus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip> o enviando una simple solicitud `POST` al [`endpoint` `/oauth/token` de nuestra Authentication API](https://auth0.com/docs/api/authentication#client-credentials).

Sin embargo, en lo que respecta al proceso manual, entendemos que ir cambiando de pantalla no siempre ofrece la mejor experiencia de usuario, por lo que estamos buscando maneras de hacer que el nuevo flujo sea más intuitivo.

<div id="the-validity-period">
  ### El periodo de validez
</div>

Con el flujo anterior, los tokens no caducaban nunca. Con el nuevo flujo, todos los tokens de Auth0 Management APIv2 caducan por defecto al cabo de 24 horas.

<div id="why-this-changed">
  #### Por qué se hizo este cambio
</div>

Tener un token que nunca caduca puede ser muy arriesgado en caso de que un atacante se haga con él. Si el token caduca en unas pocas horas, el atacante solo tendrá una pequeña ventana de tiempo para acceder a tus recursos protegidos.

Para obtener un token, debes seguir únicamente el proceso descrito en [Management API Access Tokens](/docs/es/secure/tokens/access-tokens/management-api-access-tokens).
