description: Describe cómo validar un token de acceso.
title: Validar tokens de acceso
---

Un <Tooltip tip="Token de acceso: credencial de autorización, en forma de una cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=access+token">token de acceso</Tooltip> está destinado a una API y solo debe validarlo la API para la que fue emitido.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Los tokens de acceso del proveedor de identidad (IdP) no requieren validación. Pasa el token de acceso del IdP al IdP emisor para que se encargue de la validación. Para obtener más información, consulta [Tokens de acceso del proveedor de identidad](/docs/es/secure/tokens/access-tokens/identity-provider-access-tokens).
</Callout>

Si alguna de estas comprobaciones falla, el token se considera no válido y la solicitud debe rechazarse con un resultado `401 Unauthorized`.

1. **Realiza la validación estándar de JWT.** Como el token de acceso es un JWT, debes seguir los pasos estándar de validación de JWT. Consulta [Validar JSON Web Tokens](/docs/es/secure/tokens/json-web-tokens/validate-json-web-tokens) para obtener más información.
2. **Verifica los claims de audience del token.** Si has realizado la validación estándar de JWT, ya has decodificado el payload del JWT y revisado sus claims estándar. El claim de audience del token (`aud`, array de strings) depende de la solicitud inicial del token. El campo `aud` puede contener tanto un audience correspondiente a tu API personalizada como un audience correspondiente al endpoint `/userinfo`. Al menos uno de los valores de audience del token debe coincidir con el identificador único de la API de destino, tal como se define en la [configuración de tu API](https://manage.auth0.com/#/apis), en el campo **Identifier**. Consulta [Obtener tokens de acceso](/docs/es/secure/tokens/access-tokens/get-access-tokens) para obtener más información.
3. **Verifica los permisos (scopes).** Verifica que a la aplicación se le hayan concedido los permisos necesarios para acceder a tu API. Para ello, tendrás que comprobar el claim `scope` (`scope`, lista de strings separada por espacios) en el payload del JWT decodificado. Debe coincidir con los permisos requeridos para el endpoint al que se está accediendo. Por ejemplo, si tu API personalizada proporciona tres endpoints para leer, crear o eliminar un registro de usuario, al registrar tu API en Auth0 creaste tres permisos correspondientes:

   1. `create:users` proporciona acceso al endpoint `/create`
   2. `read:users` proporciona acceso al endpoint `/read`
   3. `delete:users` proporciona acceso al endpoint `/delete`

   En este caso, si una aplicación solicita acceso al endpoint `/create`, pero el claim `scope` del token de acceso no incluye el valor `create:users`, la API debe rechazar la solicitud.
