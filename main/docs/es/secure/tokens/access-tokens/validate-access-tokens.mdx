description: Describe cómo validar un token de acceso.
title: Validar tokens de acceso
---

Un <Tooltip tip="Token de acceso: credencial de autorización, en forma de cadena opaca o JWT, que se utiliza para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=access+token">token de acceso</Tooltip> está destinado a una API y solo debe validarlo la API para la que fue emitido.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Los tokens de acceso del proveedor de identidad (IdP) no requieren validación. Pasa el token de acceso del IdP al IdP emisor para que se encargue de validarlo. Para obtener más información, consulta [Identity Provider Access Tokens](/docs/es/secure/tokens/access-tokens/identity-provider-access-tokens).
</Callout>

Si alguna de estas comprobaciones falla, el token se considera inválido y la solicitud debe rechazarse con un resultado `401 Unauthorized`.

1. **Realiza la validación estándar de JWT.** Como el token de acceso es un JWT, debes realizar los pasos estándar de validación de JWT. Consulta [Validate JSON Web Tokens](/docs/es/secure/tokens/json-web-tokens/validate-json-web-tokens) para obtener más información.
2. **Verifica las claims de audiencia del token.** Si ya realizaste la validación estándar de JWT, entonces ya decodificaste el payload del JWT y revisaste sus claims estándar. La claim de audiencia del token (`aud`, arreglo de strings) depende de la token request inicial. El campo `aud` puede contener tanto una audiencia correspondiente a tu custom API como una audiencia correspondiente al endpoint `/userinfo`. Al menos uno de los valores de audiencia del token debe coincidir con el identificador único de la API de destino, tal como se define en los [Settings](https://manage.auth0.com/#/apis) de tu API, en el campo **Identifier**. Consulta [Get Access Tokens](/docs/es/secure/tokens/access-tokens/get-access-tokens) para obtener más información.
3. **Verifica los permisos (scopes).** Verifica que a la aplicación se le hayan concedido los permisos necesarios para acceder a tu API. Para ello, tendrás que comprobar la claim `scope` (`scope`, lista de strings separada por espacios) en el payload del JWT decodificado. Debe coincidir con los permisos requeridos para el endpoint al que se está accediendo. Por ejemplo, si tu custom API proporciona tres endpoints para leer, crear o eliminar un registro de usuario, cuando registraste tu API en Auth0, creaste tres permisos correspondientes:

   1. `create:users` proporciona acceso al endpoint `/create`
   2. `read:users` proporciona acceso al endpoint `/read`
   3. `delete:users` proporciona acceso al endpoint `/delete`

   En este caso, si una aplicación solicita acceso al endpoint `/create`, pero la claim `scope` del token de acceso no incluye el valor `create:users`, entonces la API debe rechazar la solicitud.
