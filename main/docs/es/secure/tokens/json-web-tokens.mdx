<Tooltip tip="JSON Web Token (JWT): formato estándar de token de ID (y, a menudo, formato de token de acceso) que se utiliza para representar claims de forma segura entre dos partes." cta="Ver glosario" href="/docs/es/glossary?term=JSON+web+token">JSON web token</Tooltip> (JWT), pronunciado &quot;jot&quot;, es un estándar abierto ([RFC 7519](https://tools.ietf.org/html/rfc7519)) que define una forma compacta y autocontenida de transmitir información de forma segura entre partes como un objeto JSON. Una vez más, JWT es un estándar, lo que significa que todos los JWT son tokens, pero no todos los tokens son JWT.

Debido a su tamaño relativamente pequeño, un JWT puede enviarse a través de una URL, en un parámetro POST o dentro de un encabezado HTTP, y transmitirse rápidamente. Un JWT contiene toda la información necesaria sobre una entidad para evitar consultar una base de datos más de una vez. El destinatario de un JWT tampoco necesita contactar con un servidor para validar el token.

<div id="benefits">
  ## Beneficios
</div>

* **Compactos**: Los JWT son de tamaño reducido, lo que los convierte en una buena opción para transmitirse en entornos HTML y HTTP.

<Frame>![JWT codificado comparado con un token SAML](/docs/images/cdy7uua7fh8z/4S6xl4Yvi0H1CUrLB69ZtH/e3e3bf1767d2d29563a99cf71cab158d/comparing-jwt-vs-saml2.png)</Frame>

* **Seguros**: Los JWT pueden usar un par de claves pública/privada en forma de certificado X.509 para la firma. Un JWT también puede firmarse de forma simétrica con un secreto compartido mediante el algoritmo HMAC. Para obtener más información, consulta [Algoritmos de firma](/docs/es/get-started/applications/signing-algorithms).
* **Comunes**: La mayoría de los lenguajes de programación admiten analizadores de JSON.

<div id="usage">
  ## Uso
</div>

* **Autenticación**: Cuando un usuario inicia sesión correctamente con sus credenciales, se devuelve un [token de identidad](/docs/es/secure/tokens/id-tokens). Según la [especificación de OpenID Connect (OIDC)](https://openid.net/specs/openid-connect-core-1_0.html#IDToken), un <Tooltip tip="Token de ID: credencial destinada al propio cliente, en lugar de para acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=ID+token">token de identidad</Tooltip> siempre es un JWT.
* **Autorización**: Una vez que un usuario ha iniciado sesión correctamente, una aplicación puede solicitar acceso a rutas, servicios o recursos (por ejemplo, APIs) en nombre de ese usuario. Para ello, la aplicación debe enviar un <Tooltip tip="Token de acceso: credencial de autorización, en forma de una cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=Access+Token">token de acceso</Tooltip> en cada solicitud, que puede tener el formato de un JWT.
* **Intercambio de información**: Los JWT son una buena forma de transmitir información de manera segura entre partes porque pueden firmarse, lo que significa que puedes tener la certeza de que los remitentes son quienes dicen ser. Además, la estructura de un JWT te permite verificar que el contenido no se ha manipulado.

<div id="security">
  ## Seguridad
</div>

La información contenida en el objeto JSON puede verificarse y considerarse confiable porque está firmada digitalmente. Aunque los JWT también pueden cifrarse para mantener la confidencialidad entre las partes, los JWT emitidos por Auth0 son JSON Web Signatures (JWS), lo que significa que están firmados en lugar de cifrados. Por ello, nos centraremos en los tokens firmados, que permiten verificar la integridad de los claims que contienen, mientras que los tokens cifrados ocultan esos claims a otras partes.

En general, los JWT pueden firmarse usando un secreto (con el algoritmo HMAC) o un par de claves pública/privada mediante RSA o ECDSA (aunque Auth0 solo admite HMAC y RSA). Cuando los tokens se firman con pares de claves pública/privada, la firma también certifica que solo la parte que posee la clave privada es quien los firmó.

Antes de usar un JWT recibido, debe [validarse correctamente mediante su firma](/docs/es/secure/tokens/json-web-tokens/validate-json-web-tokens). Ten en cuenta que validar correctamente un token solo significa que la información que contiene no ha sido modificada por nadie más. Esto no significa que otras personas no hayan podido ver el contenido, que se almacena en texto sin formato. Por eso, nunca debes almacenar información confidencial dentro de un JWT y debes tomar otras medidas para evitar que los JWT sean interceptados, por ejemplo, enviarlos solo a través de HTTPS, seguir las [prácticas recomendadas](/docs/es/secure/tokens/token-best-practices) y usar únicamente bibliotecas seguras y actualizadas.
