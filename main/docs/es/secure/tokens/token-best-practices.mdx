---

Estas son algunas consideraciones básicas que debes tener en cuenta al usar tokens:

* **Mantenlo en secreto. Mantenlo seguro**: La clave de firma debe tratarse como cualquier otra credencial y revelarse solo a los servicios que la necesiten.
* **No agregues datos sensibles al payload**: Los tokens se firman para protegerlos contra manipulaciones y se pueden decodificar fácilmente. Agrega al payload la cantidad mínima indispensable de claims para obtener el mejor rendimiento y la mayor seguridad.
* **Establece un vencimiento para los tokens**: Técnicamente, una vez firmado, un token es válido para siempre, a menos que se cambie la clave de firma o se establezca explícitamente un vencimiento. Esto podría generar problemas, así que ten una estrategia para hacer que los tokens expiren o para revocarlos.
* **Usa HTTPS**: No envíes tokens a través de conexiones que no sean HTTPS, ya que esas solicitudes pueden ser interceptadas y los tokens verse comprometidos.
* **Considera todos tus casos de uso de autorización**: Puede ser necesario agregar un sistema secundario de verificación de tokens que garantice que los tokens se generaron en tu servidor para cumplir con tus requisitos.
* **Almacena y reutiliza:** Reduce los viajes de ida y vuelta innecesarios que amplían la superficie de ataque de tu aplicación y optimiza los límites de tokens de tu plan (cuando corresponda) almacenando <Tooltip tip="Token de acceso: Credencial de autorización, en forma de una cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=access+tokens">tokens de acceso</Tooltip> obtenidos del <Tooltip tip="Servidor de autorización: Servidor centralizado que contribuye a definir los límites del acceso de un usuario. Por ejemplo, tu servidor de autorización puede controlar los datos, tareas y funcionalidades disponibles para un usuario." cta="Ver glosario" href="/docs/es/glossary?term=authorization+server">servidor de autorización</Tooltip>. En lugar de solicitar un token nuevo, usa el token almacenado en llamadas futuras hasta que expire. La forma en que almacenes los tokens dependerá de las características de tu aplicación: las soluciones típicas incluyen bases de datos (para aplicaciones que necesitan realizar llamadas a la API independientemente de si hay una sesión activa) y sesiones HTTP (para aplicaciones cuya ventana de actividad se limita a una sesión interactiva). Para ver un ejemplo de almacenamiento del lado del servidor y reutilización de tokens, consulta [Almacenamiento de tokens](/docs/es/secure/security-guidance/data-security/token-storage).

<div id="tokens-vs-cookies">
  ## Tokens vs. cookies
</div>

Normalmente, las aplicaciones de una sola página (como React, Vue y AngularJS + Node), las aplicaciones móviles nativas (como iOS y Android) y las API web (escritas en Node, Ruby, ASP.NET o una combinación de estos) son las que más se benefician de la autenticación basada en tokens. Las aplicaciones web tradicionales del lado del servidor han usado tradicionalmente la autenticación basada en cookies.

La autenticación basada en tokens se implementa generando un token cuando el usuario se autentica y, después, estableciendo ese token en el encabezado `Authorization` de cada solicitud posterior a tu API. Lo ideal es que ese token sea un estándar, como <Tooltip tip="JSON Web Token (JWT): formato estándar de token de ID (y, a menudo, formato de token de acceso) utilizado para representar claims de forma segura entre dos partes." cta="Ver glosario" href="/docs/es/glossary?term=JSON+web+tokens">JSON web tokens</Tooltip>, ya que encontrarás bibliotecas en la mayoría de las plataformas y no querrás implementar tu propia criptografía.

Con ambos enfoques, puedes obtener la misma cantidad de información del usuario. Esto se controla con el parámetro `scope` que se envía en la solicitud de login (ya sea usando Lock, nuestra biblioteca de JavaScript, o un enlace simple). `scope` es un parámetro del método `.signin({scope: 'openid name email'})`, que termina formando parte de la cadena de consulta en la solicitud de login.

De forma predeterminada, usamos `scope=openid` en la autenticación basada en tokens para evitar tener un token demasiado grande. Puedes controlar qué claims estándar de <Tooltip tip="OpenID: estándar abierto de autenticación que permite a las aplicaciones verificar las identidades de los usuarios sin recopilar ni almacenar información de login." cta="Ver glosario" href="/docs/es/glossary?term=OpenID">OpenID</Tooltip> Connect (OIDC) quieres obtener en el token agregándolos como valores de scope. Por ejemplo, `scope=openid name email family_name address phone_number`. Para obtener más información, consulta [Standard Claims on openid.net](https://openid.net/specs/openid-connect-core-1_0.html#StandardClaims).

Puedes combinar la autenticación basada en tokens con la autenticación basada en cookies. Ten en cuenta que las cookies funcionarán perfectamente si la web app y la API se sirven desde el mismo dominio, así que quizá no necesites autenticación basada en tokens. Si la necesitas, también devolvemos un JWT en el flujo de la web app. Cada uno de nuestros SDK lo hace de forma diferente. Si quieres llamar a tus API desde JavaScript (en lugar de usar la cookie existente), debes configurar los tokens de acceso usando Web Workers o cierres de JavaScript para gestionar la transmisión y el almacenamiento de tokens. Para obtener más información, consulta la sección de escenarios en memoria del navegador de nuestra página de [almacenamiento de tokens](/docs/es/secure/security-guidance/data-security/token-storage).

<div id="refresh-token-usage">
  ## Uso del token de actualización
</div>

Solo puedes obtener un <Tooltip tip="Token de actualización: token utilizado para obtener un token de acceso renovado sin obligar a los usuarios a volver a iniciar sesión." cta="Ver glosario" href="/docs/es/glossary?term=Refresh+token">token de actualización</Tooltip> si estás implementando los siguientes flujos:

* [Flujo de código de autorización](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow/add-login-auth-code-flow)
* [Flujo de código de autorización con Proof Key for Code Exchange (PKCE)](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce/add-login-using-the-authorization-code-flow-with-pkce)
* [Flujo de contraseña del propietario del recurso](/docs/es/get-started/authentication-and-authorization-flow/resource-owner-password-flow)
* [Flujo de autorización de dispositivo](/docs/es/get-started/authentication-and-authorization-flow/device-authorization-flow)

Si limitas el acceso sin conexión a tu API mediante la opción **Allow Offline Access** en [Auth0 Dashboard &gt; Applications &gt; APIs &gt; Settings](https://manage.auth0.com/#/apis), Auth0 no devolverá un token de actualización para la API (aunque incluyas el scope `offline_access` en tu request).

Rules se ejecutará durante el intercambio de token de actualización. Para ejecutar una lógica especial, puedes revisar la propiedad `context.protocol` en tu rule. Si el valor es `oauth2-refresh-token`, la rule se está ejecutando durante el intercambio.

Al intentar obtener un token de actualización, el parámetro <Tooltip tip="Audience: identificador único del audience de un token emitido. Se denomina aud en un token; su valor contiene el ID de una aplicación (ID de cliente) para un token de ID o de una API (identificador de API) para un token de acceso." cta="Ver glosario" href="/docs/es/glossary?term=audience">audience</Tooltip> no está disponible en el objeto `context` de Rules. Si recibes un error al intentar agregar el parámetro audience, verifica que no lo tengas configurado en el token.

Si intentas hacer una redirección con `context.redirect`, el flujo de autenticación devolverá un error.

Si has agregado claims personalizados a tus tokens usando una rule, los claims personalizados aparecerán en los nuevos tokens emitidos al usar un token de actualización mientras tu rule siga vigente. Aunque los nuevos tokens no heredan automáticamente los claims personalizados, Rules se ejecuta durante el flujo del token de actualización, por lo que se ejecutará el mismo código. Esto te permite agregar o cambiar claims personalizados en tokens recién emitidos sin obligar a las aplicaciones autorizadas previamente a obtener un nuevo token de actualización.

<div id="refresh-token-limits">
  ### Límites de los tokens de actualización
</div>

Auth0 limita la cantidad de tokens de actualización activos a 200 por usuario y por aplicación. Este límite solo se aplica a los tokens activos. Si se alcanza el límite y se crea un nuevo token de actualización, el sistema revoca y elimina el token más antiguo de ese usuario y esa aplicación. Los tokens revocados y vencidos no cuentan para este límite.

<div id="automated-tests">
  #### Pruebas automatizadas
</div>

Los tokens de actualización se acumulan debido a las pruebas automatizadas y, por lo general, se usan durante toda la ejecución de la prueba. Para evitar una acumulación de tokens sujeta a los límites de los tokens de actualización, puedes usar la <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip> de Auth0 para eliminar los tokens de actualización innecesarios.

1. [Crea un usuario](https://auth0.com/docs/api/management/v2#!/Users/post_users) con Management API. Usarás este usuario para las pruebas.
2. La respuesta devuelve un `user_id` que debes conservar durante las pruebas para usarlo más adelante.
3. Una vez finalizadas las pruebas, [elimina el usuario](https://auth0.com/docs/api/management/v2#!/Users/delete_users_by_id) mediante Management API. Cuando se elimina el usuario de prueba, también se eliminan los artefactos asociados, incluidos los tokens de actualización.

<Warning>
  Para este caso de uso, no recomendamos usar un ID de usuario estático. Tampoco recomendamos conservar usuarios y artefactos de prueba, ni limpiar los tokens de actualización mediante los endpoints de credenciales de dispositivo, ya que podrías alcanzar los límites de frecuencia de la Management API. Para obtener más información, consulta [Límites de frecuencia de endpoints de Management API.](/docs/es/troubleshoot/customer-support/operational-policies/rate-limit-policy/management-api-endpoint-rate-limits)
</Warning>

Si deseas conservar el usuario de prueba para futuras pruebas:

1. Enumera los tokens de actualización del usuario usando [el endpoint de credenciales de dispositivo de Management API](https://auth0.com/docs/api/management/v2#!/Device_Credentials/get_device_credentials). El endpoint devolverá un máximo de 1000 tokens sin ningún orden específico, independientemente de la cantidad de tokens acumulados o de si se usa paginación.
2. Elimina esas credenciales [con el método DELETE](https://auth0.com/docs/api/management/v2#!/Device_Credentials/delete_device_credentials_by_id).
3. Si el usuario tiene más de 1k tokens, repite el proceso de enumerarlos y eliminarlos hasta que no queden más tokens para el usuario.

<div id="configure-expiring-refresh-tokens">
  #### Configurar la caducidad de los tokens de actualización
</div>

Cuando los usuarios inician sesión en tu aplicación con Auth0 y se solicita `offline_access` en la solicitud de autorización, se emite un nuevo token de actualización para el usuario. Si los usuarios cierran sesión y vuelven a iniciarla en el mismo dispositivo, se emite un nuevo token de actualización. Según cómo tu aplicación almacene y use los tokens de actualización, el token de actualización anterior del primer inicio de sesión puede quedar obsoleto, y lo más probable es que tu aplicación use los nuevos tokens de actualización si ambos se emiten con la misma audience. Para obtener más información, consulta [almacenamiento de tokens](/docs/es/secure/security-guidance/data-security/token-storage).

Para evitar acumular tokens de actualización obsoletos, aunque el límite de tokens de actualización elimina primero el token más antiguo, te recomendamos configurar la caducidad del token de actualización. Tanto los tokens de actualización rotativos como los no rotativos (o reutilizables) pueden configurarse para caducar por inactividad o tras un tiempo absoluto. Ambos valores de caducidad ayudan a eliminar tokens que no están en uso activo y a evitar que se acumulen tokens para el usuario. Para obtener más información, consulta [Configure Refresh Token Expiration](/docs/es/secure/tokens/refresh-tokens/configure-refresh-token-expiration).

<div id="jwt-validation">
  ## Validación de JWT
</div>

Recomendamos encarecidamente que uses middleware o alguna de las bibliotecas de código abierto de terceros ya existentes para analizar y validar JWT. En [JWT.io](https://jwt.io/#libraries-io), puedes encontrar bibliotecas para varias plataformas y lenguajes, como .NET, Python, Java, Ruby, Objective-C, Swift y PHP.

<div id="signing-algorithms">
  ## Algoritmos de firma
</div>

El algoritmo que se utiliza para firmar los tokens emitidos para tu aplicación o API. Una firma forma parte de un JWT y se usa para verificar que el emisor del token es quien dice ser y para garantizar que el mensaje no se haya modificado durante el proceso. Para obtener más información sobre los JWT, consulta [JSON Web Tokens](/docs/es/secure/tokens/json-web-tokens). Para obtener más información sobre las firmas, consulta [JSON Web Token Structure](/docs/es/secure/tokens/json-web-tokens/json-web-token-structure).

Puedes seleccionar entre los siguientes <Tooltip tip="Algoritmo de firma: algoritmo utilizado para firmar digitalmente tokens y garantizar que el token no haya sido manipulado." cta="Ver glosario" href="/docs/es/glossary?term=signing+algorithms">algoritmos de firma</Tooltip>:

* **RS256** (firma RSA con SHA-256): Un algoritmo asimétrico, lo que significa que hay dos claves: una clave pública y una clave privada que debe mantenerse en secreto. Auth0 tiene la clave privada que se usa para generar la firma, y quien consume el JWT obtiene una clave pública de los endpoints de Metadata proporcionados por Auth0 y la utiliza para [validar la firma del JWT](/docs/es/secure/tokens/json-web-tokens/validate-json-web-tokens).
* **HS256** (HMAC con SHA-256): Un algoritmo simétrico, lo que significa que solo hay una clave privada que debe mantenerse en secreto y que se comparte entre ambas partes. Como se usa la misma clave tanto para generar la firma como para validarla, se debe tener cuidado de que la clave no se vea comprometida. Esta clave privada (o secreto) se crea cuando registras tu aplicación (**<Tooltip tip="secreto de cliente: Secret utilizado por un cliente (aplicación) para autenticarse con el servidor de autorización; solo deben conocerlo el cliente y el servidor de autorización, y debe ser lo suficientemente aleatorio como para que no pueda adivinarse." cta="Ver glosario" href="/docs/es/glossary?term=Client+Secret">secreto de cliente</Tooltip>**) o API (**Signing Secret**) y eliges el algoritmo de firma HS256.

La práctica más segura, y nuestra recomendación, es usar **RS256** porque:

* Con RS256, tienes la certeza de que solo el titular de la clave privada (Auth0) puede firmar tokens, mientras que cualquiera puede comprobar si el token es válido usando la clave pública.
* Con RS256, puedes solicitar un token válido para múltiples audiencias.
* Con RS256, si la clave privada se ve comprometida, puedes implementar la rotación de claves sin tener que volver a implementar tu aplicación o API con el nuevo secreto (lo que tendrías que hacer si usaras HS256).
* Con HS256, si la clave secreta se ve comprometida, tendrías que volver a implementar la API con el nuevo secreto.

<div id="signing-keys">
  ## Claves de firma
</div>

Es una buena práctica asumir que puede haber varias claves de firma en tu JWKS. Esto puede parecer innecesario, ya que el endpoint de JWKS de Auth0 normalmente contiene una sola clave de firma; sin embargo, pueden encontrarse varias claves en el JWKS al rotar los certificados de firma.

Te recomendamos almacenar en caché tus claves de firma para mejorar el rendimiento de la aplicación y evitar exceder los límites de frecuencia, pero asegúrate de que, si falla la decodificación de un token, invalides la caché y recuperes nuevas claves de firma antes de volver a intentarlo **solo una** vez más.
