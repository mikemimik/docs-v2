title: Prácticas recomendadas para tokens
---

Estas son algunas consideraciones básicas que debes tener en cuenta al usar tokens:

* **Mantenlo en secreto. Mantenlo seguro**: La clave de firma debe tratarse como cualquier otra credencial y revelarse solo a los servicios que la necesiten.
* **No agregues datos sensibles al payload**: Los tokens se firman para protegerlos contra manipulaciones y se pueden decodificar fácilmente. Agrega la menor cantidad posible de claims al payload para obtener el mejor rendimiento y seguridad.
* **Dales una expiración a los tokens**: Técnicamente, una vez que un token está firmado, es válido para siempre, a menos que se cambie la clave de firma o se establezca explícitamente una expiración. Esto puede generar problemas, así que conviene contar con una estrategia para hacer que los tokens expiren o revocarlos.
* **Usa HTTPS**: No envíes tokens a través de conexiones que no sean HTTPS, ya que esas solicitudes pueden ser interceptadas y los tokens verse comprometidos.
* **Considera todos tus casos de uso de autorización**: Puede ser necesario agregar un sistema secundario de verificación de tokens que garantice que los tokens se generaron en tu servidor para cumplir con tus requisitos.
* **Almacena y reutiliza:** Reduce los intercambios innecesarios que amplían la superficie de ataque de tu aplicación y optimiza los límites de tokens de tu plan (cuando corresponda) almacenando <Tooltip tip="Token de acceso: Credencial de autorización, en forma de una cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=access+tokens">tokens de acceso</Tooltip> obtenidos del <Tooltip tip="Servidor de autorización: Servidor centralizado que contribuye a definir los límites del acceso de un usuario. Por ejemplo, tu servidor de autorización puede controlar los datos, las tareas y las funcionalidades disponibles para un usuario." cta="Ver glosario" href="/docs/es/glossary?term=authorization+server">servidor de autorización</Tooltip>. En lugar de solicitar un nuevo token, usa el token almacenado en futuras llamadas hasta que expire. La forma de almacenar los tokens dependerá de las características de tu aplicación: las soluciones habituales incluyen bases de datos (para aplicaciones que necesitan realizar llamadas a la API independientemente de si hay una sesión) y sesiones HTTP (para aplicaciones cuya ventana de actividad se limita a una sesión interactiva). Para ver un ejemplo de almacenamiento del lado del servidor y reutilización de tokens, consulta [almacenamiento de tokens](/docs/es/secure/security-guidance/data-security/token-storage).

<div id="tokens-vs-cookies">
  ## Tokens vs. Cookies
</div>

Normalmente, las aplicaciones de página única (como React, Vue y AngularJS + Node), las aplicaciones móviles nativas (como iOS y Android) y las API web (escritas en Node, Ruby, ASP.NET o una combinación de estos) son las que más se benefician de la autenticación basada en tokens. Las aplicaciones web tradicionales del servidor han usado tradicionalmente autenticación basada en cookies.

La autenticación basada en tokens se implementa generando un token cuando el usuario se autentica y luego estableciendo ese token en el encabezado `Authorization` de cada solicitud posterior a tu API. Conviene que ese token sea algo estándar, como <Tooltip tip="JSON Web Token (JWT): formato estándar de token de ID (y, a menudo, formato de token de acceso) utilizado para representar claims de forma segura entre dos partes." cta="Ver glosario" href="/docs/es/glossary?term=JSON+web+tokens">JSON web tokens</Tooltip>, ya que encontrarás bibliotecas en la mayoría de las plataformas y no querrás implementar tu propia criptografía.

Con ambos enfoques, puedes obtener la misma cantidad de información del usuario. Eso se controla mediante el parámetro `scope` enviado en la solicitud de login (ya sea usando Lock, nuestra biblioteca de JavaScript, o un enlace simple). `scope` es un parámetro del método `.signin({scope: 'openid name email'})`, que termina formando parte de la cadena de consulta en la solicitud de login.

De forma predeterminada, usamos `scope=openid` en la autenticación basada en tokens para evitar tener un token demasiado grande. Puedes controlar cualquier claim estándar de <Tooltip tip="OpenID: estándar abierto de autenticación que permite que las aplicaciones verifiquen las identidades de los usuarios sin recopilar ni almacenar información de login." cta="Ver glosario" href="/docs/es/glossary?term=OpenID">OpenID</Tooltip> Connect (OIDC) que quieras obtener en el token agregándolos como valores de `scope`. Por ejemplo, `scope=openid name email family_name address phone_number`. Para obtener más información, consulta [Standard Claims on openid.net](https://openid.net/specs/openid-connect-core-1_0.html#StandardClaims).

Puedes combinar la autenticación basada en tokens con la autenticación basada en cookies. Ten en cuenta que las cookies funcionarán perfectamente si la aplicación web y la API se sirven desde el mismo dominio, por lo que quizá no necesites autenticación basada en tokens. Si la necesitas, también devolvemos un JWT en el flujo de la aplicación web. Cada uno de nuestros SDKs lo hace de manera diferente. Si quieres llamar a tus API desde JavaScript (en lugar de usar la cookie existente), entonces debes configurar los tokens de acceso usando Web Workers o closures de JavaScript para gestionar la transmisión y el almacenamiento de tokens. Para obtener más información, lee la sección Browser in-memory scenarios de nuestra página [almacenamiento de tokens](/docs/es/secure/security-guidance/data-security/token-storage).

<div id="refresh-token-usage">
  ## Uso del token de actualización
</div>

Solo puedes obtener un <Tooltip tip="Token de actualización: token utilizado para obtener un token de acceso renovado sin obligar a los usuarios a volver a iniciar sesión." cta="Ver glosario" href="/docs/es/glossary?term=Refresh+token">token de actualización</Tooltip> si estás implementando los siguientes flujos:

* [Flujo de código de autorización](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow/add-login-auth-code-flow)
* [Flujo de código de autorización con Proof Key for Code Exchange (PKCE)](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce/add-login-using-the-authorization-code-flow-with-pkce)
* [Flujo de contraseña del propietario del recurso](/docs/es/get-started/authentication-and-authorization-flow/resource-owner-password-flow)
* [Flujo de autorización de dispositivos](/docs/es/get-started/authentication-and-authorization-flow/device-authorization-flow)

Si restringes el acceso sin conexión a tu API, una medida de protección configurada con el interruptor **Allow Offline Access** en [Auth0 Dashboard &gt; Applications &gt; APIs &gt; Settings](https://manage.auth0.com/#/apis), Auth0 no devolverá un token de actualización para la API (aunque incluyas el scope `offline_access` en tu solicitud).

Las Rules se ejecutarán durante el intercambio de token de actualización. Para ejecutar una lógica especial, puedes revisar la propiedad `context.protocol` en tu rule. Si el valor es `oauth2-refresh-token`, la rule se está ejecutando durante el intercambio.

Al intentar obtener un token de actualización, el parámetro <Tooltip tip="Audience: identificador único de la audiencia de un token emitido. Se denomina aud en un token, y su valor contiene el ID de una aplicación (ID de cliente) para un token de ID o de una API (identificador de API) para un token de acceso." cta="Ver glosario" href="/docs/es/glossary?term=audience">audience</Tooltip> no está disponible en el objeto `context` de Rules. Si recibes un error al intentar agregar el parámetro audience, verifica que no lo tengas configurado en el token.

Si intentas hacer una redirección con `context.redirect`, el flujo de autenticación devolverá un error.

Si has agregado claims personalizadas a tus tokens usando una rule, las claims personalizadas aparecerán en los nuevos tokens emitidos al usar un token de actualización mientras tu rule siga vigente. Aunque los nuevos tokens no heredan automáticamente las claims personalizadas, las Rules se ejecutan durante el flujo de token de actualización, por lo que se ejecutará el mismo código. Esto te permite agregar o cambiar claims personalizadas en tokens recién emitidos sin obligar a las aplicaciones autorizadas previamente a obtener un nuevo token de actualización.

<div id="refresh-token-limits">
  ### Límites de los tokens de actualización
</div>

Auth0 limita la cantidad de tokens de actualización activos a 200 por usuario y por aplicación. Este límite solo se aplica a los tokens activos. Si se alcanza el límite y se crea un nuevo token de actualización, el sistema revoca y elimina el token más antiguo de ese usuario y esa aplicación. Los tokens revocados y los tokens vencidos no cuentan para este límite.

<div id="automated-tests">
  #### Pruebas automatizadas
</div>

Los tokens de actualización se acumulan debido a las pruebas automatizadas y, por lo general, se usan durante toda la duración de la prueba. Para evitar una acumulación de tokens sujeta a los límites de los tokens de actualización, puedes usar la <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip> de Auth0 para eliminar los tokens de actualización innecesarios.

1. [Crea un usuario](https://auth0.com/docs/api/management/v2#!/Users/post_users) con Management API. Usarás este usuario para las pruebas.
2. La respuesta devuelve un `user_id` que debes guardar durante las pruebas para usarlo más adelante.
3. Cuando las pruebas hayan finalizado, [elimina el usuario](https://auth0.com/docs/api/management/v2#!/Users/delete_users_by_id) mediante Management API. Al eliminarse el usuario de prueba, también se eliminan los artefactos asociados, incluidos los tokens de actualización.

<Warning>
  Para este caso de uso, no recomendamos usar un ID de usuario estático. Tampoco recomendamos conservar los usuarios y artefactos de prueba ni limpiar los tokens de actualización usando los endpoints de credenciales de dispositivo, ya que podrías alcanzar los límites de frecuencia de la Management API. Para obtener más información, consulta [Límites de frecuencia de endpoints de Management API.](/docs/es/troubleshoot/customer-support/operational-policies/rate-limit-policy/management-api-endpoint-rate-limits)
</Warning>

Si quieres conservar el usuario de prueba para futuras pruebas:

1. Enumera los tokens de actualización del usuario usando [el endpoint de credenciales de dispositivo de Management API](https://auth0.com/docs/api/management/v2#!/Device_Credentials/get_device_credentials). El endpoint devolverá un máximo de 1000 tokens sin un orden específico, independientemente de la cantidad de tokens acumulados o del uso de paginación.
2. Elimina esas credenciales [usando el método DELETE](https://auth0.com/docs/api/management/v2#!/Device_Credentials/delete_device_credentials_by_id).
3. Si el usuario tiene más de 1k tokens, repite el proceso de enumerar y eliminar tokens hasta que no queden más tokens para el usuario.

<div id="configure-expiring-refresh-tokens">
  #### Configurar la expiración de los tokens de actualización
</div>

Cuando los usuarios inician sesión en tu aplicación con Auth0 y se solicita `offline_access` en la solicitud de autorización, se emite un nuevo token de actualización para el usuario. Si los usuarios cierran sesión y vuelven a iniciar sesión en el mismo device, se emite un nuevo token de actualización. Según cómo tu aplicación almacene y use los tokens de actualización, es posible que el token de actualización anterior del primer inicio de sesión quede obsoleto, y lo más probable es que tu aplicación use los nuevos tokens de actualización si ambos se emiten con el mismo audience. Para obtener más información, consulta [almacenamiento de tokens](/docs/es/secure/security-guidance/data-security/token-storage).

Para evitar acumular tokens de actualización obsoletos, aunque el límite de tokens de actualización elimina primero el token más antiguo, te recomendamos configurar la expiración del token de actualización. Tanto los tokens de actualización rotativos como los no rotativos (o reutilizables) pueden configurarse para que expiren con valores de expiración por inactividad o absoluta. Ambos valores de expiración ayudan a eliminar los tokens que no están en uso activo y a evitar que se acumulen tokens para el usuario. Para obtener más información, consulta [Configurar la expiración del token de actualización](/docs/es/secure/tokens/refresh-tokens/configure-refresh-token-expiration).

<div id="jwt-validation">
  ## Validación de JWT
</div>

Recomendamos encarecidamente utilizar middleware o alguna de las bibliotecas de terceros de código abierto disponibles para analizar y validar JWT. En [JWT.io](https://jwt.io/#libraries-io), puede encontrar bibliotecas para varias plataformas y lenguajes de programación, como .NET, Python, Java, Ruby, Objective-C, Swift y PHP.

<div id="signing-algorithms">
  ## Algoritmos de firma
</div>

El algoritmo utilizado para firmar los tokens emitidos para tu aplicación o API. Una firma forma parte de un JWT y se usa para verificar que el remitente del token es quien dice ser y para garantizar que el mensaje no haya sido alterado. Para obtener más información sobre los JWT, lee [JSON Web Tokens](/docs/es/secure/tokens/json-web-tokens). Para obtener más información sobre las firmas, lee [JSON Web Token Structure](/docs/es/secure/tokens/json-web-tokens/json-web-token-structure).

Puedes seleccionar entre los siguientes <Tooltip tip="Algoritmo de firma: algoritmo utilizado para firmar digitalmente tokens y garantizar que el token no haya sido manipulado." cta="Ver glosario" href="/docs/es/glossary?term=signing+algorithms">algoritmos de firma</Tooltip>:

* **RS256** (firma RSA con SHA-256): un algoritmo asimétrico, lo que significa que hay dos claves: una clave pública y una clave privada que debe mantenerse en secreto. Auth0 tiene la clave privada que se utiliza para generar la firma, y quien consume el JWT recupera una clave pública de los endpoints de metadatos proporcionados por Auth0 y la usa para [validar la firma del JWT](/docs/es/secure/tokens/json-web-tokens/validate-json-web-tokens).
* **HS256** (HMAC con SHA-256): un algoritmo simétrico, lo que significa que solo hay una clave privada que debe mantenerse en secreto y que se comparte entre las dos partes. Como la misma clave se usa tanto para generar la firma como para validarla, se debe tener cuidado de garantizar que la clave no se vea comprometida. Esta clave privada (o secreto) se crea cuando registras tu aplicación (**<Tooltip tip="secreto de cliente: secreto que usa un cliente (aplicación) para autenticarse con el servidor de autorización; solo deberían conocerlo el cliente y el servidor de autorización, y debe ser lo suficientemente aleatorio como para que no pueda adivinarse." cta="Ver glosario" href="/docs/es/glossary?term=Client+Secret">secreto de cliente</Tooltip>**) o API (**secreto de firma**) y eliges el algoritmo de firma HS256.

La práctica más segura, y nuestra recomendación, es usar **RS256** porque:

* Con RS256, tienes la certeza de que solo el titular de la clave privada (Auth0) puede firmar tokens, mientras que cualquiera puede comprobar si el token es válido usando la clave pública.
* Con RS256, puedes solicitar un token válido para múltiples audiencias.
* Con RS256, si la clave privada se ve comprometida, puedes implementar una rotación de claves sin tener que volver a desplegar tu aplicación o API con el nuevo secreto (algo que tendrías que hacer si usaras HS256).
* Con HS256, si la clave secreta se ve comprometida, tendrías que volver a desplegar la API con el nuevo secreto.

<div id="signing-keys">
  ## Claves de firma
</div>

Es una buena práctica asumir que puede haber varias claves de firma en tu JWKS. Esto puede parecer innecesario, ya que el endpoint de JWKS de Auth0 normalmente contiene una sola clave de firma; sin embargo, pueden encontrarse varias claves en el JWKS al rotar los certificados de firma.

Te recomendamos almacenar en caché tus claves de firma para mejorar el rendimiento de la aplicación y evitar alcanzar los límites de frecuencia, pero debes asegurarte de que, si falla la decodificación de un token, invalides la caché y recuperes nuevas claves de firma antes de volver a intentarlo **solo una** vez más.
