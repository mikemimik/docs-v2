---

Estas son algunas consideraciones básicas que debes tener en cuenta al usar tokens:

* **Mantenlo en secreto. Mantenlo seguro**: La clave de firma debe tratarse como cualquier otra credencial y revelarse solo a los servicios que la necesiten.
* **No agregues datos sensibles al payload**: Los tokens se firman para protegerlos contra manipulaciones y se pueden decodificar fácilmente. Agrega al payload la cantidad mínima indispensable de claims para obtener el mejor rendimiento y seguridad.
* **Haz que los tokens expiren**: Técnicamente, una vez que un token está firmado, es válido para siempre, a menos que se cambie la clave de firma o se establezca explícitamente un vencimiento. Esto puede generar problemas, así que define una estrategia para que los tokens expiren y/o puedan revocarse.
* **Usa HTTPS**: No envíes tokens a través de conexiones que no sean HTTPS, ya que esas solicitudes pueden ser interceptadas y los tokens verse comprometidos.
* **Considera todos tus casos de uso de autorización**: Puede ser necesario agregar un sistema secundario de verificación de tokens que garantice que los tokens se generaron en tu servidor para cumplir con tus requisitos.
* **Almacena y reutiliza:** Reduce los intercambios innecesarios que amplían la superficie de ataque de tu aplicación y optimiza los límites de tokens de tu plan (cuando corresponda) almacenando <Tooltip tip="Access Token: Credencial de autorización, en forma de una cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=access+tokens">tokens de acceso</Tooltip> obtenidos del <Tooltip tip="Authorization Server: Servidor centralizado que contribuye a definir los límites del acceso de un usuario. Por ejemplo, tu servidor de autorización puede controlar los datos, las tareas y las funcionalidades disponibles para un usuario." cta="Ver glosario" href="/docs/es/glossary?term=authorization+server">servidor de autorización</Tooltip>. En lugar de solicitar un token nuevo, usa el token almacenado en llamadas futuras hasta que caduque. La forma de almacenar los tokens dependerá de las características de tu aplicación: las soluciones típicas incluyen bases de datos (para aplicaciones que necesitan realizar llamadas a la API independientemente de que exista una sesión) y sesiones HTTP (para aplicaciones cuya ventana de actividad se limita a una sesión interactiva). Para ver un ejemplo de almacenamiento del lado del servidor y reutilización de tokens, consulta [almacenamiento de tokens](/docs/es/secure/security-guidance/data-security/token-storage).

<div id="tokens-vs-cookies">
  ## Tokens vs. cookies
</div>

Normalmente, las aplicaciones de página única (como React, Vue y AngularJS + Node), las aplicaciones móviles nativas (como iOS y Android) y las Web API (escritas en Node, Ruby, ASP.NET o una combinación de estos) son las que más se benefician de la autenticación basada en tokens. Las aplicaciones web tradicionales del servidor han usado tradicionalmente autenticación basada en cookies.

La autenticación basada en tokens se implementa generando un token cuando el usuario se autentica y, luego, enviando ese token en el encabezado `Authorization` de cada solicitud posterior a tu API. Lo ideal es que ese token sea un estándar, como <Tooltip tip="JSON Web Token (JWT): Formato estándar de token de ID (y, a menudo, formato de Access Token) usado para representar claims de forma segura entre dos partes." cta="Ver glosario" href="/docs/es/glossary?term=JSON+web+tokens">JSON Web Tokens</Tooltip>, ya que encontrarás bibliotecas en la mayoría de las plataformas y no querrás implementar tu propia criptografía.

Con ambos enfoques, puedes obtener la misma cantidad de información del usuario. Esto se controla mediante el parámetro `scope` enviado en la solicitud de inicio de sesión (ya sea usando Lock, nuestra biblioteca de JavaScript, o un enlace simple). `scope` es un parámetro del método `.signin({scope: 'openid name email'})`, que termina formando parte de la cadena de consulta en la solicitud de inicio de sesión.

De forma predeterminada, usamos `scope=openid` en la autenticación basada en tokens para evitar que el token sea demasiado grande. Puedes controlar qué claims estándar de <Tooltip tip="OpenID: Estándar abierto de autenticación que permite a las aplicaciones verificar las identidades de los usuarios sin recopilar ni almacenar información de inicio de sesión." cta="Ver glosario" href="/docs/es/glossary?term=OpenID">OpenID</Tooltip> Connect (OIDC) quieres obtener en el token agregándolos como valores de scope. Por ejemplo, `scope=openid name email family_name address phone_number`. Para obtener más información, consulta [Standard Claims on openid.net](https://openid.net/specs/openid-connect-core-1_0.html#StandardClaims).

Puedes combinar la autenticación basada en tokens con la autenticación basada en cookies. Ten en cuenta que las cookies funcionarán perfectamente si la aplicación web y la API se sirven desde el mismo dominio, por lo que quizá no necesites autenticación basada en tokens. Si la necesitas, también devolvemos un JWT en el flujo de la aplicación web. Cada uno de nuestros SDKs lo hace de forma distinta. Si quieres llamar a tus API desde JavaScript (en lugar de usar la cookie existente), entonces debes configurar los tokens de acceso usando Web Workers o cierres de JavaScript para gestionar la transmisión y el almacenamiento de tokens. Para obtener más información, consulta la sección Browser in-memory scenarios de nuestra página [almacenamiento de tokens](/docs/es/secure/security-guidance/data-security/token-storage).

<div id="refresh-token-usage">
  ## Uso del token de actualización
</div>

Solo puedes obtener un <Tooltip tip="Token de actualización: token usado para obtener un nuevo token de acceso sin obligar a los usuarios a volver a iniciar sesión." cta="Ver glosario" href="/docs/es/glossary?term=Refresh+token">token de actualización</Tooltip> si estás implementando uno de los siguientes flujos:

* [Flujo de código de autorización](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow/add-login-auth-code-flow)
* [Flujo de código de autorización con Proof Key for Code Exchange (PKCE)](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce/add-login-using-the-authorization-code-flow-with-pkce)
* [Flujo de contraseña del propietario del recurso](/docs/es/get-started/authentication-and-authorization-flow/resource-owner-password-flow)
* [Flujo de autorización de dispositivos](/docs/es/get-started/authentication-and-authorization-flow/device-authorization-flow)

Si limitas el acceso sin conexión a tu API, una medida de protección configurada mediante el interruptor **Allow Offline Access** en [Auth0 Dashboard &gt; Applications &gt; APIs &gt; Settings](https://manage.auth0.com/#/apis), Auth0 no devolverá un token de actualización para la API (aunque incluyas el scope `offline_access` en tu solicitud).

Rules se ejecutará durante el intercambio de token de actualización. Para ejecutar lógica especial, puedes revisar la propiedad `context.protocol` en tu rule. Si el valor es `oauth2-refresh-token`, la rule se está ejecutando durante el intercambio.

Al intentar obtener un token de actualización, el parámetro <Tooltip tip="Audience: identificador único de la audiencia de un token emitido. Se denomina aud en un token; su valor contiene el ID de una aplicación (ID de cliente) para un token de ID o de una API (identificador de API) para un token de acceso." cta="Ver glosario" href="/docs/es/glossary?term=audience">audience</Tooltip> no está disponible en el objeto `context` de Rules. Si recibes un error al intentar añadir el parámetro audience, verifica que no esté configurado en el token.

Si intentas hacer una redirección con `context.redirect`, el flujo de autenticación devolverá un error.

Si has agregado custom claims a tus tokens usando una rule, las custom claims aparecerán en los nuevos tokens emitidos al usar un token de actualización mientras tu rule siga vigente. Aunque los nuevos tokens no heredan automáticamente las custom claims, las rules se ejecutan durante el flujo del token de actualización, por lo que se ejecutará el mismo código. Esto te permite agregar o cambiar custom claims en los tokens recién emitidos sin obligar a las applications previamente autorizadas a obtener un nuevo token de actualización.

<div id="refresh-token-limits">
  ### Límites de los tokens de actualización
</div>

Auth0 limita la cantidad de tokens de actualización activos a 200 por usuario y por aplicación. Este límite solo se aplica a los tokens activos. Si se alcanza el límite y se crea un nuevo token de actualización, el sistema revoca y elimina el token más antiguo de ese usuario y esa aplicación. Los tokens revocados y expirados no cuentan para este límite.

<div id="automated-tests">
  #### Pruebas automatizadas
</div>

Los tokens de actualización se acumulan debido a las pruebas automatizadas y, por lo general, se usan durante la vigencia de la prueba. Para evitar una acumulación de tokens sujeta a los límites de los tokens de actualización, puede usar la <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip> de Auth0 para eliminar los tokens de actualización innecesarios.

1. [Create a user](https://auth0.com/docs/api/management/v2#!/Users/post_users) con Management API. Usará este usuario para las pruebas.
2. La respuesta devuelve un `user_id` que deberá conservar durante las pruebas para usarlo más adelante.
3. Cuando finalicen las pruebas, [elimine el usuario](https://auth0.com/docs/api/management/v2#!/Users/delete_users_by_id) mediante Management API. Cuando se elimina el usuario de prueba, también se eliminan los elementos asociados, incluidos los tokens de actualización.

<Warning>
  Para este caso de uso, no recomendamos usar un ID de usuario estático. Tampoco recomendamos conservar usuarios de prueba ni elementos asociados, ni limpiar los tokens de actualización mediante los endpoints de credenciales de device, ya que podría alcanzar los límites de frecuencia de Management API. Para obtener más información, consulte [los límites de frecuencia de los endpoints de Management API.](/docs/es/troubleshoot/customer-support/operational-policies/rate-limit-policy/management-api-endpoint-rate-limits)
</Warning>

Si desea conservar el usuario de prueba para futuras pruebas:

1. Enumere los tokens de actualización del usuario mediante el [endpoint de credenciales de device de Management API](https://auth0.com/docs/api/management/v2#!/Device_Credentials/get_device_credentials). El endpoint devolverá un máximo de 1000 tokens sin un orden específico, independientemente de la cantidad de tokens acumulados o del uso de la paginación.
2. Elimine esas credenciales [con el método DELETE](https://auth0.com/docs/api/management/v2#!/Device_Credentials/delete_device_credentials_by_id).
3. Si el usuario tiene más de 1k tokens, repita el proceso de enumerarlos y eliminarlos hasta que no queden más tokens para ese usuario.

<div id="configure-expiring-refresh-tokens">
  #### Configurar la expiración de los tokens de actualización
</div>

Cuando los usuarios inician sesión en tu aplicación con Auth0 y se solicita `offline_access` en la solicitud de autorización, se emite un nuevo token de actualización para el usuario. Si los usuarios cierran sesión y vuelven a iniciarla en el mismo device, se emite un nuevo token de actualización. Según cómo tu aplicación almacene y use los tokens de actualización, es posible que el token de actualización anterior, emitido durante el primer inicio de sesión, quede obsoleto, y lo más probable es que tu aplicación use los nuevos tokens de actualización si ambos se emiten con la misma audiencia. Para obtener más información, consulta [Almacenamiento de tokens](/docs/es/secure/security-guidance/data-security/token-storage).

Para evitar acumular tokens de actualización obsoletos, aunque el límite de tokens de actualización elimina primero el token más antiguo, te recomendamos configurar la expiración del token de actualización. Tanto los tokens de actualización rotativos como los no rotativos (o reutilizables) pueden configurarse para que expiren por inactividad o por expiración absoluta. Ambos valores de expiración ayudan a eliminar los tokens que no se usan activamente y a evitar que se acumulen para el usuario. Para obtener más información, consulta [Configurar la expiración del token de actualización](/docs/es/secure/tokens/refresh-tokens/configure-refresh-token-expiration).

<div id="jwt-validation">
  ## Validación de JWT
</div>

Recomendamos encarecidamente usar middleware o alguna de las bibliotecas de terceros de código abierto ya existentes para analizar y validar JWT. En [JWT.io](https://jwt.io/#libraries-io), puedes encontrar bibliotecas para varias plataformas y lenguajes de programación, como .NET, Python, Java, Ruby, Objective-C, Swift y PHP.

<div id="signing-algorithms">
  ## Algoritmos de firma
</div>

El algoritmo que se usa para firmar los tokens emitidos para tu aplicación o API. Una firma forma parte de un JWT y se usa para verificar que quien envía el token es quien dice ser, y para garantizar que el mensaje no fue modificado en el proceso. Para obtener más información sobre los JWT, lee [JSON Web Tokens](/docs/es/secure/tokens/json-web-tokens). Para obtener más información sobre las firmas, lee [JSON Web Token Structure](/docs/es/secure/tokens/json-web-tokens/json-web-token-structure).

Puedes seleccionar entre los siguientes <Tooltip tip="Algoritmo de firma: algoritmo usado para firmar digitalmente tokens y garantizar que el token no haya sido manipulado." cta="Ver glosario" href="/docs/es/glossary?term=signing+algorithms">algoritmos de firma</Tooltip>:

* **RS256** (firma RSA con SHA-256): Un algoritmo asimétrico, lo que significa que hay dos claves: una clave pública y una clave privada que debe mantenerse en secreto. Auth0 tiene la clave privada que se usa para generar la firma, y quien consume el JWT obtiene una clave pública de los endpoints de Metadata proporcionados por Auth0 y la usa para [validar la firma del JWT](/docs/es/secure/tokens/json-web-tokens/validate-json-web-tokens).
* **HS256** (HMAC con SHA-256): Un algoritmo simétrico, lo que significa que solo hay una clave privada que debe mantenerse en secreto, y se comparte entre ambas partes. Como la misma clave se usa tanto para generar la firma como para validarla, debes asegurarte de que la clave no se vea comprometida. Esta clave privada (o secreto) se crea cuando registras tu aplicación (**<Tooltip tip="Secreto de cliente: Secret usado por un cliente (aplicación) para autenticarse con el servidor de autorización; solo deberían conocerlo el cliente y el servidor de autorización, y debe ser lo suficientemente aleatorio como para que no se pueda adivinar." cta="Ver glosario" href="/docs/es/glossary?term=Client+Secret">secreto de cliente</Tooltip>**) o API (**Signing Secret**) y eliges el algoritmo de firma HS256.

La práctica más segura, y nuestra recomendación, es usar **RS256** porque:

* Con RS256, tienes la certeza de que solo quien posee la clave privada (Auth0) puede firmar tokens, mientras que cualquier persona puede comprobar si el token es válido usando la clave pública.
* Con RS256, puedes solicitar un token válido para múltiples audiencias.
* Con RS256, si la clave privada se ve comprometida, puedes implementar una rotación de claves sin tener que volver a desplegar tu aplicación o API con el nuevo secreto (algo que tendrías que hacer si usaras HS256).
* Con HS256, si la clave secreta se ve comprometida, tendrías que volver a desplegar la API con el nuevo secreto.

<div id="signing-keys">
  ## Claves de firma
</div>

Es una buena práctica asumir que puede haber varias claves de firma en tu JWKS. Esto puede parecer innecesario, ya que el endpoint JWKS de Auth0 normalmente contiene una sola clave de firma; sin embargo, es posible encontrar varias claves en el JWKS al rotar los certificados de firma.

Te recomendamos almacenar en caché tus claves de firma para mejorar el rendimiento de la aplicación y evitar alcanzar los límites de frecuencia, pero debes asegurarte de que, si falla la decodificación de un token, invalides la caché y recuperes nuevas claves de firma antes de volver a intentarlo **solo una** vez más.
