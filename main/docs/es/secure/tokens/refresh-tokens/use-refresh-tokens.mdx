<Tooltip tip="Token de actualización: token que se utiliza para obtener un nuevo token de acceso sin obligar a los usuarios a volver a iniciar sesión." cta="Ver glosario" href="/docs/es/glossary?term=Refresh+tokens">Los tokens de actualización</Tooltip> se utilizan para solicitar un nuevo <Tooltip tip="Token de actualización: token que se utiliza para obtener un nuevo token de acceso sin obligar a los usuarios a volver a iniciar sesión." cta="Ver glosario" href="/docs/es/glossary?term=access+token">token de acceso</Tooltip> y/o <Tooltip tip="Token de acceso: credencial de Authorization, en forma de una cadena opaca o JWT, que se utiliza para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=ID+token">token de identidad</Tooltip> para un usuario sin necesidad de que vuelva a autenticarse.

Por lo general, debería solicitar un nuevo token de acceso antes de que caduque el anterior (para evitar cualquier interrupción del servicio), pero no cada vez que llame a una API, ya que los intercambios de tokens están sujetos a nuestra [Política de límite de frecuencia](/docs/es/troubleshoot/customer-support/operational-policies/rate-limit-policy).

También puede usar un token de actualización para solicitar un nuevo token de identidad para un usuario, y debería hacerlo si necesita actualizar los claims del token de identidad.

<div id="call-the-api">
  ## Llama a la API
</div>

Para intercambiar el token de actualización que recibiste durante la autenticación por un nuevo token de acceso, llama al [endpoint Get Token](https://auth0.com/docs/api/authentication#refresh-token) de Auth0 Authentication API.

Para obtener más información sobre los métodos de autenticación disponibles para Authentication API, lee [Métodos de autenticación](https://auth0.com/docs/api/authentication#authentication-methods).

<div id="use-basic-authentication">
  ### Usa la autenticación Basic
</div>

<AuthCodeGroup>
  ```bash cURL
  curl --request POST \
    --url 'https://{yourDomain}/oauth/token' \
    --header 'authorization: Basic {yourApplicationCredentials}' \
    --header 'content-type: application/x-www-form-urlencoded' \
    --data grant_type=refresh_token \
    --data 'client_id={yourClientId}' \
    --data 'refresh_token={yourRefreshToken}'
  ```

  ```csharp C#
  var client = new RestClient("https://{yourDomain}/oauth/token");
  var request = new RestRequest(Method.POST);
  request.AddHeader("content-type", "application/x-www-form-urlencoded");
  request.AddHeader("authorization", "Basic {yourApplicationCredentials}");
  request.AddParameter("application/x-www-form-urlencoded", "grant_type=refresh_token&client_id={yourClientId}&refresh_token=%7ByourRefreshToken%7D", ParameterType.RequestBody);
  IRestResponse response = client.Execute(request);
  ```

  ```go Go
  package main

  import (
  	"fmt"
  	"strings"
  	"net/http"
  	"io/ioutil"
  )

  func main() {

  	url := "https://{yourDomain}/oauth/token"

  	payload := strings.NewReader("grant_type=refresh_token&client_id={yourClientId}&refresh_token=%7ByourRefreshToken%7D")

  	req, _ := http.NewRequest("POST", url, payload)

  	req.Header.Add("content-type", "application/x-www-form-urlencoded")
  	req.Header.Add("authorization", "Basic {yourApplicationCredentials}")

  	res, _ := http.DefaultClient.Do(req)

  	defer res.Body.Close()
  	body, _ := ioutil.ReadAll(res.Body)

  	fmt.Println(res)
  	fmt.Println(string(body))

  }
  ```

  ```java Java
  HttpResponse<String> response = Unirest.post("https://{yourDomain}/oauth/token")

    .body("grant_type=refresh_token&client_id={yourClientId}&refresh_token=%7ByourRefreshToken%7D")
    .asString();
  ```


  ```php PHP
    echo "cURL Error #:" . $err;
  ```

<div id="parameter-definition">
  ### Definición de parámetros
</div>

| Parámetro       | Descripción                                                                                                                                           |
| --------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------- |
| `grant_type`    | Tipo de concesión que se va a usar.                                                                                                                   |
| `client_id`     | ID de cliente de la aplicación.                                                                                                                       |
| `client_secret` | (Opcional) secreto de cliente de la aplicación. Solo se requiere para aplicaciones confidenciales que usan el método Post de autenticación de tokens. |
| `refresh_token` | Token de actualización que se va a intercambiar.                                                                                                      |

La respuesta incluirá un nuevo token de acceso, su tipo, su tiempo de validez (en segundos) y los scopes concedidos. Si el scope del token inicial incluía `openid`, la respuesta también incluirá un nuevo token de identidad.

```json lines
{
      "access_token": "eyJ...MoQ",
      "expires_in": 86400,
      "scope": "openid offline_access",
      "id_token": "eyJ...0NE",
      "token_type": "Bearer"
    }
```

<div id="bypass-mfa">
  ## Omitir MFA
</div>

Si [Autenticación multifactor (MFA)](/docs/es/secure/multi-factor-authentication) está habilitada y el flujo de intercambio de token de actualización falla, puedes usar el siguiente código de Action para omitir la lógica de <Tooltip tip="Autenticación multifactor (MFA): proceso de autenticación de usuario que utiliza un factor adicional, además del nombre de usuario y la contraseña, como un código por SMS." cta="Ver glosario" href="/docs/es/glossary?term=MFA">MFA</Tooltip>:

```js lines
exports.onExecutePostLogin = async (event, api) => {
  // Esta Action te permitirá omitir la lógica de MFA para el flujo de intercambio de token de actualización.

  if (event.transaction.protocol === "oauth2-refresh-token") {
    return;
  }

  //  Agrega tu lógica de MFA
  //  Por ejemplo: api.multifactor.enable("any");
};
```

Puede personalizar el ejemplo de código cuando sea necesario ejecutar u omitir una lógica distinta según el flujo o protocolo actual.

<div id="customize-mfa">
  ## Personalizar MFA
</div>

<Warning>
  La MFA personalizable con los flujos de contraseña del propietario del recurso, Embedded o token de actualización está en acceso anticipado. Al usar esta funcionalidad, aceptas los términos de prueba gratuita aplicables del [Master Subscription Agreement](https://www.okta.com/legal/) de Okta. Para obtener más información sobre las etapas de lanzamiento de Auth0, consulta [Product Release Stages](/docs/es/troubleshoot/product-lifecycle/product-release-stages). Para participar en el acceso anticipado, ponte en contacto con el [equipo de soporte de Auth0](https://support.auth0.com/).
</Warning>

La MFA personalizable permite a los usuarios registrar y verificar los factores de su elección compatibles con tu aplicación.

Durante la autenticación en el endpoint `oauth/token`, la respuesta devuelve el error `mfa_required`, que incluye el `mfa_token` para usar la API de MFA y el parámetro `mfa_requirements` con una lista de autenticadores:

```json
{
  "error": "mfa_required",
  "error_description": "Multifactor authentication required",
  "mfa_token": "Fe26...Ha",
  "mfa_requirements": {
    "challenge": [
      { "type": "otp" },
      { "type": "push-notification" },
      { "type": "phone" },
      { "type": "recovery-code" }
      { "type": "email"} //solo puede funcionar con challenge
    ]
  }
}
```

Utilice el `mfa_token` para llamar al endpoint [`mfa/authenticator`](/docs/es/api/authentication/muti-factor-authentication/list-authenticators) y obtener una lista de todos los factores que el usuario ha registrado, e identificar el mismo tipo que admite su aplicación.  También necesita obtener el `authenticator_type` correspondiente para emitir desafíos:

```json
[
  {
    "type": "recovery-code",
    "id": "recovery-code|dev_qpOkGUOxBpw6R16t",
    "authenticator_type": "recovery-code",
    "active": true
  },
  {
    "type": "otp",
    "id": "totp|dev_6NWz8awwC8brh2dN",
    "authenticator_type": "otp",
    "active": true
  }
]
```

Aplica el desafío MFA mediante una llamada al endpoint [`request/mfa/challenge`](/docs/es/api/authentication/muti-factor-authentication/request-mfa-challenge).

Personaliza aún más tu flujo de MFA con Auth0 Actions. Para obtener más información, consulta [Actions Triggers: post-challenge - objeto de API](/docs/es/customize/actions/explore-triggers/password-reset-triggers/post-challenge-trigger/post-challenge-api-object).
