Usar <Tooltip tip="Token de actualización: token utilizado para obtener un token de acceso renovado sin obligar a los usuarios a volver a iniciar sesión." cta="Ver glosario" href="/docs/es/glossary?term=Refresh+tokens">tokens de actualización</Tooltip> con [Actions](/docs/es/customize/actions) te permite configurar capacidades de detección y respuesta ante riesgos después de la autenticación para proteger tus aplicaciones y usuarios frente a tokens de actualización comprometidos. También puedes personalizar dinámicamente la [expiración del token de actualización](/docs/es/secure/tokens/refresh-tokens/configure-refresh-token-expiration).

Para ello, las Actions de Post Login incluyen dos objetos clave:

* **event.refresh&#95;token**: Proporciona información relevante sobre los `refresh_tokens` existentes, incluidos `id`, `created_at`, `expires_at`, `idle_expires_at`, `clients_id`, información de `device`, como `ASN`, `IP` y `User_agent`, y, en los flujos basados en navegador, `session_id`. Este objeto se rellena en los flujos de [intercambio de token de actualización](/docs/es/secure/tokens/refresh-tokens/use-refresh-token-rotation).
* **api.refreshToken**: Te permite administrar tokens de actualización existentes revocando sesiones o cambiando las fecha de expiración.

Puedes usar el objeto `event.refresh_token` para revisar la propiedad `last_exchange_at` y evaluar los riesgos asociados a las transacciones actuales. También puedes combinar el objeto `event.refresh_token` con otros objetos de evento, como `event.authentication`.

Luego, puedes usar el objeto `api.refreshToken` para establecer fechas de expiración del token de actualización o revocar el token de actualización.

Para obtener más información sobre estos objetos, consulta:

* [objeto Event](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger/post-login-event-object): Obtén información sobre el objeto Event del token de actualización y sus propiedades.
* [objeto de API](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger/post-login-api-object): Obtén información sobre el objeto de API del token de actualización y sus métodos.

<div id="revoke-refresh-tokens-with-actions">
  ## Revocar tokens de actualización con Actions
</div>

El método de Post Login **api.refreshToken.revoke(reason)** te permite responder a riesgos asociados a una transacción. Revocar el token de actualización invalida el token de actualización, devuelve un código de estado HTTP 403 para denegar la transacción actual y registra un evento de revocación del token de actualización en el [registro del tenant](/docs/es/deploy-monitor/logs/log-event-type-codes) (`srrt`).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Si quieres usar el método `api.refreshToken.revoke(reason)`, asegúrate de que exista el objeto `event.refresh_token`.
</Callout>

<div id="monitor-revoke-log-events">
  ### Supervisar eventos de log de revocación
</div>

La operación de revocación agrega el siguiente evento de log en el [registro del tenant](/docs/es/deploy-monitor/logs):

Un código de evento `srrt` que indica que se revocó un token de actualización.

Si el token de actualización está vinculado a una sesión autenticada previamente, el registro incluirá una referencia a esa sesión autenticada en el atributo `session_id`.

<div id="change-refresh-tokens-expiry-dates-with-actions">
  ## Cambiar las fechas de expiración de los tokens de actualización con Actions
</div>

Puedes modificar las fechas de expiración de los tokens de actualización con los siguientes métodos de Post Login:

* **api.refreshToken.setExpiresAt(absolute)** te permite definir una nueva fecha de expiración absoluta para un token de actualización específico.
* **api.refreshToken.setIdleExpiresAt(idle)** te permite establecer una nueva fecha de expiración por inactividad para un token de actualización específico.

Puedes usar estos métodos para personalizar dinámicamente la duración del token de actualización y las políticas de inactividad en función de:

* La organización de un usuario
* La Conexión de Auth0 de un usuario
* La pertenencia a un grupo o el perfil de un usuario específico
* Risk Assessment
* Cualquier otro criterio dinámico disponible durante la ejecución de la Action

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Los métodos `api.refreshToken.setExpiresAt(absolute)` y `api.refreshToken.setIdleExpiresAt(idle)` permiten definir la expiración de un token de actualización antes de emitirlo, o modificar la expiración de un token de actualización existente durante un flujo de [intercambio de token de actualización](/docs/es/secure/tokens/refresh-tokens/use-refresh-tokens).

  Los métodos `api.refreshToken.setExpiresAt(absolute)` y `api.refreshToken.setIdleExpiresAt(idle)` convertirán tokens de actualización sin expiración en tokens de actualización con expiración mediante la configuración predeterminada de [Expiración del token de actualización](/docs/es/secure/tokens/refresh-tokens/configure-refresh-token-expiration) como valores máximos.

  El método `api.refreshToken.setIdleExpiresAt(idle)` establece el tiempo de espera por inactividad de los tokens de actualización. Si no se llama a este método en cada intercambio exitoso, el tiempo de espera por inactividad se reemplazará con la configuración de la aplicación para la duración del token de actualización.
</Callout>

<div id="limitations">
  ## Limitaciones
</div>

* Los tokens de actualización emitidos a partir del 21-09-2023 (22-02-2024 para tenants en la región US-3) contienen la propiedad de ID de sesión (`session_id`) con el valor correspondiente. Los tokens de actualización emitidos antes de esta fecha contienen esta propiedad con un valor `null`.

* Los tokens de actualización emitidos antes del lanzamiento del método de la API de Post Login `api.refreshToken.revoke(reason)` no contendrán información de `event.refresh_token.device`.

* Los tokens de actualización que no expiran o que no se han intercambiado no contendrán la propiedad `event.refresh_token.last_exchanged_at`.

* Por motivos de seguridad, los tiempos de espera por inactividad y absolutos no pueden establecerse por encima de la configuración del token de actualización de la aplicación definida en [expiración del token de actualización](/docs/es/secure/tokens/refresh-tokens/configure-refresh-token-expiration). Si intenta establecer una fecha posterior a la configuración de expiración, los métodos de la API la ajustarán hasta [expiración del token de actualización](/docs/es/secure/tokens/refresh-tokens/configure-refresh-token-expiration) y registrarán un evento de advertencia (`w`) en los registros del tenant.

* Tanto `api.refreshToken.setExpiresAt()` como `api.refreshToken.setIdleExpiresAt()` solo pueden acortar sus respectivos periodos de validez con respecto a los valores actuales. No pueden ampliarlos ni aumentarlos.

<div id="use-cases-revoke-a-refresh-token">
  ## Casos de uso: Revocar un token de actualización
</div>

Puedes usar [Actions](/docs/es/customize/actions) para configurar las detecciones de riesgo y revocar tokens de actualización mediante el método `api.refreshToken.revoke(reason)` y los objetos de evento.

<div id="revoke-refresh-tokens-due-to-impossibletravel">
  ### Revocar tokens de actualización por ImpossibleTravel
</div>

Puedes usar el objeto de [evaluaciones](/docs/es/secure/multi-factor-authentication/adaptive-mfa/customize-adaptive-mfa#assessments-object) de <Tooltip tip="MFA adaptativo: autenticación multifactor (MFA) que solo se activa para los usuarios cuando se determina que un intento de login es de baja confianza." cta="Ver glosario" href="/docs/es/glossary?term=Adaptive+MFA">MFA adaptativo</Tooltip> para determinar si un usuario está iniciando sesión desde una ubicación que indique ImpossibleTravel y revocar el token de actualización actual asociado a la transacción.

```js lines
exports.onExecutePostLogin = async (event, api) => {
  const { riskAssessment } = event.authentication ?? {};
  const ImpossibleTravel = riskAssessment?.assessments.ImpossibleTravel;

  // Si se trata de un viaje imposible y de un intercambio de token de actualización
  if (ImpossibleTravel?.code === "impossible_travel_from_last_login") {
    if (event.refresh_token) {
      api.refreshToken.revoke("Refresh token revoked due to impossible travel");
    }
  }
};
```

En este ejemplo, se realiza una comprobación al inicio de la Action para verificar que `event.authentication.ImpossibleTravel.code` sea igual a la `propiedad impossible_travel_from_last_login`. Si es `true`, la Action llama a `api.refreshToken.revoke()` para:

* Denegar la transacción
* Revocar el token de actualización
* Devolver un error `access&#95;denied` con respuesta 403
* Generar el error “Token de actualización revocado debido a un viaje imposible”

<div id="revoke-refresh-tokens-due-to-ip-binding">
  ### Revocar tokens de actualización por vinculación de IP
</div>

Si usa las propiedades del objeto de Post Login `event.refresh_token.device.initial_ip` y `event.request.ip` para garantizar que una transacción de token de actualización permanezca en la misma dirección IP durante toda su vigencia. En este escenario, cualquier cambio de IP se considera un riesgo y se requiere un nuevo token de actualización.

```js lines
exports.onExecutePostLogin = async (event, api) => {
  const refreshTokenInitialIp = event.refresh_token?.device?.initial_ip;
  const requestCurrentIp = event.request.ip;

  // si hay un token de actualización y la IP cambia
  if (
    refreshTokenInitialIp &&
    requestCurrentIp &&
    refreshTokenInitialIp != requestCurrentIp
  ) {
    api.refreshToken.revoke("Invalid IP change");
  }
};
```

En este ejemplo, se realiza una verificación al inicio de la Action para llevar un registro de las direcciones IP con las propiedades `event.refresh_token.device.initial_ip` y `event.request.ip`. La Action determina si la dirección IP de la transacción ha cambiado. Si es `true`, la Action llama a `api.refreshToken.revoke()` para:

* Denegar la transacción
* Revocar el token de actualización
* Devolver un error `access_denied` con respuesta `403`
* Generar el error “`Invalid IP change`”

Como alternativa, si quieres una Action menos restrictiva, puedes hacer un seguimiento de las propiedades event.`request.asn` y `event.refresh_token.device.initial_asn` para supervisar cambios en el ASN en lugar de cambios en la IP.

<div id="use-cases-customize-refresh-token-expiry-dates">
  ## Casos de uso: Personalizar las fechas de expiración del token de actualización
</div>

Puedes usar [Actions](/docs/es/customize/actions) para personalizar la duración y las fechas de inactividad del token de actualización. En concreto, puedes configurar las fechas de expiración por inactividad y absoluta del token de actualización para una transacción concreta mediante los métodos de Post Login `api.refreshToken.setExpiresAt(absolute)` y `api.refreshToken.setIdleExpiresAt(idle)`.

<div id="customize-absolute-refresh-token-expiration-date-based-on-organization">
  ### Personalizar la fecha de expiración absoluta del token de actualización según la organización
</div>

Puede usar una Action de Post Login para definir la duración del token de actualización por organización. En el siguiente ejemplo, se usa la `metadata` `refresh_token_timeout` de la organización para establecer la fecha de expiración del token de actualización.

```js lines
exports.onExecutePostLogin = async (event, api) => {
  // Tiempo de espera del token de actualización (en milisegundos) configurado en las Organizations
  const organization_refresh_token_lifetime =
    event.organization?.metadata?.refresh_token_timeout;

  if (organization_refresh_token_lifetime) {
    // El token de actualización ya existe
    if (event.refresh_token) {
      const created = Date.parse(event.refresh_token.created_at);

      const new_expiration_time =
        created + Number(organization_refresh_token_lifetime);
      api.refreshToken.setExpiresAt(new_expiration_time);
    } else {
      // El token de actualización aún no existe (p. ej., se está emitiendo el token)
      const current_time = new Date().getTime();

      const new_expiration_time =
        current_time + Number(organization_refresh_token_lifetime);
      api.refreshToken.setExpiresAt(new_expiration_time);
    }
  }
};
```

En este ejemplo, si hay un tiempo de expiración absoluto específico definido para una Organización, la Action establece el tiempo de expiración absoluto del token de actualización para que sea igual a:

* Tokens emitidos recientemente: `current_time` más `organization_refresh_token_lifetime`
* Tokens existentes: `event.refresh_token.created_at` más `organization_refresh_token_lifetime`

<div id="customize-refresh-token-inactivity-timeout-based-on-membership-role">
  ### Personaliza el tiempo de espera por inactividad del token de actualización según el rol de membresía
</div>

Puedes usar una Action de Post Login para definir el tiempo de espera por inactividad de un token de actualización mediante [aplicación](/docs/es/get-started/applications/configure-application-metadata) y [metadatos de usuario](/docs/es/manage-users/user-accounts/metadata). En el siguiente ejemplo, se usan los roles de los metadatos de usuario para definir el rol de membresía del usuario y los metadatos de la aplicación para definir el tiempo de espera por inactividad esperado del token de actualización.

```js lines
exports.onExecutePostLogin = async (event, api) => {
  // los administradores están configurados con un idle timeout más corto para los tokens de actualización, en los metadatos de la aplicación
  const max_idle_lifetime =
    event.client.metadata?.admin_refresh_token_idle_timeout;

  // Verificar el atributo de roles en los metadatos de la aplicación del usuario para comprobar si es un usuario administrador.
  const isAdmin = event.user?.app_metadata?.roles?.find(
    (role) => role === "admin",
  );

  // Si la aplicación tiene un idle timeout específico definido, establecer el timeout
  if (max_idle_lifetime && isAdmin) {
    const current_time = new Date().getTime();

    api.refreshToken.setIdleExpiresAt(current_time + Number(max_idle_lifetime));
  }
};
```

En este ejemplo, si hay un tiempo de inactividad máximo específico definido para la aplicación y el usuario es un Admin, la Action establece el tiempo de espera por inactividad del token de actualización para que sea igual a `current_time` más `refresh_token_idle_timeout`. Ten en cuenta que estamos cambiando este tiempo de espera tanto para los tokens recién emitidos como para los existentes durante el intercambio de token de actualización.
