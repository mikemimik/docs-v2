Usar <Tooltip tip="Token de actualización: token usado para obtener un token de acceso renovado sin obligar a los usuarios a iniciar sesión de nuevo." cta="Ver glosario" href="/docs/es/glossary?term=Refresh+tokens">tokens de actualización</Tooltip> con [Actions](/docs/es/customize/actions) te permite configurar capacidades de detección y respuesta ante riesgos posteriores a la autenticación para proteger tus aplicaciones y usuarios frente a tokens de actualización comprometidos. También puedes personalizar dinámicamente la [expiración del token de actualización](/docs/es/secure/tokens/refresh-tokens/configure-refresh-token-expiration).

Para ello, las Actions de Post Login incluyen dos objetos clave:

* **event.refresh&#95;token**: Proporciona información relevante sobre `refresh_tokens` existentes, incluidos `id`, `created_at`, `expires_at`, `idle_expires_at`, `clients_id`, información de `device`, como `ASN`, `IP` y `User_agent`, y, en flujos basados en navegador, `session_id`. Este objeto se rellena en los flujos de [intercambio de token de actualización](/docs/es/secure/tokens/refresh-tokens/use-refresh-token-rotation).
* **api.refreshToken**: Te permite gestionar tokens de actualización existentes revocando sesiones o cambiando las fechas de expiración.

Puedes usar el objeto `event.refresh_token` para revisar la propiedad `last_exchange_at` y evaluar los riesgos asociados con las transacciones en curso. También puedes combinar el objeto `event.refresh_token` con otros objetos de evento, como `event.authentication`.

Luego, puedes usar el objeto `api.refreshToken` para establecer la fecha de expiración del token de actualización o revocarlo.

Para obtener más información sobre estos objetos, consulta:

* [Event object](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger/post-login-event-object): Obtén más información sobre el objeto Event del token de actualización y sus propiedades.
* [API object](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger/post-login-api-object): Obtén más información sobre el objeto de API del token de actualización y sus métodos.

<div id="revoke-refresh-tokens-with-actions">
  ## Revocar tokens de actualización con Actions
</div>

El método **api.refreshToken.revoke(reason)** de Post Login te permite responder a riesgos asociados a una transacción. Revocar el token de actualización invalida ese token, devuelve un código de estado HTTP 403 para denegar la transacción actual y registra un evento de revocación de token de actualización en los [registros del tenant](/docs/es/deploy-monitor/logs/log-event-type-codes) (`srrt`).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Si quieres usar el método `api.refreshToken.revoke(reason)`, asegúrate de que exista el objeto `event.refresh_token`.
</Callout>

<div id="monitor-revoke-log-events">
  ### Supervisa los eventos de log de revocación
</div>

La operación de revocación agrega el siguiente evento de log en tus [registro del tenant](/docs/es/deploy-monitor/logs):

Un código de evento `srrt` que indica que se revocó un token de actualización.

Si el token de actualización está vinculado a una sesión autenticada previamente, el registro incluirá una referencia a la sesión autenticada en el atributo `session_id`.

<div id="change-refresh-tokens-expiry-dates-with-actions">
  ## Cambiar las fechas de expiración de los tokens de actualización con Actions
</div>

Puedes modificar las fechas de expiración de los tokens de actualización con los siguientes métodos de Post Login:

* **api.refreshToken.setExpiresAt(absolute)** te permite definir una nueva fecha de expiración absoluta para un token de actualización determinado.
* **api.refreshToken.setIdleExpiresAt(idle)** te permite establecer una nueva fecha de expiración por inactividad para un token de actualización determinado.

Puedes usar estos métodos para personalizar dinámicamente la duración del token de actualización y las políticas de inactividad en función de:

* La organización de un usuario
* La Conexión de Auth0 de un usuario
* La pertenencia a grupos o el perfil de un usuario específico
* La evaluación de riesgos
* Cualquier otro criterio dinámico disponible durante la ejecución de la Action

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Los métodos `api.refreshToken.setExpiresAt(absolute)` y `api.refreshToken.setIdleExpiresAt(idle)` permiten definir la expiración de un token de actualización antes de su emisión, o modificar la expiración de un token de actualización existente durante un flujo de [intercambio de token de actualización](/docs/es/secure/tokens/refresh-tokens/use-refresh-tokens).

  Los métodos `api.refreshToken.setExpiresAt(absolute)` y `api.refreshToken.setIdleExpiresAt(idle)` convertirán los tokens de actualización que no expiran en tokens de actualización con expiración, usando la configuración predeterminada de [expiración del token de actualización](/docs/es/secure/tokens/refresh-tokens/configure-refresh-token-expiration) como valor máximo.

  El método `api.refreshToken.setIdleExpiresAt(idle)` establece el tiempo de espera por inactividad para los tokens de actualización. Si no se llama a este método en cada intercambio exitoso, el tiempo de espera por inactividad se sobrescribirá con la configuración de la aplicación para la duración del token de actualización.
</Callout>

<div id="limitations">
  ## Limitaciones
</div>

* Los tokens de actualización emitidos el 21-09-2023 o después de esa fecha (el 22-02-2024 para tenants en la región US-3) contienen la propiedad de ID de sesión (`session_id`) con el valor correspondiente. Los tokens de actualización emitidos antes de esa fecha contienen esta propiedad con un valor `null`.

* Los tokens de actualización emitidos antes del lanzamiento del método de API de Post Login `api.refreshToken.revoke(reason)` no incluirán información de `event.refresh_token.device`.

* Los tokens de actualización que no expiran o que no se han intercambiado no contendrán la propiedad `event.refresh_token.last_exchanged_at`.

* Por motivos de seguridad, los tiempos de espera por inactividad y absolutos no pueden configurarse por encima de la configuración del token de actualización de la aplicación definida en [expiración del token de actualización](/docs/es/secure/tokens/refresh-tokens/configure-refresh-token-expiration). Si intenta establecer una fecha posterior a la configuración de expiración, los métodos de API actualizarán el valor hasta [expiración del token de actualización](/docs/es/secure/tokens/refresh-tokens/configure-refresh-token-expiration) y registrarán un evento de advertencia (`w`) en los registros del tenant.

* Tanto `api.refreshToken.setExpiresAt()` como `api.refreshToken.setIdleExpiresAt()` solo pueden acortar sus respectivas duraciones respecto de los valores actuales. No pueden extender ni aumentar la duración.

<div id="use-cases-revoke-a-refresh-token">
  ## Casos de uso: Revocar un token de actualización
</div>

Puede usar [Actions](/docs/es/customize/actions) para configurar detecciones de riesgo y revocar tokens de actualización con el método `api.refreshToken.revoke(reason)` y los objetos de evento.

<div id="revoke-refresh-tokens-due-to-impossibletravel">
  ### Revocar tokens de actualización por ImpossibleTravel
</div>

Puedes usar el objeto [assessments](/docs/es/secure/multi-factor-authentication/adaptive-mfa/customize-adaptive-mfa#assessments-object) de <Tooltip tip="MFA adaptativo: autenticación multifactor (MFA) que solo se activa para los usuarios cuando se determina que un intento de inicio de sesión es de baja confianza." cta="Ver glosario" href="/docs/es/glossary?term=Adaptive+MFA">MFA adaptativo</Tooltip> para determinar si un usuario está iniciando sesión desde una ubicación que indica ImpossibleTravel y revocar el token de actualización actual asociado a la transacción.

```js lines
exports.onExecutePostLogin = async (event, api) => {
  const { riskAssessment } = event.authentication ?? {};
  const ImpossibleTravel = riskAssessment?.assessments.ImpossibleTravel;

  // Si se trata de un viaje imposible y de un intercambio de token de actualización
  if (ImpossibleTravel?.code === "impossible_travel_from_last_login") {
    if (event.refresh_token) {
      api.refreshToken.revoke("Refresh token revoked due to impossible travel");
    }
  }
};
```

En este ejemplo, se realiza una comprobación al inicio de la Action para verificar que `event.authentication.ImpossibleTravel.code` sea igual a la propiedad `impossible_travel_from_last_login`. Si es `true`, la Action llama a `api.refreshToken.revoke()` para:

* Denegar la transacción
* Revocar el token de actualización
* Devolver un error `access&#95;denied` con respuesta 403
* Generar el error “Token de actualización revocado por viaje imposible”

<div id="revoke-refresh-tokens-due-to-ip-binding">
  ### Revocar tokens de actualización debido a la vinculación de IP
</div>

Si utiliza las propiedades del objeto de Post Login `event.refresh_token.device.initial_ip` y `event.request.ip` para garantizar que la transacción de un token de actualización permanezca asociada a la misma dirección IP durante toda su vigencia. En este escenario, cualquier cambio de IP se considera un riesgo y requiere un nuevo token de actualización.

```js lines
exports.onExecutePostLogin = async (event, api) => {
  const refreshTokenInitialIp = event.refresh_token?.device?.initial_ip;
  const requestCurrentIp = event.request.ip;

  // si hay un token de actualización y la IP cambia
  if (
    refreshTokenInitialIp &&
    requestCurrentIp &&
    refreshTokenInitialIp != requestCurrentIp
  ) {
    api.refreshToken.revoke("Invalid IP change");
  }
};
```

En este ejemplo, se realiza una comprobación al inicio de la Action para hacer un seguimiento de las direcciones IP mediante las propiedades `event.refresh_token.device.initial_ip` y `event.request.ip`. La Action determina si la dirección IP de la transacción ha cambiado. Si es `true`, la Action llama a `api.refreshToken.revoke()` para:

* Denegar la transacción
* Revocar el token de actualización
* Devolver una respuesta de error `403` `access_denied`
* Generar el error “`Invalid IP change`”

Como alternativa, si quieres una Action menos restrictiva, puedes hacer un seguimiento de las propiedades event.`request.asn` y `event.refresh_token.device.initial_asn` para supervisar cambios de ASN en lugar de cambios de IP.

<div id="use-cases-customize-refresh-token-expiry-dates">
  ## Casos de uso: Personalizar las fechas de expiración del token de actualización
</div>

Puedes usar [Actions](/docs/es/customize/actions) para personalizar la duración del token de actualización y las fechas de expiración por inactividad. En concreto, puedes configurar las fechas de expiración absoluta y por inactividad del token de actualización para una transacción específica mediante los métodos de Post Login `api.refreshToken.setExpiresAt(absolute)` y `api.refreshToken.setIdleExpiresAt(idle)`.

<div id="customize-absolute-refresh-token-expiration-date-based-on-organization">
  ### Personaliza la fecha de expiración absoluta del token de actualización según la organización
</div>

Puedes usar una Action de Post Login para definir la duración del token de actualización por organización. El siguiente ejemplo usa los metadatos `refresh_token_timeout` de la organización para establecer la fecha de expiración del token de actualización.

```js lines
exports.onExecutePostLogin = async (event, api) => {
  // Tiempo de espera del token de actualización (en milisegundos) configurado en los metadatos de Organizations
  const organization_refresh_token_lifetime =
    event.organization?.metadata?.refresh_token_timeout;

  if (organization_refresh_token_lifetime) {
    // El token de actualización ya existe
    if (event.refresh_token) {
      const created = Date.parse(event.refresh_token.created_at);

      const new_expiration_time =
        created + Number(organization_refresh_token_lifetime);
      api.refreshToken.setExpiresAt(new_expiration_time);
    } else {
      // El token de actualización aún no existe (p. ej., se está emitiendo el token)
      const current_time = new Date().getTime();

      const new_expiration_time =
        current_time + Number(organization_refresh_token_lifetime);
      api.refreshToken.setExpiresAt(new_expiration_time);
    }
  }
};
```

En este ejemplo, si se define un tiempo de expiración absoluto específico para una organización, la Action establece el tiempo de expiración absoluto del token de actualización de modo que sea igual a:

* Tokens recién emitidos: `current_time` más `organization_refresh_token_lifetime`
* Tokens existentes: `event.refresh_token.created_at` más `organization_refresh_token_lifetime`

<div id="customize-refresh-token-inactivity-timeout-based-on-membership-role">
  ### Personaliza el tiempo de inactividad del token de actualización según el rol de membresía
</div>

Puedes usar una Action de Post Login para definir el tiempo de inactividad de un token de actualización mediante [aplicación](/docs/es/get-started/applications/configure-application-metadata) y [metadatos del usuario](/docs/es/manage-users/user-accounts/metadata). El siguiente ejemplo usa los roles en los metadatos del usuario para definir el rol de membresía del usuario y Application Metadata para definir el tiempo de inactividad previsto del token de actualización.

```js lines
exports.onExecutePostLogin = async (event, api) => {
  // los administradores están configurados con un idle timeout más corto para los tokens de actualización, en los metadatos de la aplicación
  const max_idle_lifetime =
    event.client.metadata?.admin_refresh_token_idle_timeout;

  // Verificar el atributo de roles en los metadatos de la aplicación del usuario para comprobar si es un usuario administrador.
  const isAdmin = event.user?.app_metadata?.roles?.find(
    (role) => role === "admin",
  );

  // Si la aplicación tiene un idle timeout específico definido, establecer el timeout
  if (max_idle_lifetime && isAdmin) {
    const current_time = new Date().getTime();

    api.refreshToken.setIdleExpiresAt(current_time + Number(max_idle_lifetime));
  }
};
```

En este ejemplo, si hay un tiempo de espera por inactividad específico definido para la aplicación y el usuario es un Admin, la Action establece el tiempo de espera por inactividad del token de actualización en `current_time` más `refresh_token_idle_timeout`. Ten en cuenta que estamos cambiando el tiempo de espera tanto para los tokens recién emitidos como para los ya existentes durante el intercambio de token de actualización.
