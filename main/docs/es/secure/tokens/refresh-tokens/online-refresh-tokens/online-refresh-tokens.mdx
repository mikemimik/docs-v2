Los Online Refresh Tokens (ORT) son un tipo de [token de actualización](/docs/es/secure/tokens/refresh-tokens) diseñado específicamente para aplicaciones basadas en navegador, como las [aplicaciones de una sola página](/docs/es/get-started/auth0-overview/create-applications/single-page-web-apps) (SPA).

Los ORT permiten mantener la sesión de un usuario en las SPA cuando las funciones de privacidad del navegador, como Intelligent Tracking Prevention (ITP), bloquean el acceso a cookies de terceros, lo que hace que la [autenticación silenciosa](/docs/es/authenticate/login/configure-silent-authentication#configure-silent-authentication) tradicional sea poco fiable.

Los ORT están vinculados a la [sesión](/docs/es/manage-users/sessions) de Auth0 del usuario y solo pueden usarse mientras esa sesión esté activa, lo que garantiza que, cuando un usuario cierre sesión, todas las aplicaciones que usan esa sesión también cerrarán sesión.

<div id="how-it-works">
  ## Cómo funciona
</div>

Cuando solicitas un ORT:

1. Tu aplicación incluye el scope `online_access` en la solicitud de autorización.
2. Auth0 devuelve un ORT vinculado a la sesión actual del usuario.
3. Cada vez que intercambias el ORT por un nuevo token de acceso, Auth0 extiende el tiempo de espera de inactividad de la sesión.
4. El token sigue siendo válido mientras la sesión de Auth0 subyacente esté activa.

<div id="comparison-of-orts-versus-refresh-tokens">
  ## Comparación entre ORTs y tokens de actualización
</div>

| **Característica**                                                                                                                       | **ORTs**                                                                                                  | **token de actualización**                                                                                     |
| ---------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------- |
| **Scope solicitado**                                                                                                                     | `online_access`                                                                                           | `offline_access`                                                                                               |
| **Vinculación con la sesión**                                                                                                            | Vinculado a la sesión de Auth0                                                                            | Independiente de la sesión                                                                                     |
| **Duración**                                                                                                                             | Determinada por la [duración de la sesión](/docs/es/manage-users/sessions/configure-session-lifetime)        | Configurada por la [duración del token](/docs/es/secure/tokens/refresh-tokens/configure-refresh-token-expiration) |
| **Rotación del token**                                                                                                                   | No rota                                                                                                   | Admite [rotación](/docs/es/secure/tokens/refresh-tokens/configure-refresh-token-rotation)                         |
| **Requisito de [Demonstration of Proof of Possession](/docs/es/secure/sender-constraining/demonstrating-proof-of-possession-dpop)  (DPoP)** | Obligatorio para clientes públicos (SPA, Native)                                                          | Opcional (configuración del servidor de recursos)                                                              |
| **Revocación**                                                                                                                           | [Revoca toda la sesión](/docs/es/manage-users/sessions/manage-sessions-actions#revoke-sessions-with-actions) | [Revoca solo el token](/docs/es/secure/tokens/refresh-tokens/revoke-refresh-tokens)                               |
| **Management API**                                                                                                                       | No visible                                                                                                | Visible mediante la Management API                                                                             |
| **Tiempo de espera por inactividad de la sesión**                                                                                        | Se extiende con cada intercambio                                                                          | No afecta a la sesión                                                                                          |

<div id="key-behavioral-differences">
  ### Diferencias clave de comportamiento
</div>

**Extensión de la sesión:** Cuando intercambias un ORT, Auth0 reinicia el tiempo de inactividad de la sesión, lo que mantiene activa la sesión del usuario. Auth0 no extiende la [duración absoluta](/docs/es/manage-users/sessions/session-lifecycle#absolute-timeout-maximum) de la sesión.

**Exigencia de DPoP:** Para clientes públicos (como las SPA), Auth0 exige DPoP al usar ORT. Como los ORT funcionan con el mismo nivel de seguridad que una cookie de sesión que da acceso a la sesión autenticada del usuario, deben protegerse con el mismo rigor. DPoP evita el robo de tokens y los ataques de repetición.

**Sin estado:** Los ORT están vinculados a una sesión, por lo que no pueden verse ni administrarse de forma independiente. No puedes verlos en la Management API ni revocarlos individualmente.

<div id="best-practices">
  ## Mejores prácticas
</div>

1. **Almacena los tokens en memoria:** Evita almacenar ORTs en `localStorage` o `sessionStorage`. Mantenlos en la memoria de JavaScript para mayor seguridad. Como los ORTs están vinculados a la sesión de Auth0, la cookie de sesión volverá a establecer el contexto en la próxima visita del usuario, lo que permitirá que tu aplicación solicite un ORT nuevo.
2. **Gestiona correctamente el vencimiento de la sesión:** Implementa un manejo de errores adecuado para los casos en que la sesión haya expirado. Redirige a los usuarios para que vuelvan a autenticarse sin perder su trabajo.
3. **Usa DPoP para clientes públicos:** Los ORTs requieren DPoP para clientes públicos. Asegúrate de que tu implementación genere y use correctamente las pruebas DPoP.
4. **No dependas de la estructura del token:** Trata los ORTs como cadenas opacas. No los analices ni dependas de ninguna estructura interna, ya que esto puede cambiar.
5. **Supervisa la duración de la sesión:** Ten en cuenta los tiempos de espera de la sesión, tanto por inactividad como absolutos. El tiempo de espera absoluto no puede ampliarse mediante el intercambio de token de actualización.

<div id="limitations">
  ## Limitaciones
</div>

<div id="unsupported-flows">
  ### Flujos no compatibles
</div>

Los ORT solo están disponibles para flujos que crean una sesión interactiva en el navegador. Los siguientes flujos **no** pueden emitir ORT:

| **Flujo**                                                                                                             | **Motivo**                                                                           |
| --------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------ |
| [Device Authorization Flow](/docs/es/get-started/authentication-and-authorization-flow/device-authorization-flow)        | La sesión se crea en el navegador, pero el token se usa en otro dispositivo          |
| [Implicit Flow](/docs/es/get-started/authentication-and-authorization-flow/implicit-flow-with-form-post)                 | La especificación de OAuth prohíbe los tokens de actualización en el flujo implícito |
| [Resource Owner Password Grant](/docs/es/get-started/authentication-and-authorization-flow/resource-owner-password-flow) | No se crea ninguna sesión en el navegador                                            |
| [Client Credentials Flow](/docs/es/get-started/authentication-and-authorization-flow/client-credentials-flow)            | No intervienen usuarios ni sesiones                                                  |
| [Custom Token Exchange](/docs/es/get-started/authentication-and-authorization-flow/token-exchange-flow)                  | No hay ninguna sesión a la que vincular el token                                     |

Si solicitas el `scope` `online_access` en cualquiera de estos flujos, el `scope` se ignora sin notificación y no se emite ningún token de actualización.

<div id="scope-conflicts">
  ### Conflictos entre scopes
</div>

No puedes solicitar tanto `online_access` como `offline_access` en la misma solicitud de autorización. Si se incluyen ambos scopes, Auth0 devuelve un error:

```json
{
  "error": "invalid_request",
  "error_description": "Cannot request both online_access and offline_access scopes"
}
```

Seleccione el scope adecuado según su caso de uso:

* Use `online_access` para aplicaciones que se ejecutan en el navegador y se benefician de la continuidad del SSO.
* Use `offline_access` para aplicaciones que necesitan tokens para funcionar de forma independiente de la sesión del navegador.
