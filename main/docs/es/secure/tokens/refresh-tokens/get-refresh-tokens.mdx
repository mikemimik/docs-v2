Para obtener un <Tooltip tip="Token de actualización: token que se usa para obtener un token de acceso renovado sin obligar a los usuarios a volver a iniciar sesión." cta="Ver glosario" href="/docs/es/glossary?term=refresh+token">token de actualización</Tooltip>, debes incluir el `offline_access` [scope](/docs/es/get-started/apis/scopes) al iniciar una solicitud de autenticación a través del endpoint `/authorize`. Asegúrate de habilitar Offline Access en tu API. Para obtener más información, consulta [Configuración de la API](/docs/es/get-started/apis/api-settings).

Por ejemplo, si estás usando el [flujo de código de autorización](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow), la solicitud de autenticación sería la siguiente:

export const codeExample1 = `https://{yourDomain}/authorize?
    audience={API_AUDIENCE}&
    scope=offline_access&
    response_type=code&
    client_id={yourClientId}&
    redirect_uri={https://yourApp/callback}&
    state={OPAQUE_VALUE}`;

<AuthCodeBlock children={codeExample1} language="http" />

El token de actualización se almacena en la sesión. Luego, cuando sea necesario renovar una sesión (por ejemplo, porque ya transcurrió un periodo preconfigurado o porque el usuario intenta realizar una operación sensible), la app usa el token de actualización en el backend para obtener un nuevo <Tooltip tip="Token de ID: Credencial destinada al propio cliente, en lugar de para acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=ID+token">token de ID</Tooltip>, mediante el endpoint `/oauth/token` con `grant_type=refresh_token`.

Una vez que el usuario se autentique correctamente, la aplicación será redirigida a `redirect_uri`, con un `code` como parte de la URL: `{https://yourApp/callback}?code=BPPLN3Z4qCTvSNOy`. Puedes intercambiar este código por un <Tooltip tip="Token de acceso: Credencial de autorización, en forma de una cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=access+token">token de acceso</Tooltip> mediante el endpoint `/oauth/token`.

La respuesta debe contener un token de acceso y un token de actualización.

```json lines
{
      "access_token": "eyJz93a...k4laUWw",
      "refresh_token": "GEbRxBN...edjnXbL",
      "token_type": "Bearer"
    }
```

Si estás solicitando un token de actualización para una aplicación móvil con el cliente nativo correspondiente (que es público), no necesitas enviar el `client_secret` en la solicitud, ya que solo se requiere para las aplicaciones confidenciales.

Los tokens de actualización deben almacenarse de forma segura en una aplicación, ya que permiten que un usuario permanezca autenticado prácticamente para siempre.

Para obtener más información sobre cómo implementar esto con el flujo de código de autorización, consulta nuestro tutorial, [Llamar a una API usando el flujo de código de autorización](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow/call-your-api-using-the-authorization-code-flow). Para otros grants, consulta [Flujos de autenticación y autorización](/docs/es/get-started/authentication-and-authorization-flow).

<div id="customize-mfa">
  ## Personalizar MFA
</div>

<Warning>
  La MFA personalizable con los flujos de Resource Owner Password Grant, Embedded o token de actualización está en acceso anticipado. Al usar esta función, aceptas los términos aplicables de la prueba gratuita del [Master Subscription Agreement](https://www.okta.com/legal/) de Okta. Para obtener más información sobre las etapas de lanzamiento de Auth0, consulta [Product Release Stages](/docs/es/troubleshoot/product-lifecycle/product-release-stages). Para participar en el acceso anticipado, ponte en contacto con el [equipo de soporte de Auth0](https://support.auth0.com/).
</Warning>

La MFA personalizable permite que los usuarios registren y usen los factores compatibles con tu aplicación que prefieran.

Durante la autenticación en el endpoint `oauth/token`, la respuesta devuelve el error `mfa_required`, que incluye el `mfa_token` para usar la API de MFA y el parámetro `mfa_requirements` con una lista de autenticadores:

```json
{
  "error": "mfa_required",
  "error_description": "Multifactor authentication required",
  "mfa_token": "Fe26...Ha",
  "mfa_requirements": {
    "challenge": [
      { "type": "otp" },
      { "type": "push-notification" },
      { "type": "phone" },
      { "type": "recovery-code" }
      { "type": "email"} //solo puede funcionar con challenge
    ]
  }
}
```

Usa el `mfa_token` para llamar al endpoint [`mfa/authenticator`](/docs/es/api/authentication/muti-factor-authentication/list-authenticators) y obtener una lista de todos los factores que el usuario haya registrado; luego, selecciona uno del mismo tipo que admita tu aplicación.  También debes obtener el `authenticator_type` correspondiente para emitir desafíos:

```json
[
  {
    "type": "recovery-code",
    "id": "recovery-code|dev_qpOkGUOxBpw6R16t",
    "authenticator_type": "recovery-code",
    "active": true
  },
  {
    "type": "otp",
    "id": "totp|dev_6NWz8awwC8brh2dN",
    "authenticator_type": "otp",
    "active": true
  }
]
```

Aplica el desafío de MFA llamando al endpoint [`request/mfa/challenge`](/docs/es/api/authentication/muti-factor-authentication/request-mfa-challenge).

Personaliza aún más tu flujo de MFA con Auth0 Actions. Para obtener más información, consulta [Actions Triggers: post-challenge - API Object](/docs/es/customize/actions/explore-triggers/password-reset-triggers/post-challenge-trigger/post-challenge-api-object).
