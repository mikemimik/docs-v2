Para obtener un <Tooltip tip="Token de actualización: token usado para obtener un token de acceso renovado sin obligar a los usuarios a iniciar sesión de nuevo." cta="Ver glosario" href="/docs/es/glossary?term=refresh+token">token de actualización</Tooltip>, debes incluir el `offline_access` [scope](/docs/es/get-started/apis/scopes) cuando inicies una solicitud de autenticación mediante el endpoint `/authorize`. Asegúrate de habilitar Offline Access en tu API. Para obtener más información, consulta [Configuración de la API](/docs/es/get-started/apis/api-settings).

Por ejemplo, si estás usando el [flujo de código de autorización](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow), la solicitud de autenticación tendría el siguiente aspecto:

export const codeExample1 = `https://{yourDomain}/authorize?
    audience={API_AUDIENCE}&
    scope=offline_access&
    response_type=code&
    client_id={yourClientId}&
    redirect_uri={https://yourApp/callback}&
    state={OPAQUE_VALUE}`;

<AuthCodeBlock children={codeExample1} language="http" />

El token de actualización se almacena en la sesión. Después, cuando sea necesario renovar una sesión (por ejemplo, porque ya transcurrió un período preconfigurado o porque el usuario intenta realizar una operación sensible), la aplicación usa el token de actualización en el backend para obtener un nuevo <Tooltip tip="token de identidad: credencial destinada al propio cliente, en lugar de acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=ID+token">token de identidad</Tooltip>, mediante el endpoint `/oauth/token` con `grant_type=refresh_token`.

Una vez que el usuario se autentica correctamente, la aplicación será redirigida a `redirect_uri`, con un `code` como parte de la URL: `{https://yourApp/callback}?code=BPPLN3Z4qCTvSNOy`. Puede intercambiar este código por un <Tooltip tip="Token de acceso: credencial de autorización, en forma de una cadena opaca o JWT, que se utiliza para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=access+token">token de acceso</Tooltip> mediante el endpoint `/oauth/token`.

La respuesta debe incluir un token de acceso y un token de actualización.

```json lines
{
      "access_token": "eyJz93a...k4laUWw",
      "refresh_token": "GEbRxBN...edjnXbL",
      "token_type": "Bearer"
    }
```

Si estás solicitando un token de actualización para una aplicación móvil mediante el cliente Native correspondiente (que es público), no necesitas enviar el `client_secret` en la solicitud, ya que solo se requiere para las aplicaciones confidenciales.

Los tokens de actualización deben almacenarse de forma segura en una aplicación, ya que permiten que un usuario permanezca autenticado prácticamente de forma indefinida.

Para obtener más información sobre cómo implementar esto mediante el flujo de código de autorización, consulta nuestro tutorial, [Llamar a la API mediante el flujo de código de autorización](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow/call-your-api-using-the-authorization-code-flow). Para otras concesiones, consulta [Flujos de autenticación y autorización](/docs/es/get-started/authentication-and-authorization-flow).

<div id="customize-mfa">
  ## Personaliza la MFA
</div>

<Warning>
  La MFA personalizable con los flujos de contraseña del propietario del recurso, Embedded o token de actualización está en acceso anticipado. Al usar esta funcionalidad, aceptas los términos aplicables de la prueba gratuita del [Master Subscription Agreement](https://www.okta.com/legal/) de Okta. Para obtener más información sobre las etapas de lanzamiento de Auth0, consulta [Etapas de lanzamiento del producto](/docs/es/troubleshoot/product-lifecycle/product-release-stages). Para participar en el acceso anticipado, contacta al [equipo de soporte de Auth0](https://support.auth0.com/).
</Warning>

La MFA personalizable permite a los usuarios registrar los factores compatibles con su aplicación que prefieran y usarlos para la verificación.

Durante la autenticación en el endpoint `oauth/token`, la respuesta devuelve el error `mfa_required`, que incluye el `mfa_token` para usar la API de MFA y el parámetro `mfa_requirements` con una lista de autenticadores:

```json
{
  "error": "mfa_required",
  "error_description": "Multifactor authentication required",
  "mfa_token": "Fe26...Ha",
  "mfa_requirements": {
    "challenge": [
      { "type": "otp" },
      { "type": "push-notification" },
      { "type": "phone" },
      { "type": "recovery-code" }
      { "type": "email"} //solo puede funcionar con challenge
    ]
  }
}
```

Usa el `mfa_token` para llamar al endpoint [`mfa/authenticator`](/docs/es/api/authentication/muti-factor-authentication/list-authenticators), obtener una lista de todos los factores que el usuario tiene registrados y hacer coincidir el mismo tipo que admite tu aplicación. También necesitas obtener el `authenticator_type` correspondiente para emitir desafíos:

```json
[
  {
    "type": "recovery-code",
    "id": "recovery-code|dev_qpOkGUOxBpw6R16t",
    "authenticator_type": "recovery-code",
    "active": true
  },
  {
    "type": "otp",
    "id": "totp|dev_6NWz8awwC8brh2dN",
    "authenticator_type": "otp",
    "active": true
  }
]
```

Haz obligatorio el desafío MFA llamando al endpoint [`request/mfa/challenge`](/docs/es/api/authentication/muti-factor-authentication/request-mfa-challenge).

Personaliza aún más tu flujo MFA con Auth0 Actions. Para más información, consulta [Actions Triggers: post-challenge - objeto de API](/docs/es/customize/actions/explore-triggers/password-reset-triggers/post-challenge-trigger/post-challenge-api-object).
