Para obtener un <Tooltip tip="Token de actualización: token que se usa para obtener un token de acceso renovado sin obligar a los usuarios a volver a iniciar sesión." cta="Ver glosario" href="/docs/es/glossary?term=refresh+token">token de actualización</Tooltip>, debes incluir el `offline_access` [scope](/docs/es/get-started/apis/scopes) cuando inicies una solicitud de autenticación a través del endpoint `/authorize`. Asegúrate de habilitar Offline Access en tu API. Para obtener más información, consulta [API Settings](/docs/es/get-started/apis/api-settings).

Por ejemplo, si estás usando el [flujo de código de autorización](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow), la solicitud de autenticación tendría este aspecto:

export const codeExample1 = `https://{yourDomain}/authorize?
    audience={API_AUDIENCE}&
    scope=offline_access&
    response_type=code&
    client_id={yourClientId}&
    redirect_uri={https://yourApp/callback}&
    state={OPAQUE_VALUE}`;

<AuthCodeBlock children={codeExample1} language="http" />

El token de actualización se almacena en la sesión. Luego, cuando es necesario renovar una sesión (por ejemplo, porque ha transcurrido un periodo preconfigurado o el usuario intenta realizar una operación sensible), la aplicación usa el token de actualización en el backend para obtener un nuevo <Tooltip tip="Token de ID: Credencial destinada al propio cliente, en lugar de acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=ID+token">token de identidad</Tooltip>, mediante el endpoint `/oauth/token` con `grant_type=refresh_token`.

Una vez que el usuario se autentica correctamente, la aplicación será redirigida a `redirect_uri`, con un `code` como parte de la URL: `{https://yourApp/callback}?code=BPPLN3Z4qCTvSNOy`. Puedes intercambiar este código por un <Tooltip tip="Token de acceso: Credencial de autorización, en forma de una cadena opaca o JWT, usada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=access+token">token de acceso</Tooltip> mediante el endpoint `/oauth/token`.

La respuesta debe contener un token de acceso y un token de actualización.

```json lines
{
      "access_token": "eyJz93a...k4laUWw",
      "refresh_token": "GEbRxBN...edjnXbL",
      "token_type": "Bearer"
    }
```

Si está solicitando un token de actualización para una aplicación móvil mediante el cliente nativo correspondiente (que es público), no necesita enviar el `client_secret` en la solicitud, ya que solo se requiere para aplicaciones confidenciales.

Los tokens de actualización deben almacenarse de forma segura en una aplicación, ya que permiten que un usuario permanezca autenticado prácticamente para siempre.

Para obtener más información sobre cómo implementar esto mediante el Flujo de Código de Autorización, consulte nuestro tutorial, [Llamar a una API mediante el Flujo de Código de Autorización](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow/call-your-api-using-the-authorization-code-flow). Para otras concesiones, consulte [Flujos de autenticación y autorización](/docs/es/get-started/authentication-and-authorization-flow).

<div id="customize-mfa">
  ## Personaliza MFA
</div>

<Warning>
  La MFA personalizable con los flujos de contraseña del propietario del recurso, Embedded o token de actualización está en acceso anticipado. Al usar esta funcionalidad, aceptas los términos aplicables de la prueba gratuita del [Master Subscription Agreement](https://www.okta.com/legal/) de Okta. Para obtener más información sobre las etapas de lanzamiento de Auth0, consulta [Etapas de lanzamiento del producto](/docs/es/troubleshoot/product-lifecycle/product-release-stages). Para participar en el acceso anticipado, contacta al [equipo de soporte de Auth0](https://support.auth0.com/).
</Warning>

La MFA personalizable permite a los usuarios inscribirse y responder a desafíos con los factores compatibles con tu aplicación que elijan.

Durante la autenticación en el endpoint `oauth/token`, la respuesta devuelve el error `mfa_required`, que incluye el `mfa_token` para usar la API de MFA y el parámetro `mfa_requirements` con una lista de autenticadores:

```json
{
  "error": "mfa_required",
  "error_description": "Multifactor authentication required",
  "mfa_token": "Fe26...Ha",
  "mfa_requirements": {
    "challenge": [
      { "type": "otp" },
      { "type": "push-notification" },
      { "type": "phone" },
      { "type": "recovery-code" }
      { "type": "email"} //solo puede funcionar con challenge
    ]
  }
}
```

Usa el `mfa_token` para llamar al endpoint [`mfa/authenticator`](/docs/es/api/authentication/muti-factor-authentication/list-authenticators) y obtener una lista de todos los factores que el usuario tiene registrados, y hazla coincidir con un tipo que admita tu aplicación. También necesitas obtener el `authenticator_type` correspondiente para emitir desafíos:

```json
[
  {
    "type": "recovery-code",
    "id": "recovery-code|dev_qpOkGUOxBpw6R16t",
    "authenticator_type": "recovery-code",
    "active": true
  },
  {
    "type": "otp",
    "id": "totp|dev_6NWz8awwC8brh2dN",
    "authenticator_type": "otp",
    "active": true
  }
]
```

Aplique el desafío MFA llamando al endpoint [`request/mfa/challenge`](/docs/es/api/authentication/muti-factor-authentication/request-mfa-challenge).

Personalice aún más su flujo MFA con Auth0 Actions. Para obtener más información, lea [Actions Triggers: post-challenge - objeto de API](/docs/es/customize/actions/explore-triggers/password-reset-triggers/post-challenge-trigger/post-challenge-api-object).
