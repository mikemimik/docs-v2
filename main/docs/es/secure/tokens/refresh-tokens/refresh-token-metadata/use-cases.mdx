Los [metadatos del token de actualización](/docs/es/secure/tokens/refresh-tokens/refresh-token-metadata) y los [metadatos de la sesión](/docs/es/manage-users/sessions/session-metadata) te permiten, en conjunto, crear y almacenar datos que persisten durante todo el ciclo de vida de la sesión de Auth0 de un usuario. Este artículo incluye ejemplos de los siguientes casos de uso:

* [Creación de claims personalizados persistentes](/docs/es/secure/tokens/refresh-tokens/refresh-token-metadata/use-cases#create-persistent-custom-claims)
* [Creación de un ID de sesión único](/docs/es/secure/tokens/refresh-tokens/refresh-token-metadata/use-cases#create-a-unique-session-id)
* [Propagación de identificadores de tenant](/docs/es/secure/tokens/refresh-tokens/refresh-token-metadata/use-cases#create-a-tenant-identifier)
* [Gestión de datos transitorios de proveedores de identidad de origen (IDPs)](/docs/es/secure/tokens/refresh-tokens/refresh-token-metadata/use-cases#manage-transient-data-from-upstream-identity-providers-idps)
* [Refuerzo de la seguridad y la detección de fraude](/docs/es/secure/tokens/refresh-tokens/refresh-token-metadata/use-cases#enhance-security-and-fraud-detection)

Para obtener más información, consulta [A guide to Auth0 Session and Refresh Token Metadata](https://auth0.com/blog/auth0-session-refresh-token-metadata-guide).

<Warning>
  Los metadatos de la sesión de Auth0 no son un repositorio seguro de datos y no deben utilizarse para almacenar información confidencial. Esto incluye secretos y PII de alto riesgo, como números de la seguridad social o de tarjetas de crédito, etc. Se recomienda encarecidamente a los clientes de Auth0 que evalúen los datos almacenados en los metadatos y que solo almacenen los necesarios para fines de gestión de identidades y accesos. Para obtener más información, consulta [Cumplimiento del Reglamento General de Protección de Datos de Auth0](/docs/es/secure/data-privacy-and-compliance/gdpr).
</Warning>

<div id="create-persistent-custom-claims">
  ## Crear claims personalizados persistentes
</div>

Los metadatos del token de actualización y los metadatos de la sesión, en conjunto, te permiten crear claims personalizados persistentes para ampliar la información contenida en los tokens de [ID](/docs/es/secure/tokens#id-tokens) y de [acceso](/docs/es/secure/tokens/access-tokens).

Con claims personalizados persistentes, puedes acceder a datos específicos de la aplicación, como:

* Roles de usuario
* Permisos
* ID del tenant
* Y otros atributos necesarios para la autorización y la personalización a lo largo de los intercambios de token de actualización.

Configura un trigger de [Post Login](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger#login-/-post-login) de [Action](/docs/es/customize/actions) para crear claims personalizados persistentes y asignarlos a los metadatos del token de actualización mediante el objeto `api.refreshToken.setMetadata()`.

```javascript custom claim Action example expandable
/**
 * @param {Event} event - Detalles sobre el usuario y la transacción de autenticación.
 * @param {PostLoginActionAPI} api - Interfaz para modificar la transacción de autenticación completada.
 */
exports.onExecutePostLogin = async (event, api) => {
  let customClaimValue1;
  let customClaimValue2;

  // --- Función auxiliar para simular el cálculo de claims personalizados ---
  // En un escenario real, esta función realizaría lógica compleja
  // basada en datos de usuario, APIs externas, etc.
  const calculateCustomClaims = (user) => {
    // Después de realizar los cálculos, retornar
    return { claim1: value1, claim2: value2 };
  };

  // --- Determinar si se trata de un inicio de sesión inicial o un intercambio de token de actualización ---
  // Si event.request.body.grant_type no es 'refresh_token', probablemente es un inicio de sesión interactivo inicial
  const isRefreshTokenGrant = event.request.body.grant_type === 'refresh_token';

  if (!isRefreshTokenGrant) {
    // --- Inicio de sesión inicial (p. ej., contraseña, Social, MFA-OOB) ---
    // Calcular los valores de los claims personalizados
    const calculatedClaims = calculateCustomClaims(event.user);
    customClaimValue1 = calculatedClaims.claim1;
    customClaimValue2 = calculatedClaims.claim2;

    // Almacenar estos valores calculados en los metadatos del token de actualización para persistirlos
    // Verificar si se emitirá un token de actualización y, de ser así, agregar metadatos
if (event.transaction.requested_scopes.indexOf('offline_access') > -1) {
api.refreshToken.setMetadata('customClaim1', customClaimValue1);
api.refreshToken.setMetadata('customClaim2', customClaimValue2);
}

  } else {
    // --- Intercambio de token de actualización ---
    // Usar los valores de claims personalizados de los metadatos del token de actualización
    customClaimValue1 = event.refresh_token?.metadata?.customClaim1;
    customClaimValue2 = event.refresh_token?.metadata?.customClaim2;

  } 
  // --- Por último, agregar los valores determinados como claims personalizados a los tokens ---
  api.idToken.setCustomClaim('custom_claim_1', customClaimValue1);
  api.accessToken.setCustomClaim('custom_claim_1', customClaimValue1);

  api.idToken.setCustomClaim('custom_claim_2', customClaimValue2);
  api.accessToken.setCustomClaim('custom_claim_2', customClaimValue2);
};
```

Durante un [intercambio de token de actualización](/docs/es/secure/tokens/refresh-tokens/use-refresh-tokens#use-refresh-tokens), un trigger posterior de la Action `post-login` puede acceder a estas claims personalizadas mediante el objeto `event.refresh_token.metadata` y aplicarlas a los tokens de actualización recién emitidos usando los objetos `api.idToken.setCustomClaim()` y `api.accessToken.setCustomClaim()`.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Una única Action `post-login` puede gestionar distintos escenarios de `grant_type` usando el objeto `event.request.body.grant_type` para administrar la persistencia de claims. El objeto `event.refresh_token` está disponible en modo de solo lectura durante los intercambios de token de actualización.
</Callout>

<div id="create-a-unique-session-id">
  ## Crear un ID de sesión único
</div>

Los metadatos del token de actualización y los metadatos de la sesión, en conjunto, te permiten crear un ID de sesión único para implementar un identificador de sesión persistente que se mantenga durante toda la duración de la sesión de un usuario, incluso durante las [rotaciones del token de actualización](docs/secure/tokens/refresh-tokens/refresh-token-rotation).

Con IDs de sesión únicos, puedes:

* Registrar con precisión la sesión de un usuario para fines de depuración y auditoría.
* Proporcionar un mecanismo para que las aplicaciones hagan seguimiento del estado interno de la sesión.
* Permitir que las [APIs](/docs/es/get-started/apis#apis) ofrezcan registros granulares, limitación de frecuencia y decisiones de autorización contextuales.
* Ofrecer experiencias de UX coherentes que abarquen múltiples ciclos de vida de los tokens.

Configura un trigger de Action de [Post Login](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger#login-/-post-login) para crear un ID de sesión único y asignarlo a la sesión del usuario mediante los objetos `api.session.setMetadata()` y `api.refreshToken.setMetadata()`.

Agrega el ID de sesión único como un claim personalizado a los tokens de ID y de acceso mediante los objetos `api.idToken.setCustomClaim()` y `api.accessToken.setCustomClaim()`.

```javascript unique session ID Action example expandable
/**
 * @param {Event} event - Detalles sobre el usuario y la transacción de autenticación.
 * @param {PostLoginActionAPI} api - Interfaz para modificar la transacción de autenticación completada.
 */
exports.onExecutePostLogin = async (event, api) => {
  let sessionId;

  // 1) Verificar si ya existe una clave de metadatos de sesión llamada 'ses_id'.
  if (event.session && event.session.metadata && event.session.metadata.ses_id) {
    sessionId = event.session.metadata.ses_id;
  }
  // Si no se encuentra en los metadatos de la sesión, verificar si está disponible en los metadatos del token de actualización.
  // Esto es especialmente relevante para los flujos ROPG donde no hubo una sesión real.
  else if (event.refresh_token && event.refresh_token.metadata && event.refresh_token.metadata.ses_id) {
    sessionId = event.refreshToken.metadata.ses_id;
  }

  // Si no existe un 'ses_id', generar uno nuevo 
  if (!sessionId) {
    sessionId = generateSesId(); // Tu propia función auxiliar para generar un UUID

    // Almacenar el 'ses_id' recién generado en los metadatos de la sesión
    // Hacer esto solo si hay una sesión activa o siendo emitida en el evento. 
    if (event.session) {
      api.session.setMetadata('ses_id', sessionId);
    }

    // Almacenar el 'ses_id' en los metadatos del token de actualización.
    // Hacer esto solo si hay un token de actualización activo o siendo emitido en el evento.
    if (event.refresh_token) {
      api.refreshToken.setMetadata('ses_id', sessionId);
    }
  } else {
    // Además, asegurarse de que los metadatos del token de actualización contengan el ses_id, por si faltaba
    // o fue actualizado en otro lugar. Esto puede ocurrir si el usuario no solicitó offline_access inicialmente pero lo agregó después.
    if (event.refresh_token && event.refresh_token.metadata && !event.refresh_token.metadata.ses_id) {
        api.refreshToken.setMetadata('ses_id', sessionId);
    }
  }


  // 2) Agregar este 'ses_id' como claim personalizado tanto al token de identidad como al access token.
  api.idToken.setCustomClaim('ses_id', sessionId);
  api.accessToken.setCustomClaim('ses_id', sessionId);
};
```

Durante un intercambio de token de actualización, un trigger posterior de Action de Post Login puede acceder a estos claims personalizados mediante el objeto `event.refresh_token.metadata` y aplicarlas a los tokens de actualización recién emitidos mediante los objetos `api.idToken.setCustomClaim()` y `api.accessToken.setCustomClaim()`.

<div id="create-a-tenant-identifier">
  ## Crear un identificador de tenant
</div>

Los metadatos del token de actualización y los metadatos de la sesión te permiten, en conjunto, crear un identificador de tenant persistente para gestionar aplicaciones de multitenencia, en las que una única instancia de una aplicación da servicio a varias organizaciones de clientes, y que se mantiene durante toda la sesión del usuario.

Con un identificador de tenant persistente, puedes:

* Añadir control de acceso dinámico para aplicar fácilmente permisos específicos del tenant en tus aplicaciones y API
* Crear una experiencia de usuario personalizada para ofrecer contenido y funcionalidades relevantes para el contexto actual de tenant del usuario
* Simplificar la lógica de multitenencia centralizando la identificación y propagación del tenant dentro de Auth0
* Mejorar la seguridad evitando la exposición accidental de datos entre tenants al garantizar un contexto de tenant coherente en todos los tokens
* Mejorar la escalabilidad al reducir la necesidad de realizar consultas repetidas a la base de datos o de usar lógica compleja para determinar el contexto del tenant en cada llamada a la API o renovación de token

Configura un trigger de Action de [Post Login](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger#login-/-post-login) para identificar el tenant activo del usuario, ya sea consultando a la aplicación un valor `ext-tenantId` proporcionado durante la solicitud de autenticación, infiriendo el tenant mediante geolocalización o pidiendo al usuario que seleccione el tenant que quiere usar.

Una vez identificado el tenant, asigna el valor del identificador de tenant a la sesión del usuario mediante los objetos `api.session.setMetadata()` y `api.refreshToken.setMetadata()`, y añádelo como un claim personalizado a los tokens de ID y de acceso mediante los objetos `api.idToken.setCustomClaim()` y `api.accessToken.setCustomClaim()`.

```javascript tenant identifier Action example expandable
/** 
 * @param {Event} event - Detalles sobre el usuario y la transacción de autenticación.
 * @param {PostLoginActionAPI} api - Interfaz para modificar la transacción de autenticación completada.
 */
exports.onExecutePostLogin = async (event, api) => {
  let tenantId;

  // 1) Comprobar si ya existe una clave de metadatos de sesión llamada 'tenant_id'.
  if (event.session && event.session.metadata && event.session.metadata.tenant_id) {
    tenantId = event.session.metadata.tenant_id;
  }
  // Si no se encuentra en los metadatos de sesión, comprobar si está disponible en los metadatos del token de actualización.
  // Esto es especialmente relevante para los flujos ROPG donde no hubo una sesión real.
  else if (event.refresh_token && event.refresh_token.metadata && event.refresh_token.metadata.tenant_id) {
    tenantId = event.refreshToken.metadata.tenant_id;
  }

  // Si aún no conocemos el 'tenant_id', lo determinamos 
  if (!tenantId) {
    // Se asume que tenant_id llegó como un parámetro ext-
    tenantId = event.request.query['ext-tenantId'];

    // También podría provenir de la geolocalización en el evento  
    // o de formularios. Si se usan formularios, se abriría el formulario
    // ahora y se ejecutaría el resto del código en la 
    // función onContinuePostLogin
  
    // Almacenar el 'tenant_id' recién generado en los metadatos de sesión 
    // Hacer esto solo si realmente se está emitiendo/existe una sesión en el evento. 
    if (event.session) {
      api.session.setMetadata('tenant_id', tenantId);
    }

    // Almacenar el 'tenant_id' en los metadatos del token de actualización.
    // Hacer esto solo si realmente se está emitiendo/existe un token de actualización en el evento.
    if (event.refresh_token) {
      api.refreshToken.setMetadata('tenant_id', tenantId);
    }
  } else {
    // Además, asegurarse de que los metadatos del token de actualización contengan el tenant_id, por si faltaba
    // o fue actualizado en otro lugar. Esto puede ocurrir si el usuario no solicitó offline_access al principio pero lo agregó después.
    if (event.refresh_token && event.refresh_token.metadata && !event.refresh_token.metadata.tenant_id) {
        api.refreshToken.setMetadata('tenant_id', tenantId);
    }
  }


  // 2) Agregar este 'tenant_id' como un claim personalizado tanto al token de identidad como al token de acceso.
  api.idToken.setCustomClaim('tenant_id', tenantId);
  api.accessToken.setCustomClaim('tenant_id', tenantId);
};
```

Durante un intercambio de token de actualización, una Action de `Post Login` posterior puede acceder a estas claims personalizadas mediante el objeto `event.refresh_token.metadata` y aplicarlas a los tokens de actualización recién emitidos usando los objetos `api.idToken.setCustomClaim()` y `api.accessToken.setCustomClaim()`.

<div id="manage-transient-data-from-upstream-identity-providers-idps">
  ## Gestiona datos transitorios de proveedores de identidad de origen (IDPs)
</div>

Los metadatos del token de actualización y los metadatos de la sesión te permiten, en conjunto, gestionar datos transitorios y contextuales de IDPs de origen durante toda la sesión de un usuario sin almacenarlos de forma permanente en su perfil de Auth0.

Con los datos transitorios, puedes:

* Mantener limpios los perfiles de usuario al evitar almacenar datos transitorios o específicos de la sesión
* Aumentar la flexibilidad al admitir distintos requisitos de datos de varios IDPs sin forzar cambios en el esquema ni sobrecargar los perfiles de usuario persistentes
* Mejorar el cumplimiento al facilitar el cumplimiento de las políticas de privacidad y retención de datos, almacenando los datos transitorios solo durante el tiempo necesario
* Reducir la carga de desarrollo al simplificar el manejo de datos transitorios de IDP, ya que Auth0 Actions y los metadatos gestionan el ciclo de vida de los datos

Configura un trigger de Action de [Post Login](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger#login-/-post-login) para identificar los datos del perfil de usuario contenidos en los objetos `event.request`, `event.user` y `event.context`.

Determina qué datos son transitorios o contextuales y asígnalos a la sesión del usuario con los objetos `api.session.setMetadata()` y `api.refreshToken.setMetadata()`. Luego, agrega los datos transitorios como un claim personalizado a los tokens de ID y de acceso mediante los objetos `api.idToken.setCustomClaim()` y `api.accessToken.setCustomClaim()`.

```javascript transient data Action example expandable
/**
 * @param {Event} event - Detalles sobre el usuario y la transacción de autenticación.
 * @param {PostLoginActionAPI} api - Interfaz para modificar la transacción de autenticación completada.
 */
exports.onExecutePostLogin = async (event, api) => {
  let deviceIdentifier;
  let groups;

  // Ejemplo: Extraer información del dispositivo de los encabezados de la solicitud o del contexto
  // Esto es ilustrativo; la identificación real del dispositivo podría ser más compleja
  if (event.request.user_agent) {
    deviceIdentifier = event.request.user_agent;
  } else {
    deviceIdentifier = 'unknown';
  }

  // Ejemplo: Extraer información del IdP desde el contexto de una conexión upstream
  // Esto depende en gran medida del IdP de origen y de cómo transmite la información.
  // Se asume un custom claim o variable de contexto de una conexión SAML/OIDC, p. ej. "groups"
  if (event.user.groups) {
    groups = event.user.groups;
  } else {
    groups = [];
  }

  // Almacenar los datos transitorios en los metadatos de la sesión
  api.session.setMetadata('deviceIdentifier', deviceIdentifier);
  api.session.setMetadata('groups', groups);

  // Almacenar los datos transitorios en los metadatos del token de actualización para mantener la persistencia entre actualizaciones
  if (event.refreshToken) {
    api.refreshToken.setMetadata('deviceIdentifier', deviceIdentifier);
    api.refreshToken.setMetadata('groups', groups);
  }

  // Opcionalmente, agregar estos como claims a los tokens de acceso si las APIs los necesitan
  // Usar namespaces personalizados es una buena práctica para claims específicos de la aplicación.
  api.accessToken.setCustomClaim('https://myapp.example.com/device_id', deviceIdentifier);
  api.accessToken.setCustomClaim('https://myapp.example.com/groups', groups);

  // Ejemplo: Si el IdP de origen proporciona un "nivel de garantía" para este evento de autenticación
  if (event.transaction && event.transaction.acr_values) { // acr: Authentication Context Class Reference (Referencia de clase de contexto de autenticación)
      api.session.setMetadata('authLevel', event.transaction.acr_values);
      if (event.refreshToken) {
        api.refreshToken.setMetadata('authLevel', event.transaction.acr_values);
      }
      api.accessToken.setCustomClaim('https://myapp.example.com/auth_level', event.transaction.acr_values);
  }
};
```

Durante un intercambio de token de actualización, el trigger de Action `Post Login` posterior puede acceder a estas claims personalizadas mediante el objeto `event.refresh_token.metadata` y aplicarlas a los tokens de actualización emitidos recientemente mediante el objeto `api.accessToken.setCustomClaim()`.

<div id="enhance-security-and-fraud-detection">
  ## Mejora la seguridad y la detección de fraude
</div>

Los metadatos del token de actualización y los metadatos de la sesión, en conjunto, le permiten implementar seguridad adaptativa al posibilitar el seguimiento y la comparación de información contextual a lo largo de la sesión de un usuario, incluidas las rotaciones del token de actualización y las solicitudes de [autenticación silenciosa](/docs/es/authenticate/login/configure-silent-authentication).

Al implementar seguridad adaptativa, puede:

* Detectar amenazas de forma proactiva mediante la identificación y respuesta automáticas ante cambios sospechosos en los datos de contexto del usuario, lo que reduce el riesgo de secuestro de sesión y acceso no autorizado.
* Reducir la fricción para los usuarios legítimos solicitando MFA o verificación adicional solo cuando se detecte una anomalía real, lo que mejora la experiencia de usuario frente a requisitos generales de MFA.

Configure un trigger de Action de [Post Login](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger#login-/-post-login) para identificar datos contextuales del usuario, como la huella digital del dispositivo, la ubicación geográfica, los atributos de red y los atributos de comportamiento. Almacene los datos contextuales en la sesión del usuario para compararlos más adelante, mediante el objeto `api.session.setMetadata()`.

```javascript security detection Action example expandable
/**
 * @param {Event} event - Detalles sobre el usuario y la transacción de autenticación.
 * @param {PostLoginActionAPI} api - Interfaz para modificar la transacción de autenticación completada.
 */
exports.onExecutePostLogin = async (event, api) => {
  // --- Capturar datos contextuales actuales ---
  // Usar las huellas digitales ja3/ja4 proporcionadas por Auth0
  const {ja3, ja4} = event.security_context;
  // Agregar ja3/ja4 a los metadatos si aún no están presentes
  // (el primer inicio de sesión no tiene metadatos configurados)
  if (event.session && !event.session.metadata) {
    api.session.setMetadata('ja3', ja3);
    api.session.setMetadata('ja4', ja4);
  } else {
    // Comparar la huella digital almacenada con la huella digital entrante 
    if(ja3 != event.session?.metadata?.ja3 || ja4 != event.session?.metadata?.ja4) {
      // Si las huellas digitales difieren, desafiar con MFA
      api.authentication.challengeWith(
        { type: 'otp'}, 
        { additionalFactors: [
          { type: 'push-notification'}, { type: 'phone' }
        ]}
      );
    }    
  }
};
```

Durante un intercambio de token de actualización o la autenticación silenciosa, un trigger posterior de `post-login Action` puede realizar una evaluación de riesgos y aplicar respuestas adaptativas.

<div id="access-metadata-with-the-management-api">
  ## Accede a los metadatos con la Auth0 Management API
</div>

Puedes usar los endpoints `GET` [/api/v2/refresh-tokens/&#123;id&#125;](/docs/es/api/management/v2/refresh-tokens/get-refresh-token) y [/api/v2/sessions/&#123;id&#125;](/docs/es/api/management/v2/sessions/get-session) de la Auth0 Management API para recuperar los datos almacenados en los metadatos de un token de actualización o de una sesión.

La respuesta incluye el campo `metadata`, que contiene los datos almacenados:

```json
{
  "id": "object_id",
  "metadata": {
    "deviceIdentifier": "deviceIdentifier"
  }
}
```
