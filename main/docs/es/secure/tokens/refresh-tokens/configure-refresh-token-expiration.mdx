<Tooltip tip="Token de actualización: token utilizado para obtener un token de acceso renovado sin obligar a los usuarios a volver a iniciar sesión." cta="Ver glosario" href="/docs/es/glossary?term=Refresh+tokens">
  Tokens de actualización
</Tooltip>

pueden ser objeto de abuso si se filtran, ya que pueden utilizarse para obtener nuevos

<Tooltip tip="Token de actualización: token utilizado para obtener un token de acceso renovado sin obligar a los usuarios a volver a iniciar sesión." cta="Ver glosario" href="/docs/es/glossary?term=access+tokens">
  tokens de acceso
</Tooltip>

. Para mitigar este riesgo, Auth0 recomienda usar la detección automática de reutilización y la

<Tooltip tip="Token de acceso: credencial de autorización, en forma de una cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=Refresh+Token+Rotation">
  rotación del Refresh Token
</Tooltip>

. La rotación del Refresh Token emite un token de actualización que vence después de una duración predefinida. Después de vencer, el usuario obtiene un nuevo token de actualización de la misma familia, es decir, tokens de actualización que comparten un ID de familia,
o un nuevo par de token de acceso/token de actualización. Para obtener más información, consulta [Refresh Token Rotation](/docs/es/secure/tokens/refresh-tokens/refresh-token-rotation).

Puedes habilitar y configurar dos ajustes de duración del token de actualización, la expiración máxima y la expiración por inactividad, mediante el <Tooltip tip="Auth0 Dashboard: el producto principal de Auth0 para configurar tus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip> o la Auth0 <Tooltip tip="Auth0 Dashboard: el producto principal de Auth0 para configurar tus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip>. Puedes usar una combinación de períodos de expiración máxima y por inactividad para lograr un equilibrio entre seguridad y experiencia de usuario que se adapte a las necesidades de tu empresa.

* **Duración máxima**: Establece la duración de un token de actualización o de una familia de tokens de actualización, tras la cual el usuario debe volver a autenticarse antes de recibir un nuevo token de acceso. Si desactivas esta configuración, la duración máxima será indefinida.
* **Duración por inactividad**: Establece la duración por inactividad de los tokens de actualización emitidos para que expiren si el usuario no está activo en tu aplicación durante un período determinado.

<div id="use-the-dashboard">
  ## Usa el Dashboard
</div>

1. Ve a [Dashboard &gt; Applications](https://manage.auth0.com/#/applications).
2. Selecciona la aplicación que quieres configurar.
3. Ve a la pestaña **Settings**.
4. En **expiración del token de actualización**, habilita **Set Idle Refresh Token Lifetime**. Cuando está habilitada, un token de actualización caduca según el tiempo de inactividad configurado y, una vez vencido, ya no puede usarse. Si la rotación está habilitada, debes establecer una duración de expiración.

   <Frame>![Dashboard Applications Applications pestaña Settings expiración del token de actualización](/docs/images/cdy7uua7fh8z/2EnpIEc4kBxjnS3T67xPzC/22fdb95e2a2f31cee6f01e02d0a709c2/Refresh_Token_Expiration_-_English.png)</Frame>
5. Introduce **Idle Refresh Token Lifetime** en segundos. El token de actualización caduca después del intervalo especificado y ya no puede usarse para obtener un nuevo token de acceso. Cuando la rotación está habilitada, el tiempo de inactividad del token de actualización también se aplica a la posibilidad de obtener nuevos tokens.

   | Duración       | Valor                        |
   | -------------- | ---------------------------- |
   | Predeterminado | 2,592,000 segundos (30 días) |
   | Mínimo         | 1 segundo                    |
   | Máximo         | 31,557,600 segundos (1 año)  |

   <Callout icon="file-lines" color="#0EA5E9" iconType="regular">
     * **Idle Refresh Token Lifetime** no puede superar **Maximum Refresh Token Lifetime**.
     * El cálculo de un año equivale a 365.25 días para tener en cuenta los años bisiestos.
   </Callout>
6. Habilita **Set Maximum Refresh Token Lifetime**. Cuando está habilitada, un token de actualización caduca según la duración máxima especificada y, una vez vencido, ya no puede usarse.
7. Introduce **Maximum Refresh Token Lifetime** en segundos. Si el token de actualización no se intercambia dentro del intervalo especificado, caduca y ya no puede usarse para obtener un nuevo token de acceso. El período de expiración del tiempo de inactividad del token de actualización se renueva cada vez que el token de actualización se intercambia por un nuevo token de acceso dentro de ese intervalo.

   | Duración | Valor                       |
   | -------- | --------------------------- |
   | Mínimo   | 1 segundo                   |
   | Máximo   | 31,557,600 segundos (1 año) |
8. Selecciona **Save Changes**.

<div id="use-the-management-api">
  ## Usa la Management API
</div>

Puedes configurar los valores de vida útil máxima y de inactividad en el payload del endpoint de la Management API [/api/v2/clients/{id}](https://auth0.com/docs/api/management/v2#!/Clients/patch_clients_by_id). Aquí tienes un ejemplo que establece el tiempo de expiración de un token de actualización sin rotación:

```json lines
PATCH /api/v2/clients/{id}
{
  "refresh_token": {
      "rotation_type": "non-rotating",
      "expiration_type": "expiring",
      "token_lifetime": 2592000,
      "infinite_token_lifetime": false,
      "idle_token_lifetime": 604800,
      "infinite_idle_token_lifetime": false
  }
}
```

<div id="support-and-limitations">
  ## Soporte y limitaciones
</div>

* La **duración máxima del token de actualización** se determina en el momento de emitirse y permanece fija durante toda su vigencia, incluso si se actualiza la duración máxima del token de actualización de la aplicación. Sin embargo, puede anular activamente la duración máxima del token de actualización en el momento de la emisión mediante [sesiones con Actions](/docs/es/manage-users/sessions/manage-sessions-actions), incluso durante intercambios de token de actualización realizados correctamente.
* La **duración inactiva del token de actualización** refleja la configuración actual de la aplicación y se actualiza con cada intercambio de token de actualización realizado correctamente.
* Si configura la Duración máxima del token de actualización para una aplicación, cualquier token de actualización sin vencimiento emitido por esa aplicación pasará a ser un token de actualización con vencimiento después de un intercambio de token de actualización realizado correctamente.
* Una vez que un token de actualización sin vencimiento pasa a ser un token de actualización con vencimiento, no se puede revertir a un token de actualización sin vencimiento.
* Puede optar por usar las capacidades de expiración del token de actualización; no se requiere ninguna acción de su parte. Los tokens de actualización existentes no se ven afectados.
* La expiración del token de actualización funciona con los siguientes flujos:

  * [Flujo de código de autorización](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow)
  * [Flujo de código de autorización con Proof Key for Code Exchange](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce)
  * [Flujo de autorización de dispositivo](/docs/es/get-started/authentication-and-authorization-flow/device-authorization-flow)
  * [Flujo de contraseña del propietario del recurso](/docs/es/get-started/authentication-and-authorization-flow/resource-owner-password-flow)
* Todos los Auth0 SDKs admiten la expiración del token de actualización.
* La funcionalidad de expiración del token de actualización cumple con las [recomendaciones de OAuth 2.0 Security BCP](https://www.ietf.org/archive/id/draft-ietf-oauth-security-topics-16.html#section-4.13.2).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  OAuth BCP establece que los tokens de actualización emitidos para aplicaciones basadas en navegador deben tener vencimiento y, además, aplicar una restricción del remitente o rotar los tokens con cada solicitud. Por lo tanto, las SPA usarán la rotación de forma predeterminada y **no admitirán** tokens de actualización sin vencimiento.
</Callout>
