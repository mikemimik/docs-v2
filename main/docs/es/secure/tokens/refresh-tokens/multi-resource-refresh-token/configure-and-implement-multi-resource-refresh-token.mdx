<div id="configure-applications-for-mrrt">
  ## Configurar aplicaciones para MRRT
</div>

Para usar <Tooltip tip="Token de actualización: token que se usa para obtener un token de acceso renovado sin obligar a los usuarios a iniciar sesión de nuevo." cta="Ver glosario" href="/docs/es/glossary?term=Refresh+Tokens">tokens de actualización</Tooltip> de múltiples recursos (MRRT), configure las políticas de token de actualización de su aplicación mediante la [Management API](https://auth0.com/docs/api/management/v2/clients/patch-clients-by-id) de Auth0. Estas políticas especifican qué API y scopes puede solicitar la aplicación durante un intercambio de token de actualización.

Puede definir políticas de MRRT en la propiedad `refresh_token.policies` de la aplicación.

| Propiedad  | Tipo               | Descripción                                                                                                                                                                |
| ---------- | ------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `audience` | string             | El [identificador](/docs/es/get-started/apis/api-settings#general-settings) de la Auth0 API de la aplicación que tendrá permiso para usar el token de actualización.          |
| `scope`    | Arreglo de strings | La lista de scopes permitidos al solicitar un token de acceso para el audience especificado. El scope debe ser igual o más restringido que los scopes definidos en la API. |

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Las propiedades audience y scope deben corresponder a una aplicación existente en su tenant; de lo contrario, se ignorarán de forma silenciosa durante el intercambio de token de actualización.
</Callout>

Para las aplicaciones existentes, haga una solicitud PATCH al endpoint [actualizar un cliente](https://auth0.com/docs/api/management/v2/clients/patch-clients-by-id). Para crear una nueva aplicación, haga una solicitud POST al endpoint [Create a Client](https://auth0.com/docs/api/management/v2/clients/post-clients):

<div id="implement-multi-resource-refresh-token">
  ## Implementa el token de actualización multirrecurso
</div>

Una vez que configures el token de actualización de tu aplicación con políticas de token de actualización de múltiples recursos, podrás empezar a intercambiar un único token de actualización por <Tooltip tip="Token de acceso: credencial de autorización, en forma de una cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=access+tokens">tokens de acceso</Tooltip> para múltiples API.
Para ello, tu aplicación debe iniciar un flujo de inicio de sesión mediante el [flujo de código de autorización](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow) o la [concesión de contraseña del propietario del recurso](/docs/es/authenticate/login/oidc-conformant-authentication/oidc-adoption-rop-flow).

<div id="step-1-authenticate-and-request-a-refresh-token">
  ### Paso 1: Autentícate y solicita un token de actualización
</div>

Para recibir un token de actualización, incluye el scope `offline_access` al iniciar la solicitud de autenticación. Para obtener más información, consulta [Obtener tokens de actualización](/docs/es/secure/tokens/refresh-tokens/get-refresh-tokens).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Si no recibes un token de actualización, confirma lo siguiente:

  * La API tiene [Allow Offline Access](/docs/es/get-started/apis/api-settings#access-settings) habilitada en su configuración.
  * `offline_access` está incluido en el scope.
  * El `audience` usado en la solicitud coincide con una API configurada en tu tenant.
</Callout>

Usa el token de acceso para llamar a la API protegida mediante el [esquema de autorización Bearer de HTTP](https://datatracker.ietf.org/doc/html/rfc6750). Para obtener más información, consulta [Usar tokens de acceso](/docs/es/secure/tokens/access-tokens/use-access-tokens).

<div id="use-multi-resource-refresh-token-with-actions">
  ## Usa el token de actualización multirrecurso con Actions
</div>

Usar token de actualización de múltiples recursos con [Actions](/docs/es/customize/actions) te permite configurar una lógica de decisión dinámica basada en las políticas de token de actualización de múltiples recursos de la aplicación. Para ello, las Actions de Post Login incluyen el objeto [`event.client.refresh_token.policies`](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger/post-login-event-object), que proporciona información relevante, como la <Tooltip tip="Audience: Identificador único de la audiencia de un token emitido. En un token, se denomina aud y su valor contiene el ID de una aplicación (ID de cliente) para un token de ID o de una API (identificador de API) para un token de acceso." cta="Ver glosario" href="/docs/es/glossary?term=audience">audience</Tooltip> y el scope.

Puedes usar el objeto `event.client.refresh_token.policies` para evaluar la audience y el scope de la aplicación al emitir o intercambiar un token de actualización, y así garantizar un control preciso sobre el acceso a la API y los scopes.

```js lines
exports.onExecutePostLogin = async (event, api) => {
  // retornar la lista de APIs permitidas en el cliente
  const allowedAPIsInTheClient = event.client.refresh_token?.policies;

  if(allowedAPIsInTheClient?.some(policy => policy?.audience?.includes('https://myapi'))){
    // lógica personalizada
  }
};
```

<div id="evaluation-logic">
  #### Lógica de evaluación
</div>

MRRT actúa como una extensión de la autenticación original, no como un reemplazo. Al intercambiar un token de actualización, Auth0 evalúa la solicitud de intercambio con la siguiente lógica:

* **Si se omite el parámetro audience**, Auth0 devuelve un token de acceso con la audience original y cualquiera de los scopes adicionales configurados en la política MRRT.
* **Si se especifica un nuevo parámetro audience**, Auth0 verifica que la audience esté incluida en la política MRRT y devuelve un token de acceso para la nueva audience con sus scopes configurados.
* **Si se omite el parámetro scope**, Auth0 combina todos los scopes permitidos de la solicitud original y de la política MRRT.
* **Si se especifica un nuevo parámetro scope**, Auth0 valida los scopes solicitados y devuelve un token de acceso con los scopes incluidos en la política MRTT. Los scopes solicitados que sean no válidos o no autorizados se ignoran silenciosamente.
* **Si el parámetro audience es el mismo que el de la solicitud original**, Auth0 aplica la política MRRT y devuelve un token de acceso para la audience con todos los scopes configurados en MRRT y los scopes de la autenticación original.

MRRT te permite ampliar el acceso del usuario a nuevas API sin emitir nuevos tokens de actualización ni requerir que el usuario inicie sesión de nuevo.

<div id="examples">
  #### Ejemplos
</div>

Un usuario inicia sesión y solicita el siguiente audience y scope:

```json lines
{
"audience": "https://api.example.com",  
"scope": "openid profile read:messages"
}
```

La política de MRRT de la aplicación está configurada para añadir un scope adicional:

```json lines
{
  "audience": "https://api.example.com",
  "scope": ["write:messages"]
}
```

Un intercambio de token de actualización con la misma audiencia y sin scopes daría como resultado un token de acceso que contiene todos los scopes configurados:

```json lines
{
  "aud": "https://api.example.com",
  "scope": "openid profile read:messages write:messages"
}
```
