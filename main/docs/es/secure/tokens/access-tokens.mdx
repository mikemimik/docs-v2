<Tooltip tip="Token de acceso: Credencial de autorización, en forma de una cadena opaca o JWT, usada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=Access+tokens">Tokens de acceso</Tooltip> se usan en la autenticación basada en tokens para permitir que una aplicación acceda a una API. La aplicación recibe un token de acceso después de que un usuario se autentica correctamente y autoriza el acceso, y luego lo envía como credencial cuando llama a la API de destino. El token enviado informa a la API de que el portador del token está autorizado para acceder a la API y realizar las acciones específicas definidas por el <Tooltip tip="Scope: Mecanismo que determina qué acciones las aplicaciones pueden realizar en nombre de un usuario con información creada previamente en un recurso en línea." cta="Ver glosario" href="/docs/es/glossary?term=Scope">Scope</Tooltip> que se concedió durante la autorización.

Además, si has elegido permitir que los usuarios inicien sesión a través de un <Tooltip tip="Proveedor de identidad (IdP): Servicio que almacena y administra identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=Identity+Provider+%28IdP%29">Proveedor de identidad (IdP)</Tooltip>, como Facebook, el <Tooltip tip="Proveedor de identidad (IdP): Servicio que almacena y administra identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=IdP">IdP</Tooltip> emitirá su propio token de acceso para permitir que tu aplicación llame a la API del IdP. Por ejemplo, si tu usuario se autentica con Facebook, el token de acceso emitido por Facebook puede usarse para llamar a la Facebook Graph API. Estos tokens los controla el IdP y pueden emitirse en cualquier formato. Consulta [Tokens de acceso del proveedor de identidad](/docs/es/secure/tokens/access-tokens/identity-provider-access-tokens) para obtener más detalles.

<div id="opaque-access-tokens">
  ## Tokens de acceso opacos
</div>

Los tokens de acceso opacos son tokens en un formato propietario cuyo contenido no puedes inspeccionar y que, por lo general, contienen algún identificador de información almacenada de forma persistente en un servidor. Para validar un token opaco, el receptor del token debe llamar al servidor que lo emitió.

En el caso de Auth0, los tokens opacos pueden usarse con el endpoint `/userinfo` para devolver el perfil de un usuario. Si recibes un token de acceso opaco, no necesitas validarlo. Puedes usarlo con el endpoint `/userinfo`, y Auth0 se encarga del resto. Para obtener más información, consulta [Obtener tokens de acceso](/docs/es/secure/tokens/access-tokens/get-access-tokens).

<div id="jwt-access-tokens">
  ## Tokens de acceso JWT
</div>

Los tokens de acceso <Tooltip tip="JSON Web Token (JWT): Formato estándar de token de ID (y, a menudo, de token de acceso) utilizado para representar claims de forma segura entre dos partes." cta="Ver glosario" href="/docs/es/glossary?term=JSON+Web+Token">JSON Web Token</Tooltip> (JWT) se ajustan al [estándar JWT](https://tools.ietf.org/html/rfc7519) y contienen información sobre una entidad en forma de claims. Son autocontenidos, por lo que no es necesario que el destinatario llame a un servidor para validar el token.

Los tokens de acceso emitidos para la <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip> y los tokens de acceso emitidos para cualquier API personalizada que haya registrado con Auth0 siguen el estándar JWT, lo que significa que su estructura básica se ajusta a la [estructura típica de JWT](/docs/es/secure/tokens/json-web-tokens/json-web-token-structure) y que contienen [claims estándar de JWT](/docs/es/secure/tokens/json-web-tokens/json-web-token-claims) sobre el propio token.

<div id="management-api-access-tokens">
  ### Tokens de acceso de la Management API
</div>

Un token de acceso emitido para Auth0 Management API debe tratarse como opaco (independientemente de si realmente lo es), por lo que no necesitas validarlo. Puedes usarlo con Auth0 Management API, y Auth0 se encarga del resto. Para obtener más información, consulta [Tokens de Auth0 Management API](/docs/es/secure/tokens/access-tokens/management-api-access-tokens).

<div id="custom-api-access-tokens">
  ### Tokens de acceso para API personalizadas
</div>

Si falla la validación de tu token de acceso para API personalizada, asegúrate de que se haya emitido con tu API personalizada como `audience`. Para obtener más información, consulta [Obtener tokens de acceso](/docs/es/secure/tokens/access-tokens/get-access-tokens).

<div id="sample-access-token">
  ## Ejemplo de token de acceso
</div>

Este ejemplo muestra el contenido de un token de acceso. Ten en cuenta que el token solo contiene información de autorización sobre las acciones que la aplicación puede realizar en la API (estos permisos se denominan `scopes`).

```json lines
{
  "iss": "https://my-domain.auth0.com/",
  "sub": "auth0|123456",
  "aud": [
    "https://example.com/health-api",
    "https://my-domain.auth0.com/userinfo"
  ],
  "azp": "my_client_id",
  "exp": 1311281970,
  "iat": 1311280970,
  "scope": "openid profile read:patients read:admin"
}
```

El token no contiene ninguna información sobre el usuario, excepto el ID de usuario (que se encuentra en el claim `sub`). En muchos casos, puede resultarle útil obtener información adicional del usuario. Puede hacerlo llamando al [endpoint de la API userinfo](https://auth0.com/docs/api/authentication#get-user-info) con el token de acceso. Asegúrese de que la API para la que se emite el token de acceso use el **RS256** [algoritmo de firma](/docs/es/get-started/applications/signing-algorithms).

<div id="access-token-security">
  ## Seguridad de los tokens de acceso
</div>

Debes seguir las [prácticas recomendadas sobre tokens](/docs/es/secure/tokens/token-best-practices) al usar tokens de acceso y, en el caso de los JWT, asegúrate de [validar un token de acceso](/docs/es/secure/tokens/access-tokens/validate-access-tokens) antes de dar por hecho que se puede confiar en su contenido.

<div id="access-token-lifetime">
  ## Duración del token de acceso
</div>

<div id="custom-api-token-lifetime">
  ### Duración del token de acceso para la API personalizada
</div>

De forma predeterminada, un token de acceso para una API personalizada es válido durante 86400 segundos (24 horas). Te recomendamos que configures el periodo de validez de tu token en función de los requisitos de seguridad de tu API. Por ejemplo, un token de acceso para una API bancaria debería caducar antes que uno para una API de lista de tareas. Para obtener más información, consulta [Actualizar la duración del token de acceso](/docs/es/secure/tokens/access-tokens/update-access-token-lifetime).

<div id="userinfo-endpoint-token-lifetime">
  ### duración del token de acceso del endpoint `/userinfo`
</div>

Los tokens de acceso emitidos exclusivamente para acceder al endpoint OIDC `/userinfo` tienen una duración predeterminada que no se puede cambiar. La duración depende del flujo utilizado para obtener el token:

| Flujo                          | Duración                  |
| ------------------------------ | ------------------------- |
| Implicit                       | 7200 segundos (2 horas)   |
| Código de autorización/Híbrido | 86400 segundos (24 horas) |
