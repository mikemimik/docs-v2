title: Tokens de actualización
---

Auth0 emite un <Tooltip tip="Token de acceso: Credencial de autorización, en forma de una cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=access+token">token de acceso</Tooltip> o un <Tooltip tip="Token de acceso: Credencial de autorización, en forma de una cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=ID+token">token de ID</Tooltip> en respuesta a una solicitud de autenticación. Puedes usar tokens de acceso para realizar llamadas autenticadas a una API protegida, mientras que el token de ID contiene atributos del perfil del usuario representados en forma de claims. Ambos son <Tooltip tip="Token de ID: Credencial destinada al propio cliente, en lugar de para acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=JSON+web+tokens">JSON Web Tokens</Tooltip> (JWT) y, por lo tanto, tienen fechas de vencimiento indicadas mediante el claim `exp`, así como medidas de seguridad, como firmas. Normalmente, un usuario necesita un nuevo token de acceso cuando accede a un recurso por primera vez o después de que expire el token de acceso anterior que se le concedió.

Un <Tooltip tip="OAuth 2.0: Marco de autorización que define protocolos y flujos de autorización." cta="Ver glosario" href="/docs/es/glossary?term=OAuth">OAuth</Tooltip> <Tooltip tip="OAuth 2.0: Marco de autorización que define protocolos y flujos de autorización." cta="Ver glosario" href="/docs/es/glossary?term=Refresh+Token">token de actualización</Tooltip> es una credencial que OAuth puede usar para obtener un nuevo token de acceso sin interacción del usuario. Esto permite que el <Tooltip tip="Servidor de autorización: Servidor centralizado que contribuye a definir los límites del acceso de un usuario. Por ejemplo, tu servidor de autorización puede controlar los datos, las tareas y las funcionalidades disponibles para un usuario." cta="Ver glosario" href="/docs/es/glossary?term=Authorization+Server">servidor de autorización</Tooltip> acorte la duración del token de acceso por motivos de seguridad sin involucrar al usuario cuando expire el token de acceso. Puedes solicitar nuevos tokens de acceso hasta que el token de actualización esté en la DenyList.

Es importante mantener la cantidad de tokens de actualización dentro de un límite razonable y manejable para asegurarte de que sea fácil administrar esas credenciales de forma segura. Las aplicaciones deben almacenar los tokens de actualización de forma segura porque, en esencia, permiten que un usuario permanezca autenticado indefinidamente.

<Card title="Acceso sin conexión">
  Si quieres permitir que los usuarios obtengan tokens de actualización mientras están sin conexión, puedes activar el interruptor **Allow Offline Access** en [Configuración de la API](/docs/es/get-started/apis/api-settings).
</Card>

Puedes aumentar la seguridad usando la [rotación del Refresh Token](/docs/es/secure/tokens/refresh-tokens/refresh-token-rotation), que emite un nuevo token de actualización e invalida el token anterior con cada solicitud realizada a Auth0 para obtener un nuevo token de acceso. Rotar el token de actualización reduce el riesgo de que se vea comprometido.

<div id="limitations">
  ### Limitaciones
</div>

* Auth0 limita a 200 la cantidad de tokens de actualización activos por usuario y por aplicación. Este límite solo se aplica a los tokens activos. Si se alcanza el límite y se crea un nuevo token de actualización, el sistema revoca y elimina el token más antiguo de ese usuario y aplicación. Los tokens revocados y los tokens vencidos no cuentan para el límite. Para consultar nuestras recomendaciones y prácticas recomendadas para evitar el exceso de tokens, lee [Prácticas recomendadas sobre tokens](/docs/es/secure/tokens/token-best-practices).

* Después de un [intercambio de token de actualización](/docs/es/secure/tokens/refresh-tokens/use-refresh-tokens), Auth0 no actualiza las propiedades `event.refresh_token.device.last*` para los tokens que no expiran. Para obtener más información sobre cómo habilitar tokens de actualización con vencimiento, lee [Configurar la expiración del token de actualización](/docs/es/secure/tokens/refresh-tokens/configure-refresh-token-expiration).

<div id="enable-oidc-flag">
  ### Habilitar el indicador OIDC
</div>

El comportamiento del token de actualización se aplica a las aplicaciones conformes con OIDC. Puedes configurar una aplicación para que sea conforme con OIDC de una de las siguientes maneras:

1. Habilitando el **indicador OIDC Conformant** para una aplicación.
2. Enviando un claim `audience` al endpoint `/authorize` de la Authentication API.

  ## Soporte para el SDK
</div>

<div id="for-web-apps">
  ### Para aplicaciones web
</div>

Los SDKs de Auth0 admiten tokens de actualización, incluidos:

* Node.js
* ASP.NET Core
* PHP
* Java

Para ver la lista completa, consulta [Quickstarts](/docs/es/quickstart/webapp).

<div id="for-single-page-apps">
  ### Para aplicaciones de página única
</div>

Proporcionar una autenticación segura en las SPA plantea varios desafíos según el caso de uso de tu aplicación. Los nuevos controles de privacidad de los navegadores, como Intelligent Tracking Prevention (ITP), afectan negativamente la experiencia del usuario en las SPA al impedir el acceso a las cookies de terceros.

Auth0 recomienda usar <Tooltip tip="rotación del Refresh Token: Estrategia que reemplaza con frecuencia los tokens de actualización para minimizar la vulnerabilidad. Con la rotación del Refresh Token, cada vez que tu aplicación intercambia un token de actualización para obtener un nuevo token de acceso, Auth0 también devuelve un nuevo token de actualización." cta="Ver glosario" href="/docs/es/glossary?term=refresh+token+rotation">rotación del Refresh Token</Tooltip>, que proporciona un método seguro para usar tokens de actualización en las SPA y, al mismo tiempo, ofrece a los usuarios finales acceso continuo a los recursos sin las interrupciones en la UX que provocan tecnologías de privacidad del navegador como ITP.

<div id="for-nativemobile-apps">
  ### Para aplicaciones nativas/móviles
</div>

Para las aplicaciones nativas, los tokens de actualización mejoran significativamente la experiencia de autenticación. El usuario solo tiene que autenticarse una vez, mediante el proceso de autenticación web. La reautenticación posterior puede realizarse sin interacción del usuario, usando el token de actualización.

Para obtener información sobre cómo usar tokens de actualización con nuestros SDKs móviles, consulta:

* [Quickstarts para móviles/nativos](/docs/es/quickstart/native)
* [Lock Android: Actualizar tokens JWT](/docs/es/libraries/lock-android/lock-android-refresh-jwt)
* [Lock iOS: Guardar y renovar tokens JWT](/docs/es/libraries/auth0-swift/auth0-swift-save-and-renew-tokens)
