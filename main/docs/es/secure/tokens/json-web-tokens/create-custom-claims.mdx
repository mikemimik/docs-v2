---

Para leer claims personalizadas en los tokens de acceso y los <Tooltip tip="token de ID: Credencial destinada al propio cliente, en lugar de para acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=ID+tokens">tokens de ID</Tooltip>, debes usar <Tooltip tip="JSON Web Token (JWT): Formato estándar de token de ID (y, a menudo, de token de acceso) utilizado para representar claims de forma segura entre dos partes." cta="Ver glosario" href="/docs/es/glossary?term=JSON+Web+Tokens">JSON Web Tokens</Tooltip> (JWT) e incluir una <Tooltip tip="audience: Identificador del destinatario previsto de un token, que normalmente representa una API o un servidor de recursos de destino." cta="Ver glosario" href="/docs/es/glossary?term=audience">audience</Tooltip> (`aud`) en un flujo de login de OIDC. Para obtener más información, consulta [tokens de acceso](/docs/es/secure/tokens/access-tokens).

Al configurar claims personalizadas en JWT, debes evitar las colisiones. Para evitar que tus claims personalizadas entren en conflicto con claims reservadas o con claims de otros recursos, asígnales un nombre resistente a colisiones. Auth0 recomienda usar un formato con espacio de nombres.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Auth0 permite claims con espacio de nombres y sin espacio de nombres, pero se aplican ciertas restricciones (consulta [Restricciones generales](/docs/es/secure/tokens/json-web-tokens/create-custom-claims#general-restrictions)). Para evitar colisiones de nombres, recomendamos usar claims con espacio de nombres. En caso de colisión, la transacción no fallará, pero tu claim personalizada no se agregará a tus tokens.
</Callout>

<div id="general-restrictions">
  ## Restricciones generales
</div>

Auth0 aplica las siguientes restricciones a los claims personalizados:

* La carga útil de los claims personalizados tiene un tamaño máximo de 100 KB
* Los [claims estándar de OPENID](https://openid.net/specs/openid-connect-core-1_0.html#IDToken) y los claims que Auth0 usa internamente no se pueden personalizar ni modificar
* Los <Tooltip tip="Token de acceso: credencial de autorización, en forma de cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=Access+tokens">tokens de acceso</Tooltip> con una audiencia de Auth0 API, excluido el endpoint `/userinfo`, no pueden tener claims personalizados privados sin espacio de nombres
* Solo se pueden agregar a los tokens de acceso los claims especificados del perfil de usuario de OIDC

<div id="non-restricted-claims">
  ### Claims no restringidos
</div>

Puedes crear claims para información sensible del usuario a fin de mejorar el perfil de usuario y la experiencia del usuario. Tu aplicación consume estos claims a partir de los tokens de ID. Para obtener más información sobre cómo usar claims no restringidos, consulta [Tokens de ID](/docs/es/secure/tokens/id-tokens) y ten en cuenta las [prácticas recomendadas para tokens](/docs/es/secure/tokens/token-best-practices) si los utilizas.

Los siguientes claims solo están sujetos a restricciones generales:

* `address`
* `birthdate`

* `email_verified`

* `gender`

* `locale`
* `middle_name`

* `picture`
* `preferred_username`
* `profile`

* `website`
* `zoneinfo`

<div id="namespaced-guidelines">
  ## Pautas para espacios de nombres
</div>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  El URN de Auth0 `urn:auth0` no se puede usar como identificador de espacio de nombres.
</Callout>

Usa las siguientes pautas para los identificadores de espacio de nombres:

* Usa cualquier URL HTTP o HTTPS que no sea de Auth0 como identificador de espacio de nombres. Los dominios de Auth0 no se pueden usar como identificadores de espacio de nombres e incluyen:

  * auth0.com
  * webtask.io
  * webtask.run
* Usa una URL que controles como identificador de espacio de nombres; así evitas el riesgo de que otra persona esté usando el mismo espacio de nombres. La URL del espacio de nombres no tiene que apuntar a un recurso real. Solo se usa como identificador; no se invocará.
* La URL debe comenzar con `http://` o `https://`.
* Como alternativa, también puedes usar identificadores de espacio de nombres basados en URN si usas una API personalizada. En ese caso, el URN de Auth0 `urn:auth0` está reservado y no se puede usar como identificador de espacio de nombres.
* Crea varios espacios de nombres según sea necesario.

Una vez que hayas elegido tu espacio de nombres, añade el claim para crear un claim con espacio de nombres, que se puede agregar a un token. Por ejemplo:

`http://www.example.com/favorite_color`

<div id="non-namespaced-guidelines">
  ## Pautas para claims sin espacio de nombres
</div>

Usa las siguientes pautas para los claims personalizados sin espacio de nombres:

* A menos que sea absolutamente necesario para tu aplicación, usa claims personalizados públicos y con espacio de nombres que resistan colisiones.
* Crea claims con nombres significativos y resistentes a colisiones. Por ejemplo, usa `employee_id` en lugar de `e_id`.
* Mantén los nombres y valores de los claims lo más ligeros posible y pasa solo los datos estrictamente necesarios para tu aplicación.
* Evita asignar payloads pesados a los claims personalizados.

Para ver más ejemplos de claims personalizados agregados a un token, consulta [Sample Use Cases: Scopes and Claims](/docs/es/get-started/apis/scopes/sample-use-cases-scopes-and-claims).

<div id="create-custom-claims">
  ## Crear claims personalizadas
</div>

Usa Auth0 Actions para crear claims personalizadas. El objeto `api` te permite usar el método `setCustomClaim` en tokens de acceso o tokens de ID.

<div id="example">
  ### Ejemplo
</div>

```js lines
exports.onExecuteCredentialsExchange = async (event, api) => {
  api.accessToken.setCustomClaim('myClaim', 'this is a private, non namespaced claim');
};
```
