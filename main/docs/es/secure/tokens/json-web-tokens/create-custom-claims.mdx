Para leer claims personalizadas en los tokens de acceso y <Tooltip tip="Token de ID: credencial destinada al propio cliente, en lugar de acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=ID+tokens">tokens de ID</Tooltip>, debes usar <Tooltip tip="Token de ID: credencial destinada al propio cliente, en lugar de acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=JSON+Web+Tokens">JSON Web Tokens</Tooltip> (JWT) e incluir un <Tooltip tip="JSON Web Token (JWT): formato estándar de token de ID (y, a menudo, formato de token de acceso) utilizado para representar claims de forma segura entre dos partes." cta="Ver glosario" href="/docs/es/glossary?term=audience">audience</Tooltip> (`aud`) en un flujo de inicio de sesión de OIDC. Para obtener más información, consulta [Tokens de acceso](/docs/es/secure/tokens/access-tokens).

Al configurar claims personalizadas en JWT, debes evitar colisiones. Para impedir que tus claims personalizadas entren en conflicto con claims reservadas o con claims de otros recursos, asígnales un nombre resistente a colisiones. Auth0 recomienda usar un formato con espacio de nombres.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Auth0 permite claims con espacio de nombres y sin espacio de nombres, pero se aplican ciertas restricciones (consulta [Restricciones generales](/docs/es/secure/tokens/json-web-tokens/create-custom-claims#general-restrictions)). Para evitar colisiones de nombres, recomendamos usar claims con espacio de nombres. En caso de colisión, la transacción no fallará, pero tu claim personalizada no se añadirá a tus tokens.
</Callout>

<div id="general-restrictions">
  ## Restricciones generales
</div>

Auth0 aplica las siguientes restricciones a las claims personalizadas:

* La carga útil de las claims personalizadas tiene un tamaño máximo de 100 KB
* Las [claims estándar de OPENID](https://openid.net/specs/openid-connect-core-1_0.html#IDToken) y las claims que Auth0 usa internamente no se pueden personalizar ni modificar
* <Tooltip tip="Token de acceso: credencial de autorización, en forma de cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=Access+tokens">Tokens de acceso</Tooltip> con una audiencia de la Auth0 API, excluido el endpoint `/userinfo`, no pueden tener claims personalizadas privadas sin espacio de nombres
* Solo se pueden agregar a los tokens de acceso las claims especificadas del perfil de usuario de OIDC

<div id="non-restricted-claims">
  ### Claims no restringidos
</div>

Puedes crear claims para información sensible del usuario a fin de enriquecer el perfil del usuario y mejorar la experiencia de usuario. Tu aplicación obtiene estos claims de los tokens de ID. Para obtener más información sobre cómo usar claims no restringidos, consulta [Tokens de ID](/docs/es/secure/tokens/id-tokens) y ten en cuenta las [prácticas recomendadas para tokens](/docs/es/secure/tokens/token-best-practices) si los utilizas.

Los siguientes claims solo están sujetos a restricciones generales:

* `address`
* `birthdate`
* `email`
* `email_verified`
* `family_name`
* `gender`
* `given_name`
* `locale`
* `middle_name`

* `phone_number_verified`
* `picture`
* `preferred_username`
* `profile`
* `updated_at`
* `website`
* `zoneinfo`

<div id="namespaced-guidelines">
  ## Directrices sobre espacios de nombres
</div>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  El URN de Auth0 `urn:auth0` no se puede usar como identificador de espacio de nombres.
</Callout>

Usa las siguientes directrices para los identificadores de espacio de nombres:

* Usa cualquier URL HTTP o HTTPS que no sea de Auth0 como identificador de espacio de nombres. Los dominios de Auth0 no se pueden usar como identificadores de espacio de nombres e incluyen:

  * auth0.com
  * webtask.io
  * webtask.run
* Usa una URL que controles como identificador de espacio de nombres; esto te permite evitar el riesgo de que otra persona use el mismo espacio de nombres. La URL del espacio de nombres no tiene que apuntar a un recurso real. Solo se usa como identificador; no se invocará.
* Comienza la URL con `http://` o `https://`.
* Como alternativa, también puedes usar identificadores de espacio de nombres basados en URN si usas una API personalizada. En ese caso, el URN de Auth0 `urn:auth0` está reservado y no se puede usar como identificador de espacio de nombres.
* Crea varios espacios de nombres según sea necesario.

Una vez que hayas elegido tu espacio de nombres, añádele el claim para crear un claim con espacio de nombres, que puede añadirse a un token. Por ejemplo:

`http://www.example.com/favorite_color`

<div id="non-namespaced-guidelines">
  ## Directrices para claims sin espacio de nombres
</div>

Usa las siguientes directrices para los claims personalizados sin espacio de nombres:

* A menos que sea absolutamente necesario para tu aplicación, usa claims personalizados públicos y con espacio de nombres que sean resistentes a colisiones.
* Crea claims con nombres significativos y resistentes a colisiones. Por ejemplo, usa `employee_id` en lugar de `e_id`.
* Mantén los nombres y valores de los claims lo más livianos posible y pasa solo los datos estrictamente necesarios para tu aplicación.
* Evita asignar payloads voluminosos a los claims personalizados.

Para ver más ejemplos de claims personalizados agregados a un token, consulta [Casos de uso de ejemplo: scopes y claims](/docs/es/get-started/apis/scopes/sample-use-cases-scopes-and-claims).

<div id="create-custom-claims">
  ## Crear claims personalizadas
</div>

Usa Auth0 Actions para crear claims personalizadas. El objeto `api` te permite usar el método `setCustomClaim` en tokens de acceso o de ID.

<div id="example">
  ### Ejemplo
</div>

```js lines
exports.onExecuteCredentialsExchange = async (event, api) => {
  api.accessToken.setCustomClaim('myClaim', 'this is a private, non namespaced claim');
};
```
