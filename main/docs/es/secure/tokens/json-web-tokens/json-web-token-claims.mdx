---
description: Describe los claims de JSON Web Token (JWT) y cómo se utilizan en Auth0.
title: Claims de JSON Web Token
---

Los claims de [JSON web tokens (JWTs)](/docs/es/secure/tokens/json-web-tokens) son fragmentos de información que se afirman sobre un sujeto. Por ejemplo, un [token de identidad](/docs/es/secure/tokens/id-tokens) (que siempre es un <Tooltip tip="Formato estándar de token de ID (y, a menudo, de token de acceso) utilizado para representar claims de forma segura entre dos partes." cta="Ver glosario" href="/docs/es/glossary?term=JWT">JWT</Tooltip>) puede contener un claim llamado `name` que indica que el nombre del usuario que se está autenticando es &quot;John Doe&quot;. En un JWT, un claim aparece como un par nombre/valor, donde el nombre siempre es una cadena y el valor puede ser cualquier valor JSON. En general, cuando hablamos de un claim en el contexto de un JWT, nos referimos al nombre (o la clave). Por ejemplo, el siguiente objeto JSON contiene tres claims (`sub`, `name`, `admin`):

```json lines
{
      "sub": "1234567890",
      "name": "John Doe",
      "admin": true
    }
```

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Para obtener más información sobre los claims estándar de OpenID Connect (OIDC), consulta [OpenID Connect Scopes](/docs/es/get-started/apis/scopes/openid-connect-scopes) y [OpenID Connect Standard Claims](https://openid.net/specs/openid-connect-core-1_0.html#StandardClaims).
</Callout>

Hay dos tipos de claims JWT:

* **Registered**: claims estándar registrados en la [Internet Assigned Numbers Authority (IANA)](https://www.iana.org/assignments/jwt/jwt.xhtml) y definidos por la [especificación JWT](https://tools.ietf.org/html/rfc7519) para garantizar la interoperabilidad con aplicaciones de terceros o externas. Los claims estándar de OIDC son claims reservados.
* **Custom**: consiste en claims públicos o privados no registrados. Los claims públicos son resistentes a colisiones, mientras que los claims privados pueden estar sujetos a colisiones.

<div id="registered-claims">
  ## Claims registrados
</div>

La especificación JWT define siete claims reservados que no son obligatorios, pero se recomiendan para permitir la interoperabilidad con [aplicaciones de terceros](/docs/es/get-started/applications/third-party-applications/configure-third-party-applications). Estos son:

* `iss` (issuer): Emisor del JWT
* `sub` (subject): Sujeto del JWT (el usuario)
* `aud` (<Tooltip tip="Audience: Identificador único del audience de un token emitido. Se denomina aud en un token, y su valor contiene el ID de una aplicación (ID de cliente) para un token de ID o de una API (identificador de API) para un token de acceso." cta="Ver glosario" href="/docs/es/glossary?term=audience">audience</Tooltip>): Destinatario al que va dirigido el JWT
* `exp` (expiration time): Momento a partir del cual el JWT expira
* `nbf` (not before time): Momento antes del cual el JWT no debe aceptarse para su procesamiento
* `iat` (issued at time): Momento en que se emitió el JWT; puede usarse para determinar la antigüedad del JWT
* `jti` (JWT ID): Identificador único; puede usarse para evitar que el JWT se reproduzca (permite que un token se use solo una vez)

Puedes ver una lista completa de los claims registrados en el [IANA JSON Web Token Claims Registry](https://www.iana.org/assignments/jwt/jwt.xhtml#claims).

<div id="custom-claims">
  ## Claims personalizadas
</div>

Puedes definir tus propias claims personalizadas, que controlas tú mismo, y agregarlas a un token mediante Actions. Aquí tienes algunos ejemplos:

* Agrega la dirección de correo electrónico de un usuario a un <Tooltip tip="Token de acceso: credencial de autorización, en forma de una cadena opaca o JWT, usada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=access+token">token de acceso</Tooltip> y úsala para identificar al usuario de forma única.
* Agrega información personalizada almacenada en el perfil de usuario de Auth0 a un <Tooltip tip="Token de identidad: credencial destinada al propio cliente, en lugar de para acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=ID+token">token de identidad</Tooltip>.

Siempre que la Action esté implementada, las claims personalizadas que agregue aparecerán en los nuevos tokens emitidos al usar un [token de actualización](/docs/es/secure/tokens/refresh-tokens).

Para ver un ejemplo de cómo agregar claims personalizadas a un token, consulta [Casos de uso de ejemplo: scopes y claims](/docs/es/get-started/apis/scopes/sample-use-cases-scopes-and-claims).

<div id="public-claims">
  ### Claims públicas
</div>

Puedes crear claims personalizadas para uso público, que pueden contener información genérica, como el nombre y el correo electrónico. Si creas claims públicas, debes registrarlas o usar nombres resistentes a colisiones mediante espacios de nombres, y tomar precauciones razonables para asegurarte de que controlas el espacio de nombres que utilizas.

En el [Registro de Claims de JSON Web Token de la IANA](https://www.iana.org/assignments/jwt/jwt.xhtml#claims), puedes ver algunos ejemplos de claims públicas registradas por <Tooltip tip="OpenID: estándar abierto de autenticación que permite a las aplicaciones verificar la identidad de los usuarios sin recopilar ni almacenar información de inicio de sesión." cta="Ver glosario" href="/docs/es/glossary?term=OpenID">OpenID</Tooltip> Connect (OIDC):

* `auth_time`
* `acr`
* `nonce`

<div id="private-claims">
  ### Claims privados
</div>

Puedes crear claims personalizados privados para compartir información específica de tu aplicación. Por ejemplo, aunque un claim público puede contener información genérica, como el nombre y el correo electrónico, los claims privados serían más específicos, como el ID de empleado y el nombre del departamento.

<div id="auth0-restrictions">
  ## Restricciones de Auth0
</div>

Auth0 aplica las restricciones generales a los claims personalizados:

* el payload de los claims personalizados tiene un tamaño máximo de 100 KB
* no se puede personalizar ni modificar un subconjunto de los claims estándar registrados de OIDC y otros claims que Auth0 utiliza internamente
* los tokens de acceso con una audience de API de Auth0, excluido el endpoint `/userinfo`, no pueden tener claims personalizados privados sin espacio de nombres
* solo se pueden agregar a los tokens de acceso los claims especificados del perfil de usuario de OIDC

Para obtener más información sobre los claims personalizados, lee [Create Custom Claims](/docs/es/secure/tokens/json-web-tokens/create-custom-claims).
