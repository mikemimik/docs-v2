description: Describe los claims de JSON Web Token (JWT) y cómo se usan en Auth0.
title: Claims de JSON Web Token
---

Los claims de los [JSON Web Tokens (JWT)](/docs/es/secure/tokens/json-web-tokens) son fragmentos de información que se afirman sobre un sujeto. Por ejemplo, un [token de identidad](/docs/es/secure/tokens/id-tokens) (que siempre es un <Tooltip tip="JSON Web Token (JWT): formato estándar de token de identidad (y, a menudo, de access token) que se usa para representar claims de forma segura entre dos partes." cta="Ver glosario" href="/docs/es/glossary?term=JWT">JWT</Tooltip>) puede contener un claim llamado `name` que afirma que el nombre del usuario que se autentica es &quot;John Doe&quot;. En un JWT, un claim aparece como un par nombre/valor, donde el nombre siempre es una cadena y el valor puede ser cualquier valor JSON. En general, cuando hablamos de un claim en el contexto de un JWT, nos referimos al nombre (o clave). Por ejemplo, el siguiente objeto JSON contiene tres claims (`sub`, `name`, `admin`):

```json lines
{
      "sub": "1234567890",
      "name": "John Doe",
      "admin": true
    }
```

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Para conocer las claims estándar de OpenID Connect (OIDC), consulta [Scopes de OpenID Connect](/docs/es/get-started/apis/scopes/openid-connect-scopes) y [Claims estándar de OpenID Connect](https://openid.net/specs/openid-connect-core-1_0.html#StandardClaims).
</Callout>

Hay dos tipos de claims en JWT:

* **Registradas**: claims estándar registradas en la [Internet Assigned Numbers Authority (IANA)](https://www.iana.org/assignments/jwt/jwt.xhtml) y definidas por la [especificación JWT](https://tools.ietf.org/html/rfc7519) para garantizar la interoperabilidad con aplicaciones de terceros o externas. Las claims estándar de OIDC son claims reservadas.
* **Custom**: consiste en claims públicas o privadas no registradas. Las claims públicas son resistentes a colisiones, mientras que las claims privadas están sujetas a posibles colisiones.

<div id="registered-claims">
  ## Claims registradas
</div>

La especificación JWT define siete claims reservadas que no son obligatorias, pero se recomiendan para permitir la interoperabilidad con [aplicaciones de terceros](/docs/es/get-started/applications/third-party-applications/configure-third-party-applications). Estas son:

* `iss` (issuer): Emisor del JWT
* `sub` (subject): Sujeto del JWT (el usuario)
* `aud` (<Tooltip tip="Audience: Identificador único de la audiencia de un token emitido. Se llama aud en un token, y su valor contiene el ID de una aplicación (ID de cliente) para un token de identidad o de una API (identificador de API) para un token de acceso." cta="Ver glosario" href="/docs/es/glossary?term=audience">audiencia</Tooltip>): Destinatario al que va dirigido el JWT
* `exp` (expiration time): Momento después del cual el JWT expira
* `nbf` (not before time): Momento antes del cual el JWT no debe aceptarse para su procesamiento
* `iat` (issued at time): Momento en el que se emitió el JWT; puede usarse para determinar la antigüedad del JWT
* `jti` (JWT ID): Identificador único; puede usarse para evitar que el JWT se reproduzca (permite que un token se use solo una vez)

Puedes ver una lista completa de claims registradas en el [IANA JSON Web Token Claims Registry](https://www.iana.org/assignments/jwt/jwt.xhtml#claims).

<div id="custom-claims">
  ## Claims personalizadas
</div>

Puedes definir tus propias claims personalizadas, sobre las que tú tienes control, y agregarlas a un token mediante Actions. Aquí tienes algunos ejemplos:

* Agrega la dirección de correo electrónico de un usuario a un <Tooltip tip="Access Token: Credencial de autorización, en forma de una cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=access+token">access token</Tooltip> y úsala para identificar al usuario de forma única.
* Agrega información personalizada almacenada en un perfil de usuario de Auth0 a un <Tooltip tip="token de identidad: Credencial destinada al propio cliente, en lugar de usarse para acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=ID+token">token de identidad</Tooltip>.

Mientras la Action esté implementada, las claims personalizadas que agregue aparecerán en los nuevos tokens emitidos al usar un [token de actualización](/docs/es/secure/tokens/refresh-tokens).

Para ver un ejemplo de cómo agregar claims personalizadas a un token, consulta [Sample Use Cases: Scopes and Claims](/docs/es/get-started/apis/scopes/sample-use-cases-scopes-and-claims).

<div id="public-claims">
  ### Claims públicos
</div>

Puedes crear claims personalizados de carácter público, que pueden contener información genérica como el nombre y el correo electrónico. Si creas claims públicos, debes registrarlos o usar nombres resistentes a colisiones mediante un espacio de nombres, y tomar precauciones razonables para asegurarte de que controlas el espacio de nombres que utilizas.

En el [IANA JSON Web Token Claims Registry](https://www.iana.org/assignments/jwt/jwt.xhtml#claims), puedes ver algunos ejemplos de claims públicos registrados por <Tooltip tip="OpenID: Estándar abierto de autenticación que permite a las aplicaciones verificar la identidad de los usuarios sin recopilar ni almacenar información de inicio de sesión." cta="Ver glosario" href="/docs/es/glossary?term=OpenID">OpenID</Tooltip> Connect (OIDC):

* `auth_time`
* `acr`
* `nonce`

<div id="private-claims">
  ### Claims privadas
</div>

Puedes crear claims personalizadas privadas para compartir información específica de tu aplicación. Por ejemplo, mientras que una claim pública podría contener información genérica, como el nombre y el correo electrónico, las claims privadas serían más específicas, como el ID del empleado y el nombre del departamento.

<div id="auth0-restrictions">
  ## Restricciones de Auth0
</div>

Auth0 aplica las restricciones generales a las claims personalizadas:

* el payload de las claims personalizadas tiene un tamaño máximo de 100 KB
* no se puede personalizar ni modificar un subconjunto de las claims estándar registradas de OIDC y otras claims que Auth0 usa internamente
* los tokens de acceso con una audiencia de Auth0 API, excluido el endpoint `/userinfo`, no pueden tener claims personalizadas privadas sin espacio de nombres
* solo se pueden agregar a los tokens de acceso las claims especificadas del perfil de usuario de OIDC

Para obtener más información sobre las claims personalizadas, consulta [Create Custom Claims](/docs/es/secure/tokens/json-web-tokens/create-custom-claims).
