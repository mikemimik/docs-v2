Algunas amenazas comunes de ciberseguridad incluyen:

* Ataques de relevo
* Ataques de repetición
* Ataques de falsificación de solicitud entre sitio (CSRF o XSRF)
* Ataques de fuerza bruta
* <Tooltip tip="Suspicious IP Throttling: Forma de protección contra ataques que protege su tenant frente a inicios de sesión sospechosos dirigidos a demasiadas cuentas desde una sola dirección IP." cta="Ver glosario" href="/docs/es/glossary?term=Suspicious+IP+throttling">Limitación de IP sospechosas</Tooltip>
* Ataques de credential stuffing
* Ataques de validación de listas

Auth0 tiene [funcionalidades de protección contra ataques](/docs/es/secure/attack-protection) que pueden protegerle frente a ciertos tipos de ataques.

<div id="bucket-brigade-attacks">
  ## Ataques de relevo
</div>

Un tipo de amenaza es el ataque de relevo, a veces llamado ataque de &quot;man-in-the-middle&quot;, en el que el atacante se sitúa entre dos partes. Cada parte cree que está interactuando a través de una conexión privada, pero en realidad la interacción está siendo controlada por un atacante externo. Para que este tipo de ataque tenga éxito, el atacante debe establecer autenticación mutua entre ambas partes.

Por lo general, los ataques de relevo implican que el atacante use un router WiFi para interceptar la comunicación de un usuario. El usuario se conecta al router del atacante, luego visita un sitio web e inicia sesión con sus credenciales confidenciales. El atacante guarda las credenciales de inicio de sesión de ese usuario, que después puede usar para hacerse pasar por él.

Algunos ataques de relevo también pueden modificar los datos transmitidos entre la aplicación y el servidor. Los ataques pueden producirse a partir de versiones de prueba y software preinstalado en equipos nuevos, herramientas de actualización de software y otras vulnerabilidades de software. En el peor de los casos, esto podría dar lugar a la ejecución remota de código, la inserción de puertas traseras en el sistema, la instalación de malware, etc. Esto también podría provocar el compromiso de otros activos de la red.

Para ayudar a defenderse contra este tipo de ataque, es importante usar un cifrado sólido y autenticación entre la aplicación y el servidor. Mediante el cifrado, el servidor autentica la solicitud de la aplicación presentando un certificado digital, y solo entonces puede establecerse la conexión. Por ejemplo, HTTPS usa la capacidad de capa de sockets seguros (SSL) del navegador para proteger el tráfico web. Para descifrar HTTPS, un atacante tendría que obtener las claves usadas para cifrar el tráfico de red.

Con la configuración de TLS en sus servidores, sugerimos usar las [recomendaciones de Mozilla OpSec](https://wiki.mozilla.org/Security/Server_Side_TLS), que usan TLSv1.2. Mozilla también proporciona un [SSL Configuration Generator](https://mozilla.github.io/server-side-tls/ssl-config-generator/); para usarlo, primero elija en qué plataforma se ejecuta su servidor y luego elija la configuración más moderna según lo que pueda admitir su aplicación. Las configuraciones más modernas proporcionan cifrados más robustos para ayudar a prevenir ataques.

<div id="replay-attacks">
  ## Ataques de repetición
</div>

Los ataques de repetición (o de reproducción) permiten a los atacantes:

* Obtener acceso a una red y a información a la que no habrían podido acceder fácilmente
* Completar una transacción duplicada

Se trata de ataques al protocolo de seguridad que reutilizan transmisiones de datos desde un remitente distinto hacia el sistema receptor previsto. Estos ataques engañan a los participantes haciéndoles creer que han completado correctamente la transmisión de datos.

Los ataques de repetición pueden evitarse usando tokens de sesión. Sin embargo, si estas credenciales se roban del almacenamiento local (como durante un ataque XSS), hay formas de evitar que alguien conserve un token válido para siempre:

* Establecer un tiempo de expiración corto para los tokens
* Proporcionar una forma de agregar a la DenyList los tokens que ya se hayan usado (y posiblemente incluso a los usuarios)
* Usar contraseñas de un solo uso

La especificación [JWT](/docs/es/secure/tokens/json-web-tokens) proporciona el campo `jti` como una forma de prevenir ataques de repetición. Aunque actualmente los tokens de Auth0 no devuelven el `jti`, puedes agregar tokens a la DenyList usando el `jti` para evitar que un token se use más veces de las permitidas. De este modo, estás implementando algo similar a un <Tooltip tip="Nonce: Número arbitrario emitido una vez en un protocolo de autenticación para detectar y prevenir ataques de repetición." cta="Ver glosario" href="/docs/es/glossary?term=nonce">nonce</Tooltip> (piensa en la firma del token como el nonce). Si se roba un token o se usa más veces de las permitidas, debe agregarse a la DenyList. Esto evita que un token válido se use de forma maliciosa. Una vez que el token expire, el atacante ya no podrá suplantar al usuario.

También puedes evitar los ataques de repetición usando contraseñas de un solo uso. Con Auth0, puedes usar la autenticación <Tooltip tip="Passwordless: Forma de autenticación que no se basa en una contraseña como primer factor." cta="Ver glosario" href="/docs/es/glossary?term=Passwordless">Passwordless</Tooltip>, que se basa en códigos de un solo uso y enlaces por correo electrónico en lugar de contraseñas tradicionales. Auth0 también proporciona <Tooltip tip="Autenticación multifactor (MFA): Proceso de autenticación de usuario que usa un factor además de nombre de usuario y contraseña, como un código por SMS." cta="Ver glosario" href="/docs/es/glossary?term=multi-factor+authentication">autenticación multifactor</Tooltip> (MFA), que usa contraseñas de un solo uso como segundo factor de autenticación y que puede enviarse mediante notificaciones push y mensajes de texto.

<div id="cross-site-request-forgery">
  ## Falsificación de solicitud entre sitio
</div>

Un ataque de falsificación de solicitud entre sitio (CSRF o XSRF) ocurre cuando un programa malicioso hace que el navegador web de un usuario realice una acción no deseada en un sitio de confianza en el que el usuario tiene una sesión autenticada en ese momento. Este tipo de ataque se dirige específicamente a solicitudes que cambian el estado para iniciar algún tipo de acción, en lugar de obtener datos de usuario, ya que el atacante no tiene forma de ver la respuesta de la solicitud falsificada.

Una forma de verificar las solicitudes que se envían es utilizar el protocolo <Tooltip tip="OAuth 2.0: Marco de autorización que define protocolos de autorización y flujos de trabajo." cta="Ver glosario" href="/docs/es/glossary?term=OAuth+2.0">OAuth 2.0</Tooltip> [state parameter](/docs/es/secure/attack-protection/state-parameters) para autenticar la respuesta.
