Algunas amenazas comunes de ciberseguridad incluyen:

* Ataques de relevo
* Ataques de repetición
* Ataques de falsificación de solicitud entre sitio (CSRF o XSRF)
* Ataques de fuerza bruta
* <Tooltip tip="Suspicious IP Throttling: Forma de protección contra ataques que protege a su tenant contra inicios de sesión sospechosos dirigidos a un número excesivo de cuentas desde una sola dirección IP." cta="Ver glosario" href="/docs/es/glossary?term=Suspicious+IP+throttling">Limitación de IP sospechosas</Tooltip>
* Ataques de credential stuffing
* Ataques de validación de listas

Auth0 tiene [funcionalidades de protección contra ataques](/docs/es/secure/attack-protection) que pueden protegerle contra ciertos tipos de ataques.

<div id="bucket-brigade-attacks">
  ## Ataques de relevo
</div>

Un tipo de amenaza es el ataque de relevo, a veces llamado ataque de &quot;man-in-the-middle&quot;, en el que el atacante se interpone entre dos partes. Cada parte cree que está interactuando a través de una conexión privada, pero en realidad la interacción está siendo controlada por un atacante externo. Para que este tipo de ataque tenga éxito, el atacante debe establecer autenticación mutua entre ambas partes.

Por lo general, los ataques de relevo implican que el atacante use un router WiFi para interceptar la comunicación de un usuario. El usuario se conecta al router del atacante, luego visita un sitio web e inicia sesión con sus credenciales. El atacante guarda las credenciales de inicio de sesión de ese usuario, que después puede usar para suplantarlo.

Algunos ataques de relevo también pueden modificar los datos transmitidos entre la aplicación y el servidor. Los ataques pueden producirse a partir de software de prueba en PC nuevos y software preinstalado, herramientas de actualización de software y otras vulnerabilidades de software. En el peor de los casos, esto podría dar lugar a la ejecución remota de código, la introducción de puertas traseras en el sistema, la instalación de malware, etc. Esto también podría llevar a que otros activos de la red se vieran comprometidos.

Para ayudar a defenderse de este tipo de ataque, es importante usar cifrado robusto y autenticación entre la aplicación y el servidor. Mediante el cifrado, el servidor autentica la solicitud de la aplicación presentando un certificado digital, y solo entonces se puede establecer la conexión. Por ejemplo, HTTPS usa la capacidad de la capa de sockets seguros (SSL) del navegador para ocultar el tráfico web. Para descifrar HTTPS, un atacante tendría que obtener las claves utilizadas para cifrar el tráfico de red.

Con la configuración de TLS en sus servidores, sugerimos usar las [recomendaciones de OpSec de Mozilla](https://wiki.mozilla.org/Security/Server_Side_TLS), que usan TLSv1.2. Mozilla también ofrece un [Generador de configuración SSL](https://mozilla.github.io/server-side-tls/ssl-config-generator/); para usarlo, primero elija en qué plataforma se ejecuta su servidor y luego elija la configuración más moderna según lo que pueda admitir su aplicación. Las configuraciones más modernas proporcionan algoritmos de cifrado más robustos para ayudar a prevenir ataques.

<div id="replay-attacks">
  ## Ataques de repetición
</div>

Los ataques de repetición (o de reproducción) permiten a los atacantes:

* Obtener acceso a una red y a información a la que, de otro modo, no habrían podido acceder fácilmente
* Completar una transaction duplicada

Se trata de ataques al protocolo de seguridad que reutilizan transmisiones de datos para reenviarlas desde un remitente distinto al sistema receptor previsto. Estos ataques engañan a los participantes haciéndoles creer que la transmisión de datos se ha completado correctamente.

Los ataques de repetición pueden evitarse usando tokens de sesión. Sin embargo, si estas credenciales se roban del almacenamiento local (por ejemplo, durante un ataque XSS), hay formas de evitar que alguien conserve un token válido indefinidamente:

* Establecer un tiempo de expiración breve para los tokens
* Proporcionar una forma de añadir a la DenyList los tokens que ya se hayan usado (y posiblemente incluso a los usuarios)
* Usar contraseñas de un solo uso

La especificación [JWT](/docs/es/secure/tokens/json-web-tokens) proporciona el campo `jti` como una forma de prevenir ataques de repetición. Aunque los tokens de Auth0 actualmente no devuelven `jti`, puedes añadir tokens a la DenyList usando `jti` para evitar que un token se utilice más de un número determinado de veces. De este modo, estás implementando algo similar a un <Tooltip tip="Nonce: Número arbitrario emitido una sola vez en un protocolo de autenticación para detectar y prevenir ataques de repetición." cta="Ver glosario" href="/docs/es/glossary?term=nonce">nonce</Tooltip> (piensa en la firma del token como si fuera el nonce). Si se roba un token o se usa más veces de las especificadas, debe añadirse a la DenyList. Esto evita que un token válido se use de forma maliciosa. Una vez que el token expire, el atacante ya no podrá hacerse pasar por el usuario.

También puedes evitar los ataques de repetición usando contraseñas de un solo uso. Con Auth0, puedes usar la autenticación <Tooltip tip="Passwordless: Forma de autenticación que no se basa en una contraseña como primer factor." cta="Ver glosario" href="/docs/es/glossary?term=Passwordless">Passwordless</Tooltip>, que se basa en códigos de un solo uso y enlaces por correo electrónico en lugar de contraseñas tradicionales. Auth0 también proporciona <Tooltip tip="Autenticación multifactor (MFA): Proceso de autenticación de usuario que usa un factor además del nombre de usuario y la contraseña, como un código por SMS." cta="Ver glosario" href="/docs/es/glossary?term=multi-factor+authentication">autenticación multifactor</Tooltip> (MFA), que usa contraseñas de un solo uso como segundo factor de autenticación y que pueden enviarse mediante notificaciones push y mensajes de texto.

<div id="cross-site-request-forgery">
  ## Falsificación de solicitud entre sitio
</div>

Un ataque de falsificación de solicitud entre sitio (CSRF o XSRF) ocurre cuando un programa malicioso hace que el navegador web de un usuario realice una acción no deseada en un sitio de confianza en el que el usuario está autenticado en ese momento. Este tipo de ataque se dirige específicamente a las solicitudes que cambian el estado para iniciar algún tipo de acción, en lugar de obtener datos de usuario, porque el atacante no tiene forma de ver la respuesta de la solicitud falsificada.

Una forma de verificar las solicitudes que se envían es utilizar el protocolo <Tooltip tip="OAuth 2.0: marco de autorización que define los protocolos de autorización y los flujos de trabajo." cta="Ver glosario" href="/docs/es/glossary?term=OAuth+2.0">OAuth 2.0</Tooltip> [state parameter](/docs/es/secure/attack-protection/state-parameters) para validar la respuesta.
