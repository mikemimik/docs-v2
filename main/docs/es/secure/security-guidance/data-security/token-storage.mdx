<Card title="Información general">
  Conceptos clave

  * La forma en que decida almacenar sus tokens es crucial para proteger su aplicación contra ataques maliciosos.
  * Revise los escenarios para cada tipo de aplicación.
  * Decida qué método se adapta mejor a su tecnología.
</Card>

La protección de las SPA que realizan llamadas a la API plantea consideraciones específicas. Deberá asegurarse de que los tokens y otros datos confidenciales no sean vulnerables a [cross-site scripting](https://owasp.org/www-community/attacks/xss/) (XSS) y no puedan ser leídos por código JavaScript malicioso.

Para obtener más información, consulte [JWT Handbook](https://auth0.com/resources/ebooks/jwt-handbook) y [The Ultimate Guide to Next.js Authentication with Auth0](https://auth0.com/blog/ultimate-guide-nextjs-authentication-auth0/?utm_source=twitter\&utm_medium=sc\&utm_campaign=nextjs_authn_guide).

<div id="nextjs-static-site-scenarios">
  #### Escenarios de sitios estáticos de Next.js
</div>

Cuando estás creando una aplicación Next.js, la autenticación puede ser necesaria en los siguientes casos:

1. Al acceder a una página
2. Al acceder a una ruta de API
3. Cuando tu aplicación hace una llamada a una API alojada fuera de tu aplicación Next.js en nombre del usuario

Cuando hay un servidor disponible, tu aplicación puede gestionar la interacción con Auth0 y crear una sesión, pero en este modelo no tenemos un backend. Todo el trabajo ocurre en el frontend:

1. El usuario es redirigido a Auth0.
2. Cuando el usuario inicia sesión correctamente, será redirigido de vuelta a la aplicación.
3. El lado del cliente completará el intercambio de código con Auth0 y recuperará el `id_token` y el `access_token` del usuario, que se almacenarán en memoria.

   <Frame>![Diagrama de prácticas recomendadas de almacenamiento de tokens en memoria](/docs/images/cdy7uua7fh8z/6a4aA0TH8PJQpvhkLaGSIp/e38aae00318515f2a0efa0dfce24dca2/in-memory-token-storage.png)</Frame>

<Tabs>
  <Tab title="Aplicación web tradicional">
    Si tu aplicación usa un escenario de inicio de sesión que no requiere llamadas a la API, solo se necesita un token de ID. No es necesario almacenarlo. Puedes validarlo y obtener de él los datos que necesites.

    Si tu aplicación necesita llamar a APIs en nombre del usuario, se necesitan tokens de acceso y, opcionalmente, tokens de actualización. Estos pueden almacenarse del lado del servidor o en una cookie de sesión. La cookie debe estar cifrada y tener un tamaño máximo de 4 KB. Si los datos que se van a almacenar son grandes, guardar tokens en la cookie de sesión no es una opción viable.

    Usa los siguientes tipos de flujo en estos escenarios:

    * [Flujo de código de autorización](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow)
    * [Quickstarts para aplicaciones web tradicionales](/docs/es/quickstart/webapp)
  </Tab>

  <Tab title="Aplicación nativa/móvil">
    Almacena los tokens en un almacenamiento seguro que ofrezca el sistema operativo y limita el acceso a ese almacenamiento. Por ejemplo, usa KeyStore para Android y Keychain para iOS.

    Usa los siguientes tipos de flujo en estos escenarios:

    * [Flujo de código de autorización con Proof Key for Code Exchange](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce)
    * [Guardar y renovar tokens para Swift](/docs/es/libraries/auth0-swift/auth0-swift-save-and-renew-tokens)
    * [Quickstarts para aplicaciones nativas/móviles](/docs/es/quickstart/native)
  </Tab>

  <Tab title="Aplicación web de página única">
    Recomendamos usar el [Auth0 SPA SDK](/docs/es/libraries/auth0-single-page-app-sdk) para que gestione por ti el almacenamiento de tokens, la administración de sesiones y otros detalles.

    Cuando la SPA llama solo a una API que se sirve desde un dominio que puede compartir cookies con el dominio de la SPA, no se necesitan tokens. OAuth agrega vectores de ataque adicionales sin aportar ningún valor extra y debe evitarse en favor de un enfoque tradicional basado en cookies.

    Cuando la SPA llama a varias APIs que residen en un dominio diferente, se necesitan tokens de acceso y, opcionalmente, tokens de actualización.

    * Si el backend de la SPA puede gestionar las llamadas a la API, entonces funciona de forma similar a una aplicación web tradicional que maneja los tokens del lado del servidor usando:

      * [Flujo de código de autorización](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow)
      * [Flujo de código de autorización con Proof Key for Code Exchange](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce)
    * Si el backend de la SPA no puede gestionar las llamadas a la API, entonces funciona de forma similar a una aplicación móvil que almacena tokens en el backend de la SPA, pero la SPA necesita obtener los tokens desde el backend para realizar solicitudes a la API. Debe establecerse un protocolo entre el backend y la SPA para permitir la transferencia segura del token desde el backend hasta la SPA.
    * Si tienes una SPA **sin** un servidor backend correspondiente, tu SPA debe solicitar nuevos tokens al iniciar sesión y almacenarlos en memoria sin persistencia. Para realizar llamadas a la API, tu SPA usaría entonces la copia del token en memoria.

    <Callout icon="file-lines" color="#0EA5E9" iconType="regular">
      De acuerdo con las especificaciones de OAuth2, cuando un navegador solicita un token de actualización al endpoint /token, Auth0 solo devolverá un Refresh Token si la [rotación del Refresh Token](/docs/es/secure/tokens/refresh-tokens/refresh-token-rotation) está habilitada para ese cliente.
    </Callout>

    Para más detalles, consulta [Auth0 SPA SDK en GitHub](https://github.com/auth0/auth0-spa-js).
  </Tab>
</Tabs>

<div id="browser-in-memory-scenarios">
  ### Escenarios de almacenamiento en memoria en el navegador
</div>

Auth0 recomienda almacenar los tokens en la memoria del navegador como la opción más segura. Usar [Web Workers](https://developer.mozilla.org/en-US/docs/Web/API/Web_Workers_API) para gestionar la transmisión y el almacenamiento de tokens es la mejor forma de protegerlos, ya que los Web Workers se ejecutan en un contexto global independiente del resto de la aplicación. Usa Auth0 SPA SDK, cuya opción de almacenamiento predeterminada es el almacenamiento en memoria mediante Web Workers.

Si no puedes usar Web Workers, Auth0 recomienda como alternativa usar [closures de JavaScript](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Guide/Closures#emulating_private_methods_with_closures) para emular métodos privados.

Usa Auth0 SPA SDK, cuya opción de almacenamiento predeterminada es el almacenamiento en memoria, para aprovechar tanto Web Workers como closures de JavaScript según el tipo de token.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  El método de almacenamiento en memoria en el navegador **no** ofrece persistencia entre recargas de página ni entre pestañas del navegador.
</Callout>

<div id="browser-local-storage-scenarios">
  ### Escenarios de almacenamiento local del navegador
</div>

Usar el almacenamiento local del navegador puede ser una alternativa viable a los mecanismos que requieren recuperar el <Tooltip tip="Token de acceso: credencial de autorización, en forma de una cadena opaca o JWT, usada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=access+token">token de acceso</Tooltip> desde un iframe y a la autenticación basada en cookies entre dominios cuando esto no es posible debido a restricciones del navegador (por ejemplo, ITP2).

<Warning>
  Almacenar tokens en el almacenamiento local del navegador proporciona persistencia entre recargas de página y pestañas del navegador. Sin embargo, si un atacante logra ejecutar JavaScript en la SPA mediante un ataque de cross-site scripting (XSS), puede recuperar los tokens almacenados en el almacenamiento local. Una vulnerabilidad que dé lugar a un ataque XSS exitoso puede estar tanto en el código fuente de la SPA como en cualquier código JavaScript de terceros (como Bootstrap, jQuery o Google Analytics) incluido en la SPA.
</Warning>

Para reducir los riesgos de seguridad si tu SPA usa flujos implícitos (recomendamos usar el flujo de código de autorización con PKCE en su lugar) o flujos híbridos, puedes reducir el tiempo absoluto de expiración del token. Esto reduce el impacto de un ataque XSS reflejado (pero no de uno persistente). Para reducir el tiempo de expiración, ve a **Dashboard &gt; APIs &gt; Settings &gt; Token Expiration For Browser Flows (Seconds)**.

Reduce al mínimo necesario la cantidad de código JavaScript de terceros incluido desde una fuente externa a tu dominio (como enlaces a jQuery, Bootstrap, Google Analytics, etc.). Reducir el código JS de terceros disminuye la posibilidad de una vulnerabilidad de XSS. También es más seguro realizar comprobaciones de [Subresource Integrity (SRI)](https://developer.mozilla.org/en-US/docs/Web/Security/Defenses/Subresource_Integrity) en scripts de terceros (cuando sea posible) para verificar que los recursos obtenidos se entregan sin manipulaciones inesperadas.
