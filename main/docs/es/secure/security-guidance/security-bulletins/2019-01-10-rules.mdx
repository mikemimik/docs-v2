description: Casos en los que el código personalizado de una rule puede crear vulnerabilidades en el flujo de autenticación.
title: Boletín de seguridad de Auth0 para Rules
---

**Publicado**: 10 de enero de 2019

<div id="overview">
  ## Resumen
</div>

Este boletín de seguridad abarca varios escenarios que los clientes deben evitar en el código personalizado de rule, ya que pueden crear vulnerabilidades en el flujo de autenticación. A continuación, se ofrece una descripción de cada problema y, cuando es posible, una forma alternativa de escribir la misma rule.

* El código personalizado de rule que contiene lógica condicional para aplicar la <Tooltip tip="Autenticación multifactor (MFA): proceso de autenticación de usuarios que utiliza un factor además de username y contraseña, como un código por SMS." cta="Ver glosario" href="/docs/es/glossary?term=multi-factor+authentication">autenticación multifactor</Tooltip> (MFA) puede hacer posible omitir por completo la MFA, especialmente al usar autenticación silenciosa.
* El código personalizado de rule que implementa controles de autorización basados en una subcadena específica sin la lógica adecuada puede dar lugar a privilegios elevados.
* El código personalizado de rule que envía información de diagnóstico a servicios de terceros, en lugar de usar las funciones de depuración integradas de Auth0, puede provocar la divulgación de información sensible.
* El código personalizado de rule que activa servicios de pago puede hacer posible que un adversario genere cargos no deseados.
* El código personalizado de rule que concede autorización basándose en direcciones de correo electrónico no verificadas puede permitir que los adversarios adquieran esos privilegios mediante un tipo de conexión secundario.
* El código personalizado de rule que contiene secretos codificados, como claves API, en lugar de usar el objeto de configuración global, aumenta el riesgo de exponer esos secretos.

<div id="improper-mfa-rules">
  ## Reglas de MFA mal configuradas
</div>

<div id="silent-authentication">
  ### Autenticación silenciosa
</div>

Dentro de una aplicación de una sola página (SPA), la autenticación silenciosa permite emitir nuevos <Tooltip tip="Token de acceso: Credencial de autorización, en forma de una cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=access+tokens">tokens de acceso</Tooltip> sin interacción del usuario, siempre que la sesión actual sea válida en el <Tooltip tip="Servidor de autorización: Servidor centralizado que contribuye a definir los límites del acceso de un usuario. Por ejemplo, su servidor de autorización puede controlar los datos, las tareas y las funciones disponibles para un usuario." cta="Ver glosario" href="/docs/es/glossary?term=authorization+server">servidor de autorización</Tooltip>.

Auth0 ofrece autenticación silenciosa mediante un parámetro especial que se agrega al endpoint de autorización de la siguiente manera:
`/authorize?prompt=none`

Sin embargo, si se agrega MFA al proceso de autenticación silenciosa, la interacción del usuario pasa a ser necesaria.

No debe usarse lógica condicional basada en la autenticación silenciosa para eludir MFA como solución alternativa. Las Rules como esta permiten omitir MFA por completo y no deben utilizarse:

```javascript lines
function (user, context, callback) {
  if (context.request.query && context.request.query.prompt === 'none') {
  // omitir MFA para solicitudes de token silenciosas
  return callback(null, user, context);
  }
...
}
```

<div id="silent-authentication-with-redirection-to-custom-mfa-provider">
  ### Autenticación silenciosa con redirección a un proveedor personalizado de MFA
</div>

Las Rules que determinan si se debe redirigir a un proveedor personalizado de autenticación multifactor en función de la autenticación silenciosa pueden permitir omitir la MFA en algunas circunstancias poco habituales. Por ejemplo, evite lo siguiente:

```javascript lines
function (user, context, callback) {
  if (context.request.query && context.request.query.prompt === 'none') {
  //redirigir a MFA personalizado
  context.redirect = {
    url: "https://example.com/"
  };
...
}
```

<div id="improper-verification">
  ### Verificación incorrecta
</div>

Usar Rules para eludir la autenticación multifactor bajo determinadas condiciones puede omitir MFA cuando las comprobaciones implementadas no bastan para determinar la veracidad del state deseado.

En estos casos, la raíz del problema es que las Rules nunca pueden determinar si MFA se ejecutó correctamente, ya que MFA ocurre después de la ejecución de la rule. Las Rules configuradas de esta manera (consulte los ejemplos a continuación) actualizan prematuramente el registro del usuario bajo la suposición incorrecta de que MFA se ejecutará correctamente, a la vez que se basan en ese mismo registro del usuario para omitir MFA en determinadas condiciones.

Los ejemplos a continuación ilustran comprobaciones de verificación incorrectas que pueden dar lugar a una omisión completa de la autenticación multifactor.

<div id="device-fingerprint">
  #### Huella del dispositivo
</div>

```javascript lines
function (user, context, callback) {

  var crypto = require('crypto');

  var deviceFingerPrint = getDeviceFingerPrint();
  user.app_metadata = user.app_metadata || {};

  // Verificación inadecuada
  if (user.app_metadata.lastLoginDeviceFingerPrint !== deviceFingerPrint) {
    user.app_metadata.lastLoginDeviceFingerPrint = deviceFingerPrint;
    context.multi-factor = {
      ...
    };
    ...
  }
  function getDeviceFingerPrint() {
    var shasum = crypto.createHash('sha1');
    shasum.update(context.request.userAgent);
    shasum.update(context.request.ip);
    return shasum.digest('hex');
  }
}
```

<div id="country-code">
  #### Código de país
</div>

```javascript lines
function (user, context, callback) {
  user.app_metadata = user.app_metadata || {};

  // Verificación insuficiente
  if (user.app_metadata.last_location !== context.request.geoip.country_code) {
    user.app_metadata.last_location = context.request.geoip.country_code;
    context.multi-factor = {
    ...
    };
  }
```

<div id="am-i-affected">
  ### ¿Me afecta?
</div>

Si utiliza cualquiera de los ejemplos de lógica de rule descritos anteriormente, es posible que un atacante que haya iniciado sesión correctamente con el primer factor de autenticación pueda eludir MFA en su aplicación.

<div id="mitigation-steps">
  ### Pasos de mitigación
</div>

**Si te afecta el escenario de autenticación silenciosa**, elimina la lógica condicional basada en `prompt === 'none'`. Esto activará la autenticación multifactor en cada llamada de autenticación silenciosa para comprobar el estado de la sesión.

**Si te afecta el escenario de autenticación silenciosa con redirección**, elimina la lógica condicional basada en `prompt === 'none'` y cambia a un proveedor de autenticación multifactor compatible con Auth0.

Para evitar pedirles a los usuarios que completen la MFA con demasiada frecuencia, puedes establecer el parámetro `allowRememberBrowser` en `true`, lo que permitirá a los usuarios finales marcar una casilla para que solo se les solicite la autenticación multifactor cada 30 días. Por ejemplo:

```js lines
context.multi-factor = {
    provider: 'guardian',
    allowRememberBrowser: true
  };
```

Puede que te convenga avisar a tus usuarios finales de que marquen la casilla para evitar que se les solicite con demasiada frecuencia.

**Si te ves afectado por reglas que utilizan una verificación inadecuada para omitir MFA**, asegúrate de aplicar comprobaciones de verificación fiables, eliminar por completo las excepciones de MFA o usar la opción de configuración `allowRememberBrowser` descrita anteriormente.

<div id="will-this-update-impact-my-users">
  ### ¿Esta actualización afectará a mis usuarios?
</div>

Según cómo ajustes tus reglas o la configuración, esta actualización puede afectar a tus usuarios al pedirles MFA con más frecuencia de la habitual.

<div id="improper-substring-verification">
  ## Verificación incorrecta de subcadenas
</div>

Si tienes lógica de `rule` que requiere un control de acceso basado en una coincidencia exacta de una cadena concreta, como un dominio de correo electrónico, pero solo estás comprobando una subcadena en lugar de una coincidencia exacta de la cadena, es posible que tu lógica no funcione como esperas. Por ejemplo:

`if( _.findIndex(connection.options.domain_aliases,function(d){ return user.email.indexOf(d) >= 0;`

La lógica anterior devolvería `true` con correos electrónicos como estos:

* `user.domain.com@not-domain.com`
* `"user@domain.com"@not-domain.com` (comillas incluidas)

<div id="am-i-affected">
  ### ¿Me afecta?
</div>

Solo se ven afectados los clientes que usan lógica condicional en las reglas, como se describió anteriormente.

<div id="mitigation-steps">
  ### Pasos de mitigación
</div>

Utiliza lógica como esta:

`const emailSplit = user.email.split('@'); const userEmailDomain = emailSplit[emailSplit.length - 1].toLowerCase();`

Consulta la [Check Domains Against Connection Aliases rule template](https://github.com/auth0/rules/blob/master/src/rules/check-domains-against-connection-aliases.js) para obtener más información. Como alternativa, en la sección **Rules** del <Tooltip tip="Auth0 Dashboard: el producto principal de Auth0 para configurar tus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip>, consulta la plantilla de Rule llamada **Check if user email domain matches configured domain**.

<div id="will-this-update-impact-my-users">
  ### ¿Esta actualización afectará a mis usuarios?
</div>

Es posible. El cambio recomendado en las reglas con lógica basada en comprobaciones de subcadenas puede afectar a los usuarios finales, aunque usted es quien mejor puede juzgar la intención de sus reglas y si es probable que alguno de los cambios recomendados afecte a sus usuarios finales.

<div id="debugging-using-external-services">
  ## Depuración con servicios externos
</div>

Si tiene una rule cuya lógica envía el objeto `context` de Auth0 a un servicio externo, está exponiendo elementos como `id_token` o `access_token` asociados a la solicitud. Por ejemplo:

```js lines
request.post({
    url: 'http://requestbin.fullcontact.com/YourBinUrl',
    json: {
      user: user,
      context: context,
    },
    timeout: 15000
  }, function(err, response, body){
    if (err) return callback(err);
    return callback(null, user, context);
  });
```

<div id="am-i-affected">
  ### ¿Me afecta?
</div>

Solo se ven afectados los clientes con lógica de rule, como se describe arriba.

<div id="mitigation-steps">
  ### Pasos de mitigación
</div>

Debe modificar su regla para que ya no envíe el objeto `context` completo a Requestbin, ya que esto puede exponer cualquier `id_token` o `access_token` asociado a cada solicitud. En su lugar, debe enviar un subconjunto de atributos del objeto `context` que sea menos sensible.

Consulte la [plantilla de regla de Requestbin](https://github.com/auth0/rules/blob/master/src/rules/requestbin.js) para obtener más información. Como alternativa, en la sección **Reglas** del Auth0 Dashboard, consulte la plantilla de regla llamada **Volcar variables de regla a RequestBin**.

Auth0 también ofrece métodos integrados para depurar reglas sin enviar información a servicios externos.

### ¿Afectará esta actualización a mis usuarios?

No. La rule afectada se usaba normalmente con fines de depuración. La modificación no debería afectar a los usuarios finales.

<div id="rules-using-a-paid-service">
  ## Rules que usan un servicio de pago
</div>

Si tiene lógica de rule que involucra un servicio de pago, como el envío de mensajes SMS a través de Twillio, como parte del proceso de autenticación, puede quedar expuesto a un aumento de costos, ya que un adversario con un nombre de usuario y una contraseña válidos podría trigger una llamada al servicio y generar costos sin tener que completar todo el proceso de autenticación especificado por la rule. Por ejemplo:

```javascript lines
//Envía SMS al usuario a través de Twilio
  function notifyUser(done) {
    const request = require('request');
    const twilioAccount = '{yourTwilioAccount}';
    const twilioAuthToken = '{yourTwilioAuthToken}';

    request.post({
      url: 'https://api.twilio.com/2010-04-01/Accounts/' + twilioAccount + '/Messages.json',
      auth: {
        'user': twilioAccount,
        'pass': twilioAuthToken,
      },
      form: {
        body: 'You\'ve logged in from a different device or location.',
        to: user.phone,
        from: '+18668888888'
      }
    }, function (error, response, body) {
      if (error) return done(error);
      if (response.statusCode !== 201) return done(new Error(response.statusCode));
      return done(null);
    });
  }
```

<div id="am-i-affected">
  ### ¿Me afecta?
</div>

Solo se ven afectados los clientes con lógica de rule, como se describe arriba, y cuya lógica de rule puede activarse por cualquier usuario no confiable.

<div id="mitigation-steps">
  ### Pasos de mitigación
</div>

Para mitigar este riesgo, considere una o varias de las siguientes medidas:

* Deshabilite los registros públicos, si no son necesarios, para reducir la cantidad de usuarios que pueden registrarse y activar llamadas a servicios de pago.
* Mitigue el riesgo de robo de credenciales para evitar la toma de control de cuentas por parte de atacantes que podrían usar cuentas comprometidas para activar llamadas a servicios de pago.
* Asegúrese de que sus usuarios tengan contraseñas seguras al usar Database Connections.
* Asegúrese de que sus usuarios utilicen autenticación multifactor.
* Asegúrese de que la rule solo se active para un subconjunto autorizado de usuarios o en otras condiciones apropiadas. Por ejemplo, puede que desee agregar lógica que verifique si un usuario tiene un dominio de correo electrónico, un rol/grupo o un nivel de suscripción determinados antes de activar la llamada al servicio de pago.

<div id="will-this-update-impact-my-users">
  ### ¿Esta actualización afectará a mis usuarios?
</div>

El impacto en los usuarios finales dependerá de cuál de las opciones descritas anteriormente elija para mitigar el problema.

<div id="authorization-based-on-unverified-email-address">
  ## Autorización basada en una dirección de correo electrónico no verificada
</div>

Si tienes lógica de rule que requiere control de acceso en función de una dirección de correo electrónico concreta o de un dominio de correo electrónico, pero no compruebas si ese usuario ha verificado su correo electrónico, un atacante podría obtener privilegios adicionales al usar un segundo tipo de conexión. Por ejemplo:

```javascript lines
var userHasAccess = allowList.some(function (email) {
    return email === user.email;
  });
```

La lógica anterior devolvería `true` si un adversario pudiera crear una cuenta con un tipo de conexión distinto (como social) usando una dirección de correo electrónico incluida en la lista de permitidos. Esto ocurre porque una misma dirección de correo electrónico puede existir en distintos tipos de conexión.

<div id="am-i-affected">
  ### ¿Me afecta?
</div>

Solo se ven afectados los clientes que usan lógica condicional en las reglas, como se describió anteriormente, y permiten varios tipos de conexión.

<div id="mitigation-steps">
  ### Pasos de mitigación
</div>

Siempre que conceda autorización en función de direcciones de correo electrónico, comience la rule con la siguiente lógica:

```javascript lines
function (user, context, callback) {
  // El acceso solo debe concederse a usuarios verificados.
  if (!user.email || !user.email_verified) {
    return callback(new UnauthorizedError('Access denied.'));
  }
```

A veces, incluso un correo electrónico verificado puede no ser una validación suficiente para la autorización que desea realizar. Para obtener más información sobre esos casos, consulte Email Verified Usage.

### ¿Afectará esta actualización a mis usuarios?

Esto solo afectará a los usuarios existentes que no hayan verificado sus direcciones de correo electrónico.

<div id="plaintext-secrets-within-rule-code">
  ## Secretos en texto plano dentro del código de rule
</div>

Si especifica información confidencial, como claves de API, en el código de una rule, debería configurar esos valores en la configuración de la rule. Por ejemplo:

`const myApiKey = 'abcdefghijklmnopqrstuvwxyz';`

Al estar presentes en el código de la rule, estos valores confidenciales permanecerán sin cifrar en nuestros sistemas y correrán el riesgo de quedar expuestos.

<div id="am-i-affected">
  ### ¿Me afecta?
</div>

Si tus reglas contienen valores sensibles en el propio código de la rule, te afecta.

<div id="mitigation-steps">
  ### Pasos de mitigación
</div>

Coloca los valores confidenciales en el área de **Settings** de la sección principal de **Rules** del Auth0 Dashboard y luego accede a ellos desde tu rule de la siguiente manera:

`const myApiKey = configuration.myApiKey;`

Esto garantiza que todos los valores confidenciales queden cifrados dentro de los sistemas de Auth0, lo que reduce el riesgo de exposición.

<div id="will-this-update-impact-my-users">
  ### ¿Esta actualización afectará a mis usuarios?
</div>

Esto no afectará a tus usuarios.
