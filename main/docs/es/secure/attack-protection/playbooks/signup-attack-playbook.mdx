La forma más sencilla de que un usuario malicioso acceda a los privilegios, servicios e información que hay detrás del inicio de sesión es crear, desde el primer día, cuentas títere bajo su control.

Luego, los atacantes pueden usar estas cuentas con diversos fines ilícitos, cuyos objetivos dependen del contexto empresarial concreto. Entre los más habituales se incluyen:

* **Beneficio económico**: Consiste en crear cuentas fraudulentas para lograr objetivos como presentar reclamaciones falsas (por ejemplo, registrarse en una cuenta de seguro en línea únicamente para presentar una reclamación fraudulenta) o aprovechar beneficios e incentivos para nuevos clientes. La creación masiva de cuentas también es un primer paso para lanzar un ataque de SMS pumping.
* **Maduración de cuentas**: Los atacantes pueden crear una gran cantidad de cuentas y dejar que maduren con el tiempo, para que parezcan más legítimas de cara a ataques futuros más sofisticados.
* **Interrupción del servicio**: Los intentos de registro de gran volumen pueden saturar los sistemas de una organización, lo que podría provocar una denegación de servicio al superar los [límites de frecuencia](/docs/es/troubleshoot/customer-support/operational-policies/rate-limit-policy) globales y dificultar que los usuarios legítimos accedan a los servicios.
* **Enumeración de nombres de usuario ([T1087](https://attack.mitre.org/techniques/T1087/))**: Los atacantes pueden intentar registrarse rápidamente en una gran cantidad de cuentas para determinar si determinados nombres de usuario ya existen en un sistema. Pueden usar esta información para ataques dirigidos, como credential stuffing o phishing.

<div id="detect-signup-attacks">
  ## Detectar ataques de registro
</div>

Independientemente del objetivo del atacante, los ataques de registro dejan un rastro de eventos de log específicos que son fundamentales para su detección y análisis.

Usa [**Security Center**](https://manage.auth0.com/#/security/center/overview) para supervisar posibles ataques contra tu tenant.
Puedes usar la [biblioteca de código abierto de Auth0 para reglas de detección](https://github.com/auth0/auth0-customer-detections) para crear y aportar tus propias reglas de detección; por ejemplo, las reglas de [fraude de registro por dominios sospechosos](https://github.com/auth0/auth0-customer-detections/blob/main/detections/risk_of_signup_fraud_by_disposable_domains.yml) y [fraude de registro por volúmenes elevados](https://github.com/auth0/auth0-customer-detections/blob/main/detections/risk_of_signup_fraud_by_volume.yml).

Los indicadores clave incluyen tipos de eventos de log como `fs`, `ss` y `signup_pwd_leak` en los [registros del tenant de Auth0](/docs/es/deploy-monitor/logs). Durante la fase de reconocimiento, podrías observar requests HTTP `POST` repetidas dirigidas al [endpoint de registro](/docs/es/api/authentication/signup/create-a-new-user) de tu tenant. Esta actividad suele indicar intentos iniciales de un atacante por sondear tu sistema en busca de vulnerabilidades o poner a prueba la eficacia de sus procesos automatizados de registro.

Para realizar análisis avanzados o correlacionar actividades de inicio de sesión con aplicaciones fuera de Auth0, habilita la [transmisión de logs](/docs/es/customize/log-streams).

<div id="mitigation-strategies">
  ## Estrategias de mitigación
</div>

Para mitigar los ataques de registro, recomendamos combinar medidas proactivas con detección y respuesta en tiempo real mediante un enfoque multicapa. Revisa las siguientes estrategias para proteger tu sistema:

* **[Suspicious IP Throttling](/docs/es/secure/attack-protection/suspicious-ip-throttling)**: Implementa umbrales de registro estrictos para direcciones IP sospechosas a fin de evitar intentos masivos de registro desde una única fuente.
* **[Bot Detection](/docs/es/secure/attack-protection/bot-detection)**: Emplea mecanismos sólidos de detección de bots para identificar y bloquear intentos de registro automatizados; para ello, habilita Bot Detection de Auth0 y configura tu Bot Detection Level en High.
* **Requisitos agresivos de CAPTCHA**: Para una respuesta más agresiva a costa de una mayor fricción, configura CAPTCHA para que se muestre siempre durante el proceso de registro. Aunque esto supone una pequeña molestia para los usuarios legítimos, puede dificultar la actividad de los bots.
* **[Autenticación passwordless](/docs/es/authenticate/passwordless#passwordless-authentication)**: Usa métodos de autenticación passwordless. Esto obliga a los atacantes a mantener acceso persistente al identificador principal (por ejemplo, correo electrónico o número de teléfono) utilizado durante el registro, lo que dificulta la creación y el control de numerosas cuentas fraudulentas.
* **Factores de autenticación mejorados**: Usa [WebAuthn](/docs/es/authenticate/login/auth0-universal-login/passwordless-login/webauthn-device-biometrics) como factor de autenticación para ofrecer resistencia al phishing y frenar por completo objetivos de los atacantes, como el fraude de SMS pumping.
* **[ACL del tenant (ACL)](/docs/es/secure/tenant-access-control-list)**: Configura [ACL del tenant](/docs/es/secure/tenant-access-control-list) para bloquear tráfico malicioso.
* **[Triggers de registro previo del usuario](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/pre-user-registration-trigger)**: Usa triggers de registro previo del usuario para aplicar medidas de verificación de identidad. Esto puede implicar pasos sencillos, como exigir la validación del correo electrónico antes de activar la cuenta, o procesos de verificación más complejos, según tu perfil de riesgo.
* **Forzar la verificación en el registro**: Configura la [verificación de correo electrónico en el registro](/docs/es/manage-users/user-accounts/verify-emails) para exigir que los usuarios verifiquen sus correos electrónicos antes de iniciar sesión, idealmente mediante una contraseña de un solo uso (OTP).
* **Reglas de Web Application Firewall (WAF)**: Si estás usando un [dominio personalizado detrás de un proxy inverso](/docs/es/customize/custom-domains/self-managed-certificates#configure-reverse-proxy), configura reglas de WAF para bloquear rangos de IP maliciosas conocidos, detectar patrones de solicitud sospechosos y mitigar vectores de ataque comunes dirigidos a tu endpoint de registro.
* **Deshabilitar el registro (la medida más extrema)**: En escenarios de ataque graves o persistentes, o en aplicaciones en las que el autorregistro de usuarios no sea esencial, la medida más extrema es deshabilitar temporal o permanentemente el registro de nuevos usuarios.
* **Eliminar o bloquear cuentas fraudulentas**: Después de identificar cuentas fraudulentas, puedes usar la <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip> de Auth0 para [eliminar](/docs/es/api/management/v2/users/delete-users-by-id) o [bloquear](/docs/es/api/management/v2/users/patch-users-by-id) cuentas fraudulentas. Para operaciones masivas, ten en cuenta los [límites de frecuencia de la Management API](/docs/es/troubleshoot/customer-support/operational-policies/rate-limit-policy) y [contacta con soporte](https://support.auth0.com/) si existe el riesgo de superar tu límite de frecuencia.

Para obtener más información sobre cómo implementar estrategias de detección y respuesta, lee [Cómo detectar el fraude de registro: 3 formas de usar los registros de Auth0 para proteger tu negocio](https://auth0.com/blog/detecting-signup-fraud-3-ways-to-use-auth0-logs-to-protect-your-business/).
