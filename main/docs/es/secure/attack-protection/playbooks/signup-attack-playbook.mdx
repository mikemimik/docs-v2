La forma más fácil de que un usuario malicioso acceda a los privilegios, servicios e información que hay detrás del inicio de sesión es crear, desde el primer día, cuentas falsas bajo su control.

Luego, los atacantes pueden usar estas cuentas para diversos fines ilícitos, cuyos objetivos dependen del contexto específico de la empresa. Entre los objetivos más comunes se incluyen:

* **Ganancia financiera**: Consiste en crear cuentas fraudulentas para lograr objetivos como presentar reclamaciones falsas (por ejemplo, registrarse en una cuenta de seguro en línea únicamente para presentar una reclamación fraudulenta) o explotar beneficios e incentivos para nuevos clientes. La creación masiva de cuentas también es un primer paso para lanzar un ataque de SMS pumping.
* **Antigüedad de las cuentas**: Los atacantes pueden crear una gran cantidad de cuentas y dejarlas envejecer con el tiempo, haciendo que parezcan más legítimas para futuros ataques más sofisticados.
* **Interrupción del servicio**: Los intentos de registro en gran volumen pueden saturar los sistemas de una organización, lo que puede provocar una denegación de servicio al superar los [límites de frecuencia](/docs/es/troubleshoot/customer-support/operational-policies/rate-limit-policy) globales y dificultar el acceso de los usuarios legítimos a los servicios.
* **Enumeración de nombres de usuario ([T1087](https://attack.mitre.org/techniques/T1087/))**: Los atacantes pueden intentar registrar rápidamente una gran cantidad de cuentas para determinar si ciertos nombres de usuario ya existen en un sistema. Pueden usar esta información para ataques dirigidos como credential stuffing o phishing.

<div id="detect-signup-attacks">
  ## Detectar ataques de registro
</div>

Independientemente del objetivo del atacante, los ataques de registro dejan un rastro de eventos de log específicos que resultan cruciales para su detección y análisis.

Usa [**Security Center**](https://manage.auth0.com/#/security/center/overview) para supervisar posibles ataques contra tu tenant.
Puedes usar la [biblioteca de código abierto de reglas de detección](https://github.com/auth0/auth0-customer-detections) de Auth0 para crear y aportar tus propias reglas de detección; por ejemplo, las reglas de [fraude de registro por dominios sospechosos](https://github.com/auth0/auth0-customer-detections/blob/main/detections/risk_of_signup_fraud_by_disposable_domains.yml) y [fraude de registro por volúmenes elevados](https://github.com/auth0/auth0-customer-detections/blob/main/detections/risk_of_signup_fraud_by_volume.yml).

Entre los indicadores clave se incluyen tipos de evento de log como `fs`, `ss` y `signup_pwd_leak` en los [registros del tenant de Auth0](/docs/es/deploy-monitor/logs). Durante la fase de reconocimiento, es posible que observes solicitudes HTTP `POST` repetidas dirigidas al [endpoint de Signup](/docs/es/api/authentication/signup/create-a-new-user) de tu tenant. Esta actividad suele indicar los intentos iniciales de un atacante de sondear tu sistema en busca de vulnerabilidades o de comprobar la eficacia de sus procesos automatizados de registro.

Para un análisis avanzado o para correlacionar actividades de inicio de sesión con aplicaciones fuera de Auth0, habilita la [transmisión de logs](/docs/es/customize/log-streams).

<div id="mitigation-strategies">
  ## Estrategias de mitigación
</div>

Para mitigar los ataques de registro, recomendamos combinar medidas proactivas con detección y respuesta en tiempo real mediante un enfoque multicapa. Revise las siguientes estrategias para proteger su sistema:

* **[Suspicious IP Throttling](/docs/es/secure/attack-protection/suspicious-ip-throttling)**: Implemente umbrales agresivos para el registro desde direcciones IP sospechosas a fin de evitar intentos de registro de gran volumen desde una sola fuente.
* **[Bot Detection](/docs/es/secure/attack-protection/bot-detection)**: Utilice mecanismos sólidos de detección de bots para identificar y bloquear intentos automatizados de registro habilitando Bot Detection de Auth0 y configurando el nivel de detección de bots en High.
* **Requisitos de CAPTCHA más estrictos**: Para una respuesta más agresiva, aunque a costa de una mayor fricción, configure CAPTCHA para que se muestre siempre durante el proceso de registro. Aunque esto introduce una ligera incomodidad para los usuarios legítimos, puede dificultar la actividad de los bots.
* **[Autenticación passwordless](/docs/es/authenticate/passwordless#passwordless-authentication)**: Use métodos de autenticación passwordless. Esto obliga a los atacantes a mantener acceso persistente al identificador principal (por ejemplo, el correo electrónico o el número de teléfono) utilizado durante el registro, lo que dificulta la creación y el control de numerosas cuentas fraudulentas.
* **Factores de autenticación mejorados**: Use [WebAuthn](/docs/es/authenticate/login/auth0-universal-login/passwordless-login/webauthn-device-biometrics) como factor de autenticación para ofrecer resistencia al phishing y bloquear por completo objetivos de los atacantes, como el fraude de SMS pumping.
* **[ACL del tenant](/docs/es/secure/tenant-access-control-list)**: Configure las [ACL del tenant](/docs/es/secure/tenant-access-control-list) para bloquear el tráfico malicioso.
* **[Triggers de registro previo del usuario](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/pre-user-registration-trigger)**: Use triggers de registro previo del usuario para aplicar medidas de verificación de identidad. Esto puede incluir pasos sencillos, como exigir la validación del correo electrónico antes de activar la cuenta, o procesos de verificación más complejos, según su perfil de riesgo.
* **Forzar la verificación en el registro**: Configure la [verificación de correo electrónico en el registro](/docs/es/manage-users/user-accounts/verify-emails) para exigir que los usuarios verifiquen sus direcciones de correo electrónico antes de iniciar sesión, idealmente mediante una contraseña de un solo uso (OTP).
* **Reglas de Web Application Firewall (WAF)**: Si está usando un [dominio personalizado detrás de un proxy inverso](/docs/es/customize/custom-domains/self-managed-certificates#configure-reverse-proxy), configure reglas de WAF para bloquear rangos de IP maliciosos conocidos, detectar patrones de solicitudes sospechosos y mitigar vectores de ataque comunes dirigidos a su endpoint de Signup.
* **Deshabilitar el registro (la medida más extrema)**: En escenarios de ataques graves o persistentes, o para aplicaciones en las que el autorregistro de usuarios no sea esencial, la medida más extrema es deshabilitar temporal o permanentemente el registro de nuevos usuarios.
* **Eliminar o bloquear cuentas fraudulentas**: Después de identificar cuentas fraudulentas, puede usar la <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip> de Auth0 para [eliminar](/docs/es/api/management/v2/users/delete-users-by-id) o [bloquear](/docs/es/api/management/v2/users/patch-users-by-id) cuentas fraudulentas. Para operaciones masivas, tenga en cuenta los [límites de frecuencia de Management API](/docs/es/troubleshoot/customer-support/operational-policies/rate-limit-policy) y [contacte con soporte](https://support.auth0.com/) si existe el riesgo de superar su límite de frecuencia.

Para obtener más información sobre cómo implementar estrategias de detección y respuesta, lea [Detecting Signup Fraud: 3 Ways to Use Auth0 Logs to Protect Your Business](https://auth0.com/blog/detecting-signup-fraud-3-ways-to-use-auth0-logs-to-protect-your-business/).
