<Card title="Antes de empezar">
  Debe habilitar [Bot Detection](/docs/es/secure/attack-protection/bot-detection) y configurar un [proveedor de CAPTCHA](/docs/es/secure/attack-protection/bot-detection/configure-captcha).
</Card>

La [funcionalidad de monitoreo de Bot Detection](/docs/es/secure/security-center#bot-detection) de Auth0 es un sistema de alerta temprana para detectar botnets y ataques. A continuación, se ofrece una guía para identificar bots que intentan iniciar sesión en su tenant.

<div id="find-log-events-of-interest">
  #### Encontrar eventos de log de interés
</div>

Al definir una respuesta, primero revisa los mensajes de registro del posible ataque. Para un análisis avanzado, habilita la [transmisión de logs](/docs/es/customize/log-streams) y conéctala a la herramienta externa que prefieras.

Los siguientes [tipos de eventos de log](/docs/es/deploy-monitor/logs/log-event-type-codes) son relevantes al investigar un aumento de la actividad de bots.

| Tipo de evento de log | Descripción                                                                                                                                                                                                              |
| --------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `pla`                 | Se genera antes del inicio de sesión y supervisa la detección de bots, incluso si la detección de bots solo está en modo de monitoreo y no usa CAPTCHA para identificarlos.                                              |
| `fu`                  | Eventos de inicio de sesión fallido de usuarios debido a un nombre de usuario no válido, lo que puede indicar intentos de enumeración de nombres de usuario o de toma de control de cuentas.                             |
| `fp`                  | Eventos de inicio de sesión fallido de usuarios debido a una contraseña no válida, lo que puede indicar intentos de ataques de credential stuffing.                                                                      |
| `pwd_leak`            | Eventos de intentos de inicio de sesión con una contraseña filtrada, lo que puede indicar intentos de ataques de credential stuffing.                                                                                    |
| `limit_wc`            | Eventos de bloqueo de IP por &gt;10 intentos de inicio de sesión fallidos en una sola cuenta, lo que indica que es probable que la dirección IP pertenezca a un bot.                                                     |
| `limit_sul`           | Eventos de bloqueo de usuarios por &gt;20 intentos de inicio de sesión por minuto desde la misma dirección IP, lo que indica una probable actividad de bots.                                                             |
| `limit_mu`            | Eventos de bloqueo de IP por &gt;100 intentos de inicio de sesión fallidos o &gt;50 intentos de registro desde la misma dirección IP, lo que indica una probable actividad de bots.                                      |
| `fcoa`                | Eventos fallidos de autenticación de origen cruzado, lo que indica que los atacantes usan automatización para tomar el control de cuentas.                                                                               |
| `scoa`                | Eventos exitosos de autenticación de origen cruzado, lo que indica que los atacantes usan automatización para tomar el control de cuentas cuando provienen de un pequeño número de direcciones IP en múltiples usuarios. |

<div id="attack-response">
  #### Respuesta ante ataques
</div>

Aunque establecer el nivel de <Tooltip tip="Bot Detection: Forma de protección contra ataques en la que Auth0 bloquea el tráfico sospechoso de bots habilitando un CAPTCHA durante el proceso de inicio de sesión." cta="Ver glosario" href="/docs/es/glossary?term=Bot+Detection">Bot Detection</Tooltip> en Alto mitiga el ataque de inmediato, una estrategia integral equilibra la tolerancia al riesgo y las capacidades técnicas de su empresa con la experiencia que tendrán sus usuarios al iniciar sesión. Al responder, tenga en cuenta dos factores principales:

* **Fricción para el usuario**: evalúe el impacto de las medidas de mitigación (p. ej., la frecuencia con que se muestra el CAPTCHA) en la experiencia del usuario.
* **Capacidad técnica**: evalúe su capacidad para implementar el bloqueo de IP, reglas de WAF y la aplicación de <Tooltip tip="Autenticación multifactor (MFA): Proceso de autenticación de usuario que utiliza un factor además de nombre de usuario y contraseña, como un código por SMS." cta="Ver glosario" href="/docs/es/glossary?term=MFA">MFA</Tooltip>.

Auth0 recomienda un enfoque de seguridad por capas que combine varias técnicas de mitigación para lograr una protección óptima.

<div id="mitigation-strategies">
  #### Estrategias de mitigación
</div>

Para una protección óptima contra ataques, considere las siguientes estrategias:

* Active CAPTCHA para uno o más flujos y aumente la frecuencia de CAPTCHA según sea necesario, pero recuerde que CAPTCHA es un elemento disuasorio, no una solución.
* Cambie su proveedor de CAPTCHA si los atacantes logran eludir el CAPTCHA actual, o considere migrar a Auth Challenge de Auth0 o a otro [proveedor compatible](/docs/es/secure/attack-protection/bot-detection/configure-captcha).
* Si sospecha una campaña de fraude en el registro, impida temporalmente que nuevos usuarios se registren en su aplicación a través de endpoints públicos sin autenticación.
* Modifique las reglas del firewall de su aplicación web con un proveedor perimetral o use [listas de control de acceso del tenant](/docs/es/secure/tenant-access-control-list) para bloquear IP abusivas, números de sistema autónomo, ubicaciones geográficas, [clientes TLS](/docs/es/customize/custom-domains/self-managed-certificates/tls-ssl) o elementos de encabezados HTTP, como cadenas `user-agent`, y considere implementar un [proxy inverso.](/docs/es/customize/custom-domains/self-managed-certificates#configure-reverse-proxy)
* Endurezca los umbrales de [Brute Force](/docs/es/secure/attack-protection/brute-force-protection) y [IP sospechosa](/docs/es/secure/attack-protection/suspicious-ip-throttling) para reducir los límites de conexiones permitidas y mitigar ataques de fuerza bruta. Para obtener más información sobre los ataques de fuerza bruta, lea el [Playbook de Brute Force](/docs/es/secure/attack-protection/playbooks/brute-force-protection-playbook).
* Deshabilite los endpoints no utilizados modificando la configuración de [Cross-Origin Authentication](/docs/es/authenticate/login/cross-origin-authentication). Si sospecha ataques con contraseñas comprometidas, lea el [Playbook de Breached Password](/docs/es/secure/attack-protection/playbooks/breached-password-playbook).
* Exija [step-up MFA](/docs/es/secure/multi-factor-authentication/step-up-authentication) para cuentas comprometidas, incluso hasta requerir MFA para cuentas potencialmente comprometidas.
* Migre a opciones de MFA más seguras para mitigar ataques de SMS pumping o fraude de peaje, reemplazando MFA basada en SMS o voz por OTP o Webauthn.
