<Warning>
  Debido a que el flujo de contraseña del propietario del recurso (ROP) requiere que la aplicación gestione la contraseña del usuario, no debe ser utilizado por clientes de terceros.
</Warning>

Aunque no lo recomendamos, las aplicaciones de alta confianza pueden llamar a APIs del lado del servidor mediante el [flujo de contraseña del propietario del recurso](/docs/es/get-started/authentication-and-authorization-flow/resource-owner-password-flow) (a veces llamado concesión de contraseña de <Tooltip tip="Resource Owner: Entidad (como un usuario o una aplicación) capaz de conceder acceso a un recurso protegido." cta="Ver glosario" href="/docs/es/glossary?term=Resource+Owner">Resource Owner</Tooltip> o ROPG), que pide a los usuarios que proporcionen sus credenciales (nombre de usuario y contraseña), normalmente mediante un formulario interactivo. Si la [protección contra ataques de fuerza bruta está habilitada](/docs/es/secure/attack-protection/brute-force-protection), cuando Auth0 valida las credenciales, también puede [verificar si hay ataques](/docs/es/secure/attack-protection) y realizar las acciones adecuadas si detecta alguno.

Desafortunadamente, al usar este flujo con <Tooltip tip="Protección contra ataques de fuerza bruta: Forma de protección contra ataques que protege contra ataques de fuerza bruta que se producen desde una sola dirección IP y se dirigen a una sola cuenta de usuario." cta="Ver glosario" href="/docs/es/glossary?term=brute-force+protection">protección contra ataques de fuerza bruta</Tooltip>, es posible que algunas funcionalidades de <Tooltip tip="Protección contra ataques de fuerza bruta: Forma de protección contra ataques que protege contra ataques de fuerza bruta que se producen desde una sola dirección IP y se dirigen a una sola cuenta de usuario." cta="Ver glosario" href="/docs/es/glossary?term=attack+protection">protección contra ataques</Tooltip> no funcionen correctamente. Sin embargo, se pueden evitar algunos problemas habituales.

<div id="attack-protection-and-server-side-apis">
  ## Protección contra ataques y API del lado del servidor
</div>

La protección contra ataques de fuerza bruta y la <Tooltip tip="Suspicious IP Throttling: Forma de protección contra ataques que protege a tu tenant frente a inicios de sesión sospechosos dirigidos a demasiadas cuentas desde una sola dirección IP." cta="Ver glosario" href="/docs/es/glossary?term=suspicious+IP+throttling">limitación de IP sospechosa</Tooltip> dependen de la dirección IP del usuario. Al llamar a una API desde tu servidor, Auth0 toma la dirección IP de tu servidor como si fuera la del usuario y la usa como dato de entrada para la protección contra ataques de fuerza bruta y la limitación de IP sospechosa. Esto podría provocar falsos positivos y hacer que la protección contra ataques bloquee a usuarios o active advertencias para solicitudes legítimas.

Para evitarlo, envía a Auth0 la dirección IP del usuario junto con sus credenciales y configura tu aplicación para que confíe en esa dirección IP.

<Warning>
  Por motivos de seguridad, solo puedes configurar de esta manera aplicaciones autenticadas (como aquellas con autenticación basada en un secreto de cliente). Las aplicaciones autenticadas solo deben usarse desde recursos protegidos, que normalmente están del lado del servidor. No las uses desde aplicaciones nativas ni aplicaciones de página única (SPA), ya que no pueden almacenar secretos.
</Warning>

<div id="configure-your-application-to-trust-the-ip-address">
  ## Configura tu aplicación para confiar en la dirección IP
</div>

Registra una [aplicación web tradicional](/docs/es/get-started/auth0-overview/create-applications/regular-web-apps) o una [aplicación de máquina a máquina](/docs/es/get-started/auth0-overview/create-applications/machine-to-machine-apps). Mientras configuras la aplicación:

1. En **Credentials**, selecciona un **Authentication Method** distinto de `None`.
2. En **Settings &gt; Advanced Settings**, localiza la pestaña **OAuth** y habilita **Trust Token Endpoint IP Header**. Esto establecerá el encabezado `auth0-forwarded-for` como origen de confianza para la dirección IP del usuario en la protección contra ataques de fuerza bruta. Esta configuración no estará disponible para aplicaciones sin autenticación.

<div id="send-the-users-ip-address-from-your-server">
  ## Envía la dirección IP del usuario desde tu servidor
</div>

1. Cuando solicites tokens mediante el [flujo de contraseña del propietario del recurso](/docs/es/get-started/authentication-and-authorization-flow/resource-owner-password-flow/call-your-api-using-resource-owner-password-flow), incluye un encabezado `auth0-forwarded-for` que contenga el valor de la dirección IP del usuario. Asegúrate de que la dirección IP que proporcionas realmente pertenezca a tu usuario.

   <Warning>
     Confiar en encabezados como `auth0-forwarded-for` (o, en general, en datos de aplicaciones) como fuentes de la dirección IP del usuario puede ser arriesgado. Como este encabezado es fácil de falsificar y permite omitir la validación de la protección contra ataques, hazlo solo si sabes que puedes confiar en ese encabezado.
   </Warning>
2. Especifica listas de permitidos de IP que deban ignorarse al activar la protección contra ataques de fuerza bruta y la limitación de IP sospechosas.

<Card title="Inclusión en lista de permitidos con protección contra ataques de fuerza bruta y limitación de IP sospechosas">
  Si tu aplicación autenticada está configurada para enviar el encabezado `auth0-forwarded-for`:

  * Solo la dirección IP contenida en el encabezado `auth0-forwarded-for` se comprueba con las listas de permitidos de la protección contra ataques de fuerza bruta y la limitación de IP sospechosas.
  * La dirección IP del proxy es ignorada por la protección contra ataques de fuerza bruta y la limitación de IP sospechosas, por lo que no es necesario agregarla a las listas de permitidos.
  * Si determinados clientes que usan el proxy no deberían estar sujetos a la protección contra ataques de fuerza bruta o a la limitación de IP sospechosas, agrégalos a las listas de permitidos.

  El encabezado `auth0-forwarded-for` solo se aceptará para llamadas autenticadas con el secreto de cliente. Si tu aplicación no está autenticada o no está configurada para enviar el encabezado `auth0-forwarded-for`:

  * La dirección IP de origen de cada request se comprueba con las listas de permitidos de la protección contra ataques de fuerza bruta y la limitación de IP sospechosas.
  * Si incluyes la IP del proxy en la lista de permitidos, todo el tráfico que pase por el proxy quedará exento de la protección contra ataques de fuerza bruta y la limitación de IP sospechosas. Probablemente esto no sea lo que quieres.
</Card>

app.post('/api/auth', function(req, res, next) {
  var options = {
    method: 'POST',
    url: 'https://{yourDomain}/oauth/token',
    headers: {
      'content-type': 'application/x-www-form-urlencoded',
      'auth0-forwarded-for': req.ip // IP del usuario final
    },
    form: {
      grant_type: 'password',
      username: 'USERNAME',
      password: 'PASSWORD',
      audience: 'YOUR_API_IDENTIFIER',
      scope: 'SCOPE',
      client_id: '{yourClientId}',
      client_secret: '{yourClientSecret}' // El cliente está autenticado
    }
  };

  request(options, function (error, response, body) {
    if (error) return next(error);

    // ...
  });
});`;

<div id="handle-breached-password-detection-responses">
  ## Gestionar las respuestas de la detección de contraseñas vulnerada
</div>

Si has habilitado la [detección de contraseñas vulnerada](/docs/es/secure/attack-protection/breached-password-detection) para tu tenant, debes configurar tu aplicación para gestionar adecuadamente las respuestas de la [Auth0 Authentication API](https://auth0.com/docs/api/authentication).

Por ejemplo, si envías una contraseña mediante el flujo ROP y Auth0 detecta que se ha visto comprometida, la Authentication API devuelve una respuesta con el código de estado HTTP `401 Unauthorized` y el siguiente cuerpo:

```json lines
{
    "error": "password_leaked",
    "error_description": "This login attempt has been blocked because the password you're using was previously disclosed through a data breach (not in this application). Please check your email for more information."
}
```

Tu aplicación debe gestionar este error, renderizar un mensaje para el usuario y [activar un flujo interactivo de restablecimiento de contraseña](/docs/es/authenticate/database-connections/password-change#authentication-api).

