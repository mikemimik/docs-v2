<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Para usar las funcionalidades de Client-Initiated Backchannel Authentication (CIBA), debes tener un Enterprise Plan o un complemento adecuado. Consulta [Auth0 Pricing](https://auth0.com/pricing/) para obtener más información.
</Callout>

Cuando usas notificaciones push en dispositivo móvil con CIBA, el usuario recibe una notificación push para autenticarse o autorizar una solicitud en su dispositivo móvil registrado. Puedes enviar notificaciones push en dispositivo móvil con CIBA usando la aplicación Auth0 Guardian o una aplicación personalizada integrada con el Auth0 Guardian SDK.

El flujo de CIBA con notificaciones push en dispositivo móvil autentica y autoriza a los usuarios en su dispositivo móvil, sin necesidad de un navegador. Como el dispositivo de consumo no requiere una sesión activa en el navegador, el usuario no necesita haber iniciado sesión antes de que se active una solicitud de CIBA. Esto también garantiza que el flujo de CIBA no afecte a ninguna sesión existente que el usuario pueda tener.

<div id="prerequisites">
  ## Requisitos previos
</div>

Para iniciar una solicitud de push de CIBA con Auth0, debes:

* [Configure Client-Initiated Backchannel Authentication](/docs/es/get-started/applications/configure-client-initiated-backchannel-authentication) para tu tenant y aplicación, incluidas las [notificaciones push en dispositivos móviles](/docs/es/get-started/applications/configure-client-initiated-backchannel-authentication/#configure-mobile-push-notifications).
* Establecer el parámetro `requested_expiry` en un valor de 300 segundos o menos. Para obtener más información, consulta [Configure notification channel](/docs/es/get-started/applications/configure-client-initiated-backchannel-authentication#configure-notification-channel).

<div id="step-1-client-application-initiates-a-ciba-request">
  ## Paso 1: La aplicación cliente inicia una solicitud de CIBA
</div>

Usa las [API de User Search](/docs/es/manage-users/user-search) para encontrar al usuario autorizador para el que deseas iniciar una solicitud de CIBA y obtener su ID de usuario.

Una vez que tengas el ID de usuario del usuario autorizador, usa la Authentication API o nuestros [SDKs](/docs/es/libraries) para enviar una solicitud de CIBA al endpoint `/bc-authorize`:

<Tabs>
  <Tab title="cURL">
    ```bash lines
    curl --location 'https://{YOUR_DOMAIN}.auth0.com/bc-authorize' \
      --header 'Content-Type: application/x-www-form-urlencoded' \
      --data-urlencode 'client_id={YOUR_CLIENT_ID}' \
      --data-urlencode 'client_secret={YOUR_CLIENT_SECRET}' \
      --data-urlencode 'login_hint={ "format": "iss_sub", "iss": "https://{YOUR_DOMAIN}.auth0.com/", "sub": "{USER_ID}" }' \
      --data-urlencode 'scope={SCOPES}' \
      --data-urlencode 'binding_message={BINDING_MESSAGE}'
    ```
  </Tab>

  <Tab title="C#">
    ```csharp lines
    var response = await authenticationApiClient.ClientInitiatedBackchannelAuthorization(
                new ClientInitiatedBackchannelAuthorizationRequest()
                {
                    ClientId = "{YOUR_CLIENT_ID}",
                    Scope = "openid",
                    ClientSecret = "{YOUR_CLIENT_SECRET}",
                    BindingMessage = "{BINDING_MESSAGE}",
                    LoginHint = new LoginHint()
                    {
                        Format = "iss_sub",
                        Issuer = "https://{YOUR_DOMAIN}.auth0.com/",
                        Subject = "{USER_ID}"
                    }
                }
            );
    ```
  </Tab>

  <Tab title="Go">
    ```go lines
    resp, err := authAPI.CIBA.Initiate(context.Background(), ciba.Request{
    		ClientID:     mgmtClientID,
    		ClientSecret: mgmtClientSecret,
    		Scope:        "openid",
    		LoginHint: map[string]string{
    			"format": "iss_sub",
    			"iss":    "https://{YOUR_DOMAIN}.auth0.com/",
    			"sub":    "{USER_ID}",
    		},
    		BindingMessage: "{BINDING_MESSAGE}",
    	})
    ```
  </Tab>

  <Tab title="Java">
    ```java lines
    //Creating AuthClient Instance
    AuthAPI auth = AuthAPI.newBuilder(domain, clientId, clientSecret).build();

    //Authorize
    Map<String, Object> loginHint = new HashMap<>();
            loginHint.put("format", "iss_sub");
            loginHint.put("iss", "https://{YOUR_DOMAIN}.auth0.com/");
            loginHint.put("sub", "{USER_ID}");

    Request<BackChannelAuthorizeResponse> request = auth.authorizeBackChannel("openid", "{BINDING_MESSAGE}", loginHint);

    BackChannelAuthorizeResponse resp = request.execute().getBody();
    ```
  </Tab>
</Tabs>

| Parámetros         | Descripción                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| ------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `tenant`           | Nombre del tenant. También puede ser un dominio personalizado. Si se usa el formato `iss_sub`, el nombre del tenant se pasa en el claim `iss`.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                  |
| `client_id`        | Identificador de la aplicación cliente.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                         |
| `client_secret`    | Método de autenticación del cliente utilizado para la autenticación de usuario con CIBA, como secreto de cliente, clave privada JWT o autenticación mTLS. Si usas clave privada JWT o mTLS, no necesitas incluir el secreto de cliente.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                         |
| `scope`            | Debe incluir `openid`.<br /><br />El scope puede incluir opcionalmente `offline_access` para solicitar un token de actualización. Sin embargo, para la autorización puntual de una transaction con el flujo de CIBA, no se necesita un token de actualización y no tiene ningún significado en este contexto.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                   |
| `user_id`          | ID de usuario del usuario autorizador que se pasa dentro de la estructura `login_hint`. Si se usa el formato `iss_sub`, el ID de usuario se pasa en el claim `sub`.<br /><br />El ID de usuario puede tener un formato diferente según el proveedor externo.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                    |
| `requested_expiry` | La duración máxima, en segundos, durante la que la sesión de CIBA debe ser válida. La expiración solicitada del flujo de CIBA debe estar entre 1 y 259200 segundos (72 horas), y el valor predeterminado es 300 segundos. Incluye el parámetro `requested_expiry` para establecer una expiración personalizada para el flujo de CIBA.<br /><br />El parámetro `requested_expiry` ayuda a determinar qué canal de notificación usa CIBA:<ul><li>Si estableces `requested_expiry` en un valor de 300 segundos o menos, CIBA usa el canal de notificación push en dispositivo móvil si está habilitado. Si no has configurado MFA para tu tenant, la solicitud de CIBA falla.</li><li>Si estableces `requested_expiry` en un valor entre 301 y 259200 segundos (72 horas), CIBA usa el canal de notificación por correo electrónico si está habilitado.</li></ul> |
| `binding_message`  | Mensaje legible para humanos que se usa para vincular el flujo de CIBA entre los dispositivos de autenticación y de consumo. El mensaje de vinculación es obligatorio y puede tener hasta 64 caracteres. Usa solo caracteres alfanuméricos y `+-_.,:#`.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                         |

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Existe un límite de frecuencia específico por usuario: no se enviarán más de 5 solicitudes por minuto al usuario autorizador.
</Callout>

<div id="step-2-auth0-tenant-acknowledges-the-ciba-request">
  ## Paso 2: El tenant de Auth0 confirma la solicitud de CIBA
</div>

Si el tenant de Auth0 recibe correctamente la solicitud `POST`, deberías recibir una respuesta con un `auth-req-id` que hace referencia a la solicitud:

```json lines
{
    "auth_req_id": "eyJh...",
    "expires_in": 300,
    "interval": 5
}
```

El valor de `auth_req_id` se pasa al endpoint `/token` para consultar si se ha completado el flujo CIBA.

<div id="step-3-client-application-polls-for-a-response">
  ## Paso 3: La aplicación cliente consulta periódicamente en busca de una respuesta
</div>

Usa la Authentication API o nuestros [SDKs](/docs/es/libraries) para llamar al endpoint `/token` con el tipo de concesión `urn:openid:params:grant-type:ciba` y el `auth_req_id` que recibiste del endpoint `/bc-authorize`:

<Tabs>
  <Tab title="cURL">
    ```bash lines
    curl --location 'https://$tenant.auth0.com/oauth/token' \
      --header 'Content-Type: application/x-www-form-urlencoded' \
      --data-urlencode 'client_id=<CLIENT_ID>' \
      --data-urlencode 'client_secret=<CLIENT_SECRET>' \
      --data-urlencode 'auth_req_id=<AUTH_REQ_ID>' \
      --data-urlencode 'grant_type=urn:openid:params:grant-type:ciba'
    ```
  </Tab>

  <Tab title="C#">
    ```csharp lines
    var token = await authenticationApiClient.GetTokenAsync(
                new ClientInitiatedBackchannelAuthorizationTokenRequest()
                {
                    AuthRequestId = response.AuthRequestId,
                    ClientId = "<CLIENT_ID>",
                    ClientSecret = "<CLIENT_SECRET>"
                }
            );
    ```
  </Tab>

  <Tab title="Go">
    ```go lines
    token, err := authAPI.OAuth.LoginWithGrant(context.Background(),
    			"urn:openid:params:grant-type:ciba",
    			url.Values{
    				"auth_req_id":   []string{resp.AuthReqID},
    				"client_id":     []string{clientID},
    				"client_secret": []string{clientSecret},
    			},
    			oauth.IDTokenValidationOptions{})
    ```
  </Tab>

  <Tab title="Java">
    ```java lines
    Request<BackChannelTokenResponse> tokenRequest = auth.getBackChannelLoginStatus(authReqId, "grant-type");

    BackChannelTokenResponse tokenResponse = tokenRequest.execute().getBody();
    ```
  </Tab>
</Tabs>

Hasta que el usuario autorizador apruebe la transacción, deberías recibir la siguiente respuesta:

```json lines
{
    "error": "authorization_pending",
    "error_description": "La autorización del usuario final está pendiente"
}
```

Hay un intervalo de espera de aproximadamente cinco segundos entre consultas. Si consultas con demasiada frecuencia, recibirás la siguiente respuesta, donde la descripción varía según el intervalo de retroceso:

```json lines
{
"error": "slow_down",
"error_description": "You are polling faster than allowed. Try again in 10 seconds."
"interval": 10
}
```

Para resolver el error, espere hasta el siguiente intervalo (en segundos) antes de consultar el endpoint `/token`.

<div id="step-4-mobile-application-receives-the-push-notification">
  ## Paso 4: La aplicación móvil recibe la notificación push
</div>

Auth0 envía una notificación push a la aplicación móvil o al dispositivo registrados del usuario a través de la app Auth0 Guardian o de una aplicación personalizada integrada con el [Auth0 Guardian SDK](/docs/es/secure/multi-factor-authentication/auth0-guardian).

Si usa una aplicación personalizada, el [Auth0 Guardian SDK](/docs/es/secure/multi-factor-authentication/auth0-guardian) proporciona métodos para analizar los datos recibidos en la notificación push y devolver una instancia de `Notification` lista para usar. La instancia de `Notification` incluye un ID de vinculación de la transacción, o `txlinkid`, que la aplicación móvil usa para recuperar los detalles del consentimiento de Auth0.

Los siguientes ejemplos de código muestran implementaciones de notificaciones push móviles para iOS y Android con el Guardian SDK:

<Tabs>
  <Tab title="iOS">
    ```swift lines
    //implementando UNUserNotificationCenterDelegate
    func userNotificationCenter(_ center: UNUserNotificationCenter, willPresent notification: UNNotification, withCompletionHandler completionHandler: (UNNotificationPresentationOptions) -> Void) {
        let userInfo = notification.request.content.userInfo
        if let notification = Guardian.notification(from: userInfo) {
             // Implemente esta función para mostrar el prompt y gestionar la aprobación o el rechazo del usuario.
             handleGuardianNotification(notification: notification)
        }
    }
    ```
  </Tab>

  <Tab title="Android">
    ```kotlin lines
    // en el listener de FCM recibes un RemoteMessage
    @Override
    public void onMessageReceived(RemoteMessage message) {
        Notification notification = Guardian.parseNotification(message.getData());
        if (notification != null) {
            // recibiste una notificación de Guardian; gestiónala
            handleGuardianNotification(notification);
            return;
        }
        /* gestiona otras notificaciones push que puedas estar usando ... */
    }
    ```
  </Tab>
</Tabs>

<div id="step-5-mobile-application-retrieves-the-consent-details">
  ## Paso 5: La aplicación móvil obtiene los detalles del consentimiento
</div>

Tu aplicación Auth0 Guardian o una aplicación personalizada integrada con el Auth0 Guardian SDK obtiene los detalles del consentimiento, es decir, el contenido de `binding_message`, desde la Auth0 Consent API.

Si usas una aplicación personalizada, los siguientes ejemplos de código muestran implementaciones para iOS y Android que recuperan datos de la Auth0 Consent API:

<Tabs>
  <Tab title="iOS">
    ```swift lines
    let device: AuthenticationDevice = // el objeto que obtuviste durante el proceso inicial de inscripción en Guardian SDK y almacenaste localmente
    if let consentId = notification.transactionLinkingId {
        Guardian
            .consent(forDomain: {yourTenantDomain}, device: device)
            .fetch(consentId: consentId, notificationToken: notification.transactionToken)
            .start{result in
                switch result {
                case .success(let payload):
                    // mostrar los detalles del consentimiento al usuario
                case .failure(let cause):
                    // algo salió mal
            }
        }
    }
    ```
  </Tab>

  <Tab title="Android">
    ```kotlin lines
    Enrollment enrollment = // el objeto que obtuviste durante el proceso inicial de inscripción en Guardian SDK y almacenaste localmente
    if (notification.getTransactionLinkingId() != null) {
        guardian
          .fetchConsent(notification, enrollment)
          .start(new Callback<Enrollment> {
            @Override
            void onSuccess(RichConsent consentDetails) {
                // mostrar los detalles del consentimiento al usuario 
            }
            @Override
            void onFailure(Throwable exception) {
                // algo salió mal 
            }
          });
    }
    ```
  </Tab>
</Tabs>

<div id="step-6-mobile-application-presents-the-consent-details-to-the-user">
  ## Paso 6: La aplicación móvil presenta los detalles del consentimiento al usuario
</div>

La Auth0 Consent API responde a la Auth0 Guardian app o a tu aplicación personalizada integrada con el Auth0 Guardian SDK con los detalles del consentimiento, incluidos `binding_message`, `scope` y `audience`. Los scopes devueltos a la aplicación móvil se filtran según tu política de RBAC. Para obtener más información, consulta [Role-Based Access Control](/docs/es/manage-users/access-control/rbac).

La aplicación móvil presenta la solicitud de autenticación y/o los detalles del consentimiento al usuario.

El siguiente fragmento de código muestra una respuesta de ejemplo de la Auth0 Consent API:

```json lines
{
  "id": "cns_abc123",
  "requested_details": {
    "audience": "https://$tenant.auth0.com/userinfo",
    "scope": ["openid"],
    "binding_message": "21-49-38"
  },
  "created_at": 1746693720
  "expires_at": 1746693750
}
```

En este punto, el usuario puede aceptar o rechazar la solicitud de autenticación.

<div id="step-7-mobile-application-sends-the-user-response-back-to-auth0">
  ## Paso 7: La aplicación móvil envía la respuesta del usuario a Auth0
</div>

La aplicación Auth0 Guardian o tu aplicación personalizada envía la respuesta del usuario a Auth0.

<div id="step-8-auth0-receives-user-response-after-the-flow-completes">
  ## Paso 8: Auth0 recibe la respuesta del usuario después de que se completa el flujo
</div>

La aplicación cliente deja de consultar al recibir una respuesta del endpoint `/token`. Un flujo CIBA siempre requiere una respuesta del usuario autorizador, ya sea de aprobación o de rechazo, y no se comprueban las concesiones existentes.

<div id="step-9-auth0-returns-access-token-to-client-application">
  ## Paso 9: Auth0 devuelve el token de acceso a la aplicación cliente
</div>

 Si el usuario rechaza la solicitud push, Auth0 devuelve a la aplicación cliente una respuesta de error como la siguiente:

```json lines
{
    "error": "access_denied",
    "error_description": "El usuario final denegó la solicitud de autorización o ha caducado"
}
```

Si el usuario aprueba la solicitud push, Auth0 devuelve a la aplicación cliente un <Tooltip tip="Token de acceso: credencial de autorización, en forma de cadena opaca o JWT, que se utiliza para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=access+token">token de acceso</Tooltip> como el siguiente:

```json lines
{
    "access_token": "eyJh...",
    "id_token": "eyJh...",
    "expires_in": 86400,
    "scope": "openid",
    "token_type": "Bearer"
}
```

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  El `refresh_token` solo se incluirá si el scope `offline_access` se incluyó en la solicitud inicial a `/bc-authorize`.
</Callout>
