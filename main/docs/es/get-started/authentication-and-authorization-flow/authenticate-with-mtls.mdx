<div id="mtls-in-oauthoidc">
  ## mTLS en OAuth/OIDC
</div>

Los flujos predeterminados de <Tooltip tip="OAuth 2.0: Marco de autorización que define protocolos y flujos de trabajo de autorización." cta="Ver glosario" href="/docs/es/glossary?term=OAuth">OAuth</Tooltip>/OIDC no siempre son seguros debido a los siguientes problemas:

* El uso de un <Tooltip tip="Client Secret: Secreto usado por un cliente (aplicación) para autenticarse con el servidor de autorización; solo deberían conocerlo el cliente y el servidor de autorización, y debe ser lo suficientemente aleatorio como para que no pueda adivinarse." cta="Ver glosario" href="/docs/es/glossary?term=Client+Secret">secreto de cliente</Tooltip> compartido como método de autenticación del cliente.
* La posibilidad de que un <Tooltip tip="Access Token: Credencial de autorización, en forma de cadena opaca o JWT, usada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=access+token">token de acceso</Tooltip> sea utilizado por partes no autorizadas.

En 2020, el Internet Engineering Task Force (IETF) publicó [RFC 8705](https://www.rfc-editor.org/rfc/rfc8705), sobre autenticación de cliente Mutual-TLS (mTLS), para abordar estos problemas. Con la autenticación mTLS, el certificado de cliente con una clave privada funciona como un secreto de cliente en un flujo de OAuth/OIDC para verificar la identidad del cliente. Si un cliente ya está autenticado en la capa de red, no hace falta un secreto de cliente en la capa de aplicación. Además, los certificados de cliente pueden usarse con varios servidores para demostrar la identidad de un cliente ante un <Tooltip tip="Resource Server: Servidor que aloja recursos protegidos. Los resource servers aceptan y responden a solicitudes de recursos protegidos." cta="Ver glosario" href="/docs/es/glossary?term=resource+server">servidor de recursos</Tooltip>. Ten en cuenta que existen otros enfoques para resolver los problemas anteriores, en concreto [Private Key JWT](/docs/es/get-started/authentication-and-authorization-flow/authenticate-with-private-key-jwt) y [DPoP](https://datatracker.ietf.org/doc/html/rfc9449), respectivamente.

Para proteger un flujo de OAuth con mTLS, los clientes envían un certificado mTLS al punto de terminación TLS de la red perimetral del cliente mientras se establece la conexión TLS. Antes de que el <Tooltip tip="Authorization Server: Servidor centralizado que ayuda a definir los límites del acceso de un usuario. Por ejemplo, tu servidor de autorización puede controlar los datos, las tareas y las funciones disponibles para un usuario." cta="Ver glosario" href="/docs/es/glossary?term=authorization+server">servidor de autorización</Tooltip> procese la solicitud, primero debe verificar el certificado mTLS del cliente.

<Frame>![](/docs/images/cdy7uua7fh8z/4SlprP2uTYMCoLIPLsOl4v/d17575139419453d8772081ad20b7499/HRI_diagrams_-_mtls_diagram_1__2_.png)</Frame>

Opcionalmente, mTLS también puede usarse para garantizar que un token de acceso sea utilizado solo por la parte prevista, lo que se conoce como Sender Constraining o Token Binding. Cuando el cliente llama al endpoint `/oauth/token` del servidor de autorización mediante una conexión mTLS, el token de acceso resultante contiene información que el servidor de recursos usa para verificar que el certificado TLS del cliente coincida con el del token de acceso.

<Frame>![](/docs/images/cdy7uua7fh8z/7qocbfqySAnu85ph6WVSGU/ee8cd3514ed1bb6fea554cbd63d230cf/HRI_diagrams_-_mtls_diagram_2__1_.png)</Frame>

**Nota**: la autenticación de cliente con mTLS y mTLS Token Binding pueden usarse de forma independiente. La autenticación de cliente con mTLS puede usarse sin mTLS Token Binding, y mTLS Token Binding puede usarse con otras formas de autenticación del cliente, como secreto de cliente o clave privada <Tooltip tip="JSON Web Token (JWT): Formato estándar de ID Token (y a menudo también de Access Token) usado para representar claims de forma segura entre dos partes." cta="Ver glosario" href="/docs/es/glossary?term=JWT">JWT</Tooltip>. Incluso si se usan otras formas de autenticación del cliente, el cliente sigue enviando el certificado de cliente al servidor de autorización para mTLS Token Binding.

<div id="mtls-at-auth0">
  ## mTLS en Auth0
</div>

mTLS para Auth0 se basa en [dominios personalizados](/docs/es/customize/custom-domains) y aprovecha la infraestructura de mTLS existente del cliente para realizar el aprovisionamiento y la verificación de certificados.

Las llamadas autenticadas del cliente a Auth0 que normalmente requieren un secreto de cliente se envían primero al customer edge. Esto ya ocurre con los <Tooltip tip="Dominio personalizado: dominio de terceros con un nombre especializado o personalizado." cta="Ver glosario" href="/docs/es/glossary?term=custom+domains">dominios personalizados</Tooltip> que usan certificados gestionados por el cliente. El customer edge realiza el handshake de mTLS con el cliente y valida el certificado del cliente. Una vez verificado el certificado del cliente, la solicitud se reenvía al dominio perimetral del tenant en Auth0, e incluye el certificado del cliente validado en un encabezado HTTP junto con la `cname-api-key` correcta, de acuerdo con la funcionalidad de dominios personalizados.

<div id="call-the-authorization-server">
  ## Llamar al servidor de autorización
</div>

Dado que mTLS sirve tanto para la autenticación del cliente como para la vinculación del token de acceso, el cliente debe saber si estas funciones están habilitadas en el servidor de autorización. Además, los endpoints de mTLS y los endpoints sin mTLS de un servidor de autorización pueden estar expuestos en dominios diferentes.

Para obtener detalles de configuración sobre el servidor de autorización, el cliente envía una solicitud GET al endpoint de [OpenID Connect Discovery](https://openid.net/specs/openid-connect-discovery-1_0.html): `https://<custom-domain>/.well-known/openid-configuration`

Una respuesta satisfactoria devuelve el documento de descubrimiento de OIDC, es decir, un objeto JSON que enumera las propiedades y endpoints del servidor de autorización, incluidos los relacionados con mTLS.

Si la autenticación de cliente con mTLS está habilitada, el documento de descubrimiento de OIDC incluye la propiedad `token_endpoint_auth_methods_supported`, que contiene `tls_client_auth` o `self_signed_tls_client_auth`:

```json lines
{
  ...
  "token_endpoint_auth_methods_supported": ["tls_client_auth"]
  ...
}
```

Si mTLS Token Binding está activado, el documento de descubrimiento de OIDC establece la propiedad `tls_client_certificate_bound_access_tokens` en `true:`

```json lines
{
  ...
  "tls_client_certificate_bound_access_tokens": true
  ...
}
```

Los entornos que admiten alias de endpoint mTLS exponen una nueva propiedad, `mtls_endpoint_aliases`, que contiene una lista de endpoints compatibles con mTLS. Para los clientes que admiten mTLS, los endpoints enumerados en `mtls_endpoint_aliases` tienen prioridad sobre los mismos endpoints expuestos fuera de `mtls_endpoint_aliases`.

En el siguiente ejemplo de código, la propiedad `token_endpoint` se expone dos veces. El endpoint que debe usarse para las llamadas mTLS aparece en `mtls_endpoint_aliases`, o `https://mtls.auth.bank.com/oauth/token`:

```json lines
{
  ...
  "mtls_endpoint_aliases": {
"token_endpoint": "https://mtls.auth.bank.com/oauth/token"
  },
  "token_endpoint": "https://auth.bank.com/oauth/token",
  "pushed_authorization_request_endpoint": "https://auth.bank.com/oauth/par",
  ...
}
```

Si un endpoint no figura en `mtls_endpoint_aliases`, usa el mismo endpoint que figura fuera de `mtls_endpoint_aliases`. En el ejemplo anterior, `pushed_authorization_request_endpoint` no figura en `mtls_endpoint_aliases`. Por lo tanto, usa el `pushed_authorization_request_endpoint` expuesto fuera de `mtls_endpoint_aliases`, es decir, `https://auth.bank.com/oauth/par`.

Para obtener más información, consulta la [sección sobre alias de endpoint](https://www.rfc-editor.org/rfc/rfc8705#name-metadata-for-mutual-tls-end) del RFC 8705.

<div id="call-the-resource-server">
  ## Llamar al servidor de recursos
</div>

Una vez que un cliente recibe un token de acceso, puede acceder a recursos protegidos en un servidor de recursos. Si mTLS Token Binding está habilitado, el servidor de autorización devuelve el documento de descubrimiento de OIDC que contiene la propiedad `tls_client_certificate_bound_access_tokens`.

Cuando el cliente llama al servidor de recursos con un token de acceso vinculado a mTLS, el servidor de recursos solicita al cliente un certificado de cliente mTLS durante el handshake de TLS. El servidor de recursos debe rechazar las solicitudes con un token de acceso que no coincida con ese certificado de cliente con un código de estado HTTP 401 y el código de error `invalid_token`. Para obtener más información, consulta [Configure Resource Server for Sender Constraining](/docs/es/secure/sender-constraining/configure-sender-constraining/configure-resource-server-for-sender-constraining).
