<div id="prerequisites">
  ## Requisitos previos
</div>

**Antes de comenzar este tutorial:**

* [Registra tu aplicación en Auth0](/docs/es/get-started/auth0-overview/create-applications/regular-web-apps).

  * Selecciona el **Application Type** adecuado.
  * Agrega una **Allowed Callback URL** de `{https://yourApp/callback}`.
  * Asegúrate de que los **Grant Types** de tu aplicación incluyan **Implicit** y **Código de autorización**. Para saber cómo hacerlo, consulta [Actualizar los tipos de concesión](/docs/es/get-started/applications/update-grant-types).
  * Si quieres que tu aplicación pueda usar tokens de actualización, asegúrate de que los **Grant Types** de la aplicación incluyan **Token de actualización**. Para saber cómo hacerlo, consulta [Actualizar los tipos de concesión](/docs/es/get-started/applications/update-grant-types). Para obtener más información sobre los tokens de actualización, consulta [Tokens de actualización](/docs/es/secure/tokens/refresh-tokens).
* [Registra tu API en Auth0](/docs/es/get-started/auth0-overview/set-up-apis)

  * Si quieres que tu API reciba tokens de actualización para poder obtener tokens nuevos cuando caduquen los anteriores, habilita **Allow Offline Access**.

<div id="steps">
  ## Pasos
</div>

1. [Autorizar al usuario](#authorize-user):
   Solicita la autorización del usuario y redirígelo a tu aplicación con un código de autorización.
2. [Solicitar tokens](#request-tokens):
   Intercambia tu código de autorización por tokens.
3. [Llamar a la API](#call-api):
   Usa el token de acceso obtenido para llamar a tu API.
4. [Actualizar tokens](#refresh-tokens):
   Usa un token de actualización para solicitar nuevos tokens cuando caduquen los actuales.

Opcional: [Explora casos de uso de muestra](#sample-use-cases)

<div id="authorize-user">
  ### Autorizar al usuario
</div>

Este paso puede incluir uno o más de los siguientes procesos:

* Autenticar al usuario
* Redirigir al usuario a un <Tooltip tip="Proveedor de identidad (IdP): Servicio que almacena y administra identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=Identity+Provider">proveedor de identidad</Tooltip> para gestionar la autenticación
* Comprobar si existen sesiones activas de <Tooltip tip="Inicio de sesión único (SSO): Servicio que, después de que un usuario inicia sesión en una aplicación, inicia sesión automáticamente en otras aplicaciones." cta="Ver glosario" href="/docs/es/glossary?term=Single+Sign-on">inicio de sesión único</Tooltip> (SSO)
* Obtener el consentimiento del usuario para el nivel de permiso solicitado, a menos que ya se haya otorgado previamente.

Para autorizar al usuario, tu aplicación debe enviar al usuario a la [URL de autorización](https://auth0.com/docs/api/authentication#authorization-code-grant).

<div id="parameters">
  ##### Parámetros
</div>

Ten en cuenta que, para autorizar a un usuario al llamar a una API personalizada, debes:

* incluir un parámetro <Tooltip tip="Audience: Identificador único de la audience de un token emitido. Se denomina aud en un token; su valor contiene el ID de una aplicación (ID de cliente) para un token de ID o de una API (identificador de API) para un token de acceso." cta="Ver glosario" href="/docs/es/glossary?term=audience">audience</Tooltip>
* incluir scopes adicionales compatibles con la API de destino

| Nombre del parámetro | Descripción                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                        |
| -------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `response_type`      | Indica el tipo de credencial que devolverá Auth0 (`code` o `token`). Para este flujo, el valor debe incluir `code`, pero también puede incluir `id_token`, `token` o `id_token token`. En concreto, `id_token` devuelve un token de ID y `token` devuelve un token de acceso.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| `response_mode`      | Especifica el método con el que deben devolverse los parámetros de respuesta. Por motivos de seguridad, el valor debe ser `form_post`. En este modo, los parámetros de respuesta se codifican como valores de formulario HTML que se transmiten mediante el método HTTP POST y se codifican en el cuerpo con el formato `application/x-www-form-urlencoded`.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                       |
| `client_id`          | El ID de cliente de tu aplicación. Puedes encontrar este valor en [Application Settings](https://manage.auth0.com/#/Applications/\{yourClientId}/settings).                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                        |
| `redirect_uri`       | La URL a la que Auth0 redirigirá el navegador después de que el usuario haya concedido la autorización. El código de autorización estará disponible en el parámetro de URL `code`. Debes especificar esta URL como una URL de devolución de llamada válida en [Application Settings](https://manage.auth0.com/#/Applications/\{yourClientId}/settings).<br /><br />**Advertencia:** Según la [especificación OAuth 2.0](https://tools.ietf.org/html/rfc6749#section-3.1.2), Auth0 elimina todo lo que aparece después del hash y *no* admite fragmentos.                                                                                                                                                                                                                                                                                                                                                                                           |
| `scope`              | Especifica los [scopes](/docs/es/get-started/apis/scopes) para los que quieres solicitar autorización, que determinan qué claims (o atributos del usuario) quieres que se devuelvan. Deben separarse con un espacio. Puedes solicitar cualquiera de los [scopes estándar de OpenID Connect (OIDC)](https://openid.net/specs/openid-connect-core-1_0.html#StandardClaims) sobre usuarios, como `profile` o `email`, [custom claims](/docs/es/secure/tokens/json-web-tokens/json-web-token-claims#custom-claims) que sigan un [formato con espacio de nombres](/docs/es/secure/tokens/json-web-tokens/create-custom-claims), o cualquier scope compatible con la API de destino (por ejemplo, `read:contacts`). Incluye `offline_access` para obtener un [token de actualización](/docs/es/glossary?term=Refresh+Token) (asegúrate de que el campo **Allow Offline Access** esté habilitado en [Application Settings](https://manage.auth0.com/#/applications)). |
| `audience`           | El identificador único de la API a la que tu aplicación quiere acceder. Usa el valor **Identifier** en la pestaña [Settings](https://manage.auth0.com/#/apis) de la API que creaste como parte de los requisitos previos de este tutorial.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                         |
| `state`              | (recomendado) Una cadena alfanumérica arbitraria y opaca que tu aplicación añade a la solicitud inicial y que Auth0 incluye al redirigir de vuelta a tu aplicación. Para ver cómo usar este valor para prevenir ataques de falsificación de solicitud entre sitio (CSRF), consulta [Mitigate CSRF Attacks With State Parameters](/docs/es/secure/attack-protection/state-parameters).                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| `nonce`              | Una cadena aleatoria criptográficamente segura que tu aplicación añade a la solicitud inicial y que Auth0 incluye dentro del token de ID, [utilizada para prevenir ataques de repetición de tokens](/docs/es/get-started/authentication-and-authorization-flow/implicit-flow-with-form-post/mitigate-replay-attacks-when-using-the-implicit-flow).                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                    |
| `organization`       | (opcional) ID de la organización que se utilizará al autenticar a un usuario. Si no se proporciona y tu aplicación está configurada para **Display Organization Prompt**, el usuario podrá introducir el nombre de la organización al autenticarse.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                |
| `invitation`         | (opcional) ID del ticket de invitación de la organización. Al [invitar a un miembro a una Organization](/docs/es/manage-users/organizations/configure-organizations/invite-members), tu aplicación debe gestionar la aceptación de la invitación reenviando los pares clave-valor `invitation` y `organization` cuando el usuario acepte la invitación.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                               |

Como ejemplo, el fragmento HTML de tu URL de autorización al agregar inicio de sesión a tu aplicación podría verse así:

export const codeExample2 = `<a href="https://{yourDomain}/authorize?
  response_type=code id_token token&
  client_id={yourClientId}&
  redirect_uri={https://yourApp/callback}&  
  scope=appointments%20contacts&
  audience=appointments:api&
  state=xyzABC123&
  nonce=eq...hPmz">
  Sign In
</a>`;

<div id="response">
  #### Respuesta
</div>

Si todo va bien, recibirás una respuesta `HTTP 302`. Las credenciales solicitadas están codificadas en el cuerpo:

```json lines
HTTP/1.1 302 Found
Content-Type: application/x-www-form-urlencoded
code=AUTHORIZATION_CODE&
access_token=ey...MhPw
&expires_in=7200
&token_type=Bearer
id_token=eyJ...acA&
state=xyzABC123
```

Ten en cuenta que los valores devueltos dependen de lo que hayas solicitado como `response_type`.

| Tipo de respuesta | Componentes                                                                |
| ----------------- | -------------------------------------------------------------------------- |
| `code `           | código de autorización                                                     |
| `id_token `       | token de ID                                                                |
| `token `          | token de acceso (más los valores `expires_in` y `token_type`)              |
| `id_token token`  | token de ID, token de acceso (más los valores `expires_in` y `token_type`) |

Auth0 también devolverá cualquier valor de `state` que hayas incluido en tu llamada a la URL de autorización.

<Warning>
  El token de acceso que recibes en esta transacción es solo el primero que recibirás. No recomendamos usarlo para llamar a API.
</Warning>

<Warning>
  Valida tus tokens antes de guardarlos. Para saber cómo hacerlo, lee [Validar tokens de ID](/docs/es/secure/tokens/id-tokens/validate-id-tokens) y [Validar tokens de acceso](/docs/es/secure/tokens/access-tokens/validate-access-tokens).
</Warning>

Cuando decodifiques y analices tu <Tooltip tip="ID Token: Credencial pensada para el propio cliente, y no para acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=ID+token">token de identidad</Tooltip>, notarás un claim adicional, `c_hash`, que contiene un hash del `code`. Este claim es obligatorio cuando se emite un token de identidad al mismo tiempo que un `code`, y debes validarlo:

1. Con el algoritmo hash especificado en el claim `alg` del header del token de ID, aplica hash a los octetos de la representación ASCII del `code`.
2. Codifica en Base64url la mitad izquierda del hash.
3. Comprueba que el resultado coincida con el valor de `c_hash`.

<div id="request-tokens">
  ### Solicita tokens
</div>

Ahora que tienes un código de autorización, debes intercambiarlo por tokens. Con el código de autorización extraído (`code`) del paso anterior, tendrás que hacer una solicitud `POST` a la [URL del token](https://auth0.com/docs/api/authentication#authorization-code).

El <Tooltip tip="Token de acceso: credencial de autorización, en forma de una cadena opaca o JWT, que se usa para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=Access+Token">Token de acceso</Tooltip> que recibas en este paso es el que debes usar para llamar a tu API. Asegúrate de mantenerlo separado del token de acceso que recibiste en el paso anterior de este tutorial.

<div id="example-post-to-token-url">

</div>

  ```bash cURL
  curl --request POST \
    --url 'https://{yourDomain}/oauth/token' \
    --header 'content-type: application/x-www-form-urlencoded' \
    --data grant_type=authorization_code \
    --data 'client_id={yourClientId}' \
    --data 'client_secret={yourClientSecret}' \
    --data 'code=yourAuthorizationCode}' \
    --data 'redirect_uri={https://yourApp/callback}'
  ```

  ```csharp C#
  var client = new RestClient("https://{yourDomain}/oauth/token");
  var request = new RestRequest(Method.POST);
  request.AddHeader("content-type", "application/x-www-form-urlencoded");
  request.AddParameter("application/x-www-form-urlencoded", "grant_type=authorization_code&client_id={yourClientId}&client_secret=%7ByourClientSecret%7D&code=yourAuthorizationCode%7D&redirect_uri={https://yourApp/callback}", ParameterType.RequestBody);
  IRestResponse response = client.Execute(request);
  ```

  ```go Go
  package main

  import (
  	"fmt"
  	"strings"
  	"net/http"
  	"io/ioutil"
  )

  func main() {

  	url := "https://{yourDomain}/oauth/token"

  	payload := strings.NewReader("grant_type=authorization_code&client_id={yourClientId}&client_secret=%7ByourClientSecret%7D&code=yourAuthorizationCode%7D&redirect_uri={https://yourApp/callback}")

  	req, _ := http.NewRequest("POST", url, payload)

  	req.Header.Add("content-type", "application/x-www-form-urlencoded")

  	res, _ := http.DefaultClient.Do(req)

  	defer res.Body.Close()
  	body, _ := ioutil.ReadAll(res.Body)

  	fmt.Println(res)
  	fmt.Println(string(body))

  }
  ```

  ```java Java

    .header("content-type", "application/x-www-form-urlencoded")
    .body("grant_type=authorization_code&client_id={yourClientId}&client_secret=%7ByourClientSecret%7D&code=yourAuthorizationCode%7D&redirect_uri={https://yourApp/callback}")
  ```

<div id="response">
  #### Respuesta
</div>

Si todo va bien, recibirás una respuesta `HTTP 200` con un payload que contiene `access_token`, `refresh_token`, `id_token` y `token_type`:

```json lines
{
  "access_token": "eyJz93a...k4laUWw",
  "refresh_token": "GEbRxBN...edjnXbL",
  "id_token": "eyJ0XAi...4faeEoQ",
  "token_type": "Bearer"
}
```

<Warning>
  Valida tus tokens antes de guardarlos. Para saber cómo, lee [Validar tokens de ID](/docs/es/secure/tokens/id-tokens/validate-id-tokens) y [Validar tokens de acceso](/docs/es/secure/tokens/access-tokens/validate-access-tokens).
</Warning>

[Los tokens de ID](/docs/es/secure/tokens/id-tokens) contienen información del usuario que debe decodificarse y extraerse.

[Los tokens de acceso](/docs/es/secure/tokens/access-tokens) se utilizan para llamar al [endpoint /userinfo de Auth0 Authentication API](https://auth0.com/docs/api/authentication#get-user-info) o a otra API. Si llamas a tu propia API, lo primero que debe hacer es [verificar el token de acceso](/docs/es/secure/tokens/access-tokens/validate-access-tokens).

[Los tokens de actualización](/docs/es/secure/tokens/refresh-tokens) se utilizan para obtener un nuevo token de acceso o token de identidad cuando el anterior haya expirado. El `refresh_token` solo estará presente en la respuesta si incluiste el scope `offline_access` y habilitaste **Allow Offline Access** para tu API en el Dashboard.

<Warning>
  Los tokens de actualización deben almacenarse de forma segura, ya que permiten que un usuario permanezca autenticado prácticamente para siempre.
</Warning>

<div id="call-api">
  ### Call API
</div>

Para llamar a tu API desde una aplicación web tradicional (o en casos similares en los que las credenciales de la aplicación pueden almacenarse de forma segura), la aplicación debe enviar el token de acceso obtenido como token Bearer en el encabezado Authorization de tu solicitud HTTP.

<AuthCodeGroup>
  ```bash cURL lines
  curl --request GET \
    --url https://myapi.com/api \
    --header 'authorization: Bearer {accessToken}' \
    --header 'content-type: application/json'
  ```

  ```csharp C# lines
  var client = new RestClient("https://myapi.com/api");
  var request = new RestRequest(Method.GET);
  request.AddHeader("content-type", "application/json");
  request.AddHeader("authorization", "Bearer {accessToken}");
  IRestResponse response = client.Execute(request);
  ```

<div id="refresh-tokens">
  ### Tokens de actualización
</div>

Ya has recibido un [token de actualización](/docs/es/secure/tokens/refresh-tokens) si has seguido este tutorial y completado lo siguiente:

* configuraste tu API para permitir acceso sin conexión
* incluiste el scope `offline_access` cuando iniciaste la solicitud de autenticación a través del [endpoint `/authorize`](https://auth0.com/docs/api/authentication/reference#authorize-application).

Puedes usar el <Tooltip tip="Token de actualización: Token utilizado para obtener un token de acceso renovado sin obligar a los usuarios a iniciar sesión de nuevo." cta="Ver glosario" href="/docs/es/glossary?term=Refresh+Token">Token de actualización</Tooltip> para obtener un nuevo token de acceso. Por lo general, un usuario solo necesitará un nuevo token de acceso después de que venza el anterior o cuando obtenga acceso a un recurso nuevo por primera vez. Es una mala práctica llamar al endpoint para obtener un nuevo token de acceso cada vez que llamas a una API, y Auth0 aplica límites de frecuencia que restringen la cantidad de solicitudes al endpoint que se pueden ejecutar con el mismo token desde la misma IP.

Para renovar tu token, haz una solicitud `POST` al endpoint `/oauth/token` de la Authentication API, usando `grant_type=refresh_token`.

<div id="parameters">
  ##### Parámetros
</div>

| Nombre del parámetro | Descripción                                                                                                                                                                                                                          |
| -------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `grant_type`         | Establece este valor en `refresh_token`.                                                                                                                                                                                             |
| `client_id`          | El ID de cliente de tu aplicación. Puedes encontrar este valor en la [Application Settings](https://manage.auth0.com/#/Applications/\{yourClientId}/settings).                                                             |
| `refresh_token`      | El token de actualización que se usará.                                                                                                                                                                                              |
| `scope`              | (opcional) Una lista de scopes solicitados, delimitada por espacios. Si no se envía, se usarán los scopes originales; de lo contrario, puedes solicitar un conjunto reducido de scopes. Ten en cuenta que debe codificarse para URL. |

<div id="response">
  #### Respuesta
</div>

Si todo sale bien, recibirás una respuesta `HTTP 200` con un payload que incluye un nuevo `access_token`, su duración en segundos (`expires_in`), los valores de `scope` otorgados y `token_type`. Si el scope del token inicial incluía `openid`, la respuesta también incluirá un nuevo `id_token`:

```json lines
{
  "access_token": "eyJ...MoQ",
  "expires_in": 86400,
  "scope": "openid offline_access",
  "id_token": "eyJ...0NE",
  "token_type": "Bearer"
}
```

<Warning>
  Valida tus tokens antes de guardarlos. Para saber cómo, lee [Validar tokens de ID](/docs/es/secure/tokens/id-tokens/validate-id-tokens) y [Validar tokens de acceso](/docs/es/secure/tokens/access-tokens/validate-access-tokens).
</Warning>

<div id="customize-tokens">
  #### Personaliza los tokens
</div>

Puedes usar Rules para cambiar los scopes que se devuelven en los Access Tokens y/o agregar claims a los Access Tokens y a los ID Tokens. (Para obtener más información sobre las Rules, consulta [Auth0 Rules](/docs/es/customize/rules).) Para ello, agrega la siguiente Rule, que se ejecutará después de que el usuario se autentique:

```js lines
exports.onExecutePostLogin = async (event, api) => {
  // Agregar claims personalizados al Access Token y al token de ID
  api.accessToken.setCustomClaim('https://foo/bar', 'value');
  api.idToken.setCustomClaim('https://fiz/baz', 'some other value');

  // Modificar el scope del Access Token
  api.accessToken.addScope('foo');
  api.accessToken.addScope('bar');
};
```

Los scopes estarán disponibles en el token una vez que se hayan ejecutado todas las reglas.

<Warning>
  Auth0 devuelve la información del perfil en un formato estructurado de claims, tal como lo define la [especificación OpenID Connect (OIDC)](https://openid.net/specs/openid-connect-core-1_0.html#StandardClaims). Esto significa que los custom claims agregados a los tokens de ID o a los tokens de acceso deben [ajustarse a las directrices y restricciones](/docs/es/secure/tokens/json-web-tokens/create-custom-claims) para evitar posibles colisiones.
</Warning>
