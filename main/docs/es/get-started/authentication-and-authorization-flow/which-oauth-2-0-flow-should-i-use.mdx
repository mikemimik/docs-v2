title: ¿Qué flujo de OAuth 2.0 debo usar?
---

El [marco de autorización de OAuth 2.0](/docs/es/authenticate/protocols/oauth) admite varios flujos diferentes (o concesiones). Los <Tooltip tip="Flow: Procesos que pueden ampliarse mediante Actions. Cada Flow está compuesto por uno o más triggers y representa el pipeline lógico por el que se mueve la información en un punto concreto del recorrido de Auth0." cta="Ver glosario" href="/docs/es/glossary?term=Flow">Flujos</Tooltip> son formas de obtener un <Tooltip tip="Flow: Procesos que pueden ampliarse mediante Actions. Cada Flow está compuesto por uno o más triggers y representa el pipeline lógico por el que se mueve la información en un punto concreto del recorrido de Auth0." cta="Ver glosario" href="/docs/es/glossary?term=Access+Token">token de acceso</Tooltip>. Decidir cuál es el más adecuado para tu caso de uso depende principalmente de tu [tipo de aplicación](/docs/es/get-started/applications), pero también influyen otros parámetros, como el nivel de confianza del cliente o la experiencia que quieres ofrecer a tus usuarios.

<div id="oauth-20-terminology">
  ## Terminología de OAuth 2.0
</div>

* **<Tooltip tip="Propietario del recurso: Entidad (como un usuario o una aplicación) capaz de otorgar acceso a un recurso protegido." cta="Ver glosario" href="/docs/es/glossary?term=Resource+Owner">Propietario del recurso</Tooltip>**: Entidad que puede otorgar acceso a un recurso protegido. Normalmente, se trata del usuario final.
* **Cliente**: Aplicación que solicita acceso a un recurso protegido en nombre del Propietario del recurso.
* **<Tooltip tip="Servidor de recursos: Servidor que aloja recursos protegidos. Los servidores de recursos aceptan y responden a solicitudes de recursos protegidos." cta="Ver glosario" href="/docs/es/glossary?term=Resource+Server">Servidor de recursos</Tooltip>**: Servidor que aloja los recursos protegidos. Es la API a la que quieres acceder.
* **<Tooltip tip="Servidor de autorización: Servidor centralizado que contribuye a definir los límites del acceso de un usuario. Por ejemplo, tu servidor de autorización puede controlar los datos, las tareas y las funcionalidades disponibles para un usuario." cta="Ver glosario" href="/docs/es/glossary?term=Authorization+Server">Servidor de autorización</Tooltip>**: Servidor que autentica al Propietario del recurso y emite tokens de acceso tras obtener la autorización adecuada. En este caso, Auth0.
* **Agente de usuario**: Agente que utiliza el Propietario del recurso para interactuar con el Cliente (por ejemplo, un navegador o una aplicación nativa).

<div id="is-the-client-the-resource-owner">
  ## ¿Es el cliente el Propietario del recurso?
</div>

La primera decisión consiste en determinar si la parte que necesita acceso a los recursos es una máquina. En el caso de la autorización de máquina a máquina, el cliente también es el Propietario del recurso, por lo que no se requiere autorización del usuario final. Un ejemplo es una tarea cron que usa una API para importar información a una base de datos. En este ejemplo, la tarea cron es el cliente y el Propietario del recurso, ya que tiene el <Tooltip tip="ID de cliente: valor de identificación asignado por Auth0 a su recurso registrado." cta="Ver glosario" href="/docs/es/glossary?term=Client+ID">ID de cliente</Tooltip> y el <Tooltip tip="secreto de cliente: Secret que usa un cliente (aplicación) para autenticarse ante el servidor de autorización; solo deben conocerlo el cliente y el servidor de autorización, y debe ser lo suficientemente aleatorio como para que no pueda adivinarse." cta="Ver glosario" href="/docs/es/glossary?term=Client+Secret">secreto de cliente</Tooltip>, y los usa para obtener un token de acceso del servidor de autorización.

Si este caso se ajusta a sus necesidades, para obtener más información sobre cómo funciona este flujo y cómo implementarlo, consulte [Client Credentials Flow](/docs/es/get-started/authentication-and-authorization-flow/client-credentials-flow).

<div id="is-the-client-a-web-app-executing-on-the-server">
  ## ¿El cliente es una aplicación web que se ejecuta en el servidor?
</div>

Si el cliente es una aplicación web tradicional que se ejecuta en un servidor, el flujo que debe usar es el flujo de código de autorización. Con este, el cliente puede obtener un token de acceso y, opcionalmente, un <Tooltip tip="token de actualización: token que se usa para obtener un token de acceso renovado sin obligar a los usuarios a volver a iniciar sesión." cta="Ver glosario" href="/docs/es/glossary?term=Refresh+Token">token de actualización</Tooltip>. Se considera la opción más segura, ya que el token de acceso se envía directamente al servidor web que aloja al cliente, sin pasar por el navegador del usuario ni exponerse.

Si este caso se ajusta a sus necesidades, para saber cómo funciona este flujo y cómo implementarlo, consulte [flujo de código de autorización](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow).

<div id="is-the-client-absolutely-trusted-with-user-credentials">
  ## ¿Se confía plenamente en el cliente para gestionar las credenciales del usuario?
</div>

Este punto de decisión puede dar lugar a la concesión de credenciales de contraseña del propietario del recurso. En este flujo, se pide al usuario final que introduzca sus credenciales (identifier/password), normalmente mediante un formulario interactivo. Esta información se envía al backend y, desde allí, a Auth0. Por lo tanto, es fundamental que se confíe plenamente en el cliente para manejar esta información.

Esta concesión solo debe usarse cuando no sea posible utilizar flujos basados en redirección (como el [flujo de código de autorización](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow)). Si este es tu caso, para obtener más información sobre cómo funciona este flujo y cómo implementarlo, consulta [flujo de contraseña del propietario del recurso](/docs/es/get-started/authentication-and-authorization-flow/resource-owner-password-flow).

<div id="is-the-client-a-single-page-app">
  ## ¿El cliente es una aplicación web de página única?
</div>

Si el cliente es una aplicación web de página única (SPA), es decir, una aplicación que se ejecuta en un navegador mediante un lenguaje de scripting como JavaScript, existen dos opciones de concesión: el flujo de código de autorización con PKCE y el flujo implícito con Form Post. En la mayoría de los casos, recomendamos usar el flujo de código de autorización con PKCE porque el token de acceso no queda expuesto del lado del cliente y este flujo puede devolver [tokens de actualización](/docs/es/secure/tokens/refresh-tokens).

Para obtener más información sobre cómo funciona este flujo y cómo implementarlo, consulta [flujo de código de autorización con PKCE](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce). El [SDK de Auth0 para aplicaciones de página única](/docs/es/libraries/auth0-single-page-app-sdk) proporciona una API de alto nivel para implementar el flujo de código de autorización con PKCE en aplicaciones SPA.

Si tu SPA no necesita un token de acceso, puedes usar el flujo implícito con Form Post. Para obtener más información sobre cómo funciona este flujo y cómo implementarlo, consulta [Implicit Flow with Form Post](/docs/es/get-started/authentication-and-authorization-flow/implicit-flow-with-form-post).

<div id="is-the-client-a-nativemobile-app">
  ## ¿El cliente es una aplicación nativa/móvil?
</div>

Si la aplicación es una aplicación nativa, usa el **flujo de código de autorización con PKCE**.

Para obtener más información sobre cómo funciona este flujo y cómo implementarlo, consulta [flujo de código de autorización con PKCE](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce).

<div id="i-have-an-application-that-needs-to-talk-to-different-resource-servers">
  ## Tengo una aplicación que necesita comunicarse con distintos servidores de recursos
</div>

Si una sola aplicación necesita tokens de acceso para distintos servidores de recursos, hay que realizar varias llamadas a `/authorize` (es decir, varias ejecuciones del mismo o de distintos <Tooltip tip="Flujo de autorización: concesión de autorización (o workflow) especificada en el framework de OAuth 2.0." cta="Ver glosario" href="/docs/es/glossary?term=Authorization+Flow">Flujo de autorización</Tooltip>). Cada autorización usará un valor distinto para `audience`, lo que dará como resultado un token de acceso diferente al final del flujo. Para obtener más información, consulta la [OAuth 2.0: Audience Information Specification](https://tools.ietf.org/html/draft-tschofenig-oauth-audience-00#section-3).

<div id="can-i-try-the-endpoints-before-i-implement-my-application">
  ## ¿Puedo probar los endpoints antes de implementar mi aplicación?
</div>

¡Claro! Puedes usar nuestra [Authentication API Debugger Extension](/docs/es/customize/extensions/authentication-api-debugger-extension). Encontrarás instrucciones detalladas para cada endpoint de `/grant` en nuestra [Authentication API Reference](https://auth0.com/docs/api/authentication).

* Para el endpoint Authorize, ve a [Authorize Application](https://auth0.com/docs/api/authentication#authorize-application) y lee el párrafo &quot;Prueba este endpoint&quot; correspondiente a la concesión que quieras probar.
* Para el <Tooltip tip="Endpoint de token: endpoint del servidor de autorización que se utiliza para solicitar tokens mediante programación." cta="Ver glosario" href="/docs/es/glossary?term=Token+endpoint">endpoint de token</Tooltip>, ve a [Get Token](https://auth0.com/docs/api/authentication#get-token) y lee la sección &quot;Prueba este endpoint&quot; correspondiente a la concesión que quieras probar.

<div id="does-the-client-application-need-to-challenge-users-for-authentication-without-browser-interaction">
  ## ¿La aplicación cliente necesita pedir a los usuarios que se autentiquen sin interacción en el navegador?
</div>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Client-Initiated Backchannel Authentication se encuentra actualmente en acceso anticipado. Para habilitar CIBA, póngase en contacto con su Technical Account Manager.
</Callout>

Client-Initiated Backchannel Authentication (CIBA) es un estándar de la <Tooltip tip="OpenID: estándar abierto de autenticación que permite a las aplicaciones verificar la identidad de los usuarios sin recopilar ni almacenar información de inicio de sesión." cta="Ver glosario" href="/docs/es/glossary?term=OpenID">OpenID</Tooltip> Foundation para implementar un flujo de autenticación alternativo a [OpenID Connect](https://openid.net/developers/how-connect-works/). CIBA se diferencia del flujo estándar de OpenID Connect en que:

* La aplicación cliente inicia el proceso de autenticación en nombre del usuario final.
* No se necesita un navegador para la interacción con el usuario.
* Hay comunicación directa entre la aplicación cliente y el proveedor de OpenID.

CIBA resulta útil cuando el usuario no confía en la aplicación cliente, la aplicación cliente no tiene navegador o el usuario no está frente a la aplicación que requiere autenticación. Estos son algunos ejemplos de casos en los que se pueden usar flujos CIBA:

* **Registro de usuarios en una terminal de punto de venta:** En escenarios de compra en línea con recogida en tienda, un usuario puede autenticarse en un quiosco público y confirmar su presencia.
* **Autenticación en un centro de llamadas o en el mostrador de un empleado:** Un agente de un centro de llamadas puede iniciar un flujo de autenticación para autenticar a la persona que llama, normalmente mediante una aplicación móvil personalizada en un smartphone.
* **Autenticación en un dispositivo sin entrada de datos:** Por ejemplo, un altavoz inteligente (u otro dispositivo conectado) puede usar un servicio de backend para contactar al usuario para autenticarse, normalmente mediante una aplicación móvil personalizada en un smartphone.

CIBA define dos dispositivos:

* **Consumption device**: El dispositivo que permite al usuario consumir un servicio.
* **Authentication device**: El dispositivo en el que el usuario se autenticará y otorgará consentimiento.

Para obtener más información, lea [Client-Initiated Backchannel Authentication Flow](/docs/es/get-started/authentication-and-authorization-flow/client-initiated-backchannel-authentication-flow).
