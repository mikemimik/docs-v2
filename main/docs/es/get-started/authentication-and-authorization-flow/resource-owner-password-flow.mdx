<Warning>
  Dado que el flujo de contraseña del propietario del recurso (ROP) implica que la aplicación gestione la contraseña del usuario, no debe utilizarse con clientes de terceros.
</Warning>

Aunque no lo recomendamos, las aplicaciones de alta confianza pueden usar el flujo de contraseña del <Tooltip tip="Propietario del recurso: Entidad (como un usuario o una aplicación) capaz de otorgar acceso a un recurso protegido." cta="Ver glosario" href="/docs/es/glossary?term=Resource+Owner">propietario del recurso</Tooltip> (definido en [OAuth 2.0 RFC 6749, sección 4.3](https://tools.ietf.org/html/rfc6749#section-4.3) y a veces denominado Resource Owner Password Grant o ROPG), que solicita a los usuarios que proporcionen credenciales (username/email/phone y contraseña), normalmente mediante un formulario interactivo. Como las credenciales se envían al backend y pueden almacenarse para su uso posterior antes de intercambiarse por un <Tooltip tip="Token de acceso: Credencial de autorización, en forma de una cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=Access+Token">token de acceso</Tooltip>, es imprescindible que la aplicación sea completamente de confianza para manejar esta información.

Incluso si se cumple esta condición, el flujo de contraseña del propietario del recurso solo debe usarse cuando no sea posible utilizar flujos basados en redirección (como el [flujo de código de autorización](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow)).

<div id="how-it-works">
  ## Cómo funciona
</div>

<Frame>![Diagrama: flujo de contraseña del propietario del recurso](/docs/images/cdy7uua7fh8z/4EeYNcnVX1RFcTy5z4lP4v/c3e4d22e6f8bf558caf07338a7388097/ROP_Grant.png)</Frame>

1. El usuario hace clic en **Login** dentro de la aplicación e introduce sus credenciales.
2. Tu aplicación envía las credenciales del usuario a tu servidor de autorización de Auth0 ([endpoint `/oauth/token`](https://auth0.com/docs/api/authentication#resource-owner-password)).
3. Tu servidor de autorización de Auth0 valida las credenciales.
4. Tu servidor de autorización de Auth0 responde con un token de acceso (y, opcionalmente, un token de actualización).
5. Tu aplicación puede usar el token de acceso para llamar a una API y acceder a información del usuario.
6. La API responde con los datos solicitados.

<div id="how-to-implement-it">
  ## Cómo implementarlo
</div>

La forma más sencilla de implementar el flujo de contraseña del propietario del recurso es seguir nuestro tutorial sobre cómo usar los endpoints de nuestra API para [llamar a tu API mediante el flujo de contraseña del propietario del recurso](/docs/es/get-started/authentication-and-authorization-flow/resource-owner-password-flow/call-your-api-using-resource-owner-password-flow).

<div id="realm-support">
  ## Compatibilidad con realm
</div>

Auth0 proporciona una concesión de extensión que ofrece una funcionalidad similar a la concesión de contraseña del propietario del recurso, pero le permite mantener directorios de usuarios separados (que corresponden a conexiones independientes) y especificar cuál usar durante el flujo.

Por ejemplo, supongamos que desea mostrar un menú desplegable en la interfaz de usuario de inicio de sesión de su aplicación para que los usuarios puedan elegir su tipo de usuario: `Employees` o `Customers`. En este caso, configuraría `Employees` y `Customers` como realms (y establecería una conexión correspondiente para cada uno), lo que permite mantener las credenciales de empleados y clientes en directorios de usuarios separados. Cuando solicite un token, enviará el valor de realm junto con las credenciales del usuario, y el realm enviado se utilizará para verificar la contraseña.

Para obtener más información sobre cómo implementar esta concesión de extensión, consulte [Llamar a su API con el flujo de contraseña del propietario del recurso: configurar la compatibilidad con realm](/docs/es/get-started/authentication-and-authorization-flow/resource-owner-password-flow/call-your-api-using-resource-owner-password-flow).

<div id="rules">
  ## Rules
</div>

Rules se ejecutará para el flujo de contraseña del propietario del recurso (incluida la concesión de la extensión Realm). Sin embargo, las reglas de redirección no funcionarán. Si intentas realizar una redirección especificando `context.redirect` en tu rule, el flujo de autenticación devolverá un error. Para obtener más información sobre Rules, lee [Auth0 Rules](/docs/es/customize/rules). Para obtener más información sobre las reglas de redirección, lee [Redirigir a los usuarios desde Rules](/docs/es/customize/rules/redirect-users).

<div id="mfa-support">
  ## Soporte para MFA
</div>

Si necesitas usar el flujo de contraseña del propietario del recurso, pero requieres una autenticación más robusta, puedes agregar <Tooltip tip="Autenticación multifactor (MFA): proceso de autenticación de usuario que utiliza un factor además de nombre de usuario y contraseña, como un código por SMS." cta="Ver glosario" href="/docs/es/glossary?term=multi-factor+authentication">autenticación multifactor</Tooltip> (MFA). Para saber cómo hacerlo, consulta [Autenticarse mediante el flujo de contraseña del propietario del recurso con MFA](/docs/es/secure/multi-factor-authentication/authenticate-using-ropg-flow-with-mfa).

<div id="attack-protection">
  ## Protección contra ataques
</div>

Al usar el flujo de contraseña del propietario del recurso con <Tooltip tip="Protección contra ataques de fuerza bruta: tipo de protección contra ataques que protege frente a ataques de fuerza bruta provenientes de una sola dirección IP y dirigidos a una sola cuenta de usuario." cta="Ver glosario" href="/docs/es/glossary?term=brute-force+protection">protección contra ataques de fuerza bruta</Tooltip>, es posible que algunas funcionalidades de <Tooltip tip="Protección contra ataques de fuerza bruta: tipo de protección contra ataques que protege frente a ataques de fuerza bruta provenientes de una sola dirección IP y dirigidos a una sola cuenta de usuario." cta="Ver glosario" href="/docs/es/glossary?term=attack+protection">protección contra ataques</Tooltip> no funcionen correctamente. Sin embargo, es posible evitar algunos problemas comunes. Para obtener más información, lee [Cómo evitar problemas comunes con el flujo de contraseña del propietario del recurso y la protección contra ataques](/docs/es/get-started/authentication-and-authorization-flow/resource-owner-password-flow/avoid-common-issues-with-resource-owner-password-flow-and-attack-protection).
