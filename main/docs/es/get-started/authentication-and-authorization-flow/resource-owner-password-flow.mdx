<Warning>
  Debido a que el flujo de contraseña del propietario del recurso (ROP) implica que la aplicación gestione la contraseña del usuario, no debe utilizarse en clientes de terceros.
</Warning>

Aunque no lo recomendamos, las aplicaciones de alta confianza pueden usar el flujo de <Tooltip tip="Propietario del recurso: Entidad (como un usuario o una aplicación) capaz de conceder acceso a un recurso protegido." cta="Ver glosario" href="/docs/es/glossary?term=Resource+Owner">propietario del recurso</Tooltip> y contraseña (definido en [OAuth 2.0 RFC 6749, sección 4.3](https://tools.ietf.org/html/rfc6749#section-4.3) y a veces denominado concesión de contraseña del propietario del recurso o ROPG), que solicita a los usuarios que proporcionen credenciales (username/email/phone y contraseña), normalmente mediante un formulario interactivo. Dado que las credenciales se envían al backend y pueden almacenarse para usarlas más adelante antes de intercambiarse por un <Tooltip tip="Token de acceso: Credencial de autorización, en forma de una cadena opaque o un JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=Access+Token">token de acceso</Tooltip>, es imprescindible que la aplicación sea absolutamente confiable para manejar esta información.

Incluso si se cumple esta condición, el flujo de contraseña del propietario del recurso solo debe usarse cuando no sea posible utilizar flujos basados en redirección (como el [flujo de código de autorización](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow)).

<div id="how-it-works">
  ## Cómo funciona
</div>

<Frame>![Diagrama: flujo de contraseña del propietario del recurso](/docs/images/cdy7uua7fh8z/4EeYNcnVX1RFcTy5z4lP4v/c3e4d22e6f8bf558caf07338a7388097/ROP_Grant.png)</Frame>

1. El usuario hace clic en **Login** en la aplicación e introduce sus credenciales.
2. Tu aplicación envía las credenciales del usuario a tu servidor de autorización de Auth0 ([endpoint `/oauth/token`](https://auth0.com/docs/api/authentication#resource-owner-password)).
3. Tu servidor de autorización de Auth0 valida las credenciales.
4. Tu servidor de autorización de Auth0 responde con un token de acceso (y, opcionalmente, un token de actualización).
5. Tu aplicación puede usar el token de acceso para llamar a una API y acceder a información sobre el usuario.
6. La API responde con los datos solicitados.

<div id="how-to-implement-it">
  ## Cómo implementarlo
</div>

La forma más sencilla de implementar el flujo de contraseña del propietario del recurso es seguir nuestro tutorial para usar nuestros endpoints de la API y [llamar a tu API mediante el flujo de contraseña del propietario del recurso](/docs/es/get-started/authentication-and-authorization-flow/resource-owner-password-flow/call-your-api-using-resource-owner-password-flow).

<div id="realm-support">
  ## Compatibilidad con realm
</div>

Auth0 proporciona una concesión de extensión que ofrece una funcionalidad similar a la de la concesión de contraseña del propietario del recurso, pero te permite mantener directorios de usuarios separados (que se corresponden con conexiones independientes) y especificar cuál usar durante el flujo.

Por ejemplo, supongamos que quieres mostrar un menú desplegable en la interfaz de usuario de inicio de sesión de tu aplicación para que los usuarios puedan elegir su tipo de usuario: `Employees` o `Customers`. En este caso, configurarías `Employees` y `Customers` como realms (y establecerías una conexión correspondiente para cada uno), lo que permite mantener las credenciales de empleados y clientes en directorios de usuarios separados. Cuando solicites un token, enviarás el valor de realm junto con las credenciales del usuario, y el realm enviado se utilizará para verificar la contraseña.

Para obtener más información sobre cómo implementar esta concesión de extensión, consulta [Call Your API Using Resource Owner Password Flow: Configure Realm Support](/docs/es/get-started/authentication-and-authorization-flow/resource-owner-password-flow/call-your-api-using-resource-owner-password-flow).

<div id="rules">
  ## Rules
</div>

Rules se ejecutará para el flujo de contraseña del propietario del recurso (incluida la concesión de extensión de realm). Sin embargo, las Rules de redirección no funcionarán. Si intenta realizar una redirección especificando `context.redirect` en su rule, el flujo de autenticación devolverá un error. Para obtener más información sobre Rules, lea [Auth0 Rules](/docs/es/customize/rules). Para obtener más información sobre las Rules de redirección, lea [Redirigir usuarios desde Rules](/docs/es/customize/rules/redirect-users).

<div id="mfa-support">
  ## Compatibilidad con MFA
</div>

Si necesita usar el flujo de contraseña del propietario del recurso, pero requiere una autenticación más robusta, puede agregar <Tooltip tip="Autenticación multifactor (MFA): proceso de autenticación de usuario que utiliza un factor adicional al nombre de usuario y la contraseña, como un código por SMS." cta="Ver glosario" href="/docs/es/glossary?term=multi-factor+authentication">autenticación multifactor</Tooltip> (MFA). Para obtener más información, lea [Autenticarse mediante el flujo de contraseña del propietario del recurso con MFA](/docs/es/secure/multi-factor-authentication/authenticate-using-ropg-flow-with-mfa).

<div id="attack-protection">
  ## Protección contra ataques
</div>

Al usar el flujo de contraseña del propietario del recurso con <Tooltip tip="Protección contra ataques de fuerza bruta: forma de protección contra ataques que protege frente a ataques de fuerza bruta que se originan desde una única dirección IP y se dirigen a una sola cuenta de usuario." cta="Ver glosario" href="/docs/es/glossary?term=brute-force+protection">protección contra ataques de fuerza bruta</Tooltip>, es posible que algunas funciones de <Tooltip tip="Protección contra ataques de fuerza bruta: forma de protección contra ataques que protege frente a ataques de fuerza bruta que se originan desde una única dirección IP y se dirigen a una sola cuenta de usuario." cta="Ver glosario" href="/docs/es/glossary?term=attack+protection">protección contra ataques</Tooltip> fallen. Sin embargo, se pueden evitar algunos problemas habituales. Para obtener más información, consulta [Evitar problemas habituales con el flujo de contraseña del propietario del recurso y la protección contra ataques](/docs/es/get-started/authentication-and-authorization-flow/resource-owner-password-flow/avoid-common-issues-with-resource-owner-password-flow-and-attack-protection).
