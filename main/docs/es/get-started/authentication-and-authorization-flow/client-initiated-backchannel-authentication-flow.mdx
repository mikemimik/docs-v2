<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Para usar las funciones de Client-Initiated Backchannel Authentication (CIBA), debes tener un Enterprise Plan o un add-on adecuado. Consulta [Auth0 Pricing](https://auth0.com/pricing/) para obtener más información.
</Callout>

El flujo Client-Initiated Backchannel Authentication (CIBA) es un estándar de la <Tooltip tip="OpenID: Estándar abierto de autenticación que permite a las aplicaciones verificar la identidad de los usuarios sin recopilar ni almacenar información de inicio de sesión." cta="Ver glosario" href="/docs/es/glossary?term=OpenID">OpenID</Tooltip> Foundation que permite una autenticación y autorización desacopladas mediante solicitudes seguras por backchannel. El flujo de CIBA está diseñado para casos de uso en los que el dispositivo que inicia una solicitud (el dispositivo de consumo) es distinto del dispositivo que usa un usuario para autenticarse (el dispositivo de autenticación).

En un flujo de CIBA, hay dos actores:

* Usuario iniciador: La entidad que inicia la autenticación o la autorización en el dispositivo de consumo. Puede ser una aplicación de backend, un usuario humano, como un agente de atención al cliente en una aplicación de atención al cliente, o un agente de IA que realiza una tarea en nombre del usuario final.
* Usuario autorizador: El usuario final que recibe la solicitud de autenticación o autorización en el dispositivo de autenticación y otorga su consentimiento.

Al desacoplar la autenticación y la autorización del dispositivo de consumo, la aplicación cliente puede llamar directamente al proveedor de OpenID mediante una solicitud por backchannel. Esto proporciona mayor seguridad, ya que puedes usar métodos más seguros para la autenticación del cliente, como [mTLS](/docs/es/get-started/authentication-and-authorization-flow/authenticate-with-mtls) y [clave privada JWT](/docs/es/get-started/authentication-and-authorization-flow/authenticate-with-private-key-jwt), con el flujo de CIBA.

<div id="use-cases">
  ## Casos de uso
</div>

Entre los casos de uso habituales del flujo CIBA se incluyen:

* Un agente de IA utiliza aprobaciones con intervención humana para solicitar el permiso del usuario para completar una acción que este ha solicitado.
* Un usuario llama por teléfono a un centro de atención, y el operador que atiende la llamada quiere acceder a la información personal del usuario en su equipo. El usuario puede dar su consentimiento para ello, por ejemplo, aprobando una notificación push en su teléfono.
* Un usuario quiere acceder a un dispositivo con capacidades de entrada limitadas, como una bicicleta de alquiler en una ciudad o un quiosco en un comercio.
* Un usuario inicia una transacción confidencial en un dispositivo relativamente inseguro y quiere autorizarla desde un dispositivo más seguro. Por ejemplo, podría autorizar un pago o un cambio en sus datos personales tras recibir una notificación push en su teléfono móvil personal.

<div id="notification-channels">
  ## Canales de notificación
</div>

Auth0 admite los siguientes canales de notificación con CIBA:

* [notificaciones push en dispositivos móviles de Auth0 Guardian](/docs/es/get-started/authentication-and-authorization-flow/client-initiated-backchannel-authentication-flow/mobile-push-notifications-with-ciba): Disponibles para todos los [planes empresariales](https://auth0.com/pricing). El usuario recibe una notificación push en su dispositivo móvil registrado para autenticarse o confirmar los detalles de la transacción. La autenticación y autorización se realizan en el dispositivo de autenticación que recibe la notificación push. Puedes habilitar las notificaciones push en dispositivos móviles con CIBA mediante:
  * La aplicación Auth0 Guardian
  * Una aplicación Custom integrada con el Auth0 Guardian SDK
* [notificaciones por correo electrónico](/docs/es/get-started/authentication-and-authorization-flow/client-initiated-backchannel-authentication-flow/email-notifications-with-ciba): Disponibles como complemento para [planes de pago](https://auth0.com/pricing). El usuario recibe un correo electrónico en su dirección de correo electrónico verificada. El correo contiene un enlace, y la autenticación y autorización se realizan a través del navegador.

De forma predeterminada, Auth0 usa y recomienda las notificaciones push de Guardian para los flujos de CIBA. Las notificaciones push de Guardian son más seguras que otros canales, como el correo electrónico, que puede ser vulnerable a ataques de phishing. Debes habilitar explícitamente las notificaciones por correo electrónico para los flujos de CIBA.

<div id="how-it-works">
  ## Cómo funciona
</div>

El siguiente diagrama describe el flujo CIBA de extremo a extremo:

<Frame>![](/docs/images/ciba/ciba_end_to_end_diagram.png)</Frame>

1. La aplicación cliente o el dispositivo de consumo solicita la autenticación o autorización del usuario.
2. El backend de la aplicación cliente envía una solicitud `POST` al endpoint `/bc-authorize`.
3. Auth0 recibe la solicitud `POST` y envía una notificación al dispositivo de autenticación.
4. El dispositivo de autenticación recupera de Auth0 los detalles de autorización y se los presenta al usuario final.
5. El usuario final revisa la solicitud, incluidos los detalles de autorización contextuales asociados a ella.
6. El usuario final proporciona su respuesta en el dispositivo de autenticación, que la envía a Auth0.
7. El backend de la aplicación cliente consulta el endpoint `/token` y recibe los tokens correspondientes cuando se completa el flujo CIBA.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Como el flujo CIBA se usa para la autenticación y autorización asíncronas del usuario en una sola ocasión, CIBA no crea ni almacena una concesión que contenga el consentimiento del usuario para que una aplicación acceda a los recursos de una API. Si más adelante el usuario se autentica mediante un flujo de autenticación diferente, y ese flujo solicita los mismos scopes para los que el usuario ya había dado su consentimiento mediante CIBA, Auth0 no tendrá ningún registro de ese consentimiento. Por lo tanto, volverá a pedirle al usuario que dé su consentimiento.
</Callout>

<div id="entity-limits">
  ## Límites de entidades
</div>

El flujo CIBA tiene los siguientes límites:

* Se pueden crear hasta 500 solicitudes CIBA por minuto y por tenant.
* Puede haber hasta 5000 solicitudes CIBA pendientes por tenant en un momento dado. Una solicitud CIBA pendiente es una solicitud que se ha iniciado, pero que aún no ha recibido respuesta del usuario. Si una solicitud CIBA vence sin recibir respuesta, puede seguir contabilizándose para el límite de 5000 durante un máximo de 24 horas.
