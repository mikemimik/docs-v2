title: Flujo de autorización de dispositivos
---

En los dispositivos con limitación de entrada que se conectan a internet, en lugar de autenticar directamente al usuario, el dispositivo le pide que vaya a un enlace en su computadora o smartphone y autorice el dispositivo. Esto evita una mala experiencia de usuario en dispositivos que no cuentan con una forma sencilla de introducir texto. Para ello, las aplicaciones del dispositivo usan el <Tooltip tip="Flujo de autorización: concesión de autorización (o workflow) especificada en el framework OAuth 2.0." cta="Ver glosario" href="/docs/es/glossary?term=Authorization+Flow">flujo de autorización</Tooltip> de dispositivos (ratificado en [OAuth 2.0](https://tools.ietf.org/html/rfc8628)), en el que envían su <Tooltip tip="ID de cliente: Valor de identificación dado a su recurso registrado por Auth0." cta="Ver glosario" href="/docs/es/glossary?term=Client+ID">ID de cliente</Tooltip> para iniciar el proceso de autorización y obtener un token.

<div id="how-it-works">
  ## Cómo funciona
</div>

El Flujo de autorización de dispositivos consta de dos rutas diferentes: una ocurre en el dispositivo que solicita la autorización y la otra, en un navegador. La ruta del flujo en el navegador, en la que un código de dispositivo se vincula a la sesión del navegador, ocurre en paralelo con parte de la ruta del flujo del dispositivo.

<div id="device-flow">
  ### Flujo del dispositivo
</div>

1. El usuario inicia la aplicación en el dispositivo.
2. La aplicación del dispositivo solicita autorización al servidor de autorización de Auth0 usando su ID de cliente (endpoint `/oauth/device/code`).
3. El servidor de autorización de Auth0 responde con un `device_code`, `user_code`, `verification_uri`, `verification_uri_complete`, `expires_in` (tiempo de vida en segundos de `device_code` y `user_code`) y el `interval` de sondeo.
4. La aplicación del dispositivo le pide al usuario que lo active usando su computadora o smartphone. La aplicación puede hacerlo de estas maneras:

   * pedirle al usuario que visite la `verification_uri` e introduzca el `user_code` después de mostrar estos valores en pantalla
   * pedirle al usuario que interactúe con un código QR o una URL acortada con el código de usuario integrado, generada a partir de `verification_uri_complete`
   * ir directamente a la página de verificación con el código de usuario integrado usando `verification_uri_complete`, si se ejecuta de forma nativa en un dispositivo basado en navegador
5. La aplicación del dispositivo comienza a consultar tu servidor de autorización de Auth0 para obtener un token de acceso (endpoint **/oauth/token**) con la frecuencia especificada por `interval`, contando desde que recibe la respuesta a la última solicitud de sondeo. La aplicación del dispositivo sigue consultando hasta que el usuario complete el flujo en el navegador o el código de usuario expire.
6. Cuando el usuario completa correctamente el flujo en el navegador, tu servidor de autorización de Auth0 responde con un token de acceso (y, opcionalmente, un token de actualización). La aplicación del dispositivo debe olvidarse entonces de su `device_code`, ya que expirará.
7. Tu aplicación del dispositivo puede usar el token de acceso para llamar a una API y acceder a información sobre el usuario.
8. La API responde con los datos solicitados.

<div id="browser-flow">
  ### Flujo del navegador
</div>

1. El usuario visita el `verification_uri` en su computadora, introduce el `user_code` y confirma que el dispositivo que se está activando muestra el `user_code`. Si el usuario visita el `verification_uri_complete` por cualquier otro medio (por ejemplo, escaneando un código QR), solo será necesaria la confirmación del dispositivo.
2. Tu servidor de autorización de Auth0 redirige al usuario al prompt de inicio de sesión y a la solicitud de consentimiento, si es necesario.
3. El usuario se autentica con una de las opciones de inicio de sesión configuradas y puede ver una página de consentimiento en la que se le pide que autorice la aplicación del dispositivo.
4. Tu aplicación del dispositivo queda autorizada para acceder a la API.

<div id="force-device-reauthorization">
  ## Forzar la reautorización del dispositivo
</div>

Para forzar al usuario a reautorizar un dispositivo, debes revocar el <Tooltip tip="token de actualización: Token utilizado para obtener un nuevo token de acceso sin obligar a los usuarios a volver a iniciar sesión." cta="Ver glosario" href="/docs/es/glossary?term=Refresh+Token">token de actualización</Tooltip> asignado al dispositivo. Para saber cómo hacerlo, consulta [Desvincular dispositivos de los usuarios](/docs/es/manage-users/user-accounts/unlink-devices-from-users). Ten en cuenta que no se obligará al dispositivo a reautorizarse hasta que el <Tooltip tip="Token de acceso: Credencial de autorización, en forma de cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=Access+Token">token de acceso</Tooltip> actual caduque y la aplicación intente usar el token de actualización revocado. Para obtener más información sobre los tokens de actualización, consulta [Tokens de actualización](/docs/es/secure/tokens/refresh-tokens).
