<Warning>
  La fecha de fin de vida útil (EOL) de Rules y Hooks será el **18 de noviembre de 2026**, y ya no están disponibles para los nuevos tenants creados a partir del **16 de octubre de 2023**. Los tenants existentes con Hooks activos conservarán el acceso al producto Hooks hasta el fin de vida útil.

  Te recomendamos encarecidamente que uses Actions para ampliar Auth0. Con Actions, tienes acceso a información de tipos detallada, documentación integrada y paquetes `npm` públicos, y puedes conectar integraciones externas que mejoran tu experiencia general de extensibilidad. Para obtener más información sobre lo que ofrece Actions, consulta [Understand How Auth0 Actions Work](/docs/es/customize/actions/actions-overview).

  Para ayudarte con la migración, ofrecemos guías que te ayudarán a [migrar de Rules a Actions](/docs/es/customize/actions/migrate/migrate-from-rules-to-actions) y [migrar de Hooks a Actions](/docs/es/customize/actions/migrate/migrate-from-hooks-to-actions). También tenemos una página específica de [Move to Actions](https://auth0.com/extensibility/movetoactions) que destaca comparaciones de funcionalidades, [una demostración de Actions](https://www.youtube.com/watch?v=UesFSY1klrI) y otros recursos para ayudarte en tu proceso de migración.

  Para obtener más información sobre la obsolescencia de Rules y Hooks, consulta nuestra entrada de blog: [Preparing for Rules and Hooks End of Life](https://auth0.com/blog/preparing-for-rules-and-hooks-end-of-life/).
</Warning>

En el punto de extensibilidad Intercambio de credenciales de cliente, los Hooks te permiten ejecutar acciones personalizadas cuando se emite un <Tooltip tip="Token de acceso: credencial de autorización, en forma de una cadena opaca o JWT, que se utiliza para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=Access+Token">token de acceso</Tooltip> a través del endpoint [`POST /oauth/token`](https://auth0.com/docs/api/authentication#client-credentials-flow) de la Authentication API mediante el Client Credentials Flow. Por ejemplo, puedes impedir que se emita el token, añadir custom claims al token de acceso o modificar sus scopes. Para obtener más información, consulta [Client Credentials Flow](/docs/es/get-started/authentication-and-authorization-flow/client-credentials-flow).

Los Hooks en este punto de extensibilidad son bloqueantes (sincrónicos), lo que significa que se ejecutan como parte del proceso del trigger y evitan que el resto del pipeline de Auth0 se ejecute hasta que el Hook se haya completado.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  El `triggerId` del punto de extensibilidad Intercambio de credenciales de cliente es `credentials-exchange`. Para aprender a crear hooks para este punto de extensibilidad, consulta [Create Hooks](/docs/es/customize/hooks/create-hooks).
</Callout>

Para obtener información sobre otros puntos de extensibilidad, consulta Extensibility Points.

<div id="starter-code-and-parameters">
  ## Código base y parámetros
</div>

Al crear un Hook que se ejecuta en el punto de extensibilidad Intercambio de credenciales de cliente, puede resultarle útil el siguiente código base. Los parámetros que pueden pasarse a la función Hook y utilizarse en ella se enumeran en la parte superior del ejemplo de código.

```javascript lines expandable
/**
@param {object} client - información del cliente
@param {string} client.name - nombre del cliente
@param {string} client.id - ID de cliente
@param {string} client.tenant - nombre del tenant de Auth0
@param {object} client.metadata - metadatos del cliente
@param {array|undefined} scope - un arreglo de strings que representa el claim de scope del token, o undefined
@param {string} audience - claim de audience del token
@param {object} context - información de contexto de Auth0
@param {object} context.webtask - contexto del Hook (webtask)
@param {function} cb - function (error, accessTokenClaims)
*/

module.exports = function(client, scope, audience, context, cb) {
  var access_token = {};
  access_token.scope = scope; // no eliminar esta línea

  // Modificar scopes o agregar claims adicionales
  // access_token['https://example.com/claim'] = 'bar';
  // access_token.scope.push('extra');

  // Denegar el token y responder con una respuesta de error de OAuth2
  // if (denyExchange) {
  //   // Para devolver un HTTP 400 con { "error": "invalid_scope", "error_description": "Not authorized for this scope." }
  //   return cb(new InvalidScopeError('Not authorized for this scope.'));
  //
  //   // Para devolver un HTTP 400 con { "error": "invalid_request", "error_description": "Not a valid request." }
  //   return cb(new InvalidRequestError('Not a valid request.'));
  //
  //   // Para devolver un HTTP 500 con { "error": "server_error", "error_description": "A server error occurred." }
  //   return cb(new ServerError('A server error occurred.'));
  // }

  cb(null, access_token);
};
```

Tenga en cuenta lo siguiente:

* La `función de devolución de llamada` (`cb`) al final del código de ejemplo indica que la ejecución ha finalizado y debe incluirse.

* La línea `access_token.scope = scope` garantiza que todos los `scopes` concedidos estén presentes en el `token de acceso`. Si la elimina, se restablecerán todos los `scopes` y el `token` incluirá solo los `scopes` que agregue con el `script`.

<div id="default-response">
  ### Respuesta predeterminada
</div>

Cuando ejecutas un Hook en el punto de extensibilidad Intercambio de credenciales de cliente, el objeto de respuesta predeterminado es:

```json lines
{
  "scope": "array of strings"
}
```

<div id="starter-code-response">
  ### Respuesta del código base
</div>

Una vez que hayas personalizado el código base con tus scopes y claims adicionales, puedes probar el Hook con el runner integrado en el Hook Editor. El runner simula una llamada al Hook con el mismo cuerpo y la misma respuesta que obtendrías con un Intercambio de credenciales de cliente.

<Warning>
  Para ejecutar el código con el runner, primero debes guardar, lo que significa que el código original se sobrescribirá.
</Warning>

Cuando ejecutas un Hook basado en el código base, el objeto de respuesta es:

```json lines

```

<div id="sample-script-add-an-additional-scope-to-the-access-token">
  ## Script de ejemplo: Añadir un `scope` adicional al `access token`
</div>

En este ejemplo, usamos un Hook para añadir un `scope` adicional a los que ya existen en el `access token`.

```js lines
module.exports = function(client, scope, audience, context, cb) {
    // Scopes a agregar
    var access_token = {};

    // Obtener el scope que está actualmente en el Access Token
    // y agregarlo al objeto con el que estamos trabajando
    // ¡No elimines esta línea!
    access_token.scope = scope;

    // Agregar el scope `read:resource`
    access_token.scope.push('read:resource');

    // Devolución de llamada para indicar la finalización y retornar el nuevo
    // arreglo de scopes
    cb(null, access_token);
};
```

Para obtener más información, consulta [Scopes](/docs/es/get-started/apis/scopes).

<div id="response">
  ### Respuesta
</div>

Al ejecutar este Hook, el objeto de respuesta es:

```json lines
{
  "scope": [
    "read:connections",
    "read:resource"
  ]
}
```

<div id="sample-script-add-a-claim-to-the-access-token">
  ## Script de ejemplo: Añadir un claim al token de acceso
</div>

En este ejemplo, añadimos un custom claim con espacio de nombres y su valor al token de acceso. Para obtener más información, consulta [Crear custom claims con espacio de nombres](/docs/es/secure/tokens/json-web-tokens/create-custom-claims).

Puedes añadir lo siguiente como claims al token emitido:

* La propiedad `scope` del objeto de respuesta
* Cualquier propiedad con nombres de propiedad con espacio de nombres

El punto de extensibilidad ignora todas las demás propiedades del objeto de respuesta.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Para acceder a un Hook Secret configurado desde un hook, usa `context.webtask.secrets.SECRET_NAME`.
</Callout>

```js lines
module.exports = function(client, scope, audience, context, cb) {
    // Claims a agregar
    var access_token = {};

    // Nuevo claim a agregar al token
    access_token['https://example.com/foo'] = 'bar';

    // Devolución de llamada para indicar la finalización y devolver el nuevo claim
    cb(null, access_token);
  };
```

<div id="response">
  ### Respuesta
</div>

Al ejecutar este Hook, el objeto de respuesta es:

```json lines
{
  "https://example.com/foo": "bar"
}
```

<div id="sample-script-raise-an-error-or-deny-an-access-token">
  ## Script de ejemplo: Generar un error o denegar un token de acceso
</div>

En este ejemplo, usamos objetos Error personalizados para generar respuestas de error de OAuth2. (Para obtener más información, consulta [OAuth2 RFC - Sección 5.2 en el IETF Datatracker](https://tools.ietf.org/html/rfc6749#section-5.2).)

Si se devuelve un error estándar de JavaScript en la función callback, como:

```js lines
module.exports = function(client, scope, audience, context, cb) {
    // Devolución de llamada para indicar la finalización y devolver un nuevo claim
    cb(new Error("Unknown error occurred.");
  };
```

Luego, cuando solicites una concesión `client_credentials` al endpoint `/oauth/token`, Auth0 responderá con:

```json lines
HTTP 500
{ "error": "server_error", "error_description": "Unknown error occurred." }
```

Sin embargo, si desea un mayor control sobre la respuesta de error de OAuth2, tiene a su disposición tres objetos Error personalizados que puede usar en su lugar.

<div id="invalidscopeerror">
  ### InvalidScopeError
</div>

```js lines
module.exports = function(client, scope, audience, context, cb) {
    const invalidScope = ...; // determinar si el scope es válido

    if(invalidScope) {
      cb(new InvalidScopeError("Scope is not permitted."));
    }
  };
```

Entonces, cuando solicitas una concesión `client_credentials` al endpoint `/oauth/token`, Auth0 responde con:

```json lines
HTTP 400
{ "error": "invalid_scope", "error_description": "Scope is not permitted." }
```

<div id="invalidrequesterror">
  ### InvalidRequestError
</div>

```js lines
module.exports = function(client, scope, audience, context, cb) {
    const invalidRequest = ...; // determinar si la solicitud es válida

    if(invalidRequest) {
      cb(new InvalidRequestError("Bad request."));
    }
  };
```

Luego, cuando solicites una concesión `client_credentials` en el endpoint `/oauth/token`, Auth0 responderá con:

```json lines
HTTP 400
{ "error": "invalid_request", "error_description": "Bad request." }
```

<div id="servererror">
  ### ServerError
</div>

```js lines
module.exports = function(client, scope, audience, context, cb) {
    callOtherService(function(err, response) {
      if(err) {
        return cb(new ServerError("Error calling remote system: " + err.message));
      }
    });
  };
```

Después, cuando solicitas una concesión `client_credentials` en el endpoint `/oauth/token`, Auth0 responde con:

```json lines
HTTP 400
{ "error": "server_error", "error_description": "Error calling remote system: ..." }
```

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Actualmente, el comportamiento de la clase integrada `Error` de JavaScript y el de `ServerError` es idéntico, pero la clase `ServerError` te permite indicar explícitamente qué error de OAuth2 se devolverá.
</Callout>
