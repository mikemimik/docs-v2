<Card title="Información general">
  Conceptos clave

  * Conozca el tipo de concesión de OAuth 2.0: el flujo de código de autorización con Proof Key for Code Exchange (PKCE).
  * Use este tipo de concesión en aplicaciones que no pueden almacenar un secreto de cliente, como las aplicaciones nativas o aplicaciones web de página única.
  * Revise distintos métodos de implementación con Auth0 SDKs.
</Card>

Cuando los <Tooltip tip="Cliente público: cliente (aplicación) que no puede mantener credenciales de forma segura. Algunos ejemplos incluyen una aplicación nativa de escritorio o móvil y una aplicación web client-side basada en JavaScript (como una aplicación web de página única (SPA))." cta="Ver glosario" href="/docs/es/glossary?term=public+clients">clientes públicos</Tooltip> (p. ej., aplicaciones nativas y aplicaciones web de página única) solicitan <Tooltip tip="Cliente público: cliente (aplicación) que no puede mantener credenciales de forma segura. Algunos ejemplos incluyen una aplicación nativa de escritorio o móvil y una aplicación web client-side basada en JavaScript (como una aplicación web de página única (SPA))." cta="Ver glosario" href="/docs/es/glossary?term=access+tokens">tokens de acceso</Tooltip>, surgen algunos problemas de seguridad adicionales que el flujo de código de autorización, por sí solo, no mitiga. Esto se debe a que:

**Aplicaciones nativas**

* No pueden almacenar de forma segura un <Tooltip tip="Secreto de cliente: Secret usado por un cliente (aplicación) para autenticarse con el servidor de autorización; solo el cliente y el servidor de autorización deben conocerlo y debe ser lo suficientemente aleatorio como para que no pueda adivinarse." cta="Ver glosario" href="/docs/es/glossary?term=Client+Secret">secreto de cliente</Tooltip>. Al descompilar la aplicación, se revelará el <Tooltip tip="Secreto de cliente: Secret usado por un cliente (aplicación) para autenticarse con el servidor de autorización; solo el cliente y el servidor de autorización deben conocerlo y debe ser lo suficientemente aleatorio como para que no pueda adivinarse." cta="Ver glosario" href="/docs/es/glossary?term=Client+Secret">secreto de cliente</Tooltip>, que está vinculado a la aplicación y es el mismo para todos los usuarios y dispositivos.
* Son vulnerables a ataques de interceptación e inyección del código de autorización. Sin un secreto de cliente, un atacante que intercepte el código de autorización puede canjearlo por tokens.
* Pueden usar un esquema de URI personalizado para capturar redirecciones (p. ej., MyApp://), lo que puede permitir que aplicaciones maliciosas reciban un <Tooltip tip="Código de autorización: cadena aleatoria generada por el servidor de autorización y devuelta a la aplicación como parte de la respuesta de autorización al usar el flujo de código de autorización (ya sea con o sin PKCE)." cta="Ver glosario" href="/docs/es/glossary?term=Authorization+Code">código de autorización</Tooltip> de su <Tooltip tip="Servidor de autorización: servidor centralizado que contribuye a definir los límites del acceso de un usuario. Por ejemplo, su servidor de autorización puede controlar los datos, las tareas y las funciones disponibles para un usuario." cta="Ver glosario" href="/docs/es/glossary?term=Authorization+Server">servidor de autorización</Tooltip>. Debido a este riesgo, **Auth0 desaconseja encarecidamente el uso de esquemas de URI personalizados**. Para obtener más información, lea [Measures Against Application Impersonation](/docs/es/secure/security-guidance/measures-against-app-impersonation.mdx).

**Aplicaciones web de página única**

* No pueden almacenar de forma segura un secreto de cliente porque todo su código fuente está disponible en el navegador.

Dadas estas situaciones, <Tooltip tip="OAuth 2.0: marco de autorización que define protocolos y flujos de trabajo de autorización." cta="Ver glosario" href="/docs/es/glossary?term=OAuth+2.0">OAuth 2.0</Tooltip> proporciona una versión del flujo de código de autorización que utiliza Proof Key for Code Exchange (PKCE) (definida en [OAuth 2.0 RFC 7636](https://tools.ietf.org/html/rfc7636)).

El flujo de código de autorización mejorado con PKCE introduce un secreto creado por la aplicación que realiza la solicitud y que el <Tooltip tip="Servidor de autorización: servidor centralizado que contribuye a definir los límites del acceso de un usuario. Por ejemplo, su servidor de autorización puede controlar los datos, las tareas y las funciones disponibles para un usuario." cta="Ver glosario" href="/docs/es/glossary?term=authorization+server">servidor de autorización</Tooltip> puede verificar; este secreto se denomina Code Verifier. Además, la aplicación solicitante crea un valor derivado del Code Verifier llamado Code Challenge y envía este valor a través de HTTPS para obtener un código de autorización. De este modo, un atacante malicioso solo puede interceptar el código de autorización y no puede canjearlo por un token sin el Code Verifier.

<div id="how-it-works">
  ## Cómo funciona
</div>

Como el flujo de código de autorización mejorado con PKCE se basa en el [flujo de código de autorización estándar](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow), los pasos son muy similares.

<Frame>![Flows - Código de autorización con PKCE - Diagrama de secuencia de autorización](/docs/images/cdy7uua7fh8z/3pstjSYx3YNSiJQnwKZvm5/33c941faf2e0c434a9ab1f0f3a06e13a/auth-sequence-auth-code-pkce.png)</Frame>

1. El usuario hace clic en **Login** dentro de la aplicación.
2. El SDK de Auth0 crea un `code_verifier` aleatorio criptográficamente seguro y, a partir de este, genera un `code_challenge`.
3. El SDK de Auth0 redirige al usuario al servidor de autorización de Auth0 ([endpoint `/authorize`](https://auth0.com/docs/api/authentication#authorization-code-grant-pkce-)) junto con el `code_challenge`.
4. Tu servidor de autorización de Auth0 redirige al usuario al prompt de inicio de sesión y autorización.
5. El usuario se autentica con una de las opciones de inicio de sesión configuradas y puede ver una página de consentimiento con la lista de permisos que Auth0 otorgará a la aplicación.
6. Tu servidor de autorización de Auth0 almacena el `code_challenge` y redirige al usuario de vuelta a la aplicación con un `code` de autorización válido para un solo uso.
7. El SDK de Auth0 envía este `code` y el `code_verifier` (creado en el paso 2) al servidor de autorización de Auth0 `(`[`/oauth/token` endpoint](https://auth0.com/docs/api/authentication?http#authorization-code-flow-with-pkce44)).
8. Tu servidor de autorización de Auth0 verifica el `code_challenge` y el `code_verifier`.
9. Tu servidor de autorización de Auth0 responde con un token de identidad y un token de acceso (y, opcionalmente, un token de actualización).
10. Tu aplicación puede usar el token de acceso para llamar a una API y acceder a información sobre el usuario.
11. La API responde con los datos solicitados.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Si tienes habilitada la [rotación del Refresh Token](/docs/es/secure/tokens/refresh-tokens/refresh-token-rotation), se genera un nuevo token de actualización con cada solicitud y se emite junto con el token de acceso. Cuando se intercambia un token de actualización, el token de actualización anterior se invalida, pero el servidor de autorización conserva información sobre esa relación.
</Callout>

<div id="how-to-implement-it">
  ## Cómo implementarlo
</div>

La forma más sencilla de implementar el flujo de código de autorización con PKCE es [seguir nuestros Quickstarts nativos](/docs/es/quickstart/native) o [seguir nuestros Quickstarts para aplicaciones web de página única](/docs/es/quickstart/spa).

Según el tipo de aplicación, también puedes usar nuestros SDK para móviles o para aplicaciones web de página única:

**Móvil**

* [SDK de Auth0 para Swift](/docs/es/libraries/auth0-swift)
* [SDK de Auth0 para Android](/docs/es/libraries/auth0-android)

**Aplicación web de página única**

* [SDK de Auth0 para aplicaciones web de página única](/docs/es/libraries/auth0-single-page-app-sdk)
* [SDK de Auth0 para React](/docs/es/libraries/auth0-react)

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Los avances recientes en los controles de privacidad de los usuarios en los navegadores afectan negativamente la experiencia del usuario al impedir el acceso a cookies de terceros; por lo tanto, los flujos basados en navegador deben usar [rotación del Refresh Token](/docs/es/secure/tokens/refresh-tokens/refresh-token-rotation), que ofrece un método seguro para usar tokens de actualización en las aplicaciones web de página única y, al mismo tiempo, proporcionar a los usuarios finales acceso fluido a los recursos sin las interrupciones en la experiencia de usuario que provocan tecnologías de privacidad del navegador como ITP.
</Callout>

Puedes seguir nuestros tutoriales para usar nuestros endpoints de la API para [Agregar inicio de sesión con el flujo de código de autorización con PKCE](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce/add-login-using-the-authorization-code-flow-with-pkce) o [Llamar a tu API con el flujo de código de autorización con PKCE](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce/call-your-api-using-the-authorization-code-flow-with-pkce).
