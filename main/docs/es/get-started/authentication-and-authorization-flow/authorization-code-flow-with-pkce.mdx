<Card title="Información general">
  Conceptos clave

  * Aprende sobre el tipo de concesión de OAuth 2.0, Flujo de código de autorización con Proof Key for Code Exchange (PKCE).
  * Usa este tipo de concesión en aplicaciones que no pueden almacenar un secreto de cliente, como las aplicaciones nativas o de página única.
  * Revisa distintos métodos de implementación con los SDKs de Auth0.
</Card>

Cuando los <Tooltip tip="Cliente público: Cliente (aplicación) que no puede conservar credenciales de forma segura. Algunos ejemplos incluyen una aplicación nativa de escritorio o móvil y una aplicación web del lado del cliente basada en JavaScript (como una aplicación de página única (SPA))." cta="Ver glosario" href="/docs/es/glossary?term=public+clients">clientes públicos</Tooltip> (por ejemplo, aplicaciones nativas y de página única) solicitan <Tooltip tip="Cliente público: Cliente (aplicación) que no puede conservar credenciales de forma segura. Algunos ejemplos incluyen una aplicación nativa de escritorio o móvil y una aplicación web del lado del cliente basada en JavaScript (como una aplicación de página única (SPA))." cta="Ver glosario" href="/docs/es/glossary?term=access+tokens">tokens de acceso</Tooltip>, surgen algunas consideraciones de seguridad adicionales que el Flujo de código de autorización por sí solo no mitiga. Esto se debe a que:

**Aplicaciones nativas**

* No pueden almacenar de forma segura un <Tooltip tip="Secreto de cliente: Secret utilizado por un cliente (aplicación) para autenticarse con el servidor de autorización; solo deberían conocerlo el cliente y el servidor de autorización, y debe ser lo bastante aleatorio como para que no pueda adivinarse." cta="Ver glosario" href="/docs/es/glossary?term=Client+Secret">secreto de cliente</Tooltip>. Descompilar la aplicación revelará el <Tooltip tip="Secreto de cliente: Secret utilizado por un cliente (aplicación) para autenticarse con el servidor de autorización; solo deberían conocerlo el cliente y el servidor de autorización, y debe ser lo bastante aleatorio como para que no pueda adivinarse." cta="Ver glosario" href="/docs/es/glossary?term=Client+Secret">secreto de cliente</Tooltip>, que está vinculado a la aplicación y es el mismo para todos los usuarios y dispositivos.
* Son vulnerables a ataques de interceptación e inyección de códigos de autorización. Sin un secreto de cliente, un atacante que intercepte el código de autorización puede intercambiarlo por tokens.
* Pueden usar un esquema de URL personalizado para capturar redirecciones (por ejemplo, MyApp://), lo que podría permitir que aplicaciones maliciosas reciban un <Tooltip tip="Código de autorización: Cadena aleatoria generada por el servidor de autorización y devuelta a la aplicación como parte de la respuesta de autorización al usar el Flujo de código de autorización (ya sea con o sin PKCE)." cta="Ver glosario" href="/docs/es/glossary?term=Authorization+Code">código de autorización</Tooltip> de tu <Tooltip tip="Servidor de autorización: Servidor centralizado que contribuye a definir los límites del acceso de un usuario. Por ejemplo, tu servidor de autorización puede controlar los datos, las tareas y las funciones disponibles para un usuario." cta="Ver glosario" href="/docs/es/glossary?term=Authorization+Server">servidor de autorización</Tooltip>. Debido a este riesgo, **Auth0 desaconseja firmemente el uso de esquemas de URI personalizados**. Para obtener más información, lee [Measures Against Application Impersonation](/docs/es/secure/security-guidance/measures-against-app-impersonation.mdx).

**Aplicaciones de página única**

* No pueden almacenar de forma segura un secreto de cliente porque todo su código fuente está disponible en el navegador.

Dadas estas situaciones, <Tooltip tip="OAuth 2.0: Marco de autorización que define protocolos y flujos de trabajo de autorización." cta="Ver glosario" href="/docs/es/glossary?term=OAuth+2.0">OAuth 2.0</Tooltip> proporciona una versión del Flujo de código de autorización que utiliza Proof Key for Code Exchange (PKCE) (definido en [OAuth 2.0 RFC 7636](https://tools.ietf.org/html/rfc7636)).

El Flujo de código de autorización mejorado con PKCE introduce un secreto creado por la aplicación solicitante que puede verificar el <Tooltip tip="Servidor de autorización: Servidor centralizado que contribuye a definir los límites del acceso de un usuario. Por ejemplo, tu servidor de autorización puede controlar los datos, las tareas y las funciones disponibles para un usuario." cta="Ver glosario" href="/docs/es/glossary?term=authorization+server">servidor de autorización</Tooltip>; este secreto se llama Code Verifier. Además, la aplicación solicitante crea un valor derivado del Code Verifier llamado Code Challenge y envía este valor por HTTPS para obtener un código de autorización. De este modo, un atacante malicioso solo puede interceptar el código de autorización, y no puede intercambiarlo por un token sin el Code Verifier.

<div id="how-it-works">
  ## Cómo funciona
</div>

Dado que el flujo de código de autorización con PKCE se basa en el [flujo de código de autorización estándar](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow), los pasos son muy similares.

<Frame>![Flows - Código de autorización con PKCE - Diagrama de secuencia de autorización](/docs/images/cdy7uua7fh8z/3pstjSYx3YNSiJQnwKZvm5/33c941faf2e0c434a9ab1f0f3a06e13a/auth-sequence-auth-code-pkce.png)</Frame>

1. El usuario hace clic en **Iniciar sesión** en la aplicación.
2. El SDK de Auth0 crea un `code_verifier` criptográficamente aleatorio y, a partir de él, genera un `code_challenge`.
3. El SDK de Auth0 redirige al usuario al servidor de autorización de Auth0 ([endpoint `/authorize`](https://auth0.com/docs/api/authentication#authorization-code-grant-pkce-)) junto con el `code_challenge`.
4. Tu servidor de autorización de Auth0 redirige al usuario al prompt de inicio de sesión y autorización.
5. El usuario se autentica con una de las opciones de inicio de sesión configuradas y puede ver una página de consentimiento con la lista de permisos que Auth0 otorgará a la aplicación.
6. Tu servidor de autorización de Auth0 almacena el `code_challenge` y redirige al usuario de vuelta a la aplicación con un `code` de autorización válido para un solo uso.
7. El SDK de Auth0 envía este `code` y el `code_verifier` (creado en el paso 2) al servidor de autorización de Auth0 `(`[`/oauth/token` endpoint](https://auth0.com/docs/api/authentication?http#authorization-code-flow-with-pkce44)).
8. Tu servidor de autorización de Auth0 verifica el `code_challenge` y el `code_verifier`.
9. Tu servidor de autorización de Auth0 responde con un token de identidad y un token de acceso (y, opcionalmente, un token de actualización).
10. Tu aplicación puede usar el token de acceso para llamar a una API y acceder a información sobre el usuario.
11. La API responde con los datos solicitados.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Si tienes habilitada la [rotación del Refresh Token](/docs/es/secure/tokens/refresh-tokens/refresh-token-rotation), se genera un nuevo token de actualización con cada solicitud y se emite junto con el token de acceso. Cuando se intercambia un token de actualización, el token de actualización anterior se invalida, pero el servidor de autorización conserva la información sobre la relación entre ambos.
</Callout>

<div id="how-to-implement-it">
  ## Cómo implementarlo
</div>

La forma más sencilla de implementar el flujo de código de autorización con PKCE es [seguir nuestros Quickstarts nativos](/docs/es/quickstart/native) o [seguir nuestros Quickstarts para aplicaciones de página única](/docs/es/quickstart/spa).

Según el tipo de aplicación, también puedes usar nuestros SDK para móviles o para aplicaciones de página única:

**Móvil**

* [SDK de Auth0 para Swift](/docs/es/libraries/auth0-swift)
* [Auth0 Android SDK](/docs/es/libraries/auth0-android)

**Página única**

* [SDK de Auth0 para aplicaciones de página única](/docs/es/libraries/auth0-single-page-app-sdk)
* [SDK de Auth0 para React](/docs/es/libraries/auth0-react)

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Los avances recientes en los controles de privacidad de los usuarios en los navegadores afectan negativamente la experiencia del usuario al impedir el acceso a cookies de terceros; por lo tanto, los flujos basados en navegador deben usar la [rotación del Refresh Token](/docs/es/secure/tokens/refresh-tokens/refresh-token-rotation), que ofrece un método seguro para usar tokens de actualización en las SPA y, al mismo tiempo, proporciona a los usuarios finales acceso ininterrumpido a los recursos sin las interrupciones en la UX que provocan tecnologías de privacidad del navegador como ITP.
</Callout>

Puedes seguir nuestros tutoriales para usar nuestros endpoints de API para [agregar Login mediante el flujo de código de autorización con PKCE](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce/add-login-using-the-authorization-code-flow-with-pkce) o [llamar a tu API mediante el flujo de código de autorización con PKCE](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce/call-your-api-using-the-authorization-code-flow-with-pkce).
