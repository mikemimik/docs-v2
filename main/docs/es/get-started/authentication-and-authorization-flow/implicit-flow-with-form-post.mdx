title: Flujo implícito con Form Post
---

<Warning>
  ¡No dejes que el término &quot;implícito&quot; te engañe! Aunque OAuth ahora desaconseja el uso de la concesión implícita para obtener tokens de acceso en las SPA, el escenario que aborda el flujo implícito con Form Post es completamente diferente y **no se ve afectado por los problemas de seguridad** que llevaron a desaconsejar su uso con las SPA. En concreto, el flujo implícito con Form Post se aplica a aplicaciones web tradicionales, a diferencia de las SPA. Obtienes tokens de ID en lugar de tokens de acceso, que tienen un propósito completamente distinto. El flujo usa POST en lugar de colocar tokens en fragmentos de URL (como ocurre con las SPA), lo que puede exponer partes del token a ataques contra el historial del navegador, encabezados de redirección, etc.
</Warning>

Puedes usar <Tooltip tip="OpenID: Estándar abierto de autenticación que permite a las aplicaciones verificar las identidades de los usuarios sin recopilar ni almacenar información de inicio de sesión." cta="Ver glosario" href="/docs/es/glossary?term=OpenID">OpenID</Tooltip> Connect (OIDC) con distintos flujos para habilitar el inicio de sesión en una aplicación web tradicional. En un flujo habitual, obtienes un <Tooltip tip="ID Token: Credencial destinada al propio cliente, en lugar de acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=ID+token">token de identidad</Tooltip> mediante el flujo de código de autorización ejecutado por el backend de la aplicación. Este método es eficaz y sólido; sin embargo, requiere que tu aplicación web obtenga y gestione un secreto. Puedes evitar esa carga si lo único que quieres es implementar el inicio de sesión y no necesitas obtener <Tooltip tip="Access Token: Credencial de autorización, en forma de una cadena opaca o JWT, usada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=access+tokens">tokens de acceso</Tooltip> para llamar a APIs.

El flujo implícito con Form Post usa OIDC para implementar un inicio de sesión web muy similar a cómo funcionan <Tooltip tip="Security Assertion Markup Language (SAML): Protocolo estandarizado que permite a dos partes intercambiar información de autenticación sin una contraseña." cta="Ver glosario" href="/docs/es/glossary?term=SAML">SAML</Tooltip> y WS-Federation. La aplicación web solicita y obtiene tokens a través del canal frontal, sin necesidad de secretos ni de llamadas adicionales al backend. Con este método, no necesitas obtener, mantener, usar ni proteger un secreto en tu aplicación.

<div id="how-it-works">
  ## Cómo funciona
</div>

<Warning>
  Debes usar este flujo exclusivamente para casos de uso de inicio de sesión; si necesitas solicitar tokens de acceso mientras el usuario inicia sesión para poder llamar a una API, usa el [flujo de código de autorización con PKCE](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce) o el [flujo híbrido](/docs/es/get-started/authentication-and-authorization-flow/hybrid-flow).
</Warning>

<Frame>![Flows - Implícito con Form Post - Diagrama de secuencia de autorización](/docs/images/cdy7uua7fh8z/6m0uE4E7Hpzbdhyh9dEuYK/e36c910ff47a7540bf27e23c02822624/auth-sequence-implicit-form-post.png)</Frame>

1. El usuario hace clic en **Iniciar sesión** en la aplicación.
2. El SDK de Auth0 redirige al usuario al servidor de autorización de Auth0 (endpoint `/authorize`) y envía un parámetro `response_type` con el valor `id_token` que indica el tipo de credencial solicitada. También envía un parámetro `response_mode` con el valor `form_post` para garantizar la seguridad.
3. Tu servidor de autorización de Auth0 redirige al usuario al prompt de inicio de sesión y autorización.
4. El usuario se autentica con una de las opciones de inicio de sesión configuradas y puede ver una página de consentimiento con la lista de los permisos que Auth0 otorgará a la aplicación.
5. Tu servidor de autorización de Auth0 redirige de nuevo al usuario a la aplicación con un token de ID.

<div id="how-to-implement-it">
  ## Cómo implementarlo
</div>

Puedes [usar nuestro SDK de Express OpenID Connect](https://www.npmjs.com/package/express-openid-connect) para implementar de forma segura el flujo implícito con Form Post.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  El [SDK de Auth0 para aplicaciones de página única](/docs/es/libraries/auth0-single-page-app-sdk) y las [Quickstarts para aplicaciones de página única](/docs/es/quickstart/spa) se ajustan a las nuevas recomendaciones y usan el [flujo de código de autorización con PKCE](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce).
</Callout>

Por último, puedes seguir nuestros tutoriales para usar nuestros endpoints de API y [agregar el inicio de sesión mediante el flujo implícito con Form Post](/docs/es/get-started/authentication-and-authorization-flow/implicit-flow-with-form-post/add-login-using-the-implicit-flow-with-form-post).
