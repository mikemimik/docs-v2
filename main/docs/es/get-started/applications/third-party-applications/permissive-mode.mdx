---
title: Modo permisivo para aplicaciones de terceros
description: "Más información sobre el modo permisivo para aplicaciones de terceros."
---

Los clientes que tuvieran al menos un tenant que usara aplicaciones de terceros antes de abril de 2026 pueden tener aplicaciones que funcionen en modo permisivo. Este modo conserva el comportamiento preexistente para mantener la compatibilidad con versiones anteriores.

<div id="who-can-use-permissive-mode">
  ## Quién puede usar el modo permisivo
</div>

El modo permisivo está disponible para los clientes que, antes de abril de 2026, tengan al menos un tenant con aplicaciones de terceros. Todos los tenants de la cuenta de ese cliente pueden crear aplicaciones de terceros en modo permisivo, incluso aquellos que anteriormente no tenían aplicaciones de terceros.

Estos clientes pueden:

* Seguir usando las aplicaciones de terceros existentes con su comportamiento actual
* Crear nuevas aplicaciones de terceros con `third_party_security_mode: "permissive"` a través de la Management API
* Crear nuevas aplicaciones con controles de seguridad reforzados en cualquier momento

Los clientes que no hayan usado antes aplicaciones de terceros siempre crean aplicaciones con controles de seguridad reforzados. No pueden usar el modo permisivo.

<div id="what-permissive-mode-means">
  ## Qué significa el modo permisivo
</div>

Las aplicaciones de terceros en modo permisivo se comportan de forma similar a las aplicaciones propias en cuanto a las funcionalidades disponibles:

* La mayoría de los tipos de concesión están disponibles (`authorization code`, `implicit`, `client credentials`, `device code`)
* Se admiten los scopes de OIDC y los tokens de ID
* El acceso a la API sigue la [política de acceso a la API](/docs/es/get-started/apis/api-access-policies-for-applications) configurada para la API (no se exige una autorización de cliente)
* Formato estándar de ID de cliente (sin el prefijo `tpc_`)
* Se pueden configurar todas las propiedades del cliente

El modo permisivo requiere un mayor esfuerzo de configuración por parte del administrador para aplicar controles de seguridad. Usted es responsable de lo siguiente:

* Configurar [políticas de acceso a la API](/docs/es/get-started/apis/api-access-policies-for-applications) en cada API para controlar qué aplicaciones de terceros pueden acceder a ellas
* Revisar qué tipos de concesión y funcionalidades están habilitados para cada aplicación
* Monitorear el comportamiento de las aplicaciones de terceros a través de los [logs del tenant](/docs/es/deploy-monitor/logs)

<div id="the-third_party_security_mode-property">
  ## La propiedad `third_party_security_mode`
</div>

Toda aplicación de terceros tiene una propiedad `third_party_security_mode` que indica su modo de seguridad. Esta propiedad se establece al crearse y no puede modificarse.

| **Valor**    | **Significado**                                                                                                                               |
| ------------ | --------------------------------------------------------------------------------------------------------------------------------------------- |
| `strict`     | Controles de seguridad reforzados. Alineación con OAuth 2.1, autorización explícita de API y un conjunto de funcionalidades más acotado.      |
| `permissive` | Comportamiento preexistente. Todas las funcionalidades están disponibles y el administrador configura los controles de seguridad manualmente. |

Las aplicaciones de terceros creadas antes de la introducción de los modos de seguridad devuelven `third_party_security_mode: "permissive"` en las respuestas de la API.

<div id="feature-comparison">
  ## Comparación de funcionalidades
</div>

| **Capacidad**                   | **Controles de seguridad reforzados (strict)**                                                                                                    | **Comportamiento preexistente (permisivo)**               |
| ------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------- |
| **Tipos de concesión**          | `authorization_code`, `refresh_token`, `client_credentials`                                                                                       | La mayoría de los tipos de concesión (excepto `password`) |
| **PKCE**                        | Obligatorio                                                                                                                                       | Opcional                                                  |
| **OIDC**                        | No es compatible. Está previsto para una futura versión.                                                                                          | Compatible                                                |
| **Autorización de API**         | Siempre requiere una autorización de cliente explícita                                                                                            | Sigue la política de acceso a la API                      |
| **Classic Login**               | No es compatible                                                                                                                                  | Compatible                                                |
| **Endpoints heredados**         | No son compatibles                                                                                                                                | Disponibles                                               |
| **Formato de ID de cliente**    | Prefijo `tpc_`                                                                                                                                    | Formato estándar                                          |
| **Propiedades configurables**   | [Conjunto seleccionado de propiedades](/docs/es/get-started/applications/third-party-applications/security-controls#restricted-client-configuration) | Todas las propiedades                                     |
| **Capacidades futuras**         | Límites de frecuencia y futuras capacidades de seguridad y gestión mejoradas                                                                      | No disponibles                                            |
| **Creación mediante Dashboard** | Compatible                                                                                                                                        | No disponible en el Dashboard                             |

<div id="dynamic-client-registration-in-permissive-mode">
  ## Registro dinámico de clientes en modo permisivo
</div>

Si su cuenta cumple los requisitos para el modo permisivo y utiliza [registro dinámico de clientes](/docs/es/get-started/applications/dynamic-client-registration), puede controlar el modo de seguridad de los clientes registrados dinámicamente mediante una configuración del tenant independiente.

<Tabs>
  <Tab title="Auth0 Dashboard">
    1. Vaya a **Settings &gt; Advanced**.
    2. En **Dynamic Client Registration (DCR) Security Mode**, seleccione **Permissive**.
    3. Seleccione **Save**.

    <Frame>![Configuración avanzada del tenant en el Dashboard con el menú desplegable DCR Security Mode](/docs/images/third-party-applications/dcr-security-mode.png)</Frame>
  </Tab>

  <Tab title="Management API">
    ```bash
    curl --request PATCH \
      --url 'https://{yourDomain}/api/v2/tenants/settings' \
      --header 'Authorization: Bearer {YOUR_MANAGEMENT_API_TOKEN}' \
      --header 'Content-Type: application/json' \
      --data '{
        "dynamic_client_registration_security_mode": "permissive"
      }'
    ```
  </Tab>
</Tabs>

La configuración `dynamic_client_registration_security_mode` es independiente del valor predeterminado para las aplicaciones creadas mediante `POST /api/v2/clients`. Puede configurarla en cualquier momento.

<Warning>
  Si establece `dynamic_client_registration_security_mode` en `strict`, primero configure los [permisos predeterminados de la API](/docs/es/get-started/applications/application-access-to-apis-client-grants#default-permissions-for-third-party-applications). Sin ellos, los clientes registrados dinámicamente no podrán acceder a ninguna API.
</Warning>

<div id="adopt-enhanced-security-controls">
  ## Adoptar controles de seguridad reforzados
</div>

Auth0 recomienda adoptar controles de seguridad reforzados para todas las aplicaciones de terceros nuevas. La propiedad `third_party_security_mode` no se puede cambiar después de crear una aplicación: no puedes convertir una aplicación permisiva existente en una `strict`, ni viceversa. En su lugar, al crear nuevas aplicaciones a partir de ahora, debes elegir qué modo de seguridad usar.

Para entender cómo está cambiando el valor predeterminado para las nuevas aplicaciones y cómo prepararte, consulta [Migración a Enhanced Security para aplicaciones de terceros](/docs/es/troubleshoot/product-lifecycle/deprecations-and-migrations/migrate-to-enhanced-security-third-party-applications).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Si necesitas reemplazar una aplicación permisiva existente por una `strict`, debes crear una aplicación nueva. Esto invalida todos los tokens de actualización y las concesiones de usuario existentes de la aplicación anterior. Coordina la transición con el tercero para minimizar las interrupciones.
</Callout>
