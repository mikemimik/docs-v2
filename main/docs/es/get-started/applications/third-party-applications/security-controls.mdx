---
title: Controles de seguridad para aplicaciones de terceros
description: "Más información sobre los controles de seguridad que Auth0 aplica a las aplicaciones de terceros, incluidos OAuth 2.1 y la autorización de API."
---

Auth0 aplica controles de seguridad reforzados a las aplicaciones de terceros para garantizar lo siguiente:

* **Seguridad a nivel de protocolo**: Ajustarse a las [prácticas recomendadas de OAuth 2.1](https://datatracker.ietf.org/doc/html/draft-ietf-oauth-v2-1) para garantizar flujos de autorización modernos y seguros.
* **Alcance del acceso**: Garantizar que las aplicaciones externas solo puedan acceder a los recursos que autorices explícitamente.

<Warning>
  Auth0 mejora periódicamente la seguridad de las aplicaciones de terceros. Solo deben usarse en producción las funcionalidades documentadas explícitamente como compatibles. Las funcionalidades no compatibles pueden modificarse o restringirse sin previo aviso en futuras actualizaciones.
</Warning>

<div id="oauth-21-standards">
  ## Estándares de OAuth 2.1
</div>

Las aplicaciones de terceros aplican estándares modernos de OAuth:

* **PKCE obligatorio**: Todos los flujos de código de autorización requieren [Proof Key for Code Exchange](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce). Esto evita ataques de interceptación de códigos de autorización.
* **Tipos de concesión admitidos**: `authorization_code`, `refresh_token` y `client_credentials`.
* **Las concesiones implícitas y de contraseña no se admiten**: Los tipos de concesión heredados que exponen tokens en la URL del navegador o requieren el manejo directo de credenciales no están disponibles para las aplicaciones de terceros.

<div id="explicit-api-authorization">
  ## Autorización explícita de la API
</div>

Las aplicaciones de terceros siempre requieren una [autorización de cliente](/docs/es/get-started/applications/application-access-to-apis-client-grants) para acceder a cualquier API, independientemente de la [política de acceso a la API](/docs/es/get-started/apis/api-access-policies-for-applications).

| **Política de acceso a la API**  | **Aplicaciones propias**         | **Aplicaciones de terceros**     |
| -------------------------------- | -------------------------------- | -------------------------------- |
| Permitir todo                    | Acceso concedido                 | Requiere autorización de cliente |
| Requerir autorización de cliente | Requiere autorización de cliente | Requiere autorización de cliente |
| Denegar                          | Acceso denegado                  | Acceso denegado                  |

Las aplicaciones de terceros deben contar con una autorización explícita, incluso cuando una API está configurada con una política de **Permitir todo**. Puedes configurar permisos por aplicación o [permisos predeterminados para aplicaciones de terceros](/docs/es/get-started/applications/application-access-to-apis-client-grants#default-permissions-for-third-party-applications).

No se puede conceder a las aplicaciones de terceros acceso a las [API del sistema](/docs/es/get-started/apis#system-apis), como Management API o My Account API.

<div id="machine-to-machine-client-credentials">
  ## Máquina a máquina (Client Credentials)
</div>

Las aplicaciones de terceros admiten el tipo de concesión `client_credentials` para el acceso de máquina a máquina. Esto permite integraciones de backend de partners y acceso a API de servidor a servidor sin intervención del usuario.

**Requisitos y restricciones:**

* Tipo de cliente: la aplicación debe ser un cliente confidencial (`token_endpoint_auth_method` no debe ser `none`).
* [Organizations](/docs/es/manage-users/organizations): se admite el acceso de máquina a máquina con Organizations. Se requiere una [autorización de cliente de organización](/docs/es/manage-users/organizations/organizations-for-m2m-applications/configure-your-application-for-m2m-access) explícita para cada organización. La opción `allow_any_organization` no está permitida para aplicaciones de terceros. Las autorizaciones de cliente predeterminadas para aplicaciones de terceros no se pueden usar para configurar `organization_usage`.
* No está disponible para aplicaciones creadas mediante [registro dinámico de clientes](/docs/es/get-started/applications/dynamic-client-registration) o [CIMD](/docs/es/get-started/auth0-overview/create-applications/register-applications-with-cimd).

**Extensibilidad:**

* Las [Actions](/docs/es/customize/actions) con el trigger `credentials-exchange` se ejecutan normalmente en los flujos de acceso de máquina a máquina.

<div id="restricted-client-configuration">
  ## Configuración restringida del cliente
</div>

Solo puedes configurar un conjunto limitado de propiedades del cliente para aplicaciones de terceros. Cuando se agregan nuevas propiedades a Auth0, no están disponibles para aplicaciones de terceros a menos que se revisen explícitamente y se añadan al conjunto compatible.

Entre las propiedades compatibles más importantes se incluyen:

| **Propiedad**                           | **Notas**                                                                             |
| --------------------------------------- | ------------------------------------------------------------------------------------- |
| `name`, `description`, `logo_uri`       | Metadatos básicos                                                                     |
| `callbacks`                             | Redirect URIs                                                                         |
| `allowed_origins`, `web_origins`        | Orígenes de CORS y `web_message`                                                      |
| `grant_types`                           | Debe ser `authorization_code`, `refresh_token` o `client_credentials`                 |
| `token_endpoint_auth_method`            | Método de autenticación para el endpoint de token                                     |
| `app_type`                              | Debe ser `regular_web`, `spa`, `native` o `non_interactive`                           |
| `client_metadata`                       | Metadatos personalizados de clave-valor                                               |
| `jwt_configuration.lifetime_in_seconds` | duración del token de acceso (el valor predeterminado es 3600)                        |
| `jwt_configuration.alg`                 | Algoritmo de firma (debe ser `RS256`; `HS256` no es compatible)                       |
| `refresh_token.*`                       | Configuración de rotación, expiración, margen y duración                              |
| `client_authentication_methods`         | clave privada JWT (solo `private_key_jwt`; sin mTLS)                                  |
| `require_proof_of_possession`           | Configuración de DPoP                                                                 |
| `redirection_policy`                    | Comportamiento de redirección para flujos de error y plantillas de correo electrónico |

Para consultar la lista completa de propiedades compatibles, consulta el [endpoint Create a Client](https://auth0.com/docs/api/management/v2/clients/post-clients) en la referencia de la Management API.

<div id="client-id-format">
  ## Formato del ID de cliente
</div>

Las aplicaciones de terceros tienen un `client_id` con el prefijo `tpc_` que se les asigna al crearse. Este prefijo permite a Auth0 clasificar y gestionar por separado el tráfico de las aplicaciones de terceros, incluidos los límites de frecuencia aplicables a estas aplicaciones.

El modo de seguridad y la titularidad de la aplicación son decisiones de diseño permanentes:

* `third_party_security_mode` no puede cambiarse después de su creación.
* Las aplicaciones de terceros no pueden convertirse en aplicaciones propias, y viceversa.

<div id="refresh-token-settings">
  ## Configuración del token de actualización
</div>

Las aplicaciones de terceros aplican una configuración segura del token de actualización:

* **Expiración obligatoria**: No se admiten tokens de actualización sin vencimiento. No se admite una duración de inactividad infinita.
* **Rotación habilitada de forma predeterminada para clientes públicos**: Las aplicaciones de terceros de tipo SPA y Native tienen habilitada de forma predeterminada la rotación del Refresh Token, en consonancia con los requisitos de [OAuth 2.1](https://datatracker.ietf.org/doc/html/draft-ietf-oauth-v2-1) y [MCP](https://modelcontextprotocol.io/specification/draft/basic/authorization).
* **Configurable**: Los administradores pueden ajustar la rotación, la tolerancia y la duración de los tokens en las aplicaciones de terceros creadas manualmente.

<div id="redirect-protection">
  ## Protección de redirección
</div>

La propiedad `redirection_policy` controla cómo Auth0 gestiona las redirecciones para aplicaciones de terceros. Acepta dos valores:

| **Valor**                                                                 | **Comportamiento**                                                                                                                                                                                 |
| ------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `open_redirect_protection` (predeterminado para aplicaciones de terceros) | Auth0 no redirige a la URI de callback de la aplicación cuando hay errores de autenticación. La variable `application.callback_domain` no está disponible en las plantillas de correo electrónico. |
| `allow_always`                                                            | Comportamiento de redirección estándar.                                                                                                                                                            |

Las redirecciones sin interacción del usuario pueden ser un vector de ataque de phishing cuando la URI de redirección está controlada por una parte no confiable (open redirect). Solo configura `redirection_policy` como `allow_always` para aplicaciones cuyas URI de callback configuradas sean de confianza.

Cuando `open_redirect_protection` está activo:

* Los errores de autenticación muestran una página de error en lugar de redirigir a la aplicación.
* Las plantillas de correo electrónico (verificación de correo electrónico, restablecimiento de contraseña, usuario bloqueado) no tendrán acceso a `{{ application.callback_domain }}`, por lo que se debe configurar un fallback junto con cualquier uso de `{{ application.callback_domain }}`. Por ejemplo:

```liquid wrap lines
{% if application.callback_domain == '' %}
  https://YOUR_FALLBACK_DOMAIN
{% endif %}
{% if application.callback_domain != '' %}
  {{ application.callback_domain }}/result-page
{% endif %}
```

<div id="authorize-parameter-validation">
  ## Validación de parámetros de `/authorize`
</div>

Auth0 valida los parámetros enviados al endpoint `/authorize` para aplicaciones de terceros. Solo se aceptan parámetros estándar de OAuth 2.0 y OpenID Connect.

**Parámetros permitidos:**

* `acr_values`
* `audience`
* `authorization_details`
* `client_id`
* `code_challenge`
* `code_challenge_method`
* `connection`
* `correlation_id`
* `display`
* `dpop_jkt`
* `ext-*` (parámetros personalizados)
* `login_hint`
* `max_age`
* `nonce`
* `prompt`
* `redirect_uri`
* `resource`
* `response_type`
* `scope`
* `state`
* `ui_locales`

**No admitidos:**

* `claims`
* `id_token_hint`
* `invitation`
* `login_ticket`
* `request` (solicitud de autorización protegida con JWT)
* `request_uri` (PAR)
* `screen_hint`

Las solicitudes que incluyan parámetros no admitidos devuelven un error `invalid_request`.

<div id="backward-compatibility">
  ## Compatibilidad con versiones anteriores
</div>

Algunos tenants que usaban aplicaciones de terceros antes de abril de 2026 pueden tener aplicaciones que funcionan con configuraciones de seguridad diferentes para mantener la compatibilidad con versiones anteriores. Para obtener más información, consulta [Modo permisivo para aplicaciones de terceros](/docs/es/get-started/applications/third-party-applications/permissive-mode).

<div id="features-not-supported">
  ## Funcionalidades no compatibles
</div>

Las siguientes funcionalidades no se admiten para las aplicaciones de terceros:

| **Funcionalidad**                                                                    | **Estado**                                                                                                                                                                           |
| ------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Scopes de OIDC y tokens de ID**                                                    | No se admite. Previsto para una versión futura.                                                                                                                                      |
| **endpoint `/userinfo`**                                                             | No se admite.                                                                                                                                                                        |
| **APIs del sistema de Auth0** (Management API, MFA API, My Account API, My Orgs API) | No se admite. Las aplicaciones de terceros no pueden acceder a las APIs del sistema en los flujos de usuario.                                                                        |
| **MFA durante el intercambio de token de actualización**                             | No se admite. Las transacciones de token de actualización que activen MFA provocarán un error.                                                                                       |
| **Rules**                                                                            | No se admite. Los tenants con Rules activas recibirán un error cuando una aplicación de terceros estricta active un flujo de login.                                                  |
| **Hooks** (credentials-exchange)                                                     | No se admite. Los tenants con un Hook `credentials-exchange` activo recibirán un error. Migra a [Actions](/docs/es/customize/actions) para la extensibilidad de `credentials-exchange`. |
| **endpoints no OAuth de Authentication API** (`/dbconnections/*`, `/passwordless/*`) | No se admite.                                                                                                                                                                        |
| **endpoints heredados** (`/delegation`, `/oauth/ro`)                                 | No se admite.                                                                                                                                                                        |
| **SAML, WsFed**                                                                      | No se admite.                                                                                                                                                                        |
| **Classic Login**                                                                    | No se admite. Usa [Universal Login](/docs/es/authenticate/login/auth0-universal-login).                                                                                                 |
| **PAR, CIBA, Device Code**                                                           | No se admite. Previsto para una versión futura.                                                                                                                                      |
| **endpoints de logout**                                                              | No se admite. Usa `POST /oauth/revoke` para revocar tokens.                                                                                                                          |
| **autenticación de origen cruzado**                                                  | No se admite.                                                                                                                                                                        |
| **logout por backchannel**                                                           | No se admite. Previsto para una versión futura.                                                                                                                                      |
| **Importación de ID de cliente**                                                     | No se admite.                                                                                                                                                                        |
| **Subdominios comodín en URL**                                                       | No se admite. Las URL de devolución de llamada, los orígenes permitidos y los orígenes web deben usar URL exactas.                                                                   |
