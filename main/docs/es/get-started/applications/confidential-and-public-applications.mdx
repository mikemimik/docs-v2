title: Aplicaciones confidenciales y públicas
---

Según la [especificación de OAuth 2.0](https://tools.ietf.org/html/rfc6749#section-2.1), las aplicaciones pueden clasificarse como confidenciales o públicas. La principal diferencia radica en si la aplicación puede almacenar credenciales (como un <Tooltip tip="ID de cliente: valor de identificación asignado por Auth0 a tu recurso registrado." cta="Ver glosario" href="/docs/es/glossary?term=client+ID">ID de cliente</Tooltip> y un secreto) de forma segura o no. Esto afecta al tipo de autenticación que pueden usar las aplicaciones.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Las clasificaciones confidencial/pública y propia/de terceros son independientes. Confidencial o pública describe la capacidad de autenticación de la aplicación (si puede almacenar un secreto). Propia o de terceros describe la relación de confianza (quién es propietario de la aplicación y quién la opera). Para obtener más información, consulta [aplicaciones propias y de terceros](/docs/es/get-started/applications/first-party-and-third-party-applications).
</Callout>

Cuando creas una aplicación en el Auth0 Dashboard, Auth0 te preguntará qué tipo de aplicación de Auth0 quieres asignarle a la nueva aplicación y usará esa información para determinar si la aplicación es confidencial o pública.

Para obtener más información, consulta [Comprueba si una aplicación es confidencial o pública](/docs/es/get-started/applications/confidential-and-public-applications/view-application-type).

<div id="confidential-applications">
  ## Aplicaciones confidenciales
</div>

Las aplicaciones confidenciales pueden almacenar credenciales de forma segura sin exponerlas a partes no autorizadas. Requieren un servidor backend de confianza para almacenar uno o varios secretos.

<div id="grant-types">
  ### Tipos de concesión
</div>

Las aplicaciones confidenciales usan un servidor backend de confianza y pueden usar tipos de concesión que requieren especificar su ID de cliente y <Tooltip tip="Secreto de cliente: Secret utilizado por un cliente (aplicación) para autenticarse con el servidor de autorización; solo deben conocerlo el cliente y el servidor de autorización, y debe ser lo suficientemente aleatorio como para que no pueda adivinarse." cta="Ver glosario" href="/docs/es/glossary?term=Client+Secret">secreto de cliente</Tooltip> (o credenciales registradas alternativas) para autenticarse al llamar al endpoint [Get Token](https://auth0.com/docs/api/authentication#get-token) de Auth0 Authentication API. Las aplicaciones confidenciales pueden usar los métodos de autenticación Client Secret Post, Client Secret Basic o [clave privada JWT](/docs/es/get-started/authentication-and-authorization-flow/authenticate-with-private-key-jwt).

Se consideran aplicaciones confidenciales las siguientes:

* Una aplicación web con un backend seguro que usa el [flujo de código de autorización](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow), el [flujo de contraseña del propietario del recurso](/docs/es/get-started/authentication-and-authorization-flow/resource-owner-password-flow) o el flujo de contraseña del <Tooltip tip="Propietario del recurso: Entidad (como un usuario o una aplicación) capaz de otorgar acceso a un recurso protegido." cta="Ver glosario" href="/docs/es/glossary?term=Resource+Owner">propietario del recurso</Tooltip> con compatibilidad con realm
* Una aplicación de máquina a máquina (M2M) que usa el [flujo de credenciales de cliente](/docs/es/get-started/authentication-and-authorization-flow/client-credentials-flow)

<div id="id-tokens">
  ### Tokens de ID
</div>

Como las aplicaciones confidenciales pueden almacenar secretos, se les pueden emitir <Tooltip tip="token de ID: credencial destinada al propio cliente, no para acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=ID+tokens">tokens de ID</Tooltip> firmados de una de estas dos maneras:

* Simétricamente, usando su secreto de cliente (`HS256`)
* Asimétricamente, usando una clave privada (`RS256`)

<div id="public-applications">
  ## Aplicaciones públicas
</div>

Las aplicaciones públicas **no pueden** guardar credenciales de forma segura.

<div id="grant-types">
  ### Tipos de concesión
</div>

Las aplicaciones públicas solo pueden usar tipos de concesión que no requieran el uso de su secreto de cliente. No pueden enviar un secreto de cliente porque no pueden mantener la confidencialidad de las credenciales necesarias.

Estas son aplicaciones públicas:

* Una aplicación nativa de escritorio o móvil que usa el [flujo de código de autorización con PKCE](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce)
* Una aplicación web basada en JavaScript que se ejecuta del lado del cliente (como una aplicación de una sola página) y usa la concesión de [flujo implícito](/docs/es/get-started/authentication-and-authorization-flow/implicit-flow-with-form-post)

<div id="id-tokens">
  ### Tokens de ID
</div>

Dado que las aplicaciones públicas no pueden almacenar secretos, los [tokens de ID](/docs/es/secure/tokens/id-tokens) emitidos para ellas deben:

* Firmarse de forma asimétrica con una clave privada (`RS256`)
* Verificarse con la clave pública correspondiente a la clave privada utilizada para firmar el token
