Puede registrar dinámicamente aplicaciones de terceros en su tenant. El registro dinámico de clientes (DCR) se basa en la [especificación de OpenID Connect para el registro dinámico de clientes](https://openid.net/specs/openid-connect-registration-1_0.html).

Todas las aplicaciones creadas mediante el registro dinámico de clientes son [aplicaciones de terceros](/docs/es/get-started/applications/third-party-applications) con [controles de seguridad reforzados](/docs/es/get-started/applications/third-party-applications/security-controls). Esto significa que los clientes DCR:

* Reciben un prefijo de ID de cliente `tpc_`
* Requieren PKCE para los flujos de código de autorización
* Solo admiten los tipos de concesión `authorization_code` y `refresh_token`. El tipo de concesión `client_credentials` no está disponible a través de DCR.
* Solo pueden acceder a las API mediante [autorizaciones de cliente](/docs/es/get-started/applications/application-access-to-apis-client-grants) explícitas
* Solo pueden usar [conexiones a nivel de dominio](/docs/es/authenticate/identity-providers/promote-connections-to-domain-level) para la autenticación

<div id="enable-dynamic-client-registration">
  ## Habilitar el registro dinámico de clientes
</div>

<Warning>
  Auth0 admite Open Dynamic Registration. Si se habilita, cualquiera podrá crear aplicaciones en su tenant sin necesidad de un token.
</Warning>

De forma predeterminada, el registro dinámico de clientes está deshabilitado para todos los tenants. Para habilitar el registro dinámico de clientes, use el Auth0 Dashboard o la Management API.

<Tabs>
  <Tab title="Auth0 Dashboard">
    1. Vaya a [Dashboard &gt; Settings &gt; Advanced](https://manage.auth0.com/#/tenant/advanced) y habilite **Registro dinámico de clientes (DCR)**.
  </Tab>

  <Tab title="Management API">
    Establezca el indicador `enable_dynamic_client_registration` en `true` en la configuración de su tenant mediante el endpoint [`/api/v2/tenants/settings`](https://auth0.com/docs/api/management/v2#!/Tenants/patch_settings).

    <AuthCodeGroup>
      ```bash cURL
      curl --request PATCH \
        --url 'https://{yourDomain}/api/v2/tenants/settings' \
        --header 'authorization: Bearer {YOUR_MANAGEMENT_API_TOKEN}' \
        --header 'content-type: application/json' \
        --data '{ "flags": { "enable_dynamic_client_registration": true } }'
      ```
    </AuthCodeGroup>

    Debe reemplazar `{YOUR_MANAGEMENT_API_TOKEN}` por un token válido con el scope `update:tenant_settings`. Para obtener más información, consulte [Management API Access Tokens](/docs/es/secure/tokens/access-tokens/management-api-access-tokens).
  </Tab>
</Tabs>

<div id="configure-api-access-for-dcr-clients">
  ## Configura el acceso a la API para clientes DCR
</div>

Antes de habilitar DCR, configura los [permisos predeterminados para aplicaciones de terceros](/docs/es/get-started/applications/application-access-to-apis-client-grants#default-permissions-for-third-party-applications) en las API a las que deban acceder los clientes registrados dinámicamente. Sin permisos predeterminados, los clientes DCR no podrán acceder a ninguna API.

Los permisos predeterminados definen un conjunto de API y scopes de línea base disponibles automáticamente para todas las aplicaciones de terceros. Esto es esencial para DCR porque no puedes configurar autorizaciones de cliente por aplicación durante el flujo de registro.

Para obtener más información sobre cómo configurar los permisos predeterminados, lee [Configure Third-Party Applications](/docs/es/get-started/applications/third-party-applications/configure-third-party-applications#default-permissions-for-all-third-party-applications).

<div id="register-an-application">
  ## Registrar una aplicación
</div>

Para registrar dinámicamente una aplicación, realiza una solicitud `POST` al endpoint `/oidc/register`. Como Auth0 admite Open Dynamic Registration, el endpoint `/oidc/register` acepta solicitudes de registro sin un token de acceso.

<AuthCodeGroup>
  ```bash cURL
  curl --request POST \
    --url 'https://{yourDomain}/oidc/register' \
    --header 'content-type: application/json' \
    --data '{
      "client_name": "My Dynamic Application",
      "redirect_uris": ["https://application.example.com/callback"],
      "token_endpoint_auth_method": "none",
      "grant_types": ["authorization_code", "refresh_token"],
      "response_types": ["code"]
    }'
  ```
</AuthCodeGroup>

| **Parameter**                | **Descripción**                                                                                                                                                                                          |
| ---------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `client_name`                | El nombre de la aplicación que se creará.                                                                                                                                                                |
| `redirect_uris` (required)   | Un arreglo de URL que Auth0 aceptará como URL de devolución de llamada válidas al final de un flujo de autenticación.                                                                                    |
| `token_endpoint_auth_method` | El método de autenticación para el endpoint de token. Usa `none` para clientes públicos (SPA, Native) o `client_secret_post` (predeterminado) para clientes confidenciales.                              |
| `grant_types`                | Los tipos de concesión que la aplicación quiere usar. Se filtran en la respuesta para reflejar los que podrá usar. Las aplicaciones creadas mediante DCR admiten `authorization_code` y `refresh_token`. |
| `response_types`             | Los tipos de respuesta que usará la aplicación. Usa `code` para el flujo de código de autorización.                                                                                                      |

Si la solicitud se realiza correctamente, Auth0 devuelve las credenciales de la aplicación:

```json
{
  "client_name": "My Dynamic Application",
  "client_id": "tpc_8SXWY6j3afl2CP5ntwEOpMdPxxy49Gt2",
  "client_secret": "Q5O...33P",
  "redirect_uris": [
    "https://application.example.com/callback"
  ],
  "client_secret_expires_at": 0,
  "grant_types": ["authorization_code", "refresh_token"],
  "token_endpoint_auth_method": "none"
}
```

| **Campo**                  | **Descripción**                                                                                                                 |
| -------------------------- | ------------------------------------------------------------------------------------------------------------------------------- |
| `client_id`                | Identificador único de la aplicación con el prefijo `tpc_`. Úselo para iniciar flujos de autenticación.                         |
| `client_secret`            | Secreto de cliente de la aplicación para clientes confidenciales. No se devuelve cuando `token_endpoint_auth_method` es `none`. |
| `client_secret_expires_at` | Momento de expiración del secreto de cliente. En Auth0, siempre es `0` (nunca expira).                                          |

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Los desarrolladores de terceros no pueden modificar la configuración de la aplicación después del registro. Si es necesario realizar cambios, deben ponerse en contacto con el propietario del tenant.
</Callout>

Después del registro, la aplicación puede iniciar un [flujo de código de autorización con PKCE](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce) con el `client_id` y los `redirect_uris` configurados.

<div id="tenant-access-control-list-acl">
  ## ACL del tenant (ACL)
</div>

Auth0 proporciona una [ACL del tenant](/docs/es/secure/tenant-access-control-list) para gestionar el tráfico al endpoint `/oidc/register`. Puede restringir quién puede enviar solicitudes de DCR configurando reglas de ACL en función de lo siguiente:

* Direcciones IP de origen y rangos CIDR
* Geolocalización
* Otras señales de la solicitud

Para configurar reglas de ACL para DCR, agregue el `scope` `dcr` a una regla de ACL. Para obtener más información, consulte la [referencia de la ACL del tenant](/docs/es/secure/tenant-access-control-list/reference).

<div id="rate-limits">
  ## Límites de frecuencia
</div>

El endpoint `/oidc/register` está limitado a 5 solicitudes por segundo por tenant. Para obtener más información sobre los límites de frecuencia, consulta [Configuración del límite de frecuencia](/docs/es/troubleshoot/customer-support/operational-policies/rate-limit-policy/rate-limit-configurations).

<div id="permissive-mode-for-dcr">
  ## Modo permisivo para DCR
</div>

Algunos clientes que ya utilizaban aplicaciones de terceros antes de abril de 2026 pueden configurar DCR para crear aplicaciones con comportamiento preexistente en lugar de controles de seguridad reforzados. Para obtener más información, consulta [Registro dinámico de clientes en modo permisivo](/docs/es/get-started/applications/third-party-applications/permissive-mode#dynamic-client-registration-in-permissive-mode).
