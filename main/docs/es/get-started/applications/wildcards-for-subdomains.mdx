title: Marcadores de posición de URL para subdominios
---

Puedes usar varios marcadores de posición como entradas de texto dinámicas en tus URL.

<div id="how-url-evaluation-works">
  ## Cómo funciona la evaluación de URL
</div>

Una URL que contiene el marcador de posición `{organization_name}` solo se evaluará cuando se cumplan todas las condiciones siguientes:

* La aplicación tiene `organization_usage` configurado como `allow` o `require`
* Se realizó una transacción en el contexto de una organización (por ejemplo, al iniciar una transacción de autorización con el parámetro `organization`: `/authorize?organization=org_bVss9Do3994SIbiH&…`)

Las URL con el marcador de posición `{organization_name}` se evaluarán además de las URL de coincidencia exacta (`https://app.exampleco.com`) y las URL con comodines (`https://*.exampleco.com`). No debe asumir ningún orden específico en la evaluación de las URL.

Evite registrar URL con comodines y marcadores de posición de organización en el mismo campo de configuración de una aplicación, ya que esto puede provocar comportamientos no deseados y dificultar la resolución de problemas. Como ejemplo, considere una aplicación con dos **Allowed Callback URLs**: `https://*.exampleco.com` y `https://{organization_name}.exampleco.com`. Un `redirect_uri` con el valor `https://company-a.exampleco.com` se consideraría válido incluso si no hubiera ninguna organización con el nombre `company-a` registrada en su tenant; esto se debe a la evaluación del marcador de posición con comodín.

<div id="wildcard-url-placeholders">
  ## Marcadores de posición comodín en URL
</div>

Los marcadores de posición comodín en subdominios **no deben** usarse en aplicaciones de producción. Auth0 recomienda usar URL con el marcador de posición `{organization_name}` cuando corresponda.

<Warning>
  Los marcadores de posición comodín en URL no son compatibles con las [aplicaciones de terceros](/docs/es/get-started/applications/third-party-applications) con controles de seguridad reforzados. Las URL de devolución de llamada, los orígenes permitidos y los orígenes web deben usar URL exactas.
</Warning>

Administra esta configuración en [Dashboard &gt; Applications &gt; Applications](https://manage.auth0.com/#/applications), en los siguientes campos:

* **Allowed Callback URLs**: Lista de URL a las que Auth0 puede redirigir a los usuarios después de autenticarse.
* **Allowed Logout URLs**: Lista de URL a las que puedes redirigir a los usuarios después de que cierren sesión en Auth0.
* **Allows Web Origins**: Lista de URL desde las que puede originarse una solicitud de autorización que use [Cross-Origin Authentication](/docs/es/authenticate/login/cross-origin-authentication), [Device Flow](/docs/es/get-started/authentication-and-authorization-flow/device-authorization-flow) y [web&#95;message como modo de respuesta](/docs/es/authenticate/protocols/oauth).
* **Allowed Origins (CORS)**: Lista de URL a las que se les permitirá realizar solicitudes desde JavaScript a Auth0 API (normalmente se usa con CORS).

Evita usar marcadores de posición comodín para subdominios en las devoluciones de llamada y los orígenes permitidos de aplicaciones de producción, ya que esto puede hacer que tu aplicación sea vulnerable a ataques.

Puedes usar el símbolo de asterisco (`*`) como comodín para subdominios, pero debe usarse de acuerdo con las siguientes reglas para que funcione correctamente:

* El protocolo de la URL **debe** ser `http` o `https`. Los protocolos como `com.example.app` y `service:jmx:rmi` no funcionarán.
* El comodín **debe** estar ubicado en un subdominio dentro del componente hostname. `https://*.com` no funcionará.
* El comodín **debe** estar ubicado en el subdominio más alejado del dominio raíz. `https://sub.*.example.com` no funcionará.
* La URL **no debe** contener más de un comodín. `https://*.*.example.com` no funcionará.
* Un comodín **puede** llevar como prefijo y/o sufijo caracteres válidos adicionales en el hostname. `https://prefix-*-suffix.example.com` funcionará.
* Una URL con un comodín válido **no** coincidirá con una URL que tenga más de un nivel de subdominio en lugar del comodín. `https://*.example.com` no funcionará con `https://sub1.sub2.example.com`.

<div id="organization-url-placeholders">
  ## Marcadores de posición de URL de la organización
</div>

<div id="organization-name-placeholder">
  ### Marcador de posición del nombre de la organización
</div>

Puedes usar `{organization_name}` como marcador de posición para especificar dinámicamente el nombre de una Organización registrada en una URL (`https://{organization_name}.exampleco.com`). Las URL con el marcador de posición `{organization_name}` solo deben usarse en dominios que controles por completo. Por ejemplo, si controlas el dominio `exampleco.com`, el marcador de posición con el nombre de tu Organización sería `https://{organization_name}.exampleco.com`.

Administra esta configuración en [Dashboard &gt; Applications &gt; Applications](https://manage.auth0.com/#/applications), en los siguientes campos:

* **Allowed Callback URLs**: Lista de URL a las que Auth0 puede redirigir a los usuarios después de autenticarse.
* **Allowed Origins (CORS)**: Lista de URL autorizadas para hacer requests desde JavaScript a Auth0 API (normalmente se usa con CORS).

Se aplican las siguientes restricciones al usar el marcador de posición `{organization_name}`:

* El protocolo de la URL debe ser `http:` o `https:`. `com.example.app://{organization_name}.exampleco.com` no funcionará.
* El marcador de posición debe estar ubicado en un subdominio dentro del componente hostname. `https://{organization_name}` y `https://exampleco.com/{organization_name}` no funcionarán.
* El marcador de posición debe estar ubicado en el subdominio más alejado del dominio raíz. `https://sub.{organization_name}.exampleco.com` no funcionará.
* La URL no debe contener más de un marcador de posición. `https://{organization_name}.{organization_name}.exampleco.com` no funcionará.
* Un marcador de posición no debe tener como prefijo ni como sufijo otros caracteres válidos de hostname. `https://prefix-{organization_name}-suffix.exampleco.com` no funcionará.
* Un marcador de posición **no debe** usarse junto con un comodín en la URL. `https://{organization_name}.*.exampleco.com` no funcionará.

<div id="organization-metadata-placeholder">
  ### Marcador de posición de metadatos de la organización
</div>

Puedes usar `{organization.metadata.KEY}` como marcador de posición para especificar dinámicamente una URL según los metadatos asociados a la organización en la solicitud actual. Esto resulta útil cuando distintas organizaciones requieren distintos URI de inicio de sesión, por ejemplo, cuando cada organización se asigna a un dominio personalizado diferente.

La `KEY` debe comenzar con `public_` o `PUBLIC_` (por ejemplo, `{organization.metadata.public_login_host}`). Las claves sin este prefijo se ignoran en runtime por motivos de seguridad.

Auth0 admite este marcador de posición para el URI de inicio de sesión de la aplicación (initiate&#95;login&#95;uri). Para obtener más información, consulta [URI de inicio de sesión dinámicos con marcadores de posición de metadatos](/docs/es/authenticate/login/auth0-universal-login/configure-default-login-routes).

Las mismas [reglas de validación](#validation-rules) que se aplican a los marcadores de posición de dominio personalizado también se aplican a los marcadores de posición de metadatos de la organización (protocolo, ubicación, anidamiento, sin comodines, tipo de dato).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Los marcadores de posición de metadatos de la organización requieren que la solicitud esté en el contexto de una organización. Si no hay ninguna organización presente, el marcador de posición no puede resolverse y Auth0 usa `default_redirection_uri` a nivel del tenant.
</Callout>

<div id="custom-domain-url-placeholders">
  ## Marcadores de posición de URL de dominio personalizado
</div>

Puedes usar `{custom_domain.metadata.KEY}` como marcador de posición para especificar dinámicamente una URL según los metadatos asociados al dominio personalizado utilizado en la solicitud. Esto te permite admitir múltiples dominios personalizados con distintas URL de aplicación dentro del mismo tenant.

Para obtener una visión completa de esta funcionalidad, consulta [múltiples dominios personalizados](/docs/es/customize/custom-domains/multiple-custom-domains)

<div id="validation-rules">
  ### Reglas de validación
</div>

<Warning>
  Los valores de metadatos usados como marcadores de posición de URL pueden quedar expuestos a los usuarios finales durante los flujos de autenticación. No uses metadatos sensibles en los marcadores de posición. Todas las claves de metadatos usadas en los marcadores de posición deben comenzar con `public_` para que esto quede explícito.
</Warning>

Las siguientes restricciones se aplican al usar marcadores de posición de dominio personalizado:

* **Prefijo `public_` obligatorio**: La clave de metadato usada en el marcador de posición debe comenzar con `public_` o `PUBLIC_` (por ejemplo, `{custom_domain.metadata.public_callback_subdomain}`). Las claves sin este prefijo se ignoran en runtime por motivos de seguridad.
  * **Protocolo**: El protocolo de la URL debe ser `http` o `https`.
  * **Ubicación**: El marcador de posición debe estar en el componente de dominio o subdominio. No puede usarse en la ruta de la URL.
    * Válido: `https://{custom_domain.metadata.public_app_url}.example.com/login`
    * No válido: `https://example.com/{custom_domain.metadata.public_path}`
  * **Anidación**: No puedes acceder a propiedades anidadas de metadatos. Solo se admiten claves de nivel superior.
  * **Sin comodines**: Un marcador de posición de dominio personalizado no debe usarse junto con un comodín (`*`) en la misma URL.
  * **Tipo de datos**: El valor del metadato del dominio personalizado debe ser un String. Si la clave no existe o el valor no es un String, la URL se ignora durante la validación.

<div id="supported-fields">
  ### Campos admitidos
</div>

Estos marcadores de posición se pueden configurar para las siguientes URLs de la aplicación:

* Allowed Callback URLs
  * Allowed Logout URLs
  * Allowed Web Origins
  * Allowed Origins (CORS)
  * Application URI de inicio de sesión (`initiate_login_uri`). Para obtener más información, consulta [URI de inicio de sesión dinámicos con marcadores de posición de metadatos](/docs/es/authenticate/login/auth0-universal-login/configure-default-login-routes#dynamic-login-uris-with-metadata-placeholders).

Para obtener más información, consulta [Application Settings](https://auth0.com/docs/get-started/applications/application-settings#application-uris).

<Note>
  Los marcadores de posición de dominio personalizado no se admiten para aplicaciones de terceros.
</Note>

<div id="using-with-organization-placeholders">
  ### Uso con marcadores de posición de organización
</div>

Puedes combinar el marcador de posición de dominio personalizado con el marcador de posición `{organization_name}`, siempre que tu flujo lo permita. Ambos marcadores de posición se evalúan y se sustituyen durante el runtime.
