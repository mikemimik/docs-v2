---

Aprende a configurar la autenticación mTLS para un tenant.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  En los siguientes ejemplos, se especifica **$management&#95;access&#95;token**, o un [token de acceso de la Management API](/docs/es/secure/tokens/access-tokens/management-api-access-tokens). Debes reemplazarlo por un token de acceso que contenga al menos los siguientes scopes:

  * `create:custom_domains`
  * `read:custom_domains`
  * `create:clients`
  * `update:clients`
  * `update:client_credentials`
  * `update:client_keys`
  * `update:tenant_settings`

  Para obtener más información sobre cómo obtener un token de acceso con los scopes requeridos, consulta [Get Access Tokens](/docs/es/secure/tokens/access-tokens/get-access-tokens).
</Callout>

Para comenzar, debes configurar y verificar un [dominio personalizado](/docs/es/customize/custom-domains).

<div id="create-a-custom-domain">
  ### Crear un dominio personalizado
</div>

A nivel del tenant, debes configurar un [dominio personalizado](/docs/es/customize/custom-domains) para aceptar encabezados mTLS con la [Management API](https://auth0.com/docs/api/management/v2). Dado que el [customer edge](/docs/es/get-started/applications/configure-mtls/set-up-the-customer-edge) se encarga de validar los certificados del cliente, establece `type` como `self_managed_certs` en la solicitud POST:

```bash lines
curl --location --request POST 'https://$tenant/api/v2/custom-domains' \
  --header 'Authorization: Bearer $management_access_token' \
  --header 'Content-Type: application/json' \
  --data-raw '{
  "domain":"string",
  "type":"self_managed_certs",
  "verification_method":"txt", 
  "tls_policy":"recommended",
  "custom_client_ip_header":"true-client-ip"
}'
```

Una solicitud exitosa devuelve un identificador que se usa para verificar el <Tooltip tip="Dominio personalizado: dominio de terceros con un nombre especializado o personalizado." cta="Ver glosario" href="/docs/es/glossary?term=custom+domain">dominio personalizado</Tooltip>. Para obtener más información, consulta la documentación de la API de [Configurar un nuevo dominio personalizado](https://auth0.com/docs/api/management/v2/custom-domains/post-custom-domains).

<div id="patch-an-existing-custom-domain">
  ### Modificar un dominio personalizado existente
</div>

Puede configurar un dominio personalizado existente para que acepte encabezados mTLS con la [Management API](https://auth0.com/docs/api/management/v2). Sin embargo, no puede actualizar el `type` de un dominio personalizado existente.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Solo los dominios personalizados con el tipo `self_managed_certs` pueden usarse para mTLS. Actualmente, Auth0 no admite el tipo `auth0_managed_certs` para mTLS.
</Callout>

La siguiente solicitud POST configura un dominio personalizado existente para que acepte encabezados mTLS:

```bash lines
curl --location --request POST 'https://$tenant/api/v2/custom-domains/:id' \
  --header 'Authorization: Bearer $management_access_token' \
  --header 'Content-Type: application/json' \
  --data-raw '{
  "tls_policy":"recommended",
  "custom_client_ip_header":"true-client-ip"
}'
```

Para obtener más información, consulta la documentación de la API para [actualizar la configuración del dominio personalizado](https://auth0.com/docs/api/management/v2/custom-domains/patch-custom-domains-by-id).

<div id="verify-the-custom-domain">
  ### Verificar el dominio personalizado
</div>

Antes de que Auth0 acepte solicitudes para crear y actualizar el dominio personalizado, primero debe verificarlo. Usa la [Management API](https://auth0.com/docs/api/management/v2) para enviar la siguiente solicitud POST y verificar el dominio personalizado:

```bash wrap lines
curl --location --request POST 'https://$tenant/api/v2/custom-domains/:id/verify'
```

Consulta el campo `status` para ver su estado de verificación. Una vez completada la verificación, pueden pasar hasta 10 minutos antes de que el dominio personalizado empiece a aceptar solicitudes.

Cuando Auth0 verifica el dominio personalizado por primera vez, la respuesta incluye la `cname_api_key`, que necesitas para configurar tu proxy perimetral/inverso. Esta clave debe mantenerse en secreto y se utiliza para validar las solicitudes reenviadas.

Para obtener más información, consulta la documentación de la API [Verify a custom domain](https://auth0.com/docs/api/management/v2/custom-domains/patch-custom-domains-by-id).

<div id="enable-mtls-endpoint-aliases">
  ### Habilitar alias de endpoint de mTLS
</div>

Cuando el handshake de mTLS solicita un certificado de cliente, el navegador web muestra a los usuarios un cuadro de diálogo modal para seleccionar un certificado. Esto genera fricción en la experiencia de usuario y debe evitarse en endpoints donde mTLS no es necesario, como el endpoint `/authorize`. Por ello, los clientes que admiten tráfico mTLS y no mTLS en dominios diferentes deben habilitar los alias de endpoint de mTLS.

Los alias de endpoint de mTLS indican que los clientes deben enviar el tráfico mTLS a los endpoints especificados en la propiedad `mtls_endpoint_aliases` del documento de descubrimiento de OIDC. Los clientes enviarán el tráfico no mTLS a los endpoints normales. Para obtener más información sobre la propiedad `mtls_endpoint_aliases`, consulta [Llamar al servidor de recursos](/docs/es/get-started/authentication-and-authorization-flow/authenticate-with-mtls#call-the-resource-server).

Puedes habilitar los alias de endpoint de mTLS con el <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip> y la <Tooltip tip="Auth0 Dashboard: el producto principal de Auth0 para configurar tus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip>.

<Tabs>
  <Tab title="Auth0 Dashboard">
    Para habilitar los alias de endpoint de mTLS con Auth0 Dashboard:

    1. Ve a [Auth0 Dashboard](https://manage.auth0.com/#/tenant/general) y selecciona **Settings** en el menú lateral.
    2. En Tenant Settings, selecciona la pestaña **Advanced**.
    3. Busca **Allow mTLS Endpoint Aliases** y actívalo. Esto redirigirá el tráfico mTLS a un endpoint llamado `mtls.<your custom domain>`**.**
  </Tab>

  <Tab title="Management API">
    Para habilitar los alias de endpoint de mTLS con la [Management API](https://auth0.com/docs/api/management/v2), establece la propiedad `enable_endpoint_aliases` en `true` para tu tenant:

    ```bash lines
    curl --location --request PATCH 'https://$tenant/api/v2/tenants/settings' \
      --header 'Authorization: Bearer $management_access_token' \
      --header 'Content-Type: application/json' \
      --data-raw '{
      "mtls": {
        "enable_endpoint_aliases": true
      }
    }'
    ```

    Los alias de endpoint de mTLS agregan el prefijo `mtls.` al dominio personalizado configurado. Por ejemplo, si el dominio personalizado configurado es `auth.saasapp.com`, los alias de endpoint de mTLS usarán el dominio `mtls.auth.saasapp.com`. Según los comentarios recibidos, es posible que en el futuro los clientes puedan configurar los alias de endpoint de mTLS.

    <Warning>
      El subdominio de mTLS está fuera del control de Auth0. Los administradores deben conocer los riesgos de los subdominios y actuar en consecuencia. Esto incluye, entre otras cosas:

      * Garantizar la propiedad y la capacidad de administrar el dominio de nivel superior y los subdominios utilizados con mTLS. De lo contrario, podría perderse tanto el controlador de inicio de sesión como el subdominio de mTLS. Esto es especialmente importante al usar [dominios personalizados](/docs/es/customize/custom-domains).
      * Asegúrate de tener un certificado SSL para el subdominio de mTLS; por ejemplo, `mtls.auth.saasapp.com`. Un certificado comodín para `*.saasapp.com` no incluye el subdominio de mTLS.

      Evitar la apropiación de subdominios debido a entradas DNS colgantes si una IP ya no está en uso. Para obtener más información, consulta la documentación de Mozilla Developer Network (MDN) sobre [Subdomain takeovers](https://developer.mozilla.org/en-US/docs/Web/Security/Subdomain_takeovers).
    </Warning>

    Para deshabilitar los alias de endpoint de mTLS, establece el valor de `enable_endpoint_aliases` en `false`. Para obtener más información, consulta la documentación de la API [Update tenant settings](https://auth0.com/docs/api/management/v2/tenants/patch-settings).
  </Tab>
</Tabs>
