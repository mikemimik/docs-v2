title: Configurar Customer Edge
---

Esta sección explica cómo configurar la red perimetral del cliente. Los detalles específicos de la configuración de distintas redes perimetrales quedan fuera del alcance de este documento. Para obtener más información, consulta la documentación de [dominios personalizados](/docs/es/customize/custom-domains).

El dominio perimetral del cliente debe coincidir con el <Tooltip tip="Dominio personalizado: Dominio de terceros con un nombre especializado o personalizado." cta="Ver glosario" href="/docs/es/glossary?term=custom+domain">dominio personalizado</Tooltip> registrado. Si los alias de endpoint de mTLS están habilitados, el Customer Edge también debe aceptar solicitudes en el subdominio que usan los alias de mTLS. Todas las solicitudes que lleguen al subdominio de mTLS deben solicitar un certificado de cliente TLS y verificar que el certificado esté registrado para su uso. Para obtener más información, consulta [Verificar el certificado de cliente](#verify-the-client-certificate).

El handshake de mTLS se produce antes de que la infraestructura perimetral pueda determinar la ruta solicitada. Una vez establecida la sesión TLS, el cliente puede querer inspeccionar la ruta y reenviar solo las solicitudes a los siguientes endpoints:

* `/oauth/token`
* `/oauth/par`
* `/userinfo`

Si tu implementación busca cumplir con FAPI, tu red perimetral debe cumplir requisitos adicionales relacionados con TLS. Para obtener más información, consulta las especificaciones [FAPI1 Baseline](https://openid.net/specs/openid-financial-api-part-1-1_0.html#tls-and-dnssec-considerations), [FAPI1 Advanced](https://openid.net/specs/openid-financial-api-part-2-1_0.html#tls-considerations) y [FAPI2 Baseline](https://openid.net/specs/fapi-2_0-baseline.html#name-network-layer-protections).

<div id="verify-the-client-certificate">
  ## Verificar el certificado de cliente
</div>

La red perimetral del cliente realiza validaciones que dependen del tipo de certificado de cliente esperado. Para evitar problemas de seguridad y errores frecuentes, use una biblioteca de validación de certificados de confianza siempre que sea posible. Si el certificado de cliente no supera la validación, el comportamiento esperado depende de la instalación y queda fuera del alcance de este documento.

<div id="ca-signed-certificates">
  ### Certificados firmados por una autoridad de certificación
</div>

Cuando un certificado está firmado por una autoridad de certificación, debe verificarse la cadena de confianza del certificado, incluido el certificado raíz. El certificado también puede compararse con una lista de permitidos o de bloqueo para garantizar que esté registrado y no haya sido revocado. Recomendamos que **no** utilice una autoridad de certificación pública para firmar sus certificados de cliente, ya que esto puede aumentar el riesgo de que se hagan pasar por sus aplicaciones cliente.

<div id="self-signed-certificates">
  ### Certificados autofirmados
</div>

Los certificados autofirmados no están respaldados por una cadena de confianza, por lo que no es posible verificar la cadena de certificados. En su lugar, se puede comprobar la huella digital del certificado con una base de datos registrada de certificados o reenviarla directamente a Auth0 para realizar esta comprobación.

<div id="forward-the-request">
  ## Reenviar la solicitud
</div>

Una vez verificado el certificado, las solicitudes se reenvían junto con varios encabezados especiales del cliente al mismo endpoint del destino de reenvío del dominio personalizado en la [red perimetral de Auth0](/docs/es/customize/custom-domains/self-managed-certificates). La solicitud reenviada debe incluir los siguientes encabezados:

* La clave API de dominio personalizado en el encabezado `cname-api-key`.
* El certificado de cliente en el encabezado `client-certificate`. **Nota**: Como los encabezados HTTP deben ser texto, el certificado debe convertirse a un PEM codificado como componente de URL. El valor del encabezado está limitado a 4096 bytes. Por lo tanto, solo debe reenviarse a Auth0 el primer certificado de la cadena.
* El estado de verificación por la autoridad de certificación del certificado de cliente en el encabezado `client-certificate-ca-verified`. El encabezado `client-certificate-ca-verified` puede tener los siguientes valores:

  * **SUCCESS**: indica que el certificado de cliente es válido y ha sido verificado por una autoridad de certificación.
  * **FAILED:** indica que el certificado de cliente presentado es válido; sin embargo, la cadena de confianza del certificado NO ha sido verificada por una autoridad de certificación. En otras palabras, es un certificado autofirmado. Puede incluir un motivo de error opcional.

Si tu configuración admite solo certificados firmados por una autoridad de certificación, no necesitas reenviar certificados autofirmados a la red perimetral de Auth0 y puedes finalizar la solicitud antes. Sin embargo, si tus clientes están configurados para autenticarse mediante certificados autofirmados, Auth0 espera que el borde de tu red envíe un encabezado `client-certificate-ca-verified:FAILED`. Según el valor de este encabezado, Auth0 sabe qué método de autenticación del cliente se ha utilizado y qué credenciales del cliente deben verificarse.
