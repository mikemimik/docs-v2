title: Configurar Customer Edge
---

Esta sección explica cómo configurar la red perimetral del cliente. Los detalles específicos de la configuración de distintas redes perimetrales quedan fuera del alcance de este documento. Para obtener más información, consulta la documentación sobre [dominios personalizados](/docs/es/customize/custom-domains).

El dominio perimetral del cliente debe coincidir con el <Tooltip tip="Dominio personalizado: dominio de terceros con un nombre especializado o personalizado." cta="Ver glosario" href="/docs/es/glossary?term=custom+domain">dominio personalizado</Tooltip> registrado. Si los alias de endpoints mTLS están habilitados, Customer Edge también debe aceptar solicitudes en el subdominio que usan los alias mTLS. Todas las solicitudes que lleguen al subdominio mTLS deben solicitar un certificado TLS del cliente y verificar que el certificado esté registrado para su uso. Para obtener más información, consulta [Verificar el certificado del cliente](#verify-the-client-certificate).

El handshake mTLS se produce antes de que la infraestructura perimetral pueda determinar la ruta solicitada. Una vez establecida la sesión TLS, es posible que el cliente quiera inspeccionar la ruta y reenviar solo las solicitudes a los siguientes endpoints:

* `/oauth/token`
* `/oauth/par`
* `/userinfo`

Si tu instalación busca cumplir con FAPI, se imponen requisitos adicionales relacionados con TLS en tu red perimetral. Para obtener más información, consulta las especificaciones [FAPI1 Baseline](https://openid.net/specs/openid-financial-api-part-1-1_0.html#tls-and-dnssec-considerations), [FAPI1 Advanced](https://openid.net/specs/openid-financial-api-part-2-1_0.html#tls-considerations) y [FAPI2 Baseline](https://openid.net/specs/fapi-2_0-baseline.html#name-network-layer-protections).

<div id="verify-the-client-certificate">
  ## Verificar el certificado del cliente
</div>

La red perimetral del cliente realiza validaciones que dependen del tipo esperado del certificado del cliente. Para evitar problemas de seguridad comunes y errores frecuentes, use una biblioteca de validación de certificados bien probada siempre que sea posible. Cuando el certificado del cliente no supera la validación, el comportamiento esperado depende de la instalación y queda fuera del alcance de este documento.

<div id="ca-signed-certificates">
  ### Certificados firmados por una autoridad de certificación
</div>

Cuando un certificado está firmado por una autoridad de certificación, debe verificarse su cadena de confianza, incluido el certificado raíz. El certificado también puede compararse con una lista de permitidos o de denegación para garantizar que esté registrado y no haya sido revocado. Le recomendamos que **no** utilice una autoridad de certificación pública para firmar sus certificados del cliente, ya que esto puede aumentar el riesgo de que se hagan pasar por sus aplicaciones cliente.

<div id="self-signed-certificates">
  ### Certificados autofirmados
</div>

Los certificados autofirmados no están respaldados por una cadena de confianza, por lo que no se puede validar una cadena de certificados. En su lugar, se puede comprobar la huella digital del certificado con una base de datos de certificados registrados o reenviarla directamente a Auth0 para realizar esta comprobación.

<div id="forward-the-request">
  ## Reenviar la solicitud
</div>

Una vez verificado el certificado, las solicitudes se reenvían, junto con varios encabezados especiales del cliente, al mismo endpoint del destino de reenvío del dominio personalizado en la [red perimetral de Auth0](/docs/es/customize/custom-domains/self-managed-certificates). La solicitud reenviada debe incluir los siguientes encabezados:

* La clave de API de dominio personalizado en el encabezado `cname-api-key`.
* El certificado del cliente en el encabezado `client-certificate`. **Nota**: Como los encabezados HTTP deben ser texto, el certificado debe convertirse a PEM codificado como componente de URL. El valor del encabezado está limitado a 4096 bytes. Por lo tanto, solo debe reenviarse a Auth0 el primer certificado de la cadena.
* El estado de verificación por CA del certificado del cliente en el encabezado `client-certificate-ca-verified`. El encabezado `client-certificate-ca-verified` puede tener los siguientes valores:

  * **SUCCESS**: indica que el certificado del cliente es válido y ha sido verificado por una autoridad de certificación.
  * **FAILED:** indica que el certificado del cliente presentado es válido; sin embargo, la cadena de confianza del certificado NO ha sido verificada por una autoridad de certificación. En otras palabras, es un certificado autofirmado. Puede incluir un motivo de error opcional.

Si tu configuración solo admite certificados firmados por una autoridad de certificación, no necesitas reenviar certificados autofirmados a la red perimetral de Auth0 y puedes dar por terminada la solicitud antes. Sin embargo, si tus clientes están configurados para autenticarse mediante certificados autofirmados, Auth0 espera que tu perímetro de red envíe un encabezado `client-certificate-ca-verified:FAILED`. Según el valor de este encabezado, Auth0 sabe qué método de autenticación del cliente se ha utilizado y con qué credenciales del cliente debe verificarse.
