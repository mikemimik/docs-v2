description: Aprende a configurar el edge del cliente para la autenticación mTLS.
title: Configurar el Customer Edge
---

En esta sección se explica cómo configurar la red perimetral del cliente. Los detalles de la configuración de las distintas redes perimetrales quedan fuera del alcance de este documento. Para obtener más información, consulta la documentación sobre [dominios personalizados](/docs/es/customize/custom-domains).

El dominio edge del cliente debe coincidir con el <Tooltip tip="Dominio personalizado: dominio de terceros con un nombre especializado o personalizado." cta="Ver glosario" href="/docs/es/glossary?term=custom+domain">dominio personalizado</Tooltip> registrado. Si los alias de endpoint de mTLS están habilitados, el edge del cliente también debe aceptar solicitudes en el subdominio que utilizan los alias de mTLS. Todas las solicitudes que lleguen al subdominio de mTLS deben requerir un certificado TLS de cliente y verificar que el certificado esté registrado para su uso. Para obtener más información, consulta [Verificar el certificado del cliente](#verify-the-client-certificate).

El handshake de mTLS se produce antes de que la infraestructura edge pueda determinar la ruta solicitada. Una vez establecida la sesión TLS, es posible que el cliente quiera inspeccionar la ruta y reenviar solo las solicitudes a los siguientes endpoints:

* `/oauth/token`
* `/oauth/par`
* `/userinfo`

Si tu instalación busca cumplir con FAPI, tu red edge debe cumplir requisitos adicionales relacionados con TLS. Para obtener más información, consulta las especificaciones [FAPI1 Baseline](https://openid.net/specs/openid-financial-api-part-1-1_0.html#tls-and-dnssec-considerations), [FAPI1 Advanced](https://openid.net/specs/openid-financial-api-part-2-1_0.html#tls-considerations) y [FAPI2 Baseline](https://openid.net/specs/fapi-2_0-baseline.html#name-network-layer-protections).

<div id="verify-the-client-certificate">
  ## Verifique el certificado del cliente
</div>

La red perimetral del cliente realiza validaciones que dependen del tipo de certificado de cliente esperado. Para evitar problemas de seguridad y errores habituales, use, siempre que sea posible, una biblioteca de validación de certificados de confianza. Cuando el certificado del cliente no supera la validación, el comportamiento esperado depende de la instalación y queda fuera del alcance de este documento.

<div id="ca-signed-certificates">
  ### Certificados firmados por una CA
</div>

Cuando un certificado está firmado por una autoridad de certificación, debe verificarse su cadena de confianza, incluido el certificado raíz. El certificado también puede compararse con una lista de certificados permitidos o denegados para garantizar que esté registrado y no haya sido revocado. Recomendamos que **no** utilice una autoridad de certificación pública para firmar sus certificados de cliente, ya que esto puede aumentar el riesgo de suplantación de sus aplicaciones cliente.

<div id="self-signed-certificates">
  ### Certificados autofirmados
</div>

Los certificados autofirmados no están respaldados por una cadena de confianza, por lo que no es posible verificar su cadena de certificados. En su lugar, la huella digital del certificado puede comprobarse con una base de datos de certificados registrados o reenviarse directamente a Auth0 para realizar esta comprobación.

<div id="forward-the-request">
  ## Reenviar la solicitud
</div>

Una vez verificado el certificado, las solicitudes se reenvían junto con varios encabezados especiales del cliente al mismo endpoint en el destino de reenvío del dominio personalizado en la [red perimetral de Auth0](/docs/es/customize/custom-domains/self-managed-certificates). La solicitud reenviada debe incluir los siguientes encabezados:

* La clave de API del dominio personalizado en el encabezado `cname-api-key`.
* El certificado del cliente en el encabezado `client-certificate`. **Nota**: Como los encabezados HTTP deben ser texto, el certificado debe convertirse a un PEM codificado como componente de URL. El valor del encabezado está limitado a 4096 bytes. Por lo tanto, solo debe reenviarse a Auth0 el primer certificado de la cadena.
* El estado de verificación de la CA del certificado del cliente en el encabezado `client-certificate-ca-verified`. El encabezado `client-certificate-ca-verified` puede tener los siguientes valores:

  * **SUCCESS**: indica que el certificado del cliente es válido y ha sido verificado por una autoridad de certificación.
  * **FAILED:** indica que el certificado del cliente presentado es válido; sin embargo, la cadena de confianza del certificado NO ha sido verificada por una autoridad de certificación. En otras palabras, es un certificado autofirmado. Puede incluir un motivo de error opcional.

Si su configuración solo admite certificados firmados por una CA, no necesita reenviar certificados autofirmados a la red perimetral de Auth0 y puede finalizar la solicitud antes. Sin embargo, si sus clientes están configurados para autenticarse con certificados autofirmados, Auth0 espera que el perímetro de su red envíe un encabezado `client-certificate-ca-verified:FAILED`. Según el valor de este encabezado, Auth0 sabe qué método de autenticación del cliente se ha utilizado y con qué credenciales del cliente debe compararse la verificación.
