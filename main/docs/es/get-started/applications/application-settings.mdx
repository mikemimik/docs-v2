En la página [Applications](https://manage.auth0.com/#/applications) del Dashboard, busca tu aplicación en la lista y haz clic en su nombre para ver la configuración disponible.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Las [aplicaciones de terceros](/docs/es/get-started/applications/third-party-applications) tienen un conjunto limitado de propiedades configurables. Las propiedades que no formen parte del conjunto admitido no se pueden configurar desde el Auth0 Dashboard ni mediante la Management API. Para obtener más información, consulta [Controles de seguridad para aplicaciones de terceros](/docs/es/get-started/applications/third-party-applications/security-controls#restricted-client-configuration).
</Callout>

<Frame>![Lista de aplicaciones del Dashboard](/docs/images/cdy7uua7fh8z/1ecNwGgFQZxdP57p0tp3jT/cd608fcfae22e195b604e2707e5a848d/App_List_-_EN.png)</Frame>

<div id="basic-settings">
  ## Configuración básica
</div>

Cuando editas la configuración de una aplicación existente o creas una nueva aplicación, introduces información sobre la aplicación en la vista **Configuración**.

<div id="basic-information">
  ### Información básica
</div>

<Frame>![Dashboard Applications Application Settings Tab Basic Information](/docs/images/cdy7uua7fh8z/2GPUw7BODYuYYH3658Upz3/92a49ec57e6b4d07be96093989baac03/2023-04-11_15-34-58.png)</Frame>

* **Nombre**: El nombre de tu aplicación. Es editable y aparecerá en el portal, los correos electrónicos, los registros, etc.
* **Dominio**: El nombre de tu tenant de Auth0. Lo eliges cuando creas un nuevo tenant de Auth0 y no se puede cambiar. Si necesitas un dominio diferente, debes registrar un nuevo tenant seleccionando **+ Create Tenant** en el menú superior derecho.
* **<Tooltip tip="ID de cliente: valor de identificación que Auth0 asigna a tu recurso registrado." cta="Ver glosario" href="/docs/es/glossary?term=Client+ID">ID de cliente</Tooltip>**: El identificador único de tu aplicación. Lo usarás al configurar la autenticación con Auth0. El sistema lo genera cuando creas una nueva aplicación y no se puede modificar.
* **<Tooltip tip="Secreto de cliente: secreto que usa un cliente (aplicación) para autenticarse con el servidor de autorización; solo deben conocerlo el cliente y el servidor de autorización, y debe ser lo suficientemente aleatorio como para que no se pueda adivinar." cta="Ver glosario" href="/docs/es/glossary?term=Client+Secret">secreto de cliente</Tooltip>**: Una cadena que se usa para firmar y validar <Tooltip tip="Secreto de cliente: secreto que usa un cliente (aplicación) para autenticarse con el servidor de autorización; solo deben conocerlo el cliente y el servidor de autorización, y debe ser lo suficientemente aleatorio como para que no se pueda adivinar." cta="Ver glosario" href="/docs/es/glossary?term=ID+Tokens">tokens de ID</Tooltip> en los flujos de autenticación y para obtener acceso a determinados endpoints de la API de Auth0. De forma predeterminada, el valor está oculto, así que marca la casilla **Reveal Client Secret** para verlo. Aunque el ID de cliente se considera información pública, el secreto de cliente **debe mantenerse confidencial**. Si alguien puede acceder a tu secreto de cliente, puede emitir tokens y acceder a recursos a los que no debería tener acceso.
* **Descripción**: Una descripción en texto libre del propósito de la aplicación. Máximo 140 caracteres.

<div id="application-properties">
  ### Propiedades de la aplicación
</div>

<Frame>![Dashboard Aplicaciones Pestaña Configuración de la aplicación Propiedades de la aplicación](/docs/images/cdy7uua7fh8z/4Z66jnQOVFwNo5iuHtdlxo/153fabafccac939aa15c23a65ba974cb/Application_Properties_-_English.png)</Frame>

* **Logotipo de la aplicación**: La URL de un logotipo (tamaño recomendado: 150x150 píxeles) que se mostrará para la aplicación. Aparece en varias áreas, incluida la lista de aplicaciones en el Dashboard y los formularios de consentimiento personalizado. Si no se establece ninguno, se mostrará la insignia predeterminada para este tipo de aplicación.
* **Propiedad de la aplicación**: Indica si la aplicación es propia o de terceros. Las aplicaciones de terceros están sujetas a [controles de seguridad reforzados](/docs/es/get-started/applications/third-party-applications/security-controls). La propiedad de la aplicación se establece al crearla y no puede cambiarse. Para obtener más información, consulta [aplicaciones propias y de terceros](/docs/es/get-started/applications/first-party-and-third-party-applications).
* **Tipo de aplicación**: El tipo de aplicación de Auth0 determina qué opciones puedes configurar desde el Dashboard. (No se puede editar para aplicaciones de máquina a máquina. A veces también está deshabilitado para otros tipos de aplicación de Auth0 si los tipos de concesión seleccionados solo están permitidos para el tipo de aplicación actualmente seleccionado). Usa el menú desplegable para seleccionar uno de los siguientes tipos:

  * [Machine to Machine](/docs/es/get-started/auth0-overview/create-applications/machine-to-machine-apps): Aplicaciones no interactivas, como herramientas de línea de comandos, daemons, dispositivos IoT o servicios que se ejecutan en tu backend. Normalmente, usarás esta opción si tienes un servicio que requiere acceso a una API.
  * [Native App](/docs/es/get-started/auth0-overview/create-applications/native-apps): Aplicaciones móviles o de escritorio que se ejecutan de forma nativa en un dispositivo (como iOS o Android).
  * [Regular Web App](/docs/es/get-started/auth0-overview/create-applications/regular-web-apps): Aplicaciones web tradicionales que realizan la mayor parte de su lógica en el servidor (como Express.js o ASP.NET).
  * [Single Page App](/docs/es/get-started/auth0-overview/create-applications/single-page-web-apps): Aplicaciones JavaScript que realizan la mayor parte de la lógica de su interfaz de usuario en un navegador web y se comunican con un servidor web principalmente mediante API (como AngularJS + Node.js o React).

<div id="application-uris">
  ### URI de la aplicación
</div>

<Frame>![Dashboard Applications Application Settings Application URIs](/docs/images/cdy7uua7fh8z/1QhW2i4fTCCp8owey2tMPI/a19f7fc3f84118920d0e5cb2357144da/Application_URIs.png)</Frame>

* **Application Login URI**: En algunos casos, Auth0 necesitará que su aplicación redirija a la página de login de la aplicación. Este URI debe apuntar a una ruta de su aplicación que redirija al endpoint `/authorize` de su tenant [endpoint](https://auth0.com/docs/api/authentication#authorize-application). Normalmente, tendría la forma `https://myapp.org/login`. Puede usar los siguientes marcadores de posición en este campo:

  * [**Marcadores de posición de metadatos de la organización**](/docs/es/get-started/applications/wildcards-for-subdomains#organization-url-placeholders): Use `{organization.metadata.KEY}` para completar dinámicamente la URL en función de los metadatos de la Auth0 Organization asociada con la solicitud (por ejemplo, `https://{organization.metadata.public_login_host}/login`).
  * [**Marcadores de posición de dominio personalizado**](/docs/es/get-started/applications/wildcards-for-subdomains#custom-domain-url-placeholders): Use `{custom_domain.metadata.KEY}` para completar dinámicamente la URL en función de los metadatos del dominio personalizado usado en la solicitud (por ejemplo, `https://{custom_domain.metadata.public_app_host}/login`).

  Para obtener más información, consulte [Configure Default Login Routes](/docs/es/authenticate/login/auth0-universal-login/configure-default-login-routes#dynamic-login-uris-with-metadata-placeholders).
* **Allowed Callback URLs**: Conjunto de URL a las que Auth0 puede redirigir a los usuarios después de autenticarse. Puede especificar varias URL válidas separándolas con comas (normalmente, para distintos entornos, como QA o pruebas). En entornos de producción, verifique que las URL no apunten a localhost. Puede usar los siguientes marcadores de posición en este campo:
  * [**Comodines**](/docs/es/get-started/applications/wildcards-for-subdomains#wildcard-url-placeholders): Use `*` para subdominios (`*.google.com`) *No se recomienda para entornos de producción.*
  * [**Marcadores de posición de organización**](/docs/es/get-started/applications/wildcards-for-subdomains#organization-url-placeholders): Use `{organization_name}` para indicar dinámicamente el nombre de una organización registrada (por ejemplo, `https://{organization_name}.example.com`).
  * [**Marcadores de posición de dominio personalizado**](/docs/es/get-started/applications/wildcards-for-subdomains#custom-domain-url-placeholders): Use `{custom.domain.metadata.KEY}` para completar dinámicamente la URL en función de los metadatos del dominio personalizado usado en la solicitud (por ejemplo, `https://{custom_domain.metadata.public_app_url}/callback`).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  La primera URL incluida en este campo se toma como la URL de devolución de llamada predeterminada cuando el flujo de protocolo correspondiente no especifica una de forma explícita. Esto se aplica específicamente a los flujos SAML, WS-Fed y SSO iniciado por IdP de SAML.
</Callout>

No use marcadores de posición comodín ni URL de localhost en las devoluciones de llamada de su aplicación ni en los campos de orígenes permitidos. El uso de URL de redirección con marcadores de posición comodín puede hacer que su aplicación sea vulnerable a ataques. Para obtener más información, consulte [Unvalidated Redirects and Forwards Cheat Sheet on owasp.org](https://www.owasp.org/index.php/Unvalidated_Redirects_and_Forwards_Cheat_Sheet). En su lugar, cuando corresponda, se deben priorizar las URL con el marcador de posición `{organization_name}`. Para obtener más información, consulte [Subdomain URL Placeholders](/docs/es/get-started/applications/wildcards-for-subdomains).

* **Allowed Logout URLs**: Después de que un usuario cierra sesión en Auth0, puedes redirigirlo con el parámetro de consulta `returnTo`. La URL que uses en `returnTo` debe figurar aquí. Puedes especificar varias URL válidas separándolas con comas. En entornos de producción, verifica que las URL no apunten a localhost.
  * [**Comodines**](/docs/es/get-started/applications/wildcards-for-subdomains#wildcard-url-placeholders): Usa `*` para subdominios (`*.google.com`) *No se recomienda para entornos de producción.*
  * [**Marcadores de posición de dominio personalizado**](/docs/es/get-started/applications/wildcards-for-subdomains#custom-domain-url-placeholders): Usa `{custom.domain.metadata.KEY}` para completar dinámicamente la URL según los metadatos del dominio personalizado usado en la solicitud (por ejemplo, `https://{custom_domain.metadata.public_app_url}/callback`).
* **Allowed Web Origins**: Lista de URL desde las que puede originarse una solicitud de autorización que use [Autenticación de origen cruzado](/docs/es/authenticate/login/cross-origin-authentication), [Flujo de dispositivo](/docs/es/get-started/authentication-and-authorization-flow/device-authorization-flow) y `web_message` como modo de respuesta. Puedes especificar varias URL válidas separándolas con comas. En entornos de producción, verifica que las URL no apunten a localhost. Las rutas, las cadenas de consulta y la información de hash no se tienen en cuenta al validar estas URL (y, de hecho, pueden hacer que la coincidencia falle). Puedes proporcionar hasta 100 URL en el campo **Allowed Web Origins**.
  * [**Comodines**](/docs/es/get-started/applications/wildcards-for-subdomains#wildcard-url-placeholders): Usa `*` para subdominios (`*.google.com`) *No se recomienda para entornos de producción.*
  * [**Marcadores de posición de dominio personalizado**](/docs/es/get-started/applications/wildcards-for-subdomains#custom-domain-url-placeholders): Usa `{custom.domain.metadata.KEY`&#125; para completar dinámicamente la URL según los metadatos del dominio personalizado usado en la solicitud (por ejemplo, `https://{custom_domain.metadata.public_app_url}/callback`).
* **Allowed Origins (CORS)**: Lista de URL autorizadas para realizar solicitudes de uso compartido de recursos entre orígenes (CORS) a Auth0.
  * [**Marcadores de posición de dominio personalizado**](/docs/es/get-started/applications/wildcards-for-subdomains#custom-domain-url-placeholders): Usa `{custom.domain.metadata.KEY}` para completar dinámicamente la URL según los metadatos del dominio personalizado usado en la solicitud (por ejemplo, `https://{custom_domain.metadata.public_app_url}/callback`).

<Warning>
  Si configuras las URL de tu aplicación exclusivamente con marcadores de posición de dominio personalizado, las solicitudes de autenticación realizadas a través del dominio canónico de tu tenant (por ejemplo, https://your-tenant.us.auth0.com) fallarán.

  Esto ocurre porque el dominio canónico no tiene los metadatos personalizados necesarios para resolver el marcador de posición. Asegúrate de que tu aplicación use el dominio personalizado específico para la autenticación, o proporciona una URL de respaldo estática si es necesario usar el dominio canónico.
</Warning>

<div id="id-token">
  ### Token de ID
</div>

En la sección **Token de ID**, introduce la **Expiración del token de ID** (en segundos), que indica cuánto tiempo transcurre antes de que caduque el `id_token` de Auth0. El valor predeterminado es de 36000 segundos, es decir, 10 horas.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  **Usa Auth0 en lugar del IdP para el inicio de sesión único**: Si se habilita, esta configuración evita que Auth0 redirija a los usuarios autenticados con sesiones válidas al proveedor de identidad (como Facebook o ADFS). **Solo para tenants Legacy.**
</Callout>

<div id="refresh-token-rotation">
  ### Rotación del token de actualización
</div>

En la sección **Rotación del <Tooltip tip="Token de actualización: token que se utiliza para obtener un token de acceso renovado sin obligar a los usuarios a iniciar sesión de nuevo." cta="Ver glosario" href="/docs/es/glossary?term=Refresh+Token">token de actualización</Tooltip>**, active o desactive la rotación. Cuando está activada, al intercambiar un token de actualización, se emitirá un nuevo token de actualización y el token existente se invalidará. Esto permite detectar automáticamente la reutilización del token si se filtra. Además, introduzca el **Período de superposición de la rotación** (en segundos). Este intervalo es el margen de tiempo permitido durante el cual el mismo `refresh_token` puede usarse para solicitar un `access_token` sin activar la detección automática de reutilización. Para obtener más información, lea [Rotación del token de actualización](/docs/es/secure/tokens/refresh-tokens/refresh-token-rotation).

<Frame>![Dashboard Applications Applications pestaña Settings Rotación del token de actualización](/docs/images/cdy7uua7fh8z/Rk6i8MUVWZG34P7K6aurb/9457a6ea7ca5942522778c7ed7925023/Screenshot_2024-11-05_at_18.14.22.png)</Frame>

<div id="refresh-token-expiration">
  ### Expiración del token de actualización
</div>

En la sección **Expiración del token de actualización**, habilite o deshabilite la expiración absoluta y por inactividad, y configure el tiempo de validez (en segundos) de cada una. Para obtener más información, lea [Configurar la expiración del token de actualización](/docs/es/secure/tokens/refresh-tokens/configure-refresh-token-expiration).

<Frame>![Dashboard Aplicaciones Aplicaciones pestaña Configuración Expiración del token de actualización](/docs/images/cdy7uua7fh8z/2EnpIEc4kBxjnS3T67xPzC/22fdb95e2a2f31cee6f01e02d0a709c2/Refresh_Token_Expiration_-_English.png)</Frame>

<div id="open-redirect-protection">
  ### Protección contra redirecciones abiertas
</div>

Controla cómo Auth0 gestiona las redirecciones de las aplicaciones de terceros. Esta configuración solo está disponible para aplicaciones de terceros con controles de seguridad reforzados.

<Frame>![Interruptor de Protección contra redirecciones abiertas en el Dashboard](/docs/images/third-party-applications/open-redirect-protection.png)</Frame>

Cuando está habilitada (opción predeterminada para aplicaciones de terceros), Auth0 no redirige a la URL de devolución de llamada de la aplicación si se producen errores de autenticación y no expone `application.callback_domain` en las plantillas de correo electrónico. Esto evita ataques de redirección abierta cuando la URI de redirección está controlada por una parte no confiable.

Solo deshabilita la protección contra redirecciones abiertas para aplicaciones de terceros cuyas URI de devolución de llamada configuradas sean de confianza.

Para obtener más información, consulta [Protección de redirección](/docs/es/get-started/applications/third-party-applications/security-controls#redirect-protection).

<div id="advanced-settings">
  ## Configuración avanzada
</div>

La sección **Configuración avanzada** le permite:

* Administrar o agregar metadatos de la aplicación, device, <Tooltip tip="OAuth 2.0: Marco de autorización que define protocolos y flujos de trabajo de autorización." cta="Ver glosario" href="/docs/es/glossary?term=OAuth">OAuth</Tooltip> y la configuración de WS-Federation
* Obtener certificados e información del <Tooltip tip="Token Endpoint: Endpoint en el servidor de autorización que se utiliza para solicitar tokens de forma programática." cta="Ver glosario" href="/docs/es/glossary?term=Token+endpoint">Token endpoint</Tooltip>
* Establecer los tipos de concesión de la aplicación

<div id="application-metadata">
  ### Metadatos de la aplicación
</div>

Los metadatos de la aplicación son claves y valores de tipo cadena personalizados (cada uno con un máximo de 255 caracteres) que se establecen por aplicación. Los metadatos se exponen en el objeto de la aplicación como `client_metadata` y en las reglas como `context.clientMetadata`. Puede crear hasta 10 conjuntos de metadatos.

<div id="oauth">
  ### OAuth
</div>

<Frame>![Dashboard Applications Application Settings Tab Advanced Settings OAuth Tab](/docs/images/cdy7uua7fh8z/OAuth-tab.png)</Frame>

* De forma predeterminada, todas las aplicaciones/API pueden hacer una solicitud de delegación, pero si quieres conceder permisos explícitamente a determinadas aplicaciones/API, puedes hacerlo en **Allowed Apps/APIs**.
* Si eres cliente y usas el add-on Identidad altamente regulada, utiliza la configuración **Compliance Enforcement Level** para establecer tu nivel de cumplimiento. Para obtener más información, consulta [Configurar el cumplimiento de FAPI](/docs/es/get-started/applications/configure-fapi-compliance).
* **Non-Verifiable Callback URI End-User Confirmation**: Usa esta configuración para controlar si se le pide al usuario que confirme el inicio de sesión cuando se utiliza una URI no verificable como devolución de llamada. Auth0 recomienda no omitir la confirmación del usuario final en estos casos. Esta configuración tiene prioridad sobre la configuración del tenant con el mismo nombre. Para obtener más información, lee [Medidas contra la suplantación de aplicaciones](/docs/es/secure/security-guidance/measures-against-app-impersonation.mdx).
* Establece el algoritmo que se usará ([**HS256**](https://en.wikipedia.org/wiki/Symmetric-key_algorithm) o [**RS256**](https://en.wikipedia.org/wiki/Public-key_cryptography)) para firmar tus <Tooltip tip="JSON Web Token (JWT): formato estándar de token de ID (y, a menudo, también de access token) que se utiliza para representar claims de forma segura entre dos partes." cta="Ver glosario" href="/docs/es/glossary?term=JSON+web+tokens">JSON web tokens</Tooltip>. Para obtener más información, lee [Algoritmos de firma de JSON Web Token](/docs/es/get-started/applications/signing-algorithms). Al seleccionar `RS256` (recomendado), el token se firmará con la clave privada de tu tenant.
* Activa o desactiva la configuración **Trust Token Endpoint IP Header**; si está habilitada, `auth0-forwarded-for` se considera confiable y se usa como fuente de información sobre la IP del usuario final para proteger el Token endpoint contra ataques de fuerza bruta. Esta configuración solo está disponible para Regular Web Apps y aplicaciones M2M.
* Activa o desactiva el interruptor para indicar si tu aplicación es **OIDC Conformant** o no. Las aplicaciones marcadas como OIDC Conformant seguirán estrictamente la especificación OIDC.
