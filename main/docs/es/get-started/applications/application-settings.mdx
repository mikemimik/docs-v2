En la página [Applications](https://manage.auth0.com/#/applications) del Dashboard, busca tu aplicación en la lista y haz clic en su nombre para ver la configuración disponible.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Las [aplicaciones de terceros](/docs/es/get-started/applications/third-party-applications) tienen un conjunto limitado de propiedades configurables. Las propiedades que no estén incluidas en el conjunto compatible no se pueden configurar desde el Auth0 Dashboard ni mediante la Management API. Para obtener más información, consulta [Security Controls for Third-Party Applications](/docs/es/get-started/applications/third-party-applications/security-controls#restricted-client-configuration).
</Callout>

<div id="basic-settings">
  ## Configuración básica
</div>

Cuando editas la configuración de una aplicación existente o creas una nueva aplicación, introduces la información de la aplicación en la vista **Configuración**.

<div id="basic-information">
  ### Información básica
</div>

<Frame>![Dashboard Aplicaciones pestaña Configuración de la aplicación Información básica](/docs/images/cdy7uua7fh8z/2GPUw7BODYuYYH3658Upz3/92a49ec57e6b4d07be96093989baac03/2023-04-11_15-34-58.png)</Frame>

* **Name**: El nombre de tu aplicación. Se puede editar y aparecerá en el portal, los correos electrónicos, los registros, etc.
* **Domain**: El nombre de tu tenant de Auth0. Lo eliges cuando creas un nuevo tenant de Auth0 y no se puede cambiar. Si necesitas un dominio distinto, debes registrar un nuevo tenant seleccionando **+ Create Tenant** en el menú superior derecho.
* **<Tooltip tip="ID de cliente: valor de identificación que Auth0 asigna a tu recurso registrado." cta="Ver glosario" href="/docs/es/glossary?term=Client+ID">ID de cliente</Tooltip>**: El identificador único de tu aplicación. Lo usarás al configurar la autenticación con Auth0. El sistema lo genera cuando creas una nueva aplicación y no se puede modificar.
* **<Tooltip tip="Secreto de cliente: secreto que usa un cliente (aplicación) para autenticarse con el servidor de autorización; solo deberían conocerlo el cliente y el servidor de autorización, y debe ser lo suficientemente aleatorio como para que no se pueda adivinar." cta="Ver glosario" href="/docs/es/glossary?term=Client+Secret">secreto de cliente</Tooltip>**: Una cadena que se usa para firmar y validar <Tooltip tip="Secreto de cliente: secreto que usa un cliente (aplicación) para autenticarse con el servidor de autorización; solo deberían conocerlo el cliente y el servidor de autorización, y debe ser lo suficientemente aleatorio como para que no se pueda adivinar." cta="Ver glosario" href="/docs/es/glossary?term=ID+Tokens">tokens de ID</Tooltip> en los flujos de autenticación y para acceder a determinados endpoints de API de Auth0. De forma predeterminada, el valor está oculto, así que marca la casilla **Reveal Client Secret** para verlo. Aunque el ID de cliente se considera información pública, el secreto de cliente **debe mantenerse confidencial**. Si alguien puede acceder a tu secreto de cliente, podrá emitir tokens y acceder a recursos a los que no debería tener acceso.
* **Description**: Una descripción en texto libre de la finalidad de la aplicación. Máximo: 140 caracteres.

<div id="application-properties">
  ### Propiedades de la aplicación
</div>

<Frame>![Dashboard Applications Application Settings Tab Application Properties](/docs/images/cdy7uua7fh8z/4Z66jnQOVFwNo5iuHtdlxo/153fabafccac939aa15c23a65ba974cb/Application_Properties_-_English.png)</Frame>

* **Logotipo de la aplicación**: La URL de un logotipo (tamaño recomendado: 150 x 150 píxeles) que se mostrará para la aplicación. Aparece en varias partes, incluida la lista de aplicaciones del Dashboard y los formularios de consentimiento personalizado. Si no se configura ninguno, se mostrará la insignia predeterminada para este tipo de aplicación.
* **Propiedad de la aplicación**: Indica si la aplicación es propia o de terceros. Las aplicaciones de terceros están sujetas a [controles de seguridad reforzados](/docs/es/get-started/applications/third-party-applications/security-controls). La propiedad de la aplicación se establece al crearla y no puede cambiarse. Para obtener más información, consulta [aplicaciones propias y de terceros](/docs/es/get-started/applications/first-party-and-third-party-applications).
* **Tipo de aplicación**: El tipo de aplicación de Auth0 determina qué opciones de configuración puedes ajustar desde el Dashboard. (No se puede editar en las aplicaciones máquina a máquina. A veces también aparece deshabilitado para otros tipos de aplicación de Auth0 si los tipos de concesión seleccionados solo se permiten para el tipo de aplicación elegido actualmente). Usa la lista desplegable para seleccionar uno de los siguientes tipos:

  * [Máquina a máquina](/docs/es/get-started/auth0-overview/create-applications/machine-to-machine-apps): Aplicaciones no interactivas, como herramientas de línea de comandos, demonios, dispositivos IoT o servicios que se ejecutan en tu backend. Normalmente, usarás esta opción si tienes un servicio que necesita acceso a una API.
  * [Aplicación nativa](/docs/es/get-started/auth0-overview/create-applications/native-apps): Aplicaciones móviles o de escritorio que se ejecutan de forma nativa en un dispositivo (como iOS o Android).
  * [Aplicación web tradicional](/docs/es/get-started/auth0-overview/create-applications/regular-web-apps): Aplicaciones web tradicionales que ejecutan la mayor parte de su lógica en el servidor (como Express.js o ASP.NET).
  * [Aplicación web de página única](/docs/es/get-started/auth0-overview/create-applications/single-page-web-apps): Aplicaciones de JavaScript que ejecutan la mayor parte de la lógica de su interfaz de usuario en un navegador web y se comunican con un servidor web principalmente mediante API (como AngularJS + Node.js o React).

<div id="application-uris">
  ### URI de la aplicación
</div>

<Frame>![Dashboard Applications Application Settings URI de la aplicación](/docs/images/cdy7uua7fh8z/1QhW2i4fTCCp8owey2tMPI/a19f7fc3f84118920d0e5cb2357144da/Application_URIs.png)</Frame>

* **Application Login URI**: En algunos casos, Auth0 necesitará que su aplicación redirija a la página de inicio de sesión de la aplicación. Esta URI debe apuntar a una ruta de su aplicación que redirija al endpoint `/authorize` de su tenant [endpoint](https://auth0.com/docs/api/authentication#authorize-application). Normalmente, tendrá un formato como `https://myapp.org/login`. Puede usar los siguientes marcadores de posición en este campo:

  * [**Marcadores de posición de metadatos de la organización**](/docs/es/get-started/applications/wildcards-for-subdomains#organization-url-placeholders): Use `{organization.metadata.KEY}` para completar dinámicamente la URL en función de los metadatos de la Auth0 Organization asociada con la solicitud (por ejemplo, `https://{organization.metadata.public_login_host}/login`).
  * [**Marcadores de posición de dominio personalizado**](/docs/es/get-started/applications/wildcards-for-subdomains#custom-domain-url-placeholders): Use `{custom_domain.metadata.KEY}` para completar dinámicamente la URL en función de los metadatos del dominio personalizado utilizado en la solicitud (por ejemplo, `https://{custom_domain.metadata.public_app_host}/login`).

  Para obtener más información, consulte [Configure Default Login Routes](/docs/es/authenticate/login/auth0-universal-login/configure-default-login-routes#dynamic-login-uris-with-metadata-placeholders).
* **Allowed Callback URLs**: Conjunto de URL a las que Auth0 puede redirigir a los usuarios después de autenticarse. Puede especificar varias URL válidas separándolas con comas (normalmente, para gestionar distintos entornos, como QA o pruebas). En entornos de producción, verifique que las URL no apunten a localhost. Puede usar los siguientes marcadores de posición en este campo:
  * [**Comodines**](/docs/es/get-started/applications/wildcards-for-subdomains#wildcard-url-placeholders): Use `*` para subdominios (`*.google.com`) *No se recomienda para entornos de producción.*
  * [**Marcadores de posición de organización**](/docs/es/get-started/applications/wildcards-for-subdomains#organization-url-placeholders): Use `{organization_name}` para especificar dinámicamente el nombre de una organización registrada (por ejemplo, `https://{organization_name}.example.com`).
  * [**Marcadores de posición de dominio personalizado**](/docs/es/get-started/applications/wildcards-for-subdomains#custom-domain-url-placeholders): Use `{custom.domain.metadata.KEY}` para completar dinámicamente la URL en función de los metadatos del dominio personalizado utilizado en la solicitud (por ejemplo, `https://{custom_domain.metadata.public_app_url}/callback`).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  La primera URL incluida en este campo se toma como la URL de devolución de llamada predeterminada cuando el flujo de protocolo correspondiente no especifica una de forma explícita. Esto se aplica específicamente a los flujos SAML, WS-Fed e IdP-Initiated SSO de SAML.
</Callout>

No use marcadores de posición comodín ni URL de localhost en las devoluciones de llamada de su aplicación ni en los campos de orígenes permitidos. El uso de URL de redirección con marcadores de posición comodín puede hacer que su aplicación sea vulnerable a ataques. Para obtener más información, consulte [Unvalidated Redirects and Forwards Cheat Sheet on owasp.org](https://www.owasp.org/index.php/Unvalidated_Redirects_and_Forwards_Cheat_Sheet). En su lugar, cuando corresponda, es preferible usar URL con el marcador de posición `{organization_name}`. Para obtener más información, consulte [Subdomain URL Placeholders](/docs/es/get-started/applications/wildcards-for-subdomains).

* **Allowed Logout URLs**: Después de que un usuario cierre sesión en Auth0, puedes redirigirlo con el parámetro de consulta `returnTo`. La URL que uses en `returnTo` debe estar incluida aquí. Puedes especificar varias URL válidas separándolas con comas. En entornos de producción, verifica que las URL no apunten a localhost.
  * [**Comodines**](/docs/es/get-started/applications/wildcards-for-subdomains#wildcard-url-placeholders): Usa `*` para subdominios (`*.google.com`) *No se recomienda para entornos de producción.*
  * [**Marcadores de posición de dominio personalizado**](/docs/es/get-started/applications/wildcards-for-subdomains#custom-domain-url-placeholders): Usa `{custom.domain.metadata.KEY}` para completar dinámicamente la URL en función de los metadatos del dominio personalizado utilizado en la solicitud (por ejemplo, `https://{custom_domain.metadata.public_app_url}/callback`).
* **Allowed Web Origins**: Lista de URL desde las que puede originarse una solicitud de autorización que use [Cross-Origin Authentication](/docs/es/authenticate/login/cross-origin-authentication), [Device Flow](/docs/es/get-started/authentication-and-authorization-flow/device-authorization-flow) y `web_message` como modo de respuesta. Puedes especificar varias URL válidas separándolas con comas. En entornos de producción, verifica que las URL no apunten a localhost. La información de rutas, cadenas de consulta y hash no se tiene en cuenta al validar estas URL (y, de hecho, puede hacer que la coincidencia falle). Puedes proporcionar hasta 100 URL en el campo **Allowed Web Origins**.
  * [**Comodines**](/docs/es/get-started/applications/wildcards-for-subdomains#wildcard-url-placeholders): Usa `*` para subdominios (`*.google.com`) *No se recomienda para entornos de producción.*
  * [**Marcadores de posición de dominio personalizado**](/docs/es/get-started/applications/wildcards-for-subdomains#custom-domain-url-placeholders): Usa `{custom.domain.metadata.KEY`&#125; para completar dinámicamente la URL en función de los metadatos del dominio personalizado utilizado en la solicitud (por ejemplo, `https://{custom_domain.metadata.public_app_url}/callback`).
* **Allowed Origins (CORS)**: Lista de URL autorizadas para realizar solicitudes de Cross-Origin Resource Sharing (CORS) a Auth0.
  * [**Marcadores de posición de dominio personalizado**](/docs/es/get-started/applications/wildcards-for-subdomains#custom-domain-url-placeholders): Usa `{custom.domain.metadata.KEY}` para completar dinámicamente la URL en función de los metadatos del dominio personalizado utilizado en la solicitud (por ejemplo, `https://{custom_domain.metadata.public_app_url}/callback`).

<Warning>
  Si configuras las URL de tu aplicación exclusivamente con marcadores de posición de dominio personalizado, las solicitudes de autenticación realizadas mediante el dominio canónico de tu tenant (por ejemplo, https://your-tenant.us.auth0.com) fallarán.

  Esto ocurre porque el dominio canónico no tiene los metadatos personalizados necesarios para resolver el marcador de posición. Asegúrate de que tu aplicación use el dominio personalizado específico para la autenticación, o proporciona una URL estática de respaldo si se requiere el uso del dominio canónico.
</Warning>

<div id="id-token">
  ### Token de ID
</div>

En la sección **Token de ID**, introduce la **Expiración del token de ID** (en segundos), que es el tiempo que transcurre antes de que caduque el `id_token` de Auth0. El valor predeterminado es de 36000 segundos, es decir, 10 horas.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  **Usa Auth0 en lugar del IdP para el inicio de sesión único**: Si está habilitada, esta configuración evita que Auth0 redirija a los usuarios autenticados con sesiones válidas al proveedor de identidad (como Facebook o ADFS). **Solo para tenants Legacy.**
</Callout>

<div id="refresh-token-rotation">
  ### Rotación del Refresh Token
</div>

En la sección **<Tooltip tip="Refresh Token: token que se usa para obtener un nuevo token de acceso sin obligar a los usuarios a volver a iniciar sesión." cta="Ver glosario" href="/docs/es/glossary?term=Refresh+Token">token de actualización</Tooltip> Rotation**, habilita o deshabilita la rotación. Cuando está habilitada, al intercambiar un token de actualización, se emitirá uno nuevo y el token existente se invalidará. Esto permite detectar automáticamente la reutilización del token si se ve comprometido. Además, introduce el **período de superposición de la rotación** (en segundos). Este intervalo es el margen de tiempo permitido durante el cual el mismo `refresh_token` puede usarse para solicitar un `access_token` sin activar la detección automática de reutilización. Para obtener más información, consulta [rotación del Refresh Token](/docs/es/secure/tokens/refresh-tokens/refresh-token-rotation).

<Frame>![Dashboard Applications Applications pestaña Settings Rotación del Refresh Token](/docs/images/cdy7uua7fh8z/Rk6i8MUVWZG34P7K6aurb/9457a6ea7ca5942522778c7ed7925023/Screenshot_2024-11-05_at_18.14.22.png)</Frame>

<div id="refresh-token-expiration">
  ### Expiración del token de actualización
</div>

En la sección **Expiración del token de actualización**, habilita o deshabilita la expiración absoluta y por inactividad, y establece el período de validez (en segundos) de cada una. Para obtener más información, consulta [Configurar la expiración del token de actualización](/docs/es/secure/tokens/refresh-tokens/configure-refresh-token-expiration).

<Frame>![Dashboard Aplicaciones Aplicaciones pestaña Configuración Expiración del token de actualización](/docs/images/cdy7uua7fh8z/2EnpIEc4kBxjnS3T67xPzC/22fdb95e2a2f31cee6f01e02d0a709c2/Refresh_Token_Expiration_-_English.png)</Frame>

<div id="open-redirect-protection">
  ### Protección contra redirecciones abiertas
</div>

Controla cómo Auth0 gestiona las redirecciones de las aplicaciones de terceros. Esta configuración solo está disponible para aplicaciones de terceros con controles de seguridad reforzados.

<Frame>![Interruptor de Protección contra redirecciones abiertas en el Dashboard](/docs/images/third-party-applications/open-redirect-protection.png)</Frame>

Cuando está habilitada (opción predeterminada para aplicaciones de terceros), Auth0 no redirige a la URL de devolución de llamada de la aplicación cuando se producen errores de autenticación y no expone `application.callback_domain` en las plantillas de correo electrónico. Esto evita ataques de redirección abierta cuando el URI de redirección está controlado por una parte no confiable.

Desactiva la protección contra redirecciones abiertas solo en aplicaciones de terceros cuyos URI de devolución de llamada configurados sean confiables.

Para obtener más información, consulta [Protección de redirección](/docs/es/get-started/applications/third-party-applications/security-controls#redirect-protection).

<div id="advanced-settings">
  ## Configuración avanzada
</div>

La sección **Configuración avanzada** le permite:

* Administrar o agregar metadatos de la aplicación y configurar device, <Tooltip tip="OAuth 2.0: Marco de autorización que define los protocolos y flujos de trabajo de autorización." cta="Ver glosario" href="/docs/es/glossary?term=OAuth">OAuth</Tooltip> y WS-Federation

* Establecer los tipos de concesión para la aplicación

<div id="application-metadata">
  ### Metadatos de la aplicación
</div>

Los metadatos de la aplicación son claves y valores de cadena personalizados (cada uno con un máximo de 255 caracteres) que se configuran para cada aplicación. Los metadatos se exponen en el objeto de la aplicación como `client_metadata` y, en las reglas, como `context.clientMetadata`. Se pueden crear hasta 10 conjuntos de metadatos.

<div id="oauth">
  ### OAuth
</div>

<Frame>![Dashboard Aplicaciones pestaña Configuración de la aplicación Configuración avanzada pestaña OAuth](/docs/images/cdy7uua7fh8z/OAuth-tab.png)</Frame>

* De forma predeterminada, todas las aplicaciones/API pueden hacer una solicitud de delegación, pero si quieres conceder permisos explícitamente a aplicaciones/API seleccionadas, puedes hacerlo en **Allowed Apps/APIs**.
* Para los clientes que usan el add-on Identidad altamente regulada, usa la configuración **Compliance Enforcement Level** para establecer tu nivel de cumplimiento. Para obtener más información, consulta [Configurar el cumplimiento de FAPI](/docs/es/get-started/applications/configure-fapi-compliance).
* **Non-Verifiable Callback URI End-User Confirmation**: Usa esta configuración para controlar si se solicita al usuario que confirme el inicio de sesión cuando se utiliza un URI no verificable como devolución de llamada. Auth0 recomienda no omitir la confirmación del usuario final en estos casos. Esta configuración tiene prioridad sobre la configuración del tenant con el mismo nombre. Para obtener más información, lee [Medidas contra la suplantación de aplicaciones](/docs/es/secure/security-guidance/measures-against-app-impersonation.mdx).
* Establece el algoritmo utilizado ([**HS256**](https://en.wikipedia.org/wiki/Symmetric-key_algorithm) o [**RS256**](https://en.wikipedia.org/wiki/Public-key_cryptography)) para firmar tus <Tooltip tip="JSON Web Token (JWT): formato estándar del token de ID (y, a menudo, también del access token) que se usa para representar claims de forma segura entre dos partes." cta="Ver glosario" href="/docs/es/glossary?term=JSON+web+tokens">JSON web tokens</Tooltip>. Para obtener más información, lee [Algoritmos de firma de JSON Web Token](/docs/es/get-started/applications/signing-algorithms). Al seleccionar `RS256` (recomendado), el token se firmará con la clave privada de tu tenant.
* Activa o desactiva la configuración **Trust Token Endpoint IP Header**; si está habilitada, `auth0-forwarded-for` se considera confiable y se usa como fuente de la IP del usuario final para la protección contra ataques de fuerza bruta en el token endpoint. Esta configuración solo está disponible para Regular Web Apps y aplicaciones máquina a máquina.
* Activa o desactiva el interruptor para indicar si tu aplicación es **OIDC Conformant** o no. Las aplicaciones marcadas como OIDC Conformant seguirán estrictamente la especificación de OIDC.
