Esta sección contiene algunos consejos sobre cómo configurar tu cliente si quieres probar tu solución con las [pruebas de conformidad de FAPI de OpenID](https://openid.net/certification/certification-fapi_op_testing/).

Para pasar las <Tooltip tip="OpenID: Estándar abierto de autenticación que permite que las aplicaciones verifiquen la identidad de los usuarios sin recopilar ni almacenar información de inicio de sesión." cta="Ver glosario" href="/docs/es/glossary?term=OpenID">OpenID</Tooltip> pruebas de conformidad de FAPI, primero configura lo siguiente:

* Establece la propiedad `compliance_level` en el perfil deseado, ya sea `fapi1_adv_pkj_par`, `fapi1_adv_mtls_par`, `fapi2_sp_pkj_mtls` o `fapi2_sp_mtls_mtls`
* [Configura mTLS](/docs/es/get-started/applications/configure-mtls) (incluidos los [alias de mTLS](/docs/es/get-started/applications/configure-mtls/configure-mtls-for-a-tenant#enable-mtls-aliases)) o [Configura clave privada JWT](/docs/es/get-started/applications/configure-private-key-jwt)
* [Configura mTLS Token Binding](/docs/es/get-started/applications/configure-mtls/configure-mtls-for-a-client#enable-token-binding)
* Configura [solicitud de autorización push](/docs/es/get-started/applications/configure-par)
* Asegúrate de que la propiedad `oidc_conformant` esté establecida en `true` para cualquier cliente usado en la prueba. Este es el valor predeterminado para los clientes creados con el Auth0 Dashboard.

Luego, sigue las instrucciones a continuación para completar la configuración de las pruebas de conformidad de OpenID FAPI:

* [Asegúrate de que Auth0 solicite el consentimiento de los usuarios](#ensure-auth0-prompts-users-for-consent)
* [Configura las claims de ACR compatibles para el tenant](#configure-supported-acr-claims-for-the-tenant)
* [Elimina la propiedad `alg` del endpoint JWKS](#remove-the-alg-property-from-jwks-endpoint)
* [Agrega una Action para requerir `scope` y `redirect_uri`](#add-action-to-require-scope-and-redirect-uri)
* [(Solo perfiles FAPI2) Asegúrate de que la claim `iss` se devuelva en las respuestas](#fapi2-profiles-only-ensure-iss-claim-is-returned-in-responses)

<div id="ensure-auth0-prompts-users-for-consent">
  ### Asegúrate de que Auth0 solicite el consentimiento de los usuarios
</div>

Debes asegurarte de que Auth0 solicite el consentimiento de los usuarios. Puedes omitir este paso si el cliente está configurado como una aplicación first-party y el <Tooltip tip="Resource Server: servidor que aloja recursos protegidos. Los Resource Servers aceptan y responden a solicitudes de recursos protegidos." cta="Ver glosario" href="/docs/es/glossary?term=Resource+Server">Resource Server</Tooltip> o la API admite omitir el consentimiento para las aplicaciones first-party. Para asegurarte de que Auth0 solicite el consentimiento de los usuarios, establece la propiedad `is_first_party` del cliente en `false`:

```bash lines
curl --location --request PATCH 'https://{YOUR_DOMAIN}/api/v2/clients/YOUR_CLIENT_ID' \
  --header 'Authorization: Bearer {YOUR_MANAGEMENT_API_ACCESS_TOKEN}' \
  --header 'Content-Type: application/json' \
  --data-raw '{
  "is_first_party": false  
}'
```

Luego, lleva tu conexión al nivel de dominio:

```bash lines
curl --location --request PATCH 'https://{YOUR_DOMAIN}/api/v2/connections/YOUR_CONNECTION_ID' \
  --header 'Authorization: Bearer {YOUR_MANAGEMENT_API_ACCESS_TOKEN}' \
  --header 'Content-Type: application/json' \
  --data-raw '{
  "is_domain_connection": true
}'
```

<div id="configure-supported-acr-claims-for-the-tenant">
  #### Configura los claims de ACR admitidos para el tenant
</div>

Las pruebas de FAPI envían un valor de ACR obligatorio de `urn:mace:incommon:iap:silver`. Para incluir el valor de ACR obligatorio en el <Tooltip tip="token de ID: credencial destinada al propio cliente, en lugar de para acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=ID+token">token de identidad</Tooltip>, agrega `urn:mace:incommon:iap:silver` a la lista de valores de ACR admitidos para el tenant:

```bash lines
curl --location --request PATCH 'https://{YOUR_DOMAIN}/api/v2/tenants/settings' \
  --header 'Authorization: Bearer {YOUR_MANAGEMENT_API_ACCESS_TOKEN}' \
  --header 'Content-Type: application/json' \
  --data-raw '{
    "acr_values_supported": ["urn:mace:incommon:iap:silver"]
}'
```

<div id="remove-the-alg-property-from-jwks-endpoint">
  #### Eliminar la propiedad alg del endpoint JWKS
</div>

Para permitir que las claves se usen con varios algoritmos, no solo con RS256, elimina la propiedad `alg` del tenant en la salida del endpoint `/.well-known/jwks.json`:

```bash lines
curl --location --request PATCH 'https://{YOUR_DOMAIN}/api/v2/tenants/settings' \
  --header 'Authorization: Bearer {YOUR_MANAGEMENT_API_ACCESS_TOKEN}' \
  --header 'Content-Type: application/json' \
  --data-raw '{
    "flags": {
        "remove_alg_from_jwks": true
    }
}'
```

<div id="add-action-to-require-scope-and-redirect_uri">
  #### Agregar una Action para requerir `scope` y `redirect_uri`
</div>

De forma predeterminada, Auth0 permite solicitudes sin `scope` y asume el scope `openid` si no se incluye ninguno. Auth0 también permite solicitudes sin `redirect_uri`, que puedes establecer en [Actions](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger/redirect-with-actions). Sin embargo, la prueba de conformidad de FAPI exige que Auth0 sea más restrictivo.

Agrega la siguiente Action para aplicar las restricciones necesarias a `scope` y `redirect_uri`:

```js lines
exports.onExecutePostLogin = async (event, api) => {
  if (!event.request.body || !event.request.body.refresh_token) {
    // Requerir un scope
    if (!event.request.query.scope) {
      api.access.deny('scope must be provided in the request');
    }
    // Para mejorar el mensaje de error si redirect_uri no está presente
    if (!event.request.query.redirect_uri) {
      api.access.deny('redirect_uri must be provided in the request');
    }
  }
};
```

<div id="fapi2-profiles-only-ensure-iss-claim-is-returned-in-responses">
  #### (Solo perfiles FAPI2) Asegúrate de que se devuelva el claim `iss` en las respuestas
</div>

El perfil de seguridad de FAPI 2.0 exige que el parámetro `iss` se devuelva en las respuestas de autorización de acuerdo con [RFC9207](https://www.rfc-editor.org/info/rfc9207). Por motivos de compatibilidad, Auth0 no lo hace de forma predeterminada. Para habilitar este comportamiento, establece la propiedad `authorization_response_iss_parameter_supported` en la Configuración del tenant como `true`.

```bash lines
curl --location --request PATCH "https://{YOUR_DOMAIN}/api/v2/tenants/settings" \
  --header "Authorization: Bearer {YOUR_MANAGEMENT_API_ACCESS_TOKEN}" \
  --header "Content-Type: application/json" \
  --data-raw '{
    "authorization_response_iss_parameter_supported": true
  }'
```
