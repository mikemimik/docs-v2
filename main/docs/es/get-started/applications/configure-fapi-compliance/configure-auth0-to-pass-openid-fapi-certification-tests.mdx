Esta sección contiene algunos consejos sobre cómo configurar su cliente si desea probar su solución con las [pruebas de conformidad de OpenID FAPI](https://openid.net/certification/certification-fapi_op_testing/).

Para pasar las <Tooltip tip="OpenID: estándar abierto de autenticación que permite a las aplicaciones verificar la identidad de los usuarios sin recopilar ni almacenar información de inicio de sesión." cta="Ver glosario" href="/docs/es/glossary?term=OpenID">OpenID</Tooltip> pruebas de conformidad de FAPI, primero configure lo siguiente:

* Establezca la propiedad `compliance_level` en el perfil deseado: `fapi1_adv_pkj_par`, `fapi1_adv_mtls_par`, `fapi2_sp_pkj_mtls` o `fapi2_sp_mtls_mtls`
* [Configure mTLS](/docs/es/get-started/applications/configure-mtls) (incluidos los [alias de mTLS](/docs/es/get-started/applications/configure-mtls/configure-mtls-for-a-tenant#enable-mtls-aliases)) o [Configure clave privada JWT](/docs/es/get-started/applications/configure-private-key-jwt)
* [Configure mTLS Token Binding](/docs/es/get-started/applications/configure-mtls/configure-mtls-for-a-client#enable-token-binding)
* Configure [solicitudes de autorización push](/docs/es/get-started/applications/configure-par)
* Asegúrese de que la propiedad `oidc_conformant` esté establecida en `true` para cualquier cliente usado en la prueba. Este es el valor predeterminado para los clientes creados en el Auth0 Dashboard.

Luego, siga las instrucciones a continuación para completar la configuración de las pruebas de conformidad de OpenID FAPI:

* [Asegúrese de que Auth0 solicite el consentimiento del usuario](#ensure-auth0-prompts-users-for-consent)
* [Configure las claims de ACR compatibles para el tenant](#configure-supported-acr-claims-for-the-tenant)
* [Quite la propiedad `alg` del endpoint JWKS](#remove-the-alg-property-from-jwks-endpoint)
* [Agregue una Action para requerir scope y redirect&#95;uri](#add-action-to-require-scope-and-redirect-uri)
* [(Solo para perfiles FAPI2) Asegúrese de que la claim `iss` se devuelva en las respuestas](#fapi2-profiles-only-ensure-iss-claim-is-returned-in-responses)

<div id="ensure-auth0-prompts-users-for-consent">
  ### Asegúrese de que Auth0 solicite el consentimiento de los usuarios
</div>

Deberá asegurarse de que Auth0 solicite el consentimiento de los usuarios. Puede omitir este paso si el cliente está configurado como una aplicación propia y el <Tooltip tip="Servidor de recursos: servidor que aloja recursos protegidos. Los servidores de recursos aceptan y responden a solicitudes de recursos protegidos." cta="Ver glosario" href="/docs/es/glossary?term=Resource+Server">Servidor de recursos</Tooltip> o la API permite omitir el consentimiento en aplicaciones propias. Para asegurarse de que Auth0 solicite el consentimiento de los usuarios, establezca la propiedad `is_first_party` del cliente en `false`:

```bash lines
curl --location --request PATCH 'https://{YOUR_DOMAIN}/api/v2/clients/YOUR_CLIENT_ID' \
  --header 'Authorization: Bearer {YOUR_MANAGEMENT_API_ACCESS_TOKEN}' \
  --header 'Content-Type: application/json' \
  --data-raw '{
  "is_first_party": false  
}'
```

Luego, promueve tu conexión a nivel de dominio:

```bash lines
curl --location --request PATCH 'https://{YOUR_DOMAIN}/api/v2/connections/YOUR_CONNECTION_ID' \
  --header 'Authorization: Bearer {YOUR_MANAGEMENT_API_ACCESS_TOKEN}' \
  --header 'Content-Type: application/json' \
  --data-raw '{
  "is_domain_connection": true
}'
```

<div id="configure-supported-acr-claims-for-the-tenant">
  #### Configurar las claims ACR admitidas para el tenant
</div>

Las pruebas de FAPI envían un valor ACR obligatorio de `urn:mace:incommon:iap:silver`. Para incluir el valor ACR obligatorio en el <Tooltip tip="token de ID: credencial destinada al propio cliente, en lugar de acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=ID+token">token de identidad</Tooltip>, agrega `urn:mace:incommon:iap:silver` a la lista de valores ACR admitidos para el tenant:

```bash lines
curl --location --request PATCH 'https://{YOUR_DOMAIN}/api/v2/tenants/settings' \
  --header 'Authorization: Bearer {YOUR_MANAGEMENT_API_ACCESS_TOKEN}' \
  --header 'Content-Type: application/json' \
  --data-raw '{
    "acr_values_supported": ["urn:mace:incommon:iap:silver"]
}'
```

<div id="remove-the-alg-property-from-jwks-endpoint">
  #### Eliminar la propiedad alg del endpoint de JWKS
</div>

Para permitir que las claves se usen con varios algoritmos, y no solo con RS256, elimine la propiedad `alg` del tenant de la respuesta del endpoint `/.well-known/jwks.json`:

```bash lines
curl --location --request PATCH 'https://{YOUR_DOMAIN}/api/v2/tenants/settings' \
  --header 'Authorization: Bearer {YOUR_MANAGEMENT_API_ACCESS_TOKEN}' \
  --header 'Content-Type: application/json' \
  --data-raw '{
    "flags": {
        "remove_alg_from_jwks": true
    }
}'
```

<div id="add-action-to-require-scope-and-redirect_uri">
  #### Agregar una Action para exigir scope y redirect_uri
</div>

De forma predeterminada, Auth0 permite solicitudes sin scope y, si no se especifica ninguno, asume el scope `openid`. Auth0 también permite solicitudes sin `redirect_uri,` que puedes configurar en [Actions](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger/redirect-with-actions). Sin embargo, las pruebas de conformidad de FAPI exigen que Auth0 sea más restrictivo.

Agrega la siguiente Action para aplicar las restricciones necesarias a scope y `redirect_uri`:

```js lines
exports.onExecutePostLogin = async (event, api) => {
  if (!event.request.body || !event.request.body.refresh_token) {
    // Requerir un scope
    if (!event.request.query.scope) {
      api.access.deny('scope must be provided in the request');
    }
    // Para mejorar el mensaje de error si redirect_uri no está presente
    if (!event.request.query.redirect_uri) {
      api.access.deny('redirect_uri must be provided in the request');
    }
  }
};
```

<div id="fapi2-profiles-only-ensure-iss-claim-is-returned-in-responses">
  #### (Solo para perfiles de FAPI2) Asegúrate de que se devuelva el claim `iss` en las respuestas
</div>

El perfil de seguridad de FAPI 2.0 requiere que el parámetro `iss` se devuelva en las respuestas de autorización según [RFC9207](https://www.rfc-editor.org/info/rfc9207). Por motivos de compatibilidad, Auth0 no lo hace de forma predeterminada. Para habilitar este comportamiento, establece la propiedad `authorization_response_iss_parameter_supported` en la configuración del tenant en `true`.

```bash lines
curl --location --request PATCH "https://{YOUR_DOMAIN}/api/v2/tenants/settings" \
  --header "Authorization: Bearer {YOUR_MANAGEMENT_API_ACCESS_TOKEN}" \
  --header "Content-Type: application/json" \
  --data-raw '{
    "authorization_response_iss_parameter_supported": true
  }'
```
