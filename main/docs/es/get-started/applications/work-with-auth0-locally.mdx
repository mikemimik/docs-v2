title: Probar aplicaciones localmente
---

En la mayoría de los casos, autenticar usuarios a través de Auth0 requiere una conexión a internet. Sin embargo, aun así puedes desarrollar y probar aplicaciones que usan Auth0 de forma local. En algunos casos, puede que ni siquiera necesites acceso a internet.

Para obtener más información sobre cómo estructurar tus entornos de desarrollo, pruebas y producción al usar Auth0, consulta [Set Up Multiple Environments](/docs/es/get-started/auth0-overview/create-tenants/set-up-multiple-environments).

<div id="use-json-web-tokens-with-client-side-applications">
  ## Usa JSON Web Tokens con aplicaciones del lado del cliente
</div>

Como los [JSON Web Tokens (JWT)](/docs/es/secure/tokens/json-web-tokens) no tienen estado (es decir, la aplicación que los consume solo se fija en su contenido, no en ninguno de sus estados anteriores), este es uno de los escenarios más fáciles de probar localmente.

Puedes obtener <Tooltip tip="JSON Web Token (JWT): Formato estándar de token de ID (y, a menudo, de token de acceso) utilizado para representar claims de forma segura entre dos partes." cta="Ver glosario" href="/docs/es/glossary?term=JWTs">JWT</Tooltip> para hacer pruebas con cualquiera de los siguientes métodos:

1. Crea un usuario de prueba para una [conexión de base de datos](/docs/es/get-started/applications/set-up-database-connections) e inicia sesión con ese usuario mediante programación. Básicamente, estás usando el [flujo de contraseña del propietario del recurso](/docs/es/get-started/authentication-and-authorization-flow/resource-owner-password-flow). Para obtener instrucciones detalladas de implementación, consulta [Llama a tu API usando el flujo de contraseña del propietario del recurso](/docs/es/get-started/authentication-and-authorization-flow/resource-owner-password-flow/call-your-api-using-resource-owner-password-flow).
2. Usa un bot de navegador para simular el papel de un usuario, iniciar sesión y recuperar un JWT.

<div id="use-sessions-with-server-side-applications">
  ## Usa sesiones con aplicaciones del lado del servidor
</div>

A menos que tu aplicación del lado del servidor permita generar sesiones artificiales para hacer pruebas, necesitarás alguna forma de iniciar sesión manualmente con Auth0.

Para ver una descripción general de cómo hacerlo, consulta [Flujo de Código de Autorización](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow). Para obtener instrucciones detalladas de implementación, consulta nuestro tutorial [Agregar inicio de sesión con el Flujo de Código de Autorización](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow/add-login-auth-code-flow).

<div id="use-local-domains-with-auth0">
  ## Usa dominios locales con Auth0
</div>

Si estás desarrollando tu aplicación localmente, puedes usar `localhost` y otros dominios a los que Auth0 no puede acceder (como los de una intranet) como [URLs de devolución de llamada](/docs/es/authenticate/login/redirect-users-after-login). Por ejemplo, durante el desarrollo, podrías usar `http://localhost:3000/callback` como URL de devolución de llamada.

1. Ve a [Auth0 Dashboard &gt; Applications &gt; Applications](https://manage.auth0.com/#/applications/\{yourClientId}/settings) y haz clic en la aplicación.
2. Agrega la URL a la lista de **Allowed Callback URLs**.

Como el protocolo de identidad principal de Auth0 es [OpenID Connect (OIDC)](/docs/es/authenticate/protocols/openid-connect-protocol), Auth0 nunca necesita comunicarse directamente con el servidor de tu aplicación. En su lugar, redirige a los usuarios al endpoint o los endpoints de tu aplicación con la información requerida en un query string o en un fragmento hash.

<div id="divert-emails-for-testing">
  ## Desviar correos electrónicos para pruebas
</div>

Si desea probar su aplicación local y no quiere que los correos electrónicos (de creación, validación, etc.) se envíen a la dirección de correo electrónico real de los usuarios que su aplicación crea o valida, Auth0 recomienda usar un proveedor de correo electrónico personalizado. Por ejemplo, un servicio como [Mailtrap](https://mailtrap.io/signin) o la implementación de su propio servidor SMTP personalizado puede aplicar la lógica que necesite para interceptar los correos electrónicos. Esto garantiza que los usuarios no reciban correos electrónicos, pero que usted pueda acceder a ellos para la validación y la resolución de problemas. Para obtener más información, lea [Configurar servidor de correo electrónico SMTP de prueba](/docs/es/customize/email/configure-test-smtp-email-servers).

Auth0 no permite ciertos dominios &quot;falsos&quot; que se usan habitualmente durante las pruebas. Use direcciones de correo electrónico reales para evitar interrupciones o errores.
