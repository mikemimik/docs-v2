La multitenencia es un enfoque arquitectónico en el que una única instancia de software se ejecuta en un servidor y puede ser utilizada por múltiples grupos de usuarios. Con la multitenencia, puede segmentar en grupos a los usuarios con características compartidas y otorgarles permisos y niveles de acceso independientes a su aplicación. Esto le permite crear y mantener experiencias adaptadas para distintos clientes, unidades de negocio u otros grupos de usuarios definidos.

En Auth0, el mejor método para implementar la multitenencia es [Auth0 Organizations](/docs/es/manage-users/organizations/organizations-overview). Si es necesario, se pueden usar otras soluciones heredadas para dar respuesta a distintos casos de uso empresariales. Si ofrece un producto o servicio [negocio a negocio (B2B)](/docs/es/get-started/architecture-scenarios/business-to-business), configurar la multitenencia para sus usuarios empresariales puede ser beneficioso para su caso de uso.

Las secciones siguientes describen las opciones disponibles para implementar la multitenencia en Auth0.

<Warning>
  En este artículo, se usa el término de arquitectura de software &quot;tenant&quot; para referirse a un grupo de usuarios que pueden acceder a su aplicación. Para referirse a su instancia de Auth0, se usa el término &quot;tenant de Auth0&quot;.
</Warning>

<div id="auth0-organizations">
  ## Auth0 Organizations
</div>

Para la mayoría de los casos de uso de multitenencia, Auth0 Organizations es la solución ideal para usted y sus usuarios. Auth0 Organizations admite implementaciones de negocio a negocio (B2B) con una o más aplicaciones a las que los usuarios finales pueden acceder.

Las funcionalidades comunes de las implementaciones B2B incluyen:

* Un producto que se concede bajo licencia a otra empresa para que lo utilicen sus empleados.
* Múltiples organizaciones que requieren su propia federación y una personalización ligera de marca de la experiencia de autenticación.
* Niveles independientes de acceso a la aplicación para distintos grupos de usuarios.

Con Auth0 Organizations, puede crear grupos únicos de usuarios y adaptar sus experiencias con [control de acceso basado en roles](/docs/es/manage-users/access-control/rbac), [páginas de inicio de sesión y plantillas de correo electrónico personalizadas](/docs/es/manage-users/organizations/create-first-organization#customize-prompts-and-email-templates), entre otras opciones.

Para obtener más información sobre cómo usar Auth0 Organizations para implementar la multitenencia, consulte [Arquitectura de múltiples organizaciones](/docs/es/get-started/architecture-scenarios/multiple-organization-architecture).

<div id="legacy-solutions">
  ## Soluciones heredadas
</div>

Si Auth0 Organizations no cubre las necesidades de tu caso de uso, puedes considerar las soluciones heredadas que se describen a continuación. Si necesitas orientación para elegir el enfoque más adecuado para tus requisitos específicos, ponte en contacto con nuestro equipo de [Professional Services](/docs/es/get-started/professional-services).

Las soluciones heredadas incluyen:

* Usar una Conexión de Auth0 para representar cada tenant.
* Usar una aplicación de Auth0 para representar cada tenant.
* Usar un tenant de Auth0 para representar cada tenant.
* Almacenar los detalles del tenant en el perfil del usuario.

<div id="use-auth0-connections">
  ### Usa conexiones de Auth0
</div>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Es posible que se apliquen límites de entidades. Para obtener más información, consulta la [Política de límites de entidades](/docs/es/troubleshoot/customer-support/operational-policies/entity-limit-policy). Si tienes una suscripción Enterprise, no te verás limitado por los límites de entidades, pero sí podrías verte limitado por una conexión que ya tenga miles de clientes habilitados.
</Callout>

Puedes representar cada uno de tus tenants con una Conexión de Auth0 independiente.

Este enfoque te permite admitir escenarios en los que:

* Tienes distintos requisitos a nivel de conexión, como diferentes políticas de contraseñas, para cada uno de tus tenants.
* Tienes grupos de usuarios de distintas conexiones. Por ejemplo, un tenant podría exigir que los usuarios proporcionen credenciales de username/password, mientras que otro tenant podría exigir que los usuarios inicien sesión a través de un proveedor de identidad empresarial <Tooltip tip="Identity Provider (IdP): Servicio que almacena y administra identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=IdP">IdP</Tooltip>.

Para pedirle a un usuario que inicie sesión a través de una conexión específica, llama al [endpoint de inicio de sesión de Auth0 Authentication API](https://auth0.com/docs/api/authentication#login) e incluye el parámetro `connection`.

<Warning>
  Si usas [Lock](/docs/es/libraries#lock) en tu aplicación, ten en cuenta que admite un máximo de 50 conexiones de base de datos por aplicación. Las conexiones sociales y empresariales no se ven afectadas por este límite, pero siguen estando sujetas a la [Política de límites de entidades](/docs/es/troubleshoot/customer-support/operational-policies/entity-limit-policy).
</Warning>

<div id="use-auth0-applications">
  ### Usa aplicaciones de Auth0
</div>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Pueden aplicarse límites de entidades. Para obtener más información, lee la [Política de límites de entidades](/docs/es/troubleshoot/customer-support/operational-policies/entity-limit-policy). Si tienes una suscripción Enterprise, no estarás limitado por los límites de entidades, pero sí podrías estarlo por una conexión que ya tenga miles de clientes habilitados.
</Callout>

Puedes representar cada uno de tus tenants con una aplicación de Auth0 distinta.

Este enfoque te permite configurar cada aplicación de Auth0 de forma específica según los distintos requisitos de cada tenant, como las conexiones disponibles.

Tendrás que hacer un seguimiento de los tenants a los que pertenecen tus usuarios dentro de tu aplicación. Cuando un usuario inicie sesión en tu aplicación, tendrás que consultar esa información y dirigirlo a la aplicación de Auth0 adecuada para completar la autenticación.

Para habilitar una conexión para varias aplicaciones con la <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip> de Auth0, llama al [endpoint para actualizar clientes habilitados para una conexión](https://auth0.com/docs/api/management/v2/connections/patch-clients) y pasa el ID de conexión correspondiente.

<div id="use-auth0-tenants">
  ### Usa tenants de Auth0
</div>

Puedes representar cada uno de tus tenants mediante un tenant de Auth0 independiente.

Este enfoque te permite compartir con los usuarios el acceso al <Tooltip tip="Auth0 Dashboard: el producto principal de Auth0 para configurar tus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip>, con acceso restringido por tenant, pero requiere que configures Auth0 de forma individual para cada tenant.

Esto significa que, además de administrar por separado las funcionalidades de cada tenant de Auth0 (como Branding, Actions y <Tooltip tip="Protección contra ataques: funcionalidades que Auth0 proporciona para detectar y mitigar ataques, incluidas la protección contra ataques de fuerza bruta, la limitación de IP sospechosas, la detección de contraseñas vulneradas, la detección de bots y la autenticación multifactor adaptativa." cta="Ver glosario" href="/docs/es/glossary?term=Attack+Protection">Protección contra ataques</Tooltip>), tu aplicación tendrá que admitir varias configuraciones de Auth0.

<div id="store-tenant-details-in-the-users-profile">
  ### Almacenar detalles del tenant en el perfil del usuario
</div>

Puede almacenar detalles del tenant en el perfil del usuario y hacer que su aplicación lea esa información después de que el usuario inicie sesión.

Este enfoque permite que todos sus usuarios, independientemente del tenant al que pertenezcan, inicien sesión con una configuración uniforme (como las conexiones disponibles).

Para implementar esto, puede almacenar los detalles del tenant en el perfil de Auth0 del usuario, en el [`objeto app_metadata`](/docs/es/manage-users/user-accounts/metadata), con un identificador de su elección (por ejemplo, `"tenant": "customer-group-12345"`). Después de que el usuario inicie sesión, su aplicación recupera la variable `tenant` y muestra una versión adecuada según el valor devuelto.
