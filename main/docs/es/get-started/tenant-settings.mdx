Usa la página **Configuración del tenant** del <Tooltip tip="Auth0 Dashboard: el producto principal de Auth0 para configurar tus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip>, en [Dashboard &gt; Settings](https://manage.auth0.com/#/tenant), para configurar varias opciones relacionadas con tu tenant de Auth0.

<div id="recommended-settings">
  ## Configuración recomendada
</div>

Cuando configures tu tenant, establece los siguientes elementos:

* **Especifica la etiqueta de entorno.** Los tenants etiquetados como **Production** tienen límites de frecuencia más altos que los etiquetados como **Development** o **Staging**. En los planes que no son Enterprise, solo un tenant por suscripción puede etiquetarse como **Production**. Para obtener más información, consulta [Set Up Multiple Environments](/docs/es/get-started/auth0-overview/create-tenants/set-up-multiple-environments).
* **Establece el correo electrónico de soporte y el URL de soporte.** Si un usuario encuentra un problema al iniciar sesión, querrá pedir ayuda. Configura estos valores para dirigirlo a una dirección de correo electrónico o a una página de destino donde pueda recibir asistencia.
* **Configura una página de error personalizada.** Si es posible, deberías alojar tu propia página de error personalizada y configurar Auth0 para que la use en lugar de la página predeterminada. Esto te permite ofrecer a los usuarios explicaciones más completas y personalizadas sobre qué hacer en caso de error.
* **Configura un <Tooltip tip="Custom Domain: Dominio de terceros con un nombre especializado o personalizado." cta="Ver glosario" href="/docs/es/glossary?term=custom+domain">dominio personalizado</Tooltip>.** Si tienes un plan de pago, puedes configurar un dominio personalizado para tu tenant de Auth0. Un dominio personalizado unifica la experiencia de inicio de sesión con tu marca y ofrece beneficios adicionales. Para obtener más información, consulta [Custom Domains](/docs/es/customize/custom-domains).
* **Establece el tiempo de espera de la sesión de <Tooltip tip="Single Sign-On (SSO): Servicio que, después de que un usuario inicia sesión en una aplicación, inicia automáticamente la sesión de ese usuario en otras aplicaciones." cta="Ver glosario" href="/docs/es/glossary?term=Single+Sign-On">inicio de sesión único</Tooltip> (SSO).** El valor del tiempo de espera de la sesión de SSO especifica cuánto tiempo pasa hasta que expira la sesión de un usuario. De forma predeterminada, el valor es de 7 días, que es el tiempo durante el cual los usuarios pueden acceder a tus aplicaciones integradas con Auth0 sin volver a introducir sus credenciales. Para obtener más información, consulta [Sessions](/docs/es/manage-users/sessions).
* **Configura miembros del tenant.** Configura usuarios adicionales de Auth0 Dashboard y habilita la <Tooltip tip="Autenticación multifactor (MFA): Proceso de autenticación de usuario que utiliza un factor además del nombre de usuario y la contraseña, como un código enviado por SMS." cta="Ver glosario" href="/docs/es/glossary?term=multi-factor+authentication">autenticación multifactor</Tooltip> (MFA). Para obtener más información, consulta [Manage Dashboard Access](/docs/es/get-started/manage-dashboard-access) y [Manage Dashboard Access with Multi-Factor Authentication](/docs/es/get-started/manage-dashboard-access/add-change-remove-mfa).
* **Desactiva la configuración Enable Application Connections.** Si esta configuración está habilitada, todas las conexiones configuradas se habilitarán automáticamente para cualquier aplicación nueva que crees. Como resultado, es posible que los usuarios puedan iniciar sesión en la aplicación mediante conexiones que no pretendías que estuvieran disponibles. Desactiva esta configuración para poder habilitar explícitamente las conexiones adecuadas para cada aplicación.
* **Habilita <Tooltip tip="Attack Protection: Funcionalidades que Auth0 proporciona para detectar y mitigar ataques, incluida la protección contra ataques de fuerza bruta, la limitación de IP sospechosas, la detección de contraseñas vulnerada, la detección de bots y la autenticación multifactor adaptativa." cta="Ver glosario" href="/docs/es/glossary?term=Attack+Protection">Attack Protection</Tooltip>.** Protege a tus usuarios contra ataques de fuerza bruta y contraseñas vulneradas. Para obtener más información, consulta [Attack Protection](/docs/es/secure/attack-protection).

<Frame>![Pestaña General Settings de Configuración del tenant en el Dashboard](/docs/images/cdy7uua7fh8z/4okToiwlkNQBxwr8QGE3Rs/7e96db0efab4082ebb9f78cb43ee302c/Basic_Settings_-_EN.png)</Frame>

* **Nombre descriptivo**: Nombre que quieres mostrar a tus usuarios en la página de <Tooltip tip="Universal Login: Tu aplicación redirige a Universal Login, alojado en el servidor de autorización de Auth0, para verificar la identidad de un usuario." cta="Ver glosario" href="/docs/es/glossary?term=Universal+Login">Universal Login</Tooltip>. * **URL de soporte**: Enlace a la página de soporte de tu empresa u organización.

  ### Configuración de autorización de API
</div>

<Frame>![Configuración de autorización de API en la pestaña General de Configuración del tenant del Dashboard](/docs/images/cdy7uua7fh8z/17yNCRrB6dWcHp6zLFb74j/d1f7c516bce3bf66e9f4bfbd201411a3/Auth_Settings_-_EN.png)</Frame>

* **Audiencia predeterminada <Tooltip tip="Audiencia: Identificador único de la audiencia de un token emitido. Se denomina aud en un token y su valor contiene el ID de una aplicación (ID de cliente) para un token de ID o de una API (identificador de API) para un token de acceso." cta="Ver glosario" href="/docs/es/glossary?term=Audience">Audience</Tooltip>**: Identificador de API que se usará en los [flujos de autorización](/docs/es/get-started/authentication-and-authorization-flow). Si introduce un valor, todos los <Tooltip tip="Token de acceso: Credencial de autorización, en forma de una cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=access+tokens">tokens de acceso</Tooltip> emitidos por Auth0 especificarán este identificador de API como audiencia. Configurar la **Audiencia predeterminada** equivale a agregar esta audiencia a cada solicitud de autorización realizada a su tenant para cada aplicación. Esto generará un nuevo comportamiento que podría provocar cambios incompatibles en algunas de sus aplicaciones. [Contacte con soporte](/docs/es/troubleshoot/customer-support/open-and-manage-support-tickets) si necesita ayuda.

Su valor debe ser el nombre exacto de una conexión existente para una de las siguientes estrategias: `auth0-adldap`, `ad`, `auth0`, `email`, `sms`, `waad` o `adfs`.

<div id="signing-keys">
  ## Claves de firma
</div>

En la pestaña **Claves de firma**, puedes administrar de forma segura la clave de firma y el certificado que se usan para firmar <Tooltip tip="token de ID: Credencial destinada al propio cliente, en lugar de para acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=ID+tokens">tokens de ID</Tooltip>, tokens de acceso, <Tooltip tip="Security Assertion Markup Language (SAML): Protocolo estandarizado que permite a dos partes intercambiar información de autenticación sin una contraseña." cta="Ver glosario" href="/docs/es/glossary?term=SAML">aserciones SAML</Tooltip> y <Tooltip tip="Security Assertion Markup Language (SAML): Protocolo estandarizado que permite a dos partes intercambiar información de autenticación sin una contraseña." cta="Ver glosario" href="/docs/es/glossary?term=WS-Fed">aserciones WS-Fed</Tooltip> que se envían a tus aplicaciones.

<Frame>![Pestaña Claves de firma de Configuración del tenant en Dashboard](/docs/images/cdy7uua7fh8z/7r8t3EGctFmvkCgPrU0i2R/f79ac74dac5cded37b39bd8a8b80951d/dashboard-tenant-settings-signing-keys.png)</Frame>

* **Configuración de rotación**: Opciones de configuración que te permiten rotar la clave de firma y el certificado de la aplicación. Puedes elegir si quieres revocar o no la clave de firma al rotarla. Para obtener más información, consulta [Claves de firma](/docs/es/get-started/tenant-settings/signing-keys).

  * **Rotar clave de firma**: Rota la clave de firma sin revocarla; en la práctica, mueve la clave actual a la posición de clave anterior. Todos los tokens firmados con la clave anterior seguirán siendo válidos hasta que se revoque.
  * **Rotar y revocar clave de firma**: Rota la clave de firma y luego la revoca; en la práctica, mueve la clave actual a la posición de clave anterior y después invalida la clave anterior. Asegúrate de haber actualizado tu aplicación con la siguiente clave de la cola antes de rotar y revocar la clave actual.
* **Lista de claves válidas**: Lista de claves de firma de aplicación válidas para tu tenant, que también están disponibles en el endpoint de metadatos de tu aplicación. Las claves válidas incluyen:

  * **Siguiente en la cola**: Clave que se usará la próxima vez que se rote la clave de firma.
  * **En uso actualmente**: Clave que está en uso actualmente.
  * **Usada anteriormente**: Clave que se utilizó anteriormente. Su presencia indica que la clave de firma se ha rotado, pero que la clave usada anteriormente aún no se ha revocado.
* **Lista de claves revocadas**: Lista de las tres últimas claves revocadas de tu tenant. Hay más información sobre las claves revocadas disponible en los registros del tenant.

<div id="login-and-logout">
  ### Inicio de sesión y cierre de sesión
</div>

<Frame>![Dashboard Configuración del tenant pestaña avanzado Inicio de sesión y cierre de sesión](/docs/images/cdy7uua7fh8z/Login_and_Logout_-_EN.png)</Frame>

* **Tenant Login URI**: URI que apunta a una ruta de su aplicación que inicia el flujo de inicio de sesión de OIDC redirigiendo al endpoint `/authorize`; debe tener el formato `https://mytenant.org/login`. Solo se utilizará en los casos en que Auth0 necesite que su tenant inicie el flujo de inicio de sesión de OIDC. Para obtener más información, consulte [Configurar rutas de inicio de sesión predeterminadas](/docs/es/authenticate/login/auth0-universal-login/configure-default-login-routes).
* **Allowed Logout URLs**: URL a las que Auth0 puede redirigir después del cierre de sesión cuando no se especifica ningún client&#95;id en la invocación del endpoint de cierre de sesión. Resulta útil como lista global cuando el inicio de sesión único (SSO) está habilitado. Para obtener más información, lea [cierre de sesión](docs/authenticate/login/logout).
* **Allowed ACR Values**: Lista de valores de Authentication Context Class Reference (ACR) permitidos. Estos valores se incluyen en el documento OpenID Configuration. Cuando esta opción está configurada, los valores que no figuren aquí se rechazarán si se usan en flujos de autenticación.
* **RP-Initiated Logout End Session Endpoint Discovery**: Controla si el endpoint de cierre de sesión se anuncia en las respuestas de OpenID Configuration como `end_session_endpoint`.
* **RP-Initiated Logout End-User Confirmation**: Controla si se solicita al usuario que confirme el inicio de sesión cuando la solicitud de cierre de sesión iniciada por RP no incluye las pistas correctas.
* **Non-Verifiable Callback URI End-User Confirmation**: Controla si se solicita al usuario que confirme el inicio de sesión cuando se usa un esquema URI personalizado como devolución de llamada. Auth0 recomienda que **no** omita la confirmación del usuario final en estos casos. Para obtener más información, lea [Measures Against Application Impersonation](docs/secure/security-guidance/measures-against-app-impersonation.mdx).

<div id="login-session-management">
  ### Gestión de sesiones de inicio de sesión
</div>

La configuración de **Gestión de sesiones de inicio de sesión** define la duración de la sesión de inicio de sesión, que corresponde a la capa de sesión del <Tooltip tip="Servidor de autorización: Servidor centralizado que contribuye a definir los límites del acceso de un usuario. Por ejemplo, su servidor de autorización puede controlar los datos, las tareas y las funcionalidades disponibles para un usuario." cta="Ver glosario" href="/docs/es/glossary?term=Authorization+Server">servidor de autorización</Tooltip> de Auth0. La capa de sesión del servidor de autorización es la que habilita el inicio de sesión único (SSO). Para obtener más información, consulta [Inicio de sesión único](/docs/es/authenticate/single-sign-on).

Los tiempos de espera de los tokens emitidos por Auth0 pueden configurarse en otro lugar. Los tiempos de espera de los tokens suelen usarse para controlar la capa de sesión de la aplicación y aparecen en los claims del token, como el claim de expiración de los token de ID de <Tooltip tip="OpenID: Estándar abierto de autenticación que permite a las aplicaciones verificar las identidades de los usuarios sin recopilar ni almacenar información de inicio de sesión." cta="Ver glosario" href="/docs/es/glossary?term=OpenID">OpenID</Tooltip> Connect (OIDC) o la aserción de vigencia para SAML.

<Frame>![Dashboard Configuración del tenant avanzado Gestión de sesiones de inicio de sesión](/docs/images/cdy7uua7fh8z/q2fGPzoUqCcXj7OxcHhjy/de15afcd26e8ea80937733034ffbd387/Screenshot_2024-11-06_at_11.02.16.png)</Frame>

* **Tiempo de inactividad**: Período (en minutos) tras el cual la sesión de un usuario expirará si no ha interactuado con el servidor de autorización. Será sustituido por los límites del sistema si supera los 4.320 minutos (3 días) para los planes no empresariales o los 144.000 minutos (100 días) para los planes Enterprise.
* **Tiempo tras el cual es necesario iniciar sesión de nuevo**: Período (en minutos) tras el cual se le pedirá al usuario que vuelva a iniciar sesión, independientemente de su actividad. Será sustituido por los límites del sistema si supera los 43.200 minutos (30 días) para los planes no empresariales o los 525.600 minutos (365 días) para los planes Enterprise.

<div id="settings-advanced">
  ### Configuración (Avanzado)
</div>

* **Flujo de cambio de contraseña v2**: Al habilitarlo, se utilizará la versión más reciente del flujo de cambio de contraseña. La versión anterior ha quedado obsoleta y recomendamos encarecidamente habilitar la v2. Este indicador se incluye solo por compatibilidad con versiones anteriores y, una vez habilitado, ya no podrá deshabilitarse. Puede personalizar la interfaz de usuario del widget de cambio de contraseña en la pestaña [Universal Login &gt; Restablecimiento de contraseña](https://manage.auth0.com/#/password_reset) del Auth0 Dashboard.
* **Registro dinámico de clientes (DCR)**: Cuando está habilitado, los desarrolladores externos podrán registrar dinámicamente aplicaciones para tus API. También puedes actualizar este indicador mediante el endpoint [`/tenant/patch_settings`](https://auth0.com/docs/api/management/v2#!/Tenants/patch_settings) de la Auth0 <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip>. De forma predeterminada, esta funcionalidad está deshabilitada. Para obtener más información, consulta [Registro dinámico de clientes](/docs/es/get-started/applications/dynamic-client-registration).
* **Perfil de compatibilidad del parámetro de recurso**: el perfil de compatibilidad del parámetro de recurso determina cómo Auth0 maneja el parámetro `resource` en las solicitudes de autorización.
  * Cuando está habilitada:
    * Auth0 usa el parámetro `resource` para especificar a qué servidor de recursos (API) quiere acceder la aplicación cliente
    * Auth0 comprueba primero el parámetro `audience`; si no se proporciona, usa el parámetro `resource`
    * Auth0 consume el parámetro `resource` y no lo reenvía al IdP de origen
  * Cuando está deshabilitada (por defecto):
    * Auth0 usa solo el parámetro `audience` para especificar el servidor de recursos
    * El parámetro `resource` se trata como un parámetro del IdP de origen y se reenviará al IdP
* **Registro mediante Documento de metadatos del ID de cliente (CIMD)**: Habilita el registro con CIMD manual importando desde una URL un Documento de metadatos del ID de cliente (CIMD) alojado externamente. CIMD es un archivo JSON que contiene metadatos del cliente y está alojado en un dominio HTTPS seguro controlado por la aplicación. Para obtener más información, consulta [Registrar aplicaciones con CIMD](/docs/es/get-started/auth0-overview/create-applications/register-applications-with-cimd).
* **Habilitar conexiones de la aplicación**: Cuando esta opción está habilitada, todas las conexiones actuales se habilitarán para cualquier aplicación nueva que se cree.
* **Usar una respuesta genérica en el mensaje de error de la API pública de registro**: Cuando esta opción está habilitada, los errores generados al usar la API pública de registro devolverán una respuesta genérica. Esto ayuda a proteger contra la enumeración de usuarios durante el registro, ya que impide que los <Tooltip tip="Actores malintencionados: Entidad (una persona o grupo) que representa una amenaza para la empresa o el entorno con la intención de causar daño." cta="Ver glosario" href="/docs/es/glossary?term=bad+actors">actores malintencionados</Tooltip> puedan deducir identificadores ya registrados (username, correo electrónico o phone) al leer códigos de respuesta de error, como `user_exists`.
* **Habilitar la publicación de información de conexiones empresariales con dominios de <Tooltip tip="Proveedor de identidad (IdP): Servicio que almacena y gestiona identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=IdP">IdP</Tooltip>**: Cuando está habilitado, admite [Home Realm Discovery](/docs/es/authenticate/login/auth0-universal-login/identifier-first) y [Auth0 Lock](/docs/es/libraries/lock) utiliza un archivo público verificado que incluye información sobre conexiones empresariales. Si no necesita esa funcionalidad, puede deshabilitarla.
* **Habilitar el flujo de verificación por correo electrónico durante el inicio de sesión para las conexiones de Azure AD y ADFS**: Al habilitarlo, se mostrará a los usuarios un prompt de verificación por correo electrónico durante su primer inicio de sesión al usar conexiones de Azure AD o ADFS.
* **<Tooltip tip="Token de actualización: token utilizado para obtener un nuevo token de acceso sin obligar a los usuarios a volver a iniciar sesión." cta="Ver glosario" href="/docs/es/glossary?term=Refresh+Token">token de actualización</Tooltip> Su revocación elimina la concesión**: Cuando está habilitado, elimina la concesión asociada al revocar un token de actualización mediante el endpoint `/oauth/revoke` de la Authentication API.

  <Callout icon="file-lines" color="#0EA5E9" iconType="regular">
    En los tenants existentes, esta funcionalidad viene habilitada de forma predeterminada para conservar el comportamiento existente. En los tenants nuevos (a partir del 13 de enero de 2021), esta funcionalidad viene deshabilitada de forma predeterminada para garantizar que la revocación de un token de actualización no revoque la concesión. Si es necesario revocar una concesión, se debe enviar una solicitud independiente mediante un endpoint de revocación de concesiones.
  </Callout>
* **Permitir nombres de organización en Authentication API**: Cuando está habilitado, los endpoints [/authorize](https://auth0.com/docs/api/authentication#authorize-application) y [SAML](https://auth0.com/docs/api/authentication#saml) pueden aceptar tanto IDs como nombres de organización. Además, los tokens de ID y de acceso incluirán los claims `org_id` y `org_name`. Antes de habilitar esta configuración, revisa [Usar nombres de organización en Authentication API](/docs/es/manage-users/organizations/configure-organizations/use-org-name-authentication-api) para conocer consideraciones importantes y posibles impactos.
* **Permitir solicitudes de autorización push (PAR):** Cuando está habilitado, el endpoint `/par` puede aceptar solicitudes de autorización enviadas mediante push desde una aplicación cliente. Esto evita que la aplicación cliente envíe solicitudes a través del front channel no seguro (es decir, el navegador).
