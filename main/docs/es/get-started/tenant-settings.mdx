Usa la página **Configuración del tenant** del <Tooltip tip="Auth0 Dashboard: el producto principal de Auth0 para configurar tus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip>, en [Dashboard &gt; Settings](https://manage.auth0.com/#/tenant), para configurar diversos ajustes relacionados con tu tenant de Auth0.

<div id="recommended-settings">
  ## Configuración recomendada
</div>

Cuando configures tu tenant, establece los siguientes elementos:

* **Especifica la etiqueta de entorno.** Los tenants etiquetados como **Production** reciben límites de frecuencia más altos que los tenants etiquetados como **Development** o **Staging**. En los planes no Enterprise, solo un tenant por suscripción puede etiquetarse como **Production**. Para obtener más información, lee [configurar varios entornos](/docs/es/get-started/auth0-overview/create-tenants/set-up-multiple-environments).
* **Establece el correo electrónico de soporte y la URL de soporte.** Si un usuario tiene un problema al iniciar sesión, querrá pedir ayuda. Establece estos valores para dirigirlo a una dirección de correo electrónico o a una página de destino donde pueda obtener asistencia.
* **Configura una página de error personalizada.** Si es posible, deberías alojar tu propia página de error personalizada y configurar Auth0 para que la use en lugar de la página predeterminada. Esto te permite ofrecer a los usuarios explicaciones más completas y personalizadas sobre qué hacer en caso de error.
* **Configura un <Tooltip tip="Custom Domain: Dominio de terceros con un nombre especializado o personalizado." cta="Ver glosario" href="/docs/es/glossary?term=custom+domain">dominio personalizado</Tooltip>.** Si tienes un plan de pago, puedes configurar un dominio personalizado para tu tenant de Auth0. Un dominio personalizado unifica la experiencia de inicio de sesión con tu marca y ofrece ventajas adicionales. Para obtener más información, lee [dominio personalizado](/docs/es/customize/custom-domains).
* **Establece el tiempo de expiración de la sesión de <Tooltip tip="Inicio de sesión único (SSO): Servicio que, después de que un usuario inicia sesión en una aplicación, inicia automáticamente la sesión de ese usuario en otras aplicaciones." cta="Ver glosario" href="/docs/es/glossary?term=Single+Sign-On">inicio de sesión único</Tooltip> (SSO).** El valor del tiempo de expiración de la sesión de SSO especifica cuánto tarda en expirar la sesión de un usuario. De forma predeterminada, el valor es de 7 días, que es el período durante el cual los usuarios pueden acceder a tus aplicaciones integradas con Auth0 sin volver a introducir sus credenciales. Para obtener más información, lee [Sessions](/docs/es/manage-users/sessions).
* **Configura miembros del tenant.** Configura usuarios adicionales de Auth0 Dashboard y habilita la <Tooltip tip="Autenticación multifactor (MFA): Proceso de autenticación de usuario que utiliza un factor además del nombre de usuario y la contraseña, como un código por SMS." cta="Ver glosario" href="/docs/es/glossary?term=multi-factor+authentication">autenticación multifactor</Tooltip> (MFA). Para obtener más información, lee [Administrar el acceso al Dashboard](/docs/es/get-started/manage-dashboard-access) y [Manage Dashboard Access with Multi-Factor Authentication](/docs/es/get-started/manage-dashboard-access/add-change-remove-mfa).
* **Desactiva la configuración Enable Application Connections.** Si esta configuración está habilitada, todas las conexiones configuradas se habilitarán automáticamente para cualquier aplicación nueva que crees. Como resultado, los usuarios podrían iniciar sesión en la aplicación mediante conexiones que no querías que estuvieran disponibles. Desactiva esta configuración para poder habilitar explícitamente las conexiones adecuadas para cada aplicación.
* **Habilita <Tooltip tip="Attack Protection: Funciones que Auth0 proporciona para detectar y mitigar ataques, incluida la protección contra ataques de fuerza bruta, la limitación de IP sospechosas, la detección de contraseñas vulnerada, Bot Detection y la autenticación multifactor adaptativa." cta="Ver glosario" href="/docs/es/glossary?term=Attack+Protection">Attack Protection</Tooltip>.** Protege a tus usuarios contra ataques de fuerza bruta y contraseñas vulneradas. Para obtener más información, lee [Attack Protection](/docs/es/secure/attack-protection).

  ### Configuración
</div>

<Frame>![Pestaña Configuración general de Configuración del tenant en el Dashboard](/docs/images/cdy7uua7fh8z/4okToiwlkNQBxwr8QGE3Rs/7e96db0efab4082ebb9f78cb43ee302c/Basic_Settings_-_EN.png)</Frame>

* **Nombre para mostrar**: Nombre que quiere que se muestre a sus usuarios en la página de <Tooltip tip="Universal Login: Su aplicación redirige a Universal Login, alojado en el servidor de autorización de Auth0, para verificar la identidad de un usuario." cta="Ver glosario" href="/docs/es/glossary?term=Universal+Login">Universal Login</Tooltip>. * **correo electrónico de soporte**: Dirección de correo electrónico que se utiliza para ponerse en contacto con su equipo de soporte.
* **URL de soporte**: Enlace a la página de soporte de su empresa u organización.

<Frame>![Dashboard Configuración del tenant Pestaña General Configuración de autorización de API](/docs/images/cdy7uua7fh8z/17yNCRrB6dWcHp6zLFb74j/d1f7c516bce3bf66e9f4bfbd201411a3/Auth_Settings_-_EN.png)</Frame>

* **Predeterminada <Tooltip tip="Audiencia: identificador único de la audiencia de un token emitido. Se denomina aud en un token; su valor contiene el ID de una aplicación (ID de cliente) para un token de ID o de una API (identificador de API) para un token de acceso." cta="Ver glosario" href="/docs/es/glossary?term=Audience">audiencia</Tooltip>**: identificador de API que se usará para los [flujos de autorización](/docs/es/get-started/authentication-and-authorization-flow). Su valor debe ser el nombre exacto de una conexión existente para una de las siguientes estrategias: `auth0-adldap`, `ad`, `auth0`, `email`, `sms`, `waad` o `adfs`.

<div id="signing-keys">
  ## Claves de firma
</div>

En la pestaña **Claves de firma**, puedes administrar de forma segura la clave de firma y el certificado que se usan para firmar <Tooltip tip="token de ID: Credencial destinada al propio cliente, en lugar de para acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=ID+tokens">tokens de ID</Tooltip>, tokens de acceso, <Tooltip tip="Security Assertion Markup Language (SAML): Protocolo estandarizado que permite a dos partes intercambiar información de autenticación sin una contraseña." cta="Ver glosario" href="/docs/es/glossary?term=SAML">aserciones SAML</Tooltip> y <Tooltip tip="Security Assertion Markup Language (SAML): Protocolo estandarizado que permite a dos partes intercambiar información de autenticación sin una contraseña." cta="Ver glosario" href="/docs/es/glossary?term=WS-Fed">aserciones WS-Fed</Tooltip> que se envían a tus aplicaciones.

<Frame>![Pestaña Claves de firma de Configuración del tenant en el Dashboard](/docs/images/cdy7uua7fh8z/7r8t3EGctFmvkCgPrU0i2R/f79ac74dac5cded37b39bd8a8b80951d/dashboard-tenant-settings-signing-keys.png)</Frame>

* **Configuración de rotación**: Configuración que te permite rotar la clave de firma y el certificado de la aplicación. Puedes elegir si quieres revocar o no la clave de firma al rotarla. Para obtener más información, consulta [Signing Keys](/docs/es/get-started/tenant-settings/signing-keys).

  * **Rotar clave de firma**: Rota la clave de firma sin revocarla; en la práctica, mueve la clave actual a la clave anterior. Todos los tokens firmados con la clave anterior seguirán siendo válidos hasta que se revoque.
  * **Rotar y revocar clave de firma**: Rota la clave de firma y luego la revoca; en la práctica, mueve la clave actual a la clave anterior y después invalida la clave anterior. Asegúrate de haber actualizado tu aplicación con la siguiente clave de la cola antes de rotar y revocar la clave actual.
* **Lista de claves válidas**: Lista de claves de firma válidas para tu tenant, que también están disponibles en el endpoint de Metadata de tu aplicación. Las claves válidas incluyen:

  * **Siguiente en la cola**: Clave que se usará la próxima vez que se rote la clave de firma.
  * **En uso actualmente**: Clave que está en uso actualmente.
  * **Usada anteriormente**: Clave que se usó anteriormente. Su presencia indica que la clave de firma se ha rotado, pero que la clave usada anteriormente aún no se ha revocado.
* **Lista de claves revocadas**: Lista de las últimas tres claves revocadas de tu tenant. Hay más información sobre las claves revocadas disponible en los registros del tenant.

<div id="login-and-logout">
  ### Inicio de sesión y cierre de sesión
</div>

<Frame>![Dashboard Configuración del tenant pestaña Avanzado Inicio de sesión y cierre de sesión](/docs/images/cdy7uua7fh8z/Login_and_Logout_-_EN.png)</Frame>

* **URI de inicio de sesión del tenant**: URI que apunta a una ruta de su aplicación que inicia el flujo de inicio de sesión de OIDC redirigiendo al endpoint `/authorize`; debe tener la forma `https://mytenant.org/login`. Esto solo se usará en los casos en que Auth0 necesite que su tenant inicie el flujo de inicio de sesión de OIDC. Para obtener más información, consulte [Configurar rutas de inicio de sesión predeterminadas](/docs/es/authenticate/login/auth0-universal-login/configure-default-login-routes).
* **Allowed Logout URLs**: URL a las que Auth0 puede redirigir después del cierre de sesión cuando no se especifica ningún client&#95;id al invocar el endpoint de cierre de sesión. Resulta útil como lista global cuando el inicio de sesión único (SSO) está habilitado. Para obtener más información, lea [cierre de sesión](docs/authenticate/login/logout).
* **Valores ACR permitidos**: Lista de valores permitidos de Authentication Context Class Reference (ACR). Estos valores se incluyen en el documento OpenID Configuration. Cuando se completa esta lista, los valores que no figuren en ella se rechazarán si se usan en flujos de autenticación.
* **Detección del endpoint de fin de sesión de RP-Initiated Logout**: Controla si el endpoint de cierre de sesión se publica en las respuestas de OpenID Configuration como `end_session_endpoint`.
* **Confirmación del usuario final de RP-Initiated Logout**: Controla si se le pide al usuario que confirme el inicio de sesión si la solicitud de cierre de sesión iniciada por RP no incluye las pistas correctas.
* **Confirmación del usuario final para Non-Verifiable Callback URI**: Controla si se le pide al usuario que confirme el inicio de sesión cuando se usa un esquema URI personalizado como devolución de llamada. Auth0 recomienda que **no** omita la confirmación del usuario final en estos casos. Para obtener más información, lea [Medidas contra la suplantación de aplicaciones](docs/secure/security-guidance/measures-against-app-impersonation.mdx).

<div id="login-session-management">
  ### Gestión de la sesión de inicio de sesión
</div>

La configuración de **Gestión de la sesión de inicio de sesión** define la duración de la sesión de inicio de sesión, que representa la capa de sesión del <Tooltip tip="Servidor de autorización: servidor centralizado que contribuye a definir los límites del acceso de un usuario. Por ejemplo, su servidor de autorización puede controlar los datos, las tareas y las funcionalidades disponibles para un usuario." cta="Ver glosario" href="/docs/es/glossary?term=Authorization+Server">servidor de autorización</Tooltip> de Auth0. La capa de sesión del servidor de autorización impulsa el inicio de sesión único (SSO). Para obtener más información, consulte [Inicio de sesión único](/docs/es/authenticate/single-sign-on).

Los tiempos de expiración de los tokens emitidos por Auth0 pueden configurarse en otra parte. Los tiempos de expiración de los tokens suelen usarse para controlar la capa de sesión de la aplicación y aparecen en los claims del token, como el claim de expiración de los tokens de ID de <Tooltip tip="OpenID: estándar abierto de autenticación que permite que las aplicaciones verifiquen la identidad de los usuarios sin recopilar ni almacenar información de inicio de sesión." cta="Ver glosario" href="/docs/es/glossary?term=OpenID">OpenID</Tooltip> Connect (OIDC) o la assertion de duración de SAML.

<Frame>![Dashboard Configuración del tenant Avanzado Gestión de la sesión de inicio de sesión](/docs/images/cdy7uua7fh8z/q2fGPzoUqCcXj7OxcHhjy/de15afcd26e8ea80937733034ffbd387/Screenshot_2024-11-06_at_11.02.16.png)</Frame>

* **Tiempo de inactividad máximo**: Periodo (en minutos) tras el cual la sesión de un usuario expirará si no ha interactuado con el servidor de autorización. Los límites del sistema prevalecerán si supera los 4.320 minutos (3 días) para planes no Enterprise o los 144.000 minutos (100 días) para planes Enterprise.
* **Tiempo tras el cual es necesario iniciar sesión de nuevo**: Periodo (en minutos) tras el cual se requerirá que un usuario inicie sesión de nuevo, independientemente de su actividad. Los límites del sistema prevalecerán si supera los 43.200 minutos (30 días) para planes no Enterprise o los 525.600 minutos (365 días) para planes Enterprise.

<div id="settings-advanced">
  ### Configuración (Avanzado)
</div>

* **Flujo de cambio de contraseña v2**: Cuando se habilita, se usa la versión más reciente del flujo de cambio de contraseña. La versión anterior ha quedado obsoleta y recomendamos encarecidamente habilitar la v2. Este indicador se incluye solo por compatibilidad con versiones anteriores y, una vez habilitado, ya no se puede deshabilitar. Puede personalizar la interfaz de usuario del widget de cambio de contraseña en la pestaña [Universal Login &gt; Restablecimiento de contraseña](https://manage.auth0.com/#/password_reset) del Auth0 Dashboard.
* **Registro dinámico de clientes (DCR)**: Cuando está habilitado, los desarrolladores externos podrán registrar dinámicamente aplicaciones para sus API. También puede actualizar este indicador mediante el endpoint [`/tenant/patch_settings`](https://auth0.com/docs/api/management/v2#!/Tenants/patch_settings) de Auth0 <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip>. De forma predeterminada, esta función está deshabilitada. Para obtener más información, consulte [Registro dinámico de clientes](/docs/es/get-started/applications/dynamic-client-registration).
* **Perfil de compatibilidad del parámetro de recurso**: El perfil de compatibilidad del parámetro de recurso determina cómo Auth0 gestiona el parámetro `resource` en las solicitudes de autorización.
  * Cuando está habilitado:
    * Auth0 usa el parámetro `resource` para especificar a qué servidor de recursos (API) quiere acceder la aplicación cliente
    * Auth0 comprueba primero el parámetro `audience`; si no se proporciona, recurre al parámetro `resource`
    * Auth0 usa internamente el parámetro `resource` y no lo reenvía al IdP ascendente
  * Cuando está deshabilitado (por defecto):
    * Auth0 usa solo el parámetro `audience` para especificar el servidor de recursos
    * El parámetro `resource` se trata como un parámetro del IdP ascendente y se reenviará al IdP
* **Registro del Documento de metadatos del ID de cliente (CIMD)**: Habilita el registro manual con CIMD importando desde una URL un Documento de metadatos del ID de cliente (CIMD) alojado externamente. CIMD es un archivo JSON que contiene metadatos del cliente alojados en un dominio HTTPS seguro controlado por la aplicación. Para obtener más información, lee [Registrar aplicaciones con CIMD](/docs/es/get-started/auth0-overview/create-applications/register-applications-with-cimd).
* **Habilitar conexiones de la aplicación**: Cuando se habilita, todas las conexiones actuales se habilitarán para cualquier aplicación nueva que se cree.
* **Usar una respuesta genérica en el mensaje de error de la API pública de registro**: Cuando está habilitada, los errores generados al usar la API pública de registro devolverán una respuesta genérica. Esto ayuda a evitar la enumeración de registros de usuarios, ya que impide que los <Tooltip tip="Actores maliciosos: Entidad (una persona o grupo) que representa una amenaza para la empresa o el entorno con la intención de causar daño." cta="Ver glosario" href="/docs/es/glossary?term=bad+actors">actores maliciosos</Tooltip> puedan deducir identificadores ya registrados (username, email o phone) a partir de los códigos de error de la respuesta, como `user_exists`.
* **Habilitar la publicación de información de conexiones empresariales con dominios de <Tooltip tip="Proveedor de identidad (IdP): servicio que almacena y administra identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=IdP">IdP</Tooltip>**: Al habilitarlo, se admite [Home Realm Discovery](/docs/es/authenticate/login/auth0-universal-login/identifier-first) y [Auth0 Lock](/docs/es/libraries/lock) depende de un archivo público verificado que incluye información sobre conexiones empresariales. Si no necesitas esa funcionalidad, puedes deshabilitarla.
* **Habilitar el flujo de verificación de correo electrónico durante el inicio de sesión para las conexiones de Azure AD y ADFS**: Al habilitar esta opción, se mostrará a los usuarios un prompt de verificación de correo electrónico durante su primer inicio de sesión cuando usen conexiones de Azure AD o ADFS.
* **<Tooltip tip="Token de actualización: token que se usa para obtener un nuevo token de acceso sin obligar a los usuarios a volver a iniciar sesión." cta="Ver glosario" href="/docs/es/glossary?term=Refresh+Token">Token de actualización</Tooltip> La revocación elimina la concesión**: Cuando está habilitado, elimina la concesión asociada cuando revocas un token de actualización mediante el endpoint `/oauth/revoke` de la Authentication API.

  <Callout icon="file-lines" color="#0EA5E9" iconType="regular">
    Para los tenants existentes, esta función viene habilitada de forma predeterminada para preservar el comportamiento existente. Para los tenants nuevos (a partir del 13 de enero de 2021), esta función viene deshabilitada de forma predeterminada para garantizar que revocar un token de actualización no revoque la concesión. Si es necesario revocar una concesión, se debe enviar una solicitud independiente mediante un endpoint de revocación de concesiones.
  </Callout>
* **Permitir nombres de organización en Authentication API**: Cuando se habilita, los endpoints [/authorize](https://auth0.com/docs/api/authentication#authorize-application) y [SAML](https://auth0.com/docs/api/authentication#saml) pueden aceptar tanto IDs como nombres de organización. Además, los tokens de ID y de acceso incluirán los claims `org_id` y `org_name`. Antes de habilitar esta configuración, revisa [Use Organization Names in Authentication API](/docs/es/manage-users/organizations/configure-organizations/use-org-name-authentication-api) para tener en cuenta consideraciones importantes y posibles impactos.
* **Permitir solicitudes de autorización push (PAR):** Cuando se habilita, el endpoint `/par` puede aceptar solicitudes de autorización enviadas por una aplicación cliente. Esto evita que la aplicación cliente envíe solicitudes a través del front channel no seguro (es decir, el navegador).
