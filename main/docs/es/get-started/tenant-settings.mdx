Usa la página **Configuración del tenant** del <Tooltip tip="Auth0 Dashboard: el producto principal de Auth0 para configurar tus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip>, en [Dashboard &gt; Settings](https://manage.auth0.com/#/tenant), para configurar varios ajustes relacionados con tu tenant de Auth0.

<div id="recommended-settings">
  ## Ajustes recomendados
</div>

Cuando configures tu tenant, establece los siguientes elementos:

* **Especifica la etiqueta de entorno.** Los tenants etiquetados como **Production** tienen límites de frecuencia más altos que los tenants etiquetados como **Development** o **Staging**. En los planes que no son Enterprise, solo un tenant por suscripción puede etiquetarse como **Production**. Para obtener más información, consulta [Set Up Multiple Environments](/docs/es/get-started/auth0-overview/create-tenants/set-up-multiple-environments).
* **Configura el correo electrónico de soporte y la URL de soporte.** Si un usuario tiene algún problema al iniciar sesión, querrá pedir ayuda. Configura estos valores para dirigirlo a una dirección de correo electrónico o a una página de destino donde pueda obtener asistencia.
* **Configura una página de error personalizada.** Si es posible, deberías alojar tu propia página de error personalizada y configurar Auth0 para que la use en lugar de la página predeterminada. Esto te permite ofrecer a los usuarios explicaciones más completas y personalizadas sobre qué hacer en caso de error.
* **Configura un <Tooltip tip="Custom Domain: Dominio de terceros con un nombre especializado o personalizado." cta="Ver glosario" href="/docs/es/glossary?term=custom+domain">dominio personalizado</Tooltip>.** Si tienes un plan de pago, puedes configurar un dominio personalizado para tu tenant de Auth0. Un dominio personalizado unifica la experiencia de inicio de sesión con tu marca y ofrece ventajas adicionales. Para obtener más información, consulta [Custom Domains](/docs/es/customize/custom-domains).
* **Configura el tiempo de espera de la sesión de <Tooltip tip="Inicio de sesión único (SSO): Servicio que, después de que un usuario inicia sesión en una aplicación, inicia automáticamente su sesión en otras aplicaciones." cta="Ver glosario" href="/docs/es/glossary?term=Single+Sign-On">inicio de sesión único</Tooltip> (SSO).** El valor del tiempo de espera de la sesión de SSO especifica cuánto tarda en expirar la sesión de un usuario. El valor predeterminado es de 7 días, que es el tiempo durante el cual los usuarios pueden acceder a tus aplicaciones integradas con Auth0 sin volver a introducir sus credenciales. Para obtener más información, consulta [Sessions](/docs/es/manage-users/sessions).
* **Configura miembros del tenant.** Configura usuarios adicionales del Auth0 Dashboard y habilita la <Tooltip tip="Autenticación multifactor (MFA): Proceso de autenticación de usuario que utiliza un factor además del nombre de usuario y la contraseña, como un código por SMS." cta="Ver glosario" href="/docs/es/glossary?term=multi-factor+authentication">autenticación multifactor</Tooltip> (MFA). Para obtener más información, consulta [Administrar el acceso al Dashboard](/docs/es/get-started/manage-dashboard-access) y [Manage Dashboard Access with Multi-Factor Authentication](/docs/es/get-started/manage-dashboard-access/add-change-remove-mfa).
* **Deshabilita la configuración Enable Application Connections.** Si esta configuración está habilitada, todas las conexiones configuradas se habilitarán automáticamente para cualquier aplicación nueva que crees. Como resultado, los usuarios podrían iniciar sesión en la aplicación mediante conexiones que no pretendías poner a su disposición. Deshabilita esta configuración para poder habilitar explícitamente las conexiones adecuadas para cada aplicación.
* **Habilita <Tooltip tip="Attack Protection: Funciones que Auth0 proporciona para detectar y mitigar ataques, incluida la protección contra ataques de fuerza bruta, la limitación de IP sospechosas, la detección de contraseñas vulneradas, la detección de bots y la autenticación multifactor adaptativa." cta="Ver glosario" href="/docs/es/glossary?term=Attack+Protection">Attack Protection</Tooltip>.** Protege a tus usuarios frente a ataques de fuerza bruta y contraseñas vulneradas. Para obtener más información, consulta [Attack Protection](/docs/es/secure/attack-protection).

</div>

* **Nombre visible**: Nombre que quieres que se muestre a tus usuarios en la página de <Tooltip tip="Universal Login: Tu aplicación redirige a Universal Login, alojado en el servidor de autorización de Auth0, para verificar la identidad de un usuario." cta="Ver glosario" href="/docs/es/glossary?term=Universal+Login">Universal Login</Tooltip>. * **Correo electrónico de soporte**: Dirección de correo electrónico que se utiliza para contactar con tu equipo de soporte.
* **URL de soporte**: Enlace a la página de soporte de tu empresa u organización.

  ### Configuración de autorización de API
</div>

<Frame>![Dashboard Configuración del tenant Pestaña General Configuración de autorización de API](/docs/images/cdy7uua7fh8z/17yNCRrB6dWcHp6zLFb74j/d1f7c516bce3bf66e9f4bfbd201411a3/Auth_Settings_-_EN.png)</Frame>

* **Audiencia predeterminada <Tooltip tip="Audiencia: Identificador único de la audiencia de un token emitido. Se denomina aud en un token; su valor contiene el ID de una aplicación (ID de cliente) para un token de ID o de una API (identificador de API) para un token de acceso." cta="Ver glosario" href="/docs/es/glossary?term=Audience">Audiencia</Tooltip>**: Identificador de API que se usará para los [flujos de autorización](/docs/es/get-started/authentication-and-authorization-flow). Esto provocará un nuevo comportamiento que podría generar cambios importantes en algunas de tus aplicaciones. Ponte en contacto con [soporte](/docs/es/troubleshoot/customer-support/open-and-manage-support-tickets) si necesitas ayuda.

Su valor debe ser el nombre exacto de una conexión existente para una de las siguientes estrategias: `auth0-adldap`, `ad`, `auth0`, `email`, `sms`, `waad` o `adfs`.

<div id="signing-keys">
  ## Claves de firma
</div>

En la pestaña **Signing Keys**, puedes gestionar de forma segura la clave de firma y el certificado que se utilizan para firmar <Tooltip tip="token de ID: Credencial destinada al propio cliente, en lugar de acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=ID+tokens">tokens de ID</Tooltip>, tokens de acceso, <Tooltip tip="Security Assertion Markup Language (SAML): Protocolo estandarizado que permite a dos partes intercambiar información de autenticación sin una contraseña." cta="Ver glosario" href="/docs/es/glossary?term=SAML">aserciones SAML</Tooltip> y <Tooltip tip="Security Assertion Markup Language (SAML): Protocolo estandarizado que permite a dos partes intercambiar información de autenticación sin una contraseña." cta="Ver glosario" href="/docs/es/glossary?term=WS-Fed">aserciones WS-Fed</Tooltip> que se envían a tus aplicaciones.

<Frame>![Pestaña Claves de firma de la Configuración del tenant en el Dashboard](/docs/images/cdy7uua7fh8z/7r8t3EGctFmvkCgPrU0i2R/f79ac74dac5cded37b39bd8a8b80951d/dashboard-tenant-settings-signing-keys.png)</Frame>

* **Configuración de rotación**: Configuración que te permite rotar la clave de firma de la aplicación y el certificado. Puedes elegir si quieres revocar o no la clave de firma al rotarla. Para obtener más información, consulta [Signing Keys](/docs/es/get-started/tenant-settings/signing-keys).

  * **Rotar clave de firma**: Rota la clave de firma sin revocarla; es decir, mueve la clave actual a la posición de clave anterior. Todos los tokens firmados con la clave anterior seguirán siendo válidos hasta que se revoque.
  * **Rotar y revocar clave de firma**: Rota la clave de firma y luego la revoca; es decir, mueve la clave actual a la posición de clave anterior y después invalida la clave anterior. Asegúrate de haber actualizado tu aplicación con la siguiente clave de la cola antes de rotar y revocar la clave actual.
* **Lista de claves válidas**: Lista de claves de firma de aplicaciones válidas de tu tenant, que también están disponibles en el endpoint de Metadata de tu aplicación. Las claves válidas incluyen:

  * **Siguiente en la cola**: Clave que se utilizará la próxima vez que se rote la clave de firma.
  * **En uso actualmente**: Clave que se está utilizando actualmente.
  * **Usada anteriormente**: Clave que se utilizó anteriormente. Su presencia indica que la clave de firma se ha rotado, pero la clave usada anteriormente aún no se ha revocado.
* **Lista de claves revocadas**: Lista de las tres últimas claves revocadas de tu tenant. Hay más información sobre las claves revocadas disponible en los registros del tenant.

<div id="login-and-logout">
  ### Inicio de sesión y cierre de sesión
</div>

<Frame>![Dashboard Configuración del tenant pestaña Advanced Inicio de sesión y cierre de sesión](/docs/images/cdy7uua7fh8z/Login_and_Logout_-_EN.png)</Frame>

* **Tenant Login URI**: URI que apunta a una ruta de tu aplicación que inicia el flujo de inicio de sesión de OIDC redirigiendo al endpoint `/authorize`; debe tener el formato `https://mytenant.org/login`. Solo se usará en los casos en que Auth0 necesite que tu tenant inicie el flujo de inicio de sesión de OIDC. Para obtener más información, consulta [Configurar rutas de inicio de sesión predeterminadas](/docs/es/authenticate/login/auth0-universal-login/configure-default-login-routes).
* **Allowed Logout URLs**: URL a las que Auth0 puede redirigir después del cierre de sesión cuando no se especifica `client_id` al invocar el endpoint de cierre de sesión. Resulta útil como lista global cuando el inicio de sesión único (SSO) está habilitado. Para obtener más información, consulta [Cierre de sesión](docs/authenticate/login/logout).
* **Allowed ACR Values**: Lista de valores permitidos de Authentication Context Class Reference (ACR). Estos valores se incluyen en el documento de configuración de OpenID. Cuando se completa esta lista, los valores que no figuren en ella se rechazarán si se usan en flujos de autenticación.
* **RP-Initiated Logout End Session Endpoint Discovery**: Controla si el endpoint de cierre de sesión se publica en las respuestas de configuración de OpenID como `end_session_endpoint`.
* **RP-Initiated Logout End-User Confirmation**: Controla si se le pide al usuario que confirme el inicio de sesión si la request de RP-Initiated Logout no incluye las pistas correctas.
* **Non-Verifiable Callback URI End-User Confirmation**: Controla si se le pide al usuario que confirme el inicio de sesión cuando se usa un custom URI scheme como devolución de llamada. Auth0 recomienda **no** omitir la confirmación del usuario final en estos casos. Para obtener más información, consulta [Measures Against Application Impersonation](docs/secure/security-guidance/measures-against-app-impersonation.mdx).

<div id="login-session-management">
  ### Gestión de sesiones de inicio de sesión
</div>

La configuración de **Gestión de sesiones de inicio de sesión** determina la duración de la sesión de inicio de sesión, que corresponde a la capa de sesión del <Tooltip tip="Servidor de autorización: servidor centralizado que contribuye a definir los límites del acceso de un usuario. Por ejemplo, su servidor de autorización puede controlar los datos, las tareas y las funcionalidades disponibles para un usuario." cta="Ver glosario" href="/docs/es/glossary?term=Authorization+Server">servidor de autorización</Tooltip> de Auth0. La capa de sesión del servidor de autorización sustenta el inicio de sesión único (SSO). Para obtener más información, consulta [inicio de sesión único](/docs/es/authenticate/single-sign-on).

Los tiempos de espera de los tokens emitidos por Auth0 pueden configurarse en otros apartados. A menudo, estos tiempos de espera se usan para controlar la capa de sesión de la aplicación y aparecen en los claims del token, como en el claim de expiración de los tokens de ID de <Tooltip tip="OpenID: estándar abierto de autenticación que permite a las aplicaciones verificar las identidades de los usuarios sin recopilar ni almacenar información de inicio de sesión." cta="Ver glosario" href="/docs/es/glossary?term=OpenID">OpenID</Tooltip> Connect (OIDC) o en la aserción de duración de SAML.

<Frame>![Dashboard Configuración del tenant Avanzado Gestión de sesiones de inicio de sesión](/docs/images/cdy7uua7fh8z/q2fGPzoUqCcXj7OxcHhjy/de15afcd26e8ea80937733034ffbd387/Screenshot_2024-11-06_at_11.02.16.png)</Frame>

* **Tiempo de espera por inactividad**: período (en minutos) tras el cual la sesión de un usuario expirará si no ha interactuado con el servidor de autorización. Se aplicarán los límites del sistema si supera los 4.320 minutos (3 días) en planes no Enterprise o los 144.000 minutos (100 días) en planes Enterprise.
* **Tiempo tras el cual es necesario iniciar sesión de nuevo**: período (en minutos) tras el cual se requerirá que un usuario inicie sesión de nuevo, independientemente de su actividad. Se aplicarán los límites del sistema si supera los 43.200 minutos (30 días) en planes no Enterprise o los 525.600 minutos (365 días) en planes Enterprise.

<div id="settings-advanced">
  ### Configuración (Avanzado)
</div>

* **Flujo de cambio de contraseña v2**: Cuando está habilitado, se utilizará la versión más reciente del flujo de cambio de contraseña. La versión anterior ha quedado obsoleta y recomendamos encarecidamente habilitar la v2. Este indicador solo se muestra por compatibilidad con versiones anteriores y, una vez habilitado, ya no podrás deshabilitarlo. Puedes personalizar la interfaz de usuario del widget de cambio de contraseña en la pestaña [Universal Login &gt; Restablecimiento de contraseña](https://manage.auth0.com/#/password_reset) del Auth0 Dashboard.
* **Registro dinámico de clientes (DCR)**: Cuando está habilitado, los desarrolladores externos podrán registrar dinámicamente aplicaciones para sus API. También puede actualizar este indicador mediante el endpoint [`/tenant/patch_settings`](https://auth0.com/docs/api/management/v2#!/Tenants/patch_settings) de la <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip> de Auth0. De forma predeterminada, esta funcionalidad está deshabilitada. Para obtener más información, consulte [Registro dinámico de clientes](/docs/es/get-started/applications/dynamic-client-registration).
* **Perfil de compatibilidad del parámetro resource**: El perfil de compatibilidad del parámetro resource determina cómo Auth0 gestiona el parámetro `resource` en las solicitudes de autorización.
  * Cuando está activado:
    * Auth0 usa el parámetro `resource` para especificar a qué servidor de recursos (API) quiere acceder la aplicación cliente
    * Auth0 comprueba primero el parámetro `audience`; si no se proporciona, recurre al parámetro `resource`
    * Auth0 procesa el parámetro `resource` y no lo reenvía al IdP de origen
  * Cuando está desactivado (predeterminado):
    * Auth0 usa solo el parámetro `audience` para especificar el servidor de recursos
    * El parámetro `resource` se trata como un parámetro del IdP de origen y se reenviará al IdP
* **Registro del Documento de metadatos del ID de cliente (CIMD)**: Habilite el registro manual con CIMD importando desde una URL un Documento de metadatos del ID de cliente (CIMD) alojado externamente. CIMD es un archivo JSON que contiene metadatos del cliente y está alojado en un dominio HTTPS seguro controlado por la aplicación. Para obtener más información, lea [Registrar aplicaciones con CIMD](/docs/es/get-started/auth0-overview/create-applications/register-applications-with-cimd).
* **Habilitar conexiones de la aplicación**: Si se habilita, todas las conexiones actuales se habilitarán para cualquier aplicación nueva que se cree.
* **Usar una respuesta genérica en el mensaje de error de la API pública de registro**: Cuando está habilitado, los errores generados al usar la API pública de registro devolverán una respuesta genérica. Esto ayuda a proteger contra la enumeración de usuarios registrados al impedir que los <Tooltip tip="Actores maliciosos: Entidad (una persona o grupo) que representa una amenaza para la empresa o el entorno con la intención de causar daño." cta="Ver glosario" href="/docs/es/glossary?term=bad+actors">actores maliciosos</Tooltip> puedan deducir identificadores ya registrados (username, email o phone) al leer códigos de respuesta de error, como `user_exists`.
* **Habilitar la publicación de información sobre conexiones empresariales con dominios de <Tooltip tip="Proveedor de identidad (IdP): Servicio que almacena y gestiona identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=IdP">IdP</Tooltip>**: Cuando está habilitado, es compatible con [Home Realm Discovery](/docs/es/authenticate/login/auth0-universal-login/identifier-first) y [Auth0 Lock](/docs/es/libraries/lock) se basa en un archivo público validado que incluye información sobre conexiones empresariales. Si no necesita esta funcionalidad, puede deshabilitarla.
* **Habilitar el flujo de verificación del correo electrónico durante el inicio de sesión para las conexiones de Azure AD y ADFS**: Al habilitar esta opción, a los usuarios se les mostrará un prompt de verificación del correo electrónico durante su primer inicio de sesión cuando usen conexiones de Azure AD o ADFS.
* **<Tooltip tip="Token de actualización: token usado para obtener un nuevo token de acceso sin obligar a los usuarios a volver a iniciar sesión." cta="Ver glosario" href="/docs/es/glossary?term=Refresh+Token">Token de actualización</Tooltip> La revocación elimina la concesión**: Cuando está habilitado, elimina la concesión subyacente al revocar un token de actualización mediante el endpoint `/oauth/revoke` de la Authentication API.

  <Callout icon="file-lines" color="#0EA5E9" iconType="regular">
    En los tenants existentes, esta funcionalidad está habilitada de forma predeterminada para preservar el comportamiento existente. En los tenants nuevos (a partir del 13 de enero de 2021), esta funcionalidad está deshabilitada de forma predeterminada para garantizar que la revocación de un token de actualización no revoque la concesión. Si es necesario revocar una concesión, se debe enviar una solicitud independiente mediante un endpoint de revocación de concesión.
  </Callout>
* **Permitir nombres de organización en Authentication API**: Cuando está habilitado, los endpoints [/authorize](https://auth0.com/docs/api/authentication#authorize-application) y [SAML](https://auth0.com/docs/api/authentication#saml) pueden aceptar tanto IDs como nombres de organización. Además, los tokens de ID y de acceso incluirán tanto el claim `org_id` como `org_name`. Antes de habilitar esta configuración, revisa [Use Organization Names in Authentication API](/docs/es/manage-users/organizations/configure-organizations/use-org-name-authentication-api) para conocer consideraciones importantes y posibles impactos.
* **Permitir solicitudes de autorización push (PAR):** Cuando está habilitado, el endpoint `/par` puede aceptar solicitudes de autorización enviadas directamente desde una aplicación cliente. Esto evita que la aplicación cliente envíe requests a través del front channel no seguro (es decir, el navegador).
