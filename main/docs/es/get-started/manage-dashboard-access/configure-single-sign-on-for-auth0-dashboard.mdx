<Card title="Antes de empezar">
  * Suscríbete a un plan Enterprise. Para obtener más información, consulta [Manage Subscriptions](/docs/es/troubleshoot/customer-support/manage-subscriptions).
  * Determina si usas Public Cloud o Private Cloud. Para conocer los requisitos previos relacionados con Private Cloud para estas instrucciones, consulta [Deploy Private Cloud](/docs/es/deploy-monitor/deploy-private-cloud).
</Card>

Puedes configurar Auth0 para permitir que los [miembros del tenant](/docs/es/get-started/manage-dashboard-access) usen su propio [proveedor de identidad empresarial](/docs/es/authenticate/identity-providers/enterprise-identity-providers) (<Tooltip tip="Proveedor de identidad (IdP): servicio que almacena y administra identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=IdP">IdP</Tooltip>) para autenticarse en <Tooltip tip="Auth0 Dashboard: el producto principal de Auth0 para configurar tus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip> mediante <Tooltip tip="Auth0 Dashboard: el producto principal de Auth0 para configurar tus servicios." cta="Ver glosario" href="/docs/es/glossary?term=single+sign-on">inicio de sesión único</Tooltip> (SSO).

<div id="how-it-works">
  ## Cómo funciona
</div>

Configurar el SSO para el Dashboard requiere trabajar con el equipo de soporte de Auth0 para añadir una conexión empresarial a la autoridad raíz del tenant (RTA), que gestiona los métodos de autenticación disponibles que un miembro del tenant puede usar para iniciar sesión en el Dashboard.

Añadir esta conexión SSO no impide que los miembros del tenant inicien sesión con los métodos de autenticación existentes (como correo electrónico/contraseña, [LinkedIn](https://marketplace.auth0.com/integrations/linkedin-social-connection), [Microsoft](https://marketplace.auth0.com/integrations/microsoft-account-social-connection), [GitHub](https://marketplace.auth0.com/integrations/github-social-connection) o [Google](https://marketplace.auth0.com/integrations/google-social-connection)).

Configurar el SSO para el Dashboard también habilita el SSO para todos los sitios públicos de Auth0, como:

* sitio web de Auth0 (https://auth0.com)
* Auth0 Community (https://community.auth0.com)
* Auth0 Docs (https://www.auth0.com/docs)
* Auth0 Support Center (https://support.auth0.com)

<div id="user-experience">
  ### Experiencia de usuario
</div>

Cuando un usuario autorizado intenta iniciar sesión en el Dashboard, introduce su dirección de correo electrónico correspondiente a un dominio registrado (por ejemplo, `user@example.com`) en la página de Auth0 <Tooltip tip="Universal Login: Su aplicación redirige a Universal Login, alojado en el servidor de autorización de Auth0, para verificar la identidad de un usuario." cta="Ver glosario" href="/docs/es/glossary?term=Universal+Login">Universal Login</Tooltip>, y luego se le redirige a su IdP para completar la autenticación.

<div id="limitations">
  ### Limitaciones
</div>

Antes de decidir si configurar SSO para el Dashboard, ten en cuenta estas restricciones:

* SSO no puede limitarse a tenants específicos.
* SSO no admite flujos de autenticación iniciados por el IdP.
* Las invitaciones de miembro del tenant no pueden automatizarse ni enviarse de forma masiva mediante la Auth0 <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip>.
* El acceso de miembro del tenant no puede administrarse mediante la pertenencia a grupos de tu IdP.
* No es posible exigir <Tooltip tip="Autenticación multifactor (MFA): Proceso de autenticación de usuario que utiliza un factor además de nombre de usuario y contraseña, como un código por SMS." cta="Ver glosario" href="/docs/es/glossary?term=MFA">MFA</Tooltip> a todos los miembros de un tenant.

<div id="considerations">
  ### Consideraciones
</div>

<div id="full-directory-access-to-dashboard">
  #### Acceso completo del directorio al Dashboard
</div>

Cuando agregas tu IdP como una conexión disponible para que los miembros del tenant inicien sesión, todos los usuarios del directorio de tu IdP pueden acceder al Dashboard, pero solo los miembros del tenant que hayan sido invitados a un tenant determinado pueden acceder a ese tenant.

Cuando los usuarios intentan acceder a un tenant en el Dashboard sin una invitación, el sistema deniega la operación. Si los usuarios no pertenecen a ningún tenant, el sistema les solicita que [completen su perfil de usuario](#complete-user-profiling) y creen un nuevo tenant de prueba. Crear un nuevo tenant de prueba no lo asocia con tu Enterprise plan.

<div id="residual-tenant-member-identities">
  #### Identidades residuales de miembro del tenant
</div>

Si se invitó a un miembro del tenant (y se le dio acceso) a un tenant en el Dashboard con una identidad distinta de la que se creó en la nueva conexión, técnicamente podrá seguir usando esa identidad para acceder al tenant.

Tendrás que decidir si quieres eliminar su identidad anterior o conservarla como un posible método de autenticación alternativo.

<div id="configure-sso-for-the-dashboard">
  ## Configurar SSO para el Dashboard
</div>

La configuración de SSO para el Dashboard requiere una serie de pasos que se reparten entre usted y el equipo de soporte de Auth0.

<div id="share-idp-configuration-data">
  ### Comparte los datos de configuración del IdP
</div>

[Abre un ticket con el equipo de soporte de Auth0](/docs/es/troubleshoot/customer-support/open-and-manage-support-tickets) para compartir los datos de configuración de tu IdP y que puedan configurar el SSO. Incluye la siguiente información al enviar tu ticket:

* Dominio(s) de correo electrónico que te gustaría asociar con la configuración de SSO
* Nombre del IdP
* Protocolo de autenticación
* Información adicional específica del IdP

Se requieren pasos de configuración adicionales según el IdP y el protocolo de autenticación que quieras usar:

<div id="adfs-saml">
  #### ADFS (SAML)
</div>

1. [Cree una Relying Party Trust](https://learn.microsoft.com/en-us/windows-server/identity/ad-fs/operations/create-a-relying-party-trust#to-create-a-claims-aware-relying-party-trust-manually) con las siguientes propiedades:

   | Propiedad                         | Valores                                    |
   | --------------------------------- | ------------------------------------------ |
   | ID de entidad                     | `urn:auth0:auth0:{assignedConnectionName}` |
   | endpoint de devolución de llamada | `https://auth0.auth0.com/login/callback`   |
2. [Agregue una descripción de claim](https://learn.microsoft.com/en-us/windows-server/identity/ad-fs/operations/add-a-claim-description) para cada uno de los siguientes claims:

   | Claim                           | Identificador del claim                                                | Valor                                      |
   | ------------------------------- | ---------------------------------------------------------------------- | ------------------------------------------ |
   | Identificador de nombre         | `http://schemas.xmlsoap.org/ws/2005/05/identity/claims/nameidentifier` | `E-Mail-Addresses` o `User-Principal-Name` |
   | Dirección de correo electrónico | `http://schemas.xmlsoap.org/ws/2005/05/identity/claims/emailaddress`   | N/A                                        |
   | Nombre                          | `http://schemas.xmlsoap.org/ws/2005/05/identity/claims/name`           | N/A                                        |
3. Habilite el endpoint de SAML 2.0.
4. Proporcione al equipo de soporte de Auth0 la siguiente información:

   * endpoint de login (por ejemplo, `https://{yourServer}/adfs/ls`)
   * certificado de firma o el archivo XML de metadata de SAML

<div id="azure-ad-oidc">
  #### Azure AD (OIDC)
</div>

1. [Cree un nuevo registro de aplicación](https://learn.microsoft.com/en-us/azure/active-directory/develop/quickstart-register-app).
2. Establezca el tipo de **Redirect URI** como Web y el valor como `https://auth0.auth0.com/login/callback`.
3. Seleccione **Register**.
4. [Habilite la concesión implícita](https://learn.microsoft.com/en-us/azure/active-directory/develop/v2-oauth2-implicit-grant-flow#send-the-sign-in-request) para el token de ID.
5. [Agregue el claim de correo electrónico](https://learn.microsoft.com/en-us/azure/active-directory/develop/active-directory-optional-claims) al token de ID.
6. Proporcione al equipo de soporte de Auth0 la siguiente información:

   * El ID de la aplicación (cliente)
   * El endpoint de metadatos de OIDC (por ejemplo, `https://login. microsoftonline.com/{yourAzureAdTenantId}/v2.0/.well-known/openid-configuration`)

<div id="azure-ad-saml">
  #### Azure AD (SAML)
</div>

1. [Cree una nueva aplicación empresarial](https://learn.microsoft.com/en-us/azure/active-directory/manage-apps/add-application-portal).

2. Configure el [inicio de sesión único para SAML](https://learn.microsoft.com/en-us/azure/active-directory/develop/single-sign-on-saml-protocol) con las siguientes propiedades (es posible que deba usar valores de marcador de posición hasta que el equipo de soporte de Auth0 pueda proporcionarle el nombre de la conexión de SSO):

   | Propiedad                     | Valor                                                                |
   | ----------------------------- | -------------------------------------------------------------------- |
   | Identificador (ID de entidad) | `urn:auth0:auth0:{assignedConnectionName}`                           |
   | URL de respuesta (ACS)        | `https://auth0.auth0.com/login/callback`                             |
   | URL de inicio de sesión       | `https://manage.auth0.com/login?connection={assignedConnectionName}` |

3. Deje la sección **Attributes &amp; Claims** —incluidos `email`, `Unique User Identifier` y, opcionalmente, `name -`— sin cambios con respecto a las sugerencias de Azure.

4. Proporcione al equipo de soporte de Auth0 los datos XML de metadatos de SAML. Puede hacer una de estas dos cosas:

   * Comparta la URL de metadatos de federación de la aplicación (por ejemplo, `https://login.microsoftonline.com/{azureAdTenantId}/federationmetadata /2007-06/federationmetadata.xml?appid={appId}`).
   * Descargue el documento XML de metadatos de federación y adjúntelo al ticket.

<div id="google-saml">
  #### Google (SAML)
</div>

Auth0 permite configurar SSO para el Dashboard con un IdP de Google <Tooltip tip="Security Assertion Markup Language (SAML): protocolo estandarizado que permite a dos partes intercambiar información de autenticación sin una contraseña." cta="Ver glosario" href="/docs/es/glossary?term=SAML">SAML</Tooltip>, pero se recomienda dirigir a los usuarios para que inicien sesión con el método de autenticación de Google existente.

Cuando un usuario inicia sesión en el IdP de SAML de Google, Auth0 crea una nueva identidad de usuario (separada de su identidad de usuario de Google existente), lo que puede resultar confuso.

Si desea configurar SSO para el Dashboard con un IdP de SAML de Google, consulte [Generic IdP (SAML)](#generic-idp-saml) para obtener instrucciones.

<div id="okta-saml">
  #### Okta (SAML)
</div>

1. Crea una aplicación SAML con las siguientes propiedades (es posible que tengas que usar valores de marcador de posición hasta que el equipo de soporte de Auth0 pueda proporcionarte el nombre de la conexión SSO):

   | Propiedad                                      | Valor                                      |
   | ---------------------------------------------- | ------------------------------------------ |
   | ID de entidad                                  | `urn:auth0:auth0:{assignedConnectionName}` |
   | endpoint de devolución de llamada (URL de ACS) | `https://auth0.auth0.com/login/callback`   |

2. Configura el **Identificador de nombre** para que envíe la dirección de correo electrónico del usuario.

3. Proporciona al equipo de soporte de Auth0 los metadatos XML de SAML. Puedes hacer una de estas dos cosas:

   * Compartir la URL de los metadatos XML de SAML:

     1. Localiza la sección **Certificados de firma SAML**.
     2. Selecciona el menú **Actions**.
     3. Selecciona **Ver metadatos del IdP** y, a continuación, **Copiar dirección del enlace**. Tendrá este formato: `https://{org}.okta.com/app/{appId}/sso/saml/metadata`.
   * Descarga el archivo de metadatos XML de SAML y adjúntalo al ticket.

<div id="idp-initiated-authentication-flows">
  ##### flujos de autenticación iniciados por el IdP
</div>

El SSO para el Dashboard no admite flujos de autenticación iniciados por el IdP. Si quieres que los usuarios puedan seleccionar un icono para iniciar sesión en el Dashboard, debes hacer lo siguiente:

1. Oculta la aplicación SAML para los usuarios.
2. [Crea una Bookmark App](https://help.okta.com/en/prod/Content/Topics/Apps/Apps_Bookmark_App.htm) que apunte a `https://manage.auth0.com/login?connection={assignedConnectionName}`. Esta es la aplicación que los usuarios podrán seleccionar para iniciar sesión.

Asegúrate de habilitar tanto la aplicación SAML como la Bookmark App para el mismo grupo de usuarios que pueden usar la aplicación.

<div id="onelogin-saml">
  #### OneLogin (SAML)
</div>

1. Cree un Conector de prueba de SAML (SP) con las siguientes propiedades (es posible que deba usar marcador de posición hasta que el equipo de soporte de Auth0 pueda proporcionarle el nombre de la conexión de SSO):

   | Propiedad                                      | Valor                                                                |
   | ---------------------------------------------- | -------------------------------------------------------------------- |
   | ID de entidad                                  | `urn:auth0:auth0:{assignedConnectionName}`                           |
   | endpoint de devolución de llamada (URL de ACS) | `https://auth0.auth0.com/login/callback`                             |
   | URL de inicio de sesión                        | `https://manage.auth0.com/login?connection={assignedConnectionName}` |
2. Proporcione al equipo de soporte de Auth0 el archivo XML de metadatos de SAML.

<div id="generic-idp-oidc">
  #### IdP genérico (OIDC)
</div>

1. Registra una aplicación (cliente) en el IdP con las siguientes propiedades:

   | Propiedad                    | Valor                                    |
   | ---------------------------- | ---------------------------------------- |
   | URL de devolución de llamada | `https://auth0.auth0.com/login/callback` |

2. [Agrega el claim de correo electrónico](/docs/es/secure/tokens/json-web-tokens/create-custom-claims) al token de ID.

3. Proporciona al equipo de soporte de Auth0 lo siguiente:

   * ID de la aplicación (cliente)
   * URL del emisor o endpoint de metadata de OIDC (por ejemplo, `https://{idpDomain}/[...]/.well-known/openid-configuration`)

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  El equipo de soporte de Auth0 usará el [Modo Implicit con flujo Form Post](/docs/es/get-started/authentication-and-authorization-flow/implicit-flow-with-form-post) si está disponible. Si tu IdP no lo admite, tendrás que proporcionarles el secreto de cliente de tu aplicación.
</Callout>

<div id="generic-idp-saml">
  #### IdP genérico (SAML)
</div>

1. El equipo de soporte de Auth0 le proporcionará el nombre de la conexión SSO.
2. Cree una aplicación SAML con las siguientes propiedades:

   | Propiedad                         | Valor                                      |
   | --------------------------------- | ------------------------------------------ |
   | ID de entidad                     | `urn:auth0:auth0:{assignedConnectionName}` |
   | endpoint de devolución de llamada | `https://auth0.auth0.com/login/callback`   |
3. Asegúrese de que las aserciones SAML contengan los siguientes claims:

   | Claim                           | Identificador del claim                                                | Valor                  |
   | ------------------------------- | ---------------------------------------------------------------------- | ---------------------- |
   | Identificador de nombre         | `http://schemas.xmlsoap.org/ws/2005/05/identity/claims/nameidentifier` | `upn` o `emailaddress` |
   | Dirección de correo electrónico | `http://schemas.xmlsoap.org/ws/2005/05/identity/claims/emailaddress`   | N/A                    |
   | Nombre                          | `http://schemas.xmlsoap.org/ws/2005/05/identity/claims/name`           | N/AA                   |
4. Proporcione al equipo de soporte de Auth0 alguno de los siguientes elementos:

   * URL de inicio de sesión y certificado de firma
   * archivo XML de metadatos SAML

<div id="set-up-sso-connection">
  ### Configura la conexión SSO
</div>

El equipo de soporte de Auth0 usa los datos de configuración que proporcionas para completar la configuración inicial de la conexión SSO.

Home Realm Discovery (HRD) no se configura durante la configuración inicial.

<div id="test-sso-connection">
  ### Probar la conexión SSO
</div>

Cuando se completa la configuración inicial de la conexión SSO, el equipo de soporte de Auth0 le pedirá que la pruebe para asegurarse de que los datos de configuración sean correctos y de que los miembros del tenant puedan autenticarse con la conexión SSO de la manera que necesita.

El equipo de soporte de Auth0 le proporcionará una URL de inicio de sesión directo que puede usar para iniciar la autenticación de la nueva conexión SSO. Por ejemplo:

`https://manage.auth0.com/login?connection={assignedConnectionName}`

<div id="enforce-sso-with-auth0-teams">
  ## Exige SSO con Auth0 Teams
</div>

Si usas Auth0 Teams con tu tenant empresarial, puedes exigir la autenticación SSO para los tenants que pertenecen a la cuenta de Teams. Para obtener más información sobre la administración y la gestión de tenants, consulta [Auth0 Teams](/docs/es/get-started/auth0-teams).

1. Abre un navegador nuevo e introduce la cuenta y el identificador de tu Teams: `https://accounts.auth0.com/teams/{team-identifier}.`

   <Callout icon="file-lines" color="#0EA5E9" iconType="regular">
     Puedes obtener el identificador de Teams en la URL o en la [**Configuración**](https://accounts.auth0.com/teams/#/settings) de Teams.
   </Callout>

2. Ve a la página **Seguridad**.

3. [Configura las Políticas de seguridad](/docs/es/get-started/auth0-teams/configure-security-policies) seleccionando **Exigir inicio de sesión único**.

<div id="enable-home-realm-discovery-optional">
  ### Habilitar Home Realm Discovery (opcional)
</div>

Si utilizas Universal Login o Classic Login, puedes solicitar que se habilite HRD para que la página de inicio de sesión detecte el dominio de la dirección de correo electrónico que introduce el miembro del tenant y, después, lo redirija a la nueva conexión de SSO.

<div id="when-to-enable-hrd">
  #### Cuándo habilitar HRD
</div>

Si HRD está habilitado, los miembros del tenant que antes usaban una identidad de correo electrónico/contraseña (con una dirección de correo electrónico que coincida con el dominio de HRD configurado) no podrán iniciar sesión a través de la página de inicio de sesión, y HRD solo es compatible con identificadores de correo electrónico.

Debido a este cambio de comportamiento, solicita habilitar HRD solo después de que al menos algunos de los miembros del tenant actuales estén al tanto del cambio y sepan que:

* Recibirán una invitación para unirse a los tenants con la nueva identidad
* Tendrán que volver a invitarse a sí mismos con la nueva identidad

<div id="how-to-bypass-hrd">
  #### Cómo eludir HRD
</div>

Si un Tenant Member necesita iniciar sesión con su identidad de correo electrónico y contraseña, puedes proporcionarle una URL de inicio de sesión directa:

`https://manage.auth0.com/login?connection=auth0`

Esta URL elude HRD y les permite iniciar sesión con su identidad de correo electrónico y contraseña.

<div id="example-of-hrd-login-behavior">
  #### Ejemplo del comportamiento del inicio de sesión con HRD
</div>

Este es un ejemplo de una lista de miembros del tenant:

| Tenant            | miembro del tenant       | Conexión       | ¿Afectado? |
| ----------------- | ------------------- | -------------- | ---------- |
| `fabrikam@us`     | `user1@example.com` | email/password | Sí         |
| `fabrikam@us`     | `user1@gmail.com`   | google-oauth2  | No         |
| `fabrikam@us`     | `user2@example.com` | github         | No         |
| `fabrikam@us`     | `user3@acme.com`    | email/password | No         |
| `fabrikam@us`     | `user4@example.com` | email/password | Sí         |
| `fabrikam-dev@us` | `user5@example.com` | email/password | Sí         |
| `fabrikam-dev@us` | `user1@example.com` | email/password | Sí         |

Si asociamos el dominio example.com a la nueva conexión, los miembros del tenant `user1@gmail.com`, `user2@example.com` y `user3@acme.com` podrán iniciar sesión como lo hacían antes, ya que usan un proveedor social o una dirección de correo electrónico con un dominio no asociado.

En cambio, los miembros del tenant `user1@example.com`, `user4@example.com` y `user5@example.com` no podrán iniciar sesión como lo hacían antes, porque sus direcciones de correo electrónico están asociadas al dominio configurado para HRD.

<div id="migrate-existing-tenant-members">
  ## Migrar miembros del tenant existentes
</div>

El proceso para migrar miembros del tenant existentes depende de si tienes habilitado HRD.

<div id="how-to-migrate-with-hrd-disabled">
  ### Cómo migrar con HRD deshabilitado
</div>

Para migrar miembros del tenant con HRD deshabilitado, debes compartir la URL de inicio de sesión directo de la nueva conexión SSO:

`https://manage.auth0.com/login?connection={assignedConnectionName}`

1. [Crea una nueva invitación para un miembro del tenant](/docs/es/get-started/manage-dashboard-access/add-dashboard-users) para el miembro del tenant.
2. Indica al miembro del tenant que:

   1. Inicie sesión en la nueva conexión SSO usando la URL de inicio de sesión directo antes de aceptar la invitación.

      <Callout icon="file-lines" color="#0EA5E9" iconType="regular">
        Si es la primera vez que inicia sesión en la nueva conexión SSO, es posible que vea una página de enriquecimiento del perfil (en https://auth.com/profile). No complete ningún campo ni seleccione **Next**. En su lugar, continúe con el siguiente paso.
      </Callout>

   2. Copie y pegue la URL de la invitación que recibió en el correo electrónico de invitación en el mismo navegador donde inició sesión en la nueva conexión SSO. Los usuarios no deben seleccionar **Create Account**.

   3. Acepte la invitación.

   4. Si el usuario tiene invitaciones para otros tenants, puede usarlas en este momento.

<div id="how-to-migrate-with-hrd-enabled">
  ### Cómo migrar con HRD habilitado
</div>

Para migrar miembros del tenant con HRD habilitado, debes seguir pasos similares a [agregar miembros del tenant](/docs/es/get-started/manage-dashboard-access/add-dashboard-users):

1. [Crear una nueva invitación para un miembro del tenant](/docs/es/get-started/manage-dashboard-access/add-dashboard-users).
2. Indica al miembro del tenant que:

   1. Cierre sesión en el Dashboard (si anteriormente había iniciado sesión con una identidad antigua).

      <Callout icon="file-lines" color="#0EA5E9" iconType="regular">
        Si esta es la primera invitación que acepta, es posible que vea una página para “Crear un nuevo tenant” o “Crear una nueva cuenta”. No seleccione **Next**. En su lugar, siga el paso siguiente.
      </Callout>

   2. Abra el enlace de invitación en el correo electrónico que recibió.

   3. Inicie sesión en la nueva conexión.

   4. Acepte la invitación.
