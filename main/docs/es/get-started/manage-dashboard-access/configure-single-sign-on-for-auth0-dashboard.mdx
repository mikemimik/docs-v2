<Card title="Antes de comenzar">
  * Suscríbase a un plan Enterprise. Para obtener más información, consulte [Manage Subscriptions](/docs/es/troubleshoot/customer-support/manage-subscriptions).
  * Determine si usa Public Cloud o Private Cloud. Para conocer los requisitos previos relacionados con Private Cloud para estas instrucciones, consulte [Deploy Private Cloud](/docs/es/deploy-monitor/deploy-private-cloud).
</Card>

Puede configurar Auth0 para permitir que los [miembros del tenant](/docs/es/get-started/manage-dashboard-access) usen su propio [proveedor de identidad empresarial](/docs/es/authenticate/identity-providers/enterprise-identity-providers) (<Tooltip tip="Identity Provider (IdP): Servicio que almacena y administra identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=IdP">IdP</Tooltip>) para autenticarse en <Tooltip tip="Auth0 Dashboard: El producto principal de Auth0 para configurar sus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip> mediante <Tooltip tip="Auth0 Dashboard: El producto principal de Auth0 para configurar sus servicios." cta="Ver glosario" href="/docs/es/glossary?term=single+sign-on">inicio de sesión único</Tooltip> (SSO).

<div id="how-it-works">
  ## Cómo funciona
</div>

Para configurar el SSO para el Dashboard, debes trabajar con el equipo de soporte de Auth0 para agregar una conexión empresarial a la autoridad del tenant raíz (RTA), que administra los métodos de autenticación disponibles que un miembro del tenant puede usar para iniciar sesión en el Dashboard.

Agregar esta conexión SSO no impide que los miembros del tenant inicien sesión con los métodos de autenticación existentes (como correo electrónico/contraseña, [LinkedIn](https://marketplace.auth0.com/integrations/linkedin-social-connection), [Microsoft](https://marketplace.auth0.com/integrations/microsoft-account-social-connection), [GitHub](https://marketplace.auth0.com/integrations/github-social-connection) o [Google](https://marketplace.auth0.com/integrations/google-social-connection)).

Configurar el SSO para el Dashboard también habilita el SSO en todos los sitios públicos de Auth0, como:

* sitio web de Auth0 (https://auth0.com)
* Auth0 Community (https://community.auth0.com)
* Auth0 Docs (https://www.auth0.com/docs)
* Auth0 Support Center (https://support.auth0.com)

<div id="user-experience">
  ### Experiencia de usuario
</div>

Cuando un usuario autorizado inicia sesión en el Dashboard, introduce su dirección de correo electrónico de un dominio registrado (por ejemplo, `user@example.com`) en la página de Auth0 <Tooltip tip="Universal Login: Tu aplicación redirige a Universal Login, alojado en el servidor de autorización de Auth0, para verificar la identidad del usuario." cta="Ver glosario" href="/docs/es/glossary?term=Universal+Login">Universal Login</Tooltip> y luego se le redirige a su IdP para completar la autenticación.

<div id="limitations">
  ### Limitaciones
</div>

Antes de decidir configurar SSO para el Dashboard, ten en cuenta estas restricciones:

* SSO no se puede limitar a tenants específicos.
* SSO no admite flujos de autenticación iniciados por el IdP.
* Las invitaciones de miembro del tenant no se pueden automatizar ni enviar de forma masiva con la <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip> de Auth0.
* El acceso de miembro del tenant no se puede administrar mediante la pertenencia a grupos de tu IdP.
* No se puede exigir <Tooltip tip="Multi-factor authentication (MFA): Proceso de autenticación de usuario que utiliza un factor además de username and password, como un código por SMS." cta="Ver glosario" href="/docs/es/glossary?term=MFA">MFA</Tooltip> para todos los miembros de un tenant.

<div id="considerations">
  ### Consideraciones
</div>

<div id="full-directory-access-to-dashboard">
  #### Acceso completo del directorio al Dashboard
</div>

Cuando agregas tu IdP como una conexión disponible para que los miembros del tenant inicien sesión, todos los usuarios del directorio de tu IdP pueden acceder al Dashboard, pero solo los miembros del tenant que hayan sido invitados a un tenant específico pueden acceder a ese tenant.

Cuando los usuarios intentan acceder a un tenant en el Dashboard sin una invitación, el sistema deniega la operación. Si los usuarios no pertenecen a ningún tenant, el sistema les pide que [completen su perfil de usuario](#complete-user-profiling) y creen un nuevo tenant de prueba. Crear un nuevo tenant de prueba no lo asocia con tu Enterprise plan.

<div id="residual-tenant-member-identities">
  #### Identidades residuales de miembro del tenant
</div>

Si se invitó a un miembro del tenant (y se le dio acceso) a un tenant en el Dashboard con una identidad distinta de la creada en la nueva conexión, técnicamente podrá seguir usando esa identidad para acceder al tenant.

Tendrás que decidir si quieres eliminar su identidad anterior o conservarla como posible método de autenticación alternativo.

<div id="configure-sso-for-the-dashboard">
  ## Configurar SSO para el Dashboard
</div>

La configuración de SSO para el Dashboard requiere una serie de pasos que deben llevarse a cabo conjuntamente entre usted y el equipo de soporte de Auth0.

<div id="share-idp-configuration-data">
  ### Compartir datos de configuración del IdP
</div>

[Abre un ticket con el equipo de soporte de Auth0](/docs/es/troubleshoot/customer-support/open-and-manage-support-tickets) para compartir los datos de configuración de tu IdP, para que puedan configurar el SSO. Incluye la siguiente información cuando envíes tu ticket:

* Dominio(s) de correo electrónico que quieras asociar con la configuración de SSO
* Nombre del IdP
* Protocolo de autenticación
* Información adicional específica del IdP

Según el IdP y el protocolo de autenticación que quieras usar, se requieren pasos de configuración adicionales:

<div id="adfs-saml">
  #### ADFS (SAML)
</div>

1. [Cree una Relying Party Trust](https://learn.microsoft.com/en-us/windows-server/identity/ad-fs/operations/create-a-relying-party-trust#to-create-a-claims-aware-relying-party-trust-manually) con las siguientes propiedades:

   | Propiedad                         | Valores                                    |
   | --------------------------------- | ------------------------------------------ |
   | ID de entidad                     | `urn:auth0:auth0:{assignedConnectionName}` |
   | endpoint de devolución de llamada | `https://auth0.auth0.com/login/callback`   |
2. [Agregue una descripción de claim](https://learn.microsoft.com/en-us/windows-server/identity/ad-fs/operations/add-a-claim-description) para cada uno de los siguientes claims:

   | Claim                           | Identificador de claim                                                 | Valor                                      |
   | ------------------------------- | ---------------------------------------------------------------------- | ------------------------------------------ |
   | Identificador de nombre         | `http://schemas.xmlsoap.org/ws/2005/05/identity/claims/nameidentifier` | `E-Mail-Addresses` o `User-Principal-Name` |
   | Dirección de correo electrónico | `http://schemas.xmlsoap.org/ws/2005/05/identity/claims/emailaddress`   | N/A                                        |
   | Nombre                          | `http://schemas.xmlsoap.org/ws/2005/05/identity/claims/name`           | N/A                                        |
3. Habilite el endpoint de SAML 2.0.
4. Proporcione al equipo de soporte de Auth0 la siguiente información:

   * endpoint de inicio de sesión (por ejemplo, `https://{yourServer}/adfs/ls`)
   * certificado de firma o el archivo XML de metadata de SAML

<div id="azure-ad-oidc">
  #### Azure AD (OIDC)
</div>

1. [Cree un nuevo registro de aplicación](https://learn.microsoft.com/en-us/azure/active-directory/develop/quickstart-register-app).
2. Establezca el tipo de **Redirect URI** en Web y el valor en `https://auth0.auth0.com/login/callback`.
3. Seleccione **Register**.
4. [Habilite la concesión implícita](https://learn.microsoft.com/en-us/azure/active-directory/develop/v2-oauth2-implicit-grant-flow#send-the-sign-in-request) para el token de ID.
5. [Agregue la claim de correo electrónico](https://learn.microsoft.com/en-us/azure/active-directory/develop/active-directory-optional-claims) al token de ID.
6. Proporcione al equipo de soporte de Auth0 la siguiente información:

   * El ID de la aplicación (cliente)
   * El endpoint de metadatos de OIDC (por ejemplo, `https://login. microsoftonline.com/{yourAzureAdTenantId}/v2.0/.well-known/openid-configuration`)

<div id="azure-ad-saml">
  #### Azure AD (SAML)
</div>

1. [Cree una nueva aplicación empresarial](https://learn.microsoft.com/en-us/azure/active-directory/manage-apps/add-application-portal).

2. Configure el [inicio de sesión único para SAML](https://learn.microsoft.com/en-us/azure/active-directory/develop/single-sign-on-saml-protocol) con las siguientes propiedades (es posible que deba usar valores provisionales hasta que el equipo de soporte de Auth0 pueda proporcionarle el nombre de la conexión SSO):

   | Propiedad                     | Valor                                                                |
   | ----------------------------- | -------------------------------------------------------------------- |
   | Identificador (ID de entidad) | `urn:auth0:auth0:{assignedConnectionName}`                           |
   | URL de respuesta (ACS)        | `https://auth0.auth0.com/login/callback`                             |
   | URL de inicio de sesión       | `https://manage.auth0.com/login?connection={assignedConnectionName}` |

3. Deje la sección **Attributes &amp; Claims** —incluidos `email`, `Unique User Identifier` y, opcionalmente, `name -`— sin cambios con respecto a las sugerencias de Azure.

4. Proporcione al equipo de soporte de Auth0 los datos XML de metadatos SAML. Puede hacer una de estas dos cosas:

   * Compartir la URL de metadatos de federación de la aplicación (por ejemplo, `https://login.microsoftonline.com/{azureAdTenantId}/federationmetadata /2007-06/federationmetadata.xml?appid={appId}`).
   * Descargar el documento XML de metadatos de federación y adjuntarlo al ticket.

<div id="google-saml">
  #### Google (SAML)
</div>

Auth0 admite configurar SSO para el Dashboard con un Google <Tooltip tip="Security Assertion Markup Language (SAML): Protocolo estandarizado que permite a dos partes intercambiar información de autenticación sin contraseña." cta="Ver glosario" href="/docs/es/glossary?term=SAML">SAML</Tooltip> IdP, pero se recomienda dirigir a los usuarios a iniciar sesión con el método de autenticación de Google existente.

Cuando un usuario inicia sesión en el IdP de SAML de Google, Auth0 crea una nueva identidad de usuario para ese usuario (separada de su identidad de usuario de Google existente), lo que puede resultar confuso.

Si desea configurar SSO para el Dashboard con un IdP de SAML de Google, consulte [Generic IdP (SAML)](#generic-idp-saml) para ver las instrucciones.

<div id="okta-saml">
  #### Okta (SAML)
</div>

1. Cree una aplicación SAML con las siguientes propiedades (es posible que tenga que usar valores de marcador de posición hasta que el equipo de soporte de Auth0 pueda proporcionarle el nombre de la conexión SSO):

   | Propiedad                                      | Valor                                      |
   | ---------------------------------------------- | ------------------------------------------ |
   | ID de entidad                                  | `urn:auth0:auth0:{assignedConnectionName}` |
   | Endpoint de devolución de llamada (URL de ACS) | `https://auth0.auth0.com/login/callback`   |

2. Configure el **Identificador de nombre** para que envíe la dirección de correo electrónico del usuario.

3. Proporcione al equipo de soporte de Auth0 los datos XML de metadatos SAML. Puede optar por una de estas opciones:

   * Compartir la URL de los metadatos XML de SAML:

     1. Localice la sección **SAML Signing Certificates**.
     2. Seleccione el menú **Actions**.
     3. Seleccione **View IdP metadata** y, a continuación, seleccione **Copy Link Address**. Tendrá este formato: `https://{org}.okta.com/app/{appId}/sso/saml/metadata`.
   * Descargue el archivo XML de metadatos SAML y adjúntelo al ticket.

<div id="idp-initiated-authentication-flows">
  ##### flujos de autenticación iniciados por el IdP
</div>

El SSO para el Dashboard no admite flujos de autenticación iniciados por el IdP. Si quiere que los usuarios puedan seleccionar un icono para iniciar sesión en el Dashboard, debe:

1. Ocultar la aplicación SAML a los usuarios.
2. [Crear una Bookmark App](https://help.okta.com/en/prod/Content/Topics/Apps/Apps_Bookmark_App.htm) que apunte a `https://manage.auth0.com/login?connection={assignedConnectionName}`. Esta es la aplicación que los usuarios podrán seleccionar para iniciar sesión.

Asegúrese de habilitar tanto la aplicación SAML como la Bookmark App para el mismo grupo de usuarios que pueden usar la aplicación.

<div id="onelogin-saml">
  #### OneLogin (SAML)
</div>

1. Cree un Conector de prueba de SAML (SP) con las siguientes propiedades (puede que deba usar valores de marcador de posición hasta que el equipo de soporte de Auth0 pueda proporcionarle el nombre de la conexión SSO):

   | Propiedad                                   | Valor                                                                |
   | ------------------------------------------- | -------------------------------------------------------------------- |
   | ID de entidad                               | `urn:auth0:auth0:{assignedConnectionName}`                           |
   | endpoint de devolución de llamada (URL ACS) | `https://auth0.auth0.com/login/callback`                             |
   | URL de inicio de sesión                     | `https://manage.auth0.com/login?connection={assignedConnectionName}` |
2. Proporcione al equipo de soporte de Auth0 el archivo XML de metadatos de SAML.

<div id="generic-idp-oidc">
  #### IdP genérico (OIDC)
</div>

1. Registre una aplicación (cliente) en el IdP con las siguientes propiedades:

   | Propiedad                    | Valor                                    |
   | ---------------------------- | ---------------------------------------- |
   | URL de devolución de llamada | `https://auth0.auth0.com/login/callback` |

2. [Agregue el claim de correo electrónico](/docs/es/secure/tokens/json-web-tokens/create-custom-claims) al token de ID.

3. Proporcione al equipo de soporte de Auth0 lo siguiente:

   * ID de la aplicación (cliente)
   * URL del emisor o endpoint de metadatos de OIDC (por ejemplo, `https://{idpDomain}/[...]/.well-known/openid-configuration`)

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  El equipo de soporte de Auth0 usará el [modo Implicit con flujo Form Post](/docs/es/get-started/authentication-and-authorization-flow/implicit-flow-with-form-post), si está disponible. Si su IdP no es compatible con esta opción, deberá proporcionarles el secreto de cliente de su aplicación.
</Callout>

<div id="generic-idp-saml">
  #### IdP genérico (SAML)
</div>

1. El equipo de soporte de Auth0 le proporcionará el nombre de la conexión de SSO.
2. Cree una aplicación SAML con las siguientes propiedades:

   | Propiedad                         | Valor                                      |
   | --------------------------------- | ------------------------------------------ |
   | ID de entidad                     | `urn:auth0:auth0:{assignedConnectionName}` |
   | Endpoint de devolución de llamada | `https://auth0.auth0.com/login/callback`   |
3. Asegúrese de que las aserciones SAML contengan los siguientes claims:

   | Claim                           | Identificador del claim                                                | Valor                  |
   | ------------------------------- | ---------------------------------------------------------------------- | ---------------------- |
   | Identificador de nombre         | `http://schemas.xmlsoap.org/ws/2005/05/identity/claims/nameidentifier` | `upn` o `emailaddress` |
   | Dirección de correo electrónico | `http://schemas.xmlsoap.org/ws/2005/05/identity/claims/emailaddress`   | N/A                    |
   | Nombre                          | `http://schemas.xmlsoap.org/ws/2005/05/identity/claims/name`           | N/AA                   |
4. Proporcione al equipo de soporte de Auth0 uno de los siguientes:

   * URL de inicio de sesión y certificado de firma
   * Archivo XML de metadatos SAML

<div id="set-up-sso-connection">
  ### Configura la conexión SSO
</div>

El equipo de soporte de Auth0 usa los datos de configuración que proporcionas para completar la configuración inicial de la conexión SSO.

Home Realm Discovery (HRD) no se configura durante la configuración inicial.

<div id="test-sso-connection">
  ### Probar la conexión SSO
</div>

Cuando se completa la configuración inicial de la conexión SSO, el equipo de soporte de Auth0 te pide que la pruebes para asegurarse de que los datos de configuración sean correctos y de que los Miembros del tenant puedan autenticarse con la conexión SSO de la forma que necesitas.

El equipo de soporte de Auth0 te proporciona una URL de inicio de sesión directa que puedes usar para iniciar la autenticación con la nueva conexión SSO. Por ejemplo:

`https://manage.auth0.com/login?connection={assignedConnectionName}`

<div id="enforce-sso-with-auth0-teams">
  ## Exigir SSO con Auth0 Teams
</div>

Si usas Auth0 Teams con tu tenant empresarial, puedes exigir la autenticación mediante SSO para los tenants que pertenecen a la cuenta de Teams. Para obtener más información sobre la administración y la gestión de tenants, consulta [Auth0 Teams](/docs/es/get-started/auth0-teams).

1. Abre un navegador nuevo e introduce tu cuenta de Teams y su identificador: `https://accounts.auth0.com/teams/{team-identifier}.`

   <Callout icon="file-lines" color="#0EA5E9" iconType="regular">
     Puedes obtener el identificador de Teams en la URL o en la página [**Settings**](https://accounts.auth0.com/teams/#/settings) de Teams.
   </Callout>

2. Ve a la página **Seguridad**.

3. [Configura las Políticas de seguridad](/docs/es/get-started/auth0-teams/configure-security-policies) seleccionando **Aplicar inicio de sesión único**.

<div id="enable-home-realm-discovery-optional">
  ### Habilitar Home Realm Discovery (opcional)
</div>

Si usas Universal Login o Classic Login, puedes solicitar que se habilite HRD para que la página de inicio de sesión reconozca el dominio de la dirección de correo electrónico que introduce el miembro del tenant y, a continuación, lo redirija a la nueva conexión SSO.

<div id="when-to-enable-hrd">
  #### Cuándo habilitar HRD
</div>

Si HRD está habilitado, los miembros del tenant que antes usaban una identidad de correo electrónico/contraseña (con una dirección de correo electrónico que coincida con el dominio de HRD configurado) no podrán iniciar sesión a través de la página de inicio de sesión, y HRD solo es compatible con identificadores de correo electrónico.

Debido a este cambio de comportamiento, solicita habilitar HRD solo después de que al menos algunos de los miembros del tenant actuales conozcan el cambio y sepan que:

* Recibirán una invitación para unirse a los tenants con la nueva identidad
* Tendrán que volver a invitarse a sí mismos con la nueva identidad

<div id="how-to-bypass-hrd">
  #### Cómo evitar HRD
</div>

Si un miembro del tenant necesita iniciar sesión con su identidad de correo electrónico/contraseña, puedes proporcionarle una URL de inicio de sesión directa:

`https://manage.auth0.com/login?connection=auth0`

Esta URL evita HRD y le permite iniciar sesión con su identidad de correo electrónico y contraseña.

<div id="example-of-hrd-login-behavior">
  #### Ejemplo del comportamiento de inicio de sesión de HRD
</div>

Aquí tienes un ejemplo de una lista de miembros del tenant:

| Tenant            | miembro del tenant       | Conexión       | ¿Afectado? |
| ----------------- | ------------------- | -------------- | ---------- |
| `fabrikam@us`     | `user1@example.com` | email/password | Sí         |
| `fabrikam@us`     | `user1@gmail.com`   | google-oauth2  | No         |
| `fabrikam@us`     | `user2@example.com` | github         | No         |
| `fabrikam@us`     | `user3@acme.com`    | email/password | No         |
| `fabrikam@us`     | `user4@example.com` | email/password | Sí         |
| `fabrikam-dev@us` | `user5@example.com` | email/password | Sí         |
| `fabrikam-dev@us` | `user1@example.com` | email/password | Sí         |

Si asociamos el dominio example.com a la nueva conexión, los miembros del tenant `user1@gmail.com`, `user2@example.com` y `user3@acme.com` pueden iniciar sesión igual que antes, porque usan un proveedor social o un correo electrónico con un dominio no asociado.

Por el contrario, los miembros del tenant `user1@example.com`, `user4@example.com` y `user5@example.com` no pueden iniciar sesión igual que antes, porque sus correos electrónicos están asociados al dominio configurado para HRD.

<div id="migrate-existing-tenant-members">
  ## Migrar miembros del tenant existentes
</div>

El proceso de migración de los miembros del tenant existentes depende de si has habilitado HRD.

<div id="how-to-migrate-with-hrd-disabled">
  ### Cómo migrar con HRD deshabilitado
</div>

Para migrar miembros del tenant con HRD deshabilitado, debes compartir la URL de inicio de sesión directa de la nueva conexión SSO:

`https://manage.auth0.com/login?connection={assignedConnectionName}`

1. [Crea una nueva invitación de miembro del tenant](/docs/es/get-started/manage-dashboard-access/add-dashboard-users) para el miembro del tenant.
2. Indica al miembro del tenant que:

   1. Inicie sesión en la nueva conexión SSO con la URL de inicio de sesión directa antes de aceptar la invitación.

      <Callout icon="file-lines" color="#0EA5E9" iconType="regular">
        Si es la primera vez que inicia sesión en la nueva conexión SSO, es posible que vea una página de enriquecimiento de perfil (en https://auth.com/profile). No complete ningún campo ni seleccione **Siguiente**. En su lugar, continúe con el siguiente paso.
      </Callout>

   2. Copie y pegue en el mismo navegador en el que inició sesión en la nueva conexión SSO la URL de invitación que recibió por correo electrónico. Los usuarios no deben seleccionar **Crear cuenta**.

   3. Acepte la invitación.

   4. Si el usuario tiene invitaciones para otros tenants, puede usarlas en este momento.

<div id="how-to-migrate-with-hrd-enabled">
  ### Cómo migrar con HRD habilitado
</div>

Para migrar Miembros del tenant con HRD habilitado, debes seguir pasos similares a los de [agregar Miembros del tenant](/docs/es/get-started/manage-dashboard-access/add-dashboard-users):

1. [Crea una nueva invitación para un Miembro del tenant](/docs/es/get-started/manage-dashboard-access/add-dashboard-users).
2. Indica al Miembro del tenant que:

   1. Cierre sesión en el Dashboard (si ya había iniciado sesión con una identidad anterior).

      <Callout icon="file-lines" color="#0EA5E9" iconType="regular">
        Si esta es la primera invitación que acepta, es posible que vea una página de “Crear un nuevo tenant” o “Crear una nueva cuenta”. No seleccione **Next**. En su lugar, continúe con el siguiente paso.
      </Callout>

   2. Abra el enlace de invitación en el correo electrónico que recibió.

   3. Inicie sesión con la nueva conexión.

   4. Acepte la invitación.
