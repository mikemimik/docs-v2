Si tu caso de uso da servicio a una <Tooltip tip="Audiencia: identificador único de la audiencia de un token emitido. Se denomina aud en un token, y su valor contiene el ID de una aplicación (ID de cliente) para un token de ID o de una API (identificador de API) para un token de acceso." cta="Ver glosario" href="/docs/es/glossary?term=audience">audiencia</Tooltip> no interactiva, como las API o los servidores backend, la incorporación se realizará con una configuración de máquina a máquina (M2M).

<div id="use-cases">
  ## Casos de uso
</div>

Usa el proceso de incorporación de máquina a máquina si:

* Admites comunicaciones de servicio a servicio
* Tienes tareas programadas o tareas cron ejecutándose en servidores que necesitan acceder a recursos protegidos o APIs
* Permites que dispositivos IoT se comuniquen con servicios de backend o APIs
* Tienes una capa de API que necesita comunicarse con otras capas de API sin intervención del usuario o después de que haya expirado el token de un usuario
* Tienes una API privilegiada que puede necesitar llamarse antes de que un usuario se haya autenticado (es decir, desde una Action o un script de base de datos personalizada en tu tenant de Auth0)
* Usas API Gateway para gestionar servicios de backend
* Usas o admites aplicaciones no interactivas u otras herramientas que no implican interacción humana, como demonios o servicios de backend

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Estos servicios seguirán necesitando un token de acceso de máquina a máquina para la autenticación.
</Callout>

<div id="architecture">
  ## Arquitectura
</div>

Antes de configurar tu cuenta y tenant de Auth0, o los grupos y estructuras de tus servicios de Auth0, crea un mapa de tu infraestructura actual para aprovechar al máximo las capacidades de Auth0 dentro de tu ecosistema existente.

Como se menciona en el apartado Escenario común, es posible que tengas otras tecnologías no interactivas en el dominio de tu aplicación, en el dominio de red o en el dominio de dispositivos máquina a máquina que debas tener en cuenta antes de configurar Auth0. Para ver un ejemplo de un escenario máquina a máquina, lee [Server + API](/docs/es/get-started/architecture-scenarios/server-application-api/part-1). Si quieres realizar un laboratorio práctico con Node y probar el despliegue de una API, visita nuestro [repositorio de GitHub](https://www.google.com/url?q=https://github.com/auth0-training/labs-node-working-with-apis\&sa=D\&source=docs\&ust=1723578190947730\&usg=AOvVaw3IXRQG9ogH4pW6LEY047T6).

También puedes crear una visualización de tu stack tecnológico actual, además de planificar cómo encaja Auth0 en tu ciclo de vida actual de desarrollo de software (SDLC). Esto puede ayudarte a determinar cuántos tenants podrías necesitar.

<div id="considerations">
  ### Consideraciones
</div>

Antes de crear una cuenta nueva o configurar tu primer tenant, quizá te convenga tener en cuenta lo siguiente:

* Cómo particionas o agrupas tus API para llamar a endpoints específicos.

  * Esto puede determinar la audience y otros claims de los tokens de acceso.
* Cualquier consumidor de terceros de tu recurso puede solicitar un token de acceso para cada call. Un volumen excesivo de calls podría afectar tu [límite de frecuencia](/docs/es/troubleshoot/customer-support/operational-policies/rate-limit-policy/rate-limit-configurations/essentials-professional-b2b).

  * Puedes usar API Gateway para limitar la cantidad de tokens de acceso que un tercero puede solicitar. Para obtener más información sobre API Gateway y Auth0, consulta [Configurar un proveedor de identidad en Access Gateway](/docs/es/get-started/auth0-overview/create-applications/configure-an-identity-provider-in-access-gateway).

<div id="create-an-account">
  ## Crea una cuenta
</div>

Ahora que ya tienes un plan para tu arquitectura, configurarás tu cuenta de Auth0 y tus tenants. Cuando te registres en los servicios de Auth0, crearás tu primer [tenant](/docs/es/get-started/auth0-overview/create-tenants). Aquí es donde configuras los recursos, servicios y demás elementos de Auth0. [Regístrate](https://auth0.com/signup) para empezar.

<Card title="Antes de empezar">
  En el Auth0 Dashboard o con la Auth0 Management API, crea:

  * Una API para representar tu API
  * Una aplicación de máquina a máquina para usar el flujo de credenciales del cliente

  Quizá te convenga planificar algunos detalles de configuración antes de crear una cuenta.

  * El nombre de tu tenant influye en tu dominio de Auth0. Antes de decidirlo, conviene que revises las [características del tenant](/docs/es/get-started/auth0-overview/create-tenants#create-a-tenant-and-domain).
  * Qué funcionalidades de Auth0 necesitas para tu caso de uso. Algunas funcionalidades solo están disponibles en los planes Professional y Enterprise.
  * Determina si necesitas admitir varios entornos, como desarrollo, staging y producción. Para obtener más información, consulta [Configurar múltiples entornos](/docs/es/get-started/auth0-overview/create-tenants/set-up-multiple-environments).
  * Si tienes un caso de uso que involucra aplicaciones de terceros que quieres registrar en un tenant, puedes usar el [registro dinámico de aplicaciones](/docs/es/get-started/applications/dynamic-client-registration) basado en la especificación [OIDC Client Registration](https://openid.net/specs/openid-connect-registration-1_0.html).
</Card>

<div id="authentication">
  ## Autenticación
</div>

Al llamar a una API desde otra API, o en cualquier situación en la que no exista el contexto de un usuario autenticado, necesitas una forma de autorizar a la aplicación en lugar de a un usuario. Este es un proceso de un solo paso en el que la aplicación se autentica (mediante un `client_id` y un `client_secret`) y luego se autoriza en una sola llamada.

Para la autenticación de aplicaciones o servicios no interactivos, debes seleccionar una autorización de cliente o un flujo de autenticación. El <Tooltip tip="OAuth 2.0: Marco de autorización que define protocolos y flujos de trabajo de autorización." cta="Ver glosario" href="/docs/es/glossary?term=OAuth+2.0">OAuth 2.0</Tooltip> [flujo de credenciales del cliente](https://tools.ietf.org/html/rfc6749#section-4.4) no requiere interacción humana y es el más adecuado para aplicaciones de máquina a máquina.

<Card title="Antes de empezar">
  En Auth0 Dashboard o Management API, harás lo siguiente:

  * Configurar tu aplicación para usar flujo de credenciales del cliente
  * Actualizar los scopes de tus tokens de acceso de máquina a máquina

  Antes de configurar tu método de autenticación:

  * Revisa el [flujo de credenciales del cliente](/docs/es/get-started/authentication-and-authorization-flow/client-credentials-flow) para la autenticación de máquina a máquina. Este es el flujo de trabajo para la autenticación y autorización no interactivas.
  * Determina el nivel de acceso de tus API. Esto te ayudará a determinar qué [scopes](/docs/es/get-started/apis/scopes/api-scopes), o permisos, configurarás al crear tu API.
</Card>

<div id="configure-the-client-credential-flow">
  ### Configurar el flujo de credenciales del cliente
</div>

Puedes usar Auth0 Dashboard o <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip> para configurar el flujo de autenticación de forma que proporcione una credencial de cliente a cambio de un token de acceso.

<div id="m2m-access-tokens">
  ### Tokens de acceso de máquina a máquina
</div>

En la autenticación basada en tokens, los clientes no interactivos proporcionan `client_id` y `client_secret` en una llamada al [endpoint de tokens de Authentication API](/docs/es/customize/actions/explore-triggers/machine-to-machine-trigger) para obtener un [token de acceso](/docs/es/secure/tokens/access-tokens). Este token de acceso permite acceder a tu API protegida.

El perfil, o formato, predeterminado es el perfil de token de Auth0, asociado a dos perfiles de token. Puedes cambiar el perfil de token a RFC 9068 si lo deseas. Para obtener más información, consulta [Perfiles de token de acceso](/docs/es/secure/tokens/access-tokens/access-token-profiles). Para verificar que el token sea válido, tu API comprobará los [Algoritmos de firma](/docs/es/get-started/applications/signing-algorithms). El <Tooltip tip="Algoritmo de firma: algoritmo utilizado para firmar digitalmente tokens y garantizar que no se hayan manipulado." cta="Ver glosario" href="/docs/es/glossary?term=signing+algorithm">algoritmo de firma</Tooltip> predeterminado es RSA256, un algoritmo basado en claves.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Auth0 admite otros métodos de autenticación del cliente además del uso de ID de cliente y secreto de cliente como credenciales. Estos métodos, incluida nuestra recomendación de [clave privada JWT](/docs/es/get-started/authentication-and-authorization-flow/authenticate-with-private-key-jwt) para configuraciones de máquina a máquina, están disponibles con un plan Enterprise. Para obtener más información, consulta [Credenciales de la aplicación](/docs/es/secure/application-credentials).
</Callout>

<div id="token-expiration">
  #### Expiración del token
</div>

Tus tokens de acceso tienen un límite de validez. Dado que tus comunicaciones se realizan por el back-channel, no puedes usar <Tooltip tip="Token de actualización: token utilizado para obtener un nuevo token de acceso sin obligar a los usuarios a volver a iniciar sesión." cta="Ver glosario" href="/docs/es/glossary?term=refresh+tokens">tokens de actualización</Tooltip> para ampliar las sesiones, y deberías considerar configurar tus tokens de acceso con una expiración de 1 hora. Puede que necesites encontrar tu propio equilibrio entre seguridad y rendimiento para tu entorno específico. Para obtener más información, consulta [Actualizar la duración del token de acceso](/docs/es/secure/tokens/access-tokens/update-access-token-lifetime).

<div id="scopes">
  ### Scopes
</div>

Antes de que cualquier cliente o servicio no interactivo llame a tu API, debes definir los permisos o Scopes que admite tu API. Puedes configurar los scopes en Auth0 Dashboard para incluirlos en tu solicitud de autenticación a Authentication API. Para ver más ejemplos de formas de usar scopes de API, consulta [Scopes de API](/docs/es/get-started/apis/scopes/api-scopes).

Para configurar scopes, sigue las instrucciones de [Agregar permisos de API](/docs/es/get-started/apis/add-api-permissions) en Auth0 Dashboard o usa el ejemplo proporcionado para Management API.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Para agregar claims personalizados a un token de acceso, puedes usar el flujo de máquina a máquina de Actions. Para obtener más información, lee Machine to Machine Flow.
</Callout>

  En Auth0 Dashboard, debes:

  * Registrar y verificar tu dominio antes de poder usarlo con tus servicios de Auth0.
  * Decidir si quieres gestionar tu propio certificado o usar un certificado gestionado por Auth0. Para obtener más información sobre los certificados, consulta [Opciones de gestión de certificados](/docs/es/customize/custom-domains#certificate-management-options).
  * Verificar que la versión de TLS (SSL) y el cifrado que quieres usar para los certificados autogestionados sean compatibles con Auth0. Para obtener más información, consulta [Versiones y cifrados de TLS (SSL)](/docs/es/customize/custom-domains/self-managed-certificates/tls-ssl).
</Card>

1. Para configurar tu dominio personalizado con certificados gestionados por Auth0, sigue las instrucciones de [Configurar dominios personalizados con certificados gestionados por Auth0](/docs/es/customize/custom-domains/auth0-managed-certificates).

   1. Si quieres gestionar tus propios certificados, sigue las instrucciones de [Configurar dominios personalizados con certificados autogestionados](/docs/es/customize/custom-domains/self-managed-certificates).

      <Callout icon="file-lines" color="#0EA5E9" iconType="regular">
        Debes tener una suscripción Enterprise para gestionar certificados en tu dominio personalizado. Para obtener más información, consulta [Auth0 Pricing](https://auth0.com/pricing/) y [Login](/docs/es/authenticate/login).
      </Callout>
2. Revisa la [Configuración de API con dominios personalizados](/docs/es/customize/custom-domains/configure-features-to-use-custom-domains#apis). Puede que tengas que ajustar la configuración de tu API para incorporar un dominio personalizado.

<div id="deployment-automation">
  ## Automatización de despliegues
</div>

Auth0 ofrece soporte para un par de opciones distintas en cuanto a los enfoques de automatización de despliegues que puede usar, y ambas pueden usarse de forma conjunta.

<Card title="Práctica recomendada">
  Independientemente de cómo configure la automatización de despliegues, le recomendamos que haga pruebas unitarias de su código personalizado y de sus Actions antes del despliegue, y que también ejecute algunas pruebas de integración en su tenant después del despliegue.
</Card>

<div id="deploy-cli-tool">
  ### Herramienta Deploy CLI
</div>

Como se recomienda en la sección Arquitectura, deberías tener tenants de Auth0 para desarrollo, pruebas y producción. Estos tenants deben compartir configuraciones idénticas para los controles de calidad y las pruebas; sin embargo, es posible que te encuentres con errores como resultado de discrepancias de configuración entre tus entornos. Por ejemplo, cada entorno tendrá distintos <Tooltip tip="ID de cliente: valor de identificación otorgado a tu recurso registrado por Auth0." cta="Ver glosario" href="/docs/es/glossary?term=Client+IDs">IDs de cliente</Tooltip> y <Tooltip tip="ID de cliente: valor de identificación otorgado a tu recurso registrado por Auth0." cta="Ver glosario" href="/docs/es/glossary?term=Client+Secrets">secretos de cliente</Tooltip>.

Para mitigar estos errores por desajustes, puedes usar la herramienta Deploy CLI para integrar tu instancia de Auth0 con tu pipeline de CI/CD actual.  Con el reemplazo dinámico de palabras clave, puedes sustituir variables de entorno en tenants que comparten configuraciones similares. Para obtener más información, consulta [Herramienta Deploy CLI](/docs/es/deploy-monitor/deploy-cli-tool) y [Reemplazo de palabras clave](/docs/es/deploy-monitor/deploy-cli-tool/keyword-replacement).

<div id="actions-real-time-logs">
  ### logs en tiempo real de Actions
</div>

Los logs en tiempo real de Actions muestran todos los registros del código personalizado en tiempo real, incluida la salida de `console.log` y otras excepciones. Si usas Auth0 Actions u otra lógica personalizada, puedes usar esta extensión para depurar y resolver problemas. Para obtener más información sobre la instalación y la configuración, consulta [logs en tiempo real de Actions](/docs/es/customize/actions/actions-real-time-logs).

<div id="quality-assurance">
  ## Control de calidad
</div>

El control de calidad es importante para identificar problemas antes de pasar a producción. Según la naturaleza de tu proyecto, hay varios tipos de pruebas de control de calidad que te conviene considerar como parte de tu integración con Auth0:

* ¿Cómo responderán tus API ante picos de carga inesperados en producción?
* ¿Cómo se verán afectados tus límites de frecuencia por las aplicaciones de terceros?

Si no usas widgets o funcionalidades de Auth0, como <Tooltip tip="Universal Login: Tu aplicación redirige a Universal Login, alojado en el servidor de autorización de Auth0, para verificar la identidad de un usuario." cta="Ver glosario" href="/docs/es/glossary?term=Universal+Login">Universal Login</Tooltip>, no contarás de entrada con prácticas recomendadas integradas de usabilidad y accesibilidad para una amplia variedad de navegadores y dispositivos. Para garantizar que se cumplan los requisitos funcionales y que los eventos inesperados se gestionen correctamente, se ofrece orientación para probar la integración entre tu aplicación o aplicaciones y Auth0, y para realizar pruebas unitarias de módulos individuales de extensibilidad, como Auth0 Actions. También te recomendamos revisar la [política de pruebas de penetración](/docs/es/troubleshoot/customer-support/operational-policies/penetration-testing-policy) de Auth0 y realizar pruebas simuladas que puedas usar junto con nuestra [política de pruebas de carga](/docs/es/troubleshoot/customer-support/operational-policies/load-testing-policy) para ayudar a garantizar que tu aplicación o aplicaciones funcionen correctamente ante cargas inesperadas.

<div id="unit-testing">
  ### Pruebas unitarias
</div>

Las pruebas unitarias consisten en verificar unidades de extensibilidad, como Auth0 Actions. Si utiliza código personalizado, le recomendamos usar un framework de pruebas (como [Mocha](https://mochajs.org/)) para probar ese código adicional antes del despliegue.

<div id="mock-testing">
  ### Prueba simulada
</div>

Como punto de equilibrio entre la [política de pruebas de carga](/docs/es/troubleshoot/customer-support/operational-policies/load-testing-policy) de Auth0 y la necesidad de realizar una prueba de carga, es habitual crear una prueba simulada de los endpoints de Auth0. Esta práctica es muy útil para garantizar que tu configuración funcione con las interfaces previstas sin tener que limitar tus pruebas, y puedes apoyarte en herramientas como [MockServer](http://www.mock-server.com/), [JSON Server](https://github.com/typicode/json-server) o incluso [Postman](https://learning.getpostman.com/docs/postman/mock_servers/setting_up_mock/).

<div id="deployment">
  ## Despliegue
</div>

Nuestra sección [Despliegue y supervisión](/docs/es/deploy-monitor) ofrece orientación sobre las prácticas recomendadas para el despliegue. Le recomendamos revisar las [Comprobaciones prev](/docs/es/deploy-monitor/pre-deployment-checks)[ias al despliegue](/docs/es/deploy-monitor/pre-deployment-checks), especialmente las [Readiness Checks integradas de Auth0 Dashboard](/docs/es/deploy-monitor/pre-deployment-checks/how-to-run-production-checks).

Para revisar la comprobación de preparación, seleccione el menú desplegable situado debajo del nombre de su tenant y de la etiqueta del entorno en [Auth0 Dashboard &gt; Run Readiness Checks](https://manage.auth0.com/#/production-checks).

<Frame>![Auth0 Dashboard > Lista de comprobación de preparación](/docs/images/cdy7uua7fh8z/7iu1CzH0NgaXJbLBNWXZli/b598b8cdaad0c676e83a0fb0595d4603/Readiness_Checks_-_English.png)</Frame>

Puede usar el filtro para aplicar las comprobaciones de preparación a las aplicaciones seleccionadas. **Estas comprobaciones no se aplican a las API que tenga configuradas**.

En el caso de las comprobaciones que no se apliquen a su configuración específica, puede seleccionar **Descartar** para quitarlas de los resultados finales.

Le recomendamos revisar las [prácticas recomendadas de despliegue](/docs/es/deploy-monitor/deployment-best-practices) para realizar las comprobaciones finales antes de pasar a producción y aprovechar los [registros](/docs/es/deploy-monitor/logs) para supervisar sus servicios.
