Si tu caso de uso está dirigido a una <Tooltip tip="Audiencia: Identificador único de la audiencia de un token emitido. <div id="use-cases">
  ## Casos de uso
</div>

Usa la ruta de incorporación de M2M si:

* Das soporte a comunicaciones de servicio a servicio
* Tienes tareas programadas o tareas cron que se ejecutan en servidores y necesitan acceder a recursos protegidos o APIs
* Permites que los dispositivos IoT se comuniquen con servicios de backend o APIs
* Tienes una capa de API que necesita comunicarse con otras capas de API sin intervención del usuario o después de que haya expirado el token de un usuario
* Tienes una API privilegiada a la que quizá haya que llamar antes de que un usuario se autentique (es decir, desde una Action o un script de base de datos personalizada en tu tenant de Auth0)
* Usas API Gateway para administrar servicios de backend
* Usas o admites aplicaciones no interactivas u otras herramientas que no impliquen interacción humana, como demonios o servicios de backend

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Estos servicios seguirán necesitando un token de acceso M2M para la autenticación.
</Callout>

<div id="architecture">
  ## Arquitectura
</div>

Antes de configurar tu cuenta de Auth0 y tu tenant, o los grupos y estructuras de tus servicios de Auth0, crea un mapa de tu infraestructura actual para que puedas aprovechar al máximo las capacidades de Auth0 dentro de tu ecosistema existente.

Como se menciona en los escenarios comunes, es posible que tengas otras tecnologías no interactivas en tu dominio de aplicación, dominio de red o dominio de device de máquina a máquina que debas tener en cuenta antes de configurar Auth0. Para ver un ejemplo de un escenario de máquina a máquina, consulta [Server + API](/docs/es/get-started/architecture-scenarios/server-application-api/part-1). Si quieres hacer un laboratorio práctico con Node y probar la implementación de una API, visita nuestro [repositorio de GitHub](https://www.google.com/url?q=https://github.com/auth0-training/labs-node-working-with-apis\&sa=D\&source=docs\&ust=1723578190947730\&usg=AOvVaw3IXRQG9ogH4pW6LEY047T6).

También puede que quieras crear una visualización de tu pila tecnológica actual y planificar cómo encaja Auth0 en tu ciclo de vida de desarrollo de software (SDLC). Esto puede ayudarte a determinar cuántos tenants podrías necesitar.

<div id="considerations">
  ### Consideraciones
</div>

Antes de crear una nueva cuenta o configurar tu primer tenant, quizá te convenga tener en cuenta lo siguiente:

* Cómo particionas o agrupas tus API para llamar a endpoints específicos.

  * Esto puede determinar el audience y otros claims de los tokens de acceso.
* Cualquier consumidor third-party de tu recurso puede solicitar un token de acceso para cada llamada. Un número excesivo de llamadas podría afectar tu [límite de frecuencia](/docs/es/troubleshoot/customer-support/operational-policies/rate-limit-policy/rate-limit-configurations/essentials-professional-b2b).

  * Puedes usar API Gateway para limitar la cantidad de tokens de acceso que un third-party puede solicitar. Para obtener más información sobre API Gateway y Auth0, consulta [Configurar un proveedor de identidad en Access Gateway](/docs/es/get-started/auth0-overview/create-applications/configure-an-identity-provider-in-access-gateway).

<div id="authentication">
  ## Autenticación
</div>

Al llamar a una API desde otra API, o en cualquier situación en la que no haya un contexto de usuario autenticado, necesita una forma de autorizar a la aplicación en lugar de a un usuario. Este es un proceso de un solo paso en el que la aplicación se autentica (mediante un `client_id` y un `client_secret`) y luego se autoriza en una sola llamada.

Para aplicaciones o servicios no interactivos de autenticación, debe seleccionar una autorización de cliente o un flujo de autenticación. El <Tooltip tip="OAuth 2.0: Marco de autorización que define protocolos y flujos de trabajo de autorización." cta="Ver glosario" href="/docs/es/glossary?term=OAuth+2.0">OAuth 2.0</Tooltip> [flujo de credenciales de cliente](https://tools.ietf.org/html/rfc6749#section-4.4) no requiere interacción humana y es el más adecuado para aplicaciones de máquina a máquina.

<Card title="Antes de comenzar">
  En Auth0 Dashboard o Management API, hará lo siguiente:

  * Configurar su aplicación para usar el flujo de credenciales de cliente
  * Actualizar los scopes de sus tokens de acceso de máquina a máquina

  Antes de configurar su método de autenticación:

  * Revise el [flujo de credenciales de cliente](/docs/es/get-started/authentication-and-authorization-flow/client-credentials-flow) para la autenticación de máquina a máquina. Este es el flujo de trabajo para la autenticación y autorización no interactivas.
  * Determine el nivel de acceso de sus API. Esto ayuda a determinar qué [scopes](/docs/es/get-started/apis/scopes/api-scopes), o permisos, configurará cuando cree su API.
</Card>

<div id="configure-the-client-credential-flow">
  ### Configurar el flujo de credenciales del cliente
</div>

Puedes usar Auth0 Dashboard o <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip> para configurar el flujo de autenticación de modo que proporcione una credencial de cliente a cambio de un token de acceso.

Sigue las instrucciones de [Actualizar tipos de concesión](/docs/es/get-started/applications/update-grant-types) para usar Auth0 Dashboard o Management API.

<div id="m2m-access-tokens">
  ### Tokens de acceso de máquina a máquina
</div>

En la autenticación basada en tokens, los clientes no interactivos proporcionan `client_id` y `client_secret` en una llamada al [endpoint de token de Authentication API](/docs/es/customize/actions/explore-triggers/machine-to-machine-trigger) para obtener un [token de acceso](/docs/es/secure/tokens/access-tokens). Este token de acceso permite acceder a tu API protegida.

El perfil, o formato, predeterminado es el perfil de token de Auth0, uno de los dos perfiles de token disponibles. Puedes cambiar el perfil de token a RFC 9068. Para obtener más información, consulta [Perfiles de tokens de acceso](/docs/es/secure/tokens/access-tokens/access-token-profiles). Para verificar que el token sea válido, tu API comprobará los [Algoritmos de firma](/docs/es/get-started/applications/signing-algorithms). El <Tooltip tip="Algoritmo de firma: algoritmo utilizado para firmar digitalmente tokens y garantizar que el token no haya sido manipulado." cta="Ver glosario" href="/docs/es/glossary?term=signing+algorithm">algoritmo de firma</Tooltip> predeterminado es RSA256, un algoritmo basado en claves.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Auth0 admite otros métodos de autenticación del cliente además de proporcionar el ID de cliente y el secreto de cliente como credenciales. Estos métodos, incluida nuestra recomendación de [clave privada JWT](/docs/es/get-started/authentication-and-authorization-flow/authenticate-with-private-key-jwt) para configuraciones de máquina a máquina, están disponibles con un plan Enterprise. Para obtener más información, consulta [Credenciales de la aplicación](/docs/es/secure/application-credentials).
</Callout>

<div id="token-expiration">
  #### Vencimiento del token
</div>

Tus tokens de acceso tienen un límite de tiempo de validez. Dado que tus comunicaciones se realizan a través del back-channel, no puedes usar <Tooltip tip="Token de actualización: token usado para obtener un token de acceso renovado sin obligar a los usuarios a iniciar sesión de nuevo." cta="Ver glosario" href="/docs/es/glossary?term=refresh+tokens">tokens de actualización</Tooltip> para prolongar las sesiones, y deberías considerar configurar tus tokens de acceso con un tiempo de vencimiento de 1 hora. Puede que necesites encontrar el equilibrio adecuado entre seguridad y rendimiento para tu entorno específico. Para obtener más información, consulta [Actualizar la duración del token de acceso](/docs/es/secure/tokens/access-tokens/update-access-token-lifetime).

<div id="scopes">
  ### Scopes
</div>

Antes de que cualquier cliente o servicio no interactivo llame a tu API, debes definir los permisos o Scopes que admite tu API. Puedes configurar los scopes en Auth0 Dashboard para incluirlos en tu solicitud de autenticación a Authentication API. Para ver más ejemplos de formas de usar scopes de API, consulta [Scopes de API](/docs/es/get-started/apis/scopes/api-scopes).

Para configurar scopes, sigue las instrucciones de [Agregar permisos de API](/docs/es/get-started/apis/add-api-permissions) en Auth0 Dashboard o usa el ejemplo proporcionado para Management API.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Para agregar claims personalizados a un token de acceso, puedes usar el flujo Machine to Machine de Actions. Para obtener más información, consulta Machine to Machine Flow.
</Callout>

  * Registrar y verificar tu dominio antes de poder usarlo con tus servicios de Auth0.
  * Decidir si quieres gestionar tu propio certificado o usar un certificado administrado por Auth0. Para obtener más información sobre los certificados, consulta [Certificate management options](/docs/es/customize/custom-domains#certificate-management-options).
  * Verificar que la versión de TLS (SSL) y el cifrado que quieres usar para los certificados autogestionados sean compatibles con Auth0. Para obtener más información, consulta [TLS (SSL) Versions and Ciphers](/docs/es/customize/custom-domains/self-managed-certificates/tls-ssl).
</Card>

1. Para configurar tu dominio personalizado con certificados administrados por Auth0, sigue las instrucciones de [Configure dominios personalizados with Auth0-Managed Certificates](/docs/es/customize/custom-domains/auth0-managed-certificates).

   1. Si quieres gestionar tus propios certificados, sigue las instrucciones de [Configure dominios personalizados with Self-Managed Certificates](/docs/es/customize/custom-domains/self-managed-certificates).

      <Callout icon="file-lines" color="#0EA5E9" iconType="regular">
        Debes tener una suscripción Enterprise para gestionar certificados en tu dominio personalizado. Para obtener más información, consulta [Auth0 Pricing](https://auth0.com/pricing/) y [Login](/docs/es/authenticate/login).
      </Callout>
2. Revisa [API configuration with dominios personalizados](/docs/es/customize/custom-domains/configure-features-to-use-custom-domains#apis). Puede que tengas que ajustar la configuración de tu API para usar un dominio personalizado.

<div id="deployment-automation">
  ## Automatización del despliegue
</div>

Auth0 ofrece un par de opciones distintas para automatizar el despliegue, y ambas pueden utilizarse en conjunto.

<Card title="Práctica recomendada">
  Independientemente de cómo configures la automatización del despliegue, te recomendamos hacer pruebas unitarias de tu código personalizado y de tus Actions antes del despliegue, y también ejecutar algunas pruebas de integración en tu tenant después del despliegue.
</Card>

<div id="deploy-cli-tool">
  ### Herramienta Deploy CLI
</div>

Como se recomienda en la sección de Arquitectura, deberías tener tenants de Auth0 para desarrollo, pruebas y producción. Estos tenants deberían compartir configuraciones idénticas para las verificaciones de calidad y las pruebas; sin embargo, podrías encontrarte con errores como resultado de discrepancias de configuración entre tus entornos. Por ejemplo, cada entorno tendrá diferentes <Tooltip tip="ID de cliente: valor de identificación asignado por Auth0 a tu recurso registrado." cta="Ver glosario" href="/docs/es/glossary?term=Client+IDs">IDs de cliente</Tooltip> y <Tooltip tip="ID de cliente: valor de identificación asignado por Auth0 a tu recurso registrado." cta="Ver glosario" href="/docs/es/glossary?term=Client+Secrets">Secretos de cliente</Tooltip>.

Para mitigar estos errores por discrepancias, puedes usar la herramienta Deploy CLI para ayudarte a integrar tu instancia de Auth0 con tu pipeline de CI/CD actual. Con el reemplazo dinámico de palabras clave, puedes sustituir variables de entorno en tenants que comparten configuraciones similares. Para obtener más información, consulta [herramienta Deploy CLI](/docs/es/deploy-monitor/deploy-cli-tool) y [reemplazo de palabras clave](/docs/es/deploy-monitor/deploy-cli-tool/keyword-replacement).

<div id="actions-real-time-logs">
  ### logs en tiempo real de Actions
</div>

Los logs en tiempo real de Actions muestran todos los logs del código personalizado en tiempo real, incluida la salida de `console.log` y otras excepciones. Si usa Auth0 Actions u otra lógica personalizada, puede usar esta extensión para depurar y solucionar problemas. Para obtener más información sobre la instalación y la configuración, lea [logs en tiempo real de Actions](/docs/es/customize/actions/actions-real-time-logs).

<div id="quality-assurance">
  ## Control de calidad
</div>

El control de calidad es importante para identificar problemas antes de pasar a producción. Según la naturaleza de tu proyecto, hay varios tipos de pruebas de control de calidad que conviene considerar como parte de tu integración con Auth0:

* ¿Cómo responderán tus API cuando se enfrenten a cargas inesperadas en producción?
* ¿Cómo se verán afectados tus límites de frecuencia por las aplicaciones de terceros?

Si no utilizas widgets ni funcionalidades de Auth0, como <Tooltip tip="Universal Login: Tu aplicación redirige a Universal Login, alojado en el servidor de autorización de Auth0, para verificar la identidad de un usuario." cta="Ver glosario" href="/docs/es/glossary?term=Universal+Login">Universal Login</Tooltip>, no contarás de forma predeterminada con prácticas recomendadas de usabilidad y accesibilidad para una amplia variedad de navegadores y dispositivos. Para garantizar que se cumplan los requisitos funcionales y que los eventos inesperados se gestionen correctamente, se ofrece orientación para probar la integración entre tus aplicaciones y Auth0, así como para realizar pruebas unitarias de módulos individuales de extensibilidad, como Auth0 Actions. También te recomendamos revisar la [política de pruebas de penetración](/docs/es/troubleshoot/customer-support/operational-policies/penetration-testing-policy) de Auth0 y realizar pruebas simuladas, que puedes aprovechar junto con nuestra [política de pruebas de carga](/docs/es/troubleshoot/customer-support/operational-policies/load-testing-policy), para ayudar a garantizar que tus aplicaciones funcionen correctamente ante cargas inesperadas.

<div id="unit-testing">
  ### Pruebas unitarias
</div>

Las pruebas unitarias sirven para verificar unidades de extensibilidad, como Auth0 Actions. Si utiliza código personalizado, le recomendamos usar un framework de pruebas (como [Mocha](https://mochajs.org/)) para probar ese código adicional antes del despliegue.

<div id="mock-testing">
  ### Prueba simulada
</div>

Para equilibrar la [política de pruebas de carga](/docs/es/troubleshoot/customer-support/operational-policies/load-testing-policy) de Auth0 con la necesidad de realizar una prueba de carga, es habitual crear una prueba simulada de los endpoints de Auth0. Esta práctica es muy útil para garantizar que tu configuración funcione con las interfaces esperadas sin necesidad de limitar tus pruebas, y puedes apoyarte en herramientas como [MockServer](http://www.mock-server.com/), [JSON Server](https://github.com/typicode/json-server), o incluso [Postman](https://learning.getpostman.com/docs/postman/mock_servers/setting_up_mock/).

<div id="deployment">
  ## Despliegue
</div>

Nuestra sección [Desplegar y supervisar](/docs/es/deploy-monitor) ofrece orientación sobre las prácticas recomendadas para el despliegue. Le recomendamos revisar las [Comprobaciones previas al despliegu](/docs/es/deploy-monitor/pre-deployment-checks)[e](/docs/es/deploy-monitor/pre-deployment-checks), especialmente las [comprobaciones de preparación integradas del Auth0 Dashboard](/docs/es/deploy-monitor/pre-deployment-checks/how-to-run-production-checks).

Para revisar la comprobación de preparación, seleccione el menú desplegable que aparece debajo del nombre de su tenant y de la etiqueta del entorno en [Auth0 Dashboard &gt; Run Readiness Checks](https://manage.auth0.com/#/production-checks).

<Frame>![Auth0 Dashboard > Lista de comprobación de preparación](/docs/images/cdy7uua7fh8z/7iu1CzH0NgaXJbLBNWXZli/b598b8cdaad0c676e83a0fb0595d4603/Readiness_Checks_-_English.png)</Frame>

Puede usar el filtro para aplicar las comprobaciones de preparación a las aplicaciones seleccionadas. **Estas comprobaciones no se aplican a las API que haya configurado**.

En el caso de las comprobaciones que no se apliquen a su configuración específica, puede seleccionar **Dismiss** para quitarlas de los resultados finales.

Le recomendamos revisar las [Prácticas recomendadas de despliegue](/docs/es/deploy-monitor/deployment-best-practices) para realizar las comprobaciones finales antes de entrar en producción y aprovechar los [Logs](/docs/es/deploy-monitor/logs) para supervisar sus servicios.
