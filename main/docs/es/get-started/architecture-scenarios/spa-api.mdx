En este escenario, construiremos una API de hojas de horas para una empresa ficticia llamada ExampleCo. La API permitirá agregar hojas de horas para un empleado o un contratista.

También construiremos una aplicación de página única (SPA) que se utilizará para registrar hojas de horas y enviarlas a la base de datos centralizada de hojas de horas mediante la API.

<Info>
  ### En resumen

  * Auth0 proporciona autenticación y autorización de API para proteger el acceso a los endpoints de la API (consulta [Autenticación y autorización de API](/docs/es/get-started/architecture-scenarios/spa-api/part-1#api-authentication-and-authorization))
  * Para autorizar a un usuario de una SPA, Auth0 admite la concesión implícita (consulta [Concesión implícita](/docs/es/get-started/architecture-scenarios/spa-api/part-1#implicit-grant))
  * Tanto la SPA como la API deben configurarse en el Auth0 Dashboard (consulta [Auth0 Configuration](/docs/es/get-started/architecture-scenarios/spa-api/part-2#auth0-configuration))
  * Los Permissions del usuario se pueden aplicar mediante la Authorization Extension (consulta [Configurar la Authorization Extension](/docs/es/get-started/architecture-scenarios/spa-api/part-2#configure-the-authorization-extension))
  * La API estará protegida al garantizar que se envíe un token de acceso válido en el encabezado HTTP Authorization cuando se hagan llamadas a la API (consulta [Implementar la API](/docs/es/get-started/architecture-scenarios/spa-api/part-3#implement-the-api))
  * La biblioteca Auth0.js se puede utilizar para autorizar al usuario de la SPA y obtener un token de acceso válido que puede usarse para llamar a la API (consulta [Autorizar al usuario](/docs/es/get-started/architecture-scenarios/spa-api/part-3#authorize-the-user))
  * La SPA puede enviar el token de acceso en el encabezado HTTP Authorization al hacer llamadas a la API (consulta [Llamar a la API](/docs/es/get-started/architecture-scenarios/spa-api/part-3#call-the-api))
  * La SPA puede mostrar elementos de la UI de forma condicional según los scopes otorgados al usuario (consulta [Mostrar elementos de la UI de forma condicional según el scope](/docs/es/get-started/architecture-scenarios/spa-api/part-3#display-ui-elements-conditionally-based-on-scope))
</Info>

<div id="the-premise">
  ## La premisa
</div>

ExampleCo es una startup de consultoría. Actualmente, tiene aproximadamente 100 empleados y además subcontrata varias actividades a contratistas externos. Todos los empleados y los contratistas externos deben completar sus hojas de horas cada semana.

La empresa ha desarrollado una aplicación de hojas de horas, un escenario que analizamos en [Inicio de sesión único para aplicaciones web tradicionales](/docs/es/get-started/architecture-scenarios/sso-for-regular-web-apps). Los empleados internos usan esta aplicación web para completar sus hojas de horas, pero la empresa quiere sustituirla por una SPA. La aplicación se usará para registrar las hojas de horas y enviar los datos a la base de datos centralizada de hojas de horas mediante la API. La aplicación también permitirá a los gerentes aprobar las hojas de horas.

<div id="goals-requirements">
  ## Objetivos y requisitos
</div>

ExampleCo quiere crear una solución flexible. Por ahora, solo se necesita una SPA para registrar hojas de horas, pero en el futuro la empresa tiene previsto lanzar más aplicaciones, como una aplicación móvil para dar servicio a sus equipos de ventas. Por ello, la empresa ha decidido desarrollar una única API de hoja de horas que se utilizará para registrar el tiempo no solo mediante este proceso del servidor, sino también desde todas las aplicaciones futuras. Quieren implantar una arquitectura de seguridad lo bastante flexible como para dar cabida a esto. ExampleCo quiere garantizar que gran parte del código y de la lógica de negocio de la aplicación pueda compartirse entre las distintas aplicaciones.

Se requiere que solo los usuarios y las aplicaciones autorizados tengan acceso a la API de hoja de horas.

Esta SPA será utilizada por dos tipos de usuarios: empleados y gerentes. Los empleados deben poder leer, crear y eliminar sus propias hojas de horas, mientras que los gerentes también deben poder aprobar hojas de hora.
