En este escenario, construiremos una API de hojas de hora para una empresa ficticia llamada ExampleCo. La API permitirá agregar entradas de hoja de hora para un Employee o un contratista.

También construiremos una aplicación de página única (SPA) que se utilizará para registrar entradas de hoja de hora y enviarlas a la base de datos centralizada de hojas de hora mediante la API.

<Info>
  ### En resumen

  * Auth0 proporciona autenticación y autorización para API como una forma de proteger el acceso a los endpoints de API (consulta [API Authentication and Authorization](/docs/es/get-started/architecture-scenarios/spa-api/part-1#api-authentication-and-authorization))
  * Para autorizar a un usuario de una SPA, Auth0 admite la concesión Implicit (consulta [Implicit Grant](/docs/es/get-started/architecture-scenarios/spa-api/part-1#implicit-grant))
  * Tanto la SPA como la API deben configurarse en el Auth0 Dashboard (consulta [Auth0 Configuration](/docs/es/get-started/architecture-scenarios/spa-api/part-2#auth0-configuration))
  * Los permisos de usuario pueden aplicarse mediante la Authorization Extension (consulta [Configure the Authorization Extension](/docs/es/get-started/architecture-scenarios/spa-api/part-2#configure-the-authorization-extension))
  * La API estará protegida asegurando que se envíe un token de acceso válido en el encabezado HTTP Authorization cuando se hagan llamadas a la API (consulta [Implement the API](/docs/es/get-started/architecture-scenarios/spa-api/part-3#implement-the-api))
  * La biblioteca Auth0.js puede utilizarse para autorizar al usuario de la SPA y obtener un token de acceso válido que pueda usarse para llamar a la API (consulta [Authorize the User](/docs/es/get-started/architecture-scenarios/spa-api/part-3#authorize-the-user))
  * La SPA puede enviar el token de acceso en el encabezado HTTP Authorization al hacer llamadas a la API (consulta [Call the API](/docs/es/get-started/architecture-scenarios/spa-api/part-3#call-the-api))
  * La SPA puede mostrar elementos de UI de forma condicional según los scopes concedidos al usuario (consulta [Display UI Elements Conditionally Based on Scope](/docs/es/get-started/architecture-scenarios/spa-api/part-3#display-ui-elements-conditionally-based-on-scope))
</Info>

<div id="the-premise">
  ## La premisa
</div>

ExampleCo es una startup de consultoría. Actualmente, tiene aproximadamente 100 empleados y además externaliza varias actividades a contratistas externos. Todos los empleados y los contratistas externos deben rellenar sus hojas de hora cada semana.

La empresa ha creado una aplicación de hojas de hora, un escenario que tratamos en [Inicio de sesión único para aplicaciones web tradicionales](/docs/es/get-started/architecture-scenarios/sso-for-regular-web-apps). Los empleados internos usan esta aplicación web para rellenar sus hojas de hora, pero la empresa quiere sustituirla por una SPA. La aplicación se usará para registrar entradas en las hojas de hora y enviar los datos a la base de datos centralizada de hojas de hora mediante la API. La aplicación también permitirá que los gerentes aprueben las entradas de las hojas de hora.

<div id="goals-requirements">
  ## Objetivos y requisitos
</div>

ExampleCo quiere crear una solución flexible. Por ahora, solo se necesita una SPA para capturar entradas de la hoja de hora, pero en el futuro la empresa planea lanzar más aplicaciones, como una aplicación móvil para dar soporte a sus equipos de ventas. Por ello, la empresa ha decidido desarrollar una única API de hoja de hora, que se utilizará para registrar el tiempo no solo mediante este proceso del servidor, sino también desde todas las aplicaciones futuras. Quieren implementar una arquitectura de seguridad lo bastante flexible como para dar cabida a esto. ExampleCo quiere asegurarse de que una gran parte del código y la lógica de negocio de la aplicación pueda compartirse entre las distintas aplicaciones.

Se requiere que solo los usuarios y las aplicaciones autorizados tengan acceso a la API de hoja de hora.

Esta SPA será utilizada por dos tipos de usuarios: empleados y gerentes. Los empleados deben poder leer, crear y eliminar sus propias entradas de la hoja de hora, mientras que los gerentes también deben poder aprobar hojas de hora.
