---

Quality Assurance es importante para identificar problemas antes de que afecten a sus clientes y, según la naturaleza de su proyecto, hay varios tipos de pruebas de aseguramiento de la calidad que le conviene tener en cuenta como parte de su integración con Auth0:

* ¿Su aplicación es fácil de entender y usar, incluso para personas con alguna discapacidad?
* ¿Su aplicación necesita funcionar en distintos navegadores y dispositivos?
* ¿Su aplicación necesita funcionar en entornos multinacionales y/o internacionales?
* ¿Cómo responderá su aplicación cuando se vea sometida a cargas inesperadas en producción?
* ¿Cómo puede asegurarse de que su aplicación esté protegida frente a vulnerabilidades de seguridad?

Auth0 [Universal Login](/docs/es/authenticate/login/auth0-universal-login) y los widgets de interfaz de usuario asociados (como [Lock](/docs/es/libraries/lock)) ya se han diseñado y desarrollado siguiendo prácticas recomendadas de usabilidad y accesibilidad, y ofrecen soporte probado y listo para usar para una amplia variedad de [navegadores y dispositivos](/docs/es/troubleshoot/customer-support/product-support-matrix). También se ofrece soporte listo para usar para la [internacionalización](/docs/es/customize/internationalization-and-localization) (I18N), con extensibilidad integrada diseñada para situaciones personalizadas de múltiples idiomas y localización (L10N).

Para garantizar que se cumplan los requisitos funcionales y que los eventos inesperados se gestionen correctamente, se ofrece orientación para probar la [integración](#integration-testing) entre sus aplicaciones y Auth0, así como para realizar [pruebas unitarias](#unit-testing) de módulos individuales de extensibilidad (como [Rules](/docs/es/customize/rules/debug-rules), [Hooks](/docs/es/customize/hooks/update-hooks) y script de Custom DB). También se ofrece orientación sobre la [política de pruebas de penetración](/docs/es/troubleshoot/customer-support/operational-policies/penetration-testing-policy) de Auth0 para ayudarle al realizar pruebas de vulnerabilidades de seguridad, así como sobre cómo se pueden aprovechar las pruebas [simuladas](#mock-testing) junto con nuestra [política de pruebas de carga](/docs/es/troubleshoot/customer-support/operational-policies/load-testing-policy) para contribuir a que sus aplicaciones rindan correctamente ante cargas inesperadas.

<div id="unit-testing">
  ## Pruebas unitarias
</div>

El objetivo de las pruebas unitarias es comprobar unidades individuales de código. Si crea código personalizado en Auth0 en forma de Rules, Hooks y/o scripts de Custom DB, debería considerar el uso de un framework de pruebas (como [Mocha](https://mochajs.org/)) para probar su código. Las empresas que han tenido más éxito con Auth0 han comprobado que es útil ejecutar estas pruebas unitarias antes de [desplegar automáticamente](/docs/es/get-started/architecture-scenarios/business-to-consumer/deployment) la configuración y los recursos asociados del tenant de Auth0.

<div id="integration-testing">
  ## Pruebas de integración
</div>

Una práctica recomendada es configurar distintos tenants para desarrollo, pruebas y producción, como se explica en la guía de Arquitectura sobre [soporte para SDLC](/docs/es/get-started/architecture-scenarios/business-to-consumer/architecture#sdlc-support). Auth0 le permite configurar variables disponibles desde la [extensibilidad](/docs/es/customize/extensions) personalizada; pueden considerarse variables de entorno para su tenant de Auth0. En lugar de codificar de forma fija referencias que cambian al mover código entre los entornos de desarrollo, pruebas y producción, puede usar un nombre de variable configurado en el tenant y referenciado por el código de extensibilidad personalizada. Esto facilita que el mismo código personalizado funcione, sin cambios, en distintos tenants, ya que puede hacer referencia a variables que se completarán con valores específicos del tenant en el momento de la ejecución:

* Para usar variables en Rules, consulte cómo [configurar valores](/docs/es/customize/rules/configuration)
* Para usar variables en Hooks, consulte cómo configurar [secrets](/docs/es/customize/hooks/hook-secrets) en el editor
* Para usar variables en Actions, consulte Explorar Flows y Triggers
* Para usar variables en scripts de Custom DB, consulte los [parámetros de configuración](/docs/es/authenticate/database-connections/custom-db/create-db-connection#step-3-add-configuration-parameters)

<Info>
  ### Práctica recomendada

  Una práctica recomendada es usar variables para almacenar valores específicos del tenant, así como cualquier Secret confidencial que no deba exponerse en su código personalizado. Si su código personalizado está desplegado en GitHub, usar una variable específica del tenant evita exponer valores confidenciales a través de su repositorio de GitHub.
</Info>

<div id="test-automation">
  ### Automatización de pruebas
</div>

Puedes automatizar tu proceso de compilación general incorporando la automatización del despliegue, así como la automatización de pruebas. Esto puede usarse para desplegar nuevas versiones de la configuración y/o del código personalizado en Auth0 y ejecutar pruebas automatizadas. Si las pruebas detectan algún fallo, la automatización del despliegue puede utilizarse para revertir a la última versión funcional. Para obtener más información, consulta la [guía sobre automatización del despliegue](/docs/es/get-started/architecture-scenarios/business-to-consumer/deployment).

<div id="mock-testing">
  ## Prueba simulada
</div>

Para equilibrar la [política de pruebas de carga](/docs/es/troubleshoot/customer-support/operational-policies/load-testing-policy) de Auth0 con la necesidad de realizar una prueba de carga, es habitual que los clientes de Auth0 simulen los endpoints de Auth0. Esta práctica resulta muy útil para garantizar que su aplicación funcione con las interfaces previstas sin tener que limitar sus pruebas, y puede apoyarse en herramientas como [MockServer](http://www.mock-server.com/), [JSON Server](https://github.com/typicode/json-server) o incluso [Postman](https://learning.getpostman.com/docs/postman/mock_servers/setting_up_mock/).
