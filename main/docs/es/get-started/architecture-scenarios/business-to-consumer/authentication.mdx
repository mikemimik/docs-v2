description: Cómo funciona la autenticación en tu implementación de B2C IAM.
title: Autenticación (B2C)
---

Para ofrecer servicios a tus usuarios, debes poder identificar quiénes son. Este proceso se llama autenticación de usuario. Existen varias formas de autenticar a un usuario: mediante cuentas de redes sociales, nombre de usuario y contraseña, <Tooltip tip="Passwordless: Forma de autenticación que no depende de una contraseña como primer factor." cta="Ver glosario" href="/docs/es/glossary?term=passwordless">passwordless</Tooltip>; y a menudo se recomienda ir más allá de un único factor de autenticación habilitando la <Tooltip tip="Passwordless: Forma de autenticación que no depende de una contraseña como primer factor." cta="Ver glosario" href="/docs/es/glossary?term=multi-factor+authentication">autenticación multifactor</Tooltip> (MFA).

<Info>
  ### Práctica recomendada

  Es importante tener en cuenta tanto la seguridad como la experiencia de usuario al diseñar cómo vas a autenticar a tus usuarios. Ofrecer varios factores primarios y/o exigir más de un factor durante la autenticación son formas de lograr ambas cosas.
</Info>

Hay varios aspectos que querrás tener en cuenta al analizar la funcionalidad y el flujo de trabajo:

* ¿Dónde introducirán los usuarios sus credenciales?
* ¿Cómo mantendrás seguras las credenciales de los usuarios?
* ¿Cómo mantendrás tu sistema de autenticación?
* ¿Cómo puedes ofrecer autenticación mediante contraseña a tus usuarios?
* ¿Cómo puedes evitar que atacantes intenten iniciar sesión haciéndose pasar por tus usuarios?
* ¿Cómo implementarás la autenticación en distintos tipos de aplicaciones?
* ¿Cómo puedes facilitar el inicio de sesión a tus usuarios cuando provienen de distintos contextos lingüísticos?
* ¿Cómo ofrecerás una buena experiencia de usuario mientras migras desde cualquier sistema de legacy authentication?
* ¿Qué debes tener en cuenta al integrar aplicaciones con Auth0?
* ¿Pueden los usuarios iniciar sesión con sus cuentas sociales existentes (por ejemplo, Facebook o Google)?
* ¿Necesitas proporcionar autenticación multifactor?
* ¿Qué haces si tienes un servicio que no permite que el usuario inicie sesión de antemano?
* ¿Puedes pasar el mismo <Tooltip tip="Token de acceso: Credencial de autorización, en forma de una cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=access+token">token de acceso</Tooltip> de usuario de una API a otra?

Auth0 [Universal Login](#universal-login) ofrece a los usuarios una experiencia segura y protegida, tanto si eliges ofrecer inicio de sesión con credenciales de ID de usuario/contraseña como si permites los llamados escenarios de trae tu propia identidad mediante [inicio de sesión social](https://auth0.com/learn/social-login/). Centralizar la experiencia de inicio de sesión con <Tooltip tip="Universal Login: Tu aplicación redirige a Universal Login, alojado en el servidor de autorización de Auth0, para verificar la identidad de un usuario." cta="Ver glosario" href="/docs/es/glossary?term=Universal+Login">Universal Login</Tooltip> también aporta ventajas de reconocimiento de marca, incluso si consideras que además tendrás requisitos de [branding](/docs/es/get-started/architecture-scenarios/business-to-consumer/branding) específicos del producto. Los widget de interfaz de usuario de Auth0 que se suelen usar con Universal Login también ofrecen soporte listo para usar en lo que respecta a la [internacionalización](/docs/es/customize/internationalization-and-localization/universal-login-internationalization) para usuarios con distintos requisitos de idioma, y ese soporte inmediato para funcionalidades de Auth0 como [MFA](#multi-factor-authentication-mfa-) y [protección contra ataques](#anomaly-detection) te permite establecer barreras para evitar que atacantes intenten acceder a las cuentas de los usuarios.

Permitir que los usuarios inicien sesión mediante credenciales de ID de usuario/contraseña significa que no dependes del estado de proveedores de identidad de terceros <Tooltip tip="Proveedor de identidad (IdP): Servicio que almacena y administra identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=identity+providers">proveedores de identidad</Tooltip> para que tus usuarios accedan a tu sistema. También te permite exigir que las credenciales utilizadas se ajusten a tus políticas corporativas. Auth0 te ayuda con esto al ofrecerte varias opciones para respaldar los inicios de sesión con ID de usuario/contraseña, y la [guía proporcionada](#username-and-password-authentication) te ayudará a comprender cómo puedes aprovecharlas. Añadir compatibilidad [social](#social-authentication) en alguna etapa, como factor principal de autenticación adicional, te ofrece más flexibilidad y puede ayudarte a comprender mejor a tus usuarios sin necesidad de hacerles más preguntas, aprovechando la información ya almacenada por los diversos [proveedores](/docs/es/authenticate/identity-providers/social-identity-providers) de inicio de sesión social.

Si ya tiene un legacy identity store, también le convendrá consultar [migración de usuario](/docs/es/get-started/architecture-scenarios/business-to-consumer/provisioning). En esta sección se analizan las ventajas de migrar al almacenamiento de identidades administrado por Auth0 en términos de protección y seguridad.

Para las aplicaciones orientadas al cliente, <Tooltip tip="OpenID: Estándar abierto de autenticación que permite a las aplicaciones verificar la identidad de los usuarios sin recopilar ni almacenar información de inicio de sesión." cta="Ver glosario" href="/docs/es/glossary?term=OpenID">OpenID</Tooltip> Connect ([OIDC](/docs/es/authenticate/protocols/openid-connect-protocol)) es el protocolo estándar del sector más utilizado, y OIDC cuenta con soporte de primer nivel en Auth0. Auth0 admite varios enfoques distintos para integrar distintas aplicaciones, por lo que le convendrá consultar la sección sobre [integración de aplicaciones](#application-integration) para obtener la información necesaria y tomar una decisión fundamentada.

Al llamar a una API desde otra API, o en cualquier situación en la que no haya un contexto de usuario autenticado —como una o más tareas cron, generadores de informes o sistemas de integración/entrega continuas—, necesitará una forma de autorizar la aplicación en lugar de un usuario. Este es un proceso de un solo paso en el que la aplicación se autentica (mediante un <Tooltip tip="ID de cliente: Valor de identificación asignado a su recurso registrado por Auth0." cta="Ver glosario" href="/docs/es/glossary?term=client+ID">ID de cliente</Tooltip> y un secreto) y luego se autoriza en una sola llamada. Puede obtener más información sobre esto en nuestro flujo de trabajo de autorización, en [autorización de máquina a máquina (m2m)](/docs/es/get-started/architecture-scenarios/business-to-consumer/authorization).

<div id="universal-login">
  ## Universal Login
</div>

¿Tiene, o tendrá, más de una aplicación en su sistema? Si la respuesta es sí, le convendrá ofrecer una experiencia de inicio de sesión centralizada. Para lograr una experiencia fluida de <Tooltip tip="Single Sign-On (SSO): Servicio que, después de que un usuario inicia sesión en una aplicación, inicia automáticamente la sesión de ese usuario en otras aplicaciones." cta="Ver glosario" href="/docs/es/glossary?term=Single+Sign-on">inicio de sesión único</Tooltip> (SSO) entre varias aplicaciones, es fundamental contar con un punto centralizado al que redirigir a sus usuarios para la autenticación. Esto le permite ofrecerles una experiencia coherente si en el futuro añade autenticación social, incorpora aplicaciones de terceros a su sistema o agrega autenticación multifactor como opción (o requisito), y además aprovechar nuevas funcionalidades para mejorar la experiencia de sus usuarios con poco o ningún esfuerzo de desarrollo adicional.

<Info>
  ### Práctica recomendada

  Si tiene más de una aplicación, la práctica recomendada es redirigir a una [ubicación centralizada](/docs/es/authenticate/login/auth0-universal-login) para autenticar al usuario. Con Auth0, esto significa aprovechar [Universal Login](/docs/es/authenticate/login/auth0-universal-login), que ofrece numerosas ventajas de seguridad y de experiencia de usuario listas para usar, incluido [SSO](/docs/es/authenticate/single-sign-on).
</Info>

Auth0 Universal Login hace que autenticar usuarios sea un proceso breve y sencillo, que puede completarse en tres pasos simples (todos nuestros Quickstarts lo demuestran, y nuestros SDKs también ocultan la complejidad por usted):

1. Determine cómo y cuándo quiere [redirigir desde su aplicación](#application-integration).
2. Configure el [branding](/docs/es/get-started/architecture-scenarios/business-to-consumer/branding) adecuado y/o el HTML personalizado en su configuración de Auth0.
3. Configure su aplicación para [recibir y gestionar la respuesta](#application-integration) del servidor de autorización.

<div id="username-and-password-authentication">
  ## Autenticación con nombre de usuario y contraseña
</div>

Casi todas las aplicaciones B2C ofrecen a sus clientes la posibilidad de crear sus propias credenciales. Esta es una forma común de autenticación con la que todos los usuarios están familiarizados.

En Auth0, la autenticación con nombre de usuario y contraseña puede implementarse de varias maneras. Si su aplicación es nueva y no tiene una base de usuarios existente, una [conexión de base de datos](/docs/es/authenticate/database-connections) de Auth0 lista para usar le dará todo lo que necesita para empezar a autenticar a sus usuarios. Sin embargo, si tiene un repositorio de usuarios heredado, como su propia base de datos de usuarios o un sistema LDAP existente, dispone de un par de opciones para migrar a sus usuarios, como se explica en nuestra guía sobre [migración de usuario](/docs/es/get-started/architecture-scenarios/business-to-consumer/provisioning).

Independientemente de cómo termine aprovisionando a los usuarios para su conexión de base de datos, la autenticación de esos usuarios es bastante similar. Para ello, debe presentarles un formulario en el que introduzcan su nombre de usuario y contraseña. Como se menciona en la guía sobre [Universal Login](#universal-login), la forma más sencilla y segura de autenticar a los usuarios con nombre de usuario y contraseña es redirigirlos a una página de inicio de sesión centralizada y recopilar allí esos datos. Esto permite a Auth0 determinar si ya se autenticaron y omitir por completo el formulario de inicio de sesión cuando no sea necesario.

<Info>
  ### Práctica recomendada

  Recopilar credenciales únicamente en la página de inicio de sesión centralizada reducirá la exposición a posibles filtraciones de secretos de usuario. También reducirá la necesidad de solicitar credenciales innecesariamente. Consulte [Universal Login](#universal-login) para obtener más información.
</Info>

<div id="application-integration">
  ## Integración de aplicaciones
</div>

Una vez que hayas decidido cómo quieres autenticar a tus usuarios, el siguiente paso es determinar cómo iniciarás esa autenticación. Normalmente, cada aplicación tendrá su propio punto de partida.

<Warning>
  Las aplicaciones móviles nativas (y las aplicaciones de escritorio) deben usar el navegador del sistema para la autenticación; de lo contrario, quedan expuestas a riesgos de seguridad adicionales. Consulta [Inicio de sesión nativo](/docs/es/authenticate/login/native-login) para obtener más información.
</Warning>

Como ya comentamos, hemos comprobado que la mayoría de nuestros clientes utilizan [OpenID Connect (OIDC)](/docs/es/authenticate/protocols/openid-connect-protocol) como protocolo estándar del sector para sus aplicaciones orientadas al cliente. Tu primera tarea es determinar qué flujo de OIDC debes usar, y te conviene empezar revisando nuestra guía sobre [mapeo de concesiones](/docs/es/get-started/applications/application-grant-types).

Si quieres permitir que usuarios anónimos accedan a alguna parte de tu aplicación, debes determinar si los redirigirás de inmediato o si solo les pedirás que se redirijan cuando sea necesario (o quizá una combinación de ambas opciones; consulta [Redirigir a los usuarios después del login](/docs/es/authenticate/login/redirect-users-after-login) para más información). Si los usuarios pueden usar [enlaces profundos](#deep-linking-to-protected-endpoints) a una versión protegida (o un área protegida) de tu sitio, tendrás que determinar qué enlaces a tu aplicación darán lugar a una redirección automática a Auth0.

<div id="anonymous-access">
  ### Acceso anónimo
</div>

Es importante tener en cuenta la experiencia de usuario cuando alguien llega por primera vez a tu aplicación. Si tu aplicación admite acceso de usuarios anónimos (algo bastante común en las aplicaciones de comercio electrónico), hay diferentes escenarios que debes considerar:

* ¿Está regresando a la aplicación después de haber iniciado sesión anteriormente, o
* Si es la primera vez que accede a la aplicación:

  * ¿Ya ha accedido a otra aplicación que usa el mismo tenant de Auth0,
  * ¿Se ha autenticado alguna vez (o quizás no en mucho tiempo) en este dispositivo o navegador?

Cuando un usuario anónimo accede a tu aplicación, a menudo puede ser conveniente que la aplicación detecte si el usuario ya inició sesión en otra aplicación de la misma familia, o que recuerde a este usuario incluso si la aplicación es una [SPA](/docs/es/quickstart/spa) sin estado. Por ejemplo, si puedes determinar que el usuario ya inició sesión, podrías decidir que el encabezado de la interfaz de la aplicación omita mostrar un botón de inicio de sesión y, en su lugar, muestre un menú de cuenta o perfil para el usuario. Para lograrlo, te convendrá utilizar la &quot;[autenticación silenciosa](/docs/es/authenticate/login/configure-silent-authentication)&quot;. La autenticación silenciosa te permitirá comprobar si el usuario ha iniciado sesión sin pedirle que lo haga si no es así. Entonces, la aplicación puede mostrar un botón de inicio de sesión si es necesario. Sin embargo, si el usuario ya inició sesión, recibirás tokens y no tendrás que volver a mostrarle un botón de inicio de sesión.

<Warning>
  Comprobar si existe una sesión de inicio de sesión redirigiendo a Auth0 puede ser útil para tu aplicación, pero si esto va a generar muchas solicitudes, debes emplear un mecanismo de limitación para evitar latencia o límites de frecuencia.

  Las llamadas a la Management API están sujetas a la [política de límite de frecuencia de Auth0](/docs/es/troubleshoot/customer-support/operational-policies/rate-limit-policy). Debes tener esto en cuenta y, como ayuda, Auth0 generalmente recomienda usar el [SDK de Auth0](/docs/es/libraries) adecuado para tu entorno de desarrollo en lugar de llamar directamente a nuestras API.
</Warning>

<div id="deep-linking-to-protected-endpoints">
  ### Enlaces profundos a endpoints protegidos
</div>

Hay varios motivos por los que alguien podría enlazar directamente a una página concreta dentro de tu aplicación a la que solo pueden acceder usuarios autenticados. Si esto es posible en tu aplicación, deberías redirigir automáticamente al usuario a Auth0 si no está autenticado. Una vez que se autentique y el <Tooltip tip="Servidor de autorización: Servidor centralizado que contribuye a definir los límites del acceso de un usuario. Por ejemplo, tu servidor de autorización puede controlar los datos, las tareas y las funcionalidades disponibles para un usuario." cta="Ver glosario" href="/docs/es/glossary?term=authorization+server">servidor de autorización</Tooltip> lo devuelva a tu aplicación, puedes [redirigirlo](/docs/es/authenticate/login/redirect-users-after-login) adonde quería ir desde el principio.

<div id="best-practice">
  ### Práctica recomendada
</div>

La mayoría de los frameworks modernos de autenticación admiten middleware para redirigir a un servidor de autorización como Auth0. Al elegir uno, ten en cuenta estas consideraciones clave:

* Compatibilidad con clientes confidenciales, clientes no confidenciales o ambos
* Compatibilidad con la configuración mediante [endpoint de descubrimiento](/docs/es/get-started/applications/configure-applications-with-oidc-discovery "Configurar aplicaciones con OIDC Discovery") o definida explícitamente en línea
* Compatibilidad con la validación de tokens, incluidas la expiración, las firmas, los claims y los scopes
* Compatibilidad con <Tooltip tip="Token de actualización: token que se usa para obtener un nuevo token de acceso sin obligar a los usuarios a volver a iniciar sesión." cta="Ver glosario" href="/docs/es/glossary?term=Refresh+Tokens">tokens de actualización</Tooltip> si es necesario

<div id="authenticating-the-user">
  ### Autenticación del usuario
</div>

La autenticación es el proceso de verificar la identidad del usuario. El resultado de la autenticación en un contexto de OIDC es un <Tooltip tip="Token de ID: Credencial destinada al propio cliente, en lugar de para acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=ID+Token">token de ID</Tooltip>. Este token contiene información sobre el usuario y solo debería poder obtenerse si el usuario se autentica mediante uno o más factores, según lo defina el servidor de autorización (la forma más habitual es la [ID de usuario y contraseña](#username-and-password-authentication)). Hay algunos aspectos que quizá también debas tener en cuenta, además de obtener un token de ID:

* ¿También necesitamos un [token de acceso](/docs/es/secure/tokens/access-tokens) para poder llamar a una API compartida?
* ¿Tu aplicación es una aplicación de página única y solo requiere un [token de ID](/docs/es/secure/tokens/id-tokens)? Consulta [concesión Authorization Code con PKCE](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce/call-your-api-using-the-authorization-code-flow-with-pkce) para más información.
* ¿Tu aplicación es una aplicación nativa (móvil o de escritorio) y/o necesitas un [token de actualización](/docs/es/secure/tokens/refresh-tokens)? Consulta [concesión Authorization Code con PKCE](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce/call-your-api-using-the-authorization-code-flow-with-pkce) para más información.

<Warning>
  Antes de pasar a producción, debes asegurarte de que **solo** estén habilitadas, para cada aplicación, las concesiones que utilizas en la [configuración de tu aplicación](/docs/es/get-started/applications/update-grant-types).
</Warning>

<div id="authorization-code-grant-with-or-without-pkce">
  ### Concesión Authorization Code (con o sin PKCE)
</div>

Si tu SDK solo admite la concesión Authorization Code, o necesitas un token de acceso o un <Tooltip tip="Token de actualización: token usado para obtener un token de acceso renovado sin obligar a los usuarios a volver a iniciar sesión." cta="Ver glosario" href="/docs/es/glossary?term=Refresh+Token">token de actualización</Tooltip>, la concesión Authorization Code (con o sin [PKCE](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce)) también puede usarse para obtener un token de ID. La concesión Authorization Code incluye una llamada adicional a la API para intercambiar el código por un token, lo que puede generar una latencia innecesaria si lo único que necesitas es el token de ID. En muchos casos, se implementa el [flujo híbrido](/docs/es/get-started/authentication-and-authorization-flow/hybrid-flow) para proporcionar un acceso óptimo al token de ID, sin dejar de aprovechar el flujo de trabajo de la concesión Authorization Code para obtener de forma segura los tokens de acceso y de actualización.

<Warning>
  Aunque Auth0 admite el uso de la concesión Implicit en aplicaciones basadas en navegador que solo requieren un token de ID, se recomienda usar la concesión Authorization Code con [PKCE](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce). Para obtener información detallada, consulta [OAuth2 Implicit Grant and SPA](https://auth0.com/blog/oauth2-implicit-grant-and-spa/) en el blog de Auth0.

  Si necesitas un [token de actualización](/docs/es/secure/tokens/refresh-tokens) para poder obtener un nuevo token de acceso o un token de ID sin tener que volver a autenticar al usuario, debes usar la [concesión Authorization Code](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow/call-your-api-using-the-authorization-code-flow).
</Warning>

<div id="attack-protection">
  ## Protección contra ataques
</div>

La razón por la que los sistemas de autenticación son importantes es que evitan que <Tooltip tip="Actores maliciosos: entidad (una persona o grupo) que representa una amenaza para la empresa o el entorno con la intención de causar daño." cta="Ver glosario" href="/docs/es/glossary?term=bad+actors">actores maliciosos</Tooltip> accedan a aplicaciones y datos de usuario a los que no deberían poder acceder. Queremos interponer tantas barreras como sea posible entre esos actores maliciosos y el acceso a nuestros sistemas. Una de las formas más sencillas de lograrlo es asegurarte de que tu [protección contra ataques](/docs/es/secure/attack-protection) con Auth0 esté configurada correctamente, así que dedica un momento a leer la guía sobre este tema y a comprobar que todo funcione correctamente para ti.

<Info>
  ### Práctica recomendada

  Auth0 gestiona la detección de anomalías entre bastidores, y esta ofrece una excelente funcionalidad de seguridad para tu producto. Si vas a utilizarla, asegúrate de haber configurado tu [proveedor de correo electrónico](/docs/es/get-started/architecture-scenarios/business-to-consumer/operations#email-provider-setup) y tus [Plantillas de correo electrónico](/docs/es/get-started/architecture-scenarios/business-to-consumer/branding#email-template-customization) antes de activar la entrega de correos electrónicos a tus usuarios.
</Info>

<div id="sso-with-legacy-systems">
  ## SSO con sistemas heredados
</div>

En una reestructuración a gran escala, no siempre es posible —o práctico— actualizar todas tus aplicaciones a la vez. De hecho, nuestra práctica recomendada es planificar un enfoque iterativo a la hora de integrar con Auth0. Si tus aplicaciones ya participan en el inicio de sesión único (SSO) y tu sistema de identidad heredado admite protocolos como OIDC o <Tooltip tip="Security Assertion Markup Language (SAML): Protocolo estandarizado que permite a dos partes intercambiar información de autenticación sin una contraseña." cta="Ver glosario" href="/docs/es/glossary?term=SAML">SAML</Tooltip>, tienes un par de opciones si quieres seguir ofreciendo SSO mientras integras con Auth0:

* Actualiza tu proveedor de identidad existente en tu sistema de SSO heredado para que redirija a Auth0 para el inicio de sesión (p. ej., usando [SAML](/docs/es/authenticate/single-sign-on/outbound-single-sign-on/configure-auth0-saml-identity-provider)), o
* Configura Auth0 para que redirija a tu sistema de SSO heredado para iniciar sesión. Esto requiere configurar tu sistema heredado como un IdP en Auth0 (es decir, ya sea usando [SAML](/docs/es/authenticate/protocols/saml/saml-sso-integrations/configure-auth0-saml-service-provider) o [OIDC](/docs/es/authenticate/identity-providers/social-identity-providers)).

<Info>
  ### Práctica recomendada

  Mantener una experiencia de SSO con tu sistema heredado puede añadir complejidad, pero puede valer la pena para ofrecer una experiencia de usuario más fluida mientras integras con Auth0. Si tienes pensado seguir este camino, planificarlo desde el principio puede ayudar a garantizar que sea posible. Si aún no tienes SSO en un servicio centralizado, es poco probable que la complejidad de añadirlo compense sus beneficios.
</Info>

Este es un tema complejo que probablemente requiera investigación adicional según tu arquitectura heredada actual, y te recomendamos que solo lo analices si actualmente tu sistema heredado admite SSO. Nota: si actualmente rediriges desde tus aplicaciones a un sistema centralizado para autenticar a tus usuarios y ese sistema solo solicita credenciales si aún no tienes una sesión con el sistema centralizado, entonces cuentas con una implementación de SSO heredada.

<div id="social-authentication">
  ## Autenticación social
</div>

El escenario de “trae tu propia identidad” que ofrecen Facebook, Google, etc. es una excelente manera de simplificar la experiencia de autenticación de usuario sin comprometer la seguridad, y usar [Universal Login](#universal-login) facilita empezar a añadir compatibilidad con [Conexiones sociales](/docs/es/authenticate/identity-providers/social-identity-providers) con cambios mínimos.

<Warning>
  Auth0 ofrece una forma sencilla de probar conexiones sociales mediante [claves de desarrollador preconfiguradas](/docs/es/authenticate/identity-providers/social-identity-providers/devkeys). Sin embargo, estas tienen [limitaciones](/docs/es/authenticate/identity-providers/social-identity-providers/devkeys#limitations-of-developer-keys), y antes de pasar a producción tendrás que configurar tus propias claves específicas de la aplicación siguiendo las [instrucciones](/docs/es/authenticate/identity-providers/social-identity-providers) del proveedor o proveedores sociales que elijas.
</Warning>

Con compatibilidad [social](https://auth0.com/learn/social-login/), las identidades y credenciales de los usuarios las gestiona el proveedor social, al igual que ciertos claims de identidad, que Auth0 usará para completar el [perfil](/docs/es/get-started/architecture-scenarios/business-to-consumer/profile-management) del usuario. Auth0 también puede proporcionar acceso a los [tokens de acceso](/docs/es/secure/tokens/access-tokens) de los proveedores de identidad social (IdP sociales), para que tu aplicación también pueda llamar a las API de IdP sociales de terceros en nombre del usuario.

<Info>
  ### Práctica recomendada

  La autenticación social es una gran funcionalidad, pero cuando ofreces más de una forma de iniciar sesión, debes tener en cuenta la posibilidad de que tus clientes realmente usen varias. De forma predeterminada, cada identidad de usuario en Auth0 tiene su propio perfil de usuario, por lo que probablemente te interese considerar la capacidad de Auth0 para [vincular cuentas de usuario](/docs/es/manage-users/user-accounts/user-account-linking) y así ofrecer una forma eficaz de asociar un perfil de usuario con múltiples identidades.
</Info>

La [extensión Custom Social Connections](/docs/es/authenticate/identity-providers/social-identity-providers) de Auth0 amplía aún más la autenticación social al permitirte conectarte con cualquier proveedor externo compatible con [OpenID Connect (OIDC)](/docs/es/authenticate/protocols/openid-connect-protocol) que no sea compatible de forma nativa. Por ejemplo, la compatibilidad con el proveedor de identidad emitida por el gobierno [SwissID](https://www.swissid.ch/) se puede configurar en Auth0 mediante una conexión social personalizada.

<div id="multi-factor-authentication-mfa">
  ## Autenticación multifactor (MFA)
</div>

En una época en la que el uso indebido de las credenciales de los usuarios está en máximos históricos, proteger sus sistemas es todo un reto, ya que es muy común que los hackers roben información de identidad de los usuarios. Sin embargo, una de las formas más eficaces de hacerlo es dar a los usuarios la posibilidad de configurar un segundo factor para proteger su cuenta, más conocido como [autenticación multifactor](/docs/es/secure/multi-factor-authentication). Esto garantiza que solo un usuario legítimo pueda acceder a su cuenta, incluso si utiliza un nombre de usuario y contraseña que podrían haberse visto comprometidos en otra aplicación.

<Info>
  ### Práctica recomendada

  Es bastante habitual que las aplicaciones orientadas al cliente ofrezcan a los usuarios la *opción* de añadir un segundo factor, en lugar de *obligarlos* a usarlo. Para obtener más información sobre esto, consulte [cómo ofrecer a sus usuarios la opción de añadir MFA](https://auth0.com/learn/multifactor-authentication-customers/).
</Info>

Auth0 admite varias opciones a la hora de habilitar MFA para proteger el acceso a las cuentas de usuario, y existen varias prácticas para asegurarse de que realmente está proporcionando una barrera de acceso flexible basada en un segundo factor:

* Auth0 [Guardian](https://auth0.com/multifactor-authentication): un servicio que proporciona tanto la generación de notificaciones Push como una aplicación para permitir o denegar solicitudes. Push envía notificaciones al dispositivo registrado previamente por el usuario —normalmente un móvil o una tableta—, desde el cual puede permitir o denegar inmediatamente el acceso a la cuenta con solo pulsar un botón.
* Time-based One-Time Password ([TOTP](https://auth0.com/blog/from-theory-to-practice-adding-two-factor-to-node-dot-js/)): le permite registrar un dispositivo —como Google Authenticator— que generará una contraseña de un solo uso que cambia con el tiempo y que puede introducirse como segundo factor para validar la cuenta de un usuario.
* SMS: para enviar un código de un solo uso por SMS que luego se pide al usuario que introduzca antes de poder completar la autenticación.
* Voz: para entregar un código de un solo uso mediante una llamada telefónica que luego se pide al usuario que introduzca antes de poder completar la autenticación.
* Duo: le permite usar su cuenta de Duo para la autenticación multifactor.
* Email: le permite usar su cuenta de correo electrónico para la autenticación multifactor.

Aunque el flujo de MFA que utiliza tecnologías como Guardian o Google Authenticator suele proporcionarse mediante una aplicación independiente que se ejecuta en un dispositivo móvil o tableta, si no quiere que sus clientes tengan que descargar una aplicación aparte, Auth0 también le proporciona un [SDK](https://auth0.com/blog/announcing-guardian-whitelabel-sdk/) que puede usar para crear flujos de segundo factor directamente en sus aplicaciones móviles existentes.
