description: Cómo funciona la autenticación en tu implementación de B2C IAM.
title: Autenticación (B2C)
---

Para ofrecer servicios a tus usuarios, debes poder identificar quiénes son. Este proceso se denomina autenticación de usuario. Existen varias formas de autenticar a un usuario: mediante cuentas de redes sociales, nombre de usuario y contraseña, <Tooltip tip="Passwordless: Forma de autenticación que no depende de una contraseña como primer factor." cta="Ver glosario" href="/docs/es/glossary?term=passwordless">passwordless</Tooltip>—y a menudo se recomienda ir más allá de un primer factor para autenticar al usuario habilitando la <Tooltip tip="Passwordless: Forma de autenticación que no depende de una contraseña como primer factor." cta="Ver glosario" href="/docs/es/glossary?term=multi-factor+authentication">autenticación multifactor</Tooltip> (MFA).

<Info>
  ### Práctica recomendada

  Es importante tener en cuenta tanto la seguridad como la experiencia de usuario al diseñar cómo autenticarás a tus usuarios. Ofrecer múltiples factores primarios y/o exigir más de un factor durante la autenticación son formas de lograr ambas.
</Info>

Hay varias cosas que querrás tener en cuenta al evaluar la funcionalidad y el flujo de trabajo:

* ¿Dónde introducirán los usuarios sus credenciales?
* ¿Cómo mantendrás seguras las credenciales de los usuarios?
* ¿Cómo mantendrás tu sistema de autenticación?
* ¿Cómo puedes ofrecer autenticación con contraseña a tus usuarios?
* ¿Cómo puedes evitar que los hackers intenten iniciar sesión haciéndose pasar por tus usuarios?
* ¿Cómo implementarás la autenticación en distintos tipos de aplicaciones?
* ¿Cómo puedes facilitar el inicio de sesión a tus usuarios cuando tienen diferentes idiomas?
* ¿Cómo ofrecerás una buena experiencia de usuario a medida que abandonas cualquier sistema de autenticación heredado?
* ¿Qué debes tener en cuenta al integrar aplicaciones con Auth0?
* ¿Pueden los usuarios iniciar sesión con sus cuentas sociales existentes (por ejemplo, Facebook o Google)?
* ¿Necesitas ofrecer autenticación multifactor?
* ¿Qué haces si tienes un servicio que no ofrece una forma de que el usuario inicie sesión de antemano?
* ¿Puedes pasar el mismo <Tooltip tip="token de acceso: Credencial de autorización, en forma de una cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=access+token">token de acceso</Tooltip> de usuario de una API a otra?

Auth0 <Accordion title="Universal Login">[Universal Login](#universal-login)</Accordion> ofrece a los usuarios una experiencia segura, independientemente de si eliges ofrecer inicio de sesión con credenciales de ID de usuario/contraseña o permitir los llamados escenarios de trae tu propia identidad mediante <Accordion title="Social Login">[Social Login](https://auth0.com/learn/social-login/)</Accordion>. También aporta beneficios de reconocimiento de marca centralizar la experiencia de inicio de sesión con <Tooltip tip="Universal Login: Tu aplicación redirige a Universal Login, alojado en el servidor de autorización de Auth0, para verificar la identidad de un usuario." cta="Ver glosario" href="/docs/es/glossary?term=Universal+Login">Universal Login</Tooltip>, incluso si consideras que también tendrás requisitos de [branding](/docs/es/get-started/architecture-scenarios/business-to-consumer/branding) específicos del producto. Los widgets de interfaz de usuario de Auth0 que normalmente se usan con Universal Login también ofrecen soporte listo para usar en materia de [internacionalización](/docs/es/customize/internationalization-and-localization/universal-login-internationalization) para usuarios con distintos requisitos de idioma, y el soporte integrado para funcionalidades de Auth0 como [MFA](#multi-factor-authentication-mfa-) y [protección contra ataques](#anomaly-detection) te permite implementar barreras para evitar que los hackers intenten acceder a las cuentas de los usuarios.

Permitir que los usuarios inicien sesión con credenciales de ID de usuario/contraseña significa que no dependes del estado de proveedores de identidad de terceros para que tus usuarios accedan a tu sistema. Además, puedes exigir que las credenciales utilizadas se ajusten a tus políticas corporativas. Auth0 te ayuda con esto al ofrecerte múltiples opciones para respaldar los inicios de sesión con ID de usuario/contraseña, y la [guía proporcionada](#username-and-password-authentication) te ayudará a comprender cómo puedes aprovechar estas opciones. Agregar soporte [social](#social-authentication) en algún momento, como un factor primario de autenticación adicional, te da más flexibilidad y puede ayudarte a comprender mejor a tus usuarios sin necesidad de hacerles más preguntas, aprovechando la información que ya almacenan los distintos [providers](/docs/es/authenticate/identity-providers/social-identity-providers) de inicio de sesión social.

Si ya cuenta con un legacy identity store, también le convendrá consultar [Migración de usuarios](/docs/es/get-started/architecture-scenarios/business-to-consumer/provisioning). En esta sección se analizan las ventajas de migrar al almacenamiento de identidades gestionado por Auth0 en términos de protección y seguridad.

Para las aplicaciones de cara al cliente, <Tooltip tip="OpenID: Estándar abierto de autenticación que permite a las aplicaciones verificar las identidades de los usuarios sin recopilar ni almacenar información de inicio de sesión." cta="Ver glosario" href="/docs/es/glossary?term=OpenID">OpenID</Tooltip> Connect ([OIDC](/docs/es/authenticate/protocols/openid-connect-protocol)) es el protocolo estándar del sector más utilizado, y OIDC cuenta con compatibilidad de primer nivel en Auth0. Auth0 ofrece distintos enfoques para integrar diferentes aplicaciones, por lo que le convendrá consultar la sección sobre [integración de aplicaciones](#application-integration) para obtener la información necesaria y tomar una decisión fundamentada.

Al llamar a una API desde otra API, o en cualquier situación en la que no haya contexto de usuario autenticado, como una o más tareas cron, generadores de informes o sistemas de integración/entrega continuas, necesitará una forma de autorizar la aplicación en lugar de un usuario. Se trata de un proceso de un solo paso en el que la aplicación se autentica (mediante un <Tooltip tip="Client ID: Valor de identificación dado por Auth0 a su recurso registrado." cta="Ver glosario" href="/docs/es/glossary?term=client+ID">ID de cliente</Tooltip> y un secreto) y luego se autoriza en una sola llamada. Puede obtener más información al respecto en nuestra sección de autorización, en [autorización de máquina a máquina (m2m)](/docs/es/get-started/architecture-scenarios/business-to-consumer/authorization).

<div id="universal-login">
  ## Universal Login
</div>

¿Tiene o tendrá más de una aplicación en su sistema? Si la respuesta es sí, le convendrá contar con una experiencia de inicio de sesión centralizada. Para lograr una experiencia fluida de <Tooltip tip="inicio de sesión único (SSO): servicio que, después de que un usuario inicia sesión en una aplicación, inicia automáticamente su sesión en otras aplicaciones." cta="Ver glosario" href="/docs/es/glossary?term=Single+Sign-on">inicio de sesión único</Tooltip> (SSO) entre varias aplicaciones, es fundamental tener un punto centralizado al que redirigir a sus usuarios para la autenticación. Esto le permite ofrecer una experiencia coherente a sus usuarios si en el futuro agrega autenticación social, incorpora aplicaciones de terceros a su sistema o añade autenticación multifactor como opción (o requisito), y también aprovechar nuevas funcionalidades para mejorar su experiencia con poco o ningún esfuerzo adicional de desarrollo.

<Info>
  ### Práctica recomendada

  Si tiene más de una aplicación, la práctica recomendada es redirigir a un [punto centralizado](/docs/es/authenticate/login/auth0-universal-login) para autenticar al usuario. Con Auth0, esto significa aprovechar [Universal Login](/docs/es/authenticate/login/auth0-universal-login), que proporciona muchas ventajas de seguridad y experiencia de usuario listas para usar, incluido [SSO](/docs/es/authenticate/single-sign-on).
</Info>

Auth0 Universal Login hace que autenticar usuarios sea un proceso breve y sencillo que puede completarse en tres pasos fáciles (todos nuestros Quickstarts lo demuestran y nuestros SDKs también le ocultan la complejidad):

1. Determine cómo y cuándo quiere [redirigir desde su aplicación](#application-integration).
2. Configure el [branding](/docs/es/get-started/architecture-scenarios/business-to-consumer/branding) adecuado y/o HTML personalizado en su configuración de Auth0.
3. Configure su aplicación para [recibir y gestionar la respuesta](#application-integration) del servidor de autorización.

<div id="username-and-password-authentication">
  ## Autenticación con nombre de usuario y contraseña
</div>

Casi todas las aplicaciones B2C permiten a sus clientes crear un nuevo conjunto de credenciales. Esta es una forma común de autenticación que resulta familiar para todos los usuarios.

En Auth0, la autenticación con nombre de usuario y contraseña puede implementarse de varias maneras. Si tu aplicación es nueva y no tiene una base de usuarios existente, una [conexión de base de datos](/docs/es/authenticate/database-connections) preconfigurada de Auth0 te dará todo lo que necesitas para empezar a autenticar a tus usuarios. Sin embargo, si tienes un repositorio de usuarios heredado, como tu propia base de datos de usuarios o un sistema LDAP existente, tienes un par de opciones para migrar tus usuarios, como se explica en nuestra guía sobre [Migración de usuarios](/docs/es/get-started/architecture-scenarios/business-to-consumer/provisioning).

Independientemente de cómo acabes aprovisionando a los usuarios en tu conexión de base de datos, la autenticación de esos usuarios es bastante similar. Para ello, debes mostrarles un formulario en el que introduzcan su nombre de usuario y contraseña. Como se menciona en la guía sobre [Universal Login](#universal-login), la forma más sencilla y segura de autenticar a los usuarios con nombre de usuario y contraseña es redirigirlos a una página de inicio de sesión centralizada y recopilar allí esas credenciales. Esto permite a Auth0 determinar si ya se han autenticado y omitir por completo el formulario de inicio de sesión cuando no sea necesario.

<Info>
  ### Práctica recomendada

  Recopilar credenciales únicamente en la página de inicio de sesión centralizada reducirá la exposición a posibles filtraciones de secretos de usuario. También evitará tener que recopilar credenciales innecesariamente. Consulta [Universal Login](#universal-login) para obtener más información.
</Info>

<div id="application-integration">
  ## Integración de aplicaciones
</div>

Una vez que hayas definido cómo quieres autenticar a tus usuarios, el siguiente paso es determinar cómo iniciarás esa autenticación. Normalmente, cada aplicación tendrá su propio punto de partida.

<Warning>
  Las aplicaciones móviles nativas (y las aplicaciones de escritorio) deben usar el navegador del sistema para la autenticación; de lo contrario, quedan expuestas a riesgos de seguridad adicionales. Consulta [Native Login](/docs/es/authenticate/login/native-login) para obtener más información.
</Warning>

Como ya comentamos, hemos comprobado que la mayoría de nuestros clientes usan [OpenID Connect (OIDC)](/docs/es/authenticate/protocols/openid-connect-protocol) como protocolo estándar del sector para sus aplicaciones orientadas al cliente. Determinar qué flujo de OIDC usar es tu primera tarea, y te conviene empezar por revisar nuestra guía sobre [mapeo de concesiones](/docs/es/get-started/applications/application-grant-types).

Si quieres permitir que usuarios anónimos accedan a alguna parte de nuestra aplicación, debes determinar si los redirigirás de inmediato o solo cuando sea necesario (o quizá una combinación de ambas opciones; consulta [Redirect Users After Login](/docs/es/authenticate/login/redirect-users-after-login) para obtener más información). Si los usuarios pueden acceder mediante [enlaces profundos](#deep-linking-to-protected-endpoints) a una versión protegida (o a un área protegida) de tu sitio, tendrás que determinar qué enlaces a tu aplicación provocarán una redirección automática a Auth0.

<div id="anonymous-access">
  ### Acceso anónimo
</div>

Es importante tener en cuenta la experiencia del usuario cuando alguien llega por primera vez a su aplicación. Si su aplicación admite acceso de usuarios anónimos (algo bastante común en las aplicaciones de comercio electrónico), hay distintos escenarios que considerar:

* Si está volviendo a la aplicación después de haber iniciado sesión anteriormente, o
* Si es la primera vez que accede a la aplicación:

  * Si ya accedió a otra aplicación que usa el mismo tenant de Auth0,
  * Si alguna vez (o quizás no desde hace mucho tiempo) se autenticó en este dispositivo o navegador.

Cuando un usuario anónimo accede a su aplicación, a menudo conviene que la aplicación detecte si el usuario ya inició sesión en otra aplicación de la misma familia, o que recuerde a este usuario aunque la aplicación sea una [SPA](/docs/es/quickstart/spa) sin estado. Por ejemplo, si puede determinar que el usuario ya inició sesión, podría decidir que el header de la UI de la aplicación no muestre un botón de inicio de sesión y que, en su lugar, muestre un menú de cuenta o perfil para el usuario. Para lograrlo, le convendrá utilizar la &quot;[autenticación silenciosa](/docs/es/authenticate/login/configure-silent-authentication)&quot;. La autenticación silenciosa le permitirá comprobar si el usuario inició sesión sin pedirle que lo haga si no es así. Entonces, la aplicación puede mostrar un botón de inicio de sesión si es necesario. Sin embargo, si el usuario ya inició sesión, recibirá tokens y no tendrá que volver a mostrarle un botón de inicio de sesión.

<Warning>
  Comprobar si hay una sesión iniciada redirigiendo a Auth0 puede ser útil para su aplicación, pero si esto generará muchas solicitudes, debe emplear un mecanismo de limitación para evitar latencia y/o límite de frecuencia.

  Las llamadas a la Management API están sujetas a la [política de límite de frecuencia de Auth0](/docs/es/troubleshoot/customer-support/operational-policies/rate-limit-policy). Debe tener esto en cuenta y, como ayuda, Auth0 generalmente recomienda usar el [SDK de Auth0](/docs/es/libraries) adecuado para su entorno de desarrollo en lugar de llamar directamente a nuestras API.
</Warning>

<div id="deep-linking-to-protected-endpoints">
  ### Enlaces directos a endpoints protegidos
</div>

Hay varios motivos por los que alguien podría enlazar directamente a una página concreta dentro de tu aplicación a la que solo pueden acceder los usuarios autenticados. Si esto es posible en tu aplicación, debes redirigir automáticamente al usuario a Auth0 si no está autenticado. Una vez que se autentique y el <Tooltip tip="Servidor de autorización: servidor centralizado que contribuye a definir los límites del acceso de un usuario. Por ejemplo, tu servidor de autorización puede controlar los datos, las tareas y las funcionalidades disponibles para un usuario." cta="Ver glosario" href="/docs/es/glossary?term=authorization+server">servidor de autorización</Tooltip> lo devuelva a tu aplicación, puedes [redirigirlo](/docs/es/authenticate/login/redirect-users-after-login) al lugar al que quería ir desde el principio.

<div id="best-practice">
  ### Práctica recomendada
</div>

La mayoría de los frameworks modernos de authentication admiten middleware para redirigir a un servidor de autorización como Auth0. Al elegir uno, ten en cuenta estas consideraciones clave:

* Compatibilidad con clientes confidenciales, clientes no confidenciales o ambos
* Compatibilidad con la configuración mediante [endpoint de descubrimiento](/docs/es/get-started/applications/configure-applications-with-oidc-discovery "Configurar Applications con OIDC Discovery") o de forma explícita e inline
* Compatibilidad con la validación de tokens, incluida la expiración, las firmas, los claims y los scopes
* Compatibilidad con <Tooltip tip="token de actualización: token usado para obtener un token de acceso renovado sin obligar a los usuarios a iniciar sesión de nuevo." cta="Ver glosario" href="/docs/es/glossary?term=Refresh+Tokens">tokens de actualización</Tooltip> si es necesario

<div id="authenticating-the-user">
  ### Autenticación del usuario
</div>

La autenticación es el proceso de determinar la identidad del usuario. El resultado de la autenticación en un contexto de OIDC es un <Tooltip tip="token de ID: credencial destinada al propio cliente, y no a acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=ID+Token">token de ID</Tooltip>. Este token contiene información sobre el usuario y solo debe poder obtenerse si el usuario se autentica mediante uno o más factores, según lo definido por el servidor de autorización (la forma más común es la [autenticación con ID de usuario y contraseña](#username-and-password-authentication)). Además de obtener un token de ID, puede que también debas tener en cuenta lo siguiente:

* ¿También necesitamos un [token de acceso](/docs/es/secure/tokens/access-tokens) para llamar a una API compartida?
* ¿Tu aplicación es una aplicación de página única y solo requiere un [token de ID](/docs/es/secure/tokens/id-tokens)? Consulta [concesión Authorization Code con PKCE](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce/call-your-api-using-the-authorization-code-flow-with-pkce) para obtener más información.
* ¿Tu aplicación es una aplicación nativa (móvil o de escritorio) y/o necesitas un [token de actualización](/docs/es/secure/tokens/refresh-tokens)? Consulta [concesión Authorization Code con PKCE](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce/call-your-api-using-the-authorization-code-flow-with-pkce) para obtener más información.

<Warning>
  Antes de pasar a producción, debes asegurarte de que **solo** estén habilitadas las concesiones que usas para cada aplicación en la [configuración de tu aplicación](/docs/es/get-started/applications/update-grant-types).
</Warning>

<div id="authorization-code-grant-with-or-without-pkce">
  ### Concesión Authorization Code (con o sin PKCE)
</div>

Si su SDK solo admite la concesión Authorization Code, o necesita un token de acceso o un <Tooltip tip="Token de actualización: token usado para obtener un token de acceso renovado sin obligar a los usuarios a volver a iniciar sesión." cta="Ver glosario" href="/docs/es/glossary?term=Refresh+Token">token de actualización</Tooltip>, la concesión Authorization Code (con o sin [PKCE](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce)) también puede usarse para obtener un token de ID. La concesión Authorization Code incluye una llamada a la API adicional para intercambiar el código por un token, lo que puede generar una latencia adicional innecesaria si lo único que necesita es el token de ID. En muchos casos, se implementa el [flujo híbrido](/docs/es/get-started/authentication-and-authorization-flow/hybrid-flow) para proporcionar un acceso óptimo al token de ID y, al mismo tiempo, aprovechar el flujo de trabajo de la concesión Authorization Code para obtener de forma segura tokens de acceso y tokens de actualización.

<Warning>
  Aunque Auth0 admite el uso de la concesión implícita para aplicaciones basadas en navegador que solo requieren un token de ID, se recomienda usar la concesión Authorization Code con [PKCE](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce). Para obtener información detallada, consulte [OAuth2 Implicit Grant and SPA](https://auth0.com/blog/oauth2-implicit-grant-and-spa/) en el blog de Auth0.

  Si necesita un [token de actualización](/docs/es/secure/tokens/refresh-tokens) para poder obtener un nuevo token de acceso o token de ID sin tener que volver a autenticar al usuario, debe usar la [concesión Authorization Code](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow/call-your-api-using-the-authorization-code-flow).
</Warning>

<div id="attack-protection">
  ## Protección contra ataques
</div>

La razón por la que los sistemas de autenticación son importantes es evitar que los <Tooltip tip="Actores malintencionados: Entidad (una persona o grupo) que representa una amenaza para la empresa o el entorno con la intención de causar daño." cta="Ver glosario" href="/docs/es/glossary?term=bad+actors">actores malintencionados</Tooltip> accedan a aplicaciones y datos de usuario a los que no deberían tener acceso. Queremos poner tantas barreras como sea posible entre esos actores malintencionados y el acceso a nuestros sistemas. Una de las formas más sencillas de hacerlo es asegurarte de que tu [protección contra ataques](/docs/es/secure/attack-protection) con Auth0 esté configurada correctamente, así que dedica un momento a leer la guía sobre este tema y a comprobar que esté funcionando correctamente para ti.

<Info>
  ### Práctica recomendada

  Auth0 gestiona la detección de anomalías entre bastidores y ofrece una excelente funcionalidad de seguridad para tu producto. Si vas a utilizarla, asegúrate de haber configurado tu [Proveedor de correo electrónico](/docs/es/get-started/architecture-scenarios/business-to-consumer/operations#email-provider-setup) y tus [Plantillas de correo electrónico](/docs/es/get-started/architecture-scenarios/business-to-consumer/branding#email-template-customization) antes de activar la entrega de correos electrónicos a tus usuarios.
</Info>

<div id="sso-with-legacy-systems">
  ## SSO con sistemas heredados
</div>

En una reestructuración a gran escala, no siempre es posible —o práctico— actualizar todas tus aplicaciones a la vez. De hecho, nuestra práctica recomendada es planificar un enfoque iterativo a la hora de integrarte con Auth0. Si tus aplicaciones ya usan inicio de sesión único (SSO) y tu sistema de identidad heredado admite protocolos como OIDC o <Tooltip tip="Security Assertion Markup Language (SAML): Protocolo estandarizado que permite a dos partes intercambiar información de autenticación sin una contraseña." cta="Ver glosario" href="/docs/es/glossary?term=SAML">SAML</Tooltip>, tienes un par de opciones si quieres seguir ofreciendo SSO mientras te integras con Auth0:

* Actualizar tu proveedor de identidad actual en tu sistema SSO heredado para que redirija a Auth0 para el inicio de sesión (p. ej., usando [SAML](/docs/es/authenticate/single-sign-on/outbound-single-sign-on/configure-auth0-saml-identity-provider)), o
* Hacer que Auth0 redirija a tu sistema SSO heredado para iniciar sesión. Para ello, debes configurar tu sistema heredado como un IdP en Auth0 (es decir, usando [SAML](/docs/es/authenticate/protocols/saml/saml-sso-integrations/configure-auth0-saml-service-provider) o [OIDC](/docs/es/authenticate/identity-providers/social-identity-providers)).

<Info>
  ### Práctica recomendada

  Ofrecer una experiencia de SSO con tu sistema heredado puede añadir complejidad, pero puede valer la pena para brindar una experiencia de usuario más fluida mientras te integras con Auth0. Si piensas seguir este camino, planificarlo desde el principio puede ayudar a garantizar que sea viable. Si aún no tienes SSO en un servicio centralizado, es poco probable que la complejidad de añadirlo compense los beneficios.
</Info>

Este es un tema complejo que probablemente requerirá investigación adicional según tu arquitectura heredada actual, y te recomendamos que solo lo explores si tu sistema heredado ya admite SSO. Nota: si actualmente rediriges a tus usuarios desde tus aplicaciones a un sistema centralizado para autenticarlos, y ese sistema solo solicita credenciales si aún no tienes una sesión con él, entonces tienes una implementación de SSO heredada.

<div id="social-authentication">
  ## Autenticación social
</div>

El escenario de “trae tu propia identidad” que ofrecen Facebook, Google, etc. es una excelente manera de simplificar la experiencia de autenticación de usuario sin comprometer la seguridad, y usar [Universal Login](#universal-login) facilita empezar a añadir compatibilidad con [Social Connections](/docs/es/authenticate/identity-providers/social-identity-providers) con cambios mínimos.

<Warning>
  Auth0 ofrece una forma sencilla de probar las conexiones sociales con [claves de desarrollador preconfiguradas](/docs/es/authenticate/identity-providers/social-identity-providers/devkeys). Sin embargo, estas tienen [limitaciones](/docs/es/authenticate/identity-providers/social-identity-providers/devkeys#limitations-of-developer-keys), y antes de entrar en producción, tendrás que configurar tus propias claves específicas de la aplicación siguiendo las [instrucciones](/docs/es/authenticate/identity-providers/social-identity-providers) para los proveedores sociales que elijas.
</Warning>

Con la compatibilidad con [inicio de sesión social](https://auth0.com/learn/social-login/), las identidades y credenciales de los usuarios son gestionadas por el proveedor social, al igual que ciertos claims de identidad, que Auth0 usará para completar el [perfil](/docs/es/get-started/architecture-scenarios/business-to-consumer/profile-management) del usuario. Auth0 también puede proporcionar acceso a los [tokens de acceso](/docs/es/secure/tokens/access-tokens) de los proveedores de identidad social (IdP sociales), para que tu aplicación también pueda llamar a las API de IdP sociales de terceros en nombre del usuario.

<Info>
  ### Práctica recomendada

  La autenticación social es una funcionalidad muy útil, pero cuando ofreces más de una forma de iniciar sesión, debes considerar la posibilidad de que tus clientes realmente utilicen más de una. De forma predeterminada, cada identidad de usuario en Auth0 tiene su propio perfil de usuario, por lo que probablemente te interese considerar la capacidad de Auth0 para [vincular cuentas de usuario](/docs/es/manage-users/user-accounts/user-account-linking) a fin de proporcionar una forma eficaz de asociar un perfil de usuario con múltiples identidades.
</Info>

La [extensión Custom Social Connections de Auth0](/docs/es/authenticate/identity-providers/social-identity-providers) amplía aún más la autenticación social al permitirte conectarte con cualquier proveedor externo compatible con [OpenID Connect (OIDC)](/docs/es/authenticate/protocols/openid-connect-protocol) que no tenga soporte listo para usar. Por ejemplo, la compatibilidad con el proveedor de identidad gubernamental [SwissID](https://www.swissid.ch/) se puede configurar en Auth0 mediante una Custom Social Connection.

<div id="multi-factor-authentication-mfa">
  ## Autenticación multifactor (MFA)
</div>

En una época en la que el uso indebido de las credenciales de los usuarios ha alcanzado niveles sin precedentes, proteger sus sistemas supone un desafío, especialmente cuando es tan habitual que los atacantes roben información de identidad de los usuarios. Sin embargo, una de las formas más eficaces de hacerlo es ofrecer a los usuarios la posibilidad de configurar un segundo factor para proteger su cuenta. Esto se conoce más comúnmente como [Autenticación multifactor](/docs/es/secure/multi-factor-authentication). Esto garantizará que solo un usuario legítimo pueda acceder a su cuenta, incluso si utiliza un nombre de usuario y una contraseña que puedan haberse visto comprometidos en otra aplicación.

<Info>
  ### Práctica recomendada

  Es bastante habitual que las aplicaciones orientadas al cliente ofrezcan a los usuarios la *opción* de añadir un segundo factor, en lugar de *obligarlos* a utilizarlo. Para obtener más información al respecto, consulte [ofrecer a sus usuarios la opción de agregar MFA](https://auth0.com/learn/multifactor-authentication-customers/).
</Info>

Auth0 admite varias opciones para habilitar MFA y proteger el acceso a las cuentas de usuario, y existen diversas prácticas que le ayudarán a ofrecer una barrera de acceso flexible basada en un segundo factor:

* Auth0 [Guardian](https://auth0.com/multifactor-authentication): un servicio que ofrece tanto la generación de notificaciones Push como una aplicación para permitir o denegar solicitudes. Push envía notificaciones al dispositivo preregistrado del usuario —normalmente un móvil o una tableta—, desde el cual puede permitir o denegar de inmediato el acceso a la cuenta con solo pulsar un botón.
* Time-based One-Time Password ([TOTP](https://auth0.com/blog/from-theory-to-practice-adding-two-factor-to-node-dot-js/)): le permite registrar un dispositivo —como Google Authenticator— que generará una contraseña de un solo uso que cambia con el tiempo y que puede introducirse como segundo factor para validar la cuenta de un usuario.
* SMS: para enviar un código de un solo uso por SMS que luego se pedirá al usuario que introduzca antes de poder completar la autenticación.
* Voz: para enviar un código de un solo uso mediante una llamada telefónica que luego se pedirá al usuario que introduzca antes de poder completar la autenticación.
* Duo: le permite usar su cuenta de Duo para la autenticación multifactor.
* Email: le permite usar su cuenta de correo electrónico para la autenticación multifactor.

Aunque el flujo de MFA con tecnologías como Guardian o Google Authenticator suele proporcionarse mediante una aplicación independiente que se ejecuta en un dispositivo móvil o tableta, si no quiere que sus clientes tengan que descargar una aplicación aparte, Auth0 también le proporciona un [SDK](https://auth0.com/blog/announcing-guardian-whitelabel-sdk/) que puede usar para crear el flujo del segundo factor directamente en sus aplicaciones móviles existentes.
