description: Cómo funciona la autenticación en tu implementación de B2C IAM.
title: Autenticación (B2C)
---

Para ofrecer servicios a tus usuarios, debes poder identificar quiénes son. Este proceso se denomina autenticación de usuario. Existen diversas formas de autenticar a un usuario —mediante cuentas de redes sociales, nombre de usuario y contraseña, <Tooltip tip="Passwordless: Forma de autenticación que no depende de una contraseña como primer factor." cta="Ver glosario" href="/docs/es/glossary?term=passwordless">passwordless</Tooltip>— y, con frecuencia, se recomienda ir más allá del primer factor de autenticación habilitando la <Tooltip tip="Passwordless: Forma de autenticación que no depende de una contraseña como primer factor." cta="Ver glosario" href="/docs/es/glossary?term=multi-factor+authentication">autenticación multifactor</Tooltip> (MFA).

<Info>
  ### Práctica recomendada

  Es importante considerar tanto la seguridad como la experiencia de usuario al diseñar cómo autenticarás a tus usuarios. Ofrecer múltiples factores primarios y/o exigir más de un factor durante la autenticación son formas de lograr ambos objetivos.
</Info>

Hay varios aspectos que deberás tener en cuenta al evaluar la funcionalidad y el flujo de trabajo:

* ¿Dónde ingresarán sus credenciales los usuarios?
* ¿Cómo mantendrás seguras las credenciales de los usuarios?
* ¿Cómo mantendrás tu sistema de autenticación?
* ¿Cómo puedes ofrecer autenticación por contraseña a tus usuarios?
* ¿Cómo puedes evitar que los hackers intenten iniciar sesión como tus usuarios?
* ¿Cómo implementarás la autenticación en distintos tipos de aplicaciones?
* ¿Cómo puedes facilitar el inicio de sesión a usuarios que provienen de diferentes contextos lingüísticos?
* ¿Cómo ofrecerás una buena experiencia de usuario al migrar desde un sistema de autenticación heredado?
* ¿Qué debes considerar al integrar aplicaciones con Auth0?
* ¿Pueden los usuarios iniciar sesión con sus cuentas sociales existentes (por ejemplo, Facebook o Google)?
* ¿Necesitas ofrecer autenticación multifactor?
* ¿Qué haces si tienes un servicio que no permite al usuario iniciar sesión de antemano?
* ¿Puedes pasar el mismo <Tooltip tip="Token de acceso: Credencial de autorización, en forma de cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=access+token">token de acceso</Tooltip> de usuario de una API a otra?

El [Universal Login](#universal-login) de Auth0 ofrece a los usuarios una experiencia segura y confiable, ya sea que elijas permitir el inicio de sesión con credenciales de ID de usuario/contraseña o habilitar los escenarios de trae tu propia identidad disponibles a través del [Social Login](https://auth0.com/learn/social-login/). Centralizar la experiencia de inicio de sesión con <Tooltip tip="Universal Login: Tu aplicación redirige al Universal Login, alojado en el servidor de autorización de Auth0, para verificar la identidad de un usuario." cta="Ver glosario" href="/docs/es/glossary?term=Universal+Login">Universal Login</Tooltip> también aporta beneficios en términos de reconocimiento de marca, incluso si prevés que tendrás requisitos de [branding](/docs/es/get-started/architecture-scenarios/business-to-consumer/branding) específicos para cada producto. Los widgets de interfaz de usuario de Auth0 que se utilizan habitualmente con Universal Login también ofrecen compatibilidad lista para usar con la [internacionalización](/docs/es/customize/internationalization-and-localization/universal-login-internationalization) para usuarios con distintos requisitos de idioma, y la compatibilidad lista para usar con funcionalidades de Auth0 como [MFA](#multi-factor-authentication-mfa-) y [protección contra ataques](#anomaly-detection) te permite establecer barreras para impedir que los hackers intenten acceder a las cuentas de tus usuarios.

Permitir que los usuarios inicien sesión con credenciales de ID de usuario/contraseña significa que no dependes del estado de los proveedores de identidad de <Tooltip tip="Proveedor de identidad (IdP): Servicio que almacena y gestiona identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=identity+providers">terceros</Tooltip> para que tus usuarios accedan a tu sistema. Además, tienes la posibilidad de exigir que las credenciales utilizadas se ajusten a tus políticas corporativas. Auth0 te ayuda con esto al ofrecerte múltiples opciones para los inicios de sesión con ID de usuario/contraseña, y la [orientación proporcionada](#username-and-password-authentication) te ayudará a entender cómo aprovechar estas opciones. Agregar compatibilidad con [redes sociales](#social-authentication) en algún momento, como factor de autenticación primario adicional, te brinda mayor flexibilidad y puede ayudarte a conocer mejor a tus usuarios sin necesidad de hacerles más preguntas, aprovechando la información ya almacenada por los distintos [proveedores](/docs/es/authenticate/identity-providers/social-identity-providers) de inicio de sesión social.

Si ya cuentas con un legacy identity store, también te convendrá consultar [Migración de usuarios](/docs/es/get-started/architecture-scenarios/business-to-consumer/provisioning). En esta sección se analizan las ventajas de migrar al almacenamiento de identidades administrado por Auth0 en términos de seguridad y protección.

Para las aplicaciones de cara al cliente, <Tooltip tip="OpenID: Estándar abierto de autenticación que permite a las aplicaciones verificar las identidades de los usuarios sin recopilar ni almacenar información de inicio de sesión." cta="Ver glosario" href="/docs/es/glossary?term=OpenID">OpenID</Tooltip> Connect ([OIDC](/docs/es/authenticate/protocols/openid-connect-protocol)) es el protocolo estándar del sector más utilizado, y OIDC cuenta con compatibilidad de primer nivel en Auth0. Auth0 ofrece distintos enfoques para integrar diferentes aplicaciones, por lo que te convendrá consultar la sección sobre [integración de la aplicación](#application-integration) para obtener la información necesaria y poder tomar una decisión informada.

Al llamar a una API desde otra API, o en cualquier situación en la que no haya contexto de usuario autenticado, como una o varias tareas cron, generadores de informes o sistemas de integración/entrega continuas, necesitarás una forma de autorizar la aplicación en lugar de a un usuario. Este es un proceso de un solo paso en el que la aplicación se autentica (mediante un <Tooltip tip="Client ID: Valor de identificación asignado a tu recurso registrado por Auth0." cta="Ver glosario" href="/docs/es/glossary?term=client+ID">ID de cliente</Tooltip> y un secreto) y luego se autoriza en una sola llamada. Puedes obtener más información sobre esto en nuestra sección de autorización, en [autorización de máquina a máquina (m2m)](/docs/es/get-started/architecture-scenarios/business-to-consumer/authorization).

<div id="universal-login">
  ## Universal Login
</div>

¿Tienes o tendrás más de una aplicación en tu sistema? Si la respuesta es sí, te convendrá contar con una experiencia de inicio de sesión centralizada. Para lograr una experiencia fluida de <Tooltip tip="Single Sign-On (SSO): servicio que, después de que un usuario inicia sesión en una aplicación, inicia automáticamente su sesión en otras aplicaciones." cta="Ver glosario" href="/docs/es/glossary?term=Single+Sign-on">inicio de sesión único</Tooltip> (SSO) entre varias aplicaciones, es fundamental contar con un lugar centralizado al que redirigir a tus usuarios para la autenticación. Esto te permite ofrecerles una experiencia coherente si en el futuro agregas autenticación social, incorporas aplicaciones de terceros a tu sistema o añades autenticación multifactor como opción (o requisito), y también aprovechar nuevas funcionalidades para mejorar su experiencia con poco o ningún esfuerzo adicional de desarrollo.

<Info>
  ### Práctica recomendada

  Si tienes más de una aplicación, la práctica recomendada es redirigir a una [ubicación centralizada](/docs/es/authenticate/login/auth0-universal-login) para autenticar al usuario. Con Auth0, esto significa aprovechar [Universal Login](/docs/es/authenticate/login/auth0-universal-login), que ofrece muchas ventajas de seguridad y de experiencia de usuario listas para usar, incluido [SSO](/docs/es/authenticate/single-sign-on).
</Info>

Auth0 Universal Login hace que autenticar a los usuarios sea un proceso breve y sencillo que puede completarse en tres simples pasos (todos nuestros Quickstarts lo muestran y nuestros SDKs también ocultan la complejidad por ti):

1. Determina cómo y cuándo quieres [redirigir desde tu aplicación](#application-integration).
2. Configura el [branding](/docs/es/get-started/architecture-scenarios/business-to-consumer/branding) adecuado y/o HTML personalizado en tu configuración de Auth0.
3. Configura tu aplicación para [recibir y gestionar la respuesta](#application-integration) del servidor de autorización.

<div id="username-and-password-authentication">
  ## Autenticación con nombre de usuario y contraseña
</div>

Casi todas las aplicaciones B2C permiten a sus clientes crear un nuevo conjunto de credenciales. Esta es una forma común de autenticación con la que todos los usuarios están familiarizados.

En Auth0, la autenticación con nombre de usuario y contraseña se presenta de varias formas. Si su aplicación es nueva y no tiene una base de usuarios existente, una simple [conexión de base de datos](/docs/es/authenticate/database-connections) lista para usar de Auth0 le dará todo lo que necesita para empezar a autenticar a sus usuarios. Sin embargo, si tiene un repositorio de usuarios heredado, como su propia base de datos de usuarios o un sistema LDAP existente, dispone de un par de opciones para migrar a sus usuarios, como se explica en nuestra guía sobre [Migración de usuarios](/docs/es/get-started/architecture-scenarios/business-to-consumer/provisioning).

Independientemente de cómo termine aprovisionando a los usuarios para su conexión de base de datos, la autenticación de esos usuarios es bastante similar. Para ello, debe mostrar a los usuarios un formulario en el que introduzcan su nombre de usuario y contraseña. Como se menciona en la guía sobre [Universal Login](#universal-login), la forma más sencilla y segura de autenticar a los usuarios con nombre de usuario y contraseña es redirigirlos a una página de inicio de sesión centralizada y recopilar allí sus credenciales. Esto permite que Auth0 determine si ya se han autenticado y omita por completo el formulario de inicio de sesión cuando no sea necesario.

<Info>
  ### Práctica recomendada

  Recopilar credenciales únicamente en la página de inicio de sesión centralizada reducirá la superficie de exposición ante posibles filtraciones de secretos de los usuarios. También reducirá la necesidad de recopilar credenciales de forma innecesaria. Consulte [Universal Login](#universal-login) para obtener más información.
</Info>

<div id="application-integration">
  ## Integración de la aplicación
</div>

Una vez que hayas decidido cómo quieres autenticar a tus usuarios, el siguiente paso es determinar cómo iniciarás esa autenticación. Normalmente, cada aplicación tendrá su propio punto de partida.

<Warning>
  Las aplicaciones móviles nativas (y las aplicaciones de escritorio) deben usar el navegador del sistema para la autenticación; de lo contrario, quedan expuestas a riesgos de seguridad adicionales. Consulta [Native Login](/docs/es/authenticate/login/native-login) para obtener más información.
</Warning>

Como ya comentamos, hemos comprobado que la mayoría de nuestros clientes usan [OpenID Connect (OIDC)](/docs/es/authenticate/protocols/openid-connect-protocol) como protocolo estándar del sector para sus aplicaciones de cara al cliente. Determinar qué flujo de OIDC usarás es tu primera tarea, y te conviene empezar por revisar nuestra guía sobre [mapeo de concesiones](/docs/es/get-started/applications/application-grant-types).

Si quieres permitir que usuarios anónimos accedan a alguna parte de tu aplicación, debes decidir si los redirigirás de inmediato o solo cuando sea necesario (o quizá alguna combinación de ambas opciones; consulta [Redirect Users After Login](/docs/es/authenticate/login/redirect-users-after-login) para más información). Si los usuarios pueden usar [enlaces profundos](#deep-linking-to-protected-endpoints) a una versión protegida (o a un área protegida) de tu sitio, tendrás que determinar qué enlaces a tu aplicación darán lugar a una redirección automática a Auth0.

<div id="anonymous-access">
  ### Acceso anónimo
</div>

Es importante tener en cuenta la experiencia del usuario cuando alguien llega por primera vez a su aplicación. Si su aplicación admite acceso de usuarios anónimos (algo bastante común en aplicaciones de comercio electrónico), hay distintos escenarios que considerar:

* ¿Vuelve a la aplicación después de haber iniciado sesión antes, o
* Si es la primera vez que accede a la aplicación:

  * ¿Ya ha accedido a otra aplicación que usa el mismo tenant de Auth0,
  * ¿Se ha autenticado alguna vez (o quizá no desde hace mucho tiempo) en este device o navegador?

Cuando un usuario anónimo accede a su aplicación, a menudo conviene que la aplicación determine si el usuario ya inició sesión en otra aplicación de la misma familia, o que recuerde a este usuario incluso si la aplicación es una [SPA](/docs/es/quickstart/spa) sin estado. Por ejemplo, si puede determinar que el usuario ya inició sesión, podría decidir que el encabezado de la interfaz de la aplicación no muestre un botón de inicio de sesión y muestre, en su lugar, un menú de cuenta o perfil para el usuario. Para lograrlo, le convendrá utilizar la &quot;[autenticación silenciosa](/docs/es/authenticate/login/configure-silent-authentication)&quot;. La autenticación silenciosa le permitirá comprobar si el usuario ha iniciado sesión sin pedirle que lo haga si no es así. Después, la aplicación puede mostrar un botón de inicio de sesión si es necesario. Sin embargo, si el usuario ya inició sesión, recibirá tokens y no tendrá que volver a mostrarle un botón de inicio de sesión.

<Warning>
  Comprobar si existe una sesión de inicio de sesión redirigiendo a Auth0 puede ser útil para su aplicación, pero si esto va a generar muchas solicitudes, debe emplear un mecanismo de limitación para evitar latencia y/o límites de frecuencia.

  Las llamadas a la Management API están sujetas a la [política de limitación de frecuencia de Auth0](/docs/es/troubleshoot/customer-support/operational-policies/rate-limit-policy). Debe tener esto en cuenta y, para ayudar, Auth0 generalmente recomienda usar el [SDK de Auth0](/docs/es/libraries) adecuado para su entorno de desarrollo en lugar de llamar directamente a nuestras API.
</Warning>

<div id="deep-linking-to-protected-endpoints">
  ### Enlaces profundos a endpoints protegidos
</div>

Hay varias razones por las que alguien podría enlazar directamente a una página específica dentro de tu aplicación a la que solo pueden acceder usuarios autenticados. Si tu aplicación permite esto, debes redirigir automáticamente al usuario a Auth0 si no está autenticado. Una vez que se autentique y el <Tooltip tip="Servidor de autorización: servidor centralizado que contribuye a definir los límites del acceso de un usuario. Por ejemplo, tu servidor de autorización puede controlar los datos, las tareas y las funcionalidades disponibles para un usuario." cta="Ver glosario" href="/docs/es/glossary?term=authorization+server">servidor de autorización</Tooltip> lo devuelva a tu aplicación, puedes [redirigirlo](/docs/es/authenticate/login/redirect-users-after-login) al lugar al que quería ir en primer lugar.

<div id="best-practice">
  ### Práctica recomendada
</div>

La mayoría de los frameworks modernos de authentication admiten middleware para redirigir a un servidor de autorización como Auth0. Al seleccionar uno, estas son algunas consideraciones clave:

* Compatibilidad con clientes confidenciales, clientes no confidenciales o ambos
* Compatibilidad con la configuración mediante [endpoint de descubrimiento](/docs/es/get-started/applications/configure-applications-with-oidc-discovery "Configurar Applications con OIDC Discovery") o de forma explícita en línea
* Compatibilidad con la validación de tokens, incluidas las expiraciones, las firmas, los claims y los scopes
* Compatibilidad con <Tooltip tip="Token de actualización: token usado para obtener un token de acceso renovado sin obligar a los usuarios a volver a iniciar sesión." cta="Ver glosario" href="/docs/es/glossary?term=Refresh+Tokens">tokens de actualización</Tooltip> si es necesario

<div id="authenticating-the-user">
  ### Autenticación del usuario
</div>

La autenticación es el proceso de determinar la identidad del usuario. El resultado de la autenticación en un contexto de OIDC es un <Tooltip tip="token de ID: credencial destinada al propio cliente, en lugar de para acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=ID+Token">token de ID</Tooltip>. Este token contiene información sobre el usuario y solo debería poder obtenerse si el usuario se autentica con uno o más factores, según lo definido por el servidor de autorización (la forma más habitual es el [ID de usuario y contraseña](#username-and-password-authentication)). Además de obtener un token de ID, hay algunos aspectos más que quizá debas tener en cuenta:

* ¿También necesitamos un [token de acceso](/docs/es/secure/tokens/access-tokens) para llamar a una API compartida?
* ¿Tu aplicación es una aplicación de página única y solo requiere un [token de ID](/docs/es/secure/tokens/id-tokens)? Consulta [Authorization Code Grant with PKCE](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce/call-your-api-using-the-authorization-code-flow-with-pkce) para obtener más información.
* ¿Tu aplicación es una aplicación nativa (móvil o de escritorio) y/o necesitas un [token de actualización](/docs/es/secure/tokens/refresh-tokens)? Consulta [Authorization Code Grant with PKCE](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce/call-your-api-using-the-authorization-code-flow-with-pkce) para obtener más información.

<Warning>
  Antes de pasar a producción, debes asegurarte de que **solo** estén habilitadas, para cada aplicación, las concesiones que estás usando en la [configuración de tu aplicación](/docs/es/get-started/applications/update-grant-types).
</Warning>

<div id="authorization-code-grant-with-or-without-pkce">
  ### Concesión Authorization Code (con o sin PKCE)
</div>

Si tu SDK solo admite la concesión Authorization Code, o necesitas un token de acceso o un <Tooltip tip="token de actualización: token que se usa para obtener un token de acceso renovado sin obligar a los usuarios a iniciar sesión de nuevo." cta="Ver glosario" href="/docs/es/glossary?term=Refresh+Token">token de actualización</Tooltip>, la concesión Authorization Code (con o sin [PKCE](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce)) también puede usarse para obtener un token de ID. La concesión Authorization Code incluye una llamada adicional a la API para intercambiar el código por un token, lo que puede generar una latencia adicional innecesaria si lo único que necesitas es el token de ID. En muchos casos, se implementa el [flujo híbrido](/docs/es/get-started/authentication-and-authorization-flow/hybrid-flow) para proporcionar un acceso óptimo al token de ID y, al mismo tiempo, seguir aprovechando el flujo de trabajo de la concesión Authorization Code para obtener de forma segura tokens de acceso y tokens de actualización.

<Warning>
  Aunque Auth0 admite el uso de la concesión implícita para aplicaciones basadas en navegador que solo requieren un token de ID, se recomienda usar la concesión Authorization Code con [PKCE](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce). Para obtener información detallada, consulta [OAuth2 Implicit Grant and SPA](https://auth0.com/blog/oauth2-implicit-grant-and-spa/) en el blog de Auth0.

  Si necesitas un [token de actualización](/docs/es/secure/tokens/refresh-tokens) para poder obtener un nuevo token de acceso o token de ID sin tener que volver a autenticar al usuario, debes usar la [concesión Authorization Code](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow/call-your-api-using-the-authorization-code-flow).
</Warning>

<div id="attack-protection">
  ## Protección contra ataques
</div>

Los sistemas de autenticación son importantes porque evitan que <Tooltip tip="Actores maliciosos: Entidad (una persona o grupo) que representa una amenaza para la empresa o el entorno con la intención de causar daño." cta="Ver glosario" href="/docs/es/glossary?term=bad+actors">actores maliciosos</Tooltip> accedan a aplicaciones y datos de usuario a los que no deberían tener acceso. Queremos poner tantas barreras como sea posible entre esos actores maliciosos y el acceso a nuestros sistemas. Una de las formas más sencillas de lograrlo es asegurarte de que la [protección contra ataques](/docs/es/secure/attack-protection) de Auth0 esté configurada correctamente, así que tómate un momento para leer la guía sobre este tema y comprobar que esté funcionando bien en tu caso.

<Info>
  ### Práctica recomendada

  Auth0 se encarga de la detección de anomalías en segundo plano, y esta ofrece una excelente funcionalidad de seguridad para tu producto. Si vas a utilizarla, asegúrate de haber configurado tu [Proveedor de correo electrónico](/docs/es/get-started/architecture-scenarios/business-to-consumer/operations#email-provider-setup) y tus [plantillas de correo electrónico](/docs/es/get-started/architecture-scenarios/business-to-consumer/branding#email-template-customization) antes de activar la entrega de correos electrónicos a tus usuarios.
</Info>

<div id="sso-with-legacy-systems">
  ## SSO con sistemas heredados
</div>

En una reestructuración a gran escala, no siempre es posible —o práctico— actualizar todas sus aplicaciones de una sola vez. De hecho, nuestra práctica recomendada es adoptar un enfoque iterativo a la hora de integrarse con Auth0. Si sus aplicaciones ya participan en el inicio de sesión único (SSO) y su sistema de identidad heredado admite protocolos como OIDC o <Tooltip tip="Security Assertion Markup Language (SAML): Protocolo estandarizado que permite a dos partes intercambiar información de autenticación sin una contraseña." cta="Ver glosario" href="/docs/es/glossary?term=SAML">SAML</Tooltip>, tiene un par de opciones si desea seguir ofreciendo SSO a medida que se integra con Auth0:

* Actualizar el proveedor de identidad existente en su sistema SSO heredado para que redirija a Auth0 para iniciar sesión (p. ej., mediante [SAML](/docs/es/authenticate/single-sign-on/outbound-single-sign-on/configure-auth0-saml-identity-provider)), o
* Hacer que Auth0 redirija a su sistema SSO heredado para iniciar sesión. Para ello, debe configurar su sistema heredado como un IdP en Auth0 (es decir, ya sea mediante [SAML](/docs/es/authenticate/protocols/saml/saml-sso-integrations/configure-auth0-saml-service-provider) o [OIDC](/docs/es/authenticate/identity-providers/social-identity-providers)).

<Info>
  ### Práctica recomendada

  Ofrecer una experiencia de SSO con su sistema heredado puede añadir complejidad, pero puede valer la pena para brindar una experiencia de usuario más fluida a medida que se integra con Auth0. Si tiene previsto seguir este camino, planificarlo desde el principio puede ayudar a garantizar que sea posible. Si aún no cuenta con SSO en un servicio centralizado, es poco probable que la complejidad de agregarlo justifique los beneficios.
</Info>

Este es un tema complejo que probablemente requiera una investigación adicional en función de su arquitectura heredada actual, y le recomendamos que solo lo considere si actualmente cuenta con soporte de SSO en su sistema heredado. Nota: si actualmente redirige sus aplicaciones a un sistema centralizado para autenticar a sus usuarios, y ese sistema solo solicita credenciales si aún no tiene una sesión con el sistema centralizado, entonces tiene una implementación de SSO heredada.

<div id="social-authentication">
  ## Autenticación social
</div>

El escenario de “trae tu propia identidad” que ofrecen Facebook, Google y otros es una forma muy útil de simplificar la experiencia de autenticación de usuario sin comprometer la seguridad, y usar [Universal Login](#universal-login) facilita empezar a añadir compatibilidad con [Social Connections](/docs/es/authenticate/identity-providers/social-identity-providers) con cambios mínimos.

<Warning>
  Auth0 ofrece una forma sencilla de probar las conexiones sociales mediante [claves de desarrollador preconfiguradas](/docs/es/authenticate/identity-providers/social-identity-providers/devkeys). Sin embargo, estas tienen [limitaciones](/docs/es/authenticate/identity-providers/social-identity-providers/devkeys#limitations-of-developer-keys), y antes de pasar a producción tendrás que configurar tus propias claves específicas de la aplicación siguiendo las [instrucciones](/docs/es/authenticate/identity-providers/social-identity-providers) del proveedor o los proveedores sociales que elijas.
</Warning>

Con la compatibilidad con [social](https://auth0.com/learn/social-login/), las identidades de usuario y las credenciales las gestiona el proveedor social, al igual que determinados claims de identidad, que Auth0 usará para completar el [perfil](/docs/es/get-started/architecture-scenarios/business-to-consumer/profile-management) del usuario. Auth0 también puede proporcionar acceso a los [Access Tokens](/docs/es/secure/tokens/access-tokens) de los proveedores de identidad social (IdP sociales), para que tu aplicación también pueda llamar a las API de proveedores de identidad social de terceros en nombre del usuario.

<Info>
  ### Práctica recomendada

  Social es una excelente funcionalidad, pero cuando ofreces más de una forma de iniciar sesión, debes tener en cuenta la posibilidad de que tus clientes realmente usen varias. De forma predeterminada, cada identidad de usuario en Auth0 tiene su propio perfil de usuario, así que probablemente te convenga considerar la capacidad de Auth0 para [vincular cuentas de usuario](/docs/es/manage-users/user-accounts/user-account-linking) y así ofrecer una forma eficaz de asociar un mismo perfil de usuario con varias identidades.
</Info>

La [extensión Custom Social Connections](/docs/es/authenticate/identity-providers/social-identity-providers) de Auth0 amplía aún más la autenticación social al permitirte conectarte con cualquier proveedor de terceros compatible con [OpenID Connect (OIDC)](/docs/es/authenticate/protocols/openid-connect-protocol) que no sea compatible de forma nativa. Por ejemplo, la compatibilidad con el proveedor de identidad gubernamental [SwissID](https://www.swissid.ch/) se puede configurar en Auth0 mediante una Custom Social Connection.

<div id="multi-factor-authentication-mfa">
  ## Autenticación multifactor (MFA)
</div>

En una época en la que el uso indebido de las credenciales de los usuarios está en su punto más alto, proteger sus sistemas es todo un desafío cuando, en general, es tan común que los atacantes roben la información de identidad de los usuarios. Aun así, una de las formas más eficaces es ofrecer a los usuarios la posibilidad de configurar un segundo factor para proteger su cuenta. Esto se conoce más comúnmente como [Autenticación multifactor](/docs/es/secure/multi-factor-authentication). Así se garantiza que solo un usuario legítimo pueda acceder a su cuenta, incluso si usa un nombre de usuario y una contraseña que podrían haberse visto comprometidos en otra aplicación.

<Info>
  ### Práctica recomendada

  Es bastante común que las aplicaciones orientadas al cliente ofrezcan a los usuarios la *opción* de añadir un segundo factor en lugar de *obligarlos* a usarlo. Para obtener más información al respecto, consulte [cómo ofrecer a sus usuarios la opción de añadir MFA](https://auth0.com/learn/multifactor-authentication-customers/).
</Info>

Auth0 admite varias opciones para habilitar MFA y proteger el acceso a las cuentas de usuario, y existen varias prácticas para asegurarse de ofrecer una barrera de acceso flexible basada en un segundo factor:

* Auth0 [Guardian](https://auth0.com/multifactor-authentication): un servicio que ofrece tanto la generación de notificaciones push como una aplicación para permitir o denegar solicitudes. push envía notificaciones al dispositivo previamente registrado del usuario —normalmente un móvil o una tableta— desde el que puede permitir o denegar de inmediato el acceso a la cuenta con solo pulsar un botón.
* One-Time Password basado en tiempo ([TOTP](https://auth0.com/blog/from-theory-to-practice-adding-two-factor-to-node-dot-js/)): le permite registrar un dispositivo —como Google Authenticator— que generará una contraseña de un solo uso que cambia con el tiempo y que puede introducirse como segundo factor para validar la cuenta de un usuario.
* SMS: para enviar un código de un solo uso por SMS que luego se le pedirá al usuario que introduzca antes de poder completar la autenticación.
* Voz: para entregar un código de un solo uso mediante una llamada telefónica que luego se le pedirá al usuario que introduzca antes de poder completar la autenticación.
* Duo: le permite usar su cuenta de Duo para la autenticación multifactor.
* Email: le permite usar su cuenta de correo electrónico para la autenticación multifactor.

Aunque el flujo de trabajo de MFA que utiliza tecnologías como Guardian o Google Authenticator suele ofrecerse mediante una aplicación independiente que se ejecuta en un dispositivo móvil o una tableta, si no quiere que sus clientes tengan que descargar una aplicación aparte, Auth0 también le proporciona un [SDK](https://auth0.com/blog/announcing-guardian-whitelabel-sdk/) que puede utilizar para crear el flujo de trabajo del segundo factor directamente en sus aplicaciones móviles existentes.
