---

Antes del lanzamiento, deberías haber completado todas las pruebas aplicables a tu entorno.

El aseguramiento de la calidad es fundamental para identificar problemas antes de que afecten a tus clientes y, según la naturaleza de tu proyecto, hay varios tipos de pruebas de calidad que conviene tener en cuenta como parte de tu integración con Auth0:

* ¿Tu aplicación es fácil de entender y usar, incluso para personas con alguna discapacidad?
* ¿Tu aplicación necesita funcionar en distintos navegadores y dispositivos?
* ¿Tu aplicación necesita funcionar en entornos multinacionales y/o internacionales?
* ¿Cómo se comportará tu aplicación cuando se vea sometida a cargas inesperadas en producción?
* ¿Cómo puedes asegurarte de que tu aplicación esté protegida frente a vulnerabilidades de seguridad?

Auth0 [Universal Login](/docs/es/authenticate/login/auth0-universal-login) y los widgets de interfaz de usuario asociados (como [Lock](/docs/es/libraries/lock)) ya se han diseñado y desarrollado siguiendo prácticas recomendadas de usabilidad y accesibilidad, y ofrecen soporte probado y listo para usar para una amplia variedad de [navegadores y dispositivos](/docs/es/troubleshoot/customer-support/product-support-matrix). El soporte para la [internacionalización](/docs/es/customize/internationalization-and-localization) (I18N) también se proporciona de forma inmediata, con extensibilidad integrada diseñada para casos personalizados de varios idiomas y localización (L10N).

Para garantizar que se cumplan los requisitos funcionales y que los eventos inesperados se gestionen correctamente, se ofrece orientación para probar la [integración](#integration-testing) entre tu(s) aplicación(es) y Auth0, y para las [pruebas unitarias](#unit-testing) de módulos individuales de extensibilidad (como [Rules](/docs/es/customize/rules/debug-rules), [Hooks](/docs/es/customize/hooks/update-hooks) y scripts de base de datos personalizada). También se ofrece orientación sobre la [política de pruebas de penetración](/docs/es/troubleshoot/customer-support/operational-policies/penetration-testing-policy) de Auth0 para ayudar con las pruebas de vulnerabilidades de seguridad, así como sobre cómo las pruebas [simuladas](#mock-testing) pueden aprovecharse junto con nuestra [política de pruebas de carga](/docs/es/troubleshoot/customer-support/operational-policies/load-testing-policy) para ayudar a garantizar que tu(s) aplicación(es) rindan bien ante cargas inesperadas.

<div id="unit-testing">
  ## Pruebas unitarias
</div>

El objetivo de las pruebas unitarias es probar unidades individuales de código. Si crea código personalizado dentro de Auth0 en forma de Rules, Hooks y/o scripts de Custom DB, debería plantearse usar un framework de pruebas (como [Mocha](https://mochajs.org/)) para probar su código. Las empresas que más éxito han tenido con Auth0 han descubierto que resulta útil ejecutar estas pruebas unitarias antes de [desplegar automáticamente](/docs/es/get-started/architecture-scenarios/business-to-consumer/deployment) la configuración y los recursos asociados del tenant de Auth0.

<div id="integration-testing">
  ## Pruebas de integración
</div>

Como práctica recomendada, configure tenants distintos para desarrollo, pruebas y producción, como se explica en la guía de arquitectura sobre [soporte para SDLC](/docs/es/get-started/architecture-scenarios/business-to-consumer/architecture#sdlc-support). Auth0 le permite configurar variables disponibles desde la [extensibilidad](/docs/es/customize/extensions) personalizada; pueden considerarse variables de entorno para su tenant de Auth0. En lugar de incluir de forma fija en el código referencias que cambian al mover código entre entornos de desarrollo, pruebas y producción, puede usar un nombre de variable configurado en el tenant y referenciado por el código de extensibilidad personalizada. Esto facilita que el mismo código personalizado funcione, sin cambios, en distintos tenants, ya que puede hacer referencia a variables que se rellenarán con valores específicos del tenant en tiempo de ejecución:

* Para usar variables en Rules, vea cómo [configurar valores](/docs/es/customize/rules/configuration)
* Para usar variables en Hooks, vea cómo configurar [secrets](/docs/es/customize/hooks/hook-secrets) en el editor
* Para usar variables en Actions, consulte Explore Flows and Triggers
* Para usar variables en scripts de Custom DB, vea los [parámetros de configuración](/docs/es/authenticate/database-connections/custom-db/create-db-connection#step-3-add-configuration-parameters)

<Info>
  ### Práctica recomendada

  Se recomienda usar variables para almacenar valores específicos del tenant, así como cualquier secret sensible que no deba exponerse en su código personalizado. Si su código personalizado está desplegado en GitHub, usar una variable específica del tenant evita exponer valores sensibles a través de su repositorio de GitHub.
</Info>

<div id="test-automation">
  ### Automatización de pruebas
</div>

Puedes automatizar tu proceso de compilación general incorporando la automatización de implementaciones, además de la automatización de pruebas. Esto puede usarse para desplegar nuevas versiones de la configuración y/o del código personalizado en Auth0 y ejecutar pruebas automatizadas. Si las pruebas detectan algún fallo, las capacidades de automatización de implementaciones pueden usarse para revertir a la última versión funcional. Para obtener más información, consulta la [guía sobre automatización de implementaciones](/docs/es/get-started/architecture-scenarios/business-to-consumer/deployment) proporcionada.

<div id="mock-testing">
  ## Prueba simulada
</div>

Para equilibrar la [política de pruebas de carga](/docs/es/troubleshoot/customer-support/operational-policies/load-testing-policy) de Auth0 con la necesidad de realizar pruebas de carga, es habitual que los clientes de Auth0 simulen los endpoints de Auth0. Esta es una práctica valiosa para garantizar que su aplicación funcione con las interfaces previstas sin tener que limitar sus pruebas, y puede apoyarse en herramientas como [MockServer](http://www.mock-server.com/), [JSON Server](https://github.com/typicode/json-server) o incluso [Postman](https://learning.getpostman.com/docs/postman/mock_servers/setting_up_mock/).

<div id="pen-testing-optional">
  ## Pruebas de penetración (opcional)
</div>

Si va a realizar pruebas de penetración, debe conocer y cumplir la [política de pruebas de penetración](/docs/es/troubleshoot/customer-support/operational-policies/penetration-testing-policy) de Auth0. Las pruebas de penetración requieren notificar a Auth0 con antelación para evitar que se confundan con actividad maliciosa y se interrumpan.

<div id="load-testing-optional">
  ## Pruebas de carga (opcional)
</div>

Si vas a realizar pruebas de carga, debes conocer la [política de pruebas de carga](/docs/es/troubleshoot/customer-support/operational-policies/load-testing-policy) de Auth0 y cumplirla. Las pruebas de carga requieren notificar a Auth0 con antelación. Al planificar tus pruebas de carga, también tendrás que tener en cuenta los [límites de frecuencia de la API](/docs/es/troubleshoot/customer-support/operational-policies/rate-limit-policy) de Auth0.

Las pruebas de carga requieren la aprobación previa de Auth0, como se explica en la política de pruebas de carga de Auth0. Asegúrate de tener en cuenta el plazo necesario para que se revise una solicitud y de dejar tiempo suficiente tanto para la revisión como para la ejecución de las pruebas. Si tu solicitud de prueba de carga ha sido aprobada, la siguiente guía puede ayudarte a evitar errores y resultados de prueba inexactos.

* Ejecuta una traza HTTP en una ejecución de prueba de tu aplicación para identificar todas las llamadas que tu aplicación o la prueba prevista debe realizar, y asegúrate de que tu prueba las incluya para que sea representativa de lo que ocurrirá en Production.
* Diseña tu prueba teniendo en cuenta los límites de frecuencia de la Auth0 API.
* El uso de cualquier código personalizado en Auth0 (Actions, Rules, Hooks, scripts de base de datos personalizados, conexiones de <Tooltip tip="OAuth 2.0: Marco de autorización que define protocolos y flujos de trabajo de autorización." cta="Ver glosario" href="/docs/es/glossary?term=OAuth">OAuth</Tooltip> personalizadas) invocará el sandbox de código personalizado de Auth0, y esto puede tener un costo mayor en términos de rendimiento. Desactiva las Rules a menos que sean esenciales para la prueba. Si están desactivadas, tendrás un mayor rendimiento que si están activadas.
* Estima la carga total esperada para tu entorno de Production y el porcentaje de llamadas a cada endpoint, y estructura tu prueba de rendimiento en consecuencia para que te proporcione un resultado realista. Los distintos endpoints tienen costos de rendimiento diferentes. Si no diseñas una prueba representativa, obtendrás resultados engañosos.
* No hagas llamadas que dependan de los resultados de llamadas anteriores sin comprobar que las llamadas o respuestas previas necesarias se hayan completado. Limitarte a introducir una demora puede no ser suficiente.
* Asegúrate de implementar un manejo de errores adecuado. Una causa frecuente de incidencias durante las pruebas son los errores en el código personalizado (Actions, Rules, Hooks, scripts de base de datos personalizados, scripts de conexión de OAuth personalizados) provocados por excepciones no controladas en ese código.
* Las pruebas de carga deben escribirse de forma que empiecen en un nivel bajo y aumenten la carga gradualmente, recopilando datos en cada nivel, para obtener los resultados más útiles. Empezar en un nivel alto y fallar de inmediato aporta menos información sobre lo que el sistema puede soportar.
* Es normal tener que ejecutar una prueba de rendimiento varias veces, posiblemente ajustando el código sometido a prueba o el arnés o la configuración de la prueba. Asegúrate de empezar las pruebas con antelación para dejar tiempo suficiente para más de una iteración.
* Usa tu propia cuenta del proveedor de correo y asegúrate de gestionar con antelación una cuota de envío suficiente, o el proveedor de correo podría aplicarte límites de frecuencia. Desactiva el envío de correo si no lo usas.
* Asegúrate de usar las credenciales de tu propia cuenta para todas las conexiones sociales en lugar de las claves de desarrollo de Auth0. En el <Tooltip tip="Auth0 Dashboard: el producto principal de Auth0 para configurar tus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Auth0+dashboard">Auth0 Dashboard</Tooltip>, ve a Connections -&gt; Social -&gt; &#123;name of connection&#125;—para ver instrucciones sobre cómo añadir a la conexión las credenciales de la cuenta de tu propio proveedor social. Nota: algunos proveedores sociales no permiten pruebas de carga. Consulta la política de tu(s) proveedor(es).
* Para evitar la limitación de frecuencia y simular con mayor precisión una carga real, tus pruebas tendrán que enviar solicitudes para distintos usuarios, no todas las solicitudes para el mismo usuario. Si usas solo uno o unos pocos usuarios, el almacenamiento en caché puede reducir la carga efectiva y no proporcionar resultados realistas.
* Asegúrate de mantenerte dentro de los parámetros acordados para la prueba y de la política de pruebas de carga de Auth0. Auth0 se reserva el derecho de interrumpir cualquier prueba de rendimiento o de carga que no se mantenga dentro de los límites de los parámetros acordados o que se extienda más allá del periodo de prueba programado.
