Antes del lanzamiento, deberías haber completado todas las pruebas aplicables a tu entorno.

La garantía de calidad es fundamental para identificar problemas antes de que afecten a tus clientes y, según la naturaleza de tu proyecto, hay varios tipos de pruebas de calidad que conviene considerar como parte de tu integración con Auth0:

* ¿Tu aplicación es fácil de entender y usar, incluso para personas con discapacidad?
* ¿Tu aplicación debe funcionar en distintos navegadores y dispositivos?
* ¿Tu aplicación debe funcionar en entornos multinacionales y/o internacionales?
* ¿Cómo responderá tu aplicación cuando se vea sometida a cargas de producción inesperadas?
* ¿Cómo puedes asegurarte de que tu aplicación esté protegida frente a vulnerabilidades de seguridad?

Auth0 [Universal Login](/docs/es/authenticate/login/auth0-universal-login) y los widget de interfaz de usuario asociados (como [Lock](/docs/es/libraries/lock)) ya se han diseñado y desarrollado siguiendo prácticas recomendadas de usabilidad y accesibilidad, y ofrecen soporte probado y listo para usar para una amplia variedad de [navegadores y dispositivos](/docs/es/troubleshoot/customer-support/product-support-matrix). También se ofrece compatibilidad inmediata con la [internacionalización](/docs/es/customize/internationalization-and-localization) (I18N), con extensibilidad integrada pensada para casos personalizados de varios idiomas y localización (L10N).

Para garantizar que se cumplan los requisitos funcionales y que los eventos inesperados se gestionen correctamente, se ofrece orientación para probar la [integración](#integration-testing) entre tu(s) aplicación(es) y Auth0, así como para las [pruebas unitarias](#unit-testing) de módulos individuales de extensibilidad (como [Rules](/docs/es/customize/rules/debug-rules), [Hooks](/docs/es/customize/hooks/update-hooks) y scripts de Custom Database). También se proporciona orientación sobre la [política de pruebas de penetración](/docs/es/troubleshoot/customer-support/operational-policies/penetration-testing-policy) de Auth0 para ayudarte a realizar pruebas de vulnerabilidades de seguridad, así como sobre cómo pueden aprovecharse las pruebas con [Mock](#mock-testing) junto con nuestra [política de pruebas de carga](/docs/es/troubleshoot/customer-support/operational-policies/load-testing-policy) para ayudar a garantizar que tu(s) aplicación(es) rindan bien ante cargas inesperadas.

<div id="unit-testing">
  ## Pruebas unitarias
</div>

El objetivo de las pruebas unitarias es probar unidades individuales de código. Si crea código personalizado en Auth0 en forma de Rules, Hooks y/o Custom DB scripts, debería considerar usar un framework de pruebas (como [Mocha](https://mochajs.org/)) para probar su código. Las empresas que han tenido más éxito con Auth0 han comprobado que resulta útil ejecutar estas pruebas unitarias antes de [desplegar automáticamente](/docs/es/get-started/architecture-scenarios/business-to-consumer/deployment) la configuración y los recursos asociados del tenant de Auth0.

<div id="integration-testing">
  ## Pruebas de integración
</div>

Una práctica recomendada es configurar distintos tenants para desarrollo, pruebas y producción, como se explica en la guía de arquitectura sobre [soporte para SDLC](/docs/es/get-started/architecture-scenarios/business-to-consumer/architecture#sdlc-support). Auth0 le permite configurar variables disponibles desde la [extensibilidad](/docs/es/customize/extensions) personalizada; puede considerarlas variables de entorno para su tenant de Auth0. En lugar de codificar de forma fija referencias que cambian al mover código entre entornos de desarrollo, pruebas y producción, puede usar un nombre de variable configurado en el tenant y al que haga referencia el código de extensibilidad personalizada. Esto facilita que el mismo código personalizado funcione sin cambios en distintos tenants, ya que puede hacer referencia a variables que se completarán con valores específicos del tenant en el momento de la ejecución:

* Para usar variables en Rules, vea cómo [configurar valores](/docs/es/customize/rules/configuration)
* Para usar variables en Hooks, vea cómo configurar [secretos](/docs/es/customize/hooks/hook-secrets) en el editor
* Para usar variables en Actions, vea Explorar Flows y triggers
* Para usar variables en Custom DB Scripts, consulte los [parámetros de configuración](/docs/es/authenticate/database-connections/custom-db/create-db-connection#step-3-add-configuration-parameters)

<Info>
  ### Práctica recomendada

  Una práctica recomendada es usar variables para almacenar valores específicos del tenant, así como cualquier secreto sensible que no deba quedar expuesto en su código personalizado. Si su código personalizado está desplegado en GitHub, usar una variable específica del tenant evita exponer valores sensibles a través de su repositorio de GitHub.
</Info>

<div id="test-automation">
  ### Automatización de pruebas
</div>

Puedes automatizar todo tu proceso de compilación incorporando tanto la automatización del despliegue como la automatización de pruebas. Esto puede usarse para desplegar nuevas versiones de la configuración y/o del código personalizado en Auth0 y ejecutar pruebas automatizadas. Si las pruebas detectan algún fallo, las capacidades de automatización del despliegue pueden usarse para revertir a la última versión funcional. Para obtener más información, consulta la [guía sobre automatización del despliegue](/docs/es/get-started/architecture-scenarios/business-to-consumer/deployment).

<div id="mock-testing">
  ## Prueba simulada
</div>

Como punto intermedio entre la [política de pruebas de carga](/docs/es/troubleshoot/customer-support/operational-policies/load-testing-policy) de Auth0 y la necesidad de realizar una prueba de carga, es habitual que los clientes de Auth0 simulen los endpoints de Auth0. Esta práctica resulta valiosa para garantizar que su aplicación funcione con las interfaces previstas sin tener que limitar sus pruebas, y pueden utilizarse herramientas como [MockServer](http://www.mock-server.com/), [JSON Server](https://github.com/typicode/json-server) o incluso [Postman](https://learning.getpostman.com/docs/postman/mock_servers/setting_up_mock/) para ello.

<div id="pen-testing-optional">
  ## Pruebas de penetración (opcional)
</div>

Si va a realizar pruebas de penetración, debe conocer y cumplir la [política de pruebas de penetración](/docs/es/troubleshoot/customer-support/operational-policies/penetration-testing-policy) de Auth0. Las pruebas de penetración requieren avisar a Auth0 con antelación para que no se confundan con actividad maliciosa ni se interrumpan.

<div id="load-testing-optional">
  ## Pruebas de carga (opcional)
</div>

Si vas a realizar pruebas de carga, debes conocer la [política de pruebas de carga](/docs/es/troubleshoot/customer-support/operational-policies/load-testing-policy) de Auth0 y cumplirla. Las pruebas de carga requieren notificar a Auth0 con antelación. Al planificar tus pruebas de carga, también debes tener en cuenta los [límites de frecuencia de la API](/docs/es/troubleshoot/customer-support/operational-policies/rate-limit-policy) de Auth0.

Las pruebas de carga requieren la aprobación previa de Auth0, como se explica en la política de pruebas de carga de Auth0. Asegúrate de tener en cuenta el tiempo de antelación necesario para que se revise una solicitud y de dejar margen suficiente tanto para la revisión como para la ejecución de las pruebas. Si se aprobó tu solicitud de prueba de carga, la siguiente guía puede ayudarte a evitar errores y resultados de prueba incorrectos.

* Ejecuta una traza HTTP durante una ejecución de prueba de tu aplicación para identificar todas las llamadas que tu aplicación o la prueba prevista deben hacer, y asegúrate de incluirlas en la prueba para que sea representativa de lo que ocurrirá en producción.
* Diseña tu prueba teniendo en cuenta los límites de frecuencia de la API de Auth0.
* El uso de cualquier código personalizado en Auth0 (Actions, Rules, Hooks, scripts personalizados de base de datos, conexiones <Tooltip tip="OAuth 2.0: Marco de autorización que define protocolos y flujos de trabajo de autorización." cta="Ver glosario" href="/docs/es/glossary?term=OAuth">OAuth</Tooltip> personalizadas) invocará el sandbox de código personalizado de Auth0, y esto puede afectar al rendimiento. Desactiva las Rules a menos que sean esenciales para la prueba. Si están desactivadas, tendrás un mayor rendimiento que si están activadas.
* Calcula la carga total esperada para tu entorno de producción y el porcentaje de llamadas a cada endpoint, y estructura tu prueba de rendimiento en consecuencia para obtener un resultado realista. Los distintos endpoints tienen costes de rendimiento diferentes. Si no diseñas una prueba representativa, obtendrás resultados engañosos.
* No hagas llamadas que dependan de los resultados de llamadas anteriores sin comprobar que las llamadas o respuestas previas necesarias se hayan completado. Añadir simplemente una demora puede no ser suficiente.
* Asegúrate de implementar un manejo de errores adecuado. Una causa frecuente de incidencias durante las pruebas son los errores en el código personalizado (Actions, Rules, Hooks, scripts personalizados de base de datos, scripts personalizados de conexión OAuth) provocados por excepciones no controladas en el código personalizado.
* Las pruebas de carga deben diseñarse para comenzar con un nivel bajo e incrementar la carga gradualmente, recopilando datos en cada nivel, para obtener los resultados más útiles. Comenzar en un nivel alto y fallar de inmediato aporta menos información sobre lo que el sistema puede soportar.
* Es normal tener que ejecutar una prueba de rendimiento varias veces, posiblemente ajustando el código que se está probando o la configuración de la herramienta de prueba. Asegúrate de comenzar las pruebas con antelación para disponer de tiempo suficiente para más de una iteración.
* Usa tu propia cuenta de proveedor de correo y asegúrate de gestionar con antelación una cuota de envío suficiente, o el proveedor de correo podría aplicarte límites de frecuencia. Desactiva el envío de correo si no lo utilizas.
* Asegúrate de usar las credenciales de tu propia cuenta para todas las conexiones sociales en lugar de las claves de desarrollo de Auth0. En el <Tooltip tip="Auth0 Dashboard: El producto principal de Auth0 para configurar tus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Auth0+dashboard">Auth0 Dashboard</Tooltip>, ve a Connections -&gt; Social -&gt; &#123;name of connection&#125;—para ver instrucciones sobre cómo añadir a la conexión las credenciales de la cuenta de tu propio proveedor social. Nota: algunos proveedores sociales no permiten pruebas de carga. Consulta la política de tus proveedores.
* Para evitar la limitación de frecuencia y simular con mayor precisión la carga real, tus pruebas deberán enviar solicitudes para distintos usuarios, no todas las solicitudes para el mismo usuario. Si usas solo uno o unos pocos usuarios, la caché puede reducir la carga efectiva y no proporcionar resultados realistas.
* Asegúrate de mantenerte dentro de los parámetros acordados para la prueba y de la política de pruebas de carga de Auth0. Auth0 se reserva el derecho de finalizar cualquier prueba de rendimiento/carga que no se mantenga dentro de los límites de los parámetros acordados o que se extienda más allá de la ventana de prueba programada.
