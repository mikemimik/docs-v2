Empecemos dando un paso atrás para hablar del control de acceso. No existe una definición única y clara del control de acceso en la industria, pero si dedicas algo de tiempo a investigar y leer, verás que la mayoría de las fuentes de referencia coinciden en que es el concepto general que reúne la Autenticación, la Autorización, el consentimiento y la aplicación de políticas para garantizar que solo las personas y los servicios adecuados tengan acceso a tus aplicaciones y API. A continuación, veamos más de cerca las diferencias entre la Autenticación, la Autorización, el consentimiento y la aplicación de políticas. Tu <Tooltip tip="Un grupo de usuarios lógicamente aislado que comparte acceso común con privilegios específicos a una sola instancia de software." cta="Ver glosario" href="/docs/es/glossary?term=Tenant">tenant</Tooltip> de Auth0 (tu <Tooltip tip="Authorization Server: Servidor centralizado que contribuye a definir los límites del acceso de un usuario. Por ejemplo, tu servidor de autorización puede controlar los datos, las tareas y las funcionalidades disponibles para un usuario." cta="Ver glosario" href="/docs/es/glossary?term=Authorization+Server">servidor de autorización</Tooltip>) normalmente es responsable de la Autenticación y del consentimiento, así como de parte o de toda la Autorización y la aplicación de políticas. Además, una <Tooltip tip="Tu software que depende de Auth0 para la autenticación y la gestión de identidades." cta="Ver glosario" href="/docs/es/glossary?term=Application">aplicación</Tooltip> o la propia API casi siempre es la principal responsable de aplicar las políticas, especialmente cuando se requiere acceso contextual:

* **Autenticación**: el proceso de determinar si un principal (un usuario o una aplicación) es quien o lo que dice ser.
* **Autorización**: el proceso de determinar qué está permitido, en función del principal, de los permisos que se le hayan concedido y/o del conjunto de criterios de acceso específicos del contexto.
* **Consentimiento**: qué permisos ha autorizado el usuario (<Tooltip tip="Resource Owner: Entidad (como un usuario o una aplicación) capaz de conceder acceso a un recurso protegido." cta="Ver glosario" href="/docs/es/glossary?term=Resource+Owner">Resource Owner</Tooltip>) para que una aplicación actúe en su nombre. Esto suele ser un requisito de la autorización delegada. El usuario tiene que dar permiso al cliente para acceder a sus datos en un sistema diferente.
* **Aplicación de políticas**: el acto de hacer cumplir las políticas de la aplicación o API, rechazando o permitiendo el acceso en función de la información de autenticación y/o autorización de un usuario.

En general, solemos agrupar los distintos tipos de control de acceso en tres categorías para que sea más fácil entender a) qué actor es responsable de almacenar la información, b) qué actor es responsable de tomar decisiones y c) cuál es responsable de aplicar las restricciones.

* La primera categoría es aquella en la que se concede o se deniega acceso a una aplicación o a una API en su totalidad. Tanto los datos necesarios para aplicarlo como el proceso de aplicación suelen definirse en el contexto del servidor de autorización. Por ejemplo, mediante el uso de [`app_metadata`](/docs/es/manage-users/user-accounts/metadata) asociado a un usuario y una [Action](/docs/es/customize/actions) definida en tu tenant de Auth0.
* La segunda categoría es aquella en la que se concede o se deniega acceso a un subconjunto específico de la funcionalidad de una aplicación o API. Los datos necesarios para aplicarlo normalmente se almacenan en el servidor de autorización. Por ejemplo, mediante el uso de `app_metadata` en un usuario de tu tenant de Auth0, mientras que el proceso de aplicación se realiza en la propia aplicación o API. En este escenario, los datos suelen comunicarse como uno o varios custom claims en un token de [`id`](/docs/es/secure/tokens/id-tokens) o de [`access`](/docs/es/secure/tokens/access-tokens).
* La tercera categoría es aquella en la que se concede o se deniega acceso en función de sobre qué puede operar el principal (subject) dentro del contexto de una aplicación o API. Tanto los datos necesarios para aplicarlo como el proceso de aplicación suelen definirse en el contexto de la aplicación o API. En este escenario, los datos comunicados como uno o varios custom claims en un token de `id` o `access` pueden consumirse con o sin datos de una fuente externa distinta de Auth0.

Además, los mecanismos de Role-based Access Control (RBAC) y Attribute-based Access Control (ABAC) pueden aplicarse en cualquiera de las categorías de control de acceso descritas anteriormente. Sea cual sea tu caso de uso, hay varios aspectos que querrás tener en cuenta al evaluar la funcionalidad y el workflow que necesitas:

* ¿Hay escenarios en los que deba rechazarse el acceso a una aplicación o API completas?
* ¿Proporcionarás API a las que puedan acceder aplicaciones de terceros?
* ¿Tus API también podrán ser accedidas por tus propias aplicaciones (first-party)?
* ¿Tu aplicación llamará a una API de terceros?
* ¿Tus aplicaciones y/o API deberían aplicar control de acceso basado en claims de usuario?

Auth0 admite la restricción de acceso para aplicaciones o APIs en función de determinadas condiciones. En algunos casos, puede que quieras crear una Action que deniegue el acceso con `api.access.deny()` cuando, por ejemplo, un usuario intenta acceder a una aplicación o a una API en un momento no permitido (como se describe en este [ejemplo](/docs/es/manage-users/access-control/sample-use-cases-actions-with-authorization#allow-access-only-on-weekdays-for-a-specific-application)) o si el usuario no tiene el <Tooltip tip="Atributo incluido en un token de seguridad que representa una claim que el proveedor del token hace sobre una entidad." cta="Ver glosario" href="/docs/es/glossary?term=Claim">claim</Tooltip>(s) adecuado(s) en su `app_metadata`. En una aplicación que use [OpenID Connect (OIDC)](/docs/es/authenticate/protocols/openid-connect-protocol), esto impediría la emisión del [token de ID](/docs/es/secure/tokens/id-tokens) utilizado para autorizar el acceso. Del mismo modo, en una API, podría impedirse la emisión de cualquier OAuth2 [token de acceso](/docs/es/secure/tokens/access-tokens) (utilizado al llamar a la API), como se describe en este [ejemplo](/docs/es/manage-users/access-control/sample-use-cases-actions-with-authorization#deny-access-to-anyone-calling-an-api).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  En general, hemos comprobado que [OIDC](/docs/es/authenticate/protocols/openid-connect-protocol) es el protocolo estándar del sector que más utilizan los clientes de Auth0 para la autenticación en sus aplicaciones. También hemos observado que, aunque [OAuth2](/docs/es/authenticate/protocols/oauth) se creó como un protocolo de delegación, suele usarse en aplicaciones de origen propio cuando hay una API que no comparte sesión con la aplicación.
</Callout>

Auth0 también puede proporcionar la información necesaria para que una aplicación aplique restricciones. Para la [integración a nivel de aplicación](#application-integration), Auth0 te permite añadir [claims personalizadas](#id-token-claims) a un <Tooltip tip="Token de ID: credencial destinada al propio cliente, en lugar de al acceso a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=ID+Token">token de ID</Tooltip>, que tu aplicación puede verificar y usar después para aplicar políticas de acceso. En este caso, tendrás que decidir qué información necesitas para que tu aplicación pueda tomar decisiones de aplicación. Si necesitas tomar decisiones en una API en lugar de en tu aplicación, probablemente tendrás que usar un <Tooltip tip="Token de acceso: credencial de autorización, en forma de una cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=Access+Token">token de acceso</Tooltip> en lugar de un token de identidad. Sigue leyendo para obtener más información.

<Warning>
  Al decidir qué datos incluir en tu token de identidad y/o token de acceso, ten en cuenta el tamaño del token, especialmente si lo pasas en la URL. Aunque no pases tokens en la URL, también tendrás que considerar la posibilidad de exponer PII (información de identificación personal) sensible. La información del token no está cifrada, así que, aunque por lo general no supone un problema de seguridad que se filtre un token de identidad, puede convertirse en un problema de privacidad dependiendo de los datos incluidos en el token.
</Warning>

Para la [integración a nivel de API](#api-integration), Auth0 admite tanto [claims personalizadas](#access-token-claims) como la reconfiguración de [scope](#access-token-scopes), ambas en el contexto de un token de acceso. De nuevo, tendrás que decidir qué información será necesaria para que tu API tome decisiones de acceso, y tu API deberá aplicarlas validando el contenido del token de acceso.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Al decidir si debes usar permisos mediante claims personalizadas o scopes, debes asegurarte de entender la naturaleza y el propósito de los scopes. Hay una buena [entrada de blog](https://auth0.com/blog/on-the-nature-of-oauth2-scopes/) sobre ello que es fácil de leer y ayuda a aclarar el tema.
</Callout>

<div id="application-integration">
  ## Integración de la aplicación
</div>

En este escenario, tu tenant de Auth0 proporciona un token como prueba de acceso autorizado a una aplicación. En las aplicaciones que utilizan [OpenID Connect (OIDC)](/docs/es/authenticate/protocols/openid-connect-protocol), el protocolo estándar del sector que normalmente vemos con más frecuencia en aplicaciones orientadas al cliente, se trataría de un token de ID expresado como un [JWT](/docs/es/secure/tokens/json-web-tokens).

<div id="id-token-claims">
  ### Claims del token de ID
</div>

Con la extensibilidad de Action, Auth0 te permite [agregar claims personalizados a un token de ID](/docs/es/get-started/apis/scopes/sample-use-cases-scopes-and-claims#add-custom-claims-to-a-token) fácilmente en función, por ejemplo, del contenido de [Metadata](/docs/es/manage-users/user-accounts/metadata) de un usuario. Luego, tu aplicación puede verificar que el token de ID contenga los claims necesarios y, según corresponda, permitir o impedir el acceso a determinadas funcionalidades. Ten en cuenta que, aunque el proceso de agregar claims personalizados mediante Action está simplificado, el runtime de Action es flexible y te permite escribir código personalizado que podría tener efectos negativos.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Si estás pensando en agregar claims personalizados, te recomendamos almacenar en el [`app_metadata`](/docs/es/manage-users/user-accounts/metadata) del usuario cualquier dato de control de acceso que necesites incluir en los claims. En primer lugar, así evitas tener que llamar a una API externa para obtener esos datos, lo que puede afectar negativamente al rendimiento y la escalabilidad de la secuencia de inicio de sesión. En segundo lugar, el usuario **no** puede modificar `app_metadata`, por lo que no puede eludir directamente ninguna restricción de control de acceso modificando sus propios <Tooltip tip="Información que los usuarios pueden actualizar, como preferencias o ajustes del perfil." cta="Ver glosario" href="/docs/es/glossary?term=Metadata">metadatos</Tooltip>. Recuerda también consultar nuestra guía de [prácticas recomendadas sobre metadatos](/docs/es/get-started/architecture-scenarios/business-to-consumer/profile-management#metadata).
</Callout>

<div id="id-token-scopes">
  ### Scopes del token de ID
</div>

Los [scopes de OIDC](/docs/es/get-started/apis/scopes/openid-connect-scopes) suelen utilizarse para que una aplicación obtenga el consentimiento necesario para acceder de forma autorizada a los datos de un usuario durante la autenticación. Cada uno de los scopes predefinidos devuelve el conjunto de claims estándar cuando corresponde, tal como se describe en la [especificación de OIDC](https://openid.net/specs/openid-connect-core-1_0.html#StandardClaims). Los scopes que solicita una aplicación dependen de los atributos de usuario que necesite. Una vez que el usuario autoriza los scopes solicitados, los claims se devuelven en el token de ID y también están disponibles a través del endpoint [/userinfo](https://auth0.com/docs/api/authentication#get-user-info).

<div id="api-integration">
  ## Integración de API
</div>

En este escenario, tu tenant de Auth0 puede proporcionar un OAuth2 [token de acceso](/docs/es/secure/tokens/access-tokens), normalmente expresado como un [JWT](/docs/es/secure/tokens/json-web-tokens), que tu API puede usar para restringir el acceso a determinadas partes. Además, Auth0 ofrece soporte para lo que, conceptualmente, se describe como [aplicaciones propias y de terceros](/docs/es/get-started/applications/first-party-and-third-party-applications).

Actuando como servidor de autorización y con el consentimiento del usuario (el Resource Owner), tu tenant de Auth0 puede usarse para proporcionar un token de acceso —normalmente expresado como un [JWT](/docs/es/secure/tokens/json-web-tokens)— a una aplicación (cliente) para que pueda acceder a un recurso protegido alojado por un <Tooltip tip="Resource Server: servidor que aloja recursos protegidos. Los resource servers aceptan y responden a solicitudes de recursos protegidos." cta="Ver glosario" href="/docs/es/glossary?term=resource+server">resource server</Tooltip> en nombre del Resource Owner. El token de acceso emitido normalmente se envía como Bearer token en el encabezado HTTP Authorization de una solicitud a una API.

Tanto si tienes una sola API como un conjunto de [API de microservicios](/docs/es/get-started/apis/set-logical-api) relacionadas lógicamente, puedes aprovechar los tokens de acceso que proporciona Auth0 para proteger el acceso a tus servicios. Aunque configurarlo es relativamente sencillo en el [Auth0 Dashboard](https://manage.auth0.com/) o mediante la [Auth0 Management API](https://auth0.com/docs/api/management/v2#!/Resource_Servers/post_resource_servers), es importante revisar los distintos escenarios de aplicación y diseños de API para determinar la mejor arquitectura para tu sistema.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Los OAuth2 tokens de acceso están diseñados principalmente para proteger API expuestas públicamente; cuando se expresan como un JWT, un token de acceso es una entidad autocontenida que puede verificarse sin necesidad de hacer ninguna llamada adicional a una API de terceros. Si tus API no entran en esta categoría, es decir, si forman parte de la propia aplicación (por ejemplo, si solo esa aplicación las invoca) o están detrás de tu firewall, protegerlas con tokens puede resultar excesivo, y tu flujo de trabajo actual basado en cookies (entre otros) probablemente sea suficiente.
</Callout>

OAuth2 se diseñó específicamente pensando en el acceso de terceros; por ejemplo, puede darse el caso de que un usuario (Resource Owner) quiera usar una aplicación (un cliente) que no pertenece a la misma organización que el servicio que proporciona los datos del usuario (el resource server). En este caso, cuando la aplicación necesita acceder a datos que pertenecen al usuario, redirige a la organización donde residen esos datos, que a su vez autentica al usuario y luego le solicita que conceda permiso a la aplicación para acceder a ellos. Esta solicitud de permiso se conoce como [consentimiento](/docs/es/get-started/applications/third-party-applications/user-consent-and-third-party-applications) y es una parte fundamental de lo que implica ofrecer soporte para [aplicaciones de terceros](/docs/es/get-started/apis/scopes/api-scopes#example-an-api-called-by-a-third-party-application). Si planeas integrar aplicaciones de terceros, es importante que las marques como de terceros desde el principio para que Auth0 gestione la solicitud de consentimiento del usuario.

Por otro lado, si tu organización posee la(s) aplicación(es), los propios datos del usuario y la(s) API a través de las cuales se accede a esos datos, normalmente no se requiere consentimiento, ya que todas las interacciones son [propias](/docs/es/get-started/apis/scopes/api-scopes#example-an-api-called-by-a-first-party-application). Si solo estás creando aplicaciones propias, puedes asegurarte de no mostrar a tus usuarios pantallas de consentimiento innecesarias [permitiendo omitir el consentimiento del usuario](/docs/es/get-started/apis) como parte de cualquier definición de servicio de recursos.

<Warning>
  Aunque puedes configurar tus aplicaciones como propias y posteriormente configurar tus API para permitir que los clientes propios omitan el consentimiento, si estás usando `localhost`, Auth0 no puede verificar que la aplicación sea realmente una aplicación propia, por lo que tus usuarios recibirán igualmente la solicitud de consentimiento. Para evitar esta limitación, al hacer pruebas en tu máquina local durante el desarrollo, crea un [hostname local falso y úsalo en su lugar](https://community.auth0.com/t/how-do-i-skip-the-consent-page-for-my-api-authorization-flow/6035).
</Warning>

Como alternativa, puede tener datos relacionados con un usuario para los que se ofrece [funcionalidad adicional](/docs/es/get-started/apis/scopes/api-scopes#example-an-api-called-by-a-back-end-service) y cuyo consentimiento explícito no puede obtenerse (es decir, no hay ningún usuario autenticado que pueda otorgarlo). En este caso, puede definirse una [lista de aplicaciones para las que está habilitada la Concesión de credenciales de cliente](/docs/es/get-started/authentication-and-authorization-flow/client-credentials-flow).

<div id="access-token-claims">
  ### Claims del token de acceso
</div>

Como sucede con los tokens de ID, puedes [agregar claims personalizadas a los tokens de acceso](/docs/es/get-started/apis/scopes/sample-use-cases-scopes-and-claims#add-custom-claims-to-a-token) mediante la extensibilidad de Action de Auth0. Una vez agregadas, tu API puede verificar si un token de acceso contiene los claims necesarios y permitir o impedir el acceso a determinadas funcionalidades, según corresponda.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Si estás pensando en agregar claims personalizadas, te recomendamos almacenar en el [`app_metadata`](/docs/es/manage-users/user-accounts/metadata) del usuario cualquier dato de control de acceso que necesites incluir en los claims. En primer lugar, esto evita tener que llamar a una API externa para obtener los datos, lo que puede afectar negativamente al rendimiento y a la escalabilidad. En segundo lugar, un usuario **no puede** modificar `app_metadata`, por lo que no puede eludir directamente ninguna restricción de control de acceso modificando su propia metadata. Recuerda también consultar nuestra guía de [prácticas recomendadas sobre metadata](/docs/es/get-started/architecture-scenarios/business-to-consumer/profile-management#metadata).
</Callout>

<div id="access-token-scopes">
  ### Scopes del token de acceso
</div>

Los [OAuth2 Scopes](/docs/es/get-started/apis/scopes/api-scopes) se utilizan normalmente como el mecanismo mediante el cual una API puede determinar qué <Tooltip tip="Funciones seguras, específicas del tenant y versionadas escritas en Node." cta="Ver glosario" href="/docs/es/glossary?term=Actions">Actions</Tooltip> se pueden realizar en nombre de un usuario. Los scopes pueden añadirse para cada API para [definir permisos de acceso específicos](/docs/es/get-started/apis/add-api-permissions) en el <Tooltip tip="Auth0 Dashboard: el producto principal de Auth0 para configurar tus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip> o mediante la <Tooltip tip="Auth0 Dashboard: el producto principal de Auth0 para configurar tus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip> de Auth0. Los scopes también pueden manipularse mediante la extensibilidad de Auth0 (por ejemplo, mediante una Action, como en este [ejemplo](/docs/es/get-started/architecture-scenarios/spa-api/part-2#create-an-action-to-validate-token-scopes)). Los scopes que una aplicación solicita para acceder a una API deben depender de las funcionalidades para las que la aplicación necesita que el usuario le conceda permiso. Una vez autorizados los scopes solicitados, se devolverán en el token de acceso, que posteriormente dicha [API podrá verificar](/docs/es/secure/tokens/access-tokens/validate-access-tokens). Un buen ejemplo de esto es cuando inicias sesión en una aplicación que usa un proveedor social para el login: la API del proveedor social requiere que la aplicación especifique si el usuario quiere que la aplicación publique contenido en su nombre. Esto permite al usuario aceptar o denegar esta solicitud. Este ejemplo demuestra cómo el usuario está delegando permisos a la aplicación, lo cual es diferente de que la API restrinja el acceso en función del <Tooltip tip="Aspecto de la identidad de un usuario asignado al usuario para indicar el nivel de acceso que debe tener al sistema." cta="Ver glosario" href="/docs/es/glossary?term=Role">rol</Tooltip> del usuario, y debe gestionarse de otra manera.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  **Práctica recomendada**

  Aunque tienes la capacidad de manipular por completo los scopes del token de acceso mediante la extensibilidad de Auth0, como práctica recomendada de seguridad solo debes eliminar los scopes que no estén autorizados y evitar añadir scopes que no se hayan solicitado.
</Callout>

Aunque los scopes suelen usarse como una forma de aplicar permisos de acceso a un usuario, hay situaciones en las que [puede resultar complicado usarlos de esta manera](https://auth0.com/blog/on-the-nature-of-oauth2-scopes/). Por lo tanto, te recomendamos que uses los scopes para su propósito previsto (es decir, delegar permisos a una aplicación) y uses [custom claims](#access-token-claims) para tus escenarios basados en roles u otros de control de acceso.

<div id="fine-grained-authorization-fga">
  ## Autorización detallada (FGA)
</div>

[Autorización detallada](https://docs.fga.dev/fga-concepts) te permite otorgar a usuarios individuales acceso a un recurso u objeto específico en función de:

* El rol de un usuario dentro de una organización, como `editor` o `admin`
* Un atributo del usuario o del objeto, como `gerente` para un usuario o `marketing` para un objeto
* Una relación entre un usuario y un objeto, como que un usuario con acceso de `view` a una carpeta principal también tenga acceso de `view` a la carpeta secundaria

Con <Tooltip tip="Autorización detallada (FGA): producto de Auth0 que permite a usuarios individuales acceder a objetos o recursos específicos." cta="Ver glosario" href="/docs/es/glossary?term=FGA">FGA</Tooltip>, puedes crear un modelo de autorización para definir qué relación quieres usar para determinar el acceso del usuario.

<div id="role-based-access-control-rbac">
  ## Control de acceso basado en roles (RBAC)
</div>

Auth0 ofrece soporte integrado para el control de acceso basado en roles ([RBAC](/docs/es/manage-users/access-control/rbac)). RBAC consiste en asignar permisos a los usuarios en función de su rol dentro de una organización, lo que simplifica el control de acceso al ofrecer un enfoque más fácil de administrar y menos propenso a errores.

<div id="machine-to-machine-m2m-authorization">
  ## Autorización de máquina a máquina (M2M)
</div>

Hay muchos escenarios en los que una aplicación sin ninguna sesión interactiva de usuario necesita obtener un token de acceso para llamar a una API. En esos casos, debes autenticar al cliente en lugar del usuario, y <Tooltip tip="OAuth 2.0: Marco de autorización que define protocolos y flujos de trabajo de autorización." cta="Ver glosario" href="/docs/es/glossary?term=OAuth">OAuth</Tooltip> 2 proporciona el tipo de concesión de [credenciales de cliente](/docs/es/get-started/authentication-and-authorization-flow/client-credentials-flow) para facilitarlo. Algunos ejemplos habituales en los que esto es necesario incluyen:

* Una tarea cron u otro servicio que necesite comunicarse con tu API (p. ej., cuando haya que generar un informe diario y enviárselo por correo electrónico a un administrador).
* Una API independiente que admita acceso privilegiado (p. ej., la API no se expone directamente a los usuarios, sino únicamente a un backend).
* En ciertas arquitecturas de microservicios, donde algunas capas de API necesitan comunicarse con otras capas de API sin intervención de un usuario, o después de que haya expirado el token de un usuario.
* Una API privilegiada a la que puede ser necesario llamar antes de que un usuario se haya autenticado (es decir, desde una Action o un script de base de datos personalizado en tu tenant de Auth0)

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Tradicionalmente, se creaba una &quot;cuenta de servicio&quot; para estos escenarios: un usuario con nombre de usuario y contraseña configurado para servicios que admitían casos de uso no interactivos. Este enfoque ya no se recomienda por muchas razones, y la práctica recomendada actual es usar [OAuth 2.0 Client Credentials Grant](/docs/es/get-started/authentication-and-authorization-flow/client-credentials-flow) en estas situaciones.
</Callout>

  ## Guía para la planificación del proyecto
