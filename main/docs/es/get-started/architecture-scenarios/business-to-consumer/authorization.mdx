Empecemos dando un paso atrás para hablar del control de acceso. No existe una definición única y clara del control de acceso en la industria, pero si dedicas algo de tiempo a investigar y leer, verás que la mayoría de las fuentes autorizadas coinciden en que es el concepto general que engloba la autenticación, la autorización, el consentimiento y la aplicación de políticas para garantizar que solo las personas y los servicios adecuados tengan acceso a tus aplicaciones y API. A continuación, veamos más de cerca las diferencias entre autenticación, autorización, consentimiento y aplicación de políticas. Tu Auth0 <Tooltip tip="Un grupo de usuarios aislado lógicamente que comparte acceso común con privilegios específicos a una única instancia de software." cta="Ver glosario" href="/docs/es/glossary?term=Tenant">tenant</Tooltip> (tu <Tooltip tip="Authorization Server: Servidor centralizado que contribuye a definir los límites del acceso de un usuario. Por ejemplo, tu servidor de autorización puede controlar los datos, las tareas y las funcionalidades disponibles para un usuario." cta="Ver glosario" href="/docs/es/glossary?term=Authorization+Server">servidor de autorización</Tooltip>) suele encargarse de la autenticación y el consentimiento, así como de parte o de toda la autorización y la aplicación de políticas. Además, una <Tooltip tip="Tu software que depende de Auth0 para la autenticación y la gestión de identidades." cta="Ver glosario" href="/docs/es/glossary?term=Application">aplicación</Tooltip> o la propia API casi siempre es la principal responsable de aplicar las políticas, especialmente cuando se requiere acceso contextual:

* **Autenticación**: el proceso de determinar si un principal (un usuario o una aplicación) es quien o lo que dice ser.
* **Autorización**: el proceso de determinar qué está permitido, en función del principal, de los permisos que se le hayan concedido y/o del conjunto de criterios de acceso específicos del contexto.
* **Consentimiento**: los permisos que el usuario (<Tooltip tip="Resource Owner: Entidad (como un usuario o una aplicación) capaz de conceder acceso a un recurso protegido." cta="Ver glosario" href="/docs/es/glossary?term=Resource+Owner">Resource Owner</Tooltip>) ha autorizado a una aplicación a realizar en su nombre. Por lo general, esto es un requisito de la autorización delegada. El usuario debe dar permiso al cliente para acceder a sus datos en otro sistema.
* **Aplicación de políticas**: el acto de aplicar las políticas de la aplicación o API, rechazando o permitiendo el acceso en función de la información de autenticación y/o autorización de un usuario.

En general, solemos agrupar los distintos tipos de control de acceso en tres categorías diferenciadas para que sea más fácil entender a) qué actor es responsable de almacenar la información, b) qué actor es responsable de tomar decisiones y c) cuál es responsable de aplicar las restricciones.

* La primera categoría es aquella en la que el acceso se concede o se deniega a una aplicación o a una API en su totalidad. Tanto los datos necesarios para hacerlo como el proceso de aplicación suelen definirse en el contexto del servidor de autorización. Por ejemplo, mediante el uso de [`app_metadata`](/docs/es/manage-users/user-accounts/metadata) asociado a un usuario y una [Action](/docs/es/customize/actions) definida en tu tenant de Auth0.
* La segunda categoría es aquella en la que el acceso se concede o se deniega a un subconjunto específico de la funcionalidad de la aplicación o API. Los datos necesarios para hacerlo suelen almacenarse en el servidor de autorización. Por ejemplo, mediante el uso de `app_metadata` en un usuario de tu tenant de Auth0, mientras que el proceso de aplicación se realiza en la propia aplicación o API. En este escenario, los datos suelen comunicarse como una o más claims personalizadas en un token de [`id`](/docs/es/secure/tokens/id-tokens) o de [`access`](/docs/es/secure/tokens/access-tokens).
* La tercera categoría es aquella en la que el acceso se concede o se deniega en función de sobre qué puede operar el principal (subject) dentro del contexto de una aplicación o API. Tanto los datos necesarios para hacerlo como el proceso de aplicación suelen definirse en el contexto de la aplicación o API. En este escenario, los datos comunicados como una o más claims personalizadas en un token `id` o `access` pueden consumirse con o sin datos de una fuente externa distinta de Auth0.

Además, los mecanismos de control de acceso basado en roles (RBAC) y control de acceso basado en atributos (ABAC) pueden aplicarse en cualquiera de las categorías de control de acceso descritas anteriormente. Sea cual sea tu caso de uso, hay una serie de aspectos que querrás tener en cuenta al evaluar la funcionalidad y el workflow que necesitas:

* ¿Hay escenarios en los que deba rechazarse el acceso a una aplicación o API completa?
* ¿Vas a proporcionar API a las que puedan acceder aplicaciones de terceros?
* ¿Tus API también serán accedidas por tus propias aplicaciones (first-party)?
* ¿Tu aplicación llamará a una API de terceros?
* ¿Tus aplicaciones y/o API deberían aplicar el control de acceso en función de las claims del usuario?

Auth0 admite la restricción de acceso a aplicaciones o APIs en función de determinadas condiciones. En ciertos escenarios, puede que quieras crear una Action que deniegue el acceso mediante `api.access.deny()` cuando, por ejemplo, un usuario intenta acceder a una aplicación o a una API en un momento no permitido (como se describe en este [ejemplo](/docs/es/manage-users/access-control/sample-use-cases-actions-with-authorization#allow-access-only-on-weekdays-for-a-specific-application)) o si el usuario no tiene el/los <Tooltip tip="Atributo empaquetado en un token de seguridad que representa una afirmación que el proveedor del token hace sobre una entidad." cta="Ver glosario" href="/docs/es/glossary?term=Claim">claim</Tooltip> adecuado(s) en su `app_metadata`. En el caso de una aplicación que use [OpenID Connect (OIDC)](/docs/es/authenticate/protocols/openid-connect-protocol), esto impediría la emisión del [token de ID](/docs/es/secure/tokens/id-tokens) utilizado para autorizar el acceso. Del mismo modo, en una API, podría impedirse la emisión de cualquier token de acceso OAuth2 [token de acceso](/docs/es/secure/tokens/access-tokens) (utilizado al llamar a la API), como se describe en este [ejemplo](/docs/es/manage-users/access-control/sample-use-cases-actions-with-authorization#deny-access-to-anyone-calling-an-api).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  En general, hemos comprobado que [OIDC](/docs/es/authenticate/protocols/openid-connect-protocol) es el protocolo estándar del sector más utilizado por los clientes de Auth0 cuando se trata de la autenticación en sus aplicaciones. También hemos comprobado que, aunque [OAuth2](/docs/es/authenticate/protocols/oauth) se creó como un protocolo de delegación, se usa habitualmente en aplicaciones propias cuando hay una API que no comparte una sesión con la aplicación.
</Callout>

Auth0 también puede proporcionar la información necesaria para que una aplicación pueda aplicar restricciones. Para la [integración a nivel de aplicación](#application-integration), Auth0 te permite añadir [claims personalizados](#id-token-claims) a un <Tooltip tip="Token de ID: credencial destinada al propio cliente, en lugar de al acceso a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=ID+Token">token de ID</Tooltip>, que tu aplicación podrá verificar y usar posteriormente para aplicar políticas. En este caso, tendrás que decidir qué información necesitas para que tu aplicación pueda tomar decisiones de aplicación de políticas. Si necesitas tomar decisiones en una API en lugar de en tu aplicación, probablemente tendrás que usar un <Tooltip tip="Token de acceso: credencial de autorización, en forma de cadena opaque o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=Access+Token">token de acceso</Tooltip> en lugar de un token de identidad. Sigue leyendo para obtener más información.

<Warning>
  Al decidir qué datos incluir en tu token de identidad o token de acceso, ten en cuenta el tamaño del token, especialmente si lo pasas en la URL. Aunque no pases tokens en la URL, también tendrás que considerar la posibilidad de exponer PII (información de identificación personal) confidencial. La información del token no está cifrada, por lo que, aunque por lo general no supone un problema de seguridad que se filtre un token de identidad, puede convertirse en un problema de privacidad según los datos que se incluyan en el token.
</Warning>

Para la [integración a nivel de API](#api-integration), Auth0 admite tanto [claims personalizados](#access-token-claims) como la reconfiguración de [scope](#access-token-scopes), ambas en el contexto de un token de acceso. De nuevo, tendrás que decidir qué información será necesaria para que tu API tome decisiones de acceso, y tu API tendrá que aplicarlas validando el contenido del token de acceso.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Al decidir si debes usar permisos mediante claims personalizados o scopes, debes asegurarte de comprender la naturaleza y el propósito de los scopes. Hay una buena [entrada de blog](https://auth0.com/blog/on-the-nature-of-oauth2-scopes/) sobre ello que es fácil de leer y ayuda a aclarar el tema.
</Callout>

<div id="application-integration">
  ## Integración de la aplicación
</div>

En este escenario, tu tenant de Auth0 proporciona un token como prueba de acceso autorizado a una aplicación. En el caso de las aplicaciones que utilizan [OpenID Connect (OIDC)](/docs/es/authenticate/protocols/openid-connect-protocol), el protocolo estándar del sector que suele usarse con más frecuencia en aplicaciones orientadas al cliente, se trataría de un token de ID expresado como un [JWT](/docs/es/secure/tokens/json-web-tokens).

<div id="id-token-claims">
  ### claims del token de ID
</div>

Con la extensibilidad de Action, Auth0 te permite [agregar claims personalizados a un token de ID](/docs/es/get-started/apis/scopes/sample-use-cases-scopes-and-claims#add-custom-claims-to-a-token) basados, por ejemplo, en el contenido de los [Metadatos](/docs/es/manage-users/user-accounts/metadata) de un usuario. Tu aplicación puede entonces verificar que el token de ID contenga los claims necesarios y permitir o impedir el acceso a determinadas funcionalidades, según corresponda. Ten en cuenta que, aunque el proceso de agregar claims personalizados mediante Action está simplificado, el runtime de Action es flexible y te permite escribir código personalizado que puede tener efectos negativos.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Si estás pensando en agregar claims personalizados, te recomendamos almacenar en el [`app_metadata`](/docs/es/manage-users/user-accounts/metadata) del usuario cualquier dato de control de acceso que necesites incluir en los claims. En primer lugar, así evitas tener que llamar a una API externa para obtener esos datos, lo que puede afectar negativamente al rendimiento y la escalabilidad del flujo de inicio de sesión. En segundo lugar, `app_metadata` **no puede** ser modificado por un usuario, por lo que este no puede eludir directamente ninguna restricción de control de acceso modificando sus propios <Tooltip tip="Información que los usuarios pueden actualizar, como preferencias o configuraciones del perfil." cta="Ver glosario" href="/docs/es/glossary?term=Metadata">metadatos</Tooltip>. Recuerda también consultar nuestra guía de [prácticas recomendadas sobre metadatos](/docs/es/get-started/architecture-scenarios/business-to-consumer/profile-management#metadata).
</Callout>

<div id="id-token-scopes">
  ### Scopes del token de ID
</div>

Los [Scopes de OIDC](/docs/es/get-started/apis/scopes/openid-connect-scopes) suelen utilizarse para que una aplicación obtenga el consentimiento para acceder de forma autorizada a los datos de un usuario durante la autenticación. Cada uno de los scopes predefinidos devuelve el conjunto de claims estándar cuando corresponde, tal como se describe en la [especificación de OIDC](https://openid.net/specs/openid-connect-core-1_0.html#StandardClaims). Los scopes que solicita una aplicación dependen de los atributos del usuario que necesite la aplicación. Una vez que el usuario autoriza los scopes solicitados, los claims se devuelven en el token de ID y también están disponibles a través del endpoint [/userinfo](https://auth0.com/docs/api/authentication#get-user-info).

<div id="api-integration">
  ## Integración de API
</div>

En este escenario, tu tenant de Auth0 puede proporcionar un OAuth2 [token de acceso](/docs/es/secure/tokens/access-tokens), normalmente expresado como un [JWT](/docs/es/secure/tokens/json-web-tokens), que tu API puede usar para restringir el acceso a determinados clientes. Además, Auth0 ofrece soporte para lo que se conoce conceptualmente como [aplicaciones propias y de terceros](/docs/es/get-started/applications/first-party-and-third-party-applications).

Al actuar como servidor de autorización, y con el consentimiento del usuario (el propietario del recurso), tu tenant de Auth0 puede usarse para proporcionar un token de acceso —normalmente expresado como un [JWT](/docs/es/secure/tokens/json-web-tokens)— a una aplicación (cliente) para que pueda acceder a un recurso protegido alojado por un <Tooltip tip="Resource Server: Servidor que aloja recursos protegidos. Los servidores de recursos aceptan y responden a solicitudes de recursos protegidos." cta="Ver glosario" href="/docs/es/glossary?term=resource+server">servidor de recursos</Tooltip> en nombre del propietario del recurso. El token de acceso emitido normalmente se envía como token Bearer en el encabezado HTTP Authorization que se manda a una API.

Tanto si tienes una sola API como quizá un conjunto de [API de microservicios](/docs/es/get-started/apis/set-logical-api) relacionadas lógicamente, puedes aprovechar los tokens de acceso que proporciona Auth0 para proteger el acceso a tus servicios. Aunque es relativamente fácil configurar esto en el [Auth0 Dashboard](https://manage.auth0.com/) o mediante la [Auth0 Management API](https://auth0.com/docs/api/management/v2#!/Resource_Servers/post_resource_servers), es importante revisar los distintos escenarios de aplicaciones y diseños de API para determinar la mejor arquitectura para tu sistema.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Los tokens de acceso de OAuth2 están diseñados principalmente para proteger API públicas; cuando se expresan como un JWT, un token de acceso es una entidad autocontenida que puede verificarse sin necesidad de hacer ninguna llamada adicional a una API de terceros. Si tus API no entran en esta categoría, es decir, si forman parte de una aplicación en sí mismas (por ejemplo, si solo esa aplicación las llama) o están detrás de tu firewall, protegerlas con tokens puede resultar excesivo, y tu flujo de trabajo actual basado en cookies (entre otros) probablemente sea suficiente.
</Callout>

OAuth2 se diseñó específicamente pensando en el acceso de terceros; por ejemplo, un escenario podría ser que un usuario (propietario del recurso) quiera usar una aplicación (un cliente) que no pertenece a la misma organización que el servicio que proporciona los datos del usuario (el servidor de recursos). En este caso, cuando la aplicación necesita acceder a datos que pertenecen al usuario, redirige a la organización donde residen los datos del usuario, que a su vez autentica al usuario y luego le solicita que conceda a la aplicación permiso para acceder a sus datos. Esta solicitud de permiso se denomina [consentimiento](/docs/es/get-started/applications/third-party-applications/user-consent-and-third-party-applications) y es una parte importante de lo que implica ofrecer soporte para [aplicaciones de terceros](/docs/es/get-started/apis/scopes/api-scopes#example-an-api-called-by-a-third-party-application). Si planeas integrar aplicaciones de terceros, es importante que las marques como de terceros desde el principio para que Auth0 gestione la solicitud de consentimiento del usuario.

Por otro lado, si tu organización posee las aplicaciones, los propios datos del usuario y las API mediante las que se accede a esos datos, normalmente no se requiere consentimiento, ya que todas las interacciones son [propias](/docs/es/get-started/apis/scopes/api-scopes#example-an-api-called-by-a-first-party-application). Si solo estás creando aplicaciones propias, puedes asegurarte de no mostrar a tus usuarios pantallas de consentimiento innecesarias [permitiendo omitir el consentimiento del usuario](/docs/es/get-started/apis) como parte de cualquier definición de servicio de recursos.

<Warning>
  Aunque puedes configurar tus aplicaciones como propias y posteriormente configurar tus API para permitir que los clientes propios ignoren el consentimiento, si estás usando `localhost` Auth0 no puede verificar que la aplicación sea realmente una aplicación propia, por lo que a tus usuarios se les solicitará el consentimiento de todos modos. Para solucionar esta limitación, al hacer pruebas en tu equipo local durante el desarrollo, crea un [hostname local falso y úsalo en su lugar](https://community.auth0.com/t/how-do-i-skip-the-consent-page-for-my-api-authorization-flow/6035).
</Warning>

Como alternativa, puede que tenga datos relacionados con un usuario para los que se ofrece [funcionalidad adicional](/docs/es/get-started/apis/scopes/api-scopes#example-an-api-called-by-a-back-end-service) y para los que no se puede obtener el consentimiento explícito del usuario (es decir, no hay ningún usuario autenticado que pueda otorgarlo). En este caso, se puede definir una [lista de aplicaciones para las que está habilitada la Concesión de credenciales de cliente](/docs/es/get-started/authentication-and-authorization-flow/client-credentials-flow).

<div id="access-token-claims">
  ### Claims del token de acceso
</div>

Al igual que ocurre con los tokens de ID, puedes [agregar claims personalizados a los tokens de acceso](/docs/es/get-started/apis/scopes/sample-use-cases-scopes-and-claims#add-custom-claims-to-a-token) mediante la extensibilidad de Auth0 Action. Una vez agregados, tu API puede verificar si un token de acceso incluye los claims necesarios y, según corresponda, permitir o impedir el acceso a determinadas funcionalidades.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Si estás pensando en agregar claims personalizados, te recomendamos almacenar en el [`app_metadata`](/docs/es/manage-users/user-accounts/metadata) del usuario cualquier dato de control de acceso que necesites incluir en los claims. En primer lugar, así evitas tener que llamar a una API externa para recuperar esos datos, lo que puede afectar negativamente al rendimiento y la escalabilidad. En segundo lugar, `app_metadata` **no puede** ser modificado por el usuario, por lo que este no puede eludir directamente ninguna restricción de control de acceso modificando sus propios metadatos. Recuerda también consultar nuestra guía de [prácticas recomendadas de metadatos](/docs/es/get-started/architecture-scenarios/business-to-consumer/profile-management#metadata).
</Callout>

<div id="access-token-scopes">
  ### Scopes del token de acceso
</div>

Los [Scopes de OAuth2](/docs/es/get-started/apis/scopes/api-scopes) se usan normalmente como el mecanismo mediante el cual una API puede determinar qué <Tooltip tip="Funciones seguras, específicas del tenant y versionadas escritas en Node." cta="Ver glosario" href="/docs/es/glossary?term=Actions">actions</Tooltip> se pueden realizar en nombre de un usuario. Los scopes pueden añadirse por API para [definir permisos de acceso específicos](/docs/es/get-started/apis/add-api-permissions) en el <Tooltip tip="Auth0 Dashboard: el producto principal de Auth0 para configurar sus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip> o a través de la <Tooltip tip="Auth0 Dashboard: el producto principal de Auth0 para configurar sus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip> de Auth0. Los scopes también pueden modificarse mediante la extensibilidad de Auth0 (por ejemplo, con una Action, como en este [ejemplo](/docs/es/get-started/architecture-scenarios/spa-api/part-2#create-an-action-to-validate-token-scopes)). Los scopes que una aplicación solicita para acceder a una API deben depender de las funciones para las que la aplicación necesita que el usuario le otorgue permiso. Una vez autorizados los scopes solicitados, se devolverán en el token de acceso, que posteriormente puede ser [validado por esa API](/docs/es/secure/tokens/access-tokens/validate-access-tokens). Un buen ejemplo de esto es cuando inicias sesión en una aplicación que usa un proveedor social para iniciar sesión: la API del proveedor social requiere que la aplicación especifique si el usuario quiere que la aplicación publique contenido en su nombre. Esto permite al usuario aceptar o rechazar esta solicitud. Este ejemplo demuestra cómo el usuario está delegando un permiso a la aplicación, lo cual es distinto de que la API restrinja el acceso según el <Tooltip tip="Aspecto de la identidad de un usuario asignado al usuario para indicar el nivel de acceso que debe tener al sistema." cta="Ver glosario" href="/docs/es/glossary?term=Role">rol</Tooltip> de un usuario, y debe gestionarse de otra manera.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  **Práctica recomendada**

  Aunque tienes la capacidad de manipular por completo los scopes del token de acceso mediante la extensibilidad de Auth0, como práctica recomendada de seguridad solo debes eliminar los scopes que no estén autorizados y evitar añadir scopes que no se hayan solicitado.
</Callout>

Aunque los scopes suelen usarse como una forma de aplicar permisos de acceso para un usuario, hay situaciones en las que puede resultar [complicado usarlos de esta manera](https://auth0.com/blog/on-the-nature-of-oauth2-scopes/). Por lo tanto, te recomendamos usar los scopes para el fin previsto (es decir, delegar permisos a una aplicación) y usar [claims personalizados](#access-token-claims) para tus escenarios basados en roles u otros de control de acceso.

<div id="fine-grained-authorization-fga">
  ## Autorización detallada (FGA)
</div>

[Autorización detallada](https://docs.fga.dev/fga-concepts) te permite conceder a usuarios individuales acceso a un recurso u objeto específico en función de lo siguiente:

* El rol de un usuario dentro de una organización, como `editor` o `admin`
* Un atributo del usuario o del objeto, como `manager` para un usuario o `marketing` para un objeto
* Una relación entre un usuario y un objeto; por ejemplo, un usuario con acceso de visualización a una carpeta principal también tiene acceso de visualización a la carpeta secundaria

Con <Tooltip tip="Autorización detallada (FGA): producto de Auth0 que permite a usuarios individuales acceder a objetos o recursos específicos." cta="Ver glosario" href="/docs/es/glossary?term=FGA">FGA</Tooltip>, puedes crear un modelo de autorización para definir qué relación quieres usar para determinar el acceso de los usuarios.

<div id="role-based-access-control-rbac">
  ## Control de acceso basado en roles (RBAC)
</div>

Auth0 ofrece compatibilidad inmediata con el control de acceso basado en roles ([RBAC](/docs/es/manage-users/access-control/rbac)). RBAC consiste en asignar permisos a los usuarios según su rol dentro de una organización y simplifica el control de acceso al ofrecer un enfoque más fácil de gestionar y menos propenso a errores.

<div id="machine-to-machine-m2m-authorization">
  ## Autorización de máquina a máquina (M2M)
</div>

Hay muchos escenarios en los que una aplicación sin ninguna sesión interactiva de usuario necesita obtener un token de acceso para poder llamar a una API. En esos casos, debes autenticar al cliente en lugar del usuario, y <Tooltip tip="OAuth 2.0: Framework de autorización que define protocolos y flujos de autorización." cta="Ver glosario" href="/docs/es/glossary?term=OAuth">OAuth</Tooltip> 2 proporciona el tipo de concesión de [credenciales de cliente](/docs/es/get-started/authentication-and-authorization-flow/client-credentials-flow) para facilitarlo. Algunos ejemplos comunes en los que esto es necesario incluyen:

* Una tarea cron u otro servicio que necesite comunicarse con tu API (p. ej., cuando sea necesario generar un informe diario y enviarlo por correo electrónico a un administrador).
* Una API independiente que admita acceso privilegiado (p. ej., la API no está expuesta directamente a los usuarios, sino únicamente a un backend).
* En ciertas arquitecturas de microservicios, donde algunas capas de API necesitan comunicarse con otras capas de API sin la intervención de un usuario, o después de que haya expirado el token de un usuario.
* Una API privilegiada que quizá deba llamarse antes de que un usuario se haya autenticado (es decir, desde una Action o un script personalizado de base de datos en tu tenant de Auth0)

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Tradicionalmente, se creaba una &quot;cuenta de servicio&quot; para cubrir estos escenarios: un usuario con nombre de usuario y contraseña configurado para servicios que admitían casos de uso no interactivos. Ese enfoque ya no se recomienda por muchas razones, y la práctica recomendada actual es usar la [Concesión de credenciales de cliente de OAuth 2.0](/docs/es/get-started/authentication-and-authorization-flow/client-credentials-flow) en estas situaciones.
</Callout>

<div id="project-planning-guide">
