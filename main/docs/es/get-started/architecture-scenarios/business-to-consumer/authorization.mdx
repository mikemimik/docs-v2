Comencemos dando un paso atrás y hablando del Control de Acceso. No existe una definición única y clara del Control de Acceso en la industria, pero si dedica algo de tiempo a buscar y leer, verá que la mayoría de las fuentes autorizadas coinciden en que es el concepto general que engloba la autenticación, la autorización, el consentimiento y la aplicación de políticas para garantizar que solo las personas y los servicios adecuados tengan acceso a sus aplicaciones y API. A continuación, veamos más de cerca las diferencias entre autenticación, autorización, consentimiento y aplicación de políticas. Su <Tooltip tip="Un grupo de usuarios lógicamente aislado que comparte acceso común con privilegios específicos a una sola instancia de software." cta="Ver glosario" href="/docs/es/glossary?term=Tenant">tenant</Tooltip> de Auth0 (su <Tooltip tip="Servidor de autorización: servidor centralizado que contribuye a definir los límites del acceso de un usuario. Por ejemplo, su servidor de autorización puede controlar los datos, las tareas y las funciones disponibles para un usuario." cta="Ver glosario" href="/docs/es/glossary?term=Authorization+Server">servidor de autorización</Tooltip>) suele ser responsable de la autenticación y el consentimiento, así como de parte o de la totalidad de la autorización y la aplicación de políticas. Además, una <Tooltip tip="Su software que depende de Auth0 para la autenticación y la gestión de identidades." cta="Ver glosario" href="/docs/es/glossary?term=Application">Aplicación</Tooltip> o la propia API casi siempre es quien aplica principalmente las políticas, especialmente cuando se requiere acceso contextual:

* **Autenticación**: el proceso de determinar si un principal (un usuario o una aplicación) es quien o lo que dice ser.
* **Autorización**: el proceso de determinar qué está permitido, en función del principal, los permisos que se le han otorgado y/o el conjunto de criterios de acceso específicos del contexto.
* **Consentimiento**: qué permisos el usuario (<Tooltip tip="Propietario del recurso: entidad (como un usuario o una aplicación) capaz de otorgar acceso a un recurso protegido." cta="Ver glosario" href="/docs/es/glossary?term=Resource+Owner">Propietario del recurso</Tooltip>) ha autorizado a una aplicación a realizar en su nombre. Por lo general, esto es un requisito de la autorización delegada. El usuario tiene que dar permiso al Cliente para acceder a sus datos en otro sistema.
* **Aplicación de políticas**: el acto de aplicar las políticas de la aplicación o la API, rechazando o permitiendo el acceso en función de la información de autenticación y/o autorización de un usuario.

En general, solemos agrupar los distintos tipos de control de acceso en tres categorías diferenciadas para que sea más fácil entender a) qué actor es responsable de almacenar la información, b) qué actor es responsable de tomar decisiones y c) cuál es responsable de aplicar las restricciones.

* La primera categoría es aquella en la que se concede o se deniega el acceso a una aplicación o a una API en su totalidad. Tanto los datos necesarios para hacerlo como el proceso de aplicación suelen definirse en el contexto del servidor de autorización. Por ejemplo, mediante el uso de [`app_metadata`](/docs/es/manage-users/user-accounts/metadata) asociado a un usuario y una [Action](/docs/es/customize/actions) definida en su tenant de Auth0.
* La segunda categoría es aquella en la que se concede o se deniega el acceso a un subconjunto específico de la funcionalidad de la aplicación o la API. Los datos necesarios para hacerlo suelen almacenarse en el servidor de autorización. Por ejemplo, mediante el uso de `app_metadata` en un usuario de su tenant de Auth0, mientras que el proceso de aplicación se lleva a cabo en la propia aplicación o API. En este escenario, los datos suelen comunicarse como uno o más claims personalizados en un token [`id`](/docs/es/secure/tokens/id-tokens) o [`access`](/docs/es/secure/tokens/access-tokens).
* La tercera categoría es aquella en la que se concede o se deniega el acceso en función de sobre qué puede operar el principal (sujeto) dentro del contexto de una aplicación o API. Tanto los datos necesarios para hacerlo como el proceso de aplicación suelen definirse en el contexto de la aplicación o la API. En este escenario, los datos comunicados como uno o más claims personalizados en un token `id` o `access` pueden consumirse con o sin datos de una fuente externa distinta de Auth0.

Además, los mecanismos de control de acceso basado en roles (RBAC) y control de acceso basado en atributos (ABAC) pueden aplicarse en cualquiera de las categorías de Control de Acceso descritas anteriormente. Sea cual sea su caso de uso, hay una serie de aspectos que querrá tener en cuenta al analizar la funcionalidad y el flujo de trabajo que necesita:

* ¿Hay escenarios en los que deba rechazarse el acceso a una aplicación o API completa?
* ¿Proporcionará API a las que puedan acceder aplicaciones de terceros?
* ¿Sus API también podrán ser accedidas por sus propias aplicaciones (propias)?
* ¿Su aplicación llamará a una API de terceros?
* ¿Sus aplicaciones y/o API deberían aplicar el control de acceso en función de los claims del usuario?

Auth0 admite restringir el acceso a aplicaciones o APIs en función de determinadas condiciones. En algunos casos, quizá quieras crear una Action que deniegue el acceso con `api.access.deny()` cuando, por ejemplo, un usuario intenta acceder a una aplicación o a una API en un momento no permitido (como se describe en este [ejemplo](/docs/es/manage-users/access-control/sample-use-cases-actions-with-authorization#allow-access-only-on-weekdays-for-a-specific-application)) o si el usuario no tiene los <Tooltip tip="Atributo empaquetado en un token de seguridad que representa un claim que el emisor del token está haciendo sobre una entidad." cta="Ver glosario" href="/docs/es/glossary?term=Claim">claim</Tooltip>(s) correctos en su `app_metadata`. En el caso de una aplicación que usa [OpenID Connect (OIDC)](/docs/es/authenticate/protocols/openid-connect-protocol), esto impediría la emisión del [token de ID](/docs/es/secure/tokens/id-tokens) utilizado para autorizar el acceso. De forma similar, en una API, podría impedirse la emisión de cualquier token OAuth2 [token de acceso](/docs/es/secure/tokens/access-tokens) (utilizado al llamar a la API), como se describe en este [ejemplo](/docs/es/manage-users/access-control/sample-use-cases-actions-with-authorization#deny-access-to-anyone-calling-an-api).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  En general, hemos comprobado que [OIDC](/docs/es/authenticate/protocols/openid-connect-protocol) es el protocolo estándar del sector que más utilizan los clientes de Auth0 para la autenticación en sus aplicaciones. También hemos comprobado que, aunque [OAuth2](/docs/es/authenticate/protocols/oauth) se creó como un protocolo de delegación, se usa habitualmente en aplicaciones propias cuando hay una API que no comparte una sesión con la aplicación.
</Callout>

Auth0 también puede proporcionar la información necesaria para que una aplicación pueda aplicar restricciones. Para la [integración a nivel de aplicación](#application-integration), Auth0 te permite añadir [claims personalizadas](#id-token-claims) a un <Tooltip tip="Token de ID: credencial destinada al propio cliente, en lugar de al acceso a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=ID+Token">token de ID</Tooltip>, que tu aplicación puede verificar y usar después para aplicar políticas. En este caso, tendrás que decidir qué información necesitas para que tu aplicación pueda tomar decisiones de control. Si necesitas tomar decisiones en una API en lugar de hacerlo en tu aplicación, probablemente tendrás que usar un <Tooltip tip="Token de acceso: credencial de Authorization, en forma de cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=Access+Token">token de acceso</Tooltip> en lugar de un token de ID. Sigue leyendo para obtener más información.

<Warning>
  Al decidir qué datos incluir en tu token de ID y/o token de acceso, ten en cuenta el tamaño del token, especialmente si lo pasas en la URL. Aunque no estés pasando tokens en la URL, también tendrás que considerar la posibilidad de exponer PII (información de identificación personal) sensible. La información del token no está cifrada, así que, aunque por lo general no supone un problema de seguridad que se filtre un token de ID, puede convertirse en un problema de privacidad según los datos que incluya el token.
</Warning>

Para la [integración a nivel de API](#api-integration), Auth0 admite tanto [claims personalizadas](#access-token-claims) como la reconfiguración de [scope](#access-token-scopes), ambas en el contexto de un token de acceso. De nuevo, tendrás que decidir qué información será necesaria para que tu API tome decisiones de acceso, y tu API deberá aplicarlas validando el contenido del token de acceso.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Al decidir si debes usar permisos mediante claims personalizadas o scopes, debes asegurarte de entender la naturaleza y el propósito de los scopes. Hay una buena [entrada de blog](https://auth0.com/blog/on-the-nature-of-oauth2-scopes/) sobre esto que es fácil de leer y ayuda a aclarar el tema.
</Callout>

<div id="application-integration">
  ## Integración de aplicaciones
</div>

En este escenario, tu tenant de Auth0 proporciona un token como indicador de acceso autorizado a una aplicación. En el caso de las aplicaciones que utilizan [OpenID Connect (OIDC)](/docs/es/authenticate/protocols/openid-connect-protocol), el protocolo estándar del sector que suele usarse con más frecuencia en aplicaciones orientadas al cliente, este sería un token de ID expresado como un [JWT](/docs/es/secure/tokens/json-web-tokens).

<div id="id-token-claims">
  ### Claims del token de ID
</div>

Con la extensibilidad de Action, Auth0 te permite [agregar claims personalizados a un token de ID](/docs/es/get-started/apis/scopes/sample-use-cases-scopes-and-claims#add-custom-claims-to-a-token) en función de, por ejemplo, el contenido de la [Metadata](/docs/es/manage-users/user-accounts/metadata) de un usuario. Luego, tu aplicación puede verificar el token de ID para comprobar los claims necesarios y permitir o impedir el acceso a determinadas funcionalidades, según corresponda. Ten en cuenta que, aunque el proceso de agregar claims personalizados mediante Action es sencillo, el entorno de ejecución de Action es flexible y te permite escribir código personalizado que podría tener efectos negativos.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Si estás pensando en agregar claims personalizados, te recomendamos almacenar en la [`app_metadata`](/docs/es/manage-users/user-accounts/metadata) del usuario cualquier dato de control de acceso que necesites incluir en los claims. En primer lugar, esto evita que tengas que llamar a una API externa para obtener los datos, lo que puede afectar negativamente al rendimiento y la escalabilidad del proceso de inicio de sesión. En segundo lugar, `app_metadata` **no puede** ser modificada por el usuario, por lo que este no puede eludir directamente ninguna restricción de control de acceso modificando su propia <Tooltip tip="Información que los usuarios pueden actualizar, como preferencias o configuraciones de perfil." cta="Ver glosario" href="/docs/es/glossary?term=Metadata">metadata</Tooltip>. Además, recuerda consultar también nuestra guía de [prácticas recomendadas para metadata](/docs/es/get-started/architecture-scenarios/business-to-consumer/profile-management#metadata).
</Callout>

<div id="id-token-scopes">
  ### Scopes del token de ID
</div>

Los [OIDC Scopes](/docs/es/get-started/apis/scopes/openid-connect-scopes) suelen utilizarse en una aplicación para obtener el consentimiento de acceso autorizado a los datos de un usuario durante la autenticación. Cada uno de los scopes predefinidos devuelve el conjunto de claims estándar cuando corresponde, tal como se describe en la [especificación de OIDC](https://openid.net/specs/openid-connect-core-1_0.html#StandardClaims). Los scopes que solicita una aplicación dependen de los atributos del usuario que necesite. Una vez que el usuario autoriza los scopes solicitados, los claims se devuelven en el token de ID y también están disponibles a través del endpoint [/userinfo](https://auth0.com/docs/api/authentication#get-user-info).

<div id="api-integration">
  ## Integración de API
</div>

En este escenario, tu tenant de Auth0 puede proporcionar un OAuth2 [token de acceso](/docs/es/secure/tokens/access-tokens), normalmente expresado como un [JWT](/docs/es/secure/tokens/json-web-tokens), que tu API puede usar para restringir el acceso a determinadas partes. Además, Auth0 ofrece soporte para lo que, conceptualmente, se describe como [aplicaciones propias y de terceros](/docs/es/get-started/applications/first-party-and-third-party-applications).

Actuando como servidor de autorización y con el consentimiento del usuario (el propietario del recurso), tu tenant de Auth0 puede usarse para proporcionar un token de acceso —normalmente expresado como un [JWT](/docs/es/secure/tokens/json-web-tokens)— a una aplicación (cliente) para que pueda acceder a un recurso protegido alojado por un <Tooltip tip="Servidor de recursos: servidor que aloja recursos protegidos. Los servidores de recursos aceptan y responden a solicitudes de recursos protegidos." cta="Ver glosario" href="/docs/es/glossary?term=resource+server">servidor de recursos</Tooltip> en nombre del propietario del recurso. El token de acceso emitido suele enviarse como token Bearer en el encabezado HTTP Authorization de una solicitud enviada a una API.

Tanto si tienes una sola API como un conjunto de [API de microservicios](/docs/es/get-started/apis/set-logical-api) relacionadas lógicamente, puedes aprovechar los tokens de acceso que proporciona Auth0 para proteger el acceso a tus servicios. Aunque configurarlo es relativamente fácil en el [Auth0 Dashboard](https://manage.auth0.com/) o mediante la [Auth0 Management API](https://auth0.com/docs/api/management/v2#!/Resource_Servers/post_resource_servers), es importante revisar los distintos escenarios de aplicaciones y diseños de API para determinar la mejor arquitectura para tu sistema.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Los tokens de acceso de OAuth2 están diseñados principalmente para proteger API públicas; cuando se expresa como JWT, un token de acceso es una entidad autocontenida que puede verificarse sin necesidad de hacer ninguna llamada adicional a una API de terceros. Si tus API no entran en esta categoría —es decir, si forman parte de una aplicación en sí misma (por ejemplo, si solo esa aplicación las llama) o están detrás de tu firewall—, entonces protegerlas con tokens puede ser excesivo, y es posible que tu flujo de trabajo actual basado en cookies (entre otros) sea suficiente.
</Callout>

OAuth2 se diseñó específicamente pensando en el acceso de terceros; por ejemplo, un escenario podría ser aquel en el que un usuario (propietario del recurso) quiere usar una aplicación (un cliente) que no pertenece a la misma organización que el servicio que proporciona los datos del usuario (el servidor de recursos). En este caso, cuando la aplicación necesita acceder a datos que pertenecen al usuario, redirige a la organización donde residen esos datos, que a su vez autentica al usuario y luego le solicita que conceda permiso a la aplicación para acceder a ellos. Esta solicitud de permiso se conoce como [consentimiento](/docs/es/get-started/applications/third-party-applications/user-consent-and-third-party-applications) y es una parte fundamental de lo que implica ofrecer soporte para [aplicaciones de terceros](/docs/es/get-started/apis/scopes/api-scopes#example-an-api-called-by-a-third-party-application). Si planeas integrar aplicaciones de terceros, es importante que las marques como de terceros desde el principio para que Auth0 gestione la solicitud de consentimiento del usuario.

Por otro lado, si tu organización posee la(s) aplicación(es), los propios datos del usuario y la(s) API a través de las cuales se accede a esos datos, normalmente no se requiere consentimiento, ya que todas las interacciones son [propias](/docs/es/get-started/apis/scopes/api-scopes#example-an-api-called-by-a-first-party-application). Si solo estás creando aplicaciones propias, puedes asegurarte de no mostrar a tus usuarios pantallas de consentimiento innecesarias [permitiendo omitir el consentimiento del usuario](/docs/es/get-started/apis) como parte de cualquier definición de servicio de recursos.

<Warning>
  Aunque puedes configurar tus aplicaciones como propias y, posteriormente, configurar tus API para permitir que los clientes propios omitan el consentimiento, si estás usando `localhost`, Auth0 no puede verificar que la aplicación sea realmente una aplicación propia, por lo que de todos modos se pedirá el consentimiento a tus usuarios. Para solucionar esta limitación, al hacer pruebas en tu máquina local durante el desarrollo, crea un [nombre de host local ficticio y úsalo en su lugar](https://community.auth0.com/t/how-do-i-skip-the-consent-page-for-my-api-authorization-flow/6035).
</Warning>

Como alternativa, puede tener datos relacionados con un usuario para los que se ofrece [funcionalidad adicional](/docs/es/get-started/apis/scopes/api-scopes#example-an-api-called-by-a-back-end-service) y para los que no puede obtenerse el consentimiento explícito del usuario (es decir, no hay ningún usuario autenticado que pueda proporcionarlo). En este caso, puede definirse una [lista de aplicaciones para las que esté habilitada la concesión de Client Credentials](/docs/es/get-started/authentication-and-authorization-flow/client-credentials-flow).

<div id="access-token-claims">
  ### Claims del token de acceso
</div>

Al igual que ocurre con los tokens de ID, puedes [agregar claims personalizadas a los tokens de acceso](/docs/es/get-started/apis/scopes/sample-use-cases-scopes-and-claims#add-custom-claims-to-a-token) mediante la extensibilidad de Auth0 Action. Una vez agregadas, tu API podrá verificar si un token de acceso contiene las claims necesarias y, según corresponda, permitir o denegar el acceso a determinadas funcionalidades.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Si estás pensando en agregar claims personalizadas, te recomendamos almacenar en la [`app_metadata`](/docs/es/manage-users/user-accounts/metadata) del usuario cualquier dato de control de acceso que necesites incluir en las claims. En primer lugar, así evitas tener que llamar a una API externa para recuperar esos datos, lo que puede afectar negativamente al rendimiento y la escalabilidad. En segundo lugar, `app_metadata` **no** puede ser modificada por el usuario, por lo que este no podrá eludir directamente ninguna restricción de control de acceso modificando sus propios metadatos. Además, recuerda consultar también nuestra guía de [buenas prácticas sobre metadatos](/docs/es/get-started/architecture-scenarios/business-to-consumer/profile-management#metadata).
</Callout>

<div id="access-token-scopes">
  ### Scopes del token de acceso
</div>

Los [OAuth2 Scopes](/docs/es/get-started/apis/scopes/api-scopes) se suelen usar como el mecanismo mediante el cual una API puede determinar qué <Tooltip tip="Funciones seguras, específicas del tenant y versionadas escritas en Node." cta="Ver glosario" href="/docs/es/glossary?term=Actions">acciones</Tooltip> se pueden realizar en nombre de un usuario. Los scopes se pueden agregar para cada API a fin de [definir permisos de acceso específicos](/docs/es/get-started/apis/add-api-permissions) en el <Tooltip tip="Auth0 Dashboard: el producto principal de Auth0 para configurar tus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip> o mediante la <Tooltip tip="Auth0 Dashboard: el producto principal de Auth0 para configurar tus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip> de Auth0. Los scopes también se pueden manipular mediante la extensibilidad de Auth0 (por ejemplo, a través de una Action, como en este [ejemplo](/docs/es/get-started/architecture-scenarios/spa-api/part-2#create-an-action-to-validate-token-scopes)). Los scopes que una aplicación solicita para acceder a una API deben depender de la funcionalidad para la que la aplicación necesite el permiso del usuario. Una vez autorizados los scopes solicitados, se devolverán en el token de acceso, que posteriormente puede ser [verificado por dicha API](/docs/es/secure/tokens/access-tokens/validate-access-tokens). Un buen ejemplo de esto es cuando inicias sesión en una aplicación que usa un proveedor social para el inicio de sesión: la API del proveedor social requiere que la aplicación especifique si el usuario quiere que la aplicación publique contenido en su nombre. Esto permite al usuario aceptar o rechazar esta solicitud. Este ejemplo demuestra cómo el usuario está delegando permisos a la aplicación, lo cual es diferente de que la API restrinja el acceso según el <Tooltip tip="Aspecto de la identidad de un usuario asignado al usuario para indicar el nivel de acceso que debe tener al sistema." cta="Ver glosario" href="/docs/es/glossary?term=Role">rol</Tooltip> de un usuario, y debe gestionarse de forma diferente.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  **Práctica recomendada**

  Aunque tienes la capacidad de manipular por completo los scopes del token de acceso mediante la extensibilidad de Auth0, como práctica recomendada de seguridad solo deberías eliminar los scopes que no estén autorizados y abstenerte de agregar scopes que no se hayan solicitado.
</Callout>

Aunque los scopes se usan a menudo como una forma de aplicar permisos de acceso para un usuario, hay situaciones en las que puede resultar [complicado usarlos de esta manera](https://auth0.com/blog/on-the-nature-of-oauth2-scopes/). Por lo tanto, te recomendamos que uses los scopes para su propósito previsto (es decir, delegar permisos a una aplicación) y uses [claims personalizados](#access-token-claims) para tus escenarios basados en roles u otros controles de acceso.

<div id="fine-grained-authorization-fga">
  ## Autorización de grano fino (FGA)
</div>

[Autorización de grano fino](https://docs.fga.dev/fga-concepts) te permite otorgar a usuarios individuales acceso a un recurso u objeto específico en función de:

* El rol de un usuario dentro de una organización, como `editor` o `admin`
* Un atributo del usuario o del objeto, como `manager` para un usuario o `marketing` para un objeto
* Una relación entre un usuario y un objeto, como cuando un usuario con acceso de visualización a una carpeta principal también tiene acceso de visualización a la carpeta secundaria

Con <Tooltip tip="Autorización de grano fino (FGA): producto de Auth0 que permite a usuarios individuales acceder a objetos o recursos específicos." cta="Ver glosario" href="/docs/es/glossary?term=FGA">FGA</Tooltip>, puedes crear un modelo de autorización para definir las relaciones con las que se determinará el acceso de los usuarios.

<div id="role-based-access-control-rbac">
  ## Control de acceso basado en roles (RBAC)
</div>

Auth0 ofrece compatibilidad lista para usar con el control de acceso basado en roles ([RBAC](/docs/es/manage-users/access-control/rbac)). RBAC consiste en asignar permisos a los usuarios en función de su rol dentro de una organización, lo que simplifica el control de acceso al ofrecer un enfoque más fácil de administrar y menos propenso a errores.

<div id="machine-to-machine-m2m-authorization">
  ## Autorización de máquina a máquina (M2M)
</div>

Hay muchos escenarios en los que una aplicación sin ninguna sesión interactiva de usuario necesita obtener un token de acceso para poder llamar a una API. En esos casos, debes autenticar al cliente en lugar de al usuario, y <Tooltip tip="OAuth 2.0: Marco de autorización que define protocolos y flujos de trabajo de autorización." cta="Ver glosario" href="/docs/es/glossary?term=OAuth">OAuth</Tooltip> 2 proporciona el tipo de concesión de [credenciales de cliente](/docs/es/get-started/authentication-and-authorization-flow/client-credentials-flow) para facilitarlo. Algunos ejemplos comunes en los que esto es necesario son:

* Un trabajo cron u otro servicio que necesite comunicarse con tu API (p. ej., cuando se debe generar un informe diario y enviárselo por correo electrónico a un administrador).
* Una API independiente que permite acceso privilegiado (p. ej., la API no está expuesta directamente a los usuarios, sino solo al backend).
* En ciertas arquitecturas de microservicios, donde algunas capas de API necesitan comunicarse con otras capas de API sin la intervención de un usuario, o después de que haya expirado el token de un usuario.
* Una API privilegiada a la que puede ser necesario llamar antes de que un usuario se haya autenticado (es decir, desde una Action o un script de base de datos personalizado en tu tenant de Auth0)

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Tradicionalmente, se creaba una &quot;cuenta de servicio&quot; especial para cubrir estos escenarios: un usuario con nombre de usuario y contraseña configurado para servicios compatibles con casos de uso no interactivos. Este ya no es un enfoque recomendado por muchas razones, y la práctica recomendada actualmente es usar [OAuth 2.0 Client Credentials Grant](/docs/es/get-started/authentication-and-authorization-flow/client-credentials-flow) en estas situaciones.
</Callout>
