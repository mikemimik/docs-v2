title: Automatización del despliegue (B2C)
---

Además de adoptar las mejores prácticas para la gestión de cambios y [QA](/docs/es/get-started/architecture-scenarios/business-to-consumer/quality-assurance), los clientes que tienen éxito también integran la gestión de recursos de Auth0 como parte de un proceso de despliegue automatizado. Como se explica en la sección de Arquitectura, en [compatibilidad con SDLC](/docs/es/get-started/architecture-scenarios/business-to-consumer/architecture#sdlc-support), le convendrá asegurarse de configurar tenants de Auth0 independientes para los entornos de desarrollo, pruebas y producción, y de que la configuración del tenant en cada entorno sea casi idéntica. El uso de la automatización del despliegue ayuda a garantizar esto, de modo que el tenant de cada entorno esté configurado de la misma forma y sea menos probable que aparezcan errores como resultado de configuraciones inconsistentes entre entornos.

<Info>
  ### Buena práctica

  Independientemente de cómo configure la automatización del despliegue, le recomendamos que haga pruebas unitarias de sus reglas, scripts de base de datos personalizados y hooks antes del despliegue, y que también ejecute algunas pruebas de integración en su tenant después del despliegue. Para obtener más información al respecto, consulte la guía de [Aseguramiento de la calidad](/docs/es/get-started/architecture-scenarios/business-to-consumer/quality-assurance).
</Info>

Auth0 ofrece dos opciones distintas en lo que respecta a los enfoques de automatización del despliegue que puede utilizar, y ambas pueden usarse conjuntamente si así lo desea:

* La [herramienta Auth0 Deploy CLI](/docs/es/deploy-monitor/deploy-cli-tool) le proporciona un script fácil de usar que puede ayudarle a integrarse con su canalización actual de integración continua/implementación continua (CI/CD).
* Si no puede integrarse directamente con una canalización de CI/CD, o si por algún motivo no tiene una, las [Extensiones de control de código fuente](/docs/es/customize/extensions) de Auth0 pueden proporcionarle un proceso básico de automatización, fácil de configurar y con muy poco mantenimiento.

<Warning>
  Tenga en cuenta que tanto Deploy CLI Tool como las extensiones de control de código fuente pueden provocar cambios destructivos; los cambios manuales realizados directamente en el panel entre despliegues automatizados podrían perderse. Por este motivo, si se utiliza cualquiera de las dos opciones, **todos** los cambios deben desplegarse desde el subsistema de control de código fuente al que hace referencia la herramienta, y no realizarse manualmente.
</Warning>

Cada entorno también puede necesitar cierta configuración específica; por ejemplo, los <Tooltip tip="Client ID: Identification value given to your registered resource from Auth0." cta="View Glossary" href="/docs/es/glossary?term=Client+ID">Client ID</Tooltip> de la aplicación y los <Tooltip tip="Client Secret: Secret used by a client (application) to authenticate with the Authorization Server; it should be known to only the client and the Authorization Server and must be sufficiently random to not be guessable." cta="View Glossary" href="/docs/es/glossary?term=Client+Secrets">Client Secrets</Tooltip> serán diferentes entre los tenants de Auth0, por lo que le convendrá contar con alguna forma de hacer referencia a ellos dinámicamente en lugar de tener valores codificados de forma fija. Auth0 ofrece compatibilidad para gestionar la información de configuración específica del entorno mediante uno de estos dos enfoques:

* Usar [variables específicas del tenant](#tenant-specific-variables)
* Usar reemplazo de palabras clave si utiliza la herramienta Auth0 Deploy CLI

<div id="tenant-specific-variables">
  ## Variables específicas del tenant
</div>

Auth0 te permite configurar variables disponibles desde la [extensibilidad](/docs/es/customize/extensions) personalizada; pueden entenderse como variables de entorno para tu tenant de Auth0. En lugar de codificar de forma fija referencias que cambian al mover código entre entornos de desarrollo, prueba y producción, puedes usar un nombre de variable configurado en el tenant al que haga referencia el código de extensibilidad personalizado. Esto facilita que el mismo código personalizado funcione sin cambios en distintos tenants, ya que el código puede hacer referencia a variables que se completarán con valores específicos del tenant en el momento de la ejecución:

* Para usar variables en Actions, consulta [Write Your First Action](/docs/es/customize/actions/write-your-first-action) para aprender a configurar secretos en el editor
* Para usar variables en Rules, consulta cómo [configurar valores](/docs/es/customize/rules/configuration)
* Para usar variables en Hooks, consulta cómo configurar [secretos](/docs/es/customize/hooks/hook-secrets) en el editor
* Para usar variables en Custom DB Scripts, consulta los [parámetros de configuración](/docs/es/authenticate/database-connections/custom-db/create-db-connection#step-3-add-configuration-parameters)

<Info>
  ### Buena práctica

  Se recomienda usar variables para almacenar valores específicos del tenant, así como cualquier secreto confidencial que no deba exponerse en tu código personalizado. Si tu código personalizado se despliega en GitHub/Gitlab/Bitbucket/VSTS, usar una variable específica del tenant evita exponer valores sensibles en tu repositorio.
</Info>
