title: Automatización de despliegues (B2C)
---

Además de adoptar prácticas recomendadas para la gestión de cambios y [QA](/docs/es/get-started/architecture-scenarios/business-to-consumer/quality-assurance), los clientes que tienen éxito también integran la gestión de recursos de Auth0 como parte de un proceso de despliegue automatizado. Como se explica en la sección de Arquitectura, en [soporte para SDLC](/docs/es/get-started/architecture-scenarios/business-to-consumer/architecture#sdlc-support), debe asegurarse de configurar tenants de Auth0 independientes para los entornos de desarrollo, pruebas y producción, y de que esa configuración sea casi idéntica en el tenant de cada entorno. Usar automatización de despliegues ayuda a garantizarlo, de modo que el tenant de cada entorno quede configurado de la misma forma y sea menos probable que aparezcan errores como resultado de configuraciones inconsistentes entre entornos.

<Info>
  ### Práctica recomendada

  Independientemente de cómo configure la automatización de despliegues, le recomendamos realizar pruebas unitarias de sus rules, scripts de Custom DB y hooks antes del despliegue, y ejecutar también algunas pruebas de integración en su tenant después del despliegue. Para obtener más información al respecto, consulte la guía de [Quality Assurance](/docs/es/get-started/architecture-scenarios/business-to-consumer/quality-assurance).
</Info>

Auth0 ofrece soporte para un par de opciones diferentes en cuanto a los enfoques de automatización de despliegues que puede usar, y cada una puede utilizarse junto con la otra si así lo desea:

* La [herramienta Auth0 Deploy CLI](/docs/es/deploy-monitor/deploy-cli-tool) le proporciona un script fácil de usar que puede ayudarle a integrarse con su pipeline actual de Integración Continua/Despliegue Continuo (CI/CD).
* Si no puede integrarse directamente con una pipeline de CI/CD o, por alguna razón, no dispone de una, las [Source Control Extensions de Auth0](/docs/es/customize/extensions) pueden ofrecerle un proceso básico de automatización fácil de configurar y con muy poco mantenimiento.

<Warning>
  Tenga en cuenta que tanto la herramienta Deploy CLI como las extensiones de control de código fuente pueden provocar cambios destructivos; los cambios manuales realizados directamente en el Dashboard entre despliegues automatizados podrían perderse. Por este motivo, si se utiliza cualquiera de las dos, **todos** los cambios deben desplegarse desde el subsistema de control de código fuente al que hace referencia la herramienta, y no realizarse manualmente.
</Warning>

Cada entorno también puede necesitar cierta configuración específica del entorno; por ejemplo, los <Tooltip tip="ID de cliente: Valor de identificación asignado por Auth0 a su recurso registrado." cta="Ver glosario" href="/docs/es/glossary?term=Client+ID">ID de cliente</Tooltip> y los <Tooltip tip="Secreto de cliente: Secreto utilizado por un cliente (aplicación) para autenticarse con el servidor de autorización; solo deberían conocerlo el cliente y el servidor de autorización, y debe ser lo bastante aleatorio como para que no pueda adivinarse." cta="Ver glosario" href="/docs/es/glossary?term=Client+Secrets">Secretos de cliente</Tooltip> de la Application serán distintos entre los tenants de Auth0, por lo que le convendrá alguna forma de hacer referencia a esto dinámicamente en lugar de usar valores codificados de forma rígida. Auth0 ofrece soporte para gestionar información de configuración específica del entorno mediante uno de los dos enfoques siguientes:

* Usar [Variables específicas del tenant](#tenant-specific-variables)
* Usar reemplazo de palabras clave si utiliza la herramienta Auth0 Deploy CLI

<div id="tenant-specific-variables">
  ## Variables específicas del tenant
</div>

Auth0 permite configurar variables que están disponibles desde la [extensibilidad](/docs/es/customize/extensions) personalizada; pueden considerarse variables de entorno para su tenant de Auth0. En lugar de incluir de forma fija referencias que cambian al mover código entre entornos de desarrollo, prueba y producción, puede usar un nombre de variable configurado en el tenant y al que haga referencia el código de extensibilidad personalizado. Esto facilita que el mismo código personalizado funcione, sin cambios, en distintos tenants, ya que puede hacer referencia a variables que se completarán con valores específicos del tenant en tiempo de ejecución:

* Para usar variables en Actions, consulte [Write Your First Action](/docs/es/customize/actions/write-your-first-action) para aprender a configurar secrets en el editor
* Para usar variables en Rules, vea cómo [configurar valores](/docs/es/customize/rules/configuration)
* Para usar variables en Hooks, vea cómo configurar [secrets](/docs/es/customize/hooks/hook-secrets) en el editor
* Para usar variables en scripts de Custom DB, consulte los [parámetros de configuración](/docs/es/authenticate/database-connections/custom-db/create-db-connection#step-3-add-configuration-parameters)

<Info>
  ### Práctica recomendada

  Se recomienda usar variables para almacenar valores específicos del tenant, así como cualquier secret sensible que no deba exponerse en su código personalizado. Si su código personalizado está en GitHub/Gitlab/Bitbucket/VSTS, usar una variable específica del tenant evita exponer valores sensibles a través de su repositorio.
</Info>
