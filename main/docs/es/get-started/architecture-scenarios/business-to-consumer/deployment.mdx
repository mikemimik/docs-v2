title: Automatización del despliegue (B2C)
---

Además de adoptar prácticas recomendadas para la gestión de cambios y [QA](/docs/es/get-started/architecture-scenarios/business-to-consumer/quality-assurance), los clientes que tienen éxito también integran la gestión de recursos de Auth0 como parte de un proceso de despliegue automatizado. Como se explica en la sección de Architecture, en [soporte para SDLC](/docs/es/get-started/architecture-scenarios/business-to-consumer/architecture#sdlc-support), le convendrá configurar tenants de Auth0 independientes para los entornos de desarrollo, pruebas y producción, y procurar que esa configuración sea casi idéntica en el tenant de cada entorno. La automatización del despliegue ayuda a garantizarlo, ya que permite que el tenant de cada entorno se configure de la misma forma y reduce la probabilidad de que aparezcan errores como resultado de configuraciones inconsistentes entre entornos.

<Info>
  ### Práctica recomendada

  Independientemente de cómo configure la automatización del despliegue, le recomendamos ejecutar pruebas unitarias para sus Rules, scripts de Custom DB y hooks antes del despliegue, y también realizar algunas pruebas de integración en su tenant después del despliegue. Para obtener más información al respecto, consulte la guía de [Quality Assurance](/docs/es/get-started/architecture-scenarios/business-to-consumer/quality-assurance).
</Info>

Auth0 ofrece soporte para un par de opciones distintas en cuanto a los enfoques de automatización del despliegue que puede utilizar, y ambas pueden usarse conjuntamente si así lo desea:

* La [herramienta Auth0 Deploy CLI](/docs/es/deploy-monitor/deploy-cli-tool) le proporciona un script fácil de usar que puede ayudarle a integrarse con su pipeline actual de Continuous Integration/Continuous Deployment (CI/CD).
* Si no puede integrarse directamente con una pipeline de CI/CD, o si por algún motivo no tiene una, entonces las [Source Control Extensions de Auth0](/docs/es/customize/extensions) pueden proporcionarle un proceso básico de automatización fácil de configurar y con muy poco mantenimiento.

<Warning>
  Tenga en cuenta que tanto la herramienta Deploy CLI como las Source Control Extensions pueden provocar cambios destructivos; los cambios manuales realizados directamente en el Dashboard entre despliegues automatizados podrían perderse. Por esta razón, si se utiliza cualquiera de las dos, **todos** los cambios deben desplegarse desde el subsistema de control de código fuente al que hacen referencia estas herramientas, y no realizarse manualmente.
</Warning>

Cada entorno también puede necesitar cierta configuración específica del entorno; por ejemplo, los <Tooltip tip="ID de cliente: Valor de identificación dado por Auth0 a su recurso registrado." cta="Ver glosario" href="/docs/es/glossary?term=Client+ID">ID de cliente</Tooltip> y los <Tooltip tip="Secreto de cliente: Secreto usado por un cliente (aplicación) para autenticarse con el servidor de autorización; solo deben conocerlo el cliente y el servidor de autorización, y debe ser lo suficientemente aleatorio como para que no pueda adivinarse." cta="Ver glosario" href="/docs/es/glossary?term=Client+Secrets">Secretos de cliente</Tooltip> de la aplicación serán distintos entre los tenants de Auth0, por lo que necesitará alguna manera de hacer referencia a ellos dinámicamente en lugar de usar valores hard-coded. Auth0 ofrece soporte para gestionar la información de configuración específica del entorno mediante uno de estos dos enfoques:

* Use [Variables específicas del tenant](#tenant-specific-variables)
* Use reemplazo de palabras clave si utiliza la herramienta Auth0 Deploy CLI

<div id="tenant-specific-variables">
  ## Variables específicas del tenant
</div>

Auth0 le permite configurar variables disponibles desde la [extensibilidad](/docs/es/customize/extensions) personalizada; puede considerarlas variables de entorno para su tenant de Auth0. En lugar de incluir de forma fija referencias que cambian al mover código entre entornos de desarrollo, pruebas y producción, puede usar un nombre de variable configurado en el tenant al que hace referencia el código de extensibilidad personalizado. Esto facilita que el mismo código personalizado funcione sin cambios en distintos tenants, ya que puede hacer referencia a variables que se rellenarán con valores específicos del tenant en el momento de la ejecución:

* Para usar variables en Actions, consulte [Write Your First Action](/docs/es/customize/actions/write-your-first-action) para aprender a configurar secretos en el editor
* Para usar variables en Rules, vea cómo [configurar valores](/docs/es/customize/rules/configuration)
* Para usar variables en Hooks, vea cómo configurar [secretos](/docs/es/customize/hooks/hook-secrets) en el editor
* Para usar variables en scripts de Custom DB, consulte los [parámetros de configuración](/docs/es/authenticate/database-connections/custom-db/create-db-connection#step-3-add-configuration-parameters)

<Info>
  ### Práctica recomendada

  Se recomienda usar variables para almacenar valores específicos del tenant, así como cualquier secreto sensible que no deba exponerse en su código personalizado. Si su código personalizado está implementado en GitHub/GitLab/Bitbucket/VSTS, usar una variable específica del tenant evita exponer valores sensibles a través de su repositorio.
</Info>
