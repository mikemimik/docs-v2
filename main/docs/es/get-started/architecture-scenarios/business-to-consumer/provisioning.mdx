Es importante definir desde el principio cómo se registrarán los usuarios, ya que las decisiones que tome aquí influirán en muchas de las que deberá tomar más adelante. Hemos visto que existe un conjunto habitual de patrones sobre cómo se incorporan los usuarios a su sistema, así como algunos aspectos que conviene tener en cuenta al diseñar los flujos de trabajo.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Aunque Auth0 admite numerosos flujos de trabajo, los flujos de trabajo web que usan Auth0 [Universal Login](/docs/es/authenticate/login/auth0-universal-login) para el registro se consideran una práctica recomendada tanto del sector como de Auth0, ya que ofrecen una funcionalidad óptima y el mejor nivel de seguridad.
</Callout>

Auth0 admite el registro de usuarios a través de varios [proveedores de identidad](/docs/es/authenticate/identity-providers). Durante el registro, Auth0 aprovisiona el perfil de usuario para que contenga la información de la cuenta del usuario. Hay varios aspectos que debe considerar al evaluar la funcionalidad y el flujo de trabajo:

* ¿Debería usar Auth0 como almacén de identidades?
* ¿Puede usar su propio almacén de identidades heredado con Auth0?
* ¿Cómo migra las identidades de los usuarios desde su almacén de identidades a Auth0?
* ¿Pueden sus usuarios registrarse con sus cuentas sociales existentes, como Google y Facebook?

Auth0 proporciona almacenamiento de identidades listo para usar que puede aprovecharse para almacenar de forma segura las credenciales de los usuarios. Consulte [autorregistro](#self-sign-up) para obtener más información. Si ya tiene un almacén de identidades heredado y quiere delegar su gestión, las capacidades de [Migración de usuarios](#user-migration) le ofrecen varias opciones para hacerlo.

Como alternativa, si debe mantener su almacén de identidades heredado —quizá porque tiene aplicaciones que aún no está listo para migrar o que no pueden migrarse—, puede usar la capacidad de [proxy de almacén de identidades](#identity-store-proxy). Permitir que sus clientes usen su propia identidad también es una opción atractiva y, aunque vemos que nuestros clientes no suelen hacerlo al principio, puede usar la capacidad de [registro con redes sociales](#social-sign-up) para ofrecerlo.

<div id="user-migration">
  ## Migración de usuarios
</div>

Además de alojar el [perfil de usuario](/docs/es/get-started/architecture-scenarios/business-to-consumer/profile-management), Auth0 también puede actuar como [proxy](#identity-store-proxy) de tu propio almacén de identidades heredado y proporcionar un reemplazo seguro alojado por Auth0. Ambas funcionalidades se admiten mediante el uso de [Database Connections](/docs/es/authenticate/database-connections) de Auth0. Si decides usar Auth0 como reemplazo de tu almacén de identidades heredado, puedes [migrar usuarios](/docs/es/manage-users/user-migration) de una sola vez mediante migración masiva, o progresivamente con migración automática.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Los clientes suelen optar por un enfoque de migración de usuarios en dos etapas: primero usan la Migración automática para migrar tantos usuarios como sea posible y luego usan la Migración masiva para los usuarios restantes. Consulta [Escenarios de migración de usuarios](/docs/es/manage-users/user-migration/user-migration-scenarios) para obtener más información.
</Callout>

Se prefiere la Migración automática, ya que permite migrar a los usuarios individualmente y también conservar su contraseña actual en casi todos los casos. Para la Migración masiva, recomendamos usar la [Management API](https://auth0.com/docs/api/management/v2#!/Jobs/post_users_imports) en lugar de la [extensión User Import/Export](/docs/es/manage-users/user-migration/user-import-export-extension) salvo en los casos más sencillos, ya que la <Tooltip tip="Management API: un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip> ofrece mayor flexibilidad y control.

Con la Migración masiva, los usuarios normalmente necesitan **restablecer su contraseña una vez completada la migración**, a menos que las contraseñas se almacenen con hash en tu almacén de identidades heredado mediante bcrypt (o que puedas generarlas en formato bcrypt). En ese caso, es posible que puedas usar la migración masiva y **conservar las contraseñas de los usuarios** como parte del proceso, según el algoritmo bcrypt y la cantidad de rondas de salt utilizadas. Consulta [Ejemplos de esquema de importación masiva de base de datos](/docs/es/manage-users/user-migration/bulk-user-import-schema) para obtener más información.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Las llamadas a la Management API están sujetas a la [política de límite de frecuencia de Auth0](/docs/es/troubleshoot/customer-support/operational-policies/rate-limit-policy). Debes tener esto en cuenta y, como ayuda, Auth0 generalmente recomienda usar el [SDK de Auth0](/docs/es/libraries) adecuado para tu entorno de desarrollo en lugar de llamar directamente a nuestras API.
</Callout>

<div id="identity-store-proxy">
  ### Proxy de almacén de identidades
</div>

Los tipos de conexión de base de datos de Auth0 también pueden configurarse para usar como proxy un almacén de identidades existente (heredado). Si necesita mantener las identidades de usuario definidas en su propio almacén heredado —por ejemplo, si tiene una o más aplicaciones esenciales para el negocio que no puede migrar a Auth0, pero que aún necesitan acceso a esas identidades—, puede integrarse fácilmente con Auth0. Consulte [Autenticar usuarios usando su base de datos](/docs/es/authenticate/database-connections/custom-db) para obtener más información.

<div id="self-sign-up">
  ## Autorregistro
</div>

El autorregistro utiliza las [Database Connections](/docs/es/authenticate/database-connections) de Auth0 para almacenar el ID de usuario, la contraseña y la información de identidad asociada al nombre de usuario (opcional) recopilada de los nuevos usuarios durante el proceso de registro. Las políticas de las conexiones de base de datos que regulan aspectos como la [longitud mínima del nombre de usuario](/docs/es/authenticate/database-connections/require-username) o la [fortaleza y complejidad de la contraseña](/docs/es/authenticate/database-connections/password-options) se pueden configurar desde el <Tooltip tip="Auth0 Dashboard: el producto principal de Auth0 para configurar sus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip>.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Auth0 [Universal Login](/docs/es/authenticate/login/auth0-universal-login), así como los widgets de Auth0 como [Lock](https://auth0.com/lock), se integran con Database Connections para ofrecer una experiencia de interfaz de usuario completa para el registro, lista para usar. Estos elementos de UI son totalmente reactivos y, gracias a su amplia configuración y personalización exhaustiva, puede implementar funcionalidades tanto de autorregistro de usuarios como de inicio de sesión.
</Callout>

<div id="social-sign-up">
  ## Registro con redes sociales
</div>

El registro con redes sociales equivale a iniciar sesión mediante [autenticación social](/docs/es/get-started/architecture-scenarios/business-to-consumer/authentication): aquí no hay ninguna distinción *propiamente dicha*, ya que la creación del [perfil](/docs/es/get-started/architecture-scenarios/business-to-consumer/profile-management) del usuario se produce automáticamente al iniciar sesión por primera vez con una cuenta social.
