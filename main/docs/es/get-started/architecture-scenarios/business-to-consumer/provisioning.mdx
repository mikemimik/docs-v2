Definir cómo se registrarán los usuarios es algo importante que conviene abordar desde el principio, y las decisiones que tome aquí influirán en muchas de las que deberá tomar más adelante. Hemos comprobado que existen varios patrones habituales sobre cómo se incorporan los usuarios a su sistema, así como algunos aspectos que conviene tener en cuenta al diseñar el flujo de trabajo.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Aunque Auth0 admite numerosos flujos de trabajo, los flujos de trabajo web que usan [Universal Login](/docs/es/authenticate/login/auth0-universal-login) para el registro se consideran una práctica recomendada tanto del sector como de Auth0, ya que ofrecen una funcionalidad óptima y la máxima seguridad.
</Callout>

Auth0 admite el registro de usuarios mediante varios [proveedores de identidad](/docs/es/authenticate/identity-providers). Durante el registro, Auth0 aprovisiona el perfil de usuario para que contenga la información de la cuenta del usuario. Hay varios aspectos que debe tener en cuenta al evaluar la funcionalidad y el flujo de trabajo:

* ¿Debería usar Auth0 como almacén de identidades?
* ¿Puede usar su propio almacén de identidades (heredado) con Auth0?
* ¿Cómo migra las identidades de los usuarios desde su almacén de identidades a Auth0?
* ¿Pueden sus usuarios registrarse con sus cuentas sociales existentes, como Google y Facebook?

Auth0 proporciona almacenamiento de identidades listo para usar que puede aprovecharse para almacenar las credenciales de los usuarios de forma segura. Consulte [autorregistro](#self-sign-up) para obtener más información. Si ya tiene un almacén de identidades heredado y desea descargar su gestión, las capacidades de [Migración de usuarios](#user-migration) le ofrecen varias opciones para hacerlo.

Como alternativa, si tiene que mantener su almacén de identidades heredado —quizá porque tiene aplicaciones que aún no está listo para migrar o que no pueden migrarse—, puede usar la capacidad de [proxy de almacén de identidades](#identity-store-proxy). Permitir que sus clientes usen su propia identidad también es una propuesta atractiva y, aunque vemos que nuestros clientes no suelen hacerlo al principio, puede usar la capacidad de [registro con redes sociales](#social-sign-up) para ofrecerlo.

<div id="user-migration">
  ## Migración de usuarios
</div>

Además de alojar el [perfil de usuario](/docs/es/get-started/architecture-scenarios/business-to-consumer/profile-management), Auth0 también puede actuar como [proxy](#identity-store-proxy) de tu almacén de identidades heredado y ofrecer un reemplazo seguro alojado por Auth0. Ambas capacidades son compatibles mediante el uso de [Database Connections](/docs/es/authenticate/database-connections) de Auth0. Si decides usar Auth0 como reemplazo de tu almacén de identidades heredado, puedes [migrar usuarios](/docs/es/manage-users/user-migration) de una sola vez mediante una migración masiva o progresivamente con migración automática.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Los clientes suelen optar por un enfoque de migración de usuarios en dos etapas: primero usan la Migración automática para migrar tantos usuarios como sea posible y luego la Migración masiva para los usuarios restantes. Consulta [Escenarios de migración de usuarios](/docs/es/manage-users/user-migration/user-migration-scenarios) para obtener más información.
</Callout>

Se prefiere la Migración automática, ya que permite migrar a los usuarios individualmente y también conservar su contraseña actual en casi todos los casos. Para la Migración masiva, recomendamos usar la [Management API](https://auth0.com/docs/api/management/v2#!/Jobs/post_users_imports) en lugar de la [extensión User Import/Export](/docs/es/manage-users/user-migration/user-import-export-extension) salvo en los casos más sencillos, ya que la <Tooltip tip="Management API: Un producto para permitir a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip> ofrece mayor flexibilidad y control.

Con la Migración masiva, por lo general los usuarios deben **restablecer su contraseña una vez completada la migración**, a menos que las contraseñas estén almacenadas como hash en tu almacén de identidades heredado mediante bcrypt (o puedas generarlas en formato bcrypt). En ese caso, puede que puedas usar la migración masiva y **conservar las contraseñas de los usuarios** como parte del proceso, según el algoritmo bcrypt y la cantidad de rondas de salt utilizadas. Consulta [Ejemplos de esquema de importación masiva de bases de datos](/docs/es/manage-users/user-migration/bulk-user-import-schema) para obtener más información.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Las llamadas a la Management API están sujetas a la [política de límite de frecuencia de Auth0](/docs/es/troubleshoot/customer-support/operational-policies/rate-limit-policy). Debes tenerlo en cuenta y, como ayuda, Auth0 recomienda en general usar el [SDK de Auth0](/docs/es/libraries) adecuado para tu entorno de desarrollo en lugar de llamar directamente a nuestras API.
</Callout>

<div id="identity-store-proxy">
  ### Proxy de almacén de identidades
</div>

Los tipos de conexión de base de datos de Auth0 también se pueden configurar para actuar como proxy de un almacén de identidades existente (heredado). Si necesita conservar las identidades de usuario definidas en su propio almacén heredado —por ejemplo, si tiene una o más aplicaciones esenciales para la empresa que no puede migrar a Auth0, pero que siguen necesitando acceso a esas identidades—, puede integrarse fácilmente con Auth0. Consulte [Autenticar usuarios con su base de datos](/docs/es/authenticate/database-connections/custom-db) para obtener más información.

<div id="self-sign-up">
  ## Autoregistro
</div>

El autoregistro aprovecha Auth0 [Database Connections](/docs/es/authenticate/database-connections) para almacenar el ID de usuario, la contraseña y la información de identidad del nombre de usuario (opcional) que se recopilan de los usuarios nuevos durante el proceso de registro. Las políticas de la conexión de base de datos que regulan aspectos como la [longitud mínima del nombre de usuario](/docs/es/authenticate/database-connections/require-username) o la [fortaleza y complejidad de la contraseña](/docs/es/authenticate/database-connections/password-options) se pueden configurar desde el <Tooltip tip="Auth0 Dashboard: el producto principal de Auth0 para configurar tus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip>.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Auth0 [Universal Login](/docs/es/authenticate/login/auth0-universal-login), así como los widgets de Auth0 como [Lock](https://auth0.com/lock), se integran con Database Connections para ofrecer de forma inmediata una funcionalidad completa de interfaz de usuario para el registro. Estos elementos de la UI son totalmente reactivos y, gracias a sus amplias opciones de configuración y personalización, puedes desplegar funcionalidades tanto para el autoregistro de usuarios como para el inicio de sesión.
</Callout>

<div id="social-sign-up">
  ## Registro con redes sociales
</div>

El registro con redes sociales es sinónimo de iniciar sesión mediante [autenticación social](/docs/es/get-started/architecture-scenarios/business-to-consumer/authentication); aquí no existe una distinción propiamente dicha, ya que el [perfil](/docs/es/get-started/architecture-scenarios/business-to-consumer/profile-management) del usuario se crea automáticamente la primera vez que inicia sesión con una cuenta social.
