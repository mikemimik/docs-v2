<div id="create-the-api">
  ## Crear la API
</div>

Haz clic en la [opción de menú APIs](https://manage.auth0.com/#/apis) de la izquierda y luego en el botón Create API.

Se te pedirá que proporciones los siguientes datos para tu API:

* **Name**: un nombre descriptivo para la API. No afecta a ninguna funcionalidad.
* **Identifier**: un identificador único para la API. Recomendamos usar una URL, pero ten en cuenta que no tiene por qué ser una URL disponible públicamente; Auth0 no llamará a tu API en ningún caso. Este valor no se puede modificar después.
* **<Tooltip tip="Algoritmo de firma: algoritmo utilizado para firmar digitalmente tokens y garantizar que el token no haya sido manipulado." cta="Ver glosario" href="/docs/es/glossary?term=Signing+Algorithm">Algoritmo de firma</Tooltip>**: el algoritmo con el que se firmarán los tokens. Los valores disponibles son `HS256` y `RS256`. Al seleccionar RS256, el token se firmará con la clave privada del tenant. Para obtener más información sobre los algoritmos de firma, consulta [Algoritmos de firma](/docs/es/get-started/applications/signing-algorithms).

Completa la información requerida y haz clic en el botón **Create**.

<div id="signing-algorithms">
  #### Algoritmos de firma
</div>

Cuando creas una API, tienes que seleccionar el algoritmo con el que se firmarán tus tokens. La firma se utiliza para verificar que el emisor del <Tooltip tip="JSON Web Token (JWT): formato estándar de token de ID (y, a menudo, formato de token de acceso) utilizado para representar claims de forma segura entre dos partes." cta="Ver glosario" href="/docs/es/glossary?term=JWT">JWT</Tooltip> es quien dice ser y para garantizar que el mensaje no se haya modificado durante el proceso.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  La firma forma parte de un JWT. Si no estás familiarizado con la estructura de un JWT, consulta [JSON Web Token Structure](/docs/es/secure/tokens/json-web-tokens/json-web-token-structure).
</Callout>

Para crear la parte de la firma, tienes que tomar el header codificado, el payload codificado, un secreto, el algoritmo especificado en el header y firmarlo. Ese algoritmo, que forma parte del header del JWT, es el que seleccionas para tu API: `HS256` o `RS256`.

* **RS256** es un [algoritmo asimétrico](https://en.wikipedia.org/wiki/Public-key_cryptography), lo que significa que hay dos claves: una pública y una privada (secreta). Auth0 tiene la clave secreta, que se utiliza para generar la firma, y el consumidor del JWT tiene la clave pública, que se utiliza para validar la firma.
* **HS256** es un [algoritmo simétrico](https://en.wikipedia.org/wiki/Symmetric-key_algorithm), lo que significa que solo hay una clave secreta, compartida entre las dos partes. La misma clave se utiliza tanto para generar la firma como para validarla. Se debe tener especial cuidado para que la clave siga siendo confidencial.

La práctica más segura, y nuestra recomendación, es usar **RS256**. Algunas de las razones son:

* Con RS256 tienes la seguridad de que solo el poseedor de la clave privada (Auth0) puede firmar tokens, mientras que cualquiera puede comprobar si el token es válido usando la clave pública.
* En HS256, si la clave secreta se ve comprometida, tendrías que volver a desplegar la API con el nuevo secreto. Con RS256 puedes solicitar un token que sea válido para múltiples audiencias.
* Con RS256 puedes implementar la rotación de claves sin tener que volver a desplegar la API con el nuevo secreto.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Para obtener una explicación más detallada de los algoritmos de firma de JWT, consulta: [JSON Web Token (JWT) Signing Algorithms Overview](https://auth0.com/blog/json-web-token-signing-algorithms-overview/).
</Callout>

<div id="create-the-application">
  ## Crear la aplicación
</div>

Hay cuatro tipos de aplicación en Auth0: **Native App** (utilizada por aplicaciones móviles o de escritorio), **aplicación web de página única**, **aplicación web tradicional** y **Machine to Machine App** (utilizada por CLI, Daemons o servicios que se ejecutan en tu backend). Para este caso, queremos crear una nueva aplicación para nuestra aplicación móvil, por lo que usaremos Native como tipo de aplicación.

Para crear una nueva aplicación, ve al [Dashboard](https://manage.auth0.com/#) y haz clic en la opción de menú [Applications](https://manage.auth0.com/#/applications) de la izquierda. Haz clic en el botón **+ Create Application**.

Asigna un nombre a tu aplicación (usaremos `Timesheets Mobile`) y selecciona `Native App` como tipo.

Haz clic en **Create**.

<div id="create-a-rule-to-validate-token-scopes">
  ### Crear una Rule para validar los scopes del token
</div>

El último paso de este proceso es crear una Rule que valide que los scopes incluidos en un <Tooltip tip="Access Token: credencial de autorización, en forma de una cadena opaca o JWT, que se utiliza para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=Access+Token">Access Token</Tooltip> sean válidos según los permisos asignados al usuario. Cualquier scope que no sea válido para un usuario debe eliminarse del Access Token.

En tu <Tooltip tip="Auth0 Dashboard: el producto principal de Auth0 para configurar tus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip>, ve a la pestaña Rules. Allí deberías ver la Rule creada por la Authorization Extension.

Haz clic en el botón **Create Rule** y selecciona la plantilla **Empty Rule**. Puedes darle un nombre a la Rule, por ejemplo **Access Token Scopes**, y luego especificar el siguiente código para la Rule:

```javascript lines
function (user, context, callback) {
  if (context.clientName !== 'Timesheets SPA') {
    return callback(null, user, context);
  }

  var permissions = user.permissions || [];
  var requestedScopes = context.request.body.scope || context.request.query.scope;
  var filteredScopes = requestedScopes.split(' ').filter( function(x) {
    return x.indexOf(':') < 0;
  });
  Array.prototype.push.apply(filteredScopes, permissions);
  context.accessToken.scope = filteredScopes.join(' ');

  callback(null, user, context);
}
```

El código anterior garantizará que todos los Access Tokens solo contengan los scopes con el formato correcto (por ejemplo, `action:area` o `delete:timesheets`) que sean válidos según los permisos de un usuario. Cuando hayas terminado, puedes hacer clic en el botón **Save**.

Las Rules se ejecutan en el orden en que se muestran en la página Rules, así que asegúrate de que la nueva rule que creaste esté ubicada debajo de la rule de Authorization Extension para que se ejecute después de la Authorization Extension rule.
