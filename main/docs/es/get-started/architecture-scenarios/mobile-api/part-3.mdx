<div id="define-the-api-endpoints">
  ## Define los endpoints de la API
</div>

Primero, necesitamos definir los endpoints de nuestra API.

<Card title="¿Qué es un endpoint de API?">
  Un **API endpoint** es una URL única que representa un objeto. Para interactuar con este objeto, debes dirigir tu aplicación a esa URL. Por ejemplo, si tuvieras una API que pudiera devolver pedidos o clientes, podrías configurar dos endpoints: `/orders` y `/customers`. Tu aplicación interactuaría con estos endpoints mediante distintos métodos HTTP; por ejemplo, `POST /orders` podría crear un nuevo pedido o `GET /orders` podría recuperar el conjunto de datos de uno o más pedidos.
</Card>

Para esta implementación, solo definiremos dos endpoints: uno para recuperar una lista de todas las hojas de hora de un Employee y otro que permitirá a un Employee crear una nueva hoja de hora.

Una solicitud `HTTP GET` al endpoint `/timesheets` permitirá que un usuario recupere sus hojas de hora, y una solicitud `HTTP POST` al endpoint `/timesheets` permitirá que un usuario agregue una nueva hoja de hora.

Consulta la implementación en [Node.js](/docs/es/get-started/architecture-scenarios/mobile-api/api-implementation-nodejs#1-define-the-api-endpoints)

<div id="secure-the-endpoints">
  ### Protege los endpoints
</div>

Cuando una API recibe una solicitud con un bearer <Tooltip tip="token de acceso: Credencial de autorización, en forma de una cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=Access+Token">token de acceso</Tooltip> como parte del header, lo primero que debe hacerse es validar el token. Esto consiste en una serie de pasos y, si alguno de ellos falla, la solicitud debe rechazarse con un mensaje de error `Missing or invalid token` para la aplicación que realiza la llamada.

Las validaciones que la API debe realizar son:

* Comprobar que el <Tooltip tip="JSON Web Token (JWT): Formato estándar de token de ID (y, a menudo, formato de token de acceso) utilizado para representar claims de forma segura entre dos partes." cta="Ver glosario" href="/docs/es/glossary?term=JWT">JWT</Tooltip> tenga un formato válido
* Verificar la firma
* Validar los claims estándar

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  [JWT.io](https://jwt.io/) proporciona una lista de bibliotecas que pueden hacer la mayor parte del trabajo por ti: analizar el JWT, verificar la firma y los claims.
</Callout>

Parte del proceso de validación también incluye comprobar los permisos del cliente (scopes), pero abordaremos esto por separado en el siguiente párrafo de este documento.

Para obtener más información sobre cómo validar tokens de acceso, consulta [Validate tokens de acceso](/docs/es/secure/tokens/access-tokens/validate-access-tokens).

Consulta la implementación en [Node.js](/docs/es/get-started/architecture-scenarios/mobile-api/api-implementation-nodejs#2-secure-the-api-endpoints)

<div id="check-the-clients-permissions">
  ### Comprueba los permisos del cliente
</div>

A estas alturas, ya hemos verificado que el JWT es válido. El último paso es comprobar que el cliente tenga los permisos necesarios para acceder a los recursos protegidos.

Para ello, la API debe comprobar los [scopes](/docs/es/get-started/apis/scopes) del JWT decodificado. Este claim forma parte del payload y es una lista de strings separada por espacios.

Consulta la implementación en [Node.js](/docs/es/get-started/architecture-scenarios/mobile-api/api-implementation-nodejs#3-check-the-client-permissions)

<div id="determine-user-identity">
  ### Determinar la identidad del usuario
</div>

Para ambos endpoints (obtener la lista de hojas de hora y agregar una nueva hoja de hora) necesitaremos determinar la identidad del usuario.

Al obtener la lista de hojas de hora, esto garantiza que solo devolvamos las hojas de hora que pertenecen al usuario que realiza la solicitud; y al agregar una nueva hoja de hora, garantiza que la hoja de hora quede asociada al usuario que realiza la solicitud.

Una de las claims estándar de JWT es la claim `sub`, que identifica al principal que es el subject de la claim. En el caso del flujo de concesión implícita, esta claim contendrá la identidad del usuario, que será el identificador único del usuario de Auth0. Puede usar esto para asociar cualquier información de sistemas externos a un usuario en particular.

También puede usar una custom claim para agregar otro atributo del usuario, como su dirección de correo electrónico, al token de acceso y usarlo para identificar de forma única al usuario.

Vea la implementación en [Node.js](/docs/es/get-started/architecture-scenarios/mobile-api/api-implementation-nodejs#4-determine-the-user-identity)

<div id="authorize-the-user">
  ### Autorizar al usuario
</div>

Para autorizar al usuario, implementaremos el [flujo de código de autorización con Proof Key for Code Exchange (PKCE)](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce/call-your-api-using-the-authorization-code-flow-with-pkce). La aplicación móvil primero debe redirigir al usuario a la [URL de autorización](https://auth0.com/docs/api/authentication#authorization-code-grant-pkce-) junto con el `code_challenge` y el método utilizado para generarlo:

export const codeExample1 = `https://{yourDomain}/authorize?
    audience=API_AUDIENCE&
    scope=SCOPE&
    response_type=code&
    client_id={yourClientId}&
    code_challenge=CODE_CHALLENGE&
    code_challenge_method=S256&
    redirect_uri=https://YOUR_APP/callback`;

<AuthCodeBlock children={codeExample1} language="http" />

La solicitud `GET` a la URL de autorización debe incluir los siguientes valores:

| Parameter                         | Description                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| --------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **client&#95;id**                 | El valor de tu ID de cliente de Auth0. Puedes obtenerlo en Settings de tu aplicación en el [Auth0 Dashboard](https://manage.auth0.com/#/applications).                                                                                                                                                                                                                                                                                                                                                                                                                           |
| **audience**                      | El valor del identificador de tu API. Puedes obtenerlo en Settings de tu API en el [Auth0 Dashboard](https://manage.auth0.com/#/apis).                                                                                                                                                                                                                                                                                                                                                                                                                                           |
| **scope**                         | Los [scopes](/docs/es/get-started/apis/scopes) que determinan los claims que se devolverán en el token de ID y el token de acceso. Por ejemplo, un scope de `openid` devolverá un token de ID en la respuesta. En nuestra aplicación móvil de ejemplo, usamos los siguientes scopes: `create:timesheets read:timesheets openid profile email offline_access`. Estos scopes permiten que la aplicación móvil llame a la API, obtenga un [token de actualización](/docs/es/glossary?term=Refresh+Token) y devuelva los claims `name`, `picture` y `email` del usuario en el token de ID. |
| **response&#95;type**             | Indica el flujo de autenticación que se debe usar. En una aplicación móvil que usa PKCE, debe establecerse en `code`.                                                                                                                                                                                                                                                                                                                                                                                                                                                            |
| **code&#95;challenge**            | El code challenge generado a partir del code verifier. Puedes encontrar instrucciones para generar un code challenge [aquí](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce/call-your-api-using-the-authorization-code-flow-with-pkce#authorize-the-user%23create-a-code-verifier).                                                                                                                                                                                                                                                    |
| **code&#95;challenge&#95;method** | Método utilizado para generar el challenge. Auth0 solo admite `S256`.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                            |
| **redirect&#95;uri**              | La URL a la que Auth0 redirigirá el navegador después de que el usuario haya autorizado el acceso. El código de autorización estará disponible en el parámetro de URL `code`. Esta URL debe especificarse como una URL de devolución de llamada válida en Settings de tu [aplicación](https://manage.auth0.com/#/applications).                                                                                                                                                                                                                                                  |

<div id="get-the-credentials">
  ### Obtén las credenciales
</div>

Tras realizar correctamente una solicitud a la URL de autorización, deberías recibir la siguiente respuesta:

export const codeExample2 = `HTTP/1.1 302 Found
Location: https://{yourDomain}/callback?code=AUTHORIZATION_CODE`;

A continuación, puedes intercambiar el `authorization_code` de la respuesta por un token de acceso que podrás usar para llamar a tu API. Haz una solicitud `POST` a la [URL del token](https://auth0.com/docs/api/authentication#authorization-code-pkce-) e incluye los siguientes datos:

  ```bash cURL
  curl --request POST \
    --url 'https://{yourDomain}/oauth/token' \
    --header 'content-type: application/x-www-form-urlencoded' \
    --data grant_type=authorization_code \
    --data 'client_id={yourClientId}' \
    --data code_verified=YOUR_GENERATED_CODE_VERIFIER \
    --data code=YOUR_AUTHORIZATION_CODE \
    --data 'redirect_uri=https://{https://yourApp/callback}'
  ```

  ```csharp C#

  request.AddHeader("content-type", "application/x-www-form-urlencoded");
  request.AddParameter("application/x-www-form-urlencoded", "grant_type=authorization_code&client_id={yourClientId}&code_verified=YOUR_GENERATED_CODE_VERIFIER&code=YOUR_AUTHORIZATION_CODE&redirect_uri=https%3A%2F%2F{https://yourApp/callback}", ParameterType.RequestBody);
  IRestResponse response = client.Execute(request);
  ```

  ```go Go
  }
  ```

| Parámetro             | Descripción                                                                                                                                                                                                            |
| --------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **grant&#95;type**    | Debe establecerse como `authorization_code`.                                                                                                                                                                           |
| **client&#95;id**     | El valor de tu ID de cliente de Auth0. Puedes obtenerlo en la sección Settings de tu aplicación en el [Auth0 Dashboard](https://manage.auth0.com/#/applications).                                                      |
| **code&#95;verifier** | Clave aleatoria criptográficamente segura que se utilizó para generar el `code_challenge` enviado a la [URL de autorización](https://auth0.com/docs/api/authentication#authorization-code-grant-pkce-) (`/authorize`). |
| **code**              | El `authorization_code` recibido de la llamada anterior a `/authorize`.                                                                                                                                                |
| **redirect&#95;uri**  | La URL debe coincidir con el `redirect_uri` enviado en la sección anterior a `/authorize`.                                                                                                                             |

La respuesta de la URL del token contendrá:

```json lines
{
  "access_token": "eyJz93a...k4laUWw",
  "refresh_token": "GEbRxBN...edjnXbL",
  "id_token": "eyJ0XAi...4faeEoQ",
  "token_type": "Bearer",
  "expires_in":86400
}
```

* **access&#95;token**: Un token de acceso para la API, especificado por la `audience`.
* **refresh&#95;token**: Un [token de actualización](/docs/es/secure/tokens/refresh-tokens) solo estará presente si incluiste el scope `offline_access` y habilitaste **Allow Offline Access** para tu API en el Dashboard.
* **id&#95;token**: Un <Tooltip tip="token de ID: credencial destinada al propio cliente, en lugar de usarse para acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=ID+Token">token de ID</Tooltip> JWT que contiene información del perfil del usuario.
* **token&#95;type**: Una cadena que contiene el tipo de token; este siempre será un token Bearer.
* **expires&#95;in**: La cantidad de segundos que faltan para que expire el token de acceso.

Deberás almacenar las credenciales anteriores en el almacenamiento local para usarlas al llamar a tu API y recuperar el perfil del usuario.

<div id="get-the-user-profile">
  ### Obtener el perfil del usuario
</div>

Para obtener el [perfil del usuario](https://auth0.com/docs/api/authentication?http#user-profile), tu aplicación móvil puede decodificar el [token de ID](/docs/es/secure/tokens/id-tokens) con una de las [bibliotecas JWT](https://jwt.io/#libraries-io). Esto se hace [verificando la firma](/docs/es/secure/tokens/id-tokens/validate-id-tokens#verify-the-signature) y [verificando los claims](/docs/es/secure/tokens/id-tokens/validate-id-tokens#verify-the-claims) del token. Después de validar el token de ID, puedes acceder a su payload, que contiene la información del usuario:

```json lines
{
  "email_verified": false,
  "email": "test.account@userinfo.com",
  "clientID": "q2hnj2iu...",
  "updated_at": "2016-12-05T15:15:40.545Z",
  "name": "test.account@userinfo.com",
  "picture": "https://s.gravatar.com/avatar/dummy.png",
  "user_id": "auth0|58454...",
  "nickname": "test.account",
  "created_at": "2016-12-05T11:16:59.640Z",
  "sub": "auth0|58454..."
}
```

Según el `scope` del usuario, es posible que quieras mostrar u ocultar determinados elementos de la interfaz de usuario. Será una cadena que contiene todos los scopes, así que tendrás que revisarla para comprobar si incluye el `scope` necesario y, en función de ello, decidir si mostrar un elemento concreto de la interfaz de usuario.

<div id="renew-the-token">
  ### Renovar el token
</div>

<Warning>
  Los tokens de actualización deben almacenarse de forma segura en una aplicación, ya que no caducan y permiten que un usuario permanezca autenticado prácticamente de forma indefinida. Si los tokens de actualización se ven comprometidos o ya no los necesita, puede revocarlos mediante la [Authentication API](https://auth0.com/docs/api/authentication#revoke-refresh-token).
</Warning>

Para actualizar su token de acceso, realice una solicitud `POST` al `endpoint` `/oauth/token` usando el <Tooltip tip="Token de actualización: token que se usa para obtener un token de acceso renovado sin obligar a los usuarios a volver a iniciar sesión." cta="Ver glosario" href="/docs/es/glossary?term=Refresh+Token">token de actualización</Tooltip> que obtuvo en el resultado de autorización.

Un [token de actualización](/docs/es/secure/tokens/refresh-tokens) solo estará presente si incluyó el `scope` `offline_access` en la solicitud de autorización anterior y habilitó **Allow Offline Access** para su API en el Dashboard.

Su solicitud debe incluir:

| **refresh&#95;token** | El token de actualización que se usará, obtenido del resultado de autenticación anterior.                                                             |

La respuesta incluirá el nuevo token de acceso:

```json lines
{
  "access_token": "eyJz93a...k4laUWw",
  "refresh_token": "GEbRxBN...edjnXbL",
  "id_token": "eyJ0XAi...4faeEoQ",
  "token_type": "Bearer",
  "expires_in":86400
}
```
