  <Accordion title="1 Define los endpoints de la API">
    Esta implementación usa el [framework web Express](http://expressjs.com/) para crear una API de Node.js.

    ##### Crea un archivo package.json

    Crea una carpeta para tu API, entra en ella y ejecuta `npm init`. Esto configura tu archivo `package.json`.

    Deja la configuración predeterminada o cámbiala como prefieras.

    El archivo `package.json` de nuestro ejemplo es el siguiente:

    ```json lines
    {
      "name": "timesheets-api",
      "version": "1.0.0",
      "description": "API used to add timesheet entries for employees and contractors",
      "main": "index.js",
      "scripts": {
        "start": "node index.js",
        "test": "echo \"Error: no test specified\" && exit 1"
      },
      "dependencies": {
        "body-parser": "^1.20.0",
        "cors": "^2.8.5",
        "express": "^4.18.0",
        "express-oauth2-jwt-bearer": "^1.6.0"
      },
      "author": "Auth0",
      "license": "MIT"
    }
    ```

    ##### Instala las dependencias

    A continuación, configura las dependencias con los siguientes módulos:

    * **express**: Este módulo añade el [framework de aplicaciones web Express](https://expressjs.com/).
    * **cors**: Este módulo añade soporte para habilitar [CORS](https://en.wikipedia.org/wiki/Cross-origin_resource_sharing), lo cual es necesario porque se llama a la API desde una aplicación de página única que se ejecuta en un dominio diferente dentro de un navegador web.
    * **jwks-rsa**: Esta biblioteca recupera claves de firma RSA desde un endpoint JWKS (JSON Web Key Set). Al usar `expressJwtSecret`, podemos generar un proveedor de secretos que proporcione la clave de firma correcta a `express-jwt` según el `kid` del encabezado del JWT. Para obtener más información, consulta el [repositorio de GitHub de node-jwks-rsa](https://github.com/auth0/node-jwks-rsa).
    * **express-jwt**: Este módulo autentica solicitudes HTTP mediante tokens JWT en tus aplicaciones de Node.js. Proporciona varias funciones que facilitan el trabajo con JWT. Para obtener más información, consulta el [repositorio de GitHub de express-jwt](https://github.com/auth0/express-jwt).
    * **body-parser**: Este es un middleware de Node.js para analizar el cuerpo de las solicitudes. Extrae todo el contenido del cuerpo de un flujo de solicitud entrante y lo expone en `req.body` de una forma más fácil de usar.

    Para instalar estas dependencias, ejecuta lo siguiente:

    ```bash wrap lines
    npm install express cors express-jwt jwks-rsa body-parser express-jwt-authz --save
    ```

    ##### Implementa los endpoints

    Ve al directorio de tu API y crea un archivo `server.js`. Tu código debe:

    * Instalar las dependencias.
    * Implementar los endpoints.
    * Iniciar el servidor de la API.

    Esta es nuestra implementación de ejemplo:

    ```javascript lines
    const express = require('express');
    const app = express();
    const { expressjwt: jwt } = require('express-jwt');
    const jwksRsa = require('jwks-rsa');
    const cors = require('cors');
    const bodyParser = require('body-parser');

    // Habilitar CORS
    app.use(cors());

    // Habilitar el uso de middleware de análisis del cuerpo de la solicitud
    app.use(bodyParser.json());
    app.use(bodyParser.urlencoded({
      extended: true
    }));

    // Crear el endpoint de la API de parte de horas
    app.post('/timesheets', function(req, res){
      res.status(201).send({message: "This is the POST /timesheets endpoint"});
    })

    // Iniciar el servidor de la API en localhost:8080
    app.listen(8080);
    ```

    Inicia tu servidor de la API con `node server` y realiza una solicitud HTTP POST a `localhost:8080/timesheets`. Deberías ver una respuesta JSON con el mensaje `This is the POST /timesheets endpoint`.

    Ahora ya tenemos nuestro endpoint, pero cualquiera puede llamarlo. Continúa con el siguiente paso para ver cómo podemos solucionar esto.

  <Accordion title="2 Protege los endpoints de la API">
    Para validar nuestro token, usa la función `jwt`, proporcionada por el [middleware express-jwt](https://github.com/auth0/express-jwt#usage), y `jwks-rsa` para recuperar nuestro secreto. Las bibliotecas hacen lo siguiente:

    1. `express-jwt` decodifica el token y pasa la solicitud, el encabezado y el payload a `jwksRsa.expressJwtSecret`.
    2. `jwks-rsa` descarga todas las claves de firma del endpoint de JWKS y comprueba si alguna de ellas coincide con el `kid` del encabezado del JWT. Si ninguna de las claves de firma coincide con el `kid` entrante, se generará un error. Si hay una coincidencia, pasa la clave de firma correcta a `express-jwt`.
    3. `express-jwt` continúa con su propia lógica para validar la firma del token, la expiración, `audience` y el `issuer`.

    Los pasos que seguiremos en nuestro código son:

    * Crear la función de middleware para validar el token de acceso.
    * Habilitar el uso del middleware en nuestras rutas.

  <Accordion title="3 Comprueba los permisos de la aplicación">
    En este paso, añadimos la capacidad de comprobar si la aplicación tiene permisos (o scopes) y de usar nuestro endpoint para crear una hoja de hora. En concreto, queremos asegurarnos de que el token tenga el scope correcto, que es `batch:upload`.

    Para ello, usamos el paquete de Node.js `express-jwt-authz`, así que agrégalo a tu proyecto:

    ```bash lines
    npm install express-jwt-authz --save
    ```

    Ahora agrega una llamada a `jwtAuthz(...)` a tu middleware para asegurarte de que el JWT incluya un scope específico para ejecutar un endpoint concreto.

    Agregamos una dependencia adicional: la biblioteca **express-jwt-authz**, que se usa junto con express-jwt, valida el [JWT](/docs/es/secure/tokens/json-web-tokens) y garantiza que tenga los permisos correctos para llamar al endpoint deseado. Para más información, consulta el [repositorio de GitHub de express-jwt-authz](https://github.com/auth0/express-jwt-authz).

    Este es nuestro ejemplo de implementación (se omite parte del código por brevedad):

    ```javascript lines
    // establecer dependencias - parte del código omitido
    const jwtAuthz = require('express-jwt-authz');

    // Habilitar CORS - código omitido

    // Crear middleware para verificar el JWT - código omitido

    // Habilitar el uso de middleware de análisis del cuerpo de la solicitud - código omitido

    // crear el endpoint de la API de parte de horas
    app.post('/timesheets', checkJwt, jwtAuthz(['create:timesheets'], { customUserKey: 'auth' }), function(req, res){
      var timesheet = req.body;

      // Guardar la hoja de hora en la base de datos...

      //enviar la respuesta
      res.status(201).send(timesheet);
    })

    // iniciar el servidor de la API en localhost:8080 - código omitido
    ```

    Si invocamos nuestra API con un token que no incluye este scope, deberíamos obtener el mensaje de error Forbidden con el código de estado HTTP `403`. Puedes comprobarlo eliminando este scope de tu API.

  <Accordion title="4 Determinar la identidad del usuario">
    El middleware `express-jwt` que se usa para validar el JWT también establece `req.user` con la información contenida en el JWT. Si quieres usar el claim `sub` para identificar al usuario de forma única, puedes usar `req.user.sub`. En la aplicación de hojas de hora, queremos usar la dirección de correo electrónico del usuario como identificador único.

    ##### Crear una Action

    Primero, [crea una nueva Action](/docs/es/customize/actions/write-your-first-action) que agregará la dirección de correo electrónico del usuario al token de acceso.

    1. Ve a [Auth0 Dashboard &gt; Actions &gt; Library](https://manage.auth0.com/#/actions/library) y selecciona **Build Custom**.
    2. Introduce un **Name** descriptivo para tu Action (por ejemplo, `Add email to access token`), selecciona el trigger **Login / Post Login** y luego **Create**.
    3. Busca el Actions Code Editor, copia en él el siguiente código JavaScript y selecciona **Save Draft** para guardar los cambios:

       ```text lines
       exports.onExecutePostLogin = async (event, api) => {
         const namespace = 'https://my-app.example.com';
         api.accessToken.setCustomClaim(`${namespace}/email`, event.user.email);
       }
       ```

       <Callout icon="file-lines" color="#0EA5E9" iconType="regular">
         El `namespace` se usa para garantizar que el claim tenga un nombre único que no entre en conflicto con los claims estándar de OIDC ni con los servicios internos. Para obtener más información sobre las restricciones y directrices de los claims con espacio de nombres y sin espacio de nombres, consulta [Create Custom Claims](/docs/es/secure/tokens/json-web-tokens/create-custom-claims).
       </Callout>
    4. En la barra lateral del Actions Code Editor, selecciona Test (icono de reproducción) y luego **Run** para [probar tu código](/docs/es/customize/actions/test-actions).
    5. Cuando la Action esté lista para entrar en producción, selecciona **Deploy**.

    ##### Agrega tu Action al trigger Post-Login

    A continuación, agrega la Action que creaste al [trigger Post-Login](https://manage.auth0.com/#/actions/triggers/post-login/). Para aprender a adjuntar Actions a triggers, consulta [Write Your First Action](/docs/es/customize/actions/write-your-first-action).

    ##### Recupera el identificador único

    Por último, dentro de tu API, recupera el valor del claim desde `req.auth`. Usa ese valor como identificador único del usuario para asociarlo con las entradas de hoja de hora.

    ```js lines
    app.get('/timesheets', checkJwt, jwtAuthz(['read:timesheets'], { customUserKey: 'auth' }), function(req, res) {
      var timesheet = req.body;

      // Asociar la hoja de horas con el usuario actual
      var userId = req.auth['https://api.exampleco.com/email'];
      timesheet.user_id = userId;

      // Guardar la hoja de horas en la base de datos...

      //enviar la respuesta
      res.status(201).send(timesheet);
    });
    ```
