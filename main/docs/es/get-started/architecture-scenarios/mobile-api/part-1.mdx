Para garantizar que solo los usuarios y las aplicaciones autorizados tengan acceso a la API de parte de horas, ExampleCo ha decidido utilizar el [marco de autorización OAuth 2.0](https://tools.ietf.org/html/rfc6749). Este marco ofrece la flexibilidad que la empresa busca, ya que las distintas concesiones le permiten autorizar fácilmente los diversos tipos de aplicaciones que necesitan comunicarse con la API de parte de horas.

<div id="api-authentication-and-authorization">
  ## Autenticación y autorización de API
</div>

Una API es una forma de exponer la funcionalidad de tu aplicación a otras aplicaciones. Una aplicación puede hacer una solicitud enviando un mensaje a un endpoint de una API y recibir información como respuesta.

Un endpoint de API puede estar protegido o no. En nuestro caso, como las hojas de hora son información sensible que afecta a las revisiones y los pagos, es importante garantizar que solo los usuarios y las aplicaciones autorizados puedan llamar a los endpoints de nuestra API. Cuando una aplicación cliente quiere acceder a endpoints protegidos de una API, debe presentar un <Tooltip tip="token de acceso: Credencial de autorización, en forma de una cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=Access+Token">token de acceso</Tooltip> como prueba de que tiene los permisos necesarios para hacer la llamada al endpoint.

Un token de acceso se obtiene autenticando al usuario con un <Tooltip tip="Authorization Server: Servidor centralizado que contribuye a definir los límites del acceso de un usuario. Por ejemplo, tu servidor de autorización puede controlar los datos, las tareas y las funcionalidades disponibles para un usuario." cta="Ver glosario" href="/docs/es/glossary?term=Authorization+Server">servidor de autorización</Tooltip>, y luego el usuario puede, a su vez, autorizar a la aplicación para acceder a la API en su nombre.

<Card title="¿Qué es un token de acceso?">
  Un token de acceso (también denominado `access_token`) es una cadena opaca que representa una autorización emitida para la aplicación. Puede ser un identificador que se usa para recuperar la información de autorización o puede contener por sí mismo la información de autorización (por ejemplo, la identidad del usuario, los permisos, etc.) de forma verificable.

  Es bastante común que los tokens de acceso se implementen como [JSON Web Tokens](/docs/es/secure/tokens/json-web-tokens).

  Para obtener más información sobre los tokens de acceso de Auth0, consulta [token de acceso](/docs/es/secure/tokens/access-tokens).
</Card>

Una API puede aplicar un control detallado sobre quién puede acceder a los distintos endpoints expuestos por la API. Estos permisos se expresan como scopes.

Cuando un usuario autoriza una aplicación cliente, la aplicación también puede indicar qué permisos necesita. El usuario puede entonces revisar y conceder esos permisos. Después, estos permisos se incluyen en el token de acceso como parte del claim `scope`.

Posteriormente, cuando el cliente envía el token de acceso al hacer solicitudes a la API, la API puede inspeccionar el claim `scope` para asegurarse de que se hayan concedido los permisos necesarios para llamar a ese endpoint de API en particular.

<Card title="¿Qué son los Scopes?">
  Cada token de acceso puede incluir una lista de los permisos que se le han concedido al cliente. Cuando un cliente se autentica con Auth0, especifica la lista de scopes (o permisos) que está solicitando. Si esos scopes se autorizan, el token de acceso contendrá una lista de scopes autorizados.

  Por ejemplo, la API de parte de horas puede aceptar cuatro niveles distintos de autorización: leer hojas de hora (scope `read:timesheets`), crear hojas de hora (scope `create:timesheets`), eliminar hojas de hora (scope `delete:timesheets`) y aprobar hojas de hora (scope `approve:timesheets`).

  Cuando un cliente le pide a la API que cree una nueva hoja de horas, el token de acceso debe contener el scope `create:timesheets`. De forma similar, para eliminar hojas de hora existentes, el token de acceso debe contener el scope `delete:timesheets`.

  Para obtener más información sobre los scopes, consulta [Scopes](/docs/es/get-started/apis/scopes).
</Card>

Al utilizar el marco de autorización <Tooltip tip="OAuth 2.0: Marco de autorización que define protocolos y flujos de autorización." cta="Ver glosario" href="/docs/es/glossary?term=OAuth+2.0">OAuth 2.0</Tooltip>, puedes dar a tus propias aplicaciones o a aplicaciones de terceros acceso limitado a tus API en nombre de la propia aplicación. Con Auth0, puedes admitir fácilmente diferentes flujos en tus propias API sin preocuparte por la especificación OAuth 2.0/<Tooltip tip="OpenID: Estándar abierto de autenticación que permite que las aplicaciones verifiquen las identidades de los usuarios sin recopilar ni almacenar información de inicio de sesión." cta="Ver glosario" href="/docs/es/glossary?term=OpenID">OpenID</Tooltip> Connect (OIDC) ni por los muchos otros aspectos técnicos de la autorización de API.

<Card title="Roles de OAuth">
  En cualquier flujo de OAuth 2.0 podemos identificar los siguientes roles:

  * **Propietario del recurso**: la entidad que puede conceder acceso a un recurso protegido. Normalmente, es el usuario final.
  * **Servidor de recursos**: el servidor que aloja los recursos protegidos. Es la API a la que quieres acceder.
  * **Cliente**: una aplicación que solicita acceso a un recurso protegido en nombre del Propietario del recurso.
  * **Servidor de autorización**: el servidor que autentica al Propietario del recurso y emite tokens de acceso tras obtener la autorización adecuada. En este caso, la Authentication API de Auth0.

  Los [tipos de concesión (o flujos)](/docs/es/get-started/authentication-and-authorization-flow/which-oauth-2-0-flow-should-i-use) determinan cómo interactúan estos participantes para conceder a las aplicaciones acceso limitado a las API que estás creando. Como resultado, la aplicación obtendrá un token de acceso que podrá usar para llamar a la API en nombre del usuario.
</Card>

<div id="proof-key-for-code-exchange-pkce">
  ## Proof Key for Code Exchange (PKCE)
</div>

OAuth 2 ofrece varios tipos de concesión para diferentes casos de uso. En este caso de uso en particular, queremos acceder a la API desde una aplicación móvil, que usará el [flujo de código de autorización con Proof Key for Code Exchange (PKCE)](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce) para hacerlo.

El [flujo de código de autorización](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow) presenta algunos problemas de seguridad cuando se implementa en aplicaciones nativas. Por ejemplo, un atacante malicioso puede interceptar el `authorization_code` devuelto por Auth0 e intercambiarlo por un [token de acceso](/docs/es/secure/tokens/access-tokens) (y posiblemente un [token de actualización](/docs/es/secure/tokens/refresh-tokens)).

Proof Key for Code Exchange (PKCE) (definido en [RFC 7636](https://tools.ietf.org/html/rfc7636)) es una técnica que se utiliza para mitigar este ataque de interceptación del código de autorización.

Con PKCE, la aplicación crea, para cada solicitud de autorización, una clave aleatoria criptográficamente segura llamada `code_verifier` y su valor transformado, llamado `code_challenge`, que se envía a Auth0 para obtener el `authorization_code`. Cuando la aplicación recibe el `authorization_code`, envía el código y el `code_verifier` al <Tooltip tip="Endpoint de token: endpoint en el servidor de autorización que se usa para solicitar tokens de manera programática." cta="Ver glosario" href="/docs/es/glossary?term=token+endpoint">endpoint de token</Tooltip> de Auth0 para intercambiarlos por los tokens solicitados.

<Frame>![Diagrama - Microsite - Código de autorización con PKCE](/docs/images/cdy7uua7fh8z/7pd2I8RXINIihsTyuGxvMb/cdbad014a576df5a15d53f84a5d97a55/authorization-code-grant-pkce.png)</Frame>

1. La app nativa inicia el flujo y redirige al usuario a Auth0 (específicamente al [endpoint `/authorize`](https://auth0.com/docs/api/authentication#authorization-code-grant-pkce-)), enviando los parámetros `code_challenge` y `code_challenge_method`.
2. Auth0 redirige al usuario a la app nativa con un `authorization_code` en la cadena de consulta.
3. La app nativa envía el `authorization_code` y el `code_verifier`, junto con el `redirect_uri` y el `client_id`, a Auth0. Esto se hace mediante el [endpoint `/oauth/token`](https://auth0.com/docs/api/authentication?http#authorization-code-pkce-).
4. Auth0 valida esta información y devuelve un token de acceso (y, opcionalmente, un token de actualización).
5. La app nativa puede usar el token de acceso para llamar a la API en nombre del usuario.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Si tienes habilitada la [rotación del Refresh Token](/docs/es/secure/tokens/refresh-tokens/refresh-token-rotation), se genera un nuevo token de actualización con cada solicitud y se emite junto con el token de acceso. Cuando se intercambia un token de actualización, el token de actualización anterior se invalida, pero el servidor de autorización conserva información sobre la relación entre ambos.
</Callout>

<div id="authorization-extension">
  ## Authorization Extension
</div>

La [Auth0 Authorization Extension](/docs/es/customize/extensions/authorization-extension) te permite proporcionar soporte de autorización en tu aplicación asignando Roles, Grupos y Permisos a Users.

La Authorization Extension crea una [Rule](/docs/es/customize/rules) que ampliará el perfil de usuario durante el flujo de autenticación con los Roles, Grupos y Permisos asignados al usuario. Después, puedes usar esta información para asegurarte de que el token de acceso emitido a un usuario solo contenga scopes permitidos de acuerdo con los permisos definidos en la Authorization Extension.
