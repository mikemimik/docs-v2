<div id="define-the-api-endpoints">
  ## Defina los endpoints de la API
</div>

Un **endpoint de API** es un URI estático que representa un recurso (colección de datos).

Por ejemplo, la API de un restaurante podría tener endpoints como `/orders` y `/customers`. Una aplicación que se conecta a esta API puede realizar operaciones CRUD (crear, leer, actualizar y eliminar) llamando a un endpoint de API con el método HTTP correspondiente (`POST`, `GET`, `PUT`, `PATCH` o `DELETE`).

Para la API de parte de horas de ExampleCo, deberá configurar un endpoint para crear hojas de horas.

| Método HTTP | API endpoint         | Descripción                  |
| ----------- | -------------------- | ---------------------------- |
| `POST`      | `/timesheets/upload` | Crea una nueva hoja de horas |

```json lines
{
  'user_id': '007',
  'date': '2017-05-10T17:40:20.095Z',
  'project': 'StoreZero',
  'hours': 5
}
```

Si la API procesa correctamente la solicitud, envía una respuesta con un código de estado `HTTP 201 Created` y un cuerpo que contiene un objeto JSON con una propiedad `message` que describe la hoja de hora recién creada:

```json lines
{
"message": "Created timesheet 14 for employee 007."
}
```

<div id="secure-the-api-endpoints">
  ### Protege los endpoints de la API
</div>

Para proteger los endpoints de tu API, debes implementar una función de middleware en tu aplicación de API para gestionar los tokens. Esta función comprueba si se incluyó un token en la request a la API, valida el token y, a continuación, confirma si están presentes los scope(s) necesarios para realizar la acción solicitada.

Si se cumplen todos los criterios, la aplicación de API responde con el mensaje mencionado anteriormente. Si hay algún problema con el <Tooltip tip="Token de acceso: credencial de autorización, en forma de una cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=access+token">token de acceso</Tooltip> proporcionado (o no se proporciona en absoluto), la aplicación de API envía una respuesta con el código de estado `HTTP 401 Unauthorized`.

**Consulta la implementación en** [**Node.js**](/docs/es/get-started/architecture-scenarios/server-application-api/api-implementation-nodejs#2-secure-the-api-endpoint)**.**

<div id="get-an-access-token">
  #### Obtener un token de acceso
</div>

Para obtener un token de acceso sin usar la implementación de ejemplo de nuestra aplicación, llama al endpoint [Get Token](https://auth0.com/docs/api/authentication#get-token54) de Auth0 Authentication API con el siguiente payload:

{/* codeblockOld.header.login.logInButton codeblockOld.header.login.configureSnippet */}

```json lines
{
  audience: "{yourApiIdentifier}",
  grant_type: "client_credentials",
  client_id: "${account.client_id}",
  client_secret: "${account.client_secret}"
}
```

<div id="check-the-application-permissions">
  ## Comprueba los permisos de la aplicación
</div>

Ahora hemos protegido el endpoint de nuestra API con un token de acceso, pero aún no hemos comprobado que el proceso que llama a la API tenga permisos para registrar una nueva hoja de horas.

Como se comentó antes, cada token de acceso puede incluir una lista de los permisos que se han concedido a la aplicación. Estos permisos se definen mediante el parámetro de solicitud `scope`. Para saber cómo configurarlo, consulta el apartado [Configure the Scopes](/docs/es/get-started/architecture-scenarios/server-application-api/part-3#configure-the-scopes).

Para nuestro endpoint, requeriremos el scope `batch:upload`.

**Consulta la implementación en** [**Node.js**](/docs/es/get-started/architecture-scenarios/server-application-api/api-implementation-nodejs#3-check-the-client-permissions)**.**

<div id="implement-the-machine-to-machine-application">
  ### Implementar la aplicación de máquina a máquina
</div>

En esta sección, veremos cómo implementar una aplicación de máquina a máquina para nuestro caso de uso.

### Obtener un token de acceso

Comenzaremos invocando el endpoint de la API de Auth0 `/oauth/token` para obtener un token de acceso.

Para ello, necesitaremos los siguientes valores de configuración, que puedes encontrar en la [configuración de la aplicación](https://manage.auth0.com/#/applications):

* **Domain**: el dominio de Auth0, así como el identificador de tu tenant. Este valor formará parte de la URL de la API: `https://{yourTenant}/oauth/token`.
* **<Tooltip tip="Audience: Identificador único de la audiencia de un token emitido. Se llama aud en un token; su valor contiene el ID de una aplicación (ID de cliente) para un token de ID o de una API (identificador de la API) para un token de acceso." cta="Ver glosario" href="/docs/es/glossary?term=Audience">Audience</Tooltip>**: identificador de la API.
* **<Tooltip tip="Client ID: Valor de identificación otorgado por Auth0 a tu recurso registrado." cta="Ver glosario" href="/docs/es/glossary?term=Client+ID">Client ID</Tooltip>**: ID de cliente de la aplicación de Auth0.
* **<Tooltip tip="Client Secret: Secret usado por un cliente (aplicación) para autenticarse con el servidor de autorización; solo el cliente y el servidor de autorización deberían conocerlo, y debe ser lo suficientemente aleatorio como para que no se pueda adivinar." cta="Ver glosario" href="/docs/es/glossary?term=Client+Secret">Client Secret</Tooltip>**: secreto de cliente de la aplicación de Auth0.

Nuestra implementación debe realizar una operación `POST` al endpoint `https://{yourDomain}/oauth/token` con un payload en el siguiente formato:

{/* codeblockOld.header.login.logInButton codeblockOld.header.login.configureSnippet */}

```json lines
{
  "audience": "{yourApiIdentifier}",
  "grant_type": "client_credentials",
  "client_id": "${account.client_id}",
  "client_secret": "${account.client_secret}"
}
```

Para obtener más información, consulta [Llama a tu API mediante Client Credentials Flow](/docs/es/get-started/authentication-and-authorization-flow/client-credentials-flow/call-your-api-using-the-client-credentials-flow).

**Consulta la implementación en** [**Python**](/docs/es/get-started/architecture-scenarios/server-application-api/cron-implementation-python#get-an-access-token).

<div id="invoke-the-api">
  ## Invocar la API
</div>

Ahora que tenemos un token de acceso que incluye los scopes válidos, podemos invocar nuestra API.

Para hacerlo, vamos a:

* Crear una hoja de horas en formato JSON con valores codificados de forma fija.
* Añadir el token de acceso en el encabezado `Authorization` de la solicitud.
* Realizar la solicitud HTTP `POST`.
* Analizar la respuesta e imprimirla en la terminal (opcional).

**Consulta la implementación en** [**Python**](/docs/es/get-started/architecture-scenarios/server-application-api/cron-implementation-python#invoke-the-api).
