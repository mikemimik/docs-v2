<div id="define-the-api-endpoints">
  ## Defina los puntos de conexión de API
</div>

Un **punto de conexión de API** es un URI estático que representa un recurso (colección de datos).

Por ejemplo, una API de restaurante podría tener puntos de conexión de API como `/orders` y `/customers`. Una aplicación que se conecta a esta API puede realizar operaciones CRUD (crear, leer, actualizar y eliminar) al llamar a un punto de conexión de API con el método HTTP correspondiente (`POST`, `GET`, `PUT`, `PATCH` o `DELETE`).

Para la API de parte de horas de ExampleCo, deberá configurar un punto de conexión de API para crear hojas de horas.

| Método HTTP | punto de conexión de API         | Descripción                     |
| ----------- | -------------------- | ------------------------------- |
| `POST`      | `/timesheets/upload` | Crea un nuevo registro de horas |

```json lines
{
  'user_id': '007',
  'date': '2017-05-10T17:40:20.095Z',
  'project': 'StoreZero',
  'hours': 5
}
```

Si la API procesa la solicitud correctamente, envía una respuesta con el código de estado `HTTP 201 Created` y un cuerpo que contiene un objeto JSON con una propiedad `message` que describe la hoja de hora recién creada:

```json lines
{
"message": "Created timesheet 14 for employee 007."
}
```

<div id="secure-the-api-endpoints">
  ### Protege los puntos de conexión de API de la API
</div>

Para proteger los puntos de conexión de API de tu API, debes implementar una función de middleware en tu aplicación de API para gestionar los tokens. Esta función comprueba si se incluyó un token en la solicitud a la API, valida el token y luego confirma si están presentes los scopes necesarios para realizar la acción solicitada.

Si se cumplen todos los criterios, la aplicación de API responde con el mensaje mencionado anteriormente. Si hay algún problema con el <Tooltip tip="Token de acceso: credencial de Authorization, en forma de una cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=access+token">token de acceso</Tooltip> proporcionado (o no se proporciona en absoluto), la aplicación de API envía una respuesta con el código de estado `HTTP 401 Unauthorized`.

**Consulta la implementación en** [**Node.js**](/docs/es/get-started/architecture-scenarios/server-application-api/api-implementation-nodejs#2-secure-the-api-endpoint)**.**

<div id="get-an-access-token">
  #### Obtener un token de acceso
</div>

Para obtener un token de acceso sin usar la implementación de ejemplo de nuestra aplicación, llama al endpoint [Get Token](https://auth0.com/docs/api/authentication#get-token54) de la Auth0 Authentication API con el siguiente payload:

{/* codeblockOld.header.login.logInButton codeblockOld.header.login.configureSnippet */}

```json lines
{
  audience: "{yourApiIdentifier}",
  grant_type: "client_credentials",
  client_id: "${account.client_id}",
  client_secret: "${account.client_secret}"
}
```

<div id="check-the-application-permissions">
  ## Comprueba los permisos de la aplicación
</div>

Ahora hemos protegido el punto de conexión de API de nuestra API con un token de acceso, pero todavía no nos hemos asegurado de que el proceso que llama a la API tenga los permisos necesarios para registrar una nueva hoja de horas.

Como se comentó antes, cada token de acceso puede incluir una lista de los permisos que se han concedido a la aplicación. Estos permisos se definen mediante el parámetro de solicitud `scope`. Para saber cómo configurar esto, consulta el párrafo [Configurar los scopes](/docs/es/get-started/architecture-scenarios/server-application-api/part-3#configure-the-scopes).

Para nuestro punto de conexión de API, requeriremos el scope `batch:upload`.

**Consulta la implementación en** [**Node.js**](/docs/es/get-started/architecture-scenarios/server-application-api/api-implementation-nodejs#3-check-the-client-permissions)**.**

<div id="implement-the-machine-to-machine-application">
  ### Implementar la aplicación de máquina a máquina
</div>

En esta sección, veremos cómo implementar una aplicación de máquina a máquina para nuestro caso.

### Obtener un token de acceso

Comenzaremos llamando al endpoint de la API de Auth0 `/oauth/token` para obtener un token de acceso.

Para ello, necesitaremos los siguientes valores de configuración, que puedes encontrar en la [configuración de la aplicación](https://manage.auth0.com/#/applications):

* **Domain**: dominio de Auth0 y también el identificador de tu tenant. Este valor formará parte de la URL de la API: `https://{yourTenant}/oauth/token`.
* **<Tooltip tip="Audience: Identificador único de la audiencia de un token emitido. Se denomina aud en un token y su valor contiene el ID de una aplicación (ID de cliente) para un token de ID o de una API (identificador de API) para un token de acceso." cta="Ver glosario" href="/docs/es/glossary?term=Audience">Audience</Tooltip>**: identificador de API.
* **<Tooltip tip="Client ID: Valor de identificación asignado por Auth0 a tu recurso registrado." cta="Ver glosario" href="/docs/es/glossary?term=Client+ID">ID de cliente</Tooltip>**: ID de cliente de la aplicación de Auth0.
* **<Tooltip tip="Client Secret: Secret utilizado por un cliente (aplicación) para autenticarse con el servidor de autorización; solo deben conocerlo el cliente y el servidor de autorización, y debe ser lo suficientemente aleatorio como para que no pueda adivinarse." cta="Ver glosario" href="/docs/es/glossary?term=Client+Secret">secreto de cliente</Tooltip>**: secreto de cliente de la aplicación de Auth0.

Nuestra implementación debe realizar una operación `POST` al endpoint `https://{yourDomain}/oauth/token` con un payload en el siguiente formato:

{/* codeblockOld.header.login.logInButton codeblockOld.header.login.configureSnippet */}

```json lines
{
  "audience": "{yourApiIdentifier}",
  "grant_type": "client_credentials",
  "client_id": "${account.client_id}",
  "client_secret": "${account.client_secret}"
}
```

Para obtener más información, consulta [Llama a tu API con Client Credentials Flow](/docs/es/get-started/authentication-and-authorization-flow/client-credentials-flow/call-your-api-using-the-client-credentials-flow).

**Consulta la implementación en** [**Python**](/docs/es/get-started/architecture-scenarios/server-application-api/cron-implementation-python#get-an-access-token).

<div id="invoke-the-api">
  ## Invocar la API
</div>

Ahora que tenemos un token de acceso que incluye los scopes válidos, podemos invocar nuestra API.

Para hacerlo, haremos lo siguiente:

* Crear una hoja de horas predefinida en formato JSON.
* Añadir el token de acceso como encabezado `Authorization` a nuestra solicitud.
* Realizar la solicitud HTTP `POST`.
* Analizar la respuesta e imprimirla en la terminal (opcional).

**Consulta la implementación en** [**Python**](/docs/es/get-started/architecture-scenarios/server-application-api/cron-implementation-python#invoke-the-api).
