Para garantizar que solo los usuarios y las aplicaciones autorizados tengan acceso a la API de parte de horas, ExampleCo ha decidido hacer uso del [marco de autorización OAuth 2.0](https://tools.ietf.org/html/rfc6749). Este marco proporciona la flexibilidad que la empresa busca, ya que las distintas concesiones le permiten autorizar fácilmente los diversos tipos de aplicaciones que necesitan comunicarse con la API de parte de horas.

<div id="oauth-20">
  ## OAuth 2.0
</div>

Al utilizar el marco de autorización de <Tooltip tip="OAuth 2.0: Marco de autorización que define protocolos y flujos de trabajo de autorización." cta="Ver glosario" href="/docs/es/glossary?term=OAuth+2.0">OAuth 2.0</Tooltip>, la aplicación Regular Web de ExampleCo y la aplicación de terceros para contratistas externos pueden tener acceso limitado a la API de parte de horas. Con Auth0, ExampleCo puede admitir fácilmente distintos [tipos de concesión](/docs/es/get-started/authentication-and-authorization-flow/which-oauth-2-0-flow-should-i-use) o flujos de autenticación sin tener que preocuparse por la especificación de OAuth 2.0/<Tooltip tip="OpenID: Estándar abierto de autenticación que permite a las aplicaciones verificar las identidades de los usuarios sin recopilar ni almacenar información de inicio de sesión." cta="Ver glosario" href="/docs/es/glossary?term=OpenID">OpenID</Tooltip> Connect (OIDC) ni por los muchos otros aspectos técnicos de la autorización de APIs.

<Card title="Roles de OAuth">
  En cualquier flujo de OAuth 2.0 podemos identificar los siguientes roles:

  * **Propietario del recurso**: la entidad que puede conceder acceso a un recurso protegido. Normalmente, es el usuario final.
  * **Servidor de recursos**: el servidor que aloja los recursos protegidos. Es la API a la que quieres acceder.
  * **Cliente**: una aplicación que solicita acceso a un recurso protegido en nombre del Propietario del recurso.
  * **Servidor de autorización**: el servidor que autentica al Propietario del recurso y emite tokens de acceso tras obtener la autorización adecuada. En este caso, la Authentication API de Auth0.

  Los [tipos de concesión (o flujos)](/docs/es/get-started/authentication-and-authorization-flow/which-oauth-2-0-flow-should-i-use) determinan cómo interactúan estos participantes para conceder a las aplicaciones acceso limitado a las APIs que estás creando. Como resultado, la aplicación obtendrá un token de acceso que puede usarse para llamar a la API en nombre del usuario.
</Card>

<div id="client-credentials-grant">
  ### Concesión de credenciales de cliente
</div>

OAuth 2 ofrece varios tipos de concesión para distintos casos de uso. En este caso de uso en particular, en el que una tarea cron cargará partes de horas mediante una API, no hay ningún usuario interactivo (ni <Tooltip tip="Resource Owner: Entidad (como un usuario o una aplicación) capaz de conceder acceso a un recurso protegido." cta="Ver glosario" href="/docs/es/glossary?term=resource+owner">propietario del recurso</Tooltip>) que pueda conceder permisos a la tarea cron para acceder a la API.

La tarea cron tampoco realiza llamadas a la API en nombre de ningún usuario. En cambio, la aplicación (la tarea cron) usa autorización de máquina a máquina y realiza llamadas al <Tooltip tip="Resource Server: Servidor que aloja recursos protegidos. Los servidores de recursos aceptan y responden a solicitudes de recursos protegidos." cta="Ver glosario" href="/docs/es/glossary?term=Resource+Server">servidor de recursos</Tooltip> (la API) en su propio nombre.

Para situaciones como esta, en las que no interviene ninguna interacción del usuario, la [Concesión de credenciales de cliente](/docs/es/get-started/authentication-and-authorization-flow/client-credentials-flow) es ideal. Con la Concesión de credenciales de cliente (definida en [RFC 6749, sección 4.4](https://tools.ietf.org/html/rfc6749#section-4.4)), una aplicación puede solicitar directamente un <Tooltip tip="Access Token: Credencial de autorización, en forma de una cadena opaca o JWT, usada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=access+token">token de acceso</Tooltip> al <Tooltip tip="Access Token: Credencial de autorización, en forma de una cadena opaca o JWT, usada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=Authorization+Server">servidor de autorización</Tooltip> usando sus credenciales de cliente (un <Tooltip tip="Authorization Server: Servidor centralizado que contribuye a definir los límites del acceso de un usuario. Por ejemplo, su servidor de autorización puede controlar los datos, tareas y funcionalidades disponibles para un usuario." cta="Ver glosario" href="/docs/es/glossary?term=Client+ID">ID de cliente</Tooltip> y un <Tooltip tip="Client Secret: Secret usado por un cliente (aplicación) para autenticarse con el servidor de autorización; solo deberían conocerlo el cliente y el servidor de autorización, y debe ser lo suficientemente aleatorio para que no pueda adivinarse." cta="Ver glosario" href="/docs/es/glossary?term=Client+Secret">secreto de cliente</Tooltip>). En lugar de identificar a un propietario del recurso, este token representará a la propia aplicación.

<Frame>![undefined](/docs/images/cdy7uua7fh8z/5CfNEkbyG1ZC5BqHwi9gEs/309babf8329b165f1241f4cbc8e002ba/client-credentials-grant.png)</Frame>

1. La aplicación se autentica con el servidor de autorización usando su ID de cliente y su secreto de cliente.
2. El servidor de autorización valida esta información y devuelve un token de acceso.
3. La aplicación puede usar el token de acceso para llamar al servidor de recursos en su propio nombre.

<div id="api-authentication-and-authorization">
  ## Autenticación y autorización de API
</div>

Una API es una forma de poner la funcionalidad de tu aplicación a disposición de otras aplicaciones. Otras aplicaciones pueden hacer una solicitud a un endpoint de API y recibir una respuesta. Del mismo modo, la aplicación externa que usan los contratistas de ExampleCo puede comunicarse tanto con la API de parte de horas como con la Regular Web Application que ExampleCo creó para los empleados internos.

Dado que la API de parte de horas procesa información confidencial (como PII e información financiera), ExampleCo debe asegurarse de que solo los usuarios y las aplicaciones autorizados puedan llamar a sus endpoints.

<div id="access-tokens-and-scopes">
  ### Tokens de acceso y scopes
</div>

Las API pueden estar protegidas o no. Cuando una aplicación quiere acceder a endpoints protegidos de una API, necesita proporcionar un token de acceso (también denominado `access_token`) como prueba de que tiene los permisos necesarios.

Un token de acceso es una cadena opaca que representa una autorización emitida a la aplicación y se obtiene autenticando al usuario con un servidor de autorización. A su vez, el usuario puede autorizar a la aplicación para que acceda a la API en su nombre. Para obtener más información, consulta [Tokens de acceso](/docs/es/secure/tokens/access-tokens).

Una API como la API de parte de horas puede aplicar un control detallado sobre quién puede acceder a los distintos endpoints expuestos por la API. Estos permisos se expresan como scopes.

Cuando la Regular Web Application de ExampleCo o la aplicación de terceros se autentica con Auth0 para obtener un token de acceso, la solicitud de autenticación incluye la lista de scopes solicitados que la aplicación necesita. Si esos scopes están permitidos, el token de acceso contendrá una lista de scopes autorizados concedidos a la aplicación.

La aplicación web tradicional o la aplicación de terceros incluye el token de acceso del servidor de autorización al realizar solicitudes a la API de parte de horas, y la API de parte de horas inspecciona el claim `scope` para asegurarse de que se hayan concedido los permisos necesarios para llamar al endpoint en cuestión.

Por ejemplo, la API de parte de horas puede aceptar cuatro niveles diferentes de autorización: leer partes de horas (scope `read:timesheets`), crear partes de horas (scope `create:timesheets`), eliminar partes de horas (scope `delete:timesheets`) y aprobar partes de horas (scope `approve:timesheets`).

Para obtener más información sobre los scopes, consulta [Scopes](/docs/es/get-started/apis/scopes).

Cuando la aplicación web tradicional envía una solicitud a la API de parte de horas para crear una nueva hoja de horas, el token de acceso debe contener el scope `create:timesheets` o la solicitud será denegada. Del mismo modo, para eliminar partes de horas existentes, el token de acceso debe contener el scope `delete:timesheets`.

Para obtener más información, consulta [Scopes](/docs/es/get-started/apis/scopes).
