Como parte del [Escenario de Arquitectura de Servidor + API](/docs/es/get-started/architecture-scenarios/server-application-api), implementaremos la API de parte de horas en Node.js. Consulta el escenario para obtener más información sobre la solución implementada.

El código fuente completo de la implementación de la API en Node.js se encuentra en [este repositorio de GitHub](https://github.com/auth0-samples/auth0-pnp-exampleco-timesheets/tree/master/timesheets-api/node).

<div id="step-1-define-the-api-endpoint">
  ## Paso 1. Defina el endpoint de la API
</div>

Usaremos el [framework para aplicaciones web Express](http://expressjs.com/) para crear nuestra API de Node.js.

<div id="create-a-packagejson-file">
  ### Crear un archivo package.json
</div>

Crea una carpeta para tu API, accede a ella y ejecuta `npm init`. Esto creará tu archivo `package.json`.

Puedes dejar la configuración predeterminada o cambiarla como prefieras.

El archivo `package.json` de nuestro ejemplo es el siguiente:

```json lines
{
  "name": "timesheets-api",
  "version": "1.0.0",

  "main": "index.js",

  },
  "homepage": "https://github.com/auth0-samples/auth0-pnp-timesheets#readme"
}
```

<div id="install-the-dependencies">
  ### Instala las dependencias
</div>

A continuación, necesitamos configurar nuestras dependencias. Usaremos los siguientes módulos:

* **express**: Este módulo agrega el [framework de aplicaciones web Express](https://expressjs.com/).
* **jwks-rsa**: Esta biblioteca recupera claves de firma RSA desde un endpoint de JWKS (JSON Web Key Set). Con `expressJwtSecret`, podemos generar un proveedor de secretos que proporcionará la clave de firma correcta a `express-jwt` según el `kid` del header del <Tooltip tip="JSON Web Token (JWT): formato estándar de token de ID (y, a menudo, formato de token de acceso) utilizado para representar claims de forma segura entre dos partes." cta="Ver glosario" href="/docs/es/glossary?term=JWT">JWT</Tooltip>. Para obtener más información, consulta el [GitHub repository de node-jwks-rsa](https://github.com/auth0/node-jwks-rsa).
* **express-jwt**: Este módulo te permite autenticar HTTP requests usando tokens JWT en tus aplicaciones de Node.js. Proporciona varias funciones que facilitan el trabajo con JWT. Para obtener más información, consulta el [GitHub repository de express-jwt](https://github.com/auth0/express-jwt).
* **body-parser**: Este es un middleware de análisis del body para Node.js. Extrae todo el contenido del body de un flujo de request entrante y lo expone en `req.body` de una manera más fácil de usar. Para obtener más información y varias alternativas, consulta el GitHub repository de body-parser.

Para instalar estas dependencias, ejecuta lo siguiente:

```bash lines
npm install express express-jwt jwks-rsa body-parser --save
```

<div id="implement-the-endpoint">
  ### Implementa el endpoint
</div>

Ve al directorio de tu API y crea un archivo `server.js`. Tu código debe:

* Definir las dependencias.
* Habilitar el middleware de análisis del cuerpo de la solicitud.
* Implementar el endpoint.
* Iniciar el servidor de la API.

Esta es nuestra implementación de ejemplo:

```javascript lines
// establecer dependencias
const express = require('express');
const app = express();
const jwt = require('express-jwt');
const jwksRsa = require('jwks-rsa');
const bodyParser = require('body-parser');

// habilitar el uso del middleware de análisis del cuerpo de la solicitud
app.use(bodyParser.json());
app.use(bodyParser.urlencoded({
  extended: true
}));

// crear el endpoint de la API para subir hojas de hora
app.post('/timesheets/upload', function(req, res){
  res.status(201).send({message: "This is the POST /timesheets/upload endpoint"});
})

// iniciar el servidor de la API en localhost:8080
app.listen(8080);
```

Inicia el servidor de la API con `node server` y haz una solicitud HTTP POST a `localhost:8080/timesheets/upload`. Deberías ver una respuesta JSON con el mensaje `This is the POST /timesheets/upload endpoint`.

Ahora ya tenemos nuestro endpoint, pero cualquiera puede llamarlo. Continúa con el siguiente párrafo para ver cómo podemos solucionar esto.

<div id="step-2-secure-the-api-endpoint">
  ## Paso 2. Protege el endpoint de la API
</div>

Para validar nuestro token usaremos la función `jwt`, proporcionada por el [middleware express-jwt](https://github.com/auth0/express-jwt#usage), y el paquete `jwks-rsa` para recuperar la clave pública de Auth0. Las bibliotecas hacen lo siguiente:

1. `express-jwt` decodificará el token y pasará la request, el header y el payload a `jwksRsa.expressJwtSecret`.
2. `jwks-rsa` descargará entonces todas las claves de firma del endpoint JWKS y comprobará si alguna de ellas coincide con el `kid` del header del JWT. Si ninguna coincide con el `kid` entrante, se generará un error. Si hay coincidencia, pasaremos la clave de firma correcta a `express-jwt`.
3. `express-jwt` continuará con su propia lógica para validate la firma del token, la expiración, `audience` y el `issuer`.

Los pasos que seguiremos en nuestro código son:

* Crear la función middleware para validate el <Tooltip tip="Access Token: credencial de autorización, en forma de una cadena opaca o JWT, utilizada para acceder a una API." cta="View Glossary" href="/docs/es/glossary?term=access+token">token de acceso</Tooltip>.
* Habilitar el uso del middleware en nuestras routes.

Este también es un buen momento para implementar la lógica para guardar las timesheet entries en una base de datos local, o cualquier otro storage mechanism que prefieras. Esta es nuestra implementación de ejemplo (se omite parte del código por brevedad):

// Crear un middleware para validar el JWT
const checkJwt = jwt({
  // Proporcionar dinámicamente una clave de firma en función del kid del header y de las claves de firma proporcionadas por el endpoint JWKS.
  secret: jwksRsa.expressJwtSecret({
    cache: true,
    rateLimit: true,
    jwksRequestsPerMinute: 5,

Si iniciamos el servidor ahora y hacemos una solicitud HTTP POST a `localhost:8080/timesheets/upload`, deberíamos recibir el mensaje de error `Missing or invalid token` (lo cual es perfectamente normal, ya que no enviamos ningún token de acceso en la solicitud).

Para probar también el caso en que sí funciona, necesitamos:

* Obtener un token de acceso. Para más información sobre cómo hacerlo, consulta: [Obtener un token de acceso](/docs/es/get-started/architecture-scenarios/server-application-api#get-an-access-token)
* Invocar la API agregando el encabezado `Authorization` a la solicitud con el valor `Bearer ACCESS_TOKEN` (donde `ACCESS_TOKEN` es el valor del token que obtuvimos en el primer paso).

<div id="step-3-check-the-client-permissions">
  ## Paso 3. Comprobar los permisos del cliente
</div>

En este paso añadiremos a nuestra implementación la capacidad de comprobar si el cliente tiene permisos (o `scope`) para usar nuestro endpoint y cargar una hoja de hora. En concreto, queremos asegurarnos de que el token tenga el `scope` correcto, que es `batch:upload`.

Para ello, utilizaremos el paquete de Node.js `express-jwt-authz`, así que añádelo a tu proyecto:

```bash lines
npm install express-jwt-authz --save
```

Ahora es tan sencillo como añadir una llamada a `jwtAuthz(...)` a tu middleware para asegurarte de que el JWT contenga un scope determinado a fin de ejecutar un endpoint concreto. Esta es nuestra implementación de ejemplo (se omite parte del código por brevedad):

```javascript lines
// establecer dependencias - parte del código omitido
const jwtAuthz = require('express-jwt-authz');

// Crear middleware para verificar el JWT

// Habilitar el uso del middleware de análisis del cuerpo de la solicitud
app.use(bodyParser.json());
app.use(bodyParser.urlencoded({
  extended: true
}));

// Endpoint de carga por lotes
app.post('/timesheets/upload', checkJwt, jwtAuthz(['batch:upload']), function(req, res){
  var timesheet = req.body;

  // Guardar la hoja de horas en la base de datos...

  //enviar la respuesta
  res.status(201).send(timesheet);
});

// iniciar el servidor de la API en localhost:8080 - código omitido
```

Si invocamos nuestra API con un token que no incluya este scope, deberíamos recibir el error message Forbidden con el status code HTTP `403`. Puedes comprobarlo quitando este scope de tu API.
