En este escenario, construiremos una API de hojas de horas para una empresa ficticia llamada ExampleCo. La API permitirá agregar entradas adicionales de hoja de horas para un empleado o un contratista.

También construiremos una tarea cron que procesará entradas de hoja de horas desde un sistema externo hacia la base de datos centralizada de hojas de horas mediante la API.

<Info>
  ### En resumen

  * Auth0 proporciona autenticación y autorización para API como medio para proteger el acceso a los endpoints de API (lea [Autenticación y autorización de API](/docs/es/get-started/architecture-scenarios/server-application-api/part-1#api-authentication-and-authorization))
  * Para autorizar una Machine-to-Machine Application (una CLI, un servicio o un daemon en el que no hay interacción del usuario), Auth0 admite la Concesión de credenciales de cliente (lea [Concesión de credenciales de cliente](/docs/es/get-started/architecture-scenarios/server-application-api/part-1#client-credentials-grant))
  * Tanto la Machine-to-Machine Application como la API deben configurarse en el Auth0 Dashboard (lea [Auth0 Configuration](/docs/es/get-started/architecture-scenarios/server-application-api/part-2))
  * La API se protegerá asegurando que se envíe un token de acceso válido (implementado como un JSON Web Token) en el encabezado HTTP Authorization cuando se hagan llamadas a la API (lea [Implementar la API](/docs/es/get-started/architecture-scenarios/server-application-api/part-3))
  * Tras una autorización correcta, se emite un token de acceso a la Machine-to-Machine Application (lea [Obtener un token de acceso](/docs/es/get-started/architecture-scenarios/server-application-api/part-3#get-an-access-token))
  * La Machine-to-Machine Application puede, a su vez, usar este token de acceso y enviarlo en el encabezado HTTP Authorization para autenticar llamadas a los endpoints de API (lea [Invocar la API](/docs/es/get-started/architecture-scenarios/server-application-api/part-3#invoke-the-api))
</Info>

<div id="the-premise">
  ## La premisa
</div>

ExampleCo es una startup de consultoría. Actualmente tiene aproximadamente 100 empleados y también recurre a contratistas externos. Todos los empleados y los contratistas externos deben completar hojas de hora cada semana. Para ello, crearon una aplicación de hojas de hora, un caso que cubrimos en [Inicio de sesión único para aplicaciones web tradicionales](/docs/es/get-started/architecture-scenarios/sso-for-regular-web-apps). Los empleados internos usan esta aplicación web para completar sus hojas de hora, pero algunos de los contratistas externos ya utilizan otra herramienta para registrar sus hojas de hora. ExampleCo necesita una solución para evitar duplicar el trabajo y decidió crear una tarea cron para leer las entradas de la hoja de hora de este sistema externo y cargarlas automáticamente en el backend de ExampleCo mediante una API.

<div id="goals-requirements">
  ### Objetivos y requisitos
</div>

ExampleCo quiere crear una solución flexible que:

* Permita procesos automatizados que se ejecuten en el backend
* Permita incorporar en la arquitectura futuros lanzamientos de aplicaciones, como una aplicación móvil
* Permita que la API de parte de horas sea segura y que solo usuarios y aplicaciones autorizados puedan acceder a ella
* Permita compartir una gran parte del código y de la lógica de negocio de la aplicación con otras aplicaciones
