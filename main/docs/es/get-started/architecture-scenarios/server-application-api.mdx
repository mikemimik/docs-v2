En este escenario, construiremos una API de Timesheet para una empresa ficticia llamada ExampleCo. La API permitirá agregar entradas adicionales de hoja de horas para un empleado o un contratista.

También construiremos una tarea cron que procesará entradas de hoja de horas desde un sistema externo hacia la base de datos centralizada de hojas de horas mediante la API.

<Info>
  ### En resumen

  * Auth0 proporciona autenticación y autorización para API como forma de proteger el acceso a los endpoints de la API (lee [Autenticación y autorización de API](/docs/es/get-started/architecture-scenarios/server-application-api/part-1#api-authentication-and-authorization))
  * Para autorizar una Aplicación de máquina a máquina (una CLI, un servicio o un daemon en el que no hay interacción del usuario), Auth0 admite la Concesión de credenciales de cliente (lee [Concesión de credenciales de cliente](/docs/es/get-started/architecture-scenarios/server-application-api/part-1#client-credentials-grant))
  * Tanto la Aplicación de máquina a máquina como la API deben configurarse en el Auth0 Dashboard (lee [Configuración de Auth0](/docs/es/get-started/architecture-scenarios/server-application-api/part-2))
  * La API se protegerá garantizando que se envíe un token de acceso válido (implementado como un JSON Web Token) en el encabezado HTTP Authorization cuando se hagan llamadas a la API (lee [Implementar la API](/docs/es/get-started/architecture-scenarios/server-application-api/part-3))
  * Tras una autorización correcta, se emite un token de acceso para la Aplicación de máquina a máquina (lee [Obtener un token de acceso](/docs/es/get-started/architecture-scenarios/server-application-api/part-3#get-an-access-token))
  * La Aplicación de máquina a máquina puede, a su vez, usar este token de acceso y enviarlo en el encabezado HTTP Authorization para autenticar llamadas a los endpoints de la API (lee [Invocar la API](/docs/es/get-started/architecture-scenarios/server-application-api/part-3#invoke-the-api))
</Info>

<div id="the-premise">
  ## La premisa
</div>

ExampleCo es una startup de consultoría. Actualmente tiene aproximadamente 100 empleados y también recurre a contratistas externos. Todos los empleados y los contratistas externos deben completar sus partes de horas cada semana. Para ello, crearon una aplicación de partes de horas, un escenario que cubrimos en [inicio de sesión único para aplicaciones web tradicionales](/docs/es/get-started/architecture-scenarios/sso-for-regular-web-apps). Los empleados internos usan esta aplicación web para completar sus partes de horas, pero algunos de los contratistas externos ya utilizan otra herramienta para registrar sus horas. ExampleCo necesita una solución para evitar el trabajo duplicado, por lo que decidió crear una tarea cron para leer las entradas de hoja de horas de ese sistema externo y cargarlas automáticamente en el backend de ExampleCo mediante una API.

<div id="goals-requirements">
  ### Objetivos y requisitos
</div>

ExampleCo quiere crear una solución flexible que:

* Permita procesos automatizados que funcionen en el backend
* Permita incorporar en la arquitectura futuras aplicaciones, como una aplicación móvil
* Permita que la API de parte de horas sea segura y que solo usuarios y aplicaciones autorizados puedan acceder a ella
* Permita que una gran parte del código y de la lógica de negocio de la aplicación se comparta con otras aplicaciones
