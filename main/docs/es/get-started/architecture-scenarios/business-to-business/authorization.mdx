Empecemos dando un paso atrás y hablando sobre el control de acceso. No existe una definición única y precisa del control de acceso en la industria, pero si dedica algo de tiempo a buscar y leer, verá que la mayoría de las fuentes autorizadas coinciden en que es el concepto general que engloba la autenticación, la autorización, el consentimiento y la aplicación de políticas para garantizar que solo las personas y los servicios adecuados tengan acceso a sus aplicaciones y API.

A continuación, veamos más de cerca las diferencias entre autenticación, autorización, consentimiento y aplicación de políticas. Su tenant de Auth0 (su <Tooltip tip="Servidor de autorización: servidor centralizado que contribuye a definir los límites del acceso de un usuario. Por ejemplo, su servidor de autorización puede controlar los datos, las tareas y las funcionalidades disponibles para un usuario." cta="Ver glosario" href="/docs/es/glossary?term=Authorization+Server">servidor de autorización</Tooltip>) suele ser responsable de la autenticación y el consentimiento, así como de parte o de la totalidad de la autorización y la aplicación de políticas. Además, una aplicación o API casi siempre es la encargada principal de aplicar las políticas, especialmente cuando se requiere acceso contextual:

* **Autenticación**: el proceso de determinar si una entidad principal (un usuario o una aplicación) es quien o lo que dice ser.
* **Autorización**: el proceso de determinar qué está permitido, en función de la entidad principal, de los permisos que se le hayan concedido y/o del conjunto de criterios de acceso específicos del contexto.
* **Consentimiento**: qué permisos el usuario (<Tooltip tip="Resource Owner: Entidad (como un usuario o una aplicación) capaz de otorgar acceso a un recurso protegido." cta="Ver glosario" href="/docs/es/glossary?term=Resource+Owner">Resource Owner</Tooltip>) ha autorizado a una aplicación a realizar en su nombre. Por lo general, esto es un requisito de la autorización delegada. El usuario debe dar permiso al cliente para acceder a sus datos en un sistema diferente.
* **Aplicación de políticas**: el acto de aplicar las políticas de la aplicación o la API, rechazando o permitiendo el acceso en función de la información de autenticación y/o autorización de un usuario.

En general, solemos agrupar los distintos tipos de control de acceso en tres categorías diferenciadas para que resulte más fácil entender a) qué actor es responsable de almacenar la información, b) qué actor es responsable de tomar decisiones y c) cuál es responsable de aplicar las restricciones.

* La primera categoría es aquella en la que el acceso se concede o se deniega a una aplicación o a una API en su totalidad. Tanto los datos necesarios para aplicarlo como el proceso de aplicación suelen definirse en el contexto del servidor de autorización. Por ejemplo, mediante el uso de [`app_metadata`](/docs/es/manage-users/user-accounts/metadata) asociado a un usuario y de una [Action](/docs/es/customize/actions) definida en su tenant de Auth0.
* La segunda categoría es aquella en la que el acceso se concede o se deniega a un subconjunto específico de la funcionalidad de una aplicación o API. Los datos necesarios para aplicarlo suelen almacenarse en el servidor de autorización. Por ejemplo, mediante el uso de `app_metadata` en un usuario de su tenant de Auth0, mientras que el proceso de aplicación se lleva a cabo en la propia aplicación o API. En este escenario, los datos suelen comunicarse como uno o varios claims personalizados en un token de [`id`](/docs/es/secure/tokens/id-tokens) o [`access`](/docs/es/secure/tokens/access-tokens).
* La tercera categoría es aquella en la que el acceso se concede o se deniega según aquello sobre lo que la entidad principal (subject) puede operar dentro del contexto de una aplicación o API. Tanto los datos necesarios para aplicarlo como el proceso de aplicación suelen definirse en el contexto de la aplicación o API. En este escenario, los datos comunicados como uno o varios claims personalizados en un token de `id` o `access` pueden consumirse con o sin datos de una fuente externa que no sea Auth0.

Además, los mecanismos de control de acceso basado en roles (RBAC) y control de acceso basado en atributos (ABAC) pueden aplicarse en cualquiera de las categorías de control de acceso descritas anteriormente. Sea cual sea su caso de uso, hay una serie de aspectos que querrá tener en cuenta al evaluar la funcionalidad y el flujo de trabajo que necesita:

* ¿Hay escenarios en los que deba rechazarse el acceso a una aplicación o API completas?
* ¿Va a proporcionar API a las que puedan acceder aplicaciones de terceros?
* ¿Sus propias aplicaciones (propio) también accederán a sus API?
* ¿Su aplicación llamará a una API de terceros?
* ¿Sus aplicaciones y/o API deberían aplicar control de acceso en función de las claims del usuario?
* ¿Qué ocurre si necesito saber con qué organización está asociado un <Tooltip tip="Token de acceso: credencial de autorización, en forma de una cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=access+token">token de acceso</Tooltip> o un <Tooltip tip="Token de acceso: credencial de autorización, en forma de una cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=ID+token">token de identidad</Tooltip>?

Auth0 admite restringir el acceso a aplicaciones o API en función de determinadas condiciones. En algunos casos, puede que quiera crear una Action que devuelva un `UnauthorizedError` cuando, por ejemplo, un usuario intente acceder a una aplicación o a una API en un momento no permitido (como se describe en este [ejemplo](/docs/es/manage-users/access-control/sample-use-cases-rules-with-authorization#allow-access-only-on-weekdays-for-a-specific-application)) o si el usuario no tiene la(s) claim(s) correcta(s) en su `app_metadata`. En una aplicación que usa [OpenID Connect (OIDC)](/docs/es/authenticate/protocols/openid-connect-protocol), esto impediría la emisión del [token de ID](/docs/es/secure/tokens/id-tokens) que se utiliza para autorizar el acceso. Del mismo modo, en una API, podría impedirse la emisión de cualquier OAuth2 [token de acceso](/docs/es/secure/tokens/access-tokens) (utilizado al llamar a la API), como se describe en este [ejemplo](/docs/es/manage-users/access-control/sample-use-cases-rules-with-authorization#deny-access-to-anyone-calling-an-api).

<Info>
  ### Práctica recomendada

  En general, hemos comprobado que [OIDC](/docs/es/authenticate/protocols/openid-connect-protocol) es el protocolo estándar del sector más utilizado por los clientes de Auth0 para la autenticación en sus aplicaciones. También hemos observado que, aunque [OAuth2](/docs/es/authenticate/protocols/oauth) se creó como un protocolo de delegación, se usa habitualmente en aplicaciones propias cuando existe una API que no comparte sesión con la aplicación.
</Info>

Auth0 también puede proporcionar la información necesaria para que una aplicación pueda aplicar restricciones. Para la [integración a nivel de aplicación](#application-integration), Auth0 le permite añadir [claims personalizadas](#id-token-claims) a un token de ID, que su aplicación puede verificar y usar posteriormente para aplicar políticas. En este caso, tendrá que decidir qué información necesita su aplicación para tomar decisiones de control. Si necesita tomar decisiones en una API en lugar de hacerlo en su aplicación, probablemente tendrá que usar un token de acceso en lugar de un token de identidad. Siga leyendo para obtener más información.

<Warning>
  Al decidir qué datos incluir en su token de ID y/o token de acceso, tenga en cuenta el tamaño del token, especialmente si va a pasarlo en la URL. Aunque no esté pasando tokens en la URL, también deberá considerar el riesgo de exponer PII (Personally Identifiable Information) confidencial. La información del token no está cifrada, por lo que, aunque por lo general no supone un problema de seguridad que se filtre un token de ID, sí puede convertirse en un problema de privacidad según los datos que incluya el token.
</Warning>

Para la [integración a nivel de API](#api-integration), Auth0 admite tanto las [claims personalizadas](#access-token-claims) como la reconfiguración de [scope](#access-token-scopes), ambas dentro del contexto de un token de acceso. De nuevo, tendrá que decidir qué información será necesaria para que su API tome decisiones de acceso, y su API deberá hacer cumplir esto validando el contenido del token de acceso.

<Info>
  ### Práctica recomendada

  Al decidir si debe usar permissions mediante claims personalizadas o scopes, asegúrese de comprender la naturaleza y el propósito de los scopes. Hay una buena [entrada de blog](https://auth0.com/blog/on-the-nature-of-oauth2-scopes/) sobre ello que es fácil de leer y ayuda a aclarar el tema.
</Info>

En escenarios con varias organizaciones, a menudo puede ser importante saber a qué organización se aplica un token de acceso (o incluso un token de identidad). Seguir las [prácticas recomendadas](#organization-data-in-an-access-token) puede ahorrarle tiempo y esfuerzo.

<div id="application-integration">
  ## Integración de aplicaciones
</div>

En este escenario, tu tenant de Auth0 proporciona un token como indicador de acceso autorizado a una aplicación. En el caso de las aplicaciones que utilizan [OpenID Connect (OIDC)](/docs/es/authenticate/protocols/openid-connect-protocol), el protocolo estándar del sector que normalmente es el más utilizado en aplicaciones orientadas al cliente, se trataría de un token de ID expresado como un [JWT](/docs/es/secure/tokens/json-web-tokens).

<div id="id-token-claims">
  ### Claims del token de ID
</div>

Gracias a la extensibilidad de Actions, Auth0 te permite [añadir claims personalizadas a un token de ID](/docs/es/get-started/apis/scopes/sample-use-cases-scopes-and-claims#add-custom-claims-to-a-token) fácilmente en función de, por ejemplo, el contenido de los [metadatos](/docs/es/manage-users/user-accounts/metadata) de un usuario. Tu aplicación puede verificar después el token de ID para comprobar si contiene los claims necesarios y, según corresponda, permitir o impedir el acceso a determinadas funcionalidades. Ten en cuenta que, aunque el proceso de añadir claims personalizadas mediante Actions está simplificado, Actions ofrece flexibilidad y te permite escribir código personalizado que podría tener efectos negativos.

<Info>
  ### Práctica recomendada

  Si estás pensando en añadir claims personalizadas, te recomendamos almacenar en el [`app_metadata`](/docs/es/manage-users/user-accounts/metadata) del usuario cualquier dato de control de acceso que necesites incluir en los claims. En primer lugar, así evitas tener que llamar a una API externa para obtener esos datos, lo que puede afectar negativamente al rendimiento y la escalabilidad de la secuencia de inicio de sesión. En segundo lugar, el usuario **no puede** modificar `app_metadata`, por lo que no puede eludir directamente ninguna restricción de control de acceso modificando sus propios metadatos. Recuerda también consultar nuestra guía de [prácticas recomendadas sobre metadatos](/docs/es/get-started/architecture-scenarios/business-to-business/profile-management#metadata).
</Info>

Si estás creando distintas instancias de tu aplicación para las organizaciones de tus clientes, una práctica habitual es crear un claim personalizado en tu token de identidad para representar la organización del usuario. Por ejemplo:

```js wrap lines
context.idToken["http://yourdomain.com/claims/organization"] = "organization A";
```

<div id="id-token-scopes">
  ### Scopes del token de ID
</div>

Los [Scopes de OIDC](/docs/es/get-started/apis/scopes/openid-connect-scopes) suelen utilizarse para que una aplicación obtenga el consentimiento para acceder de forma autorizada a la información de un usuario durante la autenticación. Cada uno de los scopes predefinidos devuelve el conjunto de claims estándar cuando corresponde, tal como se describe en la [especificación de OIDC](https://openid.net/specs/openid-connect-core-1_0.html#StandardClaims). Los scopes que solicita una aplicación dependen de los atributos del usuario que necesite. Una vez que el usuario autoriza los scopes solicitados, los claims se devuelven en el token de ID y también están disponibles a través del endpoint [/userinfo](https://auth0.com/docs/api/authentication/user-profile/get-user-info).

<div id="api-integration">
  ## Integración de API
</div>

En este escenario, tu tenant de Auth0 puede proporcionar un OAuth2 [token de acceso](/docs/es/secure/tokens/access-tokens), normalmente expresado como un [JWT](/docs/es/secure/tokens/json-web-tokens), que tu API puede usar para restringir el acceso a determinadas partes. Además, Auth0 ofrece soporte para lo que, conceptualmente, se describe como [aplicaciones propias y de terceros](/docs/es/get-started/applications/first-party-and-third-party-applications).

Actuando como servidor de autorización, y con el consentimiento del usuario (el Resource Owner), tu tenant de Auth0 puede usarse para proporcionar un token de acceso —normalmente expresado como un [JWT](/docs/es/secure/tokens/json-web-tokens)— a una aplicación (cliente) para que pueda acceder a recursos protegidos alojados en un <Tooltip tip="Resource Server: Servidor que aloja recursos protegidos. Los servidores de recursos aceptan y responden a solicitudes de recursos protegidos." cta="Ver glosario" href="/docs/es/glossary?term=resource+server">servidor de recursos</Tooltip> en nombre del Resource Owner. El token de acceso emitido suele enviarse como token Bearer en el encabezado HTTP Authorization de una solicitud a la API.

Tanto si tienes una sola API como un conjunto de [API de microservicios](/docs/es/get-started/apis/set-logical-api) relacionadas lógicamente, puedes aprovechar los tokens de acceso que proporciona Auth0 para proteger el acceso a tus servicios. Aunque es relativamente fácil configurar esto en el [Auth0 Dashboard](https://manage.auth0.com/) o mediante la [Auth0 Management API](https://auth0.com/docs/api/management/v2#!/Resource_Servers/post_resource_servers), es importante revisar los distintos escenarios de aplicaciones y diseños de API para determinar la mejor arquitectura para tu sistema.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Los tokens de acceso de OAuth2 están diseñados principalmente para proteger API expuestas públicamente; cuando se expresa como JWT, un token de acceso es una entidad autocontenida que puede verificarse sin necesidad de realizar ninguna llamada adicional a una API de terceros. Si tus API no entran en esta categoría, es decir, si forman parte de una aplicación en sí (por ejemplo, si solo esa aplicación las invoca) o están detrás de tu firewall, protegerlas con tokens puede ser excesivo, y es posible que tu flujo de trabajo actual basado en cookies (entre otros) sea suficiente.
</Callout>

OAuth2 se diseñó específicamente pensando en el acceso de terceros. Por ejemplo, un escenario podría ser que un usuario (Resource Owner) quiera usar una aplicación (un cliente) que no pertenece a la misma organización que el servicio que proporciona los datos del usuario (el servidor de recursos). En este caso, cuando la aplicación necesita acceder a datos que pertenecen al usuario, redirige a la organización donde residen esos datos, que a su vez autentica al usuario y luego le solicita permiso para que la aplicación pueda acceder a ellos. Esta solicitud de permiso se conoce como otorgar [consentimiento](/docs/es/get-started/applications/third-party-applications/user-consent-and-third-party-applications) y es una parte importante de lo que implica ofrecer soporte para [aplicaciones de terceros](/docs/es/get-started/apis/scopes/api-scopes#example-an-api-called-by-a-third-party-application). Si planeas integrar aplicaciones de terceros, es importante que las marques como de terceros desde el principio para que Auth0 gestione la solicitud de consentimiento del usuario.

Por otro lado, si tu organización posee las aplicaciones, los propios datos del usuario y las API a través de las cuales se accede a esos datos, normalmente no se requiere consentimiento, ya que todas las interacciones son [propias](/docs/es/get-started/apis/scopes/api-scopes#example-an-api-called-by-a-first-party-application). Si solo estás creando aplicaciones propias, puedes asegurarte de no mostrar a tus usuarios pantallas de consentimiento innecesarias [permitiendo omitir el consentimiento del usuario](/docs/es/get-started/apis) como parte de cualquier definición de servicio de recursos.

<Warning>
  Aunque puedes configurar tus aplicaciones como propias y posteriormente configurar tus API para permitir que los clientes propios omitan el consentimiento, si estás usando `localhost` Auth0 no puede verificar que la aplicación sea realmente una aplicación propia, por lo que de todos modos se solicitará el consentimiento a tus usuarios. Para sortear esta restricción, al hacer pruebas en tu equipo local durante el desarrollo, crea un [nombre de host local falso y úsalo en su lugar](https://community.auth0.com/t/how-do-i-skip-the-consent-page-for-my-api-authorization-flow/6035).
</Warning>

Como alternativa, puede que tenga datos relacionados con un usuario para los que se ofrece [funcionalidad adicional](/docs/es/get-started/apis/scopes/api-scopes#example-an-api-called-by-a-back-end-service) y para los que no es posible obtener el consentimiento explícito del usuario (es decir, no hay ningún usuario autenticado que pueda otorgarlo). En este caso, se puede definir una [lista de aplicaciones para las que está habilitada la Concesión de credenciales de cliente](/docs/es/get-started/authentication-and-authorization-flow/client-credentials-flow).

<div id="access-token-claims">
  ### Claims del token de acceso
</div>

Al igual que con los token de ID, puedes [agregar claims personalizadas a los tokens de acceso](/docs/es/get-started/apis/scopes/sample-use-cases-scopes-and-claims#add-custom-claims-to-a-token) mediante la extensibilidad de Auth0 Actions. Una vez agregadas, tu API puede verificar que un token de acceso contenga las claims necesarias y permitir o denegar el acceso a determinadas funcionalidades, según corresponda.

<Info>
  ### Práctica recomendada

  Si estás considerando agregar claims personalizadas, te recomendamos almacenar en el [`app_metadata`](/docs/es/manage-users/user-accounts/metadata) del usuario cualquier dato de control de acceso que necesites incluir en las claims. En primer lugar, esto evita tener que llamar a una API externa para obtener esos datos, lo que puede afectar negativamente al rendimiento y la escalabilidad. En segundo lugar, `app_metadata` **no puede** ser modificado por el usuario, por lo que no podrá eludir directamente ninguna restricción de control de acceso modificando sus propios metadatos. Recuerda también consultar nuestra guía de [prácticas recomendadas sobre metadatos](/docs/es/get-started/architecture-scenarios/business-to-business/profile-management#metadata).
</Info>

<div id="access-token-scopes">
  ### Scopes del token de acceso
</div>

Los [OAuth2 Scopes](/docs/es/get-started/apis/scopes/api-scopes) se suelen usar como el mecanismo mediante el cual una API puede determinar qué acciones se pueden realizar en nombre de un usuario. Se pueden agregar scopes por API para [definir permisos de acceso específicos](/docs/es/get-started/apis/add-api-permissions) en el <Tooltip tip="Auth0 Dashboard: el principal producto de Auth0 para configurar tus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip> o mediante la <Tooltip tip="Auth0 Dashboard: el principal producto de Auth0 para configurar tus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip> de Auth0). Los scopes también se pueden manipular mediante la extensibilidad de Auth0 (por ejemplo, con una Action, como en este [ejemplo](/docs/es/get-started/architecture-scenarios/spa-api/part-2#create-a-rule-to-validate-token-scopes)). Los scopes que una aplicación solicita para acceder a una API deben depender de la funcionalidad para la que la aplicación necesita la autorización del usuario. Una vez autorizados los scopes solicitados, se devolverán en el token de acceso, que posteriormente dicha [API puede validar](/docs/es/secure/tokens/access-tokens/validate-access-tokens). Un buen ejemplo de esto es cuando inicias sesión en una aplicación que usa un proveedor social para el inicio de sesión: la API del proveedor social requiere que la aplicación especifique si el usuario quiere que la aplicación publique contenido en su nombre. Esto permite al usuario aceptar o denegar esta solicitud. Este ejemplo demuestra cómo el usuario está delegando un permiso en la aplicación, lo cual es distinto de que la API restrinja el acceso en función del rol de un usuario, y debe manejarse de otra manera.

<Info>
  ### Práctica recomendada

  Aunque tienes la capacidad de manipular por completo los scopes del token de acceso mediante la extensibilidad de Auth0, como práctica recomendada de seguridad solo deberías eliminar los scopes que no estén autorizados y evitar agregar scopes que no se hayan solicitado.
</Info>

Aunque los scopes suelen usarse como una forma de aplicar permisos de acceso a un usuario, hay situaciones en las que puede resultar [complicado usarlos de esta manera](https://auth0.com/blog/on-the-nature-of-oauth2-scopes/). Por eso, te recomendamos usar los scopes para su propósito previsto (es decir, delegar permisos a una aplicación) y usar [custom claims](#access-token-claims) para escenarios basados en roles u otros de control de acceso.

<div id="fine-grained-authorization-fga">
  ## Autorización detallada (FGA)
</div>

La [autorización detallada](https://docs.fga.dev/fga-concepts) te permite conceder a usuarios individuales acceso a un recurso u objeto específico en función de:

* El rol de un usuario dentro de una organización, como `editor` o `admin`
* Un atributo del usuario o del objeto, como `manager` para un usuario o `marketing` para un objeto
* Una relación entre un usuario y un objeto, como cuando un usuario con acceso de visualización a una carpeta principal también tiene acceso de visualización a la carpeta secundaria

Con <Tooltip tip="Autorización detallada (FGA): producto de Auth0 que permite a usuarios individuales acceder a objetos o recursos específicos." cta="Ver glosario" href="/docs/es/glossary?term=FGA">FGA</Tooltip>, puedes crear un modelo de autorización para definir qué relación quieres usar para determinar el acceso del usuario.

<div id="role-based-access-control-rbac">
  ## Control de acceso basado en roles (RBAC)
</div>

Auth0 ofrece compatibilidad inmediata con el control de acceso basado en roles ([RBAC](/docs/es/manage-users/access-control/rbac)). RBAC consiste en asignar permisos a los usuarios en función de su rol dentro de una organización y permite un control de acceso más sencillo al ofrecer un enfoque más fácil de administrar y menos propenso a errores.

La funcionalidad principal de RBAC puede usarse en muchos escenarios con múltiples organizaciones. Consulta [Datos de organización en un token de acceso](#organization-data-in-an-access-token) para obtener más información sobre cómo garantizar que tu configuración pueda satisfacer tus necesidades de RBAC.

<div id="machine-to-machine-m2m-authorization">
  ## Autorización máquina a máquina (M2M)
</div>

Hay muchos escenarios en los que una aplicación sin ninguna sesión interactiva de usuario necesita obtener un token de acceso para llamar a una API. En esos casos, debes autenticar al cliente en lugar de al usuario, y <Tooltip tip="OAuth 2.0: Marco de autorización que define protocolos y flujos de trabajo de autorización." cta="Ver glosario" href="/docs/es/glossary?term=OAuth">OAuth</Tooltip> 2 proporciona el tipo de concesión de [credenciales de cliente](/docs/es/get-started/authentication-and-authorization-flow/client-credentials-flow) para que esto sea fácil de lograr. Algunos ejemplos habituales en los que esto es necesario son:

* Una tarea cron u otro servicio que necesite comunicarse con tu API (por ejemplo, cuando haya que generar un informe diario y enviárselo por correo electrónico a un administrador).
* Una API independiente que admita acceso privilegiado (por ejemplo, la API no se expone directamente a los usuarios, sino solo a un backend).
* En ciertas arquitecturas de microservicios, donde algunas capas de API necesitan comunicarse con otras capas de API sin intervención de un usuario, o después de que haya expirado un token de usuario.
* Una API privilegiada que quizá deba llamarse antes de que un usuario se haya autenticado (es decir, desde una Action o un script de base de datos personalizado en tu tenant de Auth0)

<Info>
  ### práctica recomendada

  Tradicionalmente, se creaba una &quot;cuenta de servicio&quot; especial para estos escenarios: un usuario con nombre de usuario y contraseña configurado para servicios compatibles con casos de uso no interactivos. Ese enfoque ya no se recomienda por muchas razones, y la práctica recomendada actual es usar la [Concesión de credenciales de cliente de OAuth 2.0](/docs/es/get-started/authentication-and-authorization-flow/client-credentials-flow) en estas situaciones.
</Info>

<div id="organization-data-in-an-access-token">
  ## Datos de la organización en un token de acceso
</div>

Si en su sistema tiene una API independiente de su aplicación que da soporte a su aplicación con varias organizaciones, es importante restringir las operaciones únicamente a la organización para la que se generó el token. Para ello, el token de acceso debe incluir algún tipo de información que indique a la API para qué organización se emitió. Esto puede hacerse de distintas formas, según las respuestas a un par de preguntas sencillas:

1. ¿Los usuarios finales de esta organización podrían pertenecer a más de una organización, o cada usuario final está limitado a una organización específica?
2. ¿Permitirá algún acceso de máquina a máquina (M2M) a su API?
3. Si permite acceso de máquina a máquina (M2M) a su API, ¿tendrá desarrolladores que necesiten un único ID de cliente y secreto para acceder a varias organizaciones (pero no a todas)?
4. ¿Permitirá la creación de aplicaciones de terceros que requieran consentimiento?

Si los usuarios finales están limitados a una sola organización **y** no permitirá acceso M2M a su API, o bien tendrá un <Tooltip tip="ID de cliente: Valor de identificación asignado a su recurso registrado por Auth0." cta="Ver glosario" href="/docs/es/glossary?term=client+ID">ID de cliente</Tooltip>/secreto independiente para cada organización que necesite acceso **y** no permitirá aplicaciones de terceros que requieran consentimiento, entonces el enfoque más sencillo es crear un claim personalizado en el token de acceso [usando Actions para los tokens basados en usuarios](#access-token-claims) y [usando el hook de credenciales de cliente para llamadas M2M](#machine-to-machine-m2m-authorization). Puede almacenar el nombre de la organización en los metadatos del cliente y extraerlo desde Actions o hooks para incluirlo en `access_token` como un claim personalizado. RBAC también funcionará de forma inmediata con este enfoque, siempre que cada usuario final solo pueda pertenecer a una organización.

Si los usuarios finales pueden pertenecer a más de una organización, o podría dar a un mismo desarrollador un ID de cliente y secreto para llamadas M2M a más de una organización, el mejor enfoque es crear un <Tooltip tip="Audience: Identificador único de la audiencia de un token emitido. Se denomina aud en un token; su valor contiene el ID de una aplicación (ID de cliente) para un token de ID o de una API (identificador de API) para un token de acceso." cta="Ver glosario" href="/docs/es/glossary?term=audience">audience</Tooltip> independiente (una instancia de API independiente en su tenant de Auth0) para cada organización. Esto le ofrece varias ventajas:

1. En primer lugar, le permite pasar la audience como un parámetro de primer nivel a Auth0 sin tener que crear un parámetro personalizado. La ventaja es que Auth0 ayudará a garantizar la existencia de la audience y la pasará a sus Actions. También garantizará que un token de actualización emitido solo funcione para la audience específica para la que se emitió originalmente.
2. Le permite restringir las autorizaciones de cliente únicamente a organizaciones específicas de forma nativa. La alternativa es crear un hook de credenciales de cliente más complejo para intentar recuperar las restricciones desde otro lugar, además de requerir una forma mucho más compleja y potencialmente problemática de indicar en la llamada de credenciales de cliente para qué organización debe emitirse el token de acceso.
3. Esto también le permite usar la funcionalidad principal de RBAC con Auth0 y garantizar que los usuarios finales que tienen acceso a más de una organización puedan tener un rol distinto para cada organización.
