Empecemos dando un paso atrás y hablando sobre el control de acceso. No existe una única definición inequívoca de control de acceso en la industria, pero, si dedicas algo de tiempo a buscar y leer, verás que la mayoría de las fuentes autorizadas coinciden en que es el concepto general que engloba la autenticación, la autorización, el consentimiento y la aplicación de políticas para garantizar que solo las personas y los servicios adecuados tengan acceso a tus aplicaciones y API.

A continuación, veamos con más detalle las diferencias entre autenticación, autorización, consentimiento y aplicación de políticas. Tu tenant de Auth0 (tu <Tooltip tip="servidor de autorización: Servidor centralizado que contribuye a definir los límites del acceso de un usuario. Por ejemplo, tu servidor de autorización puede controlar los datos, las tareas y las funcionalidades disponibles para un usuario." cta="Ver glosario" href="/docs/es/glossary?term=Authorization+Server">servidor de autorización</Tooltip>) suele ser responsable de la autenticación y del consentimiento, así como de parte o de la totalidad de la autorización y la aplicación de políticas. Además, una aplicación o la propia API casi siempre es la principal responsable de hacer cumplir las políticas, especialmente cuando se requiere acceso contextual:

* **Autenticación**: el proceso de determinar si una entidad principal (un usuario o una aplicación) es quien o lo que dice ser.
* **Autorización**: el proceso de determinar qué está permitido, en función de la entidad principal, los permisos que se le han otorgado y/o el conjunto de criterios de acceso específicos del contexto.
* **Consentimiento**: los permisos que el usuario (<Tooltip tip="Resource Owner: Entidad (como un usuario o una aplicación) capaz de conceder acceso a un recurso protegido." cta="Ver glosario" href="/docs/es/glossary?term=Resource+Owner">Resource Owner</Tooltip>) ha autorizado a una aplicación a realizar en su nombre. Por lo general, esto es un requisito de la autorización delegada. El usuario tiene que dar permiso al cliente para acceder a sus datos en un sistema diferente.
* **Aplicación de políticas**: el acto de hacer cumplir las políticas de la aplicación o API, rechazando o permitiendo el acceso en función de la información de autenticación y/o autorización de un usuario.

En general, solemos agrupar los distintos tipos de control de acceso en tres categorías diferenciadas para que sea más fácil entender a) qué actor es responsable de almacenar la información, b) qué actor es responsable de tomar decisiones y c) cuál es responsable de aplicar las restricciones.

* La primera categoría es aquella en la que se concede o se deniega acceso a una aplicación o a una API en su totalidad. Tanto los datos necesarios para hacerlo cumplir como el proceso de aplicación suelen definirse en el contexto del servidor de autorización. Por ejemplo, mediante [`app_metadata`](/docs/es/manage-users/user-accounts/metadata) asociado a un usuario y una [Action](/docs/es/customize/actions) definida en tu tenant de Auth0.
* La segunda categoría es aquella en la que se concede o se deniega acceso a un subconjunto específico de la funcionalidad de una aplicación o API. Los datos necesarios para hacerlo cumplir suelen almacenarse en el servidor de autorización. Por ejemplo, mediante `app_metadata` en un usuario de tu tenant de Auth0, mientras que el proceso de aplicación se realiza en la propia aplicación o API. En este escenario, los datos suelen comunicarse como una o varias claims personalizadas en un token de [`id`](/docs/es/secure/tokens/id-tokens) o [`access`](/docs/es/secure/tokens/access-tokens).
* La tercera categoría es aquella en la que se concede o se deniega acceso en función de aquello sobre lo que la entidad principal (subject) puede operar dentro del contexto de una aplicación o API. Tanto los datos necesarios para hacerlo cumplir como el proceso de aplicación suelen definirse en el contexto de la aplicación o API. En este escenario, los datos comunicados como una o varias claims personalizadas en un token de `id` o `access` pueden consumirse con o sin datos de una fuente externa que no sea Auth0.

Además, los mecanismos de control de acceso basado en roles (RBAC) y control de acceso basado en atributos (ABAC) pueden aplicarse en cualquiera de las categorías de control de acceso descritas anteriormente. Sea cual sea tu caso de uso, hay una serie de aspectos que querrás tener en cuenta al analizar la funcionalidad y el flujo que necesitas:

* ¿Hay situaciones en las que deba rechazarse el acceso a una aplicación o API completas?
* ¿Ofrecerá APIs a las que puedan acceder aplicaciones de terceros?
* ¿Sus APIs también serán utilizadas por sus propias aplicaciones (de primera parte)?
* ¿Su aplicación llamará a una API de terceros?
* ¿Deben sus aplicaciones y/o APIs aplicar control de acceso en función de las claims del usuario?
* ¿Qué ocurre si necesito saber a qué organización está asociado un <Tooltip tip="Token de acceso: credencial de autorización, en forma de una cadena opaca o JWT, usada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=access+token">token de acceso</Tooltip> o un <Tooltip tip="Access Token: Authorization credential, in the form of an opaque string or JWT, used to access an API." cta="Ver glosario" href="/docs/es/glossary?term=ID+token">token de ID</Tooltip>?

Auth0 admite restringir el acceso a aplicaciones o APIs en función de determinadas condiciones. En algunos casos, puede que quiera crear una Action que devuelva un `UnauthorizedError` cuando, por ejemplo, un usuario intente acceder a una aplicación o a una API en un momento no permitido (como se describe en este [ejemplo](/docs/es/manage-users/access-control/sample-use-cases-rules-with-authorization#allow-access-only-on-weekdays-for-a-specific-application)) o si el usuario no tiene las claims correctas en su `app_metadata`. En una aplicación que utilice [OpenID Connect (OIDC)](/docs/es/authenticate/protocols/openid-connect-protocol), esto impediría la emisión del [token de ID](/docs/es/secure/tokens/id-tokens) utilizado para autorizar el acceso. Del mismo modo, en una API, podría impedirse la emisión de cualquier [token de acceso](/docs/es/secure/tokens/access-tokens) de OAuth2 (utilizado al llamar a la API), como se describe en este [ejemplo](/docs/es/manage-users/access-control/sample-use-cases-rules-with-authorization#deny-access-to-anyone-calling-an-api).

<Info>
  ### Práctica recomendada

  En general, hemos observado que [OIDC](/docs/es/authenticate/protocols/openid-connect-protocol) es el protocolo estándar del sector más utilizado por los clientes de Auth0 para la autenticación en sus aplicaciones. También hemos comprobado que, aunque [OAuth2](/docs/es/authenticate/protocols/oauth) se creó como un protocolo de delegación, se usa habitualmente en aplicaciones de primera parte cuando hay una API que no comparte sesión con la aplicación.
</Info>

Auth0 también puede proporcionar la información necesaria para que una aplicación aplique restricciones. Para la [integración a nivel de aplicación](#application-integration), Auth0 le permite agregar [claims personalizadas](#id-token-claims) a un token de ID, que su aplicación puede verificar y después usar para aplicar políticas. En este caso, tendrá que decidir qué información necesita para que su aplicación pueda tomar decisiones de aplicación. Si necesita tomar decisiones en una API en lugar de en su aplicación, probablemente tendrá que usar un token de acceso en lugar de un token de ID. Siga leyendo para obtener más información.

<Warning>
  Al decidir qué datos incluir en su token de ID y/o token de acceso, tenga en cuenta el tamaño del token, especialmente si lo pasa en la URL. Aunque no esté pasando tokens en la URL, también deberá considerar la posibilidad de exponer PII (información de identificación personal) sensible. La información del token no está cifrada, por lo que, aunque en general no supone un problema de seguridad que se filtre un token de ID, puede convertirse en un problema de privacidad según los datos incluidos en el token.
</Warning>

Para la [integración a nivel de API](#api-integration), Auth0 admite tanto [claims personalizadas](#access-token-claims) como la reconfiguración de [scope](#access-token-scopes), ambas en el contexto de un token de acceso. De nuevo, tendrá que decidir qué información será necesaria para que su API tome decisiones de acceso, y su API tendrá que aplicarlo validando el contenido del token de acceso.

<Info>
  ### Práctica recomendada

  Al decidir si debe usar permisos mediante claims personalizadas o scopes, debe asegurarse de comprender la naturaleza y el propósito de los scopes. Hay una buena [entrada de blog](https://auth0.com/blog/on-the-nature-of-oauth2-scopes/) sobre ello que es fácil de leer y ayuda a aclarar el tema.
</Info>

En escenarios con varias organizaciones, a menudo puede ser importante saber a qué organización se aplica un token de acceso (o incluso un token de ID). Seguir las [prácticas recomendadas](#organization-data-in-an-access-token) puede ahorrarle tiempo y esfuerzo.

<div id="application-integration">
  ## Integración de la aplicación
</div>

En este escenario, tu tenant de Auth0 proporciona un token como indicador de acceso autorizado a una aplicación. En el caso de las aplicaciones que utilizan [OpenID Connect (OIDC)](/docs/es/authenticate/protocols/openid-connect-protocol), el protocolo estándar del sector que suele usarse con más frecuencia en aplicaciones orientadas al cliente, se trataría de un token de ID expresado como un [JWT](/docs/es/secure/tokens/json-web-tokens).

<div id="id-token-claims">
  ### Claims del token de ID
</div>

Con la extensibilidad de Actions, Auth0 te permite [agregar claims personalizadas a un token de ID](/docs/es/get-started/apis/scopes/sample-use-cases-scopes-and-claims#add-custom-claims-to-a-token) fácilmente en función de, por ejemplo, el contenido de la [metadata](/docs/es/manage-users/user-accounts/metadata) de un usuario. Luego, tu aplicación puede verificar que el token de ID contenga las claims necesarias y, según corresponda, permitir o impedir el acceso a determinadas funcionalidades. Ten en cuenta que, aunque el proceso para agregar claims personalizadas mediante Actions está simplificado, Actions ofrece flexibilidad y te permite escribir código personalizado que podría tener efectos negativos.

<Info>
  ### Práctica recomendada

  Si estás pensando en agregar claims personalizadas, te recomendamos almacenar en el [`app_metadata`](/docs/es/manage-users/user-accounts/metadata) del usuario cualquier dato de control de acceso que necesites incluir en las claims. En primer lugar, esto evita tener que llamar a una API externa para obtener los datos, lo que puede afectar negativamente al rendimiento y la escalabilidad de la secuencia de inicio de sesión. En segundo lugar, `app_metadata` **no puede** ser modificado por un usuario, por lo que este no puede eludir directamente ninguna restricción de control de acceso modificando su propia metadata. Recuerda también consultar nuestra guía de [prácticas recomendadas sobre metadata](/docs/es/get-started/architecture-scenarios/business-to-business/profile-management#metadata).
</Info>

Si estás creando distintas instancias de tu aplicación para las organizaciones de tus clientes, una práctica habitual es crear una claim personalizada en tu token de ID para representar la organización del usuario. Por ejemplo:

```js wrap lines
context.idToken["http://yourdomain.com/claims/organization"] = "organization A";
```

<div id="id-token-scopes">
  ### Scopes del token de ID
</div>

Los [OIDC Scopes](/docs/es/get-started/apis/scopes/openid-connect-scopes) suelen usarse para que una aplicación obtenga el consentimiento necesario para acceder de forma autorizada a la información del usuario durante la autenticación. Cada uno de los scopes predefinidos devuelve el conjunto de claims estándar cuando corresponde, tal como se describe en la [especificación de OIDC](https://openid.net/specs/openid-connect-core-1_0.html#StandardClaims). Los scopes que solicita una aplicación dependen de los atributos del usuario que necesite la aplicación. Una vez que el usuario autoriza los scopes solicitados, los claims se devuelven en el token de ID y también quedan disponibles a través del endpoint [/userinfo](https://auth0.com/docs/api/authentication/user-profile/get-user-info).

<div id="api-integration">
  ## Integración de API
</div>

En este escenario, tu tenant de Auth0 puede proporcionar un OAuth2 [token de acceso](/docs/es/secure/tokens/access-tokens), normalmente expresado como un [JWT](/docs/es/secure/tokens/json-web-tokens), que tu API puede usar para restringir el acceso a determinadas partes. Además, Auth0 ofrece soporte para lo que se describe conceptualmente como [aplicaciones propias y de terceros](/docs/es/get-started/applications/first-party-and-third-party-applications).

Actuando como servidor de autorización y con el consentimiento del usuario (el propietario del recurso), tu tenant de Auth0 puede utilizarse para proporcionar un token de acceso —normalmente expresado como un [JWT](/docs/es/secure/tokens/json-web-tokens)— a una aplicación (cliente) para que pueda acceder a recursos protegidos alojados por un <Tooltip tip="Resource Server: Servidor que aloja recursos protegidos. Los servidores de recursos aceptan y responden a solicitudes de recursos protegidos." cta="Ver glosario" href="/docs/es/glossary?term=resource+server">servidor de recursos</Tooltip> en nombre del propietario del recurso. El token de acceso emitido normalmente se envía como token Bearer en el encabezado HTTP Authorization de una solicitud a una API.

Tanto si tienes una sola API como un conjunto de [API de microservicios](/docs/es/get-started/apis/set-logical-api) relacionadas lógicamente, puedes aprovechar los tokens de acceso que proporciona Auth0 para proteger el acceso a tus servicios. Aunque es relativamente fácil configurar esto en el [Auth0 Dashboard](https://manage.auth0.com/) o mediante la [Auth0 Management API](https://auth0.com/docs/api/management/v2#!/Resource_Servers/post_resource_servers), es importante revisar los distintos escenarios de aplicaciones y las distintas arquitecturas de API para determinar la mejor arquitectura para tu sistema.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Los tokens de acceso de OAuth2 están diseñados principalmente para proteger API expuestas públicamente; cuando se expresan como JWT, un token de acceso es una entidad autocontenida que puede verificarse sin necesidad de realizar llamadas adicionales a API de terceros. Si tus API no entran en esta categoría —es decir, si forman parte de una aplicación en sí (por ejemplo, si solo esa aplicación las invoca) o están detrás de tu firewall—, entonces protegerlas con tokens puede ser excesivo, y tu flujo de trabajo actual basado en cookies (entre otros) puede ser suficiente.
</Callout>

OAuth2 se diseñó específicamente pensando en el acceso de terceros. Por ejemplo, un escenario posible es que un usuario (propietario del recurso) quiera usar una aplicación (un cliente) que no pertenece a la misma organización que el servicio que proporciona los datos del usuario (el servidor de recursos). En este caso, cuando la aplicación necesita acceder a datos que pertenecen al usuario, redirige a la organización donde residen esos datos, que a su vez autentica al usuario y luego le solicita permiso para que la aplicación pueda acceder a ellos. A esta solicitud de permiso se le denomina [consentimiento](/docs/es/get-started/applications/third-party-applications/user-consent-and-third-party-applications) y constituye una parte importante de lo que implica ofrecer soporte para [aplicaciones de terceros](/docs/es/get-started/apis/scopes/api-scopes#example-an-api-called-by-a-third-party-application). Si planeas integrar aplicaciones de terceros, es importante que las marques como de terceros desde el principio para que Auth0 gestione la solicitud de consentimiento del usuario.

Por otro lado, si tu organización posee la(s) aplicación(es), los propios datos del usuario y la(s) API a través de las cuales se accede a esos datos, normalmente no se requiere consentimiento, ya que todas las interacciones son [propias](/docs/es/get-started/apis/scopes/api-scopes#example-an-api-called-by-a-first-party-application). Si solo estás creando aplicaciones propias, puedes asegurarte de no mostrar a tus usuarios pantallas de consentimiento innecesarias [permitiendo omitir el consentimiento del usuario](/docs/es/get-started/apis) como parte de cualquier definición del servidor de recursos.

<Warning>
  Aunque puedes configurar tus aplicaciones como propias y posteriormente configurar tus API para permitir que los clientes propios ignoren el consentimiento, si estás usando `localhost` Auth0 no puede verificar que la aplicación sea realmente una aplicación propia, por lo que tus usuarios recibirán igualmente la solicitud de consentimiento. Para evitar esta limitación, al hacer pruebas en tu máquina local durante el desarrollo, crea un [nombre de host local falso y úsalo en su lugar](https://community.auth0.com/t/how-do-i-skip-the-consent-page-for-my-api-authorization-flow/6035).
</Warning>

Como alternativa, es posible que tenga datos relacionados con un usuario para los que se ofrece [funcionalidad adicional](/docs/es/get-started/apis/scopes/api-scopes#example-an-api-called-by-a-back-end-service) y para los que no se puede obtener el consentimiento explícito del usuario (es decir, no hay ningún usuario autenticado que pueda proporcionarlo). En este caso, se puede definir una [lista de aplicaciones para las que está habilitada la Concesión de credenciales de cliente](/docs/es/get-started/authentication-and-authorization-flow/client-credentials-flow).

<div id="access-token-claims">
  ### Claims del token de acceso
</div>

Al igual que ocurre con los tokens de ID, puedes [agregar claims personalizados a los tokens de acceso](/docs/es/get-started/apis/scopes/sample-use-cases-scopes-and-claims#add-custom-claims-to-a-token) mediante la extensibilidad de Auth0 Actions. Una vez añadidos, tu API puede verificar que un token de acceso contenga los claims necesarios y permitir o denegar el acceso a determinadas funcionalidades, según corresponda.

<Info>
  ### Práctica recomendada

  Si estás pensando en agregar claims personalizados, te recomendamos almacenar en el [`app_metadata`](/docs/es/manage-users/user-accounts/metadata) del usuario cualquier dato de control de acceso que necesites incluir en los claims. En primer lugar, así evitas tener que llamar a una API externa para obtener esos datos, lo que puede afectar negativamente al rendimiento y la escalabilidad. En segundo lugar, el usuario **no puede** modificar `app_metadata`, por lo que no puede eludir directamente ninguna restricción de control de acceso modificando sus propios metadatos. Recuerda también consultar nuestra guía sobre [prácticas recomendadas para metadatos](/docs/es/get-started/architecture-scenarios/business-to-business/profile-management#metadata).
</Info>

<div id="access-token-scopes">
  ### scopes del token de acceso
</div>

Los [OAuth2 Scopes](/docs/es/get-started/apis/scopes/api-scopes) suelen utilizarse como el mecanismo mediante el cual una API puede determinar qué acciones pueden realizarse en nombre de un usuario. Los scopes pueden agregarse por API para [definir permisos de acceso específicos](/docs/es/get-started/apis/add-api-permissions) en el <Tooltip tip="Auth0 Dashboard: el producto principal de Auth0 para configurar tus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip> o mediante la Auth0 <Tooltip tip="Auth0 Dashboard: el producto principal de Auth0 para configurar tus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip>). Los scopes también pueden manipularse mediante la extensibilidad de Auth0 (por ejemplo, con una Action, como en este [ejemplo](/docs/es/get-started/architecture-scenarios/spa-api/part-2#create-a-rule-to-validate-token-scopes)). Los scopes que una aplicación solicita para acceder a una API deben depender de la funcionalidad para la que la aplicación necesita el permiso del usuario. Una vez autorizados los scopes solicitados, se devolverán en el token de acceso, que posteriormente podrá [ser verificado por dicha API](/docs/es/secure/tokens/access-tokens/validate-access-tokens). Un buen ejemplo de esto es cuando inicias sesión en una aplicación que utiliza un proveedor social para el login: la API del proveedor social exige que la aplicación especifique si el usuario quiere que la aplicación publique contenido en su nombre. Esto permite al usuario aceptar o denegar esta solicitud. Este ejemplo muestra cómo el usuario está delegando un permiso a la aplicación, lo cual es distinto de que la API restrinja el acceso según el rol de un usuario, y debe gestionarse de otra manera.

<Info>
  ### Práctica recomendada

  Aunque tienes la capacidad de manipular por completo los scopes del token de acceso mediante la extensibilidad de Auth0, como práctica recomendada de seguridad solo deberías eliminar los scopes que no estén autorizados y evitar agregar scopes que no se hayan solicitado.
</Info>

Aunque los scopes suelen usarse como una forma de aplicar permisos de acceso a un usuario, hay situaciones en las que puede resultar [complicado usarlos de esta manera](https://auth0.com/blog/on-the-nature-of-oauth2-scopes/). Por lo tanto, te recomendamos que uses los scopes para su propósito previsto (es decir, delegar permisos a una aplicación) y que uses [claims personalizadas](#access-token-claims) para tus escenarios basados en roles u otros casos de control de acceso.

<div id="fine-grained-authorization-fga">
  ## Fine Grained Authorization (FGA)
</div>

La [Fine Grained Authorization](https://docs.fga.dev/fga-concepts) te permite conceder a usuarios individuales acceso a un recurso u objeto específico en función de:

* El rol de un usuario dentro de una organización, como `editor` o `admin`
* Un atributo del usuario o del objeto, como `manager` para un usuario o `marketing` para un objeto
* Una relación entre un usuario y un objeto, por ejemplo, que un usuario con acceso de visualización a una carpeta principal también tenga acceso de visualización a la carpeta secundaria

Con <Tooltip tip="Fine-grained Authorization (FGA): producto de Auth0 que permite a usuarios individuales acceder a objetos o recursos específicos." cta="Ver glosario" href="/docs/es/glossary?term=FGA">FGA</Tooltip>, puedes crear un modelo de autorización para definir qué relaciones determinan el acceso del usuario.

<div id="role-based-access-control-rbac">
  ## Control de acceso basado en roles (RBAC)
</div>

Auth0 ofrece soporte integrado para el control de acceso basado en roles ([RBAC](/docs/es/manage-users/access-control/rbac)). RBAC consiste en asignar permisos a los usuarios según su rol dentro de una organización y proporciona un control de acceso más sencillo al ofrecer un enfoque más fácil de gestionar y menos propenso a errores.

La funcionalidad principal de RBAC puede utilizarse en muchos escenarios con múltiples organizaciones. Consulta [Datos de la organización en un token de acceso](#organization-data-in-an-access-token) para obtener más información sobre cómo asegurarte de que tu configuración pueda satisfacer tus necesidades de RBAC.

<div id="machine-to-machine-m2m-authorization">
  ## Autorización de máquina a máquina (M2M)
</div>

Hay muchos escenarios en los que una aplicación sin ninguna sesión interactiva de usuario necesita obtener un token de acceso para llamar a una API. En esos casos, debes autenticar al cliente en lugar del usuario, y <Tooltip tip="OAuth 2.0: Marco de autorización que define protocolos y flujos de autorización." cta="Ver glosario" href="/docs/es/glossary?term=OAuth">OAuth</Tooltip> 2 proporciona el tipo de concesión de [credenciales de cliente](/docs/es/get-started/authentication-and-authorization-flow/client-credentials-flow) para que esto sea fácil de conseguir. Algunos ejemplos habituales de cuándo se necesita esto son:

* Una tarea cron u otro servicio que necesite comunicarse con tu API (p. ej., cuando hay que generar un informe diario y enviárselo por correo electrónico a un administrador).
* Una API independiente que admite acceso privilegiado (p. ej., la API no se expone directamente a los usuarios, sino solo al backend).
* En determinadas arquitecturas de microservicios, en las que algunas capas de API necesitan comunicarse con otras capas de API sin intervención de un usuario, o después de que el token de un usuario haya expirado.
* Una API privilegiada que puede necesitar invocarse antes de que un usuario se haya autenticado (es decir, desde una Action o un script de base de datos personalizado en tu tenant de Auth0)

<Info>
  ### práctica recomendada

  Tradicionalmente, se creaba una &quot;cuenta de servicio&quot; especial para estos escenarios: un usuario con nombre de usuario y contraseña configurado para servicios que admitían casos de uso no interactivos. Ese enfoque ya no se recomienda por muchas razones, y la práctica recomendada actual es usar [OAuth 2.0 Client Credentials Grant](/docs/es/get-started/authentication-and-authorization-flow/client-credentials-flow) en estas situaciones.
</Info>

<div id="organization-data-in-an-access-token">
  ## Datos de la organización en un token de acceso
</div>

Si en tu sistema tienes una API independiente de tu aplicación que da soporte a una aplicación con varias organizaciones, es importante restringir las operaciones únicamente a la organización para la que se generó el token. Para ello, el token de acceso debe incluir algún tipo de información que le indique a la API para qué organización se emitió. Esto puede hacerse de un par de formas distintas, según las respuestas a unas preguntas sencillas:

1. ¿Los usuarios finales de esta organización podrían pertenecer a más de una organización, o cada usuario final está limitado a una organización específica?
2. ¿Vas a permitir algún acceso de máquina a máquina (M2M) a tu API?
3. Si permites acceso de máquina a máquina (M2M) a tu API, ¿tendrás desarrolladores que necesiten un único ID de cliente y secreto para acceder a varias organizaciones, pero no a todas?
4. ¿Vas a permitir la creación de aplicaciones de terceros que requieran consentimiento?

Si los usuarios finales están limitados a una sola organización **y** no vas a permitir acceso de máquina a máquina a tu API, o bien tendrás un <Tooltip tip="ID de cliente: Valor de identificación dado a tu recurso registrado por Auth0." cta="Ver glosario" href="/docs/es/glossary?term=client+ID">ID de cliente</Tooltip>/secreto independiente para cada organización que necesite acceso, **y** no vas a permitir aplicaciones de terceros que requieran consentimiento, entonces el enfoque más sencillo es crear un custom claim en el token de acceso [usando Actions para los tokens basados en usuarios](#access-token-claims) y [usando el hook de credenciales de cliente para llamadas M2M](#machine-to-machine-m2m-authorization). Puedes almacenar el nombre de la organización en los client metadata y extraerlo desde Actions o hooks para incluirlo en access&#95;token como un custom claim. RBAC también funcionará de forma inmediata con este enfoque, siempre que cada usuario final solo pueda pertenecer a una organización.

Si los usuarios finales pueden pertenecer a más de una organización o podrías dar a un mismo desarrollador un ID de cliente y secreto para llamadas M2M a más de una organización, lo mejor es crear un <Tooltip tip="Audience: Identificador único del audience para un token emitido. Se llama aud en un token; su valor contiene el ID de una aplicación (ID de cliente) para un token de ID o de una API (identificador de API) para un Access Token." cta="Ver glosario" href="/docs/es/glossary?term=audience">audience</Tooltip> independiente (una instancia de API independiente en tu tenant de Auth0) para cada organización. Esto te da varias ventajas importantes:

1. En primer lugar, te permite pasar el audience a Auth0 como un parámetro de primer nivel sin tener que crear un parámetro personalizado. La ventaja es que Auth0 ayudará a garantizar la existencia del audience y lo pasará a tus Actions. También se asegurará de que un token de actualización emitido solo funcione para el audience específico para el que se emitió originalmente.
2. Te permite restringir las autorizaciones de cliente únicamente a organizaciones específicas de forma nativa. La alternativa es crear un hook de credenciales de cliente más complejo para intentar recuperar las restricciones desde otro lugar y, además, requerir una forma mucho más compleja y potencialmente problemática de indicar en la llamada de credenciales de cliente para qué organización debe emitirse el token de acceso.
3. Esto también te permite usar la funcionalidad principal de RBAC con Auth0 y garantizar que los usuarios finales que tienen acceso a más de una organización puedan tener un rol distinto para cada organización.
