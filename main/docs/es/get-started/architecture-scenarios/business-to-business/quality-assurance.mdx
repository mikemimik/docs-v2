---

El control de calidad es importante para identificar problemas antes de que afecten a sus clientes y, según la naturaleza de su proyecto, hay varios tipos de pruebas de calidad que le conviene considerar como parte de su integración con Auth0:

* ¿Su aplicación es fácil de entender y usar, incluso para personas con alguna discapacidad?
* ¿Su aplicación necesita funcionar en distintos navegadores y dispositivos?
* ¿Su aplicación necesita funcionar en entornos multinacionales o internacionales?
* ¿Cómo responderá su aplicación cuando se vea sometida a cargas inesperadas en producción?
* ¿Cómo puede asegurarse de que su aplicación esté protegida frente a vulnerabilidades de seguridad?

Auth0 [Universal Login](/docs/es/authenticate/login/auth0-universal-login) y los widgets de interfaz de usuario asociados (como [Lock](/docs/es/libraries/lock)) ya se han diseñado y desarrollado siguiendo prácticas recomendadas de usabilidad y accesibilidad, y ofrecen compatibilidad probada y lista para usar con una amplia variedad de [navegadores y dispositivos](/docs/es/troubleshoot/customer-support/product-support-matrix). También se ofrece soporte listo para usar para la [internacionalización](/docs/es/customize/internationalization-and-localization) (I18N), con extensibilidad integrada diseñada para situaciones personalizadas de múltiples idiomas y localización (L10N).

Para garantizar que se cumplan los requisitos funcionales y que los eventos inesperados se gestionen correctamente, se ofrece orientación para probar la [integración](#integration-testing) entre sus aplicaciones y Auth0, así como para realizar [pruebas unitarias](#unit-testing) de módulos individuales de extensibilidad (como [Rules](/docs/es/customize/rules/debug-rules), [Hooks](/docs/es/customize/hooks/update-hooks) y scripts de base de datos personalizada). También se ofrece orientación sobre la [política de pruebas de penetración](/docs/es/troubleshoot/customer-support/operational-policies/penetration-testing-policy) de Auth0 para ayudarle a detectar vulnerabilidades de seguridad, así como sobre cómo aprovechar las pruebas de [Mock](#mock-testing) junto con nuestra [política de pruebas de carga](/docs/es/troubleshoot/customer-support/operational-policies/load-testing-policy) para ayudar a garantizar que sus aplicaciones rindan correctamente ante cargas inesperadas.

<div id="unit-testing">
  ## Pruebas unitarias
</div>

El objetivo de las pruebas unitarias es probar unidades individuales de código. Si crea código personalizado en Auth0 en forma de Rules, Hooks y/o scripts de Custom DB, debería considerar usar un framework de pruebas (como [Mocha](https://mochajs.org/)) para probar su código. Las empresas que más éxito han tenido con Auth0 han comprobado que resulta útil ejecutar estas pruebas unitarias antes de [desplegar automáticamente](/docs/es/get-started/architecture-scenarios/business-to-business/deployment) la configuración y los recursos asociados del tenant de Auth0.

<div id="integration-testing">
  ## Pruebas de integración
</div>

Una práctica recomendada es configurar tenants distintos para desarrollo, pruebas y producción, como se explica en la guía de Architecture sobre [soporte para SDLC](/docs/es/get-started/architecture-scenarios/business-to-business/architecture#sdlc-support). Auth0 le permite configurar variables disponibles desde la [extensibilidad](/docs/es/customize/extensions) personalizada; puede considerarlas variables de entorno para su tenant de Auth0. En lugar de codificar de forma rígida referencias que cambian al mover código entre entornos de desarrollo, pruebas y producción, puede usar un nombre de variable configurado en el tenant y al que hace referencia el código de extensibilidad personalizada. Esto facilita que el mismo código personalizado funcione, sin cambios, en distintos tenants, ya que puede hacer referencia a variables que se completarán con valores específicos del tenant en el momento de la ejecución:

* Para usar variables en Rules, vea cómo [configurar valores](/docs/es/customize/rules/configuration)
* Para usar variables en Hooks, vea cómo configurar [secretos](/docs/es/customize/hooks/hook-secrets) en el editor
* Para usar variables en Actions, vea Explore Flows and Triggers
* Para usar variables en scripts de Custom DB, consulte los [parámetros de configuración](/docs/es/authenticate/database-connections/custom-db/create-db-connection#step-3-add-configuration-parameters)

<Info>
  ### Práctica recomendada

  Es una práctica recomendada usar variables para almacenar valores específicos del tenant, así como cualquier secreto sensible que no deba exponerse en su código personalizado. Si su código personalizado está desplegado en GitHub, usar una variable específica del tenant evita exponer valores sensibles a través de su repositorio de GitHub.
</Info>

<div id="test-automation">
  ### Automatización de pruebas
</div>

Puedes automatizar tu proceso general de compilación incorporando la automatización del despliegue, así como la automatización de pruebas. Esto te permite desplegar nuevas versiones de la configuración y/o del código personalizado en Auth0 y ejecutar pruebas automatizadas. Si las pruebas detectan algún fallo, las funciones de automatización del despliegue pueden usarse para revertir a la última versión funcional. Para obtener más información, consulta la [guía de automatización del despliegue](/docs/es/get-started/architecture-scenarios/business-to-business/deployment).

<div id="mock-testing">
  ## Prueba simulada
</div>

Como equilibrio entre la [política de pruebas de carga](/docs/es/troubleshoot/customer-support/operational-policies/load-testing-policy) de Auth0 y la necesidad de realizar una prueba de carga, es habitual entre los clientes de Auth0 simular los endpoints de Auth0. Esta práctica es valiosa para garantizar que su aplicación funcione con las interfaces previstas sin tener que limitar sus pruebas, y puede apoyarse en herramientas como [MockServer](http://www.mock-server.com/), [JSON Server](https://github.com/typicode/json-server) o incluso [Postman](https://learning.getpostman.com/docs/postman/mock_servers/setting_up_mock/).

<div id="multiple-organization-architecture-multitenancy">
  ## Arquitectura de múltiples organizaciones (multitenencia)
</div>

Muchas plataformas B2B implementan algún tipo de aislamiento y/o branding para las organizaciones de sus clientes, y esto puede añadir complejidad a cualquier sistema de gestión de identidades y accesos (IAM). Si este es su caso, le recomendamos que dedique algo de tiempo a leer nuestra guía y nuestras recomendaciones de prácticas recomendadas para este tipo de entorno.
