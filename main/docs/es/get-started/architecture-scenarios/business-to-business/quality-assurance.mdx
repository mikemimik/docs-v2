---

Quality Assurance es importante para identificar problemas antes de que afecten a sus clientes y, según la naturaleza de su proyecto, hay varios tipos de pruebas de aseguramiento de la calidad que conviene considerar como parte de su integración con Auth0:

* ¿Su aplicación es fácil de entender y usar, incluso para personas con discapacidad?
* ¿Su aplicación debe funcionar en distintos navegadores y dispositivos?
* ¿Su aplicación debe funcionar en entornos multinacionales y/o internacionales?
* ¿Cómo responderá su aplicación cuando se vea sometida a cargas de Production inesperadas?
* ¿Cómo puede asegurarse de que su aplicación esté protegida frente a vulnerabilidades de seguridad?

Auth0 [Universal Login](/docs/es/authenticate/login/auth0-universal-login) y los widget de interfaz de usuario asociados (como [Lock](/docs/es/libraries/lock)) ya han sido diseñados y desarrollados siguiendo prácticas recomendadas de usabilidad y accesibilidad, y ofrecen soporte probado listo para usar para una amplia variedad de [navegadores y dispositivos](/docs/es/troubleshoot/customer-support/product-support-matrix). También se ofrece compatibilidad inmediata con la [internacionalización](/docs/es/customize/internationalization-and-localization) (I18N), con extensibilidad integrada diseñada para casos personalizados de varios idiomas y localización (L10N).

Para garantizar que se cumplan los requisitos funcionales y que los eventos inesperados se gestionen correctamente, se ofrece orientación para probar la [integración](#integration-testing) entre su(s) aplicación(es) y Auth0, así como para las [pruebas unitarias](#unit-testing) de módulos individuales de extensibilidad (como [Rules](/docs/es/customize/rules/debug-rules), [Hooks](/docs/es/customize/hooks/update-hooks) y scripts de Custom Database). También se ofrece orientación sobre la [política de pruebas de penetración](/docs/es/troubleshoot/customer-support/operational-policies/penetration-testing-policy) de Auth0 para ayudarle al realizar pruebas de vulnerabilidades de seguridad, así como sobre cómo pueden aprovecharse las pruebas [simuladas](#mock-testing) junto con nuestra [política de pruebas de carga](/docs/es/troubleshoot/customer-support/operational-policies/load-testing-policy) para contribuir a garantizar que su(s) aplicación(es) rindan bien ante cargas inesperadas.

<div id="unit-testing">
  ## Pruebas unitarias
</div>

El objetivo de las pruebas unitarias es probar unidades individuales de código. Si crea código personalizado dentro de Auth0 en forma de Rules, Hooks y/o Custom DB scripts, debería considerar usar un framework de pruebas (como [Mocha](https://mochajs.org/)) para probar su código. Las empresas que han tenido más éxito con Auth0 han comprobado que resulta útil ejecutar estas pruebas unitarias antes de [desplegar automáticamente](/docs/es/get-started/architecture-scenarios/business-to-business/deployment) la configuración y los recursos asociados del tenant de Auth0.

<div id="integration-testing">
  ## Pruebas de integración
</div>

Como práctica recomendada, configure distintos tenants para desarrollo, pruebas y producción, como se indica en la guía de arquitectura sobre [soporte para SDLC](/docs/es/get-started/architecture-scenarios/business-to-business/architecture#sdlc-support). Auth0 le permite configurar variables disponibles desde la [extensibilidad](/docs/es/customize/extensions) personalizada; pueden considerarse variables de entorno para su tenant de Auth0. En lugar de incluir directamente en el código referencias que cambian al moverlo entre entornos de desarrollo, pruebas y producción, puede usar un nombre de variable configurado en el tenant al que haga referencia el código de extensibilidad personalizada. Esto facilita que el mismo código personalizado funcione, sin cambios, en distintos tenants, ya que puede hacer referencia a variables que se rellenarán con valores específicos del tenant en tiempo de ejecución:

* Para usar variables en Rules, consulte cómo [configurar valores](/docs/es/customize/rules/configuration)
* Para usar variables en Hooks, vea cómo configurar [secrets](/docs/es/customize/hooks/hook-secrets) en el editor
* Para usar variables en Actions, consulte Explore Flows and Triggers
* Para usar variables en Custom DB Scripts, consulte los [parámetros de configuración](/docs/es/authenticate/database-connections/custom-db/create-db-connection#step-3-add-configuration-parameters)

<Info>
  ### Práctica recomendada

  Es una práctica recomendada usar variables para almacenar valores específicos del tenant, así como cualquier secreto sensible que no deba exponerse en su código personalizado. Si su código personalizado está publicado en GitHub, usar una variable específica del tenant evita exponer valores sensibles a través de su repositorio de GitHub.
</Info>

<div id="test-automation">
  ### Automatización de pruebas
</div>

Puedes automatizar tu proceso general de compilación incorporando automatización de despliegues y de pruebas. Esto te permite desplegar nuevas versiones de la configuración o del código personalizado en Auth0 y ejecutar pruebas automatizadas. Si las pruebas detectan algún fallo, las capacidades de automatización de despliegues pueden usarse para revertir a la última versión que funcionaba correctamente. Para obtener más información, consulta la [guía sobre automatización de despliegues](/docs/es/get-started/architecture-scenarios/business-to-business/deployment).

<div id="mock-testing">
  ## Prueba simulada
</div>

Para equilibrar la [política de pruebas de carga](/docs/es/troubleshoot/customer-support/operational-policies/load-testing-policy) de Auth0 con la necesidad de realizar pruebas de carga, es habitual que los clientes de Auth0 simulen los endpoints de Auth0. Esta es una práctica valiosa para garantizar que su aplicación funcione con las interfaces previstas sin tener que limitar sus pruebas, y pueden utilizarse herramientas como [MockServer](http://www.mock-server.com/), [JSON Server](https://github.com/typicode/json-server) o incluso [Postman](https://learning.getpostman.com/docs/postman/mock_servers/setting_up_mock/) para ello.

<div id="multiple-organization-architecture-multitenancy">
  ## Arquitectura de múltiples organizaciones (multitenencia)
</div>

Muchas plataformas B2B implementan algún tipo de aislamiento y/o branding para las organizaciones de sus clientes, y esto puede añadir complejidad a cualquier sistema de gestión de identidades y accesos (IAM). Si este es tu caso, te recomendamos dedicar algo de tiempo a leer nuestras guías y prácticas recomendadas para este tipo de entorno.
