---

La garantía de calidad es importante para identificar problemas antes de que afecten a sus clientes y, según la naturaleza de su proyecto, hay varios tipos de pruebas de garantía de calidad que conviene tener en cuenta como parte de su integración con Auth0:

* ¿Su aplicación es fácil de entender y usar, incluso para personas con alguna discapacidad?
* ¿Su aplicación necesita funcionar en distintos navegadores y dispositivos?
* ¿Su aplicación necesita funcionar en entornos multinacionales y/o internacionales?
* ¿Cómo se comportará su aplicación cuando se vea sometida a cargas de producción inesperadas?
* ¿Cómo puede asegurarse de que su aplicación esté protegida frente a vulnerabilidades de seguridad?

Auth0 [Universal Login](/docs/es/authenticate/login/auth0-universal-login) y los widgets de interfaz de usuario asociados (como [Lock](/docs/es/libraries/lock)) ya se han diseñado y desarrollado siguiendo las mejores prácticas de usabilidad y accesibilidad, y ofrecen soporte probado y listo para usar para una amplia variedad de [navegadores y dispositivos](/docs/es/troubleshoot/customer-support/product-support-matrix). También se ofrece compatibilidad inmediata con la [internacionalización](/docs/es/customize/internationalization-and-localization) (I18N), con extensibilidad integrada diseñada para casos personalizados de varios idiomas y localización (L10N).

Para garantizar que se cumplan los requisitos funcionales y que los eventos inesperados se gestionen correctamente, se ofrece orientación para probar la [integración](#integration-testing) entre su(s) aplicación(es) y Auth0, así como para las [pruebas unitarias](#unit-testing) de módulos de extensibilidad individuales (como [Rules](/docs/es/customize/rules/debug-rules), [Hooks](/docs/es/customize/hooks/update-hooks) y script de Custom DB). También se ofrece orientación sobre la [política de pruebas de penetración](/docs/es/troubleshoot/customer-support/operational-policies/penetration-testing-policy) de Auth0 para ayudarle a evaluar vulnerabilidades de seguridad, así como sobre cómo pueden aprovecharse las pruebas con [prueba simulada](#mock-testing) junto con nuestra [política de pruebas de carga](/docs/es/troubleshoot/customer-support/operational-policies/load-testing-policy) para ayudar a garantizar que su(s) aplicación(es) funcionen correctamente ante cargas inesperadas.

<div id="unit-testing">
  ## Pruebas unitarias
</div>

El objetivo de las pruebas unitarias es comprobar unidades individuales de código. Si crea código personalizado dentro de Auth0 en forma de Rules, Hooks y/o scripts de Custom DB, debería considerar usar un framework de pruebas (como [Mocha](https://mochajs.org/)) para probar su código. Las empresas que han tenido más éxito con Auth0 han comprobado que resulta útil ejecutar estas pruebas unitarias antes de [implementar automáticamente](/docs/es/get-started/architecture-scenarios/business-to-business/deployment) la configuración del tenant de Auth0 y los recursos relacionados.

<div id="integration-testing">
  ## Pruebas de integración
</div>

Una práctica recomendada es configurar distintos tenants para desarrollo, pruebas y producción, como se explica en la guía de arquitectura sobre [compatibilidad con SDLC](/docs/es/get-started/architecture-scenarios/business-to-business/architecture#sdlc-support). Auth0 le permite configurar variables disponibles desde la [extensibilidad](/docs/es/customize/extensions) personalizada; pueden entenderse como variables de entorno para su tenant de Auth0. En lugar de codificar de forma rígida referencias que cambian al mover código entre los entornos de desarrollo, pruebas y producción, puede usar un nombre de variable configurado en el tenant y al que haga referencia el código de extensibilidad personalizado. Esto facilita que el mismo código personalizado funcione, sin cambios, en distintos tenants, ya que puede hacer referencia a variables que se completarán con valores específicos del tenant en el momento de la ejecución:

* Para usar variables en Rules, vea cómo [configurar valores](/docs/es/customize/rules/configuration)
* Para usar variables en Hooks, vea cómo configurar [secretos](/docs/es/customize/hooks/hook-secrets) en el editor
* Para usar variables en Actions, consulte Explore Flows and Triggers
* Para usar variables en script de Custom DB, vea los [parámetros de configuración](/docs/es/authenticate/database-connections/custom-db/create-db-connection#step-3-add-configuration-parameters)

<Info>
  ### Práctica recomendada

  Se recomienda usar variables para almacenar valores específicos del tenant, así como cualquier secreto sensible que no deba exponerse en el código personalizado. Si su código personalizado se implementa en GitHub, usar una variable específica del tenant evita exponer valores sensibles a través de su repositorio de GitHub.
</Info>

<div id="test-automation">
  ### Automatización de pruebas
</div>

Puedes automatizar tu proceso general de compilación incorporando tanto la automatización del despliegue como la de las pruebas. Esto puede usarse para desplegar nuevas versiones de la configuración y/o del código personalizado en Auth0 y ejecutar pruebas automatizadas. Si las pruebas detectan algún fallo, las capacidades de automatización del despliegue pueden utilizarse para volver a la última versión funcional. Para obtener más información, consulta la [guía sobre automatización del despliegue](/docs/es/get-started/architecture-scenarios/business-to-business/deployment).

<div id="mock-testing">
  ## Pruebas simuladas
</div>

Como punto intermedio entre la [política de pruebas de carga](/docs/es/troubleshoot/customer-support/operational-policies/load-testing-policy) de Auth0 y la necesidad de realizar pruebas de carga, es habitual que los clientes de Auth0 simulen los endpoints de Auth0. Esta es una práctica valiosa para garantizar que su aplicación funcione con las interfaces previstas sin tener que limitar sus pruebas, y puede apoyarse en herramientas como [MockServer](http://www.mock-server.com/), [JSON Server](https://github.com/typicode/json-server) o incluso [Postman](https://learning.getpostman.com/docs/postman/mock_servers/setting_up_mock/).

<div id="multiple-organization-architecture-multitenancy">
  ## Arquitectura de múltiples organizaciones (multitenencia)
</div>

Muchas plataformas B2B implementan algún tipo de aislamiento y/o personalización de marca para las organizaciones de sus clientes, y esto puede añadir complejidad a cualquier sistema de gestión de identidades y accesos (IAM). Si este es tu caso, te recomendamos que dediques un tiempo a leer nuestra guía y nuestras recomendaciones de buenas prácticas para este tipo de entorno.

[Arquitectura de múltiples organizaciones](/docs/es/get-started/architecture-scenarios/multiple-organization-architecture)
