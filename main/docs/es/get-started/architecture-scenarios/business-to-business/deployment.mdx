title: Automatización del despliegue (B2B)
---

Además de adoptar prácticas recomendadas para la gestión de cambios y [QA](/docs/es/get-started/architecture-scenarios/business-to-business/quality-assurance), los clientes con implementaciones exitosas también integrarán la gestión de recursos de Auth0 como parte de algún proceso de despliegue automatizado. Como se explica en la sección de Arquitectura, en [soporte para SDLC](/docs/es/get-started/architecture-scenarios/business-to-business/architecture#sdlc-support), le convendrá asegurarse de configurar tenants de Auth0 independientes para los entornos de desarrollo, pruebas y producción, y de que esa configuración sea casi idéntica en el tenant de cada entorno. El uso de la automatización del despliegue ayuda a garantizarlo, de modo que el tenant de cada entorno esté configurado de la misma forma y sea menos probable que aparezcan errores como resultado de configuraciones desalineadas entre entornos.

<Info>
  ### Práctica recomendada

  Independientemente de cómo configure la automatización del despliegue, le recomendamos realizar pruebas unitarias de sus Rules, scripts de Custom DB y Hooks antes del despliegue, y ejecutar también algunas pruebas de integración en su tenant después del despliegue. Para obtener más información al respecto, consulte la guía de [Quality Assurance](/docs/es/get-started/architecture-scenarios/business-to-business/quality-assurance).
</Info>

Auth0 ofrece soporte para un par de opciones diferentes en cuanto a los enfoques de automatización del despliegue que puede usar, y ambas pueden usarse conjuntamente si así lo desea:

* La [herramienta Deploy CLI de Auth0](/docs/es/deploy-monitor/deploy-cli-tool) le proporciona un script fácil de usar que puede ayudarle a integrarse con su pipeline actual de Integración Continua/Despliegue Continuo (CI/CD).
* Si no puede integrarse directamente con un pipeline de CI/CD, o si por alguna razón no tiene uno, las [Source Control Extensions](/docs/es/customize/extensions) de Auth0 pueden ofrecerle un proceso básico de automatización fácil de configurar y con muy poco mantenimiento.

<Warning>
  Tenga en cuenta que tanto la herramienta Deploy CLI como Source Control Extensions de Auth0 pueden provocar cambios destructivos; los cambios manuales realizados directamente en el Dashboard entre despliegues automatizados podrían perderse. Por este motivo, si se usa cualquiera de las dos, **todos** los cambios deben desplegarse desde el subsistema de control de código fuente al que hace referencia la herramienta y no realizarse manualmente.
</Warning>

Cada entorno también puede necesitar cierta configuración específica del entorno; por ejemplo, los <Tooltip tip="ID de cliente: valor de identificación dado por Auth0 a su recurso registrado." cta="Ver glosario" href="/docs/es/glossary?term=Client+ID">ID de cliente</Tooltip> y los <Tooltip tip="secreto de cliente: secreto utilizado por un cliente (aplicación) para autenticarse con el servidor de autorización; solo deberían conocerlo el cliente y el servidor de autorización, y debe ser lo suficientemente aleatorio como para que no se pueda adivinar." cta="Ver glosario" href="/docs/es/glossary?term=Client+Secrets">secretos de cliente</Tooltip> de la aplicación serán distintos entre los tenants de Auth0, por lo que le convendrá contar con alguna forma de hacer referencia a ellos dinámicamente en lugar de usar valores codificados de forma fija. Auth0 ofrece soporte para gestionar información de configuración específica del entorno mediante uno de los dos enfoques siguientes:

* Usar [variables específicas del tenant](#tenant-specific-variables)
* Usar reemplazo de palabras clave si utiliza la herramienta Deploy CLI de Auth0

<div id="tenant-specific-variables">
  ## Variables específicas del tenant
</div>

Auth0 le permite configurar variables disponibles desde la [extensibilidad](/docs/es/customize/extensions) personalizada; pueden considerarse variables de entorno para su tenant de Auth0. En lugar de incluir referencias fijas que cambian al mover código entre entornos de desarrollo, prueba y producción, puede usar un nombre de variable configurado en el tenant al que haga referencia el código de extensibilidad personalizado. Esto facilita que el mismo código personalizado funcione sin cambios en distintos tenants, ya que puede hacer referencia a variables que se rellenarán con valores específicos del tenant en tiempo de ejecución:

* Para usar variables en Actions, lea [Write Your First Action](/docs/es/customize/actions/write-your-first-action) para aprender a configurar secretos en el editor
* Para usar variables en Rules, vea cómo [configurar valores](/docs/es/customize/rules/configuration)
* Para usar variables en Hooks, vea cómo [configurar secretos](/docs/es/customize/hooks/hook-secrets) en el editor
* Para usar variables en scripts de Custom DB, consulte los [parámetros de configuración](/docs/es/authenticate/database-connections/custom-db/create-db-connection#step-3-add-configuration-parametersn#step-3-add-configuration-parameters)

<Info>
  ### Práctica recomendada

  Se recomienda usar variables para almacenar valores específicos del tenant, así como cualquier secreto sensible que no deba exponerse en su código personalizado. Si su código personalizado se despliega en GitHub/GitLab/Bitbucket/VSTS, usar una variable específica del tenant evita la exposición de valores sensibles a través de su repositorio.
</Info>

<div id="multiple-organization-architecture-multitenancy">
  ## Arquitectura de múltiples organizaciones (multitenencia)
</div>

Muchas plataformas B2B implementan algún tipo de aislamiento y/o branding para las organizaciones de sus clientes, y esto puede añadir complejidad a cualquier sistema de gestión de identidades y accesos (IAM). Si este es tu caso, te recomendamos dedicar algo de tiempo a leer nuestra guía y nuestras prácticas recomendadas para este tipo de entorno.

[Arquitectura de múltiples organizaciones](/docs/es/get-started/architecture-scenarios/multiple-organization-architecture)
