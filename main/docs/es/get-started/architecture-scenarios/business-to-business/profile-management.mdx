validatedOn: 2026-02-03
---

En algún momento, puede que necesite cambiar la información almacenada en el perfil de un usuario. El perfil de un usuario (también conocido como la cuenta del usuario) se almacena en Auth0, y puede ser necesario modificar la información que contiene por diversos motivos:

* Actualizaciones de información realizadas por el propio usuario
* Actualizaciones obligatorias relativas a los términos y condiciones de sus organizaciones
* Cambios debidos al cumplimiento normativo

<Warning>
  No puede acceder directamente al perfil de un usuario en varios tenants de Auth0. Si está implementando varios tenants de Auth0 en producción, esto es algo que debe tener en cuenta.
</Warning>

Un [proveedor de identidad](/docs/es/authenticate/identity-providers) completa el perfil de un usuario con los datos proporcionados durante el proceso de inicio de sesión, y a esto se le denomina [Perfil de usuario normalizado](/docs/es/manage-users/user-accounts/user-profiles/normalized-user-profiles).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  El Perfil de usuario normalizado se actualiza a partir del proveedor de identidad durante el inicio de sesión, y puede cambiar el conjunto limitado de información que contiene mediante la Auth0 Management API. También puede usar la extensibilidad de Auth0, como [Actions](/docs/es/customize/actions), como alternativa para sobrescribir información del Perfil de usuario normalizado. Consulte [Modificación de datos del perfil de usuario](/docs/es/manage-users/user-accounts/user-profiles#user-profile-data-modification) para obtener más información.
</Callout>

De forma predeterminada, se crea un perfil de usuario para cada identidad de usuario, y hay varios aspectos que debe tener en cuenta:

* ¿Qué debe hacer si necesita almacenar información para ayudar a personalizar la experiencia de un usuario?
* ¿Qué ocurre si necesita almacenar información del usuario que no procede de un <Tooltip tip="Proveedor de identidad (IdP): Servicio que almacena y gestiona identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=identity+provider">proveedor de identidad</Tooltip>?
* ¿Por qué necesitaría almacenar información relacionada con el usuario que este no puede modificar?
* ¿Qué debe hacer si necesita almacenar información relacionada con el usuario que este no puede modificar?
* ¿Qué sucede si un usuario olvida su contraseña?
* ¿Qué debe hacer un usuario si quiere cambiar su contraseña?
* ¿Cómo permite que un administrador de una organización de terceros gestione a sus usuarios?

Auth0 permite almacenar metadatos en el perfil de un usuario, lo que permite recopilar información adicional, como la preferencia de idioma y/o accesibilidad, para mejorar la experiencia del usuario. Los metadatos pueden usarse para almacenar tanto información que un usuario puede cambiar como información que no puede cambiar; esto último le permite asociar, por ejemplo, un perfil de usuario con registros de sus sistemas existentes sin modificar la implementación actual.

Para los usuarios que olvidan sus contraseñas o que pueden cambiar su contraseña mediante algún mecanismo de autoservicio existente (o un mecanismo de autoservicio que haya previsto), puede aprovechar la funcionalidad de [Restablecimiento de contraseña](#password-reset) proporcionada por Auth0. Esto puede integrarse con su implementación existente y ya viene incorporado en cualquier widget de interfaz de usuario de Auth0 listo para usar, incluido [Universal Login](/docs/es/authenticate/login/auth0-universal-login).

También querrá asegurarse de estar trabajando en todo momento con una [cuenta de usuario verificada](#account-verification). Auth0 también proporciona mecanismos listos para usar para ello. También debe considerar el [cumplimiento normativo](/docs/es/secure/data-privacy-and-compliance), como el [RGPD](https://gdpr-info.eu/), que establece requisitos muy específicos para proteger la privacidad de los ciudadanos de la UE y evitar violaciones de la seguridad de los datos.

Aunque Auth0 actualmente no proporciona un portal centralizado de gestión de perfiles listo para usar, para fines de gestión de perfiles de autoservicio puede usar la Auth0 <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip> para crear el suyo propio o utilizar una interfaz de usuario ya creada. Consulte nuestra guía en la [Community de Auth0](https://community.auth0.com/t/how-to-allow-the-end-user-to-update-their-own-profile-information/6228), que describe el endpoint de la Management API. Todas las llamadas a la Management API requerirán el uso de un [token de acceso](/docs/es/secure/tokens/access-tokens).

<Warning>
  La gestión de perfiles de autoservicio puede plantear problemas tanto de seguridad como de privacidad de los datos. Por ejemplo, puede que quiera permitir que un usuario cambie su dirección de correo electrónico; sin embargo, hacerlo sin seguir las prácticas recomendadas de seguridad podría provocar que un usuario pierda el acceso a su cuenta, filtre información de identificación personal (PII) o, peor aún, abra una posible brecha de seguridad.
</Warning>

Como alternativa, puedes usar el <Tooltip tip="Auth0 Dashboard: el producto principal de Auth0 para configurar tus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip> para [gestionar aspectos del perfil de un usuario](/docs/es/manage-users/user-accounts/manage-users-using-the-dashboard). Gestionar el perfil de un usuario a través del Auth0 Dashboard es más bien una tarea administrativa y **no debe** usarse para la gestión del perfil por autoservicio en un entorno de producción. Sin embargo, la interfaz que proporciona el Dashboard puede ser muy útil durante el desarrollo, ya que ofrece una forma rápida y sencilla de modificar la información del perfil de un usuario.

Si necesitas ofrecer a tus clientes una forma de permitir que un administrador gestione sus propios usuarios cuando esas credenciales se almacenan en tu sistema, puedes crear algo por tu cuenta o usar una extensión de Auth0. Consulta [Admin Portal](#admin-portal) para obtener más información.

<div id="metadata">
  ## Metadatos
</div>

Además de la información del Perfil de usuario normalizado, se pueden almacenar [metadatos](/docs/es/manage-users/user-accounts/metadata) en un perfil de usuario de Auth0. Los metadatos ofrecen una forma de almacenar información que no proviene de un proveedor de identidad, o de guardar información que sustituye la que proporciona un proveedor de identidad.

<Info>
  ### Práctica recomendada

  El uso de Metadatos debe seguir las [prácticas recomendadas de almacenamiento de datos de usuario](/docs/es/secure/security-guidance/data-security/user-data-storage#metadata) de Auth0. El almacenamiento de metadatos no está pensado como un almacén de datos de uso general, por lo que, siempre que sea posible, debería seguir utilizando su propio sistema de almacenamiento externo. Además, el tamaño y la complejidad de los metadatos deben mantenerse al mínimo, y la Auth0 Management API establece directrices estrictas en lo que respecta a la actualización y/o eliminación de los metadatos asociados a un usuario.
</Info>

Puede administrar los metadatos mediante la Auth0 Management API y la Auth0 Authentication API. Como ocurre al gestionar el Perfil de usuario normalizado, las llamadas a la Management API para administrar Metadatos requieren el uso de un [token de acceso](https://auth0.com/docs/api/management/v2/tokens).

<Warning>
  Las llamadas a la Management API están sujetas a la [política de límite de frecuencia de Auth0](/docs/es/troubleshoot/customer-support/operational-policies/rate-limit-policy). Debe tenerlo en cuenta y, para facilitar esta tarea, Auth0 suele recomendar el uso del [SDK de Auth0](/docs/es/libraries) adecuado para su entorno de desarrollo, en lugar de llamar directamente a nuestras API.
</Warning>

<div id="user-metadata">
  ### Metadatos del usuario
</div>

Los metadatos del usuario (también denominados `user_metadata`) son información que puede almacenarse en el perfil de un usuario y que este puede leer y actualizar como parte de cualquier gestión de perfiles de autoservicio. Los metadatos de este tipo pueden incluir, por ejemplo, el tratamiento de un usuario o su idioma preferido, que podría usarse para [personalizar los correos electrónicos](/docs/es/customize/email/email-templates) enviados por Auth0.

<Info>
  ### Práctica recomendada

  Guarda en los metadatos toda la información que quieras usar para personalizar los correos electrónicos de Auth0 y, de preferencia, en `user_metadata` si el usuario puede modificarla, como la información utilizada para determinar el idioma de un correo electrónico.
</Info>

<div id="app-metadata">
  ### Metadatos de la aplicación
</div>

Los metadatos de la aplicación (también denominados `app_metadata`) son, por otro lado, información que puede almacenarse en un perfil de usuario, pero que **solo puede leerse o actualizarse con la debida autorización**; `app_metadata` no es directamente accesible para el usuario. Este tipo de metadatos podría ser, por ejemplo, un indicador de que el usuario aceptó la versión más reciente de los términos y condiciones, así como la fecha en que los aceptó.

<div id="password-reset">
  ## Restablecimiento de contraseña
</div>

Para los usuarios que olvidan sus contraseñas o que pueden cambiar su contraseña mediante algún mecanismo de autoservicio existente, Auth0 ofrece la funcionalidad de [Restablecimiento de contraseña](/docs/es/authenticate/database-connections/password-change). Puede integrarla con su implementación actual y ya viene incorporada en los widgets de interfaz de usuario de Auth0 listos para usar incluidos como parte de [Universal Login](/docs/es/authenticate/login/auth0-universal-login).

<Warning>
  El cambio y el restablecimiento de contraseña solo son compatibles con los tipos de [conexión de base de datos](/docs/es/authenticate/database-connections/password-change) de Auth0.
</Warning>

Auth0 <Tooltip tip="Universal Login: Su aplicación redirige a Universal Login, alojado en el servidor de autorización de Auth0, para verificar la identidad de un usuario." cta="Ver glosario" href="/docs/es/glossary?term=Universal+Login">Universal Login</Tooltip> ofrece soporte integrado de experiencia de usuario para el restablecimiento de contraseña mediante la funcionalidad de Auth0 Authentication API. Como alternativa, puede usar la [Auth0 Authentication API](/docs/es/authenticate/database-connections/password-change#use-the-authentication-api) a través de uno de los Auth0 SDKs adecuados para su entorno de desarrollo. Las plantillas de correo electrónico utilizadas durante el flujo de trabajo de restablecimiento de contraseña también pueden personalizarse por completo, tanto si utiliza los widgets de interfaz de usuario listos para usar de Auth0 como una versión personalizada de Universal Login.

Por otro lado, puede usar Auth0 Management API para [cambiar directamente la contraseña](/docs/es/authenticate/database-connections/password-change#directly-set-the-new-password) de una identidad de usuario definida mediante un tipo de conexión de base de datos. Puede usar Auth0 Management API como parte de cualquier implementación de gestión de perfiles de autoservicio y también como parte de cualquier [personalización de la página Change Password](/docs/es/get-started/architecture-scenarios/business-to-business/branding).

<div id="account-verification">
  ## Verificación de la cuenta
</div>

También tendrás que trabajar en todo momento con una cuenta de usuario verificada y aprovechar los mecanismos que ofrece Auth0. También debes tener en cuenta el cumplimiento normativo, como el [RGPD](https://gdpr-info.eu/), que establece requisitos muy específicos para proteger la privacidad de los ciudadanos de la UE y evitar violaciones de la seguridad de los datos.

Auth0 ofrece funcionalidad lista para usar para enviar un [correo electrónico de verificación](/docs/es/customize/email/manage-email-flow) al correo electrónico del usuario y verificar su cuenta. De forma predeterminada, Auth0 envía automáticamente correos electrónicos de verificación para cualquier identidad de [conexión de base de datos](/docs/es/authenticate/database-connections/password-change) creada como parte del [autorregistro](/docs/es/get-started/architecture-scenarios/business-to-business/provisioning#self-sign-up). Sin embargo, Auth0 también proporciona un [endpoint de la Management API](https://auth0.com/docs/api/v2#!/Tickets/post_email_verification) que puedes usar para enviar correos electrónicos de verificación en los casos en que un proveedor social no valida la dirección de correo electrónico durante el registro del usuario.

<div id="blocking-users">
  ## Bloqueo de usuarios
</div>

[El bloqueo del acceso de los usuarios](/docs/es/manage-users/user-accounts/block-and-unblock-users) en Auth0 permite impedir que los usuarios inicien sesión en las aplicaciones en determinadas condiciones. De forma predeterminada, el Auth0 Dashboard ofrece un mecanismo listo para usar que permite a los administradores bloquear y desbloquear el acceso de los usuarios a todas las aplicaciones, y puedes implementar esta funcionalidad mediante la [Auth0 Management API](https://auth0.com/docs/api/management/v2#!/Users/patch_users_by_id). También puedes usar la extensibilidad de Auth0 para [deshabilitar el acceso de los usuarios a determinadas aplicaciones](/docs/es/manage-users/user-accounts/manage-user-access-to-applications), así como para proporcionar un [control de acceso](/docs/es/get-started/architecture-scenarios/business-to-business/authorization) más granular.

Además, la Auth0 Management API te permite [desbloquear](https://auth0.com/docs/api/management/v2#!/User_Blocks/delete_user_blocks_by_id) a usuarios bloqueados debido a un uso excesivo de credenciales incorrectas.

<div id="admin-portal">
  ## Portal de administración
</div>

Un portal de administración es una aplicación en la que puedes crear nuevos usuarios, editar el perfil de un usuario, ver su actividad, etc. Esta aplicación debería ser accesible solo para administradores. Aunque Auth0 proporciona su Dashboard de administración, no es recomendable dar acceso al Dashboard de administración a muchas personas, ya que hay muchas formas de que alguien pueda afectar sin querer tu tenant de Auth0. En su lugar, Auth0 ofrece otras dos opciones:

* [**Auth0 Management API**](https://auth0.com/docs/api/management/v2): Con Auth0 Management API puedes crear fácilmente una aplicación que permita a los administradores gestionar usuarios. Puedes integrarla en una aplicación existente que tus administradores ya utilicen, o crear una nueva con una UI acorde con tus aplicaciones actuales.
* [**Auth0 Delegated Administration Extension**](/docs/es/customize/extensions/delegated-administration-extension): Esta extensión potente y flexible te permite personalizar la experiencia de administración de usuarios. Puedes adaptarla para que los administradores de tus clientes puedan iniciar sesión y solo ver y gestionar usuarios dentro de su organización.

<Info>
  ### Práctica recomendada

  Si proporcionas tu propia forma de administrar usuarios, solo deberías permitir que los administradores envíen a los usuarios un enlace por correo electrónico para cambiar la contraseña, en lugar de permitirles establecer contraseñas directamente. Si debes ir en contra de esta recomendación y permitir que tus administradores establezcan la contraseña de otra persona, deberías obligar al usuario a cambiarla en su próximo inicio de sesión para que solo él la conozca (y no también un administrador).
</Info>

<div id="multiple-organization-architecture-multitenancy">
  ## Arquitectura de múltiples organizaciones (Multitenencia)
</div>

Muchas plataformas B2B implementan algún tipo de aislamiento y/o identidad visual para la organización de sus clientes, y esto puede añadir complejidad a cualquier sistema de gestión de identidades y accesos (IAM). Si este es su caso, le recomendamos que dedique algo de tiempo a leer nuestra guía y nuestras prácticas recomendadas para este tipo de entorno.

[Arquitectura de múltiples organizaciones](/docs/es/get-started/architecture-scenarios/multiple-organization-architecture)
