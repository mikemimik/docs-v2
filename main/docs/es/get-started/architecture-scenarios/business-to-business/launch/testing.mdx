---

Antes del lanzamiento, deberías haber completado todas las pruebas aplicables a tu entorno.

Quality Assurance es importante para identificar problemas antes de que afecten a tus clientes y, según la naturaleza de tu proyecto, hay varios tipos de pruebas de calidad que te conviene considerar como parte de tu integración con Auth0:

* ¿Tu aplicación es fácil de entender y usar, incluso para personas con alguna discapacidad?
* ¿Tu aplicación necesita funcionar en distintos navegadores y dispositivos?
* ¿Tu aplicación necesita funcionar en entornos multinacionales y/o internacionales?
* ¿Cómo responderá tu aplicación cuando se vea sometida a cargas inesperadas en producción?
* ¿Cómo puedes asegurarte de que tu aplicación esté protegida frente a vulnerabilidades de seguridad?

Auth0 [Universal Login](/docs/es/authenticate/login/auth0-universal-login) y los widgets de interfaz de usuario asociados (como [Lock](/docs/es/libraries/lock)) ya se han diseñado y desarrollado siguiendo prácticas recomendadas de usabilidad y accesibilidad, y ofrecen soporte probado listo para usar para toda una variedad de [navegadores y dispositivos](/docs/es/troubleshoot/customer-support/product-support-matrix). También se ofrece soporte listo para usar para la [internacionalización](/docs/es/customize/internationalization-and-localization) (I18N), con extensibilidad integrada diseñada para casos personalizados de varios idiomas y localización (L10N).

Para garantizar que se cumplan los requisitos funcionales y que los eventos inesperados se gestionen correctamente, se ofrece orientación para probar la [integración](#integration-testing) entre tus aplicaciones y Auth0, así como para realizar [pruebas unitarias](#unit-testing) de módulos individuales de extensibilidad (como [Rules](/docs/es/customize/rules/debug-rules), [Hooks](/docs/es/customize/hooks/update-hooks) y scripts de Custom Database). También se ofrece orientación sobre la [política de pruebas de penetración de Auth0](/docs/es/troubleshoot/customer-support/operational-policies/penetration-testing-policy) para ayudar con las pruebas de vulnerabilidades de seguridad, así como sobre cómo las pruebas con [Mock](#mock-testing) pueden aprovecharse junto con nuestra [política de pruebas de carga](/docs/es/troubleshoot/customer-support/operational-policies/load-testing-policy) para ayudar a garantizar que tus aplicaciones respondan bien ante cargas inesperadas.

<div id="unit-testing">
  ## Pruebas unitarias
</div>

El objetivo de las pruebas unitarias es probar componentes individuales del código. Si crea código personalizado dentro de Auth0 en forma de Rules, Hooks y/o scripts de Custom DB, debería plantearse usar un framework de pruebas (como [Mocha](https://mochajs.org/)) para probar su código. Las empresas que más éxito han tenido con Auth0 han comprobado que resulta útil ejecutar estas pruebas unitarias antes de [desplegar automáticamente](/docs/es/get-started/architecture-scenarios/business-to-business/deployment) la configuración y los recursos asociados del tenant de Auth0.

<div id="integration-testing">
  ## Pruebas de integración
</div>

Se recomienda configurar tenants distintos para desarrollo, pruebas y producción, como se explica en la guía de arquitectura sobre [soporte para SDLC](/docs/es/get-started/architecture-scenarios/business-to-business/architecture#sdlc-support). Auth0 te permite configurar variables disponibles desde la [extensibilidad](/docs/es/customize/extensions) personalizada; puedes considerarlas variables de entorno de tu tenant de Auth0. En lugar de codificar de forma fija referencias que cambian al mover código entre entornos de desarrollo, pruebas y producción, puedes usar un nombre de variable configurado en el tenant al que haga referencia el código de extensibilidad personalizado. Esto facilita que el mismo código personalizado funcione sin cambios en distintos tenants, ya que puede hacer referencia a variables que se completarán con valores específicos del tenant en el momento de la ejecución:

* Para usar variables en Rules, consulta cómo [configurar valores](/docs/es/customize/rules/configuration)
* Para usar variables en Hooks, consulta cómo configurar [secrets](/docs/es/customize/hooks/hook-secrets) en el editor
* Para usar variables en Actions, consulta Explorar Flows y Triggers
* Para usar variables en scripts de Custom DB, consulta los [parámetros de configuración](/docs/es/authenticate/database-connections/custom-db/create-db-connection#step-3-add-configuration-parameters)

<Info>
  ### Práctica recomendada

  Se recomienda usar variables para almacenar valores específicos del tenant, así como cualquier Secret sensible que no deba exponerse en tu código personalizado. Si tu código personalizado está desplegado en GitHub, usar una variable específica del tenant evita exponer valores sensibles a través de tu repositorio de GitHub.
</Info>

<div id="test-automation">
  ### Automatización de pruebas
</div>

Puedes automatizar tu proceso general de compilación incorporando tanto la automatización del despliegue como la automatización de pruebas. Esto puede usarse para desplegar nuevas versiones de la configuración y/o del código personalizado en Auth0, y ejecutar pruebas automatizadas. Si las pruebas detectan algún fallo, puedes usar las capacidades de automatización del despliegue para revertir a la última versión funcional. Para obtener más información, consulta la [guía sobre automatización del despliegue](/docs/es/get-started/architecture-scenarios/business-to-business/deployment).

<div id="mock-testing">
  ## Prueba simulada
</div>

Como equilibrio entre la [política de pruebas de carga](/docs/es/troubleshoot/customer-support/operational-policies/load-testing-policy) de Auth0 y la necesidad de realizar una prueba de carga, es habitual que los clientes de Auth0 simulen los endpoints de Auth0. Esta es una práctica valiosa para garantizar que su aplicación funcione con las interfaces previstas sin tener que limitar sus pruebas, y puede apoyarse en herramientas como [MockServer](http://www.mock-server.com/), [JSON Server](https://github.com/typicode/json-server) o incluso [Postman](https://learning.getpostman.com/docs/postman/mock_servers/setting_up_mock/) para hacerlo.

<div id="pen-testing-optional">
  ## Pruebas de penetración (opcional)
</div>

Si va a realizar pruebas de penetración, debe conocer y cumplir la [política de pruebas de penetración](/docs/es/troubleshoot/customer-support/operational-policies/penetration-testing-policy) de Auth0. Las pruebas de penetración requieren notificar a Auth0 con antelación para evitar que se confundan con actividad maliciosa y se bloqueen.

<div id="load-testing-optional">
  ## Pruebas de carga (opcional)
</div>

Si va a realizar pruebas de carga, debe conocer la [política de pruebas de carga](/docs/es/troubleshoot/customer-support/operational-policies/load-testing-policy) de Auth0 y cumplirla. Las pruebas de carga requieren notificar a Auth0 con antelación. Al planificar sus pruebas de carga, también deberá tener en cuenta los [límites de frecuencia de la API](/docs/es/troubleshoot/customer-support/operational-policies/rate-limit-policy) de Auth0.

Las pruebas de carga requieren la aprobación previa de Auth0, como se explica en la política de pruebas de carga de Auth0. Asegúrese de tener en cuenta el tiempo de antelación necesario para que se revise una solicitud y deje suficiente margen tanto para la revisión como para la ejecución de las pruebas. Si su solicitud de prueba de carga ha sido aprobada, la siguiente guía puede ayudarle a evitar errores y resultados de prueba inexactos.

* Ejecute una traza HTTP durante una ejecución de prueba de su aplicación para identificar todas las llamadas que su aplicación o la prueba prevista deben realizar, y asegúrese de incluirlas en la prueba para que sea representativa de lo que ocurrirá en producción.
* Diseñe su prueba teniendo presentes los límites de frecuencia de la Auth0 API.
* El uso de cualquier código personalizado en Auth0 (Actions, Rules, Hooks, scripts de base de datos personalizados, conexiones de <Tooltip tip="OAuth 2.0: Framework de autorización que define protocolos y flujos de trabajo de autorización." cta="Ver glosario" href="/docs/es/glossary?term=OAuth">OAuth</Tooltip> personalizadas) invocará el sandbox de código personalizado de Auth0, y esto puede tener un mayor costo en términos de rendimiento. Desactive las Rules a menos que sean esenciales para la prueba. Si están desactivadas, tendrá una mayor capacidad de procesamiento que si están activadas.
* Estime la carga total esperada para su entorno de producción y el porcentaje de llamadas a cada endpoint, y estructure su prueba de rendimiento en consecuencia para obtener un resultado realista. Los distintos endpoints tienen distintos costos de rendimiento. Si no diseña una prueba representativa, obtendrá resultados engañosos.
* No haga llamadas que dependan de los resultados de llamadas anteriores sin comprobar que las llamadas o respuestas previas necesarias se hayan completado. Simplemente añadir una demora puede no ser suficiente.
* Asegúrese de implementar un manejo de errores adecuado. Una causa frecuente de incidencias durante las pruebas son los errores en el código personalizado (Actions, Rules, Hooks, scripts de base de datos personalizados, scripts de conexión OAuth personalizados) provocados por excepciones no controladas en el código personalizado.
* Las pruebas de carga deben diseñarse para comenzar con un nivel bajo e incrementar la carga gradualmente, capturando datos en cada nivel, a fin de obtener los resultados más útiles. Comenzar con un nivel alto y fallar de inmediato ofrece menos información sobre lo que el sistema puede soportar.
* Es normal tener que ejecutar una prueba de rendimiento varias veces, posiblemente ajustando el código sometido a prueba o el entorno/configuración de prueba. Asegúrese de comenzar las pruebas con antelación suficiente para disponer de tiempo para más de una iteración.
* Use su propia cuenta de proveedor de correo y asegúrese de organizar con antelación una cuota de envío suficiente; de lo contrario, el proveedor de correo podría aplicarle límites de frecuencia. Desactive el envío de correo si no lo utiliza.
* Asegúrese de usar las credenciales de su propia cuenta para todas las conexiones sociales, en lugar de las claves de desarrollo de Auth0. En el <Tooltip tip="Auth0 Dashboard: El producto principal de Auth0 para configurar sus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Auth0+dashboard">Auth0 Dashboard</Tooltip>, vaya a Connections -&gt; Social -&gt; &#123;name of connection&#125;—para ver instrucciones sobre cómo agregar a la conexión las credenciales de su propia cuenta del proveedor social. Nota: algunos proveedores sociales no permiten pruebas de carga. Consulte la política de su(s) proveedor(es).
* Para evitar la limitación de frecuencia y simular con mayor precisión la carga real, sus pruebas deberán enviar solicitudes para distintos usuarios, no todas las solicitudes para el mismo usuario. Si usa solo uno o unos pocos usuarios, el almacenamiento en caché puede reducir la carga efectiva y no proporcionar resultados realistas.
* Asegúrese de mantenerse dentro de los parámetros acordados para la prueba y de la política de pruebas de carga de Auth0. Auth0 se reserva el derecho de interrumpir cualquier prueba de rendimiento/carga que no se mantenga dentro de los límites de los parámetros acordados o que se extienda más allá del período de prueba programado.
