---

Antes del lanzamiento, ya debería haber completado todas las pruebas aplicables a su entorno.

El aseguramiento de la calidad es fundamental para identificar problemas antes de que afecten a sus clientes y, según la naturaleza de su proyecto, hay varios tipos de pruebas de aseguramiento de la calidad que le conviene considerar como parte de su integración con Auth0:

* ¿Su aplicación es fácil de entender y usar, incluso para personas con alguna discapacidad?
* ¿Su aplicación necesita funcionar en distintos navegadores y dispositivos?
* ¿Su aplicación necesita funcionar en entornos multinacionales y/o internacionales?
* ¿Cómo se comportará su aplicación cuando se la someta a cargas inesperadas en producción?
* ¿Cómo puede asegurarse de que su aplicación esté protegida frente a vulnerabilidades de seguridad?

Auth0 [Universal Login](/docs/es/authenticate/login/auth0-universal-login) y los widgets de interfaz de usuario asociados (como [Lock](/docs/es/libraries/lock)) ya han sido diseñados y desarrollados siguiendo las prácticas recomendadas de usabilidad y accesibilidad, y ofrecen soporte probado, listo para usar, para una amplia variedad de [navegadores y dispositivos](/docs/es/troubleshoot/customer-support/product-support-matrix). También se ofrece soporte listo para usar para la [internacionalización](/docs/es/customize/internationalization-and-localization) (I18N), con extensibilidad integrada para casos personalizados de varios idiomas y localización (L10N).

Para garantizar que se cumplan los requisitos funcionales y que los eventos inesperados se gestionen correctamente, se ofrece orientación para probar la [integración](#integration-testing) entre su(s) aplicación(es) y Auth0, así como para realizar [pruebas unitarias](#unit-testing) de módulos individuales de extensibilidad (como [Rules](/docs/es/customize/rules/debug-rules), [Hooks](/docs/es/customize/hooks/update-hooks) y scripts de base de datos personalizada). También se ofrece orientación sobre la [política de pruebas de penetración](/docs/es/troubleshoot/customer-support/operational-policies/penetration-testing-policy) de Auth0 para ayudarle a detectar vulnerabilidades de seguridad, así como sobre cómo pueden utilizarse las pruebas con [prueba simulada](#mock-testing) junto con nuestra [política de pruebas de carga](/docs/es/troubleshoot/customer-support/operational-policies/load-testing-policy) para ayudar a garantizar que su(s) aplicación(es) rindan bien ante cargas inesperadas.

<div id="unit-testing">
  ## Pruebas unitarias
</div>

El objetivo de las pruebas unitarias es probar unidades individuales de código. Si crea código personalizado dentro de Auth0 en forma de Rules, Hooks y/o scripts de Custom DB, debería considerar usar un framework de pruebas (como [Mocha](https://mochajs.org/)) para poner a prueba su código. Las empresas que han tenido más éxito con Auth0 han comprobado que resulta útil ejecutar estas pruebas unitarias antes de [desplegar automáticamente](/docs/es/get-started/architecture-scenarios/business-to-business/deployment) la configuración y los recursos del tenant de Auth0.

<div id="integration-testing">
  ## Pruebas de integración
</div>

Es una práctica recomendada configurar distintos tenants para desarrollo, pruebas y producción, como se indica en la guía de [Arquitectura](/docs/es/get-started/architecture-scenarios/business-to-business/architecture#sdlc-support) sobre [soporte para SDLC](/docs/es/get-started/architecture-scenarios/business-to-business/architecture#sdlc-support). Auth0 le permite configurar variables disponibles desde la [extensibilidad](/docs/es/customize/extensions) personalizada; pueden entenderse como variables de entorno para su tenant de Auth0. En lugar de codificar de forma fija referencias que cambian al mover código entre entornos de desarrollo, pruebas y producción, puede usar un nombre de variable configurado en el tenant y referenciado por el código de extensibilidad personalizado. Esto facilita que el mismo código personalizado funcione, sin cambios, en distintos tenants, ya que puede hacer referencia a variables que se rellenarán con valores específicos de cada tenant en el momento de la ejecución:

* Para usar variables en Rules, vea cómo [configurar valores](/docs/es/customize/rules/configuration)
* Para usar variables en Hooks, vea cómo configurar [secrets](/docs/es/customize/hooks/hook-secrets) en el editor
* Para usar variables en Actions, vea Explorar Flows y Triggers
* Para usar variables en Custom DB Scripts, consulte los [parámetros de configuración](/docs/es/authenticate/database-connections/custom-db/create-db-connection#step-3-add-configuration-parameters)

<Info>
  ### Práctica recomendada

  Es una práctica recomendada usar variables para almacenar valores específicos del tenant, así como cualquier secreto sensible que no deba exponerse en su código personalizado. Si su código personalizado se almacena en GitHub, usar una variable específica del tenant evita la exposición de valores sensibles a través de su repositorio de GitHub.
</Info>

<div id="test-automation">
  ### Automatización de pruebas
</div>

Puedes automatizar todo tu proceso de compilación incorporando la automatización del despliegue, además de la automatización de pruebas. Esto puede usarse para desplegar nuevas versiones de la configuración y/o del código personalizado en Auth0 y ejecutar pruebas automatizadas. Si las pruebas detectan algún fallo, las capacidades de automatización del despliegue pueden usarse para revertir a la última versión funcional. Para más información, consulta la [guía sobre automatización del despliegue](/docs/es/get-started/architecture-scenarios/business-to-business/deployment).

<div id="mock-testing">
  ## Prueba simulada
</div>

Como equilibrio entre la [política de pruebas de carga](/docs/es/troubleshoot/customer-support/operational-policies/load-testing-policy) de Auth0 y la necesidad de realizar una prueba de carga, es habitual que los clientes de Auth0 simulen los endpoints de Auth0. Esta es una práctica valiosa para garantizar que su aplicación funcione con las interfaces previstas sin tener que limitar las pruebas, y pueden utilizarse herramientas como [MockServer](http://www.mock-server.com/), [JSON Server](https://github.com/typicode/json-server) o incluso [Postman](https://learning.getpostman.com/docs/postman/mock_servers/setting_up_mock/) como ayuda.

<div id="pen-testing-optional">
  ## Pruebas de intrusión (opcional)
</div>

Si va a realizar pruebas de intrusión, debe conocer y cumplir la [política de prueba de penetración](/docs/es/troubleshoot/customer-support/operational-policies/penetration-testing-policy) de Auth0. Estas pruebas requieren notificar a Auth0 con antelación para evitar que se confundan con actividad maliciosa y se interrumpan.

<div id="load-testing-optional">
  ## Pruebas de carga (opcional)
</div>

Si va a realizar pruebas de carga, debe conocer la [política de pruebas de carga](/docs/es/troubleshoot/customer-support/operational-policies/load-testing-policy) de Auth0 y cumplirla. Las pruebas de carga requieren notificación previa a Auth0. Al planificar sus pruebas de carga, también deberá tener en cuenta los [límites de frecuencia de la API](/docs/es/troubleshoot/customer-support/operational-policies/rate-limit-policy) de Auth0.

Las pruebas de carga requieren aprobación previa de Auth0, como se explica en la política de pruebas de carga de Auth0. Asegúrese de tener en cuenta el plazo necesario para que se revise una solicitud y de dejar tiempo suficiente tanto para la revisión como para la ejecución de las pruebas. Si su solicitud de prueba de carga ha sido aprobada, la siguiente orientación puede ayudarle a evitar errores y resultados de prueba inexactos.

* Ejecute una traza HTTP durante una ejecución de prueba de su aplicación para identificar todas las llamadas que su aplicación o la prueba prevista debe realizar, y asegúrese de que su prueba las incluya para que sea representativa de lo que ocurrirá en producción.
* Diseñe su prueba teniendo en cuenta los límites de frecuencia de la Auth0 API.
* El uso de cualquier código personalizado en Auth0 (Actions, Rules, Hooks, scripts personalizados de base de datos, conexiones <Tooltip tip="OAuth 2.0: Framework de autorización que define protocolos y flujos de trabajo de autorización." cta="Ver glosario" href="/docs/es/glossary?term=OAuth">OAuth</Tooltip> personalizadas) invocará el sandbox de código personalizado de Auth0, y esto puede tener un mayor costo en términos de rendimiento. Desactive Rules a menos que sean esenciales para la prueba. Si están desactivadas, tendrá un mayor rendimiento que si están activadas.
* Estime la carga total esperada para su entorno de producción y el porcentaje de llamadas a cada endpoint, y estructure su prueba de rendimiento en consecuencia para que le proporcione un resultado realista. Los distintos endpoints tienen diferentes costos de rendimiento. Si no diseña una prueba representativa, obtendrá resultados engañosos.
* No realice llamadas que dependan de los resultados de llamadas anteriores sin comprobar que las llamadas o respuestas previas necesarias se hayan completado. Simplemente incorporar un retraso puede no ser suficiente.
* Asegúrese de implementar un manejo de errores adecuado. Una causa frecuente de incidencias durante las pruebas son los errores en el código personalizado (Actions, Rules, Hooks, scripts personalizados de base de datos, scripts personalizados de conexiones OAuth) causados por excepciones no controladas en el código personalizado.
* Las pruebas de carga deben redactarse para comenzar en un nivel bajo e incrementar la carga gradualmente, capturando datos en cada nivel, a fin de obtener los resultados más útiles. Comenzar en un nivel alto y fallar de inmediato ofrece menos información sobre lo que el sistema puede soportar.
* Es normal tener que ejecutar una prueba de rendimiento varias veces, posiblemente ajustando el código sometido a prueba o el arnés/configuración de prueba. Asegúrese de comenzar sus pruebas con antelación para disponer de tiempo suficiente para más de una iteración.
* Use su propia cuenta de proveedor de correo y asegúrese de gestionar con antelación una cuota suficiente de envío de correo o podría verse sujeto a límites de frecuencia por parte del proveedor. Desactive el envío de correo si no lo utiliza.
* Asegúrese de usar las credenciales de su propia cuenta para todas las conexiones sociales en lugar de las claves de desarrollo de Auth0. En el <Tooltip tip="Auth0 Dashboard: El producto principal de Auth0 para configurar sus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Auth0+dashboard">Auth0 Dashboard</Tooltip>, vaya a Connections -&gt; Social -&gt; &#123;name of connection&#125;—para ver instrucciones sobre cómo agregar a la conexión las credenciales de la cuenta de su propio proveedor social. Nota: algunos proveedores sociales no permiten pruebas de carga. Consulte la política de su proveedor o proveedores.
* Para evitar la limitación de frecuencia y simular con mayor precisión una carga real, sus pruebas deberán enviar solicitudes para distintos usuarios, no todas las solicitudes para el mismo usuario. Si usa solo uno o unos pocos usuarios, el almacenamiento en caché puede reducir la carga efectiva y no proporcionar resultados realistas.
* Asegúrese de mantenerse dentro de los parámetros acordados para la prueba y de la política de pruebas de carga de Auth0. Auth0 se reserva el derecho de finalizar cualquier prueba de rendimiento o de carga que no se mantenga dentro de los límites de los parámetros acordados o que se extienda más allá del período de prueba programado.
