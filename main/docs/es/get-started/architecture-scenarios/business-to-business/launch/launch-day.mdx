<div id="notifications-announcements">
  ## Notificaciones / anuncios
</div>

Para que un lanzamiento se desarrolle sin contratiempos, conviene que todas las partes interesadas estén al tanto del lanzamiento inminente y comprendan el plan de lanzamiento, así como su función y responsabilidades. Además de notificar a los equipos que participarán activamente, también puede ser útil avisar a los equipos a los que podría ser necesario recurrir si algo sale mal. Tener a alguien de guardia durante un lanzamiento puede ayudar a agilizar la respuesta. Asegúrate de identificar y notificar a cualquier equipo que pueda tener que responder preguntas de los clientes, incluso en redes sociales.

<div id="parties-to-notify">
  ### Partes a las que hay que notificar
</div>

* Clientes
* Socios comerciales, si corresponde
* Equipos de aplicaciones afectados por el lanzamiento
* Equipos de soporte
* Equipos de red (cambios de red, de guardia, por si surge alguna incidencia)
* Equipos de seguridad (de guardia, por si surge alguna incidencia)
* Equipos de marketing (listos para hacer anuncios y responder ante incidencias)
* Equipos de redes sociales (listos para supervisar las redes sociales y responder)
* Equipos de ventas (preparados para responder preguntas de los clientes)
* Equipos de éxito del cliente (preparados para responder preguntas de los clientes)

<div id="notification-plan">
  ## Plan de notificación
</div>

Tu plan de notificación debe incluir elementos como la <Tooltip tip="Audience: Identificador único de la audiencia de un token emitido. Se denomina aud en un token; su valor contiene el ID de una aplicación (ID de cliente) para un token de ID o de una API (Identificador de API) para un token de acceso." cta="Ver glosario" href="/docs/es/glossary?term=audience">audiencia</Tooltip> objetivo, los puntos clave para la audiencia, el contenido del mensaje, el plan para distribuir la notificación y cómo probar los mensajes.

La lista de elementos que deben incluirse en el plan es:

* Audiencia objetivo (ten en cuenta tanto las audiencias internas como las externas)
* Mensaje
* Plazos
* Dependencias
* Responsables (quién lo enviará)
* Mecanismo (cómo se comunicará)
* Mensaje de prueba y entrega (si corresponde: haz una prueba para asegurarte de que las notificaciones se envíen)

<div id="notification-distribution">
  ## Distribución de notificaciones
</div>

Una táctica habitual es enviar notificaciones por lotes para repartir la carga inicial y reducir la confusión si surge algún problema imprevisto. Es más fácil corregir problemas con un grupo pequeño que durante un lanzamiento general.

* Una opción es empezar con un lote relativamente pequeño de notificaciones y, si no se detectan problemas, ir aumentando el tamaño de los lotes con el tiempo.
* También puedes enviar lotes siguiendo un calendario escalonado en todo el mundo para repartir la carga que recibe el sistema al mismo tiempo y hacer que las notificaciones lleguen en un momento óptimo dentro de cada zona horaria, lo que aumenta la probabilidad de que se lean los mensajes.
* Puedes hacer un lanzamiento gradual para una parte de los usuarios, como clientes concretos, regiones u otro tipo de agrupación que tenga sentido para tu aplicación.

<div id="outage-windows-if-needed">
  ## Ventanas de interrupción del servicio (si es necesario)
</div>

Algunas organizaciones requieren una solicitud formal para programar una ventana de interrupción del servicio si la puesta en producción requiere alguna interrupción o tiempo de inactividad. Si tu organización lo exige, asegúrate de determinar si será necesario tiempo de inactividad para el cutover o la puesta en producción (o para otros sistemas dependientes) y de enviar con antelación las solicitudes de interrupción del servicio o de cambio necesarias, teniendo en cuenta cualquier plazo de preaviso requerido.

<div id="cutover-plan-if-needed">
  ## Plan de cutover (si es necesario)
</div>

Algunos lanzamientos implican hacer un cutover de una solución anterior a una nueva. Si tu proyecto se ajusta a este escenario, debes asegurarte de identificar todo lo que debe suceder, así como cualquier dependencia, la persona responsable de cada tarea y los tiempos necesarios. También puede ser conveniente prever reemplazos para todos los roles importantes o para cada región, por si alguien se enferma inesperadamente o no está disponible por cualquier otro motivo. Una lista de verificación de aspectos que debes considerar para el plan de cutover es:

* ¿Has documentado el plan de cutover y el plan de rollback, si fuera necesario?
* ¿Se necesitan copias de seguridad de algo antes del cambio?
* ¿Hace falta realizar cambios preparatorios en los datos?
* ¿Hay registros DNS que deban cambiarse?
* ¿Hay cambios en el firewall?
* ¿Hay nuevos objetivos de monitoreo?
* ¿Hay software que deba desplegarse?

<div id="go-no-go-criteria">
  ## Criterios para avanzar / no avanzar
</div>

En tu plan general de lanzamiento, es útil contar con criterios para avanzar o no avanzar y haber analizado de antemano los tipos de problemas que podrían surgir, cuáles podrían resolverse sobre la marcha y cuáles requerirían revertir. Un plan de lanzamiento puede especificar intervalos periódicos de revisión, con criterios sobre qué evaluar en cada punto de control y cuánto tiempo permitir que un problema siga sin resolverse.

Para cada etapa del lanzamiento, conviene tener definidos criterios de éxito que indiquen que el lanzamiento avanza según lo previsto y puede continuar. Algunos criterios de ejemplo podrían ser:

* El registro de usuarios crece con errores mínimos
* Los inicios de sesión de los usuarios se producen al ritmo esperado, con errores mínimos
* Los problemas reportados al soporte se mantienen por debajo de un determinado umbral
* No se identifican problemas que puedan provocar corrupción de datos

También es útil haber identificado criterios que podrían llevar a una decisión de “no avanzar” para detener el lanzamiento. La tolerancia al riesgo varía según el entorno, pero algunos criterios de ejemplo podrían incluir:

* Un alto porcentaje de registros o inicios de sesión de usuarios que derivan en errores que no pueden resolverse rápidamente
* Un número elevado de problemas de soporte que no pueden resolverse rápidamente
* La identificación de una condición que podría provocar corrupción de datos
* El descubrimiento de un problema de seguridad grave

<div id="rollback">
  ## Rollback
</div>

Siempre es prudente contar con un plan sobre cómo hacer rollback o revertir un lanzamiento, por si ocurre algo imprevisto que no se pueda resolver. Revisar el plan de lanzamiento para cada paso que implique un cambio puede ayudar a identificar las tareas o los cambios necesarios para revertir un lanzamiento o un cutover.

El plan de rollback debe incluir los pasos que se deben seguir, la secuencia, cuánto tiempo probablemente llevará cada uno y la parte responsable. Comprender el tiempo total necesario para hacer rollback puede ayudar a determinar el momento de la decisión final de seguir o no seguir, de modo que encaje dentro de cualquier ventana de interrupción del servicio requerida.

Si se migran o modifican datos para el lanzamiento, el plan debe incluir cómo revertirlos, si fuera necesario. Revertirlos puede requerir ejecutar scripts para deshacer cambios operativos o restaurar un almacén de datos a partir de una copia de seguridad tomada antes de que comenzara el proceso de lanzamiento.

También es necesario planificar el caso en que se introduzcan datos en un sistema nuevo antes de que tenga que revertirse. ¿Será necesario abandonar esos datos / transacciones con el rollback o habrá alguna forma de capturarlos y aplicarlos en otro lugar para que no se pierdan?

Si la resolución de incidencias o el proceso para revertir podrían llegar a tomar más de un turno, querrá asegurarse de contar con una persona principal y quizá otra de apoyo disponibles y preparadas para encargarse de la situación durante cada turno de trabajo. Si una incidencia da lugar a la necesidad de una respuesta prolongada, mucho más allá de un turno, existen límites en cuanto al tiempo que las personas pueden rendir de forma realista sin un descanso. Puede ser útil estar preparado con recursos para un esfuerzo de respuesta a incidencias continuo entre husos horarios, si fuera necesario.

<div id="standby-contacts">
  ## Contactos de guardia
</div>

A medida que se acerca el día del lanzamiento, conviene identificar a todos los contactos que podrían necesitarse para la resolución de problemas o para resolverlos, y pedirles que estén de guardia y listos para ayudar si hace falta. La persona responsable del lanzamiento debe tener la información de contacto de cada persona de la lista de guardia para agilizar las comunicaciones.

Si hay una &quot;sala de lanzamiento&quot; física o virtual, las personas de guardia deben saber dónde está y estar listas para unirse si hace falta. Tener preparada una sala central o una videoconferencia puede agilizar las comunicaciones y la resolución de problemas entre todas las partes si surge algún problema.

<div id="success-criteria">
  ## Criterios de éxito
</div>

Se necesita mucha planificación para que un lanzamiento tenga éxito, pero ¿sabrás cómo evaluarlo? Si defines los criterios de éxito antes del lanzamiento, puedes determinar qué debes monitorear y si hace falta implementar monitoreo o comprobaciones adicionales para evaluarlo. Por ejemplo, si uno de los criterios de éxito es el número de registros o inicios de sesión, ¿tienes alguna forma de monitorearlo y se ha comprobado que sea preciso?

Te convendrá contar con estadísticas para poder destacar el éxito de tu lanzamiento. No querrás descubrir después del lanzamiento que no recopilaste ningún dato para cuantificar todo el arduo trabajo que tu equipo dedicó a él.

<div id="risks-mitigations-plan">
  ## Plan de riesgos y mitigaciones
</div>

No es agradable pensar en todo lo que podría salir mal, pero si ocurre algo, te alegrarás de haberlo hecho, ya que contar con un plan puede acelerar la respuesta. Algunos ejemplos para los que conviene prepararse son:

* Error en la aplicación
* Incompatibilidad de la aplicación con la configuración del navegador del usuario
* Fallo o interrupción de la red
* Ataque DoS
* Fallo del entorno de alojamiento
* Problemas de carga o capacidad
* Problemas de datos o corrupción
* Vulnerabilidad de seguridad detectada

Si tuviste un período beta, puede ser útil revisar sus resultados para identificar otros posibles escenarios de fallo.
