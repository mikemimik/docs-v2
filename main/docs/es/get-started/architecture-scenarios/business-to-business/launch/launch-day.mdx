<div id="notifications-announcements">
  ## Notificaciones / anuncios
</div>

Para que un lanzamiento se desarrolle sin contratiempos, conviene que todas las partes interesadas estén al tanto del lanzamiento inminente y comprendan el plan de lanzamiento, así como su papel y sus responsabilidades. Además de notificar a los equipos que participarán activamente, también puede ser útil avisar a los equipos que podrían hacer falta si algo sale mal. Contar con alguien de guardia durante un lanzamiento puede ayudar a agilizar la respuesta. Asegúrate de identificar y notificar a cualquier equipo que pueda tener que responder preguntas de los clientes, incluso en redes sociales.

<div id="parties-to-notify">
  ### Partes a notificar
</div>

* Clientes
* Socios comerciales, si corresponde
* Equipos de las aplicaciones afectados por el lanzamiento
* Equipos de soporte
* Equipos de red (cambios de red, de guardia en caso de incidencias)
* Equipos de seguridad (de guardia en caso de incidencias)
* Equipos de marketing (listos para hacer anuncios y responder a incidencias)
* Equipos de redes sociales (listos para supervisar las redes sociales y responder)
* Equipos de ventas (preparados para responder preguntas de los clientes)
* Equipos de éxito del cliente (preparados para responder preguntas de los clientes)

  ## Plan de notificación
</div>

Tu plan de notificación debe incluir elementos como la <Tooltip tip="Audiencia: Identificador único de la audiencia de un token emitido. Se denomina aud en un token; su valor contiene el ID de una aplicación (ID de cliente) para un token de ID o de una API (identificador de API) para un token de acceso." cta="Ver glosario" href="/docs/es/glossary?term=audience">audiencia</Tooltip> objetivo, los puntos clave para esa audiencia, el contenido del mensaje, los planes para distribuir la notificación y cómo probar los mensajes.

Una lista de elementos que se deben incluir en el plan es:

* Audiencia objetivo (considera tanto audiencias internas como externas)
* Mensaje
* Plazos
* Dependencias
* Responsables (quién lo enviará)
* Mecanismo (cómo se comunicará)
* Mensaje de prueba y entrega (si corresponde: haz una prueba para asegurarte de que se envíen las notificaciones)

<div id="notification-distribution">
  ## Distribución de notificaciones
</div>

Una táctica habitual es publicar las notificaciones por lotes para repartir la carga inicial y limitar el alcance de la confusión si surge algún problema imprevisto. Es más fácil corregir problemas con un grupo pequeño que durante un lanzamiento masivo.

* Una forma de hacerlo es empezar con un lote de notificaciones relativamente pequeño y, si no se detecta ningún problema, ir aumentando el tamaño de los lotes con el tiempo.
* También puedes enviar los lotes de forma escalonada en todo el mundo para repartir la carga que llega al sistema al mismo tiempo y hacer que las notificaciones lleguen en el momento óptimo dentro de cada zona horaria, lo que aumenta la probabilidad de que se lean los mensajes.
* También puedes hacer un lanzamiento gradual para una parte de los usuarios, como clientes individuales, regiones o cualquier otra agrupación que tenga sentido para tu aplicación.

<div id="outage-windows-if-needed">
  ## Ventanas de interrupción (si es necesario)
</div>

Algunas organizaciones requieren una solicitud formal para una ventana de interrupción si el lanzamiento exige alguna interrupción o tiempo de inactividad. Si tu organización lo requiere, asegúrate de determinar si será necesario tiempo de inactividad para la transición o el lanzamiento (o para otros sistemas dependientes) y presenta con antelación las solicitudes de interrupción o de cambio necesarias, respetando cualquier plazo de preaviso requerido.

<div id="cutover-plan-if-needed">
  ## Plan de transición (si es necesario)
</div>

Algunas puestas en producción implican la transición de una solución anterior a una nueva. Si tu proyecto encaja en este caso, debes asegurarte de identificar todo lo que debe suceder, así como cualquier dependencia, la persona responsable de cada tarea y los tiempos necesarios. También puede ser conveniente prever reemplazos para todas las funciones clave o en cada región, por si alguien se enferma de forma inesperada o no está disponible por algún otro motivo. Una lista de verificación de aspectos a tener en cuenta para el plan de transición es:

* ¿Has documentado el plan de transición y el plan de rollback, en caso de ser necesario?
* ¿Se necesitan copias de seguridad de algo antes del cambio?
* ¿Se requieren cambios preparatorios en los datos?
* ¿Hay que cambiar registros DNS?
* ¿Se necesitan cambios en el firewall?
* ¿Hay nuevos objetivos de monitoreo?
* ¿Hay software que deba desplegarse?

<div id="go-no-go-criteria">
  ## Criterios de seguir adelante / no seguir adelante
</div>

En tu plan de lanzamiento general, es útil contar con criterios de seguir adelante/no seguir adelante y haber analizado de antemano qué tipos de problemas podrían surgir, cuáles podrían resolverse sobre la marcha y cuáles requerirían revertir. Un plan de lanzamiento puede establecer revisiones periódicas con criterios sobre qué evaluar en cada punto de control y cuánto tiempo puede permanecer sin resolverse una incidencia.

Para cada etapa del lanzamiento, conviene definir criterios de éxito que indiquen que el lanzamiento avanza según lo previsto y puede continuar. Algunos ejemplos de criterios podrían ser:

* Crecimiento de los registros de usuarios con errores mínimos
* Inicios de sesión de usuarios al ritmo esperado, con errores mínimos
* Incidencias de soporte reportadas por debajo de un determinado umbral
* No se identifican problemas que puedan provocar corrupción de datos

También es útil haber identificado criterios que podrían desencadenar una decisión de “no seguir adelante” para detener el lanzamiento. La tolerancia al riesgo varía según el entorno, pero algunos criterios de ejemplo podrían incluir:

* Alto porcentaje de registros o inicios de sesión de usuarios que generan errores que no pueden resolverse rápidamente
* Alto número de incidencias de soporte que no pueden resolverse rápidamente
* Se identifica una condición que podría provocar corrupción de datos
* Se descubre un problema de seguridad de alta gravedad

<div id="rollback">
  ## Rollback
</div>

Siempre es prudente contar con un plan para hacer rollback o revertir un lanzamiento, por si ocurre algo imprevisto que no pueda resolverse. Revisar el plan de lanzamiento en cada paso que implique un cambio puede ayudar a identificar las tareas o modificaciones necesarias para revertir un lanzamiento o una transición.

El plan de rollback debe incluir los pasos que se deben seguir, la secuencia, cuánto tiempo probablemente llevará cada uno y quién será la parte responsable. Comprender el tiempo total necesario para hacer rollback puede ayudar a determinar el momento de la decisión final de seguir/no seguir, de modo que se ajuste a cualquier ventana de interrupción requerida.

Si se migran o modifican datos para el lanzamiento, el plan debe incluir cómo revertirlos, si es necesario. Revertirlos puede requerir ejecutar scripts para deshacer cambios operativos o restaurar un almacén de datos a partir de una copia de seguridad tomada antes de que comenzara el proceso de lanzamiento.

También es necesario planificar el caso en que se introduzcan datos en un sistema nuevo antes de que tenga que revertirse. ¿Será necesario abandonar esos datos / transacciones con el rollback o habrá alguna forma de capturarlos y aplicarlos en otro lugar para que no se pierdan?

Si la resolución de incidencias o el proceso para revertir podría tardar más de un turno, convendrá asegurarse de contar con una persona principal y quizá otra de respaldo, disponibles y preparadas para encargarse de la situación durante cada turno de trabajo. Si una incidencia da lugar a la necesidad de una respuesta prolongada, mucho más allá de un turno, hay límites en cuanto al tiempo que las personas pueden rendir de forma realista sin un descanso. Puede ser útil contar con recursos preparados para una respuesta a incidencias de tipo follow-the-sun, si fuera necesario.

<div id="standby-contacts">
  ## Contactos de guardia
</div>

A medida que se acerca el día del lanzamiento, conviene identificar a todos los contactos que podrían hacer falta para la resolución de problemas o de incidencias, y pedirles que estén de guardia y listos para ayudar si es necesario. La persona responsable del lanzamiento debe tener la información de contacto de cada persona de la lista de guardia para agilizar las comunicaciones.

Si hay una &quot;sala de lanzamiento&quot; física o virtual, las personas de guardia deben saber dónde está y estar listas para unirse si es necesario. Tener preparada una sala central o una videoconferencia puede agilizar las comunicaciones y la resolución de problemas entre todas las partes si surge una incidencia.

<div id="success-criteria">
  ## Criterios de éxito
</div>

Planificar bien un lanzamiento es clave para que tenga éxito, pero ¿sabrás cómo evaluarlo? Si defines los criterios de éxito antes del lanzamiento, podrás determinar qué debes monitorear y si hace falta contar con monitoreo o comprobaciones adicionales para evaluarlo. Por ejemplo, si uno de los criterios de éxito es la cantidad de registros o inicios de sesión, ¿tienes una forma de monitorearla y se ha probado para garantizar que sea precisa?

Te convendrá contar con estadísticas para poder destacar el éxito de tu lanzamiento. No querrás darte cuenta después del lanzamiento de que no recopilaste ningún dato para cuantificar todo el esfuerzo que tu equipo dedicó al lanzamiento.

<div id="risks-mitigations-plan">
  ## Plan de riesgos y mitigaciones
</div>

No es agradable pensar en todo lo que podría salir mal, pero si llega a ocurrir algo, agradecerás haberlo hecho, ya que contar con un plan puede acelerar la respuesta. Algunos ejemplos para los que conviene prepararse son:

* Error en una aplicación de software
* Incompatibilidad de la aplicación con la configuración del navegador del usuario
* Fallo/interrupción de la red
* Ataque DoS
* Fallo del entorno de alojamiento
* Problemas de carga / capacidad
* Problemas de datos / corrupción
* Vulnerabilidad de seguridad detectada

Si hubo un período beta, puede ser útil revisar sus resultados para identificar otros posibles escenarios de fallo.
