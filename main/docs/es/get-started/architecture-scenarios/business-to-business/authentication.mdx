Para poder prestar servicios a tus usuarios, debes ser capaz de identificar quiénes son. Este proceso se denomina autenticación de usuario. Existen diversas formas de autenticar a un usuario: mediante cuentas de redes sociales, nombre de usuario y contraseña, <Tooltip tip="Passwordless: Forma de autenticación que no depende de una contraseña como primer factor." cta="Ver glosario" href="/docs/es/glossary?term=passwordless">passwordless</Tooltip>; y a menudo se recomienda ir más allá del primer factor habilitando la <Tooltip tip="Passwordless: Forma de autenticación que no depende de una contraseña como primer factor." cta="Ver glosario" href="/docs/es/glossary?term=multi-factor+authentication">autenticación multifactor</Tooltip> (MFA).

<Info>
  ### Práctica recomendada

  Es importante tener en cuenta tanto la seguridad como la experiencia de usuario al diseñar la forma en que autenticarás a tus usuarios. Ofrecer múltiples factores primarios y/o exigir más de un factor durante la autenticación son maneras de lograr ambos objetivos.
</Info>

Hay una serie de aspectos que deberás considerar al analizar la funcionalidad y el flujo de trabajo:

* ¿Dónde ingresarán sus credenciales los usuarios?
* ¿Cómo mantendrás seguras las credenciales de los usuarios?
* ¿Cómo mantendrás tu sistema de autenticación?
* ¿Cómo puedes ofrecer autenticación por contraseña a tus usuarios?
* ¿Cómo puedes evitar que los hackers intenten iniciar sesión suplantando a tus usuarios?
* ¿Cómo implementarás la autenticación en distintos tipos de aplicaciones?
* ¿Cómo puedes facilitar el inicio de sesión a usuarios que provienen de diferentes contextos lingüísticos?
* ¿Cómo ofrecerás una buena experiencia de usuario mientras migras desde un sistema de autenticación heredado?
* ¿Qué debes tener en cuenta al integrar aplicaciones con Auth0?
* ¿Necesitas ofrecer autenticación multifactor?
* ¿Qué haces si tienes un servicio en el que el usuario no puede iniciar sesión de antemano?
* ¿Puedes pasar el mismo <Tooltip tip="Token de acceso: Credencial de autorización, en forma de cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=access+token">token de acceso</Tooltip> de usuario de una API a otra?
* ¿Qué haces si necesitas aislar usuarios por organización?
* ¿Cómo gestionarás la identificación de la organización a la que pertenecen los usuarios?
* ¿Cuál es el beneficio de ofrecer conexiones empresariales para tus organizaciones?

El [Universal Login](#universal-login) de Auth0 proporciona a los usuarios una experiencia segura y confiable, independientemente de si eliges ofrecer inicio de sesión con credenciales de ID de usuario/contraseña o permitir los escenarios de trae tu propia identidad disponibles a través del [inicio de sesión social](https://auth0.com/learn/social-login/). Centralizar la experiencia de inicio de sesión con <Tooltip tip="Universal Login: Tu aplicación redirige al Universal Login, alojado en el servidor de autorización de Auth0, para verificar la identidad de un usuario." cta="Ver glosario" href="/docs/es/glossary?term=Universal+Login">Universal Login</Tooltip> también aporta beneficios en términos de reconocimiento de marca, incluso si prevés que tendrás requisitos de [branding](/docs/es/get-started/architecture-scenarios/business-to-consumer/branding) específicos para cada producto. Los widgets de interfaz de usuario de Auth0 que se utilizan habitualmente con Universal Login también ofrecen compatibilidad lista para usar con la [internacionalización](/docs/es/customize/internationalization-and-localization/universal-login-internationalization) para usuarios con distintos requisitos de idioma, y la compatibilidad lista para usar con funcionalidades de Auth0 como [MFA](#multi-factor-authentication-mfa-) y [protección contra ataques](#anomaly-detection) te permite establecer barreras para impedir que los hackers intenten acceder a las cuentas de tus usuarios.

Permitir que los usuarios inicien sesión con credenciales de ID de usuario/contraseña significa que no dependes del estado de los <Tooltip tip="Proveedor de identidad (IdP): Servicio que almacena y gestiona identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=identity_providers">proveedores de identidad</Tooltip> de terceros para que tus usuarios accedan a tu sistema. Además, tienes la posibilidad de exigir que las credenciales utilizadas se ajusten a tus políticas corporativas. Auth0 te ayuda con esto al ofrecerte múltiples opciones para los inicios de sesión con ID de usuario/contraseña, y la [guía proporcionada](#username-and-password-authentication) te ayudará a entender cómo puedes aprovechar estas opciones. Agregar compatibilidad con [redes sociales](#social-authentication) en algún momento, como factor de autenticación primario adicional, te brinda mayor flexibilidad y puede ayudarte a conocer mejor a tus usuarios sin necesidad de hacerles más preguntas, aprovechando la información que ya almacenan los distintos [proveedores](/docs/es/authenticate/identity-providers/social-identity-providers) de inicio de sesión social.

Si tiene un repositorio de identidades heredado, también le convendrá consultar [Migración de usuarios](/docs/es/get-started/architecture-scenarios/business-to-consumer/provisioning). En esta sección se analizan las ventajas de migrar al almacenamiento de identidades administrado por Auth0 en términos de protección y seguridad.

Para las aplicaciones orientadas al cliente, <Tooltip tip="OpenID: Estándar abierto de autenticación que permite a las aplicaciones verificar la identidad de los usuarios sin recopilar ni almacenar información de inicio de sesión." cta="Ver glosario" href="/docs/es/glossary?term=OpenID">OpenID</Tooltip> Connect ([OIDC](/docs/es/authenticate/protocols/openid-connect-protocol)) es el protocolo estándar del sector más utilizado, y OIDC cuenta con soporte de primer nivel en Auth0. Auth0 ofrece varios enfoques para integrar distintos tipos de aplicaciones, por lo que le convendrá consultar la sección sobre [integración de aplicaciones](#application-integration) para obtener la información necesaria y tomar una decisión fundamentada.

Al llamar a una API desde otra API, o en cualquier situación en la que no haya contexto de usuario autenticado, como una o varias tareas cron, generadores de informes o sistemas de integración/entrega continuas, necesitará una forma de autorizar la aplicación en lugar de a un usuario. Este es un proceso de un solo paso en el que la aplicación se autentica (mediante un <Tooltip tip="ID de cliente: Valor de identificación proporcionado por Auth0 a su recurso registrado." cta="Ver glosario" href="/docs/es/glossary?term=client+ID">ID de cliente</Tooltip> y un secreto) y luego se autoriza en una sola llamada. Puede obtener más información sobre esto en nuestro flujo de trabajo de autorización, en [autorización de máquina a máquina (m2m)](/docs/es/get-started/architecture-scenarios/business-to-business/authorization).

A menudo, las empresas necesitan separar a sus usuarios por organización y, en ocasiones, los usuarios pueden tener acceso a más de una organización. Saber cuál de estos escenarios es relevante para su empresa ayudará a definir cómo determinar en qué conexión existe un usuario: si necesita hacerlo, cuándo debe hacerlo y cómo hacerlo. Consulte [Home Realm Discovery](#home-realm-discovery) para determinar si esto es relevante para su empresa.

<div id="universal-login">
  ## Universal Login
</div>

¿Tiene o tendrá más de una aplicación en su sistema? Si la respuesta es sí, le convendrá contar con una experiencia de inicio de sesión centralizada. Para lograr una experiencia fluida de <Tooltip tip="Inicio de sesión único (SSO): servicio que, después de que un usuario inicia sesión en una aplicación, inicia automáticamente la sesión de ese usuario en otras aplicaciones." cta="Ver glosario" href="/docs/es/glossary?term=Single+Sign-on">inicio de sesión único</Tooltip> (SSO) entre varias aplicaciones, es fundamental contar con un lugar centralizado al que redirigir a los usuarios para la autenticación. Esto le permite ofrecerles una experiencia coherente si en el futuro agrega autenticación social, incorpora aplicaciones de terceros a su sistema o añade autenticación multifactor como opción (o requisito), y además aprovechar nuevas funcionalidades para mejorar la experiencia de sus usuarios con poco o ningún esfuerzo adicional de desarrollo.

<Info>
  ### Práctica recomendada

  Si tiene más de una aplicación, la práctica recomendada es redirigir a una [ubicación centralizada](/docs/es/authenticate/login/auth0-universal-login) para autenticar al usuario. Con Auth0, esto significa aprovechar [Universal Login](/docs/es/authenticate/login/auth0-universal-login), que ofrece de forma inmediata numerosas ventajas de seguridad y experiencia de usuario, incluido el [SSO](/docs/es/authenticate/single-sign-on).
</Info>

Auth0 Universal Login hace que la autenticación de usuarios sea un proceso breve y sencillo que puede completarse en tres pasos fáciles (todos nuestros Quickstarts lo demuestran y nuestros SDKs también ocultan la complejidad):

1. Determine cómo y cuándo desea [redirigir desde su aplicación](#application-integration).
2. Configure el [branding](/docs/es/get-started/architecture-scenarios/business-to-consumer/branding) adecuado y/o el HTML personalizado en su configuración de Auth0.
3. Configure su aplicación para [recibir y procesar la respuesta](#application-integration) del servidor de autorización.

<div id="home-realm-discovery">
  ## Home Realm Discovery
</div>

Home Realm Discovery (HRD) es el proceso de identificar a qué proveedor de identidad (o a qué conexión en Auth0) pertenece el usuario antes de autenticarlo. Hay dos formas en que puede producirse el HRD:

* Ofrecer una forma de tomar la decisión en la aplicación
* Hacer que el Home Realm Discovery se produzca en la página de Universal Login

Es posible que tu sistema necesite usar uno o ambos métodos, por lo que es importante comprender todos los enfoques del HRD para que puedas aplicar el que mejor se adapte a tus aplicaciones.

<div id="application-driven-hrd">
  ### HRD basado en la aplicación
</div>

Una forma habitual de determinar a qué `realm` pertenece un usuario es cuando una aplicación está personalizada para cada organización. Como parte de esto, una organización normalmente tendrá su propia instancia de la aplicación, a la que por lo general se accede mediante una URL diferente. Esta copia o instancia puede estar aislada físicamente (ejecutándose en un conjunto independiente de servidores) o aislada virtualmente (ejecutándose en servidores compartidos) y, por lo general, se identifica mediante un nombre de host personalizado (`companyA.application1.yourcompany.com`) o una ruta (`application1.yourcompany.com/companyA`).

<Info>
  ### Práctica recomendada

  Si tu aplicación ya conoce la conexión (IdP) específica que se requiere, también puedes enviarla al redirigir al usuario a `/authorize` mediante el parámetro de consulta `connection`.
</Info>

Si este es el caso de tu aplicación o aplicaciones, entonces Home Realm Discovery consiste simplemente en almacenar el `org_id` en la configuración de la aplicación específica de la organización y enviarlo como parámetro de organización al redirigir al usuario a Universal Login. Al hacerlo, el usuario quedará limitado a esa organización específica y a su conjunto de conexiones configuradas.

<Info>
  ### Práctica recomendada

  Si una organización necesita más de un IdP, es posible que tengas que hacer una segunda ronda de Home Realm Discovery (una vez identificada la organización). Auth0 normalmente gestionará esto por ti si estás usando Organizations Feature.
</Info>

Si el usuario se comparte entre varias organizaciones usando una sola identidad, debes admitir [Home Realm Discovery en la página de Universal Login](#hrd-through-universal-login). Esto permitirá que Universal Login primero identifique al usuario y luego presente los `realms` adecuados para ese usuario, lo que dará lugar a una mejor experiencia de usuario.

Enviar los parámetros de organización y/o conexión puede lograrse agregándolos como parámetros de consulta al redirigir al usuario al `endpoint` `/authorize`. Para obtener más información, consulta la [documentación de Authentication API](https://auth0.com/docs/api/authentication#authorization-code-flow). Sin embargo, por lo general esto se hace usando el SDK del lenguaje en el que esté escrita tu aplicación.

<div id="hrd-through-universal-login">
  ### HRD mediante Universal Login
</div>

Hay tres enfoques principales para Home Realm Discovery:

* Descubrir el realm a través del subdominio del correo electrónico del usuario.
* Descubrir el realm buscando un identificador de usuario en algún tipo de mapa que relacione identificadores con realms.
* Permitir que el usuario elija o introduzca su realm (u organización).

En los dos primeros enfoques, puede considerar el uso de “Identifier First Login”. Esto significa que primero se presenta únicamente la opción de introducir un identificador. Después, se recopila el identificador del usuario y, en función de este, se redirige automáticamente al usuario o se le ofrece una forma de introducir su contraseña si la redirección no es necesaria. Auth0 proporciona una implementación lista para usar que se encarga de todo esto mediante [Universal Login](/docs/es/authenticate/login/auth0-universal-login/identifier-first).

<Warning>
  Aunque es posible implementar Identifier First Login o permitir que un usuario seleccione su organización en la aplicación en lugar de hacerlo en la página de Universal Login, esto puede añadir complejidad al inicio de sesión único, así como complicar la replicación de ese comportamiento en todas sus aplicaciones. En su lugar, Auth0 recomienda implementar algún tipo de HRD mediante Universal Login.
</Warning>

<div id="hrd-through-universal-login-using-the-email-subdomain">
  #### HRD mediante Universal Login usando el subdominio del correo electrónico
</div>

La forma más sencilla de implementar Home Realm Discovery en la página de Universal Login es usar el subdominio del correo electrónico del identificador del usuario para asociarlo con su proveedor de identidad. Esto, por supuesto, solo funciona cuando el subdominio del correo electrónico se corresponde de forma unívoca con una organización o, al menos, con un proveedor de identidad.

El widget [Lock](/docs/es/libraries/lock) de Auth0 o [Universal Login](/docs/es/authenticate/login/auth0-universal-login/universal-login-vs-classic-login/universal-experience) pueden hacerlo por ti si estás usando el mapa de dominios en una conexión empresarial; sin embargo, si quieres implementarlo tú mismo, puedes hacerlo, pero tendrás que crear un mapeo entre el subdominio del correo electrónico y la conexión.

Además, al usar la experiencia de Universal Login, la [Organizations Feature](/docs/es/manage-users/organizations/organizations-overview) te ayudará de dos maneras:

1. Si no has pasado un org&#95;id a `/authorize` al iniciar la solicitud de login, Auth0 mostrará al usuario un prompt para que introduzca la organización a la que pertenece.
2. Si la organización tiene más de un IdP asociado, Auth0 mostrará al usuario botones para elegir la organización o un formulario para introducir un nombre de usuario y una contraseña.

<div id="hrd-through-universal-login-using-the-identifier-to-realm-map">
  #### HRD mediante Universal Login usando el mapa de identificadores a realm
</div>

Una segunda alternativa, más compleja, consiste en almacenar un mapa de identificadores a IdP y proporcionar un endpoint público para acceder a esa información. Luego, en la página de Universal Login, puedes encontrar la conexión y volver a redirigir a /authorize con la conexión. Los principales inconvenientes de este enfoque son la latencia y, más importante aún, la seguridad en lo relativo al descubrimiento de identificadores: si usas direcciones de correo electrónico, esto facilita mucho que alguien averigüe si una dirección de correo electrónico concreta pertenece a uno de tus usuarios.

<Info>
  ### Práctica recomendada

  Todo endpoint público debe tener limitación de frecuencia para evitar que los atacantes lo usen para descubrir información y prevenir ataques de denegación de servicio.
</Info>

<div id="hrd-through-universal-login-using-user-choice">
  #### HRD a través de Universal Login según la elección del usuario
</div>

La otra opción es permitir que los usuarios elijan entre una lista, si no te importa hacer pública la lista de organizaciones que usan tu producto, o que introduzcan explícitamente el nombre de su organización. Normalmente, esto se haría antes de redirigir al usuario a Universal Login. Una vez que el usuario te indique a qué organización pertenece, puedes redirigir a Auth0 especificando la conexión de esa organización o hacer que Auth0 simplemente le solicite su nombre de usuario y contraseña si se trata de una conexión de base de datos.

<div id="username-and-password-authentication">
  ## Autenticación con nombre de usuario y contraseña
</div>

Casi todas las aplicaciones B2C permiten a sus clientes crear un nuevo conjunto de credenciales. Esta es una forma común de autenticación con la que todos los usuarios están familiarizados.

La autenticación con nombre de usuario y contraseña tiene varias modalidades en Auth0. Si su aplicación es nueva y no cuenta con una base de usuarios existente, una [conexión de base de datos](/docs/es/authenticate/database-connections) de Auth0 lista para usar le dará todo lo que necesita para empezar a autenticar a sus usuarios. Sin embargo, si tiene un repositorio de usuarios heredado (como su propia base de datos de usuarios o un sistema LDAP existente), dispone de un par de opciones para migrar a sus usuarios, como se explica en nuestra guía sobre [Migración de usuarios](/docs/es/get-started/architecture-scenarios/business-to-business/provisioning).

Independientemente de cómo termine aprovisionando a los usuarios para su conexión de base de datos, la autenticación de esos usuarios es bastante similar. Requiere presentarles un formulario para que introduzcan su nombre de usuario y contraseña. Como se menciona en la guía sobre [Universal Login](#universal-login), la forma más sencilla y segura de autenticar a los usuarios con un nombre de usuario y una contraseña es redirigirlos a una página de inicio de sesión centralizada y recopilar allí su nombre de usuario y contraseña. Esto permite que Auth0 determine si ya se han autenticado y omita por completo el formulario de inicio de sesión cuando no sea necesario.

<Info>
  ### Práctica recomendada

  Recopilar las credenciales solo en la página de inicio de sesión centralizada reducirá la superficie de exposición ante una posible filtración de secretos de los usuarios. También reducirá la necesidad de recopilar credenciales innecesariamente. Consulte [Universal Login](#universal-login) para obtener más información.
</Info>

<div id="application-integration">
  ## Integración de la aplicación
</div>

Una vez que hayas definido cómo quieres autenticar a tus usuarios, el siguiente paso es determinar cómo iniciarás esa autenticación. Por lo general, cada aplicación tendrá su propio punto de partida.

<Warning>
  Las aplicaciones móviles nativas (y las aplicaciones de escritorio) deben usar el navegador del sistema para la autenticación; de lo contrario, se exponen a riesgos de seguridad adicionales. Consulta [Native Login](/docs/es/authenticate/login/native-login) para obtener más información.
</Warning>

Como ya comentamos, hemos comprobado que la mayoría de nuestros clientes usan [OpenID Connect (OIDC)](/docs/es/authenticate/protocols/openid-connect-protocol) como protocolo estándar del sector para sus aplicaciones de cara al cliente. Tu primera tarea es determinar qué flujo de OIDC usar, y te convendrá empezar revisando nuestra guía de [mapeo de concesiones](/docs/es/get-started/applications/application-grant-types).

Si quieres permitir que los usuarios anónimos accedan a cualquier parte de nuestra aplicación, debes determinar si vas a redirigirlos de inmediato o si solo les pedirás que redirijan cuando sea necesario (o quizá una combinación de ambas opciones; consulta [Redirect Users After Login](/docs/es/authenticate/login/redirect-users-after-login) para obtener más información). Si los usuarios pueden usar un [enlace profundo](#deep-linking-to-protected-endpoints) a una versión protegida (o a un área protegida) de tu sitio, tendrás que determinar qué enlaces a tu aplicación darán lugar a una redirección automática a Auth0.

<div id="anonymous-access">
  ### Acceso anónimo
</div>

Es importante tener en cuenta la experiencia del usuario cuando alguien llega por primera vez a tu aplicación. Si tu aplicación admite el acceso de usuarios anónimos (algo bastante común en las aplicaciones de comercio electrónico), hay distintos escenarios que debes considerar:

* Si regresan a la aplicación después de haber iniciado sesión previamente, o
* Si es la primera vez que acceden a la aplicación:

  * Si ya han accedido a otra aplicación que usa el mismo tenant de Auth0,
  * Si alguna vez (o quizás no desde hace mucho tiempo) se autenticaron en este dispositivo o navegador.

Cuando un usuario anónimo accede a tu aplicación, a menudo puede ser conveniente que la aplicación determine si el usuario ya inició sesión en otra aplicación de la misma familia, o que recuerde a este usuario incluso si la aplicación es una [SPA](/docs/es/quickstart/spa) sin estado. Por ejemplo, si puedes determinar que el usuario ya inició sesión, podrías decidir que el encabezado de la interfaz de la aplicación no muestre un botón de inicio de sesión y que, en su lugar, presente un menú de cuenta o perfil para el usuario. Para lograrlo, te convendrá utilizar la &quot;[autenticación silenciosa](/docs/es/authenticate/login/configure-silent-authentication)&quot;. La autenticación silenciosa te permitirá comprobar si el usuario ha iniciado sesión sin pedirle que la inicie si no lo ha hecho. Luego, la aplicación puede mostrar un botón de inicio de sesión si es necesario. Sin embargo, si el usuario ya inició sesión, recibirás tokens y no tendrás que volver a mostrarle un botón de inicio de sesión.

<Warning>
  Comprobar si existe una sesión iniciada redirigiendo a Auth0 puede ser útil para tu aplicación, pero si esto va a generar muchas solicitudes, deberías implementar un mecanismo de limitación para evitar latencia o límite de frecuencia.

  Las llamadas a la Management API están sujetas a la [política de límite de frecuencia de Auth0](/docs/es/troubleshoot/customer-support/operational-policies/rate-limit-policy). Debes tener esto en cuenta y, para ayudarte, Auth0 generalmente recomienda usar el [SDK de Auth0](/docs/es/libraries) adecuado para tu entorno de desarrollo en lugar de llamar directamente a nuestras API.
</Warning>

<div id="deep-linking-to-protected-endpoints">
  ### Enlaces profundos a endpoints protegidos
</div>

Hay diversos motivos por los que alguien podría enlazar directamente a una página concreta dentro de tu aplicación a la que solo pueden acceder usuarios autenticados. Si esto es posible en tu aplicación, deberías redirigir automáticamente al usuario a Auth0 si no está autenticado. Una vez que se autentique y el <Tooltip tip="Servidor de autorización: Servidor centralizado que contribuye a definir los límites del acceso de un usuario. Por ejemplo, tu servidor de autorización puede controlar los datos, las tareas y las funcionalidades disponibles para un usuario." cta="Ver glosario" href="/docs/es/glossary?term=authorization+server">servidor de autorización</Tooltip> lo devuelva a tu aplicación, puedes [redirigirlo](/docs/es/authenticate/login/redirect-users-after-login) al lugar al que quería ir desde el principio.

<div id="best-practice">
  ### Práctica recomendada
</div>

La mayoría de los frameworks modernos de autenticación admiten middleware para redirigir a un servidor de autorización como Auth0. Al elegir uno, estas son algunas consideraciones clave:

* Compatibilidad con clientes confidenciales, clientes no confidenciales o ambos
* Compatibilidad con la configuración mediante [endpoint de descubrimiento](/docs/es/get-started/applications/configure-applications-with-oidc-discovery "Configurar Applications con OIDC Discovery") o de forma explícita e inline
* Compatibilidad con la validación de tokens, incluidas la expiración, las firmas, los claims y los scopes
* Compatibilidad con <Tooltip tip="Token de actualización: token utilizado para obtener un token de acceso renovado sin obligar a los usuarios a volver a iniciar sesión." cta="Ver glosario" href="/docs/es/glossary?term=Refresh+Tokens">tokens de actualización</Tooltip> si es necesario

<div id="authenticating-the-user">
  ### Autenticación del usuario
</div>

La autenticación es el proceso de determinar la identidad del usuario. El resultado de la autenticación en un contexto de OIDC es un <Tooltip tip="token de ID: credencial destinada al propio cliente, en lugar de usarse para acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=ID+Token">token de ID</Tooltip>. Este token contiene información sobre el usuario y solo debería poder obtenerse si el usuario se autentica mediante uno o más factores, según lo definido por el servidor de autorización (la forma más común es el [ID de usuario y contraseña](#username-and-password-authentication)). Además de obtener un token de ID, quizá también deba tener en cuenta lo siguiente:

* ¿También necesitamos un [token de acceso](/docs/es/secure/tokens/access-tokens) para llamar a una API compartida?
* ¿Su aplicación es una aplicación de página única y solo requiere un [token de ID](/docs/es/secure/tokens/id-tokens)? Consulte [concesión Authorization Code con PKCE](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce/call-your-api-using-the-authorization-code-flow-with-pkce) para obtener más información.
* ¿Su aplicación es una aplicación nativa (móvil o de escritorio) y/o necesita un [token de actualización](/docs/es/secure/tokens/refresh-tokens)? Consulte [concesión Authorization Code con PKCE](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce/call-your-api-using-the-authorization-code-flow-with-pkce) para obtener más información.

<Warning>
  Antes de pasar a producción, debe asegurarse de que **solo** estén habilitadas las concesiones que usa para cada aplicación en la [configuración de su aplicación](/docs/es/get-started/applications/update-grant-types).
</Warning>

<div id="authorization-code-grant-with-or-without-pkce">
  ### Concesión Authorization Code (con o sin PKCE)
</div>

Si tu SDK solo admite la concesión Authorization Code, o necesitas un token de acceso o un <Tooltip tip="Token de actualización: token usado para obtener un token de acceso renovado sin obligar a los usuarios a iniciar sesión de nuevo." cta="Ver glosario" href="/docs/es/glossary?term=Refresh+Token">token de actualización</Tooltip>, la concesión Authorization Code (con o sin [PKCE](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce)) también puede utilizarse para obtener un token de ID. La concesión Authorization Code incluye una llamada adicional a la API para intercambiar el código por un token, lo que puede generar una latencia adicional innecesaria si lo único que necesitas es el token de ID. En muchos casos, se implementa el [flujo híbrido](/docs/es/get-started/authentication-and-authorization-flow/hybrid-flow) para proporcionar un acceso óptimo al token de ID y, al mismo tiempo, aprovechar el flujo de trabajo de la concesión Authorization Code para obtener de forma segura tokens de acceso y tokens de actualización.

<Warning>
  Aunque Auth0 admite el uso de la concesión implícita para aplicaciones basadas en navegador que solo requieren un token de ID, se recomienda usar la concesión Authorization Code con [PKCE](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce). Para obtener información detallada, consulta [OAuth2 Implicit Grant and SPA](https://auth0.com/blog/oauth2-implicit-grant-and-spa/) en el blog de Auth0.

  Si necesitas un [token de actualización](/docs/es/secure/tokens/refresh-tokens) para poder obtener un nuevo token de acceso o token de ID sin tener que volver a autenticar al usuario, debes usar la [concesión Authorization Code](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow/call-your-api-using-the-authorization-code-flow).
</Warning>

<div id="attack-protection">
  ## Protección contra ataques
</div>

La razón por la que los sistemas de autenticación son importantes es evitar que los <Tooltip tip="Actores maliciosos: Entidad (una persona o grupo) que representa una amenaza para el negocio o el entorno con la intención de causar daño." cta="Ver glosario" href="/docs/es/glossary?term=bad+actors">actores maliciosos</Tooltip> accedan a aplicaciones y datos de usuario a los que no deberían tener acceso. Queremos poner tantas barreras como sea posible entre esos actores maliciosos y el acceso a nuestros sistemas. Una de las formas más sencillas de hacerlo es asegurarte de que tu [protección contra ataques](/docs/es/secure/attack-protection) con Auth0 esté configurada correctamente, así que tómate un momento para leer la guía sobre este tema y comprobar que todo funcione correctamente.

<Info>
  ### Práctica recomendada

  Auth0 gestiona la detección de anomalías de forma transparente y ofrece una excelente funcionalidad de seguridad para tu producto. Si vas a utilizarla, asegúrate de haber configurado tu [proveedor de correo electrónico](/docs/es/get-started/architecture-scenarios/business-to-business/operations#email-provider-setup) y tus [Plantillas de correo electrónico](/docs/es/get-started/architecture-scenarios/business-to-business/branding#email-template-customization) antes de activar la entrega de correos electrónicos a tus usuarios.
</Info>

<div id="sso-with-legacy-systems">
  ## SSO con sistemas heredados
</div>

En una reestructuración a gran escala, no siempre es posible —o práctico— actualizar todas sus aplicaciones de una sola vez. De hecho, nuestra práctica recomendada es adoptar un enfoque iterativo al integrarse con Auth0. Si sus aplicaciones ya participan en un inicio de sesión único (SSO) y su sistema de identidad heredado admite protocolos como OIDC o <Tooltip tip="Security Assertion Markup Language (SAML): Protocolo estandarizado que permite a dos partes intercambiar información de autenticación sin una contraseña." cta="Ver glosario" href="/docs/es/glossary?term=SAML">SAML</Tooltip>, tiene un par de opciones si desea seguir ofreciendo SSO mientras se integra con Auth0:

* Actualizar el proveedor de identidad existente en su sistema de SSO heredado para que redirija a Auth0 para el inicio de sesión (p. ej., usando [SAML](/docs/es/authenticate/single-sign-on/outbound-single-sign-on/configure-auth0-saml-identity-provider)), o
* Hacer que Auth0 redirija a su sistema de SSO heredado para el inicio de sesión. Esto requiere configurar su sistema heredado como un IdP en Auth0 (es decir, mediante [SAML](/docs/es/authenticate/protocols/saml/saml-sso-integrations/configure-auth0-saml-service-provider) o [OIDC](/docs/es/authenticate/identity-providers/social-identity-providers)).

<Info>
  ### Práctica recomendada

  Mantener una experiencia de SSO con su sistema heredado puede añadir complejidad, pero puede valer la pena para ofrecer una experiencia de usuario más fluida mientras se integra con Auth0. Si piensa seguir este camino, planificarlo desde el principio puede ayudar a garantizar que sea viable. Si aún no cuenta con SSO en un servicio centralizado, es poco probable que la complejidad de añadirlo compense los beneficios.
</Info>

Este es un tema complejo que probablemente requiera investigación adicional según su arquitectura heredada actual, y le recomendamos que lo evalúe solo si su sistema heredado ya admite SSO. Nota: si actualmente redirige desde sus aplicaciones a un sistema centralizado para autenticar a sus usuarios y ese sistema solo solicita credenciales si todavía no tiene una sesión con el sistema centralizado, entonces tiene una implementación heredada de SSO.

<div id="enterprise-login">
  ## Inicio de sesión empresarial
</div>

El escenario de “trae tu propia identidad” se ha convertido en un requisito indispensable para casi todas las aplicaciones B2B. La mayoría de las empresas esperan poder integrar su IdP en tu aplicación para que sus empleados no tengan que almacenar otro conjunto de credenciales. Esta es una forma muy útil de simplificar la experiencia de autenticación de usuario sin comprometer la seguridad, y usar [Universal Login](#universal-login) facilita empezar a agregar soporte para [Conexiones empresariales](/docs/es/authenticate/identity-providers/enterprise-identity-providers) con una interrupción mínima.

<Info>
  ### Práctica recomendada

  Una vez que empieces a admitir conexiones empresariales para los usuarios, debes implementar algún tipo de [Home Realm Discovery](#home-realm-discovery) para determinar a qué conexión enviar al usuario para autenticarse.
</Info>

Con soporte para conexiones empresariales, las identidades y credenciales de los usuarios son administradas por el proveedor de identidad de la organización de tus clientes, así como determinados claims de identidad, que Auth0 usará para completar el [perfil](/docs/es/get-started/architecture-scenarios/business-to-business/profile-management) del usuario.

<Info>
  ### Práctica recomendada

  “Trae tu propia identidad” es una excelente funcionalidad para ofrecer, pero si no la admites desde el primer día, y a veces incluso si lo haces, puede que tengas una organización que quiera cambiar a su propio IdP después de haber usado la aplicación durante un tiempo. Necesitarás una forma de [vincular cuentas de usuario](/docs/es/manage-users/user-accounts/user-account-linking) para asociar eficazmente la nueva identidad con la antigua identidad de base de datos.
</Info>

<div id="multi-factor-authentication-mfa">
  ## Autenticación multifactor (MFA)
</div>

En una época en la que el uso indebido de las credenciales de usuario está en niveles sin precedentes, proteger sus sistemas supone todo un desafío, especialmente cuando es tan habitual que los atacantes roben la información de identidad de los usuarios. Sin embargo, una de las formas más eficaces de hacerlo es ofrecer a los usuarios la posibilidad de configurar un segundo factor para proteger su cuenta, lo que se conoce más comúnmente como [autenticación multifactor](/docs/es/secure/multi-factor-authentication). Esto garantiza que solo un usuario legítimo pueda acceder a su cuenta, incluso si utiliza un nombre de usuario y una contraseña que se hayan visto comprometidos en otra aplicación.

<Info>
  ### Práctica recomendada

  Es bastante habitual que las aplicaciones orientadas al cliente ofrezcan a los usuarios la *opción* de añadir un segundo factor en lugar de *obligarlos* a usarlo. Para obtener más información al respecto, consulte [ofrecer a sus usuarios la opción de añadir MFA](https://auth0.com/learn/multifactor-authentication-customers/).
</Info>

Auth0 admite varias opciones para habilitar MFA y proteger el acceso a las cuentas de usuario, y existen diversas prácticas para asegurarse de ofrecer realmente una barrera de segundo factor flexible para el acceso:

* Auth0 [Guardian](https://auth0.com/multifactor-authentication): un servicio que proporciona tanto la generación de notificaciones push como una aplicación para permitir o denegar solicitudes. Push envía notificaciones al device prerregistrado del usuario —normalmente un teléfono móvil o una tableta—, desde el cual puede permitir o denegar inmediatamente el acceso a la cuenta con solo pulsar un botón.
* Contraseña de un solo uso basada en tiempo ([TOTP](https://auth0.com/blog/from-theory-to-practice-adding-two-factor-to-node-dot-js/)): le permite registrar un device, como Google Authenticator, que generará una OTP que cambia con el tiempo y que puede introducirse como segundo factor para validar la cuenta de un usuario.
* SMS: para enviar un código de un solo uso por SMS que luego se pide al usuario que introduzca antes de poder completar la autenticación.
* Voz: para entregar un código de un solo uso mediante una llamada telefónica que luego se pide al usuario que introduzca antes de poder completar la autenticación.
* Duo: le permite usar su cuenta de Duo para la autenticación multifactor.
* Email: le permite usar su cuenta de correo electrónico para la autenticación multifactor.

Aunque el workflow de MFA que utiliza tecnologías como Guardian o Google Authenticator normalmente se proporciona a través de una aplicación independiente que se ejecuta en un dispositivo móvil o una tableta, si no quiere que sus clientes tengan que descargar una aplicación independiente, Auth0 también le proporciona un [SDK](https://auth0.com/blog/announcing-guardian-whitelabel-sdk/) que puede usar para crear el workflow de segundo factor directamente en sus aplicaciones móviles existentes.
