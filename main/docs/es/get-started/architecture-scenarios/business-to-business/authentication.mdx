Para poder prestar servicios a tus usuarios, debes poder identificar quiénes son. Este proceso se llama autenticación de usuario. Hay varias formas de autenticar a un usuario: mediante cuentas de redes sociales, nombre de usuario y contraseña, <Tooltip tip="Passwordless: forma de autenticación que no depende de una contraseña como primer factor." cta="Ver glosario" href="/docs/es/glossary?term=passwordless">passwordless</Tooltip> y, a menudo, se recomienda ir más allá de un primer factor para autenticar al usuario habilitando <Tooltip tip="Passwordless: forma de autenticación que no depende de una contraseña como primer factor." cta="Ver glosario" href="/docs/es/glossary?term=multi-factor+authentication">autenticación multifactor</Tooltip> (MFA).

<Info>
  ### Práctica recomendada

  Es importante tener en cuenta tanto la seguridad como la experiencia de usuario al diseñar cómo vas a autenticar a tus usuarios. Ofrecer varios factores primarios y/o exigir más de un factor durante la autenticación son formas de lograr ambas cosas.
</Info>

Hay varias cosas que querrás tener en cuenta al evaluar la funcionalidad y el flujo de trabajo:

* ¿Dónde introducirán los usuarios sus credenciales?
* ¿Cómo mantendrás seguras las credenciales de los usuarios?
* ¿Cómo mantendrás tu sistema de autenticación?
* ¿Cómo puedes ofrecer autenticación con contraseña a tus usuarios?
* ¿Cómo puedes evitar que los hackers intenten iniciar sesión haciéndose pasar por tus usuarios?
* ¿Cómo implementarás la autenticación en distintos tipos de aplicaciones?
* ¿Cómo puedes facilitar el inicio de sesión a tus usuarios cuando provienen de distintos contextos lingüísticos?
* ¿Cómo ofrecerás una buena experiencia de usuario mientras abandonas cualquier sistema de autenticación heredado?
* ¿Qué debes tener en cuenta al integrar aplicaciones con Auth0?
* ¿Necesitas ofrecer autenticación multifactor?
* ¿Qué haces si tienes un servicio que no ofrece una forma de que el usuario inicie sesión de antemano?
* ¿Puedes pasar el mismo <Tooltip tip="token de acceso: credencial de autorización, en forma de una cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=access+token">token de acceso</Tooltip> de usuario de una API a otra?
* ¿Qué haces si necesitas aislar a los usuarios por organización?
* ¿Cómo gestionarás la identificación de la organización a la que pertenecen los usuarios?
* ¿Cuál es la ventaja de ofrecer conexiones empresariales para tus organizaciones?

Auth0 [Universal Login](#universal-login) proporciona a los usuarios una experiencia segura y protegida, tanto si eliges ofrecer inicio de sesión con credenciales de ID de usuario y contraseña como si permites los llamados escenarios de trae tu propia identidad mediante [Social Login](https://auth0.com/learn/social-login/). También hay ventajas de reconocimiento de marca al centralizar la experiencia de inicio de sesión con <Tooltip tip="Universal Login: tu aplicación redirige a Universal Login, alojado en el servidor de autorización de Auth0, para verificar la identidad de un usuario." cta="Ver glosario" href="/docs/es/glossary?term=Universal+Login">Universal Login</Tooltip>, incluso si consideras que también tendrás requisitos de [branding](/docs/es/get-started/architecture-scenarios/business-to-consumer/branding) específicos del producto. Los widgets de interfaz de usuario de Auth0 que se usan normalmente con Universal Login también ofrecen soporte inmediato en lo que respecta a la [internacionalización](/docs/es/customize/internationalization-and-localization/universal-login-internationalization) para usuarios con distintos requisitos de idioma, y el soporte inmediato para funciones de Auth0 como [MFA](#multi-factor-authentication-mfa-) y [protección contra ataques](#anomaly-detection) te permite establecer barreras para impedir que hackers intenten acceder a las cuentas de los usuarios.

Permitir que los usuarios inicien sesión mediante credenciales de ID de usuario y contraseña significa que no dependes del estado de <Tooltip tip="Identity Provider (IdP): servicio que almacena y gestiona identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=identity+providers">proveedores de identidad</Tooltip> de terceros para que tus usuarios accedan a tu sistema. También puedes exigir que las credenciales utilizadas se ajusten a tus políticas corporativas. Auth0 te ayuda con esto al ofrecerte múltiples opciones para el inicio de sesión con ID de usuario y contraseña, y la [guía proporcionada](#username-and-password-authentication) te ayudará a entender cómo puedes aprovechar estas opciones. Añadir compatibilidad [social](#social-authentication) en algún momento, como factor primario de autenticación adicional, te da más flexibilidad y puede ayudarte a conocer mejor a tus usuarios sin necesidad de hacerles más preguntas, aprovechando la información que ya almacenan los distintos [proveedores](/docs/es/authenticate/identity-providers/social-identity-providers) de inicio de sesión social.

Si ya tiene un repositorio de identidades heredado, también le convendrá consultar [migración de usuario](/docs/es/get-started/architecture-scenarios/business-to-consumer/provisioning). En esta sección se analizan las ventajas de migrar al almacenamiento de identidades administrado por Auth0 en términos de protección y seguridad.

Para las aplicaciones orientadas al cliente, <Tooltip tip="OpenID: Estándar abierto de autenticación que permite a las aplicaciones verificar la identidad de los usuarios sin recopilar ni almacenar información de inicio de sesión." cta="Ver glosario" href="/docs/es/glossary?term=OpenID">OpenID</Tooltip> Connect ([OIDC](/docs/es/authenticate/protocols/openid-connect-protocol)) es el protocolo estándar del sector más utilizado, y OIDC cuenta con soporte de primer nivel en Auth0. Auth0 ofrece varias maneras de integrar distintos tipos de aplicaciones, por lo que le convendrá consultar la sección sobre [integración de aplicaciones](#application-integration) para obtener la información que necesita y tomar una decisión informada.

Al llamar a una API desde otra API, o en cualquier situación en la que no haya contexto de usuario autenticado, como una o más tareas cron, generadores de informes o sistemas de integración/entrega continua, necesitará una forma de autorizar la aplicación en lugar de un usuario. Este es un proceso de un solo paso en el que la aplicación se autentica (mediante un <Tooltip tip="Client ID: Valor de identificación asignado a su recurso registrado por Auth0." cta="Ver glosario" href="/docs/es/glossary?term=client+ID">ID de cliente</Tooltip> y un secreto) y luego se autoriza en una sola llamada. Puede obtener más información sobre esto en nuestro flujo de trabajo de autorización en [autorización de máquina a máquina (m2m)](/docs/es/get-started/architecture-scenarios/business-to-business/authorization).

A menudo, las empresas necesitan separar a sus usuarios por organización y, en ocasiones, los usuarios pueden tener acceso a más de una organización. Saber cuál de estos escenarios es relevante para su empresa ayudará a definir cómo determinar en qué conexión existe un usuario: si necesita hacerlo, cuándo debe hacerlo y cómo lograrlo. Consulte [Home Realm Discovery](#home-realm-discovery) para determinar si esto es relevante para su empresa.

<div id="universal-login">
  ## Universal Login
</div>

¿Tiene o tendrá más de una aplicación en su sistema? Si la respuesta es sí, le convendrá contar con una experiencia de inicio de sesión centralizada. Para lograr una experiencia fluida de <Tooltip tip="Inicio de sesión único (SSO): servicio que, después de que un usuario inicia sesión en una aplicación, inicia automáticamente la sesión de ese usuario en otras aplicaciones." cta="Ver glosario" href="/docs/es/glossary?term=Single+Sign-on">inicio de sesión único</Tooltip> (SSO) entre varias aplicaciones, es fundamental contar con una ubicación centralizada a la que redirigir a sus usuarios para la autenticación. Esto le permite ofrecerles una experiencia coherente si en el futuro agrega autenticación social, incorpora aplicaciones de terceros a su sistema o añade autenticación multifactor como opción (o requisito), además de aprovechar nuevas funcionalidades para mejorar la experiencia de sus usuarios con poco o ningún esfuerzo adicional de desarrollo.

<Info>
  ### Práctica recomendada

  Si tiene más de una aplicación, la práctica recomendada es redirigir a una [ubicación centralizada](/docs/es/authenticate/login/auth0-universal-login) para autenticar al usuario. Con Auth0, esto significa aprovechar [Universal Login](/docs/es/authenticate/login/auth0-universal-login), que ofrece numerosas ventajas de seguridad y experiencia de usuario listas para usar, incluido el [inicio de sesión único](/docs/es/authenticate/single-sign-on).
</Info>

Auth0 Universal Login hace que autenticar usuarios sea un proceso breve y sencillo que puede completarse en tres pasos simples (todos nuestros Quickstarts lo demuestran y nuestros SDKs también ocultan la complejidad por usted):

1. Determine cómo y cuándo desea [redirigir desde su aplicación](#application-integration).
2. Configure el [branding](/docs/es/get-started/architecture-scenarios/business-to-consumer/branding) adecuado y/o HTML personalizado en su configuración de Auth0.
3. Configure su aplicación para [recibir y gestionar la respuesta](#application-integration) del servidor de autorización.

<div id="home-realm-discovery">
  ## Home Realm Discovery
</div>

Home Realm Discovery (HRD) es el proceso de identificar a qué proveedor de identidad (o a qué conexión en Auth0) pertenece el usuario antes de autenticarlo. Hay dos formas en que puede producirse el HRD:

* Ofrecer una manera de tomar esta decisión en la aplicación
* Hacer que Home Realm Discovery tenga lugar en la página de Universal Login

Es posible que su sistema necesite utilizar uno o ambos métodos, por lo que es importante comprender todos los enfoques de HRD para poder aplicar el que tenga más sentido para sus aplicaciones.

<div id="application-driven-hrd">
  ### HRD impulsado por la aplicación
</div>

Una forma habitual de determinar a qué realm pertenece un usuario es cuando una aplicación tiene una marca específica para cada organización. Como parte de esto, una organización normalmente tendrá su propia instancia de la aplicación, a la que por lo general se accede mediante una URL diferente. Esta copia o instancia puede estar aislada físicamente (ejecutándose en un conjunto independiente de servidores) o aislada virtualmente (ejecutándose en servidores compartidos) y, por lo general, se identifica mediante un hostname personalizado (`companyA.application1.yourcompany.com`) o una ruta (`application1.yourcompany.com/companyA`).

<Info>
  ### Práctica recomendada

  Si tu aplicación ya conoce la conexión (IdP) específica necesaria, también puedes pasarla al redirigir al usuario a `/authorize` mediante el parámetro de consulta `connection`.
</Info>

Si este es el caso de tu aplicación o aplicaciones, entonces Home Realm Discovery consiste simplemente en almacenar el `org_id` con la configuración de la aplicación específica de la organización y enviarlo como parámetro de organización al redirigir al usuario a Universal Login. Al hacerlo, limitarás al usuario a esa organización específica y a su conjunto de conexiones configuradas.

<Info>
  ### Práctica recomendada

  Si una organización necesita más de un IdP, es posible que debas hacer una segunda ronda de Home Realm Discovery (una vez identificada la organización). Auth0 normalmente gestionará esto por ti si usas Organizations Feature.
</Info>

Si el usuario se comparte entre varias organizaciones usando una sola identidad, debes admitir [Home Realm Discovery en la página de Universal Login](#hrd-through-universal-login). Esto permitirá que Universal Login identifique primero al usuario y luego presente los realms adecuados para ese usuario, lo que ofrece una mejor experiencia de usuario.

Puedes enviar los parámetros de organización y/o conexión añadiéndolos como parámetros de consulta al redirigir al usuario al endpoint `/authorize`. Para más información, consulta la [documentación de Authentication API](https://auth0.com/docs/api/authentication#authorization-code-flow). Sin embargo, por lo general lo harás usando el SDK del lenguaje en que esté escrita tu aplicación.

<div id="hrd-through-universal-login">
  ### HRD a través de Universal Login
</div>

Hay tres enfoques principales para Home Realm Discovery:

* Descubrir el realm a partir del subdominio del correo electrónico del usuario.
* Descubrir el realm buscando un identificador de usuario en algún tipo de mapa que relacione identificadores con realms.
* Permitir que el usuario elija o introduzca su realm (u organización).

En los dos primeros enfoques, puede considerar usar “Identifier First Login”. Esto significa que primero se muestra únicamente la opción de introducir un identificador. Después, se recopila el identificador del usuario y, en función de este, se le redirige automáticamente o se le muestra una forma de introducir su contraseña si la redirección no es necesaria. Auth0 proporciona una implementación lista para usar que se encarga de todo esto mediante [Universal Login](/docs/es/authenticate/login/auth0-universal-login/identifier-first).

<Warning>
  Aunque es posible implementar Identifier First Login o permitir que un usuario seleccione su organización en la aplicación en lugar de hacerlo en la Universal Login Page, esto puede añadir complejidad tanto con respecto al inicio de sesión único como a la hora de replicar ese comportamiento en todas sus aplicaciones. En su lugar, Auth0 recomienda implementar alguna forma de HRD a través de Universal Login.
</Warning>

<div id="hrd-through-universal-login-using-the-email-subdomain">
  #### HRD mediante Universal Login usando el subdominio del correo electrónico
</div>

La forma más sencilla de implementar Home Realm Discovery en la página de Universal Login es usar el subdominio del correo electrónico del identificador del usuario para asignarlo a su proveedor de identidad. Esto, por supuesto, solo funciona cuando el subdominio del correo electrónico se corresponde de forma 1:1 con una organización o, al menos, con un proveedor de identidad.

El widget [Lock](/docs/es/libraries/lock) de Auth0 o [Universal Login](/docs/es/authenticate/login/auth0-universal-login/universal-login-vs-classic-login/universal-experience) pueden hacerlo por ti si usas el mapa de dominios en una conexión empresarial; sin embargo, si quieres crear esto tú mismo, puedes hacerlo, pero tendrás que crear un mapeo entre el subdominio del correo electrónico y la conexión.

Además, al usar la experiencia de Universal Login, [Organizations Feature](/docs/es/manage-users/organizations/organizations-overview) te ayudará de dos maneras:

1. Si no has pasado un org&#95;id a `/authorize` al iniciar la solicitud de login, Auth0 mostrará al usuario un prompt para que introduzca la organización a la que pertenece.
2. Si la organización tiene más de un IdP asociado, Auth0 mostrará al usuario botones para elegir la organización o un formulario para introducir un nombre de usuario y una contraseña.

<div id="hrd-through-universal-login-using-the-identifier-to-realm-map">
  #### HRD mediante Universal Login usando el mapeo de identificadores a realm
</div>

Una segunda alternativa, más compleja, consiste en almacenar un mapeo de identificadores a IdP y proporcionar un endpoint público para acceder a esa información. Luego, en la página de Universal Login, puedes identificar la conexión y redirigir de nuevo a /authorize con la conexión. Las principales desventajas de este enfoque son la latencia y, más importante aún, la seguridad en lo que respecta al descubrimiento de identificadores: si usas direcciones de correo electrónico, esto facilita mucho que alguien descubra si una dirección de correo electrónico concreta pertenece a uno de tus usuarios.

<Info>
  ### Práctica recomendada

  A cualquier endpoint público se le debería aplicar un límite de frecuencia para evitar que los atacantes lo utilicen para descubrir información y para prevenir ataques de denegación de servicio.
</Info>

<div id="hrd-through-universal-login-using-user-choice">
  #### HRD a través de Universal Login según la elección del usuario
</div>

La otra opción es permitir que los usuarios elijan de una lista, si no te importa hacer pública la lista de organizaciones que usan tu producto, o que introduzcan explícitamente el nombre de su organización. Normalmente, esto se haría antes de redirigir al usuario a Universal Login. Una vez que el usuario te indique a qué organización pertenece, puedes redirigirlo a Auth0 con la conexión de esa organización especificada, o hacer que Auth0 simplemente le pida su username y Password si la conexión es una conexión de base de datos.

<div id="username-and-password-authentication">
  ## Autenticación con nombre de usuario y contraseña
</div>

Casi todas las aplicaciones B2C ofrecen a sus clientes la posibilidad de crear un nuevo conjunto de credenciales. Esta es una forma común de autenticación que resulta familiar para todos los usuarios.

La autenticación con nombre de usuario y contraseña tiene varias modalidades en Auth0. Si su aplicación es una aplicación nueva sin una base de usuarios existente, una sencilla [Conexión de base de datos](/docs/es/authenticate/database-connections) de Auth0 lista para usar le dará todo lo que necesita para empezar a autenticar a sus usuarios. Sin embargo, si tiene un repositorio de usuarios heredado (como su propia base de datos de usuarios o un sistema LDAP existente), dispone de un par de opciones distintas para migrar sus usuarios, como se explica en nuestra guía sobre [migración de usuario](/docs/es/get-started/architecture-scenarios/business-to-business/provisioning).

Independientemente de cómo aprovisione a los usuarios para su conexión de base de datos, la autenticación de esos usuarios es bastante similar. Requiere mostrarles un formulario para que introduzcan su nombre de usuario y contraseña. Como se menciona en la guía sobre [Universal Login](#universal-login), la forma más sencilla y segura de autenticar a los usuarios con nombre de usuario y contraseña es redirigirlos a una página de inicio de sesión centralizada y recopilar allí sus credenciales. Esto permite que Auth0 determine si ya se han autenticado y omita por completo el formulario de inicio de sesión cuando no sea necesario.

<Info>
  ### Práctica recomendada

  Recopilar credenciales solo en la página de inicio de sesión centralizada reducirá la exposición a posibles filtraciones de secretos de los usuarios. También reducirá la necesidad de recopilar credenciales innecesariamente. Consulte [Universal Login](#universal-login) para obtener más información.
</Info>

<div id="application-integration">
  ## Integración de la aplicación
</div>

Una vez que hayas definido cómo quieres autenticar a tus usuarios, el siguiente paso es determinar cómo iniciarás esa autenticación. Por lo general, cada aplicación tendrá su propio punto de partida.

<Warning>
  Las aplicaciones móviles nativas (y las aplicaciones de escritorio) deben usar el navegador del sistema para la autenticación; de lo contrario, quedan expuestas a riesgos de seguridad adicionales. Consulta [Native Login](/docs/es/authenticate/login/native-login) para obtener más información.
</Warning>

Como ya comentamos, hemos comprobado que la mayoría de nuestros clientes usan [OpenID Connect (OIDC)](/docs/es/authenticate/protocols/openid-connect-protocol) como protocolo estándar del sector para sus aplicaciones orientadas al cliente. Determinar qué flujo de OIDC usar es tu primera tarea, y te conviene empezar revisando nuestra guía sobre [mapeo de concesiones](/docs/es/get-started/applications/application-grant-types).

Si quieres permitir que los usuarios anónimos accedan a cualquier parte de nuestra aplicación, debes determinar si los redirigirás de inmediato o solo cuando sea necesario (o quizá una combinación de ambas opciones; consulta [Redirect Users After Login](/docs/es/authenticate/login/redirect-users-after-login) para ver un análisis más detallado). Si los usuarios pueden acceder mediante [enlaces profundos](#deep-linking-to-protected-endpoints) a una versión protegida (o a un área) de tu sitio, tendrás que determinar qué enlaces a tu aplicación darán lugar a una redirección automática a Auth0.

<div id="anonymous-access">
  ### Acceso anónimo
</div>

Es importante tener en cuenta la experiencia del usuario cuando alguien llega por primera vez a tu aplicación. Si tu aplicación admite acceso de usuarios anónimos (algo bastante común en aplicaciones de comercio electrónico), hay distintos escenarios que debes considerar:

* Si vuelve a la aplicación después de haber iniciado sesión previamente, o
* Si es la primera vez que accede a la aplicación:

  * Si ya ha accedido a otra aplicación que usa el mismo tenant de Auth0,
  * Si alguna vez se autenticó (o quizá no lo hace desde hace mucho tiempo) en este dispositivo o navegador.

Cuando un usuario anónimo accede a tu aplicación, a menudo es conveniente que la aplicación detecte si el usuario ya inició sesión en otra aplicación de la misma familia, o que recuerde a este usuario incluso si la aplicación es una [SPA](/docs/es/quickstart/spa) sin estado. Por ejemplo, si puedes determinar que el usuario ya inició sesión, podrías decidir que el encabezado de la interfaz de la aplicación no muestre un botón de inicio de sesión y que, en su lugar, ofrezca al usuario un menú de cuenta o perfil. Para lograrlo, te convendrá utilizar la &quot;[autenticación silenciosa](/docs/es/authenticate/login/configure-silent-authentication)&quot;. La autenticación silenciosa te permitirá comprobar si el usuario ha iniciado sesión sin pedirle que la inicie si no lo ha hecho. Después, la aplicación puede mostrar un botón de inicio de sesión si es necesario. Sin embargo, si el usuario ya inició sesión, recibirás tokens y no tendrás que volver a mostrarle un botón de inicio de sesión.

<Warning>
  Comprobar si existe una sesión de inicio de sesión redirigiendo a Auth0 puede ser útil para tu aplicación, pero si esto va a generar muchas solicitudes, debes emplear un mecanismo de limitación para evitar latencia o un límite de frecuencia.

  Las llamadas a la Management API están sujetas a la [política de límite de frecuencia de Auth0](/docs/es/troubleshoot/customer-support/operational-policies/rate-limit-policy). Debes tenerlo en cuenta y, como ayuda, Auth0 generalmente recomienda usar el [SDK de Auth0](/docs/es/libraries) adecuado para tu entorno de desarrollo en lugar de llamar directamente a nuestras API.
</Warning>

<div id="deep-linking-to-protected-endpoints">
  ### Enlaces directos a endpoints protegidos
</div>

Hay varios motivos por los que alguien podría enlazar directamente a una página específica dentro de tu aplicación a la que solo pueden acceder usuarios autenticados. Si esto es posible en tu aplicación, debes redirigir automáticamente al usuario a Auth0 si no está autenticado. Una vez que se autentique y el <Tooltip tip="Servidor de autorización: servidor centralizado que contribuye a definir los límites del acceso de un usuario. Por ejemplo, tu servidor de autorización puede controlar los datos, las tareas y las funcionalidades disponibles para un usuario." cta="Ver glosario" href="/docs/es/glossary?term=authorization+server">servidor de autorización</Tooltip> lo devuelva a tu aplicación, puedes [redirigirlo](/docs/es/authenticate/login/redirect-users-after-login) al lugar al que quería ir desde el principio.

<div id="best-practice">
  ### Práctica recomendada
</div>

La mayoría de los frameworks modernos de autenticación admiten middleware para redirigir a un servidor de autorización como Auth0. Al elegir uno, ten en cuenta estas consideraciones clave:

* Compatibilidad con clientes confidenciales, clientes no confidenciales o ambos
* Compatibilidad con la configuración mediante [endpoint de descubrimiento](/docs/es/get-started/applications/configure-applications-with-oidc-discovery "Configurar Applications con OIDC Discovery") o de forma explícita e inline
* Compatibilidad con la validación de tokens, incluidas las fechas de expiración, las firmas, los claims y los scopes
* Compatibilidad con <Tooltip tip="Token de actualización: token que se utiliza para obtener un access token renovado sin obligar a los usuarios a iniciar sesión de nuevo." cta="Ver glosario" href="/docs/es/glossary?term=Refresh+Tokens">tokens de actualización</Tooltip> si es necesario

<div id="authenticating-the-user">
  ### Autenticación del usuario
</div>

La autenticación es el proceso de determinar la identidad del usuario. El resultado de la autenticación en un contexto de OIDC es un <Tooltip tip="token de ID: credencial destinada al propio cliente, en lugar de para acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=ID+Token">token de ID</Tooltip>. Este token contiene información sobre el usuario y solo debería poder obtenerse si el usuario se autentica mediante uno o más factores, según lo defina el servidor de autorización (la forma más común es el [ID de usuario y contraseña](#username-and-password-authentication)). Hay algunas cosas que quizá también debas considerar, además de obtener un token de ID:

* ¿También necesitamos un [token de acceso](/docs/es/secure/tokens/access-tokens) para poder invocar una API compartida?
* ¿Tu aplicación es una aplicación de página única y solo requiere un [token de ID](/docs/es/secure/tokens/id-tokens)? Consulta [concesión Authorization Code con PKCE](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce/call-your-api-using-the-authorization-code-flow-with-pkce) para obtener más información.
* ¿Tu aplicación es una aplicación nativa (móvil o de escritorio) y/o necesitas un [token de actualización](/docs/es/secure/tokens/refresh-tokens)? Consulta [concesión Authorization Code con PKCE](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce/call-your-api-using-the-authorization-code-flow-with-pkce) para obtener más información.

<Warning>
  Antes de pasar a producción, debes asegurarte de que **solo** estén habilitadas las concesiones que usas para cada aplicación en la [configuración de tu aplicación](/docs/es/get-started/applications/update-grant-types).
</Warning>

<div id="authorization-code-grant-with-or-without-pkce">
  ### Concesión Authorization Code (con o sin PKCE)
</div>

Si tu SDK solo admite la concesión Authorization Code, o necesitas un token de acceso o un <Tooltip tip="Token de actualización: token usado para obtener un token de acceso renovado sin obligar a los usuarios a volver a iniciar sesión." cta="Ver glosario" href="/docs/es/glossary?term=Refresh+Token">token de actualización</Tooltip>, la concesión Authorization Code (con o sin [PKCE](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce)) también puede utilizarse para obtener un token de ID. La concesión Authorization Code incluye una llamada a la API adicional para intercambiar el código por un token, lo que puede generar una latencia innecesaria si lo único que necesitas es el token de ID. En muchos casos, se implementa el [flujo híbrido](/docs/es/get-started/authentication-and-authorization-flow/hybrid-flow) para proporcionar un acceso óptimo al token de ID y, al mismo tiempo, aprovechar el flujo de trabajo de la concesión Authorization Code para obtener de forma segura tokens de acceso y tokens de actualización.

<Warning>
  Aunque Auth0 admite el uso de la concesión implícita para aplicaciones basadas en navegador que solo requieren un token de ID, se recomienda usar la concesión Authorization Code con [PKCE](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce). Para obtener información detallada, consulta [OAuth2 Implicit Grant and SPA](https://auth0.com/blog/oauth2-implicit-grant-and-spa/) en el blog de Auth0.

  Si necesitas un [token de actualización](/docs/es/secure/tokens/refresh-tokens) para poder obtener un nuevo token de acceso o token de ID sin tener que volver a autenticar al usuario, debes usar la [concesión Authorization Code](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow/call-your-api-using-the-authorization-code-flow).
</Warning>

<div id="attack-protection">
  ## Protección contra ataques
</div>

La razón por la que los sistemas de autenticación son importantes es que evitan que los <Tooltip tip="Actores maliciosos: entidad (una persona o grupo) que representa una amenaza para la empresa o el entorno con la intención de causar daño." cta="Ver glosario" href="/docs/es/glossary?term=bad+actors">actores maliciosos</Tooltip> accedan a aplicaciones y datos de usuario a los que no deberían tener acceso. Queremos poner tantas barreras como sea posible entre esos actores maliciosos y el acceso a nuestros sistemas. Una de las formas más sencillas de hacerlo es asegurarse de que su [protección contra ataques](/docs/es/secure/attack-protection) con Auth0 esté configurada correctamente, así que tómese un momento para leer la guía sobre este tema y comprobar que todo funcione correctamente.

<Info>
  ### Práctica recomendada

  Auth0 gestiona la detección de anomalías entre bastidores y ofrece una excelente funcionalidad de seguridad para su producto. Si va a utilizarla, asegúrese de haber configurado su [proveedor de correo electrónico](/docs/es/get-started/architecture-scenarios/business-to-business/operations#email-provider-setup) y sus [plantillas de correo electrónico](/docs/es/get-started/architecture-scenarios/business-to-business/branding#email-template-customization) antes de activar la entrega de correos electrónicos a sus usuarios.
</Info>

<div id="sso-with-legacy-systems">
  ## SSO con sistemas heredados
</div>

En una reestructuración a gran escala, no siempre es posible —o práctico— actualizar todas sus aplicaciones de una sola vez. De hecho, nuestra práctica recomendada es planificar un enfoque iterativo a la hora de integrar con Auth0. Si sus aplicaciones ya participan en el inicio de sesión único (SSO) y su sistema de identidad heredado admite protocolos como OIDC o <Tooltip tip="Security Assertion Markup Language (SAML): Protocolo estandarizado que permite a dos partes intercambiar información de autenticación sin una contraseña." cta="Ver glosario" href="/docs/es/glossary?term=SAML">SAML</Tooltip>, tiene un par de opciones si desea seguir ofreciendo SSO mientras integra con Auth0:

* Actualizar su proveedor de identidad existente en su sistema de SSO heredado para que redirija a Auth0 para el inicio de sesión (por ejemplo, mediante [SAML](/docs/es/authenticate/single-sign-on/outbound-single-sign-on/configure-auth0-saml-identity-provider)), o
* Hacer que Auth0 redirija a su sistema de SSO heredado para iniciar sesión. Esto requiere configurar su sistema heredado como un IdP en Auth0 (es decir, ya sea mediante [SAML](/docs/es/authenticate/protocols/saml/saml-sso-integrations/configure-auth0-saml-service-provider) o [OIDC](/docs/es/authenticate/identity-providers/social-identity-providers)).

<Info>
  ### Práctica recomendada

  Mantener una experiencia de SSO con su sistema heredado puede añadir complejidad, pero puede valer la pena para ofrecer una experiencia de usuario más fluida mientras integra con Auth0. Si tiene pensado seguir este camino, planificarlo desde el principio puede ayudar a garantizar que sea posible. Si todavía no cuenta con SSO en un servicio centralizado, es poco probable que la complejidad de añadirlo compense los beneficios.
</Info>

Este es un tema complejo que probablemente requerirá una investigación adicional en función de su arquitectura heredada actual, y le recomendamos que solo lo considere si actualmente cuenta con compatibilidad con SSO en su sistema heredado. Nota: si actualmente redirige desde sus aplicaciones a un sistema centralizado para autenticar a sus usuarios y ese sistema solo solicita credenciales si todavía no tiene una sesión con el sistema centralizado, entonces tiene una implementación de SSO heredada.

<div id="enterprise-login">
  ## Inicio de sesión empresarial
</div>

El escenario de “trae tu propia identidad” se ha convertido en un requisito imprescindible para casi todas las aplicaciones B2B. La mayoría de las empresas esperan poder integrar su IdP en tu aplicación para que sus empleados no tengan que guardar otro conjunto de credenciales. Esta es una forma muy útil de simplificar la experiencia de autenticación sin comprometer la seguridad, y usar [Universal Login](#universal-login) facilita empezar a añadir soporte para [conexiones empresariales](/docs/es/authenticate/identity-providers/enterprise-identity-providers) con una interrupción mínima.

<Info>
  ### Práctica recomendada

  Una vez que empieces a dar soporte a conexiones empresariales para los usuarios, debes implementar algún tipo de [Home Realm Discovery](#home-realm-discovery) para poder determinar a qué conexión enviar al usuario para la autenticación.
</Info>

Con soporte para conexiones empresariales, las identidades y credenciales de los usuarios son gestionadas por el proveedor de identidad de la organización de tus clientes, así como ciertos claims de identidad, que Auth0 usará para completar el [perfil](/docs/es/get-started/architecture-scenarios/business-to-business/profile-management) del usuario.

<Info>
  ### Práctica recomendada

  &quot;Trae tu propia identidad&quot; es una excelente funcionalidad, pero si no la admites desde el primer día, y a veces incluso si lo haces, puede que alguna organización quiera cambiar a su propio IdP después de haber usado la aplicación durante un tiempo. Necesitarás una forma de [vincular cuentas de usuario](/docs/es/manage-users/user-accounts/user-account-linking) para asociar eficazmente la nueva identidad con la antigua identidad de base de datos.
</Info>

<div id="multi-factor-authentication-mfa">
  ## Autenticación multifactor (MFA)
</div>

En una época en la que el uso indebido de las credenciales de los usuarios está en su punto más alto, proteger sus sistemas supone un desafío, especialmente cuando es tan habitual que los atacantes roben información de identidad de los usuarios. Sin embargo, una de las formas más eficaces de hacerlo es ofrecer a los usuarios la posibilidad de configurar un segundo factor para proteger su cuenta. Esto se conoce más comúnmente como [Multi-Factor Authentication](/docs/es/secure/multi-factor-authentication). Así se garantiza que solo un usuario legítimo pueda acceder a su cuenta, incluso si utiliza un nombre de usuario y contraseña que podrían haberse visto comprometidos en otra aplicación.

<Info>
  ### Práctica recomendada

  Es bastante común que las aplicaciones orientadas al cliente ofrezcan a los usuarios la *opción* de añadir un segundo factor en lugar de *obligarlos* a usarlo. Para obtener más información al respecto, consulte [cómo ofrecer a sus usuarios la opción de añadir MFA](https://auth0.com/learn/multifactor-authentication-customers/).
</Info>

Auth0 admite varias opciones para habilitar MFA y proteger el acceso a las cuentas de usuario, y existen varias prácticas para garantizar que realmente proporcione una barrera de acceso flexible basada en un segundo factor:

* Auth0 [Guardian](https://auth0.com/multifactor-authentication): un servicio que proporciona tanto la generación de notificaciones Push como una aplicación para permitir o denegar solicitudes. Push envía una notificación al dispositivo prerregistrado del usuario —normalmente un móvil o una tableta—, desde el que el usuario puede permitir o denegar inmediatamente el acceso a la cuenta con solo pulsar un botón.
* Contraseña de un solo uso basada en el tiempo ([TOTP](https://auth0.com/blog/from-theory-to-practice-adding-two-factor-to-node-dot-js/)): le permite registrar un dispositivo —como Google Authenticator— que generará una contraseña de un solo uso que cambia con el tiempo y que puede introducirse como segundo factor para validar la cuenta de un usuario.
* SMS: para enviar un código de un solo uso por SMS que el usuario deberá introducir antes de poder completar la autenticación.
* Voz: para entregar un código de un solo uso mediante una llamada telefónica que el usuario deberá introducir antes de poder completar la autenticación.
* Duo: le permite usar su cuenta de Duo para la autenticación multifactor.
* Email: le permite usar su cuenta de correo electrónico para la autenticación multifactor.

Aunque el flujo de trabajo de MFA con tecnologías como Guardian o Google Authenticator normalmente se proporciona mediante una aplicación independiente que se ejecuta en un dispositivo móvil o tableta, si no quiere que sus clientes tengan que descargar otra aplicación, Auth0 también le proporciona un [SDK](https://auth0.com/blog/announcing-guardian-whitelabel-sdk/) que puede usar para crear el flujo de trabajo del segundo factor directamente en sus aplicaciones móviles existentes.
