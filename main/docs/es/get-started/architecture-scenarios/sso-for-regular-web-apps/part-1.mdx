En esta sección, explicaremos la solución que estamos implementando, incluidos los detalles sobre la gestión de identidades, los protocolos que deben utilizarse y el flujo de autenticación necesario.

<div id="identity-management">
  ## Gestión de identidades
</div>

ExampleCo decidió usar Auth0 como su proveedor de Identity as a Service (IDaaS). El motivo de esta decisión fue que la empresa no quería destinar recursos a la capacitación, la implementación y el mantenimiento de la gestión de identidades y accesos. Además, la empresa planea ampliar esta solución en el futuro, posiblemente añadiendo una aplicación nativa para móviles y una API para enviar los registros de horas aprobados a sus sistemas internos. Auth0 ofrece la flexibilidad necesaria para incorporar esos cambios en su arquitectura con un esfuerzo mínimo.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Identity-as-Service (&quot;IDaaS&quot;) es un servicio en la nube para la gestión de identidades y accesos. Los servicios ofrecidos suelen incluir inicio de sesión único (SSO), identidad federada, gestión de contraseñas y mucho más.
</Callout>

<div id="which-protocol-to-use">
  ## Qué protocolo usar
</div>

La siguiente decisión tiene que ver con qué protocolo usar: <Tooltip tip="OAuth 2.0: Marco de autorización que define protocolos de autorización y flujos de trabajo." cta="Ver glosario" href="/docs/es/glossary?term=OAuth+2.0">OAuth 2.0</Tooltip> con <Tooltip tip="OAuth 2.0: Marco de autorización que define protocolos de autorización y flujos de trabajo." cta="Ver glosario" href="/docs/es/glossary?term=OpenID">OpenID</Tooltip> Connect (OIDC) o <Tooltip tip="Security Assertion Markup Language (SAML): Protocolo estandarizado que permite a dos partes intercambiar información de autenticación sin una contraseña." cta="Ver glosario" href="/docs/es/glossary?term=SAML">SAML</Tooltip>.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Auth0 implementa protocolos de identidad probados, comunes y ampliamente usados, tanto para productos web orientados al consumidor (OAuth 2.0, OAuth 1.0, OpenID) como para implementaciones empresariales (SAML, WS-Federation, LDAP). Tienes total libertad para usar el que mejor se adapte a las necesidades de tu negocio.
</Callout>

**OpenID Connect** es un protocolo de autenticación basado en la familia de especificaciones de OAuth 2.0. Usa <Tooltip tip="ID Token: Credencial destinada al propio cliente, en lugar de para acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=identity+tokens">tokens de identidad</Tooltip> (<Tooltip tip="ID Token: Credencial destinada al propio cliente, en lugar de para acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=JWT">JWT</Tooltip>) JSON simples que se entregan a través del protocolo OAuth 2.0.

<Info>
  ### OAuth vs OpenID Connect (OIDC)

  OAuth 2.0 y OpenID Connect (OIDC) a menudo se confunden como si fueran lo mismo, pero eso no es del todo correcto.
  **OAuth 2.0** es un protocolo que te permite autorizar a un sitio web (el consumidor o la aplicación) a acceder a tus datos desde otro sitio web (el servidor de recursos o proveedor). Por ejemplo, quieres autorizar a un sitio web a acceder a algunos archivos de tu cuenta de Dropbox. El sitio web te redirigirá a Dropbox, donde te preguntará si debe dar acceso a tus archivos. Si aceptas, el sitio web quedará autorizado para acceder a tus archivos de Dropbox. En esencia, OAuth 2.0 trata del acceso a recursos y de compartirlos.
  **OpenID Connect**, por otro lado, es una capa de identidad sencilla construida sobre el protocolo OAuth 2.0. Te permite tener un único inicio de sesión para varios sitios. Cada vez que necesites iniciar sesión en un sitio web con OIDC, se te redirigirá a tu sitio OpenID, donde iniciarás sesión y luego volverás al sitio web. En esencia, OIDC se centra en la autenticación de usuario.
</Info>

**SAML** es un protocolo basado en XML que proporciona tanto autenticación como autorización entre partes de confianza.

En comparación con SAML, OpenID Connect es más ligero y más fácil de manejar. SAML está probado, es potente y flexible, pero para los requisitos de esta aplicación no hacen falta ni esa flexibilidad ni esa potencia. La federación de identidad (una de las razones más convincentes para adoptar SAML) tampoco es necesaria aquí. Y, si alguna vez llegara a ser un requisito, Auth0 puede gestionarlo fácilmente, de la misma manera que gestiona AD (que usa LDAP).

Por estas razones, ExampleCo usará OpenID Connect para su implementación.

<div id="authentication-flow">
  ## Flujo de autenticación
</div>

OpenID Connect admite más de un flujo de autenticación. Como en nuestro caso se trata de una aplicación web tradicional, usaremos el flujo de código de autorización.

El flujo funciona de la siguiente manera:

1. La aplicación web (llamada Cliente en términos de OIDC) inicia la solicitud de autenticación redirigiendo el agente de usuario (navegador) a Auth0 (el servidor de autorización en términos de OIDC).
2. Auth0 autentica al usuario (a través del agente de usuario). La primera vez que el usuario pase por este flujo, se mostrará una página de consentimiento en la que se enumeran los permisos que se otorgarán a la aplicación (por ejemplo, publicar mensajes o listar contactos). El usuario inicia sesión en el servicio (a menos que ya la haya iniciado) y autoriza el acceso de la aplicación.
3. Si el usuario concede acceso, Auth0 redirige de nuevo el agente de usuario a la aplicación junto con un código de autorización en la cadena de consulta.
4. La aplicación envía el código de autorización a Auth0, junto con las credenciales de la aplicación (`client_id` y `client_secret`), y solicita un token.
5. Auth0 autentica la aplicación (usando `client_id` y `client_secret`) y valida el código de autorización. Si es válido, Auth0 responde con un token de ID.

<Info>
  ### Modo de respuesta Form Post

  Otra opción es usar **OAuth 2.0 Form Post Response Mode** con `response_type=id_token&response_mode=form_post`. Debido al parámetro de solicitud `response_type=id_token`, la respuesta contiene directamente el token de ID, en lugar del código de autorización, mientras que `response_mode=form_post` codifica el token de ID junto con el resto de los parámetros de la respuesta de autorización como valores de un formulario HTML que se envían automáticamente en el agente de usuario. De esta manera, puedes tener un flujo de autenticación optimizado (sin necesidad de intercambiar el código por un token de ID); sin embargo, debes asegurarte de que la tecnología que usas para implementar tu aplicación sea compatible con él (el middleware de ASP.NET Core sí lo admite). Para más detalles, consulta la [especificación OAuth 2.0 Form Post Response Mode](https://openid.net/specs/oauth-v2-form-post-response-mode-1_0.html).
</Info>

El **token de ID** (normalmente llamado `id_token` en ejemplos de código) es un **JSON Web Token (JWT)** que contiene datos de identidad. La aplicación lo consume y lo utiliza para obtener información del usuario, como su nombre, correo electrónico, etc., normalmente para mostrarla en la interfaz de usuario.

<Info>
  ### Más sobre los tokens

  Los tokens son cadenas alfanuméricas que se usan en la autenticación basada en tokens. Permiten que los usuarios se autentiquen una vez con un nombre de usuario y contraseña y reciban a cambio un token que pueden usar a partir de ese momento. Tienen una duración limitada.

  Los **JSON Web Tokens (JWT)** son tokens que cumplen con el [estándar JSON Web Token](https://tools.ietf.org/html/rfc7519) y contienen información sobre una identidad en forma de claims. Son autocontenidos, lo que significa que no es necesario que el destinatario llame a un servidor para validar el token. Los JWT pueden firmarse usando un secreto (con el algoritmo **HMAC**) o un par de claves pública/privada con **RSA**. Puedes encontrar más información sobre JWT [aquí](/docs/es/secure/tokens/json-web-tokens).

  El token de ID, que es un JWT, cumple con un estándar de la industria (IETF [RFC 7519](https://tools.ietf.org/html/rfc7519)) y contiene tres partes: un encabezado, un cuerpo y una firma.

  * El encabezado contiene el tipo de token y el algoritmo hash utilizado sobre el contenido del token.
  * El cuerpo, también llamado payload, contiene claims de identidad sobre un usuario. Hay algunos claims con nombres registrados, para elementos como el emisor del token, el sujeto del token (sobre quién tratan los claims) y el momento de emisión. Se puede añadir cualquier cantidad de claims adicionales con otros nombres, aunque se debe tener cuidado de mantener el JWT dentro de las limitaciones de tamaño de las URL en el navegador.
  * La firma la utiliza el destinatario de un JWT para validar la integridad de la información transmitida en el JWT.
</Info>

<div id="how-to-validate-an-id-token">
  ### Cómo validar un token de ID
</div>

La validación de un token de ID requiere varios pasos:

1. Si el token de ID está cifrado, descífrelo con las claves y los algoritmos que especificó la aplicación.
2. El identificador del emisor del proveedor de OpenID debe coincidir con el valor del claim `iss` (issuer).
3. El claim `aud` (audience) debe contener el valor de `client_id` de la aplicación. El token de ID debe rechazarse si no incluye a la aplicación como audiencia válida o si contiene audiencias adicionales en las que la aplicación no confía.
4. Si el token de ID contiene varias audiencias, la aplicación debe verificar que esté presente un claim `azp`.
5. Si está presente un claim `azp` (authorized party), la aplicación debe verificar que su `client_id` sea el valor del claim.
6. La aplicación debe validar la firma de los token de ID de acuerdo con JWS mediante el algoritmo especificado en el parámetro de encabezado `alg` del JWT. La aplicación debe usar las claves proporcionadas por el emisor.
7. El valor de `alg` debe ser el valor predeterminado `RS256` o el algoritmo enviado por la aplicación en el parámetro `id_token_signed_response_alg` durante el registro.
8. Si el parámetro de encabezado `alg` del JWT usa un algoritmo basado en MAC, como `HS256`, `HS384` o `HS512`, los octetos de la representación UTF-8 del `client_secret` correspondiente al `client_id` incluido en el claim `aud` (audience) se usan como clave para validar la firma. En el caso de los algoritmos basados en MAC, el comportamiento no está especificado si `aud` tiene varios valores o si está presente un valor `azp` distinto del valor de `aud`.
9. La hora actual debe ser anterior a la hora representada por el claim `exp`.
10. El claim `iat` puede usarse para rechazar tokens que se hayan emitido con demasiada antelación respecto de la hora actual, lo que limita el tiempo durante el cual es necesario almacenar nonces para evitar ataques. El rango aceptable es específico de la aplicación.
11. Si se envió un valor `nonce` en la solicitud de autenticación, debe estar presente un claim `nonce` y debe comprobarse su valor para verificar que sea el mismo que se envió en la solicitud de autenticación. La aplicación debe comprobar el valor de `nonce` para detectar ataques de repetición. El método exacto para detectar ataques de repetición es específico de la aplicación.
12. Si se solicitó el claim `acr`, la aplicación debe comprobar que el valor del claim indicado sea adecuado.
13. Si se solicitó el claim `auth_time`, ya sea mediante una solicitud específica de este claim o mediante el parámetro `max_age`, la aplicación debe comprobar el valor del claim `auth_time` y solicitar una nueva autenticación si determina que ha transcurrido demasiado tiempo desde la última autenticación del usuario final.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Si almacena tokens de ID en su servidor, debe hacerlo de forma segura.
</Callout>
