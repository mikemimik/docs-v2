En esta sección, veremos la solución que estamos implementando, incluidos los detalles sobre la gestión de identidades, los protocolos que se utilizarán y el flujo de autenticación necesario.

<div id="identity-management">
  ## Gestión de identidades
</div>

ExampleCo decidió usar Auth0 como su proveedor de Identity as a Service (IDaaS). La razón de esta decisión fue que la empresa no quería dedicar recursos a la capacitación, implementación y mantenimiento de la gestión de identidades y accesos. Además, la empresa planea seguir ampliando esta solución en el futuro, posiblemente añadiendo una aplicación nativa para móviles y una API para enviar hojas de hora aprobadas a sus sistemas internos. Auth0 ofrece la flexibilidad necesaria para incorporar esos cambios en su arquitectura con el mínimo esfuerzo.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Identity-as-Service (&quot;IDaaS&quot;) es un servicio en la nube para la gestión de identidades y accesos. Los servicios ofrecidos suelen incluir inicio de sesión único (SSO), identidad federada, gestión de contraseñas, entre otros.
</Callout>

<div id="which-protocol-to-use">
  ## Qué protocolo usar
</div>

La siguiente decisión tiene que ver con qué protocolo usar: <Tooltip tip="OAuth 2.0: Marco de autorización que define protocolos de autorización y flujos de trabajo." cta="Ver glosario" href="/docs/es/glossary?term=OAuth+2.0">OAuth 2.0</Tooltip> con <Tooltip tip="OAuth 2.0: Marco de autorización que define protocolos de autorización y flujos de trabajo." cta="Ver glosario" href="/docs/es/glossary?term=OpenID">OpenID</Tooltip> Connect (OIDC) o <Tooltip tip="Security Assertion Markup Language (SAML): Protocolo estandarizado que permite a dos partes intercambiar información de autenticación sin una contraseña." cta="Ver glosario" href="/docs/es/glossary?term=SAML">SAML</Tooltip>.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Auth0 implementa protocolos de identidad consolidados, comunes y ampliamente utilizados, tanto para productos web orientados al consumidor (OAuth 2.0, OAuth 1.0, OpenID) como para implementaciones empresariales (SAML, WS-Federation, LDAP). Tiene total libertad para usar el que mejor se adapte a las necesidades de su negocio.
</Callout>

**OpenID Connect** es un protocolo de autenticación basado en la familia de especificaciones OAuth 2.0. Utiliza <Tooltip tip="ID Token: Credencial destinada al propio cliente, en lugar de al acceso a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=identity+tokens">tokens de identidad</Tooltip> JSON sencillos (<Tooltip tip="ID Token: Credencial destinada al propio cliente, en lugar de al acceso a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=JWT">JWT</Tooltip>) entregados a través del protocolo OAuth 2.0.

<Info>
  ### OAuth vs OpenID Connect (OIDC)

  OAuth 2.0 y OpenID Connect (OIDC) suelen confundirse como si fueran lo mismo, pero no es exactamente así.
  **OAuth 2.0** es un protocolo que le permite autorizar a un sitio web (el consumidor o la aplicación) a acceder a sus datos desde otro sitio web (el servidor de recursos o proveedor). Por ejemplo, quiere autorizar a un sitio web a acceder a algunos archivos de su cuenta de Dropbox. El sitio web le redirigirá a Dropbox, donde se le preguntará si debe conceder acceso a sus archivos. Si acepta, el sitio web quedará autorizado para acceder a sus archivos en Dropbox. En esencia, OAuth 2.0 se centra en el acceso a recursos y en compartirlos.
  **OpenID Connect**, por otro lado, es una capa de identidad sencilla construida sobre el protocolo OAuth 2.0. Le ofrece un único inicio de sesión para múltiples sitios. Cada vez que necesite iniciar sesión en un sitio web usando OIDC, se le redirigirá a su sitio de OpenID, donde iniciará sesión, y luego volverá al sitio web. En esencia, OIDC se centra en la autenticación de usuario.
</Info>

**SAML** es un protocolo basado en XML que proporciona tanto autenticación como autorización entre partes de confianza.

En comparación con SAML, OpenID Connect es más ligero y más fácil de manejar. SAML es una solución consolidada, potente y flexible, pero para los requisitos de esta aplicación, esa flexibilidad y esa potencia no son necesarias. La federación de identidades (una de las razones más convincentes para adoptar SAML) tampoco es necesaria aquí. Y, si alguna vez llegara a serlo, Auth0 puede gestionarla fácilmente, del mismo modo que gestiona AD (que usa LDAP).

Por estas razones, ExampleCo usará OpenID Connect en su implementación.

<div id="authentication-flow">
  ## Flujo de autenticación
</div>

OpenID Connect admite más de un flujo de autenticación. Como en nuestro caso se trata de una aplicación web tradicional, usaremos el flujo de código de autorización.

El flujo es el siguiente:

1. La aplicación web (llamada cliente en términos de OIDC) inicia la solicitud de autenticación redirigiendo el user-agent (navegador) a Auth0 (el servidor de autorización en términos de OIDC).
2. Auth0 autentica al usuario (a través del user-agent). La primera vez que el usuario pase por este flujo, se mostrará una página de consentimiento donde se enumeran los permisos que se otorgarán a la aplicación (por ejemplo, publicar mensajes o listar contactos). El usuario inicia sesión en el servicio (a menos que ya la haya iniciado) y autoriza el acceso de la aplicación.
3. Si el usuario concede acceso, Auth0 redirige el user-agent de vuelta a la aplicación, junto con un código de autorización en la cadena de consulta.
4. La aplicación envía el código de autorización a Auth0, junto con las credenciales de la aplicación (`client_id` y `client_secret`), y solicita un token.
5. Auth0 autentica la aplicación (usando `client_id` y `client_secret`) y valida el código de autorización. Si es válido, Auth0 responde con un token de ID.

<Info>
  ### Modo de respuesta Form Post

  Otra opción es usar el **OAuth 2.0 Form Post Response Mode** con `response_type=id_token&response_mode=form_post`. Debido al parámetro de solicitud `response_type=id_token`, la respuesta contiene directamente el token de ID, en lugar del código de autorización, mientras que `response_mode=form_post` codifica el token de ID junto con el resto de los parámetros de la respuesta de autorización como valores de formulario HTML que se envían automáticamente en el User Agent. De esta manera, puedes tener un flujo de autenticación optimizado (sin necesidad de intercambiar el código por un token de ID); sin embargo, debes asegurarte de que la tecnología que utilizas para implementar tu aplicación sea compatible con él (el middleware de ASP .NET Core sí lo admite). Para obtener más detalles, consulta la [especificación OAuth 2.0 Form Post Response Mode](https://openid.net/specs/oauth-v2-form-post-response-mode-1_0.html).
</Info>

El **token de ID** (normalmente denominado `id_token` en ejemplos de código) es un **JSON Web Token (JWT)** que contiene datos de identidad. La aplicación lo consume y se usa para obtener información del usuario, como su nombre, correo electrónico, etc., normalmente para mostrarla en la interfaz de usuario.

<Info>
  ### Más sobre los tokens

  Los tokens son cadenas alfanuméricas que se utilizan en la autenticación basada en tokens. Permiten a los usuarios autenticarse una vez con un nombre de usuario y contraseña, y recibir a cambio un token que pueden usar a partir de ese momento. Tienen un tiempo de vida limitado.

  Los **JSON Web Tokens (JWT)** son tokens que cumplen con el [estándar JSON Web Token](https://tools.ietf.org/html/rfc7519) y contienen información sobre una identidad en forma de claims. Son autocontenidos, en el sentido de que no es necesario que el receptor llame a un servidor para validar el token. Los JWT se pueden firmar usando un secreto (con el algoritmo **HMAC**) o un par de claves pública/privada mediante **RSA**. Puedes encontrar más información sobre JWT [aquí](/docs/es/secure/tokens/json-web-tokens).

  El token de ID, que es un JWT, se ajusta a un estándar de la industria (IETF [RFC 7519](https://tools.ietf.org/html/rfc7519)) y contiene tres partes: un header, un body y una firma.

  * El header contiene el tipo de token y el algoritmo hash utilizado sobre el contenido del token.
  * El body, también llamado payload, contiene claims de identidad sobre un usuario. Hay algunos claims con nombres registrados, para elementos como el emisor del token, el sujeto del token (sobre quién tratan los claims) y la hora de emisión. Se puede agregar cualquier cantidad de claims adicionales con otros nombres, aunque se debe tener cuidado de mantener el JWT dentro de las limitaciones de tamaño de las URL en el navegador.
  * La firma la usa el destinatario de un JWT para validar la integridad de la información transmitida en el JWT.
</Info>

<div id="how-to-validate-an-id-token">
  ### Cómo validar un token de ID
</div>

La validación de un token de ID requiere varios pasos:

1. Si el token de ID está cifrado, descífrelo con las claves y los algoritmos que especificó la aplicación.
2. El identificador del emisor del proveedor de OpenID debe coincidir con el valor del claim `iss` (issuer).
3. El claim `aud` (audience) debe contener el valor de `client_id` de la aplicación. El token de ID debe rechazarse si no incluye a la aplicación como audiencia válida o si contiene audiencias adicionales en las que la aplicación no confía.
4. Si el token de ID contiene varias audiencias, la aplicación debe verificar que esté presente un claim `azp`.
5. Si hay un claim `azp` (authorized party), la aplicación debe verificar que su `client_id` coincida con el valor del claim.
6. La aplicación debe validar la firma de los tokens de ID de acuerdo con JWS, usando el algoritmo especificado en el parámetro de encabezado `alg` del JWT. La aplicación debe usar las claves proporcionadas por el emisor.
7. El valor de `alg` debe ser el predeterminado, `RS256`, o el algoritmo enviado por la aplicación en el parámetro `id_token_signed_response_alg` durante el registro.
8. Si el parámetro de encabezado `alg` del JWT usa un algoritmo basado en MAC, como `HS256`, `HS384` o `HS512`, se usan como clave para validar la firma los octetos de la representación UTF-8 de `client_secret` correspondiente al `client_id` contenido en el claim `aud` (audience). En el caso de los algoritmos basados en MAC, el comportamiento no está especificado si `aud` tiene varios valores o si hay un valor `azp` distinto del valor de `aud`.
9. La hora actual debe ser anterior a la hora representada por el claim `exp`.
10. El claim `iat` puede usarse para rechazar tokens emitidos demasiado alejados de la hora actual, lo que limita el tiempo durante el cual deben almacenarse los nonce para prevenir ataques. El rango aceptable es específico de cada aplicación.
11. Si se envió un valor `nonce` en la solicitud de autenticación, debe estar presente un claim `nonce` y debe comprobarse su valor para verificar que sea el mismo que se envió en la solicitud de autenticación. La aplicación debe comprobar el valor `nonce` para detectar ataques de repetición. El método exacto para detectar ataques de repetición es específico de cada aplicación.
12. Si se solicitó el claim `acr`, la aplicación debe comprobar que el valor del claim indicado sea apropiado.
13. Si se solicitó el claim `auth_time`, ya sea mediante una solicitud específica de este claim o usando el parámetro `max_age`, la aplicación debe comprobar el valor del claim `auth_time` y solicitar una reautenticación si determina que ha transcurrido demasiado tiempo desde la última autenticación del usuario final.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Si almacena tokens de ID en su servidor, debe hacerlo de forma segura.
</Callout>
