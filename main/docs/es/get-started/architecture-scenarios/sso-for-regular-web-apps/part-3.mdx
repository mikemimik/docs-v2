Veamos la implementación de nuestra aplicación web tradicional. Usamos ASP .NET Core para implementarla; puedes encontrar el código en [este repositorio de GitHub](https://github.com/auth0-samples/auth0-pnp-webapp-oidc).

El ejemplo incluye una aplicación que usa la integración con Active Directory para autenticar a los empleados de la empresa y una conexión de base de datos de Auth0 para los contratistas externos. La autorización se implementa mediante reglas y claims, como veremos en detalle en esta sección.

<div id="user-login">
  ## Inicio de sesión de usuarios
</div>

Auth0 proporciona un widget de Lock que funciona como componente de inicio de sesión para su aplicación, lo que significa que no tiene que implementar su propia pantalla de inicio de sesión. El widget de Lock se integra perfectamente con todas las conexiones que configure en su <Tooltip tip="El producto principal de Auth0 para configurar sus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Auth0+dashboard">Auth0 Dashboard</Tooltip>, ya sean conexiones de base de datos, sociales o empresariales.

Hay varias formas de implementar una pantalla de inicio de sesión con una aplicación web y Auth0:

* **Lock alojado**: Use una instancia del widget de Lock alojada en la infraestructura de Auth0.
* **Embedded Lock**: Incruste el widget de Lock en una página web de su aplicación. Tiene algunas opciones de personalización para el propio widget de Lock y control total sobre el resto del HTML de la página.
* **UI personalizada**: Desarrolle una página web completamente personalizada para la pantalla de inicio de sesión. El formulario HTML personalizado enviará los datos a su servidor, que, a su vez, autenticará al usuario mediante la Authentication API. Para obtener más información sobre cuándo usar una UI personalizada, consulte [Personalice las páginas de Classic Login con Lock o SDK](/docs/es/customize/login-pages/classic-login/customize-with-lock-sdk).

<div id="automate-home-realm-discovery-hrd">
  ### Automatizar Home Realm Discovery (HRD)
</div>

De forma predeterminada, Lock mostrará todas las conexiones disponibles para iniciar sesión. Seleccionar los <Tooltip tip="Proveedor de identidad (IdP): servicio que almacena y administra identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=Identity+Providers">proveedores de identidad</Tooltip> adecuados entre varias opciones se denomina Home Realm Discovery (HRD). En nuestro caso, las opciones son autenticarse con Active Directory (para los empleados de la empresa) o usar correo electrónico y contraseña para nuestra conexión de base de datos (contratistas externos).

Sin embargo, quizá quieras evitar ese primer paso, en el que el usuario debe elegir el proveedor de identidad (IdP), y hacer que el sistema lo identifique en lugar de preguntarlo cada vez. Lock te ofrece las siguientes opciones:

* **Identificar el IdP mediante programación**: Cuando inicias una transacción de autenticación con Auth0, puedes enviar opcionalmente un parámetro `connection`. Este valor se asigna directamente a cualquier conexión definida en tu Dashboard. Al usar la versión Hosted de Lock mediante el endpoint `/authorize`, puedes pasar un parámetro `connection` en la cadena de consulta que contenga el nombre de la conexión. Como alternativa, si usas Embedded Lock, es tan sencillo como escribir `auth0.show({connections: ['{yourConnection}']});`.

  * Hay varias formas prácticas de obtener el valor de `connection`. Una de ellas es usar **vanity URLs**: por ejemplo, los empleados de la empresa usarán `https://internal.yoursite.com`, mientras que los contratistas externos usarán `https://external.yoursite.com`.
* **Usar dominios de correo electrónico**: Lock puede usar dominios de correo electrónico como una forma de enrutar solicitudes de autenticación. Las conexiones empresariales en Auth0 se pueden asignar a `domains`. Si una conexión tiene esta configuración, el campo de contraseña se desactiva automáticamente al escribir un correo electrónico con un dominio asignado. Ten en cuenta que puedes asociar varios dominios a una sola conexión.

Para obtener información adicional sobre este tema, consulta [Seleccionar entre varias opciones de conexión](/docs/es/libraries/lock/selecting-from-multiple-connection-options).

<div id="session-management">
  ## Gestión de sesiones
</div>

Cuando hablamos de gestionar sesiones, normalmente hay tres capas de sesión que debemos tener en cuenta:

* **Sesión de la aplicación**: La primera es la sesión dentro de la aplicación. Aunque tu aplicación use Auth0 para autenticar a los usuarios, igualmente necesitarás registrar que el usuario ha iniciado sesión en tu aplicación. En una aplicación web normal, esto se logra almacenando información en una cookie.
* **sesión de Auth0**: A continuación, Auth0 también mantendrá una sesión y almacenará la información del usuario en una cookie. La próxima vez que se redirija al usuario a la pantalla de Auth0 Lock, su información se recordará.
* **sesión del proveedor de identidad**: La última capa es la del proveedor de identidad, por ejemplo, Facebook o Google. Cuando permites que los usuarios inicien sesión con cualquiera de estos proveedores y ya tienen una sesión iniciada en el proveedor, no se les pedirá que vuelvan a iniciar sesión. Puede que solo se les pida dar permisos para compartir su información con Auth0 y, a su vez, con tu aplicación.

Por lo tanto, al desarrollar una aplicación web, tendrás que registrar que el usuario ha iniciado sesión en tu aplicación web. Puedes hacerlo mediante una sesión basada en cookies para mantener ese estado, y también almacenar cualquier información o token relacionado con el usuario.

<Info>
  ### ¿Cómo controlo la duración de la sesión local de la aplicación del usuario? ¿Puedo hacerlo desde Auth0?

  La aplicación web tiene control total sobre la sesión local de la aplicación del usuario. Cómo se hace normalmente depende de la pila web que se esté usando (por ejemplo, ASP.NET). En cualquier caso, todos los enfoques terminan utilizando una o más cookies para controlar la sesión. El desarrollador puede optar por usar el vencimiento del token de ID JWT devuelto por Auth0 para controlar la duración de su sesión o ignorarlo por completo. Algunos desarrolladores almacenan el propio token de ID en el estado de la sesión y finalizan la sesión del usuario cuando este expira.

  La razón para usar el vencimiento del token como referencia para determinar el vencimiento de la sesión local es que te da control centralizado sobre la duración de la sesión de un usuario desde el Auth0 Dashboard.
</Info>

El flujo de inicio de sesión es el siguiente:

<Frame>![undefined](/docs/images/cdy7uua7fh8z/4bqozVk6fF4JrWRP1BJK7Y/1403eb1c0efb12552307358a26c6e7f7/login-flow.png)</Frame>

1. **Iniciar flujo de autenticación OIDC**: El navegador del usuario enviará una solicitud a Auth0 para iniciar el flujo OIDC.
2. **Establecer cookie de SSO**: Auth0 establecerá una cookie para almacenar la información del usuario.
3. **Intercambio de código y devolución del token de ID**: Auth0 enviará una solicitud de vuelta al servidor web y devolverá el código. El servidor web intercambiará el código por un token de ID.
4. **Establecer cookie de autenticación y enviar respuesta**: El servidor web enviará una respuesta de vuelta al navegador y establecerá la cookie de autenticación de la aplicación para almacenar la información de sesión del usuario.
5. **La cookie de autenticación se envía con cada solicitud posterior**: La cookie de autenticación de la aplicación se enviará en cada solicitud posterior como prueba de que el usuario está autenticado.

<Info>
  ### ¿Cómo afecta la sesión de SSO de Auth0 a la sesión de la aplicación?

  Auth0 gestiona su propia sesión de inicio de sesión único. Las aplicaciones pueden optar por respetar o ignorar esa sesión de SSO cuando se trata de mantener su propia sesión local. El widget de Lock incluso tiene una funcionalidad especial con la que puede detectar si existe una sesión de SSO de Auth0 y preguntarle al usuario si desea volver a iniciar sesión como ese mismo usuario.

  <Frame>![Lock Widget SSO](https://cdn2.auth0.com/docs/1.14516.0/media/articles/architecture-scenarios/web-app-sso/sso-login.png)</Frame>

  Si lo hace, iniciará sesión sin tener que volver a introducir sus credenciales en el IdP real. Aunque el usuario no se haya autenticado, la aplicación sigue realizando un flujo de autenticación con Auth0 y obtiene un nuevo token de ID, que luego puede usarse para gestionar la nueva sesión local de la aplicación.
</Info>

**Consulta la implementación en** [**ASP.NET Core**](/docs/es/get-started/architecture-scenarios/sso-for-regular-web-apps/implementation-aspnetcore#configure-the-cookie-and-oidc-middleware).

<div id="user-logout">
  ## Cierre de sesión del usuario
</div>

Al cerrar la sesión del usuario, deberás volver a tener en cuenta las tres capas de sesión de las que hablamos antes:

* **sesión de la aplicación**: Debes cerrar la sesión del usuario en tu aplicación web eliminando su sesión.
* **sesión de Auth0**: Debes cerrar la sesión del usuario en Auth0. Para ello, redirige al usuario a `https://{yourDomain}/v2/logout`. Redirigir al usuario a esta URL elimina todas las cookies de <Tooltip tip="Inicio de sesión único (SSO): servicio que, después de que un usuario inicia sesión en una aplicación, inicia automáticamente la sesión de ese usuario en otras aplicaciones." cta="Ver glosario" href="/docs/es/glossary?term=single+sign-on">inicio de sesión único</Tooltip> establecidas por Auth0 para el usuario.
* **sesión del proveedor de identidad**: Aunque esto no es habitual, puedes forzar al usuario a cerrar la sesión en el proveedor de identidad utilizado, por ejemplo, Facebook o Google. Para ello, añade un parámetro `federated` de query string a la logout URL: `https://{yourDomain}/v2/logout?federated`.

Para redirigir a un usuario después del cierre de sesión, añade un parámetro `returnTo` de query string con la URL de destino como valor: `https://{yourDomain}/v2/logout?returnTo=http://www.example.com`. Ten en cuenta que tendrás que añadir la URL `returnTo` como una **Allowed Logout URLs**. Para obtener más información sobre cómo implementar esto, consulta: [Cierre de sesión](/docs/es/authenticate/login/logout).

El flujo de cierre de sesión (sin incluir el cierre de sesión federado) es el siguiente:

<Frame>![undefined](/docs/images/cdy7uua7fh8z/5t5iXTeGMUzyKHhqOAGRmp/d51797c6513686ea758f0613d01b55d4/logout-flow.png)</Frame>

1. **Iniciar el flujo de cierre de sesión**: El flujo de cierre de sesión se iniciará desde el navegador; por ejemplo, cuando el usuario haga clic en un enlace de **Cierre de sesión**. Se hará una solicitud al servidor web.
2. **Borrar la sesión local del usuario**: Se borrará la sesión de la aplicación / Cookie del usuario.
3. **Redirigir el navegador al cierre de sesión de Auth0**: El navegador del usuario será redirigido a la logout URL de Auth0.
4. **Borrar la cookie de SSO**: Auth0 borrará la cookie de SSO del usuario.
5. **Redirigir a la URL posterior al cierre de sesión**: Auth0 devolverá una respuesta de redirección y redirigirá el navegador del usuario al parámetro `returnTo` de query string.

**Consulta la implementación en** [**ASP.NET Core**](/docs/es/get-started/architecture-scenarios/sso-for-regular-web-apps/implementation-aspnetcore#implement-the-logout).

<div id="access-control">
  ## Control de acceso
</div>

La autorización se refiere al proceso de determinar qué acciones puede realizar un usuario dentro de tu aplicación.

Puedes implementar la autorización directamente en tu aplicación, de forma independiente de Auth0, o usar una de las opciones disponibles para recuperar los niveles de autorización del usuario, incluirlos como claims de autorización en el <Tooltip tip="token de ID: Credencial destinada al propio cliente, en lugar de acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=ID+Token">token de ID</Tooltip> y validar esos claims dentro de tu aplicación, una vez recuperado el token, para controlar el acceso.

Hay varias formas de recuperar y establecer los claims de autorización del usuario al usar Auth0:

* Configurando y usando la [Auth0 Authorization Extension](/docs/es/customize/extensions/authorization-extension).
* Usando grupos de Active Directory. Estos pueden usarse en combinación con la Authorization Extension al mapear los grupos de Active Directory a los grupos que defines con la Authorization extension.
* Agregando metadata al perfil del usuario mediante [reglas](/docs/es/customize/rules).
* Llamando a servicios externos desde una regla.

Dado que, en nuestro caso, la empresa ya tiene Active Directory configurado, aplicaremos el control de acceso mediante la Authorization Extension en combinación con grupos de Active Directory.

<Card title="Authorization extension">
  En este momento, la Authorization extension está diseñada principalmente para aplicar una autorización poco granular, por ejemplo, para controlar el acceso a una aplicación según la pertenencia de un usuario a un grupo. No está necesariamente diseñada para controlar un acceso más granular (como si un usuario puede realizar una acción específica dentro de la aplicación), aunque así es como la estamos utilizando en este caso.
</Card>

Todos los usuarios serán implícitamente usuarios normales, pero a los administradores de hojas de hora se les asignará un grupo `Admin` que les permitirá aprobar hojas de hora. La Authorization Extension permite mapear grupos a pertenencias a grupos ya existentes.

Todos los administradores de hojas de hora serán asignados al grupo `Timesheet Administrators` en Active Directory, que se mapeará automáticamente al grupo `Admin` dentro de la aplicación Timesheet.

Cuando instalas la Authorization Extension, esta crea una regla en segundo plano que hace lo siguiente:

1. Determinar la pertenencia del usuario a los grupos.
2. Almacenar la información de pertenencia del usuario a los grupos como parte de `app_metadata`.
3. Agregar la pertenencia del usuario a los grupos al token emitido.
4. Verificar que al usuario se le haya concedido acceso a la aplicación actual.

<div id="install-the-authorization-extension">
  ### Instalar la Authorization Extension
</div>

Para instalar la Authorization Extension, ve a la vista [Extensions](https://manage.auth0.com/#/extensions) de tu Auth0 Dashboard y selecciona e instala la extensión Auth0 Authorization.

Una vez instalada, verás la aplicación en la lista de Installed Extensions.

Cuando hagas clic en el enlace para abrir la extensión por primera vez, se te pedirá que concedas permiso para que la extensión acceda a tu cuenta de Auth0. Si lo haces, se te redirigirá al Authorization Dashboard.

Una vez en el Authorization Dashboard, ve a Groups en el menú de navegación y crea un nuevo grupo llamado Admin.

<Frame>![undefined](/docs/images/cdy7uua7fh8z/6zOF0mCrLV2rwdpxn9JD1e/5d6e227c4a96260856afd8f94c4212d9/create-admin-group.png)</Frame>

Después de agregar el grupo, puedes hacer clic en él para ir a la sección de administración del grupo. Ve a la pestaña Group Mappings y agrega un nuevo mapeo de grupo que asigne todos los usuarios de Active Directory de los grupos `Timesheet Admins` al grupo `Admin` que acabas de crear.

<Frame>![undefined](/docs/images/cdy7uua7fh8z/RaMHHJ1G9LoO5xoz3BJnN/b01c93948b1a54b599f1eb106bd8ef26/add-group-mapping.png)</Frame>

Una vez que hagas clic en **Save**, podrás ver el nuevo mapeo en la lista.

<Frame>![undefined](/docs/images/cdy7uua7fh8z/1whRHGlsRhGhA6vcrsElsv/093716ca939c843729c3022c810ee6a7/view-group-mapping.png)</Frame>

Con el mapeo configurado, solo necesitas mantener la membresía del grupo `Timesheet Admins` en Active Directory, y esos usuarios se asignarán automáticamente al grupo `Admin` dentro de nuestra aplicación.

Para obtener más información, consulta la [documentación de Authorization Extension](/docs/es/customize/extensions/authorization-extension).

<div id="enforce-permissions-in-your-application">
  ### Aplica permisos en tu aplicación
</div>

Cuando instalaste la Authorization Extension, también se creó una rule de Auth0 que añadirá un `claim` de `authorization` con toda la configuración relacionada con la autorización de un usuario en particular. Los grupos de un usuario se añadirán como un sub-`claim` del `claim` de `authorization` llamado `groups`, y todos los grupos a los que pertenezca el usuario se incluirán en este `claim` como un arreglo. Este es un ejemplo de cómo podría verse el payload JSON de un token de ID con los grupos incluidos:

```json lines
{
  "sub": "1234567890",
  "name": "John Doe",
  "authorization": {
    "groups": ["Admin"]
  }
}
```

En tu aplicación, tendrás que decodificar el token de ID que se devuelve cuando se autentica un usuario y extraer del claim `authorization` los grupos a los que pertenece. Después, puedes almacenar esos grupos, junto con otra información del usuario, en la sesión del usuario y luego consultarlos para determinar si tiene permisos para realizar una determinada acción en función de su pertenencia a esos grupos.
