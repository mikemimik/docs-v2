Veamos la implementación de nuestra aplicación web tradicional. Usamos ASP .NET Core para esta implementación; puedes encontrar el código en [este repositorio de GitHub](https://github.com/auth0-samples/auth0-pnp-webapp-oidc).

El ejemplo incluye una aplicación que usa la integración con Active Directory para autenticar a los empleados de la empresa y una conexión de base de datos de Auth0 para los contratistas externos. La autorización se implementa mediante reglas y claims, como veremos en detalle en esta sección.

<div id="user-login">
  ## Inicio de sesión de usuario
</div>

Auth0 proporciona un widget de Lock que funciona como componente de inicio de sesión para tu aplicación, lo que significa que no tienes que implementar tu propia pantalla de inicio de sesión. El widget de Lock se integra perfectamente con todas las conexiones que configures en tu <Tooltip tip="Auth0 Dashboard: el principal producto de Auth0 para configurar tus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Auth0+dashboard">Auth0 Dashboard</Tooltip>, ya sean conexiones de base de datos, sociales o empresariales.

Hay varias formas de implementar una pantalla de inicio de sesión con una aplicación web y Auth0:

* **Hosted Lock**: Usa una instancia del widget de Lock alojada en la infraestructura de Auth0.
* **Embedded Lock**: Inserta el widget de Lock en una página web de tu aplicación. Tienes algunas opciones de personalización para el propio widget de Lock y control total sobre el resto del HTML de la página.
* **Interfaz personalizada**: Desarrolla una página web completamente personalizada para la pantalla de inicio de sesión. El formulario HTML personalizado enviará una solicitud `POST` a tu servidor, que a su vez autenticará al usuario mediante la Authentication API. Para obtener más información sobre cuándo usar una interfaz personalizada, consulta [Customize Classic Login Pages with Lock or SDK](/docs/es/customize/login-pages/classic-login/customize-with-lock-sdk).

<div id="automate-home-realm-discovery-hrd">
  ### Automatiza Home Realm Discovery (HRD)
</div>

De forma predeterminada, Lock mostrará todas las conexiones disponibles para el inicio de sesión. Seleccionar los <Tooltip tip="Proveedor de identidad (IdP): servicio que almacena y administra identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=Identity+Providers">proveedores de identidad</Tooltip> adecuados entre varias opciones se denomina Home Realm Discovery (HRD). En nuestro caso, las opciones son autenticarse con Active Directory (para los empleados de la empresa) o usar correo electrónico/contraseña para nuestra conexión de base de datos (contratistas externos).

Sin embargo, quizá quieras evitar ese primer paso, en el que el usuario debe elegir el proveedor de identidad (IdP), y hacer que el sistema lo identifique en lugar de preguntárselo cada vez. Lock te ofrece las siguientes opciones:

* **Identifica el IdP de forma programática**: Cuando inicias una transacción de autenticación con Auth0, opcionalmente puedes enviar un parámetro `connection`. Este valor se corresponde directamente con cualquier conexión definida en tu Dashboard. Al usar la versión alojada de Lock mediante una llamada al endpoint `/authorize`, puedes pasar un parámetro `connection` en la query string que contenga el nombre de la conexión. Como alternativa, si estás usando Embedded Lock, es tan simple como escribir `auth0.show({connections: ['{yourConnection}']});`.

  * Hay varias formas prácticas de obtener el valor de `connection`. Una de ellas es usar **vanity URLs**: por ejemplo, los empleados de la empresa usarán `https://internal.yoursite.com`, mientras que los contratistas externos usarán `https://external.yoursite.com`.
* **Usa dominios de correo electrónico**: Lock puede usar dominios de correo electrónico como una forma de enrutar solicitudes de autenticación. Las conexiones empresariales en Auth0 se pueden asignar a `domains`. Si una conexión tiene esta configuración, el campo de contraseña se desactiva automáticamente al escribir un correo electrónico con un dominio asignado. Ten en cuenta que puedes asociar varios dominios a una sola conexión.

Para obtener información adicional sobre este tema, consulta [Select from Multiple Connection Options](/docs/es/libraries/lock/selecting-from-multiple-connection-options).

<div id="session-management">
  ## Gestión de sesiones
</div>

Cuando hablamos de gestionar sesiones, normalmente hay tres capas de sesión que debemos tener en cuenta:

* **Sesión de la aplicación**: La primera es la sesión dentro de la aplicación. Aunque tu aplicación use Auth0 para autenticar a los usuarios, igualmente tendrás que llevar un control de que el usuario ha iniciado sesión en tu aplicación. En una aplicación web normal, esto se consigue almacenando información en una cookie.
* **sesión de Auth0**: A continuación, Auth0 también mantendrá una sesión y almacenará la información del usuario en una cookie. La próxima vez que se redirija a un usuario a la pantalla de Auth0 Lock, se recordará la información del usuario.
* **sesión del proveedor de identidad**: La última capa es la del proveedor de identidad, por ejemplo, Facebook o Google. Cuando permites que los usuarios inicien sesión con cualquiera de estos proveedores y ya han iniciado sesión en el proveedor, no se les pedirá que vuelvan a iniciar sesión. Puede que simplemente se les pida conceder permisos para compartir su información con Auth0 y, a su vez, con tu aplicación.

Al desarrollar una aplicación web, por lo tanto, tendrás que llevar un control de que el usuario ha iniciado sesión en tu aplicación web. Puedes hacerlo mediante una sesión basada en cookies para registrar que el usuario ha iniciado sesión, y también almacenar cualquier información relacionada con el usuario o los tokens.

<Info>
  ### ¿Cómo controlo la duración de la sesión local de la aplicación del usuario? ¿Puedo gestionarla desde Auth0?

  La aplicación web tiene control total sobre la sesión local de la aplicación del usuario. La forma de hacerlo suele depender de la pila web que se esté usando (por ejemplo, ASP.NET). En cualquier caso, todos los enfoques terminan usando una o más cookies para controlar la sesión. El desarrollador puede optar por usar la expiración del token de ID JWT devuelto por Auth0 para controlar la duración de su sesión o ignorarla por completo. Algunos desarrolladores almacenan el propio token de ID en el estado de la sesión y finalizan la sesión del usuario cuando expira.

  La razón para usar la expiración del token para determinar la expiración de la sesión local es que te da un control centralizado sobre la duración de la sesión de un usuario desde el Auth0 Dashboard.
</Info>

El flujo de inicio de sesión es el siguiente:

<Frame>![undefined](/docs/images/cdy7uua7fh8z/4bqozVk6fF4JrWRP1BJK7Y/1403eb1c0efb12552307358a26c6e7f7/login-flow.png)</Frame>

1. **Iniciar flujo de autenticación OIDC**: El navegador del usuario enviará una solicitud a Auth0 para iniciar el flujo de OIDC.
2. **Establecer cookie de SSO**: Auth0 establecerá una cookie para almacenar la información del usuario.
3. **Intercambio de código y devolución del token de ID**: Auth0 hará una solicitud de vuelta al servidor web y devolverá el código. El servidor web intercambiará el código por un token de ID.
4. **Establecer cookie de autenticación y enviar respuesta**: El servidor web enviará una respuesta de vuelta al navegador y establecerá la cookie de autenticación de la aplicación para almacenar la información de la sesión del usuario.
5. **La cookie de autenticación se envía con cada solicitud posterior**: La cookie de autenticación de la aplicación se enviará con cada solicitud posterior como prueba de que el usuario está autenticado.

<Info>
  ### ¿Cómo afecta la sesión de SSO de Auth0 a la sesión de la aplicación?

  Auth0 gestiona su propia sesión de inicio de sesión único. Las aplicaciones pueden optar por respetar o ignorar esa sesión de SSO a la hora de mantener su propia sesión local. El widget de Lock incluso tiene una funcionalidad especial con la que puede detectar si existe una sesión de SSO de Auth0 y preguntarle al usuario si desea volver a iniciar sesión como ese mismo usuario.

  <Frame>![Lock Widget SSO](https://cdn2.auth0.com/docs/1.14516.0/media/articles/architecture-scenarios/web-app-sso/sso-login.png)</Frame>

  Si lo hace, iniciará sesión sin tener que volver a introducir sus credenciales con el IdP real. Aunque el usuario no se haya autenticado, la aplicación sigue realizando un flujo de autenticación con Auth0 y obtiene un nuevo token de ID, que luego puede usarse para gestionar la nueva sesión local de la aplicación.
</Info>

**Consulta la implementación en** [**ASP.NET Core**](/docs/es/get-started/architecture-scenarios/sso-for-regular-web-apps/implementation-aspnetcore#configure-the-cookie-and-oidc-middleware).

<div id="user-logout">
  ## Cierre de sesión del usuario
</div>

Al cerrar la sesión del usuario, una vez más deberá tener en cuenta las tres capas de sesiones de las que hablamos antes:

* **Sesión de la aplicación**: Debe cerrar la sesión del usuario en su aplicación web, borrando su sesión.
* **Sesión de Auth0**: Debe cerrar la sesión del usuario en Auth0. Para hacerlo, redirija al usuario a `https://{yourDomain}/v2/logout`. Redirigir al usuario a esta URL borra todas las cookies de <Tooltip tip="Inicio de sesión único (SSO): servicio que, después de que un usuario inicia sesión en una aplicación, le permite acceder automáticamente a otras aplicaciones." cta="Ver glosario" href="/docs/es/glossary?term=single+sign-on">inicio de sesión único</Tooltip> establecidas por Auth0 para el usuario.
* **Sesión del proveedor de identidad**: Aunque esto no es una práctica común, puede forzar al usuario a cerrar sesión en el proveedor de identidad utilizado, por ejemplo, Facebook o Google. Para hacerlo, agregue un parámetro de cadena de consulta `federated` a la URL de cierre de sesión: `https://{yourDomain}/v2/logout?federated`.

Para redirigir a un usuario después del cierre de sesión, agregue un parámetro de cadena de consulta `returnTo` con la URL de destino como valor: `https://{yourDomain}/v2/logout?returnTo=http://www.example.com`. Tenga en cuenta que deberá agregar la URL de `returnTo` a **Allowed cierre de sesión URLs**. Para obtener más información sobre cómo implementar esto, consulte: [cierre de sesión](/docs/es/authenticate/login/logout).

El flujo de cierre de sesión (sin incluir el cierre de sesión federado) es el siguiente:

<Frame>![undefined](/docs/images/cdy7uua7fh8z/5t5iXTeGMUzyKHhqOAGRmp/d51797c6513686ea758f0613d01b55d4/logout-flow.png)</Frame>

1. **Iniciar el flujo de cierre de sesión**: El flujo de cierre de sesión se iniciará desde el navegador, por ejemplo, cuando el usuario haga clic en un enlace de **Cerrar sesión**. Se hará una solicitud al servidor web.
2. **Borrar la sesión local del usuario**: Se borrará la sesión de la aplicación o la cookie del usuario.
3. **Redirigir el navegador al cierre de sesión de Auth0**: El navegador del usuario será redirigido a la URL de cierre de sesión de Auth0.
4. **Borrar la cookie de SSO**: Auth0 borrará la cookie de SSO del usuario.
5. **Redirigir a la URL posterior al cierre de sesión**: Auth0 devolverá una respuesta de redirección y redirigirá el navegador del usuario al parámetro de cadena de consulta `returnTo`.

**Vea la implementación en** [**ASP.NET Core**](/docs/es/get-started/architecture-scenarios/sso-for-regular-web-apps/implementation-aspnetcore#implement-the-logout).

<div id="access-control">
  ## Control de acceso
</div>

La autorización se refiere al proceso de determinar qué acciones puede realizar un usuario dentro de su aplicación.

Puede implementar la autorización directamente en su aplicación, de forma independiente de Auth0, o utilizar una de las opciones disponibles para recuperar los niveles de autorización del usuario, incluirlos como claims de autorización en el <Tooltip tip="token de ID: Credencial destinada al propio cliente, en lugar de para acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=ID+Token">token de ID</Tooltip> y validar esos claims dentro de su aplicación, una vez recuperado el token, para controlar el acceso.

Existen varias formas de recuperar y establecer los claims de autorización del usuario cuando se usa Auth0:

* Configurando y usando la [Auth0 Authorization Extension](/docs/es/customize/extensions/authorization-extension).
* Usando grupos de Active Directory. Estos se pueden usar en combinación con la Authorization Extension mediante el mapeo de grupos de Active Directory a grupos que usted defina con la Authorization Extension.
* Agregando metadata al perfil del usuario mediante [reglas](/docs/es/customize/rules).
* Llamando a servicios externos desde una rule.

Como en nuestro caso la empresa ya tiene Active Directory configurado, aplicaremos el control de acceso usando la Authorization Extension en combinación con grupos de Active Directory.

<Card title="Authorization Extension">
  En este momento, la Authorization Extension está diseñada principalmente para aplicar una autorización de grano grueso; por ejemplo, para controlar el acceso a una aplicación en función de la pertenencia del usuario a un grupo. No está necesariamente diseñada para controlar el acceso de grano fino (como determinar si un usuario puede realizar una acción específica dentro de la aplicación), aunque en este caso la utilizaremos de esa manera.
</Card>

Todos los usuarios serán implícitamente usuarios normales, pero los administradores de hoja de hora se asignarán a un grupo `Admin`, lo que les permitirá aprobar hojas de hora. La Authorization Extension permite mapear grupos a pertenencias de grupo existentes.

Todos los administradores de hoja de hora se asignarán al grupo `Timesheet Administrators` de Active Directory, que se mapeará automáticamente al grupo `Admin` dentro de la aplicación Timesheet.

Cuando instala la Authorization Extension, esta crea una rule en segundo plano, que hace lo siguiente:

1. Determina la pertenencia del usuario a un grupo.
2. Almacena la información sobre la pertenencia del usuario a un grupo como parte de `app_metadata`.
3. Agrega la pertenencia del usuario a un grupo al token de salida.
4. Verifica que al usuario se le haya concedido acceso a la aplicación actual.

<div id="install-the-authorization-extension">
  ### Instala la Authorization Extension
</div>

Para instalar la Authorization Extension, ve a la vista de [Extensions](https://manage.auth0.com/#/extensions) en tu Auth0 Dashboard, selecciona la Auth0 Authorization Extension e instálala.

Una vez instalada, verás la aplicación en la lista de Installed Extensions.

Cuando hagas clic en el enlace para abrir la extensión por primera vez, se te pedirá que concedas permiso para que la extensión acceda a tu cuenta de Auth0. Si lo haces, se te redirigirá al Authorization Dashboard.

Una vez en el Authorization Dashboard, ve a Groups en el menú de navegación y crea un grupo nuevo llamado Admin.

<Frame>![undefined](/docs/images/cdy7uua7fh8z/6zOF0mCrLV2rwdpxn9JD1e/5d6e227c4a96260856afd8f94c4212d9/create-admin-group.png)</Frame>

Después de agregar el grupo, puedes hacer clic en él para ir a la sección de administración del grupo. Ve a la pestaña Group Mappings y agrega un nuevo mapeo de grupos que asigne todos los usuarios de Active Directory de los grupos `Timesheet Admins` al grupo `Admin` que acabas de crear.

<Frame>![undefined](/docs/images/cdy7uua7fh8z/RaMHHJ1G9LoO5xoz3BJnN/b01c93948b1a54b599f1eb106bd8ef26/add-group-mapping.png)</Frame>

Una vez que hagas clic en **Save** podrás ver el nuevo mapeo en la lista.

<Frame>![undefined](/docs/images/cdy7uua7fh8z/1whRHGlsRhGhA6vcrsElsv/093716ca939c843729c3022c810ee6a7/view-group-mapping.png)</Frame>

Con el mapeo configurado, solo necesitas mantener la membresía del grupo `Timesheet Admins` en Active Directory, y esos usuarios se asignarán automáticamente al grupo `Admin` dentro de nuestra aplicación.

Para obtener más información, consulta la [documentación de la Authorization Extension](/docs/es/customize/extensions/authorization-extension).

<div id="enforce-permissions-in-your-application">
  ### Aplica permisos en tu aplicación
</div>

Cuando instalaste la Authorization Extension, también se creó una Rule de Auth0 que añadirá un claim `authorization` con toda la configuración relacionada con la autorización para un usuario determinado. Los grupos de un usuario se agregarán como un subclaim del claim `authorization` llamado `groups`, y todos los grupos a los que pertenece ese usuario se agregarán como un arreglo en este claim. Este es un ejemplo de cómo puede verse el payload JSON de un token de ID con los grupos incluidos:

```json lines
{
  "sub": "1234567890",
  "name": "John Doe",
  "authorization": {
    "groups": ["Admin"]
  }
}
```

En tu aplicación, por tanto, tendrás que decodificar el token de ID que se devuelve cuando se autentica a un usuario y extraer de la claim `authorization` los grupos a los que pertenece. Después, puedes almacenar esos grupos, junto con otra información del usuario, en la sesión del usuario y, posteriormente, consultarlos para determinar si tiene permisos para realizar una acción concreta en función de su pertenencia a esos grupos.
