En esta sección veremos cómo podemos implementar una API para nuestro caso de uso.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Para simplificar, mantendremos nuestra implementación centrada exclusivamente en la autenticación y autorización. Como verás en los ejemplos, la hoja de horas de entrada estará fija en el código, y la API no guardará la hoja de horas. En su lugar, simplemente devolverá parte de la información.
</Callout>

<div id="define-the-api-endpoints">
  ## Defina los endpoints de la API
</div>

Primero debemos definir los endpoints de nuestra API.

<Card title="¿Qué es un endpoint de API?">
  Un **API endpoint** es una URL única que representa un objeto. Para interactuar con este objeto, debes dirigir tu aplicación a su URL. Por ejemplo, si tuvieras una API que pudiera devolver pedidos o clientes, podrías configurar dos endpoints: `/orders` y `/customers`. Tu aplicación interactuaría con estos endpoints mediante distintos métodos HTTP; por ejemplo, `POST /orders` podría crear un pedido nuevo y `GET /orders` podría recuperar el conjunto de datos de uno o más pedidos.
</Card>

Para esta implementación, solo definiremos 2 endpoints: uno para recuperar una lista de todas las hojas de hora de un Employee y otro que permitirá a un Employee crear una nueva hoja de horas.

Una solicitud `HTTP GET` al endpoint `/timesheets` permitirá a un usuario recuperar sus hojas de hora, y una solicitud `HTTP POST` al endpoint `/timesheets` permitirá a un usuario agregar una nueva hoja de hora.

**Consulte la implementación en** [**Node.js**](/docs/es/get-started/architecture-scenarios/spa-api/api-implementation-nodejs#1-define-the-api-endpoints).

<div id="secure-the-endpoints">
  ### Protege los Endpoints
</div>

Cuando una API recibe una solicitud con un <Tooltip tip="Token de acceso: credencial de autorización, en forma de una cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=Access+Token">token de acceso</Tooltip> de tipo bearer como parte del encabezado, lo primero que debe hacer es validar el token. Esto consiste en una serie de pasos y, si alguno de ellos falla, la solicitud debe rechazarse con un mensaje de error `Missing or invalid token` para la app que realiza la llamada.

Las validaciones que debe realizar la API son:

* Comprobar que el <Tooltip tip="JSON Web Token (JWT): formato estándar de token de ID (y, a menudo, formato de token de acceso) utilizado para representar claims de forma segura entre dos partes." cta="Ver glosario" href="/docs/es/glossary?term=JWT">JWT</Tooltip> tenga el formato correcto
* Comprobar la firma
* Validar los claims estándar

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  [JWT.io](https://jwt.io/) proporciona una lista de bibliotecas que pueden hacer la mayor parte del trabajo por ti: analizar el JWT, verificar la firma y los claims.
</Callout>

Parte del proceso de validación también consiste en comprobar los permisos de la aplicación (scopes), pero lo trataremos por separado en el siguiente párrafo de este documento.

Para obtener más información sobre cómo validar tokens de acceso, consulta [Validar tokens de acceso](/docs/es/secure/tokens/access-tokens/validate-access-tokens).

**Consulta la implementación en** [**Node.js**](/docs/es/get-started/architecture-scenarios/spa-api/api-implementation-nodejs#2-secure-the-api-endpoints).

<div id="check-the-applications-permissions">
  ### Comprobar los permisos de la aplicación
</div>

A estas alturas, ya hemos verificado que el JWT es válido. El último paso es verificar que la aplicación tenga los permisos necesarios para acceder a los recursos protegidos.

Para ello, la API debe comprobar los [scopes](/docs/es/get-started/apis/scopes) del JWT decodificado. Este claim forma parte del payload y consiste en una lista de cadenas separadas por espacios.

**Consulta la implementación en** [**Node.js**](/docs/es/get-started/architecture-scenarios/spa-api/api-implementation-nodejs#3-check-the-client-permissions).

<div id="determine-user-identity">
  ### Determinar la identidad del usuario
</div>

Para ambos endpoints (recuperar la lista de hojas de hora y agregar una nueva hoja de hora), necesitaremos determinar la identidad del usuario.

Al recuperar la lista de hojas de hora, esto garantiza que solo devolvamos las hojas de hora que pertenecen al usuario que realiza la solicitud; y al agregar una nueva hoja de hora, garantiza que la hoja de hora quede asociada al usuario que realiza la solicitud.

Uno de los claims estándar de JWT es el claim `sub`, que identifica al principal que es el subject del claim. En el caso del flujo de concesión implícita, este claim contendrá la identidad del usuario, que será el identificador único del usuario de Auth0. Puedes usarlo para asociar cualquier información en sistemas externos con un usuario en particular.

También puedes usar un custom claim para agregar otro atributo del usuario, como su dirección de correo electrónico, al Access Token y usarlo para identificar al usuario de forma única.

**Consulta la implementación en** [**Node.js**](/docs/es/get-started/architecture-scenarios/spa-api/api-implementation-nodejs#4-determine-the-user-identity).

<div id="implement-the-spa">
  ## Implementar la SPA
</div>

En esta sección veremos cómo implementar una SPA para nuestro caso.

<div id="authorize-the-user">
  ### Autorizar al usuario
</div>

Para autorizar al usuario, utilizaremos la [biblioteca auth0.js](/docs/es/libraries/auth0js). Puedes inicializar una nueva instancia de la aplicación de Auth0 de la siguiente manera:

export const codeExample = `var auth0 = new auth0.WebAuth({
  clientID: '{yourClientId}',
  domain: '{yourDomain}',
  responseType: 'token id_token',
  audience: 'YOUR_API_IDENTIFIER',
  redirectUri: '{https://yourApp/callback}',
  scope: 'openid profile read:timesheets create:timesheets'
});`;

<AuthCodeBlock children={codeExample} language="javascript" />

Debes pasar los siguientes valores de configuración:

* **clientID**: El valor de tu <Tooltip tip="ID de cliente: Valor de identificación asignado por Auth0 a tu recurso registrado." cta="Ver glosario" href="/docs/es/glossary?term=Client+Id">ID de cliente</Tooltip> de Auth0. Puedes obtenerlo en Settings de tu aplicación en el [Dashboard](https://manage.auth0.com/#/applications%7D).
* **domain**: El valor de tu dominio de Auth0. Puedes obtenerlo en Settings de tu aplicación en el [Dashboard](https://manage.auth0.com/#/applications%7D).
* **responseType**: Indica el flujo de autenticación que se va a usar. Para una SPA que utiliza el **flujo implícito**, debe establecerse en `token id_token`. La parte `token` hace que el flujo devuelva un token de acceso en el fragmento de la URL, mientras que la parte `id_token` hace que el flujo también devuelva un <Tooltip tip="token de ID: Credencial destinada al propio cliente, en lugar de usarse para acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=ID+Token">token de ID</Tooltip>.
* **<Tooltip tip="Audience: Identificador único de la audiencia de un token emitido. Se llama aud en un token; su valor contiene el ID de una aplicación (ID de cliente) para un token de ID o de una API (identificador de API) para un token de acceso." cta="Ver glosario" href="/docs/es/glossary?term=audience">audience</Tooltip>**: El valor de tu identificador de API. Puedes obtenerlo en los [Settings de tu API](https://manage.auth0.com/#/apis%7D) en el Dashboard.
* **redirectUri**: La URL a la que Auth0 debe redirigir después de que el usuario se haya autenticado.
* **scope**: Los [scopes](/docs/es/get-started/apis/scopes) que determinan la información que se devolverá en el token de ID y el token de acceso. Un scope de `openid profile` devolverá toda la información del perfil del usuario en el token de ID. También debes solicitar los scopes necesarios para llamar a la API; en este caso, los scopes `read:timesheets create:timesheets`. Esto garantizará que el token de acceso tenga estos scopes.

Para iniciar el flujo de autenticación, puedes llamar al método `authorize()`:

```js lines
auth0.authorize();
```

Después de la autenticación, Auth0 te redirigirá a **redirectUri** que especificaste al configurar la nueva instancia de la aplicación de Auth0. En este punto, tendrás que llamar al método `parseHash()`, que analiza un fragmento hash de la URL para extraer el resultado de una respuesta de autenticación de Auth0.

El contenido del objeto authResult que devuelve parseHash depende de los parámetros de autenticación utilizados. Puede incluir lo siguiente:

* **idToken**: Un JWT de token de ID que contiene información del perfil del usuario
* **accessToken**: Un token de acceso para la API especificada por el **audience**.
* **expiresIn**: Una cadena que contiene el tiempo de expiración (en segundos) del token de acceso.

Determina dónde es mejor [almacenar los tokens](/docs/es/secure/security-guidance/data-security/token-storage). Si tu aplicación web de página única tiene algún servidor backend, los tokens deben gestionarse en el servidor mediante el [flujo de código de autorización](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow) o el [flujo de código de autorización con Proof Key for Code Exchange (PKCE)](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce).

Si tienes una aplicación web de página única (SPA) sin un servidor backend correspondiente, tu SPA debe solicitar nuevos tokens al iniciar sesión y almacenarlos en memoria sin persistirlos. Para hacer llamadas a la API, tu SPA usaría la copia del token almacenada en memoria.

Para ver un ejemplo de cómo manejar sesiones en las SPA, consulta la sección [Manejar tokens de autenticación](/docs/es/quickstart/spa/vanillajs#handle-authentication-tokens) del [Quickstart de aplicación web de página única en JavaScript](/docs/es/quickstart/spa/vanillajs).

**Consulta la implementación en** [**Angular 2**](/docs/es/get-started/architecture-scenarios/spa-api/spa-implementation-angular2#2-authorize-the-user).

<div id="get-the-user-profile">
  ### Obtener el perfil del usuario
</div>

<Card title="Extraer información del token">
  En esta sección se muestra cómo obtener la información del usuario mediante el token de acceso y el [endpoint /userinfo](https://auth0.com/docs/api/authentication#get-user-info). Para evitar esta llamada a la API, puedes simplemente decodificar el token de ID [con una biblioteca](https://jwt.io/#libraries-io) (asegúrate de validarlo primero). Si necesitas información adicional del usuario, considera usar [nuestra Management API](https://auth0.com/docs/api/management/v2#!/Users/get_users_by_id) desde tu backend.
</Card>

Se puede llamar al método `client.userInfo` pasándole el valor devuelto en `authResult.accessToken` para obtener la información del perfil del usuario. Este hará una solicitud al [endpoint /userinfo](https://auth0.com/docs/api/authentication#get-user-info) y devolverá el objeto `user`, que contiene la información del usuario, como en el siguiente ejemplo:

```json lines
{
    "email_verified": "false",
    "email": "test@example.com",
    "clientID": "AAAABBBBCCCCDDDDEEEEFFFFGGGGHHHH",
    "updated_at": "2017-02-07T20:50:33.563Z",
    "name": "tester9@example.com",
    "picture": "https://gravatar.com/avatar/example.png",
    "user_id": "auth0|123456789012345678901234",
    "nickname": "tester9",
    "created_at": "2017-01-20T20:06:05.008Z",
    "sub": "auth0|123456789012345678901234"
}
```

Puedes acceder a cualquiera de estas propiedades desde la función de devolución de llamada que se pasa al llamar a la función `userInfo`:

```javascript lines
const accessToken = authResult.accessToken;

auth0.client.userInfo(accessToken, (err, profile) => {
  if (profile) {
    // Obtener el nickname y la foto de perfil del usuario
    var nickname = profile.nickname;
    var picture = profile.picture;
  }
});
```

**Consulta la implementación en** [**Angular 2**](/docs/es/get-started/architecture-scenarios/spa-api/spa-implementation-angular2#3-get-the-user-profile).

<div id="display-ui-elements-conditionally-based-on-scope">
  ### Mostrar elementos de la UI de forma condicional según el scope
</div>

Según el `scope` del usuario, es posible que quieras mostrar u ocultar ciertos elementos de la UI. Para determinar el `scope` emitido para un usuario, tendrás que almacenar el `scope` que se solicitó inicialmente durante el proceso de autorización. Cuando se autorice a un usuario, el `scope` también se devolverá en `authResult`.

Si el `scope` en `authResult` está vacío, significa que se concedieron todos los scopes solicitados. Si el `scope` en `authResult` no está vacío, significa que se concedió un conjunto distinto de scopes, y debes usar los de `authResult.scope`.

**Consulta la implementación en** [**Angular 2**](/docs/es/get-started/architecture-scenarios/spa-api/spa-implementation-angular2#4-display-ui-elements-conditionally-based-on-scope).

<div id="call-the-api">
  ### Llamar a la API
</div>

Para acceder a los recursos protegidos de tu API, el token de acceso del usuario autenticado debe incluirse en las solicitudes que se envían a la API. Esto se logra enviando el token de acceso en el encabezado `Authorization` con el esquema `Bearer`.

**Consulta la implementación en** [**Angular 2**](/docs/es/get-started/architecture-scenarios/spa-api/spa-implementation-angular2#5-call-the-api).

<div id="renew-the-access-token">
  ### Renovar el token de acceso
</div>

Como medida de seguridad, se recomienda que la duración del token de acceso de un usuario sea breve. Cuando creas una API en el <Tooltip tip="Auth0 Dashboard: el producto principal de Auth0 para configurar tus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Auth0+dashboard">Auth0 Dashboard</Tooltip>, la duración predeterminada es de `7200` segundos (2 horas), aunque esto puede configurarse para cada API.

Una vez que expira, un token de acceso ya no puede usarse para acceder a una API. Para volver a obtener acceso, es necesario conseguir un nuevo token de acceso.

Para obtener un nuevo token de acceso, puedes repetir el flujo de autenticación que se utilizó para obtener el token de acceso inicial. En una SPA, esto no es lo ideal, ya que quizá no quieras redirigir al usuario fuera de la tarea que está realizando para completar de nuevo el flujo de autenticación.

En casos como este, puedes usar la [autenticación silenciosa](/docs/es/authenticate/login/configure-silent-authentication). La autenticación silenciosa te permite realizar un flujo de autenticación en el que Auth0 solo responderá con redirecciones y nunca con una página de inicio de sesión. Sin embargo, esto requiere que el usuario ya haya iniciado sesión mediante [inicio de sesión único (SSO)](/docs/es/authenticate/single-sign-on).

**Consulta la implementación en** [**Angular 2**](/docs/es/get-started/architecture-scenarios/spa-api/spa-implementation-angular2#6-renew-the-access-token).
