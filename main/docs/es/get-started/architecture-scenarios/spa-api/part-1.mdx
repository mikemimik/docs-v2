Para garantizar que solo los usuarios y las aplicaciones autorizados puedan acceder a la API de parte de horas, ExampleCo ha decidido hacer uso del [marco de autorización de OAuth 2.0](https://tools.ietf.org/html/rfc6749). Este marco ofrece la flexibilidad que la empresa busca, ya que los distintos tipos de concesión le permiten autorizar fácilmente los diversos tipos de aplicaciones que necesitan comunicarse con la API de parte de horas.

<div id="api-authentication-and-authorization">
  ## Autenticación y autorización de API
</div>

Una API es una forma de exponer la funcionalidad de tu aplicación a otras aplicaciones. Una aplicación puede hacer una solicitud enviando un mensaje a un endpoint de una API y recibir información como respuesta.

Un API endpoint puede estar protegido o no. En nuestro caso, dado que las hojas de hora son información sensible que afecta a revisiones y pagos, es importante garantizar que solo los usuarios y las aplicaciones autorizados puedan llamar a los endpoints de nuestra API. Cuando una aplicación cliente quiere acceder a endpoints protegidos de una API, necesita presentar un <Tooltip tip="Access Token: Credencial de autorización, en forma de una cadena opaque o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=Access+Token">token de acceso</Tooltip> como prueba de que tiene los permisos necesarios para llamar al endpoint.

Un token de acceso se obtiene autenticando al usuario con un <Tooltip tip="Authorization Server: Servidor centralizado que contribuye a definir los límites del acceso de un usuario. Por ejemplo, tu servidor de autorización puede controlar los datos, tareas y funcionalidades disponibles para un usuario." cta="Ver glosario" href="/docs/es/glossary?term=Authorization+Server">servidor de autorización</Tooltip>, y el usuario puede luego autorizar a la aplicación para que acceda a la API en su nombre.

<Card title="¿Qué es un token de acceso?">
  Un token de acceso (también denominado `access_token`) es una cadena opaque que representa una autorización emitida a la aplicación. Puede representar un identificador utilizado para recuperar la información de autorización o puede contener en sí mismo la información de autorización (por ejemplo, la identidad del usuario, los permisos, etc.) de forma verificable.

  Es bastante común que los tokens de acceso se implementen como [JSON Web Tokens](/docs/es/secure/tokens/json-web-tokens).

  Para obtener más información sobre los tokens de acceso de Auth0, consulta [token de acceso](/docs/es/secure/tokens/access-tokens).
</Card>

Una API puede aplicar un control detallado sobre quién puede acceder a los distintos endpoints expuestos por la API. Estos permisos se expresan como scopes.

Cuando un usuario autoriza una aplicación cliente, la aplicación también puede indicar qué permisos requiere. Luego, el usuario puede revisar y conceder esos permisos. Estos permisos se incluyen entonces en el token de acceso como parte del claim `scope`.

Posteriormente, cuando el cliente envía el token de acceso al hacer solicitudes a la API, esta puede inspeccionar el claim `scope` para garantizar que se hayan concedido los permisos necesarios para llamar al API endpoint en cuestión.

<Card title="¿Qué son los Scopes?">
  Cada token de acceso puede incluir una lista de los permisos que se le han concedido al cliente. Cuando un cliente se autentica con Auth0, especificará la lista de scopes (o permisos) que está solicitando. Si esos scopes están autorizados, entonces el token de acceso contendrá una lista de scopes autorizados.

  Por ejemplo, la API de hojas de hora puede aceptar cuatro niveles distintos de autorización: leer hojas de hora (scope `read:timesheets`), crear hojas de hora (scope `create:timesheets`), eliminar hojas de hora (scope `delete:timesheets`) y aprobar hojas de hora (scope `approve:timesheets`).

  Cuando un cliente pide a la API que cree una nueva hoja de hora, el token de acceso debe contener el scope `create:timesheets`. Del mismo modo, para eliminar hojas de hora existentes, el token de acceso debe contener el scope `delete:timesheets`.

  Para obtener más información sobre los scopes, consulta [Scopes](/docs/es/get-started/apis/scopes).
</Card>

Al usar el marco de autorización <Tooltip tip="OAuth 2.0: Marco de autorización que define protocolos y flujos de trabajo de autorización." cta="Ver glosario" href="/docs/es/glossary?term=OAuth+2.0">OAuth 2.0</Tooltip>, puedes dar a tus propias aplicaciones o a aplicaciones de terceros acceso limitado a tus APIs en nombre de la propia aplicación. Con Auth0, puedes admitir fácilmente diferentes flujos en tus propias APIs sin preocuparte por la especificación de OAuth 2.0/<Tooltip tip="OpenID: Estándar abierto de autenticación que permite a las aplicaciones verificar las identidades de los usuarios sin recopilar ni almacenar información de inicio de sesión." cta="Ver glosario" href="/docs/es/glossary?term=OpenID">OpenID</Tooltip> Connect (OIDC) ni por los muchos otros aspectos técnicos de la autorización de API.

<Card title="Roles de OAuth">
  En cualquier flujo de OAuth 2.0, podemos identificar los siguientes roles:

  * **Resource Owner**: la entidad que puede conceder acceso a un recurso protegido. Normalmente, es el usuario final.
  * **Resource Server**: el servidor que aloja los recursos protegidos. Es la API a la que quieres acceder.
  * **Cliente**: una aplicación que solicita acceso a un recurso protegido en nombre del Resource Owner.
  * **Servidor de autorización**: el servidor que autentica al Resource Owner y emite tokens de acceso tras obtener la autorización adecuada. En este caso, la Authentication API de Auth0.

  Los [tipos de concesión (o flujos)](/docs/es/get-started/authentication-and-authorization-flow/which-oauth-2-0-flow-should-i-use) determinan cómo interactúan estos participantes para conceder a las aplicaciones acceso limitado a las API que estás creando. Como resultado, la aplicación obtendrá un token de acceso que podrá usar para llamar a la API en nombre del usuario.
</Card>

<div id="implicit-grant">
  ## Concesión implícita
</div>

OAuth 2.0 ofrece varios **tipos de concesión** para diferentes casos de uso. En este caso concreto, queremos acceder a la API desde una [aplicación del lado del cliente](/docs/es/quickstart/spa).

La SPA usará el [flujo implícito (concesión implícita)](/docs/es/get-started/authentication-and-authorization-flow/implicit-flow-with-form-post) para hacerlo.

La concesión implícita (definida en [RFC 6749, sección 4.1](https://tools.ietf.org/html/rfc6749#section-4.2)) es similar a la concesión utilizada en el [flujo de código de autorización](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow), pero la principal diferencia es que la aplicación recibe un token de acceso directamente, sin necesidad de un `authorization_code`. Esto ocurre porque la aplicación, que normalmente es una aplicación JavaScript que se ejecuta en un navegador, ofrece menos garantías que una web app que se ejecuta en el servidor y, por tanto, no se le puede confiar el `client_secret` (que es obligatorio en la concesión Authorization Code).

Una vez que el usuario se autentica, la aplicación recibe el <Tooltip tip="token de ID: credencial destinada al propio cliente, en lugar de usarse para acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=ID+Token">token de ID</Tooltip> y el token de acceso en el fragmento de hash del URI. La aplicación puede usar ahora el token de ID para obtener información sobre el usuario y el token de acceso para llamar a la API en nombre del usuario.

1. La app inicia el flujo y redirige el navegador a Auth0 (específicamente al [endpoint /authorize](https://auth0.com/docs/api/authentication#implicit-grant)), para que el usuario pueda autenticarse.
2. Auth0 autentica al usuario. La primera vez que el usuario pasa por este flujo, y si la aplicación es de terceros, se mostrará una página de consentimiento en la que se enumeran los permisos que se otorgarán al cliente (por ejemplo, publicar mensajes, listar contactos, etc.).
3. Auth0 redirige al usuario a la app con un token de acceso (y, opcionalmente, un token de ID) en el fragmento de hash del URI. La app puede ahora extraer los tokens de ese fragmento.
4. La app puede usar el token de acceso para llamar a la API en nombre del usuario.

<div id="authorization-extension">
  ## Authorization Extension
</div>

La [Auth0 Authorization Extension](/docs/es/customize/extensions/authorization-extension) te permite configurar Roles, Groups y Permissions, y asignarlos a Users.

* Los Permissions son las acciones que alguien puede realizar. Para las necesidades de negocio de ExampleCo, configuraremos cuatro Permissions: leer, crear, eliminar y aprobar hojas de hora.
* Los Roles son conjuntos de Permissions. La aplicación de hojas de hora de ExampleCo será utilizada por dos tipos de usuarios (empleados y gerentes), cada uno con permisos diferentes, por lo que configuraremos dos Roles: employee y manager.

Como esto cubre nuestro caso de uso, no crearemos ningún Group.

La Authorization Extension creará una [Rule](/docs/es/customize/rules) que leerá los Roles, Groups y Permissions asignados a un usuario y añadirá esta información al [perfil de usuario](/docs/es/customize/rules#rule-syntax) durante el flujo de autenticación. Podemos usar esta información para asegurarnos de que el token de acceso emitido a un usuario solo contenga los scopes permitidos. Más adelante podremos personalizar nuestra aplicación, por ejemplo, deshabilitando la funcionalidad de aprobar hojas de hora si el usuario no tiene el permiso necesario para hacerlo.
