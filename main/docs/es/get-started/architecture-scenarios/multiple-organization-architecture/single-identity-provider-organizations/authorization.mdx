---

Al pensar en la autorización, normalmente debe considerar cómo determina qué puede hacer una persona y cómo comunica esa información a sus aplicaciones y/o API. Según las aplicaciones que tenga, es posible que esto le afecte en uno o en ambos aspectos. En nuestros escenarios de arquitectura, ofrecemos orientación general sobre la [autorización B2B](/docs/es/get-started/architecture-scenarios/business-to-business/authorization), cuya revisión recomendamos junto con la orientación que se proporciona aquí.

* Los [token de ID](/docs/es/secure/tokens/id-tokens) suelen usarse para transmitir información de autorización del usuario a las aplicaciones mediante claims personalizados, que se pueden agregar con la extensibilidad de [Rules](/docs/es/customize/rules). Los claims agregados pueden permitirle presentar una interfaz de usuario en la que los usuarios no puedan intentar hacer algo para lo que no tienen permiso. La información de autorización en un <Tooltip tip="Token de ID: Credencial destinada al propio cliente, en lugar de para acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=ID+Token">token de ID</Tooltip> también proporciona a cualquier backend de la aplicación una forma de evitar que los usuarios eludan los controles del frontend en las aplicaciones web tradicionales.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  **Práctica recomendada**

  Puede aprovechar [Auth0 Role-Based Access Control](/docs/es/manage-users/access-control/rbac) (RBAC) mediante la funcionalidad [Auth0 Authorization Core](/docs/es/manage-users/access-control/configure-core-rbac) para definir permisos de acceso, que pueden aplicarse automáticamente a los tokens de acceso. Para obtener más información, consulte [Enable Role-Based Access Control for APIs](/docs/es/get-started/apis/enable-role-based-access-control-for-apis).

  La funcionalidad RBAC de Auth0 también puede proporcionar información que puede agregar como claims personalizados a los tokens de identidad (y a los tokens de acceso, si prefiere aplicarlos manualmente). Auth0 Organizations puede aprovechar la funcionalidad RBAC de Auth0 mediante uno o varios roles asignados por membresía. Para obtener más información, consulte [Agregar roles a miembros de la organización](/docs/es/manage-users/organizations/configure-organizations/add-member-roles).
</Callout>

* Las API que ofrecen acceso público a servicios de recursos compartidos suelen estar protegidas mediante mecanismos de control de acceso. Para este fin, Auth0 permite crear un bearer token de autorización, o un <Tooltip tip="OAuth 2.0: Marco de autorización que define protocolos y flujos de trabajo de autorización." cta="Ver glosario" href="/docs/es/glossary?term=OAuth">OAuth</Tooltip> 2 [token de acceso](/docs/es/secure/tokens/access-tokens), que puede transmitir información de autorización del usuario a una API, normalmente mediante [Auth0 Role-Based Access Control (RBAC)](/docs/es/manage-users/access-control/configure-core-rbac) para aplicar uno o varios [roles asignados por membresía](/docs/es/manage-users/organizations/configure-organizations/add-member-roles), o agregando claims personalizados mediante la extensibilidad de [Rules](/docs/es/customize/rules). También puede aprovechar la funcionalidad RBAC de Auth0 para ajustar automáticamente el claim `scope` de un <Tooltip tip="Token de acceso: Credencial de autorización, en forma de una cadena opaca o JWT, usada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=access+token">token de acceso</Tooltip>. Luego, las API pueden usar esta información para aplicar el nivel adecuado de control de acceso, lo que permite que su API aplique reglas de políticas sin tener que hacer una consulta adicional para obtener información sobre el usuario.
* En algunos casos, puede que desee implementar políticas a nivel de aplicación en el tenant de Auth0; esto le permite aplicar políticas a toda una gama de aplicaciones y servicios de recursos (API) sin necesidad de modificar cada uno por separado. Normalmente, esto se implementa mediante la extensibilidad de [Rules](/docs/es/customize/rules).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  **Práctica recomendada**

  Auth0 Organizations proporciona a Auth0 Rules acceso a información centrada en la organización disponible durante la autenticación de un usuario. Esta información está disponible a través del objeto `organization` incluido en el objeto `context` de Rules. El objeto `organization` también proporciona acceso a cualquier metadato aprovisionado en una definición de organización en Auth0. Para obtener más información, consulte [Desarrollo personalizado para Organizations](/docs/es/manage-users/organizations/custom-development).
</Callout>

<div id="id-token-claims">
  ## Claims del token de identidad
</div>

Normalmente, se pueden agregar claims a los tokens de identidad, como se explica en nuestra guía de práctica recomendada sobre [Claims del token de ID](/docs/es/get-started/architecture-scenarios/business-to-consumer/authorization). Al usar la funcionalidad Auth0 Organizations, se agrega automáticamente un claim `org_id` a cualquier token de identidad (para ver un ejemplo, consulta [Trabajar con tokens y Organizations](/docs/es/manage-users/organizations/using-tokens)) emitido para usuarios que pertenecen a una organización. Los SDK de Auth0 validan este parámetro. También puedes agregar información adicional asociada a una organización de Auth0 añadiendo un claim personalizado al token de identidad:

```js wrap lines
context.idToken['http://travel0.net/multifactor'] = context.multifactor;
```

**Nota**: Si has configurado tu tenant para que admita nombres de organización en la Authentication API, el claim `org_name` se incluye automáticamente en los tokens de ID. Para obtener más información, consulta [Use Organization Names in Authentication API](/docs/es/manage-users/organizations/configure-organizations/use-org-name-authentication-api).

<div id="saml-assertion">
  ### Aserción SAML
</div>

Aunque la funcionalidad Auth0 Organization no admite aplicaciones compatibles con <Tooltip tip="Security Assertion Markup Language (SAML): Protocolo estandarizado que permite a dos partes intercambiar información de autenticación sin una contraseña." cta="Ver glosario" href="/docs/es/glossary?term=SAML">SAML</Tooltip>, se puede configurar una aserción SAML generada por un <Tooltip tip="Identity Provider (IdP): Servicio que almacena y gestiona identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=Identity+Provider">proveedor de identidad</Tooltip> (IdP) upstream para rellenar claims estándar o personalizados en un token de identidad que se consume downstream. Por ejemplo, podrías definir la sección de mapeos de una conexión empresarial SAML:

```json lines
{ 
  "user_id": [
    "http://schemas.xmlsoap.org/ws/2005/05/identity/claims/nameidentifier",
    "http://schemas.xmlsoap.org/ws/2005/05/identity/claims/upn",
    "http://schemas.xmlsoap.org/ws/2005/05/identity/claims/name"
  ],
  "email": "http://schemas.xmlsoap.org/ws/2005/05/identity/claims/emailaddress",
  "name": "http://schemas.xmlsoap.org/ws/2005/05/identity/claims/name",
  "given_name": [
    "http://schemas.xmlsoap.org/ws/2005/05/identity/claims/givenname",
    "http://schemas.xmlsoap.org/ws/2005/05/identity/claims/name"
  ], 
  "family_name": "http://schemas.xmlsoap.org/ws/2005/05/identity/claims/surname",
  "groups": "http://schemas.xmlsoap.org/claims/Group"
}
```

En este ejemplo, a cada campo se le asignará un valor en el objeto `user` dentro de una Rule, que se mapeará a claims estándar en un token de identidad o podrá mapearse mediante custom claims. Para obtener más información sobre cómo personalizar los mapeos de SAML, consulta [Conecta tu aplicación con proveedores de identidad SAML: Configura los mapeos](/docs/es/authenticate/identity-providers/enterprise-identity-providers/saml#set-up-mappings).

<div id="access-token-claims">
  ## Claims del token de acceso
</div>

Además de cualquier otro `claim` que estés agregando a tu token de acceso para tomar decisiones de control de acceso (consulta nuestra guía general de [Claims del token de acceso](/docs/es/get-started/architecture-scenarios/business-to-consumer/authorization) como práctica recomendada), normalmente querrás indicar la organización a la que pertenece el usuario.

Como ocurre con el token de identidad, al usar la funcionalidad Auth0 Organizations, el `claim` `org_id` se agrega automáticamente a cualquier token de acceso (para ver un ejemplo, consulta [Trabajar con tokens y Organizations](/docs/es/manage-users/organizations/using-tokens)) emitido para usuarios que pertenecen a una organización. También puedes agregar información complementaria asociada a una organización de Auth0 añadiendo un claim personalizado al token de acceso:

```js wrap lines
context.accessToken['http://travel0.net/multifactor'] = context.multifactor;
```

**Nota**: Si ha configurado su tenant para admitir nombres de organización en la Authentication API, el claim `org_name` se incluye automáticamente en los tokens de acceso. Para obtener más información, consulte [Use Organization Names in Authentication API](/docs/es/manage-users/organizations/configure-organizations/use-org-name-authentication-api).

Como alternativa, podría crear una <Tooltip tip="Audience: Identificador único de la audiencia de un token emitido. Llamado aud en un token, su valor contiene el ID de una aplicación (ID de cliente) para un token de ID o de una API (identificador de API) para un token de acceso." cta="Ver glosario" href="/docs/es/glossary?term=audience">audiencia</Tooltip> de API única para cada organización, lo que daría lugar a una definición de API única en Auth0. Aunque este mecanismo puede reducir la necesidad de usar extensibilidad personalizada con Rule, la complejidad que introduce puede ser difícil de gestionar. A continuación se muestra una comparación sencilla:

| Enfoque                | Ventajas                                                                                                                                                                                                                                                                       | Desventajas                                                                                                                                                                                                                                                                                     |
| ---------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Audiencia de API única | <ul><li>Compatibilidad inmediata con acceso de máquina a máquina para una sola organización.</li><li>La audiencia es un claim estándar en un token de acceso.</li><li>El procesamiento del token de actualización no requiere lógica adicional para la organización.</li></ul> | <ul><li>Se debe automatizar la creación de una API para cada organización.</li><li>Puede ser necesario crear roles independientes si se usa RBAC.</li>Se debe automatizar el aprovisionamiento de Roles a Membership.<li>La implementación de la API debe procesar varias audiencias.</li></ul> |
| Claim personalizado    | Simplifica la configuración del tenant de Auth0.                                                                                                                                                                                                                               | Se necesita código personalizado en una rule para agregar la organización al token de acceso.                                                                                                                                                                                                   |

<div id="roles">
  ### Roles
</div>

La funcionalidad Auth0 Organizations también admite [control de acceso basado en roles (RBAC)](/docs/es/manage-users/access-control/rbac) mediante la funcionalidad [Authorization Core](/docs/es/manage-users/access-control/configure-core-rbac) asociada a un tenant de Auth0. El RBAC [se aplica a nivel de la membresía de la organización de Auth0](/docs/es/manage-users/organizations/configure-organizations/add-member-roles).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Puede [habilitar Auth0 Authorization Core RBAC para una API](/docs/es/get-started/apis/enable-role-based-access-control-for-apis), lo que hará que el claim `scope` predeterminado de los tokens de acceso se modifique automáticamente y que se agregue de forma predeterminada un claim `permission` (para ver un ejemplo, consulte [Trabajar con tokens y Organizations](/docs/es/manage-users/organizations/using-tokens)). También puede agregar información de roles a los tokens de identidad como claims personalizadas accediendo al objeto `authorization` disponible en el objeto `context` de Rules. Para obtener más información, consulte [Casos prácticos de Rules con Authorization: agregar roles de usuario a los tokens](/docs/es/manage-users/access-control/sample-use-cases-rules-with-authorization).
</Callout>

<div id="access-control">
  ## Control de acceso
</div>

La aplicación de políticas a nivel de recurso es responsabilidad de las aplicaciones y/o las API de su sistema. Si intenta aplicar políticas en un <Tooltip tip="Servidor de autorización: Servidor centralizado que contribuye a definir los límites del acceso de un usuario. Por ejemplo, su servidor de autorización puede controlar los datos, las tareas y las funcionalidades disponibles para un usuario." cta="Ver glosario" href="/docs/es/glossary?term=Authorization+Server">servidor de autorización</Tooltip> centralizado, como su tenant de Auth0, pronto se encontrará con un sistema de control complejo que resulta difícil de mantener y comprender. En su lugar, su servidor de autorización centralizado puede garantizar que se incluya en los tokens la información adecuada sobre un usuario, para que sus aplicaciones y API tengan la información necesaria para tomar decisiones sobre la aplicación de políticas. Como mínimo, en situaciones en las que haya demasiada información para incluirla en un solo token (por ejemplo, permisos a nivel de recurso) o en las que la información cambie con la suficiente frecuencia como para quedar desactualizada si se accede a ella directamente en el token, sus aplicaciones y API deberían poder consultar la información correcta.

Por otro lado, cierta aplicación de políticas de alto nivel puede gestionarse de forma centralizada. Por ejemplo, al usar un contexto de tenant de Auth0, hay algunas situaciones en las que podría implementar una Rule que reduzca la necesidad de que cada aplicación y/o API aplique las mismas restricciones. Estas incluyen:

* bloquear el acceso a usuarios desde una dirección IP concreta
* implementar requisitos específicos en torno a Contextual o <Tooltip tip="MFA adaptativo: Autenticación multifactor (MFA) que solo se activa para los usuarios cuando se determina que un intento de inicio de sesión es de baja confianza." cta="Ver glosario" href="/docs/es/glossary?term=Adaptive+MFA">MFA adaptativo</Tooltip>
* restringir el inicio de sesión únicamente a los usuarios que hayan verificado su dirección de correo electrónico
* restringir el acceso a una audiencia de API determinada, para que un usuario no pueda obtener un token de acceso para ninguna otra audiencia de API o no pueda obtener un token de acceso para esa audiencia en determinadas circunstancias. En este caso, si está creando una audiencia de API personalizada para cada organización, su Rule también debe garantizar que el usuario que se autentica pertenezca a la organización que coincida con la audiencia de API correspondiente.

<Warning>
  El acceso a Auth0 Management API no está restringido por organización; el acceso a Auth0 Management API se realiza mediante un token de acceso asignado para su uso en un contexto de máquina a máquina y no puede estar vinculado a una organización de Auth0. Por lo tanto, no proporcione a sus clientes acceso directo a su instancia de Auth0 Management API. Si sus clientes deben gestionar aspectos de su organización, como las cuentas de usuario (consulte [Gestión de perfiles](/docs/es/get-started/architecture-scenarios/multiple-organization-architecture/single-identity-provider-organizations/profile-management)), entonces debería crear su propia aplicación y/o API independiente para este propósito.
</Warning>
