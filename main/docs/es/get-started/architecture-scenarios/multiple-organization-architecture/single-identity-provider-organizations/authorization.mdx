---

Al pensar en la autorización, generalmente es necesario considerar cómo se determina lo que una persona puede hacer y cómo se comunica esta información a las aplicaciones y/o APIs. Según las aplicaciones que tenga, puede verse afectado por uno o ambos aspectos. En nuestros escenarios de arquitectura, ofrecemos orientación de propósito general sobre [Autorización B2B](/docs/es/get-started/architecture-scenarios/business-to-business/authorization), que recomendamos revisar junto con la orientación proporcionada aquí.

* Los [tokens de ID](/docs/es/secure/tokens/id-tokens) se utilizan frecuentemente para transmitir información de autorización de usuarios a las aplicaciones mediante claims personalizados, que pueden añadirse usando la extensibilidad de [Rules](/docs/es/customize/rules). Los claims añadidos permiten presentar una interfaz de usuario en la que los usuarios no tienen la posibilidad de intentar algo que no tienen permitido hacer. La información de autorización en un <Tooltip tip="ID Token: Credencial destinada al propio cliente, en lugar de para acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=ID+Token">token de ID</Tooltip> también proporciona al backend de cualquier aplicación una forma de impedir que los usuarios eludan los controles del frontend en aplicaciones web tradicionales.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  **Práctica recomendada**

  Puede aprovechar el [Control de Acceso Basado en Roles de Auth0](/docs/es/manage-users/access-control/rbac) (RBAC) a través de la funcionalidad [Authorization Core de Auth0](/docs/es/manage-users/access-control/configure-core-rbac) para definir permisos de acceso, que pueden aplicarse automáticamente a los tokens de acceso. Para más detalles, consulte [Habilitar el Control de Acceso Basado en Roles para APIs](/docs/es/get-started/apis/enable-role-based-access-control-for-apis).

  La funcionalidad RBAC de Auth0 también puede proporcionar información que puede añadir como claims personalizados a los tokens de identidad (y a los tokens de acceso, si prefiere aplicarlos manualmente). Auth0 Organizations puede aprovechar la capacidad RBAC de Auth0 mediante uno o más roles asignados por membresía. Para obtener más información, consulte [Añadir roles a los miembros de una organización](/docs/es/manage-users/organizations/configure-organizations/add-member-roles).
</Callout>

* Las APIs que ofrecen acceso público a servicios de recursos compartidos generalmente están protegidas mediante mecanismos de control de acceso. Para ello, Auth0 ofrece la posibilidad de crear un bearer token de autorización, u <Tooltip tip="OAuth 2.0: Marco de autorización que define protocolos y flujos de trabajo de autorización." cta="Ver glosario" href="/docs/es/glossary?term=OAuth">OAuth</Tooltip> 2 [token de acceso](/docs/es/secure/tokens/access-tokens), que puede transmitir información de autorización de usuarios a una API, ya sea mediante el [Control de Acceso Basado en Roles (RBAC)](/docs/es/manage-users/access-control/configure-core-rbac) de Auth0 para aplicar uno o más [roles asignados por membresía](/docs/es/manage-users/organizations/configure-organizations/add-member-roles), o añadiendo claims personalizados a través de la extensibilidad de [Rules](/docs/es/customize/rules). También puede aprovechar la capacidad RBAC de Auth0 para ajustar automáticamente el claim `scope` de un <Tooltip tip="Access Token: Credencial de autorización, en forma de cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=access+token">token de acceso</Tooltip>. Las APIs pueden entonces usar esta información para aplicar el nivel adecuado de control de acceso, lo que permite que su API aplique reglas de política sin necesidad de realizar una consulta adicional para obtener información sobre el usuario.
* En ciertos casos, puede que desee implementar políticas a nivel de aplicación en el tenant de Auth0; esto le permite aplicar políticas a toda una gama de aplicaciones y servicios de recursos (APIs) sin necesidad de modificar cada uno de forma independiente. Por lo general, esto se implementa a través de la extensibilidad de [Rules](/docs/es/customize/rules).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  **Práctica recomendada**

  Auth0 Organizations proporciona a Auth0 Rules acceso a información centrada en la organización, disponible al autenticar a un usuario. Esta información está disponible a través del objeto `organization` contenido dentro del objeto `context` de Rules. El objeto `organization` también proporciona acceso a cualquier metadato aprovisionado en una definición de organización en Auth0. Para obtener más información, consulte [Desarrollo personalizado para Organizations](/docs/es/manage-users/organizations/custom-development).
</Callout>

<div id="id-token-claims">
  ## Claims del token de identidad
</div>

Normalmente, se pueden agregar claims a los tokens de identidad, como se explica en nuestra guía de práctica recomendada sobre [Claims del token de ID](/docs/es/get-started/architecture-scenarios/business-to-consumer/authorization). Al usar la funcionalidad Auth0 Organizations, se agrega automáticamente un claim `org_id` a cualquier token de identidad (para ver un ejemplo, consulta [Work with Tokens and Organizations](/docs/es/manage-users/organizations/using-tokens)) emitido para usuarios con membresía en una organización. Este parámetro es validado por los Auth0 SDKs. También puedes agregar información complementaria asociada a una organización de Auth0 añadiendo un claim personalizado al token de identidad:

```js wrap lines
context.idToken['http://travel0.net/multifactor'] = context.multifactor;
```

**Nota**: Si has configurado tu tenant para admitir el uso de nombres de organización en la Authentication API, el claim `org_name` se incluye automáticamente en los tokens de ID. Para obtener más información, consulta [Use Organization Names in Authentication API](/docs/es/manage-users/organizations/configure-organizations/use-org-name-authentication-api).

<div id="saml-assertion">
  ### Aserción SAML
</div>

Aunque la funcionalidad Auth0 Organization no admite aplicaciones compatibles con <Tooltip tip="Security Assertion Markup Language (SAML): Protocolo estandarizado que permite a dos partes intercambiar información de autenticación sin una contraseña." cta="Ver glosario" href="/docs/es/glossary?term=SAML">SAML</Tooltip>, una aserción SAML generada por un <Tooltip tip="Identity Provider (IdP): Servicio que almacena y administra identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=Identity+Provider">proveedor de identidad</Tooltip> (IdP) de origen puede configurarse para rellenar claims estándar o custom claims en un token de identidad utilizado más adelante. Por ejemplo, podrías definir la sección de mappings de una conexión empresarial SAML:

```json lines
{ 
  "user_id": [
    "http://schemas.xmlsoap.org/ws/2005/05/identity/claims/nameidentifier",
    "http://schemas.xmlsoap.org/ws/2005/05/identity/claims/upn",
    "http://schemas.xmlsoap.org/ws/2005/05/identity/claims/name"
  ],
  "email": "http://schemas.xmlsoap.org/ws/2005/05/identity/claims/emailaddress",
  "name": "http://schemas.xmlsoap.org/ws/2005/05/identity/claims/name",
  "given_name": [
    "http://schemas.xmlsoap.org/ws/2005/05/identity/claims/givenname",
    "http://schemas.xmlsoap.org/ws/2005/05/identity/claims/name"
  ], 
  "family_name": "http://schemas.xmlsoap.org/ws/2005/05/identity/claims/surname",
  "groups": "http://schemas.xmlsoap.org/claims/Group"
}
```

En este ejemplo, a cada campo se le asignará un valor en el objeto `user` dentro de una Rule, que se corresponderá con claims estándar de un token de identidad o podrá mapearse mediante claims personalizados. Para obtener más información sobre cómo personalizar los mapeos de SAML, consulta [Conecta tu aplicación con proveedores de identidad SAML: Configura los mapeos](/docs/es/authenticate/identity-providers/enterprise-identity-providers/saml#set-up-mappings).

<div id="access-token-claims">
  ## Claims del token de acceso
</div>

Además de cualquier otro claim que estés agregando a tu token de acceso para tomar decisiones de control de acceso (consulta nuestra guía general de práctica recomendada sobre [Claims del token de acceso](/docs/es/get-started/architecture-scenarios/business-to-consumer/authorization)), normalmente querrás comunicar la organización a la que pertenece el usuario.

Al igual que con el token de identidad, al usar la funcionalidad Auth0 Organizations, el claim `org_id` se agrega automáticamente a cualquier token de acceso (para ver un ejemplo, consulta [Trabajar con tokens y Organizations](/docs/es/manage-users/organizations/using-tokens)) emitido para usuarios que pertenecen a una organización. También puedes agregar información adicional asociada a una organización de Auth0 añadiendo un claim personalizado al token de acceso:

```js wrap lines
context.accessToken['http://travel0.net/multifactor'] = context.multifactor;
```

**Nota**: Si ha configurado su tenant para admitir nombres de organización en la Authentication API, el claim `org_name` se incluye automáticamente en los tokens de acceso. Para obtener más información, consulte [Use Organization Names in Authentication API](/docs/es/manage-users/organizations/configure-organizations/use-org-name-authentication-api).

Como alternativa, podría crear una <Tooltip tip="Audience: Identificador único de la audiencia de un token emitido. Se denomina aud en un token y su valor contiene el ID de una aplicación (ID de cliente) para un token de ID o de una API (identificador de API) para un token de acceso." cta="Ver glosario" href="/docs/es/glossary?term=audience">audiencia</Tooltip> de API única para cada organización, lo que daría lugar a una definición de API única en Auth0. Aunque este mecanismo puede reducir la necesidad de usar extensibilidad personalizada con Rules, la complejidad que introduce puede ser difícil de gestionar. A continuación se muestra una comparación sencilla:

| Enfoque                | Ventajas                                                                                                                                                                                                                                                                        | Desventajas                                                                                                                                                                                                                                                                                     |
| ---------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Audiencia de API única | <ul><li>Compatibilidad lista para usar con acceso de máquina a máquina para una sola organización.</li><li>La audiencia es un claim estándar en un token de acceso.</li><li>El procesamiento del token de actualización no requiere lógica adicional de organización.</li></ul> | <ul><li>Se debe automatizar la creación de una API para cada organización.</li><li>Puede ser necesario crear roles independientes si se usa RBAC.</li>Se debe automatizar el aprovisionamiento de Roles a Membership.<li>La implementación de la API debe procesar varias audiencias.</li></ul> |
| Claim personalizado    | Simplifica la configuración del tenant de Auth0.                                                                                                                                                                                                                                | Se necesita código personalizado en una rule para agregar la organización al token de acceso.                                                                                                                                                                                                   |

<div id="roles">
  ### Roles
</div>

La funcionalidad Auth0 Organizations también admite [control de acceso basado en roles (RBAC)](/docs/es/manage-users/access-control/rbac) mediante la funcionalidad [Authorization Core](/docs/es/manage-users/access-control/configure-core-rbac) asociada a un tenant de Auth0. RBAC se [aplica a nivel de la membresía de la organización en Auth0](/docs/es/manage-users/organizations/configure-organizations/add-member-roles).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Puede [habilitar Auth0 Authorization Core RBAC para una API](/docs/es/get-started/apis/enable-role-based-access-control-for-apis), lo que hará que el claim `scope` predeterminado de los tokens de acceso se modifique automáticamente y que se agregue de forma predeterminada un claim `permission` (para ver un ejemplo, consulte [Trabajar con tokens y organizaciones](/docs/es/manage-users/organizations/using-tokens)). También puede agregar información de roles a los tokens de identidad como custom claims accediendo al objeto `authorization` disponible en el objeto `context` de Rules. Para obtener más información, consulte [Casos de uso de ejemplo de Rules con Authorization: agregar roles de usuario a los tokens](/docs/es/manage-users/access-control/sample-use-cases-rules-with-authorization).
</Callout>

<div id="access-control">
  ## Control de acceso
</div>

La aplicación de políticas a nivel de recurso es responsabilidad de las aplicaciones y/o las API de tu sistema. Si intentas aplicar políticas en un <Tooltip tip="Servidor de autorización: Servidor centralizado que contribuye a definir los límites del acceso de un usuario. Por ejemplo, tu servidor de autorización puede controlar los datos, las tareas y las funcionalidades disponibles para un usuario." cta="Ver glosario" href="/docs/es/glossary?term=Authorization+Server">servidor de autorización</Tooltip> centralizado, como tu tenant de Auth0, pronto te encontrarás con un sistema de control complejo, difícil de mantener y de entender. En su lugar, tu servidor de autorización centralizado puede garantizar que los tokens incluyan la información adecuada sobre un usuario, para que tus aplicaciones y API dispongan de la información necesaria para tomar decisiones de aplicación de políticas. Como mínimo, en situaciones en las que hay demasiada información como para incluirla en un solo token (por ejemplo, permisos a nivel de recurso) o en las que la información cambia con la suficiente frecuencia como para quedar desactualizada si se consulta directamente en el token, tus aplicaciones y API deberían poder buscar la información correcta.

Por otro lado, cierta aplicación de políticas de alto nivel puede gestionarse de forma centralizada. Por ejemplo, al usar el contexto de un tenant de Auth0, hay algunas situaciones en las que podrías implementar una Rule que reduzca la necesidad de que cada aplicación y/o API aplique las mismas restricciones. Estas incluyen:

* bloquear el acceso a usuarios desde una dirección IP concreta
* implementar requisitos específicos relacionados con el MFA contextual o el <Tooltip tip="MFA adaptativo: Autenticación multifactor (MFA) que solo se activa para los usuarios cuando se determina que un intento de inicio de sesión es de baja confianza." cta="Ver glosario" href="/docs/es/glossary?term=Adaptive+MFA">MFA adaptativo</Tooltip>
* restringir el inicio de sesión solo a usuarios que hayan verificado su dirección de correo electrónico
* restringir el acceso a una audiencia de API determinada, para que un usuario no pueda obtener un token de acceso para ninguna otra audiencia de API o no pueda obtener un token de acceso para esa audiencia en determinadas circunstancias. En este caso, si estás creando una audiencia de API personalizada para cada organización, tu Rule también debe garantizar que el usuario que se autentica pertenezca a la organización que coincida con la audiencia de API correspondiente.

<Warning>
  El acceso a Auth0 Management API no está restringido por organización; el acceso a Auth0 Management API se realiza mediante un token de acceso asignado para su uso en un contexto de máquina a máquina y no puede estar vinculado a una Auth0 Organization. Por lo tanto, no proporciones a tus clientes acceso directo a tu instancia de Auth0 Management API. Si tus clientes deben gestionar aspectos de su organización, como las cuentas de usuario (consulta [Gestión de perfiles](/docs/es/get-started/architecture-scenarios/multiple-organization-architecture/single-identity-provider-organizations/profile-management)), entonces deberías crear tu propia aplicación y/o API independiente para este fin.
</Warning>
