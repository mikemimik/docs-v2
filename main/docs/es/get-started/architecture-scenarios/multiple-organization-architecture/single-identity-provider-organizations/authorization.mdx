---

Al pensar en la autorización, normalmente debes considerar cómo determinas qué puede hacer una persona y cómo comunicas esta información a tus aplicaciones y/o API. Según las aplicaciones que tengas, puede que necesites una de estas dos cosas o ambas. En nuestros escenarios de arquitectura, ofrecemos orientación general sobre la [autorización B2B](/docs/es/get-started/architecture-scenarios/business-to-business/authorization), cuya revisión recomendamos junto con la orientación que se proporciona aquí.

* Los [tokens de identidad](/docs/es/secure/tokens/id-tokens) suelen usarse para transmitir a las aplicaciones información de autorización del usuario mediante claims personalizados, que pueden añadirse usando la extensibilidad de [Rules](/docs/es/customize/rules). Las claims añadidas pueden permitirte presentar una interfaz de usuario en la que los usuarios no puedan ni siquiera intentar hacer algo para lo que no tienen permiso. La información de autorización en un <Tooltip tip="token de identidad: Credencial destinada al propio cliente, en lugar de al acceso a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=ID+Token">token de identidad</Tooltip> también ofrece a cualquier backend de la aplicación una forma de evitar que los usuarios eludan los controles del frontend en las aplicaciones web tradicionales.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  **Práctica recomendada**

  Puedes aprovechar [Auth0 control de acceso basado en roles](/docs/es/manage-users/access-control/rbac) (RBAC) mediante la funcionalidad [Auth0 Authorization Core](/docs/es/manage-users/access-control/configure-core-rbac) para definir permisos de acceso, que pueden aplicarse automáticamente a los Access Tokens. Para obtener más información, consulta [Habilitar control de acceso basado en roles para las API](/docs/es/get-started/apis/enable-role-based-access-control-for-apis).

  La funcionalidad RBAC de Auth0 también puede proporcionar información que puedes agregar como claims personalizados a los tokens de identidad (y a los tokens de acceso, si prefieres hacerlo manualmente). Auth0 Organizations puede aprovechar la capacidad RBAC de Auth0 mediante uno o más roles asignados por membresía. Para obtener más información, consulta [Agregar roles a los miembros de la organización](/docs/es/manage-users/organizations/configure-organizations/add-member-roles).
</Callout>

* Las API que ofrecen acceso público a servicios de recursos compartidos suelen estar protegidas mediante mecanismos de control de acceso. Para ello, Auth0 permite crear un bearer token de autorización, o [token de acceso](/docs/es/secure/tokens/access-tokens) de <Tooltip tip="OAuth 2.0: Marco de autorización que define protocolos y flujos de trabajo de autorización." cta="Ver glosario" href="/docs/es/glossary?term=OAuth">OAuth</Tooltip> 2, que puede transmitir información de autorización del usuario a una API, normalmente mediante el uso de [control de acceso basado en roles (RBAC)](/docs/es/manage-users/access-control/configure-core-rbac) de Auth0 para aplicar uno o más [roles asignados por membresía](/docs/es/manage-users/organizations/configure-organizations/add-member-roles), o añadiendo claims personalizados mediante la extensibilidad de [Rules](/docs/es/customize/rules). También puedes aprovechar la capacidad RBAC de Auth0 para ajustar automáticamente la claim `scope` de un <Tooltip tip="Access Token: Authorization credential, in the form of an opaque string or JWT, used to access an API." cta="View Glossary" href="/docs/es/glossary?term=access+token">token de acceso</Tooltip>. Después, las API pueden usar esta información para aplicar el nivel adecuado de control de acceso, lo que permite a tu API aplicar reglas de policy sin tener que hacer una búsqueda adicional para obtener información sobre el usuario.
* En ciertos casos, quizá quieras implementar policies a nivel de aplicación en el tenant de Auth0; esto te permite aplicar policies a toda una gama de aplicaciones y servicios de recursos (API) sin necesidad de modificar cada uno de ellos por separado. Normalmente, esto se implementa mediante la extensibilidad de [Rules](/docs/es/customize/rules).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  **Práctica recomendada**

  Auth0 Organizations proporciona a Auth0 Rules acceso a información centrada en la organización, a la que se puede acceder al autenticar a un usuario. Esta información está disponible a través del objeto `organization` contenido dentro del objeto `context` de Rules. El objeto `organization` también proporciona acceso a cualquier metadata aprovisionada en una definición de Organization en Auth0. Para obtener más información, consulta [Desarrollo personalizado para Organizations](/docs/es/manage-users/organizations/custom-development).
</Callout>

<div id="id-token-claims">
  ## Claims del token de identidad
</div>

Por lo general, se pueden agregar claims a los tokens de identidad, como se indica en nuestra guía de práctica recomendada sobre [Claims del token de identidad](/docs/es/get-started/architecture-scenarios/business-to-consumer/authorization). Al usar la funcionalidad Auth0 Organizations, se agrega automáticamente un claim `org_id` a cualquier token de identidad (para ver un ejemplo, consulta [Trabajar con tokens y Organizations](/docs/es/manage-users/organizations/using-tokens)) emitido para usuarios con membresía en una organización. Los Auth0 SDKs validan este parámetro. También puedes agregar información adicional asociada a una organización de Auth0 añadiendo un custom claim al token de identidad:

```js wrap lines
context.idToken['http://travel0.net/multifactor'] = context.multifactor;
```

**Nota**: Si has configurado tu tenant para que admita nombres de organización en la Authentication API, `org_name` se incluye automáticamente como claim en los tokens de identidad. Para obtener más información, consulta [Usar nombres de organización en Authentication API](/docs/es/manage-users/organizations/configure-organizations/use-org-name-authentication-api).

<div id="saml-assertion">
  ### Aserción SAML
</div>

Aunque la función Auth0 Organization no admite aplicaciones compatibles con <Tooltip tip="Security Assertion Markup Language (SAML): Protocolo estandarizado que permite a dos partes intercambiar información de autenticación sin una contraseña." cta="Ver glosario" href="/docs/es/glossary?term=SAML">SAML</Tooltip>, se puede configurar una aserción SAML generada por un <Tooltip tip="Identity Provider (IdP): Servicio que almacena y gestiona identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=Identity+Provider">Proveedor de identidad</Tooltip> (IdP) para completar claims estándar o personalizados en un token de identidad que se consume más adelante. Por ejemplo, podrías definir la sección de mapeo de una conexión empresarial SAML:

```json lines
{ 
  "user_id": [
    "http://schemas.xmlsoap.org/ws/2005/05/identity/claims/nameidentifier",
    "http://schemas.xmlsoap.org/ws/2005/05/identity/claims/upn",
    "http://schemas.xmlsoap.org/ws/2005/05/identity/claims/name"
  ],
  "email": "http://schemas.xmlsoap.org/ws/2005/05/identity/claims/emailaddress",
  "name": "http://schemas.xmlsoap.org/ws/2005/05/identity/claims/name",
  "given_name": [
    "http://schemas.xmlsoap.org/ws/2005/05/identity/claims/givenname",
    "http://schemas.xmlsoap.org/ws/2005/05/identity/claims/name"
  ], 
  "family_name": "http://schemas.xmlsoap.org/ws/2005/05/identity/claims/surname",
  "groups": "http://schemas.xmlsoap.org/claims/Group"
}
```

En este ejemplo, a cada campo se le asignará un valor en el objeto `user` dentro de una Rule, que se mapeará a claims estándar en un token de identidad o podrá mapearse mediante custom claims. Para obtener más información sobre cómo personalizar los mapeos de SAML, consulta [Conecta tu aplicación con proveedores de identidad SAML: Configura los mapeos](/docs/es/authenticate/identity-providers/enterprise-identity-providers/saml#set-up-mappings).

<div id="access-token-claims">
  ## Claims del token de acceso
</div>

Además de cualquier otro `claim` que estés añadiendo a tu token de acceso para tomar decisiones de control de acceso (consulta nuestra guía general de práctica recomendada sobre [Access Token Claims](/docs/es/get-started/architecture-scenarios/business-to-consumer/authorization)), normalmente querrás indicar la organización a la que pertenece el usuario.

Al igual que con el token de identidad, al usar la funcionalidad Auth0 Organizations, el `claim` `org_id` se añade automáticamente a cualquier token de acceso (para ver un ejemplo, consulta [Work with Tokens and Organizations](/docs/es/manage-users/organizations/using-tokens)) emitido para usuarios con membresía en una organización. También puedes añadir información complementaria asociada a una organización de Auth0 agregando un claim personalizado al token de acceso:

```js wrap lines
context.accessToken['http://travel0.net/multifactor'] = context.multifactor;
```

**Nota**: Si has configurado tu tenant para admitir nombres de organización en la Authentication API, el claim `org_name` se incluye automáticamente en los tokens de acceso. Para obtener más información, consulta [Use Organization Names in Authentication API](/docs/es/manage-users/organizations/configure-organizations/use-org-name-authentication-api).

Como alternativa, podrías crear una <Tooltip tip="Audiencia: identificador único de la audiencia de un token emitido. Se llama aud en un token; su valor contiene el ID de una aplicación (ID de cliente) para un token de ID o de una API (identificador de API) para un token de acceso." cta="Ver glosario" href="/docs/es/glossary?term=audience">audiencia</Tooltip> de API única para cada organización, lo que daría lugar a una definición de API única en Auth0. Aunque este mecanismo puede reducir la necesidad de usar extensibilidad personalizada con Rule, la complejidad que introduce puede resultar difícil de gestionar. A continuación, se muestra una comparación sencilla:

| Enfoque                | Ventajas                                                                                                                                                                                                                                                                  | Desventajas                                                                                                                                                                                                                                                                                        |
| ---------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Audiencia de API única | <ul><li>Compatibilidad inmediata con acceso de máquina a máquina para una sola organización.</li><li>La audiencia es un claim estándar en un token de acceso.</li><li>El procesamiento del token de actualización no requiere lógica adicional de organización.</li></ul> | <ul><li>Se debe automatizar la creación de una API para cada organización.</li><li>Puede ser necesario crear roles independientes si se usa RBAC.</li>Se debe automatizar el aprovisionamiento de Roles a Membership.<li>La implementación de la API debe procesar múltiples audiencias.</li></ul> |
| Claim personalizado    | Simplifica la configuración del tenant de Auth0.                                                                                                                                                                                                                          | Se necesita código personalizado en una rule para agregar la organización al token de acceso.                                                                                                                                                                                                      |

<div id="roles">
  ### Roles
</div>

La funcionalidad Auth0 Organizations también admite [Control de acceso basado en roles (RBAC)](/docs/es/manage-users/access-control/rbac) mediante la funcionalidad [Authorization Core](/docs/es/manage-users/access-control/configure-core-rbac) asociada a un tenant de Auth0. RBAC se [aplica a nivel de membresía de la organización de Auth0](/docs/es/manage-users/organizations/configure-organizations/add-member-roles).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Puede [habilitar Authorization Core RBAC de Auth0 para una API](/docs/es/get-started/apis/enable-role-based-access-control-for-apis), lo que hará que el claim `scope` predeterminado de los tokens de acceso se modifique automáticamente y que se agregue de forma predeterminada un claim `permission` (para ver un ejemplo, consulte [Trabajar con tokens y Organizations](/docs/es/manage-users/organizations/using-tokens)). También puede agregar información de roles a los tokens de identidad como claims personalizados accediendo al objeto `authorization` disponible en el objeto `context` de Rules. Para obtener más información, consulte [Casos de uso de ejemplo de Rules con Authorization: agregar roles de usuario a los tokens](/docs/es/manage-users/access-control/sample-use-cases-rules-with-authorization).
</Callout>

<div id="access-control">
  ## Control de acceso
</div>

La aplicación de políticas a nivel de recurso es responsabilidad de las aplicaciones y/o API de tu sistema. Si intentas aplicar políticas en un <Tooltip tip="Servidor de autorización: Servidor centralizado que contribuye a definir los límites del acceso de un usuario. Por ejemplo, tu servidor de autorización puede controlar los datos, las tareas y las funciones disponibles para un usuario." cta="Ver glosario" href="/docs/es/glossary?term=Authorization+Server">servidor de autorización</Tooltip> centralizado, como tu tenant de Auth0, rápidamente te encontrarás con un sistema de control complejo que será difícil de mantener y comprender. En cambio, tu servidor de autorización centralizado puede garantizar que se incluya en los tokens la información adecuada sobre un usuario, de modo que tus aplicaciones y API dispongan de la información necesaria para tomar decisiones sobre la aplicación de políticas. Como mínimo, en situaciones en las que haya demasiada información para incluirla en un solo token (por ejemplo, permisos a nivel de recurso) o en las que la información cambie con suficiente frecuencia como para quedar desactualizada si se accede a ella directamente en el token, tus aplicaciones y API deberían poder consultar la información correcta.

Por otro lado, cierta aplicación de políticas de alto nivel puede gestionarse de forma centralizada. Por ejemplo, al usar el contexto de un tenant de Auth0, hay algunas situaciones en las que podrías implementar una Rule para reducir la necesidad de que cada aplicación y/o API aplique las mismas restricciones. Entre ellas se incluyen:

* bloquear el acceso a usuarios desde una dirección IP concreta
* implementar requisitos específicos relacionados con Contextual o con <Tooltip tip="MFA adaptativo: Autenticación multifactor (MFA) que solo se activa para los usuarios cuando se determina que un intento de inicio de sesión es de baja confianza." cta="Ver glosario" href="/docs/es/glossary?term=Adaptive+MFA">MFA adaptativo</Tooltip>
* restringir el inicio de sesión solo a usuarios que hayan verificado su dirección de correo electrónico
* restringir el acceso a una audiencia de API determinada, para que un usuario no pueda obtener un token de acceso para ninguna otra audiencia de API o no pueda obtener un token de acceso para esa audiencia en determinadas circunstancias. En este caso, si estás creando una audiencia de API personalizada para cada organización, tu Rule también debe garantizar que el usuario que se autentica pertenezca a la organización que coincida con la audiencia de API correspondiente.

<Warning>
  El acceso a la Auth0 Management API no está restringido por organización; el acceso a la Auth0 Management API se realiza mediante un token de acceso asignado para su uso en un contexto de máquina a máquina y no puede estar limitado por una organización de Auth0. Por lo tanto, no proporciones a tus clientes acceso directo a tu instancia de la Auth0 Management API. Si tus clientes deben gestionar aspectos de su organización, como las cuentas de usuario (consulta [Gestión de perfiles](/docs/es/get-started/architecture-scenarios/multiple-organization-architecture/single-identity-provider-organizations/profile-management)), entonces debes crear tu propia aplicación y/o API independiente para este fin.
</Warning>
