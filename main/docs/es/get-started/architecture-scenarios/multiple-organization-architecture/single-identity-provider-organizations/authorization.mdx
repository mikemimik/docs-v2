---

Al pensar en la autorización, normalmente debe considerar cómo determina qué puede hacer una persona y cómo comunica esta información a sus aplicaciones y/o API. Según las aplicaciones que tenga, puede verse afectado por uno o ambos aspectos. En nuestros escenarios de arquitectura, ofrecemos orientación general sobre la [autorización B2B](/docs/es/get-started/architecture-scenarios/business-to-business/authorization), que recomendamos revisar junto con la orientación que se proporciona aquí.

* Los [tokens de ID](/docs/es/secure/tokens/id-tokens) suelen usarse para transmitir información de autorización del usuario a las aplicaciones mediante claims personalizadas, que pueden agregarse usando la extensibilidad de [Rules](/docs/es/customize/rules). Las claims agregadas pueden permitirle presentar una interfaz de usuario en la que los usuarios no tengan la posibilidad de intentar hacer algo para lo que no tienen permiso. La información de autorización en un <Tooltip tip="Token de ID: credencial destinada al propio cliente, en lugar de para acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=ID+Token">token de ID</Tooltip> también proporciona al backend de cualquier aplicación una forma de impedir que los usuarios eludan los controles del frontend en aplicaciones web tradicionales.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  **Práctica recomendada**

  Puede aprovechar [Auth0 Role-Based Access Control](/docs/es/manage-users/access-control/rbac) (RBAC) mediante la funcionalidad [Auth0 Authorization Core](/docs/es/manage-users/access-control/configure-core-rbac) para definir permisos de acceso, que pueden aplicarse automáticamente a los tokens de acceso. Para obtener más información, consulte [Habilitar el control de acceso basado en roles para las API](/docs/es/get-started/apis/enable-role-based-access-control-for-apis).

  La funcionalidad RBAC de Auth0 también puede proporcionar información que puede agregar como claims personalizadas a los tokens de identidad (y a los tokens de acceso, si prefiere aplicarla manualmente). Auth0 Organizations puede aprovechar la capacidad de RBAC de Auth0 mediante uno o varios roles asignados por membresía. Para obtener más información, consulte [Agregar roles a miembros de la organización](/docs/es/manage-users/organizations/configure-organizations/add-member-roles).
</Callout>

* Las API que ofrecen acceso público a servicios de recursos compartidos suelen estar protegidas mediante mecanismos de control de acceso. Para este propósito, Auth0 ofrece la capacidad de crear un bearer token de autorización, es decir, un [token de acceso](/docs/es/secure/tokens/access-tokens) de <Tooltip tip="OAuth 2.0: framework de autorización que define protocolos y flujos de trabajo de autorización." cta="Ver glosario" href="/docs/es/glossary?term=OAuth">OAuth</Tooltip> 2, que puede transmitir información de autorización del usuario a una API, normalmente mediante [Role-Based Access Control (RBAC)](/docs/es/manage-users/access-control/configure-core-rbac) de Auth0 para aplicar uno o más [roles asignados por membresía](/docs/es/manage-users/organizations/configure-organizations/add-member-roles), o agregando claims personalizadas mediante la extensibilidad de [Rules](/docs/es/customize/rules). También puede aprovechar la capacidad de RBAC de Auth0 para ajustar automáticamente la claim `scope` de un <Tooltip tip="Token de acceso: credencial de autorización, en forma de cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=access+token">token de acceso</Tooltip>. Luego, las API pueden usar esta información para aplicar el nivel adecuado de control de acceso, lo que permite que su API aplique reglas de política sin tener que hacer una consulta adicional para obtener información sobre el usuario.
* En ciertos casos, puede que desee implementar políticas a nivel de aplicación en el tenant de Auth0; esto le permite aplicar políticas a toda una gama de aplicaciones y servicios de recursos (API) sin necesidad de modificar cada uno por separado. Normalmente, esto se implementa mediante la extensibilidad de [Rules](/docs/es/customize/rules).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  **Práctica recomendada**

  Auth0 Organizations proporciona a Auth0 Rules acceso a información centrada en la organización que puede consultarse al autenticar a un usuario. Esta información está disponible mediante el objeto `organization` contenido dentro del objeto `context` de Rules. El objeto `organization` también proporciona acceso a cualquier metadata aprovisionada en una definición de Organization en Auth0. Para obtener más información, consulte [Desarrollo personalizado para Organizations](/docs/es/manage-users/organizations/custom-development).
</Callout>

<div id="id-token-claims">
  ## Claims del token de identidad
</div>

Por lo general, se pueden agregar claims a los tokens de identidad, como se explica en nuestra guía de práctica recomendada sobre [Claims del token de ID](/docs/es/get-started/architecture-scenarios/business-to-consumer/authorization). Al usar la funcionalidad Auth0 Organizations, se agrega automáticamente un claim `org_id` a cualquier token de identidad (para ver un ejemplo, consulta [Trabajar con tokens y Organizations](/docs/es/manage-users/organizations/using-tokens)) emitido para usuarios que pertenecen a una organización. Este parámetro lo validan los SDKs de Auth0. También puedes agregar información complementaria asociada a una organización de Auth0 añadiendo un claim personalizado al token de identidad:

```js wrap lines
context.idToken['http://travel0.net/multifactor'] = context.multifactor;
```

**Nota**: Si has configurado tu tenant para admitir el uso de nombres de organización en la Authentication API, el claim `org_name` se incluye automáticamente en los tokens de ID. Para obtener más información, consulta [Usar nombres de organización en la Authentication API](/docs/es/manage-users/organizations/configure-organizations/use-org-name-authentication-api).

<div id="saml-assertion">
  ### Aserción SAML
</div>

Aunque la funcionalidad de organización de Auth0 no admite aplicaciones compatibles con <Tooltip tip="Security Assertion Markup Language (SAML): Protocolo estandarizado que permite a dos partes intercambiar información de autenticación sin contraseña." cta="Ver glosario" href="/docs/es/glossary?term=SAML">SAML</Tooltip>, se puede configurar una aserción SAML generada por un <Tooltip tip="Identity Provider (IdP): Servicio que almacena y gestiona identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=Identity+Provider">proveedor de identidad</Tooltip> (IdP) de origen para rellenar claims estándar o personalizadas en un token de identidad que se consume más adelante. Por ejemplo, podría definir la sección de mapeos de una conexión empresarial SAML:

```json lines
{ 
  "user_id": [
    "http://schemas.xmlsoap.org/ws/2005/05/identity/claims/nameidentifier",
    "http://schemas.xmlsoap.org/ws/2005/05/identity/claims/upn",
    "http://schemas.xmlsoap.org/ws/2005/05/identity/claims/name"
  ],
  "email": "http://schemas.xmlsoap.org/ws/2005/05/identity/claims/emailaddress",
  "name": "http://schemas.xmlsoap.org/ws/2005/05/identity/claims/name",
  "given_name": [
    "http://schemas.xmlsoap.org/ws/2005/05/identity/claims/givenname",
    "http://schemas.xmlsoap.org/ws/2005/05/identity/claims/name"
  ], 
  "family_name": "http://schemas.xmlsoap.org/ws/2005/05/identity/claims/surname",
  "groups": "http://schemas.xmlsoap.org/claims/Group"
}
```

En este ejemplo, a cada campo se le asignará un valor en el objeto `user` dentro de una Rule, que luego se mapeará a claims estándar de un token de identidad o podrá mapearse mediante custom claims. Para obtener más información sobre cómo personalizar los mapeos de SAML, consulta [Conecta tu aplicación a proveedores de identidad SAML: Configura los mapeos](/docs/es/authenticate/identity-providers/enterprise-identity-providers/saml#set-up-mappings).

<div id="access-token-claims">
  ## Claims del token de acceso
</div>

Además de cualquier otro claim que añadas a tu token de acceso para tomar decisiones de control de acceso (consulta nuestra guía general de prácticas recomendadas sobre [Claims del token de acceso](/docs/es/get-started/architecture-scenarios/business-to-consumer/authorization)), normalmente querrás comunicar la organización a la que pertenece el usuario.

Al igual que con el token de identidad, al usar la funcionalidad Auth0 Organizations, el claim `org_id` se agrega automáticamente a cualquier token de acceso (para ver un ejemplo, consulta [Trabajar con tokens y organizaciones](/docs/es/manage-users/organizations/using-tokens)) emitido para usuarios con membresía en una organización. También puedes agregar información complementaria asociada a una organización de Auth0 añadiendo un claim personalizado al token de acceso:

```js wrap lines
context.accessToken['http://travel0.net/multifactor'] = context.multifactor;
```

**Nota**: Si ha configurado su tenant para admitir nombres de organización en la Authentication API, el claim `org_name` se incluye automáticamente en los tokens de acceso. Para obtener más información, consulte [Use Organization Names in Authentication API](/docs/es/manage-users/organizations/configure-organizations/use-org-name-authentication-api).

Como alternativa, podría crear una <Tooltip tip="Audiencia: identificador único de la audiencia de un token emitido. Se denomina aud en un token y su valor contiene el ID de una aplicación (ID de cliente) para un token de ID o de una API (identificador de API) para un token de acceso." cta="Ver glosario" href="/docs/es/glossary?term=audience">audiencia</Tooltip> de API única para cada organización, lo que daría lugar a una definición de API única en Auth0. Aunque este mecanismo puede reducir la necesidad de usar extensibilidad personalizada con Rule, la complejidad que introduce puede ser difícil de gestionar. A continuación, se muestra una comparación sencilla:

| Enfoque                | Ventajas                                                                                                                                                                                                                                                                        | Desventajas                                                                                                                                                                                                                                                                                                |
| ---------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Audiencia de API única | <ul><li>Compatibilidad lista para usar con acceso de máquina a máquina para una sola organización.</li><li>La audiencia es un claim estándar en un token de acceso.</li><li>El procesamiento del token de actualización no requiere lógica adicional de organización.</li></ul> | <ul><li>Se debe automatizar la creación de una API para cada organización.</li><li>Puede ser necesario crear Roles independientes si se usa RBAC.</li>Se debe automatizar el aprovisionamiento de Roles para Membership.<li>La implementación de la API debe dar soporte a múltiples audiencias.</li></ul> |
| Claim personalizado    | Simplifica la configuración del tenant de Auth0.                                                                                                                                                                                                                                | Se necesita código personalizado en una rule para agregar la organización al token de acceso.                                                                                                                                                                                                              |

<div id="roles">
  ### Roles
</div>

La funcionalidad Auth0 Organizations también admite el [control de acceso basado en roles (RBAC)](/docs/es/manage-users/access-control/rbac) mediante la funcionalidad [Authorization Core](/docs/es/manage-users/access-control/configure-core-rbac) asociada a un tenant de Auth0. RBAC [se aplica en el nivel de membresía de la organización de Auth0](/docs/es/manage-users/organizations/configure-organizations/add-member-roles).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Puede [habilitar RBAC de Auth0 Authorization Core para una API](/docs/es/get-started/apis/enable-role-based-access-control-for-apis), lo que hará que el claim `scope` predeterminado de los tokens de acceso se modifique automáticamente y que se agregue un claim `permission` de forma predeterminada (para ver un ejemplo, consulte [Uso de tokens y Organizations](/docs/es/manage-users/organizations/using-tokens)). También puede agregar información de roles a los tokens de identidad como claims personalizadas accediendo al objeto `authorization`, disponible en el objeto `context` de Rules. Para obtener más información, consulte [Casos de uso de ejemplo de Rules con Authorization: agregar roles de usuario a los tokens](/docs/es/manage-users/access-control/sample-use-cases-rules-with-authorization).
</Callout>

<div id="access-control">
  ## Control de acceso
</div>

La aplicación de políticas a nivel de recurso es responsabilidad de las aplicaciones y/o API de su sistema. Si intenta aplicar políticas en un <Tooltip tip="Servidor de autorización: servidor centralizado que contribuye a definir los límites del acceso de un usuario. Por ejemplo, su servidor de autorización puede controlar los datos, las tareas y las funcionalidades disponibles para un usuario." cta="Ver glosario" href="/docs/es/glossary?term=Authorization+Server">servidor de autorización</Tooltip> centralizado, como su tenant de Auth0, pronto se encontrará con un sistema de control complejo, difícil de mantener y de comprender. En su lugar, su servidor de autorización centralizado puede garantizar que se incluya en los tokens la información adecuada sobre un usuario, para que sus aplicaciones y API dispongan de la información necesaria para tomar decisiones sobre la aplicación de políticas. Como mínimo, en situaciones en las que hay demasiada información para incluirla en un solo token (por ejemplo, permisos a nivel de recurso) o en las que la información cambia con suficiente frecuencia como para quedar desactualizada si se consulta directamente en el token, sus aplicaciones y API deberían poder consultar la información correcta.

Por otro lado, cierta aplicación de políticas de alto nivel puede gestionarse de forma centralizada. Por ejemplo, al usar el contexto de un tenant de Auth0, hay algunas situaciones en las que podría implementar una Rule que reduzca la necesidad de que cada aplicación y/o API aplique las mismas restricciones. Entre ellas se incluyen:

* bloquear el acceso a usuarios desde una dirección IP determinada
* implementar requisitos específicos en torno al MFA contextual o <Tooltip tip="MFA adaptativo: autenticación multifactor (MFA) que solo se activa para los usuarios cuando se determina que un intento de inicio de sesión es de baja confianza." cta="Ver glosario" href="/docs/es/glossary?term=Adaptive+MFA">MFA adaptativo</Tooltip>
* restringir el inicio de sesión únicamente a los usuarios que hayan verificado su dirección de correo electrónico
* restringir el acceso a una audiencia de API determinada, de modo que un usuario no pueda obtener un token de acceso para ninguna otra audiencia de API o no pueda obtener un token de acceso para esa audiencia en determinadas circunstancias. En este caso, si está creando una audiencia de API personalizada para cada organización, su Rule también debe garantizar que el usuario que se autentica pertenezca a la organización que coincida con la audiencia de API correspondiente.

<Warning>
  El acceso a Auth0 Management API no está restringido por organización; el acceso a Auth0 Management API se realiza mediante un token de acceso asignado para su uso en un contexto de máquina a máquina y no puede restringirse mediante una organización de Auth0. Por lo tanto, no proporcione a sus clientes acceso directo a su instancia de Auth0 Management API. Si sus clientes deben gestionar aspectos de su organización, como las cuentas de usuario (consulte [Administración del perfil](/docs/es/get-started/architecture-scenarios/multiple-organization-architecture/single-identity-provider-organizations/profile-management)), entonces debería crear su propia aplicación y/o API independiente para este fin.
</Warning>
