description: Aprende sobre el aprovisionamiento de organizaciones para arquitecturas multi-tenant.
title: 'Proveedor de identidad único: Aprovisionamiento'
---

Al utilizar la funcionalidad Auth0 Organizations, se puede aprovisionar un único tenant de Auth0 para su despliegue en un entorno de producción. Salvo en los escenarios de arquitectura más complejos, se recomienda aprovisionar un único tenant de Auth0 para usarlo en un entorno de producción, ya que facilita la integración y el uso del [inicio de sesión único (SSO)](/docs/es/authenticate/single-sign-on), la [gestión de perfiles](/docs/es/get-started/architecture-scenarios/multiple-organization-architecture/single-identity-provider-organizations/profile-management) de usuario, entre otras cosas. Según la implementación, deberá abordar algunos aspectos adicionales relacionados con la configuración de su tenant de Auth0 y de la integración correspondiente.

La identidad visual asociada a una organización es extremadamente valiosa porque el uso de elementos de marca ofrece a los usuarios un entorno que conocen y en el que confían. El uso de elementos de marca reconocibles también aumenta la confianza de los usuarios en que la información que proporcionan (por ejemplo, las credenciales) se gestionará de forma segura. Por lo tanto, la identidad visual predeterminada de Auth0 lista para usar debe sustituirse. Para obtener más información, consulte [Branding](/docs/es/get-started/architecture-scenarios/multiple-organization-architecture/single-identity-provider-organizations/branding).

<div id="organizations">
  ## Organizations
</div>

Debes crear una Auth0 Organization independiente para cada una de las organizaciones a las que darás soporte. En este caso, crearemos la organización `hoekstra` para representar a Hoekstra &amp; Associates en nuestro ejemplo, y la organización `metahexa` para representar a MetaHexa Bank. Puedes [crear organizaciones](/docs/es/manage-users/organizations/configure-organizations/create-organizations) manualmente desde el <Tooltip tip="Auth0 Dashboard: el producto principal de Auth0 para configurar tus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip> o de forma programática mediante la <Tooltip tip="Management API: un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip>.

<div id="applications">
  ## Applications
</div>

Según cómo esté diseñada la implementación de su tenant de organización, tendrá distintas opciones al crear definiciones de [aplicación](/docs/es/get-started/applications) dentro de su tenant de Auth0. Independientemente de la opción que elija, el [comportamiento de la organización se define a nivel de la aplicación](/docs/es/manage-users/organizations/configure-organizations/define-organization-behavior).

Si aprovisiona un tenant de organización independiente para cada uno de sus clientes, normalmente necesitará una definición de aplicación independiente en Auth0 para cada uno. Esta configuración también suele implicar el envío tanto del parámetro `client_id` específico de la aplicación como del parámetro `organization`, que identifica qué organización de Auth0 debe usarse, en la llamada al endpoint `/authorize`. Para obtener más información, consulte [Autenticación](/docs/es/get-started/architecture-scenarios/multiple-organization-architecture/single-identity-provider-organizations/authentication).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  **Práctica recomendada**

  Para facilitar la configuración y maximizar el aislamiento de seguridad, defina las aplicaciones en Auth0 de forma independiente. Esto permite configurar por separado elementos como las URL de devolución de llamada permitidas y, siguiendo el principio de privilegio mínimo, minimiza la posible exposición del ID de cliente y del secreto de cliente.
</Callout>

Como alternativa, se admite el uso de una sola definición de aplicación en Auth0. En este caso, se solicitará al usuario que especifique la organización requerida como parte de la autenticación de primer factor. Normalmente, esto requerirá el uso de un `client_id` común de la aplicación, pero el parámetro `organization` se omitirá en la llamada al endpoint `/authorize`.

<div id="connections">
  ## Conexiones
</div>

A continuación, define las [conexiones](/docs/es/authenticate/identity-providers) que se usarán para autenticar a los usuarios. En este caso, definiremos una [conexión de base de datos](/docs/es/authenticate/database-connections) para los usuarios asociados a Hoekstra &amp; Associates y una [conexión empresarial](/docs/es/authenticate/identity-providers/enterprise-identity-providers) para los usuarios asociados a MetaHexa Bank.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  **Práctica recomendada**

  Para las Organizations con proveedor de identidad único (IdP), crea una conexión para cada organización definida a fin de contar con flexibilidad para distintos casos de uso. Por ejemplo, una sola [conexión de base de datos o de base de datos personalizada](/docs/es/authenticate/database-connections) por organización te permite eliminar fácilmente a los usuarios asociados a organizaciones que se desactivan y ofrece la máxima flexibilidad para organizaciones con distintos requisitos de complejidad de contraseña.
</Callout>

Una vez definida la conexión, se puede aprovisionar en la Auth0 Organization correspondiente mediante el Auth0 Dashboard o la Auth0 Management API. Para obtener más información, consulta [Habilitar conexiones de organización](/docs/es/manage-users/organizations/configure-organizations/enable-connections).

<div id="users">
  ## Users
</div>

En el caso de los usuarios que se autentican a través de conexiones distintas de conexiones de base de datos o conexiones de base de datos personalizadas, el usuario se aprovisiona en el <Tooltip tip="Proveedor de identidad (IdP): servicio que almacena y gestiona identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=Identity+Provider">proveedor de identidad</Tooltip> (IdP) externo de forma independiente de Auth0 y de la manera habitual. Por otro lado, los usuarios que se autentican mediante conexiones de base de datos o conexiones de base de datos personalizadas pueden aprovisionarse de varias formas. El Auth0 Dashboard y la Auth0 Management API pueden utilizarse para crear un usuario directamente en tu tenant de Auth0. También ofrecemos [migración automática](/docs/es/manage-users/user-migration/configure-automatic-migration-from-your-database) y [migración masiva](/docs/es/manage-users/user-migration/bulk-user-imports).

<Warning>
  Para que un usuario se aprovisione mediante la Management API o el Dashboard, debes habilitar directamente una conexión de base de datos o conexión de base de datos personalizada para al menos una aplicación. No basta con habilitar una conexión de base de datos o conexión de base de datos personalizada solo para una organización.
</Warning>

Luego, los usuarios se asocian a una Auth0 Organization asignándoles membresías, y una Auth0 Organization puede configurarse para [asignar automáticamente la membresía del usuario](/docs/es/manage-users/organizations/configure-organizations/grant-just-in-time-membership) o [hacerlo manualmente](/docs/es/manage-users/organizations/configure-organizations/assign-members).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Para asignar manualmente una membresía a una organización, el usuario ya debe tener un [perfil de usuario](/docs/es/manage-users/user-accounts/user-profiles) definido en Auth0. Puedes [asignar manualmente la membresía](/docs/es/manage-users/organizations/configure-organizations/assign-members) mediante el Dashboard del tenant de Auth0 o la Auth0 Management API.
</Callout>

<div id="invitation">
  ### Invitación
</div>

La funcionalidad Auth0 Organization también permite usar la invitación de miembros. En el flujo de trabajo de invitación de miembros, al invitar a un usuario a una aplicación, el usuario se aprovisiona automáticamente y su membresía se genera también de forma automática.

<div id="database-connection">
  #### Conexión de base de datos
</div>

Con nuestro ejemplo de Hoekstra &amp; Associates, veamos cómo podría ser esta implementación cuando se utiliza una conexión de base de datos como parte de una invitación de usuario; normalmente, la mayor parte del flujo de trabajo descrito se gestionará mediante el SDK de Auth0 o la biblioteca correspondiente de tu pila tecnológica:

<Frame>![Escenarios de arquitectura - MOA - Usuarios aislados, aplicaciones compartidas, flujo de invitación (conexión de base de datos)](/docs/images/cdy7uua7fh8z/2QMwMeBQ6U9TjLtbjPgNwe/1b436cd226b95355e73a93d5f9bc0be3/Isolated_Users__Shared_Apps__Invitation_Flow__Database_Connection_.png)</Frame>

1. Jennifer, de Hoekstra &amp; Associates, recibe un correo electrónico enviado desde el tenant de Auth0 de Travel0 en nombre de la instancia de Travel0 Corporate Booking de Hoekstra &amp; Associates.

   1. El correo electrónico se envió tal como se describe en [Invitar a miembros de la organización](/docs/es/manage-users/organizations/configure-organizations/invite-members) y es posible que se haya enviado desde el Auth0 Dashboard o la Auth0 Management API.
2. Jennifer abre el correo electrónico y hace clic en el enlace que contiene. Al hacerlo, su navegador es redirigido a la instancia de Travel0 Corporate Booking de Hoekstra &amp; Associates. La URL base utilizada en el enlace se especifica como el [URI de inicio de sesión de la aplicación](/docs/es/get-started/applications/application-settings), que forma parte de la definición de la aplicación Travel0 Corporate Booking en el tenant de Auth0 de Travel0 de Hoekstra &amp; Associates.

   1. El enlace contiene los parámetros `organization` y `organization_name`. El parámetro `organization` se configura con el ID de la definición de Auth0 Organization correspondiente en tu tenant de Auth0. Este valor se reenviará al tenant de Auth0 como parte del paso 3.
   2. El enlace también contiene el parámetro `invitation`, que se reenviará como parte del paso 3.
3. La instancia de Travel0 Corporate Booking de Hoekstra &amp; Associates redirige al tenant de Auth0 de Travel0 mediante el [flujo de código de autorización](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow) (con o sin [PKCE](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce)), llamando al endpoint `/authorize` y pasando parámetros similares a los siguientes, normalmente mediante el uso de un [SDK de Auth0](/docs/es/libraries) o una biblioteca de terceros:

   1. `redirect_uri`: [`https://hoekstra.corp.travel0.net/login/callback`](https://hoekstra.corp.travel0.net/login/callback)
   2. `response_type`: `code`
   3. `state`: [state](/docs/es/secure/attack-protection/state-parameters) único generado en esta sesión
   4. `scope`: `openid profile` ...
   5. cualquier [scope de OIDC](/docs/es/get-started/apis/scopes/openid-connect-scopes) adicional necesario, según la información requerida sobre el usuario.
   6. `client_id`: ID de cliente asociado a la aplicación creada en el tenant de Auth0 de Travel0 para la instancia de Travel0 Corporate Booking de Hoekstra &amp; Associates.
   7. `organization`: ID de la organización que invita, normalmente obtenido a través del enlace del correo electrónico descrito en el paso 2. Se especifica con el formato `organization=`organization&#95;id, donde organization&#95;id se establece con el identificador asociado a la definición correspondiente de Auth0 Organization en tu tenant de Auth0.
   8. `invitation`: Parámetro `invitation` adicional asociado al enlace del correo electrónico, como se describe en el paso 2.
4. El tenant de Auth0 de Travel0 redirige a `/signup/invitation` para solicitarle al usuario una contraseña.

   1. Se muestra una Universal Login Page, que puede configurar para mostrar elementos de marca específicos de la organización, como se describe en [Branding](/docs/es/get-started/architecture-scenarios/multiple-organization-architecture/single-identity-provider-organizations/branding).

      <Warning>
        El flujo de trabajo de invitación asociado con la funcionalidad Auth0 Organizations no tiene en cuenta ninguna sesión de SSO activa en Auth0. Si se invita a un usuario a registrarse y este ya ha iniciado sesión, al hacer clic en el enlace del correo electrónico siempre se mostrará la Universal Login Page asociada.
      </Warning>
5. El usuario introduce su contraseña (y cualquier credencial adicional, como el nombre de usuario) y hace clic en Continuar. El ID de usuario se establece como la dirección de correo electrónico asociada al usuario y no puede modificarse.
6. El tenant de Auth0 de Travel0 comprueba las credenciales. Si son válidas, se aprovisiona al usuario y se establece su pertenencia a Auth0 Organization. El usuario queda autenticado implícitamente y se ejecuta el pipeline de [Rules](/docs/es/customize/rules). Las Rules se pueden usar para gestionar el control de acceso, como se describe en [Authorization](/docs/es/get-started/architecture-scenarios/multiple-organization-architecture/single-identity-provider-organizations/authorization).

   1. Si las credenciales del usuario no son válidas, se le pedirá que vuelva a introducirlas.
7. Tras comprobar correctamente las credenciales y ejecutar las Rules, se redirige al usuario al `redirect_uri` ([`https://hoekstra.corp.travel0.net/login/callback`](https://hoekstra.corp.travel0.net/login/callback)) con el `state` enviado en el paso 3, así como un `code`.
8. La instancia de Travel0 Corporate Booking de Hoekstra &amp; Associates valida el `state` y luego llama al tenant de Auth0 de Travel0 en [`https://auth.travel0.net/oauth/token`](https://auth.travel0.net/oauth/token), pasando el `code`, su `ID de cliente` y su `secreto de cliente` a cambio del [token de ID](/docs/es/secure/tokens/id-tokens). Luego, el token de ID se usa para generar una sesión para [`https://hoekstra.corp.travel0.net`](https://hoekstra.corp.travel0.net).
9. La instancia de Travel0 Corporate Booking de Hoekstra &amp; Associates muestra al usuario la página correspondiente.

<div id="enterprise-connection">
  #### Conexión empresarial
</div>

Usando nuestro ejemplo de MetaHexa Bank, veamos cómo podría desarrollarse esta implementación cuando se usa una conexión empresarial como parte de una invitación de usuarios; de nuevo, la mayor parte del flujo de trabajo descrito normalmente se gestionará mediante el SDK de Auth0 o la biblioteca correspondiente de tu pila tecnológica:

<Frame>![Architecture Scenarios - MOA - Isolated Users, Shared Apps, Invitation Flow (Enterprise Connection)](/docs/images/cdy7uua7fh8z/1IsdaHpprvIp17R5zYIg7q/c2c73360fb21657ab2ae98c0981b1cc6/Isolated_Users__Shared_Apps__Invitation_Flow__Enterprise_Connection_.png)</Frame>

1. Amintha, de MetaHexa Bank, recibe un correo electrónico enviado desde el tenant de Auth0 de Travel0 en nombre de la instancia de Travel0 Corporate Booking de MetaHexa Bank.

   1. El correo electrónico se envió como se describe en [Invite Organization Members](/docs/es/manage-users/organizations/configure-organizations/invite-members) y puede haberse desencadenado mediante el Auth0 Dashboard o la Auth0 Management API.
2. Amintha abre el correo electrónico y hace clic en el enlace que contiene. Al hacerlo, su navegador se dirige a la instancia de Travel0 Corporate Booking de MetaHexa Bank. La URL base usada en el enlace se especifica como el [Application Login URI](/docs/es/get-started/applications/application-settings), que forma parte de la definición de la aplicación de la instancia de Travel0 Corporate Booking de MetaHexa Bank en el tenant de Auth0 de Travel0.

   1. El enlace contiene los parámetros `organization` y `organization_name`. El parámetro `organization` se establece en el ID de la definición de la organización de Auth0 correspondiente en tu tenant de Auth0. Esto se reenviará al tenant de Auth0 como parte del paso 3.
   2. El enlace también contiene el parámetro `invitation`, que igualmente se reenviará como parte del paso 3.
3. La instancia de Travel0 Corporate Booking de MetaHexa Bank redirige al tenant de Auth0 de Travel0 mediante el [Authorization Code Flow](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow) (con o sin [PKCE](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce)) llamando al endpoint `/authorize` y pasando parámetros similares a los siguientes, normalmente mediante el uso de un [Auth0 SDK](/docs/es/libraries) o una biblioteca de terceros:

   1. `redirect_uri`: [`https://metahexa.corp.travel0.net/login/callback`](https://metahexa.corp.travel0.net/login/callback)
   2. `response_type`: `code`
   3. `state`: Unique [state](/docs/es/secure/attack-protection/state-parameters) generado en esta sesión
   4. `scope`: `openid profile` ...
   5. any necessary additional [OIDC Scopes](/docs/es/get-started/apis/scopes/openid-connect-scopes), depending on the information required about the user.
   6. `client_id`: Client ID asociado con la aplicación creada en el tenant de Auth0 de Travel0 para la instancia de Travel0 Corporate Booking de MetaHexa Bank.
   7. `organization`: ID de la organización que invita, normalmente obtenido del enlace del correo electrónico descrito en el paso 2. Se especifica con el formato `organization=`organization&#95;id, donde organization&#95;id se establece en el identificador asociado con la definición de la organización de Auth0 correspondiente en tu tenant de Auth0.
   8. `invitation`: Parámetro `invitation` adicional asociado con el enlace del correo electrónico, como se describe en el paso 2.
4. El tenant de Auth0 de Travel0 redirige a `/invitation`, donde se informa a Amintha de que será redirigida al IdP de MetaHexa para autenticarse con las credenciales del primer factor.

   1. La usuaria confirma, y
   2. Auth0 redirige a la instancia del IdP de MetaHexa Bank, donde
   3. Se muestra la página de inicio de sesión, y la usuaria introduce sus credenciales y hace clic en `login`.
5. Si el proceso se completa correctamente, se establece la membresía en la organización de Auth0, la usuaria queda autenticada de forma implícita y se ejecuta la canalización de [Rules](/docs/es/customize/rules). Rules puede usarse para gestionar el control de acceso como se describe en [Authorization](/docs/es/get-started/architecture-scenarios/multiple-organization-architecture/single-identity-provider-organizations/authorization).

Los pasos del 6 al 8 coincidirán con los descritos en el escenario de [conexión de base de datos](#database-connection), pero con Amintha como usuaria en lugar de Jennifer, y usando MetaHexa Bank (`metahexa.corp.travel0.net`) en lugar de Hoekstra &amp; Associates.

<div id="social-connection">
  #### Conexión social
</div>

La invitación mediante una conexión social sigue un patrón similar al de una [conexión empresarial](#enterprise-connections), pero el IdP de origen está asociado al proveedor social y no a una organización específica. Para conocer otras consideraciones sobre el uso de las conexiones sociales, consulta [Autenticación](/docs/es/get-started/architecture-scenarios/multiple-organization-architecture/single-identity-provider-organizations/authentication).
