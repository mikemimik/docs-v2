---

Al usar la funcionalidad Auth0 Organizations, se puede aprovisionar un único tenant de Auth0 para su despliegue en un entorno de producción. Salvo en los escenarios de arquitectura más complejos, se recomienda aprovisionar un único tenant de Auth0 para su uso en un entorno de producción, ya que facilita la integración y el uso del [inicio de sesión único (SSO)](/docs/es/authenticate/single-sign-on), la [gestión de perfiles](/docs/es/get-started/architecture-scenarios/multiple-organization-architecture/single-identity-provider-organizations/profile-management) de los usuarios, etc. Según la implementación, deberás abordar algunos aspectos adicionales relacionados con la Setup de tu tenant de Auth0 y la integración correspondiente.

El Branding asociado a una organización es extremadamente valioso porque el uso de elementos de marca proporciona a los usuarios un entorno que conocen y en el que confían. El uso de elementos de marca reconocibles también aumenta la confianza de los usuarios en que la información que proporcionan (por ejemplo, las credenciales) se gestionará de forma segura. Por lo tanto, debe reemplazarse el Branding predeterminado de Auth0. Para obtener más información, consulta [Branding](/docs/es/get-started/architecture-scenarios/multiple-organization-architecture/single-identity-provider-organizations/branding).

<div id="organizations">
  ## Organizations
</div>

Debes crear una organización independiente para cada una de las organizaciones a las que vayas a dar soporte. En este caso, crearemos la organización `hoekstra` para representar a Hoekstra &amp; Associates en nuestro ejemplo, y la organización `metahexa` para representar a MetaHexa Bank. Puedes [crear organizaciones](/docs/es/manage-users/organizations/configure-organizations/create-organizations) ya sea manualmente a través del <Tooltip tip="Auth0 Dashboard: El producto principal de Auth0 para configurar tus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip> o de forma programática mediante la Auth0 <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip>.

<div id="applications">
  ## Applications
</div>

Según cómo esté diseñada la implementación de tu tenant de organización, tienes distintas opciones al crear definiciones de [aplicación](/docs/es/get-started/applications) dentro de tu tenant de Auth0. Independientemente de la opción que elijas, el [comportamiento de la organización se define a nivel de la aplicación](/docs/es/manage-users/organizations/configure-organizations/define-organization-behavior).

Si aprovisionas un tenant de organización independiente para cada uno de tus clientes, normalmente necesitarás una definición de aplicación independiente en Auth0 para cada uno. Esta configuración también suele implicar enviar tanto el parámetro `client_id` específico de la aplicación como el parámetro `organization`, que identifica qué Auth0 Organization se debe usar, como parte de la llamada al endpoint `/authorize`. Para obtener más información, consulta [Autenticación](/docs/es/get-started/architecture-scenarios/multiple-organization-architecture/single-identity-provider-organizations/authentication).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  **Práctica recomendada**

  Para facilitar la configuración y maximizar el aislamiento de seguridad, define las Applications en Auth0 de forma independiente. Esto permite configurar por separado elementos como las URL de devolución de llamada permitidas y, siguiendo el principio de mínimo privilegio, minimiza la posible exposición de información del ID de cliente y del secreto de cliente.
</Callout>

Como alternativa, se admite el uso de una única definición de aplicación en Auth0. En este caso, se solicitará al usuario que especifique la organización requerida como parte de la autenticación del primer factor. Esto normalmente requerirá el uso de un `client_id` de aplicación común, pero el parámetro `organization` se omitirá en la llamada al endpoint `/authorize`.

<div id="connections">
  ## Conexiones
</div>

A continuación, defina las [conexiones](/docs/es/authenticate/identity-providers) que se utilizarán para autenticar a los usuarios. En este caso, definiremos una [conexión de base de datos](/docs/es/authenticate/database-connections) para los usuarios asociados con Hoekstra &amp; Associates y una [conexión empresarial](/docs/es/authenticate/identity-providers/enterprise-identity-providers) para los usuarios asociados con MetaHexa Bank.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  **Práctica recomendada**

  Para las Organizations con un proveedor de identidad único (IdP), cree una conexión para cada organización definida a fin de ofrecer flexibilidad para distintos casos de uso. Por ejemplo, una sola [conexión de base de datos or Custom Database Connection](/docs/es/authenticate/database-connections) por organización le permite eliminar fácilmente a los usuarios asociados con organizaciones que se dan de baja y ofrece la máxima flexibilidad para aquellas organizaciones que tienen distintos requisitos de complejidad de contraseña.
</Callout>

Una vez definida la conexión, se puede aprovisionar en la Auth0 Organization correspondiente mediante el Auth0 Dashboard o la Auth0 Management API. Para obtener más información, consulte [Habilitar conexiones de organización](/docs/es/manage-users/organizations/configure-organizations/enable-connections).

<div id="users">
  ## Users
</div>

Para los usuarios que se autentican mediante conexiones distintas de conexiones de base de datos o Custom Database Connections, el usuario se aprovisiona en el <Tooltip tip="Identity Provider (IdP): Servicio que almacena y gestiona identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=Identity+Provider">proveedor de identidad</Tooltip> (IdP) externo, de forma independiente de Auth0 y de la manera habitual. Por otro lado, los usuarios que se autentican mediante conexiones de base de datos o Custom Database Connections pueden aprovisionarse de varias maneras. El Auth0 Dashboard y la Auth0 Management API pueden utilizarse para crear un usuario directamente en tu tenant de Auth0. También admitimos la [migración automática](/docs/es/manage-users/user-migration/configure-automatic-migration-from-your-database) y la [migración masiva](/docs/es/manage-users/user-migration/bulk-user-imports).

<Warning>
  Para que un usuario se aprovisione mediante la Auth0 Management API o el Auth0 Dashboard, debes habilitar directamente una conexión de base de datos o Custom Database Connection para al menos una aplicación. No basta con habilitar una conexión de base de datos o Custom Database Connection solo para una organización.
</Warning>

Luego, los usuarios se asocian con una organización de Auth0 asignándoles membresías, y una organización de Auth0 puede configurarse para [asignar automáticamente la membresía de un usuario](/docs/es/manage-users/organizations/configure-organizations/grant-just-in-time-membership) o [hacerlo de forma manual](/docs/es/manage-users/organizations/configure-organizations/assign-members).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Para asignar manualmente la membresía de una organización a un usuario, ese usuario ya debe tener un [perfil de usuario](/docs/es/manage-users/user-accounts/user-profiles) definido en Auth0. Puedes [asignar la membresía manualmente](/docs/es/manage-users/organizations/configure-organizations/assign-members) con el Dashboard del tenant de Auth0 o la Auth0 Management API.
</Callout>

<div id="invitation">
  ### Invitación
</div>

La funcionalidad organización también permite usar la Invitación de miembros. En el flujo de Invitación de miembros, al invitar a un usuario a una aplicación, el usuario se crea automáticamente y su membresía se genera de forma automática.

<div id="database-connection">
  #### Conexión de base de datos
</div>

Siguiendo con nuestro ejemplo de Hoekstra &amp; Associates, veamos cómo podría desarrollarse esta implementación cuando se utiliza una conexión de base de datos como parte de una invitación de usuarios; por lo general, la mayor parte del flujo de trabajo descrito se gestionará con el SDK de Auth0 o la biblioteca correspondientes a tu pila tecnológica:

<Frame>![Escenarios de arquitectura - MOA - Usuarios aislados, aplicaciones compartidas, flujo de invitación (conexión de base de datos)](/docs/images/cdy7uua7fh8z/2QMwMeBQ6U9TjLtbjPgNwe/1b436cd226b95355e73a93d5f9bc0be3/Isolated_Users__Shared_Apps__Invitation_Flow__Database_Connection_.png)</Frame>

1. Jennifer, de Hoekstra &amp; Associates, recibe un correo electrónico enviado desde el tenant de Auth0 de Travel0 en nombre de la instancia de Travel0 Corporate Booking de Hoekstra &amp; Associates.

   1. El correo electrónico se envió como se describe en [Invitar a miembros de la organización](/docs/es/manage-users/organizations/configure-organizations/invite-members) y puede haberse activado mediante Auth0 Dashboard o Auth0 Management API.
2. Jennifer abre el correo electrónico y hace clic en el enlace incluido. Al hacerlo, su navegador accede a la instancia de Travel0 Corporate Booking de Hoekstra &amp; Associates. La URL base utilizada en el enlace se especifica como el [Application Login URI](/docs/es/get-started/applications/application-settings), que forma parte de la definición de la aplicación Travel0 Corporate Booking de Hoekstra &amp; Associates en el tenant de Auth0 de Travel0.

   1. El enlace contiene los parámetros `organization` y `organization_name`. El parámetro `organization` se establece con el ID de la definición correspondiente de Auth0 Organization en tu tenant de Auth0. Este se enviará al tenant de Auth0 como parte del paso 3.
   2. El enlace también contiene el parámetro `invitation`, que también se enviará como parte del paso 3.
3. La instancia de Travel0 Corporate Booking de Hoekstra &amp; Associates redirige al tenant de Auth0 de Travel0 mediante el [flujo de código de autorización](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow) (con o sin [PKCE](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce)), al llamar al endpoint `/authorize` y pasar parámetros como los siguientes, normalmente con un [SDK de Auth0](/docs/es/libraries) o una biblioteca de terceros:

   1. `redirect_uri`: [`https://hoekstra.corp.travel0.net/login/callback`](https://hoekstra.corp.travel0.net/login/callback)
   2. `response_type`: `code`
   3. `state`: [state](/docs/es/secure/attack-protection/state-parameters) único generado en esta sesión
   4. `scope`: `openid profile` ...
   5. cualquier [scope de OIDC](/docs/es/get-started/apis/scopes/openid-connect-scopes) adicional que sea necesario, según la información requerida sobre el usuario.
   6. `client_id`: ID de cliente asociado a la aplicación creada en el tenant de Auth0 de Travel0 para la instancia de Travel0 Corporate Booking de Hoekstra &amp; Associates.
   7. `organization`: ID de la organización que envía la invitación, normalmente obtenido a través del enlace del correo electrónico descrito en el paso 2. Se especifica con el formato `organization=`organization&#95;id, donde organization&#95;id se establece como el identificador asociado a la definición correspondiente de Auth0 Organization en tu tenant de Auth0.
   8. `invitation`: Parámetro `invitation` adicional asociado al enlace del correo electrónico, como se describe en el paso 2.
4. El tenant de Auth0 de Travel0 redirige a `/signup/invitation` para que el usuario cree una contraseña.

   1. Se muestra una Universal Login Page, que puede configurar para que muestre elementos de marca específicos de la organización, como se describe en [Identidad visual](/docs/es/get-started/architecture-scenarios/multiple-organization-architecture/single-identity-provider-organizations/branding).

      <Warning>
        El flujo de trabajo de invitación asociado con la funcionalidad Auth0 Organizations no tiene en cuenta ninguna sesión de SSO mantenida con Auth0. Si se invita a un usuario a registrarse y este ya ha iniciado sesión, al hacer clic en el enlace del correo electrónico siempre se mostrará la Universal Login Page asociada.
      </Warning>
5. El usuario introduce su contraseña (y cualquier otra credencial adicional, como el nombre de usuario) y hace clic en Continuar. El ID de usuario se establece como la dirección de correo electrónico asociada al usuario y no se puede cambiar.
6. El tenant de Auth0 de Travel0 verifica las credenciales. Si son válidas, se aprovisiona al usuario y se establece la membresía en la organización de Auth0. El usuario queda autenticado de forma implícita y se ejecuta el pipeline de [Rules](/docs/es/customize/rules). Rules se puede usar para gestionar el control de acceso, como se describe en [Autorización](/docs/es/get-started/architecture-scenarios/multiple-organization-architecture/single-identity-provider-organizations/authorization).

   1. Si las credenciales del usuario no son válidas, se le pedirá que las vuelva a introducir.
7. Tras verificar correctamente las credenciales y ejecutar Rules, se redirige al usuario a `redirect_uri` ([`https://hoekstra.corp.travel0.net/login/callback`](https://hoekstra.corp.travel0.net/login/callback)) con el `state` enviado en el paso 3, así como un `code`.
8. La instancia de Travel0 Corporate Booking de Hoekstra &amp; Associates valida el `state` y luego llama al tenant de Auth0 de Travel0 en [`https://auth.travel0.net/oauth/token`](https://auth.travel0.net/oauth/token), pasando el `code`, su `client id` y su `client secret` para obtener el [token de ID](/docs/es/secure/tokens/id-tokens). Luego, el token de ID se utiliza para generar una sesión para [`https://hoekstra.corp.travel0.net`](https://hoekstra.corp.travel0.net).
9. La instancia de Travel0 Corporate Booking de Hoekstra &amp; Associates muestra al usuario la página correspondiente.

<div id="enterprise-connection">
  #### Conexión empresarial
</div>

Usando nuestro ejemplo de MetaHexa Bank, veamos cómo podría desarrollarse esta implementación cuando se utiliza una conexión empresarial como parte de la invitación de usuario; nuevamente, la mayor parte del flujo de trabajo descrito normalmente se gestionará mediante el SDK de Auth0 o la biblioteca correspondiente asociada a tu pila tecnológica:

<Frame>![Architecture Scenarios - MOA - Isolated Users, Shared Apps, Invitation Flow (Enterprise Connection)](/docs/images/cdy7uua7fh8z/1IsdaHpprvIp17R5zYIg7q/c2c73360fb21657ab2ae98c0981b1cc6/Isolated_Users__Shared_Apps__Invitation_Flow__Enterprise_Connection_.png)</Frame>

1. Amintha, de MetaHexa Bank, recibe un correo electrónico enviado desde el tenant de Auth0 de Travel0 en nombre de la instancia de Travel0 Corporate Booking de MetaHexa Bank.

   1. El correo electrónico fue enviado tal como se describe en [Invite Organization Members](/docs/es/manage-users/organizations/configure-organizations/invite-members) y puede haberse activado mediante el Auth0 Dashboard o la Auth0 Management API.
2. Amintha abre el correo electrónico y hace clic en el enlace que contiene. Al hacerlo, su navegador es dirigido a la instancia de Travel0 Corporate Booking de MetaHexa Bank. La URL base utilizada en el enlace se especifica como el [Application Login URI](/docs/es/get-started/applications/application-settings), que forma parte de la definición de la aplicación de la instancia de Travel0 Corporate Booking de MetaHexa Bank en el tenant de Auth0 de Travel0.

   1. El enlace contiene los parámetros `organization` y `organization_name`. El parámetro `organization` se establece con el ID de la definición de organización correspondiente en tu tenant de Auth0. Este valor se reenviará al tenant de Auth0 como parte del paso 3.
   2. El enlace también contiene el parámetro `invitation`, que igualmente se reenviará como parte del paso 3.
3. La instancia de Travel0 Corporate Booking de MetaHexa Bank redirige al tenant de Auth0 de Travel0 mediante el [Authorization Code Flow](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow) (con o sin [PKCE](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce)), llamando al endpoint `/authorize` y pasando parámetros similares a los siguientes, normalmente a través de un [SDK de Auth0](/docs/es/libraries) o una biblioteca de terceros:

   1. `redirect_uri`: [`https://metahexa.corp.travel0.net/login/callback`](https://metahexa.corp.travel0.net/login/callback)
   2. `response_type`: `code`
   3. `state`: [State](/docs/es/secure/attack-protection/state-parameters) único generado en esta sesión
   4. `scope`: `openid profile` ...
   5. cualquier [scope de OIDC](/docs/es/get-started/apis/scopes/openid-connect-scopes) adicional que sea necesario, según la información requerida sobre el usuario.
   6. `client_id`: ID de cliente asociado con la aplicación creada en el tenant de Auth0 de Travel0 para la instancia de Travel0 Corporate Booking de MetaHexa Bank.
   7. `organization`: ID de la organización que envía la invitación, obtenido normalmente a través del enlace del correo electrónico descrito en el paso 2. Se especifica con el formato `organization=`organization&#95;id, donde organization&#95;id corresponde al identificador asociado a la definición de organización correspondiente en tu tenant de Auth0.
   8. `invitation`: Parámetro `invitation` adicional asociado al enlace del correo electrónico, tal como se describe en el paso 2.
4. El tenant de Auth0 de Travel0 redirige a `/invitation`, donde se informa a Amintha de que será redirigida al IdP de MetaHexa para autenticar sus credenciales de primer factor.

   1. La usuaria confirma, y
   2. Auth0 redirige a la instancia del IdP de MetaHexa Bank, donde
   3. Se muestra la página de inicio de sesión, la usuaria introduce sus credenciales y hace clic en `login`.
5. Si el proceso es exitoso, se establece la membresía en la organización, el usuario queda autenticado de forma implícita y se ejecuta el pipeline de [Rules](/docs/es/customize/rules). Las Rules pueden utilizarse para gestionar el control de acceso tal como se describe en [Authorization](/docs/es/get-started/architecture-scenarios/multiple-organization-architecture/single-identity-provider-organizations/authorization).

Los pasos 6 al 8 coincidirán con los descritos en el escenario de [conexión de base de datos](#database-connection), pero donde Amintha es la usuaria en lugar de Jennifer, y donde se utilizará MetaHexa Bank (`metahexa.corp.travel0.net`) en lugar de Hoekstra &amp; Associates.

<div id="social-connection">
  #### Conexión social
</div>

La invitación mediante una conexión social sigue un patrón similar al de una [conexión empresarial](#enterprise-connections), pero el IdP de origen está asociado con el proveedor social, en lugar de con una organización específica. Para conocer otras consideraciones sobre el uso de conexiones sociales, consulte [Autenticación](/docs/es/get-started/architecture-scenarios/multiple-organization-architecture/single-identity-provider-organizations/authentication).
