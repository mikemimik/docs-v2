description: Aprenda a cerrar la sesión de los usuarios en Organizations en arquitecturas multi-tenant.

---

Finalizar una sesión autenticada cuando ya no es necesaria, lo que también se conoce como cierre de sesión, es una buena práctica de seguridad. Ofrecer la posibilidad de cerrar sesión ayuda a mitigar varios posibles problemas de seguridad, incluida la posibilidad de que partes no autorizadas puedan &quot;apoderarse&quot; de una sesión. En nuestros escenarios de arquitectura, ofrecemos orientación general sobre [B2B Logout](/docs/es/get-started/architecture-scenarios/business-to-business/logout), cuya revisión recomendamos junto con la orientación que se ofrece aquí. Para este escenario, el cierre de sesión es casi idéntico al de cualquier otro sistema, por lo que tendrá el mismo nivel de complejidad que se describe en nuestra documentación estándar.

<div id="database-connection">
  ## Conexión de base de datos
</div>

Con nuestro ejemplo de Hoekstra &amp; Associates, el siguiente diagrama ilustra el flujo que suele producirse cuando se trata de un usuario autenticado mediante una conexión de base de datos de Auth0. Veamos este flujo para entender qué ocurre; tenga en cuenta que la mayor parte del flujo de trabajo descrito normalmente se gestionará mediante el SDK de Auth0 o la biblioteca correspondientes a su pila tecnológica:

<Frame>![Escenarios de arquitectura - MOA - Usuarios aislados, aplicaciones compartidas, flujo de cierre de sesión](/docs/images/cdy7uua7fh8z/5fQIKm5Izpy4zKw7ZsZoAE/e3661e2fa3e5bcd2258cfb6ef895f161/Isolated_Users__Shared_Apps__Logout_Flow__1_.png)</Frame>

1. Jennifer hace clic en `logout`.
2. La instancia de Travel0 Corporate Booking de Hoekstra &amp; Associates redirige el navegador al tenant de Auth0 de Travel0 en https://auth.travel0.net con los siguientes parámetros:

   1. `returnTo`: [`https://hoekstra.corp.travel0.net/logoutComplete`](https://hoekstra.corp.travel0.net/logoutComplete)
   2. `client_id`: ID de cliente asociado a la aplicación creada en el tenant de Auth0 de Travel0 para la instancia de Travel0 Corporate Booking de Hoekstra &amp; Associates.

      <Callout icon="file-lines" color="#0EA5E9" iconType="regular">
        **Práctica recomendada**

        Defina las aplicaciones de forma independiente en Auth0 para facilitar la configuración de la URL `returnTo`, donde el `client_id` correspondiente se usa para validar correctamente la URL proporcionada. Para obtener más información, consulte [Aprovisionamiento](/docs/es/get-started/architecture-scenarios/multiple-organization-architecture/single-identity-provider-organizations/provisioning).
      </Callout>
3. El tenant de Auth0 de Travel0 finaliza la sesión de Auth0 establecida en nombre del usuario, elimina cualquier información de SSO y redirige el navegador a la URL `returnTo` especificada.
4. La instancia de Travel0 Corporate Booking de Hoekstra &amp; Associates muestra una página para informarle a Jennifer que cerró sesión correctamente, probablemente con un botón para volver a iniciar sesión si lo desea.

   1. En esta etapa, la instancia de Travel0 Corporate Booking normalmente también querrá cerrar la sesión de la aplicación asociada al usuario.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Si no desea eliminar la sesión de SSO del usuario con Auth0, la aplicación puede simplemente finalizar la sesión de la aplicación sin redirigir al endpoint `/logout` del tenant de Auth0. Aunque el usuario habrá cerrado sesión en la aplicación, seguirá teniendo una sesión de SSO con Auth0. En este caso, probablemente querrá informarle al usuario que todavía mantiene una sesión iniciada en el tenant de Auth0 y que, si hace clic en `login` de nuevo, es posible que no se le solicite ingresar las credenciales del primer factor.
</Callout>

<div id="enterprise-connection">
  ## Conexión empresarial
</div>

En este escenario, implementar el cierre de sesión puede ser un poco más complejo que cuando se usa una conexión de base de datos. Sigues teniendo la opción de cerrar sesión solo en la aplicación o de hacer que al cerrar sesión en la aplicación también se cierre la sesión en el tenant de Auth0. Sin embargo, también puedes tener la opción de permitir el cierre de sesión en el <Tooltip tip="Proveedor de identidad (IdP): servicio que almacena y administra identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=Identity+Provider">proveedor de identidad</Tooltip> (IdP) de la organización, algo que la mayoría de las aplicaciones evitan, especialmente si los usuarios tienen acceso a otras aplicaciones corporativas en las que quizá necesiten seguir con la sesión iniciada. No obstante, evitarlo puede hacer que los usuarios que posteriormente hagan clic en el botón `login` se autentiquen automáticamente sin tener que proporcionar de forma interactiva las credenciales del primer factor. Esta funcionalidad puede generar una experiencia de usuario inesperada, por lo que deberías considerar comunicárselo al usuario.

Usando nuestro ejemplo de MetaHexa Bank, veamos cómo podría desarrollarse esta implementación del cierre de sesión con un usuario autenticado mediante una conexión empresarial con MetaHexa Bank; de nuevo, la mayor parte del flujo de trabajo descrito normalmente se gestionará mediante el SDK de Auth0 o la biblioteca correspondiente de tu pila tecnológica:

<Frame>![Escenarios de arquitectura - MOA - Usuarios aislados, aplicaciones compartidas, flujo de cierre de sesión empresarial](/docs/images/cdy7uua7fh8z/36ulIzcfP3GIymWql7SDW2/b350f739f26644364400330fce0a604c/Isolated_Users__Shared_Apps__Enterprise_Logout_Flow__1_.png)</Frame>

1. Amintha hace clic en `logout`.
2. La instancia de Travel0 Corporate Booking de MetaHexa Bank redirige el navegador al tenant de Auth0 de Travel0 en https://auth.travel0.net con los siguientes parámetros:

   1. `returnTo`: [`https://metahexa.corp.travel0.net/logoutComplete`](https://metahexa.corp.travel0.net/logoutComplete)
   2. `client_id`: ID de cliente asociado a la aplicación creada en el tenant de Auth0 de Travel0 para la instancia de Travel0 Corporate Booking de MetaHexa Bank.
   3. `federated`:  Parámetro opcional que se utiliza para cerrar la sesión del usuario en el IdP de MetaHexa Bank. Cuando se especifica, el navegador se redirige al endpoint `/logout` asociado al IdP de MetaHexa Bank, que finaliza la sesión del usuario antes de redirigir el navegador de vuelta al tenant de Auth0 de Travel0.

      <Callout icon="file-lines" color="#0EA5E9" iconType="regular">
        **Práctica recomendada**

        Define las aplicaciones de forma independiente en Auth0 para que sea más fácil configurar la URL `returnTo`, donde el `client_id` correspondiente se usa para validar correctamente la URL proporcionada. Para obtener más información, consulta [Aprovisionamiento](/docs/es/get-started/architecture-scenarios/multiple-organization-architecture/single-identity-provider-organizations/provisioning).
      </Callout>

Los pasos 3 y 4 coincidirán con los descritos en el escenario de [conexión de base de datos](#database-connection), pero usarán a Amintha en lugar de Jennifer y a MetaHexa Bank (`metahexa.corp.travel0.net`) en lugar de Hoekstra &amp; Associates.

<div id="social-connection">
  ## Social Connection
</div>

En el contexto de Social Connections, el cierre de sesión sigue un patrón similar al de una [conexión empresarial](#enterprise-connection), pero el IdP de origen está asociado con el proveedor social, no con una organización específica. Es casi seguro que nunca querrás usar el cierre de sesión federado con un proveedor social, porque la experiencia del usuario resultante sería demasiado disruptiva.
