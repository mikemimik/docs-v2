---
description: Conozca las Organizaciones con múltiples proveedores de identidad (IdP), en las que cada Auth0 Organization tiene más de una Conexión de Auth0.
title: Organizaciones con múltiples proveedores de identidad
---

En el escenario de Organizaciones con múltiples <Tooltip tip="proveedor de identidad (IdP): Servicio que almacena y gestiona identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=Identity+Provider">proveedor de identidad</Tooltip> (IdP), cada organización que utiliza la funcionalidad [Auth0 Organizations](/docs/es/manage-users/organizations/organizations-overview) usa más de una [Conexión de Auth0](/docs/es/authenticate/identity-providers). En muchos sentidos, la implementación de este escenario es idéntica a la implementación de [Organizaciones con proveedor de identidad único. I](/docs/es/get-started/architecture-scenarios/multiple-organization-architecture/single-identity-provider-organizations)ncluso, la funcionalidad Auth0 Organizations permite combinarlas, así como pasar fácilmente de una a otra. Sin embargo, debe tener en cuenta algunos aspectos adicionales.

<div id="connections">
  ## Conexiones
</div>

Además de la información que normalmente se incluye como parte de [Authorization](/docs/es/get-started/architecture-scenarios/multiple-organization-architecture/single-identity-provider-organizations/authorization), puedes agregar detalles sobre la [Conexión de Auth0](/docs/es/authenticate/identity-providers) específica a través de la cual se realizó la autenticación con el primer factor del usuario. Aunque no es un requisito explícito, puedes agregar esa información a un <Tooltip tip="token de ID: Credencial destinada al propio cliente, en lugar de para acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=ID+Token">token de ID</Tooltip> y/o a un <Tooltip tip="token de acceso: Credencial de autorización, en forma de una cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=Access+Token">token de acceso</Tooltip> como una claim personalizada, para proporcionar a tu aplicación y/o API contexto adicional que pueda usarse con el control de acceso.

Como se muestra en el siguiente ejemplo, las claims personalizadas normalmente se agregan mediante la extensibilidad de Rule, que puede acceder a los detalles de la conexión mediante el uso del objeto [`context`](/docs/es/customize/rules/context-object):

```js lines
context.idToken['http://travel0.net/connection'] = context.connection;     // Claim del nombre de conexión
context.accessToken['http://travel0.net/connection'] = context.connection; // Claim del nombre de conexión
```

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  El objeto `context` de Rules proporciona acceso a varios aspectos de la conexión utilizada para autenticar a un usuario con el primer factor, incluidos el nombre de `connection`, `connectionID` y `connectionStrategy`. Esto también proporciona a la extensibilidad de Rule información adicional que puede utilizarse como parte de cualquier [aplicación centralizada de políticas](/docs/es/get-started/architecture-scenarios/multiple-organization-architecture/single-identity-provider-organizations/authorization).
</Callout>

<div id="database-connection">
  ### Conexión de base de datos
</div>

Aunque Auth0 le permite [aprovisionar](/docs/es/get-started/architecture-scenarios/multiple-organization-architecture/single-identity-provider-organizations/provisioning) varias [Database Connections y Custom Database Connections](/docs/es/authenticate/database-connections), no recomendamos asociar más de uno de estos tipos de conexión a una Auth0 Organization. Como Database Connections y Custom Database Connections no admiten actualmente Home Realm Discovery, que es un componente clave de la [autenticación con identificador primero](/docs/es/authenticate/login/auth0-universal-login/identifier-first), la nueva experiencia de <Tooltip tip="Universal Login: Su aplicación redirige a Universal Login, alojado en el servidor de autorización de Auth0, para verificar la identidad de un usuario." cta="Ver glosario" href="/docs/es/glossary?term=Universal+Login">Universal Login</Tooltip> no puede elegir entre ellas si hay más de una definida.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  **Práctica recomendada**

  Si está definiendo una custom database connection para usarla con una Auth0 Organization, asegúrese de que una, y solo una, custom database connection esté asociada a esa Auth0 Organization en un momento dado.
</Callout>

<div id="enterprise-connection">
  ### Conexión empresarial
</div>

Si tu aplicación empresarial usa [federación empresarial](/docs/es/authenticate/enterprise-connections), puedes activar [autenticación con identificador primero with Home Realm Discovery](/docs/es/authenticate/login/auth0-universal-login/identifier-first) en su perfil de autenticación. Una vez habilitado, Home Realm Discovery detecta direcciones de correo electrónico de un dominio conocido y las envía automáticamente al inicio de sesión de Workforce correspondiente.

Para obtener más información, consulta [autenticación con identificador primero](/docs/es/manage-users/organizations/login-flows-for-organizations#identifier-first-authentication) en [Login Flows for Organizations](/docs/es/manage-users/organizations/login-flows-for-organizations).

<div id="social-connection">
  ### Conexión social
</div>

Aparte de las consideraciones descritas en la sección [Autenticación](/docs/es/get-started/architecture-scenarios/multiple-organization-architecture/single-identity-provider-organizations/authentication), no hay que tener en cuenta ninguna consideración adicional cuando se utilizan conexiones sociales como parte de un escenario de organización con múltiples IdP.
