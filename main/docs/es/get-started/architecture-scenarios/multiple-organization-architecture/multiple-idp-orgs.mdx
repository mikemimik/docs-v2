description: Obtenga información sobre las Organizaciones con múltiples proveedores de identidad (IdP), en las que cada organización de Auth0 tiene más de una Conexión de Auth0.
title: Organizaciones con múltiples proveedores de identidad
---

En el escenario de Organizaciones con múltiples <Tooltip tip="Proveedor de identidad (IdP): Servicio que almacena y gestiona identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=Identity+Provider">Proveedor de identidad</Tooltip> (IdP), cada organización que usa la funcionalidad [Auth0 Organizations](/docs/es/manage-users/organizations/organizations-overview) utiliza más de una [Conexión de Auth0](/docs/es/authenticate/identity-providers). En muchos aspectos, la implementación de este escenario es idéntica a la implementación de [Organizaciones con proveedor de identidad único. I](/docs/es/get-started/architecture-scenarios/multiple-organization-architecture/single-identity-provider-organizations)n De hecho, la funcionalidad Auth0 Organizations permite combinar ambas opciones, así como pasar fácilmente de una a otra. Sin embargo, debe tener en cuenta algunos aspectos adicionales.

<div id="connections">
  ## Conexiones
</div>

Además de la información que normalmente se incluye como parte de [Authorization](/docs/es/get-started/architecture-scenarios/multiple-organization-architecture/single-identity-provider-organizations/authorization), puedes agregar detalles sobre la [Conexión de Auth0](/docs/es/authenticate/identity-providers) específica mediante la cual se llevó a cabo la autenticación de primer factor del usuario. Aunque no es un requisito explícito, puedes agregar esa información a un <Tooltip tip="token de ID: credencial destinada al propio cliente, en lugar de para acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=ID+Token">token de ID</Tooltip> y/o <Tooltip tip="Token de acceso: credencial de Authorization, en forma de cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=Access+Token">token de acceso</Tooltip> como un claim personalizado, para proporcionar a tu aplicación y/o API contexto adicional que pueda usarse en el control de acceso.

Como se muestra en el siguiente ejemplo, los claims personalizados normalmente se agregan mediante la extensibilidad de Rule, que puede acceder a detalles específicos de la conexión mediante el uso del objeto [`context`](/docs/es/customize/rules/context-object):

```js lines
context.idToken['http://travel0.net/connection'] = context.connection;     // Claim del nombre de conexión
context.accessToken['http://travel0.net/connection'] = context.connection; // Claim del nombre de conexión
```

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  El objeto `context` de Rules proporciona acceso a varios aspectos de la conexión utilizada para autenticar a un usuario con el primer factor, incluidos el nombre de `connection`, `connectionID` y `connectionStrategy`. Además, esto proporciona a la extensibilidad de Rule información adicional que puede utilizarse como parte de cualquier [aplicación centralizada de políticas](/docs/es/get-started/architecture-scenarios/multiple-organization-architecture/single-identity-provider-organizations/authorization).
</Callout>

<div id="database-connection">
  ### Conexión de base de datos
</div>

Aunque Auth0 le permite [aprovisionar](/docs/es/get-started/architecture-scenarios/multiple-organization-architecture/single-identity-provider-organizations/provisioning) varias [Database Connections y Custom Database Connections](/docs/es/authenticate/database-connections), no recomendamos asociar más de una conexión de estos tipos a una organización de Auth0. Como Database Connections y Custom Database Connections no admiten actualmente Home Realm Discovery, que es un componente clave de la [autenticación con identificador primero](/docs/es/authenticate/login/auth0-universal-login/identifier-first), la Nueva experiencia de <Tooltip tip="Universal Login: Su aplicación redirige a Universal Login, alojado en el servidor de autorización de Auth0, para verificar la identidad de un usuario." cta="Ver glosario" href="/docs/es/glossary?term=Universal+Login">Universal Login</Tooltip> no puede elegir entre ellas si hay más de una definida.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  **Práctica recomendada**

  Si está definiendo una conexión de base de datos personalizada para usarla con una organización de Auth0, asegúrese de que una, y solo una, conexión de base de datos personalizada esté asociada a esa organización de Auth0 en todo momento.
</Callout>

<div id="enterprise-connection">
  ### Conexión empresarial
</div>

Si tu aplicación empresarial usa [Federación empresarial](/docs/es/authenticate/enterprise-connections), puedes activar [Autenticación con identificador primero con Home Realm Discovery](/docs/es/authenticate/login/auth0-universal-login/identifier-first) en su Perfil de autenticación. Una vez habilitado, Home Realm Discovery detecta direcciones de correo electrónico de un dominio conocido y redirige automáticamente a los usuarios al inicio de sesión de Workforce correspondiente.

Para obtener más información, consulta [Autenticación con identificador primero](/docs/es/manage-users/organizations/login-flows-for-organizations#identifier-first-authentication) en [Flujos de inicio de sesión para Organizations](/docs/es/manage-users/organizations/login-flows-for-organizations).

<div id="social-connection">
  ### Conexión social
</div>

Aparte de las consideraciones descritas en la sección de [Autenticación](/docs/es/get-started/architecture-scenarios/multiple-organization-architecture/single-identity-provider-organizations/authentication), no es necesario tener en cuenta consideraciones adicionales cuando se utilizan Conexiones sociales como parte de un escenario de organización con múltiples IdP.
