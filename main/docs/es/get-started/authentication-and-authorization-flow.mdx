title: Flujos de autenticación y autorización
---

Auth0 utiliza el [protocolo OpenID Connect (OIDC)](/docs/es/authenticate/protocols/openid-connect-protocol) y el [marco de autorización OAuth 2.0](/docs/es/authenticate/protocols/oauth) para autenticar a los usuarios y obtener su autorización para acceder a recursos protegidos. Con Auth0, puede admitir fácilmente distintos flujos en sus propias aplicaciones y API sin tener que preocuparse por las especificaciones de OIDC/<Tooltip tip="OAuth 2.0: Marco de autorización que define los protocolos de autorización y los flujos de trabajo." cta="Ver glosario" href="/docs/es/glossary?term=OAuth+2.0">OAuth 2.0</Tooltip> ni por otros aspectos técnicos de la [autenticación y autorización](/docs/es/get-started/identity-fundamentals/authentication-and-authorization).

Admitimos escenarios para aplicaciones del lado del servidor, móviles, de escritorio, del lado del cliente, de máquina a máquina y para dispositivos.

Si no está seguro de qué flujo usar, podemos ayudarle a decidir. Para obtener más información, lea [¿Qué flujo de OAuth 2.0 debo usar?](/docs/es/get-started/authentication-and-authorization-flow/which-oauth-2-0-flow-should-i-use).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Durante la ejecución de varios flujos, su aplicación también debe autenticarse ante el servidor de autorización. Para obtener más información sobre la autenticación de aplicaciones, lea [Application Credentials](/docs/es/secure/application-credentials).
</Callout>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Según las políticas de acceso de aplicaciones a la API que configure para una API, es posible que deba crear las autorizaciones de cliente correspondientes para su aplicación. Para obtener más información, lea [Acceso de aplicaciones a las API: políticas y autorizaciones de cliente](/docs/es/get-started/applications/application-access-to-apis-client-grants).
</Callout>

<div id="authorization-code-flow">
  ## Flujo de código de autorización
</div>

Como las aplicaciones web tradicionales se ejecutan del lado del servidor y su código fuente no está expuesto públicamente, pueden usar el flujo de código de autorización, que intercambia un código de autorización por un token.

* [Flujo de código de autorización](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow)
* [Agregar inicio de sesión con el flujo de código de autorización](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow/add-login-auth-code-flow)
* [Llamar a la API con el flujo de código de autorización](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow/call-your-api-using-the-authorization-code-flow)

<div id="authorization-code-flow-with-proof-key-for-code-exchange-pkce">
  ## Flujo de código de autorización con Proof Key for Code Exchange (PKCE)
</div>

Durante la autenticación, las aplicaciones móviles y nativas pueden usar el flujo de código de autorización, pero requieren seguridad adicional. Además, las aplicaciones web de página única presentan retos particulares. Para mitigarlos, OAuth 2.0 ofrece una versión del flujo de código de autorización que usa Proof Key for Code Exchange (PKCE).

* [Flujo de código de autorización con Proof Key for Code Exchange (PKCE)](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce)
* [Agregar el inicio de sesión con el flujo de código de autorización con PKCE](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce/add-login-using-the-authorization-code-flow-with-pkce)
* [Llamar a la API con el flujo de código de autorización con PKCE](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce/call-your-api-using-the-authorization-code-flow-with-pkce)

<div id="authorization-code-flow-with-enhanced-privacy-protection">
  ## Flujo de código de autorización con mayor protección de la privacidad
</div>

Durante el proceso de autenticación y autorización, algunos casos de uso, como la [autorización transaccional](/docs/es/secure/highly-regulated-identity/transactional-authorization-with-authorization-code-flow), intercambian información contextual que puede contener datos sensibles. Para proteger los datos y la información sensible, puedes utilizar distintas mejoras del protocolo para el flujo de código de autorización:

* [Flujo de código de autorización con Rich Authorization Requests (RAR)](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow/authorization-code-flow-with-rar)
* [Flujo de código de autorización con solicitud de autorización push (PAR)](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow/authorization-code-flow-with-par)
* [Flujo de código de autorización con solicitudes de autorización protegidas con JWT (JAR)](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow/authorization-code-flow-with-jar)
* [JSON Web Encryption (JWE)](/docs/es/secure/tokens/access-tokens/json-web-encryption)

<div id="implicit-flow-with-form-post">
  ## Flujo implícito con Form Post
</div>

Como alternativa al flujo de código de autorización, OAuth 2.0 ofrece el flujo implícito, pensado para <Tooltip tip="Cliente público: Cliente (aplicación) que no puede mantener credenciales de forma segura. Algunos ejemplos son una aplicación nativa de escritorio o móvil y una aplicación web del lado del cliente basada en JavaScript (como una aplicación web de página única (SPA))." cta="Ver glosario" href="/docs/es/glossary?term=Public+Clients">clientes públicos</Tooltip> o aplicaciones que no pueden almacenar de forma segura <Tooltip tip="Cliente público: Cliente (aplicación) que no puede mantener credenciales de forma segura. Algunos ejemplos son una aplicación nativa de escritorio o móvil y una aplicación web del lado del cliente basada en JavaScript (como una aplicación web de página única (SPA))." cta="Ver glosario" href="/docs/es/glossary?term=Client+Secrets">secretos de cliente</Tooltip>. Aunque ya no se considera una práctica recomendada para solicitar <Tooltip tip="Token de acceso: Credencial de autorización, en forma de una cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=Access+Tokens">tokens de acceso</Tooltip>, cuando se usa con el modo de respuesta Form Post sí ofrece un flujo de trabajo más ágil si la aplicación solo necesita un <Tooltip tip="Token de ID: Credencial destinada al propio cliente, en lugar de al acceso a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=ID+token">token de identidad</Tooltip> para realizar la autenticación de usuario.

* [Flujo implícito con Form Post](/docs/es/get-started/authentication-and-authorization-flow/implicit-flow-with-form-post)
* [Agregar el inicio de sesión con el flujo implícito con Form Post](/docs/es/get-started/authentication-and-authorization-flow/implicit-flow-with-form-post/add-login-using-the-implicit-flow-with-form-post)
* [Autenticar SPA con cookies](/docs/es/manage-users/cookies/spa-authenticate-with-cookies)

<div id="hybrid-flow">
  ## Flujo híbrido
</div>

Las aplicaciones que pueden almacenar secretos de cliente de forma segura pueden beneficiarse del uso del flujo híbrido, que combina funcionalidades del flujo de código de autorización y del flujo implícito con Form Post para permitir que tu aplicación tenga acceso inmediato a un token de identidad y, al mismo tiempo, pueda obtener de forma segura tokens de acceso y <Tooltip tip="Token de actualización: token usado para obtener un token de acceso renovado sin obligar a los usuarios a iniciar sesión de nuevo." cta="Ver glosario" href="/docs/es/glossary?term=refresh+tokens">tokens de actualización</Tooltip>. Esto puede resultar útil cuando tu aplicación necesita acceder de inmediato a información sobre el usuario, pero debe realizar cierto procesamiento antes de acceder a recursos protegidos durante un período prolongado.

* [Flujo híbrido](/docs/es/get-started/authentication-and-authorization-flow/hybrid-flow)
* [Llamar a una API con el flujo híbrido](/docs/es/get-started/authentication-and-authorization-flow/hybrid-flow/call-api-hybrid-flow)

<div id="client-credentials-flow">
  ## flujo de credenciales de cliente
</div>

Con las aplicaciones de máquina a máquina (M2M), como CLIs, demonios o servicios que se ejecutan en su back-end, el sistema autentica y autoriza la aplicación en lugar de a un usuario. En este caso, los métodos de autenticación habituales, como identificador + contraseña o los inicios de sesión con redes sociales, no tienen sentido. En su lugar, las aplicaciones M2M usan el flujo de credenciales de cliente (definido en OAuth 2.0 RFC 6749, sección 4.4).

* [flujo de credenciales de cliente](/docs/es/get-started/authentication-and-authorization-flow/client-credentials-flow)
* [Llamar a la API con el flujo de credenciales de cliente](/docs/es/get-started/authentication-and-authorization-flow/client-credentials-flow/call-your-api-using-the-client-credentials-flow)

<div id="device-authorization-flow">
  ## Flujo de autorización de dispositivos
</div>

En los dispositivos con limitación de entrada que se conectan a internet, en lugar de autenticar al usuario directamente, el dispositivo le pide que abra un enlace en su computadora o smartphone y autorice el dispositivo. Así se evita una mala experiencia de usuario en dispositivos que no cuentan con una forma sencilla de introducir texto. Para ello, las aplicaciones del dispositivo usan el <Tooltip tip="Flujo de autorización: concesión de autorización (o flujo de trabajo) especificada en el marco de OAuth 2.0." cta="Ver glosario" href="/docs/es/glossary?term=Authorization+Flow">Flujo de autorización</Tooltip> de dispositivos (definido en OAuth 2.0). Para su uso con aplicaciones móviles/nativas.

* [Flujo de autorización de dispositivos](/docs/es/get-started/authentication-and-authorization-flow/device-authorization-flow)
* [Llamar a la API usando el Flujo de autorización de dispositivos](/docs/es/get-started/authentication-and-authorization-flow/device-authorization-flow/call-your-api-using-the-device-authorization-flow)

<div id="resource-owner-password-flow">
  ## Flujo de contraseña del propietario del recurso
</div>

Aunque no lo recomendamos, las aplicaciones de alta confianza pueden usar el flujo de contraseña de <Tooltip tip="Propietario del recurso: entidad (como un usuario o una aplicación) capaz de otorgar acceso a un recurso protegido." cta="Ver glosario" href="/docs/es/glossary?term=Resource+Owner">Resource Owner</Tooltip>, que pide a los usuarios que proporcionen sus credenciales (identificador y contraseña), normalmente mediante un formulario interactivo. El flujo de contraseña del propietario del recurso solo debe utilizarse cuando no sea posible usar flujos basados en redirección (como el [flujo de código de autorización](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow)).

* [Flujo de contraseña del propietario del recurso](/docs/es/get-started/authentication-and-authorization-flow/resource-owner-password-flow)
* [Llamar a la API con el flujo de contraseña del propietario del recurso](/docs/es/get-started/authentication-and-authorization-flow/resource-owner-password-flow/call-your-api-using-resource-owner-password-flow)

<div id="client-initiated-backchannel-authentication-flow">
  ## Client-Initiated Backchannel Authentication Flow
</div>

Con Client-Initiated Backchannel Authentication Flow (CIBA), en lugar de autenticar directamente al usuario, el backend de la aplicación cliente inicia un flujo de autenticación para solicitar su autenticación. La autenticación en sí se realiza en un dispositivo de autenticación independiente, normalmente un smartphone con una aplicación personalizada.

* [Client-Initiated Backchannel Authentication Flow](/docs/es/get-started/authentication-and-authorization-flow/client-initiated-backchannel-authentication-flow)
* [Autenticación de usuario con CIBA](/docs/es/get-started/authentication-and-authorization-flow/client-initiated-backchannel-authentication-flow/user-authentication-with-ciba)
* [Autorización de usuario con CIBA](/docs/es/get-started/authentication-and-authorization-flow/client-initiated-backchannel-authentication-flow/user-authorization-with-ciba)

<div id="custom-token-exchange">
  ## Custom Token Exchange
</div>

Custom Token Exchange (CTE) permite a las aplicaciones intercambiar tokens de identidad ya existentes por tokens de Auth0 invocando el endpoint `/oauth/token`, tal como se define en [RFC 8693](https://datatracker.ietf.org/doc/html/rfc8693). Por ejemplo, puedes usar Custom Token Exchange para intercambiar tokens de Auth0 y obtener acceso a otra audience en nombre del usuario. Para obtener más información sobre los casos de uso de Custom Token Exchange, consulta [Ejemplos de casos de uso](/docs/es/authenticate/custom-token-exchange/cte-example-use-cases).

Al asociar un [Custom Token Exchange Profile](/docs/es/authenticate/custom-token-exchange/configure-custom-token-exchange) con una Action que contiene lógica personalizada, puedes implementar flujos de trabajo de identidad altamente personalizados mediante el intercambio de un token de seguridad por otro, manteniendo al mismo tiempo el control sobre la lógica de autorización.
