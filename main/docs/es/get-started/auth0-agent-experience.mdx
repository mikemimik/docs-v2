El [Agent Experience Score](https://auth0.com/agent-experience) mide qué tan bien los agentes de IA implementan Auth0 en distintos modelos y frameworks. Te permite comparar las puntuaciones actuales de los agentes que implementan servicios y funciones de Auth0, como la autenticación multifactor (MFA) o Auth0 Actions, en entornos de desarrollo y prueba para analizar cómo las herramientas de Auth0 mejoran el rendimiento de los agentes.

Usa este recurso para conocer la metodología de puntuación, incluido cómo se calculan las puntuaciones, qué dimensiones se miden y cómo se asignan las calificaciones.

<div id="test-specifications">
  ## Especificaciones de prueba
</div>

Los agentes de IA — Claude Code, GitHub Copilot, Gemini CLI — ejecutan tareas de integración de Auth0 en entornos de desarrollo aislados. Cada agente usa las mismas herramientas que usaría un desarrollador en un entorno realista: un espacio de trabajo, una terminal y herramientas de archivos como Auth0 CLI. Los prompts son breves y realistas: &quot;añade autenticación a mi aplicación Next.js&quot;, no guías paso a paso.

Cada modelo se prueba con y sin las herramientas de Auth0 ([MCP Server](/docs/es/get-started/build-with-ai-tools#auth0-docs-mcp-server) y [Agent Skills](/docs/es/quickstart/agent-skills)). La diferencia entre esas puntuaciones es el impacto medible de las herramientas de IA de Auth0 en la experiencia del desarrollador.

<div id="score-dimensions">
  ## Dimensiones de la puntuación
</div>

Cada ejecución se puntúa en 7 dimensiones, divididas en dos categorías. Cuatro dimensiones evalúan de extremo a extremo el proceso del agente con las herramientas de Auth0. Tres dimensiones puntúan el resultado final. Cada dimensión se califica individualmente de 0 a 100 y luego se pondera y combina en la puntuación general.

| Dimensión                      | Categoría | Descripción                                                                                                                                                                            |
| ------------------------------ | --------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Fricción de configuración**  | Proceso   | Puntuación determinada por la capacidad del agente para completar la tarea de forma autónoma. Si el agente se detuvo para hacer preguntas o encontró errores, la puntuación disminuyó. |
| **Velocidad de configuración** | Proceso   | Puntuación determinada por el tiempo de ejecución activo del agente. Los resultados son comparables entre entornos.                                                                    |
| **Eficiencia**                 | Proceso   | Puntuación determinada por la cantidad de llamadas a herramientas necesarias para completar la tarea. Menos llamadas a herramientas implican menos costo y menos complejidad.          |
| **Recuperación ante errores**  | Proceso   | Puntuación determinada por errores de infraestructura (límites de frecuencia, tiempos de espera) que interrumpieron la ejecución.                                                      |
| **Corrección**                 | Resultado | Puntuación determinada por si el código generado importa paquetes reales, llama a métodos reales y conecta correctamente los componentes.                                              |
| **Alucinación**                | Resultado | Puntuación determinada por si el agente inventó paquetes que no existen o usó variantes incorrectas del SDK.                                                                           |
| **Seguridad**                  | Resultado | Puntuación determinada por si el agente dejó secretos codificados de forma fija, almacenó tokens de forma insegura o confirmó credenciales en el código fuente.                        |

<div id="grades">
  ## Calificaciones
</div>

Las puntuaciones generales se corresponden con calificaciones con letras:

| Calificación | Puntuación mínima | Descripción                                   |
| ------------ | ----------------- | --------------------------------------------- |
| A            | 90                | Lista para producción. Problemas mínimos.     |
| B            | 75                | Sólida, pero con algunos puntos por corregir. |
| C            | 60                | Utilizable, pero necesita pulirse.            |
| D            | 40                | Problemas importantes.                        |
| F            | &lt; 40           | No es útil — es más rápido empezar de cero.   |

Las calificaciones están calibradas para coincidir con la intuición de los desarrolladores. Una puntuación de 91 debería sentirse como código que aceptarías con una revisión mínima. Una puntuación de 55 debería sentirse como algo que requiere un trabajo considerable para corregirse.

<div id="result-validation">
  ## Validación de resultados
</div>

Cada evaluador verifica el código generado, no la prosa ni las explicaciones. Los evaluadores comprueban que el código compile, importe paquetes reales, llame a métodos reales del SDK y no introduzca vulnerabilidades de seguridad.

Los resultados se validan en varios niveles:

* **Comprobaciones de presencia**: Los símbolos del SDK, las importaciones y las claves de configuración requeridos están presentes en el resultado.
* **Detección de alucinaciones**: Se detectan paquetes inventados, variantes incorrectas del SDK y métodos de API inventados.
* **Comprobaciones de seguridad**: Se señalan credenciales incrustadas en el código, tokens en almacenamiento inseguro y secretos en el código fuente.
* **Validación estructural**: El código está correctamente organizado: los componentes adecuados en los archivos correctos, los hooks del ciclo de vida bien gestionados y el middleware en el orden correcto.
* **Corrección de versión**: El agente usa API actuales, no patrones obsoletos (solo se comprueba cuando el agente tiene acceso a la documentación actual).
* **Revisión integral**: Un juez LLM evalúa la corrección general de la implementación.

<div id="estimated-cost-and-time">
  ## Costo y tiempo estimados
</div>

La página de resultados muestra el costo estimado y el tiempo estimado de cada configuración. Estos valores corresponden a una sola ejecución de la evaluación con Auth0 MCP + Skills habilitados.

<div id="estimated-cost">
  ### Costo estimado
</div>

El costo se calcula a partir del total de tokens consumidos durante la ejecución de la evaluación (tokens de entrada + tokens de salida), multiplicado por el precio por token publicado por el proveedor del modelo. Auth0 no cobra por ejecutar evaluaciones; el costo refleja lo que pagarías a tu proveedor de modelos por un uso equivalente de tokens.

El precio de los tokens varía según el modelo y el proveedor. Para consultar las tarifas actuales, visita la página de precios de tu proveedor:

* [Precios de Anthropic (Claude)](https://docs.anthropic.com/en/docs/about-claude/models#model-comparison-table)
* [Precios de OpenAI (GPT)](https://openai.com/api/pricing/)
* [Precios de Google (Gemini)](https://ai.google.dev/gemini-api/docs/pricing/)

<div id="estimated-time">
  ### Tiempo estimado
</div>

El tiempo es la duración real de la ejecución de evaluación, desde el envío del prompt hasta el resultado final. Incluye toda la actividad del agente: leer archivos, hacer llamadas a herramientas, esperar respuestas de la API y escribir código.

El tiempo puede variar en función de lo siguiente:

* La latencia de la API del proveedor del modelo y los límites de frecuencia
* La cantidad de llamadas a herramientas necesarias (varía según la complejidad de la tarea)
* Las condiciones de red entre el entorno de evaluación y el proveedor del modelo
* La profundidad de la cola y la carga del proveedor

El tiempo no está normalizado entre proveedores. Un tiempo menor refleja tanto la eficiencia del modelo como el rendimiento de la infraestructura del proveedor.
