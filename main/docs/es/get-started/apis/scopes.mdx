Distintos tipos de información de los usuarios suelen almacenarse en varios recursos en línea. Los usuarios pueden subir y almacenar fotos en un servicio como Flickr, guardar archivos digitales en Dropbox y almacenar contactos y eventos en Google Calendar o en Facebook.

A menudo, las nuevas aplicaciones quieren utilizar información que ya se ha creado en un recurso en línea. Para ello, la aplicación debe solicitar autorización para acceder a esta información en nombre del usuario. Los scopes definen las acciones específicas que se puede permitir que las aplicaciones realicen en nombre del usuario.

<div id="ways-to-use-scopes">
  ## Formas de usar scopes
</div>

Cuando una app solicita permiso para acceder a un recurso a través de un <Tooltip tip="servidor de autorización: Servidor centralizado que contribuye a definir los límites del acceso de un usuario. Por ejemplo, tu servidor de autorización puede controlar los datos, las tareas y las funciones disponibles para un usuario." cta="Ver glosario" href="/docs/es/glossary?term=authorization+server">servidor de autorización</Tooltip>, usa el parámetro `scope` para especificar qué nivel de acceso necesita, y el servidor de autorización usa el parámetro `scope` para responder con el acceso que se concedió realmente (si el acceso concedido fue distinto del que se solicitó).

En general, usas scopes de tres maneras:

* Desde una aplicación, para verificar la identidad de un usuario y obtener información básica de su perfil, como su correo electrónico o foto. En este escenario, los scopes disponibles incluyen los implementados por el protocolo <Tooltip tip="OpenID: Estándar abierto para autenticación que permite a las aplicaciones verificar la identidad de los usuarios sin recopilar ni almacenar información de inicio de sesión." cta="Ver glosario" href="/docs/es/glossary?term=OpenID">OpenID</Tooltip> Connect (OIDC). Para obtener más información, consulta [Scopes de OpenID Connect](/docs/es/get-started/apis/scopes/openid-connect-scopes).
* En una [API](/docs/es/api), para implementar el control de acceso. En este caso, debes definir scopes personalizados para tu API y luego identificarlos para que las aplicaciones que la llaman puedan usarlos. Para obtener más información, consulta [Scopes de API](/docs/es/get-started/apis/scopes/api-scopes).
* Desde una aplicación, para llamar a una API que ha implementado sus propios scopes personalizados. En este caso, necesitas saber qué scopes personalizados están definidos para la API a la que llamas. Para ver ejemplos de cómo llamar a una API personalizada desde una aplicación, consulta [Casos de uso de ejemplo: Scopes y Claims](/docs/es/get-started/apis/scopes/sample-use-cases-scopes-and-claims)

<div id="best-practices">
  ## Prácticas recomendadas
</div>

Comprende tu caso de uso y elige los scopes más restrictivos posibles.

Si estás solicitando scopes, asegúrate de pedir el acceso suficiente para que tu aplicación funcione, pero solicita solo lo que realmente necesites. ¿Estás verificando la identidad de un usuario o pidiéndole permiso para interactuar con sus datos? Hay una gran diferencia entre importar la información del perfil de Facebook de un usuario y publicar en su muro. Si solicitas solo lo necesario, será más probable que obtengas el consentimiento del usuario cuando haga falta, ya que los usuarios tienden más a conceder acceso para scopes limitados y claramente definidos.

Del mismo modo, al crear scopes personalizados para una API, considera qué niveles de acceso granular pueden necesitar las aplicaciones y diseña en consecuencia.

<div id="requested-scopes-versus-granted-scopes">
  ## Scopes solicitados frente a scopes concedidos
</div>

En algunos casos, los usuarios pueden dar su consentimiento para el acceso solicitado. Aunque normalmente los scopes devueltos serán idénticos a los solicitados, los usuarios pueden modificar los scopes concedidos (tanto durante el consentimiento inicial como, a veces, después, según el recurso), lo que permite otorgar a una app menos acceso del que pidió.

Como desarrollador de aplicaciones, debes tener en cuenta esta posibilidad y contemplar estos casos en tu app. Por ejemplo, tu app podría advertir al usuario de que tendrá una funcionalidad reducida. También podría hacer que el usuario vuelva a pasar por el <Tooltip tip="Flujo de autorización: concesión de autorización (o workflow) especificada en el marco de OAuth 2.0." cta="Ver glosario" href="/docs/es/glossary?term=authorization+flow">flujo de autorización</Tooltip> para solicitar permisos adicionales. Pero, una vez más, recuerda que, cuando se les pide consentimiento, los usuarios siempre pueden negarse.

De forma predeterminada, Auth0 omite el consentimiento del usuario para las aplicaciones propias, es decir, las aplicaciones registradas en el mismo dominio de Auth0 que la API a la que llaman; sin embargo, puedes configurar tu API en Auth0 para exigir el consentimiento del usuario a las aplicaciones propias. Las aplicaciones de terceros, es decir, las aplicaciones externas, requieren el consentimiento del usuario.
