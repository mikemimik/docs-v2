Distintos tipos de información de los usuarios suelen almacenarse en diversos recursos en línea. Los usuarios pueden cargar y almacenar fotos en un servicio como Flickr, guardar archivos digitales en Dropbox y almacenar contactos y eventos en Google Calendar o en Facebook.

A menudo, las nuevas aplicaciones quieren utilizar la información que ya se ha creado en un recurso en línea. Para ello, la aplicación debe solicitar autorización para acceder a esta información en nombre del usuario. Los scopes definen las acciones específicas que se puede permitir que las aplicaciones realicen en nombre del usuario.

<div id="ways-to-use-scopes">
  ## Formas de usar scopes
</div>

Cuando una aplicación solicita permiso para acceder a un recurso a través de un <Tooltip tip="Servidor de autorización: servidor centralizado que ayuda a definir los límites del acceso de un usuario. Por ejemplo, tu servidor de autorización puede controlar los datos, las tareas y las funcionalidades disponibles para un usuario." cta="Ver glosario" href="/docs/es/glossary?term=authorization+server">servidor de autorización</Tooltip>, usa el parámetro `scope` para especificar qué acceso necesita, y el servidor de autorización usa el parámetro `scope` para responder con el acceso que realmente se concedió (si el acceso concedido fue distinto del que se solicitó).

Por lo general, los scopes se usan de tres maneras:

* Desde una aplicación, para verificar la identidad de un usuario y obtener información básica de su perfil, como su correo electrónico o foto. En este escenario, los scopes disponibles incluyen los implementados por el protocolo <Tooltip tip="OpenID: estándar abierto de autenticación que permite a las aplicaciones verificar la identidad de los usuarios sin recopilar ni almacenar información de inicio de sesión." cta="Ver glosario" href="/docs/es/glossary?term=OpenID">OpenID</Tooltip> Connect (OIDC). Para obtener más información, consulta [Scopes de OpenID Connect](/docs/es/get-started/apis/scopes/openid-connect-scopes).
* En una [API](/docs/es/api), para implementar control de acceso. En este caso, debes definir scopes personalizados para tu API y luego identificar esos scopes para que las aplicaciones que la llaman puedan usarlos. Para obtener más información, consulta [Scopes de API](/docs/es/get-started/apis/scopes/api-scopes).
* Desde una aplicación, para llamar a una API que ha implementado sus propios scopes personalizados. En este caso, necesitas saber qué scopes personalizados están definidos para la API a la que llamas. Para ver ejemplos de llamadas a una API personalizada desde una aplicación, consulta [Casos prácticos de ejemplo: Scopes y Claims](/docs/es/get-started/apis/scopes/sample-use-cases-scopes-and-claims)

<div id="best-practices">
  ## Prácticas recomendadas
</div>

Comprenda su caso de uso y elija los scopes más restrictivos posibles.

Si va a solicitar scopes, asegúrese de pedir acceso suficiente para que su aplicación funcione, pero solicite solo lo estrictamente necesario. ¿Está estableciendo la identidad de un usuario o pidiéndole que le permita interactuar con sus datos? Hay una gran diferencia entre importar la información del perfil de Facebook de un usuario y publicar en su muro. Si solicita solo lo que necesita, será más probable que obtenga el consentimiento del usuario cuando sea necesario, ya que los usuarios suelen conceder acceso a scopes limitados y claramente especificados.

Del mismo modo, al crear scopes personalizados para una API, considere qué niveles de acceso granular pueden necesitar las aplicaciones y diseñe en consecuencia.

<div id="requested-scopes-versus-granted-scopes">
  ## Scopes solicitados frente a scopes concedidos
</div>

En algunos casos, los usuarios deben dar su consentimiento para el acceso que se solicita. Aunque por lo general los scopes devueltos serán idénticos a los solicitados, los usuarios pueden modificar los scopes concedidos (tanto durante el consentimiento inicial como, en ocasiones, después, según el recurso), con lo que otorgan a una aplicación menos acceso del que pidió.

Como desarrollador de aplicaciones, debes tener presente esta posibilidad y contemplar estos casos en tu aplicación. Por ejemplo, tu aplicación podría advertir al usuario de que verá una funcionalidad reducida. También podría hacer que el usuario volviera a pasar por el <Tooltip tip="Flujo de autorización: concesión de autorización (o flujo de trabajo) especificado en el framework OAuth 2.0." cta="Ver glosario" href="/docs/es/glossary?term=authorization+flow">flujo de autorización</Tooltip> para solicitar permisos adicionales. Pero, de nuevo, recuerda que, cuando se les pide su consentimiento, los usuarios siempre pueden negarse.

De forma predeterminada, Auth0 omite el consentimiento del usuario para las aplicaciones propias, es decir, las aplicaciones registradas en el mismo dominio de Auth0 que la API a la que llaman; sin embargo, puedes configurar tu API en Auth0 para exigir el consentimiento del usuario en las aplicaciones propias. Las aplicaciones de terceros, que son aplicaciones externas, requieren el consentimiento del usuario.
