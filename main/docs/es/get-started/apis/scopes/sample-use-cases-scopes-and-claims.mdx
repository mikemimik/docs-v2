En estos ejemplos, usamos el [flujo de código de autorización](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow) para autenticar a un usuario y solicitar los permisos (scopes) y tokens necesarios. Para obtener más información sobre los parámetros de la solicitud o para aprender a implementar por completo este flujo, lee nuestro tutorial: [Agregar inicio de sesión a aplicaciones web tradicionales](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow/add-login-auth-code-flow).

<div id="authenticate-a-user-and-request-standard-claims">
  ## Autenticar a un usuario y solicitar claims estándar
</div>

En este ejemplo, queremos autenticar a un usuario y obtener sus datos para poder personalizar nuestra interfaz de usuario. Para ello, necesitamos obtener un <Tooltip tip="ID Token: Credencial destinada al propio cliente, en lugar de para acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=ID+token">token de ID</Tooltip> que contenga el nombre, el apodo, la foto de perfil y la información de correo electrónico del usuario.

1. Inicie el flujo de autenticación enviando al usuario a la URL de autorización:

   <AuthCodeBlock children={codeExample1} language="text" />

   Tenga en cuenta que, en este ejemplo:

   * El parámetro `response_type` incluye un valor:

     * `code`: como estamos usando el flujo de aplicación web clásica, nuestra solicitud inicial es para un código de autorización; cuando solicitemos los tokens usando este código, recibiremos el token de ID que necesitamos para la autenticación.
   * El parámetro `scope` incluye tres valores: los scopes de OIDC solicitados:

     * `openid`: para indicar que la aplicación pretende usar OIDC para verificar la identidad del usuario.
     * `profile`: para obtener `name`, `nickname` y `picture`.
     * `email`: para obtener `email` y `email_verified`.
2. Después de que el usuario otorgue su consentimiento (si es necesario) y Auth0 redirija de vuelta a su aplicación, [solicite tokens](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow/add-login-auth-code-flow).
3. Extraiga el token de ID de la respuesta y [decodifíquelo](/docs/es/secure/tokens/id-tokens). Debería ver los siguientes claims:

   <AuthCodeBlock children={codeExample2} language="json" />

   Su aplicación ahora puede recuperar los atributos del usuario y usarlos para personalizar la interfaz de usuario.

<div id="request-custom-api-access">
  ## Solicitar acceso a una API personalizada
</div>

En este ejemplo, solicitamos un scope personalizado para una API de calendario que autorizará a la aplicación que realiza la llamada a leer las citas del usuario. Para ello, queremos obtener un <Tooltip tip="Token de acceso: Credencial de autorización, en forma de una cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=access+token">token de acceso</Tooltip> que contenga el scope adecuado para leer citas desde la API. Ten en cuenta que solicitar un token de acceso no depende de solicitar un token de ID.

Antes de usar una API personalizada, necesitas saber qué scopes están disponibles para la API que vas a llamar. Si la API personalizada está bajo tu control, debes registrar tanto tu aplicación como la API en Auth0 y [definir los scopes de tu API con Auth0 Dashboard](/docs/es/get-started/apis/scopes/api-scopes). También puedes usar los permisos definidos para [personalizar la solicitud de consentimiento](/docs/es/customize/login-pages/customize-consent-prompts) para tus usuarios.

1. Inicia el flujo de autorización enviando al usuario a la URL de autorización:

   <AuthCodeBlock children={codeExample3} language="text" />

   Ten en cuenta que en este ejemplo:

   * El parámetro `response_type` sigue incluyendo un valor:

     * `code`: como estamos usando el flujo de aplicación web clásica, nuestra solicitud inicial es de un código de autorización; cuando solicitemos nuestros tokens con este código, recibiremos el token de acceso que podremos usar para llamar a nuestra API.
   * El parámetro `scope` incluye un valor: el scope de API solicitado:

     * `read:appointments`: para permitirnos leer las citas del usuario desde la API.
   * El parámetro `audience` es nuevo e incluye un valor:

     * El identificador único de la API desde la que queremos leer las citas del usuario.
2. Como en el ejemplo anterior, después de que el usuario otorgue su consentimiento (si es necesario) y Auth0 redirija de nuevo a tu aplicación, [solicita tokens](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow/add-login-auth-code-flow).
3. Extrae el token de acceso de la respuesta y llama a la API usando el token de acceso como credencial.

<div id="authenticate-a-user-and-request-standard-claims-and-custom-api-access">
  ## Autentica a un usuario y solicita claims estándar y acceso a una API personalizada
</div>

En este ejemplo, combinamos los dos ejemplos anteriores para autenticar a un usuario, solicitar claims estándar y también un scope personalizado para una API de calendario que permitirá a la aplicación que hace la llamada leer las citas del usuario. Para ello, obtén dos tokens:

* Token de ID que contiene:

  * Nombre del usuario
  * Apodo
  * Foto de perfil
  * Información de correo electrónico
* Token de acceso que contiene el scope adecuado para leer citas desde la API. Ten en cuenta que solicitar un token de acceso no depende de solicitar un token de ID.

Antes de usar una API personalizada, debes saber qué scopes están disponibles para la API a la que llamas. Si la API personalizada está bajo tu control, debes registrar tanto tu aplicación como tu API en Auth0 y [definir los scopes de tu API con el Auth0 Dashboard](/docs/es/get-started/apis/scopes/api-scopes). También puedes usar permissions definidas para [personalizar la solicitud de consentimiento](/docs/es/customize/login-pages/customize-consent-prompts) para tus usuarios.

1. Inicia el flujo de autenticación enviando al usuario a la URL de autorización:

   <AuthCodeBlock children={codeExample4} language="text" />

   Ten en cuenta que en este ejemplo:

   * El parámetro `response_type` sigue incluyendo un valor:

     * `code`: como estamos usando el flujo de aplicación web clásica, nuestra solicitud inicial es para un código de autorización; cuando solicitemos nuestros tokens con este código, recibiremos tanto el token de ID que necesitamos para la autenticación como el token de acceso que podemos usar para llamar a nuestra API.
   * El parámetro `scope` se usa tanto para los scopes de OIDC como para los scopes de API, por lo que ahora incluye cuatro valores:

     * `openid`: para indicar que la aplicación pretende usar OIDC para verificar la identidad del usuario.
     * `profile`: para obtener `name`, `nickname` y `picture`.
     * `email`: para obtener `email` y `email_verified`.
     * `read:appointments`: para permitirnos leer las citas del usuario desde la API.
   * el parámetro `audience` incluye un valor:

     * El identificador único de la API de la que queremos leer las citas del usuario
2. Como en los ejemplos anteriores, después de que el usuario otorgue su consentimiento (si es necesario) y Auth0 redirija de vuelta a tu aplicación, solicita los tokens.
3. Extrae el token de ID de la respuesta, decodifícalo, recupera los atributos del usuario y úsalos para personalizar tu interfaz de usuario.
4. Extrae el token de acceso de la respuesta y llama a la API usando el token de acceso como credencial.

<div id="add-custom-claims-to-a-token">
  ## Agrega claims personalizadas a un token
</div>

En este ejemplo, agregamos el color favorito de un usuario y su método de contacto preferido al ID Token. Para hacerlo, creamos una [Action](/docs/es/customize/actions) para personalizar el ID Token agregando estas [claims](/docs/es/secure/tokens/json-web-tokens/create-custom-claims). Una vez agregadas, también podremos obtener las claims personalizadas al llamar al `/userinfo` endpoint (aunque la Action solo se ejecutará durante el proceso de autenticación).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Auth0 permite claims con y sin espacio de nombres, pero se aplican ciertas restricciones (consulta [Restricciones generales](/docs/es/secure/tokens/json-web-tokens/create-custom-claims#general-restrictions)). Para evitar conflictos de nombres, recomendamos usar claims con espacio de nombres. Si se produce un conflicto, la transacción no fallará, pero la claim personalizada no se añadirá a tus tokens.
</Callout>

Supongamos que:

* En algún momento, el usuario seleccionó `email` como método `preferred_contact` y `red` como `favorite_color`, y lo guardamos como parte de su `user_metadata`.
* Hemos usado la [Management API](https://auth0.com/docs/api/management/v2#!/Users/patch_users_by_id) o el Dashboard para establecer información específica de la aplicación para este usuario.

En este caso, el [perfil de usuario normalizado](/docs/es/manage-users/user-accounts/user-profiles/normalized-user-profiles) almacenado en Auth0 es:

```json lines
{
  "email": "jane@example.com",
  "email_verified": true,
  "user_id": "custom|123",
  "favorite_color": "blue",
  "user_metadata": {
    "preferred_contact": "email"
  }
}
```

Para este perfil, Auth0 normalmente devolvería a tu aplicación los siguientes claims del token de ID:

```json lines
{
  "email": "jane@example.com",
  "email_verified": true,
  "iss": "https://my-domain.auth0.com/",
  "sub": "custom|123",
  "aud": "my_client_id",
  "iat": 1311280970,
  "exp": 1311281970
}
```

Observa que en este ejemplo:

* El claim `sub` contiene el valor de la propiedad `user_id`.
* Ni la propiedad `favorite_color` ni la propiedad `user_metadata` están presentes porque <Tooltip tip="OpenID: Estándar abierto de autenticación que permite a las aplicaciones verificar la identidad de los usuarios sin recopilar ni almacenar información de inicio de sesión." cta="Ver glosario" href="/docs/es/glossary?term=OpenID">OpenID</Tooltip> Connect (OIDC) no define claims estándar que representen `favorite_color` o `user_metadata`.

Para recibir los datos personalizados, tendremos que [crear una nueva Action](/docs/es/customize/actions/write-your-first-action) para personalizar el token con [custom claims](/docs/es/secure/tokens/json-web-tokens/create-custom-claims) que representen estas propiedades del perfil del usuario.

1. Ve a [Auth0 Dashboard &gt; Actions &gt; Library](https://manage.auth0.com/#/actions/library) y selecciona **Build Custom**.

2. Introduce un **Name** descriptivo para tu Action (por ejemplo, `Add user metadata to tokens`), selecciona el trigger `Login / Post Login` porque vas a añadir la Action al flujo de Login y, a continuación, selecciona **Create**.

3. Busca el editor de código de Actions, copia en él el siguiente código JavaScript y selecciona **Save Draft** para guardar los cambios:

   ```javascript lines
   exports.onExecutePostLogin = async (event, api) => {
     const namespace = 'https://myapp.example.com';
     const { favorite_color, preferred_contact } = event.user.user_metadata;

     if (event.authorization) {
       // Set claims 
       api.idToken.setCustomClaim(`${namespace}/favorite_color`, favorite_color);
       api.idToken.setCustomClaim(`${namespace}/preferred_contact`, preferred_contact);
     }
   };
   ```

4. En la barra lateral del editor de código de Actions, selecciona Test (icono de reproducción) y luego selecciona **Run** para [probar tu código](/docs/es/customize/actions/test-actions).

5. Cuando la Action esté lista para ponerse en producción, selecciona **Deploy**.

Por último, añade la Action que creaste al [Login Flow](https://manage.auth0.com/#/actions/flows/login/). Para obtener información sobre cómo adjuntar Actions a los Flows, lee la sección &quot;Attach the Action to a flow&quot; de [Write Your First Action](/docs/es/customize/actions/write-your-first-action).

Con esta Action habilitada, Auth0 incluirá los custom claims `favorite_color` y `preferred_contac`t en el ID Token:

```json lines
{
  "email": "jane@example.com",
  "email_verified": true,
  "iss": "https://my-domain.auth0.com/",
  "sub": "custom|123",
  "aud": "my_client_id",
  "iat": 1311280970,
  "exp": 1311281970,
  "https://myapp.example.com/favorite_color": "red",
  "https://myapp.example.com/preferred_contact": "email"
}
```

Al crear tu Action, asegúrate de definir una lógica que determine cuándo incluir claims adicionales. Inyectar claims personalizados en todos los ID Token emitidos no es lo ideal.

Este ejemplo muestra cómo se agregan claims personalizados a un ID Token mediante el método `api.idToken.setCustomClaims`. Para agregar estos claims a un token de acceso, usa el método `api.accessToken.setCustomClaim`.

Para obtener más información sobre el objeto de evento del trigger, consulta [Actions Triggers: post-login - Event Object](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger/post-login-event-object). Para obtener más información sobre los tokens, consulta [Tokens](/docs/es/secure/tokens).
