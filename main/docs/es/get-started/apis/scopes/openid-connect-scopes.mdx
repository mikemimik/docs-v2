Este documento analiza los scopes incluidos en el protocolo de autenticación <Tooltip tip="OpenID: Estándar abierto de autenticación que permite a las aplicaciones verificar la identidad de los usuarios sin recopilar ni almacenar información de inicio de sesión." cta="Ver glosario" href="/docs/es/glossary?term=OpenID">OpenID</Tooltip> Connect (OIDC). Para obtener más información sobre OIDC, consulta [Protocolo OpenID Connect](/docs/es/authenticate/protocols/openid-connect-protocol).

Los scopes de OpenID Connect (OIDC) se usan durante la autenticación para autorizar el acceso a los datos de un usuario, como su nombre y su foto. Cada scope devuelve un conjunto de atributos del usuario, llamados claims. Los scopes que una aplicación debe solicitar dependen de los atributos del usuario que necesite. Una vez que el usuario autoriza los scopes solicitados, los claims se devuelven en un <Tooltip tip="ID Token: Credencial destinada al propio cliente, en lugar de utilizarse para acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=ID+Token">token de ID</Tooltip> y también están disponibles a través del [endpoint `/userinfo`](/docs/es/api/authentication/user-profile/get-user-info).

Por ejemplo, supongamos que has creado una aplicación web tradicional, la has registrado en Auth0 y la has configurado para permitir que un usuario inicie sesión con nombre de usuario y contraseña. Una vez que un usuario inicia sesión en tu aplicación, quieres generar y enviar automáticamente un correo electrónico de bienvenida personalizado que incluya su nombre.

1. Un usuario hace clic en **Login** dentro de tu aplicación.
2. Tu aplicación redirige al usuario al servidor de autorización de Auth0 (endpoint `/authorize`), incluyendo los siguientes scopes:

   * `openid` (obligatorio; para indicar que la aplicación pretende usar OIDC para verificar la identidad del usuario)
   * `profile` (para que puedas personalizar el correo electrónico con el nombre del usuario)
   * `email` (para que sepas adónde enviar el correo electrónico de bienvenida)
3. Tu servidor de autorización de Auth0 redirige al usuario al prompt de inicio de sesión.
4. El usuario se autentica y ve una página de consentimiento con la lista de scopes que Auth0 otorgará a tu aplicación, incluidos el acceso a la información de su perfil y a su dirección de correo electrónico.
5. El usuario acepta y autoriza a tu aplicación a tener ese nivel de acceso a la información que Auth0 almacena sobre él.
6. Tu aplicación ahora tiene acceso a la información del perfil del usuario y a su dirección de correo electrónico.

<div id="standard-claims">
  ## Claims estándar
</div>

Los claims estándar están pensados para proporcionar a una aplicación información del usuario, como el nombre, el correo electrónico y la foto, y están predefinidos para el protocolo OIDC. Estos claims se devuelven en un token de ID y también están disponibles a través del [`/userinfo` endpoint](/docs/es/api/authentication/user-profile/get-user-info).

También puedes crear custom claims, es decir, claims que defines, controlas y agregas a un token mediante [Auth0 Actions](/docs/es/customize/actions/actions-overview). Para obtener más información, consulta [JSON Web Token Claims](/docs/es/secure/tokens/json-web-tokens/json-web-token-claims).

El scope básico (y obligatorio) para OIDC es `openid`, que indica que una aplicación quiere usar el protocolo OIDC para verificar la identidad de un usuario. Además, una aplicación puede solicitar scopes adicionales incluyendo los nombres de los scopes solicitados en el parámetro `scope`, separados por espacios.

A continuación, se enumeran los claims estándar incluidos en los scopes más utilizados. Sin embargo, para consultar la lista completa de claims estándar disponibles, consulta [OIDC specification: Standard Claims on openid.net](https://openid.net/specs/openid-connect-core-1_0.html#StandardClaims). Para ver la lista completa de Scopes, consulta [OIDC specification: Requesting Claims Using Scope Values on openid.net](https://openid.net/specs/openid-connect-core-1_0.html#ScopeClaims).

| Scope     | Claims                                                                                                                                                                                                                                                                                                                           |
| --------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `openid`  | (obligatorio) Devuelve el claim `sub`, que identifica de forma única al usuario. En un token de ID, también estarán presentes los claims `iss`, `aud`, `exp`, `iat` y `at_hash`. Para obtener más información sobre los claims del token de ID, consulta [ID Token Structure](/docs/es/secure/tokens/id-tokens/id-token-structure). |
| `profile` | Devuelve claims que representan información básica del perfil, incluidos `name`, `family_name`, `given_name`, `middle_name`, `nickname`, `picture` y `updated_at`.                                                                                                                                                               |
| `email`   | Devuelve el claim `email`, que contiene la dirección de correo electrónico del usuario, y `email_verified`, que es un valor booleano que indica si el usuario verificó su dirección de correo electrónico.                                                                                                                       |

Para ver un ejemplo de cómo solicitar claims estándar para tu aplicación, consulta [Ejemplos de casos de uso: Scopes and Claims](/docs/es/get-started/apis/scopes/sample-use-cases-scopes-and-claims).
