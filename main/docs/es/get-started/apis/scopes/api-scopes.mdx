Como desarrollador de APIs, necesita:

1. Decidir a qué información le gustaría que las aplicaciones pudieran acceder en nombre de un usuario.
2. Definir estos niveles de acceso como scopes personalizados. (Para saber qué son los scopes, lea [Scopes](/docs/es/get-started/apis/scopes).)
3. Identificar estos scopes para que las aplicaciones cliente puedan utilizarlos.

<div id="ways-to-use-api-scopes">
  ## Formas de usar scopes de API
</div>

Puedes usar scopes de API de distintas maneras:

* En una API en la que la aplicación que realiza la llamada es una aplicación de terceros, o externa. En este caso, la aplicación que realiza la llamada solicitará autorización al usuario para acceder a los scopes solicitados, y el usuario aprobará o denegará la solicitud.
* En una API en la que la aplicación que realiza la llamada es una aplicación propia, o una aplicación registrada en el mismo dominio de Auth0 que la API a la que llama. En este caso, de forma predeterminada no se solicita el consentimiento del usuario, pero puedes configurar que sea obligatorio.
* En una API en la que la aplicación que realiza la llamada es un servicio de back-end, ya sea de terceros o propio, y no existe ningún usuario. En este caso, nunca se solicita el consentimiento del usuario.

Todos estos ejemplos usan scopes para limitar el acceso mediante el uso de un token. Si así lo deseas, tu API también puede usar lógica adicional aparte del token para aplicar un control de acceso más amplio.

Para ver un ejemplo de cómo solicitar acceso a una API personalizada para tu aplicación, lee [Casos de uso de ejemplo: Scopes y Claims](/docs/es/get-started/apis/scopes/sample-use-cases-scopes-and-claims).

<div id="example-an-api-called-by-a-third-party-application">
  ### Ejemplo: Una API a la que llama una aplicación de terceros
</div>

Supongamos que estás creando una API que proporciona información de cuentas bancarias a aplicaciones de pago en línea. En determinados momentos, las aplicaciones pueden necesitar consultar los saldos de las cuentas o transferir fondos. Para ello, creas dos scopes para tu API: uno que autoriza el acceso de lectura al saldo de una cuenta (`read:balance`) y otro que autoriza las transferencias de fondos (`transfer:funds`). Tu API está registrada en Auth0.

La aplicación que realiza la llamada solicitará autorización al usuario para acceder a los scopes solicitados, y el usuario aprobará o denegará la solicitud. La aplicación puede solicitar acceso de lectura al saldo del usuario incluyendo el scope `read:balance` en su solicitud, acceso para realizar transferencias de fondos incluyendo el scope `transfer:funds` en su solicitud, o acceso tanto para consultar el saldo del usuario como para transferir fondos incluyendo los scopes `read:balance` y `transfer:funds` en su solicitud.

Cuando la aplicación llame a tu API, incluirá un token que verifica que el usuario ha autorizado el acceso a su contenido y que también indica qué scopes ha aprobado. Tu API debe respetar los scopes aprobados y solo devolver la información que el usuario haya autorizado para la aplicación que realiza la llamada.

<div id="example-an-api-called-by-a-first-party-application">
  ### Ejemplo: Una API invocada por una aplicación propia
</div>

Supongamos que estás creando una API que proporciona datos a una aplicación de eventos que también has desarrollado. [Implementas el control de acceso basado en roles (RBAC)](/docs/es/manage-users/access-control/rbac), creando un rol `organizer` y un rol `participant`. Los usuarios con el rol `organizer` necesitan crear y actualizar eventos, mientras que los usuarios con el rol `participant` necesitan ver eventos y registrarse en ellos. Para ello, creas cuatro scopes para tu API: uno que autoriza la creación de eventos (`create:events`), uno que autoriza la actualización de eventos (`update:events`), uno que autoriza el acceso de solo lectura a los eventos (`view:events`) y uno que autoriza el registro en eventos (`register:events`). Tanto tu API como la aplicación de eventos están registradas en Auth0, y la opción **Allow Skipping User Consent** para aplicaciones propias está habilitada para tu API. Has instalado la Authorization Extension, configurado un rol `organizer`, creado los scopes `create:events` y `update:events` para ese rol, y se lo has asignado al Usuario A. También has configurado un rol `participant`, creado los scopes `view:events` y `register:events` para ese rol, y se lo has asignado al Usuario B.

El Usuario A se autentica en la aplicación que realiza la llamada, que solicita los scopes necesarios, pero como se trata de una aplicación propia, no se solicitará el consentimiento del usuario. La aplicación puede solicitar cualquier combinación de los scopes `create:events`, `update:events`, `view:events` y `register:events`, pero se reconoce que el Usuario A tiene el rol `organizer` y, por lo tanto, solo se le conceden los scopes `create:events` y `update:events`.

Ahora, cuando la aplicación llame a tu API, incluirá un token que verifica que solo tiene autorización para los scopes asociados al rol del usuario autenticado.

<div id="example-an-api-called-by-a-back-end-service">
  ### Ejemplo: Una API llamada por un servicio de back-end
</div>

Supongamos que trabajas para un hospital y tienes una API que genera grandes cantidades de datos de diagnóstico por imagen cada vez que un paciente se hace una resonancia magnética. Almacenas esos datos localmente durante seis meses, pero el hospital necesita que las imágenes se conserven a largo plazo por motivos de cumplimiento normativo. Por eso, el hospital tiene un servicio que copia cada noche los datos de diagnóstico por imagen a una solución externa de almacenamiento en frío y elimina todos los datos médicos locales después de seis meses de almacenamiento.

Para ello, creas dos scopes para tu API: uno que autoriza acceso de lectura a tus datos de imágenes (`read:images`) y otro que autoriza acceso de eliminación a tus datos de imágenes (`delete:images`). Tu API y el servicio automatizado están registrados en Auth0, y has autorizado al servicio automatizado a solicitar tokens para tu API.

El servicio automatizado que realiza la llamada solicitará los scopes necesarios, pero como no hay ningún usuario, no se solicitará consentimiento. El servicio puede solicitar acceso de lectura a tus datos de imágenes incluyendo el scope `read:images` en su solicitud, acceso de eliminación incluyendo el scope `delete:images` en su solicitud, o ambos accesos, de lectura y de eliminación, incluyendo los scopes `read:images` y `delete:images` en su solicitud.

Ahora, cuando el servicio automatizado llame a tu API, incluirá un token que verifica que tiene autorización para los scopes solicitados.

<div id="limit-api-scopes">
  ## Limita los scopes de API
</div>

Una aplicación puede incluir cualquier scope definido para una API en su solicitud. Sin embargo, en lugar de permitir que se soliciten todos los scopes disponibles, puedes controlar cómo las aplicaciones acceden a tus APIs mediante [políticas de acceso a la API para aplicaciones](/docs/es/get-started/apis/api-access-policies-for-applications).
