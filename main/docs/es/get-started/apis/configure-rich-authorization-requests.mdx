description: Aprende a configurar Rich Authorization Requests (RAR) para un servidor de recursos.
title: Configurar Rich Authorization Requests (RAR)
---

Con [Rich Authorization Requests (RAR)](https://datatracker.ietf.org/doc/html/draft-ietf-oauth-rar), los clientes pueden solicitar y obtener datos de <Tooltip tip="Autorización granular (FGA): producto de Auth0 que permite a usuarios individuales acceder a objetos o recursos específicos." cta="Ver glosario" href="/docs/es/glossary?term=fine-grained+authorization">autorización granular</Tooltip> de <Tooltip tip="Resource Owner: entidad (como un usuario o una aplicación) capaz de conceder acceso a un recurso protegido." cta="Ver glosario" href="/docs/es/glossary?term=resource+owners">propietarios de recursos</Tooltip>, como los usuarios finales, durante el [flujo de código de autorización con solicitudes de autorización push (PAR)](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow/authorization-code-flow-with-par) y el [Client-Initiated Backchannel Authentication Flow](/docs/es/get-started/authentication-and-authorization-flow/client-initiated-backchannel-authentication-flow).

Rich Authorization Requests utiliza el parámetro `authorization_details`. Este parámetro acepta un arreglo JSON de objetos que puede incluir información detallada sobre la autorización solicitada, como los recursos o acciones específicos a los que el cliente quiere acceder en nombre del usuario.

Puedes renderizar los `authorization_details` solicitados al usuario mediante:

* el navegador, usando una [solicitud de consentimiento personalizada](#set-customized-consent-prompt)
* una aplicación móvil al usar notificaciones push con el [Auth0 Guardian SDK o la aplicación Auth0 Guardian](/docs/es/secure/multi-factor-authentication/auth0-guardian)

Para configurar Rich Authorization Requests para un <Tooltip tip="Resource Server: servidor que aloja recursos protegidos. Los servidores de recursos aceptan solicitudes de recursos protegidos y responden a ellas." cta="Ver glosario" href="/docs/es/glossary?term=resource+server">servidor de recursos</Tooltip>, debes:

1. [Registrar tipos de `authorization_details`](#register-authorization_details-types) para el servidor de recursos.
2. [Configurar las políticas de acceso a la API](#configure-the-api-access-policies) para permitir que tu aplicación solicite esos tipos.
3. [Configurar la solicitud de consentimiento personalizada](#set-customized-consent-prompt) para renderizar los `authorization_details`.
4. [Opcional: Configurar la política de consentimiento para el servidor de recursos](#optional-configure-consent-policy-for-the-resource-server).

<div id="register-authorization_details-types">
  ## Registrar tipos de `authorization_details`
</div>

Cada objeto del arreglo JSON debe tener un campo `type`, que describe la estructura del objeto. Un arreglo `authorization_details` puede contener varias entradas del mismo `type`.

<div id="auth0-guardian-app">
  ### aplicación Auth0 Guardian
</div>

Si usas aplicación Auth0 Guardian, el valor del parámetro `authorization_details` debe contener un solo objeto en el arreglo, y ese objeto debe ajustarse al siguiente esquema de Auth0:

| Campo         | Descripción                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                       | Ejemplo                                                                                                                           |
| ------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------- |
| `type`        | Especifica el tipo de solicitud de autorización:<br /><ul><li>`urn:auth0:schemas:authorization-details`: El URN de Auth0 indica que la solicitud usará el esquema de Auth0.</li><li>`v1`: La versión del esquema.</li><li>`user-profile`: Valor proporcionado por el cliente que indica que la solicitud es de información del perfil del usuario.</li></ul>                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      | `urn:auth0:schemas:authorization-details:v1:user-profile`                                                                         |
| `instruction` | Un mensaje legible para las personas para que el usuario apruebe la solicitud.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                    | `Please approve the request.`                                                                                                     |
| `properties`  | Un objeto JSON que contiene los atributos o claims específicos del usuario que se solicitan. Cada clave (por ejemplo, `email`, `full_name`) representa un campo concreto del perfil del usuario:<br /><ul><li>`display`: Un valor booleano que determina si la propiedad debe mostrarse al usuario en el cuadro de diálogo de consentimiento. Si es `true`, se mostrará; si es `false`, será una propiedad solo de uso interno que no está pensada para que la vea el usuario.</li><li>`name`: El nombre legible para las personas de la propiedad (por ejemplo, &quot;Email Address&quot;).</li><li>`display_order`: Un entero que indica el orden en que se mostrarán las propiedades en el cuadro de diálogo de consentimiento.</li><li>`description`: Una explicación breve y opcional del propósito de la propiedad.</li><li>`value`: El valor real de los datos de la propiedad (por ejemplo, &quot;user@example.com&quot;, &quot;John Doe&quot;). El tipo de dato puede variar (string, integer, boolean, etc.).</li></ul> | `"properties": { "stringPropertyForDisplay": { "display": true, "name": "A String:", "display_order": "1", "value": "Value 1"} }` |

A continuación, se muestra un ejemplo de un tipo `authorization_details` con el esquema de Auth0:

```json lines
{
    "type": "urn:auth0:schemas:authorization-details:v1:user-profile",
    "instruction": "An instruction to the user",
    "properties": {
        "stringPropertyForDisplay": {
            "display": true,
            "name": "A String:",
            "display_order": 1,
            "value": "Value 1"
        },
        "numericPropertyForDisplay": {
            "display": true,
            "name": "A Number:",
            "display_order": 2,
            "description": "An optional description",
            "value": 100.00
        },
        "booleanPropertyForDisplay": {
            "display": true,
            "name": "A Boolean:",
            "display_order": 3,
            "value": true
        },
        "hiddenProperty": {
            "display": false,
            "value": "This value should not be displayed"
        }
    }
}
```

<div id="other-notification-channels">
  ### Otros canales de notificación
</div>

El tipo `authorization_details` no necesita usar el esquema de Auth0 si no estás usando la aplicación Auth0 Guardian. Si muestras `authorization_details` mediante una solicitud de consentimiento personalizada o tu propia aplicación móvil personalizada con el Auth0 Guardian SDK, se aplican los siguientes requisitos:

* Máximo de 5 KB
* Debe ser JSON válido
* Debe ser un arreglo de objetos
* Máximo de 5 entradas en el arreglo
* Cada objeto debe tener una propiedad `type` (prerregistrada en la API)
* Máximo de 10 propiedades por objeto
* La longitud máxima de los nombres de las propiedades es de 255 caracteres
* La longitud máxima del valor de una propiedad es de 255 caracteres
* Máximo de 5 niveles de objetos anidados
* Los nombres de las propiedades solo pueden contener los siguientes caracteres: `a-zA-Z0-9_.-`

A continuación se muestra un ejemplo de `authorization_details` de tipo `money_transfer` que no usa el esquema de Auth0. Contiene los siguientes campos de objeto:

* `instructedAmount`: El monto en USD que se transferirá.
* `sourceAccount`: La cuenta bancaria de origen desde la que se transferirá el dinero.
* `destinationAccount`: La cuenta bancaria de destino a la que se transferirá el dinero.
* `beneficiary`: El destinatario de la transferencia de dinero.
* `subject`: El asunto de la transferencia de dinero.

```json lines
{
  "type": "money_transfer", 
  "instructedAmount": {"amount": 2500, "currency": "USD"},   
  "sourceAccount": "xxxxxxxxxxx1234", 
  "destinationAccount": "xxxxxxxxxxx9876", 
  "beneficiary": "Hanna Herwitz", 
  "subject": "A Lannister Always Pays His Debts"
}
```

Debes registrar tipos de `authorization_details` para un servidor de recursos, lo cual es similar a registrar scopes permitidos. Puedes registrar tipos de `authorization_details` con el [Auth0 Dashboard](https://manage.auth0.com/) o la [Management API](https://auth0.com/docs/api/management/v2).

    2. Selecciona la pestaña **Permissions**.
    3. En **Add an Authorization Details type**, puedes agregar varios tipos de `authorization_details` para tu servidor de recursos. Introduce un tipo de `authorization_details`
    4. Selecciona la opción **+Add**.

    Puedes ver los tipos de `authorization_details` para tu servidor de recursos en **List of Authorization Details Types**:

    <Frame>![](/docs/images/cdy7uua7fh8z/6qRpmbZiqaU8pSFDuWzUAy/951c827a307791de8df9cecd8f817351/Screenshot_2025-04-07_at_3.45.36_PM.png)</Frame>
  </Tab>

  <Tab title="Management API">
    Para registrar tipos de `authorization_details` en un servidor de recursos existente, realiza una solicitud `PATCH` al endpoint [Update a resource server](https://auth0.com/docs/api/management/v2/resource-servers/patch-resource-servers-by-id).

    El siguiente ejemplo de código agrega los tipos `payment_initiation` y `money_transfer` en `authorization_details` para un servidor de recursos:

    ```bash lines
    curl --location --request PATCH 'https://$tenant/api/v2/resource-servers/$resource-server-id' \
      --header 'Authorization: Bearer $management_access_token' \
      --header 'Content-Type: application/json' \
      --data-raw '{
      "authorization_details": [{"type": "payment_initiation"}, {"type": "money_transfer"}]
      }'
    ```

    Para crear un nuevo servidor de recursos con un tipo de `authorization_details` registrado, realiza una solicitud `POST` al endpoint `/resource-servers`.

    La siguiente solicitud `POST` crea un nuevo servidor de recursos con el tipo de `authorization_details` `payment_initiation`:

    ```bash lines
    curl --location --request POST 'https://$tenant/api/v2/resource-servers' \
      --header 'Authorization: Bearer $management_access_token' \
      --header 'Content-Type: application/json' \
      --data-raw '{
      "name": "Payments API",
      "identifier": "https://payments.api/",
      "authorization_details": [{"type": "payment_initiation"}]
      }'
    ```
  </Tab>
</Tabs>

<div id="configure-the-api-access-policies">
  ## Configura las políticas de acceso a la API
</div>

[API Access Policies for Applications](/docs/es/get-started/apis/api-access-policies-for-applications) controlan a qué API pueden acceder tus aplicaciones y qué scopes o tipos de `authorization_details` tienen permitido usar.

Puedes consultar la política actual de tu API mediante la Management API. Haz una solicitud `GET` al endpoint [Obtener un servidor de recursos](https://auth0.com/docs/api/management/v2/resource-servers/get-resource-servers-by-id) y revisa la propiedad `subject_type_authorization` en la respuesta:

```bash lines
curl --location --request GET 'https://$tenant/api/v2/resource-servers/$resource-server-id' \
  --header 'Authorization: Bearer $management_access_token' \
  --header 'Content-Type: application/json'
```

La propiedad `subject_type_authorization` tiene valores para `client.policy` y `user.policy`:

* Si el valor de la política es `allow_all`, las aplicaciones o los usuarios pueden solicitar todos los tipos de `authorization_details` registrados para la API.
* Si el valor de la política es `require_client_grant`, cada tipo de `authorization_details` debe estar permitido explícitamente por la autorización de cliente de esa aplicación.
* Si el valor de la política es `deny_all`, ninguna aplicación ni usuario puede solicitar ninguno de los tipos de `authorization_details` registrados para la API.

Para obtener más información sobre cómo administrar las autorizaciones de cliente para las aplicaciones, consulta [Acceso de aplicaciones a las API: autorizaciones de cliente](/docs/es/get-started/applications/application-access-to-apis-client-grants) .

<div id="set-customized-consent-prompt">
  ## Configura la solicitud de consentimiento personalizada
</div>

Puedes renderizar los `authorization_details` de una solicitud de autorización enriquecida en la solicitud de consentimiento. Para ello, configura el prompt `customized-consent` con los partials de plantilla adecuados.

Puedes configurar la solicitud de consentimiento personalizada mediante Auth0 CLI o Management API.

<div id="auth0-cli">
  ### Auth0 CLI
</div>

Para configurar los partials de consentimiento personalizado, ejecuta el comando `auth0 ul customize` con los indicadores correspondientes en tu terminal:

```bash lines
auth0 ul customize
```

Para obtener más información, consulta la [documentación sobre cómo personalizar Universal Login de Auth0](https://auth0.github.io/auth0-cli/auth0_universal-login_customize.html).

<div id="management-api">
  ### Management API
</div>

Para configurar las partials de consentimiento personalizado, haz una solicitud `PUT` al endpoint `/prompts/customized-consent/partials`:

```bash lines
curl --location --request PUT "https://$tenant/api/v2/prompts/customized-consent/partials" \
  --header "Authorization: Bearer $management_access_token" \
  --header "Content-Type: application/json" \
  --data '{
    "customized-consent": {
      "form-content": "<div style=\"font-size: 1.3em; font-weight: bold;\">Operation Details</div><hr style=\"margin: 10px 0;\"><div style=\"margin-bottom: 20px;\"></div><div style=\"font-weight: bold;\">Transaction Type</div><div>{{ transaction.params.authorization_details[0].type }}</div><div style=\"margin-bottom: 20px;\"></div><div style=\"font-weight: bold;\">Amount</div><div>{{ transaction.params.authorization_details[0].instructedAmount.amount }} {{ transaction.params.authorization_details[0].instructedAmount.currency }}</div><div style=\"margin-bottom: 20px;\"></div><div style=\"font-weight: bold;\">Recipient</div><div>{{ transaction.params.authorization_details[0].beneficiary }}</div><div style=\"margin-bottom: 20px;\"></div><div style=\"font-weight: bold;\">Destination Account</div><div>{{ transaction.params.authorization_details[0].destinationAccount }}</div><div style=\"margin-bottom: 20px;\"></div>"
    }
  }'
```

<div id="optional-configure-consent-policy-for-the-resource-server">
  ## Opcional: Configurar la política de consentimiento para el servidor de recursos
</div>

La política de consentimiento del servidor de recursos determina si Auth0 almacena los valores de `authorization_details` y los pone a disposición de las aplicaciones móviles cuando se envía una notificación push.

Revisa el comportamiento de la política de consentimiento estándar de Auth0 para una request que contiene `authorization_details`:

| Flujo                                            | Notificación push enviada | Comportamiento                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                               |
| ------------------------------------------------ | ------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Cualquiera                                       | No                        | Se muestra la solicitud de consentimiento personalizada.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| flujo de código de autorización con solicitudes de autorización push                 | Sí                        | No se muestra ninguna solicitud de consentimiento. El consentimiento debe mostrarse en la aplicación móvil que recibe el desafío de notificación push.<br />Si se usa la aplicación Auth0 Guardian, mostrará automáticamente los `authorization_details` al usuario.<br />Si se usa una aplicación móvil personalizada, los `authorization_details` se pueden recuperar con el Auth0 Guardian SDK.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                  |
| Client-Initiated Backchannel Authentication Flow | Sí                        | Si se usa la aplicación Auth0 Guardian para autorizar la CIBA request, los `authorization_details` se obtendrán y mostrarán automáticamente.<br />Si se usa una aplicación móvil personalizada para autorizar la CIBA request, los `authorization_details` se pueden recuperar con el Auth0 Guardian SDK.<br />Si la CIBA request se autoriza mediante un enlace web (por ejemplo, desde un correo electrónico), se mostrará la solicitud de consentimiento personalizada.<br />Los clientes pueden optar por activar una notificación push como segundo factor para la CIBA request cuando el usuario la aprueba mediante un enlace web; en ese caso, el comportamiento es el mismo que el anterior. La aplicación Auth0 Guardian volverá a mostrar automáticamente los `authorization_details` al usuario, mientras que las aplicaciones móviles personalizadas pueden optar por recuperarlos con el Auth0 Guardian SDK. |

Los clientes también pueden establecer `consent_policy` en `transactional-authorization-with-mfa`, que tiene el siguiente comportamiento:

| Flujo                                            | Notificación push enviada | Comportamiento                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                              |
| ------------------------------------------------ | ------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| flujo de código de autorización con solicitudes de autorización push                 | No                        | Se muestra la solicitud de consentimiento personalizada.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                    |
| flujo de código de autorización con solicitudes de autorización push                 | Sí                        | No se muestra ninguna solicitud de consentimiento. La solución del cliente debe mostrar el consentimiento mediante su propia interfaz de usuario. Auth0 asignará un ID único a la request y lo expondrá a la Post-Login Action como `event.transaction.linking_id`, junto con `event.transaction.requested_authorization_details`.<br />Si se usa la aplicación Auth0 Guardian, NO se mostrarán los `authorization_details`.<br />Si se usa una aplicación móvil personalizada, la notificación push incluirá el `linking_id`, lo que permitirá a los desarrolladores de aplicaciones recuperar los `authorization_details` desde sus propias API si es necesario. |
| Client-Initiated Backchannel Authentication Flow | Cualquiera                | El CIBA flow no es compatible con la política de consentimiento `transactional-authorization-with-mfa`                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |

Puedes establecer la política de consentimiento de un servidor de recursos con el [Auth0 Dashboard](https://manage.auth0.com/) o la [Management API](https://auth0.com/docs/api/management/v2).

  <Tab title="Auth0 Dashboard">
    Establece la política de consentimiento en la configuración de tu API usando el Auth0 Dashboard.

    1. Ve a [Auth0 Dashboard &gt; Applications &gt; APIs](https://manage.auth0.com/#/apis).
    2. Selecciona la pestaña **Settings**.
    3. En **Access Settings**, elige la política de consentimiento **Standard**.
    4. Guarda los cambios.

  </Tab>

    ```bash wrap lines

      --header 'Content-Type: application/json' \
      --data-raw '{ "consent_policy": "standard" }'
    ```
