Si tiene varias implementaciones de API diferenciadas que, lógicamente, forman parte de la misma API, puede simplificar su proceso de autorización representándolas como una sola API lógica en el <Tooltip tip="Auth0 Dashboard: El producto principal de Auth0 para configurar sus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip>. Esto le permite implementar un solo <Tooltip tip="Auth0 Dashboard: El producto principal de Auth0 para configurar sus servicios." cta="Ver glosario" href="/docs/es/glossary?term=authorization+flow">flujo de autorización</Tooltip> y, al mismo tiempo, seguir controlando el acceso a cada API mediante la asignación de los scopes adecuados.

Las siguientes secciones describen cómo usar y representar varias API como un solo <Tooltip tip="Resource Server: Servidor que hospeda recursos protegidos. Los Resource Servers aceptan y responden a requests de recursos protegidos." cta="Ver glosario" href="/docs/es/glossary?term=Resource+Server">Resource Server</Tooltip> en Auth0. En los ejemplos usaremos la siguiente aplicación de ejemplo. La aplicación usa una arquitectura de microservicios y contiene:

* 2 API de Node.js: `contacts` y `calendar` (puede considerarlas microservicios)
* 1 Resource Server que representa las 2 API
* 2 scopes con espacio de nombres: `read:contacts` y `read:calendar`
* El flujo de concesión implícita para obtener un `access_token` que funcione para ambas API

Representaremos las dos API con una sola Auth0 API llamada `Organizer Service`. Luego crearemos dos scopes para demostrar cómo puede usar el [flujo implícito](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce) para acceder a las API `calendar` y `contacts` desde la SPA.

Debe completar los siguientes pasos:

1. Habilitar una conexión para su aplicación
2. Crear un usuario de prueba
3. Registrar una API lógica en Auth0
4. Configurar scopes para la API lógica
5. Otorgar acceso a la API lógica
6. (Opcional) Implementar cierre de sesión único (SLO) o inicio de sesión único (SSO)

<div id="prerequisites">
  ## Requisitos previos
</div>

* [Registra tu aplicación](/docs/es/get-started/auth0-overview/create-applications/single-page-web-apps).

  * Selecciona **Tipo de aplicación** como **Aplicación web de página única**.
  * Agrega las **URL de devolución de llamada permitidas** `http://localhost:3000` y `http://localhost:3000/callback.html`.
* Descarga la [aplicación de ejemplo](https://github.com/auth0-samples/auth0-api-auth-implicit-sample). Para saber cómo configurarla, consulta el [README](https://github.com/auth0-samples/auth0-api-auth-implicit-sample#readme).

<div id="enable-a-connection-for-your-application">
  ## Habilita una conexión para tu aplicación
</div>

Necesitarás una fuente de usuarios para tu aplicación recién registrada, por lo que tendrás que configurar una [conexión](/docs/es/authenticate/identity-providers). Para este ejemplo, crearemos una [conexión de base de datos](/docs/es/authenticate/database-connections) sencilla que solicite únicamente la dirección de correo electrónico del usuario y una contraseña. Para obtener más información, consulta [Configurar Database Connections](/docs/es/get-started/applications/set-up-database-connections).

<div id="create-a-test-user">
  ## Crear un usuario de prueba
</div>

Como estás trabajando con una conexión recién creada, no habrá ningún usuario asociado a ella. Antes de poder probar el proceso de inicio de sesión de la aplicación de ejemplo, tendrás que crear un usuario y asociarlo con la conexión, así que asegúrate de seleccionar la conexión que acabas de crear al crear el usuario. Para obtener más información, consulta [Crear usuarios](/docs/es/manage-users/user-accounts/create-users).

<div id="register-a-logical-api-in-auth0">
  ## Registrar una API lógica en Auth0
</div>

Registra una única API lógica que usarás para representar las múltiples APIs incluidas en la aplicación de ejemplo. Para este ejemplo, llama a tu API `Organizer Service` y establece su identificador único como `organize`. De forma predeterminada, el <Tooltip tip="Algoritmo de firma: algoritmo utilizado para firmar digitalmente los tokens y garantizar que no hayan sido manipulados." cta="Ver glosario" href="/docs/es/glossary?term=signing+algorithm">algoritmo de firma</Tooltip> de los tokens obtenidos para esta API es **RS256**, que debes dejar tal cual. Para obtener más información, consulta [Registrar APIs](/docs/es/get-started/auth0-overview/set-up-apis).

<div id="configure-permissions-for-the-logical-api">
  ## Configurar permisos para la API lógica
</div>

Para permitir que la API lógica represente las API incluidas en la aplicación de ejemplo, deberás crear los permisos (scopes) adecuados.

Los scopes te permiten definir a qué acciones de la API pueden acceder las aplicaciones cliente. Cada scope representa una combinación de API/acción. Para este ejemplo, quieres que las aplicaciones cliente puedan `read` de una API llamada `calendar` y de otra llamada `contacts`, por lo que deberás crear los siguientes permisos:

* `read:calendar`
* `read:contacts`

Puedes considerar cada una como un microservicio. Para obtener más información, consulta [Agregar permisos de API](/docs/es/get-started/apis/add-api-permissions) y [Scopes de API](/docs/es/get-started/apis/scopes/api-scopes).

<div id="grant-access-to-the-logical-api">
  ## Conceder acceso a la API lógica
</div>

Ya puede proporcionar acceso a sus API permitiendo que la API lógica obtenga <Tooltip tip="Token de acceso: credencial de autorización, en forma de una cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=Access+Tokens">tokens de acceso</Tooltip>. Al incluir los scopes necesarios, puede controlar el acceso de una aplicación a las API representadas por la API lógica. Los pasos siguientes usan el [flujo implícito](/docs/es/get-started/authentication-and-authorization-flow/implicit-flow-with-form-post) para ajustarse al ejemplo. Sin embargo, puede usar el flujo que mejor se adapte a sus necesidades. Por ejemplo:

* Si tiene una **aplicación de máquina a máquina**, puede autorizarla para solicitar tokens de acceso para su API mediante un [Client Credentials Flow](/docs/es/get-started/authentication-and-authorization-flow/client-credentials-flow).
* Si está creando una **aplicación nativa**, puede implementar el [flujo de código de autorización con Proof Key for Code Exchange (PKCE)](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce).

Para obtener más información sobre los flujos de autorización, consulte [Flujos de autenticación y autorización](/docs/es/get-started/authentication-and-authorization-flow).

1. El usuario hace clic en Login dentro de la SPA y la aplicación redirige al usuario al servidor de autorización de Auth0 (endpoint `/authorize`). Para obtener más información sobre los parámetros de la llamada, consulte nuestro tutorial: [Llame a su API usando el flujo de código de autorización con PKCE](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce/call-your-api-using-the-authorization-code-flow-with-pkce).

4. Si el usuario da su consentimiento, Auth0 lo redirige de vuelta a la SPA con tokens en el fragmento hash del URI. La SPA ahora puede extraer los tokens del fragmento hash con JavaScript y usar el token de acceso para llamar a sus API en nombre del usuario.

   ```javascript lines
   function getParameterByName(name) {
     var match = RegExp('[#&]' + name + '=([^&]*)').exec(window.location.hash);
     return match && decodeURIComponent(match[1].replace(/\+/g, ' '));
   }

   function getAccessToken() {
     return getParameterByName('access_token');
   }
   ```

<div id="implement-single-logout-slo-or-single-sign-on-sso">
  ### Implementar cierre de sesión único (SLO) o inicio de sesión único (SSO)
</div>

En algunos escenarios con varias aplicaciones en los que se requiere cierre de sesión único (es decir, si un usuario cierra sesión en una aplicación, también debe cerrarla en las demás), una aplicación puede configurarse para consultar periódicamente a Auth0 mediante `checkSession()` y comprobar si existe una sesión. Si la sesión no existe, puede cerrar la sesión del usuario en la aplicación. Este mismo método de consulta periódica también puede usarse para implementar la autenticación silenciosa en un escenario de <Tooltip tip="Inicio de sesión único (SSO): Servicio que, después de que un usuario inicia sesión en una aplicación, inicia automáticamente la sesión de ese usuario en otras aplicaciones." cta="Ver glosario" href="/docs/es/glossary?term=Single+Sign-on">inicio de sesión único</Tooltip> (SSO).

El intervalo entre consultas a `checkSession()` debe ser de al menos 15 minutos para evitar posibles problemas futuros con el límite de frecuencia de esta llamada.
