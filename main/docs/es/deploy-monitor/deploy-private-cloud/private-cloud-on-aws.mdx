La opción de implementación Private Cloud en AWS es una instancia dedicada y administrada de la plataforma de identidad de Auth0 que se ejecuta en Amazon Web Services. Ofrece aislamiento, mayor rendimiento, instancias de desarrollo independientes, varios add-ons y mucho más.

<div id="operational-differences">
  ## Diferencias operativas
</div>

La siguiente tabla compara cada opción de implementación de Private Cloud en AWS.

| Característica                                 | Public Cloud                     | Private Cloud Basic on AWS * | Private Cloud Performance on AWS *                                                                                                                              |
| ---------------------------------------------- | -------------------------------- | ---------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Tenencia                                       | Múltiple                         | Única                        | Única                                                                                                                                                           |
| Solicitudes por segundo (múltiplos de 100 RPS) | 100 RPS (1x)                     | 100 RPS (1x)                 | 500* RPS (5x)<br />1,500* RPS (15x)<br />3,000* RPS (30x)<br />3,000* RPS (30x Burst)<br />6,000* RPS (60x)<br />6,000* RPS (60x Burst)<br />10,000* RPS (100x) |
| Acuerdo de nivel de servicio (SLA)             | 99.99%                           | 99.99%                       | 99.99%                                                                                                                                                          |
| Residencia de datos                            | Solo en regiones de nube pública | Sí                           | Sí                                                                                                                                                              |
| Entorno de desarrollo                          | No                               | No                           | 1                                                                                                                                                               |

*La capacidad de RPS se proporciona como guía general. El rendimiento real puede variar según los tipos y el volumen de transacciones procesadas en el entorno de Private Cloud del cliente. Las transacciones de prueba utilizadas para las referencias de capacidad indicadas podrían no coincidir con los patrones de transacción y las cargas de trabajo específicos de cada cliente. Por ejemplo, se admite la siguiente capacidad al usar el [Resource Owner Password Flow](/docs/es/get-started/authentication-and-authorization-flow/resource-owner-password-flow):

| Nivel                 | RPS  |
| --------------------- | ---- |
| Basic                 | 55   |
| Performance 5x        | 180  |
| Performance 15x       | 600  |
| Performance 30x       | 1400 |
| Performance 30x Burst | 1400 |
| Performance 60x       | 3000 |
| Performance 60x Burst | 3000 |
| Performance 100x      | 5000 |

<div id="data-residency">
  ### Residencia de datos
</div>

Con Private Cloud en AWS, usted elige la región en la que se almacenan sus datos. Auth0 puede proporcionar una lista de las regiones disponibles que utilizan varias [zonas de disponibilidad](https://aws.amazon.com/about-aws/global-infrastructure/regions_az/) para el despliegue. Puede encontrar una lista de las regiones actuales en las que ofrecemos despliegues de nube privada en nuestra Información sobre subencargados del tratamiento, publicada en nuestra [página de Trust &amp; Compliance](https://www.okta.com/trustandcompliance/). En la mayoría de los casos, Okta despliega las copias de seguridad en la misma región de AWS seleccionada.

<div id="maximum-availability">
  ### Máxima disponibilidad
</div>

Las instancias de Private Cloud en AWS cuentan con un Acuerdo de Nivel de Servicio (SLA) del 99,99 %. Los compromisos de disponibilidad no se aplican a las pruebas gratuitas, los entornos sandbox, beta ni otros entornos de preproducción.

<div id="high-demand-apps">
  ### Aplicaciones de alta demanda
</div>

Si su aplicación requiere una cantidad muy alta de solicitudes por segundo (RPS), quizá también le convenga considerar Private Cloud en AWS. Consulte las [políticas de límites de velocidad](/docs/es/troubleshoot/customer-support/operational-policies/rate-limit-policy) para obtener más información sobre los límites de velocidad estándar. Las implementaciones de Private Cloud en AWS tienen un límite de velocidad de 100 RPS para Private Basic y límites de velocidad ampliados de 500 RPS, 1,500 RPS, 3,000 RPS y 6,000 RPS para las opciones de Private Performance.

<div id="additional-dev-environments">
  ## Entornos de desarrollo adicionales
</div>

Las implementaciones de Private Cloud en AWS Performance incluyen una instancia totalmente aislada y actualizada de forma independiente para desarrollo y pruebas. Puede agregar entornos de preproducción adicionales para satisfacer los requisitos de su empresa.

Las solicitudes por segundo (RPS) garantizadas y el SLA no se aplican a los entornos no productivos.

<div id="bursty-traffic">
  ### Tráfico a ráfagas
</div>

Okta establece límites de frecuencia para las organizaciones en función del tráfico previsto y del nivel de RPS adquirido. Si su organización experimenta un volumen de tráfico superior al esperado, este uso no planificado puede afectar a los usuarios finales. La oferta de Private Cloud está diseñada para soportar aumentos graduales en la tasa de transacciones (por ejemplo, un aumento de 100 RPS a 1000 RPS en un período de 10 minutos) sin afectar al servicio. Sin embargo, los picos de tráfico repentinos y pronunciados (por ejemplo, un aumento de 100 RPS a 1000 RPS en cuestión de segundos) podrían provocar inestabilidad en el servicio y un aumento de la latencia mientras la solución se ajusta para gestionar la nueva carga.

Después de elegir Private Cloud en AWS, se seguirá un proceso de incorporación y despliegue para configurar sus entornos.

<div id="customer-onboarding-requirements">

</div>

Tras la firma del contrato, le pediremos que nos facilite información clave sobre sus requisitos de incorporación, que posteriormente validaremos.

<div id="kickoff-meeting">

</div>

Una vez que validemos sus requisitos de incorporación, celebraremos una reunión de inicio con usted para dar comienzo al proceso de implementación. Recomendamos encarecidamente que esta reunión tenga lugar, como máximo, cinco (5) días después de la firma del contrato.

<div id="implementation">
  ### Implementación
</div>

Inmediatamente después de validar el formulario de incorporación, comenzaremos a aprovisionar su entorno.

Al finalizar este proceso, todo estará listo para la entrega del entorno y su implementación de Private Cloud en AWS quedará lista para su uso.

<div id="secure-outbound-networking">
  ## Conectividad saliente segura
</div>

Algunas personalizaciones de la plataforma Auth0, como [Actions](/docs/es/customize/actions), los [webhooks personalizados](/docs/es/customize/log-streams/custom-log-streams) y los [action scripts personalizados de base de datos](/docs/es/authenticate/database-connections/custom-db/templates), te permiten realizar conexiones salientes desde la plataforma Auth0 hacia tus propios servicios. Con Private Cloud en AWS, puedes establecer conectividad de red entre tu despliegue de Private Cloud y tus propios servicios sin exponer tus datos a Internet.

La conectividad saliente segura utiliza AWS PrivateLink. Primero, compartes tu servicio a través de PrivateLink [configurando un servicio de endpoint](https://docs.aws.amazon.com/vpc/latest/privatelink/create-endpoint-service.html) en tu cuenta de AWS. El servicio subyacente puede ser un servicio nativo de AWS o un servicio que se ejecute en un centro de datos. El servicio debe estar en una VPC de la misma región de AWS que tu despliegue de Private Cloud.

A continuación, configuraremos tu despliegue de Private Cloud para que tu servicio de endpoint esté disponible. Una vez que proporciones a Auth0 la información de tu servicio de endpoint, integraremos el servicio con tu despliegue y te proporcionaremos información sobre cómo acceder a él desde tu código de personalización.

Para obtener más información sobre cómo configurar servicios de endpoint con PrivateLink, ponte en contacto con AWS. Para coordinar la incorporación del servicio con Auth0, envía una solicitud al [Support Center](https://support.auth0.com/).

<div id="load-testing">
  ### Pruebas de carga
</div>

Esta política describe los requisitos necesarios para que Auth0 realice pruebas de carga para clientes de Private Cloud en AWS que presenten una solicitud. Puedes enviar una solicitud de pruebas de carga a través del [Support Center](https://support.auth0.com/). En el campo **Issue**, selecciona **Private Cloud Support Incident**.

Si compraste un entorno dedicado para pruebas de carga, no hay límite en la frecuencia con la que puedes ejecutar estas pruebas. Los entornos estándar están limitados a dos (2) por año, siempre que se sigan los procedimientos adecuados para las pruebas de carga.

Para que la solicitud pueda ser aprobada, debe:

* Presentarse al menos dos (2) semanas antes de la fecha deseada para la prueba; en muchos casos, Auth0 recomienda avisar con un (1) mes de antelación para garantizar tiempo suficiente para una revisión exhaustiva y cualquier modificación necesaria.
* Recibir aprobación por escrito antes de que se realice cualquier prueba.
* Mantenerse dentro de nuestros [límites de velocidad de producción publicados](/docs/es/troubleshoot/customer-support/operational-policies/rate-limit-policy).

Si se solicitan cambios en la infraestructura, el costo se determinará en función de tus requisitos específicos.

<div id="testing-capacity-considerations">
  #### Consideraciones sobre la capacidad para pruebas
</div>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Los entornos adquiridos para pruebas de carga dedicadas no están sujetos a ningún acuerdo de nivel de servicio. Los problemas notificados en estos entornos tendrán menor prioridad que los problemas de producción.
</Callout>

Debe comenzar con una carga baja e ir aumentándola lentamente hasta que el entorno alcance su capacidad máxima. Si necesita una carga superior a la que el entorno puede manejar, deberá ampliarse el tamaño del entorno.

Los entornos de Private Cloud pueden ampliarse mediante un anexo contractual; póngase en contacto con su Account Executive y TAM para analizar esta compra.

| Suscripción                                    | Capacidad de prueba de carga | Incremento gradual |
| ---------------------------------------------- | ---------------------------- | ------------------ |
| Private Cloud Performance 500 RPS (5x)         | 325 RPS                      | 100 RPS/min        |
| Private Cloud Performance 1500 RPS (15x)       | 975 RPS                      | 100 RPS/min        |
| Private Cloud Performance 3000 RPS (30x)       | 1950 RPS                     | 100 RPS/min        |
| Private Cloud Performance 3000 RPS (30x Burst) | 1950 RPS                     | 100 RPS/min        |
| Private Cloud Performance 6000 RPS (60x)       | 3900 RPS                     | 100 RPS/min        |
| Private Cloud Performance 6000 RPS (60x Burst) | 3900 RPS                     | 100 RPS/min        |
| Private Cloud Performance 10,000 RPS (100x)    | 6500 RPS                     | 100 RPS/sec        |

Para obtener más información sobre las pruebas de carga en Private Cloud, consulte [Límites de solicitudes por entorno (solo Private Cloud)](/docs/es/troubleshoot/customer-support/operational-policies/rate-limit-policy#environment-request-limits-private-cloud-only-).

<div id="high-load-notifications">
  #### Notificaciones de alta demanda
</div>

En períodos de alta demanda previstos, debe informar a su equipo de cuenta con al menos 14 días de antelación al evento. La notificación permite probar adecuadamente los escenarios (si es posible) y ajustar el soporte reactivo al evento.

<div id="penetration-testing">
  ### Pruebas de penetración
</div>

Para realizar una prueba de seguridad, notifíquenos con antelación a través del Support Center de Auth0. Auth0 requiere al menos una semana (siete días) de antelación antes de la fecha de inicio prevista de la prueba.

Si la prueba se limita a su infraestructura (es decir, no se realizarán pruebas de los servicios de Auth0), no necesita notificar a Auth0.

Para conocer la información que requerimos, consulte nuestra [Política de pruebas de penetración](/docs/es/troubleshoot/customer-support/operational-policies/penetration-testing-policy).

<div id="failover-testing">
  ### Pruebas de conmutación por error
</div>

Esta política describe los requisitos necesarios para que Okta realice pruebas de conmutación por error para clientes de Private Cloud en la plataforma Auth0 de AWS o Azure con el add-on Geo Failover requerido. Puede enviar una solicitud de prueba de conmutación por error a través del [Support Center.](https://support.auth0.com/) En el campo Issue, seleccione Private Cloud Support Incident.

Para que la solicitud pueda aprobarse, debe:

* Presentarse al menos dos (2) semanas antes de la fecha y hora deseadas para la prueba (en UTC). En muchos casos, Okta recomienda avisar con un (1) mes de antelación para garantizar el tiempo necesario para una revisión exhaustiva y cualquier modificación requerida.
* No superar el límite de (2) pruebas de conmutación por error por año calendario.
* Recibir aprobación por escrito antes de que se realice cualquier prueba.
* Especificar las ventanas (en UTC) tanto para la conmutación por error como para el retorno a la región principal, teniendo en cuenta que ambas darán lugar a un tiempo de inactividad de hasta 15 minutos.
* Incluir un punto de contacto designado con quien Okta coordinará toda la logística de las pruebas

Tenga en cuenta que Okta se reserva el derecho de sugerir ventanas alternativas para la conmutación por error y el retorno, de acuerdo con la disponibilidad del personal para realizar las pruebas solicitadas. Además, cualquier interrupción del servicio que se produzca como parte de los procedimientos de conmutación por error o retorno queda exenta de cualquier disposición del SLA.

<div id="certificate-renewal-process">
  ## Proceso de renovación de certificados
</div>

Los certificados administrados por Auth0 (con el formato *.auth0app.com) son responsabilidad de Auth0, tanto su obtención como su implementación. Auth0 gestionará el proceso de principio a fin y te indicará cualquier acción necesaria.

Auth0 gestiona la renovación de los certificados emitidos por Auth0 para los <Tooltip tip="Dominio personalizado: dominio de terceros con un nombre especializado o de vanidad." cta="Ver glosario" href="/docs/es/glossary?term=custom+domains">dominios personalizados</Tooltip>.

La renovación de los certificados administrados por el cliente para los [dominios personalizados](/docs/es/customize/custom-domains) (con el formato *.&lt;CustomerName&gt;.com) es responsabilidad del cliente, quien debe obtenerlos y gestionarlos.
