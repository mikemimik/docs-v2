La opción de implementación Private Cloud en AWS es una instancia dedicada y administrada de la plataforma de identidad Auth0 que se ejecuta en Amazon Web Services. Ofrece aislamiento, mayor rendimiento, instancias de desarrollo independientes, varios complementos y mucho más.

<div id="operational-differences">
  ## Diferencias operativas
</div>

La siguiente tabla compara cada opción de implementación de Private Cloud en AWS.

| Funcionalidad                                  | Public Cloud                     | Private Cloud Basic on AWS * | Private Cloud Performance on AWS *                                                                                                                              |
| ---------------------------------------------- | -------------------------------- | ---------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Tenencia                                       | Múltiple                         | Única                        | Única                                                                                                                                                           |
| Solicitudes por segundo (múltiplos de 100 RPS) | 100 RPS (1x)                     | 100 RPS (1x)                 | 500* RPS (5x)<br />1,500* RPS (15x)<br />3,000* RPS (30x)<br />3,000* RPS (30x Burst)<br />6,000* RPS (60x)<br />6,000* RPS (60x Burst)<br />10,000* RPS (100x) |
| Acuerdo de nivel de servicio (SLA)             | 99.99%                           | 99.99%                       | 99.99%                                                                                                                                                          |
| Residencia de datos                            | Solo en regiones de nube pública | Sí                           | Sí                                                                                                                                                              |
| Entorno de desarrollo                          | No                               | No                           | 1                                                                                                                                                               |

*La capacidad de RPS se proporciona como orientación general. El rendimiento real puede variar según los tipos y el volumen de transacciones procesadas en el entorno de Private Cloud del cliente. Es posible que las transacciones de prueba utilizadas para los puntos de referencia de capacidad indicados no coincidan con los patrones de transacción ni con las cargas de trabajo específicas de cada cliente. Por ejemplo, se admite la siguiente capacidad al usar el [flujo de contraseña del propietario del recurso](/docs/es/get-started/authentication-and-authorization-flow/resource-owner-password-flow):

| Tier                  | RPS  |
| --------------------- | ---- |
| Basic                 | 55   |
| Performance 5x        | 180  |
| Performance 15x       | 600  |
| Performance 30x       | 1400 |
| Performance 30x Burst | 1400 |
| Performance 60x       | 3000 |
| Performance 60x Burst | 3000 |
| Performance 100x      | 5000 |

<div id="data-residency">
  ### Residencia de datos
</div>

Con Private Cloud en AWS, eliges la región en la que se almacenan tus datos. Auth0 puede proporcionar una lista de las regiones disponibles que usan varias [zonas de disponibilidad](https://aws.amazon.com/about-aws/global-infrastructure/regions_az/) para el despliegue. Puedes encontrar una lista de las regiones actuales en las que ofrecemos despliegues de Private Cloud en la información sobre subencargados del tratamiento publicada en nuestra [página Trust &amp; Compliance](https://www.okta.com/trustandcompliance/). En la mayoría de los casos, Okta despliega las copias de seguridad en la misma región de AWS seleccionada.

<div id="maximum-availability">
  ### Disponibilidad máxima
</div>

Las instancias de Private Cloud en AWS tienen un acuerdo de nivel de servicio (SLA) del 99,99 %. Los compromisos de disponibilidad no se aplican a las pruebas gratuitas, sandbox, Beta ni otros entornos de preproducción.

<div id="high-demand-apps">
  ### Aplicaciones de alta demanda
</div>

Si su aplicación requiere una cantidad considerablemente alta de solicitudes por segundo (RPS), quizá también le convenga considerar Private Cloud en AWS. Consulte las [políticas de límites de frecuencia](/docs/es/troubleshoot/customer-support/operational-policies/rate-limit-policy) para obtener más información sobre los límites de frecuencia estándar. Las implementaciones de Private Cloud en AWS tienen un límite de frecuencia de 100 RPS para Private Basic y límites de frecuencia ampliados de 500 RPS, 1,500 RPS, 3,000 RPS y 6,000 RPS para las opciones Private Performance.

<div id="additional-dev-environments">
  ## Entornos de desarrollo adicionales
</div>

Las implementaciones Performance de Private Cloud en AWS incluyen una instancia completamente aislada y actualizada de forma independiente para desarrollo y pruebas. Puede añadir entornos adicionales de preproducción para satisfacer las necesidades de su empresa.

Las solicitudes por segundo (RPS) garantizadas y el SLA no se aplican a los entornos no productivos.

<div id="bursty-traffic">
  ### Tráfico en ráfagas
</div>

Okta establece límites de frecuencia para las organizaciones en función del tráfico previsto y del nivel de RPS contratado. Si su organización experimenta un volumen de tráfico superior al esperado, este uso no planificado podría afectar a los usuarios finales. Private Cloud está diseñado para gestionar aumentos graduales en la tasa de transacciones (por ejemplo, un aumento de 100 RPS a 1000 RPS durante un período de 10 minutos) sin afectar al servicio. Sin embargo, los aumentos repentinos y pronunciados del tráfico (por ejemplo, pasar de 100 RPS a 1000 RPS en cuestión de segundos) podrían provocar inestabilidad en el servicio y un aumento de la latencia mientras la solución se ajusta para gestionar la nueva carga.

<div id="onboarding">
  ## Incorporación
</div>

Después de elegir Private Cloud en AWS, se seguirá un proceso de incorporación e implementación para configurar su entorno o entornos.

<div id="customer-onboarding-requirements">
  ### Requisitos de incorporación del cliente
</div>

Una vez firmado el contrato, le pediremos que nos facilite información clave sobre sus requisitos de incorporación, que luego validaremos.

<div id="kickoff-meeting">
  ### Reunión de inicio
</div>

Una vez que validemos los requisitos de su incorporación, celebraremos una reunión de inicio con usted para dar comienzo al proceso de implementación. Recomendamos encarecidamente que esta reunión se lleve a cabo a más tardar cinco (5) días después de la firma del contrato.

<div id="implementation">
  ### Implementación
</div>

Inmediatamente después de la validación del formulario de incorporación, comenzaremos el aprovisionamiento de su entorno.

Al final de este proceso, su entorno estará listo para la entrega y su implementación de Private Cloud en AWS quedará lista para su uso.

<div id="secure-outbound-networking">
  ## Redes seguras de salida
</div>

Algunas personalizaciones de la plataforma Auth0, como [Actions](/docs/es/customize/actions), [webhooks personalizados](/docs/es/customize/log-streams/custom-log-streams) y [scripts de acción de base de datos personalizados](/docs/es/authenticate/database-connections/custom-db/templates), le permiten realizar conexiones salientes desde la plataforma Auth0 a sus propios servicios. Con Private Cloud en AWS, puede establecer conectividad de red entre su despliegue de Private Cloud y sus propios servicios sin exponer sus datos a Internet.

Las redes seguras de salida utilizan AWS PrivateLink. Primero, comparta su servicio a través de PrivateLink [estableciendo un servicio de endpoint](https://docs.aws.amazon.com/vpc/latest/privatelink/create-endpoint-service.html) en su cuenta de AWS. El servicio subyacente puede ser un servicio nativo de AWS o un servicio que se ejecute en un centro de datos. El servicio debe estar en una VPC de la misma región de AWS que su despliegue de Private Cloud.

A continuación, configuraremos su despliegue de Private Cloud para que su servicio de endpoint esté disponible. Una vez que proporcione a Auth0 la información de su servicio de endpoint, integraremos el servicio con su despliegue y le indicaremos cómo acceder a él desde su código de personalización.

Para obtener más información sobre cómo configurar servicios de endpoint con PrivateLink, póngase en contacto con AWS. Para coordinar la incorporación del servicio con Auth0, envíe una solicitud al [Support Center](https://support.auth0.com/).

<div id="load-testing">
  ### Pruebas de carga
</div>

Esta política describe los requisitos necesarios para que Auth0 realice pruebas de carga para los clientes de Private Cloud en AWS que envíen una solicitud. Puede enviar una solicitud de pruebas de carga a través del [Support Center](https://support.auth0.com/). En el campo **Incidencia**, seleccione **Private Cloud Support Incident**.

Si compró un entorno dedicado para pruebas de carga, no hay límite en la frecuencia con la que puede realizar pruebas de carga. Los entornos estándar están limitados a dos (2) por año, siempre que se sigan los procedimientos adecuados para las pruebas de carga.

Para que la solicitud pueda ser aprobada, debe:

* Presentarse al menos dos (2) semanas antes de la fecha prevista para la prueba; en muchos casos, Auth0 recomienda avisar con un (1) mes de antelación para garantizar tiempo suficiente para una revisión exhaustiva y cualquier modificación necesaria.
* Recibir aprobación por escrito antes de que se realice cualquier prueba.
* Mantenerse dentro de nuestros [límites de frecuencia publicados del entorno Production](/docs/es/troubleshoot/customer-support/operational-policies/rate-limit-policy).

Si se solicitan cambios en la infraestructura, el costo se determinará en función de sus requisitos específicos.

<div id="testing-capacity-considerations">
  #### Consideraciones sobre la capacidad para pruebas
</div>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Los entornos adquiridos para pruebas de carga dedicadas no están sujetos a ningún acuerdo de nivel de servicio. Los problemas reportados en estos entornos tendrán menor prioridad que los problemas en producción.
</Callout>

Debe comenzar con una carga baja e ir aumentándola lentamente hasta que el entorno alcance su capacidad máxima. Si necesita una carga mayor de la que el entorno puede soportar, deberá aumentar el tamaño del entorno.

Los entornos de Private Cloud pueden ampliarse mediante una adenda contractual; póngase en contacto con su Account Executive y TAM para analizar esta compra.

| Suscripción                                    | Capacidad de prueba de carga | Incremento gradual |
| ---------------------------------------------- | ---------------------------- | ------------------ |
| Private Cloud Performance 500 RPS (5x)         | 325 RPS                      | 100 RPS/min        |
| Private Cloud Performance 1500 RPS (15x)       | 975 RPS                      | 100 RPS/min        |
| Private Cloud Performance 3000 RPS (30x)       | 1950 RPS                     | 100 RPS/min        |
| Private Cloud Performance 3000 RPS (30x Burst) | 1950 RPS                     | 100 RPS/min        |
| Private Cloud Performance 6000 RPS (60x)       | 3900 RPS                     | 100 RPS/min        |
| Private Cloud Performance 6000 RPS (60x Burst) | 3900 RPS                     | 100 RPS/min        |
| Private Cloud Performance 10,000 RPS (100x)    | 6500 RPS                     | 100 RPS/sec        |

Para obtener más información sobre las pruebas de carga en Private Cloud, consulte [Límites de solicitudes del entorno (Private Cloud Only)](/docs/es/troubleshoot/customer-support/operational-policies/rate-limit-policy#environment-request-limits-private-cloud-only-).

<div id="high-load-notifications">
  #### Notificaciones de alta carga
</div>

En períodos de alta carga previstos, debe informar a su equipo de cuenta con al menos 14 días de antelación al evento. La notificación permite probar adecuadamente los escenarios (si es posible) y coordinar el soporte reactivo para el evento.

<div id="penetration-testing">
  ### Pruebas de penetración
</div>

Para realizar una prueba de seguridad, notifíquenos con antelación a través del Auth0 Support Center. Auth0 requiere un aviso previo de al menos una semana (siete días) antes de la fecha de inicio prevista de la prueba.

Si la prueba se limita a su infraestructura (es decir, no se realizarán pruebas en los servicios de Auth0), no es necesario notificar a Auth0.

Para consultar la información que requerimos, vea nuestra [Política de pruebas de penetración](/docs/es/troubleshoot/customer-support/operational-policies/penetration-testing-policy).

<div id="failover-testing">
  ### Pruebas de conmutación por error
</div>

Esta política describe los requisitos necesarios para que Okta realice pruebas de conmutación por error para clientes de Private Cloud en la plataforma Auth0 de AWS o Azure con el add-on Geo Failover requerido. Puede enviar una solicitud de prueba de conmutación por error a través del [Support Center.](https://support.auth0.com/) En el campo Incidencia, seleccione Private Cloud Support Incident.

Para que la solicitud pueda considerarse para su aprobación, debe:

* Presentarse al menos dos (2) semanas antes de la fecha y hora deseadas para la prueba (en UTC). En muchos casos, Okta recomienda avisar con un (1) mes de antelación para garantizar tiempo suficiente para una revisión exhaustiva y cualquier modificación necesaria.
* Estar dentro del límite de (2) pruebas de conmutación por error por año calendario.
* Recibir aprobación por escrito antes de que se realice cualquier prueba.
* Especificar las ventanas (en UTC) tanto para la conmutación por error como para el retorno a la región principal, teniendo en cuenta que ambas ventanas supondrán un tiempo de inactividad de hasta 15 minutos.
* Incluir un punto de contacto designado con quien Okta coordinará toda la logística de las pruebas

Tenga en cuenta que Okta se reserva el derecho de sugerir ventanas alternativas para la conmutación por error y el retorno, en función de la disponibilidad del personal para realizar las pruebas solicitadas. Además, cualquier interrupción del servicio que se produzca como parte de los procedimientos de conmutación por error o retorno queda exenta de cualquier disposición del SLA.

<div id="certificate-renewal-process">
  ## Proceso de renovación de certificados
</div>

Los certificados administrados por Auth0 (con el formato *.auth0app.com) son responsabilidad de Auth0, tanto su obtención como su aplicación. Auth0 gestionará el proceso de extremo a extremo y te indicará cualquier acción necesaria.

La renovación de los certificados emitidos por Auth0 para los <Tooltip tip="Dominio personalizado: Dominio de terceros con un nombre especializado o personalizado." cta="Ver glosario" href="/docs/es/glossary?term=custom+domains">dominios personalizados</Tooltip> la gestiona Auth0.

La renovación de los certificados administrados por el cliente para los [dominios personalizados](/docs/es/customize/custom-domains) (con el formato *.&lt;CustomerName&gt;.com) es responsabilidad del cliente, que debe obtenerlos y gestionarlos.
