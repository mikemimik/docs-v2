La opción de implementación Private Cloud on Azure es una instancia dedicada y administrada de la plataforma de identidad Auth0 que se ejecuta en Microsoft Azure. Ofrece aislamiento, mayor rendimiento, instancias de desarrollo independientes, varios complementos y mucho más.

<div id="operational-differences">
  ## Diferencias operativas
</div>

La siguiente tabla compara cada opción de implementación de Private Cloud on Azure.

| Funcionalidad                      | Public Cloud                     | Private Cloud on Azure Basic * | Private Cloud on Azure Performance *                                                     |
| ---------------------------------- | -------------------------------- | ------------------------------ | ---------------------------------------------------------------------------------------- |
| Tipo de tenancy                    | Múltiple                         | Única                          | Única                                                                                    |
| Solicitudes por segundo (RPS)      | 100                              | 100                            | 500* RPS (5x)<br />1500* RPS (15x)<br />3000* RPS (30x)<br /><br />3000* RPS (30x Burst) |
| Acuerdo de nivel de servicio (SLA) | 99.99%                           | 99.99%                         | 99.99%                                                                                   |
| Residencia de datos                | Solo en regiones de nube pública | Sí                             | Sí                                                                                       |
| Entorno de desarrollo              | No                               | No                             | 1                                                                                        |

*La capacidad de solicitudes por segundo (RPS) se proporciona como guía general. El rendimiento real puede variar según los tipos y el volumen de transacciones procesadas en el entorno de Private Cloud del cliente. Las transacciones de prueba utilizadas para los valores de referencia de capacidad indicados pueden no coincidir con los patrones de transacción y las cargas de trabajo específicos de cada cliente. Por ejemplo, se admite la siguiente capacidad al usar el [flujo de contraseña del propietario del recurso](/docs/es/get-started/authentication-and-authorization-flow/resource-owner-password-flow):

| Tier                  | RPS  |
| --------------------- | ---- |
| Basic                 | 55   |
| Performance 5x        | 180  |
| Performance 15x       | 600  |
| Performance 30x       | 1400 |
| Performance 30x Burst | 1400 |

<div id="data-residency">
  ### Residencia de datos
</div>

Con Private Cloud on Azure, eliges la región en la que se almacenan tus datos. Auth0 puede proporcionarte una lista de las regiones disponibles que usan tres [zonas de disponibilidad](https://azure.microsoft.com/en-us/global-infrastructure/geographies/) para el despliegue. Puedes consultar la lista de regiones actuales en las que ofrecemos despliegues de Private Cloud en nuestra Información sobre subencargados del tratamiento, publicada en nuestra [página de Trust &amp; Compliance](https://www.okta.com/trustandcompliance/). En la mayoría de los casos, Okta despliega las copias de seguridad en la misma región de Azure seleccionada.

<div id="maximum-availability">
  ### Disponibilidad máxima
</div>

Las instancias de Private Cloud on Azure cuentan con un acuerdo de nivel de servicio (SLA) del 99.99%.  Los compromisos de disponibilidad no se aplican a las pruebas gratuitas, sandbox, beta ni a otros entornos de preproducción.

<div id="additional-dev-environments">
  ## Entornos de desarrollo adicionales
</div>

Las implementaciones de Private Cloud on Azure Performance incluyen una instancia totalmente aislada y actualizada de forma independiente para desarrollo y pruebas. Puede añadir entornos de preproducción adicionales para satisfacer sus necesidades empresariales.

Las solicitudes por segundo garantizadas (RPS) y el SLA no se aplican a los entornos no productivos.

<div id="bursty-traffic">
  ### Picos de tráfico
</div>

Okta proporciona límites de frecuencia para las organizaciones en función del tráfico que se espera que tengan y del nivel de RPS contratado. Si su organización experimenta un tráfico superior al previsto, este uso no planificado podría afectar a los usuarios finales. La oferta de Private Cloud está diseñada para gestionar aumentos graduales en la tasa de transacciones (por ejemplo, un incremento de 100 RPS a 1000 RPS durante un período de 10 minutos) sin que se produzca ningún impacto en el servicio. Sin embargo, los picos de tráfico repentinos y pronunciados (por ejemplo, un aumento de 100 RPS a 1000RPS en cuestión de segundos) podrían provocar inestabilidad en el servicio y un aumento de la latencia mientras la solución se ajusta para gestionar la nueva carga.

<div id="onboarding">
  ## Incorporación
</div>

Después de elegir Private Cloud on Azure, se seguirá un proceso de incorporación y despliegue para configurar sus entornos.

<div id="customer-onboarding-requirements">
  ### Requisitos de incorporación del cliente
</div>

Tras la firma del contrato, le pediremos que nos facilite información clave sobre sus requisitos de incorporación, que posteriormente validaremos.

<div id="kickoff-meeting">
  ### Reunión de arranque
</div>

Una vez que validemos sus requisitos de incorporación, celebraremos una reunión de arranque con usted para dar inicio al proceso de implementación. Recomendamos encarecidamente que esta reunión se celebre en un plazo máximo de cinco (5) días después de la firma del contrato.

<div id="implementation">
  ### Implementación
</div>

Inmediatamente después de la validación del formulario de incorporación, comenzaremos el aprovisionamiento de tu entorno.

Al final de este proceso, todo estará listo para la entrega del entorno y tu despliegue de Private Cloud on Azure estará listo para usarse.

<div id="secure-outbound-networking">
  ## Redes salientes seguras
</div>

Algunas personalizaciones de la plataforma Auth0, incluidas [Actions](/docs/es/customize/actions), los [webhooks personalizados](/docs/es/customize/log-streams/custom-log-streams) y los [scripts de acción de base de datos personalizados](/docs/es/authenticate/database-connections/custom-db/templates), permiten establecer conexiones salientes seguras desde la plataforma Auth0 hacia sus propios servicios y crear conectividad de red entre su implementación de Private Cloud y esos servicios.

Las redes salientes seguras usan Azure Private Link al [establecer un servicio de endpoint](https://learn.microsoft.com/en-us/azure/private-link/private-link-service-overview) en su cuenta de Azure. El servicio subyacente puede ser un servicio nativo de Azure o un servicio que se ejecute en un centro de datos, y debe estar en una Azure Virtual Network de la misma región de Azure que su implementación de Private Cloud.

Después de configurar su implementación de Private Cloud para que su servicio de endpoint esté disponible, proporcione a Auth0 la información de ese servicio para que podamos integrarlo con su implementación y brindarle información sobre cómo acceder a él desde su código de personalización.

Para obtener más información sobre cómo configurar servicios de endpoint con Private Link, póngase en contacto con Azure. Para coordinar la incorporación del servicio con Auth0, póngase en contacto con el [Support Center](https://support.auth0.com/) de Auth0.

<div id="load-testing">
  ### Pruebas de carga
</div>

Esta política describe los requisitos necesarios para que Auth0 realice pruebas de carga para clientes de Private Cloud on Azure que presenten una solicitud. Puede enviar una solicitud de prueba de carga a través del [Support Center](https://support.auth0.com/). En el campo Issue, seleccione **Private Cloud Support Incident**.

Si compró un entorno dedicado para pruebas de carga, no hay límite en la frecuencia con la que puede ejecutarlas. Los entornos estándar están limitados a dos (2) por año, siempre que se sigan los procedimientos adecuados para las pruebas de carga.

Para que la solicitud pueda considerarse para su aprobación, debe:

* Presentarse al menos dos (2) semanas antes de la fecha de prueba deseada; en muchos casos, Auth0 recomienda avisar con un (1) mes de antelación para garantizar que haya tiempo suficiente para una revisión exhaustiva y cualquier modificación necesaria.
* Recibir aprobación por escrito antes de que se realice cualquier prueba.
* Ajustarse a nuestros [límites de frecuencia de producción publicados](/docs/es/troubleshoot/customer-support/operational-policies/rate-limit-policy).

Si se solicitan cambios en la infraestructura, el costo se determinará en función de sus requisitos específicos.

<div id="testing-capacity-considerations">
  #### Consideraciones de capacidad para pruebas de carga
</div>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Los entornos adquiridos para pruebas de carga dedicadas no están sujetos a ningún acuerdo de nivel de servicio. Las incidencias notificadas en estos entornos tendrán menor prioridad que las incidencias de producción.
</Callout>

Debe comenzar con una carga baja e incrementarla lentamente hasta que el entorno alcance su capacidad máxima. Si necesita una carga superior a la que el entorno puede soportar, deberá aumentarse el tamaño del entorno.

Los entornos de Private Cloud pueden ampliarse mediante una adenda contractual; póngase en contacto con su ejecutivo de cuenta y su TAM para hablar sobre esta compra.

| Suscripción                                    | Capacidad para pruebas de carga | Aumento gradual |
| ---------------------------------------------- | ------------------------------- | --------------- |
| Private Cloud Performance 500 RPS (5x)         | 325 RPS                         | 100 RPS/min     |
| Private Cloud Performance 1500 RPS (15x)       | 975 RPS                         | 100 RPS/min     |
| Private Cloud Performance 3000 RPS (30x)       | 1950 RPS                        | 100 RPS/min     |
| Private Cloud Performance 3000 RPS (30x Burst) | 1950 RPS                        | 100 RPS/min     |

Para obtener más información sobre las pruebas de carga en Private Cloud, consulte [Límites de solicitudes del entorno (Private Cloud Only)](/docs/es/troubleshoot/customer-support/operational-policies/rate-limit-policy#environment-request-limits-private-cloud-only-).

<div id="high-load-notifications">
  #### Notificaciones de alta carga
</div>

En caso de períodos de alta carga previstos, debe informar a su equipo de cuentas con al menos 14 días de antelación al evento. La notificación permite probar adecuadamente los escenarios (si es posible) y preparar el soporte reactivo para el evento.

<div id="penetration-testing">
  ### Pruebas de penetración
</div>

Para realizar una prueba de seguridad, notifíquenos con antelación a través de Auth0 Support Center. Auth0 exige un aviso previo de al menos una semana (siete días) antes de la fecha de inicio prevista de la prueba.

Si la prueba se limita a su infraestructura (es decir, no se realizarán pruebas sobre los servicios de Auth0), no es necesario notificar a Auth0.

Para conocer la información que necesitamos, consulte nuestra [Política de pruebas de penetración](/docs/es/troubleshoot/customer-support/operational-policies/penetration-testing-policy).

<div id="failover-testing">
  ### Pruebas de conmutación por error
</div>

Esta política describe los requisitos necesarios para que Okta realice pruebas de conmutación por error para clientes de Private Cloud en la plataforma Auth0 de AWS o Azure con el add-on Geo Failover requerido. Puede enviar una solicitud de prueba de conmutación por error a través del [Support Center.](https://support.auth0.com/) En el campo Issue, seleccione Private Cloud Support Incident.

Para que la solicitud pueda aprobarse, debe:

* Presentarse al menos dos (2) semanas antes de la fecha y hora deseadas para la prueba (en UTC). En muchos casos, Okta recomienda avisar con un (1) mes de anticipación para garantizar tiempo suficiente para una revisión exhaustiva y cualquier modificación necesaria.
* No superar el límite de (2) pruebas de conmutación por error por año calendario.
* Recibir aprobación por escrito antes de que se realice cualquier prueba.
* Especificar ventanas (en UTC)  tanto para la conmutación por error como para la reversión a la región primaria, teniendo en cuenta que ambas ventanas darán lugar a un tiempo de inactividad de hasta 15 minutos.
* Incluir un punto de contacto designado con quien Okta coordinará toda la logística de las pruebas

Tenga en cuenta que Okta se reserva el derecho de sugerir ventanas alternativas para la conmutación por error y la reversión, en función de la disponibilidad del personal para realizar las pruebas solicitadas. Además, cualquier interrupción del servicio que se produzca como parte de los procedimientos de conmutación por error o reversión queda exenta de cualquier disposición del SLA.

<div id="certificate-renewal-process">
  ## Proceso de renovación de certificados
</div>

Los certificados gestionados por Auth0 (con el formato *.auth0app.com) son responsabilidad de Auth0, tanto su obtención como su instalación. Auth0 gestionará el proceso de extremo a extremo y le indicará si se requiere alguna acción por su parte.

Auth0 se encarga de la renovación de los certificados emitidos por Auth0 para <Tooltip tip="Dominio personalizado: Dominio de terceros con un nombre especializado o de vanidad." cta="Ver glosario" href="/docs/es/glossary?term=custom+domains">dominios personalizados</Tooltip>.

La renovación de los certificados gestionados por el cliente para [dominios personalizados](/docs/es/customize/custom-domains) (con el formato *.&lt;CustomerName&gt;.com) es responsabilidad del cliente, que debe obtenerlos y gestionarlos.
