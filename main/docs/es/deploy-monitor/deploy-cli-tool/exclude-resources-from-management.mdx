---
description: Aprende a excluir recursos de la administración al usar Auth0 Deploy CLI.
title: Excluir recursos de la administración
---

En algunos casos, puede resultarte útil excluir ciertos recursos de la administración. Esto puede deberse a que tu tenant tiene una gran cantidad de un tipo de recurso en particular y administrarlos supone una carga operativa, o a que tu workflow de desarrollo solo abarca un subconjunto específico de recursos y quieres omitir todos los demás por motivos de rendimiento. En cualquier caso, al usar Deploy CLI tienes varias opciones para excluir recursos.

<div id="exclude-entire-resources-by-type">
  ## Excluir tipos de recursos completos
</div>

En tenants más complejos, puede que quieras omitir tipos de recursos completos. Por ejemplo:

* Tenant empresarial con miles de organizaciones, donde gestionarlas todas supondría una carga operativa.
* El proceso de CI/CD solo se centra en gestionar roles, y quieres excluir todos los demás.
* El desarrollo de la funcionalidad se centra en Hook, y quieres excluir temporalmente todos los demás para optimizar el rendimiento.

Este tipo de exclusión se expresa pasando un arreglo de nombres de recursos a cualquiera de las propiedades de configuración `AUTH0_EXCLUDED` o `AUTH0_INCLUDED_ONLY`. La propiedad de configuración `AUTH0_EXCLUDED` excluye únicamente los tipos de recursos que se le proporcionan. En cambio, la propiedad `AUTH0_INCLUDED_ONLY` excluye todos los tipos de recursos excepto los definidos. La exclusión funciona en ambos sentidos, es decir, tanto al exportar desde Auth0 como al importar a Auth0, independientemente de si existen o no archivos de configuración de recursos.

Todos los valores de recurso admitidos para la exclusión: `actions`, `attackProtection`, `branding`, `clientGrants`, `clients`, `connections`, `customDomains`, `databases`, `emailProvider`, `emailTemplates`, `guardianFactorProviders`, `guardianFactorTemplates`, `guardianFactors`, `guardianPhoneFactorMessageTypes`, `guardianPhoneFactorSelectedProvider`, `guardianPolicies`, `logStreams`, `migrations`, `organizations`, `pages`, `prompts`, `resourceServers`, `roles`, `tenant`, `triggers`.

<div id="exclusion-example">
  ### Ejemplo de exclusión
</div>

El siguiente ejemplo muestra cómo excluir `clients`, `connections`, `databases` y `organizations` para que no los gestione Deploy CLI.

```json lines
{
  "AUTH0_DOMAIN": "example-site.us.auth0.com",
  "AUTH0_CLIENT_ID": "<YOUR_AUTH0_CLIENT_ID>",
  "AUTH0_EXCLUDED": ["clients", "connections", "databases", "organizations"]
}
```

<div id="inclusion-example">
  ### Ejemplo de inclusión
</div>

El siguiente ejemplo muestra cómo especificar que solo se administren `actions`, `clients` y `connections` mediante Deploy CLI.

```json lines
{
  "AUTH0_DOMAIN": "example-site.us.auth0.com",
  "AUTH0_CLIENT_ID": "<YOUR_AUTH0_CLIENT_ID>",
  "AUTH0_INCLUDED_ONLY": ["actions", "clients", "connections"]
}
```

<div id="exclude-single-resources-by-id">
  ## Excluir recursos específicos por ID
</div>

Algunos tipos de recursos permiten excluir recursos individuales por ID. Esto resulta útil si trabajas en un contexto multientorno y quieres omitir de tus entornos de nivel inferior un recurso específico de producción.

Este método es compatible con rules, clientes, bases de datos, conexiones y <Tooltip tip="Resource Server: servidor que aloja recursos protegidos. Los servidores de recursos aceptan y responden a solicitudes de recursos protegidos." cta="Ver glosario" href="/docs/es/glossary?term=resource+servers">servidores de recursos</Tooltip> mediante los valores de configuración `AUTH0_EXCLUDED_RULES`, `AUTH0_EXCLUDED_CLIENTS`, `AUTH0_EXCLUDED_DATABASES`, `AUTH0_EXCLUDED_CONNECTIONS` y `AUTH0_EXCLUDED_RESOURCE_SERVERS`, respectivamente.

### Ejemplo

```json lines
{
  "AUTH0_DOMAIN": "example-site.us.auth0.com",
  "AUTH0_CLIENT_ID": "<YOUR_AUTH0_CLIENT_ID>",
  "AUTH0_EXCLUDED_CLIENTS": ["Your Application Name"],
  "AUTH0_EXCLUDED_CONNECTIONS": ["con_O1H3KyRMFP1IWRq3", "con_9avEYuj19ihqKBOs"]
}
```

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Se está considerando marcar como obsoleta la exclusión de recursos por ID en futuras versiones mayores. Consulta la [propuesta de exclusión de recursos](https://github.com/auth0/auth0-deploy-cli/blob/6381ce6efa8cd59e7a5c8d37e8915b0989d677b9/docs/excluding-from-management.md?plain=1#L51) para obtener más detalles.
</Callout>

<div id="omission-and-empty-states">
  ## Omisión y estados vacíos
</div>

Además de excluir recursos, lo que obliga a ignorar las configuraciones en ambos sentidos, Auth0 Deploy CLI admite dos conceptos similares: la omisión y los estados vacíos.

<div id="omission">
  ### Omisión
</div>

La configuración de recursos que no esté presente, ya sea de forma intencional o accidental, se omitirá durante la importación. Por ejemplo, si se eliminara la configuración de tus recursos, se omitiría durante la importación y no modificaría el estado del tenant remoto.

No existe el concepto de omisión en la exportación. Salvo que se excluyan explícitamente, todas las configuraciones de tu tenant se escribirán en archivos de configuración de recursos.

<div id="example">
  #### Ejemplo
</div>

```yaml lines
roles: # la configuración de roles no se omite
  - name: Admin
    description: Can read and write things
    permissions: []
  - name: Reader
    description: Can only read things
    permissions: []
# La omisión de todas las demás configuraciones significa que serán omitidas
```

<div id="empty">
  ### Vacío
</div>

Configuración de recursos definida explícitamente como vacía. En las configuraciones basadas en conjuntos, como hooks, organizaciones y actions, establecer estas configuraciones como un conjunto vacío expresa la intención de vaciar esos recursos. Esto indicaría una eliminación, siempre que la propiedad de configuración de eliminación `AUTH0_ALLOW_DELETE` esté habilitada. Para obtener más información sobre esta propiedad, consulta [Configurar Deploy CLI](/docs/es/deploy-monitor/deploy-cli-tool/configure-the-deploy-cli).

En configuraciones de recursos no basadas en conjuntos, como tenant y branding, el concepto de vacío no se aplica y no activará ninguna eliminación ni supresión.

<div id="example-of-emptiness">
  #### Ejemplo de vacío
</div>

```yaml lines
hooks: [] # Hooks vacíos
connections: [] # Conexiones vacías
tenant: {} # En la práctica no realiza ninguna acción, el vacío no aplica a configuraciones de recursos que no son de tipo conjunto
```
