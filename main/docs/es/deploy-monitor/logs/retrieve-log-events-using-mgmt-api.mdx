---
title: Recuperar registros con la Management API
description: Aprende a recuperar registros con el endpoint get_logs de la Auth0 Management API mediante checkpoint o criterios de búsqueda.
---

Puedes usar la <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip> para recuperar hasta 100 registros por solicitud mediante el endpoint [/get&#95;logs](https://auth0.com/docs/api/management/v2/#!/Logs/get_logs), que admite dos tipos de consumo:

* **Mediante checkpoint**: Recomendado si quieres exportar registros a un servicio externo de análisis de datos.
* **Por criterios de búsqueda**: Lo utiliza el Dashboard.

Los registros almacenados por Auth0 están sujetos a un período de retención en función de tu suscripción de Auth0. Para obtener más información, consulta [Log Data Retention](/docs/es/deploy-monitor/logs/log-data-retention).

También puedes usar estos endpoints para buscar registros:

* `/api/v2/logs/{id}`: Recupera la entrada de registro asociada al ID de registro proporcionado.
* `/api/v2/users/{user_id}/logs`: Recupera registros de un user ID específico.

<div id="retrieve-logs-by-checkpoint">
  ## Recuperar registros por checkpoint
</div>

Este método le permite recuperar registros posteriores a un `log_id` determinado. Se
usa habitualmente para enviar sus registros a un sistema externo, como Splunk, Datadog,
Application Insights, Mixpanel, entre otros.

Para buscar por checkpoint, use los siguientes parámetros:

| Parámetro | Descripción                                                                                                                      |
| --------- | -------------------------------------------------------------------------------------------------------------------------------- |
| `from`    | ID del evento de log a partir del cual se empezarán a recuperar registros. Puede limitar la cantidad de registros con el parámetro `take`. |
| `take`    | La cantidad total de entradas que se recuperarán al usar el parámetro `from`.                                                    |

Se ignorará cualquier parámetro distinto de `from` y `take`. Los registros se ordenan por `log_id`.

<div id="semantics">
  #### Semántica
</div>

La recuperación de registros por checkpoint tiene la siguiente semántica:

* Se devolverán todos los registros disponibles de acuerdo con los permisos de tu rol
* Los registros se ordenan por `log_id`, en lugar de por la hora de generación del registro, para evitar omitir registros debido a retrasos en el sistema de Auth0
* La paginación no tiene límite, a diferencia del límite de `1,000` registros al recuperar registros por criterios de búsqueda

<div id="pagination">
  #### Paginación
</div>

Al recuperar registros mediante checkpoint, solo la primera solicitud debe hacerse
manualmente. Cada solicitud posterior debe realizarse usando la URL de la relación `next`
dentro del encabezado `Link`. Esto simplifica la integración con la
API y permite que Auth0 haga optimizaciones sin afectar la experiencia del
cliente.

<div id="example">
  #### Ejemplo
</div>

Supongamos que el nombre de tu tenant en `us` es `acme`. Para empezar a recuperar registros por
checkpoint:

1. Obtén el `log_id` de cualquier registro de tu sistema.
   Ejemplo: `90020211201185132572079310688835925971249535794074878050`.
2. Haz una solicitud a `https://TENANT_NAME.LOCALITY.auth0.com/api/v2/logs?from=LOG_ID&take=100` para pedir los siguientes `100` registros posteriores al `LOG_ID` indicado.
3. La respuesta HTTP contendrá

   1. Un arreglo JSON de `0` a `100` registros posteriores al `LOG_ID` indicado
   2. Una cabecera HTTP `Link` con una relación `next`
4. Si el conjunto de resultados está vacío, recibirás un arreglo vacío y la relación `next` coincidirá con la misma URL que usaste en la solicitud más reciente

<div id="retrieve-logs-by-search-criteria">
  ## Obtener registros según criterios de búsqueda
</div>

Este método le permite obtener los registros que coincidan con los criterios de búsqueda especificados o, si no se usa ningún criterio, enumerar todos los registros. Para buscar por criterios, use los siguientes parámetros:

| Parámetro        | Descripción                                                                                                                                              |
| ---------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `q`              | Criterios de búsqueda mediante la sintaxis de Query String.                                                                                              |
| `page`           | El número de página con índice inicial en cero.                                                                                                          |
| `per_page`       | El número de entradas por página.                                                                                                                        |
| `sort`           | El campo que se usará para ordenar. Use `field:order`, donde `order` es `1` para orden ascendente y `-1` para orden descendente. Por ejemplo, `date:-1`. |
| `fields`         | Una lista de campos separada por comas para incluir o excluir del resultado (según `include_fields`). Déjelo vacío para obtener todos los campos.        |
| `include_fields` | `true` si los campos especificados deben incluirse en el resultado; `false` en caso contrario. El valor predeterminado es `true`.                        |

Para ver una lista de campos y parámetros de ordenación que pueden usarse en la consulta de búsqueda, lea [sintaxis de las consultas de búsqueda en registros](/docs/es/deploy-monitor/logs/log-search-query-syntax).

<div id="limitations">
  ## Limitaciones
</div>

Además del límite de 100 registros por solicitud al recuperar registros, solo puede paginar hasta 1.000 resultados de búsqueda.

Si recibe el error `414 Request-URI Too Large`, significa que su query string supera la longitud admitida. En ese caso, refine la búsqueda.

<Card title="Limitaciones de los campos de búsqueda en Private Cloud">
  Solo se puede buscar en los siguientes campos en Private Cloud:

  * `user`
  * `connection`
  * `application`
  * `type`
  * `ip`

  Use comillas dobles para realizar búsquedas exactas (por ejemplo, `application:"test"` buscará todos los eventos de log específicos de la aplicación llamada `test`, mientras que `application:test` buscará eventos de log de aplicaciones que tengan &quot;test&quot; en su nombre.
</Card>
