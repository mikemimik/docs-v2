title: Monitoreo de aplicaciones
---

Tus aplicaciones y servicios dependen de Auth0. Monitorear el estado de Auth0 te permite informar a los clientes sobre errores específicos o reducir el impacto en los usuarios en caso de que haya un problema con Auth0.

Hay muchas formas de monitorear Auth0, y cada una complementa a las demás. Debes elegir la que mejor se adapte a tus necesidades y posibilidades de inversión.

<div id="synthetic-transactions">
  ## Transacciones sintéticas
</div>

La forma más sencilla de supervisar Auth0:

* Configura una solicitud periódica para realizar una transacción de autenticación.
* Si la solicitud se realiza correctamente, Auth0 funciona bien.
* Si la solicitud falla, esto puede indicar:

  * una incidencia con Auth0
  * una incidencia específica del tenant usado para la transacción sintética
  * o simplemente una única solicitud fallida.

Para las transacciones sintéticas, usa un entorno lo más parecido posible a la configuración de tu tenant de producción. Incluso podrías usar el mismo tenant de producción. Como configurar transacciones sintéticas con flujos de redirección y proveedores externos puede ser complicado, se recomienda usar la [concesión de contraseña del propietario del recurso](/docs/es/get-started/authentication-and-authorization-flow/resource-owner-password-flow/call-your-api-using-resource-owner-password-flow). Este flujo no implica redirecciones del navegador ni requiere una UI.

Si estás usando [rules](/docs/es/customize/rules) o [conexiones de base de datos personalizadas](/docs/es/authenticate/database-connections/custom-db) u otros puntos de extensibilidad, las transacciones sintéticas deben configurarse para usar las rules y/o los script de Custom DB y así garantizar que ese aspecto del sistema funcione.

Herramientas como [Pingdom](http://pingdom.com/) hacen que configurar transacciones sintéticas sea muy sencillo.

<div id="check-period">
  ### Intervalo de comprobación
</div>

Recomendamos ejecutar transacciones sintéticas a intervalos de un minuto. Con este sencillo enfoque, esa frecuencia no consumirá gran parte de la cuota del límite de frecuencia de Auth0 y, al mismo tiempo, proporcionará respuestas oportunas.

<div id="limitations-of-synthetic-transactions">
  ### Limitaciones de las transacciones sintéticas
</div>

Las transacciones sintéticas son una forma sencilla y económica de monitorear el estado de un tenant de Auth0. Sin embargo, tienen algunas limitaciones:

* Las transacciones sintéticas no representan la experiencia de tu usuario final. En su lugar, te proporcionan una métrica indirecta.
* Es posible que las transacciones sintéticas no utilicen los mismos flujos que tus usuarios.
* Carecen de atomicidad (normalmente se ejecutan una vez por minuto) y no informan sobre errores que tus usuarios finales podrían haber experimentado.

Si te interesa obtener datos más granulares, consulta [Seguimiento de errores](#error-tracking) y [Métricas y registros](#metrics-and-logs).

<div id="error-tracking">
  ## Seguimiento de errores
</div>

Este enfoque es útil para rastrear errores en las llamadas existentes a Auth0. Consiste en informar de los errores cada vez que falla una llamada a Auth0. [Sentry](https://sentry.io/) es una herramienta que se usa habitualmente en estos casos y que funciona tanto en escenarios de frontend como de backend.

Este enfoque es útil porque te permite conocer los errores reales que están experimentando tus usuarios finales. Sin embargo, como solo estás rastreando errores (y no todas las solicitudes), no es posible hacerse una idea precisa de cuántos usuarios finales están afectados: ¿es el 1 % o el 5 %? Además, no requiere que configures una &quot;llamada sintética&quot; aparte, que podría consumir parte de tu cuota de límite de frecuencia, especialmente si está mal configurada.

<div id="metrics-and-logs">
  ## Métricas y registros
</div>

Este enfoque es útil si las llamadas a Auth0 se realizan desde un backend que controlas. Este es el caso de:

* La mayoría de las llamadas a la [Management API de Auth0](https://auth0.com/docs/api/management/v2)
* Las llamadas a la [Authentication API](https://auth0.com/docs/api/authentication) desde aplicaciones web clásicas y aplicaciones de máquina a máquina ([más información sobre los tipos de cliente](/docs/es/get-started/applications)).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Algunos productos de observabilidad también te permiten reportar métricas/registros desde aplicaciones frontend y móviles.
</Callout>

El enfoque consiste en usar métricas y/o registros para hacer un seguimiento de las tasas de error en las llamadas a Auth0. Las métricas y los registros reflejan tasas de error que describen con precisión lo que experimentan los usuarios finales, sin necesidad de configurar una &quot;llamada sintética&quot; independiente, que podría consumir parte de tu cuota del límite de frecuencia, especialmente si está mal configurada.

<div id="alerting">
  ## Alertas
</div>

Independientemente de los enfoques de monitoreo que uses, es común recibir una alerta o aviso cuando los errores se producen con cierta frecuencia. Esa frecuencia depende de tu aplicación.

Cuando tu equipo reciba una alerta de Auth0, te recomendamos agregar un enlace a [consultar el estado de Auth0](/docs/es/deploy-monitor/monitor/check-auth0-status) al payload/Playbook de la alerta. Esto les permitirá consultar rápidamente el canal oficial de estado de Auth0 para ver si el problema proviene de Auth0 o de tu aplicación/servicio.
