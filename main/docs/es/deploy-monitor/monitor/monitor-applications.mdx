Tus aplicaciones y servicios dependen de Auth0. Monitorear el estado de Auth0 te permite informar a los clientes sobre errores específicos o reducir el impacto en los usuarios en caso de que haya un problema con Auth0.

Hay muchas formas de monitorear Auth0 y cada enfoque complementa a los demás. Debes elegir en función de tus necesidades y posibilidades de inversión.

<div id="synthetic-transactions">
  ## Transacciones sintéticas
</div>

La forma más sencilla de monitorizar Auth0:

* Configura una solicitud periódica para realizar una transacción de autenticación.
* Si la solicitud se completa correctamente, Auth0 funciona bien.
* Si la solicitud falla, esto puede indicar:

  * una incidencia en Auth0
  * una incidencia específica del tenant utilizado para la transacción sintética
  * o simplemente una solicitud fallida aislada.

Para las transacciones sintéticas, utiliza un entorno lo más parecido posible a la configuración de tu tenant de producción. Incluso podrías usar el mismo tenant de producción. Dado que configurar transacciones sintéticas con flujos de redirección y proveedores externos puede resultar complicado, se recomienda usar [Resource Owner Password Grant](/docs/es/get-started/authentication-and-authorization-flow/resource-owner-password-flow/call-your-api-using-resource-owner-password-flow). Este flujo no implica redirecciones del navegador ni requiere una UI.

Si utilizas [reglas](/docs/es/customize/rules) o [conexiones de base de datos personalizadas](/docs/es/authenticate/database-connections/custom-db) u otros puntos de extensibilidad, las transacciones sintéticas deben configurarse para utilizar las reglas y/o los scripts de Custom DB, a fin de garantizar que ese aspecto del sistema funcione.

Herramientas como [Pingdom](http://pingdom.com/) hacen que configurar transacciones sintéticas sea sencillo.

<div id="check-period">
  ### Intervalo de comprobación
</div>

Recomendamos ejecutar transacciones sintéticas a intervalos de un minuto. Con este enfoque sencillo, esa frecuencia no consumirá gran parte de la cuota del límite de frecuencia de Auth0 y, al mismo tiempo, te permitirá obtener respuestas a tiempo.

<div id="limitations-of-synthetic-transactions">
  ### Limitaciones de las transacciones sintéticas
</div>

Las transacciones sintéticas son una forma sencilla y económica de monitorear el estado de un tenant de Auth0. Sin embargo, tienen algunas limitaciones:

* Las transacciones sintéticas no representan la experiencia de tu usuario final. En cambio, te proporcionan una métrica indirecta de ella.
* Las transacciones sintéticas podrían no usar los mismos flujos que tus usuarios.
* Carecen de atomicidad (normalmente se ejecutan una vez por minuto) y no informan sobre los errores que tus usuarios finales podrían haber visto.

Si te interesa obtener datos más detallados, consulta [Error Tracking](#error-tracking) y [Métricas y registros](#metrics-and-logs).

<div id="error-tracking">
  ## Error tracking
</div>

Este enfoque es útil para rastrear errores en las llamadas existentes a Auth0. Consiste en informar de los errores cada vez que falla una llamada a Auth0. [Sentry](https://sentry.io/) es una herramienta de uso habitual en estos casos, y funciona tanto en escenarios de frontend como de backend.

Este enfoque es útil porque te permite conocer los errores reales que están experimentando tus usuarios finales. Sin embargo, como solo rastreas errores (y no todas las solicitudes), no es posible tener una idea precisa de cuántos usuarios finales se ven afectados: ¿el 1 % o el 5 %? Además, no requiere que configures una &quot;llamada sintética&quot; independiente, que podría consumir parte de tu cuota de límite de frecuencia, especialmente si está mal configurada.

<div id="metrics-and-logs">
  ## Métricas y registros
</div>

Este enfoque es útil si las llamadas a Auth0 se realizan desde un backend que controlas. Este es el caso de:

* La mayoría de las llamadas a la [Management API de Auth0](https://auth0.com/docs/api/management/v2)
* Las llamadas a la [Authentication API](https://auth0.com/docs/api/authentication) desde aplicaciones web clásicas y aplicaciones máquina a máquina ([más información sobre los tipos de cliente](/docs/es/get-started/applications)).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Algunos productos de observabilidad también te permiten enviar métricas/registros desde aplicaciones frontend y móviles.
</Callout>

Este enfoque consiste en usar métricas y/o registros para hacer un seguimiento de las tasas de error en las llamadas a Auth0. Las métricas y los registros reflejan tasas de error que describen con precisión lo que experimentan los usuarios finales, sin necesidad de configurar una &quot;llamada sintética&quot; independiente, que podría consumir parte de tu cuota del límite de frecuencia, especialmente si está mal configurada.

<div id="alerting">
  ## Alertas
</div>

Independientemente de los enfoques de monitoreo que uses, es común recibir una alerta o notificación cuando se producen errores con cierta frecuencia. Esa frecuencia depende de tu aplicación.

Cuando tu equipo recibe una alerta de Auth0, te recomendamos agregar un enlace a [Verificar el estado de Auth0](/docs/es/deploy-monitor/monitor/check-auth0-status) en el payload/Playbook de la alerta. Esto les permitirá consultar rápidamente el canal oficial de estado de Auth0 para ver si el problema proviene de Auth0 o de tu aplicación/servicio.
