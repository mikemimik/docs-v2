<div id="metric-streams-beta">
  # Flujo de métricas (Beta)
</div>

<Warning>
  Flujo de métricas actualmente está disponible en Beta. Para usar esta funcionalidad, debes tener un plan Enterprise. Al usar esta funcionalidad, aceptas los términos aplicables de la prueba gratuita incluidos en el [Master Subscription Agreement](https://www.okta.com/legal) de Okta. Para obtener más información sobre el ciclo de lanzamiento de productos de Auth0, lee [Fases de lanzamiento del producto](https://auth0.com/docs/troubleshoot/product-lifecycle/product-release-stages).
</Warning>

Envía métricas de Auth0 en tiempo real a tu plataforma de monitoreo para supervisar el rendimiento de la API, identificar problemas de límite de frecuencia y solucionar errores más rápido. Flujo de métricas admite tanto la integración nativa de Datadog como OpenTelemetry Protocol (OTLP) para plataformas como New Relic y Splunk.

<div id="what-you-can-monitor">
  ## Lo que puedes supervisar
</div>

Haz seguimiento de métricas clave sobre el uso de tu API de Auth0:

* **Volumen de solicitudes de API**: Supervisa el total de solicitudes a las APIs de Authentication and Management.
* **Tasas de error**: Identifica errores del cliente y del servidor que afectan a tu integración.
* **Incidencias de límites de frecuencia**: Detecta cuándo tu aplicación alcanza los límites de frecuencia.
* **Distribución geográfica**: Analiza los patrones de tráfico por país.
* **Flujos de autenticación**: Haz seguimiento de los tipos de concesión y del uso de conexiones.

<div id="use-cases">
  ## Casos de uso
</div>

<div id="identify-rate-limit-issues">
  ### Identificar problemas con los límites de frecuencia
</div>

Monitoriza qué API y endpoints están alcanzando sus límites de frecuencia para que puedas ajustar tu integración o pasar a un nivel superior de límite de frecuencia.

<div id="troubleshoot-api-failures">
  ### Solucionar fallos de la API
</div>

Identifica rápidamente la causa principal de los errores filtrando las métricas por API, operación y tipo de fallo (límites de frecuencia, detección de anomalías o errores generales).

<div id="plan-capacity">
  ### Capacidad del plan
</div>

Analiza el uso en picos, el promedio de RPS y las métricas de percentiles para determinar el nivel de límite de frecuencia adecuado para tu carga de trabajo.

## Requisitos previos

Antes de configurar un flujo de métricas, necesitas lo siguiente:

* **Acceso a Auth0 Management API**: Crea una aplicación de máquina a máquina (M2M) con los siguientes permisos:

  * `read:event_streams`
  * `create:event_streams`
  * `update:event_streams`
  * `delete:event_streams`
  * `read:event_deliveries`
  * `update:event_deliveries`

  Para obtener instrucciones detalladas, consulta [Obtener tokens de acceso de la Management API](docs/secure/tokens/access-tokens/management-api-access-tokens/get-management-api-access-tokens-for-production).

* **Auth0 CLI**: Instala [Auth0 CLI](https://auth0.github.io/auth0-cli/) para configurar flujos de métricas mediante comandos de la API.

* **Cuenta en una plataforma de monitoreo**: Una cuenta activa con uno de los [destinos compatibles](#supported-destinations).

<div id="metric-reference">
  ## Referencia de métricas
</div>

<div id="auth0api_requestcount">
  ### auth0.api_request.count
</div>

Cuenta las solicitudes a la API de tu tenant de Auth0 con etiquetas detalladas de filtrado y agregación. Se emite una nueva métrica cada 10 segundos y ofrece una granularidad de un segundo.

<div id="available-tags">
  #### Etiquetas disponibles
</div>

| Nombre de la etiqueta | Obligatorio | Descripción                                                                                                              |
| --------------------- | ----------- | ------------------------------------------------------------------------------------------------------------------------ |
| `api`                 | Sí          | Tipo de API: `AUTHENTICATION` o `MANAGEMENT`                                                                             |
| `operation_name`      | Sí          | Método HTTP y ruta (p. ej., `POST /oauth/token`)                                                                         |
| `operation_result`    | Sí          | Resultado: `SUCCESS` o `FAILURE`                                                                                         |
| `tenant`              | Sí          | Nombre de tu tenant de Auth0                                                                                             |
| `failure_code`        | No          | Motivo del error: `RATE_LIMIT`, `ANOMALY_DETECTION` o `CLIENT_SERVER_ERROR`                                              |
| `country_code`        | No          | Código ISO del país según la dirección IP de la solicitud                                                                |
| `client_id`           | No          | ID de cliente de la aplicación (solo Authentication API)                                                                 |
| `connection_id`       | No          | ID de conexión (solo Authentication API)                                                                                 |
| `grant_type`          | No          | Tipo de concesión de OAuth 2.0 (solo Authentication API)                                                                 |
| `batch_seq`           | Sí          | Número de secuencia para métricas con etiquetas y marcas de tiempo idénticas (temporal; se eliminará después de la Beta) |

<Warning>
  Las etiquetas `client_id` y `connection_id` pueden generar una cardinalidad alta en entornos con muchos clientes o conexiones únicos, lo que podría aumentar los costos de monitoreo. Usa estas etiquetas de forma selectiva.
</Warning>

<Note>
  Los flujos de métricas no incluyen llamadas a la API almacenadas en caché ni bloqueadas por el Web Application Firewall (WAF) de Auth0 en el perímetro para evitar la sobrecarga del sistema.
</Note>

<div id="enum-values">
  #### Valores del enum
</div>

**api**

* `AUTHENTICATION` - Solicitudes de Authentication API
* `MANAGEMENT` - Solicitudes de Management API

**operation&#95;result**

* `SUCCESS` - La solicitud se completó correctamente
* `FAILURE` - La solicitud falló debido a un error del cliente o del servidor

**failure&#95;code** (solo está presente cuando `operation_result` es `FAILURE`)

* `RATE_LIMIT` - La solicitud superó el límite de frecuencia
* `ANOMALY_DETECTION` - Bloqueado por la protección contra ataques de Auth0
* `CLIENT_SERVER_ERROR` - Error HTTP 4xx o 5xx

<div id="set-up-datadog">
  ## Configura Datadog
</div>

Transmite las métricas de Auth0 directamente a Datadog mediante su API nativa de métricas.

<div id="prerequisites">
  ### Requisitos previos
</div>

* Cuenta de Datadog activa ([Datadog](https://www.datadoghq.com/))
* Clave API de Datadog (genérala en **Configuración de la organización &gt; Claves API** de Datadog)
* Tu sitio de Datadog (identificador de región, como `US1` para app.datadoghq.com, `EU` para app.datadoghq.eu o `US3` para us3.datadoghq.com)

<div id="create-metric-stream">
  ### Crear flujo de métricas
</div>

<Tabs>
  <Tab title="Dashboard">
    **Ve a Flujo de métricas:**

    1. Ve a [Auth0 Dashboard](https://manage.auth0.com) &gt; **Monitoreo** &gt; **Flujo de métricas**.
    2. Selecciona **+ Crear flujo de métricas**.
    3. Selecciona **Datadog**.

    **Configura el flujo:**

    1. **Nombre**: Ingresa un nombre descriptivo (p. ej., `datadog-production`).
    2. **Sitio de Datadog**: Selecciona tu región de Datadog en el menú desplegable:
       * `US1` para app.datadoghq.com
       * `US3` para us3.datadoghq.com
       * `US5` para us5.datadoghq.com
       * `EU` para app.datadoghq.eu
       * `AP1` para ap1.datadoghq.com
    3. **Clave de API de Datadog**: Pega tu clave de API de Datadog.
    4. Selecciona **Guardar**.

    Esto crea y habilita automáticamente tu flujo de métricas.
  </Tab>

  <Tab title="CLI">
    Usa Auth0 CLI para crear un flujo de métricas de Datadog:

    ```bash
    auth0 api post metric-streams \
      --data '{
        "name": "datadog-production",
        "subscriptions": [{ "metric_type": "auth0.api_request.count" }],
        "destination": {
          "type": "datadog",
          "configuration": {
            "api_key": "YOUR_DATADOG_API_KEY",
            "site": "app.datadoghq.com"
          }
        }
      }'
    ```

    **Respuesta:**

    ```json
    {
      "id": "mst_abc123xyz",
      "status": "enabled",
      "name": "datadog-production",
      "created_at": "2024-02-12T10:30:00Z",
      "updated_at": "2024-02-12T10:30:00Z",
      "destination": {
        "type": "datadog",
        "configuration": {
          "api_key": "***",
          "site": "app.datadoghq.com"
        }
      }
    }
    ```

    Guarda el valor de `id` para administrar este flujo más adelante.
  </Tab>
</Tabs>

<div id="verify-in-datadog">
  ### Comprueba en Datadog
</div>

En unos minutos, las métricas de Auth0 aparecerán en Datadog. Ve a **Métricas &gt; Explorer** y busca `auth0.api_request.count` para ver tus datos.

<div id="manage-streams">
  ### Administrar flujos
</div>

<Tabs>
  <Tab title="Dashboard">
    **Ver todos los flujos:**

    1. Ve a [Auth0 Dashboard](https://manage.auth0.com) &gt; **monitoreo** &gt; **Flujos de métricas**.
    2. Todos los flujos configurados se muestran con su destino, estado, nombre y suscripción.

    **Editar un flujo:**

    1. Selecciona el nombre del flujo de la lista.
    2. Actualiza la configuración.
    3. Selecciona **Guardar**.

    **Habilitar/deshabilitar un flujo:**

    1. Usa el **interruptor** en la columna Estado para habilitar o deshabilitar el flujo.
    2. Los flujos deshabilitados dejan de enviar métricas, pero conservan su configuración.

    **Eliminar un flujo:**

    1. Selecciona el nombre del flujo de la lista.
    2. Selecciona **Eliminar** al final de la página de configuración.
    3. Confirma la eliminación.
  </Tab>

  <Tab title="CLI">
    **Listar todos los flujos:**

    ```bash
    auth0 api get /api/v2/metric-streams
    ```

    **Deshabilitar un flujo:**

    ```bash
    auth0 api patch /api/v2/metric-streams/YOUR_STREAM_ID \
      --data '{"status":"disabled"}'
    ```

    **Volver a habilitar un flujo:**

    ```bash
    auth0 api patch /api/v2/metric-streams/YOUR_STREAM_ID \
      --data '{"status":"enabled"}'
    ```
  </Tab>
</Tabs>

<div id="set-up-new-relic">
  ## Configura New Relic
</div>

Envía las métricas de Auth0 a New Relic mediante OpenTelemetry Protocol (OTLP).

<div id="get-new-relic-credentials">
  ### Obtén las credenciales de New Relic
</div>

<Steps>
  <Step title="Busca tu endpoint de OTLP">
    Consulta la [documentación de OTLP de New Relic](https://docs.newrelic.com/docs/opentelemetry/best-practices/opentelemetry-otlp/#configure-endpoint-port-protocol) para encontrar el endpoint correspondiente a tu región:

    * **US:** `https://otlp.nr-data.net`
    * **EU:** `https://otlp.eu01.nr-data.net`

    Guarda la URL de este endpoint.
  </Step>

  <Step title="Crear una clave de licencia de ingestión">
    1. Ve a **&#91;tu menú de usuario&#93;** &gt; **Claves API**.
    2. Selecciona **Crear una clave**.
    3. En **Tipo de clave**, selecciona **Ingest - License**.
    4. Introduce un nombre descriptivo y, si quieres, una nota.
    5. Selecciona **Crear una clave**.
    6. Copia la clave API de inmediato. La necesitarás en los siguientes pasos y no volverá a mostrarse.
  </Step>
</Steps>

<div id="create-metric-stream">
  ### Crear flujo de métricas
</div>

<Tabs>
  <Tab title="Dashboard">
    **Ve a Flujo de métricas:**

    1. Ve a [Auth0 Dashboard](https://manage.auth0.com) &gt; **monitoreo** &gt; **Flujo de métricas**.
    2. Selecciona **+ Crear flujo de métricas**.
    3. Selecciona **OpenTelemetry**.

    **Configura el flujo:**

    1. **Nombre**: Introduce un nombre descriptivo (p. ej., `newrelic-production`).
    2. **Protocolo**: Selecciona **HTTP** en el menú desplegable.
    3. **endpoint de OTLP**: Pega el endpoint de OTLP de New Relic (p. ej., `https://otlp.nr-data.net`).
    4. **Método de autenticación**: Selecciona **API Key** en el menú desplegable.
    5. **Nombre del encabezado**: Introduce `api-key`.
    6. **API Key**: Pega la Ingest License Key de New Relic que copiaste en los pasos anteriores.
    7. Selecciona **Guardar**.

    Esto crea y habilita automáticamente tu flujo de métricas.
  </Tab>

  <Tab title="CLI">
    ```bash
    auth0 api post metric-streams \
      --data '{
        "name": "newrelic-production",
        "subscriptions": [{ "metric_type": "metrics_bridge.api_request" }],
        "destination": {
          "type": "otlp",
          "configuration": {
            "endpoint": "YOUR_NEW_RELIC_ENDPOINT",
            "protocol": "http",
            "auth_type": "api_key",
            "api_key": "YOUR_INGEST_LICENSE_KEY",
            "auth_header_name": "api-key"
          }
        }
      }'
    ```
  </Tab>
</Tabs>

<div id="set-up-splunk">
  ## Configura Splunk
</div>

Envía métricas de Auth0 a Splunk Observability Cloud mediante OpenTelemetry Protocol (OTLP).

<div id="get-splunk-credentials">
  ### Obtén las credenciales de Splunk
</div>

<Steps>
  <Step title="Encuentra tu endpoint de OTLP">
    Consulta la [documentación de OTLP HTTP Exporter de Splunk](https://help.splunk.com/en/splunk-observability-cloud/manage-data/splunk-distribution-of-the-opentelemetry-collector/get-started-with-the-splunk-distribution-of-the-opentelemetry-collector/collector-components/exporters/otlphttp-exporter) para encontrar tu endpoint.

    El formato del endpoint es:

    ```
    https://ingest.<REALM>.signalfx.com/v2/datapoint/otlp
    ```

    Sustituye `<REALM>` por tu realm de Splunk (por ejemplo, `us1`, `eu0`, `jp0`).
  </Step>

  <Step title="Crea un access token">
    1. Ve a **Settings** &gt; **Access Tokens** &gt; **New Token**.
    2. Introduce un nombre descriptivo.
    3. Configura **Authorization Scope** como **INGEST** (obligatorio).
    4. Configura la expiración del token.
    5. Selecciona **Create**.
    6. Copia y guarda de forma segura el valor del token. Lo necesitarás en pasos posteriores.
  </Step>
</Steps>

  ### Crear flujo de métricas
</div>

<Tabs>
  <Tab title="Dashboard">
    **Ve a Flujo de métricas:**

<div id="understand-metric-cardinality">
  ## Comprender la cardinalidad de las métricas
</div>

La cardinalidad se refiere al número de series temporales únicas que generan tus métricas. Una mayor cardinalidad incrementa los costos de monitoreo.

| Etiqueta           | Cardinalidad | Impacto                                           |
| ------------------ | ------------ | ------------------------------------------------- |
| `api`              | Baja         | 2 valores únicos (Authentication, Management)     |
| `operation_name`   | Media        | Varía según las operaciones de API utilizadas     |
| `operation_result` | Baja         | 2 valores únicos (Success, Failure)               |
| `tenant`           | Media        | 1 por tenant de Auth0                             |
| `failure_code`     | Baja         | 3 valores únicos (solo en caso de fallos)         |
| `country_code`     | Media-alta   | Varía según la distribución geográfica            |
| `client_id`        | **Alta**     | 1 por cada cliente de aplicación                  |
| `connection_id`    | **Alta**     | 1 por cada conexión de proveedor de identidad     |
| `grant_type`       | Baja         | Tipos de concesión de OAuth 2.0 limitados         |
| `batch_seq`        | Baja         | Etiqueta temporal (se elimina después de la Beta) |

<Warning>
  Usa las etiquetas `client_id` y `connection_id` con cuidado. En entornos con muchas aplicaciones o conexiones, estas etiquetas pueden aumentar significativamente los costos. Considera filtrar por clientes específicos o agrupar estas etiquetas si no necesitas una granularidad detallada.
</Warning>

<div id="troubleshooting">
  ## Solución de problemas
</div>

<div id="metrics-not-appearing">
  ### Las métricas no aparecen
</div>

**Verifica el estado del flujo:**

<Tabs>
  <Tab title="Dashboard">
    1. Ve a [Auth0 Dashboard](https://manage.auth0.com) &gt; **Monitoreo** &gt; **Flujo de métricas**.
    2. Busca tu flujo de métricas en la lista.
    3. Verifica que la columna **Estado** muestre **Habilitado**.
    4. Selecciona el nombre del flujo para ver la configuración detallada y comprobar los ajustes.
  </Tab>

  <Tab title="CLI">
    ```bash
    auth0 api get /api/v2/metric-streams/YOUR_STREAM_ID
    ```

    Verifica que el campo `status` sea `enabled`.
  </Tab>
</Tabs>

**Verifica las credenciales del destino:**

* **Datadog**: Confirma que tu API key sea válida y que la URL del sitio coincida con tu región de Datadog
* **Destinos OTLP**: Verifica que la URL del endpoint, el token de autenticación y el nombre del encabezado sean correctos

**Comprueba si hay caché perimetral:**

Recuerda que las solicitudes en caché y las solicitudes bloqueadas por el WAF no se incluyen en los flujos de métricas.

<div id="high-monitoring-costs">
  ### Costos elevados de monitoreo
</div>

Si observas costos inesperadamente altos:

1. Revisa qué etiquetas usas en las consultas y los paneles.
2. Considera eliminar `client_id` y `connection_id` de las agregaciones.
3. Usa muestreo o filtrado en tu plataforma de monitoreo para reducir la ingesta de métricas.
4. Agrupa por etiquetas de nivel superior, como `api` y `operation_result`, en lugar de etiquetas granulares.
