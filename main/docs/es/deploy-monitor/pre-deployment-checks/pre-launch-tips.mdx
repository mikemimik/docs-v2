---
description: Una lista de consejos útiles para quienes empiezan a usar los servicios de Auth0, basada en comentarios y experiencias de otros usuarios
title: Consejos previos al lanzamiento
---

Aquí tienes una lista de consejos que nuestros clientes han considerado más útiles al empezar a usar los servicios de Auth0:

* Comprueba las URL de devolución de llamada permitidas, las URL de CORS y las URL de redireccionamiento permitidas para el cierre de sesión para asegurarte de que sean correctas, completas y no incluyan `file:///` ni `localhost`. Minimiza el uso de comodines y considera cuidadosamente sus implicaciones de seguridad.
* Separa el uso en producción del trabajo de desarrollo en curso [usando tenants independientes para prod, test y dev.](/docs/es/get-started/auth0-overview/create-tenants/set-up-multiple-environments)
* Configura correctamente tu red para permitir el tráfico hacia y desde nuestros conjuntos de API públicas (los entornos prod pueden tener una configuración distinta de dev/staging).
* Comprueba la fecha de vencimiento de los certificado que hayas añadido a tu configuración para asegurarte de que los certificado cargados durante los ciclos de desarrollo no caduquen durante la puesta en producción o poco después.
* Asegúrate de que todos los <Tooltip tip="Proveedor de identidad (IdP): Servicio que almacena y gestiona identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=IDPs">IDPs</Tooltip> remotos tengan NTP en funcionamiento para que la hora se sincronice correctamente.
* La cantidad de días que los datos de registro están disponibles varía según el plan, hasta un máximo de 30 días. Si necesita conservar los datos de registro durante más tiempo, debe configurar una de nuestras [soluciones de transmisión de logs](https://marketplace.auth0.com/features/log-streaming). Esto le permitirá conservar los datos de registro durante períodos más largos. Asegúrese de configurarlo antes de pasar a producción para tener datos de registro disponibles cuando los necesite.
* Asegúrese de que el uso de nuestras API se mantenga dentro de los [límites permitidos](/docs/es/troubleshoot/customer-support/operational-policies/rate-limit-policy) y de haber escrito su código para ajustarse dinámicamente a la información de límite de frecuencia devuelta en el encabezado y para gestionar errores. Si prevé que los límites de frecuencia serán un problema para su aplicación, coméntelo con nosotros con antelación para comprobar si sería posible aumentarlos temporalmente hasta que el tráfico vuelva a la normalidad. Debería considerar usar una caché de datos de usuario para no tener que consultar un endpoint de la API más de lo necesario.
* Si usa [conexiones sociales](/docs/es/authenticate/identity-providers), asegúrese de obtener sus propias credenciales del proveedor y agregarlas a la configuración de la conexión social.
* Si usa [conexiones de Custom DB](/docs/es/authenticate/database-connections/custom-db), asegúrese de que todos los script de Custom DB estén implementados y devuelvan un perfil de usuario consistente con un ID de usuario único.
* Para enviar correos electrónicos, primero asegúrese de [configurar un proveedor de correo electrónico personalizado.](/docs/es/customize/email/smtp-email-providers)
* Si usa nuestro CDN para el widget de Lock, asegúrese de fijarlo a una versión específica.
* Asegúrese de que los componentes **externos** llamados desde Actions, Rules, Hooks y los scripts de conexión de base de datos personalizada puedan manejar la carga esperada.
* Proteja debidamente cualquier valor de <Tooltip tip="secreto de cliente: Secret que utiliza un cliente (aplicación) para autenticarse con el servidor de autorización; solo el cliente y el servidor de autorización deben conocerlo, y debe ser lo bastante aleatorio para que no pueda adivinarse." cta="Ver glosario" href="/docs/es/glossary?term=client+secret">secreto de cliente</Tooltip>.
* Revisa los [tipos de concesión](/docs/es/get-started/applications/application-grant-types) de tus aplicaciones. Asegúrate de tener habilitados los correctos y, más importante aún, desactiva cualquier tipo de concesión que no necesites.
* Si usas `user_metadata`, confirma que se trata de datos que los usuarios deberían poder cambiar por sí mismos (p. ej., no el “estado de pago”).
* Revisa la [configuración de Protección contra ataques](https://manage.auth0.com/#/security/attack-protection) y lee [Protección contra ataques](/docs/es/secure/attack-protection) para entender cómo desbloquear a los usuarios que hayan sido bloqueados.
* Revise la [sección Admin de Configuración del tenant](https://manage.auth0.com/#/tenant/admins) para asegurarse de que solo los administradores autorizados tengan acceso al <Tooltip tip="Auth0 Dashboard: la interfaz principal de Auth0 para configurar sus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip>. Consulte [Administrar el acceso al Dashboard](/docs/es/get-started/manage-dashboard-access) para obtener más información.
* Asegúrate de haber probado todos los casos de uso principales de tu aplicación en todos los dispositivos que puedan usar los usuarios finales de tu aplicación. Asegúrate de probar el inicio de sesión, <Tooltip tip="Inicio de sesión único (SSO): servicio que, después de que un usuario inicia sesión en una aplicación, lo inicia automáticamente en otras aplicaciones." cta="Ver glosario" href="/docs/es/glossary?term=single+sign-on">inicio de sesión único</Tooltip> (si es compatible) y el registro, así como lo que sucede si un usuario ejecuta tu aplicación en varias pestañas del navegador.
* Revisa tu [lista de reglas](https://manage.auth0.com/#/rules) y asegúrate de que solo estén activadas las reglas adecuadas.
* Revisa el código de tus reglas, cualquier script de Custom DB y cualquier código personalizado en la página de inicio de sesión para garantizar que todas las llamadas tengan un control y manejo de errores adecuados. Además, asegúrate de que las sentencias `return` y de devolución de llamada se ejecuten correctamente.
* Configura el nombre de tu aplicación, la URL de soporte y el correo electrónico de soporte en la sección [Configuración general del tenant](https://manage.auth0.com/#/tenant) para que, cuando se produzca un error, tus usuarios finales sean dirigidos a una página adecuada.
* Asegúrate de que tu aplicación esté [obteniendo dinámicamente un token de la Management API](/docs/es/secure/tokens/access-tokens/management-api-access-tokens).
* Elimina cualquier instrucción `console.log` de tus reglas o scripts de Custom DB, especialmente las que puedan filtrar información de identificación del usuario, como el correo electrónico, el nombre de usuario o la contraseña.
* No uses secretos en texto plano en reglas o conexiones de base de datos. Deben agregarse en la sección de configuración de la interfaz. La configuración está cifrada y se proporciona justo a tiempo. No registres el objeto de configuración.
