description: Una lista de consejos útiles para dar los primeros pasos con los servicios de Auth0, basada en comentarios y experiencias de otras personas
title: Consejos previos al lanzamiento
---

Aquí tienes una lista de consejos que nuestros clientes han considerado más útiles al dar sus primeros pasos con los servicios de Auth0:

* Revise las URL de devolución de llamada permitidas, CORS y las URL de redirección permitidas para el cierre de sesión para asegurarse de que sean correctas, completas y que no impliquen el uso de `file:///` o `localhost`. Minimice el uso de comodines y considere cuidadosamente sus implicaciones de seguridad.
* Separe el uso en producción del trabajo de desarrollo en curso [usando tenants independientes para prod, test y dev.](/docs/es/get-started/auth0-overview/create-tenants/set-up-multiple-environments)
* Configure correctamente su red para permitir el tráfico hacia y desde nuestros conjuntos de API públicas (los entornos de prod pueden tener una configuración distinta de dev/staging).
* Revise la fecha de vencimiento de los certificados que haya agregado a su configuración para asegurarse de que los certificados cargados durante los ciclos de desarrollo no venzan durante la puesta en marcha o poco después.
* Asegúrate de que todos los <Tooltip tip="Proveedor de identidad (IdP): Servicio que almacena y gestiona identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=IDPs">IDPs</Tooltip> remotos usen NTP para que la hora se sincronice correctamente.
* La cantidad de días durante los que los datos de registro están disponibles varía según el plan, hasta un máximo de 30 días. Si necesita conservar los datos de registro durante más tiempo, debe configurar una de nuestras [soluciones de transmisión de logs](https://marketplace.auth0.com/features/log-streaming). Esto le permitirá conservar los datos de registro durante períodos más largos. Asegúrese de configurarlo antes de pasar a producción para tener datos de registro disponibles cuando los necesite.
* Asegúrese de que su uso de nuestras APIs se mantenga dentro de los [límites permitidos](/docs/es/troubleshoot/customer-support/operational-policies/rate-limit-policy) y de haber escrito su código para ajustarse dinámicamente a la información del límite de frecuencia devuelta en el header y para gestionar errores. Si prevé que los límites de frecuencia puedan ser un problema para su aplicación, consúltelo con nosotros con antelación para comprobar si sería posible aumentarlos temporalmente hasta que el tráfico vuelva a la normalidad. Debería considerar usar una caché de datos de usuario para no tener que consultar un endpoint de API más de lo necesario.
* Si usa [conexiones sociales](/docs/es/authenticate/identity-providers), asegúrese de obtener sus propias credenciales del proveedor y añadirlas a la configuración de la conexión social.
* Si usa [conexiones de Custom DB](/docs/es/authenticate/database-connections/custom-db), asegúrese de que todos los scripts de Custom DB estén implementados y devuelvan un perfil de usuario coherente con un ID de usuario único.
* Para enviar correos electrónicos, primero asegúrese de [configurar un proveedor de correo electrónico personalizado.](/docs/es/customize/email/smtp-email-providers)
* Si usa nuestro CDN para el widget de Lock, asegúrese de fijarlo en una versión específica.
* Asegúrese de que los componentes **externos** invocados desde Actions, Rules, Hooks y los scripts de conexiones de base de datos personalizadas puedan manejar la carga esperada.
* Proteja adecuadamente cualquier valor de <Tooltip tip="Secreto de cliente: Secret que utiliza un cliente (aplicación) para autenticarse con el servidor de autorización; solo el cliente y el servidor de autorización deben conocerlo, y debe ser lo bastante aleatorio como para que no pueda adivinarse." cta="Ver glosario" href="/docs/es/glossary?term=client+secret">secreto de cliente</Tooltip>.
* Revisa los [tipos de concesión](/docs/es/get-started/applications/application-grant-types) de tus aplicaciones. Asegúrate de tener habilitados los correctos y, lo más importante, desactiva cualquier tipo de concesión que no sea necesario.
* Si usas user&#95;metadata, confirma que se trate de datos que los usuarios deban poder cambiar por sí mismos (p. ej., no “estado de pago”).
* Revisa la [configuración de Protección contra ataques](https://manage.auth0.com/#/security/attack-protection) y lee [Protección contra ataques](/docs/es/secure/attack-protection) para saber cómo desbloquear a los usuarios que han sido bloqueados.
* Revisa la [sección Admin de la Configuración del tenant](https://manage.auth0.com/#/tenant/admins) para asegurarte de que solo los administradores autorizados tengan acceso al <Tooltip tip="Auth0 Dashboard: el producto principal de Auth0 para configurar tus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip>. Consulta [Administrar el acceso al Dashboard](/docs/es/get-started/manage-dashboard-access) para obtener más información.
* Asegúrate de haber probado todos los casos de uso principales de tu aplicación en todos los dispositivos que puedan usar los usuarios finales de tu aplicación. Asegúrate de probar el inicio de sesión, <Tooltip tip="Inicio de sesión único (SSO): servicio que, después de que un usuario inicia sesión en una aplicación, inicia automáticamente la sesión de ese usuario en otras aplicaciones." cta="Ver glosario" href="/docs/es/glossary?term=single+sign-on">inicio de sesión único</Tooltip> (si es compatible) y la generación de registros, así como lo que ocurre si un usuario usa tu aplicación en varias pestañas del navegador.
* Revisa tu [lista de reglas](https://manage.auth0.com/#/rules) y asegúrate de que solo estén activadas las adecuadas.
* Revisa el código de tu rule, cualquier script de Custom DB y cualquier código personalizado en la login page para asegurarte de que todas las llamadas tengan un control y manejo de errores adecuados. Además, comprueba que las sentencias `return`/de devolución de llamada se ejecuten correctamente.
* Configura el nombre de tu aplicación, la URL de soporte y el correo electrónico de soporte en la sección [Tenant Settings General](https://manage.auth0.com/#/tenant) para que, cuando se produzca un error, tus usuarios finales sean dirigidos a una página adecuada.
* Asegúrate de que tu aplicación esté [obteniendo dinámicamente un token de la Management API](/docs/es/secure/tokens/access-tokens/management-api-access-tokens).
* Elimina cualquier sentencia `console.log` de tus reglas o scripts de Custom DB, especialmente las que puedan exponer información identificable del usuario, como el correo electrónico, el username o la password.
* No uses secretos en texto plano en reglas ni en conexiones de base de datos. Deben añadirse en la sección de configuración de la interfaz. La configuración está cifrada y se proporciona Just In Time. No registres el objeto de configuración.
