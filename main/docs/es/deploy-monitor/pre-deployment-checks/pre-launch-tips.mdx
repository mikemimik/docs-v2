description: Una lista de consejos útiles para empezar a usar los servicios de Auth0, basada en comentarios y experiencias de otros usuarios
title: Consejos previos al lanzamiento
---

Aquí tienes una lista de consejos que nuestros clientes han considerado más útiles al empezar a usar los servicios de Auth0:

* Verifique las URLs de devolución de llamada permitidas, CORS y URLs de redirección permitidas para el cierre de sesión para asegurarse de que sean correctas, completas y no incluyan el uso de `file:///` o `localhost`. Minimice el uso de comodines y considere cuidadosamente sus implicaciones de seguridad.
* Separe el uso en producción del trabajo de desarrollo en curso [usando tenants separados para prod, test y dev.](/docs/es/get-started/auth0-overview/create-tenants/set-up-multiple-environments)
* Configure correctamente su red para permitir el tráfico entrante y saliente hacia nuestros conjuntos de API públicas (los entornos de prod pueden tener una configuración distinta de la de dev/staging).
* Verifique la fecha de expiración de los certificados que haya agregado a su configuración para asegurarse de que los certificados cargados durante los ciclos de desarrollo no expiren durante la puesta en producción o poco después.
* Asegúrese de que todos los <Tooltip tip="Proveedor de identidad (IdP): servicio que almacena y administra identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=IDPs">IDPs</Tooltip> remotos estén configurados para usar NTP, de modo que la hora se sincronice correctamente.
* La cantidad de días de datos de registro disponibles varía según el plan, con un máximo de 30 días. Si necesita conservar los datos de registro durante más tiempo, debe configurar una de nuestras [soluciones de transmisión de logs](https://marketplace.auth0.com/features/log-streaming). Esto le permitirá conservar los datos de registro durante períodos más largos. Asegúrese de hacerlo antes de pasar a producción para tener datos de registro disponibles cuando los necesite.
* Asegúrese de que el uso de nuestras API se mantenga dentro de los [límites permitidos](/docs/es/troubleshoot/customer-support/operational-policies/rate-limit-policy) y de que su código esté preparado para ajustarse dinámicamente a la información de límite de frecuencia devuelta en el header y para gestionar errores. Si prevé que los límites de frecuencia puedan ser un problema para su aplicación, consúltelo con nosotros con antelación para comprobar si sería posible aumentarlos temporalmente hasta que el tráfico vuelva a la normalidad. También debería considerar usar una caché de datos de usuario para no tener que consultar un endpoint de API más de lo necesario.
* Si utiliza [conexiones sociales](/docs/es/authenticate/identity-providers), asegúrese de obtener sus propias credenciales del proveedor y agregarlas a la configuración de la conexión social.
* Si utiliza [conexiones de Custom DB](/docs/es/authenticate/database-connections/custom-db), asegúrese de que todos los script de Custom DB estén implementados y devuelvan un perfil de usuario coherente con un ID de usuario único.
* Para enviar correos electrónicos, primero asegúrese de [configurar un proveedor de correo electrónico personalizado.](/docs/es/customize/email/smtp-email-providers)
* Si usa nuestro CDN para el widget de Lock, asegúrese de fijar una versión específica.
* Asegúrese de que los componentes **externos** a los que llaman Actions, Rules, Hooks y los scripts de conexión de base de datos personalizada puedan manejar la carga esperada.
* Proteja adecuadamente cualquier valor de <Tooltip tip="secreto de cliente: secreto que usa un cliente (aplicación) para autenticarse con el servidor de autorización; solo el cliente y el servidor de autorización deben conocerlo, y debe ser lo suficientemente aleatorio como para que no sea posible adivinarlo." cta="Ver glosario" href="/docs/es/glossary?term=client+secret">secreto de cliente</Tooltip>.
* Revisa los [tipos de concesión](/docs/es/get-started/applications/application-grant-types) de tus aplicaciones. Asegúrate de tener habilitados los correctos y, lo que es más importante, desactiva cualquier tipo de concesión que no necesites.
* Si usas user&#95;metadata, confirma que sean datos que los usuarios puedan cambiar por sí mismos (p. ej., no “estado de pago”).
* Revisa la [configuración de Attack Protection](https://manage.auth0.com/#/security/attack-protection) y lee [Attack Protection](/docs/es/secure/attack-protection) para entender cómo desbloquear a los usuarios que hayan sido bloqueados.
* Revisa la [sección Admin de Configuración del tenant](https://manage.auth0.com/#/tenant/admins) para asegurarte de que solo los administradores correspondientes tengan acceso al <Tooltip tip="Auth0 Dashboard: el principal producto de Auth0 para configurar tus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip>. Consulta [Administrar el acceso al Dashboard](/docs/es/get-started/manage-dashboard-access) para obtener más información.
* Asegúrate de haber probado todos los casos de uso principales de tu aplicación en todos los dispositivos que pueda utilizar la población de usuarios finales de tu aplicación. No dejes de probar el inicio de sesión, <Tooltip tip="Inicio de sesión único (SSO): servicio que, después de que un usuario inicia sesión en una aplicación, inicia automáticamente la sesión de ese usuario en otras aplicaciones." cta="Ver glosario" href="/docs/es/glossary?term=single+sign-on">inicio de sesión único</Tooltip> (si es compatible) y el registro de eventos, así como lo que sucede si un usuario ejecuta tu aplicación en varias pestañas del navegador.
* Revisa tu [lista de Rules](https://manage.auth0.com/#/rules) y asegúrate de que solo estén activadas las reglas adecuadas.
* Revisa el código de tus rule, cualquier script de Custom DB y cualquier código personalizado de la login page para asegurarte de que cada llamada tenga una captura y un manejo de errores adecuados. Además, comprueba que las sentencias `return`/devolución de llamada se ejecuten correctamente.
* Configura el nombre de tu aplicación, la URL de soporte y el correo electrónico de soporte en la sección [Tenant Settings General](https://manage.auth0.com/#/tenant) para que, cuando se produzca un error, tus usuarios finales sean dirigidos a una página adecuada.
* Asegúrate de que tu aplicación esté [obteniendo dinámicamente un token de la Management API](/docs/es/secure/tokens/access-tokens/management-api-access-tokens).
* Elimina cualquier sentencia `console.log` de tus reglas o scripts de Custom DB, especialmente las que puedan filtrar información de identificación del usuario, como el correo electrónico, el username o la contraseña.
* No uses secretos en texto sin formato en reglas ni en conexiones de base de datos. Deben añadirse en la sección de configuración de la interfaz. La configuración está cifrada y se proporciona justo a tiempo. No registres el objeto de configuración.
