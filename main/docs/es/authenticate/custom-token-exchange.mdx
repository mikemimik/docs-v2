<ReleaseStageNotice feature="Custom Token Exchange (CTE)" stage="ea" plans="B2C Professional, B2B Professional, and Enterprise" terms="true" />

Custom Token Exchange permite a las aplicaciones intercambiar sus tokens existentes por tokens de Auth0 al llamar al endpoint `/oauth/token`, tal como se define en [RFC 8693](https://datatracker.ietf.org/doc/html/rfc8693). Algunos casos de uso habituales de Custom Token Exchange incluyen:

* Obtener tokens de Auth0 para otra <Tooltip tip="Audience: Identificador único de la audiencia de un token emitido. Llamado aud en un token, su valor contiene el ID de una aplicación (ID de cliente) para un token de ID o de una API (identificador de API) para un token de acceso." cta="Ver glosario" href="/docs/es/glossary?term=audience">audiencia</Tooltip>
* Integrarse con un <Tooltip tip="Proveedor de identidad (IdP): Servicio que almacena y gestiona identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=identity+provider">proveedor de identidad</Tooltip> externo
* Migrar a Auth0
* Autorización delegada: un principal (como un servicio, un agente de IA o un agente de soporte) actúa en nombre de un usuario

Para obtener más información, consulta [Ejemplos de casos de uso y muestras de código](/docs/es/authenticate/custom-token-exchange/cte-example-use-cases).

Cada solicitud de Custom Token Exchange se asigna a un [Custom Token Exchange Profile](/docs/es/authenticate/custom-token-exchange/configure-custom-token-exchange#create-custom-token-exchange-profile), que está controlado por una [Action](/docs/es/customize/actions), donde puedes:

* Escribir código personalizado para decodificar y validar `subject_tokens` enviados al endpoint `/oauth/token`
* Autorizar el acceso y definir el usuario que continuará con la transacción.

Puedes [configurar varios Custom Token Exchange Profiles](/docs/es/authenticate/custom-token-exchange/configure-custom-token-exchange#create-custom-token-exchange-profile) para una aplicación. Después de que el servidor de autorización de Auth0 valide que la solicitud de Custom Token Exchange es válida y se asigne a un Custom Token Exchange Profile existente, el [trigger de Custom Token Exchange](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/custom-token-exchange-trigger) ejecuta la única Action asociada a ese perfil. A continuación, la aplicación puede usar Custom Token Exchange para autenticarse y obtener tokens de acceso, de ID y de actualización de Auth0 para el usuario.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Custom Token Exchange te ofrece la flexibilidad adicional de definir el usuario para la transacción, asumiendo al mismo tiempo la responsabilidad adicional de validar de forma segura el token de sujeto correspondiente que identifica al usuario de la transacción.

  Recuerda que los tokens de sujeto y de actor utilizados con Custom Token Exchange pueden tener cualquier formato o tipo de token que necesites, siempre que el código de tu Action pueda interpretarlos. **Debes implementar una validación sólida de los tokens que recibes y aceptas.** Si no lo haces, te expones a distintos vectores de ataque, como la suplantación o los ataques de repetición, lo que puede permitir que actores maliciosos se autentiquen con el ID de usuario de otra persona o actúen en su nombre sin autorización.

  Para conocer distintas opciones para implementar una validación segura de tus tokens de sujeto, consulta y aplica las recomendaciones incluidas en [Ejemplos de casos de uso y muestras de código](/docs/es/authenticate/custom-token-exchange/cte-example-use-cases). Asegúrate también de tener en cuenta y aplicar las capacidades de [Attack Protection](/docs/es/authenticate/custom-token-exchange/cte-attack-protection).
</Callout>

<div id="tenant-logs">
  ## Registros del tenant
</div>

Cada transacción de Custom Token Exchange genera un registro de eventos del tenant:

* Transacciones exitosas: registros `secte`
* Transacciones fallidas: registros `fecte`

Cuando se establece un actor mediante `setActor()`, la propiedad `actor` (que contiene `sub` y cualquier `actor` anidado) se incluye en las entradas de registro `secte` con fines de auditoría.

Usa los registros del tenant para solucionar cualquier problema que surja con tu intercambio de tokens.

<div id="limitations">
  ## Limitaciones
</div>

Custom Token Exchange no admite lo siguiente:

* Métodos de la API de MFA `api.authentication.challengeWith()` y `api.authentication.EnrollWith()`
* Las conexiones de base de datos personalizadas con el modo de importación `ON` no son compatibles con las operaciones `setUserByConnection()`
* clientes de terceros y clientes que no cumplen con OIDC
* La API de destino debe tener habilitado **Allow Skipping User Consent**, ya que no se puede recopilar el consentimiento en un flujo no interactivo.
