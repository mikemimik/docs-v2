<div id="configure-native-to-web-sso">
  ## Configurar SSO de nativo a web
</div>

Para usar el <Tooltip tip="Inicio de sesión único (SSO): servicio que, después de que un usuario inicia sesión en una aplicación, le inicia sesión automáticamente en otras aplicaciones." cta="Ver glosario" href="/docs/es/glossary?term=Single+Sign-On">inicio de sesión único</Tooltip> (SSO) de nativo a web, configure sus aplicaciones nativa y web (aplicación de página única o aplicación web tradicional) para crear y administrar sesiones con la <Tooltip tip="Management API: un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip> de Auth0.

Necesita un [token de acceso](/docs/es/secure/tokens/access-tokens/management-api-access-tokens) para usar la Management API o Auth0 CLI.

Para configurar SSO de nativo a web, debe crear y administrar session&#95;transfer&#95;tokens y configurar sus aplicaciones nativa y web.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  SSO de nativo a web es compatible con los siguientes SDK: [Auth0 Android SDK](https://github.com/auth0/auth0.android/blob/main/EXAMPLES.md#native-to-web-sso-login-ea), [Auth0 Swift SDK](https://github.com/auth0/Auth0.swift/blob/master/EXAMPLES.md#sso-credentials-ea) y [Auth0 React Native SDK](https://github.com/auth0/react-native-auth0/blob/master/EXAMPLES.md#native-to-web-sso-early-access).

  La compatibilidad con SSO de nativo a web está disponible en las siguientes herramientas: [Auth0 Deploy CLI](/docs/es/deploy-monitor/deploy-cli-tool). [Auth0 Terraform Provider](/docs/es/deploy-monitor/auth0-terraform-provider) y [Auth0 CLI](https://auth0.github.io/auth0-cli/).

  SSO de nativo a web es compatible con cualquier flujo de autenticación que devuelva un token de actualización, como [flujo de contraseña del propietario del recurso](/docs/es/secure/multi-factor-authentication/authenticate-using-ropg-flow-with-mfa) y [flujo de código de autorización con Proof Key for Code Exchange](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce).
</Callout>

<div id="create-and-manage-session-transfer-tokens">
  ### Crear y administrar Session Transfer Tokens
</div>

La primera llamada a la Management API permite a tus Aplicaciones nativas y web:

* Crear y administrar `session_transfer_tokens`
* Crear sesiones en un navegador web mediante cookies o un parámetro de URL
* Vincular la sesión al dispositivo de un usuario mediante una dirección IP o ASN

Para las aplicaciones existentes, realiza una llamada `PATCH` al endpoint [Update a Client](https://auth0.com/docs/api/management/v2/clients/patch-clients-by-id). Para crear una nueva aplicación, realiza una llamada `POST` al endpoint [Create a Client](https://auth0.com/docs/api/management/v2/clients/post-clients):

```json lines
{
  "session_transfer": {
    "can_create_session_transfer_token": false,
    "allowed_authentication_methods": ["cookie", "query"],
    "enforce_device_binding": "ip", // también "none" o "asn",
    "allow_refresh_token": false,
    "enforce_cascade_revocation": true,
    "enforce_online_refresh_tokens": true
  }
}
```

<div id="configure-native-applications">
  ### Configurar aplicaciones nativas
</div>

Una vez que el usuario se autentica, Auth0 devuelve un [token de acceso](/docs/es/secure/tokens/access-tokens), un [token de identidad](/docs/es/secure/tokens/id-tokens) y, opcionalmente, un [token de actualización](/docs/es/secure/tokens/refresh-tokens).

Puedes configurar tu aplicación nativa para intercambiar un <Tooltip tip="token de actualización: Token utilizado para obtener un token de acceso renovado sin obligar a los usuarios a iniciar sesión de nuevo." cta="Ver glosario" href="/docs/es/glossary?term=refresh+token">token de actualización</Tooltip> por un Session Transfer Token. Si tu aplicación web no admite la inyección de cookies, tu aplicación nativa también debe configurar el **Login URI** de la aplicación web para inyectar el Session Transfer Token como parámetro de URI.

* Actualiza tu aplicación nativa desde el <Tooltip tip="Auth0 Dashboard: El producto principal de Auth0 para configurar tus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip>:

  1. Ve a [Dashboard &gt; Applications](https://manage.auth0.com/dashboard/#/applications):
  2. Selecciona o [crea tu aplicación](/docs/es/get-started/auth0-overview/create-applications#create-applications).
  3. Selecciona la pestaña **Settings**.
  4. En **Session Transfer**, habilita **Allow Native to Web SSO**.
  5. Selecciona **Require Cascade Revocation** para revocar automáticamente la sesión de la aplicación web y los tokens asociados.
  6. Selecciona **Save** para actualizar la aplicación.

Antes de habilitar Session Transfer Token, asegúrate de haber configurado el **Application Login URI** de tu aplicación web para admitir parámetros adicionales.   7. Habilita **Allow Refresh Token Requests** para emitir nuevos tokens de actualización después de una transferencia de sesión.
  8. Selecciona **Use online refresh tokens**.
  9. Selecciona **Save** para actualizar tu aplicación web.

<div id="implement-native-to-web-sso">
  ## Implement SSO de nativo a web
</div>

El SSO de nativo a web proporciona una experiencia de usuario fluida al llevar a los usuarios autenticados de tu aplicación nativa a tu aplicación web.

Para hacerlo posible, tu aplicación nativa debe intercambiar un token de actualización por un Session Transfer Token y enviar el Session Transfer Token, mediante una URL o una cookie, a tu aplicación web para autorizar la sesión.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  * Si `allow_refresh_token` está deshabilitado en el cliente, pero la aplicación solicita `offline_access`, Auth0 no emitirá un `refresh_token`, aunque la autenticación seguirá funcionando.

  * Si la rotación del Refresh Token está habilitada, Auth0 devuelve un nuevo `refresh_token` en la llamada de intercambio de tokens. El intercambio de token de actualización debe realizarse inmediatamente antes de que tu código abra la aplicación web.
</Callout>

<div id="step-1-exchanging-a-refresh-token-for-a-session-transfer-token">
  #### Paso 1: Intercambiar un token de actualización por un Session Transfer Token
</div>

Usa el endpoint [/token](https://auth0.com/docs/api/authentication/authorization-code-flow-with-pkce/get-token-pkce) en tu aplicación nativa para intercambiar el token de actualización por un Session Transfer Token.

* Intercambia un token de actualización por un Session Transfer Token con los SDK de Swift, Android o React Native:

    ```swift lines
    // O bien, con async/await

        print("Failed with: \(error)")
    }
    ```

    ```kotlin lines
            }
    ```

Hay dos opciones para enviar el  `session_transfer_token` a tu aplicación web según los `allowed_authentication_methods` configurados.

<div id="option-1-send-the-session_transfer_token-as-a-cookie">
  ##### Opción 1: Enviar el `session_transfer_token` como una cookie
</div>

Si su aplicación web en WebView o en un navegador admite la inyección de cookies, puede configurar su aplicación nativa para:

* Agregar el `session_transfer_token` a una cookie.
* Abrir la aplicación web usando WebView o un navegador.
* Iniciar sesión en la aplicación web con su tenant de Auth0 o <Tooltip tip="Custom Domain: Dominio de terceros con un nombre especializado o personalizado." cta="Ver glosario" href="/docs/es/glossary?term=Custom+Domain">Custom Domain</Tooltip>. Como el `session_transfer_token` ya está incluido en la cookie, no se le pedirá al usuario que complete la autenticación del primer factor.

      https://{yourDomain}/authorize?{authorize_params}`;

<div id="option-2-add-the-session-transfer-token-to-web-applications-using-auth0-sdks">
  #### Opción 2: Añade el Session Transfer Token a las aplicaciones web con los SDK de Auth0
</div>

Los SDK de Auth0 no son compatibles automáticamente con Native to Web Single SSO y no incluirán el `session_transfer_token` en la solicitud al endpoint `/authorize`.

A continuación, se muestran ejemplos de aplicaciones web que usan los SDK de Auth0 para redirigir el `session_transfer_token` en la solicitud al endpoint `/authorize`:

<div id="node-expressjs">
  ##### Node (Express.js)
</div>

Si tu aplicación web usa [Express.js](https://expressjs.com/) o el [SDK de Auth0 para Express](https://github.com/auth0/express-openid-connect), puedes usar el siguiente código para añadir compatibilidad de middleware para `session_transfer_token`.

```javascript javascript lines
const baseConfig = {
  authRequired: false,
  auth0Logout: true
};

// Extensión del middleware para detectar automáticamente session_transfer_token
app.use((req, res, next) => {
  const { session_transfer_token } = req.query;

  // Crear una nueva configuración para cada solicitud y evitar filtraciones de estado
  const config = { ...baseConfig };

  if (session_transfer_token) {
    config.authorizationParams = {
      session_transfer_token,
    };
  }

  auth(config)(req, res, next);
});
```

<div id="auth0-spa-sdk-auth0auth0-spa-js">
  ##### Auth0 SPA SDK (@auth0/auth0-spa-js)
</div>

Si tu aplicación web utiliza el [Auth0 SPA SDK](https://github.com/auth0/auth0-spa-js), puedes pasar el `session_transfer_token` a `loginWithRedirect()` a través de `authorizationParams`.

```typescript typescript lines
import { Auth0Client } from '@auth0/auth0-spa-js';

const auth0 = new Auth0Client({
  domain: '{yourDomain}',
  clientId: '{yourClientId}',
});

// Obtener session_transfer_token de los parámetros de consulta de la URL (enviado desde la app nativa)
const urlParams = new URLSearchParams(window.location.search);
const sessionTransferToken = urlParams.get('session_transfer_token');

// Usar loginWithRedirect con el session_transfer_token
if (sessionTransferToken) {
  await auth0.loginWithRedirect({
    authorizationParams: {
      session_transfer_token: sessionTransferToken,
      redirect_uri: window.location.origin,
    },
  });
}
```

<div id="native-to-web-sso-with-organizations">
  ## SSO de nativo a web con Organizations
</div>

SSO de nativo a web admite [Organizations](/docs/es/manage-users/organizations). Cuando un usuario se autentica con una organización en tu aplicación nativa, el Session Transfer Token incluye el contexto de la organización.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Al usar Organizations con SSO de nativo a web, el parámetro `organization` en la solicitud `/authorize` de la aplicación web debe coincidir con la organización asociada al Session Transfer Token. Si no coincide, la autenticación fallará y se le pedirá al usuario que vuelva a iniciar sesión.
</Callout>

Para usar Organizations con SSO de nativo a web:

1. Autentica al usuario en tu aplicación nativa con una organización:

```javascript javascript lines
// Inicio de sesión en la app nativa con organización
await authorize({
  organization: 'org_abc123',
  scope: 'openid profile email offline_access'
});
```

2. Al redirigir a la aplicación web, incluye la misma organización en la solicitud de `/authorize`:

```bash bash lines
https://{yourDomain}/authorize?
  client_id={yourWebClientId}&
  redirect_uri={yourRedirectUri}&
  response_type=code&
  organization=org_abc123&
  session_transfer_token=YOUR_SESSION_TRANSFER_TOKEN
```

Si la organización de la solicitud a `/authorize` no coincide con la organización del Session Transfer Token, este se rechaza y el usuario es redirigido a la página de inicio de sesión para volver a autenticarse. Se genera un registro de advertencia en los logs de tu tenant con la descripción del evento &quot;Error de inicio de sesión único: se detectó una discrepancia entre organizaciones en el Session Transfer Token.&quot;

<div id="session-transfer-token-with-actions">
  ## Session Transfer Token con Actions
</div>

Usar `session_transfer_token` con [Actions](/docs/es/customize/actions) te permite configurar capacidades de detección y respuesta ante riesgos posteriores a la autenticación para reforzar la protección del usuario.

Para ello, el objeto de la Action de Post Login **event.session&#95;transfer&#95;token** proporciona información relevante, como un `client_id` único, `scope`, información de `request` como `ip`, `asn` y `user_agent`, e información de `geoip` como `cityName` y `countryCode`, entre otros datos. Para obtener más información, consulta [Actions Triggers: post-login - Event Object](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger/post-login-event-object).

El siguiente código de Action te permite rechazar dinámicamente una transacción en función de la información de geolocalización:

```javascript javascript lines
/**
* Manejador que será invocado durante la ejecución de un flujo PostLogin.
*
* @param {Event} event - Detalles sobre el usuario y el contexto en el que está iniciando sesión.
* @param {PostLoginAPI} api - Interfaz cuyos métodos pueden usarse para modificar el comportamiento del inicio de sesión.
*/
exports.onExecutePostLogin = async (event, api) => {
   if(
     event.session_transfer_token &&
     event.session_transfer_token.request.geoip.countryCode !== event.request.geoip.countryCode
     ) {
     api.access.deny("Network mismatch detected")
   }
};
```

<div id="access-parent-refresh-token-metadata">
  ### Acceder a los metadatos del token de actualización principal
</div>

SSO de nativo a web le permite acceder a los metadatos del token de actualización principal que se utilizó para iniciar el flujo de SSO. Esto le permite transferir información contextual recopilada en la aplicación nativa (como la integridad del dispositivo, señales de riesgo o contexto personalizado) a la sesión web creada a partir del `session_transfer_token`.

Auth0 expone esta información en las Actions de Post Login mediante el objeto `event.session.session_transfer.parent_refresh_token.metadata`. Esto permite la propagación segura y estandarizada de metadatos entre plataformas.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  **Caso de uso de ejemplo:** Una aplicación móvil recopila datos de riesgo (puntuación del dispositivo, ubicación, etc.) y los almacena en los metadatos del token de actualización. Cuando la aplicación nativa inicia un flujo de SSO de nativo a web, esos metadatos pasan a estar disponibles automáticamente en la sesión web correspondiente a través de Actions.
</Callout>

El código de Action que aparece a continuación le permite aplicar lógica de acceso condicional en función de los metadatos de confianza del dispositivo procedentes de la aplicación nativa:

```javascript javascript lines
/**
* Manejador que será llamado durante la ejecución de un flujo PostLogin.
*
* @param {Event} event - Detalles sobre el usuario y el contexto en el que está iniciando sesión.
* @param {PostLoginAPI} api - Interfaz cuyos métodos pueden usarse para cambiar el comportamiento del inicio de sesión.
*/
exports.onExecutePostLogin = async (event, api) => {
  // Acceder a los metadatos del token de actualización principal (aplicación nativa)
  const parentMetadata = event.session?.session_transfer?.parent_refresh_token?.metadata;

  if (parentMetadata) {
    const deviceTrustLevel = parentMetadata.device_trust;

    if (deviceTrustLevel !== 'high') {
      api.access.deny("Device trust level insufficient.");
    }

    // También puedes agregar claims según el contexto de la aplicación nativa
    if (parentMetadata.subscription_tier) {
      api.idToken.setCustomClaim('subscription_tier', parentMetadata.subscription_tier);
    }
  }
};
```

<div id="token-exchange-logs">
  ### Registros del intercambio de tokens
</div>

* `sertft` : Intercambio exitoso de token de actualización. Este registro corresponderá a un intercambio de SSO de nativo a web cuando el campo `audience` sea `"audience":"urn:$auth0Domain:session_transfer"`
* `fertft`: Intercambio fallido de token de actualización. Este registro corresponderá a un intercambio de SSO de nativo a web cuando el campo `audience` sea `"audience": "urn:$auth0Domain:session_transfer"`

<div id="session-transfer-validation-warning-logs">
  ### Registros de advertencia de validación de la transferencia de sesión
</div>

Auth0 emite registros de advertencia (`w`) cuando falla la validación del Session Transfer Token durante la solicitud `/authorize`. Estos registros te ayudan a resolver problemas del SSO de nativo a web:

| Descripción del evento                                                                                                   | Causa                                                                                                                                       |
| ------------------------------------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------- |
| `Single Sign-On failed: Session Transfer Token not found or expired. This may indicate token reuse or expiration.`       | No se encontró el Session Transfer Token, ya se había usado o había expirado (los tokens son de un solo uso y expiran al cabo de 1 minuto). |
| `Single Sign-On failed: Session Transfer Token device binding validation failed due to IP/ASN mismatch.`                 | La dirección IP o el ASN de la solicitud web no coinciden con la vinculación del dispositivo configurada para el Session Transfer Token.    |
| `Single Sign-On failed: Session Transfer Token organization mismatch detected.`                                          | El parámetro `organization` de la solicitud `/authorize` no coincide con la organización del Session Transfer Token.                        |
| `Single Sign-On failed: Session Transfer Token user mismatch detected.`                                                  | Una sesión de Auth0 preexistente pertenece a un usuario distinto del usuario del Session Transfer Token.                                    |
| `Single Sign-On failed: Parent refresh token not found. Session Transfer Token won't be used for session establishment.` | El token de actualización principal que se utilizó para crear el Session Transfer Token fue revocado o eliminado.                           |
