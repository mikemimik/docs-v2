<div id="configure-native-to-web-sso">
  ## Configurar SSO de nativo a web
</div>

Para usar <Tooltip tip="inicio de sesión único (SSO): Servicio que, después de que un usuario inicia sesión en una aplicación, inicia automáticamente la sesión de ese usuario en otras aplicaciones." cta="Ver glosario" href="/docs/es/glossary?term=Single+Sign-On">inicio de sesión único</Tooltip> (SSO) de nativo a web, configure sus aplicaciones nativas y web (aplicación de página única o aplicación web tradicional) para crear y administrar sesiones con la <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip> de Auth0.

Necesita un [token de acceso](/docs/es/secure/tokens/access-tokens/management-api-access-tokens) para usar la Management API o Auth0 CLI.

Para configurar SSO de nativo a web, debe crear y administrar session&#95;transfer&#95;tokens, y configurar sus aplicaciones nativas y web.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  SSO de nativo a web admite los siguientes SDKs: [Auth0 Android SDK](https://github.com/auth0/auth0.android/blob/main/EXAMPLES.md#native-to-web-sso-login-ea), [Auth0 Swift SDK](https://github.com/auth0/Auth0.swift/blob/master/EXAMPLES.md#sso-credentials-ea) y [Auth0 React Native SDK](https://github.com/auth0/react-native-auth0/blob/master/EXAMPLES.md#native-to-web-sso-early-access).

  La compatibilidad con SSO de nativo a web está disponible en las siguientes herramientas: [Auth0 Deploy CLI](/docs/es/deploy-monitor/deploy-cli-tool), [Auth0 Terraform Provider](/docs/es/deploy-monitor/auth0-terraform-provider) y [Auth0 CLI](https://auth0.github.io/auth0-cli/).

  SSO de nativo a web admite cualquier flujo de autenticación que devuelva un token de actualización, como [flujo de contraseña del propietario del recurso](/docs/es/secure/multi-factor-authentication/authenticate-using-ropg-flow-with-mfa) y [flujo de código de autorización con Proof Key for Code Exchange](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce).
</Callout>

<div id="create-and-manage-session-transfer-tokens">
  ### Crear y administrar tokens de transferencia de sesión
</div>

La primera llamada a la Management API permite que tus aplicaciones nativas y web:

* Creen y administren `session_transfer_tokens`
* Creen sesiones en un navegador web mediante cookies o un parámetro de URL
* Vinculen la sesión al dispositivo del usuario mediante una dirección IP o ASN

Para las aplicaciones existentes, haz una llamada `PATCH` al endpoint [actualizar un cliente](https://auth0.com/docs/api/management/v2/clients/patch-clients-by-id). Para crear una nueva aplicación, haz una llamada `POST` al endpoint [crear un cliente](https://auth0.com/docs/api/management/v2/clients/post-clients):

```json lines
{
  "session_transfer": {
    "can_create_session_transfer_token": false,
    "allowed_authentication_methods": ["cookie", "query"],
    "enforce_device_binding": "ip", // también "none" o "asn",
    "allow_refresh_token": false,
    "enforce_cascade_revocation": true,
    "enforce_online_refresh_tokens": true
  }
}
```

<div id="configure-native-applications">
  ### Configurar aplicaciones nativas
</div>

Una vez que el usuario se autentica, Auth0 devuelve un [token de acceso](/docs/es/secure/tokens/access-tokens), un [token de identidad](/docs/es/secure/tokens/id-tokens) y, opcionalmente, un [token de actualización](/docs/es/secure/tokens/refresh-tokens).

Puede configurar su aplicación nativa para intercambiar un <Tooltip tip="Token de actualización: token que se usa para obtener un token de acceso renovado sin obligar a los usuarios a iniciar sesión de nuevo." cta="Ver glosario" href="/docs/es/glossary?term=refresh+token">token de actualización</Tooltip> por un token de transferencia de sesión. Si su aplicación web no admite la inyección de cookies, su aplicación nativa también debe configurar el **Login URI** de la aplicación web para inyectar el token de transferencia de sesión como parámetro de URI.

* Actualice su aplicación nativa con el <Tooltip tip="Auth0 Dashboard: el producto principal de Auth0 para configurar sus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip>:

  1. Vaya a [Dashboard &gt; Applications](https://manage.auth0.com/dashboard/#/applications):
  2. Seleccione o [cree su aplicación](/docs/es/get-started/auth0-overview/create-applications#create-applications).
  3. Seleccione la pestaña **Settings**.
  4. En **Session Transfer**, habilite **Allow Native to Web SSO**.
  5. Seleccione **Require Cascade Revocation** para revocar automáticamente la sesión de la aplicación web y los tokens asociados.
  6. Seleccione **Save** para actualizar su aplicación.

  8. Selecciona **Use online refresh tokens**.
  9. Selecciona **Save** para actualizar tu aplicación web.

<div id="implement-native-to-web-sso">
  ## Implement SSO de nativo a web
</div>

El SSO de nativo a web ofrece una experiencia de usuario fluida al trasladar a usuarios autenticados de tu aplicación nativa a tu aplicación web.

Para hacerlo posible, tu aplicación nativa debe intercambiar un token de actualización por un token de transferencia de sesión y enviar el token de transferencia de sesión, a través de una URL o una cookie, a tu aplicación web para autorizar la sesión.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  * Si `allow_refresh_token` está deshabilitado en el cliente, pero la aplicación solicita `offline_access`, Auth0 no emitirá un `refresh_token`, aunque la autenticación seguirá funcionando.

  * Si la rotación del Refresh Token está habilitada, Auth0 devuelve un nuevo `refresh_token` en la llamada de intercambio de tokens. El intercambio de token de actualización debe realizarse justo antes de que tu código abra la aplicación web.
</Callout>

  ### En tu aplicación nativa
</div>

<div id="step-1-exchanging-a-refresh-token-for-a-session-transfer-token">
  #### Paso 1: Intercambiar un token de actualización por un token de transferencia de sesión
</div>

Usa el endpoint [/token](https://auth0.com/docs/api/authentication/authorization-code-flow-with-pkce/get-token-pkce) con tu aplicación nativa para intercambiar el token de actualización por un token de transferencia de sesión.

* Intercambia un token de actualización por un token de transferencia de sesión con los SDK de Swift, Android o React Native:

<Tabs>
  <Tab title="SDK de Swift">
    ```swift lines
    credentialsManager.ssoCredentials { result in
        switch result {
        case .success(let ssoCredentials):
            print("Obtained SSO credentials: \(ssoCredentials)")
        case .failure(let error):
            print("Failed with: \(error)")
        }
    }

    // O bien, usando async/await

    do {
        let ssoCredentials = try await credentialsManager.ssoCredentials()
        print("Obtained SSO credentials: \(ssoCredentials)")
    } catch {
        print("Failed with: \(error)")
    }
    ```
  </Tab>

  <Tab title="SDK de Android">

    ```kotlin lines
            override fun onSuccess(result: SSOCredentials) {
                // Usa result.sessionTransferToken para autenticar
                // al usuario en una sesión web de tu aplicación
            }

            override fun onFailure(error: CredentialsManagerException) {
                // Maneja el error
            }
        }
    )
    ```
  </Tab>

  <Tab title="SDK de React Native">

    ```typescript lines
    // Uso de la clase Auth0
    ```

<div id="option-1-send-the-session_transfer_token-as-a-cookie">
  ##### Opción 1: Envíe el session_transfer_token como una cookie
</div>

Si su aplicación web que usa WebView o el navegador admite la inyección de cookies, puede configurar su aplicación nativa para:

* Agregar el session&#95;transfer&#95;token a una cookie.
* Abrir la aplicación web usando WebView o el navegador.
* Iniciar sesión en la aplicación web con su tenant de Auth0 o <Tooltip tip="Dominio personalizado: dominio de terceros con un nombre especializado o personalizado." cta="Ver glosario" href="/docs/es/glossary?term=Custom+Domain">Dominio personalizado</Tooltip>. Como el `session_transfer_token` está incluido en la cookie, no se le solicitará al usuario la autenticación de primer factor.

<div id="option-2-add-the-session-transfer-token-to-web-applications-using-auth0-sdks">
  #### Opción 2: Agrega el token de transferencia de sesión a las aplicaciones web con SDKs de Auth0
</div>

Los SDKs de Auth0 no son compatibles automáticamente con Native to Web Single SSO y no incluirán el `session_transfer_token` en la solicitud al endpoint `/authorize`.

A continuación se muestran ejemplos de aplicaciones web que usan SDKs de Auth0 para redirigir el `session_transfer_token` en la solicitud al endpoint `/authorize`:

<div id="node-expressjs">
  ##### Node (Express.js)
</div>

Si tu aplicación web usa [Express.js](https://expressjs.com/) o el [SDK de Auth0 para Express](https://github.com/auth0/express-openid-connect), puedes usar el siguiente código para añadir compatibilidad de middleware para `session_transfer_token`.

```javascript javascript lines
const baseConfig = {
  authRequired: false,
  auth0Logout: true
};

// Extensión del middleware para detectar automáticamente session_transfer_token
app.use((req, res, next) => {
  const { session_transfer_token } = req.query;

  // Crear una nueva configuración para cada solicitud y evitar filtraciones de estado
  const config = { ...baseConfig };

  if (session_transfer_token) {
    config.authorizationParams = {
      session_transfer_token,
    };
  }

  auth(config)(req, res, next);
});
```

<div id="auth0-spa-sdk-auth0auth0-spa-js">
  ##### Auth0 SPA SDK (@auth0/auth0-spa-js)
</div>

Si tu aplicación web utiliza el [Auth0 SPA SDK](https://github.com/auth0/auth0-spa-js), puedes pasar el `session_transfer_token` a `loginWithRedirect()` mediante `authorizationParams`.

```typescript typescript lines
import { Auth0Client } from '@auth0/auth0-spa-js';

const auth0 = new Auth0Client({
  domain: '{yourDomain}',
  clientId: '{yourClientId}',
});

// Obtener session_transfer_token de los parámetros de consulta de la URL (enviado desde la app nativa)
const urlParams = new URLSearchParams(window.location.search);
const sessionTransferToken = urlParams.get('session_transfer_token');

// Usar loginWithRedirect con el session_transfer_token
if (sessionTransferToken) {
  await auth0.loginWithRedirect({
    authorizationParams: {
      session_transfer_token: sessionTransferToken,
      redirect_uri: window.location.origin,
    },
  });
}
```

  ##### SDK de Auth0 para React (@auth0/auth0-react)

Si tu aplicación web usa el [SDK de Auth0 para React](https://github.com/auth0/auth0-react), puedes usar `loginWithRedirect` del hook `useAuth0` para enviar el `session_transfer_token`.

```typescript typescript lines
import { useEffect } from 'react';
import { useAuth0 } from '@auth0/auth0-react';

function App() {
  const { loginWithRedirect, isAuthenticated, isLoading } = useAuth0();

  useEffect(() => {
    if (isLoading || isAuthenticated) return;

    const urlParams = new URLSearchParams(window.location.search);
    const sessionTransferToken = urlParams.get('session_transfer_token');

    if (sessionTransferToken) {
      loginWithRedirect({
        authorizationParams: {
          session_transfer_token: sessionTransferToken,
        },
      });
    }
  }, [loginWithRedirect, isLoading, isAuthenticated]);

  return <div>...</div>;
}
```

<div id="native-to-web-sso-with-organizations">
  ## SSO de nativo a web con Organizations
</div>

SSO de nativo a web es compatible con [Organizations](/docs/es/manage-users/organizations). Cuando un usuario se autentica con una organización en tu aplicación nativa, el token de transferencia de sesión incluye el contexto de la organización.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Al usar Organizations con SSO de nativo a web, el parámetro `organization` de la solicitud `/authorize` de la aplicación web debe coincidir con la organización asociada al token de transferencia de sesión. Si no coincide, la autenticación fallará y se le pedirá al usuario que vuelva a iniciar sesión.
</Callout>

Para usar Organizations con SSO de nativo a web:

1. Autentica al usuario en tu aplicación nativa con una organización:

```javascript javascript lines
// Inicio de sesión en la app nativa con organización
await authorize({
  organization: 'org_abc123',
  scope: 'openid profile email offline_access'
});
```

2. Al redirigir a la aplicación web, incluya la misma organización en la solicitud `/authorize`:

```bash bash lines
https://{yourDomain}/authorize?
  client_id={yourWebClientId}&
  redirect_uri={yourRedirectUri}&
  response_type=code&
  organization=org_abc123&
  session_transfer_token=YOUR_SESSION_TRANSFER_TOKEN
```

Si la organización de la solicitud a `/authorize` no coincide con la organización del token de transferencia de sesión, este se rechaza y se redirige al usuario a la página de inicio de sesión para que vuelva a autenticarse. Se genera un registro de advertencia en los logs del tenant con la descripción del evento &quot;Single Sign-On failed: Session Transfer Token organization mismatch detected.&quot;

<div id="session-transfer-token-with-actions">
  ## token de transferencia de sesión con Actions
</div>

El uso de `session_transfer_token` con [Actions](/docs/es/customize/actions) permite configurar capacidades de detección y respuesta ante riesgos posteriores a la autenticación para reforzar la protección del usuario.

Para ello, el objeto de Action Post Login **event.session&#95;transfer&#95;token** proporciona información relevante, incluido un `client_id` único, `scope`, información de `request` como `ip`, `asn` y `user_agent`, e información de `geoip` como `cityName` y `countryCode`, entre otros datos. Para obtener más información, consulte [Actions Triggers: post-login - Event Object](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger/post-login-event-object).

El código de Action que aparece a continuación permite rechazar dinámicamente una transacción en función de la información de geolocalización:

```javascript javascript lines
/**
* Manejador que se invocará durante la ejecución de un flujo Post Login.
*
* @param {Event} event - Detalles sobre el usuario y el contexto en el que está iniciando sesión.
* @param {PostLoginAPI} api - Interfaz cuyos métodos pueden usarse para cambiar el comportamiento del inicio de sesión.
*/
exports.onExecutePostLogin = async (event, api) => {
   if(
     event.session_transfer_token &&
     event.session_transfer_token.request.geoip.countryCode !== event.request.geoip.countryCode
     ) {
     api.access.deny("Network mismatch detected")
   }
};
```

<div id="access-parent-refresh-token-metadata">
  ### Acceder a los metadatos del token de actualización principal
</div>

SSO de nativo a web te permite acceder a los metadatos del token de actualización principal que se utilizó para iniciar el flujo de SSO. Esto te permite pasar información contextual recopilada en la aplicación nativa (como la integridad del dispositivo, señales de riesgo o contexto personalizado) a la sesión web creada a partir del `session_transfer_token`.

Auth0 expone esta información en las Actions de Post Login a través del objeto `event.session.session_transfer.parent_refresh_token.metadata`. Esto permite la propagación segura y estandarizada de metadatos entre plataformas.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  **Caso de uso de ejemplo:** Una aplicación móvil recopila datos de riesgo (puntuación del dispositivo, ubicación, etc.) y los almacena en los metadatos del token de actualización. Cuando la aplicación nativa inicia un flujo de SSO de nativo a web, esos metadatos pasan a estar disponibles automáticamente en la sesión web correspondiente a través de Actions.
</Callout>

El siguiente código de Action te permite aplicar lógica de acceso condicional en función de los metadatos de confianza del dispositivo de la aplicación nativa:

```javascript javascript lines
/**
* Manejador que será llamado durante la ejecución de un flujo Post Login.
*
* @param {Event} event - Detalles sobre el usuario y el contexto en el que está iniciando sesión.
* @param {PostLoginAPI} api - Interfaz cuyos métodos pueden usarse para cambiar el comportamiento del inicio de sesión.
*/
exports.onExecutePostLogin = async (event, api) => {
  // Acceder a los metadatos del token de actualización padre (aplicación nativa)
  const parentMetadata = event.session?.session_transfer?.parent_refresh_token?.metadata;

  if (parentMetadata) {
    const deviceTrustLevel = parentMetadata.device_trust;

    if (deviceTrustLevel !== 'high') {
      api.access.deny("Device trust level insufficient.");
    }

    // También puedes agregar claims según el contexto de la aplicación nativa
    if (parentMetadata.subscription_tier) {
      api.idToken.setCustomClaim('subscription_tier', parentMetadata.subscription_tier);
    }
  }
};
```

<div id="token-exchange-logs">
  ### Registros de intercambio de tokens
</div>

* `sertft` : Intercambio de token de actualización exitoso. Este registro corresponderá a un intercambio de SSO de nativo a web cuando el campo `audience` sea `"audience":"urn:$auth0Domain:session_transfer"`
* `fertft`: Intercambio de token de actualización fallido. Este registro corresponderá a un intercambio de SSO de nativo a web cuando el campo `audience` sea `"audience": "urn:$auth0Domain:session_transfer"`

<div id="session-transfer-validation-warning-logs">
  ### Registros de advertencia de validación de la transferencia de sesión
</div>

Auth0 emite registros de advertencia (`w`) cuando falla la validación del token de transferencia de sesión durante la solicitud a `/authorize`. Estos registros te ayudan a solucionar problemas de SSO de nativo a web:

| Descripción del evento                                                                                                   | Causa                                                                                                                                              |
| ------------------------------------------------------------------------------------------------------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------- |
| `Single Sign-On failed: Session Transfer Token not found or expired. This may indicate token reuse or expiration.`       | No se encontró el token de transferencia de sesión, ya se usó o ha expirado (los tokens son de un solo uso y caducan después de 1 minuto).         |
| `Single Sign-On failed: Session Transfer Token device binding validation failed due to IP/ASN mismatch.`                 | La dirección IP o el ASN de la solicitud web no coinciden con la vinculación del dispositivo configurada para el token de transferencia de sesión. |
| `Single Sign-On failed: Session Transfer Token organization mismatch detected.`                                          | El parámetro `organization` en la solicitud a `/authorize` no coincide con la organización del token de transferencia de sesión.                   |
| `Single Sign-On failed: Session Transfer Token user mismatch detected.`                                                  | Una sesión de Auth0 existente pertenece a un usuario distinto del usuario del token de transferencia de sesión.                                    |
| `Single Sign-On failed: Parent refresh token not found. Session Transfer Token won't be used for session establishment.` | El token de actualización principal usado para crear el token de transferencia de sesión ha sido revocado o eliminado.                             |
