---
title: Self-Service Enterprise Configuration
description: Aprende a usar Self-Service Enterprise Configuration para delegar la configuración de SSO a tus clientes B2B.
validatedOn: 2026-02-19
---

Self-Service Enterprise Configuration proporciona a los clientes B2B las herramientas necesarias para delegar la configuración de SSO a sus clientes empresariales. Al delegar esta tarea, puedes agilizar tu proceso de incorporación y dar a los clientes más autonomía sobre su experiencia de inicio de sesión. También puedes reducir el tiempo y los costos asociados con la gestión de SSO en toda tu base de clientes.

Self-Service Enterprise Configuration requiere una configuración mínima en tu tenant de Auth0 y ofrece a tus clientes un asistente de configuración que los guía durante el proceso de habilitación. Después de que un cliente completa la configuración, la integración de SSO se agrega automáticamente a tu tenant como una [conexión empresarial](/docs/es/authenticate/enterprise-connections).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Los usuarios con los siguientes roles del Dashboard pueden usar esta funcionalidad:

  * Los usuarios **Admin** y **Editor - Connections** pueden crear y gestionar perfiles de autoservicio.
  * Los usuarios **Viewer - Config** solo pueden ver perfiles de autoservicio.
</Callout>

<div id="self-service-enterprise-configuration-supported-providers">
  ## Proveedores compatibles con Self-Service Enterprise Configuration
</div>

Actualmente, el inicio de sesión único (SSO) es compatible con los siguientes <Tooltip tip="Proveedor de identidad (IdP): servicio que almacena y gestiona identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=identity+providers">proveedores de identidad</Tooltip>:

* Okta Workforce Identity
* Auth0
* Entra ID
* Google Workspace
* Keycloak
* Microsoft Active Directory Federation Services (ADFS)
* PingFederate
* OIDC genérico
* <Tooltip tip="Security Assertion Markup Language (SAML): protocolo estandarizado que permite a dos partes intercambiar información de autenticación sin una contraseña." cta="Ver glosario" href="/docs/es/glossary?term=SAML">SAML</Tooltip> genérico

Actualmente, el aprovisionamiento es compatible con los siguientes proveedores de identidad:

* Okta Workforce Identity
* Entra ID
* OIDC genérico
* SAML genérico

<div id="how-it-works">
  ## Cómo funciona
</div>

Self-Service Enterprise Configuration utiliza los siguientes componentes para delegar la configuración en sus clientes:

* **Perfil de autoservicio**: Define elementos clave de las implementaciones de los clientes, como los proveedores de identidad que pueden usar para SSO y los atributos de usuario que deben recopilar, como el correo electrónico. Puede crear hasta 20 perfiles en su tenant para distintos clientes o segmentos.
* **Ticket de acceso de autoservicio**: Otorga a sus clientes acceso de administrador al asistente de autoservicio y define detalles específicos de la conexión empresarial resultante. Esos administradores de clientes pueden crear nuevas conexiones o modificar las existentes.
* **Asistente de configuración de autoservicio**: Guía a los administradores de sus clientes durante el proceso de configuración de SSO. Para obtener más información sobre esta experiencia, consulte la experiencia del asistente de autoservicio.

<div id="self-service-enterprise-configuration-workflow">
  ### Flujo de trabajo de Self-Service Enterprise Configuration
</div>

Los pasos a continuación describen el flujo de trabajo general para usar Self-Service Enterprise Configuration. Estas tareas pueden completarse mediante la <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip> o el <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip>.

1. Usted (el cliente de Auth0) crea un perfil de autoservicio en su tenant.
2. Luego, crea un ticket de acceso de autoservicio asociado a ese perfil de autoservicio. Al generar el ticket, puede decidir si el administrador de su cliente creará una conexión nueva o modificará una conexión existente mediante el asistente de autoservicio.
3. Recupera la URL del ticket del recurso creado en el Paso 2 y envía el enlace al administrador de su cliente.
4. El administrador de su cliente inicia el asistente de autoservicio mediante la URL del ticket y sigue los pasos indicados para configurar su conexión y, opcionalmente, completar la verificación del dominio. Si el ticket es para una sola organización con Discovery habilitado, el asistente detecta automáticamente los dominios ya verificados para esa organización. Esto permite al administrador asociar de inmediato los dominios existentes con la nueva conexión, sin necesidad de volver a verificarlos.
5. Una conexión empresarial nueva o actualizada que apunta a la aplicación del cliente queda disponible en su tenant de Auth0.

<div id="self-service-assistant-experience">
  ## Experiencia del asistente de autoservicio
</div>

El asistente de autoservicio es una experiencia de varios pasos que guía a los administradores del cliente a través de:

* Configuración de SSO
  * Verificación de dominio
  * Configuración del aprovisionamiento

Esta experiencia incluye elementos interactivos e instrucciones para realizar los cambios necesarios en el IdP seleccionado.

Aunque los requisitos exactos para configurar SSO varían según el IdP, el flujo de trabajo general del asistente de autoservicio incluye lo siguiente:

1. **Seleccionar proveedor de identidad**: El administrador del cliente selecciona qué IdP configurar para SSO. La lista de opciones entre las que puede elegir está determinada por el perfil de autoservicio asociado.
2. **Crear aplicación**: El administrador del cliente sigue las instrucciones escritas para crear una aplicación en el sistema IdP seleccionado.
3. **Configurar conexión**: El administrador del cliente completa un breve formulario para crear o modificar una conexión en Auth0. Este formulario recopila información como el dominio del cliente, el ID de cliente y el secreto de cliente.
4. **mapeo de claims**: El administrador del cliente revisa los atributos de usuario obligatorios y opcionales que debe capturar a través de su conexión SSO. Luego, asigna esos atributos de usuario, o claims, en su sistema IdP.
5. **Asignar acceso**: El administrador del cliente sigue las instrucciones escritas para su sistema IdP a fin de conceder acceso a tu aplicación a usuarios o grupos de usuarios.
6. **Probar SSO**: El administrador del cliente usa el botón proporcionado para probar su conexión SSO en una nueva pestaña. A menos que la verificación de dominio esté habilitada, el administrador del cliente puede completar el proceso de configuración después de este paso para habilitar su conexión.
7. **(Opcional) Aprovisionamiento**: El administrador del cliente configura el aprovisionamiento de usuarios y grupos para su conexión siguiendo las instrucciones proporcionadas:
   * Crear la aplicación en su IdP.
   * Generar un bearer token de SCIM en el asistente y copiar el bearer token de SCIM y la URL del endpoint de SCIM en la configuración de su IdP.
   * Revisar los atributos obligatorios y opcionales definidos en el User Attribute Profile (UAP). Asignar estos atributos a los campos de SCIM correspondientes en su sistema IdP.
8. **(Opcional) Verificación y asociación de dominios**: Los administradores del cliente gestionan los dominios usados para la autenticación y el enrutamiento. La verificación de dominio exige que los clientes demuestren la propiedad de sus dominios. La asociación de dominios permite reutilizar identidades de confianza existentes.

Cuando se configura un ticket para **una** organización habilitada, los dominios verificados pueden sincronizarse automáticamente para [Organization Domain Discovery](/docs/es/manage-users/organizations/login-flows-for-organizations#organization-domain-discovery-optional). Una vez que un administrador de TI verifica o asocia un dominio, los usuarios finales pueden iniciar sesión con su dirección de correo electrónico (Home Realm Discovery) y evitar la necesidad de un Organization ID específico o de Magic Link.

Según cómo configures tú (el cliente de Auth0) el access ticket, la experiencia para los administradores del cliente varía. Revisa la tabla para obtener más información.

| Configuración                                                  | Descripción                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                   |
| -------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Verified Domain Association                                    | Los administradores del cliente seleccionan y asocian dominios existentes con la nueva conexión sin verificar el registro DNS TXT si el ticket está limitado a una organización y la organización:  <ul><li>Debe tener habilitada la opción **Allow Use of Domains for Organization Discovery**.</li><li>Debe tener **Domain Verification** configurado como `Optional` o `Required`.</li></ul> <p /> El asistente detecta automáticamente los dominios verificados previamente para esa organización específica.                                                                                             |
| Domain Verification set to `Required`                          | Los administradores del cliente deben verificar correctamente un dominio nuevo mediante DNS o asociar un dominio verificado existente antes de poder habilitar la conexión.                                                                                                                                                                                                                                                                                                                                                                                                                                   |
| Domain Verification set to `Optional`                          | El administrador del cliente puede optar por introducir un dominio nuevo para verificación, asociar uno existente u omitir el paso. En todos los casos, el administrador puede habilitar la conexión independientemente del estado de verificación.                                                                                                                                                                                                                                                                                                                                                           |
| Domain Verification set to `Off`                               | Los administradores del cliente no tienen que hacer nada. Este paso no aparece para los administradores del cliente y el flujo termina después de Probar SSO.                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| **Allow Use of Domains for Organization Discovery** is enabled | Esta opción está disponible al generar un ticket para **una** organización habilitada. Cuando se selecciona: <ul><li>**New Domains**: Cualquier dominio verificado recientemente completa automáticamente tanto la lista de dominios de la organización como `matching_domains` de la conexión empresarial.</li> <li>**Existing Domains**: Cualquier dominio preverificado (limitado a 5) o dominio asociado se sincroniza tanto con la conexión como con el registro de la organización al finalizar, lo que habilita el inicio de sesión inmediato mediante correo electrónico para los usuarios.</li></ul> |

Para obtener más información, consulta [Manage Self-Service Enterprise Configuration](/docs/es/authenticate/enterprise-connections/self-service-enterprise-configuration/manage-self-service-enterprise-config).
