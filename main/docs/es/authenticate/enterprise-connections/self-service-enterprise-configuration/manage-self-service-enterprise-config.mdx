Self-Service Enterprise Configuration proporciona a los clientes business-to-business (B2B) las herramientas necesarias para delegar la configuración del SSO en sus clientes empresariales y requiere una configuración mínima en su tenant de Auth0 para ofrecerles un asistente de autoservicio que los guíe durante el proceso de habilitación.

Una vez que un cliente completa la configuración, la integración de SSO se agrega automáticamente a su tenant como una [conexión empresarial](/docs/es/authenticate/enterprise-connections).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Los usuarios con los siguientes roles del Dashboard pueden usar esta funcionalidad:

  * Los usuarios **Admin** y **Editor - Connections** pueden crear y administrar perfiles de autoservicio.
  * Los usuarios **Viewer - Config** solo pueden ver perfiles de autoservicio.
</Callout>

Para facilitar Self-Service Enterprise Configuration, configurará los siguientes componentes mediante la <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip> o el <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip>:

* **Perfil de autoservicio**: Define elementos clave de las implementaciones de SSO de los clientes, incluidos los <Tooltip tip="Proveedor de identidad (IdP): Servicio que almacena y administra identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=identity+providers">proveedores de identidad</Tooltip> (IdPs) que pueden usar y los atributos de usuario que deben recopilar, como el correo electrónico. Puede crear hasta 20 perfiles en su tenant para diferentes clientes o segmentos.
* **Ticket de acceso de autoservicio**: Otorga a los administradores del cliente acceso al [**asistente de autoservicio**](/docs/es/authenticate/enterprise-connections/self-service-enterprise-configuration#self-service-assistant-experience) y establece detalles específicos para la conexión empresarial resultante. Los tickets de acceso permiten a los administradores del cliente crear conexiones nuevas o modificar las existentes.

Las siguientes secciones ofrecen pasos detallados para configurar perfiles de autoservicio y generar tickets de acceso de autoservicio para compartir con los administradores del cliente.

<div id="create-self-service-profiles">
  ## Crear perfiles de autoservicio
</div>

Puede crear perfiles de autoservicio mediante el Auth0 Dashboard o la Management API.

Los perfiles de autoservicio se utilizan para determinar elementos clave de las implementaciones de los clientes, entre ellos:

* Qué proveedores de identidad pueden usar los administradores del cliente para el SSO.
* Qué atributos de usuario deben recopilar mediante SSO, como el correo electrónico o el apellido.
* Opciones de branding que personalizan la apariencia del asistente de autoservicio.

Puede crear hasta 20 perfiles, según sea necesario, para adaptarse a distintos clientes o segmentos.

<Tabs>
  <Tab title="Auth0 Dashboard">
    Para crear un perfil de autoservicio en el Auth0 Dashboard: 

    1. Vaya a [Authentication &gt; empresarial](https://manage.auth0.com/#/connections/enterprise) y abra la sección **Configuración empresarial de autoservicio**. Luego, seleccione **Crear perfil**.
    2. En el espacio proporcionado, introduzca un nombre y una descripción opcional para el perfil. Luego, seleccione **Create**.<br />
          A. **Opcional** Adjunte un perfil de atributos de usuario.
          <Callout icon="file-lines" color="#0EA5E9" iconType="regular">
          Si crea y adjunta, o habilita, un perfil de atributos de usuario existente, no tendrá la opción de agregar atributos desde la pestaña Perfil de usuario.
          </Callout>
             1. Seleccione un UAP existente o cree uno nuevo. Para un UAP nuevo:<br />
                a. Agregue un nombre.<br />
                b. Revise las asignaciones para asegurarse de que los atributos del perfil se asignen a los atributos de Auth0 que prefiera.
                   <Frame>![Asignación de UAP Authentication > Enterprise > Self-Service](/docs/images/cdy7uua7fh8z/Authenticate>Enterprise>Self-Service>UAP.png)</Frame>
    3. En la pestaña **Settings**, complete las secciones siguientes. Luego, seleccione **Save**.
       * **Identity Providers (IdP)**: Habilite uno o más proveedores de identidad. En el asistente de autoservicio, los administradores del cliente pueden seleccionar su opción preferida de la lista de proveedores habilitados.
       * **Branding**: Proporcione un logotipo y un color principal para el asistente de autoservicio.
       * **Introducción personalizada**: Modifique o sustituya el mensaje predeterminado según sea necesario. Este texto introductorio se muestra a los administradores del cliente en la página de destino del asistente de autoservicio. El mensaje puede incluir opciones básicas de formato, como negrita o hipervínculos, y está limitado a 2000 caracteres.
    4. En la pestaña **Perfil de usuario**, agregue hasta 20 atributos de usuario que sus clientes deban capturar mediante SSO, como el correo electrónico o el apellido. Puede establecer cada atributo como `required` u `optional`.

       * Durante el flujo del asistente de autoservicio, se pedirá a los administradores del cliente que asignen estos atributos de usuario definidos a su proveedor de identidad para garantizar que los valores necesarios se transfieran a Auth0.

    #### Personaliza tu texto de introducción

    Cuando un administrador de cliente accede al asistente de autoservicio, primero llega a una página de introducción que le da la bienvenida. De forma predeterminada, se muestra el siguiente mensaje:

    &quot;Estás a pocos pasos de configurar SSO. Este proceso de configuración implica realizar algunos cambios en tu proveedor de identidad. Antes de comenzar, abre tu proveedor de identidad en una pestaña o ventana separada del navegador.&quot;

    Puedes modificar este texto realizando una llamada PUT al endpoint [Custom Text for Self-Service Profiles](https://auth0.com/docs/api/management/v2/self-service-profiles/put-self-service-profile-custom-text).

    <Warning>
      Ten en cuenta que esta llamada **sobrescribe** cualquier mensaje configurado actualmente para el perfil de autoservicio. Asegúrate de que la llamada incluya el texto completo que quieras mostrar a los administradores de tus clientes.
    </Warning>

    1. Haz una llamada a `PUT /api/v2/self-service-profiles/{id}/custom-text/{language}/{page}`, donde

       * `id` es el ID del perfil de autoservicio
       * `language` se establece en `en`
       * `page` se establece en `get-started`
    2. En el cuerpo de la solicitud, especifica lo siguiente:

    | Propiedad      | Descripción                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                     |
    | -------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
    | `introduction` | <ul><li>String. La longitud máxima es de 2000.</li><li>Texto completo de introducción que se mostrará en la página de inicio del asistente de autoservicio. El texto puede incluir opciones básicas de formato, como negrita o hipervínculos.</li><li>El texto personalizado proporcionado mediante este parámetro reemplaza por completo cualquier mensaje anterior. Para obtener los mejores resultados, asegúrese de proporcionar el mensaje completo que desea mostrar a los administradores del cliente.</li><li>Enviar un cuerpo vacío `\{}` restablece cualquier texto personalizado al texto predeterminado. </li></ul> |

    3. En la respuesta, se devuelve la entidad creada.

    **Ejemplo de llamada**

    ```js lines
    PUT /api/v2/self-service-profiles/ssp_1234567890/custom-text/en/get-started

    {
      introduction: "Welcome! With <b>only a few steps</b>, you'll be able to setup your new connection. For assistance, contact <a href="https://www.examplesupportsite.com"> our support team </a>." 
    }
    ```

    **Ejemplo de respuesta**

    ```js lines
    {
      introduction: "Welcome! With <b>only a few steps</b>, you'll be able to setup your new connection. For assistance, contact <a href="https://www.examplesupportsite.com"> our support team </a>." 
    }
    ```

<div id="manage-self-service-access-tickets">
  ## Administrar tickets de acceso de autoservicio
</div>

Después de crear al menos un perfil de autoservicio, puedes generar tickets de acceso de autoservicio desde el Auth0 Dashboard o la Management API.

Los tickets de acceso de autoservicio tienen dos finalidades principales:

* Dar a los administradores del cliente acceso al asistente de autoservicio, donde pueden configurar una nueva conexión SSO o modificar una conexión existente.
* Predefinir detalles y comportamientos clave de las nuevas conexiones SSO que configurarán los administradores del cliente, como qué aplicaciones u organizaciones se habilitarán para la nueva conexión.

Al generar tickets de acceso, también puedes habilitar ciertas funciones, como el SSO iniciado por el IdP de <Tooltip tip="Security Assertion Markup Language (SAML): Protocolo estandarizado que permite a dos partes intercambiar información de autenticación sin una contraseña." cta="Ver glosario" href="/docs/es/glossary?term=SAML">SAML</Tooltip>, configurar dominios del proveedor de identidad (que impulsan Home Realm Discovery) y la verificación de dominios.

<div id="saml-idp-initiated-sso">
  #### Inicio de sesión único de SAML iniciado por IdP
</div>

El inicio de sesión único de SAML iniciado por IdP es un tipo de implementación que permite a los proveedores de identidad iniciar el inicio de sesión único y redirigir a los usuarios al proveedor de servicios para su autenticación.

Al habilitar esta opción para Self-Service Enterprise Configuration, debe proporcionar su aplicación predeterminada y el protocolo de respuesta. También puede proporcionar una cadena de consulta opcional para personalizar aún más el comportamiento de la conexión.

Para obtener más información sobre estas opciones, consulte [Configurar el inicio de sesión único iniciado por un proveedor de identidad SAML](/docs/es/authenticate/protocols/saml/saml-sso-integrations/identity-provider-initiated-single-sign-on).

<div id="email-domain-verification-and-pre-verified-domains">
  #### Verificación del dominio de correo electrónico y dominios preverificados
</div>

Self-Service Enterprise Configuration admite tres formas de asociar dominios de correo electrónico con una conexión empresarial. Estos métodos rellenan el mismo campo: `options.domain_aliases `, que controla Home Realm Discovery (HRD) y, de forma condicional, los Dominios de organización para el descubrimiento:

1. Dominios preverificados (gestionados por el tenant): El administrador del tenant añade dominios conocidos directamente a la conexión.
2. Dominios por verificar: El administrador del tenant especifica los dominios que el administrador de TI debe verificar
   durante la configuración. Estos dominios aparecen como pendientes en la organización y/o no se asocian automáticamente a la conexión hasta que el administrador de TI complete la verificación.
3. Verificación del dominio de correo electrónico (autogestionada): El administrador de su cliente verifica el dominio durante la configuración en el asistente de autoservicio.

Estos mecanismos para asociar dominios de correo electrónico se aplican a los dominios de correo electrónico utilizados para el enrutamiento de HRD y SSO. No están relacionados con la verificación de Custom Domain a nivel del tenant.

Para que un dominio se incluya tanto en los Dominios de organización para el descubrimiento como en HRD, el ticket debe cumplir lo siguiente:

* Incluir solo un `enabled_organization`
* Tener dominios preverificados o Verificación de correo electrónico requerida como `Required` o `Optional`
* Habilitar la casilla **Allow the Use of Domains for Organization Discovery**

<div id="pre-verified-domain">
  ##### Dominio verificado previamente
</div>

Usa un dominio verificado previamente para asociar dominios de correo electrónico a una conexión empresarial en el nivel del tenant. Los dominios añadidos se consideran de confianza y se escriben directamente en `options.domain_aliases`.

Puedes proporcionar dominios al [generar tickets de acceso de autoservicio](#generate-access-tickets-for-new-connections) desde Auth0 Dashboard o la Management API.

* **Auth0 Dashboard**: En la página **Generate Ticket**, especifica tu lista de dominios en el campo **Pre-verified Domains**.
* **Management API**: Establece `connection_config.options.domain_aliases` con la lista de dominios. De forma predeterminada, `use_for_organization_discovery` se establece en `true`. Opcionalmente, selecciona **Allow the Use of Domains for Organization Discovery**.

Los dominios añadidos por los administradores del tenant se consideran **de confianza** y **no** requieren que el administrador del cliente complete la verificación. Si quieres que los administradores de TI verifiquen un dominio, en lugar de que el dominio se considere **de confianza**, establece `domain_aliases_config.domain_verification` en `required` o `optional` para solicitar la verificación desde el asistente de autoservicio. La opción **Allow Use of Domains for Organization Discovery** requiere exactamente un `enabled_organization` en el ticket.

<div id="domains-to-be-verified">
  ##### Dominios que deben verificarse
</div>

Use los dominios que deben verificarse para indicar qué dominios deben verificar los administradores de TI durante la configuración. A diferencia de los dominios preverificados, los dominios pendientes no se asocian automáticamente con la conexión ni con la Organization. Los administradores de TI deben completar la verificación en el asistente de configuración antes de que estos surtan efecto.

Puede proporcionar dominios al [generar tickets de acceso de autoservicio](#generate-access-tickets-for-new-connections) a través de Auth0 Dashboard o de la Management API:

* **Auth0 Dashboard**: En la página **Generate Ticket**, especifique la lista de dominios en el campo **Domains to be Verified**.
* **Management API**: Configure `connection_config.options.domain_aliases` con la lista de dominios. De forma predeterminada, `use_for_organization_discovery` se establece en `true`. De forma opcional, seleccione **Allow the Use of Domains for Organization Discovery**

Se aplican los siguientes límites:

* Si la Organization está asociada al ticket, el total combinado de dominios existentes y pendientes de la Organization no debe superar los 100.
* Si no hay ninguna Organization asociada al ticket, o si hay más de una, el total combinado de alias de dominio existentes y pendientes no debe superar los 1.000.

Si agregar los dominios pendientes supera cualquiera de estos límites, la creación del ticket falla con un error.

<div id="email-domain-verification">
  ##### Verificación del dominio de correo electrónico
</div>

A diferencia de los dominios preverificados, la verificación del dominio de correo electrónico confirma que un administrador de TI es propietario del dominio que proporciona durante la configuración de autoservicio. Cuando se completa la verificación, el dominio verificado se agrega al mismo arreglo `options.domain_aliases` de la conexión.

Puede habilitar la verificación para administradores de TI al generar tickets de acceso de autoservicio:

* **Auth0 Dashboard**: En la página **Generate Ticket**, use el campo **Domain Verification Requirement**. De forma opcional, seleccione **Allow the Use of Domains for Organization Discovery**.
* **Management API**: Use `domain_aliases_config.domain_verification` y, opcionalmente, `use_for_organization_discovery ` con una de las siguientes opciones:
  * `none` (predeterminado): El asistente de autoservicio no solicita a un administrador del cliente que verifique su dominio.
  * `required`: El asistente de autoservicio solicita al administrador del cliente que verifique sus dominios.
  * `optional`: El asistente de autoservicio solicita al administrador del cliente que verifique su dominio. El administrador del cliente puede elegir entre ingresar su dominio para la verificación u omitir este paso.

La opción **Allow Use of Domains for Organization Discovery** requiere exactamente **una** Organization en el campo Enabled Organizations y que Domain Verification esté configurado como `Optional` o `Required`, o bien en Pre-Verified Domain(s). En algunos casos, la verificación puede tardar hasta 48 horas, y es posible que deba emitir un ticket de acceso adicional para que el administrador del cliente pueda volver y habilitar la conexión. Los tickets de acceso vencen cinco horas después de abrirse por primera vez. Consulte [Generar tickets de acceso para conexiones existentes](https://auth0.com/docs/authenticate/enterprise-connections/self-service-enterprise-configuration/manage-self-service-enterprise-config#generate-access-tickets-for-existing-connections).

<div id="remove-a-domain">
  ##### Eliminar un dominio
</div>

Los administradores de TI pueden eliminar dominios verificados desde el asistente de autoservicio. Se aplican las siguientes reglas según el Domain Verification Requirement establecido en el ticket:

Opcional: Se pueden eliminar todos los dominios verificados.
Obligatorio: Debe permanecer al menos un dominio verificado. Si el administrador de TI intenta eliminar el último dominio verificado, aparece una advertencia.

<div id="generate-access-tickets-for-new-connections">
  ### Generar tickets de acceso para nuevas conexiones
</div>

Puede generar tickets de acceso para nuevas conexiones desde el Auth0 Dashboard o la Management API.

<Warning>
  De forma predeterminada, las URL de los tickets de acceso son válidas durante cinco días desde su generación. Una vez que se accede a la URL del ticket, el administrador del cliente dispone de cinco horas para completar la configuración. Se puede acceder a una URL de ticket de acceso un máximo de 10 veces; una vez alcanzado este límite, se debe solicitar un nuevo ticket de acceso.

  Si es necesario, puede revocar un ticket de acceso antes de su vencimiento para cancelar de inmediato el acceso al asistente de autoservicio.
</Warning>

<Tabs>
  <Tab title="Auth0 Dashboard">
    Para generar un ticket de acceso para una nueva conexión a través del Auth0 Dashboard:

    1. Vaya a [Authentication &gt; Enterprise](https://manage.auth0.com/#/connections/enterprise) y acceda a la sección **Configuración empresarial de autoservicio**. Luego, seleccione el perfil de autoservicio con el que desea crear un ticket de acceso.

    2. Seleccione **Generar ticket** para abrir el formulario del ticket. En **Seleccionar tipo de ticket**, elija **Crear una nueva conexión**.

    3. En **Configuración del ticket**, indique un nombre obligatorio para la conexión que configurará el administrador del cliente.

    4. En la sección **Configuración**, configure las opciones adicionales que necesite para la nueva conexión:

       * **Dominio**: Seleccione el dominio personalizado que se usará en la URL del ticket. Solo está disponible cuando hay múltiples dominios personalizados.
       * **Nombre para mostrar**: Un nombre descriptivo para la conexión que se muestra en los prompts de Universal Login.
       * **Clientes habilitados**: Una lista de IDs de cliente separados por comas para asociarlos con la conexión.
       * **Organizaciones habilitadas**: Una lista de IDs de organización separados por comas para asociarlos con la conexión.
       * **Mostrar la conexión como un botón**: Muestra la conexión como una opción de autenticación en la pantalla de inicio de sesión.
       * **Mostrar la conexión como un botón para organizaciones**: Muestra la conexión como una opción de autenticación en la pantalla de inicio de sesión para las organizaciones especificadas.
       * **Asignar membresía al iniciar sesión para organizaciones**: Otorga automáticamente membresía a la organización a los usuarios que se autentican con la conexión.
       * **Habilitar como conexión de nivel de dominio**: Permite que aplicaciones de terceros usen la conexión; requiere [registro dinámico de clientes](/docs/es/get-started/applications/dynamic-client-registration).
       * **Aceptar SSO iniciado por IdP SAML**: Habilita [SSO iniciado por proveedor de identidad SAML](/docs/es/authenticate/protocols/saml/saml-sso-integrations/identity-provider-initiated-single-sign-on).

    5. En **Detección basada en dominios**, opcionalmente proporcione una lista de dominios de IdP, separados por comas, ya verificados o pendientes de verificación para compararlos con los dominios de correo electrónico de los usuarios. Estos dominios se almacenan en `options.domain_aliases` y controlan el HRD. Para obtener más información, consulte [Home Realm Discovery](#home-realm-discovery).

    6. En **Requisito de verificación de dominio**, elija el nivel de verificación que desee:
       * **Desactivado**: No se solicita a los administradores del cliente que verifiquen su dominio al configurar SSO. **Desactivado** es la configuración predeterminada para los nuevos tickets de acceso.
       * **Opcional**: Se solicita a los administradores del cliente que verifiquen su dominio al configurar SSO. Sin embargo, pueden omitir este paso y habilitar su conexión sin completar la verificación.
       * **Obligatorio**: Los administradores del cliente deben verificar su dominio al configurar SSO. No podrán habilitar su conexión hasta que se complete la verificación.

    7. En **Provisioning**, opcionalmente habilite **Sincronizar usuarios y perfiles de grupo mediante aprovisionamiento**. Al habilitarse, habrá opciones de configuración adicionales:
       * **Vencimiento del token Bearer**: Defina una fecha de vencimiento para el token Bearer de SCIM. De forma predeterminada, los tokens Bearer no vencen.
       * **Permisos del token Bearer (Scopes)**: Elija qué acciones puede realizar el token. De forma predeterminada, todos los scopes de aprovisionamiento están habilitados:
         * `get:users`
         * `post:users`
         * `put:users`
         * `patch:users`
         * `delete:users`
         * `get:groups`
         * `post:groups`
         * `put:groups`
         * `patch:groups`
         * `delete:groups`

    8. En **Tiempo de vida**, establezca un período de vencimiento para el ticket de acceso en segundos. De forma predeterminada, el tiempo de vida se establece en 432000 segundos (equivalente a cinco días).

       * El tiempo de vida determina durante cuánto tiempo una URL de ticket de acceso permanece activa **antes de que** un administrador del cliente inicie el asistente de autoservicio. No determina durante cuánto tiempo el administrador del cliente tendrá acceso al asistente después de haberlo iniciado. El vencimiento del propio asistente de autoservicio es de 5 horas y no se puede configurar.

    9. En **Metadata**, agregue hasta 10 metadatos asociados con la conexión.

    10. Revise la configuración del ticket de acceso para comprobar que sea correcta. Luego, seleccione **Crear ticket**.

    A continuación, se mostrará una ventana emergente de Información del ticket que contiene la URL del ticket de acceso. Copie y guarde esta URL en un lugar seguro, ya que no podrá recuperarla después de cerrar la ventana emergente.

    Puede compartir la URL del ticket de acceso con el administrador del cliente por correo electrónico, chat u otro canal de comunicación para darle acceso al asistente de autoservicio. Luego, el asistente lo guiará en la configuración de la conexión SSO. Para obtener más información sobre esa experiencia, consulte [Experiencia del asistente de autoservicio](/docs/es/authenticate/enterprise-connections/self-service-enterprise-configuration#self-service-assistant-experience).
  </Tab>

<div id="generate-access-tickets-for-existing-connections">
  ### Generar tickets de acceso para conexiones existentes
</div>

Puede generar tickets de acceso para conexiones existentes a través de Auth0 Dashboard o de la Management API.

<Warning>
  De forma predeterminada, las URL de los tickets de acceso siguen siendo válidas durante cinco días después de generarse. Tras acceder a la URL del ticket, el administrador del cliente dispone de cinco horas para completar la configuración. Se puede acceder a una URL de ticket de acceso un máximo de 10 veces; una vez alcanzado este límite, debe solicitarse un nuevo ticket de acceso.

  Si es necesario, puede revocar un ticket de acceso antes de que venza para interrumpir de inmediato el acceso al asistente de autoservicio.
</Warning>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Si un administrador del cliente inicia la **verificación del dominio** a través del asistente de autoservicio, es posible que necesite un ticket de acceso adicional para completar el proceso de configuración.

  La verificación del dominio se realiza en el paso final del asistente de autoservicio. En este punto del flujo de trabajo, la conexión ya se ha creado, pero aún no se ha habilitado. Si se requiere la verificación del dominio, el administrador del cliente no podrá habilitar su conexión hasta que esta se complete.

  Aunque la verificación suele completarse con rapidez, en algunos casos puede tardar entre 24 y 48 horas. Si esto ocurre, el administrador del cliente no podrá usar su ticket de acceso original para habilitar su conexión, ya que los tickets vencen cinco horas después del primer acceso.

  Para completar este proceso, puede generar un ticket de acceso que permita al administrador del cliente modificar la conexión que configuró con su ticket inicial. Al crear este ticket, asegúrese de especificar el ID de la conexión que configuró con su primer ticket de acceso.
</Callout>

Si la conexión se crea sin todos los scopes de aprovisionamiento de usuarios (`get:users`,`post:users`,`put:users`, `patch:users`,`delete:users`), SCIM no se habilitará. Los scopes de grupos son opcionales. Usa [`google_workspace`](https://auth0.com/docs/api/management/v2/self-service-profiles/post-sso-ticket) para configurar [Google Workspace Directory Sync](/docs/es/authenticate/identity-providers/enterprise-identity-providers/google-directory-sync).                                                                                                                                                                                                                                                                                                                                 |
    | `ttl_sec`                               | **Opcional**. Número.<br /><br />Número de segundos que una URL de ticket de acceso permanece activa antes de que un administrador del cliente inicie el asistente de autoservicio. Si no se especifica o se establece en `0`, el valor se establece en `432000` (equivalente a 5 días).<br /><br />Ten en cuenta que este período de vencimiento no determina durante cuánto tiempo un administrador del cliente puede acceder al asistente de autoservicio después de iniciarlo. El vencimiento del asistente en sí es de cinco horas y no se puede configurar.                                                                                                                                                                                                                                                                                                                                 |
    | `domain_aliases_config`                 | **Opcional**. Objeto.<br /><br />Contiene domain&#95;verification, que se usa para determinar si la verificación del dominio es obligatoria, opcional o está desactivada.<br /><br />Las opciones de domain&#95;verification incluyen:<br /><br /><ul><li>`none`: Desactiva la verificación del dominio. También puede desactivar la verificación del dominio si omite el objeto `domain_aliases_config` de su request.</li><li>`optional`: Permite que los administradores del cliente omitan la verificación del dominio durante la configuración.</li><li>`required`: Requiere que los administradores del cliente verifiquen su dominio durante la configuración.</li></ul>                                                                                                                                                                                                                   |
    | `use_for_organization_discovery`        | **Opcional**. Boolean. <br /><br />Indica si se debe usar un dominio verificado para el descubrimiento de organizaciones durante la autenticación.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                |

    **Ejemplo de cuerpo de solicitud**

    ```json lines
    {
      "connection_id": "string",
       "ttl_sec":0,
       "domain_aliases_config": {
           "domain_verification": "string"
           "pending_domains": [
                 "acme.com"
              ]
        }
    }
    ```

    En respuesta, recibes una URL del ticket de acceso de autoservicio:

    ```json lines
    {
      "ticket": "https://{domain}/self-service/connections-flow?ticket={id}"
    }
    ```

    Una vez que reciba la URL del ticket, comparta el enlace con el administrador de su cliente para otorgarle acceso al asistente de autoservicio. El asistente lo guiará por el proceso de configuración de la conexión SSO. Para obtener más información sobre esta experiencia, consulte [Experiencia del asistente de autoservicio](/docs/es/authenticate/enterprise-connections/self-service-enterprise-configuration#self-service-assistant-experience).

    Puedes integrar la generación de tickets de acceso en tu propio portal de autoservicio o enviar las URLs de los tickets directamente a los administradores de clientes por correo electrónico, chat u otros canales de comunicación.

<div id="revoke-an-access-ticket">
  ### Revocar un ticket de acceso
</div>

De forma predeterminada, la URL de un ticket de acceso permanece válida durante cinco días. Al acceder a la URL, un administrador del cliente dispone de cinco horas para completar la configuración.

Si es necesario, puedes revocar un ticket de acceso antes de que caduque. Por ejemplo, si un ticket de acceso se comparte con la <Tooltip tip="Audience: Identificador único de la audiencia de un token emitido. Denominado aud en un token, su valor contiene el ID de una aplicación (ID de cliente) para un token de ID o de una API (identificador de API) para un token de acceso." cta="Ver glosario" href="/docs/es/glossary?term=audience">audiencia</Tooltip> incorrecta, puedes revocar el ticket para evitar el acceso no autorizado al asistente de autoservicio.

Cuando revocas un ticket de acceso, su URL se invalida de inmediato y se cierran todas las sesiones asociadas. Los administradores del cliente que tengan la URL ya no podrán acceder al asistente de autoservicio. Después, puedes generar y compartir nuevos tickets de acceso según sea necesario.

Para revocar un ticket de acceso:

1. Obtén el ID del perfil de autoservicio asociado al ticket de acceso mediante el endpoint [Retrieve Self-Service Profiles](https://auth0.com/docs/api/management/v2/self-service-profiles/get-self-service-profiles).
2. Localiza el ID del ticket de acceso que deseas revocar. Los ID se encuentran al final de la URL del ticket de acceso.
3. Llama al endpoint [Revoke SSO Access Ticket](https://auth0.com/docs/api/management/v2/self-service-profiles/post-revoke) con los ID correspondientes:

`POST  /api/v2/self-service-profiles/{id}/sso-ticket/{id}/revoke`

Como respuesta, se devuelve un `202 Accepted`.
