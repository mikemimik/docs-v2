Self-Service Enterprise Configuration proporciona a los clientes B2B las herramientas necesarias para delegar la configuración del SSO a sus clientes empresariales y requiere una configuración mínima en tu tenant de Auth0 para ofrecerles un asistente de autoservicio que los guíe durante el proceso de habilitación.

Una vez que un cliente completa la configuración, la integración de SSO se agrega automáticamente a tu tenant como una [conexión empresarial](/docs/es/authenticate/enterprise-connections).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Los usuarios con los siguientes roles del Dashboard pueden usar esta funcionalidad:

  * Los usuarios **Admin** y **Editor - Connections** pueden crear y administrar perfiles de autoservicio.
  * Los usuarios **Viewer - Config** solo pueden ver perfiles de autoservicio.
</Callout>

Para facilitar Self-Service Enterprise Configuration, configurarás los siguientes componentes mediante la <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip> o <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip>:

* **Perfil de autoservicio**: Define elementos clave de las implementaciones de SSO del cliente, incluidos los <Tooltip tip="Proveedor de identidad (IdP): Servicio que almacena y administra identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=identity+providers">proveedores de identidad</Tooltip> (IdPs) que pueden usar y los atributos de usuario que deben capturar, como el correo electrónico. Puedes crear hasta 20 perfiles en tu tenant para diferentes clientes o segmentos.
* **Ticket de acceso de autoservicio**: Otorga a los administradores del cliente acceso al [**asistente de autoservicio**](/docs/es/authenticate/enterprise-connections/self-service-enterprise-configuration#self-service-assistant-experience) y establece detalles específicos para la conexión empresarial resultante. Los tickets de acceso permiten a los administradores del cliente crear nuevas conexiones o modificar las existentes.

Las siguientes secciones ofrecen pasos más detallados para configurar perfiles de autoservicio y generar tickets de acceso de autoservicio para compartir con los administradores del cliente.

<div id="create-self-service-profiles">
  ## Crear perfiles de autoservicio
</div>

Puedes crear perfiles de autoservicio con Auth0 Dashboard o la Management API.

Los perfiles de autoservicio se utilizan para definir elementos clave de las implementaciones de los clientes, entre ellos:

* Qué proveedores de identidad pueden usar los administradores de clientes para SSO.
* Qué atributos de usuario deben recopilar mediante SSO, como el correo electrónico o el apellido.
* Opciones de branding que personalizan la apariencia y la experiencia del asistente de autoservicio.

Puedes crear hasta 20 perfiles, según sea necesario, para distintos clientes o segmentos.

<Tabs>
  <Tab title="Auth0 Dashboard">
    Para crear un perfil de autoservicio en el Auth0 Dashboard: 

    1. Vaya a [Authentication &gt; empresarial](https://manage.auth0.com/#/connections/enterprise) y abra la sección **Self-Service Enterprise Configuration**. Luego, seleccione **Create Profile**.
    2. En el espacio provisto, ingrese un nombre y una descripción opcional para el perfil. Luego, seleccione **Create**.<br />
          A. **Opcional** Adjunte un User Attribute Profile.
          <Callout icon="file-lines" color="#0EA5E9" iconType="regular">
          Si crea y adjunta, o habilita, un User Attribute Profile existente, no tendrá la opción de agregar atributos a través de la pestaña User Profile.
          </Callout>
             1. Seleccione un UAP existente o cree uno nuevo. Para un UAP nuevo:<br />
                a. Agregue un nombre.<br />
                b. Revise los mapeos para asegurarse de que los atributos del perfil se asignen a los atributos de Auth0 de su preferencia.
                   <Frame>![Mapeo de UAP Authentication > Enterprise > Self-Service](/docs/images/cdy7uua7fh8z/Authenticate>Enterprise>Self-Service>UAP.png)</Frame>
    3. En la pestaña **Settings**, complete las secciones a continuación. Luego, seleccione **Save**.
       * **Identity Providers (IdP)**: Habilite uno o más proveedores de identidad. En el asistente de autoservicio, los administradores del cliente podrán seleccionar la opción que prefieran de la lista de proveedores habilitados.
       * **Branding**: Proporcione un logotipo y un color principal para el asistente de autoservicio.
       * **Introducción personalizada**: Modifique o reemplace el mensaje predeterminado según sea necesario. Este texto introductorio se muestra a los administradores del cliente en la página de destino del asistente de autoservicio. El mensaje puede incluir opciones básicas de formato, como texto en negrita o hipervínculos, y está limitado a 2000 caracteres.
    4. En la pestaña **User Profile**, agregue hasta 20 atributos de usuario que sus clientes deban capturar mediante SSO, como el correo electrónico o el apellido. Puede configurar cada atributo como `required` u `optional`.

       * Durante el flujo del asistente de autoservicio, se pedirá a los administradores del cliente que asignen estos atributos de usuario definidos a su proveedor de identidad para garantizar que los valores necesarios se envíen a Auth0.

    #### Personaliza tu texto de introducción

    Cuando un administrador cliente accede al self-service assistant, primero llega a una página de introducción que le da la bienvenida. De forma predeterminada, se muestra el siguiente mensaje:

    &quot;Estás a pocos pasos de configurar el SSO. Este proceso de configuración implica realizar algunos cambios en tu proveedor de identidad. Antes de comenzar, abre tu proveedor de identidad en una pestaña o ventana separada del navegador.&quot;

    Puedes modificar este texto realizando una llamada PUT al endpoint [Custom Text for Self-Service Profiles](https://auth0.com/docs/api/management/v2/self-service-profiles/put-self-service-profile-custom-text).

    <Warning>
      Ten en cuenta que esta llamada **sobrescribe** cualquier mensaje configurado actualmente para el perfil de autoservicio. Asegúrate de que la llamada incluya el texto completo que quieres mostrar a los administradores de tus clientes.
    </Warning>

    1. Haz una llamada a `PUT /api/v2/self-service-profiles/{id}/custom-text/{language}/{page}`, donde

       * `id` es el ID del perfil de autoservicio
       * `language` se establece en `en`
       * `page` se establece en `get-started`
    2. En el cuerpo de la solicitud, indica lo siguiente:

    | Propiedad      | Descripción                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
    | -------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
    | `introduction` | <ul><li>String. La longitud máxima es de 2000.</li><li>Texto completo de introducción que se mostrará en la página de destino del asistente de autoservicio. El texto puede incluir opciones básicas de formato, como negrita o hipervínculos.</li><li>El texto personalizado proporcionado mediante este parámetro sobrescribe por completo cualquier texto anterior. Para obtener los mejores resultados, asegúrese de proporcionar el mensaje completo que desea mostrar a los administradores del cliente.</li><li>Enviar un cuerpo vacío `\{}` restablece cualquier texto personalizado al texto predeterminado. </li></ul> |

    3. Como respuesta, se devuelve la entidad creada.

    **Ejemplo de llamada**

    ```js lines
    PUT /api/v2/self-service-profiles/ssp_1234567890/custom-text/en/get-started

    {
      introduction: "Welcome! With <b>only a few steps</b>, you'll be able to setup your new connection. For assistance, contact <a href="https://www.examplesupportsite.com"> our support team </a>." 
    }
    ```

    **Ejemplo de respuesta**

    ```js lines
    {
      introduction: "Welcome! With <b>only a few steps</b>, you'll be able to setup your new connection. For assistance, contact <a href="https://www.examplesupportsite.com"> our support team </a>." 
    }
    ```

<div id="manage-self-service-access-tickets">
  ## Administrar tickets de acceso de autoservicio
</div>

Después de crear al menos un perfil de autoservicio, puedes generar tickets de acceso de autoservicio con Auth0 Dashboard o la Management API.

Los tickets de acceso de autoservicio tienen dos propósitos principales:

* Dar a los administradores del cliente acceso al asistente de autoservicio, donde pueden configurar una nueva SSO connection o modificar una conexión existente.
* Predefinir detalles y comportamientos clave de las nuevas SSO connections que configurarán los administradores de tu cliente, como qué aplicaciones u organizaciones se habilitarán para la nueva conexión.

Al generar access tickets, también puedes habilitar ciertas funcionalidades, como <Tooltip tip="Security Assertion Markup Language (SAML): Protocolo estandarizado que permite a dos partes intercambiar información de autenticación sin una contraseña." cta="Ver glosario" href="/docs/es/glossary?term=SAML">SAML</Tooltip> SSO iniciado por IdP, configurar dominios del proveedor de identidad (que determinan Home Realm Discovery) y la verificación de dominio.

<div id="saml-idp-initiated-sso">
  #### SSO iniciado por IdP de SAML
</div>

El SSO iniciado por IdP de SAML es un tipo de implementación que permite que los proveedores de identidad inicien el SSO y redirijan a los usuarios al proveedor de servicios para la autenticación.

Al habilitar esta opción para Self-Service Enterprise Configuration, debes proporcionar tu aplicación predeterminada y el protocolo de respuesta. También puedes proporcionar una cadena de consulta opcional para personalizar aún más el comportamiento de la conexión.

Para obtener más información sobre estas opciones, consulta [Configurar el inicio de sesión único iniciado por proveedor de identidad SAML](/docs/es/authenticate/protocols/saml/saml-sso-integrations/identity-provider-initiated-single-sign-on).

<div id="email-domain-verification-and-pre-verified-domains">
  #### Verificación del dominio de correo electrónico y dominios preverificados
</div>

Self-Service Enterprise Configuration admite tres formas de asociar dominios de correo electrónico con una conexión empresarial. Estos métodos completan el mismo campo: `options.domain_aliases `, que controla Home Realm Discovery (HRD) y, de forma condicional, los dominios de organización para descubrimiento:

1. Dominios preverificados (gestionados por el tenant): El administrador del tenant agrega directamente a la conexión los dominios conocidos.
2. Dominios por verificar: El administrador del tenant especifica los dominios que el administrador de TI debe verificar
   durante el Setup. Estos dominios figuran como pendientes en la organización y/o no se asocian automáticamente con la conexión hasta que el administrador de TI complete la verificación.
3. Verificación del dominio de correo electrónico (autogestionada): El administrador de su cliente verifica el dominio durante el Setup en el asistente de autoservicio.

Estos mecanismos para asociar dominios de correo electrónico se aplican a los dominios de correo electrónico usados para el enrutamiento de HRD y SSO. No están relacionados con la verificación de Custom Domain a nivel de tenant.

Para que un dominio se complete tanto con dominios de organización para descubrimiento como con HRD, el ticket debe:

* Incluir solo un `enabled_organization`
* Tener dominios preverificados o Email Verification Required como `Required` u `Optional`
* Habilitar la casilla **Allow the Use of Domains for Organization Discovery**

<div id="pre-verified-domain">
  ##### Dominio preverificado
</div>

Usa un dominio preverificado para asociar dominios de correo electrónico a una conexión empresarial a nivel del tenant. Los dominios añadidos se consideran de confianza y se escriben directamente en `options.domain_aliases`.

Puedes proporcionar dominios al [generar tickets de acceso de autoservicio](#generate-access-tickets-for-new-connections) a través del Auth0 Dashboard o de la Management API.

* **Auth0 Dashboard**: En la página **Generate Ticket**, especifica tu lista de dominios en el campo **Dominios preverificados**.
* **Management API**: Establece `connection_config.options.domain_aliases` con la lista de dominios. De forma predeterminada, `use_for_organization_discovery` se establece en `true`. Opcionalmente, selecciona **Allow the Use of Domains for Organization Discovery**.

Los dominios añadidos por los administradores del tenant se consideran **de confianza** y **no** requieren que el administrador del cliente complete la verificación. Si quieres que los administradores de TI verifiquen un dominio en lugar de considerarlo **de confianza**, establece `domain_aliases_config.domain_verification` en `required` u `optional` para solicitar la verificación desde el asistente de self-service. La opción **Allow Use of Domains for Organization Discovery** requiere exactamente un `enabled_organization` en el ticket.

<div id="domains-to-be-verified">
  ##### Dominios para verificar
</div>

Use los dominios para verificar para especificar qué dominios deben verificar los administradores de TI durante la configuración. A diferencia de los dominios preverificados, los dominios pendientes no se asocian automáticamente con la conexión ni con la Organización. Los administradores de TI deben completar la verificación en el asistente de configuración antes de que surtan efecto.

Puede proporcionar dominios al [generar tickets de acceso de autoservicio](#generate-access-tickets-for-new-connections) a través del Auth0 Dashboard o de la Management API:

* **Auth0 Dashboard**: En la página **Generate Ticket**, especifique la lista de dominios en el campo **Dominios para verificar**.
* **Management API**: Establezca `connection_config.options.domain_aliases` en la lista de dominios. De forma predeterminada, `use_for_organization_discovery` está establecido en `true`. Opcionalmente, seleccione **Allow the Use of Domains for Organization Discovery**

Se aplican los siguientes límites:

* Si la Organización está asociada con el ticket, el total combinado de dominios de la Organización existentes y pendientes no debe superar los 100.
* Si ninguna Organización o más de una Organización está asociada con el ticket, el total combinado de alias de dominio existentes y pendientes no debe superar los 1,000.

Si al agregar los dominios pendientes se supera cualquiera de estos límites, la creación del ticket falla con un error.

<div id="email-domain-verification">
  ##### Verificación del dominio de correo electrónico
</div>

A diferencia de los dominios preverificados, la verificación del dominio de correo electrónico confirma que un administrador de TI es propietario del dominio que proporciona durante la configuración de autoservicio. Cuando se completa la verificación, el dominio verificado se agrega al mismo arreglo `options.domain_aliases` de la conexión.

Puede habilitar la verificación para administradores de TI cuando genere tickets de acceso de autoservicio:

* **Auth0 Dashboard**: En la página **Generate Ticket**, use el campo **Domain Verification Requirement**. Opcionalmente, seleccione **Allow the Use of Domains for Organization Discovery**.
* **Management API**: Use `domain_aliases_config.domain_verification` y, opcionalmente, `use_for_organization_discovery ` con una de las siguientes opciones:
  * `none` (predeterminado): El asistente de autoservicio no solicita al administrador del cliente que verifique su dominio.
  * `required`: El asistente de autoservicio solicita al administrador del cliente que verifique sus dominios.
  * `optional`: El asistente de autoservicio solicita al administrador del cliente que verifique su dominio. El administrador del cliente puede optar por ingresar su dominio para verificarlo u omitir este paso.

Las opciones de **Allow Use of Domains for Organization Discovery** requieren exactamente **una** organización en el campo Enabled Organizations y que Domain Verification esté configurado como `Optional` o `Required`, o bien que haya dominios preverificados. En algunos casos, la verificación puede tardar hasta 48 horas, y es posible que deba emitir un ticket de acceso adicional para que el administrador del cliente pueda volver y habilitar la conexión. Los tickets de acceso vencen cinco horas después de abrirse por primera vez. Consulte [Generate access tickets for existing connections](https://auth0.com/docs/authenticate/enterprise-connections/self-service-enterprise-configuration/manage-self-service-enterprise-config#generate-access-tickets-for-existing-connections).

<div id="remove-a-domain">
  ##### Eliminar un dominio
</div>

Los administradores de TI pueden eliminar dominios verificados desde el asistente de autoservicio. Se aplican las siguientes reglas según el Domain Verification Requirement establecido en el ticket:

Opcional: se pueden eliminar todos los dominios verificados.
Obligatorio: debe mantenerse al menos un dominio verificado. Aparece una advertencia si el administrador de TI intenta eliminar el último dominio verificado.

<div id="generate-access-tickets-for-new-connections">
  ### Generar tickets de acceso para nuevas conexiones
</div>

Puede generar tickets de acceso para nuevas conexiones a través del Auth0 Dashboard o de la Management API.

<Warning>
  De forma predeterminada, las URL de los tickets de acceso siguen siendo válidas durante cinco días después de generarse. Después de acceder a la URL del ticket, el administrador del cliente dispone de cinco horas para completar la configuración. Se puede acceder a una URL de ticket de acceso un máximo de 10 veces; una vez alcanzado este límite, debe solicitarse un nuevo ticket de acceso.

  Si es necesario, puede revocar un ticket de acceso antes de que caduque para interrumpir de inmediato el acceso al asistente de autoservicio.
</Warning>

<Tabs>
  <Tab title="Auth0 Dashboard">
    Para generar un ticket de acceso para una nueva conexión a través del Auth0 Dashboard:

    1. Vaya a [Authentication &gt; Enterprise](https://manage.auth0.com/#/connections/enterprise) y acceda a la sección **Self-Service Enterprise Configuration**. Luego, seleccione el perfil de autoservicio con el que desea crear un ticket de acceso.

    2. Seleccione **Generate Ticket** para abrir el formulario del ticket. En **Select ticket type**, elija **Create a new connection**.

    3. En **Ticket configuration**, indique un nombre obligatorio para la conexión que configurará el administrador del cliente.

    4. En la sección **Settings**, configure opciones adicionales según sea necesario para la nueva conexión:

       * **Domain**: Seleccione el dominio personalizado que se usará en la URL del ticket. Solo está disponible cuando hay múltiples dominios personalizados.
       * **Display Name**: Un nombre descriptivo para la conexión que se muestra en los prompts de Universal Login.
       * **Enabled Clients**: Una lista de ID de cliente separados por comas para asociarlos con la conexión.
       * **Enabled Organizations**: Una lista de ID de organización separados por comas para asociarlos con la conexión.
       * **Display connection a as button**: Muestra la conexión como una opción de autenticación en la pantalla de login.
       * **Display connection as a button for organizations**: Muestra la conexión como una opción de autenticación en la pantalla de login para las organizaciones especificadas.
       * **Assign membership on login for organizations**: Otorga automáticamente membresía en la organización a los usuarios que se autentican con la conexión.
       * **Enable as a domain level connection**: Permite que aplicaciones de terceros usen la conexión; requiere [registro dinámico de clientes](/docs/es/get-started/applications/dynamic-client-registration).
       * **Accept SAML IdP-initiated SSO**: Habilita el [SSO iniciado por el proveedor de identidad SAML](/docs/es/authenticate/protocols/saml/saml-sso-integrations/identity-provider-initiated-single-sign-on).

    5. En **Domain-Based Discovery**, opcionalmente proporcione una lista separada por comas de dominios de IdP ya verificados o pendientes de verificación para compararlos con los dominios de correo electrónico de los usuarios. Estos dominios se almacenan en `options.domain_aliases` y controlan el HRD. Para obtener más información, consulte [Home Realm Discovery](#home-realm-discovery).

    6. En **Domain Verification Requirement**, elija el nivel de verificación que desee:
       * **Off**: No se solicita a los administradores del cliente que verifiquen su dominio al configurar SSO. **Off** es la configuración predeterminada para los tickets de acceso nuevos.
       * **Optional**: Se solicita a los administradores del cliente que verifiquen su dominio al configurar SSO. Sin embargo, pueden omitir este paso y habilitar su conexión sin completar la verificación.
       * **Required**: Los administradores del cliente deben verificar su dominio al configurar SSO. No podrán habilitar su conexión hasta que la verificación se haya completado.

    7. En **Provisioning**, opcionalmente habilite **Sync users and group profiles using provisioning**. Cuando esta opción está habilitada, hay configuraciones adicionales disponibles:
       * **Bearer Token Expiration**: Defina una fecha de vencimiento para el bearer token de SCIM. De forma predeterminada, los bearer tokens no vencen.
       * **Bearer Token Permissions (Scopes)**: Elija qué acciones puede realizar el token. De forma predeterminada, todos los scopes de aprovisionamiento están habilitados:
         * `get:users`
         * `post:users`
         * `put:users`
         * `patch:users`
         * `delete:users`
         * `get:groups`
         * `post:groups`
         * `put:groups`
         * `patch:groups`
         * `delete:groups`

    8. En **Time to Live**, establezca un período de vencimiento para el ticket de acceso en segundos. De forma predeterminada, el tiempo de vida se establece en 432000 segundos (equivalente a cinco días).

       * Time to Live determina durante cuánto tiempo una URL de ticket de acceso permanece activa **antes de que** un administrador del cliente inicie el asistente de autoservicio. No determina durante cuánto tiempo el administrador del cliente tendrá acceso al asistente después de iniciarlo. El vencimiento del asistente de autoservicio en sí es de 5 horas y no se puede configurar.

    9. En **Metadata**, agregue hasta 10 entradas de metadatos asociadas con la conexión.

    10. Revise la configuración del ticket de acceso para confirmar que sea correcta. Luego, seleccione **Create Ticket**.

    A continuación, se mostrará un popup de Ticket Information que contiene la URL del ticket de acceso. Copie y guarde esta URL en un lugar seguro, ya que no podrá recuperarla nuevamente después de cerrar el popup.

    Puede compartir la URL del ticket de acceso con el administrador de su cliente por correo electrónico, chat u otro canal de comunicación para otorgarle acceso al asistente de autoservicio. Luego, el asistente lo guiará durante la configuración de la conexión SSO. Para obtener más información sobre esa experiencia, consulte [Self-service assistant experience](/docs/es/authenticate/enterprise-connections/self-service-enterprise-configuration#self-service-assistant-experience).
  </Tab>

<div id="generate-access-tickets-for-existing-connections">
  ### Generar tickets de acceso para conexiones existentes
</div>

Puede generar tickets de acceso para conexiones existentes a través del Auth0 Dashboard o de la Management API.

<Warning>
  De forma predeterminada, las URL de los tickets de acceso siguen siendo válidas durante cinco días después de generarse. Después de acceder a la URL del ticket, el administrador del cliente tiene cinco horas para completar la configuración. Se puede acceder a la URL de un ticket un máximo de 10 veces; una vez alcanzado este límite, se debe solicitar un nuevo ticket de acceso.

  Si es necesario, puede revocar un ticket de acceso antes de que venza para interrumpir de inmediato el acceso al asistente de autoservicio.
</Warning>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Si un administrador del cliente inicia la **verificación de dominio** a través del asistente de autoservicio, es posible que necesite un ticket de acceso adicional para completar el proceso de configuración.

  La verificación de dominio se realiza como paso final del asistente de autoservicio. En este punto del flujo de trabajo, la conexión ya se creó, pero no se habilitó. Si se requiere la verificación de dominio, el administrador del cliente no podrá habilitar su conexión hasta que esta se complete.

  Aunque la verificación suele completarse en un plazo razonable, en algunos casos puede tardar entre 24 y 48 horas. Si esto ocurre, el administrador del cliente no podrá usar su ticket de acceso original para habilitar su conexión, ya que los tickets vencen cinco horas después de acceder a ellos por primera vez.

  Para completar este proceso, puede generar un ticket de acceso que permita al administrador del cliente modificar la conexión que configuró con su ticket inicial. Al crear este ticket, asegúrese de especificar el ID de la conexión que configuró con su primer ticket de acceso.
</Callout>

Si no se especifica o se establece en `0`, el valor será `432000` de forma predeterminada (equivalente a 5 días).<br /><br />Ten en cuenta que este período de vencimiento no determina durante cuánto tiempo un administrador del cliente tiene acceso al self-service assistant después de iniciarlo. El vencimiento del propio assistant es de cinco horas y no se puede configurar.                                                                                                                                                                                                                                                                                                       |
    | `domain_aliases_config`                 | **Opcional**. Objeto.<br /><br />Contiene domain&#95;verification, que se utiliza para determinar si la verificación de dominio es obligatoria, opcional o está desactivada.<br /><br />Las opciones de domain&#95;verification incluyen:<br /><br /><ul><li>`none`: Desactiva la verificación de dominio. También puede desactivar la verificación de dominio si omite el objeto `domain_aliases_config` de su solicitud.</li><li>`optional`: Permite que los administradores del cliente omitan la verificación de dominio durante la configuración.</li><li>`required`: Exige que los administradores del cliente verifiquen su dominio durante la configuración.</li></ul>                                                                                                                                                                                           |
    | `use_for_organization_discovery`        | **Opcional**. Booleano. <br /><br />Indica si debe usarse un dominio verificado para descubrir la organización durante la autenticación.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                 |

    **Ejemplo de cuerpo de solicitud**

    ```json lines
    {
      "connection_id": "string",
       "ttl_sec":0,
       "domain_aliases_config": {
           "domain_verification": "string"
           "pending_domains": [
                 "acme.com"
              ]
        }
    }
    ```

    En respuesta, recibirás una URL del ticket de acceso de autoservicio:

    ```json lines
    {
      "ticket": "https://{domain}/self-service/connections-flow?ticket={id}"
    }
    ```

    Una vez que reciba la ticket URL, comparta el enlace con el administrador de su cliente para otorgarle acceso al asistente de autoservicio. El asistente lo guiará por el proceso de configuración de la conexión SSO. Para obtener más información sobre esta experiencia, consulte [Self-service assistant experience](/docs/es/authenticate/enterprise-connections/self-service-enterprise-configuration#self-service-assistant-experience).

    Puedes integrar la generación de access tickets en tu propio portal de autoservicio o enviar las ticket URLs directamente a los administradores de clientes por correo electrónico, chat u otros canales de comunicación.
  </Tab>
</Tabs>

<div id="revoke-an-access-ticket">
  ### Revocar un ticket de acceso
</div>

Por defecto, la URL de un ticket de acceso sigue siendo válida durante cinco días. Una vez que se accede a la URL, el administrador del cliente dispone de cinco horas para completar la configuración.

Si es necesario, puedes revocar un ticket de acceso antes de que venza. Por ejemplo, si un ticket de acceso se comparte con la <Tooltip tip="Audience: Identificador único de la audiencia de un token emitido. Se llama aud en un token, y su valor contiene el ID de una aplicación (ID de cliente) para un token de ID o de una API (identificador de API) para un token de acceso." cta="Ver glosario" href="/docs/es/glossary?term=audience">audiencia</Tooltip> equivocada, puedes revocar el ticket para evitar el acceso no autorizado al asistente de autoservicio.

Cuando revocas un ticket de acceso, su URL deja de ser válida de inmediato y se cierran todas las sesiones asociadas. Los administradores del cliente que tengan la URL ya no podrán acceder al asistente de autoservicio. Después, puedes generar y compartir nuevos tickets de acceso según sea necesario.

Para revocar un ticket de acceso:

1. Recupera el ID del perfil de autoservicio asociado al ticket de acceso mediante el endpoint [Retrieve Self-Service Profiles](https://auth0.com/docs/api/management/v2/self-service-profiles/get-self-service-profiles).
2. Localiza el ID del ticket de acceso que quieres revocar. Los ID se encuentran al final de la URL del ticket de acceso.
3. Llama al endpoint [Revoke SSO Access Ticket](https://auth0.com/docs/api/management/v2/self-service-profiles/post-revoke) con los ID correspondientes:

`POST  /api/v2/self-service-profiles/{id}/sso-ticket/{id}/revoke`

Como respuesta, se devuelve un `202 Accepted`.
