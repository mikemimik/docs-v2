Self-Service Enterprise Configuration proporciona a los clientes B2B las herramientas necesarias para delegar la configuración del SSO en sus clientes empresariales y requiere una configuración mínima en tu tenant de Auth0 para ofrecerles un asistente de autoservicio que los guíe durante el proceso de habilitación.

Cuando un cliente completa la configuración, la integración de SSO se agrega automáticamente a tu tenant como una [conexión empresarial](/docs/es/authenticate/enterprise-connections).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Los usuarios con los siguientes roles del Dashboard pueden utilizar esta funcionalidad:

  * Los usuarios **Admin** y **Editor - Connections** pueden crear y gestionar perfiles de autoservicio.
  * Los usuarios **Viewer - Config** solo pueden ver perfiles de autoservicio.
</Callout>

Para facilitar Self-Service Enterprise Configuration, configurarás los siguientes componentes mediante la <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip> o el <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip>:

* **Perfil de autoservicio**: Define elementos clave de las implementaciones de SSO de los clientes, incluidos los <Tooltip tip="Proveedor de identidad (IdP): Servicio que almacena y gestiona identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=identity+providers">proveedores de identidad</Tooltip> (IdP) que pueden usar y los atributos de usuario que deben recopilar, como el correo electrónico. Puedes crear hasta 20 perfiles en tu tenant para distintos clientes o segmentos.
* **Ticket de acceso de autoservicio**: Otorga a los administradores del cliente acceso al [**asistente de autoservicio**](/docs/es/authenticate/enterprise-connections/self-service-enterprise-configuration#self-service-assistant-experience) y establece detalles específicos para la conexión empresarial resultante. Los tickets de acceso permiten a los administradores del cliente crear nuevas conexiones o modificar las existentes.

Las secciones siguientes ofrecen pasos más detallados para configurar perfiles de autoservicio y generar tickets de acceso de autoservicio para compartir con los administradores del cliente.

<div id="create-self-service-profiles">
  ## Crear perfiles de autoservicio
</div>

Puedes crear perfiles de autoservicio con Auth0 Dashboard o la Management API.

Los perfiles de autoservicio se usan para definir elementos clave de las implementaciones de los clientes, entre ellos:

* Qué proveedores de identidad pueden usar los administradores de clientes para SSO.
* Qué atributos de usuario deben recopilar mediante SSO, como el correo electrónico o el apellido.
* Opciones de branding que personalizan la apariencia del asistente de autoservicio.

Puedes crear hasta 20 perfiles, según sea necesario, para distintos clientes o segmentos.

<Tabs>
  <Tab title="Auth0 Dashboard">
    Para crear un perfil de autoservicio en el Auth0 Dashboard: 

    1. Vaya a [Authentication &gt; empresarial](https://manage.auth0.com/#/connections/enterprise) y abra la sección **Self-Service Enterprise Configuration**. Luego, seleccione **Create Profile**.
    2. En el campo correspondiente, introduzca un nombre y una descripción opcional para el perfil. Luego, seleccione **Create**.<br />
          A. **Opcional** Adjunte un User Attribute Profile.
          <Callout icon="file-lines" color="#0EA5E9" iconType="regular">
          Si crea y adjunta, o habilita, un User Attribute Profile existente, no tendrá la opción de agregar atributos desde la pestaña User Profile.
          </Callout>
             1. Seleccione un UAP existente o cree uno nuevo. Para un UAP nuevo:<br />
                a. Agregue un nombre.<br />
                b. Revise los mapeos para asegurarse de que los atributos del perfil se asignen a los atributos de Auth0 de su preferencia.
                   <Frame>![Mapeo de UAP Authentication > Enterprise > Self-Service](/docs/images/cdy7uua7fh8z/Authenticate>Enterprise>Self-Service>UAP.png)</Frame>
    3. En la pestaña **Settings**, complete las secciones siguientes. Luego, seleccione **Save**.
       * **Identity Providers (IdP)**: Habilite uno o más proveedores de identidad. En el asistente de autoservicio, los administradores de sus clientes pueden seleccionar la opción que prefieran de la lista de proveedores habilitados.
       * **Branding**: Proporcione un logotipo y un color principal para el asistente de autoservicio.
       * **Custom Introduction**: Modifique o reemplace el mensaje predeterminado según sea necesario. Este texto introductorio se muestra a los administradores de sus clientes en la página de destino del asistente de autoservicio. El mensaje puede incluir opciones básicas de formato, como negrita o hipervínculos, y está limitado a 2000 caracteres.
    4. En la pestaña **User Profile**, agregue hasta 20 atributos de usuario que sus clientes deban recopilar a través de SSO, como correo electrónico o apellido. Puede establecer cada atributo como `required` u `optional`.

       * Durante el flujo del asistente de autoservicio, se pedirá a los administradores de sus clientes que mapeen estos atributos de usuario definidos con su proveedor de identidad para garantizar que los valores necesarios se envíen a Auth0.

    #### Personaliza tu texto de introducción

    Cuando un administrador cliente accede al self-service assistant, primero llega a una página de introducción que le da la bienvenida. De forma predeterminada, se muestra el siguiente mensaje:

    &quot;Estás a pocos pasos de configurar SSO. Este proceso de configuración implica realizar algunos cambios en tu proveedor de identidad. Antes de comenzar, abre tu proveedor de identidad en una pestaña o ventana separada del navegador.&quot;

    Puedes modificar este texto realizando una llamada PUT al endpoint [Custom Text for Self-Service Profiles](https://auth0.com/docs/api/management/v2/self-service-profiles/put-self-service-profile-custom-text).

    <Warning>
      Ten en cuenta que esta solicitud **sobrescribe** cualquier mensaje configurado actualmente en el perfil de autoservicio. Asegúrate de que la solicitud incluya el texto completo que quieres mostrar a los administradores de tus clientes.
    </Warning>

    1. Haz una solicitud a `PUT /api/v2/self-service-profiles/{id}/custom-text/{language}/{page}`, donde

       * `id` es el ID del perfil de autoservicio
       * `language` está configurado en `en`
       * `page` está configurado en `get-started`
    2. En el cuerpo de la solicitud, especifica lo siguiente:

    | Propiedad      | Descripción                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                    |
    | -------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
    | `introduction` | <ul><li>String. La longitud máxima es de 2000.</li><li>Texto introductorio completo que se mostrará en la página de destino del asistente de autoservicio. El texto puede incluir opciones básicas de formato, como negrita o hipervínculos.</li><li>El texto personalizado proporcionado mediante este parámetro sobrescribe por completo cualquier texto anterior. Para obtener los mejores resultados, asegúrese de proporcionar el mensaje completo que desea mostrar a los administradores del cliente.</li><li>Enviar un cuerpo vacío `\{}` restablece cualquier texto personalizado al texto predeterminado. </li></ul> |

    3. Como respuesta, se devuelve la entidad creada.

    **Ejemplo de solicitud**

    ```js lines
    PUT /api/v2/self-service-profiles/ssp_1234567890/custom-text/en/get-started

    {
      introduction: "Welcome! With <b>only a few steps</b>, you'll be able to setup your new connection. For assistance, contact <a href="https://www.examplesupportsite.com"> our support team </a>." 
    }
    ```

    **Ejemplo de respuesta**

    ```js lines
    {
      introduction: "Welcome! With <b>only a few steps</b>, you'll be able to setup your new connection. For assistance, contact <a href="https://www.examplesupportsite.com"> our support team </a>." 
    }
    ```

<div id="manage-self-service-access-tickets">
  ## Gestionar tickets de acceso de autoservicio
</div>

Después de crear al menos un perfil de autoservicio, puedes generar tickets de acceso de autoservicio desde el Auth0 Dashboard o la Management API.

Los tickets de acceso de autoservicio tienen dos propósitos principales:

* Dar a los administradores del cliente acceso al asistente de autoservicio, donde pueden configurar una nueva conexión SSO o modificar una conexión existente.
* Predefinir detalles y comportamientos clave de las nuevas conexiones SSO que configurarán los administradores del cliente, como qué aplicaciones u organizaciones se habilitarán para la nueva conexión.

Al generar tickets de acceso, también puedes habilitar ciertas funcionalidades, como el SSO iniciado por el IdP con <Tooltip tip="Security Assertion Markup Language (SAML): Protocolo estandarizado que permite a dos partes intercambiar información de autenticación sin una contraseña." cta="Ver glosario" href="/docs/es/glossary?term=SAML">SAML</Tooltip>, configurar dominios del proveedor de identidad (que impulsan Home Realm Discovery) y la verificación de dominio.

<div id="saml-idp-initiated-sso">
  #### SSO iniciado por el IdP de SAML
</div>

El SSO iniciado por el IdP de SAML es una modalidad de implementación que permite que los proveedores de identidad inicien el SSO y redirijan a los usuarios al proveedor de servicios para completar la autenticación.

Al habilitar esta opción para Self-Service Enterprise Configuration, debes proporcionar tu aplicación predeterminada y el protocolo de respuesta. También puedes proporcionar una query string opcional para personalizar aún más el comportamiento de la conexión.

Para obtener más información sobre estas opciones, consulta [Configurar el inicio de sesión único iniciado por el proveedor de identidad SAML](/docs/es/authenticate/protocols/saml/saml-sso-integrations/identity-provider-initiated-single-sign-on).

<div id="email-domain-verification-and-pre-verified-domains">
  #### Verificación del dominio de correo electrónico y dominios preverificados
</div>

Self-Service Enterprise Configuration admite tres formas de asociar dominios de correo electrónico con una conexión empresarial. Estos métodos rellenan el mismo campo: `options.domain_aliases `, que controla Home Realm Discovery (HRD) y, de forma condicional, los dominios de organización para descubrimiento:

1. Dominios preverificados (gestionados por el tenant): el administrador del tenant agrega directamente los dominios conocidos a la conexión.
2. Dominios por verificar: el administrador del tenant especifica los dominios que el administrador de TI debe verificar
   durante el Setup. Estos dominios figuran como pendientes en la organización y/o no se asocian automáticamente con la conexión hasta que el administrador de TI complete la verificación.
3. Verificación del dominio de correo electrónico (autogestionada): el administrador de tu cliente verifica el dominio durante el Setup en el asistente de self-service.

Estos mecanismos para asociar dominios de correo electrónico se aplican a los dominios de correo electrónico usados para el enrutamiento de HRD y SSO. No están relacionados con la verificación de Custom Domain a nivel del tenant.

Para que un dominio se complete tanto con dominios de organización para descubrimiento como con HRD, el ticket debe:

* Incluir solo un `enabled_organization`
* Tener dominios preverificados o Verificación de correo electrónico obligatoria como `Required` o `Optional`
* Habilitar la casilla **Allow the Use of Domains for Organization Discovery**

<div id="pre-verified-domain">
  ##### Dominio preverificado
</div>

Usa un dominio preverificado para asociar dominios de correo electrónico a una conexión empresarial a nivel del tenant. Los dominios añadidos se consideran de confianza y se escriben directamente en `options.domain_aliases`.

Puedes proporcionar dominios al [generar tickets de acceso de autoservicio](#generate-access-tickets-for-new-connections), ya sea a través del Auth0 Dashboard o de la Management API.

* **Auth0 Dashboard**: En la página **Generate Ticket**, especifica tu lista de dominios en el campo **Dominios preverificados**.
* **Management API**: Establece `connection_config.options.domain_aliases` con la lista de dominios. De forma predeterminada, `use_for_organization_discovery` se establece en `true`. De forma opcional, selecciona **Allow the Use of Domains for Organization Discovery**.

Los dominios añadidos por los administradores del tenant se consideran **de confianza** y **no** requieren que el administrador del cliente complete la verificación. Si quieres que los administradores de TI verifiquen un dominio en lugar de tratarlo como **de confianza**, establece `domain_aliases_config.domain_verification` en `required` u `optional` para solicitar la verificación desde el asistente de autoservicio. La opción **Allow Use of Domains for Organization Discovery** requiere exactamente un `enabled_organization` en el ticket.

<div id="domains-to-be-verified">
  ##### Dominios por verificar
</div>

Usa los dominios por verificar para especificar los dominios que los administradores de TI deben verificar durante la configuración. A diferencia de los dominios preverificados, los dominios pendientes no se asocian automáticamente con la conexión ni con la Organization. Los administradores de TI deben completar la verificación en el asistente de configuración antes de que surtan efecto.

Puedes proporcionar dominios al [generar tickets de acceso de autoservicio](#generate-access-tickets-for-new-connections) a través del Auth0 Dashboard o de la Management API:

* **Auth0 Dashboard**: En la página **Generate Ticket**, especifica la lista de dominios en el campo **Dominios por verificar**.
* **Management API**: Establece `connection_config.options.domain_aliases` con la lista de dominios. De forma predeterminada, `use_for_organization_discovery` está establecido en `true`. De manera opcional, selecciona **Allow the Use of Domains for Organization Discovery**

Se aplican los siguientes límites:

* Si hay una Organization asociada al ticket, el total combinado de dominios de Organization existentes y pendientes no debe superar los 100.
* Si no hay ninguna Organization o hay más de una Organization asociada al ticket, el total combinado de alias de dominio existentes y pendientes no debe superar los 1.000.

Si añadir los dominios pendientes hiciera que se superara cualquiera de estos límites, la creación del ticket fallará con un error.

<div id="email-domain-verification">
  ##### Verificación del dominio de correo electrónico
</div>

A diferencia de los dominios preverificados, la verificación del dominio de correo electrónico confirma que un administrador de TI es propietario del dominio que proporciona durante la Setup de autoservicio. Cuando se completa la verificación, el dominio verificado se agrega al mismo arreglo `options.domain_aliases` de la conexión.

Puede habilitar la verificación para administradores de TI cuando genere tickets de acceso de autoservicio:

* **Auth0 Dashboard**: En la página **Generate Ticket**, use el campo **Domain Verification Requirement**. De forma opcional, seleccione **Allow the Use of Domains for Organization Discovery**.
* **Management API**: Use `domain_aliases_config.domain_verification` y, de forma opcional, `use_for_organization_discovery ` con una de las siguientes opciones:
  * `none` (predeterminado): El asistente de autoservicio no muestra un prompt para que un administrador del cliente verifique su dominio.
  * `required`: El asistente de autoservicio muestra un prompt para que el administrador del cliente verifique sus dominios.
  * `optional`: El asistente de autoservicio muestra un prompt para que el administrador del cliente verifique su dominio. El administrador del cliente puede optar por introducir su dominio para verificarlo u omitir este paso.

La opción **Allow Use of Domains for Organization Discovery** requiere exactamente **una** Organization en el campo Enabled Organizations y que Domain Verification esté configurado como `Optional` o `Required`, o bien Pre-Verified Domain(s). En algunos casos, la verificación puede tardar hasta 48 horas, y es posible que deba emitir un ticket de acceso adicional para que el administrador del cliente vuelva y habilite la conexión. Los tickets de acceso vencen cinco horas después de abrirse por primera vez. Consulte [Generate access tickets for existing connections](https://auth0.com/docs/authenticate/enterprise-connections/self-service-enterprise-configuration/manage-self-service-enterprise-config#generate-access-tickets-for-existing-connections).

<div id="remove-a-domain">
  ##### Eliminar un dominio
</div>

Los administradores de TI pueden eliminar dominios verificados desde el asistente de autoservicio. Se aplican las siguientes reglas según el Domain Verification Requirement configurado en el ticket:

Optional: Se pueden eliminar todos los dominios verificados.
Required: Debe permanecer al menos un dominio verificado. Aparece una advertencia si el administrador de TI intenta eliminar el último dominio verificado.

<div id="generate-access-tickets-for-new-connections">
  ### Generar tickets de acceso para nuevas conexiones
</div>

Puede generar tickets de acceso para nuevas conexiones a través de Auth0 Dashboard o de la Management API.

<Warning>
  De forma predeterminada, las URL de los tickets de acceso siguen siendo válidas durante cinco días desde su generación. Después de acceder a la URL del ticket, el administrador del cliente tiene cinco horas para completar la configuración. Se puede acceder a una URL de ticket de acceso un máximo de 10 veces; una vez alcanzado este límite, se debe solicitar un nuevo ticket de acceso.

  Si es necesario, puede revocar un ticket de acceso antes de su expiración para interrumpir de inmediato el acceso al asistente de autoservicio.
</Warning>

<Tabs>
  <Tab title="Auth0 Dashboard">
    Para generar un ticket de acceso para una nueva conexión a través del Auth0 Dashboard:

    1. Vaya a [Authentication &gt; Enterprise](https://manage.auth0.com/#/connections/enterprise) y acceda a la sección **Self-Service Enterprise Configuration**. Luego, seleccione el self-service profile con el que desea crear un ticket de acceso.

    2. Seleccione **Generate Ticket** para abrir el formulario del ticket. En **Select ticket type**, elija **Create a new connection**.

    3. En **Ticket configuration**, proporcione el nombre obligatorio de la conexión que configurará el administrador del cliente.

    4. En la sección **Settings**, configure opciones adicionales según sea necesario para la nueva conexión:

       * **Domain**: Seleccione el dominio personalizado que se usará en la URL del ticket. Solo está disponible cuando hay múltiples dominios personalizados.
       * **Display Name**: Un nombre descriptivo para la conexión que se muestra en los prompts de Universal Login.
       * **Enabled Clients**: Una lista de IDs de cliente separados por comas para asociar con la conexión.
       * **Enabled Organizations**: Una lista de IDs de organización separados por comas para asociar con la conexión.
       * **Display connection a as button**: Muestra la conexión como una opción de autenticación en la pantalla de login.
       * **Display connection as a button for organizations**: Muestra la conexión como una opción de autenticación en la pantalla de login para las organizaciones especificadas.
       * **Assign membership on login for organizations**: Asigna automáticamente membresía en la organización a los usuarios que se autentican con la conexión.
       * **Enable as a domain level connection**: Permite que aplicaciones de terceros usen la conexión; requiere [registro dinámico de clientes](/docs/es/get-started/applications/dynamic-client-registration).
       * **Accept SAML IdP-initiated SSO**: Habilita el [SSO iniciado por el proveedor de identidad SAML](/docs/es/authenticate/protocols/saml/saml-sso-integrations/identity-provider-initiated-single-sign-on).

    5. En **Domain-Based Discovery**, opcionalmente proporcione una lista separada por comas de dominios de IdP ya verificados o por verificar para compararlos con los dominios de correo electrónico de los usuarios. Estos dominios se almacenan en `options.domain_aliases` y controlan HRD. Para obtener más información, consulte [Home Realm Discovery](#home-realm-discovery).

    6. En **Domain Verification Requirement**, elija el nivel de verificación deseado:
       * **Off**: No se solicita a los administradores del cliente que verifiquen su dominio al configurar SSO. **Off** es la configuración predeterminada para los nuevos tickets de acceso.
       * **Optional**: Se solicita a los administradores del cliente que verifiquen su dominio al configurar SSO. Sin embargo, pueden omitir este paso y habilitar su conexión sin completar la verificación.
       * **Required**: Los administradores del cliente deben verificar su dominio al configurar SSO. No podrán habilitar su conexión hasta que se complete la verificación.

    7. En **Provisioning**, habilite opcionalmente **Sync users and group profiles using provisioning**. Cuando esta opción está habilitada, hay configuración adicional disponible:
       * **Bearer Token Expiration**: Defina una fecha de expiración para el bearer token de SCIM. De manera predeterminada, los bearer tokens no expiran.
       * **Bearer Token Permissions (Scopes)**: Elija qué acciones puede realizar el token. De manera predeterminada, todos los scopes de aprovisionamiento están habilitados:
         * `get:users`
         * `post:users`
         * `put:users`
         * `patch:users`
         * `delete:users`
         * `get:groups`
         * `post:groups`
         * `put:groups`
         * `patch:groups`
         * `delete:groups`

    8. En **Time to Live**, establezca un período de expiración para el ticket de acceso en segundos. De manera predeterminada, el tiempo de vida está establecido en 432000 segundos (lo que equivale a cinco días).

       * Time to Live determina durante cuánto tiempo una URL de ticket de acceso permanece activa **antes de que** un administrador del cliente inicie el self-service assistant. No determina durante cuánto tiempo el administrador del cliente tendrá acceso al assistant después de iniciarlo. La expiración del self-service assistant en sí es de 5 horas y no se puede configurar.

    9. En **Metadata**, agregue hasta 10 metadatos asociados con la conexión.

    10. Revise la configuración de su ticket de acceso para verificar que sea correcta. Luego, seleccione **Create Ticket**.

    A continuación, se muestra una ventana emergente de Ticket Information que contiene la URL del ticket de acceso. Copie y guarde esta URL en un lugar seguro, ya que no podrá volver a recuperarla después de cerrar la ventana emergente.

    Puede compartir la URL del ticket de acceso con el administrador del cliente por correo electrónico, chat u otro canal de comunicación para otorgarle acceso al self-service assistant. Luego, el assistant lo guiará durante la configuración de la conexión SSO. Para obtener más información sobre esa experiencia, consulte [Self-service assistant experience](/docs/es/authenticate/enterprise-connections/self-service-enterprise-configuration#self-service-assistant-experience).
  </Tab>

<div id="generate-access-tickets-for-existing-connections">
  ### Generar tickets de acceso para conexiones existentes
</div>

Puede generar tickets de acceso para conexiones existentes a través del Auth0 Dashboard o de la Management API.

<Warning>
  De forma predeterminada, las URL de los tickets de acceso siguen siendo válidas durante cinco días después de generarse. Después de acceder a la URL del ticket, el administrador del cliente tiene cinco horas para completar la configuración. Se puede acceder a la URL de un ticket un máximo de 10 veces; una vez alcanzado este límite, debe solicitarse un nuevo ticket de acceso.

  Si es necesario, puede revocar un ticket de acceso antes de su expiración para interrumpir de inmediato el acceso al asistente de autoservicio.
</Warning>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Si un administrador del cliente inicia la **verificación de dominio** a través del asistente de autoservicio, es posible que necesite un ticket de acceso adicional para completar el proceso de configuración.

  La verificación de dominio ocurre como paso final del asistente de autoservicio. En este punto del flujo de trabajo, la conexión ya se ha creado, pero no se ha habilitado. Si se requiere la verificación de dominio, el administrador del cliente no podrá habilitar su conexión hasta que se complete la verificación.

  Aunque la verificación normalmente se realiza en poco tiempo, en algunos casos puede tardar entre 24 y 48 horas. Si esto ocurre, el administrador del cliente no podrá usar su ticket de acceso original para habilitar su conexión, ya que los tickets expiran cinco horas después del primer acceso.

  Para completar este proceso, puede generar un ticket de acceso que permita al administrador del cliente modificar la conexión que configuró con su ticket inicial. Al crear este ticket, asegúrese de especificar el ID de la conexión que configuró con su primer ticket de acceso.
</Callout>

Si no se especifica o se establece en `0`, el valor predeterminado es `432000` (equivalente a 5 días).<br /><br />Ten en cuenta que este período de expiración no determina durante cuánto tiempo un administrador del cliente tiene acceso al self-service assistant después de iniciarlo. La expiración del assistant en sí es de cinco horas y no se puede configurar.                                                                                                                                                                                                                                                                                                                                                         |
    | `domain_aliases_config`                 | **Opcional**. Objeto.<br /><br />Contiene domain&#95;verification, que se usa para determinar si la verificación de dominio es obligatoria, opcional o está desactivada.<br /><br />Las opciones de domain&#95;verification incluyen:<br /><br /><ul><li>`none`: Desactiva la verificación de dominio. También puede desactivar la verificación de dominio si omite el objeto `domain_aliases_config` de su solicitud.</li><li>`optional`: Permite que los administradores del cliente omitan la verificación de dominio durante Setup.</li><li>`required`: Requiere que los administradores del cliente verifiquen su dominio durante Setup.</li></ul>                                                                                                                                                                                                                                                         |
    | `use_for_organization_discovery`        | **Opcional**. Booleano. <br /><br />Indica si debe usarse un dominio verificado para el descubrimiento de la organización durante la autenticación.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                             |

    **Ejemplo de cuerpo de solicitud**

    ```json lines
    {
      "connection_id": "string",
       "ttl_sec":0,
       "domain_aliases_config": {
           "domain_verification": "string"
           "pending_domains": [
                 "acme.com"
              ]
        }
    }
    ```

    En respuesta, recibes una URL del ticket de acceso de autoservicio:

    ```json lines
    {
      "ticket": "https://{domain}/self-service/connections-flow?ticket={id}"
    }
    ```

    Una vez que reciba la ticket URL, comparta el enlace con el administrador de su cliente para otorgarle acceso al asistente de autoservicio. El asistente lo guiará por el proceso de configuración de la conexión SSO. Para obtener más información sobre esta experiencia, consulte [Self-service assistant experience](/docs/es/authenticate/enterprise-connections/self-service-enterprise-configuration#self-service-assistant-experience).

    Puede integrar la generación de access tickets en su propio portal de autoservicio o enviar las ticket URLs directamente a los administradores de clientes por correo electrónico, chat u otros canales de comunicación.
  </Tab>
</Tabs>

<div id="revoke-an-access-ticket">
  ### Revocar un ticket de acceso
</div>

De forma predeterminada, una URL de ticket de acceso sigue siendo válida durante cinco días. Al acceder a la URL, un administrador del cliente dispone de cinco horas para completar la configuración.

Si es necesario, puedes revocar un ticket de acceso antes de su expiración. Por ejemplo, si un ticket de acceso se comparte con la <Tooltip tip="Audience: Identificador único de la audiencia de un token emitido. Denominado aud en un token, su valor contiene el ID de una aplicación (Client ID) para un token de ID o de una API (API Identifier) para un token de acceso." cta="Ver glosario" href="/docs/es/glossary?term=audience">audiencia</Tooltip> incorrecta, puedes revocar el ticket para evitar el acceso no autorizado al asistente de autoservicio.

Cuando revocas un ticket de acceso, su URL se invalida de inmediato y todas las sesiones asociadas se cierran. Los administradores del cliente que tengan la URL ya no podrán acceder al asistente de autoservicio. Después, puedes generar y compartir nuevos tickets de acceso según sea necesario.

Para revocar un ticket de acceso:

1. Recupera el ID del perfil de autoservicio asociado al ticket de acceso mediante el endpoint [Retrieve Self-Service Profiles](https://auth0.com/docs/api/management/v2/self-service-profiles/get-self-service-profiles).
2. Localiza el ID del ticket de acceso que quieras revocar. Los ID se encuentran al final de la URL del ticket de acceso.
3. Llama al endpoint [Revoke SSO Access Ticket](https://auth0.com/docs/api/management/v2/self-service-profiles/post-revoke) con los ID correspondientes:

`POST  /api/v2/self-service-profiles/{id}/sso-ticket/{id}/revoke`

Como respuesta, se devuelve un `202 Accepted`.
