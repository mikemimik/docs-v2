<ReleaseStageNotice feature="perfil de atributos de usuario con Configuración empresarial de autoservicio" stage="ea" plans="B2B Professional and B2B Enterprise" terms="true" />

perfil de atributos de usuario (UAP) proporciona una forma coherente de definir, gestionar y mapear atributos de usuario en distintos protocolos, como [SCIM](/docs/es/authenticate/protocols/scim), [SAML](/docs/es/authenticate/protocols/saml) y [OIDC](/docs/es/authenticate/login/oidc-conformant-authentication). UAP con [Configuración empresarial de autoservicio](/docs/es/authenticate/enterprise-connections/self-service-enterprise-configuration) ofrece a los administradores un mayor control sobre los datos de identidad del usuario al definir atributos de usuario y aplicar el perfil en todos los protocolos de autenticación.

<div id="how-it-works">
  ## Cómo funciona
</div>

* **Definición de perfil**
  Un administrador crea un perfil de atributos de usuario para definir atributos, entre ellos:
  * Cómo mostrar los atributos
  * Cómo hacer que los atributos sean obligatorios
  * Cómo se mapean los atributos en Auth0 y en sistemas de identidad externos

* **Alcance flexible**
  Los perfiles están vinculados a los flujos de Configuración empresarial de autoservicio, pero están diseñados para el aprovisionamiento, la incorporación y la gestión de derechos.

* **Capa de mapeo unificada**
  Cada atributo admite mapeos entre protocolos de autenticación, con la opción de reemplazar valores para proveedores específicos o estrategias de conexión, como Okta y Entra ID.

<div id="attribute-mapping-and-override">
  ## Mapeo de atributos y sobrescritura
</div>

UAP admite definiciones de atributos para varios protocolos y sobrescrituras de estrategia para necesidades específicas de cada proveedor.

**Mapeo de atributos**

| **Protocolo**  | **Descripción**                                                                                                                                                                                                                          |
| -------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Mapeo de Auth0 | Atributo canónico almacenado en Auth0 (`email`, `name`, `app_metadata.department`).                                                                                                                                                      |
| Mapeo de OIDC  | `claims` estándar de OIDC (`sub`, `preferred_username`, `zoneinfo`). Para obtener más información sobre los `claims` estándar de OIDC, consulta [Standard Claims](https://openid.net/specs/openid-connect-core-1_0.html#StandardClaims). |
| Mapeo de SAML  | Admite uno o más URI de `assertion` (`http://schemas.xmlsoap.org/ws/2005/05/identity/claims/emailaddress`).                                                                                                                              |
| Mapeo de SCIM  | Atributos de aprovisionamiento (`name.familyName`, `addresses[type eq "work"].country`).                                                                                                                                                 |

**Sobrescrituras de estrategia**

Algunos proveedores usan mapeos no estándar. UAP permite sobrescrituras:

| **Protocolo**   | **Descripción**                                                                              |
| --------------- | -------------------------------------------------------------------------------------------- |
| SAML            | Asigna `userName` en lugar de `externalId`.                                                  |
| WAAD (Entra ID) | Usa `oid` como identificador de OIDC.                                                        |
| Okta            | Asigna atributos como `middleNam`e o `federated_groups` mediante claims específicos de Okta. |

<div id="user-id">
  ### ID de usuario
</div>

La propiedad user&#95;id define cómo mapear claims de OIDC, atributos de SAML o atributos de SCIM al ID de usuario de Auth0. Todos los usuarios de Auth0 deben tener un ID, por lo que este mapeo es obligatorio.

* En OIDC, las opciones son fijas (normalmente `sub`, `oid` para Azure AD o `email` para Google).
  * En SAML y SCIM, el mapeo es más flexible y puede apuntar a varios atributos posibles.

<div id="user-attributes">
  ### Atributos de usuario
</div>

La propiedad `user_attributes` contiene la información de mapeo que permite al sistema interpretar los claims entrantes del IdP y almacenarlos como atributos del perfil de usuario de Auth0.

Cada atributo debe proporcionarse como un par clave/valor:

* La clave corresponde al nombre del atributo.
  * El valor es un objeto con:
    * `label`
    * `description`
    * `profile_required`
    * `auth0_mapping`
    * `saml_mapping`
    * `scim_mapping`
    * `oidc_mapping`, un objeto con propiedades
      * `mapping` representa el claim entrante del IdP (valor literal, objeto `context` dinámico o ambos mediante la sintaxis `${variable}` compatible con el [objeto `context`](/docs/es/authenticate/identity-providers/enterprise-identity-providers/configure-pkce-claim-mapping-for-oidc#context-object))
      * `display_name`, la etiqueta que se muestra a los usuarios finales en flujos de autoservicio

<div id="strategy-overrides">
  ### Sobrescrituras de estrategia
</div>

La propiedad `strategy_overrides` te permite especificar excepciones para proveedores de identidad (IdPs) concretos, ya que no todos los IdPs exponen los mismos identificadores o claims.

Cada sobrescritura define mapeos específicos del protocolo que reemplazan los valores predeterminados definidos en `user_id` o `user_attributes`.

<div id="examples">
  ## Ejemplos
</div>

<div id="user-identifier">
  ### Identificador de usuario
</div>

```json
"user_id": {
  "oidc_mapping": "sub",
  "saml_mapping": [
    "http://schemas.xmlsoap.org/ws/2005/05/identity/claims/nameidentifier",
    "http://schemas.xmlsoap.org/ws/2005/05/identity/claims/upn",
    "http://schemas.xmlsoap.org/ws/2005/05/identity/claims/name"
  ],
  "scim_mapping": "externalId",
  "strategy_overrides": {
    "waad": {
      "oidc_mapping": "oid"
    },
    "samlp": {
      "scim_mapping": "userName"
    },
    "google-apps": {
      "oidc_mapping": "email"
    }
  }
},
```

* **Identificador predeterminado**: `externalId` mediante SCIM.
* **SAML**: Admite varios URI de identificador.
* **OIDC**: Usa `sub`.
* **Sobrescrituras**: SAML y WAAD personalizan los mapeos.

<div id="email-attribute">
  ### Atributo de correo electrónico
</div>

```json
"email": {
  "description": "Email",
  "label": "Email",
  "profile_required": true,
  "auth0_mapping": "email",
  "scim_mapping": "emails[primary eq true].value",
  "oidc_mapping": {
    "mapping": "${context.tokenset.email}",
    "display_name": "email"
  },
  "saml_mapping": [
    "http://schemas.xmlsoap.org/ws/2005/05/identity/claims/emailaddress"
  ],
  "strategy_overrides": {
    "waad": {
      "scim_mapping": "emails[type eq \"work\"].value"
    }
  }
}
```

* Recomendado para la mayoría de los perfiles.
* Unificado en Auth0, OIDC, SAML y SCIM.
* La excepción de WAAD garantiza el mapeo correcto a los correos electrónicos laborales.

<div id="create-a-user-attribute-profile">
  ## Crear un perfil de atributos de usuario
</div>

Puede definir un UAP a través de Configuración empresarial de autoservicio desde Auth0 Dashboard o mediante la Management API. Actualmente, se puede configurar desde la experiencia de Configuración empresarial de autoservicio.

<div id="configure-with-auth0-dashboard">
  ### Configurar con Auth0 Dashboard
</div>

1. Ve a [**Authentication &gt; empresarial &gt; Configuración empresarial de autoservicio**](http://manage.auth0.com/*/connections/enterprise/self-service-profiles).
2. Selecciona **+Create Profile**.
3. Indica un **Name** y una **Description** opcional para el nuevo perfil.
4. Añade una entrada de perfil de atributos de usuario seleccionando un perfil existente o eligiendo **+Create New**.
   * Para un perfil nuevo, indica un **User Profile Attribute Name**.
   * Revisa los mapeos para asegurarte de que los atributos del perfil se asignen a los atributos de Auth0 que prefieras.
5. Elige **Create**.

Tu nuevo UAP ya está disponible para configurarse para SSO.

<div id="configure-with-management-api">
  ### Configurar con Management API
</div>

Para gestionar los perfiles de atributos de usuario, están disponibles los siguientes endpoints de la [Management API](https://auth0.com/docs/api/management/v2/introduction):

* `POST` `/api/v2/user-attribute-profiles`
* `GET` `/api/v2/user-attribute-profiles`
* `PATCH` `/api/v2/user-attribute-profiles/{id}`
* `GET` `/api/v2/user-attribute-profiles/{id}`
* `GET` `/api/v2/user-attribute-profiles/templates`
* `GET` `/api/v2/user-attribute-profiles/templates/{id}`
