---
description: Aprende a proteger las conexiones empresariales de OIDC y Okta con vinculación criptográfica de tokens.
title: Configurar conexiones empresariales con Demonstrating Proof-of-Possession (DPoP)
---

Demonstrating Proof-of-Possession (DPoP) es una [extensión del framework OAuth 2.0](https://datatracker.ietf.org/doc/draft-ietf-oauth-dpop/). Para obtener más información sobre el protocolo DPoP y cómo se usa para aplicar Sender Constraining a los tokens, consulta [Demonstrating Proof-of-Possession (DPoP)](/docs/es/secure/sender-constraining). Si usas Okta u OpenID Connect (OIDC) como proveedor de identidad (IdP) y los has configurado en Auth0 como una conexión empresarial, puedes habilitar y configurar DPoP para vincular los tokens de acceso de tu IdP a claves criptográficas. El uso de DPoP evita los ataques de repetición de tokens y ayuda a cumplir requisitos normativos, como [Interoperability Profiling for Secure Identity in the Enterprise (IPSIE) OIDC](https://openid.net/specs/ipsie-openid-connect-sl1-profile-1_0.html) Security Level 1 o [Financial-grade API (FAPI) 2.0](https://openid.net/specs/fapi-security-profile-2_0-final.html).

<Card title="Antes de empezar">
  Antes de habilitar DPoP en Auth0:

  * Tu proveedor de identidad ascendente debe admitir DPoP de acuerdo con la especificación [RFC-9449](https://www.rfc-editor.org/rfc/rfc9449.html).
  * Debes tener una conexión empresarial existente de OIDC u Okta, o poder crear una. Para obtener información sobre cómo crear una conexión empresarial en Auth0, consulta [Conexiones empresariales](/docs/es/authenticate/enterprise-connections).
  * La conexión no debe estar configurada para usar [Token Vault](/docs/es/secure/call-apis-on-users-behalf/token-vault/configure-token-vault).
  * La conexión debe usar Proof Key for Code Exchange (PKCE) [flujo de código de autorización + PKCE](docs/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce#authorization-code-flow-with-proof-key-for-code-exchange-pkce), que se habilita en el proveedor ascendente si tu proveedor de identidad admite PKCE.
  * La conexión debe ser de tipo `back_channel`.
</Card>

<div id="verify-upstream-idp-support">
  ## Verifique la compatibilidad del IdP ascendente
</div>

Revise el documento de descubrimiento de OIDC de su IdP para comprobar si admite DPoP:

```curl
curl https://YOUR_IDP_DOMAIN/.well-known/openid-configuration
```

En la respuesta, busca el valor de compatibilidad con DPoP `dpop_signing_alg_values_supported`.

**Ejemplo**

```
{
  "dpop_signing_alg_values_supported": ["ES256", "ES384", "Ed25519", "ES512", "RS256"] 
},
```

<div id="choose-a-signing-algorithm">
  ## Elige un algoritmo de firma
</div>

Antes de configurar DPoP, elige uno de los [algoritmos de firma](/docs/es/get-started/applications/signing-algorithms) compatibles:

| **Algoritmo** | **Descripción**                 | **Cuándo usarlo**                                                                 |
| ------------- | ------------------------------- | --------------------------------------------------------------------------------- |
| ES256         | ECDSA con curva P-256 y SHA-256 | Tu proveedor de identidad admite ES256.                                           |
| ES384         | ECDSA con curva P-384 y SHA-384 | Tu proveedor de identidad requiere ES384.                                         |
| ES512         | ECDSA con curva P-521 y SHA-512 | Tu proveedor de identidad requiere ES512.                                         |
| Ed25519       | EdDSA con Curve25519            | Tu proveedor de identidad requiere Ed25519 por motivos de cumplimiento normativo. |

Elige ES256, a menos que tu proveedor de identidad requiera específicamente otro algoritmo.

<div id="enable-dpop">
  ## Habilitar DPoP
</div>

Para habilitar DPoP y elegir un algoritmo, usa Auth0 Dashboard o la Management API.

<Tabs>
  <Tab title="Auth0 Dashboard">
    En Auth0 Dashboard:

    1. Ve a **[Authentication &gt; empresarial](https://manage.auth0.com/dashboard/#/connections/enterprise/)**. Selecciona la conexión que quieras configurar.
    2. Selecciona la pestaña **Credenciales**.
    3. Marca la casilla **Habilitar Demonstrating Proof of Possession (DPoP)**.
    4. En el menú Signing Algorithms para DPoP, elige tu algoritmo.
       <Frame>![Habilitar DPoP y seleccionar un algoritmo](/docs/images/cdy7uua7fh8z/enable-dpop-dashboard.png)</Frame>
    5. Selecciona **Guardar**.
  </Tab>

  <Tab title="Management API">
    Para usar la Management API, debes obtener un [token de acceso de la Management API](/docs/es/secure/tokens/access-tokens/management-api-access-tokens).

    Haz una solicitud `PATCH` al endpoint [actualizar una conexión](https://auth0.com/docs/api/management/v2/connections/patch-connections-by-id) con `dpop_signing_alg_values_supported` en el objeto `options`:

    ```curl
    PATCH https://YOUR_DOMAIN/api/v2/connections/YOUR_CONNECTION_ID
    Content-Type: application/json
    Authorization: Bearer YOUR_MANAGEMENT_API_TOKEN

    {
      "options": {
        "dpop_signing_alg": "ES256"
      }
    }
    ```

    <Callout icon="file-lines" color="#0EA5E9" iconType="regular">
      Si haces `PATCH` del parámetro `options`, se sobrescribirá todo el objeto `options`. Para evitar datos incompletos u otros problemas, asegúrate de incluir todas las propiedad deseadas al hacer `PATCH` de `options`.
    </Callout>

    Reemplaza los valores de marcador de posición:

    * **YOUR&#95;DOMAIN**: El dominio de tu tenant de Auth0. Ejemplo: `travel0.us.auth0.com`.
    * **YOUR&#95;CONNECTION&#95;ID**: El ID de tu conexión empresarial de OIDC u Okta.
    * **YOUR&#95;MANAGEMENT&#95;API&#95;TOKEN**: Un token de la Management API con el scope `update:connections`
  </Tab>
</Tabs>

<div id="test-dpop">
  ## Probar DPoP
</div>

Después de habilitar DPoP, pruebe la configuración iniciando un flujo de inicio de sesión:

1. Vaya a su aplicación.
2. Inicie el flujo de inicio de sesión con la conexión empresarial que configuró.
3. Complete el inicio de sesión con su proveedor de identidad ascendente.
4. [Revise los registros de Auth0](/docs/es/deploy-monitor/logs#logs) en [**Auth0 Dashboard &gt; monitoreo &gt; registros**](https://manage.auth0.com/#/logs) para confirmarlo.

Una transacción exitosa en una entrada de registro debería ser similar a:

```
{
  "type": "s",
  "description": "Success Login",
  "details": {
    "dpop_signing_alg": "ES256",
    "idp_token_type": "dpop",
    "upstream_userinfo_fetch": {
      "status": "SUCCESS",
      "dpop_bound": true
    }
   }
}
```

Los valores `dpop_signing_alg` e `idp_token_type: "dpop"` confirman que Auth0 envió una prueba DPoP con el algoritmo configurado y que tu IdP emitió tokens vinculados a DPoP. El objeto `upstream_userinfo_fetch` solo está presente si se llama al endpoint de [Información del usuario](https://auth0.com/docs/api/authentication/user-profile/get-user-info). El campo `dpop_bound` solo está presente si la solicitud `GET` al endpoint `/userinfo` se vincula correctamente a DPoP.

<div id="disable-dpop">
  ## Deshabilitar DPoP
</div>

Puede deshabilitar DPoP desde el Auth0 Dashboard o mediante la Management API.

<Tabs>
  <Tab title="Auth0 Dashboard">
    1. Vaya a [**Authentication &gt; empresarial**](https://manage.auth0.com/dashboard/#/connections/enterprise/). Seleccione la conexión que desea configurar.
    2. Seleccione la pestaña **credencial**.
    3. Desmarque la casilla **Enable Demonstrating Proof of Possession (DPoP)**.
    4. Seleccione **Save**.
  </Tab>

  <Tab title="Management API">
    Para deshabilitar DPoP, elimine la propiedad `dpop_signing_alg` de la configuración de su conexión:

    ```curl
    PATCH https://YOUR_DOMAIN/api/v2/connections/YOUR_CONNECTION_ID
    Content-Type: application/json
    Authorization: Bearer YOUR_MANAGEMENT_API_TOKEN

    {
      "options": {

      }
    }
    ```

    <Callout icon="file-lines" color="#0EA5E9" iconType="regular">
      Si aplica `PATCH` al parámetro `options`, se sobrescribirá todo el objeto `options`. Para evitar datos parciales u otros problemas, asegúrese de que todas las propiedades deseadas estén presentes al aplicar `PATCH` a `options`.
    </Callout>
  </Tab>
</Tabs>

<div id="troubleshoot">
  ## Solución de problemas
</div>

Usa las siguientes recomendaciones para diagnosticar y resolver problemas con la configuración de DPoP para OIDC y las conexiones empresariales de Okta.

<div id="check-auth0-configuration">
  ### Comprueba la configuración de Auth0
</div>

Antes de empezar a solucionar problemas, verifica la configuración de DPoP en Auth0:

1. Ve a [**Auth0 Dashboard &gt; Authentication &gt; Empresarial**](https://manage.auth0.com/#/connections/enterprise).
2. Selecciona tu conexión de Okta u OIDC.
3. Verifica que la conexión **no** esté configurada con Token Vault; para ello, ve a **Configuración avanzada &gt; Tipos de concesión**. Asegúrate de que Token Vault no esté seleccionado.
4. Usa el endpoint [actualizar una conexión](https://auth0.com/docs/api/management/v2/connections/patch-connections-by-id) de Management API para comprobar la configuración `dpop_signing_alg`:

```curl
GET https://YOUR_DOMAIN/api/v2/connections/YOUR_CONNECTION_ID
Authorization: Bearer YOUR_MANAGEMENT_API_TOKEN
```

En la respuesta de `dpop_signing_alg`, compruebe lo siguiente:

* Verifique que el algoritmo sea uno de los valores admitidos: **ES256**, **ES384**, **ES512** o **Ed25519**.
* Verifique que la conexión use el intercambio de tokens por back-channel mediante [flujo de código de autorización + PKCE](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce). DPoP no es compatible con las comunicaciones por front-channel, como en [flujo implícito](/docs/es/get-started/authentication-and-authorization-flow/implicit-flow-with-form-post), y se desactiva de forma silenciosa cuando la conexión usa el front-channel.

<div id="dpop-fields-missing-in-tenant-logs">
  #### Faltan campos de DPoP en los registros del tenant
</div>

Si los logs de éxito (`s`) o de error (`f`) de Auth0 para la conexión empresarial no contienen los campos `dpop_signing_alg` o `idp_token_type`, podría deberse a alguna de las siguientes causas:

* DPoP no está configurado. Verifique que `dpop_signing_alg` esté establecido en el objeto `options` de la conexión mediante el endpoint [actualizar una conexión](https://auth0.com/docs/api/management/v2/connections/patch-connections-by-id) de la Management API, como se describió anteriormente.
* Algoritmo no compatible. Auth0 admite ES256, ES384, ES512 y Ed25519. Si `dpop_signing_alg` está establecido en un valor no compatible (por ejemplo, RS256), DPoP se desactiva silenciosamente. No se registra ningún error. Actualice la conexión para usar ES256, ES384, ES512 o Ed25519.
* Conexión front-channel. DPoP requiere un tipo de conexión de intercambio de tokens `back_channel`. Es posible que deba [actualizar el tipo de concesión](/docs/es/get-started/applications/update-grant-types#update-grant-types) a un flujo back-channel, como Authorization Code Flow o flujo de código de autorización + PKCE.

<div id="authentication-fails-after-enabling-dpop">
  ### La autenticación falla después de habilitar DPoP
</div>

Revise las siguientes técnicas de solución de problemas si sus usuarios no pueden completar la autenticación después de habilitar DPoP en su conexión empresarial de Okta u OIDC.

Los logs del tenant deberían mostrar un evento de fallo (`f`) con `dpop_signing_alg` similar a:

```bash
{
  "type": "f",
  "description": "Failed Login",
  "details": {
    "error": "dpop_signing_alg"
  }
}
```

Ten en cuenta que el error no incluye `idp_token_type` porque Auth0 no recibió un token de tu IdP.

<div id="identity-provider-rejects-dpop-proof">
  #### El proveedor de identidad rechaza la prueba DPoP
</div>

Es posible que el IdP rechace activamente la prueba DPoP que Auth0 envía durante el intercambio de tokens. El IdP puede devolver un error `invalid_dpop_proof`, lo que provoca que falle la autenticación.

Comprueba que el IdP sea compatible con DPoP y que el algoritmo configurado (ES256, ES384, ES512 o Ed25519) esté en la lista de algoritmos compatibles. Puedes comprobarlo accediendo al documento de descubrimiento de OpenID Connect del IdP:

```curl
curl https://YOUR_IDP_DOMAIN/.well-known/openid-configuration
```

En la respuesta del IdP, busque `dpop_signing_alg_values_supported`. Si falta este campo, es posible que el IdP no sea compatible con DPoP. Si el campo enumera solo algoritmos que Auth0 no admite (por ejemplo, solo RS256), no se podrá usar DPoP con esta conexión. Debe deshabilitar DPoP para esta conexión o ponerse en contacto con su IdP para habilitar la compatibilidad con ES256, ES384, ES512 o Ed25519.

<div id="token-exchange-fails-for-a-non-dpop-reason">
  #### El intercambio de tokens falla por un motivo ajeno a DPoP
</div>

Si encuentras un registro de error con `dpop_signing_alg` presente, eso no significa necesariamente que DPoP haya causado el fallo. Auth0 adjunta metadatos de DPoP a todos los registros de error cuando DPoP está configurado, incluso si la causa raíz no está relacionada. Por ejemplo, la autenticación podría fallar debido a un código de autorización vencido o a credenciales de cliente no válidas.

Examina la descripción del error en el registro para determinar la causa real. Entre los errores comunes no relacionados con DPoP se incluyen: `invalid_grant`, `invalid_client` y errores de verificación de la firma del token de identidad.

<div id="dpop-key-generation-fails">
  #### La generación de claves DPoP falla
</div>

Auth0 genera un par de claves efímero para cada prueba DPoP. Si la generación de claves falla, la autenticación falla antes de que se envíe la solicitud de token. Se trata de un problema transitorio del lado del servidor. Pide al usuario que vuelva a intentar la autenticación.

<div id="idp-token-binding">
  #### Vinculación de tokens del IdP
</div>

Si la autenticación de usuario se realiza correctamente, pero los registros del tenant de Auth0 muestran `"idp_token_type": "bearer"`, es posible que tu IdP no vincule tokens con DPoP incluso cuando Auth0 envía pruebas de DPoP. Según la RFC 9449, este comportamiento es conforme. El IdP tiene control total sobre si emite tokens vinculados a DPoP y puede no vincular los tokens por alguno de estos motivos:

* La política del IdP no requiere DPoP para el recurso o la aplicación solicitados.
* El IdP no admite DPoP, aunque acepte la prueba sin errores.
* El IdP encontró un problema interno al procesar la prueba de DPoP.

Auth0 registra esto como un evento de &quot;downgrade&quot;. La autenticación se completa correctamente con tokens Bearer estándar.

Si necesitas tokens vinculados a DPoP por motivos de cumplimiento normativo, te recomendamos que te pongas en contacto con tu IdP para entender por qué no se están vinculando los tokens. Auth0 no puede forzar la vinculación DPoP en la respuesta de tokens del proveedor de identidad.

<div id="dpop-nonce-handling">
  #### Gestión del nonce de DPoP
</div>

Algunos IdP requieren un `nonce` en la [prueba DPoP según §8](https://www.rfc-editor.org/rfc/rfc9449.html#name-authorization-server-provid). Cuando tu IdP devuelve un HTTP `400` con un encabezado de respuesta `DPoP-Nonce`, Auth0 reintenta automáticamente la solicitud de token con el nonce proporcionado. Esto se gestiona de forma transparente y no aparece como un fallo en los registros del tenant.

El código de error `use_dpop_nonce` es una señal interna del protocolo entre Auth0 y el IdP. No indica ningún problema. No deberías ver `use_dpop_nonce` como motivo de una entrada de registro de error. Si el reintento con el nonce también falla (por ejemplo, si el proveedor de identidad devuelve `invalid_dpop_proof` en el segundo intento), el error final es el que aparece en el registro de fallos.

Si observas fallos de autenticación repetidos en una conexión en la que el proveedor de identidad requiere nonces, comprueba la conectividad de red entre Auth0 y el proveedor de identidad. El intercambio de nonce requiere dos idas y vueltas al endpoint de token, lo que aumenta la sensibilidad a los tiempos de espera de red.
