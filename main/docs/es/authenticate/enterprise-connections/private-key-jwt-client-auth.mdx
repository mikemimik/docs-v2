La autenticación de cliente con clave privada <Tooltip tip="JSON Web Token (JWT): formato estándar del token de ID (y a menudo del token de acceso) que se utiliza para representar claims de forma segura entre dos partes." cta="Ver glosario" href="/docs/es/glossary?term=JWT">JWT</Tooltip> es un método alternativo de autenticación del cliente para las conexiones empresariales de <Tooltip tip="JSON Web Token (JWT): formato estándar del token de ID (y a menudo del token de acceso) que se utiliza para representar claims de forma segura entre dos partes." cta="Ver glosario" href="/docs/es/glossary?term=OpenID">OpenID</Tooltip> Connect (OIDC) y Okta Workforce. Aunque la autenticación del cliente suele gestionarse mediante el envío de un <Tooltip tip="Client Secret: Secret usado por un cliente (aplicación) para autenticarse con el servidor de autorización; solo deberían conocerlo el cliente y el servidor de autorización, y debe ser lo suficientemente aleatorio como para que no pueda adivinarse." cta="Ver glosario" href="/docs/es/glossary?term=client+secret">secreto de cliente</Tooltip> compartido, la autenticación de cliente con clave privada JWT utiliza en su lugar un JWT firmado para mejorar la seguridad de la aplicación.

Al usar esta funcionalidad, puedes evitar algunas deficiencias de seguridad comunes que suelen presentarse con la autenticación estándar mediante secreto de cliente, como:

* Un mayor riesgo de interceptación y reutilización, ya que los secretos de cliente deben transmitirse entre las partes en cada solicitud.
* Mecanismos limitados para aplicar vencimientos y evitar la reutilización por parte de actores maliciosos.
* Un mayor riesgo de filtraciones o exposición, ya que ambas partes poseen el secreto de cliente.

Puedes configurar la autenticación de cliente con clave privada JWT para tus conexiones empresariales de OIDC y Okta Workforce a través del <Tooltip tip="Auth0 Dashboard: el principal producto de Auth0 para configurar tus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip> o de la <Tooltip tip="Auth0 Dashboard: el principal producto de Auth0 para configurar tus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip>.

<div id="how-it-works">
  ## Cómo funciona
</div>

El flujo de conexión OIDC usa endpoints autenticados como `/oauth/token` o `/oauth/par` para verificar la identidad de un cliente ante el <Tooltip tip="Servidor de autorización: Servidor centralizado que ayuda a definir los límites del acceso de un usuario. Por ejemplo, su servidor de autorización puede controlar los datos, las tareas y las funcionalidades disponibles para un usuario." cta="Ver glosario" href="/docs/es/glossary?term=authorization+server">servidor de autorización</Tooltip> o proveedor de OpenID. Con autenticación de cliente con clave privada JWT, se envía una aserción JWT de cliente firmada al proveedor de OpenID en lugar de un secreto de cliente.

La aserción JWT de cliente contiene los siguientes claims:

* Un `aud` (<Tooltip tip="Audience: Identificador único de la audience de un token emitido. Se denomina aud en un token y su valor contiene el ID de una aplicación (ID de cliente) para un token de ID o de una API (API Identifier) para un access token." cta="Ver glosario" href="/docs/es/glossary?term=audience">audience</Tooltip>) que identifica el <Tooltip tip="Audience: Identificador único de la audience de un token emitido. Se denomina aud en un token y su valor contiene el ID de una aplicación (ID de cliente) para un token de ID o de una API (API Identifier) para un access token." cta="Ver glosario" href="/docs/es/glossary?term=token+endpoint">endpoint de token</Tooltip> del proveedor de OpenID.
* Un `jti` (ID de JWT) para permitir un solo uso o proteger contra ataques de repetición.
* Un `exp` (tiempo de expiración) que limita la ventana de validez del token.
* Un `sub` y `iss` que identifican el <Tooltip tip="ID de cliente: Valor de identificación asignado a su recurso registrado por Auth0." cta="Ver glosario" href="/docs/es/glossary?term=client+ID">ID de cliente</Tooltip>.

La autenticación de cliente con clave privada JWT ofrece un método de autenticación más seguro al eliminar el uso de secretos de cliente compartidos. En su lugar, el JWT se firma con la clave privada del cliente, y el proveedor de OpenID solo tiene acceso a la clave pública.

<div id="private-key-jwt-client-authentication-flow">
  ### Flujo de autenticación de cliente con clave privada JWT
</div>

Después de que un usuario completa la autenticación con un <Tooltip tip="Proveedor de identidad (IdP): servicio que almacena y administra identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=identity+provider">proveedor de identidad</Tooltip> (IdP) externo, se le redirige a Auth0 con un código de autorización que se intercambia por tokens en el endpoint de token del proveedor de OpenID. Cuando la clave privada JWT está habilitada en una conexión, la llamada al endpoint de token del proveedor de OpenID usa una aserción de cliente en lugar de un secreto de cliente para una autenticación más segura.

Los siguientes pasos muestran un flujo típico de autenticación de cliente con clave privada JWT.

<Frame>![Un diagrama que muestra el flujo de autenticación de cliente con clave privada JWT. ](/docs/images/cdy7uua7fh8z/4JOxbdG7aweqpDHCJ8VOZq/aea18936c0cbdbfbb39d0cd79af3987e/Client_Assertion_JWT_-_Diagram.png)</Frame>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Para completar este flujo, primero debes configurar una conexión nueva o existente de OIDC u Okta Workforce con `token_endpoint_auth_method=private_key_jwt` a través de Auth0 Dashboard o de la Management API. Para obtener más información, consulta la sección [Configurar la autenticación de cliente con clave privada JWT](#configure-private-key-jwt-client-authentication).
</Callout>

1. Después de configurar tu conexión, Auth0 genera y almacena automáticamente dos pares de claves públicas y privadas.

   * Un par de claves corresponde al conjunto activo `current`, mientras que el otro está etiquetado como `next` para admitir la [rotación de claves](#rotate-signing-keys).
2. Según tu IdP, a continuación debes hacer una de estas dos cosas:

   * Descarga la clave pública `current` y carga el archivo en el servidor de autorización, o:
   * Copia y pega el [`jwks_uri`](#retrieve-signing-keys) en el servidor de autorización.
3. Un usuario realiza una acción que requiere autenticación, como iniciar sesión en tu aplicación.
4. Auth0 envía una solicitud al servidor de autorización para iniciar la autenticación.
5. El servidor de autorización muestra al usuario las pantallas de autenticación y consentimiento.
6. El usuario se autentica y otorga su consentimiento al servidor de autorización.
7. El servidor de autorización envía un código de autorización a Auth0.
8. Auth0 genera una aserción JWT de cliente y la firma con la clave privada `current`.
9. Auth0 envía la aserción JWT de cliente al servidor de autorización.
10. El servidor de autorización identifica al cliente según el `client_id` proporcionado.
11. El servidor de autorización obtiene las claves públicas de Auth0 si se proporcionó un `jwks_uri`; de lo contrario, identifica la clave pública registrada en el paso 2.
12. Si se solicitó el `jwks_uri`, Auth0 devuelve las claves públicas como JWKS.
13. El servidor de autorización valida el JWT verificando la firma con la clave pública `current`, identificada por `kid` en el encabezado del JWT `client_assertion`.
14. El servidor de autorización genera un token de acceso.
15. El servidor de autorización entrega el token de acceso a Auth0.
16. Con el token de acceso, Auth0 solicita un recurso al Resource Server.
17. El Resource Server proporciona el recurso para completar el flujo.

<div id="configure-private-key-jwt-client-authentication">
  ## Configurar la autenticación de cliente con clave privada JWT
</div>

Puede configurar las conexiones empresariales de OIDC y Okta Workforce para usar la autenticación de cliente con clave privada JWT mediante el Auth0 Dashboard o Management API. A continuación, se indican los pasos para cada método.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  * Auth0 genera automáticamente un par de claves de firma pública y privada para cada conexión.
  * Puede usar los siguientes algoritmos para firmar aserciones JWT de cliente: `RS256`, `RS384`, `RS512`, `PS256`, `PS384`, `ES256` y `ES384` para las conexiones empresariales de Okta y OIDC. Si no se especifica ninguno, se usa `RS256` de forma predeterminada.
  * Los JWT firmados vencen automáticamente a los 60 segundos.
</Callout>

<div id="auth0-dashboard">
  #### Auth0 Dashboard
</div>

Puedes usar el Auth0 Dashboard para configurar la autenticación de cliente con clave privada JWT en conexiones OIDC y Okta Workforce, tanto nuevas como existentes.

<Tabs>
  <Tab title="Nueva conexión">
    1. En tu Auth0 Dashboard, ve a [Authentication &gt; Enterprise](https://manage.auth0.com/#/connections/enterprise).
    2. Junto a **OpenID Connect** o **Okta Workforce**, selecciona **Crear**.
    3. En la sección **General**, completa los datos de tu nueva conexión, incluidos su nombre y la URL de descubrimiento.
    4. Configura los siguientes campos para habilitar la autenticación con clave privada JWT:

       * Establece **Communication Channel** en **Back Channel**.
       * Establece **Authentication Method** en **Private Key JWT**.
    5. Selecciona **Crear** para guardar tu nueva conexión.
  </Tab>

  <Tab title="Conexión existente">
    1. En tu Auth0 Dashboard, ve a [Authentication &gt; Enterprise](https://manage.auth0.com/#/connections/enterprise).
    2. Junto a **OpenID Connect** o **Okta Workforce**, selecciona **Examinar**.
    3. Elige la conexión correspondiente. Luego, ve a su pestaña **Credenciales**.
    4. En **Authentication Settings**, configura lo siguiente:

       * Establece **Communication Channel** en **Back Channel**.
       * Establece **Authentication Method** en **Private Key JWT**.
    5. Selecciona **Guardar**.
    6. En la ventana emergente de confirmación, selecciona **Cambiar** para aplicar los cambios.
  </Tab>
</Tabs>

  <Accordion title="URI JWKS pública">
    Algunos proveedores de identidad permiten proporcionar claves públicas para `private_key_jwt` en forma de una URI JWKS (JSON Web Key Set) pública.

    Si se han generado claves públicas para una conexión, puedes recuperarlas agregando la siguiente URI a la configuración de tu IdP:

    ```http wrap lines
    https://{auth0 domain}/oauth/connection/{connection name}/.well-known/jwks.json
    ```

    <Warning>
      Las URI de JWKS cuentan para los límites de frecuencia globales. Puedes usar caché de claves públicas para evitar alcanzar estos límites. Como práctica recomendada, Auth0 recomienda un intervalo de caché de al menos 5-10 minutos para evitar llamar al endpoint de la URI JWKS en cada intento de inicio de sesión.

<div id="rotate-signing-keys">
  ## Rotar las claves de firma
</div>

La autenticación de cliente con clave privada JWT admite la rotación de claves de firma para ofrecer mayor seguridad frente a la naturaleza estática y duradera de los secretos de cliente compartidos. Rotar las claves de firma mejora la seguridad al limitar el tiempo de exposición de cada clave y reducir la ventana de oportunidad para que un atacante la comprometa. También permite responder con rapidez tras un incidente de seguridad.

Para evitar interrupciones, Auth0 recomienda rotar las claves de firma después de un año. Puede usar el Auth0 Dashboard o la Management API para rotar las claves de firma:

<AccordionGroup>
  <Accordion title="Auth0 Dashboard">
    Para rotar sus claves de firma desde el Auth0 Dashboard:

    1. Vaya a [Autenticación &gt; empresarial](https://manage.auth0.com/#/connections/enterprise).
    2. Junto a **OpenID Connect** u **Okta Workforce**, seleccione **Examinar**.
    3. Elija la conexión correspondiente. Luego, abra la pestaña **Credenciales**.
    4. En la sección **Credenciales**, seleccione **Rotar claves**.
    5. En la ventana emergente, seleccione **Guardar** para confirmar la rotación.

    Después de la rotación, cualquier JWT en tránsito firmado con la clave anterior quedará inactivo de inmediato y podría no superar la verificación con su IdP.
  </Accordion>

  <Accordion title="Management API">
    Para ver las claves públicas a través de la Management API, llame al endpoint Rotate Connection Signing Keys con el ID de su conexión.

    <Warning>
      El uso de este endpoint requiere ambos scopes: `create:connections_keys` y `update:connections_keys`.
    </Warning>

    ```js lines
    POST /v2/connections/{id}/keys/rotate
    ```

    Después de la rotación, cualquier JWT en tránsito firmado con la clave anterior quedará inactivo de inmediato y podría no superar la verificación con su IdP.
  </Accordion>
</AccordionGroup>

<Card title="Comprender la rotación de claves">
  En su conexión OIDC u Okta Workforce, a sus claves de firma se les asigna uno de los siguientes estados:

  * **Actual**: La clave de firma que la aplicación usa actualmente.
  * **Siguiente**: La siguiente clave de firma que usará la aplicación después de que se revoque la clave actual.
  * **Anterior**: Una clave de firma caducada o revocada de otro modo que ya no se usa.

  Cuando la autenticación de cliente con clave privada JWT se habilita por primera vez para una conexión, solo se genera un par de claves `current` y `next`. Una clave solo se marca como `previous` después de que se produce una rotación.

  Al rotar las claves de firma, se producen los siguientes cambios:

  1. La clave `current` se elimina y se revoca, y cualquier JWT firmado con esta clave no superará la verificación con el IdP si el IdP se configuró con `jwks_uri`.
  2. A la clave `current` se le asigna el estado `previous`.
  3. La clave `next` pasa a ser la clave activa y recibe el estado `current`. A partir de ese momento, las aserciones JWT de cliente se firmarán con esta clave.
  4. Se genera automáticamente una nueva clave de firma para reemplazar la clave rotada. La nueva clave de firma tiene el estado `next`.
</Card>
