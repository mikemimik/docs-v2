<Card title="Overview">
  Conceptos clave

  * Auth0 admite el protocolo OAuth 2.0 elaborado por el Grupo de Trabajo de Ingeniería de Internet (IETF).
  * Lee sobre roles, tipos de concesión (o flujos de trabajo) y endpoints de la especificación OAuth 2.0.
</Card>

El marco de autorización [OAuth 2.0](https://tools.ietf.org/html/rfc6749) es un protocolo que permite a un usuario otorgar a un sitio web o aplicación de terceros acceso a los recursos protegidos del usuario, sin necesidad de revelar sus credenciales a largo plazo ni siquiera su identidad.

<Tooltip tip="OAuth 2.0: Marco de autorización que define protocolos y flujos de trabajo de autorización." cta="Ver glosario" href="/docs/es/glossary?term=OAuth">
  OAuth
</Tooltip>

introduce una capa de autorización y separa el rol del cliente del del

<Tooltip tip="Resource Owner: Entidad (como un usuario o una aplicación) capaz de otorgar acceso a un recurso protegido." cta="Ver glosario" href="/docs/es/glossary?term=resource+owner">
  propietario del recurso
</Tooltip>

. En OAuth, el cliente solicita acceso a los recursos controlados por el propietario del recurso y alojados en el

<Tooltip tip="Resource Server: Servidor que aloja recursos protegidos. Los servidores de recursos aceptan y responden a solicitudes de recursos protegidos." cta="Ver glosario" href="/docs/es/glossary?term=resource+server">
  servidor de recursos
</Tooltip>

y recibe un conjunto de credenciales distinto al del propietario del recurso. En lugar de usar las credenciales del propietario del recurso para acceder a los recursos protegidos, el cliente obtiene un

<Tooltip tip="Access Token: Credencial de autorización, en forma de cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=access+token">
  access token
</Tooltip>

: una cadena que indica un scope específico, un tiempo de vida y otros atributos de acceso. Los access tokens son emitidos a clientes de terceros por un

<Tooltip tip="Authorization Server: Servidor centralizado que contribuye a definir los límites del acceso de un usuario. Por ejemplo, tu servidor de autorización puede controlar los datos, las tareas y las funcionalidades disponibles para un usuario." cta="Ver glosario" href="/docs/es/glossary?term=authorization+server">
  servidor de autorización
</Tooltip>

con la aprobación del propietario del recurso. Luego, el cliente usa el access token para acceder a los recursos protegidos alojados en el servidor de recursos.

Auth0 genera access tokens para escenarios de autorización de API, en [formato JSON web token (JWT)](/docs/es/secure/tokens/json-web-tokens/json-web-token-structure). Los permisos representados por el access token, en términos de OAuth, se conocen como [scopes](/docs/es/get-started/apis/scopes). Cuando una aplicación se autentica con Auth0, especifica los scopes que desea. Si el usuario autoriza esos scopes, el access token representará dichos scopes autorizados.

<div id="roles">
  ## Roles
</div>

Un flujo de OAuth 2.0 tiene los siguientes roles:

* **Propietario del recurso**: entidad que puede conceder acceso a un recurso protegido. Normalmente, es el usuario final.
* **Servidor de recursos**: servidor que aloja los recursos protegidos. Es la API a la que quieres acceder.
* **Cliente**: aplicación que solicita acceso a un recurso protegido en nombre del Propietario del recurso.
* **Servidor de autorización**: servidor que autentica al Propietario del recurso y emite tokens de acceso tras obtener la autorización adecuada. En este caso, Auth0.

<div id="grant-types">
  ## Tipos de concesión
</div>

OAuth 2.0 define cuatro flujos para obtener un token de acceso. Estos flujos se denominan tipos de concesión. [Decidir cuál es el más adecuado para tu caso](/docs/es/get-started/authentication-and-authorization-flow/which-oauth-2-0-flow-should-i-use) depende principalmente del tipo de aplicación.

* [Flujo de código de autorización](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow): lo utilizan las aplicaciones web que se ejecutan en un servidor. También lo utilizan las aplicaciones móviles mediante la técnica [Proof Key for Code Exchange (PKCE)](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce).
* [Flujo implícito con Form Post](/docs/es/get-started/authentication-and-authorization-flow/implicit-flow-with-form-post): lo utilizan las aplicaciones basadas en JavaScript (aplicaciones de página única) que se ejecutan en el navegador del usuario.
* [Flujo de contraseña del propietario del recurso](/docs/es/get-started/authentication-and-authorization-flow/resource-owner-password-flow): lo utilizan las aplicaciones de alta confianza.
* [Flujo de credenciales de cliente](/docs/es/get-started/authentication-and-authorization-flow/client-credentials-flow): se utiliza para la comunicación de máquina a máquina.

La especificación también proporciona un mecanismo de extensibilidad para definir tipos de concesión adicionales. Para obtener más información sobre cómo funciona cada tipo de concesión y cuándo debe utilizarse, consulta [Flujos de autenticación y autorización](/docs/es/get-started/authentication-and-authorization-flow).

<div id="endpoints">
  ## Endpoints
</div>

OAuth 2.0 utiliza dos endpoints: el endpoint `/authorize` y el endpoint `/oauth/token`.

<div id="authorization-endpoint">
  ### Endpoint de autorización
</div>

El endpoint `/authorize` se utiliza para interactuar con el propietario del recurso y obtener autorización para acceder al recurso protegido. Para entenderlo mejor, imagina que quieres iniciar sesión en un servicio con tu cuenta de Google. Primero, el servicio te redirige a Google para autenticarte (si aún no has iniciado sesión) y luego verás una pantalla de consentimiento, donde se te pedirá que autorices al servicio a acceder a algunos de tus datos (recursos protegidos); por ejemplo, tu dirección de correo electrónico y tu lista de contactos.

Los parámetros de solicitud del endpoint `/authorize` son:

| Parámetro       | Descripción                                                                                                                                                                                                                                                                                                                                                                                                                                                                         |
| --------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `response_type` | Indica al servidor de autorización qué concesión debe ejecutar.                                                                                                                                                                                                                                                                                                                                                                                                                     |
| `response_mode` | (Opcional) Cómo se formatea el resultado de la solicitud de autorización. Valores:<br />- `query`: para la concesión Authorization Code. `302 Found` desencadena una redirección.<br />- `fragment`: para la concesión Implicit. `302 Found` desencadena una redirección.<br />- `form_post`: `200 OK` con los parámetros de respuesta incrustados en un formulario HTML como parámetros ocultos.<br />- `web_message`: para autenticación silenciosa. Usa mensajería web de HTML5. |
| `client_id`     | El ID de la aplicación que solicita autorización.                                                                                                                                                                                                                                                                                                                                                                                                                                   |
| `redirect_uri`  | Contiene una URL. Una respuesta correcta de este endpoint da como resultado una redirección a esta URL.                                                                                                                                                                                                                                                                                                                                                                             |
| `scope`         | Una lista de permisos separada por espacios que la aplicación requiere.                                                                                                                                                                                                                                                                                                                                                                                                             |
| `state`         | Un valor opaco, usado con fines de seguridad. Si este parámetro de solicitud se establece en la solicitud, se devuelve a la aplicación como parte de la `redirect_uri`.                                                                                                                                                                                                                                                                                                             |
| `connection`    | Especifica el tipo de conexión para las conexión passwordless                                                                                                                                                                                                                                                                                                                                                                                                                       |

Puedes configurar parámetros de consulta personalizados cuando tu aplicación haga la llamada inicial al endpoint `/authorize` para autenticar a un usuario. Puedes usar parámetros de consulta personalizados para proporcionar contexto adicional a la plantilla de página de la experiencia de <Tooltip tip="Universal Login: Tu aplicación redirige a Universal Login, alojado en el servidor de autorización de Auth0, para verificar la identidad de un usuario." cta="Ver glosario" href="/docs/es/glossary?term=Universal+Login">Universal Login</Tooltip>.

Debes habilitar ID First para usar el parámetro `connection`. Para obtener más información sobre el parámetro `connection` y la experiencia de Universal Login, consulta [Passwordless for Universal Login](/docs/es/authenticate/passwordless/passwordless-with-universal-login).

Los parámetros de consulta con el prefijo `ext-` aparecen automáticamente en el [contexto de la plantilla de página](/docs/es/customize/login-pages/universal-login/customize-templates#custom-query-parameters).

Este endpoint se utiliza con los tipos de concesión Authorization Code e Implicit. El servidor de autorización necesita saber qué tipo de concesión quiere usar la aplicación, ya que eso afecta al tipo de credencial que emitirá:

* Para la concesión Authorization Code, emitirá un código de autorización (que más adelante puede intercambiarse por un token de acceso en el endpoint `/oauth/token`).
* Para la concesión Implicit, emitirá un token de acceso, que es una cadena opaca (o un <Tooltip tip="JSON Web Token (JWT): Formato estándar de token de ID (y a menudo formato de Access Token) utilizado para representar reclamaciones de forma segura entre dos partes." cta="Ver glosario" href="/docs/es/glossary?term=JWT">JWT</Tooltip> en una implementación de Auth0) que indica quién ha autorizado qué permisos (scopes) a qué aplicación.

Para indicar al servidor de autorización qué tipo de concesión debe usar, el parámetro de solicitud `response_type` se utiliza de la siguiente manera:

* Para la concesión Authorization Code, usa `response_type=code` para incluir el código de autorización.
* Para la concesión Implicit, usa `response_type=token` para incluir un token de acceso. Una alternativa es usar `response_type=id_token token` para incluir tanto un token de acceso como un <Tooltip tip="ID Token: Credencial destinada al propio cliente, en lugar de para acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=ID+token">token de identidad</Tooltip>.

Un token de identidad es un JWT que contiene información sobre el usuario que ha iniciado sesión. Fue introducido por <Tooltip tip="OpenID: Estándar abierto de autenticación que permite a las aplicaciones verificar la identidad de los usuarios sin recopilar ni almacenar información de inicio de sesión." cta="Ver glosario" href="/docs/es/glossary?term=OpenID">OpenID</Tooltip> Connect (OIDC).

La [especificación OAuth 2.0 Multiple Response Type Encoding Practices](https://openid.net/specs/oauth-v2-multiple-response-types-1_0.html) añadió un parámetro que especifica cómo se formatea el resultado de la solicitud de autorización. Este parámetro se llama `response_mode`. Es opcional y puede tomar los siguientes valores:

| Valor         | Descripción                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                  |
| ------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `query`       | Este es el valor predeterminado para la concesión Authorization Code. Una respuesta exitosa es `302 Found`, lo que desencadena una redirección a `redirect_uri`. Los parámetros de respuesta se incluyen en el componente de consulta (la parte que va después de `?`) de `redirect_uri`, en el encabezado `Location`.<br />Por ejemplo:<br />`HTTP/1.1 302 Found`<br />`Location: https://my-redirect-uri.callback?code=js89p2x1`, donde el código de autorización es `js89p21`.                                                                                                                            |
| `fragment`    | Este es el valor predeterminado para la concesión Implicit. Una respuesta exitosa es `302 Found`, lo que desencadena una redirección a `redirect_uri` (que es un parámetro de solicitud). Los parámetros de respuesta se incluyen en el componente de fragmento (la parte que va después de `#`) de `redirect_uri`, en el encabezado `Location`.<br />Por ejemplo:<br />`HTTP/1.1 302 Found`<br />`Location: https://my-redirect-uri/callback#access_token=eyB...78f&token_type=Bearer&expires_in=3600`.                                                                                                     |
| `form_post`   | El modo de respuesta está definido por la [especificación OAuth 2.0 Form Post Response Mode](https://openid.net/specs/oauth-v2-form-post-response-mode-1_0.html). Una respuesta exitosa es `200 OK` y los parámetros se incluyen en un formulario HTML como `params` ocultos. El `action` del formulario es `redirect_uri` y el atributo `onload` está configurado para enviar el formulario. Después de que el navegador carga el HTML, se realiza una redirección a `redirect_uri`.                                                                                                                        |
| `web_message` | Este modo de respuesta está definido en la [especificación OAuth 2.0 Web Message Response Mode](https://tools.ietf.org/html/draft-sakimura-oauth-wmrm-00). Usa HTML5 Web Messaging en lugar de la redirección para la respuesta de autorización desde el endpoint de autorización `/authorization`. Esto resulta especialmente útil cuando se usa autenticación silenciosa. Para usar este modo de respuesta, debes registrar la URL de tu aplicación en el campo **Allowed Web Origins** de la [configuración de la aplicación](https://manage.auth0.com/#/applications/\{yourClientId}/settings) de Auth0. |

<div id="token-endpoint">
  ### Endpoint de token
</div>

La aplicación usa el endpoint `/oauth/token` para obtener un token de acceso o un <Tooltip tip="Token de actualización: token usado para obtener un token de acceso renovado sin obligar a los usuarios a volver a iniciar sesión." cta="Ver glosario" href="/docs/es/glossary?term=refresh+token">token de actualización</Tooltip>. Se usa en todos los flujos, excepto en el flujo implícito, porque en ese caso el token de acceso se emite directamente.

* En el flujo de código de autorización, la aplicación intercambia el código de autorización que obtuvo del endpoint de autorización por un token de acceso.
* En el flujo de credenciales de cliente y en el intercambio de concesión de credenciales de contraseña del propietario del recurso, la aplicación se autentica con un conjunto de credenciales y luego obtiene un token de acceso.

<div id="state-parameters">
  ## Parámetros de `state`
</div>

Los protocolos de autorización proporcionan un parámetro `state` que le permite restaurar el estado anterior de su aplicación. El parámetro `state` conserva un objeto de estado establecido por el cliente en la solicitud de autorización y lo pone a disposición del cliente en la respuesta. La principal razón para usar el parámetro `state` es mitigar los ataques CSRF. Consulte [Use OAuth 2.0 State Parameters](/docs/es/secure/attack-protection/state-parameters) para obtener más información.
