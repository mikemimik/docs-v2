import {AuthLink} from "/snippets/AuthLink.jsx";

Para aumentar la seguridad de tus transacciones, puedes firmar o cifrar tanto tus solicitudes como tus respuestas en el protocolo <Tooltip tip="Security Assertion Markup Language (SAML): Protocolo estandarizado que permite a dos partes intercambiar información de autenticación sin una contraseña." cta="Ver glosario" href="/docs/es/glossary?term=SAML">SAML</Tooltip>. En este artículo, encontrarás configuraciones para escenarios específicos, organizadas en dos casos de uso:

* Auth0 como proveedor de servicios SAML (por ejemplo, una conexión SAML)
* Auth0 como <Tooltip tip="Identity Provider (IdP): Servicio que almacena y gestiona identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=identity+provider">proveedor de identidad</Tooltip> SAML (por ejemplo, una aplicación configurada con el addon SAML Web App)

<div id="auth0-as-the-saml-service-provider">
  ## Auth0 como proveedor de servicios SAML
</div>

Estos escenarios corresponden a cuando Auth0 actúa como proveedor de servicios SAML, lo que significa que Auth0 se conecta a un proveedor de identidad SAML mediante la creación de una conexión SAML.

<div id="sign-the-saml-authentication-request">
  ### Firmar la solicitud de autenticación SAML
</div>

Si Auth0 es el **proveedor de servicios** de SAML, puede firmar la solicitud de autenticación que Auth0 envía al IdP de la siguiente manera:

1. Vaya a [Auth0 Dashboard &gt; Autenticación &gt; Empresarial](https://manage.auth0.com/#/connections/enterprise) y seleccione **SAML**.
2. Seleccione el nombre de la conexión para ver sus detalles.
3. Busque **Sign Request** y active su interruptor.
4. Descargue el certificado que aparece debajo del interruptor **Sign Request** y proporcióneselo al IdP para que pueda validar la firma.

<div id="enabledisable-deflate-encoding">
  #### Habilitar/deshabilitar la codificación deflate
</div>

De forma predeterminada, las solicitudes de autenticación SAML se envían mediante HTTP-Redirect y usan codificación deflate, lo que coloca la firma en un parámetro de consulta.

Para desactivar la codificación deflate, puedes hacer una [llamada PATCH al endpoint Actualizar una conexión de la Management API](https://auth0.com/docs/api/management/v2#!/Connections/patch_connections_by_id) y establecer la opción `deflate` en `false`.

Al actualizar el objeto `options` de una conexión, se sobrescribe todo el objeto `options`. Para conservar las opciones de conexión anteriores, obtén el objeto `options` existente y agrégale nuevos pares clave-valor.

Endpoint: `https://{yourDomain}/api/v2/connections/{yourConnectionId}`

Payload:

```json lines
{
	{ 
		"options" : {
			[...], // todas las demás opciones de conexión
		  "deflate": false
    }
  }
}
```

<div id="use-a-custom-key-to-sign-requests">
  ### Usa una clave personalizada para firmar solicitudes
</div>

De forma predeterminada, Auth0 usa la clave privada del tenant para firmar solicitudes SAML (cuando el interruptor **Sign Request** está habilitado). También puedes proporcionar tu propio par de clave pública y privada para firmar solicitudes procedentes de una conexión específica.

Puedes generar tu propio certificado y clave privada con este comando:

```bash wrap lines
openssl req -x509 -nodes -sha256 -days 3650 -newkey rsa:2048 -keyout private_key.key -out certificate.crt
```

No es posible cambiar la clave usada para firmar requests en la conexión desde la UI del Dashboard, así que tendrás que usar el [endpoint actualizar una conexión](https://auth0.com/docs/api/management/v2#!/Connections/patch_connections_by_id) de la <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip> v2 y añadir una propiedad `signing_key` al objeto `options`, como se muestra en el ejemplo de payload a continuación.

Al actualizar el objeto `options` de una conexión, se sobrescribe todo el objeto `options`. Para conservar las opciones anteriores de la conexión, obtén el objeto `options` existente y añádele nuevos pares clave/valor.

Endpoint: `https://{yourDomain}/api/v2/connections/{yourConnectionId}`

Payload:

```json lines
{
	{ 
		"options" : {
			[...], // todas las demás opciones de conexión
		  "signing_key": {
				"key":"-----BEGIN PRIVATE KEY-----\n...{your private key here}...\n-----END PRIVATE KEY-----",
				"cert":"-----BEGIN CERTIFICATE-----\n...{your public key cert here}...\n-----END CERTIFICATE-----"
			}
    }
	}
}
```

Para saber cómo obtener la clave privada y el certificado en formato de cadena JSON para usarlos en el payload, consulta [Trabajar con certificados, claves y cadenas](/docs/es/authenticate/protocols/saml/saml-sso-integrations/work-with-certificates-and-keys-as-strings).

<div id="receive-signed-saml-authentication-responses">
  ### Recibir respuestas de autenticación SAML firmadas
</div>

Si Auth0 es el **proveedor de servicios** SAML, todas las respuestas SAML de tu proveedor de identidad deben estar firmadas para indicar que no han sido manipuladas por un tercero no autorizado.

Deberás configurar Auth0 para validar las firmas de las respuestas. Para ello, obtén un certificado de firma del proveedor de identidad y carga ese certificado en tu Conexión de Auth0:

1. Ve a [Auth0 Dashboard &gt; Autenticación &gt; empresarial](https://manage.auth0.com/#/connections/enterprise) y selecciona **SAML**.
2. Selecciona el nombre de la conexión que quieres ver.
3. Busca **X509 Signing Certificate** y carga el certificado.
4. Selecciona **Guardar cambios**.

Auth0 puede aceptar una respuesta firmada para la aserción, la respuesta o ambas.

<div id="receive-encrypted-saml-authentication-assertions">
  ### Recibir aserciones de autenticación SAML cifradas
</div>

Si Auth0 es el proveedor de servicios SAML, puede que necesite recibir aserciones cifradas de un proveedor de identidad. Para ello, debe proporcionar al IdP el certificado de clave pública del tenant. El IdP cifra la aserción SAML con la clave pública y la envía a Auth0, que la descifra con la clave privada del tenant.

Use los siguientes enlaces para obtener el certificado de clave pública en distintos formatos:

* <AuthLink href="https://{yourDomain}/cer?cert=connection">CER</AuthLink>
* <AuthLink href="https://{yourDomain}/pem?cert=connection">PEM</AuthLink>
* <AuthLink href="https://{yourDomain}/rawpem?cert=connection">PEM sin formato</AuthLink>
* <AuthLink href="https://{yourDomain}/pb7?cert=connection">PKCS#7</AuthLink>
* <AuthLink href="https://{yourDomain}/fingerprint?cert=connection">Huella digital</AuthLink>

Descargue el certificado en el formato solicitado por el IdP.

<div id="set-content-decryption-algorithms">
  #### Establecer algoritmos de descifrado de contenido
</div>

De forma predeterminada, Auth0 admite automáticamente los algoritmos enumerados en el [Perfil de algoritmos](/docs/es/authenticate/protocols/saml/saml-sso-integrations/algorithm-profiles) más reciente para descifrar aserciones SAML.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Si la aserción está cifrada con un algoritmo que no figura en la lista, Auth0 rechazará la aserción.
</Callout>

Para especificar un perfil diferente o usar un algoritmo que no figure en la lista, debes actualizar la conexión mediante el endpoint [actualizar una conexión](https://auth0.com/docs/api/management/v2/connections/patch-connections-by-id) y cambiar la propiedad `assertion_decryption_settings`, como se muestra en el ejemplo de payload a continuación.

Cuando actualizas el objeto options de una conexión, la nueva configuración reemplaza por completo el objeto `options`. Para conservar las opciones anteriores de la conexión, obtén el objeto options existente y agrégale nuevos pares clave/valor.

Endpoint: `https://{yourDomain}/api/v2/connections/{yourConnectionId}`
Payload:

```json
{
  "options": {
    [...], // todas las demás opciones de conexión
    "assertion_decryption_settings": {
      "algorithm_profile": "v2026-1",
      "algorithm_exceptions": []
    }
  }
}
```

<div id="use-your-key-pair-to-decrypt-encrypted-responses">
  ### Usa tu par de claves para descifrar respuestas cifradas
</div>

Como se indicó anteriormente, Auth0 usará de forma predeterminada el par de claves privada/pública de tu tenant para gestionar el cifrado. También puedes proporcionar tu propio par de claves pública/privada si algún escenario avanzado lo requiere.

No es posible cambiar en la interfaz del Dashboard el par de claves que se usa para cifrar y descifrar solicitudes en la conexión, por lo que tendrás que usar el [endpoint actualizar una conexión](https://auth0.com/docs/api/management/v2#!/Connections/patch_connections_by_id) de la Management API v2 y añadir una propiedad `decryptionKey` al objeto `options`, como se muestra en el ejemplo de payload a continuación.

Actualizar el objeto `options` de una conexión sobrescribe por completo todo el objeto `options`. Para conservar las opciones de conexión anteriores, obtén el objeto `options` actual y añádele nuevos pares clave/valor.

Endpoint: `https://{yourDomain}/api/v2/connections/{yourConnectionId}`

Payload:

```json lines
{
	{ 
		"options" : {
			[...], // todas las demás opciones de conexión
		  "decryptionKey": {
				"key":"-----BEGIN PRIVATE KEY-----\n...{your private key here}...\n-----END PRIVATE KEY-----",
				"cert":"-----BEGIN CERTIFICATE-----\n...{your public key cert here}...\n-----END CERTIFICATE-----"
			}
	}
}
```

Los metadatos SAML disponibles para la conexión se actualizarán con el certificado proporcionado para que el proveedor de identidad pueda recuperarlo y usarlo para firmar la respuesta SAML.

<div id="auth0-as-the-saml-identity-provider">
  ## Auth0 como proveedor de identidad SAML
</div>

Este escenario aplica cuando Auth0 actúa como proveedor de identidad SAML para una aplicación. En el Dashboard, esto se representa mediante una **aplicación** con el SAML Web App Addon habilitado.

<div id="sign-the-saml-responsesassertions">
  ### Firmar las respuestas/aserciones SAML
</div>

Si Auth0 es el proveedor de identidad SAML, firmará las aserciones SAML con la clave privada del tenant y proporcionará al proveedor de servicios la clave pública/el certificado necesarios para validar la firma.

Para firmar las aserciones SAML:

1. Ve a [Auth0 Dashboard &gt; Applications](https://manage.auth0.com/#/applications) y selecciona el nombre de la aplicación que quieres ver.
2. Desplázate hasta la parte inferior de la página **Settings**, selecciona **Show Advanced Settings** y, a continuación, la vista **Certificates**.
3. Selecciona **Download Certificate** y elige el formato en el que quieres recibir tu certificado de firma.
4. Envía tu certificado al proveedor de servicios.

De forma predeterminada, Auth0 firma la **aserción** SAML dentro de la respuesta. Para firmar la **respuesta** SAML en su lugar:

1. Ve a [Auth0 Dashboard &gt; Applications](https://manage.auth0.com/#/applications) y selecciona el nombre de la aplicación que quieres ver.
2. Selecciona la vista **Addons**.
3. Selecciona **SAML2 Web App** para ver su configuración y localiza el bloque de código **Settings**.
4. Localiza la clave **`"signResponse"`**. Descoméntala (o agrégala, si es necesario) y luego establece su valor en `true` (el valor predeterminado es `false`). La configuración debería verse así:

   ```json lines
   {
     [...], // otra configuración
     "signResponse": true
   }
   ```

<div id="change-the-signing-key-for-saml-responses">
  #### Cambiar la clave de firma de las respuestas SAML
</div>

De forma predeterminada, Auth0 usará el par de claves privada/pública asignado a tu tenant para firmar respuestas SAML o aserciones. En casos muy concretos, quizá quieras proporcionar tu propio par de claves. Puedes hacerlo con una rule como esta:

```javascript lines expandable
/**
* Manejador que se invocará durante la ejecución de un flujo Post Login.
*
* @param {Event} event - Detalles sobre el usuario y el contexto en el que está iniciando sesión.
* @param {PostLoginAPI} api - Interfaz cuyos métodos pueden usarse para cambiar el comportamiento del inicio de sesión.
*/
exports.onExecutePostLogin = async (event, api) => {

    // reemplazar con el ID de la aplicación que tiene el SAML Web App Addon habilitado
      // y para la cual se desea cambiar el par de claves de firma.
      const samlIdpClientId = 'YOUR_SAML_APP_CLIENT_ID';

    // hacer esto solo para el ID de cliente específico. Si tiene varios IdPs que requieren
    // certificados personalizados, agregue una sentencia "if" para cada uno.  
    if (event.client.client_id === samlIdpClientId) {

    // proporcione aquí su propia clave privada y certificado
    // consulte https://auth0.com/docs/authenticate/protocols/saml/saml-sso-integrations/work-with-certificates-and-keys-as-strings
    // para instrucciones de formato: básicamente, comience con un certificado en formato PEM y
    // reemplace los saltos de línea con "\n"
    const signingCert = "-----BEGIN CERTIFICATE-----\nnMIIC8jCCAdqgAwIBAgIJObB6jmhG0QIEMA0GCSqGSIb3DQEBBQUAMCAxHjAcBgNV[..all the other lines..]-----END CERTIFICATE-----\n";
    const signingKey = "-----BEGIN PRIVATE KEY-----\nnMIIC8jCCAdqgAwIBAgIJObB6jmhG0QIEMA0GCSqGSIb3DQEBBQUAMCAxHjAcBgNV[..all the other lines..]-----END PRIVATE KEY-----\n";

    api.samlResponse.setCert(signingCert)    
    api.samlResponse.setKey(signingKey);

  }
  };
```

Para saber cómo convertir la clave privada y los archivos de certificado en cadenas que puedas usar en una rule, consulta [Trabajar con certificados, claves y cadenas](/docs/es/authenticate/protocols/saml/saml-sso-integrations/work-with-certificates-and-keys-as-strings).

<div id="receive-signed-saml-authentication-requests">
  ### Recibir solicitudes de autenticación SAML firmadas
</div>

Si Auth0 es el proveedor de identidad SAML, puede recibir solicitudes firmadas con la clave privada del proveedor de servicios. Auth0 usa la clave pública o el certificado para validar la firma.

Para configurar la validación de la firma:

1. Descargue el certificado del proveedor de servicios con la clave pública.
2. Vaya a [Auth0 Dashboard &gt; Applications](https://manage.auth0.com/#/applications) y seleccione el nombre de la aplicación para verla.
3. Seleccione la vista **Addons**.
4. Seleccione **SAML2 Web App** para ver su configuración y busque el bloque de código **Settings**.
5. Busque la clave **`"signingCert"`**. Descoméntela (o agréguela, si es necesario) y luego establezca su valor en el certificado que descargó del proveedor de servicios. La configuración debería verse así:

   ```json lines
   {
     [...], // otros ajustes
     "signingCert": "-----BEGIN CERTIFICATE-----\nMIIC8jCCAdqgAwIBAgIJObB6jmhG0QIEMA0GCSqGSIb3DQEBBQUAMCAxHjAcBgNV\n[..all the other lines..]-----END CERTIFICATE-----\n"
   }
   ```

<div id="send-encrypted-saml-authentication-assertions">
  ### Enviar aserciones de autenticación SAML cifradas
</div>

Si Auth0 es el proveedor de identidad SAML, puedes usar [Actions](/docs/es/customize/actions) para cifrar las aserciones SAML que envía. También puedes seleccionar el algoritmo de cifrado que se usa para cifrar las aserciones. Auth0 recomienda usar `aes256-gcm` para una configuración de seguridad más sólida.

Debes obtener el certificado y la clave pública del proveedor de servicios. Si solo obtuviste el certificado, puedes extraer la clave pública con `openssl`. Suponiendo que el archivo del certificado se llame `certificate.pem`, puedes ejecutar:

`openssl x509 -in certificate.pem -pubkey -noout > public_key.pem`

Una vez que tengas los archivos del certificado y la clave pública, debes [convertirlos en cadenas](/docs/es/authenticate/protocols/saml/saml-sso-integrations/work-with-certificates-and-keys-as-strings) para usarlos en una Action. La Action tendrá este aspecto:

```js lines
exports.onExecutePostLogin = async (event, api) => {

// esta Action configura una clave pública específica para cifrar la aserción SAML generada por Auth0
  if (
    event.client.client_id ===
    "THE_CLIENT_ID_OF_THE_APP_WITH_THE_SAML_APP_ADDON"
  ) {
    const encryptionCert =
      "-----BEGIN CERTIFICATE-----\nnMIIC8jCCAdqgAwIBAgIJObB6jmhG0QIEMA0GCSqGSIb3DQEBBQUAMCAxHjAcBgNV[..all the other lines..]-----END CERTIFICATE-----\n";
    const encryptionPublicKey =
      "-----BEGIN PUBLIC KEY-----\nnMIIC8jCCAdqgAwIBAgIJObB6jmhG0QIEMA0GCSqGSIb3DQEBBQUAMCAxHjAcBgNV[..all the other lines..]-----END PUBLIC KEY-----\n";

    api.samlResponse.setEncryptionCert(encryptionCert);
    api.samlResponse.setEncryptionPublicKey(encryptionPublicKey);
    api.samlResponse.setEncryptionAlgorithm("aes256-gcm");
  }
};
```

Auth0 admite los siguientes algoritmos para el cifrado de aserciones:

* [`aes256-gcm`](https://www.w3.org/TR/xmlenc-core1/#sec-AES-GCM) **(recomendado)**: Cifrado autenticado que proporciona tanto confidencialidad como integridad. Es resistente a ataques de oráculo de validación de formato.
* [`aes256-cbc`](https://www.w3.org/TR/xmlenc-core1/#sec-AES) (predeterminado): No ofrece garantías de integridad. Cuando una Action no usa el objeto [`api.samlResponse.setEncryptionAlgorithm`](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger/post-login-api-object#api-samlresponse-setencryptioncert-encryptioncert) para establecer el algoritmo de cifrado, Auth0 usa de forma predeterminada el algoritmo `aes256-cbc` y registra una advertencia de obsolescencia en los registros de tu tenant.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Para el transporte de claves, Auth0 usa [`rsa-oaep`](https://www.w3.org/TR/xmlenc-core1/#rsa-oaep), incluidas las funciones MGF1 y SHA1.
</Callout>

<Warning>
  Auth0 tiene previsto actualizar el algoritmo de cifrado predeterminado a `aes256-gcm`.

  Para mantener un comportamiento coherente después de que cambie el algoritmo predeterminado, te recomendamos cambiar a `aes256-gcm`:

  1. Verifica que tu proveedor de servicios SAML admita `aes256-gcm` y, si no es así, ponte en contacto con su equipo de soporte.
  2. Establece el algoritmo de cifrado en el código de tu Action con `api.samlResponse.setEncryptionAlgorithm("aes256-gcm");`.
</Warning>
