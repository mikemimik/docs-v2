Muchas instrucciones para configurar una federación <Tooltip tip="Security Assertion Markup Language (SAML): Protocolo estandarizado que permite a dos partes intercambiar información de autenticación sin una contraseña." cta="Ver glosario" href="/docs/es/glossary?term=SAML">SAML</Tooltip> comienzan con el <Tooltip tip="Security Assertion Markup Language (SAML): Protocolo estandarizado que permite a dos partes intercambiar información de autenticación sin una contraseña." cta="Ver glosario" href="/docs/es/glossary?term=Single+Sign-on">inicio de sesión único</Tooltip> (SSO) iniciado por el proveedor de servicios. El proveedor de servicios redirige al usuario al <Tooltip tip="Single Sign-On (SSO): Servicio que, después de que un usuario inicia sesión en una aplicación, inicia automáticamente la sesión de ese usuario en otras aplicaciones." cta="Ver glosario" href="/docs/es/glossary?term=identity+provider">proveedor de identidad</Tooltip> (IdP) con fines de autenticación. Este proceso se usa habitualmente en escenarios orientados al consumidor.

Sin embargo, en entornos empresariales, también es habitual que el SSO lo inicie el IdP en lugar del proveedor de servicios. Por ejemplo, una empresa puede configurar un portal para asegurarse de que los usuarios accedan a la aplicación correcta después de iniciar sesión en él.

<div id="risks-and-considerations">
  ## Riesgos y consideraciones
</div>

Los flujos iniciado por IdP conllevan un riesgo de seguridad y, por lo tanto, no se recomiendan. Se recomienda usar flujos SP-Initiated siempre que sea posible.

Asegúrate de comprender los riesgos antes de habilitar iniciado por IdP SSO. En este escenario, Auth0 recibe la respuesta no solicitada del IdP y la aplicación recibe la respuesta no solicitada de Auth0. Ninguna de las dos entidades puede verificar que el usuario haya iniciado el flujo. Por ello, habilitar este flujo abre la posibilidad de sufrir un ataque de [Login CSRF](https://support.detectify.com/support/solutions/articles/48001048951-login-csrf), en el que un atacante puede engañar a un usuario legítimo para que, sin darse cuenta, inicie sesión en la aplicación con la identidad del atacante.

<div id="openid-connect-idp-initiated-flow">
  ### Flujo iniciado por el IdP con OpenID Connect
</div>

<Tooltip tip="OpenID: Estándar abierto de autenticación que permite a las aplicaciones verificar la identidad de los usuarios sin recopilar ni almacenar información de inicio de sesión." cta="Ver glosario" href="/docs/es/glossary?term=OpenID">OpenID</Tooltip> Connect (OIDC) no admite el concepto de un flujo iniciado por IdP. Por eso, aunque Auth0 ofrece la posibilidad de transformar un flujo SAML iniciado por IdP (desde una conexión SAML) en una respuesta OIDC para una aplicación, cualquier aplicación que implemente correctamente el protocolo OIDC/OAuth2 rechazará una respuesta no solicitada.

Al usar aplicaciones OIDC, la mejor opción es que tu aplicación cree un endpoint de login. La única finalidad de este endpoint es iniciar la redirección de vuelta al IdP (tu tenant de Auth0).

Si utilizas varios IdPs, asegúrate de que el endpoint de login sea específico para el proveedor de identidad o de que pueda aceptar un parámetro para identificar qué IdP inicia el flujo de trabajo.

Como alternativa, puedes hacer que los usuarios inicien el flujo de login desde la aplicación.

<div id="post-back-url">
  ### URL de retorno
</div>

Al usar iniciado por IdP SSO, asegúrate de incluir el parámetro de conexión en la URL de retorno:

export const codeExample1 = `https://{yourDomain}/login/callback?connection={yourConnectionName}`;

<AuthCodeBlock children={codeExample1} language="http" />

Si utiliza la funcionalidad [Organizations](/docs/es/manage-users/organizations), opcionalmente puede incluir un parámetro organization que contenga el ID de la organización deseada:

export const codeExample2 = `https://{yourDomain}/login/callback?connection={yourConnectionName}&organization={yourCustomersOrganizationId}`;

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Para que los usuarios puedan iniciar sesión correctamente con este método, la conexión debe estar habilitada para la organización. Además, debes configurar la [membresía automática](/docs/es/manage-users/organizations/configure-organizations/grant-just-in-time-membership) para la conexión habilitada o asegurarte de que los usuarios pertenezcan a la organización.
</Callout>

<div id="lockauth0js">
  ### Lock/Auth0.js
</div>

Si tu aplicación es una aplicación de página única que usa Lock o Auth0.js para procesar los resultados de la autenticación, debes indicar explícitamente que quieres permitir los flujos iniciado por IdP y, por lo tanto, exponer la aplicación a posibles ataques Login CSRF.

Si usas Auth0.js, debes actualizar `webAuth.parseHash` de la biblioteca y establecer el indicador `__enableIdPInitiatedLogin` en `true`.

```javascript lines
var data = webAuth.parseHash(
      {
        ...
        __enableIdPInitiatedLogin: true
        ...
      }
```

Si estás usando Lock, puedes incluir el indicador mediante el parámetro options que se envía al constructor.

`const lock = new Auth0Lock(clientID, domain, options)`

Este es el indicador en sí:

`var options = { _enableIdPInitiatedLogin: true };`

Ten en cuenta que el indicador `enableIdPInitiatedLogin` va precedido de un guion bajo cuando se usa con Lock y de dos guiones bajos cuando se usa con la biblioteca auth0.js.

<div id="set-up-idp-initiated-sso">
  ## Configurar SSO iniciado por IdP
</div>

1. Ve al [Dashboard &gt; Authentication &gt; Enterprise](https://manage.auth0.com/#/connections/enterprise) y elige **SAMLP Identity Provider**.
2. En **Settings** puedes ver la configuración de iniciado por IdP SSO.

   <Frame>![Pantalla de configuración de protocolos de iniciado por IdP SSO](/docs/images/cdy7uua7fh8z/1HORCAp4fZQg0BOoIopvoS/0009d4b5f405c2436af5729d52a3867d/2025-01-28_13-47-26.png)</Frame>

* **Comportamiento de iniciado por IdP SSO**: Esta opción te permite habilitar inicios de sesión iniciados por IdP para la conexión SAML. Selecciona **Accept Requests** y completa todos los campos obligatorios.
* **Aplicación predeterminada:** Cuando el inicio de sesión iniciado por IdP se completa correctamente, esta es la aplicación a la que se redirige a los usuarios. Esta configuración muestra las aplicaciones disponibles habilitadas para esta conexión. Selecciona en la lista desplegable la aplicación con la que quieres que los usuarios inicien sesión mediante IdP. Solo se puede seleccionar una aplicación para un inicio de sesión iniciado por IdP por conexión SAML.
* **Protocolo de respuesta:** Este es el protocolo que se usa para conectar la **Aplicación predeterminada** seleccionada. Lo más habitual es que las aplicaciones se configuren con el protocolo OpenID Connect (consulta arriba). Sin embargo, si has configurado el addon SAML2 Web App para tu aplicación y quieres enrutar la aserción SAML, tendrás que seleccionar SAML. Una vez que se haya enviado una SAML Assertion válida a la URL de retorno, Auth0 envía una respuesta de inicio de sesión a la primera URL de devolución de llamada permitida de la aplicación predeterminada configurada mediante el protocolo de respuesta elegido, que puede modificarse usando el campo query string para especificar un `redirect_uri` si usas OIDC.
  * Si la URL de devolución de llamada configurada para la aplicación incluye un marcador de posición de múltiples dominios personalizados (MCD), el sistema lo rellena dinámicamente con el valor de metadata que corresponde al dominio personalizado de la URL de retorno que recibió la solicitud inicial del IdP. Para más información, consulta [múltiples dominios personalizados](/docs/es/customize/custom-domains/multiple-custom-domains).
* **Query String:** Las opciones de query string ayudan a personalizar el comportamiento cuando se usa el protocolo OpenID Connect. Puedes establecer varias opciones, de forma similar a configurar parámetros con un [query string](https://en.wikipedia.org/wiki/Query_string). Puedes establecer:

| Configuración   | Descripción                                                                                                                                                                                                                                                                                                                                                                                         |
| --------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `redirect_uri`  | Cuando el inicio de sesión iniciado por IdP se haya completado, la solicitud se redirige a la primera URL que aparece en **Allowed Callback URLs** de la aplicación. Sin embargo, si estableces un `redirect_uri`, el IdP redirigirá a esa URL. Esto aporta flexibilidad en casos como cuando tienes un esquema de subdominios con un wildcard y solo quieres redirigir a un subdominio específico. |
| `scope`         | Define scopes para el token de ID enviado. Puedes establecer varios scopes.                                                                                                                                                                                                                                                                                                                         |
| `response_type` | Establece el token para el flujo de concesión implícita para SPA. Puedes establecer code para el flujo de concesión Authorization Code para las aplicaciones web tradicionales.                                                                                                                                                                                                                     |

<Warning>
  En un flujo iniciado por IdP, los servidores de Auth0 eliminan los scopes de un token si la URL de devolución de llamada es un dominio no verificado. Auth0 define solo `localhost` y [127.0.0.1](http://127.0.0.1/) como dominios no verificados. Si usas cualquiera de los dos como URL de devolución de llamada, los tokens del endpoint `/userinfo` devolverán una respuesta vacía. Para obtener una respuesta de token con los scopes solicitados, usa un dominio verificado.
</Warning>

Ejemplo de Query String:
`redirect_uri=https://jwt.io&scope=openid email&response_type=token`

<div id="application-drop-down-list-limited-to-100">
  ## Lista desplegable de aplicaciones limitada a 100
</div>

Si quieres seleccionar una aplicación como la **Aplicación predeterminada** en el SSO iniciado por el IdP y la aplicación no está entre las primeras 100 que aparecen en la lista desplegable de tu tenant, debes usar la <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip> para seleccionar esa aplicación. Debes realizar una solicitud PATCH:

```json lines
{
"options": {
"signInEndpoint": "yourIdpSignInUrl",
"idpinitiated": {
"client_id": "yourClientId",
"client_protocol": "saml",
"client_authorizequery": ""
},
"signingCert": "[copied-from-GET]"
}
}
```
