Muchas instrucciones para configurar una federación de <Tooltip tip="Security Assertion Markup Language (SAML): Protocolo estandarizado que permite a dos partes intercambiar información de autenticación sin contraseña." cta="Ver glosario" href="/docs/es/glossary?term=SAML">SAML</Tooltip> comienzan con el <Tooltip tip="Security Assertion Markup Language (SAML): Protocolo estandarizado que permite a dos partes intercambiar información de autenticación sin contraseña." cta="Ver glosario" href="/docs/es/glossary?term=Single+Sign-on">inicio de sesión único</Tooltip> (SSO) iniciado por el proveedor de servicios. El proveedor de servicios redirige al usuario al <Tooltip tip="Single Sign-On (SSO): Servicio que, después de que un usuario inicia sesión en una aplicación, lo inicia automáticamente en otras aplicaciones." cta="Ver glosario" href="/docs/es/glossary?term=identity+provider">proveedor de identidad</Tooltip> (IdP) con fines de autenticación. Este proceso se usa habitualmente en escenarios orientados al consumidor.

Sin embargo, en escenarios empresariales, también es habitual que el SSO lo inicie el IdP en lugar del proveedor de servicios. Por ejemplo, una empresa puede configurar un portal para asegurarse de que los usuarios accedan a la aplicación correcta después de iniciar sesión en el portal.

<div id="risks-and-considerations">
  ## Riesgos y consideraciones
</div>

Los flujos IdP-Initiated conllevan un riesgo de seguridad y, por lo tanto, no se recomiendan. Se recomienda usar flujos SP-Initiated siempre que sea posible.

Asegúrese de comprender los riesgos antes de habilitar SSO iniciado por IdP. En este escenario, Auth0 recibe la respuesta no solicitada del IdP y la aplicación recibe la respuesta no solicitada de Auth0. Ninguna de las dos entidades puede verificar que el usuario haya iniciado el flujo. Por ello, habilitar este flujo abre la posibilidad de sufrir un [ataque de Login CSRF](https://support.detectify.com/support/solutions/articles/48001048951-login-csrf), en el que un atacante puede engañar a un usuario legítimo para que inicie sesión en la aplicación sin saberlo con la identidad del atacante.

<div id="openid-connect-idp-initiated-flow">
  ### Flujo de OpenID Connect iniciado por IdP
</div>

<Tooltip tip="OpenID: estándar abierto de autenticación que permite a las aplicaciones verificar la identidad de los usuarios sin recopilar ni almacenar información de inicio de sesión." cta="Ver glosario" href="/docs/es/glossary?term=OpenID">OpenID</Tooltip> Connect (OIDC) no admite el concepto de flujo iniciado por IdP. Por lo tanto, aunque Auth0 ofrece la posibilidad de convertir un flujo SAML iniciado por IdP (desde una conexión SAML) en una respuesta OIDC para una aplicación, cualquier aplicación que implemente correctamente el protocolo OIDC/OAuth2 rechazará una respuesta no solicitada.

Al usar aplicaciones OIDC, la mejor opción es que su aplicación cree un endpoint de inicio de sesión. El único propósito de este endpoint es iniciar la redirección de regreso al IdP (su tenant de Auth0).

Si usa varios IdP, asegúrese de que el endpoint de inicio de sesión sea específico para el proveedor de identidad o pueda aceptar un parámetro para identificar qué IdP inicia el workflow.

Como alternativa, puede hacer que los usuarios inicien el flujo de inicio de sesión desde la aplicación.

<div id="post-back-url">
  ### URL de retorno
</div>

Al usar SSO iniciado por IdP, asegúrate de incluir el parámetro de conexión en la URL de retorno:

export const codeExample1 = `https://{yourDomain}/login/callback?connection={yourConnectionName}`;

<AuthCodeBlock children={codeExample1} language="http" />

Si usa la función [Organizations](/docs/es/manage-users/organizations), también puede incluir un parámetro `organization` con el ID de la organización deseada:

export const codeExample2 = `https://{yourDomain}/login/callback?connection={yourConnectionName}&organization={yourCustomersOrganizationId}`;

<AuthCodeBlock children={codeExample2} language="http" />

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Para que los usuarios puedan iniciar sesión correctamente con este método, la conexión debe estar habilitada para la organización. Además, debes configurar la [membresía automática](/docs/es/manage-users/organizations/configure-organizations/grant-just-in-time-membership) para la conexión habilitada o asegurarte de que los usuarios pertenezcan a la organización.
</Callout>

<div id="lockauth0js">
  ### Lock/Auth0.js
</div>

Si su aplicación es una aplicación de página única que usa Lock o Auth0.js para procesar los resultados de la autenticación, debe indicar explícitamente que desea permitir flujos IdP-Initiated y, por lo tanto, exponer la aplicación a posibles ataques de Login CSRF.

Si usa Auth0.js, debe actualizar el método `webAuth.parseHash` de la biblioteca y establecer el indicador `__enableIdPInitiatedLogin` en `true`.

```javascript lines
var data = webAuth.parseHash(
      {
        ...
        __enableIdPInitiatedLogin: true
        ...
      }
```

Si estás usando Lock, puedes incluir el indicador mediante el parámetro options que se envía al constructor.

`const lock = new Auth0Lock(clientID, domain, options)`

Este es el indicador:

`var options = { _enableIdPInitiatedLogin: true };`

Ten en cuenta que el indicador `enableIdPInitiatedLogin` va precedido por un guion bajo cuando se usa con Lock y por dos guiones bajos cuando se usa con la biblioteca auth0.js.

<div id="set-up-idp-initiated-sso">
  ## Configura SSO iniciado por IdP
</div>

1. Ve a [Dashboard &gt; Autenticación &gt; empresarial](https://manage.auth0.com/#/connections/enterprise) y elige **SAMLP Identity Provider**.
2. En **Configuración** puedes ver la configuración de SSO iniciado por IdP.

   <Frame>![Pantalla de configuración de protocolos de SSO iniciado por IdP](/docs/images/cdy7uua7fh8z/1HORCAp4fZQg0BOoIopvoS/0009d4b5f405c2436af5729d52a3867d/2025-01-28_13-47-26.png)</Frame>

* **Comportamiento de SSO iniciado por IdP**: Esta opción te permite habilitar inicios de sesión iniciados por IdP para la conexión SAML. Selecciona **Accept Requests** y completa todos los campos obligatorios.
* **Aplicación predeterminada:** Cuando el inicio de sesión iniciado por IdP se realiza correctamente, esta es la aplicación a la que se redirige a los usuarios. Esta configuración muestra las aplicaciones disponibles habilitadas para esta conexión. Selecciona en la lista desplegable la aplicación con la que quieres que los usuarios inicien sesión mediante IdP. Solo se puede seleccionar una aplicación para un inicio de sesión iniciado por IdP por conexión SAML.
* **Protocolo de respuesta:** Este es el protocolo que se usa para conectar la **Aplicación predeterminada** seleccionada. Lo más habitual es que las aplicaciones estén configuradas con el protocolo OpenID Connect (consulta arriba). Sin embargo, si has configurado un addon SAML2 Web App para tu aplicación y quieres enrutar la aserción SAML, tendrás que seleccionar SAML. Una vez que se haya enviado una aserción SAML válida a la URL de retorno, Auth0 envía una respuesta de inicio de sesión a la primera URL de devolución de llamada permitida de la aplicación predeterminada configurada mediante el protocolo de respuesta elegido, que puede modificarse con el campo de cadena de consulta para especificar un `redirect_uri` si usas OIDC.
  * Si la URL de devolución de llamada configurada para la aplicación incluye un marcador de posición de múltiples dominios personalizados (MCD), el sistema lo completa dinámicamente con el valor de metadata correspondiente al dominio personalizado de la URL de retorno que recibió la solicitud inicial del IdP. Para obtener más información, consulta [múltiples dominios personalizados](/docs/es/customize/custom-domains/multiple-custom-domains).
* **Query String:** Las opciones de la cadena de consulta ayudan a personalizar el comportamiento cuando se usa el protocolo OpenID Connect. Puedes establecer varias opciones, de forma similar a configurar parámetros con una [query string](https://en.wikipedia.org/wiki/Query_string). Puedes establecer:

| Configuración   | Descripción                                                                                                                                                                                                                                                                                                                                                                                      |
| --------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `redirect_uri`  | Cuando se completa el inicio de sesión iniciado por IdP, la solicitud se redirige a la primera URL que aparece en **Allowed Callback URLs** de la aplicación. Sin embargo, si estableces un `redirect_uri`, el IdP redirigirá a esta URL. Esto aporta flexibilidad en casos como cuando tienes un esquema fijo de subdominios con un wildcard y solo quieres redirigir a un subdominio concreto. |
| `scope`         | Define scopes para el token de identidad enviado. Puedes establecer varios scopes.                                                                                                                                                                                                                                                                                                               |
| `response_type` | Establece el token para el flujo de concesión implícita para las SPA. Puedes establecer `code` para el flujo de concesión Authorization Code para aplicaciones web tradicionales.                                                                                                                                                                                                                |

<Warning>
  En un flujo iniciado por IdP, los servidores de Auth0 eliminan los scopes dentro de un token si la URL de devolución de llamada es un dominio no verificado. Auth0 define solo `localhost` y [127.0.0.1](http://127.0.0.1/) como dominios no verificados. Si usas cualquiera de los dos como URL de devolución de llamada, los tokens del endpoint `/userinfo` devolverán una respuesta vacía. Para obtener una respuesta de token con los scopes solicitados, usa un dominio verificado.
</Warning>

Ejemplo de Query String:
`redirect_uri=https://jwt.io&scope=openid email&response_type=token`

<div id="application-drop-down-list-limited-to-100">
  ## Lista desplegable de aplicaciones limitada a 100
</div>

Si quieres seleccionar una aplicación como la **Aplicación predeterminada** en el SSO iniciado por IdP, y la aplicación no se encuentra entre las primeras 100 aplicaciones de la lista desplegable de tu tenant, debes usar la <Tooltip tip="Management API: Un producto para permitir a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip> para seleccionar esa aplicación. Debes ejecutar una solicitud PATCH:

```json lines
{
"options": {
"signInEndpoint": "yourIdpSignInUrl",
"idpinitiated": {
"client_id": "yourClientId",
"client_protocol": "saml",
"client_authorizequery": ""
},
"signingCert": "[copied-from-GET]"
}
}
```
