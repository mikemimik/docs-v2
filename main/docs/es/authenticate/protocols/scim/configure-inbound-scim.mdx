<Card title="Antes de comenzar">
  Tu plan de Auth0 o acuerdo personalizado debe incluir conexiones empresariales para usar esta funcionalidad. Para obtener más información, consulta la [página de precios de Auth0](https://auth0.com/pricing).

  Debes tener [Postman](https://www.postman.com/downloads/) instalado en un equipo local para probar tus conexiones de SCIM entrante.
</Card>

[SCIM](/docs/es/authenticate/protocols/scim) es un estándar de [protocolo](https://tools.ietf.org/html/rfc7644) y [esquema](https://tools.ietf.org/html/rfc7643) que las organizaciones empresariales utilizan para aprovisionar, desaprovisionar y gestionar información relacionada con la identidad de los usuarios.

La funcionalidad SCIM entrante de Auth0 admite integraciones para desarrolladores de SaaS B2B con <Tooltip tip="Proveedor de identidad (IdP): servicio que almacena y gestiona identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=identity+providers">proveedores de identidad</Tooltip> empresariales. Actualmente, se admiten los siguientes tipos de conexión de Auth0: [SAML](/docs/es/authenticate/identity-providers/enterprise-identity-providers/saml), [OpenID Connect](/docs/es/authenticate/identity-providers/enterprise-identity-providers/oidc), [Okta Workforce Identity](/docs/es/authenticate/identity-providers/enterprise-identity-providers/okta) y [Microsoft Azure AD / Entra ID](/docs/es/authenticate/identity-providers/enterprise-identity-providers/azure-active-directory/v2).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  El aprovisionamiento de usuarios de Google Workspace es compatible a través de Directory Sync. Para sincronizar usuarios de Google Workspace con Auth0, consulta [Sincronizar usuarios de Google Workspace con Auth0 mediante Directory Sync](/docs/es/authenticate/identity-providers/enterprise-identity-providers/google-directory-sync).
</Callout>

<div id="enable-inbound-scim-for-an-enterprise-connection">
  ## Habilitar SCIM entrante para una conexión empresarial
</div>

1. Abre el [Auth0 Dashboard](https://manage.auth0.com/#/), y luego elige el tenant que quieres configurar.
2. Ve a [**Autenticación &gt; empresarial**](https://manage.auth0.com/#/connections/enterprise/)**,** y luego selecciona **SAML, OpenID Connect, Okta Workforce** o **Microsoft Azure AD.**
3. Elige una conexión existente o crea una nueva con [Crear conexión](/docs/es/authenticate/database-connections/custom-db/create-db-connection).
4. En la pestaña **Aprovisionamiento** de tu conexión, cambia **Sincronizar atributos del perfil de usuario en cada inicio de sesión** a **Desactivado** y **Sincronizar usuarios y grupos con SCIM** a **Activado.**
5. Selecciona la pestaña **Setup** en **Sincronizar usuarios y grupos con SCIM** para obtener la URL del endpoint de SCIM y los tokens de SCIM necesarios para hacer pruebas con Postman.

<div id="test-with-postman">
  ### Probar con Postman
</div>

Puedes descargar la colección de Postman a continuación para probar tu configuración de SCIM:

* [Colección de Postman de SCIM 2.0](https://cdn.auth0.com/scim/collections/scim_postman_collection_v2.json)

1. Inicia Postman, selecciona **File &gt; Import…** y arrastra el archivo descargado al cuadro de diálogo de importación.
2. Selecciona la colección **SCIM 2.0 Tests** y luego elige la pestaña **Variables**.
3. Copia el valor de **URL del endpoint de SCIM** desde el Auth0 Dashboard y pégalo en el campo **Current Value** junto a la variable **SCIM-ENDPOINT-URL**.
4. En el Auth0 Dashboard, elige **Generate New Token**, luego selecciona **Generate Token** y **Copy and Close.**
5. Vuelve a Postman, selecciona la pestaña **Authorization** y pega el valor del token en el campo **Token**.
6. Elige **Save.**
7. Ejecuta las pruebas en el orden indicado. Ten en cuenta que el comando **Users - POST** debe ejecutarse antes que las pruebas de grupos para garantizar que exista un usuario válido. El resultado de cada operación de usuario y grupo puede verse en el Auth0 Dashboard, en [**User Management &gt; Users**](https://manage.auth0.com/#/users)  y [**User Management &gt; Enterprise Groups**](https://manage.auth0.com/#/enterprise-groups). Los detalles de cada operación de SCIM pueden verse en los logs del tenant, en [**Monitoreo &gt; Logs**](https://manage.auth0.com/#/logs).

Para probar con proveedores de identidad específicos, consulta [configurar SCIM entrante para proveedores de identidad que usan SAML u OpenID](/docs/es/authenticate/protocols/scim/configure-inbound-scim-for-identity-providers-using-saml-or-openid).

<div id="supported-scim-operations">
  ## Operaciones SCIM admitidas
</div>

Auth0 admite las siguientes operaciones de SCIM 2.0 para administrar usuarios y grupos:

| Recurso | Comando              | Descripción                                                                                                                                                                                                              |
| ------- | -------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Usuario | `POST`               | Crea un usuario en Auth0 con atributos del esquema principal de SCIM 2.0 y de la extensión del esquema empresarial, como se describe en [RFC7644 Section 3.3](https://tools.ietf.org/html/rfc7644#section-3.3).          |
| Usuario | `GET`                | Recupera un usuario ya creado en Auth0 mediante su valor de ID de usuario en Auth0, como se describe en [RFC7644 Section 3.4.1](https://tools.ietf.org/html/rfc7644#section-3.4.1).                                      |
| Usuario | `PUT`                | Reemplaza un usuario en Auth0 con atributos del esquema principal de SCIM 2.0 y de la extensión del esquema empresarial, como se describe en [RFC7644 Section 3.5.1](https://tools.ietf.org/html/rfc7644#section-3.5.1). |
| Usuario | `PATCH`              | Actualiza un usuario en Auth0 con atributos del esquema principal de SCIM 2.0 y de la extensión del esquema empresarial, como se describe en [RFC7644 Section 3.5.2](https://tools.ietf.org/html/rfc7644#section-3.5.2). |
| Usuario | `PUT` (Desactivar)   | Reemplaza un usuario almacenado en Auth0 y establece el atributo SCIM active en false, lo que hace que el usuario quede bloqueado en Auth0.                                                                              |
| Usuario | `PATCH` (Desactivar) | Actualiza un usuario almacenado en Auth0 y establece el atributo SCIM active en false, lo que hace que el usuario quede bloqueado en Auth0.                                                                              |
| Usuario | `SEARCH`             | Busca un usuario en Auth0 mediante un filtro SCIM, como se describe en [RFC7644 Section 3.4.2](https://tools.ietf.org/html/rfc7644#section-3.4.2). Los operadores admitidos son Equals (EQ), and (AND) y or (OR).        |
| Usuario | `DELETE`             | Elimina un usuario en Auth0 mediante su valor de ID de usuario en Auth0, como se describe en [RFC7644 Section 3.6](https://tools.ietf.org/html/rfc7644#section-3.6).                                                     |
| Grupo   | `POST`               | Crea un grupo en Auth0 con atributos del esquema principal de SCIM 2.0, como se describe en [RFC7644 Section 3.3](https://tools.ietf.org/html/rfc7644#section-3.3).                                                      |
| Grupo   | `GET`                | Recupera un grupo ya creado en Auth0 mediante su valor de ID de grupo en Auth0, como se describe en [RFC7644 Section 3.4.1](https://tools.ietf.org/html/rfc7644#section-3.4.1).                                          |
| Grupo   | `PUT`                | Reemplaza un grupo en Auth0 con atributos del esquema principal de SCIM 2.0, como se describe en [RFC7644 Section 3.5.1](https://tools.ietf.org/html/rfc7644#section-3.5.1).                                             |
| Grupo   | `PATCH`              | Actualiza un grupo en Auth0 con atributos del esquema principal de SCIM 2.0, como se describe en [RFC7644 Section 3.5.2](https://tools.ietf.org/html/rfc7644#section-3.5.2).                                             |
| Grupo   | `SEARCH`             | Busca un grupo en Auth0 mediante un filtro SCIM, como se describe en [RFC7644 Section 3.4.2](https://tools.ietf.org/html/rfc7644#section-3.4.2). Admite el operador Equals (EQ).                                         |
| Grupo   | `DELETE`             | Elimina un grupo en Auth0 mediante el valor de ID del grupo, como se describe en [RFC7644 Section 3.6](https://tools.ietf.org/html/rfc7644#section-3.6).                                                                 |

Auth0 admite el [esquema principal](https://tools.ietf.org/html/rfc7643) y el [esquema empresarial](https://tools.ietf.org/html/rfc7643) de SCIM 2.0 para los recursos de usuario.

Auth0 admite el [esquema principal](https://tools.ietf.org/html/rfc7643) de SCIM 2.0 para los recursos de grupo; sin embargo, solo se admiten tipos de miembro “user”. Los valores de “displayName” de los grupos deben ser únicos por conexión, tal como se requiere para garantizar la compatibilidad con proveedores de identidad empresariales como Microsoft Entra ID.

<div id="scim-endpoints-and-tokens">
  ### Endpoints de SCIM y tokens
</div>

Cada cliente empresarial obtiene un endpoint de SCIM y un token específicos de la conexión que le permiten aprovisionar, desaprovisionar y administrar de forma segura los usuarios y grupos de esa conexión almacenados en el tenant de Auth0.

Los endpoints y tokens son visibles y configurables en el <Tooltip tip="Auth0 Dashboard: el principal producto de Auth0 para configurar sus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip>, en **Authentication &gt; empresarial &gt; [connecton-type] &gt; [your-connection] &gt; aprovisionamiento &gt;  sincronizar usuarios y grupos con SCIM &gt;  Setup**.

<Frame>![Captura de pantalla del Dashboard con la configuración de SCIM](/docs/images/cdy7uua7fh8z/4jsLx8sZiy9WRhqaV5uCqt/09e0bc4d690742b16b8535f07c3a64a9/setup.png)</Frame>

La **URL del endpoint de SCIM** permite a los clientes de SCIM administrar los usuarios y grupos almacenados en Auth0 para un proveedor de identidad determinado.

La opción **Generate New Token** genera hasta dos tokens activos para este endpoint de SCIM, lo que permite actualizar el token que usa el cliente de SCIM sin tiempo de inactividad. Los tokens también pueden revocarse en esta pantalla seleccionando **Delete**.

<Frame>![Pantalla del Dashboard para la configuración del token de SCIM](/docs/images/cdy7uua7fh8z/5TUDyHPvAKFffK76gxUHY4/5ab3e28268afa11abb31c679e2e64bcc/Tokens.jpg)</Frame>

Están disponibles las siguientes opciones de generación de tokens:

* **Sin fecha de expiración:** Seleccione si el token expira o no.
* **Fecha de expiración en segundos**: si el token no tiene fecha de expiración, puede elegir un tiempo de expiración para el token. Cuando el token expire, el endpoint de SCIM devolverá una respuesta de error la próxima vez que se use. El tiempo mínimo de expiración permitido es de 900 segundos.
* **Lista de permisos (scopes):** detalla qué operaciones de SCIM pueden realizarse con este token. Los permisos disponibles son:

  * **get:users** - Permite recuperar y buscar usuarios.
  * **post:users** - Permite crear usuarios
  * **put:users** - Permite actualizar usuarios mediante el método PUT.
  * **patch:users** - Permite actualizar usuarios mediante el método PATCH.
  * **delete:users** - Permite eliminar usuarios.
  * **get:groups** - Permite recuperar y buscar grupos.
  * **post:groups** - Permite crear grupos
  * **put:groups** - Permite actualizar grupos mediante el método PUT.
  * **patch:groups** - Permite actualizar grupos mediante el método PATCH.
  * **delete:groups** - Permite eliminar grupos.

<div id="session-revocation-and-backchannel-logout">
  ### Revocación de sesiones y cierre de sesión por back-channel
</div>

Cuando Auth0 recibe un mensaje SCIM para desactivar y bloquear a un usuario, termina todas las sesiones de Auth0 de ese usuario, revoca los <Tooltip tip="Token de actualización: token usado para obtener un token de acceso renovado sin obligar a los usuarios a volver a iniciar sesión." cta="Ver glosario" href="/docs/es/glossary?term=refresh+tokens">tokens de actualización</Tooltip> y, si está configurado, inicia el [cierre de sesión por back-channel de OpenID Connect](/docs/es/authenticate/login/logout/back-channel-logout/configure-back-channel-logout) para tus aplicaciones.

<div id="logs">
  ### Logs
</div>

Puedes ver los detalles completos de todas las solicitudes SCIM que recibe Auth0 en la sección [**Monitoreo &gt; Logs**](https://manage.auth0.com/#/logs) de Auth0 Dashboard. Ten en cuenta que cualquier operación POST o PUT para un grupo con más de 1.000 miembros en el cuerpo de la solicitud no mostrará el atributo members en los logs debido a las limitaciones de tamaño.

Para obtener información sobre cómo sincronizar desde Auth0 hacia sistemas externos usuarios, grupos y membresías de grupo aprovisionados mediante SCIM, consulta [Sync Users and Groups to External Systems](/docs/es/authenticate/protocols/scim/configure-inbound-scim#sync-users-and-groups-to-external-systems).

<div id="user-provisioning-options">
  ## Opciones de aprovisionamiento de usuarios
</div>

Consulta las siguientes secciones para obtener orientación sobre cómo configurar el mapeo de atributos entre el esquema SCIM y los perfiles de usuario de Auth0, y sobre cómo los usuarios aprovisionados mediante SCIM pueden usarse con otras funcionalidades de Auth0.

<div id="user-attribute-mapping">
  ### Mapeo de atributos de usuario
</div>

Cada nueva conexión usa el mapa de atributos predeterminado que se encuentra en el Auth0 Dashboard en **Authentication &gt; empresarial &gt; [connecton-type] &gt; [your-connection] &gt; aprovisionamiento &gt;  sincronizar usuarios y grupos con SCIM &gt;  Mapping,** donde el mapa puede editarse y personalizarse según las necesidades de su conexión.

Se pueden usar los atributos principales de usuario de SCIM definidos en [las secciones 3.1, 4.1 y 4.3 de la RFC 7643](https://tools.ietf.org/html/rfc7643).

| Notas                     |                                                                                                                                                                                                                                                                                         |
| ------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Mapeo uno a uno           | Un atributo SCIM seleccionado (o subatributo) solo puede mapearse a un atributo de Auth0.                                                                                                                                                                                               |
| Mapeo de subatributos     | Se puede hacer referencia a valores de subatributos dentro de atributos SCIM multivalor, como emails y phoneNumbers, mediante la sintaxis de filtros de SCIM con el operador eq. Consulte los mapas de atributos predeterminados para ver ejemplos.                                     |
| Atributos no mapeables    | Los atributos id y meta de SCIM no pueden mapearse, ya que Auth0 solo los envía en las respuestas del protocolo SCIM. El valor id en las respuestas SCIM siempre se establece en el user&#95;id de Auth0, y el atributo password de SCIM no puede usarse para conexiones empresariales. |
| Omisión de atributos SCIM | Si un atributo SCIM específico no está configurado para gestionarse en su mapa de atributos, se ignorará en todas las solicitudes y respuestas SCIM.                                                                                                                                    |

Estos atributos SCIM pueden mapearse a atributos de [raíz](/docs/es/manage-users/user-accounts/user-profiles/root-attributes) y [metadata](/docs/es/manage-users/user-accounts/metadata/metadata-fields-data) en el perfil de usuario de Auth0.

| Notas                                 |                                                                                                                                                                                                                                                                                                                                                                                                                                                |
| ------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Mapeo uno a uno                       | Un atributo de Auth0 seleccionado solo puede mapearse a un atributo SCIM.                                                                                                                                                                                                                                                                                                                                                                      |
| Mapeo de atributos de raíz            | Al mapear a atributos de [raíz](/docs/es/manage-users/user-accounts/user-profiles/root-attributes), tenga en cuenta que solo los atributos de raíz enumerados en [Atributos del perfil de usuario](/docs/es/manage-users/user-accounts/user-profiles/user-profile-structure#user-profile-attributes) pueden buscarse mediante una consulta SCIM. Si necesita que un atributo se pueda buscar y no está en esta lista, colóquelo en app&#95;metadata. |
| Mapeo de atributos de metadata        | Se permite usar user&#95;metadata, pero por lo general no se recomienda para atributos sincronizados, ya que está pensado para almacenar atributos que el usuario final puede editar directamente. En su lugar, use app&#95;metadata o atributos de raíz.                                                                                                                                                                                      |
| Comportamiento del atributo `blocked` | El mapeo al atributo `blocked` de Auth0 tiene un comportamiento especial cuando se mapea al atributo active de SCIM. Cuando active contiene un valor de true o false, Auth0 invierte ese valor y establece el atributo `blocked` de Auth0 en false o true, respectivamente.                                                                                                                                                                    |

Auth0 también le permite aprovisionar atributos mediante SCIM en paralelo con los atributos aprovisionados durante el inicio de sesión. Para obtener más información, consulte la siguiente sección.

<div id="sync-additional-attributes-at-login">
  ### Sincronice atributos adicionales al iniciar sesión
</div>

Si está integrando un proveedor de identidad que no admite el mismo conjunto de atributos de usuario en <Tooltip tip="Security Assertion Markup Language (SAML): Protocolo estandarizado que permite a dos partes intercambiar información de autenticación sin contraseña." cta="Ver glosario" href="/docs/es/glossary?term=SAML">SAML</Tooltip>/OIDC y SCIM, puede acceder a los atributos adicionales habilitando **Sincronizar atributos del perfil de usuario en cada inicio de sesión** junto con el aprovisionamiento de SCIM.

Cuando **Sincronizar atributos del perfil de usuario en cada inicio de sesión** está habilitado, todos los atributos [raíz](/docs/es/manage-users/user-accounts/user-profiles/root-attributes) del perfil de usuario se sobrescriben cada vez que un usuario inicia sesión. Para evitar posibles conflictos entre SCIM y la sincronización al iniciar sesión, siga estas pautas:

* Asegúrese de que los atributos raíz comunes de Auth0, como `email` y `username`, estén mapeados en el mapeo de atributos de [SAML](/docs/es/authenticate/protocols/saml/saml-configuration) o [OpenID Connect](/docs/es/authenticate/identity-providers/enterprise-identity-providers/configure-pkce-claim-mapping-for-oidc) si también están presentes en el mapeo de atributos de SCIM.
* En el mapeo de atributos de SCIM, mapee todos los demás atributos de SCIM (excepto el atributo `active`) a valores dentro del atributo `app_metadata` de Auth0.

<div id="sync-roles-as-a-user-attribute">
  ### Sincronizar roles como atributo de usuario
</div>

Auth0 admite sincronizar el atributo de usuario `roles` de SCIM 2.0, tal como se define en [RFC7643](https://datatracker.ietf.org/doc/html/rfc7643). Para aceptar roles, asegúrate de que el mapeo de atributos de SCIM de tu conexión asigne el atributo `roles` de SCIM a un atributo de usuario de Auth0 como `app_metadata.roles`.

Para saber cómo sincronizar roles específicos de la aplicación desde Okta Workforce Identity, consulta [How to Add Multi-value Roles in SCIM Cloud Integration](https://support.okta.com/help/s/article/How-to-add-multivalue-roles-in-SCIM-Cloud-integration). Para los roles de Microsoft Entra ID, consulta [Customize user provisioning attribute-mappings for SaaS applications in Microsoft Entra ID](https://learn.microsoft.com/en-us/entra/identity/app-provisioning/customize-application-attributes#provisioning-a-role-to-a-scim-app).

Para saber cómo asignar roles de Auth0 a grupos aprovisionados mediante SCIM, consulta [Opciones de aprovisionamiento de grupos](/docs/es/authenticate/protocols/scim/configure-inbound-scim#group-provisioning-options).

<div id="organization-membership">
  ### Membresía en la organización
</div>

Para que los usuarios aprovisionados mediante SCIM pasen a ser miembros de una organización, la conexión debe configurarse para **Habilitar membresía automática**, como se describe en [Conceder membresía Just-In-Time a una conexión de organización](/docs/es/manage-users/organizations/configure-organizations/grant-just-in-time-membership).

<div id="user-account-linking">
  ### Vinculación de cuentas de usuario
</div>

Al usar la [vinculación de cuentas de usuario](/docs/es/manage-users/user-accounts/user-account-linking), la cuenta de usuario aprovisionada mediante SCIM debe configurarse como la cuenta de usuario principal. Si se configura como cuenta secundaria, cambiará el atributo `id` de SCIM, lo que contraviene la [especificación básica del esquema de SCIM 2.0](https://datatracker.ietf.org/doc/html/rfc7643#section-3.1). No se recomienda vincular cuentas de usuario empresariales con cuentas de usuario sociales y personales.

<div id="group-provisioning-options">
  ## Opciones de aprovisionamiento de grupos
</div>

Consulta las secciones siguientes para saber cómo usar grupos aprovisionados mediante SCIM para la autorización y cómo pueden usarse con Auth0 Organizations.

<div id="use-groups-with-roles">
  ### Usar grupos con roles
</div>

Los grupos sincronizados desde proveedores de identidad empresariales se pueden asignar a [roles principales de autorización](/docs/es/manage-users/access-control/configure-core-rbac) en Auth0. Una vez asignados, todos los miembros del grupo heredan el rol asignado al iniciar sesión. Para obtener más información, consulta [Asignar roles para grupos empresariales](/docs/es/manage-users/access-control/configure-core-rbac/rbac-users/assign-roles-to-groups).

Ten en cuenta los límites de entidades a nivel del tenant para grupos empresariales y roles principales de autorización en la [Política de límite de entidades](/docs/es/troubleshoot/customer-support/operational-policies/entity-limit-policy#entity-limit-policy).

<div id="use-groups-with-organizational-roles">
  ### Usar grupos con roles de organización
</div>

De forma similar a lo anterior, los grupos sincronizados desde proveedores de identidad empresariales pueden [asignarse a roles dentro de una organización](/docs/es/manage-users/organizations/organizations-overview). Cuando un miembro de la organización inicia sesión, hereda cualquier rol de organización asignado a los grupos sincronizados de los que forma parte.

Si busca representar clientes o partners en una aplicación B2B o SaaS, le recomendamos representarlos como [Organizations](/docs/es/manage-users/organizations/organizations-overview) y usar grupos y roles en este contexto.

Para que los usuarios aprovisionados mediante SCIM se conviertan en miembros de una organización, primero se debe configurar la conexión para **habilitar la membresía automática**, como se describe en [Conceder membresía Just-In-Time a una conexión de organización](/docs/es/manage-users/organizations/configure-organizations/grant-just-in-time-membership).

Luego, se pueden asignar roles de organización a cualquier grupo sincronizado desde las conexiones empresariales asociadas con la organización. Para obtener más información, consulte [Asignar roles a grupos empresariales](/docs/es/manage-users/access-control/configure-core-rbac/rbac-users/assign-roles-to-groups).

Tenga en cuenta los límites de entidad a nivel de organización para los grupos empresariales y los roles principales de autorización en la [Política de límites de entidades](/docs/es/troubleshoot/customer-support/operational-policies/entity-limit-policy#entity-limit-policy).

<div id="use-groups-in-a-post-login-action">
  ### Usa grupos en una Action de Post Login
</div>

Puedes usar información de grupos procedente de proveedores de identidad empresariales en tus Actions de Post Login para tomar decisiones personalizadas de control de acceso y autorización en Auth0. Para obtener más información, consulta la documentación de las funciones `api.groups.getUserGroups()` y `api.groups.hasGroupMembership()` en [Actions Triggers: Post Login - objeto de API](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger/post-login-api-object).

<div id="sync-users-and-groups-to-external-systems">
  ## Sincronizar usuarios y grupos con sistemas externos
</div>

Usa la funcionalidad [Events](/docs/es/customize/events) de Auth0 para sincronizar usuarios, grupos y membresías de grupos desde Auth0 hacia sistemas externos. Esto te permite enviar notificaciones en tiempo real sobre cambios en usuarios y grupos a un endpoint de webhook personalizado, a AWS EventBridge o a una Auth0 Action personalizada.

Para configurar la sincronización de salida de usuarios y grupos:

1. Sigue las instrucciones de [Create an Event Stream](/docs/es/customize/events/create-an-event-stream) para crear un flujo de eventos en el Auth0 Dashboard. Elige un endpoint de webhook personalizado, AWS EventBridge o una Auth0 Action personalizada como destino de los eventos.

2. En la página de configuración de tu flujo de eventos, selecciona los eventos de Users y Groups que quieras transmitir. En la mayoría de los casos, se recomienda seleccionar todos los eventos de User y Group. Para ver la información incluida en cada evento de User y Group, consulta el [Events Catalog](/docs/es/events).

3. Después de configurar tu flujo de eventos y el destino deseado, puedes generar eventos de usuario y grupo para hacer pruebas usando la colección de Postman proporcionada anteriormente, o bien usando un proveedor de identidad empresarial para crear, actualizar o eliminar usuarios y grupos mediante SCIM.

<div id="deployment-guidelines">
  ## Pautas de despliegue
</div>

<div id="leverage-integration-galleries-for-streamlined-setup">
  ### Aprovecha las galerías de integraciones para simplificar la configuración
</div>

Para ofrecer a tus clientes una experiencia personalizada al configurar tanto SCIM como <Tooltip tip="Inicio de sesión único (SSO): servicio que, después de que un usuario inicia sesión en una aplicación, lo inicia automáticamente en otras aplicaciones." cta="Ver glosario" href="/docs/es/glossary?term=SSO">SSO</Tooltip>, considera incluir tu aplicación en la [Okta Integration Network](https://www.okta.com/integrations/) y en otras galerías de integraciones de proveedores de identidad para la fuerza laboral con las que planeas integrarte.

Actualmente, Auth0 es compatible con [Express Configuration with Okta](/docs/es/authenticate/identity-providers/enterprise-identity-providers/okta/express-configuration), que automatiza la configuración de OpenID Connect, SCIM y Universal Logout para las aplicaciones publicadas en la [Okta Integration Network](https://www.okta.com/integrations/).

<div id="use-auth0-self-service-sso-and-user-provisioning">
  ### Usa Auth0 Self-Service SSO y el aprovisionamiento de usuarios
</div>

Si no utilizas una galería de integraciones, usa [Auth0 Self-Service Enterprise Configuration](/docs/es/authenticate/enterprise-connections/self-service-enterprise-configuration) para permitir que tus clientes o los administradores del IDP configuren SCIM directamente.

<div id="test-scim-in-lower-level-environments-first">
  ### Prueba primero SCIM en entornos inferiores
</div>

No habilites SCIM en un tenant de Auth0 de producción sin antes probar exhaustivamente tu integración en un tenant de Auth0 de desarrollo o de staging.

<div id="send-scim-tokens-securely">
  ### Envía los tokens SCIM de forma segura
</div>

Nunca envíes tokens SCIM en texto sin formato, por correo electrónico ni por canales inseguros. Usa un servicio de comunicación seguro como [SendSafely](https://www.sendsafely.com/) o [Auth0 Self-Service Enterprise Configuration](/docs/es/authenticate/enterprise-connections/self-service-enterprise-configuration) para emitir tokens SCIM directamente a tus clientes.

Al usar [Express Configuration with Okta](/docs/es/authenticate/identity-providers/enterprise-identity-providers/okta/express-configuration), la configuración del token se automatiza y no se muestra.

<div id="management-api">
  ## Management API
</div>

La configuración de SCIM para una conexión empresarial se puede automatizar mediante la Auth0 Management API. Para obtener más información, lee [Gestiona una implementación de SCIM entrante con la Management API](/docs/es/authenticate/protocols/scim/manage-an-inbound-scim-deployment-with-the-management-api).

Todos los usuarios y grupos empresariales creados en tu tenant de Auth0 mediante SCIM también se pueden buscar, consultar o eliminar a través de la Auth0 Management API. Los siguientes endpoints de la Auth0 Management API están disponibles:

* [GET /api/v2/users](https://auth0.com/docs/api/management/v2/users/get-users)
* [GET /api/v2/users/:id](https://auth0.com/docs/api/management/v2/users/get-users-by-id)
* [GET /api/v2/users/:id/groups](https://auth0.com/docs/api/management/v2/users/get-user-groups)
* [GET /api/v2/groups](https://auth0.com/docs/api/management/v2/groups/get-groups)
* [GET /api/v2/groups/:id](https://auth0.com/docs/api/management/v2/groups/get-group)
* [GET /api/v2/groups/:id/members](https://auth0.com/docs/api/management/v2/groups/get-group-members)
* [DELETE /api/v2/users/:id](https://auth0.com/docs/api/management/v2/users/delete-users-by-id)
* [DELETE /api/v2/groups/:id](https://auth0.com/docs/api/management/v2/groups/delete-group)
