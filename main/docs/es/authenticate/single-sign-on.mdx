title: Inicio de sesión único
description: Descubra qué es el inicio de sesión único (SSO) y cómo funciona.
---

<Tooltip tip="Inicio de sesión único (SSO): servicio que, después de que un usuario inicia sesión en una aplicación, le inicia sesión automáticamente en otras aplicaciones." cta="Ver glosario" href="/docs/es/glossary?term=Single+Sign-on">Inicio de sesión único</Tooltip> (SSO) se produce cuando un usuario inicia sesión en una aplicación y, a continuación, queda conectado automáticamente a otras aplicaciones, independientemente de la plataforma, la tecnología o el dominio que utilice. El usuario inicia sesión una sola vez; de ahí el nombre de esta funcionalidad (inicio de sesión único).

Por ejemplo, si inicia sesión en un servicio de Google como Gmail, quedará autenticado automáticamente en YouTube, AdSense, Google Analytics y otras aplicaciones de Google. Del mismo modo, si cierra sesión en Gmail o en otras aplicaciones de Google, la sesión se cerrará automáticamente en todas las aplicaciones; esto se conoce como cierre de sesión único.

SSO ofrece una experiencia fluida a los usuarios al utilizar sus aplicaciones y servicios. En lugar de tener que recordar distintos conjuntos de credenciales para cada aplicación o servicio, los usuarios solo tienen que iniciar sesión una vez para acceder a todo su conjunto de aplicaciones.

Siempre que los usuarios accedan a un dominio que requiera autenticación, se les redirigirá al dominio de autenticación, donde puede que se les pida iniciar sesión. Si el usuario ya ha iniciado sesión en el dominio de autenticación, se le puede redirigir inmediatamente al dominio original sin tener que volver a iniciar sesión.

<div id="how-it-works">
  ## Cómo funciona
</div>

El inicio de sesión único y el cierre de sesión único son posibles mediante el uso de [sesiones](/docs/es/manage-users/sessions). Puede haber hasta tres sesiones diferentes para un usuario con SSO:

* Sesión local mantenida por la aplicación
* Sesión del <Tooltip tip="Servidor de autorización: Servidor centralizado que contribuye a definir los límites del acceso de un usuario. Por ejemplo, tu servidor de autorización puede controlar los datos, las tareas y las funciones disponibles para un usuario." cta="Ver glosario" href="/docs/es/glossary?term=Authorization+Server">Servidor de autorización</Tooltip>, si SSO está habilitado
* Sesión del <Tooltip tip="Proveedor de identidad (IdP): Servicio que almacena y administra identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=Identity+Provider+%28IdP%29">Proveedor de identidad (IdP)</Tooltip>, si el usuario eligió iniciar sesión a través de un <Tooltip tip="Proveedor de identidad (IdP): Servicio que almacena y administra identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=Identity+Provider">Proveedor de identidad</Tooltip> (como Google, Facebook o un Proveedor de identidad <Tooltip tip="Security Assertion Markup Language (SAML): Protocolo estandarizado que permite a dos partes intercambiar información de autenticación sin una contraseña." cta="Ver glosario" href="/docs/es/glossary?term=SAML">SAML</Tooltip> empresarial)

Con SSO, un dominio central realiza la autenticación y luego comparte la sesión con otros dominios. La forma de compartir una sesión puede variar según el protocolo de SSO, pero el concepto general es el mismo.

Por ejemplo, el dominio de autenticación puede generar un [JSON Web Token (JWT)](/docs/es/secure/tokens/json-web-tokens) firmado (cifrado mediante JSON Web Encryption (JWE)), que contiene toda la información necesaria para identificar al usuario en cualquier otro dominio que requiera autenticación. Este token se transmite al cliente, pero, como está firmado, el cliente no puede modificarlo de ninguna manera. El token puede volver a enviarse al dominio original mediante una redirección y ser utilizado por el dominio de autenticación y cualquier otro dominio para identificar al usuario.

<div id="sso-with-universal-login">
  ## SSO con Universal Login
</div>

La forma más fácil y segura de implementar el inicio de sesión único (SSO) con Auth0 es usar [Universal Login](/docs/es/authenticate/login/auth0-universal-login) para la autenticación. De hecho, actualmente el SSO solo es posible en plataformas nativas (como iOS o Android) si la aplicación usa <Tooltip tip="Universal Login: Su aplicación redirige a Universal Login, alojado en el servidor de autorización de Auth0, para verificar la identidad de un usuario." cta="Ver glosario" href="/docs/es/glossary?term=Universal+Login">Universal Login</Tooltip>. Las guías de inicio rápido de [Swift](/docs/es/quickstart/native/ios-swift) y [Android](/docs/es/quickstart/native/ios-swift) ofrecen algunos ejemplos de uso de Universal Login.

Si no puede usar Universal Login con su aplicación, consulte lo siguiente para obtener más información sobre la autenticación integrada:

* [Lock](/docs/es/libraries/lock/lock-api-reference)
* [Auth0.js](/docs/es/libraries/auth0js)

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Si usa [Passwordless](/docs/es/authenticate/passwordless/passwordless-with-universal-login) con inicio de sesión único, los parámetros de conexión `sms` y `email` no usan la sesión existente de Auth0, y se le pedirá al usuario que inicie sesión.
</Callout>

<div id="sso-on-first-login">
  ### SSO en el primer inicio de sesión
</div>

Para SSO con Auth0, el **servicio central** es el <Tooltip tip="Servidor de autorización: Servidor centralizado que ayuda a definir los límites del acceso de un usuario. Por ejemplo, su servidor de autorización puede controlar los datos, las tareas y las funciones disponibles para un usuario." cta="Ver glosario" href="/docs/es/glossary?term=Authorization+Server">servidor de autorización</Tooltip> de Auth0.

Veamos un ejemplo del flujo de SSO cuando un usuario inicia sesión por primera vez:

1. Su aplicación redirige al usuario a la página de inicio de sesión.
2. Auth0 comprueba si ya existe una cookie de SSO.
3. Como es la primera vez que el usuario visita la página de inicio de sesión y no hay ninguna cookie de SSO, se le pedirá que inicie sesión con una de las conexiones que haya configurado.

   <Frame>![Pantalla de inicio de sesión de la aplicación de hojas de hora de ejemplo](/docs/images/cdy7uua7fh8z/6m01sxT4xI0oUC6ox3vb4Z/ca72f1208d952a9a43f24f425d360e48/Screenshot_2024-07-11_at_11.57.33.png)</Frame>
4. Una vez que el usuario haya iniciado sesión, Auth0 establecerá una cookie de SSO y lo redirigirá a su aplicación, devolviendo un token de ID que contiene información de identidad del usuario.

<div id="sso-on-subsequent-logins">
  ### SSO en inicios de sesión posteriores
</div>

Veamos un ejemplo del flujo de SSO cuando un usuario vuelve a tu sitio web en una visita posterior:

1. Tu aplicación redirige al usuario a la página de inicio de sesión.
2. Auth0 comprueba si ya existe una cookie de SSO.
3. Auth0 encuentra la cookie de SSO y, si es necesario, la actualiza. No se muestra la pantalla de inicio de sesión.
4. Auth0 redirige al usuario a tu aplicación y devuelve un token de ID que contiene información de identidad del usuario.

<div id="check-users-sso-status">
  ### Comprueba el estado de SSO del usuario
</div>

Puedes comprobar el estado de SSO de un usuario desde una aplicación llamando al método `checkSession` del SDK `auth0.js`, que intentará [autenticar silenciosamente](/docs/es/authenticate/login/configure-silent-authentication) al usuario dentro de un iframe. Que la autenticación se complete correctamente o no indica si el usuario tiene una cookie de SSO activa.

<div id="protocols">
  ## Protocolos
</div>

<div id="saml-and-ws-federation">
  ### SAML y WS-Federation
</div>

Security Assertion Markup Language (SAML) y Web Services Federation (<Tooltip tip="Web Service Federation (WS-Fed): protocolo para gestionar identidades de usuario entre dominios." cta="Ver glosario" href="/docs/es/glossary?term=WS-Fed">WS-Fed</Tooltip>) son [protocolos](/docs/es/authenticate/protocols) ampliamente utilizados en implementaciones de SSO. Tanto SAML como WS-Fed intercambian datos de autorización y autenticación en formato XML; los principales elementos de este intercambio son el usuario, el proveedor de identidad y el proveedor de servicios.

Con SAML o WS-Fed:

1. Un usuario solicita un recurso al proveedor de servicios.
2. El proveedor de servicios consulta al proveedor de identidad para determinar si el usuario debe tener acceso al recurso.
3. El proveedor de identidad verifica la identidad del usuario y, si es válida, le indica al proveedor de servicios que el usuario debe tener acceso.

<div id="openid-connect">
  ### OpenID Connect
</div>

<Tooltip tip="OpenID: Estándar abierto de autenticación que permite a las aplicaciones verificar la identidad de los usuarios sin recopilar ni almacenar información de inicio de sesión." cta="Ver glosario" href="/docs/es/glossary?term=OpenID">OpenID</Tooltip> Connect (OIDC) es un protocolo de autenticación que se usa habitualmente en implementaciones de SSO dirigidas al consumidor. El protocolo OIDC gestiona la autenticación mediante <Tooltip tip="JSON Web Token (JWT): Formato estándar de token de ID (y, a menudo, también de token de acceso) que se utiliza para representar claims de forma segura entre dos partes." cta="Ver glosario" href="/docs/es/glossary?term=JSON+Web+Tokens">JSON Web Tokens</Tooltip> y un proveedor de identidad central.

Con OIDC:

1. Un usuario solicita acceso a una aplicación.
2. La aplicación redirige al usuario al proveedor de identidad para autenticarse.
3. El proveedor de identidad verifica al usuario y, si la verificación es correcta, le solicita que conceda a la aplicación acceso a sus datos.
4. Si se concede el acceso, el proveedor de identidad genera un token de ID, que contiene información sobre la identidad del usuario que la aplicación puede utilizar.
5. El proveedor de identidad devuelve al usuario a la aplicación.

<div id="adldap">
  ### AD/LDAP
</div>

Lightweight Directory Access Protocol (LDAP) es un protocolo de aplicación que se utiliza para acceder a un directorio de credenciales que puede compartirse entre varias aplicaciones; se usa habitualmente en intranets. Cuando se combina con Active Directory (AD), LDAP proporciona una ubicación centralizada para la identidad del usuario, por lo que la aplicación envía una solicitud de autenticación al servidor LDAP/AD. El protocolo LDAP intercambia información en formato LDAP Data Interchange Format (LDIF).

<div id="service-provider-initiated-sso">
  ## SSO iniciado por el proveedor de servicios
</div>

En el [SSO iniciado por el proveedor de servicios](/docs/es/authenticate/single-sign-on/inbound-single-sign-on), Auth0 es el proveedor de servicios (SP) del SSO.

Cuando un usuario inicia sesión en una aplicación:

1. La aplicación le presenta al usuario uno o más proveedores de identidad externos.
2. El usuario selecciona un proveedor de identidad con el que autenticarse e inicia sesión.
3. Tras autenticarse correctamente, el usuario vuelve a la aplicación.

En Auth0, el SSO iniciado por el SP se gestiona mediante conexiones.

<div id="identity-provider-initiated-sso">
  ## SSO iniciado por el proveedor de identidad
</div>

Para el [SSO iniciado por el proveedor de identidad](/docs/es/authenticate/single-sign-on/outbound-single-sign-on), un proveedor de identidad (IdP) de terceros actúa como proveedor de SSO.

Cuando un usuario inicia sesión en una aplicación:

1. La aplicación redirige al usuario a un proveedor de identidad.
2. El proveedor de identidad de terceros realiza la autenticación y la autorización.
3. Tras autenticarse correctamente, el usuario regresa a la aplicación.

Al planificar una implementación de SSO iniciado por el IdP, puede optar por usar la [SSO Dashboard Extension](/docs/es/customize/extensions/single-sign-on-dashboard-extension/create-sso-dashboard-application), que le permite crear un Dashboard con una lista de varias aplicaciones empresariales que pueden habilitarse para SSO. Luego, este Dashboard se presenta a sus usuarios para que inicien sesión.

<div id="use-cases">
  ## Casos de uso
</div>

<div id="business-to-business">
  ### Business to Business
</div>

Para escenarios Business to Business (B2B), SSO puede simplificar la preparación de tu aplicación para su uso en entornos empresariales. Con Auth0, tus aplicaciones pueden admitir escenarios comunes de federación empresarial, como Active Directory (AD), Lightweight Directory Access Protocol (LDAP), Ping o Security Assertion Markup Language (SAML). Esto permite que tus partners y clientes empresariales inicien sesión con sus tecnologías de identidad empresarial preferidas.

* [Caso de estudio: O&#39;Reilly](https://auth0.com/case-studies/oreilly)

<div id="business-to-consumer-ciam">
  ### Business to Consumer CIAM
</div>

En escenarios de Business to Consumer (B2C) o Customer Identity Access Management (CIAM), SSO puede ofrecer acceso fluido a tus aplicaciones o servicios. Puedes permitir que los clientes se autentiquen mediante proveedores de identidad social populares, como Google, Facebook, LinkedIn, X y Microsoft, en lugar de pedirles que creen otra cuenta.

* [Caso de estudio: Giving Compass](https://auth0.com/case-studies/giving-compass)
