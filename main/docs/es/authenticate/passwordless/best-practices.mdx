<div id="use-sms-and-email-as-authentication-factors">
  ## Usa SMS y correo electrónico como factores de autenticación
</div>

La implementación de passwordless de Auth0 permite autenticar a los usuarios con un solo factor. Ese factor único puede ser un código de un solo uso enviado por correo electrónico o SMS, o un Magic Link enviado por correo electrónico.

Aunque usar correo electrónico o SMS puede ser más seguro que una contraseña débil, ambos presentan problemas conocidos:

* Los números de teléfono no son suficientes para la autenticación de usuario. El [sistema de enrutamiento telefónico SS7 que usan las redes celulares tiene vulnerabilidades comprobadas](https://thehackernews.com/2017/05/ss7-vulnerability-bank-hacking.html), por lo que no se recomienda como factor de autenticación. Existen muchos vectores de ataque, desde el uso de ingeniería social hasta el intercambio de tarjetas SIM y la compra de acceso a la red SS7.
* Tener acceso a una dirección de correo electrónico no es suficiente para la autenticación de usuario (por ejemplo, alias, reenvío y varias personas usando una misma cuenta). Los proveedores de correo electrónico varían en sus prácticas de seguridad, y algunos no exigen verificar la identidad del usuario. SMTP es un protocolo muy antiguo, y muchos proveedores aún enrutan el tráfico SMTP sin cifrar, lo que aumenta la probabilidad de sufrir un ataque de interceptación.

Por estas razones, recomendamos que, si usas autenticación passwordless, también implementes [Autenticación multifactor (MFA)](/docs/es/secure/multi-factor-authentication) con un factor distinto cuando el usuario realice una operación sensible desde el punto de vista de la seguridad.

<div id="prevent-phishing-attacks">
  ## Prevenir ataques de phishing
</div>

Un ejemplo de cómo podría producirse un ataque de phishing es el siguiente:

1. El usuario hace clic en un enlace de un correo electrónico o un sitio web malicioso.
2. El usuario llega al sitio falso del atacante, donde se le pide que introduzca su número de teléfono para autenticarse.
3. El usuario introduce su número de teléfono y el atacante introduce ese mismo número en la aplicación legítima.
4. La aplicación legítima envía un SMS al usuario.
5. El usuario escribe el código de un solo uso en el sitio web del atacante.
6. El atacante ya puede iniciar sesión en el sitio web legítimo.

Para reducir las probabilidades de éxito de este ataque, el usuario debería esperar que el SMS identifique claramente la aplicación. Debes configurar la plantilla de SMS para que mencione el nombre del tenant y/o el Nombre de la aplicación:

```http wrap lines
Your verification code for accessing's Acme @@application.name@@ is @@code@@
```

<div id="prevent-brute-force-attacks">
  ## Prevenir ataques de fuerza bruta
</div>

Auth0 cuenta con las siguientes protecciones contra ataques de fuerza bruta:

* Solo se aceptará el código de un solo uso (o enlace) emitido más recientemente. En cuanto se emite uno nuevo, los anteriores quedan invalidados. Una vez utilizado, el más reciente también queda invalidado.
* Solo se permiten tres intentos fallidos de introducir un mismo código de un solo uso. Después de eso, será necesario solicitar un código nuevo.
* El código de un solo uso emitido será válido durante tres minutos (de forma predeterminada) antes de caducar.
* Si un usuario passwordless ha sido bloqueado [administrativamente](/docs/es/manage-users/user-accounts/block-and-unblock-users), Auth0 no enviará ningún código OTP por SMS o correo electrónico hasta que se desbloquee al usuario. Este comportamiento evita que tus proveedores de SMS y correo electrónico reciban solicitudes innecesarias.

El tiempo de expiración del código de un solo uso puede modificarse en [Auth0 Dashboard &gt; Authentication &gt; Passwordless](https://manage.auth0.com/#/connections/passwordless).

<div id="prevent-user-enumeration-attacks">
  ## Prevenir ataques de enumeración de usuarios
</div>

La enumeración de usuarios se produce cuando un actor malicioso utiliza técnicas de fuerza bruta para adivinar o confirmar qué usuarios son válidos en un sistema.

Cuando **Disable Sign Ups** está habilitado, tu aplicación puede volverse vulnerable a ataques de enumeración de usuarios. Auth0 recomienda no habilitar esta configuración para garantizar la máxima seguridad de tu aplicación y sus usuarios.

Si habilitas **Disable Sign Ups**, la [protección contra ataques de fuerza bruta](/docs/es/secure/attack-protection/brute-force-protection) puede mitigar la amenaza de algunos de estos ataques, pero no protege por completo tu aplicación.

<div id="link-accounts">
  ## Vincular cuentas
</div>

Es posible que los usuarios quieran autenticarse con distintos factores passwordless a lo largo del tiempo. Por ejemplo, podrían registrarse inicialmente con un SMS y, más adelante, empezar a autenticarse con un correo electrónico. Puede lograrlo permitiéndoles vincular sus distintos perfiles mediante la [vinculación de cuentas](/docs/es/manage-users/user-accounts/user-account-linking/link-user-accounts).

<div id="set-the-auth0-forwarded-for-header-for-rate-limit-purposes">
  ## Establece el encabezado auth0-forwarded-for a efectos del límite de frecuencia
</div>

El endpoint `/passwordless/start` tiene un [límite de frecuencia](/docs/es/troubleshoot/customer-support/operational-policies/rate-limit-policy) de 50 solicitudes por hora y por dirección IP. Si llamas a la API desde el servidor, la IP de tu backend puede alcanzar fácilmente estos límites de frecuencia. Para saber cómo resolver este problema, consulta la sección Limitación de frecuencia en endpoints de Passwordless de [Uso de las API de Passwordless](/docs/es/authenticate/passwordless/implement-login/embedded-login/relevant-api-endpoints).
