  Si tienes [Token Vault](/docs/es/secure/tokens/token-vault) habilitado para tu conexión, los tokens de acceso y de actualización ya no se almacenarán en el arreglo `identities` del usuario. En su lugar, se guardarán en un conjunto seguro de tokens dentro de Token Vault. Para habilitar Token Vault, consulta [Configurar Token Vault](/docs/es/secure/tokens/token-vault/configure-token-vault).

Una vez que autentiques correctamente a un usuario con un <Tooltip tip="Proveedor de identidad (IdP): servicio que almacena y gestiona identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=Identity+Provider">proveedor de identidad</Tooltip> (IdP) externo, como Facebook o GitHub, el IdP suele incluir un <Tooltip tip="Token de acceso: credencial de autorización, en forma de una cadena opaca o JWT, usada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=access+token">token de acceso</Tooltip> en el perfil de usuario que devuelve a Auth0.

Puedes recuperar y usar este token para llamar a la API del IdP.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Este artículo asume que ya has configurado la conexión con el IdP que elijas. Si no es así, ve a [Identity Providers Supported by Auth0](/docs/es/authenticate/identity-providers/social-identity-providers), selecciona el IdP que quieras y sigue los pasos de configuración.
</Callout>

El proceso que seguirás varía según si tu código se ejecuta en el backend o en el frontend:

* Si tu código se ejecuta en el backend, podemos asumir que tu servidor es de confianza y puede almacenar secretos de forma segura (como verás, usamos un secreto en el escenario de backend). Si ese es el caso, continúa en la [sección de backend](#from-the-backend) de este artículo.
* Si tu código se ejecuta en el frontend (por ejemplo, si es una SPA o una aplicación nativa de escritorio o móvil), tu aplicación no puede almacenar credenciales de forma segura y debe seguir un enfoque alternativo. En este caso, continúa en la [sección de frontend](#from-the-frontend) de este artículo.

<div id="from-the-backend">
  ## Desde el backend
</div>

Una vez que autenticas a un usuario, el IdP suele incluir un token de acceso en el perfil del usuario que devuelve a Auth0.

Por motivos de seguridad y cumplimiento normativo, Auth0 no envía este token a tu aplicación como parte del perfil del usuario. Para obtenerlo, debes acceder a la Auth0 <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip> y recuperar el perfil completo del usuario:

1. Obtén un token de acceso que te permita llamar a la [Auth0 Management API](https://auth0.com/docs/api/management/v2).
2. Llama al [endpoint Get Users by ID de Auth0 Management API](https://auth0.com/docs/api/management/v2#!/Users/get_users_by_id) con el token de acceso obtenido en el paso 1. Este endpoint devuelve el perfil completo del usuario, que contiene el token de acceso del IdP.
3. Extrae el token de acceso del IdP de la respuesta y úsalo para llamar a la API del IdP.

<div id="step-1-get-a-token">
  ### Paso 1: Obtener un token
</div>

Necesitarás un token de acceso para llamar a la [Management API](https://auth0.com/docs/api/management/v2).

<div id="create-a-test-application-for-the-management-api">
  #### Cree una aplicación de prueba para la Management API
</div>

Si esta es la primera vez que solicita un [Management APIv2 Token](https://auth0.com/docs/api/management/v2/tokens), deberá crear y configurar una aplicación que pueda usarse para llamar a la Management API:

1. Vaya a [Auth0 Dashboard &gt; Applications &gt; APIs](https://manage.auth0.com/#/apis) y seleccione la [Auth0 Management API](https://manage.auth0.com/#/apis/management/authorized-clients).
2. Seleccione la vista **API Explorer** y haga clic en **Create &amp; Authorize a Test Application**.

Esto creará una nueva aplicación y concederá todos los scopes de la Management API, lo que significa que los tokens generados para esta aplicación podrán acceder a todos los endpoints de la Management API.

<Card title="¿No ve el botón?">
  Si no ve este botón, significa que ya tiene al menos una aplicación autorizada para la Management API. En este caso, puede actualizar los scopes de la aplicación existente y usar esa, o crear una nueva siguiendo estos pasos:

  1. Vaya a [Auth0 Dashboard &gt; Applications &gt; Applications](https://manage.auth0.com/#/applications) y seleccione **Create Application**.
  2. Seleccione **Machine to Machine Applications** y, a continuación, **Create**.
  3. En el menú desplegable **Select an API**, seleccione `Auth0 Management API`.
  4. Habilite los scopes requeridos y seleccione **Authorize**.
  5. Seleccione la vista **APIs** y active el toggle de **Auth0 Management API**.
</Card>

<Warning>
  Por motivos de seguridad, le recomendamos asignar únicamente los scopes requeridos a la aplicación que vaya a usar. En este caso concreto, los scopes que necesita son: `read:users`, `read:user_idp_tokens`. Los scopes requeridos se enumeran para cada endpoint en el [Management API Explorer](https://auth0.com/docs/api/management/v2).
</Warning>

Para conceder o quitar scopes de la [Auth0 Management API registrada](https://manage.auth0.com/#/apis/management/authorized-clients), seleccione la [vista **Machine to Machine Applications**](https://manage.auth0.com/#/apis/management/authorized-clients):

<Frame>![Edite los scopes concedidos a la aplicación](/docs/images/cdy7uua7fh8z/6nuMjQMrTWhoVqtRwJBQz7/28a0f7c89c696426e273f43bd849b5e5/2025-02-25_11-11-36.png)</Frame>

<div id="get-the-management-api-token">
  #### Obtén el token de la Management API
</div>

Ya terminaste la configuración y estás listo para obtener tu token de la Management API:

1. En la [Auth0 Management API registrada](https://manage.auth0.com/#/apis/management/authorized-clients), selecciona la [vista **Test**](https://manage.auth0.com/#/apis/management/test).
2. Elige tu aplicación en el menú desplegable **Application** para completar automáticamente los fragmentos listos para usar con variables personalizadas.
3. Elige el idioma que prefieras para el fragmento, cópialo y ejecútalo.
4. Extrae la propiedad `access_token` de la respuesta. Esto es lo que usarás para acceder a la Management API.

<Card title="¿Qué hacen los fragmentos?">
  Los fragmentos realizan una operación `POST` al [endpoint `/oauth/token` de la Auth0 Authentication API](https://auth0.com/docs/api/authentication#client-credentials), usando la **concesión de credenciales de cliente de OAuth 2.0**. Esta es la concesión que usan los procesos de máquina a máquina para acceder a una API. Para obtener más información sobre el flujo, consulta [Flujo de credenciales de cliente](/docs/es/get-started/authentication-and-authorization-flow/client-credentials-flow).
</Card>

<div id="token-expiration">
  #### Expiración del token
</div>

De forma predeterminada, el token que recibiste caduca en 24 horas (86.400 segundos). Para cambiarlo:

1. Ve a [Auth0 Dashboard &gt; Applications &gt; APIs](https://manage.auth0.com/#/apis) y selecciona la [Auth0 Management API](https://manage.auth0.com/#/apis/management/authorized-clients).
2. Selecciona la vista **Settings**, busca el campo **Token Expiration (Seconds)**, introduce un valor nuevo y haz clic en **Save**. El valor máximo que puedes establecer es 2.592.000 segundos (30 días), aunque te recomendamos mantener el valor predeterminado.

El siguiente token que generes usará el tiempo de expiración actualizado.

<Warning>
  Estos tokens **no se pueden revocar**. Para minimizar el riesgo, te recomendamos emitir tokens de corta duración y conceder solo los scopes necesarios para cada aplicación. En un entorno de producción, puedes configurar una CLI sencilla para obtener un nuevo token cuando caduque el anterior.
</Warning>

<div id="step-2-get-the-full-user-profile">
  ### Paso 2: Obtén el perfil de usuario completo
</div>

Para obtener el perfil de un usuario, llama al [endpoint Get a User](https://auth0.com/docs/api/management/v2#!/Users/get_users_by_id) de la Management API con el token de acceso que extrajiste en la sección anterior:

Reemplaza estos valores:

* `{userId}`: ID del usuario para el que quieres llamar a la API del IdP.
* `{yourAccessToken}`: token de acceso que extrajiste en la sección anterior.

<Card title="¿Dónde encuentro el ID de usuario?">
  - Para hacer pruebas, puedes encontrar un ID de usuario en [Auth0 Dashboard &gt; User Management &gt; Users](https://manage.auth0.com/#/users). Busca un usuario y copia el valor del campo **user&#95;id**.

  - Para tu implementación, puedes extraer el ID de usuario del claim `sub` del [token de ID](/docs/es/secure/tokens/id-tokens), o llamar al [endpoint /userinfo](https://auth0.com/docs/api/authentication#get-user-info) de la Authentication API y extraerlo de la propiedad `user_id` de la respuesta.
</Card>

<div id="step-3-extract-the-idp-access-token">
  ### Paso 3: Extraer el token de acceso del IdP
</div>

Puede encontrar el token de acceso que se usa para llamar a la API del IdP en el arreglo `identities` del usuario: `user.identities[0].access_token`.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Para ciertos proveedores de identidad, Auth0 también almacena un token de actualización, que puede usar para obtener un nuevo token de acceso del IdP. Esto funciona con: BitBucket, Google (OAuth 2.0), OAuth 2.0, SharePoint y Azure AD. Para obtener más información, consulte [Tokens de acceso del proveedor de identidad](/docs/es/secure/tokens/access-tokens/identity-provider-access-tokens).
</Callout>

En la mayoría de los casos, el usuario solo tendrá una identidad, pero si ha iniciado sesión varias veces a través de diferentes conexiones y ha utilizado la [vinculación de cuentas](/docs/es/manage-users/user-accounts/user-account-linking/link-user-accounts), puede que haya más.

En esta respuesta de ejemplo, vemos que nuestro usuario solo tiene una identidad: `google-oauth2`.

<div id="from-the-frontend">
  ## Desde el frontend
</div>

Si trabajas con una aplicación pública (SPA, aplicación nativa de escritorio o aplicación móvil), este es el lugar adecuado.

Cuando trabajas con una aplicación de frontend, el proceso para llamar a las API del IdP es distinto del proceso del backend, porque las aplicaciones de frontend son aplicaciones públicas que **no pueden almacenar credenciales de forma segura**. Dado que el código de una SPA puede verse y modificarse, y que las aplicaciones nativas o móviles pueden descompilarse e inspeccionarse, no es seguro confiarles información sensible como claves secretas o contraseñas.

En concreto, no pueden almacenar de forma segura el **<Tooltip tip="secreto de cliente: Secret que usa un cliente (aplicación) para autenticarse con el servidor de autorización; solo deben conocerlo el cliente y el servidor de autorización, y debe ser lo suficientemente aleatorio como para que no se pueda adivinar." cta="Ver glosario" href="/docs/es/glossary?term=Client+Secret">secreto de cliente</Tooltip>** de la aplicación de máquina a máquina, que usas para llamar a `/oauth/token` durante el primer paso del proceso del backend.

En su lugar, debes crear un proxy para tu backend y exponerlo a tu aplicación como una API.

<div id="build-a-proxy">
  ### Crear un proxy
</div>

Primero, crearás un proceso en tu backend que implemente los pasos incluidos en [la sección de backend](#from-the-backend) de este artículo y lo expondrás a tu aplicación como una API.

Llamarás a la API del IdP desde ese mismo proceso del backend, para que el token de acceso nunca quede expuesto a tu aplicación pública.

Luego, llamarás a tu API proxy desde tu aplicación pública mediante el [flujo de código de autorización con Proof Key for Code Exchange (PKCE)](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce).
