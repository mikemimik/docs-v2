Los [proveedores de identidad (IdP)](/docs/es/authenticate/identity-providers) más comunes están disponibles en el [Auth0 Dashboard](https://manage.auth0.com/#) y en el [Auth0 Marketplace](https://marketplace.auth0.com/features/social-connections). Sin embargo, puede agregar cualquier proveedor de <Tooltip tip="OAuth 2.0: marco de autorización que define protocolos y flujos de trabajo de autorización." cta="Ver glosario" href="/docs/es/glossary?term=OAuth+2.0">OAuth 2.0</Tooltip> como una **conexión social personalizada** en el <Tooltip tip="Auth0 Dashboard: el producto principal de Auth0 para configurar sus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip>.

1. En el Dashboard, vaya a [Authentication &gt; Social](https://manage.auth0.com/#/connections/social).
2. Seleccione **Create Connection**, vaya al final de la lista y, a continuación, seleccione **Create Custom**.

El formulario que aparece contiene varios campos que debe usar para configurar la conexión personalizada:

* **Nombre de la conexión**: Identificador lógico de la conexión que está creando. Este nombre no se puede cambiar, debe comenzar y terminar con un carácter alfanumérico y solo puede contener caracteres alfanuméricos y guiones.
* **URL de autorización**: URL a la que se redirige a los usuarios para iniciar sesión.

  <Callout icon="file-lines" color="#0EA5E9" iconType="regular">
    No intente establecer el parámetro OAuth2 `response_mode` en la URL de autorización. Esta conexión solo admite el valor predeterminado de `response_mode` (`query`).
  </Callout>
* **URL del token**: URL que se usa para intercambiar el código de autorización recibido por <Tooltip tip="Access Token: credencial de autorización, en forma de una cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=access+tokens">tokens de acceso</Tooltip> y, si se solicitan, <Tooltip tip="ID Token: credencial destinada al propio cliente, en lugar de para acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=ID+tokens">tokens de ID</Tooltip>.
* **Scope**: Parámetros `scope` que se envían con la solicitud de autorización. Separe varios scopes con espacios.
* **Separate scopes using a space**: Interruptor que determina cómo se delimitan los scopes si se incluye el parámetro `connection_scope` al [llamar a la API del IdP](/docs/es/authenticate/identity-providers/calling-an-external-idp-api). De forma predeterminada, los scopes se delimitan con una coma. Si el interruptor está habilitado, los scopes se delimitan con un espacio. Para obtener más información, lea [Agregar scopes/permisos para llamar a las API de proveedores de identidad](/docs/es/authenticate/identity-providers/adding-scopes-for-an-external-idp).
* **<Tooltip tip="ID de cliente: valor de identificación dado a su recurso registrado por Auth0." cta="Ver glosario" href="/docs/es/glossary?term=Client+ID">ID de cliente</Tooltip>**: ID de cliente de Auth0 como aplicación que se utiliza para solicitar autorización e intercambiar el código de autorización. Para obtener un ID de cliente, deberá registrarse con el <Tooltip tip="proveedor de identidad (IdP): servicio que almacena y administra identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=identity+provider">proveedor de identidad</Tooltip>.
* **<Tooltip tip="secreto de cliente: Secret utilizado por un cliente (aplicación) para autenticarse con el servidor de autorización; solo deberían conocerlo el cliente y el servidor de autorización, y debe ser lo suficientemente aleatorio como para que no pueda adivinarse." cta="Ver glosario" href="/docs/es/glossary?term=Client+Secret">secreto de cliente</Tooltip>**: Secreto de cliente de Auth0 como aplicación que se utiliza para intercambiar el código de autorización. Para obtener un secreto de cliente, deberá registrarse con el proveedor de identidad.
* **Script para obtener el perfil del usuario**: Script de Node.js que se usa para llamar a una URL de userinfo con el token de acceso proporcionado. Para obtener más información sobre este script, consulte [Script para obtener el perfil del usuario](#fetch-user-profile-script).
* **Propósito**: Habilita la conexión social para la autenticación, Connected Accounts for Token Vault o ambas. Para obtener más información, lea [autenticación de usuario vs Connected Accounts](/docs/es/secure/tokens/token-vault/connected-accounts-for-token-vault#user-authentication-vs-connected-accounts).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Al configurar el proveedor de identidad personalizado, usa la URL de devolución de llamada `https://{yourDomain}/login/callback`.
</Callout>

Una vez que hayas creado la conexión personalizada, verás la vista **Applications** y tu conexión estará sujeta a la [Política de límite de tasa](/docs/es/troubleshoot/customer-support/operational-policies/rate-limit-policy) de Auth0. Aquí puedes habilitar y deshabilitar las aplicaciones en las que quieras que aparezca la conexión.

<div id="update-authentication-flow">
  ## Actualizar el flujo de autenticación
</div>

Cuando crea una conexión, el tipo de concesión de OAuth 2.0 predeterminado asignado a la conexión es el Flujo de código de autorización. Si tiene una aplicación pública que no puede almacenar un secreto de cliente, como una aplicación de una sola página o una aplicación nativa, puede usar la <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip> para actualizar la conexión y que use [Flujo de código de autorización + PKCE](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce). Para obtener más información sobre los <Tooltip tip="Flujo de autorización: Concesión de autorización (o flujo de trabajo) especificada en el framework de OAuth 2.0." cta="Ver glosario" href="/docs/es/glossary?term=authorization+flows">flujos de autorización</Tooltip>, consulte [¿Qué flujo de OAuth 2.0 debería usar?](/docs/es/get-started/authentication-and-authorization-flow/which-oauth-2-0-flow-should-i-use)

1. Haga una solicitud `GET` al endpoint [`/get-connections-by-id`](https://auth0.com/docs/api/management/v2/connections/get-connections-by-id). La respuesta será similar a esta:

   ```json lines
   {
     "id": "[connectionID]",
     "options": {
       "email": true,
       "scope": [
         "email",
         "profile"
       ],
       "profile": true
     },
     "strategy": "google-oauth2",
     "name": "google-oauth2",
     "is_domain_connection": false,
     "realms": [
       "google-oauth2"
     ]
   }
   ```

2. Copie el objeto `options` completo.

3. Haga una solicitud [`PATCH`](https://auth0.com/docs/api/management/v2/connections/patch-connections-by-id) con el objeto `options` y agregue `"pkce_enabled":` `true`.

<Warning>
  Si no incluye el objeto `options` completo, se perderá información y la conexión dejará de funcionar.
</Warning>

<div id="fetch-user-profile-script">
  ## Script para obtener el perfil del usuario
</div>

El script para obtener el perfil del usuario se ejecuta después de que el usuario haya iniciado sesión con el proveedor de OAuth2. Auth0 ejecuta este script para llamar a la API del proveedor de OAuth2 y obtener el perfil del usuario:

```javascript lines expandable
function fetchUserProfile(accessToken, context, callback) {
  request.get(
    {
      url: 'https://auth.example.com/userinfo',
      headers: {
        'Authorization': 'Bearer ' + accessToken,
      }
    },
    (err, resp, body) => {
      if (err) {
        return callback(err);
      }
      if (resp.statusCode !== 200) {
        return callback(new Error(body));
      }
      let bodyParsed;
      try {
        bodyParsed = JSON.parse(body);
      } catch (jsonError) {
        return callback(new Error(body));
      }
      const profile = {
        user_id: bodyParsed.account.uuid,
        email: bodyParsed.account.email
      };
      callback(null, profile);
    }
  );
}
```

La propiedad `user_id` en el perfil devuelto es obligatoria, y la propiedad `email` es opcional, pero muy recomendable. Para obtener más información sobre qué atributos se pueden devolver, consulta [Atributos raíz del perfil de usuario](/docs/es/manage-users/user-accounts/user-profiles/root-attributes/update-root-attributes-for-users).

Puedes filtrar, agregar o eliminar cualquier elemento del perfil devuelto por el proveedor. Sin embargo, se recomienda mantener este script lo más simple posible. Las manipulaciones más complejas de la información del usuario se pueden realizar mediante [Rules](/docs/es/customize/rules). Una ventaja de usar Rules es que se aplican a cualquier conexión.

  ## Pasar parámetros específicos de cada proveedor
</div>

Puedes pasar parámetros específicos de cada proveedor al endpoint de autorización de los proveedores de OAuth 2.0. Estos pueden ser estáticos o dinámicos.

<div id="pass-static-parameters">
  ### Pasar parámetros estáticos
</div>

<div id="pass-dynamic-parameters">
  ### Pasar parámetros dinámicos
</div>

En determinadas circunstancias, puede que quieras pasar un valor dinámico a un proveedor de identidad de OAuth 2.0. En ese caso, puedes usar el elemento `authParamsMap` de `options` para especificar un mapeo entre uno de los parámetros adicionales existentes que acepta el [endpoint de Auth0 `/authorize`](https://auth0.com/docs/api/authentication#social) y el parámetro que acepta el proveedor de identidad.

Siguiendo con el ejemplo anterior, supongamos que quieres pasar el parámetro `custom_param` al endpoint de autorización, pero quieres especificar el valor real del parámetro al llamar al endpoint de Auth0 `/authorize`.

En este caso, puedes usar uno de los parámetros adicionales existentes que acepta el endpoint `/authorize`, como `access_type`, y mapearlo al parámetro `custom_param`:

<AuthCodeGroup>
  ```bash cURL
  curl --request PATCH \
    --url 'https://{yourDomain}/api/v2/connections/CONNECTION-ID' \
    --header 'content-type: application/json' \
    --data '{ "options": { "client_id": "...", "client_secret": "...", "authParamsMap": { "custom_param": "access_type" }, "scripts": { "fetchUserProfile": "..." }, "authorizationURL": "https://auth.example.com/oauth2/authorize", "tokenURL": "https://auth.example.com/oauth2/token", "scope": "auth" } }'
  ```

<div id="pass-extra-headers">
  ## Envía encabezados adicionales
</div>

En algunos casos, tendrás que enviar encabezados adicionales al <Tooltip tip="Token Endpoint: Endpoint en el servidor de autorización que se usa para solicitar tokens mediante programación." cta="Ver glosario" href="/docs/es/glossary?term=Token+endpoint">Token endpoint</Tooltip> de un proveedor de OAuth 2.0. Para configurar encabezados adicionales, abre Settings de la conexión y, en el campo **Custom Headers**, especifica un objeto JSON con los encabezados personalizados como pares clave-valor:

```json lines
{
    "Header1" : "Value",
    "Header2" : "Value"
}
```

Usemos un ejemplo en el que un proveedor de identidad puede requerir que envíes un encabezado `Authorization` con credenciales de [autenticación básica](https://en.wikipedia.org/wiki/Basic_access_authentication). En este caso, puedes especificar el siguiente objeto JSON en el campo **Custom Headers**:

```json lines
{
  "Authorization": "Basic [your credentials]"
}
```

donde `[your credentials]` son las credenciales reales que se deben enviar al proveedor de identidad.
