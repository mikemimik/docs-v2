Los [proveedores de identidad (IdP)](/docs/es/authenticate/identity-providers) más comunes están disponibles en el [Auth0 Dashboard](https://manage.auth0.com/#) y en el [Auth0 Marketplace](https://marketplace.auth0.com/features/social-connections). Sin embargo, puedes agregar cualquier proveedor de <Tooltip tip="OAuth 2.0: Marco de autorización que define protocolos y flujos de trabajo de autorización." cta="Ver glosario" href="/docs/es/glossary?term=OAuth+2.0">OAuth 2.0</Tooltip> como una **Custom Social Connection** en el <Tooltip tip="Auth0 Dashboard: El producto principal de Auth0 para configurar tus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip>.

1. En el Dashboard, ve a [Authentication &gt; Social](https://manage.auth0.com/#/connections/social).
2. Selecciona **Create Connection**, desplázate hasta el final de la lista y luego selecciona **Create Custom**.

El formulario que aparece contiene varios campos que debes usar para configurar la conexión personalizada:

* **Connection Name**: Identificador lógico de la conexión que estás creando. Este nombre no se puede cambiar, debe comenzar y terminar con un carácter alfanumérico y solo puede contener caracteres alfanuméricos y guiones.
* **Authorization URL**: URL a la que se redirige a los usuarios para iniciar sesión.

  <Callout icon="file-lines" color="#0EA5E9" iconType="regular">
    No intentes establecer el parámetro OAuth2 `response_mode` en la URL de autorización. Esta conexión solo admite el valor predeterminado de `response_mode` (`query`).
  </Callout>
* **Token URL**: URL que se usa para intercambiar el código de autorización recibido por <Tooltip tip="Access Token: Credencial de autorización, en forma de cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=access+tokens">tokens de acceso</Tooltip> y, si se solicitan, <Tooltip tip="ID Token: Credencial destinada al propio cliente, en lugar de para acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=ID+tokens">token de ID</Tooltip>.
* **Scope**: Parámetros `scope` que se enviarán con la solicitud de autorización. Separa varios scopes con espacios.
* **Separate scopes using a space**: Control que determina cómo se delimitan los scopes si se incluye el parámetro `connection_scope` al [llamar a la API del IdP](/docs/es/authenticate/identity-providers/calling-an-external-idp-api). De forma predeterminada, los scopes se delimitan con una coma. Si el control está habilitado, los scopes se delimitan con un espacio. Para obtener más información, consulta [Add Scopes/Permissions to Call Identity Provider APIs](/docs/es/authenticate/identity-providers/adding-scopes-for-an-external-idp).
* **<Tooltip tip="Client ID: Valor de identificación otorgado a tu recurso registrado por Auth0." cta="Ver glosario" href="/docs/es/glossary?term=Client+ID">ID de cliente</Tooltip>**: ID de cliente de Auth0 como aplicación utilizada para solicitar autorización e intercambiar el código de autorización. Para obtener un ID de cliente, deberás registrarte con el <Tooltip tip="Identity Provider (IdP): Servicio que almacena y administra identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=identity+provider">proveedor de identidad</Tooltip>.
* **<Tooltip tip="Client Secret: Secret utilizado por un cliente (aplicación) para autenticarse con el servidor de autorización; solo deberían conocerlo el cliente y el servidor de autorización, y debe ser lo suficientemente aleatorio como para que no pueda adivinarse." cta="Ver glosario" href="/docs/es/glossary?term=Client+Secret">secreto de cliente</Tooltip>**: Secreto de cliente de Auth0 como aplicación utilizada para intercambiar el código de autorización. Para obtener un secreto de cliente, deberás registrarte con el proveedor de identidad.
* **Fetch User Profile Script**: Script de Node.js que se usa para llamar a una URL de userinfo con el token de acceso proporcionado. Para obtener más información sobre este script, consulta [Fetch User Profile Script](#fetch-user-profile-script).
* **Purpose**: Habilita la conexión social para autenticación, Connected Accounts for Token Vault o ambas opciones. Para obtener más información, consulta [User authentication vs Connected Accounts](/docs/es/secure/tokens/token-vault/connected-accounts-for-token-vault#user-authentication-vs-connected-accounts).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Al configurar el proveedor de identidad personalizado, use la URL de devolución de llamada `https://{yourDomain}/login/callback`.
</Callout>

Una vez que cree la conexión personalizada, verá la vista **Applications** y su conexión estará sujeta a la [Política de límite de tasa](/docs/es/troubleshoot/customer-support/operational-policies/rate-limit-policy). Aquí puede habilitar y deshabilitar las aplicaciones en las que quiera que aparezca la conexión.

<div id="update-authentication-flow">
  ## Actualizar flujo de autenticación
</div>

Cuando creas una conexión, el tipo de concesión de OAuth 2.0 predeterminado asignado a la conexión es Authorization Code Flow. Si tienes una aplicación pública que no puede almacenar un secreto de cliente, como una aplicación de una sola página o una aplicación nativa, puedes usar la <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip> para actualizar la conexión a fin de que use [Authorization Code Flow + PKCE](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce). Para obtener más información sobre los <Tooltip tip="Flujo de autorización: Concesión de autorización (o workflow) especificada en el framework de OAuth 2.0." cta="Ver glosario" href="/docs/es/glossary?term=authorization+flows">flujos de autorización</Tooltip>, consulta [¿Qué flujo de OAuth 2.0 debo usar?](/docs/es/get-started/authentication-and-authorization-flow/which-oauth-2-0-flow-should-i-use)

1. Haz una solicitud `GET` al endpoint [`/get-connections-by-id`](https://auth0.com/docs/api/management/v2/connections/get-connections-by-id). La respuesta será similar a:

   ```json lines
   {
     "id": "[connectionID]",
     "options": {
       "email": true,
       "scope": [
         "email",
         "profile"
       ],
       "profile": true
     },
     "strategy": "google-oauth2",
     "name": "google-oauth2",
     "is_domain_connection": false,
     "realms": [
       "google-oauth2"
     ]
   }
   ```

2. Copia todo el objeto `options`.

3. Haz una solicitud [`PATCH`](https://auth0.com/docs/api/management/v2/connections/patch-connections-by-id) con el objeto `options` y agrega `"pkce_enabled":` `true`.

<Warning>
  Si no incluyes el objeto `options` completo, se perderá información y la conexión dejará de funcionar.
</Warning>

<div id="fetch-user-profile-script">
  ## Script de obtención del perfil de usuario
</div>

El script de obtención del perfil de usuario se ejecuta después de que el usuario inicie sesión con el proveedor OAuth2. Auth0 ejecuta este script para llamar a la API del proveedor OAuth2 y obtener el perfil del usuario:

```javascript lines expandable
function fetchUserProfile(accessToken, context, callback) {
  request.get(
    {
      url: 'https://auth.example.com/userinfo',
      headers: {
        'Authorization': 'Bearer ' + accessToken,
      }
    },
    (err, resp, body) => {
      if (err) {
        return callback(err);
      }
      if (resp.statusCode !== 200) {
        return callback(new Error(body));
      }
      let bodyParsed;
      try {
        bodyParsed = JSON.parse(body);
      } catch (jsonError) {
        return callback(new Error(body));
      }
      const profile = {
        user_id: bodyParsed.account.uuid,
        email: bodyParsed.account.email
      };
      callback(null, profile);
    }
  );
}
```

La propiedad `user_id` del perfil devuelto es obligatoria, y la propiedad `email` es opcional, pero muy recomendable. Para obtener más información sobre qué atributos se pueden devolver, consulta [Atributos raíz del perfil de usuario](/docs/es/manage-users/user-accounts/user-profiles/root-attributes/update-root-attributes-for-users).

Puedes filtrar, añadir o eliminar cualquier elemento del perfil devuelto por el proveedor. Sin embargo, se recomienda que este script sea lo más sencillo posible. La manipulación más sofisticada de la información del usuario se puede lograr mediante [Rules](/docs/es/customize/rules). Una ventaja de usar Rules es que se aplican a cualquier conexión.

  ## Pasa parámetros específicos del proveedor
</div>

Puedes pasar parámetros específicos del proveedor al endpoint de autorización de los proveedores de OAuth 2.0. Pueden ser estáticos o dinámicos.

<div id="pass-static-parameters">
  ### Pasar parámetros estáticos
</div>

Para pasar parámetros estáticos (parámetros que se envían con cada solicitud de autorización), puedes usar el elemento `authParams` de `options` al configurar una conexión de OAuth 2.0 mediante la [Management API](https://auth0.com/docs/api/management/v2#!/Connections/patch_connections_by_id). La siguiente llamada configurará el parámetro estático `custom_param` con el valor `custom.param.value` en todas las solicitudes de autorización:

<AuthCodeGroup>

  ```bash cURL
    --header 'content-type: application/json' \
  ```

<div id="pass-dynamic-parameters">
  ### Pasar parámetros dinámicos
</div>

En determinadas circunstancias, puede que quieras pasar un valor dinámico a un proveedor de identidad de OAuth 2.0. En ese caso, puedes usar el elemento `authParamsMap` de `options` para especificar un mapeo entre uno de los parámetros adicionales ya admitidos por el [endpoint `/authorize` de Auth0](https://auth0.com/docs/api/authentication#social) y el parámetro admitido por el proveedor de identidad.

Siguiendo con el mismo ejemplo anterior, supongamos que quieres pasar el parámetro `custom_param` al endpoint de autorización, pero quieres especificar el valor real del parámetro al llamar al endpoint `/authorize` de Auth0.

En ese caso, puedes usar uno de los parámetros adicionales ya admitidos por el endpoint `/authorize`, como `access_type`, y mapearlo al parámetro `custom_param`:

<AuthCodeGroup>
  ```bash cURL
  curl --request PATCH \
    --url 'https://{yourDomain}/api/v2/connections/CONNECTION-ID' \
    --header 'content-type: application/json' \
    --data '{ "options": { "client_id": "...", "client_secret": "...", "authParamsMap": { "custom_param": "access_type" }, "scripts": { "fetchUserProfile": "..." }, "authorizationURL": "https://auth.example.com/oauth2/authorize", "tokenURL": "https://auth.example.com/oauth2/token", "scope": "auth" } }'
  ```

  ```csharp C#
  var client = new RestClient("https://{yourDomain}/api/v2/connections/CONNECTION-ID");
  ```

<div id="pass-extra-headers">
  ## Enviar encabezados adicionales
</div>

En algunos casos, tendrás que enviar encabezados adicionales al <Tooltip tip="Token endpoint: Endpoint en el servidor de autorización que se utiliza para solicitar tokens mediante programación." cta="Ver glosario" href="/docs/es/glossary?term=Token+endpoint">Token endpoint</Tooltip> de un proveedor de OAuth 2.0. Para configurar encabezados adicionales, abre Settings de la conexión y, en el campo **Custom Headers**, especifica un objeto JSON con los encabezados personalizados como pares clave-valor:

```json lines
{
    "Header1" : "Value",
    "Header2" : "Value"
}
```

Veamos un ejemplo en el que un proveedor de identidad puede requerir que envíes un encabezado `Authorization` con credenciales de [autenticación básica de acceso](https://en.wikipedia.org/wiki/Basic_access_authentication). En este caso, puedes especificar el siguiente objeto JSON en el campo **Custom Headers**:

```json lines
{
  "Authorization": "Basic [your credentials]"
}
```

Donde `[your credentials]` son las credenciales reales que se envían al proveedor de identidad.
