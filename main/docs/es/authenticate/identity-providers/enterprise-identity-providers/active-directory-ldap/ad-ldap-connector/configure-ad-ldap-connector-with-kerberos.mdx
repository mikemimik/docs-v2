title: Configurar la autenticación de Kerberos con el Conector AD/LDAP
---

Puedes federar con Active Directory a través del Conector AD/LDAP si usas Classic Login. El Conector AD/LDAP permite que tus usuarios se autentiquen cuando usan un equipo unido al dominio dentro de la red corporativa.

<div id="configure-active-directory">
  ## Configurar Active Directory
</div>

1. Ve a [Auth0 Dashboard &gt; Autenticación &gt; Empresarial &gt; Active Directory/LDAP](https://manage.auth0.com/#/connections/enterprise/ad) y selecciona la conexión que quieras configurar.
2. Activa el interruptor **Use Windows Integrated Auth (Kerberos)**.

<div id="auto-detected-range-for-kerberos">
  ### Rango detectado automáticamente para Kerberos
</div>

Cuando la autenticación de Kerberos está habilitada, la dirección IP visible del servidor donde se ejecuta el Conector AD se agrega implícitamente al rango de IP de la red. Esto significa que, si las solicitudes de un usuario se originan desde la misma dirección IP visible que la del Conector AD, se intentará la autenticación de Kerberos.

1. Configure los **Rangos de IP**. Use la [notación CIDR](https://en.wikipedia.org/wiki/Classless_Inter-Domain_Routing). Deben ser rangos visibles para Auth0. Cuando Auth0 se ejecuta en la nube, no podrá ver la dirección IP interna del usuario. En ese caso, debe configurar la(s) dirección(es) IP públicas/WAN de su empresa.
2. Recomendamos reiniciar el servicio de Windows que aloja el Conector AD cada vez que se cambie esta configuración. De esta manera, los cambios surtirán efecto de inmediato.

<div id="authentication-flow">
  ## Flujo de autenticación
</div>

Dependiendo de la ubicación de los usuarios, el flujo de autenticación será diferente cuando se configuren rangos de IP. Usando Fabrikam como ejemplo, como utiliza la versión SaaS de Auth0, configuraron su dirección IP pública (`24.12.34.56/32`) en la conexión. Los usuarios que se conecten desde dentro del edificio procederán todos de `24.12.34.56` (tal como está configurado en la conexión). Al autenticarse, los usuarios pueden seguir el flujo nativo de AD/LDAP y tener una experiencia de <Tooltip tip="Single Sign-On (SSO): Servicio que, después de que un usuario inicia sesión en una aplicación, inicia automáticamente la sesión de ese usuario en otras aplicaciones." cta="Ver glosario" href="/docs/es/glossary?term=SSO">SSO</Tooltip> fluida.

Para que esto funcione, la red debe permitir que los usuarios se conecten al Conector AD/LDAP a través del puerto configurado en el archivo `config.json`. En implementaciones de alta disponibilidad del Conector, la dirección a la que se conectarán los usuarios es la del equilibrador de carga de red situado frente a todas las instancias del conector.

<div id="auto-login-with-lock">
  ## Inicio de sesión automático con Lock
</div>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Detectar rangos de IP en una conexión AD/LDAP y usar esos rangos con Lock para permitir Windows Authentication es una funcionalidad que funciona en Lock 10, pero en Lock 11 solo puede usarse en escenarios de Universal Login. Esta funcionalidad está deshabilitada en Lock 11 cuando se usa en escenarios de Embedded Login.
</Callout>

Cuando una aplicación usa Lock 10 u 11 dentro de la página de inicio de sesión alojada por Auth0 (normalmente usada para los protocolos <Tooltip tip="Security Assertion Markup Language (SAML): Protocolo estandarizado que permite a dos partes intercambiar información de autenticación sin una contraseña." cta="Ver glosario" href="/docs/es/glossary?term=SAML">SAML</Tooltip>/WS-Federation e integraciones de inicio de sesión único (SSO)), habrá un botón que permitirá a los usuarios autenticarse mediante &quot;Windows Authentication&quot;.

En algunos casos, puede ser necesario iniciar sesión automáticamente al usuario si Kerberos es posible (según la dirección IP del usuario final). Se pueden agregar los siguientes cambios a la página de inicio de sesión de Auth0 para iniciar sesión automáticamente al usuario si Kerberos es posible:

```html lines expandable
<script src="https://cdn.auth0.com/js/lock/11.x.x/lock.min.js"></script>
<script src="https://cdn.auth0.com/js/auth0/9.x/auth0.min.js"></script>
<script src="https://cdn.auth0.com/js/polyfills/1.0/object-assign.min.js"></script>

<script>
  var config = JSON.parse(decodeURIComponent(escape(window.atob('@@config@@'))));

  var lock = new Auth0Lock(config.clientID, config.auth0Domain, {
      //...configuración adicional
  });

  function handleError(err) {
    // agregar el manejo de errores adecuado
    console.log(err);
  };

  var params = Object.assign({
      scope: config.internalOptions.scope,
      _csrf: config.internalOptions._csrf,
      state: config.internalOptions.state,
    },
    {
      /* configuración adicional necesaria para el uso de dominios personalizados 
      overrides: {
        __tenant: config.auth0Tenant,
        __token_issuer: '{yourCustomDomain}'
      }, */
      domain: config.auth0Domain,
      clientID: config.clientID,
      redirectUri: config.callbackURL,
      responseType: 'code'
    });

  var webAuth = new auth0.WebAuth(params);

  /*
   * Verificar si Kerberos es posible; si lo es, intentar autenticar al usuario.
   *
   * la respuesta de getSSOData solo tendrá una conexión y estrategia si
   * la dirección IP está dentro del rango de Kerberos en la configuración de la conexión
   */
  webAuth.client.getSSOData(true, function(err, data) {
    if (err) handleError(err);

    if (data.connection && data.strategy === 'ad') {
      webAuth.authorize({connection: data.connection}, function(err) {
        if (err) handleError(err);
      });
    } else {
      lock.show();
    }
  });
</script>
```

<div id="bypass-kerberos-at-runtime">
  ## Omitir Kerberos en runtime
</div>

Puede evitar que se utilice Kerberos, incluso si el usuario inicia sesión desde una dirección IP dentro del rango configurado en la configuración de la conexión, pasando `rememberLastLogin: false` a `lock.show()`.

```javascript lines
function useKerberos() {
      // devuelve true para usar Kerberos, false para omitirlo
    };
    
    lock.show({rememberLastLogin: useKerberos()});
```

<div id="enable-kerberos-request-logging">
  ## Habilitar el registro de solicitudes de Kerberos
</div>

1. Para habilitar el registro detallado de las solicitudes de Kerberos, agregue una variable de entorno a nivel del sistema: `DEBUG=kerberos-server`.
2. Reinicie el Conector.
3. Vuelva a iniciar sesión y revise los registros para obtener más información.

<div id="firefox-support-for-kerberos">
  ## Compatibilidad de Firefox con Kerberos
</div>

De forma predeterminada, [Firefox](https://www.mozilla.org/firefox) [rechaza todas las solicitudes &quot;negotiate&quot; necesarias para autenticar a los usuarios con Kerberos](https://developer.mozilla.org/en-US/docs/Mozilla/Integrated_authentication). Si deseas usar Firefox con Kerberos, debes añadir a la lista de permitidos el servidor donde está instalado el Conector.

1. Abre una pestaña de Firefox y escribe `about:config` en la barra de direcciones.
2. Descarta cualquier mensaje de advertencia y, en el cuadro de búsqueda, escribe `negotiate`.
3. Busca el elemento `network.negotiate-auth.trusted-uris` y haz doble clic para cambiar su valor.
4. Escribe el nombre de dominio del servidor donde está instalado el Conector. Si tienes varias instancias del Conector detrás de un equilibrador de carga, agrega el nombre DNS del equilibrador.
   El valor admite una lista de prefijos de URL o dominios separados por comas con el formato `mydomain.com, https://myotherdomain.com`.
5. Haz clic en **Ok**. No necesitas reiniciar el servidor para que los cambios surtan efecto.

<div id="https-support-for-kerberos">
  ## Compatibilidad con HTTPS para Kerberos
</div>

La autenticación de Kerberos funciona a través de HTTP (no de HTTPS). Es posible que Microsoft Office 365 y otros productos modernos no funcionen con HTTP.

Para resolver esta limitación:

1. Configure un proxy inverso y exponga el Conector AD/LDAP en un dominio HTTPS. Puede usar el parámetro `SERVER_URL` (URL expuesta externamente) para publicar la ubicación pública en la que el Conector AD/LDAP escuchará las solicitudes entrantes.
2. Haga que `SERVER_URL` en el proxy inverso apunte a todas las instancias internas de los conectores desplegados.
