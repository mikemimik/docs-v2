Para conectar su aplicación a Active Directory Federation Services (ADFS) de Microsoft, deberá proporcionar la siguiente información a su administrador de ADFS:

* Identificador de realm: `urn:auth0:{yourTenant}`
* Endpoint: `https://{yourDomain}/login/callback` o `https://<YOUR CUSTOM DOMAIN>/login/callback`, si está usando un [dominio personalizado](/docs/es/customize/custom-domains).

<Card title="Metadatos de federación">
  El archivo de metadatos de federación contiene información sobre los certificados del servidor ADFS. Si el endpoint de metadatos de federación (`/FederationMetadata/2007-06/FederationMetadata.xml`) está habilitado en ADFS, Auth0 puede buscar periódicamente (una vez al día) cambios en la configuración, como la incorporación de un nuevo certificado de firma de cara a una rotación. Por eso, es preferible habilitar el endpoint de metadatos de federación en lugar de proporcionar un archivo de metadatos independiente. Si proporciona un archivo de metadatos independiente, le notificaremos por correo electrónico cuando los certificados estén cerca de su fecha de expiración.
</Card>

Puede usar un script para configurar la conexión o hacerlo manualmente.

  ## Configuración mediante script
</div>

Ejecute los dos comandos siguientes en la ventana de Windows PowerShell.

  Debe ejecutar este script como administrador del sistema.

export const codeExample1 = `AddRelyingParty "urn:auth0:{yourTenant}" "https://{yourDomain}/login/callback"`;

<AuthCodeBlock children={codeExample1} language="powershell" />

Para la integración automatizada, este script usa [ADFS PowerShell SnapIn](http://technet.microsoft.com/en-us/library/adfs2-powershell-basics.aspx) para crear y configurar un **<Tooltip tip="Relying Party: Entidad (como un servicio o una aplicación) que depende de un proveedor de identidad de terceros para autenticar a un usuario." cta="Ver glosario" href="/docs/es/glossary?term=Relying+Party">Relying Party</Tooltip>** que emitirá, para el usuario autenticado, los siguientes claims: **email**, **upn**, **nombre de pila** y **apellido**.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Si usas la funcionalidad de [dominios personalizados](/docs/es/customize/custom-domains), tendrás que sustituir el valor de `$webAppEndpoint` por `https://<YOUR CUSTOM DOMAIN>/login/callback`.
</Callout>

El script crea el Relying Party Trust en ADFS de la siguiente manera:

export const codeExample2 = `$realm = "urn:auth0:{yourTenant}";
$webAppEndpoint = "https://{yourDomain}/login/callback";
Add-PSSnapin Microsoft.Adfs.Powershell
Add-ADFSRelyingPartyTrust -Name $realm -Identifier $realm -WSFedEndpoint $webAppEndpoint
$rp = Get-ADFSRelyingPartyTrust -Name $realm`;

<AuthCodeBlock children={codeExample2} language="powershell" />

El script también crea reglas para mostrar los atributos más comunes, como correo electrónico, UPN, nombre de pila o apellido:

```powershell lines
$rules = @'
@RuleName = "Store: ActiveDirectory -> Mail (ldap attribute: mail), Name (ldap attribute: displayName), Name ID (ldap attribute: userPrincipalName), GivenName (ldap attribute: givenName), Surname (ldap attribute: sn)"
c:[Type == "http://schemas.microsoft.com/ws/2008/06/identity/claims/windowsaccountname", Issuer == "AD AUTHORITY"]
=> issue(store = "Active Directory",
    types = ("http://schemas.xmlsoap.org/ws/2005/05/identity/claims/emailaddress",
             "http://schemas.xmlsoap.org/ws/2005/05/identity/claims/name",
             "http://schemas.xmlsoap.org/ws/2005/05/identity/claims/nameidentifier",
             "http://schemas.xmlsoap.org/ws/2005/05/identity/claims/givenname",
             "http://schemas.xmlsoap.org/ws/2005/05/identity/claims/surname"), query = ";mail,displayName,userPrincipalName,givenName,sn;{0}", param = c.Value);
'@
Set-ADFSRelyingPartyTrust –TargetName $realm -IssuanceTransformRules $rules
$rSet = New-ADFSClaimRuleSet –ClaimRule '=> issue(Type = "http://schemas.microsoft.com/authorization/claims/permit", Value = "true");'
Set-ADFSRelyingPartyTrust –TargetName $realm –IssuanceAuthorizationRules $rSet.ClaimRulesString
```

<div id="manual-setup-part-1-add-a-relying-party-trust">
  ## Configuración manual, parte 1: Agregar un Relying Party Trust
</div>

1. Abra la consola de administración de ADFS.
2. En la parte derecha de la consola, haga clic en **Add Relying Party Trust***
3. Haga clic en **Start**.
4. Seleccione **Enter data about the relying party manually** y haga clic en **Next**.
5. Escriba un nombre (como `{yourAppName}`) y haga clic en **Next**.
6. Use la opción predeterminada (`ADFS 2.0 profile`) y haga clic en **Next**.
7. Use la opción predeterminada (`no encryption certificate`) y haga clic en **Next**.
8. Marque **Enable support for the WS-Federation...** y escriba este valor en el cuadro de texto:
   `https://{yourDomain}/login/callback`, o, si está usando un [dominio personalizado](/docs/es/customize/custom-domains), use `https://<YOUR CUSTOM DOMAIN>/login/callback`
9. Haga clic en **Next**.
10. Agregue un identificador de Relying Party Trust con este valor:
    `urn:auth0:{yourTenant}`
11. Haga clic en **Add** y luego en **Next**.
12. Deje la opción predeterminada `Permit all users...` y haga clic en **Next**.
13. Haga clic en **Next** y luego en **Close**.

<div id="manual-setup-part-2-add-a-claim-issuance-policy-rule">
  ## Configuración manual, parte 2: Agregar una regla de directiva de emisión de claims
</div>

1. Si usas Windows Server 2019, el cuadro de diálogo Edit Claim Issuance Policy se abre automáticamente al terminar el asistente Add Relying Party Trust. Si usas Windows 2012 o 2016, sigue estos pasos:

| En Windows Server 2012                                                                                                                                               | En Windows Server 2016                                                                                                                                                                                                                          |
| -------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| En el panel Actions, en el lado derecho de la consola, busca el Relying Party Trust que acabas de crear. Debajo de este, haz clic en **Edit Claim Issuance Policy**. | En el árbol de la consola, en ADFS, haz clic en **Relying Party Trusts**. En el lado derecho de la consola, busca el Relying Party Trust que acabas de crear. Haz clic con el botón derecho sobre él y luego en **Edit Claim Issuance Policy**. |

2. En la ventana Edit Claim Issuance Policy, en Issuance Transform Rules, haz clic en **Add Rule...**.
3. Deja el valor predeterminado `Send LDAP Attributes as Claims`.
4. Asigna a la regla un nombre que describa lo que hace.
5. En Attribute Store, selecciona **Active Directory**.
6. Selecciona estos mapeos en `Mapping of LDAP attributes to outgoing claim types` y haz clic en **Finish**.

| Atributo LDAP       | Tipo de claim saliente          |
| ------------------- | ------------------------------- |
| E-Mail-Addresses    | Dirección de correo electrónico |
| Display-Name        | Nombre                          |
| User-Principal-Name | ID de nombre                    |
| Given-Name          | Nombre de pila                  |
| Surname             | Apellido                        |

<div id="add-additional-ldap-attributes">
  ### Agregar atributos LDAP adicionales
</div>

Los mapeos de los pasos anteriores son los más utilizados, pero si necesita atributos LDAP adicionales con información sobre el usuario, puede agregar más mapeos de claims.

1. Si cerró la ventana en el paso anterior, seleccione **Edit Claim Rules** en el menú contextual de la Relying Party Trust que creó y edite la rule.
2. Cree una fila adicional por cada atributo LDAP que necesite; elija el nombre del atributo en la columna izquierda y el tipo de claim deseado en la columna derecha.
3. Si el tipo de claim que busca no existe, tiene dos opciones:

   1. Escriba un [nombre calificado con espacio de nombres](/docs/es/secure/tokens/json-web-tokens/create-custom-claims) para el nuevo claim (por ejemplo, `http://schemas.xmlsoap.org/ws/2005/05/identity/claims/department`).
   2. Registre un nuevo tipo de claim (en **ADFS &gt; Services &gt; Claim Descriptions** de la consola de administración de ADFS) y use el nombre del claim en el mapeo.
      Auth0 usa la parte del nombre del tipo de claim (por ejemplo, `department` en `http://schemas.xmlsoap.org/ws/2005/05/identity/claims/department`) como nombre del atributo del perfil de usuario.
