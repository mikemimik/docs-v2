title: Express Configuration with Okta
---

La [Okta Integration Network (OIN)](https://www.okta.com/integrations/) es un catálogo de aplicaciones SaaS para las que Okta ofrece una experiencia de configuración exprés que permite habilitar el inicio de sesión único (SSO) con OpenID Connect, el aprovisionamiento automatizado de usuarios con SCIM y [Universal Logout](https://developer.okta.com/docs/guides/oin-universal-logout-overview). Los administradores de Okta usan la [Consola de administración de Okta](https://www.okta.com/okta-administrator-experience) para configurar estas integraciones dentro de su tenant de Okta:

<Frame>![Consola de administración de Okta](/docs/images/cdy7uua7fh8z/okta-admin-dash.png)</Frame>

Express Configuration permite a los clientes empresariales configurar de forma segura integraciones de identidad con aplicaciones SaaS sin necesidad de copiar y pegar valores de configuración específicos de cada protocolo.

Express Configuration ofrece estas ventajas a los administradores de Okta y a los desarrolladores de aplicaciones SaaS:

* Reduce el tiempo necesario para configurar una instancia de aplicación al automatizar el intercambio de información de configuración entre Okta y Auth0.
* Utiliza flujos de consentimiento de OAuth 2.0 para compartir de forma segura y autorizada datos confidenciales de configuración, lo que reduce posibles errores relacionados con las credenciales y los ajustes de configuración.
* Simplifica y estandariza el proceso de despliegue de integraciones. El workflow automatizado permite realizar despliegues coherentes y repetibles de integraciones de aplicaciones en múltiples clientes o entornos, lo que favorece un ecosistema de aplicaciones escalable y reduce la posibilidad de errores humanos.
* Elimina la complejidad de la configuración manual, lo que permite a los administradores de clientes de Okta añadir rápidamente instancias de integraciones OIN habilitadas con Auth0.

<div id="how-it-works">
  ## Cómo funciona
</div>

La API de Express Configuration permite que las aplicaciones de Auth0 publicadas en la OIN para los clientes usen Express Configuration con una [conexión de Okta](/docs/es/authenticate/identity-providers/enterprise-identity-providers/okta). Express Configuration es compatible con OpenID Connect, SCIM y Universal Logout dentro de una organización de Auth0.

Revise el flujo de trabajo de Express Configuration para una aplicación de Auth0 en la OIN:

<Frame>![flujo de trabajo de Express Configuration para una aplicación de Auth0 en la OIN.](/docs/images/cdy7uua7fh8z/express-config-workflow.png)</Frame>

* Un administrador de Okta inicia sesión en el portal de Okta y selecciona en la OIN la aplicación habilitada para Express Configuration.
* El administrador de Okta va a la sección **Sign On** y selecciona **Express Configure SSO &amp; UL**. Esto lo redirige a una pantalla de [Auth0 Universal Login](/docs/es/authenticate/login/auth0-universal-login/universal-login-vs-classic-login/universal-experience#universal-login-experience).

<Frame>![Consola de administrador de Okta > Sign On > Express Configuration](/docs/images/cdy7uua7fh8z/okta-express-config-app.png)</Frame>

* El administrador de Okta introduce las credenciales de un usuario de la aplicación que tiene permiso para realizar la Express Configuration. En Auth0, esto se refiere a un usuario que es miembro de una [organización](/docs/es/manage-users/organizations) y está autorizado para realizar la Express Configuration mediante un [rol organizacional](/docs/es/manage-users/organizations/configure-organizations/add-member-roles) u otro método de autorización.

<Frame>![Inicio de sesión en Auth0 Organization](/docs/images/cdy7uua7fh8z/okta-express-config-auth0-login.png)</Frame>

* Después de autenticarse, Auth0 solicita el consentimiento del administrador de Okta.

<Frame>![consentimiento de Auth0](/docs/images/cdy7uua7fh8z/express-config-auth0-consent.png)</Frame>

* Después de otorgar su consentimiento, Okta usa la API de Express Configuration para configurar automáticamente una [conexión de Okta](/docs/es/authenticate/identity-providers/enterprise-identity-providers/okta) dentro de la organización de Auth0 a la que pertenece el administrador de Okta.

* Después, el administrador de Okta puede asignar usuarios a la instancia de la aplicación y comprobar de inmediato que el inicio de sesión único funciona.

Los desarrolladores de Auth0 también pueden configurar su integración de OIN para permitir la Express Configuration de SCIM y Universal Logout, ambos recomendados:

* Cuando SCIM está habilitado, los administradores de Okta pueden configurar SCIM yendo a la sección **Provisioning** de los detalles de la aplicación y seleccionando **Express Configure SCIM**.
* Cuando Universal Logout está habilitado, se configurará automáticamente como parte de la integración de OpenID Connect.

<div id="prerequisites">
  ## Requisitos previos
</div>

Para publicar una aplicación en la OIN con Express Configuration habilitada, debe contar con lo siguiente:

* Una [Okta Integrator Free Plan org](https://developer.okta.com/signup/), con acceso al rol Super Admin o a los roles App Admin y Org Admin.
* Una [suscripción de Auth0](https://auth0.com/pricing) que permita usar el tipo de conexión Okta y la característica Organizations para tantos clientes como necesite.
* Una aplicación SaaS integrada con Auth0 mediante una arquitectura multiorganización.
  * Esto incluye registrar su aplicación como una [Regular Web Application](/docs/es/get-started/auth0-overview/create-applications/regular-web-apps) o una [Aplicación de página única](/docs/es/get-started/auth0-overview/create-applications/single-page-web-apps) en Auth0, y permitir que cada uno de sus clientes inicie sesión en esa aplicación con sus propios [proveedores de identidad](/docs/es/authenticate/enterprise-connections).
* Debe haberse implementado, o poder implementarse, una [Auth0 Organization](/docs/es/manage-users/organizations) para cada cliente que use Express Configuration. Organizations y los roles de la organización se usan para autorizar a usuarios específicos a realizar la configuración de Express Configuration y crear conexiones de Okta solo dentro de las organizaciones a las que pertenecen.
* Su tenant de Auth0 debe tener deshabilitado el ajuste de tenant [Enable Application Connections](/docs/es/get-started/tenant-settings#recommended-settings).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  **Consejo**: Para ver una aplicación de ejemplo que implementa una arquitectura multi-tenant que incluye Auth0 Organizations y roles de la organización, consulte la [aplicación de referencia SaaStart](https://auth0.com/blog/speed-up-your-customer-identity-journey-with-auth0-saastart/).
</Callout>

<div id="configure-your-application-for-express-configuration">
  ## Configura tu aplicación para Express Configuration
</div>

Auth0 Dashboard guía a un desarrollador de Auth0 para habilitar Express Configuration en su aplicación y publicarla en la OIN.

Se requiere un [rol de Auth0 Admin](/docs/es/get-started/manage-dashboard-access/feature-access-by-role) en un tenant de Auth0 de producción para completar el proceso de configuración y publicación de extremo a extremo.
<Frame>![OIN en Auth0 Dashboard](/docs/images/cdy7uua7fh8z/dashboard-auth0-oin.png)</Frame>
Configura los siguientes componentes de Auth0 para implementar Express Configuration en tu tenant:

* Una aplicación registrada con Initiate Login URI Template
* Un [perfil de conexión (CP)](/docs/es/authenticate/enterprise-connections/connection-profile)
* Un [perfil de atributos de usuario](/docs/es/authenticate/enterprise-connections/user-attribute-profile)
* Configuración de inicio de sesión de la organización
* Configuración de inicio de sesión y consentimiento del administrador

<div id="register-an-application-with-initiate-login-uri-template">
  ### Registrar una aplicación con Initiate Login URI Template
</div>

1. Registra tu aplicación como una [Regular Web Application](/docs/es/get-started/auth0-overview/create-applications/regular-web-apps) o una [Aplicación de página única](/docs/es/get-started/auth0-overview/create-applications/single-page-web-apps) en Auth0 Dashboard.
2. Una vez registrada, selecciona la aplicación que creaste y ve a la pestaña **Okta Integration Network** para iniciar el asistente de configuración de Express Configuration.
3. Selecciona **Get Started**.
4. Revisa los requisitos previos y selecciona **Continuar**.
5. Registra un **Initiate Login URI Template**. Esta plantilla implementa un endpoint en tu aplicación para redirigir automáticamente a los usuarios finales al endpoint `/authorize` de Auth0 para la autenticación. Para ver un ejemplo de un endpoint de login, consulta la ruta `/login` en el [Express SDK Quickstart](/docs/es/quickstart/webapp/express/interactive).
   * Después de que Express Configuration se inicia para una instancia de la aplicación, Okta usa el **Initiate Login URI Template** para abrir la aplicación desde el [end-user dashboard](https://help.okta.com/en-us/content/topics/settings/new-dashboard-overview.htm).
   * (Opcional) Establece `organization_name`, `organization_id,` y `connection_name` para el endpoint `/authorize` de Auth0 a fin de identificar la organización o conexión que se debe usar. Okta reemplaza dinámicamente estas variables cuando la URL se abre desde el end-user dashboard. Ejemplo: `https://{organization_name}.your-app.com?connection={connection_name}`.

<Frame>![Dashboard>Applications>OIN>URI-template](/docs/images/cdy7uua7fh8z/Dashboard>Applications>OIN>URI-template.png)</Frame>

**Ejemplos de Initiate Login URI:**

`https://your-app.com/login` <br />
`https://your-app.com/login?connection={connection_name}` <br />
`https://{organization_name}.your-app.com?connection={connection_name}` <br />

<div id="connection-profile">
  #### Perfil de conexión
</div>

En Initiate Login URI Template, puedes agregar un **perfil de conexión**. El [Perfil de conexión](/docs/es/authenticate/enterprise-connections/connection-profile) permite a Auth0 especificar cómo deben configurarse los ajustes privados de tus conexiones cuando se crean con Express Configuration, incluidos los ajustes que determinan cómo interactúa la conexión con las funciones Universal Login y Organizations de Auth0:

* Opciones sobre cómo debe crearse el nombre de la conexión en Auth0
  * Opciones para usar SCIM y/o Universal Logout con la conexión (recomendado)
  * Opciones para permitir que los usuarios finales de la conexión se conviertan automáticamente en miembros de la organización del administrador que otorgó su consentimiento
  * Opciones para la configuración **Mostrar como botón** de la conexión en la página de Universal Login de Auth0

Si no tienes configurado un perfil de conexión, Auth0 proporciona un perfil de conexión predeterminado con ajustes comunes y recomendados para las conexiones configuradas con Express.

Para saber cómo personalizar el CP predeterminado, consulta [Perfil de conexión](/docs/es/authenticate/enterprise-connections/connection-profile).

<div id="user-attribute-profile">
  #### perfil de atributos de usuario
</div>

En Initiate Login URI Template, puedes agregar un **perfil de atributos de usuario**. El [perfil de atributos de usuario (UAP)](/docs/es/authenticate/enterprise-connections/user-attribute-profile) permite a los desarrolladores de Auth0 definir, gestionar y mapear atributos de usuario de forma uniforme en los distintos protocolos compatibles con Auth0. Cuando se usa con Express Configuration, el perfil de atributos de usuario permite a los desarrolladores personalizar los mapeos de atributos de OpenID Connect y SCIM que se escribirán en la conexión de Okta generada por Express Configuration.

Si no tienes configurado un perfil de atributos de usuario, Auth0 proporciona un perfil de atributos de usuario predeterminado con mapeos comunes y recomendados para todos los protocolos, incluidos OpenID Connect y SCIM, que utiliza el tipo de conexión de Okta.

Para saber cómo personalizar el UAP predeterminado, consulta [perfil de atributos de usuario](/docs/es/authenticate/enterprise-connections/user-attribute-profile).

<div id="organization-login-settings">
  ### Configuración de inicio de sesión de la organización
</div>

Auth0 Organizations es necesario para autorizar a usuarios específicos de la organización a crear conexiones aisladas, pero no es necesario modificar el flujo de inicio de sesión actual de tu aplicación para admitir el [flujo de inicio de sesión basado en organización para usuarios empresariales](/docs/es/manage-users/organizations/configure-organizations/define-organization-behavior).

* Si tu aplicación usa actualmente una [experiencia de inicio de sesión Identifier-First](/docs/es/authenticate/login/auth0-universal-login/identifier-first) y Home Realm Discovery en tus aplicaciones, consulta la sección [Habilitar Home Realm Discovery](/docs/es/authenticate/identity-providers/enterprise-identity-providers/okta/express-configuration#enable-home-realm-discovery) para obtener información sobre cómo habilitar Home Realm Discovery.
* Si deseas habilitar varias aplicaciones en `enabled_clients` para usarlas con tus conexiones de Express Configuration, consulta Habilitar varias aplicaciones de Auth0 en una sola integración.

Si la configuración actual de tu aplicación no usa un flujo de inicio de sesión basado en organización, puedes seleccionar la opción **Habilitar esta aplicación para las conexiones creadas**. Al habilitarla, cada Express Configuration de una conexión de Okta se asocia directamente con tu aplicación. Usa la configuración obligatoria `enabled_clients` en la conexión.

<div id="administrator-login-and-consent-settings">
  ### Configuración del inicio de sesión y el consentimiento del administrador
</div>

Cuando Express Configuration está habilitado, Auth0 crea un ID adicional de aplicación cliente que la OIN usa durante el flujo de consentimiento del administrador entre Okta y Auth0. Okta crea un cliente de la OIN para cada integración de aplicación distinta publicada en la OIN.

Configure los ajustes de inicio de sesión y consentimiento del administrador en la aplicación que quiera publicar en la OIN, en **Configure Integration Profile**.

1. Vaya a [Auth0 Dashboard &gt; Applications](https://manage.auth0.com/dashboard/#/applications).
2. Elija la aplicación que quiere publicar en la OIN.
3. Seleccione Okta Integration Network.
4. Verifique que cumple con los requisitos previos necesarios y seleccione **Continuar**.
5. En la sección **Configure Integration Profile**, configure las opciones en **Admin Settings**.

<Frame>![Configuración de administrador de Express Configuration](/docs/images/cdy7uua7fh8z/Express-config-admin-settings.png)</Frame>

* **Admin Login Domain**: Dominio del tenant de Auth0 al que Okta redirige como parte del flujo de consentimiento en un navegador web. Use el dominio personalizado si tiene uno configurado en Auth0. Para obtener más información, consulte [Custom Domains](docs/customize/custom-domains).
  * **Display Name of the OIN Express Configuration Application**: Nombre de la aplicación cliente de la OIN que se muestra en el cuadro de diálogo de consentimiento.

  * **Admin Login Flow**: Define el flujo de [inicio de sesión de la organización](/docs/es/manage-users/organizations/login-flows-for-organizations) para la aplicación cliente de la OIN y se usa cuando un administrador de Okta ejecuta Express Configuration desde la consola de Okta. Seleccione una de estas opciones:

    * **Prompt for Organization**: Primero se les pide a los administradores que seleccionen su organización y luego se les presenta la experiencia de inicio de sesión de su organización de Auth0. Los administradores pueden iniciar sesión con una [conexión preexistente](/docs/es/manage-users/organizations/configure-organizations/enable-connections) configurada en esa organización.

    * **Prompt for Credentials**: Primero se les pide a los administradores que proporcionen sus credenciales de inicio de sesión. Cuando se selecciona esta opción, aparecerá un botón que le permitirá seleccionar las conexiones que contienen a los usuarios administradores. Puede tratarse de una conexión de base de datos compartida, una conexión de correo electrónico passwordless u otra conexión que pueda asociarse con todas las Auth0 Organizations.

  * **Admin Role**:  Para realizar Express Configuration, el administrador debe tener asignados los permisos adecuados. Lea [Asignar permisos de Express Configuration a los usuarios](/docs/es/authenticate/identity-providers/enterprise-identity-providers/okta/express-configuration#assign-permissions-to-users) para obtener instrucciones sobre cómo autorizar a los administradores de la forma que mejor se adapte a su despliegue actual de Auth0.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Para ofrecer la mejor experiencia de consentimiento, establezca la marca `use_scope_descriptions_for_consent` de su tenant en `true`, como se describe en [Customize Consent Prompts](/docs/es/customize/login-pages/customize-consent-prompts). Más adelante, cuando pruebe y verifique su integración, tendrá la oportunidad de ver su solicitud de consentimiento.
</Callout>

<div id="assign-permissions-to-users">
  ## Asignar permisos a los usuarios
</div>

En Okta, los administradores deben tener una cuenta de usuario de la aplicación con los permisos necesarios para realizar la Express Configuration de una aplicación SaaS con tecnología de Auth0.

En Auth0, se puede conceder este permiso a cualquier usuario de la aplicación, siempre que sea miembro de una [Auth0 Organization](/docs/es/manage-users/organizations) en la que se creen las conexiones configuradas con Express, entre ellas:

* Un usuario en una conexión de base de datos dedicada creada exclusivamente para una sola Organization
* Un usuario en una conexión de base de datos compartida que sea miembro de una o más Organizations
* Un usuario con una conexión passwordless por correo electrónico que sea miembro de una o más Organizations
* Un usuario en una conexión social o en una conexión empresarial existente que sea miembro de una o más Organizations

Una vez cumplidas estas condiciones, Auth0 ofrece formas flexibles de asignar permisos de Express Configuration y permite a los desarrolladores elegir el método más adecuado para su implementación actual de Auth0:

| **Método**                                                                                                                                                                                                                                                | **Recomendado para...**                                                                                                                                                                                                                                                                                                                |
| --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Asignar permisos de Express Configuration a un rol de usuario existente                                                                                                                                                                                   | Clientes con un [rol](/docs/es/manage-users/access-control/configure-core-rbac/roles) de usuario de aplicación ya existente en su tenant de Auth0.                                                                                                                                                                                        |
| Asignar un nuevo rol a los usuarios de la aplicación que necesiten permisos de Express Configuration, como se ejemplifica en la [aplicación de referencia SaaStart](https://auth0.com/blog/speed-up-your-customer-identity-journey-with-auth0-saastart/). | Clientes que quieren usar la funcionalidad de [rol](/docs/es/manage-users/access-control/configure-core-rbac/roles) de Auth0 para conceder permisos de Express Configuration a usuarios individuales. Esto puede hacerse con Auth0 Dashboard o la Management API.                                                                         |
| Usar una Post-Login Action para asignar permisos dinámicamente mediante control de acceso basado en atributos                                                                                                                                             | Clientes que actualmente no usan la funcionalidad de [rol](/docs/es/manage-users/access-control/configure-core-rbac/roles) de Auth0 y prefieren usar una Post-Login Action para asignar permisos dinámicamente según los atributos del usuario, en lugar de usar Auth0 Dashboard o hacer llamadas a la Management API para asignar roles. |

Para obtener consejos sobre cómo aprovisionar cuentas de administrador para usar Express Configuration, lee [Customer Enablement](/docs/es/authenticate/identity-providers/enterprise-identity-providers/okta/express-configuration#customer-enablement).

<div id="assign-permissions-to-a-pre-existing-application-user-role">
  ### Asignar permisos a un rol de usuario de aplicación existente
</div>

Utiliza este método para asignar permisos de Express Configuration si usas la [función RBAC de Auth0](/docs/es/manage-users/access-control/configure-core-rbac/roles) para los roles de la aplicación y los permisos de la API, y ya tienes uno o más roles que corresponden a un usuario administrador y que se asignarían al administrador de TI encargado de desplegar la aplicación.

Permisos de API requeridos:

| **Nombre del permiso**   | **Identificador de API (Servidor de recursos)** |
| ------------------------ | ----------------------------------------------- |
| `express_configure:sso`  | `urn:auth0:express-configure`                   |
| `express_configure:scim` | `urn:auth0:express-configure`                   |

Para obtener más información sobre cómo asignar roles en Auth0 Dashboard y Management API, consulta [Agregar permisos a roles](/docs/es/manage-users/access-control/configure-core-rbac/roles/add-permissions-to-roles).

<div id="assign-a-new-role-to-application-users">
  ### Asignar un nuevo rol a los usuarios de la aplicación
</div>

Usa este método para asignar permisos de Express Configuration si utilizas la [función RBAC de Auth0](/docs/es/manage-users/access-control/configure-core-rbac/roles) para los roles de la aplicación y los permisos de la API, pero no tienes un rol que represente a un usuario administrador, como un administrador de TI, al que se le asignaría desplegar la aplicación.

Este método también es adecuado para los desarrolladores de Auth0 que actualmente no utilizan la [función RBAC de Auth0](/docs/es/manage-users/access-control/configure-core-rbac/roles), pero quieren aprovecharla para obtener un control granular sobre qué usuarios tienen acceso, ya sea mediante el Auth0 Dashboard o la Management API.

<div id="create-a-role">
  #### Crear un rol
</div>

Usa Auth0 Dashboard, Management API o [Auth0 CLI](https://auth0.github.io/auth0-cli/auth0_roles_create.html) para [crear roles](/docs/es/manage-users/access-control/configure-core-rbac/roles/create-roles). Este ejemplo usa Auth0 CLI:

```bash
auth0 roles create \
  --name "EXPRESS_CONFIGURE_ADMIN_ROLE" \
  --description "Administrator role for Express Configuration"
```

<div id="assign-permissions-to-the-role">
  #### Asignar permisos al rol
</div>

Asigna los permisos `express_configuration:sso` y `express_configuration:scim` al rol especificado. Sustituye `$ROLE_ID` por el ID del rol al que quieres asignar permisos.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Al usar este método, tendrás que actualizar tu proceso de incorporación de clientes para asignar este rol a cualquier usuario nuevo de la organización que necesite permisos de Express Configuration.
</Callout>

```bash
auth0 roles permissions add "$ROLE_ID" \
  --api-id "urn:auth0:express-configure" \
  --permissions "express_configure:sso"
```

```bash
auth0 roles permissions add "$ROLE_ID" \
  --api-id "urn:auth0:express-configure" \
  --permissions "express_configure:scim"
```

Para obtener más información sobre cómo asignar roles a usuarios existentes de la organización, consulta [Agregar roles de miembros](/docs/es/manage-users/organizations/configure-organizations/add-member-roles).

<div id="assign-permissions-based-on-user-attributes-using-a-post-login-action">
  ### Asigna permisos según los atributos del usuario mediante una Action de Post Login
</div>

Usa este método para asignar permisos en Express Configuration si estás usando una Auth0 [Action de Post Login](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger) para asignar permisos dinámicamente según los atributos del usuario, en lugar de usar Role-Based Access Control (RBAC), Management API o Auth0 Dashboard para asignar roles de usuario.

Esto es adecuado para clientes de Auth0 que han implementado Auth0 con un modelo de autorización basado en atributos que usa permisos personalizados en la [metadata de Auth0](/docs/es/manage-users/user-accounts/metadata).

**Ejemplo** <br />
El siguiente ejemplo usa una Action de Post Login para asignar permisos según el valor de un atributo personalizado existente del usuario, `user.app_metadata.is_admin`.

```bash
/**
* Asigna permisos de Express Configuration basados en lógica personalizada en lugar de asignación de roles
*/
exports.onExecutePostLogin = async (event, api) => {


 //verificar si la solicitud es para un token de acceso a la API de EC
 if (event.resource_server && event.resource_server.identifier === "urn:auth0:express-configure") {


   //agregar permisos según una condición personalizada
   if (event.user.app_metadata && event.user.app_metadata.is_admin === true) {
     api.accessToken.addScope("express_configure:sso");
     api.accessToken.addScope("express_configure:scim");
   }
   //de lo contrario, denegar el acceso si se solicita el token de acceso de EC
   else {
     api.access.deny('Access denied');
   }
 }
};
```

<div id="enable-home-realm-discovery">
  ## Habilitar Home Realm Discovery
</div>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Antes de ejecutar Express Configuration, debe recopilar y verificar las direcciones de correo electrónico de los clientes como parte del proceso de incorporación para habilitar HRD.
</Callout>

Puede usar Express Configuration con Auth0 Actions si su aplicación se basa en [Identifier-First login experience](/docs/es/authenticate/login/auth0-universal-login/identifier-first) y utiliza Home Realm Discovery (HRD) para asociar a los usuarios con sus proveedores de identidad (IdP) en función de su dirección de correo electrónico.

Las direcciones de correo electrónico deben almacenarse en una ubicación a la que las Actions de Post Login puedan acceder durante el flujo de trabajo de Express Configuration, por ejemplo:

* Puede almacenar uno o más dominios de correo electrónico en la [metadata de la organización](/docs/es/manage-users/organizations/configure-organizations/create-organizations) de la Auth0 Organization del cliente.
* Su Action de Post Login puede hacer una llamada a la API para recuperar dominios verificados desde cualquier sistema de su entorno donde estén almacenados.

Cuando el usuario administrador inicia Express Configuration en el portal de Okta y se autentica, estas Actions agregan los dominios de correo electrónico al token que recibe Okta, que luego extrae y utiliza para configurar la conexión de Okta con la configuración de HRD correcta.

**Ejemplo 1** <br />
En este ejemplo, los dominios verificados se almacenan en la metadata de la conexión de la Auth0 Organization, lo que requiere una cadena delimitada por comas con uno o más dominios de correo electrónico, almacenada en la metadata de la organización bajo una clave llamada `domains`.

Valores de ejemplo: `test.com`,`test2.com`

```javascript
exports.onExecutePostLogin = async (event, api) => {
if (event?.resource_server?.identifier === "urn:auth0:express-configure") {
  if (event.organization && event.organization.metadata && event.organization.metadata.domains)
  {
    var domain_aliases = event.organization.metadata.domains.replace(/ /g,'').split(',');
    var express_configuration = {
      "domain_aliases": domain_aliases
     };
     api.accessToken.setCustomClaim("express_configuration", express_configuration);
  }
}
};
```

**Ejemplo 2** <br />
En este ejemplo, la Action de Post Login hace una llamada a la API para obtener los dominios verificados de un sistema de almacenamiento de tu entorno.

```javascript
/**
*/
exports.onExecutePostLogin = async (event, api) => {
 if (event?.resource_server?.identifier === "urn:auth0:express-configure") {
    const axios = require("axios");
    // configura aquí tu llamada a la API personalizada 
    const domains = await axios.get("https://example.org/endpoint");
     var express_configuration = {
     "domain_aliases": domains
      };
      api.accessToken.setCustomClaim("express_configuration", express_configuration);
  
 }
};
```

<div id="maintain-admin-account-matching-across-connections">
  ### Mantener la correspondencia de cuentas de administrador entre conexiones
</div>

Auth0 permite aprovisionar cuentas de usuario con la misma dirección de correo electrónico en distintas conexiones. Un usuario final puede tener una conexión de base de datos, una cuenta social o passwordless con su dirección de correo electrónico del trabajo, y también tener esa misma dirección de correo electrónico en su cuenta federada de Okta.

Al usar la opción **Prompt for Credentials** en el [flujo de inicio de sesión y consentimiento del administrador](/docs/es/authenticate/identity-providers/enterprise-identity-providers/okta/express-configuration#provision-admin-accounts), Home Realm Discovery empieza a asociar a los usuarios con un sufijo de dominio específico con la nueva conexión de Okta, en lugar de hacerlo con otros tipos de conexión, después de que se agregue a la organización. Para obtener más información sobre este comportamiento de inicio de sesión de la organización, consulta [Identifier First Authentication with prompt for credentials](/docs/es/manage-users/organizations/login-flows-for-organizations#identifier-first-authentication-with-prompt-for-credentials).

Este comportamiento solo surte efecto cuando los clientes ejecutan Express Configuration por segunda vez después de que caduque su sesión inicial de usuario administrador de Auth0. Si el identificador de la cuenta de administrador es una dirección de correo electrónico que coincide con la nueva conexión de Okta, el administrador será redirigido allí.

Este comportamiento se puede gestionar o evitar con cualquiera de los siguientes métodos:

* Usa la opción **Prompt for Organization** en el [flujo de inicio de sesión y consentimiento del administrador](/docs/es/authenticate/identity-providers/enterprise-identity-providers/okta/express-configuration#provision-admin-accounts).
  * Habilita los permisos de Express Configuration para la nueva cuenta de Okta que contiene la dirección de correo electrónico verificada correspondiente después de que se aprovisione.
  * Si solo usas Auth0 Organizations para el flujo de consentimiento del administrador de Express Configuration y no en la experiencia de tu aplicación, puedes resolver el problema deshabilitando la propiedad `enable_organization` descrita en [Configure SaaS Application Properties](/docs/es/authenticate/identity-providers/enterprise-identity-providers/okta/express-configuration#configure-saas-application-properties).
  * Esto no supone un problema si el identificador del usuario administrador no es una dirección de correo electrónico, como un nombre de usuario en una conexión de base de datos.

<div id="enable-multiple-applications-under-a-single-integration">
  ## Habilitar varias aplicaciones en una sola integración
</div>

Si quieres que los usuarios finales de una sola Express Configuration en una conexión de Okta puedan acceder a varias aplicaciones de tu tenant, puedes establecer la propiedad `linked_clients` en tu aplicación web registrada.

Para modificar esta configuración, usa la Auth0 Management API para [obtener tu aplicación web registrada](https://auth0.com/docs/api/management/v2/clients/get-clients-by-id). Reemplaza `{yourAppId}` por el ID de cliente de tu aplicación registrada y `{yourAccessToken}` por un token de acceso de la Management API v2. Para saber cómo obtener un token de acceso para usar la Management API, consulta [Management API Access Tokens](/docs/es/secure/tokens/access-tokens/management-api-access-tokens).

```curl
curl --request GET \
  --url 'https://{yourDomain}/api/v2/clients/{yourAppId}' \
  --header 'authorization: Bearer {yourAccessToken}'
```

Extrae la propiedad `express_configuration` de la respuesta, añade un arreglo de IDs de aplicaciones adicionales a la propiedad `linked_clients` y [actualiza tu aplicación web registrada](https://auth0.com/docs/api/management/v2/clients/patch-clients-by-id):

```curl
curl --request PATCH \
  --url 'https://{yourDomain}/api/v2/clients/{yourAppId}' \
  --data '{"express_configuration": 
  {"admin_login_domain":"TENANT.auth0.com",
    "connection_profile_id":"cop_xxxxxxxxxxxxxxx",
    "enable_client":true,
    "enable_organization":true,
    "initiate_login_uri_template":"https://example.org",
    "okta_oin_client_id":"LDiGbUeiAYjRB5a4yOGfBvxxxxxxxxxxx",
    "user_attribute_profile_id":"uap_1ctMVQUg8jxxxxxxxxxxxx",   

     "linked_clients": [ 
        { "client_id": "KJM86F2susguvsasSeAsIxxxxxxxxxxx" }, 
	  { "client_id": "FIXwZCf5iUElvqy6eidlfxxxxxxxxxxx" }
      ] 
     }
  }' \
  --header 'cache-control: no-cache' \
  --header 'content-type: application/json' \
  --header 'authorization: Bearer {yourAccessToken}'
```

Cuando se ejecute Express Configuration, todos estos ID de aplicación de la propiedad linked&#95;clients se añadirán a la propiedad `enabled_clients` de la conexión de Okta.

<div id="customer-enablement">
  ## Habilitación del cliente
</div>

Cuando tu aplicación se publique en el OIN, puedes actualizar la documentación de tu producto para indicar que es compatible con Express Configuration with Okta. La documentación de tu producto debe indicar qué roles o tipos de usuario tienen permiso para realizar la Express Configuration.

Si tu aplicación ya utiliza Auth0 Organizations con roles de administrador de organización, como se muestra en la [aplicación de referencia SaaStart](https://auth0.com/blog/speed-up-your-customer-identity-journey-with-auth0-saastart/), puedes agregar los permisos de Express Configuration a ese rol.

Si aún no has desplegado Auth0 Organizations para tus clientes, o tu aplicación no incluye usuarios administrativos, consulta el flujo de ejemplo para la incorporación de clientes.

<div id="example-customer-enablement-flow">
  ### Flujo de ejemplo para la habilitación de clientes
</div>

En tu proceso de incorporación de clientes, debes recopilar:

* El nombre de la organización del cliente
* La dirección de correo electrónico del administrador del cliente que debe tener permisos para realizar la Express Configuration
* Los dominios de correo electrónico verificados que el IdP del cliente emite para los usuarios, incluidos los dominios del administrador

Recopila esta información manualmente o a través de la experiencia de registro o incorporación de tus clientes. Usa esta información para crear una Auth0 Organization y una cuenta de usuario administrador para este cliente:

1. Crea una Auth0 Organization
2. Crea la cuenta de usuario administrador en una conexión de Auth0, como Database o email passwordless.
3. Asocia la conexión con la Auth0 Organization. Desactiva **Membership On Authentication**.
4. Agrega la cuenta de usuario administrador como miembro de la Auth0 Organization.
5. Asigna al usuario administrador un rol organizacional con permisos para realizar la Express Configuration.

<div id="auth0-cli-example">
  #### Ejemplo con Auth0 CLI
</div>

**Inicializa Auth0 CLI y autentícate**
Autentícate en Auth0 con [Auth0 CLI](https://auth0.github.io/auth0-cli/auth0_roles_create.html).

```bash
auth0 login --scopes "create:users,create:organizations,create:organization_members,create:organization_member_roles,create:organization_connections"
```

**Crea una Auth0 Organization**
Introduce el nombre de la organización y un nombre corto sin espacios. Si has recopilado varios sufijos de correo electrónico, añádelos como metadatos de la organización.

```bash
auth0 orgs create --json \
--display "organization_display_name" \
--name "organization_short_name" \
--metadata 'domains=["domain1.com", "domain2.com"]'
```

Recoge el `org_id` devuelto.

**Crea la cuenta de administrador de la organización**
En este ejemplo, creas un usuario en una conexión passwordless por correo electrónico, para lo que se requiere el tipo de conexión [`email`](/docs/es/authenticate/passwordless/authentication-methods/email-otp). El comando requiere introducir la dirección de correo electrónico del usuario al que autorizas para realizar la Express Configuration.

```bash
auth0 api users \
--data '{"email":"user@example.com","connection":"email", "email_verified":true}'
```

<Card title="Consejo">
  Si va a crear esta organización para proporcionar al equipo de operaciones de Okta OIN una cuenta de prueba con nombre de usuario y contraseña, cree la cuenta de prueba en una conexión de base de datos compartida o dedicada:

  ```bash
  auth0 api users \
  --data '{"email":"user@example.com","connection":"db_connection_name_here", "password": "add_a_long_password_here"}'
  ```
</Card>

Anote el `user_id` devuelto.

**Asocie la conexión con la organización**

Obtenga el ID de la conexión:

```bash
auth0 api get "connections" -q "name=connection_name_here"
```

Asocia el ID de la conexión con el ID de la organización, `org_id`.

```bash
auth0 api post "organizations/org_id_here/enabled_connections" \
--data '{ "connection_id": "connection_id_here" };
```

**Añade al usuario administrador como miembro de la Organización**
Necesitas `org_id` y `user_id`.

```bash
auth0 api post "organizations/org_id_here/members" \
--data '{ "members": ["user_id__here"] }'

```

**Asigne el rol de organización con permisos de Express Configuration**
Recupere el ID del rol con permisos de Express Configuration, según se indica en [Asignar permisos de Express Configuration a los usuarios](/docs/es/authenticate/identity-providers/enterprise-identity-providers/okta/express-configuration#assign-permissions-to-users).

```bash
auth0 api get "roles" -q "name_filter=role_name_here"
```

Asigna el ID del role a `user_id` y `org_id`.

```bash
auth0 api post "organizations/org_id_here/members/user_id_here/roles" --data '{ "roles": ["role_id_here"] }'
```

Una vez hecho esto, su cliente puede usar la cuenta de administrador de la organización para realizar la Express Configuration en su organización de Okta con su [integración de OIN](/docs/es/authenticate/identity-providers/enterprise-identity-providers/okta/express-configuration#publish-your-integration-to-the-oin).

<div id="recommendations-to-provision-admin-accounts">
  ### Recomendaciones para aprovisionar cuentas de administrador
</div>

Tanto si usa cuentas de usuario existentes como si aprovisiona otras nuevas para habilitar Express Configuration, siga las recomendaciones que se indican a continuación.

<div id="use-work-email">
  #### Usa el correo electrónico laboral
</div>

Tanto si la cuenta de administrador no federada es una cuenta de usuario de base de datos, passwordless por correo electrónico o social, lo mejor es que el valor del atributo `email` coincida con la dirección de correo electrónico laboral del usuario que se aprovisionará con su cuenta de usuario de Okta Enterprise (por ejemplo: `john@mycompany.com`).

Auth0 también recomienda verificar estas direcciones de correo electrónico mediante un [flujo de verificación del correo electrónico](/docs/es/manage-users/user-accounts/verify-emails).

<div id="use-multi-factor-authentication">
  #### Usar autenticación multifactor
</div>

Puedes añadir una capa adicional de seguridad al flujo de Express Configuration exigiendo autenticación multifactor en cada inicio de sesión.

La [Action de Post Login](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger) de ejemplo muestra cómo puedes solicitar de forma condicional factores como [WebAuthn con biometría del dispositivo](/docs/es/secure/multi-factor-authentication/fido-authentication-with-webauthn/configure-webauthn-device-biometrics-for-mfa) y [contraseña de un solo uso enviada por correo electrónico](/docs/es/secure/multi-factor-authentication/multi-factor-authentication-factors/configure-email-notifications-for-mfa) cada vez que se ejecuta Express Configuration.

La Action de ejemplo requiere [habilitar estos factores](/docs/es/secure/multi-factor-authentication/enable-mfa#enable-mfa-in-the-auth0-dashboard) en tu tenant de Auth0 y seleccionar la opción **Personalizar factores de MFA mediante Actions**.

```javascript
exports.onExecutePostLogin = async (event, api) => {


if (event.resource_server && event.resource_server.identifier === "urn:auth0:express-configure") {


   api.multifactor.enable('any');
  api.authentication.challengeWith({ type: 'email' });


  if (!event.user.enrolledFactors.some(m => m.type === 'webauthn-platform')) {
     api.authentication.enrollWith({type: 'webauthn-platform'});
   } else {
     api.authentication.challengeWith({type: 'webauthn-platform'});
  }
 }
}
```

<div id="monitor-express-configuration-usage">
  ### Monitorear el uso de Express Configuration
</div>

El uso de Express Configuration puede observarse en los [registros del tenant de Auth0](/docs/es/deploy-monitor/logs) filtrando por el **OIN Client ID** de cada aplicación que hayas configurado. Esto incluye toda la actividad de inicio de sesión de los usuarios administradores, así como todas las operaciones de la API para crear y administrar las conexiones empresariales de Okta.

El OIN Client ID de tu aplicación puede verse en estos lugares:

* En la sección **Applications &gt; [Application] &gt; Okta Integration Network &gt; Create OIN Integration** del Auth0 Dashboard.

  * En la propiedad `express_configuration.okta_oin_client_id` de tu aplicación al verla en la [Auth0 Management API](https://auth0.com/docs/api/management/v2/clients/get-clients-by-id).

Para obtener más información sobre cómo buscar logs con el Client ID, consulta [sintaxis de las consultas de búsqueda en logs](/docs/es/deploy-monitor/logs/log-search-query-syntax).

<div id="limitations">
  ## Limitaciones
</div>

* Solo se puede usar un tenant de Auth0 en una única integración de OIN. Si la misma aplicación se implementa en varias regiones de Auth0, cada región requiere su propia integración de OIN.
* Una integración de OIN en una organización de Okta puede crear una única conexión de Okta en la organización de Auth0 correspondiente. Express Configuration no es compatible con más de una instancia de una aplicación de OIN dentro de una organización de Okta.
* Cada integración única incluida en el OIN crea una conexión de Okta independiente. Si tiene varias aplicaciones que requieren una [experiencia de inicio de sesión Identifier-First](https://auth0.com/docs/authenticate/login/auth0-universal-login/identifier-first) con Home Realm Discovery, le recomendamos publicar [varias aplicaciones en una sola integración](/docs/es/authenticate/identity-providers/enterprise-identity-providers/okta/express-configuration#enable-multiple-applications-under-a-single-integration).
* Al habilitar [varias aplicaciones en una sola integración](/docs/es/authenticate/identity-providers/enterprise-identity-providers/okta/express-configuration#enable-multiple-applications-under-a-single-integration), Okta solo muestra la aplicación principal de OIN en su [dashboard de usuario final](https://help.okta.com/en-us/content/topics/settings/new-dashboard-overview.htm).

<div id="create-the-okta-oin-express-configuration-system-api">

<div id="create-user-attribute-profile">

<div id="create-connection-profile">

<div id="create-oin-express-configuration-client">
  ### Crear cliente de Express Configuration de OIN
</div>

Cree la aplicación de servicio que Okta usará durante Express Configuration para una aplicación determinada. Este cliente debe crearse con las propiedades que se muestran en el ejemplo siguiente.

El atributo `organization_require_behavior` puede personalizarse con uno de los valores que aparecen a continuación, que también se explican en [Configuración de inicio de sesión y consentimiento del administrador](/docs/es/authenticate/identity-providers/enterprise-identity-providers/okta/express-configuration#administrator-login-and-consent-settings).

* `pre_login_prompt`: Solicita la organización del administrador antes de pedir sus credenciales.
  * `post_login_prompt`: Solicita las credenciales del administrador. Para esta opción, use el endpoint `PATCH /api/v2/connections/{id}` para agregar el ID de cliente del cliente OIN a la propiedad `enabled_clients` de las conexiones que contienen las cuentas de administrador.

Ejemplo:

```curl
POST /api/v2/clients
{
    "name": "Okta OIN Express Configuration",
    "app_type": "express_configuration",
    "organization_require_behavior": "pre_login_prompt",
    "client_authentication_methods": {
      "private_key_jwt": {
        "credentials": []
      }
    }
}
```

[Referencia del endpoint](https://auth0.com/docs/api/management/v2/clients/post-clients)

<div id="configure-saas-application-properties">
  ### Configurar las propiedades de la aplicación SaaS
</div>

Establece los valores de configuración de la propiedad `express_configuration` en la aplicación registrada que se publicará en el OIN. Sustituye los valores de los ejemplos por un ID de perfil de conexión válido, un ID de User Attribute Profile, un ID de aplicación cliente de OIN, un URI de inicio de sesión y el dominio del tenant de Auth0 que se usarán para esta integración.

El atributo `linked_clients` corresponde a la configuración descrita en [Habilitar varias aplicaciones en una sola integración](/docs/es/authenticate/identity-providers/enterprise-identity-providers/okta/express-configuration#enable-multiple-applications-under-a-single-integration).

El atributo `enable_client` corresponde a la configuración de `enabled_clients` descrita en [Configuración de inicio de sesión de la organización](/docs/es/authenticate/identity-providers/enterprise-identity-providers/okta/express-configuration#organization-login-settings).

El atributo `enable_organization` normalmente debe establecerse en `true`, a menos que no estés asignando las conexiones creadas a sus organizaciones. La instancia de la aplicación de Okta puede seguir administrando la conexión, pero los usuarios de Okta no pasarán a ser miembros de la Auth0 Organization. Establecerlo en `false` puede ser útil en el caso descrito en [Mantener la coincidencia de cuentas de administrador entre conexiones](/docs/es/authenticate/identity-providers/enterprise-identity-providers/okta/express-configuration#maintain-admin-account-matching-across-connections).

Ejemplo:

```curl
PATCH /api/v2/clients/{id}
{
  "express_configuration": {
      "admin_login_domain": "yourAuth0TenantDomain",
      "connection_profile_id": "yourConnectionProfileId",
      "enable_client": true,
      "enable_organization": true,
      "initiate_login_uri_template": "yourInitiateLoginUriTemplate",
      "okta_oin_client_id": "yourOinClientId",
      "user_attribute_profile_id": "yourConnectionProfileId",
"linked_clients": [ 
    { "client_id": "KJM86F2susguvsasSeAsIxxxxxxxxxxx" }, 
    { "client_id": "FIXwZCf5iUElvqy6eidlfxxxxxxxxxxx" }
 ] 
    }
}
```

[Referencia del endpoint](https://auth0.com/docs/api/management/v2/clients/patch-clients-by-id)
