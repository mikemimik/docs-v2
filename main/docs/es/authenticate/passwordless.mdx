<Tooltip tip="Passwordless: Forma de autenticación que no depende de una contraseña como primer factor." cta="Ver glosario" href="/docs/es/glossary?term=Passwordless">Passwordless</Tooltip>, como método de autenticación, ofrece a los usuarios una experiencia de inicio de sesión fluida y más segura. A medida que avanza la tecnología, los métodos tradicionales de autenticación, como los nombres de usuario y las contraseñas, se vuelven más vulnerables a ciberataques (como phishing o captura de teclas) y a posibles brechas de seguridad.

Con la autenticación passwordless, los usuarios ya no necesitan recordar ni introducir manualmente una contraseña para acceder a una aplicación. En su lugar, pueden usar distintos métodos de autenticación basados en enlaces de acceso temporales y tokens, claves de acceso almacenadas, biometría o cuentas sociales.

Al eliminar la dependencia de las contraseñas y otros códigos memorizados, puede ofrecer a los usuarios una forma más cómoda de acceder a las aplicaciones y, al mismo tiempo, reducir la probabilidad de violaciones de la seguridad de los datos por contraseñas robadas o débiles.

Auth0 actualmente admite los siguientes métodos de autenticación passwordless:

* [Inicio de sesión social](#social-login)
* [Claves de acceso](#passkeys)
* [Autenticación passwordless por SMS](#sms-based-passwordless-authentication)
* [Autenticación passwordless por correo electrónico](#email-based-passwordless-authentication)
* [Biometría (WebAuthn)](#biometrics-webauthn-)

Cada uno de estos métodos ofrece a los usuarios una experiencia de inicio de sesión passwordless diferente. Para obtener más información sobre qué método se adapta mejor a sus necesidades, revise las secciones a continuación.

<div id="passwordless-connections-vs-passwordless-authentication">
  ### Conexiones passwordless vs. autenticación passwordless
</div>

Una conexión passwordless es un tipo de conexión distinto de las conexiones de base de datos, sociales o empresariales. Las conexiones passwordless se usan específicamente para:

* [autenticación passwordless por SMS](#sms-based-passwordless-authentication)
* [autenticación passwordless por correo electrónico](#email-based-passwordless-authentication)

Aunque un usuario de una base de datos de usuarios de Auth0 o de un proveedor social pueda tener la misma dirección de correo electrónico, la identidad asociada a su conexión passwordless es distinta. La [vinculación de cuentas](/docs/es/manage-users/user-accounts/user-account-linking) puede usarse para asociar identidades de conexiones passwordless con identidades de otras conexiones.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  No puedes crear usuarios passwordless desde el Auth0 Dashboard. Créelos directamente desde la [Management API](https://auth0.com/docs/api/management/v2/users/post-users) si Signup está deshabilitado. En el campo Connection, usa email para los usuarios passwordless que utilizan una dirección de correo electrónico y SMS para los usuarios passwordless que utilizan un número de teléfono móvil.
</Callout>

<div id="social-login">
  ## Inicio de sesión social
</div>

El inicio de sesión social es un método de autenticación que permite a los usuarios iniciar sesión en una aplicación con credenciales existentes de un <Tooltip tip="Proveedor de identidad (IdP): servicio que almacena y gestiona identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=identity+provider">proveedor de identidad</Tooltip> social, como Google o Facebook. Como los usuarios suelen tener sus credenciales sociales almacenadas en su navegador o dispositivo, el inicio de sesión social ofrece una experiencia de usuario fluida que requiere una interacción manual mínima con su aplicación.

El flujo de trabajo general del inicio de sesión social es el siguiente:

1. Un usuario llega a la pantalla de inicio de sesión de su aplicación y selecciona la opción **Continuar con [provider]**.
2. El usuario sigue las indicaciones del proveedor seleccionado para autenticarse con sus credenciales sociales existentes.
3. El usuario es redirigido a su aplicación para completar el proceso de inicio de sesión.

Para las aplicaciones web, Auth0 admite múltiples proveedores de identidad social de forma nativa. Visite la sección [Social Login](https://marketplace.auth0.com/categories/social-login) de Auth0 Marketplace para consultar la lista completa de proveedores compatibles. Si un proveedor específico no está disponible, también puede configurar [conexiones personalizadas de inicio de sesión social](/docs/es/authenticate/identity-providers/social-identity-providers/oauth2) a través del <Tooltip tip="Auth0 Dashboard: el producto principal de Auth0 para configurar sus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip>.

Para aplicaciones nativas, Auth0 admite el inicio de sesión social para determinados proveedores:

* [Apple](/docs/es/authenticate/identity-providers/social-identity-providers/apple-native)
* [Facebook](/docs/es/authenticate/identity-providers/social-identity-providers/facebook-native)
* Google (¡próximamente!)

Para estas aplicaciones, Auth0 admite el intercambio de tokens para conexiones sociales, lo que permite la autenticación sin necesidad de redirecciones web.

<div id="passkeys">
  ## Claves de acceso
</div>

Las claves de acceso son una alternativa resistente al phishing a los factores de autenticación tradicionales, basadas en las especificaciones de FIDO® W3C Web Authentication (WebAuthn) y Client to Authenticator Protocol (CTAP).

Auth0 admite las claves de acceso como método de autenticación para conexiones de base de datos y ofrece dos métodos de implementación:

* [Claves de acceso de Universal Login](/docs/es/authenticate/database-connections/passkeys) para aplicaciones web
* [Claves de acceso nativas](/docs/es/authenticate/database-connections/passkeys/native-passkeys-for-mobile-applications) para aplicaciones de Android e iOS

Las claves de acceso reducen la fricción de los métodos de autenticación de un solo dispositivo al permitir que las credenciales se sincronicen entre dispositivos. La autenticación entre dispositivos elimina la necesidad de que los usuarios tengan que registrar de nuevo cada uno de sus dispositivos.

Un ejemplo de flujo de trabajo de claves de acceso para aplicaciones web es el siguiente:

1. Un usuario llega a la página de Universal Login de tu aplicación e introduce su dirección de correo electrónico.
2. Luego, el usuario puede elegir crear una clave de acceso o continuar con la autenticación tradicional.
3. Cuando el usuario selecciona **Crear una nueva clave de acceso**, se inicia el flujo de claves de acceso del navegador y se le solicita que se autentique con las credenciales de su dispositivo, como la biometría.
4. Una vez completada la autenticación local, la clave de acceso se guarda en el dispositivo del usuario y se sincroniza con su proveedor de claves de acceso (como iCloud o Google).

Una vez completado este proceso, el usuario puede autenticarse con la clave de acceso guardada en los siguientes inicios de sesión en tu aplicación.

<div id="sms-based-passwordless-authentication">
  ## Autenticación passwordless basada en SMS
</div>

La autenticación passwordless basada en SMS permite a los usuarios acceder a aplicaciones mediante códigos de un solo uso, conocidos como contraseñas de un solo uso (OTP). Con esta implementación, los OTP se envían por SMS a los números de teléfono de los usuarios.

El flujo de trabajo general de los OTP basados en SMS es el siguiente:

1. Un usuario llega a la pantalla de inicio de sesión de tu aplicación e ingresa su número de teléfono.
2. Auth0 envía al usuario un OTP por SMS.
3. El usuario copia el código y vuelve a la pantalla de inicio de sesión.
4. El usuario ingresa el código en el campo correspondiente e inicia sesión en tu aplicación.

Para obtener más información sobre cómo configurar la autenticación passwordless basada en SMS para distintos tipos de login, consulta los recursos a continuación:

* [Autenticación passwordless con Universal Login](/docs/es/authenticate/passwordless/passwordless-with-universal-login)
* [Autenticación passwordless con Classic Login](/docs/es/authenticate/passwordless/implement-login/universal-login)
* [Autenticación passwordless con Embedded Login](/docs/es/authenticate/passwordless/implement-login/embedded-login)

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Actualmente, esta categoría de autenticación passwordless se considera un tipo de conexión único en tu tenant, separado de otras conexiones de base de datos, sociales o empresariales.

  Cuando un usuario se autentica con este método, su perfil se crea en la conexión passwordless usando Auth0 como proveedor de identidad (IdP). Como no puedes garantizar que los usuarios iniciarán sesión siempre con el mismo correo electrónico o número de teléfono, es posible que acaben teniendo varios perfiles de usuario en el Auth0 datastore. Si se crean duplicados, puedes asociar varios perfiles de usuario [vinculando sus cuentas](/docs/es/manage-users/user-accounts/user-account-linking).

  En algunos casos, un perfil de usuario creado mediante este tipo de conexión passwordless puede compartir un identificador (como el correo electrónico o el número de teléfono) con un perfil asociado a otro tipo de conexión, como una conexión empresarial o social. Si esto ocurre, puedes usar la [vinculación de cuentas](/docs/es/manage-users/user-accounts/user-account-linking) para asociar perfiles passwordless con identidades de otras conexiones.
</Callout>

<div id="email-based-passwordless-authentication">
  ## Autenticación passwordless por correo electrónico
</div>

La autenticación passwordless por correo electrónico abarca dos métodos:

* [Contraseñas de un solo uso](#one-time-passwords)
* [Magic Links](#magic-links)

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Actualmente, esta categoría de autenticación passwordless se considera un tipo de conexión independiente en tu tenant, separado de otras conexiones de base de datos, sociales o empresariales.

  Cuando un usuario se autentica con este método, su perfil se crea en la conexión passwordless con Auth0 como proveedor de identidad (IdP). Como no puedes garantizar que los usuarios siempre iniciarán sesión con el mismo correo electrónico o número de teléfono, es posible que terminen con varios perfiles de usuario en el datastore de Auth0. Si se crean duplicados, puedes asociar varios perfiles de usuario [vinculando sus cuentas](/docs/es/manage-users/user-accounts/user-account-linking).

  En algunos casos, un perfil de usuario creado mediante este tipo de conexión passwordless puede compartir un identificador (como el correo electrónico o el número de teléfono) con un perfil asociado a otro tipo de conexión, como una conexión empresarial o social. Si esto ocurre, puedes usar la [vinculación de cuentas](/docs/es/manage-users/user-accounts/user-account-linking) para asociar perfiles passwordless con identidades de otras conexiones.
</Callout>

<div id="one-time-passwords">
  ### Contraseñas de un solo uso
</div>

Este método de autenticación passwordless permite a los usuarios acceder a las aplicaciones mediante códigos de un solo uso, conocidos como OTP (contraseñas de un solo uso). Con esta implementación, los OTP se envían a los usuarios por correo electrónico.

El flujo de trabajo general de los OTP por correo electrónico es el siguiente:

1. Un usuario llega a la pantalla de inicio de sesión de su aplicación e introduce su correo electrónico.
2. Auth0 envía un OTP al correo electrónico del usuario.
3. El usuario copia el código y vuelve a la pantalla de inicio de sesión.
4. El usuario introduce el código en el campo correspondiente e inicia sesión en su aplicación.

Para obtener más información sobre cómo configurar la autenticación passwordless por correo electrónico para diferentes tipos de inicio de sesión, consulte los recursos a continuación:

* [Autenticación Passwordless con Universal Login](/docs/es/authenticate/passwordless/passwordless-with-universal-login)
* [Autenticación Passwordless con Classic Login](/docs/es/authenticate/passwordless/implement-login/universal-login)
* [Autenticación Passwordless con Embedded Login](/docs/es/authenticate/passwordless/implement-login/embedded-login)

<div id="magic-links">
  ### Magic Links
</div>

Los Magic Links son un método de autenticación passwordless que permite a los usuarios acceder a una aplicación mediante un enlace enviado a su correo electrónico. Si el usuario no accede al enlace dentro de un plazo determinado, el enlace deja de estar disponible.

El flujo de trabajo general de los Magic Links es el siguiente:

1. Un usuario llega a la pantalla de inicio de sesión de tu aplicación e introduce su correo electrónico.
2. Auth0 envía un Magic Link al usuario al correo electrónico proporcionado.
3. El usuario selecciona el Magic Link en su correo electrónico e inicia sesión automáticamente en tu aplicación.

Los Magic Links solo se pueden implementar en Classic Login. Para obtener más información, consulta los siguientes recursos:

* [Autenticación passwordless con Magic Links](/docs/es/authenticate/passwordless/authentication-methods/email-magic-link)
* [Autenticación passwordless con Classic Login](/docs/es/authenticate/passwordless/implement-login/universal-login)

<div id="biometrics-webauthn">
  ## Biometría (WebAuthn)
</div>

La biometría es un método de autenticación passwordless que utiliza los rasgos físicos de una persona para verificar su identidad y permitirle acceder a una aplicación. Actualmente, Auth0 admite la autenticación biométrica mediante huellas dactilares y reconocimiento facial.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Aunque Auth0 sigue admitiendo el método de autenticación heredado Identifier First with Biometrics, se recomienda encarecidamente usar en su lugar la [autenticación basada en clave de acceso](/docs/es/authenticate/passwordless#passkeys).
</Callout>

Para implementar la biometría en su aplicación, se requieren las siguientes configuraciones:

* Habilitar [Universal Login](/docs/es/authenticate/login/auth0-universal-login).
* Configurar una [conexión de base de datos](/docs/es/authenticate/database-connections).
* Configurar la [autenticación Identifier First + Biometrics](/docs/es/authenticate/login/auth0-universal-login/identifier-first).

Una vez implementada, los usuarios pueden optar por registrar sus dispositivos para usar biometría después de iniciar sesión en su aplicación con credenciales tradicionales. Para registrarse en biometría, el usuario debe utilizar un dispositivo con capacidades biométricas, como un dispositivo móvil inteligente.

El flujo de trabajo general de la autenticación biométrica es el siguiente:

1. Un usuario llega a la página de Universal Login de su aplicación e introduce su dirección de correo electrónico.
2. El usuario primero se autentica con sus credenciales tradicionales. Después, se le solicita que registre su dispositivo para usar biometría.
3. Si el usuario decide registrarse, sigue las indicaciones de su dispositivo para completar el proceso de configuración biométrica.

Después de que el usuario registre su dispositivo, podrá usar la biometría como método principal de autenticación en los siguientes inicios de sesión en su aplicación.

Para obtener más información sobre la autenticación biométrica, consulte [Configurar WebAuthn with Device Biometrics para la autenticación passwordless](/docs/es/authenticate/login/auth0-universal-login/passwordless-login/webauthn-device-biometrics).
