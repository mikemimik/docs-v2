title: Forzar la reautenticación en OIDC
---

El mecanismo `prompt=login` puede vulnerarse simplemente eliminando el parámetro a su paso por el agente de usuario (navegador), y solo sirve para proporcionar una pista de UX al proveedor <Tooltip tip="OpenID: Estándar abierto de autenticación que permite a las aplicaciones verificar las identidades de los usuarios sin recopilar ni almacenar información de inicio de sesión." cta="Ver glosario" href="/docs/es/glossary?term=OpenID">OpenID</Tooltip> (OP) en los casos en que la <Tooltip tip="Relying Party: Entidad (como un servicio o una aplicación) que depende de un proveedor de identidad externo para autenticar a un usuario." cta="Ver glosario" href="/docs/es/glossary?term=relying+party">parte de confianza</Tooltip> (RP) quiera mostrar un enlace como:

**“Hola, Josh. ¿No eres tú? Haz clic aquí.”**

Sin embargo, no debes confiar en él para validar que se haya producido una autenticación reciente. Para mitigar esto, el cliente debe validar que la reautenticación se haya producido usando el claim `auth_time`. Este claim se incluirá automáticamente en el <Tooltip tip="ID Token: Credencial destinada al propio cliente, en lugar de para acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=ID+token">token de identidad</Tooltip> cuando se proporcionen los parámetros `prompt=login` o `max_age=0` en la solicitud de autenticación.

Debes pasar el parámetro `max_age` al endpoint [`/authorize` de la API de autorización](https://auth0.com/docs/api/authentication). Si usas [Auth0.js](/docs/es/libraries/auth0js) o [Lock](/docs/es/libraries/lock/lock-authentication-parameters), puedes establecer el parámetro en las opciones correspondientes de la biblioteca.

La forma de implementar la reautenticación depende de tu caso de uso concreto. Distingue entre la reautenticación simple para operaciones sensibles y el [step-up](/docs/es/secure/multi-factor-authentication/step-up-authentication) (es decir, la <Tooltip tip="Multi-factor authentication (MFA): Proceso de autenticación de usuario que utiliza un factor además de username y contraseña, como un código por SMS." cta="Ver glosario" href="/docs/es/glossary?term=multi-factor+authentication">autenticación multifactor</Tooltip>) para operaciones sensibles. Ambas son medidas de seguridad válidas. La primera requiere que el usuario final vuelva a introducir su contraseña, mientras que la segunda requiere que también use un método de autenticación multifactor previamente configurado.

<div id="limitations-of-promptlogin-parameters">
  ## Limitaciones de los parámetros prompt=login
</div>

La [especificación de OIDC](https://openid.net/specs/openid-connect-core-1_0.html#AuthRequest) define el parámetro `prompt=login`, que puede usarse para activar la interfaz de reautenticación (normalmente, un prompt de inicio de sesión):

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  **prompt**

  OPCIONAL: Lista de valores String ASCII delimitados por espacios y que distingue entre mayúsculas y minúsculas, que especifica si el servidor de autorización solicita al usuario final la reautenticación y el consentimiento. Los valores definidos son:

  **login**

  El servidor de autorización debe solicitar al usuario final que vuelva a autenticarse. Si no puede volver a autenticar al usuario final, debe devolver un error, normalmente `login_required`.
</Callout>

Sin embargo, hay un problema al usar este parámetro para garantizar la reautenticación: **el RP no tiene forma de validate que realmente se haya llevado a cabo una reautenticación**. Veamos el tráfico para entender por qué. El flujo de una Authentication Request del RP es el siguiente:

```http lines
https://mydomain.auth0.com/authorize?
client_id=abcd1234
&redirect_uri= https://mydomain.com/callback
&scope=openid profile
&response_type=id_token
&prompt=login
```

Una vez completada correctamente la autenticación por parte del AS, el RP recibirá un token de identidad:

```json JSON lines
{
  "nickname": "user",
  "name": "user@mydomain.auth0.com",
  "updated_at": "2019-04-01T14:43:03.445Z",
  "iss": "https://jcain0.auth0.com/",
  "sub": "auth0|l33t",
  "aud": "abcd1234",
  "iat": 1554129793,
  "exp": 1554165793
}
```

El documento de identidad confiable devuelto por el AS **no incluye claims que permitan validar cuándo se produjo el último inicio de sesión**. Esto se convierte en un problema cuando la solicitud de autorización inicial se realiza en forma de una redirección 302 a través del navegador del usuario final. Si un actor malicioso quiere omitir el paso de reautenticación solicitado por el RP, simplemente tiene que eliminar el parámetro `prompt=login`, y el RP no puede detectar la diferencia entre los campos contenidos en el token de identidad.

Aquí tienes un diagrama de un flujo implícito simplificado que usa el parámetro `prompt=login`:

<Frame>![Forzar la reautenticación en el flujo implícito de OIDC](/docs/images/cdy7uua7fh8z/7lhntbIKJ25JqQ1M9uB6rJ/26163ab92ac6e289e1185bcb50db2a72/simplified-implicit-flow-with-prompt-login.png)</Frame>

Ten en cuenta que lo único que tiene que hacer el usuario final es eliminar el parámetro `prompt=login` para omitir el paso de reautenticación:

<Frame>![Flujo implícito simplificado sin prompt=login](/docs/images/cdy7uua7fh8z/3hye4mnbcsny7oT0L2kxeq/029a66b8f4f98ee56c5d45a2da617e88/simplified-implicit-flow-remove-prompt.png)</Frame>

El token o los tokens devueltos por el primer flujo anterior serán idénticos al token o los tokens devueltos por el segundo flujo. El RP no tiene ninguna forma definida por la especificación de verificar que la reautenticación se haya realizado y, por lo tanto, no puede confiar en que `prompt=login` realmente haya dado lugar a una reautenticación.

<div id="max_age-authentication-request-parameter">
  ## parámetro de solicitud de autenticación `max_age`
</div>

A diferencia de `prompt=login`, el parámetro de solicitud de autenticación `max_age` ofrece un mecanismo mediante el cual los RP pueden confirmar de forma fehaciente que la reautenticación se ha realizado dentro de un intervalo de tiempo determinado. La [especificación de OIDC](https://openid.net/specs/openid-connect-core-1_0.html#AuthRequest) establece lo siguiente:

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  **max&#95;age**

  OPCIONAL: Antigüedad máxima de autenticación. Especifica el tiempo transcurrido permitido, en segundos, desde la última vez que el usuario final fue autenticado activamente por el OP. Si el tiempo transcurrido es mayor que este valor, el OP debe intentar reautenticar activamente al usuario final. (El parámetro de solicitud `max_age` corresponde al parámetro de solicitud `max_auth_age` de OpenID 2.0 PAPE). Cuando se usa `max_age`, el token de identidad devuelto debe incluir un valor de claim `auth_time`.
</Callout>

La última frase de la definición es la parte más importante. Cuando el RP solicita `max_age`, debe haber un claim `auth_time` presente en el RP. Esto significa que `max_age` puede usarse de una de estas dos maneras:

* **Para exigir una antigüedad máxima mínima de la sesión**: Si una aplicación exige que los usuarios vuelvan a autenticarse una vez al día, esto puede hacerse cumplir en el contexto de una sesión de <Tooltip tip="inicio de sesión único (SSO): Servicio que, después de que un usuario inicia sesión en una aplicación, inicia automáticamente la sesión de ese usuario en otras aplicaciones." cta="Ver glosario" href="/docs/es/glossary?term=SSO">SSO</Tooltip> mucho más larga proporcionando un valor para `max_age`. Estos valores se definen en segundos.
* **Para forzar una reautenticación inmediata**: Si una aplicación requiere que un usuario se reautentique antes de obtener acceso, proporcione un valor de 0 para el parámetro `max_age` y el AS forzará un nuevo login.

Este requisito se describe de la siguiente manera:

<Frame>![flujo de reautenticación con max\_age de OIDC](/docs/images/cdy7uua7fh8z/2IHX6TjuCEcMrPZxoHkC41/11a32321789feee0eac6e71e9c64b307/max-age-flow.png)</Frame>

Tenga en cuenta que el RP recibe un token con la información necesaria para validar si la reautenticación se ha realizado o no. Ahora el RP puede consultar el claim `auth_time` en el token de identidad para determinar si se cumplió o no el parámetro `max_age` que solicitó. De esta manera, el parámetro `max_age=0` es inmune al mismo tipo de manipulación del cliente que podría vulnerar el parámetro `prompt=login`.

<Warning>
  Tenga en cuenta que depende exclusivamente del RP validar que está recibiendo un token de identidad con un `auth_time` adecuado. Los autores de aplicaciones y los frameworks que usen el parámetro `max_age` deberán contemplar esta validación adicional.
</Warning>

<div id="use-auth_time-claims">
  ## Usa claims de auth_time
</div>

Hemos establecido que la especificación de OIDC proporciona el parámetro `max_age` como una forma de confirmar de manera fehaciente que se ha producido una reautenticación, pero `prompt=login` no lo hace. Esto no ofrece opciones muy seguras si quieres forzar una reautenticación:

* **prompt=login**: Incluye solo el parámetro `prompt` y no valida que el AS realmente haya vuelto a autenticar.
* **prompt=login &amp; max&#95;age=999999**: Incluye un `max_age` arbitrario para que esté presente un claim `auth_time`. Puedes validar que se produjo una reautenticación, pero los parámetros se vuelven engorrosos.
* **max&#95;age=0**: Fuerza de forma efectiva un prompt de inicio de sesión usando solo el parámetro `max_age`. Ten en cuenta que una actualización reciente de la especificación aclaró aún más este parámetro, indicando que es, en la práctica, lo mismo que `prompt=login`. Esta opción no es viable, ya que mezcla lo que debería ser un parámetro de UX con un parámetro de mantenimiento de sesión.

En su lugar, Auth0 ha optado por enviar el claim `auth_time` en el token de identidad al responder a un parámetro de solicitud `prompt=login`. Esto significa que tienes la opción de usar `prompt=login` Y validar que se produjo una reautenticación.

<div id="auth_time-validation-example">
  ### Ejemplo de validación de auth_time
</div>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Debe asegurarse de implementar la validación para confirmar que se haya producido una reautenticación. Debe validar que se haya devuelto un `auth_time` válido.
</Callout>

El siguiente ejemplo usa el módulo [passport-auth0-openidconnect](https://github.com/auth0/passport-auth0-openidconnect) para mostrar cómo validar la reautenticación. La primera forma (y la más sencilla) es agregar la opción `max_age=0` a `Auth0OidcStrategy`:

```javascript JavaScript lines
var strategy = new Auth0OidcStrategy(
  {
    domain: process.env.AUTH0_DOMAIN,
    clientID: process.env.AUTH0_CLIENT_ID,
    clientSecret: process.env.AUTH0_CLIENT_SECRET,
    callbackURL: process.env.AUTH0_CALLBACK_URL || 'http://localhost:5000/callback',
    max_age: 0
  },
  function(req, issuer, audience, profile, accessToken, refreshToken, params, cb) {
    // ¡No se necesita validación adicional!
    return cb(null, profile);
  });
```

Ten en cuenta que no se necesitan más pasos de validación, ya que la estrategia ya se encarga de validar el parámetro `max_age`:

```javascript JavaScript lines
// https://openid.net/specs/openid-connect-basic-1_0.html#IDTokenValidation - verificación 8.
if (meta.params.max_age && (!jwtClaims.auth_time || ((meta.timestamp - meta.params.max_age) > jwtClaims.auth_time))) {
  return self.error(new Error('auth_time in id_token not included or too old'));
}
```

También puedes usar `prompt=login` en el mismo contexto, pero como el estándar no exige que un `auth_time` acompañe la respuesta del token de identidad, debes realizar la validación manualmente. Por lo tanto, el constructor de la estrategia sería:

```javascript JavaScript lines
var strategy = new Auth0OidcStrategy(
  {
    domain: process.env.AUTH0_DOMAIN,
    clientID: process.env.AUTH0_CLIENT_ID,
    clientSecret: process.env.AUTH0_CLIENT_SECRET,
    callbackURL: process.env.AUTH0_CALLBACK_URL || 'http://localhost:5000/callback',
    prompt: 'login'
  },
  function(req, issuer, audience, profile, accessToken, refreshToken, params, cb) {
    const tenSecondsAgo = (Date.now() / 1000) - 10;
    if (isNaN(profile.auth_time) || profile.auth_time < tenSecondsAgo) {
      return cb('prompt=login requested, but auth_time is greater than 10 seconds old', null);
    }

    return cb(null, profile);
  });
```

A diferencia de `max_age=0`, el cliente debe validar manualmente el parámetro `auth_time`. Para obtener más información, consulta [Usar claims de auth&#95;time](/docs/es/authenticate/login/max-age-reauthentication#use-auth_time-claims).

<Warning>
  El ejemplo anterior representa una prueba de concepto simplificada (debe haberse autenticado en los últimos 10 segundos). Idealmente, si quieres validar que se ha producido una reautenticación, tendrías que:

  1. Almacenar la hora en que se realizó la solicitud de autenticación inicial.
  2. Tras la respuesta de autenticación, recuperar la hora a la que se envió la solicitud.
  3. Comparar la hora original de la solicitud de autenticación con el claim `auth_time` para asegurarte de que `auth_time` sea posterior.

  **Auth0 no recomienda seguir el enfoque utilizado en el ejemplo en ningún sistema de producción.**
</Warning>

<div id="known-issues">
  ## Problemas conocidos
</div>

Auth0 solo puede garantizar que se haya producido un intercambio con el <Tooltip tip="Proveedor de identidad (IdP): servicio que almacena y administra identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=identity+provider">proveedor de identidad</Tooltip> upstream. Esto puede deberse a que el usuario realmente inició sesión en un proveedor de identidad de terceros, o quizás a que ya tenía una sesión activa y no tuvo que volver a iniciar sesión. En cualquier caso, el intercambio de Auth0 con el proveedor de identidad upstream dará como resultado una actualización de `auth_time`.

Auth0 no admite forzar la reautenticación dentro del proveedor de identidad upstream, ya que no todos los proveedores lo permiten.

El siguiente diagrama muestra un flujo de ejemplo para un usuario que decide volver a autenticarse con una conexión federada:

<Frame>![Diagrama de que las conexiones federadas no fuerzan la reautenticación](/docs/images/cdy7uua7fh8z/8o7GZWQo6LKRTwuYC6dDS/119eeb33dcafcdc5e8d992f284fd5134/federated-connection-flow.png)</Frame>

Este método supone que usa [conexiones de base de datos](/docs/es/authenticate/database-connections). Los proveedores de identidad externos pueden o no admitir el forzado de la reautenticación. Usar `prompt=login` o `prompt=consent` suele ser una forma de indicar a un proveedor de identidad externo (social) que vuelva a autenticar a un usuario, pero Auth0 no puede forzarlo.

<Warning>
  No confíe en la verificación del lado del cliente (es decir, en el navegador) del token de identidad o `auth_time` para evitar que se realicen operaciones sensibles.
</Warning>
