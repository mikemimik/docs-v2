---
description: Describe cómo las partes de confianza pueden confirmar que se ha realizado una reautenticación dentro de un intervalo de tiempo específico mediante el parámetro de solicitud `max_age`.
title: Forzar la reautenticación en OIDC
---

El mecanismo `prompt=login` puede verse comprometido simplemente eliminando el parámetro mientras pasa por el agente de usuario (navegador) y solo sirve para dar una pista de UX al proveedor de <Tooltip tip="OpenID: Estándar abierto de autenticación que permite a las aplicaciones verificar la identidad de los usuarios sin recopilar ni almacenar información de inicio de sesión." cta="Ver glosario" href="/docs/es/glossary?term=OpenID">OpenID</Tooltip> (OP) en los casos en que la <Tooltip tip="Relying Party: Entidad (como un servicio o aplicación) que depende de un proveedor de identidad de terceros para autenticar a un usuario." cta="Ver glosario" href="/docs/es/glossary?term=relying+party">parte de confianza</Tooltip> (RP) quiere mostrar un enlace como:

**“Hola, Josh. ¿No eres tú? Haz clic aquí.”**

Sin embargo, no debe confiar en él para validar que se haya producido una autenticación reciente. Para mitigar esto, el cliente debe validar que la reautenticación se haya producido mediante el claim `auth_time`. Este claim se incluirá automáticamente en el <Tooltip tip="ID Token: Credencial destinada al propio cliente, en lugar de para acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=ID+token">token de identidad</Tooltip> cuando se proporcionen los parámetros `prompt=login` o `max_age=0` en la solicitud de autenticación.

Debe pasar el parámetro `max_age` al endpoint [`/authorize` de la API de autorización](https://auth0.com/docs/api/authentication). Si usa [Auth0.js](/docs/es/libraries/auth0js) o [Lock](/docs/es/libraries/lock/lock-authentication-parameters), puede establecer el parámetro en las opciones adecuadas de la biblioteca.

La forma de implementar la reautenticación depende de su caso de uso específico. Distinga entre la reautenticación simple para operaciones sensibles y el [step-up](/docs/es/secure/multi-factor-authentication/step-up-authentication) (es decir, la <Tooltip tip="Multi-factor authentication (MFA): Proceso de autenticación de usuario que utiliza un factor además de nombre de usuario y contraseña, como un código por SMS." cta="Ver glosario" href="/docs/es/glossary?term=multi-factor+authentication">autenticación multifactor</Tooltip>) para operaciones sensibles. Ambas son medidas de seguridad válidas. La primera requiere que el usuario final vuelva a introducir su contraseña, mientras que la segunda también exige que use un método de autenticación multifactor configurado previamente.

<div id="limitations-of-promptlogin-parameters">
  ## Limitaciones de los parámetros `prompt=login`
</div>

La [especificación OIDC](https://openid.net/specs/openid-connect-core-1_0.html#AuthRequest) define el parámetro `prompt=login`, que puede usarse para mostrar la UI de reautenticación (normalmente, un prompt de inicio de sesión):

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  **prompt**

  OPCIONAL: lista de valores de tipo cadena ASCII delimitados por espacios y sensibles a mayúsculas y minúsculas que especifica si el servidor de autorización solicita al usuario final que vuelva a autenticarse y dé su consentimiento. Los valores definidos son:

  **login**

  El servidor de autorización debe solicitar al usuario final que vuelva a autenticarse. Si no puede volver a autenticar al usuario final, debe devolver un error, normalmente `login_required`.
</Callout>

Sin embargo, hay un problema al usar este parámetro para garantizar la reautenticación: **el RP no tiene forma de validar que realmente se haya producido una reautenticación**. Veamos el tráfico para entender por qué. El flujo de una solicitud de autenticación desde el RP es el siguiente:

```http lines
https://mydomain.auth0.com/authorize?
client_id=abcd1234
&redirect_uri= https://mydomain.com/callback
&scope=openid profile
&response_type=id_token
&prompt=login
```

Una vez que el AS complete correctamente la autenticación, el RP recibirá un token de identidad:

```json JSON lines
{
  "nickname": "user",
  "name": "user@mydomain.auth0.com",
  "updated_at": "2019-04-01T14:43:03.445Z",
  "iss": "https://jcain0.auth0.com/",
  "sub": "auth0|l33t",
  "aud": "abcd1234",
  "iat": 1554129793,
  "exp": 1554165793
}
```

El documento de identidad de confianza devuelto por el AS **no incluye claims que permitan validar cuándo se produjo el último inicio de sesión**. Esto se convierte en un problema cuando la solicitud de autorización inicial se realiza en forma de una redirección 302 a través del navegador del usuario final. Si un actor malicioso quiere omitir el paso de reautenticación solicitado por el RP, simplemente tiene que eliminar el parámetro `prompt=login`, y el RP no puede distinguir la diferencia en los campos incluidos en el token de identidad.

Aquí tienes un diagrama de un flujo implícito simplificado que utiliza el parámetro `prompt=login`:

<Frame>![Forzar la reautenticación en un flujo implícito de OIDC](/docs/images/cdy7uua7fh8z/7lhntbIKJ25JqQ1M9uB6rJ/26163ab92ac6e289e1185bcb50db2a72/simplified-implicit-flow-with-prompt-login.png)</Frame>

Ten en cuenta que lo único que tiene que hacer el usuario final es eliminar el parámetro `prompt=login`, y el paso de reautenticación puede omitirse:

<Frame>![Flujo implícito simplificado sin prompt=login](/docs/images/cdy7uua7fh8z/3hye4mnbcsny7oT0L2kxeq/029a66b8f4f98ee56c5d45a2da617e88/simplified-implicit-flow-remove-prompt.png)</Frame>

Los token(s) devueltos por el primer flujo anterior serán idénticos a los token(s) devueltos por el segundo flujo. El RP no tiene ninguna forma definida por la especificación de verificar que la reautenticación se haya producido y, por lo tanto, no puede confiar en que `prompt=login` haya dado lugar realmente a una reautenticación.

<div id="max_age-authentication-request-parameter">
  ## parámetro de solicitud de autenticación `max_age`
</div>

A diferencia de `prompt=login`, el parámetro de solicitud de autenticación `max_age` proporciona un mecanismo mediante el cual los RP pueden confirmar de forma fehaciente que la reautenticación se ha producido dentro de un intervalo de tiempo determinado. La [especificación OIDC](https://openid.net/specs/openid-connect-core-1_0.html#AuthRequest) establece lo siguiente:

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  **max&#95;age**

  OPCIONAL: Antigüedad máxima de autenticación. Especifica el tiempo transcurrido máximo permitido, en segundos, desde la última vez que el usuario final fue autenticado activamente por el OP. Si el tiempo transcurrido es mayor que este valor, el OP debe intentar reautenticar activamente al usuario final. (El parámetro de solicitud `max_age` corresponde al parámetro de solicitud `max_auth_age` de OpenID 2.0 PAPE). Cuando se usa `max_age`, el token de identidad devuelto debe incluir un valor de claim `auth_time`.
</Callout>

La última oración de la definición es la parte más importante. Cuando el RP solicita `max_age`, debe haber un claim `auth_time` presente en el RP. Esto significa que `max_age` puede usarse de una de estas dos maneras:

* **Para exigir una vigencia mínima de la sesión**: Si una app exige que los usuarios se reautentiquen una vez al día, esto puede aplicarse en el contexto de una sesión de <Tooltip tip="Inicio de sesión único (SSO): servicio que, después de que un usuario inicia sesión en una aplicación, inicia automáticamente la sesión de ese usuario en otras aplicaciones." cta="Ver glosario" href="/docs/es/glossary?term=SSO">SSO</Tooltip> mucho más larga proporcionando un valor para `max_age`. Estos valores se definen en segundos.
* **Para forzar una reautenticación inmediata**: Si una app requiere que un usuario se reautentique antes de obtener acceso, proporcione un valor de 0 para el parámetro `max_age` y el AS forzará un nuevo inicio de sesión.

Este requisito se describe de la siguiente manera:

<Frame>![Flujo de reautenticación de OIDC con max\_age](/docs/images/cdy7uua7fh8z/2IHX6TjuCEcMrPZxoHkC41/11a32321789feee0eac6e71e9c64b307/max-age-flow.png)</Frame>

Tenga en cuenta que el RP recibe un token con la información necesaria para validar si se ha producido o no una reautenticación. Ahora el RP puede consultar el claim `auth_time` en el token de identidad para determinar si se cumplió o no el parámetro `max_age` que solicitó. De este modo, el parámetro `max_age=0` es inmune al mismo tipo de manipulación del cliente que podría vulnerar el parámetro `prompt=login`.

<Warning>
  Tenga en cuenta que depende exclusivamente del RP validar que está recibiendo un token de identidad con un `auth_time` adecuado. Esta validación adicional deberá ser contemplada por los autores de aplicaciones y los frameworks que utilicen el parámetro `max_age`.
</Warning>

<div id="use-auth_time-claims">
  ## Usa claims `auth_time`
</div>

Hemos establecido que la especificación de OIDC proporciona el parámetro `max_age` como una forma de confirmar de manera inequívoca que se ha producido una reautenticación, pero `prompt=login` no. Esto no ofrece opciones muy seguras si quieres forzar una reautenticación:

* **prompt=login**: Incluye solo el parámetro `prompt` y no valida que el AS realmente haya vuelto a autenticar al usuario.
* **prompt=login &amp; max&#95;age=999999**: Incluye un `max_age` arbitrario para que esté presente un claim `auth_time`. Puedes validar que se produjo una reautenticación, pero los parámetros se vuelven engorrosos.
* **max&#95;age=0**: Fuerza de forma efectiva un prompt de inicio de sesión usando solo el parámetro `max_age`. Ten en cuenta que una actualización reciente de la especificación aclaró aún más este parámetro, indicando que, en la práctica, es lo mismo que `prompt=login`. Esta opción no es viable, ya que mezcla lo que debería ser un parámetro de UX con un parámetro de mantenimiento de sesión.

En su lugar, Auth0 ha optado por enviar el claim `auth_time` en el token de identidad al responder a un parámetro de solicitud `prompt=login`. Esto significa que tienes la opción de usar `prompt=login` Y validar que se produjo una reautenticación.

<div id="auth_time-validation-example">
  ### Ejemplo de validación de auth_time
</div>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Debes asegurarte de validar que se haya producido una reautenticación. Debes validar que se haya devuelto un `auth_time` válido.
</Callout>

El siguiente ejemplo usa el módulo [passport-auth0-openidconnect](https://github.com/auth0/passport-auth0-openidconnect) para mostrar cómo validar la reautenticación. La primera forma (y la más sencilla) es añadir la opción `max_age=0` a `Auth0OidcStrategy`:

```javascript JavaScript lines
var strategy = new Auth0OidcStrategy(
  {
    domain: process.env.AUTH0_DOMAIN,
    clientID: process.env.AUTH0_CLIENT_ID,
    clientSecret: process.env.AUTH0_CLIENT_SECRET,
    callbackURL: process.env.AUTH0_CALLBACK_URL || 'http://localhost:5000/callback',
    max_age: 0
  },
  function(req, issuer, audience, profile, accessToken, refreshToken, params, cb) {
    // ¡No se necesita validación adicional!
    return cb(null, profile);
  });
```

Ten en cuenta que no se necesitan más pasos de validación, ya que la estrategia ya se encarga de validar el parámetro `max_age`:

```javascript JavaScript lines
// https://openid.net/specs/openid-connect-basic-1_0.html#IDTokenValidation - verificación 8.
if (meta.params.max_age && (!jwtClaims.auth_time || ((meta.timestamp - meta.params.max_age) > jwtClaims.auth_time))) {
  return self.error(new Error('auth_time in id_token not included or too old'));
}
```

También puedes usar `prompt=login` en el mismo contexto, pero como el estándar no exige que la respuesta del token de identidad vaya acompañada de un `auth_time`, debes encargarte de la validación manualmente. Por lo tanto, el constructor de la estrategia sería:

```javascript JavaScript lines
var strategy = new Auth0OidcStrategy(
  {
    domain: process.env.AUTH0_DOMAIN,
    clientID: process.env.AUTH0_CLIENT_ID,
    clientSecret: process.env.AUTH0_CLIENT_SECRET,
    callbackURL: process.env.AUTH0_CALLBACK_URL || 'http://localhost:5000/callback',
    prompt: 'login'
  },
  function(req, issuer, audience, profile, accessToken, refreshToken, params, cb) {
    const tenSecondsAgo = (Date.now() / 1000) - 10;
    if (isNaN(profile.auth_time) || profile.auth_time < tenSecondsAgo) {
      return cb('prompt=login requested, but auth_time is greater than 10 seconds old', null);
    }

    return cb(null, profile);
  });
```

A diferencia de `max_age=0`, el cliente debe validar manualmente el parámetro `auth_time`. Para obtener más información, consulta [Usar claims de auth&#95;time](/docs/es/authenticate/login/max-age-reauthentication#use-auth_time-claims).

<Warning>
  El ejemplo anterior es una prueba de concepto simplificada (debe haberse autenticado en los últimos 10 segundos). Lo ideal es que, si quieres validar que se ha producido una reautenticación, debas:

  1. Almacenar la hora en que se hizo la solicitud de autenticación inicial.
  2. Al recibir la respuesta de autenticación, recuperar la hora en que se envió la solicitud.
  3. Comparar la hora original de la solicitud de autenticación con el claim `auth_time` para asegurarte de que `auth_time` sea posterior.

  **Auth0 no recomienda seguir el enfoque utilizado en el ejemplo en sistemas de producción.**
</Warning>

<div id="known-issues">
  ## Problemas conocidos
</div>

Auth0 solo puede garantizar que se haya producido un intercambio con el <Tooltip tip="Proveedor de identidad (IdP): servicio que almacena y gestiona identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=identity+provider">proveedor de identidad</Tooltip> upstream. Esto puede deberse a que el usuario haya iniciado sesión realmente en un proveedor de identidad de terceros o, quizá, a que ya tuviera una sesión y no necesitara volver a iniciar sesión. En cualquier caso, el intercambio de Auth0 con el proveedor de identidad upstream dará como resultado un `auth_time` actualizado.

Auth0 no admite forzar la reautenticación dentro del proveedor de identidad upstream, porque no todos los proveedores lo permiten.

El siguiente diagrama muestra un ejemplo de flujo para un usuario que decide reautenticarse con una conexión federada:

<Frame>![Diagrama de que las conexiones federadas no fuerzan la reautenticación](/docs/images/cdy7uua7fh8z/8o7GZWQo6LKRTwuYC6dDS/119eeb33dcafcdc5e8d992f284fd5134/federated-connection-flow.png)</Frame>

Este método supone que utilizas [conexiones de base de datos](/docs/es/authenticate/database-connections). Los proveedores de identidad externos pueden admitir o no que se fuerce la reautenticación. Usar `prompt=login` o `prompt=consent` suele ser una forma de indicarle a un proveedor de identidad externo (social) que reautentique a un usuario, pero Auth0 no puede imponerlo.

<Warning>
  No confíes en la verificación del lado del cliente (es decir, en el navegador) del token de identidad o de `auth_time` para impedir operaciones sensibles.
</Warning>
