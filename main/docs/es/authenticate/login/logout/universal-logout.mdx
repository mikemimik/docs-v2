---
description: Aprende a configurar la revocación global de tokens para las conexiones de Auth0.
title: Universal Logout
---

Auth0 admite integraciones de Universal Logout con Okta Workforce Identity, que cierran la sesión de los usuarios en las aplicaciones cuando se produce un evento administrativo o de seguridad.

Universal Logout implementa la especificación [Global Token Revocation](https://www.ietf.org/archive/id/draft-parecki-oauth-global-token-revocation-02.html), que va más allá de los estándares establecidos de cierre de sesión por back-channel de <Tooltip tip="OpenID: Estándar abierto para la autenticación que permite a las aplicaciones verificar las identidades de los usuarios sin recopilar ni almacenar información de inicio de sesión." cta="Ver glosario" href="/docs/es/glossary?term=OpenID">OpenID</Tooltip> Connect al revocar <Tooltip tip="OpenID: Estándar abierto para la autenticación que permite a las aplicaciones verificar las identidades de los usuarios sin recopilar ni almacenar información de inicio de sesión." cta="Ver glosario" href="/docs/es/glossary?term=refresh+tokens">tokens de actualización</Tooltip> además de las sesiones de usuario. Esta solución integral de cierre de sesión abarca aplicaciones web tradicionales, aplicaciones de página única (SPA) y aplicaciones nativas que usan una combinación de tokens de actualización, sesiones de aplicación y sesiones del <Tooltip tip="Identity Provider (IdP): Servicio que almacena y gestiona identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=identity+provider">proveedor de identidad</Tooltip> para obtener nuevos tokens y mantener la sesión del usuario iniciada.

Si usas los tipos de conexión [Okta](/docs/es/authenticate/identity-providers/enterprise-identity-providers/okta), [SAML](/docs/es/authenticate/identity-providers/enterprise-identity-providers/saml) u [OpenID Connect](/docs/es/authenticate/identity-providers/enterprise-identity-providers/oidc) para federarte con Workforce Identity, ya no necesitas crear un endpoint de [global token revocation](https://developer.okta.com/docs/guides/oin-universal-logout-overview/) para usar Okta Universal Logout. En su lugar, puedes proporcionar al administrador de Okta Workforce la URL del endpoint específico de la conexión de Auth0 y aprovechar [OpenID Connect Back-Channel Logout](/docs/es/authenticate/login/logout/back-channel-logout) de Auth0 para finalizar las sesiones de la aplicación si es necesario.

<div id="how-it-works">
  ## Cómo funciona
</div>

El endpoint de global token revocation de Auth0 sigue el perfil documentado en [Build Logout for your app](https://developer.okta.com/docs/guides/oin-universal-logout-overview/). Usa el formato `iss_sub` para identificar a los usuarios en la solicitud de cierre de sesión y el siguiente formato de URL:

`https://{yourDomain}/oauth/global-token-revocation/connection/{yourConnectionName}`

Cuando Auth0 recibe una solicitud de cierre de sesión de un usuario, valida la solicitud con el mismo conjunto de claves que se usa para validar <Tooltip tip="Token de ID: Credencial destinada al propio cliente, en lugar de para acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=ID+tokens">tokens de ID</Tooltip> o aserciones de <Tooltip tip="Token de ID: Credencial destinada al propio cliente, en lugar de para acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=SAML">SAML</Tooltip> emitidas por Okta Workforce Identity. A continuación, finaliza todas las [sesiones de Auth0](/docs/es/manage-users/sessions/session-layers) del usuario, revoca los [tokens de actualización](/docs/es/secure/tokens/refresh-tokens) emitidos por Auth0 y, si está configurado, activa [OpenID Connect Back-Channel Logout](/docs/es/authenticate/login/logout/back-channel-logout) para revocar las sesiones de la aplicación.

<Frame>![Flujo de trabajo del usuario con Universal Logout](/docs/images/cdy7uua7fh8z/4tcRn1VmydyBGORCHlKuvt/725b95df1cddae8ed3b2d36f0c3495bb/universal_logout_flow_diagram.png)</Frame>

El tiempo que tarda un usuario de una aplicación en perder el acceso depende del tipo de aplicación y de cómo esté integrada con Auth0. Auth0 admite una amplia variedad de [arquitecturas de aplicaciones](/docs/es/get-started/applications) gracias a su compatibilidad con los estándares de identidad OIDC y <Tooltip tip="OAuth 2.0: Framework de autorización que define protocolos y flujos de trabajo de autorización." cta="Ver glosario" href="/docs/es/glossary?term=OAuth+2.0">OAuth 2.0</Tooltip>, así como con los [Quickstarts y SDKs](/docs/es/quickstarts) de Auth0. Esto incluye:

* Las aplicaciones web tradicionales que crean sus propias [sesiones de la aplicación](/docs/es/manage-users/sessions/session-layers) pueden usar tokens de actualización y <Tooltip tip="Access Token: Credencial de autorización, en forma de cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=access+tokens">tokens de acceso</Tooltip> para acceder a las API a través de un backend seguro.
* Las aplicaciones de JavaScript basadas en navegador que aprovechan la [capa de sesión de Auth0](/docs/es/authenticate/login/configure-silent-authentication) o usan técnicas como la [rotación del token de actualización](/docs/es/secure/tokens/refresh-tokens/refresh-token-rotation) para obtener los tokens de acceso necesarios para acceder a las API desde un navegador web.
* Las aplicaciones nativas o móviles que no se ejecutan en un navegador web y usan tokens de actualización y tokens de acceso como método principal para mantener a los usuarios con la sesión iniciada.

<div id="revoke-refresh-tokens-and-auth0-user-sessions">
  ### Revocar tokens de actualización y sesiones de usuario de Auth0
</div>

Las aplicaciones que usan tokens de actualización o sesiones de Auth0 obtienen los siguientes beneficios de seguridad cuando la integración de Universal Logout está habilitada:

* En las aplicaciones basadas en navegador que usan [la sesión de Auth0](/docs/es/authenticate/login/configure-silent-authentication), el usuario pierde el acceso la próxima vez que la aplicación [consulta la sesión de Auth0](/docs/es/authenticate/login/configure-silent-authentication#poll-with-checksession-), y Auth0 le pide que vuelva a iniciar sesión cuando se le redirige
* En las aplicaciones que usan tokens de actualización, el usuario pierde el acceso en cuanto expire su token de acceso actual, que puede durar desde unos pocos segundos hasta la [duración máxima del token de acceso](/docs/es/secure/tokens/access-tokens/update-access-token-lifetime) configurada en Auth0.

<div id="revoke-application-user-sessions">
  ### Revocar las sesiones de usuario de la aplicación
</div>

Para las sesiones creadas por aplicaciones web, debes usar la funcionalidad existente de [OpenID Connect Back-Channel Logout](/docs/es/authenticate/login/logout/back-channel-logout/configure-back-channel-logout) de Auth0 para cerrar esas sesiones cuando Universal Logout cierre la sesión de usuario de Auth0. Para obtener más información, consulta los [ejemplos de implementación del SDK de Auth0](/docs/es/authenticate/login/logout/back-channel-logout#auth0-sdks).

<div id="configure-universal-logout-in-auth0">
  ## Configurar Universal Logout en Auth0
</div>

Configure Universal Logout en función del tipo de conexión de Auth0.

<div id="okta-workforce">
  ### Okta Workforce
</div>

1. [Conecta tu tenant de Auth0 con Okta Workforce Identity](/docs/es/authenticate/identity-providers/enterprise-identity-providers/okta).
2. En Auth0 Dashboard, ve a [**Autenticación &gt; Empresarial &gt; Okta Workforce**](https://manage.auth0.com/#/connections/enterprise/okta). Selecciona tu conexión y elige **Configuración**.
3. Copia la **URL del endpoint de revocación** que proporcionarás al administrador de Okta Workforce.

<div id="openid-connect">
  ### OpenID Connect
</div>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Recomendamos encarecidamente usar el tipo de conexión de **Okta Workforce** para las nuevas integraciones de OpenID Connect. Sin embargo, Universal Logout aún puede habilitarse en integraciones anteriores mediante el tipo de conexión genérico de OpenID Connect en **Settings** de OpenID Connect en Auth0 Dashboard.
</Callout>

1. En Auth0 Dashboard, ve a [**Authentication &gt; Enterprise &gt; OpenID Connect**](https://manage.auth0.com/#/connections/enterprise/oidc)**.** Selecciona tu conexión y elige **Settings**.
2. Copia la **URL del endpoint de revocación** que proporcionarás al administrador de Okta Workforce.

<div id="saml">
  ### SAML
</div>

1. [Configura Okta como proveedor de identidad SAML](/docs/es/authenticate/protocols/saml/saml-sso-integrations/configure-auth0-saml-service-provider/configure-okta-as-saml-identity-provider).
2. En Auth0 Dashboard, ve a [**Autenticación &gt; Empresarial &gt; SAML**](https://manage.auth0.com/#/connections/enterprise/samlp). Selecciona tu conexión y elige **Configuración**.
3. En **Subject**, introduce el Application ID de la aplicación SAML registrada en Okta Workforce Identity. Ejemplo: `0oagcc12354688xxxx`. Para obtener más información, consulta [Cómo obtener un Application ID](https://support.okta.com/help/s/article/How-to-obtain-an-application-ID?language=en_US).
4. En **Issuer**, ve a la aplicación SAML registrada en la organización de Okta Workforce Identity y copia el [URI del emisor](https://developer.okta.com/docs/guides/oin-universal-logout-overview/#endpoint-authentication).

   <Callout icon="file-lines" color="#0EA5E9" iconType="regular">
     Para obtener este valor, los administradores de WIC pueden ir a la sección **Applications** &gt; **Applications &gt; [Application] &gt; Authentication &gt; Sign-on settings &gt; Sign-on methods &gt; SAML 2.0 &gt; More details** del portal de Okta.
   </Callout>
5. Copia la **URL del endpoint de revocación**, que proporcionarás al administrador de Okta Workforce.

<div id="configure-universal-logout-in-okta-workforce-identity">
  ## Configurar Universal Logout en Okta Workforce Identity
</div>

Configure Universal Logout en Okta Workforce Identity para enviar señales de cierre de sesión a la aplicación mediante la conexión de Auth0. Esto debe hacerlo un administrador de Okta.

<div id="prerequisites">
  ### Requisitos previos
</div>

Necesitas:

* Una organización de Okta Workforce Identity con [Identity Threat Protection](https://www.okta.com/products/identity-threat-protection/) habilitado, o una [prueba gratuita](https://www.okta.com/free-trial/).
* Debe estar habilitado el acceso anticipado a Universal Logout para aplicaciones Generic SAML y OIDC en tu organización de Okta Workforce Identity. Para obtener más información, consulta [Configurar Universal Logout para aplicaciones compatibles](https://help.okta.com/oie/en-us/content/topics/itp/config-universal-logout.htm).

<div id="configure-okta">
  ### Configurar Okta
</div>

Habilita Universal Logout en una [organización de Okta Workforce](/docs/es/authenticate/identity-providers/enterprise-identity-providers/okta) para la conexión de Auth0.

1. En el portal de Okta, selecciona **Applications** &gt; **Applications**.
2. Selecciona la aplicación que registraste para la integración de Auth0.
3. En la pestaña **General**, selecciona **Logout &gt; Edit**.

   <Frame>![Imagen del Dashboard de Okta para la configuración de Universal Logout](/docs/images/cdy7uua7fh8z/M957prYUhjyMxXgpdA8pB/224812ce3a76ac9d032abe50c4be3a1d/2025-01-23_13-03-21.png)</Frame>
4. Selecciona **Okta system or admin initiates logout**.
5. Introduce la URL copiada de Auth0 en **Logout Endpoint URL**.
6. Selecciona **Issuer and Subject Identifier** como **Subject Format**.
7. Selecciona **Save**.

<div id="test-universal-logout">
  ### Probar Universal Logout
</div>

Pruebe Universal Logout en Okta Workforce Identity revocando las sesiones del usuario seleccionado.

1. En el portal de Okta, seleccione **Directory &gt; People**.
2. Seleccione un usuario que haya iniciado sesión en una aplicación de Auth0.
3. Seleccione **More Actions &gt; Clear User Sessions**.

   <Frame>![Probar Universal Logout en Okta Dashboard](/docs/images/cdy7uua7fh8z/Z8an5lN6WAlSAdgiBkKLa/fde264f71615c5eb7e07e0c48d8426bf/2025-01-23_13-08-29.png)</Frame>
4. En el cuadro de diálogo, seleccione **Also include logout enabled apps and Okta API tokens**.
5. Seleccione **Clear and Revoke**.

<div id="logs-and-notifications">
  ## Logs y notificaciones
</div>

Los administradores del tenant de Auth0 pueden consultar el estado y los detalles de cada solicitud de Universal Logout que recibe Auth0 en [Auth0 Dashboard &gt; Monitoreo &gt; Logs](https://manage.auth0.com/#/*/logs). Los tipos de eventos de log emitidos para Universal Logout se documentan en [Códigos de tipos de evento de log](/docs/es/deploy-monitor/logs/log-event-type-codes).

También puede integrar [flujos de logs personalizados](/docs/es/customize/log-streams) para notificar a sistemas externos cuando se produzcan eventos de Universal Logout.

<div id="management-api">
  ## Management API
</div>

También puedes usar la [Management API](https://auth0.com/docs/api/management/v2) para configurar Universal Logout de forma programática. Esto resulta útil para automatizar configuraciones o crear tu propia configuración fuera del <Tooltip tip="Auth0 Dashboard: El producto principal de Auth0 para configurar tus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip>.

Para Okta Workforce y OpenID Connections, no es necesario proporcionar ninguna información en la <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip>, ya que solo es necesario presentar el endpoint de revocación global de tokens de Auth0 al administrador de Okta Workforce.

Para las conexiones SAML, además de compartir la URL del endpoint, Universal Logout debe configurarse estableciendo estos atributos en el objeto `connection`:

* **options.global&#95;token&#95;revocation&#95;jwt&#95;iss** - El [ID del emisor](https://developer.okta.com/docs/guides/oin-universal-logout-overview/#endpoint-authentication) de la aplicación SAML registrada en Okta Workforce Identity. Valor de ejemplo: `http://www.okta.com/exkhwkmkwhZUnuA6xxxx`.
* **options.global&#95;token&#95;revocation&#95;jwt&#95;sub** - El [ID de la aplicación](https://support.okta.com/help/s/article/How-to-obtain-an-application-ID?language=en_US) de la aplicación SAML registrada en Okta Workforce Identity. Valor de ejemplo: `0oagcc12354688xxxx`.

Puedes usar estos atributos en llamadas a la Management API para [actualizar](https://auth0.com/docs/api/management/v2#!/Connections/patch_connections_by_id) o [crear](https://auth0.com/docs/api/management/v2/connections/post-connections)  conexiones SAML. Consulta el siguiente ejemplo de JSON para crear una solicitud:

```json JSON lines
{
	"strategy": "samlp",
  	"name": "CONNECTION_NAME",
  	"options": {
    	"global_token_revocation_jwt_iss": "ISS_VALUE", 
     	"global_token_revocation_jwt_sub":  "SUB_VALUE",
    	"signInEndpoint": "SIGN_IN_ENDPOINT_URL",
    	"signOutEndpoint": "SIGN_OUT_ENDPOINT_URL",
    	"signatureAlgorithm": "rsa-sha256",
    	"digestAlgorithm": "sha256",
    	"fieldsMap": {
     		...
    	},
    	"signingCert": "BASE64_SIGNING_CERT"
  	}
}
```

Para obtener más información, consulta [Crear una conexión empresarial con la Management API](/docs/es/authenticate/identity-providers/enterprise-identity-providers/saml#create-an-enterprise-connection-using-the-management-api).
