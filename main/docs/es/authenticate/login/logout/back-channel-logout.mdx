title: cierre de sesión OIDC por back-channel
description: Describe la funcionalidad de cierre de sesión OIDC por back-channel de Auth0.
---

Auth0 es compatible con la [especificación OpenID Connect Back-Channel Logout 1.0](https://openid.net/specs/openid-connect-backchannel-1_0.html#Backchannel) en todos los tenants con una suscripción al plan Enterprise.

Esta especificación utiliza el ID de sesión (`sid`) incluido en los <Tooltip tip="token de ID: Credencial destinada al propio cliente, y no al acceso a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=ID+tokens">tokens de ID</Tooltip> y los tokens de cierre de sesión para coordinar el cierre de la sesión mediante comunicación por back-channel. Los distintos ID de sesión representan sesiones individuales de un agente de usuario o dispositivo en tu tenant. Los tokens de cierre de sesión identifican al usuario final y la sesión que se debe cerrar.

<div id="back-channel-communications">
  ## Comunicaciones de back-channel
</div>

Para usar el cierre de sesión por back-channel, una aplicación debe exponer un URI de cierre de sesión por back-channel, accesible desde el servidor del tenant, donde la aplicación espera recibir las solicitudes con el token de cierre de sesión. Cuando una aplicación recibe esta solicitud, debe borrar el estado de la sesión local que coincida con los claims del token.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Para que el cierre de sesión por back-channel funcione, las aplicaciones deben poder recibir comunicaciones de back-channel.
</Callout>

<div id="tokens-in-oidc-back-channel-logout-communications">
  ### Tokens en las comunicaciones de cierre de sesión OIDC por back-channel
</div>

Las aplicaciones no pueden basarse en las <Tooltip tip="Cookie de sesión: Entidad que, cuando está presente, permite considerar al usuario como autenticado." cta="Ver glosario" href="/docs/es/glossary?term=session+cookies">cookies de sesión</Tooltip> para determinar qué sesión deben finalizar cuando las comunicaciones se realizan por back-channel. En su lugar, el servicio depende de un identificador de sesión compartido (`sid`) en los tokens de ID y de cierre de sesión.

Cuando los usuarios finales se autentican correctamente con Auth0 durante el inicio de sesión, el <Tooltip tip="Servidor de autorización: Servidor centralizado que contribuye a definir los límites del acceso de un usuario. Por ejemplo, su servidor de autorización puede controlar los datos, las tareas y las funcionalidades disponibles para un usuario." cta="Ver glosario" href="/docs/es/glossary?term=authorization+server">servidor de autorización</Tooltip> emite tokens de acceso y de ID. Los tokens de cierre de sesión se generan cuando se destruye una sesión, por ejemplo, mediante una acción de cierre de sesión o la revocación de una sesión. Tanto los tokens de ID como los de cierre de sesión contienen los claims que su aplicación necesita para facilitar el workflow de cierre de sesión por back-channel. Para obtener más información sobre los claims, lea [Claims de JSON Web Token](/docs/es/secure/tokens/json-web-tokens/json-web-token-claims).

<Frame>![Workflow del cierre de sesión por back-channel](/docs/images/cdy7uua7fh8z/5jQ5HogFNeD9tqUGEJuJHE/663f1f52b285f7019fd1157f5a26caf0/2023-09-29_09-32-00.png)</Frame>

1. Inicio de sesión - Durante la autenticación de usuario, el tenant de Auth0 agrega el `sid` al token de ID.
2. Inicio de sesión - La aplicación almacena el identificador de sesión recibido en su propio session store y lo asocia con la sesión específica de la aplicación.
3. Cierre de sesión - El IdP llama a la URL de devolución de llamada de cierre de sesión prerregistrada y envía el token de cierre de sesión a este endpoint. El token contiene el `user_id` (`sub`) y el `sid`, junto con otros parámetros.
4. Cierre de sesión - El backend de la aplicación debe validar el token de cierre de sesión según la especificación de OIDC y extraer el `sid`. Luego, el backend puede usar este token para encontrar la sesión asociada al identificador y finalizarla según sea necesario.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  La respuesta esperada es `HTTP 200` para un cierre de sesión correcto. Si recibe `HTTP 400`, una request incorrecta o malinterpretada, puede usar nuestros consejos para solucionar el problema. Para obtener más información, lea [Configurar el cierre de sesión por back-channel](/docs/es/authenticate/login/logout/back-channel-logout/configure-back-channel-logout).
</Callout>

<div id="how-it-works">
  ### Cómo funciona
</div>

El caso de uso de ejemplo muestra cómo funciona el cierre de sesión por back-channel con más de una aplicación:

<Frame>![Caso de uso de cierre de sesión por back-channel con varias aplicaciones](/docs/images/cdy7uua7fh8z/54mbNobsvLec0A2tz0DXUG/99ce79c9e93ffe526472d6250373778c/2023-06-20_09-39-12.png)</Frame>

1. Durante la configuración de la aplicación, la Aplicación A registra un URI de cierre de sesión por back-channel en Auth0.
2. Durante la configuración de la aplicación, la Aplicación B registra un URI de cierre de sesión por back-channel en Auth0.

   <Callout icon="file-lines" color="#0EA5E9" iconType="regular">
     Las URL de cierre de sesión OIDC por back-channel deben:

     * Ser accesibles desde el IdP
     * Usar endpoints con cifrado TLS
     * [Validar los tokens de cierre de sesión](https://openid.net/specs/openid-connect-backchannel-1_0.html#Validation)
   </Callout>
3. Durante el login del usuario final, este se autentica con Auth0 para acceder a la Aplicación A.
4. Auth0 envía un token de identidad con `sid` a la Aplicación A. Para obtener más información, consulta [Estructura del token de ID](/docs/es/secure/tokens/id-tokens/id-token-structure).
5. El usuario se autentica con Auth0 para acceder a la Aplicación B.
6. Auth0 envía un token de identidad con el mismo `sid` a la Aplicación B. Tu aplicación debe almacenar la información de la sesión.
7. Durante el cierre de sesión, la Aplicación A u otras entidades inician el cierre de sesión en el front-channel.
8. Auth0 termina la capa de sesión de Auth0 mediante la cookie de sesión.
9. Auth0 llama al URI de cierre de sesión por back-channel de la Aplicación A y envía el token de cierre de sesión.
10. La Aplicación A valida el token de cierre de sesión y termina la sesión.
11. Auth0 llama al URI de cierre de sesión por back-channel de la Aplicación B y envía el token de cierre de sesión.
12. La Aplicación B valida el token de cierre de sesión y termina la sesión.

Las solicitudes de cierre de sesión por back-channel se colocan en una cola asíncrona y se procesan lo antes posible. En situaciones de alta carga, puede haber un ligero retraso antes de que se ejecute la solicitud de cierre de sesión, así que diseña tus aplicaciones para contemplar esta consistencia eventual.

<div id="session-storage">
  ### Almacenamiento de sesiones
</div>

El ejemplo de almacenamiento de sesiones está desarrollado en Node (Express) y se basa en [Express OpenID Connect Web App Sample](https://github.com/auth0-samples/auth0-express-webapp-sample/tree/master/01-Login).

En la pestaña de sesiones de tu aplicación, expón la ruta que configuraste para recibir el token de cierre de sesión. Valida el token y finaliza la sesión del usuario.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Este ejemplo utiliza un almacenamiento de sesiones en memoria con fines de demostración.
</Callout>

```javascript routes/index.js lines expandable
const express = require('express');
const router = express.Router();
const { requiresAuth } = require('express-openid-connect');

// middleware para validar el token de cierre de sesión
const requiresValidLogoutToken = require('../middlewares/validateLogoutToken');

// función auxiliar para eliminar sesiones de usuario
const deleteUserSessions = require('../utils/sessions');

// nueva ruta para recibir tokens de cierre de sesión por backchannel
// debe configurarse en la pestaña Application -> Sessions
// en el Auth0 Management Dashboard
router.post(
  '/backchannel-logout',
  requiresValidLogoutToken,
  function (req, res, next) {
    // en este punto el token de cierre de sesión es válido, verificado por el middleware requiresValidLogoutToken
    // se puede acceder a él desde el objeto de solicitud: req.logoutToken

    // eliminar la sesión del usuario para cerrar su sesión
    deleteUserSessions(
      req.app.locals.sessionStore,
      req.logoutToken.sub,
      req.logoutToken.sid
    );

    res.sendStatus(200);
  }
);

router.get('/', function (req, res, next) {
  res.render('index', {
    title: 'Auth0 Webapp sample Nodejs',
    isAuthenticated: req.oidc.isAuthenticated(),
    headline: process.env.APP_NAME,
    backgroundColor: process.env.BACKGROUND_COLOR,
    baseURL: process.env.BASE_URL,
  });
});

router.get('/profile', requiresAuth(), function (req, res, next) {
  res.render('profile', {
    userProfile: JSON.stringify(req.oidc.user, null, 2),
    title: 'Profile page',
    headline: process.env.APP_NAME,
    backgroundColor: process.env.BACKGROUND_COLOR,
    baseURL: process.env.BASE_URL,
  });
});

module.exports = router;
```

```javascript middlewares/validateLogoutToken.js lines expandable
// Este middleware valida el token de cierre de sesión según se define aquí:
// https://openid.net/specs/openid-connect-backchannel-1_0.html#Validation

const jose = require('jose');

async function requiresValidLogoutToken(req, res, next) {

  // obtener el conjunto de claves remoto para la verificación del token
  const JWKS = jose.createRemoteJWKSet(
    new URL(process.env.ISSUER_BASE_URL + '/.well-known/jwks.json')
  );

  const logoutToken = req.body.logout_token;

  if (!logoutToken) {
    res.status(400).send('Need logout token');
  }

  try {
    const { payload, protectedHeader } = await jose.jwtVerify(
      logoutToken,
      JWKS,
      {
        issuer: process.env.ISSUER_BASE_URL + '/',
        audience: process.env.CLIENT_ID,
        typ: 'JWT',
        maxTokenAge: '2 minutes',
      }
    );

    // Verificar que el token de cierre de sesión contenga un claim sub, un claim sid, o ambos
    if (!payload.sub && !payload.sid) {
      res
        .status(400)
        .send(
          'Error: Logout token must contain either sub claim or sid claim, or both'
        );
    }

    // Verificar que el token de cierre de sesión contenga un claim events
    // cuyo valor sea un objeto JSON con el nombre de miembro http://schemas.openid.net/event/backchannel-logout
    if (!payload.events['http://schemas.openid.net/event/backchannel-logout']) {
      res
        .status(400)
        .send(
          'Error: Logout token must contain events claim with correct schema'
        );
    }

    // Verificar que el token de cierre de sesión no contenga un claim nonce.
    if (payload.nonce) {
      res
        .status(400)
        .send('Error: Logout token must not contain a nonce claim');
    }

    // adjuntar el token de cierre de sesión válido al objeto de solicitud
    req.logoutToken = payload;

    // el token es válido, invocar el siguiente middleware
    next();
  } catch (error) {
    res.status(400).send(`Error:  ${error.message}`);
  }
}

module.exports = requiresValidLogoutToken;
```

<div id="logout-token-store">
  ### Almacenamiento de tokens de cierre de sesión
</div>

Un enfoque común para el almacenamiento de tokens consiste en definir un almacenamiento de cierre de sesión como alternativa al modelo de almacenamiento de sesiones. La aplicación mantiene una colección de tokens de cierre de sesión en la capa de persistencia.

Cada vez que la aplicación necesita comprobar su estado de autenticación, consulta el almacenamiento de tokens de cierre de sesión para determinar si su sesión sigue activa. El almacenamiento de cierre de sesión depura periódicamente la información obsoleta para conservar solo la necesaria.

<div id="security-considerations">
  ## Consideraciones de seguridad
</div>

Los tokens de cierre de sesión por back-channel se entregan a través de internet; por lo tanto, los endpoints de devolución de llamada que los reciben deben seguir prácticas recomendadas para garantizar un funcionamiento fiable y seguro. La siguiente lista de recomendaciones no es exhaustiva, y siempre debe tener en cuenta las situaciones específicas de despliegue y operación para adaptarse según corresponda. En la lista siguiente, cualquier aplicación que gestione los tokens de cierre de sesión por back-channel se denomina “apps”.

* Las apps deben poder almacenar el ID de sesión (claim `sid`) recibido durante el inicio de sesión del usuario para poder recuperarlo más adelante al recibir un token de cierre de sesión por back-channel.
* Las apps deben verificar todos los tokens recibidos según las [prácticas recomendadas para la validación de JWT](/docs/es/secure/tokens/json-web-tokens/validate-json-web-tokens).
* Las apps deben aceptar únicamente tokens emitidos por tenants de confianza. Un actor malicioso puede intentar enviar tokens emitidos por otros tenants de Auth0; esos intentos deben rechazarse.
* Las apps deben aceptar tokens solo cuando contengan un valor `sid` (ID de sesión) que la app reconozca. Los tokens que contengan un ID de sesión no válido, ya sea porque haya expirado o porque no se reconozca, deben rechazarse.
* Las apps deben exponer los endpoints de devolución de llamada únicamente a través de TLS. No se permiten canales de comunicación sin cifrar.
* Se recomienda que las apps acepten requests solo desde la lista publicada de [direcciones IP salientes](/docs/es/secure/security-guidance/data-security/allowlist).
* Se recomienda que las apps sigan las prácticas recomendadas generales en materia de monitoreo, registro y límite de frecuencia; sin embargo, los detalles de estos aspectos quedan fuera del alcance de este documento.
* Se recomienda que las apps limpien periódicamente las sesiones obsoletas o expiradas.
* Cualquier cambio en la dirección del endpoint debe sincronizarse con la configuración del tenant para garantizar que los tokens de cierre de sesión siempre se entreguen a la URL de devolución de llamada correcta del cierre de sesión por back-channel.
