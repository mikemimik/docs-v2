<Card title="Antes de empezar">
  Para usar el cierre de sesión por back-channel, tu aplicación debe cumplir los siguientes requisitos:

  * El endpoint URI de cierre de sesión OIDC por canal secundario debe estar expuesto en internet para que tu tenant de Auth0 pueda acceder a él.
  * Las aplicaciones listas para producción deben usar TLS. Para obtener más información, consulta [Versiones de TLS (SSL) y algoritmos de cifrado](/docs/es/customize/custom-domains/self-managed-certificates/tls-ssl).
  * La aplicación debe poder almacenar el identificador de sesión (`sid`) proporcionado y asociarlo a las sesiones de usuario que crea. Recomendamos que las aplicaciones de producción usen un almacenamiento de sesiones persistente.
  * La aplicación debe poder recuperar una sesión existente mediante el `sid` durante el proceso de cierre de sesión sin usar cookies del lado del cliente. Las cookies existen dentro del navegador y no son accesibles para el endpoint de devolución de llamada de cierre de sesión.
</Card>

<div id="availability">
  ## Disponibilidad
</div>

El cierre de sesión OIDC por canal secundario está disponible para todos los clientes del plan Enterprise. Debe consultar el endpoint de metadatos estándar de OIDC `/.well-known/*` para determinar si su aplicación cumple con los requisitos.

```json lines
GET https://acme.eu.auth0.com/.well-known/openid-configuration

HTTP 200
{ ..., "backchannel_logout_supported": true, 
  "backchannel_logout_session_supported": true }
```

<div id="back-channel-logout-restrictions">
  ## Restricciones del cierre de sesión por back-channel
</div>

Las URL de cierre de sesión por back-channel se invocan en endpoints expuestos públicamente y deben cumplir ciertas restricciones:

1. Debe usar el protocolo HTTPS. No se permiten HTTP sin cifrar ni otros protocolos.
2. No debe usar subdominios de Auth0. Algunos subdominios de Auth0 son:

   * auth0.com
   * auth0app.com
   * webtask.io
   * webtask.run
3. No recomendamos usar direcciones IP sin dominio. Para usar el cierre de sesión por back-channel, las direcciones IP deben ser públicas. No se permiten direcciones IP de rangos internos, reservados o de loopback.

<div id="configure-auth0">
  ## Configurar Auth0
</div>

Registre su aplicación para recibir tokens de cierre de sesión a través de <Tooltip tip="Auth0 Dashboard: el producto principal de Auth0 para configurar sus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip> o <Tooltip tip="Auth0 Dashboard: el producto principal de Auth0 para configurar sus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip>.

<Tabs>
  <Tab title="Auth0 Dashboard">
    #### Suscribir aplicaciones

    1. Vaya a [**Auth0 Dashboard &gt; Applications**](https://manage.auth0.com/#/applications).
    2. Elija la aplicación que desea registrar.
    3. Seleccione la pestaña **Settings**.
    4. En **OpenID Connect Back-Channel Logout &gt; Back-Channel Logout URI**, agregue el URI de cierre de sesión de la aplicación que recibirá los tokens de cierre de sesión
    5. Cuando termine, seleccione **Save Changes**.

       <Frame>![Auth0 Dashboard > Applications > Settings](/docs/images/cdy7uua7fh8z/19uQRkVswdmaO8gHTVTXCc/0564bfeda05bc7df90af9e8ee852047f/Screenshot_2024-11-05_at_13.31.54.png)</Frame>

    #### Cancelar la suscripción de aplicaciones

    Al cancelar la suscripción de su aplicación, el servicio deja de hacer seguimiento de los nuevos inicios de sesión y de enviar eventos de cierre de sesión. El servicio descarta los eventos de cierre de sesión pendientes una vez que se cancela la suscripción de su aplicación.

    Para cancelar la suscripción de su aplicación, elimine la URL de cierre de sesión por back-channel.

    1. Vaya a [Auth0 Dashboard &gt; Applications](http://manage.auth0.com/#/applications).
    2. Elija la aplicación que desea registrar.
    3. Seleccione la pestaña **Settings**.
    4. En **OpenID Connect Back-Channel Logout &gt; Back-Channel Logout URI**
    5. Elimine la URL de back-channel.
    6. Cuando termine, seleccione **Save Changes**.

       <Frame>![Auth0 Dashboard > Applications > Settings](/docs/images/cdy7uua7fh8z/5SlrBhwrNbJx0Wdw2kQ19t/9b69eb926e44c35b426daa76a851a5db/Screenshot_2024-11-05_at_13.31.54.png)</Frame>

    Con fines de auditoría, el servicio siempre registra la suscripción o la cancelación de la suscripción de las URL de cierre de sesión por back-channel como `API Operation Event` en los registros del tenant de Auth0. Para obtener más información, consulte [Logs](/docs/es/deploy-monitor/logs).
  </Tab>

  <Tab title="Management API">
    #### Suscribir aplicación

    Agregue la propiedad `oidc_logout` para enviar un `PATCH` a un cliente mediante Management API:

    ```bash cURL lines
    PATCH /api/v2/clients/{id}
    "oidc_logout": {
      "backchannel_logout_urls": ["http://acme.eu.auth0.com/events"]
    }
    ```

    <Callout icon="file-lines" color="#0EA5E9" iconType="regular">
      La solicitud puede incluir los demás parámetros de la aplicación.
    </Callout>

    #### Cancelar la suscripción de la aplicación

    Elimine el parámetro `oidc_logout`:

    ```bash cURL lines
    PATCH https://acme.eu.auth0.com/api/v2/clients/{id}
    {
      "oidc_logout": {
        "backchannel_logout_urls": []
      }
    }
    ```
  </Tab>
</Tabs>

<div id="configure-your-application">
  ## Configure su aplicación
</div>

Una vez que haya configurado el cierre de sesión por back-channel a través de Auth0 Dashboard o Management API, configure su aplicación para usar el servicio según la tecnología o el framework.

1. Implemente la autenticación de usuario de acuerdo con el tipo de aplicación.

   1. Los usuarios finales deben poder iniciar sesión en la aplicación y debe crearse una sesión.
   2. Auth0 debe emitir un token de identidad, y este debe estar accesible en el backend de la aplicación para su posterior procesamiento.
2. Extienda el proceso de inicio de sesión para guardar el `sid` y, opcionalmente, los claims `sub` después de validar el token de identidad.

   1. Estos claims deben guardarse en la sesión actual de la aplicación.
   2. Las funciones de gestión de sesiones deben poder recuperar una sesión específica a partir del valor de `sid`.
3. Configure el endpoint de cierre de sesión por back-channel:

   1. El endpoint debe procesar solo solicitudes `HTTP POST`.
   2. Extraiga el parámetro `logout_token` y valídelo como un JWT normal, de acuerdo con la especificación.
   3. Verifique que el token contenga un claim `events` con un valor de objeto JSON y un miembro llamado `http://schemas.openid.net/event/backchannel-logout`.
   4. Verifique que el token contenga los claims `sid` y/o `sub`.
   5. Verifique que el token NO contenga el claim `nonce`. Esto es necesario para evitar abusos, al distinguir el token de cierre de sesión del token de identidad.
   6. Una vez validado el token, recupere la sesión correspondiente al valor de `sid` y/o `sub` recibido y termínela. El proceso exacto para finalizar la sesión de la aplicación depende de los detalles de la implementación. Por ejemplo, puede que sea necesario comunicar este evento al front-end.

<div id="expected-responses">
  ## Respuestas esperadas
</div>

* **HTTP 200**:  Confirma el cierre de sesión del usuario en la aplicación específica.
* **HTTP 400**: Indica una solicitud incorrecta. La solicitud no se entiende o el token no superó la validación. Auth0 registra el problema en los registros del tenant, pero no realiza más solicitudes para esta sesión específica.

<div id="troubleshoot-issues">
  ## Solucionar incidencias
</div>

<div id="application-did-not-receive-the-logout-events">
  ### La aplicación no recibió los eventos de cierre de sesión
</div>

Si tu aplicación no recibió una solicitud de cierre de sesión tras un evento de cierre de sesión.

1. Asegúrate de que tu aplicación tenga registrada una URL de cierre de sesión por back-channel en Auth0 Dashboard.
2. Asegúrate de que se pueda acceder a la URL de cierre de sesión por back-channel desde el tenant de Auth0.
3. Asegúrate de que haya una sesión válida establecida. El usuario final debe iniciar sesión en tu aplicación específica mediante Auth0.

   <Callout icon="file-lines" color="#0EA5E9" iconType="regular">
     Las aplicaciones reciben eventos de cierre de sesión solo si los usuarios finales inician sesión en esa aplicación específica con Auth0.  Si el usuario final inicia sesión en otras aplicaciones, los eventos de cierre de sesión no se activan.
   </Callout>
4. Revisa los registros del tenant de Auth0 para ver si hay mensajes sobre errores en la entrega de mensajes de cierre de sesión.
5. Asegúrate de que el cierre de sesión se inicie a través del endpoint de cierre de sesión estándar. Otros eventos no activan los eventos de cierre de sesión.
6. Si es posible, revisa si hay requests bloqueadas en el servidor web y/o en los registros del firewall de la aplicación.

<div id="i-cant-find-oidc-back-channel-logout-tenant-logs">
  #### No encuentro los registros del tenant del cierre de sesión OIDC por canal secundario
</div>

Esta funcionalidad se implementará gradualmente en todos los tenants a partir del 3 de octubre de 2023. Es posible que siga recibiendo [códigos de eventos de log del tenant](/docs/es/deploy-monitor/logs/log-event-type-codes) `sslo` para `oidc_backchannel_logout_succeeded` o `fslo` para `oidc_backchannel_logout_failed`.

<div id="client-app-cannot-verify-the-received-logout-token">
  ### La aplicación cliente no puede verificar el token de cierre de sesión recibido
</div>

Si la transacción sigue fallando con un error 400:

1. Compruebe si el token es un JWT estándar codificado en base64. Algunos servidores web pueden truncar parámetros largos. Para obtener más información, consulte [Algoritmos de firma](/docs/es/get-started/applications/signing-algorithms).
2. Si es posible, capture un token y verifique que sea un JWT. Use una fuente confiable, como [JWT.IO](http://jwt.io).
3. Asegúrese de que la función de verificación obtenga dinámicamente la clave de firma del tenant mediante [conjunto de claves web JSON](/docs/es/secure/tokens/json-web-tokens/json-web-key-sets) (JWKS).

   <Callout icon="file-lines" color="#0EA5E9" iconType="regular">
     No recomendamos usar una clave estática hardcoded. Si su configuración requiere que la clave estática esté hardcoded, asegúrese de que la configuración contenga la clave pública más reciente del tenant de Auth0.
   </Callout>

<div id="response-timeout-errors">
  ### Errores por tiempo de espera en la respuesta
</div>

Auth0 esperará hasta cinco segundos a que responda la URL de cierre de sesión OIDC por canal secundario de la aplicación. Transcurrido ese tiempo, registrará un &quot;Failed OIDC Back-Channel Logout request&quot; en el registro del tenant con una descripción de respuesta vacía.
