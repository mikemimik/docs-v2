El [protocolo OpenID Connect](/docs/es/authenticate/protocols/openid-connect-protocol) admite un parámetro `prompt=none` en la solicitud de autenticación que permite a las aplicaciones indicar que el <Tooltip tip="Authorization Server: Centralized server that contributes to defining the boundaries of a user’s access. For example, your authorization server can control the data, tasks, and features available to a user." cta="View Glossary" href="/docs/es/glossary?term=authorization+server">servidor de autorización</Tooltip> no debe mostrar ninguna interacción del usuario (como autenticación, consentimiento o <Tooltip tip="Authorization Server: Centralized server that contributes to defining the boundaries of a user’s access. For example, your authorization server can control the data, tasks, and features available to a user." cta="View Glossary" href="/docs/es/glossary?term=MFA">MFA</Tooltip>). Auth0 devolverá la respuesta solicitada a la aplicación o devolverá un error si el usuario aún no se ha autenticado o si se requiere algún tipo de consentimiento o prompt antes de continuar.

El uso del [flujo implícito](/docs/es/get-started/authentication-and-authorization-flow/implicit-flow-with-form-post) en las SPA plantea desafíos de seguridad que requieren estrategias de mitigación explícitas. Puede usar el [flujo de código de autorización con PKCE](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce) junto con la autenticación silenciosa para renovar sesiones en las SPA.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Los avances recientes en los controles de privacidad de los usuarios en los navegadores afectan negativamente la experiencia del usuario al impedir el acceso a cookies de terceros; por lo tanto, los flujos basados en navegador deben usar la [rotación del Refresh Token](/docs/es/secure/tokens/refresh-tokens/refresh-token-rotation), que proporciona un método seguro para usar tokens de actualización en las SPA y, al mismo tiempo, ofrece a los usuarios finales acceso fluido a los recursos sin las interrupciones en la experiencia de usuario causadas por tecnologías de privacidad del navegador como ITP.
</Callout>

<div id="initiate-silent-authentication-requests">
  ## Iniciar solicitudes de autenticación silenciosa
</div>

Para iniciar una solicitud de autenticación silenciosa, agrega el parámetro `prompt=none` cuando redirijas a un usuario al [endpoint `/authorize` de la Authentication API de Auth0](https://auth0.com/docs/api/authentication#authorize-application). (Los parámetros de la solicitud de autenticación variarán según las necesidades específicas de tu aplicación).

Por ejemplo:

export const codeExample = `GET https://{yourDomain}/authorize
    ?response_type=id_token token&
    client_id=...&
    redirect_uri=...&
    state=...&
    scope=openid...&
    nonce=...&
    audience=...&
    response_mode=...&
    prompt=none`;

<AuthCodeBlock children={codeExample} language="json" />

El parámetro `prompt=none` hace que Auth0 envíe de inmediato un resultado al `redirect_uri` especificado (URL de devolución de llamada) mediante el `response_mode` indicado, con una de estas dos posibles respuestas: éxito o error.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Todas las [reglas](/docs/es/customize/rules) aplicables se ejecutarán como parte del proceso de autenticación silenciosa.
</Callout>

<div id="response-modes">
  ### Modos de respuesta
</div>

El parámetro `response_mode` determina cómo Auth0 entrega la respuesta de autorización a tu aplicación. Para la autenticación silenciosa, puedes usar:

| Modo          | Método de entrega                      | Caso de uso                                                  |
| ------------- | -------------------------------------- | ------------------------------------------------------------ |
| `query`       | Query string (`?code=...`)             | flujo de código de autorización con gestión en el servidor   |
| `fragment`    | Fragmento de URL (`#access_token=...`) | flujo implícito (legacy)                                     |
| `web_message` | API `postMessage()`                    | **Recomendado para SPAs** - no requiere navegación de página |

Al usar `web_message`, Auth0 renderiza una página HTML dentro de un iframe oculto que utiliza la [API de Web Messaging de HTML5](https://developer.mozilla.org/en-US/docs/Web/API/Window/postMessage) para comunicar el resultado a tu aplicación. Esto permite la autenticación silenciosa sin redirecciones visibles ni pérdida de estado.

<Note>
  Para usar `response_mode=web_message`, debes agregar la URL de tu aplicación al campo **Allowed Web Origins** en la [configuración de la aplicación](https://manage.auth0.com/#/applications). Para obtener más información sobre todos los modos de respuesta, consulta [Marco de autorización OAuth 2.0](/docs/es/authenticate/protocols/oauth#authorization-endpoint).
</Note>

<div id="successful-authentication-responses">
  ### Respuestas de autenticación satisfactorias
</div>

Si el usuario ya había iniciado sesión en Auth0 y no se requiere ningún otro prompt interactivo, Auth0 responderá exactamente igual que si el usuario se hubiera autenticado manualmente a través de la página de inicio de sesión. El formato de la respuesta depende del `response_mode` utilizado:

<Tabs>
  <Tab title="web_message">
    Al usar `response_mode=web_message` con el flujo de código de autorización con PKCE (`response_type=code`), Auth0 devuelve una página HTML que envía el código de autorización a tu aplicación:

    ```html
    <script>
      window.parent.postMessage({
        type: 'authorization_response',
        response: {
          code: 'SplX...GT',
          state: 'your_state_value'
        }
      }, 'https://yourApp.com');
    </script>
    ```

    Tu aplicación (o SDK) escucha este mensaje e intercambia el código por tokens. Si el usuario no tiene una sesión válida, Auth0 envía un error en su lugar:

    ```html
    <script>
      window.parent.postMessage({
        type: 'authorization_response',
        response: {
          error: 'login_required',
          error_description: 'Login required',
          state: 'your_state_value'
        }
      }, 'https://yourApp.com');
    </script>
    ```
  </Tab>

  <Tab title="fragment">
    Al usar `response_mode=fragment` (predeterminado para el flujo implícito con `response_type=id_token token`), Auth0 redirige con tokens en el fragmento de la URL:

    ```text
    GET https://yourApp.com/callback
        #id_token=eyJhbG...&
        access_token=eyJhbG...&
        state=your_state_value&
        expires_in=86400
    ```
  </Tab>

  <Tab title="query">
    Al usar `response_mode=query` (predeterminado para el flujo de código de autorización con `response_type=code`), Auth0 redirige con el código en la cadena de consulta:

    ```text
    GET https://yourApp.com/callback
        ?code=SplX...GT&
        state=your_state_value
    ```
  </Tab>
</Tabs>

Estas respuestas tienen el mismo formato que las de un inicio de sesión realizado directamente sin el parámetro `prompt=none`. La única diferencia es que, con `prompt=none`, la respuesta es inmediata y no requiere interacción del usuario.

<div id="error-responses">
  ### Respuestas de error
</div>

Si el usuario no había iniciado sesión mediante <Tooltip tip="Inicio de sesión único (SSO): servicio que, después de que un usuario inicia sesión en una aplicación, lo autentica automáticamente en otras aplicaciones." cta="Ver glosario" href="/docs/es/glossary?term=Single+Sign-on">inicio de sesión único</Tooltip> (SSO) o su sesión de SSO había expirado, Auth0 devuelve un error con el mismo `response_mode`. Para los modos basados en redirección (`fragment` o `query`):

```text
GET https://your_callback_url/
    #error=ERROR_CODE&
    error_description=ERROR_DESCRIPTION&
    state=...
```

Para el modo `web_message`, el error se envía mediante `postMessage()`, como se muestra en el ejemplo anterior.

Los posibles valores de `ERROR_CODE` están definidos por la [especificación de OpenID Connect](https://openid.net/specs/openid-connect-core-1_0.html#AuthError):

| Respuesta              | Descripción                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                              |
| ---------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `login_required`       | El usuario no había iniciado sesión en Auth0, por lo que la autenticación silenciosa no es posible. Este error puede producirse según cómo estén configurados los ajustes de **Log In Session Management** a nivel del tenant; en concreto, puede ocurrir una vez transcurrido el período establecido en la configuración **tiempo tras el cual es necesario iniciar sesión de nuevo**. Consulta [Configurar los ajustes del tiempo de vida de la sesión](/docs/es/manage-users/sessions/configure-session-lifetime-settings) para obtener más información. |
| `consent_required`     | El usuario había iniciado sesión en Auth0, pero necesita dar su consentimiento para autorizar la aplicación.                                                                                                                                                                                                                                                                                                                                                                                                                                             |
| `interaction_required` | El usuario había iniciado sesión en Auth0 y ha autorizado la aplicación, pero debe ser redirigido a otro lugar antes de que se pueda completar la autenticación; por ejemplo, al usar una [redirect rule](/docs/es/customize/rules/redirect-users).                                                                                                                                                                                                                                                                                                         |

Si se devuelve cualquiera de estos errores, se debe redirigir al usuario a la página de inicio de sesión de Auth0 sin el parámetro `prompt=none` para autenticarse.

<div id="renew-expired-tokens">
  ## Renovar tokens vencidos
</div>

Puede realizar una solicitud de autenticación silenciosa para obtener nuevos tokens siempre que el usuario siga teniendo una sesión válida en Auth0. El [método `checkSession` de auth0.js](/docs/es/libraries/auth0js) utiliza una solicitud de token silenciosa en combinación con `response_mode=web_message` para las SPA, de modo que la solicitud se realiza en un iframe oculto. En las SPA, Auth0.js procesa el resultado (ya sea el token o el código de error) y pasa la información mediante una función de devolución de llamada proporcionada por la aplicación. Esto evita cualquier interrupción en la experiencia de usuario (sin recargar la página ni perder el estado).

<div id="access-token-expiration">
  ### Expiración del token de acceso
</div>

<Tooltip tip="token de acceso: Credencial de autorización, en forma de una cadena opaca o un JWT, que se usa para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=Access+Tokens">tokens de acceso</Tooltip> son opacos para las aplicaciones. Esto significa que las aplicaciones no pueden inspeccionar el contenido de los tokens de acceso para determinar su fecha de expiración.

Hay dos opciones para determinar cuándo expira un token de acceso:

* Leer el parámetro de respuesta `expires_in` que devuelve Auth0.
* Ignorar por completo las fechas de expiración. En su lugar, renueva el token de acceso si tu API rechaza una solicitud de la aplicación (por ejemplo, con un 401).

En el caso del [flujo implícito](/docs/es/get-started/authentication-and-authorization-flow/implicit-flow-with-form-post), Auth0 devuelve el parámetro `expires_in` como parámetro hash después de una autenticación correcta. En el [flujo de código de autorización con PKCE](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce), se devuelve al servidor backend al realizar el intercambio del código de autorización.

El parámetro `expires_in` indica cuántos segundos será válido el token de acceso y puede usarse para prever su expiración.

<div id="error-response">
  ### Respuesta de error
</div>

Es posible que reciba la respuesta de error `timeout`, que indica que se agotó el tiempo de espera durante la ejecución de la comunicación `web_message`. Este error suele estar asociado con el uso de la autenticación de origen cruzado como alternativa. Para solucionarlo, asegúrese de agregar todas las URL desde las que quiera realizar autenticación silenciosa en el campo **Allowed Web Origins** de su **aplicación** en el <Tooltip tip="Auth0 Dashboard: el producto principal de Auth0 para configurar sus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip>.

<div id="poll-with-checksession">
  ## Consulta con checkSession()
</div>

En algunos escenarios con varias aplicaciones en los que se desea el cierre de sesión único (es decir, que cuando un usuario cierre sesión en una aplicación también se cierre en las demás), se puede configurar una aplicación para que consulte periódicamente a Auth0 mediante `checkSession()` si existe una sesión. Si la sesión no existe, puedes cerrar la sesión del usuario en la aplicación. El mismo método de consulta puede utilizarse para implementar autenticación silenciosa en un escenario de inicio de sesión único (SSO).

El intervalo entre las comprobaciones con `checkSession()` debe ser de al menos 15 minutos entre llamadas para evitar posibles problemas futuros con el límite de frecuencia de esta llamada.

<div id="silent-authentication-with-multi-factor-authentication">
  ## Autenticación silenciosa con Autenticación multifactor
</div>

En algunos casos, quizá quieras evitar pedirle al usuario [Autenticación multifactor (MFA)](/docs/es/secure/multi-factor-authentication) cada vez que inicie sesión desde el mismo navegador. Para ello, configura una rule para que MFA solo se solicite una vez por sesión. Esto resulta útil al realizar autenticación silenciosa (`prompt=none`) para renovar tokens de acceso de corta duración en una SPA durante la sesión del usuario, sin tener que depender de establecer `allowRememberBrowser` en `true`.

```js lines
exports.onExecutePostLogin = async (event, api) => {
  const authMethods = event.authentication?.methods || []

  const completedMfa = !!authMethods.find((method) => method.name === 'mfa')

  if (!completedMfa) {
    api.multifactor.enable('any', { allowRememberBrowser: true })
  }
};
```

Para obtener más información, consulta [Cambiar la frecuencia de la solicitud de autenticación](/docs/es/secure/multi-factor-authentication/customize-mfa).
