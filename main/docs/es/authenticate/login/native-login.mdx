---

Las aplicaciones móviles nativas pueden usar flujos de inicio de sesión nativos o basados en navegador.

En un flujo de **inicio de sesión basado en navegador**, se le muestra al usuario un navegador web y se le redirige a la página de inicio de sesión de Auth0 para registrarse o iniciar sesión. Por ejemplo, una aplicación de iOS abre un SafariViewController o una aplicación de Android abre una pestaña personalizada de Chrome.

Con un flujo de inicio de sesión **nativo**, el usuario se registra o introduce sus credenciales directamente en la aplicación.

Independientemente de la opción que elija, Auth0 admite ambas.

<div id="considerations">
  ### Consideraciones
</div>

* Riesgos de phishing/seguridad: una parte no autorizada podría descompilar o interceptar el tráfico hacia o desde su aplicación para obtener el <Tooltip tip="ID de cliente: valor de identificación dado por Auth0 a su recurso registrado." cta="Ver glosario" href="/docs/es/glossary?term=Client+ID">ID de cliente</Tooltip> y la URL de autenticación. Con esta información, la parte no autorizada podría crear una aplicación maliciosa, subirla a una tienda de aplicaciones y usarla para hacer phishing para obtener nombres de usuario, contraseñas y <Tooltip tip="Tokens de acceso: credencial de autorización, en forma de cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=Access+Tokens">Tokens de acceso</Tooltip>.
* <Tooltip tip="inicio de sesión único (SSO): servicio que, después de que un usuario inicia sesión en una aplicación, inicia automáticamente la sesión de ese usuario en otras aplicaciones." cta="Ver glosario" href="/docs/es/glossary?term=SSO">inicio de sesión único</Tooltip>: los usuarios tienen que introducir sus credenciales en cada aplicación.

  * Se puede implementar inicio de sesión único con aplicaciones nativas almacenando tokens de actualización en un Keychain compartido, pero esto no cumple con las especificaciones de OAuth 2.0.
* Requiere más tiempo de implementación
* No hay mejoras automáticas cuando Auth0 agrega nuevas funcionalidades; hay que actualizar el código de la aplicación para aprovecharlas, a diferencia de UL
* No cumple con las prácticas recomendadas de <Tooltip tip="OAuth 2.0: framework de autorización que define protocolos y flujos de trabajo de autorización." cta="Ver glosario" href="/docs/es/glossary?term=OAuth+2.0">OAuth 2.0</Tooltip>

  * [RFC 8252 OAuth 2.0 para aplicaciones nativas](https://tools.ietf.org/html/rfc8252): las solicitudes de autorización de aplicaciones nativas solo deben hacerse a través de agentes de usuario externos, principalmente el navegador del usuario

<div id="rate-limits">
  ### Límites de frecuencia
</div>

Los límites solo se aplican a las solicitudes relacionadas con los flujos de inicio de sesión social nativo, que se identifican según el cuerpo de las solicitudes con los siguientes criterios iniciales:

| Tipo de solicitud    | Cuerpo                                               |
| -------------------- | ---------------------------------------------------- |
| `grant_type`         | `urn:ietf:params:oauth:grant-type:token-exchange`    |
| `subject_token_type` | `http://auth0.com/oauth/token-type/apple-authz-code` |

<div id="limits-for-production-tenants-of-paying-customers">
