El [flujo de contraseña del propietario del recurso](/docs/es/get-started/authentication-and-authorization-flow/resource-owner-password-flow) (a veces llamado concesión de contraseña del <Tooltip tip="Propietario del recurso: Entidad (como un usuario o una aplicación) capaz de otorgar acceso a un recurso protegido." cta="Ver glosario" href="/docs/es/glossary?term=Resource+Owner">propietario del recurso</Tooltip> o ROPG) lo utilizan aplicaciones de alta confianza para realizar autenticación activa. A diferencia del código de autorización y de las concesiones implícitas, este mecanismo de autenticación no redirige a los usuarios a Auth0. Autentica a los usuarios con una sola solicitud, intercambiando sus credenciales de contraseña por un token.

El flujo compatible con OIDC afecta al flujo de contraseña del propietario del recurso en las siguientes áreas:

* Solicitud de autenticación
* Respuesta de autenticación
* Estructura del <Tooltip tip="Token de ID: Credencial destinada al propio cliente, en lugar de acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=ID+token">token de identidad</Tooltip>
* Estructura del <Tooltip tip="Token de acceso: Credencial de autorización, en forma de una cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=Access+token">token de acceso</Tooltip>

<div id="authentication-request">
  ## Solicitud de autenticación
</div>

<div id="legacy">
  ### Antiguo
</div>

```json lines
POST /oauth/ro HTTP 1.1
Content-Type: application/json
{
  "grant_type": "password",
  "client_id": "123",
  "username": "alice",
  "password": "A3ddj3w",
  "connection": "my-database-connection",
  "scope": "openid email favorite_color offline_access",
  "device": "my-device-name"
}
```

El parámetro `device` solo es necesario si se solicita un <Tooltip tip="Token de actualización: token utilizado para obtener un nuevo token de acceso sin obligar a los usuarios a volver a iniciar sesión." cta="Ver glosario" href="/docs/es/glossary?term=refresh+token">token de actualización</Tooltip> al pasar el scope `offline_access`.

<div id="oidc-conformant">
  ### compatible con OIDC
</div>

```json lines
POST /oauth/token HTTP 1.1
Content-Type: application/x-www-form-urlencoded
grant_type=http%3A%2F%2Fauth0.com%2Foauth%2Fgrant-type%2Fpassword-realm&client_id=123&username=alice&password=A3ddj3w&realm=my-database-connection&scope=openid+email+offline_access&audience=https%3A%2F%2Fapi.example.com
```

* El endpoint para realizar intercambios de credenciales es `/oauth/token`.
* El tipo de concesión propio de Auth0 se usa para autenticar usuarios desde una conexión específica (`realm`). También se admite la concesión estándar de contraseña de OIDC, pero no acepta parámetros específicos de Auth0 como `realm`.
* `favorite_color` ya no es un scope válido.
* Se ha eliminado el parámetro `device`.
* El parámetro `audience` es opcional.

<div id="authentication-response">
  ## Respuesta de autenticación
</div>

### Legado

```json lines
HTTP/1.1 200 OK
Content-Type: application/json
Cache-Control: no-store
Pragma: no-cache
{
    "access_token": "SlAV32hkKG",
    "token_type": "Bearer",
    "refresh_token": "8xLOxBtZp8",
    "expires_in": 3600,
    "id_token": "eyJ..."
}
```

* El token de acceso devuelto solo es válido para llamar al endpoint [`/userinfo`](https://auth0.com/docs/api/authentication#get-user-info).
* Solo se devolverá un token de actualización si se envió el parámetro `device` y se solicitó el scope `offline_access`.

<div id="oidc-conformant">
  ### compatible con OIDC
</div>

```json lines
HTTP/1.1 200 OK
Content-Type: application/json
Cache-Control: no-store
Pragma: no-cache
{
    "access_token": "eyJ...",
    "token_type": "Bearer",
    "refresh_token": "8xLOxBtZp8",
    "expires_in": 3600,
    "id_token": "eyJ..."
}
```

* El token de acceso devuelto es válido para llamar al `endpoint` `/userinfo` (siempre que la API especificada por el parámetro `audience` use `RS256` como [algoritmo de firma](/docs/es/get-started/applications/signing-algorithms)) y, opcionalmente, al <Tooltip tip="Resource Server: Servidor que aloja recursos protegidos. Los servidores de recursos aceptan y responden a solicitudes de recursos protegidos." cta="Ver glosario" href="/docs/es/glossary?term=resource+server">servidor de recursos</Tooltip> especificado por el parámetro `audience`.
* El token de identidad se firmará obligatoriamente con `RS256` si lo solicita una aplicación pública. Para obtener más información, consulta [Aplicaciones confidenciales y públicas](/docs/es/get-started/applications/confidential-and-public-applications).
* Solo se devolverá un token de actualización si se concedió el scope `offline_access`.

<div id="access-token-structure-optional">
  ## Estructura del token de acceso (opcional)
</div>

### Legado

```json JSON lines
SlAV32hkKG
```

El token de acceso devuelto es opaco y solo es válido para llamar al endpoint `/userinfo`.

<div id="oidc-conformant">
  ### compatible con OIDC
</div>

export const codeExample3 = `{
    "sub": "auth0|alice",
    "iss": "https://{yourDomain}/",
    "aud": [
        "https://api.example.com",
        "https://{yourDomain}/userinfo"
    ],
    "azp": "123",
    "exp": 1482816809,
    "iat": 1482809609,
    "scope": "openid email"
}`;

<AuthCodeBlock children={codeExample3} language="json" filename="JSON" />

* El token de acceso devuelto es un <Tooltip tip="JSON Web Token (JWT): formato estándar de token de ID (y, a menudo, de token de acceso) que se utiliza para representar claims de forma segura entre dos partes." cta="Ver glosario" href="/docs/es/glossary?term=JWT">JWT</Tooltip> válido para llamar al endpoint `/userinfo` (siempre que la API especificada por el parámetro `audience` use `RS256` como <Tooltip tip="Signing Algorithm: algoritmo utilizado para firmar digitalmente tokens y garantizar que no hayan sido manipulados." cta="Ver glosario" href="/docs/es/glossary?term=signing+algorithm">algoritmo de firma</Tooltip>), así como al servidor de recursos especificado por el parámetro `audience`.
* Ten en cuenta que aún podría devolverse un token de acceso opaco si `/userinfo` es la única <Tooltip tip="Audience: identificador único de la audiencia de un token emitido. Llamado aud en un token, su valor contiene el ID de una aplicación (ID de cliente) para un token de ID o de una API (identificador de API) para un token de acceso." cta="Ver glosario" href="/docs/es/glossary?term=audience">audiencia</Tooltip> especificada.

<div id="standard-password-grant-requests">
  ## Solicitudes estándar para la concesión de contraseña
</div>

La concesión de contraseña con `realm` de Auth0 no está definida por el estándar OIDC, pero se sugiere como alternativa al endpoint del propietario del recurso heredado porque admite el parámetro `realm` específico de Auth0. El [flujo estándar de OIDC también es compatible](/docs/es/get-started/authentication-and-authorization-flow/resource-owner-password-flow) cuando se usa autenticación OIDC.
