---
description: Aprende cómo el flujo compatible con OIDC afecta a los tokens utilizados para proteger las API, incluidos los scopes y claims.
title: Tokens de acceso con OIDC
---

Como las aplicaciones y las API (recursos) se definen como entidades independientes de Auth0 en el flujo compatible con OIDC, puedes obtener <Tooltip tip="Token de acceso: credencial de autorización, en forma de una cadena opaca o JWT, que se utiliza para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=access+tokens">tokens de acceso</Tooltip> para tus API. En consecuencia, todas las API deben protegerse con tokens de acceso en lugar de <Tooltip tip="Token de acceso: credencial de autorización, en forma de una cadena opaca o JWT, que se utiliza para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=ID+tokens">tokens de identidad</Tooltip>. Para obtener más información, consulta [Tokens de acceso](/docs/es/secure/tokens/access-tokens) y [Tokens de identidad](/docs/es/secure/tokens/id-tokens).

<div id="claims">
  ## Claims
</div>

El flujo compatible con OIDC estandariza los claims que puedes agregar a los tokens de identidad y de acceso.

<div id="standard-claims">
  ### Claims estándar
</div>

La especificación de OIDC define un [](https://openid.net/specs/openid-connect-core-1_0.html#StandardClaims) conjunto de claims estándar de los usuarios, como el perfil y el correo electrónico, que se pueden devolver en tokens de identidad o en la respuesta del endpoint `/userinfo`.

Para obtener más información sobre la especificación de claims estándar de OIDC, lee la [Sección 5.1 Claims estándar](https://openid.net/specs/openid-connect-core-1_0.html#StandardClaims) de la [Especificación OpenID](https://openid.net).

<div id="custom-claims">
  ### Claims personalizadas
</div>

Con el flujo compatible con OIDC, se pueden seguir agregando claims personalizadas a los tokens de identidad o a los tokens de acceso, pero deben tener un formato con espacio de nombres para evitar posibles colisiones con las claims estándar de OIDC.

Para aprender a agregar una claim personalizada en el flujo compatible con OIDC, lee [Create Namespaced Custom Claims](/docs/es/secure/tokens/json-web-tokens/create-custom-claims).

<div id="scopes">
  ## Scopes
</div>

En el flujo compatible con OIDC, puedes configurar tus aplicaciones en Auth0 para usar scopes y solicitar lo siguiente:

* Que los claims estándar de OIDC, como `profile` y `email`, se incluyan en el token de identidad (si el usuario da su consentimiento para proporcionar esta información a la aplicación).
* Que los permisos admitidos por la API a la que se quiere acceder se incluyan en el token de acceso. Por ejemplo, puedes definir la <Tooltip tip="Audience: Identificador único de la audience de un token emitido. Se denomina aud en un token; su valor contiene el ID de una aplicación (Client ID) para un token de identidad o de una API (API Identifier) para un token de acceso." cta="Ver glosario" href="/docs/es/glossary?term=audience">audience</Tooltip> de tu API personalizada y los scopes requeridos, lo que te permitirá delimitar el acceso a distintas operaciones dentro de tu API.

Para obtener más información, consulta [Scopes de OpenID Connect](/docs/es/get-started/apis/scopes/openid-connect-scopes).
