Puedes redirigir a los usuarios a páginas específicas (URL) dentro de tu aplicación después de validar sus <Tooltip tip="token de ID: credencial destinada al propio cliente, en lugar de utilizarse para acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=ID+Tokens">tokens de ID</Tooltip> (autenticación). Para ver un ejemplo de cómo funciona, prueba el [React: Login Quickstart](/docs/es/quickstart/spa/react).

<div id="redirect-users-to-callback-urls-on-the-allowlist">
  ## Redirigir a los usuarios a URL de devolución de llamada de la lista de permitidos
</div>

Debido a que las URL de devolución de llamada pueden ser manipuladas por partes no autorizadas, Auth0 solo reconoce como válidas las URL incluidas en la lista de permitidos establecida en el campo **Allowed Callback URLs** de la [configuración de una aplicación](https://manage.auth0.com/#/applications/\{yourClientId}/settings). Para devolver a los usuarios a URL de devolución de llamada de la lista de permitidos, tu aplicación debe saber cómo hacer que el usuario continúe su recorrido.

Hay dos métodos para hacerlo:

* Usar cookies y sesiones del navegador
* Usar parámetros `state`

Durante la autenticación de un usuario, el parámetro de solicitud `redirect_uri` se usa como URL de devolución de llamada. Aquí es donde tu aplicación recibe y procesa la respuesta de Auth0, y suele ser la URL a la que se redirige a los usuarios una vez completada la autenticación. Para obtener más información sobre cómo funciona `redirect_uri`, consulta el [marco de autorización de OAuth 2.0](/docs/es/authenticate/protocols/oauth).

<Tabs>
  <Tab title="Cookie o sesión del navegador">
    Puedes usar una cookie o la sesión del navegador para almacenar un valor de URL de retorno. Esta es una solución sencilla de implementar; sin embargo, puede causar problemas en los casos en que una cookie no persista. En esta situación se inician dos sesiones de usuario independientes. Cada una cumple una función distinta y requiere cierta consideración para lograr la experiencia de usuario deseada.

    * **Sesión de SSO proporcionada por Auth0**: Auth0 proporciona una sesión para habilitar el [inicio de sesión único (SSO)](/docs/es/authenticate/single-sign-on) y permitir que tu usuario mantenga una sesión de autenticación sin que se le soliciten credenciales más de una vez. Auth0 mantiene esta sesión y la almacena como una cookie vinculada al dominio de tu tenant (o `CNAME`). Hay dos opciones de [Configuración del tenant](/docs/es/manage-users/sessions/configure-session-lifetime-settings) que determinan la duración de la sesión de Auth0:

      * `idle_session_lifetime` indica cuánto tiempo permanecerá activa la sesión sin interacción.
      * `session_lifetime` es la duración máxima durante la que se permite que la sesión permanezca activa.

      Estas opciones se aplican a todas las aplicaciones dentro de tu tenant y deben configurarse de modo que se ajusten al modelo de seguridad correspondiente a tu caso de uso.
    * **Sesión de la aplicación**: Tu aplicación también debe mantener una sesión propia. A lo largo de la sesión del usuario, es posible que tu aplicación necesite solicitar tokens adicionales o renovar los que hayan caducado. Debes almacenar estos tokens en tu aplicación y referenciarlos mediante un identificador devuelto al navegador mediante una cookie segura.

    Una vez que tu usuario se haya autenticado con Auth0, depende de tu aplicación determinar durante cuánto tiempo persiste esta sesión.
  </Tab>

  <Tab title="Parámetros de estado">
    Como método alternativo, puedes crear un enlace profundo usando el parámetro `state`, que tu devolución de llamada interpretaría para determinar una ruta de reenvío. Esta solución requiere un poco más de trabajo para implementarse, pero garantiza que la aplicación tenga la información que necesita una vez completada la redirección. Para obtener más información, lee [Prevenir ataques y redirigir usuarios con parámetros de estado de OAuth0 2.0](/docs/es/secure/attack-protection/state-parameters).

    Con este método, envías un valor aleatorio al iniciar una solicitud de autenticación y validas el valor recibido al procesar la respuesta (esto implica que almacenas algo del lado de la aplicación cliente, en la sesión u otro medio, que te permita realizar la validación). Si recibes una respuesta con un valor de `state` que no coincide, es probable que hayas sido objetivo de un ataque, porque se trata de una respuesta a una solicitud no solicitada o de alguien que intenta falsificar la respuesta real.

    El tipo de aplicación determina el mejor lugar para guardar los datos que permiten a tu aplicación validar la respuesta. Por ejemplo, si una aplicación web progresiva usa un framework de SPA, podría almacenar esto en local storage, mientras que un framework de aplicación web tradicional lo almacenaría en una sesión del servidor.
  </Tab>
</Tabs>

<div id="redirect-users-to-other-urls">
  ## Redirigir a los usuarios a otras URL
</div>

A veces, la URL de devolución de llamada no es necesariamente adonde quieres redirigir a los usuarios después de la autenticación. Por ejemplo, si un usuario intenta acceder a una página protegida de tu aplicación y esa acción desencadena la solicitud de autenticación, puedes almacenar esa URL para redirigirlo de vuelta a la página a la que quería ir una vez finalizada la autenticación. Almacena la URL deseada mediante los siguientes métodos:

* [Redirigir a los usuarios con parámetros de estado](/docs/es/secure/attack-protection/state-parameters)
* [Redirigir a los usuarios desde reglas](/docs/es/customize/rules/redirect-users)

Elige la opción que mejor funcione para tu tipo de aplicación y el tipo de [flujo](/docs/es/get-started/authentication-and-authorization-flow/which-oauth-2-0-flow-should-i-use) que estés usando. Crea la lógica necesaria en tu aplicación para recuperar la URL almacenada y redirigir a tus usuarios adonde quieras que vayan. Los [SDKs de Auth0](/docs/es/libraries) también incluyen soporte para URL de redirección.
