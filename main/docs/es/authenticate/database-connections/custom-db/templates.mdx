Si tienes tu propia base de datos (conocida en Auth0 como un almacén de datos heredado) que contiene datos de identidad de usuario, puedes usarla como un <Tooltip tip="Proveedor de identidad (IdP): servicio que almacena y gestiona identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=identity+provider">proveedor de identidad</Tooltip> para autenticar usuarios.
Creas y configuras la conexión a tu almacén de datos heredado como una base de datos personalizada en Auth0. Puedes optar por migrar gradualmente los datos de tu base de datos heredada al almacén de datos de Auth0 con el tiempo, o seguir usándola sin migrar datos. Proporcionamos plantillas de scripts para realizar funciones en la base de datos personalizada que puedes usar y personalizar.

Hay dos tipos diferentes de scripts de acción de base de datos personalizada:

* **Migración automática**: Cada vez que un usuario inicia sesión en Auth0, si el usuario aún no está en Auth0, el script comprobará la base de datos heredada para ver si el usuario existe allí. Si lo encuentra y el indicador **Import users to Auth0** está activado, los datos del usuario se migran al almacén de datos de Auth0. Esta capacidad a veces se denomina **migración gradual** o **migración perezosa**.
* **Base de datos heredada**: Auth0 siempre consultará la base de datos subyacente cuando un usuario intente iniciar sesión, se cree, cambie su contraseña, verifique su correo electrónico o se elimine. Si se encuentra y el indicador **Import users to Auth0** **no** está activado, los datos de usuario permanecen en la base de datos heredada y **no** migran a Auth0.

<div id="script-execution">
  ## Ejecución de scripts
</div>

Como se describe en [Custom Database Connections](/docs/es/authenticate/database-connections/custom-db), un tipo de conexión de base de datos personalizada le permite configurar scripts de acción: código personalizado que se utiliza para interactuar con su repositorio de identidades heredado. Cada script de acción es, en esencia, una función JavaScript con nombre a la que se le pasa una serie de parámetros; tanto el nombre de la función como los parámetros que se pasan dependen del script.

<div id="limits">
  ### Límites
</div>

La ejecución de scripts de acción admite la naturaleza asíncrona de JavaScript, y se pueden usar construcciones como los objetos [Promise](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/Promise). El procesamiento asíncrono implica, en la práctica, una suspensión mientras se completa una operación, y un contenedor serverless de Webtask de Auth0 suele tener un límite de ejecución de 20 segundos, tras el cual el contenedor puede reciclarse. Si un contenedor se recicla debido a este límite, la operación finalizará prematuramente, lo que en última instancia hará que se devuelva un error (además de poder provocar un posible restablecimiento del objeto `global`).

<div id="completion-and-the-callback-function">
  ### Finalización y la función `callback`
</div>

La función `callback` proporcionada a cada script de acción actúa, en la práctica, como una señal para indicar que la operación ha finalizado. Un script de acción debe completarse inmediatamente después de llamar a la función `callback` (ya sea de forma implícita o ejecutando explícitamente una instrucción `return` de JavaScript) y no debe realizar ninguna otra operación.

<Warning>
  La función `callback` proporcionada por Auth0 debe llamarse exactamente **una** vez; llamar a la función más de una vez dentro de un script de acción dará lugar a resultados impredecibles y/o errores.
</Warning>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Cuando `callback` se ejecuta sin parámetros, como en `callback()`, esto implica que la función se ha llamado como si se hubiera ejecutado `callback(null)`.
</Callout>

Si un script de acción usa procesamiento asíncrono, la llamada a la función `callback` debe posponerse hasta que dicho procesamiento finalice, y debe ser lo último que se llame. La ejecución asíncrona hará que una `callback` de JavaScript se ejecute una vez completada la operación asíncrona; normalmente, esta devolución de llamada se activa en algún momento después de que finaliza el cuerpo principal (sincrónico) de una función de JavaScript.

<Warning>
  No ejecutar la función `callback` provocará que la ejecución se bloquee y, en última instancia, se devuelva una condición de error. El script de acción debe llamar a la función `callback` exactamente una vez: debe llamarse al menos una vez para evitar que la ejecución se bloquee; sin embargo, no debe llamarse más de una vez, ya que de lo contrario se producirán resultados impredecibles y/o errores.
</Warning>
