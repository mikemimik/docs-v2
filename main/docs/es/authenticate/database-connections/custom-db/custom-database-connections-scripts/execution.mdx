Un tipo de conexión de base de datos personalizada le permite configurar scripts de acción que contienen código personalizado que Auth0 utiliza para interactuar con su repositorio de identidades heredado. Los scripts de acción son funciones de JavaScript con nombre que aceptan un conjunto predefinido de parámetros.

<div id="webtask-containers">
  ## Contenedores de Webtask
</div>

Los scripts de acción se ejecutan en un contenedor de Webtask individual, con un límite de ejecución de aproximadamente 20 segundos. Una vez que las funciones se ejecutan, el contenedor se recicla.

Cuando un contenedor se recicla, finaliza las operaciones pendientes del script de acción. Esto puede hacer que se devuelva un error y que el objeto `global` se restablezca. Para obtener más información sobre el objeto `global`, consulta [Prácticas recomendadas del entorno de scripts de acción de base de datos personalizada](/docs/es/authenticate/database-connections/custom-db/custom-database-connections-scripts/environment).

<div id="asynchronous-features">
  ## Características asíncronas
</div>

Las características asíncronas de JavaScript, incluidos los objetos [Promise](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/Promise) y las [funciones async](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Statements/async_function), son compatibles con los scripts de acción.

Puede usar características asíncronas para ejecutar operaciones sin bloqueo dentro de un script de acción, pero asegúrese de que estas operaciones no superen el límite de ejecución del contenedor de Webtask. Cuando se recicle el contenedor de Webtask, se finalizarán las operaciones pendientes, lo que puede provocar comportamientos inesperados o errores.

Si realiza una llamada a un servicio externo o a una API dentro de su script de acción, configure la función para que agote el tiempo de espera tras un intervalo razonable y [devuelva un error](#error-handling) si no se puede acceder al servicio externo o a la API.

<div id="examples">
  ### Ejemplos
</div>

Auth0 permite usar objetos [Promise](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/Promise) y [funciones async](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Statements/async_function) en los scripts de acción.

<div id="promise-object">
  #### Objeto Promise
</div>

Este ejemplo utiliza el método `fetch` integrado de JavaScript, que obtiene un recurso de una red y luego devuelve un objeto Promise. Está escrito con [encadenamiento de promesas](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Guide/Using_promises#chaining).

```javascript lines expandable
function login(userNameOrEmail, password, callback) {
	const hashedPassword = hash(password);
	const apiEndpoint = 'https://example.com/api/authenticate';
	const options = {
		method: 'POST',
		body: {
			email: userNameOrEmail,
			password: hashedPassword
		}
	};

	fetch(apiEndpoint, options) 
		.then((response) => {
			if (!response.ok) {
				return callback(new Error(`HTTP error! Status: ${response.status}`));
			}

			return response.json();
		})
		.then((response) => {
			if (response.err) {
				return callback(new Error(`Error authenticating user: ${err}`));
			}

			let profile = {
				email: response.profileData.email,
				username: response.profileData.username
			};

			return callback(null, profile);
		})
		.catch((err) => {
			return callback(new Error(`An error occurred: ${err}`));
		});
  }
```

<div id="async-function">
  #### Función async
</div>

Este ejemplo utiliza el método `fetch` integrado de JavaScript, que obtiene un recurso de la red y luego devuelve un objeto Promise, y está escrito con [funciones async](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Statements/async_function).

```js lines expandable
async function login(userNameOrEmail, password, callback) {
	const hashedPassword = hash(password);
	const apiEndpoint = 'https://example.com/api/authenticate';
	const options = {
		method: 'POST',
		body: {
			email: userNameOrEmail,
			password: hashedPassword
		}
	};

	const response = await fetch(apiEndpoint, options);

	if (!response.ok) {
		return callback(new Error(`HTTP error! Status: ${response.status}`));
	}

	const result = response.json();

	if (result.err) {
		return callback(new Error(`Error authenticating user: ${err}`));
	}

	let profile = {
		email: response.profileData.email,
		username: response.profileData.username
	};

	return callback(null, profile);
}
```

<div id="callback-function">
  ## Función de devolución de llamada
</div>

La función `callback` indica que la operación del script de acción ha finalizado y debe llamarse exactamente una vez. Un script de acción debería finalizar inmediatamente después de llamar a la función `callback`, preferiblemente mediante el uso explícito de la instrucción `return`.

<Warning>
  Si la función `callback` se llama más de una vez, puede dar lugar a resultados inesperados y/o errores.

  Si no se llama a la función `callback`, la ejecución del script de acción se quedará bloqueada y se devolverá un error cuando se recicle el contenedor de Webtask.
</Warning>

<div id="asynchronous-processing">
  ### Procesamiento asíncrono
</div>

Si un script de acción utiliza procesamiento asíncrono, la función `callback` debe llamarse una vez que se hayan completado todas las operaciones asíncronas.

<div id="parameters">
  ### Parámetros
</div>

Si se llama a la función `callback` sin ningún parámetro, se ejecutará como si se hubiera pasado un parámetro `null`.

<div id="size">
  ## Tamaño
</div>

El tamaño total de la implementación de cualquier script de acción no debe superar los 100 kB. Esta limitación de tamaño no incluye los módulos `npm` importados. Para obtener más información sobre los módulos `npm`, consulta [Prácticas recomendadas del entorno de scripts de acción de Custom Database](/docs/es/authenticate/database-connections/custom-db/custom-database-connections-scripts/environment).

Cuanto mayor sea el tamaño de un script, mayor será la latencia debido al proceso de empaquetado y transporte que utiliza la plataforma Webtask. El tamaño afecta al rendimiento del sistema.

<div id="anonymous-functions">
  ## Funciones anónimas
</div>

Los scripts de acción pueden implementarse como funciones anónimas, pero no se recomienda hacerlo. Las funciones anónimas dificultan la depuración del script de acción y la interpretación de la pila de llamadas que se genera cuando se produce un error excepcional. Para obtener más información sobre las funciones anónimas, consulta [IIFE en MDN Web Docs](https://developer.mozilla.org/en-US/docs/Glossary/IIFE).

<div id="error-handling">
  ## Manejo de errores
</div>

Pasa un objeto `Error` a la función `callback` con un mensaje descriptivo:

```js lines
return callback(new Error('My custom error message'));
```

<div id="security">
  ## Seguridad
</div>

<div id="database-interface-vs-api">
  ### Interfaz de base de datos vs. API
</div>

Asegúrese de proteger todas las comunicaciones entre Auth0 y su repositorio de identidades heredado. Si su repositorio de identidades heredado aún no tiene una API implementada, es muy recomendable que implemente una.

Si su repositorio de identidades heredado tiene una API disponible, puede [registrar la API](/docs/es/get-started/auth0-overview/set-up-apis) a través de Auth0 y [crear una Action](/docs/es/manage-users/access-control/sample-use-cases-actions-with-authorization#deny-access-to-anyone-calling-an-api) para restringir el acceso de los usuarios finales.

Si su repositorio de identidades heredado no tiene una API disponible, y no es viable implementar una, puede seguir comunicándose directamente con su base de datos. Asegúrese de [agregar las direcciones IP de Auth0 a la lista de permitidos de su firewall](/docs/es/secure/security-guidance/data-security/allowlist) para permitir el tráfico entrante desde Auth0.

<div id="identity-provider-tokens">
  ## Tokens del proveedor de identidad
</div>

Si el objeto `user` devuelve las propiedades `access_token` y `refresh_token`, Auth0 las gestiona de forma diferente a otros tipos de información del usuario. Auth0 las almacena en la propiedad `identities` del objeto `user`:

```json lines
{
	"email": "you@example.com",
	"updated_at": "2019-03-15T15:56:44.577Z",
	"user_id": "auth0|some_unique_id",
	"nickname": "a_nick_name",
	"identities": [ 
		{
			"user_id": "some_unique_id",
			"access_token": "e1b5.................92ba",
			"refresh_token": "a90c.................620b",
			"provider": "auth0", 
			"connection": "custom_db_name",
			"isSocial": false 
		}
  ], 
  "created_at": "2019-03-15T15:56:44.577Z",
  "last_ip": "192.168.1.1",
  "last_login": "2019-03-15T15:56:44.576Z",
  "logins_count": 3
}
```

Si quieres recuperar cualquiera de estas propiedades con la <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip> de Auth0, incluye el scope `read:user_idp_tokens` al [solicitar un token de acceso](/docs/es/secure/tokens/access-tokens/management-api-access-tokens/get-management-api-access-tokens-for-production).
