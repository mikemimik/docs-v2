<div id="access-legacy-identity-storage-via-custom-api">
  ## Acceso al almacenamiento de identidades heredadas mediante una API personalizada
</div>

Recomendamos implementar una API para aplicar el principio de mínimo privilegio a tu almacenamiento de identidades heredadas, en lugar de simplemente abrir un acceso general a través de internet.

Proteger el almacenamiento de identidades heredadas frente al acceso general es una práctica recomendada. Exponer directamente a internet una base de datos (de identidades heredadas), por ejemplo, puede ser extremadamente problemático: las interfaces de bases de datos para SQL y similares son extremadamente abiertas en términos de funcionalidad, lo que vulnera el principio de mínimo privilegio en materia de seguridad.

La alternativa es crear una API (personalizada) sencilla, protegida mediante el uso de un <Tooltip tip="Token de acceso: credencial de autorización, en forma de una cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=access+token">token de acceso</Tooltip>, a la que pueda llamar cada script de acción. Esto actuaría como interfaz para el almacén de identidades heredado. Luego, se puede usar el flujo de concesión de credenciales de cliente para obtener un token de acceso desde un script, y este puede almacenarse posteriormente en caché para reutilizarlo dentro del objeto `global` y mejorar el rendimiento. La API puede entonces ofrecer un número reducido de endpoints protegidos que realicen únicamente la funcionalidad de gestión de identidades heredadas necesaria (por ejemplo, leer un usuario, cambiar la contraseña).

De forma predeterminada, Auth0 te dará un token para cualquier API si te autenticas correctamente e incluyes el <Tooltip tip="Audience: identificador único de la audiencia de un token emitido. Llamado aud en un token, su valor contiene el ID de una aplicación (ID de cliente) para un token de ID o de una API (identificador de API) para un token de acceso." cta="Ver glosario" href="/docs/es/glossary?term=audience">audience</Tooltip> adecuado. Restringir el acceso a la API del almacén de identidades heredado limitando la asignación de tokens de acceso mediante el uso de una rule evitará usos no autorizados y mitigará varios escenarios de ataque, como aquellos en los que se intercepta la redirección a `/authorize` y se añade el audience de la API.

<Warning>
  Restringir el acceso a la API mediante Rule mitigará escenarios de ataque, como aquellos en los que se intercepta la redirección a `/authorize` y se añade el audience de la API, y garantizará que solo se conceda acceso mediante credenciales de cliente específicas.
</Warning>

<div id="access-to-legacy-identity-storage">
  ## Acceso al almacenamiento de identidades heredadas
</div>

Tanto si administra el acceso al almacenamiento de identidades heredadas mediante una API personalizada como si usa la interfaz nativa disponible, debe restringirlo a la lista de direcciones IP asociadas a su tenant de Auth0. Para ver la lista de direcciones IP, consulte [Direcciones IP de Auth0 para listas de permitidos](/docs/es/secure/security-guidance/data-security/allowlist). Agregar URL a la lista de permitidos restringe el acceso al almacén de identidades heredado y garantiza que solo se permitan los scripts de acciones de base de datos personalizada definidos en Auth0.

<Warning>
  La lista de permitidos de direcciones IP de Auth0 se comparte entre todos los tenants de Auth0 asignados a una región. Nunca use la lista de permitidos como único método para proteger el acceso a su almacén de identidades heredado; hacerlo podría crear vulnerabilidades de seguridad que permitan el acceso no autorizado a sus usuarios.
</Warning>
