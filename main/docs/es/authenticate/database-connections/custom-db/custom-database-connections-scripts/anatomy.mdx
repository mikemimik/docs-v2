---

Normalmente, usa una conexión de base de datos personalizada para proporcionar acceso a su propio almacén de identidades heredado para la autenticación (a veces denominada **autenticación heredada**) o [realizar la importación de usuarios mediante migración automática](/docs/es/manage-users/user-migration/configure-automatic-migration-from-your-database) (a menudo denominada migración **trickle** o **lazy**). También puede usar conexiones de base de datos personalizadas para actuar como proxy de acceso a un tenant de Auth0 en escenarios en los que usa una arquitectura multi-tenant de Auth0. Para obtener más información, lea [Prácticas recomendadas para aplicaciones multi-tenant](/docs/es/get-started/auth0-overview/create-tenants/multi-tenant-apps-best-practices).

Normalmente crea y configura conexiones de base de datos personalizadas mediante el <Tooltip tip="Auth0 Dashboard: El producto principal de Auth0 para configurar sus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip>. Crea una conexión de base de datos y luego activa **Use my own database** para habilitar la edición de los scripts de acción de base de datos. Una conexión de base de datos personalizada también se puede crear y configurar con el <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip> de Auth0, el endpoint [Create a connection](https://auth0.com/docs/api/management/v2#!/Connections/post_connections) y la estrategia `auth0`.

Como se muestra a continuación, usa conexiones de base de datos personalizadas como parte de un flujo de inicio de sesión para obtener información sobre la identidad del usuario desde su propio almacén de identidades heredado para la autenticación o la importación de usuarios.

<Frame>![Flujo de conexión de base de datos personalizada](/docs/images/cdy7uua7fh8z/1H2Ky1WyPlZ9zHZrEVlFhP/4e4856395fac6d6339c883c35d4b4ac0/custom-database-connections.png)</Frame>

Además de los artefactos comunes a todos los tipos de conexiones de base de datos, una conexión de base de datos personalizada le permite configurar scripts de acción: código personalizado que se usa al interactuar con almacenes de identidades heredados. Los scripts que elija configurar dependen de si está creando una conexión para autenticación heredada o para migración automática.

Los scripts de acción pueden implementarse como funciones anónimas; sin embargo, las funciones anónimas dificultan la depuración cuando se trata de interpretar el call-stack generado como resultado de cualquier condición de error excepcional. Para mayor comodidad, recomendamos proporcionar un nombre de función para cada script de acción. Para ver algunos nombres recomendados, lea [Prácticas recomendadas de ejecución de scripts de acción de base de datos personalizada](/docs/es/authenticate/database-connections/custom-db/custom-database-connections-scripts/execution).

En un escenario de autenticación heredada, no se crea ningún nuevo registro de usuario; el usuario permanece en el almacén de identidades heredado y Auth0 usa la identidad que contiene al autenticar al usuario. Las conexiones de base de datos personalizadas también se usan fuera del flujo de <Tooltip tip="Universal Login: Su aplicación redirige a Universal Login, alojado en el servidor de autorización de Auth0, para verificar la identidad de un usuario." cta="Ver glosario" href="/docs/es/glossary?term=Universal+Login">Universal Login</Tooltip>. Por ejemplo, el script de acción `changePassword` de una conexión se llama cuando se realiza una operación de cambio de contraseña para un usuario que reside en un almacén de identidades heredado.

<div id="automatic-migration">
  ## Migración automática
</div>

Durante la migración automática, Auth0 crea un usuario nuevo en un repositorio de identidades (base de datos) administrado por Auth0. Auth0 utiliza la identidad del repositorio de identidades administrado por Auth0 al autenticar al usuario. Para que esto ocurra, Auth0 primero requiere que el usuario se autentique en el almacén de identidades heredado, y solo si esto se realiza correctamente se creará el nuevo usuario en la base de datos administrada por Auth0. Auth0 crea el nuevo usuario con el mismo ID y contraseña que se proporcionaron durante la autenticación.

La creación de usuarios en un escenario de migración automática suele producirse después de que finalice el script de acción Login. Recomendamos no intentar eliminar usuarios de un almacén de identidades heredado como una operación en línea dentro del script Login, sino hacerlo como un proceso independiente. Esto evita la eliminación accidental de usuarios si se produce algún error durante la migración.

Con la migración automática, los usuarios permanecen en el almacén de identidades heredado y pueden eliminarse o archivarse si es necesario. Puede producirse un efecto secundario por el que un usuario se elimina de Auth0, pero permanece en el almacén de datos heredado. En este caso, un inicio de sesión del usuario eliminado podría hacer que se ejecute el script Login o Get User y que el usuario vuelva a migrarse desde el almacén de identidades heredado.

Recomendamos marcar las identidades de usuario del repositorio heredado como migradas antes de que finalicen los scripts Login o Get User, y antes de cualquier eliminación posterior del repositorio heredado, para evitar la recreación accidental de usuarios eliminados intencionadamente.

<div id="size">
  ## Tamaño
</div>

Recomendamos que el tamaño total de cualquier script de acción no supere los 100 KB. Cuanto mayor sea el tamaño, mayor será la latencia introducida por el proceso de empaquetado y transporte que utiliza la plataforma Auth0, lo que afectará al rendimiento de tu sistema. Ten en cuenta que el límite de 100 KB no incluye ningún módulo de `npm` al que se haga referencia en instrucciones `require`.
