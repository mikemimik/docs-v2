Normalmente, usa una conexión de Custom Database para proporcionar acceso a su propio almacén de identidades heredado para la autenticación (a veces denominada **autenticación heredada**) o [realizar la importación de usuarios mediante migración automática](/docs/es/manage-users/user-migration/configure-automatic-migration-from-your-database) (a menudo denominada migración **trickle** o **lazy**). También puede usar conexiones de Custom Database para actuar como proxy del acceso a un tenant de Auth0 en escenarios en los que usa una arquitectura multi-tenant de Auth0. Para obtener más información, lea [Prácticas recomendadas para aplicaciones multi-tenant](/docs/es/get-started/auth0-overview/create-tenants/multi-tenant-apps-best-practices).

Normalmente crea y configura conexiones de Custom Database mediante el <Tooltip tip="Auth0 Dashboard: El producto principal de Auth0 para configurar sus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip>. Crea una conexión de base de datos y luego activa **Use my own database** para habilitar la edición de los scripts de acción de base de datos. Una conexión de Custom Database también se puede crear y configurar con la Auth0 <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip>, el [endpoint Create a connection](https://auth0.com/docs/api/management/v2#!/Connections/post_connections) y la estrategia `auth0`.

Como se muestra a continuación, usa conexiones de Custom Database como parte de un workflow de login para obtener información de identidad del usuario de su propio almacén de identidades heredado para la autenticación o la importación de usuarios.

<Frame>![Flujo de conexión de Custom Database](/docs/images/cdy7uua7fh8z/1H2Ky1WyPlZ9zHZrEVlFhP/4e4856395fac6d6339c883c35d4b4ac0/custom-database-connections.png)</Frame>

Además de los artefactos comunes a todos los tipos de conexiones de base de datos, una conexión de Custom Database le permite configurar scripts de acción: código personalizado usado al interactuar con almacenes de identidades heredados. Los scripts que elija configurar dependen de si está creando una conexión para autenticación heredada o para migración automática.

Los scripts de acción se pueden implementar como funciones anónimas; sin embargo, las funciones anónimas dificultan la depuración a la hora de interpretar la pila de llamadas generada como resultado de cualquier condición de error excepcional. Para mayor comodidad, recomendamos proporcionar un nombre de función para cada script de acción. Para ver algunos nombres recomendados, lea [Prácticas recomendadas para la ejecución de scripts de acción de base de datos personalizados](/docs/es/authenticate/database-connections/custom-db/custom-database-connections-scripts/execution).

En un escenario de autenticación heredada, no se crea ningún registro de usuario nuevo; el usuario permanece en el almacén de identidades heredado y Auth0 usa la identidad que contiene al autenticar al usuario. Las conexiones de Custom Database también se usan fuera del workflow de <Tooltip tip="Universal Login: Su aplicación redirige a Universal Login, alojado en el servidor de autorización de Auth0, para verificar la identidad de un usuario." cta="Ver glosario" href="/docs/es/glossary?term=Universal+Login">Universal Login</Tooltip>. Por ejemplo, el script de acción `changePassword` de una conexión se llama cuando se produce una operación de cambio de contraseña para un usuario que reside en un almacén de identidades heredado.

<div id="automatic-migration">
  ## Migración automática
</div>

Durante la migración automática, Auth0 crea un usuario nuevo en un almacén de identidades (base de datos) gestionado por Auth0. Auth0 utiliza la identidad del almacén de identidades gestionado por Auth0 al autenticar al usuario. Para que esto ocurra, Auth0 primero debe autenticar al usuario en el almacén de identidades heredado, y solo si esto se realiza correctamente se creará el nuevo usuario en la base de datos gestionada por Auth0. Auth0 crea el nuevo usuario con el mismo ID y contraseña que se proporcionaron durante la autenticación.

La creación de usuarios en un escenario de migración automática suele producirse después de que finaliza el script de acción Login. Recomendamos no intentar eliminar usuarios de un almacén de identidades heredado como una operación en línea dentro del script Login, sino hacerlo como un proceso independiente. Esto evita la eliminación accidental de usuarios si se produce algún error durante la migración.

Con la migración automática, los usuarios permanecen en el almacén de identidades heredado y pueden eliminarse o archivarse si es necesario. Puede producirse el efecto secundario de que un usuario se elimine de Auth0, pero permanezca en el almacén de datos heredado. En este caso, un login del usuario eliminado podría hacer que se ejecute el script Login o Get User y que el usuario vuelva a migrarse desde el almacén de identidades heredado.

Recomendamos marcar como migradas las identidades de usuario del almacén heredado antes de que finalicen los scripts Login o Get User, y antes de cualquier eliminación posterior del almacén heredado, para evitar la recreación no intencionada de usuarios eliminados deliberadamente.

<div id="size">
  ## Tamaño
</div>

Recomendamos que el tamaño total de cualquier script de acción no supere los 100 KB. Cuanto mayor sea su tamaño, mayor será la latencia que se introduzca debido al proceso de empaquetado y transporte que utiliza la plataforma Auth0, lo que afectará al rendimiento de su sistema. Tenga en cuenta que el límite de 100 KB no incluye ningún módulo de `npm` al que se haga referencia en una instrucción `require`.
