<Card title="La disponibilidad varía según el plan de Auth0">
  La disponibilidad de esta función depende de tu plan de Auth0 o de tu acuerdo personalizado. Para obtener más información, consulta [Pricing](https://auth0.com/pricing).
</Card>

Usa una conexión de base de datos personalizada cuando quieras dar acceso a tu propio almacén independiente de datos de identidad (legacy) para los siguientes fines:

* **Autenticación**: Usa tu base de datos como <Tooltip tip="Proveedor de identidad (IdP): servicio que almacena y administra identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=identity+provider">proveedor de identidad</Tooltip> en Auth0 para autenticar usuarios. (Esto se conoce como autenticación heredada).
* **Importar usuarios**: Usa la migración automática (migración progresiva o diferida)
* **Acceso mediante proxy a un tenant de Auth0**: Usa la arquitectura multi-tenant de Auth0.

Puedes crear y configurar una conexión de base de datos personalizada realizando una de las siguientes tareas:

1. Usa el endpoint [Create connections](https://auth0.com/docs/api/management/v2#!/Connections/post_connections) con la strategy `auth0`.
2. Ve a [Auth0 Dashboard &gt; Authentication &gt; Database](https://manage.auth0.com/#/connections/database), crea la conexión y habilita la opción **Use my own database** para permitir la edición del script de acción de la base de datos.

   <Frame>![Habilitar la opción Use my own database en Custom Database](/docs/images/cdy7uua7fh8z/3kgHDpBFdVWNq9XOfhsTXI/2efa43e44793b8dd69b9f9e8b54bf752/2025-02-25_10-20-20.png)</Frame>

<div id="how-it-works">
  ## Cómo funciona
</div>

Como se muestra en el diagrama siguiente, utilizas conexiones de base de datos personalizadas como parte del flujo de <Tooltip tip="Universal Login: Tu aplicación redirige a Universal Login, alojado en el servidor de autorización de Auth0, para verificar la identidad de un usuario." cta="Ver glosario" href="/docs/es/glossary?term=Universal+Login">Universal Login</Tooltip> para obtener información sobre la identidad del usuario desde tu propio almacén de identidades heredado.

<Frame>![Anatomía de las conexiones de base de datos personalizadas](/docs/images/cdy7uua7fh8z/2lHqvZKFiEbAXURU2gmchc/626cac94211c266f2135a41456b2e49d/custom-database-connections.png)</Frame>

Además de los elementos comunes a todos los tipos de [conexión de base de datos](/docs/es/authenticate/database-connections), una conexión de base de datos personalizada te permite configurar scripts de acción: código personalizado que se utiliza para interactuar con tu almacén de identidades heredado. Auth0 proporciona [plantillas de scripts de acción de base de datos personalizada](/docs/es/authenticate/database-connections/custom-db/templates) para la configuración, y las que utilices dependerán de si estás creando una conexión de base de datos personalizada para autenticación heredada o para migración automática.

<Card title="Práctica recomendada">
  Los scripts de acción pueden implementarse como funciones anónimas; sin embargo, estas dificultan la depuración a la hora de interpretar la pila de llamadas generada como resultado de cualquier error excepcional. Por comodidad, recomendamos asignar un nombre de función a cada script de acción.
</Card>

<div id="legacy-authentication-scenario">
  ### Escenario de autenticación heredada
</div>

En un escenario de autenticación heredada, se crea un nuevo registro de usuario en Auth0 durante el primer inicio de sesión del usuario, pero Auth0 no almacena un hash de la contraseña del usuario. Auth0 siempre usará el almacén de identidades heredado y la identidad que contiene al autenticar al usuario.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Las conexiones de base de datos personalizadas también se usan fuera del flujo de trabajo de Universal Login. Por ejemplo, el script de acción `changePassword` de una conexión se invoca cuando se realiza una operación de cambio de contraseña para un usuario que se encuentra en un almacén de identidades heredado.
</Callout>

<div id="automatic-migration-scenario">
  ### Escenario de migración automática
</div>

Durante la migración automática o gradual, Auth0 crea un nuevo usuario en un almacén de identidades (base de datos) administrado por Auth0. A partir de ese momento, Auth0 siempre utiliza la identidad del almacén de identidades administrado por Auth0 para autenticar al usuario. Para que esto ocurra, primero Auth0 requiere que el usuario se autentique en el almacén de identidades heredado, y solo si esto se realiza correctamente se creará el nuevo usuario. El nuevo usuario se creará con el mismo ID y contraseña proporcionados durante la autenticación.

<Card title="práctica recomendada">
  La creación de un usuario en un escenario de migración automática suele producirse después de que finaliza el script de acción `login`. Por lo tanto, recomendamos no intentar eliminar a un usuario del almacén de identidades heredado como una operación en línea (es decir, dentro del script `login`), sino realizar la eliminación como un proceso independiente. Esto evitará la eliminación accidental de un usuario si se produce un error durante el proceso de migración.
</Card>

En un escenario de migración automática, los usuarios permanecen en el almacén de identidades heredado y pueden eliminarse o archivarse si es necesario. Uno de los efectos secundarios puede darse si un usuario se elimina de Auth0, pero sigue presente en el almacén de identidades heredado. En este caso, un inicio de sesión del usuario eliminado podría hacer que se ejecute el script `login` y/o `getUser`, y que el usuario vuelva a migrarse desde el almacén de identidades heredado.

<Card title="práctica recomendada">
  Recomendamos marcar cualquier identidad de usuario heredada como migrada antes de que finalicen `login` o `getUser` y antes de cualquier posible eliminación del almacén heredado.
</Card>

<div id="size">
  ## Tamaño
</div>

El tamaño total de la implementación de cualquier script de acción no debe superar los 100 kB. Cuanto mayor sea su tamaño, más latencia se introducirá debido al proceso de empaquetado y transporte que utiliza la plataforma serverless Webtask de Auth0, y esto afectará al rendimiento de su sistema.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  El límite de 100 kB no incluye ningún módulo de `npm` que se referencie en alguna instrucción `require`.
</Callout>

<div id="environment">
  ## Entorno
</div>

Los scripts de acción se ejecutan como una serie de funciones de JavaScript invocadas en una instancia de un contenedor serverless de Webtask. En este contexto, se proporciona un entorno específico, junto con una serie de artefactos suministrados tanto por el contenedor de Webtask como por el servidor de <Tooltip tip="Servidor de autenticación: servidor que confirma o rechaza la identidad de un usuario." cta="Ver glosario" href="/docs/es/glossary?term=authentication+server">autenticación</Tooltip> de Auth0 (tu tenant de Auth0).

<div id="npm-modules">
  ### módulos de npm
</div>

Los contenedores serverless de Webtask de Auth0 pueden usar una [amplia gama de módulos de npm](https://auth0-extensions.github.io/canirequire/); los módulos de `npm` no solo reducen el tamaño total del código de implementación de los scripts de acción, sino que también proporcionan acceso a una amplia variedad de funcionalidades ya creadas.

Muchos módulos de `npm` de acceso público son compatibles de forma predeterminada. La lista se ha recopilado y revisado para detectar posibles problemas de seguridad. Si necesitas un módulo de `npm` que no sea compatible de forma predeterminada, puedes enviar una solicitud a través del [portal de soporte de Auth0](https://support.auth0.com) o de tu representante de Auth0. Auth0 evaluará tu solicitud para determinar si es adecuado. Actualmente, Auth0 no ofrece soporte para el uso de módulos de `npm` de repositorios privados.

<div id="variables">
  ### Variables
</div>

Los scripts de acción de Auth0 admiten variables de entorno, a las que se accede mediante el objeto `configuration`, disponible globalmente. Para obtener más información, consulta la sección Agregar parámetros de configuración en [Crear conexiones de base de datos personalizadas](/docs/es/authenticate/database-connections/custom-db/create-db-connection).

<Card title="Práctica recomendada">
  El objeto `configuration` debe tratarse como de solo lectura y usarse para almacenar información confidencial, como credenciales o claves de API, para acceder a almacenes de identidades externos. Esto evita tener valores sensibles de seguridad codificados directamente en un script de acción.
</Card>

El objeto de configuración también puede usarse para respaldar cualquier estrategia del [ciclo de vida de desarrollo de software (SDLC)](/docs/es/get-started/auth0-overview/create-tenants/set-up-multiple-environments) que emplees, ya que te permite definir variables con valores específicos del tenant. Esto evita valores codificados directamente en un script de acción que pueden cambiar según el tenant que lo ejecute.

<div id="global-object">
  ### objeto global
</div>

Los contenedores serverless de Webtask de Auth0 se aprovisionan a partir de un grupo asociado a cada tenant de Auth0. Cada instancia de contenedor pone a disposición el objeto `global`, al que se puede acceder desde todos los scripts de acción que se ejecutan dentro de esa instancia. El objeto `global` actúa como una variable global exclusiva del contenedor y puede usarse para definir información o funciones que utilicen todos los scripts de acción que se ejecutan en la instancia del contenedor.

Esto significa que el objeto `global` puede usarse para almacenar en caché recursos costosos, siempre que no sean específicos del usuario. Por ejemplo, podrías usarlo para almacenar un <Tooltip tip="Access Token: Credencial de autorización, en forma de una cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=Access+Token">Access Token</Tooltip> para una API de registro de terceros que proporcione funcionalidad no específica del usuario. O bien podrías almacenar un Access Token para tu propia API no específica del usuario definida en Auth0 y obtenida mediante el [Client Credentials Flow](/docs/es/get-started/authentication-and-authorization-flow/client-credentials-flow).

<Warning>
  Un script de acción puede ejecutarse en cualquiera de las instancias de contenedor que ya estén en ejecución o en una instancia de contenedor recién creada (que posteriormente puede añadirse al grupo). No existe afinidad de contenedor para la ejecución de scripts de acción en Auth0. Esto significa que debes evitar almacenar información específica del usuario en el objeto `global` y asegurarte siempre de que cualquier declaración realizada dentro del objeto `global` contemple también la inicialización.
</Warning>

Cada vez que se recicla un contenedor de Webtask, o cada vez que se crea una nueva instancia de un contenedor de Webtask, el objeto `global` que define se restablece. Por lo tanto, cualquier declaración de asignación dentro del objeto `global` asociado a un contenedor también debe contemplar la inicialización.

<Card title="Práctica recomendada">
  Para ofrecer flexibilidad de rendimiento, los contenedores serverless de Webtask se aprovisionan en Auth0 según sea necesario y también están sujetos a diversas políticas de reciclaje. En general, recomendamos no asumir que la vida útil de un objeto `global` sea superior a 20 minutos.
</Card>

<div id="security">
  ## Seguridad
</div>

<div id="access-legacy-identity-storage-via-custom-api">
  ### Acceda al almacenamiento de identidades heredado mediante una API personalizada
</div>

Proteger el almacenamiento de identidades heredado frente al acceso general es una práctica recomendada. Exponer una base de datos directamente a internet, por ejemplo, puede ser extremadamente problemático: las interfaces de base de datos, como las de SQL, son extremadamente abiertas en cuanto a funcionalidad, lo que viola el principio de privilegio mínimo desde el punto de vista de la seguridad.

<Card title="práctica recomendada">
  Le recomendamos que implemente una API que aplique el principio de privilegio mínimo a su almacén de identidades heredado (base de datos), en lugar de simplemente abrir un acceso general a través de internet.
</Card>

Una alternativa es crear una API (personalizada) sencilla, protegida mediante un token de acceso, a la que pueda llamar cada script de acción. Esta serviría como interfaz para el almacén de identidades heredado. Después, se puede usar el flujo de concesión de credenciales de cliente para obtener un token de acceso desde un script, que posteriormente puede almacenarse en caché en el objeto `global` para reutilizarlo y mejorar el rendimiento. La API puede proporcionar entonces un número limitado de endpoints protegidos que realicen únicamente la funcionalidad de administración heredada necesaria (por ejemplo, `read user`, `change password`).

<Card title="práctica recomendada">
  De forma predeterminada, Auth0 le dará un token para cualquier API si se autentica correctamente e incluye la audience adecuada. Restringir el acceso a la API del almacén de identidades heredado limitando la asignación de tokens de acceso mediante el uso de una Rule evitará usos no autorizados y mitigará diversos escenarios de ataque, como aquellos en los que se intercepta la redirección a `/authorize` y se añade la audience de la API.
</Card>

<div id="allowlist-access-to-legacy-identity-storage">
  ### Incluir en la lista de permitidos el acceso al almacenamiento de identidades heredado
</div>

Tanto si gestiona el acceso a un almacén de identidades heredado mediante una API personalizada como si usa la interfaz nativa proporcionada, restrinja el acceso a la lista de direcciones IP asociadas a su tenant de Auth0. La lista de permitidos restringe el acceso al almacén de identidades heredado y garantiza que solo estén permitidos los scripts de acción de la base de datos personalizada definidos en Auth0.

<Warning>
  La lista de permitidos de direcciones IP de Auth0 se comparte entre todos los tenants de Auth0 de una misma región. Nunca use la lista de permitidos como único método para proteger el acceso a su almacén de identidades heredado; hacerlo podría introducir vulnerabilidades de seguridad y permitir el acceso no autorizado a sus usuarios.
</Warning>
