Las claves de acceso son un método de autenticación seguro y passwordless basado en los [estándares FIDO2 (WebAuthn y CTAP)](https://en.wikipedia.org/wiki/WebAuthn). Tienen varias ventajas frente a la autenticación tradicional con identificador y contraseña:

* **Las claves de acceso permiten a los usuarios autenticarse con biometría o credenciales vinculadas al dispositivo** (como una huella digital, un PIN o un patrón), por lo que el inicio de sesión es más rápido y no requiere recordar una contraseña.

* **Las claves de acceso sincronizan las credenciales entre dispositivos** para que los usuarios no tengan que volver a registrarlas en cada dispositivo nuevo.

* **Las claves de acceso son resistentes al phishing** porque usan criptografía de clave pública, así que no hay secretos compartidos y el dispositivo del usuario genera claves únicas para cada cuenta.

* **Las claves de acceso ofrecen una recuperación más confiable** porque las credenciales almacenadas pueden seguir utilizándose aunque se pierda el dispositivo original.

* **Las claves de acceso vinculan las credenciales a un dominio específico** para que los usuarios puedan autenticarse en todo un dominio con una sola clave de acceso.

Para obtener más información sobre las claves de acceso, consulta [la descripción general sobre claves de acceso de FIDO Alliance](https://fidoalliance.org/passkeys).

<div id="about-passkeys-on-auth0">
  ## Acerca de las claves de acceso en Auth0
</div>

Auth0 admite claves de acceso como método de autenticación para conexiones de base de datos, con tres métodos de implementación según el tipo de aplicación:

* Claves de acceso de Universal Login para aplicaciones web
* [Embedded Login para aplicaciones nativas](/docs/es/authenticate/passwordless/implement-login/embedded-login/native)
* [Claves de acceso nativas para aplicaciones Android e iOS](/docs/es/authenticate/database-connections/passkeys/native-passkeys-for-mobile-applications)

Auth0 tiene un límite de 20 claves de acceso por usuario.

Cuando habilitas las claves de acceso para tu conexión de base de datos, estas pasan a estar disponibles para los usuarios durante el registro y el inicio de sesión.

Las claves de acceso no sustituyen ni invalidan las credenciales existentes de un usuario. Cuando un usuario crea su clave de acceso, esta se añade a su cuenta como método de autenticación, pero las credenciales existentes de correo electrónico o nombre de usuario y contraseña siguen siendo válidas.

<div id="passkeys-with-mfa-enabled">
  ### Claves de acceso con MFA habilitado
</div>

Si <Tooltip tip="Autenticación multifactor (MFA): proceso de autenticación de usuario que utiliza un factor adicional al nombre de usuario y la contraseña, como un código por SMS." cta="Ver glosario" href="/docs/es/glossary?term=MFA">MFA</Tooltip> está habilitado, es posible que se solicite al usuario completar un desafío MFA después de autenticarse con una clave de acceso, según la configuración y la evaluación de riesgos.

El comportamiento predeterminado es exigir que se complete un desafío MFA, independientemente de si el método de autenticación utilizado fue una contraseña o una clave de acceso. Dado el alto nivel de seguridad que proporcionan las claves de acceso, puede omitir MFA para los usuarios que se hayan autenticado con una clave de acceso a fin de reducir la fricción. Esto puede lograrse mediante una Action de Post Login.

Para obtener más información, consulte [Reducir la fricción con claves de acceso](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger#reduce-friction-with-passkeys) y [Autenticación multifactor](/docs/es/secure/multi-factor-authentication).

<div id="passkeys-with-multiple-custom-domains-mcd">
  ### Claves de acceso con múltiples dominios personalizados (MCD)
</div>

<ReleaseStageNotice feature="Múltiples dominios personalizados" stage="ea" plans="Empresarial" terms="true" />

Si tiene habilitados múltiples dominios personalizados en su tenant, Auth0 mantiene una relación uno a uno entre un dominio y la clave de acceso de ese dominio. Los usuarios con una base de datos con claves de acceso habilitadas pueden registrarse e iniciar sesión con una clave de acceso vinculada al dominio específico en el que se creó. Los usuarios solo pueden registrar una clave de acceso para un único dominio (el primero con el que la registren, entre los múltiples dominios personalizados del tenant).

Para el inicio de sesión passwordless, el dominio personalizado seleccionado debe reflejarse en el Magic Link del flujo de inicio de sesión passwordless.

<div id="relying-party-id-for-passkeys">
  ### ID de la parte de confianza para claves de acceso
</div>

El [identificador de la parte de confianza (RP ID)](https://www.w3.org/TR/webauthn-2/#relying-party-identifier) es un dominio al que WebAuthn vincula credenciales como las claves de acceso. El RP ID define qué orígenes de solicitud están permitidos para la autenticación.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  El dominio de tu aplicación debe agregarse a la lista de **Allowed Origins (CORS)** en la [configuración de la aplicación](https://manage.auth0.com/#/applications) para que el RP ID funcione correctamente.
</Callout>

Definir el RP ID como sufijo del origen permite a los usuarios autenticarse en distintos subdominios con una sola clave de acceso. Por ejemplo, si tu aplicación web se sirve desde `login.example.com` y tu aplicación nativa se sirve desde `app.example.com`, puedes configurar el RP ID como `example.com` para que los usuarios finales puedan autenticarse en ambas aplicaciones (y en cualquier otro subdominio de `example.com`) con una sola clave de acceso.

| Entorno | Dominio raíz                | RP ID         |
| :------ | :-------------------------- | :------------ |
| Web     | `https://login.example.com` | `example.com` |
| iOS     | `app.example.com`           | `example.com` |
| Android | `assetlinks.json`           | `example.com` |

Con Auth0, puedes personalizar el RP ID para el dominio raíz o el dominio principal, de modo que los usuarios puedan autenticarse en aplicaciones móviles o aplicaciones web con la misma clave de acceso. Si usas múltiples dominios personalizados, también puedes establecer `rp.id` para cada dominio personalizado.

Para obtener más información sobre cómo personalizar el RP ID, consulta [Configure Passkey Policy](/docs/es/authenticate/database-connections/passkeys/configure-passkey-policy).
