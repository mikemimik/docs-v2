<Warning>
  Habilitar Flexible Identifiers en su tenant puede introducir cambios incompatibles en su entorno de Production. Pruebe esta funcionalidad exhaustivamente en un entorno de desarrollo y tome nota de la configuración actual de su conexión antes de implementarla de forma generalizada.
</Warning>

Un **identificador flexible** es el atributo que un usuario introduce en una pantalla de inicio de sesión para autenticarse. Puede elegir entre correo electrónico, nombre de usuario, teléfono o una combinación de dos o más.

<div id="attribute-and-identifier-definitions">
  ### Definiciones de atributo e identificador
</div>

En este producto, un **atributo** es un dato de usuario que se puede almacenar, como el correo electrónico, el número de teléfono y el nombre de usuario. Todos los identificadores son atributos, pero solo algunos atributos son identificadores.

Un **identificador** es un atributo único que permite reconocer a un usuario concreto dentro de una conexión determinada. El correo electrónico, el teléfono y el nombre de usuario pueden identificar de forma única a una persona y servir como identificadores, mientras que otros atributos contribuyen al perfil del usuario sin identificarlo de forma única.

<div id="use-flexible-identifiers">
  ### Usar Flexible Identifiers
</div>

Flexible Identifiers es para acceso general con las siguientes limitaciones:

* Flexible Identifiers, incluido el atributo phone, solo está disponible con [Universal Login](/docs/es/authenticate/login/auth0-universal-login/universal-login-vs-classic-login/universal-experience) y debes [configurar un proveedor telefónico](/docs/es/customize/phone-messages/configure-phone-messaging-providers).
* Debes configurar [Identifier First](/docs/es/authenticate/login/auth0-universal-login/identifier-first) para usar la verificación por teléfono en Signup.
* El atributo de dirección de correo electrónico debe estar habilitado para usar [MFA adaptativo](/docs/es/secure/multi-factor-authentication/adaptive-mfa).
* Debes tener un correo electrónico en el perfil de usuario para usar las invitaciones de Signup de [Organizations](/docs/es/manage-users/organizations/organizations-overview).
* Los usuarios finales bloqueados por <Tooltip tip="Brute Force Protection: Forma de protección contra ataques que protege contra ataques de fuerza bruta que se producen desde una única dirección IP y tienen como objetivo una sola cuenta de usuario." cta="Ver glosario" href="/docs/es/glossary?term=Brute+Force+Protection">Brute Force Protection</Tooltip> no pueden desbloquearse por sí mismos mediante un mensaje SMS. Hay otros métodos disponibles; para obtener más información, [consulta Brute Force Protection](/docs/es/secure/attack-protection/brute-force-protection).
* Flexible Identifiers mueve el campo del identificador a la primera pantalla de inicio de sesión y cambia el prompt de restablecimiento de contraseña de correo electrónico a nombre de usuario.
* Los tokens OTP para la verificación de identificadores por teléfono y correo electrónico tienen una validez de 900 segundos.

<div id="issues-using-flexible-identifiers">
  ### Problemas al usar Flexible Identifiers
</div>

A continuación, se incluye una lista de posibles problemas que puedes encontrar al configurar y administrar Flexible Identifiers:

* Si tu aplicación no especifica el scope `phone` en la solicitud de autorización, no recibirás el claim `phone_number`. Para obtener más información sobre los scopes, [consulta Scopes](/docs/es/get-started/apis/scopes).
* Tu script de acción de base de datos personalizada Get User debe ser válido cuando **Import Users to Auth0** esté **activado.** Para obtener más información, [consulta Configurar la migración automática desde tu base de datos](/docs/es/manage-users/user-migration/configure-automatic-migration-from-your-database).
* A cada usuario se le debe asignar un nombre de usuario, una dirección de correo electrónico y un número de teléfono únicos, independientemente del tipo de conexión. El número de teléfono debe ser único aunque no se haya agregado como atributo.
* Si usas el script de acción de base de datos personalizada Change Password y quieres establecer `email` y `email_verified` en `True`, debes devolver en el objeto el estado deseado de `email_verified`. Para obtener más información, [consulta Change Password](/docs/es/authenticate/database-connections/custom-db/templates/change-password).
* Si usas una conexión de base de datos personalizada con Import Users to Auth0 desactivado, debes alinear las propiedades de tu perfil de usuario con el perfil de usuario normalizado de Auth0. Para obtener más información, [consulta Perfil de usuario normalizado](/docs/es/manage-users/user-accounts/user-profiles/normalized-user-profiles).
* Si usas una conexión de base de datos personalizada con Import Users to Auth0 activado, Auth0 comprobará la unicidad de `phone_number` y `phone_verified.`
* Los prompts de Identifier First muestran todos los identificadores en la primera pantalla y eliminan tu configuración anterior, y el prompt Reset Password mostrará el campo de entrada para Username en lugar de Email.
* Familiarízate con las prácticas recomendadas para evitar ataques de SMS Pumping. Para obtener más información, consulta nuestro [documento técnico sobre SMS Pumping](https://www.okta.com/resources/whitepaper-handling-toll-fraud-and-sms-pumping-with-twilio-in-the-okta-customer-identity/).
