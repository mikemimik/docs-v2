<Warning>
  Habilitar Identificadores flexibles en tu tenant puede introducir cambios incompatibles en tu entorno de producción. Prueba esta función a fondo en un entorno de desarrollo y toma nota de la configuración actual de tu conexión antes de implementarla de forma generalizada.
</Warning>

Un **Identificador flexible** es el atributo que un usuario introduce en una pantalla de inicio de sesión para autenticarse. Puedes elegir entre correo electrónico, nombre de usuario, número de teléfono o una combinación de dos o más.

<div id="attribute-and-identifier-definitions">
  ### Definiciones de atributos e identificadores
</div>

Para este producto, un **Atributo** es un dato de usuario que puede almacenarse, como el correo electrónico, el número de teléfono y el nombre de usuario. Todos los identificadores son atributos, pero solo algunos atributos son identificadores.

Un **Identificador** es un atributo único que permite identificar a un usuario concreto dentro de una conexión determinada. Email, phone y el nombre de usuario pueden identificar de forma única a una persona y servir como identificadores, mientras que otros atributos contribuyen al perfil del usuario sin identificarlo de forma única.

<div id="use-flexible-identifiers">
  ### Usar Identificadores flexibles
</div>

Identificadores flexibles es para acceso general con las siguientes limitaciones:

* Identificadores flexibles, incluido el atributo phone, solo está disponible con [Universal Login](/docs/es/authenticate/login/auth0-universal-login/universal-login-vs-classic-login/universal-experience) y debes [configurar un phone provider](/docs/es/customize/phone-messages/configure-phone-messaging-providers).
* Debes configurar [Identifier First](/docs/es/authenticate/login/auth0-universal-login/identifier-first) para usar la verificación por teléfono en Signup.
* El atributo de dirección de correo electrónico debe estar habilitado para usar [MFA adaptativo](/docs/es/secure/multi-factor-authentication/adaptive-mfa).
* Debes tener correo electrónico en el perfil de usuario para usar las invitaciones de Signup para [Organizations](/docs/es/manage-users/organizations/organizations-overview).
* Los usuarios finales bloqueados por <Tooltip tip="Brute Force Protection: Forma de protección contra ataques que protege frente a ataques de fuerza bruta que se producen desde una única dirección IP y se dirigen a una sola cuenta de usuario." cta="Ver glosario" href="/docs/es/glossary?term=Brute+Force+Protection">Brute Force Protection</Tooltip> no pueden desbloquearse por sí solos mediante un mensaje SMS. Hay otros métodos disponibles; para obtener más información, [consulta Brute Force Protection](/docs/es/secure/attack-protection/brute-force-protection).
* Identificadores flexibles mueve el campo de identificador a la primera pantalla de inicio de sesión y cambia el prompt de restablecimiento de contraseña de correo electrónico a nombre de usuario.
* Los tokens OTP para la verificación del identificador por teléfono y correo electrónico tienen una vigencia de 900 segundos.

<div id="issues-using-flexible-identifiers">
  ### Problemas al usar Identificadores flexibles
</div>

A continuación, se muestra una lista de posibles problemas que puedes encontrar al configurar y gestionar Identificadores flexibles:

* Si tu aplicación no especifica el scope `phone` en la solicitud de autorización, no recibirás el claim `phone_number`. Para obtener más información sobre los scopes, [consulta Scopes](/docs/es/get-started/apis/scopes).
* Tu script de acción de base de datos personalizada Get User debe ser válido cuando **Import Users to Auth0** está **activado.** Para obtener más información, [consulta Configure Automatic Migration from Your Database](/docs/es/manage-users/user-migration/configure-automatic-migration-from-your-database).
* A cada usuario se le debe asignar un nombre de usuario, una dirección de correo electrónico y un número de teléfono únicos, independientemente del tipo de conexión. El número de teléfono debe ser único aunque no se haya añadido como atributo.
* Si usas el script de acción de base de datos personalizada Change Password y quieres establecer `email` y `email_verified` en `True`, debes devolver en el objeto el valor deseado de `email_verified`. Para obtener más información, [consulta Change Password](/docs/es/authenticate/database-connections/custom-db/templates/change-password).
* Si usas una conexión de base de datos personalizada con Import Users to Auth0 desactivado, debes alinear las propiedades de tu perfil de usuario con el perfil de usuario normalizado de Auth0. Para obtener más información, [consulta Normalized User Profile](/docs/es/manage-users/user-accounts/user-profiles/normalized-user-profiles).
* Si usas una conexión de base de datos personalizada con Import Users to Auth0 activado, Auth0 comprobará que `phone_number` y `phone_verified` sean únicos.
* Los prompts Identifier First muestran todos los identificadores en la primera pantalla y eliminan tu configuración anterior, y el prompt Reset Password mostrará el campo de entrada de Username en lugar de Email.
* Familiarízate con las prácticas recomendadas para evitar ataques de SMS Pumping. Para obtener más información, lee nuestro [documento técnico sobre SMS Pumping](https://www.okta.com/resources/whitepaper-handling-toll-fraud-and-sms-pumping-with-twilio-in-the-okta-customer-identity/).
