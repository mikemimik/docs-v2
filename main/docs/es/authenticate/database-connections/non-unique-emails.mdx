---
description: Cómo configurar correos electrónicos no únicos y sus identificadores relacionados
title: Correos electrónicos no únicos
---

Los correos electrónicos no únicos permiten que varias cuentas de usuario que comparten una conexión de base de datos usen una sola dirección de correo electrónico, mientras que otro atributo (como `username` o un número de teléfono) actúa como identificador principal. Tanto si eres una madre o un padre que gestiona varias cuentas de menores con una sola dirección de correo electrónico como si tienes una pequeña empresa con un correo electrónico por ubicación, los correos electrónicos no únicos te permiten mantener una experiencia segura de inicio de sesión y Restablecimiento de contraseña específica para cada cuenta.

<div id="considerations">
  ### Consideraciones
</div>

Revisa lo siguiente para confirmar que Non-Unique Emails se ajusta a tu caso de uso.

<div id="primary-identifier-requirements">
  #### Requisitos del identificador principal
</div>

El correo electrónico no puede usarse como identificador principal cuando se utilizan correos electrónicos no únicos. Debe configurar otro atributo como identificador principal, que se usará para la autenticación, el restablecimiento de contraseña y la gestión de la cuenta.

Para obtener más información sobre identificadores y atributos, consulte [Identificadores flexibles](/docs/es/authenticate/database-connections/flexible-identifiers-and-attributes).

<div id="password-resets">
  #### Restablecimiento de contraseña
</div>

Los usuarios finales deben proporcionar su nombre de usuario, su número de teléfono o el atributo que su administrador haya configurado como atributo principal para restablecer la contraseña. Auth0 usa ese identificador principal para localizar y restablecer la cuenta asociada a una dirección de correo electrónico compartida.

<div id="irreversible-settings">
  #### Configuraciones irreversibles
</div>

Una vez que el atributo de correo electrónico se configura como no único en una conexión, no se puede volver a cambiar para que sea único. Además, el soporte para correo electrónico no único solo puede habilitarse al crear nuevas conexiones de base de datos; no es posible modificar una conexión existente, y debe actualizar su aplicación para usar el identificador principal que elija.

<div id="flexible-identifiers">
  #### Identificadores flexibles
</div>

Los identificadores flexibles deben estar habilitados en la conexión de base de datos para usar Non-Unique Emails y no se puede deshabilitar después de crear la conexión. Cuando habilitas Non-Unique Emails con la <Tooltip tip="" cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip>, los identificadores flexibles se configuran automáticamente.

<div id="api-behavior-changes">
  ### Cambios en el comportamiento de la API
</div>

`GET /api/v2/users-by-email` devuelve todos los usuarios que comparten la misma dirección de correo electrónico.

`DELETE /api/v2/connections/{id}/users` no es compatible con conexiones con correo electrónico no único.

`POST /dbconnections/change_password` no funciona con conexiones con correo electrónico no único porque requiere una dirección de correo electrónico única para encontrar la cuenta de usuario. Los usuarios deben usar flujos que utilicen el identificador principal para restablecer sus contraseñas.

<div id="enable-non-unique-emails-in-the-auth0-dashboard">
  ### Habilita correos electrónicos no únicos en el Auth0 Dashboard
</div>

1. Ve a **Authentication &gt; Database** y crea una nueva conexión.
2. Ve a la sección **Choose one or more attributes as user identifiers** y activa **Email Address** en **On,** luego habilita el interruptor **Allow non-unique email addresses** que aparece.
3. Elige **username** o **phone number** y actívalo también en **On** para que funcione como identificador principal para el login y los flujos de restablecimiento de contraseña.
4. Después de confirmar que el correo electrónico no se usará como identificador, selecciona **Create** para guardar la conexión.

<div id="enable-non-unique-emails-via-the-management-api">
  ### Habilitar correos electrónicos no únicos mediante la Management API
</div>

Usa el endpoint `POST /api/v2/connections` de la Management API para crear una conexión de base de datos que admita correos electrónicos no únicos.

Al crear la conexión:

* Establece **unique: false** en el objeto `options.attributes.email` para permitir varias cuentas con la misma dirección de correo electrónico. Establece **identifier.active: false** para asegurarte de que el correo electrónico no pueda actuar como identificador principal cuando no sea único.
* Elige otro atributo como identificador principal y establece **identifier.active: true** en el atributo que hayas elegido.

Sin un identificador principal aparte del correo electrónico, la autenticación y los flujos de restablecimiento de contraseña no funcionan correctamente. Asegúrate de que al menos un atributo esté configurado como identificador activo.

<div id="example-request">
  #### Ejemplo de solicitud
</div>

A continuación se muestra un ejemplo del cuerpo de una solicitud para crear una conexión de base de datos que usa un nombre de usuario como identificador principal y admite correos electrónicos no únicos:

```json lines expandable
{
 "name": "new-non-unique",
 "strategy": "authO",
 "options": {
  "attributes": {
   "email": {
     "unique": false,
     "signup" : {
       "status": "required"
     },
     "identifier": {
      "active": false
     },
     "profile_required": true
   ｝，
   "username": {
    "signup": {
     "status": "required"
   },
   "identifier": {
    "active": true
   ｝，
   "profile_required": true
   }
  }
 }
}
```

<div id="shared-email-risk-disclaimer">
  ### Aviso sobre los riesgos del correo electrónico compartido
</div>

Aunque la funcionalidad **Non-Unique Emails** incluye salvaguardas, como impedir el uso del correo electrónico como identificador principal y exigir que los restablecimientos de contraseña se realicen con un nombre de usuario o un número de teléfono**,** sigue existiendo un riesgo inherente cuando varias cuentas de usuario comparten la misma dirección de correo electrónico. Por ejemplo:

* Todas las comunicaciones por correo electrónico (p. ej., enlaces de restablecimiento de contraseña, notificaciones) llegarán a la misma bandeja de entrada, independientemente de qué usuario haya iniciado la acción.
* Esto podría generar confusión entre los usuarios o provocar acceso no intencionado a enlaces enviados por correo electrónico si la bandeja de entrada es compartida.

Al habilitar esta funcionalidad, usted reconoce y acepta la responsabilidad de garantizar que:

* Los escenarios de correo electrónico compartido sean adecuados para su caso de uso.
* Los usuarios finales estén debidamente informados y capacitados.
* El diseño de su aplicación contemple posibles solapamientos en los flujos de trabajo basados en correo electrónico.

Esta contrapartida ofrece flexibilidad, pero requiere una implementación cuidadosa y una comunicación clara con sus usuarios.
