<ReleaseStageNotice feature="Claves de acceso nativas" stage="ea" contact="equipo de soporte de Auth0" />

Las claves de acceso son una alternativa resistente al phishing frente a las formas tradicionales de autenticación (como nombre de usuario y contraseña) y ofrecen una experiencia de usuario más sencilla y segura. Para consultar todos los detalles de implementación, revisa [Claves de acceso nativas para aplicaciones móviles](/docs/es/authenticate/database-connections/passkeys/native-passkeys-for-mobile-applications).

Las claves de acceso nativas usan una combinación de Auth0 y APIs nativas de iOS y Android para integrar directamente en tu aplicación móvil los flujos de desafío. Los endpoints que se enumeran a continuación son un subconjunto de la Auth0 Authentication API. Para obtener más información sobre cómo usar esta API, revisa la [Introducción a la Authentication API](https://auth0.com/docs/api/authentication#introduction).

Las claves de acceso tienen tres flujos relacionados:

* **registro**: Crea una nueva cuenta de usuario con una clave de acceso como método principal de autenticación.
* **inscripción**: Agrega una clave de acceso como método de autenticación a la cuenta de un usuario existente.
* **inicio de sesión**: Solicita a un usuario existente que se autentique con una clave de acceso asociada a su cuenta.

<div id="signup-flow">
  ## Flujo de registro
</div>

<div id="request-signup-challenge">
  ### Request Signup Challenge
</div>

`POST /passkey/register`

Inicia el flujo de registro con clave de acceso para un usuario nuevo.

En respuesta, Auth0 devuelve [PublicKeyCredentialCreationOptions](https://www.w3.org/TR/webauthn-3/#dictdef-publickeycredentialcreationoptions) y un ID de sesión. Consulta `timeout` en `authn_params_public_key` de la respuesta para ver el tiempo de espera de la sesión.

El flujo de registro con clave de acceso admite Organizations mediante el parámetro `organization`, siguiendo el comportamiento descrito en [Flujos de inicio de sesión para Organizations](/docs/es/manage-users/organizations/login-flows-for-organizations). Si tu aplicación está configurada para **Business Users**, debes proporcionar el parámetro `organization` y un valor válido de nombre o identificador de organización. Después de que el usuario registre una clave de acceso, Auth0 lo inscribe en la organización proporcionada.

#### Parámetros de solicitud

| Parámetro      | Descripción                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                       |
| -------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `client_id`    | **Obligatorio**. El `client_id` de tu aplicación.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| `realm`        | **Opcional**. El nombre de la conexión que se asociará a este usuario.<br /><br />Si no se especifica una conexión, se usa el directorio predeterminado de tu tenant.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                             |
| `user_profile` | **Obligatorio**. Un objeto que contiene la información de identificación del usuario. De forma predeterminada, incluye un `email` válido y un `name` opcional para mostrar.<br /><br />Si has habilitado [Flexible Identifiers](/docs/es/authenticate/database-connections/activate-and-configure-attributes-for-flexible-identifiers) para tu conexión de base de datos, puedes usar una combinación de `email`, `phone_number` o `username` como identificadores. Estas opciones pueden ser obligatorias u opcionales y deben coincidir con la configuración de Flexible Identifiers.<br /><br />Si el identificador proporcionado (como `email`) ya existe en el directorio, se debe pedir al usuario que complete el flujo de inicio de sesión en su lugar. |

<div id="remarks">
  #### Observaciones
</div>

* Una vez completada la solicitud del desafío, tu aplicación puede continuar con el proceso de registro del usuario mediante las API nativas de [Android](https://developer.android.com/identity/passkeys/create-passkeys) o [iOS](https://developer.apple.com/documentation/authenticationservices/supporting-passkeys#Register-a-new-account-on-a-service).
* Luego debes [autenticar al nuevo usuario](#authenticate-new-user) usando la información obtenida a través de las API nativas para completar el flujo.

<div id="authenticate-new-user">
  ### Autenticar a un usuario nuevo
</div>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Actualmente no se admite el registro nativo con clave de acceso cuando se requiere la verificación por OTP de SMS/Email en la misma conexión durante el registro.
</Callout>

`POST /oauth/token`

Usa el <Tooltip tip="endpoint de token: Endpoint en el servidor de autorización que se utiliza para solicitar tokens mediante programación." cta="Ver glosario" href="/docs/es/glossary?term=Token+endpoint">endpoint de token</Tooltip> para autenticar al usuario con las credenciales proporcionadas, crear su cuenta y devolver los tokens solicitados.

El parámetro `authn_response` se basa en la [especificación de la API de autenticación web](https://www.w3.org/TR/webauthn-3/). En el flujo nativo de clave de acceso, la información que se envía a este endpoint puede recuperarse mediante las API nativas de tu aplicación móvil:

* [Documentación de registro para Android](https://developer.android.com/identity/passkeys/create-passkeys)
* [Documentación de registro para iOS](https://developer.apple.com/documentation/authenticationservices/supporting-passkeys#Register-a-new-account-on-a-service)

<div id="enrollment-flow">
  ## Flujo de inscripción
</div>

La inscripción de una nueva clave de acceso para un usuario implica un proceso de dos pasos mediante la My Account API. Este flujo garantiza que la inscripción de la clave de acceso se inicie de forma segura y luego se verifique.

Antes de iniciar el flujo de inscripción, asegúrate de tener un <Tooltip tip="Token de acceso: Credencial de Authorization, en forma de una cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=access+token">token de acceso</Tooltip> con el scope `create:me:authentication_methods` para el endpoint `/me`.

<div id="initiate-passkey-enrollment">
  ### Iniciar la inscripción de la clave de acceso
</div>

`POST /me/v1/authentication-methods`

El primer paso es iniciar el proceso de inscripción. Para ello, se realiza una solicitud POST al endpoint `/me/v1/authentication-methods`.

<div id="request-parameters">
  #### Parámetros de la solicitud
</div>

| Parámetro    | Descripción                                                                    |
| ------------ | ------------------------------------------------------------------------------ |
| `type`       | **Obligatorio**. Incluya el valor `public-key`.                                |
| `connection` | **Opcional**. El nombre de la conexión en la que se creará la clave de acceso. |
| `identity`   | **Opcional**. La identidad del usuario. Se usa con cuentas vinculadas.         |

<div id="code-samples">
  #### Ejemplos de código
</div>

<div id="request">
  ##### Solicitud
</div>

```json lines
{
  "type": "passkey",
  "connection": "CONNECTION_NAME",
  "identity": "IDENTITY_USER_ID"
}
```

      "residentKey": "required",
      "userVerification": "preferred"
    },
    "user": {
      "id": "GENERATED_ID",
      "name": "USER_ENTERED_IDENTIFIER",
      "displayName": "USER_ENTERED_DISPLAY_NAME_OR_IDENTIFIER_IF_MISSING"

<div id="remarks">
  #### Observaciones
</div>

* La propiedad `auth_session` en el cuerpo de la respuesta es el identificador de la sesión de autenticación actual. Debe pasarse al endpoint `/verify`.
* Una vez completada la solicitud del desafío, tu aplicación puede continuar el proceso de inscripción del usuario mediante las API nativas de [Android](https://developer.android.com/identity/passkeys/create-passkeys) o [iOS](https://developer.apple.com/documentation/authenticationservices/supporting-passkeys#Register-a-new-account-on-a-service). Se le pedirá al usuario que cree una clave de acceso con su autenticador (como un lector de huellas dactilares, una llave de seguridad o el teléfono).

<div id="verify-passkey-enrollment">
  ### Verificar la inscripción de la clave de acceso
</div>

`POST /me/v1/authentication-methods/passkey|new/verify`

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  El ID de la ruta siempre es `passkey|new` para nuevas inscripciones.
</Callout>

Una vez que el usuario haya creado correctamente la clave de acceso con su autenticador, la aplicación cliente recibirá un `AuthenticatorAttestationResponse` de la API de WebAuthn. Esta respuesta debe enviarse al servicio de Auth0 para completar y verificar la inscripción.

* [Documentación de registro de Android](https://developer.android.com/identity/passkeys/create-passkeys)
* [Documentación de registro de iOS](https://developer.apple.com/documentation/authenticationservices/supporting-passkeys#Register-a-new-account-on-a-service)

<div id="login-flow">
  ## Flujo de inicio de sesión
</div>

<div id="request-login-challenge">
  ### Solicitar desafío de inicio de sesión
</div>

`POST /passkey/challenge`

Inicia el flujo de inicio de sesión con clave de acceso para un usuario existente que guardó una clave de acceso en su cuenta durante el registro inicial.

Como respuesta, Auth0 devuelve [PublicKeyCredentialRequestOptions](https://www.w3.org/TR/webauthn-3/#dictdef-publickeycredentialrequestoptions), un ID de sesión y un identificador de Relying Party, `rpId`. Consulta `timeout` en `authn_params_public_key` de la respuesta para ver el tiempo de espera de la sesión. El `rpId` devuelto en la respuesta es el identificador que el dispositivo nativo (iOS/Android) usa para buscar credenciales guardadas vinculadas a un dominio. Para que una clave de acceso creada en una aplicación web (por ejemplo, `example.com`) esté disponible en el flujo nativo, el `rpId` devuelto debe coincidir con el `rpId` de la web. Para obtener más información sobre cómo personalizar el RP ID para tu tenant, consulta [Configure Passkey Policy](/docs/es/authenticate/database-connections/passkeys/configure-passkey-policy#configure-relying-party-id-rp-id).

El flujo de inicio de sesión con clave de acceso es compatible con organizaciones mediante el parámetro `organization`, según el comportamiento descrito en [Login Flows for Organizations](/docs/es/manage-users/organizations/login-flows-for-organizations). Si tu aplicación está configurada para **Business Users**, debes proporcionar el parámetro `organization` y un nombre de organización o un valor de identificador válidos.

Todos los tokens emitidos están en el contexto de la organización proporcionada. Si habilitaste la [membresía automática](/docs/es/manage-users/organizations/login-flows-for-organizations#auto-membership) para tu organización, el usuario se agrega automáticamente a la organización después de autenticarse correctamente.

<div id="authenticate-existing-user">
  ### Autenticar a un usuario existente
</div>

`POST /oauth/token`

* [documentación de inicio de sesión en Android](https://developer.android.com/identity/passkeys/sign-in-with-passkeys)
* [documentación de inicio de sesión en iOS](https://developer.apple.com/documentation/authenticationservices/supporting-passkeys#Connect-to-a-service-with-an-existing-account)

Incluye los valores:<ul><li>`platform`</li><li>`cross-platform`</li></ul>                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                   |
| `authn_response.response`                | **Obligatorio**. Un objeto que contiene los siguientes elementos: <ul><li>`authenticatorData`: Contiene [datos del autenticador](https://www.w3.org/TR/webauthn-3/#authenticator-data) devueltos por el autenticador.</li><li>`clientDataJSON`: Contiene la serialización compatible con JSON de los datos del cliente; heredada de [AuthenticatorResponse](https://www.w3.org/TR/webauthn-3/#authenticatorresponse).</li><li>`signature`: Firma Base64URL devuelta por el autenticador.</li><li>`userHandle`: Identificador Base64URL de la cuenta de usuario, devuelto como `user.id` en el paso de registro.</li></ul> |
| `authn_response.clientExtensionResults`  | **Opcional**. Contiene los resultados del procesamiento de las extensiones de cliente solicitadas por la parte de confianza.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                              |
