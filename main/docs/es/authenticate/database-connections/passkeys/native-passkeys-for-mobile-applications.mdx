Las claves de acceso son una alternativa resistente al phishing a las formas tradicionales de autenticación (como nombre de usuario y contraseña) y ofrecen una experiencia de usuario más sencilla y segura. Se basan en las [especificaciones](https://fidoalliance.org/specs/fido-v2.1-ps-20210615/fido-client-to-authenticator-protocol-v2.1-ps-errata-20220621.html#intro) de FIDO® W3C Web Authentication (WebAuthn) y Client to Authenticator Protocol (CTAP).

Actualmente, Auth0 admite claves de acceso como método de autenticación para conexiones de base de datos y ofrece dos métodos de implementación:

* [Claves de acceso en Universal Login](/docs/es/authenticate/database-connections/passkeys) para aplicaciones web.
* Claves de acceso nativas para aplicaciones Android e iOS.
* [Embedded Login para aplicaciones web y nativas](/docs/es/authenticate/passwordless/implement-login/embedded-login)

<div id="how-it-works">
  ## Cómo funciona
</div>

Las claves de acceso nativas usan una combinación de la [Auth0 Authentication API](/docs/es/authenticate/database-connections/passkeys/native-passkeys-api) y las API nativas de iOS o Android para integrar directamente los flujos de verificación en su aplicación móvil. Esto le permite crear una experiencia integrada de registro e inicio de sesión para su aplicación sin depender de redirigir a los usuarios a sus navegadores para completar la autenticación.

El siguiente ejemplo muestra lo que puede experimentar un usuario nuevo durante el flujo de registro con clave de acceso:

1. Un usuario nuevo abre su aplicación móvil y accede a la pantalla de inicio de sesión. Como es un usuario nuevo, selecciona el botón de Registro.
2. En la siguiente pantalla, el usuario introduce su dirección de correo electrónico y selecciona Crear cuenta.
3. A continuación, se le pregunta al usuario si quiere crear una clave de acceso para su aplicación. Para continuar, el usuario selecciona Continuar.
4. Para generar una clave de acceso, el usuario debe autenticarse localmente en su device usando biometría u otro método de autenticación, como introducir un PIN.
5. Una vez completada la autenticación local, se guarda una nueva clave de acceso en el device del usuario y se sincroniza con su proveedor de claves de acceso, como iCloud o Google.
6. Después de guardar la clave de acceso, el usuario continúa con el proceso de registro para finalizar su cuenta.

Una vez completado este proceso, el usuario puede autenticarse con su clave de acceso guardada la próxima vez que inicie sesión en su aplicación.

<div id="before-you-begin">
  ## Antes de empezar
</div>

<div id="configure-a-custom-domain">
  ### Configura un dominio personalizado
</div>

Las claves de acceso nativas requieren el uso de un <Tooltip tip="Dominio personalizado: dominio de terceros con un nombre específico o de marca." cta="Ver glosario" href="/docs/es/glossary?term=custom+domain">dominio personalizado</Tooltip>. Antes de continuar, asegúrate de haber configurado un dominio personalizado para tu tenant. Para obtener más información, consulta [Dominios personalizados](/docs/es/customize/custom-domains).

<div id="configure-your-passkey-policy">
  ### Configura tu política de claves de acceso
</div>

Antes de poder implementar claves de acceso nativas para aplicaciones de Android o iOS, debes configurar una política de claves de acceso en tu tenant de Auth0. Para preparar tu tenant, sigue los pasos de [Configurar la política de claves de acceso](/docs/es/authenticate/database-connections/passkeys/configure-passkey-policy).

<div id="prepare-your-application">
  ### Prepara tu aplicación
</div>

Para preparar tu aplicación para las claves de acceso nativas, debes configurar Device Settings y agregar la concesión `Passkey`. Puedes completar esta configuración desde tu <Tooltip tip="Auth0 Dashboard: el producto principal de Auth0 para configurar tus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip> o la <Tooltip tip="Management API: un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip>.

<div id="associated-domains">
  #### Dominios asociados
</div>

Para que los usuarios finales puedan autenticarse con una sola clave de acceso en distintos tipos de aplicación o en aplicaciones con subdominios diferentes, establece el ID de Relying Party en el dominio raíz o en el dominio principal. Para obtener más información, consulta [Claves de acceso](/docs/es/authenticate/database-connections/passkeys).

<div id="configure-auth0-dashboard">
  ### Configurar Auth0 Dashboard
</div>

1. Vaya a [Applications &gt; Applications](https://manage.auth0.com/#/applications) y seleccione la aplicación que desea actualizar.
2. En la parte inferior de la pestaña Settings, seleccione **Advanced Settings**. Luego, elija la pestaña **Device Settings**.
3. Complete las secciones de **iOS** y **Android** según sea necesario para su aplicación. Luego, haga clic en **Save Changes**.
4. En la sección **Advanced Settings**, seleccione la pestaña **Grant Types**.
5. Habilite la concesión **Passkey** y, a continuación, seleccione **Save Changes**.

<div id="management-api">
  #### Management API
</div>

Llama al endpoint [Actualizar un cliente](https://auth0.com/docs/api/management/v2/clients/patch-clients-by-id) y:

* Actualiza `grant_types` para incluir `urn:okta:params:oauth:grant-type:webauthn`.
* Usa el objeto `mobile` para especificar la configuración de dispositivos iOS y Android según sea necesario.

<div id="implement-passkey-flows">
  ## Implementa flujos de claves de acceso
</div>

Puedes definir los siguientes flujos de claves de acceso para tu aplicación:

* [Flujo de registro](#signup-flow): Permite a los nuevos usuarios generar y guardar una clave de acceso durante el proceso de registro.
* [Flujo de inicio de sesión](#login-flow): Permite que un usuario existente que ya haya registrado una clave de acceso se autentique con su clave de acceso guardada durante el proceso de inicio de sesión.

<div id="signup-flow">
  ### Flujo de registro
</div>

Un usuario inicia el flujo de registro con clave de acceso la primera vez que intenta iniciar sesión en tu aplicación.

Si el usuario proporciona un identificador que ya existe, se recomienda indicarle que complete el flujo de inicio de sesión en su lugar. De lo contrario, la Action fallará.

<div id="flow-steps">
  #### Pasos del flujo
</div>

1. Un usuario visita su aplicación y elige crear una cuenta nueva. El usuario proporciona un identificador solicitado por su aplicación, como su dirección de correo electrónico.
2. Su aplicación inicia el desafío de registro llamando al endpoint [Solicitar desafío de registro](/docs/es/authenticate/database-connections/passkeys/native-passkeys-api#request-signup-challenge) del Auth0 Authentication API:

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  * Si no especifica un `realm`, se usa el directorio predeterminado de su tenant.
  * De forma predeterminada, `email` es el identificador obligatorio. Si ha habilitado [Flexible Identifiers](/docs/es/authenticate/database-connections/activate-and-configure-attributes-for-flexible-identifiers) para su conexión de base de datos, puede usar en su lugar una combinación de `email`, `phone_number` o `username`. Estas opciones pueden ser obligatorias u opcionales y deben coincidir con su configuración de Flexible Identifiers.
</Callout>

```text lines
   POST /passkey/register
   Content-Type: application/json

   {
     "client_id": "{YOUR_CLIENT_ID}",
     "realm": "{OPTIONAL_CONNECTION}",
     "user_profile": {
   	  "email": "{VALID_EMAIL_ADDRESS}",
   	  "name": "{OPTIONAL_USER_DISPLAY_NAME}",
     }
   }
```

3. En respuesta, Auth0 devuelve [`PublicKeyCredentialCreationOptions`](https://www.w3.org/TR/webauthn-3/#dictdef-publickeycredentialcreationoptions) junto con un ID de `auth_session` y `rpId` como `rp.id`:

   ```text lines expandable
   HTTP/1.1 200 OK
   Content-Type: application/json

   {
     "authn_params_public_key": {
       "challenge": "{GENERATED_CHALLENGE_FOR_THIS_SESSION}",
       "timeout": {MILLISECONDS},
       "rp": {
         "id": "{YOUR_CUSTOM_DOMAIN}",
         "name": "{YOUR_CUSTOM_DOMAIN}"
       },
       "pubKeyCredParams": [
         { type: 'public-key', alg: -8 },
         { type: 'public-key', alg: -7 },
         { type: 'public-key', alg: -257 }
       ],
       "authenticatorSelection": {
         "residentKey": "required",
         "userVerification": "preferred"
       },
       "user": {
         "id": "{GENERATED_ID}",
         "name": "{USER-ENTERED_IDENTIFIER}",
         "displayName": "{USER-ENTERED_DISPLAY_NAME_OR_IDENTIFIER_IF_MISSING}"
       }
     },
     "auth_session": "{SESSION_ID}"
    }
   ```

   <Callout icon="file-lines" color="#0EA5E9" iconType="regular">
     El `rpId` devuelto en la respuesta coincide con el dominio utilizado en el flujo web para que las claves de acceso funcionen en dominios asociados.
   </Callout>

4. Luego, la aplicación completa el proceso de registro del usuario mediante las API nativas correspondientes:

   * Revise la [documentación de registro para Android](https://developer.android.com/identity/passkeys/create-passkeys).
   * Revise la [documentación de registro para iOS](https://developer.apple.com/documentation/authenticationservices/supporting-passkeys#Register-a-new-account-on-a-service).

5. Luego, la aplicación usa la información de la credencial obtenida durante el proceso de registro, incluidos los detalles de `authn_response`, para llamar al endpoint [Token](/docs/es/authenticate/database-connections/passkeys/native-passkeys-api#authenticate-new-user):

   ```text lines
   POST /oauth/token
   Content-Type: application/json

   {
     "grant_type": "urn:okta:params:oauth:grant-type:webauthn",
     "client_id": "{YOUR_CLIENT_ID}",
     "realm": "{OPTIONAL_CONNECTION}",
     "scope": "{OPTIONAL_REQUESTED_SCOPE}",
     "audience": "{OPTIONAL_REQUESTED_AUDIENCE}"
     "auth_session": "{SESSION_ID_FROM_THE_FIRST_REQUEST}",
     "authn_response": {
       "id": "{BASE64URL_ID}",
       "rawId": "{BASE64URL_RAWID}",
       "type": "public-key",
       "authenticatorAttachment": "platform|cross-platform",
       "response": {
         "clientDataJSON": "{BASE64URL_CLIENT_DATA_JSON}",
         "attestationObject": "{BASE64URL_ATTESTATION_OBJECT}",
         {OTHER_PROPERTIES}
       },
     },
   }
   ```

6. Auth0 crea una nueva cuenta de usuario y devuelve los tokens solicitados para completar el flujo:

   ```text lines
   HTTP/1.1 200 OK
   Content-Type: application/json

   {
     "access_token": "{BASE64_TOKEN}",
     "refresh_token": "{BASE64_TOKEN}",
     "id_token": "{BASE64_TOKEN}",
     "token_type": "Bearer",
     "expires_in": {SECONDS}
   }
   ```

<div id="login-flow">
  ### Flujo de inicio de sesión
</div>

Un usuario que ya tiene una cuenta inicia el flujo de inicio de sesión con clave de acceso cuando intenta iniciar sesión en tu aplicación. Este flujo solo se aplica a usuarios existentes que guardaron claves de acceso en sus cuentas durante el registro inicial.

<div id="flow-steps">
  #### Pasos del flujo
</div>

1. Un usuario abre tu aplicación y comienza el proceso de inicio de sesión. A continuación, tu aplicación inicia el desafío de inicio de sesión llamando al endpoint [Solicitar desafío de inicio de sesión](/docs/es/authenticate/database-connections/passkeys/native-passkeys-api#request-login-challenge) de la Authentication API:

   <Callout icon="file-lines" color="#0EA5E9" iconType="regular">
     Si no especificas un `realm`, se usará el directorio predeterminado de tu tenant.
   </Callout>

   ```text lines
   POST /passkey/challenge
   Content-Type: application/json

   {
     "client_id": "{YOUR_CLIENT_ID}",
     "realm": "{OPTIONAL_CONNECTION}"
   }
   ```

2. En respuesta, Auth0 devuelve [`PublicKeyCredentialRequestOptions`](https://www.w3.org/TR/webauthn-3/#dictdef-publickeycredentialrequestoptions) junto con un `auth_session` para continuar el flujo en tu aplicación:

   ```text lines
   HTTP/1.1 200 OK
   Content-Type: application/json

   {
     "authn_params_public_key": {
       "challenge": "{GENERATED_CHALLENGE_FOR_THIS_SESSION}",
       "timeout": {AUTH_TIMEOUT_IN_MILLISECONDS},
       "rpId": "{CUSTOM_DOMAIN}",
       "userVerification": "preferred"
     },
     "auth_session": "{SESSION_ID}"
   }
   ```

3. A continuación, tu aplicación completa el proceso de inicio de sesión mediante las API nativas correspondientes:

   * Consulta la [documentación de inicio de sesión de Android](https://developer.android.com/identity/passkeys/sign-in-with-passkeys)
   * Consulta la [documentación de inicio de sesión de iOS](https://developer.apple.com/documentation/authenticationservices/supporting-passkeys#Connect-to-a-service-with-an-existing-account).

4. A continuación, tu aplicación usa la información de credenciales obtenida durante el proceso de inicio de sesión, incluidos los detalles de `authn_response`, para llamar al endpoint [Token](/docs/es/authenticate/database-connections/passkeys/native-passkeys-api#authenticate-existing-user):

   ```text lines
   POST /oauth/token
   Content-Type: application/json

   {
     "grant_type": "urn:okta:params:oauth:grant-type:webauthn",
     "client_id": "{YOUR_CLIENT_ID}",
     "realm": "{OPTIONAL_CONNECTION}",
     "scope": "{OPTIONAL_REQUESTED_SCOPE}",
     "audience": "{OPTIONAL_REQUESTED_AUDIENCE}"
     "auth_session": "{SESSION_ID_FROM_THE_FIRST_REQUEST}",
     "authn_response": {
       "id": "{BASE64URL_ID}",
       "rawId": "{BASE64URL_RAWID}",
       "type": "public-key",
       "authenticatorAttachment": "platform|cross-platform",
       "response": {
         "authenticatorData": "{BASE64URL_AUTHENTICATORDATA}",
         "clientDataJSON": "{BASE64URL_CLIENTDATAJSON}",
         "signature": "{BASE64URL_SIGNATURE}",
         "userHandle": "{BASE64URL_USERHANDLE}"
       },
       "clientExtensionResults": {OPTIONAL_OBJECT}
      }
   }
   ```

5. Auth0 autentica las credenciales y devuelve los tokens solicitados para completar el flujo:

   ```text lines
   HTTP/1.1 200 OK
   Content-Type: application/json

   {
     "access_token": "{BASE64_TOKEN}",
     "refresh_token": "{BASE64_TOKEN}",
     "id_token": "{BASE64_TOKEN}",
     "token_type": "Bearer",
     "expires_in": {SECONDS}
   }
   ```
