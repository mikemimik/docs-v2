<div id="prerequisites">
  ## Requisitos previos
</div>

Antes de poder habilitar las claves de acceso para una conexión de base de datos, debes cumplir los siguientes requisitos previos.

En tu tenant, debes:

* [Habilitar Universal Login](/docs/es/authenticate/login/auth0-universal-login/universal-login-vs-classic-login/universal-experience).

* [Habilitar la autenticación Identifier First](/docs/es/authenticate/login/auth0-universal-login/identifier-first).

* Deshabilitar las páginas de inicio de sesión personalizadas. En el Auth0 Dashboard, ve a [Branding &gt; Universal Login &gt; Manage Classic Login &gt; Login](https://manage.auth0.com/#/login_page) y desactiva el interruptor **Custom Login Page**.

En tu conexión de base de datos, debes:

* Deshabilitar [el requisito de nombre de usuario](/docs/es/authenticate/database-connections/require-username) para tu conexión de base de datos o, alternativamente, [habilitar Flexible Identifiers](/docs/es/authenticate/database-connections/flexible-identifiers-and-attributes).

* Deshabilitar [el uso de tu propia base de datos](/docs/es/authenticate/database-connections/custom-db/create-db-connection) o, alternativamente, habilitar la [importación de usuarios](/docs/es/authenticate/database-connections/custom-db/templates) (**Import Users to Auth0**).

  Para usar tu propia base de datos sin tener habilitada la importación de usuarios (**Import Users to Auth0**), sigue los [requisitos previos adicionales que se indican a continuación para actualizar tus scripts de acción de base de datos](#early-access-use-your-own-database-without-user-import-enabled).

Las conexiones de base de datos con claves de acceso habilitadas también deben seguir teniendo contraseñas habilitadas. Esto garantiza que los usuarios puedan seguir accediendo a sus cuentas desde navegadores y dispositivos más antiguos que quizá todavía no sean compatibles con las claves de acceso.

### <Badge color="yellow">Acceso anticipado</Badge> Usa tu propia base de datos sin habilitar la importación de usuarios

La autenticación con clave de acceso ahora admite el uso de tu propia base de datos sin habilitar la importación de usuarios. Esto te permite ofrecer autenticación con clave de acceso en conexiones de base de datos personalizadas mientras sigues autenticando a los usuarios con tu repositorio de usuarios externo.

Como requisito previo, debes actualizar los scripts de acción de base de datos [Get User y Create](/docs/es/authenticate/database-connections/custom-db/templates) para admitir la gestión de usuarios tanto por identificador como por `user_id`.

<Accordion title="Cómo actualizar los scripts de acción de base de datos para habilitar claves de acceso en tu propia base de datos sin habilitar la importación de usuarios">
  <ReleaseStageNotice feature="passkeys for using your own database wihtout user import enabled" stage="ea" terms="true" />

  1. Primero, confirma que la importación de usuarios esté deshabilitada para tu conexión de base de datos.

  2. [Habilita la compatibilidad con el objeto `context`](/docs/es/authenticate/database-connections/custom-db/create-db-connection#enable-context-object). Esto hace que el parámetro `context` esté disponible en tus scripts de acción de base de datos, lo cual es necesario para admitir búsquedas tanto por identificador como por `user_id`.

  3. El script Get User debe admitir tanto la búsqueda por identificador (`context.identifierType` no está definido) como la búsqueda por `user_id` (`context.identifierType` está definido como `user_id`).

     Actualiza tu [script de acción de base de datos Get User](/docs/es/authenticate/database-connections/custom-db/templates/get-user) de `getByEmail` a la función `getUser` proporcionada:

     ```js
     function getByEmail(email, callback) { // [!code --] 
     function getUser(identifierValue, context, callback) { // [!code ++] 
         const axios = require('axios');

         // Obtén el identificador de usuario desde context // [!code ++]
         const identifierType = context.identifierType || 'email'; // [!code ++]

         // Sustituye por tu endpoint externo de recuperación de usuarios 
         const url = configuration.baseAPIUrl
                     + `/get_user/${email}?type=email`; // [!code --]
                     + `/get_user/${identifierValue}?type=${identifierType}`; // [!code ++]

         axios.get(url)
             .then(response => {
                 // Devuelve el objeto de usuario si se encuentra
                 const user = response.data; 
                 return callback(null, {
                     user_id: user.user_id,
                     email: user.email
                     // Opcionalmente, incluye otros atributos del usuario 
                 });
             })
             .catch(error => {
                 // Gestiona la respuesta de error 
                 return callback(null);
             });
     }
     ```

  4. El script Create debe devolver un perfil de usuario válido. Cuando los atributos están habilitados, el perfil de usuario devuelto debe seguir la misma validación que los scripts Login y Get User. Cuando los atributos no están habilitados, el perfil de usuario devuelto debe incluir el valor `user_id`.

     Actualiza tu [script de acción de base de datos Create](/docs/es/authenticate/database-connections/custom-db/templates/create) para devolver un perfil de usuario válido:

     ```js
     function create(user, callback) {
         const axios = require('axios');

         // Sustituye por tu endpoint externo de creación de usuarios
         const url = 'https://example.com/api/create'; 

         const payload = {
             email: user.email
             // Agrega otras propiedades del usuario según sea necesario
         };

         axios.post(url, payload)
             .then(response => {
                 // Devuelve el objeto de usuario si se crea
                 const createdUser = response.data; 

                 return callback(null, {
                     user_id: createdUser.user_id,
                     email:createdUser.email
                     // Opcionalmente, incluye otros atributos del usuario
                 });
             })
             .catch(error => {
                 // Gestiona la respuesta de error 
                 return callback(error);
             }); 
     }
     ```
</Accordion>

<div id="configure-passkeys">
  ## Configurar claves de acceso
</div>

Una vez que completes los requisitos previos, puedes usar el Auth0 Dashboard para habilitar y configurar las claves de acceso.

<Steps titleSize="h3">
  <Step title="Abrir la configuración de las claves de acceso">
    Ve a [Authentication &gt; Database](https://manage.auth0.com/#/connections/database) y selecciona el nombre de la conexión de base de datos que quieras editar.

    Selecciona la pestaña **Métodos de autenticación**. Luego, en la sección **Passkey**, selecciona **Configurar** para abrir el panel **Passkey**.

    Si aún no está marcada, marca la casilla **Habilitar claves de acceso**.
  </Step>

  <Step title="Elegir la interfaz de usuario de autenticación con clave de acceso">
    La interfaz de usuario de autenticación con clave de acceso determina cómo los usuarios pueden iniciar la autenticación con clave de acceso durante el inicio de sesión y el registro.

    En la sección **Interfaz de usuario de autenticación con clave de acceso**, elige una de las tres opciones:

    | Interfaz de usuario de autenticación con clave de acceso | Descripción                                                                                                                                                                                                                                                         |
    | -------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
    | Botón de clave de acceso y autocompletado                | Los usuarios pueden autenticarse con claves de acceso mediante el autocompletado o el botón de clave de acceso.                                                                                                                                                     |
    | Autocompletado                                           | Los usuarios deben iniciar sesión con la función de autocompletado de su navegador para usar claves de acceso. El autocompletado permite a los usuarios seleccionar una cuenta guardada en un menú desplegable en lugar de introducir manualmente sus credenciales. |
    | Botón de clave de acceso                                 | Los usuarios deben seleccionar el botón **Continuar con una clave de acceso** en el prompt de inicio de sesión.                                                                                                                                                     |

    Si el autocompletado no está disponible en el navegador del usuario, este puede iniciar sesión con el botón de clave de acceso (si está habilitado) o con credenciales con contraseña.
  </Step>

  <Step title="Habilitar el registro progresivo (opcional)">
    El registro progresivo solicita a los usuarios que creen una clave de acceso (si aún no lo han hecho) después de iniciar sesión con un identificador y una contraseña. Este paso no es obligatorio, y los usuarios pueden optar por posponer esta acción cada 30 días.

    El registro progresivo puede ser útil al migrar usuarios a claves de acceso, ya que facilita la transición entre métodos de autenticación.

    <Callout icon="file-lines" color="#0EA5E9" iconType="regular">
      Al crear una cuenta mediante un [correo electrónico de invitación a una organización](/docs/es/manage-users/organizations/configure-organizations/invite-members), los usuarios no pueden elegir la autenticación con clave de acceso. Puedes habilitar el registro progresivo para que estos usuarios puedan crear claves de acceso después de iniciar sesión con una contraseña.
    </Callout>

    La casilla **Registro progresivo** está marcada de forma predeterminada, pero puedes desmarcarla para deshabilitarla.
  </Step>

  <Step title="Habilitar el registro local (opcional)">
    Cuando un usuario de clave de acceso inicia sesión en un dispositivo nuevo usando una clave de acceso multidispositivo, el registro local le solicita que cree una clave de acceso local en el nuevo dispositivo. Esto no es obligatorio, por lo que puede optar por omitir la creación de una clave de acceso local.

    La casilla **Registro local** está marcada de forma predeterminada, pero puedes desmarcarla para deshabilitarla.
  </Step>

  <Step title="Guardar la configuración">
    Selecciona **Guardar** para guardar los cambios de configuración.

    Si el botón Guardar está deshabilitado, usa la sección **Requisitos previos de autenticación con clave de acceso** en la parte superior del panel **Passkey** para confirmar que tu tenant y tu conexión de base de datos estén configurados correctamente.
  </Step>
</Steps>

<div id="next-steps">
  ## Próximos pasos
</div>

Para garantizar la mejor experiencia de los usuarios finales al usar claves de acceso, quizá le convenga tener en cuenta los siguientes pasos adicionales.

<div id="configure-relying-party-id-rp-id">
  ### Configurar el ID de la parte de confianza (RP ID)
</div>

Para permitir que los usuarios usen una sola clave de acceso para autenticarse en todos los subdominios (por ejemplo, tanto en una aplicación nativa alojada en `app.example.com` como en una aplicación web alojada en `login.example.com`), puedes establecer el RP ID en el dominio principal o dominio raíz.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Cuando personalizas el RP ID en el Auth0 Dashboard o la Management API, todas las claves de acceso de otros dominios dejarán de funcionar y tus usuarios finales deberán volver a registrarlas.
</Callout>

El dominio de tu aplicación debe agregarse a la lista de **Allowed Origins (CORS)** en la [configuración de la aplicación](https://manage.auth0.com/#/applications) para que el RP ID funcione correctamente.

Para establecer el RP ID en Auth0 Dashboard:

1. Ve a **[Auth0 Dashboard &gt; Configuración del tenant](https://manage.auth0.com/#/tenant/relying_party_ids)**.
2. Selecciona la pestaña **Relying Party IDs**.
3. Elige **Edit Relying Party ID**.
4. Introduce el dominio.
5. Si este RP ID es el dominio raíz, marca la casilla para establecerlo como predeterminado para tu tenant.
6. Selecciona **Save**.
7. A continuación, ve a la aplicación para la que quieres establecer un ID de parte de confianza.
8. Selecciona la configuración de la aplicación.
9. En la sección **Application URIs**, agrega el dominio de tu aplicación en Allowed Origins (CORS).
10. Selecciona **Save**.

<div id="configure-a-custom-domain">
  ### Configurar un dominio personalizado
</div>

Cuando un usuario registra una clave de acceso, esta se asocia con el dominio de la <Tooltip tip="Entidad (como un servicio o una aplicación) que depende de un proveedor de identidad de terceros para autenticar a un usuario." cta="Ver glosario" href="/docs/es/glossary?term=relying+party">parte de confianza</Tooltip>. Si el nombre de dominio cambia, cualquier clave de acceso asociada al dominio anterior dejará de ser válida.

Considera configurar un [dominio personalizado](/docs/es/customize/custom-domains) para tu tenant antes de habilitar las claves de acceso, a fin de evitar interrupciones para los usuarios finales.

Si tienes [múltiples dominios personalizados](/docs/es/customize/custom-domains/multiple-custom-domains) habilitados en tu tenant, Auth0 mantiene una relación uno a uno entre un dominio y la clave de acceso de ese dominio.

Los usuarios solo pueden registrar una clave de acceso para uno de los múltiples dominios personalizados del tenant (el primero con el que la registren). Para el inicio de sesión passwordless, el dominio personalizado seleccionado debe reflejarse en el Magic Link del flujo de inicio de sesión passwordless.

<div id="bypass-multi-factor-authentication">
  ### Omitir la autenticación multifactor
</div>

Si tienes habilitada la [autenticación multifactor](/docs/es/secure/multi-factor-authentication), el comportamiento predeterminado es exigir que se complete un desafío MFA, sin importar si el método de autenticación fue una contraseña o una clave de acceso.

Dado el alto nivel de seguridad que proporcionan las claves de acceso, puedes [omitir el MFA en la autenticación con clave de acceso mediante una Action de Post Login](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger#reduce-friction-with-passkeys).
