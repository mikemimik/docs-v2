<ReleaseStageNotice feature="Autenticación Passwordless en Database Connections" stage="ea" contact="support" terms="true" />

Auth0 te permite configurar la autenticación passwordless por correo electrónico y teléfono directamente en una [conexión de base de datos](/docs/es/authenticate/database-connections). En lugar de crear una conexión independiente para la autenticación por contraseña de un solo uso (OTP), puedes ofrecer el inicio de sesión passwordless directamente desde tu conexión de base de datos. Esto reduce la complejidad de la implementación y simplifica la experiencia de inicio de sesión para los usuarios finales.

<Callout icon="triangle-exclamation" color="#F59E0B" iconType="regular">
  Passwordless para las conexiones de base de datos no es compatible con Classic Login.
</Callout>

<div id="how-it-works">
  ## Cómo funciona
</div>

Auth0 usa un enfoque de [Identifier-First Authentication](/docs/es/authenticate/login/auth0-universal-login/identifier-first) en el que los usuarios se verifican mediante autenticación OTP:

1. Un usuario final introduce un identificador para iniciar sesión con [Universal Login de Auth0](/docs/es/authenticate/login/auth0-universal-login).
2. Auth0 busca los métodos de autenticación de cada identificador configurado en tu conexión de base de datos.
3. Al usuario se le muestra la opción más adecuada según `default_method` y los métodos de autenticación disponibles (por ejemplo, recibir un OTP por correo electrónico o por teléfono, o introducir una contraseña).
4. Si OTP es el método de autenticación seleccionado, Auth0 envía un código al correo electrónico o al teléfono del usuario.
5. El usuario introduce el código y se autentica. Si has configurado claves de acceso, se le pedirá al usuario que complete la inscripción progresiva de clave de acceso.

<Card title="Antes de empezar">
  * Configura tu tenant para usar [Identifier-First Authentication Profile](https://auth0.com/docs/authenticate/login/auth0-universal-login/identifier-first).
  * Si planeas usar OTP por teléfono (SMS o voz), debes habilitar la configuración [Unified Phone Experience](/docs/es/customize/phone-messages/unified-phone/configure-unified-phone) en [**Branding &gt; Phone Provider**](https://manage.auth0.com/#/phone/templates/phone/provider).
  * [Atributos](/docs/es/authenticate/database-connections/activate-and-configure-attributes-for-flexible-identifiers) debe estar habilitado en la conexión de base de datos.
  * Si planeas usar conexiones passwordless existentes (heredadas) junto con la autenticación passwordless en conexiones de base de datos, y no quieres configurar tus conexiones passwordless existentes para usar Unified Phone Provider:
    1. Ve a [**Auth0 Dashboard &gt; Authentication &gt; Passwordless**](https://manage.auth0.com/#/connections/passwordless).
    2. Selecciona **Configurar** en SMS.
    3. Asegúrate de que la configuración **Usar proveedor de mensajería a nivel de tenant** esté deshabilitada.
</Card>

<div id="configure-attributes-identifiers">
  ## Configurar atributos (identificadores)
</div>

Al configurar la autenticación passwordless por correo electrónico y teléfono en conexiones de base de datos, primero debes determinar qué atributo(s) quieres que los usuarios finales proporcionen durante el registro y el inicio de sesión.

Consulta el tipo de identificador y el método de autenticación correspondiente:

| Identificador     | Método(s) de autenticación                  |
| ----------------- | ------------------------------------------- |
| Email             | Password, Email OTP, Clave de acceso        |
| Teléfono          | Password, OTP por teléfono, Clave de acceso |
| Nombre de usuario | Password                                    |

Para los identificadores de correo electrónico y teléfono, si Password **no** está habilitado, debes tener habilitada la verificación por OTP para el registro. Puedes configurar los atributos de correo electrónico y teléfono como opcionales en el registro, para que los usuarios puedan registrarse solo con un correo electrónico o solo con un número de teléfono.

Cuando un usuario se autentica con Email OTP, `email_verified` se establece automáticamente en `true` en su perfil. Cuando un usuario se autentica con OTP por teléfono, `phone_verified` se establece automáticamente en `true`.

<Warning>
  Si quieres que los usuarios finales se autentiquen con claves de acceso, **debes** tener configurado un método de autenticación alternativo, como correo electrónico o teléfono. La autenticación únicamente con clave de acceso no es compatible.
</Warning>

Para obtener más información, consulta [Activar y configurar atributos para identificadores flexibles](/docs/es/authenticate/database-connections/activate-and-configure-attributes-for-flexible-identifiers).

<div id="create-a-new-database-connection">
  ## Crear una nueva conexión de base de datos
</div>

Si no tiene una conexión de base de datos existente, cree una con el Auth0 Dashboard o la Management API.

<Tabs>
  <Tab title="Auth0 Dashboard" icon="browser">
    1. Vaya a [**Auth0 Dashboard &gt; Authentication &gt; Database**](https://manage.auth0.com/#/connections/database). Seleccione **Create DB Connection** para crear la conexión.
    2. Introduzca un nombre único para la conexión.
    3. Elija uno o más atributos para que los usuarios finales inicien sesión o se registren.
    4. Seleccione los métodos de autenticación. Podrá configurarlos con más detalle una vez creada la conexión.
    5. Active **Disable Sign Ups** si no quiere que los usuarios se registren mediante endpoints públicos.
    6. Active **Promote Connection to Domain Level** si quiere usar esta conexión con aplicaciones de terceros.
    7. Seleccione **Create**.

    Para una conexión realmente passwordless, complete estos pasos adicionales:

    1. En la nueva conexión, seleccione la pestaña **Attributes**.
    2. Para desactivar Username como identificador, seleccione **Configure** y desactive **Use Username as Identifier**.
    3. Para configurar los identificadores Email y Phone, seleccione **Configure**.

    * Para los atributos de Email, seleccione **One-Time Password (OTP)** en Verification Methods y active **Verify email on sign up** para una configuración realmente passwordless. Esto garantiza que `email_verified` se establezca automáticamente para que siempre se solicite un OTP a los usuarios al iniciar sesión y registrarse.

    4. Seleccione **Save**.

    5. Configure los ajustes de los métodos de autenticación correspondientes a los identificadores seleccionados. No puede desactivar la contraseña a menos que tenga configurados `phone_otp` y/o `email_otp`.

    6. En la configuración de Password, seleccione **Policy** y elija **Block** para:

       * Password on Login
       * Password on Signup
       * cambiar la contraseña de autoservicio (se actualiza automáticamente)

    7. Active **Support users without a password**.
       <Warning>
         Es posible que reciba un error si no activa **Support users without a password**.
       </Warning>

    8. Seleccione **Save**. En el prompt, seleccione **Continue** para confirmar que entiende que los usuarios existentes pueden verse afectados.

    9. Vaya a la pestaña **Applications** y habilite la conexión para su aplicación o API.
  </Tab>

  <Tab title="Management API" icon="code">
    <Callout icon="file-lines" color="#0EA5E9" iconType="regular">
      Para usar la Management API, necesita un [token de acceso de la Management API](/docs/es/secure/tokens/access-tokens/management-api-access-tokens/get-management-api-access-tokens-for-production).
    </Callout>

    Llame al endpoint [Create a Connection](https://auth0.com/docs/api/management/v2/connections/post-connections). Use la opción `default_method` para establecer el primer método de autenticación de cada identificador.

    ```bash expandable lines
    curl --location 'https://YOUR_AUTH0_TENANT/api/v2/connections' \
    --header 'Authorization: Bearer YOUR_MANAGEMENT_API_TOKEN' \
    --header 'Accept: application/json, text/plain, */*' \
    --header 'Content-Type: application/json' \
    --data '{
      "name": "YOUR_CONNECTION_NAME",
      "strategy": "auth0",
      "is_domain_connection": false,
      "options": {
        "disable_signup": false,
        "attributes": {
          "email": {
            "identifier": {
              "active": true,
              "default_method": "email_otp"
            },
            "profile_required": true,
            "signup": {
              "status": "required",
              "verification": {
                "active": true
              }
            },
            "verification_method": "otp",
            "unique": true
          },
          "phone_number": {
            "identifier": {
              "active": true,
              "default_method": "password"
            },
            "profile_required": true,
            "signup": {
              "status": "required",
              "verification": {
                "active": true
              }
            }
          },
          "username": {
            "identifier": {
              "active": true
            },
            "profile_required": true,
            "signup": {
              "status": "required"
            },
            "validation": {
              "min_length": 1,
              "max_length": 15,
              "allowed_types": {
                "email": false,
                "phone_number": false
              }
            }
          }
        },
        "authentication_methods": {
          "password": {
            "enabled": true
          },
          "passkey": {
            "enabled": false
          },
          "email_otp": {
            "enabled": true
          },
          "phone_otp": {
            "enabled": true
          }
        }
      }
    }'
    ```
  </Tab>
</Tabs>

<div id="update-existing-connections">
  ## Actualizar conexiones existentes
</div>

Si ya tienes conexiones de base de datos, actualiza la configuración de passwordless en el Auth0 Dashboard o en la Management API.

<Tabs>
  <Tab title="Auth0 Dashboard" icon="browser">
    1. **Abre la conexión**: Ve a [**Auth0 Dashboard &gt; Authentication &gt; Database**](https://manage.auth0.com/#/connections/database) y selecciona la conexión que quieres actualizar.
    2. **Activa Attributes**: En la pestaña **Attributes**, selecciona **Activate** para habilitar la nueva configuración de atributos.
    3. **Añade los atributos de Email y Phone**: Selecciona **+ Add Attributes** y añade Email y Phone Number si todavía no están presentes.

       <Callout icon="file-lines" color="#0EA5E9" iconType="regular">
         En una conexión passwordless (OTP) propiamente dicha, el identificador username no es compatible.
       </Callout>
    4. **Habilita los métodos de autenticación OTP**: En la pestaña **Authentication Methods**:
       * Configura Phone para **Allow** Phone OTP y guarda los cambios.
       * Configura Email para **Allow** Email OTP y guarda los cambios.
    5. **Bloquea la autenticación con contraseña**: En la configuración de Password, elige **Policy** y selecciona **Block** para lo siguiente:

       * Password on Login
       * Password on Signup
       * Self-service change password

       Activa **Support users without a password**.

       <Warning>
         Es posible que se produzca un error si no activas **Support users without a password**.
       </Warning>
    6. **Guardar**: Selecciona **Save**.
  </Tab>

  <Tab title="Management API" icon="code">
    <Callout icon="file-lines" color="#0EA5E9" iconType="regular">
      Para usar la Management API, necesitas un [token de acceso de la Management API](/docs/es/secure/tokens/access-tokens/get-management-api-access-tokens-for-production).
    </Callout>

    Llama al endpoint [actualizar una conexión](https://auth0.com/docs/api/management/v2/connections/patch-connections-by-id). Añade la opción `default_method` para establecer el primer método de autenticación. Para los atributos de Email, establece `password` o `email_otp`. Para los atributos de Phone, establece `password` o `phone_otp`.

    ```bash
    curl -L -X PATCH "https://YOUR_AUTH0_TENANT/api/v2/connections/<connection_id>" \
    -H "Content-Type: application/json" \
    -H "Authorization: Bearer YOUR_MANAGEMENT_API_TOKEN" \
    -d '{
      "options": {
        "passwordPolicy": "none",
        "authentication_methods": {
          "password": { "enabled": true },
          "passkey": { "enabled": true },
          "email_otp": { "enabled": true },
          "phone_otp": { "enabled": true }
        },
        "attributes": {
          "email": {
            "signup": {
              "status": "required",
              "verification": { "active": true }
            },
            "unique": true,
            "identifier": {
              "active": true,
              "default_method": "email_otp"
            },
            "profile_required": true,
            "verification_method": "otp"
          },
          "phone_number": {
            "signup": {
              "status": "required",
              "verification": { "active": true }
            },
            "identifier": {
              "active": true,
              "default_method": "phone_otp"
            },
            "profile_required": true
          }
        }
      }
    }'
    ```
  </Tab>
</Tabs>

<div id="use-auth0-actions">
  ## Usa Auth0 Actions
</div>

Para obtener más información sobre los factores de autenticación passwordless en conexiones de base de datos, configura atributos con Auth0 Actions.

<div id="post-login-trigger">
  ### trigger Post Login
</div>

El trigger `post-login` se activa después de que un usuario se autentica, pero antes de que el servidor de autorización devuelva un token. El objeto `event.authentication` en [Auth0 Dashboard &gt; Actions &gt; Triggers &gt; Post Login](https://manage.auth0.com/#/actions/flows) expone los siguientes métodos:

| Método                     | Parámetro | Descripción                                                                       |
| -------------------------- | --------- | --------------------------------------------------------------------------------- |
| Email OTP                  | `email`   | Email OTP utilizado para autenticar al usuario como primer factor.                |
| OTP por teléfono con texto | `sms`     | OTP por teléfono (SMS) utilizado para autenticar al usuario como primer factor.   |
| OTP por teléfono con voz   | `tel`     | OTP por teléfono con voz utilizado para autenticar al usuario como primer factor. |
| Password                   | `pwd`     | Password utilizada para autenticar al usuario como primer factor.                 |

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  `email_verified` y `phone_verified` se establecen automáticamente en el perfil de usuario cuando un usuario se autentica mediante Email OTP u OTP por teléfono. Si antes usabas una Action de Post Login para establecer manualmente estos indicadores, puedes eliminar esa solución alternativa.
</Callout>

Usa `event.authentication` para:

* Detectar qué factor passwordless completó el usuario (`email`, `sms`, `tel` o `pwd`)
* Añadir claims personalizadas a los tokens en función del método de autenticación
* Ejecutar lógica condicional según cómo se autenticó el usuario

<div id="example">
  #### Ejemplo
</div>

El siguiente ejemplo lee `event.authentication.methods` para detectar qué factor passwordless completó el usuario y lo añade como claim personalizado al token de identidad.

```javascript
exports.onExecutePostLogin = async (event, api) => {
  if (!event.authentication?.methods || event.authentication.methods.length === 0) {
    return;
  }

  if (event.connection.name !== 'YOUR_AUTH0_CONNECTION') {
    return;
  }

  const firstFactor = event.authentication.methods[0];

  if (firstFactor.name === 'email') {
    api.idToken.setCustomClaim('https://your-app.com/auth_method', 'email_otp');
  } else if (firstFactor.name === 'sms' || firstFactor.name === 'tel') {
    api.idToken.setCustomClaim('https://your-app.com/auth_method', 'phone_otp');
  } else if (firstFactor.name === 'pwd') {
    api.idToken.setCustomClaim('https://your-app.com/auth_method', 'password');
  }
};
```

<div id="post-challenge-trigger">
  ### trigger post-challenge
</div>

El trigger `post-challenge` se ejecuta después de que los usuarios completan una verificación, como un restablecimiento de contraseña, la validación del número de teléfono o MFA. El objeto `event.authentication` en [Auth0 Dashboard &gt; Actions &gt; Triggers &gt; password-reset-post-challenge](https://manage.auth0.com/#/actions/flows) expone los siguientes atributos:

| Atributo | Parámetro      | Descripción                                                                 |
| -------- | -------------- | --------------------------------------------------------------------------- |
| Email    | `email`        | Restablecimiento de contraseña con OTP por correo electrónico o Magic Link. |
| Teléfono | `phone_number` | Restablecimiento de contraseña con OTP por teléfono.                        |

<div id="example">
  #### Ejemplo
</div>

El siguiente ejemplo lee `event.authentication.methods` para detectar qué factor passwordless completó la verificación y lo agrega como un claim personalizado al token de identidad.

```javascript
exports.onExecutePostChallenge = async (event, api) => {
  if (!event.authentication?.methods || event.authentication.methods.length === 0) {
    return;
  }

  if (event.connection.name !== 'YOUR_AUTH0_CONNECTION') {
    return;
  }

  const firstFactor = event.authentication.methods[0];

  if (firstFactor.name === 'email') {
    api.idToken.setCustomClaim('https://your-app.com/challenge_method', 'email_otp');
  } else if (firstFactor.name === 'phone_number') {
    api.idToken.setCustomClaim('https://your-app.com/challenge_method', 'phone_otp');
  }
};
```

<div id="benefits">
  ## Beneficios
</div>

* **Implementación simplificada:** Menos conexiones que configurar y mantener. No es necesario vincular cuentas, salvo que se usen conexiones sociales/federadas junto con conexiones de base de datos.
* **Experiencia de usuario mejorada**: Ofrece combinaciones de OTP por correo electrónico y por teléfono con contraseñas, claves de acceso e inicio de sesión social/federado, todo desde una misma experiencia de Universal Login en la que primero se introduce el identificador.
* **Flujos de registro flexibles**: Configura los atributos de correo electrónico y teléfono como opcionales en el registro, lo que permite a los usuarios registrarse solo con un correo electrónico o solo con un número de teléfono, ideal para experiencias centradas en dispositivos móviles o solo por correo electrónico.
* **OTP por voz**: El OTP por voz se incluye como primer factor cuando se configura en Unified Phone Experience.

<div id="limitations">
  ## Limitaciones
</div>

* Disponible solo para flujos basados en Universal Login; todavía no es compatible con la autenticación basada en API.
* Passwordless para conexiones de base de datos no es compatible con Classic Login.
* Passwordless para conexiones de base de datos no es compatible con Implicit Signup &amp; Login.
* En las conexiones passwordless heredadas, no había diferencias en la experiencia de usuario entre el registro y el inicio de sesión. Las conexiones de base de datos distinguen entre la experiencia de registro y la de inicio de sesión, lo que requiere un registro y un inicio de sesión *explícitos*.
  * Si un usuario sin una identidad en Auth0 entra en el flujo de inicio de sesión, el sistema no lo registrará automáticamente. El usuario recibe un error después de validar el OTP.
  * Si un usuario con una identidad en Auth0 entra en el flujo de registro, recibe un error después de validar el OTP.
