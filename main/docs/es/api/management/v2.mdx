**La Auth0 Management API** es una colección de endpoints para realizar tareas administrativas de forma programática y debe ser utilizada por servidores de back-end o entidades de confianza. En términos generales, cualquier cosa que se pueda hacer a través del Auth0 Dashboard también se puede hacer a través de esta API.

Esta API es independiente de la [Auth0 Authentication API de acceso público](https://auth0.com/docs/api/authentication), que está pensada para ser utilizada por frontends y partes no confiables.

Al utilizar los ejemplos de código incluidos en esta documentación de la API, las solicitudes deben enviarse con un Content-Type de `application/json`. Todos los endpoints aceptan un tamaño máximo de payload de 1 megabyte.

La documentación de la Auth0 Management API sigue el [esquema OpenAPI v3.1 de la Auth0 Management API](https://auth0.com/docs/api/management/openapi.json). Ten en cuenta que la compatibilidad con el esquema OpenAPI v3.1 se encuentra actualmente en Beta.

<div id="authentication">
  ## Autenticación
</div>

El uso de la Auth0 Management API requiere un token de acceso de la Management API. Para obtener información sobre cómo solicitar este token, lee [Management API Access Tokens](https://auth0.com/docs/secure/tokens/access-tokens/management-api-access-tokens).

La Auth0 Management API utiliza JSON Web Tokens (JWT) para autenticar solicitudes. La claim `scopes` del token de acceso de la Management API indica qué métodos de solicitud se pueden usar al llamar a esta API. El token de ejemplo deserializado en esta página concede acceso de solo lectura a los usuarios y acceso de lectura y escritura a las conexiones. Si intentas usar cualquier método de solicitud que no esté permitido dentro de los scopes establecidos, el resultado será una respuesta **403 Forbidden**.

```bash lines
{
  "aud": "m8DAxghyfE0KdpzogfXgMSxrkCSdKVEF",
  "scopes": {
    "connections": {
      "actions": ["read", "update"]
    }
  },
  "iat": "1446056652",
  "jti": "7e9c6a991f5a227fb7ebaa522536ae4c"
}
```

Para llamar a la API, envía el token de la API en el encabezado HTTP Authorization mediante el [esquema de autenticación Bearer](https://tools.ietf.org/html/draft-ietf-oauth-v2-bearer-20#section-2.1).

```bash lines
curl -H "Authorization: Bearer eyJhb..." https://@@TENANT@@/api/v2/users
```

<div id="request-correlation">
  ## Correlación de solicitudes
</div>

Un ID de correlación es un identificador único (de hasta 64 caracteres) de una sola operación de la Management API y permite rastrear esas operaciones en los registros del tenant. Para obtener más información, consulta [Logs](https://auth0.com/docs/deploy-monitor/logs).

La API acepta un ID de correlación proporcionado por el cliente si se envía en el encabezado HTTP `X-Correlation-ID` en los métodos `POST`, `PUT`, `PATCH` y `DELETE`.

```bash lines
curl -H "Authorization: Bearer eyJhb..." -H "x-correlation-id: client1_xyz" https://@@TENANT@@/api/v2/users
```

Si se proporciona un valor del encabezado `X-Correlation-ID` de más de 64 caracteres, solo se mostrarán los primeros 64 caracteres en los logs.

```bash lines
"references": {
  "correlation_id": "client1_xyz"
}
```

<div id="pagination">
  ## Paginación
</div>

La paginación es una técnica que usan las API para dividir grandes conjuntos de datos en páginas manejables, lo que reduce la cantidad de datos que se devuelve en cada respuesta. En las API, los dos tipos principales de paginación son la **basada en offset** y la **basada en checkpoint**. Cada una tiene ventajas y casos de uso específicos según el tamaño del conjunto de datos y los requisitos de recuperación.

La Auth0 Management API admite ambos tipos de paginación en muchos endpoints, como `GET /api/v2/clients` y `GET /api/v2/logs`. Cuando ambas opciones están disponibles, se recomienda la **paginación basada en checkpoint** por su mayor eficiencia y estabilidad al trabajar con grandes conjuntos de datos.

<div id="offset-based-pagination">
  ### Paginación basada en offset
</div>

La paginación basada en offset es un método sencillo y muy utilizado para paginar conjuntos de datos de hasta aproximadamente 1.000 elementos. Este enfoque utiliza los parámetros `page` y `per_page` para definir el punto de inicio y la cantidad de elementos de cada página.

* **Parámetros:**
  * `page`: El número de página, con índice inicial cero, que se va a recuperar. El valor predeterminado es `0` si no se especifica.
  * `per_page`: La cantidad de elementos que se devolverán por página. Para los tenants de Public Cloud, [el máximo es `50`](https://auth0.com/docs/troubleshoot/product-lifecycle/past-migrations/migrate-to-paginated-queries); para Private Cloud, el máximo es `100`. Si no se especifica, el valor predeterminado es la mitad del máximo.

**Ejemplo de solicitud de paginación basada en offset:**

```bash lines
curl -L "https://@@TENANT@@/api/v2/clients?per_page=10&page=2" \
-H 'Authorization: Bearer {ACCESS_TOKEN}' \
-H 'Accept: application/json'
```

En la paginación por offset:

* Si `page * per_page` supera el número total de resultados, se devuelve un arreglo vacío.
* Cada solicitud de página recalcula el offset, lo que puede afectar el rendimiento con conjuntos de datos más grandes. Por lo general, la paginación por offset es más adecuada para colecciones que probablemente no superen los 1.000 elementos.

<div id="checkpoint-based-pagination">
  ### Paginación basada en checkpoint
</div>

La paginación basada en checkpoint, también conocida como paginación basada en cursores o en tokens, está optimizada para grandes conjuntos de datos. Este método utiliza un ID de checkpoint `next` proporcionado por el servidor para recuperar las páginas siguientes en una secuencia solo hacia adelante. El ID de checkpoint `next` se incluye en la respuesta cuando hay más resultados disponibles.

Para continuar con la paginación, usa el ID de checkpoint `next` en el parámetro de consulta `from` de la siguiente solicitud. Este ID es opaco y debe pasarse sin modificaciones.

* **Parámetros:**
  * `from`: El siguiente ID de checkpoint de la respuesta anterior, que se usa para recuperar la siguiente página de resultados.
  * `take`: El número de elementos que se devolverán por página. Para los tenants de Public Cloud, [el máximo es `50`](https://auth0.com/docs/troubleshoot/product-lifecycle/past-migrations/migrate-to-paginated-queries); para Private Cloud, el máximo es `100`. Si no se proporciona, el valor predeterminado es la mitad del máximo.

**Ejemplo de solicitud de paginación basada en checkpoint:**

```bash lines
curl -L "https://@@TENANT@@/api/v2/clients?take=10&from=Cg1HRUY3NEszUERFME40GgAiAQgCEj..." \
-H 'Authorization: Bearer {ACCESS_TOKEN}' \
-H 'Accept: application/json'
```

<div id="checkpoint-id-expiry">
  #### Caducidad del ID de checkpoint
</div>

Al usar la paginación basada en checkpoint, es importante tener en cuenta el período de validez de cada ID de checkpoint `next`. El ID de checkpoint está diseñado para usarse de forma secuencial, y cada ID solo es válido durante un tiempo limitado para garantizar la coherencia de los datos.

<Note>
  <p class="uppercase font-bold">Nota</p>

  El ID de checkpoint `next` es válido durante **24 horas** desde su emisión. Si caduca, se requiere una nueva solicitud para volver a empezar desde el inicio del conjunto de datos. Considera almacenar en caché los resultados si puede transcurrir un período prolongado entre solicitudes.
</Note>

<div id="forward-only-constraints">
  #### Restricciones de solo avance
</div>

La paginación basada en checkpoint es de solo avance. Evita usar el ID del checkpoint para navegar hacia atrás o para solicitudes fuera de secuencia, ya que esto puede provocar errores. Usa siempre el ID del checkpoint `next` de la respuesta anterior.

<div id="choosing-between-offset-and-checkpoint-pagination">
  ### Elegir entre la paginación por offset y la paginación por checkpoint
</div>

Cuando se admiten ambos tipos de paginación:

* **Usa la paginación basada en checkpoint** para gestionar grandes conjuntos de datos de forma eficiente.
* Usa la paginación basada en offset para conjuntos de datos más pequeños (normalmente, menos de 1.000 elementos), ya que es más sencilla de implementar, aunque menos eficiente para colecciones grandes.

<div id="best-practices-for-handling-pagination">
  ### Prácticas recomendadas para gestionar la paginación
</div>

* **Consistencia de los datos:** Cada solicitud paginada refleja el estado de los datos en el momento en que se realiza. Si los datos se actualizan o eliminan, puede haber elementos omitidos o duplicados. La paginación basada en checkpoint puede ayudar a que la paginación sea más fluida en conjuntos de datos dinámicos.
* **Almacenamiento de checkpoints:** Para recuperar grandes volúmenes de datos, considera almacenar checkpoints después de cada página para poder reanudar desde el último checkpoint en caso de interrupción.

Este enfoque permite recuperar datos de forma eficiente y estable en conjuntos de datos grandes, en línea con las opciones de paginación de Auth0 Management API.
