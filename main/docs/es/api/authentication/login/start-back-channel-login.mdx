<div id="endpoint">
  ## Endpoint
</div>

`POST /bc-authorize`

El endpoint de Back-Channel Login permite a las aplicaciones enviar una solicitud de autenticación al teléfono de un usuario o al dispositivo de autenticación, siempre que tengan una aplicación instalada y estén inscritos para [recibir notificaciones push con Guardian SDK](https://auth0.com/docs/secure/multi-factor-authentication/auth0-guardian#enroll-in-push-notifications).

Utilice el endpoint de Back-Channel Login para autenticar a los usuarios en los siguientes casos de uso:

* Los usuarios no están frente a la aplicación que requiere autenticación, por ejemplo, cuando llaman por teléfono a un centro de atención.
* El dispositivo de consumo, es decir, el dispositivo que permite al usuario utilizar un servicio, no es seguro para operaciones sensibles (por ejemplo, un navegador web para transacciones financieras).
* El dispositivo de consumo tiene una capacidad de interacción limitada (por ejemplo, bicicletas eléctricas o patinetes eléctricos).

<div id="response-body">
  ### Cuerpo de la respuesta
</div>

Si la solicitud se procesa correctamente, deberías recibir una respuesta como la siguiente:

```http
{
  "auth_req_id": "eyJh...",
  "expires_in": 300,
  "interval": 5
}
```

El valor de `auth_req_id` debe conservarse, ya que se utiliza más adelante en el flujo para identificar la solicitud de autenticación.

El valor de `expires_in` indica cuántos segundos faltan para que caduque la solicitud de autenticación.

El valor de `interval` indica cuántos segundos debes esperar entre cada solicitud de consulta.

La solicitud debe aprobarse o rechazarse en el dispositivo de autenticación del usuario mediante el Guardian SDK.

<div id="remarks">
  ### Observaciones
</div>

* El valor de `login_hint` sigue el formato `iss_sub`. Por ejemplo:

```http
{ "format": "iss_sub", "iss": "https://[TENANT_DOMAIN]/", "sub": "auth0|[USER ID]" }
```

* La autenticación puede realizarse con secreto de cliente (mediante HTTP Basic Auth o Post), clave privada JWT o mTLS. Consulta los ejemplos detallados anteriores.

<div id="body-parameters">
  ## Parámetros del Body
</div>

<ParamField body="client_id" type="string" required>
  El `client_id` de tu aplicación.
</ParamField>

<ParamField body="binding_message" type="string" required>
  Una cadena legible para que el usuario se asegure de aprobar la solicitud correcta.
</ParamField>

<ParamField body="login_hint" type="string" required>
  Un objeto JSON que contiene los detalles del usuario para la authentication en el formato `iss_sub`.
</ParamField>

<ParamField body="scope" type="string" required>
  Lista de scopes de OIDC y de custom API separada por espacios, que debe incluir como mínimo `openid`.
</ParamField>

<ParamField body="audience" type="string">
  Identificador único opcional de la API de destino para los access tokens.
</ParamField>

<ParamField body="resource" type="string">
  El identificador de la API de destino (resource server) a la que quieres acceder. Debe coincidir con un API identifier registrado en tu tenant de Auth0. Se usa como alternativa a `audience` cuando el [Resource Parameter Compatibility Profile](https://auth0.com/docs/get-started/tenant-settings#settings-advanced) del tenant está configurado en `compatibility`.
</ParamField>

<ParamField body="requested_expiry" type="string">
  Tiempo de expiración personalizado de la solicitud en segundos (1-300). El valor predeterminado es 300 segundos.
</ParamField>

<ParamField body="authorization_details" type="array">
  Un arreglo JSON opcional de objetos que describen los permisos que se autorizarán. El valor `type` de cada objeto debe haberse registrado previamente en el resource server mediante el parámetro `authorization_details` del Resource Server. Para obtener más información, consulta la documentación de Management API sobre [Actualizar un resource server](https://auth0.com/docs/api/management/v2/resource-servers/patch-resource-servers-by-id#body-parameters).
</ParamField>

<div id="response-messages">
  ## Mensajes de respuesta
</div>

| Estado | Descripción                                    |
| ------ | ---------------------------------------------- |
| 200    | Solicitud de autenticación iniciada con éxito. |
| 400    | Solicitud incorrecta - Parámetros no válidos   |
| 500    | Error interno del servidor                     |
