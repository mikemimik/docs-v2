title: "Obtener token"
description: "El intercambio de tokens On-Behalf-Of (OBO) permite que un servicio de nivel intermedio intercambie un token de usuario entrante por un nuevo token con alcance limitado al servicio de destino."
---

`POST /oauth/token`

El intercambio de tokens On-Behalf-Of (OBO) ([RFC 8693](https://datatracker.ietf.org/doc/html/rfc8693)) permite que los servicios de nivel intermedio intercambien un token de usuario con alcance específico por un nuevo token para llamar a servicios de destino.

El nuevo token mantiene la identidad y los permisos del usuario original, a la vez que limita su alcance específicamente al servicio de destino, lo que permite a ese servicio tomar decisiones de autorización en función del usuario final. La cadena de delegación se registra en el claim `act` (actor), donde cada nivel representa un servicio dentro de la cadena de llamadas. Para obtener más información, consulta la [documentación de On-Behalf-Of Token Exchange](https://auth0.com/docs/secure/call-apis-on-users-behalf/on-behalf-of-token-exchange).

<div id="remarks">
  ## Observaciones
</div>

* Solo los clientes de API personalizada asociados a un servidor de recursos pueden usar el intercambio de tokens OBO. Un cliente de API personalizada debe cumplir los siguientes requisitos:
  * Establecer `app_type` en `resource_server`.
  * Establecer `resource_server_identifier` en un servidor de recursos válido, es decir, `https://my-api.example.com`. Auth0 usa el identificador del servidor de recursos como parámetro `audience` en las llamadas de autorización.

* Los scopes emitidos para la aplicación pueden diferir de los scopes solicitados. En este caso, se incluirá un parámetro `scope` en la respuesta JSON. Los scopes se basan en las [políticas de control de acceso basado en roles (RBAC)](/docs/es/manage-users/access-control/rbac) del usuario.

* Los intercambios de tokens OBO activan el [trigger de Action `Post Login`](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger), donde `event.transaction.protocol` se establece en `oauth2-token-exchange` y `event.transaction.actor` registra toda la cadena de delegación.

* La cadena de delegación está limitada a cinco niveles anidados. El intercambio de tokens OBO falla si el token de sujeto ya tiene cinco niveles `act` anidados.

* Almacena en caché los tokens de acceso durante la duración del token en lugar de solicitar uno nuevo para cada llamada a la API. Los tokens de acceso pueden reutilizarse hasta que caduquen; repetir intercambios de tokens desperdicia recursos, aumenta la latencia y puede activar límites de frecuencia.

<div id="parameters">
  ## Parámetros
</div>

<ParamField header="DPoP" type="string">
  Una prueba DPoP para la solicitud. Es opcional y solo se requiere si tu aplicación usa [Demonstrating Proof-of-Possession](/docs/es/secure/sender-constraining/demonstrating-proof-of-possession-dpop).
</ParamField>

<ParamField header="auth0-forwarded-for" type="string">
  La IP del usuario final como una cadena de texto. Establece este valor si quieres que la protección [Suspicious IP Throttling](/docs/es/secure/attack-protection/suspicious-ip-throttling) funcione en escenarios del lado del servidor.
</ParamField>

<div id="request-body">
  ## Cuerpo de la solicitud
</div>

<ParamField body="grant_type" type="string" required>
  Indica el flujo que estás usando. Para On-Behalf-Of Token Exchange, usa `urn:ietf:params:oauth:grant-type:token-exchange`.
</ParamField>

<ParamField body="subject_token_type" type="string" required>
  El tipo del token de sujeto. Para On-Behalf-Of Token Exchange, usa `urn:ietf:params:oauth:token-type:access_token`.
</ParamField>

<ParamField body="subject_token" type="string" required>
  El token de acceso de Auth0 entrante del usuario o del servicio upstream que el servicio de nivel intermedio tiene actualmente.
</ParamField>

<ParamField body="requested_token_type" type="string" required>
  Indica qué tipo de token quieres recibir. Para On-Behalf-Of Token Exchange, usa `urn:ietf:params:oauth:token-type:access_token`.
</ParamField>

<ParamField body="client_id" type="string" required>
  El ID de cliente de tu cliente de API personalizada. El cliente de API personalizada debe estar asociado a un servidor de recursos (el mismo identificador). Al igual que con otros tipos de concesión, también puedes pasar el ID de cliente en el encabezado Authorization mediante HTTP Basic Auth.
</ParamField>

<ParamField body="client_secret" type="string" required>
  El secreto de cliente de tu cliente de API personalizada. Al igual que con otros tipos de concesión, también puedes pasar el secreto de cliente en el encabezado Authorization mediante `HTTP` Basic Auth. Consulta las alternativas en la [documentación de referencia de Auth0 Authentication API](https://auth0.com/docs/api/authentication#authentication-methods). Ten en cuenta que no puedes establecer `token_endpoint_auth_method` en `none` para el intercambio de tokens OBO.
</ParamField>

<ParamField body="audience" type="string" required>
  El identificador único de la API de destino a la que quieres acceder. Este es el identificador del servicio de destino que recibe y valida el nuevo token.
</ParamField>

<ParamField body="scope" type="string">
  (Opcional) Una lista de permisos específicos para la llamada descendente, delimitada por espacios. Si no se especifica, se incluirán todos los scopes concedidos al usuario para la audiencia de destino según las políticas de RBAC.
</ParamField>

<div id="response">
  ## Respuesta
</div>

| Estado | Descripción                                                                                                                                                                   |
| ------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 200    | Respuesta correcta. Devuelve un token de acceso para la API de destino y conserva la misma identidad de usuario.                                                              |
| 400    | Solicitud incorrecta. Puede ocurrir si la profundidad de la cadena de delegación supera el límite máximo permitido de 4 niveles anidados o si faltan parámetros obligatorios. |
| 401    | No autorizado. El token de sujeto no es válido, ha caducado o las credenciales de cliente son incorrectas.                                                                    |
| 403    | Prohibido. El cliente no tiene permiso para realizar el intercambio de tokens, o el usuario no tiene los scopes solicitados para la audiencia de destino.                     |

<ResponseExample>
  ```json 200 Response
  {
    "access_token": "eyJ...",
    "expires_in": 86400,
    "token_type": "Bearer",
    "issued_token_type": "urn:ietf:params:oauth:token-type:access_token"
  }
  ```

  ```json 400 Response (Delegation chain limit exceeded)
  {
    "error": "invalid_request",
    "error_description": "Delegation chain (`act` claim) depth exceeds the maximum allowed limit of 5"
  }
  ```
</ResponseExample>

<div id="response-fields">
  ## Campos de respuesta
</div>

<ResponseField name="access_token" type="string">
  El nuevo token de acceso de Auth0 con scope para la API de destino. Este [JSON Web Token (JWT)](/docs/es/secure/tokens/json-web-tokens#json-web-tokens) contiene el mismo `sub` (identidad del usuario) que el token de sujeto, con `aud` establecido en la audience de la API de destino solicitada. El claim `act` rastrea la cadena de delegación.
</ResponseField>

<ResponseField name="issued_token_type" type="string">
  Confirma el formato del token devuelto. Esto coincide con el `requested_token_type` de tu solicitud.

  Valor: `urn:ietf:params:oauth:token-type:access_token`
</ResponseField>

<ResponseField name="token_type" type="string">
  Especifica el esquema de autenticación que se debe usar en el encabezado Authorization. Para OBO, este es `Bearer`, a menos que se use DPoP, en cuyo caso se usará `DPoP`.
</ResponseField>

<ResponseField name="expires_in" type="number">
  La duración del token en segundos.
</ResponseField>

<ResponseField name="scope" type="string">
  (Opcional) Solo se incluye si los scopes concedidos difieren de los scopes solicitados. Lista de scopes concedidos realmente, delimitada por espacios.
</ResponseField>
