El [Client Credentials Flow](/docs/es/get-started/authentication-and-authorization-flow/client-credentials-flow) permite que las aplicaciones de máquina a máquina (M2M), como CLI, daemons o servicios de backend, obtengan <Tooltip tip="Token de acceso: credencial de autorización, en forma de una cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=access+tokens">tokens de acceso</Tooltip> en su propio nombre sin interacción del usuario.

Aplicar las prácticas recomendadas para el almacenamiento en caché de tokens de acceso puede minimizar los viajes de ida y vuelta a Auth0 y controlar el volumen de emisión de tokens. Sin embargo, algunas aplicaciones, como sistemas de terceros o las que forman parte de implementaciones grandes y complejas, pueden no aplicar un almacenamiento en caché adecuado y mostrar un comportamiento en ráfagas. Esto puede hacer que Auth0 genere un gran volumen de tokens M2M en períodos cortos, lo que afecta a la cuota general de tokens M2M de tu tenant y provoca un comportamiento inesperado del servicio.

Las cuotas granulares de tokens M2M de Auth0 te permiten establecer límites horarios y diarios de tokens de acceso M2M para aplicaciones y Auth0 Organizations. Esto proporciona un control operativo granular sobre la emisión de tokens para evitar un consumo excesivo.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Las cuotas granulares de tokens M2M son distintas de la asignación mensual total de tokens M2M de tu tenant. Puedes usar las cuotas granulares de tokens como medida de protección frente a solicitudes rápidas de tokens.
</Callout>

<div id="use-cases">
  ## Casos de uso
</div>

* **Controla las aplicaciones descontroladas**: Aplica cuotas a nivel de aplicación para evitar que las aplicaciones M2M soliciten tokens con demasiada frecuencia (por errores, falta de caché, etc.) sin afectar la cuota general de tokens M2M de tu tenant.
* **Garantiza un uso justo en entornos multi-tenant**: Asigna cuotas a nivel de organización a distintos clientes (Auth0 Organizations) para evitar que uno consuma la cuota de los demás.
* **Establece salvaguardas para todo el tenant**: Usa cuotas predeterminadas a nivel del tenant como límite de línea base frente a solicitudes excesivas de tokens M2M por parte de aplicaciones y organizaciones.
* **Supervisa antes de aplicar restricciones**: Despliega cuotas con `enforce: false` para observar el consumo de tokens mediante [registros de advertencia de consumo](#consumption-warning-logs) y [encabezados de cuota de Auth0](#auth0-quota-headers). Luego, establece límites fundamentados antes del bloqueo activo.

<div id="how-it-works">
  ## Cómo funciona
</div>

Puedes aplicar cuotas granulares de token M2M a las siguientes entidades:

* **Aplicación**: Limita los tokens generados para un cliente para controlar el comportamiento de cada aplicación.
* **Organización**: Limita los tokens de las solicitudes realizadas con una organización para ayudar a gestionar la actividad de máquina a máquina del cliente de SaaS o de la unidad de negocio en las aplicaciones que utilizan tokens asociados con un `org_id`.

Auth0 admite dos tipos de cuotas granulares de token M2M:

* **Por hora:** La cantidad máxima de tokens que se pueden obtener en una sola hora. La cuota se restablece al comienzo de cada hora UTC.
* **Diaria:** La cantidad máxima de tokens que se pueden obtener en un período de 24 horas. La cuota se restablece al comienzo de cada día UTC.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  El mecanismo de recuento de tokens de Auth0 es de consistencia eventual, por lo que puede haber excesos leves y breves antes de que la cuota se aplique a solicitudes posteriores.
</Callout>

<div id="quota-evaluation-and-enforcement-rules">
  ### Reglas de evaluación y aplicación de cuotas
</div>

Auth0 determina y aplica cuotas granulares de token M2M para aplicaciones y organizaciones asociadas con tu tenant:

1. **Cuota a nivel de cliente:**

   * Si has configurado una **cuota específica para la aplicación** del `client_id`, Auth0 aplica la cuota al cliente.
   * De lo contrario, Auth0 aplica al cliente la **cuota predeterminada a nivel del tenant para clientes**. Si no has configurado un valor predeterminado a nivel del tenant para clientes, Auth0 no aplica ninguna cuota granular al cliente.
2. **Cuota a nivel de organización:**

   * Si has configurado una **cuota específica para la organización** del `org_id` asociado con la solicitud de token, Auth0 aplica la cuota a la organización.
   * De lo contrario, Auth0 aplica al `org_id` asociado con la solicitud de token la **cuota predeterminada a nivel del tenant para Organizations**. Si no has configurado un valor predeterminado a nivel del tenant para Organizations, Auth0 no aplica ninguna cuota granular al `org_id` asociado con la solicitud de token.
3. Según si la solicitud de token tiene una organización asociada, Auth0 comprueba si existe una cuota aplicable:

   * Si la solicitud de token tiene una organización asociada definida explícitamente o mediante la configuración predeterminada de la organización, Auth0 comprueba de forma simultánea las cuotas a nivel de cliente y de organización.
   * Si la solicitud de token no tiene una organización asociada, Auth0 solo comprueba la cuota a nivel de cliente.

Auth0 evalúa de forma simultánea todas las cuotas aplicables a la transacción y hace cumplir aquellas que tienen el indicador `enforce` establecido en `true` en la configuración de la cuota:

* **Cuando se hace cumplir una cuota (**`enforce: true`**):** Si se supera una cuota aplicada, la solicitud de token se **rechaza** con un error HTTP 429 Too Many Requests. Para obtener más información, consulta [Respuestas de error por cuotas superadas](#error-responses-for-exceeded-quotas).
* **Cuando no se hace cumplir una cuota (**`enforce: false`**):** Las solicitudes de token **no se rechazan** por esta cuota, aunque se supere su límite. Auth0 igualmente contabiliza los tokens y genera [registros de advertencia de consumo](#consumption-warning-logs) en los umbrales, lo que permite el monitoreo antes de habilitar la aplicación de la cuota.

<div id="observability">
  ### Observabilidad
</div>

Para proporcionar visibilidad sobre el consumo de cuota, Auth0 genera [registros de advertencia de consumo](#consumption-warning-logs). Estos registros se generan cuando una cuota alcanza el 60 %, el 80 % y el 100 % de su límite. Puedes usar estos registros para supervisar de forma proactiva el uso de tokens e identificar posibles incidencias.

Las aplicaciones también pueden determinar mediante programación la cuota que les queda usando [encabezados de cuota de Auth0](#auth0-quota-headers). Estos encabezados HTTP se incluyen tanto en las respuestas correctas como en las de error a las solicitudes del Client Credentials Flow, y proporcionan información en tiempo real sobre el uso de la cuota.

<div id="configure-m2m-token-quotas">
  ## Configurar cuotas de token M2M
</div>

Puedes configurar cuotas granulares de token M2M mediante la [Management API](https://auth0.com/docs/api/management/v2).

<div id="tenant-level-default-quotas">
  ### Cuotas predeterminadas a nivel del tenant
</div>

Las cuotas a nivel del tenant son la configuración predeterminada para todas las aplicaciones y organizaciones dentro de tu tenant de Auth0. Puedes usar cuotas a nivel del tenant para establecer límites de línea base para el uso de tokens de máquina a máquina. Estas cuotas se sustituyen por las cuotas específicas de la aplicación y de la organización.

Puedes especificar cuotas predeterminadas de tokens para las aplicaciones y/o organizaciones asociadas a tu tenant. Para configurar cuotas a nivel del tenant, usa el endpoint [Actualizar un tenant](https://auth0.com/docs/api/management/v2/tenants/patch-settings):

```bash lines
curl --request PATCH 'https://YOUR_DOMAIN/api/v2/tenants/settings' \
  --header 'Authorization: Bearer YOUR_MANAGEMENT_API_TOKEN' \
  --header 'Content-Type: application/json' \
  --data '{
    "default_token_quota": {
      "clients": {
        "client_credentials": {
          "per_hour": 100,
          "per_day": 1000,
          "enforce": true
        }
      },
      "organizations": {
        "client_credentials": {
          "per_hour": 500,
          "per_day": 5000,
          "enforce": false
        }
      }
    }
  }'
```

El objeto `default_token_quota` contiene las siguientes configuraciones de cuota de tokens a nivel de aplicación y de organización:

* `clients.client_credentials`: (Opcional) Define cuotas de tokens M2M para aplicaciones individuales.
* `organizations.client_credentials`: (Opcional) Define cuotas de tokens M2M para organizaciones.
* `per_hour`: (Opcional) Establece la cuota horaria de tokens.
* `per_day`: (Opcional) Establece la cuota diaria de tokens.
* `enforce`: (Opcional) Determina si las solicitudes de token se rechazan cuando se supera la cuota. El valor predeterminado es true.

En el ejemplo, todas las aplicaciones del tenant están limitadas de forma predeterminada a 100 tokens M2M por hora y 1000 por día. Para Organizations, el valor predeterminado es de 500 tokens por hora y 5000 por día, pero la aplicación de estos límites está deshabilitada actualmente (`enforce: false`). Esto le permite supervisar el uso de tokens de la organización antes de aplicar los límites.

<div id="application-specific-quotas">
  ### Cuotas específicas por aplicación
</div>

Puedes configurar cuotas para aplicaciones específicas, anulando los valores predeterminados a nivel del tenant. Esto resulta útil para controlar con mayor precisión el comportamiento de cada aplicación.

Puedes establecer cuotas específicas por aplicación al crear o actualizar una aplicación.

Para crear una aplicación con una cuota de token, usa el endpoint [Crear un cliente](https://auth0.com/docs/api/management/v2/clients/post-clients):

```bash lines
curl --location 'https://YOUR_DOMAIN/api/v2/clients' \
--header 'Authorization: Bearer YOUR_MANAGEMENT_API_TOKEN' \
--header 'Content-Type: application/json' \
--data '{
    "name": "APP_NAME",
    "app_type": "non_interactive",
    "token_quota": {
      "client_credentials": {
        "per_hour": 10,
        "per_day": 50,
        "enforce": true
      }
    }
}'
```

Para actualizar la cuota de tokens de una aplicación, utilice el endpoint [Actualizar un cliente](https://auth0.com/docs/api/management/v2/clients/patch-clients-by-id):

```bash lines
curl --location --request PATCH 'https://YOUR_DOMAIN/api/v2/clients/a01FUMJHEtb0q8jcXm7y2k9EAGe5fcxZ' \
--header 'Content-Type: application/json' \
--header 'Authorization: Bearer YOUR_MANAGEMENT_API_TOKEN' \
--data '{
    "token_quota": {
        "client_credentials": {
            "per_hour": 5,
            "enforce": true
        }
    }
}'
```

El objeto `token_quota` contiene la configuración de la cuota de tokens de la aplicación:

* `client_credentials`: Define las cuotas de tokens de máquina a máquina.
* `per_hour`: Establece la cuota de tokens por hora.
* `per_day`: Establece la cuota diaria de tokens.
* `enforce`: Determina si se rechazan las solicitudes de tokens cuando se supera la cuota. El valor predeterminado es `true`.

En este ejemplo, la aplicación con `{yourClientId}` está limitada a 10 tokens de máquina a máquina por hora y 50 por día, lo que anula cualquier configuración a nivel del tenant.

Puede obtener la configuración de la cuota de tokens de una aplicación mediante el endpoint [Obtener un cliente por ID](https://auth0.com/docs/api/management/v2/clients/get-clients-by-id).

  ## Supervisar el consumo de la cuota de tokens
</div>

<div id="auth0-quota-headers">
  ### Encabezados de cuota de Auth0
</div>

Auth0 incluye encabezados HTTP en las respuestas del Client Credentials Flow (tanto en las respuestas correctas como en las respuestas de error 429) para ofrecer información en tiempo real sobre el consumo de cuota:

* `Auth0-Client-Quota-Limit`: Proporciona información de cuota de la aplicación.
* `Auth0-Organization-Quota-Limit`: Proporciona información de cuota de la organización.

Auth0 solo incluye en la respuesta los encabezados correspondientes a las cuotas de la solicitud de token.

El siguiente fragmento de código muestra un ejemplo de encabezado de cuota de Auth0:

```http lines
Auth0-Client-Quota-Limit: b=per_hour;q=10;r=7;t=3540,b=per_day;q=50;r=47;t=43200
Auth0-Organization-Quota-Limit: b=per_hour;q=50;r=47;t=3540,b=per_day;q=250;r=247;t=43200
```

Los valores de los encabezados son listas de buckets de cuota separadas por comas. Cada bucket se representa como una lista de pares clave-valor separados por punto y coma:

* `b` (bucket&#95;name): El nombre del bucket de cuota (`per_hour` o `per_day`).
* `q` (quota): El límite de cuota configurado para el bucket.
* r (remaining): El número de tokens restantes en el bucket.
* `t` (time): El número de segundos hasta que el bucket se restablezca.

En el ejemplo de `Auth0-Client-Quota-Limit`:

* La aplicación tiene una cuota por hora (`b=per_hour`) de 10 tokens (`q=10`). Le quedan 7 tokens (`r=7`) y la cuota se restablece en 3540 segundos (`t=3540`).
* La aplicación también tiene una cuota diaria (`b=per_day`) de 50 tokens (`q=50`). Le quedan 47 tokens (`r=47`) y la cuota se restablece en 43200 segundos (`t=43200`).

Los SDKs de Auth0 proporcionan utilidades integradas para leer y analizar encabezados de cuota, lo que permite a las aplicaciones aprovechar información de cuota en tiempo real para depurar, implementar estrategias de ralentización y mucho más. A continuación se muestra un ejemplo con Node.js y el SDK [node-auth0](https://github.com/auth0/node-auth0):

export const codeExample1 = `const {
    AuthenticationClient,
    HttpResponseHeadersUtils,
  } = require("auth0");

const auth0 = new AuthenticationClient({
    domain: '{{yourDomain}}',
    clientId: '{{yourClientId}}',
    clientSecret: '{{yourSecret}}'
});

async function getAccessToken() {
    try {
        //obtener el token
        const response = await auth0.oauth.clientCredentialsGrant({
            audience: '{{yourAudience}}'
        });

        // Mostrar a la aplicación información sobre la cuota del token de máquina a máquina para supervisarla
        const clientQuota = HttpResponseHeadersUtils.getClientQuotaLimit(response.headers);
        console.log("clientQuota ", clientQuota);
        console.log("clientQuota por día - cuota:", clientQuota?.perDay?.quota);
        console.log("clientQuota por día - restante:", clientQuota?.perDay?.remaining);
        console.log("clientQuota por día - se restablece después de:", clientQuota?.perDay?.resetAfter);
        console.log("clientQuota por hora - cuota:", clientQuota?.perHour?.quota);
        console.log("clientQuota por hora - restante:", clientQuota?.perHour?.remaining);
        console.log("clientQuota por hora - se restablece después de:", clientQuota?.perHour?.resetAfter);

        return response.data.access_token;

    } catch (error) {

        //gestiona el error aquí

    }
}

getAccessToken();`;

<div id="error-responses-for-exceeded-quotas">
  ### Respuestas de error por cuotas superadas
</div>

Cuando se supera una cuota aplicada, Auth0 Authentication API devuelve un error `HTTP 429 Too Many Requests`. En el cuerpo de la respuesta, Auth0 devuelve el código de error con una descripción más detallada. Auth0 también genera un registro de eventos de tipo `feccft` (por un intercambio fallido de un token de acceso por una Concesión de credenciales de cliente).

La siguiente muestra de código es un ejemplo de respuesta de error para una cuota superada:

```json lines
{
  "error": "too_many_requests",
  "error_description": "Cuota de cliente excedida"
}
```

Además del cuerpo de la respuesta, Auth0 devuelve los siguientes encabezados:

* `Auth0-Client-Quota-Limit` o `Auth0-Organization-Quota-Limit`: El encabezado de cuota de Auth0 correspondiente a la cuota consumida de la aplicación o la organización. Para obtener más información sobre el formato del encabezado, consulta [encabezados de cuota de Auth0](#auth0-quota-headers).
* `X-RateLimit-Limit`: El límite configurado para la cuota que se ha consumido.
* `X-RateLimit-Remaining`: Se establece en cero, lo que indica que la cuota se ha consumido por completo.
* `X-RateLimit-Reset`: Una marca de tiempo UNIX (en segundos) que representa el momento en que se espera que la cuota se restablezca y se permitan más solicitudes.
* `Retry-After`: El número de segundos que faltan para que la cuota se restablezca y las solicitudes posteriores se procesen correctamente.

Para obtener más información sobre los encabezados de límite de frecuencia, consulta [Predict when requests to a tenant will be rate-limited](/docs/es/troubleshoot/customer-support/operational-policies/rate-limit-policy/rate-limit-use-cases#predict-when-requests-to-a-tenant-will-be-rate-limited-).

Usa el encabezado `Retry-After` para determinar el tiempo de espera necesario para obtener un nuevo M2M token, de modo que tu aplicación pueda decidir si reintentar o lanzar un error. Esto también es importante para evitar consumir el límite de frecuencia de tu tenant.

El siguiente ejemplo de código en Node.js intenta obtener un nuevo M2M token, recibe un error 429 y vuelve a intentarlo si el tiempo de espera es inferior a 60 segundos. En caso contrario, lanza un error:

export const codeExample2 = `const {
    AuthenticationClient,
    HttpResponseHeadersUtils,
  } = require("auth0");

const auth0 = new AuthenticationClient({
    domain: '{{yourDomain}}',
    clientId: '{{yourClientId}}',
    clientSecret: '{{yourSecret}}'
});

const MAX_RETRY_AFTER = 60;

async function sleep(seconds) {
    return new Promise(resolve => setTimeout(resolve, seconds * 1000));
}

async function getAccessToken(retry = true) {
    try {
        // obtener el token
        const response = await auth0.oauth.clientCredentialsGrant({
            audience: '{{yourAudience}}'
        });

        return response.data.access_token;

    } catch (error) {

        if (error.statusCode === 429 && retry) {
            // si se produce un error 429, calcula cuánto tiempo debo esperar para obtener un nuevo token
            const retryAfter = parseInt(error.headers.get('retry-after'), 10);
            if (retryAfter < MAX_RETRY_AFTER) {
                // El tiempo de reintento es inferior al número máximo de segundos
                console.warn(\`Se alcanzó el límite de frecuencia. Reintentando en \${retryAfter} segundos...\`);
                await sleep(retryAfter);
                return getAccessToken(false); // Reintentar solo una vez
            }
        }

        console.error('Error al obtener el token de acceso:', error.response?.data || error.message);
    }
}

getAccessToken();`;

<div id="consumption-warning-logs">
  ## Registros de advertencia de consumo
</div>

Auth0 genera eventos de log `token_quota_consumption_warning` cuando el consumo de una cuota alcanza el 60 %, el 80 % y el 100 %. Puedes analizar estos logs para supervisar los patrones de uso de tokens, identificar posibles incidencias antes de que esos límites se apliquen de forma estricta y decidir valores de cuota adecuados.

El siguiente ejemplo de código muestra un registro de advertencia de consumo como una entrada de registro en JSON:

```json lines expandable
{
  "date": "2025-05-08T08:39:10.838Z",
  "type": "token_quota_consumption_warning",
  "description": "60% of client per_day quota consumed",
  "connection_id": "",
  "client_id": "QAxE5Z8LrvmQ2jxlzzEACeo39hO3xjFV",
  "client_name": "My_M2M_App",
  "ip": "xxxx",
  "client_ip": "xxxxx",
  "user_agent": "Other 0.0.0 / Other 0.0.0",
  "details": {
    "bucket": "per_day",
    "entity_type": "client",
    "entity_id": "QAxE5Z8LrvmQ2jxlzzEACeo39hO3xjFV",
    "quota": 15,
    "quota_consumption_percentage": 60,
    "quota_consumption": 11
  },
  "hostname": "xxxxx",
  "user_id": "",
  "user_name": "",
  "$event_schema": {
    "version": "1.0.0"
  },
  "log_id": "90020250508083910862978000000000000001223372036854918178",

  "_id": "90020250508083910862978000000000000001223372036854918178",
  "isMobile": false,
  "id": "90020250508083910862978000000000000001223372036854918178"
}
```

El registro de advertencia de consumo de ejemplo contiene los siguientes campos:

* **type**: `token_quota_consumption_warning`. Identifica la advertencia de consumo de la cuota de tokens de máquina a máquina que está alcanzando un umbral.
* **description**: (String) Resumen legible para las personas con el formato `<percent>% <entity_type> <bucket>` de cuota consumida. Por ejemplo, &quot;Se ha consumido el 60% de la cuota diaria del cliente.&quot;
* **details**: (Object) Describe los detalles específicos de la cuota y su consumo:

  * `bucket`: (String) `per_day` o `per_hour`.
  * `entity_type`: (String) cliente u organización.
  * `entity_id`: (String) ID del cliente o de la organización.
  * `quota`: (Integer) Límite configurado para este bucket.
  * `quota_consumption_percentage`: (Integer) Porcentaje consumido (por ejemplo, 60, 80, 100).
  * `quota_consumption`: (Integer) Cantidad real de tokens contabilizados.

Otros campos, como `date`, `client_name`, `ip` y `log_id`, son campos de registro estándar de Auth0.

<div id="best-practices">
  ## Prácticas recomendadas
</div>

* **Prioriza el almacenamiento en caché de tokens**: Destaca el almacenamiento en caché adecuado de tokens de acceso en aplicaciones de máquina a máquina. Las cuotas son una medida de protección, no un sustituto de una gestión eficiente de tokens.
* **Niveles de cuota adecuados**: Establece cuotas en función de patrones de tráfico legítimos y una capacidad de ráfaga razonable.
* **Supervisa antes de aplicar**: Comienza con `enforce: false` para observar el uso mediante registros y encabezados, y luego define límites fundamentados antes de habilitar `enforce: true`.
* **Manejo del lado del cliente**: Las aplicaciones deben manejar los errores 429 correctamente, usando `Retry-After` para las estrategias de espera.
* **Verifica proactivamente los encabezados de cuota**: Supervisa el uso de tokens para identificar posibles problemas.
* **Revisa los registros regularmente**: Supervisa los registros `token_quota_consumption_warning` para ajustar las cuotas de forma proactiva.

<div id="limitations">
  ## Limitaciones
</div>

* **No son límites de llamadas a la API**: Las cuotas granulares de tokens M2M se aplican a la cantidad de tokens de acceso obtenidos mediante Client Credentials Flow, no a las llamadas realizadas a tus APIs con esos tokens.
* **Consistencia eventual:** El recuento de tokens tiene consistencia eventual; puede haber excesos menores y breves antes de que el límite se aplique en solicitudes posteriores.
* Las solicitudes que den como resultado una respuesta `429` cuentan para el límite de frecuencia de la Auth0 Authentication API de tu tenant. Para obtener más información, consulta la [Política de límite de tasa](/docs/es/troubleshoot/customer-support/operational-policies/rate-limit-policy).

<div id="troubleshoot">
  ## Solución de problemas
</div>

* **Tráfico legítimo bloqueado:**

  * Considera establecer `enforce: false`.
  * Revisa los registros del tenant de Auth0 para `token_quota_consumption_warning` y los eventos de intercambio fallidos (`feccft`).
  * Examina los encabezados `Auth0-...-Quota-Limit` en las respuestas.
  * Revisa los patrones de solicitud de tokens de la aplicación y el almacenamiento en caché.
  * Considera si es necesario ajustar el límite de cuota.
* **Interpretar errores 429**: Usa `X-RateLimit-*`, `Retry-After` y los encabezados `Auth0-...-Quota-Limit` específicos para identificar el límite consumido y el tiempo de restablecimiento.
