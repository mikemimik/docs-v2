title: Guía de migración de redirecciones de cierre de sesión

El **01 de diciembre de 2021**, el comportamiento de cierre de sesión cambiará para redirigir siempre a los usuarios al URI enviado a las API de cierre de sesión de Auth0, en lugar de usar el parámetro de consulta `returnTo` enviado por <Tooltip tip="proveedor de identidad (IdP): Servicio que almacena y administra identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=Identity+Providers">proveedores de identidad</Tooltip> a `/login/callback` durante la ejecución del cierre de sesión. El URI utilizado dependerá de la API a la que se llame para solicitar el cierre de la sesión:

* `https://{yourDomain}/logout`: Auth0 usará el parámetro `returnTo`
* `https://{yourDomain}/v2/logout?federated`: Auth0 usará el parámetro `returnTo`
* `https://{yourDomain}/wsfed/{yourClientId}?wa=wsignout1.0`: el parámetro `wreply`

Si Auth0 no tiene constancia de una llamada previa a una de estas API, el cierre de sesión se completará, pero no se realizará la redirección y se mostrará una página de error a los usuarios finales.

<div id="affected-endpoints">
  ### Endpoints afectados
</div>

Los siguientes endpoints se ven afectados por este cambio:

* `https://{yourDomain}/v2/logout` al usar el parámetro de consulta `federated` para [cerrar la sesión de los usuarios en proveedores de identidad](/docs/es/authenticate/login/logout/log-users-out-of-idps)
* `https://{yourDomain}/wsfed/{yourClientId}?wa=wsignout1.0` al usar el parámetro de consulta `wreply` para [cerrar sesión en Auth0 mediante el protocolo WS-Fed](/docs/es/authenticate/protocols/ws-fed-protocol)
* `https://{yourDomain}/login/callback` al usar el parámetro de consulta `returnTo`.

Los tenants que dependan de los siguientes patrones de uso no podrán redirigir a la URI de redirección deseada después del **01 de diciembre de 2021**.

1. Aplicaciones que llaman a `/login/callback?returnTo` sin una llamada previa a `/v2/logout or /wsfed/{yourClient}?wa=wsignout1.0` para especificar la URL de cierre de sesión.
2. Proveedores de identidad que llaman a `/login/callback?returnTo` con un valor en `returnTo` distinto del valor proporcionado originalmente en una llamada previa a `/v2/logout` o `/wsfed/{yourClientId}`.
3. Aplicaciones que realizan llamadas simultáneas a una de las API afectadas mientras proporcionan diferentes URI de redirección en esas llamadas.
4. Aplicaciones o usuarios que realizan llamadas a la API `/authorize` de forma simultánea o intercalada con llamadas a una de las API afectadas, independientemente de si hubo una llamada previa a `/v2/logout` o `/wsfed/{yourClientId}?wa=wsignout1.0`.

<div id="actions">
  ### Actions
</div>

1. Ve a [Auth0 Dashboard &gt; Monitoreo &gt; Logs](https://manage.auth0.com/#/logs) y busca `type:depnote AND description:*unvalidated*redirects*` para encontrar las aplicaciones que dependen del comportamiento obsoleto.
2. Para cada aplicación afectada, asegúrate de que:

   1. la aplicación o la Relying Party inicie el cierre de sesión mediante una de las API públicas de Auth0: [Authentication API: Logout](https://auth0.com/docs/api/authentication#logout) o [Authentication API: WS-Federation](https://auth0.com/docs/api/authentication#ws-federation).
   2. la aplicación no dependa de que los proveedores de identidad modifiquen la URL de cierre de sesión que se devuelve a `/login/callback?returnTo`, ya que esas modificaciones dejarán de tenerse en cuenta.
   3. la aplicación no haga llamadas simultáneas a las API de cierre de sesión con distintas URL de cierre de sesión. Auth0 almacenará solo una URL de cierre de sesión por agente de usuario en un momento dado, lo que significa que los procesos de cierre de sesión simultáneos no podrán redirigir.
   4. la aplicación no haga llamadas a `/authorize` de forma simultánea ni intercaladas con llamadas a las API de cierre de sesión. Al completar una transacción de inicio de sesión, se borrará la URL de cierre de sesión almacenada para un agente de usuario, lo que significa que los procesos de cierre de sesión simultáneos no podrán redirigir.
   5. la aplicación y los usuarios no intercalen llamadas a `/authorize` con llamadas a una de las API afectadas.

Una vez completadas y probadas las acciones de migración, debes verificar la migración.

<div id="verify-migration">
  ### Verificar la migración
</div>

Una vez que hayas migrado tus aplicaciones y estés seguro de que ya no dependen de los patrones de uso obsoletos, comprueba los cambios desactivando el comportamiento obsoleto en el momento que elijas y antes del **01 de diciembre de 2021**.

1. Ve a [Auth0 Dashboard &gt; Configuración del tenant &gt; Advanced](https://manage.auth0.com/#/tenant/advanced) y busca la sección **Migración**.
2. Desactiva el interruptor **Unvalidated redirects from /login/callback**. Al desactivar este interruptor, se deshabilita el comportamiento obsoleto para tu tenant y se evita que siga utilizándose.

Si la redirección a la URL de cierre de sesión no funciona como se espera después de desactivar este interruptor, es señal de que tu aplicación todavía depende del comportamiento obsoleto.

Una vez que las migraciones se hayan realizado correctamente y se hayan confirmado en entornos de producción, podrás desactivar el interruptor de forma permanente para asegurarte de que las funcionalidades obsoletas ya no puedan utilizarse. El **01 de diciembre de 2021**, Auth0 eliminará por completo el comportamiento obsoleto junto con el interruptor asociado.
