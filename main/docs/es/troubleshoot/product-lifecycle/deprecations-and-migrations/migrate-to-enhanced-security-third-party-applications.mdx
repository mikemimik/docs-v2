A partir del 23 de octubre de 2026, Auth0 cambiará el modo de seguridad predeterminado para las aplicaciones de terceros que se creen a través de la Management API. De forma predeterminada, las aplicaciones de terceros usarán controles de seguridad reforzados alineados con las [prácticas recomendadas de OAuth 2.1](https://datatracker.ietf.org/doc/html/draft-ietf-oauth-v2-1).

<Warning>
  Este cambio solo afecta a los tenants que ya usaban aplicaciones de terceros antes del 23 de abril de 2026, y únicamente repercute en las aplicaciones nuevas. Sus aplicaciones de terceros existentes seguirán funcionando como hasta ahora, sin necesidad de realizar cambios.
</Warning>

Auth0 recomienda encarecidamente adoptar controles de seguridad reforzados para todas las nuevas aplicaciones de terceros. Los controles reforzados proporcionan autorización explícita para la API, uso obligatorio de PKCE y un conjunto de funcionalidades más acotado, alineado con OAuth 2.1 y las prácticas recomendadas de seguridad. Las aplicaciones con controles reforzados también se beneficiarán de futuras capacidades, como límites de frecuencia a nivel de aplicación y herramientas de gestión mejoradas. Sin embargo, puede conservar el comportamiento existente si lo necesita para casos de uso concretos.

Para obtener más información sobre las aplicaciones de terceros, consulte [Aplicaciones de terceros](/docs/es/get-started/applications/third-party-applications). Para ver una comparación detallada de lo que ofrece cada modo, consulte [Comparación de funcionalidades](#feature-comparison). Para obtener orientación sobre patrones de integración específicos, consulte [Escenarios comunes](#common-scenarios).

<div id="how-are-you-affected">
  ## ¿Cómo te afecta?
</div>

Esta migración te afecta de dos maneras:

<div id="1-management-api-default-the-deprecation">
  ### 1. Valor predeterminado de Management API (obsolescencia)
</div>

Al crear aplicaciones de terceros mediante `POST /api/v2/clients`, el valor predeterminado de `third_party_security_mode` cambiará de `permissive` (comportamiento existente) a `strict` (controles de seguridad reforzados) el 23 de octubre de 2026.

Todas las aplicaciones de terceros creadas a través del Auth0 Dashboard ya usan controles de seguridad reforzados. Esto no se puede configurar desde el Auth0 Dashboard.

<div id="2-dynamic-client-registration-independent-configuration">
  ### 2. Registro dinámico de clientes (configuración independiente)
</div>

Si utiliza [registro dinámico de clientes](/docs/es/get-started/applications/dynamic-client-registration), los clientes de DCR se rigen por una configuración independiente del tenant: `dynamic_client_registration_security_mode`. Esto es independiente de la obsolescencia y requiere una decisión de configuración propia.

<div id="migration-tasks">
  ## Tareas de migración
</div>

<div id="review-your-third-party-applications">
  ### Revisa tus aplicaciones de terceros
</div>

Antes de elegir una ruta de migración, revisa cómo funcionan tus aplicaciones de terceros para entender sus requisitos actuales. Ten en cuenta lo siguiente:

* ¿Qué tipos de concesión usan? (`authorization code`, `implicit`, `client credentials`, etc.)
* ¿Requieren scopes de OIDC (`openid`, `profile`, `email`) o tokens de ID?
* ¿Usan Classic Login o endpoints heredados?
* ¿Cómo se crean? (manualmente mediante Dashboard/API o dinámicamente mediante DCR)

Si tienes pocas aplicaciones de terceros, puedes revisarlas individualmente con Auth0 Dashboard o Management API. Cada aplicación incluye una propiedad `third_party_security_mode` que indica su modo de seguridad actual (`strict` o `permissive`).

**Refuerza las aplicaciones permisivas existentes**: Considera revisar las [políticas de acceso a API](/docs/es/get-started/apis/api-access-policies-for-applications) de tus API y configurarlas como **Requerir autorización de cliente** cuando corresponda. Esto garantiza que las aplicaciones de terceros permisivas deban contar con una concesión explícita para acceder a esas API. Ten en cuenta que esta política también se aplica a las aplicaciones propias, así que revisa tus integraciones existentes antes de cambiarla.

Para entender qué significan los controles de seguridad reforzados para tus integraciones, revisa la [Comparación de funcionalidades](#feature-comparison) a continuación y comprueba si tu caso coincide con alguno de los [Escenarios comunes](#common-scenarios). La sección de [FAQ](#frequently-asked-questions) también responde a preguntas frecuentes sobre la migración.

<div id="step-1-choose-how-to-create-new-third-party-applications">
  ### Paso 1: Elige cómo crear nuevas aplicaciones de terceros
</div>

Decide si las nuevas aplicaciones de terceros creadas mediante la Management API deben usar controles de seguridad reforzados o mantener el comportamiento existente. Esta opción se aplica solo a `POST /api/v2/clients`. Las aplicaciones creadas desde el Auth0 Dashboard siempre usan controles de seguridad reforzados.

<div id="option-a-complete-the-migration-recommended">
  #### Opción A: Completa la migración (Recomendada)
</div>

Completa la migración antes del 23 de octubre de 2026 para que los controles de seguridad reforzados sean la opción predeterminada. Este enfoque alinea tus aplicaciones de terceros con las prácticas recomendadas de OAuth 2.1 y las prepara para capacidades futuras.

<div id="1-test-enhanced-security-controls">
  ##### 1. Prueba los controles de seguridad reforzados
</div>

Crea una aplicación externa de prueba con controles de seguridad reforzados para validar la compatibilidad:

<AuthCodeGroup>
  ```bash cURL
  curl --request POST \
    --url 'https://{yourDomain}/api/v2/clients' \
    --header 'Authorization: Bearer {YOUR_MANAGEMENT_API_TOKEN}' \
    --header 'Content-Type: application/json' \
    --data '{
      "name": "Test Third-Party App",
      "is_first_party": false,
      "third_party_security_mode": "strict",
      "app_type": "regular_web",
      "callbacks": ["https://partner.example.com/callback"],
      "grant_types": ["authorization_code", "refresh_token"]
    }'
  ```
</AuthCodeGroup>

La respuesta incluirá un `client_id` con el prefijo `tpc_` y `third_party_security_mode: "strict"`.

<div id="2-set-up-default-api-permissions">
  ##### 2. Configura los permisos predeterminados de API
</div>

Las aplicaciones de terceros con controles de seguridad reforzados requieren autorizaciones de cliente explícitas para acceder a las API. Los permisos predeterminados definen un conjunto base de API y scopes al que todas las aplicaciones de terceros pueden acceder automáticamente. Esto es importante para los clientes creados dinámicamente, ya que no puedes configurar permisos para cada aplicación de forma individual.

También puedes definir permisos específicos para aplicaciones individuales (por `client_id`) para conceder un acceso más amplio o más restringido que el predeterminado. Cuando existen ambos, los permisos por aplicación tienen prioridad sobre los predeterminados.

<Tabs>
  <Tab title="Auth0 Dashboard">
    1. Ve a **Applications &gt; APIs**.
    2. Selecciona la API a la que quieres que accedan las aplicaciones de terceros.
    3. En la pestaña **Settings**, desplázate hasta **Default Permissions for Third Party Apps**.
    4. Selecciona **Authorized** para User Access y/o acceso de cliente.
    5. Selecciona los scopes que quieres conceder.
    6. Haz clic en **Save**.
  </Tab>

  <Tab title="Management API">
    <AuthCodeGroup>
      ```bash cURL
      curl --request POST \
        --url 'https://{yourDomain}/api/v2/client-grants' \
        --header 'Authorization: Bearer {YOUR_MANAGEMENT_API_TOKEN}' \
        --header 'Content-Type: application/json' \
        --data '{
          "default_for": "third_party_clients",
          "audience": "https://api.example.com",
          "scope": ["read:items", "write:items"],
          "subject_type": "user"
        }'
      ```
    </AuthCodeGroup>
  </Tab>
</Tabs>

Puedes configurar permisos predeterminados independientes para las aplicaciones de terceros para el acceso de usuario (`subject_type: "user"`) y el acceso de máquina a máquina (`subject_type: "client"`).

Para obtener más información, consulta [Permisos predeterminados para aplicaciones de terceros](/docs/es/get-started/applications/application-access-to-apis-client-grants#default-permissions-for-third-party-applications).

<div id="3-validate-compatibility">
  ##### 3. Validar la compatibilidad
</div>

Prueba tus flujos de trabajo para crear aplicaciones de terceros con los controles de seguridad reforzados habilitados. Confirma que:

* Tus aplicaciones pueden usar los tipos de concesión `authorization_code`, `refresh_token` y `client_credentials`
* PKCE está implementado en tus flujos de autorización
* No necesitas scopes de OIDC
* No necesitas Classic Login ni endpoints heredados
* Tu tenant no tiene [Rules](/docs/es/customize/rules) activas que deban ejecutarse en los flujos de login de terceros. Rules no es compatible con aplicaciones de terceros strict y dará lugar a un error. Si usas Rules, considera [migrar a Actions](/docs/es/customize/actions/migrate/migrate-from-rules-to-actions) o usar el modo permisivo.

Si identificas problemas de compatibilidad, consulta [Solución de problemas de aplicaciones de terceros](/docs/es/get-started/applications/third-party-applications/troubleshooting).

<div id="4-complete-the-migration">
  ##### 4. Completa la migración
</div>

Una vez que hayas validado la compatibilidad, completa la migración desactivando la opción **Create Permissive Third-Party Clients by Default**:

<Frame>![Create Permissive Third-Party Clients Bt Default](/docs/images/third-party-applications/create_permissive_3p_clients_by_default.png)</Frame>

En el Auth0 Dashboard:

1. Ve a **Settings &gt; Advanced**.
2. Desplázate hasta la sección **Migración**.
3. Desactiva **Create Permissive Third-Party Clients by Default**.
4. Selecciona **Save**.

Después de completar la migración, cuando crees aplicaciones de terceros mediante `POST /api/v2/clients`, puedes:

* Omitir el parámetro `third_party_security_mode` (los controles mejorados se aplican de forma predeterminada), o
* Establecer explícitamente `third_party_security_mode: "strict"`

**Después del 23 de octubre de 2026**: La opción **Create Permissive Third-Party Clients by Default** se desactivará automáticamente para todos los tenants aptos. Para crear aplicaciones con el comportamiento existente, debes especificar explícitamente `third_party_security_mode: "permissive"` en la solicitud `POST` al endpoint `/api/v2/clients`.

<div id="option-b-preserve-existing-behavior-as-the-default">
  #### Opción B: Mantener el comportamiento existente como predeterminado
</div>

Si necesita seguir creando aplicaciones de terceros con el comportamiento existente como predeterminado, puede mantener activado el interruptor **Create Permissive Third-Party Clients by Default** hasta que esté listo para adoptar controles de seguridad reforzados.

<Warning>
  Esta opción conserva sus flujos de trabajo actuales, pero no ofrece las ventajas de seguridad de los controles reforzados. Auth0 recomienda encarecidamente adoptar controles de seguridad reforzados para todas las nuevas aplicaciones de terceros.
</Warning>

<div id="before-the-deadline">
  ##### Antes de la fecha límite
</div>

La opción **Create Permissive Third-Party Clients by Default** sigue habilitada (no es necesario hacer nada con la opción en sí). Sin embargo, debes preparar tus flujos de trabajo para especificar explícitamente el modo de seguridad después de la fecha límite.

Actualiza el código de creación de tu aplicación para pasar explícitamente `third_party_security_mode: "permissive"`:

<AuthCodeGroup>
  ```bash cURL
  curl --request POST \
    --url 'https://{yourDomain}/api/v2/clients' \
    --header 'Authorization: Bearer {YOUR_MANAGEMENT_API_TOKEN}' \
    --header 'Content-Type: application/json' \
    --data '{
      "name": "Partner Integration",
      "is_first_party": false,
      "third_party_security_mode": "permissive",
      "app_type": "regular_web",
      "callbacks": ["https://partner.example.com/callback"]
    }'
  ```
</AuthCodeGroup>

Prueba este enfoque antes de la fecha límite para asegurarte de que tus flujos de trabajo procesen correctamente el parámetro explícito.

<div id="after-the-deadline">
  ##### Después de la fecha límite
</div>

El 23 de octubre de 2026, la opción **Create Permissive Third-Party Clients by Default** se desactivará automáticamente. Para seguir creando aplicaciones con el comportamiento existente, deberás establecer explícitamente `third_party_security_mode: "permissive"` en cada solicitud `POST /api/v2/clients`.

Si omites el parámetro `third_party_security_mode`, se aplicarán controles de seguridad reforzados de forma predeterminada.

<div id="step-2-choose-how-to-handle-dynamic-client-registration">
  ### Paso 2: Elige cómo gestionar el registro dinámico de clientes
</div>

Si utilizas el [registro dinámico de clientes](/docs/es/get-started/applications/dynamic-client-registration), configura por separado el modo de seguridad para los clientes de DCR. Esto es independiente del [cambio en el valor predeterminado de Management API](/docs/es/troubleshoot/product-lifecycle/deprecations-and-migrations/migrate-to-enhanced-security-third-party-applications#1-management-api-default-the-deprecation) y puedes configurarlo en cualquier momento.

<Warning>
  Antes de habilitar los controles de seguridad reforzados para DCR, asegúrate de haber configurado los [permisos predeterminados de API](/docs/es/get-started/applications/application-access-to-apis-client-grants#default-permissions-for-third-party-applications) para aplicaciones de terceros. Sin permisos predeterminados, los clientes de DCR no podrán acceder a ninguna API.
</Warning>

<div id="review-current-dcr-behavior">
  #### Revisa el comportamiento actual de DCR
</div>

Comprueba la configuración actual del modo de seguridad de DCR:

<Tabs>
  <Tab title="Auth0 Dashboard">
    1. Ve a **Settings &gt; Advanced**. En **Dynamic Client Registration (DCR) Security Mode**, verifica el valor actual.

    <Frame>![Configuración avanzada del tenant en el Dashboard con el menú desplegable de DCR Security Mode](/docs/images/third-party-applications/dcr-security-mode.png)</Frame>
  </Tab>

  <Tab title="Management API">
    Haz una solicitud `GET` al endpoint `/api/v2/tenants/settings` para obtener la configuración de tu tenant:

    <AuthCodeGroup>
      ```bash cURL
      curl --request GET \
        --url 'https://{yourDomain}/api/v2/tenants/settings' \
        --header 'Authorization: Bearer {YOUR_MANAGEMENT_API_TOKEN}'
      ```
    </AuthCodeGroup>

    Busca la propiedad `dynamic_client_registration_security_mode` en la respuesta. Si no aparece, los clientes de DCR usan actualmente el comportamiento existente de forma predeterminada.
  </Tab>
</Tabs>

<div id="configure-dcr-security-mode">
  #### Configurar el modo de seguridad de DCR
</div>

Elige el modo de seguridad que quieres para los clientes registrados de forma dinámica:

**Opción A: Controles de seguridad reforzados para los clientes de DCR** (recomendado)

<Note>
  Antes de habilitar el modo `strict` para DCR, configura los [permisos predeterminados de API](/docs/es/get-started/applications/application-access-to-apis-client-grants#default-permissions-for-third-party-applications) para aplicaciones de terceros. Sin permisos predeterminados, los clientes de DCR no podrán acceder a ninguna API.
</Note>

Establece `dynamic_client_registration_security_mode` en `strict`:

<AuthCodeGroup>
  ```bash cURL
  curl --request PATCH \
    --url 'https://{yourDomain}/api/v2/tenants/settings' \
    --header 'Authorization: Bearer {YOUR_MANAGEMENT_API_TOKEN}' \
    --header 'Content-Type: application/json' \
    --data '{
      "dynamic_client_registration_security_mode": "strict"
    }'
  ```
</AuthCodeGroup>

**Opción B: Conservar el comportamiento existente para los clientes de DCR**

Mantén `dynamic_client_registration_security_mode` en `permissive` o establécelo en ese valor:

<AuthCodeGroup>
  ```bash cURL
  curl --request PATCH \
    --url 'https://{yourDomain}/api/v2/tenants/settings' \
    --header 'Authorization: Bearer {YOUR_MANAGEMENT_API_TOKEN}' \
    --header 'Content-Type: application/json' \
    --data '{
      "dynamic_client_registration_security_mode": "permissive"
    }'
  ```
</AuthCodeGroup>

Para obtener más información, consulta [Registro dinámico de clientes](/docs/es/get-started/applications/dynamic-client-registration).

  ## Comparación de funcionalidades
</div>

La siguiente tabla compara las capacidades disponibles en cada modo de seguridad:

| Capacidad                       | Controles de seguridad reforzados                                                                                                             | Comportamiento existente                 |
| ------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------- |
| **Tipos de concesión**          | `authorization_code`, `refresh_token`, `client_credentials`                                                                                   | Todos los tipos de concesión disponibles |
| **PKCE**                        | Obligatorio                                                                                                                                   | Opcional                                 |
| **OIDC**                        | No disponible. Previsto para una futura versión.                                                                                              | Compatible                               |
| **Autorización de API**         | Siempre requiere una autorización de cliente explícita                                                                                        | Sigue la política de acceso de la API    |
| **Classic Login**               | No compatible                                                                                                                                 | Compatible                               |
| **Endpoints heredados**         | No disponible                                                                                                                                 | Disponible                               |
| **Formato de ID de cliente**    | Prefijo `tpc_`                                                                                                                                | Formato estándar                         |
| **Propiedades configurables**   | [Conjunto limitado de propiedades](/docs/es/get-started/applications/third-party-applications/security-controls#restricted-client-configuration) | Todas las propiedades                    |

| **Creación desde el Dashboard** | Siempre usa controles de seguridad reforzados                                                                                                 | No disponible desde el Dashboard         |

<div id="common-scenarios">
  ## Escenarios habituales
</div>

<div id="scenario-1-partner-integrations-using-modern-oauth">
  ### Escenario 1: Integraciones de partners que usan OAuth moderno
</div>

**Situación**: Tiene integraciones de partners que usan el flujo de código de autorización con PKCE y acceden a sus APIs.

**Recomendación**: Adopte controles de seguridad reforzados (Opción A). Esto es totalmente compatible con las implementaciones modernas de OAuth y ofrece ventajas de seguridad adicionales.

**Pasos**:

1. Configure los permisos predeterminados de API para sus APIs
2. Pruebe a crear una aplicación de partner con `third_party_security_mode: "strict"`
3. Complete la migración desactivando el selector de migración

<div id="scenario-2-applications-requiring-oidc">
  ### Escenario 2: Aplicaciones que requieren OIDC
</div>

**Situación**: Sus aplicaciones de terceros requieren scopes de OIDC (openid, profile, email) o tokens de ID.

**Recomendación**: El soporte de OIDC para aplicaciones de terceros está previsto para una futura versión. Hasta entonces, preserve el comportamiento existente (Opción B) o migre a tokens de acceso con scope de API.

**Pasos**:

* Si necesita usar OIDC, mantenga activado el toggle de migración y envíe explícitamente `third_party_security_mode: "permissive"` al crear aplicaciones
* Como alternativa, actualice sus integraciones para usar scopes de API en lugar de scopes de OIDC

<div id="scenario-3-dynamic-client-registration-mcp-ai-agents">
  ### Escenario 3: registro dinámico de clientes (MCP, agentes de IA)
</div>

**Situación**: Usas DCR para agentes de IA, servidores MCP o aplicaciones del portal de desarrolladores.

**Recomendación**: Configura `dynamic_client_registration_security_mode: "strict"` y define permisos predeterminados de la API. Los clientes MCP (Claude Code, VS Code) son compatibles con los controles de seguridad reforzados.

**Pasos**:

1. Configura los permisos predeterminados de la API
2. Establece `dynamic_client_registration_security_mode: "strict"` mediante la Management API
3. Haz una prueba con un registro de DCR
4. Verifica que los clientes DCR puedan obtener tokens de acceso

<div id="scenario-4-applications-using-classic-login">
  ### Escenario 4: Aplicaciones que usan Classic Login
</div>

**Situación**: Tus aplicaciones de terceros usan Classic Login en lugar de Universal Login.

**Recomendación**: Classic Login no es compatible con aplicaciones de terceros con controles de seguridad reforzados. Migra a Universal Login o conserva el comportamiento existente.

**Pasos**:

* Recomendado: Migra a Universal Login antes de adoptar los controles de seguridad reforzados
* Alternativa: Mantén habilitado el selector de migración y usa explícitamente `third_party_security_mode: "permissive"`

<div id="frequently-asked-questions">
  ## Preguntas frecuentes
</div>

<div id="can-i-change-the-security-mode-on-an-existing-application">
  ### ¿Puedo cambiar el modo de seguridad de una aplicación existente?
</div>

No. `third_party_security_mode` se establece al crear la aplicación y no puede cambiarse posteriormente. Para usar un modo de seguridad diferente, cree una aplicación nueva.

<div id="what-happens-to-my-existing-third-party-applications">
  ### ¿Qué sucede con mis aplicaciones de terceros existentes?
</div>

Nada. Las aplicaciones de terceros existentes siguen funcionando exactamente igual que hoy. Esta migración solo afecta a la configuración predeterminada de las aplicaciones recién creadas.

<div id="can-i-use-both-security-modes-in-the-same-tenant">
  ### ¿Puedo usar ambos modos de seguridad en el mismo tenant?
</div>

Sí. Puedes tener algunas aplicaciones de terceros con controles de seguridad reforzados y otras con el comportamiento anterior. El modo de seguridad se configura por aplicación.

<div id="what-about-dynamic-client-registration">
  ### ¿Qué ocurre con el registro dinámico de clientes?
</div>

DCR se controla mediante una configuración independiente del tenant, `dynamic_client_registration_security_mode`. Esto es independiente de la obsolescencia y requiere una decisión de configuración propia. Para obtener más información, consulta [Registro dinámico de clientes](/docs/es/get-started/applications/dynamic-client-registration).

<div id="can-i-create-applications-with-existing-behavior-after-the-deadline">
  ### ¿Puedo crear aplicaciones con el comportamiento anterior después de la fecha límite?
</div>

Sí, pero solo mediante la Management API. Establece explícitamente `third_party_security_mode: "permissive"` al crear cada aplicación. El Auth0 Dashboard no permite crear aplicaciones con el comportamiento anterior.

<div id="will-future-features-work-with-existing-behavior">
  ### ¿Serán compatibles las futuras funciones con el comportamiento existente?
</div>

Algunas funciones futuras (como los límites de frecuencia por aplicación) solo estarán disponibles para las aplicaciones con controles de seguridad reforzados.
