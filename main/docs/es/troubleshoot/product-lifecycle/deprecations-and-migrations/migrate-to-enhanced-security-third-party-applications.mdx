A partir del 23 de octubre de 2026, Auth0 cambiará el modo de seguridad predeterminado de las aplicaciones de terceros creadas mediante la Management API. De forma predeterminada, las aplicaciones de terceros usarán controles de seguridad reforzados alineados con las [prácticas recomendadas de OAuth 2.1](https://datatracker.ietf.org/doc/html/draft-ietf-oauth-v2-1).

<Warning>
  Este cambio solo afecta a los tenants que ya usaban aplicaciones de terceros antes del 23 de abril de 2026, y solo afecta a las aplicaciones recién creadas. Tus aplicaciones de terceros existentes seguirán funcionando como hasta ahora, sin que tengas que hacer ningún cambio.
</Warning>

Auth0 recomienda encarecidamente adoptar controles de seguridad reforzados para todas las nuevas aplicaciones de terceros. Los controles reforzados ofrecen autorización explícita de API, uso obligatorio de PKCE y un conjunto de funcionalidades más acotado, alineado con OAuth 2.1 y las prácticas recomendadas de seguridad. Las aplicaciones con controles reforzados también se beneficiarán de futuras capacidades, como los límites de frecuencia a nivel de aplicación y herramientas de gestión mejoradas. Sin embargo, si lo necesitas para casos de uso específicos, puedes mantener el comportamiento existente.

Para obtener más información sobre las aplicaciones de terceros, consulta [Third-Party Applications](/docs/es/get-started/applications/third-party-applications). Para ver una comparación detallada de lo que&#39;s disponible en cada modo, consulta [Comparación de funcionalidades](#feature-comparison). Para obtener orientación sobre patrones de integración específicos, consulta [Escenarios comunes](#common-scenarios).

<div id="how-are-you-affected">
  ## ¿Cómo te afecta?
</div>

Esta migración te afecta de dos formas:

<div id="1-management-api-default-the-deprecation">
  ### 1. Valor predeterminado de Management API (obsolescencia)
</div>

Cuando cree aplicaciones de terceros mediante `POST /api/v2/clients`, el valor predeterminado de `third_party_security_mode` cambiará de `permissive` (comportamiento existente) a `strict` (controles de seguridad reforzados) el 23 de octubre de 2026.

Todas las aplicaciones de terceros creadas en el Auth0 Dashboard ya usan controles de seguridad reforzados. Esto no se puede configurar en el Auth0 Dashboard.

<div id="2-dynamic-client-registration-independent-configuration">
  ### 2. Registro dinámico de clientes (configuración independiente)
</div>

Si usa [registro dinámico de clientes](/docs/es/get-started/applications/dynamic-client-registration), los clientes DCR se controlan mediante una configuración independiente del tenant: `dynamic_client_registration_security_mode`. Esto es independiente de la obsolescencia y requiere una decisión de configuración propia.

<div id="migration-tasks">
  ## Tareas de migración
</div>

<div id="review-your-third-party-applications">
  ### Revisa tus aplicaciones de terceros
</div>

Antes de elegir una opción de migración, revisa cómo funcionan tus aplicaciones de terceros para entender sus requisitos actuales. Ten en cuenta lo siguiente:

* ¿Qué tipos de concesión usan? (`authorization code`, `implicit`, `credenciales de cliente`, etc.)
* ¿Necesitan scopes de OIDC (`openid`, `profile`, `email`) o tokens de ID?
* ¿Usan Classic Login o endpoints heredados?
* ¿Cómo se crean? (manualmente a través del Dashboard o la API, o dinámicamente mediante DCR)

Si tienes pocas aplicaciones de terceros, puedes revisarlas individualmente en Auth0 Dashboard o Management API. Cada aplicación incluye una propiedad `third_party_security_mode` que indica su modo de seguridad actual (`strict` o `permissive`).

**Refuerza las aplicaciones permisivas existentes**: Considera revisar las [políticas de acceso a la API](/docs/es/get-started/apis/api-access-policies-for-applications) de tus API y configurarlas en **Requerir autorización de cliente** cuando corresponda. Esto garantiza que las aplicaciones de terceros permisivas deban contar con una concesión explícita para acceder a esas API. Ten en cuenta que esta política también se aplica a las aplicaciones propias, así que revisa tus integraciones existentes antes de cambiarla.

Para entender qué implican los controles de seguridad reforzados para tus integraciones, revisa la [Comparación de funcionalidades](#feature-comparison) a continuación y comprueba si tu caso coincide con alguno de los [Escenarios comunes](#common-scenarios). La sección de [Preguntas frecuentes](#frequently-asked-questions) también responde preguntas frecuentes sobre la migración.

<div id="step-1-choose-how-to-create-new-third-party-applications">
  ### Paso 1: Elegir cómo crear nuevas aplicaciones de terceros
</div>

Decide si las nuevas aplicaciones de terceros que se creen mediante la Management API deben usar controles de seguridad reforzados o conservar el comportamiento existente. Esta elección se aplica solo a `POST /api/v2/clients`. Las aplicaciones creadas con el Auth0 Dashboard siempre usan controles reforzados.

<div id="option-a-complete-the-migration-recommended">
  #### Opción A: Completar la migración (Recomendado)
</div>

Completa la migración antes del 23 de octubre de 2026 para que los controles de seguridad reforzados sean la configuración predeterminada. Este enfoque alinea tus aplicaciones de terceros con las prácticas recomendadas de OAuth 2.1 y las prepara para futuras funcionalidades.

<div id="1-test-enhanced-security-controls">
  ##### 1. Pruebe los controles de seguridad reforzados
</div>

Cree una aplicación de terceros de prueba con controles de seguridad reforzados para validar la compatibilidad:

<AuthCodeGroup>
  ```bash cURL
  curl --request POST \
    --url 'https://{yourDomain}/api/v2/clients' \
    --header 'Authorization: Bearer {YOUR_MANAGEMENT_API_TOKEN}' \
    --header 'Content-Type: application/json' \
    --data '{
      "name": "Test Third-Party App",
      "is_first_party": false,
      "third_party_security_mode": "strict",
      "app_type": "regular_web",
      "callbacks": ["https://partner.example.com/callback"],
      "grant_types": ["authorization_code", "refresh_token"]
    }'
  ```
</AuthCodeGroup>

La respuesta incluirá un `client_id` con el prefijo `tpc_` y `third_party_security_mode: "strict"`.

<div id="2-set-up-default-api-permissions">
  ##### 2. Configura los permisos predeterminados de la API
</div>

Las aplicaciones de terceros con controles de seguridad reforzados requieren autorizaciones de cliente explícitas para acceder a las API. Los permisos predeterminados definen un conjunto básico de API y scopes a los que todas las aplicaciones de terceros pueden acceder automáticamente. Esto es importante para los clientes creados de forma dinámica, ya que no puedes configurar permisos para cada aplicación individualmente.

También puedes definir permisos específicos para aplicaciones individuales (por `client_id`) para conceder un acceso más amplio o más restringido que el predeterminado. Cuando existen ambos, los permisos por aplicación tienen prioridad sobre los predeterminados.

<Tabs>
  <Tab title="Auth0 Dashboard">
    1. Ve a **Applications &gt; APIs**.
    2. Selecciona la API a la que quieres que accedan las aplicaciones de terceros.
    3. En la pestaña **Settings**, desplázate hasta **Default Permissions for Third Party Apps**.
    4. Selecciona **Authorized** para **User Access** y/o **Client Access**.
    5. Selecciona los scopes que quieres conceder.
    6. Haz clic en **Save**.
  </Tab>

  <Tab title="Management API">
    <AuthCodeGroup>
      ```bash cURL
      curl --request POST \
        --url 'https://{yourDomain}/api/v2/client-grants' \
        --header 'Authorization: Bearer {YOUR_MANAGEMENT_API_TOKEN}' \
        --header 'Content-Type: application/json' \
        --data '{
          "default_for": "third_party_clients",
          "audience": "https://api.example.com",
          "scope": ["read:items", "write:items"],
          "subject_type": "user"
        }'
      ```
    </AuthCodeGroup>
  </Tab>
</Tabs>

Puedes configurar permisos predeterminados independientes para las aplicaciones de terceros para el acceso de usuario (`subject_type: "user"`) y el acceso de máquina a máquina (`subject_type: "client"`).

Para obtener más información, consulta [Permisos predeterminados para aplicaciones de terceros](/docs/es/get-started/applications/application-access-to-apis-client-grants#default-permissions-for-third-party-applications).

<div id="3-validate-compatibility">
  ##### 3. Validar la compatibilidad
</div>

Prueba tus flujos de trabajo para crear aplicaciones de terceros con los controles de seguridad reforzados activados. Confirma que:

* Tus aplicaciones pueden usar los tipos de concesión `authorization_code`, `refresh_token` y `client_credentials`
* PKCE está implementado en tus flujos de autorización
* No necesitas scopes de OIDC
* No necesitas Classic Login ni endpoints heredados
* Tu tenant no tiene [Rules](/docs/es/customize/rules) activas que deban ejecutarse en los flujos de inicio de sesión de terceros. Rules no son compatibles con las aplicaciones de terceros en modo `strict` y provocarán un error. Si usas Rules, considera [migrar a Actions](/docs/es/customize/actions/migrate/migrate-from-rules-to-actions) o usar el modo permisivo.

Si identificas problemas de compatibilidad, consulta la [Solución de problemas de aplicaciones de terceros](/docs/es/get-started/applications/third-party-applications/troubleshooting).

<div id="4-complete-the-migration">
  ##### 4. Completa la migración
</div>

Una vez que hayas validado la compatibilidad, completa la migración desactivando el interruptor **Create Permissive Third-Party Clients by Default**:

<Frame>![Create Permissive Third-Party Clients Bt Default](/docs/images/third-party-applications/create_permissive_3p_clients_by_default.png)</Frame>

En el Auth0 Dashboard:

1. Ve a **Settings &gt; Advanced**.
2. Desplázate hasta la sección **Migración**.
3. Desactiva **Create Permissive Third-Party Clients by Default**.
4. Selecciona **Save**.

Después de completar la migración, cuando crees aplicaciones de terceros mediante `POST /api/v2/clients`, puedes:

* Omitir el parámetro `third_party_security_mode` (los controles reforzados se aplican de forma predeterminada), o
* Establecer explícitamente `third_party_security_mode: "strict"`

**Después del 23 de octubre de 2026**: El interruptor **Create Permissive Third-Party Clients by Default** se desactivará automáticamente para todos los tenants aptos. Para crear aplicaciones con el comportamiento existente, debes especificar explícitamente `third_party_security_mode: "permissive"` en la solicitud `POST` al endpoint `/api/v2/clients`.

<div id="option-b-preserve-existing-behavior-as-the-default">
  #### Opción B: Mantener el comportamiento existente como opción predeterminada
</div>

Si necesitas seguir creando aplicaciones de terceros con el comportamiento existente como opción predeterminada, puedes mantener activado el toggle **Create Permissive Third-Party Clients by Default** hasta que estés listo para adoptar controles de seguridad reforzados.

<Warning>
  Esta opción conserva tus flujos de trabajo actuales, pero no ofrece las ventajas de seguridad de los controles reforzados. Auth0 recomienda encarecidamente adoptar controles de seguridad reforzados para todas las nuevas aplicaciones de terceros.
</Warning>

<div id="before-the-deadline">
  ##### Antes de la fecha límite
</div>

La opción **Create Permissive Third-Party Clients by Default** permanece habilitada (no es necesario realizar ninguna acción en la propia opción). Sin embargo, debe preparar sus flujos de trabajo para especificar explícitamente el modo de seguridad después de la fecha límite.

Actualice el código de creación de su aplicación para incluir explícitamente `third_party_security_mode: "permissive"`:

<AuthCodeGroup>
  ```bash cURL
  curl --request POST \
    --url 'https://{yourDomain}/api/v2/clients' \
    --header 'Authorization: Bearer {YOUR_MANAGEMENT_API_TOKEN}' \
    --header 'Content-Type: application/json' \
    --data '{
      "name": "Partner Integration",
      "is_first_party": false,
      "third_party_security_mode": "permissive",
      "app_type": "regular_web",
      "callbacks": ["https://partner.example.com/callback"]
    }'
  ```
</AuthCodeGroup>

Pruebe este enfoque antes de la fecha límite para asegurarse de que sus flujos de trabajo procesen correctamente el parámetro explícito.

<div id="after-the-deadline">
  ##### Después de la fecha límite
</div>

El 23 de octubre de 2026, el toggle **Create Permissive Third-Party Clients by Default** se desactivará automáticamente. Para seguir creando aplicaciones con el comportamiento actual, debes establecer explícitamente `third_party_security_mode: "permissive"` en cada solicitud a `POST /api/v2/clients`.

Si omites el parámetro `third_party_security_mode`, se aplicarán controles de seguridad reforzados de forma predeterminada.

<div id="step-2-choose-how-to-handle-dynamic-client-registration">
  ### Paso 2: Elige cómo gestionar el registro dinámico de clientes
</div>

Si usas [registro dinámico de clientes](/docs/es/get-started/applications/dynamic-client-registration), configura por separado el modo de seguridad para los clientes DCR. Esto es independiente del [cambio en el valor predeterminado de Management API](/docs/es/troubleshoot/product-lifecycle/deprecations-and-migrations/migrate-to-enhanced-security-third-party-applications#1-management-api-default-the-deprecation) y puedes configurarlo en cualquier momento.

<Warning>
  Antes de habilitar los controles de seguridad reforzados para DCR, asegúrate de haber configurado los [permisos predeterminados de la API](/docs/es/get-started/applications/application-access-to-apis-client-grants#default-permissions-for-third-party-applications) para aplicaciones de terceros. Sin permisos predeterminados, los clientes DCR no podrán acceder a ninguna API.
</Warning>

<div id="review-current-dcr-behavior">
  #### Revisa el comportamiento actual de DCR
</div>

Comprueba la configuración actual del modo de seguridad de DCR:

<Tabs>
  <Tab title="Auth0 Dashboard">
    1. Ve a **Settings &gt; Advanced**. En **Dynamic Client Registration (DCR) Security Mode**, comprueba el valor actual.

    <Frame>![Configuración avanzada del tenant en el Dashboard con el menú desplegable de DCR Security Mode](/docs/images/third-party-applications/dcr-security-mode.png)</Frame>
  </Tab>

  <Tab title="Management API">
    Haz una solicitud `GET` al endpoint `/api/v2/tenants/settings` para obtener la configuración de tu tenant:

    <AuthCodeGroup>
      ```bash cURL
      curl --request GET \
        --url 'https://{yourDomain}/api/v2/tenants/settings' \
        --header 'Authorization: Bearer {YOUR_MANAGEMENT_API_TOKEN}'
      ```
    </AuthCodeGroup>

    Busca la propiedad `dynamic_client_registration_security_mode` en la respuesta. Si no está presente, los clientes DCR usan actualmente el comportamiento existente de forma predeterminada.
  </Tab>
</Tabs>

<div id="configure-dcr-security-mode">
  #### Configurar el modo de seguridad de DCR
</div>

Elige el modo de seguridad que quieras para los clientes registrados dinámicamente:

**Opción A: Controles de seguridad reforzados para clientes DCR** (recomendado)

<Note>
  Antes de habilitar el modo `strict` para DCR, configura los [permisos predeterminados de la API](/docs/es/get-started/applications/application-access-to-apis-client-grants#default-permissions-for-third-party-applications) para las aplicaciones de terceros. Sin permisos predeterminados, los clientes DCR no podrán acceder a ninguna API.
</Note>

Establece `dynamic_client_registration_security_mode` en `strict`:

<AuthCodeGroup>
  ```bash cURL
  curl --request PATCH \
    --url 'https://{yourDomain}/api/v2/tenants/settings' \
    --header 'Authorization: Bearer {YOUR_MANAGEMENT_API_TOKEN}' \
    --header 'Content-Type: application/json' \
    --data '{
      "dynamic_client_registration_security_mode": "strict"
    }'
  ```
</AuthCodeGroup>

**Opción B: Mantener el comportamiento existente para clientes DCR**

Mantén o establece `dynamic_client_registration_security_mode` en `permissive`:

<AuthCodeGroup>
  ```bash cURL
  curl --request PATCH \
    --url 'https://{yourDomain}/api/v2/tenants/settings' \
    --header 'Authorization: Bearer {YOUR_MANAGEMENT_API_TOKEN}' \
    --header 'Content-Type: application/json' \
    --data '{
      "dynamic_client_registration_security_mode": "permissive"
    }'
  ```
</AuthCodeGroup>

Para obtener más información, consulta [Registro dinámico de clientes](/docs/es/get-started/applications/dynamic-client-registration).

<div id="common-scenarios">
  ## Escenarios comunes
</div>

<div id="scenario-1-partner-integrations-using-modern-oauth">
  ### Escenario 1: Integraciones de partners que usan OAuth moderno
</div>

**Situación**: Tiene integraciones de partners que usan el flujo de código de autorización con PKCE y acceden a sus API.

**Recomendación**: Adopte controles de seguridad reforzados (Opción A). Esto es totalmente compatible con implementaciones modernas de OAuth y ofrece beneficios de seguridad adicionales.

**Pasos**:

1. Configure los permisos predeterminados de API para sus API
2. Pruebe a crear una aplicación de partner con `third_party_security_mode: "strict"`
3. Complete la migración desactivando el interruptor de migración

<div id="scenario-2-applications-requiring-oidc">
  ### Escenario 2: Aplicaciones que requieren OIDC
</div>

**Situación**: Tus aplicaciones de terceros requieren scopes de OIDC (openid, profile, email) o tokens de ID.

**Recomendación**: La compatibilidad con OIDC para aplicaciones de terceros está prevista para una próxima versión. Hasta entonces, conserva el comportamiento existente (Opción B) o migra a tokens de acceso con scope de API.

**Pasos**:

* Si debes usar OIDC, mantén activado el toggle de migración y pasa explícitamente `third_party_security_mode: "permissive"` al crear aplicaciones
* Como alternativa, actualiza tus integraciones para usar scopes de API en lugar de scopes de OIDC

<div id="scenario-3-dynamic-client-registration-mcp-ai-agents">
  ### Escenario 3: registro dinámico de clientes (MCP, agentes de IA)
</div>

**Situación**: Utilizas DCR para agentes de IA, servidores MCP o aplicaciones del portal para desarrolladores.

**Recomendación**: Configura `dynamic_client_registration_security_mode: "strict"` y establece los permisos predeterminados de la API. Los clientes MCP (Claude Code, VS Code) son compatibles con los controles de seguridad reforzados.

**Pasos**:

1. Configura los permisos predeterminados de la API
2. Establece `dynamic_client_registration_security_mode: "strict"` mediante la Management API
3. Haz una prueba con un registro DCR
4. Verifica que los clientes DCR puedan obtener tokens de acceso

<div id="scenario-4-applications-using-classic-login">
  ### Escenario 4: Aplicaciones que usan Classic Login
</div>

**Situación**: Tus aplicaciones de terceros usan Classic Login en lugar de Universal Login.

**Recomendación**: Classic Login no es compatible con las aplicaciones de terceros que usan controles de seguridad reforzados. Migra a Universal Login o conserva el comportamiento existente.

**Pasos**:

* Recomendado: Migra a Universal Login antes de adoptar los controles reforzados
* Alternativa: Mantén habilitado el toggle de migración y usa explícitamente `third_party_security_mode: "permissive"`

<div id="frequently-asked-questions">
  ## Preguntas frecuentes
</div>

<div id="can-i-change-the-security-mode-on-an-existing-application">
  ### ¿Puedo cambiar el modo de seguridad de una aplicación existente?
</div>

No. `third_party_security_mode` se establece cuando se crea la aplicación y no se puede cambiar después. Para usar un modo de seguridad distinto, cree una aplicación nueva.

<div id="what-happens-to-my-existing-third-party-applications">
  ### ¿Qué sucede con mis aplicaciones de terceros existentes?
</div>

Nada. Las aplicaciones de terceros existentes seguirán funcionando exactamente igual que hoy. Esta migración solo afecta al valor predeterminado de las aplicaciones que se creen a partir de ahora.

<div id="can-i-use-both-security-modes-in-the-same-tenant">
  ### ¿Puedo usar ambos modos de seguridad en el mismo tenant?
</div>

Sí. Puede tener algunas aplicaciones de terceros con controles de seguridad reforzados y otras con el comportamiento actual. El modo de seguridad se configura por aplicación.

<div id="what-about-dynamic-client-registration">
  ### ¿Qué pasa con el registro dinámico de clientes?
</div>

DCR se controla mediante una configuración independiente del tenant llamada `dynamic_client_registration_security_mode`. Esto es independiente de la obsolescencia y requiere una decisión de configuración por separado. Para obtener más información, consulta [Registro dinámico de clientes](/docs/es/get-started/applications/dynamic-client-registration).

<div id="can-i-create-applications-with-existing-behavior-after-the-deadline">
  ### ¿Puedo crear aplicaciones con el comportamiento anterior después de la fecha límite?
</div>

Sí, pero solo con la Management API. Establece explícitamente `third_party_security_mode: "permissive"` al crear cada aplicación. El Auth0 Dashboard no permite crear aplicaciones con el comportamiento anterior.

<div id="will-future-features-work-with-existing-behavior">
  ### ¿Serán compatibles las funcionalidades futuras con el comportamiento actual?
</div>

Algunas funcionalidades futuras (como los límites de frecuencia a nivel de aplicación) solo estarán disponibles para las aplicaciones con controles de seguridad reforzados.
