---
description: Describe cómo migrar las llamadas y respuestas de la API de contraseñas de `/oauth/ro` a `/oauth/token`.
title: Migra tu flujo de contraseña del propietario del recurso
---

Se añadió compatibilidad con la contraseña del <Tooltip tip="Propietario del recurso: Entidad (como un usuario o una aplicación) capaz de conceder acceso a un recurso protegido." cta="Ver glosario" href="/docs/es/glossary?term=resource+owner">propietario del recurso</Tooltip> en `/oauth/token`. El uso del endpoint `/oauth/ro` quedó obsoleto el 08 de julio de 2017. Anteriormente, el endpoint `/oauth/ro` se usaba para intercambiar una contraseña de un solo uso (OTP) que el usuario final recibía por correo electrónico o SMS por un <Tooltip tip="Token de ID: Credencial destinada al propio cliente, en lugar de acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=ID+token">token de identidad</Tooltip> y un <Tooltip tip="Access Token: Credencial de autorización, en forma de una cadena opaca o JWT, usada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=access+token">token de acceso</Tooltip>. Auth0 ha implementado una nueva API que sustituye a `/oauth/ro` para este caso de uso, y te recomendamos migrar al nuevo endpoint.

<div id="features-affected">
  ## Funciones afectadas
</div>

Este cambio le afecta si utiliza el [flujo de contraseña del propietario del recurso](/docs/es/get-started/authentication-and-authorization-flow/resource-owner-password-flow)(a veces llamado concesión de contraseña del propietario del recurso o ROPG) y llama a `/oauth/ro` directamente sin usar ninguna biblioteca ni SDK de Auth0. Las bibliotecas de Auth0, como Lock o Auth0.js, se han actualizado para dejar de usar `/oauth/ro` internamente. Si utiliza la biblioteca lock-<Tooltip tip="Passwordless: Forma de autenticación que no depende de una contraseña como primer factor." cta="Ver glosario" href="/docs/es/glossary?term=passwordless">passwordless</Tooltip>, ahora puede usar el modo Passwordless de Lock.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Cuando el token de acceso basado en `/oauth/ro` de un usuario haya expirado, Auth0 le obligará a volver a autenticarse (es necesario forzar el cierre de sesión), ya que el token de actualización de `/oauth/ro` no puede usarse para llamar a `/oauth/token` y obtener un nuevo token de acceso. Todos los usuarios que hayan iniciado sesión tendrán que volver a iniciar sesión durante una migración de `/oauth/ro` a `/oauth/token`.
</Callout>

<div id="actions">
  ## Actions
</div>

<div id="change-requests">
  ### Cambios en las solicitudes
</div>

Anteriormente, el payload de una solicitud a `/oauth/ro` tenía este aspecto:

```json lines
{
  "grant_type": "password",
  "client_id": "123",
  "username": "alice",
  "password": "A3ddj3w", 
  "connection": "my-database-connection",
  "scope": "openid email favorite_color offline_access",
  "device": "my-device-name"
}
```

La nueva implementación incluye los siguientes cambios:

* El endpoint para ejecutar intercambios de tokens ahora es `/oauth/token`.
* El propio tipo de concesión de Auth0 se utiliza para autenticar a los usuarios mediante una conexión específica (o realm).
* Auth0 admite los scopes estándar de OIDC, además de los scopes que hayas definido en tu API personalizada.
* Un scope que no pertenezca a una de estas categorías, como `favorite_color`, ya no es un scope válido.
* Se elimina el parámetro `device`.
* El parámetro `audience` es opcional.

Aquí tienes un ejemplo del cuerpo de una solicitud a `/oauth/token`:

```json lines
{
  "grant_type": "http://auth0.com/oauth/grant-type/password-realm",
  "client_id": "123",
  "username": "alice",
  "password": "A3ddj3w",
  "realm": "my-database-connection",
  "scope": "openid email offline_access",
  "audience": "https://api.example.com"
}
```

* El tipo de concesión se especifica aquí como `password-realm`, en lugar del `password` estándar.
* Los parámetros `client_id`, `username` y `password` no cambian.
* Se incluye `realm` porque estamos usando el tipo de concesión Password Realm y sustituye al parámetro `connection` de llamadas anteriores.
* El parámetro `scope` es prácticamente el mismo, pero no acepta valores que no sean de OIDC.
* Se puede agregar el parámetro `audience`, que indica la API <Tooltip tip="Audience: Identificador único del audience de un token emitido. Denominado aud en un token, su valor contiene el ID de una aplicación (ID de cliente) para un token de ID o de una API (API Identifier) para un Access Token." cta="Ver glosario" href="/docs/es/glossary?term=audience">audience</Tooltip> a la que estará destinado el token.

<div id="response-changes">
  ### Cambios en la respuesta
</div>

Las respuestas de `/oauth/ro` tenían un formato similar a este:

```json lines
{
  "access_token": "SlAV32hkKG",
  "token_type": "Bearer",
  "refresh_token": "8xLOxBtZp8",
  "expires_in": 3600,
  "id_token": "eyJ..."
}
```

* El token de acceso devuelto es válido para llamar al endpoint `/userinfo` (siempre que la API especificada por el parámetro `audience` utilice RS256 como <Tooltip tip="Algoritmo de firma: algoritmo usado para firmar digitalmente tokens y garantizar que el token no haya sido manipulado." cta="Ver glosario" href="/docs/es/glossary?term=signing+algorithm">algoritmo de firma</Tooltip>) y, opcionalmente, para la API personalizada si se especificó alguna.
* El token de identidad se firmará obligatoriamente con RS256 si lo solicita un <Tooltip tip="Cliente público: cliente (aplicación) que no puede almacenar credenciales de forma segura. Algunos ejemplos incluyen una aplicación nativa de escritorio o móvil y una aplicación web del lado del cliente basada en JavaScript (como una aplicación web de página única (SPA))." cta="Ver glosario" href="/docs/es/glossary?term=public+client">cliente público</Tooltip>.
* Solo se devolverá un <Tooltip tip="Token de actualización: token usado para obtener un token de acceso renovado sin obligar a los usuarios a iniciar sesión de nuevo." cta="Ver glosario" href="/docs/es/glossary?term=Refresh+Token">token de actualización</Tooltip> si se concedió el scope `offline_access` y la API tiene activada la opción **Allow offline access**.

A continuación, se muestra un ejemplo de la respuesta compatible con OIDC de `/oauth/token`:

```json lines
{
  "access_token": "eyJ...",
  "token_type": "Bearer",
  "refresh_token": "8xLOxBtZp8",
  "expires_in": 3600,
  "id_token": "eyJ..."
}
```

<div id="verify-migration">
  ### Verificar la migración
</div>

1. Una vez que hayas migrado tu base de código y estés seguro de que tus aplicaciones no están llamando al endpoint, ve a [Dashboard &gt; Configuración del tenant &gt; Advanced](https://manage.auth0.com/#/tenant/advanced).
2. Desplázate hacia abajo hasta **Migración** y desactiva **Legacy** `/oauth/ro` **Endpoint**. Al desactivar este interruptor, se inhabilita el endpoint obsoleto de tu tenant, lo que evita que se siga utilizando.

Si al desactivar este interruptor se producen inicios de sesión fallidos, es una señal de que aún no has eliminado por completo todas las instancias de código heredado de tus aplicaciones.

Una vez que las migraciones se hayan realizado correctamente en entornos de producción, el interruptor puede desactivarse y mantenerse así para garantizar que las funcionalidades obsoletas ya no puedan utilizarse.
