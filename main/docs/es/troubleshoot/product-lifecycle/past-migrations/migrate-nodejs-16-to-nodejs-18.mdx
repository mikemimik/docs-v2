description: Describe los elementos de Auth0 Actions afectados por la migración de Node.js v12+6 a Node.js v18.
title: Migrar de Node 12 y 16 a Node 18
---

El soporte a largo plazo (LTS) de Node.js 12 y 16 finalizó en 2023, lo que significa que el equipo de desarrollo de Node.js ya no incorpora de forma retroactiva correcciones de seguridad críticas a estas versiones. Usar runtimes de Node 12 o 16 podría exponer tu código de extensibilidad a vulnerabilidades de seguridad.

El runtime de extensibilidad de Node 18 está disponible de forma general (GA) en toda nuestra suite de opciones de extensibilidad. Esto incluye Actions, Rules, Hooks, scripts de base de datos y Conexiones sociales personalizadas. Recomendamos encarecidamente a todos actualizar a Node 18 lo antes posible para seguir las mejores prácticas de seguridad del código.

<div id="general-considerations">
  ## Consideraciones generales
</div>

<div id="migrate-rules-and-hooks-to-actions">
  ### Migrar Rules y Hooks a Actions
</div>

Si estás usando un runtime de extensibilidad discontinuado, te recomendamos aprovechar la revisión de cómo tienes implementados tus Rules y Hooks para migrarlos a Actions (Node 18). Determina cuáles de tus Rules y Hooks puedes migrar a Actions visitando [Actions Limitations](/docs/es/customize/actions/limitations). Para obtener más información sobre cómo migrar tus Rules y Hooks a Actions, consulta [Migrate to Actions](/docs/es/customize/actions/migrate).

<div id="marketplace-integrations">
  ### Integraciones del Marketplace
</div>

<div id="social-connections-integrations">
  #### Integraciones de conexión social
</div>

Use la [Management API](https://auth0.com/docs/api/management/v2) para identificar una lista completa de las conexiones sociales que podrían verse afectadas por un cambio de versión del runtime de Node. En particular, todas las conexiones sociales potencialmente afectadas, ya sea creadas explícitamente como una conexión social personalizada o añadidas inicialmente a través del Marketplace, tienen el atributo `strategy` con un valor de `oauth1` o `oauth2`.

Luego puede recorrer con paginación todas las conexiones sociales personalizadas existentes en un tenant determinado mediante el endpoint de conexiones `GET` de la <Tooltip tip="Management API: Un producto para permitir a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip>. Por ejemplo, las siguientes opciones de consulta devuelven los nombres y los identificadores de hasta 100 conexiones sociales personalizadas:

```http wrap lines
/api/v2/connections?strategy=oauth1&strategy=oauth2&include_totals=true&fields=name&per_page=100
```

El <Tooltip tip="Auth0 Dashboard: el principal producto de Auth0 para configurar tus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip> no permite actualizar los scripts de las conexiones sociales personalizadas agregadas desde Marketplace. Si es necesario modificar un script para que sea compatible con Node 18, debes usar Management API.

<div id="migration-tasks">
  ## Tareas de migración
</div>

<div id="create-new-custom-actions">
  ### Crear nuevas Actions personalizadas
</div>

Para crear una nueva Action personalizada con Node 18 a través del Auth0 Dashboard:

1. Vaya a [Auth0 Dashboard &gt; Actions &gt; Biblioteca](https://manage.auth0.com/#/dashboard/actions/library).
2. Seleccione **Crear Action &gt; Crear desde cero**.
3. En el campo **Runtime***, seleccione **Node 18 (Recomendado)**.
4. Escriba sus Actions personalizadas en Node 18, pruébelas y despliegúelas cuando estén listas.

<div id="upgrade-existing-custom-actions">
  ### Actualizar las Actions personalizadas existentes
</div>

Puedes actualizar individualmente las Actions personalizadas existentes creadas en Node 12 o 16 a Node 18, y [revertir a la versión anterior](/docs/es/customize/actions/manage-versions#revert-actions) usando el runtime anterior. Para actualizar las Actions a Node 18, crea e implementa una nueva versión de la implementación existente con los cambios necesarios y configúrala para usar Node 18 como runtime.

<div id="choose-node-18-for-other-extensibility-products">
  ### Elige Node 18 para otros productos de extensibilidad
</div>

El runtime que se usa para las demás ofertas de extensibilidad (que no son Actions) se define globalmente en la configuración avanzada del tenant. Cambiar esta configuración afecta al mismo tiempo a la siguiente funcionalidad:

* reglas
* hooks
* scripts de Custom DB
* scripts de conexiones sociales personalizadas

Para cambiar la configuración del runtime de extensibilidad del tenant en el Auth0 Dashboard:

1. Ve a **Dashboard &gt; Settings &gt; Advanced**.
2. Desplázate hasta la sección **Extensibility**.
3. En **Runtime**, selecciona **Node 18**.

Dado que se trata de una configuración global que afecta simultáneamente a varias funcionalidades de extensibilidad, te recomendamos realizar este paso primero en tu tenant de desarrollo, completar las pruebas de todas las funcionalidades de extensibilidad correspondientes y pasar a tu tenant de producción solo cuando no detectes problemas en desarrollo.

En el caso específico de los scripts de Custom DB, puedes [seguir los pasos que se explican en esta página](/docs/es/authenticate/database-connections/custom-db/custom-database-connections-scripts/environment) para verificar individualmente un script con una versión específica del runtime antes de cambiar la versión global del runtime.

<div id="known-breaking-changes">
  ## Cambios incompatibles conocidos
</div>

<div id="magic-npm-modules">
  ### Módulos mágicos de npm
</div>

El runtime de extensibilidad de Node 12 admite el uso de módulos `npm` específicos sin necesidad de requerirlos explícitamente en el código de extensibilidad. A partir del runtime de Node 16, eliminamos la compatibilidad con este tipo de uso para los siguientes módulos:

* &#95;
* async
* Auth0
* azure&#95;storage
* bcrypt
* crypto
* couchbase
* cql
* ip
* <Tooltip tip="JSON Web Token (JWT): Formato estándar del token de ID (y, a menudo, del token de acceso) que se utiliza para representar claims de forma segura entre dos partes." cta="Ver glosario" href="/docs/es/glossary?term=jwt">jwt</Tooltip>
* Knex
* mongo
* mysql
* mysql&#95;pool
* ObjectID
* pbkdf2
* pg
* postgres
* Pubnub
* q
* querystring
* sqlserver
* uuid
* xml2js
* xmldom
* xpath
* xtend

Si todavía tienes extensibilidad ejecutándose en Node 12, ten en cuenta lo anterior al actualizar el código directamente a Node 18. Antes de usar un módulo, debes asegurarte de requerirlo explícitamente.

En el contexto de Rules, Custom Database Connections y conexiones sociales personalizadas, debes requerir explícitamente una versión del módulo que [figure como disponible para Node 18](https://auth0-extensions.github.io/canirequire/).

En Hooks y Actions, debes agregar la versión de destino prevista como una dependencia explícita antes de requerir el módulo.

Este cambio afecta al código de extensibilidad asociado con Rules, los scripts de Custom Database Connection y los scripts de conexión social personalizada.

<div id="secure-renegotiation-required-by-default-for-tls-connections">
  ### Se requiere renegociación segura de forma predeterminada para las conexiones TLS
</div>

Node.js 18 requiere de forma predeterminada la renegociación segura (RFC 5746) para las conexiones TLS, como consecuencia de que este requisito se introdujo en la dependencia subyacente de OpenSSL.

Si tu código de extensibilidad realiza llamadas de red externas, los servidores de destino deben admitir la renegociación segura; de lo contrario, las solicitudes fallarán y recibirás un error similar a:

```text wrap lines
Error: write EPROTO C0BAF076:error:0A000152:SSL routines:final_renegotiate:unsafe legacy renegotiation disabled:../deps/openssl/openssl/ssl/statem/extensions.c:922:
```

Dado este cambio relacionado con la seguridad, le recomendamos asegurarse de que todos los servidores de destino estén actualizados para admitir la renegociación segura. Si los servidores en cuestión son servidores de terceros que no están bajo su control, puede valorar la posibilidad de optar por el comportamiento anterior.

Por ejemplo, en la biblioteca `axios`, el siguiente fragmento de código muestra cómo optar por el comportamiento Legacy:

```javascript lines
const axios = require('axios');
const https = require('https');
const crypto = require('crypto');

axios.get(
  'https://[LEGACY_SERVER]', 
  {
    httpsAgent: new https.Agent(
      {
        secureOptions: crypto.constants.SSL_OP_LEGACY_SERVER_CONNECT
      }
    )
  })
```
