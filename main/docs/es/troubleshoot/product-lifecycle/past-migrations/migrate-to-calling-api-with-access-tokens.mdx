El uso de <Tooltip tip="token de ID: credencial destinada al propio cliente, en lugar de para acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=ID+tokens">tokens de ID</Tooltip> para llamar a endpoints de <Tooltip tip="token de ID: credencial destinada al propio cliente, en lugar de para acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip> está quedando obsoleto. Debe utilizar <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=access+tokens">tokens de acceso</Tooltip>. El período de gracia para esta migración comenzó el **31 de marzo de 2018**.

Después de completar la migración a tokens de acceso, desactive el interruptor **Allow ID Tokens for Management API v2 Authentication** en el Dashboard.

Si utiliza tokens de ID para llamar a cualquiera de los siguientes endpoints, esta migración le afecta. Estos endpoints ahora pueden aceptar tokens de acceso normales. No cambia nada más en el funcionamiento de los endpoints. Puede esperar los mismos esquemas de solicitud y respuesta, y solo necesita actualizar el token que usa para la autorización.

<div id="endpoints-affected">
  ## Endpoints afectados
</div>

| Endpoint                                                  | Caso de uso                                                      |
| --------------------------------------------------------- | ---------------------------------------------------------------- |
| GET /api/v2/users/{id}                                    | Obtener la información de un usuario                             |
| GET /api/v2/users/{id}/enrollments                        | Obtener todas las inscripciones de MFA de Guardian de un usuario |
| PATCH /api/v2/users/{id}                                  | Actualizar la información de un usuario                          |
| DELETE /api/v2/users/{id}/multifactor/{provider}          | Eliminar la configuración del proveedor de MFA de un usuario     |
| POST /api/v2/device-credentials                           | Crear una clave pública para un device                           |
| DELETE /api/v2/device-credentials/{id}                    | Eliminar una credencial de device                                |
| POST/api/v2/users/{id}/identities                         | Vincular cuentas de usuario de varios proveedores de identidad   |
| DELETE /api/v2/users/{id}/identities/{provider}/{user_id} | Desvincular cuentas de usuario                                   |

<div id="actions">
  ## Actions
</div>

<div id="scope-changes">
  ### Cambios en los scopes
</div>

Las acciones que puedes realizar con la Management API dependen de los scopes que contiene tu access token. Con esta migración, puedes obtener un access token limitado que solo puede actualizar los datos del usuario que ha iniciado sesión, o bien un access token que puede actualizar los datos de cualquier usuario. En la siguiente matriz, puedes ver qué scopes debe tener tu token en cada caso y para cada endpoint.

Por ejemplo, si obtienes un access token que contiene el scope `read:users`, puedes recuperar los datos de cualquier usuario mediante el endpoint `GET /api/v2/users/{id}`. Sin embargo, si tu token contiene el scope `read:current_user`, solo puedes recuperar la información del usuario que ha iniciado sesión en ese momento (aquel para el que se emitió el token).

| Endpoint                                                  | Scope para el usuario actual             | Scope para cualquier usuario |
| --------------------------------------------------------- | ---------------------------------------- | ---------------------------- |
| GET /api/v2/users/{id}                                    | `read:current_user`                      | `read:users`                 |
| GET /api/v2/users/{id}/enrollments                        | `read:current_user`                      | `read:users`                 |
| POST/api/v2/users/{id}/identities                         | `update:current_user_identities`         | `update:users`               |
| DELETE /api/v2/users/{id}/identities/{provider}/{user_id} | `update:current_user_identities`         | `update:users`               |
| PATCH /api/v2/users/{id}                                  | `update:current_user_metadata`           | `update:users`               |
| PATCH /api/v2/users/{id}                                  | `create:current_user_metadata`           | `update:users`               |
| DELETE /api/v2/users/{id}/multifactor/{provider}          | `delete:current_user_metadata`           | `update:users`               |
| POST /api/v2/device-credentials                           | `create:current_user_device_credentials` | `create:device_credentials`  |
| DELETE /api/v2/device-credentials/{id}                    | `delete:current_user_device_credentials` | `delete:device_credentials`  |

<div id="get-access-tokens">
  ### Obtener tokens de acceso
</div>

Auth0 ha cambiado la forma de obtener un token para los endpoints mencionados anteriormente. Hay varias formas de autenticar a un usuario y obtener tokens, según la tecnología y el flujo de <Tooltip tip="OAuth 2.0: Marco de autorización que define protocolos y flujos de trabajo de autorización." cta="Ver glosario" href="/docs/es/glossary?term=OAuth+2.0">OAuth 2.0</Tooltip> que utilices para autenticar:

* **SPA que se ejecuta en un navegador**: Usa el endpoint de autorización.
* **Aplicación web que se ejecuta en un servidor, una aplicación móvil, un proceso del servidor o una aplicación de alta confianza**: Usa el <Tooltip tip="Token endpoint: Endpoint en el servidor de autorización que se utiliza para solicitar tokens de forma programática." cta="Ver glosario" href="/docs/es/glossary?term=Token+endpoint">endpoint de token</Tooltip>.
* **Autenticación cruzada**: Usa Embedded Lock o auth0.js para autenticar a los usuarios cuando las solicitudes provienen de dominios distintos.

<div id="authorization-endpoint">
  #### Endpoint de autorización
</div>

En esta sección, usaremos un ejemplo para mostrar las diferencias en la forma de obtener un token con el endpoint de autorización. Ten en cuenta que, independientemente del endpoint que quieras migrar, los cambios son los mismos; lo único que varía son los scopes que especifiques en la solicitud.

En el ejemplo siguiente, usas el endpoint `GET User by ID` para recuperar toda la información del perfil del usuario que ha iniciado sesión. Para ello, primero autenticaremos al usuario mediante la concesión Implicit y recuperaremos el token o los tokens. A continuación, puedes ver una implementación del enfoque anterior que obtiene un token de ID y luego lo usa para llamar al endpoint.

export const codeExample1 = `https://{yourDomain}/authorize?
      scope=openid
      &response_type=id_token
      &client_id={yourClientId}
      &redirect_uri=https://{yourApp}/callback
      &nonce={nonce}
      &state={opaqueValue}`;

<AuthCodeBlock children={codeExample1} language="http" />

En el ejemplo siguiente, puedes ver el nuevo enfoque para obtener un token de acceso.

export const codeExample2 = `https://{yourDomain}/authorize?
      audience=https://{yourDomain}/api/v2/
      &scope=read:current_user
      &response_type=token%20id_token
      &client_id={yourClientId}
      &redirect_uri=https://{yourApp}/callback
      &nonce={nonce}
      &state={opaqueValue}`;

<AuthCodeBlock children={codeExample2} language="http" />

Para obtener un token de acceso que permita acceder a la Management API:

* Establece la `audience` en `https://{yourDomain}/api/v2/`
* Solicita el scope `${scope}`
* Establece `response_type` en `id_token token` para que Auth0 envíe tanto un token de identidad como un token de acceso

Si decodificas el token de acceso recibido y revisas su contenido, verás lo siguiente:

export const codeExample3 = `{
      "iss": "https://{yourDomain}/",
      "sub": "auth0|5a620d29a840170a9ef43672",
      "aud": "https://{yourDomain}/api/v2/",
      "iat": 1521031317,
      "exp": 1521038517,
      "azp": "{yourClientId}",
      "scope": "\${scope}"
    }`;

<AuthCodeBlock children={codeExample3} language="json" />

Ten en cuenta que `aud` está configurado con el URI de la API de tu tenant, `scope` está configurado como `${scope}` y `sub` está configurado con el ID de usuario del usuario que ha iniciado sesión.

Una vez que tengas el token de acceso, puedes usarlo para llamar al endpoint. Esta parte sigue siendo la misma; no cambia nada más en la solicitud, excepto el valor que usas como token `Bearer`. La respuesta también sigue siendo la misma.

<div id="token-endpoint">
  #### Token endpoint
</div>

En esta sección, usaremos un ejemplo para mostrar las diferencias a la hora de obtener un token con el Token endpoint. Ten en cuenta, sin embargo, que independientemente del endpoint que quieras migrar, los cambios son los mismos; lo único que varía son los scopes que especifiques en la solicitud.

En el ejemplo siguiente, quieres usar el endpoint `GET User by ID` para recuperar toda la información del perfil del usuario que ha iniciado sesión. Primero, autentica al usuario mediante la concesión Password Exchange y, después, obtén el token o los tokens. A continuación, puedes ver una implementación del enfoque anterior que obtiene un token de ID (y luego lo usa para llamar al endpoint).

export const codeExample4 = `POST https://{yourDomain}/oauth/token
    Content-Type: application/x-www-form-urlencoded
    {
      "grant_type": "password",
      "username": "{yourUsername}",
      "password": "{yourPassword}",
      "scope": "openid",
      "client_id": "{yourClientId}",
      "client_secret": "{yourClientSecret}",
    }`;

<AuthCodeBlock children={codeExample4} language="json" />

En el ejemplo siguiente, también puedes ver el nuevo enfoque con el que se obtiene un access token.

export const codeExample5 = `POST https://{yourDomain}/oauth/token
    Content-Type: application/x-www-form-urlencoded
    {
      "grant_type": "password",
      "username": "{yourUsername}",
      "password": "{yourPassword}",
      "audience": "https://{yourDomain}/api/v2/",
      "scope": "read:current_user",
      "client_id": "{yourClientId}",
      "client_secret": "{yourClientSecret}",
    }`;

<AuthCodeBlock children={codeExample5} language="json" />

Para obtener un token de acceso para acceder a la Management API:

* Establece `aud` como `https://{yourDomain}/api/v2/`
* Solicita el scope `read:current_user`

Una vez que tengas el token de acceso, puedes usarlo para llamar al endpoint. Esta parte sigue siendo la misma; no cambia nada más en la solicitud, excepto el valor que uses como token `Bearer`. La respuesta también sigue siendo la misma.

<div id="embedded-lock-or-auth0js">
  #### Embedded Lock o auth0.js
</div>

Si incrustas Lock o auth0.js v9 en tu aplicación, estás usando autenticación de origen cruzado. Se utiliza para autenticar a los usuarios cuando las solicitudes provienen de dominios distintos.

Si usas auth0.js para acceder a la Management API y gestionar tus usuarios, tendrás que actualizar tu script.

En el ejemplo siguiente, puedes ver el enfoque anterior.

export const codeExample6 = `// obtener un token de ID
    var webAuth = new auth0.WebAuth({
      clientID: '{yourClientId}',
      domain: '{yourDomain}',
      redirectUri: 'https://{yourApp}/callback',
      scope: 'openid',
      responseType: 'id_token'
    });
    // crear una instancia nueva
    var auth0Manage = new auth0.Management({
      domain: '{yourDomain}',
      token: '{yourIdToken}'
    });`;

<AuthCodeBlock children={codeExample6} language="javascript" />

En este ejemplo, se puede ver el nuevo enfoque.

export const codeExample7 = `// obtener un token de acceso
    var webAuth = new auth0.WebAuth({
      clientID: '{yourClientId}',
      domain: '{yourDomain}',
      redirectUri: 'https://{yourApp}/callback',
      audience: 'https://{yourDomain}/api/v2/',
      scope: 'read:current_user',
      responseType: 'token id_token'
    });
    // crear una nueva instancia
    var auth0Manage = new auth0.Management({
      domain: '{yourDomain}',
      token: '{yourMgmtApiAccessToken}'
    });`;

<AuthCodeBlock children={codeExample7} language="javascript" />

* Solicita tanto un token de identidad como un token de acceso en la respuesta

  `responseType: 'token id_token'`
* Configura la Management API como la <Tooltip tip="Audiencia: identificador único de la audiencia de un token emitido. Se denomina aud en un token; su valor contiene el ID de una aplicación (ID de cliente) para un token de identidad o de una API (identificador de API) para un token de acceso." cta="Ver glosario" href="/docs/es/glossary?term=audience">audiencia</Tooltip> de destino del token

  `audience: 'https://YOUR_DOMAIN/api/v2/'`
* Solicita el permiso necesario

  `scope: 'read:current_user'`
* Autentícate en la Management API con el token de acceso

<div id="account-linking-changes">
  ### Cambios en Account Linking
</div>

Los cambios en esta funcionalidad son los siguientes:

* Ya no puedes usar un token de ID en el encabezado `Authorization`
* Si usas un token de acceso en el encabezado `Authorization`, con `update:users` como permiso concedido, puedes enviar en el cuerpo de la solicitud el `user_id` o el token de ID de la cuenta secundaria
* Si usas un token de acceso en el encabezado `Authorization`, con `update:current_user_metadata` como permiso concedido, solo puedes enviar el token de ID de la cuenta secundaria en el cuerpo de la solicitud. Debe cumplirse lo siguiente:

  * El token de identidad debe estar firmado con `RS256` (puedes establecer este valor en **Dashboard &gt; Applications &gt; Application Settings &gt; Advanced Settings &gt; OAuth**)
  * El claim `aud` del token de identidad debe identificar la aplicación y tener el mismo valor que el claim `azp` del token de acceso

<div id="restrictions">
  ## Restricciones
</div>

Los tokens de acceso que se usan para acceder a la Management API solo deben tener un valor en el claim `aud`. Si tu token contiene más de un valor, tu solicitud a la Management API dará error.
