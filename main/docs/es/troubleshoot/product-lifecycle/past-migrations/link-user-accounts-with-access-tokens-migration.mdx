Anteriormente podías usar <Tooltip tip="Token de ID: Credencial destinada al propio cliente, en lugar de para acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=ID+tokens">tokens de ID</Tooltip> para vincular y desvincular cuentas de usuario en algunos casos de uso. Auth0 está dejando obsoleta esta funcionalidad. Ahora tendrás que usar <Tooltip tip="Token de ID: Credencial destinada al propio cliente, en lugar de para acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=access+tokens">tokens de acceso</Tooltip> en todos los casos.

<Warning>
  Esta obsolescencia responde a una posible vulnerabilidad de seguridad. Auth0 recomienda encarecidamente que actualices tu código lo antes posible.
</Warning>

<div id="features-affected">
  ## Funcionalidades afectadas
</div>

Los cambios en la vinculación de cuentas son los siguientes:

* Ya no puede usar un token de identidad en el encabezado `Authorization`; en su lugar, debe usar un token de acceso.
* Si usa un token de acceso en el encabezado `Authorization` con `update:users` como permiso concedido, puede enviar en el cuerpo de la solicitud el `user_id` o el token de identidad de la cuenta secundaria.
* Si usa un token de acceso en el encabezado `Authorization` con `update:current_user_metadata` como permiso concedido, solo puede enviar el token de identidad de la cuenta secundaria en el cuerpo de la solicitud.
* Si envía el token de identidad de la cuenta secundaria en el cuerpo de la solicitud (en los casos de uso descritos en los dos puntos anteriores), debe cumplirse lo siguiente:

  * El token de identidad debe estar firmado con `RS256` (puede establecer este valor en **Dashboard &gt; Clients &gt; Client Settings &gt; Advanced Settings &gt; OAuth**.
  * El claim `aud` del token de ID debe identificar al cliente y tener el mismo valor que el claim `azp` del token de acceso.
* Para desvincular cuentas, ya no puede usar un token de identidad en el encabezado `Authorization`. En su lugar, debe usar un token de acceso.

Hay varias formas de vincular y desvincular cuentas. En la siguiente lista puede ver los casos de uso y cómo les afectan estos cambios.

| Caso de uso                                                                                                                                                                                                                               | Estado            |
| ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------- |
| Use el endpoint `POST /api/v2/users/{id}/identities` de la Management API y envíe el token de identidad de la cuenta principal en el encabezado `Authorization`.                                                                          | Afectado          |
| Use el endpoint `POST /api/v2/users/{id}/identities` de la Management API y envíe un token de acceso (con scope `update:users`) en el encabezado `authorization`, y el `user_id` de la cuenta secundaria en el payload.                   | No afectado       |

| Use el endpoint `POST /api/v2/users/{id}/identities` de la Management API y envíe un token de acceso en el encabezado `Authorization` y el token de identidad de la cuenta secundaria en el payload.                                      | Nuevo caso de uso |
| Use la biblioteca auth0.js y el token de identidad de la cuenta principal para instanciar `auth0.Management`.                                                                                                                             | Afectado          |
| Use la biblioteca auth0.js y un token de acceso (con scope `update:users`) para instanciar `auth0.Management`.                                                                                                                            | No afectado       |

<div id="actions">
  ## Actions
</div>

Revisa todas tus llamadas al [endpoint Identities](https://auth0.com/docs/api/management/v2/#!/Users/post_identities) de vinculación de cuentas y actualiza aquellas que utilicen el flujo vulnerable descrito anteriormente. Puedes actualizar tus llamadas de cualquiera de las siguientes maneras:

* **Escenarios de vinculación iniciados por el usuario / del lado del cliente:** En los escenarios de vinculación del lado del cliente, realiza la llamada al endpoint Identities con un token de acceso que tenga el scope `update:current_user_identities` y proporciona el token de identidad de la cuenta secundaria en el payload (`link_with`). Este token de identidad debe obtenerse mediante un flujo compatible con <Tooltip tip="OAuth 2.0: marco de autorización que define los protocolos y flujos de trabajo de autorización." cta="Ver glosario" href="/docs/es/glossary?term=OAuth">OAuth</Tooltip>/OIDC.
* **Escenarios de vinculación del lado del servidor**: En los escenarios de vinculación del lado del servidor, realiza la llamada al endpoint Identities con un token de acceso que tenga el scope `update:users` y proporciona el `user_id` de la cuenta secundaria en el payload.

Consulta [Vincular cuentas de usuario](/docs/es/manage-users/user-accounts/user-account-linking/link-user-accounts) para obtener más detalles.

<div id="link-user-accounts">
  ### Vincular cuentas de usuario
</div>

Para vincular cuentas de usuario, puedes llamar al [Link a User Account endpoint](https://auth0.com/docs/api/management/v2#!/Users/post_identities) de la <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip> o usar la [biblioteca Auth0.js](/docs/es/libraries/auth0js).

<div id="link-current-user-accounts-with-the-management-api">
  #### Vincula las cuentas del usuario actual con la Management API
</div>

Un caso de uso común es permitir que el usuario que ha iniciado sesión vincule sus cuentas mediante tu aplicación.

Antes de la obsolescencia, podías usar el token de identidad o el token de acceso del usuario principal (que contenía el `scope` `update:current_user_identities`) para autenticarte con la Management API y usar el [endpoint Link a User Account](https://auth0.com/docs/api/management/v2#!/Users/post_identities).

Ahora debes obtener un token de acceso (que contenga el `scope` `update:current_user_identities`) y usarlo para autenticarte con la API y usar el endpoint Link a User Account. El payload debe ser el token de identidad del usuario secundario.

1. Obtén un token de acceso con el `scope` `update:current_user_identities`, como se muestra en el siguiente ejemplo. El ejemplo usa el [flujo implícito](/docs/es/get-started/authentication-and-authorization-flow/implicit-flow-with-form-post); sin embargo, puedes [obtener tokens de acceso](/docs/es/secure/tokens/access-tokens/get-access-tokens) para cualquier tipo de aplicación.

<div id="link-any-user-account-with-the-management-api">
  #### Vincula cualquier cuenta de usuario con la Management API
</div>

Si obtienes un token de acceso para la vinculación de cuentas que contiene el scope `update:users` y envías el `user_id` y el `provider` de la cuenta secundaria en la solicitud, no tienes que hacer ningún cambio.

<div id="security-considerations">
  ## Consideraciones de seguridad
</div>

Hemos identificado una vulnerabilidad en un flujo concreto de vinculación de cuentas que podría permitir su uso indebido en determinadas circunstancias. No hemos encontrado indicios de que se haya utilizado de forma maliciosa, pero hemos decidido dejar obsoleto este flujo para evitar que eso pueda ocurrir.

Por lo tanto, Auth0 exige a los clientes que utilizan el flujo de vinculación de cuentas afectado que migren a una implementación más segura antes del 19 de octubre de 2018. En esta guía se indican las rutas de migración, que no deberían implicar ninguna pérdida de funcionalidad.

A partir del 19 de octubre de 2018, el flujo de vinculación de cuentas afectado se deshabilitará y se producirán errores en tiempo de ejecución.

Se verá afectado si hace una llamada al [endpoint Post Identities](https://auth0.com/docs/api/management/v2#!/Users/post_identities) con un token (ID o token de acceso) con el scope `update:current_user_identities` en el encabezado Authorization e incluye el `user_id` de la cuenta secundaria en el payload. No se verá afectado ningún otro caso de uso.
