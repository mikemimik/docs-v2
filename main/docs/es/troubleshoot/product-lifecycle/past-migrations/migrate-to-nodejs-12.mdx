---
description: Describe las funcionalidades de Auth0 afectadas por la migración de Node.js v8 a Node.js v12 y ofrece recomendaciones para llevarla a cabo.
title: Migrar de Node.js 8 a Node.js 12
---

A partir del 31 de diciembre de 2019, [Node.js v8 dejó de contar con soporte a largo plazo (LTS)](https://github.com/nodejs/Release#release-schedule), lo que significa que el equipo de desarrollo de Node.js ya no aplica retroactivamente correcciones críticas de seguridad a esta versión. Esto podría exponer tu código de extensibilidad a vulnerabilidades de seguridad. Por ello, Auth0 migró de Node 8 a Node 12.

<div id="features-affected">
  ## Funcionalidades afectadas
</div>

Las siguientes funcionalidades de Auth0 utilizan Node 8:

* Rules
* Hooks
* conexiones de bases de datos personalizadas
* conexiones sociales personalizadas
* Extensions

Si no utilizas ninguna de las funcionalidades de extensibilidad mencionadas anteriormente, esta migración no te afecta.

<div id="tasks">
  ## Tareas
</div>

Como parte del proceso de incorporación de Node 12 a nuestro runtime de Webtask, realizamos una serie de pruebas para determinar qué módulos no son compatibles con versiones posteriores al pasar de Node 8 a Node 12. La mayoría de los clientes deberían poder actualizar a Node 12 sin ningún problema.

Dicho esto, antes de migrar, recomendamos encarecidamente que pruebe todos sus:

* Rules
* Hooks
* conexiones de base de datos personalizadas y scripts
* conexiones sociales personalizadas
* Extensions

Auth0 recomienda que primero cambie su tenant de desarrollo al runtime de Node 12, complete las pruebas en su tenant de desarrollo y migre su tenant de producción solo cuando no detecte ningún problema en desarrollo.

<div id="enable-node-12-runtime">
  ### Habilitar el runtime de Node 12
</div>

Como resultado de esta migración, puede haber cambios adicionales en el comportamiento, por lo que hemos incorporado un switch de migración que te permite controlar la migración de tu entorno al nuevo runtime de Webtask con Node 12.

Auth0 recomienda que primero cambies tu tenant de desarrollo al runtime de Node 12, completes las pruebas en tu tenant de desarrollo y migres tu tenant de producción solo cuando no detectes incidencias en desarrollo.

Puedes consultar la [Management API](/docs/es/api) para tus Rules, Hooks, scripts de base de datos personalizados y conexiones sociales personalizadas. Esto te facilitará mover elementos de tu tenant de producción a tu tenant de desarrollo para realizar pruebas.

Al usar los endpoints de Connections en la <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip>, los scripts de base de datos personalizados se pueden recuperar o actualizar mediante `options.customScripts`. Del mismo modo, puedes encontrar conexiones sociales personalizadas en `options.scripts.fetchUserProfile`.

1. Habilita Node 12 en tu tenant de desarrollo desde el nuevo panel de **Extensibilidad** en la página [Configuración avanzada del tenant](https://manage.auth0.com/#/tenant/advanced) del Dashboard. Elige **Node 12** en la lista desplegable **Runtime**.
2. Haz clic en **Save**.
3. Si usas los elementos que se enumeran a continuación, completa los pasos de migración para cada uno de ellos.
4. Prueba tu configuración.
5. Cuando estés seguro de que nada se ha roto, usa los pasos 1 y 2 anteriores para habilitar Node 12 en tu tenant de producción.

<div id="allowlist-new-urls">
  ### Agregar nuevas URL a la lista de permitidos
</div>

La extensión de Administración delegada y la extensión de Dashboard de <Tooltip tip="Inicio de sesión único (SSO): servicio que, después de que un usuario inicia sesión en una aplicación, lo inicia automáticamente en otras aplicaciones." cta="Ver glosario" href="/docs/es/glossary?term=Single+Sign-on">inicio de sesión único</Tooltip> (SSO) requieren agregar a la lista de permitidos las URL que se usan para acceder a las extensiones y a los Webtasks personalizados. Cuando actualice a Node 12, cambiarán las URL que usa para acceder a las extensiones y a los Webtasks personalizados. Este cambio puede causar incompatibilidades en estas extensiones.

Si usa alguna de estas extensiones, debe agregar las nuevas URL a la lista de permitidos tanto en **Allowed Callback URLs** como en **Allowed Logout URLs**.

La parte de la región de la URL cambiará de 8 a 12. Si accede a una extensión mediante la URL:

`https://{yourTenant}.us8.webtask.io/dummy-extension-url`

cuando actualice a Node 12, la URL será:

`https://{yourTenant}.us12.webtask.io/dummy-extension-url`

1. Vaya a [Dashboard &gt; Applications &gt; Applications &gt; Settings](https://manage.auth0.com/#/applications/\{yourClientId}/settings) y agregue la URL a los campos **Allowed Callback URLs** y **Allowed Logout URLs**.
2. Las URL de ejecución de los Webtasks personalizados en su contenedor de Auth0 también cambiarán. Debe actualizar cualquier aplicación externa que llame a esos Webtasks.

<div id="republish-authorization-extension-rule">
  ### Volver a publicar la rule de Authorization Extension
</div>

Si usas Authorization Extension, se genera una rule `auth0-authorization-extension`. Vuelve a publicar esta rule desde Authorization Extension para actualizar las URL automáticamente.

1. Asegúrate de haber actualizado a la versión más reciente de Authorization Extension desde la pestaña **Installed Extensions**. Si el botón **Upgrade** está disponible, haz clic en él para actualizar. Si el botón no aparece, ya tienes la versión más reciente de la extension.
2. Abre la página de configuración de **Authorization Extension**.
3. Para actualizar la URL en la rule, vuelve a publicarla haciendo clic en el botón **Publish Rule**.
4. Haz una prueba para asegurarte de que todo siga funcionando. Si ves el error Invalid API Key después de la actualización, haz clic en el botón **Rotate** para generar una nueva API key.

</div>

<div id="update-extensions">
  ### Actualizar Extensions
</div>

La mayoría de las extensiones usan el secreto oculto `PUBLIC_WT_URL` para la autorización. Este secreto depende de la versión del runtime y no se actualiza automáticamente.

Para actualizarlo, debes guardar la configuración de la extensión (sin necesidad de hacer cambios). Para ello, después de cambiar el runtime a `Node 12`, debes abrir la configuración de la extensión en el Dashboard de Extensions (icono de engranaje) y hacer clic en `Save`. Después, la galería de extensiones actualizará el secreto `PUBLIC_WT_URL` según el runtime seleccionado.

Si no actualizas el secreto oculto `PUBLIC_WT_URL`, recibirás el siguiente error:

<div id="update-pinned-modules">
  ### Actualizar módulos con versión fijada
</div>

Si estás usando los siguientes módulos integrados (es decir, módulos que no cargaste explícitamente con `require`), ten en cuenta que algunas versiones se actualizaron para funcionar con Node 12. La siguiente tabla resume los cambios. Estas nuevas versiones deberían seguir siendo compatibles con las versiones anteriores.

| Nombre del módulo | Versión anterior | Nueva versión |
| ----------------- | ---------------- | ------------- |
| couchbase         | ~2.5.1           | 2.6.10        |
| bcrypt            | 1.0.3            | 3.0.8         |

Si has fijado manualmente la versión de algún módulo, puede que tengas que actualizarla también para que tu código se ejecute con Node 12.

Por ejemplo, debes cambiar:

`var bcrypt = require(‘bcrypt@1.0.3’);`

por

`var bcrypt = require(‘bcrypt’);`

o, si el módulo debe fijarse a una versión específica:

`var bcrypt = require(‘bcrypt@3.0.8’);`
