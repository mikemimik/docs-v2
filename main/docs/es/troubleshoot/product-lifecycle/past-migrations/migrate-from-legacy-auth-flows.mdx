title: Migrar desde flujos de autenticación heredados
---

Si usas versiones de Lock anteriores a la 11 y versiones de Auth0.js anteriores a la 9, es posible que estés usando flujos de autenticación heredados, que están obsoletos. Auth0 recomienda migrar el código de versiones anteriores de Auth0.js y Lock a las nuevas API conformes con OIDC.

<div id="renew-tokens">
  ## Renovar tokens
</div>

Las aplicaciones Legacy usaban <Tooltip tip="Token de actualización: token usado para obtener un token de acceso renovado sin obligar a los usuarios a iniciar sesión de nuevo." cta="Ver glosario" href="/docs/es/glossary?term=Refresh+Tokens">tokens de actualización</Tooltip> y la función `refreshToken()` para obtener nuevos tokens cuando caducaban (a continuación se muestra un ejemplo).

En auth0.js v9 y Lock 11 debes usar [autenticación silenciosa](/docs/es/authenticate/login/configure-silent-authentication) y `checkSession()`(a continuación se muestra un ejemplo).

<div id="call-apis">
  ## Llamar a APIs
</div>

Las aplicaciones heredadas usaban un <Tooltip tip="token de identidad: Credencial destinada al propio cliente, en lugar de usarse para acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=ID+token">token de identidad</Tooltip> para invocar APIs. Esto es una mala práctica, y te recomendamos que uses únicamente <Tooltip tip="Token de acceso: Credencial de autorización, en forma de una cadena opaca o JWT, que se utiliza para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=access+tokens">tokens de acceso</Tooltip>.

Para llamar a una API, tendrás que especificar el identificador de la API como parámetro `audience` al inicializar auth0.js o Lock.

Si especificas un <Tooltip tip="Audiencia: Identificador único de la audiencia de un token emitido. Se denomina aud en un token y su valor contiene el ID de una aplicación (ID de cliente) para un token de identidad o de una API (identificador de API) para un token de acceso." cta="Ver glosario" href="/docs/es/glossary?term=audience">audiencia</Tooltip>, se activará el flujo OIDC y los datos del perfil de usuario que Auth0 devuelve en los tokens de identidad o desde `/userinfo` serán conformes con OIDC. Si tu aplicación utiliza algún claim no estándar del perfil de usuario, dejará de funcionar.

Consulta la sección **Llamar a una API** de nuestros Quickstarts de SPA para obtener más información sobre cómo llamar a APIs desde SPAs. También tendrás que migrar la implementación de tu backend API para usar tokens de acceso. Consulta los Quickstarts de API para ver las instrucciones sobre cómo hacerlo.

<div id="user-profiles">
  ## Perfiles de usuario
</div>

Los flujos de autenticación heredados que permiten que los tokens de ID y el endpoint `/userinfo` incluyan el perfil de usuario completo están quedando obsoletos. Asegúrate de que el selector `Legacy User Profile` esté desactivado después de completar la migración a las nuevas API conformes con OIDC.

Al usar los flujos de autenticación heredados, el perfil de usuario completo se devuelve en los tokens de ID y desde `/userinfo`, como se muestra a continuación.

El nuevo perfil de usuario cumple con la especificación de OIDC, que permite que ciertos [claims estándar](https://openid.net/specs/openid-connect-core-1_0.html#StandardClaims) estén disponibles en la respuesta.

El contenido variará según los scopes que se soliciten. Tendrás que ajustar los scopes que solicitas al configurar Auth0.js o Lock para que todos los claims que necesitas estén disponibles en tu aplicación. Ten en cuenta que puedes agregar claims personalizados para devolver cualquier dato que quieras (por ejemplo, metadatos de usuario).

Otra forma de obtener el perfil de usuario completo es usar la <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip> (en lugar de obtener el perfil a través del flujo de autenticación), como se describe en la siguiente sección.

<div id="user-profile-with-management-api">
  ## Perfil de usuario con Management API
</div>

En los flujos heredados, la Management API admitía la autenticación con un token de ID. Este enfoque ha quedado obsoleto, y ahora debes llamarla con un token de acceso.

Para obtener un token de acceso, debes solicitarle uno a Auth0 usando la audiencia `https://{yourDomain}/api/v2/`. Actualmente, Auth0 no admite especificar dos audiencias al autenticar, por lo que tendrás que seguir usando la audiencia de la API de tu aplicación al inicializar Lock o auth0.js. Una vez que el usuario se haya autenticado, puedes usar `checkSession` para obtener un `access_token` de la Management API y luego llamar al endpoint `getUser()`.

Puedes solicitar los siguientes scopes:

* `read:current_user`
* `update:current_user_identities`
* `create:current_user_metadata`
* `update:current_user_metadata`
* `delete:current_user_metadata`
* `create:current_user_device_credentials`
* `delete:current_user_device_credentials`

Es posible que recibas un error `consent_required` al llamar a `checkSession()`. Si sucede, asegúrate de tener habilitada la opción **Allow Skipping User Consent** para la Management API y de no estar ejecutándote desde localhost.
