Auth0 está dejando en desuso el uso del endpoint `/passwordless/start` desde aplicaciones confidenciales cuando Auth0 no puede autenticar que la llamada se hace en nombre de la aplicación.

<Tooltip tip="OAuth 2.0: Marco de autorización que define protocolos de autorización y flujos de trabajo." cta="Ver glosario" href="/docs/es/glossary?term=OAuth">OAuth</Tooltip> usa el término &#39;confidential&#39; para las aplicaciones que pueden almacenar secrets. En Auth0, estas son las &#39;aplicaciones web clásicas&#39;, que sirven páginas web desde una aplicación backend. Las aplicaciones de página única y las aplicaciones nativas se consideran aplicaciones &#39;públicas&#39; y no se ven afectadas por este cambio.

Auth0 puede autenticar llamadas a `/passwordless/start` cuando incluyen un `client_secret` como parámetro, o cuando se hacen desde la página de inicio de sesión personalizada en Classic Login y reenvían el parámetro `state`.

<div id="features-affected">
  ## Funcionalidades afectadas
</div>

Si alguna de sus aplicaciones llama actualmente al endpoint `/passwordless/start` directamente para iniciar la <Tooltip tip="Passwordless: Forma de autenticación que no depende de una contraseña como primer factor." cta="Ver glosario" href="/docs/es/glossary?term=passwordless">passwordless</Tooltip> autenticación desde una aplicación web y no está enviando el `client_secret` como parámetro, esta obsolescencia sí le afecta.

Si está implementando autenticación passwordless a través de la página de Classic Login y cambió la forma predeterminada en que se inicializan las bibliotecas de Auth0, también podría verse afectado.

Puede comprobar si se ve afectado revisando el [registro del tenant](https://manage.auth0.com/#/logs), filtrando por &quot;Aviso de obsolescencia&quot; y comprobando si hay logs que indiquen &quot;Aplicar autenticación del cliente para conexiones passwordless&quot;. También puede realizar esta búsqueda directamente con la siguiente consulta: `type:depnote AND description:*passwordless*`. Tenga en cuenta que esta consulta específica solo funcionará para tenants de Public Cloud, ya que en Private Cloud no se puede buscar en el campo description dentro de los logs.

<div id="actions">
  ## Actions
</div>

Si llamas al endpoint `/passwordless/start` sin la autenticación adecuada de la aplicación, debes:

* Seguir las instrucciones que se describen a continuación para ajustar el código y llamar correctamente a `/passwordless/start`.
* Revisar tus [registros del tenant](https://manage.auth0.com/#/logs) para verificar que el cambio se haya realizado correctamente y que no se estén generando registros de obsolescencia para &quot;Aplicar autenticación del cliente para conexiones passwordless&quot;.
* En la sección de **Migración** de **Configuración avanzada del tenant**, activa el toggle **Aplicar autenticación del cliente para conexiones passwordless**.

Hay algunos casos de uso que podrían verse afectados, pero, en cada uno de ellos, la ruta de migración es bastante sencilla:

<div id="api-calls-from-backend">
  ### Llamadas a la API desde el backend
</div>

Para cualquier llamada desde tu backend al endpoint `/passwordless/start`, la solicitud debe incluir el <Tooltip tip="Secreto de cliente: Secret que usa un cliente (aplicación) para autenticarse con el servidor de autorización; solo el cliente y el servidor de autorización deben conocerlo, y debe ser lo suficientemente aleatorio para que no se pueda adivinar." cta="Ver glosario" href="/docs/es/glossary?term=client+secret">secreto de cliente</Tooltip> como parámetro.

Si haces una solicitud POST directamente a `/passwordless/start`, incluye `client_secret` como parte del payload:

Si usas un SDK, agrega el parámetro al método que inicia el flujo passwordless. Esto varía según el SDK, y no todos los SDKs se han actualizado todavía. Si usas un SDK que aún no se ha actualizado, puedes hacer la llamada HTTP directamente hasta que ese trabajo se haya completado.

<div id="auth0js-or-lockjs-in-the-classic-login-page">
  ### Auth0.js o Lock.js en la página de Classic Login
</div>

Si la página de Classic Login se usa para la autenticación passwordless en una aplicación web, hará llamadas al endpoint `/passwordless/start`, ya sea con Lock.js o con Auth0.js.

Dado que no puedes almacenar un secreto de cliente en una página web, la forma de autenticar la llamada es reenviar el parámetro `state` que se recibe en la página de Classic Login al endpoint `/passwordless/start`. Ese parámetro se almacena en el campo `config.internalOptions` de la página de inicio de sesión personalizada.

Las plantillas predeterminadas para personalizar la página de inicio de sesión lo usan de la siguiente manera al inicializar Lock.js o auth0.js:

```javascript lines
var lock = new Auth0Lock(
  config.clientID, 
  config.auth0Domain, 
  {
    auth: {
      // .. otros campos configurados
      params: {
       scope: config.internalOptions.scope,
       _csrf: config.internalOptions._csrf,
       state: config.internalOptions.state,
      }
    }
  });
```

```javascript lines
var params = Object.assign({
    scope: config.internalOptions.scope,
    _csrf: config.internalOptions._csrf,
    state: config.internalOptions.state,
  }, {
    // ...parámetros de autenticación
  });

var webAuth = new auth0.WebAuth(params);
```

Compruebe en la implementación de su página personalizada que no haya eliminado ese código.

<div id="call-passwordlessstart-from-the-client-in-a-web-application">
  ### Llama a /passwordless/start desde el cliente en una aplicación web
</div>

Si llamas al endpoint `/passwordless/start` desde una página con JavaScript (por ejemplo, con Auth0.js en la propia página) en aplicaciones web tradicionales, no podrás especificar un secreto de cliente en una llamada hecha con JavaScript. Si este es el caso de tu aplicación, tendrás que modificar tus aplicaciones para que `/passwordless/start` se llame desde el backend de tu aplicación web, en lugar de desde el frontend.

<div id="rate-limits">
  ### Límites de frecuencia
</div>

Una consecuencia de agregar autenticación del cliente a `/passwordless/start` es que Auth0 puede confiar en los encabezados enviados con la solicitud. Si establece el encabezado `auth0-forwarded-for`, la dirección IP aparecerá en los logs y se usará con fines de <Tooltip tip="Protección contra ataques: funcionalidades que Auth0 proporciona para detectar y mitigar ataques, incluida la protección contra ataques de fuerza bruta, Suspicious IP Throttling, la detección de contraseñas vulneradas, Bot Detection y la autenticación multifactor adaptativa." cta="Ver glosario" href="/docs/es/glossary?term=attack+protection">protección contra ataques</Tooltip>.

Las solicitudes autenticadas a `/passwordless/start` se tratarán como solicitudes autenticadas normales a la API, y se aplicará el correspondiente límite de frecuencia global de Authentication API.
