Auth0 está dejando de admitir el uso del endpoint `/passwordless/start` desde aplicaciones confidenciales cuando no puede autenticar que la llamada se hace en nombre de la aplicación.

<Tooltip tip="OAuth 2.0: Marco de autorización que define protocolos y flujos de trabajo de autorización." cta="Ver glosario" href="/docs/es/glossary?term=OAuth">OAuth</Tooltip> usa el término &#39;confidencial&#39; para las aplicaciones que pueden almacenar secretos. En Auth0, esas son las &#39;aplicaciones web clásicas&#39;, que sirven páginas web desde una aplicación de backend. Las Single Page Applications y las Native Applications se consideran aplicaciones &#39;públicas&#39; y no se ven afectadas por este cambio.

Auth0 puede autenticar las llamadas a `/passwordless/start` cuando incluyen un `client_secret` como parámetro, o cuando se hacen desde la página de inicio de sesión personalizada en Classic Login y transmiten el parámetro `state`.

<div id="features-affected">
  ## Funcionalidades afectadas
</div>

Si alguna de sus aplicaciones llama actualmente al endpoint `/passwordless/start` directamente para iniciar la autenticación <Tooltip tip="Passwordless: Forma de autenticación que no depende de una contraseña como primer factor." cta="Ver glosario" href="/docs/es/glossary?term=passwordless">passwordless</Tooltip> desde una Aplicación web y no está enviando `client_secret` como parámetro, esta obsolescencia sí le afecta.

Si está implementando la autenticación passwordless a través de la página de Classic Login y cambió la forma predeterminada en que se inicializan las bibliotecas de Auth0, es posible que esto también le afecte.

Puede comprobar si se ve afectado revisando los [registros del tenant](https://manage.auth0.com/#/logs), filtrando por &quot;Deprecation Notice&quot; y buscando registros que indiquen &quot;Enforce client authentication for passwordless connections&quot;. También puede realizar esta búsqueda directamente con la siguiente consulta: `type:depnote AND description:*passwordless*`. Tenga en cuenta que esta consulta específica solo funcionará para tenants de Public Cloud, ya que en Private Cloud no se pueden buscar registros por el campo description.

<div id="actions">
  ## Actions
</div>

Si llamas al endpoint `/passwordless/start` sin la autenticación adecuada de la aplicación, debes:

* Seguir las instrucciones que se describen a continuación para ajustar el código y llamar correctamente a `/passwordless/start`.
* Revisar tus [registros del tenant](https://manage.auth0.com/#/logs) para verificar que el cambio se haya realizado correctamente y que no se estén generando registros de obsolescencia para &quot;Enforce client authentication for passwordless connections&quot;.
* En la sección **Migración** de **Configuración avanzada del tenant**, activa el interruptor **Enforce client authentication for passwordless connections**.

Hay algunos casos de uso que podrían verse afectados, pero la ruta de migración para cada uno es bastante sencilla:

<div id="api-calls-from-backend">
  ### Llamadas a la API desde el backend
</div>

En cualquier llamada desde tu backend al endpoint `/passwordless/start`, debes incluir el <Tooltip tip="Secreto de cliente: secreto que usa un cliente (aplicación) para autenticarse con el servidor de autorización; solo deberían conocerlo el cliente y el servidor de autorización, y debe ser lo suficientemente aleatorio como para que no pueda adivinarse." cta="Ver glosario" href="/docs/es/glossary?term=client+secret">secreto de cliente</Tooltip> como parámetro.

Si haces una solicitud POST directamente a `/passwordless/start`, incluye `client_secret` en el payload:

Si estás usando un SDK, agrega el parámetro al método que inicia el flujo passwordless. Esto varía según el SDK, y no todos los SDKs se han actualizado todavía. Si estás usando un SDK que aún no se ha actualizado, puedes hacer la llamada HTTP directamente hasta que se complete esa actualización.

<div id="auth0js-or-lockjs-in-the-classic-login-page">
  ### Auth0.js o Lock.js en la página de Classic Login
</div>

Si la página de Classic Login se utiliza para autenticación passwordless en una aplicación web, hará llamadas al endpoint `/passwordless/start`, ya sea con Lock.js o Auth0.js.

Como no puedes almacenar un secreto de cliente en una página web, la forma de autenticar la llamada es reenviar el parámetro `state` que se recibe en la página de Classic Login al endpoint `/passwordless/start`. Ese parámetro se almacena en el campo `config.internalOptions` de la página de inicio de sesión personalizada.

Las plantillas predeterminadas para personalizar la página de inicio de sesión lo usan de la siguiente manera al inicializar Lock.js o auth0.js:

```javascript lines
var lock = new Auth0Lock(
  config.clientID, 
  config.auth0Domain, 
  {
    auth: {
      // .. otros campos establecidos
      params: {
       scope: config.internalOptions.scope,
       _csrf: config.internalOptions._csrf,
       state: config.internalOptions.state,
      }
    }
  });
```

```javascript lines
var params = Object.assign({
    scope: config.internalOptions.scope,
    _csrf: config.internalOptions._csrf,
    state: config.internalOptions.state,
  }, {
    // ...parámetros de autenticación
  });

var webAuth = new auth0.WebAuth(params);
```

Revise la implementación de su página personalizada para verificar que no haya eliminado ese código.

<div id="call-passwordlessstart-from-the-client-in-a-web-application">
  ### Llamar a `/passwordless/start` desde el cliente en una aplicación web
</div>

Si llamas al endpoint `/passwordless/start` desde una página que usa JavaScript (por ejemplo, con Auth0.js en la página) en aplicaciones web tradicionales, no podrás especificar un secreto de cliente en una llamada hecha con JavaScript. Si ese es el caso de tu aplicación, tendrás que modificarla para que `/passwordless/start` se llame desde el backend de tu aplicación web, y no desde el frontend.

<div id="rate-limits">
  ### Límites de frecuencia
</div>

Una consecuencia de agregar autenticación del cliente a `/passwordless/start` es que Auth0 puede confiar en los encabezados enviados con la solicitud. Si configura el encabezado `auth0-forwarded-for`, la dirección IP aparecerá en los registros y se usará con fines de <Tooltip tip="Protección contra ataques: Funcionalidades que Auth0 proporciona para detectar y mitigar ataques, incluida la protección contra ataques de fuerza bruta, la limitación de IP sospechosas, la detección de contraseñas vulnerada, la detección de bots y la autenticación multifactor adaptativa." cta="Ver glosario" href="/docs/es/glossary?term=attack+protection">protección contra ataques</Tooltip>.

Las solicitudes autenticadas a `/passwordless/start` se tratarán como solicitudes autenticadas normales de la API y se les aplicará el límite de frecuencia global correspondiente de la Authentication API.
