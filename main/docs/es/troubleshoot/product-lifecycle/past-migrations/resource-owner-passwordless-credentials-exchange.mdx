Se agregó compatibilidad con la contraseña de <Tooltip tip="Resource Owner: Entidad (como un usuario o una aplicación) capaz de conceder acceso a un recurso protegido." cta="Ver glosario" href="/docs/es/glossary?term=resource+owner">propietario del recurso</Tooltip> a `/oauth/token`. El endpoint `/oauth/ro` quedó obsoleto el 08 de julio de 2017. Anteriormente, el endpoint `/oauth/ro` se usaba para intercambiar una contraseña de un solo uso (OTP) recibida por correo electrónico o SMS por el usuario final por un <Tooltip tip="Token de identidad: Credencial destinada al cliente mismo, en lugar de para acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=ID+token">token de identidad</Tooltip> y un <Tooltip tip="Token de acceso: Credencial de autorización, en forma de una cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=access+token">token de acceso</Tooltip>. Auth0 ha implementado una nueva API que reemplaza `/oauth/ro` para este caso de uso y te recomendamos migrar al nuevo endpoint.

<div id="features-affected">
  ## Características afectadas
</div>

Este cambio le afecta si usa el intercambio de credenciales passwordless de propietario del recurso <Tooltip tip="Passwordless: Forma de autenticación que no depende de una contraseña como primer factor." cta="Ver glosario" href="/docs/es/glossary?term=passwordless">passwordless</Tooltip> y llama a `/oauth/ro` directamente sin usar ninguna biblioteca ni SDK de Auth0.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Cuando vence el token de acceso de un usuario basado en `/oauth/ro`, Auth0 lo obliga a volver a autenticarse (se requiere cierre de sesión forzado) porque el token de actualización de `/oauth/ro` no puede usarse para llamar a `/oauth/token` y obtener un nuevo token de acceso. Todos los usuarios que hayan iniciado sesión tendrán que volver a iniciar sesión durante una migración de `/oauth/ro` a `/oauth/token`.
</Callout>

<div id="actions">
  ## Actions
</div>

<div id="request-changes">
  ### Cambios en las solicitudes
</div>

Anteriormente, el payload de una solicitud a `/oauth/ro` era similar a este:

```json lines
{
  "grant_type": "password",
  "client_id": "123",
  "username": "alice",
  "password": "A3ddj3w", 
  "connection": "my-database-connection",
  "scope": "openid email favorite_color offline_access",
  "device": "my-device-name"
}
```

Estos son los cambios de la nueva implementación:

* El endpoint para ejecutar el intercambio de tokens ahora es `/oauth/token`.
* El propio tipo de concesión de Auth0 se usa para autenticar a usuarios desde una conexión (o realm) específica.
* Auth0 admite los scopes estándar de OIDC, junto con los scopes que hayas definido en tu API personalizada.
* Un scope que no encaje en una de estas categorías, como `favorite_color`, ya no es un scope válido.
* Se elimina el parámetro `device`.
* El parámetro `audience` es opcional.

Aquí tienes un ejemplo del payload de una solicitud a `/oauth/token`:

export const codeExample = `{
  "grant_type" : "http://auth0.com/oauth/grant-type/passwordless/otp",
  "client_id": "{yourClientId}",
  "client_secret": "{yourClientSecret}", // solo para aplicaciones web; las aplicaciones nativas no tienen secreto de cliente
  "username": "{userEmailAddress}", // o "{userPhoneNumber}"
  "otp": "CÓDIGO",
  "realm": "email", // o "sms" 
  "audience" : "{yourApiIdentifier}", // en caso de que necesites un token de acceso para una API específica
  "scopes": "openid profile email" // los scopes que necesites
}`;

<AuthCodeBlock children={codeExample} language="json" />

* El tipo de concesión se especifica aquí como `http://auth0.com/oauth/grant-type/passwordless/otp`.
* Los parámetros `client_id` y `username` no cambian.
* Se debe especificar `client_secret` para los <Tooltip tip="Confidential Client: Un cliente (aplicación) que puede almacenar credenciales de forma segura mediante un servidor backend de confianza. Algunos ejemplos son una aplicación web con un backend seguro y una aplicación de máquina a máquina (M2M)." cta="Ver glosario" href="/docs/es/glossary?term=confidential+clients">clientes confidenciales</Tooltip> (por ejemplo, aplicaciones web tradicionales).
* La contraseña de un solo uso debe enviarse en el parámetro `otp` en lugar de en el parámetro `password`.
* `realm` se usa para identificar la conexión y sustituye al parámetro `connection` de llamadas anteriores.
* El parámetro `scope` es prácticamente el mismo, pero no acepta valores que no sean OIDC.
* Se puede añadir el parámetro `audience`, que indica el <Tooltip tip="Audience: Identificador único del destinatario de un token emitido. Se denomina aud en un token; su valor contiene el ID de una aplicación (Client ID) para un token de ID o de una API (API Identifier) para un access token." cta="Ver glosario" href="/docs/es/glossary?term=audience">destinatario</Tooltip> de la API a la que irá dirigido el token.

<div id="response-changes">
  ### Cambios en las respuestas
</div>

Las respuestas de `/oauth/ro` tenían un formato similar al siguiente:

```json lines
{
  "access_token": "SlAV32hkKG",
  "token_type": "Bearer",
  "refresh_token": "8xLOxBtZp8",
  "expires_in": 3600,
  "id_token": "eyJ..."
}
```

* El token de acceso devuelto es válido para llamar al endpoint `/userinfo` (siempre que la API especificada por el parámetro `audience` use RS256 como <Tooltip tip="Algoritmo de firma: algoritmo utilizado para firmar digitalmente tokens y garantizar que no hayan sido manipulados." cta="Ver glosario" href="/docs/es/glossary?term=signing+algorithm">algoritmo de firma</Tooltip>) y, opcionalmente, a la API personalizada, si se especificó alguna.
* El token de identidad se firmará obligatoriamente con RS256 si lo solicita un <Tooltip tip="Cliente público: cliente (aplicación) que no puede almacenar credenciales de forma segura. Algunos ejemplos son una aplicación nativa de escritorio o móvil y una aplicación web del lado del cliente basada en JavaScript (como una aplicación de una sola página, o SPA)." cta="Ver glosario" href="/docs/es/glossary?term=public+client">cliente público</Tooltip>.
* Solo se devolverá un <Tooltip tip="Token de actualización: token utilizado para obtener un token de acceso renovado sin obligar a los usuarios a volver a iniciar sesión." cta="Ver glosario" href="/docs/es/glossary?term=refresh+token">token de actualización</Tooltip> si se concedió el scope `offline_access` y la API tiene activada la opción **Allow offline access**.

A continuación, se muestra un ejemplo de la respuesta compatible con OIDC de `/oauth/token`:

```json lines
{
  "access_token": "eyJ...",
  "token_type": "Bearer",
  "refresh_token": "8xLOxBtZp8",
  "expires_in": 3600,
  "id_token": "eyJ..."
}
```

<div id="code-changes-when-using-the-sdks">
  ### Cambios en el código al usar los SDKs
</div>

Si su aplicación utiliza las bibliotecas nativas de Auth0 para Android o iOS, asegúrese de que la versión de la biblioteca incluida sea, como mínimo, la que se indica a continuación (o una superior). Además, asegúrese de establecer la marca `OIDC Conformant` en `true` al configurar las bibliotecas.

| Biblioteca   | Versión mínima |
| ------------ | -------------- |
| Android SDK  | 1.2            |
| Lock Android | 2.17           |
| Swift SDK    | 1.20.0         |
| Lock iOS     | 2.14.0         |

<div id="verify-migration">
  ### Verificar la migración
</div>

1. Para comprobar si está utilizando el endpoint obsoleto, revise los [registros del tenant](https://manage.auth0.com/#/logs), filtre por **Aviso de obsolescencia** y luego busque registros que indiquen `oauth/ro` **passwordless: Esta funcionalidad está quedando obsoleta**. También puede realizar esta búsqueda directamente con la siguiente consulta: `type:depnote AND description:*passwordless*`.
2. Una vez que haya migrado su base de código y esté seguro de que sus aplicaciones no están invocando el endpoint, vaya al [Dashboard](https://manage.auth0.com/#/tenant/advanced), en **Configuración del tenant &gt; Advanced**.
3. Desplácese hacia abajo hasta **Migrations** y desactive **Legacy** `/oauth/ro` **Endpoint**. Al desactivar este interruptor, se deshabilita el endpoint obsoleto para su tenant, lo que impide que se siga utilizando.

Si al desactivar este interruptor se producen inicios de sesión fallidos, es señal de que todavía no ha eliminado por completo todas las instancias de código heredado de sus aplicaciones.

Una vez que las migraciones se hayan realizado correctamente en los entornos de producción, el interruptor puede desactivarse y dejarse así para garantizar que las funcionalidades obsoletas ya no puedan utilizarse.
