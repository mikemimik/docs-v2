Se añadió compatibilidad con la contraseña de <Tooltip tip="Resource Owner: Entidad (como un usuario o una aplicación) capaz de conceder acceso a un recurso protegido." cta="Ver glosario" href="/docs/es/glossary?term=resource+owner">propietario del recurso</Tooltip> en `/oauth/token`. El uso del endpoint `/oauth/ro` quedó obsoleto el 08 de julio de 2017. Anteriormente, el endpoint `/oauth/ro` se usaba para intercambiar una contraseña de un solo uso (OTP) que el usuario final recibía por correo electrónico o SMS por un <Tooltip tip="ID Token: Credencial destinada al cliente mismo, en lugar de para acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=ID+token">token de identidad</Tooltip> y un <Tooltip tip="Access Token: Credencial de autorización, en forma de una cadena opaca o JWT, usada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=access+token">token de acceso</Tooltip>. Auth0 ha implementado una nueva API que reemplaza `/oauth/ro` para este caso de uso y le recomendamos migrar al nuevo endpoint.

<div id="features-affected">
  ## Funciones afectadas
</div>

Este cambio le afecta si utiliza el intercambio de credenciales del propietario del recurso <Tooltip tip="Passwordless: Form of authentication that does not rely on a password as the first factor." cta="View Glossary" href="/docs/es/glossary?term=passwordless">passwordless</Tooltip> y llama a `/oauth/ro` directamente sin usar ninguna biblioteca ni SDK de Auth0.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Cuando el token de acceso basado en `/oauth/ro` de un usuario caduca, Auth0 le obliga a volver a autenticarse (se requiere un cierre de sesión forzado), ya que el token de actualización de `/oauth/ro` no puede usarse para llamar a `/oauth/token` y obtener un nuevo token de acceso. Todos los usuarios que tengan una sesión iniciada actualmente deberán volver a iniciar sesión durante una migración de `/oauth/ro` a `/oauth/token`.
</Callout>

<div id="actions">
  ## Actions
</div>

<div id="request-changes">
  ### Cambios en las solicitudes
</div>

Anteriormente, el payload de una solicitud a `/oauth/ro` era similar a este:

```json lines
{
  "grant_type": "password",
  "client_id": "123",
  "username": "alice",
  "password": "A3ddj3w", 
  "connection": "my-database-connection",
  "scope": "openid email favorite_color offline_access",
  "device": "my-device-name"
}
```

Estos son los cambios de la nueva implementación:

* El endpoint para realizar intercambios de tokens ahora es `/oauth/token`.
* El tipo de concesión propio de Auth0 se utiliza para autenticar a los usuarios desde una conexión (o realm) específica.
* Auth0 admite los scopes estándar de OIDC, junto con los scopes que hayas definido en tu API personalizada.
* Un scope que no encaja en una de estas categorías, como `favorite_color`, ya no es válido.
* Se elimina el parámetro `device`.
* El parámetro `audience` es opcional.

Aquí tienes un ejemplo del payload de una solicitud a `/oauth/token`:

export const codeExample = `{
  "grant_type" : "http://auth0.com/oauth/grant-type/passwordless/otp",
  "client_id": "{yourClientId}",
  "client_secret": "{yourClientSecret}", // solo para web apps; las aplicaciones nativas no tienen secreto de cliente
  "username": "{userEmailAddress}", // o "{userPhoneNumber}"
  "otp": "CODE",
  "realm": "email", // o "sms" 
  "audience" : "{yourApiIdentifier}", // en caso de que necesites un token de acceso para una API concreta
  "scopes": "openid profile email" // los scopes que necesites
}`;

<AuthCodeBlock children={codeExample} language="json" />

* El tipo de concesión se especifica aquí como `http://auth0.com/oauth/grant-type/passwordless/otp`.
* Los parámetros `client_id` y `username` no cambian.
* Se debe especificar `client_secret` para los <Tooltip tip="Cliente confidencial: un cliente (aplicación) que puede mantener las credenciales de forma segura mediante un backend de confianza. Algunos ejemplos incluyen una aplicación web con un backend seguro y una aplicación de máquina a máquina (M2M)." cta="Ver glosario" href="/docs/es/glossary?term=confidential+clients">clientes confidenciales</Tooltip> (por ejemplo, aplicaciones web tradicionales).
* La contraseña de un solo uso debe enviarse en el parámetro `otp` en lugar de en el parámetro `password`.
* El `realm` se usa para identificar la conexión y sustituye al parámetro `connection` de llamadas anteriores.
* El parámetro `scope` es prácticamente el mismo, pero no acepta valores ajenos a OIDC.
* Se puede agregar el parámetro `audience`, que indica la <Tooltip tip="Audiencia: identificador único de la audiencia de un token emitido. Denominado aud en un token, su valor contiene el ID de una aplicación (ID de cliente) para un token de ID o de una API (identificador de API) para un token de acceso." cta="Ver glosario" href="/docs/es/glossary?term=audience">audiencia</Tooltip> de la API a la que se destinará el token.

<div id="response-changes">
  ### Cambios en las respuestas
</div>

Las respuestas de `/oauth/ro` tenían un formato similar al siguiente:

```json lines
{
  "access_token": "SlAV32hkKG",
  "token_type": "Bearer",
  "refresh_token": "8xLOxBtZp8",
  "expires_in": 3600,
  "id_token": "eyJ..."
}
```

* El token de acceso devuelto es válido para llamar al endpoint `/userinfo` (siempre que la API especificada por el parámetro `audience` use RS256 como <Tooltip tip="Signing Algorithm: Algoritmo utilizado para firmar digitalmente los tokens y garantizar que el token no haya sido manipulado." cta="Ver glosario" href="/docs/es/glossary?term=signing+algorithm">algoritmo de firma</Tooltip>) y, opcionalmente, a la API personalizada, si se especificó una.
* El token de identidad se firmará obligatoriamente con RS256 si lo solicita un <Tooltip tip="Cliente público: Cliente (aplicación) que no puede almacenar credenciales de forma segura. Algunos ejemplos son una aplicación nativa de escritorio o móvil y una aplicación web del lado del cliente basada en JavaScript (como una aplicación web de página única (SPA))." cta="Ver glosario" href="/docs/es/glossary?term=public+client">cliente público</Tooltip>.
* Solo se devolverá un <Tooltip tip="Refresh Token: Token utilizado para obtener un Access Token renovado sin obligar a los usuarios a iniciar sesión de nuevo." cta="Ver glosario" href="/docs/es/glossary?term=refresh+token">token de actualización</Tooltip> si se concedió el scope `offline_access` y la API tiene activada la opción **Permitir acceso sin conexión**.

Aquí tienes un ejemplo de la respuesta compatible con OIDC de `/oauth/token`:

```json lines
{
  "access_token": "eyJ...",
  "token_type": "Bearer",
  "refresh_token": "8xLOxBtZp8",
  "expires_in": 3600,
  "id_token": "eyJ..."
}
```

<div id="code-changes-when-using-the-sdks">
  ### Cambios en el código al usar los SDKs
</div>

Si su aplicación utiliza las bibliotecas nativas de Auth0 para Android o iOS, asegúrese de que la versión de la biblioteca que incluye sea, como mínimo, la que se indica a continuación (o una superior). Además, asegúrese de establecer el indicador `OIDC Conformant` en `true` al configurar las bibliotecas.

| Biblioteca   | Versión mínima |
| ------------ | -------------- |
| Android SDK  | 1.2            |
| Lock Android | 2.17           |
| Swift SDK    | 1.20.0         |
| Lock iOS     | 2.14.0         |

<div id="verify-migration">
  ### Verificar la migración
</div>

1. Para verificar si estás usando el endpoint obsoleto, revisa los [registros del tenant](https://manage.auth0.com/#/logs), filtra por **Deprecation Notice** y luego busca registros que digan `oauth/ro` **passwordless: This feature is being deprecated**. También puedes realizar esta búsqueda directamente con la siguiente consulta: `type:depnote AND description:*passwordless*`.
2. Una vez que hayas migrado tu base de código y estés seguro de que tus aplicaciones no están llamando al endpoint, ve al [Dashboard](https://manage.auth0.com/#/tenant/advanced), en **Configuración del tenant &gt; Advanced**.
3. Desplázate hacia abajo hasta **Migración** y desactiva **Legacy** `/oauth/ro` **Endpoint**. Al desactivar este switch, el endpoint obsoleto se inhabilita para tu tenant y se evita que pueda usarse.

Si al desactivar este switch se producen inicios de sesión fallidos, es señal de que todavía no has eliminado por completo todas las instancias de código heredado de tus aplicaciones.

Una vez que las migraciones se hayan realizado correctamente en entornos de producción, el switch puede desactivarse y dejarse así para garantizar que las funciones obsoletas ya no puedan usarse.
