Creemos que el control de versiones es una parte fundamental de nuestra oferta y trabajamos para ofrecer un esquema de control de versiones coherente para nuestros productos, de modo que nuestros usuarios puedan gestionar y prever cómo afectarán nuestros cambios al uso.

<div id="semantic-versioning">
  ## Control de versiones semántico
</div>

El [control de versiones semántico](http://semver.org) (también conocido como semver) es una estrategia de control de versiones cuya principal funcionalidad es hacer que los cambios incompatibles sean fáciles de identificar. Una versión se compone de 3 números separados por puntos: {major}.{minor}.{patch}. Por ejemplo, 2.12.5, 0.1.0 y 10.5.35 son números de semver válidos.

* El primer número representa un **cambio mayor**: la API de la biblioteca ha cambiado de una forma incompatible con versiones anteriores. Cuando se incrementa la parte mayor de una versión, la API pública de esa biblioteca ha cambiado. Por ejemplo, el código y la funcionalidad marcados previamente como obsoletos se eliminan de la base de código.
* El segundo número representa un **cambio menor**: se ha añadido nueva funcionalidad a la API de la biblioteca o se ha marcado como obsoleta, manteniendo la compatibilidad con versiones anteriores. Se espera que la nueva versión menor sea segura de usar y animamos a los clientes a actualizar. Sin embargo, como es imposible conocer todas las formas en que los clientes usan un componente, siempre existe la posibilidad de que los cambios afecten al uso actual del componente. Por lo tanto, recomendamos verificar y probar antes de realizar una actualización.
* El tercer número representa un **cambio de parche**: se ha corregido un error y no debería tener ningún impacto en la API expuesta al usuario. Esto debería ser seguro de actualizar, pero siempre se recomienda hacer pruebas.

<div id="production-usage">
  ## Uso en producción
</div>

Auth0 proporciona enlaces a nuestra [red de distribución de contenido (CDN)](https://en.wikipedia.org/wiki/Content_delivery_network), donde alojamos algunas de nuestras bibliotecas. La forma en que hagas referencia a un componente en tu código influirá en si los cambios se incorporan automáticamente y en qué momento. Por ejemplo, si apuntas a la versión mayor desde un script y se publica una nueva versión menor, recibirás la actualización en cuanto esté disponible. Recomendamos esta práctica para entornos de desarrollo y para experimentar con Auth0. Aquí tienes ejemplos de cómo incluir los archivos fuente:

```html lines
<!-- versión mayor -->
<script src="https://cdn.auth0.com/example/1/library.js"></script>

<!-- versión menor -->
<script src="https://cdn.auth0.com/example/1.0/library.js"></script>

<!-- versión de parche (recomendada para producción) -->
<script src="https://cdn.auth0.com/example/1.0.1/library.js"></script>
```

Cuando los componentes de Auth0 se despliegan en producción, recomendamos a nuestros usuarios fijar una versión completa y probar exhaustivamente con esa versión. Aunque añadir nuevas funcionalidades de forma retrocompatible no debería romper su código, las interacciones con el resultado del componente pueden ser difíciles de predecir. Incluso las correcciones de errores triviales pueden introducir cambios en las suposiciones que estaba haciendo sobre el componente y que puede que ya no sean ciertas.

Cada componente de código abierto de Auth0 que sigue semver tendrá una etiqueta que coincida con una versión publicada en su repositorio de git. Como algunos de los proyectos usan herramientas de [npm](https://npmjs.com) para publicar versiones, las etiquetas llevarán la letra `v` como prefijo. Por ejemplo, la etiqueta de la versión 5.2.3 será v5.2.3.
