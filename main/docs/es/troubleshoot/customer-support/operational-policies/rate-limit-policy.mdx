Auth0 limita el uso de sus servicios para garantizar un rendimiento óptimo y protegerse frente a <Tooltip tip="Actores maliciosos: Entidad (una persona o grupo) que representa una amenaza para la empresa o el entorno con la intención de causar daño." cta="Ver glosario" href="/docs/es/glossary?term=bad+actors">actores maliciosos</Tooltip>, errores técnicos o tráfico legítimo excesivo.  Le recomendamos revisar cómo Auth0 aplica estos límites para configurar su aplicación y ofrecer la mejor experiencia de usuario.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Para consultar los límites de frecuencia asociados a su cuenta, vea [Configuraciones de límites de frecuencia](/docs/es/troubleshoot/customer-support/operational-policies/rate-limit-policy/rate-limit-configurations), donde encontrará una matriz de todas las políticas de límite de frecuencia.
</Callout>

<div id="introduction-to-rate-limits">
  ## Introducción a los límites de frecuencia
</div>

Auth0 aplica limitaciones para proteger los servicios frente a un exceso de solicitudes y proteger a los clientes de interrupciones o degradación del servicio.

Auth0 supervisa y, en muchos casos, aplica una serie de límites, entre ellos:

* Solicitudes a entornos (solo para Private Cloud)
* Solicitudes a tenants mediante la API o un endpoint de API
* Límites varios

<div id="environment-request-limits-private-cloud-only">
  ### Límites de solicitudes del entorno (Solo para Private Cloud)
</div>

En Private Cloud, los límites de solicitudes del entorno se basan en el nivel de rendimiento de Private Cloud.  Para obtener más información, consulta [Private Cloud en AWS](/docs/es/deploy-monitor/deploy-private-cloud/private-cloud-on-aws) o [Private Cloud on Azure](/docs/es/deploy-monitor/deploy-private-cloud/private-cloud-on-azure).

Actualmente,  los límites de frecuencia del entorno de Private Cloud representan la carga máxima bajo la cual el producto Auth0 cumplirá los SLA.  Sin embargo, en este momento, Auth0 solo aplica estos límites y notifica a los clientes cuando se han excedido en un tenant específico dentro del entorno.  En la mayoría de los casos de uso de Private Cloud, en los que los clientes mantienen un único tenant de producción, esto no supone un problema; sin embargo, los casos de uso que aprovisionan más de un tenant de producción deben considerar la carga esperada de todos los tenants del entorno e implementar monitoreo adicional según sea necesario.

| Característica                                      | Authentication API                                                                                                                                                                                                                                                                                           | Todas las demás API (incluida Management API)                                                                                                                                                                                                                                                   |
| --------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Límites de frecuencia del tenant                    | Todos los tenants consumen de un límite global compartido para toda la API en el entorno, pero el límite de frecuencia del entorno es un tope estricto. Superar el límite del entorno debido a la carga combinada entre tenants dará lugar a que las solicitudes excedentes queden limitadas por frecuencia. | Los límites de frecuencia del tenant se aplican de forma independiente, y el límite de frecuencia del entorno actúa como un umbral de posible degradación del rendimiento e impacto en el SLA, pero **no** provoca directamente limitación por frecuencia a nivel de tenant.                    |
| Límite de frecuencia del entorno superado (ejemplo) | Con un límite del entorno de 1500 rps, Tenant 1 con 1400 rps y Tenant 2 con 900 rps (2300 rps en conjunto) darán lugar a que 800 solicitudes queden limitadas por frecuencia.                                                                                                                                | Con un límite del entorno de 1500 rps, Tenant 1 con 1400 rps y Tenant 2 con 900 rps (2300 rps en conjunto) **no** hará que los tenants individuales queden limitados por frecuencia, pero el entorno en general puede experimentar una degradación del rendimiento y posibles problemas de SLA. |

<Warning>
  Superar un límite de frecuencia del entorno invalida el Acuerdo de nivel de servicio (SLA).
</Warning>

Para conocer las consideraciones sobre pruebas de carga de Private Cloud, consulta [Private Cloud en AWS](/docs/es/deploy-monitor/deploy-private-cloud/private-cloud-on-aws) y [Private Cloud on Azure](/docs/es/deploy-monitor/deploy-private-cloud/private-cloud-on-azure).

<div id="tenant-request-limits">
  ### Límites de solicitudes del tenant
</div>

Auth0 limita la cantidad de solicitudes que se pueden realizar en un tenant. Estos límites se configuran en función de la API y, dentro de cada una, de endpoints específicos.

<div id="api-rate-limits">
  #### Límites de frecuencia de la API
</div>

Auth0 limita el número de solicitudes a una API específica, independientemente del endpoint de la API. Los límites de la API pueden variar según:

* API

  * Autenticación
  * Management
* Tipo de tenant (Production vs. Development o Staging)
* Nivel de suscripción (Free, Essential, Professional, Enterprise Public vs. Private)

Por ejemplo, un tenant gratuito de no producción podría tener límites diferentes a los de un tenant de producción con una suscripción de pago. Para consultar configuraciones específicas de límites de frecuencia, consulta [Configuraciones de límites de tasa](/docs/es/troubleshoot/customer-support/operational-policies/rate-limit-policy/rate-limit-configurations).

<div id="user-requests">
  ### Solicitudes de usuario
</div>

Una sola solicitud de un usuario final (p. ej., inicio de sesión o registro) suele generar múltiples solicitudes a los endpoints de la Authentication API. La proporción real entre las solicitudes de los usuarios finales y la Authentication API depende de varios factores:

* Entidad que se autentica (p. ej., máquina o aplicación móvil o de escritorio del usuario final)
* Experiencia de autenticación (p. ej., <Tooltip tip="Universal Login: su aplicación redirige a Universal Login, alojado en el servidor de autorización de Auth0, para verificar la identidad de un usuario." cta="Ver glosario" href="/docs/es/glossary?term=Universal+Login">Universal Login</Tooltip> nuevo o Classic Login)
* Flujo de autenticación (p. ej., inicio de sesión, registro o cambiar la contraseña)
* Tipo de flujo de autenticación (p. ej., inicio de sesión con nombre de usuario / contraseña; inicio de sesión mediante Social Login; inicio de sesión cuando ya existe un token de autenticación)

Los clientes que aprovechan la Extensibilidad podrían añadir aún más solicitudes, no solo a la Authentication API, sino también a la <Tooltip tip="Management API: un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip>, según la configuración de Extensibilidad.
Consulte [Casos de uso del límite de frecuencia](/docs/es/troubleshoot/customer-support/operational-policies/rate-limit-policy/rate-limit-use-cases) para obtener orientación sobre cómo estimar el impacto de su configuración de Auth0 en el uso de la API.

<div id="endpoint-rate-limits">
  #### Límites de frecuencia de los endpoints
</div>

Auth0 limita la cantidad de solicitudes realizadas a los endpoints de API y, en algunos casos, la cantidad de operaciones de los endpoints.  Los límites de los endpoints de API también varían según:

* API
* Tipo de tenant
* Nivel de suscripción

Por ejemplo, un tenant Free de no producción tiene límites diferentes a los de un tenant de producción con una suscripción de pago.

<div id="tenant-request-limits-order">
  ### Orden de aplicación de los límites de solicitudes del tenant
</div>

A medida que se reciben solicitudes en tu tenant, Auth0 las evalúa primero según el límite global de la API y luego según el límite de frecuencia de endpoints específicos de la API.

  ### Otros límites
</div>

<div id="database-login-limits">
  #### Límites de inicio de sesión de base de datos
</div>

En las conexiones de base de datos, Auth0 limita ciertos tipos de intentos repetidos de inicio de sesión según la cuenta de usuario y la dirección IP. Para proteger la estabilidad general del sistema, Auth0 aplica límites de frecuencia por usuario/contraseña para mitigar la carga. El alto grado de personalización de Auth0 puede exponernos al riesgo de degradación del servicio. Algunas de las causas pueden ser:

* Pruebas de estrés de alta carga
* Pruebas de rendimiento
* Código ineficiente que hace que los usuarios inicien sesión varias veces

Las solicitudes están sujetas a límites, según se describe en las políticas específicas de las API de Auth0.

Además, existe un límite de frecuencia de inicio de sesión para un mismo usuario: si una dirección IP realiza 20 intentos de inicio de sesión en un minuto en la misma cuenta de usuario, el límite de frecuencia entra en vigor. A partir de ese momento, Auth0 permite al usuario 10 intentos por minuto. Cualquier combinación de intentos de inicio de sesión correctos y fallidos cuenta para este límite.

<Card title="Límites que protegen a los usuarios">
  La [protección contra ataques de fuerza bruta](/docs/es/secure/attack-protection/brute-force-protection) y la [limitación de IP sospechosas](/docs/es/secure/attack-protection/suspicious-ip-throttling) de Auth0 también pueden limitar los inicios de sesión y los registros, pero son independientes de los límites de frecuencia. Para obtener más información sobre cómo Auth0 detecta y maneja anomalías potencialmente maliciosas, consulta [Protección contra ataques](/docs/es/secure/attack-protection).
</Card>

<div id="sms-message-limits-for-multi-factor-authentication-end-users-only">
  #### Límites de mensajes SMS para la autenticación multifactor (solo para usuarios finales)
</div>

Si intentas enviar más de 10 mensajes SMS a tu dispositivo en el plazo de una hora, recibirás un mensaje de error por exceder el límite de frecuencia.

Cuando superes tu límite de mensajes, deberás esperar al menos una hora desde la primera solicitud de mensaje antes de poder pedir otro. Recibirás un intento adicional por cada hora adicional que transcurra.

<div id="native-social-login-limits">
  #### Límites del inicio de sesión social nativo
</div>

Los límites que se aplican a las solicitudes del flujo de inicio de sesión social nativo se identifican según el cuerpo de las solicitudes, con los siguientes criterios iniciales:

| Tipo de solicitud    | Cuerpo                                               |
| -------------------- | ---------------------------------------------------- |
| `grant_type`         | `urn:ietf:params:oauth:grant-type:token-exchange`    |
| `subject_token_type` | `http://auth0.com/oauth/token-type/apple-authz-code` |

<div id="public-performance-burst">
  #### Public Performance Burst
</div>

La oferta Public Performance Burst es un add-on disponible para suscripciones empresariales que mejora un despliegue existente de Public Cloud. Esta oferta permite un aumento dinámico del límite de requests de Authentication API hasta un multiplicador del límite predeterminado de requests empresariales de 100 RPS durante un máximo de 48 horas al mes.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Este add-on solo amplía los límites de requests de Authentication API y NO se aplica a Management API ni a otros endpoints que tienen límites de frecuencia fuera del alcance de Authentication API.
</Callout>

Actualmente, hay tres modificadores de Public Performance Burst (2x, 3x y 4x) que permiten 200, 300 y 400 RPS, respectivamente, para Authentication API durante un máximo de 48 horas al mes. Estas 48 horas se contabilizan y descuentan de la cuota mensual permitida en intervalos de 1 minuto, lo que permite usar la API en el nivel del multiplicador durante 2880 intervalos de 1 minuto cada mes.

Cuando el volumen de requests para Authentication API supera el valor predeterminado de 100 RPS, se descuenta un intervalo de 5 minutos de la asignación mensual y se permite tráfico a la tasa asociada con el multiplicador. El tráfico puede mantenerse dentro del rango del multiplicador durante los 5 minutos continuos completos a partir de ese momento sin generar descuentos adicionales.

Es posible monitorear los eventos de descuento de intervalos a través de los [logs del tenant](/docs/es/deploy-monitor/logs/log-event-type-codes). Cada descuento genera un evento de log del tenant `appi` asociado que contiene información sobre la asignación ya consumida y la restante.

Para obtener más información, consulta la sección de Authentication API en [Límites de frecuencia - Empresarial](/docs/es/troubleshoot/customer-support/operational-policies/rate-limit-policy/rate-limit-configurations/enterprise-public).

<div id="private-performance-burst">
  #### Private Performance Burst
</div>

La oferta Private Performance Burst (disponible actualmente en AWS para los niveles 30x y 60x) incluye una capacidad de rendimiento en ráfaga (pico) de hasta 30x (3.000 RPS) o 60x (6.000 RPS) durante un máximo de 80 horas al mes. La capacidad de rendimiento base, que equivale a la mitad de la capacidad de rendimiento en ráfaga, está disponible durante el resto del mes.

En otras palabras, Private Performance Burst 30x tiene:

* Capacidad base: 1.500 RPS durante todo el mes
* Capacidad de ráfaga/pico: 3.000 RPS durante un máximo de 80 horas al mes

Si las transacciones de autenticación superan el límite base de solicitudes de API, se descuenta una hora de la asignación mensual. El tráfico puede mantenerse a esa tasa elevada durante la hora completa e ininterrumpida a partir de ese momento. Se aplican deducciones adicionales de la misma forma cada vez que se vuelve a superar el umbral.

Una vez agotada la asignación de 80 horas, el tráfico de autenticación quedará sujeto a un límite de frecuencia correspondiente a la capacidad base hasta que entre en vigor la nueva asignación mensual.

<div id="extensibility-concurrency-limits">
  #### Límites de concurrencia de la extensibilidad
</div>

Para garantizar la disponibilidad del sistema y el uso justo de sus recursos, Auth0 limita la cantidad de solicitudes simultáneas en curso en todos los productos de extensibilidad: Actions, Hooks, Rules, Custom Database Connections, Extensions y conexiones OAuth2 personalizadas. Los tenants que superen sus límites de solicitudes simultáneas pueden esperar errores en las nuevas solicitudes hasta que se completen las que están en curso. Los límites de concurrencia se definen a continuación:

| Suscripción                                                             | Límite de concurrencia (por Tenant) |
| ----------------------------------------------------------------------- | ----------------------------------- |
| Public Cloud                                                            | 250                                 |
| Tier Dev Private Cloud                                                  | 100                                 |
| Private Cloud Basic 100 RPS (1x)                                        | 200                                 |
| Private Cloud Performance 500 RPS (5x)                                  | 400                                 |
| Private Cloud Performance 1500 RPS (15x)                                | 1200                                |
| Private Cloud Performance 3000 RPS (30x) and 3000 RPS Burst (30x Burst) | 1200                                |
| Private Cloud Performance 6000 RPS (60x) and 6000 RPS Burst (60x Burst) | 1200                                |

La concurrencia puede calcularse multiplicando los RPS esperados por la latencia de cada solicitud. Por ejemplo, un tenant con dos Actions de Post Login vinculadas, que tardan 250 ms cada una, y con un total de 400 RPS de inicio de sesión, tiene una concurrencia esperada de (2 * (400 requests / 1 second) * (.25 seconds / 1 request)) = 200.

Para asegurarse de que su tenant no se vea afectado por estos límites de concurrencia, asegúrese de que la lógica de extensibilidad potencialmente de larga duración, como las llamadas a API externas, tenga timeouts razonables.

<div id="rate-limit-algorithm">
  ## Algoritmo de límite de frecuencia
</div>

Auth0 establece límites de frecuencia y límites de ráfaga para sus API. Mientras que el límite de frecuencia es la cantidad máxima de tráfico sostenido que el sistema permite de forma continua, el límite de ráfaga es el volumen máximo de tráfico a corto plazo que el sistema permite dentro de un intervalo de tiempo. Los límites de frecuencia y los límites de ráfaga de Auth0 funcionan conjuntamente para ofrecer una limitación más eficaz en volúmenes de tráfico dinámicos.

Los límites de frecuencia de Auth0 usan un algoritmo de bucket de tokens con las siguientes configuraciones:

* **Claves de límite:**

  * Normalmente, una clave de límite de frecuencia se basa en dos factores principales:

    * API y endpoint
    * Tipo de tenant
  * En algunos casos, también se incluyen factores adicionales:

    * IP de origen
    * ID del usuario de destino
* **Valores de límite:**

  * **Tamaño del bucket:** El número máximo de solicitudes que una API o un endpoint recibe en general, o de un usuario o una dirección IP específicos, antes de que se agreguen nuevas solicitudes.
  * **Tasa de recarga:** La tasa a la que se agregan nuevas solicitudes al bucket.

A partir de estos dos valores, Auth0 calcula el límite de ráfaga y el límite de frecuencia sostenida:

* **Límite de ráfaga:**  Igual al tamaño del bucket.
* **Límite de frecuencia sostenida:** Tasa de recarga en solicitudes por minuto o por segundo.

Si el límite de frecuencia sostenida se calcula en solicitudes por segundo, las nuevas solicitudes se agregan por milisegundo. Si el límite de frecuencia sostenida se calcula en solicitudes por minuto, las nuevas solicitudes se agregan por segundo.
