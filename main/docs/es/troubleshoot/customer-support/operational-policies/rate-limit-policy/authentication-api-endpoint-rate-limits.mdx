title: Política de límites de tasa
---

Auth0 limita el uso de sus servicios para garantizar un rendimiento óptimo y protegerse frente a <Tooltip tip="Actores maliciosos: entidad (una persona o grupo) que representa una amenaza para la empresa o el entorno con la intención de causar daño." cta="Ver glosario" href="/docs/es/glossary?term=bad+actors">actores maliciosos</Tooltip>, errores técnicos o tráfico legítimo excesivo.  Le recomendamos revisar cómo Auth0 aplica estos límites para configurar su aplicación y ofrecer la mejor experiencia de usuario.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Para consultar los límites de tasa asociados a su cuenta, vea [Configuraciones de límites de tasa](/docs/es/troubleshoot/customer-support/operational-policies/rate-limit-policy/rate-limit-configurations), donde encontrará una matriz con todas las políticas de límites de tasa.
</Callout>

<div id="introduction-to-rate-limits">
  ## Introducción a los límites de velocidad
</div>

Auth0 aplica limitaciones para proteger los servicios de solicitudes excesivas y a los clientes de interrupciones o degradación del servicio.

Auth0 supervisa y, en muchos casos, aplica una serie de límites, entre ellos:

* Solicitudes a entornos (solo en Private Cloud)
* Solicitudes a tenants a través de la API o de un endpoint de API
* Límites diversos

<div id="environment-request-limits-private-cloud-only">
  ### Límites de solicitudes del entorno (solo para Private Cloud)
</div>

En Private Cloud, los límites de solicitudes del entorno se basan en el nivel de rendimiento de Private Cloud. Para obtener más información, consulta [Private Cloud for AWS](/docs/es/deploy-monitor/deploy-private-cloud/private-cloud-on-aws) o [Private Cloud for Azure](/docs/es/deploy-monitor/deploy-private-cloud/private-cloud-on-azure).

Actualmente, los rate limits del entorno de Private Cloud representan la carga máxima bajo la cual el producto Auth0 cumplirá los SLA. Sin embargo, por el momento, Auth0 solo aplica estos límites y notifica a los clientes cuando se superan los rate limits en un tenant específico dentro del entorno. En la mayoría de los casos de uso de Private Cloud, en los que los clientes mantienen un único Production Tenant, esto no supone un problema; sin embargo, los casos de uso que aprovisionan más de un tenant de producción deben tener en cuenta la carga prevista en todos los tenants del entorno e implementar supervisión adicional según sea necesario.

| Funcionalidad                             | Authentication API                                                                                                                                                                                                                                                                                     | Todas las demás API (incluida Management API)                                                                                                                                                                                                                                             |
| ----------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Rate limits del tenant                    | Todos los tenants consumen de un límite global compartido de la API para todo el entorno, pero el rate limit del entorno es un límite estricto. Superar el límite del entorno debido a la carga combinada entre tenants hará que las solicitudes que excedan el límite queden sujetas a rate limiting. | Los rate limits del tenant se aplican de forma independiente, y el rate limit del entorno actúa como umbral de una posible degradación del rendimiento y del impacto en el SLA, pero **no** provoca directamente rate limiting a nivel de tenant.                                         |
| Rate limit del entorno superado (ejemplo) | Con un límite de entorno de 1500 rps, el Tenant 1 con 1400 rps y el Tenant 2 con 900 rps (2300 rps en total) harán que 800 solicitudes queden sujetas a rate limiting.                                                                                                                                 | Con un límite de entorno de 1500 rps, el Tenant 1 con 1400 rps y el Tenant 2 con 900 rps (2300 rps en total) **no** harán que se apliquen rate limits a tenants individuales, pero el entorno general puede experimentar una degradación del rendimiento y posibles problemas con el SLA. |

<Warning>
  Superar un rate limit del entorno invalida el Service Level Agreement (SLA).
</Warning>

Para conocer las consideraciones sobre las pruebas de carga en Private Cloud, consulta [Private Cloud on AWS](/docs/es/deploy-monitor/deploy-private-cloud/private-cloud-on-aws) y [Private Cloud on Azure](/docs/es/deploy-monitor/deploy-private-cloud/private-cloud-on-azure).

<div id="tenant-request-limits">
  ### Límites de solicitudes del tenant
</div>

Auth0 limita la cantidad de solicitudes que se pueden realizar en un tenant. Estos límites se configuran según la API y también según endpoints específicos de cada API.

<div id="api-rate-limits">
  #### Límites de solicitudes de la API
</div>

Auth0 limita la cantidad de solicitudes que se pueden hacer a una API específica, independientemente del endpoint de la API.  Los límites de la API pueden variar según:

* API

  * Autenticación
  * Administración
* Tipo de tenant (Producción vs. Desarrollo o Staging)
* Nivel de suscripción (Free, Essential, Professional, Enterprise Public vs. Private)

Por ejemplo, un tenant gratuito de no producción podría tener límites diferentes a los de un tenant de producción con una suscripción de pago. Para ver configuraciones específicas de límites de solicitudes, consulta [Configuraciones de límites de solicitudes](/docs/es/troubleshoot/customer-support/operational-policies/rate-limit-policy/rate-limit-configurations).

<div id="user-requests">
  ### Solicitudes de usuario
</div>

Una sola solicitud de un usuario final (p. ej., Login o Signup) normalmente inicia múltiples solicitudes a los endpoints de Authentication API. La proporción real entre las solicitudes del usuario final y Authentication API depende de varios factores:

* Entidad que se autentica (p. ej., máquina o aplicación móvil o de escritorio del usuario final)
* Experiencia de autenticación (p. ej., nuevo <Tooltip tip="Universal Login: Su aplicación redirige a Universal Login, alojado en el servidor de autorización de Auth0, para verificar la identidad de un usuario." cta="Ver glosario" href="/docs/es/glossary?term=Universal+Login">Universal Login</Tooltip> o Classic Login)
* Flujo de autenticación (p. ej., Login, Signup o cambiar la contraseña)
* Tipo de flujo de autenticación (p. ej., Login con Username / Password; Login con Social Login; Login cuando ya existe un token de autenticación)

Los clientes que usan Extensibility podrían agregar aún más solicitudes, no solo a Authentication API, sino también a <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip>, según la configuración de Extensibility.
Consulte [Rate Limit Use Cases](/docs/es/troubleshoot/customer-support/operational-policies/rate-limit-policy/rate-limit-use-cases) para obtener orientación sobre cómo estimar de qué manera su configuración de Auth0 puede afectar el uso que hace de la API.

<div id="endpoint-rate-limits">
  #### Límites de tasa por endpoint
</div>

Auth0 limita la cantidad de solicitudes que se pueden hacer a los endpoints de la API y, en algunos casos, la cantidad de operaciones por endpoint.  Los límites de los endpoints de la API también varían según:

* API
* Tipo de tenant
* Nivel de suscripción

Por ejemplo, un tenant gratuito no destinado a producción tiene límites distintos de los de un tenant de producción con una suscripción de pago.

<div id="tenant-request-limits-order">
  ### Orden de aplicación de los límites de solicitudes del tenant
</div>

A medida que se realizan solicitudes a tu tenant, Auth0 primero las evalúa según el límite global de la API y luego según el límite de solicitudes de endpoints específicos de la API.

<div id="miscellaneous-limits">
  ### Otros límites
</div>

<div id="database-login-limits">
  #### Límites de inicio de sesión en conexiones de base de datos
</div>

Para las conexiones de base de datos, Auth0 limita ciertos tipos de intentos repetidos de inicio de sesión en función de la cuenta de usuario y la dirección IP. Para proteger la estabilidad general del sistema, Auth0 aplica límites de frecuencia por usuario/contraseña para mitigar la carga. El alto grado de personalización de Auth0 puede exponernos al riesgo de degradación del servicio. Entre las causas se incluyen:

* Pruebas de estrés con alta carga
* Pruebas de rendimiento
* Código ineficiente que hace que los usuarios inicien sesión varias veces

Las solicitudes están sujetas a límites, tal como se describe en las políticas individuales de las API de Auth0.

Además, existe un límite de frecuencia de inicio de sesión para un mismo usuario: si una dirección IP realiza 20 intentos de inicio de sesión en un minuto sobre la misma cuenta de usuario, el límite de frecuencia entra en vigor. Después de eso, Auth0 permite al usuario 10 intentos por minuto. Cualquier combinación de intentos de inicio de sesión correctos e intentos fallidos de inicio de sesión cuenta para este límite.

<Card title="Límites que protegen a los usuarios">
  La [protección contra ataques de fuerza bruta](/docs/es/secure/attack-protection/brute-force-protection) y la [limitación de IP sospechosas](/docs/es/secure/attack-protection/suspicious-ip-throttling) de Auth0 también pueden limitar los inicios de sesión y los registros, pero son independientes de los límites de frecuencia. Para obtener más información sobre cómo Auth0 detecta y gestiona anomalías potencialmente maliciosas, consulta [Protección contra ataques](/docs/es/secure/attack-protection).
</Card>

<div id="sms-message-limits-for-multi-factor-authentication-end-users-only">
  #### Límites de mensajes SMS para la autenticación multifactor (solo usuarios finales)
</div>

Si intenta enviar más de 10 mensajes SMS a su dispositivo en el transcurso de una hora, recibirá un mensaje de error indicando que se ha superado un límite de frecuencia.

Cuando supere el límite de mensajes, deberá esperar al menos una hora desde la primera solicitud de mensaje antes de pedir otro. Recibirá un intento adicional por cada hora adicional que transcurra.

<div id="native-social-login-limits">
  #### Límites del inicio de sesión social nativo
</div>

Los límites que se aplican a las solicitudes del flujo de inicio de sesión social nativo se determinan según el cuerpo de las solicitudes, con los siguientes criterios iniciales:

| Tipo de solicitud    | Cuerpo                                               |
| -------------------- | ---------------------------------------------------- |
| `grant_type`         | `urn:ietf:params:oauth:grant-type:token-exchange`    |
| `subject_token_type` | `http://auth0.com/oauth/token-type/apple-authz-code` |

<div id="public-performance-burst">
  #### Public Performance Burst
</div>

La oferta Public Performance Burst es un add-on disponible para suscripciones Enterprise que mejora una implementación existente de Public Cloud. Esta oferta permite aumentar dinámicamente el límite de requests de Authentication API hasta un multiplicador del límite de requests Enterprise predeterminado de 100 RPS durante un máximo de 48 horas al mes.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Este add-on solo amplía los límites de requests de Authentication API y NO se aplica a Management API ni a otros endpoints sujetos a rate limits fuera del scope de Authentication API.
</Callout>

Actualmente, hay tres modificadores de Public Performance Burst (2x, 3x y 4x) que permiten 200, 300 y 400 RPS, respectivamente, para Authentication API durante un máximo de 48 horas al mes. Estas 48 horas se contabilizan y descuentan de la cuota mensual permitida en intervalos de 5 minutos, lo que permite usar la API al nivel del multiplicador durante 576 intervalos de 5 minutos al mes.

Cuando el volumen de requests de Authentication API supera el valor predeterminado de 100 RPS, se descuenta un intervalo de 5 minutos de la cuota mensual y se permite tráfico a la tasa asociada con el multiplicador. El tráfico puede mantenerse dentro del rango del multiplicador durante los 5 minutos continuos completos a partir de ese momento sin generar descuentos adicionales.

Es posible monitorear los eventos de descuento de intervalos mediante los [logs del tenant](/docs/es/deploy-monitor/logs/log-event-type-codes). Cada descuento genera un tipo de evento de log del tenant `appi` asociado que contiene información sobre la cuota ya consumida y la restante.

Para obtener más información, consulta la sección de Authentication API en [Rate Limits - Enterprise](/docs/es/troubleshoot/customer-support/operational-policies/rate-limit-policy/rate-limit-configurations/enterprise-public).

<div id="private-performance-burst">
  #### Private Performance Burst
</div>

La oferta Private Performance Burst (disponible actualmente en AWS para los niveles 30x y 60x) incluye una capacidad de rendimiento en ráfaga (pico) de hasta 30x (3.000 RPS) o 60x (6.000 RPS) durante un máximo de 80 horas al mes. La capacidad de rendimiento base, que equivale a la mitad de la capacidad de rendimiento en ráfaga, está disponible durante el resto del mes.

En otras palabras, Private Performance Burst 30x tiene:

* Capacidad base: 1.500 RPS durante todo el mes
* Capacidad en ráfaga/pico: 3.000 RPS durante un máximo de 80 horas al mes

Si las transacciones de autenticación superan el límite base de solicitudes de la API, se descuenta una hora de la asignación mensual. A partir de ese momento, el tráfico puede mantenerse a una tasa elevada durante toda esa hora consecutiva. Se aplican descuentos adicionales de la misma manera cada vez que se vuelve a superar el umbral.

Una vez agotada la asignación de 80 horas, el tráfico de autenticación quedará limitado a la capacidad base hasta que entre en vigor la nueva asignación mensual.

<div id="extensibility-concurrency-limits">
  #### Límites de concurrencia de extensibilidad
</div>

Para garantizar la disponibilidad del sistema y el uso equitativo de sus recursos, Auth0 limita la cantidad de solicitudes simultáneas en curso en todos los productos de extensibilidad: Actions, Hooks, Rules, Custom Database Connections, Extensions y conexiones OAuth2 personalizadas. Los tenants que superen sus límites de solicitudes simultáneas pueden esperar errores en las nuevas solicitudes hasta que se completen las solicitudes en curso. Los límites de concurrencia se definen a continuación:

| Suscripción                                                          | Límite de concurrencia (por tenant) |
| -------------------------------------------------------------------- | ----------------------------------- |
| Nube pública                                                         | 250                                 |
| Nube privada Tier Dev                                                | 100                                 |
| Nube privada Basic 100 RPS (1x)                                      | 200                                 |
| Nube privada Performance 500 RPS (5x)                                | 400                                 |
| Nube privada Performance 1500 RPS (15x)                              | 1200                                |
| Nube privada Performance 3000 RPS (30x) y 3000 RPS Burst (30x Burst) | 1200                                |
| Nube privada Performance 6000 RPS (60x) y 6000 RPS Burst (60x Burst) | 1200                                |

La concurrencia puede calcularse multiplicando los RPS esperados por la latencia de cada solicitud. Por ejemplo, un tenant que tiene dos post-login Actions asociadas, cada una con una duración de 250 ms, y un RPS total de inicio de sesión de 400, tendrá una concurrencia esperada de (2 * (400 requests / 1 second) * (.25 seconds / 1 request)) = 200.

Para asegurarse de que su tenant no se vea afectado por estos límites de concurrencia, procure que la lógica de extensibilidad potencialmente de larga duración, como las llamadas a API externas, tenga tiempos de espera razonables.

<div id="rate-limit-algorithm">
  ## Algoritmo de limitación de tasa
</div>

Auth0 establece límites de tasa y límites de ráfaga para sus API. Mientras que el límite de tasa es la cantidad máxima de tráfico sostenido que el sistema permite de forma continua, el límite de ráfaga es el volumen máximo de tráfico a corto plazo que el sistema permite dentro de un intervalo de tiempo. Los límites de tasa y los límites de ráfaga de Auth0 funcionan conjuntamente para ofrecer una mejor limitación en volúmenes de tráfico dinámicos.

Los límites de tasa de Auth0 utilizan un algoritmo de bucket de tokens con las siguientes configuraciones:

* **Claves de límite:**

  * Normalmente, una clave de límite de tasa se basa en dos factores principales:

    * API y endpoint
    * Tipo de tenant
  * En algunos casos, los factores adicionales incluyen:

    * IP de origen
    * ID del usuario de destino
* **Valores de límite:**

  * **Tamaño del bucket:** El número máximo de solicitudes que una API o un endpoint recibe en general, o que recibe de un usuario o una dirección IP específicos, antes de que se añadan nuevas solicitudes.
  * **Tasa de reposición:** La velocidad a la que se añaden nuevas solicitudes al bucket.

A partir de estos dos valores, Auth0 calcula el límite de ráfaga y el límite de tasa sostenido:

* **Límite de ráfaga:**  Igual al tamaño del bucket.
* **Límite de tasa sostenido:** Tasa de reposición en solicitudes por minuto o por segundo.

Si el límite de tasa sostenido se calcula en solicitudes por segundo, las nuevas solicitudes se añaden en milisegundos. Si el límite de tasa sostenido se calcula en solicitudes por minuto, las nuevas solicitudes se añaden por segundos.
