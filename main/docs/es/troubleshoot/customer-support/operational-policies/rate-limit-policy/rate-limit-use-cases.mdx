<div id="discover-when-requests-to-a-tenant-are-rate-limited">
  ## Cómo saber cuándo las solicitudes a un tenant están sujetas a límite de frecuencia
</div>

Hay varias formas de determinar si Auth0 está aplicando un límite de frecuencia a un producto del cliente. Consulta a continuación las posibles causas de la limitación por frecuencia.

<div id="tenant-logs">
  ### Registros del tenant
</div>

Al suscribirte a distintos registros del tenant, puedes hacer seguimiento de los problemas relacionados con el volumen de solicitudes. Para entender cómo funcionan en Auth0 los registros de eventos del tenant y de operaciones, consulta [Registros](/docs/es/deploy-monitor/logs).

<div id="api_limit">
  #### api_limit
</div>

El evento `api_limit` se activa inmediatamente después de que se supera un límite de frecuencia en el bucket global de límite de frecuencia de Authentication o de la <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip>. Si la cantidad de tokens de solicitud utilizados sigue por encima del 80 % después de un minuto para el mismo bucket de límite de frecuencia, se genera un segundo registro de advertencia. Si se supera un límite de frecuencia en un bucket de límite de frecuencia diferente, se genera un nuevo evento `api_limit`. Esto ayuda a los clientes a identificar qué configuración de límite de frecuencia están alcanzando sus llamadas a la API, lo que constituye un primer paso fundamental para diagnosticar la causa raíz.

<div id="api_limit_warning">
  #### api_limit_warning
</div>

El registro `api_limit_warning` se activa cuando la tasa de solicitudes de un cliente consume el 80 % de los tokens de solicitud de un bucket de límite de frecuencia determinado. Si la cantidad de tokens de solicitud utilizados se mantiene por encima del 80 % después de un minuto para el mismo bucket de límite de frecuencia, se genera un segundo registro de advertencia. Si se supera el umbral del 80 % para un bucket de límite de frecuencia diferente, se crea un nuevo registro `api_limit_warning`.

<div id="appi-public-performance-burst-only">
  #### appi (Solo para Public Performance Burst)
</div>

El registro `appi` se activa cuando el tenant de un cliente con el add-on Public Performance Burst supera su límite sostenido de solicitudes de 100 RPS para la Authentication API, consumiendo un bloque de un minuto de su asignación de ráfaga de 48 horas. Si, después de 15 minutos, la frecuencia de solicitudes vuelve a superar el límite sostenido de solicitudes de 100 RPS, se activa un segundo registro `appi`.

<div id="api-responses">
  ### Respuestas de la API
</div>

La Auth0 API devuelve respuestas [HTTP 429 (Demasiadas solicitudes)](http://tools.ietf.org/html/rfc6585#section-4) con el límite de frecuencia excedido. Esto permite a los clientes ver en tiempo real cuándo se aplica el límite de frecuencia. Sin embargo, esto solo resulta útil para aplicaciones personalizadas de los clientes que interactúan directamente con la Auth0 API.

<div id="sdk-error-handling">
  ### Gestión de errores del SDK
</div>

Si utiliza un SDK, consulte las páginas de errores de las [bibliotecas del SDK de la Management API](/docs/es/libraries#mgmt).

<div id="error-pages">
  ### Páginas de error
</div>

La respuesta de la página de error se envía para los endpoints que renderizan contenido HTML para el usuario final. Si su tenant está configurado para usar páginas genéricas (alojadas por Auth0), Auth0 renderiza la página de error en lugar del contenido esperado cuando se supera el límite de respuesta. Si su tenant está configurado para usar [páginas de error personalizadas](/docs/es/customize/login-pages/custom-error-pages), se redirige al usuario a la URL de la página de error personalizada con el error correspondiente en el parámetro de la query string `error_description`. Para obtener más información, consulte las descripciones de [Endpoints afectados](/docs/es/troubleshoot/customer-support/operational-policies/rate-limit-policy/authentication-api-endpoint-rate-limits#affected-endpoints) y [JSON Error](/docs/es/troubleshoot/customer-support/operational-policies/rate-limit-policy/authentication-api-endpoint-rate-limits#json-error).

<div id="find-out-why-a-tenant-is-being-rate-limited">
  ## Descubra por qué se está aplicando el límite de frecuencia a un tenant
</div>

Si cree que las solicitudes del tenant están sujetas al límite de frecuencia y necesita ayuda para entender por qué, abra una solicitud a través del [Support Center](http://support.auth0.com).  Como parte de su solicitud, incluya el registro completo sin procesar en el que se observó el problema.

<div id="predict-when-requests-to-a-tenant-will-be-rate-limited">
  ## Prediga cuándo las solicitudes a un tenant estarán sujetas a límites de frecuencia
</div>

Auth0 informa datos actualizados sobre el status actual de sus límites de frecuencia mediante encabezados de respuesta HTTP de endpoints que tienen policies de límite de frecuencia configuradas. Este status se comunica de la siguiente manera:

* `x-ratelimit-limit`: Número máximo de solicitudes disponibles.
* `x-ratelimit-remaining`: Número de solicitudes restantes disponibles hasta que el bucket se vuelva a llenar con solicitudes adicionales.
* `x-ratelimit-reset`: [marca de tiempo UNIX](https://en.wikipedia.org/wiki/Unix_time), en segundos, de la hora estimada en la que se agregarán solicitudes adicionales al bucket.

Por ejemplo:

Una API tiene el siguiente límite de frecuencia:

* Burst Limit: `1000`
* Límite de frecuencia sostenido: `100` `solicitudes por segundo` (en una ventana fija)

A partir de esta información, puede deducir lo siguiente:

* El límite de frecuencia sostenido es de `100 solicitudes por segundo` en una ventana fija.
* Debido a la ventana fija, el bucket de solicitudes se vuelve a llenar cada segundo.

Si recibe los siguientes encabezados `x-` en la respuesta de su API:

* `x-ratelimit-limit: 1000`
* `x-ratelimit-remaining: 50`
* `x-ratelimit-reset: 1675452600`

Ahora sabe que:

* Su tenant ha consumido 950 de las 1000 solicitudes permitidas para esa API, y solo le quedan 50 solicitudes hasta que se agreguen solicitudes adicionales.
* Se agregarán nuevas solicitudes en `1675452600`, o a las 7:30:00 p. m. UTC del 3 de febrero de 2023.
* Se agregará 1 nueva solicitud en ese momento

Por lo tanto, si está haciendo solicitudes a una tasa mayor que la descrita arriba, es de esperar que se aplique una limitación de frecuencia. La rapidez con la que estará sujeto a límites de frecuencia depende del Burst Limit y de cuánto esté superando el límite de frecuencia sostenido.

<div id="examples-of-how-rate-limits-are-enforced">
  ## Ejemplos de cómo se aplican los límites de frecuencia
</div>

<div id="requests-per-second-example">
  ### Ejemplo de solicitudes por segundo
</div>

Supongamos que Auth0 lanza una nueva API llamada `/ratelimitexample` con los siguientes valores de límite de frecuencia:

* Límite de ráfaga: cinco (5) solicitudes
* Límite de frecuencia sostenido: 10 solicitudes por segundo.

Puntos clave:

* La API comienza con cinco tokens de solicitud y nunca superará esa cantidad, que equivale al límite de ráfaga.
* El bucket de 10 tokens se rellena cada segundo mediante una “ventana fija”. Se agregan nuevos tokens al bucket, rellenándolo al “inicio” de cada segundo.

 Escenario de ejemplo con límites de frecuencia:

<Frame>![](/docs/images/cdy7uua7fh8z/30m5xST81Db6mROVGCV5Bj/34d9638eb7467f02c2d0ecbe87ff0fd8/Examples_of_how_rate_limits_are_enforced_-_Page_1.png)</Frame>

En este escenario:

* T0 - T1sec:  El usuario final realiza seis solicitudes en el primer segundo.  Cinco solicitudes, equivalentes al límite de ráfaga, reciben una respuesta `200`.  La sexta solicitud recibe un error `429` porque no quedan tokens de solicitud en el bucket.
* T1sec - T2sec:  Auth0 vuelve a llenar por completo el bucket de tokens de solicitud debido al algoritmo de ventana fija. Como resultado, las solicitudes séptima a undécima se procesan correctamente, agotando el bucket con la duodécima solicitud, lo que da como resultado un error `429`.
* T2sec - T3sec:  Auth0 vuelve a rellenar el bucket de tokens una vez más, y la siguiente solicitud (13) recibe una respuesta `200`.

<div id="requests-per-minute-example">
  ### Ejemplo de solicitudes por minuto
</div>

Supongamos que Auth0 lanza una nueva API llamada `/ratelimitexample2` con los siguientes valores de límite de frecuencia:

* Límite de ráfaga:  Cinco (5) solicitudes
* Límite de frecuencia sostenido:  Seis (6) solicitudes por minuto.

Puntos clave:

* La API comienza con cinco tokens de solicitud, lo que equivale al límite de ráfaga.
* El bucket de seis tokens se rellena cada minuto mediante una “ventana fija”. Se agregan nuevos tokens al bucket y se rellena al inicio de cada minuto.

Ejemplo de escenario con límites de frecuencia:

<Frame>![](/docs/images/cdy7uua7fh8z/PhtwGBwS9PfEpNQbPeA1o/dc2b05c72ef960a19958dbafa0295e70/Examples_of_how_rate_limits_are_enforced_-_Page_1__1_.png)</Frame>

En este escenario:

* T0 - T+1min:  El usuario final realiza seis solicitudes en el primer minuto. Cinco solicitudes, equivalentes al límite de ráfaga, reciben una respuesta `200`.  La sexta solicitud recibe un error `429` porque no quedan tokens de solicitud.
* T+1min - T+2min:  Auth0 vuelve a rellenar el bucket de tokens debido al algoritmo de ventana fija. Como resultado, de la séptima a la undécima solicitudes se procesan correctamente, y el bucket se agota con la duodécima solicitud, lo que da lugar a un error `429`.
* T+2min: Auth0 vuelve a rellenar el bucket de tokens una vez más, y la siguiente solicitud (13) recibe una respuesta `200`.

<div id="other-scenarios">
  ### Otros escenarios
</div>

En ocasiones, Auth0 asigna dos límites de frecuencia a una sola API.  Esto se hace para configurar un límite de ráfaga y un límite de frecuencia sostenido más adaptados a las necesidades del servicio.  En la práctica, el primer límite de frecuencia se convierte en el límite de ráfaga efectivo, y el segundo límite de frecuencia se convierte en el límite de frecuencia sostenido efectivo.  En este escenario, Auth0 solo publica los límites efectivos de ráfaga y de frecuencia sostenida, en lugar de comunicar los límites de ráfaga y de frecuencia sostenida reales.

<div id="end-user-login-and-signup-api-usage">
  ## Uso de la API para el login y el registro del usuario final
</div>

Hay varios flujos de autenticación, entre ellos Login, Signup y Change Password. Los más comunes suelen ser Login, seguido de Signup.

El login de un usuario final inicia múltiples llamadas a la API a endpoints de Authentication API para determinar si el usuario final está autorizado para recibir un token de autorización y, de ese modo, acceder a la aplicación que ha solicitado.

La cantidad exacta de llamadas a la API depende de varias configuraciones:

* Experiencia de autenticación (p. ej., New <Tooltip tip="Universal Login: Su aplicación redirige a Universal Login, alojado en el servidor de autorización de Auth0, para verificar la identidad de un usuario." cta="Ver glosario" href="/docs/es/glossary?term=Universal+Login">Universal Login</Tooltip> o Classic Login)
* Flujo de autenticación (p. ej., Login, Signup o Change Password)
* Tipo de flujo de autenticación (p. ej., Login con Username / Password; Login con Social Login; Login cuando ya existe un Authentication Token)

A continuación, describimos algunas configuraciones habituales de los clientes y su impacto en el uso de la API.

<div id="universal-login">
  ### Universal Login
</div>

[Auth0 Universal Login](/docs/es/authenticate/login/auth0-universal-login) ofrece una funcionalidad esencial de un <Tooltip tip="Servidor de autorización: Servidor centralizado que contribuye a definir los límites del acceso de un usuario. Por ejemplo, tu servidor de autorización puede controlar los datos, las tareas y las funcionalidades disponibles para un usuario." cta="Ver glosario" href="/docs/es/glossary?term=authorization+server">servidor de autorización</Tooltip>: el flujo de inicio de sesión. Cuando un usuario necesita demostrar su identidad para obtener acceso a tu aplicación, puedes redirigirlo a Universal Login y dejar que Auth0 gestione el proceso de autenticación.

| Flujo de autenticación | Tipo de flujo                                                    | Solicitudes a endpoints de la Authentication API |
| ---------------------- | ---------------------------------------------------------------- | ------------------------------------------------ |
| Inicio de sesión       | Verificación de nombre de usuario/contraseña*                    | 5                                                |
| Inicio de sesión       | proveedor de identidad de terceros – p. ej., Social o Work Login | 6                                                |
| Inicio de sesión       | Ya existe una sesión de autenticación de Auth0                   | 1                                                |
| Registro               | mediante nombre de usuario/contraseña                            | 6                                                |

<div id="modifiers">
  #### Modificadores
</div>

Ciertas configuraciones de autenticación modifican la cantidad base de solicitudes. Estos ajustes dependen de medidas de seguridad adicionales o de flujos de autenticación:

| Modificador                 | Descripción                                                                                                   | Solicitudes adicionales |
| --------------------------- | ------------------------------------------------------------------------------------------------------------- | ----------------------- |
| **ID First**                | Identifica al usuario antes de solicitar las credenciales.                                                    | +2                      |
| **MFA**                     | Añade autenticación multifactor.                                                                              | +2 por factor           |
| **OTP**                     | Contraseña de un solo uso para la autenticación                                                               | +2                      |
| **Enterprise Login**        | Autenticación mediante una conexión empresarial (p. ej., SAML, OIDC, LDAP).                                   | +1                      |
| **Credenciales de cliente** | Se utiliza para la autenticación máquina a máquina. Se aplica de forma universal, incluso si se usan Actions. | +1                      |

*Cualquier elemento usado en combinación se suma al total de solicitudes.

<div id="classic-login">
  ### Classic login
</div>

[Classic Login](/docs/es/authenticate/login/auth0-universal-login/universal-login-vs-classic-login/classic-experience) es una experiencia de inicio de sesión alojada por Auth0 que se basa en JavaScript para su personalización. Implementar Classic Login es menos complejo que integrar directamente en tu aplicación el proceso de autenticación y puede ayudarte a evitar los riesgos de la autenticación de origen cruzado.

| Flujo de autenticación | Tipo de flujo                                                    | Solicitudes |
| ---------------------- | ---------------------------------------------------------------- | ----------- |
| Inicio de sesión       | Verificación con nombre de usuario y contraseña                  | 8           |
| Inicio de sesión       | Proveedor de identidad de terceros – p. ej., Social o Work Login | 8           |
| Inicio de sesión       | Ya existe una sesión de autenticación de Auth0                   | 2           |
| Registro               | Nombre de usuario/contraseña                                     | 8           |

<Warning>
  Los clientes que configuren Custom Databases deben agregar dos (2) llamadas adicionales a la Authentication API en cada flujo de autenticación descrito anteriormente. Para obtener más detalles, consulta [Custom Database Connections](/docs/es/authenticate/database-connections/custom-db).
</Warning>

<div id="modifiers">
  #### Modificadores
</div>

Los siguientes factores aumentan el número de solicitudes para Classic Login:

| Modificador                        | Descripción                                                                               | Solicitudes adicionales |
| ---------------------------------- | ----------------------------------------------------------------------------------------- | ----------------------- |
| **Solo autenticación por SMS**     | Cuando se usa SMS como método principal de autenticación.                                 | +7                      |
| **Inicio de sesión social nativo** | Inicio de sesión con un proveedor social nativo (p. ej., Google, Facebook).               | +1                      |
| **Redirecciones**                  | Las redirecciones adicionales durante la autenticación aumentan el número de solicitudes. | +1                      |
