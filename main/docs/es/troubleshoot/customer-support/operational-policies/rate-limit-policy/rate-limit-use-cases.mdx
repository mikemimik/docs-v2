<div id="discover-when-requests-to-a-tenant-are-rate-limited">
  ## Descubre cuándo las solicitudes a un tenant están sujetas a limitación de velocidad
</div>

Hay varias formas de determinar si Auth0 está limitando la tasa de solicitudes de un producto del cliente. A continuación, se indican las posibles causas de esta limitación.

<div id="tenant-logs">
  ### Logs del tenant
</div>

Al suscribirte a distintos logs del tenant, puedes hacer seguimiento de problemas relacionados con el volumen de solicitudes. Para entender cómo funcionan en Auth0 los logs de eventos del tenant y de operaciones, consulta [Logs](/docs/es/deploy-monitor/logs).

<div id="api_limit">
  #### api_limit
</div>

El evento `api_limit` se activa inmediatamente después de que se supera el límite de tasa del bucket global de límite de tasa de Authentication o de la <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip>. Si, al cabo de un minuto, la cantidad de tokens de solicitud utilizados sigue por encima del 80 % para el mismo bucket de límite de tasa, se genera un segundo registro de advertencia. Si se supera el límite de tasa de un bucket de límite de tasa diferente, se genera un nuevo evento `api_limit`. Esto ayuda a los clientes a identificar qué configuración de límite de tasa están activando sus llamadas a la API, lo cual es un primer paso fundamental para diagnosticar la causa principal.

<div id="api_limit_warning">
  #### api_limit_warning
</div>

El registro `api_limit_warning` se activa cuando la tasa de solicitudes de un cliente consume el 80 % de los tokens de solicitud de un bucket de límite de tasa determinado. Si la cantidad de tokens de solicitud utilizados se mantiene por encima del 80 % después de un minuto para el mismo bucket de límite de tasa, se genera un segundo registro de advertencia. Si se supera el umbral del 80 % en un bucket de límite de tasa diferente, se crea un nuevo registro `api_limit_warning`.

<div id="appi-public-performance-burst-only">
  #### appi (Solo para Public Performance Burst)
</div>

El registro `appi` se activa cuando un tenant de cliente con el complemento Public Performance Burst supera el límite sostenido de 100 RPS de la Authentication API, consumiendo un bloque de un minuto de su asignación de ráfaga de 48 horas. Si, transcurridos 15 minutos, la tasa de solicitudes vuelve a superar ese límite sostenido de 100 RPS, se activa un segundo registro `appi`.

<div id="api-responses">
  ### Respuestas de la API
</div>

Las respuestas de la API de Auth0 devuelven respuestas [HTTP 429 (Too Many Requests)](http://tools.ietf.org/html/rfc6585#section-4) cuando se supera el límite de solicitudes. Esto permite a los clientes ver en tiempo real cuándo se aplica ese límite. Sin embargo, esto solo resulta útil para aplicaciones de clientes desarrolladas a medida que interactúan directamente con la API de Auth0.

<div id="sdk-error-handling">
  ### Gestión de errores del SDK
</div>

Si usa un SDK, consulte las páginas de errores de las [bibliotecas del SDK de Management API](/docs/es/libraries#mgmt).

<div id="error-pages">
  ### Páginas de error
</div>

La respuesta de la página de error se envía para los endpoints que muestran contenido HTML al usuario final. Si tu tenant está configurado para usar páginas genéricas (alojadas por Auth0), Auth0 muestra la página de error en lugar del contenido esperado cuando superas el límite de respuesta. Si tu tenant está configurado para usar [páginas de error personalizadas](/docs/es/customize/login-pages/custom-error-pages), se redirige al usuario a la URL de la página de error personalizada con el error correspondiente en el parámetro de cadena de consulta `error_description`. Para obtener más información, consulta [Endpoints afectados](/docs/es/troubleshoot/customer-support/operational-policies/rate-limit-policy/authentication-api-endpoint-rate-limits#affected-endpoints) y las descripciones de [Error JSON](/docs/es/troubleshoot/customer-support/operational-policies/rate-limit-policy/authentication-api-endpoint-rate-limits#json-error).

<div id="find-out-why-a-tenant-is-being-rate-limited">
  ## Averigua por qué un tenant está siendo limitado por tasa
</div>

Si crees que las solicitudes del tenant están siendo limitadas por tasa y necesitas ayuda para entender por qué, abre una solicitud a través del [Support Center](http://support.auth0.com). Como parte de tu solicitud, incluye el registro sin procesar completo en el que se observó el problema.

<div id="predict-when-requests-to-a-tenant-will-be-rate-limited">
  ## Prediga cuándo las solicitudes a un tenant estarán sujetas a límites de tasa
</div>

Auth0 proporciona información actualizada sobre el estado actual de sus límites de tasa mediante encabezados de respuesta HTTP de endpoints que tienen configuradas políticas de límites de tasa. Este estado se comunica de la siguiente manera:

* `x-ratelimit-limit`: Cantidad máxima de solicitudes disponibles.
* `x-ratelimit-remaining`: Cantidad de solicitudes restantes disponibles hasta que el bucket se vuelva a llenar con solicitudes adicionales.
* `x-ratelimit-reset`: [marca de tiempo UNIX](https://en.wikipedia.org/wiki/Unix_time), en segundos, de la hora estimada en la que se agregarán solicitudes adicionales al bucket.

Por ejemplo:

Una API tiene el siguiente límite de tasa:

* Límite de ráfaga: `1000`
* Límite de tasa sostenida: `100` `solicitudes por segundo` (en una ventana fija)

A partir de esta información, puede deducir lo siguiente:

* El límite de tasa sostenida es de `100 solicitudes por segundo` en una ventana fija.
* Debido a la ventana fija, el bucket de solicitudes se vuelve a llenar cada segundo.

Si recibe los siguientes encabezados `x-` en la respuesta de su API:

* `x-ratelimit-limit: 1000`
* `x-ratelimit-remaining: 50`
* `x-ratelimit-reset: 1675452600`

Ahora sabe que:

* Su tenant ha consumido 950 de las 1000 solicitudes permitidas para esa API, y solo le quedan 50 solicitudes hasta que se agreguen solicitudes adicionales.
* Se agregarán nuevas solicitudes en `1675452600`, es decir, a las 7:30:00 p. m. UTC del 3 de febrero de 2023.
* En ese momento se agregará 1 nueva solicitud

Por lo tanto, si está realizando solicitudes a una tasa mayor que la descrita anteriormente, es de esperar que se aplique una limitación de tasa. Qué tan pronto se le aplicarán límites de tasa depende del límite de ráfaga y de en qué medida esté superando el límite sostenido.

<div id="examples-of-how-rate-limits-are-enforced">
  ## Ejemplos de cómo se aplican los límites de tasa
</div>

<div id="requests-per-second-example">
  ### Ejemplo de solicitudes por segundo
</div>

Supongamos que Auth0 lanza una nueva API llamada `/ratelimitexample` con los siguientes valores de límite de tasa:

* Límite de ráfaga: cinco (5) solicitudes
* Límite de tasa sostenida: 10 solicitudes por segundo.

Puntos clave:

* La API comienza con cinco tokens de solicitud y nunca superará esa cantidad, que equivale al límite de ráfaga.
* El bucket de 10 tokens se repone cada segundo mediante una “ventana fija”. Se agregan nuevos tokens al bucket, rellenándolo al “inicio” de cada segundo.

 Ejemplo de escenario con límites de tasa:

<Frame>![](/docs/images/cdy7uua7fh8z/30m5xST81Db6mROVGCV5Bj/34d9638eb7467f02c2d0ecbe87ff0fd8/Examples_of_how_rate_limits_are_enforced_-_Page_1.png)</Frame>

En este escenario:

* T0 - T1sec:  El usuario final hace seis solicitudes en el primer segundo.  Cinco solicitudes, equivalentes al límite de ráfaga, reciben una respuesta `200`.  La sexta solicitud recibe un error `429` porque no quedan tokens de solicitud en el bucket.
* T1sec - T2sec:  Auth0 vuelve a llenar el bucket de tokens de solicitud debido al algoritmo de ventana fija. Como resultado, las solicitudes séptima a undécima se completan correctamente, agotando el bucket en la solicitud 12, lo que genera un error `429`.
* T2sec - T3sec:  Auth0 vuelve a rellenar el bucket de tokens una vez más, y la siguiente solicitud (13) recibe una respuesta `200`.

<div id="requests-per-minute-example">
  ### Ejemplo de solicitudes por minuto
</div>

Supongamos que Auth0 lanza una nueva API llamada `/ratelimitexample2` con los siguientes valores de límite de tasa:

* Límite de ráfaga:  Cinco (5) solicitudes
* Límite de tasa sostenida:  Seis (6) solicitudes por minuto.

Puntos clave:

* La API comienza con cinco tokens de solicitud, lo que equivale al límite de ráfaga.
* El bucket de seis tokens se rellena cada minuto mediante una “ventana fija”. Se agregan nuevos tokens al bucket, rellenándolo al inicio de cada minuto.

Escenario de ejemplo con límites de tasa:

<Frame>![](/docs/images/cdy7uua7fh8z/PhtwGBwS9PfEpNQbPeA1o/dc2b05c72ef960a19958dbafa0295e70/Examples_of_how_rate_limits_are_enforced_-_Page_1__1_.png)</Frame>

En este escenario:

* T0 - T+1min:  El usuario final hace seis solicitudes en el primer minuto. Cinco solicitudes —equivalentes al límite de ráfaga— reciben una respuesta `200`.  La sexta solicitud recibe un error `429` porque no quedan tokens de solicitud.
* T+1min - T+2min:  Auth0 rellena el bucket de tokens debido al algoritmo de ventana fija. Como resultado, las solicitudes de la séptima a la undécima se completan correctamente, y la duodécima agota el bucket, lo que da lugar a un error `429`.
* T+2min: Auth0 vuelve a rellenar el bucket de tokens, y la siguiente solicitud (13) recibe una respuesta `200`.

<div id="other-scenarios">
  ### Otros escenarios
</div>

En ocasiones, Auth0 asigna dos límites de tasa a una sola API.  Esto se hace para configurar un límite de ráfaga y un límite de tasa sostenida más adaptado a las necesidades del servicio.  En la práctica, el primer límite de tasa se convierte en el límite de ráfaga efectivo y el segundo límite de tasa se convierte en el límite de tasa sostenida efectivo.  En este escenario, Auth0 solo publica los límites de ráfaga y de tasa sostenida efectivos, en lugar de comunicar los límites de ráfaga y de tasa sostenida reales.

<div id="end-user-login-and-signup-api-usage">
  ## Uso de la API para el inicio de sesión y el registro de usuarios finales
</div>

Hay varios flujos de autenticación, entre ellos Inicio de sesión, Registro y Cambio de contraseña. Los más comunes suelen ser el Inicio de sesión y, después, el Registro.

El inicio de sesión de un usuario final desencadena múltiples llamadas a endpoints de Authentication API para determinar si el usuario final está autorizado a recibir un token de autorización y, por lo tanto, acceder a la aplicación que ha solicitado.

El número exacto de llamadas a la API depende de varias configuraciones:

* Experiencia de autenticación (p. ej., la nueva <Tooltip tip="Universal Login: Tu aplicación redirige a Universal Login, alojado en el servidor de autorización de Auth0, para verificar la identidad del usuario." cta="Ver glosario" href="/docs/es/glossary?term=Universal+Login">Universal Login</Tooltip> o Classic Login)
* Flujo de autenticación (p. ej., Inicio de sesión, Registro o Cambio de contraseña)
* Tipo de flujo de autenticación (p. ej., Inicio de sesión con nombre de usuario / contraseña; Inicio de sesión mediante redes sociales; Inicio de sesión cuando ya existe un token de autenticación)

A continuación, describimos algunas configuraciones habituales de los clientes y su impacto en el uso de la API.

<div id="universal-login">
  ### Universal Login
</div>

[Auth0 Universal Login](/docs/es/authenticate/login/auth0-universal-login) proporciona la función esencial de un <Tooltip tip="Servidor de autorización: servidor centralizado que contribuye a definir los límites del acceso de un usuario. Por ejemplo, tu servidor de autorización puede controlar los datos, las tareas y las funciones disponibles para un usuario." cta="Ver glosario" href="/docs/es/glossary?term=authorization+server">servidor de autorización</Tooltip>: el flujo de inicio de sesión. Cuando un usuario necesita demostrar su identidad para acceder a tu aplicación, puedes redirigirlo a Universal Login y dejar que Auth0 gestione el proceso de autenticación.

| Flujo de autenticación | Tipo de flujo                                                                 | Solicitudes a los endpoints de Authentication API |
| ---------------------- | ----------------------------------------------------------------------------- | ------------------------------------------------- |
| Inicio de sesión       | desafío de nombre de usuario y contraseña*                                    | 5                                                 |
| Inicio de sesión       | proveedor de identidad de terceros; p. ej., inicio de sesión social o corporativo | 6                                                 |
| Inicio de sesión       | Ya existe una sesión de autenticación de Auth0                                | 1                                                 |
| Registro               | mediante nombre de usuario/contraseña                                         | 6                                                 |

<div id="modifiers">
  #### Modificadores
</div>

Ciertas configuraciones de autenticación modifican el número base de solicitudes. Estos ajustes dependen de medidas de seguridad adicionales o de flujos de autenticación:

| Modificador            | Descripción                                                                                                     | Solicitudes adicionales |
| ---------------------- | --------------------------------------------------------------------------------------------------------------- | ----------------------- |
| **ID First**           | Identifica al usuario antes de solicitar las credenciales.                                                      | +2                      |
| **MFA**                | Añade autenticación multifactor.                                                                                | +2 por factor           |
| **OTP**                | Contraseña de un solo uso para la autenticación                                                                 | +2                      |
| **Enterprise Login**   | Autenticación mediante una conexión empresarial (p. ej., SAML, OIDC, LDAP).                                     | +1                      |
| **Client Credentials** | Se utiliza para la autenticación de máquina a máquina. Se aplica de forma general, incluso si se usan acciones. | +1                      |

*Cualquier elemento que se use en combinación se suma al total de solicitudes.

<div id="classic-login">
  ### Inicio de sesión clásico
</div>

[Classic Login](/docs/es/authenticate/login/auth0-universal-login/universal-login-vs-classic-login/classic-experience) es una experiencia de inicio de sesión alojada por Auth0 que se basa en JavaScript para su personalización. Implementar Classic Login es menos complejo que integrar el proceso de autenticación directamente en tu aplicación y puede ayudar a evitar los riesgos de la autenticación cross-origin.

| Flujo de autenticación | Tipo de flujo                                                                     | Solicitudes |
| ---------------------- | --------------------------------------------------------------------------------- | ----------- |
| Inicio de sesión       | Desafío de nombre de usuario y contraseña                                         | 8           |
| Inicio de sesión       | Proveedor de identidad de terceros; p. ej., inicio de sesión social o corporativo | 8           |
| Inicio de sesión       | Existe una sesión de autenticación de Auth0                                       | 2           |
| Registro               | Nombre de usuario y contraseña                                                    | 8           |

<Warning>
  Los clientes que configuren bases de datos personalizadas deben agregar dos (2) llamadas adicionales a la Authentication API a cada flujo de autenticación descrito anteriormente. Para más detalles, consulta [Conexiones de bases de datos personalizadas](/docs/es/authenticate/database-connections/custom-db).
</Warning>

<div id="modifiers">
  #### Modificadores
</div>

Los siguientes factores aumentan la cantidad de solicitudes para el inicio de sesión clásico:

| Modificador                        | Descripción                                                                                 | Solicitudes adicionales |
| ---------------------------------- | ------------------------------------------------------------------------------------------- | ----------------------- |
| **Autenticación solo por SMS**     | Cuando se usa SMS como método principal de autenticación.                                   | +7                      |
| **Inicio de sesión social nativo** | Inicio de sesión mediante un proveedor social nativo (p. ej., Google, Facebook).            | +1                      |
| **Redirecciones**                  | Las redirecciones adicionales durante la autenticación aumentan la cantidad de solicitudes. | +1                      |
