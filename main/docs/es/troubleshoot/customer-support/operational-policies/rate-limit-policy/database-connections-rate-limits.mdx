Auth0 limita el uso de los servicios para garantizar un rendimiento óptimo y protegerse frente a <Tooltip tip="Actores maliciosos: Entidad (una persona o grupo) que representa una amenaza para el negocio o el entorno con la intención de causar daño." cta="Ver glosario" href="/docs/es/glossary?term=bad+actors">actores maliciosos</Tooltip>, errores técnicos o tráfico legítimo excesivo.  Te recomendamos revisar cómo Auth0 aplica estos límites para configurar tu aplicación y ofrecer la mejor experiencia de usuario.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Para consultar los límites de frecuencia asociados a tu cuenta, consulta [Configuraciones de límites de tasa](/docs/es/troubleshoot/customer-support/operational-policies/rate-limit-policy/rate-limit-configurations), donde encontrarás una matriz de todas las políticas de límite de tasa.
</Callout>

<div id="introduction-to-rate-limits">
  ## Introducción a los límites de frecuencia
</div>

Auth0 impone limitaciones para proteger sus servicios frente a un exceso de solicitudes y evitar que los clientes sufran interrupciones o degradaciones del servicio.

Auth0 supervisa y, en muchos casos, aplica una serie de límites, entre ellos:

* Solicitudes a entornos (Private Cloud Only)
* Solicitudes a tenants mediante la API o un endpoint de API
* Límites varios

<div id="environment-request-limits-private-cloud-only">
  ### Límites de solicitudes del entorno (solo para Private Cloud)
</div>

En Private Cloud, los límites de solicitudes del entorno se basan en el Tier de rendimiento de Private Cloud. Para obtener más información, consulta [Private Cloud en AWS](/docs/es/deploy-monitor/deploy-private-cloud/private-cloud-on-aws) o [Private Cloud on Azure](/docs/es/deploy-monitor/deploy-private-cloud/private-cloud-on-azure).

Actualmente, los límites de frecuencia del entorno de Private Cloud representan la carga máxima bajo la cual el producto Auth0 cumplirá los SLA. Sin embargo, por el momento, Auth0 solo aplica estos límites y notifica a los clientes cuando se han superado en un tenant específico dentro del entorno. En la mayoría de los casos de uso de Private Cloud, en los que los clientes mantienen un único tenant de producción, esto no supone un problema; sin embargo, los casos de uso que aprovisionan más de un tenant de producción deben tener en cuenta la carga esperada en todos los tenants del entorno e implementar monitoreo adicional según sea necesario.

| Funcionalidad                                       | Authentication API                                                                                                                                                                                                                                                                                                   | Todas las demás API (incluida Management API)                                                                                                                                                                                                                                                                              |
| --------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Límites de frecuencia del tenant                    | Todos los tenants consumen de un límite global compartido para toda la API en el entorno, pero el límite de frecuencia del entorno es un límite estricto. Superar el límite del entorno debido a la carga combinada de varios tenants hará que las solicitudes excedentes queden sujetas a limitación de frecuencia. | Los límites de frecuencia del tenant se aplican de forma independiente, y el límite de frecuencia del entorno actúa como un umbral de posible degradación del rendimiento y afectación del SLA, pero **no** provoca directamente limitación de frecuencia a nivel de tenant.                                               |
| Límite de frecuencia del entorno superado (ejemplo) | Con un límite del entorno de 1500 rps, un Tenant 1 con 1400 rps y un Tenant 2 con 900 rps (2300 rps en total) darán lugar a que 800 solicitudes queden sujetas a limitación de frecuencia.                                                                                                                           | Con un límite del entorno de 1500 rps, un Tenant 1 con 1400 rps y un Tenant 2 con 900 rps (2300 rps en total) **no** harán que se aplique limitación de frecuencia a tenants individuales, pero el entorno en su conjunto puede experimentar una degradación del rendimiento y posibles problemas relacionados con el SLA. |

<Warning>
  Superar un límite de frecuencia del entorno invalida el Acuerdo de Nivel de Servicio (SLA).
</Warning>

Para consultar consideraciones sobre pruebas de carga en Private Cloud, consulta [Private Cloud en AWS](/docs/es/deploy-monitor/deploy-private-cloud/private-cloud-on-aws) y [Private Cloud on Azure](/docs/es/deploy-monitor/deploy-private-cloud/private-cloud-on-azure).

<div id="tenant-request-limits">
  ### Límites de solicitudes por tenant
</div>

Auth0 limita la cantidad de solicitudes que se pueden realizar en un tenant.  Estos límites se configuran según la API y también según endpoints específicos de cada API.

<div id="api-rate-limits">
  #### Límites de frecuencia de API
</div>

Auth0 limita la cantidad de solicitudes a una API específica, independientemente del endpoint de la API.  Los límites de la API pueden variar según:

* API

  * Authentication
  * Management
* Tipo de tenant (Producción vs. Desarrollo o Staging)
* Nivel de suscripción (Free, Essential, Professional, Enterprise Public vs. Private)

Por ejemplo, un tenant Free que no es de producción podría tener límites distintos a los de un tenant de producción con una suscripción de pago. Para conocer configuraciones específicas de límites de frecuencia, consulta [Configuraciones de límites de tasa](/docs/es/troubleshoot/customer-support/operational-policies/rate-limit-policy/rate-limit-configurations).

<div id="user-requests">
  ### Solicitudes de los usuarios
</div>

Una sola solicitud de un usuario final (p. ej., inicio de sesión o registro) suele generar múltiples requests a los endpoints de Authentication API. La proporción real entre las solicitudes de los usuarios finales y Authentication API depende de varios factores:

* Entidad que se autentica (p. ej., máquina o aplicación móvil o de escritorio del usuario final)
* Experiencia de autenticación (p. ej., nuevo <Tooltip tip="Universal Login: Su aplicación redirige a Universal Login, alojado en el servidor de autorización de Auth0, para verificar la identidad de un usuario." cta="Ver glosario" href="/docs/es/glossary?term=Universal+Login">Universal Login</Tooltip> o Classic Login)
* Flujo de autenticación (p. ej., inicio de sesión, registro o cambiar la contraseña)
* Tipo de flujo de autenticación (p. ej., inicio de sesión con nombre de usuario / contraseña; inicio de sesión mediante Social login; inicio de sesión cuando ya existe un token de autenticación)

Los clientes que aprovechan la extensibilidad podrían agregar aún más requests, no solo a Authentication API, sino también a la <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip>, según la configuración de extensibilidad.
Consulte [Rate Limit Use Cases](/docs/es/troubleshoot/customer-support/operational-policies/rate-limit-policy/rate-limit-use-cases) para obtener orientación sobre cómo estimar de qué manera su configuración de Auth0 puede afectar el uso de su API.

<div id="endpoint-rate-limits">
  #### Límites de frecuencia por endpoint
</div>

Auth0 limita el número de solicitudes realizadas a los endpoints de API y, en algunos casos, el número de operaciones por endpoint.  Los límites de los endpoints de API también varían según:

* API
* Tipo de tenant
* Nivel de suscripción

Por ejemplo, un tenant Free de no producción tiene límites distintos a los de un tenant de producción con una suscripción de pago.

<div id="tenant-request-limits-order">
  ### Orden de aplicación de los límites de solicitudes del tenant
</div>

A medida que se hacen solicitudes a tu tenant, Auth0 las evalúa primero con respecto al límite global de la API y después con respecto al límite de frecuencia de endpoints específicos de la API.

<div id="miscellaneous-limits">
  ### Otros límites
</div>

<div id="database-login-limits">
  #### Límites de inicio de sesión para bases de datos
</div>

En las conexiones de base de datos, Auth0 limita ciertos tipos de intentos repetidos de inicio de sesión según la cuenta de usuario y la dirección IP. Para proteger la estabilidad general del sistema, Auth0 aplica límites de frecuencia de usuario/contraseña para mitigar la carga. El alto grado de personalización de Auth0 puede exponernos al riesgo de degradación del servicio. Entre las causas se incluyen:

* Pruebas de estrés con alta carga
* Pruebas de rendimiento
* Código ineficiente que hace que los usuarios inicien sesión varias veces

Las solicitudes están sujetas a los límites descritos en las políticas individuales de las API de Auth0.

Además, existe un límite de frecuencia de inicio de sesión para un mismo usuario: si una dirección IP realiza 20 intentos de inicio de sesión en un minuto en la misma cuenta de usuario, el límite de frecuencia se aplica. A partir de ese momento, Auth0 permite al usuario 10 intentos por minuto. Cualquier combinación de intentos de inicio de sesión correctos y fallidos cuenta para este límite.

<Card title="Límites que protegen a los usuarios">
  La [protección contra ataques de fuerza bruta](/docs/es/secure/attack-protection/brute-force-protection) de Auth0 y la [limitación de direcciones IP sospechosas](/docs/es/secure/attack-protection/suspicious-ip-throttling) también pueden limitar los inicios de sesión y los registros, pero son independientes de los límites de frecuencia. Para obtener más información sobre cómo Auth0 detecta y gestiona anomalías potencialmente maliciosas, consulta [Protección contra ataques](/docs/es/secure/attack-protection).
</Card>

<div id="sms-message-limits-for-multi-factor-authentication-end-users-only">
  #### Límites de mensajes SMS para la autenticación multifactor (solo usuarios finales)
</div>

Si intentas enviar más de 10 mensajes SMS a tu dispositivo en el plazo de una hora, recibirás un mensaje de error sobre una excepción de límite de frecuencia.

Cuando superes el límite de mensajes, deberás esperar al menos una hora desde la primera solicitud de mensaje antes de solicitar otro. Recibirás un intento adicional por cada hora adicional que transcurra.

<div id="native-social-login-limits">
  #### Límites del inicio de sesión social nativo
</div>

Los límites aplicables a las solicitudes del flujo de inicio de sesión social nativo se determinan en función del cuerpo de las solicitudes según los siguientes criterios iniciales:

| Tipo de solicitud    | Cuerpo                                               |
| -------------------- | ---------------------------------------------------- |
| `grant_type`         | `urn:ietf:params:oauth:grant-type:token-exchange`    |
| `subject_token_type` | `http://auth0.com/oauth/token-type/apple-authz-code` |

<div id="public-performance-burst">
  #### Public Performance Burst
</div>

La oferta Public Performance Burst es un add-on disponible para suscripciones empresariales que mejora un deployment existente de Public Cloud. Esta oferta permite un aumento dinámico del límite de solicitudes de la Authentication API hasta un multiplicador del límite de solicitudes empresarial predeterminado de 100 RPS durante un máximo de 48 horas al mes.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Este add-on solo amplía los límites de solicitudes de la Authentication API y NO se aplica a la Management API ni a otros endpoints sujetos a límites de frecuencia fuera del scope de la Authentication API.
</Callout>

Actualmente, hay tres modificadores de Public Performance Burst (2x, 3x y 4x) que permiten 200, 300 y 400 RPS, respectivamente, para la Authentication API durante un máximo de 48 horas al mes. Las 48 horas se contabilizan y se descuentan de la cuota mensual permitida en intervalos de 5 minutos, lo que permite usar la API al nivel del multiplicador durante 576 intervalos de 5 minutos al mes.

Cuando el volumen de requests para la Authentication API supera el valor predeterminado de 100 RPS, se descuenta un intervalo de 5 minutos de la cuota mensual y se permite tráfico a la frecuencia asociada con el multiplicador. El tráfico puede mantenerse dentro del rango del multiplicador durante 5 minutos consecutivos completos a partir de ese momento sin generar descuentos adicionales.

Es posible monitorear los eventos de deducción de intervalos a través de los [logs del tenant](/docs/es/deploy-monitor/logs/log-event-type-codes). Cada deducción genera un evento de log del tenant asociado de tipo `appi` que contiene información sobre la cuota ya consumida y la restante.

Para obtener más información, consulta la sección de la Authentication API en [Límites de frecuencia - Empresarial](/docs/es/troubleshoot/customer-support/operational-policies/rate-limit-policy/rate-limit-configurations/enterprise-public).

<div id="private-performance-burst">
  #### Private Performance Burst
</div>

La oferta Private Performance Burst (disponible actualmente en AWS para los niveles 30x y 60x) incluye una capacidad de rendimiento en ráfaga (pico) de hasta 30x (3.000 RPS) o 60x (6.000 RPS) durante un máximo de 80 horas al mes. La capacidad de rendimiento base, que equivale a la mitad de la capacidad de rendimiento en ráfaga, está disponible durante el resto del mes.

En otras palabras, Private Performance Burst 30x tiene:

* Capacidad base: 1.500 RPS durante todo el mes
* Capacidad en ráfaga/pico: 3.000 RPS durante un máximo de 80 horas al mes

Si las transacciones de autenticación superan el límite base de solicitudes de la API, se descuenta una hora de la asignación mensual. A partir de ese momento, el tráfico puede mantenerse a una tasa elevada durante una hora completa ininterrumpida. Se realizarán deducciones adicionales de la misma manera cada vez que se vuelva a superar el umbral.

Una vez agotada la asignación de 80 horas, al tráfico de autenticación se le aplicará un límite de frecuencia correspondiente a la capacidad base hasta que entre en vigor la nueva asignación mensual.

<div id="extensibility-concurrency-limits">
  #### Límites de concurrencia de la extensibilidad
</div>

Para garantizar la disponibilidad del sistema y un uso equitativo de sus recursos, Auth0 limita la cantidad de solicitudes simultáneas en curso en todos los productos de extensibilidad: Actions, Hooks, Rules, Custom Database Connections, Extensions y conexiones OAuth2 personalizadas. Los tenants que superen sus límites de solicitudes simultáneas pueden recibir errores en las solicitudes nuevas hasta que se completen las solicitudes en curso. Los límites de concurrencia se definen a continuación:

| Suscripción                                                           | Límite de concurrencia (por tenant) |
| --------------------------------------------------------------------- | ----------------------------------- |
| Public Cloud                                                          | 250                                 |
| Tier Dev Private Cloud                                                | 100                                 |
| Private Cloud Basic 100 RPS (1x)                                      | 200                                 |
| Private Cloud Performance 500 RPS (5x)                                | 400                                 |
| Private Cloud Performance 1500 RPS (15x)                              | 1200                                |
| Private Cloud Performance 3000 RPS (30x) y 3000 RPS Burst (30x Burst) | 1200                                |
| Private Cloud Performance 6000 RPS (60x) y 6000 RPS Burst (60x Burst) | 1200                                |

La concurrencia se puede calcular multiplicando el RPS esperado por la latencia de cada solicitud. Por ejemplo, un tenant que tiene vinculadas dos Actions de Post Login, cada una con una duración de 250 ms, y un RPS total de login de 400, tiene una concurrencia esperada de (2 * (400 solicitudes / 1 segundo) * (.25 segundos / 1 solicitud)) = 200.

Para asegurarse de que su tenant no se vea afectado por estos límites de concurrencia, asegúrese de que la lógica de extensibilidad potencialmente de larga duración, como las llamadas a API externas, tenga tiempos de espera razonables.

<div id="rate-limit-algorithm">
  ## Algoritmo de límite de frecuencia
</div>

Auth0 establece límites de frecuencia y límites de ráfaga para sus API. Mientras que el límite de frecuencia es la cantidad máxima de tráfico que el sistema puede sostener de forma continua, el límite de ráfaga es el volumen máximo de tráfico a corto plazo que el sistema permite dentro de un intervalo de tiempo. Los límites de frecuencia y los límites de ráfaga de Auth0 funcionan conjuntamente para ofrecer una limitación más eficaz ante volúmenes de tráfico dinámicos.

Los límites de frecuencia de Auth0 usan un algoritmo de token bucket con las siguientes configuraciones:

* **Claves de límite:**

  * Normalmente, una clave de límite de frecuencia se basa en dos factores principales:

    * API y endpoint
    * Tipo de tenant
  * En algunos casos, también se incluyen factores adicionales:

    * IP de origen
    * ID del usuario de destino
* **Valores de límite:**

  * **Tamaño del bucket:** La cantidad máxima de solicitudes que una API o un endpoint recibe en general, o que recibe de un usuario o una dirección IP específicos, antes de que se agreguen nuevas solicitudes.
  * **Tasa de recarga:** La velocidad a la que se agregan nuevas solicitudes al bucket.

A partir de estos dos valores, Auth0 calcula el límite de ráfaga y el límite de tasa sostenida:

* **Límite de ráfaga:**  Igual al tamaño del bucket.
* **Límite de tasa sostenida:** Tasa de recarga en solicitudes por minuto o por segundo.

Si el límite de tasa sostenida se calcula en Requests per Second, las nuevas solicitudes se agregan en milisegundos. Si el límite de tasa sostenida se calcula en Requests per Minute, las nuevas solicitudes se agregan por segundos.
