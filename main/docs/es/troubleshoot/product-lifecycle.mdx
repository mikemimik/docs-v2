Al desarrollar productos de Auth0, nos comprometemos a:

* Ofrecer valor a los clientes desde el principio y con frecuencia, iterando en función de sus comentarios
* Buscar una comprensión profunda de nuestros clientes y tenerlos en cuenta en cada decisión
* Recopilar y analizar datos de forma constante para poder tomar mejores decisiones
* Visualizar y diseñar las versiones actuales, idealizadas y futuras de todo nuestro producto al incorporar funcionalidades

Este documento proporciona orientación sobre qué se considera un cambio retrocompatible (no disruptivo) y un cambio no retrocompatible (disruptivo) en la plataforma de Auth0. La distinción entre estos cambios ayuda a los desarrolladores a anticipar posibles impactos en sus aplicaciones al utilizar los servicios de Auth0.

Este documento describe cambios comunes y su compatibilidad para ayudar a establecer expectativas sobre las integraciones. Los cambios descritos no son exhaustivos ni completos; para obtener más detalles sobre los cambios disruptivos, consulte los temas a continuación.

<div id="auth0-apis">
  ## APIs de Auth0
</div>

Auth0 expone y mantiene APIs como las de autenticación y administración, cada una de las cuales define un contrato entre Auth0 y sus clientes en función de las especificaciones documentadas de la API y de los artefactos asociados. Auth0 sigue estándares reconocidos, como los de IETF y OIDC; cuando no existen estándares, desarrolla APIs propietarias conforme a las prácticas recomendadas.

Los cambios en estas APIs, ya sean retrocompatibles o no retrocompatibles, son necesarios de vez en cuando para mejorar su funcionalidad, seguridad o rendimiento.

<div id="backward-compatible-non-breaking-changes">
  ### Cambios retrocompatibles, no disruptivos
</div>

Un cambio retrocompatible no interrumpe la interoperabilidad entre la plataforma Auth0 y las aplicaciones de los clientes, y no exige que los clientes tomen medidas inmediatas ni participen en el proceso de migración.

Algunos ejemplos de cambios no disruptivos incluyen:

* **Cadenas opacas**: El formato y el tamaño de las cadenas opacas (por ejemplo, tokens, IDs) pueden cambiar. Los clientes no deben asumir un tamaño ni un formato fijos. El tamaño máximo de las cadenas opacas no superará los 4096 caracteres.
* **Tamaño de <Tooltip tip="JSON Web Token (JWT): formato estándar de token de ID (y, a menudo, formato de access token) utilizado para representar claims de forma segura entre dos partes." cta="Ver glosario" href="/docs/es/glossary?term=JWT">JWT</Tooltip>**: Según la especificación de <Tooltip tip="JSON Web Token (JWT): formato estándar de token de ID (y, a menudo, formato de access token) utilizado para representar claims de forma segura entre dos partes." cta="Ver glosario" href="/docs/es/glossary?term=OAuth">OAuth</Tooltip> (RFC6749), el tamaño de las credenciales JWT no está definido. Auth0 puede emitir JWT de tamaño variable, y los clientes no deben asumir un tamaño específico.
* **Tamaño del código de autorización**: Los clientes deben prever que los códigos de autorización, según la especificación de OAuth, pueden variar de tamaño.
* **Parámetros de respuesta no reconocidos**: Los clientes deben ignorar los parámetros de respuesta no reconocidos, lo que permite a Auth0 añadir nuevas funcionalidades sin afectar el funcionamiento actual.
* **Nuevos recursos, campos, headers o scopes**: La incorporación de nuevos recursos, campos, headers o scopes de API no afectará a los clientes existentes que no reconozcan ni utilicen estos elementos.

Para obtener una explicación más detallada de los cambios retrocompatibles, consulta las directrices de cambios de API de Auth0 en las [etapas del ciclo de vida del producto](/docs/es/troubleshoot/product-lifecycle/product-release-stages).

<div id="backward-incompatible-breaking-changes">
  ### Cambios no retrocompatibles que causan cambios disruptivos
</div>

Un cambio no retrocompatible puede provocar fallos cuando altera la interoperabilidad entre la plataforma Auth0 y las aplicaciones de los clientes. Cuando este tipo de cambio es necesario, sigue el [proceso de obsolescencia y migración](/docs/es/troubleshoot/product-lifecycle/deprecations-and-migrations) para notificarlo y ofrecer asistencia a los clientes a fin de que adapten sus implementaciones en el tenant.

Algunos ejemplos de cambios disruptivos son:

* **Eliminación de recursos de la API**: Si se elimina o se cambia el nombre de un recurso de la API, los clientes que dependen de ese recurso se verán afectados por un cambio disruptivo.
* **Cambios en la estructura de la URI**: Modificar la estructura de una URI existente puede afectar a los clientes que dependen de ella.
* **Eliminación de métodos, parámetros o campos**: Si se elimina o se cambia el nombre de un método, parámetro o campo, esto dará lugar a un cambio disruptivo para los clientes que utilizan estos elementos.
* **Cambios en los valores predeterminados**: Los cambios en el valor predeterminado de un campo pueden afectar a las integraciones existentes y constituir un cambio disruptivo.
* **Cambios en las respuestas de error y los códigos de estado**: Las modificaciones en los formatos de respuesta de error, los códigos de error o los códigos de estado pueden afectar al comportamiento actual de los clientes.
* **Formato de JWT**: Los cambios en el formato JWT de los tokens son cambios disruptivos
* **Formato JSON**: Cambiar el tipo de un valor JSON es un cambio disruptivo.

Cualquier cambio disruptivo activa el proceso de obsolescencia, en el que se notifica a los clientes y se les concede un plazo de al menos seis meses para migrar al nuevo comportamiento. Para obtener más detalles, consulta el [proceso de obsolescencia y fin de vida útil](/docs/es/troubleshoot/product-lifecycle/deprecations-and-migrations).

<div id="auth0s-commitment">
  ## El compromiso de Auth0
</div>

Auth0 minimiza las interrupciones causadas por cambios no retrocompatibles mediante el siguiente proceso:

* **Anuncio de obsolescencia**: Auth0 anuncia la obsolescencia para informar a los clientes de un cambio próximo.
* **Ventana de migración**: Los clientes disponen de un plazo mínimo de seis meses para migrar a la funcionalidad actualizada, con orientación en nuestro [proceso de migración](/docs/es/troubleshoot/product-lifecycle/migration-process).
* **Fin de vida útil**: Cuando se cierra la ventana de migración, la funcionalidad obsoleta pasa a la etapa de fin de vida útil y deja de estar disponible.

<div id="undocumented-apis">
  ## APIs no documentadas
</div>

Las API de Auth0 que no están documentadas se consideran privadas y pueden cambiar sin previo aviso. Estas API no están cubiertas por el proceso de obsolescencia, y los clientes deben evitar depender de ellas en sistemas de producción.
