Al crear productos de Auth0, nos proponemos:

* Ofrecer valor a los clientes desde el principio y con frecuencia, iterando en función de sus comentarios
* Buscar una comprensión profunda de nuestros clientes y tenerlos presentes en cada decisión
* Recopilar y analizar datos de forma constante para poder tomar mejores decisiones
* Visualizar y diseñar las versiones actuales, ideales y futuras de todo nuestro producto al agregar funcionalidades

Este documento ofrece orientación sobre qué se considera un cambio compatible con versiones anteriores (no disruptivo) y un cambio no retrocompatible (disruptivo) en la plataforma Auth0. La distinción entre estos cambios ayuda a los desarrolladores a anticipar posibles impactos en sus aplicaciones al consumir servicios de Auth0.

Este documento describe cambios comunes y su compatibilidad para ayudar a establecer expectativas sobre las integraciones. Los cambios descritos no son exhaustivos ni abarcan todos los casos; para obtener más detalles sobre los cambios disruptivos, lea los temas a continuación.

<div id="auth0-apis">
  ## APIs de Auth0
</div>

Auth0 expone y mantiene APIs como la API de autenticación y la Management API, cada una de las cuales define un contrato entre Auth0 y sus clientes basado en las especificaciones documentadas de la API y sus artefactos asociados. Auth0 sigue estándares reconocidos de organismos como IETF y OIDC; cuando no existen estándares, desarrolla APIs propietarias conforme a las prácticas recomendadas.

Los cambios en estas APIs, ya sean compatibles con versiones anteriores o no retrocompatibles, son necesarios de vez en cuando para mejorar su funcionalidad, seguridad o rendimiento.

<div id="backward-compatible-non-breaking-changes">
  ### Cambios compatibles con versiones anteriores y no disruptivos
</div>

Un cambio compatible con versiones anteriores no interrumpe la interoperabilidad entre la plataforma Auth0 y las aplicaciones de los clientes, ni requiere que los clientes tomen medidas inmediatas o participen en el proceso de migración.

Algunos ejemplos de cambios no disruptivos incluyen:

* **Cadenas opacas**: El formato y el tamaño de las cadenas opacas (p. ej., tokens, IDs) pueden cambiar. Los clientes no deben asumir un tamaño ni un formato fijos. El tamaño máximo de las cadenas opacas no superará los 4096 caracteres.
* **Tamaño de <Tooltip tip="JSON Web Token (JWT): Formato estándar de token de ID (y, con frecuencia, formato de access token) utilizado para representar claims de forma segura entre dos partes." cta="Ver glosario" href="/docs/es/glossary?term=JWT">JWT</Tooltip>**: Según la especificación <Tooltip tip="JSON Web Token (JWT): Formato estándar de token de ID (y, con frecuencia, formato de access token) utilizado para representar claims de forma segura entre dos partes." cta="Ver glosario" href="/docs/es/glossary?term=OAuth">OAuth</Tooltip> (RFC6749), el tamaño de las credenciales JWT no está definido. Auth0 puede emitir JWT de tamaño variable, y los clientes no deben asumir un tamaño específico.
* **Tamaño del código de autorización**: Los clientes deben prever que los códigos de autorización, según la especificación de OAuth, pueden variar de tamaño.
* **Parámetros de respuesta no reconocidos**: Los clientes deben ignorar los parámetros de respuesta no reconocidos, lo que permite a Auth0 añadir nuevas funcionalidades sin afectar la funcionalidad actual.
* **Nuevos recursos, campos, encabezados o scopes**: La incorporación de nuevos recursos, campos, encabezados o scopes de API no afectará a los clientes existentes que no reconozcan ni utilicen estos elementos.

Para obtener una explicación más detallada de los cambios compatibles con versiones anteriores, consulta las directrices sobre cambios en la API de Auth0 en las [etapas del ciclo de vida del producto](/docs/es/troubleshoot/product-lifecycle/product-release-stages).

<div id="backward-incompatible-breaking-changes">
  ### Cambios no retrocompatibles y disruptivos
</div>

Un cambio no retrocompatible puede provocar fallos cuando altera la interoperabilidad entre la plataforma Auth0 y las aplicaciones de los clientes. Cuando este tipo de cambio es necesario, sigue el [proceso de obsolescencia y migración](/docs/es/troubleshoot/product-lifecycle/deprecations-and-migrations) para avisar a los clientes y ofrecerles soporte a fin de que adapten sus implementaciones del tenant.

Algunos ejemplos de cambios disruptivos incluyen:

* **Eliminación de recursos de API**: Si se elimina o se cambia el nombre de un recurso de API, los clientes que dependen de ese recurso experimentarán un cambio disruptivo.
* **Cambios en la estructura de URI**: Modificar la estructura de un URI existente puede afectar a los clientes que dependen de él.
* **Eliminación de métodos, parámetros o campos**: Si se elimina o se cambia el nombre de un método, parámetro o campo, esto dará lugar a un cambio disruptivo para los clientes que usan estos elementos.
* **Cambios en valores predeterminados**: Los cambios en el valor predeterminado de un campo pueden afectar a las integraciones existentes y constituir un cambio disruptivo.
* **Cambios en respuestas de error y códigos de estado**: Las modificaciones en los formatos de respuesta de error, los códigos de error o los códigos de estado pueden alterar el comportamiento existente de los clientes.
* **Formato JWT**: Los cambios en el formato JWT de los tokens son cambios disruptivos
* **Formato JSON**: Cambiar el tipo de un valor JSON es un cambio disruptivo.

Cualquier cambio disruptivo desencadena el proceso de obsolescencia, en el que se notifica a los clientes y se les da un plazo de al menos seis meses para migrar al nuevo comportamiento. Para obtener más detalles, consulta el [proceso de obsolescencia y fin de vida](/docs/es/troubleshoot/product-lifecycle/deprecations-and-migrations).

<div id="auth0s-commitment">
  ## El compromiso de Auth0
</div>

Auth0 minimiza las interrupciones causadas por cambios no retrocompatibles mediante el siguiente proceso:

* **Anuncio de obsolescencia**: Auth0 anuncia la obsolescencia para avisar a los clientes de un cambio inminente.
* **Ventana de migración**: Los clientes disponen de un plazo mínimo de seis meses para migrar a la funcionalidad actualizada, con orientación en nuestro [proceso de migración](/docs/es/troubleshoot/product-lifecycle/migration-process).
* **Fin de vida útil**: Cuando se cierra la ventana de migración, la funcionalidad obsoleta pasa a la etapa de fin de vida útil y deja de estar disponible.

<div id="undocumented-apis">
  ## APIs no documentadas
</div>

Las APIs no documentadas de Auth0 se consideran privadas y pueden cambiar sin previo aviso. Estas APIs no están cubiertas por el proceso de obsolescencia, y los clientes deben evitar basar en ellas sus sistemas de producción.
