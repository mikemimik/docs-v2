Al desarrollar productos de Auth0, nos proponemos:

* Ofrecer valor a los clientes de forma temprana y frecuente, iterando en función de sus comentarios
* Procurar una comprensión profunda de nuestros clientes y tenerlos en cuenta en cada decisión
* Recopilar y analizar datos de forma constante para poder tomar mejores decisiones
* Visualizar y diseñar versiones actuales, idealizadas y futuras de todo nuestro producto al añadir funcionalidades

Este documento ofrece orientación sobre qué constituye cambios retrocompatibles (no disruptivos) y cambios no retrocompatibles (disruptivos) en la plataforma Auth0. La distinción entre estos cambios ayuda a los desarrolladores a anticipar posibles impactos en sus aplicaciones al utilizar los servicios de Auth0.

Este documento describe cambios habituales y su compatibilidad para ayudar a establecer expectativas sobre las integraciones. Los cambios descritos no son exhaustivos ni completos; para obtener más información sobre los cambios disruptivos, consulte los temas a continuación.

<div id="auth0-apis">
  ## APIs de Auth0
</div>

Auth0 expone y mantiene APIs como las API de Authentication y Management, cada una de las cuales define un contrato entre Auth0 y sus clientes conforme a las especificaciones documentadas de la API y a sus artefactos asociados. Auth0 sigue estándares reconocidos de organismos como IETF y OIDC; cuando no existen estándares, desarrolla APIs propietarias siguiendo prácticas recomendadas.

Los cambios en estas APIs, ya sean retrocompatibles o no retrocompatibles, son necesarios de vez en cuando para mejorar su funcionalidad, seguridad o rendimiento.

<div id="backward-compatible-non-breaking-changes">
  ### Cambios retrocompatibles y no disruptivos
</div>

Un cambio retrocompatible no altera la interoperabilidad entre la plataforma Auth0 y las aplicaciones de los clientes, y no exige que los clientes tomen medidas inmediatas ni participen en el proceso de migración.

Entre los ejemplos de cambios no disruptivos se incluyen:

* **Cadenas opacas**: El formato y el tamaño de las cadenas opacas (por ejemplo, tokens, IDs) pueden cambiar. Los clientes no deben asumir un tamaño ni un formato fijos. El tamaño máximo de las cadenas opacas no superará los 4096 caracteres.
* **Tamaño de <Tooltip tip="JSON Web Token (JWT): formato estándar del token de ID (y, con frecuencia, también del token de acceso) utilizado para representar claims de forma segura entre dos partes." cta="Ver glosario" href="/docs/es/glossary?term=JWT">JWT</Tooltip>**: Según la especificación de <Tooltip tip="JSON Web Token (JWT): formato estándar del token de ID (y, con frecuencia, también del token de acceso) utilizado para representar claims de forma segura entre dos partes." cta="Ver glosario" href="/docs/es/glossary?term=OAuth">OAuth</Tooltip> (RFC6749), el tamaño de las credenciales JWT no está definido. Auth0 puede emitir JWT de tamaño variable, y los clientes no deben asumir un tamaño concreto.
* **Tamaño del código de autorización**: Los clientes deben prever que los códigos de autorización, según la especificación de OAuth, pueden variar en tamaño.
* **Parámetros de respuesta no reconocidos**: Los clientes deben ignorar los parámetros de respuesta no reconocidos, lo que permite a Auth0 añadir nuevas funcionalidades sin afectar a la funcionalidad actual.
* **Nuevos recursos, campos, encabezados o scopes**: La incorporación de nuevos recursos, campos, encabezados o scopes de API no afectará a los clientes existentes que no reconozcan o utilicen estos elementos.

Para obtener una explicación más detallada de los cambios retrocompatibles, consulta las directrices sobre cambios de API de Auth0 en las [etapas del ciclo de vida del producto](/docs/es/troubleshoot/product-lifecycle/product-release-stages).

<div id="backward-incompatible-breaking-changes">
  ### Cambios no retrocompatibles y disruptivos
</div>

Un cambio no retrocompatible puede provocar fallos cuando altera la interoperabilidad entre la plataforma Auth0 y las aplicaciones de los clientes. Cuando este tipo de cambio es necesario, se sigue el [proceso de obsolescencia y migración](/docs/es/troubleshoot/product-lifecycle/deprecations-and-migrations) para notificarlo y ofrecer asistencia a los clientes a fin de que ajusten las implementaciones de su tenant.

Entre los ejemplos de cambios disruptivos se incluyen:

* **Eliminación de recursos de API**: Si se elimina o se cambia el nombre de un recurso de API, los clientes que dependan de ese recurso experimentarán un cambio disruptivo.
* **Cambios en la estructura de URI**: Modificar la estructura de una URI existente puede afectar a los clientes que dependen de ella.
* **Eliminación de métodos, parámetros o campos**: Si se elimina o se cambia el nombre de un método, parámetro o campo, esto dará lugar a un cambio disruptivo para los clientes que utilicen estos elementos.
* **Cambios en los valores predeterminados**: Los cambios en el valor predeterminado de un campo pueden afectar a las integraciones existentes y constituir un cambio disruptivo.
* **Cambios en las respuestas de error y los códigos de estado**: Las modificaciones en los formatos de respuesta de error, los códigos de error o los códigos de estado pueden romper el comportamiento actual del cliente.
* **Formato JWT**: Los cambios en el formato JWT de los tokens son cambios disruptivos
* **Formato JSON**: Cambiar el tipo de un valor JSON es un cambio disruptivo.

Cualquier cambio disruptivo desencadena el proceso de obsolescencia, en el que se notifica a los clientes y se les concede un plazo mínimo de seis meses para migrar al nuevo comportamiento. Para obtener más detalles, consulta el [proceso de obsolescencia y fin de vida útil](/docs/es/troubleshoot/product-lifecycle/deprecations-and-migrations).

<div id="auth0s-commitment">
  ## El compromiso de Auth0
</div>

Auth0 minimiza las interrupciones causadas por cambios no retrocompatibles mediante el siguiente proceso:

* **Anuncio de obsolescencia**: Auth0 anuncia esta obsolescencia para avisar a los clientes de un cambio próximo.
* **Ventana de migración**: Los clientes disponen de una ventana mínima de seis meses para migrar a la funcionalidad actualizada, con orientación disponible en nuestro [proceso de migración](/docs/es/troubleshoot/product-lifecycle/migration-process).
* **Fin de vida útil**: Cuando se cierra la ventana de migración, la función obsoleta pasa a la etapa de fin de vida útil y deja de estar disponible.

<div id="undocumented-apis">
  ## APIs no documentadas
</div>

Las APIs no documentadas de Auth0 se consideran privadas y están sujetas a cambios sin previo aviso. Estas APIs no están cubiertas por el proceso de obsolescencia, y los clientes deben evitar depender de ellas en sistemas de producción.
