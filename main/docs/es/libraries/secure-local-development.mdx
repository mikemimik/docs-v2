Los entornos de desarrollo local suelen ejecutarse con HTTP (es decir, `http://localhost`) de forma predeterminada. Esta guía explica cuándo deberías ejecutar un servidor local con HTTPS y cómo configurar HTTPS en localhost.

<div id="when-to-use-a-local-https-server">
  ## Cuándo usar un servidor https local
</div>

Los navegadores web se comportan de forma diferente en http y en https; para adaptarte a estas diferencias, puede que necesites que tu servidor de desarrollo sirva páginas a través de https. Por ejemplo:

* Cuando tu página se sirve a través de https, se bloquearán las solicitudes para cargar recursos desde una URL http. Puede que quieras este comportamiento en tu entorno local para evitar errores que solo se producirían en el entorno de producción, o porque no quieres hacer por error solicitudes http que contengan información sensible.
* Las cookies que usan un atributo Secure o un atributo [SameSite](https://auth0.com/blog/browser-behavior-changes-what-developers-need-to-know/) establecido en None no se enviarán por canales inseguros, lo que interrumpirá la autenticación y otras funcionalidades.

<div id="how-to-set-up-a-secure-local-server">
  ## Cómo configurar un servidor local seguro
</div>

La idea es ejecutar un servidor web delante de tu servidor de desarrollo local para que acepte solicitudes HTTPS y las reenvíe al servidor web de tu aplicación a través de HTTP, de modo que las solicitudes de tu aplicación se sirvan por HTTPS mediante un proxy. Esto se conoce como un proxy inverso HTTPS, y podemos usar un producto ya preparado llamado [Caddy](https://caddyserver.com/v2) para crear uno.

El proceso para hacerlo es:

1. Descargar el binario de Caddy
2. Ejecutar el servidor web de tu aplicación a través de HTTP como de costumbre
3. Ejecutar el binario de Caddy como proxy inverso, indicándole que reenvíe el tráfico al servidor web de tu aplicación
4. Visitar el servidor web del proxy inverso a través de HTTPS

<div id="step-1-download-the-caddy-binary">
  ### Paso 1. Descargar el binario de Caddy
</div>

Visita https://caddyserver.com/download, selecciona tu plataforma y descarga el archivo en tu equipo.

<div id="step-2-run-your-applications-web-server-over-http-as-normal">
  ### Paso 2. Ejecuta el servidor web de tu aplicación mediante HTTP, como de costumbre
</div>

Ejecuta el servidor de desarrollo local de tu aplicación como de costumbre; por ejemplo, $ PORT=3000 npm start

<div id="step-3-run-the-caddy-binary-as-a-reverse-proxy-telling-it-to-forward-traffic-to-your-applications-web-server">
  ### Paso 3. Ejecuta el binario de Caddy como proxy inverso para reenviar el tráfico al servidor web de tu aplicación
</div>

Busca dónde está el binario de Caddy que descargaste y ejecuta el siguiente comando para crear un proxy inverso y reenviar las solicitudes del puerto 443 (el puerto predeterminado para https) al puerto de tu aplicación (en este caso, 3000):

```bash wrap lines
/path/to/caddy reverse-proxy --from localhost:443 --to localhost:3000
```

<div id="step-4-now-visit-your-web-server-over-https">
  ### Paso 4. Ahora visita tu servidor web por HTTPS
</div>

Visita https://localhost y servirá las solicitudes de tu aplicación por HTTPS

<div id="running-your-application-behind-a-proxy">
  ## Ejecutar tu aplicación detrás de un proxy
</div>

Puede que tengas que hacer algunos ajustes menores en tu aplicación cuando se ejecute detrás de un proxy.

Por ejemplo, en Express.js, aunque la aplicación no dejará de ejecutarse, registrará por error la dirección IP del proxy como la dirección IP del cliente, a menos que se configure `trust proxy`; consulta &lt;https://expressjs.com/en/guide/behind-proxies.html&gt;
