<div id="auth0lock">
  ## Auth0Lock
</div>

`new Auth0Lock(clientID, domain, options)`

Inicializa una nueva instancia de `Auth0Lock` configurada con el `clientID` de tu aplicación y el `domain` de tu cuenta en tu [Dashboard de administración de Auth0](https://manage.auth0.com/#/). El tercer parámetro, opcional, es un objeto `options` que se utiliza para configurar Lock según las necesidades de tu aplicación. Puedes encontrar esta información en la [configuración de la aplicación](https://manage.auth0.com/#/applications).

* **clientId `{String}`**: Parámetro obligatorio. El clientId de tu aplicación en Auth0.
* **domain `{String}`**: Parámetro obligatorio. Tu dominio de Auth0. Normalmente, tu-cuenta.auth0.com.
* **options `{Object}`**: Parámetro opcional. Permite configurar la apariencia y el comportamiento de Lock. Consulta [la página de opciones de configuración](/docs/es/libraries/lock/lock-configuration) para obtener más información.

export const codeExample1 = `var Auth = (function() {

  var privateStore = {};

  function Auth() {
    // Crear una instancia de Lock, sin opciones personalizadas
    this.lock = new Auth0Lock(
      '<{yourClientId}>',
      '<{yourDomain}>'
    );
  }

  Auth.prototype.getProfile = function() {
    return privateStore.profile;
  };

  Auth.prototype.authn = function() {
    // Escuchar el evento authenticated y obtener el perfil
    this.lock.on("authenticated", function(authResult) {
      // Usar el token de authResult para getUserInfo() y guardarlo si es necesario
      this.getUserInfo(authResult.accessToken, function(error, profile) {
        if (error) {
          // Manejar el error
          return;
        }

        // guardar el token de acceso solo si es necesario
        privateStore.accessToken = accessToken;
        privateStore.profile = profile;

        // Actualizar el DOM
      });
    });
  };
  return Auth;
}());`;

<div id="getuserinfo">
  ## getUserInfo()
</div>

`getUserInfo(accessToken, callback)`

Una vez que el usuario haya iniciado sesión y dispongas de un token, puedes usarlo para obtener el perfil del usuario con `getUserInfo`. Este método reemplaza al método obsoleto `getProfile()`.

* **accessToken &#123;String&#125;**: Token del usuario.
* **callback &#123;Function&#125;**: Se invocará cuando se haya recuperado el perfil del usuario.

```js lines
lock.getUserInfo(accessToken, function(error, profile) {
  if (!error) {
    alert("hello " + profile.name);
  }
});
```

<div id="show">
  ## show()
</div>

`show(options)`

El método `show` muestra el widget. A partir de la versión 10.2.0 de Lock, el método `show` también puede aceptar un objeto `options` como parámetro. Ten en cuenta que este parámetro está pensado para usarse como una forma de sobrescribir las `options` de tu Lock para esta visualización concreta del widget; las options deben establecerse al crear una instancia de Lock y sobrescribirse aquí solo si es necesario para tu caso de uso específico.

Se puede sobrescribir el siguiente subconjunto de `options` respecto de los valores asignados (o sus valores predeterminados) cuando se creó una instancia de Lock:

* allowedConnections
* auth.params
* allowLogin
* allowSignUp
* allowForgotPassword
* initialScreen
* rememberLastLogin

Para obtener más información sobre la lista completa de opciones configurables que pueden elegirse al crear una instancia de Lock, en lugar del subconjunto limitado anterior que puede sobrescribirse en el método `show`, consulta la [página de opciones configurables por el usuario](/docs/es/libraries/lock/lock-configuration).

Ejemplos de sobrescritura de options:

```js lines
// Mostrar el widget de Lock sin sobreescribir ninguna opción
lock.show();

// Mostrar el widget de Lock sobreescribiendo algunas opciones
lock.show({
  allowedConnections: ["twitter", "facebook"],
  allowSignUp: false
});
```

Las opciones deben configurarse al crear una instancia de Lock por primera vez `var lock = new Auth0Lock(clientId, domain, options);`. Las opciones solo deben pasarse a `show` para modificar temporalmente las opciones configuradas previamente al mostrar el widget en ese momento y lugar concretos.

Hay una opción adicional que puede configurarse en el método `show`, llamada `flashMessage`.

<div id="flashmessage">
  ### flashMessage
</div>

Este objeto solo está disponible como opción para el método `show`; no se usa en el objeto `options` normal al crear una instancia de Lock. El objeto `flashMessage` muestra un mensaje flash de error o de éxito cuando se muestra Lock. Tiene los siguientes parámetros:

* **type** &#123;String&#125;: El tipo de mensaje; debe ser `error` o `success`.
* **text** &#123;String&#125;: El texto que se mostrará.

```js lines
lock.show({
  flashMessage:{
    type: 'success',
    text: 'Amazing Success!!'
  }
});
```

Una aplicación práctica de la opción `flashMessage` es gestionar errores de autorización. El `flashMessage` puede contener texto con la descripción del error.

```js lines
lock.on('authorization_error', function(error) {
  lock.show({
    flashMessage: {
      type: 'error',
      text: error.errorDescription
    }
  });
});
```

Entonces, si `tester@example.com` intentara ahora iniciar sesión, al ser un usuario bloqueado, se le volvería a mostrar Lock, con una barra superior que muestra el mensaje de error, en lugar de que simplemente falle el inicio de sesión y Lock se cierre.

<div id="hide">
  ## hide()
</div>

`hide()`

El método `hide` cierra el widget si está abierto en ese momento. El widget se cierra por sí solo en la mayoría de los casos, por lo que este método suele usarse principalmente en casos de uso específicos. Por ejemplo, puede que quieras escuchar el evento `unrecoverable_error` y luego aplicar `hide` a Lock y redirigir al usuario a una página de error personalizada. Otro ejemplo es el de los usuarios que están implementando el [modo popup](/docs/es/libraries/lock/lock-authentication-modes) y que podrían necesitar ejecutar manualmente `hide` en el widget después de que se dispare el evento `authenticated`.

Ejemplo de uso para ocultar (cerrar) el widget de Lock en modo popup:

```js lines
// Escuchar el evento authenticated y ocultar Lock
lock.on("authenticated", function() {
  lock.hide();

  // Cualquier otra acción que desees realizar en el evento authenticated

});
```

<div id="on">
  ## on()
</div>

Lock emite eventos a lo largo de su ciclo de vida. El método `on` puede usarse para escuchar eventos concretos y reaccionar ante ellos.

* `show`: se emite cuando se muestra Lock. No tiene argumentos.
* `hide`: se emite cuando se oculta Lock. No tiene argumentos.
* `unrecoverable_error`: se emite cuando se produce un error irrecuperable, por ejemplo, cuando no hay ninguna conexión disponible. Tiene el error como único argumento.
* `authenticated`: se emite después de una autenticación correcta. Tiene el resultado de la autenticación como único argumento. El resultado de la autenticación contiene el token, que puede usarse para obtener el perfil del usuario o almacenarse para iniciar sesión en comprobaciones posteriores.
* `authorization_error`: se emite cuando falla la autorización. Tiene el error como único argumento.
* `hash_parsed`: cada vez que se inicializa un nuevo objeto Auth0Lock en modo de redirección (el valor predeterminado), intentará analizar la parte hash de la URL en busca del resultado de un intento de inicio de sesión. Este es un evento de bajo nivel para casos de uso avanzados y, siempre que sea posible, deben preferirse `authenticated` y `authorization_error`. Después, este evento se emitirá con `null` si no pudo encontrar nada en el hash. Se emitirá con el mismo argumento que el evento `authenticated` después de un inicio de sesión correcto o con el mismo argumento que `authorization_error` si algo salió mal. Este evento no se emitirá en [modo popup](/docs/es/libraries/lock/lock-authentication-modes) porque no es necesario analizar la parte hash de la URL.
* `forgot_password ready`: se emite cuando se muestra la pantalla &quot;Olvidé mi contraseña&quot;. (Solo en la versión &gt;`10.18`)
* `forgot_password submit`: se emite cuando el usuario hace clic en el botón de envío de la pantalla &quot;Olvidé mi contraseña&quot;. (Solo en la versión &gt;`10.14`)
* `signin ready`: se emite cuando se muestra la pantalla &quot;Iniciar sesión&quot;.
* `signup ready`: se emite cuando se muestra la pantalla &quot;Registrarse&quot;.
* `signin submit`: se emite cuando el usuario hace clic en el botón de envío de la pantalla &quot;Login&quot;. (Solo en la versión &gt;`10.18`)
* `signup submit`: se emite cuando el usuario hace clic en el botón de envío de la pantalla &quot;Registrarse&quot;. (Solo en la versión &gt;`10.18`)
* `federated login`: se emite cuando el usuario hace clic en un botón de conexión social. Tiene como argumentos el nombre de la conexión y la estrategia. (Solo en la versión &gt;`10.18`)
* `socialOrPhoneNumber ready`: se emite cuando se muestra la pantalla <Tooltip tip="Passwordless: Forma de autenticación que no se basa en una contraseña como primer factor." cta="Ver glosario" href="/docs/es/glossary?term=Passwordless">Passwordless</Tooltip> con Social + número de teléfono
* `socialOrPhoneNumber submit`: se emite cuando se envía la pantalla Passwordless con Social + número de teléfono
* `socialOrEmail ready`: se emite cuando se muestra la pantalla Passwordless con Social + correo electrónico
* `socialOrEmail submit`: se emite cuando se envía la pantalla Passwordless con Social + correo electrónico
* `vcode ready`: se emite cuando se muestra la pantalla Passwordless con la contraseña de un solo uso
* `vcode submit`: se emite cuando se envía la pantalla Passwordless con la contraseña de un solo uso

El listener del evento `authenticated` tiene un único argumento: un objeto `authResult`. Este objeto contiene las siguientes propiedades: `accessToken`, `idToken`, `state`, `refreshToken` e `idTokenPayload`.

Un ejemplo de uso del evento `authenticated`:

export const codeExample2 = `var Auth = (function() {

<div id="resumeauth">
  ## resumeAuth()
</div>

Este método solo puede usarse cuando estableces la opción [auth.autoParseHash](/docs/es/libraries/lock/lock-configuration) en `false`. Tendrás que llamar a `resumeAuth` para completar el flujo de autenticación. Este método es útil cuando usas un router del lado del cliente que utiliza `#` para gestionar las URL (angular2 con `useHash` o react-router con `hashHistory`).

* **hash** &#123;String&#125;: El fragmento hash recibido de la redirección.
* **callback** &#123;Function&#125;: Se invocará una vez finalizado el análisis. Recibe un error (si lo hay) como primer argumento y el resultado de la autenticación como segundo. Si no hay ningún hash disponible, ambos argumentos serán `null`.

```js lines
lock.resumeAuth(hash, function(error, authResult) {
  if (error) {
    alert("Could not parse hash");
  }
  //Este es solo un ejemplo; no debes registrar Access Tokens en producción.
  console.log(authResult.accessToken);
});
```

<div id="checksession">
  ## checkSession()
</div>

El método `checkSession` te permite obtener un nuevo token de Auth0 para un usuario que ya está autenticado con Auth0 en tu dominio. Acepta los siguientes parámetros:

* **options** &#123;Object&#125;: Opcional. Acepta cualquier parámetro válido de OAuth2 que normalmente se enviaría a `/authorize`. Si los omites, usará los que se proporcionaron al inicializar Auth0.
* **callback** &#123;Function&#125;: Se invocará con el resultado de la renovación del token. Recibe un error (si lo hay) como primer argumento y el resultado de la autenticación como segundo.

```js lines
lock.checkSession({}, function(err, authResult) {
  // manejar el error o los nuevos tokens
});
```

<div id="logout">
  ## logout()
</div>

Cierra la sesión del usuario.

* **options** &#123;Object&#125;: Es opcional y sigue las mismas reglas que `logout()` de auth0.js.

```js lines
lock.logout({
  returnTo: 'https://myapp.com/bye-bye'
});
```
