title: Parámetros de autenticación de Lock
---

Puedes enviar parámetros al iniciar sesión agregándolos al objeto `options`. El siguiente ejemplo agrega un parámetro `state` con un valor de `'foo'`.

```js lines
var options = {
  auth: {
    params: {state: 'foo'},
  }
};
```

El ejemplo anterior sería equivalente a iniciar el inicio de sesión con `https://{yourDomain}/authorize?state=foo&...`.

Se admiten los siguientes parámetros: `scope`, `device`, `nonce` y `state`.

<div id="supported-parameters">
  ## Parámetros admitidos
</div>

### scope {string}

```js lines
var options = {
  auth: {
    params: {scope: 'openid email user_metadata app_metadata picture'},
  }
};
```

Hay distintos valores admitidos para `scope`. Ten en cuenta que los <Tooltip tip="JSON Web Token (JWT): formato estándar del token de ID (y a menudo del token de acceso) utilizado para representar de forma segura claims entre dos partes." cta="Ver glosario" href="/docs/es/glossary?term=JWTs">JWT</Tooltip> se envían en cada solicitud a la API, por lo que conviene mantenerlos lo más pequeños posible.

El valor predeterminado de `scope` en Lock es `openid profile email`. Este valor mínimo de `scope` es necesario para que la funcionalidad **La última vez que iniciaste sesión con** funcione correctamente.

<Card title="Ejecutar Lock localmente">
  Si no especificas manualmente al menos el valor predeterminado de `scope`, `openid profile email`, al inicializar Lock, y ejecutas tu sitio web desde `http://localhost` o `http://127.0.0.1`, obtendrás el siguiente error en la consola del navegador:

  `Consent required. When using getSSOData, the user has to be authenticated with the following scope: openid profile email`

  Eso **no** ocurrirá cuando ejecutes tu aplicación en Production o si especificas el `scope` `openid profile email`. Puedes obtener más información en el documento [consentimiento del usuario y aplicaciones de terceros](/docs/es/get-started/applications/third-party-applications/user-consent-and-third-party-applications).
</Card>

Para obtener más información sobre los scopes, consulta la [página de documentación sobre scopes](/docs/es/get-started/apis/scopes).

<div id="example-retrieve-a-token">
  #### Ejemplo: obtener un token
</div>

En Lock, si deseas recibir un token que permita obtener los datos del perfil del usuario, debes agregar el parámetro `scope`.

```js lines
var options = {
  auth: {
    params: {
        scope: 'openid profile'
    }
  }
};
```

También existe una opción de configuración `connectionScopes` para Lock que te permite especificar scopes para una conexión concreta. Esto resulta útil si quieres empezar con un conjunto de scopes (definidos en el Dashboard) y, más adelante, solicitar permisos o atributos adicionales de una conexión específica. Obtén más información en la página [Lock Configuration Options](/docs/es/libraries/lock/lock-configuration).

### state {string}

El parámetro `state` es un valor arbitrario que se mantendrá entre redirecciones. Es útil para mitigar los [ataques XSRF](https://en.wikipedia.org/wiki/Cross-site_request_forgery) y para cualquier información de contexto, [como una URL de retorno](/docs/es/authenticate/login/logout/redirect-users-after-logout), que puedas necesitar una vez finalizado el proceso de autenticación. Si no se proporciona un parámetro `state` personalizado, Lock generará uno automáticamente. Para obtener más información, consulta [Parámetro state](/docs/es/secure/attack-protection/state-parameters).

### nonce {string}

El parámetro `nonce` se utiliza para prevenir ataques de repetición y Lock lo generará automáticamente si no se proporciona un valor personalizado.

### device {string}

El parámetro `device` indica el nombre del dispositivo o navegador que solicita la autenticación.
