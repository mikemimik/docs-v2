Con Auth0 puedes ofrecer a los usuarios múltiples métodos de autenticación. Esto es importante en aplicaciones SaaS o multi-tenant, donde muchas organizaciones usan una sola aplicación. Cada organización puede usar sistemas diferentes, como LDAP, Active Directory, Google Workspace o almacenes de nombre de usuario/contraseña.

En Auth0, puedes asociar diferentes conexiones (métodos de autenticación) a aplicaciones específicas, o directamente a un tenant (como [conexiones de dominio](/docs/es/authenticate/identity-providers/promote-connections-to-domain-level)). Cuando un usuario inicia sesión, será necesario seleccionar una de estas conexiones para usarla.

<Frame>![Home Realm Discovery en Lock](/docs/images/cdy7uua7fh8z/1wnZRx9V1cyyV1nRgsu9fD/f2569441334d3b277702db225045e102/2023-09-22_14-18-30.png)</Frame>

La selección del <Tooltip tip="Proveedor de identidad (IdP): Servicio que almacena y administra identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=Identity+Providers">Identity Providers</Tooltip> adecuado entre varias opciones se denomina &quot;Home Realm Discovery&quot;.

Si usas como máximo una conexión de base de datos y cero o más conexiones sociales, el proceso de selección es sencillo. El usuario hará una de estas dos cosas:

* Hará clic en uno de los botones de proveedores de identidad social (por ejemplo, &quot;Iniciar sesión con Google&quot;)
* Introducirá su correo electrónico y contraseña (es decir, &quot;Usaré la conexión de base de datos&quot;).

Pero si la aplicación o el tenant tiene habilitados otros tipos de conexión (como conexiones empresariales o varias conexiones de base de datos), el proceso de selección puede ser más complejo. ¿Cómo indicas que un usuario quiere usar una conexión de base de datos específica si hay más de una habilitada? ¿Y si un usuario quiere usar una conexión empresarial para iniciar sesión mediante <Tooltip tip="Inicio de sesión único (SSO): Servicio que, después de que un usuario inicia sesión en una aplicación, lo inicia automáticamente en otras aplicaciones." cta="Ver glosario" href="/docs/es/glossary?term=Single+Sign-on">inicio de sesión único</Tooltip> (SSO)?

Si implementas [una interfaz de inicio de sesión personalizada](/docs/es/customize/login-pages/classic-login/customize-with-lock-sdk), tienes control total sobre el flujo de autenticación. Puedes elegir la conexión según el contexto (como la dirección de correo electrónico proporcionada) o preguntárselo al usuario y luego pasar el parámetro `connection` a uno de los [métodos de login](/docs/es/libraries/auth0js) de Auth0.js.

<div id="lock-and-multiple-connections">
  ## Lock y múltiples conexiones
</div>

Lock tiene funcionalidad integrada para seleccionar el proveedor de identidad. En las conexiones sociales, muestra los logotipos de todas las que estén habilitadas en una aplicación determinada. También proporciona campos de username/email y contraseña si está habilitada una conexión de base de datos o una conexión de Active Directory.

Solo verá un botón de inicio de sesión si es la única conexión habilitada para la aplicación (en la experiencia de [Classic Universal Login](/docs/es/authenticate/login/auth0-universal-login/universal-login-vs-classic-login/classic-experience)). De lo contrario, deberá usar una [UI personalizada](/docs/es/customize/login-pages/classic-login/customize-with-lock-sdk) o [New Universal Login](/docs/es/authenticate/login/auth0-universal-login/universal-login-vs-classic-login/universal-experience), que sí admite tener un botón para cada conexión social y empresarial habilitada.

<div id="using-email-domains-with-enterprise-connections">
  ## Uso de dominios de correo electrónico con conexiones empresariales
</div>

Una función adicional de Lock es el uso de dominios de correo electrónico para enrutar solicitudes de autenticación. Las conexiones empresariales en Auth0 pueden mapearse a `domains`. Por ejemplo, al configurar ADFS o un proveedor de identidad <Tooltip tip="Security Assertion Markup Language (SAML): Protocolo estandarizado que permite a dos partes intercambiar información de autenticación sin una contraseña." cta="Ver glosario" href="/docs/es/glossary?term=SAML">SAML</Tooltip>-P:

<Frame>![Configuración de proveedores de identidad ADFS o SAML-P](/docs/images/cdy7uua7fh8z/6c8fqJpKCPNSqlop6ZJGYK/b20b1125302b8f5511825be1299e0f96/enterprise-connection.png)</Frame>

Si una conexión tiene dominios asociados, el campo de contraseña se desactiva automáticamente cuando un usuario escribe una dirección de correo electrónico con un dominio asociado.

En el ejemplo anterior, el dominio `auth0.com` se ha asociado a una conexión empresarial.

Ten en cuenta que puedes asociar varios dominios a una sola conexión.

<div id="selecting-among-multiple-database-connections">
  ## Seleccionar entre varias conexiones de base de datos
</div>

Si tu aplicación tiene habilitadas varias conexiones de base de datos, Lock necesita saber cuál debe usar. Puedes proporcionar la [opción `connectionResolver`](https://github.com/auth0/lock#other-options), que recibe una función para decidir qué conexión usar en función de la entrada del usuario y del contexto. En este ejemplo, se usa una conexión de base de datos alternativa si el dominio del correo electrónico es &quot;auth0.com&quot;:

```js lines
var options = {
  connectionResolver: function (username, context, cb) {
    var domain = username.indexOf('@') !== -1 && username.split('@')[1];
    if (domain && domain ==='auth0.com') {
      // Si el username es test@auth0.com, la conexión utilizada será la conexión `auth0-users`.
      cb({ type: 'database', name: 'auth0-users' });
    } else {
      // Usar el enfoque predeterminado para determinar la conexión
      cb(null);
    }
  }
}
```

Puedes usar la opción [`defaultDatabaseConnection`](/docs/es/libraries/lock/lock-configuration#defaultdatabaseconnection-string-) para especificar la conexión de base de datos que se usará de forma predeterminada.

<div id="filtering-available-connections-programmatically">
  ## Filtrar mediante programación las conexiones disponibles
</div>

La [opción `allowedConnections`](/docs/es/libraries/lock/lock-configuration#allowedconnections-array-) de Lock te permite indicar cuáles de las conexiones disponibles se deben mostrar al usuario como opción.

Esto te permite adaptar la experiencia en función de información o contexto adicionales (p. ej., &quot;Haz clic aquí para iniciar sesión como estudiante, o aquí para iniciar sesión como profesor&quot;).

export const codeExample = `var lock = new Auth0Lock(
  '{yourClientId}',
  '{yourDomain}',
  {
    allowedConnections: ['yourConnection'];
  }
);`;

Consulta la [documentación de la API](/docs/es/libraries/lock/lock-api-reference) del método `show` para obtener más información.

<div id="sending-realm-information-from-the-application">
  ## Envío de información de realm desde la aplicación
</div>

A veces, la aplicación que solicita la autenticación puede saber de antemano qué realm pretende usar el usuario. Por ejemplo, una aplicación multi-tenant podría usar URL con un formato como: `https://{customer}.yoursite.com` o `https://www.yoursite.com/{customer}`. Cuando un usuario llega a tu aplicación mediante la vanity URL, puedes tomar ese valor de `tenant` y pasarlo como `login_hint` en la solicitud `authorize`:

`https://{yourDomain}/authorize?client_id=[...]&login_hint={customer}`

`login_hint` es una pista para el <Tooltip tip="Servidor de autorización: servidor centralizado que contribuye a definir los límites del acceso de un usuario. Por ejemplo, tu servidor de autorización puede controlar los datos, las tareas y las funciones disponibles para un usuario." cta="Ver glosario" href="/docs/es/glossary?term=authorization+server">servidor de autorización</Tooltip> (Auth0) que indica qué podría usar el usuario para iniciar sesión. En este caso, según la URL a la que llegó el usuario, tratamos &quot;customer&quot; como el realm.

El código predeterminado de la hosted login page lo usa para completar previamente el campo de correo electrónico en Lock, pero podemos modificar el código para cambiar la conexión de base de datos predeterminada que se usará si se proporciona un realm en lugar de una dirección de correo electrónico real:

```js lines expandable
// de la plantilla predeterminada de la página de inicio de sesión alojada
var config = JSON.parse(decodeURIComponent(escape(window.atob('@@config@@'))));
[...]

var loginHint = config.extraParams.login_hint;
var realmHint;

// si el indicio de inicio de sesión no es una dirección de correo electrónico, lo tratamos como un indicio de realm
if (loginHint && loginHint.indexOf('@') < 0) {
  realmHint = loginHint;
  loginHint = null;
}

// ahora asignamos el realm a una base de datos real
var defaultDatabaseConnection;
if (realmHint === 'acme') {
  defaultDatabaseConnection = 'acme-users';
} else if (realmHint === 'auth0') {
  defaultDatabaseConnection = 'auth0-DB';
}
    
// Al configurar Lock, proporcionamos los valores obtenidos anteriormente
var lock = new Auth0Lock(config.clientID, config.auth0Domain, {
  [...] // otras opciones
  prefill: loginHint ? { email: loginHint, username: loginHint } : null,
  defaultDatabaseConnection: defaultDatabaseConnection
}
```

El código anterior es, por supuesto, solo un ejemplo. Podrías ampliar esta lógica para filtrar las conexiones sociales o para establecer una conexión predeterminada que se use incluso si se proporciona una dirección de correo electrónico como `login_hint`.

Mapear &quot;customer&quot; como un realm es una decisión de diseño arbitraria para este ejemplo. Pero, en general, conviene aislar las aplicaciones del concepto concreto de &quot;conexión&quot; que se usa en Auth0 y utilizar en su lugar el concepto más abstracto de &quot;realm&quot;, posiblemente haciendo un mapeo de realm a conexión dentro de la página de login alojada (donde es más fácil hacer cambios si es necesario).
