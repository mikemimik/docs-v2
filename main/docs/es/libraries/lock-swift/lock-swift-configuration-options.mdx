A continuación, se enumeran numerosas opciones para configurar el comportamiento de Lock. Además, en la página de [Opciones de personalización de estilo](/docs/es/libraries/lock-swift/lock-swift-customization) también encontrarás varias opciones para modificar la apariencia y el estilo de Lock.

<div id="configuring-locks-behavior">
  ## Configuración del comportamiento de Lock
</div>

Puedes añadir opciones de configuración a la inicialización de Lock mediante `withOptions`.

```swift lines
Lock
  .classic()
  .withOptions {
    $0.closable = true
    $0.usernameStyle = [.Username]
    $0.allow = [.Login, .ResetPassword]
  }
  .present(from: self)
```

<div id="behavior-options">
  ## Opciones de comportamiento
</div>

<div id="closable">
  ### closable
</div>

Permite que el usuario pueda cerrar Lock. De forma predeterminada, es `false`.

```swift lines
.withOptions {
  $0.closable = true
}
```

<div id="scope">
  ### scope
</div>

Scope utilizado para la autenticación. El valor predeterminado es `openid`. Devuelve no solo el **<Tooltip tip="Token de acceso: credencial de autorización, en forma de cadena opaca o JWT, que se usa para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=Access+Token">token de acceso</Tooltip>**, sino también un **<Tooltip tip="Token de identidad: credencial destinada al propio cliente, en lugar de para acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=ID+Token">token de identidad</Tooltip>**, que es un <Tooltip tip="Token de identidad: credencial destinada al propio cliente, en lugar de para acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=JSON+Web+Token">JSON Web Token</Tooltip> (JWT) que contiene información del usuario. Consulta la documentación sobre [Scopes](/docs/es/get-started/apis/scopes) para obtener más información sobre los scopes de autenticación.

```swift lines
.withOptions {
  $0.scope = "openid name email picture"
}
```

<div id="refresh-tokens">
  #### Tokens de actualización
</div>

Especificar el scope `offline_access` en las opciones de Lock permitirá que se devuelva un [token de actualización](/docs/es/secure/tokens/refresh-tokens) junto con el access&#95;token y el id&#95;token. <Tooltip tip="Token de actualización: token que se usa para obtener un nuevo token de acceso sin obligar a los usuarios a iniciar sesión de nuevo." cta="Ver glosario" href="/docs/es/glossary?term=Refresh+Tokens">Tokens de actualización</Tooltip> se pueden guardar y usar para obtener un nuevo token de acceso cuando el anterior caduque. Para obtener más información sobre el uso de tokens de actualización para la autenticación de Auth0, consulta la documentación de referencia del [SDK de Auth0.swift](/docs/es/libraries/auth0-swift), que usarías para implementar tokens de actualización, o la [Guía Quickstart de Swift](/docs/es/quickstart/native/ios-swift), que ofrece un ejemplo completo del uso de Auth0 en el desarrollo con Swift, incluida la gestión de tokens de actualización.

<div id="termsofservice">
  ### termsOfService
</div>

De forma predeterminada, Lock utilizará los Términos de servicio y la política de privacidad de Auth0, pero se pueden indicar otras URL para enlazar a otros términos y políticas.

```swift lines
.withOptions {
  $0.termsOfService = "https://mycompany.com/terms"
  $0.privacyPolicy = "https://mycompany.com/privacy"
}
```

<div id="show-terms-of-service">
  ### Mostrar los Términos de servicio
</div>

Las conexiones de base de datos muestran el cuadro de diálogo de Términos de servicio. De forma predeterminada, es `true`. Ten en cuenta que los Términos de servicio siempre se mostrarán si el indicador `mustAcceptTerms` está habilitado.

```swift lines
.withOptions {
    $0.showTerms = true
}
```

<div id="require-users-to-accept-the-terms-of-service">
  ### Exigir que los usuarios acepten los Términos de servicio
</div>

La conexión de base de datos requiere la aceptación explícita de los Términos de servicio.

```swift lines
.withOptions {
    $0.mustAcceptTerms = true
}
```

<div id="web-authentication-options">
  ## Opciones de Web Authentication
</div>

<div id="leeway">
  ### leeway
</div>

Desajuste horario utilizado para la validación del token de identidad. Amplía el intervalo de tiempo durante el cual el token de identidad seguirá considerándose válido, para tener en cuenta la diferencia entre la hora del servidor y la del cliente. De forma predeterminada, es de **60000 milisegundos** (60 segundos).

```swift lines
.withOptions {
  $0.leeway = 30000 // 30 segundos
}
```

<div id="maxage">
  ### maxAge
</div>

Tiempo máximo permitido (en milisegundos) desde la última vez que el usuario se autenticó. Se usa para la validación del token de identidad. Si se establece, el token de identidad contendrá un claim `auth_time` con la marca de tiempo de la autenticación. El valor predeterminado es `nil`.

```swift lines
.withOptions {
  $0.maxAge = 86400000 // 1 día
}
```

<div id="database-options">
  ## Opciones de Database
</div>

<div id="allow">
  ### permitir
</div>

A qué pantallas de Database se podrá acceder; de forma predeterminada, se habilitan todas, como `.Login, .Signup, .ResetPassword`.

```swift lines
.withOptions {
  $0.allow = [.Login, .ResetPassword]
}
```

<div id="initialscreen">
  ### initialScreen
</div>

La primera pantalla que se le mostrará al usuario. La predeterminada es `.Login`; otras opciones son `.Signup` y `ResetPassword`.

```swift lines
.withOptions {
  $0.initialScreen = .Login
}
```

<div id="usernamestyle">
  ### usernameStyle
</div>

Especifique el tipo de identificador que requerirá el inicio de sesión. De forma predeterminada, es `[.Username, .Email]`, pero también puede aceptar `[.Username]` o `[.Email]`. Sin embargo, es importante tener en cuenta que esta opción solo está activa si ha establecido el indicador `requires_username` en `true` en su [Auth0 Dashboard](https://manage.auth0.com/#/).

```swift lines
.withOptions {
  $0.usernameStyle = [.Username]
}
```

<div id="custom-signup-fields">
  #### Campos de registro personalizados
</div>

Al registrarse, la información predeterminada requerida es el correo electrónico y la contraseña del usuario. Puede ampliar los datos que recopila según sea necesario. Los campos de registro adicionales que capture aquí se almacenarán en `user_metadata`; puede obtener más información al respecto en [Metadata](/docs/es/manage-users/user-accounts/metadata). Tenga en cuenta que debe especificar el ícono que se usará con su campo de texto personalizado.

```swift lines
.withOptions {
  $0.customSignupFields = [
    CustomTextField(name: "first_name", placeholder: "First Name", icon: LazyImage(name: "ic_person", bundle: Lock.bundle)),
    CustomTextField(name: "last_name", placeholder: "Last Name", icon: LazyImage(name: "ic_person", bundle: Lock.bundle))
  ]
}
```

También puedes especificar iconos de otros bundles, como en el siguiente ejemplo:
CustomTextField(name: &quot;slack&#95;handle&quot;, placeholder: &quot;Nombre de usuario de Slack&quot;, icon: LazyImage(name: &quot;ic&#95;slack&quot;, bundle: Bundle(identifier: &quot;CustomBundle&quot;)))

<div id="enterprise-options">
  ## Opciones de las conexiones empresariales
</div>

También hay opciones de configuración específicas para las conexiones empresariales:

<div id="enterpriseconnectionusingactiveauth">
  ### enterpriseConnectionUsingActiveAuth
</div>

De forma predeterminada, las conexiones empresariales usarán Web Authentication. Sin embargo, puedes especificar qué conexiones usarán en su lugar la autenticación mediante credenciales y pedirán un nombre de usuario y una contraseña.

```swift lines
.withOptions {
  $0.enterpriseConnectionUsingActiveAuth = ["enterprisedomain.com"]
}
```

<div id="activedirectoryemailasusername">
  ### activeDirectoryEmailAsUsername
</div>

En el modo de autenticación por credenciales, ¿se debe exigir al usuario que use su correo electrónico como identificador? El valor predeterminado es `false`; en su lugar, se requiere un nombre de usuario.

```swift lines
.withOptions {
  $0.activeDirectoryEmailAsUsername = true
}
```

<div id="logging-options">
  ## Opciones de registro
</div>

Lock ofrece opciones para activar y desactivar fácilmente el registro, así como para ajustar otras configuraciones relacionadas.

<div id="loglevel">
  ### logLevel
</div>

De forma predeterminada, es `.off`; se admiten los niveles de registro de Syslog.

```swift lines
.withOptions {
  $0.logLevel = .all
}
```

<div id="loghttprequest">
  ### logHttpRequest
</div>

Indica si se registran o no las solicitudes de la API de Auth0.swift. De forma predeterminada, es `false`.

```swift lines
.withOptions {
  $0.logHttpRequest = true
}
```

<div id="loggeroutput">
  ### loggerOutput
</div>

Especifica el controlador de salida del logger; de forma predeterminada, usa la instrucción `print`.

```swift lines
.withOptions {
  $0.loggerOutput = CleanroomLockLogger()
}
```

En el código anterior, se ha configurado loggerOutput para usar [CleanroomLogger](https://github.com/emaloney/CleanroomLogger). Esto normalmente se consigue implementando el protocolo loggerOutput. Por supuesto, puedes usar tu biblioteca de logging favorita. A continuación se muestra un ejemplo de uso para gestionar la salida del logger con CleanroomLogger.

```swift lines
class CleanroomLockLogger: LoggerOutput {
  func message(_ message: String, level: LoggerLevel, filename: String, line: Int) {
    let channel: LogChannel?
    switch level {
    case .debug:
        channel = Log.debug
    case .error:
        channel = Log.error
    case .info:
        channel = Log.info
    case .verbose:
        channel = Log.verbose
    case .warn:
        channel = Log.warning
    default:
        channel = nil
    }
    channel?.message(message, filePath: filename, fileLine: line)
  }
}
```
