このドキュメントは、API から予期しないレスポンスが返ってきた場合に、設定の問題を切り分けるのに役立ちます。ご利用のアカウント向けに設定されたサンプルを使ってこのクイックスタートを進められるよう、ログインしておくことをおすすめします。

JWT の検証が正しく設定されていれば、リクエスト時に API から適切なレスポンスを取得できます。ただし、API から 401 (Unauthorized) が返される場合は、JWT ミドルウェアの設定が渡された JWT と一致していないことが原因です。

このドキュメントは、JWT ミドルウェアの設定に関するトラブルシューティングに役立ちます。

<div id="how-does-a-token-get-validated">
  ## トークンはどのように検証されるのでしょうか？
</div>

JWT を検証する際には、いくつか確認すべき点があります。

1. **トークンの形式は正しいですか？** つまり、このトークンは JSON Web Token (JWT) の構造に準拠していますか？ JWT の構造について詳しくは、[JWT の構造に関するこのセクション](/docs/ja/secure/tokens/json-web-tokens#what-is-the-json-web-token-structure-)を参照してください。
2. **トークンは改ざんされていませんか？** JWT の最後の部分は署名です。この署名は、トークンが実際に送信者によって署名されたものであり、途中で一切変更されていないことを確認するために使われます。
3. **トークンは有効期間内に受信されていますか？** JWT は、指定された期間内でのみ有効です (`exp` クレームで表されます) 。
4. **トークンは想定された Authority から送られてきていますか？** これには 2 つの要素があります

   * **署名の検証**: JWT が、発行元 Authority によって発行された鍵を使って正しく署名されていることを確認できますか？
   * **Issuer 値**: Issuer は `iss` クレームで定義されます。このクレームの値は、アプリケーションが想定しているものと一致していますか？
5. **トークンは現在のアプリケーション向けのものですか？** つまり、JWT の `aud` クレームは、アプリケーションが想定している値と一致していますか？

<div id="inspecting-a-token">
  ## トークンの確認
</div>

JWT を手早く確認するには、[JWT.io](https://jwt.io/) のウェブサイトを使うのが簡単です。ここには便利なデバッガーがあり、JWT の形式が正しいかをすぐに確認できるほか、各クレームの値も確認できます。

<Frame>![JWT.io で JWT をデバッグする](https://cdn2.auth0.com/docs/1.14550.0/media/articles/server-apis/aspnet-core-webapi/jwt-io-debugger-rs256.png)</Frame>

上のスクリーンショットを見ると、このトークンが **RS256** アルゴリズムで署名されていることがわかります。トークンの **Issuer** は **https://jerrie.auth0.com/**、**Audience** は **https://rs256.test.api** です。

つまり、JWT 検証の設定登録にあるこれらの値は **完全に** 一致している必要があります。たとえば Issuer の末尾のスラッシュも含めて、次のように指定します。

```
audience = "https://rs256.test.api",
domain = "https://jerrie.auth0.com/"
};
```

HS256 で署名されたトークンの場合、デバッガービューは少し異なります。

上のスクリーンショットから、トークンが **HS256** アルゴリズムで署名されていることがわかります。トークンの **Issuer** は **https://jerrie.auth0.com/**、**Audience** は **https://hs256.test.api** です。
