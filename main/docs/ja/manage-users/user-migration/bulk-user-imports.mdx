[Create Import Users Job](https://auth0.com/docs/api/management/v2#!/Jobs/post_users_imports) エンドポイントを使用すると、ユーザーデータを Auth0 に一括インポートできます。一括インポートは、既存のデータベースやサービスから Auth0 へユーザーを移行する際に便利です。

<Warning>
  複数の移行方法 (たとえば、自動移行の後にユーザーの一括インポートを行うなど) を使用しようとすると、`DUPLICATED_USER` エラーが発生する場合があります。このエラーは、ユーザーが Auth0 の内部ユーザーストアには存在している一方で、テナントには存在していないことを示します。このエラーを解消するには、Auth0 Management API の [Delete a Connection User](https://auth0.com/docs/api/management/v2#!/Connections/delete_users_by_email) エンドポイントを使用してそのユーザーを削除し、再度インポートを試してください。
</Warning>

<div id="prerequisites">
  ## 前提条件
</div>

ユーザーのインポートジョブを実行する前に、以下を行ってください。

* ユーザーのインポート先となるデータベース接続を設定し、少なくとも 1 つのアプリケーションで有効にします。
* パスワードをインポートする場合は、サポート対象のアルゴリズムのいずれかでハッシュ化されていることを確認してください。サポートされていないアルゴリズムでハッシュ化されたパスワードを持つユーザーは、一括インポート後の初回ログイン時にパスワードをリセットする必要があります。
* <Tooltip tip="多要素認証（MFA）: SMS で送信されるコードなど、ユーザー名とパスワードに加えて認証要素を使用するユーザー認証プロセス。" cta="用語集を見る" href="/docs/ja/glossary?term=MFA">MFA</Tooltip> の登録情報をインポートする場合は、サポート対象の種類 (`email`、`phone`、または `totp`) であることを確認してください。
* ジョブのエンドポイントへのリクエスト用に、[Management API トークンを取得](/docs/ja/secure/tokens/access-tokens/management-api-access-tokens)します。

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Auth0 テナントのエクスポートファイルを使用している場合は、エクスポートしたファイルを `ndjson` から [JSON](/docs/ja/manage-users/user-migration/bulk-user-exports#json-compatible-format) に変換する必要があります。同じユーザー ID を維持するには、インポートするすべてのユーザー ID から `auth0| プレフィックス` を削除する必要があります。

  インポート処理では、インポートされたユーザー ID に自動的に `auth0| プレフィックス` が追加されます。インポート前に `auth0|` プレフィックスを削除しないと、ユーザー ID は `auth0|auth0|...` として返されます。
</Callout>

<div id="create-users-json-file">
  ## ユーザーのJSONファイルを作成
</div>

Auth0 にインポートするユーザーデータを含む JSON ファイルを作成します。ユーザーデータを JSON ファイルにエクスポートする方法は、現在使用しているユーザーデータベースによって異なります。<Tooltip tip="Management API: 顧客が管理タスクを実行できるようにする製品です。" cta="用語集を表示" href="/docs/ja/glossary?term=Management+API">Management API</Tooltip> エンドポイントは、JSON ファイルを分割して送信することを想定しています。そのため、`fs.readFileSync` ではなく `fs.createReadStream` を使用する必要があります。エンドポイントが想定しているのは、JSON ファイル全体ではなく、パイプされた読み取りストリームです。

JSON ファイルの schema の詳細や例については、[Bulk Import Database Schema and Examples](/docs/ja/manage-users/user-migration/bulk-user-import-database-schema-and-examples) を参照してください。

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  一括インポートのファイルサイズ上限は 500KB です。データがこのサイズを超える場合は、複数回に分けてインポートを開始する必要があります。
</Callout>

<div id="request-bulk-user-import">
  ## 一括ユーザーインポートをリクエストする
</div>

一括ユーザーインポートジョブを開始するには、[Create Import Users Job](https://auth0.com/docs/api/management/v2#!/Jobs/post_users_imports)エンドポイントに `POST` リクエストを送信します。`MGMT_API_ACCESS_TOKEN`、`USERS_IMPORT_FILE.json`、`CONNECTION_ID`、`EXTERNAL_ID` のプレースホルダー値は、それぞれ Management API の<Tooltip tip="Access Token: API へのアクセスに使用される、オペークな文字列または JWT の形式の Authorization 資格情報。" cta="用語集を見る" href="/docs/ja/glossary?term=Access+Token">アクセストークン</Tooltip>、ユーザー JSON ファイル、データベースの[接続 ID](/docs/ja/authenticate/identity-providers/locate-the-connection-id)、external ID に必ず置き換えてください。

| パラメータ                   | 説明                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| ----------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `users`                 | インポートするユーザーを含む JSON 形式のファイルです。                                                                                                                                                                                                                                                                                                                                                                                                         |
| `connection_id`         | ユーザーの挿入先となる接続の ID です。この ID は、[GET /api/v2/connections](https://auth0.com/docs/api/management/v2#!/Connections/get_connections) エンドポイントを使用して取得できます。                                                                                                                                                                                                                                                                                     |
| `upsert`                | ブール値です。デフォルトは `false` です。`false` に設定すると、メールアドレス、ユーザー ID、`phone`、またはユーザー名が一致する既存ユーザーは失敗します。`true` に設定すると、メールアドレスが一致する既存ユーザーは更新されますが、更新できるのはアップサート可能な属性のみです。インポート時にアップサートできるユーザープロファイルのフィールド一覧については、[User Profile Structure: User profile attributes](/docs/ja/manage-users/user-accounts/user-profiles/user-profile-structure#user-profile-attributes) を参照してください。注: インポートファイルに重複したユーザーエントリが含まれていると、エラーになります。この場合、Auth0 は挿入後に更新を行うことはありません。 |
| `external_id`           | 複数のジョブを関連付けるために使用できる、ユーザー定義の任意の文字列です。ジョブのステータスレスポンスの一部として返されます。                                                                                                                                                                                                                                                                                                                                                                        |
| `send_completion_email` | ブール値です。デフォルトは `true` です。`true` に設定すると、インポートジョブの完了時に、すべてのテナント所有者に完了メールが送信されます。メールを送信したくない場合は、このパラメータを明示的に `false` に設定する必要があります。                                                                                                                                                                                                                                                                                                         |

リクエストが成功すると、次のようなレスポンスが返されます。

```json lines
{
  "status": "pending",
  "type": "users_import",
  "created_at": "",
  "id": "job_abc123",
  "connection_id": "CONNECTION_ID",
  "upsert": false,
  "external_id": "EXTERNAL_ID",
  "send_completion_email": true
}
```

返されたエンティティは、インポートジョブを表します。

ユーザーのインポートジョブが完了し、`send_completion_email` が `true` に設定されている場合は、テナント管理者に、そのジョブが成功したか失敗したかを知らせるメールが送信されます。失敗したジョブに関するメールでは、ユーザーのインポート時に users JSON ファイルを解析できなかったことが管理者に通知される場合があります。

<div id="concurrent-import-jobs">
  ### 同時実行されるインポートジョブ
</div>

[Create Import Users Job](https://auth0.com/docs/api/management/v2#!/Jobs/post_users_imports) endpoint では、同時に実行できるインポートジョブは 2 つまでに制限されています。保留中のジョブが 2 つある状態でさらにジョブを request すると、`429 Too Many Requests` レスポンスが返されます。

```json lines
{
  "statusCode": 429,
  "error": "Too Many Requests",
  "message": "There are 2 active import users jobs, please wait until some of them are finished and try again
}
```

**保留中**

```json lines
{
  "status": "pending",
  "type": "users_import",
  "created_at": "",
  "id": "job_abc123",
  "connection_id": "CONNECTION_ID",
  "external_id": "EXTERNAL_ID"
}
```

**完了**

ジョブが完了すると、ジョブのステータスレスポンスには、成功、失敗、追加、更新されたレコードの合計数が含まれます。

```json lines
{
  "status": "completed",
  "type": "users_import",
  "created_at": "",
  "id": "job_abc123",
  "connection_id": "CONNECTION_ID",
  "external_id": "EXTERNAL_ID",
  "summary": {
    "failed": 0,
    "updated": 0,
    "inserted": 1,
    "total": 1
  }
}
```

**失敗**

ジョブでエラーが発生すると、ステータスは失敗として返されます。ただし、無効なメールアドレスなど、ユーザー情報に不備があっても、ジョブ全体が失敗することはありません。

```json lines
{
  "status": "failed",
  "type": "users_import",
  "created_at": "",
  "id": "job_abc123",
  "connection_id": "CONNECTION_ID",
  "external_id": "EXTERNAL_ID",
}
```

失敗したエントリの詳細については、以下の**失敗したエントリの取得**を参照してください。

<div id="job-timeouts">
  ## ジョブ のタイムアウト
</div>

すべてのユーザーインポート ジョブ は、**2 時間**で timeout になります。この時間内に ジョブ が完了しない場合は、失敗としてマークされます。

さらに、ジョブ 関連のデータはすべて 24 時間後に自動的に削除され、その後はアクセスできなくなります。そのため、**ジョブ の結果は、お使いの任意の保存方法で保存しておくことを強く推奨します**。

<div id="best-practices-for-large-scale-migrations">
  ## 大規模な移行のベストプラクティス
</div>

大量のユーザーをインポートする際 (10件以上のジョブが必要な場合) は、移行を成功させるため、以下の推奨事項に従ってください：

<div id="file-size-and-chunking">
  ### ファイルサイズとチャンク分割
</div>

ユーザーレコードは、1ファイルあたり400KB未満の小さなチャンクに分割してください。こうすることで、500KBのファイルサイズ上限を十分下回りつつ、各ユーザーのメタデータ項目数が10未満であれば、1ジョブあたり約1,000ユーザーを扱えます。

<div id="job-scheduling-and-concurrency">
  ### ジョブのスケジューリングと同時実行
</div>

Auth0 では、テナントごとに同時実行できるユーザーインポートのジョブ数が 2 つまでに制限されているため、ジョブスケジューラフレームワーク ([Bull](https://github.com/OptimalBits/bull) や [Agenda](https://github.com/agenda/agenda) など) を使用して、次のことを行ってください。

* インポートジョブを計画的にスケジュールする
* 必要な同時実行制限を適用する
* 中断された場合に途中から再開できるよう、インポートの進行状況を追跡する
* ジョブの履歴と現在のステータスを管理する

<div id="error-handling-strategy">
  ### エラー処理戦略
</div>

* 1 人のユーザーで失敗しても、インポート処理を中断しないでください。すべてのユーザーをインポートし、問題には後から対処します。
* 生成されたすべてのジョブを確認し、失敗したレコードを新しいファイルにまとめる「finalizer」ジョブを実装します。
* まとめた失敗レコードのエラーを修正し、新しいジョブとしてインポートします。
* ネットワーク接続の切断や一時的な障害に対処するための再試行戦略を含めます。

<div id="upsert-mode-considerations">
  ### Upsertモードに関する注意点
</div>

`upsert` モードを有効にする際は、以下の点に注意してください。

* upsert を使用したインポートは、通常のインポートよりも時間がかかります。
* Upsert モードでは、`user_metadata` や `app_metadata` をマージできません。既存のメタデータは新しい値で完全に上書きされます。
* upsert モードは、既存のユーザーを更新する必要があり、これらの制限を理解している場合にのみ使用してください。

<div id="data-retention">
  ### データ保持
</div>

ジョブに関連するデータは、すべて24時間後に自動的に削除されます。長期的に追跡・分析できるよう、ジョブの結果とエラーの詳細は独自のストレージに保存してください。
