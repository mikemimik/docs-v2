<div id="step-5-use-identity-tokens-to-flow-identity">
  ## ステップ 5 - ID トークンを使用してアイデンティティ情報を渡す
</div>

この最後のステップでは、次のことを行います。

* <Tooltip tip="OpenID: アプリケーションがログイン情報を収集・保存することなく、ユーザーのアイデンティティを検証できるようにする認証のためのオープン標準です。" cta="用語集を表示" href="/docs/ja/glossary?term=OpenID">OpenID</Tooltip> <Tooltip tip="OpenID: アプリケーションがログイン情報を収集・保存することなく、ユーザーのアイデンティティを検証できるようにする認証のためのオープン標準です。" cta="用語集を表示" href="/docs/ja/glossary?term=JSON+Web+Token">JSON Web トークン</Tooltip> (JWT) を渡して、サービスにアイデンティティ情報を渡す。
* トークンを検証する。
* ペットの購入者を割り当てるため、プロファイル情報を抽出する。

<div id="use-an-identity-token">
  ## ID トークンを使用する
</div>

Lambda 関数を使用して、ユーザーに関する情報を処理したり取得したりできます。たとえば、購入トランザクション中に、<Tooltip tip="ID トークン: リソースへのアクセスではなく、クライアント自体を対象とした認証情報。" cta="用語集を見る" href="/docs/ja/glossary?term=identity+token">ID トークン</Tooltip> とともに返されたプロファイルからユーザー名を取得しました。ただし、ユーザー情報をアイデンティティ自体に埋め込むこともできます。このアイデンティティは JSON Web トークン (JWT) です。

JWT を使用する利点は次のとおりです。

1. JWT の真正性を検証できる。
2. 呼び出し元のユーザーが認証済みであることを確認できる (改ざんされている可能性のあるプレーンテキストのパラメータに依存せずに済む) 。

さらに、JWT を認可に使用すれば、Amazon API Gateway との IAM 連携を回避できます。ただし、認可に API Gateway を使用すると、Lambda 関数が呼び出される前に API 呼び出しを停止できる点に注意してください。

<div id="add-information-to-the-jwt">
  ### JWT に情報を追加する
</div>

ユーザー情報を JWT に追加する方法はいくつかあります。次の例では、ユーザーのメールアドレスを JWT に追加していますが、同じ考え方を他のユーザー情報にも適用できます。

<div id="use-rules">
  #### ルールを使用する
</div>

JWT にユーザーのメールアドレスを追加する方法の 1 つは、[ルール](/docs/ja/customize/rules)を使用することです。これは、認証するユーザーの JWT でこの値を常に利用できるようにしたい場合に適した方法です。

`login.js` では、このスコープが `auth.signin` に渡されるパラメータで指定されていることがわかります。

```js lines
$scope.login = function() {
    var params = {
        authParams: {
          scope: 'openid email'
        }
      };

    auth.signin(params, function(profile, token) {
      ...
    }
  }
```

JWT にはユーザーの完全なプロファイルを含めることもできますが、JWT は通常すべてのリクエストで送信されるため、本当に必要な情報だけを含めるようにしてください。

<div id="validate-the-jwt-token">
  ## JWT トークンを検証する
</div>

AWS Lambda コンソールでは、ブラウザーコンソールから Node.js コードを入力する場合に利用できる Node モジュールが限られているため、ID トークンを処理するには追加のモジュールを含めたうえで、Lambda 関数をパッケージとしてアップロードする必要があります。

詳細については、[Creating Deployment Packages using Node.js](http://docs.aws.amazon.com/lambda/latest/dg/nodejs-create-deployment-pkg.html) および [Uploading Deployment Packages and Testing](http://docs.aws.amazon.com/lambda/latest/dg/walkthrough-s3-events-adminuser-create-test-function-upload-zip-test.html) を参照してください。

以下のシードプロジェクトには、更新した AWS Lambda 関数に必要なコードが含まれています。

```js wrap lines
<%= include('../../../_includes/_package', { org: 'auth0', repo: 'auth0-aws', path: 'examples/api-gateway/lambda' }) %>
```

シードプロジェクトには、2 つのカスタム JavaScript ファイルがあります。

* `index.js`: メインのコードが含まれています。
* `auth0-variables`: 更新が必要なコードが含まれています。

これらのカスタムファイルに加えて、標準の Node.js `package.json` ファイルもあります。

このコードは、JWT から情報を抽出して検証する機能を追加します。デフォルトでは、Auth0 は JWT の署名に対称キーを使用しますが、必要に応じて非対称キーを使用することもできます (トークンをサードパーティが検証できるようにする必要がある場合は、非対称キーを使用し、公開鍵のみを共有してください) 。

トークンの検証について詳しくは、[Auth0 でサポートされているアイデンティティプロトコル](/docs/ja/authenticate/protocols) を参照してください。

`auth0-variables.js` をシークレットキーで更新します。このキーは、<Tooltip tip="Auth0 のサービスを設定するためのメイン製品である Auth0 Dashboard。" cta="用語集を見る" href="/docs/ja/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip> のアプリケーションの [設定] タブで確認できます。

```javascript lines
var env={};
env.AUTH0_SECRET='{yourAuth0Secret}';
module.exports = env;
```

ファイルがあるディレクトリで **npm install** を実行し、内容を zip 化して (`index.js` は zip のルートに配置する必要があります) 、`PurchasePet` Lambda Function で使用できるようにアップロードします。これをテストすると、JWT がメッセージ本文に含まれていないため、認可エラーが発生するはずです。

`index.js` のロジックを確認してください。60 行目あたりに、トークンを検証し、購入ロジックで使用されるアイデンティティ情報を含むデコード済み情報を抽出する処理があることがわかります:

```js lines
if(event.authToken) {
     jwt.verify(event.authToken, secret, function(err, decoded) {
         if(err) {
           console.log('failed jwt verify: ', err, 'auth: ', event.authToken);
           context.done('authorization failure', null);
         } else if(!decoded.email)
         {
           console.log('err, email missing in jwt', 'jwt: ', decoded);
           context.done('authorization failure', null);
         } else {
           userEmail = decoded.email;
           console.log('authorized, petId', petId, 'userEmail:', userEmail);
           dynamo.getItem({TableName:"Pets", Key:{username:"default"}}, readcb);
         }
     });
  } else {
     console.log('invalid authorization token', event.authToken);
     context.done('authorization failure', null);
  }
    ...
```

<div id="extract-profile-information-to-assign-a-buyer">
  ### 購入者を割り当てるためのプロファイル情報の抽出
</div>

最後のステップでは、JWT をブラウザのクライアントが使用するメソッドに渡します。

標準の方法では、ベアラートークンとして `Authorization` header を使用します。IAM の認可を無効にし、認可を OpenID トークンのみに依存させれば、この方法を利用できます (あわせて、`Authorization` header を AWS Lambda 関数に渡されるイベントデータにマッピングする必要があります) 。

ただし、IAM を使用している場合、AWS API Gateway は `Authorization` header にメッセージの署名を格納するため、この header に JWT を入れると認証が機能しなくなります。その場合は、次のいずれかの方法を使用できます。

* JWT 用のカスタム header を追加する。
* カスタム header をメッセージ本文に含める。

カスタム header を使用する場合は、`pets/purchase` の POST メソッドの Integration Request に対して、追加のマッピングも必要になります。

バリデーションを簡単にするため、JWT は POST の本文で AWS Lambda 関数に渡します。これを行うには、`home.js` の `buyPet` メソッドを更新し、本文から `userName` を削除して、次のように `authToken` を追加します。

```javascript lines
function buyPet(user, id) {
    var apigClient = getSecureApiClient();
    var body = {
      petId:id,
      authToken: store.get('token')
    };

    apigClient.petsPurchasePost({}, body)
      .then(function(response) {
        console.log(response);
        $scope.pets = response.data;
        $scope.$apply();
      }).catch(function (response) {
        alert('buy pets failed');
        showError(response);
    });
}
```

コードをS3 bucketにアップロードし、ペットを購入してみてください。表示されるメッセージに購入者のメールアドレスが表示されます。

エラーが発生した場合は、シークレットキーが正しく設定されているか再確認してください。トークンのデコードに関する問題を確認するのに便利なツールとして、[jwt.io](http://jwt.io/) があります。

<div id="summary">
  ## 概要
</div>

このチュートリアルでは、次のことを行いました。

* AWS Lambda 関数を使用するメソッドを含む API を、AWS API Gateway を使って作成したこと;
* IAM ロールを使用して API へのアクセスを保護したこと;
* <Tooltip tip="IDプロバイダー (IdP): デジタルアイデンティティを保存および管理するサービス。" cta="用語集を見る" href="/docs/ja/glossary?term=SAML">SAML</Tooltip> <Tooltip tip="Security Assertion Markup Language (SAML): パスワードなしで 2 者間で認証情報をやり取りできる標準化されたプロトコル。" cta="用語集を見る" href="/docs/ja/glossary?term=identity+provider">アイデンティティプロバイダー</Tooltip> を IAM と統合し、API へのアクセスをユーザーベースに関連付けたこと;
* ユーザーが Database と Social Connection のどちらで認証されたかに応じて、異なるレベルのアクセスを提供したこと;
* ロールの割り当てを強制するために Auth0 ルールを使用したこと;
* 追加の認可コンテキストを提供し、適切な Lambda 関数にアイデンティティ情報を渡すために JWT を使用したこと。
