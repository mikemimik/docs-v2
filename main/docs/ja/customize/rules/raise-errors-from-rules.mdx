Auth0ログインページでは、Ruleからのエラーメッセージを表示できません。Ruleで発生させたカスタムメッセージをユーザーに表示したい場合は、アプリケーション側でそのメッセージを表示する必要があります。

Ruleによって発生したエラーは、プロトコルの標準レスポンスモードでコールバックURLに送られます。たとえば、アプリケーションが OIDC/OAuth2 を使って認証をリクエストした場合、`error` と `error_description` という結果値を含む標準的な OAuth2 エラーを受け取ります。

コールバックURLが https://travel0.com/callback で、標準の query レスポンスモードを使用しているとします。この場合、次のレスポンスを受け取ります。

```http wrap lines
https://travel0.com/callback?error=access_denied&error_description=[The message you wrote in the rule]
```

このメッセージをユーザーに表示するのは、たとえば次のようにアプリ側の役割です。

<div id="retry-login-after-a-rule-error">
  ## Rule エラー後にログインを再試行する
</div>

次に考えられる対応としては、新たに認可をリクエストすることです。しかし、多くの場合、Auth0 はすでにユーザーを特定しているため、ログイン情報の入力を求めません。代わりに Rule がすぐに実行され、再び認可が拒否されます。すると、これが無限ループになることがあります。

ユーザーには、新しい認証情報を入力する機会が必要です。アプリからログインページを再表示する方法は 2 つあります。

* リクエストに指定を追加して、ログインページを強制的に表示します。

  * OIDC/OAuth2 リクエストで `prompt=login` パラメーターを使用します。
  * SAML リクエストで `ForceAuthn="true"` 属性を使用します。

—or—

* 新しいリクエストを送信する前に、ユーザーを Auth0 からログアウトさせます。
