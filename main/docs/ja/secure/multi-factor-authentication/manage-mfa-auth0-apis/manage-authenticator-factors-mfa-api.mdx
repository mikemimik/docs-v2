<Card title="始める前に">
  * アプリケーションで **MFA** グラントタイプを有効にします。詳しくは、[グラントタイプを更新する](/docs/ja/get-started/applications/update-grant-types)をご覧ください。

Auth0 には、アプリケーションで使用している <Tooltip tip="多要素認証（MFA）: SMS で送信されるコードのように、ユーザー名とパスワードに加えて認証要素を使用するユーザー認証プロセス。" cta="用語集を見る" href="/docs/ja/glossary?term=multi-factor+authentication">多要素認証</Tooltip> (MFA) 用の認証器を管理するための [API エンドポイント](https://auth0.com/docs/api/authentication#multi-factor-authentication) がいくつか用意されています。これらのエンドポイントを使用すると、ユーザーが自分の認証器を管理できる完全なユーザーインターフェースを構築できます。

<div id="get-mfa-api-access-tokens">
  ## MFA API のアクセストークンを取得する
</div>

登録を管理するために MFA API を呼び出すには、まず MFA API 用の <Tooltip tip="アクセストークン: API へのアクセスに使用される認可資格情報で、不透明な文字列または JWT の形式を取ります。" cta="用語集を見る" href="/docs/ja/glossary?term=access+token">アクセストークン</Tooltip> を取得する必要があります。

認証フローの一部として MFA API を使用するには、[Resource Owner Password Grant と MFA を使用して認証する](/docs/ja/secure/multi-factor-authentication/authenticate-using-ropg-flow-with-mfa) に記載された手順に従ってください。認証要素を管理するユーザーインターフェースを構築する場合は、認証時だけでなく、いつでも MFA API で使用できるトークンを取得する必要があります。

MFA アクセストークン、または `https://{yourDomain}/mfa/` の <Tooltip tip="Audience: 発行された token の audience を一意に識別する値です。token では aud という名前で、その値には ID トークンの場合はアプリケーション (Client ID)、アクセストークンの場合は API (API Identifier) の ID が含まれます。" cta="用語集を見る" href="/docs/ja/glossary?term=audience">audience</Tooltip> を持つアクセストークンは、認証を行うと同時に MFA の [チャレンジリクエスト](https://auth0.com/docs/api/authentication/muti-factor-authentication/request-mfa-challenge) を開始します。MFA チャレンジが開始されるかどうかは、テナント での認証要素の有効化状況と、ユーザーの登録状況によって決まります。

* テナント で複数の認証要素の登録が許可されており、ユーザーがワンタイムパスワード (OTP) などの有効な認証要素 (`email` を除く) を登録している場合、ログイン時に MFA チャレンジがユーザーに送信されます。
* ユーザーがどの認証要素にも登録していない場合、または `email` 認証要素のみに登録している場合は、MFA トークンが発行され、ユーザーにチャレンジは送信されません。

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  `https://{yourDomain}/mfa/*` audience を持つアクセストークンのデフォルトの有効期限は 10 分です。この値は設定できません。
</Callout>

<div id="universal-login">
  ### Universal Login
</div>

[Universal Login](/docs/ja/authenticate/login/auth0-universal-login) を使用している場合は、`https://{yourDomain}/mfa/` を audience に指定して、Authorize エンドポイントにリダイレクトします。

<Warning>
  audience に `https://{yourDomain}/mfa/` を指定すると、MFA が強制されます。`.../mfa/` を audience に指定している場合、エンドユーザーが **Remember this browser** を有効にしても、その設定は反映されません。

  Auth0 では、テナント管理者が `allowRememberBrowser` を false に設定する [Action を作成する](/docs/ja/customize/actions/write-your-first-action) ことを推奨しています。これにより、エンドユーザーの画面には **Remember this browser** が表示されなくなります。
</Warning>

<div id="resource-owner-password-grant">
  ### リソース所有者パスワードグラント
</div>

<Tooltip tip="リソース所有者: 保護されたリソースへのアクセスを許可できる主体（ユーザーやアプリケーションなど）。" cta="用語集を見る" href="/docs/ja/glossary?term=Resource+Owner">リソース所有者</Tooltip>パスワードグラント (ROPG) を使用している場合、次の 3 つの方法があります。

* ログイン時に `https://{yourDomain}/mfa/` の audience を要求し、後で [リフレッシュトークン](/docs/ja/secure/tokens/refresh-tokens) を使って更新します。
* 認証器の一覧表示と削除が必要な場合は、`https://{yourDomain}/mfa/` の audience を指定して、`/oauth/token` でユーザーに [再度認証](/docs/ja/secure/multi-factor-authentication/authenticate-using-ropg-flow-with-mfa) してもらいます。認証要素を一覧表示または削除するには、ユーザーは先に MFA を完了する必要があります。
* 認証器の一覧表示だけが必要な場合は、ユーザー名/パスワードを使って `/oauth/token` でユーザーに [再度認証](/docs/ja/secure/multi-factor-authentication/authenticate-using-ropg-flow-with-mfa) してもらいます。エンドポイントからは `mfa_required` エラーと、認証器の一覧表示に使える `mfa_token` が返されます。ユーザーが自分の認証器を表示するには、パスワードを入力する必要があります。

<div id="scopes">
  ### スコープ
</div>

MFA audience 向けのトークンをリクエストする際は、次のスコープを指定できます。

| スコープ                    | 説明               |
| ----------------------- | ---------------- |
| `enroll`                | 新しい認証器を登録するため。   |
| `read:authenticators`   | 既存の認証器を一覧表示するため。 |
| `remove:authenticators` | 認証器を削除するため。      |

<div id="list-authenticators">
  ## 認証器を一覧表示する
</div>

ユーザーの認証器の一覧を取得するには、MFA Authenticators エンドポイントを呼び出します。

  ```bash cURL
  curl --request GET \
    --url 'https://{yourDomain}/mfa/authenticators' \
    --header 'authorization: Bearer MFA_TOKEN'
  ```

  ```csharp C#
  var client = new RestClient("https://{yourDomain}/mfa/authenticators");
  var request = new RestRequest(Method.GET);
  request.AddHeader("authorization", "Bearer MFA_TOKEN");
  IRestResponse response = client.Execute(request);
  ```

  ```go Go
  package main

  import (
  	"fmt"
  	"net/http"
  	"io/ioutil"
  )

  func main() {

  	url := "https://{yourDomain}/mfa/authenticators"

  	req, _ := http.NewRequest("GET", url, nil)

  	req.Header.Add("authorization", "Bearer MFA_TOKEN")

  	res, _ := http.DefaultClient.Do(req)

  	defer res.Body.Close()
  	body, _ := ioutil.ReadAll(res.Body)

  	fmt.Println(res)
  	fmt.Println(string(body))

  }
  ```


  ```javascript Node.JS
  var axios = require("axios").default;

    method: 'GET',

    console.log(response.data);
  }).catch(function (error) {
  ```

  ```php PHP

    CURLOPT_HTTP_VERSION => CURL_HTTP_VERSION_1_1,
    CURLOPT_CUSTOMREQUEST => "GET",
    CURLOPT_HTTPHEADER => [
      "authorization: Bearer MFA_TOKEN"
    ],
  ]);

  $response = curl_exec($curl);
  $err = curl_error($curl);
  ```

  ```python Python
  data = res.read()

  print(data.decode("utf-8"))
  ```


レスポンスでは、認証器の種類に関する情報を受け取れます。

```json lines
[
  {
    "authenticator_type": "recovery-code",
    "id": "recovery-code|dev_IsBj5j3H12VAdOIj",
    "active": true
  },
  {
    "authenticator_type": "otp",
    "id": "totp|dev_nELLU4PFUiTW6iWs",
    "active": true,
  },
  {
    "authenticator_type": "oob",
    "oob_channel": "sms",
    "id": "sms|dev_sEe99pcpN0xp0yOO",
    "name": "+1123XXXXX",
    "active": true
  }
]
```

エンドユーザーが自分の認証要素を管理するためのユーザーインターフェースを構築する場合は、`active` が `false` の認証器を無視してください。これらの認証器はユーザーによる確認が完了していないため、MFA のチャレンジには使用できません。

MFA API では、認証器の種類に応じて次の登録が一覧表示されます。

| 認証器               | 内容                                                                                                                  |
| ----------------- | ------------------------------------------------------------------------------------------------------------------- |
| **Push and OTP**  | Push が有効になっている場合、Auth0 は OTP の登録も作成します。登録を一覧表示すると、両方が表示されます。                                                        |
| **SMS and Voice** | SMS と音声通話の両方が有効になっている場合、ユーザーが SMS または音声通話のいずれかで登録すると、Auth0 はその電話番号に対して 2 つの認証器を自動的に作成します。1 つは SMS 用、もう 1 つは音声通話用です。 |
| **Email**         | 確認済みのすべてのメールアドレスが認証器として一覧表示されます。                                                                                    |

<div id="delete-authenticators">
  ## 認証器を削除する
</div>

関連付けられている認証器を削除するには、MFA Authenticators エンドポイントに `DELETE` リクエストを送信し、`AUTHENTICATOR_ID` を対象の認証器 ID に置き換えます。この ID は、認証器を一覧表示した際に取得できます。

認証器の一覧表示に `mfa_token` を使用した場合、認証器を削除するには、audience が `https://{yourDomain}/mfa/` の[アクセストークンを取得するために MFA を完了する](https://auth0.com/docs/api/authentication/muti-factor-authentication/verify-mfa-with-otp)必要があります。

  ```bash cURL
    --header 'authorization: Bearer ACCESS_TOKEN'
  ```


認証器が削除されると、204 レスポンスが返されます。

認証器を削除すると、認証器の種類に応じて次の動作が発生します。

| 認証器               | 動作                                                                                            |
| ----------------- | --------------------------------------------------------------------------------------------- |
| **Push and OTP**  | ユーザーが push 認証器を登録すると、Auth0 は OTP も登録します。どちらか一方を削除すると、もう一方も削除されます。                             |
| **SMS and Voice** | ユーザーが SMS または Voice のいずれかを登録すると、Auth0 は SMS と voice の 2 つの認証器を作成します。どちらか一方を削除すると、もう一方も削除されます。 |
| **Email**         | 検証済みのメールアドレスはすべて認証器として一覧表示されますが、削除することはできません。削除できるのは、明示的に登録された email 認証器のみです。                 |

<div id="regenerate-recovery-codes">
  ## リカバリーコードを再生成する
</div>
