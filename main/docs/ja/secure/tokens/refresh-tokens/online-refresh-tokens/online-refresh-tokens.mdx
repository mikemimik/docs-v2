Online Refresh Tokens (ORT) は、[リフレッシュトークン](/docs/ja/secure/tokens/refresh-tokens)の一種で、[シングルページアプリケーション](/docs/ja/get-started/auth0-overview/create-applications/single-page-web-apps) (SPA) などのブラウザベースのアプリケーション向けに特化して設計されています。

ORTは、Intelligent Tracking Prevention (ITP) などのブラウザのプライバシー機能によってサードパーティ Cookie へのアクセスがブロックされ、従来の[サイレント認証](/docs/ja/authenticate/login/configure-silent-authentication#configure-silent-authentication)が信頼できなくなる場合でも、SPAでのユーザーセッションを維持できます。

ORTはユーザーのAuth0 [セッション](/docs/ja/manage-users/sessions)に紐付けられており、そのセッションが有効な間だけ使用できます。これにより、ユーザーがログアウトすると、そのセッションを使用しているすべてのアプリケーションもまとめてログアウトされます。

<div id="how-it-works">
  ## 仕組み
</div>

ORT をリクエストすると、次のように動作します。

1. アプリケーションは、認可リクエストに `online_access` スコープを含めます。
2. Auth0 は、ユーザーの現在のセッションに関連付けられた ORT を返します。
3. ORT を新しい access token と交換するたびに、Auth0 はセッションのアイドルタイムアウトを延長します。
4. 基盤となる Auth0 セッションがアクティブである限り、そのトークンは有効なままです。

<div id="comparison-of-orts-versus-refresh-tokens">
  ## ORT とリフレッシュトークンの比較
</div>

| **機能**                                                                                                                          | **ORTs**                                                                                          | **リフレッシュトークン**                                                                          |
| ------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------- |
| **要求される スコープ**                                                                                                                 | `online_access`                                                                                   | `offline_access`                                                                        |
| **セッションへの紐付け**                                                                                                                  | Auth0 セッションに紐付く                                                                                   | セッションに依存しない                                                                             |
| **有効期間**                                                                                                                        | [セッションの有効期間](/docs/ja/manage-users/sessions/configure-session-lifetime)によって決まる                       | [トークンの有効期間](/docs/ja/secure/tokens/refresh-tokens/configure-refresh-token-expiration)で設定する |
| **トークンのローテーション**                                                                                                                | ローテーションされない                                                                                       | [ローテーション](/docs/ja/secure/tokens/refresh-tokens/configure-refresh-token-rotation)に対応       |
| **[Demonstration of Proof of Possession](/docs/ja/secure/sender-constraining/demonstrating-proof-of-possession-dpop)  (DPoP) の要件** | パブリッククライアント (SPA、ネイティブ) の場合は必須                                                                    | 任意 (Resource Server の設定による)                                                             |
| **失効**                                                                                                                          | [セッション全体を失効させる](/docs/ja/manage-users/sessions/manage-sessions-actions#revoke-sessions-with-actions) | [トークンのみを失効させる](/docs/ja/secure/tokens/refresh-tokens/revoke-refresh-tokens)                |
| **Management API**                                                                                                              | 表示されない                                                                                            | Management API で表示可能                                                                    |
| **セッションのアイドルタイムアウト**                                                                                                            | 交換のたびに延長される                                                                                       | セッションに影響しない                                                                             |

<div id="key-behavioral-differences">
  ### 主な挙動の違い
</div>

**セッションの延長:** ORT を交換すると、Auth0 はセッションのアイドルタイムアウトをリセットし、ユーザーセッションを維持します。一方で、セッションの[絶対有効期間](/docs/ja/manage-users/sessions/session-lifecycle#absolute-timeout-maximum)は延長されません。

**DPoP の適用:** パブリッククライアント (SPA など) の場合、Auth0 では ORT の使用時に DPoP が必須です。ORT は、ユーザーの認証済みセッションへのアクセスを許可するセッション Cookie と同等のセキュリティレベルで扱われるため、同じ厳密さで保護する必要があります。DPoP はトークンの窃取やリプレイ攻撃を防ぎます。

**ステートレス:** ORT はセッションに紐づいているため、個別に表示したり管理したりすることはできません。Management API で確認することも、個別に失効させることもできません。

<div id="best-practices">
  ## ベストプラクティス
</div>

1. **トークンはメモリに保存する:** ORT は `localStorage` や `sessionStorage` に保存しないでください。セキュリティを高めるため、JavaScript のメモリ内に保持してください。ORT は Auth0 セッションに紐づいているため、ユーザーが次回アクセスした際にはセッションクッキーによってコンテキストが再確立され、アプリで新しい ORT をリクエストできます。
2. **セッションの有効期限切れを適切に処理する:** セッションの有効期限が切れた場合に備えて、適切なエラーハンドリングを実装してください。ユーザーの作業内容を失わないように、再認証へリダイレクトしてください。
3. **パブリッククライアントでは DPoP を使用する:** ORT ではパブリッククライアントに DPoP が必要です。実装で DPoP 証明を正しく生成し、使用していることを確認してください。
4. **トークンの構造に依存しない:** ORT は不透明な文字列として扱ってください。解析したり、内部構造に依存したりしないでください。内部構造は変更される可能性があります。
5. **セッションの有効期間を監視する:** アイドルタイムアウトと絶対セッションタイムアウトの両方を把握しておいてください。絶対タイムアウトは、リフレッシュトークン交換で延長することはできません。

<div id="limitations">
  ## 制限事項
</div>

<div id="unsupported-flows">
  ### サポートされていないフロー
</div>

ORT は、対話型のブラウザーセッションを作成するフローでのみ利用できます。次のフローでは ORT を**発行できません**。

| **フロー**                                                                                                  | **理由**                                            |
| -------------------------------------------------------------------------------------------------------- | ------------------------------------------------- |
| [デバイス認可フロー](/docs/ja/get-started/authentication-and-authorization-flow/device-authorization-flow)           | セッションはブラウザー内で作成されますが、トークンは別のデバイスで使用されます           |
| [インプリシットフロー](/docs/ja/get-started/authentication-and-authorization-flow/implicit-flow-with-form-post)       | OAuth の仕様では、インプリシットフローでリフレッシュトークンを発行することは禁止されています |
| [リソース所有者パスワードグラント](/docs/ja/get-started/authentication-and-authorization-flow/resource-owner-password-flow) | ブラウザーセッションが作成されません                                |
| [クライアント認証情報フロー](/docs/ja/get-started/authentication-and-authorization-flow/client-credentials-flow)         | ユーザーもセッションも関与しません                                 |
| [カスタムトークン交換](/docs/ja/get-started/authentication-and-authorization-flow/token-exchange-flow)                | トークンを関連付けるセッションがありません                             |

これらのフローのいずれかで `online_access` スコープをリクエストした場合、そのスコープは暗黙的に無視され、リフレッシュトークンは発行されません。

<div id="scope-conflicts">
  ### スコープの競合
</div>

同じ認可リクエストで `online_access` と `offline_access` の両方を同時に要求することはできません。両方のスコープが含まれている場合、Auth0 はエラーを返します。

```json
{
  "error": "invalid_request",
  "error_description": "Cannot request both online_access and offline_access scopes"
}
```

ユースケースに応じて適切な スコープ を選択してください。

* SSO を継続して利用できるブラウザベースのアプリケーションには、`online_access` を使用します。
* ブラウザのセッションとは独立して token を利用する必要があるアプリケーションには、`offline_access` を使用します。
