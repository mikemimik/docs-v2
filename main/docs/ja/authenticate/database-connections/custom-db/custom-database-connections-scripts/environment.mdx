アクションスクリプトは、serverless Webtask コンテナーのインスタンス内で、呼び出された一連の JavaScript 関数として実行されます。その際には、特定の実行環境に加え、Webtask コンテナーと Auth0 の <Tooltip tip="認証サーバー: ユーザーのアイデンティティを確認または拒否するサーバー。" cta="用語集を見る" href="/docs/ja/glossary?term=authentication+server">認証サーバー</Tooltip> (Auth0 テナント自体) から提供されるさまざまなアーティファクトも利用できます。

<div id="npm-modules">
  ## npm モジュール
</div>

Auth0 の serverless Webtask コンテナーでは、幅広い [`npm`](https://www.npmjs.com/) モジュールを利用できます。`npm` モジュールは、アクションスクリプトの実装コード全体のサイズを抑えられるだけでなく、あらかじめ用意されたさまざまな機能も利用できます。

デフォルトでは、一般公開されている多数の `npm` モジュールが標準でサポートされています。この一覧は、潜在的なセキュリティ上の懸念がないか確認したうえで作成されています。どの `npm` モジュールがサポートされているかを確認するには、[Can I require: Auth0 Extensibility](https://auth0-extensions.github.io/canirequire/) を参照してください。

標準ではサポートされていない `npm` モジュールが必要な場合は、[Auth0 Extensibility portal](https://auth0-extensions.github.io/canirequire/) または Auth0 の担当者を通じてリクエストできます。Auth0 はそのリクエストを評価し、適切かどうかを判断します。未解決のリクエストをエスカレーションする必要がある場合は、[Auth0 support portal](https://support.auth0.com/) からサポートチケットを作成してください。現在、Auth0 ではプライベートリポジトリーの `npm` モジュールの利用はサポートされていません。

<div id="variables">
  ## 変数
</div>

Auth0 のアクションスクリプトは環境変数をサポートしており、グローバルに利用可能な `configuration` オブジェクトを介してアクセスできます。`configuration` オブジェクトは読み取り専用として扱う必要があり、外部アイデンティティストアへのアクセスに使用する資格情報や API キーなどの機密情報を保存するために使用します。これにより、セキュリティ上重要な値をアクションスクリプトにハードコードせずに済みます。

また、`configuration` オブジェクトを使うと、変数にテナント固有の値を設定できるため、[複数の環境の設定](/docs/ja/get-started/auth0-overview/create-tenants/set-up-multiple-environments) など、採用しているソフトウェア開発ライフサイクル (SDLC) のベストプラクティス戦略にも対応できます。これにより、実行するテナントに応じて変わる可能性のある値をアクションスクリプトにハードコードせずに済みます。

<div id="global-object">
  ## グローバルオブジェクト
</div>

Auth0 の serverless コンテナーは、各 Auth0 テナントに関連付けられたプールからプロビジョニングされます。各コンテナインスタンスではグローバルオブジェクトが利用でき、そのコンテナインスタンス内で実行されるすべてのアクションスクリプトからアクセスできます。グローバルオブジェクトは、そのコンテナに固有のグローバル変数として機能し、そのコンテナインスタンス内で実行されるすべてのアクションスクリプトで利用できる情報や、関数さえも定義できます。

つまり、ユーザー固有でないリソースであれば、グローバルオブジェクトを使ってコストの高いリソースをキャッシュできます。たとえば、ユーザー固有ではない機能を提供するサードパーティ (例: ログ) API 用の <Tooltip tip="アクセストークン: API にアクセスするために使用される認可資格情報で、不透明な文字列または JWT の形式を取ります。" cta="用語集を見る" href="/docs/ja/glossary?term=Access+Token">アクセストークン</Tooltip> を保存できます。また、Auth0 で定義され、クライアント認証情報フローを使って取得した、ユーザー固有ではない独自の API 用アクセストークンを保存することもできます。

<Warning>
  アクションスクリプトは、すでに実行中のいずれかのコンテナインスタンスで実行されることもあれば、新しく作成されたコンテナインスタンスで実行されることもあります (その後、そのインスタンスがプールに追加される場合があります) 。Auth0 では、アクションスクリプトの実行先コンテナは固定されません。したがって、グローバルオブジェクトにはユーザー固有の情報を保存しないようにし、グローバルオブジェクト内で行う宣言には必ず初期化処理も含めるようにしてください。
</Warning>

コンテナーが再利用されるたび、または新しい コンテナーがインスタンス化されるたびに、そのコンテナで定義されるグローバルオブジェクトはリセットされます。そのため、コンテナに関連付けられたグローバルオブジェクト内で代入を宣言する場合は、初期化の考慮も含める必要があります。パフォーマンスの柔軟性を確保するため、serverless コンテナーは Auth0 でアドホックにプロビジョニングされ、さまざまな再利用ポリシーの対象にもなります。一般に、グローバルオブジェクトの寿命は 20 分を超えるものと考えないことをお勧めします。

<div id="custom-database-connection-environment-checklist">
  ## カスタムデータベース接続環境のチェックリスト
</div>

* データベースに、**id**、**nickname**、**email**、**password** などのユーザープロファイル属性を保存するための適切なフィールドがあることを確認してください。Auth0 のユーザープロファイルのスキーマと想定されるフィールドについて詳しくは、[Normalized User Profiles](/docs/ja/manage-users/user-accounts/user-profiles/normalized-user-profiles) を参照してください。ユーザープロファイルの更新方法については、[Update User Profiles Using Your Database](/docs/ja/manage-users/user-accounts/user-profiles/update-user-profiles-using-your-database) を参照してください。
* トラブルシューティングのために、カスタムデータベース接続で発生したエラーを返すことができます。
* 返されるユーザープロファイル内の `id` (または `user_id`) プロパティは、Auth0 がユーザーを識別するために使用します。複数のカスタムデータベース接続を使用している場合は、**ユーザーID** の衝突を避けるため、**id** の値が **すべてのカスタムデータベース接続で一意である必要があります**。推奨される方法は、**id** の値の先頭に接続名 (空白を除く) を付けることです。ユーザーID の詳細については、[Identify Users](/docs/ja/manage-users/user-accounts/identify-users) を参照してください。
* レイテンシーは、Auth0 がホストするユーザーストアと比べて大きくなります。
* データベースまたはサービスには Auth0 サーバーからアクセスできる必要があります。ユーザーストアがファイアウォールの背後にある場合は、受信接続を設定する必要があります。

<div id="testing-a-specific-runtime-version">
  ## 特定のランタイムバージョンをテストする
</div>

カスタムデータベーススクリプトのランタイムバージョンは、**Auth0 Dashboard &gt; 設定 &gt; Advanced** の **Extensibility** セクションで定義されます。

特定のランタイムバージョンで単一のカスタムデータベーススクリプトをテストするには、次の手順に従ってください。

1. **Authentication &gt; Database** に移動します。
2. カスタムスクリプトを定義しているデータベース接続を選択します。
3. 選択したデータベース接続のページで、**Custom Database** タブを選択します。
4. ページ内の **Database Action Scripts** セクションまでスクロールします。
5. 該当するタブで、確認したいスクリプト (例: Login、Create など) を選択します。
6. **Save and Try** を選択します。すると、テスト用のパラメータとサンプルのコンテキスト詳細が表示されたモーダルが開きます。必要に応じて更新します。
7. モーダルで **試す** を選択し、特定の Node バージョンを選ぶためのドロップダウンセレクタを開きます。
8. 目的の Node バージョンを選択するとテストが実行され、結果は同じ画面上のメッセージに表示されます。
