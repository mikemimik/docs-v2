description: OIDC 準拠パイプラインが委譲の利用にどのような影響を与えるかを説明します。
title: OIDC での委譲
---

<Warning>
  デフォルトでは、2017 年 6 月 8 日時点で使用中のアドオンがないテナントでは、委譲は無効になっています。現在、委譲が必要なアドオンを使用しているレガシーテナントは、この機能を引き続き利用できます。今後、委譲機能が変更されたりサービスから削除されたりする場合は、現在利用しているお客様に事前に通知し、移行のための十分な時間が提供されます。
</Warning>

従来、委譲は次の目的で使用されてきました。

* あるアプリケーションに発行された <Tooltip tip="ID トークン: リソースへのアクセスではなく、クライアント自体を対象とした資格情報です。" cta="用語集を見る" href="/docs/ja/glossary?term=ID+token">ID トークン</Tooltip> を、別のアプリケーション向けに発行された新しいものと交換する。
* [リフレッシュトークン](/docs/ja/secure/tokens/refresh-tokens) を使用して新しい ID トークンを取得する。
* ID トークンをサードパーティ (例: Firebase、AWS) の API トークンと交換する。

OIDC 準拠パイプラインでは、ID トークンを API の保護に使用できず、<Tooltip tip="リフレッシュトークン: ユーザーに再度ログインを求めることなく、新しいアクセストークンを取得するために使用されるトークンです。" cta="用語集を見る" href="/docs/ja/glossary?term=refresh+tokens">リフレッシュトークン</Tooltip> は [`/oauth/token`](https://auth0.com/docs/api/authentication#refresh-token) エンドポイントでのみ使用する必要があるため、[`/delegation`](https://auth0.com/docs/api/authentication#delegation) エンドポイントは非推奨です。

OIDC 準拠アプリケーションは、委譲リクエストの送信元にも送信先にもできません。

<div id="third-party-apis">
  ## サードパーティAPI
</div>

サードパーティ (例: Firebase、AWS) のAPIトークンを取得するためのOIDC準拠の仕組みは存在しないため、サードパーティAPIトークンの取得には引き続き委譲を利用できます。
