<div id="auth0lock">
  ## Auth0Lock
</div>

`new Auth0Lock(clientID, domain, options)`

Inicializa una nueva instancia de `Auth0Lock` configurada con el `clientID` de tu aplicación y el `domain` de tu cuenta desde el [Dashboard](https://manage.auth0.com/#/) de administración de Auth0. El tercer parámetro, opcional, es un objeto `options` que se usa para configurar Lock según las necesidades de tu aplicación. Puedes encontrar esta información en la [configuración de la aplicación](https://manage.auth0.com/#/applications).

* **clientId `{String}`**: Parámetro obligatorio. El clientId de tu aplicación en Auth0.
* **domain `{String}`**: Parámetro obligatorio. Tu dominio de Auth0. Normalmente, your-account.auth0.com.
* **options `{Object}`**: Parámetro opcional. Permite configurar la apariencia y el comportamiento de Lock. Consulta [la página de opciones de configuración](/es/docs/libraries/lock/lock-configuration) para más detalles.

export const codeExample1 = `var Auth = (function() {

  var privateStore = {};

  function Auth() {
    // Instanciar Lock, sin opciones personalizadas
    this.lock = new Auth0Lock(
      '<{yourClientId}>',
      '<{yourDomain}>'
    );
  }

  Auth.prototype.getProfile = function() {
    return privateStore.profile;
  };

  Auth.prototype.authn = function() {
    // Escuchar el evento authenticated y obtener el perfil
    this.lock.on("authenticated", function(authResult) {
      // Usa el token de authResult para getUserInfo() y guárdalo si es necesario
      this.getUserInfo(authResult.accessToken, function(error, profile) {
        if (error) {
          // Gestionar el error
          return;
        }

        //guardar el token de acceso solo si es necesario
        privateStore.accessToken = accessToken;
        privateStore.profile = profile;

        // Actualizar el DOM
      });
    });
  };
  return Auth;
}());`;

<div id="getuserinfo">
  ## getUserInfo()
</div>

`getUserInfo(accessToken, callback)`

Una vez que el usuario haya iniciado sesión y dispongas de un token, puedes usarlo para obtener el perfil del usuario con `getUserInfo`. Este método reemplaza al obsoleto `getProfile()`.

* **accessToken &#123;String&#125;**: Token de usuario.
* **callback &#123;Function&#125;**: Se invocará después de recuperar el perfil del usuario.

```js lines
lock.getUserInfo(accessToken, function(error, profile) {
  if (!error) {
    alert("hello " + profile.name);
  }
});
```

<div id="show">
  ## show()
</div>

`show(options)`

El método `show` muestra el widget. A partir de la versión 10.2.0 de Lock, el método `show` puede aceptar un objeto `options` como parámetro. Ten en cuenta que este parámetro está pensado para usarse como una forma de sobrescribir las `options` de tu Lock para esta visualización concreta del widget; las opciones deben configurarse al instanciar Lock y sobrescribirse aquí solo si es necesario para tu caso de uso específico.

El siguiente subconjunto de `options` puede sobrescribirse con respecto a los valores que se les asignaron (o a sus valores predeterminados) cuando se instanció Lock:

* allowedConnections
* auth.params
* allowLogin
* allowSignUp
* allowForgotPassword
* initialScreen
* rememberLastLogin

Para obtener más información sobre la lista completa de opciones configurables que pueden elegirse al instanciar Lock, a diferencia del subconjunto limitado anterior que puede sobrescribirse en el método `show`, consulta la [página de opciones configurables por el usuario](/es/docs/libraries/lock/lock-configuration).

Ejemplos de sobrescritura de opciones:

```js lines
// Mostrar el widget de Lock sin sobreescribir ninguna opción
lock.show();

// Mostrar el widget de Lock sobreescribiendo algunas opciones
lock.show({
  allowedConnections: ["twitter", "facebook"],
  allowSignUp: false
});
```

Las opciones deben establecerse al instanciar Lock por primera vez: `var lock = new Auth0Lock(clientId, domain, options);`. Las opciones solo deben pasarse a `show` para sobrescribir las que configuraste previamente al mostrar el widget en ese momento y lugar concretos.

Hay una opción adicional que puede establecerse en el método `show`, llamada `flashMessage`.

<div id="flashmessage">
  ### flashMessage
</div>

Este objeto solo está disponible como opción para el método `show`, no para usarlo en el objeto `options` habitual al instanciar Lock. El objeto `flashMessage` muestra un mensaje flash de error o de éxito cuando se muestra Lock. Tiene los siguientes parámetros:

* **type** &#123;String&#125;: El tipo de mensaje; debe ser `error` o `success`.
* **text** &#123;String&#125;: El texto que se mostrará.

```js lines
lock.show({
  flashMessage:{
    type: 'success',
    text: 'Amazing Success!!'
  }
});
```

Una forma práctica de usar la opción `flashMessage` es gestionar errores de autorización. `flashMessage` se puede rellenar con un texto descriptivo del error.

```js lines
lock.on('authorization_error', function(error) {
  lock.show({
    flashMessage: {
      type: 'error',
      text: error.errorDescription
    }
  });
});
```

Así que, si `tester@example.com` intentara ahora iniciar sesión, como es un usuario bloqueado, se le volverá a mostrar Lock, con una barra superior en la que aparece el mensaje de error, en lugar de que simplemente falle el inicio de sesión y Lock se cierre.

<div id="hide">
  ## hide()
</div>

`hide()`

El método `hide` cierra el widget si está abierto en ese momento. El widget se cierra por sí solo en la mayoría de los casos, por lo que este método normalmente solo se usa en casos de uso específicos. Por ejemplo, puede que quieras escuchar el evento `unrecoverable_error` y luego aplicar `hide` a Lock y redirigir al usuario a una página de error personalizada. Otro ejemplo son los usuarios que están implementando el [modo popup](/es/docs/libraries/lock/lock-authentication-modes) y que podrían necesitar aplicar `hide` manualmente al widget después de que se dispare el evento `authenticated`.

Ejemplo de uso de `hide` para ocultar (cerrar) el widget de Lock en modo popup:

```js lines
// Escuchar el evento authenticated y ocultar Lock
lock.on("authenticated", function() {
  lock.hide();

  // Cualquier otra acción que desees realizar en el evento authenticated

});
```

<div id="on">
  ## on()
</div>

Lock emite eventos durante su ciclo de vida. El método `on` puede usarse para escuchar eventos concretos y reaccionar a ellos.

* `show`: se emite cuando se muestra Lock. No tiene argumentos.
* `hide`: se emite cuando se oculta Lock. No tiene argumentos.
* `unrecoverable_error`: se emite cuando se produce un error irrecuperable, por ejemplo, cuando no hay ninguna conexión disponible. Tiene el error como único argumento.
* `authenticated`: se emite después de una autenticación correcta. Tiene el resultado de la autenticación como único argumento. El resultado de la autenticación contiene el token, que puede usarse para obtener el perfil del usuario o almacenarse para iniciar sesión al usuario en verificaciones posteriores.
* `authorization_error`: se emite cuando falla la autorización. Tiene el error como único argumento.
* `hash_parsed`: cada vez que se inicializa un nuevo objeto Auth0Lock en modo de redirección (el predeterminado), intentará analizar la parte hash de la URL en busca del resultado de un intento de inicio de sesión. Este es un evento de bajo nivel para casos de uso avanzados y, cuando sea posible, se recomienda usar `authenticated` y `authorization_error`. Después, este evento se emitirá con `null` si no pudo encontrar nada en el hash. Se emitirá con el mismo argumento que el evento `authenticated` después de un inicio de sesión correcto o con el mismo argumento que `authorization_error` si algo salió mal. Este evento no se emitirá en [modo popup](/es/docs/libraries/lock/lock-authentication-modes) porque no es necesario analizar la parte hash de la URL.
* `forgot_password ready`: se emite cuando se muestra la pantalla &quot;Olvidé mi contraseña&quot;. (Solo en la versión &gt;`10.18`)
* `forgot_password submit`: se emite cuando el usuario hace clic en el botón de envío de la pantalla &quot;Olvidé mi contraseña&quot;. (Solo en la versión &gt;`10.14`)
* `signin ready`: se emite cuando se muestra la pantalla &quot;Iniciar sesión&quot;.
* `signup ready`: se emite cuando se muestra la pantalla &quot;Registrarse&quot;.
* `signin submit`: se emite cuando el usuario hace clic en el botón de envío de la pantalla &quot;Login&quot;. (Solo en la versión &gt;`10.18`)
* `signup submit`: se emite cuando el usuario hace clic en el botón de envío de la pantalla &quot;Registrarse&quot;. (Solo en la versión &gt;`10.18`)
* `federated login`: se emite cuando el usuario hace clic en un botón de conexión social. Tiene el nombre de la conexión y la estrategia como argumentos. (Solo en la versión &gt;`10.18`)
* `socialOrPhoneNumber ready`: se emite cuando se muestra la pantalla <Tooltip tip="Passwordless: Forma de autenticación que no depende de una contraseña como primer factor." cta="Ver glosario" href="/es/docs/glossary?term=Passwordless">Passwordless</Tooltip> con Social + Número de teléfono
* `socialOrPhoneNumber submit`: se emite cuando se envía la pantalla Passwordless con Social + Número de teléfono
* `socialOrEmail ready`: se emite cuando se muestra la pantalla Passwordless con Social + Email
* `socialOrEmail submit`: se emite cuando se envía la pantalla Passwordless con Social + Email
* `vcode ready`: se emite cuando se muestra la pantalla Passwordless con la contraseña de un solo uso
* `vcode submit`: se emite cuando se envía la pantalla Passwordless con la contraseña de un solo uso

El listener del evento `authenticated` tiene un solo argumento: un objeto `authResult`. Este objeto contiene las siguientes propiedades: `accessToken`, `idToken`, `state`, `refreshToken` e `idTokenPayload`.

Un ejemplo de uso del evento `authenticated`:

<div id="resumeauth">
  ## resumeAuth()
</div>

Este método solo se puede usar cuando configuras la opción [auth.autoParseHash](/es/docs/libraries/lock/lock-configuration) en `false`. Tendrás que llamar a `resumeAuth` para completar el flujo de autenticación. Este método resulta útil cuando usas un router en el cliente que utiliza `#` para gestionar URL (angular2 con `useHash` o react-router con `hashHistory`).

* **hash** &#123;String&#125;: El fragmento hash recibido en la redirección.
* **callback** &#123;Function&#125;: Se invocará una vez completado el análisis. Recibe un error (si lo hay) como primer argumento y el resultado de la autenticación como segundo. Si no hay ningún hash disponible, ambos argumentos serán `null`.

```js lines
lock.resumeAuth(hash, function(error, authResult) {
  if (error) {
    alert("Could not parse hash");
  }
  //Este es solo un ejemplo; no debes registrar Access Tokens en producción.
  console.log(authResult.accessToken);
});
```

<div id="checksession">
  ## checkSession()
</div>

El método `checkSession` te permite obtener un nuevo token de Auth0 para un usuario que ya se ha autenticado con Auth0 en tu dominio. Acepta los siguientes parámetros:

* **options** &#123;Object&#125;: Opcional. Acepta cualquier parámetro válido de OAuth2 que normalmente se enviaría a `/authorize`. Si los omites, usará los que se proporcionaron al inicializar Auth0.
* **callback** &#123;Function&#125;: Se invocará con el resultado de la renovación del token. Recibe un error (si lo hay) como primer argumento y el resultado de la autenticación como segundo.

```js lines
lock.checkSession({}, function(err, authResult) {
  // manejar el error o los nuevos tokens
});
```

<div id="logout">
  ## logout()
</div>

Cierra la sesión del usuario.

* **options** &#123;Object&#125;: Es opcional y sigue las mismas reglas que `auth0.js logout()`.

```js lines
lock.logout({
  returnTo: 'https://myapp.com/bye-bye'
});
```
