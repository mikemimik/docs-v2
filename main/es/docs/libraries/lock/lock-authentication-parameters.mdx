Puedes enviar parámetros al iniciar sesión añadiéndolos al objeto `options`. En el siguiente ejemplo, se añade un parámetro `state` con un valor igual a `'foo'`.

```js lines
var options = {
  auth: {
    params: {state: 'foo'},
  }
};
```

El ejemplo anterior sería equivalente a iniciar sesión con `https://{yourDomain}/authorize?state=foo&...`.

Se admiten los siguientes parámetros: `scope`, `device`, `nonce` y `state`.

<div id="supported-parameters">
  ## Parámetros admitidos
</div>

### scope {string}

```js lines
var options = {
  auth: {
    params: {scope: 'openid email user_metadata app_metadata picture'},
  }
};
```

Hay diferentes valores admitidos para `scope`. Ten en cuenta que los <Tooltip tip="JSON Web Token (JWT): formato estándar del token de ID (y, a menudo, del access token) que se usa para representar claims de forma segura entre dos partes." cta="Ver glosario" href="/docs/es/glossary?term=JWTs">JWTs</Tooltip> se envían en cada solicitud a la API, por lo que conviene mantenerlos lo más pequeños posible.

El valor predeterminado de `scope` en Lock es `openid profile email`. Este valor mínimo de `scope` es necesario para que la funcionalidad **Última vez que iniciaste sesión con** funcione correctamente.

<Card title="Ejecutar Lock localmente">
  Si no especificas manualmente al menos el `scope` predeterminado `openid profile email` al inicializar Lock, y ejecutas tu sitio web desde `http://localhost` o `http://127.0.0.1`, verás el siguiente error en la consola del navegador:

  `Consent required. When using getSSOData, the user has to be authenticated with the following scope: openid profile email`

  Eso **no** ocurrirá cuando ejecutes tu aplicación en producción o si especificas el `scope` `openid profile email`. Puedes leer más sobre esto en el documento [consentimiento del usuario y aplicaciones de terceros](/docs/es/get-started/applications/third-party-applications/user-consent-and-third-party-applications).
</Card>

Para obtener más información sobre los scopes, consulta la [página de documentación sobre scopes](/docs/es/get-started/apis/scopes).

<div id="example-retrieve-a-token">
  #### Ejemplo: recuperar un token
</div>

En Lock, si deseas recibir un token que permita obtener los datos del perfil del usuario, debes agregar el parámetro `scope`.

```js lines
var options = {
  auth: {
    params: {
        scope: 'openid profile'
    }
  }
};
```

También existe una opción de configuración de `connectionScopes` para Lock que te permite especificar scopes para una conexión concreta. Esto resulta útil si quieres empezar con un conjunto de scopes (definidos en el Dashboard) y, más adelante, solicitar permisos o atributos adicionales de una conexión específica. Obtén más información en la página [opciones de configuración de Lock](/docs/es/libraries/lock/lock-configuration).

### state {string}

El parámetro `state` es un valor arbitrario que se mantiene entre redirecciones. Resulta útil para mitigar los [ataques XSRF](https://en.wikipedia.org/wiki/Cross-site_request_forgery) y para conservar información contextual, [como una URL de retorno](/docs/es/authenticate/login/logout/redirect-users-after-logout), que puedas necesitar una vez finalizado el proceso de autenticación. Si no se proporciona un parámetro `state` personalizado, Lock generará uno automáticamente. Para obtener más información, consulta [Parámetro state](/docs/es/secure/attack-protection/state-parameters).

### nonce {string}

El parámetro `nonce` se utiliza para ayudar a prevenir ataques de repetición y Lock lo genera automáticamente si no se proporciona un valor personalizado.

### device {string}

El parámetro `device` indica el nombre del dispositivo o navegador que solicita autenticación.
