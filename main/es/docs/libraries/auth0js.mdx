La biblioteca Auth0.js es una biblioteca JavaScript del lado del cliente para Auth0. Admite casos de uso tanto de [inicio de sesión alojado y inicio de sesión integrado](/es/docs/authenticate/login/universal-vs-embedded-login). Este artículo trata sobre la **v10**, la versión actual.

Consulta la [documentación completa de la API de la biblioteca](https://auth0.github.io/auth0.js/index.html).

<div id="migrate-from-v9-to-v10">
  ## Migrar de v9 a v10
</div>

Auth0.js v10 incluye una corrección de seguridad para CVE-2026-42280 y un cambio incompatible:

| Afectado                    | Detalles                                                                                                                                                                                                                                                                                             |
| --------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Aplicaciones que usan RS256 | No hay cambios incompatibles                                                                                                                                                                                                                                                                         |
| Aplicaciones que usan HS256 | `parseHash()` ahora devuelve un error `invalid_token` en lugar de completarse correctamente sin avisar. Cambia a RS256 en el Auth0 Dashboard: **Applications** &gt; tu aplicación &gt; **Settings** &gt; **Advanced Settings** &gt; **OAuth** &gt; **JsonWebToken Signature Algorithm** &gt; `RS256` |

Si necesitas la referencia de la API de v9, consulta el [paquete auth0-js en npm](https://www.npmjs.com/package/auth0-js?activeTab=versions) y selecciona tu versión de v9, o examina el [código fuente y el registro de cambios de v9 en GitHub](https://github.com/auth0/auth0.js).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  El inicio de sesión integrado para aplicaciones web usa [autenticación de origen cruzado](/es/docs/authenticate/login/cross-origin-authentication), a menos que [configures un dominio personalizado](/es/docs/customize/custom-domains) para tu tenant. La autenticación de origen cruzado usa cookies de terceros para permitir transacciones de autenticación seguras entre distintos orígenes.
</Callout>

<div id="configure-your-auth0-application-for-embedded-login">
  ### Configura tu aplicación de Auth0 para el inicio de sesión integrado
</div>

Al implementar el inicio de sesión integrado, la biblioteca utiliza llamadas de origen cruzado dentro de iframes ocultos para realizar la autenticación. Para garantizar que esto pueda hacerse de forma segura, Auth0 necesita conocer los dominios en los que alojas tus aplicaciones.

Agrega el dominio al campo **Allowed Web Origins**. Puedes encontrar este campo en la sección [Configuración de la aplicación](https://manage.auth0.com/#/applications) de tu Dashboard.

<div id="installation-options">
  ### Opciones de instalación
</div>

Tienes varias opciones para usar Auth0.js en tu proyecto. Elige una de las siguientes según tus necesidades:

Instálalo mediante [npm](https://npmjs.org) o [yarn](https://yarnpkg.com):

```bash lines
npm install auth0-js

yarn add auth0-js
```

Después de instalar el módulo `auth0-js`, empaquétalo junto con todas sus dependencias o impórtalo usando:

```js lines
import auth0 from 'auth0-js';
```

Como alternativa, incluye el script mediante la CDN:

```html wrap lines
<script src="https://cdn.auth0.com/js/auth0/10.0.0/auth0.min.js"></script>
```

<div id="available-parameters">
  #### Parámetros disponibles
</div>

Hay dos parámetros obligatorios que deben pasarse en el objeto `options` al instanciar `webAuth`, y varios más que son opcionales.

| **Parámetro**                 | **Obligatorio** | **Descripción**                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                         |
| ----------------------------- | --------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `domain`                      | obligatorio     | (String) El dominio de tu cuenta de Auth0 (p. ej., myaccount.auth0.com)                                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| `clientID`                    | obligatorio     | (String) Tu ID de cliente de Auth0                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| `redirectUri`                 | opcional*       | (String) El valor predeterminado de `redirectUri`. De forma predeterminada, es una cadena vacía (ninguno). **Si no proporcionas aquí un valor global de `redirectUri`, tendrás que proporcionar un valor de `redirectUri` para *cada* método que uses.**                                                                                                                                                                                                                                                                                |
| `scope`                       | opcional        | (String) El <Tooltip tip="Scope: Mecanismo que determina qué acciones pueden realizar las aplicaciones en nombre de un usuario con información creada previamente en un recurso en línea." cta="Ver glosario" href="/es/docs/glossary?term=scope%28s%29">scope(s)</Tooltip> predeterminado que usa la aplicación. El uso de scopes puede permitirte devolver claims específicos para campos concretos de tu solicitud. Te recomendamos leer nuestra [documentación sobre scopes](/es/docs/get-started/apis/scopes) para obtener más detalles. |
| `audience`                    | opcional        | (String) La audiencia predeterminada que se usará para solicitar acceso a la API.                                                                                                                                                                                                                                                                                                                                                                                                                                                       |
| `responseType`                | opcional*       | (String) El valor predeterminado de `responseType`. Puede ser cualquier lista de valores separados por espacios entre `code`, `token` e `id_token`. Su valor predeterminado es `'token'`, a menos que se proporcione un `redirectUri`; en ese caso, el valor predeterminado es `'code'`. **Si no proporcionas un valor global de `responseType`, tendrás que proporcionar un valor de `responseType` para *cada* método que uses.**                                                                                                     |
| `responseMode`                | opcional        | (String) Esta opción se omite de forma predeterminada. Puede establecerse en `'form_post'` para enviar el token o el código a `'redirectUri'` mediante POST. Los valores admitidos son `query`, `fragment` y `form_post`.                                                                                                                                                                                                                                                                                                               |
| `leeway`                      | opcional        | (Integer) Un valor en segundos; margen para tolerar la desviación del reloj con respecto a los tiempos de vencimiento del token de ID.                                                                                                                                                                                                                                                                                                                                                                                                  |
| `_disableDeprecationWarnings` | opcional        | (Boolean) Desactiva las advertencias de obsolescencia; el valor predeterminado es `false`.                                                                                                                                                                                                                                                                                                                                                                                                                                              |

Debido a problemas de desviación del reloj, es posible que ocasionalmente aparezca el error `The token was issued in the future`. El parámetro `leeway` puede usarse para permitir unos segundos de margen en los tiempos de vencimiento del token de ID y evitar así que eso ocurra.

<div id="webauthauthorize">
  ### webAuth.authorize()
</div>

El método `authorize()` se puede utilizar para iniciar sesión de usuarios mediante <Tooltip tip="Universal Login: Tu aplicación redirige a Universal Login, alojado en el servidor de autorización de Auth0, para verificar la identidad de un usuario." cta="Ver glosario" href="/es/docs/glossary?term=Universal+Login">Universal Login</Tooltip> o conexiones sociales, como se muestra en los ejemplos a continuación. Este método invoca el endpoint `/authorize` de la Authentication API y puede recibir diversos parámetros a través del objeto `options`.

| **Parámetro**  | **Requerido** | **Descripción**                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                               |
| -------------- | ------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `audience`     | opcional      | (String) La audiencia predeterminada que se utilizará para solicitar acceso a la API.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                         |
| `connection`   | opcional      | (String) Especifica la conexión que se debe usar, en lugar de mostrar todas las conexiones disponibles para la aplicación.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                    |
| `scope`        | opcional      | (String) Los scopes para los que deseas solicitar autorización. Deben estar separados por espacios. Puedes solicitar cualquiera de los scopes estándar de OIDC relacionados con usuarios, como `profile` y `email`, custom claims que deben [ajustarse a un formato con espacio de nombres](/es/docs/secure/tokens/json-web-tokens/create-custom-claims), o cualquier scope compatible con la API de destino (por ejemplo, `read:contacts`). Incluye `offline_access` para obtener un <Tooltip tip="Refresh Token: Token utilizado para obtener un token de acceso renovado sin obligar a los usuarios a iniciar sesión de nuevo." cta="Ver glosario" href="/es/docs/glossary?term=Refresh+Token">token de actualización</Tooltip>. |
| `responseType` | opcional      | (String) Puede ser cualquier lista de los valores `code`, `token`, `id_token`, separados por espacios. De forma predeterminada, es `'token'`, a menos que se proporcione un `redirectUri`; en ese caso, el valor predeterminado será `'code'`.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                |
| `clientID`     | opcional      | (String) Tu ID de cliente de Auth0.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                           |
| `redirectUri`  | opcional      | (String) La URL a la que Auth0 redirigirá el navegador después de que se haya concedido la autorización para el usuario.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| `state`        | opcional      | (String) Un valor arbitrario que debe mantenerse entre redirecciones. Resulta útil para mitigar ataques CSRF y para cualquier información contextual (por ejemplo, una URL de retorno) que puedas necesitar una vez finalizado el proceso de autenticación. Para obtener más información, consulta [State Parameter](/es/docs/secure/attack-protection/state-parameters). Auth0.js, cuando se usa en aplicaciones web de página única, gestiona automáticamente la generación y validación del state si no se especifica.                                                                                                                                                                                                        |
| `prompt`       | opcional      | (String) Un valor de `login` forzará que se muestre la página de inicio de sesión independientemente de la sesión actual. Un valor de `none` intentará omitir los prompts de inicio de sesión si ya existe una sesión (consulta la documentación de [autenticación silenciosa](/es/docs/authenticate/single-sign-on#silent-authentication) para obtener más detalles).                                                                                                                                                                                                                                                                                                                                                           |

Para el inicio de sesión alojado, se debe llamar al método `/authorize()`.

`webAuth.authorize({//Any additional options can go here});`

Para los inicios de sesión sociales, será necesario especificar el parámetro `connection`:

`webAuth.authorize({connection: 'twitter'});`

<div id="webauthpopupauthorize">
  ### webAuth.popup.authorize()
</div>

Para la autenticación con `popup`, se puede usar el método `popup.authorize`. La autenticación con `popup` no se puede usar en páginas de inicio de sesión alojadas. Normalmente, la autenticación con `popup` la usan las aplicación web de página única para no perder el estado actual al hacer una redirección completa de página.

Autorización predeterminada con `popup` (Universal Login):

```js lines
webAuth.popup.authorize({
  responseType: 'token'
  redirectUri: 'https://YOUR_APP/popup_response_handler.html'
  //Aquí pueden ir opciones adicionales
}, function(err, authResult) {
  //hacer algo
});
```

Y para el inicio de sesión con redes sociales mediante una ventana emergente con `authorize`:

```js lines
webAuth.popup.authorize({
  responseType: 'token'
  redirectUri: 'https://{yourApp}/popup_response_handler.html',
  connection: 'twitter'
}, function(err, authResult) {
  //hacer algo
});
```

<div id="handle-popup-authentication-results">
  #### Gestionar los resultados de la autenticación con popup
</div>

Al usar la autenticación con popup, tendrás que proporcionar un `redirectUri` donde la página de destino comunique los resultados de la autorización a la devolución de llamada mediante el método `webAuth.popup.callback`. Una implementación sencilla sería algo así:

```html lines
<!-- popup_response_handler.html -->
<html>
  <body>
    <script src="https://cdn.auth0.com/js/auth0/10.0.0/auth0.min.js"></script>
    <script type="text/javascript">
      var webAuth = new auth0.WebAuth({
        domain:       'YOUR_DOMAIN',
        clientID:     'YOUR_CLIENT_ID'
      });
      webAuth.popup.callback();
    </script>
  </body>
</html>
```

Un controlador ideal solo debería contener esta funcionalidad mínima (es decir, evita volver a cargar toda la aplicación solo para procesar la respuesta).

Tendrás que agregar la `redirectUri` a la lista de **Allowed Callback URLs** en la página de configuración de la aplicación en el Dashboard.

<div id="single-sign-on-with-embedded-authentication">
  ### Inicio de sesión único con autenticación integrada
</div>

El inicio de sesión integrado permite <Tooltip tip="Inicio de sesión único (SSO): Servicio que, una vez que un usuario inicia sesión en una aplicación, lo inicia automáticamente en otras aplicaciones." cta="Ver glosario" href="/es/docs/glossary?term=Single+Sign-on">inicio de sesión único</Tooltip> (SSO) cuando tus aplicaciones comparten la siguiente arquitectura:

1. Las aplicaciones que intentan usar SSO son aplicaciones propias. No se admite compartir sesiones integradas con aplicaciones de terceros.
2. Las aplicaciones y tu tenant de Auth0 comparten un dominio de nivel superior mediante un [dominio personalizado](/es/docs/customize/custom-domains). Los dominios tradicionales de Auth0 usan el formato `foo.auth0.com`; los dominios personalizados permiten que tus aplicaciones y tu tenant de Auth0 compartan el mismo dominio de nivel superior, lo que también ayuda a prevenir ataques CSRF.

Cuando tu arquitectura cumple estos criterios, el SSO integrado con Auth0.js encaja bien. Universal Login gestiona automáticamente el SSO en varios dominios o con [aplicaciones de terceros](/es/docs/get-started/applications/third-party-applications/configure-third-party-applications) a través de su capa de sesión, y ambos pueden coexistir en la misma aplicación; por ejemplo, Universal Login para el inicio de sesión principal y Auth0.js para la inscripción integrada de factores o los flujos de step-up. Para comparar las ventajas y desventajas, lee [Inicio de sesión alojado vs. inicio de sesión integrado](/es/docs/authenticate/login/universal-vs-embedded-login).

<div id="passwordless-login">
  ## Inicio de sesión passwordless
</div>

La autenticación <Tooltip tip="Passwordless: Forma de autenticación que no depende de una contraseña como primer factor." cta="Ver glosario" href="/es/docs/glossary?term=Passwordless">Passwordless</Tooltip> permite a los usuarios iniciar sesión recibiendo una contraseña de un solo uso por correo electrónico o mensaje de texto. Para ello, deberás iniciar el proceso passwordless, generar y enviar un código al usuario (o un código dentro de un enlace) y, a continuación, aceptar sus credenciales mediante el método de verificación. Esto puede hacerse mediante una pantalla de inicio de sesión que solicite su correo electrónico o número de teléfono y el código que acabas de enviarle. También puede implementarse mediante un enlace passwordless, en lugar de enviarle un código. El usuario solo tendría que hacer clic en el enlace del correo electrónico o mensaje de texto para acceder a tu endpoint y verificar estos datos automáticamente con el mismo método de verificación (simplemente sin que tenga que introducir manualmente un código).

Para usar la autenticación passwordless, inicializa Auth0.js con un `redirectUri` y establece `responseType: 'token'`.

export const codeExample3 = `var webAuth = new auth0.WebAuth({
  clientID: '{yourClientId}',
  domain: '{yourDomain}',
  redirectUri: 'http://example.com',
  responseType: 'token id_token'
});`;

<div id="start-passwordless-authentication">
  ### Iniciar la autenticación passwordless
</div>

El primer paso de la autenticación passwordless con Auth0.js es el método `passwordlessStart`, que tiene varios parámetros que pueden pasarse dentro de su objeto `options`:

| **Parámetro** | **Requerido** | **Descripción**                                                                                   |
| ------------- | ------------- | ------------------------------------------------------------------------------------------------- |
| `connection`  | requerido     | (String) Especifica cómo enviar el código/enlace al usuario. El valor debe ser `email` o `sms`.   |
| `send`        | requerido     | (String) El valor debe ser `code` o `link`. Si es `null`, se enviará un enlace.                   |
| `phoneNumber` | opcional      | (String) El número de teléfono del usuario para enviar un código o enlace por SMS.                |
| `email`       | opcional      | (String) El correo electrónico del usuario para enviar un código o enlace por correo electrónico. |

Ten en cuenta que, para iniciar la transacción passwordless, debe enviarse exactamente uno de los parámetros opcionales `phoneNumber` y `email`.

```js lines
webAuth.passwordlessStart({
    connection: 'email',
    send: 'code',
    email: 'foo@bar.com'
  }, function (err,res) {
    // manejar errores o continuar
  }
);
```

<div id="complete-passwordless-authentication">
  ### Completar la autenticación passwordless
</div>

Si envías un código, después tendrás que pedirle al usuario que lo introduzca. Procesarás el código y autenticarás al usuario con el método `passwordlessLogin`, que tiene varios parámetros que se pueden enviar en su objeto `options`:

| **Parámetro**      | **Obligatorio** | **Descripción**                                                                                                                                        |
| ------------------ | --------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `connection`       | obligatorio     | (String) Especifica cómo enviar el código/enlace al usuario. El valor debe ser `email` o `sms`, y debe ser el mismo que se pasó a `passwordlessStart`. |
| `verificationCode` | obligatorio     | (String) El código enviado al usuario, ya sea como código o incluido en un enlace.                                                                     |
| `phoneNumber`      | opcional        | (String) El número de teléfono del usuario al que se envió el código o enlace por SMS.                                                                 |
| `email`            | opcional        | (String) El correo electrónico del usuario al que se envió el código o enlace por email.                                                               |

Al igual que con `passwordlessStart`, se debe enviar exactamente uno de los parámetros opcionales `phoneNumber` y `email` para verificar la transacción passwordless.

Para usar `passwordlessLogin`, especifica `redirectUri` y `responseType` al inicializar WebAuth.

```js lines
webAuth.passwordlessLogin({
    connection: 'email',
    email: 'foo@bar.com',
    verificationCode: '389945'
  }, function (err,res) {
    // manejar errores o continuar
  }
);
```

<div id="extract-the-authresult-and-get-user-info">
  ## Extrae el authResult y obtén información del usuario
</div>

Después de la autenticación, puedes usar el método `parseHash` para analizar el fragmento hash de una URL cuando se redirige al usuario de vuelta a tu aplicación, a fin de extraer el resultado de una respuesta de autenticación de Auth0. Puedes gestionar esto en una página de devolución de llamada que luego redirija a tu aplicación principal o directamente en la página, según convenga.

El método `parseHash` toma un objeto `options` que contiene los siguientes parámetros:

| **Parameter**                                                                                                                                                                                                     | **Required** | **Description**                                                                                                                                                                       |
| ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `state`                                                                                                                                                                                                           | optional     | (String) Un valor opaco que la aplicación agrega a la request inicial y que Auth0 incluye al redirigir de vuelta a la aplicación. Auth0.js usa este valor para prevenir ataques CSRF. |
| <Tooltip tip="Nonce: Número arbitrario emitido una sola vez en un protocolo de autenticación para detectar y prevenir ataques de repetición." cta="Ver glosario" href="/es/docs/glossary?term=nonce">nonce</Tooltip> | optional     | (String) Se usa para verificar el token de ID                                                                                                                                         |
| `hash`                                                                                                                                                                                                            | optional     | (String) El hash de la URL (si no se proporciona, se usará `window.location.hash` de forma predeterminada)                                                                            |

El contenido del objeto authResult que devuelve `parseHash` depende de los parámetros de autenticación que se hayan usado. Puede incluir lo siguiente:

| **Item**      | **Description**                                                                                                                                                                                                                                            |
| ------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `accessToken` | Un <Tooltip tip="Access Token: Credencial de autorización, en forma de una cadena opaca o JWT, usada para acceder a una API." cta="Ver glosario" href="/es/docs/glossary?term=Access+Token">Access Token</Tooltip> para la API especificada por el `audience` |
| `expiresIn`   | Una cadena que contiene el tiempo de expiración (en segundos) del `accessToken`                                                                                                                                                                            |
| `idToken`     | Un JWT de token de ID que contiene información del perfil del usuario                                                                                                                                                                                      |

```js lines
webAuth.parseHash({ hash: window.location.hash }, function(err, authResult) {
  if (err) {
    return console.log(err);
  }

  webAuth.client.userInfo(authResult.accessToken, function(err, user) {
    // Ahora tienes la información del usuario
  });
});
```

Como se muestra arriba, el método `client.userInfo` se puede llamar pasando el `accessToken` devuelto. Hará una solicitud al endpoint `/userinfo` y devolverá el objeto `user`, que contiene la información del usuario, con un formato similar al del ejemplo siguiente.

```json lines
{
    "sub": "auth0|123456789012345678901234",
    "nickname": "johnfoo",
    "name": "johnfoo@gmail.com",
    "picture": "https://gravatar.com/avatar/example.png",
    "updated_at": "2018-05-07T14:16:52.013Z",
    "email": "johnfoo@gmail.com",
    "email_verified": "false"
}
```

Ahora puedes hacer algo más con esta información según las necesidades de tu aplicación, como obtener toda la información del perfil del usuario con la <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/es/docs/glossary?term=Management+API">Management API</Tooltip>, como se describe a continuación.

<div id="using-nonces">
  ## Uso de nonces
</div>

De forma predeterminada (y si `responseType` contiene `id_token`), Auth0.js generará un `nonce` aleatorio cuando llames a `webAuth.authorize`, lo almacenará en local storage y lo recuperará en `webAuth.parseHash`. El comportamiento predeterminado debería funcionar en la mayoría de los casos, pero algunos casos de uso pueden requerir que el desarrollador controle el `nonce`.
Si quieres usar un `nonce` generado por el desarrollador, debes proporcionarlo como opción tanto a `webAuth.authorize` como a `webAuth.parseHash`.
`webAuth.authorize({<Tooltip tip="Nonce: Número arbitrario emitido una sola vez en un protocolo de autenticación para detectar y prevenir ataques de repetición." cta="Ver glosario" href="/docs/glossary?term=nonce">nonce</Tooltip>: '1234', responseType: 'token id_token'});
webAuth.parseHash({nonce: '1234'}, callback);`

Si llamas a `webAuth.checkSession` en lugar de `webAuth.authorize`, solo tienes que especificar tu `nonce` personalizado como una opción de `checkSession`:

```js lines
webAuth.checkSession({
  nonce: '1234',
}, function (err, authResult) {
    ...
});
```

El método `webAuth.checkSession` verificará automáticamente que el claim `nonce` del <Tooltip tip="ID Token: Credencial destinada al propio cliente, en lugar de usarse para acceder a un recurso." cta="Ver glosario" href="/es/docs/glossary?term=ID+Token">token de ID</Tooltip> devuelto coincida con la opción.

<div id="logout">
  ## Cierre de sesión
</div>

Para cerrar la sesión de un usuario, usa el método `logout()`. Este método acepta un objeto de opciones, que puede incluir los siguientes parámetros.

Si se incluye el parámetro `clientID`, la URL `returnTo` proporcionada debe figurar en **Allowed Logout URLs** de la aplicación en el [Auth0 Dashboard](https://manage.auth0.com/#). Sin embargo, si no se incluye el parámetro `clientID`, la URL `returnTo` debe figurar en **Allowed Logout URLs** a nivel de cuenta en el [Auth0 Dashboard](https://manage.auth0.com/#).

```js lines
webAuth.logout({
  returnTo: 'some url here',
  clientID: 'some client ID here'
});
```

<div id="signup">
  ## Registro
</div>

Para registrar un usuario, use el método `signup`. Este método acepta un objeto de opciones, que puede incluir los siguientes parámetros.

| **Parámetro**   | **Obligatorio** | **Descripción**                                                                                                                                       |
| --------------- | --------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------- |
| `email`         | required        | (String) Dirección de correo electrónico del usuario                                                                                                  |
| `password`      | required        | (String) Contraseña que desea usar el usuario                                                                                                         |
| `username`      | required*       | (String) Nombre de usuario que desea usar el usuario. *Obligatorio si usa una conexión de base de datos y ha habilitado **Requires Username**         |
| `connection`    | required        | (String) Nombre de la conexión de base de datos de su aplicación en la que se intentará crear la cuenta de usuario                                    |
| `user_metadata` | optional        | (JSON object) Atributos adicionales para la información del usuario. Se almacenarán en [user&#95;metadata](/es/docs/manage-users/user-accounts/metadata) |

Los registros deben realizarse en conexiones de base de datos. A continuación, se muestra un ejemplo del método `signup` y código de ejemplo para un formulario.

```html lines
<h2>Signup Database Connection</h2>
<input class="signup-email" />
<input type="password" class="signup-password" />
<input type="button" class="signup-db" value="Signup!" />
<script type="text/javascript">
    $('.signup-db').click(function (e) {
        e.preventDefault();
        webAuth.signup({
            connection: 'Username-Password-Authentication',
            email: $('.signup-email').val(),
            password: $('.signup-password').val(),
            user_metadata: { plan: 'silver', team_id: 'a111' }
        }, function (err) {
            if (err) return alert('Something went wrong: ' + err.message);
            return alert('success signup without login!')
        });
    });
</script>
```

<div id="using-checksession-to-acquire-new-tokens">
  ## Uso de checkSession para obtener nuevos tokens
</div>

El método `checkSession` te permite obtener un nuevo token de Auth0 para un usuario que ya se haya autenticado con Auth0 en tu dominio. El método acepta cualquier parámetro válido de OAuth2 que normalmente se enviaría a `authorize`. Si los omites, usará los que se proporcionaron al inicializar Auth0.

La llamada a `checkSession` puede usarse para obtener un nuevo token para la API que se especificó como la <Tooltip tip="Audiencia: identificador único de la audiencia de un token emitido. Se denomina aud en un token, y su valor contiene el ID de una aplicación (ID de cliente) para un token de ID o de una API (identificador de API) para un token de acceso." cta="Ver glosario" href="/es/docs/glossary?term=audience">audiencia</Tooltip> cuando se inicializó `webAuth`:

```js lines
webAuth.checkSession({}, function (err, authResult) {
  // err si el parseHash automático falla
  ...
});
```

Consulta [Extraer el AuthResult y Get User Info](#extract-the-authresult-and-get-user-info) para conocer el formato de `authResult`.

O bien, el token puede obtenerse para una API distinta de la que se utilizó al inicializar `webAuth`, especificando un `audience` y un `scope`:

```js lines
webAuth.checkSession(
  {
    audience: `https://mydomain/another-api/˜`,
    scope: 'read:messages'
  }, function (err, authResult) {
  // err si el parseHash automático falla
  ...
});
```

Ten en cuenta que `checkSession()` ejecuta cualquier [regla](/es/docs/customize/rules) que hayas configurado, así que deberías revisar tus reglas en el [Dashboard](https://manage.auth0.com/#/rules) antes de usarlo.

La redirección a `/authorize` se realiza dentro de un iframe, por lo que no recargará tu aplicación ni te sacará de ella.

Sin embargo, el navegador **debe** tener habilitadas las cookies de terceros. De lo contrario, **checkSession()** no podrá acceder a la sesión del usuario actual (lo que hace imposible obtener un nuevo token sin mostrar nada al usuario). Lo mismo ocurrirá si los usuarios tienen [ITP de Safari habilitado](https://support.auth0.com/center/s/article/troubleshoot-auth0-token-renewal-issues-in-safari-with-itp-enabled).

Recuerda agregar la URL desde la que se origina la solicitud de autorización a la lista de **Allowed Web Origins** de tu aplicación de Auth0 en el [Dashboard](https://manage.auth0.com/#), dentro de **Settings** de tu aplicación.

<Warning>
  Si la conexión es una conexión social y estás usando claves de desarrollo de Auth0, la llamada a `checkSession` siempre devolverá `login_required`.
</Warning>

<div id="polling-with-checksession">
  ### Sondeo con checkSession()
</div>

En algunos escenarios con múltiples aplicaciones, en los que se desea el cierre de sesión único (si un usuario cierra sesión en una aplicación, debe cerrarla también en las demás), se puede configurar una aplicación para que realice consultas periódicas a Auth0 mediante `checkSession()` y compruebe si existe una sesión. Si la sesión no existe, puede cerrar la sesión del usuario en la aplicación. El mismo método de sondeo puede usarse para implementar autenticación silenciosa en un escenario de inicio de sesión único (SSO).

El intervalo de sondeo entre comprobaciones con `checkSession()` debe ser de al menos 15 minutos entre llamadas para evitar posibles problemas futuros con el límite de frecuencia de esta llamada.

Si estás intentando configurar una funcionalidad de restablecimiento de contraseña, usarás el método `changePassword` y pasarás un objeto `options`, con un parámetro `connection` y un parámetro `email`.

```js lines
$('.change_password').click(function () {
```

El usuario recibe un correo electrónico con un enlace para restablecer su contraseña.

La Management API proporciona funcionalidades que te permiten vincular y desvincular cuentas de usuario independientes de distintos proveedores, y actualizar los metadatos del usuario. Puedes hacerlo especificando la audience `https://{yourDomain}/api/v2/` al inicializar auth0.js, en cuyo caso obtendrás el access token como parte del flujo de autenticación.

  redirectUri: 'http://example.com',
  audience: \`https://{yourDomain}/api/v2/\`,
  scope: 'read:current_user',
  responseType: 'token id_token'
});`;

  ### Obtener el perfil del usuario
</div>

Para obtener los datos del perfil del usuario, usa el método `getUser()` con `userId` y una devolución de llamada como parámetros. El método devuelve el perfil del usuario. Ten en cuenta que el `userID` necesario aquí será el mismo que se obtuvo del método `client.userInfo`.
`auth0Manage.getUser(userId, cb);`

<div id="update-the-user-profile">
  ### Actualizar el perfil del usuario
</div>

Al actualizar los metadatos del usuario, primero deberás crear un objeto `userMetadata` y luego llamar al método `patchUserMetadata`, pasándole el ID del usuario y el objeto `userMetadata` que creaste. Los valores de este objeto sobrescribirán los valores existentes que tengan la misma clave o añadirán otros nuevos si aún no existen en los metadatos del usuario. Para obtener más información, consulta [Metadatos](/es/docs/manage-users/user-accounts/metadata).
`auth0Manage.patchUserMetadata(userId, userMetadata, cb);`

<div id="link-users">
  ### Vincular usuarios
</div>

Vincular cuentas de usuario permite que un usuario se autentique con cualquiera de sus cuentas y, sin importar cuál use, acceda al mismo perfil al iniciar sesión. Auth0 trata todas estas cuentas como perfiles separados de forma predeterminada, así que, si quieres vincular las cuentas de un usuario, esta es la forma de hacerlo.

El método `linkUser` acepta dos parámetros: el `userId` principal y el token de ID del usuario secundario (el token obtenido después de iniciar sesión con esta identidad). El ID de usuario en cuestión es el identificador único de la cuenta de usuario principal. Al usar este método, el ID debe pasarse con el prefijo del proveedor, por ejemplo, `auth0|1234567890` o `facebook|1234567890`. Consulta [Vinculación de cuentas de usuario](/es/docs/manage-users/user-accounts/user-account-linking) para obtener más información.
`auth0Manage.linkUser(userId, secondaryUserToken, cb);`

Después de vincular las cuentas, la segunda cuenta dejará de existir como una entrada independiente en la base de datos de usuarios y solo se podrá acceder a ella como parte de la principal.
Cuando las cuentas están vinculadas, los metadatos de la cuenta secundaria no se fusionan con los de la cuenta principal y, si en algún momento se desvinculan, la cuenta secundaria tampoco conservará los metadatos de la cuenta principal cuando vuelva a separarse.
