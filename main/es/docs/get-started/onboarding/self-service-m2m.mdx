Si su caso de uso está dirigido a una <Tooltip tip="Audiencia: identificador único de la audiencia de un token emitido. <div id="use-cases">
  ## Casos de uso
</div>

Usa el proceso de incorporación de M2M si:

* Das soporte a comunicaciones de servicio a servicio
* Tienes tareas programadas o tareas `cron` ejecutándose en servidores que necesitan acceder a recursos protegidos o a APIs
* Permites que dispositivos IoT se comuniquen con servicios de backend o APIs
* Tienes una capa de API que necesita comunicarse con otras capas de API sin intervención del usuario o después de que haya expirado el token de un usuario
* Tienes una API privilegiada que puede necesitar ser llamada antes de que un usuario se haya autenticado (es decir, desde una Action o un script de base de datos personalizada en tu tenant de Auth0)
* Usas un API Gateway para gestionar servicios de backend
* Usas o das soporte a aplicaciones no interactivas u otras herramientas sin interacción humana, como daemons o servicios de backend

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Estos servicios seguirán necesitando un token de acceso de M2M para la autenticación.
</Callout>

Como se menciona en los escenarios comunes, puede que tenga otras tecnologías no interactivas en el dominio de su aplicación, el dominio de su red o el dominio de dispositivos M2M que deba tener en cuenta antes de configurar Auth0. <div id="considerations">
  ### Consideraciones
</div>

Antes de crear una cuenta nueva o configurar tu primer tenant, quizá te convenga tener en cuenta lo siguiente:

* Cómo particionas o agrupas tus API para llamar a endpoints específicos.

  * Esto puede determinar la audiencia y otros claims de los tokens de acceso.
* Cualquier consumidor de terceros de tu recurso puede solicitar un token de acceso para cada llamada. Un número excesivo de llamadas podría afectar tu [límite de frecuencia](/es/docs/troubleshoot/customer-support/operational-policies/rate-limit-policy/rate-limit-configurations/essentials-professional-b2b).

  * Puedes usar un API Gateway para limitar la cantidad de tokens de acceso que un tercero puede solicitar. Para obtener más información sobre API Gateways y Auth0, consulta [Configurar un proveedor de identidad en Access Gateway](/es/docs/get-started/auth0-overview/create-applications/configure-an-identity-provider-in-access-gateway).

<div id="authentication">
  ## Autenticación
</div>

Cuando una API llama a otra API, o en cualquier situación en la que no haya un contexto de usuario autenticado, necesitas una forma de autorizar a la aplicación en lugar de a un usuario. Este es un proceso de un solo paso en el que la aplicación se autentica (mediante un `client_id` y un `client_secret`) y luego se autoriza en una sola llamada.

Para aplicaciones o servicios no interactivos, debes seleccionar una autorización de cliente o un flujo de autenticación. El <Tooltip tip="OAuth 2.0: marco de autorización que define protocolos y flujos de trabajo de autorización." cta="Ver glosario" href="/es/docs/glossary?term=OAuth+2.0">OAuth 2.0</Tooltip> [flujo de credenciales de cliente](https://tools.ietf.org/html/rfc6749#section-4.4) no requiere interacción humana y es la opción más adecuada para las aplicaciones M2M.

<Card title="Antes de comenzar">
  En Auth0 Dashboard o Management API, harás lo siguiente:

  * Configurar tu aplicación para usar flujo de credenciales de cliente
  * Actualizar los scopes de tus tokens de acceso M2M

  Antes de configurar tu método de autenticación:

  * Revisa el [flujo de credenciales de cliente](/es/docs/get-started/authentication-and-authorization-flow/client-credentials-flow) para la autenticación de máquina a máquina. Este es el flujo de trabajo para la autenticación y autorización no interactivas.
  * Determina el nivel de acceso de tus API. Esto te ayudará a determinar qué [scopes](/es/docs/get-started/apis/scopes/api-scopes), o permisos, configurarás cuando crees tu API.
</Card>

<div id="configure-the-client-credential-flow">

</div>

<div id="m2m-access-tokens">
  ### Tokens de acceso M2M
</div>

En la autenticación basada en tokens, los clientes no interactivos proporcionan `client_id` y `client_secret` en una llamada al [endpoint de token de Authentication API](/es/docs/customize/actions/explore-triggers/machine-to-machine-trigger) para obtener un [token de acceso](/es/docs/secure/tokens/access-tokens). Este token de acceso permite acceder a tu API protegida.

El perfil predeterminado, o formato, es el perfil de token de Auth0 asociado a uno de los dos perfiles de token. Puedes cambiar el perfil del token a RFC 9068. Para obtener más información, consulta [Perfiles de token de acceso](/es/docs/secure/tokens/access-tokens/access-token-profiles). Para verificar que el token sea válido, tu API comprobará los [Algoritmos de firma](/es/docs/get-started/applications/signing-algorithms). El <Tooltip tip="Algoritmo de firma: algoritmo usado para firmar digitalmente tokens y garantizar que el token no haya sido manipulado." cta="Ver glosario" href="/es/docs/glossary?term=signing+algorithm">algoritmo de firma</Tooltip> predeterminado es RSA256, un algoritmo basado en claves.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Auth0 admite otros métodos de autenticación de clientes además de usar el ID de cliente y el secreto de cliente como credenciales. Estos métodos, incluida nuestra recomendación de [clave privada JWT](/es/docs/get-started/authentication-and-authorization-flow/authenticate-with-private-key-jwt) para configuraciones M2M, están disponibles con un plan empresarial. Para obtener más información, consulta [Credenciales de la aplicación](/es/docs/secure/application-credentials).
</Callout>

<div id="token-expiration">
  #### Vencimiento del token
</div>

Tus tokens de acceso tienen un tiempo de validez limitado. Como tus comunicaciones se realizan a través del back-channel, no puedes usar <Tooltip tip="token de actualización: token utilizado para obtener un token de acceso renovado sin obligar a los usuarios a iniciar sesión de nuevo." cta="Ver glosario" href="/es/docs/glossary?term=refresh+tokens">tokens de actualización</Tooltip> para extender las sesiones, por lo que deberías considerar configurar tus tokens de acceso para que venzan al cabo de 1 hora. Es posible que debas encontrar el equilibrio adecuado entre seguridad y rendimiento para tu entorno específico. Para obtener más información, consulta [Actualizar la duración del token de acceso](/es/docs/secure/tokens/access-tokens/update-access-token-lifetime).

<div id="scopes">
  ### Scopes
</div>

Antes de que cualquier cliente o servicio no interactivo llame a tu API, debes definir los permisos o scopes que admite tu API. Puedes configurar los scopes en Auth0 Dashboard para incluirlos en tu solicitud de authentication a Authentication API. Para ver más ejemplos de cómo usar scopes de API, consulta [API Scopes](/es/docs/get-started/apis/scopes/api-scopes).

Para configurar scopes, sigue las instrucciones de [Agregar permisos de API](/es/docs/get-started/apis/add-api-permissions) en Auth0 Dashboard o usa el ejemplo proporcionado para Management API.

  Para agregar claims personalizadas a un token de acceso, puedes usar el flujo de máquina a máquina de Actions. Para obtener más información, consulta Flujo de máquina a máquina.
</Callout>

</div>

Auth0 permite usar dominios personalizados cuando llame al `endpoint` `/authorize` para solicitar tokens de acceso.

<Card title="Incorporación de M2M - dominios personalizados">
  En Auth0 Dashboard, debe:

  * Registrar y verificar su dominio antes de poder usarlo con sus servicios de Auth0.
  * Decidir si desea administrar su propio certificado o usar un certificado gestionado por Auth0. Para obtener más información sobre los certificados, consulte [Certificate management options](/es/docs/customize/custom-domains#certificate-management-options).
  * Verificar que la versión de TLS (SSL) y el cifrado que desea usar para los certificados autogestionados sean compatibles con Auth0. Para obtener más información, consulte [TLS (SSL) Versions and Ciphers](/es/docs/customize/custom-domains/self-managed-certificates/tls-ssl).
</Card>

1. Para configurar su dominio personalizado con certificados gestionados por Auth0, siga las instrucciones de [Configure Custom Domains with Auth0-Managed Certificates](/es/docs/customize/custom-domains/auth0-managed-certificates).

   1. Si desea administrar sus propios certificados, siga las instrucciones de [Configure Custom Domains with Self-Managed Certificates](/es/docs/customize/custom-domains/self-managed-certificates).

      <Callout icon="file-lines" color="#0EA5E9" iconType="regular">
        Debe tener una suscripción Enterprise para administrar certificados en su dominio personalizado. Para obtener más información, consulte [Auth0 Pricing](https://auth0.com/pricing/) y [Login](/es/docs/authenticate/login).
      </Callout>
2. Revise [API configuration with custom domains](/es/docs/customize/custom-domains/configure-features-to-use-custom-domains#apis). Es posible que deba ajustar la configuración de su API para usar un dominio personalizado.

<div id="deployment-automation">
  ## Automatización del despliegue
</div>

Auth0 admite un par de opciones distintas para automatizar el despliegue, y ambas pueden utilizarse en conjunto.

<Card title="Práctica recomendada">
  Independientemente de cómo configure la automatización del despliegue, le recomendamos realizar pruebas unitarias de su código personalizado y de Actions antes del despliegue, y también ejecutar algunas pruebas de integración en su tenant después del despliegue.
</Card>

<div id="deploy-cli-tool">
  ### Herramienta Deploy CLI
</div>

Como se recomienda en la sección de Arquitectura, deberías tener tenants de Auth0 para desarrollo, pruebas y producción. Estos tenants deben compartir configuraciones idénticas para los controles de calidad y las pruebas; sin embargo, puedes encontrarte con errores debido a discrepancias de configuración entre tus entornos. Por ejemplo, cada entorno tendrá diferentes <Tooltip tip="ID de cliente: valor de identificación asignado a tu recurso registrado por Auth0." cta="Ver glosario" href="/es/docs/glossary?term=Client+IDs">IDs de cliente</Tooltip> y <Tooltip tip="ID de cliente: valor de identificación asignado a tu recurso registrado por Auth0." cta="Ver glosario" href="/es/docs/glossary?term=Client+Secrets">secretos de cliente</Tooltip>.

Para mitigar estos errores de discrepancia, puedes usar la herramienta Deploy CLI para integrar tu instancia de Auth0 con tu pipeline de CI/CD actual. Con el reemplazo dinámico de palabras clave, puedes sustituir variables de entorno en tenants que comparten configuraciones similares. Para obtener más información, consulta [Herramienta Deploy CLI](/es/docs/deploy-monitor/deploy-cli-tool) y [Reemplazo de palabras clave](/es/docs/deploy-monitor/deploy-cli-tool/keyword-replacement).

<div id="actions-real-time-logs">
  ### logs en tiempo real de Actions
</div>

Los logs en tiempo real de Actions muestran todos los logs del código personalizado en tiempo real, incluida la salida de `console.log` y otras excepciones. Si usas Auth0 Actions u otra lógica personalizada, puedes usar esta extensión para depurar y solucionar problemas. Para obtener más información sobre la instalación y la configuración, consulta [logs en tiempo real de Actions](/es/docs/customize/actions/actions-real-time-logs).

<div id="quality-assurance">
  ## Control de calidad
</div>

El control de calidad es importante para identificar problemas antes de poner en marcha tu implementación. Según la naturaleza de tu proyecto, hay varios tipos de pruebas de calidad que conviene considerar como parte de tu integración con Auth0:

* ¿Cómo responderán tus API ante cargas de producción inesperadas?
* ¿Cómo afectarán las aplicaciones de terceros a tus límites de frecuencia?

Si no usas widgets o funcionalidades de Auth0, como <Tooltip tip="Universal Login: Tu aplicación redirige a Universal Login, alojado en el servidor de autorización de Auth0, para verificar la identidad de un usuario." cta="Ver glosario" href="/es/docs/glossary?term=Universal+Login">Universal Login</Tooltip>, no contarás de entrada con prácticas recomendadas integradas de usabilidad y accesibilidad para una amplia variedad de navegadores y dispositivos. Para garantizar que se cumplan los requisitos funcionales y que los eventos inesperados se gestionen correctamente, se ofrece orientación para probar la integración entre tu aplicación o aplicaciones y Auth0, así como para realizar pruebas unitarias de módulos individuales de extensibilidad, como Auth0 Actions. También te recomendamos revisar la [política de pruebas de penetración](/es/docs/troubleshoot/customer-support/operational-policies/penetration-testing-policy) de Auth0 y realizar pruebas simuladas que puedas aprovechar junto con nuestra [política de pruebas de carga](/es/docs/troubleshoot/customer-support/operational-policies/load-testing-policy) para ayudar a garantizar que tu aplicación o aplicaciones funcionen correctamente ante cargas inesperadas.

<div id="unit-testing">
  ### Pruebas unitarias
</div>

Las pruebas unitarias sirven para verificar unidades de extensibilidad, como Auth0 Actions. Si utiliza código personalizado, le recomendamos usar un framework de pruebas (como [Mocha](https://mochajs.org/)) para probar ese código adicional antes del despliegue.

<div id="mock-testing">
  ### Prueba simulada
</div>

Como punto medio entre la [política de pruebas de carga](/es/docs/troubleshoot/customer-support/operational-policies/load-testing-policy) de Auth0 y la necesidad de realizar pruebas de carga, es habitual crear una prueba simulada de los endpoints de Auth0. Esta práctica resulta muy útil para asegurarte de que tu configuración funcione con las interfaces previstas sin tener que limitar tus pruebas, y puedes apoyarte en herramientas como [MockServer](http://www.mock-server.com/), [JSON Server](https://github.com/typicode/json-server) o incluso [Postman](https://learning.getpostman.com/docs/postman/mock_servers/setting_up_mock/).

<Frame>![Auth0 Dashboard > Lista de comprobación de preparación](/docs/images/cdy7uua7fh8z/7iu1CzH0NgaXJbLBNWXZli/b598b8cdaad0c676e83a0fb0595d4603/Readiness_Checks_-_English.png)</Frame>
