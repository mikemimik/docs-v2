En este escenario construiremos una API de hojas de hora para una empresa ficticia llamada ExampleCo. La API permitirá agregar registros de hoja de hora para un empleado o un contratista.

También construiremos una tarea cron que procesará registros de hoja de hora desde un sistema externo hacia la base de datos centralizada de hojas de hora mediante la API.

<Info>
  ### Resumen rápido

  * Auth0 proporciona authentication y authorization para API como una forma de proteger el acceso a los endpoints de la API (lea [Autenticación y autorización de API](/es/docs/get-started/architecture-scenarios/server-application-api/part-1#api-authentication-and-authorization))
  * Para autorizar una aplicación de máquina a máquina (una CLI, un servicio o un demonio sin interacción del usuario), Auth0 admite la Concesión de credenciales de cliente (lea [Concesión de credenciales de cliente](/es/docs/get-started/architecture-scenarios/server-application-api/part-1#client-credentials-grant))
  * Tanto la aplicación de máquina a máquina como la API deben configurarse en el Auth0 Dashboard (lea [Configuración de Auth0](/es/docs/get-started/architecture-scenarios/server-application-api/part-2))
  * La API estará protegida al garantizar que se envíe un access token válido (implementado como un JSON Web Token) en el encabezado HTTP Authorization al realizar llamadas a la API (lea [Implementar la API](/es/docs/get-started/architecture-scenarios/server-application-api/part-3))
  * Tras una autorización correcta, se emite un access token para la aplicación de máquina a máquina (lea [Obtener un access token](/es/docs/get-started/architecture-scenarios/server-application-api/part-3#get-an-access-token))
  * A su vez, la aplicación de máquina a máquina puede usar este access token y enviarlo en el encabezado HTTP Authorization para autenticar las llamadas a los endpoints de la API (lea [Invocar la API](/es/docs/get-started/architecture-scenarios/server-application-api/part-3#invoke-the-api))
</Info>

<div id="the-premise">
  ## La premisa
</div>

ExampleCo es una startup de consultoría. Actualmente tiene aproximadamente 100 empleados y también recurre a contratistas externos. Todos los empleados y contratistas externos deben completar hojas de hora cada semana. Para ello, crearon una aplicación de hojas de hora, un escenario que cubrimos en [Inicio de sesión único para aplicaciones web tradicionales](/es/docs/get-started/architecture-scenarios/sso-for-regular-web-apps). Los empleados internos usan esta aplicación web para completar sus hojas de hora, pero algunos de los contratistas externos ya usan otra herramienta para registrar sus hojas de hora. ExampleCo necesita una solución para evitar el trabajo duplicado y decidió crear una tarea cron para leer los registros de la hoja de hora de este sistema externo y enviarlos automáticamente al backend de ExampleCo mediante una API.

<div id="goals-requirements">
  ### Objetivos y requisitos
</div>

ExampleCo quiere crear una solución flexible que:

* Permita procesos automatizados que se ejecuten en el backend
* Permita incorporar en la arquitectura futuros lanzamientos de aplicaciones, como una aplicación móvil
* Permita que la API de hojas de hora sea segura y que solo puedan acceder a ella usuarios y aplicaciones autorizados
* Permita compartir gran parte del código y la lógica de negocio de la aplicación con otras aplicaciones
