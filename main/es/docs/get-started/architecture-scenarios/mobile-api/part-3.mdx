<div id="define-the-api-endpoints">
  ## Define los endpoints de la API
</div>

Primero, necesitamos definir los endpoints de nuestra API.

<Card title="¿Qué es un endpoint de la API?">
  Un **endpoint de la API** es una URL única que representa un objeto. Para interactuar con este objeto, debes apuntar tu aplicación a su URL. Por ejemplo, si tuvieras una API que pudiera devolver pedidos o clientes, podrías configurar dos endpoints: `/orders` y `/customers`. Tu aplicación interactuaría con estos endpoints usando distintos métodos HTTP; por ejemplo, `POST /orders` podría crear un nuevo pedido o `GET /orders` podría recuperar los datos de uno o varios pedidos.
</Card>

Para esta implementación, solo definiremos dos endpoints: uno para recuperar una lista de todas las hojas de horas de un empleado y otro que permitirá a un empleado crear una nueva entrada de hoja de horas.

Una solicitud `HTTP GET` al endpoint `/timesheets` permitirá a un usuario recuperar sus hojas de horas, y una solicitud `HTTP POST` al endpoint `/timesheets` permitirá a un usuario añadir una nueva hoja de horas.

Consulta la implementación en [Node.js](/es/docs/get-started/architecture-scenarios/mobile-api/api-implementation-nodejs#1-define-the-api-endpoints)

<div id="secure-the-endpoints">
  ### Proteger los endpoints
</div>

Cuando una API recibe una solicitud con un <Tooltip tip="Token de acceso: Credencial de autorización, en forma de una cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/es/docs/glossary?term=Access+Token">token de acceso</Tooltip> bearer en la cabecera, lo primero que debe hacer es validar el token. Esto consiste en una serie de pasos y, si alguno falla, la solicitud debe rechazarse con el mensaje de error `Missing or invalid token` dirigido a la aplicación que realiza la llamada.

Las validaciones que la API debe realizar son:

* Comprobar que el <Tooltip tip="JSON Web Token (JWT): Formato estándar de token de ID (y, a menudo, de token de acceso) utilizado para representar claims de forma segura entre dos partes." cta="Ver glosario" href="/es/docs/glossary?term=JWT">JWT</Tooltip> tenga un formato correcto
* Comprobar la firma
* Validar las claims estándar

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  [JWT.io](https://jwt.io/) proporciona una lista de bibliotecas que pueden hacer la mayor parte del trabajo por ti: analizar el JWT, verificar la firma y las claims.
</Callout>

Parte del proceso de validación también consiste en comprobar los permisos del cliente (scopes), pero lo abordaremos por separado en el siguiente párrafo de este documento.

Para obtener más información sobre cómo validar tokens de acceso, consulta [Validate Access Tokens](/es/docs/secure/tokens/access-tokens/validate-access-tokens).

Consulta la implementación en [Node.js](/es/docs/get-started/architecture-scenarios/mobile-api/api-implementation-nodejs#2-secure-the-api-endpoints)

<div id="check-the-clients-permissions">
  ### Comprueba los permisos del cliente
</div>

A estas alturas, ya hemos verificado que el JWT es válido. El último paso es comprobar que el cliente tenga los permisos necesarios para acceder a los recursos protegidos.

Para ello, la API debe comprobar los [scopes](/es/docs/get-started/apis/scopes) del JWT decodificado. Este claim forma parte del payload y es una lista de cadenas separadas por espacios.

Consulta la implementación en [Node.js](/es/docs/get-started/architecture-scenarios/mobile-api/api-implementation-nodejs#3-check-the-client-permissions)

<div id="determine-user-identity">
  ### Determinar la identidad del usuario
</div>

Para ambos endpoints (obtener la lista de hojas de horas y añadir una nueva hoja de horas), necesitaremos determinar la identidad del usuario.

Al obtener la lista de hojas de horas, esto garantiza que solo devolvamos las hojas de horas que pertenecen al usuario que realiza la solicitud; y al añadir una nueva hoja de horas, garantiza que la hoja de horas quede asociada al usuario que realiza la solicitud.

Uno de los `claims` estándar de JWT es el `claim` `sub`, que identifica al principal que es el sujeto del `claim`. En el caso del flujo Implicit Grant, este `claim` contendrá la identidad del usuario, que será el identificador único del usuario de Auth0. Puede usarlo para asociar cualquier información en sistemas externos con un usuario concreto.

También puede usar un `claim` personalizado para añadir otro atributo del usuario, como su dirección de correo electrónico, al token de acceso y usarlo para identificar al usuario de forma única.

Consulte la implementación en [Node.js](/es/docs/get-started/architecture-scenarios/mobile-api/api-implementation-nodejs#4-determine-the-user-identity)

<div id="authorize-the-user">
  ### Autorizar al usuario
</div>

Para autorizar al usuario, implementaremos el [Authorization Code Flow with Proof Key for Code Exchange (PKCE)](/es/docs/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce/call-your-api-using-the-authorization-code-flow-with-pkce). La aplicación móvil primero debe redirigir al usuario a la [URL de autorización](https://auth0.com/docs/api/authentication#authorization-code-grant-pkce-) junto con el `code_challenge` y el método utilizado para generarlo:

export const codeExample1 = `https://{yourDomain}/authorize?
    audience=API_AUDIENCE&
    scope=SCOPE&
    response_type=code&
    client_id={yourClientId}&
    code_challenge=CODE_CHALLENGE&
    code_challenge_method=S256&
    redirect_uri=https://YOUR_APP/callback`;

<AuthCodeBlock children={codeExample1} language="http" />

La solicitud `GET` a la URL de autorización debe incluir los siguientes valores:

| Parameter                         | Description                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| --------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **client&#95;id**                 | El valor del ID de cliente de Auth0. Puedes obtenerlo en la configuración de tu aplicación en el [Auth0 Dashboard](https://manage.auth0.com/#/applications).                                                                                                                                                                                                                                                                                                                                                                                                |
| **audience**                      | El valor del identificador de tu API. Puedes obtenerlo en la configuración de tu API en el [Auth0 Dashboard](https://manage.auth0.com/#/apis).                                                                                                                                                                                                                                                                                                                                                                                                              |
| **scope**                         | Los [scopes](/es/docs/get-started/apis/scopes) que determinan los claims que se devolverán en el ID Token y el token de acceso. Por ejemplo, un scope de `openid` devolverá un ID Token en la respuesta. En nuestra aplicación móvil de ejemplo, usamos los siguientes scopes: `create:timesheets read:timesheets openid profile email offline_access`. Estos scopes permiten que la aplicación móvil llame a la API, obtenga un [Refresh Token](/es/docs/glossary?term=Refresh+Token) y devuelva los claims `name`, `picture` y `email` del usuario en el ID Token. |
| **response&#95;type**             | Indica el flujo de autenticación que se debe usar. En una aplicación móvil que usa PKCE, este valor debe establecerse en `code`.                                                                                                                                                                                                                                                                                                                                                                                                                            |
| **code&#95;challenge**            | El code challenge generado a partir del code verifier. Puedes encontrar instrucciones para generar un code challenge [aquí](/es/docs/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce/call-your-api-using-the-authorization-code-flow-with-pkce#authorize-the-user%23create-a-code-verifier).                                                                                                                                                                                                                               |
| **code&#95;challenge&#95;method** | Método utilizado para generar el challenge. Auth0 solo admite `S256`.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                       |
| **redirect&#95;uri**              | La URL a la que Auth0 redirigirá el navegador después de que el usuario haya concedido la autorización. El Authorization Code estará disponible en el parámetro `code` de la URL. Esta URL debe especificarse como una URL de callback válida en la [configuración de tu aplicación](https://manage.auth0.com/#/applications).                                                                                                                                                                                                                              |

A continuación, puedes canjear el `authorization_code` de la respuesta por un token de acceso que podrás usar para hacer llamadas a tu API. Realiza una solicitud `POST` a la [URL del token](https://auth0.com/docs/api/authentication#authorization-code-pkce-) e incluye los siguientes datos:

  ```bash cURL
    --data 'client_id={yourClientId}' \
    --data code_verified=YOUR_GENERATED_CODE_VERIFIER \
    --data code=YOUR_AUTHORIZATION_CODE \
    --data 'redirect_uri=https://{https://yourApp/callback}'
  ```

  ```csharp C#

  IRestResponse response = client.Execute(request);
  ```

| Parámetro             | Descripción                                                                                                                                                                                                            |
| --------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |

| **client&#95;id**     | El valor de tu Id. de cliente de Auth0. Puedes obtenerlo en la Configuración de tu aplicación en el [Auth0 Dashboard](https://manage.auth0.com/#/applications).                                                        |
| **code&#95;verifier** | Clave aleatoria criptográficamente segura que se utilizó para generar el `code_challenge` enviado a la [URL de autorización](https://auth0.com/docs/api/authentication#authorization-code-grant-pkce-) (`/authorize`). |

La respuesta de la URL del token contendrá:

```json lines
{
  "access_token": "eyJz93a...k4laUWw",
  "refresh_token": "GEbRxBN...edjnXbL",
  "id_token": "eyJ0XAi...4faeEoQ",
  "token_type": "Bearer",
  "expires_in":86400
}
```

* **access&#95;token**: Un token de acceso para la API, especificado por `audience`.
* **refresh&#95;token**: Un [Refresh Token](/es/docs/secure/tokens/refresh-tokens) solo estará presente si incluiste el scope `offline_access` Y habilitaste **Allow Offline Access** para tu API en el Dashboard.
* **id&#95;token**: Un <Tooltip tip="ID Token: Credencial destinada al propio cliente, en lugar de para acceder a un recurso." cta="Ver glosario" href="/es/docs/glossary?term=ID+Token">ID Token</Tooltip> JWT que contiene información del perfil del usuario.
* **token&#95;type**: Una cadena que indica el tipo de token; siempre será un token Bearer.
* **expires&#95;in**: La cantidad de segundos hasta que expire el token de acceso.

Tendrás que almacenar las credenciales anteriores en el almacenamiento local para usarlas al llamar a tu API y recuperar el perfil del usuario.

<div id="get-the-user-profile">
  ### Obtener el perfil de usuario
</div>

Para obtener el [perfil de usuario](https://auth0.com/docs/api/authentication?http#user-profile), tu aplicación móvil puede decodificar el [ID Token](/es/docs/secure/tokens/id-tokens) con una de las [bibliotecas JWT](https://jwt.io/#libraries-io). Esto se hace [verificando la firma](/es/docs/secure/tokens/id-tokens/validate-id-tokens#verify-the-signature) y [verificando los claims](/es/docs/secure/tokens/id-tokens/validate-id-tokens#verify-the-claims) del token. Después de validar el ID Token, puedes acceder a su payload, que contiene la información del usuario:

```json lines
{
  "email_verified": false,

}
```

<div id="display-ui-elements-conditionally-based-on-scope">
  ### Mostrar elementos de la interfaz de usuario de forma condicional según el `scope`
</div>

Según el `scope` del usuario, puede que quieras mostrar u ocultar ciertos elementos de la interfaz de usuario. Será una cadena que contiene todos los scopes, así que tendrás que inspeccionarla para ver si contiene el `scope` necesario y, en función de eso, decidir si mostrar un elemento concreto de la interfaz de usuario.

<div id="call-the-api">
  ### Llamar a la API
</div>

Para acceder a los recursos protegidos de tu API, el token de acceso del usuario autenticado debe incluirse en las solicitudes que se envían a ella. Para ello, envía el token de acceso en un encabezado `Authorization` con el esquema `Bearer`.

<div id="renew-the-token">
  ### Renovar el token
</div>

<Warning>
  Los Refresh Tokens deben almacenarse de forma segura en una aplicación, ya que no vencen y permiten que un usuario permanezca autenticado prácticamente para siempre. Si los Refresh Tokens se ven comprometidos o ya no los necesitas, puedes revocarlos mediante la [Authentication API](https://auth0.com/docs/api/authentication#revoke-refresh-token).
</Warning>

Para renovar tu token de acceso, realiza una solicitud `POST` al endpoint `/oauth/token` con el <Tooltip tip="Refresh Token: token que se utiliza para obtener un token de acceso renovado sin obligar a los usuarios a volver a iniciar sesión." cta="Ver glosario" href="/es/docs/glossary?term=Refresh+Token">Refresh Token</Tooltip> del resultado de autorización.

Un [Refresh Token](/es/docs/secure/tokens/refresh-tokens) solo estará presente si incluiste el scope `offline_access` en la solicitud de autorización anterior y habilitaste **Allow Offline Access** para tu API en el Dashboard.

Tu solicitud debe incluir:
