---

Antes del lanzamiento, deberías haber completado todas las pruebas aplicables a tu entorno.

El aseguramiento de la calidad es fundamental para identificar problemas antes de que afecten a tus clientes y, según la naturaleza de tu proyecto, hay varios tipos de pruebas de calidad que te conviene considerar como parte de tu integración con Auth0:

* ¿Tu aplicación es fácil de entender y usar, incluso para personas con alguna discapacidad?
* ¿Tu aplicación necesita funcionar en distintos navegadores y dispositivos?
* ¿Tu aplicación necesita funcionar en entornos multinacionales o internacionales?
* ¿Cómo responderá tu aplicación cuando se vea sometida a cargas de producción inesperadas?
* ¿Cómo puedes asegurarte de que tu aplicación esté protegida frente a vulnerabilidades de seguridad?

Auth0 [Universal Login](/es/docs/authenticate/login/auth0-universal-login) y los widgets de UI asociados (como [Lock](/es/docs/libraries/lock)) ya se han diseñado y desarrollado siguiendo prácticas recomendadas de usabilidad y accesibilidad, y ofrecen soporte probado listo para usar para una amplia variedad de [navegadores y dispositivos](/es/docs/troubleshoot/customer-support/product-support-matrix). También se ofrece soporte listo para usar para la [internacionalización](/es/docs/customize/internationalization-and-localization) (I18N), con extensibilidad integrada diseñada para situaciones personalizadas de varios idiomas y localización (L10N).

Para garantizar que se cumplan los requisitos funcionales y que los eventos inesperados se gestionen correctamente, se ofrece orientación para probar la [integración](#integration-testing) entre tu(s) aplicación(es) y Auth0, así como para las [pruebas unitarias](#unit-testing) de módulos individuales de extensibilidad (como [Rules](/es/docs/customize/rules/debug-rules), [Hooks](/es/docs/customize/hooks/update-hooks) y scripts de base de datos personalizada). También se ofrece orientación sobre la [política de pruebas de penetración de Auth0](/es/docs/troubleshoot/customer-support/operational-policies/penetration-testing-policy) para ayudarte a detectar vulnerabilidades de seguridad, así como sobre cómo pueden aprovecharse las pruebas con [prueba simulada](#mock-testing) junto con nuestra [política de pruebas de carga](/es/docs/troubleshoot/customer-support/operational-policies/load-testing-policy) para ayudar a garantizar que tu(s) aplicación(es) rindan bien ante cargas inesperadas.

<div id="unit-testing">
  ## Pruebas unitarias
</div>

El objetivo de las pruebas unitarias es comprobar unidades individuales de código. Si crea código personalizado dentro de Auth0 en forma de Rules, Hooks y/o scripts de Custom DB, debería considerar usar un framework de pruebas (como [Mocha](https://mochajs.org/)) para probar su código. Las empresas que han tenido más éxito con Auth0 han comprobado que resulta útil ejecutar estas pruebas unitarias antes de [implementar automáticamente](/es/docs/get-started/architecture-scenarios/business-to-business/deployment) la configuración y otros recursos del tenant de Auth0.

<div id="integration-testing">
  ## Pruebas de integración
</div>

Una práctica recomendada es configurar distintos tenants para desarrollo, pruebas y producción, como se explica en la guía de Architecture para [soporte para SDLC](/es/docs/get-started/architecture-scenarios/business-to-business/architecture#sdlc-support). Auth0 le permite configurar variables disponibles desde la [extensibilidad](/es/docs/customize/extensions) personalizada; puede considerarlas variables de entorno para su tenant de Auth0. En lugar de codificar de forma fija referencias que cambian al mover código entre entornos de desarrollo, pruebas y producción, puede usar un nombre de variable configurado en el tenant y referenciado por el código de extensibilidad personalizado. Esto facilita que el mismo código personalizado funcione sin cambios en distintos tenants, ya que puede hacer referencia a variables que se completarán con valores específicos del tenant en tiempo de ejecución:

* Para usar variables en Rules, vea cómo [configurar valores](/es/docs/customize/rules/configuration)
* Para usar variables en Hooks, vea cómo configurar [secrets](/es/docs/customize/hooks/hook-secrets) en el editor
* Para usar variables en Actions, vea Explore Flows and Triggers
* Para usar variables en scripts de Custom DB, vea los [parámetros de configuración](/es/docs/authenticate/database-connections/custom-db/create-db-connection#step-3-add-configuration-parameters)

<Info>
  ### Práctica recomendada

  Se recomienda usar variables para almacenar valores específicos del tenant, así como cualquier secreto sensible que no deba exponerse en su código personalizado. Si su código personalizado se despliega en GitHub, usar una variable específica del tenant evita exponer valores sensibles a través de su repositorio de GitHub.
</Info>

<div id="test-automation">
  ### Automatización de pruebas
</div>

Puedes automatizar todo tu proceso de compilación incorporando tanto la automatización de despliegues como la automatización de pruebas. Esto puede utilizarse para desplegar nuevas versiones de la configuración y/o del código personalizado en Auth0 y ejecutar pruebas automatizadas. Si las pruebas detectan algún fallo, las capacidades de automatización de despliegues pueden utilizarse para revertir a la última versión que funcionaba. Para obtener más información, consulta la [guía sobre automatización de despliegues](/es/docs/get-started/architecture-scenarios/business-to-business/deployment).

<div id="mock-testing">
  ## Prueba simulada
</div>

Como equilibrio entre la [política de pruebas de carga](/es/docs/troubleshoot/customer-support/operational-policies/load-testing-policy) de Auth0 y la necesidad de realizar una prueba de carga, es habitual que los clientes de Auth0 simulen los endpoints de Auth0. Esta es una práctica valiosa para garantizar que tu aplicación funcione con las interfaces previstas sin tener que limitar tus pruebas, y puedes apoyarte en herramientas como [MockServer](http://www.mock-server.com/), [JSON Server](https://github.com/typicode/json-server) o incluso [Postman](https://learning.getpostman.com/docs/postman/mock_servers/setting_up_mock/) para hacerlo.

<div id="pen-testing-optional">
  ## Pruebas de penetración (opcional)
</div>

Si va a realizar pruebas de penetración, debe conocer la [política de pruebas de penetración](/es/docs/troubleshoot/customer-support/operational-policies/penetration-testing-policy) de Auth0 y cumplirla. Las pruebas de penetración requieren avisar a Auth0 con antelación para que no se confundan con actividad maliciosa y se suspendan.

<div id="load-testing-optional">
  ## Pruebas de carga (opcional)
</div>

Si va a realizar pruebas de carga, debe conocer la [política de pruebas de carga](/es/docs/troubleshoot/customer-support/operational-policies/load-testing-policy) de Auth0 y cumplirla. Las pruebas de carga requieren notificación previa a Auth0. Al planificar sus pruebas de carga, también deberá tener en cuenta los [límites de frecuencia de la API](/es/docs/troubleshoot/customer-support/operational-policies/rate-limit-policy) de Auth0.

Las pruebas de carga requieren la aprobación previa de Auth0, como se explica en la política de pruebas de carga de Auth0. Asegúrese de tener en cuenta el tiempo de antelación necesario para que se revise una solicitud y de dejar margen suficiente tanto para la revisión como para la realización de las pruebas. Si se ha aprobado su solicitud de prueba de carga, las siguientes recomendaciones pueden ayudarle a evitar errores y resultados de prueba inexactos.

* Ejecute una traza HTTP en una ejecución de prueba de su aplicación para identificar todas las llamadas que su aplicación o la prueba prevista deben realizar, y asegúrese de que su prueba las incluya para que sea representativa de lo que ocurrirá en producción.
* Diseñe su prueba teniendo en cuenta los límites de frecuencia de la Auth0 API.
* El uso de cualquier código personalizado en Auth0 (Actions, Rules, Hooks, scripts de base de datos personalizada, conexiones <Tooltip tip="OAuth 2.0: Marco de autorización que define los protocolos y flujos de autorización." cta="Ver glosario" href="/es/docs/glossary?term=OAuth">OAuth</Tooltip> personalizadas) invocará el sandbox de código personalizado de Auth0, y esto puede suponer un mayor costo en términos de rendimiento. Desactive las Rules, salvo que sean esenciales para la prueba. Si están desactivadas, tendrá un mayor rendimiento que si están activadas.
* Estime la carga total esperada para su entorno de producción y el porcentaje de llamadas a cada endpoint, y estructure su prueba de rendimiento en consecuencia para obtener un resultado realista. Los distintos endpoints tienen diferentes costos de rendimiento. Si no diseña una prueba representativa, obtendrá resultados engañosos.
* No haga llamadas que dependan de los resultados de llamadas anteriores sin comprobar que las llamadas o respuestas previas necesarias se hayan completado. Limitarse a añadir una demora puede no ser suficiente.
* Asegúrese de implementar un manejo de errores adecuado. Una causa frecuente de problemas durante las pruebas son los errores en el código personalizado (Actions, Rules, Hooks, scripts de base de datos personalizada, scripts personalizados de conexión OAuth) provocados por excepciones no controladas en el código personalizado.
* Las pruebas de carga deben diseñarse para comenzar con un nivel bajo e incrementar la carga gradualmente, capturando datos en cada nivel, a fin de obtener los resultados más útiles. Comenzar con un nivel alto y fallar de inmediato proporciona menos información sobre lo que el sistema puede soportar.
* Es normal tener que ejecutar una prueba de rendimiento varias veces, posiblemente ajustando el código sometido a prueba o la configuración del entorno o de la propia prueba. Asegúrese de comenzar sus pruebas con antelación para disponer de tiempo suficiente para más de una iteración.
* Use su propia cuenta de proveedor de correo y asegúrese de gestionar con antelación una cuota suficiente de envío de correo, o su proveedor de correo puede aplicarle límites de frecuencia. Desactive el envío de correo si no lo utiliza.
* Asegúrese de usar las credenciales de su propia cuenta para todas las conexiones sociales en lugar de las claves de desarrollo de Auth0. En el <Tooltip tip="Auth0 Dashboard: el producto principal de Auth0 para configurar sus servicios." cta="Ver glosario" href="/es/docs/glossary?term=Auth0+dashboard">Auth0 Dashboard</Tooltip>, vaya a Connections -&gt; Social -&gt; &#123;name of connection&#125;—para ver instrucciones sobre cómo agregar a la conexión las credenciales de la cuenta de su propio proveedor social. Nota: algunos proveedores sociales no permiten pruebas de carga. Consulte la política de su(s) proveedor(es)
* Para evitar la limitación de frecuencia y simular con mayor precisión la carga real, sus pruebas deberán enviar solicitudes para distintos usuarios, no todas las solicitudes para el mismo usuario. Si utiliza solo uno o unos pocos usuarios, el almacenamiento en caché puede reducir la carga efectiva y no proporcionar resultados realistas.
* Asegúrese de mantenerse dentro de los parámetros acordados para la prueba y de la política de pruebas de carga de Auth0. Auth0 se reserva el derecho de finalizar cualquier prueba de rendimiento/carga que no se mantenga dentro de los límites de los parámetros acordados o que se extienda más allá del período programado para la prueba.
