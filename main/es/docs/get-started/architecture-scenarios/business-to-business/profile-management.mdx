validatedOn: 2026-02-03

En algún momento, puede que necesite cambiar la información almacenada en el perfil de un usuario. El perfil de un usuario (también conocido como la cuenta del usuario) se almacena en Auth0, y puede ser necesario modificar la información que contiene por diversos motivos:

* Actualizaciones de información autogestionadas
* Actualizaciones obligatorias relacionadas con los Términos y Condiciones de sus organizaciones
* Cambios debidos al cumplimiento normativo

<Warning>
  No puede acceder directamente a un perfil de usuario en varios tenants de Auth0. Si está implementando varios tenants de Auth0 en producción, esto es algo que debe tener en cuenta.
</Warning>

Un [proveedor de identidad](/es/docs/authenticate/identity-providers) completa el perfil de un usuario con los datos proporcionados durante el proceso de inicio de sesión, y esto se conoce como el [perfil de usuario normalizado](/es/docs/manage-users/user-accounts/user-profiles/normalized-user-profiles).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  El perfil de usuario normalizado se actualiza a partir del proveedor de identidad durante el inicio de sesión, y puede modificar el conjunto limitado de información que contiene mediante la Auth0 Management API. También puede usar la extensibilidad de Auth0, como [Actions](/es/docs/customize/actions), como alternativa para sobrescribir información en el perfil de usuario normalizado. Consulte [Modificación de datos del perfil de usuario](/es/docs/manage-users/user-accounts/user-profiles#user-profile-data-modification) para obtener más información.
</Callout>

De forma predeterminada, se crea un perfil de usuario para cada identidad de usuario, y hay varios aspectos que debe tener en cuenta:

* ¿Qué debería hacer si necesita almacenar información para ayudar a personalizar la experiencia de un usuario?
* ¿Qué ocurre si necesita almacenar información del usuario que no proviene de un <Tooltip tip="Identity Provider (IdP): Service that stores and manages digital identities." cta="View Glossary" href="/es/docs/glossary?term=identity+provider">proveedor de identidad</Tooltip>?
* ¿Por qué necesitaría almacenar información relacionada con el usuario que este no puede modificar?
* ¿Qué debe hacer si necesita almacenar información relacionada con el usuario que este no puede modificar?
* ¿Qué sucede si un usuario olvida su contraseña?
* ¿Qué debería hacer un usuario si quiere cambiar su contraseña?
* ¿Cómo permite que un administrador de una organización de terceros gestione a sus usuarios?

Auth0 permite almacenar metadatos en el perfil de un usuario, lo que permite recopilar información adicional, como preferencias de idioma y/o accesibilidad, para mejorar la experiencia del usuario. Los metadatos pueden usarse para almacenar tanto información que un usuario puede cambiar como información que no puede cambiar; esto último le permite asociar, por ejemplo, un perfil de usuario con registros de sus sistemas existentes sin modificar la implementación actual.

Para los usuarios que olvidan sus contraseñas o a quienes se les permite cambiarla mediante algún mecanismo de autoservicio existente (o uno que haya planificado), puede aprovechar la funcionalidad de [restablecimiento de contraseña](#password-reset) proporcionada por Auth0. Esto puede integrarse con su implementación existente y ya viene incorporado en cualquiera de los widgets de interfaz de usuario listos para usar de Auth0, incluido [Universal Login](/es/docs/authenticate/login/auth0-universal-login).

También querrá asegurarse de trabajar en todo momento con una [cuenta de usuario verificada](#account-verification). Auth0 también proporciona mecanismos listos para usar para ello. También debe tener en cuenta el [cumplimiento normativo](/es/docs/secure/data-privacy-and-compliance), como [GDPR](https://gdpr-info.eu/), que tiene requisitos muy específicos en lo que respecta a la protección de los ciudadanos de la UE frente a vulneraciones de la privacidad y filtraciones de datos.

Aunque Auth0 no proporciona actualmente un portal centralizado de gestión de perfiles listo para usar, para fines de gestión de perfiles de autoservicio, puede usar la Auth0 <Tooltip tip="Management API: A product to allow customers to perform administrative tasks." cta="View Glossary" href="/es/docs/glossary?term=Management+API">Management API</Tooltip> para crear el suyo propio o utilizar una interfaz ya creada. Consulte nuestra [guía de la comunidad](https://community.auth0.com/t/how-to-allow-the-end-user-to-update-their-own-profile-information/6228) de Auth0, que describe el endpoint de la Management API. Todas las llamadas a la Management API requerirán el uso de un [token de acceso](/es/docs/secure/tokens/access-tokens).

<Warning>
  La gestión de perfiles de autoservicio puede plantear problemas tanto de seguridad como de privacidad de los datos. Por ejemplo, puede que quiera permitir que un usuario cambie su dirección de correo electrónico; sin embargo, hacerlo sin seguir las prácticas recomendadas de seguridad podría hacer que un usuario se quede sin acceso a su cuenta, exponga información de identificación personal (PII) o, peor aún, abra una posible brecha de seguridad.
</Warning>

Como alternativa, puedes usar el <Tooltip tip="Auth0 Dashboard: el producto principal de Auth0 para configurar tus servicios." cta="Ver glosario" href="/es/docs/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip> para [administrar aspectos del perfil de un usuario](/es/docs/manage-users/user-accounts/manage-users-using-the-dashboard). Administrar el perfil de un usuario a través del Auth0 Dashboard es más bien una tarea administrativa y **no debe** usarse para la autogestión del perfil en un entorno de producción. Sin embargo, la interfaz que proporciona el Dashboard puede ser sumamente útil durante el desarrollo, ya que ofrece una forma rápida y sencilla de modificar la información del perfil de un usuario.

Si necesitas ofrecer a tus clientes una forma de que un administrador gestione sus propios usuarios cuando esas credenciales se almacenan en tu sistema, puedes crear algo tú mismo o usar una Extensión de Auth0. Consulta [Portal de administración](#admin-portal) para obtener más información.

<div id="metadata">
  ## Metadatos
</div>

Además de la información del Perfil de Usuario Normalizado, se pueden almacenar [metadatos](/es/docs/manage-users/user-accounts/metadata) en un perfil de usuario de Auth0. Los metadatos permiten almacenar información que no proviene de un proveedor de identidad o información que reemplaza la que proporciona un proveedor de identidad.

<Info>
  ### Práctica recomendada

  El uso de metadatos debe seguir las [prácticas recomendadas de Auth0 para el almacenamiento de datos de usuarios](/es/docs/secure/security-guidance/data-security/user-data-storage#metadata). El almacenamiento de metadatos no está diseñado para ser un almacén de datos de uso general, por lo que, siempre que sea posible, debes seguir utilizando tu propio sistema de almacenamiento externo. El tamaño y la complejidad de los metadatos también deben mantenerse al mínimo, y la Auth0 Management API establece directrices estrictas para actualizar y/o eliminar los metadatos asociados a un usuario.
</Info>

Puedes manipular metadatos tanto mediante la Auth0 Management API como la Auth0 Authentication API. Al igual que al gestionar el Perfil de Usuario Normalizado, las llamadas a la Management API para manipular metadatos requieren el uso de un [token de acceso](https://auth0.com/docs/api/management/v2/tokens).

<Warning>
  Las llamadas a la Management API están sujetas a la [política de limitación de tasa de Auth0](/es/docs/troubleshoot/customer-support/operational-policies/rate-limit-policy). Debes tener esto en cuenta y, para facilitarlo, Auth0 generalmente recomienda usar el [SDK de Auth0](/es/docs/libraries) adecuado para tu entorno de desarrollo en lugar de llamar directamente a nuestras API.
</Warning>

<div id="user-metadata">
  ### Metadatos del usuario
</div>

Los metadatos del usuario (también denominados `user_metadata`) son información que puede almacenarse en el perfil de un usuario y que este puede consultar y actualizar como parte de cualquier gestión de perfil de autoservicio. Este tipo de metadatos puede incluir, por ejemplo, el tratamiento de un usuario o su idioma preferido, que podría usarse para [personalizar los correos electrónicos](/es/docs/customize/email/email-templates) enviados por Auth0.

<Info>
  ### Buena práctica

  Almacena en los metadatos toda la información que quieras usar para personalizar los correos electrónicos de Auth0 y, de preferencia, en `user_metadata` si el usuario puede cambiarla, como la información utilizada para determinar el idioma de un correo electrónico.
</Info>

<div id="app-metadata">
  ### Metadatos de la aplicación
</div>

Los metadatos de la aplicación (también denominados `app_metadata`) son, por otro lado, información que puede almacenarse en el perfil de un usuario, pero que **solo puede leerse o actualizarse con la autorización adecuada**; `app_metadata` no es directamente accesible para el usuario. Este tipo de metadatos podría ser, por ejemplo, una marca que indique que el usuario aceptó la última versión válida de los términos y condiciones, así como una fecha que indique cuándo los aceptó.

<div id="password-reset">
  ## Restablecimiento de contraseña
</div>

Para los usuarios que olvidan su contraseña o que pueden cambiarla mediante algún mecanismo de autoservicio ya existente, Auth0 ofrece la funcionalidad de [restablecimiento de contraseña](/es/docs/authenticate/database-connections/password-change). Puedes integrarla con tu implementación actual, y ya está incorporada en los widgets de interfaz de usuario prediseñados de Auth0 que se incluyen como parte de [Universal Login](/es/docs/authenticate/login/auth0-universal-login).

<Warning>
  El cambio y el restablecimiento de contraseña solo son compatibles con los tipos de [Database Connection](/es/docs/authenticate/database-connections/password-change) de Auth0.
</Warning>

Auth0 <Tooltip tip="Universal Login: Tu aplicación redirige a Universal Login, alojado en el servidor de autorización de Auth0, para verificar la identidad de un usuario." cta="Ver glosario" href="/es/docs/glossary?term=Universal+Login">Universal Login</Tooltip> ofrece soporte de experiencia de usuario integrado para el restablecimiento de contraseña mediante la funcionalidad de la Auth0 Authentication API. Como alternativa, puedes usar la [Auth0 Authentication API](/es/docs/authenticate/database-connections/password-change#use-the-authentication-api) a través de uno de los SDK de Auth0 adecuados para tu entorno de desarrollo. Las plantillas de correo electrónico que se usan durante el flujo de restablecimiento de contraseña también se pueden personalizar por completo, tanto si utilizas los widgets de interfaz de usuario prediseñados de Auth0 como un Universal Login personalizado.

Por otro lado, puedes usar la Auth0 Management API para [cambiar directamente la contraseña](/es/docs/authenticate/database-connections/password-change#directly-set-the-new-password) de una identidad de usuario definida mediante un tipo de Database Connection. Puedes usar la Auth0 Management API como parte de cualquier implementación de administración de perfiles de autoservicio y también como parte de cualquier [personalización de la página de cambio de contraseña](/es/docs/get-started/architecture-scenarios/business-to-business/branding).

<div id="account-verification">
  ## Verificación de la cuenta
</div>

También tendrás que trabajar en todo momento con una cuenta de usuario verificada y utilizar los mecanismos que proporciona Auth0. También debes tener en cuenta normativas como el [GDPR](https://gdpr-info.eu/), que establece requisitos muy específicos para proteger a los ciudadanos de la UE frente a vulneraciones de la privacidad y filtraciones de datos.

Auth0 ofrece funcionalidad lista para usar para enviar un [correo electrónico de verificación](/es/docs/customize/email/manage-email-flow) a la dirección de correo electrónico de un usuario y verificar su cuenta. De forma predeterminada, Auth0 envía automáticamente correos electrónicos de verificación para cualquier identidad de [Database Connection](/es/docs/authenticate/database-connections/password-change) creada como parte del [self sign-up](/es/docs/get-started/architecture-scenarios/business-to-business/provisioning#self-sign-up). Sin embargo, Auth0 también proporciona un [endpoint de la Management API](https://auth0.com/docs/api/v2#!/Tickets/post_email_verification) que puedes utilizar para enviar correos electrónicos de verificación en los casos en que un proveedor social no valide la dirección de correo electrónico durante el registro del usuario.

<div id="blocking-users">
  ## Bloqueo de usuarios
</div>

El [bloqueo del acceso de usuarios](/es/docs/manage-users/user-accounts/block-and-unblock-users) en Auth0 permite impedir el inicio de sesión de los usuarios en las aplicaciones en determinadas circunstancias. De forma predeterminada, el Auth0 Dashboard proporciona un mecanismo integrado que permite a los administradores bloquear y desbloquear el acceso de los usuarios a todas las aplicaciones, y puedes implementar esta funcionalidad mediante la [Auth0 Management API](https://auth0.com/docs/api/management/v2#!/Users/patch_users_by_id). También puedes usar la extensibilidad de Auth0 para [deshabilitar el acceso de los usuarios a determinadas aplicaciones](/es/docs/manage-users/user-accounts/manage-user-access-to-applications), así como proporcionar un [control de acceso](/es/docs/get-started/architecture-scenarios/business-to-business/authorization) más granular.

Además, la Auth0 Management API te permite [desbloquear](https://auth0.com/docs/api/management/v2#!/User_Blocks/delete_user_blocks_by_id) a usuarios deshabilitados por el uso excesivo de credenciales incorrectas.

<div id="admin-portal">
  ## Portal de administración
</div>

Un portal de administración es una aplicación en la que puedes crear usuarios nuevos, editar el perfil de un usuario, ver la actividad de un usuario, etc. Esta aplicación solo debería ser accesible para administradores. Aunque Auth0 proporciona su Dashboard de administración, no se recomienda dar acceso al Dashboard de administración a muchas personas, ya que hay muchas formas en las que alguien podría alterar tu tenant de Auth0 sin querer. En su lugar, Auth0 ofrece otras dos opciones:

* [**Auth0 Management API**](https://auth0.com/docs/api/management/v2): Con la Management API, puedes crear fácilmente una aplicación que permita a los administradores gestionar usuarios. Puedes integrarla en una aplicación existente que ya utilicen tus administradores o crear una nueva con una interfaz de usuario acorde con tus aplicaciones actuales.
* [**Auth0 Delegated Administration Extension**](/es/docs/customize/extensions/delegated-administration-extension): Esta potente y flexible extensión te permite personalizar la experiencia de administración de usuarios. Puedes adaptarla para que los administradores de tus clientes inicien sesión y solo puedan ver y gestionar a los usuarios de su organización.

<Info>
  ### Práctica recomendada

  Si ofreces tu propio método para que un administrador gestione usuarios, solo deberías permitir que los administradores envíen a los usuarios un enlace por correo electrónico para cambiar la contraseña, en lugar de permitirles establecer contraseñas directamente. Si debes ir en contra de esta recomendación y permitir que tus administradores establezcan la contraseña de otra persona, deberías obligar al usuario a cambiar su contraseña en el siguiente inicio de sesión para que solo esa persona la conozca (y no también un administrador).
</Info>

<div id="multiple-organization-architecture-multitenancy">
  ## Arquitectura de múltiples organizaciones (multitenencia)
</div>

Muchas plataformas B2B implementan algún tipo de aislamiento y/o personalización de marca para las organizaciones de sus clientes, y esto puede añadir complejidad a cualquier sistema de gestión de identidades y accesos (IAM). Si este es tu caso, te recomendamos dedicar algo de tiempo a leer nuestra guía y nuestras recomendaciones de buenas prácticas para este tipo de entorno.

[Arquitectura de múltiples organizaciones](/es/docs/get-started/architecture-scenarios/multiple-organization-architecture)
