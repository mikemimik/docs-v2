En esta sección, veremos la solución que estamos implementando, incluidos los detalles sobre la gestión de identidades, los protocolos que se utilizarán y el flujo de autenticación necesario.

<div id="identity-management">
  ## Gestión de identidades
</div>

ExampleCo decidió usar Auth0 como proveedor de Identity as a Service (IDaaS). El motivo de esta decisión fue que la empresa no quería destinar recursos a la capacitación, implementación y mantenimiento de la gestión de identidades y accesos. Además, la empresa tiene previsto ampliar esta solución en el futuro, posiblemente añadiendo una aplicación móvil nativa y una API para enviar las hojas de horas aprobadas a sus sistemas internos. Auth0 ofrece la flexibilidad necesaria para incorporar esos cambios en su arquitectura con el mínimo esfuerzo.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Identity-as-a-Service (&quot;IDaaS&quot;) es un servicio en la nube para la gestión de identidades y accesos. Los servicios ofrecidos suelen incluir inicio de sesión único (SSO), identidad federada, gestión de contraseñas, entre otros.
</Callout>

<div id="which-protocol-to-use">
  ## Qué protocolo usar
</div>

La siguiente decisión consiste en elegir qué protocolo usar: <Tooltip tip="OAuth 2.0: Marco de autorización que define protocolos y flujos de trabajo de autorización." cta="Ver glosario" href="/es/docs/glossary?term=OAuth+2.0">OAuth 2.0</Tooltip> con <Tooltip tip="OAuth 2.0: Marco de autorización que define protocolos y flujos de trabajo de autorización." cta="Ver glosario" href="/es/docs/glossary?term=OpenID">OpenID</Tooltip> Connect (OIDC) o <Tooltip tip="Security Assertion Markup Language (SAML): Protocolo estandarizado que permite a dos partes intercambiar información de autenticación sin una contraseña." cta="Ver glosario" href="/es/docs/glossary?term=SAML">SAML</Tooltip>.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Auth0 implementa protocolos de identidad probados, comunes y populares, tanto para productos web orientados al consumidor (OAuth 2.0, OAuth 1.0, OpenID) como para implementaciones empresariales (SAML, WS-Federation, LDAP). Tienes total libertad para usar el que mejor se adapte a las necesidades de tu negocio.
</Callout>

**OpenID Connect** es un protocolo de autenticación basado en la familia de especificaciones de OAuth 2.0. Utiliza <Tooltip tip="ID Token: Credencial destinada al propio cliente, en lugar de para acceder a un recurso." cta="Ver glosario" href="/es/docs/glossary?term=identity+tokens">tokens de identidad</Tooltip> (<Tooltip tip="ID Token: Credencial destinada al propio cliente, en lugar de para acceder a un recurso." cta="Ver glosario" href="/es/docs/glossary?term=JWT">JWT</Tooltip>) en formato JSON, entregados a través del protocolo OAuth 2.0.

<Info>
  ### OAuth vs OpenID Connect (OIDC)

  OAuth 2.0 y OpenID Connect (OIDC) suelen confundirse como si fueran lo mismo, pero no es así.
  **OAuth 2.0** es un protocolo que te permite autorizar a un sitio web (el consumidor o la aplicación) a acceder a tus datos desde otro sitio web (el servidor de recursos o proveedor). Por ejemplo, quieres autorizar a un sitio web para que acceda a algunos archivos de tu cuenta de Dropbox. El sitio web te redirigirá a Dropbox, que te preguntará si debe conceder acceso a tus archivos. Si aceptas, el sitio web quedará autorizado para acceder a tus archivos de Dropbox. En esencia, OAuth 2.0 trata del acceso y el uso compartido de recursos.
  **OpenID Connect**, por otro lado, es una capa de identidad sencilla construida sobre el protocolo OAuth 2.0. Te ofrece un único inicio de sesión para varios sitios. Cada vez que necesites iniciar sesión en un sitio web mediante OIDC, se te redirige a tu sitio OpenID, donde inicias sesión, y luego vuelves al sitio web. En esencia, OIDC se centra en la autenticación del usuario.
</Info>

**SAML** es un protocolo basado en XML que proporciona tanto autenticación como autorización entre partes de confianza.

En comparación con SAML, OpenID Connect es más ligero y más sencillo de gestionar. SAML es una solución probada, potente y flexible, pero para los requisitos de esta aplicación no se necesitan ni esa flexibilidad ni esa potencia. La federación de identidades (una de las razones más convincentes para adoptar SAML) tampoco es necesaria aquí. Y si alguna vez llegara a ser un requisito, Auth0 puede gestionarlo fácilmente, de la misma forma en que lo hace con AD (que usa LDAP).

Por estas razones, ExampleCo usará OpenID Connect para su implementación.

<div id="authentication-flow">
  ## Flujo de autenticación
</div>

OpenID Connect admite más de un flujo de autenticación. Dado que nuestro caso de uso involucra una aplicación web tradicional, usaremos el Authorization Code Flow.

El flujo funciona de la siguiente manera:

1. La aplicación web (denominada Client en términos de OIDC) inicia la solicitud de autenticación redirigiendo el user-agent (navegador) a Auth0 (el Authorization Server en términos de OIDC).
2. Auth0 autentica al usuario (a través del user-agent). La primera vez que el usuario pase por este flujo, se mostrará una página de consentimiento en la que se enumeran los permisos que se concederán a la aplicación (por ejemplo, publicar mensajes o listar contactos). El usuario inicia sesión en el servicio (a menos que ya la haya iniciado) y autoriza el acceso de la aplicación.
3. Si el usuario concede acceso, Auth0 redirige el user-agent de vuelta a la aplicación junto con un código de autorización en la cadena de consulta.
4. La aplicación envía el código de autorización a Auth0, junto con las credenciales de la aplicación (`client_id` y `client_secret`), y solicita un token.
5. Auth0 autentica la aplicación (usando `client_id` y `client_secret`) y valida el código de autorización. Si es válido, Auth0 responde con un ID Token.

<Info>
  ### Modo de respuesta Form Post

  Otra opción es usar el **OAuth 2.0 Form Post Response Mode** con `response_type=id_token&response_mode=form_post`. Debido al parámetro de solicitud `response_type=id_token`, la respuesta contiene el ID Token directamente, en lugar del código de autorización, mientras que `response_mode=form_post` codifica el ID Token junto con el resto de los parámetros de la Authorization Response como valores de formulario HTML que se envían automáticamente en el User Agent. De esta manera, puede tener un flujo de autenticación optimizado (sin necesidad de intercambiar el código por un ID Token); sin embargo, debe asegurarse de que la tecnología que está usando para implementar su aplicación lo admita (el middleware de ASP .NET Core sí lo admite). Para obtener más detalles, consulte la [especificación de OAuth 2.0 Form Post Response Mode](https://openid.net/specs/oauth-v2-form-post-response-mode-1_0.html).
</Info>

El **ID Token** (normalmente denominado `id_token` en ejemplos de código) es un **JSON Web Token (JWT)** que contiene datos de identidad. Lo consume la aplicación y se usa para obtener información del usuario, como su nombre, correo electrónico, etc., normalmente para mostrarla en la interfaz de usuario.

<Info>
  ### Más sobre los tokens

  Los tokens son cadenas alfanuméricas que se usan en la autenticación basada en tokens. Permiten que los usuarios se autentiquen con un nombre de usuario y una contraseña una sola vez y, a cambio, reciban un token que pueden usar a partir de ese momento. Tienen una vigencia limitada.

  Los **JSON Web Tokens (JWTs)** son tokens que cumplen con el [estándar JSON Web Token](https://tools.ietf.org/html/rfc7519) y contienen información sobre una identidad en forma de claims. Son autocontenidos, ya que no es necesario que el destinatario llame a un servidor para validar el token. Los JWT pueden firmarse usando un secreto (con el algoritmo **HMAC**) o un par de claves pública/privada usando **RSA**. Puede encontrar más información sobre los JWT [aquí](/es/docs/secure/tokens/json-web-tokens).

  El ID Token, que es un JWT, cumple con un estándar de la industria (IETF [RFC 7519](https://tools.ietf.org/html/rfc7519)) y contiene tres partes: un encabezado, un cuerpo y una firma.

  * El encabezado contiene el tipo de token y el algoritmo hash usado en el contenido del token.
  * El cuerpo, también llamado payload, contiene claims de identidad sobre un usuario. Hay algunos claims con nombres registrados para elementos como el emisor del token, el sujeto del token (sobre quién tratan los claims) y el momento de emisión. Se puede agregar cualquier cantidad de claims adicionales con otros nombres, aunque se debe tener cuidado de mantener el JWT dentro de las limitaciones de tamaño de las URL en el navegador.
  * La firma la usa el destinatario de un JWT para validar la integridad de la información transmitida en el JWT.
</Info>

<div id="how-to-validate-an-id-token">
  ### Cómo validar un ID Token
</div>

La validación de un ID Token requiere varios pasos:

1. Si el ID Token está cifrado, descífrelo con las claves y los algoritmos que haya especificado la aplicación.
2. El identificador del emisor del OpenID Provider debe coincidir con el valor del claim `iss` (issuer).
3. El claim `aud` (audience) debe contener el valor de `client_id` de la aplicación. El ID Token debe rechazarse si no incluye a la aplicación como audiencia válida o si contiene audiencias adicionales en las que la aplicación no confía.
4. Si el ID Token contiene varias audiencias, la aplicación debe verificar que esté presente un claim `azp`.
5. Si hay un claim `azp` (authorized party), la aplicación debe verificar que su `client_id` coincida con el valor del claim.
6. La aplicación debe validar la firma de los ID Tokens de acuerdo con JWS, usando el algoritmo especificado en el parámetro de encabezado JWT `alg`. La aplicación debe usar las claves proporcionadas por el emisor.
7. El valor de `alg` debe ser el predeterminado, `RS256`, o el algoritmo enviado por la aplicación en el parámetro `id_token_signed_response_alg` durante el registro.
8. Si el parámetro de encabezado JWT `alg` usa un algoritmo basado en MAC, como `HS256`, `HS384` o `HS512`, se usan como clave para validar la firma los octetos de la representación UTF-8 del `client_secret` correspondiente al `client_id` contenido en el claim `aud` (audience). En el caso de algoritmos basados en MAC, el comportamiento no está especificado si `aud` tiene varios valores o si hay un valor `azp` distinto del valor de `aud`.
9. La hora actual debe ser anterior a la hora representada por el claim `exp`.
10. El claim `iat` puede usarse para rechazar tokens emitidos con una diferencia excesiva respecto de la hora actual, lo que limita el tiempo durante el cual es necesario almacenar nonces para prevenir ataques. El intervalo aceptable depende de cada aplicación.
11. Si se envió un valor `nonce` en la Authentication Request, debe estar presente un claim `nonce`, y debe comprobarse su valor para verificar que coincida con el que se envió en la Authentication Request. La aplicación debe comprobar el valor de `nonce` para detectar ataques de repetición. El método exacto para detectar ataques de repetición depende de cada aplicación.
12. Si se solicitó el claim `acr`, la aplicación debe comprobar que el valor del claim indicado sea adecuado.
13. Si se solicitó el claim `auth_time`, ya sea mediante una solicitud específica de este claim o mediante el parámetro `max_age`, la aplicación debe comprobar el valor del claim `auth_time` y solicitar una nueva autenticación si determina que ha transcurrido demasiado tiempo desde la última autenticación del End-User.

  Si almacena ID Tokens en su servidor, debe hacerlo de forma segura.
</Callout>
