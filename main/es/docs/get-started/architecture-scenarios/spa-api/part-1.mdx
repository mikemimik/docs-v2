Para garantizar que solo los usuarios y las aplicaciones autorizados puedan acceder a la API de Timesheets, ExampleCo ha decidido utilizar el [marco de autorización OAuth 2.0](https://tools.ietf.org/html/rfc6749). Este marco ofrece la flexibilidad que la empresa busca, ya que los distintos tipos de concesión le permiten autorizar fácilmente los diversos tipos de aplicaciones que necesitan comunicarse con la API de Timesheets.

<div id="api-authentication-and-authorization">
  ## Autenticación y autorización de API
</div>

Una API es una forma de exponer la funcionalidad de tu aplicación a otras aplicaciones. Una aplicación puede hacer una solicitud enviando un mensaje a un endpoint de una API y recibir información como respuesta.

Un endpoint de API puede estar protegido o no. En nuestro caso, como las hojas de horas son información sensible que afecta las revisiones y los pagos, es importante garantizar que solo los usuarios y las aplicaciones autorizados puedan llamar a los endpoints de nuestra API. Cuando una aplicación cliente quiere acceder a endpoints protegidos de una API, necesita presentar un <Tooltip tip="Access Token: Credencial de autorización, en forma de una cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/es/docs/glossary?term=Access+Token">token de acceso</Tooltip> como prueba de que tiene los permisos necesarios para llamar al endpoint.

Un token de acceso se obtiene autenticando al usuario con un <Tooltip tip="Authorization Server: Servidor centralizado que contribuye a definir los límites del acceso de un usuario. Por ejemplo, tu servidor de autorización puede controlar los datos, las tareas y las funciones disponibles para un usuario." cta="Ver glosario" href="/es/docs/glossary?term=Authorization+Server">servidor de autorización</Tooltip> y, a su vez, el usuario puede autorizar a la aplicación para que acceda a la API en su nombre.

<Card title="¿Qué es un token de acceso?">
  Un token de acceso (también denominado `access_token`) es una cadena opaca que representa una autorización emitida para la aplicación. Puede representar un identificador utilizado para recuperar la información de autorización o puede contener en sí mismo la información de autorización (por ejemplo, la identidad del usuario, los permisos, etc.) de forma verificable.

  Es bastante común que los tokens de acceso se implementen como [JSON Web Tokens](/es/docs/secure/tokens/json-web-tokens).

  Para obtener más información sobre los tokens de acceso de Auth0, consulta [token de acceso](/es/docs/secure/tokens/access-tokens).
</Card>

Una API puede aplicar un control detallado sobre quién puede acceder a los distintos endpoints expuestos por la API. Estos permisos se expresan como scopes.

Cuando un usuario autoriza una aplicación cliente, la aplicación también puede indicar qué permisos necesita. El usuario puede entonces revisar y conceder esos permisos. Luego, estos permisos se incluyen en el token de acceso como parte del claim `scope`.

Posteriormente, cuando el cliente envía el token de acceso al hacer solicitudes a la API, la API puede inspeccionar el claim `scope` para asegurarse de que se hayan concedido los permisos necesarios para llamar a ese endpoint específico de la API.

<Card title="¿Qué son los Scopes?">
  Cada token de acceso puede incluir una lista de los permisos que se le han concedido al cliente. Cuando un cliente se autentica con Auth0, especifica la lista de scopes (o permisos) que solicita. Si esos scopes se autorizan, el token de acceso contendrá una lista de scopes autorizados.

  Por ejemplo, la API de hojas de horas puede aceptar cuatro niveles distintos de autorización: leer hojas de horas (scope `read:timesheets`), crear hojas de horas (scope `create:timesheets`), eliminar hojas de horas (scope `delete:timesheets`) y aprobar hojas de horas (scope `approve:timesheets`).

  Cuando un cliente solicita a la API crear una nueva entrada de hoja de horas, el token de acceso debe contener el scope `create:timesheets`. De forma similar, para eliminar hojas de horas existentes, el token de acceso debe contener el scope `delete:timesheets`.

  Para obtener más información sobre scopes, consulta [Scopes](/es/docs/get-started/apis/scopes).
</Card>

Al usar el marco de autorización <Tooltip tip="OAuth 2.0: Marco de autorización que define protocolos y flujos de trabajo de autorización." cta="Ver glosario" href="/es/docs/glossary?term=OAuth+2.0">OAuth 2.0</Tooltip>, puedes dar a tus propias aplicaciones o a aplicaciones de terceros acceso limitado a tus APIs en nombre de la propia aplicación. Con Auth0, puedes admitir fácilmente distintos flujos en tus propias APIs sin preocuparte por la especificación OAuth 2.0/<Tooltip tip="OpenID: Estándar abierto de autenticación que permite a las aplicaciones verificar las identidades de los usuarios sin recopilar ni almacenar información de inicio de sesión." cta="Ver glosario" href="/es/docs/glossary?term=OpenID">OpenID</Tooltip> Connect (OIDC) ni por muchos de los demás aspectos técnicos de la autorización de API.

<Card title="Roles de OAuth">
  En cualquier flujo de OAuth 2.0 podemos identificar los siguientes roles:

  * **Propietario del recurso**: la entidad que puede conceder acceso a un recurso protegido. Normalmente, se trata del usuario final.
  * **Servidor de recursos**: el servidor que aloja los recursos protegidos. Es la API a la que quieres acceder.
  * **Cliente**: una aplicación que solicita acceso a un recurso protegido en nombre del Propietario del recurso.
  * **Servidor de autorización**: el servidor que autentica al Propietario del recurso y emite tokens de acceso tras obtener la autorización correspondiente. En este caso, la Authentication API de Auth0.

  Los [tipos de concesión (o flujos)](/es/docs/get-started/authentication-and-authorization-flow/which-oauth-2-0-flow-should-i-use) determinan cómo interactúan estos participantes para otorgar a las aplicaciones acceso limitado a las API que estás creando. Como resultado, la aplicación obtendrá un token de acceso que podrá usarse para llamar a la API en nombre del usuario.
</Card>

<div id="implicit-grant">
  ## Concesión implícita
</div>

OAuth 2.0 proporciona varios **tipos de concesión** para distintos casos de uso. En este caso concreto, queremos acceder a la API desde una [aplicación del lado del cliente](/es/docs/quickstart/spa).

La SPA usará el [flujo implícito (concesión implícita)](/es/docs/get-started/authentication-and-authorization-flow/implicit-flow-with-form-post) para hacerlo.

La concesión implícita (definida en [RFC 6749, sección 4.1](https://tools.ietf.org/html/rfc6749#section-4.2)) es similar a la concesión que se usa en el [flujo de código de autorización](/es/docs/get-started/authentication-and-authorization-flow/authorization-code-flow), pero la principal diferencia es que la aplicación recibe un token de acceso directamente, sin necesidad de un `authorization_code`. Esto ocurre porque la aplicación, que normalmente es una aplicación JavaScript que se ejecuta en un navegador, es menos confiable que una aplicación web que se ejecuta en el servidor y, por lo tanto, no se le puede confiar el `client_secret` (que es obligatorio en la concesión de código de autorización).

Una vez que el usuario se autentica, la aplicación recibe el <Tooltip tip="ID Token: Credencial destinada al propio cliente, en lugar de usarse para acceder a un recurso." cta="Ver glosario" href="/es/docs/glossary?term=ID+Token">ID Token</Tooltip> y el token de acceso en el fragmento hash del URI. La aplicación ahora puede usar el ID Token para obtener información sobre el usuario y el token de acceso para llamar a la API en nombre del usuario.

1. La aplicación inicia el flujo y redirige el navegador a Auth0 (específicamente al [endpoint /authorize](https://auth0.com/docs/api/authentication#implicit-grant)) para que el usuario pueda autenticarse.
2. Auth0 autentica al usuario. La primera vez que el usuario pasa por este flujo, y si la aplicación es de terceros, se mostrará una página de consentimiento en la que se enumeran los permisos que se otorgarán al cliente (por ejemplo, publicar mensajes, listar contactos, etc.).
3. Auth0 redirige al usuario a la aplicación con un token de acceso (y, opcionalmente, un ID Token) en el fragmento hash del URI. La aplicación ahora puede extraer los tokens de ese fragmento hash.
4. La aplicación puede usar el token de acceso para llamar a la API en nombre del usuario.

<div id="authorization-extension">
  ## Extensión de autorización
</div>

La [Extensión de autorización de Auth0](/es/docs/customize/extensions/authorization-extension) te permite configurar Roles, Grupos y Permisos, y asignárselos a los Usuarios.

* Los Permisos son acciones que alguien puede realizar. Para cubrir las necesidades de negocio de ExampleCo, configuraremos cuatro Permisos: leer, crear, eliminar y aprobar partes de horas.
* Los Roles son conjuntos de Permisos. La aplicación de partes de horas de ExampleCo la usarán dos tipos de usuarios (empleados y gerentes), cada uno con permisos distintos, por lo que configuraremos dos Roles: empleado y gerente.

Como esto cubre nuestro caso de uso de negocio, no crearemos ningún Grupo.

La Extensión de autorización creará una [Regla](/es/docs/customize/rules) que leerá los Roles, Grupos y Permisos asignados a un usuario y añadirá esta información al [perfil de usuario](/es/docs/customize/rules#rule-syntax) durante el flujo de autenticación. Podemos usar esta información para asegurarnos de que el token de acceso emitido a un usuario solo contenga los scopes permitidos. Más adelante, podremos personalizar nuestra aplicación; por ejemplo, deshabilitar la funcionalidad de aprobar partes de horas si el usuario no tiene el permiso necesario para hacerlo.
