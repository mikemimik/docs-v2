<div id="nodejs-api-implementation-spas-api">
  # Implementación de la API en Node.js (SPAs + API)
</div>

Este documento forma parte del escenario de arquitectura de SPA + API y explica cómo implementar la API en Node.js. Consulta el escenario para obtener más información sobre la solución implementada.

El código fuente completo de la implementación de la API en Node.js está disponible en [este repositorio de GitHub](https://github.com/auth0-samples/auth0-pnp-exampleco-timesheets/tree/master/timesheets-api/node).

<div id="step-1-define-the-api-endpoints">
  ## Paso 1. Definir los endpoints de la API
</div>

Usaremos el [Express web application framework](http://expressjs.com/) para desarrollar nuestra API en Node.js.

<div id="create-a-packagejson-file">
  ### Crear un archivo package.json
</div>

Crea una carpeta para tu API, accede a ella y ejecuta `npm init`. Esto configurará tu archivo `package.json`.

Puedes dejar la configuración predeterminada o cambiarla como prefieras.

El archivo `package.json` de nuestro ejemplo se ve así:

```json lines
{
  "name": "timesheets-api",
  "version": "1.0.0",
  "description": "API used to add timesheet entries for employees and contractors",
  "main": "index.js",
  "scripts": {

  },

    "type": "git",
    "url": "git+https://github.com/auth0-samples/auth0-pnp-timesheets.git"

  },
  "homepage": "https://github.com/auth0-samples/auth0-pnp-timesheets#readme"
}
```

<div id="install-the-dependencies">
  ### Instalar las dependencias
</div>

A continuación, debemos configurar las dependencias. Usaremos los siguientes módulos:

* **express**: Este módulo agrega el [framework de aplicaciones web Express](https://expressjs.com/).
* **cors**: Este módulo agrega compatibilidad para habilitar [CORS](https://en.wikipedia.org/wiki/Cross-origin_resource_sharing), lo cual es necesario, ya que se llamará a la API desde una aplicación de una sola página que se ejecuta en un dominio distinto dentro de un navegador web.
* **jwks-rsa**: Esta biblioteca recupera claves de firma RSA desde un endpoint JWKS (JSON Web Key Set). Con `expressJwtSecret`, podemos generar un proveedor de secretos que proporcionará la clave de firma correcta a `express-jwt` según el `kid` del encabezado <Tooltip tip="JSON Web Token (JWT): Formato estándar de ID Token (y, a menudo, de Access Token) que se usa para representar claims de forma segura entre dos partes." cta="Ver glosario" href="/docs/es/glossary?term=JWT">JWT</Tooltip>. Para obtener más información, consulta el [repositorio de GitHub de node-jwks-rsa](https://github.com/auth0/node-jwks-rsa).
* **express-jwt**: Este módulo te permite autenticar solicitudes HTTP con tokens JWT en tus aplicaciones de Node.js. Proporciona varias funciones que facilitan el trabajo con JWT. Para obtener más información, consulta el [repositorio de GitHub de express-jwt](https://github.com/auth0/express-jwt).
* **body-parser**: Este es un middleware de análisis del cuerpo para Node.js. Extrae todo el contenido del cuerpo de un flujo de solicitud entrante y lo expone en `req.body` de una forma más fácil de usar. Para obtener más información y ver varias alternativas, consulta el repositorio de GitHub de body-parser.

Para instalar estas dependencias, ejecuta lo siguiente:

```bash wrap lines
npm install express cors express-jwt jwks-rsa body-parser express-jwt-authz --save
```

<div id="implement-the-endpoints">
  ### Implementa los endpoints
</div>

Ve al directorio de tu API y crea un archivo `server.js`. Tu código debe:

* Instalar las dependencias.
* Implementar el/los endpoint(s).
* Iniciar el servidor de la API.

Esta es nuestra implementación de ejemplo:

```javascript lines
const express = require('express');
const app = express();
const { expressjwt: jwt } = require('express-jwt');
const jwksRsa = require('jwks-rsa');
const cors = require('cors');
const bodyParser = require('body-parser');

// Habilitar CORS
app.use(cors());

// Habilitar el uso del middleware de análisis del cuerpo de la solicitud
app.use(bodyParser.json());
app.use(bodyParser.urlencoded({
  extended: true
}));

// Crear el endpoint de la API de hojas de hora
app.post('/timesheets', function(req, res){
  res.status(201).send({message: "This is the POST /timesheets endpoint"});
})

// Iniciar el servidor de la API en localhost:8080
app.listen(8080);
```

Inicia el servidor de tu API con `node server` y realiza una solicitud HTTP POST a `localhost:8080/timesheets`. Deberías ver una respuesta JSON con el mensaje `This is the POST /timesheets endpoint`.

Ahora ya tenemos nuestro endpoint, pero cualquiera puede llamarlo. Continúa con el siguiente párrafo para ver cómo podemos solucionar esto.

<div id="step-2-secure-the-api-endpoints">
  ## Paso 2. Protege los endpoints de la API
</div>

Para validar nuestro token, usaremos la función `jwt`, proporcionada por el [middleware express-jwt](https://github.com/auth0/express-jwt#usage), y `jwks-rsa` para obtener nuestra clave secreta. Las bibliotecas hacen lo siguiente:

1. `express-jwt` decodificará el token y pasará la solicitud, el encabezado y el payload a `jwksRsa.expressJwtSecret`.
2. `jwks-rsa` descargará todas las claves de firma del endpoint JWKS y comprobará si alguna coincide con el `kid` del encabezado del JWT. Si ninguna de las claves de firma coincide con el `kid` recibido, se producirá un error. Si hay coincidencia, pasaremos la clave de firma correcta a `express-jwt`.
3. `express-jwt` continuará con su propia lógica para validar la firma del token, la expiración, `audience` y el `issuer`.

Los pasos que seguiremos en nuestro código son:

* Crear la función de middleware para validar el <Tooltip tip="Token de acceso: Credencial de autorización, en forma de una cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=Access+Token">token de acceso</Tooltip>.
* Habilitar el uso del middleware en nuestras rutas.

También puedes escribir código para guardar la hoja de hora en una base de datos. Esta es nuestra implementación de ejemplo (se omite parte del código por brevedad):

export const codeExample = `// configurar dependencias - código omitido

// Habilitar CORS - código omitido

// Crear el middleware para verificar el JWT
const checkJwt = jwt({
  // Proporcionar dinámicamente una clave de firma según el kid del encabezado y las claves de firma proporcionadas por el endpoint JWKS
  secret: jwksRsa.expressJwtSecret({
    cache: true,
    rateLimit: true,
    jwksRequestsPerMinute: 5,
    jwksUri: \`https://{yourDomain}/.well-known/jwks.json\`

Si iniciamos nuestro servidor ahora y hacemos un HTTP POST a `localhost:8080/timesheets`, deberíamos recibir el mensaje de error `Missing or invalid token` (lo cual es perfectamente normal, ya que no enviamos ningún token de acceso en la solicitud).

Para probar también el caso en que sí funciona, necesitamos:

* Obtener un token de acceso. Para más detalles sobre cómo hacerlo, consulta: [Obtener un token de acceso](/docs/es/get-started/architecture-scenarios/server-application-api#get-an-access-token).
* Invocar la API agregando a la solicitud un encabezado `Authorization` con el valor `Bearer ACCESS_TOKEN` (donde `ACCESS_TOKEN` es el valor del token que obtuvimos en el primer paso).

<div id="step-3-check-the-application-permissions">
  ## Paso 3. Comprueba los permisos de la aplicación
</div>

En este paso, añadiremos a nuestra implementación la capacidad de comprobar si la aplicación tiene permisos (o `scope`) para usar nuestro endpoint y crear una hoja de horas. En concreto, queremos asegurarnos de que el token tenga el `scope` correcto, que es `batch:upload`.

Para ello, utilizaremos el paquete de Node.js `express-jwt-authz`, así que añádelo a tu proyecto:

```bash lines
npm install express-jwt-authz --save
```

Ahora, solo tienes que añadir una llamada a `jwtAuthz(...)` a tu middleware para asegurarte de que el JWT incluya un scope determinado y así poder ejecutar un endpoint específico.

Añadiremos una dependencia adicional. La biblioteca **express-jwt-authz**, que se usa junto con express-jwt, valida el [JWT](/docs/es/secure/tokens/json-web-tokens) y se asegura de que tenga los permisos correctos para llamar al endpoint deseado. Para obtener más información, consulta el [repositorio de GitHub de express-jwt-authz](https://github.com/auth0/express-jwt-authz).

Esta es nuestra implementación de ejemplo (se omite parte del código por brevedad):

```javascript lines
// establecer dependencias - parte del código omitido
const jwtAuthz = require('express-jwt-authz');

// Habilitar CORS - código omitido

// Crear middleware para verificar el JWT - código omitido

// Habilitar el uso del middleware de análisis del cuerpo de la solicitud - código omitido

// crear el endpoint de la API de hojas de horas
app.post('/timesheets', checkJwt, jwtAuthz(['create:timesheets'], { customUserKey: 'auth' }), function(req, res){
  var timesheet = req.body;

  // Guardar la hoja de horas en la base de datos...

  //enviar la respuesta
  res.status(201).send(timesheet);
})

// iniciar el servidor de la API en localhost:8080 - código omitido
```

Si invocamos nuestra API con un token que no incluye este scope, deberíamos obtener el mensaje de error Forbidden con el código de estado HTTP `403`. Puedes comprobarlo quitando este scope de tu API.

<div id="step-4-determine-the-user-identity">
  ## Paso 4. Determinar la identidad del usuario
</div>

El middleware `express-jwt` que se utiliza para validar el JWT también establece `req.auth` con la información contenida en él. Si quieres usar el claim `sub` para identificar al usuario de forma única, simplemente puedes usar `req.auth.sub`.

Sin embargo, en el caso de la aplicación de hojas de horas, queremos usar la dirección de correo electrónico del usuario como identificador único.

Lo primero que debemos hacer es escribir una regla que añada la dirección de correo electrónico del usuario al token de acceso. Ve a la [sección Rules](https://manage.auth0.com/#/rules%7D) del Dashboard y haz clic en el botón **Create Rule**.

Puedes darle a la regla un nombre descriptivo, por ejemplo `Add email to Access Token`, y luego usar el siguiente código para la regla:

```javascript lines
function (user, context, callback) {
  const namespace = 'https://api.exampleco.com/';
  context.accessToken[namespace + 'email'] = user.email;
  callback(null, user, context);
}
```

El `namespace` se usa para garantizar que el claim tenga un nombre único y no entre en conflicto con los nombres de ninguno de los claims estándar de OIDC. Sin embargo, Auth0 admite claims personalizados con y sin namespace. Para obtener más información sobre los claims personalizados, consulta [Create Custom Claims](/docs/es/secure/tokens/json-web-tokens/create-custom-claims).

A continuación, en tu API, puedes recuperar el valor del claim desde `req.auth` y usarlo como la identidad única del usuario, que puedes asociar con las entradas de la hoja de horas.

```js lines
app.get('/timesheets', checkJwt, jwtAuthz(['read:timesheets'], { customUserKey: 'auth' }), function(req, res) {
  var timesheet = req.body;

  // Asociar la entrada de la hoja de horas con el usuario actual
  var userId = req.auth['https://api.exampleco.com/email'];
  timesheet.user_id = userId;

  // Guardar la hoja de horas en la base de datos...

  //enviar la respuesta
  res.status(201).send(timesheet);
});
```
