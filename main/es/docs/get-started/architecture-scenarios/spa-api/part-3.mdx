En esta sección veremos cómo implementar una API para nuestro caso.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Para simplificar, nuestra implementación se centrará únicamente en la autenticación y la autorización. Como verá en los ejemplos, la entrada del registro de horas estará codificada de forma fija y la API no la almacenará. En su lugar, simplemente devolverá parte de la información.
</Callout>

<div id="define-the-api-endpoints">
  ## Definir los endpoints de la API
</div>

Primero debemos definir los endpoints de nuestra API.

<Card title="¿Qué es un endpoint de API?">
  Un **endpoint de API** es una URL única que representa un objeto. Para interactuar con este objeto, debes dirigir tu aplicación a esa URL. Por ejemplo, si tuvieras una API que pudiera devolver pedidos o clientes, podrías configurar dos endpoints: `/orders` y `/customers`. Tu aplicación interactuaría con estos endpoints mediante distintos métodos HTTP; por ejemplo, `POST /orders` podría crear un nuevo pedido o `GET /orders` podría recuperar el conjunto de datos de uno o varios pedidos.
</Card>

Para esta implementación, solo definiremos 2 endpoints: uno para recuperar una lista de todos los registros de horas de un empleado y otro para permitir que un empleado cree una nueva entrada de registro de horas.

Una solicitud `HTTP GET` al endpoint `/timesheets` permitirá a un usuario recuperar sus registros de horas, y una solicitud `HTTP POST` al endpoint `/timesheets` le permitirá añadir un nuevo registro de horas.

**Consulta la implementación en** [**Node.js**](/es/docs/get-started/architecture-scenarios/spa-api/api-implementation-nodejs#1-define-the-api-endpoints).

<div id="secure-the-endpoints">
  ### Protege los endpoints
</div>

Cuando una API recibe una solicitud que incluye un <Tooltip tip="Token de acceso: Credencial de autorización, en forma de una cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/es/docs/glossary?term=Access+Token">token de acceso</Tooltip> bearer en el encabezado, lo primero que debe hacer es validar el token. Esto consiste en una serie de pasos y, si alguno falla, la solicitud debe rechazarse con el mensaje de error `Missing or invalid token` para la aplicación que realiza la llamada.

Las validaciones que debe realizar la API son:

* Comprobar que el <Tooltip tip="JSON Web Token (JWT): Formato estándar de ID Token (y, a menudo, de Access Token) utilizado para representar claims de forma segura entre dos partes." cta="Ver glosario" href="/es/docs/glossary?term=JWT">JWT</Tooltip> tenga un formato válido
* Comprobar la firma
* Validar las claims estándar

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  [JWT.io](https://jwt.io/) proporciona una lista de bibliotecas que pueden hacer la mayor parte del trabajo por ti: analizar el JWT y verificar la firma y las claims.
</Callout>

Parte del proceso de validación también consiste en comprobar los permisos de la aplicación (scopes), pero trataremos esto por separado en el siguiente párrafo de este documento.

Para obtener más información sobre cómo validar tokens de acceso, consulta [Validate Access Tokens](/es/docs/secure/tokens/access-tokens/validate-access-tokens).

**Consulta la implementación en** [**Node.js**](/es/docs/get-started/architecture-scenarios/spa-api/api-implementation-nodejs#2-secure-the-api-endpoints).

<div id="check-the-applications-permissions">
  ### Comprobar los permisos de la aplicación
</div>

A estas alturas, ya hemos comprobado que el JWT es válido. El último paso es verificar que la aplicación tenga los permisos necesarios para acceder a los recursos protegidos.

Para ello, la API debe comprobar los [scopes](/es/docs/get-started/apis/scopes) del JWT decodificado. Este claim forma parte del payload y consiste en una lista de cadenas separadas por espacios.

**Consulta la implementación en** [**Node.js**](/es/docs/get-started/architecture-scenarios/spa-api/api-implementation-nodejs#3-check-the-client-permissions).

<div id="determine-user-identity">
  ### Determinar la identidad del usuario
</div>

Para ambos endpoints (obtener la lista de registros de horas y agregar un nuevo registro de horas), tendremos que determinar la identidad del usuario.

Al obtener la lista de registros de horas, esto garantiza que solo devolvamos los registros de horas que pertenecen al usuario que realiza la solicitud. Y al agregar un nuevo registro de horas, garantiza que el registro de horas quede asociado al usuario que realiza la solicitud.

Uno de los claims estándar de JWT es el claim `sub`, que identifica a la entidad principal a la que hace referencia el claim. En el caso del flujo Implicit Grant, este claim contendrá la identidad del usuario, que será el identificador único del usuario en Auth0. Puedes usarlo para asociar cualquier información en sistemas externos a un usuario concreto.

También puedes usar un claim personalizado para agregar otro atributo del usuario, como su dirección de correo electrónico, al token de acceso y usarlo para identificar de forma única al usuario.

**Consulta la implementación en** [**Node.js**](/es/docs/get-started/architecture-scenarios/spa-api/api-implementation-nodejs#4-determine-the-user-identity).

<div id="implement-the-spa">
  ## Implementar la SPA
</div>

En esta sección veremos cómo podemos implementar una SPA para nuestro caso.

<div id="authorize-the-user">
  ### Autoriza al usuario
</div>

Para autorizar al usuario, usaremos la [biblioteca auth0.js](/es/docs/libraries/auth0js). Puedes inicializar una nueva instancia de la aplicación Auth0 de la siguiente manera:

export const codeExample = `var auth0 = new auth0.WebAuth({
  clientID: '{yourClientId}',
  domain: '{yourDomain}',
  responseType: 'token id_token',
  audience: 'YOUR_API_IDENTIFIER',
  redirectUri: '{https://yourApp/callback}',
  scope: 'openid profile read:timesheets create:timesheets'
});`;

<AuthCodeBlock children={codeExample} language="javascript" />

Debes pasar los siguientes valores de configuración:

* **clientID**: El valor de tu Auth0 <Tooltip tip="Client ID: Valor de identificación que Auth0 asigna a tu recurso registrado." cta="Ver glosario" href="/es/docs/glossary?term=Client+Id">Id. de cliente</Tooltip>. Puedes obtenerlo en la sección Settings de tu aplicación en el [Dashboard](https://manage.auth0.com/#/applications%7D).
* **domain**: El valor de tu dominio de Auth0. Puedes obtenerlo en la sección Settings de tu aplicación en el [Dashboard](https://manage.auth0.com/#/applications%7D).
* **responseType**: Indica el flujo de autenticación que se debe usar. En el caso de una SPA que usa el **flujo implícito**, debe establecerse en `token id_token`. La parte `token` hace que el flujo devuelva un token de acceso en el fragmento de la URL, mientras que la parte `id_token` hace que el flujo devuelva también un <Tooltip tip="ID Token: Credencial destinada al propio cliente, en lugar de al acceso a un recurso." cta="Ver glosario" href="/es/docs/glossary?term=ID+Token">ID Token</Tooltip>.
* **<Tooltip tip="Audience: Identificador único de la audiencia de un token emitido. Se llama aud en un token y su valor contiene el ID de una application (Client ID) para un ID Token o de una API (API Identifier) para un token de acceso." cta="Ver glosario" href="/es/docs/glossary?term=audience">audience</Tooltip>**: El valor del identificador de tu API. Puedes obtenerlo en la [sección Settings de tu API](https://manage.auth0.com/#/apis%7D) en el Dashboard.
* **redirectUri**: La URL a la que Auth0 debe redirigir una vez que el usuario se haya autenticado.
* **scope**: Los [scopes](/es/docs/get-started/apis/scopes) que determinan la información que se devolverá en el ID Token y en el token de acceso. Un scope de `openid profile` devolverá toda la información del perfil del usuario en el ID Token. También debes solicitar los scopes necesarios para llamar a la API; en este caso, los scopes `read:timesheets create:timesheets`. Esto garantizará que el token de acceso tenga esos scopes.

Para iniciar el flujo de autenticación, puedes llamar al método `authorize()`:

```js lines
auth0.authorize();
```

Después de la autenticación, Auth0 te redirigirá de nuevo al **redirectUri** que especificaste al configurar la nueva instancia de la aplicación de Auth0. En este punto, tendrás que llamar al método `parseHash()`, que analiza un fragmento hash de la URL para extraer el resultado de una respuesta de autenticación de Auth0.

El contenido del objeto authResult que devuelve parseHash depende de los parámetros de autenticación que se hayan utilizado. Puede incluir lo siguiente:

* **idToken**: un JWT de token de ID que contiene información del perfil del usuario
* **accessToken**: un token de acceso para la API, especificado por la **audience**.
* **expiresIn**: una cadena que contiene el tiempo de expiración (en segundos) del token de acceso.

Determina cuál es la mejor forma de [almacenar los tokens](/es/docs/secure/security-guidance/data-security/token-storage). Si tu aplicación de página única tiene un servidor backend, los tokens deben gestionarse del lado del servidor mediante el [Authorization Code Flow](/es/docs/get-started/authentication-and-authorization-flow/authorization-code-flow) o [Authorization Code Flow with Proof Key for Code Exchange (PKCE)](/es/docs/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce).

Si tienes una aplicación de página única (SPA) sin un servidor backend correspondiente, tu SPA debe solicitar nuevos tokens al iniciar sesión y almacenarlos en memoria sin persistirlos. Para hacer llamadas a la API, tu SPA usaría entonces la copia del token en memoria.

Para ver un ejemplo de cómo gestionar sesiones en aplicaciones SPA, consulta la sección [Handle Authentication Tokens](/es/docs/quickstart/spa/vanillajs#handle-authentication-tokens) de la [JavaScript Single-Page App Quickstart](/es/docs/quickstart/spa/vanillajs).

**Consulta cómo implementarlo en** [**Angular 2**](/es/docs/get-started/architecture-scenarios/spa-api/spa-implementation-angular2#2-authorize-the-user).

<div id="get-the-user-profile">
  ### Obtener el perfil del usuario
</div>

<Card title="Extraer información del token">
  Esta sección muestra cómo obtener la información del usuario mediante el token de acceso y el [endpoint /userinfo](https://auth0.com/docs/api/authentication#get-user-info). Para evitar esta llamada a la API, también puedes decodificar el ID Token [con una biblioteca](https://jwt.io/#libraries-io) (asegúrate de validarlo primero). Si necesitas información adicional del usuario, considera usar [nuestra Management API](https://auth0.com/docs/api/management/v2#!/Users/get_users_by_id) desde tu backend.
</Card>

Se puede llamar al método `client.userInfo` pasando el `authResult.accessToken` devuelto para obtener la información del perfil del usuario. Esto hará una solicitud al [endpoint /userinfo](https://auth0.com/docs/api/authentication#get-user-info) y devolverá el objeto `user`, que contiene la información del usuario, como en el siguiente ejemplo:

```json lines
{
    "email_verified": "false",
    "email": "test@example.com",
    "clientID": "AAAABBBBCCCCDDDDEEEEFFFFGGGGHHHH",
    "updated_at": "2017-02-07T20:50:33.563Z",
    "name": "tester9@example.com",
    "picture": "https://gravatar.com/avatar/example.png",
    "user_id": "auth0|123456789012345678901234",
    "nickname": "tester9",
    "created_at": "2017-01-20T20:06:05.008Z",
    "sub": "auth0|123456789012345678901234"
}
```

Puedes acceder a cualquiera de estas propiedades en la función de devolución de llamada que se pasa al invocar la función `userInfo`:

```javascript lines
const accessToken = authResult.accessToken;

auth0.client.userInfo(accessToken, (err, profile) => {
  if (profile) {
    // Obtener el apodo y la imagen de perfil del usuario
    var nickname = profile.nickname;
    var picture = profile.picture;
  }
});
```

**Consulta la implementación en** [**Angular 2**](/es/docs/get-started/architecture-scenarios/spa-api/spa-implementation-angular2#3-get-the-user-profile).

<div id="display-ui-elements-conditionally-based-on-scope">
  ### Mostrar elementos de la interfaz de usuario de forma condicional según el `scope`
</div>

Según el `scope` del usuario, puede que quieras mostrar u ocultar ciertos elementos de la interfaz de usuario. Para determinar el `scope` emitido para un usuario, tendrás que almacenar el `scope` que se solicitó inicialmente durante el proceso de autorización. Cuando se autoriza a un usuario, el `scope` también se devuelve en el `authResult`.

Si el `scope` en el `authResult` está vacío, significa que se concedieron todos los `scopes` solicitados. Si el `scope` en el `authResult` no está vacío, significa que se concedió un conjunto distinto de `scopes`, y deberías usar los de `authResult.scope`.

**Consulta la implementación en** [**Angular 2**](/es/docs/get-started/architecture-scenarios/spa-api/spa-implementation-angular2#4-display-ui-elements-conditionally-based-on-scope).

<div id="call-the-api">
  ### Llamar a la API
</div>

Para acceder a los recursos protegidos de tu API, el token de acceso del usuario autenticado debe incluirse en las solicitudes que se envían a la API. Esto se logra enviando el token de acceso en un encabezado `Authorization` con el esquema `Bearer`.

**Consulta la implementación en** [**Angular 2**](/es/docs/get-started/architecture-scenarios/spa-api/spa-implementation-angular2#5-call-the-api).

<div id="renew-the-access-token">
  ### Renovar el token de acceso
</div>

Como medida de seguridad, se recomienda que la vigencia del token de acceso de un usuario sea breve. Cuando creas una API en el <Tooltip tip="Auth0 Dashboard: el producto principal de Auth0 para configurar tus servicios." cta="Ver glosario" href="/es/docs/glossary?term=Auth0+dashboard">Auth0 Dashboard</Tooltip>, la duración predeterminada es `7200` segundos (2 horas), pero esto puede configurarse para cada API.

Una vez que expira, un token de acceso ya no puede usarse para acceder a una API. Para volver a obtener acceso, es necesario conseguir un nuevo token de acceso.

Para obtener un nuevo token de acceso, puedes repetir el flujo de autenticación que se usó para obtener el token de acceso inicial. En una SPA esto no es lo ideal, ya que quizá no quieras redirigir al usuario y apartarlo de su tarea actual para que vuelva a completar el flujo de autenticación.

En casos como este, puedes usar la [autenticación silenciosa](/es/docs/authenticate/login/configure-silent-authentication). La autenticación silenciosa te permite realizar un flujo de autenticación en el que Auth0 solo responde con redirecciones y nunca con una página de inicio de sesión. Sin embargo, esto requiere que el usuario ya haya iniciado sesión mediante [inicio de sesión único (SSO)](/es/docs/authenticate/single-sign-on).

**Consulta la implementación en** [**Angular 2**](/es/docs/get-started/architecture-scenarios/spa-api/spa-implementation-angular2#6-renew-the-access-token).
