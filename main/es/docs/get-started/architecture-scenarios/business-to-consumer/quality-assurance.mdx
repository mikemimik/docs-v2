---

El aseguramiento de la calidad es fundamental para identificar problemas antes de que afecten a sus clientes y, según la naturaleza de su proyecto, hay varios tipos de pruebas de aseguramiento de la calidad que conviene tener en cuenta como parte de su integración con Auth0:

* ¿Es su aplicación fácil de entender y usar, incluso para personas con discapacidad?
* ¿Debe su aplicación funcionar en distintos navegadores y dispositivos?
* ¿Debe su aplicación funcionar en entornos multinacionales y/o internacionales?
* ¿Cómo se comportará su aplicación cuando se vea sometida a cargas de producción inesperadas?
* ¿Cómo puede garantizar que su aplicación esté protegida frente a vulnerabilidades de seguridad?

Auth0 [Universal Login](/es/docs/authenticate/login/auth0-universal-login) y los widgets de interfaz de usuario asociados (como [Lock](/es/docs/libraries/lock)) ya se han diseñado y desarrollado siguiendo las mejores prácticas de usabilidad y accesibilidad, y ofrecen compatibilidad probada lista para usar con una amplia variedad de [navegadores y dispositivos](/es/docs/troubleshoot/customer-support/product-support-matrix). También se ofrece compatibilidad inmediata con la [internacionalización](/es/docs/customize/internationalization-and-localization) (I18N), con extensibilidad integrada diseñada para casos personalizados de varios idiomas y localización (L10N).

Para garantizar que se cumplan los requisitos funcionales y que los eventos inesperados se gestionen correctamente, se ofrece orientación para probar la [integración](#integration-testing) entre su(s) aplicación(es) y Auth0, así como para las [pruebas unitarias](#unit-testing) de módulos de extensibilidad individuales (como [Rules](/es/docs/customize/rules/debug-rules), [Hooks](/es/docs/customize/hooks/update-hooks) y scripts de base de datos personalizada). También se ofrece orientación sobre la [política de pruebas de penetración](/es/docs/troubleshoot/customer-support/operational-policies/penetration-testing-policy) de Auth0 para ayudar en las pruebas de vulnerabilidades de seguridad, así como sobre cómo pueden aprovecharse las pruebas [Mock](#mock-testing) junto con nuestra [política de pruebas de carga](/es/docs/troubleshoot/customer-support/operational-policies/load-testing-policy) para ayudar a garantizar que su(s) aplicación(es) rindan bien ante cargas inesperadas.

<div id="unit-testing">
  ## Pruebas unitarias
</div>

El objetivo de las pruebas unitarias es probar unidades individuales de código. Si crea código personalizado en Auth0 en forma de Rules, Hooks y/o scripts de Custom DB, debería considerar usar un framework de pruebas (como [Mocha](https://mochajs.org/)) para probar su código. Las empresas que han tenido más éxito con Auth0 han descubierto que resulta útil ejecutar estas pruebas unitarias antes de [implementar automáticamente](/es/docs/get-started/architecture-scenarios/business-to-consumer/deployment) la configuración del tenant de Auth0 y los recursos asociados.

<div id="integration-testing">
  ## Pruebas de integración
</div>

Como práctica recomendada, conviene configurar distintos tenants para desarrollo, pruebas y producción, como se explica en la guía de arquitectura sobre [compatibilidad con SDLC](/es/docs/get-started/architecture-scenarios/business-to-consumer/architecture#sdlc-support). Auth0 le permite configurar variables disponibles desde la [extensibilidad](/es/docs/customize/extensions) personalizada; pueden entenderse como variables de entorno para su tenant de Auth0. En lugar de codificar de forma fija referencias que cambian al mover código entre los entornos de desarrollo, pruebas y producción, puede usar un nombre de variable configurado en el tenant y al que haga referencia el código de extensibilidad personalizada. Esto facilita que el mismo código personalizado funcione sin cambios en distintos tenants, ya que puede hacer referencia a variables que se rellenarán con valores específicos del tenant en el momento de la ejecución:

* Para usar variables en Rules, vea cómo [configurar valores](/es/docs/customize/rules/configuration)
* Para usar variables en Hooks, vea cómo configurar [secrets](/es/docs/customize/hooks/hook-secrets) en el editor
* Para usar variables en Actions, consulte Explore Flows and Triggers
* Para usar variables en scripts de Custom DB, vea los [parámetros de configuración](/es/docs/authenticate/database-connections/custom-db/create-db-connection#step-3-add-configuration-parameters)

<Info>
  ### Buena práctica

  Se recomienda usar variables para almacenar valores específicos del tenant, así como cualquier secreto confidencial que no deba exponerse en el código personalizado. Si su código personalizado se implementa en GitHub, usar una variable específica del tenant evita exponer valores confidenciales a través de su repositorio de GitHub.
</Info>

<div id="test-automation">
  ### Automatización de pruebas
</div>

Puedes automatizar todo tu proceso de compilación incorporando tanto la automatización del despliegue como la de las pruebas. Esto puede servir para desplegar nuevas versiones de la configuración y/o del código personalizado en Auth0 y ejecutar pruebas automatizadas. Si las pruebas detectan algún fallo, las capacidades de automatización del despliegue pueden utilizarse para volver a la última versión funcional. Para obtener más información, consulta la [guía sobre automatización del despliegue](/es/docs/get-started/architecture-scenarios/business-to-consumer/deployment).

<div id="mock-testing">
  ## Pruebas con simulación
</div>

Como forma de equilibrar la [política de pruebas de carga](/es/docs/troubleshoot/customer-support/operational-policies/load-testing-policy) de Auth0 con la necesidad de realizar pruebas de carga, es habitual que los clientes de Auth0 simulen los endpoints de Auth0. Esta es una práctica muy útil para garantizar que tu aplicación funcione con las interfaces esperadas sin tener que limitar tus pruebas, y puedes apoyarte en herramientas como [MockServer](http://www.mock-server.com/), [JSON Server](https://github.com/typicode/json-server) o incluso [Postman](https://learning.getpostman.com/docs/postman/mock_servers/setting_up_mock/).
