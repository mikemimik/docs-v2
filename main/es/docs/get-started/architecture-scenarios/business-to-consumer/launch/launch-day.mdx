<div id="notifications-announcements">
  ## Notificaciones / anuncios
</div>

Para que un lanzamiento se desarrolle sin contratiempos, es importante que todas las partes interesadas estén al tanto del lanzamiento inminente y comprendan el plan de lanzamiento, así como su función y responsabilidades. Además de notificar a los equipos que participarán activamente, también puede ser útil avisar a los equipos que podrían necesitarse si algo sale mal. Tener a alguien disponible durante un lanzamiento puede ayudar a agilizar la respuesta. Asegúrate de identificar y notificar a cualquier equipo que pueda tener que responder preguntas de los clientes, incluso en redes sociales.

<div id="parties-to-notify">
  ### Partes a las que se debe notificar
</div>

* Clientes
* Socios comerciales, si corresponde
* Equipos de la aplicación afectados por la puesta en producción
* Equipos de soporte
* Equipos de red (cambios de red, en espera, en caso de problemas)
* Equipos de seguridad (en espera, en caso de problemas)
* Equipos de marketing (listos para los anuncios y para responder ante problemas)
* Equipos de redes sociales (listos para monitorear las redes sociales y responder)
* Equipos de ventas (preparados para responder preguntas de los clientes)
* Equipos de éxito del cliente (preparados para responder preguntas de los clientes)

  ## Plan de notificación
</div>

Tu plan de notificación debe incluir elementos como la <Tooltip tip="Audiencia: identificador único de la audiencia de un token emitido. Se denomina aud en un token, y su valor contiene el ID de una aplicación (ID de cliente) para un token de ID o de una API (identificador de API) para un token de acceso." cta="Ver glosario" href="/es/docs/glossary?term=audience">audiencia</Tooltip> objetivo, los puntos clave para la audiencia, el contenido del mensaje, los planes para distribuir la notificación y cómo probar los mensajes.

La lista de elementos que debes incluir en el plan es:

* Audiencia objetivo (ten en cuenta tanto las audiencias internas como las externas)
* Mensaje
* Calendario
* Dependencias
* Responsables (quién lo enviará)
* Mecanismo (cómo se comunicará)
* Mensaje de prueba y entrega (si corresponde, haz una prueba para asegurarte de que se envíen las notificaciones)

<div id="notification-distribution">
  ## Distribución de notificaciones
</div>

Una práctica habitual es publicar las notificaciones por lotes para repartir la carga inicial y limitar el alcance de la confusión si surge algún problema imprevisto. Es más fácil corregir problemas con un grupo pequeño que durante un lanzamiento a gran escala.

* Una opción es empezar con un lote de notificaciones relativamente pequeño y, si no se detectan problemas, ir aumentando el tamaño de los lotes con el tiempo.
* También puedes enviar lotes de forma escalonada en distintas partes del mundo para repartir la carga que recibe el sistema al mismo tiempo y hacer que las notificaciones lleguen en el momento óptimo dentro de cada zona horaria, lo que aumenta la probabilidad de que se lean los mensajes.
* Puedes hacer un lanzamiento gradual para una parte de los usuarios, como clientes individuales, regiones o algún otro tipo de agrupación que tenga sentido para tu aplicación.

<div id="outage-windows-if-needed">
  ## Ventanas de inactividad (si es necesario)
</div>

Algunas organizaciones requieren una solicitud formal para programar una ventana de inactividad si el lanzamiento exige alguna interrupción o tiempo de inactividad. Si tu organización lo requiere, asegúrate de determinar si será necesario tiempo de inactividad para el cutover, el lanzamiento o cualquier otro sistema dependiente, y presenta con antelación las solicitudes de interrupción o de cambio necesarias, teniendo en cuenta cualquier plazo de preaviso requerido.

<div id="cutover-plan-if-needed">
  ## Plan de cutover (si es necesario)
</div>

Algunas puestas en producción implican hacer un cutover de una solución anterior a una nueva. Si tu proyecto se ajusta a este escenario, asegúrate de identificar todo lo que debe suceder, así como cualquier dependencia, la persona responsable de cada tarea y los plazos necesarios. También puede convenirte prever reemplazos para todas las funciones clave o para cada región, por si alguien se enferma inesperadamente o no está disponible por cualquier otro motivo. Una lista de verificación de aspectos que debes considerar para el plan de cutover es:

* ¿Has documentado el plan de cutover y el plan de rollback, en caso de ser necesario?
* ¿Es necesario hacer copias de seguridad de algo antes del cambio?
* ¿Se requiere algún cambio previo en los datos?
* ¿Hay que cambiar algún registro DNS?
* ¿Se necesita algún cambio en el firewall?
* ¿Hay algún nuevo objetivo de monitoreo?
* ¿Hay que desplegar algún software?

<div id="go-no-go-criteria">
  ## Criterios para seguir adelante / no seguir adelante
</div>

En el plan general de lanzamiento, es útil contar con criterios para seguir adelante o no y haber analizado de antemano qué tipos de problemas podrían surgir, cuáles podrían resolverse sobre la marcha y cuáles requerirían revertir. Un plan de lanzamiento puede especificar intervalos de revisión periódicos, con criterios sobre qué evaluar en cada punto de control y cuánto tiempo permitir que un problema siga sin resolverse.

Para cada etapa del lanzamiento, conviene definir criterios de éxito que indiquen que el lanzamiento avanza según lo previsto y puede continuar. Algunos criterios de ejemplo podrían ser:

* Registros de usuarios en aumento y con errores mínimos
* Inicios de sesión de usuarios al ritmo esperado, con errores mínimos
* Problemas de soporte reportados por debajo de un determinado umbral
* Ningún problema identificado que pueda provocar corrupción de datos

También es útil haber identificado criterios que podrían desencadenar una decisión de “no seguir adelante” para detener el lanzamiento. La tolerancia al riesgo en cada entorno varía, pero algunos criterios de ejemplo podrían incluir:

* Un alto porcentaje de registros o inicios de sesión de usuarios que den lugar a errores que no puedan resolverse rápidamente
* Una gran cantidad de problemas de soporte que no puedan resolverse rápidamente
* Una condición identificada que podría provocar corrupción de datos
* El descubrimiento de un problema de seguridad de alta gravedad

<div id="rollback">
  ## Rollback
</div>

Siempre es prudente contar con un plan para hacer rollback o revertir una puesta en producción, por si ocurre algo imprevisto que no pueda resolverse. Revisar el plan de lanzamiento en cada paso que implique un cambio puede ayudar a identificar las tareas o modificaciones necesarias para revertir una puesta en producción o un cutover.

El plan de rollback debe incluir los pasos que se deben seguir, la secuencia, cuánto tiempo probablemente llevará cada uno y la parte responsable. Comprender el tiempo total necesario para hacer rollback puede ayudar a determinar el momento de la decisión final de seguir/no seguir, de modo que se ajuste a cualquier ventana de inactividad requerida.

Si se migra o modifica algún dato para la puesta en producción, el plan debe incluir cómo revertirlo, si fuera necesario. Revertirlo puede requerir ejecutar scripts para deshacer cambios operativos o restaurar un almacén de datos a partir de una copia de seguridad realizada antes de que comenzara el proceso de puesta en producción.

También es necesario planificar el caso en que se introduzcan datos en un sistema nuevo antes de que tenga que revertirse. ¿Será necesario abandonar esos datos / transacciones con el rollback o habrá alguna forma de capturarlos y aplicarlos en otro lugar para que no se pierdan?

Si la resolución de los problemas o el proceso para revertir pudiera tardar más de un turno, convendrá asegurarse de contar con una persona principal y quizá una secundaria disponibles y preparadas para encargarse de todo durante cada turno de trabajo. Si un problema da lugar a la necesidad de una respuesta prolongada, mucho más allá de un turno, hay límites en cuanto al tiempo que las personas pueden seguir rindiendo de forma realista sin descansar. Puede ser útil contar con recursos preparados para una respuesta a incidentes de tipo follow-the-sun, si fuera necesario.

<div id="standby-contacts">
  ## Contactos de guardia
</div>

A medida que se acerca el día del lanzamiento, conviene identificar a todos los contactos que podrían ser necesarios para diagnosticar o resolver incidencias y pedirles que estén de guardia y listos para ayudar si hace falta. La persona responsable del lanzamiento debe tener la información de contacto de cada persona de la lista de guardia para agilizar las comunicaciones.

Si hay una &quot;sala de lanzamiento&quot; física o virtual, las personas de guardia deben saber dónde está y estar preparadas para unirse si es necesario. Tener preparada una sala central o una videoconferencia puede agilizar las comunicaciones y la resolución de incidencias entre todas las partes si surge algún problema.

<div id="success-criteria">
  ## Criterios de éxito
</div>

Planificar bien un lanzamiento es clave para que tenga éxito, pero ¿sabrás cómo evaluarlo? Si defines criterios de éxito antes del lanzamiento, podrás determinar qué monitorear y si hace falta implementar monitoreo o verificaciones adicionales para evaluarlo. Por ejemplo, si uno de los elementos de los criterios de éxito es la cantidad de registros o inicios de sesión, ¿tienes una forma de monitorearlo y se ha probado para garantizar su precisión?

Te convendrá contar con estadísticas para poder destacar el éxito de tu lanzamiento. No querrás descubrir después del lanzamiento que no recopilaste datos para cuantificar todo el arduo trabajo que tu equipo dedicó a hacerlo posible.

<div id="risks-mitigations-plan">
  ## Plan de riesgos y mitigación
</div>

No es agradable pensar en las cosas que podrían salir mal, pero si ocurre algo, te alegrarás de haberlo hecho, ya que contar con un plan puede agilizar la respuesta. Algunos ejemplos para los que conviene prepararse incluyen:

* Error en una aplicación de software
* Incompatibilidad de la aplicación con la configuración del navegador del usuario
* Fallo o interrupción de la red
* Ataque DoS
* Fallo del entorno de alojamiento
* Problemas de carga o capacidad
* Problemas de datos o corrupción
* Vulnerabilidad de seguridad detectada

Si hubo un período beta, puede ser útil revisar sus resultados para identificar otros posibles escenarios de fallo.
