---

Antes del lanzamiento, debería haber completado todas las pruebas aplicables a su entorno.

El aseguramiento de la calidad es importante para identificar problemas antes de que afecten a sus clientes y, según la naturaleza de su proyecto, hay varios tipos de pruebas de calidad que le convendrá considerar como parte de su integración con Auth0:

* ¿Su aplicación es fácil de entender y usar, incluso para personas con alguna discapacidad?
* ¿Su aplicación debe funcionar en distintos navegadores y dispositivos?
* ¿Su aplicación debe funcionar en entornos multinacionales o internacionales?
* ¿Cómo responderá su aplicación ante cargas de producción inesperadas?
* ¿Cómo puede asegurarse de que su aplicación esté protegida frente a vulnerabilidades de seguridad?

Auth0 [Universal Login](/es/docs/authenticate/login/auth0-universal-login) y los widgets de interfaz de usuario asociados (como [Lock](/es/docs/libraries/lock)) ya han sido diseñados y desarrollados siguiendo las prácticas recomendadas de usabilidad y accesibilidad, y ofrecen soporte probado listo para usar para una amplia variedad de [navegadores y dispositivos](/es/docs/troubleshoot/customer-support/product-support-matrix). También se ofrece soporte listo para usar para la [internacionalización](/es/docs/customize/internationalization-and-localization) (I18N), con extensibilidad integrada diseñada para casos personalizados de varios idiomas y localización (L10N).

Para garantizar que se cumplan los requisitos funcionales y que los eventos inesperados se gestionen correctamente, se ofrece orientación para probar la [integración](#integration-testing) entre su(s) aplicación(es) y Auth0, así como para realizar [pruebas unitarias](#unit-testing) de módulos individuales de extensibilidad (como [Rules](/es/docs/customize/rules/debug-rules), [Hooks](/es/docs/customize/hooks/update-hooks) y script de Custom DB). También se proporciona orientación sobre la [política de pruebas de penetración](/es/docs/troubleshoot/customer-support/operational-policies/penetration-testing-policy) de Auth0 para ayudarle a detectar vulnerabilidades de seguridad, así como sobre cómo las pruebas con [Mock](#mock-testing) pueden aprovecharse junto con nuestra [política de pruebas de carga](/es/docs/troubleshoot/customer-support/operational-policies/load-testing-policy) para ayudar a garantizar que su(s) aplicación(es) rindan correctamente ante cargas inesperadas.

<div id="unit-testing">
  ## Pruebas unitarias
</div>

El objetivo de las pruebas unitarias es probar unidades individuales de código. Si crea código personalizado dentro de Auth0 en forma de Rules, Hooks y/o scripts de Custom DB, debería considerar utilizar un framework de pruebas (como [Mocha](https://mochajs.org/)) para probar su código. Las empresas que han tenido más éxito con Auth0 han comprobado que es útil ejecutar estas pruebas unitarias antes de [implementar automáticamente](/es/docs/get-started/architecture-scenarios/business-to-consumer/deployment) la configuración y los recursos relacionados del tenant de Auth0.

<div id="integration-testing">
  ## Pruebas de integración
</div>

Como práctica recomendada, configura distintos tenants para desarrollo, pruebas y producción, como se explica en la guía de arquitectura sobre [soporte para SDLC](/es/docs/get-started/architecture-scenarios/business-to-consumer/architecture#sdlc-support). Auth0 te permite configurar variables disponibles dentro de la [extensibilidad](/es/docs/customize/extensions) personalizada; puedes considerarlas variables de entorno para tu tenant de Auth0. En lugar de codificar de forma fija referencias que cambian al mover código entre los entornos de desarrollo, pruebas y producción, puedes usar un nombre de variable configurado en el tenant y referenciado por el código de extensibilidad personalizada. Esto facilita que el mismo código personalizado funcione, sin cambios, en distintos tenants, ya que puede hacer referencia a variables que se completarán con valores específicos de cada tenant en el momento de la ejecución:

* Para usar variables en Rules, consulta cómo [configurar valores](/es/docs/customize/rules/configuration)
* Para usar variables en Hooks, consulta cómo configurar [secrets](/es/docs/customize/hooks/hook-secrets) en el editor
* Para usar variables en Actions, consulta Explora Flows y Triggers
* Para usar variables en script de Custom DB, consulta los [parámetros de configuración](/es/docs/authenticate/database-connections/custom-db/create-db-connection#step-3-add-configuration-parameters)

<Info>
  ### Práctica recomendada

  Se recomienda usar variables para almacenar valores específicos del tenant, así como cualquier Secret sensible que no deba exponerse en tu código personalizado. Si tu código personalizado está desplegado en GitHub, usar una variable específica del tenant evita exponer valores sensibles a través de tu repositorio de GitHub.
</Info>

<div id="test-automation">
  ### Automatización de pruebas
</div>

Puede automatizar todo su proceso de compilación incorporando tanto la automatización del despliegue como la automatización de pruebas. Esto le permite desplegar nuevas versiones de la configuración y/o del código personalizado en Auth0 y ejecutar pruebas automatizadas. Si las pruebas detectan fallos, las capacidades de automatización del despliegue pueden utilizarse para revertir a la última versión que funcionaba. Para obtener más información, consulte la [guía de automatización del despliegue](/es/docs/get-started/architecture-scenarios/business-to-consumer/deployment).

<div id="mock-testing">
  ## Pruebas simuladas
</div>

Como punto medio entre la [política de pruebas de carga](/es/docs/troubleshoot/customer-support/operational-policies/load-testing-policy) de Auth0 y la necesidad de realizar una prueba de carga, es habitual que los clientes de Auth0 simulen los endpoints de Auth0. Esta práctica es útil para garantizar que su aplicación funcione con las interfaces previstas sin tener que limitar sus pruebas, y puede apoyarse en herramientas como [MockServer](http://www.mock-server.com/), [JSON Server](https://github.com/typicode/json-server) o incluso [Postman](https://learning.getpostman.com/docs/postman/mock_servers/setting_up_mock/).

<div id="pen-testing-optional">
  ## Pruebas de penetración (opcional)
</div>

Si va a realizar pruebas de penetración, debe conocer y cumplir la [política de pruebas de penetración](/es/docs/troubleshoot/customer-support/operational-policies/penetration-testing-policy) de Auth0. Las pruebas de penetración requieren notificación previa a Auth0 para que no se confundan con actividad maliciosa ni se suspendan.

<div id="load-testing-optional">
  ## Pruebas de carga (opcional)
</div>

Si va a realizar pruebas de carga, debe conocer la [política de pruebas de carga](/es/docs/troubleshoot/customer-support/operational-policies/load-testing-policy) de Auth0 y cumplirla. Las pruebas de carga requieren aviso previo a Auth0. Al planificar sus pruebas de carga, también deberá tener en cuenta los [límites de frecuencia de la API](/es/docs/troubleshoot/customer-support/operational-policies/rate-limit-policy) de Auth0.

Las pruebas de carga requieren la aprobación previa de Auth0, como se explica en la política de pruebas de carga de Auth0. Asegúrese de tener en cuenta el tiempo de antelación necesario para revisar una solicitud y de dejar suficiente margen tanto para la revisión como para la realización de las pruebas. Si su solicitud de prueba de carga ha sido aprobada, la siguiente guía puede ayudarle a evitar errores y resultados de prueba incorrectos.

* Ejecute una traza HTTP en una ejecución de prueba de su aplicación para identificar todas las llamadas que su aplicación o la prueba prevista deben realizar, y asegúrese de que su prueba las incluya para que represente lo que sucederá en Production.
* Diseñe la prueba teniendo en cuenta los límites de frecuencia de la Auth0 API.
* El uso de cualquier código personalizado en Auth0 (Actions, Rules, Hooks, scripts de base de datos personalizada, conexiones <Tooltip tip="OAuth 2.0: Marco de autorización que define protocolos y flujos de autorización." cta="Ver glosario" href="/es/docs/glossary?term=OAuth">OAuth</Tooltip> personalizadas) invocará el sandbox de código personalizado de Auth0, lo que puede suponer un mayor costo en términos de rendimiento. Desactive las Rules, a menos que sean esenciales para la prueba. Si están desactivadas, tendrá un mayor rendimiento que si están activadas.
* Estime la carga total prevista para su entorno de Production y el porcentaje de llamadas a cada endpoint, y estructure su prueba de rendimiento en consecuencia para obtener un resultado realista. Los distintos endpoints tienen distintos costos de rendimiento. No diseñar una prueba representativa dará lugar a resultados engañosos.
* No realice llamadas que dependan de los resultados de llamadas anteriores sin comprobar que las llamadas o respuestas previas necesarias se hayan completado. Simplemente añadir una demora puede no ser suficiente.
* Asegúrese de implementar un manejo de errores adecuado. Una causa frecuente de problemas durante las pruebas son los errores en código personalizado (Actions, Rules, Hooks, scripts de base de datos personalizada, scripts personalizados de conexión OAuth) provocados por excepciones no controladas en el código personalizado.
* Las pruebas de carga deben diseñarse para comenzar en un nivel bajo e incrementar la carga gradualmente, recopilando datos en cada nivel, para obtener los resultados más útiles. Comenzar en un nivel alto y fallar de inmediato proporciona menos información sobre lo que el sistema puede sostener.
* Es normal tener que ejecutar una prueba de rendimiento varias veces, posiblemente ajustando el código que se está probando o el arnés/la configuración de la prueba. Asegúrese de comenzar las pruebas con antelación para dejar tiempo suficiente para más de una iteración.
* Use su propia cuenta de proveedor de correo y asegúrese de prever con antelación una cuota de envío suficiente, o el proveedor de correo podría aplicarle límites de frecuencia. Desactive el envío de correo si no lo utiliza.
* Asegúrese de usar las credenciales de su propia cuenta para todas las conexiones sociales en lugar de las claves de desarrollo de Auth0. En el <Tooltip tip="Auth0 Dashboard: el producto principal de Auth0 para configurar sus servicios." cta="Ver glosario" href="/es/docs/glossary?term=Auth0+dashboard">Auth0 Dashboard</Tooltip>, vaya a Connections -&gt; Social -&gt; &#123;name of connection&#125;—para ver instrucciones sobre cómo agregar a la conexión las credenciales de su propia cuenta del proveedor social. Nota: algunos proveedores sociales no permiten pruebas de carga. Consulte la política de su(s) proveedor(es)
* Para evitar la limitación de frecuencia y simular con mayor precisión la carga real, sus pruebas deberán enviar solicitudes para distintos usuarios, no todas las solicitudes para el mismo usuario. Si usa solo uno o unos pocos usuarios, el almacenamiento en caché puede reducir la carga efectiva y no proporcionar resultados realistas.
* Asegúrese de mantenerse dentro de los parámetros acordados para la prueba y de la política de pruebas de carga de Auth0. Auth0 se reserva el derecho de interrumpir cualquier prueba de rendimiento/carga que no se mantenga dentro de los límites de los parámetros acordados o que se extienda más allá del período de prueba programado.
