<div id="configure-the-api">
  ## Configura la API
</div>

Para proteger una API, debes registrarla en el <Tooltip tip="Auth0 Dashboard: el principal producto de Auth0 para configurar tus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip>. Sigue las instrucciones de [Registrar API](/docs/es/get-started/auth0-overview/set-up-apis) para configurar la API de Timesheets.

<div id="json-web-token-profile">
  ### Perfil de JSON Web Token
</div>

<Tooltip tip="Token de acceso: Credencial de autorización, en forma de cadena opaca o JWT, que se usa para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=OAuth+2.0">OAuth 2.0</Tooltip> define los tipos de perfiles, o dialectos, de un <Tooltip tip="OAuth 2.0: Marco de autorización que define protocolos y flujos de autorización." cta="Ver glosario" href="/docs/es/glossary?term=access+token">token de acceso</Tooltip>. Auth0 admite dos perfiles distintos:

* Perfil de token de Auth0
* Perfil RFC 9068

El perfil predeterminado es el de Auth0. La principal diferencia entre estos dos perfiles está en los nombres de los claims. El perfil de Auth0 usa `azp` en lugar de `client_id`. Puedes obtener más información sobre estas diferencias en [Perfiles de token de acceso](/docs/es/secure/tokens/access-tokens/access-token-profiles).

<div id="signing-algorithms">
  ### Algoritmos de firma
</div>

Cuando creas una API, debes seleccionar el algoritmo con el que se firmarán tus tokens. La firma se utiliza para verificar que el emisor del <Tooltip tip="JSON Web Token (JWT): formato estándar de ID Token (y, a menudo, formato de Access Token) utilizado para representar claims de forma segura entre dos partes." cta="Ver glosario" href="/docs/es/glossary?term=JWT">JWT</Tooltip> es quien dice ser y para garantizar que el mensaje no haya sido modificado durante el trayecto.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  La firma forma parte de un JWT. Si no estás familiarizado con la estructura de un JWT, consulta [Estructura de JSON Web Token](/docs/es/secure/tokens/json-web-tokens/json-web-token-structure).
</Callout>

Para crear la firma, debes tomar el encabezado codificado, el payload codificado, un secreto y el algoritmo especificado en el encabezado, y firmarlo. Ese algoritmo, que forma parte del encabezado del JWT, es el que seleccionas para tu API: `HS256,` `RS256,` o `PS256`.

La práctica más segura, y nuestra recomendación, es usar **RS256**. Algunas de las razones son:

* Con RS256, tienes la certeza de que solo quien posee la clave privada (Auth0) puede firmar tokens, mientras que cualquiera puede comprobar si el token es válido usando la clave pública.
* Con HS256, si la clave privada se ve comprometida, tendrías que volver a desplegar la API con el nuevo secreto. Con RS256, puedes solicitar un token que sea válido para múltiples audiencias.
* Con RS256, puedes implementar la rotación de claves sin tener que volver a desplegar la API con el nuevo secreto.

<div id="configure-the-scopes">
  ## Configura los scopes
</div>

En la configuración de tu API, tendrás que configurar los scopes que las aplicaciones pueden solicitar durante la autorización. Ve a la pestaña Permissions.

En esta sección, puedes agregar los cuatro scopes mencionados anteriormente: `batch:upload`, `read:timesheets`, `create:timesheets`, `delete:timesheets` y `approve:timesheets`. Agrega también un scope adicional: `batch:upload`.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  A efectos de este documento, solo nos ocuparemos del scope `batch:upload`, porque es lo único que requiere la tarea cron. Sin embargo, para que quede completo, estamos agregando los scopes necesarios que requerirán futuras aplicaciones.
</Callout>

<div id="create-the-application">
  ## Crear la aplicación
</div>

Al crear una API en el Auth0 Dashboard, se generará automáticamente una aplicación de prueba para la API. En el Auth0 Dashboard, ve a la [sección Applications](https://manage.auth0.com/#/applications) y verás la aplicación de prueba para la API de Timesheets.

<Frame>![undefined](/docs/images/cdy7uua7fh8z/7lnuzFgs3IuWyL4bK5aVZV/05abca8035c011bf018b27e31f77a055/2024-06-28_17-13-34.png)</Frame>

Ve a la configuración de la aplicación seleccionando la opción de menú al final. Cámbiale el nombre a `Timesheets Import Job`.

Para la tarea cron, necesitarás una Machine-to-Machine Application. La aplicación de prueba que se generó al crear la API se configuró automáticamente como una Machine-to-Machine Application.

<div id="configure-applications-access-to-the-api">
  ## Configurar el acceso de la aplicación a la API
</div>

Por último, debes permitir que la aplicación acceda a la API de Timesheets. Vuelve a la configuración de la API y selecciona la pestaña Machine to Machine Application.

Verás la aplicación **Timesheets Import Job** en la lista, y debería tener acceso a la API, como se indica en el interruptor situado a la derecha del nombre de la aplicación, cuyo valor debe ser `Authorized`. Si no indica que la aplicación está autorizada, simplemente cambia el valor del interruptor de `Unauthorized` a `Authorized`.

También tendrás que especificar qué scopes se incluirán en los tokens de acceso emitidos para la aplicación cuando esta se autorice con Auth0.

Expande la configuración de la aplicación haciendo clic en la flecha hacia abajo situada en el extremo derecho, y verás la lista de scopes disponibles. La tarea cron solo requerirá el scope `batch:upload`, ya que simplemente creará nuevas hojas de horas basadas en las entradas de hojas de horas del sistema externo.

Una vez que hayas seleccionado el scope `batch:upload`, guarda la configuración haciendo clic en el botón **Update**.
