En este escenario, construiremos una API de hoja de hora para una empresa ficticia llamada ExampleCo. La API permitirá agregar entradas de hoja de hora para un empleado o un contratista.

También construiremos una aplicación de una sola página (SPA) que se utilizará para registrar entradas de hoja de hora y enviarlas a la base de datos centralizada de hojas de hora mediante la API.

<Info>
  ### En resumen

  * Auth0 proporciona autenticación y autorización de API como un medio para proteger el acceso a los endpoints de API (consulte [API Authentication and Authorization](/es/docs/get-started/architecture-scenarios/spa-api/part-1#api-authentication-and-authorization))
  * Para autorizar a un usuario de una SPA, Auth0 admite la concesión implícita (consulte [Implicit Grant](/es/docs/get-started/architecture-scenarios/spa-api/part-1#implicit-grant))
  * Tanto la SPA como la API deben configurarse en el Auth0 Dashboard (consulte [Auth0 Configuration](/es/docs/get-started/architecture-scenarios/spa-api/part-2#auth0-configuration))
  * Los permisos de usuario se pueden aplicar mediante la Authorization Extension (consulte [Configure the Authorization Extension](/es/docs/get-started/architecture-scenarios/spa-api/part-2#configure-the-authorization-extension))
  * La API estará protegida al garantizar que se envíe un token de acceso válido en el header HTTP Authorization al hacer llamadas a la API (consulte [Implement the API](/es/docs/get-started/architecture-scenarios/spa-api/part-3#implement-the-api))
  * La biblioteca Auth0.js puede utilizarse para autorizar al usuario de la SPA y obtener un token de acceso válido que pueda usarse para llamar a la API (consulte [Authorize the User](/es/docs/get-started/architecture-scenarios/spa-api/part-3#authorize-the-user))
  * La SPA puede enviar el token de acceso en el header HTTP Authorization al hacer llamadas a la API (consulte [Call the API](/es/docs/get-started/architecture-scenarios/spa-api/part-3#call-the-api))
  * La SPA puede mostrar elementos de la interfaz de usuario de forma condicional según los scopes concedidos al usuario (consulte [Display UI Elements Conditionally Based on Scope](/es/docs/get-started/architecture-scenarios/spa-api/part-3#display-ui-elements-conditionally-based-on-scope))
</Info>

<div id="the-premise">
  ## La premisa
</div>

ExampleCo es una startup de consultoría. Actualmente, cuenta con aproximadamente 100 empleados y también externaliza varias actividades a contratistas externos. Todos los empleados y contratistas externos deben completar sus hojas de hora cada semana.

La empresa ha desarrollado una aplicación de hojas de hora, un escenario que analizamos en [Inicio de sesión único para aplicaciones web tradicionales](/es/docs/get-started/architecture-scenarios/sso-for-regular-web-apps). Los empleados internos usan esta aplicación web para completar sus hojas de hora, pero la empresa quiere reemplazarla por una SPA. La aplicación se utilizará para registrar entradas en las hojas de hora y enviar los datos a la base de datos centralizada de hojas de hora mediante la API. La aplicación también permitirá que los gerentes aprueben las entradas de las hojas de hora.

<div id="goals-requirements">
  ## Objetivos y requisitos
</div>

ExampleCo quiere desarrollar una solución flexible. Por el momento, solo se necesita una SPA para registrar entradas de hoja de hora, pero en el futuro la empresa tiene previsto lanzar más aplicaciones, como una aplicación móvil para dar soporte a sus equipos de ventas. Por eso, la empresa ha decidido desarrollar una única API de hoja de hora que se utilizará para registrar el tiempo no solo mediante este proceso del servidor, sino también desde todas las aplicaciones futuras. Quieren implementar una arquitectura de seguridad lo bastante flexible como para dar cabida a esto. ExampleCo quiere asegurarse de que gran parte del código y de la lógica de negocio de la aplicación pueda compartirse entre las distintas aplicaciones.

Se requiere que solo los usuarios y las aplicaciones autorizados puedan acceder a la API de hoja de hora.

Esta SPA será utilizada por dos tipos de usuarios: empleados y gerentes. Los empleados deben poder leer, crear y eliminar sus propias entradas de hoja de hora, mientras que los gerentes también deben poder aprobar hojas de hora.
