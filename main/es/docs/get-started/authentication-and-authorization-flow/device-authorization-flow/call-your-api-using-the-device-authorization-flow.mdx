<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Este tutorial te ayudará a llamar a tu propia API desde un dispositivo con opciones de entrada limitadas mediante el flujo de autorización de dispositivos. Si quieres saber cómo funciona este flujo y por qué deberías usarlo, consulta [flujo de autorización de dispositivos](/es/docs/get-started/authentication-and-authorization-flow/device-authorization-flow).
</Callout>

Auth0 facilita que tu aplicación implemente el Device <Tooltip tip="Flujo de autorización: concesión de autorización (o workflow) especificada en el marco de OAuth 2.0." cta="Ver glosario" href="/es/docs/glossary?term=Authorization+flow">Authorization flow</Tooltip> mediante lo siguiente:

* [Authentication API](https://auth0.com/docs/api/authentication): Sigue leyendo para aprender a llamar directamente a nuestra API. Si prefieres una experiencia interactiva, consulta [Device Flow Playground](https://auth0.github.io/device-flow-playground/).

<div id="prerequisites">
  ## Requisitos previos
</div>

Antes de comenzar este tutorial:

* Revisa las limitaciones (a continuación) para asegurarte de que el flujo de autorización de dispositivos sea adecuado para tu implementación.
* [Registra la aplicación en Auth0](/es/docs/get-started/auth0-overview/create-applications/native-apps).

  * Selecciona **Native** como **Tipo de aplicación**.
  * Si es necesario, configura **Allowed Web Origins**. Puedes usar esta opción para permitir localhost como origen durante el desarrollo local o para establecer un origen permitido para software de TV específico con una arquitectura sujeta a CORS (por ejemplo, HTML5 + JS). La mayoría de las aplicaciones no usarán esta configuración.
  * Asegúrate de que el interruptor **OIDC Conformant** esté habilitado. Esta configuración se encuentra en el [Dashboard](https://manage.auth0.com/#), en **Applications &gt; Application &gt; Advanced Settings &gt; OAuth**.
  * Asegúrate de que los **Tipos de concesión** de la aplicación incluyan **Device Code**. Para saber cómo hacerlo, consulta [Actualizar tipos de concesión](/es/docs/get-started/applications/update-grant-types).
  * Si quieres que tu aplicación pueda usar tokens de actualización, asegúrate de que los **Tipos de concesión** de la aplicación incluyan **token de actualización**. Para saber cómo hacerlo, consulta [Actualizar tipos de concesión](/es/docs/get-started/applications/update-grant-types). Para obtener más información sobre los tokens de actualización, consulta [Tokens de actualización](/es/docs/secure/tokens/refresh-tokens).
* Configura y habilita al menos una conexión para la aplicación: [Conexiones de base de datos](/es/docs/get-started/applications/set-up-database-connections), [Conexiones sociales](/es/docs/authenticate/identity-providers/social-identity-providers)
* [Registra tu API en Auth0](/es/docs/get-started/architecture-scenarios/mobile-api/part-2#create-the-api)

  * Si quieres que tu API reciba tokens de actualización para poder obtener nuevos tokens cuando caduquen los anteriores, habilita **Allow Offline Access**. Para obtener más información sobre los tokens de actualización, consulta [Tokens de actualización](/es/docs/secure/tokens/refresh-tokens).
* [Configura los ajustes del código de usuario del dispositivo](/es/docs/get-started/tenant-settings/configure-device-user-code-settings) para definir el conjunto de caracteres, el formato y la longitud del código de usuario generado aleatoriamente.

<div id="steps">
  ## Pasos
</div>

1. [Solicitar código de dispositivo](#request-device-code) (Flujo de dispositivo): Solicita un código de dispositivo que el usuario pueda usar para autorizar el dispositivo.
2. [Solicitar activación del dispositivo](#request-device-activation) (Flujo de dispositivo): Solicita que el usuario autorice el dispositivo desde su portátil o smartphone.
3. [Solicitar tokens](#request-tokens) (Flujo de dispositivo): Consulta el endpoint de token para solicitar un token.
4. [Autorizar al usuario](#authorize-user) (Flujo del navegador): El usuario autoriza el dispositivo para que este pueda recibir tokens.
5. [Recibir tokens](#receive-tokens) (Flujo de dispositivo): Después de que el usuario autorice correctamente el dispositivo, recibe los tokens.
6. [Llamar a la API](#call-your-api) (Flujo de dispositivo): Usa el token de acceso obtenido para llamar a tu API.
7. [Renovar tokens](#refresh-tokens) (Flujo de dispositivo): Usa un token de actualización para solicitar nuevos tokens cuando caduquen los actuales.

Opcional: [Ver casos de uso de ejemplo](#sample-use-cases).

Opcional: [Solucionar problemas](#troubleshoot).

<div id="request-device-code">
  ### Solicitar un código de dispositivo
</div>

Una vez que el usuario haya iniciado la aplicación en su dispositivo y quiera autorizar el dispositivo, tendrás que obtener un código de dispositivo. Cuando el usuario inicie su sesión en el dispositivo con navegador, este código quedará vinculado a esa sesión.

Para obtener el código de dispositivo, tu aplicación debe solicitar un código en la [URL de código de dispositivo](https://auth0.com/docs/api/authentication#get-device-code), incluido el <Tooltip tip="ID de cliente: Valor de identificación que Auth0 proporciona a tu recurso registrado." cta="Ver glosario" href="/es/docs/glossary?term=Client+ID">ID de cliente</Tooltip>.

<div id="example-post-to-device-code-url">
  #### Ejemplo de solicitud POST a la URL del código de dispositivo
</div>

<AuthCodeGroup>
  ```bash cURL
  curl --request POST \
    --url 'https://{yourDomain}/oauth/device/code' \
    --header 'content-type: application/x-www-form-urlencoded' \
    --data 'client_id={yourClientId}' \
    --data 'scope={scope}' \
    --data 'audience={audience}'
  ```

  ```csharp C#
  var client = new RestClient("https://{yourDomain}/oauth/device/code");
  var request = new RestRequest(Method.POST);
  request.AddHeader("content-type", "application/x-www-form-urlencoded");
  request.AddParameter("application/x-www-form-urlencoded", "client_id={yourClientId}&scope=%7Bscope%7D&audience=%7Baudience%7D", ParameterType.RequestBody);
  IRestResponse response = client.Execute(request);
  ```

  ```go Go
  package main

  import (
  	"fmt"
  	"strings"
  	"net/http"
  	"io/ioutil"
  )

  func main() {

  	url := "https://{yourDomain}/oauth/device/code"

  	payload := strings.NewReader("client_id={yourClientId}&scope=%7Bscope%7D&audience=%7Baudience%7D")

  	req, _ := http.NewRequest("POST", url, payload)

  	req.Header.Add("content-type", "application/x-www-form-urlencoded")

  	res, _ := http.DefaultClient.Do(req)

  	defer res.Body.Close()
  	body, _ := ioutil.ReadAll(res.Body)

  	fmt.Println(res)
  	fmt.Println(string(body))

  }
  ```

  ```java Java
  HttpResponse response = Unirest.post("https://{yourDomain}/oauth/device/code")
    .header("content-type", "application/x-www-form-urlencoded")
    .body("client_id={yourClientId}&scope=%7Bscope%7D&audience=%7Baudience%7D")
    .asString();
  ```

  ```javascript Node.JS
  var axios = require("axios").default;

  var options = {
    method: 'POST',
    url: 'https://{yourDomain}/oauth/device/code',
    headers: {'content-type': 'application/x-www-form-urlencoded'},
    data: {client_id: '{yourClientId}', scope: '{scope}', audience: '{audience}'}
  };

  axios.request(options).then(function (response) {
    console.log(response.data);
  }).catch(function (error) {
    console.error(error);
  });
  ```

  ```php PHP
  $curl = curl_init();

  curl_setopt_array($curl, [
    CURLOPT_URL => "https://{yourDomain}/oauth/device/code",
    CURLOPT_RETURNTRANSFER => true,
    CURLOPT_ENCODING => "",
    CURLOPT_MAXREDIRS => 10,
    CURLOPT_TIMEOUT => 30,
    CURLOPT_HTTP_VERSION => CURL_HTTP_VERSION_1_1,
    CURLOPT_CUSTOMREQUEST => "POST",
    CURLOPT_POSTFIELDS => "client_id={yourClientId}&scope=%7Bscope%7D&audience=%7Baudience%7D",
    CURLOPT_HTTPHEADER => [
      "content-type: application/x-www-form-urlencoded"
    ],
  ]);

  $response = curl_exec($curl);
  $err = curl_error($curl);

  curl_close($curl);

  if ($err) {
    echo "cURL Error #:" . $err;
  } else {
    echo $response;
  }
  ```

  ```python Python

  conn = http.client.HTTPSConnection("")

  payload = "client_id={yourClientId}&scope=%7Bscope%7D&audience=%7Baudience%7D"

  headers = { 'content-type': "application/x-www-form-urlencoded" }

  ```

  ```ruby Ruby
  response = http.request(request)
  puts response.read_body
  ```

<div id="device-code-parameters">
  ##### Parámetros del device code
</div>

Ten en cuenta que, al solicitar un device code para llamar a una API personalizada:

* debes incluir un parámetro <Tooltip tip="Audience: Identificador único de la audience de un token emitido. En un token se denomina aud; su valor contiene el ID de una aplicación (ID de cliente) para un token de ID o de una API (identificador de API) para un token de acceso." cta="Ver glosario" href="/es/docs/glossary?term=audience">audience</Tooltip>
* puedes incluir scopes adicionales admitidos por la API de destino

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Si tu aplicación quiere un token de acceso solo para obtener información sobre el usuario autenticado, no se requiere ningún parámetro audience.
</Callout>

| Nombre del parámetro | Descripción                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                             |
| -------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `client_id`          | El ID de cliente de tu aplicación. Puedes encontrar este valor en [configuración de la aplicación](https://manage.auth0.com/#/Applications/\{yourClientId}/settings).                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                             |
| `scope`              | Los [scopes](/es/docs/get-started/apis/scopes) para los que quieres solicitar autorización. Deben estar separados por espacios. Puedes solicitar cualquiera de los [scopes estándar de OIDC](https://openid.net/specs/openid-connect-core-1_0.html#StandardClaims) sobre usuarios, como `profile` y `email`, [custom claims](/es/docs/secure/tokens/json-web-tokens/json-web-token-claims#custom-claims) que sigan un [formato con espacio de nombres](/es/docs/secure/tokens/json-web-tokens/create-custom-claims), o cualquier [scope admitido por la API de destino](/es/docs/get-started/apis/scopes/api-scopes) (por ejemplo, `read:contacts`). Incluye `openid` para obtener un token de ID o para poder usar el [endpoint /userinfo](https://auth0.com/docs/api/authentication#user-profile) para recuperar la información de perfil del usuario. Incluye `offline_access` para obtener un token de actualización (asegúrate de que el campo **Allow Offline Access** esté habilitado en [API Settings](https://manage.auth0.com/#/apis)). Ten en cuenta que debe estar codificado como URL. |
| `audience`           | El identificador único de la API a la que tu aplicación quiere acceder. Usa el valor **Identifier** en la pestaña [Settings](https://manage.auth0.com/#/apis) de la API que creaste como parte de los requisitos previos de este tutorial. Ten en cuenta que debe estar codificado como URL.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                            |

<div id="device-code-response">
  #### Respuesta de código de dispositivo
</div>

Si todo va bien, recibirás una respuesta `HTTP 200` con un payload que contiene los valores `device_code`, `user_code`, `verification_uri`, `expires_in`, `interval` y `verification_uri_complete`:

```json lines
{
  "device_code": "Ag_EE...ko1p",
  "user_code": "QTZL-MCBW",
  "verification_uri": "https://accounts.acmetest.org/activate",
  "verification_uri_complete": "https://accounts.acmetest.org/activate?user_code=QTZL-MCBW",
  "expires_in": 900,
  "interval": 5
}
```

* `device_code` es el código único del dispositivo. Cuando el usuario vaya a la `verification_uri` desde un dispositivo con navegador, este código se asociará a su sesión.
* `user_code` contiene el código que debe introducirse en la `verification_uri` para autorizar el dispositivo.
* `verification_uri` contiene la URL que el usuario debe visitar para autorizar el dispositivo.
* `verification_uri_complete` contiene la URL completa que el usuario debe visitar para autorizar el dispositivo. Esto permite que tu aplicación incluya el `user_code` en la URL, si así lo deseas.
* `expires_in` indica el tiempo de validez (en segundos) del `device_code` y del `user_code`.
* `interval` indica el intervalo (en segundos) con el que la aplicación debe consultar la URL del token para solicitar un token.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Puedes [configurar el conjunto de caracteres, el formato y la longitud de tu código de usuario generado aleatoriamente](/es/docs/get-started/tenant-settings/configure-device-user-code-settings) en la Configuración del tenant.

  Para evitar ataques de fuerza bruta, aplicamos los siguientes límites a `user_code`:

  **Longitud mínima**:

  * Letras BASE20: 8 caracteres
  * Números: 9 caracteres

  **Longitud máxima**:

  * 20 caracteres (incluidos los guiones y espacios, que pueden añadirse como separadores para facilitar la lectura)

  **Tiempo de expiración**:

  * 15 minutos
</Callout>

<div id="request-device-activation">
  ### Solicitar la activación del dispositivo
</div>

Una vez que hayas recibido un `device_code` y un `user_code`, debes pedirle al usuario que vaya a `verification_uri` desde su portátil o smartphone e introduzca el `user_code`:

<Frame>![Auth0 Flows Solicitud de autorización del dispositivo, página de ejemplo que muestra dos métodos de activación: user\_code y código QR](/docs/images/cdy7uua7fh8z/2WzaeNXIYCVduRuzyRd0Sb/cdb4d59b657166d0a9a555a662b9ed63/request-device-activation.png)</Frame>

El `device_code` no está pensado para que lo use directamente el usuario y no debe mostrarse durante la interacción para evitar confusiones.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Al crear una CLI, puedes omitir este paso y abrir de inmediato el navegador con `verification_uri_complete`.
</Callout>

<div id="request-tokens">
  ### Solicitar tokens
</div>

Mientras esperas a que el usuario active el dispositivo, comienza a hacer polling en la URL del token para solicitar un <Tooltip tip="Token de acceso: credencial de autorización, en forma de una cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/es/docs/glossary?term=Access+Token">token de acceso</Tooltip>. Con el intervalo de polling (`interval`) obtenido en el paso anterior, tendrás que hacer un `POST` a la [URL del token](https://auth0.com/docs/api/authentication#device-auth) enviando el `device_code`.

Para evitar errores debidos a la latencia de la red, debes empezar a contar cada intervalo después de recibir la respuesta de la última solicitud de polling.

<div id="token-request-parameters">
  ##### Parámetros de la solicitud de token
</div>

| Nombre del parámetro | Descripción                                                                                                                                                                                                                                                                          |
| -------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `grant_type`         | Establece este valor en &quot;urn:ietf:params:oauth:grant-type:device&#95;code&quot;. Este es un tipo de concesión de extensión (según se define en la Sección 4.5 de [RFC6749](https://tools.ietf.org/html/rfc6749#section-4.5)). Ten en cuenta que debe estar codificado como URL. |
| `device_code`        | El `device_code` obtenido en el paso anterior de este tutorial.                                                                                                                                                                                                                      |
| `client_id`          | El ID de cliente de tu aplicación. Puedes encontrar este valor en [Configuración de la aplicación](https://manage.auth0.com/#/Applications/\{yourClientId}/settings).                                                                                                                |

<div id="token-responses">
  #### Respuestas de token
</div>

Mientras esperas a que el usuario autorice el dispositivo, puede que recibas distintas respuestas `HTTP 4xx`:

<div id="authorization-pending">
  ##### Autorización pendiente
</div>

Verás este error mientras esperas a que el usuario realice una acción. Sigue consultando periódicamente con el intervalo sugerido obtenido en el paso anterior de este tutorial.

```json lines
HTTP/1.1 403 Forbidden
{
  "error": "authorization_pending",
  "error_description": "..."
}
```

<div id="slow-down">
  ##### Ve más despacio
</div>

Estás haciendo sondeos demasiado rápido. Ve más despacio y usa el intervalo sugerido que obtuviste en el paso anterior de este tutorial. Para evitar este error debido a la latencia de la red, debes empezar a contar cada intervalo después de recibir la respuesta de la última solicitud de sondeo.

```json lines
HTTP/1.1 429 Too Many Requests
{
  "error": "slow_down",
  "error_description": "..."
}
```

<div id="expired-token">
  ##### Token caducado
</div>

El usuario no ha autorizado el dispositivo con la suficiente rapidez, por lo que el `device_code` ha caducado. Tu aplicación debe notificarle que el flujo ha caducado e indicarle que lo reinicie.

<Warning>
  El error `expired_token` se devolverá exactamente una vez; después de eso, se devolverá `invalid_grant`. Tu dispositivo **debe** dejar de consultar.
</Warning>

```json lines
HTTP/1.1 403 Bad Request
{ 
  "error": "expired_token",
  "error_description": "..."
}
```

<div id="access-denied">
  ##### Acceso denegado
</div>

Por último, si se te deniega el acceso, recibirás:

```json lines
HTTP/1.1 403 Forbidden
{
  "error": "access_denied",
  "error_description": "..."
}
```

Esto puede ocurrir por varias razones, entre ellas:

* el usuario se negó a autorizar el dispositivo
* el <Tooltip tip="Servidor de autorización: Servidor centralizado que contribuye a definir los límites del acceso de un usuario. Por ejemplo, su servidor de autorización puede controlar los datos, las tareas y las funciones disponibles para un usuario." cta="Ver glosario" href="/es/docs/glossary?term=authorization+server">servidor de autorización</Tooltip> denegó la transacción
* una rule configurada denegó el acceso (Para obtener más información, lea [Auth0 Rules](/es/docs/customize/rules).)

<div id="authorize-user">

<div id="receive-tokens">
  ### Recibir tokens
</div>

Mientras el usuario se autentica y autoriza el dispositivo, la aplicación del dispositivo ha seguido consultando la URL del token para solicitar un token de acceso.

Una vez que el usuario haya autorizado correctamente el dispositivo, recibirás una respuesta `HTTP 200` con un payload que contiene los valores `access_token`, `refresh_token` (opcionalmente), `id_token` (opcionalmente), `token_type` y `expires_in`:

```json lines
{
  "access_token":"eyJz93a...k4laUWw",
  "refresh_token":"GEbRxBN...edjnXbL",
  "id_token": "eyJ0XAi...4faeEoQ",
  "token_type":"Bearer",
  "expires_in":86400
}
```

<Warning>
  Valida tus tokens antes de guardarlos. Para saber cómo hacerlo, consulta [Validar tokens de ID](/es/docs/secure/tokens/id-tokens/validate-id-tokens) y [Validar tokens de acceso](/es/docs/secure/tokens/access-tokens/validate-access-tokens).
</Warning>

Los tokens de acceso se usan para llamar al endpoint [`/userinfo`](https://auth0.com/docs/api/authentication#get-user-info) de la Auth0 Authentication API o a otra API. (Para obtener más información sobre los tokens de acceso, consulta [Tokens de acceso](/es/docs/secure/tokens/access-tokens).) Solo podrás usar el token de acceso para llamar a `/userinfo` si incluiste el scope `openid`. Si llamas a tu propia API, lo primero que deberá hacer es [verificar el token de acceso](/es/docs/secure/tokens/access-tokens/validate-access-tokens).

<Tooltip tip="token de ID: Credencial destinada al propio cliente, en lugar de usarse para acceder a un recurso." cta="Ver glosario" href="/es/docs/glossary?term=ID+Tokens">Tokens de ID</Tooltip> contienen información del usuario que debe decodificarse y extraerse. (Para obtener más información sobre los tokens de ID, consulta [Tokens de ID](/es/docs/secure/tokens/id-tokens).) El `id_token` solo estará presente en la respuesta si incluiste el scope `openid`.

<Tooltip tip="token de actualización: Token usado para obtener un token de acceso renovado sin obligar a los usuarios a iniciar sesión de nuevo." cta="Ver glosario" href="/es/docs/glossary?term=Refresh+Tokens">Tokens de actualización</Tooltip> se usan para obtener un nuevo token de acceso o token de ID una vez que el anterior haya expirado. (Para obtener más información sobre los tokens de actualización, consulta [Tokens de actualización](/es/docs/secure/tokens/refresh-tokens).) El `refresh_token` solo estará presente en la respuesta si incluiste el scope `offline_access` y habilitaste **Allow Offline Access** para tu API en el Dashboard.

<Warning>
  Los tokens de actualización deben almacenarse de forma segura, ya que permiten que un usuario permanezca autenticado prácticamente indefinidamente.
</Warning>

<div id="refresh-tokens">
  ### Tokens de actualización
</div>

Ya has recibido un token de actualización si has seguido este tutorial y has completado lo siguiente:

* configuraste tu API para permitir el acceso sin conexión
* incluiste el scope `offline_access` cuando iniciaste la solicitud de autenticación a través del [endpoint de autorización](https://auth0.com/docs/api/authentication/reference#authorize-application)

Puedes usar el token de actualización para obtener un nuevo token de acceso. Por lo general, un usuario solo necesitará un nuevo token de acceso después de que el anterior caduque o cuando obtenga acceso a un nuevo recurso por primera vez. No es una buena práctica llamar al endpoint para obtener un nuevo token de acceso cada vez que llamas a una API, y Auth0 aplica límites de frecuencia que restringen la cantidad de solicitudes al endpoint que pueden ejecutarse con el mismo token desde la misma IP.

Para actualizar tu token, haz una solicitud `POST` al endpoint `/oauth/token` de la Authentication API, usando `grant_type=refresh_token`.

<div id="refresh-token-request-parameters">
  ##### Parámetros de la solicitud de token de actualización
</div>

| Nombre del parámetro | Descripción                                                                                                                                                                                                                               |
| -------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `grant_type`         | Establézcalo en &quot;refresh&#95;token&quot;.                                                                                                                                                                                            |
| `client_id`          | El ID de cliente de su aplicación. Puede encontrar este valor en la [Configuración de la aplicación](https://manage.auth0.com/#/Applications/\{yourClientId}/settings).                                                                   |
| `client_secret`      | El secreto de cliente de su aplicación. Puede encontrar este valor en la [Configuración de la aplicación](https://manage.auth0.com/#/Applications/\{yourClientSecret}/settings).                                                          |
| `refresh_token`      | El token de actualización que se usará.                                                                                                                                                                                                   |
| `scope`              | (Opcional) Una lista de scopes solicitados, separados por espacios. Si no se envía, se usarán los scopes originales; de lo contrario, puede solicitar un conjunto reducido de scopes. Tenga en cuenta que debe estar codificado como URL. |

```json lines
  "token_type": "Bearer"

```

<div id="limitations">
  ### Limitaciones
</div>

Para usar el flujo de autorización de dispositivos, los dispositivos deben:

* Admitir Server Name Indication (SNI) cuando se usan [Custom Domains](/es/docs/customize/custom-domains)
* Tener un [tipo de aplicación de Auth0](/es/docs/get-started/applications) **Native**
* Tener el [Token Endpoint Authentication Method](/es/docs/get-started/applications/application-settings) configurado como **None**
* Ser [compatibles con OIDC](/es/docs/get-started/applications/application-settings)
* No haberse creado mediante [registro dinámico de clientes](/es/docs/get-started/applications/dynamic-client-registration)

Además, el flujo de autorización de dispositivos no permite:

* [Social Connections](/es/docs/authenticate/identity-providers/social-identity-providers) que usen [developer keys de Auth0](/es/docs/authenticate/identity-providers/social-identity-providers/devkeys), a menos que estés usando la [experiencia de Universal Login](/es/docs/authenticate/login/auth0-universal-login/universal-login-vs-classic-login/universal-experience)
* Acceder a parámetros de la cadena de consulta desde la página de inicio de sesión alojada, Rules o Actions
* [La vinculación de cuentas de usuario](/es/docs/manage-users/user-accounts/user-account-linking)

Admitimos el Draft 15 completo, excepto <Tooltip tip="Cliente confidencial: un cliente (aplicación) que puede almacenar credenciales de forma segura mediante el uso de un servidor backend de confianza. Algunos ejemplos son una aplicación web con un backend seguro y una aplicación de máquina a máquina (M2M)." cta="Ver glosario" href="/es/docs/glossary?term=confidential+Clients">Clientes confidenciales</Tooltip>. Para obtener más información, consulta [OAuth 2.0 Device Authorization Grant Draft 15 en ietf.org](https://tools.ietf.org/html/draft-ietf-oauth-device-flow-15).
