[Pushed Authorization Request (PAR)](https://datatracker.ietf.org/doc/html/rfc9126) es un protocolo de backend para enviar solicitudes de autorización directamente al <Tooltip tip="Servidor de autorización: Servidor centralizado que ayuda a definir los límites del acceso de un usuario. Por ejemplo, tu servidor de autorización puede controlar los datos, las tareas y las funciones disponibles para un usuario." cta="Ver glosario" href="/es/docs/glossary?term=authorization+server">servidor de autorización</Tooltip>. Es un componente técnico de [Financial-Grade API (FAPI) Security Profile 1.0](https://openid.net/specs/openid-financial-api-part-2-1_0.html) diseñado para proteger las API en escenarios de alto valor.

<div id="how-it-works">
  ## Cómo funciona
</div>

PAR permite que tu aplicación envíe directamente al endpoint PAR del servidor de autorización los parámetros de las solicitudes de autorización de <Tooltip tip="OAuth 2.0: Marco de autorización que define protocolos y flujos de trabajo de autorización." cta="Ver glosario" href="/es/docs/glossary?term=OAuth+2.0">OAuth 2.0</Tooltip> **(1)**. Como respuesta, el servidor de autorización envía un valor URI de solicitud, `request_uri` **(2)**, para usarlo cuando llames al endpoint `/authorize` **(3)**. `request_uri` es una referencia a las solicitudes de autorización almacenadas en el endpoint `/par`, por lo que esas solicitudes no quedan expuestas **(4)**. Para obtener más información, consulta [Configurar solicitudes de autorización push](/es/docs/get-started/applications/configure-par).

<div id="benefits">
  ## Beneficios
</div>

Uno de los beneficios de usar PAR es la validación anticipada. En otros flujos de OAuth 2.0, como el [flujo de código de autorización](/es/docs/get-started/authentication-and-authorization-flow/authorization-code-flow), los usuarios finales son redirigidos al servidor de autorización para su validación. En PAR, los parámetros de la solicitud se validan al inicio de la solicitud de autorización, antes de redirigir al usuario final. No es recomendable redirigir a los usuarios solo para mostrarles una página de error.

PAR también envía las solicitudes de autorización por el back-channel. Las comunicaciones por front-channel dependen de un intermediario (p. ej., un navegador) mediante parámetros de consulta HTTPS añadidos (GET, POST). Los mensajes no se envían directamente. Las comunicaciones por back-channel se transmiten en el cuerpo de una solicitud autenticada de backend, lo que ofrece un enfoque más directo.

Las solicitudes de autorización enviadas mediante push viajan a través del back-channel, lo que significa que:

* El servidor de autorización puede confiar en el origen de la solicitud, y que las solicitudes no han sido modificadas por un usuario final.
* Los detalles de la solicitud no quedan expuestos en la barra ni en el historial del navegador, por lo que la privacidad se preserva en ese punto del proceso.
* Las restricciones de longitud de la URL no suponen una limitación.

<div id="limitations">
  ## Limitaciones
</div>

* El tamaño máximo del payload de la solicitud es de 10 KB.
* Actualmente no se admiten las aplicaciones públicas. Para obtener más información, consulta [Public and Confidential Applications](/es/docs/get-started/applications/confidential-and-public-applications).

<div id="call-the-par-endpoint">
  ## Llamar al endpoint PAR
</div>

<div id="requirements">
  ### Requisitos
</div>

Para llamar al endpoint PAR, debes:

* Establecer el tipo de contenido de la solicitud como `application/x-www-form-urlencoded`.
* Usar cadenas para todos los parámetros que se envíen.
* Incluir en la solicitud un parámetro adicional para el método de autenticación de la aplicación. Solo los <Tooltip tip="Cliente confidencial: un cliente (aplicación) que puede almacenar credenciales de forma segura mediante un servidor backend de confianza. Algunos ejemplos son una aplicación web con un backend seguro y una aplicación de máquina a máquina (M2M)." cta="Ver glosario" href="/es/docs/glossary?term=confidential+clients">clientes confidenciales</Tooltip> admiten PAR, por lo que están disponibles los siguientes [métodos de autenticación de la aplicación](https://auth0.com/docs/api/authentication#authentication-methods): <Tooltip tip="Secreto de cliente: Secret que utiliza un cliente (aplicación) para autenticarse con el servidor de autorización; solo deben conocerlo el cliente y el servidor de autorización, y debe ser lo suficientemente aleatorio como para que no pueda adivinarse." cta="Ver glosario" href="/es/docs/glossary?term=Client+Secret">secreto de cliente</Tooltip>, clave privada <Tooltip tip="Secreto de cliente: Secret que utiliza un cliente (aplicación) para autenticarse con el servidor de autorización; solo deben conocerlo el cliente y el servidor de autorización, y debe ser lo suficientemente aleatorio como para que no pueda adivinarse." cta="Ver glosario" href="/es/docs/glossary?term=JWT">JWT</Tooltip> y mTLS. Debes usar el mismo método de autenticación de la aplicación para el endpoint `/token` al recuperar un <Tooltip tip="Token de acceso: credencial de autorización, en forma de una cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/es/docs/glossary?term=access+token">token de acceso</Tooltip>.

<div id="supported-parameters">
  ### Parámetros admitidos
</div>

El endpoint PAR solo almacena y procesa:

* Los parámetros estándar de OAuth 2.0 y las extensiones aplicables que reconocemos en el endpoint de autorización.
* Hasta 10 parámetros de autorización personalizados con el prefijo `ext-`.

PAR ignora los parámetros de autorización personalizados adicionales. Los parámetros de autorización personalizados no están disponibles en [Auth0 Actions](/es/docs/customize/actions) ni en [Logs](/es/docs/deploy-monitor/logs).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Si usa parámetros de autorización personalizados en Actions, debe anteponerles el prefijo `ext-`. De lo contrario, no estarán disponibles.
</Callout>

<div id="example-par-request">
  ### Ejemplo de solicitud PAR
</div>

```bash lines
curl --location --request POST https://$tenant/oauth/par \
  -H "content-type: application/x-www-form-urlencoded" \
  -d "client_id=CLIENT_ID"\
"&client_secret=CLIENT_SECRET"\
"&redirect_uri=https://jwt.io"\
"&audience=urn:my-notes-api"\
"&scope=openid%20profile%20read:notes"\
"&response_type=code"
```

<div id="example-par-response">
  ### Ejemplo de respuesta PAR
</div>

En el siguiente ejemplo de respuesta PAR:

* `request_uri` es una referencia a las solicitudes de autorización almacenadas. Los valores de la solicitud se envían al endpoint GET `/authorize` como parámetro `request_uri`.
* `expires_in` es la cantidad de segundos durante los cuales `request_uri` es válido. Transcurrido ese plazo, `request_uri` caduca si no se utiliza. El tiempo de caducidad de treinta segundos es un valor estático y no se puede configurar.

```json lines
HTTP/1.1 201 Created
 Content-Type: application/json

 {
  "request_uri":
    "urn:ietf:params:oauth:request_uri:6esc_11ACC5bwc014ltc14eY22c",
  "expires_in": 30
 }
```

<div id="rate-limits">
  ### Límites de frecuencia
</div>

Para los tenants de producción de Essential, Professional y Enterprise, las llamadas al endpoint PAR se incluyen en el límite de frecuencia estándar de la Authentication API. Para obtener más información, consulta [Configuraciones de límites de tasa](/es/docs/troubleshoot/customer-support/operational-policies/rate-limit-policy/rate-limit-configurations) y haz clic en tu tipo de suscripción. Luego, haz clic en **Authentication API**.

<div id="call-the-authorization-endpoint">
  ## Llama al endpoint de autorización
</div>

Tu aplicación usa el valor `request_uri` que devuelve el endpoint `/oauth/par` en la solicitud de autorización y redirige el agente de usuario al endpoint de autorización. Para obtener más información sobre el parámetro `request_uri`, consulta [Configure Push Authorization Requests](/es/docs/get-started/applications/configure-par).

El siguiente ejemplo hace que el agente de usuario realice la siguiente solicitud HTTP:

```http wrap lines
GET /authorize?client_id=CLIENT_ID&request_uri=urn%3Aietf%3Aparam...qrwSI HTTP/1.1 Host: TENANT.auth0.com
```

En el caso de un `request_uri` válido, el resto del <Tooltip tip="Flujo de autorización: concesión de autorización (o flujo de trabajo) especificado en el marco de OAuth 2.0." cta="Ver glosario" href="/es/docs/glossary?term=authorization+flow">flujo de autorización</Tooltip> es igual.

<div id="validation">
  ### Validación
</div>

* El PAR vuelve a ser validado por el servidor de autorización en esta etapa, como cualquier otra solicitud de autorización.
* El valor de `request_uri` solo puede usarse una vez.
* El servidor de autorización rechazará cualquier `request_uri` caducado.
* Se rechazará una solicitud que no sea PAR si PAR es obligatorio, ya sea a nivel de tenant o de cliente.
