<Warning>
  La fecha de fin de vida útil (EOL) de Rules y Hooks será el **18 de noviembre de 2026** y ya no están disponibles para nuevos tenants creados a partir del **16 de octubre de 2023**. Los tenants existentes con Hooks activos conservarán el acceso al producto Hooks hasta su fin de vida útil.

  Recomendamos encarecidamente que utilice Actions para extender Auth0. Con Actions, tiene acceso a información de tipos detallada, documentación integrada y paquetes públicos de `npm`, y puede conectar integraciones externas que mejoran su experiencia general de extensibilidad. Para obtener más información sobre lo que ofrece Actions, lea [Understand How Auth0 Actions Work](/docs/es/customize/actions/actions-overview).

  Para ayudarle con su migración, ofrecemos guías que le ayudarán a [migrar de Rules a Actions](/docs/es/customize/actions/migrate/migrate-from-rules-to-actions) y [migrar de Hooks a Actions](/docs/es/customize/actions/migrate/migrate-from-hooks-to-actions). También tenemos una página dedicada de [Move to Actions](https://auth0.com/extensibility/movetoactions) que destaca comparativas de funcionalidades, [una demostración de Actions](https://www.youtube.com/watch?v=UesFSY1klrI) y otros recursos para ayudarle en su proceso de migración.

  Para obtener más información sobre la obsolescencia de Rules y Hooks, lea nuestra entrada de blog: [Preparing for Rules and Hooks End of Life](https://auth0.com/blog/preparing-for-rules-and-hooks-end-of-life/).
</Warning>

En el punto de extensibilidad Intercambio de credenciales de cliente, Hooks le permite ejecutar acciones personalizadas cuando se emite un <Tooltip tip="Access Token: Credencial de autorización, en forma de una cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=Access+Token">Access Token</Tooltip> a través del endpoint [`POST /oauth/token` de la Authentication API](https://auth0.com/docs/api/authentication#client-credentials-flow) mediante el Client Credentials Flow. Por ejemplo, puede impedir que se emita el token, agregar claims personalizadas al token de acceso o modificar sus scopes. Para obtener más información, lea [Client Credentials Flow](/docs/es/get-started/authentication-and-authorization-flow/client-credentials-flow).

Los Hooks en este punto de extensibilidad son bloqueantes (sincrónicos), lo que significa que se ejecutan como parte del proceso del trigger e impiden que el resto del pipeline de Auth0 se ejecute hasta que el Hook se complete.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  El `triggerId` para el punto de extensibilidad Intercambio de credenciales de cliente es `credentials-exchange`. Para aprender a crear hooks para este punto de extensibilidad, lea [Create Hooks](/docs/es/customize/hooks/create-hooks).
</Callout>

Para conocer otros puntos de extensibilidad, consulte Extensibility Points.

<div id="starter-code-and-parameters">
  ## Código de inicio y parámetros
</div>

Al crear un Hook que se ejecuta en el punto de extensibilidad Intercambio de credenciales de cliente, el siguiente código de inicio puede resultar útil. Los parámetros que pueden pasarse a la función del Hook y usarse en ella se enumeran al comienzo del ejemplo de código.

```javascript lines expandable
/**
@param {object} client - información del cliente
@param {string} client.name - nombre del cliente
@param {string} client.id - ID de cliente
@param {string} client.tenant - nombre del tenant de Auth0
@param {object} client.metadata - metadatos del cliente
@param {array|undefined} scope - un arreglo de strings que representa el claim de scope del token, o undefined
@param {string} audience - claim de audience del token
@param {object} context - información de contexto de Auth0
@param {object} context.webtask - contexto del Hook (Webtask)
@param {function} cb - function (error, accessTokenClaims)
*/

module.exports = function(client, scope, audience, context, cb) {
  var access_token = {};
  access_token.scope = scope; // no eliminar esta línea

  // Modificar los scopes o agregar claims adicionales
  // access_token['https://example.com/claim'] = 'bar';
  // access_token.scope.push('extra');

  // Denegar el token y responder con una respuesta de error de OAuth2
  // if (denyExchange) {
  //   // Para devolver un HTTP 400 con { "error": "invalid_scope", "error_description": "Not authorized for this scope." }
  //   return cb(new InvalidScopeError('Not authorized for this scope.'));
  //
  //   // Para devolver un HTTP 400 con { "error": "invalid_request", "error_description": "Not a valid request." }
  //   return cb(new InvalidRequestError('Not a valid request.'));
  //
  //   // Para devolver un HTTP 500 con { "error": "server_error", "error_description": "A server error occurred." }
  //   return cb(new ServerError('A server error occurred.'));
  // }

  cb(null, access_token);
};
```

Ten en cuenta:

* La función de devolución de llamada (`cb`) al final del código de ejemplo indica que la ejecución ha finalizado y debe incluirse.

* La línea `access_token.scope = scope` garantiza que todos los scopes concedidos estén presentes en el token de acceso. Si la eliminas, se reiniciarán todos los scopes y el token incluirá solo los scopes que agregues con el script.

<div id="default-response">
  ### Respuesta predeterminada
</div>

Cuando ejecutas un Hook en el punto de extensibilidad de Intercambio de credenciales de cliente, el objeto de respuesta predeterminado es:

```json lines
{
  "scope": "array of strings"
}
```

<div id="starter-code-response">
  ### Respuesta del código de inicio
</div>

Una vez que hayas personalizado el código de inicio con tus scopes y claims adicionales, puedes probar el Hook con el ejecutor integrado en el Hook Editor. El ejecutor simula una llamada al Hook con el mismo cuerpo y el mismo objeto de respuesta que obtendrías con un Intercambio de credenciales de cliente.

<Warning>
  Ejecutar el código con el ejecutor requiere guardar los cambios, lo que significa que el código original se sobrescribirá.
</Warning>

Cuando ejecutas un Hook basado en el código de inicio, el objeto de respuesta es:

```json lines
{
  "audience": "https://my-tenant.auth0.com/api/v2/",
  "client": {
    "id": "xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx",
    "name": "client-name",
    "tenant": "my-tenant",
    "metadata": {
      "plan": "full"
    }
  },
  "scope": [
    "read:connections"
  ]
}
```

<div id="sample-script-add-an-additional-scope-to-the-access-token">
  ## Script de ejemplo: Agregar un scope adicional al token de acceso
</div>

En este ejemplo, usamos un Hook para agregar un scope adicional a los que ya tiene el token de acceso.

```js lines
module.exports = function(client, scope, audience, context, cb) {
    // Scopes a agregar
    var access_token = {};

    // Obtener el scope que está actualmente en el Access Token
    // y agregarlo al objeto con el que estamos trabajando
    // ¡No elimines esta línea!
    access_token.scope = scope;

    // Agregar el scope `read:resource`
    access_token.scope.push('read:resource');

    // Devolución de llamada para indicar la finalización y retornar el nuevo
    // arreglo de scopes
    cb(null, access_token);
};
```

Para obtener más información, lee [Scopes](/docs/es/get-started/apis/scopes).

<div id="response">
  ### Respuesta
</div>

Cuando ejecutamos este Hook, el objeto de respuesta es el siguiente:

```json lines
{
  "scope": [
    "read:connections",
    "read:resource"
  ]
}
```

<div id="sample-script-add-a-claim-to-the-access-token">
  ## Script de ejemplo: Añadir un claim al token de acceso
</div>

En este ejemplo, añadimos un custom claim con espacio de nombres y su valor al token de acceso. Para obtener más información, consulta [Create Namespaced Custom Claims](/docs/es/secure/tokens/json-web-tokens/create-custom-claims).

Puedes añadir lo siguiente como claims al token emitido:

* La propiedad `scope` del objeto de respuesta
* Cualquier propiedad con un nombre de propiedad con espacio de nombres

El punto de extensibilidad ignora todas las demás propiedades del objeto de respuesta.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Para acceder a un Hook Secret configurado desde un hook, usa `context.webtask.secrets.SECRET_NAME`.
</Callout>

```js lines
module.exports = function(client, scope, audience, context, cb) {
    // Claims a agregar
    var access_token = {};

    // Nuevo claim a agregar al token
    access_token['https://example.com/foo'] = 'bar';

    // Devolución de llamada para indicar la finalización y devolver el nuevo claim
    cb(null, access_token);
  };
```

<div id="response">
  ### Respuesta
</div>

Cuando ejecutamos este Hook, el objeto de respuesta es:

```json lines
{
  "https://example.com/foo": "bar"
}
```

<div id="sample-script-raise-an-error-or-deny-an-access-token">
  ## Script de ejemplo: Generar un error o denegar un token de acceso
</div>

En este ejemplo, usamos objetos `Error` personalizados para generar respuestas de error de OAuth2. (Para obtener más información, consulta [OAuth2 RFC - Sección 5.2 en el IETF Datatracker](https://tools.ietf.org/html/rfc6749#section-5.2).)

Si se devuelve un error simple de Javascript en la devolución de llamada, como:

```js lines
module.exports = function(client, scope, audience, context, cb) {
    // Devolución de llamada para indicar la finalización y devolver un nuevo claim
    cb(new Error("Unknown error occurred.");
  };
```

Entonces, cuando solicites una concesión `client_credentials` al endpoint `/oauth/token`, Auth0 responderá con:

```json lines
HTTP 500
{ "error": "server_error", "error_description": "Unknown error occurred." }
```

Sin embargo, si desea tener un mayor control sobre la respuesta de error de OAuth2, puede usar en su lugar tres objetos Error personalizados.

<div id="invalidscopeerror">
  ### InvalidScopeError
</div>

```js lines
module.exports = function(client, scope, audience, context, cb) {
    const invalidScope = ...; // determinar si el scope es válido

    if(invalidScope) {
      cb(new InvalidScopeError("Scope is not permitted."));
    }
  };
```

Luego, al solicitar una concesión de `client_credentials` en el endpoint `/oauth/token`, Auth0 responde con:

```json lines
HTTP 400
{ "error": "invalid_scope", "error_description": "Scope is not permitted." }
```

<div id="invalidrequesterror">
  ### InvalidRequestError
</div>

```js lines
module.exports = function(client, scope, audience, context, cb) {
    const invalidRequest = ...; // determinar si la solicitud es válida

    if(invalidRequest) {
      cb(new InvalidRequestError("Bad request."));
    }
  };
```

Entonces, cuando solicites una concesión `client_credentials` al endpoint `/oauth/token`, Auth0 responderá con:

```json lines
HTTP 400
{ "error": "invalid_request", "error_description": "Bad request." }
```

<div id="servererror">
  ### ServerError
</div>

```js lines
module.exports = function(client, scope, audience, context, cb) {
    callOtherService(function(err, response) {
      if(err) {
        return cb(new ServerError("Error calling remote system: " + err.message));
      }
    });
  };
```

Luego, cuando solicitas una concesión `client_credentials` al endpoint `/oauth/token`, Auth0 responde con:

```json lines
HTTP 400
{ "error": "server_error", "error_description": "Error calling remote system: ..." }
```

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Actualmente, el comportamiento de la clase `Error` incluida en JavaScript y de `ServerError` es idéntico, pero la clase `ServerError` permite especificar explícitamente qué error de OAuth2 se devolverá.
</Callout>
