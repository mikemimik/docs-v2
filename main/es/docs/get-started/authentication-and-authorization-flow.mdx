title: Flujos de autenticación y autorización
---

Auth0 usa el [protocolo OpenID Connect (OIDC)](/docs/es/authenticate/protocols/openid-connect-protocol) y el [marco de autorización de OAuth 2.0](/docs/es/authenticate/protocols/oauth) para autenticar a los usuarios y obtener su autorización para acceder a recursos protegidos. Con Auth0, puede admitir fácilmente distintos flujos en sus propias aplicaciones y API sin preocuparse por las especificaciones de OIDC/<Tooltip tip="OAuth 2.0: Marco de autorización que define protocolos y flujos de trabajo de autorización." cta="Ver glosario" href="/docs/es/glossary?term=OAuth+2.0">OAuth 2.0</Tooltip> ni por otros aspectos técnicos de la [autenticación y autorización](/docs/es/get-started/identity-fundamentals/authentication-and-authorization).

Admitimos escenarios para aplicaciones del lado del servidor, móviles, de escritorio, del lado del cliente, de máquina a máquina y de dispositivos.

Si no está seguro de qué flujo usar, podemos ayudarle a decidir. Para obtener más información, lea [¿Qué flujo de OAuth 2.0 debo usar?](/docs/es/get-started/authentication-and-authorization-flow/which-oauth-2-0-flow-should-i-use).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Durante la ejecución de varios flujos, su aplicación también debe autenticarse ante el servidor de autorización. Para obtener más información sobre la autenticación de aplicaciones, lea [Credenciales de la aplicación](/docs/es/secure/application-credentials).
</Callout>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Según las políticas de acceso de aplicaciones a la API que configure para una API, es posible que deba crear las autorizaciones de cliente correspondientes para su aplicación. Para obtener más información, lea [Acceso de aplicaciones a las API: políticas y autorizaciones de cliente](/docs/es/get-started/applications/application-access-to-apis-client-grants).
</Callout>

<div id="authorization-code-flow">
  ## Flujo de código de autorización
</div>

Como las aplicaciones web tradicionales son aplicaciones del lado del servidor, donde el código fuente no está expuesto públicamente, pueden usar el flujo de código de autorización, que intercambia un código de autorización por un token.

* [Flujo de código de autorización](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow)
* [Agregar inicio de sesión mediante el flujo de código de autorización](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow/add-login-auth-code-flow)
* [Llamar a la API mediante el flujo de código de autorización](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow/call-your-api-using-the-authorization-code-flow)

<div id="authorization-code-flow-with-proof-key-for-code-exchange-pkce">
  ## Flujo de código de autorización con PKCE
</div>

Durante la autenticación, las aplicaciones móviles y nativas pueden usar el flujo de código de autorización, pero requieren seguridad adicional. Además, las aplicaciones web de página única plantean desafíos particulares. Para mitigarlos, OAuth 2.0 proporciona una versión del flujo de código de autorización que utiliza PKCE.

* [Flujo de código de autorización con PKCE](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce)
* [Agregar inicio de sesión con el flujo de código de autorización con PKCE](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce/add-login-using-the-authorization-code-flow-with-pkce)
* [Llamar a la API con el flujo de código de autorización con PKCE](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce/call-your-api-using-the-authorization-code-flow-with-pkce)

<div id="authorization-code-flow-with-enhanced-privacy-protection">
  ## Flujo de código de autorización con protección reforzada de la privacidad
</div>

Durante el proceso de autenticación y autorización, algunos casos de uso, como la [autorización transaccional](/docs/es/secure/highly-regulated-identity/transactional-authorization-with-authorization-code-flow), intercambian información contextual que puede contener datos sensibles. Para proteger los datos y la información sensible, puedes usar distintas mejoras del protocolo para el flujo de código de autorización:

* [Flujo de código de autorización con Rich Authorization Requests (RAR)](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow/authorization-code-flow-with-rar)
* [Flujo de código de autorización con solicitudes de autorización push (PAR)](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow/authorization-code-flow-with-par)
* [Flujo de código de autorización con solicitudes de autorización protegidas con JWT (JAR)](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow/authorization-code-flow-with-jar)
* [JSON Web Encryption (JWE)](/docs/es/secure/tokens/access-tokens/json-web-encryption)

<div id="implicit-flow-with-form-post">
  ## Flujo implícito con Form Post
</div>

Como alternativa al flujo de código de autorización, OAuth 2.0 ofrece el flujo implícito, pensado para <Tooltip tip="Cliente público: Cliente (aplicación) que no puede mantener las credenciales de forma segura. Algunos ejemplos incluyen una aplicación nativa de escritorio o móvil y una aplicación web del lado del cliente basada en JavaScript (como una aplicación web de página única (SPA))." cta="Ver glosario" href="/docs/es/glossary?term=Public+Clients">clientes públicos</Tooltip> o aplicaciones que no pueden almacenar de forma segura <Tooltip tip="Cliente público: Cliente (aplicación) que no puede mantener las credenciales de forma segura. Algunos ejemplos incluyen una aplicación nativa de escritorio o móvil y una aplicación web del lado del cliente basada en JavaScript (como una aplicación web de página única (SPA))." cta="Ver glosario" href="/docs/es/glossary?term=Client+Secrets">secretos de cliente</Tooltip>. Aunque esto ya no se considera una práctica recomendada para solicitar <Tooltip tip="Token de acceso: Credencial de autorización, en forma de una cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=Access+Tokens">tokens de acceso</Tooltip>, al usarse con el modo de respuesta Form Post ofrece un flujo de trabajo más ágil si la aplicación solo necesita un <Tooltip tip="Token de ID: Credencial destinada al propio cliente, en lugar de usarse para acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=ID+token">token de identidad</Tooltip> para autenticar al usuario.

* [Flujo implícito con Form Post](/docs/es/get-started/authentication-and-authorization-flow/implicit-flow-with-form-post)
* [Agregar inicio de sesión mediante el flujo implícito con Form Post](/docs/es/get-started/authentication-and-authorization-flow/implicit-flow-with-form-post/add-login-using-the-implicit-flow-with-form-post)
* [Autenticar aplicaciones web de página única con cookies](/docs/es/manage-users/cookies/spa-authenticate-with-cookies)

<div id="hybrid-flow">
  ## Flujo híbrido
</div>

Las aplicaciones que pueden almacenar secretos de cliente de forma segura pueden beneficiarse del uso del flujo híbrido, que combina características del flujo de código de autorización y del flujo implícito con Form Post para permitir que tu aplicación tenga acceso inmediato a un token de identidad, sin dejar de permitir la obtención segura de tokens de acceso y <Tooltip tip="Token de actualización: token que se usa para obtener un token de acceso renovado sin obligar a los usuarios a volver a iniciar sesión." cta="Ver glosario" href="/docs/es/glossary?term=refresh+tokens">tokens de actualización</Tooltip>. Esto puede ser útil en situaciones en las que tu aplicación necesita acceder de inmediato a información sobre el usuario, pero debe realizar cierto procesamiento antes de obtener acceso a recursos protegidos durante un período prolongado.

* [Flujo híbrido](/docs/es/get-started/authentication-and-authorization-flow/hybrid-flow)
* [Llamar a la API mediante el flujo híbrido](/docs/es/get-started/authentication-and-authorization-flow/hybrid-flow/call-api-hybrid-flow)

<div id="client-credentials-flow">
  ## Client Credentials Flow
</div>

Con las aplicaciones de máquina a máquina (M2M), como CLIs, daemons o servicios que se ejecutan en tu back-end, el sistema autentica y autoriza la aplicación en lugar de a un usuario. En este caso, los esquemas de autenticación habituales, como identificador + contraseña o los inicios de sesión con redes sociales, no tienen sentido. En su lugar, las aplicaciones M2M usan el Client Credentials Flow (definido en OAuth 2.0 RFC 6749, sección 4.4).

* [Client Credentials Flow](/docs/es/get-started/authentication-and-authorization-flow/client-credentials-flow)
* [Llamar a la API con Client Credentials Flow](/docs/es/get-started/authentication-and-authorization-flow/client-credentials-flow/call-your-api-using-the-client-credentials-flow)

<div id="device-authorization-flow">
  ## Flujo de autorización de dispositivos
</div>

En los dispositivos con limitación de entrada que se conectan a internet, en lugar de autenticar al usuario directamente, el dispositivo le pide que visite un enlace desde su computadora o smartphone y autorice el dispositivo. Esto evita una mala experiencia de usuario en dispositivos que no cuentan con una forma sencilla de introducir texto. Para ello, las aplicaciones para dispositivos usan el <Tooltip tip="Flujo de autorización: concesión de autorización (o workflow) especificada en el marco de OAuth 2.0." cta="Ver glosario" href="/docs/es/glossary?term=Authorization+Flow">Flujo de autorización</Tooltip> de dispositivos (definido en OAuth 2.0). Para su uso con aplicaciones móviles o nativas.

* [Flujo de autorización de dispositivos](/docs/es/get-started/authentication-and-authorization-flow/device-authorization-flow)
* [Llamar a la API usando el flujo de autorización de dispositivos](/docs/es/get-started/authentication-and-authorization-flow/device-authorization-flow/call-your-api-using-the-device-authorization-flow)

<div id="resource-owner-password-flow">
  ## Flujo de contraseña del propietario del recurso
</div>

Aunque no lo recomendamos, las aplicaciones de alta confianza pueden usar el flujo de contraseña del <Tooltip tip="Resource Owner: Entidad (como un usuario o una aplicación) capaz de conceder acceso a un recurso protegido." cta="Ver glosario" href="/docs/es/glossary?term=Resource+Owner">Resource Owner</Tooltip>, que solicita a los usuarios que proporcionen credenciales (identificador y contraseña), normalmente mediante un formulario interactivo. El flujo de contraseña del propietario del recurso solo debe usarse cuando no sea posible usar flujos basados en redirección (como el [flujo de código de autorización](/docs/es/get-started/authentication-and-authorization-flow/authorization-code-flow)).

* [Flujo de contraseña del propietario del recurso](/docs/es/get-started/authentication-and-authorization-flow/resource-owner-password-flow)
* [Llamar a la API con el flujo de contraseña del propietario del recurso](/docs/es/get-started/authentication-and-authorization-flow/resource-owner-password-flow/call-your-api-using-resource-owner-password-flow)

<div id="client-initiated-backchannel-authentication-flow">
  ## Client-Initiated Backchannel Authentication Flow
</div>

Con Client-Initiated Backchannel Authentication Flow (CIBA), en lugar de autenticar al usuario directamente, el backend de la aplicación cliente inicia un flujo de autenticación para pedir al usuario que se autentique. La autenticación en sí se realiza en un dispositivo de autenticación independiente, normalmente un smartphone con una aplicación personalizada.

* [Client-Initiated Backchannel Authentication Flow](/docs/es/get-started/authentication-and-authorization-flow/client-initiated-backchannel-authentication-flow)
* [Autenticación de usuario con CIBA](/docs/es/get-started/authentication-and-authorization-flow/client-initiated-backchannel-authentication-flow/user-authentication-with-ciba)
* [Autorización de usuario con CIBA](/docs/es/get-started/authentication-and-authorization-flow/client-initiated-backchannel-authentication-flow/user-authorization-with-ciba)

<div id="custom-token-exchange">
  ## Custom Token Exchange
</div>

Custom Token Exchange (CTE) permite a las aplicaciones intercambiar tokens de identidad existentes por tokens de Auth0 invocando el endpoint `/oauth/token`, tal como se define en [RFC 8693](https://datatracker.ietf.org/doc/html/rfc8693). Por ejemplo, puedes usar Custom Token Exchange para intercambiar tokens de Auth0 y acceder a otra audience en nombre del usuario. Para obtener más información sobre los casos de uso de Custom Token Exchange, consulta [Casos de uso de ejemplo](/docs/es/authenticate/custom-token-exchange/cte-example-use-cases).

Al asociar un [Custom Token Exchange Profile](/docs/es/authenticate/custom-token-exchange/configure-custom-token-exchange) con una Action que contiene lógica personalizada, puedes implementar flujos de trabajo de identidad altamente personalizados mediante el intercambio de un token de seguridad por otro, todo ello sin perder el control sobre la lógica de autorización.
