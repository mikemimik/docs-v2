<Card title="Antes de empezar">
  * Suscríbete a un plan Enterprise. Para obtener más información, consulta [Manage Subscriptions](/docs/es/troubleshoot/customer-support/manage-subscriptions).
  * Comprueba si usas Public Cloud o Private Cloud. Para conocer los requisitos previos de Private Cloud para estas instrucciones, consulta [Deploy Private Cloud](/docs/es/deploy-monitor/deploy-private-cloud).
</Card>

Puedes configurar Auth0 para permitir que los [miembros del tenant](/docs/es/get-started/manage-dashboard-access) usen su propio [proveedor de identidad empresarial](/docs/es/authenticate/identity-providers/enterprise-identity-providers) (<Tooltip tip="Proveedor de identidad (IdP): servicio que almacena y administra identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=IdP">IdP</Tooltip>) para autenticarse en el <Tooltip tip="Auth0 Dashboard: el producto principal de Auth0 para configurar tus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip> mediante <Tooltip tip="Auth0 Dashboard: el producto principal de Auth0 para configurar tus servicios." cta="Ver glosario" href="/docs/es/glossary?term=single+sign-on">inicio de sesión único</Tooltip> (SSO).

<div id="how-it-works">
  ## Cómo funciona
</div>

Configurar el SSO para el Dashboard requiere trabajar con el equipo de soporte de Auth0 para agregar una conexión empresarial a la autoridad del tenant raíz (RTA), que gestiona los métodos de autenticación disponibles que un miembro del tenant puede usar para iniciar sesión en el Dashboard.

Agregar esta conexión SSO no impide que los miembros del tenant inicien sesión con los métodos de autenticación existentes (como email/password, [LinkedIn](https://marketplace.auth0.com/integrations/linkedin-social-connection), [Microsoft](https://marketplace.auth0.com/integrations/microsoft-account-social-connection), [GitHub](https://marketplace.auth0.com/integrations/github-social-connection) o [Google](https://marketplace.auth0.com/integrations/google-social-connection)).

Configurar el SSO para el Dashboard también habilita el SSO para todos los sitios públicos de Auth0, como:

* sitio web de Auth0 (https://auth0.com)
* Auth0 Community (https://community.auth0.com)
* documentación de Auth0 (https://www.auth0.com/docs)
* Auth0 Support Center (https://support.auth0.com)

<div id="user-experience">
  ### Experiencia de usuario
</div>

Cuando un usuario autorizado accede al Dashboard para iniciar sesión, introduce la dirección de correo electrónico de un dominio registrado (por ejemplo, `user@example.com`) en la página de Auth0 <Tooltip tip="Universal Login: Tu aplicación redirige a Universal Login, alojado en el servidor de autorización de Auth0, para verificar la identidad de un usuario." cta="Ver glosario" href="/docs/es/glossary?term=Universal+Login">Universal Login</Tooltip> y, a continuación, se le redirige a su IdP para completar la autenticación.

<div id="limitations">
  ### Limitaciones
</div>

Antes de decidir si configurar SSO para el Dashboard, ten en cuenta estas limitaciones:

* SSO no se puede limitar a tenants específicos.
* SSO no admite flujos de autenticación iniciados por el IdP.
* Las invitaciones a miembros del tenant no pueden automatizarse ni enviarse en bloque con la <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip> de Auth0.
* El acceso de los miembros del tenant no se puede gestionar en función de los grupos de tu IdP.
* <Tooltip tip="Autenticación multifactor (MFA): Proceso de autenticación de usuarios que utiliza un factor adicional al nombre de usuario y la contraseña, como un código por SMS." cta="Ver glosario" href="/docs/es/glossary?term=MFA">MFA</Tooltip> no se puede exigir para todos los miembros de un tenant.

<div id="considerations">
  ### Consideraciones
</div>

<div id="full-directory-access-to-dashboard">
  #### Acceso completo al directorio desde el Dashboard
</div>

Cuando agregas tu IdP como una conexión disponible para que los miembros del tenant inicien sesión, todos los usuarios del directorio de tu IdP pueden acceder al Dashboard, pero solo los miembros del tenant que hayan sido invitados a un tenant determinado pueden acceder a ese tenant.

Cuando los usuarios intentan acceder a un tenant en el Dashboard sin una invitación, el sistema rechaza la operación. Si los usuarios no pertenecen a ningún tenant, el sistema les pide que [completen su perfil de usuario](#complete-user-profiling) y creen un nuevo tenant de prueba. Crear un nuevo tenant de prueba no lo asocia a tu plan Enterprise.

<div id="residual-tenant-member-identities">
  #### Identidades residuales de miembros del tenant
</div>

Si se invitó a un miembro del tenant (y se le dio acceso) a un tenant en el Dashboard con una identidad distinta de la que se creó en la nueva conexión, técnicamente podrá seguir usando esa identidad para acceder al tenant.

Tendrás que decidir si quieres eliminar su identidad anterior o conservarla como un posible método de autenticación alternativo.

<div id="configure-sso-for-the-dashboard">
  ## Configurar el SSO para el Dashboard
</div>

Configurar el SSO para el Dashboard requiere una serie de pasos compartidos entre usted y el equipo de soporte de Auth0.

<div id="share-idp-configuration-data">
  ### Compartir los datos de configuración del IdP
</div>

[Abre un ticket con el equipo de soporte de Auth0](/docs/es/troubleshoot/customer-support/open-and-manage-support-tickets) para compartir los datos de configuración de tu IdP, de modo que puedan configurar el SSO. Incluye la siguiente información al enviar el ticket:

* Los dominios de correo electrónico que quieras asociar con la configuración de SSO
* Nombre del IdP
* Protocolo de autenticación
* Información adicional específica del IdP

Según el IdP y el protocolo de autenticación que quieras usar, se requieren pasos de configuración adicionales:

<div id="adfs-saml">
  #### ADFS (SAML)
</div>

1. [Cree un Relying Party Trust](https://learn.microsoft.com/en-us/windows-server/identity/ad-fs/operations/create-a-relying-party-trust#to-create-a-claims-aware-relying-party-trust-manually) con las siguientes propiedades:

   | Propiedad                         | Valores                                    |
   | --------------------------------- | ------------------------------------------ |
   | ID de entidad                     | `urn:auth0:auth0:{assignedConnectionName}` |
   | Endpoint de devolución de llamada | `https://auth0.auth0.com/login/callback`   |
2. [Agregue una descripción de claim](https://learn.microsoft.com/en-us/windows-server/identity/ad-fs/operations/add-a-claim-description) para cada uno de los siguientes claims:

   | Claim                           | Identificador del claim                                                | Valor                                      |
   | ------------------------------- | ---------------------------------------------------------------------- | ------------------------------------------ |
   | Identificador de nombre         | `http://schemas.xmlsoap.org/ws/2005/05/identity/claims/nameidentifier` | `E-Mail-Addresses` o `User-Principal-Name` |
   | Dirección de correo electrónico | `http://schemas.xmlsoap.org/ws/2005/05/identity/claims/emailaddress`   | N/A                                        |
   | Nombre                          | `http://schemas.xmlsoap.org/ws/2005/05/identity/claims/name`           | N/A                                        |
3. Habilite el endpoint de SAML 2.0.
4. Proporcione al equipo de soporte de Auth0 la siguiente información:

   * Endpoint de inicio de sesión (por ejemplo, `https://{yourServer}/adfs/ls`)
   * Certificado de firma o el archivo XML de metadatos de SAML

<div id="azure-ad-oidc">
  #### Azure AD (OIDC)
</div>

1. [Cree un nuevo registro de aplicación](https://learn.microsoft.com/en-us/azure/active-directory/develop/quickstart-register-app).
2. Establezca el tipo de **Redirect URI** como Web y el valor en `https://auth0.auth0.com/login/callback`.
3. Seleccione **Register**.
4. [Habilite la concesión implícita](https://learn.microsoft.com/en-us/azure/active-directory/develop/v2-oauth2-implicit-grant-flow#send-the-sign-in-request) para el token de ID.
5. [Agregue el claim de correo electrónico](https://learn.microsoft.com/en-us/azure/active-directory/develop/active-directory-optional-claims) al token de ID.
6. Proporcione al equipo de soporte de Auth0 la siguiente información:

   * El ID de la aplicación (cliente)
   * El endpoint de metadatos de OIDC (por ejemplo, `https://login. microsoftonline.com/{yourAzureAdTenantId}/v2.0/.well-known/openid-configuration`)

<div id="azure-ad-saml">
  #### Azure AD (SAML)
</div>

1. [Cree una nueva aplicación empresarial](https://learn.microsoft.com/en-us/azure/active-directory/manage-apps/add-application-portal).

2. Configure el [inicio de sesión único para SAML](https://learn.microsoft.com/en-us/azure/active-directory/develop/single-sign-on-saml-protocol) con las siguientes propiedades (puede que tenga que usar valores provisionales hasta que el equipo de soporte de Auth0 pueda proporcionarle el nombre de la conexión de SSO):

   | Propiedad                 | Valor                                                                |
   | ------------------------- | -------------------------------------------------------------------- |
   | Identificador (Entity ID) | `urn:auth0:auth0:{assignedConnectionName}`                           |
   | URL de respuesta (ACS)    | `https://auth0.auth0.com/login/callback`                             |
   | URL de inicio de sesión   | `https://manage.auth0.com/login?connection={assignedConnectionName}` |

3. Deje la sección **Attributes &amp; Claims** —incluidos `email`, `Unique User Identifier` y, opcionalmente, `name -`— sin cambios con respecto a las sugerencias de Azure.

4. Proporcione al equipo de soporte de Auth0 los datos XML de los metadatos SAML. Puede optar por una de las siguientes opciones:

   * Comparta la URL de los metadatos de federación de la aplicación (por ejemplo, `https://login.microsoftonline.com/{azureAdTenantId}/federationmetadata /2007-06/federationmetadata.xml?appid={appId}`).
   * Descargue el documento XML de metadatos de federación y adjúntelo al ticket.

<div id="google-saml">
  #### Google (SAML)
</div>

Auth0 admite configurar SSO para el Dashboard con un <Tooltip tip="Security Assertion Markup Language (SAML): protocolo estandarizado que permite a dos partes intercambiar información de autenticación sin una contraseña." cta="Ver glosario" href="/docs/es/glossary?term=SAML">SAML</Tooltip> IdP de Google, pero se recomienda indicar a los usuarios que inicien sesión con el método de autenticación de Google existente.

Cuando un usuario inicia sesión en el IdP de SAML de Google, Auth0 le crea una nueva identidad de usuario (independiente de su identidad de usuario de Google existente), lo que puede resultar confuso.

Si quieres configurar SSO para el Dashboard con un IdP de SAML de Google, consulta [IdP genérico (SAML)](#generic-idp-saml) para obtener instrucciones.

<div id="okta-saml">
  #### Okta (SAML)
</div>

1. Cree una aplicación SAML con las siguientes propiedades (es posible que deba usar valores de marcador de posición hasta que el equipo de soporte de Auth0 pueda proporcionarle el nombre de la conexión SSO):

   | Propiedad                                   | Valor                                      |
   | ------------------------------------------- | ------------------------------------------ |
   | ID de entidad                               | `urn:auth0:auth0:{assignedConnectionName}` |
   | endpoint de devolución de llamada (URL ACS) | `https://auth0.auth0.com/login/callback`   |

2. Configure el **Identificador de nombre** para que envíe la dirección de correo electrónico del usuario.

3. Proporcione al equipo de soporte de Auth0 los metadatos SAML en formato XML. Puede hacer una de estas dos cosas:

   * Compartir la URL de los metadatos SAML en formato XML:

     1. Busque la sección **SAML Signing Certificates**.
     2. Seleccione el menú **Actions**.
     3. Seleccione **View IdP metadata** y, a continuación, **Copy Link Address**. Tendrá este formato: `https://{org}.okta.com/app/{appId}/sso/saml/metadata`.
   * Descargue el archivo XML de metadatos SAML y adjúntelo al ticket.

<div id="idp-initiated-authentication-flows">
  ##### Flujos de autenticación iniciados por el IdP
</div>

El SSO para el Dashboard no admite flujos de autenticación iniciados por el IdP. Si quieres que los usuarios puedan seleccionar un icono para iniciar sesión en el Dashboard, debes:

1. Ocultar la aplicación SAML a los usuarios.
2. [Crear una Bookmark App](https://help.okta.com/en/prod/Content/Topics/Apps/Apps_Bookmark_App.htm) que apunte a `https://manage.auth0.com/login?connection={assignedConnectionName}`. Esta es la aplicación que los usuarios podrán seleccionar para iniciar sesión.

Asegúrate de habilitar tanto la aplicación SAML como la Bookmark App para el mismo conjunto de usuarios que pueden usar la aplicación.

<div id="onelogin-saml">
  #### OneLogin (SAML)
</div>

1. Cree un Conector de prueba SAML (SP) con las siguientes propiedades (puede que necesite usar valores provisionales hasta que el equipo de soporte de Auth0 pueda proporcionarle el nombre de la conexión SSO):

   | Propiedad                                      | Valor                                                                |
   | ---------------------------------------------- | -------------------------------------------------------------------- |
   | ID de entidad                                  | `urn:auth0:auth0:{assignedConnectionName}`                           |
   | endpoint de devolución de llamada (URL de ACS) | `https://auth0.auth0.com/login/callback`                             |
   | URL de inicio de sesión                        | `https://manage.auth0.com/login?connection={assignedConnectionName}` |
2. Proporcione al equipo de soporte de Auth0 el archivo XML de metadatos SAML.

<div id="generic-idp-oidc">
  #### Generic IdP (OIDC)
</div>

1. Registra una aplicación (cliente) con el IdP con las siguientes propiedades:

   | Propiedad                    | Valor                                    |
   | ---------------------------- | ---------------------------------------- |
   | URL de devolución de llamada | `https://auth0.auth0.com/login/callback` |

2. [Agrega el claim de correo electrónico](/docs/es/secure/tokens/json-web-tokens/create-custom-claims) al token de ID.

3. Proporciona al equipo de soporte de Auth0 lo siguiente:

   * ID de la aplicación (cliente)
   * URL del emisor o endpoint de metadatos de OIDC (por ejemplo, `https://{idpDomain}/[...]/.well-known/openid-configuration`)

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  El equipo de soporte de Auth0 usará el [modo implícito con flujo Form Post](/docs/es/get-started/authentication-and-authorization-flow/implicit-flow-with-form-post), si está disponible. Si tu IdP no lo admite, tendrás que proporcionarles el secreto de cliente de tu aplicación.
</Callout>

<div id="generic-idp-saml">
  #### IdP genérico (SAML)
</div>

1. El equipo de soporte de Auth0 le proporcionará el nombre de la conexión SSO.
2. Cree una aplicación SAML con las siguientes propiedades:

   | Propiedad                         | Valor                                      |
   | --------------------------------- | ------------------------------------------ |
   | ID de entidad                     | `urn:auth0:auth0:{assignedConnectionName}` |
   | endpoint de devolución de llamada | `https://auth0.auth0.com/login/callback`   |
3. Asegúrese de que las aserciones SAML contengan los siguientes claims:

   | Claim                           | Identificador del claim                                                | Valor                   |
   | ------------------------------- | ---------------------------------------------------------------------- | ----------------------- |
   | Identificador de nombre         | `http://schemas.xmlsoap.org/ws/2005/05/identity/claims/nameidentifier` | `upn` or `emailaddress` |
   | Dirección de correo electrónico | `http://schemas.xmlsoap.org/ws/2005/05/identity/claims/emailaddress`   | N/A                     |
   | Nombre                          | `http://schemas.xmlsoap.org/ws/2005/05/identity/claims/name`           | N/AA                    |
4. Proporcione al equipo de soporte de Auth0 uno de los siguientes elementos:

   * URL de inicio de sesión y certificado de firma
   * Archivo XML de metadatos SAML

<div id="set-up-sso-connection">
  ### Configurar la conexión SSO
</div>

El equipo de soporte de Auth0 utiliza los datos de configuración que proporcionas para completar la configuración inicial de la conexión SSO.

Home Realm Discovery (HRD) no se configura durante la configuración inicial.

<div id="test-sso-connection">
  ### Probar la conexión SSO
</div>

Una vez completada la configuración inicial de la conexión SSO, el equipo de soporte de Auth0 le pedirá que la pruebe para asegurarse de que los datos de configuración sean correctos y de que los miembros del tenant puedan autenticarse en la conexión SSO de la forma que necesita.

El equipo de soporte de Auth0 le proporcionará una URL directa de inicio de sesión que puede usar para iniciar la autenticación de la nueva conexión SSO. Por ejemplo:

`https://manage.auth0.com/login?connection={assignedConnectionName}`

<div id="enforce-sso-with-auth0-teams">
  ## Exigir SSO con Auth0 Teams
</div>

Si usas Auth0 Teams con tu tenant empresarial, puedes exigir la autenticación SSO para los tenants que pertenecen a la cuenta de Teams. Para obtener más información sobre la administración y la gestión de tenants, consulta [Auth0 Teams](/docs/es/get-started/auth0-teams).

1. Abre una nueva ventana del navegador e introduce la cuenta y el identificador de tu Teams: `https://accounts.auth0.com/teams/{team-identifier}.`

   <Callout icon="file-lines" color="#0EA5E9" iconType="regular">
     Puedes obtener el identificador de Teams en la URL o en la [**Configuración**](https://accounts.auth0.com/teams/#/settings) de Teams.
   </Callout>

2. Ve a la página **Seguridad**.

3. [Configura las políticas de seguridad](/docs/es/get-started/auth0-teams/configure-security-policies) seleccionando **Exigir inicio de sesión único**.

<div id="enable-home-realm-discovery-optional">
  ### Habilitar Home Realm Discovery (opcional)
</div>

Si usas Universal Login o Classic Login, puedes solicitar que se habilite HRD para que la página de inicio de sesión reconozca el dominio de la dirección de correo electrónico que introduce el miembro del tenant y luego lo redirija a la nueva conexión SSO.

<div id="when-to-enable-hrd">
  #### Cuándo habilitar HRD
</div>

Si HRD está habilitado, los miembros del tenant que antes usaban una identidad de correo electrónico/contraseña (con una dirección de correo electrónico que coincida con el dominio de HRD configurado) no podrán iniciar sesión a través de la página de inicio de sesión, y HRD solo es compatible con identificadores de correo electrónico.

Debido a este cambio de comportamiento, solicita habilitar HRD solo después de que al menos algunos de los miembros del tenant actuales estén al tanto del cambio y sepan que:

* Recibirán una invitación para unirse a los tenants con la nueva identidad
* Tendrán que volver a invitarse con la nueva identidad

<div id="how-to-bypass-hrd">
  #### Cómo omitir HRD
</div>

Si un miembro del tenant necesita iniciar sesión con su identidad de correo electrónico/contraseña, puedes proporcionarle una URL de inicio de sesión directa:

`https://manage.auth0.com/login?connection=auth0`

Esta URL omite HRD y le permite iniciar sesión con su identidad de correo electrónico/contraseña.

<div id="example-of-hrd-login-behavior">
  #### Ejemplo del comportamiento de inicio de sesión de HRD
</div>

Aquí tienes un ejemplo de una lista de miembros del tenant:

| Tenant            | Miembro del tenant  | Conexión       | ¿Afectado? |
| ----------------- | ------------------- | -------------- | ---------- |
| `fabrikam@us`     | `user1@example.com` | email/password | Sí         |
| `fabrikam@us`     | `user1@gmail.com`   | google-oauth2  | No         |
| `fabrikam@us`     | `user2@example.com` | github         | No         |
| `fabrikam@us`     | `user3@acme.com`    | email/password | No         |
| `fabrikam@us`     | `user4@example.com` | email/password | Sí         |
| `fabrikam-dev@us` | `user5@example.com` | email/password | Sí         |
| `fabrikam-dev@us` | `user1@example.com` | email/password | Sí         |

Si asociamos el dominio example.com a la nueva conexión, los miembros del tenant `user1@gmail.com`, `user2@example.com` y `user3@acme.com` podrán iniciar sesión como lo hacían antes, ya que usan un proveedor social o un correo electrónico con un dominio no asociado.

Por el contrario, los miembros del tenant `user1@example.com`, `user4@example.com` y `user5@example.com` no podrán iniciar sesión como lo hacían antes, porque sus correos electrónicos están asociados al dominio configurado para HRD.

<div id="migrate-existing-tenant-members">
  ## Migrar miembros existentes del tenant
</div>

El proceso para migrar miembros existentes del tenant depende de si has habilitado HRD.

<div id="how-to-migrate-with-hrd-disabled">
  ### Cómo migrar con HRD deshabilitado
</div>

Para migrar miembros del tenant con HRD deshabilitado, debes compartir la URL de inicio de sesión directa para la nueva conexión SSO:

`https://manage.auth0.com/login?connection={assignedConnectionName}`

1. [Crea una nueva invitación para un miembro del tenant](/docs/es/get-started/manage-dashboard-access/add-dashboard-users) para el miembro del tenant.
2. Indica al miembro del tenant que:

   1. Inicie sesión en la nueva conexión SSO mediante la URL de inicio de sesión directa antes de aceptar la invitación.

      <Callout icon="file-lines" color="#0EA5E9" iconType="regular">
        Si es la primera vez que inicia sesión en la nueva conexión SSO, es posible que vea una página de enriquecimiento del perfil (en https://auth.com/profile). No completes ningún campo ni selecciones **Siguiente**. En su lugar, pasa al siguiente paso.
      </Callout>

   2. Copie y pegue la URL de la invitación que recibió en el correo electrónico de invitación en el mismo navegador donde inició sesión en la nueva conexión SSO. Los usuarios no deben seleccionar **Crear cuenta**.

   3. Acepte la invitación.

   4. Si el usuario tiene invitaciones para otros tenants, puede usarlas en ese momento.

<div id="how-to-migrate-with-hrd-enabled">
  ### Cómo migrar con HRD habilitado
</div>

Para migrar miembros del tenant con HRD habilitado, debes seguir pasos similares a los de [agregar miembros del tenant](/docs/es/get-started/manage-dashboard-access/add-dashboard-users):

1. [Crea una nueva invitación para un miembro del tenant](/docs/es/get-started/manage-dashboard-access/add-dashboard-users).
2. Indica al miembro del tenant que:

   1. Cierre sesión en el Dashboard (si anteriormente había iniciado sesión con una identidad antigua).

      <Callout icon="file-lines" color="#0EA5E9" iconType="regular">
        Si esta es la primera invitación que acepta, es posible que vea una página de “Create a new tenant” o “Create a new account”. No seleccione **Next**. En su lugar, siga el paso siguiente.
      </Callout>

   2. Abra el enlace de invitación en el correo electrónico que recibió.

   3. Inicie sesión con la nueva conexión.

   4. Acepte la invitación.
