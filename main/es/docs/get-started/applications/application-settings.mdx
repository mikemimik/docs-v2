En la página [Aplicaciones](https://manage.auth0.com/#/applications) del Dashboard, busca tu aplicación en la lista y haz clic en su nombre para ver las opciones de configuración disponibles.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Las [aplicaciones de terceros](/es/docs/get-started/applications/third-party-applications) tienen un conjunto limitado de propiedades configurables. Las propiedades que no formen parte del conjunto admitido no pueden configurarse desde el Auth0 Dashboard ni mediante la Management API. Para obtener más información, consulta [Controles de seguridad para aplicaciones de terceros](/es/docs/get-started/applications/third-party-applications/security-controls#restricted-client-configuration).
</Callout>

<Frame>![Lista de aplicaciones del Dashboard](/docs/images/cdy7uua7fh8z/1ecNwGgFQZxdP57p0tp3jT/cd608fcfae22e195b604e2707e5a848d/App_List_-_EN.png)</Frame>

<div id="basic-settings">
  ## Configuración básica
</div>

Al editar la configuración de una aplicación existente o crear una nueva, introduces información sobre la aplicación en la vista **Settings**.

<div id="basic-information">
  ### Información básica
</div>

<Frame>![Dashboard Applications configuración de la aplicación Tab Basic Information](/docs/images/cdy7uua7fh8z/2GPUw7BODYuYYH3658Upz3/92a49ec57e6b4d07be96093989baac03/2023-04-11_15-34-58.png)</Frame>

* **Name**: El nombre de tu aplicación. Es editable y aparecerá en el portal, los correos electrónicos, los registros, etc.
* **Domain**: El nombre de tu tenant de Auth0. Lo eliges cuando creas un nuevo tenant de Auth0 y no se puede cambiar. Si necesitas un dominio diferente, debes registrar un nuevo tenant seleccionando **+ Create Tenant** en el menú superior derecho.
* **<Tooltip tip="ID de cliente: valor de identificación dado a tu recurso registrado por Auth0." cta="Ver glosario" href="/es/docs/glossary?term=Client+ID">ID de cliente</Tooltip>**: El identificador único de tu aplicación. Lo usarás al configurar la autenticación con Auth0. Lo genera el sistema cuando creas una aplicación nueva y no puede modificarse.
* **<Tooltip tip="Secreto de cliente: secreto que usa un cliente (aplicación) para autenticarse con el servidor de autorización; solo el cliente y el servidor de autorización deben conocerlo y debe ser lo suficientemente aleatorio como para que no pueda adivinarse." cta="Ver glosario" href="/es/docs/glossary?term=Client+Secret">Secreto de cliente</Tooltip>**: Una cadena que se usa para firmar y validar <Tooltip tip="Secreto de cliente: secreto que usa un cliente (aplicación) para autenticarse con el servidor de autorización; solo el cliente y el servidor de autorización deben conocerlo y debe ser lo suficientemente aleatorio como para que no pueda adivinarse." cta="Ver glosario" href="/es/docs/glossary?term=ID+Tokens">tokens de ID</Tooltip> en los flujos de autenticación y para obtener acceso a endpoints específicos de la API de Auth0. De forma predeterminada, este valor está oculto, así que marca la casilla **Reveal Client Secret** para verlo. Aunque el ID de cliente se considera información pública, el secreto de cliente **debe mantenerse confidencial**. Si alguien puede acceder a tu secreto de cliente, puede emitir tokens y acceder a recursos a los que no debería tener acceso.
* **Description**: Una descripción en texto libre del propósito de la aplicación. Máximo 140 caracteres.

<div id="application-properties">
  ### Propiedades de la aplicación
</div>

<Frame>![Dashboard Aplicaciones Configuración de la aplicación Pestaña Propiedades de la aplicación](/docs/images/cdy7uua7fh8z/4Z66jnQOVFwNo5iuHtdlxo/153fabafccac939aa15c23a65ba974cb/Application_Properties_-_English.png)</Frame>

* **Logo de la aplicación**: La URL de un logotipo (tamaño recomendado: 150x150 píxeles) que se mostrará para la aplicación. Aparece en varias áreas, incluida la lista de aplicaciones en el Dashboard y los formularios de consentimiento personalizado. Si no se configura ninguno, se mostrará la insignia predeterminada para este tipo de aplicación.
* **Titularidad de la aplicación**: Indica si la aplicación es propia o de terceros. Las aplicaciones de terceros están sujetas a [controles de seguridad reforzados](/es/docs/get-started/applications/third-party-applications/security-controls). La titularidad de la aplicación se establece al crearla y no se puede cambiar. Para obtener más información, consulta [aplicaciones propias y de terceros](/es/docs/get-started/applications/first-party-and-third-party-applications).
* **Tipo de aplicación**: El tipo de aplicación de Auth0 determina qué opciones puedes configurar desde el Dashboard. (No se puede editar en las aplicaciones M2M. A veces también está deshabilitado para otros tipos de aplicación de Auth0 si los tipos de concesión seleccionados solo se permiten para el tipo de aplicación seleccionado actualmente). Usa el menú desplegable para seleccionar uno de los siguientes tipos:

  * [Máquina a máquina](/es/docs/get-started/auth0-overview/create-applications/machine-to-machine-apps): Aplicaciones no interactivas, como herramientas de línea de comandos, demonios, dispositivos IoT o servicios que se ejecutan en tu backend. Normalmente, usarás esta opción si tienes un servicio que necesita acceso a una API.
  * [Aplicación nativa](/es/docs/get-started/auth0-overview/create-applications/native-apps): Aplicaciones móviles o de escritorio que se ejecutan de forma nativa en un dispositivo (como iOS o Android).
  * [Aplicación web tradicional](/es/docs/get-started/auth0-overview/create-applications/regular-web-apps): Aplicaciones web tradicionales que ejecutan la mayor parte de su lógica en el servidor (como Express.js o ASP.NET).
  * [Aplicación de página única](/es/docs/get-started/auth0-overview/create-applications/single-page-web-apps): Aplicaciones JavaScript que ejecutan la mayor parte de la lógica de su interfaz de usuario en un navegador web y se comunican con un servidor web principalmente mediante API (como AngularJS + Node.js o React).

<div id="application-uris">
  ### URI de la aplicación
</div>

<Frame>![Dashboard Applications configuración de la aplicación URI de la aplicación](/docs/images/cdy7uua7fh8z/1QhW2i4fTCCp8owey2tMPI/a19f7fc3f84118920d0e5cb2357144da/Application_URIs.png)</Frame>

* **Application Login URI**: En algunos casos, Auth0 necesitará que tu aplicación redirija a la página de login de tu aplicación. Esta URI debe apuntar a una ruta de tu aplicación que redirija al endpoint `/authorize` de tu tenant [endpoint](https://auth0.com/docs/api/authentication#authorize-application). Normalmente, tendrá una forma como `https://myapp.org/login`. Puedes usar los siguientes marcadores de posición en este campo:

  * [**Marcadores de posición de metadatos de la organización**](/es/docs/get-started/applications/wildcards-for-subdomains#organization-url-placeholders): Usa `{organization.metadata.KEY}` para completar dinámicamente la URL en función de los metadatos de la Auth0 Organization asociada a la solicitud (por ejemplo, `https://{organization.metadata.public_login_host}/login`).
  * [**Marcadores de posición de dominio personalizado**](/es/docs/get-started/applications/wildcards-for-subdomains#custom-domain-url-placeholders): Usa `{custom_domain.metadata.KEY}` para completar dinámicamente la URL en función de los metadatos del dominio personalizado utilizado en la solicitud (por ejemplo, `https://{custom_domain.metadata.public_app_host}/login`).

  Para obtener más información, consulta [Configure Default Login Routes](/es/docs/authenticate/login/auth0-universal-login/configure-default-login-routes#dynamic-login-uris-with-metadata-placeholders).
* **Allowed Callback URLs**: Conjunto de URL a las que Auth0 puede redirigir a los usuarios después de autenticarse. Puedes especificar varias URL válidas separándolas con comas (normalmente, para gestionar distintos entornos, como QA o pruebas). En entornos de producción, verifica que las URL no apunten a localhost. Puedes usar los siguientes marcadores de posición en este campo:
  * [**Comodines**](/es/docs/get-started/applications/wildcards-for-subdomains#wildcard-url-placeholders): Usa `*` para subdominios (`*.google.com`) *No se recomienda para entornos de producción.*
  * [**Marcadores de posición de organización**](/es/docs/get-started/applications/wildcards-for-subdomains#organization-url-placeholders): Usa `{organization_name}` para especificar dinámicamente el nombre de una organización registrada (por ejemplo, `https://{organization_name}.example.com`).
  * [**Marcadores de posición de dominio personalizado**](/es/docs/get-started/applications/wildcards-for-subdomains#custom-domain-url-placeholders): Usa `{custom.domain.metadata.KEY}` para completar dinámicamente la URL en función de los metadatos del dominio personalizado utilizado en la solicitud (por ejemplo, `https://{custom_domain.metadata.public_app_url}/callback`).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  La primera URL incluida en este campo se toma como la URL de devolución de llamada predeterminada cuando el flujo de protocolo correspondiente no especifica una de forma explícita. Esto se aplica específicamente a los flujos de SSO iniciados por IdP de SAML, SAML y WS-Fed.
</Callout>

No uses marcadores de posición comodín ni URL de localhost en las devoluciones de llamada de tu aplicación ni en los campos de orígenes permitidos. El uso de URL de redirección con marcadores de posición comodín puede hacer que tu aplicación sea vulnerable a ataques. Para obtener más información, consulta [Unvalidated Redirects and Forwards Cheat Sheet on owasp.org](https://www.owasp.org/index.php/Unvalidated_Redirects_and_Forwards_Cheat_Sheet). En su lugar, cuando corresponda, se recomienda usar URL con el marcador de posición `{organization_name}`. Para obtener más información, consulta [Subdomain URL Placeholders](/es/docs/get-started/applications/wildcards-for-subdomains).

* **Allowed Logout URLs**: Después de que un usuario cierre sesión en Auth0, puedes redirigirlo con el parámetro de consulta `returnTo`. La URL que uses en `returnTo` debe aparecer aquí. Puedes especificar varias URL válidas separándolas con comas. En entornos de producción, verifica que las URL no apunten a localhost.
  * [**Comodines**](/es/docs/get-started/applications/wildcards-for-subdomains#wildcard-url-placeholders): Usa `*` para subdominios (`*.google.com`) *No se recomienda para entornos de producción.*
  * [**Marcadores de posición de dominio personalizado**](/es/docs/get-started/applications/wildcards-for-subdomains#custom-domain-url-placeholders): Usa `{custom.domain.metadata.KEY}` para completar dinámicamente la URL en función de los metadatos del Custom Domain utilizado en la solicitud (por ejemplo, `https://{custom_domain.metadata.public_app_url}/callback`).
* **Orígenes web permitidos**: Lista de URL desde las que puede originarse una solicitud de autorización que use [Autenticación de origen cruzado](/es/docs/authenticate/login/cross-origin-authentication), [Flujo de dispositivo](/es/docs/get-started/authentication-and-authorization-flow/device-authorization-flow) y `web_message` como modo de respuesta. Puedes especificar varias URL válidas separándolas con comas. En entornos de producción, verifica que las URL no apunten a localhost. Las rutas, las cadenas de consulta y la información de hash no se tienen en cuenta al validar estas URL (y, de hecho, pueden hacer que la coincidencia falle). Puedes proporcionar hasta 100 URL en el campo **Orígenes web permitidos**.
  * [**Comodines**](/es/docs/get-started/applications/wildcards-for-subdomains#wildcard-url-placeholders): Usa `*` para subdominios (`*.google.com`) *No se recomienda para entornos de producción.*
  * [**Marcadores de posición de dominio personalizado**](/es/docs/get-started/applications/wildcards-for-subdomains#custom-domain-url-placeholders): Usa `{custom.domain.metadata.KEY`&#125; para completar dinámicamente la URL en función de los metadatos del Custom Domain utilizado en la solicitud (por ejemplo, `https://{custom_domain.metadata.public_app_url}/callback`).
* **Orígenes permitidos (CORS)**: Lista de URL autorizadas para realizar solicitudes de uso compartido de recursos entre orígenes (CORS) a Auth0.
  * [**Marcadores de posición de dominio personalizado**](/es/docs/get-started/applications/wildcards-for-subdomains#custom-domain-url-placeholders): Usa `{custom.domain.metadata.KEY}` para completar dinámicamente la URL en función de los metadatos del Custom Domain utilizado en la solicitud (por ejemplo, `https://{custom_domain.metadata.public_app_url}/callback`).

<Warning>
  Si configuras las URL de tu aplicación exclusivamente con marcadores de posición de dominio personalizado, las solicitudes de autenticación realizadas a través del dominio canónico de tu tenant (por ejemplo, https://your-tenant.us.auth0.com) fallarán.

  Esto ocurre porque el dominio canónico no tiene los metadatos personalizados necesarios para resolver el marcador de posición. Asegúrate de que tu aplicación use el Custom Domain específico para la autenticación, o proporciona una URL de respaldo estática si se requiere el uso del dominio canónico.
</Warning>

<div id="id-token">
  ### Token de ID
</div>

En la sección **Token de ID**, introduce la **Expiración del token de ID** (en segundos), es decir, el tiempo que transcurre antes de que caduque el `id_token` de Auth0. El valor predeterminado es de 36000 segundos, lo que equivale a 10 horas.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  **Usar Auth0 en lugar del IdP para el inicio de sesión único**: Si se habilita, esta configuración evita que Auth0 redirija a los usuarios autenticados con sesiones válidas al proveedor de identidad (como Facebook o ADFS). **Solo para tenants Legacy.**
</Callout>

<div id="refresh-token-rotation">
  ### Rotación del Refresh Token
</div>

En la sección **rotación del <Tooltip tip="token de actualización: token que se usa para obtener un token de acceso renovado sin obligar a los usuarios a volver a iniciar sesión." cta="Ver glosario" href="/es/docs/glossary?term=Refresh+Token">token de actualización</Tooltip>**, habilita o deshabilita la rotación. Cuando está habilitada, al intercambiar un token de actualización, se emitirá uno nuevo y el token existente se invalidará. Esto permite detectar automáticamente la reutilización del token si se filtra. Además, introduce el **Período de superposición de la rotación** (en segundos). Este intervalo es el margen de tiempo permitido durante el cual el mismo `refresh_token` puede usarse para solicitar un `access_token` sin activar la detección automática de reutilización. Para obtener más información, consulta [rotación del Refresh Token](/es/docs/secure/tokens/refresh-tokens/refresh-token-rotation).

<Frame>![Dashboard Applications Applications Settings Tab Rotación del Refresh Token](/docs/images/cdy7uua7fh8z/Rk6i8MUVWZG34P7K6aurb/9457a6ea7ca5942522778c7ed7925023/Screenshot_2024-11-05_at_18.14.22.png)</Frame>

<div id="refresh-token-expiration">
  ### Expiración del token de actualización
</div>

En la sección **Expiración del token de actualización**, habilita o deshabilita la expiración absoluta y por inactividad, y establece la duración (en segundos) de cada una. Para obtener más información, consulta [Configurar la expiración del token de actualización](/es/docs/secure/tokens/refresh-tokens/configure-refresh-token-expiration).

<Frame>![Dashboard Aplicaciones Aplicaciones pestaña Configuración Expiración del token de actualización](/docs/images/cdy7uua7fh8z/2EnpIEc4kBxjnS3T67xPzC/22fdb95e2a2f31cee6f01e02d0a709c2/Refresh_Token_Expiration_-_English.png)</Frame>

<div id="open-redirect-protection">
  ### Protección contra redirecciones abiertas
</div>

Controla cómo Auth0 gestiona las redirecciones de las aplicaciones de terceros. Esta configuración solo está disponible para aplicaciones de terceros con controles de seguridad reforzados.

<Frame>![Interruptor de Protección contra redirecciones abiertas en el Dashboard](/docs/images/third-party-applications/open-redirect-protection.png)</Frame>

Cuando está habilitada (opción predeterminada para las aplicaciones de terceros), Auth0 no redirige a la URL de devolución de llamada de la aplicación cuando se producen errores de autenticación y no expone `application.callback_domain` en las plantillas de correo electrónico. Esto evita ataques de redirección abierta cuando la URI de redirección está controlada por una parte no confiable.

Desactiva la protección contra redirecciones abiertas solo en aplicaciones de terceros cuyas URI de devolución de llamada configuradas sean de confianza.

Para obtener más información, consulta [Protección frente a redirecciones](/es/docs/get-started/applications/third-party-applications/security-controls#redirect-protection).

<div id="advanced-settings">
  ## Configuración avanzada
</div>

La sección **Configuración avanzada** le permite:

* Administrar o agregar metadatos de la aplicación y configuraciones de dispositivos, <Tooltip tip="OAuth 2.0: Marco de autorización que define protocolos y flujos de trabajo de autorización." cta="Ver glosario" href="/es/docs/glossary?term=OAuth">OAuth</Tooltip> y WS-Federation
* Obtener certificados e información del <Tooltip tip="Token Endpoint: Endpoint del servidor de autorización que se usa para solicitar tokens de forma programática." cta="Ver glosario" href="/es/docs/glossary?term=Token+endpoint">Token endpoint</Tooltip>
* Establecer los tipos de concesión para la aplicación

<div id="application-metadata">
  ### Metadata de la aplicación
</div>

Los metadatos de la aplicación son claves y valores de cadena personalizados (cada uno con un máximo de 255 caracteres) que se establecen para cada aplicación. Los metadatos se exponen en el objeto de la aplicación como `client_metadata` y en las reglas como `context.clientMetadata`. Puede crear hasta 10 conjuntos de metadatos.

<div id="oauth">
  ### OAuth
</div>

<Frame>![Pestaña OAuth de configuración avanzada en Application Settings de Applications del Dashboard](/docs/images/cdy7uua7fh8z/OAuth-tab.png)</Frame>

* De forma predeterminada, todas las aplicaciones/API pueden hacer una solicitud de delegación, pero si desea otorgar permisos explícitamente a aplicaciones/API seleccionadas, puede hacerlo en **Allowed Apps/APIs**.
* Los clientes que usan el add-on Highly Regulated Identity pueden usar la configuración **Compliance Enforcement Level** para establecer su nivel de cumplimiento. Para obtener más información, consulte [Configure FAPI Compliance](/es/docs/get-started/applications/configure-fapi-compliance).
* **Non-Verifiable Callback URI End-User Confirmation**: use esta configuración para controlar si se solicita al usuario que confirme el inicio de sesión cuando se usa un URI no verificable como devolución de llamada. Auth0 recomienda no omitir la confirmación del usuario final en estos casos. Esta configuración tiene prioridad sobre la configuración del tenant con el mismo nombre. Para obtener más información, lea [Measures Against Application Impersonation](/es/docs/secure/security-guidance/measures-against-app-impersonation.mdx).
* Establezca el algoritmo que se usa ([**HS256**](https://en.wikipedia.org/wiki/Symmetric-key_algorithm) o [**RS256**](https://en.wikipedia.org/wiki/Public-key_cryptography)) para firmar sus <Tooltip tip="JSON Web Token (JWT): formato estándar de token de ID (y, a menudo, formato de token de acceso) utilizado para representar claims de forma segura entre dos partes." cta="Ver glosario" href="/es/docs/glossary?term=JSON+web+tokens">JSON Web Tokens</Tooltip>. Para obtener más información, lea [JSON Web Token Signing Algorithms](/es/docs/get-started/applications/signing-algorithms). Al seleccionar `RS256` (recomendado), el token se firmará con la clave privada de su tenant.
* Active o desactive la opción **Trust Token Endpoint IP Header**; si está habilitada, `auth0-forwarded-for` se considera confiable y se usa como fuente de información sobre la IP del usuario final para la protección contra ataques de fuerza bruta en el endpoint de token. Esta configuración solo está disponible para aplicaciones web tradicionales y aplicaciones M2M.
* Active o desactive el interruptor para indicar si su aplicación es **OIDC Conformant** o no. Las aplicaciones marcadas como OIDC Conformant seguirán estrictamente la especificación OIDC.
