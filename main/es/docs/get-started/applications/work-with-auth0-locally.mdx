---

En la mayoría de los casos, autenticar usuarios a través de Auth0 requiere una conexión a internet. Sin embargo, aún puedes desarrollar y probar aplicaciones que usen Auth0 localmente. En algunos casos, puede que ni siquiera necesites conexión a internet.

Para aprender a estructurar tus entornos de desarrollo, pruebas y producción al usar Auth0, consulta [Configurar varios entornos](/es/docs/get-started/auth0-overview/create-tenants/set-up-multiple-environments).

<div id="use-json-web-tokens-with-client-side-applications">
  ## Usa JSON Web Tokens con aplicaciones del lado del cliente
</div>

Dado que los [JSON Web Tokens (JWTs)](/es/docs/secure/tokens/json-web-tokens) no tienen estado (es decir, a la aplicación que los consume solo le importa su contenido, no sus estados anteriores), este es uno de los escenarios más fáciles de probar localmente.

Puedes obtener <Tooltip tip="JSON Web Token (JWT): Formato estándar de token de ID (y, a menudo, de token de acceso) que se utiliza para representar claims de forma segura entre dos partes." cta="Ver glosario" href="/es/docs/glossary?term=JWTs">JWTs</Tooltip> para pruebas mediante cualquiera de los siguientes métodos:

1. Crea un usuario de prueba para una [conexión de base de datos](/es/docs/get-started/applications/set-up-database-connections) e inicia sesión con ese usuario mediante programación. Básicamente, estás usando el [flujo de contraseña del propietario del recurso](/es/docs/get-started/authentication-and-authorization-flow/resource-owner-password-flow). Para obtener instrucciones detalladas de implementación, lee [Llama a tu API usando el flujo de contraseña del propietario del recurso](/es/docs/get-started/authentication-and-authorization-flow/resource-owner-password-flow/call-your-api-using-resource-owner-password-flow).
2. Usa un bot de navegador para simular ser un usuario, iniciar sesión y recuperar un JWT.

<div id="use-sessions-with-server-side-applications">
  ## Usa sesiones con aplicaciones del lado del servidor
</div>

A menos que tu aplicación del lado del servidor permita generar sesiones artificiales para pruebas, necesitarás una forma de iniciar sesión manualmente a través de Auth0.

Para obtener una descripción general de cómo hacerlo, consulta [Flujo de código de autorización](/es/docs/get-started/authentication-and-authorization-flow/authorization-code-flow). Para ver instrucciones de implementación detalladas, consulta nuestro tutorial, [Agregar inicio de sesión mediante el flujo de código de autorización](/es/docs/get-started/authentication-and-authorization-flow/authorization-code-flow/add-login-auth-code-flow).

<div id="use-local-domains-with-auth0">
  ## Usa dominios locales con Auth0
</div>

Si estás desarrollando tu aplicación de forma local, puedes usar `localhost` y otros dominios a los que Auth0 no puede acceder (como los de una intranet) como [URL de devolución de llamada](/es/docs/authenticate/login/redirect-users-after-login). Por ejemplo, durante el desarrollo, podrías usar `http://localhost:3000/callback` como URL de devolución de llamada.

1. Ve a [Auth0 Dashboard &gt; Applications &gt; Applications](https://manage.auth0.com/#/applications/\{yourClientId}/settings) y haz clic en la aplicación.
2. Agrega la URL a la lista de **Allowed Callback URLs**.

Como el principal protocolo de identidad de Auth0 es [OpenID Connect (OIDC)](/es/docs/authenticate/protocols/openid-connect-protocol), Auth0 nunca necesita llamar directamente al servidor de tu aplicación. En cambio, Auth0 redirige a los usuarios a los endpoints de tu aplicación con la información requerida en una cadena de consulta o un fragmento hash.

<div id="divert-emails-for-testing">
  ## Desviar correos electrónicos para pruebas
</div>

Si quieres probar tu aplicación local y no quieres que los correos electrónicos (de creación, validación, etc.) se envíen a la dirección de correo electrónico real de los usuarios que tu aplicación crea o valida, Auth0 recomienda usar un proveedor de correo electrónico personalizado. Por ejemplo, un servicio como [Mailtrap](https://mailtrap.io/signin) o tu propia implementación de un servidor SMTP personalizado pueden aplicar la lógica que necesites para interceptar los correos electrónicos. Así te aseguras de que los usuarios no los reciban, pero tú sí puedas acceder a ellos para tareas de validación y resolución de problemas. Para obtener más información, consulta [Configure Test SMTP Email Server](/es/docs/customize/email/configure-test-smtp-email-servers).

Auth0 no permite ciertos dominios &quot;falsos&quot; que se usan habitualmente durante las pruebas. Usa direcciones de correo electrónico reales para evitar interrupciones o errores.
