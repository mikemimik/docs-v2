Usa esta página para resolver errores comunes al integrar aplicaciones de terceros. Para obtener una visión general de las capacidades y restricciones de las aplicaciones de terceros, consulta [Controles de seguridad para aplicaciones de terceros](/es/docs/get-started/applications/third-party-applications/security-controls).

<div id="identify-third-party-application-issues">
  ## Identificar problemas de aplicaciones de terceros
</div>

Si encuentras un error durante un flujo de OAuth, comprueba si la aplicación es de terceros:

* **Prefijo del ID de cliente**: Las aplicaciones de terceros tienen un `client_id` que empieza por `tpc_`.
* **Logs del tenant**: En [Auth0 Dashboard &gt; Monitoring &gt; Logs](https://manage.auth0.com/#/logs), filtra por la aplicación para revisar los eventos de error.

<div id="common-errors">
  ## Errores comunes
</div>

<div id="unauthorized_client-when-requesting-tokens">
  ### `unauthorized_client` al solicitar tokens
</div>

**Causa**: La aplicación de terceros no tiene una autorización de cliente para la API solicitada. Las aplicaciones de terceros siempre requieren una autorización de cliente explícita, incluso cuando la política de acceso a la API está configurada como **Allow All**.

**Solución**: Cree una autorización de cliente para la aplicación o configure permisos predeterminados para aplicaciones de terceros. Para obtener más información, consulte [Acceso de aplicaciones a las API: autorizaciones de cliente](/es/docs/get-started/applications/application-access-to-apis-client-grants).

```bash
curl --request POST \
  --url 'https://YOUR_DOMAIN/api/v2/client-grants' \
  --header 'Authorization: Bearer YOUR_MANAGEMENT_API_TOKEN' \
  --header 'Content-Type: application/json' \
  --data '{
    "default_for": "third_party_clients",
    "audience": "https://api.example.com",
    "scope": ["read:items", "write:items"],
    "subject_type": "user"
  }'
```

<div id="unauthorized_client-even-with-allow-all-api-policy">
  ### `unauthorized_client` incluso con la política de API &quot;Allow All&quot;
</div>

**Causa**: La configuración de la política de acceso de la API **Allow All** solo se aplica a las aplicaciones propias. Las aplicaciones de terceros siempre requieren una autorización de cliente explícita, independientemente de esta configuración.

**Solución**: Configura los [permisos predeterminados para aplicaciones de terceros](/es/docs/get-started/applications/application-access-to-apis-client-grants#default-permissions-for-third-party-applications) o crea permisos específicos para cada aplicación.

<div id="invalid_request-on-authorize-with-unsupported-parameters">
  ### `invalid_request` en `/authorize` con parámetros no admitidos
</div>

**Causa**: Las aplicaciones de terceros realizan una validación estricta de los parámetros en el endpoint `/authorize`. No se admiten parámetros como `screen_hint`, `login_ticket`, `invitation`, `request` (JAR) y `request_uri` (PAR).

**Solución**: Elimina los parámetros no admitidos de tu solicitud de autorización. Para consultar la lista de parámetros permitidos, lee [Controles de seguridad para aplicaciones de terceros](/es/docs/get-started/applications/third-party-applications/security-controls#authorize-parameter-validation).

<div id="unsupported_response_type-for-id_token-or-token">
  ### `unsupported_response_type` para `id_token` o `token`
</div>

**Causa**: El flujo implícito (`response_type=token` o `response_type=id_token`) no está disponible para aplicaciones de terceros.

**Solución**: Usa `response_type=code` con [PKCE](/es/docs/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce).

<div id="no-id-token-returned-from-oauthtoken">
  ### No se devuelve ningún ID token desde `/oauth/token`
</div>

**Causa**: Las aplicaciones de terceros con controles de seguridad reforzados no devuelven ID tokens ni procesan scopes de OIDC (`openid`, `profile`, `email`) en esta versión. El endpoint de token devolverá un token de acceso, pero no `id_token`.

**Solución**: Usa tokens de acceso con scopes de API para obtener la información que necesita tu aplicación. La compatibilidad con OIDC para aplicaciones de terceros está prevista para una versión futura.

<div id="grant-type-not-supported">
  ### Tipo de concesión no admitido
</div>

**Causa**: Solo se admiten los tipos de concesión `authorization_code`, `refresh_token` y `client_credentials`. Los tipos de concesión como `implicit`, `password` y `urn:ietf:params:oauth:grant-type:device_code` no están disponibles.

**Solución**: Para los flujos de usuario, usa el [Authorization Code Flow with PKCE](/es/docs/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce). Para el acceso de máquina a máquina, usa el [Client Credentials Flow](/es/docs/get-started/authentication-and-authorization-flow/client-credentials-flow) con un cliente confidencial (`token_endpoint_auth_method` no debe ser `none`).

<div id="classic-login-not-working">
  ### Classic Login no funciona
</div>

**Causa**: [Classic Login](/es/docs/authenticate/login/auth0-universal-login/universal-login-vs-classic-login/classic-experience) no es compatible con las aplicaciones de terceros.

**Solución**: Usa [Universal Login](/es/docs/authenticate/login/auth0-universal-login). Universal Login es la experiencia de inicio de sesión recomendada para todas las aplicaciones.

<div id="client-id-starts-with-tpc_">
  ### El ID de cliente empieza por `tpc_`
</div>

**Causa**: Las aplicaciones de terceros reciben automáticamente el prefijo `tpc_` en su ID de cliente para la clasificación del tráfico. Se asigna al crearse y no puede modificarse.

**Solución**: Este comportamiento es el esperado. Actualiza cualquier validación del lado del cliente o restricción de la base de datos para admitir el formato más largo del ID de cliente.

<div id="cannot-change-is_first_party-or-security-mode">
  ### No se puede cambiar `is_first_party` ni el modo de seguridad
</div>

**Causa**: El modo de seguridad y la propiedad de la aplicación son decisiones de diseño permanentes que se establecen al crearla. No se pueden cambiar después.

**Solución**: Cree una nueva aplicación con la configuración deseada. No puede convertir una aplicación existente de propia a de terceros, ni cambiarla de un modo de seguridad a otro.

<div id="email-verification-or-password-reset-shows-an-error-page">
  ### La verificación del correo electrónico o el restablecimiento de la contraseña muestra una página de error
</div>

**Causa**: La `redirection_policy` de la aplicación está configurada como `open_redirect_protection`, lo que impide que Auth0 exponga `application.callback_domain` en las plantillas de correo electrónico.

**Solución**: Actualiza tus plantillas de correo electrónico con una condición Liquid que proporcione una alternativa para las aplicaciones de terceros:

```liquid wrap lines
{% if application.callback_domain == '' %}
  https://YOUR_FALLBACK_DOMAIN
{% endif %}
{% if application.callback_domain != '' %}
  {{ application.callback_domain }}/result-page
{% endif %}
```

Como alternativa, configura `redirection_policy` como `allow_always` para aplicaciones de terceros de confianza creadas mediante el Dashboard o la Management API. Para obtener más información, consulta [Controles de seguridad para aplicaciones de terceros](/es/docs/get-started/applications/third-party-applications/security-controls#redirect-protection).

<div id="dcr-client-cannot-access-any-api">
  ### El cliente DCR no puede acceder a ninguna API
</div>

**Causa**: Los clientes registrados dinámicamente requieren que se configuren permisos predeterminados antes de poder solicitar tokens. Sin permisos predeterminados, los clientes DCR de terceros no tienen acceso a ninguna API.

**Solución**: Configure los permisos predeterminados para las aplicaciones de terceros en cada API a la que deban acceder los clientes DCR. Para obtener más información, lea [Configure Third-Party Applications](/es/docs/get-started/applications/third-party-applications/configure-third-party-applications#default-permissions-for-all-third-party-applications).

<div id="userinfo-returns-error">
  ### `/userinfo` devuelve un error
</div>

**Causa**: El endpoint `/userinfo` no está disponible para las aplicaciones de terceros en esta versión.

**Solución**: Usa token de acceso con scope de API para obtener la información que necesita tu aplicación. La compatibilidad con OIDC, incluido `/userinfo`, está prevista para una próxima versión.

<div id="oauthrevoke-works-but-logout-endpoints-do-not">
  ### `/oauth/revoke` funciona, pero no los endpoints de cierre de sesión
</div>

**Causa**: Los endpoints de cierre de sesión (`/v2/logout`) no están disponibles para aplicaciones de terceros.

**Solución**: Usa `POST /oauth/revoke` para revocar los tokens de actualización. La aplicación debe encargarse de borrar su propio estado de sesión.

<div id="connection-not-available-for-a-third-party-application">
  ### Conexión no disponible para una aplicación de terceros
</div>

**Causa**: La conexión no se ha promovido al nivel de dominio. Las aplicaciones de terceros solo pueden autenticar usuarios mediante conexiones a nivel de dominio.

**Solución**: Promueva la conexión al nivel de dominio. Para obtener más información, consulte [Promover conexiones al nivel de dominio](/es/docs/authenticate/identity-providers/promote-connections-to-domain-level).

<div id="refresh-token-rotation-causing-issues">
  ### La rotación de tokens de actualización causa problemas
</div>

**Causa**: La rotación de tokens de actualización está habilitada de forma predeterminada para aplicaciones públicas (SPA, nativas) de terceros, en consonancia con los requisitos de OAuth 2.1.

**Solución**: Asegúrate de que tu aplicación gestione correctamente la rotación de tokens de actualización: cada intercambio de token devuelve un nuevo token de actualización y el anterior queda invalidado. Los administradores pueden ajustar la configuración de rotación de las aplicaciones creadas manualmente a través del Dashboard o la Management API.
