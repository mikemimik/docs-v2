Esta sección contiene algunas recomendaciones sobre cómo configurar tu cliente si quieres probar tu solución con las [pruebas de conformidad de FAPI](https://openid.net/certification/certification-fapi_op_testing/).

Para aprobar las <Tooltip tip="OpenID: Estándar abierto de autenticación que permite a las aplicaciones verificar la identidad de los usuarios sin recopilar ni almacenar información de inicio de sesión." cta="Ver glosario" href="/docs/es/glossary?term=OpenID">OpenID</Tooltip> pruebas de conformidad de FAPI, primero configura lo siguiente:

* Establece la propiedad `compliance_level` en el perfil deseado: `fapi1_adv_pkj_par`, `fapi1_adv_mtls_par`, `fapi2_sp_pkj_mtls` o `fapi2_sp_mtls_mtls`
* [Configura mTLS](/docs/es/get-started/applications/configure-mtls) (incluidos los [alias de mTLS](/docs/es/get-started/applications/configure-mtls/configure-mtls-for-a-tenant#enable-mtls-aliases)) o [configura clave privada JWT](/docs/es/get-started/applications/configure-private-key-jwt)
* [Configura el enlace de token mTLS](/docs/es/get-started/applications/configure-mtls/configure-mtls-for-a-client#enable-token-binding)
* Configura [Pushed Authorization Requests](/docs/es/get-started/applications/configure-par)
* Asegúrate de que la propiedad `oidc_conformant` esté establecida en `true` para todos los clientes que se utilicen en la prueba. Este es el valor predeterminado para los clientes creados con Auth0 Dashboard.

Luego, sigue las instrucciones a continuación para completar la configuración de las pruebas de conformidad de FAPI de OpenID:

* [Asegúrate de que Auth0 solicite el consentimiento del usuario](#ensure-auth0-prompts-users-for-consent)
* [Configura los claims de ACR compatibles para el tenant](#configure-supported-acr-claims-for-the-tenant)
* [Elimina la propiedad alg del endpoint JWKS](#remove-the-alg-property-from-jwks-endpoint)
* [Agrega una Action para exigir scope y redirect&#95;uri](#add-action-to-require-scope-and-redirect-uri)
* [(Solo para perfiles FAPI2) Asegúrate de que se devuelva el claim `iss` en las respuestas](#fapi2-profiles-only-ensure-iss-claim-is-returned-in-responses)

<div id="ensure-auth0-prompts-users-for-consent">
  ### Asegúrate de que Auth0 solicite el consentimiento del usuario
</div>

Debes asegurarte de que Auth0 solicite el consentimiento del usuario. Puedes omitir este paso si el cliente está configurado como una aplicación first-party y el <Tooltip tip="Servidor de recursos: servidor que aloja recursos protegidos. Los servidores de recursos aceptan y responden a solicitudes de recursos protegidos." cta="Ver glosario" href="/docs/es/glossary?term=Resource+Server">Servidor de recursos</Tooltip> o la API permite omitir el consentimiento para aplicaciones first-party. Para asegurarte de que Auth0 solicite el consentimiento del usuario, establece la propiedad `is_first_party` del cliente en `false`:

```bash lines
curl --location --request PATCH 'https://{YOUR_DOMAIN}/api/v2/clients/YOUR_CLIENT_ID' \
  --header 'Authorization: Bearer {YOUR_MANAGEMENT_API_ACCESS_TOKEN}' \
  --header 'Content-Type: application/json' \
  --data-raw '{
  "is_first_party": false  
}'
```

Luego, eleva tu conexión a nivel de dominio:

```bash lines
curl --location --request PATCH 'https://{YOUR_DOMAIN}/api/v2/connections/YOUR_CONNECTION_ID' \
  --header 'Authorization: Bearer {YOUR_MANAGEMENT_API_ACCESS_TOKEN}' \
  --header 'Content-Type: application/json' \
  --data-raw '{
  "is_domain_connection": true
}'
```

<div id="configure-supported-acr-claims-for-the-tenant">
  #### Configura los claims de ACR compatibles para el tenant
</div>

Las pruebas de FAPI envían un valor de ACR obligatorio: `urn:mace:incommon:iap:silver`. Para incluir el valor de ACR obligatorio en el <Tooltip tip="ID Token: Credencial destinada al propio cliente, en lugar de para acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=ID+token">token de ID</Tooltip>, agrega `urn:mace:incommon:iap:silver` a la lista de valores de ACR compatibles para el tenant:

```bash lines
curl --location --request PATCH 'https://{YOUR_DOMAIN}/api/v2/tenants/settings' \
  --header 'Authorization: Bearer {YOUR_MANAGEMENT_API_ACCESS_TOKEN}' \
  --header 'Content-Type: application/json' \
  --data-raw '{
    "acr_values_supported": ["urn:mace:incommon:iap:silver"]
}'
```

<div id="remove-the-alg-property-from-jwks-endpoint">
  #### Elimina la propiedad alg del endpoint JWKS
</div>

Para permitir que las claves se usen con varios algoritmos, no solo con RS256, elimina la propiedad `alg` del tenant de la respuesta del endpoint `/.well-known/jwks.json`:

```bash lines
curl --location --request PATCH 'https://{YOUR_DOMAIN}/api/v2/tenants/settings' \
  --header 'Authorization: Bearer {YOUR_MANAGEMENT_API_ACCESS_TOKEN}' \
  --header 'Content-Type: application/json' \
  --data-raw '{
    "flags": {
        "remove_alg_from_jwks": true
    }
}'
```

<div id="add-action-to-require-scope-and-redirect_uri">
  #### Agregar una Action para exigir `scope` y `redirect_uri`
</div>

De forma predeterminada, Auth0 permite solicitudes sin un `scope` y, si no se especifica ninguno, asume el `scope` `openid`. Auth0 también permite solicitudes sin un `redirect_uri,` que puedes configurar en [Actions](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger/redirect-with-actions). Sin embargo, las pruebas de conformidad de FAPI exigen que Auth0 sea más restrictivo.

Agrega la siguiente Action para aplicar las restricciones necesarias a `scope` y `redirect_uri`:

```js lines
exports.onExecutePostLogin = async (event, api) => {
  if (!event.request.body || !event.request.body.refresh_token) {
    // Requerir un scope
    if (!event.request.query.scope) {
      api.access.deny('scope must be provided in the request');
    }
    // Para mejorar el mensaje de error si redirect_uri no está presente
    if (!event.request.query.redirect_uri) {
      api.access.deny('redirect_uri must be provided in the request');
    }
  }
};
```

<div id="fapi2-profiles-only-ensure-iss-claim-is-returned-in-responses">
  #### (Solo para perfiles FAPI2) Asegúrate de que se incluya el claim `iss` en las respuestas
</div>

El perfil de seguridad FAPI 2.0 exige que el parámetro `iss` se incluya en las respuestas de autorización de acuerdo con [RFC9207](https://www.rfc-editor.org/info/rfc9207). Por motivos de compatibilidad, Auth0 no lo hace de forma predeterminada. Para habilitar este comportamiento, establece la propiedad `authorization_response_iss_parameter_supported` en la configuración del tenant en `true`.

```bash lines
curl --location --request PATCH "https://{YOUR_DOMAIN}/api/v2/tenants/settings" \
  --header "Authorization: Bearer {YOUR_MANAGEMENT_API_ACCESS_TOKEN}" \
  --header "Content-Type: application/json" \
  --data-raw '{
    "authorization_response_iss_parameter_supported": true
  }'
```
