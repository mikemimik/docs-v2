La multitenencia es un enfoque arquitectónico en el que una única instancia de software se ejecuta en un servidor y es accesible para varios grupos de usuarios. Con la multitenencia, puede segmentar a los usuarios con características compartidas en grupos y otorgarles permisos y niveles de acceso independientes a su aplicación. Esto le permite crear y mantener experiencias personalizadas para distintos clientes, unidades de negocio u otros grupos de usuarios definidos.

En Auth0, el mejor método para implementar la multitenencia es [Auth0 Organizations](/docs/es/manage-users/organizations/organizations-overview). Si es necesario, también pueden usarse otras soluciones heredadas para adaptarse a distintos casos de uso empresariales. Si ofrece un producto o servicio [business-to-business (B2B)](/docs/es/get-started/architecture-scenarios/business-to-business), configurar la multitenencia para sus usuarios de negocio puede ser beneficioso para su caso de uso.

Las siguientes secciones describen las opciones disponibles para implementar la multitenencia en Auth0.

<Warning>
  En este artículo, el término de arquitectura de software &quot;tenant&quot; se usa para referirse a un grupo de usuarios que puede acceder a su aplicación. Al referirse a su instancia de Auth0, se utiliza el término &quot;tenant de Auth0&quot;.
</Warning>

<div id="auth0-organizations">
  ## Organizaciones de Auth0
</div>

Para la mayoría de los casos de uso de multitenencia, Auth0 Organizations es la solución ideal para ti y tus usuarios. Auth0 Organizations admite implementaciones entre empresas (B2B) con una o más aplicaciones a las que los usuarios finales pueden acceder.

Las características comunes de las implementaciones B2B incluyen:

* Un producto con licencia para otra empresa, destinado al uso por parte de sus empleados.
* Varias organizaciones que requieren su propia federación y una personalización básica de la marca en la experiencia de autenticación.
* Distintos niveles de acceso a la aplicación para diferentes grupos de usuarios.

Con Auth0 Organizations, puedes crear grupos de usuarios diferenciados y adaptar sus experiencias con [control de acceso basado en roles](/docs/es/manage-users/access-control/rbac), [páginas de inicio de sesión y plantillas de correo electrónico personalizadas](/docs/es/manage-users/organizations/create-first-organization#customize-prompts-and-email-templates), entre otras opciones.

Para obtener más información sobre cómo usar Auth0 Organizations para implementar la multitenencia, consulta [Arquitectura de múltiples organizaciones](/docs/es/get-started/architecture-scenarios/multiple-organization-architecture).

<div id="legacy-solutions">
  ## Soluciones heredadas
</div>

Si Auth0 Organizations no satisface los requisitos de tu caso de uso, puedes considerar las soluciones heredadas que se describen a continuación. Si necesitas orientación para elegir el enfoque más adecuado para tus requisitos específicos, ponte en contacto con nuestro equipo de [Professional Services](/docs/es/get-started/professional-services).

Las soluciones heredadas incluyen:

* Usar una conexión de Auth0 para representar cada tenant.
* Usar una aplicación de Auth0 para representar cada tenant.
* Usar un tenant de Auth0 para representar cada tenant.
* Almacenar los detalles del tenant en el perfil del usuario.

<div id="use-auth0-connections">
  ### Usar conexiones de Auth0
</div>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Pueden aplicarse límites de entidades. Para obtener más información, consulta la [Política de límites de entidades](/docs/es/troubleshoot/customer-support/operational-policies/entity-limit-policy). Si tienes una suscripción Enterprise, no estarás sujeto a límites de entidades, pero sí podrías verte limitado por una conexión que ya tenga miles de clientes habilitados.
</Callout>

Puedes representar cada uno de tus tenants con una conexión de Auth0 independiente.

Este enfoque te permite admitir escenarios en los que:

* Tienes distintos requisitos a nivel de conexión para cada uno de tus tenants, como políticas de contraseñas diferentes.
* Tienes grupos de usuarios de distintas conexiones. Por ejemplo, un tenant podría requerir que los usuarios proporcionen credenciales de nombre de usuario y contraseña, mientras que otro podría requerir que inicien sesión a través de un <Tooltip tip="Identity Provider (IdP): Servicio que almacena y gestiona identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=IdP">IdP</Tooltip> empresarial.

Para pedir a un usuario que inicie sesión a través de una conexión específica, llama al [endpoint de Login de Auth0 Authentication API](https://auth0.com/docs/api/authentication#login) e incluye el parámetro `connection`.

<Warning>
  Si usas [Lock](/docs/es/libraries#lock) en tu aplicación, ten en cuenta que admite un máximo de 50 conexiones de base de datos por aplicación. Las conexiones sociales y empresariales no se ven afectadas por este límite, pero siguen estando sujetas a la [Política de límites de entidades](/docs/es/troubleshoot/customer-support/operational-policies/entity-limit-policy).
</Warning>

<div id="use-auth0-applications">
  ### Usar aplicaciones de Auth0
</div>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Es posible que se apliquen límites de entidades. Para obtener más información, consulta la [Política de límites de entidades](/docs/es/troubleshoot/customer-support/operational-policies/entity-limit-policy). Si tienes una suscripción Enterprise, no estarás limitado por los límites de entidades, pero sí podrías estarlo por una conexión que ya tenga miles de clientes habilitados.
</Callout>

Puedes representar cada uno de tus tenants con una aplicación de Auth0 independiente.

Este enfoque te permite configurar cada aplicación de Auth0 de forma específica según los distintos requisitos de cada tenant, como las conexiones disponibles.

Tendrás que hacer un seguimiento de los tenants a los que pertenecen tus usuarios dentro de tu aplicación. Cuando un usuario inicie sesión en tu aplicación, tendrás que leer esa información y dirigirlo a la aplicación de Auth0 correspondiente para completar la autenticación.

Para habilitar una conexión para varias aplicaciones con la <Tooltip tip="Management API: Un producto para permitir que los clientes realicen tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip> de Auth0, llama al [endpoint para actualizar los clientes habilitados de una conexión](https://auth0.com/docs/api/management/v2/connections/patch-clients) y proporciona el ID de conexión correspondiente.

<div id="use-auth0-tenants">
  ### Usa tenants de Auth0
</div>

Puedes representar cada uno de tus tenants con un tenant de Auth0 independiente.

Este enfoque te permite compartir con los usuarios el acceso al <Tooltip tip="Auth0 Dashboard: el principal producto de Auth0 para configurar tus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip>, restringido por tenant, pero requiere que configures Auth0 de forma individual para cada tenant.

Esto significa que, además de gestionar de forma individual las funciones de cada tenant de Auth0 (como Branding, Actions y <Tooltip tip="Attack Protection: funciones que Auth0 ofrece para detectar y mitigar ataques, incluida la protección contra fuerza bruta, la limitación de IP sospechosas, la detección de contraseñas comprometidas, la detección de bots y la autenticación multifactor adaptativa." cta="Ver glosario" href="/docs/es/glossary?term=Attack+Protection">Attack Protection</Tooltip>), tu aplicación tendrá que admitir varias configuraciones de Auth0.

<div id="store-tenant-details-in-the-users-profile">
  ### Almacena los detalles del tenant en el perfil del usuario
</div>

Puedes almacenar los detalles del tenant en el perfil del usuario y hacer que tu aplicación lea esa información después de que el usuario inicie sesión.

Este enfoque permite que todos tus usuarios, independientemente del tenant al que pertenezcan, inicien sesión con una configuración uniforme (como las conexiones disponibles).

Para implementar esto, podrías almacenar los detalles del tenant en el perfil de Auth0 del usuario, en el [objeto `app_metadata`](/docs/es/manage-users/user-accounts/metadata), usando un identificador de tu elección (por ejemplo, `"tenant": "customer-group-12345"`). Después de que el usuario inicie sesión, tu aplicación recupera la variable `tenant` y muestra una versión adecuada según el valor devuelto.
