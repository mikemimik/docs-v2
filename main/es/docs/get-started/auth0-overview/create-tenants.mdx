Te guiaremos en los primeros pasos para empezar a usar Auth0 y familiarizarte con los conceptos clave del servicio de Auth0. Usaremos la empresa **Example-Co** para ilustrar algunos de los pasos del proceso.

<div id="set-up-an-auth0-account">
  ## Configura una cuenta de Auth0
</div>

Si aún no has [creado](https://auth0.com/signup) una **cuenta** de Auth0, hazlo (es gratis). Puedes usar nombre de usuario/correo electrónico/teléfono y contraseña, o iniciar sesión con un proveedor social (como LinkedIn, Microsoft, GitHub o Google).

<div id="create-a-tenant-and-domain">
  ## Crear un tenant y un dominio
</div>

Una vez creada su cuenta, se le pedirá que cree un **tenant**. Todo comienza con un tenant de Auth0. Aquí es donde configura su uso de Auth0 y donde se definen, administran y almacenan los recursos de Auth0, como [aplicaciones](/es/docs/get-started/applications), [conexiones](/es/docs/authenticate/identity-providers) y [perfiles de usuario](/es/docs/get-started/architecture-scenarios/business-to-business/profile-management). Puede acceder a un tenant de Auth0 a través del [Dashboard](/es/docs/get-started/auth0-overview/dashboard) de Auth0, donde también puede crear tenants adicionales asociados. Puede crear más de un tenant de Auth0 para organizar sus tenants de forma que aíslen distintos dominios de usuarios y, además, den soporte a su ciclo de vida de desarrollo de software (SDLC).

**Los nombres de los tenants no pueden cambiarse ni reutilizarse una vez eliminados.** Así que asegúrese de estar conforme con el nombre o los nombres antes de crear sus tenants de Auth0.

Determinar el nivel de aislamiento que necesita para sus dominios de usuario es un paso importante y, junto con sus requisitos de branding, le ayuda a definir cuántos tenants de Auth0 necesita en su entorno. La cantidad de tenants de Auth0 que necesita administrar puede crecer rápidamente, así que piénselo bien antes de crear varios tenants de Auth0 para producción.

Características del tenant:

* El nombre del tenant debe ser único. Se usará para crear su dominio personal.
* El nombre del tenant solo puede contener caracteres alfanuméricos en minúscula y guiones (&quot;-&quot;). No puede comenzar ni terminar con un guion.
* El nombre del tenant debe tener un mínimo de 3 caracteres y un máximo de 63 caracteres.
* El nombre del tenant no puede cambiarse después de su creación.
* Puede crear más de un tenant; de hecho, se recomienda hacerlo para cada entorno que pueda tener, como desarrollo, staging o producción. Para obtener más información, lea [Set Up Multiple Environments](/es/docs/get-started/auth0-overview/create-tenants/set-up-multiple-environments).

El nombre de su tenant forma parte de su dominio de Auth0. Puede personalizarlo usando un <Tooltip tip="Custom Domain: Dominio de terceros con un nombre especializado o personalizado." cta="Ver glosario" href="/es/docs/glossary?term=custom+domain">dominio personalizado</Tooltip>. Este dominio es la URL base que se utiliza para acceder a la API de Auth0 y la URL donde sus usuarios se autentican.

<div id="region-locality-and-sub-locality">
  ### Región, localidad y sublocalidad
</div>

El nombre de dominio también se compone del valor de localidad de una **región**. Admitimos los siguientes valores de localidad para la [opción de implementación de Public Cloud](/es/docs/deploy-monitor/deployment-options):

| Región                    | Localidad | Sublocalidades       |
| ------------------------- | --------- | -------------------- |
| Australia                 | AU        | AU                   |
| Canadá                    | CA        | CA                   |
| Europa                    | EU        | EU, EU-2             |
| Japón                     | JP        | JP                   |
| Reino Unido               | UK        | UK                   |
| Estados Unidos de América | US        | US, US-3, US-4, US-5 |

Cada una de estas localidades se divide en una sublocalidad (o **entorno del tenant**) con un dígito después de la localidad; por ejemplo, `EU-2`. Los entornos del tenant no se pueden elegir manualmente, pero sí se pueden especificar localidades según la región seleccionada, lo que determina el dominio del tenant asignado y la región donde se alojarán tus datos.

En nuestro ejemplo, **Example-Co** eligió el nombre `example-co` y AU como región. Por lo tanto, su dominio es `example-co.au.auth0.com`.

<div id="custom-domains">
  ## Dominios personalizados
</div>

Recomendamos usar dominios personalizados, como `example-co.com`, en entornos de producción para ofrecer a los usuarios la experiencia más segura y fluida posible.

Los certificados de dominio personalizado pueden ser administrados por Auth0 o administrados por usted.

Para obtener más información, consulte [Dominios personalizados](/es/docs/customize/custom-domains).

<div id="whats-next">
  ## Qué sigue
</div>

* **Crea y registra aplicaciones**: Ahora que tienes una cuenta y un dominio, debes registrar cada aplicación que usará nuestros servicios en el [Auth0 Dashboard](https://manage.auth0.com/#/applications). Para obtener más información, consulta [Aplicaciones en Auth0](/es/docs/get-started/applications) y [Crear aplicaciones](/es/docs/get-started/auth0-overview/create-applications).
* **Configura conexiones**: A continuación, debes configurar cómo se autenticarán tus usuarios al iniciar sesión. Auth0 se encuentra entre tu aplicación y el <Tooltip tip="Proveedor de identidad (IdP): Servicio que almacena y gestiona identidades digitales." cta="Ver glosario" href="/es/docs/glossary?term=identity+provider">proveedor de identidad</Tooltip> que autentica a tus usuarios (como Google o Facebook). La relación entre Auth0 y el proveedor de identidad se denomina **conexión**. Al usar esta capa de conexión, Auth0 aísla tu aplicación de cualquier cambio en la implementación del proveedor de identidad. Para obtener más información, consulta [Autenticación y autorización](/es/docs/get-started/identity-fundamentals/authentication-and-authorization) y [Conexiones](/es/docs/authenticate/identity-providers).

<div id="extend-auth0s-functionality">
  ## Amplía la funcionalidad de Auth0
</div>

Auth0 ofrece varias formas de ampliar la funcionalidad de la plataforma:

* [Actions](/es/docs/customize/actions): Actions son funciones seguras, específicas del tenant y versionadas, escritas en Node.js, que se ejecutan en determinados puntos de la plataforma Auth0. Usa Actions para personalizar y ampliar las capacidades de Auth0 con un login personalizado.
* [Rules](/es/docs/customize/rules): Rules son funciones escritas en JavaScript o C# que se ejecutan en Auth0 justo después de una autenticación correcta y antes de que el control vuelva a tu aplicación. Rules se pueden encadenar para lograr un código modular y se pueden activar y desactivar de forma individual. Puedes usar Rules para:

  * Control de acceso
  * Webhooks
  * Enriquecimiento del perfil
  * Autenticación multifactor (MFA)
* [Hooks](/es/docs/customize/hooks): Hooks te permiten personalizar el comportamiento de Auth0 mediante código en Node.js que se ejecuta en puntos de extensibilidad (similares a los webhooks que vienen con un servidor). Son funciones seguras y autocontenidas asociadas a puntos de extensibilidad específicos de la plataforma Auth0. Auth0 invoca los Hooks en runtime para ejecutar tu lógica personalizada.
* [Extensions](/es/docs/customize/extensions): Las Extensions de Auth0 te permiten instalar aplicaciones o ejecutar comandos/scripts que amplían la funcionalidad básica de Auth0. Puedes usar una de las extensiones predefinidas proporcionadas por Auth0 o crear la tuya propia. Algunas de las acciones que puedes realizar con extensiones incluyen:

  * Gestionar las autorizaciones de los usuarios (mediante grupos, roles y permisos)
  * Importar/exportar usuarios
  * Exportar registros a otros servicios
  * Implementar scripts desde repositorios externos
