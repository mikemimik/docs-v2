Puedes usar Custom Token Exchange para resolver casos avanzados de integración en los que no es posible aplicar estrategias habituales de inicio de sesión federado basadas en redirigir al usuario final, debido a limitaciones técnicas o de experiencia de usuario. El código proporcionado para los casos de uso está incompleto y solo busca mostrar los pasos lógicos que puedes seguir con tu código para abordar cada caso de uso. Consulta los [ejemplos de código](#code-samples) para ver ejemplos más detallados.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Auth0 recomienda usar el inicio de sesión federado habitual, listo para usar, siempre que sea posible. Al permitirte establecer el usuario de la transacción, Custom Token Exchange te ofrece más flexibilidad, pero también te asigna la responsabilidad adicional de validar y gestionar la transacción de forma segura.
</Callout>

<div id="use-cases">
  ## Casos de uso
</div>

Esta sección describe casos de uso de ejemplo y ejemplos de código específicos, junto con recomendaciones para implementar su caso. Para ilustrar los casos de uso, usaremos GearUp, una empresa ficticia de alquiler de coches.

<div id="use-case-seamless-migration-into-auth0">
  ### Caso de uso: migración fluida a Auth0
</div>

GearUp tiene una aplicación móvil que usan millones de personas y necesita modernizar su solución de identidad, por lo que ha decidido pasarse a Auth0. Sin embargo, quiere evitar obligar a los usuarios a volver a autenticarse mientras migran desde su proveedor de identidad heredado, o IdP, ya que eso añade fricción a la experiencia de usuario.

Para resolverlo y limitar los riesgos, GearUp está migrando de forma gradual. Para cada usuario, quiere intercambiar el token de actualización de su IdP heredado por un conjunto de token de acceso de Auth0, token de actualización y <Tooltip tip="token de ID: Credencial destinada al propio cliente, en lugar de usarse para acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=ID+token">token de identidad</Tooltip>. Esto permite que su aplicación empiece a usar Auth0 sin fricciones como IdP de ese usuario, así como consumir las API de GearUp con tokens emitidos por Auth0. Una vez completado el intercambio para todos los usuarios, la aplicación habrá migrado por completo y el IdP anterior podrá desconectarse, todo ello sin afectar a los usuarios finales ni al negocio de GearUp.

<Frame>![](/docs/images/cdy7uua7fh8z/2Ke6p3yZl06KT4HHqtaVu9/5d9c5feb98d614d6d793fb01ccc03e92/Screenshot_2025-02-03_at_5.00.32_PM.png)</Frame>

Como requisito previo, GearUp ha realizado una [importación masiva de usuarios](/docs/es/manage-users/user-migration/bulk-user-imports) en su tenant de Auth0 y la aplicación móvil tiene un token de actualización heredado válido para cada usuario que se va a migrar.

1. La aplicación móvil hace una solicitud a Auth0 para intercambiar el token de actualización heredado, estableciéndolo como token de sujeto.
2. Se ejecuta la Action correspondiente del perfil de Custom Token Exchange. Valida el token de actualización con el IdP heredado y obtiene el ID de usuario externo del perfil de usuario. Luego aplica la política de autorización requerida y, por último, establece el usuario.
3. Auth0 responde con un token de acceso de Auth0, un token de ID y un token de actualización.
4. La aplicación móvil ahora puede usar las API del cliente con tokens de Auth0 sin que el usuario tenga que volver a autenticarse.

* No queremos crear el usuario.
* No queremos actualizar el perfil de usuario.

Usamos el ID de usuario del IdP externo para establecer el usuario en la conexión correspondiente.

<div id="use-case-re-use-an-external-authentication-provider">
  ### Caso de uso: reutilizar un proveedor de autenticación externo
</div>

Otro caso de uso consiste en que GearUp se asocie con Air0, un proveedor líder de viajes, para ofrecer sus servicios de alquiler de coches directamente en la aplicación de página única de Air0. GearUp ofrece una biblioteca de Javascript que encapsula el uso de sus API. De este modo, las API de GearUp pueden ser consumidas fácilmente por el sitio web de Air0, donde se ofrecen servicios de alquiler de coches.

Una vez más, la solución debe ser invisible para los usuarios finales, evitando que tengan que volver a autenticarse en GearUp. Para resolver este problema, la biblioteca de Javascript de GearUp puede realizar un intercambio de tokens usando como entrada el token de identidad externo de Air0. Esto da como resultado un token de acceso de Auth0 que se genera y se asocia con el usuario correspondiente de GearUp en función de su dirección de correo electrónico. Una vez que la biblioteca de GearUp obtiene el token de acceso, puede empezar a usar las API de GearUp para ofrecer servicios de alquiler de coches directamente en el sitio web de Air0.

<Frame>![](/docs/images/cdy7uua7fh8z/34AVzwyYARK6fn2IEnLsQn/409082d736d8495b637626406977fb1f/Screenshot_2025-02-03_at_5.08.47_PM.png)</Frame>

Como requisito previo, GearUp ha configurado Air0 IdP como una conexión federada empresarial o Social, para que el usuario pueda autenticarse mediante inicio de sesión federado o, alternativamente, mediante Custom Token Exchange de la siguiente manera:

1. La aplicación de página única obtiene el token de identidad del IdP externo una vez que el usuario se autentica.
2. Luego solicita el intercambio del token de identidad, estableciéndolo como token de sujeto.
3. Se ejecuta la Action del perfil de Custom Token Exchange correspondiente. Valida el token de identidad y obtiene de él el ID del usuario y otros atributos del perfil. Luego aplica la política de autorización requerida y, por último, asigna el usuario.
4. Auth0 responde con token de acceso de Auth0, token de identidad y token de actualización.
5. El código Javascript que se ejecuta en la SPA ahora puede usar las API de Customer mediante tokens de Auth0 sin que el usuario tenga que volver a autenticarse.

* No queremos reemplazar el perfil del usuario si se obtiene un conjunto de atributos más completo mediante inicio de sesión federado, en caso de que el usuario ya exista.
* No queremos verificar las direcciones de correo electrónico cuando se crean usuarios.

<div id="use-case-get-auth0-tokens-for-another-audience">
  ### Caso de uso: Obtener tokens de Auth0 para otra audiencia
</div>

GearUp quiere mejorar cómo autoriza las llamadas entre sus microservicios internos para procesar solicitudes de API. Quiere una política centralizada que controle los recursos que cada servicio puede consumir. Esto también puede resolverse mediante intercambio de tokens.

Cuando la solicitud de API llega por primera vez al servicio A, este intercambia el token de acceso recibido por uno nuevo que le permite consumir el servicio B como nueva audiencia. Si la política de autorización que rige el intercambio de tokens lo permite, el servicio A recibe el nuevo token y ahora puede consumir el servicio B. El ID de usuario se mantiene sin cambios en el nuevo token, por lo que el contexto de usuario adecuado se conserva durante todo el proceso.

<Frame>![](/docs/images/cdy7uua7fh8z/5Zw7yaJGct9eHAl4rdf72D/42274a5896851a16bea402ac52037f52/Screenshot_2025-02-03_at_5.17.14_PM.png)</Frame>

La aplicación GearUp ha obtenido inicialmente un token de acceso para consumir la API A en nombre de un usuario:

1. La aplicación envía la solicitud con el token de acceso inicial a la API A.
2. El servicio de backend de la API A valida el token de acceso y solicita intercambiarlo estableciéndolo como token de sujeto para obtener un nuevo token de acceso con el que consumir la API B.
3. Se ejecuta la Action correspondiente del perfil de Custom Token Exchange. Valida el token de acceso y obtiene del token el ID de usuario de Auth0. Luego aplica la política de autorización requerida y, por último, establece el usuario.
4. Auth0 responde con un token de acceso de Auth0 para consumir la audiencia de la API B.
5. El servicio de backend de la API A llama a la API B con el nuevo token de acceso, que sigue asociado al mismo usuario.

El siguiente código ejemplifica cómo implementar esto en la Action de Custom Token Exchange. En este caso:

* Usamos el ID de usuario de Auth0 para establecer el usuario, por lo que no es necesario definirlo en el alcance de ninguna conexión.
* No queremos crear ni actualizar al usuario.

Consulta [Validate JWTs signed with asymmetric keys](#validate-jwts-signed-with-asymmetric-keys) para ver ejemplos de código ampliados sobre este caso de uso.

<div id="use-case-perform-mfa-during-custom-token-exchange">
  ### Caso de uso: Realizar MFA durante Custom Token Exchange
</div>

Partiendo del [Caso de uso: Reutilizar un proveedor de autenticación externo](#use-case%3A-re-use-an-external-authentication-provider), GearUp ahora quiere confirmar que el usuario está presente cuando se utiliza un token del proveedor de autenticación externo. Esto es necesario para mitigar riesgos de seguridad, como el robo de tokens o escenarios en los que el autenticador externo no admite MFA. GearUp tiene dos opciones para lograrlo: implementar una política de MFA para toda la organización o activar MFA mediante programación con una Action de Post Login.

El siguiente ejemplo utiliza una Action de Post Login para activar la autenticación MFA durante la transacción de Custom Token Exchange. Para obtener más información sobre el uso de la concesión de MFA a través de APIs embebidas, consulte la [documentación](/docs/es/secure/multi-factor-authentication/authenticate-using-ropg-flow-with-mfa) sobre el uso de ROPG con MFA, ya que Custom Token Exchange sigue el mismo modelo.

Primero, defina la Action con `api.multifactor.enable()` para activar un desafío MFA. Esta función se describe en la [documentación de la API de Post Login](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger/post-login-api-object).

```js lines
exports.onExecutePostLogin = async (event, api) => {
  api.multifactor.enable('any');
};

Ahora podemos enviar una solicitud de intercambio de tokens:

curl --location 'https://{yourDomain}/oauth/token' \
--header 'Content-Type: application/x-www-form-urlencoded' \
--data-urlencode 'grant_type=urn:ietf:params:oauth:grant-type:token-exchange' \
--data-urlencode 'audience=https://api.gearup.com' \
--data-urlencode 'scopes=openid offline_access gearup-scope1 gearup-scope2' \
--data-urlencode 'subject_token_type=urn:gearup:external-idp' \
--data-urlencode 'subject_token=t8e7S2D9trQm73e .... iqBR3GjxDtbDVjpfQU' \
--data-urlencode 'client_id=<YOUR_CLIENT_ID>' \
--data-urlencode 'client_secret=<YOUR_CLIENT_SECRET>'
```

Esto da como resultado un error `mfa_required` que devuelve un token de MFA:

```json lines
403 Forbidden
{
  "error": "mfa_required",
  "error_description": "Multifactor authentication required",
  "mfa_token": "<YOUR_MFA_TOKEN>"
}
```

Con el `mfa_token` devuelto, la aplicación puede llamar a la API de MFA para iniciar la comprobación y verificar un factor:

Primero, obtenga una lista de autenticadores:

```bash lines
curl --location 'https://{yourDomain}.auth0.com/mfa/authenticators' \
--header 'Authorization: Bearer <YOUR_MFA_TOKEN>' \


[
    {
        "id": "sms|dev_1MHoE3huPRB5dcDJ",
        "authenticator_type": "oob",
        "active": true,
        "oob_channel": "sms",
        "name": "XXXXXXXX6220"
    },
    {
        "id": "email|dev_QLGL8cGsvFFnOloK",
        "authenticator_type": "oob",
        "active": true,
        "oob_channel": "email",
        "name": "dloz********@gmai*****"
    }
]
```

En segundo lugar, usa el ID del autenticador para activar un desafío:

```bash lines
curl --location 'https://{yourDomain}.auth0.com/mfa/challenge' \
--header 'Content-Type: application/x-www-form-urlencoded' \
--data-urlencode 'client_id=<YOUR_CLIENT_ID>' \
--data-urlencode 'client_secret=<YOUR_CLIENT_SECRET>'
--data-urlencode 'mfa_token=<YOUR_MFA_TOKEN>' \
--data-urlencode 'authenticator_id=sms|dev_1MHoE3huPRB5dcDJ' \
--data-urlencode 'challenge_type=oob'
```

La solicitud de verificación genera la siguiente respuesta:

```json lines
{
  "challenge_type": "oob",
  "oob_code": "<YOUR_OOB_CODE>",
  "binding_method": "prompt"
}
```

Usa `mfa_token` y `oob_code` (si se proporcionan) para completar el proceso de verificación mediante el endpoint de token y recibir tokens:

```bash lines
curl --location 'https://{yourDomain}.auth0.com/oauth/token' \
--header 'Content-Type: application/x-www-form-urlencoded' \
--data-urlencode 'grant_type=http://auth0.com/oauth/grant-type/mfa-oob' \
--data-urlencode 'mfa_token=<YOUR_MFA_TOKEN>' \
--data-urlencode 'oob_code=<YOUR_OOB_CODE>' \
--data-urlencode 'binding_code=<YOUR_USER_CODE>'
```

<div id="use-case-support-agent-acting-on-behalf-of-an-end-user">
  ### Caso de uso: Agente de soporte que actúa en nombre de un usuario final
</div>

Los agentes de soporte de GearUp necesitan acceder a los datos del usuario final y realizar acciones a través de las API de backend de GearUp en nombre del usuario final. La herramienta de soporte autentica al agente y luego usa Custom Token Exchange para obtener un token de acceso que represente al usuario final, con el agente registrado como actor.

<Frame>![](/docs/images/custom-token-exchange/Support-agent-acting-on-behalf-of-an-end-user.png)</Frame>

En este caso, el token de identidad de Auth0 del agente se envía como `actor_token` en la solicitud, y un JWT firmado que identifica al usuario final se envía como `subject_token`. Cuando `actor_token_type` se establece en `urn:ietf:params:oauth:token-type:id_token`, Auth0 valida automáticamente el token (firma, vencimiento, emisor) y rellena `event.transaction.actor_token_user` con el perfil del agente. Esto elimina la necesidad de usar código de validación personalizado para el token de actor.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  No es obligatorio usar un token de identidad de Auth0 como `actor_token`. Cuando `actor_token_type` es un valor personalizado, la Action debe validar el token de actor mediante código personalizado, de forma similar a como se validan los tokens de sujeto. El rellenado automático de `event.transaction.actor_token_user` solo se aplica a los tokens de identidad de Auth0.
</Callout>

1. La herramienta de soporte autentica al agente con Auth0 y obtiene su token de identidad.
2. La herramienta de soporte llama al endpoint `/oauth/token` de Auth0 mediante una [solicitud de Custom Token Exchange](/docs/es/get-started/authentication-and-authorization-flow/token-exchange-flow/call-your-api-using-the-custom-token-exchange-flow), que incluye un JWT firmado con el identificador del usuario final como `subject_token` y el token de identidad del agente como `actor_token`.
3. La [Action de Custom Token Exchange](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/custom-token-exchange-trigger) valida el token de sujeto, verifica que el actor tenga derecho a actuar en nombre del usuario final y llama a `api.authentication.setActor()`.
4. Auth0 emite tokens con el claim `act` que identifica al agente de soporte.
5. El agente de soporte consume la API en nombre del usuario final. Las API pueden inspeccionar el claim `act` para aplicar políticas de autorización específicas para el acceso delegado, como restringir operaciones de escritura o registrar actividad con fines de auditoría.

La Action de Custom Token Exchange decide qué incluir en el objeto actor, incluidas las propiedades personalizadas y los niveles de anidación. Consulta la [documentación del objeto de API de Custom Token Exchange](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/custom-token-exchange-trigger/custom-token-exchange-api-object) para conocer las restricciones.

<div id="important-considerations-for-delegated-authorization">
  ### Consideraciones importantes para la autorización delegada
</div>

Al implementar la autorización delegada con Custom Token Exchange, sigue estas pautas:

* **Implementa la lógica de autorización dentro de tu Action de Custom Token Exchange** para verificar que el actor esté autorizado para acceder a la cuenta de usuario específica. Por ejemplo, quizá quieras definir reglas de autorización para que solo determinados actores puedan realizar acceso delegado, o podrías comprobar que el usuario de destino tenga un ticket de soporte activo para evitar el acceso arbitrario a usuarios.

* **Valida los scopes solicitados** para garantizar que solo se emita el conjunto mínimo de scopes necesarios para la autorización delegada. Quizá quieras asegurarte de que ciertas operaciones sensibles nunca puedan realizarse en el contexto de la autorización delegada.

* **Asegúrate de que tus APIs utilicen el contexto de delegación** del claim `act` del token de acceso. Debes mantener registros de auditoría en tus APIs sobre las acciones realizadas por un actor delegado y asegurarte de poder auditar claramente qué actor realizó operaciones en nombre del usuario.

* **Con fines de auditoría**, puedes usar los detalles del actor en los registros del tenant de Auth0. Las transacciones exitosas de Custom Token Exchange (eventos de log `secte`) incluyen la propiedad `actor` con `sub` y cualquier información `actor` anidada.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Auth0 no notifica al usuario final cuando se emite en su nombre un token de autorización delegada. Si tu caso de uso requiere notificar al usuario u obtener su consentimiento explícito antes de que se produzca el acceso delegado, considera usar [Client Initiated Backchannel Authentication (CIBA)](/docs/es/get-started/authentication-and-authorization-flow/client-initiated-backchannel-authentication-ciba-flow) para enviar una solicitud de consentimiento al dispositivo del usuario final antes de realizar el intercambio de tokens. Para necesidades de notificación más simples, puedes implementar la lógica de notificación dentro de tu Action de Custom Token Exchange, en una Action de Post Login o en tus servicios de destino.
</Callout>

<div id="code-samples">
  ## Ejemplos de código
</div>

Los siguientes ejemplos de código muestran prácticas recomendadas para escenarios comunes de validación segura y eficiente de tokens de sujeto entrantes.

Usa algoritmos y claves asimétricos siempre que puedas, ya que así no necesitas compartir ningún secreto con Auth0. Esto también simplifica la rotación de claves, por ejemplo, al exponer un endpoint URI de JWKS para publicar las claves públicas correspondientes.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Es tu responsabilidad garantizar que los tokens de sujeto estén protegidos con un algoritmo robusto y con claves o secretos con suficiente entropía.
</Callout>

<div id="validate-jwts-signed-with-asymmetric-keys">
  ### Validar JWT firmados con claves asimétricas
</div>

Ten en cuenta las siguientes recomendaciones:

* Usa los métodos de Actions [`api.cache ()`](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/custom-token-exchange-trigger/custom-token-exchange-api-object) para evitar tener que obtener las claves de firma en cada transacción.
* Sigue las prácticas recomendadas de [RFC8725](https://www.rfc-editor.org/rfc/rfc8725.txt)
* Usa algoritmos RS*, PS*, ES* o Ed25519
* No uses ni aceptes el algoritmo none
* Usa RSA con una longitud mínima de 2048 bits.

```javascript lines expandable
const { jwtVerify, importJWK } = require("jose");

const jwksUri = "https://example.com/.well-known/jwks.json";
const fetchTimeout = 5000; // 5 segundos

const validIssuer = "urn:my-issuer"; // Reemplaza con tu emisor

/**
 * Controlador que se ejecuta al procesar una solicitud de intercambio de token personalizado
 * @param {Event} event - Detalles sobre la solicitud de intercambio de tokens entrante.
 * @param {CustomTokenExchangeAPI} api - Métodos y utilidades para definir el proceso de intercambio de tokens.
 */
exports.onExecuteCustomTokenExchange = async (event, api) => {
  const { isValid, payload } = await validateToken(
    event.transaction.subject_token,
  );

  // Aplica tu política de autorización según sea necesario para determinar si la solicitud es válida.
  // Usa api.access.deny() para rechazar la transacción en esos casos.

  if (!isValid) {
    // Marca el token de sujeto como inválido y falla la transacción.
    api.access.rejectInvalidSubjectToken("Invalid subject_token");
  } else {
    // Establece el usuario de la solicitud actual como autenticado, usando el user ID del token de sujeto.
    api.authentication.setUserById(payload.sub);
  }

  /**
   * Valida el token de sujeto
   * @param {string} subjectToken
   * @returns {Promise<{ isValid: boolean, payload?: object }>} Payload del token
   */
  async function validateToken(subjectToken) {
    try {
      const { payload, protectedHeader } = await jwtVerify(
        subjectToken,
        async (header) => await getPublicKey(header.kid),
        {
          issuer: validIssuer,
        },
      );

      // Realiza validaciones adicionales sobre el payload del token según sea necesario

      return { isValid: true, payload };
    } catch (/** @type {any} */ error) {
      if (error.message === "Error fetching JWKS") {
        throw new Error("Internal error - retry later");
      } else {
        console.log("Token validation failed:", error.message);
        return { isValid: false };
      }
    }
  }

  /**
   * Obtiene la clave pública para la verificación. La carga desde la caché de Actions si está disponible;
   * de lo contrario, la obtiene desde el endpoint JWKS y la almacena en la caché.
   * @param {string} kid - kid (ID de la clave) de la clave que se usará para la verificación
   * @returns {Promise<Object>}
   */
  async function getPublicKey(kid) {
    const cachedKey = api.cache.get(kid);
    let keyData;

    if (!cachedKey) {
      console.log(`Clave ${kid} no encontrada en caché`);
      keyData = await fetchKeyFromJWKS(kid);
      // Almacena en caché la versión serializada
      api.cache.set(kid, JSON.stringify(keyData), { ttl: 600000 });
    } else {
      // Parsea el objeto JWK sin procesar desde la caché
      keyData = JSON.parse(cachedKey.value);
    }

    //Convierte el objeto JWK sin procesar a un objeto KeyLike
    return await importJWK(keyData, keyData.alg);
  }

  /**
   * Obtiene la clave de firma pública desde el endpoint JWKS proporcionado, para usarla en la verificación de tokens
   * @param {string} kid - kid (ID de la clave) de la clave que se usará para la verificación
   * @returns {Promise<object>}
   */
  async function fetchKeyFromJWKS(kid) {
    const controller = new AbortController();
    setTimeout(() => controller.abort(), fetchTimeout);

    /** @type {any} */
    const response = await fetch(jwksUri);

    if (!response.ok) {
      console.log(`Error al obtener JWKS. Estado de la respuesta: ${response.status}`);
      throw new Error("Error fetching JWKS");
    }
    const jwks = await response.json();
    const key = jwks.keys.find((key) => key.kid === kid);
    if (!key) {
      throw new Error("Key not found in JWKS");
    }
    return key;
  }
};
```

<div id="validate-jwts-signed-with-symmetric-keys">
  ### Validar JWT firmados con claves simétricas
</div>

Considere las siguientes recomendaciones:

* Use [Secrets de Actions](/docs/es/customize/actions/write-your-first-action#add-a-secret) para almacenar de forma segura sus secretos simétricos.
* Siga las prácticas recomendadas de [RFC8725](https://www.rfc-editor.org/rfc/rfc8725.txt)
* Use algoritmos seguros, como HS256, junto con secretos aleatorios de alta entropía (por ejemplo, de al menos 256 bits de longitud)

```javascript lines expandable
const { jwtVerify } = require("jose");

const validIssuer = "urn:my-issuer"; // Reemplaza con tu emisor

/**
 * Controlador que se ejecuta al procesar una solicitud de intercambio de token personalizado
 * @param {Event} event - Detalles sobre la solicitud de intercambio de tokens entrante.
 * @param {CustomTokenExchangeAPI} api - Métodos y utilidades para definir el proceso de intercambio de tokens.
 */
exports.onExecuteCustomTokenExchange = async (event, api) => {
  // Inicializa la clave simétrica compartida desde los Secrets de Actions
  const encoder = new TextEncoder();
  const symmetricKey = encoder.encode(event.secrets.SHARED_SECRET);

  const { isValid, payload } = await validateToken(
    event.transaction.subject_token,
    symmetricKey,
  );

  // Aplica tu política de autorización según sea necesario para determinar si la solicitud es válida.
  // Usa api.access.deny() para rechazar la transacción en esos casos.

  if (!isValid) {
    // Marca el token de sujeto como inválido y falla la transacción.
    api.access.rejectInvalidSubjectToken("Invalid subject_token");
  } else {
    // Establece el usuario de la solicitud actual como autenticado, usando el user ID del token de sujeto.
    api.authentication.setUserById(payload.sub);
  }
};

/**
 * Valida el token de sujeto
 * @param {string} subjectToken
 * @param {Uint8Array} symmetricKey
 * @returns {Promise<{ isValid: boolean, payload?: object }>} Payload del token
 */
async function validateToken(subjectToken, symmetricKey) {
  try {
    // Valida que el token esté correctamente firmado con la clave simétrica compartida
    // También verifica que no haya expirado, siempre que incluya un atributo 'exp'.
    const { payload, protectedHeader } = await jwtVerify(
      subjectToken,
      symmetricKey,
      {
        issuer: validIssuer,
      },
    );

    return { isValid: true, payload };
  } catch (/** @type {any} */ error) {
    console.log("Token validation failed:", error.message);
    return { isValid: false };
  }
}
```

<div id="validate-opaque-token-with-an-external-service">
  ### Validar token opaco con un servicio externo
</div>

Usa [Action Secrets](/docs/es/customize/actions/write-your-first-action#add-a-secret) para almacenar de forma segura el <Tooltip tip="secreto de cliente: secreto que usa un cliente (aplicación) para autenticarse con el servidor de autorización; solo deben conocerlo el cliente y el servidor de autorización, y debe ser lo suficientemente aleatorio como para que no pueda adivinarse." cta="Ver glosario" href="/docs/es/glossary?term=client+secret">secreto de cliente</Tooltip> de tu IdP externo.

```javascript lines expandable
const tokenEndpoint = "EXTERNAL_TOKEN_ ENDPOINT";
const userInfoEndpoint = "EXTERNAL_USER_INFO_ENDPOINT";
const clientId = "EXTERNAL_CLIENT_ID";
const connectionName = "YOUR_CONNECTION_NAME";
const fetchTimeout = 5000; // 5 segundos

/**
 * Controlador que se ejecuta al procesar una solicitud de intercambio de token personalizado
 * @param {Event} event - Detalles sobre la solicitud de intercambio de tokens entrante.
 * @param {CustomTokenExchangeAPI} api - Métodos y utilidades para definir el proceso de intercambio de tokens.
 */
exports.onExecuteCustomTokenExchange = async (event, api) => {
  const { isValid, user } = await getUserProfile(
    event.transaction.subject_token,
    event.secrets.CLIENT_SECRET,
  );

  if (!isValid) {
    // Marcar el token de sujeto como inválido y hacer fallar la transacción.
    api.access.rejectInvalidSubjectToken("Invalid subject_token");
    return;
  }

  // Aplica tu política de autorización según sea necesario para determinar si la solicitud es válida.
  // Usa api.access.deny() para rechazar la transacción en esos casos.

  // Cuando tenemos el perfil, establecemos el usuario en la conexión de destino
  api.authentication.setUserByConnection(
    connectionName,
    {
      // solo se necesita el user_id en la conexión, ya que no estamos
      // creando ni actualizando el usuario
      user_id: user.sub,
    },
    {
      creationBehavior: "none",
      updateBehavior: "none",
    },
  );
};

/**
 * Intercambia el token de actualización y carga el perfil de usuario desde el IdP heredado
 * @param {string} refreshToken
 * @param {string} clientSecret
 * @returns {Promise<{ isValid: boolean, user?: object }>} Si el token de actualización se intercambió correctamente, devuelve el perfil de usuario
 */
async function getUserProfile(refreshToken, clientSecret) {
  const { isValid, accessToken } = await refreshAccessToken(
    refreshToken,
    clientSecret,
  );
  if (!isValid) {
    return { isValid: false };
  }

  const controller = new AbortController();
  setTimeout(() => controller.abort(), fetchTimeout);

  /** @type {any} */
  const response = await fetch(userInfoEndpoint, {
    method: "GET",
    headers: {
      Authorization: `Bearer ${accessToken}`,
      "Content-Type": "application/json",
    },
  });

  if (!response.ok) {
    console.log(`Failed to fetch user info. Status: ${response.status}`);
    throw new Error("Error fetching user info");
  }

  const userProfile = await response.json();

  return { isValid: true, user: userProfile };
}

/**
 * Usa el token de actualización con el IdP heredado para validarlo y obtener un access token
 * @param {string} refreshToken
 * @param {string} clientSecret
 * @returns {Promise<{ isValid: boolean, accessToken?: string }>} Si el token de actualización se intercambió correctamente, devuelve el access token
 */
async function refreshAccessToken(refreshToken, clientSecret) {
  const controller = new AbortController();
  setTimeout(() => controller.abort(), fetchTimeout);

  /** @type {any} */
  let response;

  try {
    response = await fetch(tokenEndpoint, {
      method: "POST",
      headers: {
        "Content-Type": "application/x-www-form-urlencoded",
      },
      body: new URLSearchParams({
        grant_type: "refresh_token",
        refresh_token: refreshToken,
        client_id: clientId,
        client_secret: clientSecret,
      }).toString(),
    });
  } catch (error) {
    console.error("Error refreshing token");
    throw error;
  }

  if (!response.ok) {
    const errorBody = await response.json();
    console.error("Error refreshing token:", errorBody.error);

    // Si recibimos un error que indica que el token de actualización es inválido (por ejemplo, un error invalid_grant),
    // debemos indicar explícitamente un token inválido usando api.access.rejectInvalidSubjectToken
    // para protegerse contra ataques de fuerza bruta sobre el token de actualización activando Suspicious IP Throttling.
    // Para otros errores que indican un fallo genérico al realizar la solicitud al IdP, debemos lanzar
    // un error para indicar un fallo transitorio.
    if (errorBody.error === "invalid_grant") {
      return { isValid: false };
    } else {
      throw new Error("Error refreshing token");
    }
  }

  // Analiza la respuesta, con la forma { access_token: "...", expires_in: ..., }
  const data = await response.json();
  console.log("Successfully exchanged refresh token");
  return { isValid: true, accessToken: data.access_token };
}
```
