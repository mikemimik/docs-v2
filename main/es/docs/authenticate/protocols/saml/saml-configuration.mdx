<div id="saml-service-providers">
  ## Proveedores de servicios SAML
</div>

Las aplicaciones, especialmente las personalizadas, pueden autenticar a los usuarios mediante un <Tooltip tip="Proveedor de identidad (IdP): servicio que almacena y administra identidades digitales." cta="Ver glosario" href="/es/docs/glossary?term=IdP">IdP</Tooltip> externo usando protocolos como <Tooltip tip="OpenID: estándar abierto de autenticación que permite a las aplicaciones verificar las identidades de los usuarios sin recopilar ni almacenar información de inicio de sesión." cta="Ver glosario" href="/es/docs/glossary?term=OpenID">OpenID</Tooltip> Connect (OIDC) u <Tooltip tip="OpenID: estándar abierto de autenticación que permite a las aplicaciones verificar las identidades de los usuarios sin recopilar ni almacenar información de inicio de sesión." cta="Ver glosario" href="/es/docs/glossary?term=OAuth+2.0">OAuth 2.0</Tooltip>. Sin embargo, puede que quieras usar un proveedor <Tooltip tip="Security Assertion Markup Language (SAML): protocolo estandarizado que permite a dos partes intercambiar información de autenticación sin una contraseña." cta="Ver glosario" href="/es/docs/glossary?term=SAML">SAML</Tooltip> empresarial para la autenticación, aunque hayas desarrollado tu aplicación para usar cualquiera de estos dos protocolos.

<div id="saml-identity-providers">
  ## proveedores de identidad SAML
</div>

Algunas aplicaciones (como Salesforce, Box y Workday) permiten a los usuarios autenticarse mediante un IdP externo usando el protocolo SAML. Luego puedes integrar la aplicación con Auth0, que actúa como el IdP de SAML de la aplicación. Los usuarios de la aplicación serán redirigidos a Auth0 para iniciar sesión, y Auth0 puede autenticarlos mediante cualquier conexión de autenticación de backend, como un directorio LDAP, una base de datos u otro IdP de SAML o proveedor social. Una vez autenticado el usuario, Auth0 devuelve a la aplicación una aserción SAML que así lo indica.

<div id="auth0-as-service-provider">
  ## Auth0 como proveedor de servicios
</div>

Si Auth0 actúa como proveedor de servicios en una federación SAML, puede enrutar solicitudes de autenticación a un proveedor de identidad sin necesidad de tener una cuenta creada de antemano para un usuario específico. A partir de la aserción devuelta por el proveedor de identidad, Auth0 puede recopilar la información necesaria para crear un perfil de usuario para ese usuario (a este proceso a veces se le llama aprovisionamiento just-in-time). Para obtener más información, lee [Select from Multiple Connection Options](/es/docs/libraries/lock/selecting-from-multiple-connection-options).

Aunque Auth0 no requiere cuentas de usuario creadas previamente antes del proceso de autenticación, la aplicación integrada con Auth0 podría requerirlas. Si este es el caso, tienes varias opciones para gestionarlo:

* Después de que el proveedor de identidad cree al usuario, puedes usar un proceso fuera de banda para crear el usuario correspondiente en la aplicación (o en Auth0) y añadir los atributos del perfil de usuario que la aplicación necesite. Si, después de la autenticación, falta algún atributo en el perfil, la aplicación puede obtenerlo de la fuente adecuada y almacenarlo en el perfil de usuario de Auth0. La próxima vez que el usuario inicie sesión, esos atributos adicionales se enviarán a la aplicación (además de los que haya agregado el proveedor de identidad).
* Puedes usar una rule de Auth0 para llamar a una API, recuperar cualquier información que falte y añadirla dinámicamente al perfil de Auth0 (que luego se devuelve a la aplicación). Las Rules se ejecutan después de una autenticación correcta, y tu aplicación puede recuperar los atributos del perfil cada vez o puedes guardar los atributos en el perfil de Auth0.
* Auth0 puede pasar a la aplicación la información básica del perfil desde el proveedor de identidad, y luego la aplicación recupera cualquier dato faltante de otra fuente. Con estos dos conjuntos de información, la aplicación crea un perfil de usuario local.

Puedes especificar dominios de correo electrónico como parte de la configuración de la conexión SAMLP de Auth0 para controlar el IdP que gestiona a un grupo determinado de usuarios. Por ejemplo, si agregas el dominio de correo electrónico `example.com` a la configuración de la conexión SAMLP de Auth0 para la Empresa X, todos los usuarios con direcciones de correo electrónico del dominio `example.com` serán gestionados por el IdP específico de la Empresa X.

<div id="auth0-as-identity-provider">
  ## Auth0 como proveedor de identidad
</div>

Si Auth0 actúa como proveedor de identidad en una federación SAML, las cuentas de usuario pueden crearse de varias formas:

* Usando un sistema de autenticación de backend, como un directorio LDAP, una base de datos u otro proveedor de identidad SAML.
* Usando el <Tooltip tip="Auth0 Dashboard: El producto principal de Auth0 para configurar sus servicios." cta="Ver glosario" href="/es/docs/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip>.
* Llamando a la <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/es/docs/glossary?term=Management+API">Management API</Tooltip> de Auth0.
* Implementando el registro de usuarios por autoservicio.

Si la aplicación está diseñada para obtener información del perfil de usuario desde un almacén local, deberá crear el perfil local después de que las cuentas se hayan creado en Auth0. Algunas formas de hacerlo incluyen:

* Un proceso fuera de banda que cree perfiles de usuario en la aplicación;
* Una rule de Auth0 que se ejecute en el primer inicio de sesión y llame a una API de la aplicación para crear el perfil de usuario en ella;
* Modificar la aplicación para crear perfiles de usuario de forma dinámica, basándose en la información de la aserción SAML.
