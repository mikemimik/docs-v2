<Card title="Overview">
  Conceptos clave

  * Auth0 es compatible con el protocolo OAuth 2.0 elaborado por el Internet Engineering Task Force (IETF).
  * Conoce los roles, los tipos de concesión (o flujos de trabajo) y los endpoints de la especificación OAuth 2.0.
</Card>

El marco de autorización [OAuth 2.0](https://tools.ietf.org/html/rfc6749) es un protocolo que permite a un usuario otorgar a un sitio web o aplicación de terceros acceso a sus recursos protegidos, sin necesidad de revelar sus credenciales a largo plazo ni su identidad.

<Tooltip tip="OAuth 2.0: Marco de autorización que define protocolos y flujos de trabajo de autorización." cta="Ver glosario" href="/es/docs/glossary?term=OAuth">
  OAuth
</Tooltip>

introduce una capa de autorización y separa el rol del cliente del

<Tooltip tip="Propietario del recurso: Entidad (como un usuario o una aplicación) capaz de otorgar acceso a un recurso protegido." cta="Ver glosario" href="/es/docs/glossary?term=resource+owner">
  propietario del recurso
</Tooltip>

. En OAuth, el cliente solicita acceso a los recursos controlados por el propietario del recurso y alojados en el

<Tooltip tip="Servidor de recursos: Servidor que aloja recursos protegidos. Los servidores de recursos aceptan y responden a las solicitudes de recursos protegidos." cta="Ver glosario" href="/es/docs/glossary?term=resource+server">
  servidor de recursos
</Tooltip>

y recibe un conjunto de credenciales distinto al del propietario del recurso. En lugar de usar las credenciales del propietario del recurso para acceder a los recursos protegidos, el cliente obtiene un

<Tooltip tip="Token de acceso: Credencial de autorización, en forma de cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/es/docs/glossary?term=access+token">
  token de acceso
</Tooltip>

, una cadena que indica un scope específico, un tiempo de vida y otros atributos de acceso. Los tokens de acceso son emitidos a clientes de terceros por un

<Tooltip tip="Servidor de autorización: Servidor centralizado que contribuye a definir los límites del acceso de un usuario. Por ejemplo, tu servidor de autorización puede controlar los datos, las tareas y las funciones disponibles para un usuario." cta="Ver glosario" href="/es/docs/glossary?term=authorization+server">
  servidor de autorización
</Tooltip>

con la aprobación del propietario del recurso. Luego, el cliente usa el token de acceso para acceder a los recursos protegidos alojados en el servidor de recursos.

Auth0 genera tokens de acceso para escenarios de autorización de API en [formato JSON web token (JWT)](/es/docs/secure/tokens/json-web-tokens/json-web-token-structure). Los permisos representados por el token de acceso, en términos de OAuth, se conocen como [scopes](/es/docs/get-started/apis/scopes). Cuando una aplicación se autentica con Auth0, especifica los scopes que desea. Si el usuario autoriza esos scopes, el token de acceso representará dichos scopes autorizados.

<div id="roles">
  ## Roles
</div>

Un flujo de OAuth 2.0 tiene los siguientes roles:

* **Propietario del recurso**: Entidad que puede conceder acceso a un recurso protegido. Por lo general, es el usuario final.
* **Servidor de recursos**: Servidor que aloja los recursos protegidos. Esta es la API a la que quieres acceder.
* **Cliente**: Aplicación que solicita acceso a un recurso protegido en nombre del propietario del recurso.
* **Servidor de autorización**: Servidor que autentica al propietario del recurso y emite tokens de acceso después de obtener la autorización correspondiente. En este caso, Auth0.

<div id="grant-types">
  ## Tipos de concesión
</div>

OAuth 2.0 define cuatro flujos para obtener un token de acceso. Estos flujos se denominan tipos de concesión. [Decidir cuál es el más adecuado para su caso](/es/docs/get-started/authentication-and-authorization-flow/which-oauth-2-0-flow-should-i-use) depende principalmente del tipo de aplicación.

* [Flujo de código de autorización](/es/docs/get-started/authentication-and-authorization-flow/authorization-code-flow): lo utilizan las aplicaciones web que se ejecutan en un servidor. También lo utilizan las aplicaciones móviles, mediante la [técnica Proof Key for Code Exchange (PKCE)](/es/docs/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce).
* [Flujo implícito con envío de formulario](/es/docs/get-started/authentication-and-authorization-flow/implicit-flow-with-form-post): lo utilizan las aplicaciones basadas principalmente en JavaScript (aplicaciones de una sola página) que se ejecutan en el navegador del usuario.
* [Flujo de contraseña del propietario del recurso](/es/docs/get-started/authentication-and-authorization-flow/resource-owner-password-flow): lo utilizan aplicaciones de alta confianza.
* [Flujo de credenciales del cliente](/es/docs/get-started/authentication-and-authorization-flow/client-credentials-flow): se utiliza para la comunicación entre máquinas.

La especificación también proporciona un mecanismo de extensibilidad para definir tipos de concesión adicionales. Para obtener más información sobre cómo funciona cada tipo de concesión y cuándo debe utilizarse, consulte [Flujos de autenticación y autorización](/es/docs/get-started/authentication-and-authorization-flow).

<div id="endpoints">
  ## Endpoints
</div>

OAuth 2.0 usa dos endpoints: el endpoint `/authorize` y el endpoint `/oauth/token`.

<div id="authorization-endpoint">
  ### Endpoint de autorización
</div>

El endpoint `/authorize` se utiliza para interactuar con el propietario del recurso y obtener autorización para acceder al recurso protegido. Para entenderlo mejor, imagina que quieres iniciar sesión en un servicio con tu cuenta de Google. Primero, el servicio te redirige a Google para autenticarte (si aún no has iniciado sesión) y, después, verás una pantalla de consentimiento en la que se te pedirá autorizar al servicio para acceder a algunos de tus datos (recursos protegidos); por ejemplo, tu dirección de correo electrónico y tu lista de contactos.

Los parámetros de solicitud del endpoint `/authorize` son:

| Parámetro       | Descripción                                                                                                                                                                                                                                                                                                                                                                                                                                                                        |
| --------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `response_type` | Indica al servidor de autorización qué concesión debe ejecutar.                                                                                                                                                                                                                                                                                                                                                                                                                    |
| `response_mode` | (Opcional) Cómo se da formato al resultado de la solicitud de autorización. Valores:<br />- `query`: para la concesión Authorization Code. `302 Found` desencadena una redirección.<br />- `fragment`: para la concesión Implicit. `302 Found` desencadena una redirección.<br />- `form_post`: `200 OK` con los parámetros de respuesta incrustados en un formulario HTML como parámetros ocultos.<br />- `web_message`: para Silent Authentication. Usa mensajería web de HTML5. |
| `client_id`     | El ID de la aplicación que solicita autorización.                                                                                                                                                                                                                                                                                                                                                                                                                                  |
| `redirect_uri`  | Contiene una URL. Una respuesta correcta de este endpoint da lugar a una redirección a esta URL.                                                                                                                                                                                                                                                                                                                                                                                   |
| `scope`         | Una lista de permisos separada por espacios que requiere la aplicación.                                                                                                                                                                                                                                                                                                                                                                                                            |
| `state`         | Un valor opaco que se utiliza con fines de seguridad. Si este parámetro de solicitud se establece en la solicitud, se devuelve a la aplicación como parte de la `redirect_uri`.                                                                                                                                                                                                                                                                                                    |
| `connection`    | Especifica el tipo de conexión para las conexiones Passwordless                                                                                                                                                                                                                                                                                                                                                                                                                    |

Puedes configurar parámetros de consulta personalizados cuando tu aplicación hace la llamada inicial al endpoint `/authorize` para autenticar a un usuario. Puedes usar parámetros de consulta personalizados para proporcionar contexto adicional a la plantilla de página de la experiencia de <Tooltip tip="Universal Login: Tu aplicación redirige a Universal Login, alojado en el servidor de autorización de Auth0, para verificar la identidad de un usuario." cta="Ver glosario" href="/es/docs/glossary?term=Universal+Login">Universal Login</Tooltip>.

Debes habilitar ID First para usar el parámetro `connection`. Para obtener más información sobre el parámetro `connection` y la experiencia de Universal Login, consulta [Passwordless for Universal Login](/es/docs/authenticate/passwordless/passwordless-with-universal-login).

Los parámetros de consulta con el prefijo `ext-` aparecen automáticamente en el [contexto de la plantilla de página](/es/docs/customize/login-pages/universal-login/customize-templates#custom-query-parameters).

Este endpoint se utiliza en los tipos de concesión Authorization Code e Implicit. El servidor de autorización necesita saber qué tipo de concesión quiere usar la aplicación, ya que esto afecta al tipo de credencial que emitirá:

* En la concesión Authorization Code, emitirá un código de autorización (que más tarde puede intercambiarse por un token de acceso en el endpoint `/oauth/token`).
* En la concesión Implicit, emitirá un token de acceso, que es una cadena opaca (o un <Tooltip tip="JSON Web Token (JWT): Formato estándar de ID Token (y, a menudo, también de Access Token) usado para representar claims de forma segura entre dos partes." cta="Ver glosario" href="/es/docs/glossary?term=JWT">JWT</Tooltip> en una implementación de Auth0) que indica quién ha autorizado qué permisos (scopes) para qué aplicación.

Para indicar al servidor de autorización qué tipo de concesión debe usar, el parámetro de solicitud `response_type` se utiliza de la siguiente manera:

* Para la concesión Authorization Code, usa `response_type=code` para incluir el código de autorización.
* Para la concesión Implicit, usa `response_type=token` para incluir un token de acceso. Otra opción es usar `response_type=id_token token` para incluir tanto un token de acceso como un <Tooltip tip="ID Token: Credencial destinada al propio cliente, en lugar de para acceder a un recurso." cta="Ver glosario" href="/es/docs/glossary?term=ID+token">ID token</Tooltip>.

Un ID token es un JWT que contiene información sobre el usuario que ha iniciado sesión. Fue introducido por <Tooltip tip="OpenID: Estándar abierto para autenticación que permite a las aplicaciones verificar la identidad de los usuarios sin recopilar ni almacenar información de inicio de sesión." cta="Ver glosario" href="/es/docs/glossary?term=OpenID">OpenID</Tooltip> Connect (OIDC).

La especificación [OAuth 2.0 Multiple Response Type Encoding Practices](https://openid.net/specs/oauth-v2-multiple-response-types-1_0.html) añadió un parámetro que especifica cómo se da formato al resultado de la solicitud de autorización. Este parámetro se llama `response_mode`. Es opcional y puede tomar los siguientes valores:

| Valor         | Descripción                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                       |
| ------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `query`       | Este es el valor predeterminado para la concesión Authorization Code. Una respuesta correcta es `302 Found`, lo que desencadena una redirección a `redirect_uri`. Los parámetros de la respuesta se incluyen en el componente de consulta (la parte después de `?`) de `redirect_uri`, en el encabezado `Location`.<br />Por ejemplo:<br />`HTTP/1.1 302 Found`<br />`Location: https://my-redirect-uri.callback?code=js89p2x1`, donde el código de autorización es `js89p21`.                                                                                                        |
| `fragment`    | Este es el valor predeterminado para la concesión Implicit. Una respuesta correcta es `302 Found`, lo que desencadena una redirección a `redirect_uri` (que es un parámetro de la solicitud). Los parámetros de la respuesta se incluyen en el componente de fragmento (la parte después de `#`) de `redirect_uri`, en el encabezado `Location`.<br />Por ejemplo:<br />`HTTP/1.1 302 Found`<br />`Location: https://my-redirect-uri/callback#access_token=eyB...78f&token_type=Bearer&expires_in=3600`.                                                                              |
| `form_post`   | El modo de respuesta está definido por la [especificación OAuth 2.0 Form Post Response Mode](https://openid.net/specs/oauth-v2-form-post-response-mode-1_0.html). Una respuesta correcta es `200 OK` y los parámetros se incluyen en un formulario HTML como parámetros ocultos. La `action` del formulario es `redirect_uri` y el atributo `onload` está configurado para enviar el formulario. Después de que el navegador carga el HTML, se redirige a `redirect_uri`.                                                                                                         |
| `web_message` | Este modo de respuesta está definido en la [especificación OAuth 2.0 Web Message Response Mode](https://tools.ietf.org/html/draft-sakimura-oauth-wmrm-00). Utiliza HTML5 Web Messaging en lugar de la redirección para la respuesta de autorización del endpoint /authorization. Esto es especialmente útil cuando se usa Silent Authentication. Para usar este modo de respuesta, debes registrar la URL de tu aplicación en el campo **Allowed Web Origins** de la [configuración de la aplicación](https://manage.auth0.com/#/applications/\{yourClientId}/settings) de Auth0. |

<div id="token-endpoint">
  ### Endpoint de token
</div>

La aplicación utiliza el endpoint `/oauth/token` para obtener un token de acceso o un <Tooltip tip="Refresh Token: Token utilizado para obtener un nuevo token de acceso sin obligar a los usuarios a volver a iniciar sesión." cta="Ver glosario" href="/es/docs/glossary?term=refresh+token">token de actualización</Tooltip>. Se utiliza en todos los flujos excepto en el Flujo Implícito, ya que en ese caso el token de acceso se emite directamente.

* En el Flujo de Código de Autorización, la aplicación intercambia el código de autorización que obtuvo del endpoint de autorización por un token de acceso.
* En el Flujo de Credenciales de Cliente y en el grant de credenciales de contraseña del propietario del recurso, la aplicación se autentica con un conjunto de credenciales y luego obtiene un token de acceso.

<div id="state-parameters">
  ## Parámetros de estado
</div>

Los protocolos de autorización proporcionan un parámetro `state` que le permite restaurar el estado previo de su aplicación. El parámetro `state` conserva un objeto de estado establecido por el cliente en la solicitud de autorización y lo pone a disposición del cliente en la respuesta. La razón principal para usar el parámetro `state` es mitigar los ataques CSRF. Consulte [Uso de los parámetros de estado de OAuth 2.0](/es/docs/secure/attack-protection/state-parameters) para obtener más información.
