Las claves de acceso son una alternativa resistente al phishing frente a las formas tradicionales de authentication (como nombre de usuario y contraseña) y ofrecen una experiencia de usuario más sencilla y segura. Se basan en las [especificaciones](https://fidoalliance.org/specs/fido-v2.1-ps-20210615/fido-client-to-authenticator-protocol-v2.1-ps-errata-20220621.html#intro) FIDO® W3C Web Authentication (WebAuthn) y Client to Authenticator Protocol (CTAP).

Actualmente, Auth0 admite claves de acceso como método de authentication para conexiones de base de datos y ofrece dos métodos de implementación:

* [Claves de acceso de Universal Login](/es/docs/authenticate/database-connections/passkeys) para aplicaciones web.
* Claves de acceso nativas para aplicaciones Android e iOS.
* [Embedded Login para aplicaciones web y nativas](/es/docs/authenticate/passwordless/implement-login/embedded-login)

<div id="how-it-works">
  ## Cómo funciona
</div>

Las claves de acceso nativas usan una combinación de la [Auth0 Authentication API](/es/docs/authenticate/database-connections/passkeys/native-passkeys-api) y las API nativas de iOS o Android para integrar directamente en tu aplicación móvil los flujos de verificación. Esto te permite crear una experiencia integrada de registro e inicio de sesión para tu aplicación sin depender de redirigir a los usuarios a sus navegadores para completar la authentication.

El siguiente ejemplo muestra lo que puede experimentar un usuario nuevo durante el flujo de registro con clave de acceso:

1. Un usuario nuevo abre tu aplicación móvil y accede a la pantalla de inicio de sesión. Como es un usuario nuevo, selecciona el botón Registrarse.
2. En la siguiente pantalla, el usuario introduce su dirección de correo electrónico y selecciona Crear cuenta.
3. A continuación, se le pregunta al usuario si quiere crear una clave de acceso para tu aplicación. Para continuar, el usuario selecciona Continuar.
4. Para generar una clave de acceso, el usuario debe autenticarse localmente en su dispositivo mediante biometría u otro método de autenticación, como introducir un PIN.
5. Una vez completada la autenticación local, se guarda una nueva clave de acceso en el dispositivo del usuario y se sincroniza con su proveedor de claves de acceso, como iCloud o Google.
6. Después de guardar la clave de acceso, el usuario continúa con el proceso de registro para finalizar la creación de su cuenta.

Una vez completado este proceso, el usuario puede autenticarse con su clave de acceso guardada la próxima vez que inicie sesión en tu aplicación.

<div id="before-you-begin">
  ## Antes de comenzar
</div>

<div id="configure-a-custom-domain">
  ### Configura un dominio personalizado
</div>

Las claves de acceso nativas requieren el uso de un <Tooltip tip="Dominio personalizado: dominio de terceros con un nombre especializado o distintivo." cta="Ver glosario" href="/es/docs/glossary?term=custom+domain">dominio personalizado</Tooltip>. Antes de continuar, asegúrate de haber configurado un dominio personalizado para tu tenant. Para obtener más información, consulta [Custom Domains](/es/docs/customize/custom-domains).

<div id="configure-your-passkey-policy">
  ### Configura tu política de claves de acceso
</div>

Antes de implementar claves de acceso nativas para aplicaciones de Android o iOS, debes configurar una política de claves de acceso en tu tenant de Auth0. Para preparar tu tenant, sigue los pasos de [Configurar la política de claves de acceso](/es/docs/authenticate/database-connections/passkeys/configure-passkey-policy).

<div id="prepare-your-application">
  ### Prepara tu aplicación
</div>

Para preparar tu aplicación para claves de acceso nativas, debes configurar tus Device Settings y añadir la concesión `Passkey`. Puedes completar estas configuraciones desde tu <Tooltip tip="Auth0 Dashboard: el principal producto de Auth0 para configurar tus servicios." cta="Ver glosario" href="/es/docs/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip> o la <Tooltip tip="Management API: un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/es/docs/glossary?term=Management+API">Management API</Tooltip>.

<div id="associated-domains">
  #### Dominios asociados
</div>

Para que los usuarios finales puedan autenticarse con una sola clave de acceso en distintos tipos de aplicaciones o en aplicaciones con diferentes subdominios, establece el ID de la Relying Party en el dominio raíz o dominio principal. Para obtener más información, consulta [Passkeys](/es/docs/authenticate/database-connections/passkeys).

<div id="configure-auth0-dashboard">
  ### Configura Auth0 Dashboard
</div>

1. Ve a [Applications &gt; Applications](https://manage.auth0.com/#/applications) y selecciona la aplicación que quieres actualizar.
2. En la parte inferior de la pestaña **Settings**, selecciona **Advanced Settings**. Luego, elige la pestaña **Device Settings**.
3. Completa las secciones de **iOS** y **Android** según sea necesario para tu aplicación. Luego, haz clic en **Save Changes**.
4. En la sección **Advanced Settings**, selecciona la pestaña **Grant Types**.
5. Habilita la concesión **Passkey** y, a continuación, selecciona **Save Changes**.

<div id="management-api">
  #### Management API
</div>

Llama al endpoint [Actualizar un cliente](https://auth0.com/docs/api/management/v2/clients/patch-clients-by-id) y:

* Actualiza `grant_types` para incluir `urn:okta:params:oauth:grant-type:webauthn`.
* Usa el objeto `mobile` para especificar la configuración de los dispositivos iOS y Android según sea necesario.

<div id="implement-passkey-flows">
  ## Implementa flujos de claves de acceso
</div>

Puedes definir los siguientes flujos de claves de acceso para tu aplicación:

* [Flujo de registro](#signup-flow): Permite a los nuevos usuarios generar y guardar una clave de acceso durante el proceso de registro.
* [Flujo de inicio de sesión](#login-flow): Permite a un usuario existente que ya haya registrado claves de acceso autenticarse con su clave de acceso guardada durante el proceso de inicio de sesión.

<div id="signup-flow">
  ### Flujo de registro
</div>

Un usuario inicia el flujo de registro con clave de acceso cuando intenta iniciar sesión en tu aplicación por primera vez.

Si el usuario proporciona un identificador que ya existe, se recomienda pedirle que, en su lugar, complete el flujo de inicio de sesión. De lo contrario, la Action fallará.

<div id="flow-steps">
  #### Pasos del flujo
</div>

1. Un usuario visita tu aplicación y elige crear una cuenta nueva. El usuario proporciona un identificador solicitado por tu aplicación, como su dirección de correo electrónico.
2. Tu aplicación inicia el desafío de registro llamando al endpoint [Request Signup Challenge](/es/docs/authenticate/database-connections/passkeys/native-passkeys-api#request-signup-challenge) de Auth0 Authentication API:

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  * Si no especificas un `realm`, se utilizará el directorio predeterminado de tu tenant.
  * De forma predeterminada, el correo electrónico es el identificador obligatorio. Si has habilitado [Flexible Identifiers](/es/docs/authenticate/database-connections/activate-and-configure-attributes-for-flexible-identifiers) para tu conexión de base de datos, puedes usar en su lugar una combinación de `email`, `phone_number` o `username`. Estas opciones pueden ser obligatorias u opcionales y deben coincidir con tu configuración de Flexible Identifiers.
</Callout>

```text lines
   POST /passkey/register
   Content-Type: application/json

   {
     "client_id": "{YOUR_CLIENT_ID}",
     "realm": "{OPTIONAL_CONNECTION}",
     "user_profile": {
   	  "email": "{VALID_EMAIL_ADDRESS}",
   	  "name": "{OPTIONAL_USER_DISPLAY_NAME}",
     }
   }
```

3. Como respuesta, Auth0 devuelve [`PublicKeyCredentialCreationOptions`](https://www.w3.org/TR/webauthn-3/#dictdef-publickeycredentialcreationoptions) junto con un ID de `auth_session` y `rpId` como `rp.id`:

   ```text lines expandable
   HTTP/1.1 200 OK
   Content-Type: application/json

   {
     "authn_params_public_key": {
       "challenge": "{GENERATED_CHALLENGE_FOR_THIS_SESSION}",
       "timeout": {MILLISECONDS},
       "rp": {
         "id": "{YOUR_CUSTOM_DOMAIN}",
         "name": "{YOUR_CUSTOM_DOMAIN}"
       },
       "pubKeyCredParams": [
         { type: 'public-key', alg: -8 },
         { type: 'public-key', alg: -7 },
         { type: 'public-key', alg: -257 }
       ],
       "authenticatorSelection": {
         "residentKey": "required",
         "userVerification": "preferred"
       },
       "user": {
         "id": "{GENERATED_ID}",
         "name": "{USER-ENTERED_IDENTIFIER}",
         "displayName": "{USER-ENTERED_DISPLAY_NAME_OR_IDENTIFIER_IF_MISSING}"
       }
     },
     "auth_session": "{SESSION_ID}"
    }
   ```

   <Callout icon="file-lines" color="#0EA5E9" iconType="regular">
     El `rpId` que se devuelve en la respuesta coincide con el dominio usado en el flujo web para que las claves de acceso funcionen en los dominios asociados.
   </Callout>

4. A continuación, su aplicación completa el proceso de registro del usuario mediante las API nativas correspondientes:

   * Consulte la [documentación de registro para Android](https://developer.android.com/identity/passkeys/create-passkeys).
   * Consulte la [documentación de registro para iOS](https://developer.apple.com/documentation/authenticationservices/supporting-passkeys#Register-a-new-account-on-a-service).

5. A continuación, su aplicación usa la información de credenciales obtenida durante el proceso de registro, incluidos los detalles de `authn_response`, para llamar al endpoint [Token](/es/docs/authenticate/database-connections/passkeys/native-passkeys-api#authenticate-new-user):

   ```text lines
   POST /oauth/token
   Content-Type: application/json

   {
     "grant_type": "urn:okta:params:oauth:grant-type:webauthn",
     "client_id": "{YOUR_CLIENT_ID}",
     "realm": "{OPTIONAL_CONNECTION}",
     "scope": "{OPTIONAL_REQUESTED_SCOPE}",
     "audience": "{OPTIONAL_REQUESTED_AUDIENCE}"
     "auth_session": "{SESSION_ID_FROM_THE_FIRST_REQUEST}",
     "authn_response": {
       "id": "{BASE64URL_ID}",
       "rawId": "{BASE64URL_RAWID}",
       "type": "public-key",
       "authenticatorAttachment": "platform|cross-platform",
       "response": {
         "clientDataJSON": "{BASE64URL_CLIENT_DATA_JSON}",
         "attestationObject": "{BASE64URL_ATTESTATION_OBJECT}",
         {OTHER_PROPERTIES}
       },
     },
   }
   ```

6. Auth0 crea una nueva cuenta de usuario y devuelve los tokens solicitados para completar el flujo:

   ```text lines
   HTTP/1.1 200 OK
   Content-Type: application/json

   {
     "access_token": "{BASE64_TOKEN}",
     "refresh_token": "{BASE64_TOKEN}",
     "id_token": "{BASE64_TOKEN}",
     "token_type": "Bearer",
     "expires_in": {SECONDS}
   }
   ```

<div id="login-flow">
  ### Flujo de inicio de sesión
</div>

Un usuario existente inicia el flujo de inicio de sesión con clave de acceso cuando intenta iniciar sesión en tu aplicación. Este flujo solo se aplica a los usuarios existentes que guardaron claves de acceso en sus cuentas durante el registro inicial.

<div id="flow-steps">
  #### Pasos del flujo
</div>

1. Un usuario abre tu aplicación y comienza el proceso de inicio de sesión. A continuación, tu aplicación inicia el desafío de inicio de sesión llamando al endpoint [Request Login Challenge](/es/docs/authenticate/database-connections/passkeys/native-passkeys-api#request-login-challenge) de la Authentication API:

   <Callout icon="file-lines" color="#0EA5E9" iconType="regular">
     Si no especificas un `realm`, se usa el directorio predeterminado de tu tenant.
   </Callout>

   ```text lines
   POST /passkey/challenge
   Content-Type: application/json

   {
     "client_id": "{YOUR_CLIENT_ID}",
     "realm": "{OPTIONAL_CONNECTION}"
   }
   ```

2. En respuesta, Auth0 devuelve [`PublicKeyCredentialRequestOptions`](https://www.w3.org/TR/webauthn-3/#dictdef-publickeycredentialrequestoptions) junto con una `auth_session` para continuar el flujo en tu aplicación:

   ```text lines
   HTTP/1.1 200 OK
   Content-Type: application/json

   {
     "authn_params_public_key": {
       "challenge": "{GENERATED_CHALLENGE_FOR_THIS_SESSION}",
       "timeout": {AUTH_TIMEOUT_IN_MILLISECONDS},
       "rpId": "{CUSTOM_DOMAIN}",
       "userVerification": "preferred"
     },
     "auth_session": "{SESSION_ID}"
   }
   ```

3. A continuación, tu aplicación completa el proceso de inicio de sesión con las API nativas correspondientes:

   * Revisa la [documentación de inicio de sesión de Android](https://developer.android.com/identity/passkeys/sign-in-with-passkeys)
   * Revisa la [documentación de inicio de sesión de iOS](https://developer.apple.com/documentation/authenticationservices/supporting-passkeys#Connect-to-a-service-with-an-existing-account).

4. A continuación, tu aplicación usa la información de credenciales obtenida durante el proceso de inicio de sesión, incluidos los detalles de `authn_response`, para llamar al endpoint [Token](/es/docs/authenticate/database-connections/passkeys/native-passkeys-api#authenticate-existing-user):

   ```text lines
   POST /oauth/token
   Content-Type: application/json

   {
     "grant_type": "urn:okta:params:oauth:grant-type:webauthn",
     "client_id": "{YOUR_CLIENT_ID}",
     "realm": "{OPTIONAL_CONNECTION}",
     "scope": "{OPTIONAL_REQUESTED_SCOPE}",
     "audience": "{OPTIONAL_REQUESTED_AUDIENCE}"
     "auth_session": "{SESSION_ID_FROM_THE_FIRST_REQUEST}",
     "authn_response": {
       "id": "{BASE64URL_ID}",
       "rawId": "{BASE64URL_RAWID}",
       "type": "public-key",
       "authenticatorAttachment": "platform|cross-platform",
       "response": {
         "authenticatorData": "{BASE64URL_AUTHENTICATORDATA}",
         "clientDataJSON": "{BASE64URL_CLIENTDATAJSON}",
         "signature": "{BASE64URL_SIGNATURE}",
         "userHandle": "{BASE64URL_USERHANDLE}"
       },
       "clientExtensionResults": {OPTIONAL_OBJECT}
      }
   }
   ```

5. Auth0 autentica las credenciales y devuelve los tokens solicitados para completar el flujo:

   ```text lines
   HTTP/1.1 200 OK
   Content-Type: application/json

   {
     "access_token": "{BASE64_TOKEN}",
     "refresh_token": "{BASE64_TOKEN}",
     "id_token": "{BASE64_TOKEN}",
     "token_type": "Bearer",
     "expires_in": {SECONDS}
   }
   ```
