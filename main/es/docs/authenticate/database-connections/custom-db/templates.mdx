Si tienes tu propia base de datos (conocida en Auth0 como un almacén de datos heredado) que contiene datos de identidad de usuario, puedes usarla como <Tooltip tip="proveedor de identidad (IdP): Servicio que almacena y gestiona identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=identity+provider">proveedor de identidad</Tooltip> para autenticar usuarios.
Creas y configuras la conexión con tu almacén de datos heredado como una base de datos personalizada en Auth0. Puedes optar por migrar los datos desde tu base de datos heredada al almacén de datos de Auth0 de forma gradual con el tiempo, o seguir usándola sin migrar los datos. Proporcionamos plantillas de scripts para realizar funciones en la base de datos personalizada que puedes usar y personalizar.

Hay dos tipos de scripts de acción de base de datos personalizada:

* **Migración automática**: Cada vez que un usuario inicia sesión en Auth0, si el usuario aún no está en Auth0, el script comprobará la base de datos heredada para ver si el usuario existe allí. Si lo encuentra y la marca **Import users to Auth0** está activada, los datos del usuario se migran al almacén de datos de Auth0. A esta capacidad a veces se la denomina **migración gradual** o **migración diferida**.
* **Base de datos heredada**: Auth0 siempre consultará la base de datos subyacente cuando un usuario intente iniciar sesión, se cree, cambie su contraseña, verifique su correo electrónico o sea eliminado. Si lo encuentra y la marca **Import users to Auth0** **no** está activada, los datos del usuario permanecen en la base de datos heredada y **no** migran a Auth0.

<div id="script-execution">
  ## Ejecución de scripts
</div>

Como se describe en [Conexiones de base de datos personalizadas](/docs/es/authenticate/database-connections/custom-db), un tipo de conexión de base de datos personalizada le permite configurar scripts de acción: código personalizado que se usa al interactuar con su repositorio de identidades heredado. Cada script de acción es, básicamente, una función de JavaScript con nombre a la que se le pasan varios parámetros; el nombre de la función y los parámetros que se pasan dependen del script.

<div id="limits">
  ### Límites
</div>

La ejecución del script de acción admite la naturaleza asíncrona de JavaScript, y se pueden usar construcciones como los objetos [Promise](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/Promise). El procesamiento asíncrono implica, en la práctica, una suspensión mientras se espera a que se complete una operación, y un contenedor serverless de Webtask de Auth0 suele tener un límite de ejecución de 20 segundos, tras el cual el contenedor puede reciclarse. Si un contenedor se recicla debido a este límite, la operación finalizará prematuramente, lo que en última instancia hará que se devuelva un error (además de poder restablecer el objeto `global`).

<div id="completion-and-the-callback-function">
  ### Finalización y la función callback
</div>

La función `callback` proporcionada a cada script de acción actúa, en la práctica, como una señal que indica que la operación ha finalizado. Un script de acción debe terminar inmediatamente después de llamar a la función `callback` (ya sea de forma implícita o mediante la ejecución explícita de una instrucción `return` de JavaScript) y no debe realizar ninguna otra operación.

<Warning>
  La función `callback` proporcionada por Auth0 debe llamarse exactamente **una** vez; si se llama más de una vez dentro de un script de acción, se producirán resultados impredecibles y/o errores.
</Warning>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Cuando `callback` se ejecuta sin parámetros, como en `callback()`, esto implica que la función se ha llamado como si se hubiera ejecutado `callback(null)`.
</Callout>

Si un script de acción utiliza procesamiento asíncrono, la llamada a la función `callback` debe posponerse hasta que ese procesamiento finalice y debe ser lo último que se llame. La ejecución asíncrona hace que un `callback` de JavaScript se ejecute una vez completada la operación asíncrona; normalmente, este callback se activa en algún momento después de que finaliza el cuerpo principal (sincrónico) de una función de JavaScript.

<Warning>
  No ejecutar la función `callback` hará que la ejecución se quede bloqueada y, en última instancia, que se devuelva un error. El script de acción debe llamar a la función `callback` exactamente una vez: debe llamarse al menos una vez para evitar que la ejecución se bloquee; sin embargo, no debe llamarse más de una vez, ya que, de lo contrario, se producirán resultados impredecibles y/o errores.
</Warning>
