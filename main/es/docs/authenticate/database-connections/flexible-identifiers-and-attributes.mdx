<Warning>
  Habilitar Identificadores Flexibles en tu tenant puede introducir cambios incompatibles en tu entorno de producción. Prueba esta función exhaustivamente en un entorno de desarrollo y toma nota de la configuración actual de tu conexión antes de implementarla de forma generalizada.
</Warning>

Un **Identificador Flexible** es el atributo que un usuario introduce en una pantalla de inicio de sesión para autenticarse. Puedes elegir entre email, nombre de usuario, teléfono o una combinación de dos o más.

<div id="attribute-and-identifier-definitions">
  ### Definiciones de Atributo e Identificador
</div>

Para este producto, un **Atributo** es un dato de usuario que puede almacenarse, como el correo electrónico, el número de teléfono y el nombre de usuario. Todos los Identificadores son Atributos, pero solo algunos atributos son Identificadores.

Un **Identificador** es un Atributo único que permite identificar de forma inequívoca a un usuario dentro de una conexión determinada. El correo electrónico, el teléfono y el nombre de usuario pueden identificar de forma única a una persona y servir como Identificadores, mientras que otros atributos contribuyen al perfil del usuario sin identificarlo de manera única.

<div id="use-flexible-identifiers">
  ### Usa Flexible Identifiers
</div>

Flexible Identifiers es para uso general con las siguientes limitaciones:

* Flexible Identifiers, incluido el atributo teléfono, solo está disponible con [Universal Login](/es/docs/authenticate/login/auth0-universal-login/universal-login-vs-classic-login/universal-experience) y debes [configurar un proveedor de telefonía](/es/docs/customize/phone-messages/configure-phone-messaging-providers).
* Debes configurar [Identifier First](/es/docs/authenticate/login/auth0-universal-login/identifier-first) para usar la verificación por teléfono en el registro.
* El atributo de dirección de correo electrónico debe estar habilitado para usar [MFA adaptativo](/es/docs/secure/multi-factor-authentication/adaptive-mfa).
* Debes tener una dirección de correo electrónico en el perfil de usuario para usar las indicaciones de Signup para [Organizations](/es/docs/manage-users/organizations/organizations-overview).
* Los usuarios finales bloqueados por <Tooltip tip="Brute-force Protection: Forma de protección contra ataques que protege contra ataques de fuerza bruta que se producen desde una sola dirección IP y tienen como objetivo una sola cuenta de usuario." cta="Ver glosario" href="/es/docs/glossary?term=Brute+Force+Protection">Brute Force Protection</Tooltip> no pueden desbloquear su cuenta mediante un mensaje SMS. Hay otros métodos disponibles; para obtener más información, [consulta Brute Force Protection](/es/docs/secure/attack-protection/brute-force-protection).
* Flexible Identifiers mueve el campo de identificador a la primera pantalla de inicio de sesión y cambia el prompt de restablecimiento de contraseña para que use nombre de usuario en lugar de correo electrónico.
* Los tokens OTP para la verificación del identificador por teléfono y correo electrónico tienen una vigencia de 900 segundos.

<div id="issues-using-flexible-identifiers">
  ### Problemas al usar Flexible Identifiers
</div>

La siguiente es una lista de posibles problemas que puedes encontrar al configurar y gestionar Flexible Identifiers:

* Si tu aplicación no especifica el scope `phone` en la solicitud de autorización, no recibirás el claim `phone_number`. Para obtener más información sobre los scopes, [lee Scopes](/es/docs/get-started/apis/scopes).
* Tu script de acción de base de datos personalizada Get User debe ser válido cuando **Import Users to Auth0** está **activado.** Para obtener más información, [lee Configure Automatic Migration from Your Database](/es/docs/manage-users/user-migration/configure-automatic-migration-from-your-database).
* A cada usuario se le debe asignar un nombre de usuario, una dirección de correo electrónico y un número de teléfono únicos, independientemente del tipo de conexión. El número de teléfono debe ser único aunque no se agregue como atributo.
* Si usas el script de acción de base de datos personalizada Change Password y quieres establecer `email` y `email_verified` en `True`, debes devolver el estado preferido de `email_verified` en el objeto. Para obtener más información, [lee Change Password](/es/docs/authenticate/database-connections/custom-db/templates/change-password).
* Si usas una conexión de base de datos personalizada con Import Users to Auth0 desactivado, debes alinear las propiedades de tu perfil de usuario con el perfil de usuario normalizado de Auth0. Para obtener más información, [lee Normalized User Profile](/es/docs/manage-users/user-accounts/user-profiles/normalized-user-profiles).
* Si usas una conexión de base de datos personalizada con Import Users to Auth0 activado, Auth0 comprobará que `phone_number` y `phone_verified` sean únicos.
* Los prompts de Identifier First muestran todos los identificadores en la primera pantalla y eliminan tu configuración anterior, y el prompt de restablecimiento de contraseña mostrará el campo de entrada para Username en lugar de Email.
* Familiarízate con las prácticas recomendadas para evitar ataques de SMS Pumping. Para obtener más información, lee nuestro [documento técnico sobre SMS Pumping](https://www.okta.com/resources/whitepaper-handling-toll-fraud-and-sms-pumping-with-twilio-in-the-okta-customer-identity/).
