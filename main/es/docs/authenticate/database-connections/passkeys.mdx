Las claves de acceso son un método de autenticación seguro y passwordless basado en los [estándares FIDO2 (WebAuthn y CTAP)](https://en.wikipedia.org/wiki/WebAuthn). Tienen varias ventajas frente a la autenticación tradicional con identificador/contraseña:

* **Las claves de acceso permiten a los usuarios autenticarse con biometría o credenciales vinculadas al dispositivo** (como una huella digital, un PIN o un patrón), por lo que el inicio de sesión es más rápido y no requiere recordar una contraseña.

* **Las claves de acceso sincronizan las credenciales entre dispositivos** para que los usuarios no tengan que volver a registrarlas en cada dispositivo nuevo.

* **Las claves de acceso son resistentes al phishing** porque usan criptografía de clave pública, por lo que no hay secretos compartidos, y el dispositivo del usuario genera claves únicas para cada cuenta.

* **Las claves de acceso permiten una recuperación más fiable** porque las credenciales almacenadas pueden sobrevivir a la pérdida del dispositivo de origen.

* **Las claves de acceso vinculan las credenciales a un dominio específico** para que los usuarios puedan autenticarse en todo un dominio con una sola clave de acceso.

Para obtener más información sobre las claves de acceso, lee [la descripción general sobre claves de acceso de FIDO Alliance](https://fidoalliance.org/passkeys).

<div id="about-passkeys-on-auth0">
  ## Acerca de las claves de acceso en Auth0
</div>

Auth0 admite claves de acceso como método de autenticación para conexiones de base de datos, con tres métodos de implementación según el tipo de aplicación:

* Claves de acceso de Universal Login para aplicaciones web
* [Aplicaciones nativas con Embedded Login](/es/docs/authenticate/passwordless/implement-login/embedded-login/native)
* [Claves de acceso nativas para aplicaciones Android e iOS](/es/docs/authenticate/database-connections/passkeys/native-passkeys-for-mobile-applications)

Auth0 tiene un límite de 20 claves de acceso por usuario.

Cuando habilitas las claves de acceso para tu conexión de base de datos, quedan disponibles para los usuarios durante el registro y el inicio de sesión.

Las claves de acceso no sustituyen ni invalidan las credenciales existentes de un usuario. Cuando un usuario crea su clave de acceso, esta se añade a su cuenta como método de autenticación, pero cualquier credencial existente de correo electrónico/nombre de usuario y contraseña sigue siendo válida.

<div id="passkeys-with-mfa-enabled">
  ### Claves de acceso con MFA habilitado
</div>

Si <Tooltip tip="Autenticación multifactor (MFA): proceso de autenticación de usuarios que utiliza un factor adicional, además del nombre de usuario y la contraseña, como un código por SMS." cta="Ver glosario" href="/es/docs/glossary?term=MFA">MFA</Tooltip> está habilitado, es posible que se pida al usuario completar un desafío de MFA después de autenticarse con una clave de acceso, según la configuración y la evaluación de riesgos.

El comportamiento predeterminado es exigir que se complete un desafío de MFA independientemente de si el método de autenticación utilizado fue una contraseña o una clave de acceso. Dado el alto nivel de seguridad que ofrecen las claves de acceso, puede omitir MFA para los usuarios que se hayan autenticado con una clave de acceso para reducir la fricción. Esto se puede lograr con una Action de Post Login.

Para obtener más información, consulte [Reducir la fricción con claves de acceso](/es/docs/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger#reduce-friction-with-passkeys) y [Autenticación multifactor](/es/docs/secure/multi-factor-authentication).

<div id="passkeys-with-multiple-custom-domains-mcd">
  ### Claves de acceso con múltiples dominios personalizados (MCD)
</div>

<ReleaseStageNotice feature="múltiples dominios personalizados" stage="ea" plans="Empresarial" terms="true" />

Si tienes habilitados múltiples dominios personalizados en tu tenant, Auth0 mantiene una relación uno a uno entre un dominio y la clave de acceso de ese dominio. Los usuarios con una base de datos habilitada para claves de acceso pueden registrarse e iniciar sesión con una clave de acceso vinculada al dominio específico en el que se creó. Los usuarios solo pueden registrar una clave de acceso para un dominio (el primero con el que la registren, entre los múltiples dominios personalizados del tenant).

Para el login passwordless, el dominio personalizado seleccionado debe reflejarse en el Magic Link del flujo de login passwordless.

<div id="relying-party-id-for-passkeys">
  ### ID de Relying Party para claves de acceso
</div>

El [identificador de la relying party (RP ID)](https://www.w3.org/TR/webauthn-2/#relying-party-identifier) es un dominio al que WebAuthn vincula credenciales como las claves de acceso. El RP ID define qué orígenes de solicitud están permitidos para la autenticación.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  El dominio de tu aplicación debe añadirse a la lista de **Allowed Origins (CORS)** en la [configuración de la aplicación](https://manage.auth0.com/#/applications) para que el RP ID funcione correctamente.
</Callout>

Definir el RP ID como un sufijo del origen permite a los usuarios autenticarse en distintos subdominios con una sola clave de acceso. Por ejemplo, si tu aplicación web está disponible en `login.example.com` y tu aplicación nativa está disponible en `app.example.com`, puedes configurar el RP ID como `example.com` para que los usuarios finales puedan autenticarse en ambas aplicaciones (y en cualquier otro subdominio de `example.com`) con una sola clave de acceso.

| Entorno | Dominio raíz                | RP ID         |
| :------ | :-------------------------- | :------------ |
| Web     | `https://login.example.com` | `example.com` |
| iOS     | `app.example.com`           | `example.com` |
| Android | `assetlinks.json`           | `example.com` |

Con Auth0, puedes personalizar el RP ID para que use el dominio raíz o el dominio padre, de modo que los usuarios puedan autenticarse en aplicaciones móviles o aplicaciones web con la misma clave de acceso. Si usas Múltiples dominios personalizados, también puedes establecer `rp.id` para cada dominio personalizado.

Para obtener más información sobre cómo personalizar el RP ID, consulta [Configure Passkey Policy](/es/docs/authenticate/database-connections/passkeys/configure-passkey-policy).
