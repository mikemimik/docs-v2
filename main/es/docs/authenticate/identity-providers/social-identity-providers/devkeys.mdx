<Card title="Descripción general">
  Conceptos clave

  * Revisa las limitaciones de las claves de desarrollador de Auth0.
</Card>

Al usar cualquiera de los [proveedores de identidad social](/es/docs/authenticate/identity-providers/social-identity-providers), debes registrar tu aplicación con el <Tooltip tip="Proveedor de identidad (IdP): servicio que almacena y administra identidades digitales." cta="Ver glosario" href="/es/docs/glossary?term=identity+provider">proveedor de identidad</Tooltip> correspondiente para obtener un <Tooltip tip="ID de cliente: un identificador único asignado a tu aplicación cuando la registras con un proveedor de identidad." cta="Ver glosario" href="/es/docs/glossary?term=Client+ID">ID de cliente</Tooltip> y un <Tooltip tip="Secreto de cliente: un valor confidencial que se usa junto con el ID de cliente para autenticar tu aplicación con un proveedor de identidad." cta="Ver glosario" href="/es/docs/glossary?term=Client+Secret">secreto de cliente</Tooltip>.

<Warning>
  Auth0 te permite probar un proveedor de identidad social sin especificar tu propio ID de cliente ni secreto de cliente, mediante las claves de desarrollador de Auth0. Esto te permite habilitar y probar rápidamente un proveedor de identidad social específico, pero no debe usarse en producción.
</Warning>

Las claves de desarrollador de Auth0 no están disponibles en las [implementaciones de Private Cloud](/es/docs/deploy-monitor).

Para entornos de producción, asegúrate de [seguir los pasos del proveedor que hayas elegido](/es/docs/authenticate/identity-providers) para obtener del proveedor el ID de cliente y el secreto de cliente, y así evitar las [limitaciones](#limitations-of-developer-keys) de usar claves de desarrollador. Para ver instrucciones sobre cómo convertir las claves de desarrollador de Google en claves de producción, [consulta este Auth0 Developer Lab.](https://developer.auth0.com/resources/labs/authentication/google-social-connection-to-login#set-up-google-production-keys)

<Card title="Claves de desarrollador personalizadas">
  Una o más conexiones están usando claves de desarrollo de Auth0, que solo están pensadas para desarrollo y pruebas. Las conexiones deben configurarse con tus propias claves de desarrollador para que la página de consentimiento muestre tu logotipo en lugar del de Auth0 y para configurar el inicio de sesión único (SSO) en estas conexiones. No se recomienda usar las claves de desarrollo de Auth0 en entornos de producción.
</Card>

<Card title="ID de cliente y secreto de cliente">
  La terminología exacta de ID de cliente / secreto de cliente puede variar entre distintos proveedores de identidad. Por ejemplo, X se refiere a ellos como Consumer Key / Consumer Secret, y LinkedIn como API Key / Secret Key.
</Card>

<div id="limitations-of-developer-keys">
  ## Limitaciones de las claves de desarrollador
</div>

Las claves de desarrollador de Auth0 están pensadas exclusivamente para pruebas. Usarlas en lugar de tu propio ID de cliente y secreto de cliente puede provocar comportamientos inesperados, funciones limitadas o errores en la aplicación.

Al usar las claves de desarrollador de Auth0, el flujo de autenticación de los distintos proveedores de identidad puede mostrar a tus usuarios el nombre, el logotipo y la información de Auth0. En cambio, cuando registras tu propia aplicación, puedes usar tu propio logotipo y otra información de la aplicación.

<div id="limitations-of-developer-keys-when-using-universal-login">
  ## Limitaciones de las claves de desarrollador al usar Universal Login
</div>

Si utiliza la [experiencia Classic Login](/es/docs/authenticate/login/auth0-universal-login/universal-login-vs-classic-login/classic-experience) o la [experiencia Universal Login](/es/docs/authenticate/login/auth0-universal-login/universal-login-vs-classic-login/universal-experience), se aplican las siguientes limitaciones:

1. No puede usar claves de desarrollador con [dominios personalizados](/es/docs/customize/custom-domains).
2. El [inicio de sesión único](/es/docs/authenticate/single-sign-on) no funcionará correctamente al usar las claves de desarrollador de Auth0. Las aplicaciones de desarrollador de Auth0 están configuradas para redirigir a la URL `https://login.auth0.com/login/callback` en lugar de a la **URL de devolución de llamada** de su propio tenant, por ejemplo `https://YOUR_AUTH0_DOMAIN/login/callback`.
   Como resultado, la cookie de SSO no se establece en el dominio de su propio tenant, por lo que la próxima vez que un usuario se autentique no se detectará ninguna cookie de SSO, incluso si configuró su aplicación para **Use Auth0 instead of the Identity Provider to do Single Sign-on** (solo en tenants heredados).
3. [Redirigir usuarios con Actions](/es/docs/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger/redirect-with-actions) no funcionará correctamente. Las Actions de redirección se reanudan en el endpoint `https://YOUR_AUTH0_DOMAIN/continue`. Al usar las claves de desarrollador de Auth0, la sesión se establece en un endpoint especial, genérico e independiente del tenant, y al llamar a `/continue` no se encontrará la sesión anterior, lo que dará como resultado un error.
4. El [cierre de sesión federado](/es/docs/authenticate/login/logout) no funciona. Al usar las claves de desarrollador de Auth0, llamar a `/v2/logout?federated` cerrará la sesión del usuario en Auth0, pero no en el proveedor de identidad social.
5. `prompt=none` no funcionará en el endpoint [/authorize](https://auth0.com/docs/api/authentication/reference#social). El método checkSession() de [Auth0.js](/es/docs/libraries/auth0js) usa `prompt=none` internamente, por lo que tampoco funcionará.
6. Si Auth0 actúa como un proveedor de identidad SAML y usa una conexión social con las claves de desarrollador de Auth0, la respuesta SAML generada tendrá algunos errores, como la ausencia del atributo `InResponseTo` o un elemento `AudienceRestriction` vacío.
7. La [autenticación multifactor](/es/docs/secure/multi-factor-authentication) no funcionará correctamente. Cuando la autenticación MFA se completa correctamente, se envía una solicitud POST a `https://YOUR_AUTH0_DOMAIN/mf`. Al usar las claves de desarrollador de Auth0, la sesión se establece en un endpoint especial, genérico e independiente del tenant, y al llamar a `/mf` no se encontrará la sesión anterior, lo que dará como resultado un error.
