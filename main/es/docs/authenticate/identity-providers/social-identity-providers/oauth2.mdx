Los [proveedores de identidad (IdP)](/es/docs/authenticate/identity-providers) más comunes están disponibles en [Auth0 Dashboard](https://manage.auth0.com/#) y en el [Auth0 Marketplace](https://marketplace.auth0.com/features/social-connections). Sin embargo, puedes agregar cualquier proveedor de <Tooltip tip="OAuth 2.0: Marco de autorización que define protocolos y flujos de autorización." cta="Ver glosario" href="/es/docs/glossary?term=OAuth+2.0">OAuth 2.0</Tooltip> como una **conexión social personalizada** en el <Tooltip tip="Auth0 Dashboard: El producto principal de Auth0 para configurar tus servicios." cta="Ver glosario" href="/es/docs/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip>.

1. En el Dashboard, ve a [Authentication &gt; Social](https://manage.auth0.com/#/connections/social).
2. Selecciona **Create Connection**, ve al final de la lista y, a continuación, selecciona **Create Custom**.

El formulario que aparece contiene varios campos que debes usar para configurar la conexión personalizada:

* **Connection Name**: Identificador lógico de la conexión que estás creando. Este nombre no se puede cambiar; debe empezar y terminar con un carácter alfanumérico, y solo puede contener caracteres alfanuméricos y guiones.
* **Authorization URL**: URL a la que se redirige a los usuarios para iniciar sesión.

  <Callout icon="file-lines" color="#0EA5E9" iconType="regular">
    No intentes establecer el parámetro OAuth2 `response_mode` en la URL de autorización. Esta conexión solo admite el valor predeterminado de `response_mode` (`query`).
  </Callout>
* **Token URL**: URL que se usa para intercambiar el código de autorización recibido por <Tooltip tip="Access Token: Credencial de autorización, en forma de una cadena opaca o JWT, usada para acceder a una API." cta="Ver glosario" href="/es/docs/glossary?term=access+tokens">tokens de acceso</Tooltip> y, si se solicitan, <Tooltip tip="ID Token: Credencial destinada al propio cliente, en lugar de acceder a un recurso." cta="Ver glosario" href="/es/docs/glossary?term=ID+tokens">tokens de ID</Tooltip>.
* **Scope**: Parámetros `scope` que se envían con la solicitud de autorización. Separa varios scopes con espacios.
* **Separate scopes using a space**: Interruptor que determina cómo se delimitan los scopes si el parámetro `connection_scope` se incluye al [llamar a la API del IdP](/es/docs/authenticate/identity-providers/calling-an-external-idp-api). De forma predeterminada, los scopes se delimitan con una coma. Si el interruptor está habilitado, los scopes se delimitan con un espacio. Para obtener más información, consulta [Add Scopes/Permissions to Call Identity Provider APIs](/es/docs/authenticate/identity-providers/adding-scopes-for-an-external-idp).
* **<Tooltip tip="Client ID: Valor de identificación dado a tu recurso registrado por Auth0." cta="Ver glosario" href="/es/docs/glossary?term=Client+ID">ID de cliente</Tooltip>**: ID de cliente de Auth0 como aplicación para solicitar autorización e intercambiar el código de autorización. Para obtener un ID de cliente, tendrás que registrarte con el <Tooltip tip="Identity Provider (IdP): Servicio que almacena y administra identidades digitales." cta="Ver glosario" href="/es/docs/glossary?term=identity+provider">proveedor de identidad</Tooltip>.
* **<Tooltip tip="Client Secret: Secret usado por un cliente (aplicación) para autenticarse con el servidor de autorización; solo deberían conocerlo el cliente y el servidor de autorización, y debe ser lo suficientemente aleatorio como para que no pueda adivinarse." cta="Ver glosario" href="/es/docs/glossary?term=Client+Secret">secreto de cliente</Tooltip>**: Secreto de cliente de Auth0 como aplicación para intercambiar el código de autorización. Para obtener un secreto de cliente, tendrás que registrarte con el proveedor de identidad.
* **Fetch User Profile Script**: Script de Node.js que se usa para llamar a una URL de userinfo con el token de acceso proporcionado. Para obtener más información sobre este script, consulta [Fetch User Profile Script](#fetch-user-profile-script).
* **Purpose**: Habilita la conexión social para Authentication, Connected Accounts para Token Vault, o ambas opciones. Para obtener más información, consulta [User authentication vs Connected Accounts](/es/docs/secure/tokens/token-vault/connected-accounts-for-token-vault#user-authentication-vs-connected-accounts).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Al configurar el proveedor de identidad personalizado, usa la URL de devolución de llamada `https://{yourDomain}/login/callback`.
</Callout>

Una vez que crees la conexión personalizada, verás la vista de **Aplicaciones** y tu conexión estará sujeta a la [Política de limitación de tasa](/es/docs/troubleshoot/customer-support/operational-policies/rate-limit-policy) de Auth0. Aquí puedes habilitar y deshabilitar las aplicaciones en las que quieres que aparezca la conexión.

<div id="update-authentication-flow">
  ## Actualizar el flujo de autenticación
</div>

Cuando creas una conexión, el grant de OAuth 2.0 predeterminado asignado a la conexión es Authorization Code Flow. Si tienes una aplicación pública que no puede almacenar un secreto de cliente, como una aplicación de una sola página o una aplicación nativa, puedes usar la <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/es/docs/glossary?term=Management+API">Management API</Tooltip> para actualizar la conexión y hacer que use [Authorization Code Flow + PKCE](/es/docs/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce). Para obtener más información sobre los <Tooltip tip="Flujo de autorización: concesión de autorización (o workflow) especificada en el marco de OAuth 2.0." cta="Ver glosario" href="/es/docs/glossary?term=authorization+flows">flujos de autorización</Tooltip>, consulta [¿Qué flujo de OAuth 2.0 debería usar?](/es/docs/get-started/authentication-and-authorization-flow/which-oauth-2-0-flow-should-i-use)

1. Haz una solicitud `GET` al endpoint [`/get-connections-by-id`](https://auth0.com/docs/api/management/v2/connections/get-connections-by-id). La respuesta será similar a esta:

   ```json lines
   {
     "id": "[connectionID]",
     "options": {
       "email": true,
       "scope": [
         "email",
         "profile"
       ],
       "profile": true
     },
     "strategy": "google-oauth2",
     "name": "google-oauth2",
     "is_domain_connection": false,
     "realms": [
       "google-oauth2"
     ]
   }
   ```

2. Copia el objeto `options` completo.

3. Haz una solicitud [`PATCH`](https://auth0.com/docs/api/management/v2/connections/patch-connections-by-id) con el objeto `options` y agrega `"pkce_enabled":` `true`.

<Warning>
  Si no incluyes el objeto `options` completo, se perderá información y la conexión dejará de funcionar.
</Warning>

<div id="fetch-user-profile-script">
  ## Script para obtener el perfil de usuario
</div>

El script para obtener el perfil de usuario se ejecuta después de que el usuario haya iniciado sesión con el proveedor de OAuth2. Auth0 ejecuta este script para llamar a la API del proveedor de OAuth2 y obtener el perfil de usuario:

```javascript lines expandable
function fetchUserProfile(accessToken, context, callback) {
  request.get(
    {
      url: 'https://auth.example.com/userinfo',
      headers: {
        'Authorization': 'Bearer ' + accessToken,
      }
    },
    (err, resp, body) => {
      if (err) {
        return callback(err);
      }
      if (resp.statusCode !== 200) {
        return callback(new Error(body));
      }
      let bodyParsed;
      try {
        bodyParsed = JSON.parse(body);
      } catch (jsonError) {
        return callback(new Error(body));
      }
      const profile = {
        user_id: bodyParsed.account.uuid,
        email: bodyParsed.account.email
      };
      callback(null, profile);
    }
  );
}
```

La propiedad `user_id` del perfil devuelto es obligatoria, y la propiedad `email` es opcional, pero muy recomendable. Para obtener más información sobre qué atributos se pueden devolver, consulta [Atributos raíz del perfil de usuario](/es/docs/manage-users/user-accounts/user-profiles/root-attributes/update-root-attributes-for-users).

Puedes filtrar, agregar o quitar cualquier elemento del perfil que devuelve el proveedor. Sin embargo, se recomienda mantener este script lo más simple posible. Para realizar manipulaciones más avanzadas de la información del usuario, puedes usar [Rules](/es/docs/customize/rules). Una ventaja de usar Rules es que se aplican a cualquier conexión.

</Warning>

<div id="pass-provider-specific-parameters">
  ## Pasar parámetros específicos del proveedor
</div>

Puedes pasar parámetros específicos del proveedor al endpoint de autorización de los proveedores de OAuth 2.0. Pueden ser estáticos o dinámicos.

<div id="pass-static-parameters">
  ### Pasar parámetros estáticos
</div>

Para pasar parámetros estáticos (parámetros que se envían con cada solicitud de autorización), puedes usar el elemento `authParams` de `options` al configurar una conexión OAuth 2.0 mediante la [Management API](https://auth0.com/docs/api/management/v2#!/Connections/patch_connections_by_id). La siguiente llamada establecerá un parámetro estático, `custom_param`, con el valor `custom.param.value` en todas las solicitudes de autorización:

<div id="pass-dynamic-parameters">
  ### Pasar parámetros dinámicos
</div>

En determinadas circunstancias, puede que quieras pasar un valor dinámico a un proveedor de identidad de OAuth 2.0. En ese caso, puedes usar el elemento `authParamsMap` de `options` para especificar una asignación entre uno de los parámetros adicionales existentes que acepta el [endpoint `/authorize` de Auth0](https://auth0.com/docs/api/authentication#social) y el parámetro que acepta el proveedor de identidad.

Siguiendo con el mismo ejemplo anterior, supongamos que quieres pasar el parámetro `custom_param` al endpoint de autorización, pero quieres indicar el valor real del parámetro al llamar al endpoint `/authorize` de Auth0.

En ese caso, puedes usar uno de los parámetros adicionales existentes que acepta el endpoint `/authorize`, como `access_type`, y asignarlo al parámetro `custom_param`:

<AuthCodeGroup>
  ```bash cURL
  curl --request PATCH \
    --url 'https://{yourDomain}/api/v2/connections/CONNECTION-ID' \
    --header 'content-type: application/json' \
    --data '{ "options": { "client_id": "...", "client_secret": "...", "authParamsMap": { "custom_param": "access_type" }, "scripts": { "fetchUserProfile": "..." }, "authorizationURL": "https://auth.example.com/oauth2/authorize", "tokenURL": "https://auth.example.com/oauth2/token", "scope": "auth" } }'
  ```

  ```csharp C#
  var client = new RestClient("https://{yourDomain}/api/v2/connections/CONNECTION-ID");
  ```

<div id="pass-extra-headers">
  ## Pasa encabezados adicionales
</div>

En algunos casos, tendrás que pasar encabezados adicionales al <Tooltip tip="Endpoint de token: endpoint del servidor de autorización que se utiliza para solicitar tokens de forma programática." cta="Ver glosario" href="/es/docs/glossary?term=Token+endpoint">endpoint de token</Tooltip> de un proveedor de OAuth 2.0. Para configurar encabezados adicionales, abre la configuración de la conexión y, en el campo **Custom Headers**, especifica un objeto JSON con los encabezados personalizados como pares clave-valor:

```json lines
{
    "Header1" : "Value",
    "Header2" : "Value"
}
```

Veamos un ejemplo en el que un proveedor de identidad puede requerir que envíes un encabezado `Authorization` con credenciales de [autenticación de acceso básica](https://en.wikipedia.org/wiki/Basic_access_authentication). En este caso, puedes especificar el siguiente objeto JSON en el campo **Custom Headers**:

```json lines
{
  "Authorization": "Basic [your credentials]"
}
```

Donde `[your credentials]` son las credenciales reales que deben enviarse al proveedor de identidad.
