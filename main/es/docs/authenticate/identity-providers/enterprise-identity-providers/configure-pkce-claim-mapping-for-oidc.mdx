Las conexiones empresariales que usan [OpenID Connect](/docs/es/authenticate/identity-providers/enterprise-identity-providers/oidc) u O[kta Workforce](/docs/es/authenticate/identity-providers/enterprise-identity-providers/okta) como <Tooltip tip="Proveedor de identidad (IdP): servicio que almacena y gestiona identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=identity+provider">proveedor de identidad</Tooltip> admiten Proof Key for Code Exchange (PKCE), así como el mapeo de atributos y tokens.

<div id="configure-pkce-for-oidc-connections">
  ## Configurar PKCE para conexiones OIDC
</div>

<Tooltip tip="OpenID: Estándar abierto de autenticación que permite a las aplicaciones verificar la identidad de los usuarios sin recopilar ni almacenar información de inicio de sesión." cta="Ver glosario" href="/docs/es/glossary?term=OpenID">OpenID</Tooltip> Connect y las conexiones de Okta Workforce se configuran automáticamente para admitir Proof Key for Code Exchange (PKCE).

Si tu proveedor de identidad (IdP) OIDC admite PKCE a través de los metadatos de OIDC Discovery, Auth0 usará de forma predeterminada el algoritmo más robusto disponible. Para obtener más información sobre los metadatos de OIDC Discovery, consulta la [documentación de OpenID](https://openid.net/specs/openid-connect-discovery-1_0.html#ProviderMetadata).

<div id="view-pkce-configuration-for-a-connection">
  ### Ver la configuración de PKCE para una conexión
</div>

Puedes ver la configuración de PKCE de una conexión específica desde el <Tooltip tip="Auth0 Dashboard: el producto principal de Auth0 para configurar tus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip>:

1. Ve a **Authentication &gt; Enterprise** y elige tu proveedor de OIDC (OpenID Connect u Okta Workforce).
2. Selecciona la pestaña **Settings**.
3. En la sección **General**, busca el campo **Connection Profile**.

<Tabs>
  <Tab title="Auth0 Dashboard">
    Puedes administrar la configuración de PKCE de una conexión desde el Auth0 Dashboard:

    1. Ve a [**Dashboard &gt; Authentication &gt; Enterprise**](https://manage.auth0.com/#/connections/enterprise) y elige tu proveedor de OIDC (OpenID Connect u Okta Workforce).
    2. Selecciona la pestaña **Settings** y busca el campo **Connection Profile**.
    3. Establece la propiedad `pkce` en uno de los [valores admitidos](#supported-pkce-configuration-values) que se indican a continuación.
    4. Selecciona **Save**.
  </Tab>

<div id="supported-pkce-configuration-values">
  ### Valores de configuración de PKCE compatibles
</div>

Auth0 admite los siguientes valores para la configuración de PKCE:

| Valor      | Descripción                                                                                                                 |
| ---------- | --------------------------------------------------------------------------------------------------------------------------- |
| `auto`     | Valor predeterminado. Usa el algoritmo más sólido disponible.                                                               |
| `s256`     | Usa el algoritmo SHA-256. Actualmente, Auth0 no admite tokens RS512.                                                        |
| `plain`    | Usa texto sin formato, como se describe en la [especificación de PKCE](https://www.rfc-editor.org/rfc/rfc7636#section-4.2). |
| `disabled` | Desactiva la compatibilidad con PKCE.                                                                                       |

<Warning>
  Configurar la propiedad `pkce` con un valor distinto de `auto` puede impedir que una conexión funcione correctamente si el proveedor de identidad no admite el valor seleccionado.

  No configure la propiedad como `disabled`, salvo para solucionar problemas de autenticación.
</Warning>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  **Limitación de Microsoft Entra ID**

  Si usa una conexión OpenID Connect para Microsoft Entra ID, debe establecer `pkce` en `s256`, ya que los metadatos de la conexión no exponen el algoritmo de hash utilizado. Actualmente, la [conexión empresarial de Microsoft Entra ID](/docs/es/authenticate/identity-providers/enterprise-identity-providers/azure-active-directory/v2) no admite PKCE.
</Callout>

<div id="map-claims-for-oidc-connections">
  ## Asignar claims para conexiones OIDC
</div>

Las conexiones de OpenID Connect y Okta Workforce pueden asignar automáticamente los claims recibidos del proveedor de identidad (IdP). Puedes configurar este mapeo mediante una plantilla de la biblioteca proporcionada por Auth0 o introduciendo directamente tu propia plantilla.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Los claims asignados no se agregan automáticamente a un token de ID de Auth0. Para agregar claims a un token de ID, consulta [Create custom claims](/docs/es/secure/tokens/json-web-tokens/create-custom-claims#create-custom-claims).
</Callout>

<div id="mapping-template-properties">
  ### Propiedades de la plantilla de mapeo
</div>

Las plantillas de mapeo admiten las propiedades del objeto `options.attribute_map` que se enumeran a continuación. Las plantillas deben estar en formato JSON con pares clave/valor válidos.

| Propiedad        | ¿Obligatoria? | Descripción                                                                                                                   |
| ---------------- | ------------- | ----------------------------------------------------------------------------------------------------------------------------- |
| `mapping_mode`   | Obligatoria   | Método que se usa para mapear los claims entrantes.                                                                           |
| `userinfo_scope` | Opcional      | Scopes que se solicitan al IdP durante la autorización y que determinan qué claims están disponibles en el endpoint UserInfo. |
| `attributes`     | Obligatoria   | Objeto que contiene los detalles de mapeo de los claims entrantes.                                                            |

<div id="mapping-mode">
  ### Modo de mapeo
</div>

La propiedad `mapping_mode` define el método que se utiliza para mapear los claims entrantes del IdP al perfil de usuario de Auth0. `mapping_mode` admite los siguientes valores:

| Valor      | Descripción                                                   |
| ---------- | ------------------------------------------------------------- |
| `use_map`  | Usa la plantilla proporcionada para mapear los datos.         |
| `bind_all` | Copia todos los elementos de datos proporcionados por el IdP. |

<div id="restricted-claims">
  #### Claims restringidos
</div>

Algunos `claims` están reservados para su uso por Auth0; esos `claims` no pueden usarse como claves de atributos para perfiles de usuario.

Si estableces la propiedad `mapping_mode` en `bind_all`, tu IdP puede intentar asignar valores a uno o más de estos `claims` restringidos. Aunque esto no impide que los usuarios se autentiquen en tu conexión, los valores asociados a `claims` restringidos **no** se asignan al perfil de usuario de Auth0.

Si estableces `mapping_mode` en `use_map`, puedes asignar el `claim` restringido entrante a uno válido:

```json lines
"attribute_map": {
        "mapping_mode": "use_map",
        "attributes": {
            "amr": "{context.tokenset.amr}", // `amr` es un claim restringido y no será mapeado
            "federated_amr": "{context.tokenset.amr}" // `federated_amr` no es un claim restringido y será mapeado
        }
    }
```

Para ver una lista completa de los claims restringidos, consulta [Crear claims personalizados](/docs/es/secure/tokens/json-web-tokens/create-custom-claims).

<div id="userinfo-scope">
  ### Alcance de UserInfo
</div>

La propiedad `userinfo_scope` define los scopes que Auth0 incluye en la solicitud de autorización al IdP. Estos scopes determinan qué claims pone a disposición el IdP en su endpoint `UserInfo`. Cuando el mapa de atributos hace referencia a `context.userinfo properties`, Auth0 llama al endpoint `UserInfo` del IdP usando el token de acceso concedido con estos scopes.

Por ejemplo, si quieres solicitar los scopes estándar de OIDC y el scope `groups` para que los claims de grupo estén disponibles en el endpoint `UserInfo`, puedes configurarlo de la siguiente manera:

```json lines
"attribute_map": {
    "mapping_mode": "use_map",
    "userinfo_scope": "openid email profile groups",
    "attributes": {
        "name": "${context.tokenset.name}",
        "groups": "${context.userinfo.groups}"
    }
}
```

<div id="attributes">
  ### Atributos
</div>

La propiedad `attributes` es un objeto que contiene información de mapeo que permite a Auth0 interpretar los claims entrantes del IdP. La información de mapeo debe proporcionarse como pares clave/valor.

La clave de la izquierda corresponde a un atributo del perfil de usuario de Auth0. El valor de la derecha representa el claim entrante del IdP, que puede expresarse como un valor literal, un objeto de contexto dinámico o una combinación de ambos. Los objetos de contexto dinámico son expresiones de plantilla escritas en el conocido formato `${variable}`.

```json lines
"attribute_map": {
    . . .
    "attributes": {
        "name": "${context.tokenset.name}",
        "email": "${context.tokenset.email}",
        "username": "${context.tokenset.preferred_username}"
    }
}
```

<div id="literal-values">
  #### Valores literales
</div>

Un valor literal es un valor estático asociado a un atributo específico del perfil para todos los usuarios de tu conexión.

Por ejemplo, si estás configurando una conexión OIDC de SalesForce y quieres asignar el mismo ID de comunidad de SFDC a todos los perfiles de usuario, puedes hacerlo de la siguiente manera:

```json lines
"attribute_map": {
    . . .
    "attributes": {
        …
        "sf_community_id": "3423409219032-32"
    }
}
```

<div id="context-object">
  #### Objeto de contexto
</div>

Puedes asignar valores dinámicos a atributos del perfil de usuario mediante el objeto `context`. Esto te permite almacenar valores únicos para perfiles individuales, en lugar de valores literales estáticos en todos los perfiles.

El objeto `context` admite las siguientes propiedades:

| Propiedad            | Descripción                                                                                                                                                                                                                 |
| -------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `context.connection` | Contiene las siguientes propiedades:<br /><br /><li> `id`: El identificador único de la conexión (por ejemplo, `con_4423423423432423`).</li><br /><li> `strategy`: La estrategia de la conexión (por ejemplo, `oidc`).</li> |
| `context.tokenset`   | Contiene las siguientes propiedades:<br /><br /><li> `access_token`: El token de acceso validado completo enviado por el IdP.</li><br /><li>`<claim name>`: Cualquier claim del token de ID enviado por el IdP.</li>        |
| `context.userinfo`   | Contiene las siguientes propiedades:<br /><br /><li> `<claim name>`: Cualquier claim disponible proporcionado por el endpoint UserInfo del IdP.</li>                                                                        |

<div id="examples">
  ### Ejemplos
</div>

<div id="simple-user-claim-mapping">
  #### Mapeo sencillo de claims de usuario
</div>

Este ejemplo muestra cómo mapear claims comunes de usuario al perfil de usuario de Auth0 con datos del <Tooltip tip="token de ID: Credencial destinada al propio cliente, en lugar de usarse para acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=ID+token">token de ID</Tooltip>:

```json lines
"attribute_map": {
    . . .
    "attributes": {
        "name": "${context.tokenset.name}",
        "email": "${context.tokenset.email}",
        "username": "${context.tokenset.preferred_username}"
    }
}
```

<div id="group-claim-mapping">
  #### Mapeo del claim de grupos
</div>

Este ejemplo muestra cómo asignar grupos al perfil de usuario de Auth0 a partir del IdP entrante:

```json lines
"attribute_map": {
    . . .
    "attributes": {
        "federated_groups": "${context.userinfo.groups}",
        "federated_locale": "${context.userinfo.locale}",
        "federated_zoneinfo": "${context.userinfo.zoneinfo}"
    }
}
```

<div id="combining-literal-values-and-context-objects">
  #### Combinar valores literales y objetos de contexto
</div>

Este ejemplo muestra cómo combinar valores literales y expresiones dinámicas de plantilla para asignar un valor complejo a un atributo del perfil de usuario de Auth0:

```json lines
"attribute_map":{
    . . .
    "attributes": {
        "alt_id": "user_email|${context.tokenset.email}",
        . . .
    }
}
```
