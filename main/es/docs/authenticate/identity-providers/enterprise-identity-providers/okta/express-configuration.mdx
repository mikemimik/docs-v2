[Okta Integration Network (OIN)](https://www.okta.com/integrations/) es un catálogo de aplicaciones SaaS para las que Okta ofrece una experiencia de configuración exprés que permite habilitar el inicio de sesión único (SSO) con OpenID Connect, el aprovisionamiento automatizado de usuarios con SCIM y [Universal Logout](https://developer.okta.com/docs/guides/oin-universal-logout-overview). Los administradores de Okta usan la [Consola de administración de Okta](https://www.okta.com/okta-administrator-experience) para configurar estas integraciones dentro de su tenant de Okta:

<Frame>![Consola de administración de Okta](/docs/images/cdy7uua7fh8z/okta-admin-dash.png)</Frame>

Express Configuration permite a los clientes empresariales configurar de forma segura integraciones de identidad con aplicaciones SaaS sin necesidad de copiar y pegar valores de configuración específicos del protocolo.

Express Configuration ofrece estos beneficios a los administradores de Okta y a los desarrolladores de aplicaciones SaaS:

* Reduce el tiempo necesario para configurar una instancia de una aplicación al automatizar el intercambio de información de configuración entre Okta y Auth0.
* Utiliza flujos de consentimiento de OAuth 2.0 para compartir de forma segura y autorizada datos de configuración sensibles, lo que reduce los posibles errores relacionados con las credenciales y la configuración.
* Simplifica y estandariza el proceso de despliegue de la integración. El flujo de trabajo automatizado permite despliegues coherentes y repetibles de integraciones de aplicaciones en múltiples clientes o entornos, lo que favorece un ecosistema de aplicaciones escalable con menos probabilidades de error humano.
* Elimina la complejidad de la configuración manual, lo que permite a los administradores de clientes de Okta agregar rápidamente instancias de integraciones de OIN habilitadas para Auth0.

<div id="how-it-works">
  ## Cómo funciona
</div>

La API de Express Configuration permite que las aplicaciones de Auth0 publicadas en el OIN para clientes usen Express Configuration en una [conexión de Okta](/es/docs/authenticate/identity-providers/enterprise-identity-providers/okta). Express Configuration admite OpenID Connect, SCIM y Universal Logout dentro de una organización de Auth0.

Consulta el flujo de trabajo de Express Configuration para una aplicación de Auth0 en el OIN:

<Frame>![flujo de trabajo de Express Configuration para una aplicación de Auth0 en el OIN.](/docs/images/cdy7uua7fh8z/express-config-workflow.png)</Frame>

* Un administrador de Okta inicia sesión en el portal de Okta y selecciona en el OIN la aplicación con Express Configuration habilitada.
* El administrador de Okta va a la sección **Sign On** y selecciona **Express Configure SSO &amp; UL**. Esto lo redirige a una pantalla de [Auth0 Universal Login](/es/docs/authenticate/login/auth0-universal-login/universal-login-vs-classic-login/universal-experience#universal-login-experience).

<Frame>![consola de administrador de Okta > Sign On > Express Configuration](/docs/images/cdy7uua7fh8z/okta-express-config-app.png)</Frame>

* El administrador de Okta introduce las credenciales de un usuario de la aplicación que tiene permiso para realizar Express Configuration. En Auth0, esto se refiere a un usuario que es miembro de una [organización](/es/docs/manage-users/organizations) y está autorizado para realizar Express Configuration mediante un [rol organizacional](/es/docs/manage-users/organizations/configure-organizations/add-member-roles) u otro método de autorización.

<Frame>![inicio de sesión en una organización de Auth0](/docs/images/cdy7uua7fh8z/okta-express-config-auth0-login.png)</Frame>

* Después de autenticarse, Auth0 solicita el consentimiento del administrador de Okta.

<Frame>![consentimiento de Auth0](/docs/images/cdy7uua7fh8z/express-config-auth0-consent.png)</Frame>

* Después de dar su consentimiento, Okta usa la API de Express Configuration para configurar automáticamente una [conexión de Okta](/es/docs/authenticate/identity-providers/enterprise-identity-providers/okta) dentro de la organización de Auth0 a la que pertenece el administrador de Okta.

* Después, el administrador de Okta puede asignar usuarios a la instancia de la aplicación y ver el inicio de sesión único en funcionamiento de inmediato.

Los desarrolladores de Auth0 también pueden configurar su integración del OIN para permitir la Express Configuration de SCIM y Universal Logout, ambos recomendados:

* Cuando SCIM está habilitado, los administradores de Okta pueden configurar SCIM yendo a la sección **Provisioning** de los detalles de la aplicación y seleccionando **Express Configure SCIM**.
* Cuando Universal Logout está habilitado, se configurará automáticamente como parte de la integración de OpenID Connect.

<div id="prerequisites">
  ## Requisitos previos
</div>

Para publicar una aplicación en el OIN con Express Configuration habilitada, debe contar con lo siguiente:

* Una [organización de Okta Integrator Free Plan](https://developer.okta.com/signup/), con acceso al rol Super Admin o a los roles App Admin y Org Admin.
* Una [suscripción de Auth0](https://auth0.com/pricing) que permita usar el tipo de conexión Okta y la funcionalidad Organizations para tantos clientes como necesite.
* Una aplicación SaaS integrada con Auth0 mediante una arquitectura de múltiples organizaciones.
  * Esto incluye registrar su aplicación como una [Regular Web Application](/es/docs/get-started/auth0-overview/create-applications/regular-web-apps) o una [Aplicación de página única](/es/docs/get-started/auth0-overview/create-applications/single-page-web-apps) en Auth0, y permitir que cada uno de sus clientes inicie sesión en esa aplicación con sus propios [proveedores de identidad](/es/docs/authenticate/enterprise-connections).
* Debe haber una [organización de Auth0](/es/docs/manage-users/organizations) implementada, o que pueda implementarse, para cada cliente que use Express Configuration. Organizations y los roles de organización se utilizan para autorizar a usuarios específicos a realizar Express Configuration y crear conexiones de Okta solo dentro de las organizaciones a las que pertenecen.
* Su tenant de Auth0 debe tener deshabilitada la opción de tenant [Enable Application Connections](/es/docs/get-started/tenant-settings#recommended-settings).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  **Consejo**: Para ver una aplicación de ejemplo que implementa una arquitectura multi-tenant, incluidas Auth0 Organizations y los roles de organización, consulte la [aplicación de referencia SaaStart](https://auth0.com/blog/speed-up-your-customer-identity-journey-with-auth0-saastart/).
</Callout>

<div id="configure-your-application-for-express-configuration">
  ## Configure su aplicación para Express Configuration
</div>

Auth0 Dashboard guía al desarrollador de Auth0 para habilitar Express Configuration en su aplicación y publicarla en el OIN.

Se requiere un [rol de Auth0 Admin](/es/docs/get-started/manage-dashboard-access/feature-access-by-role) en un tenant de Auth0 en Production para completar el proceso de configuración y publicación de extremo a extremo.
<Frame>![OIN en Auth0 Dashboard](/docs/images/cdy7uua7fh8z/dashboard-auth0-oin.png)</Frame>
Configure los siguientes componentes de Auth0 para configurar Express Configuration en su tenant:

* Una aplicación registrada con Initiate Login URI Template
* Un [perfil de conexión (CP)](/es/docs/authenticate/enterprise-connections/connection-profile)
* Un [User Attribute Profile](/es/docs/authenticate/enterprise-connections/user-attribute-profile)
* Configuración de inicio de sesión de la organización
* Configuración de inicio de sesión y consentimiento del administrador

<div id="register-an-application-with-initiate-login-uri-template">
  ### Registrar una aplicación con una Initiate Login URI Template
</div>

1. Registra tu aplicación como una [Regular Web Application](/es/docs/get-started/auth0-overview/create-applications/regular-web-apps) o [aplicación de página única](/es/docs/get-started/auth0-overview/create-applications/single-page-web-apps) en Auth0 Dashboard.
2. Una vez registrada, selecciona la aplicación que creaste y ve a la pestaña **Okta Integration Network** para iniciar el asistente de configuración de Express Configuration.
3. Selecciona **Get Started**.
4. Revisa los requisitos previos y selecciona **Continuar**.
5. Registra una **Initiate Login URI Template**. Esta plantilla implementa un endpoint en tu aplicación para redirigir automáticamente a los usuarios finales al endpoint `/authorize` de Auth0 para la autenticación. Para ver un ejemplo de un endpoint de login, consulta la ruta `/login` en el [Express SDK Quickstart](/es/docs/quickstart/webapp/express/interactive).
   * Después de que Express Configuration se inicia para una instancia de aplicación, Okta usa la **Initiate Login URI Template** para iniciar la aplicación desde el [dashboard del usuario final](https://help.okta.com/en-us/content/topics/settings/new-dashboard-overview.htm).
   * (Opcional) Configura `organization_name`, `organization_id,` y `connection_name` para el endpoint `/authorize` de Auth0 con el fin de identificar la organización o conexión que se usará. Okta reemplaza dinámicamente estas variables cuando la URL se inicia desde el dashboard del usuario final. Ejemplo: `https://{organization_name}.your-app.com?connection={connection_name}`.

<Frame>![Dashboard>Applications>OIN>URI-template](/docs/images/cdy7uua7fh8z/Dashboard>Applications>OIN>URI-template.png)</Frame>

**Ejemplos de URI de inicio de sesión:**

`https://your-app.com/login` <br />
`https://your-app.com/login?connection={connection_name}` <br />
`https://{organization_name}.your-app.com?connection={connection_name}` <br />

<div id="connection-profile">
  #### Perfil de conexión
</div>

En la plantilla de URI de inicio de sesión, tienes la opción de agregar un **perfil de conexión**. El [perfil de conexión](/es/docs/authenticate/enterprise-connections/connection-profile) permite a Auth0 especificar cómo debe configurarse la configuración privada de tus conexiones cuando se crean mediante Express Configuration, incluida la configuración que determina cómo interactúa la conexión con las funciones Universal Login y Organizations de Auth0:

* Opciones sobre cómo debe crearse el nombre de la conexión en Auth0
  * Opciones para usar SCIM y/o Universal Logout con la conexión (recomendado)
  * Opciones para permitir que los usuarios finales de la conexión se conviertan automáticamente en miembros de la organización del administrador que otorgó el consentimiento
  * Opciones para la configuración **Mostrar como botón** de la conexión en la página de Universal Login de Auth0

Si no tienes un perfil de conexión configurado, Auth0 proporciona un perfil de conexión predeterminado con configuraciones comunes y recomendadas implementadas para las conexiones configuradas con Express.

Para obtener más información sobre cómo personalizar el CP predeterminado, consulta [perfil de conexión](/es/docs/authenticate/enterprise-connections/connection-profile).

<div id="user-attribute-profile">
  #### User Attribute Profile
</div>

En Initiate Login URI Template, tienes la opción de agregar un **User Attribute Profile**. El [User Attribute Profile (UAP)](/es/docs/authenticate/enterprise-connections/user-attribute-profile) permite a los desarrolladores de Auth0 definir, gestionar y mapear atributos de usuario de forma coherente en los distintos protocolos compatibles con Auth0. Cuando se utiliza con Express Configuration, User Attribute Profile permite a los desarrolladores personalizar los mapeos de atributos de OpenID Connect y SCIM que se escribirán en la conexión de Okta generada por Express Configuration.

Si no tienes configurado un User Attribute Profile, Auth0 proporciona un User Attribute Profile predeterminado con mapeos comunes y recomendados para todos los protocolos, incluidos OpenID Connect y SCIM, que utiliza el tipo de conexión de Okta.

Para saber cómo personalizar el UAP predeterminado, consulta [User Attribute Profile](/es/docs/authenticate/enterprise-connections/user-attribute-profile).

<div id="organization-login-settings">
  ### Configuración de inicio de sesión de la organización
</div>

Auth0 Organizations es necesario para autorizar a usuarios específicos de una organización a crear conexiones aisladas, pero no es necesario cambiar el flujo de inicio de sesión actual de su aplicación para admitir el [flujo de inicio de sesión basado en organizaciones para usuarios empresariales](/es/docs/manage-users/organizations/configure-organizations/define-organization-behavior).

* Si su aplicación usa actualmente una [experiencia de inicio de sesión Identifier-First](/es/docs/authenticate/login/auth0-universal-login/identifier-first) y Home Realm Discovery para sus aplicaciones, consulte la sección [Habilitar Home Realm Discovery](/es/docs/authenticate/identity-providers/enterprise-identity-providers/okta/express-configuration#enable-home-realm-discovery) para obtener información sobre cómo habilitar Home Realm Discovery.
* Si desea habilitar varias aplicaciones en `enabled_clients` para usarlas con sus conexiones de Express Configuration, consulte Habilitar varias aplicaciones de Auth0 en una sola integración.

Si la configuración de su aplicación no usa actualmente un flujo de inicio de sesión basado en organizaciones, puede seleccionar la opción **Habilitar esta aplicación para las conexiones creadas**. Al habilitarla, cada Express Configuration de una conexión de Okta se asocia directamente con su aplicación. Use la configuración obligatoria `enabled_clients` en la conexión.

<div id="administrator-login-and-consent-settings">
  ### Configuración del inicio de sesión y consentimiento del administrador
</div>

Cuando Express Configuration está habilitado, Auth0 crea un ID adicional de aplicación cliente que la OIN usa durante el flujo de consentimiento del administrador entre Okta y Auth0. Okta crea un cliente OIN por cada integración de aplicación distinta publicada en la OIN.

Configure los ajustes de inicio de sesión y consentimiento del administrador en la aplicación que desea publicar en la OIN, en **Configure Integration Profile**.

1. Vaya a [Auth0 Dashboard &gt; Applications](https://manage.auth0.com/dashboard/#/applications).
2. Elija la aplicación que desea publicar en la OIN.
3. Seleccione Okta Integration Network.
4. Verifique que cumple con los requisitos previos necesarios y seleccione **Continuar**.
5. En la sección **Configure Integration Profile**, configure las opciones en **Admin Settings**.

<Frame>![Configuración de administrador de Express Configuration](/docs/images/cdy7uua7fh8z/Express-config-admin-settings.png)</Frame>

* **Admin Login Domain**: Dominio del tenant de Auth0 al que Okta redirige como parte del flujo de consentimiento en un navegador web. Use el nombre de dominio personalizado si tiene uno configurado en Auth0. Para obtener más información, consulte [Custom Domains](docs/customize/custom-domains).
  * **Display Name of the OIN Express Configuration Application**: Nombre de la aplicación cliente de OIN que se muestra en el cuadro de diálogo de consentimiento.

  * **Admin Login Flow**: Define el flujo de [inicio de sesión de la organización](/es/docs/manage-users/organizations/login-flows-for-organizations) para la aplicación cliente OIN y se usa cuando un administrador de Okta ejecuta Express Configuration desde la consola de Okta. Seleccione una de estas opciones:

    * **Prompt for Organization**: Primero se pide a los administradores que seleccionen su organización y se les muestra la experiencia de inicio de sesión de su Auth0 Organization. Los administradores pueden iniciar sesión con una [conexión preexistente](/es/docs/manage-users/organizations/configure-organizations/enable-connections) configurada en esa organización.

    * **Prompt for Credentials**: Primero se pide a los administradores que proporcionen sus credenciales de inicio de sesión. Cuando se selecciona esta opción, aparecerá un botón que le permitirá seleccionar las conexiones que contienen a los usuarios administradores. Puede tratarse de una conexión de base de datos compartida, una conexión de correo electrónico passwordless u otra conexión que pueda asociarse a todas las Auth0 Organizations.

  * **Admin Role**:  Para realizar Express Configuration, al administrador se le deben asignar los permisos adecuados. Consulte [Asignar permisos de Express Configuration a los usuarios](/es/docs/authenticate/identity-providers/enterprise-identity-providers/okta/express-configuration#assign-permissions-to-users) para obtener instrucciones sobre cómo autorizar a los administradores de la forma que mejor se adapte a su despliegue actual de Auth0.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Para ofrecer la mejor experiencia de consentimiento, establezca el indicador `use_scope_descriptions_for_consent` de su tenant en `true`, como se describe en [Customize Consent Prompts](/es/docs/customize/login-pages/customize-consent-prompts). Más adelante, cuando pruebe y verifique su integración, tendrá la oportunidad de ver su solicitud de consentimiento.
</Callout>

<div id="assign-permissions-to-users">
  ## Asignar permisos a los usuarios
</div>

En Okta, los administradores deben tener una cuenta de usuario de la aplicación con los permisos necesarios para realizar la Express Configuration de una aplicación SaaS con tecnología de Auth0.

En Auth0, este permiso puede concederse a cualquier usuario de la aplicación, siempre que sea miembro de una [Auth0 Organization](/es/docs/manage-users/organizations) en la que se haya creado la conexión configurada con Express, incluidos los siguientes casos:

* Un usuario en una conexión de base de datos dedicada creada exclusivamente para una sola Organization
* Un usuario en una conexión de base de datos compartida que sea miembro de una o más Organizations
* Un usuario con una conexión de correo electrónico passwordless que sea miembro de una o más Organizations
* Un usuario en una conexión social o en una conexión empresarial existente que sea miembro de una o más Organizations

Una vez cumplidas estas condiciones, Auth0 ofrece formas flexibles de asignar permisos de Express Configuration y permite a los desarrolladores elegir el método más adecuado para su despliegue actual de Auth0:

| **Método**                                                                                                                                                                                                                                            | **Recomendado para...**                                                                                                                                                                                                                                                                                                                   |
| ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Asignar permisos de Express Configuration a un rol de usuario existente                                                                                                                                                                               | Clientes con un [rol](/es/docs/manage-users/access-control/configure-core-rbac/roles) de usuario de aplicación ya existente en su tenant de Auth0.                                                                                                                                                                                           |
| Asignar un nuevo rol a los usuarios de la aplicación que necesiten permisos de Express Configuration, como se muestra en la [aplicación de referencia SaaStart](https://auth0.com/blog/speed-up-your-customer-identity-journey-with-auth0-saastart/). | Clientes que quieran usar la funcionalidad de [rol](/es/docs/manage-users/access-control/configure-core-rbac/roles) de Auth0 para conceder permisos de Express Configuration a usuarios individuales. Esto puede hacerse con Auth0 Dashboard o la Management API.                                                                            |
| Usar una Action de Post Login para asignar permisos dinámicamente mediante control de acceso basado en atributos                                                                                                                                         | Clientes que actualmente no usan la funcionalidad de [rol](/es/docs/manage-users/access-control/configure-core-rbac/roles) de Auth0 y prefieren usar una Action de Post Login para asignar permisos dinámicamente según los atributos del usuario, en lugar de usar el Auth0 Dashboard o hacer llamadas a la Management API para asignar roles. |

Para obtener consejos sobre el aprovisionamiento de cuentas de administrador para usar Express Configuration, consulta [Customer Enablement](/es/docs/authenticate/identity-providers/enterprise-identity-providers/okta/express-configuration#customer-enablement).

<div id="assign-permissions-to-a-pre-existing-application-user-role">
  ### Asignar permisos a un rol de usuario de aplicación existente
</div>

Usa este método para asignar permisos de Express Configuration si utilizas la [función RBAC de Auth0](/es/docs/manage-users/access-control/configure-core-rbac/roles) para los roles de aplicación y los permisos de API, y ya tienes uno o más roles que representan a un usuario administrador y que se asignarían al administrador de TI encargado de implementar la aplicación.

Permisos de API requeridos:

| **Nombre del permiso**   | **Identificador de API (Resource Server)** |
| ------------------------ | ------------------------------------------ |
| `express_configure:sso`  | `urn:auth0:express-configure`              |
| `express_configure:scim` | `urn:auth0:express-configure`              |

Para obtener más información sobre cómo asignar roles en Auth0 Dashboard y Management API, consulta [Agregar permisos a roles](/es/docs/manage-users/access-control/configure-core-rbac/roles/add-permissions-to-roles).

<div id="assign-a-new-role-to-application-users">
  ### Asignar un nuevo rol a los usuarios de la aplicación
</div>

Usa este método para asignar permisos de Express Configuration si utilizas la [función RBAC de Auth0](/es/docs/manage-users/access-control/configure-core-rbac/roles) para los roles de la aplicación y los permisos de la API, pero no tienes un rol que represente a un usuario administrador, como un administrador de TI, a quien se le asignaría desplegar la aplicación.

Este método también es adecuado para los desarrolladores de Auth0 que actualmente no usan la [función RBAC de Auth0](/es/docs/manage-users/access-control/configure-core-rbac/roles), pero quieren utilizarla para tener un control granular sobre qué usuarios pueden acceder, ya sea mediante el Auth0 Dashboard o la Management API.

<div id="create-a-role">
  #### Crear un rol
</div>

Usa Auth0 Dashboard, la Management API o [Auth0 CLI](https://auth0.github.io/auth0-cli/auth0_roles_create.html) para [crear roles](/es/docs/manage-users/access-control/configure-core-rbac/roles/create-roles). Este ejemplo usa Auth0 CLI:

```bash
auth0 roles create \
  --name "EXPRESS_CONFIGURE_ADMIN_ROLE" \
  --description "Administrator role for Express Configuration"
```

<div id="assign-permissions-to-the-role">
  #### Asigna permisos al rol
</div>

Asigna los permisos `express_configuration:sso` y `express_configuration:scim` al rol especificado. Reemplaza `$ROLE_ID` por el ID del rol al que quieres conceder los permisos.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Al usar este método, tendrás que actualizar tu proceso de incorporación de clientes para asignar este rol a cualquier usuario nuevo de la organización que necesite permisos de Express Configuration.
</Callout>

```bash
auth0 roles permissions add "$ROLE_ID" \
  --api-id "urn:auth0:express-configure" \
  --permissions "express_configure:sso"
```

```bash
auth0 roles permissions add "$ROLE_ID" \
  --api-id "urn:auth0:express-configure" \
  --permissions "express_configure:scim"
```

Para obtener más información sobre cómo asignar roles a los usuarios existentes de la organización, consulta [Agregar roles de miembro](/es/docs/manage-users/organizations/configure-organizations/add-member-roles).

<div id="assign-permissions-based-on-user-attributes-using-a-post-login-action">
  ### Asigna permisos según los atributos del usuario usando una Action de Post Login
</div>

Usa este método para asignar permisos en Express Configuration si estás usando una [Action de Post Login](/es/docs/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger) de Auth0 para asignar permisos dinámicamente según los atributos del usuario, en lugar de usar Role-Based Access Control (RBAC), Management API o Auth0 Dashboard para asignar roles a los usuarios.

Esto es adecuado para los clientes de Auth0 que han implementado Auth0 con un modelo de autorización basado en atributos y permisos personalizados en los [metadatos de Auth0](/es/docs/manage-users/user-accounts/metadata).

**Ejemplo** <br />
El siguiente ejemplo usa una Action de Post Login para asignar permisos según el valor de un atributo personalizado existente del usuario, `user.app_metadata.is_admin`.

```bash
/**
* Asigna permisos de Express Configuration basados en lógica personalizada en lugar de asignación de roles
*/
exports.onExecutePostLogin = async (event, api) => {


 //verificar si la solicitud es para un token de acceso a la API de EC
 if (event.resource_server && event.resource_server.identifier === "urn:auth0:express-configure") {


   //agregar permisos según una condición personalizada
   if (event.user.app_metadata && event.user.app_metadata.is_admin === true) {
     api.accessToken.addScope("express_configure:sso");
     api.accessToken.addScope("express_configure:scim");
   }
   //de lo contrario, denegar el acceso si se solicita el token de acceso de EC
   else {
     api.access.deny('Access denied');
   }
 }
};
```

<div id="enable-home-realm-discovery">
  ## Habilitar Home Realm Discovery
</div>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Antes de ejecutar Express Configuration, debes recopilar y verificar las direcciones de correo electrónico de los clientes como parte del proceso de incorporación para habilitar HRD.
</Callout>

Puedes usar Express Configuration con Auth0 Actions si tu aplicación se basa en la [experiencia de login Identifier-First](/es/docs/authenticate/login/auth0-universal-login/identifier-first) y usa Home Realm Discovery (HRD) para hacer coincidir a los usuarios con sus proveedores de identidad (IdP) según su dirección de correo electrónico.

Las direcciones de correo electrónico deben almacenarse en una ubicación a la que puedan acceder las Actions de Post Login durante el workflow de Express Configuration, lo que incluye:

* Puedes almacenar uno o varios dominios de correo electrónico en la [metadata de la organización](/es/docs/manage-users/organizations/configure-organizations/create-organizations) de la Auth0 Organization del cliente.
* Tu Action de Post Login puede hacer una llamada a la API para recuperar dominios verificados de cualquier sistema de tu entorno donde estén almacenados.

Cuando el usuario administrador inicia la configuración exprés en el portal de Okta y se autentica, estas Actions agregan los dominios de correo electrónico al token que recibe Okta, que luego los extrae y los usa para configurar la conexión de Okta con la configuración de HRD correcta.

**Ejemplo 1** <br />
En este ejemplo, los dominios verificados se almacenan en la metadata de la conexión de la Auth0 Organization, lo que requiere una cadena delimitada por comas con uno o varios dominios de correo electrónico almacenados en la metadata de la organización bajo una clave llamada `domains`.

Valores de ejemplo: `test.com`,`test2.com`

```javascript
exports.onExecutePostLogin = async (event, api) => {
if (event?.resource_server?.identifier === "urn:auth0:express-configure") {
  if (event.organization && event.organization.metadata && event.organization.metadata.domains)
  {
    var domain_aliases = event.organization.metadata.domains.replace(/ /g,'').split(',');
    var express_configuration = {
      "domain_aliases": domain_aliases
     };
     api.accessToken.setCustomClaim("express_configuration", express_configuration);
  }
}
};
```

**Ejemplo 2** <br />
En este ejemplo, la Action de Post Login realiza una llamada a la API para recuperar dominios verificados desde un sistema de almacenamiento de tu entorno.

```javascript
/**

exports.onExecutePostLogin = async (event, api) => {
 if (event?.resource_server?.identifier === "urn:auth0:express-configure") {

    // configura aquí tu llamada a la API personalizada 
    const domains = await axios.get("https://example.org/endpoint");
```

<div id="maintain-admin-account-matching-across-connections">
  ### Mantener la correspondencia de las cuentas de administrador entre conexiones
</div>

Auth0 permite aprovisionar cuentas de usuario con la misma dirección de correo electrónico en diferentes conexiones. Un usuario final puede tener una conexión de base de datos, una cuenta social o passwordless con su dirección de correo electrónico del trabajo, y también tener esa misma dirección en su cuenta federada de Okta.

Al usar la opción **Prompt for Credentials** en el [flujo de inicio de sesión y consentimiento del administrador](/es/docs/authenticate/identity-providers/enterprise-identity-providers/okta/express-configuration#provision-admin-accounts), Home Realm Discovery comienza a asociar a los usuarios con un sufijo de dominio específico a la nueva conexión de Okta, en lugar de a otros tipos de conexión, después de que se agrega a la organización. Para obtener más información sobre este comportamiento de inicio de sesión de la organización, consulta [Identifier First Authentication with prompt for credentials](/es/docs/manage-users/organizations/login-flows-for-organizations#identifier-first-authentication-with-prompt-for-credentials).

Este comportamiento solo surte efecto cuando los clientes ejecutan Express Configuration por segunda vez después de que caduque la sesión inicial del usuario administrador de Auth0. Si el identificador de la cuenta de administrador es una dirección de correo electrónico que coincide con la nueva conexión de Okta, se redirigirá al administrador allí.

Este comportamiento se puede gestionar o evitar con cualquiera de los siguientes métodos:

* Usa la opción **Prompt for Organization** en el [flujo de inicio de sesión y consentimiento del administrador](/es/docs/authenticate/identity-providers/enterprise-identity-providers/okta/express-configuration#provision-admin-accounts).
  * Habilita los permisos de Express Configuration para la nueva cuenta de Okta que contiene la dirección de correo electrónico verificada correspondiente después de aprovisionarla.
  * Si solo usas Auth0 Organizations para el flujo de consentimiento del administrador de Express Configuration y no en la experiencia de tu aplicación, puedes resolver el problema deshabilitando la propiedad `enable_organization` descrita en [Configure SaaS Application Properties](/es/docs/authenticate/identity-providers/enterprise-identity-providers/okta/express-configuration#configure-saas-application-properties).
  * Esto no supone un problema si el identificador del usuario administrador no es una dirección de correo electrónico, como un nombre de usuario en una conexión de base de datos.

<div id="enable-multiple-applications-under-a-single-integration">
  ## Habilita varias aplicaciones en una sola integración
</div>

Si quieres que los usuarios finales de una única Express Configuration en una conexión de Okta accedan a varias aplicaciones de tu tenant, puedes establecer una propiedad `linked_clients` en tu aplicación web registrada.

Para modificar esta configuración, usa Auth0 Management API para [obtener tu aplicación web registrada](https://auth0.com/docs/api/management/v2/clients/get-clients-by-id). Reemplaza `{yourAppId}` por el ID de cliente de tu aplicación registrada y `{yourAccessToken}` por un token de acceso de la Management API v2. Para saber cómo obtener un token de acceso para usar la Management API, consulta [Management API Access Tokens](/es/docs/secure/tokens/access-tokens/management-api-access-tokens).

```curl
curl --request GET \
  --url 'https://{yourDomain}/api/v2/clients/{yourAppId}' \
  --header 'authorization: Bearer {yourAccessToken}'
```

Extrae la propiedad `express_configuration` de la respuesta, añade un arreglo de IDs de aplicación adicionales a la propiedad `linked_clients` y [actualiza tu aplicación web registrada](https://auth0.com/docs/api/management/v2/clients/patch-clients-by-id):

```curl
curl --request PATCH \
  --url 'https://{yourDomain}/api/v2/clients/{yourAppId}' \
  --data '{"express_configuration": 
  {"admin_login_domain":"TENANT.auth0.com",
    "connection_profile_id":"cop_xxxxxxxxxxxxxxx",
    "enable_client":true,
    "enable_organization":true,
    "initiate_login_uri_template":"https://example.org",
    "okta_oin_client_id":"LDiGbUeiAYjRB5a4yOGfBvxxxxxxxxxxx",
    "user_attribute_profile_id":"uap_1ctMVQUg8jxxxxxxxxxxxx",   

     "linked_clients": [ 
        { "client_id": "KJM86F2susguvsasSeAsIxxxxxxxxxxx" }, 
	  { "client_id": "FIXwZCf5iUElvqy6eidlfxxxxxxxxxxx" }
      ] 
     }
  }' \
  --header 'cache-control: no-cache' \
  --header 'content-type: application/json' \
  --header 'authorization: Bearer {yourAccessToken}'
```

Cuando se ejecute Express Configuration, todos estos ID de aplicación de la propiedad linked&#95;clients se agregarán a la propiedad `enabled_clients` de la conexión de Okta.

<div id="example-customer-enablement-flow">
  ### Flujo de ejemplo de habilitación del cliente
</div>

En tu proceso de incorporación de clientes, debes recopilar:

* El nombre de la organización del cliente
* La dirección de correo electrónico del administrador del cliente que debe tener permisos para realizar la Express Configuration
* Los dominios de correo electrónico verificados que el IdP del cliente emite para los usuarios, incluidos los del administrador

Recopila esta información manualmente o mediante la experiencia de registro o incorporación de tus clientes. Usa esta información para crear una organización de Auth0 y una cuenta de usuario administrador para este cliente:

1. Crea una organización de Auth0.
2. Crea la cuenta de usuario administrador en una conexión de Auth0, como una base de datos o email passwordless.
3. Asocia la conexión con la organización de Auth0. Desactiva **Membership On Authentication**.
4. Agrega la cuenta de usuario administrador como miembro de la organización de Auth0.
5. Asigna al usuario administrador un rol organizacional con permisos para realizar la Express Configuration.

<div id="auth0-cli-example">
  #### Ejemplo de Auth0 CLI
</div>

**Inicializa Auth0 CLI y autentícate**
Autentícate en Auth0 con [Auth0 CLI](https://auth0.github.io/auth0-cli/auth0_roles_create.html).

```bash
auth0 login --scopes "create:users,create:organizations,create:organization_members,create:organization_member_roles,create:organization_connections"
```

**Crea una organización de Auth0**
Introduce el nombre de la organización y un nombre corto sin espacios. Si recopilaste varios sufijos de correo electrónico, agrégalos como metadatos de la organización.

```bash
auth0 orgs create --json \
--display "organization_display_name" \
--name "organization_short_name" \
--metadata 'domains=["domain1.com", "domain2.com"]'
```

Recopila el `org_id` devuelto.

**Crea la cuenta de administrador de la organización**
En este ejemplo, creas un usuario en una conexión passwordless por correo electrónico, lo que requiere el tipo de conexión [`email`](/es/docs/authenticate/passwordless/authentication-methods/email-otp). El comando requiere que introduzcas la dirección de correo electrónico del usuario al que autorizas para realizar Express Configuration.

```bash
auth0 api users \
--data '{"email":"user@example.com","connection":"email", "email_verified":true}'
```

<Card title="Consejo">
  Si está creando esta organización para proporcionar una cuenta de prueba con nombre de usuario y contraseña al equipo de operaciones de Okta OIN, cree la cuenta de prueba en una conexión de base de datos compartida o específica:

  ```bash
  auth0 api users \
  --data '{"email":"user@example.com","connection":"db_connection_name_here", "password": "add_a_long_password_here"}'
  ```
</Card>

Anote el `user_id` devuelto.

**Asocie la conexión con la organización**

Obtenga el ID de la conexión:

```bash
auth0 api get "connections" -q "name=connection_name_here"
```

Asocie el ID de conexión con el Organization ID, `org_id`.

```bash
auth0 api post "organizations/org_id_here/enabled_connections" \
--data '{ "connection_id": "connection_id_here" };
```

**Añade el usuario Admin como miembro de la organización**
Necesitas el `org_id` y el `user_id`.

```bash
auth0 api post "organizations/org_id_here/members" \
--data '{ "members": ["user_id__here"] }'

```

**Asigna el rol de la organización con permisos de Express Configuration**
Obtén el ID del rol con permisos de Express Configuration, tal como se configuró en [Asignar permisos de Express Configuration a los usuarios](/es/docs/authenticate/identity-providers/enterprise-identity-providers/okta/express-configuration#assign-permissions-to-users).

```bash
auth0 api get "roles" -q "name_filter=role_name_here"
```

Asigna el ID del rol a `user_id` y `org_id`.

```bash
auth0 api post "organizations/org_id_here/members/user_id_here/roles" --data '{ "roles": ["role_id_here"] }'
```

Una vez completado, su cliente podrá usar la cuenta de administrador de la organización para realizar Express Configuration en su organización de Okta con su [integración de OIN](/es/docs/authenticate/identity-providers/enterprise-identity-providers/okta/express-configuration#publish-your-integration-to-the-oin).

<div id="recommendations-to-provision-admin-accounts">
  ### Recomendaciones para aprovisionar cuentas de Admin
</div>

Tanto si utilizas cuentas de usuario existentes como si implementas otras nuevas para habilitar Express Configuration, sigue las recomendaciones que se indican a continuación.

<div id="use-work-email">
  #### Usar el correo electrónico laboral
</div>

Tanto si la cuenta de administrador no federada es una cuenta de usuario de base de datos, passwordless por correo electrónico o social, lo mejor es que el valor del atributo `email` coincida con la dirección de correo electrónico laboral del usuario que se aprovisionará con su cuenta de usuario empresarial de Okta (por ejemplo: `john@mycompany.com`).

Auth0 también recomienda verificar estas direcciones de correo electrónico mediante un [flujo de verificación de correo electrónico](/es/docs/manage-users/user-accounts/verify-emails).

<div id="use-multi-factor-authentication">
  #### Usa autenticación multifactor
</div>

Puedes añadir una capa adicional de seguridad al flujo de Express Configuration exigiendo autenticación multifactor en cada inicio de sesión.

El ejemplo de [Action de Post Login](/es/docs/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger) muestra cómo puedes requerir de forma condicional factores como [WebAuthN con biometría del dispositivo](/es/docs/secure/multi-factor-authentication/fido-authentication-with-webauthn/configure-webauthn-device-biometrics-for-mfa) y [contraseña de un solo uso enviada por correo electrónico](/es/docs/secure/multi-factor-authentication/multi-factor-authentication-factors/configure-email-notifications-for-mfa) cada vez que se ejecuta Express Configuration.

La Action de ejemplo requiere [habilitar estos factores](/es/docs/secure/multi-factor-authentication/enable-mfa#enable-mfa-in-the-auth0-dashboard) en tu tenant de Auth0 y seleccionar la opción **Personalizar los factores de MFA con Actions**.

```javascript
exports.onExecutePostLogin = async (event, api) => {


if (event.resource_server && event.resource_server.identifier === "urn:auth0:express-configure") {


   api.multifactor.enable('any');
  api.authentication.challengeWith({ type: 'email' });


  if (!event.user.enrolledFactors.some(m => m.type === 'webauthn-platform')) {
     api.authentication.enrollWith({type: 'webauthn-platform'});
   } else {
     api.authentication.challengeWith({type: 'webauthn-platform'});
  }
 }
}
```

<div id="monitor-express-configuration-usage">
  ### Supervisa el uso de Express Configuration
</div>

El uso de Express Configuration puede observarse en los [registros del tenant de Auth0](/es/docs/deploy-monitor/logs) filtrando por el **OIN Client ID** de cada aplicación que hayas configurado. Esto incluye toda la actividad de inicio de sesión de los usuarios administradores, así como todas las operaciones de la API para crear y gestionar las conexiones empresariales de Okta.

El OIN Client ID de tu aplicación puede consultarse en estos lugares:

* En la sección **Applications &gt; [Application] &gt; Okta Integration Network &gt; Create OIN Integration** del Auth0 Dashboard.

  * En la propiedad `express_configuration.okta_oin_client_id` de tu aplicación al consultarla en la [Auth0 Management API](https://auth0.com/docs/api/management/v2/clients/get-clients-by-id).

Para obtener más información sobre cómo buscar logs con el ID de cliente, consulta [Sintaxis de las consultas de búsqueda en logs](/es/docs/deploy-monitor/logs/log-search-query-syntax).

<div id="limitations">
  ## Limitaciones
</div>

* Solo se puede usar un tenant de Auth0 en una sola integración de OIN. Si la misma aplicación se implementa en varias regiones de Auth0, cada región requiere su propia integración de OIN
* Una integración de OIN en una organización de Okta puede crear una única conexión de Okta en la organización de Auth0 correspondiente. Express Configuration no admite más de una instancia de una aplicación de OIN dentro de una organización de Okta
* Cada integración única incluida en OIN crea una conexión de Okta independiente. Si tiene varias aplicaciones que requieren una [experiencia de inicio de sesión Identifier-First](https://auth0.com/docs/authenticate/login/auth0-universal-login/identifier-first) con Home Realm Discovery, le recomendamos publicar [varias aplicaciones en una sola integración](/es/docs/authenticate/identity-providers/enterprise-identity-providers/okta/express-configuration#enable-multiple-applications-under-a-single-integration).
* Al habilitar [varias aplicaciones en una sola integración](/es/docs/authenticate/identity-providers/enterprise-identity-providers/okta/express-configuration#enable-multiple-applications-under-a-single-integration), Okta solo muestra la aplicación principal de OIN en su [panel para el usuario final](https://help.okta.com/en-us/content/topics/settings/new-dashboard-overview.htm).

<div id="management-api-reference">
  ## Referencia de la Management API
</div>

Para usar la [Management API](https://auth0.com/docs/api/management/v2) en lugar de Auth0 Dashboard para habilitar Express Configuration en una aplicación determinada, usa los ejemplos siguientes.

<div id="create-the-okta-oin-express-configuration-system-api">
  ### Crear la API del sistema de Express Configuration de Okta OIN
</div>

Registra el servidor de recursos que se utilizará para autenticar a los administradores de la organización. Esto solo es necesario si el Dashboard nunca se usó para configurar Express Configuration para aplicaciones en un tenant.

Ejemplo:

```curl
POST /api/v2/resource-servers
{
    "name": "Okta OIN Express Configuration API",
    "identifier": "urn:auth0:express-configure"
  }

```

[Referencia del endpoint](/es/docs/api/management/v2/resource-servers/post-resource-servers)

<div id="create-user-attribute-profile">
  ### Create perfil de atributos de usuario
</div>

Crea un [perfil de atributos de usuario](/es/docs/authenticate/enterprise-connections/user-attribute-profile). Este paso solo es necesario si no existen perfiles o si el desarrollador quiere usar un perfil personalizado.

Para comenzar con un conjunto de valores predeterminados para el perfil de atributos de usuario, llama primero al endpoint `GET /api/v2/user-attribute-profiles/templates` y usa la respuesta en el cuerpo de la solicitud para crear el perfil de atributos de usuario.

Ejemplo:

```curl
GET /api/v2/user-attribute-profiles/templates
```

[Referencia del endpoint](https://auth0.com/docs/api/management/v2/user-attribute-profiles/post-user-attribute-profiles)

```curl
POST /api/v2/user-attribute-profiles
{
    "name": "Okta OIN Express Configuration API",
    "identifier": "urn:auth0:express-configure"
}
```

[Referencia del endpoint](https://auth0.com/docs/api/management/v2/user-attribute-profiles/post-user-attribute-profiles)

<div id="create-connection-profile">
  ### Crear perfil de conexión
</div>

Cree un perfil de conexión. Este paso solo es necesario si no hay perfiles existentes o si el desarrollador quiere usar un perfil personalizado.

Para partir de un conjunto de valores predeterminados para el perfil de conexión, llame al endpoint `GET /api/v2/connection-profiles/templates` y use la respuesta en el cuerpo de la llamada para crear el perfil de conexión.

Ejemplo:

```curl
GET /api/v2/connection-profiles/templates
```

[Referencia del endpoint](https://auth0.com/docs/api/management/v2/connection-profiles/get-connection-profile-templates)

```curl
POST /api/v2/connection-profile
{
    "name": "Okta OIN Express Configuration API",
    "identifier": "urn:auth0:express-configure"
}
```

[Referencia del endpoint](https://auth0.com/docs/api/management/v2/connection-profiles/post-connection-profiles)

<div id="create-oin-express-configuration-client">
  ### Crear el cliente de OIN Express Configuration
</div>

Cree la aplicación de servicio que Okta usará durante Express Configuration para una aplicación determinada. Este cliente debe crearse con las propiedades que se muestran en el ejemplo siguiente.

El atributo `organization_require_behavior` puede personalizarse con uno de los siguientes valores, que también se explican en [Configuración de inicio de sesión y consentimiento del administrador](/es/docs/authenticate/identity-providers/enterprise-identity-providers/okta/express-configuration#administrator-login-and-consent-settings).

* `pre_login_prompt`: Solicita la organización del administrador antes de pedir sus credenciales.
  * `post_login_prompt`: Solicita las credenciales del administrador. Para esta opción, use el endpoint `PATCH /api/v2/connections/{id}` para agregar el ID de cliente del cliente OIN a la propiedad `enabled_clients` de la(s) conexión(es) que contienen las cuentas de administrador.

Ejemplo:

```curl
POST /api/v2/clients
{
    "name": "Okta OIN Express Configuration",
    "app_type": "express_configuration",
    "organization_require_behavior": "pre_login_prompt",
    "client_authentication_methods": {
      "private_key_jwt": {
        "credentials": []
      }
    }
}
```

[Referencia del endpoint](https://auth0.com/docs/api/management/v2/clients/post-clients)

<div id="configure-saas-application-properties">
  ### Configurar las propiedades de la aplicación SaaS
</div>

Establece los valores de configuración de la propiedad `express_configuration` en la aplicación registrada que se publicará en la OIN. Sustituye los valores de los ejemplos por un ID de perfil de conexión válido, un ID de perfil de atributos de usuario, un ID de la aplicación cliente de OIN, un URI de inicio de sesión y el dominio del tenant de Auth0 que se utilizarán para esta integración.

El atributo `linked_clients` corresponde a la configuración descrita en [Habilitar varias aplicaciones en una sola integración](/es/docs/authenticate/identity-providers/enterprise-identity-providers/okta/express-configuration#enable-multiple-applications-under-a-single-integration).

El atributo `enable_client` corresponde a la configuración de `enabled_clients` descrita en [Configuración de inicio de sesión de la organización](/es/docs/authenticate/identity-providers/enterprise-identity-providers/okta/express-configuration#organization-login-settings).

Normalmente, el atributo `enable_organization` debe establecerse en `true`, a menos que no estés asignando las conexiones creadas a sus organizaciones. La instancia de la aplicación de Okta puede seguir administrando la conexión, pero los usuarios de Okta no pasarán a ser miembros de la organización de Auth0. Establecerlo en `false` puede resultar útil en el caso descrito en [Mantener la coincidencia de cuentas de administrador entre conexiones](/es/docs/authenticate/identity-providers/enterprise-identity-providers/okta/express-configuration#maintain-admin-account-matching-across-connections).

Ejemplo:

```curl
PATCH /api/v2/clients/{id}
{
  "express_configuration": {
      "admin_login_domain": "yourAuth0TenantDomain",
      "connection_profile_id": "yourConnectionProfileId",
      "enable_client": true,
      "enable_organization": true,
      "initiate_login_uri_template": "yourInitiateLoginUriTemplate",
      "okta_oin_client_id": "yourOinClientId",
      "user_attribute_profile_id": "yourConnectionProfileId",
"linked_clients": [ 
    { "client_id": "KJM86F2susguvsasSeAsIxxxxxxxxxxx" }, 
    { "client_id": "FIXwZCf5iUElvqy6eidlfxxxxxxxxxxx" }
 ] 
    }
}
```

[Referencia del endpoint](https://auth0.com/docs/api/management/v2/clients/patch-clients-by-id)

[Referencia del endpoint](https://auth0.com/docs/api/management/v2/clients/post-credentials)

<div id="assign-keys-to-oin-client-application">
  ### Asigna claves a la aplicación cliente de OIN
</div>

Asigna las claves públicas cargadas desde Okta al cliente de servicio de OIN. Sustituye `yourCredentialId` por el ID de la credencial recibido en el paso anterior.

Ejemplo:

```curl
PATCH /api/v2/clients/{oin_client_id}
{
 "client_authentication_methods": {
    "private_key_jwt": {
      "credentials": [
        {
          "id": "yourCredentialId",
        }
      ]
    }
  }
}
```

[Referencia del endpoint](https://auth0.com/docs/api/management/v2/clients/patch-clients-by-id)

<div id="customize-your-tenants-consent-prompt">
  ### Personaliza la solicitud de consentimiento de tus tenants
</div>

Actualiza la Configuración del tenant para mostrar los detalles de los scopes en la página de consentimiento. Estas opciones mejoran la experiencia del usuario al proporcionar información sobre los permisos que se van a conceder. No es obligatorio, pero sí recomendable.

Ejemplo:

```curl
PATCH /api/v2/tenants/settings
{ 
  "flags": { "use_scope_descriptions_for_consent": true } 
}
```

[Referencia del endpoint](https://auth0.com/docs/api/management/v2/tenants/patch-settings)
