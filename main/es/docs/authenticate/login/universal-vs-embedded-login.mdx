---
title: Inicio de sesión alojado vs. inicio de sesión integrado
description: Describe las diferencias entre el inicio de sesión alojado (Universal Login) y el inicio de sesión integrado.
---

Cuando diseñes la experiencia de inicio de sesión de tu aplicación, tendrás que decidir si quieres que sea alojada ([Universal Login](/es/docs/authenticate/login/auth0-universal-login)) o integrada.

<div id="hosted-login">
  ## Inicio de sesión alojado
</div>

En un flujo de inicio de sesión alojado, la aplicación redirige al usuario al <Tooltip tip="Servidor de autorización: servidor centralizado que contribuye a definir los límites del acceso de un usuario. Por ejemplo, su servidor de autorización puede controlar los datos, las tareas y las funciones disponibles para un usuario." cta="Ver glosario" href="/es/docs/glossary?term=authorization+server">servidor de autorización</Tooltip> para autenticarse, y luego el servidor de autorización lo redirige de vuelta a la aplicación.

<div id="universal-login-flow">
  ### Flujo de Universal Login
</div>

En la experiencia típica de autenticación de usuarios con <Tooltip tip="Universal Login: Tu aplicación redirige a Universal Login, alojado en el servidor de autorización de Auth0, para verificar la identidad de un usuario." cta="Ver glosario" href="/es/docs/glossary?term=Universal+Login">Universal Login</Tooltip>:

1. Tu aplicación inicia una solicitud de autenticación al invocar el método de inicio de sesión de un SDK de Auth0 o al llamar directamente al [endpoint de inicio de sesión de Auth0 Authentication API](https://auth0.com/docs/api/authentication/login/).
2. Tu aplicación redirige al usuario al servidor de autorización de Auth0 (tu tenant de Auth0).
3. Auth0 comprueba si existe una cookie de sesión en la [capa de sesión de Auth0](/es/docs/manage-users/sessions/session-layers).

   * Si no encuentra una cookie de sesión, Auth0 redirige al usuario a Universal Login para que se autentique y crea la cookie de sesión correspondiente.
   * Si encuentra una cookie de sesión, Auth0 la utiliza para autenticar al usuario.
4. Auth0 redirige de vuelta al usuario a tu aplicación con un [ID token](/es/docs/secure/tokens/id-tokens) y un [token de acceso](/es/docs/secure/tokens/access-tokens).

<div id="embedded-login">
  ## Inicio de sesión integrado
</div>

En un flujo de inicio de sesión integrado, el usuario se autentica directamente en su aplicación. La aplicación muestra el formulario de inicio de sesión, recopila las credenciales del usuario, envía esos datos al proveedor de autenticación y luego recibe una respuesta con el <Tooltip tip="ID Token: Credencial destinada al propio cliente, en lugar de para acceder a un recurso." cta="Ver glosario" href="/es/docs/glossary?term=ID+token">token de ID</Tooltip> y el <Tooltip tip="ID Token: Credencial destinada al propio cliente, en lugar de para acceder a un recurso." cta="Ver glosario" href="/es/docs/glossary?term=access+token">token de acceso</Tooltip> del usuario.

En las aplicaciones web, el inicio de sesión integrado usa la [autenticación entre orígenes](/es/docs/authenticate/login/cross-origin-authentication), a menos que [configure un dominio personalizado](/es/docs/customize/custom-domains) para su tenant de Auth0. La autenticación entre orígenes usa cookies de terceros para permitir transacciones de autenticación seguras entre distintos orígenes.

<div id="feature-comparison">
  ## Comparación de funcionalidades
</div>

| Funcionalidad                                           | Alojado                                                                                                                                                                                                                                                                                                             | Integrado                                                                                                                                                                                                                                                                                                                                                                                             |
| ------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **[Single Sign-on](/es/docs/authenticate/single-sign-on)** | Compatibilidad total con Universal Login mediante el uso de [cookies de sesión](/es/docs/manage-users/sessions) en el servidor de autorización de Auth0 (tu tenant de Auth0).                                                                                                                                                 | Compatibilidad limitada. Las aplicaciones web que usan las bibliotecas [Lock](/es/docs/libraries/lock/lock-api-reference#checksession-) o [Auth0.js](/es/docs/libraries/auth0js#using-checksession-to-acquire-new-tokens) pueden compartir sesiones. Las aplicaciones nativas pueden compartir sesiones con aplicaciones web mediante [Native to Web SSO](/es/docs/authenticate/single-sign-on/native-to-web). |
| **[Personalización](/es/docs/customize)**                  | Universal Login te permite personalizar fácilmente muchas partes de la experiencia (como temas, plantillas de página, elementos de texto y prompts).<br /><br />La personalización total es compatible mediante [Advanced Customizations for Universal Login.](/es/docs/customize/login-pages/advanced-customizations) | El mayor nivel de personalización, ya que controlas por completo la UI/UX de tu aplicación.                                                                                                                                                                                                                                                                                                           |
| **Gestión de funcionalidades**                          | Las funcionalidades pueden gestionarse de forma centralizada desde el Auth0 Dashboard o mediante la Auth0 Management API.<br /><br />Por ejemplo, cuando habilitas o deshabilitas MFA en tu Dashboard, el cambio se reflejará de inmediato en los siguientes inicios de sesión de los usuarios.                     | Las funcionalidades deben gestionarse de forma individual para cada aplicación.<br /><br />Por ejemplo, si quisieras implementar MFA para tu aplicación web y tu aplicación nativa, tendrías que actualizar y publicar nuevas versiones de ambas.                                                                                                                                                     |
| **Experiencia de usuario**                              | Los usuarios son redirigidos entre tu aplicación y el servidor de autorización de Auth0 durante la autenticación.<br /><br />En las aplicaciones nativas, esto requiere implementar enlaces universales o enlaces profundos.                                                                                               | Los usuarios permanecen en tu aplicación durante la autenticación.                                                                                                                                                                                                                                                                                                                                    |
| **Seguridad y mantenimiento**                           | Auth0 supervisa las tendencias de seguridad y actualiza Universal Login en consecuencia.                                                                                                                                                                                                                            | Eres responsable de seguir las mejores prácticas de seguridad e implementar adecuadamente las tecnologías proporcionadas.                                                                                                                                                                                                                                                                             |

<div id="best-practice">
  ## Práctica recomendada
</div>

Según [RFC 8252: OAuth 2.0 for Native Apps on IETF](https://www.rfc-editor.org/rfc/rfc8252.txt), las aplicaciones nativas solo deben utilizar agentes de usuario externos (como el navegador) para los flujos de autenticación.
