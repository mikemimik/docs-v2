Tus aplicaciones y servicios dependen de Auth0. Supervisar el estado de Auth0 te permite comunicar errores específicos a los clientes o reducir el impacto en los usuarios en caso de que haya algún problema con Auth0.

Hay muchas formas de supervisar Auth0, y cada enfoque complementa a los demás. Debes elegir la que mejor se adapte a tus necesidades y posibilidades de inversión.

<div id="synthetic-transactions">
  ## Transacciones sintéticas
</div>

La forma más sencilla de supervisar Auth0:

* Configura una solicitud periódica para realizar una transacción de autenticación.
* Si la solicitud se completa correctamente, Auth0 funciona bien.
* Si la solicitud falla, esto puede indicar:

  * un problema con Auth0
  * un problema específico del tenant usado para la transacción sintética
  * o simplemente una solicitud fallida aislada.

Para las transacciones sintéticas, usa un entorno lo más parecido posible a la configuración de tu tenant de producción. Incluso podrías usar el mismo tenant de producción. Como configurar transacciones sintéticas con flujos de redirección y proveedores externos puede ser complicado, se recomienda usar el [Resource Owner Password Grant](/es/docs/get-started/authentication-and-authorization-flow/resource-owner-password-flow/call-your-api-using-resource-owner-password-flow). Este flujo no implica redirecciones del navegador ni requiere una interfaz de usuario.

Si usas [Rules](/es/docs/customize/rules) o [conexiones de base de datos personalizadas](/es/docs/authenticate/database-connections/custom-db) u otros puntos de extensibilidad, las transacciones sintéticas deben configurarse para usar las Rules y/o los scripts de Custom DB, a fin de garantizar que esa parte del sistema funcione correctamente.

Herramientas como [Pingdom](http://pingdom.com/) hacen que configurar transacciones sintéticas sea muy sencillo.

<div id="check-period">
  ### Intervalo de verificación
</div>

Recomendamos ejecutar transacciones sintéticas a intervalos de un minuto. Con este enfoque sencillo, esa frecuencia no consumirá gran parte de tu cuota de límite de solicitudes de Auth0 y, al mismo tiempo, te proporcionará respuestas oportunas.

<div id="limitations-of-synthetic-transactions">
  ### Limitaciones de las transacciones sintéticas
</div>

Las transacciones sintéticas son una forma sencilla y económica de supervisar el estado de un tenant de Auth0. Sin embargo, tienen algunas limitaciones:

* Las transacciones sintéticas no reflejan la experiencia de sus usuarios finales, sino que ofrecen una métrica indirecta de ella.
* Es posible que las transacciones sintéticas no utilicen los mismos flujos que sus usuarios.
* Carecen de atomicidad (por lo general, se ejecutan una vez por minuto) y no informan sobre errores que sus usuarios finales podrían haber experimentado.

Si le interesa obtener datos más detallados, lea sobre [Seguimiento de errores](#error-tracking) y [Métricas y logs](#metrics-and-logs).

<div id="error-tracking">
  ## Seguimiento de errores
</div>

Este enfoque es útil para hacer seguimiento de los errores en las llamadas existentes a Auth0. Implica reportar errores cada vez que falla una llamada a Auth0. [Sentry](https://sentry.io/) es una herramienta que se usa habitualmente en estos casos y funciona tanto en entornos de frontend como de backend.

Este enfoque es útil porque te permite conocer los errores reales que están experimentando tus usuarios finales. Sin embargo, como solo haces seguimiento de los errores (y no de todas las solicitudes), no es posible tener una idea precisa de cuántos usuarios finales se ven afectados: ¿el 1 % o el 5 %? Además, no requiere configurar por separado una &quot;llamada sintética&quot;, que podría consumir parte de tu cuota de límite de solicitudes, especialmente si está mal configurada.

<div id="metrics-and-logs">
  ## Métricas y logs
</div>

Este enfoque es útil si las llamadas a Auth0 se realizan desde un backend que controlas. Este es el caso de:

* La mayoría de las llamadas a la [Management API de Auth0](https://auth0.com/docs/api/management/v2)
* Las llamadas a la [Authentication API](https://auth0.com/docs/api/authentication) desde Regular Web Applications y aplicaciones de máquina a máquina ([más información sobre los tipos de cliente](/es/docs/get-started/applications)).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Algunos productos de observabilidad también te permiten enviar métricas y logs desde aplicaciones frontend y móviles.
</Callout>

El enfoque consiste en usar métricas y/o logs para hacer seguimiento de las tasas de error en las llamadas a Auth0. Las métricas y los logs reflejan tasas de error que describen con precisión lo que experimentan los usuarios finales, sin necesidad de configurar una &quot;llamada sintética&quot; independiente, que podría consumir parte de tu cuota de límite de solicitudes, especialmente si está mal configurada.

<div id="alerting">
  ## Alertas
</div>

Independientemente de los enfoques de monitoreo que uses, es común recibir una alerta o una notificación cuando los errores se producen con cierta frecuencia. Esa frecuencia depende de tu aplicación.

Cuando tu equipo reciba una alerta de Auth0, te recomendamos agregar un enlace a [Consultar el estado de Auth0](/es/docs/deploy-monitor/monitor/check-auth0-status) en el payload/playbook de la alerta. Esto les permitirá consultar rápidamente el canal oficial de estado de Auth0 para comprobar si el problema proviene de Auth0 o de tu aplicación/servicio.
