---
title: Recuperar logs con la Management API
description: Aprende a recuperar logs con el endpoint get_logs de la Auth0 Management API por checkpoint o mediante criterios de búsqueda.
---

Puedes usar la <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/es/docs/glossary?term=Management+API">Management API</Tooltip> para recuperar hasta 100 logs por request mediante el endpoint [/get&#95;logs](https://auth0.com/docs/api/management/v2/#!/Logs/get_logs), que admite dos tipos de consumo:

* **Por checkpoint**: Recomendado si quieres exportar logs a un servicio externo de análisis de datos.
* **Por criterios de búsqueda**: Lo usa el Dashboard.

Los logs almacenados por Auth0 están sujetos a un período de retención según tu Auth0 subscription. Para obtener más información, consulta [Retención de datos de logs](/es/docs/deploy-monitor/logs/log-data-retention).

También puedes usar estos endpoints para buscar logs:

* `/api/v2/logs/{id}`: Recupera la entrada de log asociada al ID de log proporcionado.
* `/api/v2/users/{user_id}/logs`: Recupera logs de un ID de usuario específico.

<div id="retrieve-logs-by-checkpoint">
  ## Recuperar logs por checkpoint
</div>

Este método le permite recuperar logs posteriores a un `log_id` determinado. Se
usa habitualmente para enviar sus logs a un sistema externo, como Splunk, Datadog,
Application Insights, Mixpanel, entre otros.

Para buscar por checkpoint, use los siguientes parámetros:

| Parámetro | Descripción                                                                                                                        |
| --------- | ---------------------------------------------------------------------------------------------------------------------------------- |
| `from`    | ID del evento del log a partir del cual comenzar a recuperar logs. Puede limitar la cantidad de logs mediante el parámetro `take`. |
| `take`    | La cantidad total de entradas que se recuperarán al usar el parámetro `from`.                                                      |

Se ignorará cualquier parámetro distinto de `from` y `take`. Los logs se ordenan por `log_id`.

<div id="semantics">
  #### Semántica
</div>

La recuperación de logs mediante checkpoint tiene la siguiente semántica:

* Se devolverán todos los logs disponibles según los permisos de tu rol
* Los logs se ordenan por `log_id`, en lugar de por la hora de generación del log, para evitar omitir registros anteriores debido a retrasos del sistema de Auth0
* La paginación no tiene límite, a diferencia del límite de `1,000` registros al recuperar logs mediante criterios de búsqueda

<div id="pagination">
  #### Paginación
</div>

Al recuperar logs por checkpoint, solo la primera solicitud debe realizarse
manualmente. Cada solicitud posterior debe hacerse usando la URL de la relación `next`
dentro del encabezado `Link`. Esto simplifica la integración con la
API y permite que Auth0 realice optimizaciones sin afectar la experiencia del
cliente.

<div id="example">
  #### Ejemplo
</div>

Supongamos que el nombre de tu tenant en `us` es `acme`. Para empezar a recuperar registros mediante
checkpoint:

1. Toma el `log_id` de cualquier log de tu sistema.
   Ejemplo: `90020211201185132572079310688835925971249535794074878050`.
2. Haz una solicitud a `https://TENANT_NAME.LOCALITY.auth0.com/api/v2/logs?from=LOG_ID&take=100` para pedir los siguientes `100` logs posteriores al `LOG_ID` indicado.
3. La respuesta HTTP contendrá

   1. Un arreglo JSON de `0` a `100` logs posteriores al `LOG_ID` indicado
   2. Un encabezado HTTP `Link` con una relación `next`
4. Si el conjunto de resultados está vacío, recibirás un arreglo vacío y la relación `next` coincidirá con la misma URL que usaste en la solicitud más reciente

<div id="retrieve-logs-by-search-criteria">
  ## Recuperar logs según criterios de búsqueda
</div>

Este método permite recuperar los logs que coincidan con los criterios de búsqueda especificados o, si no se usa ningún criterio, listar todos los logs. Para buscar por criterios, use los siguientes parámetros:

| Parámetro        | Descripción                                                                                                                                              |
| ---------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `q`              | Criterios de búsqueda mediante la sintaxis de cadena de consulta.                                                                                        |
| `page`           | El número de página basado en cero.                                                                                                                      |
| `per_page`       | El número de entradas por página.                                                                                                                        |
| `sort`           | El campo que se usará para ordenar. Use `field:order`, donde `order` es `1` para orden ascendente y `-1` para orden descendente. Por ejemplo, `date:-1`. |
| `fields`         | Una lista de campos separados por comas para incluir o excluir del resultado (según `include_fields`). Déjelo vacío para recuperar todos los campos.     |
| `include_fields` | `true` si los campos especificados deben incluirse en el resultado; en caso contrario, `false`. El valor predeterminado es `true`.                       |

Para ver una lista de campos y parámetros de ordenación que pueden usarse en la consulta de búsqueda, lea [Sintaxis de consulta de búsqueda de logs](/es/docs/deploy-monitor/logs/log-search-query-syntax).

<div id="limitations">
  ## Limitaciones
</div>

Además del límite de 100 registros por solicitud al recuperar registros, solo puedes paginar hasta 1.000 resultados de búsqueda.

Si obtienes el error `414 Request-URI Too Large`, significa que la cadena de consulta supera la longitud admitida. En ese caso, limita más la búsqueda.

<Card title="Limitaciones de los campos de búsqueda en Private Cloud">
  Los siguientes campos solo se pueden buscar en Private Cloud:

  * `user`
  * `connection`
  * `application`
  * `type`
  * `ip`

  Usa comillas dobles para búsquedas exactas (por ejemplo, `application:\"test\"` buscará todos los eventos de log específicos de la aplicación llamada `test`, mientras que `application:test` buscará eventos de log de aplicaciones que tengan &quot;test&quot; en su nombre.
</Card>
