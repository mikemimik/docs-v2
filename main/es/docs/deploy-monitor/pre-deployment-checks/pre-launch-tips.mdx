description: Una lista de consejos útiles para empezar a usar los servicios de Auth0, basada en los comentarios y la experiencia de otros usuarios
title: Consejos previos al lanzamiento
---

Aquí tienes una lista de consejos que nuestros clientes han encontrado más útiles al empezar a usar los servicios de Auth0:

* Verifica las URL de callback permitidas, las URL de CORS permitidas y las URL de redirección permitidas para Logout para asegurarte de que sean correctas, completas y que no impliquen el uso de `file:///` o `localhost`. Minimiza el uso de comodines y considera cuidadosamente sus implicaciones de seguridad.
* Separa el uso en producción del trabajo de desarrollo en curso [usando tenants separados para prod, test y dev.](/docs/es/get-started/auth0-overview/create-tenants/set-up-multiple-environments)
* Configura correctamente tu red para permitir el tráfico hacia y desde nuestras APIs públicas (los entornos de prod pueden tener una configuración diferente a la de dev/staging).
* Verifica la fecha de vencimiento de los certificados que has agregado a tu configuración para asegurarte de que los certificados cargados durante los ciclos de desarrollo no venzan durante el lanzamiento ni poco después.
* Asegúrate de que todos los <Tooltip tip="Proveedor de identidad (IdP): Servicio que almacena y gestiona identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=IDPs">IDPs</Tooltip> remotos estén usando NTP para que la hora se sincronice correctamente.
* La cantidad de días durante los que los datos de registro están disponibles varía según el plan, hasta un máximo de 30 días. Si necesita conservar los datos de registro durante más tiempo, debe configurar una de nuestras [soluciones de transmisión de registros](https://marketplace.auth0.com/features/log-streaming). Esto le permitirá conservar los datos de registro por períodos más largos. Asegúrese de hacerlo antes de pasar a producción para tener datos de registro disponibles cuando los necesite.
* Asegúrese de que el uso de nuestras API se mantenga dentro de los [límites permitidos](/docs/es/troubleshoot/customer-support/operational-policies/rate-limit-policy) y de que su código esté preparado para ajustarse dinámicamente a la información de rate limit devuelta en el encabezado, así como para gestionar errores. Si prevé que los rate limits serán un problema para su aplicación, consúltelo con nosotros de antemano para comprobar si sería posible aumentarlos temporalmente hasta que el tráfico vuelva a la normalidad. Debería considerar usar una caché de datos de usuario para no tener que consultar un endpoint de API más de lo necesario.
* Si utiliza [conexiones sociales](/docs/es/authenticate/identity-providers), asegúrese de obtener sus propias credenciales del proveedor y agregarlas a la configuración de la conexión social.
* Si utiliza [conexiones de Custom DB](/docs/es/authenticate/database-connections/custom-db), asegúrese de que todos los script de Custom DB estén implementados y devuelvan un perfil de usuario coherente con un ID de usuario único.
* Para enviar correos electrónicos, primero asegúrese de [configurar un proveedor de correo electrónico personalizado.](/docs/es/customize/email/smtp-email-providers)
* Si utiliza nuestro CDN para el widget Lock, asegúrese de fijar una versión específica.
* Asegúrese de que los componentes **externos** llamados desde Actions, Rules, Hooks y los scripts de conexiones de base de datos personalizadas puedan manejar la carga esperada.
* Proteja adecuadamente todos los valores de <Tooltip tip="Secreto del cliente: secreto utilizado por un cliente (aplicación) para autenticarse con el servidor de autorización; solo el cliente y el servidor de autorización deben conocerlo y debe ser lo suficientemente aleatorio como para que no se pueda adivinar." cta="Ver glosario" href="/docs/es/glossary?term=client+secret">secreto del cliente</Tooltip>.
* Revisa los [tipos de concesión](/docs/es/get-started/applications/application-grant-types) de tus aplicaciones. Asegúrate de tener habilitados los correctos y, lo que es más importante, desactiva cualquier tipo de concesión que no necesites.
* Si usas user&#95;metadata, confirma que se trata de datos que los usuarios deberían poder cambiar por sí mismos (p. ej., no el “estado de pago”).
* Revisa la configuración de [Protección contra ataques](https://manage.auth0.com/#/security/attack-protection) y lee [Protección contra ataques](/docs/es/secure/attack-protection) para entender cómo desbloquear a los usuarios que hayan sido bloqueados.
* Revisa la [sección Administradores de la configuración del tenant](https://manage.auth0.com/#/tenant/admins) para asegurarte de que solo los administradores correspondientes tengan acceso al <Tooltip tip="Auth0 Dashboard: el producto principal de Auth0 para configurar tus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip>. Consulta [Administrar el acceso al Dashboard](/docs/es/get-started/manage-dashboard-access) para obtener más información.
* Asegúrate de haber probado todos los casos de uso principales de tu aplicación en todos los dispositivos que puedan utilizar los usuarios finales. No dejes de probar el inicio de sesión, <Tooltip tip="Inicio de sesión único (SSO): servicio que, después de que un usuario inicia sesión en una aplicación, inicia automáticamente la sesión de ese usuario en otras aplicaciones." cta="Ver glosario" href="/docs/es/glossary?term=single+sign-on">inicio de sesión único</Tooltip> (si es compatible) y el registro, así como lo que ocurre si un usuario ejecuta tu aplicación en varias pestañas del navegador.
* Revisa tu [lista de Rules](https://manage.auth0.com/#/rules) y asegúrate de que solo estén activadas las Rules adecuadas.
* Revisa el código de tus Rules, cualquier script de Custom DB y cualquier código personalizado de la página de inicio de sesión para asegurarte de que todas las llamadas tengan un manejo y control de errores adecuados. Además, verifica que las sentencias `return` y callback se estén invocando correctamente.
* Configura el nombre de tu aplicación, la URL de soporte y el correo electrónico de soporte en la sección [Tenant Settings General](https://manage.auth0.com/#/tenant) para que, cuando ocurra un error, tus usuarios finales sean dirigidos a una página adecuada.
* Asegúrate de que tu aplicación [obtenga dinámicamente un token de Management API](/docs/es/secure/tokens/access-tokens/management-api-access-tokens).
* Elimina cualquier sentencia `console.log` de tus Rules o scripts de Custom DB, especialmente las que puedan filtrar información que permita identificar al usuario, como el correo electrónico, el nombre de usuario o la contraseña.
* No uses secretos en texto plano en Rules ni en conexiones de base de datos. Deben agregarse en la sección de configuración de la interfaz. La configuración está cifrada y se proporciona en el momento necesario. No registres el objeto de configuración.
