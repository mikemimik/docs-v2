description: Aprende a excluir recursos de la gestión al usar Auth0 Deploy CLI.
title: Excluir recursos de la gestión
---

En algunos casos, puede resultarte útil excluir ciertos recursos de la gestión. Esto puede deberse a que tu tenant tiene una gran cantidad de un tipo concreto de recurso y gestionarlos supone una carga operativa, o a que tu flujo de trabajo de desarrollo solo se centra en un subconjunto específico de recursos y quieres omitir todos los demás por motivos de rendimiento. En cualquier caso, hay varias opciones para excluir recursos al usar Deploy CLI.

<div id="exclude-entire-resources-by-type">
  ## Excluir recursos completos por tipo
</div>

Para tenants más complejos, quizá quieras omitir tipos de recursos completos. Por ejemplo:

* Tenant empresarial con miles de organizaciones, en el que gestionarlas todas supondría una carga operativa.
* El proceso de CI/CD solo se centra en gestionar roles y quieres excluir todos los demás.
* El desarrollo de funcionalidades se refiere a hook y quieres excluir temporalmente todos los demás para optimizar el rendimiento.

Este tipo de exclusión se expresa pasando un array de nombres de recursos a cualquiera de las propiedades de configuración `AUTH0_EXCLUDED` o `AUTH0_INCLUDED_ONLY`. La propiedad de configuración `AUTH0_EXCLUDED` excluye únicamente los tipos de recursos que se le proporcionan. En cambio, la propiedad `AUTH0_INCLUDED_ONLY` excluye todas las propiedades excepto las que se definen. La exclusión funciona en ambas direcciones, es decir, tanto al exportar desde Auth0 como al importar a Auth0, independientemente de si existen o no archivos de configuración de recursos.

Todos los valores de recursos compatibles para la exclusión: `actions`, `attackProtection`, `branding`, `clientGrants`, `clients`, `connections`, `customDomains`, `databases`, `emailProvider`, `emailTemplates`, `guardianFactorProviders`, `guardianFactorTemplates`, `guardianFactors`, `guardianPhoneFactorMessageTypes`, `guardianPhoneFactorSelectedProvider`, `guardianPolicies`, `logStreams`, `migrations`, `organizations`, `pages`, `prompts`, `resourceServers`, `roles`, `tenant`, `triggers`.

<div id="exclusion-example">
  ### Ejemplo de exclusión
</div>

El siguiente ejemplo muestra cómo excluir `clients`, `connections`, `databases` y `organizations` de la administración mediante Deploy CLI.

```json lines
{
  "AUTH0_DOMAIN": "example-site.us.auth0.com",
  "AUTH0_CLIENT_ID": "<YOUR_AUTH0_CLIENT_ID>",
  "AUTH0_EXCLUDED": ["clients", "connections", "databases", "organizations"]
}
```

<div id="inclusion-example">
  ### Ejemplo de inclusión
</div>

El siguiente ejemplo muestra cómo especificar que solo `actions`, `clients` y `connections` se gestionen con Deploy CLI.

```json lines
{
  "AUTH0_DOMAIN": "example-site.us.auth0.com",
  "AUTH0_CLIENT_ID": "<YOUR_AUTH0_CLIENT_ID>",
  "AUTH0_INCLUDED_ONLY": ["actions", "clients", "connections"]
}
```

<div id="exclude-single-resources-by-id">
  ## Excluir recursos individuales por ID
</div>

Algunos tipos de recursos permiten excluir recursos individuales por ID. Esto resulta útil si trabajas en varios entornos y deseas omitir de los entornos de nivel inferior un recurso específico de producción.

Este método se admite para reglas, clientes, bases de datos, conexiones y <Tooltip tip="Servidor de recursos: servidor que aloja recursos protegidos. Los servidores de recursos aceptan y responden a solicitudes de recursos protegidos." cta="Ver glosario" href="/es/docs/glossary?term=resource+servers">servidores de recursos</Tooltip> con los valores de configuración `AUTH0_EXCLUDED_RULES`, `AUTH0_EXCLUDED_CLIENTS`, `AUTH0_EXCLUDED_DATABASES`, `AUTH0_EXCLUDED_CONNECTIONS`, `AUTH0_EXCLUDED_RESOURCE_SERVERS`, respectivamente.

### Ejemplo

```json lines
{
  "AUTH0_DOMAIN": "example-site.us.auth0.com",
  "AUTH0_CLIENT_ID": "<YOUR_AUTH0_CLIENT_ID>",
  "AUTH0_EXCLUDED_CLIENTS": ["Your Application Name"],
  "AUTH0_EXCLUDED_CONNECTIONS": ["con_O1H3KyRMFP1IWRq3", "con_9avEYuj19ihqKBOs"]
}
```

  Se está considerando dejar de admitir la exclusión de recursos por ID en futuras versiones principales. Consulta la [propuesta de exclusión de recursos](https://github.com/auth0/auth0-deploy-cli/blob/6381ce6efa8cd59e7a5c8d37e8915b0989d677b9/docs/excluding-from-management.md?plain=1#L51) para obtener más detalles.
</Callout>

<div id="omission-and-empty-states">
  ## Omisión y estados vacíos
</div>

Además de excluir recursos, lo que obliga a ignorar las configuraciones en ambos sentidos, Auth0 Deploy  CLI admite dos conceptos similares: la omisión y los estados vacíos.

<div id="omission">
  ### Omisión
</div>

La configuración de recursos que falte, ya sea intencionalmente o no, se omitirá durante la importación. Por ejemplo, si se eliminara la configuración de tu recurso, se omitiría durante la importación y no alteraría el estado del tenant remoto.

No existe el concepto de omisión en la exportación. A menos que se excluyan específicamente, todas las configuraciones de tu tenant se escribirán en archivos de configuración de recursos.

<div id="example">
  #### Ejemplo
</div>

```yaml lines
roles: # la configuración de roles no se omite
  - name: Admin
    description: Can read and write things
    permissions: []
  - name: Reader
    description: Can only read things
    permissions: []
# La omisión de todas las demás configuraciones significa que serán omitidas
```

<div id="empty">
  ### Vacío
</div>

Configuración de recursos definida explícitamente como vacía. En las configuraciones basadas en conjuntos, como hooks, organizaciones y acciones, establecer estas configuraciones como un conjunto vacío expresa la intención de vaciar esos recursos. Esto indicaría una eliminación, siempre que la propiedad de configuración de eliminación `AUTH0_ALLOW_DELETE` esté habilitada. Para obtener más información sobre esta propiedad, consulta [Configure the Deploy CLI](/es/docs/deploy-monitor/deploy-cli-tool/configure-the-deploy-cli).

En las configuraciones de recursos que no se basan en conjuntos, como tenant y branding, el concepto de vacío no se aplica y no activará ninguna eliminación ni supresión.

<div id="example-of-emptiness">
  #### Ejemplo de vacío
</div>

```yaml lines
hooks: [] # Hooks vacíos
connections: [] # Conexiones vacías
tenant: {} # Efectivamente un no-op, el vacío no se aplica a la configuración de recursos no definidos
```
