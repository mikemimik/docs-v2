<Tooltip tip="JSON Web Token (JWT): Formato estándar de token de identidad (y, a menudo, de token de acceso) que se utiliza para representar claims de forma segura entre dos partes." cta="Ver glosario" href="/docs/es/glossary?term=JSON+web+token">JSON web token</Tooltip> (JWT), que se pronuncia &quot;jot&quot;, es un estándar abierto ([RFC 7519](https://tools.ietf.org/html/rfc7519)) que define una forma compacta y autosuficiente de transmitir información de forma segura entre partes en un objeto JSON. Una vez más, JWT es un estándar, lo que significa que todos los JWT son tokens, pero no todos los tokens son JWT.

Debido a su tamaño relativamente pequeño, un JWT puede enviarse a través de una URL, mediante un parámetro POST o dentro de un encabezado HTTP, y se transmite rápidamente. Un JWT contiene toda la información necesaria sobre una entidad para evitar consultar una base de datos más de una vez. El destinatario de un JWT tampoco necesita hacer una llamada a un servidor para validar el token.

<div id="benefits">
  ## Beneficios
</div>

* **Compactos**: Los JWT son pequeños, lo que los convierte en una buena opción para transmitirse en entornos HTML y HTTP.

<Frame>![JWT codificado comparado con un token SAML](/docs/images/cdy7uua7fh8z/4S6xl4Yvi0H1CUrLB69ZtH/e3e3bf1767d2d29563a99cf71cab158d/comparing-jwt-vs-saml2.png)</Frame>

* **Seguros**: Los JWT pueden usar un par de claves pública y privada en forma de certificado X.509 para firmarse. Un JWT también puede firmarse de forma simétrica con un secreto compartido mediante el algoritmo HMAC. Para obtener más información, consulta [Algoritmos de firma](/docs/es/get-started/applications/signing-algorithms).
* **Comunes**: Los analizadores de JSON son compatibles con la mayoría de los lenguajes de programación.

<div id="usage">
  ## Uso
</div>

* **Autenticación**: Cuando un usuario inicia sesión correctamente con sus credenciales, se devuelve un [token de identidad](/docs/es/secure/tokens/id-tokens). Según la [especificación OpenID Connect (OIDC)](https://openid.net/specs/openid-connect-core-1_0.html#IDToken), un <Tooltip tip="Token de identidad: credencial destinada al propio cliente, en lugar de usarse para acceder a un recurso." cta="Ver glosario" href="/docs/es/glossary?term=ID+token">token de identidad</Tooltip> siempre es un JWT.
* **Autorización**: Una vez que un usuario ha iniciado sesión correctamente, una aplicación puede solicitar acceso a rutas, servicios o recursos (por ejemplo, API) en nombre de ese usuario. Para ello, la aplicación debe enviar un <Tooltip tip="Token de acceso: credencial de autorización, en forma de una cadena opaca o de un JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=Access+Token">token de acceso</Tooltip> en cada solicitud, que puede tener forma de JWT.
* **Intercambio de información**: Los JWT son una buena forma de transmitir información de manera segura entre partes porque pueden estar firmados, lo que significa que puedes tener la certeza de que los remitentes son quienes dicen ser. Además, la estructura de un JWT te permite verificar que el contenido no se haya manipulado.

<div id="security">
  ## Seguridad
</div>

La información contenida en el objeto JSON puede verificarse y considerarse fiable porque está firmada digitalmente. Aunque los JWT también pueden cifrarse para proporcionar confidencialidad entre las partes, los JWT emitidos por Auth0 son JSON Web Signatures (JWS), lo que significa que están firmados, no cifrados. Por ello, nos centraremos en los tokens firmados, que permiten verificar la integridad de los claims que contienen, mientras que los tokens cifrados ocultan esos claims a otras partes.

En general, los JWT pueden firmarse con un secreto (mediante el algoritmo HMAC) o con un par de claves pública y privada mediante RSA o ECDSA (aunque Auth0 solo admite HMAC y RSA). Cuando los tokens se firman con pares de claves pública y privada, la firma también certifica que solo la parte que posee la clave privada es quien lo firmó.

Antes de usar un JWT recibido, debe [validarse correctamente mediante su firma](/docs/es/secure/tokens/json-web-tokens/validate-json-web-tokens). Ten en cuenta que validar correctamente un token solo significa que la información que contiene no ha sido modificada por nadie más. Esto no significa que otros no hayan podido ver el contenido, que se almacena en texto plano. Por ello, nunca debes almacenar información sensible dentro de un JWT y debes tomar otras medidas para asegurarte de que los JWT no sean interceptados; por ejemplo, enviándolos solo a través de HTTPS, siguiendo las [prácticas recomendadas](/docs/es/secure/tokens/token-best-practices) y usando únicamente bibliotecas seguras y actualizadas.
