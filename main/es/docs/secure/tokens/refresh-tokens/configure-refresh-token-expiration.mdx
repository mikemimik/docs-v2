<Tooltip tip="Refresh Token: Token usado para obtener un token de acceso renovado sin obligar a los usuarios a volver a iniciar sesión." cta="Ver glosario" href="/es/docs/glossary?term=Refresh+tokens">
  Tokens de actualización
</Tooltip>

pueden ser objeto de uso indebido si se filtran, ya que pueden usarse para obtener nuevos

<Tooltip tip="Refresh Token: Token usado para obtener un token de acceso renovado sin obligar a los usuarios a volver a iniciar sesión." cta="Ver glosario" href="/es/docs/glossary?term=access+tokens">
  tokens de acceso
</Tooltip>

. Para mitigar este riesgo, Auth0 recomienda usar la detección automática de reutilización y la

<Tooltip tip="Access Token: Credencial de autorización, en forma de una cadena opaca o JWT, usada para acceder a una API." cta="Ver glosario" href="/es/docs/glossary?term=Refresh+Token+Rotation">
  rotación del Refresh Token
</Tooltip>

. La rotación del Refresh Token emite un token de actualización que expira después de un período predefinido. Tras la expiración, el usuario obtiene un nuevo token de actualización de la misma familia, o tokens de actualización que comparten un ID de familia,
o un nuevo par de token de acceso/token de actualización. Para obtener más información, consulta [Refresh Token Rotation](/es/docs/secure/tokens/refresh-tokens/refresh-token-rotation).

Puedes habilitar y configurar dos opciones de duración del token de actualización, expiración máxima y expiración por inactividad, usando el <Tooltip tip="Auth0 Dashboard: El producto principal de Auth0 para configurar tus servicios." cta="Ver glosario" href="/es/docs/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip> o la <Tooltip tip="Auth0 Dashboard: El producto principal de Auth0 para configurar tus servicios." cta="Ver glosario" href="/es/docs/glossary?term=Management+API">Management API</Tooltip> de Auth0. Puedes usar una combinación de períodos de expiración máxima y expiración por inactividad del token de actualización para lograr un equilibrio entre seguridad y experiencia de usuario que se adapte a las necesidades de tu negocio.

* **Tiempo de vida máximo**: Establece el tiempo de vida de un token de actualización o de una familia de tokens de actualización, tras el cual el usuario debe volver a autenticarse antes de recibir un nuevo token de acceso. Si deshabilitas esta configuración, el tiempo de vida máximo será indefinido.
* **Tiempo de vida inactivo**: Establece el tiempo de inactividad tras el cual expiran los tokens de actualización emitidos si el usuario no está activo en tu aplicación durante un período determinado.

<div id="use-the-dashboard">
  ## Usa el Dashboard
</div>

1. Ve a [Dashboard &gt; Applications](https://manage.auth0.com/#/applications).
2. Selecciona la aplicación que quieres configurar.
3. Ve a la pestaña **Settings**.
4. En **Refresh Token Expiration**, habilita **Set Idle Refresh Token Lifetime**. Cuando está habilitado, un token de actualización caduca según el tiempo de vida inactivo del token de actualización, después de lo cual ya no se puede usar. Si la rotación está habilitada, se debe establecer un tiempo de vida de expiración.

   <Frame>![Dashboard Applications Applications pestaña Settings Expiración del token de actualización](/docs/images/cdy7uua7fh8z/2EnpIEc4kBxjnS3T67xPzC/22fdb95e2a2f31cee6f01e02d0a709c2/Refresh_Token_Expiration_-_English.png)</Frame>
5. Introduce **Idle Refresh Token Lifetime** en segundos. El token de actualización caduca después del intervalo especificado y ya no se puede usar para obtener un nuevo token de acceso. Cuando la rotación está habilitada, el tiempo de vida inactivo del token de actualización también se aplica a la capacidad de obtener nuevos tokens.

   | Tiempo de vida | Valor                        |
   | -------------- | ---------------------------- |
   | Predeterminado | 2,592,000 segundos (30 días) |
   | Mínimo         | 1 segundo                    |
   | Máximo         | 31,557,600 segundos (1 año)  |

   <Callout icon="file-lines" color="#0EA5E9" iconType="regular">
     * **Idle Refresh Token Lifetime** no puede superar **Maximum Refresh Token Lifetime**.
     * El cálculo de un año equivale a 365.25 días para tener en cuenta los años bisiestos.
   </Callout>
6. Habilita **Set Maximum Refresh Token Lifetime**. Cuando está habilitado, un token de actualización caduca según un tiempo de vida máximo especificado, después de lo cual ya no se puede usar.
7. Introduce **Maximum Refresh Token Lifetime** en segundos. Si el token de actualización no se intercambia dentro del intervalo especificado, caduca y ya no se puede usar para obtener un nuevo token de acceso. El período de expiración del tiempo de vida inactivo del token de actualización se renueva cada vez que el token de actualización se intercambia por un nuevo token de acceso dentro del intervalo.

   | Tiempo de vida | Valor                       |
   | -------------- | --------------------------- |
   | Mínimo         | 1 segundo                   |
   | Máximo         | 31,557,600 segundos (1 año) |
8. Selecciona **Save Changes**.

<div id="use-the-management-api">
  ## Usa la Management API
</div>

Puedes configurar los ajustes de duración máxima y de inactividad en el payload del endpoint [/api/v2/clients/{id}](https://auth0.com/docs/api/management/v2#!/Clients/patch_clients_by_id) de la Management API. Aquí tienes un ejemplo que establece el tiempo de expiración de un token de actualización sin rotación:

```json lines
PATCH /api/v2/clients/{id}
{
  "refresh_token": {
      "rotation_type": "non-rotating",
      "expiration_type": "expiring",
      "token_lifetime": 2592000,
      "infinite_token_lifetime": false,
      "idle_token_lifetime": 604800,
      "infinite_idle_token_lifetime": false
  }
}
```

<div id="support-and-limitations">
  ## Soporte y limitaciones
</div>

* La **vida útil máxima del token de actualización** se determina en el momento de su emisión y permanece fija durante toda su vigencia, incluso si se actualiza la vida útil máxima del token de actualización de la aplicación. Sin embargo, puedes reemplazar activamente la vida útil máxima del token de actualización en el momento de su emisión mediante [sesiones con Actions](/es/docs/manage-users/sessions/manage-sessions-actions), incluso durante intercambios exitosos de token de actualización.
* La **vida útil inactiva del token de actualización** refleja la configuración actual de la aplicación y se actualiza con cada intercambio exitoso de token de actualización.
* Si configuras la vida útil máxima del token de actualización para una aplicación, cualquier token de actualización sin vencimiento emitido por esa aplicación pasará a ser un token de actualización con vencimiento después de un intercambio exitoso de token de actualización.
* Una vez que un token de actualización sin vencimiento pasa a ser un token de actualización con vencimiento, no se puede revertir a un token de actualización sin vencimiento.
* Puedes optar por usar las capacidades de expiración del token de actualización; no necesitas realizar ninguna acción. Los tokens de actualización existentes no se ven afectados.
* La expiración del token de actualización funciona con los siguientes flujos:

  * [Flujo de código de autorización](/es/docs/get-started/authentication-and-authorization-flow/authorization-code-flow)
  * [Flujo de código de autorización con Proof Key for Code Exchange](/es/docs/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce)
  * [Flujo de autorización de dispositivos](/es/docs/get-started/authentication-and-authorization-flow/device-authorization-flow)
  * [Flujo de contraseña del propietario del recurso](/es/docs/get-started/authentication-and-authorization-flow/resource-owner-password-flow)
* Todos los SDK de Auth0 admiten la expiración del token de actualización.
* La funcionalidad de expiración del token de actualización cumple con las [recomendaciones de OAuth 2.0 Security BCP](https://www.ietf.org/archive/id/draft-ietf-oauth-security-topics-16.html#section-4.13.2).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  El OAuth BCP establece que los tokens de actualización emitidos para aplicaciones basadas en navegador deben tener vencimiento y, además, aplicar restricciones del remitente o rotar los tokens con cada solicitud. Por lo tanto, las SPA usarán la rotación de forma predeterminada y **no admitirán** tokens de actualización sin vencimiento.
</Callout>
