<Warning>
  Si necesitas configurar un dominio personalizado, hazlo **antes** de implementar WebAuthn en entornos de producción. Si configuras o cambias un dominio personalizado, los usuarios inscritos previamente no podrán autenticarse.

  Puedes usar la configuración [Relying Party](#configure-relying-party) para especificar el dominio que se utiliza para autenticar a los usuarios.
</Warning>

<div id="use-the-dashboard">
  ## Usa el Dashboard
</div>

1. Habilita **WebAuthn with clave de seguridad** en [Dashboard &gt; Security &gt; Multi-factor Auth](https://manage.auth0.com/#/security/mfa).
2. Configura cómo quieres gestionar la verificación del usuario. En las claves de seguridad, lo habitual es que se pida a los usuarios que introduzcan un PIN para completar el desafío de WebAuthn.

   1. Nunca: a los usuarios no se les pedirá nunca que introduzcan un PIN. Este es el valor predeterminado y, por lo general, suele ser suficiente cuando se usan claves de seguridad para MFA. Los usuarios ya han introducido su contraseña, así que ya han proporcionado cierta verificación.
   2. Si es compatible: se pedirá a los usuarios que introduzcan un PIN si ya han configurado uno en la clave.
   3. Obligatorio: se pedirá a los usuarios que configuren un PIN si aún no lo han hecho, y que lo introduzcan cada vez. Esta es la opción que ofrece el mayor nivel de seguridad. Algunos navegadores no lo implementan correctamente (por ejemplo, [Brave](https://brave.com/) en iOS), por lo que la autenticación fallará y Auth0 pedirá a los usuarios que utilicen otro navegador.

Ten en cuenta que solo las claves de seguridad compatibles con FIDO-2 admiten la verificación del usuario. Las claves FIDO-1 pueden usarse con WebAuthn, pero no se pueden usar si estableces la verificación del usuario como Obligatorio.

<div id="configure-relying-party">
  ## Configurar Relying Party
</div>

WebAuthn hace que el phishing sea imposible al vincular las credenciales con el origen del navegador. Los usuarios tampoco pueden usar WebAuthn en un sitio en el que no se registraron.

Vincular las credenciales al origen significa que, si configuras un <Tooltip tip="Custom Domain: Dominio de terceros con un nombre especializado o personalizado." cta="Ver glosario" href="/es/docs/glossary?term=custom+domain">dominio personalizado</Tooltip> o lo cambias, los usuarios que se registraron antes del cambio no podrán autenticarse.

WebAuthn define un [atributo de ID de Relying Party](https://www.w3.org/TR/webauthn-2/#relying-party-identifier), que te permite especificar el dominio que se usa para autenticar a los usuarios. Puedes establecerlo en cualquier sufijo de dominio registrable del origen del navegador. Por ejemplo, si el dominio personalizado es `login.example.com`, puedes configurar el ID de <Tooltip tip="Relying Party: Entidad (como un servicio o aplicación) que depende de un proveedor de identidad de terceros para autenticar a un usuario." cta="Ver glosario" href="/es/docs/glossary?term=Relying+Party">Relying Party</Tooltip> como `example.com`. Esto permite a los usuarios autenticarse en cualquier dominio `example.com` con sus credenciales de WebAuthn.

Auth0 te permite especificar el ID de Relying Party solo si tienes configurado un dominio personalizado. Si el dominio personalizado cambia, debes actualizar el ID de Relying Party.

<div id="device-support">
  ## Compatibilidad de dispositivos
</div>

Para usar claves de seguridad, el navegador debe tener JavaScript habilitado y ser compatible con WebAuthn. Si no se cumplen esas condiciones, Auth0 no ofrecerá la opción de registrar ni de autenticarse con claves de seguridad. Auth0 solicitará otro factor o un código de recuperación (si no tienen otro factor registrado).

Las versiones más recientes de los navegadores y sistemas operativos más populares son compatibles con WebAuthn with claves de seguridad. Para obtener más información, consulta la [sección de compatibilidad de navegadores de webauthn.me](https://webauthn.me/browser-support).

<div id="limitations">
  ## Limitaciones
</div>

* Al usar la [API de MFA](/es/docs/secure/multi-factor-authentication/multi-factor-authentication-developer-resources/mfa-api), puedes listar y eliminar registros WebAuthn, pero no registrarlos.
