Las aplicaciones confidenciales, a diferencia de las aplicaciones públicas, pueden almacenar credenciales de forma segura. Cuando las aplicaciones confidenciales solicitan acceso o <Tooltip tip="Token de ID: credencial destinada al propio cliente, en lugar de acceder a un recurso." cta="Ver glosario" href="/es/docs/glossary?term=ID+tokens">tokens de ID</Tooltip> en el [endpoint de token](https://auth0.com/docs/api/authentication#get-token), la aplicación debe autenticarse ante el <Tooltip tip="Token de ID: credencial destinada al propio cliente, en lugar de acceder a un recurso." cta="Ver glosario" href="/es/docs/glossary?term=authorization+server">servidor de autorización</Tooltip>. Durante esta solicitud de tokens, la aplicación presenta sus credenciales. Además, las credenciales de la aplicación también pueden ofrecer protección de autenticidad e integridad para los parámetros de solicitud enviados al endpoint [`/authorize`](https://auth0.com/docs/api/authentication#authorize-application).

Para obtener más información sobre las aplicaciones confidenciales y las aplicaciones públicas, consulta [Confidential and Public Applications](/es/docs/get-started/applications/confidential-and-public-applications).

<div id="application-authentication-methods">
  ## Métodos de autenticación de la aplicación
</div>

Para obtener tokens de Auth0, tu aplicación debe autenticarse a través de la [Authentication API](https://auth0.com/docs/api/authentication). Auth0 admite las siguientes formas de autenticación para tu aplicación:

* **<Tooltip tip="Client Secret: Secreto que usa un cliente (aplicación) para autenticarse con el servidor de autorización; solo deben conocerlo el cliente y el servidor de autorización, y debe ser lo suficientemente aleatorio como para que no pueda adivinarse." cta="Ver glosario" href="/es/docs/glossary?term=Client+Secret">Secreto de cliente</Tooltip>:** Un método de autenticación simétrico. En la autenticación con secreto de cliente, usas el secreto de cliente que Auth0 generó cuando creaste la aplicación.
* **Clave privada <Tooltip tip="JSON Web Token (JWT): Formato estándar de token de ID (y, a menudo, de token de acceso) que se usa para representar claims de forma segura entre dos partes." cta="Ver glosario" href="/es/docs/glossary?term=JWT">JWT</Tooltip>:** Un método de autenticación asimétrico. En clave privada JWT, generas un par de claves, una pública y otra privada, para usarlas como credenciales. Proporcionas la clave pública y almacenas de forma segura la clave privada en tus propios sistemas, sin compartirla con Auth0.
* **mTLS para <Tooltip tip="OAuth 2.0: Marco de autorización que define protocolos y flujos de trabajo de autorización." cta="Ver glosario" href="/es/docs/glossary?term=OAuth">OAuth</Tooltip>:** Un método de autenticación asimétrico. En mTLS para OAuth, registras en Auth0 un certificado de cliente X.509 estándar. Luego, usas la clave privada correspondiente para establecer de forma segura el túnel mTLS y enviar solicitudes a los endpoints de tu tenant de Auth0.

<div id="client-secret-authentication">
  ### Autenticación con secreto de cliente
</div>

La autenticación con secreto de cliente es un método de autenticación simétrica incluido en la [especificación de OAuth 2.0](https://www.rfc-editor.org/rfc/rfc6749#section-2.3). La autenticación con secreto de cliente es el método de autenticación predeterminado en Auth0.

Este método de autenticación es compatible con todas las aplicaciones y herramientas existentes. El secreto de cliente es un valor de alta entropía generado por Auth0 cuando creas una aplicación, y tanto tu aplicación como Auth0 lo conocen. Tu aplicación se autentica incluyendo el secreto de cliente en la solicitud al servidor de autorización.

El uso del secreto de cliente como credencial conlleva algunos riesgos de seguridad, especialmente en escenarios con mayores exigencias de seguridad:

* El secreto que usa la aplicación se comparte con Auth0.
* El secreto se envía por la red y podría ser interceptado en caso de ataques de tipo man-in-the-middle.

<Warning>
  Para mejorar tu postura de seguridad, te recomendamos usar el método de autenticación con clave privada JWT.
</Warning>

Una aplicación puede tener un único secreto de cliente. No es posible rotar el secreto mientras actualizas tu implementación para usar el nuevo secreto. Para obtener más información, consulta [Rotate Client Secrets](/es/docs/get-started/applications/rotate-client-secret).

<div id="private-key-jwt-authentication">
  ### Autenticación con clave privada JWT
</div>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  La clave privada JWT está disponible para clientes con el plan Enterprise. Para cambiar de plan, contacta con [el equipo de precios de Auth0](https://auth0.com/pricing/).
</Callout>

La autenticación con clave privada JWT es un método de autenticación asimétrico que se basa en pares de claves públicas y privadas. Para obtener más información, lee [JSON Web Token (JWT) Profile for OAuth 2.0 Client Authentication and Authorization Grants](http://tools.ietf.org/html/draft-ietf-oauth-jwt-bearer).

Puedes usar el <Tooltip tip="Auth0 Dashboard: el producto principal de Auth0 para configurar tus servicios." cta="Ver glosario" href="/es/docs/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip> o la <Tooltip tip="Auth0 Dashboard: el producto principal de Auth0 para configurar tus servicios." cta="Ver glosario" href="/es/docs/glossary?term=Management+API">Management API</Tooltip> de Auth0 para configurar un tenant para usar clave privada JWT. Para obtener más información, consulta [Configure Private Key JWT Authentication.](/es/docs/get-started/applications/configure-private-key-jwt)

En la clave privada JWT, una solicitud al servidor de autorización consta de dos pasos principales:

1. Configura las claves pública y privada:

   1. [Genera un par de claves](/es/docs/secure/application-credentials/generate-rsa-key-pair) (una clave pública y una clave privada).
   2. Registra la clave privada en la aplicación que realiza la solicitud de autenticación y registra la clave pública en el proveedor de identidad (IdP).
2. Crea assertions para las solicitudes al servidor de autorización:

   1. Crea una nueva assertion con los claims especificados en formato JWT y fírmala con la clave privada. Incluye esta assertion como parte de la solicitud al IdP.
   2. El IdP valida la assertion con la clave pública.

Para configurar la clave privada JWT para Auth0, consulta [Configure Private Key JWT Authentication](/es/docs/get-started/applications/configure-private-key-jwt). Para obtener más información sobre cómo crear una assertion para clave privada JWT, consulta [Authenticate with Private Key JWT](/es/docs/get-started/authentication-and-authorization-flow/authenticate-with-private-key-jwt).

Usar clave privada JWT ofrece algunas ventajas de seguridad:

* La clave privada no se transmite por la red, lo que reduce el riesgo de exposición de las credenciales de tu aplicación. Los <Tooltip tip="Identity Provider (IdP): servicio que almacena y administra identidades digitales." cta="Ver glosario" href="/es/docs/glossary?term=Identity+providers">proveedores de identidad</Tooltip> como Auth0 no conocen la clave privada, y solo las aplicaciones que tienen acceso a ella pueden crear solicitudes de autenticación.
* Las assertions firmadas tienen un tiempo de expiración breve, lo que limita la ventana de oportunidad para ataques de repetición.

<div id="mtls-for-oauth">
  ### mTLS para OAuth
</div>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Para usar las funciones de Highly Regulated Identity, debes tener un plan Enterprise con el add-on Highly Regulated Identity. Consulta [Auth0 Pricing](https://auth0.com/pricing/) para obtener más información.
</Callout>

[mTLS para OAuth](https://www.rfc-editor.org/rfc/rfc8705) autentica las solicitudes al servidor de autorización mediante TLS mutuo basado en certificados autofirmados o infraestructura de clave pública (PKI). Lee [Authenticate with mTLS](/es/docs/get-started/authentication-and-authorization-flow/authenticate-with-mtls) para obtener más información sobre cómo funciona la autenticación mTLS en Auth0.

mTLS para OAuth de Auth0 está dirigido inicialmente a clientes de sectores altamente regulados, como finanzas y salud, que probablemente ya cuenten con despliegues de mTLS. Para facilitar su adopción, la función de mTLS se basa en [dominios personalizados](/es/docs/customize/custom-domains) y aprovecha la infraestructura mTLS existente del cliente para realizar el aprovisionamiento y la verificación de certificados. Para obtener más información sobre cómo autenticar con mTLS y configurar tu red perimetral, lee [Authenticate with mTLS](/es/docs/get-started/authentication-and-authorization-flow/authenticate-with-mtls) y [Set up your customer edge](/es/docs/get-started/applications/configure-mtls/set-up-the-customer-edge).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Puedes configurar [alias de endpoint de mTLS](/es/docs/get-started/applications/configure-mtls/configure-mtls-for-a-tenant#enable-mtls-endpoint-aliases) para usar un subdominio específico para mTLS para OAuth.
</Callout>

Para aprender a configurar mTLS, lee [Configure mTLS Authentication](/es/docs/get-started/applications/configure-mtls). Una vez que hayas configurado tu red perimetral y mTLS, tu aplicación debe establecer el túnel mTLS para enviar solicitudes a Auth0, como se explica en [Call the authorization server](/es/docs/get-started/authentication-and-authorization-flow/authenticate-with-mtls#call-the-authorization-server).

En mTLS, la clave privada del certificado del cliente no se transmite por la red, lo que reduce el riesgo de exponer las credenciales de tu aplicación. Los proveedores de identidad como Auth0 no tienen acceso a la clave privada. Solo las aplicaciones que tienen acceso a la clave privada pueden autenticarse.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  mTLS también es compatible con Sender Constraining o Token Binding para proteger los tokens de acceso frente a atacantes. Para obtener más información, lee [Configure Sender Constraining](/es/docs/secure/sender-constraining/configure-sender-constraining). Token Binding no requiere credenciales de aplicación registradas previamente, como un certificado del cliente, para usarse con mTLS.
</Callout>

<div id="jwt-secured-authorization-request-jar">
  ### Solicitud de autorización protegida con JWT (JAR)
</div>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Para usar las funciones de Highly Regulated Identity, debes tener un plan Enterprise con el add-on Highly Regulated Identity. Consulta [Auth0 Pricing](https://auth0.com/pricing/) para obtener más información.
</Callout>

La [solicitud de autorización protegida con JWT (JAR)](https://datatracker.ietf.org/doc/rfc9101/) es una extensión del protocolo OAuth2 que refuerza la seguridad de las solicitudes de autorización. Para ello, utiliza un parámetro de solicitud JSON Web Token (JWT) para proteger la integridad y la confidencialidad de los parámetros de la solicitud de autorización.

Puedes usar la [Auth0 Management API](https://auth0.com/docs/api/management/v2) para configurar JAR en tu aplicación. La implementación de Auth0 para JAR utiliza criptografía asimétrica: registras la clave pública y mantienes la clave privada almacenada de forma segura de tu lado. Para obtener más información, consulta [Configure JWT-Secured Authorization Requests](/es/docs/get-started/applications/configure-jar).

Al usar JAR, el cliente crea un JWT que incluye los parámetros de la solicitud de autorización, lo firma con su clave privada y lo envía al servidor de autorización. Luego, el servidor de autorización verifica la firma con la clave pública del cliente y, si la firma es válida, extrae del JWT los parámetros de la solicitud de autorización y procesa la solicitud como de costumbre. Para obtener más información sobre cómo usar JAR, consulta [Authorization Code Flow with JWT-Secured Authorization Requests (JAR)](/es/docs/get-started/authentication-and-authorization-flow/authorization-code-flow/authorization-code-flow-with-jar).

<div id="key-and-certificate-registration">
  ### Registro de claves y certificados
</div>

<Warning>
  Debe generar un par de claves independiente para cada tipo de uso de credenciales. Por ejemplo, no reutilice los mismos pares de claves para JAR y la autenticación con clave privada JWT.
</Warning>

Puede registrar dos claves públicas para una aplicación al mismo tiempo. Auth0 gestiona la verificación con la clave adecuada y le permite rotarlas sin tiempo de inactividad. Una vez que la clave anterior se haya eliminado o desactivado, todas las solicitudes firmadas con la clave privada correspondiente dejarán de ser válidas.

**Nota:** Auth0 admite los siguientes algoritmos para la autenticación de aplicaciones y la firma de solicitudes de autorización: RS256, RS384 y PS256. Asegúrese de proporcionar las claves adecuadas para cada uno. Para obtener más información, consulte [Configure Private JWT Authentication](/es/docs/get-started/applications/configure-private-key-jwt) y [Configure JWT-Secured Authorization Requests](/es/docs/get-started/authentication-and-authorization-flow/authorization-code-flow/authorization-code-flow-with-jar).

Del mismo modo, para los certificados de cliente de mTLS, puede registrar dos certificados X.509 de cliente (autofirmados o con el Subject DN del certificado de la CA) al mismo tiempo para una aplicación. Auth0 gestiona la verificación con ambos certificados de cliente, lo que le permite rotar certificados sin tiempo de inactividad.
