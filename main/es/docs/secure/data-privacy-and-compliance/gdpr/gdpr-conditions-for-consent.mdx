De acuerdo con el artículo 7 del RGPD, debe solicitar a los usuarios su consentimiento para el tratamiento de sus datos personales de forma clara y fácilmente accesible. También debe poder demostrar que el usuario ha dado su consentimiento y ofrecer una forma sencilla de retirarlo en cualquier momento.

Este artículo explica cómo puede utilizar las funcionalidades de Auth0 para cumplir estos requisitos.

</Callout>

<div id="ask-for-consent">
  ## Pedir consentimiento
</div>

Durante el registro, debes pedir el consentimiento de tus usuarios. Con Auth0, puedes guardar esta información en los metadatos del usuario. Hay varias opciones disponibles según cómo uses Auth0 para autenticar a tus usuarios. Antes de diseñar tu solución con metadatos, asegúrate de conocer las restricciones. Auth0 limita el tamaño total de `user_metadata` a **16 MB**. Para obtener más información, consulta [Nombres de campos y tipos de datos de los metadatos](/es/docs/manage-users/user-accounts/metadata/metadata-fields-data).

<Warning>
  Los metadatos de Auth0 no son un almacenamiento de datos seguro y no deben usarse para guardar información confidencial, como secretos de alto riesgo e información de identificación personal (PII), como números de la seguridad social o de tarjetas de crédito. Se recomienda encarecidamente a los clientes de Auth0 que evalúen los datos almacenados en los metadatos y que guarden solo lo necesario para fines de gestión de identidades y accesos.
</Warning>

<div id="use-lock">
  ### Usa Lock
</div>

Puedes personalizar la UI de Lock para mostrar enlaces a tus páginas de términos y condiciones y/o declaración de privacidad, así como una casilla de consentimiento que el usuario debe marcar para poder registrarse. Esto se puede hacer con la opción de Lock `mustAcceptTerm`s. Esta propiedad, cuando se establece en `true`, muestra una casilla junto a los términos y condiciones que debe marcarse antes de registrarse. Los términos y condiciones se pueden especificar mediante la opción languageDictionary. Para obtener más información, consulta [Opciones de configuración de Lock](/es/docs/libraries/lock/lock-configuration).

Una vez que el usuario acepte y se registre, guarda la información de consentimiento en `user_metadata` mediante una `rule` que se ejecutará en el primer inicio de sesión. Para obtener más información sobre las rules, consulta [Auth0 Rules](/es/docs/customize/rules).

Si quieres obtener más información de los usuarios durante el registro y autenticas a los usuarios con una conexión de base de datos, puedes agregar campos personalizados a la UI de Lock. Esto se puede hacer con la opción de Lock `additionalSignUpFields`. Todos los campos personalizados se agregan automáticamente a `user_metadata`.

Si usas inicios de sesión sociales, agregar campos personalizados no es una opción, pero puedes redirigir al usuario a otra página para solicitar el consentimiento y cualquier información adicional, y luego redirigirlo de vuelta para finalizar la transacción de authentication. Esto se puede hacer con redirect rules. Para obtener más información, consulta [Redirigir a los usuarios desde Rules](/es/docs/customize/rules/redirect-users). Una vez que el proceso de registro se haya completado, guarda la información de consentimiento en `user_metadata` llamando a la <Tooltip tip="Management API: Un producto para permitir a los clientes realizar tareas administrativas." cta="Ver glosario" href="/es/docs/glossary?term=Management+API">Management API</Tooltip> [**endpoint Update User**](https://auth0.com/docs/api/management/v2#!/Users/patch_users_by_id).

Para aprender a implementar cualquiera de estos escenarios, consulta [RGPD: Seguimiento del consentimiento con Lock](/es/docs/secure/data-privacy-and-compliance/gdpr/gdpr-track-consent-with-lock).

<div id="use-custom-ui">
  ### Usa una interfaz de usuario personalizada
</div>

Si usas un formulario de registro personalizado con una conexión de base de datos, debes añadir un campo adicional a la pantalla de registro para recoger el consentimiento del usuario. Después, llama al [**endpoint Signup**](https://auth0.com/docs/api/authentication#signup) de la Authentication API para crear al usuario en Auth0. En este punto, puedes guardar la información del consentimiento como parte de `user_metadata`.

Como alternativa, si usas Auth0.js desde una SPA, puedes [usar el método `signup`](/es/docs/libraries/auth0js) para crear al usuario en Auth0 y guardar la información del consentimiento como parte de `user_metadata`.

Si usas un formulario de registro personalizado con proveedores sociales, no puedes guardar la información del consentimiento del usuario durante el registro, pero puedes actualizarla en cuanto se cree el usuario. Guarda la información del consentimiento en `user_metadata` llamando al [**endpoint Update User**](https://auth0.com/docs/api/management/v2#!/Users/patch_users_by_id) de la Management API.

Para obtener más información sobre cómo implementar cualquiera de estos escenarios, consulta [RGPD: Seguimiento del consentimiento con una interfaz de usuario personalizada](/es/docs/secure/data-privacy-and-compliance/gdpr/gdpr-track-consent-with-custom-ui).

<div id="re-consent-and-user-migration">
  ### Renovación del consentimiento y migración de usuarios
</div>

Si necesitas solicitar el consentimiento de los usuarios existentes y decides migrar tus usuarios desde una base de datos existente a Auth0, puedes [usar nuestra función de Migración automática de usuarios](/es/docs/manage-users/user-migration/configure-automatic-migration-from-your-database). Al activarla, cada vez que un usuario inicie sesión por primera vez (desde su activación), se creará en Auth0 sin tener que restablecer su contraseña. Para ello, debes:

* Redactar el aviso que verán los usuarios sobre cómo se usarán sus datos, durante cuánto tiempo se usarán, cuáles son sus derechos, etc., además de personalizar el cuadro de registro de la interfaz.
* Determinar si es necesario volver a solicitar el consentimiento de tus usuarios, en función de tus antiguos términos y condiciones y de certificaciones de privacidad previas.

Ten en cuenta que cada vez que cambien tus Términos y Condiciones, **debes** volver a pedir el consentimiento de los usuarios.

<div id="track-consent">
  ## Registrar el consentimiento
</div>

De acuerdo con el RGPD, debes poder demostrar que el usuario ha dado su consentimiento para el tratamiento de sus datos personales.

Con Auth0, puedes guardar la información sobre el consentimiento del usuario como parte de `user_metadata`. Puedes guardar solo un indicador que muestre si el usuario ha dado su consentimiento o no, o bien un conjunto de información y preferencias de consentimiento (incluidos, por ejemplo, el día en que el usuario dio su consentimiento, los términos que aceptó, etc.). Después, puedes acceder a esta información y gestionarla mediante la Management API.

La Management API también ofrece varias opciones en lo que respecta a la búsqueda de usuarios y a los endpoints para actualizar los metadatos del usuario o exportar usuarios en lote.

Para acceder a la Management API, necesitarás un <Tooltip tip="Token de acceso: credencial de autorización, en forma de una cadena opaca o JWT, que se utiliza para acceder a una API." cta="Ver glosario" href="/es/docs/glossary?term=access+token">token de acceso</Tooltip>. Para saber cómo obtener un token de acceso para la Management API, consulta [Tokens de acceso de la Management API](/es/docs/secure/tokens/access-tokens/management-api-access-tokens).

<div id="search-for-users-by-email-address">
  ### Buscar usuarios por dirección de correo electrónico
</div>

Para buscar un usuario por su dirección de correo electrónico, usa el [endpoint **Search User by Email**](/es/docs/manage-users/user-search/user-search-best-practices).

Establece el parámetro de solicitud **fields** en `user_metadata` para limitar los campos que se devuelven. De esta forma, solo se devolverá `user_metadata` en lugar del perfil de usuario completo.

Ejemplo de solicitud:

<AuthCodeGroup>
  ```bash cURL
  curl --request GET \
    --url 'https://{yourDomain}/api/v2/users-by-email?email=USER_EMAIL_ADDRESS&fields=user_metadata' \
    --header 'authorization: Bearer YOUR_MGMT_API_ACCESS_TOKEN'
  ```

  ```csharp C#
  var client = new RestClient("https://{yourDomain}/api/v2/users-by-email?email=USER_EMAIL_ADDRESS&fields=user_metadata");
  var request = new RestRequest(Method.GET);
  request.AddHeader("authorization", "Bearer YOUR_MGMT_API_ACCESS_TOKEN");
  IRestResponse response = client.Execute(request);
  ```

  ```go Go
  package main

  import (
  	"fmt"
  	"net/http"
  	"io/ioutil"
  )

  func main() {

  	url := "https://{yourDomain}/api/v2/users-by-email?email=USER_EMAIL_ADDRESS&fields=user_metadata"

  	req, _ := http.NewRequest("GET", url, nil)

  	req.Header.Add("authorization", "Bearer YOUR_MGMT_API_ACCESS_TOKEN")

  	res, _ := http.DefaultClient.Do(req)

  	defer res.Body.Close()
  	body, _ := ioutil.ReadAll(res.Body)

  	fmt.Println(res)
  	fmt.Println(string(body))

  }
  ```

  ```java Java

    .header("authorization", "Bearer YOUR_MGMT_API_ACCESS_TOKEN")
    .asString();
  ```

  ```javascript Node.JS
  var axios = require("axios").default;

  var options = {
    method: 'GET',
    url: 'https://{yourDomain}/api/v2/users-by-email',
    params: {email: 'USER_EMAIL_ADDRESS', fields: 'user_metadata'},

  };

  axios.request(options).then(function (response) {
    console.log(response.data);
  }).catch(function (error) {
    console.error(error);
  });
  ```

  ```php PHP

    CURLOPT_URL => "https://{yourDomain}/api/v2/users-by-email?email=USER_EMAIL_ADDRESS&fields=user_metadata",

    CURLOPT_ENCODING => "",
    CURLOPT_MAXREDIRS => 10,
    CURLOPT_TIMEOUT => 30,

    echo "cURL Error #:" . $err;
  } else {

  }
  ```


<div id="update-consent-information">
  ### Actualizar la información del consentimiento
</div>

Para actualizar el `user_metadata` de un usuario, usa el endpoint [**Update a User**](https://auth0.com/docs/api/management/v2#!/Users/patch_users_by_id).

La forma de estructurar tu solicitud depende de cómo hayas estructurado tus metadatos: en la raíz o como propiedades internas.

Si tus metadatos se almacenan como propiedades de la raíz:

```json lines
{
  "consentGiven": true,
  "consentDetails": "some-url"
}
```

Si tus metadatos se almacenan en propiedades internas:

```json lines
{
  "consent": {
    "given": true,
    "text_details": "some-url"
  }
}
```

<div id="update-root-property">
  #### Actualizar la propiedad raíz
</div>

Las actualizaciones de las propiedades del nivel raíz se fusionan, por lo que solo debes enviar el valor del campo que quieres actualizar. Por ejemplo, supongamos que queremos agregar una fecha de consentimiento y establecerla en `01/23/2018`.

<AuthCodeGroup>
  ```bash cURL
  curl --request PATCH \
    --url 'https://{yourDomain}/api/v2/users/USER_ID' \
    --header 'authorization: Bearer YOUR_MGMT_API_ACCESS_TOKEN' \
    --header 'content-type: application/json' \
    --data '{"user_metadata":{"consentDate":"01/24/2018"}}'
  ```

  ```csharp C#
  var client = new RestClient("https://{yourDomain}/api/v2/users/USER_ID");
  var request = new RestRequest(Method.PATCH);
  request.AddHeader("authorization", "Bearer YOUR_MGMT_API_ACCESS_TOKEN");
  request.AddHeader("content-type", "application/json");
  request.AddParameter("application/json", "{"user_metadata":{"consentDate":"01/24/2018"}}", ParameterType.RequestBody);
  IRestResponse response = client.Execute(request);
  ```

  ```go Go
  package main

  import (
  	"fmt"
  	"strings"
  	"net/http"
  	"io/ioutil"
  )

  func main() {

  	url := "https://{yourDomain}/api/v2/users/USER_ID"

  	payload := strings.NewReader("{"user_metadata":{"consentDate":"01/24/2018"}}")

  	req, _ := http.NewRequest("PATCH", url, payload)

  	req.Header.Add("authorization", "Bearer YOUR_MGMT_API_ACCESS_TOKEN")
  	req.Header.Add("content-type", "application/json")

  	res, _ := http.DefaultClient.Do(req)

  	defer res.Body.Close()
  	body, _ := ioutil.ReadAll(res.Body)

  	fmt.Println(res)
  	fmt.Println(string(body))

  }
  ```

  ```java Java
  HttpResponse response = Unirest.patch("https://{yourDomain}/api/v2/users/USER_ID")
    .header("authorization", "Bearer YOUR_MGMT_API_ACCESS_TOKEN")
    .header("content-type", "application/json")
    .body("{"user_metadata":{"consentDate":"01/24/2018"}}")
    .asString();
  ```

  ```javascript Node.JS

    console.error(error);
  ```

  ```python Python

      'authorization': "Bearer YOUR_MGMT_API_ACCESS_TOKEN",
  ```

</AuthCodeGroup>

Esto añadirá una nueva propiedad al perfil del usuario, **user&#95;metadata.consentDate**, que contendrá la fecha en la que el cliente otorgó su consentimiento. La respuesta será el perfil del usuario completo. Los metadatos actualizados se verán así:

```json lines
{
  "consentGiven": true,
  "consentDate": "01/23/2018",
  "consentDetails": "some-url"
}
```

  #### Actualizar una propiedad interna
</div>

<div id="export-consent-information">
  ### Exportar información de consentimiento
</div>

Para exportar una lista de tus usuarios mediante la Management API, usa el [endpoint **User Export**](/es/docs/manage-users/user-search/user-search-best-practices).

Este endpoint crea una tarea que exporta todos los usuarios asociados a una conexión. Necesitarás el ID de la conexión. Para encontrarlo, usa el [endpoint **Get Connections**](https://auth0.com/docs/api/management/v2#!/Connections/get_connections) (puedes establecer el parámetro **name** con el nombre de la conexión para recuperar solo esa).

Una vez que tengas el ID de la conexión y un token de acceso de la Management API, podrás comenzar a exportar usuarios. Para ver una solicitud y una respuesta de ejemplo, consulta [Import and Export Users](/es/docs/manage-users/user-migration). Para obtener información sobre cómo conseguir un token de acceso de la Management API, consulta [Management API Access Tokens](/es/docs/secure/tokens/access-tokens/management-api-access-tokens).

También debes:

* Determinar cómo quieres hacer el seguimiento del consentimiento. Recomendamos incluir información no solo sobre la fecha en que el usuario dio su consentimiento, sino también sobre la versión de los términos y condiciones que aceptó. También recomendamos incluir un arreglo para guardar información sobre los usuarios que retiren su permiso (recuerda que el usuario puede dar y retirar su consentimiento varias veces).
* Elegir dónde quieres almacenar el consentimiento: en la base de datos de Auth0 o en otro lugar.

<div id="withdraw-consent">
  ## Retirar el consentimiento
</div>

El usuario debe tener la opción de retirar su consentimiento desde su aplicación. Esta opción debe ser fácilmente accesible y fácil de identificar. Una vez que el usuario decida retirar su consentimiento, debe actuar.

Primero, debe decidir cómo gestionará la retirada del consentimiento: ¿eliminará a los usuarios o los marcará como eliminados?

<div id="flag-user-as-deleted">
  ### Marcar al usuario como eliminado
</div>

Si no quieres eliminar al usuario, marca su perfil como eliminado mediante el [endpoint de **app&#95;metadata**](https://auth0.com/docs/api/management/v2#!/Users/patch_users_by_id). <div id="disable-login-for-flagged-users">
  #### Deshabilitar el inicio de sesión de los usuarios marcados
</div>

A continuación, debes deshabilitar el inicio de sesión de los usuarios marcados como eliminados. Para ello, añadirás una Rule (un fragmento de JavaScript que se ejecuta como parte del pipeline de autenticación).

1. Ve a [Auth0 Dashboard &gt; pipeline de autenticación &gt; Rules](https://manage.auth0.com/#/rules) y crea una Rule.
2. Copia el siguiente script:

   ```javascript lines
   function (user, context, callback) {
     user.app_metadata = user.app_metadata || {};
     if (user.app_metadata.deleted){
     	return callback(new UnauthorizedError('Access denied (deleted user)'));
     }
     callback(null, user, context);
   }
   ```

   El script hace lo siguiente:

   1. Comprueba el valor de la propiedad de metadato **deleted** (`user.app_metadata.deleted`).
   2. Devuelve un error `Access denied (deleted user)` a tu aplicación si `user.app_metadata.deleted = true`
3. Asigna un nombre a tu Rule y guarda los cambios.

También debes:

* Asegurarte de que el proceso de retirada del consentimiento sea lo suficientemente granular.
* Configurar en la aplicación el área donde los clientes retirarán su consentimiento.
