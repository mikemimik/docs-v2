Algunas amenazas comunes de ciberseguridad incluyen:

* Ataques de relevo
* Ataques de repetición
* Ataques de falsificación de solicitud entre sitio (CSRF o XSRF)
* Ataques de fuerza bruta
* <Tooltip tip="Suspicious IP Throttling: Forma de protección contra ataques que protege su tenant frente a inicios de sesión sospechosos dirigidos a un número elevado de cuentas desde una sola dirección IP." cta="Ver glosario" href="/es/docs/glossary?term=Suspicious+IP+throttling">Suspicious IP throttling</Tooltip>
* Ataques de credential stuffing
* Ataques de validación de listas

Auth0 tiene [funciones de protección contra ataques](/es/docs/secure/attack-protection) que pueden proteger frente a ciertos tipos de ataques.

<div id="bucket-brigade-attacks">
  ## Ataques de relevo
</div>

Un tipo de amenaza es el ataque de relevo, a veces llamado ataque de &quot;man-in-the-middle&quot;, en el que el atacante se interpone entre dos partes. Cada parte cree que está interactuando a través de una conexión privada, pero en realidad la interacción está siendo controlada por un atacante externo. Para que este tipo de ataque tenga éxito, el atacante debe establecer autenticación mutua entre ambas partes.

Por lo general, los ataques de relevo implican que el atacante use un router WiFi para interceptar la comunicación de un usuario. El usuario se conecta al router del atacante, luego visita un sitio web e inicia sesión con sus credenciales confidenciales. El atacante guarda las credenciales de inicio de sesión de ese usuario, que luego puede utilizar para hacerse pasar por él.

Algunos ataques de relevo también pueden modificar los datos transmitidos entre la aplicación y el servidor. Estos ataques pueden producirse a través de versiones de prueba en PC nuevas, software preinstalado, herramientas de actualización de software y otras vulnerabilidades de software. En el peor de los casos, esto podría dar lugar a la ejecución remota de código, la instalación de puertas traseras en el sistema, la instalación de malware, etc. Esto también podría comprometer otros activos de la red.

Para ayudar a defenderse contra este tipo de ataque, es importante usar cifrado sólido y autenticación entre la aplicación y el servidor. Al usar cifrado, el servidor autentica la solicitud de la aplicación presentando un certificado digital, y solo entonces puede establecerse la conexión. Por ejemplo, HTTPS utiliza la funcionalidad de capa de sockets seguros (SSL) del navegador para ocultar el tráfico web. Para descifrar HTTPS, un atacante tendría que obtener las claves utilizadas para cifrar el tráfico de la red.

En la configuración de TLS en sus servidores, sugerimos usar las [recomendaciones de Mozilla OpSec](https://wiki.mozilla.org/Security/Server_Side_TLS), que utilizan TLSv1.2. Mozilla también proporciona un [Generador de configuración SSL](https://mozilla.github.io/server-side-tls/ssl-config-generator/); para usarlo, primero elija en qué plataforma se ejecuta su servidor y luego elija la configuración más moderna según lo que pueda admitir su aplicación. Las configuraciones más modernas proporcionan cifrados más robustos para ayudar a prevenir ataques.

<div id="replay-attacks">
  ## Ataques de repetición
</div>

Los ataques de repetición (o de reproducción) permiten a los atacantes:

* Obtener acceso a una red y a información a la que, de otro modo, no habría sido fácil acceder
* Completar una transaction duplicada

Se trata de ataques al protocolo de seguridad en los que se repite una transmisión de datos de un remitente distinto al sistema receptor previsto. Estos ataques engañan a los participantes para que crean que la transmisión de datos se ha completado correctamente.

Los ataques de repetición pueden evitarse mediante el uso de tokens de sesión. Sin embargo, si estas credenciales se roban del almacenamiento local (por ejemplo, durante un ataque XSS), hay formas de evitar que alguien conserve un token válido indefinidamente:

* Establecer un tiempo de expiración breve para los tokens
* Proporcionar una forma de agregar a la DenyList los tokens que ya se hayan usado (y posiblemente incluso a los usuarios)
* Usar contraseñas de un solo uso

La especificación [JWT](/es/docs/secure/tokens/json-web-tokens) proporciona el campo `jti` como una forma de prevenir ataques de repetición. Aunque actualmente los tokens de Auth0 no devuelven el `jti`, puedes agregar tokens a la DenyList usando el `jti` para evitar que un token se use más de un número determinado de veces. De esta forma, estás implementando algo similar a un <Tooltip tip="Nonce: Número arbitrario emitido una sola vez en un protocolo de autenticación para detectar y prevenir ataques de repetición." cta="Ver glosario" href="/es/docs/glossary?term=nonce">nonce</Tooltip> (piensa en la firma del token como si fuera el nonce). Si roban un token o se usa más veces de las especificadas, debe agregarse a la DenyList. Esto evita que un token válido se utilice con fines maliciosos. Una vez que el token expire, el atacante ya no podrá suplantar al usuario.

También puedes evitar ataques de repetición usando contraseñas de un solo uso. Con Auth0, puedes usar la autenticación <Tooltip tip="Passwordless: Forma de autenticación que no depende de una contraseña como primer factor." cta="Ver glosario" href="/es/docs/glossary?term=Passwordless">Passwordless</Tooltip>, que se basa en códigos de un solo uso y enlaces por correo electrónico en lugar de contraseñas tradicionales. Auth0 también ofrece <Tooltip tip="Autenticación multifactor (MFA): Proceso de autenticación de usuarios que utiliza un factor adicional al username y la Password, como un code por SMS." cta="Ver glosario" href="/es/docs/glossary?term=multi-factor+authentication">autenticación multifactor</Tooltip> (MFA), que utiliza contraseñas de un solo uso como segundo factor de autenticación y que puede enviarse mediante notificaciones push y mensajes de texto.

<div id="cross-site-request-forgery">
  ## Falsificación de solicitudes entre sitios
</div>

Un ataque de falsificación de solicitudes entre sitios (CSRF o XSRF) se produce cuando un programa malicioso hace que el navegador web de un usuario realice una acción no deseada en un sitio de confianza en el que el usuario tiene una sesión autenticada en ese momento. Este tipo de ataque se dirige específicamente a solicitudes que cambian el estado para ejecutar una acción, en lugar de obtener datos del usuario, ya que el atacante no tiene forma de ver la respuesta de la solicitud falsificada.

Una forma de verificar las solicitudes que se envían es utilizar el parámetro <Tooltip tip="OAuth 2.0: Marco de autorización que define protocolos y flujos de autorización." cta="Ver glosario" href="/es/docs/glossary?term=OAuth+2.0">OAuth 2.0</Tooltip> [state parameter](/es/docs/secure/attack-protection/state-parameters) para autenticar la respuesta.
