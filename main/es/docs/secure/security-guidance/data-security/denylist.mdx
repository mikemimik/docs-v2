Si hay campos de usuario que no deberían almacenarse en las bases de datos de Auth0 por motivos de privacidad, puedes agregarlos a la Lista de denegación. Para agregar atributos a la Lista de denegación, haz una llamada `PATCH` al endpoint Update Connection de la <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/es/docs/glossary?term=Management+API">Management API</Tooltip>.

1. Obtén un token de acceso válido para acceder al endpoint [endpoint](https://auth0.com/docs/api/management/v2#!/Connections/patch_connections_by_id) `/patch_connections_by_id`. El token debe incluir el scope `update:connections`. Consulta [Management API Access Tokens](/es/docs/secure/tokens/access-tokens/management-api-access-tokens) para obtener más información.
2. Con el token de acceso y la lista de atributos que deben denegarse, llama a la API. Aquí tienes un ejemplo de solicitud HTTP que deniega dos atributos: ethnicity y gender. Ten en cuenta que debes recuperar el objeto `options` y enviar el objeto completo en tu solicitud PATCH, ya que no existe ningún &quot;merge&quot; cuando solo actualizas uno o dos valores.

Donde:

1. `{yourConnectionId}` es el [ID de la conexión](/es/docs/authenticate/identity-providers/locate-the-connection-id) para la que se denegarán estos atributos.
   2. `{yourToken}` es el token de acceso que recibiste en el paso anterior.
   3. El objeto `options.non_persistent_attrs` contiene un array con los atributos que se denegarán. Si el claim que quieres denegar lo envía un proveedor de identidad (IdP) de origen, debes indicar el claim exactamente como lo envió ese IdP. Por ejemplo, para un claim recibido como `https://acme.com/temporary_idtoken`, el objeto `non_persistent_attrs` del ejemplo anterior quedaría así:

   ```json wrap lines
   {"non_persistent_attrs": ["ethnicity", "gender", "https://acme.com/temporary_idtoken"]}
   ```

<div id="limitations">
  ## Limitaciones
</div>

* Solo se pueden denegar los [campos raíz](/es/docs/manage-users/user-accounts/user-profiles/user-profile-structure) (como `user.name` o `user.phone_number`).

  * Si se deniegan `user.name` o `user.nickname`, no se incluirán en los tokens.
  * Si se deniega `user.email`, el valor no se puede asignar a un claim personalizado. Por ejemplo, en una regla, `context.idToken[namespace + 'work_email'] = user.email` no funcionaría.
* Cuando deniegas atributos, seguirán estando disponibles a través de reglas y en los tokens salientes. Sin embargo, si se cumple cualquiera de las siguientes condiciones, los atributos de la lista de denegación **no** se incluirán en los tokens:

  * Has habilitado la autenticación multifactor (MFA)
  * Has realizado una redirección mediante reglas
  * Tu aplicación usa delegación (y no has establecido `scope = passthrough`)
  * Tu aplicación usa suplantación
  * Has habilitado la opción de configuración **Usar Auth0 en lugar del IdP para realizar el inicio de sesión único** (solo para tenants Legacy)
* En las conexiones SAMLP, si habilitas el modo Debug, tus logs contendrán información sobre los atributos de la lista de denegación

Si alguna de estas limitaciones no te resulta aceptable, puedes escribir una [rule](/es/docs/customize/rules) para cifrar los datos y hacer que se conserven en el objeto `user.app_metadata`.
