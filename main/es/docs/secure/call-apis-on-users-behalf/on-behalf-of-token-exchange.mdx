El intercambio de tokens On-Behalf-Of (OBO) ([RFC 8693](https://www.rfc-editor.org/rfc/rfc8693.html)) permite que los servicios de nivel intermedio conserven la identidad y los permisos del usuario al llamar a APIs de destino.

Cuando una aplicación necesita llamar a una API de destino, puede usar:

* [Client Credentials Flow](/es/docs/get-started/authentication-and-authorization-flow/client-credentials-flow): La aplicación actúa en su propio nombre y se autentica como sí misma. La request puede haber sido iniciada por un usuario, pero ese contexto se perderá. El servicio de destino solo conoce la identidad de la aplicación que realiza la llamada.
* On-Behalf-Of (OBO) Token Exchange: La aplicación recibe un token con alcance de usuario y puede intercambiarlo por un nuevo token para llamar a servicios de destino. Esto preserva la identidad y el contexto del usuario final original a lo largo de toda la cadena de llamadas.

Por ejemplo, si un usuario inicia una llamada al Servicio A, que luego llama al Servicio B, el intercambio de tokens OBO permite que el Servicio A intercambie el token de acceso del usuario por un nuevo token que:

* Mantiene la identidad y los permisos del usuario original
* Tiene alcance específico para el Servicio B
* Permite que el Servicio B tome decisiones de autorización basadas en el usuario final

Los intercambios de tokens OBO activan el trigger de Action [`Post Login`](/es/docs/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger), donde:

* El [`event.transaction.protocol`](/es/docs/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger/post-login-event-object#param-protocol) se establece en `oauth2-token-exchange`.
* El [`event.transaction.actor`](/es/docs/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger/post-login-event-object#param-actor) rastrea toda la cadena de delegación.

Al igual que en un flujo de inicio de sesión estándar, los scopes devueltos para las llamadas a APIs de destino se basan en las políticas de [Role-Based Access Control (RBAC)](/es/docs/manage-users/access-control/rbac) del usuario.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Al adquirir el add-on Auth0 for AI Agents, puedes usar el límite de frecuencia máximo de Authentication API de tu nivel de suscripción para los intercambios de tokens OBO. Por ejemplo, si tienes [Private Cloud 100 RPS](/es/docs/troubleshoot/customer-support/operational-policies/rate-limit-policy/rate-limit-configurations/tier-100-rps-private-cloud), puedes superar el límite de frecuencia de 30 RPS del intercambio de tokens OBO y aprovechar toda la capacidad de 100 RPS para tus requests de intercambio de tokens OBO. Ponte en contacto con tu Technical Account Manager para obtener más información.
</Callout>

<div id="use-cases">
  ## Casos de uso
</div>

Entre los casos de uso más comunes del intercambio de tokens OBO se incluyen:

* Servidores MCP que necesitan llamar a API propias en nombre del usuario
* Microservicios que necesitan llamar a servicios de destino en nombre del usuario

Para permitir que sus aplicaciones llamen a API de terceros en nombre del usuario, use [Token Vault](/es/docs/secure/call-apis-on-users-behalf/token-vault).

<div id="how-it-works">
  ## Cómo funciona
</div>

El OBO token exchange permite que los servicios de nivel intermedio intercambien un token de usuario recibido por un nuevo token con permisos para un servicio de destino. El nuevo token conserva la identidad del usuario original, a la vez que registra la cadena de servicios implicados en el payload del JSON Web Token (JWT).

<div id="example-mcp-server-calls-first-party-api">
  ### Ejemplo: el servidor MCP llama a una API propia
</div>

Un usuario se autentica con Auth0 en una aplicación cliente, que después llama a un servidor MCP, el cual a su vez necesita llamar a una API propia.

<div id="step-1-user-authentication">
  #### Paso 1: Autenticación del usuario
</div>

Cuando el usuario inicia sesión, Auth0 emite un token de acceso para el servidor MCP con los siguientes claims en el payload del JWT:

```json
{
  "sub": "auth0|user123",
  "aud": "https://mcp-server.example.com",
  "azp": "spa_client_id" // o "client_id" según el dialecto del token
}
```

| Claim                                                                                                                | Valor                            | Descripción                                 |
| -------------------------------------------------------------------------------------------------------------------- | -------------------------------- | ------------------------------------------- |
| `sub`                                                                                                                | `auth0\|user123`                 | La identidad del usuario final              |
| `aud`                                                                                                                | `https://mcp-server.example.com` | Token con alcance para el servidor MCP      |
| `azp` (o `client_id` según el [Perfil del token de acceso](/es/docs/secure/tokens/access-tokens/access-token-profiles)) | `spa_client_id`                  | La aplicación cliente que solicitó el token |

<div id="step-2-obo-exchange">
  #### Paso 2: intercambio OBO
</div>

Mediante el intercambio de tokens OBO, el servidor MCP presenta el token del usuario a Auth0 y solicita un token de acceso con el alcance de la API propia. Auth0 emite un nuevo token de acceso para la API con los siguientes claims:

```json
{
  "sub": "auth0|user123",
  "aud": "https://first-party-api.example.com",
  "azp": "mcp_server_client_id", // o "client_id" según el dialecto del token
  "act": {
    "sub": "mcp_server_client_id",
    "act": {
      "sub": "spa_client_id"
    }
  }
}
```

| Claim                                                                                                                | Valor                                                                     | Descripción                                                                |
| -------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------- | -------------------------------------------------------------------------- |
| `sub`                                                                                                                | `auth0\|user123`                                                          | Se conserva la misma identidad del usuario                                 |
| `aud`                                                                                                                | `https://first-party-api.example.com`                                     | Token con alcance para la API propia                                       |
| `azp` (o `client_id` según el [perfil del token de acceso](/es/docs/secure/tokens/access-tokens/access-token-profiles)) | `mcp_server_client_id`                                                    | Cliente que solicitó el token (el servidor MCP que realizó el intercambio) |
| `act`                                                                                                                | `{"sub": "mcp_server_client_id",`<br />`"act": {"sub": "spa_client_id"}}` | Cadena de delegación que muestra todos los actores que intervienen         |

<div id="the-act-claim">
  #### El `claim` `act`
</div>

El `claim` `act` (actor) rastrea toda la cadena de delegación. Cada nivel de `act` representa un servicio de la cadena de llamadas, y el `act.sub` más externo identifica al actor actual que realizó el intercambio de tokens.

En nuestro ejemplo:

* `act.sub` más externo: `mcp_server_client_id` (el servidor MCP que acaba de intercambiar el token)
* `act.sub` anidado: `spa_client_id` (la aplicación cliente original)

El `claim` `azp` debe coincidir con el valor del `act.sub` más externo, identificando el servicio que realizó más recientemente el intercambio de tokens.

Si la API propia llama a otro servicio de destino (`https://calendar-api.acme.com`), la cadena de delegación se ampliaría:

```json
{
  "sub": "auth0|user123",
  "aud": "https://calendar-api.acme.com",
  "azp": "first_party_api_client_id",
  "act": {
    "sub": "first_party_api_client_id",
    "act": {
      "sub": "mcp_server_client_id",
      "act": {
        "sub": "spa_client_id"
      }
    }
  }
}
```

La cadena de delegación se limita a cinco niveles anidados. El intercambio de tokens OBO fallará si el token de sujeto ya contiene cinco niveles `act` anidados.

```json
400 Bad Request
{
  "error": "invalid_request",
  "error_description": "Delegation chain (`act` claim) depth exceeds the maximum allowed limit of 4"
}
```

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Almacena en caché los tokens de acceso durante la vida útil del token, en lugar de solicitar uno nuevo para cada llamada a la API. Los tokens de acceso se pueden reutilizar hasta que caduquen; los intercambios de tokens repetidos desperdician recursos, aumentan la latencia y pueden provocar límites de frecuencia.
</Callout>

<div id="user-mcp-server-api-flow">
  ### Usuario &gt; servidor MCP &gt; flujo de la API
</div>

El siguiente diagrama muestra un flujo de intercambio de tokens OBO de extremo a extremo en el que un servidor MCP invoca una API de primera parte en nombre del usuario:

```mermaid
sequenceDiagram
    participant User
    participant Client App
    participant Auth0
    participant MCP Server
    participant First-Party API

    Note over User,Auth0: Step 1: User Authentication
    User->>Client App: Authenticates (Login)
    Client App->>Auth0: Requests User Token
    Auth0-->>Client App: Issues Token A<br/>(aud: MCP Server, sub: UserX)

    Note over Client App,MCP Server: Step 2: Initial Request
    Client App->>MCP Server: Call Endpoint (Token A)
    Note right of Client App: Authorization: Bearer [Token A]

    Note over MCP Server,Auth0: Step 3: Validation and Token Exchange
    activate MCP Server
    MCP Server->>MCP Server: Validates Token A
    MCP Server->>Auth0: POST /oauth/token (Token Exchange)
    Note right of MCP Server: client_id: MCP_Server_Client_ID<br/>subject_token: [Token A]<br/>audience: First-Party API
    
    Note over Auth0: Step 4: Token Issuance
    activate Auth0
    Auth0-->>MCP Server: Issues Token B<br/>(aud: First-Party API, sub: UserX)
    deactivate Auth0
    Note right of Auth0: Token B contains act claim<br/>showing MCP Server as the actor

    Note over MCP Server,First-Party API: Step 5: Downstream Call
    MCP Server->>First-Party API: Call Endpoint (Token B)
    Note right of MCP Server: Authorization: Bearer [Token B]
    
    activate First-Party API
    First-Party API->>First-Party API: Validates Token B<br/>Identifies subject as UserX
    
    First-Party API-->>MCP Server: 200 OK (Data)
    deactivate First-Party API

    MCP Server-->>Client App: 200 OK (Processed Data)
    deactivate MCP Server
```

1. **Autenticación de usuarios**: El usuario se autentica con la aplicación cliente. El servidor de autorización de Auth0 emite el Token A, con los permisos del servidor MCP.
2. **Solicitud inicial**: La aplicación cliente llama al servidor MCP y pasa el Token A en el encabezado `Authorization: Bearer`.
3. **Validación e intercambio de tokens**: El servidor MCP recibe el Token A, lo valida y lo envía al endpoint `/oauth/token` del servidor de autorización de Auth0. Mediante el intercambio de tokens OBO, el servidor MCP presenta el Token A como `subject_token` y solicita un nuevo token para la API propia.
4. **Emisión de tokens**: El servidor de autorización de Auth0 emite el Token B. El Token B tiene el mismo `sub` (ID de usuario) que el Token A, pero el `aud` (`audience`) ahora es la API propia.
5. **Llamada descendente**: El servidor MCP llama a la API propia con el Token B. La API valida el Token B y comprueba que la solicitud se está realizando legítimamente «en nombre del» usuario original.

<div id="user-api1-api2-api3">
  ### Usuario &gt; API1 &gt; API2 &gt; API3
</div>

El siguiente diagrama muestra un flujo de extremo a extremo de una cadena de microservicios que realizan llamadas a servicios posteriores en nombre del usuario:

```mermaid
sequenceDiagram
    participant User
    participant Client App
    participant Auth0
    participant API1
    participant API2
    participant API3

    Note over User,Auth0: Paso 1: Autenticación del usuario
    User->>Client App: Se autentica (Login)
    Client App->>Auth0: Solicita token de usuario
    Auth0-->>Client App: Emite Token A<br/>(aud: API1, sub: UserX)

    Note over Client App,API1: Paso 2: Solicitud inicial
    Client App->>API1: Llama al endpoint (Token A)
    Note right of Client App: Authorization: Bearer [Token A]

    Note over API1,Auth0: Paso 3: API1 delega en API2
    activate API1
    API1->>API1: Valida Token A
    API1->>Auth0: POST /oauth/token (intercambio de tokens)
    Note right of API1: client_id: API1_ID<br/>subject_token: [Token A]<br/>audience: API2

    Note over Auth0: Paso 4: Emisión de token
    activate Auth0
    Auth0-->>API1: Emite Token B<br/>(aud: API2, sub: UserX)
    deactivate Auth0
    Note right of Auth0: Token B contiene el claim act<br/>que indica a API1 como actor

    Note over API1,API2: Paso 5: Llamada descendente
    API1->>API2: Llama al endpoint (Token B)
    Note right of API1: Authorization: Bearer [Token B]
    activate API2
    API2->>API2: Valida Token B<br/>Identifica al subject como UserX

    Note over API2,Auth0: Paso 6: API2 delega en API3
    API2->>Auth0: POST /oauth/token (intercambio de tokens)
    Note right of API2: client_id: API2_ID<br/>subject_token: [Token B]<br/>audience: API3
    activate Auth0

    Note over Auth0: Paso 7: Emisión de token
    Auth0-->>API2: Emite Token C<br/>(aud: API3, sub: UserX)
    deactivate Auth0
    Note right of Auth0: Token C contiene el claim act<br/>que indica a API2 como actor

    Note over API2,API3: Paso 8: Llamada descendente
    API2->>API3: Llama al endpoint (Token C)
    Note right of API2: Authorization: Bearer [Token C]
    activate API3
    API3->>API3: Valida Token C<br/>Identifica al subject como UserX
    API3-->>API2: 200 OK
    deactivate API3

    API2-->>API1: 200 OK (Datos)
    deactivate API2
    API1-->>Client App: 200 OK (Datos procesados)
    deactivate API1
```

1. **Autenticación del usuario**: El usuario se autentica correctamente en una aplicación cliente. El servidor de autorización de Auth0 emite el token A, con alcance para API1.
2. **Solicitud inicial**: La aplicación cliente llama a API1 y envía el token A en el encabezado `Authorization: Bearer`.
3. **API1 delega en API2**: API1 recibe el token A, lo valida y luego lo envía al endpoint `/oauth/token` del servidor de autorización de Auth0. Mediante el intercambio de tokens OBO, API1 presenta el token A como `subject_token` y solicita un nuevo token para API2.
4. **Emisión del token**: El servidor de autorización de Auth0 concede un nuevo token de acceso, el token B, a API1. El token B tiene el mismo `sub` (ID de usuario) que el token A, pero el `aud` (audiencia) ahora es API2.
5. **Llamada descendente**: API1 hace una solicitud a API2 con el token B.
6. **API2 delega en API3**: API2 recibe el token B, lo valida y luego lo envía al endpoint `/oauth/token` del servidor de autorización de Auth0. Mediante el intercambio de tokens OBO, API2 presenta el token B como `subject_token` y solicita un nuevo token para API3.
7. **Emisión del token**: El servidor de autorización de Auth0 concede un nuevo token de acceso, el token C, a API2. El token C tiene el mismo `sub` (ID de usuario) que los tokens A y B, pero el `aud` (audiencia) ahora es API3.
8. **Llamada descendente**: API2 hace una solicitud a API3 con el token C. API3 valida el token C y comprueba que la solicitud se está realizando legítimamente &quot;en nombre de&quot; el usuario original.

<div id="prerequisites">
  ## Requisitos previos
</div>

Solo los clientes de API personalizada asociados a un servidor de recursos pueden usar el intercambio de tokens OBO. Un cliente de API personalizada se vincula a un servidor de recursos cuando comparten el mismo identificador.

Los clientes de API personalizada tienen los siguientes requisitos:

* Configura `app_type` como `resource_server`.
* Configura `resource_server_identifier` como el servidor de recursos válido, es decir, `https://my-api.example.com`. Auth0 usa el identificador del servidor de recursos como parámetro audience en las llamadas de autorización.

Como los clientes de API personalizada son clientes first-party, asegúrate de [omitir el consentimiento del usuario](/es/docs/get-started/applications/confidential-and-public-applications/user-consent-and-third-party-applications#skip-consent-for-first-party-applications) para las API a las que tu cliente first-party necesita acceder.

  ### Crear cliente de API personalizada
</div>

Puedes crear un cliente de API personalizada con Auth0 Dashboard o Management API.

<Tabs>

    Una vez que la aplicación se haya creado correctamente, revísala seleccionando **Configure Application** y, después, desplázate hasta **Application Properties**. El **Application Type** es **Custom API Client**.

    <Frame>![My Test OBO API](/docs/images/call_apis_on_users_behalf/create_custom_api_client.png)</Frame>
  </Tab>

  <Tab title="Management API">
    Para crear un cliente de API personalizada con el mismo identificador que tu servidor de recursos, realiza una solicitud `POST` al endpoint [`/api/v2/clients`](https://auth0.com/docs/api/management/v2/clients/post-clients) con el siguiente cuerpo de la solicitud:

    ```bash
    curl --request POST 'https://{yourDomain}/api/v2/clients' \
      --header 'Content-Type: application/json' \
      --header 'Authorization: Bearer YOUR_MANAGEMENT_API_TOKEN' \

        "app_type": "resource_server",
        "resource_server_identifier": "https://my-api.example.com"
      }'
    ```

    | Parámetro                    | Descripción                                                                                                                                                 |
    | ---------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------- |
    | `name`                       | Nombre de tu cliente de API personalizada.                                                                                                                  |
    | `app_type`                   | El tipo de aplicación de tu cliente de API personalizada. Configúralo como `resource_server`.                                                               |
    | `resource_server_identifier` | El identificador único de tu cliente de API personalizada. Configúralo con la audiencia de tu servidor de recursos; es decir, `https://my-api.example.com`. |
  </Tab>
</Tabs>

<div id="create-client-grant">
  ### Crear autorización de cliente
</div>

Debes crear una autorización de cliente delegada por el usuario entre el cliente de API personalizada y la API de destino para autorizar el acceso.

<Tabs>
  <Tab title="Auth0 Dashboard">
    1. Ve a [**Applications &gt; Applications**](https://manage.auth0.com/#/applications) y selecciona tu cliente de API personalizada.
    2. En **API Access**, busca tu servidor de recursos (es decir, `https://my-api.example.com`) y selecciona **Edit**.
    3. En **User-Delegated Access**, selecciona **Grant Access** y luego los permisos que quieras conceder, o **Always grant all permissions**.
    4. Selecciona **Save**.
  </Tab>

  <Tab title="Management API">
    Haz una solicitud `POST` al endpoint [`/api/v2/client-grants`](https://auth0.com/docs/api/management/v2/client-grants/post-client-grants) con el siguiente cuerpo de la solicitud:

    ```bash
    curl --location 'https://{yourDomain}/api/v2/client-grants' \
      --header 'Content-Type: application/json' \
      --header 'Authorization: Bearer YOUR_MANAGEMENT_API_TOKEN' \
      --data '{
        "client_id": "YOUR_CLIENT_ID",
        "audience": "https://my-api.example.com",
        "scope": [
          "read:item"
        ],
        "subject_type": "user"
      }'
    ```
  </Tab>
</Tabs>

    ```bash

      --header 'Authorization: Bearer YOUR_MANAGEMENT_API_TOKEN' \
      --data '{
        "token_exchange": {
          "allow_any_profile_of_type": ["on_behalf_of_token_exchange"]
        }
      }'
    ```
  </Tab>
</Tabs>

<div id="perform-obo-token-exchange">
  ## Realizar un intercambio de tokens OBO
</div>

Para realizar un intercambio de tokens OBO, puedes usar [`auth0-api-js`](https://github.com/auth0/auth0-auth-js), [`auth0_api_python`](https://github.com/auth0/auth0-api-python) o la [Authentication API](https://auth0.com/docs/api/authentication).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Almacena en caché los tokens de acceso durante la vigencia del token, en lugar de solicitar uno nuevo para cada llamada a la API. Los tokens de acceso se pueden reutilizar hasta que expiren; repetir intercambios de tokens desperdicia recursos, aumenta la latencia y puede activar límites de frecuencia.
</Callout>

<Tabs>
  <Tab title="JavaScript">
    Antes de empezar, asegúrate de haber instalado la biblioteca [`auth0-api-js`](https://github.com/auth0/auth0-auth-js) y sus dependencias.

    Primero, inicializa `ApiClient` con las credenciales de tu servidor MCP:

    ```javascript
    import { ApiClient } from '@auth0/auth0-api-js';

    const apiClient = new ApiClient({
      domain: 'YOUR_AUTH0_DOMAIN',
      audience: 'YOUR_MCP_SERVER_AUDIENCE',
      clientId: 'YOUR_CLIENT_ID',
      clientSecret: 'YOUR_CLIENT_SECRET',
    });
    ```

    A continuación, usa el método `getTokenOnBehalfOf()` para intercambiar tokens:

    ```javascript
    const result = await apiClient.getTokenOnBehalfOf(accessToken, {
      audience: 'YOUR_DOWNSTREAM_API_AUDIENCE',
      scope: 'read:private',  // Opcional
    });
    ```

    `getTokenOnBehalfOf()` devuelve un objeto que contiene:

    * `accessToken`: El nuevo token para tu API downstream
    * `scope`: Los scopes concedidos
    * `expiresIn`: El tiempo de expiración del token en segundos
  </Tab>

  <Tab title="Python">
    Antes de empezar, asegúrate de haber instalado la biblioteca [`auth0_api_python`](https://github.com/auth0/auth0-api-python) y sus dependencias.

    Primero, importa las clases necesarias e inicializa `ApiClient` con las credenciales de tu servidor MCP:

    ```python
    from auth0_api_python import ApiClient, ApiClientOptions

    api_client = ApiClient(
        ApiClientOptions(
            domain='YOUR_AUTH0_DOMAIN',
            audience='YOUR_MCP_SERVER_AUDIENCE',
            client_id='YOUR_CLIENT_ID',
            client_secret='YOUR_CLIENT_SECRET',
        )
    )
    ```

    A continuación, usa el método `get_token_on_behalf_of()` para intercambiar tokens:

    ```python
    result = await api_client.get_token_on_behalf_of(
        access_token=access_token,
        audience='YOUR_DOWNSTREAM_API_AUDIENCE',
        scope='read:private'  # Opcional
    )
    ```

    `get_token_on_behalf_of()` devuelve un diccionario que contiene:

    * `access_token`: El nuevo token para tu downstream API
    * `scope`: Los scopes otorgados
    * `expires_in`: Tiempo de expiración del token en segundos
  </Tab>

  <Tab title="cURL">
    Envía una solicitud `POST` al endpoint `/oauth/token` con el siguiente cuerpo de la solicitud:

    ```bash
    curl --location 'https://YOUR_DOMAIN.us.auth0.com/oauth/token' \
      --header 'Content-Type: application/json' \
      --data '{
        "client_id": "YOUR_CLIENT_ID",
        "client_secret": "YOUR_CLIENT_SECRET",
        "subject_token": "AUTH0_SUBJECT_TOKEN",
        "grant_type": "urn:ietf:params:oauth:grant-type:token-exchange",
        "subject_token_type": "urn:ietf:params:oauth:token-type:access_token",
        "requested_token_type": "urn:ietf:params:oauth:token-type:access_token",
        "audience": "https://my-api.example.com"
      }'
    ```

    | Parámetro              | Ejemplo                                           | Descripción                                                                                                                                                                                                                              |
    | ---------------------- | ------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
    | `grant_type`           | `urn:ietf:params:oauth:grant-type:token-exchange` | Obligatorio. Indica al servidor de autorización que realice un intercambio en lugar de un inicio de sesión estándar.                                                                                                                     |
    | `client_id`            | `<custom_api_client_id>`                          | Obligatorio. El ID único del servicio de nivel intermedio que realiza la solicitud.                                                                                                                                                      |
    | `client_secret`        | `<custom_api_client_secret>`                      | Opcional. El secreto (o aserción) que se usa para autenticar al propio servicio de nivel intermedio. Puede usar cualquier método de autenticación de cliente; sin embargo, no puede configurar `token_endpoint_auth_method` como `none`. |
    | `subject_token`        | `<auth0_access_token>`                            | Obligatorio. El token recibido del usuario/cliente que el servicio de nivel intermedio tiene actualmente.                                                                                                                                |
    | `subject_token_type`   | `urn:ietf:params:oauth:token-type:access_token`   | Obligatorio. Define el formato del `subject_token` (por ejemplo, un token de acceso frente a un token de ID).                                                                                                                            |
    | `requested_token_type` | `urn:ietf:params:oauth:token-type:access_token`   | Obligatorio. Indica qué tipo de token quiere recibir (normalmente, un token de acceso para la siguiente API).                                                                                                                            |
    | `audience`             | `https://my-api.example.com`                      | Obligatorio. El identificador del servicio de destino que recibirá y validará el nuevo token.                                                                                                                                            |
    | `scope`                | `read:data write:data`                            | Opcional. Una lista de permisos específicos solicitados para la llamada al servicio de destino, delimitada por espacios.                                                                                                                 |

    Si la operación se realiza correctamente, debería recibir una respuesta como la siguiente:

    ```json
    {
      "access_token": "YOUR_AUTH0_ACCESS_TOKEN",
      "expires_in": 86400,
      "token_type": "Bearer",
      "issued_token_type": "urn:ietf:params:oauth:token-type:access_token"
    }
    ```

    | Parámetro           | Ejemplo                                         | Descripción                                                                                                                                                                                                                                                   |
    | ------------------- | ----------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
    | `access_token`      | `eyJ...`                                        | El token de acceso de Auth0 &quot;nuevo&quot;. Es el JWT o la cadena opaca que el servicio de nivel intermedio usará para llamar a la API de destino.                                                                                                         |
    | `issued_token_type` | `urn:ietf:params:oauth:token-type:access_token` | Confirma el formato del token devuelto. Coincide con `requested_token_type` de tu solicitud (o es un subconjunto de este).                                                                                                                                    |
    | `token_type`        | `Bearer`                                        | Especifica el esquema de autenticación en el header `Authorization`. Para OBO, es `Bearer`, a menos que el servicio de nivel intermedio y la API de destino usen DPoP, en cuyo caso se usará `DPoP`.                                                          |
    | `expires_in`        | `3600`                                          | La duración del token en segundos depende de la configuración de la API de destino. Ten en cuenta que suele ser menor que la del token original del usuario.                                                                                                  |
    | `scope`             | `read:data`                                     | Los permisos específicos otorgados al token. Debes habilitar estos permisos mediante una [autorización de cliente para acceso delegado por el usuario](/es/docs/get-started/applications/application-access-to-apis-client-grants#user-access-vs-client-access). |

<div id="organizations-support">
  ## Compatibilidad con Organizations
</div>

Cuando un usuario se autentica a través de una organización, el token de acceso incluye un claim `org_id`. El intercambio de tokens OBO conserva este contexto de organización a lo largo de la cadena de delegación.

Cuando Auth0 recibe una solicitud de intercambio de tokens OBO con un token de acceso vinculado a una organización, valida lo siguiente:

* Que el `org_id` exista en tu tenant
* Que el usuario (identificado por `sub`) sea miembro de esa organización

Si la validación falla, Auth0 rechaza la solicitud de intercambio de tokens. Si se realiza correctamente, Auth0 emite un nuevo token de acceso que:

* Contiene el mismo claim `org_id` que el token original
* Aplica las mismas políticas de RBAC específicas de la organización
* Pone el contexto de la organización a disposición en el [trigger de Actions `Post Login`](/es/docs/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger/post-login-event-object#event-organization) mediante la propiedad `event.organization`
