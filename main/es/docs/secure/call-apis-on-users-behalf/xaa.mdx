Conectar aplicaciones de terceros y agentes de IA en un entorno empresarial plantea dos problemas clave: poca visibilidad de TI sobre cómo se comparten los datos y flujos de consentimiento repetitivos para los usuarios.

Cross App Access (XAA) aborda estos desafíos al permitir que los administradores de TI definan de forma centralizada controles de acceso sobre cómo las aplicaciones SaaS, como los agentes de IA, se conectan en nombre de un usuario. Los administradores gestionan estas conexiones desde un panel central, como la Consola de administración de Okta, lo que elimina las molestas solicitudes de consentimiento de OAuth para los usuarios finales. El resultado es una mejora en la seguridad de la organización, la gobernanza y la experiencia del usuario.

XAA implementa la [Identity Assertion Authorization Grant](https://datatracker.ietf.org/doc/draft-ietf-oauth-identity-assertion-authz-grant/), una extensión de OAuth en desarrollo que permite que un agente de IA o una aplicación (aplicación solicitante) obtenga un token seguro a través del IdP empresarial. Este token permite que la aplicación solicitante llame a las API de otra aplicación (aplicación de recursos) en nombre del usuario final. Para obtener más información, consulta [How it works](#how-it-works).

<div id="key-benefits">
  ## Beneficios clave
</div>

XAA ofrece beneficios clave para cada rol de su ecosistema empresarial:

* Para los administradores de TI de empresas: control centralizado, visibilidad y aplicación de políticas sobre el acceso de las aplicaciones a los datos de la empresa y de los usuarios.
* Para los proveedores de SaaS y los desarrolladores: integración estandarizada y segura para la IA empresarial que impulsa el crecimiento del ecosistema.
* Para los usuarios finales: conexiones fluidas y sin fricciones entre aplicaciones, eliminando los complejos flujos de consentimiento de OAuth.

<div id="use-cases">
  ## Casos de uso
</div>

Algunos casos de uso habituales de XAA incluyen:

* Conectar agentes de IA con aplicaciones empresariales: un empleado usa un agente de IA para leer su aplicación de calendario y publicar una actualización sobre su disponibilidad en la aplicación empresarial de mensajería. En lugar de obligar al empleado a pasar por flujos de redirección y solicitudes de consentimiento, el agente de IA usa XAA para obtener un token de acceso del IdP empresarial y llamar de forma segura a las API tanto de la aplicación de calendario como de la aplicación de mensajería, si la política de acceso empresarial lo permite.
* Conectar aplicaciones SaaS: en nuestro ejemplo anterior, tanto la aplicación empresarial de calendario como la de mensajería son compatibles con XAA. Los empleados pueden conectar sin problemas la aplicación de mensajería para acceder a la API de la aplicación de calendario sin redirección del usuario ni consentimiento, mientras cumplen las políticas de acceso empresariales.

<div id="how-it-works">
  ## Cómo funciona
</div>

El flujo XAA involucra a los siguientes actores:

* Aplicación solicitante: La aplicación o agente de IA que necesita acceder a un recurso.
* Aplicación de recursos: La aplicación que posee el recurso protegido y lo expone mediante una API
* IdP empresarial: El IdP, como Okta, que autentica a los empleados.

Después de que el usuario final se autentica con el IdP empresarial, la aplicación solicitante se pone en contacto con el IdP empresarial para solicitar acceso a la aplicación de recursos en nombre del usuario. Tras aplicar su política de acceso para comprobar si esta conexión entre aplicaciones está permitida, el IdP empresarial genera una aserción llamada ID-JAG, que la aplicación solicitante presenta a la aplicación de recursos para obtener un token de acceso para consumir la API. 

En el siguiente diagrama, Acme es el cliente empresarial cuyos empleados se autentican con su IdP empresarial, como Okta, para acceder a la aplicación solicitante (Agent0) y a la aplicación de recursos (Todo0):

<Frame>![](/docs/images/xaa/xaa_high_level_diagram.png)</Frame>

* El servidor de autorización de la aplicación de recursos (Todo0) está federado con el IdP empresarial mediante OIDC para que pueda generar tokens de acceso para usuarios finales autenticados por ese IdP.
* La aplicación solicitante (Agent0) está registrada en el servidor de autorización de la aplicación de recursos como un cliente OAuth 2.0 con un `client_id` válido y credenciales para solicitar tokens de acceso al servidor de autorización de la aplicación de recursos.
* El administrador de TI de Acme ha definido controles de acceso XAA entre Agent0 y Todo0.

<div id="end-to-end-xaa-flow">
  ## Flujo XAA de extremo a extremo
</div>

Teniendo en cuenta nuestro ejemplo de Acme, el flujo XAA de extremo a extremo consta de los siguientes pasos:

1. El empleado de Acme inicia sesión en la aplicación solicitante (Agent0) mediante SSO con el IdP empresarial. La aplicación solicitante obtiene un token de identidad para verificar la identidad del empleado de Acme.
2. La aplicación solicitante realiza una solicitud de intercambio de tokens al IdP para canjear el token de identidad por una concesión de autorización JWT de aserción de identidad entre dominios, también conocida como ID-JAG. El IdP valida la solicitud y comprueba la política XAA definida por el administrador de TI de Acme.
3. Si la política XAA lo permite, el IdP devuelve el ID-JAG a la aplicación solicitante.
4. La aplicación solicitante realiza una solicitud de token al servidor de autorización de la aplicación de recursos (Todo0) mediante el ID-JAG.
5. El servidor de autorización de la aplicación de recursos valida el ID-JAG con la clave pública que también utiliza para su flujo de OpenID Connect con el IdP. Si es válido, el servidor de autorización devuelve un token de acceso.
6. La aplicación solicitante realiza una solicitud a la API de la aplicación de recursos con el token de acceso.

Al aprovechar el flujo XAA, las políticas del administrador de TI de Acme controlan el acceso de Agent0 a Todo0 sin necesidad de redirigir al usuario final ni de que interactúe.

<div id="beta-limitations">
  ## Limitaciones de la Beta
</div>

XAA Beta tiene las siguientes limitaciones:

* La aplicación solicitante debe ser un cliente confidencial y una aplicación propia en tu tenant de Auth0. Los clientes públicos, como las SPA y las aplicaciones nativas, no se admiten.
* La Administración delegada no es compatible. El cliente empresarial no puede configurar directamente conexiones SSO en tu tenant de Auth0. Está previsto incorporar compatibilidad con [SSO de autoservicio](/es/docs/authenticate/enterprise-connections/self-service-enterprise-configuration) en una versión posterior.
* Solo puede haber una conexión habilitada para XAA por cada emisor de IdP de origen. El mismo Okta tenant no puede usarse para más de una conexión empresarial habilitada para XAA.
* La compatibilidad con las organizaciones es limitada:
  * Una conexión tiene una asignación 1:1 con una organización. Varias organizaciones no pueden asignarse a la misma conexión para el acceso XAA.
  * Cuando la aplicación solicitante está configurada para requerir el uso de Organizations, los usuarios ya deben ser miembros de la organización de destino.
* Si no se especifica el parámetro `resource` en la solicitud de ID-JAG, la API de destino se determina según `tenant.default_audience`.
* No hay creación dinámica de usuarios: el usuario debe haber iniciado sesión previamente en tu aplicación de recursos usando la conexión de Okta configurada. De lo contrario, la solicitud para intercambiar la aserción de ID-JAG por un token de acceso fallará con un error `User not found error`.

<div id="rate-limits">
  ## Límites de frecuencia
</div>

En XAA Beta, los intercambios de ID-JAG en el endpoint `/token` de tu tenant de Auth0 estarán sujetos a un límite de 5 RPS.
