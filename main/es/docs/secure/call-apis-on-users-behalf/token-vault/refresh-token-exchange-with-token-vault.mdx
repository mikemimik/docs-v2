---
description: Aprende cómo una aplicación puede acceder a Token Vault para intercambiar un token de actualización de Auth0 por un token de acceso para invocar APIs externas.
title: Intercambio de token de actualización con Token Vault
---

Token Vault admite el intercambio de token de actualización, que permite a una aplicación cliente acceder a Token Vault para intercambiar un token de actualización de Auth0 (token de sujeto) por el token de acceso de un proveedor externo (token solicitado).

Como los tokens de actualización solo se intercambian a través de un canal seguro entre el cliente y el servidor de autorización, nunca se exponen al usuario final. Como resultado, los clientes pueden mantener la sesión de un usuario sin necesidad de que este vuelva a autorizar la conexión.

<div id="use-cases">
  ## Casos de uso
</div>

Entre los casos de uso habituales del intercambio de token de actualización se incluyen:

* Aplicación web: una aplicación web de productividad se conecta al Google Calendar de un usuario y realiza tareas en su nombre, como programar reuniones, sin necesidad de que el usuario vuelva a autenticarse.
* Aplicación móvil: una aplicación móvil de galería de fotos se conecta a la cuenta de Google Photos de un usuario y sube fotos a medida que se toman, manteniendo la sesión del usuario iniciada al actualizar el token de acceso en segundo plano.

<div id="prerequisites">
  ## Requisitos previos
</div>

Antes de empezar, debes [configurar el intercambio de token de actualización con Token Vault](/docs/es/secure/tokens/token-vault/configure-token-vault#configure-refresh-token-exchange).

<div id="step-1-connect-and-authorize-access">
  ## Paso 1: Conectar y autorizar el acceso
</div>

Para programar la reunión, la aplicación web necesita conectarse con Google a través de Auth0 y luego obtener el permiso del usuario para acceder a la API de Google Calendar.

El usuario inicia sesión en la aplicación con Google mediante el [Connected Accounts flow](/docs/es/secure/tokens/token-vault/connected-accounts-for-token-vault#how-it-works), que utiliza la [My Account API](/docs/es/manage-users/my-account-api). Si la aplicación usa [Organizations](/docs/es/manage-users/organizations), el usuario inicia sesión en la organización de destino antes de continuar. Una vez que la My Account API valida y completa la solicitud de Connected Accounts, almacena los tokens de acceso y de actualización de Google con los scopes de calendario solicitados en el Token Vault.

<div id="step-2-perform-refresh-token-exchange">
  ## Paso 2: Realizar el intercambio de token de actualización
</div>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Aunque Token Vault no admite la rotación del Refresh Token, puedes usar [DPoP](/docs/es/secure/sender-constraining/demonstrating-proof-of-possession-dpop) para vincular los tokens emitidos por Auth0 a tu cliente y añadir una capa adicional de seguridad. Para realizar correctamente el intercambio de token de actualización con Token Vault, desactiva Allow Refresh Token Rotation para tu aplicación en el Auth0 Dashboard.
</Callout>

La aplicación puede usar un token de actualización de Auth0 válido para solicitar un token de acceso de Google desde Token Vault con los scopes otorgados en el flujo de Connected Accounts. Este proceso permite que la aplicación obtenga un nuevo token de acceso sin necesidad de que el usuario vuelva a autorizar la conexión.

Para realizar el intercambio de token de actualización, la aplicación usa Auth0 SDKs para hacer una solicitud `POST` al endpoint `/oauth/token` con los siguientes parámetros:

```bash lines
curl --request POST 'https://{yourDomain}/oauth/token' \
--header 'Content-Type: application/json' \
--data '{
  "client_id": "<YOUR_CLIENT_ID>",
  "client_secret": "<YOUR_CLIENT_SECRET>",
  "subject_token": "<YOUR_AUTH0_REFRESH_TOKEN>",
  "grant_type": "urn:auth0:params:oauth:grant-type:token-exchange:federated-connection-access-token",
  "subject_token_type": "urn:ietf:params:oauth:token-type:refresh_token",
  "requested_token_type": "http://auth0.com/oauth/token-type/federated-connection-access-token",
  "connection": "google-oauth2"
}'
```

| Parámetro              | Descripción                                                                                                                                                                                                                                                                                                                                    |
| ---------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `grant_type`           | El tipo de concesión. Para Token Vault, configúrelo en `urn:auth0:params:oauth:grant-type:token-exchange:federated-connection-access-token`                                                                                                                                                                                                    |
| `client_id`            | ID de la aplicación cliente                                                                                                                                                                                                                                                                                                                    |
| `client_secret`        | Secreto de cliente. **Nota:** Puede usar cualquier método de autenticación de cliente para obtener el token de acceso de un proveedor externo.                                                                                                                                                                                                 |
| `subject_token_type`   | Tipo de token de sujeto. Para Token Vault, configúrelo como token de actualización: `urn:ietf:params:oauth:token-type:refresh_token`                                                                                                                                                                                                           |
| `subject_token`        | El token de actualización de Auth0 que el servidor de autorización de Auth0 valida para identificar al usuario.                                                                                                                                                                                                                                |
| `requested_token_type` | El tipo de token solicitado. Para Token Vault, configúrelo como el token de acceso del proveedor externo o `http://auth0.com/oauth/token-type/federated-connection-access-token`                                                                                                                                                               |
| `connection`           | El nombre de la conexión, en este caso, `google-oauth2`.                                                                                                                                                                                                                                                                                       |
| `login_hint`           | (Opcional) Use `login_hint` solo si el usuario tiene varias cuentas de la misma conexión, como una cuenta de Google del trabajo y una cuenta personal de Google. Cuando pasa un valor para `login_hint` durante el intercambio de tokens, identifica explícitamente para cuál de las múltiples cuentas vinculadas del usuario es la solicitud. |

<div id="step-3-auth0-authorization-server-validates-refresh-token">
  ## Paso 3: El servidor de autorización de Auth0 valida el token de actualización
</div>

El servidor de autorización de Auth0 valida y carga el perfil de usuario asociado al token de actualización de Auth0:

1. Auth0 verifica si el arreglo `connected_accounts` del perfil de usuario contiene una cuenta de usuario con el nombre de la conexión que se pasó en la solicitud de autorización.
2. Si la solicitud de autorización contiene `login_hint`, Auth0 busca una identidad que coincida tanto con el nombre de la conexión como con `login_hint`.
3. Si Auth0 no puede encontrar al usuario, devuelve un código de estado `401` con un mensaje de error.

Una vez que el servidor de autorización de Auth0 valida al usuario, localiza el token de acceso de Google en Token Vault. Si aún es válido, Auth0 devuelve el token de acceso de Google con sus scopes y su tiempo de expiración:

```json lines
{
  "access_token": "<YOUR_GOOGLE_ACCESS_TOKEN>",
  "scope": "https://www.googleapis.com/auth/calendar https://www.googleapis.com/auth/calendar.addons.execute https://www.googleapis.com/auth/calendar.events https://www.googleapis.com/auth/calendar.events.readonly https://www.googleapis.com/auth/calendar.settings.readonly https://www.googleapis.com/auth/userinfo.email https://www.googleapis.com/auth/userinfo.profile openid",
  "expires_in": 1377,
  "issued_token_type": "http://auth0.com/oauth/token-type/federated-connection-access-token",
  "token_type": "Bearer"
}
```

Si el token de acceso de Google ha caducado, Auth0 usa el token de actualización de Google almacenado en Token Vault para obtener un nuevo token de acceso de Google con los mismos scopes.

Con el token de acceso de Google, la aplicación llama a la API de Google Calendar en nombre del usuario.

<div id="external-provider-refresh-token-expiration-policy">
  ## Política de expiración del token de actualización del proveedor externo
</div>

Auth0 elimina los tokens de actualización de un proveedor externo cuando caducan, según la fecha de expiración establecida por dicho proveedor. Los tokens también se eliminan si no se han usado en un intercambio de tokens durante más de un año.
