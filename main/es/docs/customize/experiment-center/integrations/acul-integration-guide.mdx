Cuando un experimento está activo y se asigna una variación, Experiment Center inyecta un objeto `ExperimentContext` en tu componente [ACUL](/docs/es/customize/login-pages/advanced-customizations) como `experiment`.

<Warning>
  Durante la Beta, Experiment Center funciona solo en tenants de desarrollo. Los tenants de producción no están admitidos.
</Warning>

La inyección solo ocurre en las pantallas para las que te hayas suscrito. La suscripción se hace por pantalla agregando `"experiment"` al arreglo `context_configuration` de esa pantalla.

Para suscribir una pantalla y que reciba el contexto del experimento, haz una llamada `PATCH` al endpoint [`/api/v2/prompts/{prompt}/screen/{screen}/rendering`](/docs/es/api/management/v2/prompts/patch-rendering).

```json Example
    "context_configuration": ["experiment"]
```

Reemplaza `{prompt}` por el nombre del prompt (por ejemplo, `login`) y `{screen}` por el nombre de la pantalla (por ejemplo, `login`).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  La resolución del experimento y el enriquecimiento del registro del tenant siempre se ejecutan, independientemente de si una pantalla se ha habilitado o no. La habilitación solo controla si las propiedades de `experiment` se pasan a tu componente ACUL. Esta es una medida de minimización de datos: no habilites pantallas que no necesiten el contexto del experimento.
</Callout>

<div id="the-experiment-context-shape">
  ## La estructura del contexto del experimento
</div>

Cuando una pantalla se ha habilitado y hay un experimento activo, puedes acceder al contexto del experimento mediante `window.universal_login_context.experiment`.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Durante Experiment Center (Beta), el SDK de ACUL no añade automáticamente la propiedad `experiment`, por lo que debe definirse mediante `window.universal_login_context.experiment`.
</Callout>

```typescript
const experiment = window.universal_login_context.experiment;
```

El componente ACUL recibe un contexto de `experiment`:

```typescript
interface ExperimentContext {
  experiment_id: string;   // El ID del experimento activo
  variation_id: string;    // El ID de la variación asignada
  config: {                // Configuración combinada: línea base + sobrescritura
    [paramName: string]: { value: unknown };
  };
  is_control: boolean;     // True cuando esta es la variación de control
}
```

Cuando no hay ningún experimento activo (o cuando la funcionalidad no está habilitada para el tenant), `experiment` es `null`.

El parámetro `config` contiene la configuración completa resultante de la combinación para la variación asignada. Experiment Center toma los parámetros de línea base del flag de funcionalidad y les aplica las sobrescrituras de la variación asignada. Cada parámetro definido en el flag de funcionalidad siempre tiene un valor en `config`.

Por ejemplo, si tu flag de funcionalidad tiene un parámetro `button_label` con un valor de línea base de `"Sign in"` y la variación asignada lo sobrescribe con `"Continue"`, entonces `config.button_label.value` es `"Continue"`.
Para la variación de control (sin sobrescrituras), `config.button_label.value` es `"Sign in"`.

<div id="read-a-parameter-value">
  ## Leer el valor de un parámetro
</div>

Puedes acceder al valor de un parámetro mediante `config[paramName].value`:

```typescript
const experiment = window.universal_login_context.experiment;
const label = experiment?.config?.button_label?.value;
```

Usa el encadenamiento opcional (`?.`) en todos los casos. La prop `experiment` es `undefined` cuando no hay ningún experimento activo.

<div id="use-is_control">
  ## Usar is_control
</div>

El parámetro `is_control` es `true` cuando el usuario está en el grupo de control (recibió la línea base, sin sobrescrituras aplicadas). Úsalo cuando necesites identificar qué usuarios vieron la experiencia sin modificar o cuando quieras omitir el procesamiento opcional para los usuarios del grupo de control.

```typescript
if (!experiment?.is_control) {
  // Solo se ejecuta para usuarios en el grupo de tratamiento
  trackExperimentImpression(experiment.experiment_id, experiment.variation_id);
}
```

Para la renderización por bifurcación, comprueba el valor del parámetro directamente (`config.my_param.value`) en lugar de `is_control`.
Las comprobaciones basadas en parámetros son más legibles y siguen funcionando correctamente incluso si cambias qué variación es el control estadístico en un experimento futuro.

<div id="example-copy-variant-experiment">
  ## Ejemplo: experimento de variantes de texto
</div>

Este ejemplo muestra un flag de funcionalidad con dos variaciones que cambia el texto del botón. La versión de control usa la etiqueta estándar; el tratamiento usa una alternativa.

**Parámetros del flag de funcionalidad:**

```json
{
  "button_label": {
    "type": "string",
    "value": "Sign in",
    "description": "Label for the primary login button"
  }
}
```

**Variación de control:** sin sobrescrituras (hereda `button_label: "Sign in"`)

**Variación de tratamiento:**

```json
{
  "overrides": {
    "button_label": { "value": "Continue to your account" }
  }
}
```

**Componente ACUL que lee el parámetro:**

```typescript
// LoginScreen.tsx
export default function LoginScreen({...props }) {
  // config siempre tiene un valor — no se necesita valor de respaldo
  const experiment = window.universal_login_context.experiment;
  const buttonLabel = experiment?.config?.button_label?.value ?? "Sign in";

  return (
    <div>
      <h1>Welcome back</h1>
      <form onSubmit={props.onSubmit}>
        <input type="email" name="email" placeholder="Email" />
        <input type="password" name="password" placeholder="Password" />
        <button type="submit">{buttonLabel}</button>
      </form>
    </div>
  );
}
```

El valor alternativo `?? "Sign in"` de la última línea contempla el caso en que no haya ningún experimento activo (en cuyo caso `experiment` es `undefined` y `config?.button_label?.value` da como resultado `undefined`). Si lo prefieres, puedes usar una comprobación de `null` por separado:

```typescript
const buttonLabel = experiment
  ? experiment.config.button_label.value
  : "Sign in"; // Sin experimento activo; usar el valor predeterminado
```

<div id="example-boolean-feature-rollout">
  ## Ejemplo: despliegue gradual de una función con valor booleano
</div>

Este ejemplo usa un parámetro booleano para mostrar de forma condicional un nuevo elemento de la interfaz de usuario.

```typescript
// PostLoginScreen.tsx
export default function PostLoginScreen({...props }) {
  const experiment = window.universal_login_context.experiment;
  const showPasskeyBanner = experiment?.config?.show_passkey_banner?.value === true;

  return (
    <div>
      <p>You're signed in.</p>
      {showPasskeyBanner && (
        <div className="banner">
          <p>Set up a passkey for faster sign-in next time.</p>
          <button onClick={props.onEnrollPasskey}>Set up passkey</button>
          <button onClick={props.onDismiss}>Not now</button>
        </div>
      )}
    </div>
  );
}
```

La comprobación `=== true` (en lugar de una comprobación de veracidad) es intencional: garantiza que el banner solo se muestre cuando el parámetro sea explícitamente `true`, no cuando `experiment` sea `undefined` o falte `config`.

<div id="troubleshoot">
  ## Solución de problemas
</div>

La propiedad `experiment` es `undefined` en tres situaciones:

1. No hay ningún experimento activo actualmente en el tenant
2. La pantalla no se ha activado mediante `context_configuration`
3. Experiment Center no está habilitado en el tenant

Trata siempre `experiment` como si pudiera ser `undefined`. El patrón más seguro:

```typescript
const experiment = window.universal_login_context.experiment;
const myParam = experiment?.config?.my_param?.value;
// myParam es undefined cuando no hay ningún experimento activo
// Usa un valor por defecto: myParam ?? "your-default"
```

Nunca des por sentado que la propiedad `experiment` existe. El código que depende de que `experiment` esté definido provocará errores cuando no se esté ejecutando ningún experimento, que es lo habitual.
