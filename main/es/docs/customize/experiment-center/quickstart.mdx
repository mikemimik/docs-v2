<div id="prerequisites">
  ## Requisitos previos
</div>

Para empezar a usar Experiment Center, necesitas:

* **Un tenant de Auth0 de [desarrollo](/es/docs/get-started/tenant-settings#environment-tag)**

* **Una application de [Machine-to-Machine](/es/docs/get-started/apis/create-m2m-app-test#create-machine-to-machine-applications-for-testing)** con los siguientes scopes de Management API:

  ```text wrap lines
  read:experimentation
  create:experimentation
  update:experimentation
  delete:experimentation
  ```

<div id="1-create-and-activate-the-feature-flag">
  ## 1. Crea y activa la feature flag
</div>

Una feature flag define qué estás probando y las posibles variaciones.

Para crear una feature flag, realiza una llamada `POST` al endpoint [`/api/v2/experimentation/feature-flags`](/es/docs/api/management/v2/experimentation/create-feature-flag).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  * La respuesta incluye un valor `feature_flag_id`; necesitarás ese valor en los pasos posteriores.
  * La feature flag comienza con el estado `draft`.
</Callout>

<div id="add-two-variations">
  ### Añade dos variaciones
</div>

La feature flag necesita al menos dos variaciones antes de poder activarse: una de control y una de tratamiento.

<div id="create-the-control-variation">
  #### Crear la variación de control
</div>

La variación de control tiene sobrescrituras vacías. Representa la base sin cambios.
Para agregar una variación de control, realiza una llamada `POST` al endpoint [`/api/v2/experimentation/feature-flags/{feature_flag_id}/variations`](/es/docs/api/management/v2/experimentation/create-variation).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  La respuesta incluye un valor `variation_id` para el control. Lo necesitarás al configurar las asignaciones.
</Callout>

<div id="create-the-treatment-variation">
  #### Crear la variación de tratamiento
</div>

La variación de tratamiento realiza la sobrescritura de los parámetros que deseas cambiar.

Para añadir la variación de tratamiento, haz una llamada `POST` al [endpoint `/api/v2/experimentation/feature-flags/{feature_flag_id}/variations`](/es/docs/api/management/v2/experimentation/create-variation) y configura el objeto de sobrescritura `overrides`.

```json Example
    "overrides": {
      "show_passkey_prompt": { "value": true },
      "prompt_style": { "value": "modal" }
    }
```

<div id="activate-the-feature-flag">
  ### Activa la feature flag
</div>

Ahora que tienes dos variaciones, activa la feature flag. Un experimento no puede activarse a menos que la feature flag a la que hace referencia esté en estado `active`.

Para cambiar el estado de la feature flag a `active`, haz una llamada `POST` al endpoint [`/api/v2/experimentation/feature-flags/{feature_flag_id}/status`](/es/docs/api/management/v2/experimentation/update-feature-flag-status).

<div id="2-create-a-segment-optional">
  ## 2. Crea un segmento (opcional)
</div>

Si quieres dirigirte a un tráfico específico, crea un segmento. Omite este paso si quieres una división porcentual simple para todo el tráfico.

Para crear un segmento, realiza una llamada `POST` al endpoint [`/api/v2/experimentation/segments`](https://auth0.com/docs/api/management/v2/experimentation/create-segment).

Este ejemplo crea un segmento que selecciona a usuarios móviles en Estados Unidos:

```json Example
    "name": "mobile-us-users",
    "description": "Mobile users from the United States",
    "rules": [
      {
        "match_type": "all",
        "conditions": [
          { "attribute": "device_type", "operator": "equals", "value": "mobile" },
          { "attribute": "country", "operator": "equals", "value": "US" }
        ]
      }
    ]
```

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  La respuesta incluye un valor `segment_id`. Los segmentos solo se aplican cuando usas la estrategia de asignación [segment](/es/docs/api/management/v2/experimentation/create-experiment#body-allocation-strategy) en el experimento.
</Callout>

<div id="3-create-and-activate-an-experiment">
  ## 3. Crear y activar un experimento
</div>

Crea un experimento que haga referencia a tu feature flag y defina cómo se divide el tráfico.

Para crear un experimento, haz una llamada `POST` al endpoint [`/api/v2/experimentation/experiments`](/es/docs/api/management/v2/experimentation/create-experiment#body-allocations).

Este ejemplo usa una división porcentual de 90/10 mediante el objeto `allocations`: el 90 % de los usuarios recibe el control y el 10 % recibe el tratamiento. Este es un punto de partida habitual para un despliegue cauteloso.

```json Example
    "allocations": [
      {
        "variation_id": "var_Id",
        "weight": 90,
        "is_control": true
      },
      {
        "variation_id": "var_Id",
        "weight": 10,
        "is_control": false
      }
    ]

```

El experimento comienza en estado `draft` con `is_valid: false`. Esto es normal; la comprobación completa de disponibilidad se ejecuta en el siguiente paso.

<div id="validate-the-experiment">
  ### Validar el experimento
</div>

Antes de activarlo, confirma que el experimento esté listo. El endpoint `/validate` devuelve las mismas comprobaciones que se ejecutan durante la activación.

Para confirmar el experimento, realiza una solicitud `POST` al endpoint [`/api/v2/experimentation/experiments/{experiment_id}/validate`](/es/docs/api/management/v2/experimentation/validate-experiment).

La respuesta incluye el valor `is_valid` para confirmar si todo está listo para activarlo.

<div id="activate-the-experiment">
  ### Activa el experimento
</div>

Cuando el experimento sea válido y hayas probado ambas variaciones, actívalo.

Para cambiar el estado del experimento a `active`, haz una solicitud `POST` al endpoint [`/api/v2/experimentation/experiments/{experiment_id}/status`](/es/docs/api/management/v2/experimentation/update-experiment-status).

El valor `started_at` del experimento se establece la primera vez que se activa y no cambia si lo pausas y lo vuelves a activar.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  **Un experimento activo por tenant.** Si ya hay otro experimento activo, la activación devuelve `400 experiment_active_limit_exceeded`. Primero pausa o completa el otro experimento.
</Callout>

<div id="4-trigger-an-auth-event-and-observe-results">
  ## 4. Activar un evento de autenticación y observar los resultados
</div>

Activa un inicio de sesión de prueba en tu tenant de prueba. Por ejemplo, abre un navegador y ve a la URL de `/authorize` sin ningún parámetro de anulación.

Completa el flujo de inicio de sesión. El Experiment Center resuelve el experimento activo, asigna una variación mediante hash determinista, inyecta el contexto del experimento y enriquece el evento de autenticación resultante.

<div id="verify-enriched-logs">
  ### Verifica los logs enriquecidos
</div>

Puedes revisar los logs de Auth0 para ver los eventos del experimento:

* Ve a Auth0 [**Dashboard &gt; Monitoring &gt; Logs**](https://manage.auth0.com/dashboard/#/logs).

* Usa Log streaming: si ya envías los logs del tenant a una herramienta de análisis (Datadog, Splunk, Segment, etc.), los metadatos enriquecidos del experimento se transmiten automáticamente por el mismo flujo.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  El evento incluye metadatos del experimento en el objeto `details.experiment`. El campo `variation_id` indica qué variación se le asignó al usuario.
</Callout>

<Check>
  Has ejecutado correctamente tu primer experimento cuando:

  1. Tus eventos de autenticación en los logs del tenant incluyen `details.experiment` con `experiment_id` y `variation_id`
  2. Ambas variaciones producen el comportamiento correcto cuando se prueban con anulaciones de parámetros de consulta
  3. El mismo dispositivo o usuario recibe sistemáticamente el mismo `variation_id` en varios eventos de autenticación (asignación determinista)
</Check>
