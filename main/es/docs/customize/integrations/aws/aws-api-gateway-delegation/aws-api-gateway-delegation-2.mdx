<div id="step-2-secure-and-deploy-the-amazon-api-gateway">
  ## Paso 2: proteger e implementar Amazon API Gateway
</div>

Ahora que tu API ya está en funcionamiento, necesitas añadir seguridad. En este paso, harás lo siguiente:

* Proteger la API de actualización para limitar el acceso a usuarios autenticados con un rol específico de AWS IAM;
* Configurar la delegación de Auth0 para usar las capacidades de federación de AWS IAM;
* Obtener un AWS <Tooltip tip="Access Token: Credencial de autorización, en forma de una cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/docs/es/glossary?term=Access+Token">Access Token</Tooltip> que use el rol de AWS IAM.

Una vez protegida tu API, crearás una aplicación sin servidor y de página única (SPA). La SPA dependerá de la federación de identidades para determinar qué usuarios pueden acceder. Al combinar la integración de AWS IAM con Amazon API Gateway, la federación de identidades de AWS IAM para <Tooltip tip="Security Assertion Markup Language (SAML): Protocolo estandarizado que permite a dos partes intercambiar información de autenticación sin una contraseña." cta="Ver glosario" href="/docs/es/glossary?term=SAML">SAML</Tooltip> y Auth0 delegación para AWS, puedes permitir que usuarios de muchas fuentes distintas, incluidos proveedores sociales o conexiones empresariales, accedan a tus API. El siguiente diagrama ilustra un flujo de ejemplo que utiliza un <Tooltip tip="Identity Provider (IdP): Servicio que almacena y administra identidades digitales." cta="Ver glosario" href="/docs/es/glossary?term=Identity+Provider">proveedor de identidad</Tooltip> basado en SAML, junto con la federación SAML y delegación de Auth0 para AWS.

Verás dos formas de implementar este flujo:

1. Usar Auth0 delegación con AWS IAM;
2. Añadir un token de identidad para llevar la identidad a la función Lambda.

Delegación te permite obtener fácilmente tokens de AWS para acceder a los servicios de AWS en tu aplicación.

<div id="ways-to-secure-the-amazon-api-gateway">
  ### Formas de proteger Amazon API Gateway
</div>

AWS API Gateway ofrece varios métodos para proteger tus API:

1. claves de API;
2. IAM;
3. [Amazon Cognito](/docs/es/customize/integrations/aws/aws-api-gateway-cognito).

El uso de claves de API suele ser adecuado para interacciones de servicio a servicio, como se muestra a continuación. Sin embargo, este enfoque tiene varias desventajas:

* Incluir un secreto de larga duración en la aplicación es arriesgado (las aplicaciones son más fáciles de comprometer);
* Crear un sistema para emitir y administrar claves de API requiere una implementación segura que puede ser difícil de desarrollar.

En esta sección del tutorial se usarán roles y políticas de IAM para proteger tu API en API Gateway, aunque también puedes hacerlo con User Pools de Amazon Cognito. Para consultar instrucciones detalladas sobre cómo proteger tu API de AWS, lee [Protege AWS API Gateway con Cognito](/docs/es/customize/integrations/aws/aws-api-gateway-cognito). Para obtener más información sobre el uso de roles y políticas de IAM, consulta el artículo de Amazon [Controla el acceso a una API con permisos de IAM](http://docs.aws.amazon.com/apigateway/latest/developerguide/permissions.html). Para obtener más información sobre los User Pools en Cognito, visita [User Pool de Amazon Cognit](http://docs.aws.amazon.com/cognito/latest/developerguide/cognito-user-identity-pools.html).

<div id="step-1-configure-iam-and-auth0-for-saml-integration-with-the-api-gateway">
  ### Paso 1. Configurar IAM y Auth0 para la integración de SAML con API Gateway
</div>

Puedes especificar un rol de IAM de AWS para el token SAML que intercambias por un token de AWS. Según los permisos concedidos a ese rol de IAM (que se establecen mediante tu proveedor de identidad), el token recibido tendrá esos mismos permisos. Al emitir distintos tokens SAML, cada uno con su propio rol de IAM de AWS, puedes controlar los niveles de acceso de tus usuarios.

Por ejemplo, el IdP podría especificar el rol de IAM en función de la pertenencia a grupos (por ejemplo, un administrador en Active Directory) o del origen de la autenticación (por ejemplo, una conexión de base de datos o un proveedor social como Facebook). Este enfoque te permite diferenciar el acceso de los usuarios a tus métodos de Amazon API Gateway cuando están protegidos con AWS IAM.

<div id="configure-auth0">
  #### Configura Auth0
</div>

Inicia sesión en tu cuenta de Auth0. Se te redirigirá al Management Dashboard. Haz clic en **+ New Application**, que se encuentra en la esquina superior derecha de la página.

Ponle a tu nueva aplicación el nombre AWS API Gateway e indica que esta aplicación será una aplicación de página única. Haz clic en **Create**.

Ve a la pestaña Addons de la aplicación que acabas de crear. Con el control deslizante correspondiente, habilita Amazon Web Services. Esto activa AWS delegación.

<div id="configure-aws">
  #### Configurar AWS
</div>

Sigue el tutorial [Cómo configurar AWS para la autenticación delegada](/docs/es/customize/integrations/aws/how-to-set-up-aws-for-delegated-authentication) para configurar AWS para el acceso delegado, que utiliza SAML. Ten en cuenta lo siguiente:

* Sigue las [instrucciones que aparecen a continuación](#setting-the-permissions-policy-on-your-iws-iam-role) para adjuntar la política de permisos a tu rol, en lugar de las del tutorial enlazado;
* Asigna el nombre `auth0` al proveedor de SAML que crees;
* Asigna el nombre `auth0-api-role` al rol de AWS IAM.

<div id="set-the-permissions-policy-on-your-iws-iam-role">
  ##### Establezca la política de permisos en su rol de IAM de IWS
</div>

Una vez que haya configurado el rol de IAM de AWS, agregará una política a `auth0-api-role` que le permitirá ejecutar los métodos de su API Gateway. Para obtener más información sobre este proceso, consulte [User Access Permissions for Amazon API Gateway](http://docs.aws.amazon.com/apigateway/latest/developerguide/permissions.html).

<div id="getting-the-gateway-api-arn">
  #### Obtener el ARN de la API de Gateway
</div>

Antes de comenzar, necesitarás el ARN de tu API de Gateway:

1. Ve a [Amazon API Gateway Console](https://console.aws.amazon.com/apigateway) e inicia sesión.
2. Selecciona la API correspondiente.
3. Haz clic en cualquiera de los **Methods** asociados a la API para abrir la página **Method Execution**.
4. En la página **Method Execution**, el cuadro **Method Request** de la esquina superior izquierda muestra el **ARN** de la API, aunque incluye el nombre del método:

`arn:aws:execute-api:us-east-2:484857107747:97i1dwv0j4/*/POST/`

Deberás quitar el nombre del método para obtener el ARN base de la API:

`arn:aws:execute-api:us-east-2:484857107747:97i1dwv0j4/*`

El comodín (`*`) en el ARN anterior habilita permisos para tu API en todas las etapas, pero puedes implementar distintas etapas por separado (por ejemplo, desarrollo, luego pruebas y después producción).

Selecciona el rol `auth0-api-role` que acabas de crear para abrir su página **Summary**.

Expande **Inline Policies** y haz clic en **click here**.

Selecciona **Custom Policy** y haz clic en **Select**.

Edita tu documento de política. Puedes establecer **Policy Name** como quieras, pero te sugerimos algo como `api-gateway-policy`. Para habilitar el acceso a los métodos de la API para este rol, aplica la siguiente política después de actualizar el ARN con el de tu API.

```json lines
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Effect": "Allow",
            "Action": [
                "execute-api:*"
            ],
            "Resource": [
                "arn:[{yourArn}]"
            ]
        }
    ]
}
```

Haz clic en **Aplicar política**.

Como API Gateway asumirá este rol en nombre del usuario, la política de confianza debe permitir esta acción. Para ello, edita las relaciones de confianza del rol y ve a esta pestaña en la página de Resumen del rol.

La relación de confianza final debería verse similar a la siguiente:

En este punto, tendrás que configurar la autorización en [API Gateway](https://console.aws.amazon.com/apigateway).

En la vista **Resources**, selecciona el método POST de `/pets`.

Haz clic en el enlace **Method Request**.

<div id="step-2-set-up-cors-and-deploy-the-api">
  ### Paso 2. Configura CORS y despliega la API
</div>

Nuestra aplicación de página única (SPA) accederá a métodos de la API web desde un dominio distinto del de la página. La configuración de Cross-Origin Resource Sharing debe permitir explícitamente esta acción para que el navegador autorice el acceso a AWS API Gateway. Normalmente, el navegador primero enviará una solicitud `OPTIONS` para ver qué acciones permite el sitio.

Selecciona `/pets` en **Resources** y haz clic en **Create Method**. En el menú desplegable, selecciona **OPTIONS** y haz clic en la **marca de verificación** para guardar la configuración.

El navegador usa el método Options para obtener los encabezados HTTP necesarios, pero la función necesita instrucciones adicionales sobre qué hacer. En la pantalla de Setup de `OPTIONS`, configura las siguientes variables/parámetros:

* **Integration Type**: Lambda Function;
* **Use Lambda Proxy Integration**: déjalo sin marcar;
* **Lambda Region**: selecciona tu región;
* **Lambda Function**: NoOp.

Haz clic en **Save**. En la siguiente ventana emergente, concede a tu función Lambda los permisos que necesita.

Luego se te redirigirá automáticamente a la página `OPTIONS` Method Execution. Abre la página Method Response.

Expande la sección **200** ubicada debajo de la barra HTTP Status y agrega los siguientes encabezados de respuesta:

* Access-Control-Allow-Headers;
* Access-Control-Allow-Methods;
* Access-Control-Allow-Origin.

A continuación, asigna los valores correspondientes a cada uno de los encabezados de respuesta. Después de volver a la página Method Execution, haz clic en **Integration Response**. Después de expandir la fila asociada con el estado de respuesta del método **200**, expande **Header Mappings** y aplica los siguientes mapeos:

* Access-Control-Allow-Headers: `'Content-Type,X-Amz-Date,Authorization,x-api-key,x-amz-security-token'`;
* Access-Control-Allow-Origin: `'*'`
* Access-Control-Allow-Methods: `'POST, GET, OPTIONS'`

Por último, repite los pasos anteriores para habilitar CORS en los métodos POST y GET. Sin embargo, para estos dos métodos, agregarás un encabezado, Access-Control-Allow-Origin, y su valor debe establecerse en `'*'`.

<div id="deploy-the-api">
  ### Despliega la API
</div>

Vuelve a la vista de **Resources** de tu API. Haz clic en **Actions** y selecciona **DEPLOY API**.

Selecciona **New Stage** en **Deploy State** y asigna a la stage el nombre `Test`. Haz clic en el botón **Deploy**.

En la página de resultados, ve a **SDK Generation**. Selecciona JavaScript como **Platform**. Haz clic en el botón **Generate SDK**.

Guarda el archivo ZIP descargado para usarlo más adelante.
