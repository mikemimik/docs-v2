<div id="step-4-use-multiple-roles-with-amazon-api-gateway">
  ## Paso 4: usa varios roles con Amazon API Gateway
</div>

En este paso, asignarás diferentes roles de AWS IAM a los usuarios según la información de autenticación:

* Los usuarios que se autentiquen con Social Connections serán tratados como compradores;
* Los usuarios que se autentiquen con Database Connections serán tratados como administradores.

Implementarás esta lógica de asignación de roles de dos maneras diferentes:

* JavaScript;
* rules de Auth0.

En muchas aplicaciones de Auth0, querrás que distintos usuarios tengan diferentes niveles de acceso, y querrás usar información adicional sobre una identidad determinada en la lógica de tu servicio. En los casos en que baste con restringir el acceso a nivel de API, puedes usar diferentes roles de AWS IAM (por ejemplo, los administradores pueden usar la función de actualización para agregar y eliminar mascotas, mientras que los usuarios de conexiones sociales solo pueden comprar mascotas).

El siguiente diagrama ilustra las asignaciones de roles de AWS IAM para dos clases distintas de usuarios: usuarios autenticados mediante Social Connections y usuarios autenticados mediante Database Connections. También muestra que los roles de AWS IAM pueden asignarse a otras entidades, como las funciones de AWS Lambda, para controlar los permisos que se asignan a esas entidades dentro de una cuenta. En resumen, un rol de IAM es un conjunto de permisos sobre capacidades de AWS que se define mediante una o más políticas y luego se asigna a una entidad.

En los casos en que quieras tomar decisiones dentro de tu código (por ejemplo, podrías necesitar una verificación de crédito para un usuario que compra una mascota), también querrás transferir la identidad. Esto se demostrará a continuación en [Paso 5: uso de Identity Tokens para transferir la identidad](/docs/es/customize/integrations/aws/aws-api-gateway-delegation/aws-api-gateway-delegation-5).

<div id="step-1-create-the-petpurchase-api-resource">
  ### Paso 1. Crear el recurso de la API PetPurchase
</div>

En la [Amazon API Gateway Console](https://console.aws.amazon.com/apigateway), selecciona tu API de Pets. Se te llevará a su página Resources.

Haz clic en **Actions** y en **Create Resource**. Asigna a New Child Resource el nombre `Purchase`. Haz clic en **Create Resource**.

Agrega un método OPTIONS para el recurso `purchase`, como se indicó anteriormente para `pets` en la [sección Configurar CORS e implementar la API del Paso 2: Protección e implementación de Amazon API Gateway](/docs/es/customize/integrations/aws/aws-api-gateway-delegation/aws-api-gateway-delegation-2#set-up-cors-and-deploy-the-api).

Crea una nueva función de AWS Lambda para la compra de una mascota llamada `PetPurchase`, que agrega los atributos `isSold` y `soldTo` a una mascota de la siguiente manera:

```javascript lines expandable
var AWS = require('aws-sdk');
var DOC = require('dynamodb-doc');
var dynamo = new DOC.DynamoDB();

exports.handler = function(event, context) {
   var petId = event.petId;
   var user = event.userName;
   var pets = {};
   console.log('start PetsPurchase, petId', petId, ' userName', user);

   var writecb = function(err, data) {
      if(!err) {
          context.done(null, pets);
      } else {
          console.log('error on GetPetsInfo: ',err);
          context.done('failed on update', null);
      }
   };

   var readcb = function(err, data) {
      if(err) {
          console.log('error on GetPetsInfo: ',err);
          context.done('failed to retrieve pet information', null);
      } else {
          // verificar que tenemos mascotas
          if(data.Item && data.Item.pets) {
              pets = data.Item.pets;
              var found = false;

              for(var i = 0; i < pets.length && !found; i++) {
                  if(pets[i].id === petId) {
                     if(!pets[i].isSold) {
                        pets[i].isSold = true;
                        pets[i].soldTo = user;
                        var item = { username:"default",pets: pets};
                        dynamo.putItem({TableName:"Pets", Item:item}, writecb);
                        found = true;
                     }
                  }
               }
               if(!found) {
                 console.log('pet not found');
                 context.done('That pet is not available.', null);
               }
           } else {
              console.log('pet already sold');
              context.done('That pet is not available.', null);
           }
       }
   };

   dynamo.getItem({TableName:"Pets", Key:{username:"default"}}, readcb);
};
```

Una vez que haya definido la función Lambda, [agregue un método POST al recurso `purchase`](/docs/es/customize/integrations/aws/aws-api-gateway-delegation/aws-api-gateway-delegation-1#method-post-pet-information) que invoque la Lambda `PetPurchase`. Asegúrese también de agregar el encabezado `Access-Control-Allow-Origin` con un valor de `*` al método POST mediante la configuración de respuesta del método/respuesta de integración que se encuentra en la [sección Set Up Cors and Deploy the API del Paso 2: Securing and Deploying the Amazon API Gateway](/docs/es/customize/integrations/aws/aws-api-gateway-delegation/aws-api-gateway-delegation-2#set-up-cors-and-deploy-the-api).

Pruebe el método de API Gateway proporcionando lo siguiente como mensaje de entrada:

```json lines
{
    "petId": 1,
    "userName": "fred flintstone"
 }
```

En la respuesta de prueba, deberías ver que la mascota con ID 1 ahora se ha vendido a Fred Flintstone:

```json lines
[
  {
    "id": 1,
    "price": 249.99,
    "type": "dog",
    "isSold": true,
    "soldTo": "fred flintstone"
  },

  ...
```

<div id="update-iam">
  #### Actualizar IAM
</div>

Para proteger tu API, sigue el mismo proceso para agregar un nuevo rol que [seguiste en la Parte 2](/docs/es/customize/integrations/aws/aws-api-gateway-delegation/aws-api-gateway-delegation-2) de este tutorial. Asigna al nuevo rol el nombre `auth0-api-social-role`.

El ARN del método que vas a proteger en la política de IAM debería verse más o menos así:

```bash wrap lines
arn:aws:execute-api:us-east-1:your-accountid:your-api-id/*/pets/purchase
```

Asegúrate de actualizar también la política de confianza.

Ve a la [Amazon API Gateway Console](https://console.aws.amazon.com/apigateway) y selecciona el método POST para el recurso `/pets/purchase`. Selecciona **Method Request** y cambia **Authorization Type** a AWS&#95;IAM. Haz clic en la marca de verificación para guardar la configuración.

En este punto, has definido dos roles que puedes usar con API Gateway:

* `auth0-api-role`: permite actualizar mascotas
* `auth0-api-social-role`: permite comprar una mascota

<div id="configure-login-with-amazon-and-update-auth0">
  #### Configura Login with Amazon y actualiza Auth0
</div>

Puedes crear un rol social con Login with Amazon (LWA).

Aunque este tutorial incluye instrucciones para usar Login with Amazon, ten en cuenta que también puedes usar otros proveedores sociales.

1. Ve a [Auth0 Dashboard &gt; Authentication &gt; Social](https://manage.auth0.com/#/connections/social) y selecciona **Create Connection**.
2. Elige la conexión que quieras configurar y da tu consentimiento.
3. Copia y pega el `Client ID` y el `Client Secret` de tu proveedor de identidad social, selecciona los **Attributes** (y **Permissions**, cuando corresponda) y haz clic en **Save**.
4. Selecciona la vista **Applications**, habilita el interruptor de cada una de tus aplicaciones de Auth0 que deban poder usar esta conexión y selecciona **Save**.

Una vez que hayas introducido la información correspondiente, selecciona **Try Connection** para asegurarte de que todo esté configurado correctamente.

Cuando configures LWA desde la consola de Amazon, asegúrate de introducir en Allowed Return URLs la URL de devolución de llamada de tu aplicación de Auth0, que debería verse como `https://johndoe.auth0.com/login/callback`. La página de ayuda de Auth0 te mostrará exactamente qué debes introducir.

Ve a [Auth0 Dashboard &gt; Applications &gt; Applications](https://manage.auth0.com/#/applications) y selecciona tu aplicación para ver su configuración. Selecciona la vista **Connections**, localiza la sección **Social** y asegúrate de que **Amazon** esté habilitado.

<div id="enforce-role-assignment-with-auth0-rules">
  ### Aplicar la asignación de roles con Auth0 Rules
</div>

En algunos casos, puedes determinar el rol adecuado usando la aplicación (como se muestra aquí), pero por motivos de seguridad (por ejemplo, para evitar que el usuario asuma un rol con más privilegios de los necesarios), quizá te convenga determinar los privilegios del usuario del lado del servidor.

Con Auth0, esto se hace mediante rules, que son instrucciones de lógica de servicio que defines y que luego se ejecutan durante el proceso de autenticación de Auth0. Por ejemplo, podrías crear rules para:

* Evitar pasar información de roles del navegador a la aplicación;
* Insertar información de roles en la solicitud de delegación según la fuente de autenticación.

<div id="enforce-role-assignment">
  #### Aplicar la asignación de roles
</div>

Agregará una rule que comprobará si el role solicitado por el usuario está permitido, en función de su asociación con una conexión Social o una conexión de base de datos.

1. Vaya a [Auth0 Dashboard &gt; pipeline de autenticación &gt; Rules](https://manage.auth0.com/#/rules) y seleccione **Create Rule**.
2. Elija la plantilla **Empty rule**
3. Asigne a la rule el nombre **AWS Pets** (o algo similar) y, a continuación, complete el cuerpo de la rule con el siguiente código JavaScript:

export const codeExample = `   function (user, context, callback) {
  if(context.clientID === '{yourClientId}') {
    var socialRoleInfo = {
      role:"arn:aws:iam::<your account>:role/auth0-api-social-role",
      principal: "arn:aws:iam::your account>:saml-provider/auth0"
    };

    var adminRoleInfo = {
      role:"arn:aws:iam::<your account>:role/auth0-api-role",
      principal: "arn:aws:iam::<your account>:saml-provider/auth0"
    };

    var requestRole = context.request.body.role;
    var requestPrincipal = context.request.body.principal;
    var allowedRole = null;

    if(user.identities[0].isSocial === false) {
      allowedRole = adminRoleInfo;
    } else {
      allowedRole = socialRoleInfo;
    }

    if((requestRole && requestRole !== allowedRole.role) ||
      (requestPrincipal && requestPrincipal !== allowedRole.principal)) {
        console.log('el rol solicitado no coincide:',requestRole, ':', requestPrincipal);
        console.log('sobrescribiendo');
    } else {
      console.log('es válido o no se solicitó ningún rol para Delegation');
    }

    context.addonConfiguration = context.addonConfiguration || {};
    context.addonConfiguration.aws = context.addonConfiguration.aws || {};
    context.addonConfiguration.aws.role = allowedRole.role;
    context.addonConfiguration.aws.principal = allowedRole.principal;
    callback(null, user, context);

  } else {
    callback(null, user, context);
  }
}
`;

Asegúrese de ajustar el código anterior con los valores correctos para su integración. Los campos son **ARN principal**, **ARN del rol** y **secreto de cliente**.
4. **Guarde** los cambios.

<div id="caveats">
  #### Advertencias
</div>

* Rules se ejecuta con alcance global en cada autenticación. Solo debes ejecutar la lógica en las solicitudes de autenticación asociadas a una aplicación determinada (por eso el script utilizado pide el clientID). Sin esta información, la lógica se ejecuta en cada solicitud de autenticación asociada a tu cuenta de Auth0.
* La información se pasa a la rule mediante el contexto y el usuario.
* Puedes ampliar los objetos que se pasan a la rule. En el código anterior, la rule comprueba el cuerpo de la solicitud para obtener la información del rol. El rol se establece en el addonConfiguration del contexto como el rol permitido, que siempre sobrescribe la configuración del cuerpo de la solicitud.
