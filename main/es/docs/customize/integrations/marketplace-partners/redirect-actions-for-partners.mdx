title: Actions de redirección para partners
---

Cuando un usuario inicia sesión o se registra, una Action de redirección te permite redirigirlo a una página externa (como un formulario de consentimiento) y luego devolverlo a Auth0 para completar su inicio de sesión o registro. Puedes redirigir a los usuarios para que accedan a una aplicación externa que les solicite:

* Realizar una acción, como la verificación de identidad
* Proporcionar información, como el progressive profiling
* Aceptar algo, como el consentimiento o los Términos de servicio

Este proceso ocurre en una Action de redirección:

1. La **aplicación del cliente** redirige al usuario a Auth0 para iniciar sesión.
2. Si el inicio de sesión se realiza correctamente, se ejecutan todas las Actions del [trigger Post Login](/es/docs/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger) (esto sucede antes de MFA, si está activo).
3. Si una Action activa una redirección, el usuario es enviado a la URL designada junto con un parámetro state. Esta URL debe estar alojada por tu servicio o por el cliente.
4. El usuario es redirigido o enviado mediante POST de vuelta a Auth0 en una ruta específica de su dominio, junto con el valor state original, y la Action ejecuta cualquier código que exista en `onContinuePostLogin`.
5. El usuario es enviado de vuelta a la aplicación con su identidad o, si algo falló, con un mensaje de error.

Una vez que estés listo para integrar tu servicio en el proceso, hay elementos importantes que debes tener en cuenta:

* ¿Cómo determinas cuándo redirigir fuera de Auth0?

  * ¿Con un indicador en el [app&#95;metadata del usuario](/es/docs/manage-users/user-accounts/metadata/manage-metadata-rules)?
  * ¿En función de un campo de metadata específico en el cliente?
* ¿Cómo gestionas los datos existentes del perfil de usuario que deban verificarse? (Estos datos pueden haber sido proporcionados por el usuario o provenir de una fuente de identidad federada, como Google, Facebook o Azure AD).
* ¿Qué datos necesitas de Auth0 en tu servicio y cómo los haces llegar allí de forma segura?
* ¿Cómo conservas el valor state de Auth0 en tu servicio?
* ¿Cómo obtienes y conservas la URL `/continue` a la que quieres hacer POST o redirigir?
* ¿Qué envías de vuelta a Auth0 y cómo lo haces de forma segura?
* ¿Cómo indicas que la verificación de identidad se completó y transmites el status?
* ¿Cómo almacenas la información necesaria en el [app&#95;metadata del usuario](/es/docs/manage-users/user-accounts/metadata/manage-metadata-rules) o en el [perfil de usuario normalizado](/es/docs/manage-users/user-accounts/user-profiles/user-profile-structure)
* Ten en cuenta los [límites de frecuencia](/es/docs/troubleshoot/customer-support/operational-policies/rate-limit-policy/management-api-endpoint-rate-limits) y actualiza solo cuando sea necesario
* ¿Cómo envías información de vuelta a la aplicación solicitante usando [claims de token personalizadas](/es/docs/get-started/apis/scopes/sample-use-cases-scopes-and-claims)?

Para ayudarte a responder todas estas preguntas y más, lee [Redirección con Actions](/es/docs/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger/redirect-with-actions). Cuando estés listo para enviar tu integración de Action, sigue los pasos 4-6 de [Integraciones de Action para partners.](/es/docs/customize/integrations/marketplace-partners/actions-integrations-for-partners)
