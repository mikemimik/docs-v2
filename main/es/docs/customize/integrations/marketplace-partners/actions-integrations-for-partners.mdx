description: Instrucciones para partners empresariales de Auth0 sobre cómo crear soluciones sin código que conecten productos de terceros con los flujos de trabajo de Auth0
title: integraciones de Actions para partners
---

Partners como tú usan integraciones de Actions para crear soluciones sin código que conectan sus productos con los flujos de trabajo de Auth0. integraciones de Actions permiten a los clientes de Auth0 implementar soluciones como la verificación de identidad o la gestión del consentimiento sin necesidad de escribir código. Tú desarrollas la integración de Action; los clientes la arrastran y la sueltan en su flujo de trabajo.

Las integraciones de Actions son funciones autocontenidas que se ejecutan en puntos específicos de la plataforma Auth0. Escritas en JavaScript, son de código cerrado; los clientes no pueden modificarlo.

<div id="create-and-publish-an-actions-integration">
  ## Crear y publicar una integración de Action
</div>

Crear una integración de Action implica planificación, desarrollo y pruebas. Antes de ponerla a disposición de los clientes en el [Auth0 Marketplace](https://marketplace.auth0.com/), Auth0 revisará la integración de Action. Para obtener información sobre el Auth0 Marketplace, consulta [Marketplace Partners](/es/docs/customize/integrations/marketplace-partners).

<div id="step-1-determine-your-use-case">
  ### Paso 1. Determina tu caso de uso
</div>

Identifica qué caso de uso abordará tu integración de Action. ¿Qué problema resolverá para los clientes? ¿Cómo lo resolverá? Para obtener más información, lee [Definir un caso de uso de la integración](/es/docs/customize/integrations/marketplace-partners/defining-an-integration-use-case).

<div id="step-2-choose-a-flow">
  ### Paso 2. Elige un flujo
</div>

Las integraciones de Actions constan de una sola [Action](/es/docs/customize/actions). Lo que una Action puede hacer depende de dónde se ejecute dentro del entorno de ejecución de Auth0, es decir, de su [flujo](/es/docs/customize/actions/explore-triggers) de Action. Los flujos de Action disponibles son los siguientes:

* **Login flow** se ejecuta de forma sincrónica después de que un usuario se registra o inicia sesión correctamente. Los eventos de login incluyen flujos interactivos (como username/password o inicio de sesión federado) y flujos no interactivos (como comprobaciones de sesión e intercambios de <Tooltip tip="Refresh Token: Token utilizado para obtener un Access Token renovado sin obligar a los usuarios a iniciar sesión de nuevo." cta="Ver glosario" href="/es/docs/glossary?term=refresh+token">token de actualización</Tooltip>). Para obtener más información sobre este flujo, consulta [Login Flow](/es/docs/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger).
* **máquina a máquina flow** se ejecuta de forma sincrónica cuando una aplicación solicita un <Tooltip tip="Access Token: Credencial de autorización, en forma de una cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/es/docs/glossary?term=access+token">access token</Tooltip> mediante la Concesión de credenciales de cliente. Para obtener más información sobre este flujo, consulta [máquina a máquina Flow](/es/docs/customize/actions/explore-triggers/machine-to-machine-trigger).
* **Registro previo de usuario flow** se ejecuta de forma sincrónica antes de que Auth0 agregue un usuario a una conexión de base de datos o <Tooltip tip="Passwordless: Forma de authentication que no se basa en una contraseña como primer factor." cta="Ver glosario" href="/es/docs/glossary?term=passwordless">passwordless</Tooltip>. Este flujo no se ejecuta para inicios de sesión federados, como conexiones empresariales o sociales. Para obtener más información sobre este flujo, consulta [Registro previo de usuario Flow](/es/docs/customize/actions/explore-triggers/signup-and-login-triggers/pre-user-registration-trigger).
* **Post User Registration flow** se ejecuta de forma asincrónica después de que Auth0 agrega un usuario a una conexión de base de datos o passwordless. Este flujo no se ejecuta para inicios de sesión federados, como conexiones empresariales o sociales. Para obtener más información sobre este flujo, consulta [Post User Registration Flow](/es/docs/customize/actions/explore-triggers/signup-and-login-triggers/post-user-registration-trigger).
* **Post Change Password flow** se ejecuta de forma asincrónica después de que el cliente cambia la contraseña de un usuario de una conexión de base de datos. Este flujo no se ejecuta para cambios de contraseña en otros tipos de conexiones. Para obtener más información sobre este flujo, consulta [Post Change Password Flow](/es/docs/customize/actions/explore-triggers/password-reset-triggers/post-change-password-trigger).
* **Enviar mensaje telefónico** **flow** envía un mensaje telefónico o un SMS como parte de un proveedor personalizado de <Tooltip tip="Multi-factor authentication (MFA): Proceso de authentication del usuario que utiliza un factor además de username y contraseña, como un código por SMS." cta="Ver glosario" href="/es/docs/glossary?term=MFA">MFA</Tooltip>. Para obtener información sobre proveedores de MFA personalizados, consulta [Customize Multi-factor Authentication SMS and Voice Messages](/es/docs/customize/customize-sms-or-voice-messages). Para obtener más información sobre este flujo, consulta [Enviar mensaje telefónico Flow](/es/docs/customize/actions/explore-triggers/mfa-notifications-trigger).

Para obtener más información sobre los flujos de Action, consulta [Explore Flows and Triggers](/es/docs/customize/actions/explore-triggers). Si necesitas redirigir a los usuarios a un endpoint durante el login, consulta [Redirect Actions for Partners](/es/docs/customize/integrations/marketplace-partners/redirect-actions-for-partners).

<div id="step-3-build-an-actions-integration">
  ## Paso 3. Crea una integración de Actions
</div>

Una vez que hayas decidido qué flujo se ajusta al caso de uso de tu integración de Action, crea tu Action en un tenant de prueba. Para obtener instrucciones, consulta [Write Your First Action](/es/docs/customize/actions/write-your-first-action). Esta Action será la base del código de tu integración de Action.

Además, para ayudarte a crear tu integración, puedes usar nuestras plantillas de integración de Action:

* [Plantilla de integración de Action de Post Login](https://github.com/auth0-marketplace/TEMPLATE-action-post-login)
* [Plantilla de integración de Action de máquina a máquina](https://github.com/auth0-marketplace/TEMPLATE-action-client-credentials)
* [Plantilla de integración de Action de Registro previo de usuario](https://github.com/auth0-marketplace/TEMPLATE-action-pre-user-registration)
* [Plantilla de integración de Action de Post User Registration](https://github.com/auth0-marketplace/TEMPLATE-action-post-user-registration)
* [Plantilla de integración de Action de Post Change Password](https://github.com/auth0-marketplace/TEMPLATE-action-post-password-change)
* [Plantilla de integración de Action de Enviar mensaje telefónico](https://github.com/auth0-marketplace/TEMPLATE-action-phone-message)

Para aprender a hacer que tu integración de Actions sea segura y estable, consulta [Guías de codificación de Actions](/es/docs/customize/actions/action-coding-guidelines). Sigue también estas pautas específicas para las integraciones de Actions:

* Al crear tu Action en un tenant de prueba, usa secrets para todos los valores que un cliente deba proporcionar (como identificadores, nombres de tenant y umbrales). Cuando envías tu integración de Actions, Auth0 te pide que indiques qué campos deben estar cifrados en reposo (seguir siendo secrets) y cuáles pueden ser campos de texto editables.
* Usa un indicador de configuración para el modo de depuración/prueba del logging que ayude con el Setup o la resolución de problemas. Las integraciones de Actions rara vez deberían generar logs en todas las ejecuciones.
* Escribe los nombres de los secrets en `ALL_CAPS_SNAKE_CASE` para facilitar la revisión.
* Las integraciones de Actions son de código cerrado; los clientes no pueden ver ni modificar el código. Cualquier dato específico del cliente necesario para controlar la integración de Action debe provenir de secrets o de la configuración.

<div id="step-4-test-your-actions-integrations">
  ### Paso 4. Prueba tus integraciones de Actions
</div>

Como los revisores de Auth0 no tienen acceso a tu servicio, debes probar tu Action antes de enviarla para su revisión. Prueba manualmente tanto las ejecuciones correctas como los casos límite. Incluye estas comprobaciones:

* ¿La Action falla silenciosamente o bloquea la ejecución posterior?
* ¿Cómo funciona la Action si falla una petición HTTP?
* ¿Qué mensajes de error, si los hay, se muestran a los usuarios?
* ¿Qué información se registra?
* ¿Se puede eludir o falsificar la Action?

Durante el proceso de envío, Auth0 solicita cobertura de pruebas unitarias para el caso de éxito. Auth0 proporciona todas las herramientas y ejemplos necesarios para que esto sea lo más sencillo posible.

<div id="step-5-submit-your-actions-integration">
  ### Paso 5. Envía tu integración de Action
</div>

Una vez que hayas escrito y probado exhaustivamente tu integración de Action, envíala a Auth0 para su revisión: envía una [solicitud de servicio de Auth0 Marketplace](https://autheco.atlassian.net/servicedesk/customer/portal/1/group/1/create/43). Tras una revisión inicial satisfactoria, la integración se somete a una revisión de cumplimiento de seguridad.

<div id="step-6-publish-your-actions-integration">
  ### Paso 6. Publica tu integración de Action
</div>

Una vez que Auth0 complete las revisiones de tu integración de Action, te notificaremos si la integración las ha superado y te informaremos sobre los plazos de publicación.
