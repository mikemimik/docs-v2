Para configurar Cloudflare como proxy inverso con el enfoque recomendado, se requiere un plan Enterprise de Cloudflare con las siguientes funciones:

| Función               | Descripción                                                                                                                                                                                                                                                                                                               |
| --------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Host Header Override  | Reescribe los encabezados `Host` mediante distintas reglas de Cloudflare. Para obtener más información, consulta [Rewrite Host headers en la documentación de Cloudflare](https://developers.cloudflare.com/rules/origin-rules/tutorials/change-uri-path-and-host-header/).                                               |
| True-Client-IP Header | Al habilitar el encabezado `True-Client-IP`, este se añade a todas las solicitudes a tu servidor de origen e incluye la dirección IP del usuario final. Para obtener más información, consulta [Understanding the True-Client-IP Header en Cloudflare](https://developers.cloudflare.com/network/true-client-ip-header/). |

<div id="configure-cloudflare">
  ## Configurar Cloudflare
</div>

<Warning>
  Como requisito previo, el dominio principal del dominio personalizado que elijas debe estar [agregado y activado en el Dashboard de Cloudflare](https://developers.cloudflare.com/learning-paths/get-started/add-domain-to-cf/minimize-downtime/#activate-your-domain). Además, confirma que el dominio personalizado que quieres usar no exista ya en tu zona de Cloudflare. Si ya existe, la verificación de Cloudflare fallará.
</Warning>

Para configurar Cloudflare como proxy inverso, tendrás que crear un registro CNAME, una Page Rule y una Transform Rule en Cloudflare.

1. [Configura y verifica un dominio personalizado con Self-Managed Certificates](/es/docs/customize/custom-domains/self-managed-certificates) si aún no lo has hecho. Anota los valores de **Origin Domain Name** y **cname-api-key**, ya que los necesitarás más adelante.
2. En el Dashboard de Cloudflare de la zona de destino, [crea un registro CNAME](https://developers.cloudflare.com/dns/manage-dns-records/how-to/create-dns-records/#create-dns-records) con la siguiente configuración:

   | Setting      | Value                                                          |
   | ------------ | -------------------------------------------------------------- |
   | Name         | El nombre del dominio personalizado.                           |
   | Target       | El valor de **Origin Domain Name** que anotaste anteriormente. |
   | Proxy Status | `Proxied`                                                      |
3. [Crea una Page Rule](https://developers.cloudflare.com/rules/page-rules/) que se aplique a todas las URL del dominio personalizado elegido y con la siguiente configuración:

   | Setting              | Value                                                          |
   | -------------------- | -------------------------------------------------------------- |
   | Host Header Override | El valor de **Origin Domain Name** que anotaste anteriormente. |
   | True-Client-IP       | `Enable`                                                       |
4. [Crea una Transform Rule](https://developers.cloudflare.com/rules/transform/request-header-modification/create-dashboard/):

   <Callout icon="file-lines" color="#0EA5E9" iconType="regular">
     Aunque es posible usar Cloudflare Workers en lugar de Page Rules y Transform Rules para configurar un proxy inverso que cumpla los requisitos de un dominio personalizado con certificados autogestionados, recomendamos usar el enfoque basado en rules porque elimina la necesidad de código personalizado.
   </Callout>

   1. Cambia a la vista **Modify Request Header**.
   2. Selecciona **Create Rule** y asígnale el nombre que prefieras.
   3. En **When incoming requests match**, selecciona **Custom filter expression** y configura una expresión que limite la Rule a las solicitudes asociadas con el dominio personalizado elegido. Por ejemplo, usa una coincidencia exacta en el campo **Hostname**.
   4. En Modify request header, selecciona **Set static** y luego configura los siguientes campos:

      | Field       | Value                                                     |
      | ----------- | --------------------------------------------------------- |
      | Header name | `cname-api-key`                                           |
      | Value       | El valor de **cname-api-key** que anotaste anteriormente. |
5. Asegúrate de que [Always Use HTTPS](https://developers.cloudflare.com/ssl/edge-certificates/additional-options/always-use-https/) esté habilitado y de que el [modo de cifrado](https://developers.cloudflare.com/ssl/origin-configuration/ssl-modes/) esté configurado, como mínimo, en **Full** para el dominio personalizado elegido.

<div id="use-managed-challenges">
  ## Usa Managed Challenges
</div>

Los [Managed Challenges](https://developers.cloudflare.com/cloudflare-challenges/challenge-types/challenge-pages/#managed-challenges) de Cloudflare te permiten filtrar el tráfico de bots antes de que las solicitudes lleguen a Auth0 Universal Login. Cuando una solicitud coincide con tu rule, Cloudflare la intercepta y muestra un desafío de verificación. Como las páginas de desafío devuelven HTML, Managed Challenges solo son compatibles con flujos basados en navegador; aplicarlos a endpoints de API o a flujos sin navegador hará que esos flujos fallen, ya que el cliente recibe una página HTML de desafío en lugar de la respuesta esperada.

<div id="endpoints-to-exclude">
  ### Endpoints para excluir
</div>

No apliques un Managed Challenge a los siguientes endpoints. Los llaman servidores, SDKs o servidores de recursos y no pueden completar un desafío interactivo:

| Endpoint                            | Descripción                                                                     |
| ----------------------------------- | ------------------------------------------------------------------------------- |
| `/oauth/token`                      | endpoint de token                                                               |
| `/oauth/revoke`                     | endpoint de revocación de token                                                 |
| `/userinfo`                         | endpoint de UserInfo                                                            |
| `/.well-known/openid-configuration` | documento de descubrimiento de OIDC                                             |
| `/.well-known/jwks.json`            | JSON Web Key Set; obtenido por servidores de recursos para validar tokens       |
| `/api/v2/*`                         | Management API                                                                  |
| `/co/authenticate`                  | autenticación de origen cruzado                                                 |
| `/dbconnections/signup`             | conexiones de base de datos: registro                                           |
| `/dbconnections/change_password`    | conexiones de base de datos: cambiar la contraseña                              |
| `/usernamepassword/login`           | envío del formulario de Classic Universal Login                                 |
| `/mfa/challenge`                    | solicitud de desafío                                                            |
| `/mfa/associate`                    | asociación de autenticador                                                      |
| `/passwordless/start`               | Passwordless: solicitud de inicio                                               |
| `/samlp/*`                          | endpoints del protocolo SAML                                                    |
| `/wsfed/*`                          | endpoints de WS-Federation                                                      |
| `/v2/logout`                        | Puede llamarse del lado del servidor en flujos de cierre de sesión back-channel |

<div id="example-rule">
  ### Ejemplo de rule
</div>

Para aplicar Managed Challenges solo a los flujos de Universal Login en navegador, crea una [rule personalizada de WAF](https://developers.cloudflare.com/waf/custom-rules/) en Cloudflare. Configura la acción de la rule como **Managed Challenge** y usa la siguiente expresión; sustituye `YOUR_CUSTOM_DOMAIN` por tu dominio personalizado (por ejemplo, `login.example.com`):

```text
(http.host eq "YOUR_CUSTOM_DOMAIN" and (
  http.request.uri.path eq "/authorize" or
  starts_with(http.request.uri.path, "/u/") or
  http.request.uri.path eq "/login"
))
```

Esto limita el desafío solo a los endpoints de Universal Login con interfaz de usuario y evita interrupciones en el tráfico de API y de máquina a máquina.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Algunos casos de uso pueden comportarse de manera diferente:

  * **Persistencia de la cookie de validación**: Una vez que un navegador resuelve un Managed Challenge, Cloudflare emite una cookie de validación que normalmente se mantiene durante la sesión. Según tu configuración, limitar la rule solo a `/authorize` puede ser suficiente para cubrir todo el flujo de Universal Login sin aplicarla a cada ruta `/u/*`.
  * **Puntos de entrada que no usan OAuth**: Los flujos que comienzan desde puntos de entrada iniciados por SAML SP o WS-Federation usan `/samlp/*` o `/wsfed/*` en lugar de `/authorize`. Estas rutas están en la lista de exclusión y no se les debe aplicar un Managed Challenge.
</Callout>

<div id="configure-auth0">
  ## Configura Auth0
</div>

Llama al endpoint [Actualizar la configuración del dominio personalizado](https://auth0.com/docs/api/management/v2#!/Custom_Domains/patch_custom_domains_by_id) de la <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/es/docs/glossary?term=Management+API">Management API</Tooltip> de Auth0 con el siguiente payload en el cuerpo:

```json lines
{
  "custom_client_ip_header": "true-client-ip"
}
```
