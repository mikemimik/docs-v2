Esta guía te ayuda a migrar de una configuración con un solo dominio personalizado a [múltiples dominios personalizados](/docs/es/customize/custom-domains/multiple-custom-domains). Tanto si vas a añadir dominios para distintas marcas, regiones o segmentos de clientes, esta guía ofrece instrucciones paso a paso para que la transición sea fluida.

<div id="migration-scenarios">
  ## Escenarios de migración
</div>

Elige el escenario que mejor se adapte a tu situación:

| Escenario                            | Descripción                                                                                                                | Caso de uso                                                           | Complejidad  | Tiempo de inactividad                |
| ------------------------------------ | -------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------- | ------------ | ------------------------------------ |
| **Agregar dominios adicionales**     | Actualmente tienes un dominio personalizado y quieres añadir más sin dejar de usar el dominio actual                       | Expansión para dar servicio a varias marcas o regiones                | Baja         | Ninguno                              |
| **Reemplazar un dominio existente**  | Quieres sustituir tu dominio personalizado actual por uno nuevo                                                            | Cambio de marca o de titularidad del dominio                          | Media        | Mínimo (durante el cambio de DNS)    |
| **Migrar desde el dominio canónico** | Actualmente usas tu dominio canónico de Auth0 (por ejemplo, `tenant.auth0.com`) y quieres migrar a dominios personalizados | Implementación inicial de dominios personalizados con varios dominios | Media a alta | Ninguno (funcionamiento en paralelo) |

<div id="pre-migration-checklist">
  ## Lista de verificación previa a la migración
</div>

Antes de iniciar tu migración, asegúrate de haber completado lo siguiente:

* Haber verificado la titularidad del dominio de todos los nuevos dominios personalizados
* Haber revisado los procesos actuales de autenticación y las integraciones con la API
* Haber identificado todas las aplicaciones que usan el dominio actual
* Haber documentado las plantillas de correo electrónico actuales y sus enlaces
* Haber obtenido los certificados SSL/TLS (si usas certificados autogestionados)
* Haber probado la configuración del nuevo dominio personalizado en un entorno de desarrollo o de staging
* Haber preparado un plan de reversión
* Haber programado la migración durante un período de poco tráfico (si corresponde)
* Haber notificado a las partes interesadas y a los usuarios (si es necesario)

<div id="migration-steps">
  ## Pasos de migración
</div>

<div id="add-your-new-custom-domains">
  ### Añade tus nuevos dominios personalizados
</div>

Añade tus nuevos dominios personalizados con el [Auth0 Dashboard](/docs/es/customize/custom-domains/multiple-custom-domains#configure-multiple-custom-domains) o la [Management API](https://auth0.com/docs/api/management/v2).

<AuthCodeGroup>
  ```bash cURL
  curl --request POST \
    --url 'https://{yourDomain}/api/v2/custom-domains' \
    --header 'authorization: Bearer {yourMgmtApiAccessToken}' \
    --header 'content-type: application/json' \
    --data '{
      "domain": "new-domain.example.com",
      "type": "auth0_managed_certs",
      "domain_metadata": {
        "purpose": "new-brand"
      }
    }'
  ```

  ```javascript Node.js
  const newDomain = await management.customDomains.create({
    domain: 'new-domain.example.com',
    type: 'auth0_managed_certs',
    domain_metadata: {
      purpose: 'new-brand'
    }
  });

  console.log('Domain added:', newDomain.custom_domain_id);
  ```
</AuthCodeGroup>

<div id="verify-domain-ownership">
  ### Verificar la titularidad del dominio
</div>

Completa el proceso de verificación del dominio para cada nuevo dominio personalizado:

<div id="for-auth0-managed-certificates">
  #### Para certificados administrados por Auth0
</div>

1. Anota el registro CNAME proporcionado por Auth0
2. Añade el registro CNAME en tu proveedor de DNS
3. Verifica el dominio desde el Dashboard o la API

```bash
# Obtener detalles de verificación
curl --request GET \
  --url 'https://{yourDomain}/api/v2/custom-domains/{customDomainId}' \
  --header 'authorization: Bearer {yourMgmtApiAccessToken}'

# Verificar después de agregar el registro DNS
curl --request POST \
  --url 'https://{yourDomain}/api/v2/custom-domains/{customDomainId}/verify' \
  --header 'authorization: Bearer {yourMgmtApiAccessToken}'
```

<div id="for-self-managed-certificates">
  #### Para certificados autogestionados
</div>

1. Añade a tu DNS el registro TXT requerido
2. Configura tu proxy inverso o CDN
3. Sube tu certificado SSL
4. Verifica el dominio

<div id="configure-default-domain-optional">
  ### Configurar el dominio predeterminado (opcional)
</div>

Si quieres usar un mismo dominio de forma predeterminada para los correos electrónicos y las llamadas a la API, [establécelo como dominio predeterminado](/docs/es/customize/custom-domains/multiple-custom-domains/default-domain):

```bash
curl --request PATCH \
  --url 'https://{yourDomain}/api/v2/custom-domains/{customDomainId}' \
  --header 'authorization: Bearer {yourMgmtApiAccessToken}' \
  --header 'content-type: application/json' \
  --data '{
    "is_default": true
  }'
```

<div id="update-application-configurations">
  ### Actualiza la configuración de las aplicaciones
</div>

Actualiza tus aplicaciones para que usen el dominio personalizado adecuado:

<div id="sdk-configuration">
  #### Configuración del SDK
</div>

Actualiza la inicialización del SDK de Auth0:

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  El único cambio necesario es actualizar el parámetro `domain`: sustituye el dominio canónico de Auth0 por tu nuevo dominio personalizado.
</Callout>

<AuthCodeGroup>
  ```javascript Auth0 SPA SDK
  // Antes
  const auth0 = await createAuth0Client({
    domain: 'tenant.auth0.com',
    client_id: '{yourClientId}'
  });

  // Después
  const auth0 = await createAuth0Client({
    domain: 'new-domain.example.com', // Usa tu nuevo dominio personalizado
    client_id: '{yourClientId}'
  });
  ```

  ```javascript Auth0.js
  // Antes
  const webAuth = new auth0.WebAuth({
    domain: 'tenant.auth0.com',
    clientID: '{yourClientId}'
  });

  // Después
  const webAuth = new auth0.WebAuth({
    domain: 'new-domain.example.com',
    clientID: '{yourClientId}'
  });
  ```

  ```javascript Node.js
  import { ManagementClient } from "auth0";

  // Antes
  const auth0 = new ManagementClient({
    domain: 'tenant.auth0.com',
    clientId: '{yourClientId}',
    clientSecret: '{yourClientSecret}',
  });

  // Después
  const auth0 = new ManagementClient({
    domain: 'new-domain.example.com',
    clientId: '{yourClientId}',
    clientSecret: '{yourClientSecret}',
  });
  ```
</AuthCodeGroup>

<div id="callback-urls">
  #### URLs de devolución de llamada
</div>

Actualiza las URL de devolución de llamada de tu aplicación en el Auth0 Dashboard:

1. Ve a [**Auth0 Dashboard** &gt; **Applications**](https://manage.auth0.com/#/applications). Elige la aplicación que quieres configurar y selecciona la pestaña **Settings**.
2. Actualiza **Allowed Callback URLs** para incluir el nuevo dominio:
   ```
   https://new-domain.example.com/callback
   ```
3. Actualiza **Allowed Logout URLs**:
   ```
   https://new-domain.example.com/logout
   ```
4. Actualiza **Allowed Web Origins**:
   ```
   https://new-domain.example.com
   ```

<div id="update-email-templates">
  ### Actualizar las plantillas de correo electrónico
</div>

Para que la información del dominio personalizado esté disponible en tus plantillas de correo electrónico:

1. Ve a **Branding** &gt; **Custom Domains**
2. Establece como predeterminado el dominio que quieras
3. Personaliza tus plantillas de correo electrónico para usar la información del dominio personalizado en la dirección &quot;From&quot;, el asunto y el cuerpo, según sea necesario

<Note>
  Establecer un dominio como predeterminado no cambia automáticamente tus correos electrónicos. Lo que hace es poner el contexto del dominio predeterminado a disposición de las plantillas de correo electrónico. Debes personalizar tus plantillas para usar esa información. El contexto del dominio predeterminado estará disponible cuando no se proporcione un dominio específico mediante el encabezado `auth0-custom-domain`.
</Note>

<div id="update-social-identity-providers-idp">
  ### Actualizar proveedores de identidad social (IdP)
</div>

Actualiza las URI de redirección de tus proveedores de identidad social:

<div id="google">
  #### Google
</div>

1. Ve a [Google Cloud Console](https://console.cloud.google.com)
2. Ve a **APIs &amp; Services** &gt; **Credenciales**
3. Agrega `https://new-domain.example.com/login/callback` a **URI de redireccionamiento autorizados**

<div id="facebook">
  #### Facebook
</div>

1. Ve a [Facebook Developers](https://developers.facebook.com)
2. Ve a tu aplicación &gt; **Facebook Login** &gt; **Settings**
3. Añade `https://new-domain.example.com/login/callback` a **Valid OAuth Redirect URIs**

<div id="other-providers">
  #### Otros proveedores
</div>

Actualiza las URI de redirección de todos los proveedores sociales configurados según la documentación de cada proveedor.

<div id="update-enterprise-connections-if-applicable">
  ### Actualiza las conexiones empresariales (si aplica)
</div>

Si usas SAML, WS-Fed, Azure AD u otras conexiones empresariales, actualiza su configuración:

<div id="saml-connections">
  #### Conexiones SAML
</div>

Actualiza la URL del Servicio de Consumo de Aserciones (ACS):

```
https://new-domain.example.com/login/callback?connection={connectionName}
```

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  No necesita actualizar manualmente la URL de ACS en su IdP si utiliza solicitudes SAML iniciadas por el SP y su IdP acepta ACS dinámico.
</Callout>

<div id="azure-ad-connections">
  #### Conexiones de Azure AD
</div>

1. Ve a Azure Active Directory &gt; **App registrations**
2. Selecciona tu aplicación &gt; **Authentication**
3. Agrega `https://new-domain.example.com/login/callback` a **Redirect URIs**

<div id="adfs-connections">
  #### Conexiones de ADFS
</div>

Actualiza el endpoint en la configuración de ADFS para usar el nuevo dominio personalizado.

<div id="test-authentication">
  ### Probar la autenticación
</div>

Antes de completar la migración, prueba exhaustivamente lo siguiente:

1. **Inicio de sesión de usuario**: Verifica que los usuarios puedan autenticarse a través del nuevo dominio
2. **Restablecimiento de contraseña**: Comprueba que los correos electrónicos de restablecimiento de contraseña usen el dominio correcto
3. **Verificación del correo electrónico**: Verifica que los enlaces de verificación del correo electrónico funcionen
4. **Inicios de sesión sociales**: Prueba cada proveedor social configurado
5. **Llamadas a la API**: Verifica que las llamadas a la API funcionen con el nuevo dominio
6. **Validación de token**: Asegúrate de que los JWT tengan el claim `iss` correcto

<div id="monitor-and-verify">
  ### Supervisa y verifica
</div>

Después de la migración:

1. Supervisa los registros de autenticación para detectar cualquier error
2. Comprueba la entrega de correos electrónicos y el funcionamiento de los enlaces
3. Verifica la validez del certificado SSL y las fechas de vencimiento
4. Haz pruebas desde distintas regiones geográficas (si corresponde)
5. Confirma que todas las aplicaciones estén usando los dominios personalizados previstos

<div id="decommission-old-domain-if-applicable">
  ### Retira el dominio antiguo (si corresponde)
</div>

Si vas a reemplazar un dominio personalizado existente:

1. Asegúrate de que todas las aplicaciones se hayan migrado al nuevo dominio
2. Supervisa el tráfico del dominio antiguo para confirmar que ya no se utiliza
3. Considera mantener activo el dominio antiguo durante un período de gracia
4. Elimina el dominio personalizado antiguo cuando todo esté listo:

```bash
curl --request DELETE \
  --url 'https://{yourDomain}/api/v2/custom-domains/{oldCustomDomainId}' \
  --header 'authorization: Bearer {yourMgmtApiAccessToken}'
```

<div id="migration-patterns">
  ## Patrones de migración
</div>

<AccordionGroup>
  <Accordion title="Operación en paralelo (sin tiempo de inactividad)">
    Ejecute los dominios personalizados antiguo y nuevo simultáneamente:

    1. Agregue un nuevo dominio personalizado
    2. Actualice las nuevas aplicaciones para que usen el nuevo dominio
    3. Mantenga las aplicaciones existentes en el dominio antiguo
    4. Migre gradualmente las aplicaciones
    5. Retire el dominio antiguo cuando la migración se haya completado por completo

    **Ventajas**: Sin tiempo de inactividad, implementación gradual, reversión sencilla

    **Desventajas**: Sobrecarga temporal de gestión
  </Accordion>

  <Accordion title="Migración por fases por aplicación">
    Migre las aplicaciones una por una:

    1. Identifique las aplicaciones por prioridad o riesgo
    2. Migre primero las aplicaciones de bajo riesgo
    3. Supervise la aparición de problemas antes de continuar
    4. Migre las aplicaciones restantes
    5. Limpie las configuraciones antiguas

    **Ventajas**: Implementación controlada, detección temprana de problemas

    **Desventajas**: Plazo de migración más largo
  </Accordion>

  <Accordion title="Despliegue blue-green">
    Use entornos separados para las pruebas:

    1. Configure el nuevo dominio personalizado en el entorno de staging
    2. Pruebe toda la funcionalidad exhaustivamente
    3. Haga el cambio en producción en una sola operación
    4. Mantenga el dominio antiguo como respaldo
    5. Retírelo después del período de verificación

    **Ventajas**: Pruebas exhaustivas, reversión rápida

    **Desventajas**: Requiere un entorno separado
  </Accordion>
</AccordionGroup>

<div id="handling-existing-user-sessions">
  ## Gestión de las sesiones de usuario existentes
</div>

Al migrar dominios personalizados, las sesiones de usuario existentes pueden verse afectadas:

<div id="session-considerations">
  ### Consideraciones sobre las sesiones
</div>

* Las sesiones creadas con el dominio anterior seguirán siendo válidas hasta que caduquen
* Los nuevos inicios de sesión crearán sesiones con el nuevo dominio
* Las sesiones entre dominios requieren una planificación minuciosa

<div id="recommended-approach">
  ### Enfoque recomendado
</div>

1. **Notificar a los usuarios**: Informe a los usuarios de que quizá deban iniciar sesión de nuevo
2. **Período de gracia**: Mantenga activo el dominio antiguo durante la transición
3. **Transferencia de sesión**: Use Universal Login para gestionar la migración de la sesión
4. **Instrucciones claras**: Proporcione instrucciones claras si los usuarios tienen problemas

<div id="rollback-plan">
  ## Plan de reversión
</div>

Si surge algún problema durante la migración:

<div id="immediate-rollback">
  ### Reversión inmediata
</div>

1. Revertir la configuración de la aplicación para usar el dominio anterior
2. Mantener el nuevo dominio personalizado configurado para futuros intentos
3. Documentar el problema para solución de problemas

<div id="partial-rollback">
  ### Reversión parcial
</div>

1. Identifica las aplicaciones afectadas
2. Revierte solo esas aplicaciones al dominio anterior
3. Investiga y corrige los problemas
4. Vuelve a migrar cuando estés listo

<div id="complete-rollback">
  ### Reversión completa
</div>

1. Actualiza todas las configuraciones de las aplicaciones para que usen el dominio anterior
2. Si configuraste un dominio predeterminado, vuelve a cambiarlo al dominio anterior
3. Notifica a los usuarios cualquier acción necesaria
4. Programa otro intento de migración

<div id="troubleshooting">
  ## Solución de problemas
</div>

<div id="common-issues-and-solutions">
  ### Problemas comunes y soluciones
</div>

| Problema                                                     | Causa                                             | Solución                                                                                                      |
| ------------------------------------------------------------ | ------------------------------------------------- | ------------------------------------------------------------------------------------------------------------- |
| La verificación del dominio falla                            | Los registros DNS no se han propagado             | Espera a que se propaguen los DNS (puede tardar hasta 48 horas) y verifica que el registro CNAME sea correcto |
| El inicio de sesión redirige al dominio anterior             | La configuración de la aplicación no se actualizó | Actualiza la inicialización del SDK y las URL de devolución de llamada                                        |
| Los enlaces de correo electrónico usan el dominio equivocado | El dominio predeterminado no está configurado     | Configura el dominio predeterminado o el enrutamiento del correo electrónico                                  |
| El inicio de sesión social falla                             | Las URI de redirección no se actualizaron         | Agrega el nuevo dominio personalizado a las URI de redirección permitidas del proveedor social                |
| Emisor de token no válido                                    | La validación de JWT espera el dominio anterior   | Actualiza la validación del token para aceptar el nuevo dominio como emisor                                   |
| Errores de aserción SAML                                     | La URL de ACS no se actualizó                     | Actualiza la configuración de SAML con la nueva URL del dominio personalizado                                 |
| Errores de certificado                                       | El certificado no se aprovisionó o venció         | Verifica el dominio, espera a que se aprovisione el certificado o actualízalo                                 |

<div id="getting-help">
  ### Obtener ayuda
</div>

Si tienes problemas durante la migración:

1. Consulta [Auth0 Community](https://community.auth0.com/) para ver si hay problemas similares
2. Revisa la [documentación de solución de problemas](https://support.auth0.com/center/s/knowledge?selectedTopics=Custom%20Domains\&isTopicFilter=true)
3. Ponte en contacto con el [equipo de soporte de Auth0](https://support.auth0.com/) e incluye:
   * El nombre de tu tenant
   * Los ID de dominios personalizados
   * Una descripción detallada del problema
   * Los pasos para reproducirlo
   * Mensajes de error o logs

<div id="post-migration-best-practices">
  ## Prácticas recomendadas después de la migración
</div>

Después de completar la migración:

1. **Documenta la configuración**: Registra qué aplicaciones usan qué dominios personalizados
2. **Supervisa el vencimiento de los certificados**: Configura alertas para la renovación de certificados
3. **Revísalo periódicamente**: Asegúrate de que la configuración de los dominios personalizados se ajuste a las necesidades del negocio
4. **Actualiza los runbooks**: Actualiza la documentación operativa con la nueva información sobre los dominios personalizados
5. **Capacita a los miembros del equipo**: Asegúrate de que tu equipo entienda la nueva configuración de varios dominios
6. **Planifica el escalado**: Considera cómo gestionarás dominios adicionales en el futuro
