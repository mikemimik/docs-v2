Migra tu SAML Federation Connection actual de un dominio personalizado heredado de Auth0 a uno nuevo, para que tus tenants puedan aprovechar las capacidades de Multiple Custom Domain (MCD).

Los proveedores de identidad externos (IdPs) suelen estar codificados de forma fija con tu dominio anterior, por lo que necesitas implementar un proxy inverso inteligente que intercepte la respuesta SAML enviada a tu dominio anterior y la redirija de forma segura a un nuevo dominio personalizado.

Esto garantiza una migración sin interrupciones para los usuarios federados, permitiéndote usar de inmediato tu nuevo dominio sin tener que esperar actualizaciones de configuración obligatorias de tu IdP. 

<div id="prerequisites">
  ## Requisitos previos
</div>

Este proceso requiere configurar mediante Terraform los componentes de Auth0 y desplegar un Cloudflare Worker para la lógica de proxy.

Antes de comenzar tu migración, revisa los siguientes requisitos:

* Un tenant con un plan Enterprise y MCD habilitado.
* Dos dominios personalizados verificados configurados en tu tenant de Auth0:
  1. **Dominio heredado antiguo** (para el proxy)
  2. **Nuevo dominio de destino** (para la aplicación)
* Terraform CLI y `Node.js/npm` instalados.
* Una cuenta de Cloudflare con acceso al dominio que aloja tus dominios personalizados.

<div id="how-it-works">
  ## Cómo funciona
</div>

Esta estrategia de migración utiliza un proxy inverso inteligente para cerrar la brecha entre tu dominio personalizado heredado y tu nuevo dominio. Este proxy se despliega en el dominio antiguo para interceptar la respuesta de autenticación SAML enviada por tu proveedor de identidad (IdP) externo.

Esto es necesario porque la configuración del IdP está codificada de forma rígida en el endpoint de tu dominio antiguo. El proxy modifica los campos de control del payload de SAML (como `Destination` y `Recipient`) para que reflejen con precisión el nuevo dominio personalizado.

Por último, el proxy reenvía este payload corregido al endpoint de inicio de sesión del nuevo dominio. Esto permite una transición sin tiempo de inactividad a tu nuevo dominio, sin requerir cambios manuales de configuración por parte de tus partners de IdP.

<div id="setup-and-configuration">
  ## Setup y configuración
</div>

Para configurar tu migración:

1. Clona el repositorio de migración:

```bash
git clone https://github.com/abbaspour/auth0-mcd-federation-migration.git
cd auth0-mcd-federation-migration
```

2. Instala las dependencias:

```bash
npm install
```

3. Crea un archivo `terraform.auto.tfvars` en tu directorio `tf` con las credenciales y la información del dominio requeridas:

```bash
# Variables de Auth0 Service Provider (SP)
auth0_domain = "your-sp-tenant.auth0.com"
auth0_existing_custom_domain = "oldfed.example.com"
auth0_new_custom_domain = "id.example2.com"
auth0_tf_client_id = "your-sp-client-id"
auth0_tf_client_secret = "your-sp-client-secret"

# Variables de Auth0 Identity Provider (IDP)
auth0_idp_domain = "your-idp-tenant.auth0.com"
auth0_idp_tf_client_id = "your-idp-client-id"
auth0_idp_tf_client_secret = "your-idp-client-secret"

# Variables de Cloudflare
cloudflare_api_key = "your-cloudflare-api-key"
cloudflare_email = "your-cloudflare-email"
cloudflare_zone_id = "your-cloudflare-zone-id"
```

4. Inicializa y aplica Terraform:

```bash
cd tf
terraform init
terraform apply
```

Esto crea la aplicación SAML y la conexión necesarias, configura el DNS con Cloudflare y prepara las variables de entorno para el worker.

<div id="deploy-the-cloudflare-worker">
  ### Despliega el Cloudflare Worker
</div>

Este proxy se encarga de interceptar la respuesta SAML y de la lógica de redirección. Para desplegarlo:

1. Despliega el proxy de Cloudflare:

```bash
cd ..
npx wrangler deploy
```

2. El worker recibe automáticamente las variables de entorno necesarias (como `AUTH0_EDGE_LOCATION` y `NEW_SP_DOMAIN`) a partir de la salida de Terraform.

<div id="temporarily-update-saml-connection-parameters">
  ### Actualizar temporalmente los parámetros de la conexión SAML
</div>

Tu dominio antiguo recibe la respuesta SAML antes de volver a enviarla al dominio nuevo. Por lo tanto, los parámetros de validación esperados de la conexión SAML deben apuntar temporalmente a la URL de devolución de llamada del dominio antiguo para evitar errores de discrepancia.

1. Obtén un token de acceso de la Management API para el tenant de tu proveedor de servicios, con los scopes `read:connections` y `update:connections`:

   ```bash
   cd bin/
   export access_token='<sp-tenant-management-api-token>'
   ```

2. Actualiza la URL de destino:

   ```bash
   ./sp-set-destination-url.sh -i <saml-connection-id> -d https://oldfed.example.com/login/callback
   ```

3. Actualiza la URL del destinatario:

   ```bash
   ./sp-set-recipient-url.sh -i <saml-connection-id> -r https://oldfed.example.com/login/callback
   ```
