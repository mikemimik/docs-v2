Al usar [múltiples dominios personalizados](/es/docs/customize/custom-domains/multiple-custom-domains), debes configurar tus SDK de Auth0 para que utilicen el dominio personalizado adecuado para la autenticación. En esta guía se explica cómo configurar los SDK para distintas plataformas y escenarios.

<div id="key-concepts">
  ## Conceptos clave
</div>

<div id="domain-parameter">
  ### Parámetro `domain`
</div>

Todos los SDK de Auth0 requieren un parámetro `domain` que indica qué dominio de Auth0 se debe usar para la autenticación. Si usas dominios personalizados, configura este parámetro con tu dominio personalizado en lugar del dominio canónico de Auth0.

**Sin dominio personalizado:**

```
domain: 'tenant.auth0.com'
```

**Con dominio personalizado:**

```
domain: 'login.example.com'
```

<div id="token-issuer">
  ### Emisor del token
</div>

Al usar un dominio personalizado, el claim `iss` (emisor) de los tokens se establecerá en tu dominio personalizado:

```json
{
  "iss": "https://login.example.com/",
  "sub": "auth0|123456",
  "aud": "your-client-id"
}
```

Debes configurar la validación de tus tokens para que acepte tu dominio personalizado como emisor válido.

<div id="authentication-sdks">
  ## SDKs de autenticación
</div>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Al usar MCD, el Cliente es responsable de proporcionar y validar todos los dominios personalizados. Al configurar los SDKs para resolver dominios personalizados del tenant mediante las funciones de resolución de dominios, usted es responsable de garantizar que todos los dominios resueltos sean de confianza. Una configuración incorrecta del resolvedor de dominios puede provocar una elusión de la autenticación en la parte usuaria o exponer la aplicación a ataques de falsificación de solicitudes del lado del servidor. No configurar correctamente sus dominios y servidores proxy puede generar vulnerabilidades de seguridad críticas por las que Okta no asume responsabilidad.
</Callout>

  ```javascript Configuración dinámica
  function getAuth0Domain() {
    const hostname = window.location.hostname;

    // Asigna los dominios de las aplicaciones a dominios personalizados de Auth0

      return 'login.brand1.com';
    } else if (hostname.includes('app2.example.com')) {
      return 'login.brand2.com';
    }

    // Dominio personalizado predeterminado
    return 'login.example.com';

  const auth0 = await createAuth0Client({
    domain: getAuth0Domain(),
    clientId: 'YOUR_CLIENT_ID',
    authorizationParams: {
      redirect_uri: window.location.origin
    }
  });
  ```

  ```javascript Variables de entorno
  import { createAuth0Client } from '@auth0/auth0-spa-js';

      redirect_uri: window.location.origin
  ```

  ```typescript Configuración dinámica (multimarca)
      // Asigna hosts de aplicaciones a dominios personalizados de Auth0
      const host = headers.get('host') || '';

      if (host.includes('brand1.example.com')) {
        return 'login.brand1.com';
      } else if (host.includes('brand2.example.com')) {
        return 'login.brand2.com';
      }

      return 'login.example.com';  // Dominio predeterminado
    },
    clientId: process.env.AUTH0_CLIENT_ID,
    clientSecret: process.env.AUTH0_CLIENT_SECRET,
    appBaseUrl: process.env.APP_BASE_URL,
    secret: process.env.AUTH0_SECRET
  });
  ```

  ```typescript Configuración dinámica (SaaS B2B)
  import { Auth0Client } from '@auth0/nextjs-auth0/server';

  // Asigna identificadores de tenant a dominios personalizados (desde base de datos, caché, etc.)
  async function resolveTenantDomain(host: string): Promise<string> {
    // Extrae el identificador del tenant del subdominio
    const subdomain = host.split('.')[0];

    // Busca el dominio personalizado para este tenant (por ejemplo, en la base de datos)
    const tenantConfig = await getTenantConfig(subdomain);

    return tenantConfig?.customDomain || 'login.example.com';
  }

  const auth0 = new Auth0Client({
    domain: async ({ headers }) => {
      const host = headers.get('host') || '';
      return resolveTenantDomain(host);
    },
    clientId: process.env.AUTH0_CLIENT_ID,
    clientSecret: process.env.AUTH0_CLIENT_SECRET,
    appBaseUrl: process.env.APP_BASE_URL,
    secret: process.env.AUTH0_SECRET
  });
  ```

  ```bash Variables de entorno
  # .env.local

  # Configuración de Auth0 (un solo tenant con múltiples dominios personalizados)
  AUTH0_DOMAIN=login.example.com           # Dominio personalizado predeterminado o de respaldo
  AUTH0_CLIENT_ID=your_client_id
  AUTH0_CLIENT_SECRET=your_client_secret
  APP_BASE_URL=http://localhost:3000
  AUTH0_SECRET=your-long-random-secret
  ```
</AuthCodeGroup>

**Conceptos clave de MCD con Next.js:**

* **Un solo tenant de Auth0, múltiples dominios**: Todos los dominios personalizados comparten el mismo `clientId` y `clientSecret` porque pertenecen al mismo tenant de Auth0.
* **Función `DomainResolver`**: El parámetro `domain` acepta una función `(config: { headers: Headers; url?: URL }) => Promise<string> | string`. Esto permite resolver el dominio dinámicamente en cada solicitud según los headers de la solicitud entrante.
* **Caché de instancias**: El SDK almacena automáticamente en caché las instancias de `Auth0Client` por dominio mediante una caché LRU acotada (máximo 100 entradas) para mejorar el rendimiento.
* **Aislamiento de sesiones**: Las sesiones creadas mediante un dominio personalizado quedan aisladas en ese dominio y no pueden usarse indistintamente con sesiones de otro dominio.
* **Parámetro de URL**: El parámetro `url` en la función de resolución es `undefined` en Server Components y Server Actions; solo está disponible en middleware o rutas de API.
* **Ajuste de la caché de discovery**: Configura el almacenamiento en caché de metadatos de OIDC con la opción `discoveryCache`:
  ```typescript
  const auth0 = new Auth0Client({
    // ... otra configuración
    discoveryCache: {
      ttl: 600,      // Caché durante 10 minutos (predeterminado)
      maxEntries: 100  // Número máximo de issuers en caché (predeterminado: 100, expulsión LRU)
    }
  });
  ```

```javascript
app.use((req, res, next) => {
```

<div id="nodejs-management-sdk">

</div>

```javascript
import { ManagementClient, CustomDomainHeader } from "auth0";

// Dominio personalizado global (enviado en todas las solicitudes permitidas)
const management = new ManagementClient({
  domain: 'tenant.auth0.com',
  clientId: 'YOUR_M2M_CLIENT_ID',
  clientSecret: 'YOUR_M2M_CLIENT_SECRET',
  withCustomDomainHeader: 'login.example.com',

// Listar usuarios (endpoint permitido - el encabezado se envía automáticamente)
const users = await management.users.getAll();

// Anulación por solicitud (tiene precedencia sobre la configuración global)

  ...CustomDomainHeader("specific-user-request.exampleco.com"),
};
const response = await management.users.getAll({}, reqOptions);
```

</div>

Cuando uses dominios personalizados, actualiza la validación de tokens para que acepte el dominio personalizado como emisor.

<div id="nodejs-express">
  ### Node.js (Express)
</div>

Con [express-jwt](https://github.com/auth0/express-jwt) o [jose](https://github.com/panva/jose):

```javascript
app.use(

    secret: expressJwtSecret({
      cache: true,
      rateLimit: true,
      jwksUri: 'https://login.example.com/.well-known/jwks.json'  // Custom domain
    }),

    issuer: 'https://login.example.com/',  // Custom domain como emisor

```

Para varios dominios personalizados:

```javascript
const validIssuers = [
  'https://login.brand1.com/',
  'https://login.brand2.com/',
  'https://login.example.com/'
];

app.use(
  expressjwt({
    secret: expressJwtSecret({
      cache: true,
      rateLimit: true,
      jwksRequestsPerMinute: 5,
      jwksUri: (req) => {
        // Extraer el emisor del token para determinar el URI de JWKS
        const token = req.headers.authorization?.split(' ')[1];
        if (token) {
          const payload = JSON.parse(Buffer.from(token.split('.')[1], 'base64').toString());
          return `${payload.iss}.well-known/jwks.json`;
        }
        return 'https://login.example.com/.well-known/jwks.json';
      }
    }),
    audience: 'YOUR_API_IDENTIFIER',
    issuer: validIssuers,  // Aceptar múltiples emisores
    algorithms: ['RS256']
  })
);
```

<div id="python-flask">
  ### Python (Flask)
</div>

Con [python-jose](https://github.com/mpdavis/python-jose):

```python
from jose import jwt
from functools import wraps
from flask import request, jsonify

def get_token_auth_header():
    auth = request.headers.get('Authorization', None)
    if not auth:
        raise Exception('Authorization header is expected')

    parts = auth.split()
    if parts[0].lower() != 'bearer':
        raise Exception('Authorization header must start with Bearer')
    elif len(parts) == 1:
        raise Exception('Token not found')
    elif len(parts) > 2:
        raise Exception('Authorization header must be Bearer token')

    return parts[1]

def requires_auth(f):
    @wraps(f)
    def decorated(*args, **kwargs):
        token = get_token_auth_header()

        # Compatibilidad con múltiples dominios personalizados
        valid_issuers = [
            'https://login.brand1.com/',
            'https://login.brand2.com/',
            'https://login.example.com/'
        ]

        try:
            # Obtener JWKS del dominio personalizado
            unverified = jwt.get_unverified_header(token)
            issuer = jwt.get_unverified_claims(token)['iss']

            if issuer not in valid_issuers:
                raise Exception('Invalid issuer')

            jwks_uri = f"{issuer}.well-known/jwks.json"
            jwks = requests.get(jwks_uri).json()

            payload = jwt.decode(
                token,
                jwks,
                algorithms=['RS256'],
                audience='YOUR_API_IDENTIFIER',
                issuer=valid_issuers
            )
        except Exception as e:
            return jsonify({'error': str(e)}), 401

        return f(*args, **kwargs)

    return decorated
```

<div id="java-spring-boot">
  ### Java (Spring Boot)
</div>

Con Spring Security:

```java
@Configuration
@EnableWebSecurity
public class SecurityConfig extends WebSecurityConfigurerAdapter {

    @Value("${auth0.audience}")
    private String audience;

    @Value("${auth0.custom-domain}")
    private String customDomain;

    @Override
    protected void configure(HttpSecurity http) throws Exception {
        http
            .authorizeRequests()
            .mvcMatchers("/api/public").permitAll()
            .mvcMatchers("/api/private").authenticated()
            .and()
            .oauth2ResourceServer()
            .jwt()
            .decoder(jwtDecoder());
    }

    @Bean
    JwtDecoder jwtDecoder() {
        String issuerUri = "https://" + customDomain + "/";

        NimbusJwtDecoder jwtDecoder = JwtDecoders.fromIssuerLocation(issuerUri);

        // Validar el audience
        OAuth2TokenValidator<Jwt> audienceValidator = new AudienceValidator(audience);
        OAuth2TokenValidator<Jwt> withIssuer = JwtValidators.createDefaultWithIssuer(issuerUri);
        OAuth2TokenValidator<Jwt> withAudience = new DelegatingOAuth2TokenValidator<>(withIssuer, audienceValidator);

        jwtDecoder.setJwtValidator(withAudience);

        return jwtDecoder;
    }
}
```

      <td>La validación de tokens espera el dominio canónico, pero recibe un dominio personalizado</td>
      <td>Actualiza la validación de tokens para aceptar el dominio personalizado como emisor</td>
    </tr>

    <tr>
      <td>Error al obtener el JWKS</td>
      <td>El URI de JWKS apunta al dominio canónico</td>

    </tr>

      <td>La URL de devolución de llamada no coincide con los Redirect URIs configurados</td>

<div id="best-practices">
  ## Prácticas recomendadas
</div>

1. **Use variables de entorno**: Almacene los dominios personalizados en archivos de configuración específicos del entorno
2. **Valide múltiples emisores**: Si usa múltiples dominios personalizados, configure la validación de tokens para aceptar todos como emisores válidos
3. **Actualice las URL de devolución de llamada**: Asegúrese de agregar todos los dominios personalizados a Allowed Callback URLs en la configuración de la aplicación
4. **Pruebe exhaustivamente**: Pruebe la autenticación a través de cada dominio personalizado antes de pasar a Production
5. **Supervise los emisores de tokens**: Registre y supervise el claim `iss` en los tokens para garantizar el uso correcto del dominio personalizado
6. **Documente los mapeos de dominios**: Mantenga una documentación clara de qué aplicaciones usan qué dominios personalizados
7. **Gestione los fallos correctamente**: Implemente un manejo adecuado de errores para los fallos de autenticación
8. **Almacene JWKS en caché**: Almacene en caché los datos de JWKS para mejorar el rendimiento y reducir las solicitudes
