---

Implementar [múltiples dominios personalizados](/es/docs/customize/custom-domains/multiple-custom-domains) (MCD) de forma eficaz requiere una planificación cuidadosa y seguir las prácticas recomendadas. Si sigues estas pautas, puedes garantizar la escalabilidad, mantener una seguridad sólida y ofrecer una experiencia de marca coherente en todos tus dominios personalizados.

Esta guía abarca prácticas recomendadas de planificación, seguridad, rendimiento, operaciones y experiencia del usuario al usar múltiples dominios personalizados.

<div id="verify-ownership-of-domains">
  ## Verifique la titularidad de los dominios
</div>

Asegúrese de estar listo para verificar a tiempo la titularidad de sus dominios personalizados. Dejar dominios sin verificar durante demasiado tiempo puede llenar su espacio y complicar la gestión.

<div id="planning-and-architecture">
  ## Planificación y arquitectura
</div>

<div id="choose-a-domain-strategy">
  ### Elige una estrategia de dominio
</div>

Antes de implementar MCD, define tu estrategia de dominio:

* **Por marca**: Dominios personalizados independientes para cada marca (p. ej., `login.brand1.com`, `login.brand2.com`)
* **Por región**: Dominios regionales por motivos de cumplimiento o rendimiento (p. ej., `login.us.example.com`, `login.eu.example.com`)
* **Por cliente**: Dominios dedicados para clientes empresariales (p. ej., `login.customer1.com`, `login.customer2.com`)
* **Híbrida**: Combinación de estrategias (p. ej., marca + región: `login-us.brand1.com`)

<div id="configure-a-default-domain">
  ### Configura un dominio predeterminado
</div>

Configura siempre un [dominio personalizado predeterminado](/es/docs/customize/custom-domains/multiple-custom-domains/default-domain) para:

* Simplificar la configuración de las aplicaciones que no requieren un comportamiento específico según el dominio
* Servir de respaldo para las notificaciones por correo electrónico
* Reducir la necesidad de especificar explícitamente el dominio en las llamadas a la API

**Práctica recomendada**: Usa un dominio genérico o administrativo como predeterminado (por ejemplo, `login.company.com` o `admin.company.com`) en lugar de uno específico de la marca.

<div id="plan-for-scale">
  ### Planifica para crecer
</div>

Ten en cuenta el crecimiento futuro al diseñar tu arquitectura MCD:

* **Documenta las asignaciones de dominios**: Mantén una documentación clara sobre qué aplicaciones usan qué dominios
* **Reserva patrones de dominios**: Registra los dominios que puedas necesitar en el futuro
* **Supervisa los límites**: Lleva un control de la cantidad de dominios en relación con los límites permitidos
* **Planifica la expansión**: Diseña tu arquitectura para dar cabida a dominios adicionales

<div id="use-metadata-to-stay-organized">
  ## Usa metadatos para mantener todo organizado
</div>

Aprovecha los campos de metadatos disponibles para cada dominio personalizado para facilitar una gestión y personalización eficientes:

**Campos de metadatos recomendados**:

* `brand`: Identificador de marca (p. ej., &quot;BrandA&quot;, &quot;BrandB&quot;)
* `region`: Región geográfica (p. ej., &quot;us-east&quot;, &quot;eu-west&quot;)
* `environment`: Tipo de entorno (p. ej., &quot;production&quot;, &quot;staging&quot;)
* `customer_id`: Identificador del cliente o del tenant
* `support_email`: Contacto de soporte específico del dominio
* `purpose`: Finalidad del dominio (p. ej., &quot;customer-portal&quot;, &quot;admin-portal&quot;)

**Ejemplo de estructura de metadatos**:

```json
{
  "brand": "BrandA",
  "region": "us-east",
  "environment": "production",
  "customer_id": "cust_12345",
  "support_email": "support@brand-a.com",
  "tier": "enterprise"
}
```

Usa metadatos para:

* **Personalización del correo electrónico**: Personaliza las plantillas de correo electrónico en función de los metadatos
* **Lógica de Actions**: Implementa reglas de autenticación específicas del dominio
* **Filtrado y búsqueda**: Organiza los dominios en tus herramientas de gestión
* **Informes**: Supervisa el uso y el rendimiento por marca, región o cliente

<div id="security">
  ## Seguridad
</div>

Al implementar múltiples dominios personalizados, ten en cuenta estos patrones de seguridad según tu caso de uso:

* **Validación de dominio**: Evita el uso no autorizado de dominios restringiendo la autenticación a dominios personalizados concretos
* **Aislamiento de organizaciones**: Asegúrate de que los usuarios solo puedan acceder a las organizaciones a través de dominios aprobados
* **Gestión de certificados**: Mantén una gestión segura del ciclo de vida de los certificados

<div id="domain-validation-in-actions">
  ### Validación de dominios en Actions
</div>

Usa [Actions](/es/docs/customize/actions) para restringir la autenticación a dominios personalizados específicos. Esto resulta útil cuando quieres impedir que los usuarios se autentiquen a través del dominio canónico de Auth0 o restringir el acceso a dominios de marca específicos:

```javascript
exports.onExecutePostLogin = async (event, api) => {
  const domain = event.custom_domain?.domain;

  // Lista de dominios personalizados permitidos
  const allowedDomains = [
    'login.brand1.com',
    'login.brand2.com',
    'login.example.com'
  ];

  if (!domain || !allowedDomains.includes(domain)) {
    return api.access.deny(
      `Authentication from ${domain} is not permitted.`
    );
  }
};
```

<div id="organization-based-access-control">
  ### Control de acceso basado en organizaciones
</div>

Si usas [Auth0 Organizations](/es/docs/manage-users/organizations), puedes restringir qué dominios personalizados se permiten para cada organización. Esto resulta útil en escenarios B2B en los que cada cliente tiene su propio dominio de marca y solo debería acceder a su organización a través de ese dominio:

```javascript
exports.onExecutePostLogin = async (event, api) => {
  const domain = event.custom_domain?.domain;

  if (!event.organization) {
    return;
  }

  // Verificar los dominios permitidos de la organización
  const allowedDomains = event.organization.metadata?.allowed_domains?.split(',') || [];

  if (allowedDomains.length > 0 && !allowedDomains.includes(domain)) {
    return api.access.deny(
      `Access to ${event.organization.name} requires authentication via an approved domain.`
    );
  }
};
```

<div id="certificate-management">
  ### Gestión de certificados
</div>

* **Supervisar el vencimiento**: Configura alertas para las fechas de vencimiento de los certificados
* **Automatizar la renovación**: Usa certificados gestionados por Auth0 para renovarlos automáticamente
* **Documentar los procesos autogestionados**: Si usas certificados autogestionados, mantén procedimientos operativos claros

<div id="performance">
  ## Rendimiento
</div>

<div id="use-regional-domains">
  ### Usa dominios regionales
</div>

Para aplicaciones globales, considera usar dominios personalizados específicos de cada región para cumplir los requisitos de residencia de datos y adaptar la imagen de marca por región:

* Ejemplos: `login-us.example.com`, `login-eu.example.com`, `login-ap.example.com`

<Note>
  La latencia de la autenticación depende de dónde esté ubicado tu tenant de Auth0, no del nombre de dominio personalizado. Elige la región de tu tenant en función de dónde se encuentren tus usuarios.
</Note>

<div id="branding-and-user-experience">
  ## Branding y experiencia de usuario
</div>

<div id="keep-branding-consistent">
  ### Mantén un branding coherente
</div>

MCD añade otra dimensión a tus esfuerzos de branding. Los usuarios esperan coherencia en todos los puntos de contacto:

* **Universal Login**: Personaliza las páginas de inicio de sesión por dominio mediante metadatos del dominio
* **Plantillas de correo electrónico**: Usa variables de dominio personalizado para personalizar los correos
* **Interfaz de usuario de la aplicación**: Alinea el branding de tu aplicación con la experiencia de autenticación

Para obtener una guía completa, consulta las pautas de branding de [B2B](/es/docs/get-started/architecture-scenarios/business-to-business/branding) y [B2C](/es/docs/get-started/architecture-scenarios/business-to-consumer/branding).

<div id="passkey-management">
  ### Gestión de claves de acceso
</div>

* **Comunique esto con claridad**: Informe a los usuarios de que las claves de acceso son específicas de cada dominio
* **Fomente el registro**: Anime a los usuarios a registrar una clave de acceso después del tercer inicio de sesión
* **Haga seguimiento del registro**: Supervise en qué dominios los usuarios han registrado claves de acceso
* **Proporcione ayuda**: Ofrezca documentación clara sobre la gestión de claves de acceso

Consulte [Claves de acceso con múltiples dominios personalizados](/es/docs/customize/custom-domains/multiple-custom-domains/passkeys) para obtener orientación detallada.

<div id="user-communication">
  ### Comunicación con los usuarios
</div>

Cuando los usuarios interactúan con múltiples dominios personalizados:

* **Gestiona las expectativas**: Explica que distintas marcas o portales pueden tener diferentes URL de inicio de sesión
* **Brinda orientación**: Ayuda a los usuarios a entender qué dominio deben usar
* **Gestiona los errores adecuadamente**: Muestra mensajes de error claros si los usuarios intentan acceder al dominio equivocado
* **Documentación de soporte**: Mantén una documentación clara sobre la estructura de tus dominios

<div id="operations-and-maintenance">
  ## Operación y mantenimiento
</div>

<div id="monitoring-and-alerting">
  ### Monitoreo y alertas
</div>

Configure el monitoreo de lo siguiente:

* **Vencimiento del certificado**: Genere alertas 30, 15 y 7 días antes del vencimiento
* **Estado de verificación del dominio**: Supervise los errores de verificación
* **Errores de autenticación**: Haga un seguimiento de los errores por dominio personalizado
* **Estado de DNS**: Supervise la resolución de DNS de todos los dominios personalizados
* **Rendimiento de la API**: Haga un seguimiento de la latencia de Management API en las operaciones de dominio

<div id="documentation">
  ### Documentación
</div>

Mantén una documentación completa:

* **Inventario de dominios**: Lista de todos los dominios personalizados con metadatos y finalidad
* **Mapeo de aplicaciones**: Qué aplicaciones usan qué dominios
* **Procedimientos operativos**: Procedimientos para operaciones comunes (agregar dominios, renovar certificados, solucionar problemas)
* **Diagramas de arquitectura**: Representación visual de la estructura de dominios
* **Información de contacto**: Equipos o personas responsables de cada dominio

<div id="testing">
  ### Pruebas
</div>

Pruebe exhaustivamente antes de pasar a producción:

* **Pruebas funcionales**: Verifique la autenticación de cada dominio personalizado
* **Pruebas de integración**: Pruebe todas las aplicaciones con los dominios personalizados que tengan asignados
* **Pruebas de correo electrónico**: Verifique que los correos usen los dominios personalizados correctos
* **Pruebas de conmutación por error**: Pruebe la reversión al dominio predeterminado
* **Pruebas de rendimiento**: Realice una prueba de carga de la autenticación a través de dominios personalizados

<div id="change-management">
  ### Gestión de cambios
</div>

Al realizar cambios en dominios personalizados:

* **Usa primero staging**: Prueba los cambios en entornos no productivos
* **Despliegue gradual**: Implementa primero los cambios en un subconjunto de dominios
* **Supervisa de cerca**: Mantente atento a posibles problemas durante el despliegue
* **Ten un plan de rollback**: Prepárate para revertir los cambios si es necesario
* **Comunica los cambios**: Informa a las partes interesadas sobre los cambios previstos

<div id="common-pitfalls-to-avoid">
  ## Errores comunes que conviene evitar
</div>

<div id="dont-over-complicate-domain-structure">
  ### No compliques demasiado la estructura de los dominios
</div>

* **Antipatrón**: Crear un dominio personalizado para cada variación posible
* **Mejor enfoque**: Empieza con unos pocos dominios principales y amplía según sea necesario

<div id="dont-forget-to-update-all-integration-points">
  ### No olvides actualizar todos los puntos de integración
</div>

Al agregar nuevos dominios personalizados, actualiza:

* URLs de devolución de llamada de las aplicaciones
* URI de redirección de los proveedores sociales
* endpoints de las conexiones empresariales
* Lógica de validación de tokens
* Reglas de monitoreo y alertas

<div id="dont-neglect-documentation">
  ### No descuides la documentación
</div>

* **Antipatrón**: Depender del conocimiento institucional
* **Mejor enfoque**: Documentar la asignación de dominios, el significado de los metadatos y los procedimientos operativos

<div id="dont-ignore-certificate-management">
  ### No descuides la gestión de certificados
</div>

* **Anti-pattern**: Dejar que los certificados caduquen sin previo aviso
* **Better approach**: Automatizar el monitoreo y los procesos de renovación

<div id="dont-mix-authentication-contexts">
  ### No mezcles contextos de autenticación
</div>

* **Patrón no recomendado**: usar el mismo dominio personalizado para marcas o clientes no relacionados
* **Mejor enfoque**: mantener una separación clara entre distintos contextos de autenticación
