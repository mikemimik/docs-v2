[Claves de acceso](/docs/es/secure/multi-factor-authentication/fido-authentication-with-webauthn) ofrecen una experiencia de authentication passwordless resistente al phishing mediante WebAuthn. Al usar [múltiples dominios personalizados](/docs/es/customize/custom-domains/multiple-custom-domains), las claves de acceso se registran para cada dominio debido al modelo de seguridad de WebAuthn.

<div id="how-passkeys-work-with-custom-domains">
  ## Cómo funcionan las claves de acceso con los dominios personalizados
</div>

<div id="webauthn-relying-party-id-rp-id">
  ### ID de Relying Party (RP ID) de WebAuthn
</div>

WebAuthn usa un identificador de Relying Party (RP ID) para definir el alcance de las credenciales de clave de acceso. El RP ID determina lo siguiente:

* **Dónde se pueden usar las claves de acceso**: Las claves de acceso están vinculadas al dominio en el que se crearon
* **Límites de seguridad**: Evita que las claves de acceso se usen en dominios no autorizados
* **Experiencia de usuario**: Los usuarios deben registrar claves de acceso por separado para cada dominio personalizado

<div id="per-domain-enrollment">
  ### Registro por dominio
</div>

Con múltiples dominios personalizados, cada dominio tiene su propio ID de RP, lo que significa lo siguiente:

* Una clave de acceso registrada en `login.brand1.com` **no** puede usarse en `login.brand2.com`
* Los usuarios que se autentican a través de distintos dominios personalizados deben registrar una clave de acceso para cada dominio
* Las claves de acceso de cada dominio se administran de forma independiente

<div id="understanding-the-passkey-user-experience">
  ## Comprender la experiencia de usuario con la clave de acceso
</div>

<div id="single-brand-single-domain">
  ### Una sola marca, un solo dominio
</div>

**Setup**: Un dominio personalizado para una sola marca

**Experiencia del usuario**:

1. El usuario visita `login.example.com`
2. El usuario registra una clave de acceso
3. El usuario puede usar la clave de acceso en todos los futuros inicios de sesión a través de `login.example.com`

**Complejidad**: Baja: experiencia sencilla con clave de acceso

<div id="multi-brand-separate-domains">
  ### Varias marcas, dominios separados
</div>

**Setup**: Varias marcas, cada una con su propio dominio personalizado

**Experiencia del usuario**:

1. El usuario visita `login.brand1.com` y registra una clave de acceso
2. Más tarde, el mismo usuario visita `login.brand2.com` (una marca diferente)
3. La clave de acceso registrada previamente no está disponible
4. El usuario debe registrar una nueva clave de acceso para `login.brand2.com`

**Complejidad**: Media: los usuarios necesitan claves de acceso distintas para cada marca

**Práctica recomendada**: Comunica a los usuarios que cada marca requiere registrar una clave de acceso por separado

<div id="multi-tenant-with-common-domain">
  ### Multi-tenant con dominio común
</div>

**Setup**: Varios clientes, con un dominio personalizado común para servicios compartidos.

**Experiencia del usuario**:

1. La mayoría de los usuarios se autentican a través del dominio común
2. Los usuarios registran sus claves de acceso una sola vez para el dominio común
3. Las claves de acceso funcionan de forma coherente en la mayoría de los escenarios de autenticación
4. Los casos especiales (dominios específicos de cada cliente) requieren un registro por separado

**Complejidad**: Baja a media: la mayoría de los usuarios tienen una experiencia coherente

<div id="enable-passkeys-for-your-tenant">
  ### Activa claves de acceso para tu tenant
</div>

Antes de usar claves de acceso con dominios personalizados, asegúrate de tenerlas activadas:

1. Ve a **Auth0 Dashboard** &gt; **Security** &gt; **Multi-factor Auth**
2. Activa **WebAuthn with FIDO Security Keys**
3. Configura las opciones de las claves de acceso

<div id="configure-custom-domains-for-passkeys">
  ### Configurar dominios personalizados para claves de acceso
</div>

Cada dominio personalizado obtiene automáticamente su propio ID de RP:

* **Formato del ID de RP**: el propio dominio personalizado (por ejemplo, `login.example.com`)
* **No se requiere configuración adicional**: Auth0 configura automáticamente el ID de RP para cada dominio personalizado verificado

<div id="verify-rp-id-configuration">
  ### Verificar la configuración del RP ID
</div>

Para verificar el RP ID de un dominio personalizado:

1. Ve a **Auth0 Dashboard** &gt; **Branding** &gt; **Custom Domains**
2. Selecciona tu dominio personalizado
3. En los detalles del dominio, se mostrará el RP ID

  ## Patrones de implementación

<div id="prompt-for-passkey-enrollment-per-domain">
  ### Solicitar el registro de claves de acceso por dominio
</div>

Guía a los usuarios para registrar claves de acceso en cada dominio personalizado que usen:

```javascript
import { createAuth0Client } from '@auth0/auth0-spa-js';

async function setupPasskeyEnrollment() {
  const auth0 = await createAuth0Client({
    domain: 'login.example.com',
    clientId: 'YOUR_CLIENT_ID'
  });

  // Verificar si el usuario está autenticado
  const isAuthenticated = await auth0.isAuthenticated();

  if (isAuthenticated) {
    // Verificar si la clave de acceso está registrada para este dominio
    const user = await auth0.getUser();

    if (!user.passkey_enrolled) {
      // Solicitar al usuario que registre la clave de acceso
      showPasskeyEnrollmentPrompt();
    }
  }
}

function showPasskeyEnrollmentPrompt() {
  // Mostrar interfaz para fomentar el registro de la clave de acceso
  const banner = document.createElement('div');
  banner.innerHTML = `
    <div class="passkey-prompt">
      <p>Set up passkey for faster, more secure login on this site</p>
      <button onclick="enrollPasskey()">Set Up Passkey</button>
    </div>
  `;
  document.body.prepend(banner);
}
```

<div id="track-passkey-enrollment-by-domain">
  ### Seguimiento del registro de claves de acceso por dominio
</div>

Almacena en qué dominios un usuario ha registrado claves de acceso:

```javascript
// En tu Auth0 Action (Post Login)
exports.onExecutePostLogin = async (event, api) => {
  const domain = event.custom_domain?.domain;
  const authMethods = event.authentication?.methods || [];

  // Verificar si el usuario se autenticó con clave de acceso
  const usedPasskey = authMethods.some(method =>
    method.name === 'webauthn' || method.name === 'passkey'
  );

  if (usedPasskey) {
    // Registrar en qué dominios el usuario tiene claves de acceso registradas
    const enrolledDomains = event.user.app_metadata?.passkey_domains || [];

    if (domain && !enrolledDomains.includes(domain)) {
      enrolledDomains.push(domain);
      api.user.setAppMetadata('passkey_domains', enrolledDomains);
    }

    // Agregar claim al token
    api.idToken.setCustomClaim('passkey_enrolled', true);
    api.idToken.setCustomClaim('passkey_domain', domain);
  } else {
    // El usuario no usó clave de acceso
    api.idToken.setCustomClaim('passkey_enrolled', false);
  }
};
```

A continuación, en tu aplicación:

```javascript
async function checkPasskeyEnrollment() {
  const auth0 = await createAuth0Client({
    domain: window.CUSTOM_DOMAIN,
    clientId: 'YOUR_CLIENT_ID'
  });

  const isAuthenticated = await auth0.isAuthenticated();

  if (isAuthenticated) {
    const user = await auth0.getUser();
    const claims = await auth0.getIdTokenClaims();

    // Verificar si la clave de acceso está registrada para el dominio actual
    const passkeyEnrolledHere = claims.passkey_enrolled &&
                                 claims.passkey_domain === window.CUSTOM_DOMAIN;

    if (!passkeyEnrolledHere) {
      // Solicitar al usuario que registre la clave de acceso para este dominio
      promptPasskeyEnrollment();
    }
  }
}
```

<div id="domain-specific-enrollment-pages">
  ### Páginas de registro específicas para cada dominio
</div>

Crea páginas de registro específicas para cada dominio personalizado:

```javascript
// Página de registro para Brand 1
// URL: https://login.brand1.com/enroll-passkey

import { createAuth0Client } from '@auth0/auth0-spa-js';

async function enrollPasskeyForBrand1() {
  const auth0 = await createAuth0Client({
    domain: 'login.brand1.com',
    clientId: 'YOUR_CLIENT_ID'
  });

  try {
    // Trigger para el registro de clave de acceso
    await auth0.loginWithPopup({
      authorizationParams: {
        acr_values: 'http://schemas.openid.net/pape/policies/2007/06/multi-factor',
        prompt: 'login'
      }
    });

    alert('Passkey enrolled successfully for Brand 1!');
  } catch (error) {
    console.error('Passkey enrollment failed:', error);
  }
}
```

<div id="contextual-enrollment-prompts">
  ### Prompts contextuales de registro
</div>

Muestra prompts de registro de clave de acceso en función del comportamiento del usuario. Aspectos clave:

* Registra cuándo los usuarios descartan los prompts de registro (guárdalo en `localStorage`)
* Comprueba `logins_count` en los metadatos del usuario para mostrar prompts después de varias visitas
* Verifica que la clave de acceso no esté ya registrada para el dominio actual

```javascript
async function shouldShowEnrollmentPrompt(auth0, customDomain) {
  const storageKey = `passkey_prompt_dismissed_${customDomain}`;

  // No mostrar si el usuario lo descartó
  if (localStorage.getItem(storageKey)) return false;

  const claims = await auth0.getIdTokenClaims();
  const passkeyEnrolled = claims.passkey_enrolled &&
                          claims.passkey_domain === customDomain;

  if (passkeyEnrolled) return false;

  // Mostrar prompt después del 3er inicio de sesión
  const user = await auth0.getUser();
  return (user.logins_count || 0) >= 3;
}
```

<div id="help-documentation">
  ### Documentación de ayuda
</div>

Incluye una documentación de ayuda clara:

**Ejemplo de entrada de preguntas frecuentes**:

**P: ¿Por qué tengo que volver a configurar una clave de acceso?**

R: Por seguridad, las claves de acceso están vinculadas a dominios específicos. Si inicias sesión a través de un portal distinto (p. ej., Marca A frente a Marca B), tendrás que configurar una clave de acceso para cada uno. Así se protege la seguridad de tus cuentas, ya que las claves de acceso solo funcionan donde corresponde.

**P: ¿Necesito un dispositivo distinto para cada clave de acceso?**

R: ¡No! Puedes usar el mismo dispositivo (teléfono, ordenador o llave física de seguridad) para claves de acceso en distintos dominios. Cada clave de acceso es simplemente una credencial independiente almacenada en tu dispositivo.

<div id="related-origins-future-feature">
  ### Orígenes relacionados (función futura)
</div>

Auth0 tiene previsto admitir los orígenes relacionados de WebAuthn, lo que permitirá compartir claves de acceso entre los dominios especificados. Esta función:

* Te permitirá configurar dominios como &quot;relacionados&quot; para las claves de acceso
* Permitirá a los usuarios usar una clave de acceso registrada en `login.brand1.com` en `login.brand2.com` si están configurados como relacionados
* Ofrecerá mayor flexibilidad para implementaciones con varias marcas

**Estado**: Previsto para una versión posterior a la disponibilidad general

  ## Escenarios de migración

<div id="migrating-from-single-custom-domain-to-multiple">
  ### Migración de un único dominio personalizado a varios
</div>

**Antes**: Un único dominio personalizado con claves de acceso registradas

**Después**: Múltiples dominios personalizados para distintas marcas

**Desafío**: Las claves de acceso existentes solo funcionan en el dominio original

**Enfoque de migración**:

1. **Mantener activo el dominio original**: Mantén el dominio personalizado original como dominio común
2. **Implementación gradual**: Introduce nuevos dominios personalizados de forma gradual
3. **Notificación a los usuarios**: Informa a los usuarios de que tendrán que registrar claves de acceso en los nuevos dominios
4. **Proporcionar un flujo de nuevo registro**: Facilita que los usuarios registren claves de acceso en los nuevos dominios
5. **Supervisar la adopción**: Haz un seguimiento de las tasas de registro de claves de acceso por dominio

**Plantilla de comunicación**:

> &quot;¡Estamos lanzando páginas de inicio de sesión específicas para cada marca! Tu clave de acceso actual seguirá funcionando en [original domain]. Cuando visites nuestras nuevas páginas de inicio de sesión, se te pedirá que configures una clave de acceso para iniciar sesión más rápido también allí.&quot;

<div id="migrating-between-custom-domains">
  ### Migración entre dominios personalizados
</div>

**Escenario**: Cambiar de `old-domain.com` a `new-domain.com`

**Problema**: Las claves de acceso no se pueden transferir

**Pasos de la migración**:

1. **Funcionamiento en paralelo**: Mantén ambos dominios activos simultáneamente durante la transición
2. **Detectar claves de acceso registradas**: Haz un seguimiento de qué usuarios tienen claves de acceso en el dominio anterior
3. **Solicitar un nuevo registro**: Cuando los usuarios inicien sesión a través del nuevo dominio, pídeles que vuelvan a registrar su clave de acceso
4. **Período de gracia**: Mantén activo el dominio anterior durante un período de transición
5. **Retirar el dominio anterior**: Tras la adopción, desactiva el dominio anterior

```javascript
// En tu Auth0 Action
exports.onExecutePostLogin = async (event, api) => {
  const domain = event.custom_domain?.domain;
  const oldDomain = 'old-domain.com';
  const newDomain = 'new-domain.com';

  // Verificar si el usuario tenía una clave de acceso en el dominio anterior
  const hadOldPasskey = event.user.app_metadata?.passkey_domains?.includes(oldDomain);

  // El usuario está en el nuevo dominio pero aún no tiene una clave de acceso registrada
  if (domain === newDomain && hadOldPasskey) {
    const newDomainPasskeys = event.user.app_metadata?.passkey_domains?.includes(newDomain);

    if (!newDomainPasskeys) {
      // Establecer un indicador para solicitar el nuevo registro
      api.idToken.setCustomClaim('should_enroll_passkey', true);
      api.idToken.setCustomClaim('migrated_from', oldDomain);
    }
  }
};
```

<div id="best-practices">
  ## Prácticas recomendadas
</div>

1. **Usa un dominio común**: Usa un dominio personalizado común para minimizar la cantidad de dominios que requieren el registro de claves de acceso
2. **Comunicación clara**: Informa a los usuarios sobre los requisitos de registro por dominio
3. **Usa prompt estratégicamente**: Muestra prompts de registro después de que los usuarios demuestren interacción (p. ej., 3+ inicios de sesión)
4. **Haz seguimiento del registro**: Supervisa qué usuarios han registrado claves de acceso en qué dominios
5. **Proporciona ayuda**: Ofrece documentación clara y soporte para la gestión de claves de acceso
6. **Prueba exhaustivamente**: Prueba los flujos de claves de acceso en todos los dominios personalizados antes del despliegue en producción
7. **Planifica las migraciones**: Al cambiar dominios personalizados, planifica que los usuarios vuelvan a registrarse
8. **Supervisa la adopción**: Haz seguimiento de las tasas de registro y uso de claves de acceso por dominio

<div id="troubleshooting">
  ## Solución de problemas
</div>

<div id="passkey-not-working-on-custom-domain">
  ### La clave de acceso no funciona en un dominio personalizado
</div>

**Síntomas**: El usuario registró una clave de acceso, pero no puede usarla

**Posibles causas**:

* El usuario está en un dominio personalizado distinto de aquel en el que registró la clave de acceso
* Problemas de compatibilidad del navegador
* La clave de acceso se eliminó del dispositivo

**Resolución**:

1. Confirma que el usuario esté en el dominio personalizado correcto
2. Verifica la compatibilidad del navegador con WebAuthn
3. Indica al usuario que vuelva a registrar la clave de acceso si es necesario

<div id="user-confused-about-multiple-enrollments">
  ### Usuario confundido por múltiples registros
</div>

**Síntomas**: El usuario informa que la &quot;clave de acceso no funciona&quot; al cambiar de dominio

**Causa**: El usuario no entiende que el registro es por dominio

**Resolución**:

1. Proporcionar mensajes claros sobre las claves de acceso por dominio
2. Mostrar en qué dominios el usuario tiene claves de acceso registradas
3. Solicitar el registro cuando el usuario visite un dominio nuevo
