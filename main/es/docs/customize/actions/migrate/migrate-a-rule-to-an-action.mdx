description: Descripción de cómo usar la herramienta de migración de Rules en el Auth0 Dashboard
title: Herramienta de migración
---

Simplifica el proceso de migración de Rules a Actions con la herramienta de migración de Rules del [Auth0 Dashboard](https://manage.auth0.com/). Además de facilitar el paso de una Rule a una Action, la nueva herramienta también ofrece orientación integrada y correcciones para patrones comunes de Rule.

<div id="before-migration">
  ## Antes de la migración
</div>

Antes de comenzar la migración, consulta [Migrar de Rules a Actions](/es/docs/customize/actions/migrate/migrate-from-rules-to-actions). Esta guía de migración es un excelente recurso que explica las diferencias entre Rules y Actions y te será de gran ayuda durante el proceso de migración.

<div id="rule-migration-order">
  ### Orden de migración de Rules
</div>

Las Rules se ejecutan antes que las Actions, lo que implica que deben migrarse en el orden inverso al de ejecución. Migra la última Rule que se ejecuta en tu flujo actual y ve avanzando en sentido inverso hasta la primera Rule, en orden cronológico inverso.

No seguir este orden de migración puede generar errores por efectos secundarios imprevistos derivados de dependencias en el orden de las Rules y las Actions.

<div id="safely-rolling-back">
  ### Revertir de forma segura
</div>

Las herramientas de migración te permiten revertir los cambios de forma segura y sencilla si algo sale mal. Vuelve a habilitar tu Rule y omite la lógica de la Action correspondiente.

<div id="migrating-a-rule">
  ## Migrar una Rule
</div>

Para empezar a migrar tu Rule, selecciona **Auth Pipeline &gt; Rules**. Una vez que lo selecciones, verás tus Rules con la opción de migrar cada una de ellas de forma individual.

<Frame>![](/docs/images/cdy7uua7fh8z/79DmJp8C2vYZpdos51Z0nM/5a665d04c66dd9f0b8f0e25198cb9e4e/SS1_Dashboard_migration_screenshot.png)</Frame>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Al elegir Migrate to Action, solo se inicia la migración. Como desarrollador, puedes decidir cuándo activar la nueva Action.
</Callout>

<div id="migrate-to-action">
  ### Migrar a Action
</div>

Después de seleccionar **Migrate to Action**, la herramienta te pide que crees una nueva Action.

<Frame>![](/docs/images/cdy7uua7fh8z/1A8FnVyvqAwW8hbrkOQDoG/f83ba9d5157db14ac2a3a666b2280574/SS2_Migrate_to_Action_screenshot.png)</Frame>

Después de seleccionar **Create,** se crea la nueva Action y se te muestra el editor de Actions.

<Frame>![](/docs/images/cdy7uua7fh8z/2A10fE69aVngEcL2ZQCqWX/720dca2a8e6005c9378efad3bc140af0/Screenshot_2024-06-06_at_11.55.38_AM.png)</Frame>

Si la Rule se migra a Actions y sigue activa, la Rule finaliza antes de que se ejecute la lógica.

Esto se hace llamando a `api.rules.wasExecuted()`. Si la función `wasExecuted` recibe una cadena que contiene el ID de la Rule, devuelve `true` si la Rule ya se ejecutó (y, en caso contrario, devuelve `false`).

Esta es una herramienta útil para evitar que tu lógica se ejecute dos veces. En cuanto deshabilitas la Rule referenciada, la función devuelve `false` y la lógica de negocio pasa a ejecutarse en la Action.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  El valor del ID de la Rule se rellena automáticamente para cualquier rule que se migre con la función **Migrate to Action**.
</Callout>

Para obtener más información sobre la función `api.rules.wasExecuted()`, consulta [Triggers de Actions: Post Login - Objeto de API](/es/docs/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger/post-login-api-object).

<div id="rewrite-rule-code">
  ## Reescribe el código de la Rule
</div>

Selecciona el botón **Rules** en el editor de Actions para cargar la herramienta de migración de Rules. Allí, puedes consultar rápidamente el código de la Rule migrada. Debes revisar el código de la Rule seleccionada en la barra lateral.

Copia el contenido de la función de la Rule y pégalo debajo del comentario `// YOUR CODE HERE` en la Action. Es probable que este código no esté adaptado al modelo de programación de Actions.

<Frame>![](/docs/images/cdy7uua7fh8z/2jWCZFnfERuN7NzBPrpCDn/80ba271699893a007dac7baaf2dd44cb/Screenshot_2024-06-06_at_11.57.03_AM.png)</Frame>

El editor de Actions indica dónde debe refactorizarse el código. También ofrece sugerencias sobre cómo corregir automáticamente la mayoría de las situaciones; para revisar la sugerencia o la corrección, pasa el cursor sobre el código subrayado:

<Frame>![](/docs/images/cdy7uua7fh8z/5jRGI4iDX9ZgwcPRADTbZ1/caa90923a76800526871b137806e2af3/Screenshot_2024-06-06_at_11.58.07_AM.png)</Frame>

Es posible que puedas aplicar actualizaciones específicas de la Action seleccionando una opción en la ventana modal **Quick Fix**.

<Frame>![](/docs/images/cdy7uua7fh8z/qTS4RD6cjWFVfI0T9Uph3/45776c464f7247ddd8e91860f4646f2e/Screenshot_2024-06-06_at_11.58.48_AM.png)</Frame>

Cuando seleccionas la corrección recomendada, el código se actualiza automáticamente para que sea compatible.

<Frame>![](/docs/images/cdy7uua7fh8z/1cAq5A1MNVLPAsinHjtROh/c71276afd1f6df9e01d71cc869778907/Screenshot_2024-06-06_at_11.59.37_AM.png)</Frame>

Hay situaciones en las que las correcciones no pueden aplicarse automáticamente, pero aun así se ofrecen indicaciones:

<Frame>![](/docs/images/cdy7uua7fh8z/4Lx3RxJh8v4tj1J2SpPVvU/0ff35b0f40bb24d195ab2f4b06817d86/Screenshot_2024-06-06_at_12.00.22_PM.png)</Frame>

Por ejemplo, en el caso de la función `callback` de la Rule, se recomienda usar [`api.access.deny`](/es/docs/customize/actions/explore-triggers/signup-and-login-triggers/pre-user-registration-trigger/pre-user-registration-api-object) para los errores, o simplemente `return` cuando el resultado es satisfactorio.

<div id="test-the-new-action">
  ### Probar la nueva Action
</div>

Las Actions se pueden probar directamente en el editor, ya que son una parte esencial del proceso de migración. Para obtener más información, consulta [Probar Actions](/es/docs/customize/actions/test-actions).

<Warning>
  Te recomendamos encarecidamente que valides toda la lógica de migración en un tenant de desarrollo antes de habilitarla en producción.
</Warning>

<div id="automatically-enabling-the-action">
  ### Habilitar automáticamente la Action
</div>

Cuando se actualice el código de la Action y esté listo para realizar el cambio:

1. Seleccione **Deploy**.
2. En la alerta que confirma su despliegue, seleccione **Add to flow**.
3. Agregue al flujo la Action que acaba de crear y, a continuación, seleccione **Apply.**
4. Seleccione **Continue**, **and disable any Rules selected** para habilitar simultáneamente la nueva Action y deshabilitar la Rule original. Si aún no está listo, seleccione **Cancel** y podrá deshabilitar la Rule más adelante para activar la lógica de la Action migrada.

Para obtener más información sobre cómo agregar su Action al flujo, consulte [Escriba su primera Action](/es/docs/customize/actions/write-your-first-action).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Si vuelve a habilitar una Rule, se omitirá la lógica de la Action correspondiente.
</Callout>

<div id="migrate-remaining-rules">
  ## Migrar las reglas restantes
</div>

Este proceso puede repetirse para migrar cada Rule de tu flujo de inicio de sesión. Recuerda que las Rules deben migrarse en el orden inverso al que se ejecutan durante el flujo de inicio de sesión.
