Según en qué punto de la implementación te encuentres, tienes varias opciones para probar y depurar Auth0 Actions.

Te recomendamos que pruebes las acciones individualmente desde el <Tooltip tip="Auth0 Dashboard: el principal producto de Auth0 para configurar tus servicios." cta="Ver glosario" href="/docs/es/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip> a medida que las implementas. Cuando esté todo listo, debes verificar que el flujo de extremo a extremo funcione como se espera usando los logs del tenant a través del Auth0 Dashboard.

<div id="test-individual-actions">
  ## Probar Actions individuales
</div>

Puedes probar Actions individuales con el Actions Code Editor. La función de prueba del editor simula una llamada a la Action con un payload de ejemplo basado en el flujo al que está asociada. Para probar una Action individual:

1. Ve a [Auth0 Dashboard &gt; Actions](https://manage.auth0.com/#/actions) y elige el flujo y la Action que quieras editar.
2. Busca el Actions Code Editor y selecciona **Test** (el icono de reproducción) en su barra lateral. Edita el payload en la barra lateral de Test para analizar el resultado de distintos payloads.
3. Selecciona **Run**.

Los resultados de la prueba muestran los pasos que siguió la Action, la salida de la consola, los errores que se produjeron y estadísticas útiles sobre la ejecución de la Action.

<Frame>![Captura de pantalla del módulo de pruebas dentro del Actions Code Editor](/docs/images/cdy7uua7fh8z/7I4CRM8SmRFiibjuk55L86/794042f525cddd55a8005cc646b3c9c2/test-action-code-editor.png)</Frame>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  También puedes probar una Action con el endpoint [**Test an Action**](https://auth0.com/docs/api/management/v2#!/Actions/post_test_action) de la Management API, proporcionando un payload correspondiente al flujo al que está asociada. Para obtener un payload de ejemplo, prueba la Action con la función Test del Actions Code Editor.
</Callout>

<div id="debug-deployed-actions">
  ## Depurar Actions implementadas
</div>

Ahora que has probado tu Action, es momento de implementarla y observar su ejecución. Suponiendo que ya has implementado tu Action y la has añadido a un flujo (para aprender a adjuntar Actions a Flows, consulta la sección &quot;Adjuntar la Action a un flujo&quot; en [Write Your First Action](/docs/es/customize/actions/write-your-first-action)), puedes depurar tu Action en producción en unos pocos pasos.

<div id="verify-end-to-end-login-flow">
  ### Verificar el flujo de inicio de sesión de principio a fin
</div>

Para una Action de `post-login`, puedes verificar el flujo completo de inicio de sesión realizando un intento de inicio de sesión en tu tenant:

1. Ve a [Auth0 Dashboard &gt; Authentication &gt; Authentication Profile](https://manage.auth0.com/#/authentication-profiles) y selecciona **Try**. Se abrirá una ventana con un inicio de sesión de ejemplo.
2. Completa el flujo de inicio de sesión. Se ejecutarán todas las Actions configuradas.

Una vez completado, se te redirigirá a una página que mostrará los atributos del perfil de usuario que recibirán tus aplicaciones o un mensaje de error que explicará qué salió mal.

<div id="analyze-tenant-logs">
  ### Analizar los logs del tenant
</div>

En la lista de logs del tenant, puedes ver los logs asociados a Actions y optar por visualizarlos. En el siguiente ejemplo, el tipo de log `Success Login` está asociado a una Action en el trigger `post-login` de este tenant.

<div id="unit-test-actions">
  ## Pruebas unitarias de Actions
</div>

​Puedes hacer pruebas unitarias de tus Auth0 Actions simulando el `event` y la `api` que se pasan a tu función Action. Tus Actions pueden estar en un sistema de control de versiones o en un directorio local, y puedes usar una herramienta como Jest para probarlas automáticamente. En el siguiente ejemplo, usamos el trigger [Login / Post Login](/docs/es/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger) y [Jest](https://jestjs.io/) para simular y realizar las pruebas.

**Código de la Action:**

```javascript lines
// action.js

exports.onExecutePostLogin = async (event, api) => {
  const namespace = "https://my-app.example.com";
  if (event.authorization) {
    api.idToken.setCustomClaim(`${namespace}/favorite_color`, "pink");
  }
};
```

**Código del archivo de prueba:**

```javascript lines
// action.spec.js

const { onExecutePostLogin } = require("./action.js");
test("sets favorite_color ID token claim to pink", async () => {
  const namespace = "https://my-app.example.com";
  const mockFunction = jest.fn();
  const mockApi = { idToken: { setCustomClaim: mockFunction } };
  const mockEvent = {authorization: {}};

  await onExecutePostLogin(mockEvent, mockApi);
  expect(mockFunction).toBeCalledWith(
    `${namespace}/favorite_color`,
    "pink"
  );
});
```

<div id="best-practices">
  ## Prácticas recomendadas
</div>

Al crear Actions en Auth0, [se recomienda crear un tenant de Auth0 específico para cada entorno](/docs/es/get-started/auth0-overview/create-tenants/set-up-multiple-environments). Esto le permite aislar su base de usuarios de producción de la de otros entornos, así como configurar distintos administradores para cada entorno (por ejemplo, es posible que los ingenieros no tengan acceso a la configuración de producción).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Si desea administrar, probar e implementar por completo sus Actions mediante una CLI o una canalización de CI/CD, puede usar la [Deploy CLI Tool](/docs/es/deploy-monitor/deploy-cli-tool).
</Callout>
