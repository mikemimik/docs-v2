Cuando se produce un error de autorización y tu URL de callback es válida, el <Tooltip tip="Servidor de autorización: servidor centralizado que ayuda a definir los límites del acceso de un usuario. Por ejemplo, tu servidor de autorización puede controlar los datos, las tareas y las funciones disponibles para un usuario." cta="Ver glosario" href="/es/docs/glossary?term=Authorization+Server">Servidor de autorización</Tooltip> devuelve a tu URL de callback los parámetros de error y estado correspondientes. Si tu URL de callback no es válida, tu aplicación mostrará la [página de error predeterminada de Auth0](/es/docs/authenticate/login/auth0-universal-login/error-pages).

Tu aplicación también puede mostrar la página de error predeterminada de Auth0 por motivos distintos de una URL de callback no válida, por ejemplo:

* Faltan parámetros obligatorios al llamar al [endpoint de Login](https://auth0.com/docs/api/authentication#login) de la Auth0 Authentication API.
* El usuario abre un enlace caducado de restablecimiento de contraseña (cuando se usa la experiencia de Classic Login).
* El usuario accede a una página de inicio de sesión guardada en marcadores y no se ha especificado una [ruta de inicio de sesión predeterminada](/es/docs/authenticate/login/auth0-universal-login/configure-default-login-routes).

<div id="parameters">
  ## Parámetros
</div>

Si eliges configurar una página de error personalizada, el servidor de autorización devolverá parámetros añadidos a la URL como una cadena de consulta.

| Parámetro           | Descripción                                                                    |
| ------------------- | ------------------------------------------------------------------------------ |
| `client_id`         | Identificador de la aplicación de Auth0.                                       |
| `connection`        | Conexión utilizada en el momento del error.                                    |
| `lang`              | Idioma configurado para usarse en el momento del error.                        |
| `error`             | Código del error.                                                              |
| `error_description` | Descripción del error.                                                         |
| `tracking`          | Identificador utilizado por Auth0 para localizar errores en los logs internos. |

Los parámetros que se muestran varían según el tipo de error y son específicos de la solicitud. Por ejemplo, si la solicitud que produjo el error no contenía un `client_id`, el servidor de autorización no devolverá el parámetro `client_id`.

<div id="display-a-custom-error-page">
  ## Mostrar una página de error personalizada
</div>

Si deseas mostrar una página de error personalizada, tienes dos opciones:

1. Redirigir a los usuarios a una página de error personalizada mediante Auth0 Dashboard o la Management API de Auth0.
2. Configurar Auth0 para que muestre una página de error personalizada en tu lugar mediante la Management API.

<div id="redirect-users-to-a-custom-error-page-using-the-dashboard">
  ### Redirige a los usuarios a una página de error personalizada con el Dashboard
</div>

Usa el Dashboard para configurar Auth0 de forma que redirija a los usuarios a una página de error personalizada:

1. Ve a [Auth0 Dashboard &gt; Tenant Settings](https://manage.auth0.com/#/tenant/).
2. Busca la sección **Error Pages**.
3. Selecciona la opción **Redirect users to your own error page**.
4. Introduce la URL de la página de error que quieres que vean tus usuarios y selecciona **Save**.

<div id="redirect-users-to-a-custom-error-page-using-the-management-api">
  ### Redirige a los usuarios a una página de error personalizada con la Management API
</div>

Usa el <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/es/docs/glossary?term=Management+API">Management API</Tooltip> endpoint [Actualizar la configuración del tenant](https://auth0.com/docs/api/management/v2/#!/Tenants/patch_settings). Reemplaza el valor del marcador de posición `{mgmtApiAccessToken}` por tu <Tooltip tip="Access Token: Credencial de autorización, en forma de una cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/es/docs/glossary?term=Access+Token">token de acceso</Tooltip> de la Management API y actualiza el valor del campo `url` en el cuerpo JSON para que apunte a la ubicación de la página de error.

  	fmt.Println(string(body))

  }

  require 'uri'

  response = http.request(request)
  puts response.read_body

| Valor                   | Descripción                                                                                                                      |
| ----------------------- | -------------------------------------------------------------------------------------------------------------------------------- |
| `MGMT_API_ACCESS_TOKEN` | [Token de acceso para la Management API](https://auth0.com/docs/api/management/v2/tokens) con el scope `update:tenant_settings`. |
| `show_log_link`         | Indica si se muestra un enlace al error en los logs de tu tenant. Los valores válidos son `true` y `false`.                      |
| `url`                   | Ubicación de la página de error personalizada a la que quieres redirigir.                                                        |

<div id="render-a-custom-error-page">
  ### Mostrar una página de error personalizada
</div>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Cuando los errores se muestran dentro del widget de Classic Login (por ejemplo, un enlace de restablecimiento de contraseña vencido), no se mostrará una página de error personalizada alojada por Auth0, aunque esté configurada.
</Callout>

Usa el endpoint [Actualizar la configuración del tenant](https://auth0.com/docs/api/management/v2/#!/Tenants/patch_settings) de la Management API. Reemplaza el valor del marcador de posición `{mgmtApiAccessToken}` por tu token de acceso de la Management API y actualiza el valor del campo `html` en el cuerpo JSON con una cadena que contenga el HTML de tu página.

Puedes usar la sintaxis de Liquid para incluir las siguientes variables:

* `{client_id}`
* `{connection}`
* `{lang}`
* `{error}`
* `{error_description}`
* `{tracking}`
