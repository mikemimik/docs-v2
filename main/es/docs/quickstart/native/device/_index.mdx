    <Section id={sections[1].id} title={sections[1].title} stepNumber="2" isSingleColumn>
      Cuando el usuario inicia la aplicación del dispositivo y desea autorizarla, su aplicación debe solicitar un código de dispositivo a la Auth0 Authentication API para asociarlo con la sesión del usuario.

      Para obtener el device code, tu aplicación debe llamar al [Authorize endpoint](https://auth0.com/docs/api/authentication#-post-oauth-device-code-) del Device Authorization Flow de la Authentication API:

      Mientras su aplicación del dispositivo espera a que el usuario la active, debe llamar de forma intermitente al endpoint [POST /oauth/token de la Authentication API](https://auth0.com/docs/api/authentication#-post-oauth-token-)
      y gestionar la respuesta de forma adecuada.

      <Info>
        Asegúrese de que su aplicación del dispositivo espere el tiempo indicado por `interval` (en segundos) o hasta
        que reciba una respuesta satisfactoria, lo que sea más largo, para evitar problemas de latencia de red.
      </Info>

      ```bash
      curl --request POST \ 
        --url 'https://{yourDomain}/oauth/token' \
        --header 'content-type: application/x-www-form-urlencoded' \
        --data grant_type=urn:ietf:params:oauth:grant-type:device_code \
        --data device_code=AUTH0_SCOPES \
        --data 'client_id={yourClientId}'
      ```

    <Section id={sections[6].id} title={sections[6].title} stepNumber="7" isSingleColumn>
      Después de que el usuario autorice la aplicación del dispositivo, esta recibe una respuesta HTTP 200 y el siguiente payload:

      ```json lines
      {
        "access_token": "eyJz93a...k4laUWw",
        "refresh_token": "GEbRxBN...edjnXbL",
        "id_token": "eyJ0XAi...4faeEoQ",
        "token_type": "Bearer",
        "expires_in": 86400
      }
      ```

      <Info>
        Debes validar tus tokens antes de guardarlos. Para saber cómo hacerlo, consulta Validar tokens de acceso y Validar tokens de ID.
      </Info>

      Los tokens de acceso se usan para llamar al [endpoint Get User Info de la Authentication API](https://auth0.com/docs/api/authentication#get-user-info) (si tu
      aplicación para dispositivos solicitó el scope `openid`) o a la API especificada por el parámetro `audience`.
      Si llamas a tu propia API, tu aplicación para dispositivos debe validar el token de acceso antes de usarlo.

      Los tokens de ID contienen información del usuario que debe [decodificarse y extraerse](/es/docs/secure/tokens/id-tokens#id-token-payload). La Authentication API
      solo devuelve un `id_token` si tu aplicación para dispositivos solicitó el scope `openid`.

      Los tokens de actualización se usan para obtener un nuevo token de acceso o
      token de ID después de que el anterior haya expirado. La Authentication API solo devuelve un `refresh_token` si
      la configuración **Allow Offline Access** está habilitada para la API especificada por el parámetro `audience`
      y tu aplicación para dispositivos solicitó el scope `offline_access`.

      Para llamar a tu API, la aplicación de tu dispositivo debe enviar el token de acceso como token Bearer en el
      encabezado `Authorization` de tu solicitud HTTP.

      ```bash lines
      curl --request GET \
        --url https://myapi.com/api \
        --header 'authorization: Bearer AUTH0_API_ACCESS_TOKEN' \
        --header 'content-type: application/json'
      ```

    <Section id={sections[8].id} title={sections[8].title} stepNumber="9" isSingleColumn>
      Para obtener un nuevo token de acceso para un usuario, la aplicación del dispositivo puede llamar al [endpoint POST /oauth/token de la Authentication API](https://auth0.com/docs/api/authentication#-post-oauth-token-) con el
      parámetro `refresh_token`.

      ```bash lines
      curl --request POST \
        --url 'https://{yourDomain}/oauth/token' \
        --header 'content-type: application/x-www-form-urlencoded' \
        --data grant_type=refresh_token \
        --data 'client_id={yourClientId}' \
        --data 'client_secret={yourClientSecret}' \
        --data refresh_token=AUTH0_REFRESH_TOKEN
      ```

      Si la solicitud se realizó correctamente, la aplicación de tu dispositivo recibe una respuesta HTTP 200 con la siguiente carga útil:

      ```json lines
      {
        "access_token": "eyJ...MoQ",
        "expires_in": 86400,
        "scope": "openid offline_access",
        "id_token": "eyJ...0NE",
        "token_type": "Bearer"
      }
      ```

      Para obtener más información sobre los tokens de actualización, consulta [Tokens de actualización](https://auth0.com/docs/secure/tokens/refresh-tokens).

    <Section id={sections[9].id} title={sections[9].title} stepNumber="10" isSingleColumn>
      Los registros del tenant se generan con cualquier interacción que se produzca y pueden usarse para solucionar problemas.

      ### Respuestas de token

      Mientras esperas a que el usuario autorice el dispositivo, es posible que recibas varias respuestas HTTP 4xx.

      #### Autorización pendiente

      Verás este error mientras esperas a que el usuario realice una acción. Sigue haciendo polling con el
      `interval` sugerido obtenido en el paso anterior de este tutorial.

      ```json lines
      `HTTP 403`
      {
      "error": "authorization_pending",
      "error_description": "..."
      }
      ```

      #### Más despacio

      Estás haciendo polling demasiado rápido. Ve más despacio y usa el intervalo sugerido obtenido en el paso anterior de este
      tutorial. Para evitar que este error se produzca debido a la latencia de la red, debes empezar a contar cada intervalo después de
      recibir la respuesta de la última solicitud de polling.

      ```json lines
      `HTTP 429`
      {
      "error": "slow_down",
      "error_description": "..."
      }
      ```

      #### Token expirado

      El usuario no autorizó el dispositivo con la suficiente rapidez, por lo que el `device_code` ha caducado. Su
      aplicación debe notificar al usuario que el flujo ha caducado y pedirle que lo reinicie.

      <Info>
        El error `expired_token` se devuelve exactamente una vez. Después de eso, el endpoint devuelve el
        error `invalid_grant`.
      </Info>

      ```json lines
      `HTTP 403`
      {
      "error": "expired_token",
      "error_description": "..."
      }
      ```

      #### Acceso denegado

      Si se deniega el acceso, verá:

      ```json lines
      `HTTP 403`
      {
      "error": "access_denied",
      "error_description": "..."
      }
      ```

      Esto puede ocurrir por diversos motivos, entre ellos:

      * El usuario se negó a autorizar el dispositivo.
      * El servidor de autorización denegó la transacción.
      * Una Action configurada denegó el acceso.

    <Section id={sections[11].id} title={sections[11].title} stepNumber="12" isSingleColumn>
      Para usar el flujo de autorización de dispositivos, una aplicación de dispositivo debe:

      * Admitir Server Name Indication (SNI)
      * Ser una Aplicación nativa
      * Tener el [Método de autenticación](/es/docs/secure/application-credentials#application-authentication-methods) configurado como **None**
      * Ser [conforme con OIDC](/es/docs/get-started/applications/application-settings#oauth)
      * No haber sido creada mediante registro dinámico de clientes

      Además, el flujo de autorización de dispositivos no permite:

      * Social Connections que usen claves de desarrollador de Auth0, a menos que se use la nueva experiencia de Universal Login
      * Acceder a parámetros de la cadena de consulta desde una página de inicio de sesión alojada o desde Actions.
