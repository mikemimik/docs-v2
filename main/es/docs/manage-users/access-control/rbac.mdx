description: Comprende el concepto del control de acceso basado en roles y cómo se aplica en Auth0.
title: Control de acceso basado en roles
---

El control de acceso basado en roles (RBAC) se refiere a la asignación de permisos a los usuarios en función de su rol dentro de una organización. Ofrece un enfoque simple y fácil de gestionar para la administración del acceso, y es menos propenso a errores que asignar permisos a los usuarios de forma individual.

Al usar RBAC para la gestión de roles, analizas las necesidades de tus usuarios y los agrupas en roles según responsabilidades compartidas. Después, asignas uno o más roles a cada usuario y uno o más permisos a cada rol. Las relaciones entre usuario y rol, y entre rol y permisos, simplifican la asignación de usuarios, ya que dejan de gestionarse de forma individual y, en su lugar, reciben privilegios conforme a los permisos asignados a sus roles.

Por ejemplo, si usaras RBAC para controlar el acceso a una aplicación de RR. HH., podrías asignar a los gerentes de RR. HH. un rol que les permita actualizar los datos de los empleados, mientras que el resto de los empleados solo podría ver sus propios datos.

Al definir tu estrategia de control de acceso, se recomienda asignar a los usuarios la menor cantidad de permisos necesaria para que puedan realizar su trabajo.

  Para un modelo de autorización más sólido, consulta [Fine-Grained Authorization (FGA)](https://docs.fga.dev/authorization-concepts) para la autorización basada en atributos y relaciones.
</Callout>

<div id="benefits-of-rbac">
  ## Beneficios de RBAC
</div>

Con RBAC, la gestión del acceso es más sencilla siempre que te ciñas estrictamente a los requisitos de los roles. RBAC te ayuda a:

* crear una asignación de permisos sistemática y repetible
* auditar fácilmente los privilegios de los usuarios y corregir los problemas detectados
* añadir y modificar roles rápidamente, así como implementarlos en las API
* reducir la posibilidad de errores al asignar permisos a los usuarios
* integrar a usuarios de terceros dándoles roles predefinidos
* cumplir de forma más eficaz los requisitos normativos y legales de confidencialidad y privacidad

<div id="rbac-model">
  ## Modelo RBAC
</div>

<div id="roles">
  ### Roles
</div>

En esencia, un rol es un conjunto de permisos que puedes aplicar a los usuarios. El uso de roles facilita agregar, quitar y ajustar permisos, en lugar de asignarlos individualmente a cada usuario. A medida que tu base de usuarios crece en escala y complejidad, los roles resultan especialmente útiles.

También puedes usar roles para agrupar permisos definidos para varias API. Por ejemplo, supongamos que tienes un módulo de marketing que permite a los usuarios crear y distribuir boletines para los clientes. Tu especialista en contenidos de marketing crea todos los boletines y los prepara para su distribución. Del mismo modo, tienes un módulo de eventos que permite a los usuarios crear, publicar y gestionar el registro en eventos. Tu coordinador de eventos crea los eventos. Una vez que el vicepresidente de Marketing aprueba los boletines y los eventos, su asistente publica los eventos y distribuye los boletines. En este caso, tu API de boletines podría tener un permiso `distribute:newsletters` y tu API de eventos podría tener un permiso `publish:events`. Luego, estos permisos podrían agruparse en un rol llamado `Marketing Publisher` y asignarse al asistente del vicepresidente de Marketing.

Además, se pueden [agregar roles específicos de la organización a los miembros de la organización](/es/docs/manage-users/organizations/configure-organizations/add-member-roles) y usarse para permitir el acceso en tu aplicación según las organizaciones con las que un usuario final inicia sesión. Esto es particularmente útil al dar soporte a productos SaaS y multi-tenant, donde un usuario determinado puede tener un rol con privilegios en una organización, pero no en otras.

<div id="overlapping-role-assignments">
  ## Asignaciones de roles superpuestas
</div>

RBAC es un modelo aditivo, así que, si tienes asignaciones de roles superpuestas, tus permisos efectivos son la unión de tus asignaciones de roles.

Por ejemplo, supongamos que tienes una API que proporciona datos para una aplicación para eventos. Creas un rol de `Organizer` y le asignas permisos que le permiten ver, crear y editar eventos. También creas un rol de `Registrant` y le asignas permisos que le permiten ver eventos y registrarse en ellos. Cualquier usuario que tenga los roles `Organizer` y `Registrant` podrá ver, crear, editar y registrarse en eventos.

<div id="role-based-access-control-in-auth0">
  ## Control de acceso basado en roles en Auth0
</div>

Actualmente, ofrecemos dos formas de implementar el control de acceso basado en roles (RBAC), que puedes usar en lugar del sistema interno de control de acceso de tu API o en combinación con él:

* [Authorization Core](/es/docs/manage-users/access-control/configure-core-rbac)
* [Authorization Extension](/es/docs/customize/extensions/authorization-extension)

Estamos ampliando el conjunto de funcionalidades de Authorization Core para que iguale la funcionalidad de Authorization Extension. Nuestra nueva implementación central de RBAC mejora el rendimiento y la escalabilidad, y con el tiempo ofrecerá un sistema de RBAC más flexible que Authorization Extension.

Por ahora, ambas implementan las funcionalidades clave de RBAC y te permiten restringir los scopes personalizados definidos para una API a aquellos que se hayan asignado al usuario como permisos. Para ver una comparación, consulta [Authorization Core vs. Authorization Extension](/es/docs/manage-users/access-control/authorization-core-vs-authorization-extension).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  El conjunto de funcionalidades de [Authorization Core](/es/docs/manage-users/access-control/rbac) y [Authorization Extension](/es/docs/customize/extensions/authorization-extension) son funciones completamente independientes. Para administrar grupos, roles o permisos, debes usar la función en la que se crearon originalmente.
</Callout>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Aunque [Delegated Administration Extension (DAE)](/es/docs/customize/extensions/delegated-administration-extension) y el conjunto de funcionalidades de [Authorization Core](/es/docs/manage-users/access-control/rbac) son funciones completamente independientes, puedes usar el conjunto de funcionalidades de Authorization Core para crear y administrar roles para DAE mediante Actions. Para saber cómo hacerlo, consulta [Casos de uso de ejemplo: Actions con autorización](/es/docs/manage-users/access-control/sample-use-cases-actions-with-authorization).
</Callout>

<div id="extending-rbac">
  ### Extender RBAC
</div>

Puede lograr un mayor control usando [Actions](/es/docs/customize/actions) para restringir el acceso en función de una combinación de atributos, como el departamento del usuario, la hora del día, la ubicación desde la que se accede o cualquier otro atributo del usuario o de la API (por ejemplo, username, nivel de autorización de seguridad o nombre de la API).

Para obtener más información sobre cómo usar Actions con políticas de autorización, consulte [Casos de uso de ejemplo: Actions con autorización](/es/docs/manage-users/access-control/sample-use-cases-actions-with-authorization).
