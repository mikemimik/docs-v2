Las cookies, que se usan para la autenticación y el mantenimiento de sesiones, pueden protegerse configurando atributos. Auth0 usa cookies para lo siguiente:

* OIDC empresarial con `form_post`
* <Tooltip tip="Security Assertion Markup Language (SAML): Protocolo estandarizado que permite a dos partes intercambiar información de autenticación sin contraseña." cta="Ver glosario" href="/es/docs/glossary?term=SAML">SAML</Tooltip> HTTP-POST Binding
* Mensaje web (también conocido como `checkSession`)

<div id="samesite-attributes">
  ## Atributos SameSite
</div>

Puede agregar atributos de [cookies SameSite](https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers/Set-Cookie/SameSite) en el encabezado de respuesta HTTP `set-cookie` para restringir el comportamiento del navegador. Esto puede impedir que el navegador envíe el par `key=value` de la cookie según el tipo de interacción que activó la solicitud HTTP.

Los valores de atributo admitidos son los siguientes:

| Atributo | Descripción                                                                                                                                                                                         |
| -------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `strict` | Envía la cookie si el usuario navega dentro de los límites del origen del sitio web                                                                                                                 |
| `lax`    | Envía la cookie si el usuario navega entre dominios, pero no en contextos de terceros (iframes o publicaciones)                                                                                     |
| `none`   | Envía la cookie con solicitudes que cruzan los límites del origen del sitio web. A menos que se den otras condiciones (es decir, que las cookies de terceros estén bloqueadas), no envíe la cookie. |

Algunos atributos de cookie con los que quizá esté familiarizado son:

| Atributo            | Descripción                                                                                                                                                                                                  |
| ------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `httpOnly`          | Permite que una cookie se envíe solo con solicitudes HTTP; no puede leerse mediante `document.cookie` de Javascript                                                                                          |
| `secure`            | Permite que el navegador envíe la cookie solo a un contexto seguro; que el contexto se considere seguro o no depende del navegador, pero normalmente esto requiere el uso de HTTPS                           |
| `max-age / expires` | Controla si la cookie es de **sesión** (por ejemplo, se elimina cuando el navegador finaliza su sesión) o **persistente** (por ejemplo, la cookie permanece después de que finaliza la sesión del navegador) |

Cuando lo recibe, el navegador analiza los encabezados y actualiza su almacén de cookies en consecuencia.

<div id="browser-cookie-changes">
  ## Cambios en las cookies del navegador
</div>

A partir de febrero de 2020, Google Chrome v80 cambió la forma en que gestiona las cookies. Auth0 implementó los siguientes cambios en la forma en que gestiona las cookies:

* Las cookies sin el atributo `SameSite` se configurarán como `lax`
* Las cookies con `SameSite=none` deben ser seguras; de lo contrario, no pueden guardarse en el almacén de cookies del navegador

El objetivo de estos cambios es mejorar la seguridad y ayudar a mitigar los ataques CSRF.

Estos cambios afectan a las siguientes cookies:

* `auth0` (gestiona las sesiones de usuario)
* `auth0-mf` (gestiona información relevante para la <Tooltip tip="Autenticación multifactor (MFA): proceso de autenticación del usuario que usa un factor además del nombre de usuario y la contraseña, como un código vía SMS." cta="Ver glosario" href="/es/docs/glossary?term=multi-factor+authentication">autenticación multifactor</Tooltip>)
* `did` (el identificador de un dispositivo/agente de usuario)

Para estas cookies, Auth0 hará lo siguiente:

* Configurar el atributo `SameSite` como `none`, por lo que la cookie requerirá el uso de HTTPS (independientemente del entorno)
* Configurar cookies de respaldo en caso de que un navegador heredado no admita que `SameSite` se establezca en `None`. Estas cookies de respaldo son `auth0_compat`, `auth0-mf_compat` y `did_compat`

El servidor cambia la forma en que la presenta cuando el visitante regresa y establece una cookie de visitado. <div id="features-affected">
  ## Funcionalidades afectadas
</div>

La siguiente tabla muestra cómo los cambios en el atributo `SameSite` pueden afectar a tus aplicaciones.

| Comportamiento de la aplicación                                                                                                                                                        | Afectado por el cambio |
| -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------- |
| Cookies configuradas como `sameSite=none` cuando el sitio web no es `https://`                                                                                                         | Sí                     |
| Las cookies no tienen configurado un valor explícito para el atributo `sameSite` y se necesitan en un contexto de origen cruzado (como HTTP form&#95;post o la inserción de un iframe) | Sí                     |
| Aplicaciones nativas (todo lo que no sea cookies + basado en web)                                                                                                                      | No (M2M)               |
| Ya se está configurando un valor explícito para el atributo `sameSite` de la cookie                                                                                                    | No                     |
| Subdominio diferente dentro del mismo eTLD+1 (la aplicación está en el mismo eTLD+1 que el tenant de Auth0 con dominio personalizado)                                                  | Potencialmente         |

Si usas una aplicación web con sesiones (p. ej., para guardar preferencias de usuario, carritos de compra, etc.) y permites que los usuarios inicien sesión usando <Tooltip tip="Proveedor de identidad (IdP): servicio que almacena y gestiona identidades digitales." cta="Ver glosario" href="/es/docs/glossary?term=identity+providers">proveedores de identidad</Tooltip> como Google, Github o Auth0, entonces dependes de las cookies para que esa funcionalidad sea posible. Hay cambios en el comportamiento de las cookies del navegador que pueden afectar la experiencia del usuario. Google Chrome, por ejemplo, es el primer proveedor de navegadores en implementar un cambio que podría no ser compatible con tu aplicación web.

Puede que notes que las especificaciones de Google Chrome y Microsoft Edge para establecer `SameSite` como undefined han cambiado: `SameSite` ya no toma `none` como valor predeterminado, sino `lax`.

Por ejemplo, supongamos que creas una nueva interfaz de usuario y tienes varios servicios a los que accedes a través de un proxy mediante una puerta de enlace de Auth0. En esta puerta de enlace, creas una sesión basada en cookies. Si haces una solicitud de origen cruzado, puede que veas esta advertencia en la consola de Javascript:

`A cookie associated with a cross-site resource (URL) was set without the SameSite attribute. A future release of Chrome will only deliver cookies with cross-site requests if they are set with SameSite=None and Secure. You can review cookies in developer tools under Application>Storage>Cookies and see more details at https://www.chromestatus.com/feature/5088147346030592 and https://www.chromestatus.com/feature/5633521622188032`

<div id="actions-you-need-to-take">
  ## Acciones que debe realizar
</div>

Para prepararse para este cambio, debe:

* Revisar la lista de [navegadores no compatibles](https://www.chromium.org/updates/same-site/incompatible-clients).
* Configurar su aplicación para que use `SameSite=none` si utiliza `response_mode=form_post` al interactuar con Auth0 (tenga en cuenta que Chrome no hace excepciones, ni siquiera para `localhost`)
* Configurar su cookie como segura si su atributo `SameSite` es igual a `None.` De lo contrario, el navegador la rechazará. Si utiliza HTTP para sus URL de devolución de llamada, estas dejarán de funcionar si usa esas cookies para asociar el `state` de la solicitud de autorización/<Tooltip tip="Nonce: Número arbitrario emitido una sola vez en un protocolo de autenticación para detectar y prevenir ataques de repetición." cta="Ver glosario" href="/es/docs/glossary?term=nonce">nonce</Tooltip>. Por lo tanto, debe usar HTTPS o establecer `SameSite=lax`
