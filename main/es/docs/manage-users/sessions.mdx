title: Sesiones

Una sesión es un conjunto de interacciones entre un usuario y una aplicación durante un período determinado. Una sola sesión puede incluir varias actividades (como vistas de página, eventos, interacciones sociales y transacciones de comercio electrónico) y puede almacenar esta información temporalmente mientras el usuario está conectado.

Con una implementación estándar del encabezado Set-Cookie, una sesión termina cuando un usuario abandona un sitio web o cierra el navegador. Para evitar que los usuarios tengan que iniciar sesión cada vez, las aplicaciones pueden prolongar las sesiones al establecer una duración máxima para la <Tooltip tip="Cookie de sesión: entidad que, cuando está presente, permite considerar al usuario autenticado." cta="Ver glosario" href="/es/docs/glossary?term=session+cookie">cookie de sesión</Tooltip>. Las sesiones terminan cuando un usuario cierra sesión o cuando se alcanza el límite de duración de la sesión.

Para obtener más información, consulta la [Política de Privacidad y Cookies de Auth0](https://auth0.com/privacy).

<div id="session-use-cases">
  ## Casos de uso de la sesión
</div>

Auth0 mantiene una sesión de inicio de sesión para cualquier usuario que se autentique a través de una aplicación. Cuando un usuario realiza un nuevo inicio de sesión estándar, Auth0 restablece la sesión de inicio de sesión. Actualizar una contraseña, el correo electrónico, el número de teléfono o el username también hace que caduque la sesión de Auth0 del usuario.

Cuando creas una aplicación que requiere autenticación, puedes usar sesiones para determinar si un usuario está autenticado cada vez que se produce una solicitud. Según cómo se haya creado tu aplicación, se recomiendan distintos <Tooltip tip="Flujo de autorización: concesión de autorización (o flujo de trabajo) especificada en el marco de OAuth 2.0." cta="Ver glosario" href="/es/docs/glossary?term=authorization+flows">flujos de autorización</Tooltip> para ofrecer una experiencia más segura a los usuarios.

Por ejemplo, considera un sitio web compatible con OIDC (<Tooltip tip="OpenID: estándar abierto de autenticación que permite a las aplicaciones verificar las identidades de los usuarios sin recopilar ni almacenar información de inicio de sesión." cta="Ver glosario" href="/es/docs/glossary?term=OpenID">OpenID</Tooltip> Connect) llamado storezero.io.

<Frame>![Ejemplo de sitio web de comercio electrónico Storezero.io](/docs/images/cdy7uua7fh8z/5XXxdX4fuApQtAapQfZU1b/2fd9161af60962e3de3fc951d95b83d1/use-case-storezero.png)</Frame>

Storezero.io no requiere que sus usuarios inicien sesión para completar compras. Sin embargo, los usuarios deben iniciar sesión para ver la sección Mi cuenta del sitio.

Para los casos de uso que se enumeran a continuación, considera un escenario en el que un usuario quiere revisar sus pedidos anteriores antes de completar la compra. Para hacerlo, navega a la página Todos los pedidos de la sección Mi cuenta y se le pide que inicie sesión.

<div id="login-flows">
  ### Flujos de inicio de sesión
</div>

La mayoría de los tipos de aplicaciones (como las aplicaciones web, las aplicaciones de página única y las aplicaciones nativas) deben usar el [flujo de código de autorización con PKCE](/es/docs/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce) para gestionar la autenticación de inicio de sesión. Este flujo consiste en intercambiar un código de autorización por tokens.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  El flujo de código de autorización con PKCE reemplaza el uso anterior del flujo implícito para las aplicaciones de página única sin backend. Las implementaciones nuevas deben usar este flujo para garantizar una seguridad óptima. También se recomienda encarecidamente migrar las aplicaciones existentes que usan flujo implícito al flujo de código de autorización con PKCE.
</Callout>

<div id="user-logs-in-with-username-and-password">
  #### El usuario inicia sesión con nombre de usuario y contraseña
</div>

En este ejemplo, un usuario inicia sesión manualmente con su nombre de usuario y contraseña:

1. El SDK de Auth0 crea una sesión local y redirige al usuario al servidor de autorización de Auth0 (endpoint `/authorize`).
2. El servidor de autorización crea una sesión y luego redirige al usuario al prompt de inicio de sesión y autorización.
3. El usuario se autentica con su nombre de usuario y contraseña.
4. El servidor de autorización de Auth0 actualiza la sesión del usuario creada previamente para indicar que ha iniciado sesión.
5. Según el flujo utilizado, el servidor de autorización redirige al usuario de vuelta a tu aplicación junto con un token de identidad o un código de autorización.
6. Tu aplicación intercambia el token o el código de autorización por un token de acceso y completa el flujo.

Con este flujo, se crean dos sesiones:

* La **sesión local** (storezero.io), que indica a la aplicación si un usuario está autenticado.
* La **sesión del <Tooltip tip="Servidor de autorización: servidor centralizado que contribuye a definir los límites del acceso de un usuario. Por ejemplo, tu servidor de autorización puede controlar los datos, las tareas y las funciones disponibles para un usuario." cta="Ver glosario" href="/es/docs/glossary?term=authorization+server">servidor de autorización</Tooltip>** (storezero.auth0.com), que indica al servidor si un usuario está autenticado. La sesión del servidor también puede, opcionalmente, registrar detalles sobre la autenticación.

  * Por ejemplo, el servidor de autorización puede rastrear si un usuario utilizó [autenticación multifactor (MFA)](/es/docs/secure/multi-factor-authentication). Esta información puede usarse para determinar si se le debe pedir a un usuario que inicie sesión o use MFA la próxima vez que llegue al servidor de autorización.

<div id="user-logs-in-with-identity-provider">
  #### El usuario inicia sesión con un proveedor de identidad
</div>

En este ejemplo, el usuario opta por iniciar sesión con Facebook en lugar de usar su nombre de usuario y contraseña:

1. El SDK de Auth0 crea una sesión local y redirige al usuario al servidor de autorización de Auth0 (endpoint `/authorize`).
2. El servidor de autorización crea una sesión y luego redirige al usuario al prompt de inicio de sesión y autorización.
3. Al elegir iniciar sesión con Facebook, el servidor de autorización redirige al usuario a Facebook.
4. Facebook crea una sesión y autentica al usuario. Luego, actualiza su sesión para indicar que el usuario ha iniciado sesión.
5. Facebook devuelve al usuario al servidor de autorización de Auth0. Luego, el servidor de autorización actualiza su sesión para indicar que el usuario ha iniciado sesión.
6. Según el flujo utilizado, el servidor de autorización devuelve al usuario a tu aplicación, junto con un token de identidad o un código de autorización.
7. Tu aplicación intercambia el token o el código de autorización por un token de acceso y completa el flujo.

En este escenario, se crean tres sesiones: la **sesión local** (storezero.io), la **sesión del servidor de autorización** (storezero.auth0.com) y una **sesión de <Tooltip tip="Identity Provider (IdP): Servicio que almacena y administra identidades digitales." cta="Ver glosario" href="/es/docs/glossary?term=identity+provider">proveedor de identidad</Tooltip> (IdP)** (facebook.com).

La sesión del IdP en el servidor de Facebook autentica al usuario y proporciona una experiencia fluida de <Tooltip tip="Single Sign-On (SSO): Servicio que, después de que un usuario inicia sesión en una aplicación, inicia automáticamente la sesión de ese usuario en otras aplicaciones." cta="Ver glosario" href="/es/docs/glossary?term=SSO">inicio de sesión único</Tooltip>. Como es muy probable que los usuarios ya hayan iniciado sesión en Facebook, a menudo se les autentica sin que tengan que proporcionar manualmente sus credenciales de Facebook.

<div id="session-management-for-spas">
  ### Gestión de sesiones para SPAs
</div>

En los ejemplos anteriores, se crea una sesión local cuando el usuario inicia cualquiera de los dos flujos de inicio de sesión. Esta sesión local puede mantener la sesión iniciada de los usuarios y determinar cuándo deben volver a autenticarse.

Sin embargo, las sesiones locales no están disponibles para aplicaciones sin backend, como las aplicaciones de página única (SPA). En su lugar, estas aplicaciones usan un enfoque distinto conocido como [autenticación silenciosa](/es/docs/authenticate/login/configure-silent-authentication) para mantener la sesión iniciada de los usuarios.

La autenticación silenciosa usa la sesión en el servidor de autorización para determinar cuándo un usuario debe volver a autenticarse. Un iframe oculto redirige las solicitudes de autenticación al servidor de autorización junto con el parámetro `prompt=none`. Este parámetro impide que el servidor solicite información al usuario.

* Si la sesión en el servidor de autorización no ha expirado, la transacción continúa sin interrupciones. El servidor envía un <Tooltip tip="Token de acceso: Credencial de autorización, en forma de una cadena opaca o JWT, utilizada para acceder a una API." cta="Ver glosario" href="/es/docs/glossary?term=access+token">token de acceso</Tooltip> mediante WMRM (modo de respuesta de mensajes web), que utiliza `postMessage`.
* Si la sesión en el servidor de autorización ha expirado o el usuario cierra sesión, la redirección en el iframe devuelve un error. La aplicación debe entonces redirigir al usuario al servidor de autorización para que vuelva a autenticarse.
