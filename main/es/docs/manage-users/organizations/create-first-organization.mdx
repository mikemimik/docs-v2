Para empezar a usar la función [Organizations](/docs/es/manage-users/organizations/organizations-overview), debes crear y configurar tus organizaciones.

<Card title="La disponibilidad varía según el plan de Auth0">
  Tu plan de Auth0 o tu acuerdo personalizado determinan si esta función está disponible. Para obtener más información, consulta [Precios](https://auth0.com/pricing).
</Card>

<div id="create-an-organization">
  ## Crear una organización
</div>

Cuando [creas una organización](/docs/es/manage-users/organizations/configure-organizations/create-organizations), la configuración que definas se usa para personalizar la página de inicio de sesión y las plantillas de correo electrónico que ven los usuarios finales cuando se autentican en el contexto de la organización.

Al usar Organizations de forma predeterminada, esta configuración anulará la de las páginas de <Tooltip tip="Universal Login: Tu aplicación redirige a Universal Login, alojado en el servidor de autorización de Auth0, para verificar la identidad de un usuario." cta="Ver glosario" href="/docs/es/glossary?term=Universal+Login">Universal Login</Tooltip>. Para obtener más información sobre cómo personalizar aún más la página de inicio de sesión o las plantillas de correo electrónico, consulta [Personalizar prompts y plantillas de correo electrónico](#customize-prompts-and-email-templates).

<div id="define-organization-behavior">
  ## Definir el comportamiento de la organización
</div>

Puede que conozcas aplicaciones como Heroku que presentan comportamientos distintos según si un usuario inicia sesión con una cuenta personal o selecciona una organización asociada a su cuenta empresarial. Auth0 te permite definir un comportamiento similar para las organizaciones a nivel de aplicación.

Para cada aplicación que crees en Auth0, puedes decidir si los usuarios finales deben iniciar sesión directamente o si deben autenticarse en el contexto de una organización. En las aplicaciones que requieren que los usuarios inicien sesión a través de una organización, también puedes especificar qué ocurre si no se proporciona ninguna en el flujo de inicio de sesión.

También puede que quieras actualizar los **Allowed Callback URLs** y **Allowed Origins (CORs)** de tu aplicación en la [configuración de la aplicación](/docs/es/get-started/applications/application-settings) para incluir [marcadores de posición para subdominios de organización](/docs/es/get-started/applications/wildcards-for-subdomains).

<div id="customize-prompts-and-email-templates">
  ## Personaliza los prompts y las plantillas de correo electrónico
</div>

Al usar Organizations con los prompts predeterminados de Universal Login, la configuración de marca que estableces al [crear organizaciones](/docs/es/manage-users/organizations/configure-organizations/create-organizations) anula la configuración de marca de las páginas de Universal Login y de las plantillas de correo electrónico. Si quieres modificar aún más las páginas de Universal Login y los correos electrónicos que recibe el usuario final, puedes personalizar las plantillas de página y de correo electrónico.

<div id="page-templates">
  ### Plantillas de página
</div>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Para acceder a la personalización de plantillas de página, debes usar dominios personalizados, una función incluida en cualquier plan de pago.
</Callout>

Para modificar la página de Universal Login, personaliza una plantilla de página con el [lenguaje de plantillas Liquid](https://shopify.github.io/liquid/) y variables de plantilla, y después aplícala mediante la API de plantillas de página de Universal Login. Para obtener más información, consulta [Plantillas de página de Universal Login](/docs/es/customize/login-pages/universal-login/customize-templates).

Como se utiliza la misma plantilla para todas las páginas, puedes implementar páginas de inicio de sesión coherentes con un esfuerzo mínimo.

La plantilla más sencilla que puedes escribir es:

```html lines
<!DOCTYPE html>
<html>
  <head>
    {%- auth0:head -%}
  </head>
  <body>
    {%- auth0:widget -%}
  </body>
</html>
```

Las siguientes etiquetas deben estar presentes en la plantilla:

* `auth0:widget`: Contiene el HTML que define la estructura del widget que se muestra en cada tipo de página (p. ej., Login, Reset Password).
* **`auth0:head`**: Contiene las etiquetas necesarias para mostrar el widget.

Para centrar el widget en la página, sustituye la etiqueta `<body>` por `<body class="_widget-auto-layout">`.

<div id="email-templates">
  ### Plantillas de correo electrónico
</div>

Para modificar los correos electrónicos que recibe el usuario final, personaliza una plantilla de correo electrónico con el [lenguaje de plantillas Liquid](https://shopify.github.io/liquid/) y variables de plantilla. Para obtener más información, consulta [Personalizar plantillas de correo electrónico](/docs/es/customize/email/email-templates).

Las siguientes plantillas de correo electrónico pueden personalizarse aún más para las organizaciones:

* **Welcome**: El usuario final la recibe una vez que verifica su dirección de correo electrónico o, si la verificación del correo electrónico está deshabilitada, cuando se registra (o inicia sesión por primera vez).
* **Password Change**: El usuario final la recibe cuando solicita un cambio de contraseña. Contiene un enlace que lo redirige a la página de restablecimiento de contraseña.
* **Invite User**: El usuario final la recibe cuando lo invitan a una organización. Contiene un enlace que lo redirige a la ruta predeterminada de inicio de sesión que hayas configurado. Para obtener más información, consulta [Invitar a miembros de la organización](/docs/es/manage-users/organizations/configure-organizations/invite-members).

<div id="template-variables">
  ### Variables de plantilla
</div>

Las plantillas de página y de correo electrónico pueden acceder a un conjunto de variables de contexto que puedes usar para influir en cómo se muestra la plantilla.

Para ver las listas de variables disponibles, consulta [Universal Login Page Templates](/docs/es/customize/login-pages/universal-login/customize-templates) para las variables de las plantillas de página y [Customize Email Templates](/docs/es/customize/email/email-templates) para las variables comunes de las plantillas de correo electrónico.

Hay variables adicionales disponibles para la plantilla de página cuando un usuario inicia sesión a través de una organización:

* `organization.id`
* `organization.display_name`
* `organization.name`
* `organization.metadata`
* `organization.branding.logo_url`
* `organization.branding.colors.primary`
* `organization.branding.colors.page_background`

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Los parámetros de la organización solo se exponen a una plantilla de correo electrónico cuando el correo asociado se activa en el contexto de una organización. Por ejemplo, el correo de invitación de usuario siempre se activa en el contexto de una organización, por lo que los parámetros de la organización siempre se exponen en su plantilla. Sin embargo, los correos de bienvenida pueden activarse o no en el contexto de una organización, por lo que los parámetros de la organización no siempre estarán disponibles en su plantilla.
</Callout>

<div id="configure-organization-domains">
  ## Configurar dominios de la organización
</div>

<Warning>
  Organization Domains for Discovery se encuentra actualmente en acceso anticipado. Al usar esta función, aceptas los términos aplicables de la prueba gratuita del [Master Subscription Agreement de Okta](https://www.okta.com/legal). Para obtener más información sobre las etapas de lanzamiento, consulta [Product Release Stages](/docs/es/troubleshoot/product-lifecycle/product-release-stages).
</Warning>

Antes de iniciar sesión, Auth0 puede detectar las Organizations de un usuario mediante el nombre de la organización o el dominio de correo electrónico que introduzca. Para asociar una organización con un dominio, puedes crear dominios de organización. Auth0 los dirige automáticamente al proveedor de identidad (IdP) correcto.

<Note>
  Nota: Los usuarios deben seguir introduciendo su dirección de correo electrónico completa o el nombre de la organización al iniciar sesión. Auth0 usa esa información para determinar la organización asociada.
</Note>

<div id="add-a-domain-to-an-organization">
  ### Añadir un dominio a una organización
</div>

Los administradores del tenant pueden añadir y gestionar dominios en la sección **Domains** de una organización en el Auth0 Dashboard o mediante la Management API.

Para añadir un dominio en el Auth0 Dashboard:

1. Ve a [Auth0 Dashboard &gt; Organizations](https://manage.auth0.com/#/organizations).
2. Selecciona el nombre de la organización. Luego, selecciona **Domains**.
3. Elige **Add Domain**.
4. Introduce el dominio (por ejemplo, `travel0.com`)
5. Una vez verificado, el dominio aparecerá como **Verified** y se utilizará para el descubrimiento de organizaciones antes del inicio de sesión.

<Note>
  Solo se utilizan los dominios verificados durante el descubrimiento de dominios. Los dominios pendientes o no verificados no activarán el enrutamiento.
</Note>

Para obtener más información sobre cómo el descubrimiento de dominios afecta al comportamiento de inicio de sesión, consulta [Login Flows for Organizations](https://auth0.com/docs/manage-users/organizations/login-flows-for-organizations).

<div id="configure-organization-membership">
  ## Configurar la membresía de la organización
</div>

Cada organización tiene sus propios miembros asociados, que son los usuarios que pueden acceder a sus aplicaciones iniciando sesión a través de la página de inicio de sesión configurada de la organización.

Las aplicaciones SaaS y B2B normalmente se ofrecen a equipos en lugar de a usuarios individuales. En este contexto, un equipo puede ser una empresa entera, un pequeño grupo de empleados (por ejemplo, el departamento de marketing) o incluso un grupo temporal de usuarios organizados en torno a un propósito (por ejemplo, un grupo de vecinos que se han inscrito para colaborar con una organización benéfica y compiten como equipo para recaudar la mayor cantidad de dinero para una causa).

Las Organizations en Auth0 son lo suficientemente flexibles como para admitir todos estos casos de uso, pero requieren cierta planificación al crear una aplicación compatible con equipos. Una de las primeras cosas que debe considerar es cómo administrar la membresía del equipo, lo cual puede lograr de las siguientes maneras:

* [Invitar usuarios por correo electrónico](/docs/es/manage-users/organizations/configure-organizations/invite-members)
* [Conceder membresía just-in-time](/docs/es/manage-users/organizations/configure-organizations/grant-just-in-time-membership) a los usuarios que inician sesión mediante una conexión habilitada
* Administrar directamente la membresía mediante la <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip> o el <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip> [asignando miembros](/docs/es/manage-users/organizations/configure-organizations/assign-members) o [eliminando miembros](/docs/es/manage-users/organizations/configure-organizations/remove-members)

Una vez definida la membresía, puede [obtener los miembros de las organizaciones](/docs/es/manage-users/organizations/configure-organizations/retrieve-members) o [obtener la membresía de un usuario en una organización](/docs/es/manage-users/organizations/configure-organizations/retrieve-user-membership).

<div id="assign-roles-to-members">
  ## Asignar roles a los miembros
</div>

A cada miembro de la organización se le puede asignar uno o más roles, que se aplican cuando los usuarios inician sesión a través de la organización. En concreto, puedes [agregar roles a los miembros de la organización](/docs/es/manage-users/organizations/configure-organizations/add-member-roles), [eliminar roles de los miembros de la organización](/docs/es/manage-users/organizations/configure-organizations/remove-member-roles) o [obtener los roles de los miembros de una organización](/docs/es/manage-users/organizations/configure-organizations/retrieve-member-roles). Puedes usar los roles y los permisos asociados para controlar el acceso a [las aplicaciones que desarrollas](/docs/es/manage-users/organizations/custom-development). Para obtener más información sobre los roles, consulta [Control de acceso basado en roles](/docs/es/manage-users/access-control/rbac).
