title: Sesiones con Actions
---

Usar sesiones con [Actions](/es/docs/customize/actions) te permite configurar capacidades de detección y respuesta ante riesgos posteriores a la autenticación para proteger tus aplicaciones y usuarios contra el secuestro de sesiones. También puedes personalizar dinámicamente los [límites de duración de la sesión](/es/docs/manage-users/sessions/configure-session-lifetime-settings).

Para ello, las Actions de Post Login incluyen dos objetos clave:

* **event.session**: Proporciona información relevante, como `id`, `created_at`, `expires_at`, `idle_expires_at`, `updated_at`, `clients`, `authentication_at` y datos de `device`, como `ASN`, `IP` y `User_agent`.
* **api.session:** Te permite gestionar sesiones existentes revocándolas o cambiando las fechas de `expiry`.

Los objetos `event.session` y `api.session` son compatibles con flujos interactivos basados en web, incluidos el [flujo de código de autorización](/es/docs/get-started/authentication-and-authorization-flow/authorization-code-flow), el flujo implícito, el flujo de código de dispositivo, así como <Tooltip tip="Security Assertion Markup Language (SAML): Protocolo estandarizado que permite a dos partes intercambiar información de autenticación sin una contraseña." cta="Ver glosario" href="/es/docs/glossary?term=SAML">SAML</Tooltip> y <Tooltip tip="Security Assertion Markup Language (SAML): Protocolo estandarizado que permite a dos partes intercambiar información de autenticación sin una contraseña." cta="Ver glosario" href="/es/docs/glossary?term=WS-Fed">WS-Fed</Tooltip>.

Puedes usar el objeto `event.session` para revisar las marcas de tiempo de las interacciones más recientes y evaluar los riesgos asociados con las transacciones actuales. También puedes combinar el objeto `event.session` con otros objetos de evento, como `event.authentication` o `event.request`.

Luego, puedes usar el objeto `api.session` para restablecer las fechas de expiración de la sesión existente o revocar la sesión.

Para obtener más información sobre estos objetos, consulta:

* [objeto Event](/es/docs/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger/post-login-event-object): Obtén información sobre el objeto Event de sesión y sus propiedades.
* [objeto de API](/es/docs/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger/post-login-api-object): Obtén información sobre el objeto de API de sesión y sus métodos.

<div id="revoke-sessions-with-actions">
  ## Revocar sesiones con Actions
</div>

El método **api.session.revoke(reason, options)** de Post Login le permite reaccionar ante riesgos asociados a una transacción. Este método incluye una opción que le permite conservar los <Tooltip tip="token de actualización: Token utilizado para obtener un token de acceso renovado sin obligar a los usuarios a iniciar sesión de nuevo." cta="Ver glosario" href="/es/docs/glossary?term=refresh+tokens">tokens de actualización</Tooltip> vinculados a la transacción revocada.

Además de revocar la sesión, el método también iniciará un [iniciador de cierre de sesión OIDC por canal secundario](/es/docs/authenticate/login/logout/back-channel-logout/oidc-back-channel-logout-initiators) `session-revoked` para cerrar la sesión de los usuarios en todas las aplicaciones vinculadas a la sesión actual y registrar un evento [session&#95;revoked](/es/docs/deploy-monitor/logs/log-event-type-codes) en los registros del tenant.

Puede usar este método para:

* Invalidar la transacción de la sesión actual en Auth0
* Denegar la transacción actual
* Revocar todos los [tokens de actualización](/es/docs/secure/tokens/refresh-tokens) asociados a la sesión existente con un valor `session_id` coincidente.

  * Esta es una opción personalizable; puede optar por conservar los tokens de actualización en lugar de revocarlos. Esta operación se ejecuta de forma asíncrona y con el tiempo alcanza la consistencia.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Si quiere usar el método `api.session.revoke(reason,options)`, asegúrese de que exista la propiedad `event.session.id`.

  A diferencia de `api.access.deny()`, `api.session.revoke()` denegará la transacción actual y también revocará la sesión; por lo tanto, será necesario volver a realizar la autenticación del primer factor.
</Callout>

<div id="monitor-revoke-log-events">
  ### Supervisa los eventos de log de revocación
</div>

La operación de revocación añade el siguiente evento de log a los [registro del tenant](/es/docs/deploy-monitor/logs):

Un código de evento `session_revoked` que indica que se revocó una sesión, junto con su atributo `session_id` asociado.

<div id="change-sessions-expiry-dates-with-actions">
  ## Cambiar las fechas de expiración de las sesiones con Actions
</div>

Puedes modificar las [fechas de expiración](/es/docs/manage-users/sessions/session-lifetime-limits) de la sesión con los siguientes métodos de Post Login:

* **api.session.setExpiresAt(absolute)** te permite definir una nueva fecha de expiración absoluta de la sesión (tiempo tras el cual es necesario iniciar sesión de nuevo) para una sesión determinada.
* **api.session.setIdleExpiresAt(idle)** te permite establecer una nueva fecha de expiración por inactividad para una sesión determinada.

Puedes usar estos métodos para personalizar dinámicamente el tiempo de vida de la sesión y las políticas de inactividad en función de:

* La organización del usuario
* La Conexión de Auth0 del usuario
* La pertenencia a un grupo o el perfil de un usuario específico
* La evaluación de riesgos
* Cualquier otro criterio dinámico disponible durante la ejecución de la Action

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Si quieres usar los métodos `api.session.setExpiresAt(absolute)` y `api.session.setIdleExpiresAt(idle)`, asegúrate de que exista una propiedad del objeto `event.session`, como `event.session.id`.

  El método `api.session.setIdleExpiresAt(idle)` establece el tiempo de espera por inactividad de la sesión para la interacción actual. Si no se vuelve a aplicar, las interacciones exitosas posteriores reemplazarán ese tiempo de espera por la configuración de inactividad de la sesión.
</Callout>

<div id="set-session-cookie-persistence-with-actions">
  ## Configura la persistencia de la cookie de sesión con Actions
</div>

El método **api.session.setCookieMode(options)** de Post Login te permite modificar la persistencia de la cookie de sesión. Puedes configurar si una cookie de sesión es persistente o no persistente.

Las sesiones no persistentes (efímeras) mejoran la seguridad porque existen solo en memoria. Estas sesiones se eliminan cuando se cierra el navegador o la aplicación, lo que las hace ideales para flujos de trabajo sensibles, como el acceso desde dispositivos no confiables o escenarios de autenticación escalonada.

Puedes usar este método para:

* Aplicar sesiones efímeras a usuarios con roles o grupos de alto riesgo

* Exigir una nueva autenticación cuando se cierre el navegador en determinados dispositivos o rangos de IP

* Acortar la duración de la sesión en entornos no confiables o compartidos (por ejemplo, equipos públicos)

Este método acepta los siguientes parámetros:

| Propiedad        | Tipo   | Descripción                                                                                                                          |
| ---------------- | ------ | ------------------------------------------------------------------------------------------------------------------------------------ |
| `persistent`     | String | La sesión se almacena en una cookie persistente y se mantiene después de reiniciar el navegador, a menos que se elimine manualmente. |
| `non-persistent` | String | La sesión se almacena solo en memoria y se elimina cuando se cierra el navegador o la aplicación.                                    |

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  El método `api.session.setCookieMode()` reemplaza la configuración predeterminada del tenant para la sesión actual.

  Auth0 conserva la configuración de persistencia durante toda la sesión y no es necesario volver a establecerla en futuros flujos de autenticación silenciosa.

  El método `api.session.setCookieMode()` debe usarse dentro de una Action de Post Login. Si se usa en un contexto donde `api.session` no está disponible, la llamada falla silenciosamente.
</Callout>

Para obtener más información sobre cómo implementar la persistencia de la cookie de sesión, consulta [Configure Keep Me Signed In using Sessions](/es/docs/manage-users/sessions/configure-keep-me-signed-in-sessions).

<div id="limitations">
  ## Limitaciones
</div>

Las sesiones emitidas antes del lanzamiento de los métodos de la API de Post Login `api.session.setExpiresAt(absolute)` y `api.session.setIdleExpiresAt(idle)` no incluirán la siguiente propiedad de event.session: `last_interacted_at.`

Las sesiones emitidas antes del lanzamiento del método de la API de Post Login `api.session.revoke(reason, options)` no incluirán las siguientes propiedades de event.session.device:

* `initial_ip`
* `initial_asn`
* `initial_user_agent`

Por motivos de seguridad, los tiempos de espera por inactividad y los tiempos de espera absolutos no pueden establecerse por encima de la configuración de sesión definida en los [límites de duración de la sesión](/es/docs/manage-users/sessions/configure-session-lifetime-settings) del tenant. Si intentas establecer una fecha que supere los límites de duración, los métodos de la API la ajustarán a esos límites y registrarán un evento de warning (`w`) en el registro del tenant.

<div id="use-cases-revoke-a-session">
  ## Casos de uso: Revocar una sesión
</div>

Puedes usar [Actions](/es/docs/customize/actions) para configurar detecciones de riesgo y revocar sesiones consideradas de riesgo, junto con sus tokens de actualización asociados, mediante el método de Post Login `api.session.revoke(reason, options)` y el objeto `event.session`.

<div id="revoke-a-session-due-to-asn-network-binding">
  ### Revocar una sesión debido a una vinculación de red ASN
</div>

Puedes usar las propiedades del objeto Post Login, `event.session.device.initial_asn` y `event.request.asn`, para vincular las transacciones de la sesión a una red específica de [número de sistema autónomo (ASN)](https://www.arin.net/resources/guide/asn/) mientras dure la sesión y exigir una nueva autenticación si cambia la red ASN.

```js lines
exports.onExecutePostLogin = async (event, api) => {
  const sessionInitialAsn = event.session?.device?.initial_asn;
  const sessionCurrentAsn = event.request.asn;

  // si existe una sesión y el ASN cambia
  if (
    sessionInitialAsn &&
    sessionCurrentAsn &&
    sessionInitialAsn != sessionCurrentAsn
  ) {
    api.session.revoke( "Invalid network change. Login again from a trusted network" )
  }
};
```

En este ejemplo, se realiza una comprobación al inicio de la Action para verificar que las propiedades `event.session.device.initial_asn` y `event.request.asn` se mantengan dentro de la misma red ASN durante la transacción. Si esta comprobación falla, la Action llama a  `api.session.revoke()` para:

* Invalidar la sesión
* Denegar la transacción actual
* Revocar todos los tokens de actualización asociados
* Solicitar una nueva autenticación

<div id="revoke-a-session-due-to-an-ip-binding">
  ### Revocar una sesión por vinculación a una IP
</div>

Puedes usar las propiedades del objeto Post Login `event.session.device.initial_ip` y `event.request.ip` para garantizar que la transacción de la sesión mantenga la misma dirección IP durante toda su vigencia. En este escenario, cualquier cambio de IP se considera un riesgo y se solicitará al usuario que se autentique de nuevo.

```js lines
exports.onExecutePostLogin = async (event, api) => {
  const sessionInitialIp = event.session?.device?.initial_ip;
  const sessionCurrentIp = event.request.ip;

  // si hay una sesión y la IP cambia
  if (
    sessionInitialIp &&
    sessionCurrentIp &&
    sessionInitialIp != sessionCurrentIp
  ) {
    api.session.revoke("Invalid IP change")
  }
};
```

En este ejemplo, se realiza una comprobación al inicio de la Action para verificar que las propiedades `event.session.device.initial_ip` y `event.request.ip` mantengan la misma dirección IP durante la transacción. Si la comprobación falla, la Action llama entonces a  `api.session.revoke()` para:

* Invalidar la sesión
* Denegar la transacción actual
* Revocar todos los tokens de actualización asociados
* Solicitar una nueva autenticación

<div id="use-cases-customize-a-session-expiry-dates">
  ## Casos de uso: Personalizar las fechas de expiración de una sesión
</div>

Puede usar [Actions](/es/docs/customize/actions) para personalizar las fechas de expiración absoluta y por inactividad de la sesión. En concreto, puede configurar las fechas de expiración de una transacción de sesión específica mediante los métodos de Post Login `api.session.setExpiresAt(absolute)` y `api.session.setIdleExpiresAt(idle)`, y el objeto `event.session`.

<div id="customize-absolute-session-expiration-time-based-on-connections">
  ### Personaliza el tiempo de expiración absoluto de la sesión según las conexiones
</div>

Puedes usar las siguientes propiedades del objeto Post Login para definir un tiempo de vida para la conexión utilizada para autenticar a un usuario.

* event.session.created&#95;at
* event.session.expires&#95;at

Y, al usar la Auth0 [Management API](https://auth0.com/docs/api/management/v2/connections/patch-connections-by-id) para crear metadatos de la conexión, `event.connection.metadata.session_timeout` define un tiempo de espera específico para esa conexión.

```js lines
exports.onExecutePostLogin = async (event, api) => {
  const created = Date.parse(event.session?.created_at ?? "");

  // duración de sesión deseada para esta conexión en milisegundos, configurada como metadatos de la conexión
  const connection_lifetime = event.connection?.metadata?.session_timeout;

  // si hay una duración de sesión definida para la conexión, establecerla
  if (event.session?.id && connection_lifetime) {
    api.session.setExpiresAt(created + Number(connection_lifetime));
  }
};
```

En este ejemplo, se realiza una comprobación al inicio de la Action para verificar que haya un `session_timeout` definido en la conexión actual. En ese caso, la Action establece el vencimiento de la sesión para que coincida con el momento en que se `created` la sesión más el `connection_lifetime`.

<div id="customize-session-inactivity-timeout-based-on-the-organization">
  ### Personaliza el tiempo de espera por inactividad de la sesión según la organización
</div>

Puedes definir una variable `current_time` y, mediante un nuevo metadato de la organización llamado `idle_session_timeout`, establecer el tiempo de espera por inactividad que desees para una organización.

```js lines
exports.onExecutePostLogin = async (event, api) => {
  // Los metadatos de la organización están configurados con un tiempo de inactividad más corto para las sesiones (en milisegundos)
  const idle_organization_lifetime =
    event.organization?.metadata?.idle_session_timeout;

  // Si la organización tiene un tiempo de inactividad específico definido, establecer el tiempo de espera
  if (event.session?.id && idle_organization_lifetime) {
    const current_time = new Date().getTime();

    api.session.setIdleExpiresAt(
      current_time + Number(idle_organization_lifetime),
    );
  }
};
```

En este ejemplo, si hay un tiempo de espera de inactividad específico definido para la organización, la Action establece el tiempo de inactividad de la sesión para que sea igual a `current_time` más `idle_organization_lifetime`.
