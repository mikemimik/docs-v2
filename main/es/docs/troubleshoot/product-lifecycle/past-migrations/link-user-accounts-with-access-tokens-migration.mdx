Anteriormente, podías usar <Tooltip tip="token de identidad: credencial destinada al propio cliente, en lugar de para acceder a un recurso." cta="Ver glosario" href="/es/docs/glossary?term=ID+tokens">tokens de identidad</Tooltip> para vincular y desvincular cuentas de usuario en ciertos casos de uso. Auth0 está marcando esta funcionalidad como obsoleta. A partir de ahora, tendrás que usar <Tooltip tip="token de identidad: credencial destinada al propio cliente, en lugar de para acceder a un recurso." cta="Ver glosario" href="/es/docs/glossary?term=access+tokens">tokens de acceso</Tooltip> en todos los casos.

<Warning>
  Esta obsolescencia responde a una posible vulnerabilidad de seguridad. Auth0 recomienda encarecidamente que actualices tu código lo antes posible.
</Warning>

<div id="features-affected">
  ## Funcionalidades afectadas
</div>

Los cambios en la vinculación de cuentas son los siguientes:

* Ya no puedes usar un token de identidad en el encabezado `Authorization`; en su lugar, debes usar un token de acceso.
* Si usas un token de acceso en el encabezado `Authorization` con `update:users` como permiso concedido, puedes enviar en el cuerpo de la request el `user_id` o el token de ID de la cuenta secundaria.
* Si usas un token de acceso en el encabezado `Authorization` con `update:current_user_metadata` como permiso concedido, solo puedes enviar el token de ID de la cuenta secundaria en el cuerpo de la request.
* Si envías el token de ID de la cuenta secundaria en el cuerpo de la request (los casos de uso descritos en los dos puntos anteriores), debe cumplirse lo siguiente:

  * El token de ID debe estar firmado con `RS256` (puedes establecer este valor en **Dashboard &gt; Clients &gt; Client Settings &gt; Advanced Settings &gt; OAuth**.
  * El claim `aud` del token de ID debe identificar al cliente y tener el mismo valor que el claim `azp` del token de acceso.
* Para desvincular cuentas, ya no puedes usar un token de identidad en el encabezado `Authorization`. Debes usar un token de acceso en su lugar.

Hay varias formas de vincular y desvincular cuentas. En la siguiente lista puedes ver los casos de uso y cómo les afectan estos cambios.

| Caso de uso                                                                                                                                                                                                                               | Estado            |
| ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------- |
| Usa el endpoint `POST /api/v2/users/{id}/identities` de la Management API y envía el token de identidad de la cuenta principal en el encabezado `Authorization`.                                                                          | Afectado          |
| Usa el endpoint `POST /api/v2/users/{id}/identities` de la Management API y envía un token de acceso (con scope `update:users`) en el encabezado `authorization`, y el `user_id` de la cuenta secundaria en el payload.                   | No afectado       |

| Usa el endpoint `POST /api/v2/users/{id}/identities` de la Management API y envía un token de acceso en el encabezado `Authorization` y el token de ID de la cuenta secundaria en el payload.                                             | Nuevo caso de uso |
| Usa la biblioteca auth0.js y el token de identidad de la cuenta principal para crear una instancia de `auth0.Management`.                                                                                                                 | Afectado          |
| Usa la biblioteca auth0.js y un token de acceso (con scope `update:users`) para crear una instancia de `auth0.Management`.                                                                                                                | No afectado       |

<div id="actions">
  ## Actions
</div>

Revise todas las llamadas a la [endpoint de Identities](https://auth0.com/docs/api/management/v2/#!/Users/post_identities) para la vinculación de cuentas y actualice aquellas que usan el flujo vulnerable descrito anteriormente. Puede actualizar sus llamadas de cualquiera de las siguientes maneras:

* **Escenarios de vinculación iniciados por el usuario / del lado del cliente:** En los escenarios de vinculación del lado del cliente, haga la llamada a la endpoint de Identities con un token de acceso que tenga el scope `update:current_user_identities` y proporcione el token de identidad de la cuenta secundaria en el payload (`link_with`). Este token de identidad debe obtenerse mediante un flujo compatible con <Tooltip tip="OAuth 2.0: Marco de autorización que define protocolos y flujos de trabajo de autorización." cta="Ver glosario" href="/es/docs/glossary?term=OAuth">OAuth</Tooltip>/OIDC.
* **Escenarios de vinculación del lado del servidor**: Para los escenarios de vinculación del lado del servidor, haga la llamada a la endpoint de Identities con un token de acceso que tenga el scope `update:users` y proporcione el `user_id` de la cuenta secundaria en el payload.

Consulte [Vincular cuentas de usuario](/es/docs/manage-users/user-accounts/user-account-linking/link-user-accounts) para obtener más detalles.

<div id="link-user-accounts">
  ### Vincular cuentas de usuario
</div>

Para vincular cuentas de usuario, puedes llamar al [endpoint Link a User Account](https://auth0.com/docs/api/management/v2#!/Users/post_identities) de la <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/es/docs/glossary?term=Management+API">Management API</Tooltip> o usar la [biblioteca auth0.js](/es/docs/libraries/auth0js).

<div id="link-current-user-accounts-with-the-management-api">
  #### Vincular las cuentas del usuario actual con la Management API
</div>

Un caso de uso común es permitir que el usuario que ha iniciado sesión vincule sus cuentas desde tu aplicación.

Antes de la obsolescencia, podías usar el token de identidad o el token de acceso del usuario principal (que contenía el `scope` `update:current_user_identities`) para autenticarte con la Management API y usar el [endpoint Link a User Account](https://auth0.com/docs/api/management/v2#!/Users/post_identities).

Ahora debes obtener un token de acceso (que contenga el `scope` `update:current_user_identities`) y usarlo para autenticarte con la API y usar el endpoint Link a User Account. El payload debe ser el token de identidad del usuario secundario.

1. Obtén un token de acceso con el `scope` `update:current_user_identities`, como se muestra en el siguiente ejemplo. El ejemplo usa el [flujo implícito](/es/docs/get-started/authentication-and-authorization-flow/implicit-flow-with-form-post); sin embargo, puedes [obtener tokens de acceso](/es/docs/secure/tokens/access-tokens/get-access-tokens) para cualquier tipo de aplicación.

<div id="link-any-user-account-with-the-management-api">
  #### Vincula cualquier cuenta de usuario con la Management API
</div>

Si obtienes un token de acceso para la vinculación de cuentas que contiene el `scope` `update:users` y envías el `user_id` y el `provider` de la cuenta secundaria en la solicitud, no tienes que hacer ningún cambio.

<div id="security-considerations">
  ## Consideraciones de seguridad
</div>

Hemos identificado una debilidad en un flujo concreto de vinculación de cuentas que podría permitir su uso indebido en determinadas circunstancias. No hemos encontrado indicios de que se haya utilizado de forma maliciosa, pero hemos decidido deprecar el flujo para evitar que eso pueda suceder.

Por lo tanto, Auth0 exige a los clientes que utilizan el flujo de vinculación de cuentas afectado que migren a una implementación más segura antes del 19 de octubre de 2018. En esta guía se indican las rutas de migración, que no deberían implicar ninguna pérdida de funcionalidad.

A partir del 19 de octubre de 2018, el flujo de vinculación de cuentas afectado se deshabilitará y se producirán errores en tiempo de ejecución.

Te afecta si llamas al [endpoint Post Identities](https://auth0.com/docs/api/management/v2#!/Users/post_identities) usando un token (ID o token de acceso) con el scope `update:current_user_identities` en el header Authorization e incluyes el `user_id` de la cuenta secundaria en el payload. Ningún otro caso de uso se ve afectado.
