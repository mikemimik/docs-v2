A partir del 28 de julio de 2022, Auth0 permitirá agregar claims personalizados privados sin espacio de nombres a los tokens de acceso y a los <Tooltip tip="token de ID: credencial destinada al propio cliente, en lugar de a acceder a un recurso." cta="Ver glosario" href="/es/docs/glossary?term=ID+tokens">tokens de ID</Tooltip>. Estos mismos claims también se agregarán a la respuesta del [endpoint `/userinfo`](https://auth0.com/docs/api/authentication#get-user-info). Para obtener más información sobre los tipos de claims en <Tooltip tip="token de ID: credencial destinada al propio cliente, en lugar de a acceder a un recurso." cta="Ver glosario" href="/es/docs/glossary?term=JWT">JWT</Tooltip>, consulta [Claims de JSON Web Token](/es/docs/secure/tokens/json-web-tokens/json-web-token-claims).

<Warning>
  Aunque se permite el uso de claims personalizados privados sin espacio de nombres, Auth0 recomienda encarecidamente usar claims personalizados públicos con espacio de nombres siempre que sea posible. Los claims personalizados públicos con espacio de nombres son la mejor manera de evitar conflictos con futuros claims añadidos al estándar.
</Warning>

<div id="example">
  #### Ejemplo
</div>

Anteriormente, Auth0 solo permitía claims con espacio de nombres en los tokens de acceso y de ID. Con la migración a claims personalizadas, ahora se pueden usar claims sin espacio de nombres en los <Tooltip tip="Token de acceso: credencial de autorización, en forma de una cadena opaca o JWT, que se utiliza para acceder a una API." cta="Ver glosario" href="/es/docs/glossary?term=access+tokens">tokens de acceso</Tooltip>, los tokens de ID y el endpoint `/userinfo` de Authentication API de Auth0.

```javascript lines
// una Action de Auth0 
exports.onExecutePostLogin = async (event, api) => {

  // claims Custom públicos con espacio de nombres 
  api.accessToken.setCustomClaim('https://myDomain.com/myClaim', 'this is a public, namespaced claim');
  api.idToken.setCustomClaim('https://myDomain.com/myClaim', 'this is a public, namespaced claim');

  // claims Custom sin espacio de nombres
  api.accessToken.setCustomClaim('myClaim', 'this is a private, non namespaced claim');
  api.idToken.setCustomClaim('myClaim', 'this is a private, non namespaced claim');
};
```

<div id="affected-flows">
  ## Flujos afectados
</div>

Todos los flujos de <Tooltip tip="OpenID: estándar abierto de autenticación que permite a las aplicaciones verificar la identidad de los usuarios sin recopilar ni almacenar información de inicio de sesión." cta="Ver glosario" href="/es/docs/glossary?term=OpenID">OpenID</Tooltip> Connect (OIDC) compatibles con Auth0 se ven afectados por esta migración. Para consultar la lista de flujos, lee [Authentication and Authorization Flows](/es/docs/get-started/authentication-and-authorization-flow).

Las siguientes funciones también se ven afectadas:

* [Inicio de sesión social nativo](/es/docs/get-started/authentication-and-authorization-flow)
* [tokens de actualización](/es/docs/secure/tokens/refresh-tokens)

Las siguientes funciones se ven afectadas solo cuando se usan junto con [Auth0 Rules](/es/docs/customize/rules) y el mapeo de atributos:

* [Protocolo de federación de servicios web (WS-Fed)](/es/docs/authenticate/protocols/ws-fed-protocol)
* [complemento SAML2 para aplicaciones web](/es/docs/authenticate/protocols/saml/saml-sso-integrations/enable-saml2-web-app-addon)

  ## Restricciones
</div>

<div id="maximum-token-size">
  ### Tamaño máximo del token
</div>

Auth0 restringirá el payload de claims personalizados a un máximo de 100KB. Es importante asegurarse de que el payload no supere este límite; de lo contrario, la transacción de autenticación fallará con un error. Te recomendamos revisar tu uso de código de extensibilidad (es decir, [Rules](/es/docs/customize/rules), [Hooks](/es/docs/customize/hooks) o [Actions](/es/docs/customize/actions)). En particular, revisa los payloads grandes de APIs externas.

Para evitar errores, Auth0 recomienda usar el payload de token más pequeño necesario para que tu aplicación funcione. Puede que necesites eliminar las propiedades que no sean esenciales antes de establecer el valor del claim personalizado.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Esta restricción se aplica al tamaño total del payload de todos tus claims personalizados. Esto incluye los nombres de los claims personalizados, así como los valores asociados, ya sean públicos con espacio de nombres o privados sin espacio de nombres.
</Callout>

El límite de 100KB se aplica por separado a los tokens de acceso y a los token de ID. Por ejemplo, en la misma transacción pueden devolverse un token de acceso de 100KB y un token de ID de 100KB.

<div id="examples">
  #### Ejemplos
</div>

```javascript lines
// una Action de Auth0 
exports.onExecutePostLogin = async (event, api) => {

  // obteniendo un payload superior a 100KB
  const aHeavyPayload = getHeavyPayload();

  // esto fallará la autenticación
  api.idToken.setCustomClaim('myclaim', aHeavyPayload);

};
```

```javascript lines
// una Action de Auth0 
exports.onExecutePostLogin = async (event, api) => {

  // obteniendo un payload de 50KB
  const a50KBPayload = getHeavyPayload();

  // obteniendo otro payload de 50KB
  const another50KBPayload = getHeavyPayload();

  // esto fallará la autenticación
  api.idToken.setCustomClaim('myclaim', a50KBPayload);
  api.idToken.setCustomClaim('https://myDomain.com/myClaim', another50KBPayload);

};
```

```javascript lines
// una Action de Auth0 
exports.onExecutePostLogin = async (event, api) => {

  // obteniendo un payload de 50KB
  const a50KBPayload = getHeavyPayload();

  // obteniendo otro payload de 50KB
  const another50KBPayload = getHeavyPayload();

  // esto se ejecutará correctamente
  api.accessToken.setCustomClaim('myclaim', a50KBPayload);
  api.idToken.setCustomClaim('https://myDomain.com/myClaim', another50KBPayload);

};
```

<div id="restricted-token-audience">
  ### Audiencia restringida del token
</div>

Auth0 restringirá la creación de claims personalizadas privadas sin espacio de nombres en los tokens de acceso cuya <Tooltip tip="Audiencia: Identificador único de la audiencia de un token emitido. Se denomina aud en un token y su valor contiene el ID de una aplicación (ID de cliente) para un token de ID o de una API (identificador de API) para un token de acceso." cta="Ver glosario" href="/es/docs/glossary?term=audience">audiencia</Tooltip> sea la Auth0 API. Se ignorará cualquier intento de establecer una claim personalizada privada sin espacio de nombres en un token de acceso cuya audiencia sea la Auth0 API. La transacción no fallará, pero la claim no se agregará a tu token. Auth0 recomienda no establecer claims personalizadas en tokens que vayan a consumir las API de Auth0.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  * Los tokens de ID no se ven afectados por esta restricción.
  * Las claims personalizadas públicas con espacio de nombres no se ve afectadas por esta restricción.
</Callout>

La siguiente audiencia restringirá la creación de claims personalizadas privadas sin espacio de nombres:

* `https://YOUR_TENANT.auth0.com/api` o `https://YOUR_TENANT.auth0app.com/api`
* `https://YOUR_TENANT.auth0.com/api/v2` o `https://YOUR_TENANT.auth0app.com/api/v2`
* `https://YOUR_TENANT.auth0.com/mfa` o `https://YOUR_TENANT.auth0app.com/mfa`

La excepción a esta restricción es la audiencia `/userinfo` de Auth0. Se permiten claims personalizadas privadas sin espacio de nombres en la siguiente audiencia:

* `https://YOUR_TENANT.auth0.com/userinfo`
* `https://YOUR_TENANT.auth0app.com/userinfo`

<div id="examples">
  #### Ejemplos
</div>

```javascript lines
// una Action de Auth0 
exports.onExecutePostLogin = async (event, api) => {

  // estos serán ignorados si el audience es un audience de Auth0
  api.accessToken.setCustomClaim('myATclaim', 'this is a claim');
  api.accessToken.setCustomClaim('https://myDomain.com/myATclaim', 'this is a claim');

  // estos se ejecutarán correctamente, no se ven afectados por la restricción de audience
  api.idToken.setCustomClaim('myIdTclaim', 'this is a claim');
  api.idToken.setCustomClaim('https://myDomain.com/myIdTclaim', 'this is a claim');

};
```

<div id="restriction-on-auth0-and-webtask-namespaces">
  ## Restricción sobre los espacios de nombres de Auth0 y Webtask
</div>

Auth0 restringirá la creación de `claims` personalizados con espacio de nombres que usen un dominio de Auth0 como identificador de espacio de nombres. Los dominios de Auth0 son:

* auth0.com
* webtask.io
* webtask.run

Se ignorará cualquier intento de establecer un `claim` personalizado con espacio de nombres en un token usando como identificador uno de los dominios anteriores. La transacción no fallará, pero el `claim` no se agregará a tu token.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Antes de esta migración, establecer un `claim` personalizado con espacio de nombres con un identificador de dominio de Auth0 hacía que el `claim` apareciera en la respuesta de `/userinfo`. Este comportamiento desaparece después de la migración y esos `claims` personalizados se ignoran por completo.
</Callout>

```javascript lines
// una Action de Auth0 
exports.onExecutePostLogin = async (event, api) => {

  // ninguno de estos se agregará a los tokens ni a la respuesta de /userinfo
  api.idToken.setCustomClaim('https://example.auth0.com', 'this is a claim');
  api.idToken.setCustomClaim('https://example.webtask.io', 'this is a claim');
  api.idToken.setCustomClaim('https://example.webtask.run', 'this is a claim');

};
```

<div id="oidc-user-profile-claims">
  ### Claims del perfil de usuario de OIDC
</div>

Auth0 ahora permite agregar claims del perfil de usuario de OIDC a los tokens de acceso.

Antes de esta migración, los intentos de agregar claims del perfil de usuario de OIDC al token de acceso se ignoraban sin mostrar ningún aviso. Con este comportamiento actualizado, los tokens de acceso contendrán estos claims del perfil de usuario de OIDC.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Si agrega claims del perfil de usuario de OIDC a los tokens de acceso, se aplican las mismas restricciones de `scope` que a los tokens de ID. Por ejemplo, para agregar el claim `email` a los tokens de acceso, el flujo debe activarse con un `scope` que contenga `email`.
</Callout>

Puede agregar los siguientes claims del perfil de usuario de OIDC a los tokens de acceso:

* `address`
* `birthdate`
* `email`
* `email_verified`
* `family_name`
* `gender`
* `given_name`
* `locale`
* `middle_name`
* `name`
* `nickname`
* `phone_number`
* `phone_number_verified`
* `picture`
* `preferred_username`
* `profile`
* `updated_at`
* `website`
* `zoneinfo`

<div id="example">
  #### Ejemplo
</div>

```javascript lines
// una Action de Auth0 
exports.onExecutePostLogin = async (event, api) => {

  // esto se ignoraba hasta ahora. A partir de esta migración, el claim se añadirá a los token de acceso 
  // si el scope contiene 'email'
  api.accessToken.setCustomClaim('email', 'myemail@domin.com');

  // esto se ignoraba hasta ahora. A partir de esta migración, el claim se añadirá a los token de acceso 
  // si el scope contiene 'profile'
  api.accessToken.setCustomClaim('family_name', 'A family name');

};
```

<div id="saml2-add-on-and-web-service-federation-protocol-ws-fed-attribute-mapping-with-auth0-rules">
  ### Mapeo de atributos con complemento SAML2 y el protocolo Web Service Federation (WS-Fed) mediante Auth0 Rules
</div>

Al igual que cuando se usan Auth0 Rules para realizar cambios en el objeto de usuario, los claims anteriores a la migración de `app_metadata` o `user_metadata` también fusionan su contenido cuando el claim se establece en el objeto `context.idToken` y los nombres entran en conflicto. Para obtener más información sobre las propiedades del objeto, consulta [Propiedades del objeto de usuario en Rules](/es/docs/customize/rules/user-object-in-rules).

Sin embargo, cuando se usan claims personalizados, Auth0 da prioridad al claim establecido en el objeto `context.idToken`.

Este cambio afecta a las Auth0 Rules que establecen `app_metadata` y `user_metadata` mediante `context.id_token` (asignándoles objetos) y, al mismo tiempo, usan estos campos en el mapeo de atributos para el add-on de <Tooltip tip="Security Assertion Markup Language (SAML): Protocolo estandarizado que permite a dos partes intercambiar información de autenticación sin una contraseña." cta="Ver glosario" href="/es/docs/glossary?term=SAML">SAML</Tooltip> o el protocolo <Tooltip tip="Web Service Federation (WS-Fed): Protocolo para gestionar identidades de usuario entre dominios." cta="Ver glosario" href="/es/docs/glossary?term=Web+Service+Federation">Web Service Federation</Tooltip> (WS-Fed).

Ejemplo 1: Auth0 ignora el mapeo de atributos cuando `context.idToken.app_metadata` se establece como un objeto vacío.

```javascript lines
// un Rule de Auth0
function (user, context, callback) {

  user.app_metadata.a_claim = 'This is a claim';
  user.app_metadata.another_claim = 'This is a another claim';

  context.samlConfiguration = context.samlConfiguration || {};

  context.samlConfiguration.mappings = {
    "a_claim": "app_metadata.a_claim",
    "another_claim": "app_metadata.another_claim"
  };

  context.idToken.app_metadata = {};

  return callback(null, user, context);
}
```

Respuesta de SAML antes de esta migración:

```xml lines
<samlp:Response>
    (...)
    <saml:Assertion>
        (...)
        <saml:AttributeStatement xmlns:xs="http://www.w3.org/2001/XMLSchema" xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance">
            <saml:Attribute Name="a_claim" NameFormat="urn:oasis:names:tc:SAML:2.0:attrname-format:basic">
                <saml:AttributeValue xsi:type="xs:string">
                    This is a claim
                </saml:AttributeValue>
            </saml:Attribute>
            <saml:Attribute Name="another_claim" NameFormat="urn:oasis:names:tc:SAML:2.0:attrname-format:basic">
                <saml:AttributeValue xsi:type="xs:string">
                    This is a another claim
                </saml:AttributeValue>
            </saml:Attribute>
        </saml:AttributeStatement>
    </saml:Assertion>
</samlp:Response>
```

Respuesta SAML con el comportamiento mejorado:

```xml lines

    (...)
    <saml:Assertion>
        (...)
        <saml:AttributeStatement xmlns:xs="http://www.w3.org/2001/XMLSchema" xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"/>
    </saml:Assertion>
</samlp:Response>
```

<div id="add-private-non-namespace-claims-to-tokens">
  ### Agrega claims privadas sin espacio de nombres a los tokens
</div>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  El comportamiento de las claims personalizadas no cambiará para los miembros del programa beta de claims personalizadas. Esta función ya está habilitada.
</Callout>

Ahora puedes agregar claims personalizadas privadas sin espacio de nombres al payload de los tokens de acceso y de ID.

<div id="example">
  #### Ejemplo
</div>

```javascript lines
// una Action de Auth0 
exports.onExecutePostLogin = async (event, api) => {

  // ignorado anteriormente
  // A partir de esta migración, el claim se agregará a los token de acceso
  api.accessToken.setCustomClaim('myATclaim', 'this is a claim');

  // ignorado anteriormente
  // A partir de esta migración, el claim se agregará a los tokens de ID
  api.idToken.setCustomClaim('myIdTclaim', 'this is a claim');

};
```

</div>

<div id="review-tenant-logs">
  ### Revisa los registros del tenant
</div>

Primero, revisa los registros del tenant para identificar avisos de obsolescencia y determinar si tu tenant está afectado por la migración.

1. Ve a Auth0 [Dashboard &gt; Monitoreo &gt; Registros](https://manage.auth0.com/#/logs).
2. Busca en los registros `type: depnote AND description: *Custom*claims*`.

<div id="fix-auth0-rules-for-saml2-add-on-and-web-service-federation-protocol-ws-fed">
  ### Corregir Auth0 Rules para el complemento SAML2 y protocolo Web Service Federation (Ws-Fed)
</div>

Si estableces claims `app_metadata` o `user_metadata` en el objeto `context.idToken` mediante el complemento SAML2 o protocolo Web Service Federation (Ws-Fed) con Auth0 Rules junto con el mapeo de atributos, tendrás que actualizar tu configuración para adaptarla a la forma en que Auth0 evalúa los nombres de claims en conflicto entre estos objetos. Hay varias soluciones posibles:

* Asegúrate de que el código de tu Rule de Auth0 siempre dé prioridad al contenido de los objetos establecidos en `context.id_token`:

  ```javascript lines
  // my_claim se ignorará, esta línea de código ya no es relevante,
  // es preferible establecer my_claim en `context.idToken`
  user.app_metadata.my_claim = 'a value'; 

  // esta versión de app_metadata tendrá prioridad sobre cualquier otro cambio 
  context.idToken.app_metadata = {
    another_claim: 'another value'
  };

  // Solo `another_claim` aparecerá en las respuestas SAML/WsFed
  ```

* Si estás usando el mapeo de atributos del complemento SAML2 o de protocolo Web Service Federation (Ws-Fed), evita establecer claims `app_metadata` o `user_metadata` en el objeto `context.idToken`. Sustituye estos claims por claims con espacio de nombres siempre que sea posible:

  ```js lines
  context.idToken['https://mydomain.com/app_metadata'] = {
    my_claim: 'my claim'
  };
  ```

* Usa una condición basada en el protocolo actual o en el cliente actual para excluir las instrucciones que establecen `app_metadata` o `user_metadata` cuando el protocolo sea `samlp` o `wsfed`.

  ```js lines
  if (!['samlp', 'wsfed'].includes(context.protocol)) {
      context.idToken.app_metadata = {
        claim_set_via_id_token: "This is a claim which was set via context.idToken"
      };
  }
  ```

<div id="disable-legacy-behavior">
  ### Deshabilitar el comportamiento heredado
</div>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Antes de deshabilitar el comportamiento heredado, te recomendamos revisar la lista de cambios y verificar que tus aplicaciones e integraciones sean compatibles.
</Callout>

<Warning>
  Si tu tenant no tiene la opción del interruptor, no se ve afectado y no es necesario realizar ninguna otra acción.
</Warning>

1. Ve a [Auth0 Dashboard &gt; Configuración del tenant &gt; Advanced](https://manage.auth0.com/dashboard/#/tenant/advanced) y busca **Migración**.
2. Usa el interruptor para deshabilitar **Los claims personalizados deben usar un espacio de nombres**.
