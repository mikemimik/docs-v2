---
description: Describe las funcionalidades de Auth0 afectadas por la migración de Node.js v8 a Node.js v12 y ofrece recomendaciones para la migración.
title: Migrar de Node.js 8 a Node.js 12
---

A partir del 31 de diciembre de 2019, [Node.js v8 dejó de tener soporte a largo plazo (LTS)](https://github.com/nodejs/Release#release-schedule), lo que significa que el equipo de desarrollo de Node.js ya no incorpora correcciones críticas de seguridad a esta versión. Esto podría exponer su código de extensibilidad a vulnerabilidades de seguridad. Por lo tanto, Auth0 migró de Node 8 a Node 12.

<div id="features-affected">
  ## Funciones afectadas
</div>

Las siguientes funciones de Auth0 utilizan Node 8:

* Reglas
* Hooks
* conexiones de base de datos personalizadas
* conexiones sociales personalizadas
* Extensions

Si no utilizas ninguna de las funciones de extensibilidad mencionadas anteriormente, esta migración no te afecta.

<div id="tasks">
  ## Tareas
</div>

Como parte del proceso de incorporación de Node 12 a nuestro runtime de Webtask, realizamos una serie de pruebas para determinar qué módulos no son compatibles al pasar de Node 8 a Node 12. La mayoría de los clientes deberían poder actualizar a Node 12 sin problemas.

Dicho esto, antes de migrar, le recomendamos encarecidamente que pruebe todos sus:

* Reglas
* Hooks
* conexiones de base de datos personalizadas y scripts
* conexiones sociales personalizadas
* Extensions

Auth0 recomienda que primero cambie su tenant de desarrollo al runtime de Node 12, complete las pruebas en ese tenant y migre su tenant de producción solo cuando no detecte problemas en desarrollo.

<div id="enable-node-12-runtime">
  ### Habilitar el runtime de Node 12
</div>

Como resultado de esta migración, puede haber cambios adicionales en el comportamiento, por lo que hemos proporcionado una opción de migración que le permite controlar la migración de su entorno al nuevo runtime de Webtask con Node 12.

Auth0 recomienda que primero cambie su tenant de desarrollo al runtime de Node 12, complete las pruebas en ese tenant de desarrollo y migre su tenant de producción solo cuando compruebe que no hay problemas en desarrollo.

Puede consultar la [Management API](/es/docs/api) para ver sus reglas, hooks, scripts de base de datos personalizada y conexiones sociales personalizadas. Esto le facilitará mover elementos de su tenant de producción al tenant de desarrollo con fines de prueba.

Al usar los endpoints de Connections en la <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/es/docs/glossary?term=Management+API">Management API</Tooltip>, los scripts de base de datos personalizada pueden recuperarse o actualizarse con `options.customScripts`. Del mismo modo, puede encontrar las conexiones sociales personalizadas en `options.scripts.fetchUserProfile`.

1. Habilite Node 12 en su tenant de desarrollo mediante el nuevo panel de **extensibilidad** en la página [Configuración avanzada del tenant](https://manage.auth0.com/#/tenant/advanced) del Dashboard. Elija **Node 12** en el menú desplegable **Runtime**.
2. Haga clic en **Guardar**.
3. Si usa los elementos que se enumeran a continuación, complete los pasos de migración correspondientes para cada uno.
4. Pruebe su configuración.
5. Cuando esté seguro de que nada se ha roto, use los pasos 1 y 2 anteriores para habilitar Node 12 en su tenant de producción.

<div id="allowlist-new-urls">
  ### Agregar nuevas URL a la lista de permitidos
</div>

La Extensión de Administración delegada y la extensión Dashboard de <Tooltip tip="Single Sign-On (SSO): servicio que, después de que un usuario inicia sesión en una aplicación, inicia automáticamente la sesión de ese usuario en otras aplicaciones." cta="Ver glosario" href="/es/docs/glossary?term=Single+Sign-on">inicio de sesión único</Tooltip> (SSO) requieren agregar a la lista de permitidos las URL que se usan para acceder a extensiones y Webtasks personalizados. Cuando actualices a Node 12, cambiarán las URL que usas para acceder a extensiones y Webtasks personalizados. Este es un cambio incompatible para estas extensiones.

Si usas alguna de estas extensiones, debes agregar las nuevas URL a la lista de permitidos tanto en Allowed Callback URLs como en Allowed Logout URLs.

La parte de la región de la URL cambiará de 8 a 12. Si accedes a una extensión con la URL:

`https://{yourTenant}.us8.webtask.io/dummy-extension-url`

cuando actualices a Node 12, la URL será:

`https://{yourTenant}.us12.webtask.io/dummy-extension-url`

1. Ve a [Dashboard &gt; Applications &gt; Applications &gt; Settings](https://manage.auth0.com/#/applications/\{yourClientId}/settings) y agrega la URL a los campos **Allowed Callback URLs** y **Allowed Logout URLs**.
2. Las URL de ejecución también cambiarán para los Webtasks personalizados en tu contenedor de Auth0. Debes actualizar cualquier aplicación externa que llame a esos Webtasks.

<div id="republish-authorization-extension-rule">
  ### Volver a publicar la rule de Authorization Extension
</div>

Si usas Authorization Extension, se genera una rule `auth0-authorization-extension`. Vuelve a publicar esta rule desde Authorization Extension para actualizar las URL automáticamente.

1. Asegúrate de haber actualizado Authorization Extension a la versión más reciente desde la pestaña **Installed Extensions**. Si aparece el botón **Upgrade**, haz clic en él para actualizar. Si no aparece, ya tienes la versión más reciente de la extensión.
2. Abre la página de configuración de **Authorization Extension**.
3. Para actualizar la URL en la rule, vuelve a publicarla haciendo clic en el botón **Publish Rule**.
4. Haz una prueba para asegurarte de que todo siga funcionando. Si ves un error de API key no válida después de la actualización, haz clic en el botón **Rotate** para generar una nueva API key.

<div id="update-extensions">
  ### Actualizar extensiones
</div>

La mayoría de las extensiones usan el Secret oculto `PUBLIC_WT_URL` para la autorización. Este Secret depende de la versión del runtime y no se actualiza automáticamente.

Para actualizarlo, debes guardar la configuración de la extensión (no hace falta realizar ningún cambio). Para ello, después de cambiar el runtime a `Node 12`, debes abrir la configuración de la extensión en el Dashboard de extensiones (icono de engranaje) y hacer clic en `Save`. Después, la galería de extensiones actualizará el Secret `PUBLIC_WT_URL` en función del runtime seleccionado.

Si no actualizas el Secret oculto `PUBLIC_WT_URL`, recibirás el siguiente error:

<div id="update-pinned-modules">
  ### Actualizar módulos con versión fijada
</div>

Si está utilizando los siguientes módulos integrados (es decir, módulos que no incluyó explícitamente con `require`), tenga en cuenta que algunas versiones se actualizaron para que funcionen con Node 12. La siguiente tabla resume los cambios. Estas nuevas versiones deberían seguir siendo compatibles con versiones anteriores.

| Nombre del módulo | Versión anterior | Nueva versión |
| ----------------- | ---------------- | ------------- |
| couchbase         | ~2.5.1           | 2.6.10        |
| bcrypt            | 1.0.3            | 3.0.8         |

Si ha fijado manualmente la versión de algunos módulos, es posible que tenga que actualizarlos también manualmente para que su código se ejecute con Node 12.

Por ejemplo, debe cambiar:

`var bcrypt = require(‘bcrypt@1.0.3’);`

por

`var bcrypt = require(‘bcrypt’);`

o, si el módulo debe fijarse a una versión específica:

`var bcrypt = require(‘bcrypt@3.0.8’);`
