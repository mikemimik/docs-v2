Auth0 limita el uso de sus servicios para garantizar un rendimiento óptimo y protegerse frente a <Tooltip tip="Actores maliciosos: Entidad (una persona o grupo) que representa una amenaza para la empresa o el entorno con la intención de causar daño." cta="Ver glosario" href="/docs/es/glossary?term=bad+actors">actores maliciosos</Tooltip>, errores técnicos o tráfico legítimo excesivo.  Te recomendamos revisar cómo Auth0 aplica estos límites para configurar tu aplicación y ofrecer la mejor experiencia de usuario.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Para consultar los límites de tasa asociados a tu cuenta, consulta [Configuraciones de límites de tasa](/docs/es/troubleshoot/customer-support/operational-policies/rate-limit-policy/rate-limit-configurations), donde encontrarás una matriz con todas las políticas de límites de tasa.
</Callout>

<div id="introduction-to-rate-limits">
  ## Introducción a los límites de tasa
</div>

Auth0 establece límites para proteger los servicios frente a un exceso de solicitudes y a los clientes frente a interrupciones o degradación del servicio.

Auth0 supervisa y, en muchos casos, aplica una serie de límites, entre ellos:

* Solicitudes a entornos (solo Private Cloud)
* Solicitudes a tenants mediante la API o endpoints de API
* Otros límites

<div id="environment-request-limits-private-cloud-only">
  ### Límites de solicitudes del entorno (solo para Private Cloud)
</div>

En Private Cloud, los límites de solicitudes del entorno se basan en el nivel de rendimiento de Private Cloud. Para obtener más información, consulta [Private Cloud for AWS](/docs/es/deploy-monitor/deploy-private-cloud/private-cloud-on-aws) o [Private Cloud for Azure](/docs/es/deploy-monitor/deploy-private-cloud/private-cloud-on-azure).

Actualmente, los límites de tasa del entorno de Private Cloud representan la carga máxima bajo la cual el producto Auth0 cumplirá los SLA. Sin embargo, por ahora, Auth0 solo aplica estos límites y notifica a los clientes cuando se superan los límites de tasa en un tenant específico dentro del entorno. En la mayoría de los casos de uso de Private Cloud, en los que los clientes mantienen un único tenant de producción, esto no supone un problema; sin embargo, los casos de uso que aprovisionan más de un tenant de producción deben tener en cuenta la carga prevista en todos los tenants del entorno e implementar supervisión adicional según sea necesario.

| Característica                            | Authentication API                                                                                                                                                                                                                                                                                 | Todas las demás API (incluida Management API)                                                                                                                                                                                                                                     |
| ----------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Límites de tasa por tenant                | Todos los tenants consumen de un límite global compartido para toda la API en el entorno, pero el límite de tasa del entorno es un tope estricto. Superar el límite del entorno debido a la carga combinada de varios tenants hará que las solicitudes que excedan ese límite queden sujetas a limitación. | Los límites de tasa por tenant se aplican de forma independiente, y el límite de tasa del entorno actúa como un umbral de posible degradación del rendimiento e impacto en el SLA, pero **no** provoca directamente limitación a nivel de tenant.                                                 |
| Se supera el límite de tasa del entorno (ejemplo) | Con un límite del entorno de 1500 rps, un Tenant 1 a 1400 rps y un Tenant 2 a 900 rps (2300 rps en total) darán como resultado que 800 solicitudes queden sujetas a limitación.                                                                                                                    | Con un límite del entorno de 1500 rps, un Tenant 1 a 1400 rps y un Tenant 2 a 900 rps (2300 rps en total) **no** harán que se limiten tenants individuales, pero el entorno en su conjunto puede experimentar una degradación del rendimiento y posibles incumplimientos del SLA. |

<Warning>
  Superar un límite de tasa del entorno invalida el Acuerdo de Nivel de Servicio (SLA).
</Warning>

Para conocer las consideraciones sobre las pruebas de carga en Private Cloud, consulta [Private Cloud on AWS](/docs/es/deploy-monitor/deploy-private-cloud/private-cloud-on-aws) y [Private Cloud on Azure](/docs/es/deploy-monitor/deploy-private-cloud/private-cloud-on-azure).

<div id="tenant-request-limits">
  ### Límites de solicitudes por tenant
</div>

Auth0 limita la cantidad de solicitudes que se pueden realizar en un tenant. Estos límites se configuran en función de la API y, además, de endpoints específicos de cada API.

<div id="api-rate-limits">
  #### Límites de tasa de la API
</div>

Auth0 limita la cantidad de solicitudes a una API específica, independientemente del endpoint de la API.  Los límites de la API pueden variar según:

* API

  * Autenticación
  * Administración
* Tipo de tenant (Producción vs. Desarrollo o Staging)
* Nivel de suscripción (Free, Essential, Professional, Enterprise Public vs. Private)

Por ejemplo, un tenant gratuito que no sea de producción podría tener límites distintos de los de un tenant de producción con una suscripción de pago. Para ver configuraciones específicas de límites de tasa, consulta [Rate Limit Configurations](/docs/es/troubleshoot/customer-support/operational-policies/rate-limit-policy/rate-limit-configurations).

<div id="user-requests">
  ### Solicitudes de usuario
</div>

Una sola solicitud de un usuario final (p. ej., Login o Signup) normalmente inicia múltiples solicitudes a endpoints de Authentication API. La proporción real entre las solicitudes de usuarios finales y Authentication API depende de varios factores:

* Entidad que se está autenticando (p. ej., una máquina o una aplicación móvil o de escritorio del usuario final)
* Experiencia de autenticación (p. ej., <Tooltip tip="Universal Login: Tu aplicación redirige a Universal Login, alojado en el servidor de autorización de Auth0, para verificar la identidad de un usuario." cta="Ver glosario" href="/docs/es/glossary?term=Universal+Login">Universal Login</Tooltip> nuevo o Classic Login)
* Flujo de autenticación (p. ej., Login, Signup o cambiar la contraseña)
* Tipo de flujo de autenticación (p. ej., Login con nombre de usuario / contraseña; Login mediante inicio de sesión Social; Login cuando ya existe un token de autenticación)

Los clientes que aprovechan extensibilidad podrían agregar aún más solicitudes, no solo a Authentication API, sino también a <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/docs/es/glossary?term=Management+API">Management API</Tooltip>, según la configuración de extensibilidad.
Consulta [Rate Limit Use Cases](/docs/es/troubleshoot/customer-support/operational-policies/rate-limit-policy/rate-limit-use-cases) para obtener orientación sobre cómo estimar de qué manera la configuración de Auth0 puede afectar el uso de tu API.

<div id="endpoint-rate-limits">
  #### Límites de tasa de los endpoints
</div>

Auth0 limita la cantidad de solicitudes realizadas a los endpoints de API y, en algunos casos, la cantidad de operaciones de endpoint.  Los límites de los endpoints de API también varían según:

* API
* Tipo de tenant
* Nivel de suscripción

Por ejemplo, un tenant gratuito de no producción tiene límites distintos a los de un tenant de producción con una suscripción de pago.

<div id="tenant-request-limits-order">
  ### Orden de evaluación de los límites de solicitudes del tenant
</div>

A medida que se reciben solicitudes en tu tenant, Auth0 las evalúa primero con respecto al límite global de la API y luego con respecto al límite de solicitudes de endpoints específicos de la API.

<div id="miscellaneous-limits">
  ### Límites varios
</div>

<div id="database-login-limits">
  #### Límites de inicio de sesión para conexiones de base de datos
</div>

En las conexiones de base de datos, Auth0 limita ciertos tipos de intentos repetidos de inicio de sesión según la cuenta de usuario y la dirección IP. Para proteger la estabilidad general del sistema, Auth0 aplica límites de tasa para usuario/contraseña que ayudan a mitigar la carga. El alto grado de personalización de Auth0 puede aumentar el riesgo de degradación del servicio. Algunas causas pueden ser:

* Pruebas de estrés con alta carga
* Pruebas comparativas de rendimiento
* Código ineficiente que hace que los usuarios inicien sesión varias veces

Las solicitudes están sujetas a límites, tal como se describe en las políticas individuales de las API de Auth0.

Además, existe un límite de tasa de inicio de sesión para un mismo usuario: si una dirección IP realiza 20 intentos de inicio de sesión en un minuto sobre la misma cuenta de usuario, el límite entra en vigor. Después de eso, Auth0 permite al usuario 10 intentos por minuto. Cualquier combinación de intentos de inicio de sesión exitosos y fallidos cuenta para este límite.

<Card title="Límites que protegen a los usuarios">
  La [protección contra fuerza bruta](/docs/es/secure/attack-protection/brute-force-protection) y la [limitación de IP sospechosas](/docs/es/secure/attack-protection/suspicious-ip-throttling) de Auth0 también pueden limitar los inicios de sesión y los registros, pero son independientes de los límites de tasa. Para obtener más información sobre cómo Auth0 detecta y gestiona anomalías potencialmente maliciosas, consulta [Protección contra ataques](/docs/es/secure/attack-protection).
</Card>

<div id="sms-message-limits-for-multi-factor-authentication-end-users-only">
  #### Límites de mensajes SMS para la autenticación multifactor (solo usuarios finales)
</div>

Si intenta enviar más de 10 mensajes SMS a su dispositivo en el transcurso de una hora, recibirá un mensaje de error por haber superado el límite.

Cuando supere su límite de mensajes, deberá esperar al menos una hora desde la primera solicitud de mensaje antes de pedir otro. Recibirá un intento adicional por cada hora adicional que transcurra.

<div id="native-social-login-limits">
  #### Límites del inicio de sesión social nativo
</div>

Los límites aplicados a las solicitudes del flujo de inicio de sesión social nativo se determinan en función del cuerpo de la solicitud, según los siguientes criterios iniciales:

| Tipo de solicitud    | Cuerpo                                               |
| -------------------- | ---------------------------------------------------- |
| `grant_type`         | `urn:ietf:params:oauth:grant-type:token-exchange`    |
| `subject_token_type` | `http://auth0.com/oauth/token-type/apple-authz-code` |

<div id="public-performance-burst">
  #### Public Performance Burst
</div>

La oferta Public Performance Burst es un add-on disponible para suscripciones empresariales que mejora una implementación existente de Public Cloud. Esta oferta permite aumentar dinámicamente el límite de solicitudes de la Authentication API hasta un múltiplo del límite empresarial predeterminado de 100 RPS durante un máximo de 48 horas al mes.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Este add-on solo amplía los límites de solicitudes de la Authentication API y NO se aplica a la Management API ni a otros endpoints sujetos a límites de tasa fuera del alcance de la Authentication API.
</Callout>

Actualmente, hay tres modificadores de Public Performance Burst (2x, 3x y 4x) que permiten 200, 300 y 400 RPS, respectivamente, para la Authentication API durante un máximo de 48 horas al mes. Estas 48 horas se contabilizan y descuentan de la cuota mensual permitida en intervalos de 1 minuto, lo que permite usar la API en el nivel del multiplicador durante 2880 intervalos de 1 minuto cada mes.

Cuando el volumen de solicitudes de la Authentication API supera el valor predeterminado de 100 RPS, se descuenta un intervalo de 5 minutos de la asignación mensual y se permite tráfico a la tasa asociada con el multiplicador. El tráfico puede mantenerse dentro del rango del multiplicador durante 5 minutos consecutivos completos a partir de ese momento sin que se apliquen descuentos adicionales.

Es posible supervisar los eventos de descuento de intervalos mediante los [logs del tenant](/docs/es/deploy-monitor/logs/log-event-type-codes). Cada descuento genera un evento de log del tenant asociado de tipo `appi` que contiene información sobre la asignación ya consumida y la restante.

Para obtener más información, consulta la sección de Authentication API en [Límites de tasa - Enterprise](/docs/es/troubleshoot/customer-support/operational-policies/rate-limit-policy/rate-limit-configurations/enterprise-public).

<div id="private-performance-burst">
  #### Private Performance Burst
</div>

La oferta Private Performance Burst (disponible actualmente en AWS para los niveles 30x y 60x) incluye una capacidad de rendimiento en ráfaga (pico) de hasta 30x (3,000 RPS) o 60x (6,000 RPS) durante un máximo de 80 horas al mes. La capacidad de rendimiento base, que es la mitad de la capacidad de rendimiento en ráfaga, está disponible durante el resto del mes.

En otras palabras, 30x Private Performance Burst tiene:

* Capacidad base: 1,500 RPS durante todo el mes
* Capacidad de ráfaga/pico: 3,000 RPS durante un máximo de 80 horas al mes

Si las transacciones de autenticación superan el límite base de solicitudes de API, se descuenta una hora de la asignación mensual. A partir de ese momento, el tráfico puede mantenerse a una tasa elevada durante toda esa hora consecutiva. Se aplican deducciones adicionales de la misma manera cada vez que se vuelve a superar el umbral.

Una vez agotada la asignación de 80 horas, al tráfico de autenticación se le aplicará el límite de tasa correspondiente a la capacidad base hasta que entre en vigor la nueva asignación mensual.

<div id="extensibility-concurrency-limits">
  #### Límites de concurrencia de extensibilidad
</div>

Para garantizar la disponibilidad del sistema y el uso equitativo de sus recursos, Auth0 limita la cantidad de solicitudes simultáneas en curso en todos los productos de extensibilidad: Actions, Hooks, Rules, Custom Database Connections, Extensions y conexiones OAuth2 personalizadas. Los tenants que superen sus límites de solicitudes simultáneas pueden esperar errores en las nuevas solicitudes hasta que se completen las que están en curso. Los límites de concurrencia se definen a continuación:

| Suscripción                                                           | Límite de concurrencia (por tenant) |
| --------------------------------------------------------------------- | ----------------------------------- |
| Public Cloud                                                          | 250                                 |
| Tier Dev Private Cloud                                                | 100                                 |
| Private Cloud Basic 100 RPS (1x)                                      | 200                                 |
| Private Cloud Performance 500 RPS (5x)                                | 400                                 |
| Private Cloud Performance 1500 RPS (15x)                              | 1200                                |
| Private Cloud Performance 3000 RPS (30x) y 3000 RPS Burst (30x Burst) | 1200                                |
| Private Cloud Performance 6000 RPS (60x) y 6000 RPS Burst (60x Burst) | 1200                                |

La concurrencia puede calcularse multiplicando los RPS esperados por la latencia de cada solicitud. Por ejemplo, un tenant que tiene dos Actions de Post Login asociadas, cada una con una duración de 250 ms, y un total de 400 RPS de inicio de sesión, tiene una concurrencia esperada de (2 * (400 requests / 1 second) * (.25 seconds / 1 request)) = 200.

Para asegurarte de que tu tenant no se vea afectado por estos límites de concurrencia, verifica que la lógica de extensibilidad que pueda tardar más en ejecutarse, como las llamadas a API externas, tenga tiempos de espera razonables.

<div id="rate-limit-algorithm">
  ## Algoritmo de limitación de tasa
</div>

Auth0 establece límites de tasa y límites de ráfaga para sus API. Mientras que el límite de tasa es la cantidad máxima de tráfico sostenido que el sistema permite de forma continua, el límite de ráfaga es el volumen máximo de tráfico a corto plazo que el sistema permite dentro de un intervalo de tiempo. Los límites de tasa y los límites de ráfaga de Auth0 funcionan conjuntamente para ofrecer una mejor limitación en volúmenes de tráfico dinámicos.

Auth0 utiliza un algoritmo de token bucket para los límites de tasa con las siguientes configuraciones:

* **Claves de límite:**

  * Normalmente, una clave de límite de tasa se basa en dos factores principales:

    * API y endpoint
    * Tipo de tenant
  * En algunos casos, los factores adicionales incluyen:

    * IP de origen
    * ID de usuario de destino
* **Valores de límite:**

  * **Tamaño del bucket:** El número máximo de solicitudes que una API o un endpoint recibe en general, o de un usuario o una dirección IP específicos, antes de que se agreguen nuevas solicitudes.
  * **Tasa de recarga:** La velocidad a la que se agregan nuevas solicitudes al bucket.

A partir de estos dos valores, Auth0 calcula el límite de ráfaga y el límite de tasa sostenido:

* **Límite de ráfaga:**  Igual al tamaño del bucket.
* **Límite de tasa sostenido:** Tasa de recarga en solicitudes por minuto o por segundo.

Si el límite de tasa sostenido se calcula en solicitudes por segundo, las nuevas solicitudes se agregan en milisegundos. Si el límite de tasa sostenido se calcula en solicitudes por minuto, las nuevas solicitudes se agregan por segundos.
