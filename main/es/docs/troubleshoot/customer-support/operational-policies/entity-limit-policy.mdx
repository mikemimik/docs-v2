---
description: Describe la política de límites de entidades del tenant de Auth0 para suscriptores.
title: Política de límites de entidades
---

Las entidades en Auth0 son elementos de configuración del tenant, como aplicaciones, conexiones, Organizations, API <Tooltip tip="Resource Server: Servidor que aloja recursos protegidos. Los servidores de recursos aceptan y responden a solicitudes de recursos protegidos." cta="Ver glosario" href="/es/docs/glossary?term=resource+servers">servidores de recursos</Tooltip>, permisos, Roles y Actions.

<Warning>
  Los límites de entidades son límites estrictos aplicados a las entidades para preservar la estabilidad de nuestra plataforma; no forman parte del precio, aunque pueden variar según el grupo de planes de precios. Los planes de precios individuales tienen limitaciones independientes, que se refieren a los beneficios incluidos, como MAU y conexiones empresariales, que recibes al suscribirte a un plan determinado. (Para conocer las limitaciones de tu plan de precios, consulta [Auth0 Pricing](https://auth0.com/pricing/).)
</Warning>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Cualquier uso del término &quot;ilimitado&quot; significa que no hay límites estrictos, pero Okta establece limitaciones del sistema para ese umbral. Ponte en contacto con soporte si tienes alguna pregunta sobre estos límites.
</Callout>

Los clientes con planes Enterprise pueden solicitar aumentos en los límites de entidades para Organizations por tenant y miembros de la organización por organización [poniéndose en contacto con soporte](https://support.auth0.com/).

Estos límites pueden aumentarse a 2,000,000 Organizations por tenant y 2,000,000 miembros de la organización por organización en la nube pública. En las instancias de nube privada, estas entidades son ilimitadas.

<div id="authorization-core-role-based-access-control-rbac">
  ### Authorization Core Role-Based Access Control (RBAC)
</div>

| Entity                                                    | Limit  |
| --------------------------------------------------------- | ------ |
| Roles por Tenant                                          | 1000   |
| Scopes por API (Resource Servers)                         | 1000   |
| Roles por usuario                                         | 50     |
| Roles por grupo empresarial                               | 10     |
| Roles únicas que se pueden asignar a grupos empresariales | 100    |

| Permisos por rol                                          | 1000   |

Los límites anteriores de grupos empresariales y Roles se aplican al usar RBAC sin la funcionalidad de Organizations. El uso de la funcionalidad de Organizations permite que estos límites se apliquen por Organización en lugar de por Tenant.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Ten en cuenta que las limitaciones de Permisos por usuario afectan a los Permisos asignados directamente.

  Técnicamente, un usuario podría tener más Permisos de los indicados si asignas los Permisos a diferentes Roles y luego asignas esos Roles al usuario.
</Callout>

| Entity                                           | Limit |
| ------------------------------------------------ | ----- |
| token de actualización por usuario y Application | 200   |

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Si alcanzas el límite de <Tooltip tip="Refresh Token: token utilizado para obtener un nuevo token de acceso sin obligar a los usuarios a volver a iniciar sesión." cta="Ver glosario" href="/es/docs/glossary?term=Refresh+tokens">tokens de actualización</Tooltip> y creas un nuevo token de actualización,
  el sistema revoca o elimina el token más antiguo de ese usuario en la
  Application.

  Los tokens revocados y los tokens vencidos no cuentan para el límite.
</Callout>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Los límites de Actions y de módulos de Actions incluyen tanto las entidades desplegadas como las no desplegadas. Cuando se eliminan, dejan de contabilizarse para ese límite específico.

  Los límites de las versiones de Action y de las versiones de módulos de Action incluyen tanto las versiones en borrador como las activas. Una vez que alcanzas el límite, cada versión adicional que agregas hace que Auth0 elimine la versión no utilizada más antigua.
</Callout>
