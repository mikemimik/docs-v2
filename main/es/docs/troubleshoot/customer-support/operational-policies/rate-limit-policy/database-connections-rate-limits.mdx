title: Política de límites de tasa
---

Auth0 limita el uso de los servicios para garantizar un rendimiento óptimo y la protección frente a <Tooltip tip="Actores maliciosos: Entidad (una persona o un grupo) que representa una amenaza para la empresa o el entorno con la intención de causar daño." cta="Ver glosario" href="/es/docs/glossary?term=bad+actors">actores maliciosos</Tooltip>, errores técnicos o tráfico legítimo excesivo. Recomendamos revisar cómo Auth0 aplica estos límites para configurar tu aplicación y ofrecer la mejor experiencia de usuario.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Para consultar los límites de tasa asociados a tu cuenta, consulta [Configuraciones de límites de tasa](/es/docs/troubleshoot/customer-support/operational-policies/rate-limit-policy/rate-limit-configurations), donde encontrarás una matriz con todas las políticas de límites de tasa.
</Callout>

<div id="introduction-to-rate-limits">
  ## Introducción a los límites de tasa
</div>

Auth0 aplica límites para proteger los servicios frente a un exceso de solicitudes y a los clientes frente a interrupciones o degradación del servicio.

Auth0 supervisa y, en muchos casos, aplica una serie de límites, entre ellos:

* Solicitudes a entornos (solo en Private Cloud)
* Solicitudes a tenants a través de la API o de un endpoint de API
* Límites varios

<div id="environment-request-limits-private-cloud-only">
  ### Límites de solicitudes del entorno (solo para Private Cloud)
</div>

En Private Cloud, los límites de solicitudes del entorno se basan en el Private Cloud Performance Tier. Para obtener más información, consulta [Private Cloud for AWS](/es/docs/deploy-monitor/deploy-private-cloud/private-cloud-on-aws) o [Private Cloud for Azure](/es/docs/deploy-monitor/deploy-private-cloud/private-cloud-on-azure).

Actualmente, los límites de solicitudes del entorno en Private Cloud representan la carga máxima bajo la cual el producto Auth0 cumplirá los SLA. Sin embargo, por el momento, Auth0 solo aplica estos límites y notifica a los clientes cuando se superan en un tenant específico dentro del entorno. En la mayoría de los casos de uso de Private Cloud, en los que los clientes mantienen un único tenant de producción, esto no supone un problema; sin embargo, los casos de uso que aprovisionan más de un tenant de producción deben tener en cuenta la carga esperada en todos los tenants del entorno e implementar monitoreo adicional según sea necesario.

| Funcionalidad                             | Authentication API                                                                                                                                                                                                                                                                      | Todas las demás API (incluida Management API)                                                                                                                                                                                                                                                     |
| ----------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Tenant Rate Limits                        | Todos los tenants consumen de un límite global compartido para toda la API a nivel de entorno, pero el límite del entorno es un tope estricto. Superar el límite del entorno debido a la carga combinada entre tenants hará que las solicitudes excedentes queden sujetas a limitación. | Los límites de los tenants se aplican de forma independiente, y el límite del entorno actúa como un umbral de posible degradación del rendimiento e impacto en el SLA, pero **no** provoca directamente la limitación de solicitudes a nivel de tenant.                                           |
| Environment Rate Limit Exceeded (Example) | Con un límite de entorno de 1500 rps, un Tenant 1 a 1400 rps y un Tenant 2 a 900 rps (2300 rps en total) harán que se limiten 800 solicitudes.                                                                                                                                          | Con un límite de entorno de 1500 rps, un Tenant 1 a 1400 rps y un Tenant 2 a 900 rps (2300 rps en total) **no** hará que se limite la tasa de solicitudes de tenants individuales, pero el entorno en general puede experimentar una degradación del rendimiento y posibles problemas con el SLA. |

<Warning>
  Superar el límite de solicitudes del entorno invalida el Acuerdo de Nivel de Servicio (SLA).
</Warning>

Para conocer las consideraciones sobre pruebas de carga en Private Cloud, consulta [Private Cloud on AWS](/es/docs/deploy-monitor/deploy-private-cloud/private-cloud-on-aws) y [Private Cloud on Azure](/es/docs/deploy-monitor/deploy-private-cloud/private-cloud-on-azure).

<div id="tenant-request-limits">
  ### Límites de solicitudes del tenant
</div>

Auth0 limita la cantidad de solicitudes que se pueden realizar en un tenant. Estos límites se configuran según la API y, dentro de cada una, según endpoints específicos.

<div id="api-rate-limits">
  #### Límites de tasa de la API
</div>

Auth0 limita el número de solicitudes a una API específica, independientemente del endpoint de la API. Los límites de la API pueden variar según:

* API

  * Autenticación
  * Administración
* Tipo de tenant (Producción vs. Desarrollo o Staging)
* Nivel de suscripción (Free, Essential, Professional, Enterprise Public vs. Private)

Por ejemplo, un tenant gratuito y de no producción podría tener límites distintos de los de un tenant de producción con una suscripción de pago. Para consultar configuraciones específicas de límites de tasa, consulta [Rate Limit Configurations](/es/docs/troubleshoot/customer-support/operational-policies/rate-limit-policy/rate-limit-configurations).

<div id="user-requests">
  ### Solicitudes de usuario
</div>

Una sola solicitud de usuario final (p. ej., Login o Signup) normalmente inicia varias solicitudes a los endpoints de Authentication API.  La proporción real entre las solicitudes de usuario final y las solicitudes a Authentication API depende de varios factores:

* Entidad que se autentica (p. ej., máquina o aplicación móvil o de escritorio del usuario final)
* Experiencia de autenticación (p. ej., <Tooltip tip="Universal Login: Su aplicación redirige a Universal Login, alojado en el servidor de autorización de Auth0, para verificar la identidad de un usuario." cta="Ver glosario" href="/es/docs/glossary?term=Universal+Login">nuevo Universal Login</Tooltip> o Classic Login)
* Flujo de autenticación (p. ej., Login, Signup o Change Password)
* Tipo de flujo de autenticación (p. ej., Login con nombre de usuario / contraseña; Login mediante Social Login; Login cuando ya existe un Authentication Token)

Los clientes que aprovechan la extensibilidad podrían agregar aún más solicitudes, no solo a Authentication API, sino también a la <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/es/docs/glossary?term=Management+API">Management API</Tooltip>, según la configuración de extensibilidad.
Consulte [Casos de uso de los límites de tasa](/es/docs/troubleshoot/customer-support/operational-policies/rate-limit-policy/rate-limit-use-cases) para obtener orientación sobre cómo calcular de qué manera su configuración de Auth0 puede afectar el uso de la API.

<div id="endpoint-rate-limits">
  #### Límites de tasa por endpoint
</div>

Auth0 limita la cantidad de solicitudes realizadas a los endpoints de la API y, en algunos casos, la cantidad de operaciones por endpoint.  Los límites de los endpoints de la API también varían según:

* API
* Tipo de tenant
* Nivel de suscripción

Por ejemplo, un tenant gratuito que no es de producción tiene límites distintos a los de un tenant de producción con una suscripción de pago.

<div id="tenant-request-limits-order">
  ### Orden de aplicación de los límites de solicitudes del tenant
</div>

A medida que se envían solicitudes a su tenant, Auth0 primero las evalúa según el límite global de la API y luego según el límite de tasa de endpoints específicos de la API.

<div id="miscellaneous-limits">
  ### Otros límites
</div>

<div id="database-login-limits">
  #### Límites de inicio de sesión para bases de datos
</div>

En el caso de las conexiones de base de datos, Auth0 limita ciertos tipos de intentos repetidos de inicio de sesión según la cuenta de usuario y la dirección IP. Para proteger la estabilidad general del sistema, Auth0 aplica límites de tasa para usuario/contraseña que mitigan la carga. El alto grado de personalización de Auth0 puede exponernos al riesgo de degradación del servicio. Algunas de las causas pueden ser:

* Pruebas de carga intensiva
* Pruebas comparativas
* Código ineficiente que hace que los usuarios inicien sesión varias veces

Las solicitudes están sujetas a límites, tal como se describe en las políticas individuales de las API de Auth0.

Además, existe un límite de tasa para el inicio de sesión de un mismo usuario: si una dirección IP realiza 20 intentos de inicio de sesión en un minuto en la misma cuenta de usuario, el límite entra en vigor. A partir de ese momento, Auth0 permite al usuario 10 intentos por minuto. Para este límite, cuenta cualquier combinación de intentos de inicio de sesión exitosos y fallidos.

<Card title="Límites que protegen a los usuarios">
  La [protección contra ataques de fuerza bruta](/es/docs/secure/attack-protection/brute-force-protection) de Auth0 y la [limitación de IP sospechosas](/es/docs/secure/attack-protection/suspicious-ip-throttling) también pueden limitar los inicios de sesión y los registros, pero son independientes de los límites de tasa. Para obtener más información sobre cómo Auth0 detecta y gestiona anomalías potencialmente maliciosas, consulta [Protección contra ataques](/es/docs/secure/attack-protection).
</Card>

<div id="sms-message-limits-for-multi-factor-authentication-end-users-only">
  #### Límites de mensajes SMS para la autenticación multifactor (solo para usuarios finales)
</div>

Si intentas enviar más de 10 mensajes SMS a tu dispositivo en una hora, recibirás un mensaje de error por exceder el límite de tasa.

Cuando superes el límite de envío de mensajes, deberás esperar al menos una hora desde la primera solicitud de mensaje antes de solicitar otro. Se te permitirá un intento adicional por cada hora adicional que transcurra.

<div id="native-social-login-limits">
  #### Límites del inicio de sesión social nativo
</div>

Los límites aplicados a las solicitudes del flujo de inicio de sesión social nativo se identifican según el cuerpo de las solicitudes con los siguientes criterios iniciales:

| Tipo de solicitud    | Cuerpo                                               |
| -------------------- | ---------------------------------------------------- |
| `grant_type`         | `urn:ietf:params:oauth:grant-type:token-exchange`    |
| `subject_token_type` | `http://auth0.com/oauth/token-type/apple-authz-code` |

<div id="public-performance-burst">
  #### Public Performance Burst
</div>

La oferta Public Performance Burst es un complemento disponible para las suscripciones Enterprise que mejora una implementación existente de Public Cloud. Esta oferta permite aumentar dinámicamente el límite de solicitudes de la Authentication API hasta un multiplicador del límite de solicitudes Enterprise predeterminado de 100 RPS durante un máximo de 48 horas al mes.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Este complemento solo amplía los límites de solicitudes de la Authentication API y NO se aplica a la Management API ni a otros endpoints sujetos a rate limits fuera del alcance de la Authentication API.
</Callout>

Actualmente, hay tres modificadores de Public Performance Burst (2x, 3x y 4x) que permiten 200, 300 y 400 RPS, respectivamente, para la Authentication API durante un máximo de 48 horas al mes. Las 48 horas se contabilizan y descuentan de la cuota mensual permitida en intervalos de 5 minutos, lo que permite usar la API en el nivel del multiplicador durante 576 intervalos de 5 minutos cada mes.

Cuando el volumen de solicitudes de la Authentication API supera el valor predeterminado de 100 RPS, se descuenta un intervalo de 5 minutos de la cuota mensual y se permite tráfico a la tasa asociada al multiplicador. El tráfico puede mantenerse dentro del rango del multiplicador durante 5 minutos consecutivos completos a partir de ese momento sin que se apliquen descuentos adicionales.

Es posible supervisar los eventos de descuento de intervalos a través de los [logs del tenant](/es/docs/deploy-monitor/logs/log-event-type-codes). Cada descuento genera un tipo de evento `appi` en los logs del tenant que contiene información sobre la cuota ya consumida y la restante.

Para obtener más información, consulta la sección Authentication API en [Rate Limits - Enterprise](/es/docs/troubleshoot/customer-support/operational-policies/rate-limit-policy/rate-limit-configurations/enterprise-public).

<div id="private-performance-burst">
  #### Ráfaga de rendimiento privada
</div>

La oferta Private Performance Burst (disponible actualmente en AWS para los niveles 30x y 60x) incluye una capacidad de rendimiento en ráfaga (pico) de hasta 30x (3.000 RPS) o 60x (6.000 RPS) durante un máximo de 80 horas al mes. La capacidad de rendimiento base, que equivale a la mitad de la capacidad de rendimiento en ráfaga, está disponible durante el resto del mes.

En otras palabras, Private Performance Burst 30x tiene:

* Capacidad base: 1.500 RPS durante todo el mes
* Capacidad de ráfaga/pico: 3.000 RPS durante un máximo de 80 horas al mes

Si las transacciones de autenticación superan el límite base de solicitudes a la API, se descuenta una hora de la asignación mensual. A partir de ese momento, el tráfico puede mantenerse en una tasa elevada durante esa hora completa consecutiva. Se aplican descuentos adicionales de la misma manera cada vez que se vuelve a superar el umbral.

Una vez agotadas las 80 horas asignadas, el tráfico de autenticación quedará limitado a la capacidad base hasta que entre en vigor la nueva asignación mensual.

<div id="extensibility-concurrency-limits">
  #### Límites de concurrencia de la extensibilidad
</div>

Para garantizar la disponibilidad del sistema y el uso equitativo de sus recursos, Auth0 limita el número de solicitudes simultáneas en curso en todos los productos de extensibilidad: Actions, Hooks, Rules, Custom Database Connections, Extensions y conexiones OAuth2 personalizadas. Los tenants que superen sus límites de solicitudes simultáneas pueden esperar errores en las solicitudes nuevas hasta que se completen las que estén en curso. Los límites de concurrencia se definen a continuación:

| Suscripción                                                          | Límite de concurrencia (por Tenant) |
| -------------------------------------------------------------------- | ----------------------------------- |
| Public Cloud                                                         | 250                                 |
| Private Cloud Tier Dev                                               | 100                                 |
| Private Cloud Basic 100 RPS (1x)                                     | 200                                 |
| Private Cloud Performance 500 RPS (5x)                               | 400                                 |
| Private Cloud Performance 1500 RPS (15x)                             | 1200                                |
| Private Cloud Performance 3000 RPS (30x) y 3000 RPS Burst (30x Burst) | 1200                                |
| Private Cloud Performance 6000 RPS (60x) y 6000 RPS Burst (60x Burst) | 1200                                |

La concurrencia puede calcularse multiplicando los RPS esperados por la latencia de cada solicitud. Por ejemplo, un tenant que tiene dos Actions vinculadas posteriores al inicio de sesión, cada una de las cuales tarda 250 ms, con un total de 400 RPS de inicio de sesión, tiene una concurrencia esperada de (2 * (400 solicitudes / 1 segundo) * (.25 segundos / 1 solicitud)) = 200.

Para asegurarse de que su tenant no se vea afectado por estos límites de concurrencia, asegúrese de que la lógica de extensibilidad que pueda tardar más tiempo en ejecutarse, como las llamadas a API externas, tenga tiempos de espera razonables.

<div id="rate-limit-algorithm">
  ## Algoritmo de límites de tasa
</div>

Auth0 establece límites de tasa y límites de ráfaga para sus API. Mientras que el límite de tasa es la cantidad máxima de tráfico que el sistema permite de forma sostenida, el límite de ráfaga es el volumen máximo de tráfico a corto plazo que el sistema permite dentro de un intervalo de tiempo. Los límites de tasa y los límites de ráfaga de Auth0 funcionan conjuntamente para ofrecer una limitación más eficaz ante volúmenes de tráfico dinámicos.

Los límites de tasa de Auth0 utilizan un algoritmo de token bucket con las siguientes configuraciones:

* **Claves de límite:**

  * Por lo general, una clave de límite de tasa se basa en dos factores principales:

    * API y endpoint
    * Tipo de tenant
  * En algunos casos, los factores adicionales incluyen:

    * IP de origen
    * ID del usuario de destino
* **Valores de límite:**

  * **Tamaño del bucket:** El número máximo de solicitudes que una API o un endpoint recibe en general, o de un usuario o una dirección IP específicos, antes de que se agreguen nuevas solicitudes.
  * **Tasa de recarga:** La velocidad a la que se agregan nuevas solicitudes al bucket.

A partir de estos dos valores, Auth0 calcula el límite de ráfaga y el límite de tasa sostenida:

* **Límite de ráfaga:**  Igual al tamaño del bucket.
* **Límite de tasa sostenida:** Tasa de recarga en solicitudes por minuto o por segundo.

Si el límite de tasa sostenida se calcula en solicitudes por segundo, las nuevas solicitudes se agregan en milisegundos. Si el límite de tasa sostenida se calcula en solicitudes por minuto, las nuevas solicitudes se agregan por segundos.
