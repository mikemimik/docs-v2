Auth0 limita el uso de los servicios para garantizar un rendimiento óptimo y protegerse de <Tooltip tip="Actores maliciosos: entidad (una persona o un grupo) que representa una amenaza para la empresa o el entorno con la intención de causar daño." cta="Ver glosario" href="/es/docs/glossary?term=bad+actors">actores maliciosos</Tooltip>, errores técnicos o tráfico legítimo excesivo. Recomendamos revisar cómo Auth0 aplica estos límites para configurar su aplicación y ofrecer la mejor experiencia de usuario.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Para consultar los límites de tasa asociados a su cuenta, vea [Configuraciones de límites de tasa](/es/docs/troubleshoot/customer-support/operational-policies/rate-limit-policy/rate-limit-configurations), donde encontrará una matriz de todas las políticas de límites de tasa.
</Callout>

<div id="introduction-to-rate-limits">
  ## Introducción a los límites de tasa
</div>

Auth0 aplica limitaciones para proteger los servicios de las solicitudes excesivas y a los clientes de interrupciones o degradación del servicio.

Auth0 supervisa y, en muchos casos, aplica una serie de límites, entre ellos:

* Solicitudes a entornos (solo en Private Cloud)
* Solicitudes a tenants mediante la API o un endpoint de API
* Límites varios

<div id="environment-request-limits-private-cloud-only">
  ### Límites de solicitudes del entorno (solo para Private Cloud)
</div>

En Private Cloud, los límites de solicitudes del entorno se basan en el nivel de rendimiento de Private Cloud. Para obtener más información, consulta [Private Cloud for AWS](/es/docs/deploy-monitor/deploy-private-cloud/private-cloud-on-aws) o [Private Cloud for Azure](/es/docs/deploy-monitor/deploy-private-cloud/private-cloud-on-azure).

Actualmente, los límites de velocidad del entorno de Private Cloud representan la carga máxima con la que el producto Auth0 cumplirá los SLA. Sin embargo, por el momento, Auth0 solo aplica estos límites y notifica a los clientes cuando se superan en un tenant específico dentro del entorno. En la mayoría de los casos de uso de Private Cloud, donde los clientes mantienen un único tenant de producción, esto no supone un problema; no obstante, los casos de uso que aprovisionan más de un tenant de producción deben tener en cuenta la carga esperada en todos los tenants del entorno e implementar monitorización adicional según sea necesario.

| Característica                                     | Authentication API                                                                                                                                                                                                                                                                                                               | Todas las demás API (incluida Management API)                                                                                                                                                                                                                                                                       |
| -------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Límites de velocidad del tenant                    | Todos los tenants consumen de un límite global compartido para toda la API a nivel de entorno, pero el límite de velocidad del entorno es un tope estricto. Si se supera el límite del entorno debido a la carga combinada de varios tenants, las solicitudes que excedan ese límite quedarán sujetas a limitación de velocidad. | Los límites de velocidad del tenant se aplican de forma independiente, y el límite de velocidad del entorno actúa como un umbral de posible degradación del rendimiento y de impacto en el SLA, pero **no** provoca directamente la limitación de velocidad a nivel de tenant.                                      |
| Límite de velocidad del entorno superado (ejemplo) | Con un límite del entorno de 1500 rps, si el Tenant 1 está a 1400 rps y el Tenant 2 a 900 rps (2300 rps en total), 800 solicitudes quedarán sujetas a limitación de velocidad.                                                                                                                                                   | Con un límite del entorno de 1500 rps, si el Tenant 1 está a 1400 rps y el Tenant 2 a 900 rps (2300 rps en total), **no** se aplicarán límites de velocidad a tenants individuales, pero el entorno en su conjunto puede experimentar una degradación del rendimiento y posibles problemas relacionados con el SLA. |

<Warning>
  Superar un límite de velocidad del entorno invalida el Acuerdo de Nivel de Servicio (SLA).
</Warning>

Para conocer las consideraciones sobre las pruebas de carga en Private Cloud, consulta [Private Cloud on AWS](/es/docs/deploy-monitor/deploy-private-cloud/private-cloud-on-aws) y [Private Cloud on Azure](/es/docs/deploy-monitor/deploy-private-cloud/private-cloud-on-azure).

<div id="tenant-request-limits">
  ### Límites de solicitudes del tenant
</div>

Auth0 limita la cantidad de solicitudes que se pueden realizar en un tenant. Estos límites se configuran según la API y también según los endpoints específicos de cada API.

<div id="api-rate-limits">
  #### Límites de tasa de la API
</div>

Auth0 limita la cantidad de solicitudes a una API específica, independientemente del endpoint de la API.  Los límites de la API pueden variar según:

* API

  * Autenticación
  * Gestión
* Tipo de tenant (producción vs. desarrollo o staging)
* Nivel de suscripción (Free, Essential, Professional, Enterprise Public vs. Private)

Por ejemplo, un tenant gratuito que no es de producción podría tener límites diferentes a los de un tenant de producción con una suscripción de pago. Para ver configuraciones específicas de límites de tasa, consulta [Rate Limit Configurations](/es/docs/troubleshoot/customer-support/operational-policies/rate-limit-policy/rate-limit-configurations).

<div id="user-requests">
  ### Solicitudes de usuario
</div>

Una sola solicitud de un usuario final (p. ej., Login o Signup) normalmente inicia múltiples solicitudes a endpoints de la Authentication API.  La proporción real entre las solicitudes de los usuarios finales y la Authentication API depende de varios factores:

* Entidad que se autentica (p. ej., una máquina o una aplicación móvil o de escritorio del usuario final)
* Experiencia de autenticación (p. ej., <Tooltip tip="Universal Login: Su aplicación redirige a Universal Login, alojado en el servidor de autorización de Auth0, para verificar la identidad de un usuario." cta="Ver glosario" href="/es/docs/glossary?term=Universal+Login">Universal Login</Tooltip> nuevo o Classic Login)
* Flujo de autenticación (p. ej., Login, Signup o Change Password)
* Tipo de flujo de autenticación (p. ej., Login con nombre de usuario / contraseña; Login con Social Login; Login cuando ya existe un token de autenticación)

Los clientes que utilizan Extensibility pueden agregar aún más solicitudes, no solo a la Authentication API, sino también a la <Tooltip tip="Management API: Un producto que permite a los clientes realizar tareas administrativas." cta="Ver glosario" href="/es/docs/glossary?term=Management+API">Management API</Tooltip>, según la configuración de Extensibility.
Consulte [Rate Limit Use Cases](/es/docs/troubleshoot/customer-support/operational-policies/rate-limit-policy/rate-limit-use-cases) para obtener orientación sobre cómo estimar de qué manera su configuración de Auth0 puede afectar el uso de su API.

<div id="endpoint-rate-limits">
  #### Límites de tasa de los endpoints
</div>

Auth0 limita la cantidad de solicitudes realizadas a los endpoints de la API y, en algunos casos, la cantidad de operaciones del endpoint.  Los límites de los endpoints de la API también varían según:

* API
* Tipo de tenant
* Nivel de suscripción

Por ejemplo, un tenant gratuito que no es de producción tiene límites diferentes a los de un tenant de producción con una suscripción de pago.

<div id="tenant-request-limits-order">
  ### Orden de aplicación de los límites de solicitudes del tenant
</div>

A medida que se hacen solicitudes a tu tenant, Auth0 primero las evalúa según el límite global de la API y luego según el rate limit de los endpoints específicos de la API.

<div id="miscellaneous-limits">
  ### Otros límites
</div>

<div id="database-login-limits">
  #### Límites de inicio de sesión para conexiones de base de datos
</div>

En las conexiones de base de datos, Auth0 limita ciertos tipos de intentos repetidos de inicio de sesión en función de la cuenta de usuario y la dirección IP. Para proteger la estabilidad general del sistema, Auth0 aplica límites de frecuencia para usuario/contraseña que ayudan a mitigar la carga. El alto grado de personalización de Auth0 puede exponernos al riesgo de degradación del servicio. Entre las causas posibles se incluyen:

* Pruebas de estrés con alta carga
* Pruebas comparativas de rendimiento
* Código ineficiente que hace que los usuarios inicien sesión varias veces

Las solicitudes están sujetas a límites, como se describe en las políticas individuales de las API de Auth0.

Además, existe un límite de frecuencia para el inicio de sesión del mismo usuario: si una dirección IP realiza 20 intentos de inicio de sesión en un minuto en la misma cuenta de usuario, el límite entra en vigor. Después de eso, Auth0 permite al usuario 10 intentos por minuto. Cualquier combinación de intentos de inicio de sesión correctos y fallidos cuenta para este límite.

<Card title="Límites que protegen a los usuarios">
  La [protección contra fuerza bruta](/es/docs/secure/attack-protection/brute-force-protection) y la [limitación de IP sospechosas](/es/docs/secure/attack-protection/suspicious-ip-throttling) de Auth0 también pueden limitar los inicios de sesión y los registros, pero son independientes de los límites de frecuencia. Para obtener más información sobre cómo Auth0 detecta y gestiona anomalías potencialmente maliciosas, consulta [Attack Protection](/es/docs/secure/attack-protection).
</Card>

<div id="sms-message-limits-for-multi-factor-authentication-end-users-only">
  #### Límites de mensajes SMS para la autenticación multifactor (solo para usuarios finales)
</div>

Si intenta enviar más de 10 mensajes SMS a su dispositivo en el plazo de una hora, recibirá un mensaje de error indicando que se ha superado el límite.

Cuando supere el límite de mensajes, deberá esperar al menos una hora desde la primera solicitud de mensaje antes de poder solicitar otro. Recibirá un intento adicional por cada hora adicional que transcurra.

<div id="native-social-login-limits">
  #### Límites del inicio de sesión social nativo
</div>

Los límites que se aplican a las solicitudes del flujo de inicio de sesión social nativo se determinan según el cuerpo de la solicitud, con los siguientes criterios iniciales:

| Tipo de solicitud    | Cuerpo                                               |
| -------------------- | ---------------------------------------------------- |
| `grant_type`         | `urn:ietf:params:oauth:grant-type:token-exchange`    |
| `subject_token_type` | `http://auth0.com/oauth/token-type/apple-authz-code` |

<div id="public-performance-burst">
  #### Public Performance Burst
</div>

La oferta Public Performance Burst es un add-on disponible para suscripciones empresariales que mejora una implementación existente de Public Cloud. Esta oferta permite aumentar dinámicamente el límite de solicitudes de Authentication API hasta un multiplicador del límite predeterminado de solicitudes empresariales de 100 RPS durante un máximo de 48 horas al mes.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Este add-on solo amplía los límites de solicitudes de Authentication API y NO se aplica a Management API ni a otros endpoints sujetos a rate limits fuera del alcance de Authentication API.
</Callout>

Actualmente, hay tres modificadores de Public Performance Burst (2x, 3x y 4x) que permiten 200, 300 y 400 RPS, respectivamente, para Authentication API durante un máximo de 48 horas al mes. Estas 48 horas se contabilizan y descuentan de la cuota mensual permitida en intervalos de 5 minutos, lo que permite usar la API en el nivel del multiplicador durante 576 intervalos de 5 minutos al mes.

Cuando el volumen de solicitudes de Authentication API supera el valor predeterminado de 100 RPS, se descuenta un intervalo de 5 minutos de la cuota mensual permitida y se permite tráfico a la tasa asociada con el multiplicador. El tráfico puede mantenerse dentro del rango del multiplicador durante 5 minutos continuos completos a partir de ese momento sin generar descuentos adicionales.

Es posible monitorear los eventos de descuento de intervalos mediante los [logs del tenant](/es/docs/deploy-monitor/logs/log-event-type-codes). Cada descuento genera un evento de log del tenant `appi` asociado que contiene información sobre la cuota ya consumida y la restante.

Para obtener más información, consulta la sección de Authentication API en [Rate Limits - Enterprise](/es/docs/troubleshoot/customer-support/operational-policies/rate-limit-policy/rate-limit-configurations/enterprise-public).

<div id="private-performance-burst">
  #### Private Performance Burst
</div>

La oferta Private Performance Burst (disponible actualmente en AWS para los niveles 30x y 60x) incluye una capacidad de rendimiento en ráfaga (pico) de hasta 30x (3.000 RPS) o 60x (6.000 RPS) durante un máximo de 80 horas al mes. La capacidad de rendimiento base, que equivale a la mitad de la capacidad de rendimiento en ráfaga, está disponible durante el resto del mes.

En otras palabras, Private Performance Burst 30x ofrece:

* Capacidad base: 1.500 RPS durante todo el mes
* Capacidad en ráfaga/pico: 3.000 RPS durante un máximo de 80 horas al mes

Si las transacciones de autenticación superan el límite base de solicitudes a la API, se descuenta una hora de la asignación mensual. El tráfico puede mantenerse a una tasa elevada durante toda esa hora consecutiva a partir de ese momento. Se aplican descuentos adicionales de la misma forma cada vez que se vuelve a superar el umbral.

Una vez agotada la asignación de 80 horas, al tráfico de autenticación se le aplicará el límite de tasa correspondiente a la capacidad base hasta que entre en vigor la nueva asignación mensual.

<div id="extensibility-concurrency-limits">
  #### Límites de concurrencia de la extensibilidad
</div>

Para garantizar la disponibilidad del sistema y el uso equitativo de sus recursos, Auth0 limita la cantidad de solicitudes simultáneas activas en todos los productos de extensibilidad: Actions, Hooks, Rules, Custom Database Connections, Extensions y conexiones OAuth2 personalizadas. Los tenants que superen sus límites de solicitudes simultáneas pueden recibir errores en las nuevas solicitudes hasta que se completen las que están en curso. Los límites de concurrencia se definen a continuación:

| Suscripción                                                           | Límite de concurrencia (por tenant) |
| --------------------------------------------------------------------- | ----------------------------------- |
| Public Cloud                                                          | 250                                 |
| Tier Dev Private Cloud                                                | 100                                 |
| Private Cloud Basic 100 RPS (1x)                                      | 200                                 |
| Private Cloud Performance 500 RPS (5x)                                | 400                                 |
| Private Cloud Performance 1500 RPS (15x)                              | 1200                                |
| Private Cloud Performance 3000 RPS (30x) y 3000 RPS Burst (30x Burst) | 1200                                |
| Private Cloud Performance 6000 RPS (60x) y 6000 RPS Burst (60x Burst) | 1200                                |

La concurrencia puede calcularse multiplicando los RPS esperados por la latencia de cada solicitud. Por ejemplo, un tenant que tiene asociadas dos Actions de Post Login, cada una de las cuales tarda 250 ms, con un total de 400 RPS de inicio de sesión, tiene una concurrencia esperada de (2 * (400 solicitudes / 1 segundo) * (.25 segundos / 1 solicitud)) = 200.

Para asegurarse de que su tenant no se vea afectado por estos límites de concurrencia, asegúrese de que cualquier lógica de extensibilidad que pueda tardar mucho en ejecutarse, como las llamadas a APIs externas, tenga tiempos de espera razonables.

<div id="rate-limit-algorithm">
  ## Algoritmo de rate limit
</div>

Auth0 establece rate limits y burst limits para sus API. Mientras que el rate limit es la cantidad máxima de tráfico sostenido que el sistema permite de forma continua, el burst limit es el volumen máximo de tráfico a corto plazo que el sistema permite dentro de un intervalo de tiempo. Los rate limits y burst limits de Auth0 funcionan conjuntamente para ofrecer una mejor capacidad de limitación ante volúmenes de tráfico dinámicos.

Los rate limits de Auth0 utilizan un algoritmo de token bucket con las siguientes configuraciones:

* **Claves de límite:**

  * Normalmente, una clave de rate limit se basa en dos factores principales:

    * API y endpoint
    * Tipo de tenant
  * En algunos casos, los factores adicionales incluyen:

    * IP de origen
    * ID del usuario de destino
* **Valores de límite:**

  * **Tamaño del bucket:** El número máximo de solicitudes que una API o un endpoint recibe en general, o de un usuario o una dirección IP específicos, antes de que se agreguen nuevas solicitudes.
  * **Tasa de recarga:** La velocidad a la que se agregan nuevas solicitudes al bucket.

A partir de estos dos valores, Auth0 calcula el burst limit y el rate limit sostenido:

* **Burst limit:**  Igual al tamaño del bucket.
* **Rate limit sostenido:** Tasa de recarga en solicitudes por minuto o por segundo.

Si el rate limit sostenido se calcula en solicitudes por segundo, las nuevas solicitudes se agregan en milisegundos. Si el rate limit sostenido se calcula en solicitudes por minuto, las nuevas solicitudes se agregan cada segundo.
