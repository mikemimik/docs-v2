---

<div id="endpoint">
  ## Endpoint
</div>

`POST /bc-authorize`

El endpoint de inicio de sesión por canal secundario permite a las aplicaciones enviar una solicitud de autenticación al teléfono del usuario o al dispositivo de autenticación, siempre que tengan una aplicación instalada y estén inscritas para [recibir notificaciones push mediante el SDK de Guardian](https://auth0.com/docs/secure/multi-factor-authentication/auth0-guardian#enroll-in-push-notifications).

Usa el endpoint de inicio de sesión por canal secundario para autenticar a los usuarios en los siguientes casos de uso:

* Los usuarios no están delante de la aplicación que requiere autenticación, como cuando llaman a un centro de atención telefónica.
* El dispositivo de consumo, o el dispositivo que ayuda al usuario a consumir un servicio, no es seguro para operaciones sensibles (por ejemplo, un navegador web para transacciones financieras).
* El dispositivo de consumo tiene una capacidad de interacción limitada (por ejemplo, bicicletas o patinetes eléctricos).

<div id="response-body">
  ### Cuerpo de la respuesta
</div>

Si la solicitud se realiza correctamente, deberías recibir una respuesta similar a la siguiente:

```http
{
  "auth_req_id": "eyJh...",
  "expires_in": 300,
  "interval": 5
}
```

El valor de `auth_req_id` debe conservarse, ya que se utiliza más adelante en el flujo para identificar la solicitud de autenticación.

El valor de `expires_in` te indica cuántos segundos faltan para que expire la solicitud de autenticación.

El valor de `interval` te indica cuántos segundos debes esperar entre cada solicitud de consulta.

La solicitud debe aprobarse o rechazarse en el dispositivo de autenticación del usuario mediante el SDK de Guardian.

<div id="remarks">
  ### Observaciones
</div>

* El valor de `login_hint` sigue el formato `iss_sub`. Por ejemplo:

```http
{ "format": "iss_sub", "iss": "https://[TENANT_DOMAIN]/", "sub": "auth0|[USER ID]" }
```

* La autenticación puede realizarse con Client Secret (mediante HTTP Basic Auth o POST), Private Key JWT o mTLS. Consulta los ejemplos detallados anteriores.

<div id="body-parameters">
  ## Parámetros del cuerpo
</div>

<ParamField body="client_id" type="string" required>
  El `client_id` de tu aplicación.
</ParamField>

<ParamField body="binding_message" type="string" required>
  Una cadena legible para el usuario, para asegurarse de que apruebe la solicitud correcta.
</ParamField>

<ParamField body="login_hint" type="string" required>
  Un objeto JSON que contiene los datos del usuario para la autenticación en formato `iss_sub`.
</ParamField>

<ParamField body="scope" type="string" required>
  Lista de scopes de OIDC y de API personalizadas, separados por espacios, que debe incluir `openid` como mínimo.
</ParamField>

<ParamField body="audience" type="string">
  Identificador único opcional de la API de destino para los tokens de acceso.
</ParamField>

<ParamField body="resource" type="string">
  El identificador de la API de destino (servidor de recursos) a la que deseas acceder. Debe coincidir con un identificador de API registrado en tu tenant de Auth0. Se usa como alternativa a `audience` cuando el [Resource Parameter Compatibility Profile](https://auth0.com/docs/get-started/tenant-settings#settings-advanced) del tenant está configurado como `compatibility`.
</ParamField>

<ParamField body="requested_expiry" type="string">
  Tiempo de expiración personalizado de la solicitud, en segundos (1-300). El valor predeterminado es 300 segundos.
</ParamField>

<ParamField body="authorization_details" type="array">
  Un arreglo JSON opcional de objetos que describe los permisos que se van a autorizar. El valor de `type` de cada objeto debe haberse registrado previamente en el servidor de recursos mediante el parámetro `authorization_details` del servidor de recursos. Para obtener más información, consulta la documentación de la Management API sobre [Update a resource server](https://auth0.com/docs/api/management/v2/resource-servers/patch-resource-servers-by-id#body-parameters).
</ParamField>

<div id="response-messages">
  ## Mensajes de respuesta
</div>

| Estado | Descripción                                            |
| ------ | ------------------------------------------------------ |
| 200    | La solicitud de autenticación se inició correctamente. |
| 400    | Solicitud incorrecta - Parámetros no válidos           |
| 500    | Error interno del servidor                             |
