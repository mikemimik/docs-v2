---
title: "Flujo de Back-Channel Login - Verificación de estado"
description: "Consulta periódicamente el endpoint de token para verificar el estado de una solicitud pendiente de Back-Channel Login."
---

<div id="endpoint">
  ## Endpoint
</div>

`POST /oauth/token`

Para comprobar el estado de un flujo de Back-Channel Login, consulta periódicamente el endpoint `/oauth/token` pasando lo siguiente:

* `auth_req_id` devuelto por la llamada a `/bc-authorize`
* tipo de concesión `urn:openid:params:grant-type:ciba`

<div id="response-body">
  ### Cuerpo de la respuesta
</div>

Si el usuario autorizador aún no ha aprobado ni rechazado la solicitud, deberías recibir una respuesta como la siguiente:

```http
{ 
  "error": "authorization_pending", 
  "error_description": "The end-user authorization is pending"
}
```

Si el usuario que autoriza rechaza la solicitud, deberías recibir una respuesta como la siguiente:

```http
{
  "error": "access_denied",
  "error_description": "The end-user denied the authorization request or it has been expired"
}
```

Si estás realizando consultas con demasiada frecuencia (más rápido que el valor del intervalo devuelto por `/bc-authorize`), deberías recibir una respuesta como la siguiente:

```http
{
  "error": "slow_down",
  "error_description": "You are polling faster than allowed. Try again in 10 seconds."
}
```

Además, Auth0 agregará el encabezado [Retry-After](https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers/Retry-After) a la respuesta para indicar cuántos segundos debe esperar antes de volver a consultar. Si consulta con demasiada frecuencia de forma reiterada, aumentará la cantidad de segundos que debe esperar.

Si el usuario que autoriza ha aprobado la notificación push, la llamada devuelve el token de ID y el token de acceso (y, potencialmente, un token de actualización):

```http
{
  "access_token": "eyJh...",
  "id_token": "eyJh...",
  "expires_in": 86400,
  "scope": "openid"
}
```

Una vez que hayas intercambiado un `auth_req_id` por un token de ID o de acceso, ya no podrá usarse.

<div id="remarks">
  ### Observaciones
</div>

Incluya un parámetro opcional para la autenticación de la aplicación en la solicitud:

* Client Secret con autenticación HTTP Basic, en cuyo caso no se requiere ningún parámetro. El `client_id` y el `client_secret` se envían en un encabezado.
* Client Secret Post, en cuyo caso se requieren `client_id` y `client_secret`.
* Private Key JWT, para el que se requieren `client_id`, `client_assertion` y el tipo de `client_assertion`.
* mTLS, para el que se requiere el parámetro `client_id` y los encabezados `client-certificate` y `client-certificate-ca-verified`.

<div id="body-parameters">
  ## Parámetros del cuerpo
</div>

<ParamField body="client_id" type="string" required>
  El `client_id` de tu aplicación.
</ParamField>

<ParamField body="auth_req_id" type="string" required>
  El `auth_req_id` devuelto por el endpoint `/bc-authorize`.
</ParamField>

<ParamField body="grant_type" type="string" required>
  Debe establecerse como `urn:openid:params:grant-type:ciba`.
</ParamField>

<div id="response-messages">
  ## Mensajes de respuesta
</div>

| Estado | Descripción                                 |
| ------ | ------------------------------------------- |
| 200    | Se devolvió el estado de autenticación.     |
| 400    | Solicitud incorrecta: parámetros no válidos |
| 500    | Error interno del servidor                  |
