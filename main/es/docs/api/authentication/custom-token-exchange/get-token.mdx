<div id="endpoint">
  ## Endpoint
</div>

`POST /oauth/token`

Custom Token Exchange (CTE) ofrece un mecanismo para que las aplicaciones intercambien tokens de identidad preexistentes por tokens de Auth0 invocando el endpoint `/oauth/token`, de conformidad con las especificaciones de [RFC 8693](https://datatracker.ietf.org/doc/html/rfc8693). Esta funcionalidad es clave para satisfacer requisitos de integración avanzados, como intercambiar tokens existentes de Auth0 para acceder a otra audiencia en nombre del mismo usuario, facilitar la integración de proveedores de identidad externos o permitir una migración fluida de usuarios a la plataforma de Auth0. El proceso de intercambio puede controlarse por completo, ya que los desarrolladores pueden definir sus detalles mediante lógica personalizada que se ejecuta en una Auth0 Action dedicada al caso de uso correspondiente.

<Note>
  Custom Token Exchange se encuentra actualmente en Early Access. Al utilizar esta funcionalidad, aceptas los términos aplicables de la Free Trial incluidos en el [Master Subscription Agreement de Okta](https://www.okta.com/legal/). Es tu responsabilidad validar de forma segura el `subject_token` del usuario. Consulta la [guía del usuario](https://auth0.com/docs/authenticate/custom-token-exchange) para obtener más información.
</Note>

<div id="remarks">
  ### Observaciones
</div>

* Los scopes emitidos para la aplicación pueden diferir de los scopes solicitados. En este caso, se incluirá un parámetro `scope` en el JSON de respuesta. Los scopes se filtran como se explica en [API Access Policies for Applications](https://auth0.com/docs/ja-jp/get-started/apis/api-access-policies-for-applications)

* `subject_token_type` debe corresponder a un Custom Token Exchange Profile existente y a una [Action](https://auth0.com/docs/customize/actions) asociada.

* En el caso de las aplicaciones no confidenciales que no pueden mantener seguro el Client Secret (por ejemplo, las aplicaciones nativas), el endpoint permite no enviar ningún Client Secret, pero la propia aplicación debe tener la propiedad `tokenEndpointAuthMethod` establecida en `none`. Puedes hacerlo desde la interfaz de usuario (**Dashboard &gt; Applications &gt; Application Settings**) o mediante la Management API.

* Debes [habilitar Custom Token Exchange para tu aplicación](https://auth0.com//docs/authenticate/custom-token-exchange/configure-custom-token-exchange#enable-custom-token-exchange-for-your-application). Para obtener más información, consulta la [documentación de Custom Token Exchange](https://auth0.com/docs/authenticate/custom-token-exchange).

* `actor_token` y `actor_token_type` deben estar ambos presentes o ambos ausentes. Si proporcionas solo uno, se devuelve un error `400` con el código `invalid_request`.

* Cuando `actor_token` está presente, no se emiten tokens de actualización. El scope `offline_access` se excluye de la respuesta.

* MFA no es compatible con transacciones en las que se establece un actor mediante `setActor()`. Si MFA es obligatorio y la Action de Custom Token Exchange establece un actor, la solicitud devuelve un error `400`: `MFA is not supported using actor_token with the requested token exchange profile.`

* Cuando la Action llama a `setActor()`, los tokens de acceso y los tokens de identidad emitidos incluyen un claim `act` que representa la [cadena de delegación](/es/docs/secure/call-apis-on-users-behalf/on-behalf-of-token-exchange#the-act-claim). El claim `act` también se incluye en la respuesta de userinfo.

<div id="headers">
  ## Encabezados
</div>

<ParamField header="DPoP" type="string">
  Una prueba DPoP para la solicitud. Es opcional y solo se requiere si tu aplicación usa Demonstrating Proof-of-Possession.
</ParamField>

<ParamField header="auth0-forwarded-for" type="string">
  La IP del usuario final como un valor de cadena. Configúralo si quieres que la protección de Suspicious IP Throttling funcione en escenarios del lado del servidor.
</ParamField>

<div id="body-parameters">
  ## Parámetros del cuerpo
</div>

<ParamField body="grant_type" type="string" required>
  Indica el flujo que estás utilizando. Para Custom Token Exchange, usa `urn:ietf:params:oauth:grant-type:token-exchange`.

  Valores permitidos: `urn:ietf:params:oauth:grant-type:token-exchange`
</ParamField>

<ParamField body="subject_token_type" type="string" required>
  El tipo del token de sujeto. Para Custom Token Exchange, puede ser cualquier URI dentro de un espacio de nombres de tu propiedad, como `http://acme.com/legacy-token` o `urn:acme:legacy-token`. Los siguientes espacios de nombres están reservados y no se pueden usar: `http://auth0.com`, `https://auth0.com`, `http://okta.com`, `https://okta.com`, `urn:ietf`, `urn:auth0`, `urn:okta`.
</ParamField>

<ParamField body="subject_token" type="string" required>
  El token de sujeto, que tu acción debe validar y usar para identificar al usuario.
</ParamField>

<ParamField body="client_id" type="string" required>
  El Client ID de tu aplicación. Al igual que con otros tipos de grant, también puedes pasar el client ID en el encabezado Authorization mediante HTTP Basic Auth.
</ParamField>

<ParamField body="client_secret" type="string">
  (Opcional) El Client Secret de tu aplicación. Al igual que con otros tipos de grant, también puedes pasar el client secret en el encabezado Authorization mediante HTTP Basic Auth. También hay otras alternativas disponibles, como se explica en la [documentación de referencia de Auth0 Authentication API](https://auth0.com/docs/api/authentication#authentication-methods). Dado que Custom Token Exchange puede ser utilizado por aplicaciones públicas, consulta [Attack Protection](https://auth0.com/docs/authenticate/custom-token-exchange/cte-attack-protection) para implementar medidas de seguridad adicionales.
</ParamField>

<ParamField body="audience" type="string">
  (Opcional) El identificador único de la API de destino a la que quieres acceder. Si no se proporciona, se usará la audience predeterminada del tenant, según la configuración de [Tenant Settings](https://auth0.com/docs/get-started/tenant-settings).
</ParamField>

<ParamField body="resource" type="string">
  (Opcional) El identificador de la API de destino (servidor de recursos) a la que quieres acceder. Debe coincidir con un API Identifier registrado en tu tenant de Auth0. Se usa como alternativa a `audience` cuando el [Resource Parameter Compatibility Profile](https://auth0.com/docs/get-started/tenant-settings#settings-advanced) del tenant está configurado como `compatibility`.
</ParamField>

<ParamField body="scope" type="string">
  (Opcional) El parámetro scope de OAuth2.
</ParamField>

<ParamField body="organization" type="string">
  (Opcional) La organización o el identificador con el que quieres asociar la solicitud. Como alternativa, puedes especificar un nombre de organización si [Use Organization Names in Authentication API](https://auth0.com/docs/manage-users/organizations/configure-organizations/use-org-name-authentication-api) está configurado.
</ParamField>

<ParamField body="actor_token" type="string">
  (Opcional) Un token que identifica al actor que realiza la delegación en nombre del usuario sujeto. Debe proporcionarse junto con `actor_token_type`. Si está presente, no se emitirán tokens de actualización.
</ParamField>

<ParamField body="actor_token_type" type="string">
  (Opcional) El tipo del token del actor. Debe proporcionarse junto con `actor_token`. Para los tokens de identidad de Auth0, usa `urn:ietf:params:oauth:token-type:id_token` para la validación automática del lado del servidor (firma, vencimiento, emisor, búsqueda de usuario). Para otros valores, se aplican las mismas restricciones de espacio de nombres que a `subject_token_type`.
</ParamField>

  </ResponseField>

  <ResponseField name="id_token" type="string">

  </ResponseField>

    El tipo de token.
