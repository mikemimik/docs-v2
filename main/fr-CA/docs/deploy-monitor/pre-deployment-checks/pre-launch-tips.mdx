description: Une liste de conseils utiles pour bien démarrer avec les services Auth0, fondée sur les commentaires et l’expérience d’autres utilisateurs
title: Conseils avant le lancement
---

Voici une liste des conseils que nos clients ont jugés les plus utiles lorsqu’ils ont commencé à utiliser les services Auth0 :

* Vérifiez les URL dans vos URL de rappel autorisées, vos origines CORS autorisées et vos URL de redirection autorisées pour la déconnexion afin de vous assurer qu’elles sont exactes, complètes et qu’elles n’utilisent ni `file:///` ni `localhost`. Réduisez au minimum l’utilisation des caractères génériques et évaluez attentivement les répercussions de sécurité qu’ils peuvent avoir.
* Séparez l’utilisation en production du travail de développement en cours en [utilisant des tenants distincts pour prod, test et dev.](/docs/fr-CA/get-started/auth0-overview/create-tenants/set-up-multiple-environments)
* Configurez correctement votre réseau pour autoriser le trafic à destination et en provenance de nos différentes API publiques (les environnements prod peuvent avoir une configuration différente de dev/staging).
* Vérifiez la date d’expiration des certificats que vous avez ajoutés à votre configuration pour vous assurer que les certificats téléversés au cours des cycles de développement n’expireront pas pendant le lancement ou peu après.
* Assurez-vous que tous les <Tooltip tip="Fournisseur d’identité (IdP) : service qui stocke et gère les identités numériques." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=IDPs">IDPs</Tooltip> distants utilisent NTP pour que l’heure soit correctement synchronisée.
* Le nombre de jours de données de journalisation disponibles varie selon le forfait, jusqu’à un maximum de 30 jours. Si vous devez conserver des données de journalisation plus longtemps, vous devriez configurer l’une de nos [solutions de diffusion continue des journaux](https://marketplace.auth0.com/features/log-streaming). Cela vous permettra de les conserver sur une plus longue période. Assurez-vous de le configurer avant la mise en production afin d’avoir des données de journalisation lorsque vous en aurez besoin.
* Assurez-vous que votre utilisation de nos API demeure dans les [limites autorisées](/docs/fr-CA/troubleshoot/customer-support/operational-policies/rate-limit-policy) et que votre code est conçu pour s’adapter dynamiquement aux informations sur les limites de débit renvoyées dans l’en-tête, ainsi que pour gérer les erreurs. Si vous prévoyez que les limites de débit poseront problème pour votre application, parlez-nous-en à l’avance pour vérifier s’il serait possible de les augmenter temporairement jusqu’à ce que le trafic revienne à la normale. Vous devriez envisager d’utiliser un cache des données utilisateur afin de ne pas interroger un point de terminaison d’API plus souvent que nécessaire.
* Si vous utilisez des [connexions sociales](/docs/fr-CA/authenticate/identity-providers), assurez-vous d’obtenir vos propres identifiants auprès du fournisseur et de les ajouter à la configuration de la connexion sociale.
* Si vous utilisez des [connexions DB personnalisées](/docs/fr-CA/authenticate/database-connections/custom-db), assurez-vous que tous les scripts DB personnalisés sont mis en place et renvoient un profil utilisateur cohérent avec un identifiant utilisateur unique.
* Pour l’envoi de courriels, assurez-vous d’abord de [configurer un fournisseur de courriel personnalisé.](/docs/fr-CA/customize/email/smtp-email-providers)
* Si vous utilisez notre CDN pour le widget Lock, assurez-vous de fixer une version précise.
* Assurez-vous que les composants **externes** appelés depuis Actions, Rules, Hooks et les scripts de connexion à une base de données personnalisée peuvent prendre en charge la charge prévue.
* Protégez adéquatement toute valeur de <Tooltip tip="Client Secret : secret utilisé par un client (application) pour s’authentifier auprès de l’Authorization Server; il ne doit être connu que du client et de l’Authorization Server et doit être suffisamment aléatoire pour qu’il soit impossible à deviner." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=client+secret">client secret</Tooltip>.
* Vérifiez les [types d’autorisation](/docs/fr-CA/get-started/applications/application-grant-types) de vos applications. Assurez-vous que les bons sont activés et, surtout, désactivez tous les types d’autorisation inutiles.
* Si vous utilisez user&#95;metadata, assurez-vous qu’il s’agit de données que les utilisateurs peuvent modifier eux-mêmes (p. ex., pas le « statut du paiement »).
* Passez en revue vos [paramètres d’Attack Protection](https://manage.auth0.com/#/security/attack-protection) et lisez la documentation sur [Attack Protection](/docs/fr-CA/secure/attack-protection) pour comprendre comment débloquer les utilisateurs qui ont été bloqués.
* Vérifiez la [section Admin de vos Tenant Settings](https://manage.auth0.com/#/tenant/admins) pour vous assurer que seuls les administrateurs appropriés ont accès au <Tooltip tip="Auth0 Dashboard : le produit principal d’Auth0 pour configurer vos services." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip>. Consultez [Manage Dashboard Access](/docs/fr-CA/get-started/manage-dashboard-access) pour en savoir plus.
* Assurez-vous d’avoir testé tous les principaux cas d’utilisation de votre application sur tous les appareils susceptibles d’être utilisés par ses utilisateurs finaux. Assurez-vous de tester la connexion, <Tooltip tip="Authentification unique (SSO) : service qui, après qu’un utilisateur se connecte à une application, le connecte automatiquement aux autres applications." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=single+sign-on">authentification unique</Tooltip> (si prise en charge) et la journalisation, ainsi que ce qui se passe si un utilisateur exécute votre application dans plusieurs onglets de navigateur.
* Passez en revue votre [liste de Rules](https://manage.auth0.com/#/rules) et assurez-vous que seules les Rules appropriées sont activées.
* Passez en revue le code de vos Rules, tous les custom DB scripts, ainsi que tout code personnalisé dans la page de login afin de vous assurer que chaque appel comprend un traitement adéquat des erreurs. Vérifiez également que les instructions return/callback sont appelées correctement.
* Configurez le nom de votre application, l’URL de soutien et le courriel de soutien dans la section [Tenant Settings General](https://manage.auth0.com/#/tenant) afin que, lorsqu’une erreur survient, vos utilisateurs finaux soient redirigés vers la page appropriée.
* Assurez-vous que votre application [obtient dynamiquement un jeton pour la Management API](/docs/fr-CA/secure/tokens/access-tokens/management-api-access-tokens).
* Supprimez toutes les instructions `console.log` de vos Rules ou custom DB scripts, surtout celles qui pourraient divulguer des renseignements permettant d’identifier un utilisateur, comme son courriel, son username ou son Password.
* N’utilisez pas de secrets en texte clair dans les Rules ou les connexions de base de données. Ils doivent être ajoutés dans la partie configuration de l’interface. La configuration est chiffrée et fournie en Just In Time. N’inscrivez pas l’objet de configuration dans le journal.
