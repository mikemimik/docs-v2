title: Exclure des ressources de la gestion
---

Dans certains cas, il peut être utile d’exclure certaines ressources de la gestion. Par exemple, votre tenant peut contenir un grand nombre d’un type de ressource donné, et leur gestion peut représenter une charge opérationnelle importante. Ou encore, votre workflow de développement peut ne porter que sur un sous-ensemble précis de ressources, et vous pourriez vouloir laisser de côté toutes les autres pour des raisons de performance. Quoi qu’il en soit, plusieurs options s’offrent à vous pour exclure des ressources lorsque vous utilisez le Deploy CLI.

<div id="exclude-entire-resources-by-type">
  ## Exclure des types de ressources entiers
</div>

Pour les tenants plus complexes, vous pourriez vouloir omettre des types de ressources entiers. Par exemple :

* Tenant Enterprise avec des milliers d’organizations, pour lequel tout gérer représenterait un lourd fardeau opérationnel.
* Le processus CI/CD se concentre uniquement sur la gestion des rôles, et vous voulez exclure tous les autres.
* Le développement d’une fonctionnalité concerne un hook, et vous voulez temporairement exclure tous les autres afin d’optimiser les performances.

Ce type d’exclusion s’exprime en transmettant un tableau de noms de ressources à l’une ou l’autre des propriétés de configuration `AUTH0_EXCLUDED` ou `AUTH0_INCLUDED_ONLY`. La propriété de configuration `AUTH0_EXCLUDED` exclut uniquement les types de ressources qui lui sont fournis. À l’inverse, la propriété `AUTH0_INCLUDED_ONLY` exclut tous les types de ressources, sauf ceux qui sont définis. L’exclusion fonctionne dans les deux sens, c’est-à-dire à la fois lors de l’exportation depuis Auth0 et de l’importation vers Auth0, que des fichiers de configuration de ressources existent ou non.

Toutes les valeurs de ressource prises en charge pour l’exclusion : `actions`, `attackProtection`, `branding`, `clientGrants`, `clients`, `connections`, `customDomains`, `databases`, `emailProvider`, `emailTemplates`, `guardianFactorProviders`, `guardianFactorTemplates`, `guardianFactors`, `guardianPhoneFactorMessageTypes`, `guardianPhoneFactorSelectedProvider`, `guardianPolicies`, `logStreams`, `migrations`, `organizations`, `pages`, `prompts`, `resourceServers`, `roles`, `tenant`, `triggers`.

<div id="exclusion-example">
  ### Exemple d’exclusion
</div>

L’exemple suivant montre comment exclure `clients`, `connections`, `databases` et `organizations` afin qu’ils ne soient pas gérés par le Deploy CLI.

```json lines
{
  "AUTH0_DOMAIN": "example-site.us.auth0.com",
  "AUTH0_CLIENT_ID": "<YOUR_AUTH0_CLIENT_ID>",
  "AUTH0_EXCLUDED": ["clients", "connections", "databases", "organizations"]
}
```

<div id="inclusion-example">
  ### Exemple d’inclusion
</div>

L’exemple suivant montre comment indiquer au Deploy CLI de ne gérer que les `actions`, `clients` et `connections`.

```json lines
{
  "AUTH0_DOMAIN": "example-site.us.auth0.com",
  "AUTH0_CLIENT_ID": "<YOUR_AUTH0_CLIENT_ID>",
  "AUTH0_INCLUDED_ONLY": ["actions", "clients", "connections"]
}
```

<div id="exclude-single-resources-by-id">
  ## Exclure des ressources uniques par ID
</div>

Certains types de ressources permettent d’exclure une ressource précise par son ID. Cette option est utile si vous travaillez dans un contexte multienvironnement et souhaitez omettre, dans vos environnements de niveau inférieur, une ressource propre à l’environnement Production.

Cette méthode est prise en charge pour les rules, les clients, les bases de données, les connections et les <Tooltip tip="Serveur de ressources : serveur hébergeant des ressources protégées. Les serveurs de ressources acceptent les requêtes de ressources protégées et y répondent." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=resource+servers">serveurs de ressources</Tooltip> au moyen des valeurs de configuration `AUTH0_EXCLUDED_RULES`, `AUTH0_EXCLUDED_CLIENTS`, `AUTH0_EXCLUDED_DATABASES`, `AUTH0_EXCLUDED_CONNECTIONS`, `AUTH0_EXCLUDED_RESOURCE_SERVERS`, respectivement.

### Exemple

```json lines
{
  "AUTH0_DOMAIN": "example-site.us.auth0.com",
  "AUTH0_CLIENT_ID": "<YOUR_AUTH0_CLIENT_ID>",
  "AUTH0_EXCLUDED_CLIENTS": ["Your Application Name"],
  "AUTH0_EXCLUDED_CONNECTIONS": ["con_O1H3KyRMFP1IWRq3", "con_9avEYuj19ihqKBOs"]
}
```

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  L’exclusion de ressources par ID pourrait être dépréciée dans de futures versions majeures. Consultez la [proposition d’exclusion de ressources](https://github.com/auth0/auth0-deploy-cli/blob/6381ce6efa8cd59e7a5c8d37e8915b0989d677b9/docs/excluding-from-management.md?plain=1#L51) pour en savoir plus.
</Callout>

<div id="omission-and-empty-states">
  ## Omission et états vides
</div>

En plus d’exclure des ressources, ce qui force l’ignorance bidirectionnelle des configurations, la Auth0 Deploy CLI prend en charge deux concepts similaires : l’omission et les états vides.

<div id="omission">
  ### Omission
</div>

Toute configuration de ressource absente, que ce soit intentionnellement ou non, sera ignorée à l’importation. Par exemple, si votre configuration de ressource était supprimée, elle serait ignorée à l’importation et ne modifierait pas l’état du tenant distant.

Il n’existe pas de notion d’omission à l’exportation. Sauf exclusion explicite, toutes les configurations de votre tenant seront écrites dans des fichiers de configuration des ressources.

<div id="example">
  #### Exemple
</div>

```yaml lines
roles: # la configuration des rôles n'est pas omise
  - name: Admin
    description: Can read and write things
    permissions: []
  - name: Reader
    description: Can only read things
    permissions: []
# L'omission de toutes les autres configurations signifie qu'elles seront ignorées
```

<div id="empty">
  ### Vide
</div>

Configuration de ressource explicitement définie comme vide. Pour les configurations basées sur des ensembles, comme les hooks, les organizations et les actions, définir ces configurations comme un ensemble vide exprime une volonté explicite de vider ces ressources. Cela signale une suppression, à condition que la propriété de configuration de suppression AUTH0&#95;ALLOW&#95;DELETE soit activée. Pour en savoir plus sur cette propriété, consultez [Configurer le Deploy CLI](/fr-CA/docs/deploy-monitor/deploy-cli-tool/configure-the-deploy-cli).

Pour les configurations de ressources qui ne sont pas basées sur des ensembles, comme tenant et branding, la notion de vide ne s’applique pas et ne déclenche aucune suppression ni aucun retrait.

<div id="example-of-emptiness">
  #### Exemple d’état vide
</div>

```yaml lines
hooks: [] # Hooks vides
connections: [] # Connexions vides
tenant: {} # Équivaut à un no-op, l'état vide ne s'applique pas aux configurations de ressources non ensemblistes
```
