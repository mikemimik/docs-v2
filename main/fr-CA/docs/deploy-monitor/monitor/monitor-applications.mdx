Vos applications et services dépendent d’Auth0. Surveiller l’état d’Auth0 vous permet de communiquer des erreurs précises aux clients ou de limiter les répercussions pour les utilisateurs en cas de problème avec Auth0.

Il existe plusieurs façons de surveiller Auth0, et chacune complète les autres. Choisissez l’approche qui convient le mieux à vos besoins et aux ressources que vous pouvez y consacrer.

<div id="synthetic-transactions">
  ## Transactions synthétiques
</div>

L’approche la plus simple pour surveiller Auth0 :

* Configurez une requête périodique pour effectuer une transaction d’authentication.
* Si la requête réussit, Auth0 fonctionne correctement.
* Si la requête échoue, cela peut indiquer :

  * un problème avec Auth0
  * un problème propre au tenant utilisé pour la transaction synthétique
  * ou simplement l’échec d’une seule requête.

Pour les transactions synthétiques, utilisez un environnement dont la configuration se rapproche le plus possible de celle de votre tenant de production. Vous pourriez même utiliser ce même tenant de production. Comme la configuration de transactions synthétiques avec des flux de redirection et des fournisseurs tiers peut être complexe, il est recommandé d’utiliser le [Resource Owner Password Grant](/fr-CA/docs/get-started/authentication-and-authorization-flow/resource-owner-password-flow/call-your-api-using-resource-owner-password-flow). Ce flux n’implique pas de redirections dans le navigateur et ne nécessite pas d’UI.

Si vous utilisez des [rules](/fr-CA/docs/customize/rules), des [custom database connections](/fr-CA/docs/authenticate/database-connections/custom-db) ou d’autres points d’extensibilité, les transactions synthétiques doivent être configurées de façon à utiliser les rules et/ou les custom DB scripts afin de s’assurer que cet aspect du système fonctionne.

Des outils comme [Pingdom](http://pingdom.com/) simplifient la configuration des transactions synthétiques.

<div id="check-period">
  ### Fréquence de vérification
</div>

Nous recommandons d’exécuter des transactions synthétiques toutes les minutes. Cette approche simple consommera peu de votre quota de limitation du débit d’Auth0, tout en fournissant des réponses en temps utile.

<div id="limitations-of-synthetic-transactions">
  ### Limites des transactions synthétiques
</div>

Les transactions synthétiques constituent un moyen simple et peu coûteux de surveiller l’état de santé d’un tenant Auth0. Toutefois, elles comportent certaines limites :

* Les transactions synthétiques ne reflètent pas l’expérience de votre utilisateur final. Elles vous fournissent plutôt un indicateur indirect.
* Les transactions synthétiques peuvent ne pas utiliser les mêmes flux que vos utilisateurs.
* Elles manquent d’atomicité (elles s’exécutent généralement une fois par minute) et ne signalent pas les erreurs que vos utilisateurs finaux ont pu voir.

Si vous souhaitez obtenir des données plus détaillées, consultez [Suivi des erreurs](#error-tracking) et [Métriques et journaux](#metrics-and-logs).

<div id="error-tracking">
  ## Suivi des erreurs
</div>

Cette approche est utile pour suivre les erreurs dans les appels existants vers Auth0. Elle consiste à signaler les erreurs chaque fois qu’un appel à Auth0 échoue. [Sentry](https://sentry.io/) est un outil couramment utilisé dans ce genre de cas, et il fonctionne autant côté frontend que backend.

Cette approche est utile parce qu’elle vous permet de connaître les erreurs réelles que rencontrent vos utilisateurs finaux. Cependant, comme vous ne suivez que les erreurs (et non toutes les requêtes), il n’est pas possible d’avoir une idée précise du nombre d’utilisateurs finaux touchés : est-ce 1 % ou 5 % ? Elle ne vous oblige pas non plus à configurer un « appel synthétique » distinct, qui pourrait consommer une partie de votre quota de limitation du débit, surtout s’il est mal configuré.

<div id="metrics-and-logs">
  ## Métriques et journaux
</div>

Cette approche est utile si les appels à Auth0 sont effectués à partir d’un backend que vous contrôlez. C’est notamment le cas pour :

* La plupart des appels à l’[API de gestion d’Auth0](https://auth0.com/docs/api/management/v2)
* Les appels à l’[API d’authentification](https://auth0.com/docs/api/authentication) effectués depuis des applications Web traditionnelles et des applications machine à machine ([en savoir plus sur les types de clients](/fr-CA/docs/get-started/applications)).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Certains produits d’observabilité permettent aussi de recueillir des métriques et des journaux à partir d’applications frontales et mobiles.
</Callout>

Cette approche consiste à utiliser des métriques et/ou des journaux pour suivre les taux d’erreur des appels à Auth0. Les métriques et les journaux fournissent des taux d’erreur qui reflètent fidèlement l’expérience des utilisateurs finaux, sans nécessiter la configuration d’un « appel synthétique » distinct, qui pourrait consommer une partie de votre quota de limitation du débit, surtout s’il est mal configuré.

<div id="alerting">
  ## Alertes
</div>

Peu importe les méthodes de surveillance que vous utilisez, il est courant de recevoir une alerte ou un appel de garde lorsque des erreurs se produisent à une certaine fréquence. Ce seuil dépend de votre application.

Lorsque votre équipe reçoit une alerte d’Auth0, nous vous recommandons d’ajouter un lien vers [Vérifier l’état d’Auth0](/fr-CA/docs/deploy-monitor/monitor/check-auth0-status) au payload/playbook de l’alerte. Cela permettra à votre équipe de consulter rapidement le canal officiel de signalement de l’état d’Auth0 afin de vérifier si le problème provient d’Auth0 ou de votre application/service.
