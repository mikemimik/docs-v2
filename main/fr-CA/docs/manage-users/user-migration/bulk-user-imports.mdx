Vous pouvez effectuer une importation en bloc de données utilisateur dans Auth0 à l’aide du point de terminaison [Create Import Users Job](https://auth0.com/docs/api/management/v2#!/Jobs/post_users_imports). Les importations en bloc sont utiles pour migrer des utilisateurs d’une base de données ou d’un service existant vers Auth0.

<Warning>
  Si vous essayez d’utiliser plus d’une méthode de migration (par exemple, une migration automatique, puis une importation en bloc d’utilisateurs), vous risquez d’obtenir une erreur `DUPLICATED_USER`. Cette erreur indique que l’utilisateur existe dans le user store interne d’Auth0, mais pas dans votre tenant. Pour corriger cette erreur, supprimez l’utilisateur à l’aide du point de terminaison [Delete a Connection User](https://auth0.com/docs/api/management/v2#!/Connections/delete_users_by_email) de l’Auth0 Management API, puis relancez l’importation.
</Warning>

<div id="prerequisites">
  ## Prérequis
</div>

Avant de lancer la tâche d’importation d’utilisateurs :

* Configurez une connexion de base de données dans laquelle importer les utilisateurs et activez-la pour au moins une application.
* Si vous importez des mots de passe, assurez-vous qu’ils sont hachés à l’aide de l’un des algorithmes pris en charge. Les utilisateurs dont les mots de passe sont hachés avec des algorithmes non pris en charge devront réinitialiser leur mot de passe à leur première connexion après l’importation en bloc.
* Si vous importez des inscriptions <Tooltip tip="Authentification multifacteur (MFA) : processus d’authentification de l’utilisateur qui utilise un facteur en plus du nom d’utilisateur et du mot de passe, comme un code envoyé par SMS." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=MFA">MFA</Tooltip>, assurez-vous qu’elles sont d’un type pris en charge : `email`, `phone` ou `totp`.
* [Obtenez un jeton de la Management API](/fr-CA/docs/secure/tokens/access-tokens/management-api-access-tokens) pour les requêtes vers le point de terminaison de la tâche.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Si vous utilisez un fichier d’exportation provenant d’un tenant Auth0, vous devez convertir le fichier exporté de `ndjson` en [JSON](/fr-CA/docs/manage-users/user-migration/bulk-user-exports#json-compatible-format). Pour conserver les mêmes ID utilisateur, vous devez supprimer le préfixe `auth0|` de tous les ID utilisateur importés.

  Le processus d’importation ajoute automatiquement le préfixe `auth0|` aux ID utilisateur importés. Si vous ne supprimez pas le préfixe `auth0|` avant l’importation, les ID utilisateur seront renvoyés sous la forme `auth0|auth0|...`
</Callout>

<div id="create-users-json-file">
  ## Créer le fichier JSON des utilisateurs
</div>

Créez un fichier JSON contenant les données des utilisateurs que vous souhaitez importer dans Auth0. La façon d’exporter ces données vers un fichier JSON varie selon votre base de données utilisateur actuelle. Le point de terminaison <Tooltip tip="Management API : Un produit permettant aux clients d’effectuer des tâches administratives." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=Management+API">Management API</Tooltip> attend des segments du fichier JSON. Ainsi, au lieu d’utiliser `fs.readFileSync`, vous devez utiliser `fs.createReadStream`. Le point de terminaison attend un flux de lecture transmis par pipeline plutôt que le fichier JSON au complet.

Pour en savoir plus sur le schéma du fichier JSON et voir des exemples, consultez [Bulk Import Database Schema and Examples](/fr-CA/docs/manage-users/user-migration/bulk-user-import-database-schema-and-examples).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  La taille maximale d’un fichier pour une importation en lot est de 500KB. Vous devrez lancer plusieurs importations si vos données dépassent cette limite.
</Callout>

<div id="request-bulk-user-import">
  ## Demander l’importation groupée d’utilisateurs
</div>

Pour lancer une tâche d’importation en bloc d’utilisateurs, envoyez une requête `POST` au point de terminaison [Create Import Users Job](https://auth0.com/docs/api/management/v2#!/Jobs/post_users_imports). Assurez-vous de remplacer les valeurs fictives `MGMT_API_ACCESS_TOKEN`, `USERS_IMPORT_FILE.json`, `CONNECTION_ID` et `EXTERNAL_ID` par votre <Tooltip tip="Jeton d’accès : identifiant d’autorisation, sous la forme d’une chaîne opaque ou d’un JWT, utilisé pour accéder à une API." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=Access+Token">jeton d’accès</Tooltip> de la Management API, le fichier JSON des utilisateurs, l’[ID de connexion](/fr-CA/docs/authenticate/identity-providers/locate-the-connection-id) de la base de données et l’ID externe, respectivement.

| Paramètre               | Description                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                         |
| ----------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `users`                 | Fichier au format JSON contenant les utilisateurs à importer.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                       |
| `connection_id`         | ID de la connexion dans laquelle les utilisateurs seront ajoutés. Vous pouvez récupérer cet ID à l’aide du point de terminaison [GET /api/v2/connections](https://auth0.com/docs/api/management/v2#!/Connections/get_connections).                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                  |
| `upsert`                | Valeur booléenne; `false` par défaut. Lorsqu’elle est définie sur `false`, l’importation échoue pour les utilisateurs existants qui correspondent à une adresse courriel, à un ID utilisateur, à `phone` ou à un nom d’utilisateur. Lorsqu’elle est définie sur `true`, les utilisateurs existants qui correspondent à une adresse courriel sont mis à jour, mais uniquement avec les attributs pouvant être mis à jour par upsert. Pour obtenir la liste des champs du profil utilisateur qui peuvent être mis à jour par upsert pendant l’importation, consultez [User Profile Structure: User profile attributes](/fr-CA/docs/manage-users/user-accounts/user-profiles/user-profile-structure#user-profile-attributes). Remarque : fournir une entrée utilisateur en double dans le fichier d’importation entraîne une erreur. Dans ce cas, Auth0 n’effectue pas d’insertion suivie d’une mise à jour. |
| `external_id`           | Chaîne facultative définie par l’utilisateur pouvant servir à corréler plusieurs tâches. Elle est renvoyée dans la réponse de status de la tâche.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                          |
| `send_completion_email` | Valeur booléenne; `true` par défaut. Lorsqu’elle est définie sur `true`, un courriel de confirmation est envoyé à tous les propriétaires du tenant une fois la tâche d’importation terminée. Si vous ne voulez *pas* que des courriels soient envoyés, vous devez définir explicitement ce paramètre sur `false`.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |

Si la request réussit, vous recevrez une réponse semblable à la suivante :

```json lines
{
  "status": "pending",
  "type": "users_import",
  "created_at": "",
  "id": "job_abc123",
  "connection_id": "CONNECTION_ID",
  "upsert": false,
  "external_id": "EXTERNAL_ID",
  "send_completion_email": true
}
```

L’entité renvoyée représente la tâche d’importation.

Lorsque la tâche d’importation d’utilisateurs se termine et que `send_completion_email` est défini à `true`, le ou les administrateurs du tenant recevront un courriel les informant que la tâche a réussi ou a échoué. Dans le cas d’une tâche ayant échoué, le courriel peut aviser le ou les administrateurs qu’il a été impossible d’analyser le fichier JSON des utilisateurs lors de l’importation.

<div id="concurrent-import-jobs">
  ### Tâches d’importation simultanées
</div>

Le point de terminaison [Create Import Users Job](https://auth0.com/docs/api/management/v2#!/Jobs/post_users_imports) est limité à deux tâches d’importation simultanées. Si vous demandez des tâches supplémentaires alors que deux sont déjà en attente, une réponse `429 Too Many Requests` est renvoyée :

```json lines
{
  "statusCode": 429,
  "error": "Too Many Requests",
  "message": "There are 2 active import users jobs, please wait until some of them are finished and try again
}
```

Selon l’état de la tâche d’importation d’utilisateurs, vous recevrez une réponse similaire à l’une des suivantes :

**Pending**

```json lines
{
  "status": "pending",
  "type": "users_import",
  "created_at": "",
  "id": "job_abc123",
  "connection_id": "CONNECTION_ID",
  "external_id": "EXTERNAL_ID"
}
```

**Terminée**

Si une tâche est terminée, la réponse sur l’état de la tâche comprendra le total des enregistrements réussis, échoués, insérés et mis à jour.

```json lines
{
  "status": "completed",
  "type": "users_import",
  "created_at": "",
  "id": "job_abc123",
  "connection_id": "CONNECTION_ID",
  "external_id": "EXTERNAL_ID",
  "summary": {
    "failed": 0,
    "updated": 0,
    "inserted": 1,
    "total": 1
  }
}
```

**Échec**

Si une erreur survient dans la tâche, son état sera « failed ». Notez toutefois que des renseignements utilisateur invalides, comme une adresse courriel invalide, ne feront pas échouer l’ensemble de la tâche.

```json lines
{
  "status": "failed",
  "type": "users_import",
  "created_at": "",
  "id": "job_abc123",
  "connection_id": "CONNECTION_ID",
  "external_id": "EXTERNAL_ID",
}
```

Pour obtenir plus de détails sur les entrées ayant échoué, consultez **Récupérer les entrées ayant échoué** ci-dessous.

<div id="job-timeouts">
  ## Délais d’expiration des tâches
</div>

Toutes les tâches d’importation d’utilisateurs expirent après **deux (2) heures**. Si votre tâche ne se termine pas dans ce délai, elle est considérée comme ayant échoué.

De plus, toutes les données associées à votre tâche sont automatiquement supprimées après 24 heures et ne sont plus accessibles par la suite. Par conséquent, **nous vous recommandons fortement d’enregistrer les résultats de la tâche à l’aide du mécanisme de stockage de votre choix**.

<div id="retrieve-failed-entries">
  ## Récupérer les entrées en échec
</div>

<Warning>
  Toutes les données liées à la tâche sont automatiquement supprimées après 24 heures et ne sont plus accessibles après ce délai. C’est pourquoi nous vous recommandons fortement de stocker les résultats de la tâche à l’aide du mécanisme de stockage de votre choix.
</Warning>

<div id="best-practices-for-large-scale-migrations">
  ## Pratiques exemplaires pour les migrations à grande échelle
</div>

Lors de l’importation d’un grand nombre d’utilisateurs (nécessitant 10 tâches ou plus), suivez ces recommandations pour assurer le succès de la migration :

<div id="file-size-and-chunking">
  ### Taille des fichiers et partitionnement
</div>

Répartissez les enregistrements d’utilisateurs en petits lots de moins de 400 KB par fichier. Vous pourrez ainsi traiter environ 1 000 utilisateurs par tâche avec moins de 10 champs de métadonnées par utilisateur, tout en restant bien en deçà de la limite de 500 KB par fichier.

<div id="job-scheduling-and-concurrency">
  ### Ordonnancement des tâches et concurrence
</div>

Comme Auth0 limite les importations d’utilisateurs à deux tâches simultanées par tenant, utilisez un framework d’ordonnancement de tâches (comme [Bull](https://github.com/OptimalBits/bull) ou [Agenda](https://github.com/agenda/agenda)) pour :

* Planifier systématiquement les tâches d’importation
* Faire respecter la limite de concurrence requise
* Suivre la progression de l’importation pour reprendre là où vous vous êtes arrêté en cas d’interruption
* Conserver l’historique des tâches et les status actuels

<div id="error-handling-strategy">
  ### Stratégie de gestion des erreurs
</div>

* N’interrompez pas le processus d’importation en cas d’échec d’un seul utilisateur. Importez tous les utilisateurs et réglez les problèmes par la suite.
* Mettez en place une tâche de « finalisation » qui passe en revue toutes les tâches générées et regroupe les enregistrements en échec dans un nouveau fichier.
* Corrigez les erreurs dans les enregistrements en échec ainsi recueillis et importez-les dans le cadre d’une nouvelle tâche.
* Prévoyez une stratégie de reprise pour gérer les pertes de connexion réseau et les défaillances temporaires.

<div id="upsert-mode-considerations">
  ### Points à considérer concernant le mode `upsert`
</div>

Soyez prudent lorsque vous activez le mode `upsert` :

* Les importations en mode `upsert` sont plus lentes que les importations standard.
* Le mode `upsert` ne permet pas de fusionner `user_metadata` ni `app_metadata`. Les métadonnées existantes seront entièrement remplacées par les nouvelles valeurs.
* Utilisez le mode `upsert` uniquement si vous devez mettre à jour des utilisateurs existants et que vous comprenez bien ces limites.

<div id="data-retention">
  ### Conservation des données
</div>

N’oubliez pas que toutes les données liées aux tâches sont automatiquement supprimées après 24 heures. Conservez les résultats des tâches et les détails des erreurs dans votre propre système de stockage afin d’en assurer le suivi et l’analyse à long terme.
