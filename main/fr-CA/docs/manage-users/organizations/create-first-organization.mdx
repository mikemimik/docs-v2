Pour commencer à utiliser la fonctionnalité [Organizations](/docs/fr-CA/manage-users/organizations/organizations-overview), vous devez créer et configurer vos organisations.

<Card title="La disponibilité varie selon votre forfait Auth0">
  La disponibilité de cette fonctionnalité dépend de votre forfait Auth0 ou de votre entente personnalisée. Pour en savoir plus, consultez [Tarification](https://auth0.com/pricing).
</Card>

<div id="create-an-organization">
  ## Créer une organisation
</div>

Lorsque vous [créez une organisation](/docs/fr-CA/manage-users/organizations/configure-organizations/create-organizations), les paramètres que vous définissez servent à personnaliser la page de connexion et les modèles de courriel que les utilisateurs finaux voient lorsqu’ils s’authentifient dans le contexte de l’organisation.

Lorsque vous utilisez Organizations prêt à l’emploi, ces paramètres remplacent ceux des pages <Tooltip tip="Universal Login : votre application redirige vers Universal Login, hébergé sur l’Authorization Server d’Auth0, pour vérifier l’identité d’un utilisateur." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Universal+Login">Universal Login</Tooltip>. Pour en savoir plus sur la personnalisation de la page de connexion ou des modèles de courriel, consultez [Personnaliser les prompts et les modèles de courriel](#customize-prompts-and-email-templates).

<div id="define-organization-behavior">
  ## Définir le comportement de l’organisation
</div>

Vous connaissez peut-être des applications comme Heroku, dont le comportement varie selon qu’un utilisateur se connecte avec un compte personnel ou choisit une organisation associée à son compte professionnel. Auth0 vous permet de définir un comportement semblable pour les organisations au niveau de l’application.

Pour chaque application que vous créez dans Auth0, vous pouvez décider si les utilisateurs finaux doivent se connecter directement ou s’ils doivent obligatoirement s’authentifier dans le contexte d’une organisation. Pour les applications qui exigent que les utilisateurs se connectent par l’intermédiaire d’une organisation, vous pouvez aussi préciser ce qui se passe si aucune organisation n’est fournie au flux de connexion.

Vous pouvez aussi vouloir mettre à jour les **Allowed Callback URLs** et les **Allowed Origins (CORs)** de votre application dans les [Application Settings](/docs/fr-CA/get-started/applications/application-settings) afin d’inclure les [espaces réservés d’organisation pour les sous-domaines](/docs/fr-CA/get-started/applications/wildcards-for-subdomains).

<div id="customize-prompts-and-email-templates">
  ## Personnaliser les prompts et les modèles de courriel
</div>

Lorsque vous utilisez Organizations avec les prompts prêts à l’emploi de Universal Login, les paramètres de marque que vous configurez lorsque vous [créez des organisations](/docs/fr-CA/manage-users/organizations/configure-organizations/create-organizations) remplacent la marque des pages Universal Login et des modèles de courriel. Si vous souhaitez personnaliser davantage les pages Universal Login et les courriels que reçoit l’utilisateur final, vous pouvez personnaliser les modèles de page et de courriel.

<div id="page-templates">
  ### Gabarits de page
</div>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Pour accéder à la personnalisation des gabarits de page, vous devez utiliser des domaines personnalisés, une fonctionnalité incluse dans tout forfait payant.
</Callout>

Pour modifier la page Universal Login, personnalisez un gabarit de page à l’aide du [langage de gabarits Liquid](https://shopify.github.io/liquid/) et des variables de gabarit, puis appliquez-le à l’aide de l’API Universal Login Page Templates. Pour en savoir plus, consultez [Universal Login Page Templates](/docs/fr-CA/customize/login-pages/universal-login/customize-templates).

Comme le même gabarit est utilisé pour toutes les pages, vous pouvez créer des pages de connexion cohérentes avec un minimum d’effort.

Le gabarit le plus simple que vous puissiez écrire est :

```html lines
<!DOCTYPE html>
<html>
  <head>
    {%- auth0:head -%}
  </head>
  <body>
    {%- auth0:widget -%}
  </body>
</html>
```

Les balises suivantes doivent être présentes dans le modèle :

* `auth0:widget` : Contient le HTML qui structure le widget affiché sur chaque type de page (p. ex., Login, Reset Password).
* **`auth0:head`** : Contient les balises nécessaires pour afficher le widget.

Pour centrer le widget sur la page, remplacez la balise `<body>` par `<body class="_widget-auto-layout">`.

<div id="email-templates">
  ### Modèles de courriel
</div>

Pour modifier les courriels que reçoit l’utilisateur final, personnalisez un modèle de courriel à l’aide du [langage de gabarits Liquid](https://shopify.github.io/liquid/) et des variables de gabarit. Pour en savoir plus, consultez [Personnaliser les modèles de courriel](/docs/fr-CA/customize/email/email-templates).

Les modèles de courriel suivants peuvent être personnalisés davantage pour les organisations :

* **Bienvenue** : Reçu par l’utilisateur final après qu’il a vérifié son adresse courriel ou, si la vérification du courriel est désactivée, au moment de son inscription (ou de sa première connexion).
* **Changement de mot de passe** : Reçu par l’utilisateur final lorsqu’il demande un changement de mot de passe. Contient un lien qui le redirige vers la page de réinitialisation du mot de passe.
* **Inviter un utilisateur** : Reçu par l’utilisateur final lorsqu’il est invité dans une organisation. Contient un lien qui le redirige vers la route de connexion par défaut que vous avez configurée. Pour en savoir plus, consultez [Inviter des membres d’une organisation](/docs/fr-CA/manage-users/organizations/configure-organizations/invite-members).

<div id="template-variables">
  ### Variables de modèle
</div>

Les modèles de page et de courriel peuvent accéder à un ensemble de variables de contexte que vous pouvez utiliser pour modifier le rendu du modèle.

Pour obtenir la liste des variables disponibles, consultez [Universal Login Page Templates](/docs/fr-CA/customize/login-pages/universal-login/customize-templates) pour les variables de gabarit de page et [Personnaliser les modèles de courriel](/docs/fr-CA/customize/email/email-templates) pour les variables communes des modèles de courriel.

Des variables supplémentaires sont accessibles au gabarit de page lorsqu’un utilisateur se connecte par l’intermédiaire d’une organisation :

* `organization.id`
* `organization.display_name`
* `organization.name`
* `organization.metadata`
* `organization.branding.logo_url`
* `organization.branding.colors.primary`
* `organization.branding.colors.page_background`

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Les paramètres d’organisation ne sont exposés à un modèle de courriel que lorsque le courriel associé est déclenché dans le contexte d’une organisation. Par exemple, le courriel d’invitation d’un utilisateur est toujours déclenché dans le contexte d’une organisation; les paramètres d’organisation sont donc toujours exposés à son modèle. Toutefois, les courriels de bienvenue peuvent être déclenchés ou non dans le contexte d’une organisation; les paramètres d’organisation ne seront donc pas toujours accessibles dans leur modèle.
</Callout>

<div id="configure-organization-domains">
  ## Configurer les domaines d’Organization
</div>

<Warning>
  Les domaines d’Organization pour Discovery sont actuellement en Early Access. En utilisant cette fonctionnalité, vous acceptez les conditions applicables de l’essai gratuit énoncées dans le [Master Subscription Agreement d’Okta](https://www.okta.com/legal). Pour en savoir plus sur les étapes de publication du produit, consultez [Product Release Stages](/docs/fr-CA/troubleshoot/product-lifecycle/product-release-stages).
</Warning>

Pendant le pre-login, Auth0 peut détecter les Organizations d’un utilisateur soit à partir du nom de l’Organization, soit à partir du domaine de courriel saisi. Pour associer une Organization à un domaine, vous pouvez créer des domaines d’Organization. Auth0 les dirige ensuite automatiquement vers le bon Identity Provider (IdP).

<Note>
  Remarque : les utilisateurs doivent tout de même saisir leur adresse courriel complète ou le nom de l’Organization pendant le login. Auth0 utilise cette information pour déterminer l’Organization associée.
</Note>

<div id="add-a-domain-to-an-organization">
  ### Ajouter un domaine à une Organization
</div>

Les administrateurs de tenant peuvent ajouter et gérer des domaines dans la section Domains d’une Organization dans l’Auth0 Dashboard ou au moyen de la Management API.

Pour ajouter un domaine dans l’Auth0 Dashboard :

1. Accédez à [Auth0 Dashboard &gt; Organizations](https://manage.auth0.com/#/organizations).
2. Sélectionnez le nom de l’Organization, puis sélectionnez **Domains**.
3. Choisissez **Add Domain**.
4. Saisissez le domaine (par exemple, `travel0.com`)
5. Une fois vérifié, le domaine s’affichera comme Verified et sera utilisé pour la découverte d’Organization en pre-login.

<Note>
  Seuls les domaines vérifiés sont utilisés lors de la découverte de domaine. Les domaines en attente ou non vérifiés ne déclencheront pas le routage.
</Note>

Pour savoir comment la découverte de domaine influe sur le comportement de login, consultez [Login Flows for Organizations](https://auth0.com/docs/manage-users/organizations/login-flows-for-organizations).

<div id="configure-organization-membership">
  ## Configurer l’appartenance à l’organisation
</div>

Chaque organisation a ses propres membres, c’est-à-dire les utilisateurs qui peuvent accéder à vos applications en se connectant au moyen de la page de connexion configurée pour l’organisation.

Les applications SaaS et B2B sont généralement offertes à des équipes plutôt qu’à des utilisateurs individuels. Dans ce contexte, une équipe peut correspondre à une entreprise entière, à un petit groupe d’employés (par exemple, le département du marketing), ou même à un groupe temporaire d’utilisateurs réunis autour d’un objectif commun (par exemple, un groupe de voisins qui se sont inscrits pour collaborer avec un organisme de bienfaisance et qui rivalisent en équipe pour amasser le plus d’argent pour une cause).

Les Organizations dans Auth0 sont assez souples pour prendre en charge tous ces cas d’utilisation, mais la création d’une application conçue pour des équipes demande une certaine planification. L’un des premiers éléments à considérer est la façon de gérer l’appartenance à l’équipe, ce que vous pouvez faire en :

* [Invitant des utilisateurs par courriel](/docs/fr-CA/manage-users/organizations/configure-organizations/invite-members)
* [Accordant une appartenance juste-à-temps](/docs/fr-CA/manage-users/organizations/configure-organizations/grant-just-in-time-membership) aux utilisateurs qui se connectent au moyen d’une connexion activée
* En gérant directement l’appartenance au moyen de la <Tooltip tip="Management API : Un produit permettant aux clients d’effectuer des tâches administratives." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Management+API">Management API</Tooltip> ou du <Tooltip tip="Management API : Un produit permettant aux clients d’effectuer des tâches administratives." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip> en [attribuant des membres](/docs/fr-CA/manage-users/organizations/configure-organizations/assign-members) ou en [supprimant des membres](/docs/fr-CA/manage-users/organizations/configure-organizations/remove-members)

Une fois l’appartenance établie, vous pouvez [récupérer les membres des organisations](/docs/fr-CA/manage-users/organizations/configure-organizations/retrieve-members) ou [récupérer l’appartenance d’un utilisateur à une organisation](/docs/fr-CA/manage-users/organizations/configure-organizations/retrieve-user-membership).

<div id="assign-roles-to-members">
  ## Attribuer des rôles aux membres
</div>

Chaque membre de l’organisation peut se voir attribuer un ou plusieurs rôles, qui s’appliquent lorsque les utilisateurs se connectent par l’intermédiaire de l’organisation. Plus précisément, vous pouvez [ajouter des rôles à des membres de l’organisation](/docs/fr-CA/manage-users/organizations/configure-organizations/add-member-roles), [retirer des rôles à des membres de l’organisation](/docs/fr-CA/manage-users/organizations/configure-organizations/remove-member-roles) ou [récupérer les rôles des membres d’une organisation](/docs/fr-CA/manage-users/organizations/configure-organizations/retrieve-member-roles). Vous pouvez utiliser les rôles et les permissions qui y sont associées pour contrôler l’accès aux [applications que vous développez](/docs/fr-CA/manage-users/organizations/custom-development). Pour en savoir plus sur les rôles, consultez [le contrôle d’accès basé sur les rôles](/docs/fr-CA/manage-users/access-control/rbac).
