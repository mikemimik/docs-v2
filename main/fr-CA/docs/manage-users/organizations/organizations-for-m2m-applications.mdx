Le [Client Credentials Flow](/fr-CA/docs/get-started/authentication-and-authorization-flow/client-credentials-flow) permet à une application d’obtenir des <Tooltip tip="Jeton d’accès : information d’autorisation, sous la forme d’une chaîne opaque ou d’un JWT, utilisée pour accéder à une API." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=access+tokens">jetons d’accès</Tooltip> pour elle-même plutôt que pour un utilisateur. Cela est utile dans les cas d’utilisation machine à machine, comme les robots, les CLI, les processus backend ou démons, ou toute application qui consomme des API pour son propre compte. Même si les utilisateurs finaux n’interviennent pas, le Client Credentials Flow doit tout de même être limité à une organisation afin de garantir que seules les applications autorisées peuvent accéder aux données de chaque organisation.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Pour limiter chaque requête à une organisation donnée, incluez le claim `org_id` correspondant dans les jetons machine à machine. Les serveurs d’API doivent tenir compte de ce claim et l’appliquer lorsqu’ils autorisent l’accès aux ressources et aux données de l’API. Pour en savoir plus, consultez [Utiliser les jetons et les Organizations](/fr-CA/docs/manage-users/organizations/using-tokens).
</Callout>

L’accès machine à machine pour Organizations vous permet de définir, pour chaque API, les organisations auxquelles une application donnée peut accéder à l’aide du Client Credentials Flow. Pour en savoir plus, consultez ce qui suit :

<div id="open-up-apis-to-third-party-applications">
  ## Ouvrir les API aux applications tierces
</div>

À l’aide d’Auth0 Organizations et de l’<Tooltip tip="Management API : Un produit permettant aux clients d’effectuer des tâches administratives." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=Management+API">Management API</Tooltip>, Travel0 offre un portail en libre-service où les clients peuvent créer et gérer leur propre organisation. Travel0 veut permettre aux clients de créer facilement des bots pour aider les utilisateurs finaux à trouver et à acheter des aventures. Dans le cadre de ce portail, Travel0 permet donc aux clients d’enregistrer leurs applications (p. ex. les bots) pour utiliser la Travel0 API en leur propre nom au moyen d’un accès machine à machine. Dans ce cas d’utilisation, l’accès entre les organisations doit être contrôlé adéquatement afin que les applications appartenant à une organisation ne puissent accéder qu’aux données de cette organisation par l’intermédiaire de la Travel0 API.

**Machine-to-Machine Access for Organizations vous permet de configurer l’accès Client Credentials pour chaque API en l’associant à une organisation précise**. Dans le diagramme suivant, les applications de notre organisation d’exemple, org&#95;X, ne peuvent accéder à la Travel0 API que dans le contexte de org&#95;X. Vous pouvez également configurer l’accès d’une même application à plusieurs organisations lorsque des agrégateurs sont utilisés. Avec M2M Access for Organizations, vous contrôlez, pour chaque API, quelles applications peuvent accéder à une organisation précise au moyen d’un accès machine à machine.

<div id="isolation-of-organizations-for-internal-applications">
  ## Isolation des organisations pour les applications internes
</div>

Travel0 dispose aussi de certains processus internes et d’outils CLI qui doivent accéder à l’API Travel0 au moyen du Client Credentials Flow. Pour appliquer à l’API une stratégie unifiée de contrôle d’accès, Travel0 veut que les requêtes provenant de ses propres applications soient restreintes à une organisation précise afin de garantir que seules les données appropriées sont consultées dans chaque cas.

<Frame>![](/docs/images/cdy7uua7fh8z/5bMEPlmTLGgBn8fq5Fp59S/fe874efcfba549114ad062bd7e3d874d/image7.png)</Frame>

Machine-to-Machine Access to Organizations vous permet de configurer une application pour accéder à n’importe quelle organisation pour chacune des API de votre tenant au moyen du Client Credentials Flow.
