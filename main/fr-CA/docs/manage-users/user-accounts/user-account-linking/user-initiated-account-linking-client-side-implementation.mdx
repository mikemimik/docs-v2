title: 'Liaison de comptes initiée par l’utilisateur : implémentation côté client'
---

Auth0 prend en charge la liaison de comptes d’utilisateurs provenant de divers <Tooltip tip="Identity Provider (IdP) : service qui stocke et gère les identités numériques." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=identity+providers">fournisseurs d’identité</Tooltip>. Une façon de mettre en œuvre cette fonctionnalité consiste à permettre à l’utilisateur de lier explicitement des comptes. Dans ce scénario, l’utilisateur s’authentifie au moyen de l’interface utilisateur de votre application monopage (SPA), puis peut utiliser un lien ou un bouton pour lier un autre compte au premier. Lorsque l’utilisateur clique sur ce lien ou ce bouton, votre application effectue un appel afin que, lorsque l’utilisateur ouvre une session avec le deuxième fournisseur, le deuxième compte soit lié au premier.

Lorsque vous initiez la liaison de comptes, vous pouvez choisir quelle identité utiliser comme compte principal et laquelle utiliser comme compte secondaire. Ce choix dépend de l’ensemble d’attributs que vous souhaitez conserver dans le profil principal, puisque seuls les attributs du compte principal seront conservés.

Créez un exemple d’application monopage React à l’aide de ce projet [sur GitHub](https://github.com/auth0-samples/auth0-link-accounts-sample/tree/master/SPA).

Dans ce projet, ouvrez le fichier `auth_config.json` dans votre éditeur de texte et mettez à jour le domaine et le `client_id` avec vos informations.

Ajoutez `http://localhost:3000` dans les champs « Allowed Callback URLs », « Allowed Logout URLs » et « Allowed Web Origins ».

1. Connectez l’utilisateur à votre application.<br /><br />

   L’utilisateur s’authentifie à votre SPA au moyen de [Universal Login](/fr-CA/docs/authenticate/login/auth0-universal-login), en demandant un [Access Token for the Management API](https://auth0.com/docs/api/management/v2/get-access-tokens-for-spas).<br /><br />

   Dans un scénario de connexion SPA typique, le callback est géré côté client par la même page, et un JWT est reçu après une authentification réussie. Pour en savoir plus, consultez le [Quickstart pour les single-page apps](/fr-CA/docs/quickstart/spa).
2. L’utilisateur lance la liaison de comptes. Votre SPA doit fournir une interface utilisateur permettant à l’utilisateur de lancer une association avec ses autres comptes. Par exemple, votre SPA pourrait contenir une page des paramètres de l’utilisateur :<br /><br />

   Lorsque l’utilisateur clique sur le bouton **Associer un compte**, votre application le redirige vers la page Universal Login, où il se connecte avec la connexion qu’il souhaite associer. Après une authentification réussie, utilisez le token obtenu pour associer les comptes.<br /><br />

   Vous pourriez aussi ajouter un bouton pour chaque connexion (p. ex. « Associer un compte Facebook », « Associer un compte Google ») et rediriger l’utilisateur vers `/authorize` avec le paramètre `connection` défini (p. ex. `/authorize?connection=facebook`).

   <Frame>![Exemple de page de profil avec association de comptes utilisateur](/docs/images/cdy7uua7fh8z/4qxdXGBnlVRDaTkw2wv9WP/8792cff7c5fc21910ec12495828539da/account-linking-spa.png)</Frame>
3. Associez les comptes en appelant le [point de terminaison Link a User Account](https://auth0.com/docs/api/v2#!/Users/post_identities) de l’Auth0 Management API.

   <Warning>
     Pour conserver et fusionner le `user_metadata` du compte secondaire, vous devez le récupérer et le fusionner dans les métadonnées du compte principal avant d’appeler le point de terminaison de l’API. Une fois les comptes associés, les métadonnées du compte secondaire sont supprimées.

     Lorsque vous lancez la liaison de comptes, vous pouvez choisir quelle identité sera utilisée comme compte principal et laquelle sera secondaire. Ce choix dépendra de l’ensemble des attributs que vous souhaitez conserver dans le profil principal.
   </Warning>
4. Dans la fonction `linkAccount`, appelez la Management API. Authentifiez-vous auprès de l’API au moyen du JWT principal, qui est l’Access Token, puis effectuez la liaison au moyen de l’ID de l’utilisateur principal et du JWT secondaire, qui est l’ID Token de l’utilisateur secondaire.

   ```javascript lines expandable
   const linkAccount = async () => {
     const accessToken = await auth0.getTokenSilently();
     const { sub } = await auth0.getUser();
     // authenticateUser doit authentifier l'utilisateur avec le compte à associer
     // Consultez l'exemple Github pour plus de détails
     const {
       __raw: targetUserIdToken,
       email_verified,
       email,
     } = await authenticateUser();
     if (!email_verified) {
       throw new Error(
         `L'association de comptes est autorisée uniquement avec un compte vérifié. Veuillez vérifier votre courriel ${email}.`
       );
     }
     await fetch(`https://${config.domain}/api/v2/users/${sub}/identities`, {
       method: "POST",
       headers: {
         "Content-Type": "application/json",
         Authorization: `Bearer ${accessToken}`,
       },
       body: JSON.stringify({
         link_with: targetUserIdToken,
       }),
     });
   };
   ```

<Card title="Liaison de comptes : Access Tokens vs. ID Tokens">
  Auparavant, dans certains cas, vous pouviez utiliser des ID Tokens pour associer et dissocier des comptes utilisateur. Cette fonctionnalité est en voie d’être deprecated. Vous devrez utiliser des Access Tokens dans tous les cas. Le changement concernant la dissociation de comptes est que vous ne pouvez plus utiliser un ID Token dans l’en-tête Authorization. Un Access Token doit être utilisé à la place. La fonctionnalité est toujours offerte, mais elle n’est pas recommandée, et les utilisateurs concernés sont encouragés à migrer. Consultez le [Guide de migration : liaison de comptes avec Access Tokens vs. ID Tokens](/fr-CA/docs/troubleshoot/product-lifecycle/past-migrations/link-user-accounts-with-access-tokens-migration) pour plus de détails.
</Card>
