**Date d’entrée en vigueur** : 4 février 2019

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Pour les clients de Private Cloud, veuillez consulter la [Private Cloud Load Testing Policy](/fr-CA/docs/deploy-monitor/deploy-private-cloud/private-cloud-on-aws#load-testing).
</Callout>

Auth0 reconnaît que ses clients peuvent parfois avoir besoin d’effectuer des tests de charge sur son service infonuagique de production. Afin d’assurer le bon déroulement des tests et de maintenir un niveau de service élevé pour tous les clients, Auth0 a établi les lignes directrices suivantes. Tout test de charge effectué dans Auth0 doit respecter la présente politique.

Seuls les clients ayant souscrit un abonnement Enterprise peuvent effectuer des tests de charge. Les clients ayant un abonnement Enterprise peuvent demander un test de charge (avec un maximum de 2 répétitions) par année sur un tenant de production Auth0. Les tests de performance et de charge ne sont autorisés qu’avec l’approbation écrite préalable d’Auth0. Une fois approuvés, les tests ne peuvent viser que les tenants que nous avons approuvés.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Auth0 se réserve le droit de rejeter la demande de test de charge ou d’exiger des modifications. Le non-respect de cette politique peut entraîner le blocage temporaire de l’accès à un tenant jusqu’à ce que le problème soit corrigé.
</Callout>

<div id="acceptable-testing-windows">
  ## Fenêtres de test autorisées
</div>

Les clients du cloud public doivent s’organiser pour effectuer leurs tests de charge pendant les fenêtres de test hors pointe suivantes :

| Région    | Fenêtre de test   |
| --------- | ----------------- |
| `prod-au` | 9:00 - 17:00 UTC  |
| `prod-eu` | 21:00 - 5:00 UTC  |
| `prod-us` | 2:00 - 10:00 UTC  |
| `prod-jp` | 14:00 - 22:00 UTC |
| `prod-ca` | 2:00 - 10:00 UTC  |
| `prod-uk` | 21:00 - 5:00 UTC  |

Dans des circonstances très exceptionnelles, Auth0 peut accorder des dérogations à cette règle.

<div id="change-freeze-policy">
  ## Politique relative au gel des changements
</div>

<Warning>
  Les tests de charge ne sont pas autorisés pendant les périodes de gel des changements.
</Warning>

Pour consulter les périodes de gel des changements actuellement prévues, consultez la [Politique relative au gel des changements](/fr-CA/docs/troubleshoot/customer-support/operational-policies/change-freeze-policy).

<div id="holiday-load-testing-policy">
  ## Politique relative aux tests de charge pendant le temps des Fêtes
</div>

**Mise à jour :** 1 novembre 2022

Pendant le temps des Fêtes, les demandes de tests de charge doivent respecter les exigences suivantes :

* Être soumises au moins 2 semaines avant la date de test souhaitée. Un préavis de quatre (4) semaines ou plus est préférable afin de prévoir le temps nécessaire pour un examen approfondi et toute modification requise.
* Les tests de charge approuvés sont permis uniquement les mardis, mercredis et jeudis, durant les heures creuses de l’environnement ciblé.
* Si toutes les demandes de RPS sont jugées comme respectant les limites de RPS de façon sécuritaire, le test de charge sera autorisé, à condition qu’il ait été examiné et approuvé par toutes les équipes produit/plateforme responsables des points de terminaison visés par le test.

  * Les tests qui dépassent les seuils sécuritaires de RPS seront rejetés.
* Un seul test de charge par client, par environnement, sera planifié dans une même journée.

  * S’il est nécessaire d’exécuter plus d’un test à la fois, la demande doit être transmise à un échelon supérieur et approuvée par un directeur de plateforme ou un vice-président.

<div id="submit-load-testing-request">
  ## Soumettre une demande de test de charge
</div>

Vous pouvez soumettre une demande de test de charge dans le [Auth0 Support Center](https://support.auth0.com).

[Ouvrir une nouvelle demande](/fr-CA/docs/troubleshoot/customer-support/open-and-manage-support-tickets), sélectionnez **Soutien produit** dans le menu déroulant **Sur quoi pouvons-nous vous aider?**, puis fournissez tous les renseignements indiqués ci-dessous.

<Warning>
  Pour que votre demande puisse être approuvée, vous devez :

  * Soumettre votre demande au moins 2 semaines avant la date prévue du test.
  * Obtenir une approbation écrite avant d’effectuer tout test.
  * Respecter nos [limites de débit publiées en production](/fr-CA/docs/troubleshoot/customer-support/operational-policies/rate-limit-policy).
</Warning>

<div id="required-load-testing-information">
  ### Renseignements requis pour la demande de test de charge
</div>

Votre demande de test de charge doit inclure les renseignements suivants.

<div id="general">
  #### Généralités
</div>

* Une description du test à effectuer
* Les personnes-ressources qui seront disponibles pendant le test et la façon de les joindre
* La date et l’heure souhaitées pour le test, y compris le fuseau horaire
* La durée souhaitée du test (maximum de 2 heures)
* Les plateformes à utiliser pour le test (ordinateur de bureau/portable, iOS, Android, autres)

<div id="tenant">
  #### Tenant
</div>

* Le nom et la région du tenant Auth0 à utiliser lors du test

<div id="features">
  #### Fonctionnalités
</div>

* Les fonctionnalités Auth0 (comme Actions, Rules ou Email) utilisées pendant le test
* Les types de connexions Auth0 utilisés dans le test
* Quelle base de données personnalisée, le cas échéant, sera utilisée
* Si vous utilisez un <Tooltip tip="Domaine personnalisé : domaine tiers avec un nom spécialisé ou personnalisé." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=custom+domain">domaine personnalisé</Tooltip>
* Quelles Auth0 Rules ou Actions, le cas échéant, seront exécutées pendant le test
* Quels Auth0 Webtasks, le cas échéant, seront utilisés
* Si des courriels de vérification, de bienvenue ou autres seront envoyés

<div id="apis">
  #### API
</div>

* Les méthodes et les points de terminaison de l’Auth0 API à utiliser (par exemple `GET /api/v2/clients`)
* Le nombre maximal de requêtes par seconde pour chaque type de requête ou de point de terminaison

<div id="users">
  #### Utilisateurs
</div>

* Nombre d’utilisateurs distincts prenant part au test de charge

<div id="load-estimates">
  #### Estimations de charge
</div>

* La charge de pointe, exprimée en requêtes par seconde, attendue pour chaque point de terminaison d’API ou chaque fonctionnalité Auth0 visée par le test
* Une explication ou justification des valeurs de charge de pointe, y compris la taille de la population d’utilisateurs cible et des estimations réalistes du nombre de connexions par heure
* Le taux de montée en charge du test
* Le nombre d’utilisateurs uniques participant au test de charge

<div id="test-requirements">
  ## Exigences relatives aux tests
</div>

Les créneaux de test de charge sont offerts selon les disponibilités; il est donc fortement recommandé d’aviser à l’avance. Une fois approuvés, les créneaux de test de charge auront une heure de début et une heure de fin prévues, pour une durée maximale de deux (2) heures. Tous les tests doivent commencer et se terminer pendant ce créneau. Pendant le test de charge, faites alterner vos adresses IP afin que votre test ne se heurte pas aux [limites de taux basées sur l’IP](/fr-CA/docs/troubleshoot/customer-support/operational-policies/rate-limit-policy).

Auth0 recommande fortement d’inclure une courte période de « montée en charge » jusqu’aux valeurs cibles du test de charge. Par exemple, un test de charge de 100 RPS pourrait être précédé de trois périodes de cinq minutes : 5 minutes à 25 RPS, 5 minutes à 50 RPS et 5 minutes à 75 RPS. Cette période de montée en charge permet à Auth0 et au client d’observer et de comparer les effets à des niveaux de RPS croissants avant d’atteindre le RPS de pointe. S’il n’est pas possible de prévoir une période de montée en charge, veuillez indiquer pourquoi.

<div id="configure-email-provider">
  ### Configurer le fournisseur de courriel
</div>

Avant d’effectuer tout test, vous devez :

* Configurer votre propre fournisseur de courriel dans Auth0
* Obtenir l’approbation de votre fournisseur de courriel pour envoyer le volume de courriels prévu
* Prévoir le traitement des courriels rejetés
* Mettre en place un mécanisme pour vérifier que les courriels ont bien été reçus
