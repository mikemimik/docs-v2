title: Politique sur les limites d’entités
---

Les entités dans Auth0 sont des éléments de configuration du tenant, comme Applications, Connections, Organizations, les API <Tooltip tip="Serveur de ressources : serveur hébergeant des ressources protégées. Les serveurs de ressources acceptent les requêtes visant des ressources protégées et y répondent." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=resource+servers">serveurs de ressources</Tooltip>, Permissions, Roles et Actions.

<Warning>
  Les limites d’entités sont des limites strictes appliquées aux entités pour assurer le bon fonctionnement de notre plateforme; elles ne font pas partie de la tarification, bien qu’elles puissent varier selon les groupes de plans tarifaires. Les plans tarifaires individuels comportent des limites distinctes, qui correspondent aux droits inclus, comme les MAU et les Enterprise Connections, que vous recevez lorsque vous vous abonnez à un plan donné. (Pour connaître les limites de votre plan tarifaire, consultez [Auth0 Pricing](https://auth0.com/pricing/).)
</Warning>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Toute utilisation du terme « illimité » signifie qu’il n’y a pas de limites strictes, mais qu’Okta applique tout de même des limites système quant au seuil. Veuillez communiquer avec le soutien si vous avez des questions sur ces limites.
</Callout>

<div id="global-subscription-plan-limits">
  ## Limites globales des plans d’abonnement
</div>

Cette section énumère les limites qui s’appliquent à tous les plans d’abonnement.

<div id="organizations">
  ### Organizations
</div>

| Entité                                                      | Limite  |
| ----------------------------------------------------------- | ------- |
| Organizations par tenant                                    | 100,000 |
| Membres par Organization                                    | 100,000 |
| Connexions par Organization                                 | 10      |
| Attributions de rôles par membre d’Organization             | 50      |
| Attributions de rôles par Enterprise Group                  | 10      |
| Rôles uniques pouvant être attribués aux Enterprise Groups  | 100     |
| Enterprise Groups auxquels des rôles peuvent être attribués | 10,000  |
| Client Grants M2M par Organization (Public Cloud)           | 100     |
| Client Grants M2M par Organization (Private Cloud)          | 1000    |
| Profils d’échange de jetons personnalisés                   | 100     |

Les clients abonnés aux forfaits Enterprise peuvent demander une augmentation des limites d’entités pour les Organizations par tenant et les membres par Organization en [contactant le soutien](https://support.auth0.com/).

Ces limites peuvent être augmentées à 2,000,000 Organizations par tenant et à 2,000,000 membres par Organization dans le Public Cloud. Dans les instances Private Cloud, ces entités sont illimitées.

<div id="authorization-core-role-based-access-control-rbac">
  ### Contrôle d’accès basé sur les rôles (RBAC) d’Authorization Core
</div>

| Élément                                                      | Limite |
| ------------------------------------------------------------ | ------ |
| Rôles par tenant                                             | 1000   |
| Scopes par API (serveurs de ressources)                      | 1000   |
| Rôles par utilisateur                                        | 50     |
| Rôles par Enterprise Group                                   | 10     |
| Rôles uniques pouvant être attribués à des Enterprise Groups | 100    |
| Enterprise Groups auxquels des rôles peuvent être attribués  | 10,000 |
| Permissions par utilisateur                                  | 1000   |
| Permissions par rôle                                         | 1000   |

Les limites d’Enterprise Group et de rôle ci-dessus s’appliquent à l’utilisation de RBAC sans la fonctionnalité Organizations. L’utilisation de la fonctionnalité Organizations permet d’appliquer ces limites par Organization plutôt que par tenant.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Notez que les limites relatives aux Permissions par utilisateur s’appliquent aux Permissions attribuées directement.

  Techniquement, un utilisateur pourrait avoir plus de Permissions que ce qui est indiqué si vous attribuez les Permissions à différents rôles, puis attribuez ces rôles à l’utilisateur.
</Callout>

| Élément                                               | Limite |
| ----------------------------------------------------- | ------ |
| Jeton d’actualisation par utilisateur par Application | 200    |

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Si vous atteignez la limite de <Tooltip tip="Jeton d’actualisation : jeton utilisé pour obtenir un nouveau jeton d’accès sans obliger les utilisateurs à se reconnecter." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=Refresh+tokens">jetons d’actualisation</Tooltip> et créez un nouveau jeton d’actualisation,
  le système révoque ou supprime le jeton le plus ancien de cet utilisateur dans
  l’Application.

  Les jetons révoqués et les jetons expirés ne comptent pas dans la limite.
</Callout>

<div id="actions">
  ### Actions
</div>

| Entité                                      | Limite |
| ------------------------------------------- | ------ |
| Actions par tenant                          | 100    |
| Actions par Trigger par tenant              | 20     |
| Actions Modules par tenant                  | 100    |
| Actions Modules par Action                  | 3      |
| Versions par Action                         | 50     |
| Versions par Action Module                  | 50     |
| Secrets par Action                          | 30     |
| Secrets par Action Module                   | 30     |
| Dépendances (modules NPM) par Action        | 10     |
| Dépendances (modules NPM) par Action Module | 10     |

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Les limites des Actions et des Actions Modules incluent à la fois les entités déployées et non déployées. Lorsque vous les supprimez, elles ne sont plus prises en compte dans la limite correspondante.

  Les limites des versions d’Action et d’Action Module incluent à la fois les versions en brouillon et les versions actives. Une fois la limite atteinte, l’ajout de chaque version supplémentaire entraîne la suppression par Auth0 de la plus ancienne version inutilisée.
</Callout>
