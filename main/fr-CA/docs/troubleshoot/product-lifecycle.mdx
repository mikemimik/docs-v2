Dans le développement des produits Auth0, nous nous engageons à :

* Offrir rapidement et régulièrement de la valeur à nos clients, en itérant selon leurs commentaires
* Chercher à bien comprendre nos clients et en tenir compte dans chacune de nos décisions
* Recueillir et analyser des données sans relâche afin de prendre de meilleures décisions
* Visualiser et concevoir les versions actuelles, idéalisées et futures de l’ensemble de notre produit lors de l’ajout de fonctionnalités

Ce document explique ce qui constitue des changements rétrocompatibles (sans rupture) et des changements non rétrocompatibles (avec rupture de compatibilité) dans la plateforme Auth0. La distinction entre ces types de changements aide les développeurs à anticiper les répercussions possibles sur leurs applications lorsqu’ils utilisent les services Auth0.

Ce document présente les changements courants et leur compatibilité afin d’aider à établir les attentes en matière d’intégration. Les changements décrits ne sont pas exhaustifs; pour en savoir plus sur les changements avec rupture de compatibilité, consultez les sujets ci-dessous.

<div id="auth0-apis">
  ## API Auth0
</div>

Auth0 offre et maintient des API comme les API Authentication et Management, qui définissent chacune un contrat entre Auth0 et ses clients selon les spécifications d’API documentées et les artefacts connexes. Auth0 suit des normes reconnues provenant d’organismes comme l’IETF et OIDC ; lorsqu’il n’existe pas de norme, des API propriétaires sont développées conformément aux pratiques exemplaires.

Des modifications à ces API, qu’elles soient rétrocompatibles ou non rétrocompatibles, sont parfois nécessaires afin d’améliorer leur fonctionnalité, leur sécurité ou leurs performances.

<div id="backward-compatible-non-breaking-changes">
  ### Changements rétrocompatibles et sans rupture
</div>

Un changement rétrocompatible ne perturbe pas l’interopérabilité entre la plateforme Auth0 et les applications clientes, et n’oblige pas les clients à prendre des mesures immédiates ni à participer au processus de migration.

Voici quelques exemples de changements sans rupture :

* **Chaînes opaques** : Le format et la taille des chaînes opaques (p. ex., tokens, IDs) peuvent changer. Les clients ne doivent pas supposer une taille ou un format fixes. La taille maximale des chaînes opaques ne dépassera pas 4096 caractères.
* **Taille des <Tooltip tip="JSON Web Token (JWT) : Format standard de ID Token (et souvent format de Access Token) utilisé pour représenter de manière sécuritaire des claims entre deux parties." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=JWT">JWT</Tooltip>** : Conformément à la spécification <Tooltip tip="JSON Web Token (JWT) : Format standard de ID Token (et souvent format de Access Token) utilisé pour représenter de manière sécuritaire des claims entre deux parties." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=OAuth">OAuth</Tooltip> (RFC6749), la taille des credentials JWT n’est pas définie. Auth0 peut émettre des JWT de taille variable, et les clients ne doivent pas supposer une taille précise.
* **Taille du Authorization Code** : Les clients doivent s’attendre à ce que les authorization codes, conformément à la spécification OAuth, varient en taille.
* **Paramètres de réponse non reconnus** : Les clients doivent ignorer les paramètres de réponse non reconnus, ce qui permet à Auth0 d’ajouter de nouvelles fonctionnalités sans incidence sur le fonctionnement actuel.
* **Nouvelles ressources, nouveaux champs, headers ou scopes** : L’ajout de nouvelles ressources API, de nouveaux champs, headers ou scopes n’aura pas d’incidence sur les clients existants qui ne reconnaissent pas ou n’utilisent pas ces éléments.

Pour une explication plus détaillée des changements rétrocompatibles, consultez les directives d’Auth0 sur les changements d’API dans les [étapes du cycle de vie du produit](/fr-CA/docs/troubleshoot/product-lifecycle/product-release-stages).

<div id="backward-incompatible-breaking-changes">
  ### Changements non rétrocompatibles entraînant une rupture de compatibilité
</div>

Un changement non rétrocompatible peut entraîner des défaillances s’il modifie l’interopérabilité entre la plateforme Auth0 et les applications clientes. Lorsqu’un tel changement est nécessaire, il suit le [processus de dépréciation et de migration](/fr-CA/docs/troubleshoot/product-lifecycle/deprecations-and-migrations) afin d’aviser les clients et de les accompagner pour qu’ils puissent adapter les implémentations de leur tenant.

Voici quelques exemples de changements entraînant une rupture de compatibilité :

* **Suppression d’une ressource d’API** : Si une ressource d’API est supprimée ou renommée, les clients qui en dépendent subiront une rupture de compatibilité.
* **Modifications de la structure d’URI** : Modifier la structure d’un URI existant peut perturber les clients qui en dépendent.
* **Suppression d’une méthode, d’un paramètre ou d’un champ** : Si une méthode, un paramètre ou un champ est supprimé ou renommé, cela entraînera une rupture de compatibilité pour les clients qui utilisent ces éléments.
* **Modifications des valeurs par défaut** : Les changements apportés à la valeur par défaut d’un champ peuvent avoir une incidence sur les intégrations existantes et constituer une rupture de compatibilité.
* **Modifications des réponses d’erreur et des codes d’état** : Les modifications apportées aux formats de réponse d’erreur, aux codes d’erreur ou aux codes d’état peuvent perturber le comportement existant des clients.
* **Format JWT** : Les modifications au format JWT des tokens constituent des ruptures de compatibilité
* **Format JSON** : Modifier le type d’une valeur JSON constitue une rupture de compatibilité.

Toute rupture de compatibilité déclenche le processus de dépréciation, dans le cadre duquel les clients sont avisés et disposent d’au moins six mois pour migrer vers le nouveau comportement. Pour en savoir plus, consultez le [processus de dépréciation et de fin de vie](/fr-CA/docs/troubleshoot/product-lifecycle/deprecations-and-migrations).

<div id="auth0s-commitment">
  ## L’engagement d’Auth0
</div>

Auth0 réduit au minimum les perturbations causées par des changements non rétrocompatibles en appliquant le processus suivant :

* **Annonce de dépréciation** : Auth0 annonce la dépréciation afin d’aviser les clients d’un changement à venir.
* **Fenêtre de migration** : Les clients disposent d’une période minimale de six mois pour migrer vers la fonctionnalité mise à jour, et des directives sont fournies dans notre [processus de migration](/fr-CA/docs/troubleshoot/product-lifecycle/migration-process).
* **Fin de vie** : Lorsque la fenêtre de migration se termine, la fonctionnalité dépréciée passe à l’étape de fin de vie et n’est plus disponible.

<div id="undocumented-apis">
  ## API non documentées
</div>

Les API Auth0 non documentées sont considérées comme privées et peuvent être modifiées sans préavis. Elles ne sont pas couvertes par le processus de dépréciation, et les clients devraient éviter de s’y fier pour leurs systèmes de production.
