Auth0 abandonne progressivement l’utilisation du point de terminaison `/passwordless/start` par les applications confidentielles lorsqu’Auth0 ne peut pas authentifier que l’appel est effectué au nom de l’application.

<Tooltip tip="OAuth 2.0 : Cadre d’autorisation qui définit les protocoles d’autorisation et les flux de travail." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=OAuth">OAuth</Tooltip> utilise le terme &#39;confidential&#39; pour désigner les applications qui peuvent stocker des secrets. Dans Auth0, il s’agit des &#39;Regular Web Applications&#39;, qui servent des pages Web à partir d’une application backend. Les Single Page Applications et les Native Applications sont considérées comme des applications &#39;public&#39; et ne sont pas touchées par ce changement.

Auth0 peut authentifier les appels à `/passwordless/start` lorsqu’ils incluent un `client_secret` comme paramètre, ou lorsqu’ils sont effectués à partir de la page de connexion personnalisée dans Classic Login et transmettent le paramètre `state`.

<div id="features-affected">
  ## Fonctionnalités concernées
</div>

Si l’une de vos applications appelle actuellement directement le point de terminaison `/passwordless/start` pour démarrer l’<Tooltip tip="Passwordless : forme d’authentification qui ne repose pas sur un mot de passe comme premier facteur." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=passwordless">authentification passwordless</Tooltip> à partir d’une application Web, et que vous n’envoyez pas le paramètre `client_secret`, cette dépréciation vous concerne.

Si vous mettez en œuvre l’authentification passwordless au moyen de la Classic Login page et que vous avez modifié la méthode d’initialisation par défaut des bibliothèques Auth0, cela pourrait aussi vous concerner.

Vous pouvez vérifier si vous êtes concerné en consultant les [journaux du tenant](https://manage.auth0.com/#/logs), en filtrant par &quot;Deprecation Notice&quot; et en recherchant des journaux indiquant &quot;Enforce client authentication for passwordless connections&quot;. Vous pouvez aussi effectuer cette recherche directement avec la requête suivante : `type:depnote AND description:*passwordless*`. Notez que cette requête précise ne fonctionnera que pour les tenants du cloud public, puisque les journaux du cloud privé ne peuvent pas être recherchés dans le champ description.

<div id="actions">
  ## Actions
</div>

Si vous appelez le point de terminaison `/passwordless/start` sans authentification d’application appropriée, vous devez :

* Suivre les instructions ci-dessous pour ajuster le code afin d’appeler correctement `/passwordless/start`.
* Vérifier vos [journaux du tenant](https://manage.auth0.com/#/logs) pour confirmer que la modification a bien été apportée et qu’aucun journal de dépréciation n’est généré pour « Exiger l’authentification du client pour les connexions Passwordless ».
* Dans la section **Migrations** des **Paramètres avancés du tenant**, activez la bascule **Exiger l’authentification du client pour les connexions Passwordless**.

Quelques cas d’utilisation pourraient être touchés, mais pour chacun, le parcours de migration est assez simple :

<div id="api-calls-from-backend">
  ### Appels API à partir du backend
</div>

Pour tout appel effectué depuis votre backend vers le point de terminaison `/passwordless/start`, votre appel doit inclure le <Tooltip tip="Secret du client : secret utilisé par un client (application) pour s’authentifier auprès du serveur d’autorisation; il ne devrait être connu que du client et du serveur d’autorisation et doit être suffisamment aléatoire pour ne pas pouvoir être deviné." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=client+secret">secret du client</Tooltip> comme paramètre.

Si vous envoyez une requête POST directement à `/passwordless/start`, incluez `client_secret` dans la charge utile :

Si vous utilisez un SDK, ajoutez le paramètre à la méthode qui lance le processus sans mot de passe. Cela varie d’un SDK à l’autre, et tous les SDK n’ont pas encore été mis à jour. Si vous utilisez un SDK qui n’a pas encore été mis à jour, vous pouvez faire l’appel HTTP directement en attendant que ce soit fait.

<div id="auth0js-or-lockjs-in-the-classic-login-page">
  ### Auth0.js ou Lock.js dans la page Classic Login
</div>

Si la page Classic Login est utilisée pour l’authentification Passwordless d’une application Web, elle fera des appels au point de terminaison `/passwordless/start`, au moyen de Lock.js ou d’Auth0.js.

Comme vous ne pouvez pas stocker de client secret dans une page Web, pour authentifier l’appel, il faut transmettre au point de terminaison `/passwordless/start` le paramètre `state` reçu dans la page Classic Login. Ce paramètre est stocké dans le champ `config.internalOptions` de la page de connexion personnalisée.

Les modèles par défaut pour personnaliser la page de connexion l’utilisent de la façon suivante lors de l’initialisation de Lock.js ou d’auth0.js :

```javascript lines
var lock = new Auth0Lock(
  config.clientID, 
  config.auth0Domain, 
  {
    auth: {
      // .. autres champs définis
      params: {
       scope: config.internalOptions.scope,
       _csrf: config.internalOptions._csrf,
       state: config.internalOptions.state,
      }
    }
  });
```

```javascript lines
var params = Object.assign({
    scope: config.internalOptions.scope,
    _csrf: config.internalOptions._csrf,
    state: config.internalOptions.state,
  }, {
    // ...paramètres d'authentification
  });

var webAuth = new auth0.WebAuth(params);
```

Veuillez vérifier dans l’implémentation de votre page personnalisée que vous n’avez pas supprimé ce code.

<div id="call-passwordlessstart-from-the-client-in-a-web-application">
  ### Appeler /passwordless/start depuis le client dans une application Web
</div>

Si vous appelez le point de terminaison `/passwordless/start` depuis une page qui utilise JavaScript (par exemple, en utilisant auth0.js sur la page) dans des applications de type Regular Web Apps, vous ne pourrez pas préciser de client secret dans un appel effectué en JavaScript. Si c’est le cas pour votre application, vous devrez modifier vos applications pour que `/passwordless/start` soit appelé depuis le backend de votre application Web plutôt que depuis le frontend.

<div id="rate-limits">
  ### Limites de débit
</div>

L’ajout de l’authentification du client à `/passwordless/start` a pour effet qu’Auth0 peut se fier aux en-têtes envoyés avec la requête. Si vous définissez l’en-tête `auth0-forwarded-for`, l’adresse IP apparaîtra dans les journaux et sera utilisée à des fins de <Tooltip tip="Protection contre les attaques : fonctionnalités qu’Auth0 fournit pour détecter et atténuer les attaques, y compris la protection contre les attaques par force brute, la limitation des adresses IP suspectes, la détection des mots de passe compromis, la détection des robots et l’authentification multifacteur adaptative." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=attack+protection">protection contre les attaques</Tooltip>.

Les requêtes authentifiées vers `/passwordless/start` seront traitées comme des requêtes authentifiées normales à l’API, et la limite de débit globale correspondante de l’API d’authentification s’appliquera.
