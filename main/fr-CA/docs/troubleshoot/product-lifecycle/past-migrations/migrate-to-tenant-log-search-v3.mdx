---
description: Décrit comment migrer d’Auth0 Logs Search v2 vers Tenant Log Search v3.
title: Migrer vers Tenant Log Search v3
---

Afin d’offrir à nos clients la solution la plus fiable et la plus évolutive, Auth0 a déprécié Tenant Logs Search Engine v2 au profit de v3.

Auth0 migre de manière proactive les clients qui ne sont pas touchés par ce changement, tandis que ceux qui pourraient l’être sont avisés qu’ils doivent activer explicitement v3 pendant la période de transition prévue.

<div id="features-affected">
  ## Fonctionnalités touchées
</div>

Vous êtes concerné si vous répondez à tous les critères suivants :

* Tenants créés le 21 mai 2019 ou avant
* Tenants hébergés dans le Public Cloud d&#39;Auth0, dans les régions AU ou EU
* Vous utilisez le endpoint GET /api/v2/logs ou le endpoint GET `/api/v2/users/{user_id}/logs` avec le paramètre `include_totals=true` ou le paramètre `q`.
* Vous parcourez plus de 1000 résultats.
* Vous utilisez la Delegated Admin Extension. Les anciennes versions de l&#39;extension continueront de fonctionner après la migration de votre tenant vers Search Engine v3 pour les logs, mais il se peut que les totaux de pagination affichés dans les logs soient inexacts. La mise à jour vers la version 3.7 de l&#39;extension corrige ce problème.

Les tenants suivants ne sont pas concernés :

* Les tenants Cloud dans la région US. La région US a été entièrement migrée et utilise déjà Search Engine v3.
* Les tenants Private Cloud (la migration pour les clients de Private Cloud débutera à une date ultérieure.)
* Les tenants Cloud dans les régions EU et AU qui :

  * n&#39;utilisent pas du tout les endpoints `GET /api/v2/logs` ou `GET /api/v2/users/{user_id}/logs` de la Management API.
  * consultent les logs uniquement dans la section Dashboard Logs.
  * utilisent le `GET /api/v2/logs endpoint` avec la méthode by checkpoint (en utilisant le paramètre `from`).
  * consultent les logs à l&#39;aide de l&#39;une des extensions Auth0 Logs to External Service Dashboard (qui utilisent la méthode by checkpoint).

<div id="verify-query-migration">
  ## Migration des requêtes Verify
</div>

Auth0 ne génère qu’un seul log du même type et avec la même description toutes les 60 minutes. Peu importe le nombre d’appels que vous faites aux endpoints concernés à l’aide de fonctionnalités obsolètes, vous ne verrez quand même qu’un seul log par fonctionnalité obsolète chaque heure.

Si vous modifiez vos requêtes, vous devrez attendre 60 minutes avant de pouvoir conclure avec certitude que l’absence de nouveaux logs `depnote` signifie que le comportement obsolète a été retiré de votre code.

Vous pouvez effectuer la recherche suivante dans les logs de votre tenant pour repérer les requêtes qui généreraient des erreurs après la migration vers la v3 :

`type:depnote AND description:*logs*`

Ces entrées de log comprennent un champ `description` qui précise le comportement obsolète que vous utilisez.

Vous pouvez aussi vérifier les champs `details.request.path` et `client_name` pour voir quelle application appelle soit `GET /api/v2/logs`, soit `GET /api/v2/users/{user_id}/logs`.

<div id="changes">
  ## Changements
</div>

Les changements non rétrocompatibles sont mineurs, mais vous devriez vérifier vos requêtes pour vous assurer que les résultats obtenus sont bien ceux attendus.

Les changements non rétrocompatibles sont liés à :

<div id="pagination">
  ### Pagination
</div>

* Lorsque votre tenant sera migré vers logs v3, la valeur du champ `total` renvoyée dans le résultat sommaire lors d’un appel à `GET /api/v2/logs` ou à `GET /api/v2/users/{user_id}/logs` changera. Lorsque vous recherchez des logs à l’aide du moteur de recherche v2, le champ totals dans vos résultats indique le nombre de logs correspondant à la requête fournie. En revanche, dans v3, le champ totals indique combien de logs sont renvoyés dans la page (un peu comme le fait le champ length). Afin d’éviter toute perturbation, si votre application s’appuie sur le champ total pour la pagination, vous devriez mettre à jour votre logique pour tenir compte de ce changement.
* Il existe déjà une limite de 100 logs par requête. Lorsque votre tenant sera migré vers logs v3, vous ne pourrez paginer que dans un maximum de 1 000 résultats de recherche; par conséquent, les appels portant sur plus de 1 000 résultats renverront une erreur. Afin d’éviter toute perturbation, vous devriez revoir vos requêtes pour éviter cette limite ou gérer les erreurs en conséquence.

<div id="q-parameter-validation">
  ### validation du paramètre q
</div>

* La syntaxe des requêtes utilisant le paramètre `q` dans `GET /api/v2/logs` comporte des changements mineurs dont il faut tenir compte. Lorsque votre tenant sera migré vers logs v3, cette validation sera appliquée, et cette requête renverra alors une erreur. Pour éviter toute interruption, vous devriez passer en revue vos requêtes pour vous assurer qu’elles respectent la syntaxe de requête prise en charge.
* Le paramètre `q` contient un champ invalide. Lorsque votre tenant sera migré vers logs v3, cette validation sera appliquée, et cet appel renverra alors une erreur. Pour éviter toute interruption, vous devriez passer en revue vos requêtes pour vous assurer qu’elles n’incluent que des champs interrogeables.

<div id="enable-tenant-log-search-v3">
  ## Activer Tenant Log Search v3
</div>

Après avoir examiné vos requêtes, vous pouvez activer Tenant Logs Search Engine v3 à partir du Dashboard.

1. Accédez à [Tenant Settings &gt; Advanced](https://manage.auth0.com/#/tenant/advanced).
2. Faites défiler la page jusqu&#39;à **Migrations**.
3. Désactivez le commutateur **Legacy Logs Search V2**. Lorsque ce commutateur est désactivé, le moteur de recherche de logs v2 obsolète est désactivé et l&#39;utilisation de Search Engine v3 est forcée. Si vous ne voyez pas la bascule **Legacy Logs Search V2**, c&#39;est que vous avez déjà été migré vers v3. Aucune autre action n&#39;est requise.

Si vous avez besoin d&#39;aide pour la migration, communiquez avec nous par l&#39;entremise du [Support Center](https://support.auth0.com/).
