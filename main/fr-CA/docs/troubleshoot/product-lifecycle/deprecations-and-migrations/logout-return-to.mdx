title: Guide de migration des redirections de déconnexion
---

Le **1 décembre 2021**, le comportement de déconnexion changera pour toujours rediriger les utilisateurs vers l’URI transmis aux API de déconnexion d’Auth0, au lieu d’utiliser le paramètre de requête `returnTo` transmis par les <Tooltip tip="Fournisseur d’identité (IdP) : service qui stocke et gère les identités numériques." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=Identity+Providers">fournisseurs d’identité</Tooltip> à `/login/callback` pendant l’exécution de la déconnexion. L’URI utilisé dépendra de l’API appelée pour demander la déconnexion de la session :

* `https://{yourDomain}/logout` : Auth0 utilisera le paramètre `returnTo`
* `https://{yourDomain}/v2/logout?federated` : Auth0 utilisera le paramètre `returnTo`
* `https://{yourDomain}/wsfed/{yourClientId}?wa=wsignout1.0` : le paramètre `wreply`

Si Auth0 n’a aucune trace d’un appel précédent à l’une de ces API, la déconnexion se terminera, mais la redirection n’aura pas lieu et une page d’erreur s’affichera pour les utilisateurs finaux.

<div id="affected-endpoints">
  ### Points de terminaison touchés
</div>

Les points de terminaison suivants sont touchés par ce changement :

* `https://{yourDomain}/v2/logout` lors de l’utilisation du paramètre de requête `federated` pour [déconnecter les utilisateurs des fournisseurs d’identité](/fr-CA/docs/authenticate/login/logout/log-users-out-of-idps)
* `https://{yourDomain}/wsfed/{yourClientId}?wa=wsignout1.0` lors de l’utilisation du paramètre de requête `wreply` pour [se déconnecter d’Auth0 à l’aide du protocole WS-Fed](/fr-CA/docs/authenticate/protocols/ws-fed-protocol)
* `https://{yourDomain}/login/callback` lors de l’utilisation du paramètre de requête `returnTo`.

Les tenants qui s’appuient sur les scénarios d’utilisation suivants ne pourront pas rediriger vers l’URI de redirection souhaité après le **01 décembre 2021**.

1. Les applications qui appellent `/login/callback?returnTo` sans appel préalable à `/v2/logout or /wsfed/{yourClient}?wa=wsignout1.0` pour préciser l’URL de déconnexion.
2. Les fournisseurs d’identité qui appellent `/login/callback?returnTo` avec une valeur dans `returnTo` différente de celle fournie initialement dans un appel précédent à `/v2/logout` ou `/wsfed/{yourClientId}`.
3. Les applications qui effectuent des appels simultanés à l’une des API touchées en fournissant des URI de redirection différentes d’un appel à l’autre.
4. Les applications ou les utilisateurs qui effectuent des appels à l’API `/authorize` simultanément ou en alternance avec des appels à l’une des API touchées, qu’il y ait eu ou non un appel précédent à `/v2/logout` ou `/wsfed/{yourClientId}?wa=wsignout1.0`.

<div id="actions">
  ### Actions
</div>

1. Accédez à [Auth0 Dashboard &gt; Monitoring &gt; Logs](https://manage.auth0.com/#/logs), puis recherchez `type:depnote AND description:*unvalidated*redirects*` pour trouver les applications qui dépendent du comportement déprécié.
2. Pour chaque application touchée, assurez-vous que :

   1. l’application ou la Relying Party lance la déconnexion au moyen de l’une des API publiques d’Auth0 : [Authentication API: Logout](https://auth0.com/docs/api/authentication#logout) ou [Authentication API: WS-Federation](https://auth0.com/docs/api/authentication#ws-federation).
   2. l’application ne dépend pas des fournisseurs d’identité pour modifier la Logout URL renvoyée à `/login/callback?returnTo`, car ces modifications ne seront plus prises en compte.
   3. l’application n’effectue pas d’appels simultanés aux API de déconnexion avec des Logout URL différentes. Auth0 ne stockera qu’une seule Logout URL par agent utilisateur à la fois, ce qui signifie que les processus de déconnexion simultanés ne pourront pas rediriger.
   4. l’application n’effectue pas d’appels à `/authorize` simultanément ou de façon entrelacée avec des appels aux API de déconnexion. Lorsqu’une transaction de connexion se termine, la Logout URL stockée pour un agent utilisateur est effacée, ce qui signifie que les processus de déconnexion simultanés ne pourront pas rediriger.
   5. l’application et les utilisateurs n’entrelacent pas les appels à `/authorize` avec les appels à l’une des API touchées.

Une fois les actions de migration terminées et testées, vous devriez vérifier la migration.

<div id="verify-migration">
  ### Vérifier la migration
</div>

Une fois vos applications migrées et une fois que vous êtes certain qu’elles ne reposent plus sur des modes d’utilisation obsolètes, vérifiez vos modifications en désactivant le comportement obsolète au moment de votre choix, mais avant le **01 décembre 2021**.

1. Accédez à [Auth0 Dashboard &gt; Tenant Settings &gt; Advanced](https://manage.auth0.com/#/tenant/advanced), puis repérez la section **Migrations**.
2. Désactivez le commutateur **Unvalidated redirects from /login/callback**. La désactivation de ce commutateur désactive le comportement obsolète pour votre tenant, ce qui empêche son utilisation.

Si la redirection vers le **Logout URL** ne fonctionne pas comme prévu après la désactivation de ce commutateur, c’est le signe que votre application repose encore sur le comportement obsolète.

Une fois les migrations effectuées avec succès et confirmées dans les environnements de production, vous pouvez désactiver le commutateur de façon permanente afin de vous assurer que les fonctionnalités obsolètes ne puissent plus être utilisées. Le **01 décembre 2021**, Auth0 supprimera complètement le comportement obsolète ainsi que le commutateur associé.
