title: "Obtenir un jeton"
description: "L’échange de jeton On-Behalf-Of (OBO) permet à un service intermédiaire d’échanger un jeton utilisateur entrant contre un nouveau jeton dont la portée est limitée au service en aval."
---

`POST /oauth/token`

L’échange de jeton On-Behalf-Of (OBO) ([RFC 8693](https://datatracker.ietf.org/doc/html/rfc8693)) permet aux services intermédiaires d’échanger un jeton associé à l’utilisateur contre un nouveau jeton pour appeler des services en aval.

Le nouveau jeton conserve l’identité et les permissions de l’utilisateur d’origine, tout en étant limité précisément au service en aval, ce qui permet à ce service de prendre des décisions d’autorisation en fonction de l’utilisateur final. La chaîne de délégation est consignée dans la claim `act` (actor), chaque niveau représentant un service dans la chaîne d’appel. Pour en savoir plus, consultez la [documentation sur l’échange de jeton On-Behalf-Of](https://auth0.com/docs/secure/call-apis-on-users-behalf/on-behalf-of-token-exchange).

<div id="remarks">
  ## Remarques
</div>

* Seuls les clients d’API personnalisée associés à un serveur de ressources peuvent utiliser l’échange de jetons OBO. Un client d’API personnalisée doit répondre aux exigences suivantes :
  * Définir `app_type` sur `resource_server`.
  * Définir `resource_server_identifier` sur un serveur de ressources valide, c.-à-d. `https://my-api.example.com`. Auth0 utilise l’identifiant du serveur de ressources comme paramètre d’audience dans les appels d’autorisation.

* Les scopes accordés à l’application peuvent différer des scopes demandés. Dans ce cas, un paramètre `scope` sera inclus dans la réponse JSON. Les scopes sont fondés sur les [politiques de contrôle d’accès basé sur les rôles (RBAC)](/fr-CA/docs/manage-users/access-control/rbac) de l’utilisateur.

* Les échanges de jetons OBO déclenchent le [`post-login` Action trigger](/fr-CA/docs/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger), où `event.transaction.protocol` est défini sur `oauth2-token-exchange` et `event.transaction.actor` assure le suivi de l’ensemble de la chaîne de délégation.

* La chaîne de délégation est limitée à cinq niveaux imbriqués. L’échange de jetons OBO échoue si le jeton de sujet comporte déjà cinq niveaux `act` imbriqués.

* Mettez en cache les jetons d’accès pendant toute la durée de vie du jeton au lieu de demander un nouveau jeton pour chaque appel d’API. Les jetons d’accès peuvent être réutilisés jusqu’à leur expiration; les échanges de jetons répétés gaspillent des ressources, augmentent la latence et peuvent déclencher des limites de débit.

<div id="parameters">
  ## Paramètres
</div>

<ParamField header="DPoP" type="string">
  Une preuve DPoP pour la requête. Ce paramètre est facultatif et n’est requis que si votre application utilise [Demonstrating Proof-of-Possession](/fr-CA/docs/secure/sender-constraining/demonstrating-proof-of-possession-dpop).
</ParamField>

<ParamField header="auth0-forwarded-for" type="string">
  L’adresse IP de l’utilisateur final, sous forme de chaîne de caractères. Définissez ce paramètre si vous voulez que la protection [Suspicious IP Throttling](/fr-CA/docs/secure/attack-protection/suspicious-ip-throttling) fonctionne dans les scénarios côté serveur.
</ParamField>

<div id="request-body">
  ## Corps de la requête
</div>

<ParamField body="grant_type" type="string" required>
  Indique le flux utilisé. Pour l’échange de jeton On-Behalf-Of, utilisez `urn:ietf:params:oauth:grant-type:token-exchange`.
</ParamField>

<ParamField body="subject_token_type" type="string" required>
  Le type du jeton sujet. Pour l’échange de jeton On-Behalf-Of, utilisez `urn:ietf:params:oauth:token-type:access_token`.
</ParamField>

<ParamField body="subject_token" type="string" required>
  Le jeton d’accès Auth0 entrant provenant de l’utilisateur ou du service en amont que le service intermédiaire détient actuellement.
</ParamField>

<ParamField body="requested_token_type" type="string" required>
  Indique le type de jeton que vous souhaitez obtenir en retour. Pour l’échange de jeton On-Behalf-Of, utilisez `urn:ietf:params:oauth:token-type:access_token`.
</ParamField>

<ParamField body="client_id" type="string" required>
  Le Client ID de votre client d’API personnalisée. Le client d’API personnalisée doit être associé à un serveur de ressources (même identifiant). Comme pour les autres types d’octroi, vous pouvez aussi transmettre le Client ID dans l’en-tête Authorization au moyen de l’authentification Basic HTTP.
</ParamField>

<ParamField body="client_secret" type="string" required>
  Le Client Secret de votre client d’API personnalisée. Comme pour les autres types d’octroi, vous pouvez aussi transmettre le Client Secret dans l’en-tête Authorization au moyen de l’authentification Basic `HTTP`. Consultez les options offertes dans la [documentation de référence de l’Auth0 Authentication API](https://auth0.com/docs/api/authentication#authentication-methods). Notez que vous ne pouvez pas définir `token_endpoint_auth_method` sur `none` pour l’échange de jetons OBO.
</ParamField>

<ParamField body="audience" type="string" required>
  L’identifiant unique de l’API en aval à laquelle vous voulez accéder. Il s’agit de l’identifiant du service en aval qui reçoit et valide le nouveau jeton.
</ParamField>

<ParamField body="scope" type="string">
  (Facultatif) Liste de permissions précises, séparées par des espaces, demandées pour l’appel en aval. Si elle n’est pas précisée, tous les scopes accordés à l’utilisateur pour l’audience cible seront inclus selon les politiques RBAC.
</ParamField>

<div id="response">
  ## Réponse
</div>

| Statut | Description                                                                                                                                                                                     |
| ------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 200    | Réponse réussie. Renvoie un jeton d’accès pour l’API en aval tout en conservant la même identité utilisateur.                                                                                   |
| 400    | Requête incorrecte. Peut se produire si la profondeur de la chaîne de délégation dépasse la limite maximale autorisée de 4 niveaux imbriqués, ou si des paramètres obligatoires sont manquants. |
| 401    | Non autorisé. Le jeton sujet est invalide ou expiré, ou les identifiants du client sont incorrects.                                                                                          |
| 403    | Interdit. Le client n’est pas autorisé à effectuer l’échange de jeton, ou l’utilisateur ne dispose pas des scopes demandés pour l’audience cible.                                               |

<ResponseExample>
  ```json 200 Réponse
  {
    "access_token": "eyJ...",
    "expires_in": 86400,
    "token_type": "Bearer",
    "issued_token_type": "urn:ietf:params:oauth:token-type:access_token"
  }
  ```

  ```json 400 Réponse (Limite de la chaîne de délégation dépassée)
  {
    "error": "invalid_request",
    "error_description": "Delegation chain (`act` claim) depth exceeds the maximum allowed limit of 5"
  }
  ```
</ResponseExample>

<div id="response-fields">
  ## Champs de réponse
</div>

<ResponseField name="access_token" type="string">
  Le nouvel Auth0 jeton d’accès limité à l’API en aval. Ce [JSON Web Token (JWT)](/fr-CA/docs/secure/tokens/json-web-tokens#json-web-tokens) contient le même `sub` (identité de l’utilisateur) que le jeton sujet, avec `aud` défini sur l’audience de l’API en aval demandée. Le claim `act` assure le suivi de la chaîne de délégation.
</ResponseField>

<ResponseField name="issued_token_type" type="string">
  Confirme le format du jeton renvoyé. Il correspond au `requested_token_type` de votre requête.

  Valeur : `urn:ietf:params:oauth:token-type:access_token`
</ResponseField>

<ResponseField name="token_type" type="string">
  Précise le schéma d’authentification à utiliser dans l’en-tête Authorization. Pour OBO, il s’agit de `Bearer`, sauf si vous utilisez DPoP, auquel cas `DPoP` sera utilisé.
</ResponseField>

<ResponseField name="expires_in" type="number">
  La durée de vie du jeton, en secondes.
</ResponseField>

<ResponseField name="scope" type="string">
  (Facultatif) Inclus uniquement si les scopes accordés diffèrent des scopes demandés. Liste des scopes réellement accordés, délimitée par des espaces.
</ResponseField>
