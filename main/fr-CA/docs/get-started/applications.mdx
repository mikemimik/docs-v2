Dans Auth0, le terme **application** ou **app** ne renvoie à aucune caractéristique d’implémentation particulière. Par exemple, il peut s’agir d’une application native qui s’exécute sur un appareil mobile, d’une application web monopage qui s’exécute dans un navigateur, ou d’une application web traditionnelle qui s’exécute sur un serveur.

Auth0 classe les apps selon ces caractéristiques :

* **Type d’application** : Pour ajouter l’authentification à votre application, vous devez l’enregistrer dans le <Tooltip tip="Auth0 Dashboard : le principal produit d’Auth0 pour configurer vos services." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip> et sélectionner l’un des types d’application suivants :

  * **Application web traditionnelle** : Applications web traditionnelles qui exécutent la majeure partie de leur logique sur le serveur (comme Express.js ou ASP.NET). Pour savoir comment configurer une application web traditionnelle, consultez [Register Regular Web Applications](/fr-CA/docs/get-started/auth0-overview/create-applications/regular-web-apps).
  * **Application web monopage (SPA)** : Applications JavaScript qui exécutent la majeure partie de leur logique d’interface utilisateur dans un navigateur web et communiquent avec un serveur web principalement au moyen d’API (comme AngularJS + Node.js ou React). Pour savoir comment configurer une application web monopage, consultez [Register Single-Page Web Applications](/fr-CA/docs/get-started/auth0-overview/create-applications/single-page-web-apps).
  * **Application native** : Applications mobiles ou de bureau qui s’exécutent nativement sur un appareil (comme iOS ou Android). Pour savoir comment configurer une application native, consultez [Register Native Applications](/fr-CA/docs/get-started/auth0-overview/create-applications/native-apps).
  * **Application machine à machine (M2M)** : Applications non interactives, comme des outils en ligne de commande, des démons, des appareils IoT ou des services qui s’exécutent sur votre back-end. En général, vous utiliserez cette option si vous avez un service qui doit accéder à une API. Pour savoir comment configurer une application machine à machine, consultez [Register Machine-to-Machine Applications](/fr-CA/docs/get-started/auth0-overview/create-applications/machine-to-machine-apps).
* **Sécurité des identifiants** : Selon la [spécification OAuth 2.0](https://tools.ietf.org/html/rfc6749#section-2.1), les apps peuvent être classées comme publiques ou confidentielles; les apps confidentielles peuvent conserver leurs identifiants en toute sécurité, tandis que les apps publiques ne le peuvent pas. Pour en savoir plus, consultez [Confidential and Public Applications](/fr-CA/docs/get-started/applications/confidential-and-public-applications).
* **Propriété** : Le fait qu’une app soit classée comme application de première partie ou application de tierce partie dépend de sa propriété et de son contrôle. Les applications de première partie sont contrôlées par la même organisation ou la même personne qui possède le domaine Auth0. Les applications de tierce partie permettent à des parties externes ou à des partenaires d’accéder de façon sécurisée aux ressources protégées derrière votre API. Pour en savoir plus, consultez [First-Party and Third-Party Applications](/fr-CA/docs/get-started/applications/first-party-and-third-party-applications).

<div id="manage-applications-settings">
  ## Gérer les paramètres des applications
</div>

Vous enregistrez les applications dans [Dashboard &gt; Applications &gt; Applications](https://manage.auth0.com/#/applications/\{yourClientId}/settings). En plus de configurer les applications dans le Dashboard, vous pouvez aussi configurer les applications par programmation, comme décrit dans la spécification [OpenID Connect (OIDC) Dynamic Client Registration 1.0](https://openid.net/specs/openid-connect-registration-1_0.html).

Vous pouvez mettre en place une configuration plus complexe qui permet aux utilisateurs d’ouvrir une session différemment selon l’application. Pour en savoir plus, consultez [Multi-Tenant Application Best Practices](/fr-CA/docs/get-started/auth0-overview/create-tenants/multi-tenant-apps-best-practices) et [Create Multiple Tenants](/fr-CA/docs/get-started/auth0-overview/create-tenants/create-multiple-tenants).

Par défaut, Auth0 active toutes les connexions associées à votre tenant lorsque vous créez une nouvelle application. Pour modifier ce comportement, [mettez à jour les connexions de l’application](/fr-CA/docs/get-started/applications/update-application-connections) dans les **Application Settings** du Dashboard.

<div id="monitor-applications">
  ## Surveiller les applications
</div>

Vous pouvez [surveiller les applications](/fr-CA/docs/deploy-monitor/monitor/monitor-applications) et effectuer des tests de bout en bout en utilisant vos propres tests. Auth0 stocke des [données de journalisation](/fr-CA/docs/deploy-monitor/logs), y compris les actions des administrateurs du Dashboard, les authentifications réussies et échouées des utilisateurs, ainsi que les demandes de changement de mot de passe. Vous pouvez utiliser la transmission en continu des logs dans [Auth0 Marketplace](https://marketplace.auth0.com/features/log-streaming) pour exporter vos données de journalisation et vous servir d’outils comme Sumo Logic, Splunk ou Mixpanel pour les analyser et les stocker.

<div id="remove-applications">
  ## Supprimer des applications
</div>

Vous pouvez [supprimer une application](/fr-CA/docs/get-started/applications/remove-applications) à l’aide du Dashboard ou de la <Tooltip tip="Management API : Produit permettant aux clients d’effectuer des tâches administratives." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=Management+API">Management API</Tooltip>.

<div id="manage-client-secrets">
  ## Gérer les secrets client
</div>

Un <Tooltip tip="Serveur d’autorisation : serveur centralisé qui contribue à définir les limites de l’accès d’un utilisateur. Par exemple, votre serveur d’autorisation peut contrôler les données, les tâches et les fonctionnalités auxquelles un utilisateur a accès." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=client+secret">secret client</Tooltip> est un secret connu uniquement de votre application et du <Tooltip tip="Secret client : secret utilisé par un client (application) pour s’authentifier auprès du serveur d’autorisation; il ne doit être connu que du client et du serveur d’autorisation et doit être suffisamment aléatoire pour ne pas pouvoir être deviné." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=authorization+server">serveur d’autorisation</Tooltip>. Il protège vos ressources en n’accordant des [jetons](/fr-CA/docs/secure/tokens) qu’aux entités autorisées.

Protégez vos secrets client et ne les incluez **jamais** dans des applications mobiles ou des applications qui s’exécutent dans le navigateur. Si votre secret client est compromis, vous devriez [le remplacer par un nouveau](/fr-CA/docs/get-started/applications/rotate-client-secret) et mettre à jour toutes les applications autorisées avec le nouveau secret client.

<div id="grant-types">
  ## Types d’autorisation
</div>

Auth0 offre de nombreux types d’autorisation ou flux d’authentification et d’autorisation, et vous permet d’indiquer quels types d’autorisation sont appropriés selon la propriété `grant_types` de votre application enregistrée dans Auth0. Pour en savoir plus, consultez [type d’autorisation de l’application](/fr-CA/docs/get-started/applications/application-grant-types).
