title: Applications confidentielles et publiques
---

Selon la [spécification OAuth 2.0](https://tools.ietf.org/html/rfc6749#section-2.1), les applications peuvent être classées comme confidentielles ou publiques. La principale différence tient à la capacité de l’application à conserver des informations d’identification (comme un <Tooltip tip="Client ID : valeur d’identification attribuée par Auth0 à votre ressource enregistrée." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=client+ID">ID client</Tooltip> et un secret) en toute sécurité. Cela a une incidence sur le type d’authentification que les applications peuvent utiliser.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Les classifications confidentielle/publique et first-party/third-party sont indépendantes. Le caractère confidentiel ou public décrit la capacité d’authentification de l’application (autrement dit, si elle peut conserver un secret). Le caractère first-party ou third-party décrit la relation de confiance (qui possède et exploite l’application). Pour en savoir plus, consultez [Applications propriétaires et applications tierces](/fr-CA/docs/get-started/applications/first-party-and-third-party-applications).
</Callout>

Lorsque vous créez une application dans l’Auth0 Dashboard, Auth0 vous demande quel type d’Auth0 application vous souhaitez attribuer à la nouvelle application et utilise cette information pour déterminer si l’application est confidentielle ou publique.

Pour en savoir plus, consultez [Vérifier si une application est confidentielle ou publique](/fr-CA/docs/get-started/applications/confidential-and-public-applications/view-application-type).

<div id="confidential-applications">
  ## Applications confidentielles
</div>

Les applications confidentielles peuvent conserver des informations d’authentification de manière sécurisée sans les exposer à des tiers non autorisés. Elles nécessitent un serveur backend fiable pour stocker le ou les secrets.

<div id="grant-types">
  ### Types d’octroi
</div>

Les applications confidentielles utilisent un serveur backend de confiance et peuvent utiliser des types d’octroi qui exigent la spécification de leur Client ID et de leur <Tooltip tip="Client Secret : secret utilisé par un client (application) pour s’authentifier auprès du serveur d’autorisation; il ne doit être connu que du client et du serveur d’autorisation et doit être suffisamment aléatoire pour ne pas pouvoir être deviné." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=Client+Secret">Client Secret</Tooltip> (ou d’autres identifiants enregistrés) pour l’authentification lorsqu’elles appellent l’endpoint [Get Token](https://auth0.com/docs/api/authentication#get-token) de l’Auth0 Authentication API. Les applications confidentielles peuvent utiliser les méthodes d’authentification Client Secret Post, Client Secret Basic ou [Private Key JWT](/fr-CA/docs/get-started/authentication-and-authorization-flow/authenticate-with-private-key-jwt).

Voici les applications considérées comme confidentielles :

* Une application web dotée d’un backend sécurisé qui utilise le [Authorization Code Flow](/fr-CA/docs/get-started/authentication-and-authorization-flow/authorization-code-flow), le [Resource Owner Password Flow](/fr-CA/docs/get-started/authentication-and-authorization-flow/resource-owner-password-flow) ou le <Tooltip tip="Resource Owner: Entity (such as a user or application) capable of granting access to a protected resource." cta="View Glossary" href="/fr-CA/docs/glossary?term=Resource+Owner">Resource Owner</Tooltip> Password Flow avec prise en charge du realm
* Une application machine-to-machine (M2M) qui utilise le [Client Credentials Flow](/fr-CA/docs/get-started/authentication-and-authorization-flow/client-credentials-flow)

<div id="id-tokens">
  ### Jetons d’identité
</div>

Comme les applications confidentielles peuvent conserver des secrets, vous pouvez obtenir pour elles des <Tooltip tip="ID Token : information d’identification destinée au client lui-même, plutôt qu’à l’accès à une ressource." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=ID+tokens">jetons d’identité</Tooltip> signés de l’une des deux façons suivantes :

* De façon symétrique, à l’aide de leur secret client (`HS256`)
* De façon asymétrique, à l’aide d’une clé privée (`RS256`)

<div id="public-applications">
  ## Applications publiques
</div>

Les applications publiques **ne peuvent pas** conserver des informations d’identification en toute sécurité.

<div id="grant-types">
  ### Types d’octroi
</div>

Les applications publiques peuvent uniquement utiliser des types d’octroi qui n’exigent pas l’utilisation de leur secret client. Elles ne peuvent pas envoyer de secret client, car elles ne peuvent pas garantir la confidentialité des identifiants requis.

Voici des applications publiques :

* Une application native de bureau ou mobile qui utilise le [flux du code d’autorisation avec PKCE](/fr-CA/docs/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce)
* Une application Web côté client basée sur JavaScript (comme une application monopage) qui utilise le type d’octroi [flux implicite](/fr-CA/docs/get-started/authentication-and-authorization-flow/implicit-flow-with-form-post)

### jetons d’identité

Comme les applications publiques ne peuvent pas conserver de secrets, les [jetons d’identité](/fr-CA/docs/secure/tokens/id-tokens) qui leur sont émis doivent être :

* Signés de façon asymétrique à l’aide d’une clé privée (`RS256`)
* Vérifiés à l’aide de la clé publique correspondant à la clé privée utilisée pour signer le jeton
