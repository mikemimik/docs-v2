<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  L’onglet identifiants შეიძლება ne pas s’afficher dans votre tenant si votre application est publique. Pour en savoir plus sur les applications confidentielles et les applications publiques, consultez [Applications confidentielles et publiques](/fr-CA/docs/get-started/applications/confidential-and-public-applications).
</Callout>

<div id="authentication-methods">
  ## Méthodes d’authentification
</div>

Auth0 offre les méthodes suivantes pour permettre à votre application de s’authentifier auprès du <Tooltip tip="Serveur d’autorisation : serveur centralisé qui contribue à définir les limites de l’accès d’un utilisateur. Par exemple, votre serveur d’autorisation peut contrôler les données, les tâches et les fonctionnalités accessibles à un utilisateur." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=authorization+server">serveur d’autorisation</Tooltip> :

* **Private Key <Tooltip tip="JSON Web Token (JWT) : format standard d’ID Token (et souvent de jeton d’accès) utilisé pour représenter de façon sécurisée des déclarations entre deux parties." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=JWT">JWT</Tooltip>** : Une méthode d’authentification asymétrique pour les applications confidentielles. Avec Private Key JWT, vous générez une paire de clés, une publique et une privée, qui servent d’identifiants. Vous fournissez la clé publique et conservez la clé privée en lieu sûr dans votre propre système, sans la partager avec Auth0. Vous utilisez la clé privée pour signer la requête envoyée au serveur d’autorisation. Vous pouvez gérer les clés publiques ci-dessous, dans la section **Identifiants disponibles**.
* **<Tooltip tip="Client Secret : secret utilisé par un client (application) pour s’authentifier auprès du serveur d’autorisation; il ne devrait être connu que du client et du serveur d’autorisation et doit être suffisamment aléatoire pour ne pas pouvoir être deviné." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=Client+Secret">Client Secret</Tooltip> :** Une méthode d’authentification symétrique. Avec l’authentification Client Secret, vous fournissez le Client Secret qu’Auth0 a attribué lors de la création de l’application. Vous pouvez afficher et copier le `client_secret` directement dans cette section.

  * **Client Secret (Basic)** : Utilisez Basic pour une application confidentielle qui emploie le schéma d’authentification `HTTP BASIC` pour envoyer un Client Secret.
  * **Client Secret (Post)** : Utilisez Post pour une application confidentielle qui utilise les paramètres du corps de la requête pour envoyer un Client Secret.

Pour en savoir plus sur les méthodes d’authentification et les identifiants associés, consultez [Identifiants d’application](/fr-CA/docs/secure/application-credentials).

<div id="available-credentials">
  ## Identifiants disponibles
</div>

<Frame>![](/docs/images/cdy7uua7fh8z/7vzjXXZ0A9pDj5qsXmOTwh/b4b299c6316216376f7e2858007df0b1/Private_Key_JWT_-_English.png)</Frame>

**Ajouter un nouvel identifiant** : ajoute une nouvelle entrée à la liste si vous n’avez pas atteint le nombre maximal d’identifiants (2) pour l’application.

* **Nom** : nom de l’identifiant. Par exemple : 4096.
* **ID de clé** : identifiant unique de l’identifiant généré par Auth0. Le même identifiant ne peut pas être téléversé plus d’une fois.
* **Algorithme** : algorithme que vous sélectionnez pour chaque identifiant. Nous prenons en charge RS256, PS256 et RS384.
* **Date d’expiration** : `datetime` à partir duquel l’identifiant n’est plus valide pour l’utilisation déclarée. L’identifiant ne sera pas supprimé, mais il ne pourra plus être utilisé. Cela peut être configuré dans l’interface utilisateur en activant la case à cocher **Définir une date d’expiration explicite pour cet identifiant**.

Choisissez le menu latéral pour activer, désactiver ou supprimer des identifiants.
