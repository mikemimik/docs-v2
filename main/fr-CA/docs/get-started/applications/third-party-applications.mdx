Les applications tierces sont des applications détenues et exploitées par une organisation externe — un partenaire, un développeur indépendant ou un agent d’IA — qui authentifient vos utilisateurs et accèdent à vos API.

Contrairement aux [applications de première partie](/docs/fr-CA/get-started/applications/first-party-and-third-party-applications#first-party-applications), que vous contrôlez directement, les applications tierces fonctionnent de manière autonome : vous leur accordez l’accès à vos ressources, mais vous ne pouvez pas contrôler ce qu’elles en font. Pour en savoir plus sur les différences entre les applications de première partie et les applications tierces, consultez [Applications de première partie et applications tierces](/docs/fr-CA/get-started/applications/first-party-and-third-party-applications).

Les applications tierces ont les caractéristiques particulières suivantes :

* **Contrôles de sécurité renforcés** : Auth0 applique des [contrôles de sécurité renforcés](/docs/fr-CA/get-started/applications/third-party-applications/security-controls) aux applications tierces afin de garantir que les applications externes ne peuvent accéder qu’aux ressources que vous autorisez explicitement. Les fonctionnalités destinées aux cas d’utilisation des applications de première partie ne sont pas offertes.
* **Consentement de l’utilisateur** : Auth0 exige toujours le consentement de l’utilisateur lorsqu’une application tierce demande l’accès aux API. Le consentement ne peut pas être contourné.
* **Connexions** : Les applications tierces ne peuvent authentifier les utilisateurs qu’au moyen de connexions de niveau domaine. Pour en savoir plus, consultez [Promouvoir les connexions au niveau du domaine](/docs/fr-CA/authenticate/identity-providers/promote-connections-to-domain-level).

<div id="use-cases">
  ## Cas d’utilisation
</div>

Les cas d’utilisation courants des applications tierces comprennent :

* **Intégrations partenaires** : Des partenaires externes créent des applications qui appellent vos API au nom de l’utilisateur. Par exemple, un fournisseur de CRM intègre votre plateforme afin que des clients communs puissent synchroniser des données entre les deux produits.
* **Agents IA et clients MCP** : Des outils propulsés par l’IA, comme Claude Code, VS Code avec Copilot ou des serveurs MCP personnalisés, se connectent à vos API pour effectuer des actions au nom de l’utilisateur. Pour en savoir plus, consultez [Auth for MCP](https://auth0.com/ai/docs/mcp/intro/overview).
* **Écosystèmes de développeurs** : Vous exposez des API pour permettre à des développeurs externes de créer des applications sur votre plateforme, que ce soit au moyen d’un portail développeur, d’un Marketplace ou de la [Dynamic Client Registration](/docs/fr-CA/get-started/applications/dynamic-client-registration).
* **Cross App Access (XAA)** : Une application destinée au personnel dans le tenant d’une autre organisation accède à vos API par l’intermédiaire d’une relation de confiance, où l’application demandeuse est modélisée comme une application tierce dans votre tenant. Pour en savoir plus, consultez [Cross App Access](/docs/fr-CA/secure/call-apis-on-users-behalf/xaa).

<div id="supported-client-types">
  ## Types de clients pris en charge
</div>

Les applications tierces prennent en charge les types de clients confidentiel et public :

| **Type de client** | **Type d’application**         | **Cas d’utilisation**                  |
| ------------------ | ------------------------------ | -------------------------------------- |
| Confidentiel       | Application Web traditionnelle | Intégrations partenaires côté serveur  |
| Public             | Application monopage           | Widgets partenaires dans le navigateur |
| Public             | Native                         | Applications mobiles partenaires       |

<div id="supported-grant-types">
  ## Types de grant pris en charge
</div>

Les applications tierces prennent en charge :

* `authorization_code` avec PKCE obligatoire
* `refresh_token`
* `client_credentials` (clients confidentiels uniquement)

<div id="get-started">
  ## Premiers pas
</div>

Pour configurer une application tierce dans Auth0, suivez les étapes suivantes :

<div id="step-1-create-the-application">
  ### Étape 1 : Créer l’application
</div>

[Créer une application tierce](/docs/fr-CA/get-started/applications/third-party-applications/configure-third-party-applications#create-a-third-party-application) dans l’Auth0 Dashboard ou au moyen de la Management API.

<div id="step-2-configure-api-access">
  ### Étape 2 : Configurer l’accès à l’API
</div>

Les applications tierces exigent toujours une autorisation explicite pour accéder à vos API, même lorsque la politique d’accès de l’API est définie sur **Allow All.** Vous configurez les politiques d’accès aux API à l’aide des [client grants](/docs/fr-CA/get-started/applications/application-access-to-apis-client-grants).

Vous pouvez configurer des autorisations par défaut qui s’appliquent automatiquement à toutes les applications tierces. C’est utile lorsque vous gérez un grand nombre d’applications tierces ou que vous utilisez [Dynamic Client Registration](/docs/fr-CA/get-started/applications/dynamic-client-registration). Pour en savoir plus, consultez [Autorisations par défaut pour les applications tierces](/docs/fr-CA/get-started/applications/application-access-to-apis-client-grants#default-permissions-for-third-party-applications).

Vous pouvez aussi définir des autorisations précises pour certaines applications à l’aide des [client grants](/docs/fr-CA/get-started/applications/application-access-to-apis-client-grants). Les autorisations par application priment sur les valeurs par défaut. Pour en savoir plus, consultez [Créer un client grant](/docs/fr-CA/get-started/applications/application-access-to-apis-client-grants#create-client-grant).

<div id="step-3-configure-connections">
  ### Étape 3 : Configurer les connexions
</div>

Les applications tierces ne peuvent authentifier les utilisateurs qu’au moyen de connexions promues au niveau du domaine. Les connexions au niveau du domaine sont accessibles à toutes les applications tierces du tenant.

Pour promouvoir une connexion au niveau du domaine, consultez [Promouvoir les connexions au niveau du domaine](/docs/fr-CA/authenticate/identity-providers/promote-connections-to-domain-level).

<div id="step-4-users-provide-consent">
  ### Étape 4 : Les utilisateurs donnent leur consentement
</div>

Lorsqu’un utilisateur s’authentifie au moyen d’une application tierce, Auth0 affiche une boîte de dialogue de consentement pour lui demander d’approuver les autorisations demandées. Le consentement est toujours requis pour les applications tierces et ne peut pas être contourné.

Pour en savoir plus, consultez [Consentement de l’utilisateur et applications tierces](/docs/fr-CA/get-started/applications/third-party-applications/user-consent-and-third-party-applications).

<div id="dynamic-client-registration">
  ## Dynamic Client Registration
</div>

[Dynamic Client Registration](/docs/fr-CA/get-started/applications/dynamic-client-registration) crée par défaut des applications tierces dotées de contrôles de sécurité renforcés. Avant d’activer DCR pour les applications tierces, vous devez [configurer les autorisations d’API par défaut](/docs/fr-CA/get-started/applications/dynamic-client-registration#configure-api-access-for-dcr-clients) afin que les clients enregistrés dynamiquement puissent accéder à vos API.
