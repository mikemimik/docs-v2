Dans la plupart des cas, l’authentification des utilisateurs par l’intermédiaire d’Auth0 nécessite une connexion Internet. Toutefois, vous pouvez quand même développer et tester localement des applications qui utilisent Auth0. Dans certains cas, vous pourriez même ne pas avoir besoin d’une connexion Internet.

Pour savoir comment structurer vos environnements de développement, de test et de production lorsque vous utilisez Auth0, consultez [Set Up Multiple Environments](/docs/fr-CA/get-started/auth0-overview/create-tenants/set-up-multiple-environments).

<div id="use-json-web-tokens-with-client-side-applications">
  ## Utiliser les JSON Web Tokens avec des applications côté client
</div>

Comme les [JSON Web Tokens (JWT)](/docs/fr-CA/secure/tokens/json-web-tokens) sont sans état (c’est-à-dire que l’application qui les utilise ne tient compte que de leur contenu, et non de leurs états précédents), il s’agit de l’un des scénarios les plus faciles à tester localement.

Vous pouvez obtenir des <Tooltip tip="JSON Web Token (JWT) : format standard de ID Token (et souvent de Access Token) utilisé pour représenter de façon sécurisée des claims entre deux parties." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=JWTs">JWT</Tooltip> pour vos tests à l’aide de l’une des méthodes suivantes :

1. Créez un utilisateur de test pour une [connexion de base de données](/docs/fr-CA/get-started/applications/set-up-database-connections), puis connectez cet utilisateur par programmation. Essentiellement, vous utilisez le [Resource Owner Password Flow](/docs/fr-CA/get-started/authentication-and-authorization-flow/resource-owner-password-flow). Pour obtenir des instructions détaillées sur l’implémentation, consultez [Call Your API Using the Resource Owner Password Flow](/docs/fr-CA/get-started/authentication-and-authorization-flow/resource-owner-password-flow/call-your-api-using-resource-owner-password-flow).
2. Utilisez un robot de navigateur pour jouer le rôle d’un utilisateur, ouvrir une session et récupérer un JWT.

<div id="use-sessions-with-server-side-applications">
  ## Utiliser des sessions avec des applications côté serveur
</div>

À moins que votre application côté serveur ne permette de générer des sessions artificielles à des fins de test, vous devrez trouver un moyen d’effectuer manuellement une connexion avec Auth0.

Pour un aperçu général de la marche à suivre, consultez [Flux du code d’autorisation](/docs/fr-CA/get-started/authentication-and-authorization-flow/authorization-code-flow). Pour des instructions détaillées sur la mise en œuvre, consultez notre tutoriel, [Ajouter la connexion à l’aide du flux du code d’autorisation](/docs/fr-CA/get-started/authentication-and-authorization-flow/authorization-code-flow/add-login-auth-code-flow).

<div id="use-local-domains-with-auth0">
  ## Utiliser des domaines locaux avec Auth0
</div>

Si vous développez votre application localement, vous pouvez utiliser `localhost` et d’autres domaines auxquels Auth0 n’a pas accès (comme ceux d’un intranet) comme [URL de rappel](/docs/fr-CA/authenticate/login/redirect-users-after-login). Par exemple, pendant le développement, vous pourriez utiliser `http://localhost:3000/callback` comme URL de rappel.

1. Accédez à [Auth0 Dashboard &gt; Applications &gt; Applications](https://manage.auth0.com/#/applications/\{yourClientId}/settings) et cliquez sur l’application.
2. Ajoutez l’URL à la liste **Allowed Callback URLs**.

Comme le principal protocole d’identité d’Auth0 est [OpenID Connect (OIDC)](/docs/fr-CA/authenticate/protocols/openid-connect-protocol), Auth0 n’a pas à appeler directement le serveur de votre application. À la place, Auth0 redirige les utilisateurs vers le ou les points de terminaison de votre application avec les informations requises dans une chaîne de requête ou un fragment de hachage.

<div id="divert-emails-for-testing">
  ## Rediriger les courriels pour les tests
</div>

Si vous voulez tester votre application locale sans que les courriels (création, validation, etc.) soient envoyés à l’adresse courriel réelle des utilisateurs que votre application crée ou valide, Auth0 recommande d’utiliser un fournisseur de courriel personnalisé. Par exemple, un service comme [Mailtrap](https://mailtrap.io/signin) ou votre propre implémentation de serveur SMTP peut appliquer toute la logique nécessaire pour intercepter les courriels. Ainsi, les utilisateurs ne reçoivent pas de courriels, mais vous pouvez y accéder pour la validation et le dépannage. Pour en savoir plus, consultez [Configurer le serveur de courriel SMTP de test](/docs/fr-CA/customize/email/configure-test-smtp-email-servers).

Auth0 n’autorise pas certains domaines « faux » couramment utilisés pendant les tests. Utilisez de vraies adresses courriel pour éviter les problèmes ou les erreurs.
