Si votre cas d’utilisation concerne une <Tooltip tip="Audience : identifiant unique de l’audience d’un jeton émis. Nommée aud dans un jeton, sa valeur contient l’ID d’une application (Client ID) pour un ID Token ou d’une API (API Identifier) pour un jeton d’accès." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=audience">audience</Tooltip> non interactive, comme des API ou des serveurs backend, vous ferez l’intégration à l’aide d’une configuration machine-to-machine (M2M).

<div id="use-cases">
  ## Cas d’utilisation
</div>

Utilisez le parcours d’intégration M2M si vous :

* prenez en charge des communications de service à service
* avez des jobs planifiés ou des tâches cron exécutées sur des serveurs qui doivent accéder à des ressources protégées ou à des API
* permettez à des appareils IoT de communiquer avec des services backend ou des API
* avez une couche API qui doit communiquer avec d’autres couches API sans intervention de l’utilisateur ou après l’expiration d’un jeton utilisateur
* avez une API privilégiée qui peut devoir être appelée avant qu’un utilisateur soit authentifié (c.-à-d. à partir d’une Action ou d’un script de base de données personnalisé dans votre tenant Auth0)
* utilisez API Gateway pour gérer des services backend
* utilisez ou prenez en charge des applications non interactives ou d’autres outils ne nécessitant aucune interaction humaine, comme des démons ou des services backend

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Ces services auront quand même besoin d’un jeton d’accès M2M pour l’authentification.
</Callout>

<div id="how-to-use-this-guide">
  ## Comment utiliser ce guide
</div>

Ce guide vous aide à créer votre mise en œuvre M2M dans Auth0. Nous y présentons des éléments à prendre en compte, des bonnes pratiques et des concepts à passer en revue.

* Dans Architecture, nous vous conseillons de configurer Auth0 de façon à prendre en charge votre cycle de vie du développement logiciel et votre infrastructure existante.
* Dans Créer un compte, nous fournissons des instructions pour créer votre instance d’API dans Auth0 ainsi qu’une application prenant en charge le flux d’authentification (ou `grant`) nécessaire à l’authentification machine à machine.
* Dans Authentification, nous vous expliquons le `grant` à utiliser pour l’authentification, ainsi que les <Tooltip tip="Jeton d’accès : justificatif d’autorisation, sous la forme d’une chaîne opaque ou d’un JWT, utilisé pour accéder à une API." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=access+tokens">jetons d’accès</Tooltip> et les permissions (ou `scopes`) que vous pouvez définir.
* Dans Branding, nous vous indiquons où trouver l’information nécessaire pour configurer les <Tooltip tip="Domaine personnalisé : domaine tiers ayant un nom spécialisé ou personnalisé." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Custom+Domains">Custom Domains</Tooltip> selon la façon dont vous comptez gérer les certificats.
* Dans Automatisations de déploiement, vous pouvez en apprendre davantage sur nos outils conçus pour faciliter le déploiement.
* Dans Assurance qualité, vous pouvez en apprendre davantage sur les tests unitaires et sur les vérifications de préparation que nous fournissons dans le <Tooltip tip="Auth0 Dashboard : principal produit d’Auth0 pour configurer vos services." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip>.

<div id="architecture">
  ## Architecture
</div>

Avant de configurer votre compte et votre tenant Auth0, ou encore les groupes et les structures de vos services Auth0, dressez une cartographie de votre infrastructure existante afin de tirer le meilleur parti des capacités d’Auth0 dans votre écosystème actuel.

Comme indiqué dans les scénarios courants, il se peut que vous ayez d’autres technologies non interactives dans le domaine de votre application, de votre réseau ou de vos appareils M2M à prendre en compte avant de configurer Auth0. Pour voir un exemple de scénario M2M, consultez [Server + API](/docs/fr-CA/get-started/architecture-scenarios/server-application-api/part-1). Pour suivre un laboratoire pratique avec Node et tester le déploiement d’API, consultez notre [dépôt GitHub](https://www.google.com/url?q=https://github.com/auth0-training/labs-node-working-with-apis\&sa=D\&source=docs\&ust=1723578190947730\&usg=AOvVaw3IXRQG9ogH4pW6LEY047T6).

Vous pourriez également créer une visualisation de votre pile technologique actuelle et planifier l’intégration d’Auth0 à votre cycle de développement logiciel (SDLC) actuel. Cela peut vous aider à déterminer le nombre de tenants dont vous pourriez avoir besoin.

<div id="considerations">
  ### Considérations
</div>

Avant de créer un nouveau compte ou de configurer votre premier tenant, il peut être utile de réfléchir aux points suivants :

* La façon dont vous partitionnez ou regroupez vos API pour appeler des endpoints précis.

  * Cela peut déterminer l’audience et d’autres claims dans les jetons d’accès.
* Des consommateurs tiers de votre ressource peuvent demander un jeton d’accès pour chaque appel. Un volume d’appels trop élevé pourrait avoir une incidence sur votre [limite de débit](/docs/fr-CA/troubleshoot/customer-support/operational-policies/rate-limit-policy/rate-limit-configurations/essentials-professional-b2b).

  * Vous pouvez utiliser API Gateway pour limiter le nombre de jetons d’accès qu’un tiers peut demander. Pour en savoir plus sur API Gateway et Auth0, consultez [Configurer un fournisseur d’identité dans Access Gateway](/docs/fr-CA/get-started/auth0-overview/create-applications/configure-an-identity-provider-in-access-gateway).

<div id="create-an-account">
  ## Créer un compte
</div>

Maintenant que vous avez défini votre architecture, vous allez configurer votre compte Auth0 et vos tenants. Lorsque vous vous inscrivez aux services Auth0, vous créez votre premier [tenant](/docs/fr-CA/get-started/auth0-overview/create-tenants). C’est là que vous configurez les composantes, les services et les ressources Auth0. [Inscrivez-vous](https://auth0.com/signup) pour commencer.

<Card title="Avant de commencer">
  Dans l’Auth0 Dashboard ou avec l’Auth0 Management API, créez :

  * Une API pour représenter votre API
  * Une application M2M pour utiliser le flux d’identifiants du client

  Vous voudrez peut-être planifier certains détails de configuration avant de créer un compte.

  * Le nom de votre tenant joue un rôle dans votre Auth0 domain. Avant d’en choisir un, vous devriez consulter les [caractéristiques du tenant](/docs/fr-CA/get-started/auth0-overview/create-tenants#create-a-tenant-and-domain).
  * Les fonctionnalités Auth0 dont vous avez besoin pour votre cas d’utilisation. Certaines fonctionnalités sont offertes uniquement avec les forfaits Professional et Enterprise.
  * Déterminez si vous devez prendre en charge plusieurs environnements, comme le développement, le staging et la production. Pour en savoir plus, consultez [Set Up Multiple Environments](/docs/fr-CA/get-started/auth0-overview/create-tenants/set-up-multiple-environments).
  * Si votre cas d’utilisation implique des applications tierces que vous souhaitez enregistrer dans un tenant, vous pouvez utiliser la [Dynamic Application Registration](/docs/fr-CA/get-started/applications/dynamic-client-registration) selon la spécification [OIDC Client Registration](https://openid.net/specs/openid-connect-registration-1_0.html).
</Card>

<div id="authentication">
  ## Authentification
</div>

Lorsqu’une API appelle une autre API, ou dans tout contexte où aucun utilisateur n’est authentifié, vous avez besoin d’un moyen d’autoriser l’application plutôt qu’un utilisateur. Il s’agit d’un processus en une seule étape où l’application est authentifiée (à l’aide d’un `client_id` et d’un `client_secret`), puis autorisée en un seul appel.

Pour les applications non interactives ou les services qui doivent s’authentifier, vous devez sélectionner un client grant, ou un flux d’authentification. Le <Tooltip tip="OAuth 2.0 : cadre d’autorisation qui définit les protocoles et les workflows d’autorisation." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=OAuth+2.0">OAuth 2.0</Tooltip> [flux d’identifiants du client](https://tools.ietf.org/html/rfc6749#section-4.4) ne nécessite aucune interaction humaine et convient le mieux aux applications M2M.

<Card title="Avant de commencer">
  Dans Auth0 Dashboard ou Management API, vous devrez :

  * Configurer votre application pour utiliser le flux d’identifiants du client
  * Mettre à jour les scopes de vos jetons d’accès M2M

  Avant de configurer votre méthode d’authentification :

  * Consultez le [flux d’identifiants du client](/docs/fr-CA/get-started/authentication-and-authorization-flow/client-credentials-flow) pour l’authentification machine à machine. Il s’agit du processus d’authentification et d’autorisation non interactives.
  * Déterminez le niveau d’accès de vos API. Cela vous aidera à déterminer quels [scopes](/docs/fr-CA/get-started/apis/scopes/api-scopes), ou permissions, vous configurerez lorsque vous créerez votre API.
</Card>

<div id="configure-the-client-credential-flow">
  ### Configurer le flux d’identifiants du client
</div>

Vous pouvez utiliser l’Auth0 Dashboard ou <Tooltip tip="Management API : un produit qui permet aux clients d’effectuer des tâches administratives." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Management+API">Management API</Tooltip> pour configurer le flux d’authentification afin d’échanger les informations d’identification du client contre un jeton d’accès.

Suivez les instructions de [Mettre à jour les types d’autorisation](/docs/fr-CA/get-started/applications/update-grant-types) pour utiliser l’Auth0 Dashboard ou Management API.

<div id="m2m-access-tokens">
  ### Jetons d’accès M2M
</div>

Dans le cadre de l’authentification par jeton, les clients non interactifs fournissent `client_id` et `client_secret` lors d’un appel au [point de terminaison de jeton de l’Authentication API](/docs/fr-CA/customize/actions/explore-triggers/machine-to-machine-trigger) pour obtenir un [jeton d’accès](/docs/fr-CA/secure/tokens/access-tokens). Ce jeton d’accès permet d’accéder à votre API protégée.

Le profil, ou format, par défaut est le profil de jeton Auth0, qui fait partie des deux profils de jeton offerts. Vous pouvez choisir d’utiliser le profil de jeton RFC 9068. Pour en savoir plus, consultez [Profils de jetons d’accès](/docs/fr-CA/secure/tokens/access-tokens/access-token-profiles). Pour vérifier que le jeton est valide, votre API vérifiera les [algorithmes de signature](/docs/fr-CA/get-started/applications/signing-algorithms). L’<Tooltip tip="Algorithme de signature : algorithme utilisé pour signer numériquement les jetons afin de garantir qu’ils n’ont pas été altérés." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=signing+algorithm">algorithme de signature</Tooltip> par défaut est RSA256, un algorithme fondé sur une clé.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Auth0 prend en charge d’autres méthodes d’authentification du client que la fourniture d’un Client ID et d’un Client Secret comme informations d’identification. Ces méthodes, y compris notre recommandation d’utiliser [Private Key JWT](/docs/fr-CA/get-started/authentication-and-authorization-flow/authenticate-with-private-key-jwt) pour les configurations M2M, sont offertes avec un plan Enterprise. Pour en savoir plus, consultez [Application Credentials](/docs/fr-CA/secure/application-credentials).
</Callout>

<div id="token-expiration">
  #### Expiration des jetons
</div>

La durée de validité de vos jetons d’accès est limitée. Comme vos communications se font par le back-channel, vous ne pouvez pas utiliser les <Tooltip tip="Refresh Token : jeton utilisé pour obtenir un nouveau jeton d’accès sans obliger les utilisateurs à se reconnecter." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=refresh+tokens">jetons de renouvellement</Tooltip> pour prolonger les sessions; vous devriez donc envisager de configurer vos jetons d’accès avec une durée d’expiration d’une heure. Vous devrez peut-être trouver le bon équilibre entre sécurité et performance pour votre environnement. Pour en savoir plus, consultez [Mettre à jour la durée de validité des jetons d’accès](/docs/fr-CA/secure/tokens/access-tokens/update-access-token-lifetime).

<div id="scopes">
  ### Scopes
</div>

Avant que des clients ou des services non interactifs appellent votre API, vous devez définir les permissions ou les scopes qu’elle autorise. Vous pouvez définir les scopes dans l’Auth0 Dashboard afin de les inclure dans votre requête d’authentification à l’Authentication API. Pour voir d’autres exemples d’utilisation des scopes d’API, consultez [API Scopes](/docs/fr-CA/get-started/apis/scopes/api-scopes).

Pour configurer les scopes, suivez les instructions de [Ajouter des permissions d’API](/docs/fr-CA/get-started/apis/add-api-permissions) dans l’Auth0 Dashboard ou utilisez l’exemple fourni pour la Management API.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Pour ajouter des claims personnalisés à un jeton d’accès, vous pouvez utiliser le flux Machine to Machine d’Actions. Pour en savoir plus, consultez Machine to Machine Flow.
</Callout>

  ## Image de marque

Même si vous prenez en charge des clients non interactifs ou des services qui communiquent par le back-channel, vous pouvez tout de même personnaliser l’expérience pour qu’elle reflète l’image de votre marque actuelle.

<div id="custom-domains">
  ### Domaines personnalisés
</div>

Auth0 prend en charge l’utilisation de domaines personnalisés lorsque vous appelez le point de terminaison `/authorize` pour demander des jetons d’accès.

<Card title="Onboarding M2M - domaines personnalisés">
  Dans Auth0 Dashboard, vous devez :

  * Enregistrer et vérifier votre domaine avant de pouvoir l’utiliser avec vos services Auth0.
  * Déterminer si vous souhaitez gérer votre propre certificat ou utiliser un certificat géré par Auth0. Pour en savoir plus sur les certificats, consultez [Options de gestion des certificats](/docs/fr-CA/customize/custom-domains#certificate-management-options).
  * Vérifier que la version TLS (SSL) et l’algorithme de chiffrement que vous souhaitez utiliser pour les certificats autogérés sont pris en charge par Auth0. Pour en savoir plus, consultez [Versions et algorithmes de chiffrement TLS (SSL)](/docs/fr-CA/customize/custom-domains/self-managed-certificates/tls-ssl).
</Card>

1. Pour configurer votre domaine personnalisé avec des certificats gérés par Auth0, suivez les instructions de [Configurer des domaines personnalisés avec des certificats gérés par Auth0](/docs/fr-CA/customize/custom-domains/auth0-managed-certificates).

   1. Si vous souhaitez gérer vos propres certificats, suivez les instructions de [Configurer des domaines personnalisés avec des certificats autogérés](/docs/fr-CA/customize/custom-domains/self-managed-certificates).

      <Callout icon="file-lines" color="#0EA5E9" iconType="regular">
        Vous devez avoir un abonnement Enterprise pour gérer les certificats de votre domaine personnalisé. Pour en savoir plus, consultez [Auth0 Pricing](https://auth0.com/pricing/) et [Login](/docs/fr-CA/authenticate/login).
      </Callout>
2. Consultez [Configuration de l’API avec des domaines personnalisés](/docs/fr-CA/customize/custom-domains/configure-features-to-use-custom-domains#apis). Vous devrez peut-être ajuster les paramètres de votre API pour y intégrer un domaine personnalisé.

<div id="deployment-automation">
  ## Automatisation du déploiement
</div>

Auth0 prend en charge quelques options d’automatisation du déploiement, et elles peuvent être utilisées conjointement.

<Card title="Pratique exemplaire">
  Quelle que soit la façon dont vous configurez l’automatisation du déploiement, nous vous recommandons de faire des tests unitaires de votre code personnalisé et de vos Actions avant le déploiement, puis d’exécuter également quelques tests d’intégration sur votre tenant après le déploiement.
</Card>

<div id="deploy-cli-tool">
  ### Outil Deploy CLI
</div>

Comme recommandé dans la section Architecture, vous devriez avoir des tenants Auth0 pour le développement, le test et la production. Ces tenants devraient partager des configurations identiques pour l’assurance qualité et les tests; toutefois, vous pourriez rencontrer des erreurs en raison d’écarts de configuration entre vos environnements. Par exemple, chaque environnement aura des <Tooltip tip="Client ID : valeur d’identification attribuée à votre ressource enregistrée par Auth0." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Client+IDs">Client IDs</Tooltip> et des <Tooltip tip="Client ID : valeur d’identification attribuée à votre ressource enregistrée par Auth0." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Client+Secrets">Client Secrets</Tooltip> différents.

Pour atténuer ces erreurs de non-concordance, vous pouvez utiliser l’outil Deploy CLI pour intégrer votre instance Auth0 à votre pipeline CI/CD existant. Grâce au remplacement dynamique de mots-clés, vous pouvez remplacer les variables d’environnement dans des tenants qui partagent des configurations semblables. Pour en savoir plus, consultez [Outil Deploy CLI](/docs/fr-CA/deploy-monitor/deploy-cli-tool) et [Remplacement de mots-clés](/docs/fr-CA/deploy-monitor/deploy-cli-tool/keyword-replacement).

<div id="actions-real-time-logs">
  ### Actions Real-time Logs
</div>

Actions Real-time Logs affiche tous les journaux du code personnalisé en temps réel, y compris la sortie de `console.log` et d’autres exceptions. Si vous utilisez Auth0 Actions ou une autre logique personnalisée, vous pouvez utiliser cette extension pour faire du débogage et résoudre les problèmes. Pour en savoir plus sur l’installation et la configuration, consultez [Actions Real-time Logs](/docs/fr-CA/customize/actions/actions-real-time-logs).

<div id="quality-assurance">
  ## Assurance qualité
</div>

L’assurance qualité est essentielle pour repérer les problèmes avant la mise en production. Selon la nature de votre projet, vous voudrez envisager plusieurs types de tests d’assurance qualité dans le cadre de votre intégration avec Auth0 :

* Comment vos API se comporteront-elles si elles sont soumises à des charges de production imprévues ?
* Quel effet les applications tierces auront-elles sur vos limites de débit ?

Si vous n’utilisez pas les widgets ou les fonctionnalités d’Auth0, comme <Tooltip tip="Universal Login : votre application est redirigée vers Universal Login, hébergé sur l’Authorization Server d’Auth0, afin de vérifier l’identité d’un utilisateur." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Universal+Login">Universal Login</Tooltip>, vous ne bénéficierez pas d’emblée des bonnes pratiques intégrées en matière de convivialité et d’accessibilité sur un large éventail de navigateurs et d’appareils. Pour vous assurer que les exigences fonctionnelles sont respectées et que les événements imprévus sont gérés correctement, des conseils sont fournis pour tester l’intégration entre votre ou vos applications et Auth0, ainsi que pour effectuer des tests unitaires sur des modules d’extensibilité individuels, comme Auth0 Actions. Nous vous recommandons également de consulter la [politique de tests d’intrusion](/docs/fr-CA/troubleshoot/customer-support/operational-policies/penetration-testing-policy) d’Auth0 et d’effectuer des tests simulés que vous pouvez utiliser conjointement avec notre [politique de tests de charge](/docs/fr-CA/troubleshoot/customer-support/operational-policies/load-testing-policy) afin de contribuer à garantir le bon fonctionnement de votre ou vos applications en cas de charge imprévue.

<div id="unit-testing">
  ### Tests unitaires
</div>

Les tests unitaires servent à vérifier des unités d’extensibilité, comme les Actions Auth0. Si vous utilisez du code personnalisé, nous vous recommandons d’utiliser un framework de test (comme [Mocha](https://mochajs.org/)) pour tester le code ajouté avant le déploiement.

<div id="mock-testing">
  ### Tests simulés
</div>

Pour concilier la [politique de tests de charge](/docs/fr-CA/troubleshoot/customer-support/operational-policies/load-testing-policy) d’Auth0 et le besoin d’effectuer des tests de charge, il est courant de créer une version simulée des points de terminaison d’Auth0. C’est une pratique utile pour vous assurer que votre configuration fonctionne avec les interfaces attendues sans avoir à limiter vos tests, et des outils comme [MockServer](http://www.mock-server.com/), [JSON Server](https://github.com/typicode/json-server) ou même [Postman](https://learning.getpostman.com/docs/postman/mock_servers/setting_up_mock/) peuvent vous aider.

<div id="deployment">
  ## Déploiement
</div>

Notre section [Déployer et surveiller](/docs/fr-CA/deploy-monitor) fournit des conseils sur les meilleures pratiques de déploiement. Nous vous recommandons de consulter les [Vérifications avant le déploi](/docs/fr-CA/deploy-monitor/pre-deployment-checks)[ement](/docs/fr-CA/deploy-monitor/pre-deployment-checks), en particulier les [vérifications de préparation intégrées dans l’Auth0 Dashboard](/docs/fr-CA/deploy-monitor/pre-deployment-checks/how-to-run-production-checks).

Pour consulter la vérification de préparation, sélectionnez le menu déroulant sous le nom de votre tenant et l’étiquette d’environnement dans [Auth0 Dashboard &gt; Exécuter les vérifications de préparation](https://manage.auth0.com/#/production-checks).

<Frame>![Auth0 Dashboard > Liste de vérification de préparation](/docs/images/cdy7uua7fh8z/7iu1CzH0NgaXJbLBNWXZli/b598b8cdaad0c676e83a0fb0595d4603/Readiness_Checks_-_English.png)</Frame>

Vous pouvez utiliser le filtre pour appliquer les vérifications de préparation aux applications sélectionnées. **Ces vérifications ne s’appliquent pas aux API que vous avez configurées**.

Pour les vérifications qui ne s’appliquent pas à votre configuration particulière, vous pouvez sélectionner **Supprimer** pour les retirer des résultats finaux.

Nous vous recommandons de consulter les [meilleures pratiques de déploiement](/docs/fr-CA/deploy-monitor/deployment-best-practices) pour effectuer les dernières vérifications avant la mise en service et d’utiliser les [journaux](/docs/fr-CA/deploy-monitor/logs) pour surveiller vos services.
