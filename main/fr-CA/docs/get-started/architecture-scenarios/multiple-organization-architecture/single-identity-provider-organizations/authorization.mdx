Lorsqu’on réfléchit à l’autorisation, il faut généralement se demander comment déterminer ce qu’une personne est autorisée à faire et comment transmettre cette information à vos applications et/ou API. Selon les applications que vous avez, l’un ou l’autre de ces aspects — ou les deux — peut vous concerner. Dans nos scénarios d’architecture, nous fournissons des conseils généraux sur [l’autorisation B2B](/fr-CA/docs/get-started/architecture-scenarios/business-to-business/authorization), que nous vous recommandons de consulter en complément des conseils fournis ici.

* Les [ID Tokens](/fr-CA/docs/secure/tokens/id-tokens) servent souvent à transmettre aux applications des informations d’autorisation utilisateur au moyen de claims personnalisées, qui peuvent être ajoutées à l’aide de l’extensibilité [Rules](/fr-CA/docs/customize/rules). Les claims ajoutées peuvent vous permettre de présenter une interface utilisateur dans laquelle les utilisateurs ne peuvent pas essayer d’effectuer une action qu’ils n’ont pas la permission d’exécuter. Les informations d’autorisation contenues dans un <Tooltip tip="ID Token : justificatif destiné au client lui-même plutôt qu’à l’accès à une ressource." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=ID+Token">ID Token</Tooltip> permettent aussi au backend d’une application d’empêcher les utilisateurs de contourner les contrôles du frontend dans les applications Web traditionnelles.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  **Pratique exemplaire**

  Vous pouvez tirer parti de [Auth0 Role-Based Access Control](/fr-CA/docs/manage-users/access-control/rbac) (RBAC) à l’aide de la fonctionnalité [Auth0 Authorization Core](/fr-CA/docs/manage-users/access-control/configure-core-rbac) pour définir des permissions d’accès, qui peuvent être appliquées automatiquement aux jetons d’accès. Pour en savoir plus, consultez [Enable Role-Based Access Control for APIs](/fr-CA/docs/get-started/apis/enable-role-based-access-control-for-apis).

  La fonctionnalité RBAC d’Auth0 peut également fournir des informations que vous pouvez ajouter sous forme de claims personnalisées aux jetons d’identité (et aux jetons d’accès, si vous préférez les appliquer manuellement). Auth0 Organizations peut aussi tirer parti des capacités RBAC d’Auth0 au moyen d’un ou de plusieurs rôles attribués selon l’appartenance. Pour en savoir plus, consultez [Add Roles to Organization Members](/fr-CA/docs/manage-users/organizations/configure-organizations/add-member-roles).
</Callout>

* Les API qui offrent un accès public à des services de ressources partagées sont généralement protégées par des mécanismes de contrôle d’accès. À cette fin, Auth0 offre la possibilité de créer un bearer token d’autorisation, ou <Tooltip tip="OAuth 2.0 : cadre d’autorisation qui définit les protocoles et workflows d’autorisation." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=OAuth">OAuth</Tooltip> 2 [jeton d’accès](/fr-CA/docs/secure/tokens/access-tokens), qui peut transmettre à une API des informations d’autorisation utilisateur, généralement en utilisant Auth0 [Role-Based Access Control (RBAC)](/fr-CA/docs/manage-users/access-control/configure-core-rbac) pour appliquer un ou plusieurs [rôles attribués selon l’appartenance](/fr-CA/docs/manage-users/organizations/configure-organizations/add-member-roles), ou en ajoutant des claims personnalisées au moyen de l’extensibilité [Rules](/fr-CA/docs/customize/rules). Vous pouvez aussi tirer parti des capacités RBAC d’Auth0 pour ajuster automatiquement la claim `scope` d’un <Tooltip tip="Jeton d’accès : justificatif d’autorisation, sous la forme d’une chaîne opaque ou d’un JWT, utilisé pour accéder à une API." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=access+token">jeton d’accès</Tooltip>. Les API peuvent ensuite utiliser cette information pour appliquer le niveau de contrôle d’accès approprié, ce qui permet à votre API d’appliquer les règles de policy sans devoir effectuer une recherche supplémentaire pour obtenir de l’information sur l’utilisateur.
* Dans certains cas, vous pourriez vouloir mettre en œuvre des policies au niveau de l’application dans le tenant Auth0; cela vous permet d’appliquer des policies à tout un ensemble d’applications et de services de ressources (API) sans avoir à modifier chacune d’elles individuellement. En général, cela se fait au moyen de l’extensibilité [Rules](/fr-CA/docs/customize/rules).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  **Pratique exemplaire**

  Auth0 Organizations donne à Auth0 Rules accès à des informations liées à l’organisation qui peuvent être consultées lors de l’authentification d’un utilisateur. Cette information est accessible au moyen de l’objet `organization` contenu dans l’objet `context` de Rules. L’objet `organization` donne aussi accès à toutes les métadonnées provisionnées dans une définition d’Organization dans Auth0. Pour en savoir plus, consultez [Custom Development for Organizations](/fr-CA/docs/manage-users/organizations/custom-development).
</Callout>

<div id="id-token-claims">
  ## Claims du jeton d’identité
</div>

En général, des claims peuvent être ajoutées aux jetons d’identité, comme l’expliquent nos directives de bonnes pratiques sur les [Claims du jeton d’identité](/fr-CA/docs/get-started/architecture-scenarios/business-to-consumer/authorization). Lorsque vous utilisez la fonctionnalité Auth0 Organizations, une revendication `org_id` est automatiquement ajoutée à tout jeton d’identité émis pour les utilisateurs ayant une appartenance à une organisation (pour un exemple, consultez [Utiliser les jetons et les organisations](/fr-CA/docs/manage-users/organizations/using-tokens)). Ce paramètre est validé par les SDK Auth0. Vous pouvez également ajouter des renseignements supplémentaires associés à une Auth0 Organization en ajoutant une revendication personnalisée au jeton d’identité :

```js wrap lines
context.idToken['http://travel0.net/multifactor'] = context.multifactor;
```

**Remarque** : Si vous avez configuré votre tenant pour qu’il prenne en charge les noms d’organisation dans l’Authentication API, la revendication `org_name` est automatiquement incluse dans les jetons d’identité. Pour en savoir plus, consultez [Utiliser les noms d’organisation dans l’Authentication API](/fr-CA/docs/manage-users/organizations/configure-organizations/use-org-name-authentication-api).

<div id="saml-assertion">
  ### Assertion SAML
</div>

Bien que la fonctionnalité Auth0 Organization ne prenne pas en charge les applications compatibles avec <Tooltip tip="Security Assertion Markup Language (SAML) : Protocole normalisé permettant à deux parties d’échanger des informations d’authentification sans mot de passe." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=SAML">SAML</Tooltip>, une assertion SAML générée par un <Tooltip tip="Fournisseur d’identité (IdP) : Service qui stocke et gère les identités numériques." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=Identity+Provider">fournisseur d’identité</Tooltip> (IdP) en amont peut être configurée pour alimenter des claims standards ou personnalisés dans un jeton d’identité utilisé en aval. Par exemple, vous pourriez définir la section des mappings d’une connexion d’entreprise SAML :

```json lines
{ 
  "user_id": [
    "http://schemas.xmlsoap.org/ws/2005/05/identity/claims/nameidentifier",
    "http://schemas.xmlsoap.org/ws/2005/05/identity/claims/upn",
    "http://schemas.xmlsoap.org/ws/2005/05/identity/claims/name"
  ],
  "email": "http://schemas.xmlsoap.org/ws/2005/05/identity/claims/emailaddress",
  "name": "http://schemas.xmlsoap.org/ws/2005/05/identity/claims/name",
  "given_name": [
    "http://schemas.xmlsoap.org/ws/2005/05/identity/claims/givenname",
    "http://schemas.xmlsoap.org/ws/2005/05/identity/claims/name"
  ], 
  "family_name": "http://schemas.xmlsoap.org/ws/2005/05/identity/claims/surname",
  "groups": "http://schemas.xmlsoap.org/claims/Group"
}
```

Dans cet exemple, chaque champ se verra attribuer une valeur dans l’objet `user` d’une Rule; celle-ci sera ensuite soit associée à des claims standard dans un jeton d’identité, soit mappée au moyen de claims personnalisés. Pour en savoir plus sur la personnalisation des mappages SAML, consultez [Connectez votre application aux fournisseurs d’identité SAML : configurer les mappages](/fr-CA/docs/authenticate/identity-providers/enterprise-identity-providers/saml#set-up-mappings).

<div id="access-token-claims">
  ## Claims du jeton d’accès
</div>

En plus de toute autre claim que vous ajoutez à votre jeton d’accès pour prendre des décisions de contrôle d’accès (consultez nos recommandations générales sur les [Claims du jeton d’accès](/fr-CA/docs/get-started/architecture-scenarios/business-to-consumer/authorization)), vous voudrez généralement indiquer l’organisation à laquelle l’utilisateur appartient.

Comme pour le jeton d’identité, lorsque vous utilisez la fonctionnalité Auth0 Organizations, la revendication `org_id` est automatiquement ajoutée à tout jeton d’accès (pour un exemple, consultez [Utiliser les jetons et les organisations](/fr-CA/docs/manage-users/organizations/using-tokens)) émis pour des utilisateurs ayant une appartenance à une organisation. Vous pouvez aussi ajouter des renseignements supplémentaires associés à une Auth0 Organization en ajoutant une revendication personnalisée au jeton d’accès :

```js wrap lines
context.accessToken['http://travel0.net/multifactor'] = context.multifactor;
```

**Remarque** : Si vous avez configuré votre tenant pour prendre en charge les noms d’organisation dans l’Authentication API, la claim `org_name` est automatiquement incluse dans les jetons d’accès. Pour en savoir plus, consultez [Use Organization Names in Authentication API](/fr-CA/docs/manage-users/organizations/configure-organizations/use-org-name-authentication-api).

Autrement, vous pouvez créer une <Tooltip tip="Audience : Identifiant unique de l’audience d’un token émis. Nommée aud dans un token, sa valeur contient l’ID d’une application (Client ID) pour un ID Token ou d’une API (API Identifier) pour un Access Token." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=audience">audience</Tooltip> d’API unique pour chaque organisation, ce qui donnerait lieu à une définition d’API distincte dans Auth0. Bien que ce mécanisme puisse réduire le besoin de recourir à l’extensibilité personnalisée des Rules, la complexité qu’il introduit peut être difficile à gérer. Voici une comparaison simple :

| Approche              | Avantages                                                                                                                                                                                                                                                                               | Inconvénients                                                                                                                                                                                                                                                                                           |
| --------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Audience d’API unique | <ul><li>Prise en charge prête à l’emploi de l’accès machine-to-machine pour une seule organisation.</li><li>L’audience est une claim standard dans un jeton d’accès.</li><li>Le traitement du refresh token ne nécessite aucune logique supplémentaire liée à l’organisation.</li></ul> | <ul><li>Il faut automatiser la création d’une API pour chaque organisation.</li><li>Il peut être nécessaire de créer des rôles distincts si vous utilisez RBAC.</li>Il faut automatiser l’attribution des Roles à la Membership.<li>L’implémentation de l’API doit gérer plusieurs audiences.</li></ul> |
| Claim personnalisée   | Simplifie la configuration du tenant Auth0.                                                                                                                                                                                                                                             | Du code personnalisé est nécessaire dans une rule pour ajouter l’organisation au jeton d’accès.                                                                                                                                                                                                         |

<div id="roles">
  ### Rôles
</div>

La fonctionnalité Auth0 Organizations prend également en charge le [Role-Based Access Control (RBAC)](/fr-CA/docs/manage-users/access-control/rbac) au moyen de la fonctionnalité [Authorization Core](/fr-CA/docs/manage-users/access-control/configure-core-rbac) associée à un tenant Auth0. Le RBAC est [appliqué au niveau de l’appartenance à une organisation Auth0](/fr-CA/docs/manage-users/organizations/configure-organizations/add-member-roles).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Vous pouvez [activer Auth0 Authorization Core RBAC pour une API](/fr-CA/docs/get-started/apis/enable-role-based-access-control-for-apis), ce qui modifiera automatiquement la claim `scope` par défaut dans les jetons d’accès et ajoutera également, par défaut, une claim `permission` (pour un exemple, consultez [Work with Tokens and Organizations](/fr-CA/docs/manage-users/organizations/using-tokens)). Vous pouvez aussi ajouter des renseignements sur les rôles aux jetons d’identité sous forme de claims personnalisées en accédant à l’objet `authorization` disponible dans l’objet `context` de Rules. Pour en savoir plus, consultez [Rules with Authorization Sample Use Cases: Add User Roles to Tokens](/fr-CA/docs/manage-users/access-control/sample-use-cases-rules-with-authorization).
</Callout>

<div id="access-control">
  ## Contrôle d’accès
</div>

L’application des politiques au niveau des ressources relève de la responsabilité des applications et/ou des API de votre système. Si vous tentez d’appliquer les politiques dans un <Tooltip tip="Authorization Server : Serveur centralisé qui contribue à définir les limites de l’accès d’un utilisateur. Par exemple, votre serveur d’autorisation peut contrôler les données, les tâches et les fonctionnalités accessibles à un utilisateur." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=Authorization+Server">Authorization Server</Tooltip> centralisé, comme votre tenant Auth0, vous vous retrouverez rapidement avec un système de contrôle complexe, difficile à maintenir et à comprendre. Votre Authorization Server centralisé peut plutôt s’assurer que les renseignements appropriés sur un utilisateur sont inclus dans les jetons, afin que vos applications et vos API disposent des renseignements nécessaires pour prendre des décisions d’application des politiques. À tout le moins, dans les situations où il y a trop de renseignements pour tenir dans un seul jeton (par exemple, des permissions au niveau des ressources) ou lorsque les renseignements changent assez souvent pour devenir obsolètes s’ils sont consultés directement dans le jeton, vos applications et vos API devraient pouvoir aller chercher les bons renseignements.

En revanche, certaines politiques de haut niveau peuvent être appliquées de façon centralisée. Par exemple, lorsque vous utilisez le contexte d’un tenant Auth0, il existe certaines situations où vous pourriez implémenter une Rule qui évite à chaque application et/ou API d’avoir à appliquer les mêmes restrictions. Cela comprend :

* bloquer l’accès des utilisateurs provenant d’une adresse IP particulière
* implémenter des exigences précises liées au Contextual ou à <Tooltip tip="Adaptive Multi-factor Authentication : Authentification multifacteur (MFA) qui n’est déclenchée pour les utilisateurs que lorsqu’une tentative de connexion est jugée peu fiable." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=Adaptive+MFA">Adaptive MFA</Tooltip>
* limiter la connexion aux seuls utilisateurs qui ont vérifié leur adresse courriel
* limiter l’accès à une audience d’API donnée, afin qu’un utilisateur ne puisse pas obtenir de jeton d’accès pour une autre audience d’API ou ne puisse pas obtenir de jeton d’accès pour cette audience dans certaines circonstances. Dans ce cas, si vous créez une audience d’API personnalisée pour chaque organisation, votre Rule doit aussi s’assurer que l’utilisateur qui s’authentifie appartient à l’organisation correspondant à l’audience d’API en question.

<Warning>
  L’accès à l’Auth0 Management API n’est pas limité par organisation; l’accès à l’Auth0 Management API se fait au moyen d’un jeton d’accès attribué pour une utilisation dans un contexte machine-to-machine et ne peut pas être restreint par une Auth0 Organization. Par conséquent, n’accordez pas à vos clients un accès direct à votre instance de l’Auth0 Management API. Si vos clients doivent gérer certains aspects de leur organisation, comme les comptes d’utilisateur (consultez [Gestion du profil](/fr-CA/docs/get-started/architecture-scenarios/multiple-organization-architecture/single-identity-provider-organizations/profile-management)), vous devriez créer votre propre application et/ou API indépendante à cette fin.
</Warning>
