Commençons par prendre un peu de recul et par parler du contrôle d’accès. Il n’existe pas de définition unique et parfaitement arrêtée du contrôle d’accès dans l’industrie, mais si vous prenez le temps de faire quelques recherches, vous verrez que la plupart des sources faisant autorité s’entendent pour dire qu’il s’agit du concept global qui regroupe l’authentification, l’autorisation, le consentement et l’application des politiques afin de garantir que seules les bonnes personnes et les bons services ont accès à vos applications et à vos API.

Examinons maintenant de plus près les distinctions entre l’authentification, l’autorisation, le consentement et l’application des politiques. Votre tenant Auth0 (votre <Tooltip tip="Serveur d’autorisation : serveur centralisé qui contribue à définir les limites de l’accès d’un utilisateur. Par exemple, votre serveur d’autorisation peut contrôler les données, les tâches et les fonctionnalités accessibles à un utilisateur." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=Authorization+Server">Serveur d’autorisation</Tooltip>) est généralement responsable de l’authentification et du consentement, ainsi que d’une partie ou de la totalité de l’autorisation et de l’application des politiques. De plus, l’application ou l’API elle-même constitue presque toujours le principal mécanisme d’application des politiques, surtout lorsqu’un accès contextuel est requis :

* **Authentification**: processus qui consiste à déterminer si un principal (un utilisateur ou une application) est bien qui ou ce qu’il prétend être.
* **Autorisation**: processus qui consiste à déterminer ce qui est autorisé, en fonction du principal, des permissions qui lui ont été accordées et/ou de l’ensemble des critères d’accès propres au contexte.
* **Consentement**: permissions que l’utilisateur (<Tooltip tip="Propriétaire de la ressource : entité (comme un utilisateur ou une application) capable d’accorder l’accès à une ressource protégée." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=Resource+Owner">Propriétaire de la ressource</Tooltip>) a accordées à une application pour agir en son nom. Il s’agit généralement d’une exigence propre à l’autorisation déléguée. L’utilisateur doit autoriser le client à accéder à ses données dans un autre système.
* **Application des politiques**: action d’appliquer les politiques de l’application ou de l’API, en refusant ou en autorisant l’accès selon les renseignements d’authentification et/ou d’autorisation d’un utilisateur.

De façon générale, nous regroupons les différents types de contrôle d’accès en trois catégories distinctes afin de mieux comprendre a) quel acteur est responsable du stockage de l’information, b) quel acteur est responsable de la prise de décisions, et c) quel acteur est responsable de l’application des restrictions.

* La première catégorie est celle où l’accès à une application ou à une API est accordé ou refusé dans son ensemble. Les données nécessaires à cette application ainsi que le processus d’application lui-même sont généralement définis dans le contexte du Serveur d’autorisation. Par exemple, en utilisant [`app_metadata`](/fr-CA/docs/manage-users/user-accounts/metadata) associé à un utilisateur et une [Action](/fr-CA/docs/customize/actions) définie dans votre tenant Auth0.
* La deuxième catégorie est celle où l’accès à un sous-ensemble précis des fonctionnalités d’une application ou d’une API est accordé ou refusé. Les données nécessaires à cette application sont généralement stockées dans le Serveur d’autorisation. Par exemple, en utilisant `app_metadata` sur un utilisateur dans votre tenant Auth0, tandis que l’application des politiques est assurée par l’application ou l’API elle-même. Dans ce scénario, les données sont généralement transmises sous forme d’une ou de plusieurs claims personnalisées dans un jeton [`id`](/fr-CA/docs/secure/tokens/id-tokens) ou [`access`](/fr-CA/docs/secure/tokens/access-tokens).
* La troisième catégorie est celle où l’accès est accordé ou refusé selon ce sur quoi le principal (sujet) peut agir dans le contexte d’une application ou d’une API. Les données nécessaires à cette application, de même que le processus d’application, sont généralement définis dans le contexte de l’application ou de l’API. Dans ce scénario, les données transmises sous forme d’une ou de plusieurs claims personnalisées dans un jeton `id` ou `access` peuvent être utilisées seules ou avec des données provenant d’une source externe autre qu’Auth0.

De plus, les mécanismes de contrôle d’accès basé sur les rôles (RBAC) et de contrôle d’accès basé sur les attributs (ABAC) peuvent être appliqués dans n’importe laquelle des catégories de contrôle d’accès décrites ci-dessus. Quel que soit votre cas d’utilisation, vous devrez tenir compte d’un certain nombre d’éléments lorsque vous évaluerez les fonctionnalités et le flux de travail dont vous avez besoin :

* Existe-t-il des scénarios où l’accès à une application entière ou à une API devrait être refusé ?
* Allez-vous fournir des API auxquelles des applications tierces pourront accéder ?
* Vos API seront-elles aussi accessibles par vos propres applications (de première partie) ?
* Votre application appellera-t-elle une API tierce ?
* Vos applications et/ou API devraient-elles appliquer le contrôle d’accès en fonction des claims de l’utilisateur ?
* Que faire si j’ai besoin de savoir à quelle organisation un <Tooltip tip="Jeton d’accès : justificatif d’autorisation, sous la forme d’une chaîne opaque ou d’un JWT, utilisé pour accéder à une API." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=access+token">jeton d’accès</Tooltip> ou un <Tooltip tip="Jeton d’accès : justificatif d’autorisation, sous la forme d’une chaîne opaque ou d’un JWT, utilisé pour accéder à une API." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=ID+token">jeton ID</Tooltip> est associé ?

Auth0 prend en charge la restriction de l’accès aux applications ou aux API selon certaines conditions. Dans certains cas, vous pourriez vouloir créer une Action qui renvoie une `UnauthorizedError` lorsque, par exemple, un utilisateur tente d’accéder à une application ou à une API au mauvais moment (comme décrit dans cet [exemple](/fr-CA/docs/manage-users/access-control/sample-use-cases-rules-with-authorization#allow-access-only-on-weekdays-for-a-specific-application)) — ou si l’utilisateur n’a pas le ou les claims requis dans son `app_metadata`. Pour une application qui utilise [OpenID Connect (OIDC)](/fr-CA/docs/authenticate/protocols/openid-connect-protocol), cela empêcherait l’émission du [jeton ID](/fr-CA/docs/secure/tokens/id-tokens) utilisé pour autoriser l’accès. De même, pour une API, l’émission de tout [jeton d’accès](/fr-CA/docs/secure/tokens/access-tokens) OAuth2 (utilisé pour appeler l’API) pourrait être empêchée, comme décrit dans cet [exemple](/fr-CA/docs/manage-users/access-control/sample-use-cases-rules-with-authorization#deny-access-to-anyone-calling-an-api).

<Info>
  ### Bonne pratique

  Dans l’ensemble, nous avons constaté que [OIDC](/fr-CA/docs/authenticate/protocols/openid-connect-protocol) est le protocole standard de l’industrie le plus couramment utilisé par les clients d’Auth0 pour l’authentification dans leurs applications. Nous avons aussi constaté que, même si [OAuth2](/fr-CA/docs/authenticate/protocols/oauth) a été conçu comme un protocole de délégation, il est couramment utilisé dans les applications de première partie lorsqu’il existe une API qui ne partage pas de session avec l’application.
</Info>

Auth0 peut également fournir l’information nécessaire pour qu’une application puisse appliquer des restrictions. Pour l’[intégration au niveau de l’application](#application-integration), Auth0 vous permet d’ajouter des [claims personnalisés](#id-token-claims) à un jeton ID, que votre application peut ensuite vérifier et utiliser pour appliquer une politique. Dans ce cas, vous devrez déterminer de quelle information votre application a besoin pour prendre des décisions d’application. Si vous devez prendre des décisions au niveau d’une API plutôt que dans votre application, vous devrez probablement utiliser un jeton d’accès plutôt qu’un jeton ID. Poursuivez votre lecture pour en savoir plus.

<Warning>
  Lorsque vous décidez quelles données inclure dans votre jeton ID et/ou votre jeton d’accès, tenez compte de la taille du jeton, surtout si vous le transmettez dans l’URL. Même si vous ne transmettez pas de jetons dans l’URL, vous devrez aussi tenir compte du risque d’exposer des PII (renseignements personnels identifiables) sensibles. L’information contenue dans le jeton n’est pas chiffrée. Ainsi, même si la divulgation d’un jeton ID ne pose généralement pas un problème de sécurité, elle peut devenir un enjeu de confidentialité selon les données incluses dans le jeton.
</Warning>

Pour l’[intégration au niveau de l’API](#api-integration), Auth0 prend en charge à la fois les [claims personnalisés](#access-token-claims) et la reconfiguration du [scope](#access-token-scopes), dans le contexte d’un jeton d’accès. Encore une fois, vous devrez déterminer de quelle information votre API aura besoin pour prendre des décisions d’accès, et votre API devra les appliquer en validant le contenu du jeton d’accès.

<Info>
  ### Bonne pratique

  Lorsque vous décidez si vous devriez utiliser des permissions au moyen de claims personnalisés ou de scopes, assurez-vous de bien comprendre la nature et le rôle des scopes. Il existe un excellent [article de blogue](https://auth0.com/blog/on-the-nature-of-oauth2-scopes/) à ce sujet, facile à lire, qui aide à clarifier la question.
</Info>

Dans les scénarios multi-organisations, il peut souvent être important de savoir à quelle organisation un jeton d’accès (ou même un jeton ID) s’applique. Le fait de suivre les [bonnes pratiques](#organization-data-in-an-access-token) peut vous faire gagner du temps et des efforts.

<div id="application-integration">
  ## Intégration d’application
</div>

Dans ce scénario, votre tenant Auth0 fournit un jeton comme preuve d’accès autorisé à une application. Pour les applications qui utilisent [OpenID Connect (OIDC)](/fr-CA/docs/authenticate/protocols/openid-connect-protocol), le protocole standard de l’industrie que nous considérons généralement comme le plus utilisé pour les applications destinées aux clients, il s’agirait d’un jeton ID présenté sous forme de [JWT](/fr-CA/docs/secure/tokens/json-web-tokens).

<div id="id-token-claims">
  ### Claims du jeton ID
</div>

Grâce à l’extensibilité Actions, Auth0 vous permet facilement d’[ajouter des claims personnalisés à un jeton ID](/fr-CA/docs/get-started/apis/scopes/sample-use-cases-scopes-and-claims#add-custom-claims-to-a-token) en fonction, par exemple, du contenu des [métadonnées](/fr-CA/docs/manage-users/user-accounts/metadata) d’un utilisateur. Votre application peut ensuite vérifier que le jeton ID contient les claims nécessaires, puis autoriser ou refuser l’accès à certaines fonctionnalités, selon le besoin. Notez que, même si le processus d’ajout de claims personnalisés au moyen d’Actions est simplifié, Actions offre beaucoup de souplesse et vous permet d’écrire du code personnalisé qui pourrait avoir des effets indésirables.

<Info>
  ### Bonne pratique

  Lorsque vous envisagez d’ajouter des claims personnalisés, nous vous recommandons de stocker toutes les données de contrôle d’accès que vous pourriez devoir inclure dans des claims dans le [`app_metadata`](/fr-CA/docs/manage-users/user-accounts/metadata) de l’utilisateur. D’abord, cela vous évite d’avoir à appeler une API externe pour récupérer ces données, ce qui peut nuire aux performances et à l’évolutivité du processus de connexion. Ensuite, `app_metadata` **ne peut pas** être modifié par un utilisateur — celui-ci ne peut donc pas contourner directement les restrictions de contrôle d’accès en modifiant ses propres métadonnées. N’oubliez pas non plus de consulter nos conseils sur les [pratiques exemplaires relatives aux métadonnées](/fr-CA/docs/get-started/architecture-scenarios/business-to-business/profile-management#metadata).
</Info>

Si vous créez différentes instances de votre application pour les organisations de vos clients, il est courant de créer un claim personnalisé dans votre jeton ID pour représenter l’organisation de l’utilisateur. Par exemple :

```js wrap lines
context.idToken["http://yourdomain.com/claims/organization"] = "organization A";
```

<div id="id-token-scopes">
  ### Scopes du jeton ID
</div>

Les [scopes OIDC](/fr-CA/docs/get-started/apis/scopes/openid-connect-scopes) sont généralement utilisés par une application pour obtenir le consentement nécessaire à l’accès autorisé aux renseignements d’un utilisateur pendant l’authentification. Chacun des scopes prédéfinis renvoie l’ensemble des claims standard lorsqu’ils sont définis, comme indiqué dans la [spécification OIDC](https://openid.net/specs/openid-connect-core-1_0.html#StandardClaims). Les scopes qu’une application demande dépendent des attributs utilisateur dont elle a besoin. Une fois les scopes demandés autorisés par l’utilisateur, les claims sont renvoyés dans le jeton ID et sont aussi accessibles par l’endpoint [/userinfo](https://auth0.com/docs/api/authentication/user-profile/get-user-info).

<div id="api-integration">
  ## Intégration d’API
</div>

Dans ce scénario, votre tenant Auth0 peut fournir un [jeton d’accès](/fr-CA/docs/secure/tokens/access-tokens) OAuth2, généralement présenté sous la forme d’un [JWT](/fr-CA/docs/secure/tokens/json-web-tokens), que votre API peut utiliser pour restreindre l’accès à certaines parties. De plus, Auth0 prend en charge ce qu’on appelle les [applications de première partie et tierces](/fr-CA/docs/get-started/applications/first-party-and-third-party-applications).

En agissant comme serveur d’autorisation, et avec le consentement de l’utilisateur (le propriétaire de la ressource), votre tenant Auth0 peut fournir un jeton d’accès — généralement présenté sous la forme d’un [JWT](/fr-CA/docs/secure/tokens/json-web-tokens) — à une application (client) afin qu’elle puisse accéder à des ressources protégées hébergées par un <Tooltip tip="Serveur de ressources : serveur hébergeant des ressources protégées. Les serveurs de ressources acceptent les requêtes de ressources protégées et y répondent." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=resource+server">serveur de ressources</Tooltip> au nom du propriétaire de la ressource. Le jeton d’accès émis est généralement transmis comme jeton Bearer dans l’en-tête HTTP Authorization envoyé à une API.

Que vous ayez une seule API ou une suite d’[API de microservices](/fr-CA/docs/get-started/apis/set-logical-api) logiquement liées, vous pouvez tirer parti des jetons d’accès fournis par Auth0 pour sécuriser l’accès à votre ou vos services. Même s’il est relativement facile de configurer cela dans l’[Auth0 Dashboard](https://manage.auth0.com/) ou au moyen de l’[Auth0 Management API](https://auth0.com/docs/api/management/v2#!/Resource_Servers/post_resource_servers), il est important d’examiner les différents scénarios d’application et architectures d’API afin de déterminer celle qui convient le mieux à votre système.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Les jetons d’accès OAuth2 sont principalement conçus pour sécuriser les API exposées publiquement; lorsqu’il est présenté sous la forme d’un JWT, un jeton d’accès est une entité autonome qui peut être vérifiée sans qu’il soit nécessaire d’effectuer un appel supplémentaire à une API tierce. Si vos API n’entrent pas dans cette catégorie, c.-à-d. qu’elles font partie d’une application elle-même (autrement dit, qu’elles ne sont appelées que par cette application) ou qu’elles se trouvent derrière votre pare-feu, les protéger au moyen de jetons pourrait donc être excessif, et votre workflow actuel fondé sur des cookies (entre autres mécanismes) pourrait très bien suffire.
</Callout>

OAuth2 a été conçu précisément en fonction de l’accès par des tiers. Par exemple, un utilisateur (propriétaire de la ressource) peut vouloir utiliser une application (un client) qui n’appartient pas à la même organisation que le service qui fournit les données de l’utilisateur (le serveur de ressources). Dans ce cas, lorsque l’application doit accéder à des données appartenant à l’utilisateur, elle redirige vers l’organisation où se trouvent les données de l’utilisateur, laquelle authentifie ensuite l’utilisateur et l’invite à accorder à l’application l’autorisation d’accéder à ses données. Cette demande d’autorisation est appelée [consentement](/fr-CA/docs/get-started/applications/third-party-applications/user-consent-and-third-party-applications) et constitue une grande partie de ce qu’implique la prise en charge des [applications tierces](/fr-CA/docs/get-started/apis/scopes/api-scopes#example-an-api-called-by-a-third-party-application). Si vous prévoyez intégrer des applications tierces, il est important de les marquer comme tierces dès le départ afin qu’Auth0 gère la demande de consentement de l’utilisateur.

D’autre part, si votre organisation possède la ou les applications, les données utilisateur elles-mêmes et la ou les API par lesquelles ces données sont consultées, le consentement n’est généralement pas requis puisque toutes les interactions sont [de première partie](/fr-CA/docs/get-started/apis/scopes/api-scopes#example-an-api-called-by-a-first-party-application). Si vous ne créez que des applications de première partie, vous pouvez éviter de présenter à vos utilisateurs des écrans de consentement inutiles en [permettant d’ignorer le consentement de l’utilisateur](/fr-CA/docs/get-started/apis) dans toute définition de service de ressources.

<Warning>
  Bien que vous puissiez configurer vos applications comme étant de première partie puis configurer vos API pour permettre aux clients de première partie d’ignorer le consentement, si vous utilisez `localhost`, Auth0 ne peut pas vérifier que l’application est réellement une application de première partie; vos utilisateurs seront donc quand même invités à donner leur consentement. Pour contourner cette contrainte, lorsque vous faites des tests sur votre machine locale pendant le développement, créez un [faux nom d’hôte local et utilisez-le à la place](https://community.auth0.com/t/how-do-i-skip-the-consent-page-for-my-api-authorization-flow/6035).
</Warning>

Autrement, vous pouvez avoir des données liées à un utilisateur pour lesquelles des [fonctionnalités supplémentaires sont offertes](/fr-CA/docs/get-started/apis/scopes/api-scopes#example-an-api-called-by-a-back-end-service) et pour lesquelles il est impossible d’obtenir le consentement explicite de l’utilisateur (c.-à-d. qu’il n’y a aucun utilisateur authentifié pouvant le fournir). Dans ce cas, une [liste d’applications pour lesquelles le grant Client Credentials est activé](/fr-CA/docs/get-started/authentication-and-authorization-flow/client-credentials-flow) peut être définie.

<div id="access-token-claims">
  ### Claims du jeton d’accès
</div>

Comme pour les jetons ID, vous pouvez [ajouter des claims personnalisés aux jetons d’accès](/fr-CA/docs/get-started/apis/scopes/sample-use-cases-scopes-and-claims#add-custom-claims-to-a-token) à l’aide de la fonctionnalité d’extensibilité Auth0 Actions. Une fois ajoutés, votre API peut ensuite vérifier qu’un jeton d’accès contient les claims requis et autoriser ou refuser l’accès à certaines fonctionnalités, selon le cas.

<Info>
  ### Bonnes pratiques

  Si vous envisagez d’ajouter des claims personnalisés, nous vous recommandons de stocker toutes les données de contrôle d’accès que vous devrez inclure dans les claims dans le [`app_metadata`](/fr-CA/docs/manage-users/user-accounts/metadata) de l’utilisateur. D’une part, cela vous évite d’avoir à appeler une API externe pour récupérer ces données, ce qui peut nuire aux performances et à l’évolutivité. D’autre part, `app_metadata` **ne peut pas** être modifié par un utilisateur; celui-ci ne peut donc pas contourner directement les restrictions de contrôle d’accès en modifiant ses propres métadonnées. N’oubliez pas non plus de consulter nos [bonnes pratiques relatives aux métadonnées](/fr-CA/docs/get-started/architecture-scenarios/business-to-business/profile-management#metadata).
</Info>

<div id="access-token-scopes">
  ### Scopes du jeton d’accès
</div>

Les [scopes OAuth2](/fr-CA/docs/get-started/apis/scopes/api-scopes) sont généralement utilisées comme mécanisme permettant à une API de déterminer quelles actions peuvent être effectuées au nom d’un utilisateur. Des scopes peuvent être ajoutées pour chaque API afin de [définir des permissions d’accès précises](/fr-CA/docs/get-started/apis/add-api-permissions) dans le <Tooltip tip="Auth0 Dashboard : le principal produit d’Auth0 pour configurer vos services." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip> ou au moyen de l’Auth0 <Tooltip tip="Auth0 Dashboard : le principal produit d’Auth0 pour configurer vos services." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=Management+API">Management API</Tooltip>). Les scopes peuvent aussi être modifiées au moyen de l’extensibilité d’Auth0 (par ex., au moyen d’une Action, comme dans cet [exemple](/fr-CA/docs/get-started/architecture-scenarios/spa-api/part-2#create-a-rule-to-validate-token-scopes)). Les scopes qu’une application demande pour accéder à une API devraient dépendre des fonctionnalités pour lesquelles l’utilisateur doit autoriser l’application à les utiliser. Une fois les scopes demandées autorisées, elles seront renvoyées dans le jeton d’accès, qui pourra ensuite être [validé par l’API en question](/fr-CA/docs/secure/tokens/access-tokens/validate-access-tokens). Un bon exemple est lorsque vous vous connectez à une application qui utilise un fournisseur d’identité sociale pour la connexion : l’API de ce fournisseur exige que l’application précise si l’utilisateur veut qu’elle publie des éléments en son nom. Cela permet à l’utilisateur d’accepter ou de rejeter cette requête. Cet exemple montre comment l’utilisateur délègue une permission à l’application, ce qui diffère d’une API qui restreint l’accès selon le rôle d’un utilisateur, et doit être géré différemment.

<Info>
  ### Bonne pratique

  Même si vous pouvez entièrement modifier les scopes du jeton d’accès au moyen de l’extensibilité d’Auth0, par bonne pratique de sécurité, vous ne devriez retirer que les scopes qui ne sont pas autorisées et vous abstenir d’en ajouter qui n’ont pas été demandées.
</Info>

Bien que les scopes soient souvent utilisées comme moyen d’appliquer des permissions d’accès à un utilisateur, il existe des situations où cela peut devenir [complexe lorsqu’on les utilise de cette façon](https://auth0.com/blog/on-the-nature-of-oauth2-scopes/). Nous vous recommandons donc d’utiliser les scopes pour l’usage auquel elles sont destinées (c.-à-d. déléguer une permission à une application) et d’utiliser des [claims personnalisées](#access-token-claims) pour vos scénarios d’accès fondés sur les rôles ou autres scénarios de contrôle d’accès.

<div id="fine-grained-authorization-fga">
  ## Autorisation granulaire (FGA)
</div>

[L’autorisation granulaire](https://docs.fga.dev/fga-concepts) vous permet d’accorder à des utilisateurs un accès à une ressource ou à un objet précis en fonction des éléments suivants :

* le rôle d’un utilisateur au sein d’une organisation, comme `editor` ou `admin`
* un attribut de l’utilisateur ou de l’objet, comme `manager` pour un utilisateur ou `marketing` pour un objet
* une relation entre un utilisateur et un objet; par exemple, un utilisateur qui a un accès en consultation à un dossier parent a aussi un accès en consultation au dossier enfant

Avec <Tooltip tip="Autorisation granulaire (FGA) : produit Auth0 permettant à des utilisateurs d’accéder à des objets ou à des ressources précis." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=FGA">FGA</Tooltip>, vous pouvez créer un modèle d’autorisation pour définir la relation à utiliser afin de déterminer l’accès des utilisateurs.

<div id="role-based-access-control-rbac">
  ## Contrôle d’accès basé sur les rôles (RBAC)
</div>

Auth0 prend en charge nativement le contrôle d’accès basé sur les rôles ([RBAC](/fr-CA/docs/manage-users/access-control/rbac)). Le RBAC consiste à attribuer des permissions aux utilisateurs en fonction de leur rôle au sein d’une organisation et simplifie le contrôle d’accès en proposant une approche plus facile à administrer et moins sujette aux erreurs.

La fonctionnalité de base du RBAC peut être utilisée dans de nombreux scénarios impliquant plusieurs organisations. Consultez [les données d’organisation dans un jeton d’accès](#organization-data-in-an-access-token) pour en savoir plus sur la façon de vous assurer que votre configuration répond à vos besoins en matière de RBAC.

<div id="machine-to-machine-m2m-authorization">
  ## Autorisation machine à machine (M2M)
</div>

De nombreux scénarios exigent qu’une application sans session utilisateur interactive obtienne un jeton d’accès afin d’appeler une API. Dans ces cas, vous devez authentifier le client plutôt que l’utilisateur, et <Tooltip tip="OAuth 2.0 : Cadre d’autorisation qui définit les protocoles et workflows d’autorisation." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=OAuth">OAuth</Tooltip> 2 fournit le type d’octroi [client credentials](/fr-CA/docs/get-started/authentication-and-authorization-flow/client-credentials-flow), ce qui facilite grandement les choses. Voici quelques exemples courants où cela s’applique :

* Un cron job ou un autre service qui doit communiquer avec votre API (par exemple, lorsqu’un rapport quotidien doit être généré et envoyé par courriel à un administrateur).
* Une API distincte qui prend en charge un accès privilégié (par exemple, l’API n’est pas exposée directement aux utilisateurs, mais uniquement à un backend).
* Dans certaines architectures de microservices, où certaines couches d’API doivent communiquer avec d’autres couches d’API sans intervention d’un utilisateur, ou après l’expiration du jeton d’un utilisateur.
* Une API privilégiée qui peut devoir être appelée avant qu’un utilisateur soit authentifié (c.-à-d. à partir d’une Action ou d’un script de BD personnalisé dans votre tenant Auth0)

<Info>
  ### bonne pratique

  Traditionnellement, on créait un « compte de service » spécial pour répondre à ces scénarios : un utilisateur doté d’un nom d’utilisateur et d’un mot de passe, configuré pour des services prenant en charge des cas d’utilisation non interactifs. Cette approche n’est toutefois plus recommandée pour de nombreuses raisons, et la bonne pratique actuelle consiste à utiliser [OAuth 2.0 Client Credentials Grant](/fr-CA/docs/get-started/authentication-and-authorization-flow/client-credentials-flow) dans ces situations.
</Info>

<div id="organization-data-in-an-access-token">
  ## Données d’organisation dans un jeton d’accès
</div>

Si votre système comprend une API distincte de votre application pour prendre en charge une application multi-organisation, il est important de limiter les opérations à l’organisation pour laquelle le jeton a été généré. Pour ce faire, le jeton d’accès doit contenir une forme d’information indiquant à l’API pour quelle organisation il a été émis. Cela peut se faire de différentes façons, selon les réponses à quelques questions simples :

1. Les utilisateurs finaux de cette organisation pourraient-ils potentiellement appartenir à plus d’une organisation, ou chaque utilisateur final est-il limité à une organisation précise ?
2. Allez-vous autoriser l’accès machine à machine (M2M) à votre API ?
3. Si vous autorisez l’accès machine à machine (M2M) à votre API, aurez-vous des développeurs qui auront besoin d’un seul client ID et secret pour accéder à plusieurs organisations (mais pas à toutes) ?
4. Allez-vous autoriser la création d’applications tierces nécessitant le consentement ?

Si les utilisateurs finaux sont limités à une seule organisation **et** que vous n’autoriserez pas l’accès M2M à votre API, ou que vous aurez un <Tooltip tip="Client ID : valeur d’identification donnée à votre ressource enregistrée par Auth0." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=client+ID">client ID</Tooltip>/secret distinct pour chaque organisation qui a besoin d’y accéder **et** que vous n’autoriserez pas les applications tierces nécessitant le consentement, l’approche la plus simple consiste à créer un custom claim dans le jeton d’accès [à l’aide d’Actions pour les jetons basés sur l’utilisateur](#access-token-claims) et [à l’aide du hook client credentials pour les appels M2M](#machine-to-machine-m2m-authorization). Vous pouvez stocker le nom de l’organisation dans les client metadata et l’extraire à partir d’Actions ou de hooks afin de l’inclure dans `access_token` comme custom claim. RBAC fonctionnera également d’emblée avec cette approche, tant que chaque utilisateur final ne peut appartenir qu’à une seule organisation.

Si les utilisateurs finaux peuvent appartenir à plus d’une organisation, ou si vous pourriez donner à un même développeur un client ID et secret pour des appels M2M vers plus d’une organisation, la meilleure approche consiste à créer une <Tooltip tip="Audience : identifiant unique de l’audience d’un jeton émis. Nommée aud dans un jeton, sa valeur contient l’ID soit d’une application (Client ID) pour un jeton ID, soit d’une API (API Identifier) pour un jeton d’accès." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=audience">audience</Tooltip> distincte (une instance d’API distincte dans votre Auth0 tenant) pour chaque organisation. Cela vous donne quelques avantages intéressants :

1. D’abord, cela vous permet de transmettre l’audience à Auth0 comme paramètre de première classe, sans avoir à créer un paramètre personnalisé. L’avantage est qu’Auth0 vous aidera à en imposer la présence et la transmettra à vos Actions. Cela garantira aussi qu’un refresh token émis ne fonctionnera que pour l’audience précise pour laquelle il a été initialement émis.
2. Cela vous permet de restreindre les client grants à des organisations précises, d’emblée. L’autre option consiste à créer un hook client credentials plus complexe pour tenter de récupérer ces restrictions ailleurs, tout en exigeant également une méthode beaucoup plus complexe et potentiellement problématique pour indiquer à l’appel client credentials pour quelle organisation émettre le jeton d’accès.
3. Cela vous permet aussi d’utiliser la fonctionnalité RBAC de base avec Auth0 et de vous assurer que les utilisateurs finaux qui ont accès à plus d’une organisation peuvent avoir un rôle différent pour chaque organisation.
