Déterminer dès le départ comment les utilisateurs s’inscriront est important, et les décisions que vous prenez ici influeront sur bon nombre de celles que vous devrez prendre par la suite. Nous avons constaté qu’il existe un ensemble de modèles typiques pour la façon dont les utilisateurs sont ajoutés à votre système, ainsi que certains points à prendre en compte dans la conception du workflow.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Bien qu’Auth0 prenne en charge de nombreux workflows, les workflows Web qui utilisent [Universal Login](/fr-CA/docs/authenticate/login/auth0-universal-login) pour l’inscription sont considérés comme une pratique exemplaire, tant dans l’industrie que chez Auth0, puisqu’ils offrent une fonctionnalité optimale et le meilleur niveau de sécurité.
</Callout>

Auth0 prend en charge l’inscription des utilisateurs au moyen de différents [fournisseurs d’identité](/fr-CA/docs/authenticate/identity-providers). Pendant l’inscription, Auth0 provisionne le profil utilisateur afin qu’il contienne les renseignements du compte de l’utilisateur. Il y a plusieurs éléments à prendre en compte en ce qui concerne la fonctionnalité et le workflow :

* Devriez-vous utiliser Auth0 comme référentiel d’identités ?
* Pouvez-vous utiliser votre propre référentiel d’identités (hérité) avec Auth0 ?
* Comment migrer les identités utilisateur de votre référentiel d’identités vers Auth0 ?
* Vos utilisateurs peuvent-ils s’inscrire au moyen de leurs comptes sociaux existants, comme Google et Facebook ?

Auth0 fournit un stockage d’identités prêt à l’emploi qui peut être utilisé pour stocker les informations d’authentification des utilisateurs de façon sûre et sécurisée. Consultez [Self Sign Up](#self-sign-up) pour en savoir plus. Si vous avez déjà un référentiel d’identités hérité et souhaitez en transférer la gestion, les fonctionnalités de [User Migration](#user-migration) vous offrent plusieurs options pour le faire.

Sinon, si vous devez conserver votre référentiel d’identités hérité — par exemple parce que vous avez des applications que vous n’êtes pas prêt à migrer ou qui ne peuvent pas l’être — vous pouvez utiliser la fonctionnalité de [proxy du référentiel d’identités](#identity-store-proxy). Permettre à vos clients d’utiliser leur propre identité est également une option intéressante et, même si nous constatons que nos clients ne le font généralement pas au départ, vous pouvez utiliser la fonctionnalité de [Social Sign Up](#social-sign-up) pour l’offrir.

<div id="user-migration">
  ## Migration des utilisateurs
</div>

En plus d’héberger le [profil utilisateur](/fr-CA/docs/get-started/architecture-scenarios/business-to-consumer/profile-management), Auth0 peut aussi à la fois servir de [proxy](#identity-store-proxy) pour votre ancien référentiel d’identités et fournir une solution de remplacement sécurisée hébergée par Auth0. Ces deux capacités sont prises en charge grâce aux [Database Connections](/fr-CA/docs/authenticate/database-connections) d’Auth0. Si vous décidez d’utiliser Auth0 comme solution de remplacement de votre ancien référentiel d’identités, vous pouvez [migrer les utilisateurs](/fr-CA/docs/manage-users/user-migration) soit en une seule fois au moyen d’une migration en bloc, soit progressivement avec une migration automatique.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Les clients optent souvent pour une approche de migration des utilisateurs en deux étapes : ils utilisent d’abord la migration automatique pour migrer le plus d’utilisateurs possible, puis la migration en bloc pour ceux qui restent. Consultez [Scénarios de migration des utilisateurs](/fr-CA/docs/manage-users/user-migration/user-migration-scenarios) pour en savoir plus.
</Callout>

La migration automatique est à privilégier, car elle permet de migrer les utilisateurs individuellement et de leur permettre, dans presque tous les cas, de conserver leur mot de passe actuel. Pour la migration en bloc, nous recommandons d’utiliser la [Management API](https://auth0.com/docs/api/management/v2#!/Jobs/post_users_imports) plutôt que l’[extension User Import/Export](/fr-CA/docs/manage-users/user-migration/user-import-export-extension) dans tous les cas sauf les plus simples, car la <Tooltip tip="Management API : un produit qui permet aux clients d’effectuer des tâches administratives." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=Management+API">Management API</Tooltip> offre plus de flexibilité et de contrôle.

Avec la migration en bloc, les utilisateurs doivent généralement **réinitialiser leur mot de passe une fois la migration terminée**, sauf si les mots de passe sont stockés sous forme hachée dans votre ancien référentiel d’identités au moyen de bcrypt (ou si vous pouvez les générer au format bcrypt). Dans ce cas, vous pourriez être en mesure d’utiliser la migration en bloc et de **conserver les mots de passe des utilisateurs** dans le cadre du processus, selon l’algorithme bcrypt et le nombre de rondes de salage utilisées. Consultez [Exemples de schéma d’importation en bloc pour la base de données](/fr-CA/docs/manage-users/user-migration/bulk-user-import-schema) pour en savoir plus.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Les appels à la Management API sont assujettis à la [politique de limitation du taux de requêtes d’Auth0](/fr-CA/docs/troubleshoot/customer-support/operational-policies/rate-limit-policy). Vous devez en tenir compte et, pour vous aider, Auth0 recommande généralement d’utiliser le [SDK Auth0](/fr-CA/docs/libraries) approprié à votre environnement de développement plutôt que d’appeler directement nos API.
</Callout>

<div id="identity-store-proxy">
  ### Proxy du référentiel d’identités
</div>

Les types de connexion Database d’Auth0 peuvent aussi être configurés pour servir de proxy à un référentiel d’identités existant (hérité). Si vous devez conserver des identités d’utilisateurs définies dans votre propre référentiel hérité — par exemple, si vous avez une ou plusieurs applications essentielles à l’entreprise que vous ne pouvez pas migrer vers Auth0, mais qui doivent tout de même avoir accès à ces identités — vous pouvez facilement les intégrer à Auth0. Consultez [Authentifier les utilisateurs à l’aide de votre base de données](/fr-CA/docs/authenticate/database-connections/custom-db) pour en savoir plus.

<div id="self-sign-up">
  ## Inscription libre-service
</div>

L’inscription libre-service s’appuie sur les [Database Connections](/fr-CA/docs/authenticate/database-connections) d’Auth0 pour stocker le user ID, le mot de passe et les renseignements d’identité associés au nom d’utilisateur (facultatif) recueillis auprès des nouveaux utilisateurs pendant le processus d’inscription. Les politiques de connexion de base de données régissant, par exemple, la [longueur minimale du nom d’utilisateur](/fr-CA/docs/authenticate/database-connections/require-username) ou la [robustesse et la complexité du mot de passe](/fr-CA/docs/authenticate/database-connections/password-options) peuvent être configurées dans le <Tooltip tip="Auth0 Dashboard : le principal produit d’Auth0 pour configurer vos services." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip>.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Le [Universal Login](/fr-CA/docs/authenticate/login/auth0-universal-login) d’Auth0, tout comme les widgets Auth0 tels que [Lock](https://auth0.com/lock), s’intègrent aux Database Connections pour offrir, dès le départ, des fonctionnalités complètes d’interface utilisateur pour l’inscription. Ces éléments d’interface sont entièrement réactifs et, grâce à une configuration riche et à des options de personnalisation étendues, vous pouvez déployer des fonctionnalités d’inscription libre-service ainsi que de login.
</Callout>

<div id="social-sign-up">
  ## Inscription sociale
</div>

L’inscription sociale est synonyme de connexion au moyen de l’[authentification sociale](/fr-CA/docs/get-started/architecture-scenarios/business-to-consumer/authentication) — il n’y a pas de distinction à faire ici *per se*, puisque la création du [profil](/fr-CA/docs/get-started/architecture-scenarios/business-to-consumer/profile-management) utilisateur se fait automatiquement lors de la première connexion sociale.
