title: De l’entreprise aux employés
---

Le scénario B2E (Business to Employees) concerne des applications utilisées par des employés. Il s’agit d’applications destinées à des utilisateurs qui agissent généralement au nom d’une organization, comme un employeur, une université ou un groupe dont ils sont membres, plutôt qu’en leur propre nom.

Ces applications, lorsqu’elles sont développées sur mesure par l’organization, peuvent utiliser le protocole OIDC/<Tooltip tip="OAuth 2.0 : Cadre d’autorisation qui définit les protocoles et les workflows d’autorisation." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=OAuth">OAuth</Tooltip> pour externaliser l’authentification, tandis que celles qui ont été achetées utilisent souvent le protocole <Tooltip tip="Security Assertion Markup Language (SAML) : Protocole normalisé permettant à deux parties d’échanger des renseignements d’authentification sans mot de passe." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=SAML">SAML</Tooltip>. Dans les deux cas, l’entreprise voudra généralement utiliser un type d’Enterprise connection, comme un <Tooltip tip="Identity Provider (IdP) : Service qui stocke et gère les identités numériques." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=Identity+Provider">fournisseur d’identité</Tooltip> SAML, ADFS, Google Workspace, Azure AD ou un service d’annuaire comme AD ou OpenLDAP, et plus rarement une base de données personnalisée, pour l’authentification des utilisateurs d’entreprise.

Pour une entreprise qui crée ou intègre des applications à Auth0 dans un environnement B2E, plusieurs exigences reviennent fréquemment dans ce scénario. Ce guide résume les besoins les plus courants des applications B2E et explique les fonctionnalités d’Auth0 qui permettent d’y répondre.

<div id="enterprise-providers">
  ## Fournisseurs d’identité d’entreprise
</div>

La plupart des entreprises disposent déjà d’un référentiel d’identités d’entreprise qui contient des renseignements sur tous les employés ainsi que les données du profil utilisateur. Il peut aussi contenir des renseignements sur les partenaires et les sous-traitants. Une exigence courante dans un scénario B2E consiste donc à permettre à ces utilisateurs de se connecter au moyen des [connexions d’entreprise Auth0](/fr-CA/docs/authenticate/identity-providers/enterprise-identity-providers), comme des fournisseurs SAML2, ADFS, Google Workspace, Azure AD ou un service d’annuaire d’entreprise sur site. C’est avantageux pour les utilisateurs, car cela leur évite d’avoir à créer un autre nom d’utilisateur et mot de passe pour chaque application et leur permet plutôt d’utiliser les mêmes identifiants de connexion dans toutes leurs applications d’entreprise.

C’est particulièrement intéressant du point de vue de la sécurité au sein de l’entreprise, parce que les identifiants des utilisateurs ne sont exposés qu’à la couche d’identité plutôt qu’à chaque application. De plus, cette architecture permet à l’entreprise de garder le contrôle de l’accès aux applications, puisque le fournisseur d’identité d’entreprise offre un point de coupure unique. Si un utilisateur quitte l’organisation, les administrateurs peuvent simplement désactiver son compte dans le fournisseur d’identité d’entreprise, et l’utilisateur ne peut alors plus se connecter à aucune des applications au moyen de ce fournisseur d’identité.

Auth0 facilite l’activation de la connexion au moyen d’une grande variété de fournisseurs d’identité d’entreprise en quelques étapes de configuration simples.

<div id="groups-and-roles">
  ## Groupes et rôles
</div>

Lorsque vous avez beaucoup d’utilisateurs, vous pouvez configurer des groupes et des rôles pour gérer l’accès et les privilèges. Souvent, ceux-ci sont stockés et gérés dans un service d’annuaire.

Auth0 peut récupérer des attributs utilisateur, comme les groupes et les rôles, à partir d’un service d’annuaire ou d’un fournisseur d’identité d’entreprise pendant l’authentification. Vous pouvez ensuite rendre ces attributs disponibles dans les jetons renvoyés à l’application ou au moyen de l’API Auth0 <Tooltip tip="Management API : un produit permettant aux clients d’effectuer des tâches administratives." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=Management+API">Management API</Tooltip>.

<div id="profile-translation">
  ## Traduction du profil
</div>

Il arrive qu’un annuaire ou un fournisseur d’identité renvoie des attributs dans un certain format, alors que votre application en utilise un autre. À l’aide des [Rules](/fr-CA/docs/manage-users/user-accounts/metadata/manage-metadata-rules) d’Auth0, vous pouvez [mapper et traduire les attributs du profil utilisateur](https://auth0.com/rules/saml-attribute-mapping). Vous pouvez même faire la traduction entre OIDC/OAuth, SAML, <Tooltip tip="Web Service Federation (WS-Fed) : protocole de gestion des identités d’utilisateurs entre domaines." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=WS-Fed">WS-Fed</Tooltip> et LDAP.

Par exemple, vous récupérez des attributs au format d’assertion SAML depuis un fournisseur d’identité SAML. Avec une règle, vous pouvez ensuite convertir ces attributs en claims personnalisés dans un <Tooltip tip="ID Token : justificatif destiné au client lui-même, plutôt qu’à l’accès à une ressource." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=ID+Token">ID Token</Tooltip> pour une application OIDC/OAuth.

Vous pouvez aussi mapper des attributs SAML vers le profil utilisateur Auth0 depuis le dashboard. Pour ce faire, accédez à [Connections &gt; Enterprise &gt; SAMLP Identity Provider](https://manage.auth0.com/#/connections/enterprise), sélectionnez votre connexion SAML, puis définissez vos mappages d’attributs dans l’onglet **Mappings**.

<div id="extensibility-with-augmented-user-profiles">
  ## Extensibilité avec des profils utilisateur enrichis
</div>

Vous souhaiterez peut-être enrichir les profils utilisateur avec des attributs ou des données provenant d’autres services. Par exemple, vous pourriez recevoir une adresse ou un numéro de téléphone et vouloir l’associer à une région géographique. Les [Rules Auth0](/fr-CA/docs/customize/rules) vous permettent d’écrire de petits extraits de code qui s’exécutent pendant la transaction d’authentification. Vous pouvez ainsi exécuter une logique ou appeler d’autres services pour obtenir des renseignements sur l’utilisateur, puis ajouter des [métadonnées utilisateur](/fr-CA/docs/manage-users/user-accounts/metadata) au profil utilisateur Auth0 et, au besoin, aux jetons envoyés à vos applications.

<div id="single-sign-on">
  ## Authentification unique
</div>

Si vous avez plusieurs applications internes, vous pouvez configurer l’[authentification unique (SSO)](/fr-CA/docs/authenticate/single-sign-on) entre elles afin que les utilisateurs n’aient à se connecter qu’une seule fois.

Auth0 prend en charge l’intégration avec des applications qui externalisent l’authentification à l’aide de protocoles d’identité standard de l’industrie :

* OIDC/OAuth
* SAML2
* WS-Fed

Après quelques étapes de configuration, toutes vos applications peuvent tirer parti de votre fournisseur d’identité d’entreprise. Dans cette configuration, Auth0 sert d’intermédiaire entre vos applications et les fournisseurs d’identité d’entreprise.

Ainsi, lorsqu’un utilisateur se connecte à une application, il peut accéder aux autres applications intégrées à Auth0 sans avoir à se reconnecter. Ce sera le cas jusqu’à l’expiration de sa session <Tooltip tip="Authentification unique (SSO) : service qui, après qu’un utilisateur s’est connecté à une application, connecte automatiquement cet utilisateur aux autres applications." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=SSO">SSO</Tooltip>. Vous devriez configurer la durée de la session SSO dans Auth0 afin de respecter vos politiques de sécurité.

<div id="single-sign-on-integrations">
  ## Intégrations d’authentification unique
</div>

Vous pouvez également intégrer à Auth0 des applications acquises pour l’authentification unique (SSO). Auth0 offre des [intégrations prédéfinies](/fr-CA/docs/customize/integrations/sso-integrations) pour des applications comme :

* Salesforce
* Zendesk
* Slack
* New Relic

<div id="branding">
  ## Image de marque
</div>

L’image de marque est un élément important de toute application. Votre logo, vos couleurs et vos styles devraient être cohérents dans l’ensemble de l’application. Vous pouvez [personnaliser](/fr-CA/docs/libraries/custom-signup) les pages de connexion, d’inscription et d’erreur affichées par Auth0 afin qu’elles s’harmonisent avec votre application. Ajoutez votre propre logo, votre texte et vos couleurs. La prise en charge de l’I18N/L10N est également offerte pour les déploiements à l’échelle mondiale. Les [courriels de vérification ou de réinitialisation du mot de passe](/fr-CA/docs/customize/email/email-templates) peuvent aussi être personnalisés.

Les [écrans de connexion](/fr-CA/docs/libraries/lock/lock-ui-customization) devraient sembler provenir du domaine associé à l’image de marque de votre application. Pour assurer cette cohérence, vous pouvez définir un [domaine personnalisé](/fr-CA/docs/customize/custom-domains) pour l’écran de connexion affiché par Auth0.

<div id="multi-factor-authentication">
  ## Authentification multifactorielle
</div>

Les applications internes ou destinées aux employés traitent souvent du contenu sensible. [L’authentification multifactorielle (MFA)](/fr-CA/docs/secure/multi-factor-authentication) aide à protéger vos données et vos applications. Auth0 offre plusieurs façons de mettre en place <Tooltip tip="Multi-factor authentication (MFA) : processus d’authentification qui utilise un facteur supplémentaire en plus du nom d’utilisateur et du mot de passe, comme un code envoyé par SMS." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=MFA">MFA</Tooltip>. Et pour plus de souplesse, vous pouvez utiliser Rules pour l’activer seulement pour les applications ou les groupes d’utilisateurs qui en ont besoin.

<div id="logs-export">
  ## Export des logs
</div>

Vous devez analyser les logs ou les conserver à long terme ? Auth0 propose des extensions pour [exporter les logs vers des outils externes](/fr-CA/docs/deploy-monitor/logs) afin de les analyser et de les conserver. Vous pouvez aussi récupérer les données de logs avec la Management API.

<div id="audit">
  ## Audit
</div>

Les entreprises utilisent les données de journalisation à diverses fins, notamment pour les rapports d’audit. Auth0 consigne divers types de données dans les fichiers journaux, ce qui peut être utile pour vos rapports d’audit. Les journaux contiennent des renseignements sur les utilisateurs authentifiés, le fournisseur d’identité utilisé, ainsi que sur le moment où des modifications administratives importantes sont apportées dans le <Tooltip tip="Auth0 Dashboard : le principal produit d’Auth0 pour configurer vos services." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=Auth0+dashboard">Auth0 Dashboard</Tooltip>.

Chaque entrée de journal est associée à un type d’événement. Vous pouvez utiliser les types d’événements comme filtres lorsque vous interrogez les données de journalisation avec la Management API, ou lorsque vous exportez les journaux vers des outils d’analyse.

<div id="monitoring">
  ## Surveillance
</div>

La surveillance de l’infrastructure et des services dont dépendent vos applications est essentielle. Auth0 met à votre disposition une page [Auth0 Status](https://status.auth0.com/) à laquelle vous pouvez vous abonner.

Auth0 met tout en œuvre pour réduire au minimum les interruptions de service, mais toute perturbation sera affichée sur la page d’état. Pour répondre aux exigences de soutien liées à la documentation de l’analyse des causes profondes après une perturbation, Auth0 effectue une analyse interne et publie les résultats dans l’avis de perturbation une fois l’analyse terminée.

<div id="attack-protection">
  ## Protection contre les attaques
</div>

Une réalité malheureuse de la vie moderne sur Internet, ce sont les pirates informatiques. Ils cherchent constamment des moyens de s’introduire dans les applications. Par exemple, ils peuvent tenter de se connecter en utilisant des mots de passe courants. Ils peuvent aussi utiliser des identifiants volés ailleurs, en espérant que les utilisateurs aient réutilisé les mêmes mots de passe sur d’autres sites.

La [Protection contre les attaques](/fr-CA/docs/secure/attack-protection) d’Auth0 détecte ces situations pour les connexions de base de données Auth0 et offre des options quant à la façon d’y répondre. Activez <Tooltip tip="Protection contre les attaques : fonctionnalités qu’Auth0 fournit pour détecter et atténuer les attaques, y compris la protection contre les attaques par force brute, la limitation des IP suspectes, la détection des mots de passe compromis, la détection des robots et l’authentification multifacteur adaptative." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=Attack+Protection">Protection contre les attaques</Tooltip> et configurez les options de réponse afin de pouvoir réagir adéquatement si un tel événement se produit.

<div id="github-deployment">
  ## Déploiement Github
</div>

Gérez-vous une grande partie du code de votre application dans Github ? Vous pouvez y déployer du code pour les Actions, les Rules, les Hooks ou l’accès à une base de données personnalisée à l’aide de l’[extension Github Deployment d’Auth0](https://marketplace.auth0.com/integrations/github-actions).

Si vous disposez d’un pipeline complet d’intégration continue et de déploiement continu, utilisez l’outil [Auth0 Deploy CLI](https://github.com/auth0/auth0-deploy-cli) pour plus de flexibilité.
