title: Consulter l’Auth0 Agent Experience Score
description: Découvrez comment Auth0 évalue les performances des agents d’IA selon différents modèles et frameworks, y compris les dimensions, le système de notation et la méthodologie de l’Agent Experience Score.
---

L’[Agent Experience Score](https://auth0.com/agent-experience) mesure l’efficacité avec laquelle les agents d’IA intègrent Auth0 dans différents modèles et frameworks. Il vous permet de comparer les scores actuels d’agents qui intègrent les services et fonctionnalités d’Auth0 — comme l’authentification multifacteur (MFA) ou les Auth0 Actions — dans des environnements de développement et de test, afin d’évaluer dans quelle mesure les outils Auth0 améliorent les performances des agents.

Utilisez cette ressource pour en savoir plus sur la méthodologie de notation, notamment sur le calcul des scores, les dimensions mesurées et l’attribution des notes.

<div id="test-specifications">
  ## Spécifications des tests
</div>

Les agents IA — Claude Code, GitHub Copilot, Gemini CLI — exécutent des tâches d’intégration Auth0 dans des environnements de développement isolés. Chaque agent utilise les mêmes outils qu’utiliserait un développeur dans un environnement réaliste : un espace de travail, un shell et des outils comme Auth0 CLI. Les prompts sont courtes et réalistes : « ajoute l’authentification à mon application Next.js », pas des guides détaillés étape par étape.

Chaque modèle est testé avec et sans les outils d’Auth0 ([MCP Server](/fr-CA/docs/get-started/build-with-ai-tools#auth0-docs-mcp-server) et [Agent Skills](/fr-CA/docs/quickstart/agent-skills)). La différence entre ces scores représente l’impact mesurable des outils d’IA d’Auth0 sur l’expérience des développeurs.

<div id="score-dimensions">
  ## Dimensions du score
</div>

Chaque exécution est évaluée selon 7 dimensions réparties en deux catégories. Quatre dimensions portent sur le processus de l’agent de bout en bout avec les outils Auth0. Trois dimensions évaluent le résultat final. Chaque dimension reçoit individuellement un score de 0 à 100, puis ces scores sont pondérés et combinés pour former le score global.

| Dimension                     | Catégorie | Description                                                                                                                                                                             |
| ----------------------------- | --------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Friction de configuration** | Processus | Score déterminé par la capacité de l’agent à accomplir la tâche de manière autonome. Si l’agent a dû s’arrêter pour poser des questions ou a rencontré des erreurs, le score diminuait. |
| **Vitesse de configuration**  | Processus | Score déterminé par le temps d’exécution actif de l’agent. Les résultats sont comparables d’un environnement à l’autre.                                                                 |
| **Efficacité**                | Processus | Score déterminé par le nombre d’appels aux outils nécessaires pour accomplir la tâche. Moins d’appels aux outils signifie moins de coûts et moins de complexité.                        |
| **Reprise après erreur**      | Processus | Score déterminé par les erreurs d’infrastructure (limites de débit, délais d’expiration) qui ont perturbé l’exécution.                                                                  |
| **Exactitude**                | Résultat  | Score déterminé selon que le code généré importe de vrais packages, appelle de vraies méthodes et connecte correctement les composants.                                                 |
| **Hallucination**             | Résultat  | Score déterminé selon que l’agent a inventé des packages inexistants ou utilisé des variantes de SDK incorrectes.                                                                       |
| **Sécurité**                  | Résultat  | Score déterminé selon que l’agent a codé en dur des secrets, stocké des jetons de façon non sécurisée ou inclus des identifiants d’authentification dans le code source.                |

<div id="grades">
  ## Notes
</div>

Les scores globaux correspondent aux notes alphabétiques :

| Note | Score min. | Description                                         |
| ---- | ---------- | --------------------------------------------------- |
| A    | 90         | Prêt pour la mise en production. Problèmes minimes. |
| B    | 75         | Solide, mais comporte quelques lacunes à corriger.  |
| C    | 60         | Utilisable, mais nécessite un peu de nettoyage.     |
| D    | 40         | Problèmes importants.                               |
| F    | &lt; 40    | Inutilisable — mieux vaut repartir de zéro.         |

Les notes sont calibrées pour correspondre à l&#39;intuition des développeurs. Un score de 91 devrait donner l&#39;impression d&#39;un code que vous accepteriez avec très peu de révision. Un score de 55 devrait donner l&#39;impression de quelque chose qui demande un vrai travail de correction.

<div id="result-validation">
  ## Validation des résultats
</div>

Chaque évaluateur vérifie le code généré — pas la prose ni les explications. Les évaluateurs s’assurent que le code compile, importe de vrais packages, appelle de véritables méthodes du SDK et n’introduit pas de vulnérabilités de sécurité.

Les résultats sont validés à plusieurs niveaux :

* **Vérifications de présence** : les symboles du SDK, les importations et les clés de configuration requis sont présents dans le résultat.
* **Détection d’hallucinations** : les packages inventés, les mauvaises versions de SDK et les méthodes d’API inventées sont détectés.
* **Vérifications de sécurité** : les informations d’authentification codées en dur, les tokens stockés dans un emplacement non sécurisé et les secrets dans le code source sont signalés.
* **Validation structurelle** : le code est correctement intégré — les bons composants dans les bons fichiers, les hooks du cycle de vie sont bien gérés et les intergiciels sont dans le bon ordre.
* **Exactitude des versions** : l’agent utilise les API actuelles, et non des approches dépréciées (vérifié seulement lorsque l’agent a accès à la documentation à jour).
* **Examen d’ensemble** : un juge LLM évalue l’exactitude globale de l’implémentation.

<div id="estimated-cost-and-time">
  ## Coût et temps estimés
</div>

La page des résultats affiche le coût estimé et le temps estimé pour chaque configuration. Ces valeurs correspondent à une seule exécution d’évaluation avec Auth0 MCP + Skills activés.

<div id="estimated-cost">
  ### Coût estimé
</div>

Le coût est calculé en fonction du nombre total de tokens consommés pendant l’exécution d’évaluation (tokens d’entrée + tokens de sortie), multiplié par le tarif par token publié par le fournisseur du modèle. Auth0 ne facture pas l’exécution des évaluations — ce coût correspond à ce que vous paieriez à votre fournisseur de modèle pour une utilisation équivalente de tokens.

Le prix des tokens varie selon le modèle et le fournisseur. Pour connaître les tarifs en vigueur, consultez la page de tarification de votre fournisseur :

* [Tarification d’Anthropic (Claude)](https://docs.anthropic.com/en/docs/about-claude/models#model-comparison-table)
* [Tarification d’OpenAI (GPT)](https://openai.com/api/pricing/)
* [Tarification de Google (Gemini)](https://ai.google.dev/gemini-api/docs/pricing/)

<div id="estimated-time">
  ### Temps estimé
</div>

Le temps correspond à la durée réelle de l’exécution d’évaluation, depuis l’envoi du prompt jusqu’au résultat final. Il comprend toute l’activité de l’agent : lecture de fichiers, appels d’outils, attente des réponses de l’API et écriture de code.

Le temps peut varier en fonction des éléments suivants :

* la latence de l’API du fournisseur du modèle et les limites de débit
* le nombre d’appels d’outils nécessaires (qui varie selon la complexité de la tâche)
* les conditions réseau entre l’environnement d’évaluation et le fournisseur du modèle
* la profondeur de la file d’attente et la charge du côté du fournisseur

Le temps n’est pas normalisé d’un fournisseur à l’autre. Un temps plus court reflète à la fois l’efficacité du modèle et les performances de l’infrastructure du fournisseur.
