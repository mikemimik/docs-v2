title: Intégrations Actions pour les partenaires
---

Des partenaires comme vous utilisent les intégrations Actions pour créer des solutions sans code qui intègrent leurs produits aux workflows Auth0. Les intégrations Actions permettent aux clients d’Auth0 de mettre en œuvre des solutions comme la vérification d’identité ou la gestion du consentement sans écrire une seule ligne de code. Vous créez l’intégration Actions; les clients la glissent-déposent dans leur workflow.

Les intégrations Actions sont des fonctions autonomes qui s’exécutent à des points précis de la plateforme Auth0. Écrites en JavaScript, elles sont à code source fermé; les clients ne peuvent pas en modifier le code.

<div id="create-and-publish-an-actions-integration">
  ## Créer et publier une intégration Actions
</div>

La création d’une intégration Actions comprend la planification, le développement et les tests. Avant de la proposer aux clients dans l’[Auth0 Marketplace](https://marketplace.auth0.com/), Auth0 examine l’intégration Actions. Pour en savoir plus sur l’Auth0 Marketplace, consultez [Partenaires Marketplace](/docs/fr-CA/customize/integrations/marketplace-partners).

<div id="step-1-determine-your-use-case">
  ### Étape 1. Déterminez votre cas d’utilisation
</div>

Déterminez le cas d’utilisation auquel votre intégration Actions répondra. Quel problème résoudra-t-elle pour les clients? Comment y répondra-t-elle? Pour en savoir plus, consultez [Définir un cas d’utilisation d’intégration](/docs/fr-CA/customize/integrations/marketplace-partners/defining-an-integration-use-case).

<div id="step-2-choose-a-flow">
  ### Étape 2. Choisissez un flux
</div>

Les intégrations Actions se composent d’une seule [Action](/docs/fr-CA/customize/actions). Ce qu’une Action peut faire dépend de l’endroit où elle s’exécute dans l’environnement d’exécution d’Auth0, c’est-à-dire de son [flux](/docs/fr-CA/customize/actions/explore-triggers) d’Action. Les flux d’Action offerts sont les suivants :

* Le **flux Login** s’exécute de façon synchrone après l’inscription ou la connexion réussie d’un utilisateur. Les événements de connexion comprennent des flux interactifs (comme nom d’utilisateur/mot de passe ou la connexion fédérée) et des flux non interactifs (comme les vérifications de session et les échanges de <Tooltip tip="Jeton d’actualisation : jeton utilisé pour obtenir un nouveau jeton d’accès sans obliger les utilisateurs à se reconnecter." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=refresh+token">jeton d’actualisation</Tooltip>). Pour en savoir plus sur ce flux, consultez [Login Flow](/docs/fr-CA/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger).
* Le **flux Machine to Machine** s’exécute de façon synchrone lorsqu’une application demande un <Tooltip tip="Jeton d’accès : information d’identification d’autorisation, sous la forme d’une chaîne opaque ou d’un JWT, utilisée pour accéder à une API." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=access+token">jeton d’accès</Tooltip> au moyen du grant client credentials. Pour en savoir plus sur ce flux, consultez [Machine to Machine Flow](/docs/fr-CA/customize/actions/explore-triggers/machine-to-machine-trigger).
* Le **flux Pre User Registration** s’exécute de façon synchrone avant qu’Auth0 ajoute un utilisateur à une connexion de base de données ou <Tooltip tip="Sans mot de passe : forme d’authentification qui ne repose pas sur un mot de passe comme premier facteur." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=passwordless">sans mot de passe</Tooltip>. Ce flux ne s’exécute pas pour les connexions fédérées, comme les connexions d’entreprise ou sociales. Pour en savoir plus sur ce flux, consultez [Pre User Registration Flow](/docs/fr-CA/customize/actions/explore-triggers/signup-and-login-triggers/pre-user-registration-trigger).
* Le **flux Post User Registration** s’exécute de façon asynchrone après qu’Auth0 ajoute un utilisateur à une connexion de base de données ou sans mot de passe. Ce flux ne s’exécute pas pour les connexions fédérées, comme les connexions d’entreprise ou sociales. Pour en savoir plus sur ce flux, consultez [Post User Registration Flow](/docs/fr-CA/customize/actions/explore-triggers/signup-and-login-triggers/post-user-registration-trigger).
* Le **flux Post Change Password** s’exécute de façon asynchrone après qu’un client a changé le mot de passe d’un utilisateur d’une connexion de base de données. Ce flux ne s’exécute pas pour les changements de mot de passe sur d’autres types de connexions. Pour en savoir plus sur ce flux, consultez [Post Change Password Flow](/docs/fr-CA/customize/actions/explore-triggers/password-reset-triggers/post-change-password-trigger).
* Le **flux** **Send Phone Message** envoie un message vocal ou SMS dans le cadre d’un fournisseur <Tooltip tip="Authentification multifacteur (MFA) : processus d’authentification utilisateur qui utilise un facteur en plus du nom d’utilisateur et du mot de passe, comme un code envoyé par SMS." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=MFA">MFA</Tooltip> personnalisé. Pour en savoir plus sur les fournisseurs MFA personnalisés, consultez [Customize Multi-factor Authentication SMS and Voice Messages](/docs/fr-CA/customize/customize-sms-or-voice-messages). Pour en savoir plus sur ce flux, consultez [Send Phone Message Flow](/docs/fr-CA/customize/actions/explore-triggers/mfa-notifications-trigger).

Pour en savoir plus sur les flux d’Action, consultez [Explore Flows and Triggers](/docs/fr-CA/customize/actions/explore-triggers). Si vous devez rediriger des utilisateurs vers un point de terminaison pendant la connexion, consultez [Redirect Actions for Partners](/docs/fr-CA/customize/integrations/marketplace-partners/redirect-actions-for-partners).

<div id="step-3-build-an-actions-integration">
  ## Étape 3. Créer une intégration Actions
</div>

Une fois que vous avez déterminé quel flux convient au cas d’utilisation de votre intégration Actions, créez votre Action dans un tenant de test. Pour savoir comment faire, consultez [Write Your First Action](/docs/fr-CA/customize/actions/write-your-first-action). Cette Action servira de base au code de votre intégration Actions.

De plus, pour vous aider à créer votre intégration, vous pouvez utiliser nos modèles d’intégration d’Action :

* [Modèle d’intégration d’Action Post Login](https://github.com/auth0-marketplace/TEMPLATE-action-post-login)
* [Modèle d’intégration d’Action Machine-to-Machine](https://github.com/auth0-marketplace/TEMPLATE-action-client-credentials)
* [Modèle d’intégration d’Action Pre User Registration](https://github.com/auth0-marketplace/TEMPLATE-action-pre-user-registration)
* [Modèle d’intégration d’Action Post User Registration](https://github.com/auth0-marketplace/TEMPLATE-action-post-user-registration)
* [Modèle d’intégration d’Action Post Change Password](https://github.com/auth0-marketplace/TEMPLATE-action-post-password-change)
* [Modèle d’intégration d’Action Send Phone Message](https://github.com/auth0-marketplace/TEMPLATE-action-phone-message)

Pour savoir comment rendre votre intégration Actions sécuritaire et fiable, consultez les [Directives de codage des Actions](/docs/fr-CA/customize/actions/action-coding-guidelines). Suivez aussi les directives suivantes, propres aux intégrations Actions :

* Lorsque vous créez votre Action dans un tenant de test, utilisez des secrets pour toutes les valeurs qu’un client doit fournir (comme les Identifiers, les noms de tenant et les Thresholds). Lorsque vous soumettez votre intégration Actions, Auth0 vous demande quels champs doivent être chiffrés au repos (et donc rester des secrets) et lesquels peuvent être des champs de texte modifiables.
* Utilisez un flag de configuration pour un mode de débogage/test de la journalisation afin de faciliter la configuration ou le troubleshooting. Les intégrations Actions devraient rarement produire des logs pour toutes les exécutions.
* Écrivez les noms des secrets en `ALL_CAPS_SNAKE_CASE` pour faciliter la révision.
* Les intégrations Actions sont à code source fermé; les clients ne peuvent ni voir ni modifier le code. Toute donnée propre au client requise pour contrôler l’intégration Actions doit provenir des secrets ou de la configuration.

<div id="step-4-test-your-actions-integrations">
  ### Étape 4. Testez vos intégrations Actions
</div>

Comme les réviseurs d’Auth0 n’ont pas accès à votre service, vous devez tester votre Action avant de la soumettre pour examen. Testez manuellement à la fois les exécutions réussies et les cas limites. Incluez ces vérifications :

* L’Action échoue-t-elle silencieusement ou bloque-t-elle la suite de l’exécution ?
* Comment l’Action se comporte-t-elle si une requête HTTP échoue ?
* Quels messages d’erreur, le cas échéant, sont exposés aux utilisateurs ?
* Quelles informations sont consignées dans les journaux ?
* L’Action peut-elle être contournée ou usurpée ?

Pendant le processus de soumission, Auth0 demande une couverture de tests unitaires pour le scénario de réussite. Auth0 fournit tous les outils et exemples nécessaires pour rendre cela aussi simple que possible.

<div id="step-5-submit-your-actions-integration">
  ### Étape 5. Soumettez votre intégration Actions
</div>

Une fois que vous avez rédigé et mis à l’essai minutieusement votre intégration Actions, soumettez-la à Auth0 pour examen : envoyez une [demande de service Auth0 Marketplace](https://autheco.atlassian.net/servicedesk/customer/portal/1/group/1/create/43). Après un premier examen concluant, l’intégration fait l’objet d’une vérification de conformité en matière de sécurité.

<div id="step-6-publish-your-actions-integration">
  ### Étape 6. Publiez votre intégration Actions
</div>

Une fois qu’Auth0 aura terminé l’évaluation de votre intégration Actions, nous vous informerons si elle a été approuvée ainsi que des échéances de publication.
