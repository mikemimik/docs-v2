<div id="step-4-use-multiple-roles-with-amazon-api-gateway">
  ## Étape 4 - Utiliser plusieurs rôles avec Amazon API Gateway
</div>

Dans cette étape, vous attribuerez différents rôles AWS IAM aux utilisateurs en fonction des renseignements d’authentification :

* Les utilisateurs qui s’authentifient à l’aide de connexions sociales seront considérés comme des acheteurs ;
* Les utilisateurs qui s’authentifient à l’aide de connexions de base de données seront considérés comme des administrateurs.

Vous mettrez en œuvre cette logique d’attribution des rôles de deux façons différentes :

* JavaScript ;
* règles Auth0.

Pour de nombreuses applications Auth0, vous voudrez que les utilisateurs aient des niveaux d’accès différents et pouvoir exploiter des renseignements supplémentaires sur une identité donnée dans la logique de votre service. Lorsqu’il suffit de restreindre l’accès au niveau de l’API, vous pouvez utiliser différents rôles AWS IAM (par exemple, les administrateurs peuvent utiliser la fonction de mise à jour pour ajouter ou supprimer des animaux, tandis que les utilisateurs des connexions sociales peuvent seulement en acheter).

Le schéma suivant illustre l’attribution de rôles AWS IAM à deux catégories d’utilisateurs : les utilisateurs authentifiés au moyen de connexions sociales et les utilisateurs authentifiés au moyen de connexions de base de données. Il montre également que des rôles AWS IAM peuvent être attribués à d’autres entités, comme des fonctions AWS Lambda, afin de contrôler les permissions accordées à ces entités pour un compte. Bref, un rôle IAM est un ensemble de permissions liées aux capacités d’AWS, défini par une ou plusieurs politiques, puis attribué à une entité.

Lorsque vous souhaitez prendre des décisions dans votre code (par exemple, effectuer une vérification de solvabilité pour un utilisateur qui achète un animal), vous voudrez aussi transmettre l’identité. Cela sera démontré ci-dessous dans [Étape 5 - Utiliser des jetons d’identité pour transmettre l’identité](/fr-CA/docs/customize/integrations/aws/aws-api-gateway-delegation/aws-api-gateway-delegation-5).

<div id="step-1-create-the-petpurchase-api-resource">
  ### Étape 1. Créer la ressource API PetPurchase
</div>

Dans la [Amazon API Gateway Console](https://console.aws.amazon.com/apigateway), sélectionnez votre API Pets. Vous serez redirigé vers la page Resources.

Cliquez sur **Actions**, puis sur **Create Resource**. Nommez la nouvelle ressource enfant `Purchase`. Cliquez sur **Create Resource**.

Ajoutez une méthode OPTIONS pour la ressource `purchase`, comme indiqué précédemment pour `pets` dans la [section Set Up CORS and Deploy the API de l’étape 2 - Securing and Deploying the Amazon API Gateway](/fr-CA/docs/customize/integrations/aws/aws-api-gateway-delegation/aws-api-gateway-delegation-2#set-up-cors-and-deploy-the-api).

Créez une nouvelle fonction AWS Lambda pour l’achat d’un animal de compagnie, appelée `PetPurchase`, qui ajoute les attributs `isSold` et `soldTo` à un animal de compagnie, comme suit :

```javascript lines expandable
var AWS = require('aws-sdk');
var DOC = require('dynamodb-doc');
var dynamo = new DOC.DynamoDB();

exports.handler = function(event, context) {
   var petId = event.petId;
   var user = event.userName;
   var pets = {};
   console.log('start PetsPurchase, petId', petId, ' userName', user);

   var writecb = function(err, data) {
      if(!err) {
          context.done(null, pets);
      } else {
          console.log('error on GetPetsInfo: ',err);
          context.done('failed on update', null);
      }
   };

   var readcb = function(err, data) {
      if(err) {
          console.log('error on GetPetsInfo: ',err);
          context.done('failed to retrieve pet information', null);
      } else {
          // vérifier que nous avons des animaux
          if(data.Item && data.Item.pets) {
              pets = data.Item.pets;
              var found = false;

              for(var i = 0; i < pets.length && !found; i++) {
                  if(pets[i].id === petId) {
                     if(!pets[i].isSold) {
                        pets[i].isSold = true;
                        pets[i].soldTo = user;
                        var item = { username:"default",pets: pets};
                        dynamo.putItem({TableName:"Pets", Item:item}, writecb);
                        found = true;
                     }
                  }
               }
               if(!found) {
                 console.log('pet not found');
                 context.done('That pet is not available.', null);
               }
           } else {
              console.log('pet already sold');
              context.done('That pet is not available.', null);
           }
       }
   };

   dynamo.getItem({TableName:"Pets", Key:{username:"default"}}, readcb);
};
```

Une fois que vous avez défini la fonction Lambda, [ajoutez une méthode POST à la ressource `purchase`](/fr-CA/docs/customize/integrations/aws/aws-api-gateway-delegation/aws-api-gateway-delegation-1#method-post-pet-information) qui appelle la fonction Lambda `PetPurchase`. Assurez-vous également d’ajouter l’en-tête `Access-Control-Allow-Origin` avec la valeur `*` à la méthode POST à l’aide de la configuration de réponse de méthode et de réponse d’intégration décrite dans la [section Set Up Cors and Deploy the API de l’étape 2 - Securing and Deploying the Amazon API Gateway](/fr-CA/docs/customize/integrations/aws/aws-api-gateway-delegation/aws-api-gateway-delegation-2#set-up-cors-and-deploy-the-api).

Testez la méthode d’API Gateway en fournissant le message d’entrée suivant :

```json lines
{
    "petId": 1,
    "userName": "fred flintstone"
 }
```

Dans la réponse de test, vous devriez voir que l’animal ayant l’ID 1 a maintenant été vendu à Fred Flintstone :

```json lines
[
  {
    "id": 1,
    "price": 249.99,
    "type": "dog",
    "isSold": true,
    "soldTo": "fred flintstone"
  },

  ...
```

<div id="step-2-use-iam-to-secure-the-purchasepet-api">
  ### Étape 2. Utilisez IAM pour sécuriser l’API PurchasePet
</div>

<div id="update-iam">
  #### Mettre à jour IAM
</div>

Pour sécuriser votre API, suivez le même processus que celui utilisé pour ajouter un nouveau rôle dans la [partie 2](/fr-CA/docs/customize/integrations/aws/aws-api-gateway-delegation/aws-api-gateway-delegation-2) de ce tutoriel. Nommez le nouveau rôle `auth0-api-social-role`.

L’ARN de la méthode que vous allez sécuriser dans la stratégie IAM devrait ressembler à ceci :

```bash wrap lines
arn:aws:execute-api:us-east-1:your-accountid:your-api-id/*/pets/purchase
```

Assurez-vous également de mettre à jour la politique de confiance.

Accédez à la [Amazon API Gateway Console](https://console.aws.amazon.com/apigateway), puis sélectionnez la méthode POST pour la ressource `/pets/purchase`. Sélectionnez **Method Request** et remplacez **Authorization Type** par AWS&#95;IAM. Cliquez sur la coche pour enregistrer ce paramètre.

À ce stade, vous avez défini deux rôles que vous pouvez utiliser avec API Gateway :

* `auth0-api-role` : permet de mettre à jour les animaux de compagnie
* `auth0-api-social-role` : permet d’acheter un animal de compagnie

<div id="configure-login-with-amazon-and-update-auth0">
  #### Configurer Login with Amazon et mettre à jour Auth0
</div>

Vous pouvez créer un rôle social à l’aide de Login with Amazon (LWA).

Bien que ce tutoriel comprenne des instructions sur l’utilisation de Login with Amazon, veuillez noter que vous pouvez aussi utiliser d’autres fournisseurs sociaux.

1. Accédez à [Auth0 Dashboard &gt; Authentication &gt; Social](https://manage.auth0.com/#/connections/social), puis sélectionnez **Create Connection**.
2. Choisissez la connexion que vous souhaitez configurer, puis donnez votre consentement.
3. Copiez-collez le `Client ID` et le `Client Secret` de votre fournisseur d’identité sociale, sélectionnez les **Attributes** (et les **Permissions**, le cas échéant), puis cliquez sur **Save**.
4. Sélectionnez la vue **Applications**, activez le commutateur pour chacune de vos applications Auth0 qui doivent pouvoir utiliser cette connexion, puis sélectionnez **Save**.

Une fois les renseignements appropriés saisis, sélectionnez **Try Connection** pour vous assurer que tout est configuré correctement.

Lorsque vous configurez LWA à l’aide de la console Amazon, assurez-vous d’entrer, dans Allowed Return URLs, l’URL de rappel de votre application Auth0, qui devrait ressembler à `https://johndoe.auth0.com/login/callback`. La page d’aide d’Auth0 vous indiquera précisément quoi saisir.

Accédez à [Auth0 Dashboard &gt; Applications &gt; Applications](https://manage.auth0.com/#/applications), puis sélectionnez votre Application pour afficher ses paramètres. Sélectionnez la vue **Connections**, repérez la section **Social** et assurez-vous que **Amazon** est activé.

<div id="update-the-login-controller-logic-to-choose-different-roles-for-different-types-of-users">
  ##### Mettre à jour la logique du contrôleur Login pour choisir différents rôles selon le type d’utilisateur
</div>

La logique du contrôleur de connexion utilise `getOptionsForRole` pour sélectionner différents rôles selon l’utilisateur. Lorsque vous obtenez le jeton de délégation, vous pouvez indiquer à Auth0 quel rôle utiliser (c’est-à-dire si l’utilisateur est administrateur ou non).

Dans le fichier `pets/login/login.js`, modifiez les valeurs `role` et `principal` de l’utilisateur non administrateur pour utiliser le rôle IAM de l’utilisateur social que vous venez de créer.

À ce stade, vous devriez pouvoir vous connecter à l’aide d’identifiants Amazon **ou** de l’utilisateur de base de données que vous avez créé précédemment. Remarquez que l’interface utilisateur permet à un utilisateur social d’acheter des animaux de compagnie, tandis qu’un utilisateur administrateur peut en ajouter et en supprimer.

Pour tester cette fonctionnalité, vous pouvez masquer temporairement le bouton de suppression dans l’interface utilisateur en retirant `ng-show="isAdmin"` dans `/pets/home/home.html` :

```html wrap lines
<button ng-show="isAdmin" class="btn delete-btn" ng-click="removePet(pet.id)">remove</button>
```

Après avoir copié les modifications dans votre bucket S3, essayez de supprimer un animal de compagnie tout en étant connecté comme utilisateur social.

Dans `home.js`, modifiez la fonction `buyPet` pour permettre l’achat d’animaux de compagnie :

```javascript lines
        $scope.$apply();
      }).catch(function (response) {
        alert('buy pets failed');
        showError(response);
    });
}
…
```

Copiez le code dans votre bucket S3, déconnectez-vous, puis reconnectez-vous comme utilisateur social en cliquant sur l’icône Amazon dans la boîte de dialogue de connexion Lock. Vous devrez peut-être cliquer sur **SHOW ALL** si votre connexion précédente est toujours affichée dans le panneau Lock.

Notez qu’en tant qu’utilisateur Amazon, vous pouvez acheter un animal, mais pas en ajouter ni en supprimer. Cependant, si vous vous connectez avec un utilisateur associé à une connexion de base de données, vous pouvez ajouter et supprimer des animaux, mais pas en acheter.

<div id="enforce-role-assignment-with-auth0-rules">
  ### Renforcer l’attribution des rôles avec Auth0 Rules
</div>

Dans certains cas, vous pouvez déterminer le rôle approprié à l’aide de l’Application (comme illustré ici), mais pour des raisons de sécurité (par exemple, pour empêcher l’utilisateur d’assumer un rôle plus privilégié que nécessaire), il peut être préférable de déterminer les privilèges de l’utilisateur côté serveur.

Avec Auth0, cela se fait au moyen de rules, c’est-à-dire des règles de logique de service que vous définissez, puis qui sont exécutées pendant le processus d’authentification Auth0. Par exemple, vous pourriez créer des rules pour :

* Éviter de transmettre les informations de rôle du navigateur à l’Application;
* Insérer les informations de rôle dans la requête de délégation selon la source d’authentification.

<div id="enforce-role-assignment">
  #### Appliquer l’attribution des rôles
</div>

Vous allez ajouter une règle qui vérifiera si le rôle demandé par l’utilisateur est autorisé, selon son association avec une connexion Social ou connexion de base de données.

1. Accédez à [Auth0 Dashboard &gt; Auth Pipeline &gt; Rules](https://manage.auth0.com/#/rules), puis sélectionnez **Create Rule**.
2. Choisissez le modèle **Empty rule**
3. Nommez la règle **AWS Pets** (ou un nom semblable), puis remplissez le corps de la règle avec le code JavaScript suivant :

export const codeExample = `   function (user, context, callback) {
  if(context.clientID === '{yourClientId}') {
    var socialRoleInfo = {
      role:"arn:aws:iam::<your account>:role/auth0-api-social-role",
      principal: "arn:aws:iam::your account>:saml-provider/auth0"
    };

    var adminRoleInfo = {
      role:"arn:aws:iam::<your account>:role/auth0-api-role",
      principal: "arn:aws:iam::<your account>:saml-provider/auth0"
    };

    var requestRole = context.request.body.role;
    var requestPrincipal = context.request.body.principal;
    var allowedRole = null;

    if(user.identities[0].isSocial === false) {
      allowedRole = adminRoleInfo;
    } else {
      allowedRole = socialRoleInfo;
    }

    if((requestRole && requestRole !== allowedRole.role) ||
      (requestPrincipal && requestPrincipal !== allowedRole.principal)) {
        console.log('rôle demandé non conforme :',requestRole, ':', requestPrincipal);
        console.log('remplacement');
    } else {
      console.log('rôle valide ou aucun rôle demandé pour la délégation');
    }

    context.addonConfiguration = context.addonConfiguration || {};
    context.addonConfiguration.aws = context.addonConfiguration.aws || {};
    context.addonConfiguration.aws.role = allowedRole.role;
    context.addonConfiguration.aws.principal = allowedRole.principal;
    callback(null, user, context);

  } else {
    callback(null, user, context);
  }
}
`;

Assurez-vous de modifier le code ci-dessus avec les valeurs appropriées pour votre intégration. Les champs sont **Princial ARN**, **Role ARN** et **Client Secret**.
4. **Enregistrer** vos modifications.

<div id="caveats">
  #### Mises en garde
</div>

* Les règles s’exécutent globalement pour chaque authentification. Vous devriez exécuter cette logique uniquement pour les requêtes d’authentification associées à une application donnée (c’est pourquoi le script utilisé demande le clientID). Sans cette information, la logique s’exécute pour chaque requête d’authentification associée à votre compte Auth0.
* Les informations sont transmises à la règle avec le contexte et l’utilisateur.
* Vous pouvez enrichir les objets transmis à la règle. Dans le code ci-dessus, la règle vérifie les informations de rôle dans le corps de la requête. Le rôle est défini dans le addonConfiguration du contexte du rôle autorisé, ce qui remplace toujours les paramètres du corps de la requête.
