Auth0 Experiment Center est le moteur d’expérimentation d’Auth0. Il vous permet d’effectuer des tests A/B sur des changements apportés à votre expérience d’[authentification](/fr-CA/docs/authenticate/login/auth0-universal-login) et d’en voir l’impact dans vos [logs](/fr-CA/docs/deploy-monitor/logs) d’événements d’authentification.

Avec Experiment Center, vous définissez votre test A/B dans Auth0, le trafic est réparti de façon déterministe, et vos événements d’authentification existants sont enrichis de métadonnées d’expérience afin que vous puissiez analyser les résultats dans vos propres outils.

<div id="how-it-works">
  ## Fonctionnement
</div>

Experiment Center vous permet d’exécuter des tests A/B contrôlés sur votre pipeline d’authentification Auth0. Au lieu de déployer une modification pour tous les utilisateurs à la fois, vous exposez un nouveau comportement à un pourcentage contrôlé du trafic, mesurez les résultats à l’aide d’événements d’authentification enrichis, puis déployez la variation gagnante lorsque vous êtes prêt.

Experiment Center repose sur trois entités :

* **Experiment** : définit comment le trafic est réparti et quand le test s’exécute.
* **Feature flag** : définit ce que vous testez et les variations possibles.
* **Segment** : définit un ensemble de règles pour acheminer les expériences vers des variations précises.

<Warning>
  Pendant la période Beta, Experiment Center fonctionne uniquement sur les tenants de développement. Les tenants de production ne sont pas pris en charge.
</Warning>

<div id="experiment">
  ## Expérience
</div>

Une expérience sert de cadre de mesure pour un feature flag. Elle définit :

* Quel feature flag est testé
* Comment le trafic est réparti entre les variations
* Quand le test se déroule

Chaque expérience fait référence à un seul feature flag.

<div id="experiment-lifecycle">
  ### Cycle de vie de l’expérience
</div>

Les expériences ont cinq états :

| Statut      | Signification                                                                                                                              |
| ----------- | ------------------------------------------------------------------------------------------------------------------------------------------ |
| `draft`     | Créé, mais non actif. Peut être testé sans risque à l’aide de remplacements de paramètres de requête.                                      |
| `active`    | En cours d’exécution. L’attribution des variantes et l’injection de contexte sont actives pour toutes les transactions d’authentification. |
| `paused`    | Temporairement suspendu. Aucune nouvelle attribution. Les sessions en cours conservent la variation qui leur a été attribuée.              |
| `completed` | Terminé. Aucune nouvelle attribution. La configuration est conservée à titre de référence pendant la promotion manuelle.                   |
| `archived`  | Supprimé de façon logique. Masqué des vues de liste par défaut. Les données analytiques sont conservées.                                   |

<div id="allocation-strategies">
  ### Stratégies d’allocation
</div>

Une expérience utilise l’une des deux stratégies d’allocation suivantes :

**Basée sur un pourcentage :** le trafic est réparti entre les variations selon leur pondération. La somme de toutes les pondérations doit être de 100. Une pondération de 0 est valide (la variation figure dans la définition de l’expérience, mais ne reçoit aucun trafic).

**Basée sur des segments (ciblée) :** le trafic est acheminé vers les variations en fonction de l’appartenance à un segment. Les segments sont évalués par ordre de priorité. Le premier segment correspondant est retenu. Si aucun segment ne correspond, l’allocation `is_fallback` reçoit la requête.

Pour en savoir plus sur l’entité d’expérience, consultez [Détails des entités](/fr-CA/docs/customize/experiment-center/entities#experiment).

<div id="experiment-context">
  ### Contexte de l’expérience
</div>

Lorsqu’une expérience est active et qu’une variation est attribuée, Experiment Center injecte un objet `ExperimentContext` dans vos environnements d’exécution.

Les écrans [ACUL](/fr-CA/docs/customize/login-pages/advanced-customizations#advanced-customizations-for-universal-login-acul) le reçoivent uniquement si vous activez cette option pour l’écran au moyen de `context_configuration`.
Pour en savoir plus, consultez le [guide d’intégration ACUL](/fr-CA/docs/customize/experiment-center/integrations/acul-integration-guide).

Les [Actions](/fr-CA/docs/customize/actions/actions-overview) et les [modèles de page](/fr-CA/docs/customize/login-pages/universal-login/customize-templates#customize-universal-login-page-templates) le reçoivent automatiquement dès qu’une expérience est active. Voici la structure de l’objet :

```typescript
{
  experiment_id: string;   // L'ID d'expérience actif
  variation_id: string;    // L'ID de variation assigné
  config: {                // Configuration fusionnée (référence de base + substitutions)
    [paramName]: { value: unknown }
  };
  is_control: boolean;     // Vrai si c'est la variation de contrôle
}
```

Le champ `config` contient l’intégralité de la configuration fusionnée pour la variation attribuée. Chaque paramètre défini dans le feature flag a toujours une valeur. Vous n’avez pas besoin d’écrire de logique de secours.

<div id="assignment">
  ### Affectation
</div>

Une affectation correspond à la manière dont un utilisateur est dirigé vers une variation précise pendant une transaction d’authentification. Les affectations sont déterministes et persistantes : un même utilisateur voit toujours la même variation pour une même expérience sur le même appareil, ce qui garantit une expérience stable d’une connexion à l’autre.

* **Répartition en pourcentage** divise le trafic entre les variations selon leur pondération. Un sujet donné est toujours affecté à la même variation.
* **Répartition par segment** évalue les propriétés de la requête en fonction des règles de votre segment, selon l’ordre de priorité. Le premier segment correspondant détermine la variation, et une requête qui correspond au même segment mènera toujours à la même variation.

Les résultats d’affectation sont consignés dans les journaux du tenant sous `details.experiment`; ils ne sont pas exposés par une API distincte.

Pour en savoir plus sur l’entité d’affectation, consultez [Détails des entités](/fr-CA/docs/customize/experiment-center/entities#assignment).

<div id="feature-flag">
  ## Feature flag
</div>

Un feature flag est l’unité de contrôle de ce qui est testé. Il contient :

* Une **configuration de base** : des paramètres typés et leurs valeurs par défaut
* Une ou plusieurs **variations** : des configurations de rechange qui diffèrent de la configuration de base

Les feature flags sont propres à un tenant et réutilisables. Le même drapeau peut être référencé par plusieurs expériences au fil du temps (par exemple, un test du T1 et un raffinement du T2 sur la même fonctionnalité).

<div id="feature-flag-lifecycle">
  ### Cycle de vie des feature flags
</div>

Les feature flags ont un cycle de vie enregistré comportant trois états :

| Statut     | Signification                                                                                          |
| ---------- | ------------------------------------------------------------------------------------------------------ |
| `draft`    | Créé, mais pas encore actif. Ne peut pas être référencé par une expérience active.                     |
| `active`   | Prêt à être utilisé. Obligatoire avant qu’une expérience qui référence ce flag puisse être activée.    |
| `archived` | État terminal. Aucune nouvelle expérience ne peut référencer ce flag. Créez un nouveau flag au besoin. |

Pour en savoir plus sur l’entité feature flag, consultez [Détails des entités](/fr-CA/docs/customize/experiment-center/entities#feature-flag).

<div id="variation">
  ### Variation
</div>

Une variation est l’une des versions de l’expérience définies dans un feature flag. Elle précise quels paramètres de configuration diffèrent de la référence et dans quelle mesure.

* La **variation témoin** correspond à la référence; ses remplacements sont vides (aucun paramètre n’est modifié par rapport aux valeurs par défaut du flag).
* Les **variations de traitement** précisent chacune un ou plusieurs remplacements de paramètres.

Les variations ne comportent pas de marqueur `is_control`. Le fait qu’une variation soit le témoin statistique d’une expérience donnée est défini au niveau de l’allocation, et non de la variation. Une même variation peut être le témoin dans une expérience et une variation de traitement dans une autre.

Pour en savoir plus sur l’entité de variation, consultez [Détails des entités](/fr-CA/docs/customize/experiment-center/entities#variation).

<div id="segment">
  ## Segment
</div>

Un segment est un groupe nommé de requêtes d’authentification qui correspondent à un ensemble de règles. Les segments servent, dans des expériences avec allocation ciblée, à acheminer des cohortes de trafic précises vers des variations précises.

Les segments sont limités à un seul tenant et peuvent être réutilisés d’une expérience à l’autre.

Pour en savoir plus sur l’entité Segment, consultez [Détails des entités](/fr-CA/docs/customize/experiment-center/entities#segment).

<div id="limitations">
  ## Limites
</div>

Les limites suivantes s&#39;appliquent durant la bêta :

* **Une seule expérience active par tenant.** Vous pouvez avoir plusieurs expériences dans les états `draft`, `paused` ou `completed`, mais une seule peut être `active` à la fois.
* **Paramètres structurés uniquement.** Les paramètres des feature flags utilisent la structure clé/type/valeur.
* **Trois déclencheurs Actions.** Le contexte de l&#39;expérience est disponible dans [`post_login`](/fr-CA/docs/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger), [`pre_user_registration`](/fr-CA/docs/customize/actions/explore-triggers/signup-and-login-triggers/pre-user-registration-trigger) et [`post_user_registration`](/fr-CA/docs/customize/actions/explore-triggers/signup-and-login-triggers/post-user-registration-trigger).
* **Trafic de test uniquement.** La bêta fonctionne sur des tenants de développement avec le trafic que vous générez. Les données des utilisateurs finaux de l&#39;environnement Production ne transitent pas par la bêta.
* **Promotion manuelle.** Lorsqu&#39;une expérience est terminée, vous appliquez manuellement à votre tenant la configuration de la variation gagnante. La promotion automatisée sera offerte après la disponibilité générale.
