Migrez votre connexion de fédération SAML existante d’un ancien domaine personnalisé Auth0 vers un nouveau, afin de permettre à vos tenants de tirer parti des fonctionnalités de Multiple Custom Domain (MCD).

Les fournisseurs d’identité (IdPs) externes sont souvent codés en dur avec votre ancien domain, ce qui vous oblige à déployer un proxy inverse intelligent qui intercepte la réponse SAML envoyée à votre ancien domain et la redirige de façon sécurisée vers un nouveau domaine personnalisé.

Cela garantit une migration fluide pour les utilisateurs fédérés et vous permet d’utiliser immédiatement votre nouveau domain sans attendre les mises à jour de configuration obligatoires de votre IdP. 

<div id="prerequisites">
  ## Prérequis
</div>

Ce processus nécessite une configuration au moyen de Terraform pour mettre en place les composants Auth0, ainsi que le déploiement d’un Cloudflare Worker pour la logique de proxy.

Avant de commencer votre migration, examinez les exigences ci-dessous :

* Un tenant avec un forfait Enterprise et MCD activé.
* Deux domaines personnalisés vérifiés configurés dans votre tenant Auth0 :
  1. **Ancien domaine Legacy** (pour le proxy)
  2. **Nouveau domaine cible** (pour l’application)
* Terraform CLI et `Node.js/npm` installés.
* Un compte Cloudflare ayant accès au domaine qui héberge vos domaines personnalisés.

<div id="how-it-works">
  ## Fonctionnement
</div>

Cette stratégie de migration utilise un proxy inverse intelligent pour faire le lien entre votre ancien domaine personnalisé et votre nouveau domaine. Ce proxy est déployé sur l’ancien domaine afin d’intercepter la réponse d’authentification SAML envoyée par votre fournisseur d’identité externe (IdP).

C’est nécessaire parce que la configuration de l’IdP est codée en dur pour le point de terminaison de votre ancien domaine. Le proxy modifie les champs de contrôle de la charge utile SAML (comme `Destination` et `Recipient`) pour qu’ils correspondent correctement au nouveau domaine personnalisé.

Enfin, le proxy achemine cette charge utile corrigée vers le point de terminaison de connexion du nouveau domaine. Cela permet un basculement vers votre nouveau domaine sans interruption de service, sans exiger de changements manuels à la configuration de vos partenaires IdP.

<div id="setup-and-configuration">
  ## Mise en place et configuration
</div>

Pour mettre en place et configurer votre migration :

1. Clonez le dépôt de la migration :

```bash
git clone https://github.com/abbaspour/auth0-mcd-federation-migration.git
cd auth0-mcd-federation-migration
```

2. Installez les dépendances :

```bash
npm install
```

3. Créez un fichier `terraform.auto.tfvars` dans votre répertoire `tf` avec les identifiants requis et les renseignements sur le domaine :

```bash
# Variables du fournisseur de services (SP) Auth0
auth0_domain = "your-sp-tenant.auth0.com"
auth0_existing_custom_domain = "oldfed.example.com"
auth0_new_custom_domain = "id.example2.com"
auth0_tf_client_id = "your-sp-client-id"
auth0_tf_client_secret = "your-sp-client-secret"

# Variables du fournisseur d'identité (IDP) Auth0
auth0_idp_domain = "your-idp-tenant.auth0.com"
auth0_idp_tf_client_id = "your-idp-client-id"
auth0_idp_tf_client_secret = "your-idp-client-secret"

# Variables Cloudflare
cloudflare_api_key = "your-cloudflare-api-key"
cloudflare_email = "your-cloudflare-email"
cloudflare_zone_id = "your-cloudflare-zone-id"
```

4. Initialisez et appliquez Terraform :

```bash
cd tf
terraform init
terraform apply
```

Cela crée l’application SAML et la connexion nécessaires, configure le DNS dans Cloudflare et prépare les variables d’environnement pour le worker.

<div id="deploy-the-cloudflare-worker">
  ### Déployer le Cloudflare Worker
</div>

Ce proxy gère l’interception de la réponse SAML et la logique de redirection. Pour le déployer :

1. Déployez le proxy Cloudflare :

```bash
cd ..
npx wrangler deploy
```

2. Le worker reçoit automatiquement les variables d’environnement requises (comme `AUTH0_EDGE_LOCATION` et `NEW_SP_DOMAIN`) depuis la sortie de Terraform.

<div id="temporarily-update-saml-connection-parameters">
  ### Mettre temporairement à jour les paramètres de la connexion SAML
</div>

Votre ancien domaine reçoit la réponse SAML avant qu’elle soit retransmise au nouveau domaine. Par conséquent, les paramètres de validation attendus par la connexion SAML doivent temporairement pointer vers l’URL de rappel de l’ancien domaine afin d’éviter les erreurs de non-correspondance.

1. Obtenez un jeton d’accès à l’API de gestion pour le tenant de votre fournisseur de services, avec les scopes `read:connections` et `update:connections` :

   ```bash
   cd bin/
   export access_token='<sp-tenant-management-api-token>'
   ```

2. Mettez à jour l’URL de destination :

   ```bash
   ./sp-set-destination-url.sh -i <saml-connection-id> -d https://oldfed.example.com/login/callback
   ```

3. Mettez à jour l’URL du destinataire :

   ```bash
   ./sp-set-recipient-url.sh -i <saml-connection-id> -r https://oldfed.example.com/login/callback
   ```
