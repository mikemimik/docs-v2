Pour configurer les fonctionnalités d’Auth0 afin qu’elles utilisent votre <Tooltip tip="Domaine personnalisé : domaine tiers avec un nom spécialisé ou personnalisé." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=custom+domain">domaine personnalisé</Tooltip>, vous devrez peut-être suivre des étapes supplémentaires selon les fonctionnalités que vous utilisez. Par exemple, vous devrez peut-être apporter des modifications avant de pouvoir utiliser votre domaine personnalisé sur votre page de connexion ou pour appeler vos API.

Si vous utilisez Auth0 depuis un certain temps et décidez d’activer un domaine personnalisé, vous devrez migrer vos applications existantes et mettre à jour les paramètres, comme décrit ci-dessous, y compris ceux de tout VPN ou pare-feu que vous utilisez. Notez que les sessions existantes créées sur `{yourDomain}` ne seront plus valides une fois que vous commencerez à utiliser votre domaine personnalisé; les utilisateurs devront donc se reconnecter.

</div>

<div id="universal-login">
  ## Universal Login
</div>

Si vous utilisez [Auth0 Universal Login](/fr-CA/docs/authenticate/login/auth0-universal-login) et que vous avez personnalisé la page de connexion, vous devez mettre à jour le code pour utiliser votre domaine personnalisé. Si vous utilisez la page de connexion **par défaut** sans l’avoir personnalisée, vous n’avez rien à modifier. Pour en savoir plus, consultez le <Tooltip tip="Universal Login : votre application redirige l’utilisateur vers Universal Login, hébergé sur l’Authorization Server d’Auth0, afin de vérifier son identité." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=Universal+Login">Universal Login</Tooltip> d’Auth0.

Si vous utilisez [Lock for Web](/fr-CA/docs/libraries/lock), vous devez définir les options `configurationBaseUrl` et `overrides`, comme dans l’exemple de script suivant :

```javascript lines
var lock = new Auth0Lock(config.clientID, config.auth0Domain, {
  //code omis par souci de concision
  configurationBaseUrl: config.clientConfigurationBaseUrl,
  overrides: {
  	__tenant: config.auth0Tenant,
  	__token_issuer: config.authorizationServer.issuer
  },
  //code omis par souci de concision
});
```

Si vous utilisez [Auth0.js](/fr-CA/docs/libraries/auth0js) sur la page Universal Login, vous devez définir l’option `overrides`.

```javascript lines
var webAuth = new auth0.WebAuth({
  clientID: config.clientID,
  domain: config.auth0Domain,
  //code omitted par souci de concision
  overrides: {
  	__tenant: config.auth0Tenant,
  	__token_issuer: config.authorizationServer.issuer
  },
  //code omitted par souci de concision
});
```

Dans la plupart des cas, les bibliothèques Auth0.js et Lock récupèrent le nom du tenant (requis pour `/usernamepassword/login`) et l’émetteur (requis pour la validation de `id_token`) à partir du domain. Toutefois, si vous êtes un client Private Cloud qui utilise un proxy ou un nom de domaine personnalisé dont le nom de domaine est différent du tenant ou de l’émetteur, vous pouvez utiliser `__tenant` et `__token_issuer` pour fournir vos propres valeurs.

<div id="embedded-lock">
  ## Embedded Lock
</div>

Si vous utilisez [Lock for Web](/fr-CA/docs/libraries/lock) intégré dans votre application, vous devez mettre à jour le code pour utiliser votre domaine personnalisé lors de l’initialisation de Lock. Vous devrez également définir le `configurationBaseUrl` avec l’URL CDN appropriée.

{/* codeblockOld.header.login.logInButton codeblockOld.header.login.configureSnippet */}

export const codeExample1 = `var lock = new Auth0Lock('{yourClientId}', '{yourCustomDomain}', {
  //code omis par souci de concision
  configurationBaseUrl: 'https://cdn.us.auth0.com'
  //code omis par souci de concision
});`;

L’URL du CDN varie selon la région. Utilisez `https://cdn.[us|eu|au|jp].auth0.com` (`us` pour les États-Unis, `eu` pour l’Europe, `au` pour l’Australie ou `jp` pour le Japon).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  L’URL du CDN varie selon la région. Les tenants créés avant le 11 juin 2020 doivent utiliser `https://cdn.auth0.com` si la région correspond aux États-Unis, ou ajouter `eu`, `au` ou `jp` pour l’Europe, l’Australie ou le Japon. Si votre tenant a été créé après le 11 juin 2020, utilisez `https://cdn.us.auth0.com` si la région correspond aux États-Unis.
</Callout>

<div id="use-custom-domains-in-emails-and-phone-notifications">
  ## Utiliser des domaines personnalisés dans les courriels et les notifications téléphoniques
</div>

Si vous souhaitez utiliser votre domaine personnalisé avec vos courriels Auth0 ou vos notifications téléphoniques, vous devez activer cette fonctionnalité.

1. Accédez à [Auth0 Dashboard &gt; Branding &gt; Custom Domains](https://manage.auth0.com/#/branding/custom_domains).
2. Activez la bascule **Utiliser le domaine personnalisé dans les courriels**.

<div id="configure-social-identity-providers">
  ## Configurer les fournisseurs d’identité sociaux
</div>

Si vous souhaitez utiliser votre domaine personnalisé avec des <Tooltip tip="Fournisseur d’identité (IdP) : service qui stocke et gère les identités numériques." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=identity+providers">fournisseurs d’identité</Tooltip> sociaux (IdP), vous devez mettre à jour la liste des URI de redirection autorisées de votre IdP pour y inclure votre domaine personnalisé (par exemple, `https://login.northwind.com/login/callback`).

Vous ne pouvez pas utiliser les [clés de développeur Auth0](/fr-CA/docs/authenticate/identity-providers/social-identity-providers/devkeys) avec des domaines personnalisés.

<div id="configure-google-workspace-connections">
  ## Configurer les connexions Google Workspace
</div>

Si vous souhaitez utiliser votre domaine personnalisé avec les connexions Google Workspace, vous devez mettre à jour l’URI de redirection autorisée dans les paramètres de votre client <Tooltip tip="OAuth 2.0 : Cadre d’autorisation qui définit les protocoles et les workflows d’autorisation." cta="Consulter le glossaire" href="/fr-CA/docs/glossary?term=OAuth">OAuth</Tooltip>. Dans Google Cloud Console, accédez à **Credentials**, sélectionnez votre client OAuth dans la liste, puis vous verrez une page de paramètres contenant le <Tooltip tip="Client ID : Valeur d’identification attribuée à votre ressource enregistrée par Auth0." cta="Consulter le glossaire" href="/fr-CA/docs/glossary?term=Client+ID">Client ID</Tooltip> de l’application, le secret et d’autres champs. Dans le champ **Authorized redirect URIs**, ajoutez une URL au format `https://<YOUR-CUSTOM-DOMAIN>/login/callback` qui inclut votre domaine personnalisé (par exemple `https://login.northwind.com/login/callback`).

<div id="apis">
  ## APIs
</div>

Les identifiants d’API (c.-à-d. `audience`) ne changeront pas. Il s’agit d’une valeur constante pour chaque API et, même s’il est courant d’utiliser un URI, elle est complètement indépendante du domaine utilisé pour obtenir le jeton.

Auth0 émet des jetons avec le claim `iss` du domaine que vous avez utilisé pour obtenir le jeton.

<div id="auth0-apis">
  ##### API Auth0
</div>

Continuez d’utiliser le nom de domaine par défaut de votre tenant (comme `https://{yourDomain}/userinfo` et `https://{yourDomain}/api/v2/`) plutôt que votre domaine personnalisé lorsque vous spécifiez une `audience`. C’est le seul endroit où vous devez utiliser le domaine par défaut de votre tenant.

Toutes les requêtes (c.-à-d. l’obtention du jeton et l’appel proprement dit de l’API) doivent utiliser le même domaine. Les jetons obtenus au moyen d’un domaine personnalisé doivent être utilisés avec une API Auth0 qui utilise ce même domaine personnalisé.

Si vous utilisez un flux d’authentification avec votre <Tooltip tip="Jeton d’accès : information d’autorisation, sous la forme d’une chaîne opaque ou d’un JWT, utilisée pour accéder à une API." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=Access+Tokens">domaine personnalisé</Tooltip> pour demander des <Tooltip tip="Jeton d’accès : information d’autorisation, sous la forme d’une chaîne opaque ou d’un JWT, utilisée pour accéder à une API." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=Access+Tokens">jetons d’accès</Tooltip> afin d’accéder à la <Tooltip tip="Management API : un produit qui permet aux clients d’effectuer des tâches administratives." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=Management+API">Management API</Tooltip>, vous devez aussi appeler le point de terminaison de la Management API avec votre domaine personnalisé.

```json lines
POST https://mycustomdomain.com/oauth/token
... // autres paramètres 
...
audience:https://defaulttenant.eu.auth0.com/api/v2/
```

Votre requête de jeton d’accès devrait ressembler à ceci

```json lines
GET https://mycustomdomain.com/api/v2/clients

Headers:
Authorization: Bearer <access_token>
```

<div id="custom-apis">
  ##### API personnalisées
</div>

Si vous utilisez Auth0 avec un domaine personnalisé pour émettre des jetons d’accès pour vos API, vous devez valider le ou les émetteurs du <Tooltip tip="JSON Web Token (JWT) : format standard des ID token (et souvent des jetons d’accès) utilisé pour représenter des claims de façon sécurisée entre deux parties." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=JWT">JWT</Tooltip> en fonction de votre domaine personnalisé. Par exemple, si vous utilisez le middleware [express-jwt](https://github.com/auth0/express-jwt), vous devez apporter la modification suivante :

```js lines
app.use(jwt({
  issuer: 'https://<YOUR-CUSTOM-DOMAIN>/',
  //code omis par souci de concision
}));
```

<div id="configure-saml-identity-providers">
  ## Configurer les fournisseurs d&#39;identité SAML
</div>

Pour utiliser votre domaine personnalisé avec les <Tooltip tip="Security Assertion Markup Language (SAML) : protocole normalisé permettant à deux parties d'échanger des renseignements d'authentification sans mot de passe." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=SAML">fournisseurs d&#39;identité</Tooltip> (IdP) SAML, vous devez mettre à jour vos **URL(s) Assertion Consumer Service (ACS)** auprès du ou des fournisseurs d&#39;identité. Selon ce que l&#39;IdP prend en charge, vous pouvez procéder de l&#39;une des deux façons suivantes :

1. Vous pouvez obtenir les métadonnées du fournisseur de services depuis Auth0 à l&#39;adresse `https://<YOUR-CUSTOM-DOMAIN>/samlp/metadata?connection=<YOUR-CONNECTION-NAME>`. Elles incluront l&#39;URL ACS mise à jour. Vous devrez ensuite mettre cette valeur à jour manuellement dans les paramètres de votre ou vos IdP. Cette modification doit être apportée à votre ou vos IdP au même moment où vous commencez à utiliser votre domaine personnalisé dans vos applications. Cela peut poser problème s&#39;il y a plusieurs IdP à configurer.
2. Si l&#39;IdP le prend en charge, vous pouvez utiliser des requêtes signées pour répondre à cette exigence :

* Téléchargez le certificat de signature depuis `https://<TENANT>.auth0.com/pem`. Notez que `https://<YOUR-CUSTOM-DOMAIN>.com/pem` renverra le même certificat
* Remettez le certificat au(x) IdP afin qu&#39;il soit téléversé. Cela permet à l&#39;IdP de valider la signature du message `AuthnRequest` qu&#39;Auth0 envoie à l&#39;IdP
* L&#39;IdP importera le certificat et, au besoin, la vérification de la signature devra être activée (les étapes exactes varient selon l&#39;IdP)
* Activez le bouton bascule **Sign Request** dans le Dashboard, sous **Connections &gt; Enterprise &gt; SAML &gt; CONNECTION**. Cela amènera Auth0 à signer les messages SAML `AuthnRequest` qu&#39;il envoie à l&#39;IdP.

Une fois cela fait, lorsque vous commencerez à utiliser votre domaine personnalisé pour lancer une requête d&#39;authentification dans votre application, l&#39;IdP recevra ce domaine personnalisé dans votre requête signée. Comme la requête signée de votre application est approuvée, l&#39;IdP devrait automatiquement remplacer toute valeur configurée comme URL ACS par la valeur envoyée dans la requête signée. Toutefois, certains IdP n&#39;acceptent **pas** l&#39;URL ACS dans la requête signée; vous devez donc d&#39;abord vérifier auprès du vôtre si cette fonctionnalité est prise en charge.

Si c&#39;est le cas, cela vous évitera d&#39;avoir à modifier un ou plusieurs paramètres d&#39;IdP exactement au même moment et vous permettra de les préparer à accepter vos requêtes signées à l&#39;avance. Vous pourrez ensuite également modifier plus tard l&#39;URL ACS configurée statiquement dans les paramètres de votre IdP.

Notez que si votre fournisseur d&#39;identité SAML est configuré pour utiliser votre domaine personnalisé, tester la connexion à l&#39;aide du bouton **Try** dans le Dashboard **ne** fonctionnera **pas** et les liens par défaut pour télécharger les métadonnées depuis Auth0 afficheront toujours le domaine par défaut, et non le domaine personnalisé.

Si vous avez un flux d&#39;authentification initié par l&#39;IdP, vous devrez mettre à jour le ou les IdP et votre ou vos applications en même temps pour utiliser le domaine personnalisé.

<div id="configure-saml-applications">
  ## Configurer les applications SAML
</div>

Si vous voulez utiliser votre domaine personnalisé avec des applications SAML (lorsqu’Auth0 est l’IdP), vous devez mettre à jour votre fournisseur de services avec les nouvelles métadonnées du fournisseur d’identité d’Auth0. Vous pouvez obtenir les métadonnées à jour correspondant au domaine personnalisé à l’adresse `https://<YOUR-CUSTOM-DOMAIN>/samlp/metadata/<YOUR-CLIENT-ID>`. Notez que l’ID d’entité de l’émetteur de l’assertion renvoyée par Auth0 changera lorsque vous utiliserez un domaine personnalisé (par exemple, de `urn:northwind.auth0.com` à une valeur utilisant le domaine personnalisé, comme `urn:login.northwind.com`).

Si vous avez un flux d’authentification initié par l’IdP, vous devrez mettre à jour l’URL utilisée pour lancer ce flux afin qu’elle reflète le domaine personnalisé. Au lieu de `https://<TENANT>.auth0.com/samlp/<YOUR-CLIENT-ID>`, vous devriez utiliser `https://<YOUR-CUSTOM-DOMAIN>/samlp/<YOUR-CLIENT-ID>`.

<div id="configure-ws-fed-applications">
  ## Configurer les applications WS-Fed
</div>

Si vous souhaitez utiliser votre domaine personnalisé avec des <Tooltip tip="Web Service Federation (WS-Fed) : protocole de gestion des identités utilisateur entre domaines." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=WS-Fed">applications WS-Fed</Tooltip> lorsque Auth0 agit comme IdP, vous devez mettre à jour votre fournisseur de services avec les nouvelles métadonnées du fournisseur d’identité d’Auth0. Vous pouvez obtenir les métadonnées correspondant au domaine personnalisé à l’adresse `https://<YOUR-CUSTOM-DOMAIN>/wsfed/FederationMetadata/2007-06/FederationMetadata.xml`.

<div id="configure-azure-ad-connections">
  ## Configurer les connexions Azure AD
</div>

Si vous souhaitez utiliser votre domaine personnalisé avec les connexions Azure AD, vous devez mettre à jour l’**Allowed Reply URL** dans vos paramètres Azure AD. Dans votre Azure Active Directory, accédez à **Apps registrations** et sélectionnez votre application. Cliquez ensuite sur **Settings -&gt; Reply URLs** et ajoutez une URL avec votre domaine personnalisé au format `https://<YOUR-CUSTOM-DOMAIN>/login/callback` (par exemple `https://login.northwind.com/login/callback`).

<div id="configure-adfs-connections">
  ## Configurer les connexions ADFS
</div>

Si vous souhaitez utiliser votre domaine personnalisé avec des connexions ADFS, vous devez mettre à jour le point de terminaison dans vos paramètres ADFS. Vous devrez le mettre à jour pour utiliser votre domaine personnalisé dans l’URL de rappel au format `https://<YOUR-CUSTOM-DOMAIN>/login/callback` (par exemple `https://login.northwind.com/login/callback`).

<div id="configure-adldap-connections">
  ## Configurer les connexions AD/LDAP
</div>

Si vous n’avez pas besoin de la prise en charge de Kerberos, les connexions AD/LDAP ne requièrent aucune configuration supplémentaire.

Pour utiliser des connexions AD/LDAP avec la prise en charge de Kerberos, vous devrez mettre à jour le point de terminaison de ticket pour qu’il fonctionne avec le domaine personnalisé. Comme il est indiqué dans la [documentation du connecteur AD/LDAP d’Auth0](/fr-CA/docs/authenticate/identity-providers/enterprise-identity-providers/active-directory-ldap/ad-ldap-connector/ad-ldap-connector-to-auth0#point-an-ad-ldap-connector-to-a-new-connection), le fichier `config.json` doit être modifié en remplaçant la valeur `PROVISIONING_TICKET` afin d’utiliser votre domaine personnalisé au format `https://<YOUR-CUSTOM-DOMAIN>/p/ad/jUG0dN0R`.

Une fois cette modification enregistrée, vous devez redémarrer le service AD/LDAP Connector pour qu’elle prenne effet.
