Suivez les directives ci-dessous pour rédiger un code d’Actions performant, sécuritaire et clair, afin d’assurer un environnement de production efficace.

<div id="actions-basics">
  ## Notions de base sur Actions
</div>

* Utilisez le moins de requêtes HTTP possible et définissez un délai d’attente raisonnable (moins de 10 secondes) pour éviter l’accumulation de requêtes pendant le login.
* Utilisez les [métadonnées de l’application](/docs/fr-CA/get-started/applications/configure-application-metadata) pour filtrer des applications précises afin de déterminer si une Action doit être exécutée.

  * Exemple : des valeurs propres à l’application ou des valeurs de regroupement d’applications servant à faire varier partiellement ou totalement l’exécution de l’Action.
* Utilisez `user_metadata`/`app_metadata` pour conserver des données personnalisées dans le profil de l’utilisateur.

  * Exemple : préférences de l’utilisateur ou valeurs d’état.
* Utilisez les [métadonnées de transaction](/docs/fr-CA/customize/actions/transaction-metadata) pour partager des données entre les [déclencheurs post-login](/docs/fr-CA/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger) au cours d’une même exécution.

  * Exemple : valeurs récupérées à partir de services externes ou calculées, qui doivent être réutilisées dans des Actions subséquentes et qui sont liées à l’exécution en cours.
* Utilisez le cache pour les données devant être partagées par des Actions appartenant à différentes exécutions.

  * Exemple : valeurs récupérées à partir de services externes ou calculées, qui doivent être réutilisées par des Actions et qui ne sont liées ni à l’utilisateur ni à l’exécution en cours.
* Assurez-vous que les Actions qui effectuent une vérification ou déclenchent l’authentification multifacteur <Tooltip tip="MFA : processus d’authentification de l’utilisateur qui utilise un facteur en plus du username et du mot de passe, comme un code par SMS." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=MFA">MFA</Tooltip> ne puissent pas être contournées, volontairement ou non.
* Les Actions ne devraient jamais générer intentionnellement une erreur; si un processus doit être interrompu en raison d’une erreur ou d’une condition, utilisez la méthode `api` appropriée, comme `api.access.deny()`.
* Utilisez `event.request.hostname` pour le domaine utilisé dans les appels à l’Authentication API; il peut s’agir du domaine par défaut du tenant Auth0 ou d’un [domaine personnalisé](/docs/fr-CA/customize/custom-domains).

<div id="actions-modules">
  ## Modules d’Actions
</div>

* Utilisez les Modules d’Actions lorsque vous devez réutiliser des fonctions dans différentes Actions, afin d’éviter la duplication du code.
* Réduisez le temps d’instanciation des objets lorsque possible en appliquant le [patron Singleton](https://en.wikipedia.org/wiki/Singleton_pattern) pour réutiliser des instances d’objet dans le Module d’Action.
* Suivez les directives applicables présentées dans ce document concernant la programmation, la sécurité, la journalisation, les dépendances et les données utilisateur.
* Lorsque vous utilisez un Module d’Action à partir d’une Action, portez toujours attention à la version du module utilisée.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Lorsque vous publiez une nouvelle version d’un Module d’Action, Auth0 ne met pas automatiquement à niveau la version à laquelle les Actions font référence, afin d’éviter des problèmes potentiels si la mise à niveau n’est pas compatible avec l’Action.
</Callout>

<div id="coding-basics">
  ## Notions de base du code
</div>

* Utilisez des comparaisons strictes `===` pour toutes les données entrantes ou stockées.
* Utilisez une instruction `return` lorsque l’exécution de l’Action doit s’arrêter.
* Exécutez un linter de code, comme [ESLint](https://eslint.org), ainsi qu’un analyseur, comme [Semgrep](https://semgrep.dev/docs/getting-started/), pour améliorer la qualité du code et repérer automatiquement les problèmes.

<div id="security-basics">
  ## Principes de base de la sécurité
</div>

* N’écrivez pas de secrets ni d’artefacts de code sensibles en clair dans le code de vos Actions. Utilisez plutôt le [Secrets Manager](/docs/fr-CA/customize/actions/write-your-first-action#add-a-secret) ou votre propre gestionnaire en l’intégrant au code des Actions.
* Ne transmettez pas de renseignements personnels identifiables (PII) non chiffrés à découvert, par exemple dans les URL ou les messages d’erreur.
* Utilisez toujours des URL HTTPS pour les redirections et les appels d’API.
* Placez les adresses IP dans une AllowList lorsque possible.
* Soyez attentif aux données entrantes qui peuvent être altérées (paramètres d’URL, agent utilisateur, etc.).

<div id="defensive-coding">
  ## Programmation défensive
</div>

* [Interceptez les erreurs](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Guide/Control_flow_and_error_handling#exception_handling_statements) et gérez-les au besoin.
* Utilisez des clauses de garde et retournez sans attendre si le traitement de l’Action ne doit pas se poursuivre.

<div id="logging">
  ## Journalisation
</div>

* Ne consignez jamais de données sensibles, de secrets ou de PII.
* Limitez-vous à un maximum de 256 caractères consignés par Action.

<div id="dependencies">
  ## Dépendances
</div>

* Utilisez des paquets fiables et maintenus.
* Vérifiez s’il existe des CVE non corrigées à l’aide de la [fonction d’audit de `npm`](https://docs.npmjs.com/cli/v6/commands/npm-audit) ou d’un outil automatisé de vérification des dépendances connecté à un dépôt.
* Utilisez la version la plus récente d’un paquet lorsque possible.

<div id="user-data">
  ## Données utilisateur
</div>

* Vérifiez si une adresse courriel est vérifiée avec `event.user.email_verified` lorsqu’elle est utilisée dans un contexte sensible ou hautement sécurisé.
* Selon la connexion, les données de profil utilisateur peuvent varier; le seul [champ du profil utilisateur](/docs/fr-CA/manage-users/user-accounts/user-profiles/user-profile-structure) garanti est le `user_id`.

<div id="redirect-actions-in-the-login-flow">
  ## Redirect Actions dans le flux de connexion
</div>

* Le jeton renvoyé par `api.redirect.encodeToken` est signé, mais non chiffré; il ne faut donc pas inclure de données sensibles ni de renseignements personnels identifiables (PII) dans le payload.
* Le flux de connexion s’exécute après une connexion réussie, y compris dans les cas suivants :

  * SSO (aucun formulaire de connexion n’est affiché)
  * authentification silencieuse (vérification d’une session à l’aide de `prompt=none` dans l’URL d’autorisation)
  * échange de jeton d’actualisation (aucune interaction utilisateur)
  * octrois RO par mot de passe (identifiants recueillis à partir d’une application et échangés avec le point de terminaison de jeton)
* Les Actions qui effectuent une redirection doivent tenir compte des cas ci-dessus et soit refuser l’accès si une interaction est nécessaire, soit autoriser délibérément le contournement, ce qui reporte la responsabilité sur l’application qui demande la connexion.
