L’objet API du déclencheur d’Actions custom-token-exchange comprend :

<div id="apiaccess">
  ## `api.access`
</div>

Modifiez l’accès associé à la requête d’échange de jetons, par exemple en la rejetant.

<div id="apiaccessdenycode-reason">
  ### `api.access.deny(code, reason)`
</div>

Marquez l’échange de jetons en cours comme refusé.

Si la requête est refusée en raison d’un jeton du sujet invalide, nous recommandons d’utiliser plutôt api.access.rejectInvalidSubjectToken
afin de distinguer les tentatives d’attaque par force brute sur le jeton du sujet des autres motifs de refus de la requête.

<ResponseField name="code" type="string">
  Le code d’erreur justifiant le refus de l’échange de jetons. Il peut s’agir de invalid&#95;request, server&#95;error ou de tout code personnalisé
</ResponseField>

<ResponseField name="reason" type="string">
  Une explication en clair du refus de la requête d’échange de jetons.
</ResponseField>

```js
exports.onExecuteCustomTokenExchange = async (event, api) => {

  // 1. Valider le subject_token
  const subject_token = await validateToken(event.transaction.subject_token, jwksUri);

  // 2. Appliquer votre politique d'autorisation à l'utilisateur
  const isAuthorized = await authorizeAccess(subject_token.sub);
  if (!isAuthorized) {
    api.access.deny('Unauthorized_login', 'User cannot login due to reason: X');
  }

  // si l'utilisateur est autorisé, continuer comme indiqué ici

};
```

<div id="apiaccessrejectinvalidsubjecttokenreason">
  ### `api.access.rejectInvalidSubjectToken(reason)`
</div>

Indiquez que le jeton du sujet fourni dans la requête n’est pas valide. La requête sera alors
rejetée avec le code d’erreur &quot;invalid&#95;request&quot;.

Cela indiquera aux fonctionnalités Attack Protection qu’un jeton du sujet invalide a été fourni,
afin que les protections contre les attaques par force brute visant le jeton du sujet puissent être appliquées.

<ResponseField name="reason" type="string">
  Une explication en langage clair du rejet de la requête d’échange de jetons.
</ResponseField>

```js
exports.onExecuteCustomTokenExchange = async (event, api) => {

  try {
    // Valider le subject_token
    const subject_token = await validateToken(event.transaction.subject_token, jwksUri);
    // définir l'utilisateur pour la transaction
    api.authentication.setUserById(subject_token.id);

  } catch (error) {
    if (error.message === 'Invalid Token') {
      // Si le problème vient précisément du fait que le subject_token est invalide
      console.error('Invalid Token error');
      api.access.rejectInvalidSubjectToken('Invalid subject_token');
    } else {
      // pour toute autre erreur inattendue, lancer une erreur serveur
      throw error;
    }
  }

};
```

<div id="apiauthentication">
  ## `api.authentication`
</div>

Indique le résultat de l’authentification du jeton du sujet afin de préciser pour quel utilisateur des jetons seront émis.

<div id="apiauthenticationsetuserbyiduser_id">
  ### `api.authentication.setUserById(user_id)`
</div>

Indiquez l’utilisateur correspondant au `subject&#95;token` en fournissant le userId. La demande d’échange de jetons émettra des jetons pour cet utilisateur.
Il doit s’agir d’un utilisateur existant.
Remarque : une seule des méthodes api.authentication.setUserByConnection et api.authentication.setUserById doit être appelée par l’action Échange de jetons personnalisé.

<ResponseField name="user_id" type="string">
  ID de l’utilisateur; doit correspondre à un utilisateur existant.
</ResponseField>

```js
exports.onExecuteCustomTokenExchange = async (event, api) => {

  // 1. Valider le subject_token
  const subject_token = await validateToken(event.transaction.subject_token, jwksUri);

  // 2. Appliquer votre politique d'autorisation à l'utilisateur
  const isAuthorized = await authorizeAccess(subject_token.sub);
  if (!isAuthorized) {
    api.access.deny('Unauthorized_login', 'User cannot login due to reason: X');
  }

  // 3. Définir l'utilisateur pour la transaction
  api.authentication.setUserById(subject_token.sub);

  return;
};
```

<div id="apiauthenticationsetuserbyconnectionconnection_name-user_attributes-options">
  ### `api.authentication.setUserByConnection(connection_name, user_attributes, options)`
</div>

Indiquez l’utilisateur correspondant au `subject_token` en fournissant une connexion et des attributs utilisateur.
La requête d’échange de jetons émettra des jetons pour cet utilisateur.

Il peut s’agir d’un utilisateur existant ou d’un nouvel utilisateur. Si l’utilisateur n’existe pas, il sera créé.
La propriété user&#95;id du user&#95;profile sera utilisée pour déterminer si l’utilisateur existe déjà.

Remarque : Exactement un de api.authentication.setUserByConnection ou api.authentication.setUserById doit être appelé par l’action Échange de jetons personnalisé.

<ResponseField name="connection_name" type="string">
  Nom de la connexion dans laquelle l’utilisateur doit être enregistré.
</ResponseField>

<ResponseField name="user_attributes" type="customtokenexchangesetuserbyconnectionuserattributes">
  Les attributs du profil de l’utilisateur, y compris user&#95;id et, au besoin, d’autres attributs comme l’email, le nom, etc.

  Le champ user&#95;id est requis et doit être l’identifiant unique de l’utilisateur dans la connexion;
  il servira à déterminer si l’utilisateur existe ou s’il doit être créé. Pour les utilisateurs existants, ce user&#95;id
  se trouve dans le tableau identities du profil utilisateur normalisé.

  Si l’utilisateur existe déjà, les attributs utilisateur suivants ne peuvent pas être mis à jour : email, email&#95;verified, phone, phone&#95;verified, username.
  S’ils ne correspondent pas à ceux de l’utilisateur existant, une erreur sera renvoyée.

  <Expandable title="propriétés de user_attributes" defaultOpen>
    <ResponseField name="email" type="string" post={["optional"]}>
      L’adresse courriel de l’utilisateur.
    </ResponseField>

    <ResponseField name="email_verified" type="boolean" post={["optional"]}>
      Indique si cette adresse courriel est vérifiée (true) ou non vérifiée (false).
    </ResponseField>

    <ResponseField name="family_name" type="string" post={["optional"]}>
      Le ou les noms de famille de l’utilisateur.
    </ResponseField>

    <ResponseField name="given_name" type="string" post={["optional"]}>
      Le ou les prénoms de l’utilisateur.
    </ResponseField>

    <ResponseField name="name" type="string" post={["optional"]}>
      Le nom complet de l’utilisateur.
    </ResponseField>

    <ResponseField name="nickname" type="string" post={["optional"]}>
      Le surnom de l’utilisateur.
    </ResponseField>

    <ResponseField name="phone_number" type="string" post={["optional"]}>
      Le numéro de téléphone de l’utilisateur (conformément à la recommandation E.164).
    </ResponseField>

    <ResponseField name="phone_verified" type="boolean" post={["optional"]}>
      Indique si ce numéro de téléphone a été vérifié (true) ou non (false).
    </ResponseField>

    <ResponseField name="picture" type="string" post={["optional"]}>
      Un URI pointant vers l’image de l’utilisateur.
    </ResponseField>

    <ResponseField name="user_id" type="string">
      L’identifiant unique de l’utilisateur dans la connexion.
    </ResponseField>

    <ResponseField name="username" type="string" post={["optional"]}>
      Le nom d’utilisateur de l’utilisateur.
    </ResponseField>

    <ResponseField name="verify_email" type="boolean" post={["optional"]}>
      Indique si l’utilisateur recevra un courriel de vérification après sa création (true) ou aucun courriel (false).
    </ResponseField>
  </Expandable>
</ResponseField>

<ResponseField name="options" type="customtokenexchangesetuserbyconnectionoptions">
  Options permettant de contrôler le comportement de la commande setUserByConnection.

  * `creationBehavior` - comportement à appliquer si aucun utilisateur avec le user&#95;id spécifié n’existe dans la connexion.
    Peut être &#39;create&#95;if&#95;not&#95;exists&#39;, ce qui entraînera la création d’un nouvel utilisateur à l’aide des attributs utilisateur fournis;
    ou &#39;none&#39;, ce qui signifie qu’aucun utilisateur ne sera créé et qu’une erreur sera renvoyée si aucun utilisateur n’existe.

  * `updateBehavior` - comportement à appliquer si un utilisateur avec le user&#95;id spécifié existe déjà dans la connexion.
    Peut être &#39;replace&#39;, ce qui fera en sorte que les attributs de l’utilisateur existant soient remplacés par les
    attributs utilisateur spécifiés; ou &#39;none&#39;, ce qui signifie que l’utilisateur existant ne sera pas modifié.

  <Expandable title="propriétés des options" defaultOpen>
    <ResponseField name="creationBehavior" type="string">
      Comportement à appliquer si aucun utilisateur avec le user&#95;id spécifié n’existe dans la connexion.

      Valeurs autorisées : `create_if_not_exists`, `none`
    </ResponseField>

    <ResponseField name="updateBehavior" type="string">
      Comportement à appliquer si un utilisateur avec le user&#95;id spécifié existe déjà dans la connexion.

      Valeurs autorisées : `none`, `replace`
    </ResponseField>
  </Expandable>
</ResponseField>

```js Set user by connection with full profile attributes
exports.onExecuteCustomTokenExchange = async (event, api) => {

  // 1. Valider le subject_token
  const subject_token = await validateToken(event.transaction.subject_token, jwksUri);

  // 2. Appliquer votre politique d'autorisation à l'utilisateur
  const isAuthorized = await authorizeAccess(subject_token.sub);
  if (!isAuthorized) {
    api.access.deny('Unauthorized_login', 'User cannot login due to reason: X');
  }

  // 3. Définir l'utilisateur pour la transaction
  api.authentication.setUserByConnection(
    'My Connection',
    {
      user_id: subject_token.sub,
      email: subject_token.email,
      email_verified: subject_token.email_verified,
      phone_number: subject_token.phone_number,
      phone_verified: subject_token.phone_number_verified,
      username: subject_token.preferred_username,
      name: subject_token.name,
      given_name: subject_token.given_name,
      family_name: subject_token.family_name,
      nickname: subject_token.nickname,
      verify_email: false
    },
    {
      creationBehavior: 'create_if_not_exists',
      updateBehavior: 'none'
    }
  );

  return;
};
```

<div id="apiauthenticationsetactoractor">
  ### `api.authentication.setActor(actor)`
</div>

Définit l’acteur pour l’échange de jetons afin de représenter l’entité qui agit au nom du sujet.
Doit être utilisé avec les commandes setUserById ou SetUserByConnection. L’appel à setActor est facultatif.
Le fait de recevoir un actor&#95;token dans la requête ne produit pas automatiquement de claim act; l’Action doit appeler explicitement cette méthode.
Aucun jeton d’actualisation n’est émis lorsqu’un acteur est défini pour la transaction.

<ResponseField name="actor" type="actorparams">
  Un objet imbriqué représentant une chaîne de délégation. Jusqu’à 4 niveaux act supplémentaires sont autorisés
  (5 acteurs au total, y compris l’acteur racine). Pour chaque niveau, le champ `sub` est requis; jusqu’à 5 propriétés
  personnalisées supplémentaires (valeurs `string`, `boolean` ou `number`) peuvent être fournies.

  <Expandable title="propriétés de l’acteur" defaultOpen>
    <ResponseField name="sub" type="string" />

    <ResponseField name="act" type="dictionary" post={["optional"]}>
      <Expandable title="propriétés de act" defaultOpen>
        <ResponseField name="sub" type="string" />

        <ResponseField name="act" type="dictionary" post={["optional"]}>
          <Expandable title="propriétés de act" defaultOpen>
            <ResponseField name="sub" type="string" />

            <ResponseField name="act" type="dictionary" post={["optional"]}>
              <Expandable title="propriétés de act">
                <ResponseField name="sub" type="string" />

                <ResponseField name="act" type="dictionary" post={["optional"]}>
                  <Expandable title="propriétés de act">
                    <ResponseField name="sub" type="string" />

                    <ResponseField name="act" type="undefined" post={["optional"]} />
                  </Expandable>
                </ResponseField>
              </Expandable>
            </ResponseField>
          </Expandable>
        </ResponseField>
      </Expandable>
    </ResponseField>
  </Expandable>
</ResponseField>

```js
exports.onExecuteCustomTokenExchange = async (event, api) => {

  // 1. Valider le subject_token
  const subject_token = await validateToken(event.transaction.subject_token, jwksUri);
  const actor_token = await validateToken(event.transaction.actor_token, jwksUri);

  // 2. Définir l'acteur pour la transaction
  api.authentication.setActor({ sub: actor_token.sub });

  // 3. Définir l'utilisateur pour la transaction
  api.authentication.setUserById(subject_token.sub);

  return;
};
```

<div id="apicachesetkey-value-options">
  ### `api.cache.set(key, value, options)`
</div>

Stocke ou met à jour une chaîne de caractères dans le cache à la clé spécifiée.

Les valeurs stockées dans ce cache sont propres au Trigger dans lequel
elles sont définies. Elles sont assujetties aux [Actions Cache Limits](https://auth0.com/docs/customize/actions/limitations).

Les valeurs stockées de cette façon auront une durée de vie pouvant aller *jusqu&#39;à* la
valeur `ttl` ou `expires_at` spécifiée. Si aucune durée de vie n&#39;est spécifiée, une
durée de vie par défaut de 15 minutes sera utilisée. Les durées de vie ne peuvent pas dépasser la
durée maximale indiquée dans [Actions Cache Limits](https://auth0.com/docs/customize/actions/limitations).

**Important** : Ce cache est conçu pour des données éphémères de courte durée. Il se peut que des éléments ne soient pas
disponibles dans des transactions ultérieures, même s&#39;ils sont encore dans leur durée de vie.

<ResponseField name="key" type="string">
  La clé de l&#39;enregistrement à stocker.
</ResponseField>

<ResponseField name="value" type="string">
  La valeur de l&#39;enregistrement à stocker.
</ResponseField>

<ResponseField name="options" type="cachesetoptions" post={["optional"]}>
  Options pour ajuster le comportement du cache.

  <Expandable title="propriétés des options" defaultOpen>
    <ResponseField name="expires_at" type="number" post={["optional"]}>
      L&#39;heure d&#39;expiration absolue en millisecondes depuis l&#39;époque Unix.
      Bien que les enregistrements mis en cache puissent être évincés plus tôt, ils ne
      demeureront jamais au-delà de la valeur `expires_at` fournie.

      *Remarque* : Cette valeur ne doit pas être fournie si une valeur a aussi été
      fournie pour `ttl`. Si les deux options sont fournies, la date
      d&#39;expiration la plus rapprochée des deux sera utilisée.
    </ResponseField>

    <ResponseField name="ttl" type="number" post={["optional"]}>
      La durée de vie de cette entrée de cache en millisecondes.
      Bien que les valeurs mises en cache puissent être évincées plus tôt, elles ne
      demeureront jamais au-delà de la valeur `ttl` fournie.

      *Remarque* : Cette valeur ne doit pas être fournie si une valeur a aussi été
      fournie pour `expires_at`. Si les deux options sont fournies, la date
      d&#39;expiration la plus rapprochée des deux sera utilisée.
    </ResponseField>
  </Expandable>
</ResponseField>
