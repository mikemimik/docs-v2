<Warning>
  Si vous devez configurer un domaine personnalisé, faites-le **avant** de déployer WebAuthn dans un environnement de production. Si vous configurez ou modifiez un domaine personnalisé, les utilisateurs déjà inscrits ne pourront pas s’authentifier.

  Vous pouvez utiliser le paramètre [Relying Party](#configure-relying-party) pour préciser le domaine utilisé pour authentifier les utilisateurs.
</Warning>

<div id="use-the-dashboard">
  ## Utiliser le Dashboard
</div>

1. Activez **WebAuthn with clés de sécurité** en accédant à [Dashboard &gt; Security &gt; Multi-factor Auth](https://manage.auth0.com/#/security/mfa).
2. Configurez la façon dont vous souhaitez gérer la user verification. Pour les clés de sécurité, la user verification consiste généralement à demander aux utilisateurs d’entrer un NIP pour terminer le WebAuthn challenge.

   1. Never : Les utilisateurs ne seront jamais invités à entrer un NIP. Il s’agit de la valeur par défaut, et elle est habituellement suffisante lorsque vous utilisez des clés de sécurité pour le MFA. Les utilisateurs ont déjà entré leur mot de passe et ont donc déjà fourni un certain niveau de vérification.
   2. If supported : Les utilisateurs seront invités à entrer un NIP s’ils en ont déjà configuré un sur la key.
   3. Required : Les utilisateurs devront définir un NIP s’il n’est pas déjà configuré, puis l’entrer chaque fois. C’est l’option qui offre le niveau de sécurité le plus élevé. Certains navigateurs ne l’implémentent pas correctement (par exemple, [Brave](https://brave.com/) sur iOS), donc l’authentification échouera et Auth0 demandera aux utilisateurs d’utiliser un autre navigateur.

Notez que seules les clés de sécurité conformes à FIDO-2 prennent en charge la user verification. Les keys FIDO-1 peuvent être utilisées avec WebAuthn, mais elles ne peuvent pas être utilisées si vous définissez User Verification sur Required.

<div id="configure-relying-party">
  ## Configurer le Relying Party
</div>

WebAuthn rend l’hameçonnage impossible en liant les informations d’authentification à l’origine du navigateur. Les utilisateurs ne peuvent pas non plus utiliser WebAuthn pour un site sur lequel ils ne se sont pas inscrits.

Le fait de lier les informations d’authentification à l’origine signifie que, si vous configurez un <Tooltip tip="Custom Domain : domaine tiers avec un nom spécialisé ou personnalisé." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=custom+domain">domaine personnalisé</Tooltip> ou que vous le modifiez, les utilisateurs inscrits avant ce changement ne pourront pas s’authentifier.

WebAuthn définit un [attribut d’identifiant du Relying Party](https://www.w3.org/TR/webauthn-2/#relying-party-identifier), qui vous permet de préciser le domaine utilisé pour authentifier les utilisateurs. Vous pouvez le définir sur n’importe quel suffixe de domaine enregistrable de l’origine du navigateur. Par exemple, si le domaine personnalisé est `login.example.com`, vous pouvez configurer l’identifiant du <Tooltip tip="Relying Party : entité (comme un service ou une application) qui s’appuie sur un fournisseur d’identité tiers pour authentifier un utilisateur." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Relying+Party">Relying Party</Tooltip> sur `example.com`. Les utilisateurs peuvent ainsi s’authentifier sur n’importe quel domaine `example.com` avec leurs informations d’authentification WebAuthn.

Auth0 vous permet de préciser l’identifiant du Relying Party seulement si vous avez configuré un domaine personnalisé. Si le domaine personnalisé change, vous devez mettre à jour l’identifiant du Relying Party.

<div id="device-support">
  ## Prise en charge des appareils
</div>

Pour utiliser des clés de sécurité, un navigateur doit avoir Javascript activé et prendre en charge WebAuthn. Si ces conditions ne sont pas remplies, Auth0 n’offrira pas l’option d’enregistrer ou d’utiliser des clés de sécurité pour l’authentification. Auth0 demandera plutôt une vérification avec un autre facteur ou avec un code de récupération (si aucun autre facteur n’est enregistré).

Les versions les plus récentes des navigateurs et systèmes d’exploitation courants prennent en charge WebAuthn with clés de sécurité. Pour plus de détails, consultez la [section sur la compatibilité des navigateurs sur webauthn.me](https://webauthn.me/browser-support).

  ## Limites
