---

Voici quelques menaces courantes de cybersécurité :

* attaque de type Bucket brigade
* attaques par rejeu
* attaques par falsification de requête intersites (CSRF ou XSRF)
* attaques par force brute
* <Tooltip tip="Suspicious IP Throttling : forme de protection contre les attaques qui protège votre tenant contre les connexions suspectes visant un trop grand nombre de comptes à partir d’une seule adresse IP." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=Suspicious+IP+throttling">Suspicious IP throttling</Tooltip>
* attaques par credential stuffing
* attaques de validation de listes

Auth0 offre des [fonctionnalités de protection contre les attaques](/fr-CA/docs/secure/attack-protection) qui peuvent aider à contrer certains types d’attaques.

<div id="bucket-brigade-attacks">
  ## Attaques de type Bucket brigade
</div>

Un type de menace est l’attaque de type Bucket brigade, parfois appelée attaque de « l’homme du milieu », où l’attaquant s’interpose entre deux parties. Chaque partie croit interagir au moyen d’une connexion privée, mais l’échange est en réalité contrôlé par un attaquant tiers. Pour que ce type d’attaque réussisse, l’attaquant doit établir une authentification mutuelle entre les deux parties.

Habituellement, les attaques de type Bucket brigade impliquent que l’attaquant utilise un routeur WiFi pour intercepter les communications d’un utilisateur. L’utilisateur se connecte au routeur de l’attaquant, puis visite un site Web et se connecte avec ses identifiants confidentiels. L’attaquant enregistre alors les identifiants de connexion de cet utilisateur, qu’il peut ensuite utiliser pour se faire passer pour lui.

Certaines attaques de type Bucket brigade peuvent aussi modifier les données transmises entre l’application et le serveur. Les attaques peuvent provenir de logiciels d’essai sur de nouveaux PC, de logiciels préinstallés, d’outils de mise à jour logicielle et d’autres vulnérabilités logicielles. Dans le pire des cas, cela pourrait entraîner l’exécution de code à distance, l’installation de portes dérobées dans le système, l’installation de logiciels malveillants, etc. Cela pourrait également mener à la compromission d’autres ressources réseau.

Pour aider à se défendre contre ce type d’attaque, il est important d’utiliser un chiffrement robuste et une authentification entre l’application et le serveur. Grâce au chiffrement, le serveur authentifie la requête de l’application en présentant un certificat numérique, et ce n’est qu’ensuite que la connexion peut être établie. Par exemple, HTTPS utilise la technologie SSL (Secure Sockets Layer) du navigateur pour masquer le trafic Web. Pour déchiffrer HTTPS, un attaquant devrait obtenir les clés utilisées pour chiffrer le trafic réseau.

Pour configurer TLS sur vos serveurs, nous vous suggérons d’utiliser les [recommandations OpSec de Mozilla](https://wiki.mozilla.org/Security/Server_Side_TLS), qui utilisent TLSv1.2. Mozilla fournit également un [générateur de configuration SSL](https://mozilla.github.io/server-side-tls/ssl-config-generator/) ; pour l’utiliser, choisissez d’abord la plateforme sur laquelle votre serveur fonctionne, puis la configuration la plus moderne que vos applications peuvent prendre en charge. Les configurations les plus modernes offrent des algorithmes de chiffrement plus robustes pour aider à prévenir les attaques.

<div id="replay-attacks">
  ## Attaques par rejeu
</div>

Les attaques par rejeu (ou par relecture) permettent aux attaquants de :

* Accéder à un réseau et à des renseignements qui n’auraient pas été faciles d’accès
* Effectuer une transaction en double

Il s’agit d’attaques visant le protocole de sécurité, où des données transmises par un autre expéditeur sont rejouées dans le système de réception visé. Ces attaques amènent les participants à croire que la transmission des données s’est déroulée avec succès.

Les attaques par rejeu peuvent être évitées à l’aide de jetons de session. Cependant, si ces informations d’identification sont volées à partir du stockage local (par exemple lors d’une attaque XSS), il existe des moyens d’empêcher qu’une personne conserve indéfiniment un jeton valide :

* Définir une courte durée d’expiration pour les jetons
* Prévoir un moyen d’ajouter à la DenyList les jetons qui ont été utilisés (et possiblement même les utilisateurs)
* Utiliser des mots de passe à usage unique

La spécification [JWT](/fr-CA/docs/secure/tokens/json-web-tokens) fournit le champ `jti` comme moyen de prévenir les attaques par rejeu. Bien qu’à l’heure actuelle, les jetons Auth0 ne renvoient pas le `jti`, vous pouvez ajouter des jetons à la DenyList à l’aide du `jti` afin d’empêcher qu’un jeton soit utilisé plus qu’un nombre de fois déterminé. De cette façon, vous mettez en œuvre quelque chose de semblable à un <Tooltip tip="Nonce : nombre arbitraire émis une seule fois dans un protocole d’authentification pour détecter et prévenir les attaques par rejeu." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=nonce">nonce</Tooltip> (considérez la signature du jeton comme le nonce). Si un jeton est volé ou utilisé plus souvent que le nombre de fois permis, il devrait être ajouté à la DenyList. Cela empêche qu’un jeton valide soit utilisé de façon malveillante. Une fois le jeton expiré, l’attaquant ne pourra plus se faire passer pour l’utilisateur.

Vous pouvez aussi éviter les attaques par rejeu en utilisant des mots de passe à usage unique. Avec Auth0, vous pouvez utiliser l’authentification <Tooltip tip="Passwordless : forme d’authentification qui ne repose pas sur un mot de passe comme premier facteur." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=Passwordless">Passwordless</Tooltip>, qui repose sur des codes à usage unique et des liens envoyés par courriel plutôt que sur des mots de passe traditionnels. Auth0 offre aussi l’<Tooltip tip="Authentification multifacteur (MFA) : processus d’authentification de l’utilisateur qui utilise un facteur en plus du nom d’utilisateur et du mot de passe, comme un code envoyé par SMS." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=multi-factor+authentication">authentification multifacteur</Tooltip> (MFA), qui utilise des mots de passe à usage unique comme deuxième facteur d’authentification pouvant être envoyés par notifications push et par texto.

<div id="cross-site-request-forgery">
  ## Falsification de requête intersites
</div>

Une attaque par falsification de requête intersites (CSRF ou XSRF) se produit lorsqu’un programme malveillant amène le navigateur Web d’un utilisateur à effectuer une action non désirée sur un site de confiance où l’utilisateur est actuellement authentifié. Ce type d’attaque cible précisément les requêtes qui modifient l’état afin de déclencher une action, plutôt que d’obtenir des données utilisateur, puisque l’attaquant n’a aucun moyen de voir la réponse à la requête falsifiée.

Une façon de vérifier les requêtes envoyées consiste à utiliser le <Tooltip tip="OAuth 2.0 : cadre d’autorisation qui définit les protocoles et les flux d’autorisation." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=OAuth+2.0">OAuth 2.0</Tooltip> [paramètre state](/fr-CA/docs/secure/attack-protection/state-parameters) du protocole pour authentifier la réponse.
