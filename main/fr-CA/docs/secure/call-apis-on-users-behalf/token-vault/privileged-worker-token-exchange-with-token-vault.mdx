Token Vault prend en charge le Privileged Worker Token Exchange, qui permet à une application cliente d’échanger un JWT signé (jeton de sujet) contre le jeton d’accès ou d’actualisation d’un fournisseur externe (jeton demandé).

Après l’authentification et l’autorisation réussies de l’utilisateur, une application cliente transmet généralement le contexte utilisateur, qui contient l’identité, les permissions et l’état de la session de l’utilisateur, sous forme de jeton d’accès ou d’actualisation pour effectuer l’échange de jetons avec Token Vault. Dans les flux de service à service, une application cliente, comme une application backend ou un service worker, peut avoir besoin d’accéder à des ressources au nom de l’utilisateur, mais comme « l’utilisateur n’est pas présent » dans une session interactive, l’application cliente n’a pas accès au contexte utilisateur.

Dans ces scénarios de service à service, l’application cliente peut générer un JWT bearer token signé et l’utiliser comme jeton de sujet pour effectuer l’échange de jetons et recevoir les jetons nécessaires pour appeler des API externes. Cela signifie que l’application cliente peut exécuter des actions au nom de l’utilisateur sans interaction ni session utilisateur active.

Pour utiliser Privileged Worker Token Exchange with Token Vault, l’application cliente doit être un client hautement privilégié qui peut aussi demander des jetons d’actualisation à des fournisseurs externes via Token Vault. Elle doit s’authentifier auprès de Token Vault à l’aide de méthodes cryptographiques asymétriques, comme l’assertion [Private Key JWT](/fr-CA/docs/get-started/authentication-and-authorization-flow/authenticate-with-private-key-jwt) ou l’[authentification TLS mutuelle](/fr-CA/docs/get-started/authentication-and-authorization-flow/authenticate-with-mtls).

<div id="prerequisites">
  ## Prérequis
</div>

Seuls certains types de clients peuvent utiliser Privileged Worker Token Exchange with Token Vault :

* Le client doit être un client first-party, c.-à-d. que la propriété `is_first_party` doit être définie sur `true`.
* Le client doit être un client confidentiel doté d’un mécanisme d’authentification valide, c.-à-d. que la propriété `token_endpoint_auth_method` ne doit pas être définie sur `none`.
* Le client doit être conforme à OIDC, c.-à-d. que la propriété `oidc_conformant` doit être définie sur `true`.

Avant de configurer Privileged Worker Token Exchange pour votre application cliente :

1. [Activez le type d’attribution Token Vault](/fr-CA/docs/secure/tokens/token-vault/configure-token-vault#configure-application) pour votre application cliente.
2. Configurez [Private Key JWT](/fr-CA/docs/get-started/authentication-and-authorization-flow/authenticate-with-private-key-jwt) ou [l’authentification TLS mutuelle](/fr-CA/docs/get-started/authentication-and-authorization-flow/authenticate-with-mtls) pour votre application cliente.

<div id="configure-client-application">
  ## Configurer l’application cliente
</div>

Pour configurer l’accès privilégié de l’application cliente à Token Vault, vous devez fournir une clé publique qui servira à vérifier un JWT signé en tant que jeton de sujet.

Comme pour la [configuration de JAR](/fr-CA/docs/get-started/applications/configure-jar#configure-jwt-secured-authorization-requests-jar), vous pouvez définir la clé publique d’accès privilégié à Token Vault lors de la création d’un nouveau client :

```bash lines
POST https://{yourDomain}.auth0.com/api/v2/clients
Authorization: Bearer <YOUR_MANAGEMENT_API_ACCESS_TOKEN>
Content-Type: application/json
{
  "name": "My App using JAR",
   “grant_types”: [“urn:auth0:params:oauth:grant-type:token-exchange:federated-connection-access-token”],
     “oidc_conformant”: true,
           “is_first_party”: true,
           “jwt_configuration”: {
             “alg”: 'RS256',
           },

  "token_vault_privileged_access": {
"credentials": [{
        "name": "My credential for Token Vault Privileged Access",
        "credential_type": "public_key",
        "pem": "<YOUR PEM FILE CONTENT>",
        "alg": "RS256"
}]
  },
}
```

Vous pouvez également mettre à jour un client existant à l’aide de la clé publique d’accès privilégié de Token Vault :

```bash lines
PATCH https://{yourDomain}.auth0.com/api/v2/clients/{yourClientId}
Authorization: Bearer <YOUR_MANAGEMENT_API_ACCESS_TOKEN>
Content-Type: application/json
{
  "token_vault_privileged_access": {
    "credentials": [{"id": "<YOUR CREDENTIAL ID>"}]
  }
}
```

<div id="create-signed-jwt-subject-token">
  ## Créer un jeton de sujet JWT signé
</div>

Après [avoir configuré votre application cliente avec la clé publique](#configure-client-application), vous devez créer le jeton de sujet qui sera échangé contre un jeton d’accès pour une API externe. Le jeton de sujet est un JSON Web Token (JWT) contenant les assertions nécessaires. Il est signé à l’aide de la clé privée.

Le JWT a un format standard et des assertions standard, où :

* Le `typ` de l’en-tête est `token-vault-req+jwt`
* Le `kid` de l’en-tête est facultatif si vous n’avez configuré qu’une seule clé publique
* Le `sub` de la charge utile correspond à l’identifiant de l’utilisateur pour lequel vous voulez obtenir le jeton
* Le `aud` de la charge utile est l’hôte de votre tenant
* Le `iss` de la charge utile est l’identifiant client qui effectue la requête

Voici un exemple de JWT :

```json lines
{
    alg: "RS256"  
    typ: "token-vault-req+jwt"
}
.
{
    sub: "auth0|000012030101231",
    aud: "https://{yourDomain}.auth0.com/",
    iss: "<YOUR_CLIENT_ID>",
    iat: 1758799540,
    exp: 1758800540,
    nbf: 1758799540
}
```

L’exemple de code suivant est un script qui génère un JWT signé servant de jeton de sujet :

```tsx lines
import * as jwt from 'jsonwebtoken';
   const privateKey = ‘-----BEGIN RSA PRIVATE KEY-----........’;
   const subjectToken = jwt.sign(
     {
       iss: CLIENT_ID,
       aud: 'https://' + TENANT_DOMAIN + '/',
       sub: USER_ID,
     },
     privateKey,
     {
       algorithm: 'RS256',
       header: {
         typ: 'token-vault-req+jwt',
       },
     }
   );
```

<div id="request-token-for-external-api">
  ## Demander un jeton d’accès pour l’API externe
</div>

Une fois que vous avez le JWT signé, vous pouvez faire une requête afin d’obtenir le jeton d’accès pour l’API externe :

```bash lines
curl --request POST 'https://{yourDomain}/oauth/token' \
--header 'Content-Type: application/json' \
--data '{
  "client_id": "<YOUR_CLIENT_ID>",
  "client_secret": "<YOUR_CLIENT_SECRET>",
  "subject_token": "<YOUR_SIGNED_JWT_BEARER>",
  "grant_type": "urn:auth0:params:oauth:grant-type:token-exchange:federated-connection-access-token",
  "subject_token_type": "urn:ietf:params:oauth:token-type:jwt",
  "requested_token_type": "http://auth0.com/oauth/token-type/token-vault-access-token",
  "connection": "google-oauth2"
}'
```

| Paramètre              | Description                                                                                                                                      |
| ---------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------ |
| `grant_type`           | Le type d’attribution. Pour Token Vault, définissez-le sur `urn:auth0:params:oauth:grant-type:token-exchange:federated-connection-access-token`       |
| `client_id`            | ID de l’application cliente                                                                                                                      |
| `client_secret`        | Secret du client. **Remarque :** Pour Privileged Worker Token Exchange, nous recommandons d’utiliser Private Key JWT ou l’authentification mTLS. |
| `subject_token_type`   | Type du jeton de sujet. Pour Privileged Worker Token Exchange, définissez-le sur JWT : `urn:ietf:params:oauth:token-type:jwt`                     |
| `subject_token`        | Le JWT bearer token signé que l’Auth0 Authorization Server valide pour identifier l’utilisateur.                                                 |
| `requested_token_type` | Le type de jeton demandé. Pour Privileged Worker Token Exchange, vous pouvez demander un jeton d’accès ou un jeton d’actualisation.              |
| `connection`           | Le nom de la connexion, dans ce cas-ci, `google-oauth2`.                                                                                         |

Vous devriez recevoir le jeton d’accès stocké dans Token Vault. Vous pouvez aussi demander le jeton d’actualisation pour l’API externe :

```bash lines
curl --request POST 'https://{yourDomain}/oauth/token' \
--header 'Content-Type: application/json' \
--data '{
  "client_id": "<YOUR_CLIENT_ID>",
  "client_secret": "<YOUR_CLIENT_SECRET>",
  "subject_token": "<YOUR_SIGNED_JWT_BEARER>",
  "grant_type": "urn:auth0:params:oauth:grant-type:token-exchange:federated-connection-access-token",
  "subject_token_type": "urn:ietf:params:oauth:token-type:jwt",
  "requested_token_type": "http://auth0.com/oauth/token-type/token-vault-refresh-token",
  "connection": "google-oauth2"
}'
```
