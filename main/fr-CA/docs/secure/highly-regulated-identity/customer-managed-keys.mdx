L’utilisation d’Auth0 Customer Managed Keys vous permet de configurer le cycle de vie de votre Tenant Master Key et d’utiliser votre propre Customer Provided Root Key pour remplacer l’Environment Root Key de votre tenant Auth0.

Customer Managed Keys offre deux méthodes pour gérer vos clés :

* **Control Your Own Key:** Permet aux utilisateurs ayant le [Key Management Editor role](/fr-CA/docs/get-started/manage-dashboard-access/add-dashboard-users) de personnaliser le cycle de vie de la Tenant Master Key dans Auth0 KMS.
* **Bring Your Own Key:** Permet aux utilisateurs ayant le [Key Management Editor role](/fr-CA/docs/get-started/manage-dashboard-access/add-dashboard-users) de remplacer l’Auth0 Environment Root Key et d’importer leur propre Wrapped Encryption Key dans le Hardware Security Module (HSM) d’Auth0 Cloud correspondant.

Pour en savoir plus sur les clés d’Auth0, [consultez la key hierarchy d’Auth0](/fr-CA/docs/secure/highly-regulated-identity/customer-managed-keys#auth0-key-hierarchy).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  La fonctionnalité Customer Managed Keys est offerte aux utilisateurs ayant le Key Management Editor role. Ce rôle n’est pas attribué aux utilisateurs par défaut et doit être accordé explicitement à un tenant member. Pour en savoir plus, consultez [Add Tenant Members](/fr-CA/docs/get-started/manage-dashboard-access/add-dashboard-users).
</Callout>

<div id="control-your-own-key">
  ## Control Your Own Key
</div>

Vous pouvez gérer le cycle de vie de votre Tenant Master Key à l’aide du point de terminaison [Rekey](https://auth0.com/docs/api/management/v2/keys/post-encryption-rekey) de l’Auth0 <Tooltip tip="Management API : un produit permettant aux clients d’effectuer des tâches administratives." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=Management+API">Management API</Tooltip> pour :

* Faire la rotation de l’ancienne Tenant Master Key en la remplaçant par une Tenant Master Key nouvellement créée.
* Faire la rotation des Namespace Keys et les rechiffrer avec la nouvelle Tenant Master Key.

<div id="permissions">
  ### Permissions
</div>

Utilisez les permissions suivantes pour accorder l’accès aux endpoints Rekey :

| Permissions                                       | Descriptions                                                                                                                                            |
| ------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `create:encryption_keys` `update:encryption_keys` | Les utilisateurs peuvent effectuer la rotation et rechiffrer la hiérarchie de clés d’un tenant Auth0 au moyen de l’endpoint Rekey de la Management API. |

<div id="endpoint">
  ### Point de terminaison
</div>

Utilisez le point de terminaison [Rekey](https://auth0.com/docs/api/management/v2/keys/post-encryption-rekey) de la Management API pour faire la rotation de la Tenant Master Key, ainsi que la rotation et le rechiffrement des Namespace Keys.

* Faire la rotation de la Tenant Master Key : désactive la Tenant Master Key actuellement active et crée une nouvelle Tenant Master Key.
* Faire la rotation de toutes les Namespace Keys du tenant : désactive les clés actuellement actives et active de nouvelles clés.

  * Les nouvelles Namespace Keys sont utilisées pour les nouvelles opérations de chiffrement.
  * Les Namespace Keys désactivées sont utilisées pour déchiffrer les données chiffrées précédemment.
* Rechiffre toutes les Namespace Keys existantes avec la nouvelle Tenant Master Key.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Le point de terminaison est accessible uniquement aux membres du tenant disposant de scopes d’autorisation précis dans Auth0 ; par défaut, ces scopes sont accordés aux utilisateurs ayant le [Key Management Editor role](/fr-CA/docs/get-started/manage-dashboard-access/add-dashboard-users).
</Callout>

<div id="bring-your-own-key">
  ## Bring Your Own Key
</div>

<Warning>
  En important votre propre Customer Provided Root Key avec Bring Your Own Key, vous retirez implicitement à Auth0 l’autorisation de gérer le cycle de vie de la Customer Provided Root Key, à l’exception de sa suppression.
</Warning>

Bring Your Own Key permet aux Key Management Editors d’importer de façon sécuritaire une Wrapped Encryption Key (Customer Provided Root Key) dans un Hardware Security Module (HSM) FIPS 140-2 L3 du Auth0 Cloud correspondant.

En utilisant Bring Your Own Key, vous pouvez :

* Remplacer l’Environment Root Key générée par défaut par Auth0 par une nouvelle Customer Provided Root Key.
* Effectuer une rotation et rechiffrer la key hierarchy avec la Customer Provided Root Key. Par exemple : créer et rechiffrer une nouvelle Tenant Master Key ainsi qu’une nouvelle Namespace Key.

Vous pouvez utiliser le [Dashboard](/fr-CA/docs/secure/highly-regulated-identity/customer-managed-keys/customer-managed-keys-dashboard) ou la [Management API](/fr-CA/docs/secure/highly-regulated-identity/customer-managed-keys/customer-managed-keys-management-api) pour activer Bring Your Own Key.

<div id="monitor-customer-managed-keys-log-events">
  ## Surveiller les événements de journalisation de Customer Managed Keys
</div>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Auth0 fait automatiquement la rotation des clés de chiffrement du tenant une fois par année, en ajoutant les événements de journalisation suivants dans les journaux du tenant :

  * `kms_key_state_changed`
  * `kms_key_management_success`
</Callout>

Les opérations de Customer Managed Keys ajoutent les événements de journalisation suivants dans vos [journaux du tenant](/fr-CA/docs/deploy-monitor/logs) :

Un code d’événement `sapi` indiquant :

* la création de la nouvelle clé de chiffrement
* la création de la clé d’enveloppement publique
* l’importation de la clé de chiffrement
* la suppression de la clé de chiffrement selon son ID de clé
* le renouvellement de la hiérarchie de clés

Un code d’événement `kms_key_management_success` indiquant une opération KMS réussie.

Un code d’événement `kms_key_management_failure` indiquant l’échec d’une opération KMS.

Un code d’événement `kms_key_state_changed` indiquant un changement d’état d’une clé KMS.

<div id="auth0-key-hierarchy">
  ## Hiérarchie des clés d’Auth0
</div>

Dans la couche applicative d’Auth0, Auth0 protège les secrets et les données des clients au moyen du chiffrement par enveloppe.

La hiérarchie de chiffrement par enveloppe d’Auth0 se compose des clés suivantes, chacune étant chiffrée à l’aide de la clé qui la précède. Le tableau ci-dessous résume la hiérarchie des clés :

| Clé                            | Algorithme                                                       | Stockage                                  |
| ------------------------------ | ---------------------------------------------------------------- | ----------------------------------------- |
| Clé racine d’environnement     | RSA 2048 OAEP (Auth0 sur Azure)<br />AES-256-GCM (Auth0 sur AWS) | Module de sécurité matériel FIPS 140-2 L3 |
| Clé maître du tenant           | AES-256-GCM                                                      | Base de données Auth0 KMS                 |
| Clé d’espace de noms           | AES-256-GCM                                                      | Base de données Auth0 KMS                 |
| Clé de chiffrement des données | AES-256-GCM                                                      | Stockée avec les données                  |

<div id="environment-root-key">
  ### Environment Root Key
</div>

L’Environment Root Key représente le sommet de la hiérarchie et sert à envelopper la Tenant Master Key afin d’éviter qu’elle ne soit divulguée ou altérée à l’extérieur d’Auth0.

Une Auth0 Environment Root Key distincte est générée pour chaque environnement Auth0 et stockée dans un HSM adjacent. Les HSM sont déployés dans une configuration hautement disponible répartie sur plusieurs zones géographiques. Cela signifie qu’en cas d’incident majeur touchant toute une région, les HSM basculeront vers une autre région.

L’Auth0 Environment Root Key est partagée entre tous les tenants. Les clients peuvent utiliser la fonctionnalité Bring Your Own Key pour disposer d’une Environment Root Key dédiée à leur tenant.

Auth0 utilise les algorithmes suivants pour envelopper la Tenant Master Key avec l’Environment Root Key, selon votre Auth0 Cloud Service Provider :

* Auth0 on Azure : RSA 2048 OAEP
* Auth0 on AWS : AES 256 GCM

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  À l’aide de l’Auth0 Dashboard ou de la Management API, les Admins du tenant peuvent remplacer l’Auth0 Environment Root Key par leur propre Customer Provided Root Key.
</Callout>

<div id="tenant-master-keys">
  ### Tenant Master Keys
</div>

Chaque tenant possède une Tenant Master Key chiffrée, stockée dans le Auth0 Key Management Service, qui chiffre les Namespace Keys.

L’algorithme utilisé pour chiffrer la Tenant Master Key est AES256 GCM.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Lorsqu’un Admin de tenant utilisant le Auth0 Dashboard ou la Management API fournit sa propre Customer Provided Root Key, une nouvelle Tenant Master Key est créée.
</Callout>

<div id="namespace-keys">
  ### Clés d’espace de noms
</div>

Les clés d’espace de noms servent à séparer les clés de chiffrement utilisées à différentes fins au sein du tenant. Le nombre de clés d’espace de noms et leur usage sont configurés en interne par Auth0 et ne peuvent pas être personnalisés.

Le chiffrement et le déchiffrement des clés d’espace de noms nécessitent l’accès à la clé maître du tenant. Les clés d’espace de noms ne quittent jamais l’Auth0 Key Management Service et ne sont accessibles ni aux développeurs ni aux administrateurs.

Les clés d’espace de noms se trouvent dans l’Auth0 Key Management Service et sont chiffrées à l’aide de l’algorithme  AES256 GCM.

<div id="data-encryption-keys">
  ### Clés de chiffrement des données
</div>

Auth0 Key Management Service génère de façon sécurisée différentes clés de chiffrement des données pour les requêtes de chiffrement de données ultérieures. Auth0 Key Management Service optimise la sécurité et les performances en générant régulièrement de nouvelles clés de chiffrement des données.

Le chiffrement et le déchiffrement des clés de chiffrement des données exigent l’accès à la Namespace Key qui leur est attribuée. Les clés de chiffrement des données ne peuvent pas être déchiffrées à l’extérieur d’Auth0 Key Management Service ni par les Key Management Editors.

Les clés de chiffrement des données sont stockées à proximité des données et sont chiffrées à l’aide de l’algorithme AES256 GCM.
