***

La fonctionnalité Tenant Access Control List (ACL) gère le trafic vers les ressources de votre tenant en fonction des règles que vous définissez.

L’utilisation de la Tenant ACL peut vous aider à ménager vos limites de taux face à des menaces potentielles (comme les attaques par déni de service) et à faire en sorte que seuls les utilisateurs légitimes accèdent à vos applications.

<div id="overview">
  ## Aperçu
</div>

Chaque Tenant ACL se compose d’un ensemble de règles. Lorsque votre tenant ou l’<Tooltip tip="Management API : un produit permettant aux clients d’effectuer des tâches administratives." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=Management+API">Management API</Tooltip> reçoit une requête, les règles de votre Tenant ACL déterminent la façon dont la requête est traitée.

Les options de configuration granulaires de Tenant ACL vous permettent d’appliquer des règles à différents types de signaux et de scopes. Par exemple :

* Vous pouvez utiliser le [`hostnames` scope](./tenant-access-control-list/reference#param-hostnames) pour désactiver ou restreindre l’accès à votre domaine canonique, tout en autorisant le trafic via des domaines personnalisés.

* Si vous adoptez les Model Context Protocols (MCP) dans votre tenant, vous pouvez utiliser le [Dynamic Client Registration (`dcr`) scope](./tenant-access-control-list/reference#param-dcr) pour réduire les risques, comme l’enregistrement non autorisé d’applications ou les tentatives d’hameçonnage au moyen de noms d’application trompeurs.

À des fins de surveillance et d’audit, un [tenant log event](./tenant-access-control-list/tenant-log-reference) (`acls_summary`) est créé toutes les 10 minutes pour chaque règle de Tenant ACL, avec des détails sur son incidence sur le trafic. Vous pouvez activer le mode de surveillance pour les règles afin de tester et de consigner leur comportement sans appliquer réellement la règle au trafic.
