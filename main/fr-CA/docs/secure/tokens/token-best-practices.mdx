description: Énumère les bonnes pratiques à suivre lors de l’utilisation de jetons dans l’authentification et l’autorisation.
title: Token Best Practices
---

Voici quelques points de base à garder à l’esprit lorsque vous utilisez des jetons :

* **Gardez-la secrète. Gardez-la en lieu sûr** : La clé de signature doit être traitée comme toute autre donnée d’authentification et ne doit être divulguée qu’aux services qui en ont besoin.
* **N’ajoutez pas de données sensibles à la charge utile** : Les jetons sont signés pour les protéger contre l’altération et peuvent être décodés facilement. Ajoutez le minimum de claims à la charge utile pour optimiser les performances et la sécurité.
* **Définissez une expiration pour les jetons** : Techniquement, une fois qu’un jeton est signé, il reste valide indéfiniment, à moins que la clé de signature ne soit modifiée ou qu’une date d’expiration soit explicitement définie. Cela peut poser certains problèmes ; prévoyez donc une stratégie pour faire expirer et/ou révoquer les jetons.
* **Utilisez HTTPS** : N’envoyez pas de jetons sur des connexions non HTTPS, car ces requêtes peuvent être interceptées et les jetons compromis.
* **Tenez compte de tous vos cas d’utilisation en matière d’autorisation** : Il peut être nécessaire d’ajouter un système secondaire de vérification des jetons pour s’assurer qu’ils ont bien été générés par votre serveur afin de répondre à vos exigences.
* **Stockez et réutilisez :** Réduisez les allers-retours inutiles qui augmentent la surface d’attaque de votre application, et optimisez les limites de jetons de votre plan (le cas échéant) en stockant les <Tooltip tip="Jeton d’accès : information d’autorisation, sous la forme d’une chaîne opaque ou d’un JWT, utilisée pour accéder à une API." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=access+tokens">jetons d’accès</Tooltip> obtenus auprès du <Tooltip tip="Serveur d’autorisation : serveur centralisé qui aide à définir les limites de l’accès d’un utilisateur. Par exemple, votre serveur d’autorisation peut contrôler les données, les tâches et les fonctionnalités accessibles à un utilisateur." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=authorization+server">serveur d’autorisation</Tooltip>. Au lieu de demander un nouveau jeton, utilisez le jeton stocké lors des appels ultérieurs jusqu’à son expiration. La façon de stocker les jetons dépend des caractéristiques de votre application : les solutions courantes comprennent les bases de données (pour les applications qui doivent effectuer des appels d’API, même en l’absence de session) et les sessions HTTP (pour les applications dont la fenêtre d’activité est limitée à une session interactive). Pour un exemple de stockage côté serveur et de réutilisation des jetons, consultez [Stockage des jetons](/fr-CA/docs/secure/security-guidance/data-security/token-storage).

<div id="tokens-vs-cookies">
  ## Jetons vs cookies
</div>

En règle générale, les applications monopages (comme React, Vue et AngularJS + Node), les applications mobiles natives (comme iOS et Android) et les API Web (écrites en Node, Ruby, ASP.NET ou dans un mélange de ces langages) bénéficient le plus de l’authentification par jeton. Les applications Web traditionnelles côté serveur ont, quant à elles, traditionnellement utilisé l’authentification par cookie.

L’authentification par jeton consiste à générer un jeton lorsque l’utilisateur s’authentifie, puis à inclure ce jeton dans l’en-tête `Authorization` de chaque requête subséquente vers votre API. Idéalement, ce jeton devrait être dans un format standard, comme les <Tooltip tip="JSON Web Token (JWT) : format standard d’ID Token (et souvent d’Access Token) utilisé pour représenter de façon sécurisée des revendications entre deux parties." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=JSON+web+tokens">JSON Web Tokens</Tooltip>, puisque vous trouverez des bibliothèques pour la plupart des plateformes et que vous ne voudrez pas implémenter votre propre chiffrement.

Avec les deux approches, vous pouvez obtenir la même quantité d’information sur l’utilisateur. Cela est déterminé par le paramètre `scope` envoyé dans la requête de login (soit au moyen de Lock, notre bibliothèque JavaScript, soit d’un simple lien). Le `scope` est un paramètre de la méthode `.signin({scope: 'openid name email'})`, qui se retrouve ensuite dans la chaîne de requête de la requête de login.

Par défaut, nous utilisons `scope=openid` dans l’authentification par jeton pour éviter d’avoir un jeton trop volumineux. Vous pouvez contrôler les revendications standard d’<Tooltip tip="OpenID : norme ouverte d’authentication qui permet aux applications de vérifier l’identité des utilisateurs sans recueillir ni stocker d’informations de login." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=OpenID">OpenID</Tooltip> Connect (OIDC) que vous souhaitez inclure dans le jeton en les ajoutant comme valeurs de portée. Par exemple, `scope=openid name email family_name address phone_number`. Pour en savoir plus, consultez [Standard Claims on openid.net](https://openid.net/specs/openid-connect-core-1_0.html#StandardClaims).

Vous pouvez combiner l’authentification par jeton avec l’authentification par cookie. Gardez à l’esprit que les cookies fonctionnent très bien si l’application Web et l’API sont servies à partir du même domaine; vous n’aurez donc peut-être pas besoin d’une authentification par jeton. Si nécessaire, nous retournons aussi un JWT dans le flux de l’application Web. Chacun de nos SDKs le fait différemment. Si vous souhaitez appeler vos API à partir de JavaScript (au lieu d’utiliser le cookie existant), vous devez alors définir les jetons d’accès à l’aide de Web Workers ou de fermetures JavaScript afin de gérer la transmission et le stockage des jetons. Pour en savoir plus, consultez la section sur les scénarios en mémoire du navigateur de notre page [Stockage des jetons](/fr-CA/docs/secure/security-guidance/data-security/token-storage).

<div id="refresh-token-usage">
  ## Utilisation du jeton d’actualisation
</div>

Vous ne pouvez obtenir un <Tooltip tip="Refresh Token : jeton utilisé pour obtenir un nouvel Access Token sans obliger les utilisateurs à se reconnecter." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=Refresh+token">jeton d’actualisation</Tooltip> que si vous mettez en œuvre les flows suivants :

* [Authorization Code Flow](/fr-CA/docs/get-started/authentication-and-authorization-flow/authorization-code-flow/add-login-auth-code-flow)
* [Authorization Code Flow with Proof Key for Code Exchange (PKCE)](/fr-CA/docs/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce/add-login-using-the-authorization-code-flow-with-pkce)
* [Resource Owner Password Flow](/fr-CA/docs/get-started/authentication-and-authorization-flow/resource-owner-password-flow)
* [Device Authorization Flow](/fr-CA/docs/get-started/authentication-and-authorization-flow/device-authorization-flow)

Si vous limitez l’accès hors ligne à votre API, au moyen de la protection configurée avec le commutateur **Allow Offline Access** dans [Auth0 Dashboard &gt; Applications &gt; APIs &gt; Settings](https://manage.auth0.com/#/apis), Auth0 ne renverra pas de jeton d’actualisation pour l’API (même si vous incluez la portée `offline_access` dans votre requête).

Les Rules s’exécutent lors du jeton d’actualisation exchange. Pour exécuter une logique particulière, vous pouvez consulter la propriété `context.protocol` dans votre rule. Si la valeur est `oauth2-refresh-token`, cela signifie que la rule s’exécute pendant l’échange.

Lorsque vous tentez d’obtenir un jeton d’actualisation, le paramètre <Tooltip tip="Audience : identifiant unique de l’audience d’un jeton émis. Nommée aud dans un jeton, sa valeur contient l’ID soit d’une application (Client ID) pour un ID Token, soit d’une API (API Identifier) pour un Access Token." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=audience">audience</Tooltip> n’est pas disponible dans le context object des Rules. Si vous recevez une erreur lorsque vous tentez d’ajouter le paramètre audience, vérifiez qu’il n’est pas défini dans le jeton.

Si vous essayez d’effectuer une redirection avec `context.redirect`, le flux d’authentification retournera une erreur.

Si vous avez ajouté des claims personnalisées à vos jetons à l’aide d’une rule, elles apparaîtront dans les nouveaux jetons émis lorsque vous utilisez un jeton d’actualisation, tant que votre rule est en place. Bien que les nouveaux jetons n’héritent pas automatiquement des claims personnalisées, les rules s’exécutent pendant le jeton d’actualisation flow; le même code sera donc exécuté. Cela vous permet d’ajouter ou de modifier des claims personnalisées dans les jetons nouvellement émis sans obliger les applications déjà autorisées à obtenir un nouveau jeton d’actualisation.

<div id="refresh-token-limits">
  ### Limites des jetons d’actualisation
</div>

Auth0 limite à 200 le nombre de jetons d’actualisation actifs par utilisateur et par application. Cette limite s’applique uniquement aux jetons actifs. Si la limite est atteinte et qu’un nouveau jeton d’actualisation est créé, le système révoque et supprime le jeton le plus ancien pour cet utilisateur et cette application. Les jetons révoqués et expirés ne sont pas pris en compte dans cette limite.

<div id="automated-tests">
  #### Tests automatisés
</div>

Les jetons d’actualisation s’accumulent à cause des tests automatisés et sont généralement utilisés pendant toute la durée des tests. Pour éviter d’accumuler un trop grand nombre de jetons soumis aux limites des jetons d’actualisation, vous pouvez utiliser l’Auth0 <Tooltip tip="Management API : un produit qui permet aux clients d’effectuer des tâches administratives." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=Management+API">Management API</Tooltip> pour supprimer les jetons d’actualisation inutiles.

1. [Créer un utilisateur](https://auth0.com/docs/api/management/v2#!/Users/post_users) avec la Management API. Vous utiliserez cet utilisateur pour les tests.
2. La réponse renvoie un `user_id` que vous devez enregistrer pendant les tests afin de l’utiliser plus tard.
3. Une fois les tests terminés, [supprimez l’utilisateur](https://auth0.com/docs/api/management/v2#!/Users/delete_users_by_id) au moyen de la Management API. Lorsque l’utilisateur de test est supprimé, les éléments associés le sont aussi, y compris les jetons d’actualisation.

<Warning>
  Pour ce cas d’utilisation, nous ne recommandons pas d’utiliser un ID utilisateur statique. Nous ne recommandons pas non plus de conserver les utilisateurs de test et les éléments associés, ni de supprimer les jetons d’actualisation à l’aide des points de terminaison des informations d’identification d’appareil, car vous pourriez atteindre les limites de débit de la Management API. Pour en savoir plus, consultez [les limites de débit des points de terminaison de la Management API.](/fr-CA/docs/troubleshoot/customer-support/operational-policies/rate-limit-policy/management-api-endpoint-rate-limits)
</Warning>

Si vous souhaitez conserver l’utilisateur de test pour de futurs tests :

1. Répertoriez les jetons d’actualisation de l’utilisateur à l’aide du [point de terminaison des informations d’identification d’appareil de la Management API](https://auth0.com/docs/api/management/v2#!/Device_Credentials/get_device_credentials). Le point de terminaison renverra un maximum de 1000 jetons, sans ordre particulier, peu importe le nombre de jetons accumulés ou l’utilisation de la pagination.
2. Supprimez ces informations d’identification [à l’aide de la méthode DELETE](https://auth0.com/docs/api/management/v2#!/Device_Credentials/delete_device_credentials_by_id).
3. Si l’utilisateur a plus de 1k jetons, répétez l’opération de répertorier et de supprimer les jetons jusqu’à ce qu’il n’en reste plus pour cet utilisateur.

<div id="configure-expiring-refresh-tokens">
  #### Configurer l’expiration des jetons d’actualisation
</div>

Lorsque des utilisateurs se connectent à votre application avec Auth0 et que `offline_access` est demandé dans la demande d’autorisation, un nouveau jeton d’actualisation leur est émis. S’ils se déconnectent puis se reconnectent sur le même appareil, un nouveau jeton d’actualisation leur est de nouveau émis. Selon la façon dont votre application stocke et utilise les jetons d’actualisation, l’ancien jeton d’actualisation de la première connexion peut devenir obsolète, et votre application utilisera très probablement les nouveaux jetons d’actualisation si les deux jetons sont émis pour la même audience. Pour en savoir plus, consultez [Stockage des jetons](/fr-CA/docs/secure/security-guidance/data-security/token-storage).

Pour éviter l’accumulation de jetons d’actualisation obsolètes, même si la limite de jetons supprime d’abord le plus ancien, nous vous recommandons de configurer l’expiration des jetons d’actualisation. Les jetons d’actualisation rotatifs et non rotatifs (ou réutilisables) peuvent être configurés pour expirer selon un délai d’inactivité ou une durée de vie absolue. Ces deux valeurs d’expiration aident à supprimer les jetons qui ne sont plus activement utilisés et à éviter qu’ils ne s’accumulent pour l’utilisateur. Pour en savoir plus, consultez [Configurer l’expiration du jeton d’actualisation](/fr-CA/docs/secure/tokens/refresh-tokens/configure-refresh-token-expiration).

  ## Validation des JWT
</div>

Nous vous recommandons fortement d’utiliser un middleware ou l’une des bibliothèques tierces open source existantes pour analyser et valider les JWT. Sur [JWT.io](https://jwt.io/#libraries-io), vous trouverez des bibliothèques pour diverses plateformes et divers langages, comme .NET, Python, Java, Ruby, Objective-C, Swift et PHP.

<div id="signing-algorithms">
  ## Algorithmes de signature
</div>

L’algorithme utilisé pour signer les jetons émis pour votre application ou votre API. Une signature fait partie d’un JWT et sert à vérifier que l’expéditeur du jeton est bien celui qu’il prétend être, et à s’assurer que le message n’a pas été modifié en cours de route. Pour en savoir plus sur les JWT, consultez [JSON Web Tokens](/fr-CA/docs/secure/tokens/json-web-tokens). Pour en savoir plus sur les signatures, consultez [JSON Web Token Structure](/fr-CA/docs/secure/tokens/json-web-tokens/json-web-token-structure).

Vous pouvez choisir parmi les <Tooltip tip="Algorithme de signature : algorithme utilisé pour signer numériquement des jetons afin de s’assurer que le jeton n’a pas été altéré." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=signing+algorithms">algorithmes de signature</Tooltip> suivants :

* **RS256** (signature RSA avec SHA-256) : un algorithme asymétrique, ce qui signifie qu’il y a deux clés : une clé publique et une clé privée qui doit être gardée secrète. Auth0 possède la clé privée utilisée pour générer la signature, et le consommateur du JWT récupère une clé publique à partir des endpoints de métadonnées fournis par Auth0 et l’utilise pour [valider la signature du JWT](/fr-CA/docs/secure/tokens/json-web-tokens/validate-json-web-tokens).
* **HS256** (HMAC avec SHA-256) : un algorithme symétrique, ce qui signifie qu’il n’y a qu’une seule clé privée qui doit être gardée secrète et qu’elle est partagée entre les deux parties. Comme la même clé est utilisée à la fois pour générer la signature et pour la valider, il faut veiller à ce qu’elle ne soit pas compromise. Cette clé privée (ou ce secret) est créée lorsque vous enregistrez votre Application (**<Tooltip tip="Secret client : secret utilisé par un client (application) pour s’authentifier auprès du serveur d’autorisation; il ne doit être connu que du client et du serveur d’autorisation et doit être suffisamment aléatoire pour ne pas pouvoir être deviné." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=Client+Secret">Secret client</Tooltip>**) ou votre API (**Signing Secret**) et que vous choisissez l’algorithme de signature HS256.

La pratique la plus sécuritaire, et celle que nous recommandons, consiste à utiliser **RS256** parce que :

* Avec RS256, vous avez l’assurance que seul le détenteur de la clé privée (Auth0) peut signer des jetons, tandis que n’importe qui peut vérifier si le jeton est valide à l’aide de la clé publique.
* Avec RS256, vous pouvez demander un jeton valide pour plusieurs audiences.
* Avec RS256, si la clé privée est compromise, vous pouvez effectuer une rotation des clés sans avoir à redéployer votre application ou votre API avec le nouveau secret (ce que vous devriez faire si vous utilisiez HS256).
* Avec HS256, si la clé secrète est compromise, vous devrez redéployer l’API avec le nouveau secret.

<div id="signing-keys">
  ## Clés de signature
</div>

Il est recommandé de supposer que plusieurs clés de signature peuvent être présentes dans votre JWKS. Cela peut sembler inutile, puisque le point de terminaison JWKS d’Auth0 contient généralement une seule clé de signature; toutefois, il peut y avoir plusieurs clés dans le JWKS lors de la rotation des certificats de signature.

Nous vous recommandons de mettre en cache vos clés de signature pour améliorer les performances de l’application et éviter de vous heurter aux limites de débit, mais assurez-vous que si le décodage d’un jeton échoue, vous invalidiez le cache et récupériez de nouvelles clés de signature avant de réessayer **une seule** fois.
