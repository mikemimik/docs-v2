<div id="configure-applications-for-mrrt">
  ## Configurer les applications pour les MRRT
</div>

Pour utiliser les <Tooltip tip="Jeton d’actualisation : jeton utilisé pour obtenir un nouveau jeton d’accès sans obliger les utilisateurs à se reconnecter." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=Refresh+Tokens">jetons d’actualisation</Tooltip> multi-ressources (MRRT), configurez les politiques de jeton d’actualisation de votre application à l’aide de l’API [Management API](https://auth0.com/docs/api/management/v2/clients/patch-clients-by-id). Ces politiques préciseront quelle API et quelles portées l’application est autorisée à demander lors d’un échange de jeton d’actualisation.

Vous pouvez définir des politiques MRRT dans la propriété `refresh_token.policies` de l’application.

| Propriété  | Type                             | Description                                                                                                                                                                     |
| ---------- | -------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `audience` | chaîne de caractères             | L’[identifiant](/fr-CA/docs/get-started/apis/api-settings#general-settings) de l’API Auth0 de l’application qui sera autorisée à utiliser le jeton d’actualisation.                   |
| `scope`    | tableau de chaînes de caractères | La liste des portées autorisées lors de la demande d’un jeton d’accès pour l’audience indiquée. La portée doit être identique ou plus restreinte que les portées définies pour l’API. |

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Les propriétés audience et scope doivent correspondre à une application existante dans votre tenant, sinon l’échange de jeton d’actualisation les ignorera sans avertissement.
</Callout>

Pour les applications existantes, effectuez un appel PATCH vers le point de terminaison [Update a Client](https://auth0.com/docs/api/management/v2/clients/patch-clients-by-id). Pour créer une nouvelle application, effectuez un appel POST vers le point de terminaison [Create a Client](https://auth0.com/docs/api/management/v2/clients/post-clients) :

<div id="implement-multi-resource-refresh-token">
  ## Mettre en œuvre un jeton d’actualisation multi-ressource
</div>

Une fois que vous avez configuré le jeton d’actualisation de votre application avec des politiques MRRT, vous pouvez commencer à échanger un seul jeton d’actualisation contre des <Tooltip tip="Jeton d’accès : information d’autorisation, sous la forme d’une chaîne opaque ou d’un JWT, utilisée pour accéder à une API." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=access+tokens">jetons d’accès</Tooltip> pour plusieurs API.
Pour ce faire, votre application doit lancer un flux de connexion à l’aide du [Flux de code d’autorisation](/fr-CA/docs/get-started/authentication-and-authorization-flow/authorization-code-flow) ou de l’[Octroi par mot de passe du propriétaire de la ressource](/fr-CA/docs/authenticate/login/oidc-conformant-authentication/oidc-adoption-rop-flow).

<div id="step-1-authenticate-and-request-a-refresh-token">
  ### Étape 1 : Authentifiez-vous et demandez un jeton d’actualisation
</div>

Pour recevoir un jeton d’actualisation, incluez la portée `offline_access` au moment de lancer la requête d’authentification. Pour en savoir plus, consultez [Obtenir des jetons d’actualisation](/fr-CA/docs/secure/tokens/refresh-tokens/get-refresh-tokens).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Si vous ne recevez pas de jeton d’actualisation, assurez-vous que :

  * L’option [Allow Offline Access](/fr-CA/docs/get-started/apis/api-settings#access-settings) est activée dans les paramètres de l’API.
  * `offline_access` est inclus dans la portée.
  * La valeur `audience` utilisée dans la requête correspond à une API configurée dans votre tenant.
</Callout>

  ### Étape 3 : Appeler l’API à l’aide du jeton d’accès

Utilisez le jeton d’accès pour appeler l’API sécurisée au moyen du [schéma d’autorisation HTTP Bearer](https://datatracker.ietf.org/doc/html/rfc6750). Pour en savoir plus, consultez [Use Access Tokens](/fr-CA/docs/secure/tokens/access-tokens/use-access-tokens).

<div id="use-multi-resource-refresh-token-with-actions">
  ## Utiliser le jeton d’actualisation multi-ressource avec Actions
</div>

L’utilisation du MRRT avec [Actions](/fr-CA/docs/customize/actions) vous permet de configurer une logique de décision dynamique en fonction des politiques MRRT de l’application. À cette fin, les Actions post-login incluent l’objet [`event.client.refresh_token.policies`](/fr-CA/docs/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger/post-login-event-object), qui fournit des renseignements pertinents, notamment l’<Tooltip tip="Audience : identifiant unique de l’audience d’un token émis. Nommée aud dans un token, sa valeur contient l’ID d’une application (Client ID) pour un ID Token ou d’une API (API Identifier) pour un Access Token." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=audience">audience</Tooltip> et la portée.

Vous pouvez utiliser l’objet `event.client.refresh_token.policies` pour évaluer l’audience et la portée de l’application lors de l’émission ou de l’échange d’un jeton d’actualisation, et assurer ainsi un contrôle précis de l’accès à l’API et des portées.

```js lines
exports.onExecutePostLogin = async (event, api) => {
  // retourner la liste des API autorisées dans le client
  const allowedAPIsInTheClient = event.client.refresh_token?.policies;

  if(allowedAPIsInTheClient?.some(policy => policy?.audience?.includes('https://myapi'))){
    // logique personnalisée
  }
};
```

<div id="evaluation-logic">
  #### Logique d’évaluation
</div>

Le MRRT agit comme un prolongement de l’authentification initiale, et non comme un remplacement. Lors de l’échange d’un jeton d’actualisation, Auth0 évalue la requête d’échange selon la logique suivante :

* **Si le paramètre audience est omis**, Auth0 renvoie un jeton d’accès avec l’audience initiale et toutes les portées supplémentaires configurées dans la politique MRRT.
* **Si un nouveau paramètre audience est spécifié**, Auth0 vérifie que l’audience est incluse dans la politique MRRT et renvoie un jeton d’accès pour la nouvelle audience avec les portées configurées.
* **Si le paramètre scope est omis**, Auth0 combine toutes les portées autorisées de la requête initiale et de la politique MRRT.
* **Si un nouveau paramètre scope est spécifié**, Auth0 valide les portées demandées et renvoie un jeton d’accès avec les portées incluses dans la politique MRTT. Les portées demandées qui sont invalides ou non autorisées sont ignorées sans avertissement.
* **Si le paramètre audience est le même que dans la requête initiale**, Auth0 applique la politique MRRT et renvoie un jeton d’accès pour l’audience avec toutes les portées configurées dans MRRT ainsi que les portées d’authentification initiales.

Le MRRT vous permet d’étendre l’accès des utilisateurs à de nouvelles API sans émettre de nouveaux jetons d’actualisation ni obliger l’utilisateur à se reconnecter.

  #### Exemples
</div>

Un utilisateur se connecte en demandant l’audience et la portée suivantes :

```json lines
{
"audience": "https://api.example.com",  
"scope": "openid profile read:messages"
}
```

La politique MRRT de l’application est configurée pour ajouter une `portée` additionnelle :

```json lines
{
  "audience": "https://api.example.com",
  "scope": ["write:messages"]
}
```

Un échange de jeton d’actualisation avec la même audience et sans aucune portée produirait un jeton d’accès contenant toutes les portées configurées :

```json lines
{
  "aud": "https://api.example.com",
  "scope": "openid profile read:messages write:messages"
}
```
