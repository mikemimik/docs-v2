Pour obtenir un <Tooltip tip="Jeton d’actualisation : jeton utilisé pour obtenir un nouveau jeton d’accès sans obliger les utilisateurs à se reconnecter." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=refresh+token">jeton d’actualisation</Tooltip>, vous devez inclure la portée `offline_access` [scope](/fr-CA/docs/get-started/apis/scopes) lorsque vous lancez une demande d’authentification au point de terminaison `/authorize`. Assurez-vous d’activer l’accès hors ligne dans votre API. Pour en savoir plus, consultez les [paramètres de l’API](/fr-CA/docs/get-started/apis/api-settings).

Par exemple, si vous utilisez le [flux du code d’autorisation](/fr-CA/docs/get-started/authentication-and-authorization-flow/authorization-code-flow), la demande d’authentification ressemblerait à ceci :

export const codeExample1 = `https://{yourDomain}/authorize?
    audience={API_AUDIENCE}&
    scope=offline_access&
    response_type=code&
    client_id={yourClientId}&
    redirect_uri={https://yourApp/callback}&
    state={OPAQUE_VALUE}`;

<AuthCodeBlock children={codeExample1} language="http" />

Le refresh token est stocké dans la session. Ensuite, lorsqu’une session doit être actualisée (par exemple, lorsqu’une période prédéfinie s’est écoulée ou que l’utilisateur tente d’effectuer une opération sensible), l’application utilise le refresh token dans le backend pour obtenir un nouveau <Tooltip tip="ID Token : Jeton destiné au client lui-même, plutôt qu’à l’accès à une ressource." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=ID+token">ID token</Tooltip> au moyen du point de terminaison `/oauth/token` avec `grant_type=refresh_token`.

Une fois l’utilisateur authentifié avec succès, l’application sera redirigée vers le `redirect_uri`, avec un `code` dans l’URL : `{https://yourApp/callback}?code=BPPLN3Z4qCTvSNOy`. Vous pouvez échanger ce code contre un <Tooltip tip="Access Token : Jeton d’autorisation, sous la forme d’une chaîne opaque ou d’un JWT, utilisé pour accéder à une API." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=access+token">access token</Tooltip> au moyen du point de terminaison `/oauth/token`.

La réponse doit contenir un jeton d’accès et un jeton d’actualisation.

```json lines
{
      "access_token": "eyJz93a...k4laUWw",
      "refresh_token": "GEbRxBN...edjnXbL",
      "token_type": "Bearer"
    }
```

Si vous demandez un jeton d’actualisation pour une application mobile au moyen du client natif correspondant (qui est public), vous n’avez pas à envoyer le `client_secret` dans la requête, car il n’est requis que pour les applications confidentielles.

Les jetons d’actualisation doivent être stockés de manière sécurisée par une application, puisqu’ils permettent à un utilisateur de rester authentifié pratiquement indéfiniment.

Pour en savoir plus sur la façon de mettre cela en œuvre à l’aide du flux du code d’autorisation, consultez notre tutoriel, [Appeler une API à l’aide du flux du code d’autorisation](/fr-CA/docs/get-started/authentication-and-authorization-flow/authorization-code-flow/call-your-api-using-the-authorization-code-flow). Pour les autres types d’octroi, consultez [Flux d’authentification et d’autorisation](/fr-CA/docs/get-started/authentication-and-authorization-flow).

<div id="customize-mfa">
  ## Personnaliser la MFA
</div>

<Warning>
  La MFA personnalisable avec les flux Resource Owner Password Grant, Embedded ou jeton d’actualisation est en Early Access. En utilisant cette fonctionnalité, vous acceptez les conditions applicables de l’essai gratuit prévues dans le [Master Subscription Agreement](https://www.okta.com/legal/) d’Okta. Pour en savoir plus sur les phases de lancement d’Auth0, consultez [Product Release Stages](/fr-CA/docs/troubleshoot/product-lifecycle/product-release-stages). Pour participer à l’accès anticipé, contactez [Auth0 Support](https://support.auth0.com/).
</Warning>

La MFA personnalisable permet aux utilisateurs d’enregistrer les facteurs de leur choix pris en charge par votre application et de répondre aux demandes de vérification avec ceux-ci.

Pendant l’authentification au point de terminaison `oauth/token`, la réponse renvoie l’erreur `mfa_required`, qui inclut le `mfa_token` à utiliser avec la MFA API ainsi que le paramètre `mfa_requirements`, qui contient une liste d’authentificateurs :

```json
{
  "error": "mfa_required",
  "error_description": "Multifactor authentication required",
  "mfa_token": "Fe26...Ha",
  "mfa_requirements": {
    "challenge": [
      { "type": "otp" },
      { "type": "push-notification" },
      { "type": "phone" },
      { "type": "recovery-code" }
      { "type": "email"} //fonctionne uniquement avec challenge
    ]
  }
}
```

Utilisez le `mfa_token` pour appeler le point de terminaison [`mfa/authenticator`](/fr-CA/docs/api/authentication/muti-factor-authentication/list-authenticators) afin d’obtenir la liste de tous les facteurs que l’utilisateur a configurés et de repérer celui du même type que prend en charge votre application.  Vous devez aussi obtenir le `authenticator_type` correspondant pour lancer des demandes de vérification :

```json
[
  {
    "type": "recovery-code",
    "id": "recovery-code|dev_qpOkGUOxBpw6R16t",
    "authenticator_type": "recovery-code",
    "active": true
  },
  {
    "type": "otp",
    "id": "totp|dev_6NWz8awwC8brh2dN",
    "authenticator_type": "otp",
    "active": true
  }
]
```

Forcez la vérification MFA en appelant le point de terminaison [`request/mfa/challenge`](/fr-CA/docs/api/authentication/muti-factor-authentication/request-mfa-challenge).

Personnalisez davantage votre processus MFA avec Auth0 Actions. Pour en savoir plus, consultez [Déclencheurs d’Actions : post-challenge - Objet d’API](/fr-CA/docs/customize/actions/explore-triggers/password-reset-triggers/post-challenge-trigger/post-challenge-api-object).
