title: Modifications apportées aux jetons Auth0 Management APIv2
---

Il y a quelque temps, nous avons modifié le processus d’obtention d’un jeton Auth0 Management APIv2. Cet article explique ce qui a changé, pourquoi nous l’avons fait et comment le contourner (ce n’est pas recommandé).

<div id="what-changed-and-why">
  ## Ce qui a changé et pourquoi
</div>

<div id="the-user-experience">
  ### L’expérience utilisateur
</div>

Jusqu’à récemment, vous pouviez générer un Management APIv2 Token directement depuis l’Explorateur de la <Tooltip tip="Management API : un produit qui permet aux clients d’effectuer des tâches administratives." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Management+API">Management API</Tooltip>. Vous sélectionniez les scopes selon le point de terminaison que vous vouliez appeler, puis vous obteniez un jeton sur cette même page.

Cette méthode était très simple, mais elle était aussi très peu sûre. Nous avons donc changé d’approche.

La nouvelle méthode utilise le [Client Credentials Processus](/docs/fr-CA/get-started/authentication-and-authorization-flow/client-credentials-flow). Pour savoir comment utiliser ce nouveau processus, consultez [Management API Access Tokens](/docs/fr-CA/secure/tokens/access-tokens/management-api-access-tokens).

<div id="why-this-changed">
  #### Pourquoi cela a changé
</div>

Pour générer le jeton, la Management API nécessitait l’accès à votre <Tooltip tip="Client Secret : secret utilisé par un client (application) pour s’authentifier auprès de l’Authorization Server; il ne doit être connu que du client et de l’Authorization Server et doit être suffisamment aléatoire pour ne pas pouvoir être deviné." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Client+Secret">Client Secret</Tooltip> global (utilisé pour signer le jeton). Il s’agit d’une information qui ne devrait pas être exposée dans les navigateurs Web.

De plus, l’API Explorer ne permet aucunement d’effectuer l’autorisation. Cela signifie que si un utilisateur pouvait se connecter et accéder à l’API Explorer, il pourrait générer un jeton avec n’importe quel scope, même s’il n’était pas autorisé à l’avoir.

La nouvelle implémentation ne présente pas ce type de risque. Une fois la configuration initiale effectuée, vous pouvez obtenir un jeton soit en visitant le <Tooltip tip="Auth0 Dashboard : principal produit d’Auth0 pour configurer vos services." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip>, soit en envoyant une simple requête `POST` au [point de terminaison `/oauth/token` de notre Authentication API](https://auth0.com/docs/api/authentication#client-credentials).

Toutefois, en ce qui concerne le processus manuel, nous comprenons que changer d’écran n’offre pas toujours la meilleure expérience utilisateur. Nous cherchons donc des moyens de rendre le nouveau processus plus intuitif.

<div id="the-validity-period">
  ### La durée de validité
</div>

Avec le processus précédent, les jetons n’expiraient jamais. Avec le nouveau processus, tous les jetons Management APIv2 expirent par défaut au bout de 24 heures.

<div id="why-this-changed">
  #### Pourquoi cela a changé
</div>

Avoir un jeton qui n’expire jamais peut être très risqué si un attaquant met la main dessus. Si le jeton expire au bout de quelques heures, l’attaquant ne dispose que d’une courte période pour accéder à vos ressources protégées.

Pour obtenir un jeton, vous devez suivre uniquement le processus décrit dans [Management API Access Tokens](/docs/fr-CA/secure/tokens/access-tokens/management-api-access-tokens).
