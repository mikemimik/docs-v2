  <Content>
    Ce tutoriel montre comment appeler votre API depuis un appareil à saisie limitée à l&#39;aide du Device Authorization Flow. Nous vous recommandons de vous connecter pour
    suivre ce Quickstart avec des exemples configurés pour votre compte.

    Pour une expérience interactive, vous pouvez utiliser le [Device Flow Playground](https://auth0.github.io/device-flow-playground/).

    <Section id={sections[0].id} title={sections[0].title} stepNumber="1">
      * Enregistrez une application native.
      * Assurez-vous que le bouton bascule **OIDC Conformant** est activé. Pour en savoir plus, consultez OIDC-Conformant Authentication.
      * Ajoutez **Device Code** aux types d’octroi de l’application. Pour en savoir plus, consultez Update Grant Types.
      * Ajoutez **Refresh Token** aux types d’octroi de l’application si vous souhaitez activer les refresh tokens.
      * Configurez et activez au moins une connexion pour
        l’application.
      * Enregistrez votre API dans Auth0.
      * Activez **Allow Offline Access** si vous utilisez des refresh tokens. Pour en savoir plus, consultez API Settings.
      * Configurez les paramètres de Device User Code pour définir le
        jeu de caractères, le format et la longueur de votre user code généré aléatoirement.
    </Section>

    <Section id={sections[1].id} title={sections[1].title} stepNumber="2">
      Lorsque l&#39;utilisateur démarre l&#39;application de l&#39;appareil et souhaite l&#39;autoriser, votre application doit demander un code d&#39;appareil à l&#39;Auth0 Authentication API pour l&#39;associer à la session de l&#39;utilisateur.

      Pour obtenir le code d&#39;appareil, votre application doit appeler l&#39;[Authorize endpoint](https://auth0.com/docs/api/authentication#-post-oauth-device-code-) du Device Authorization Flow de l&#39;Authentication API :

    <Section id={sections[2].id} title={sections[2].title} stepNumber="3">
      L’application sur l’appareil devrait recevoir une réponse HTTP 200 et un payload semblable à celui-ci :

      ```json lines
      {
        "device_code": "GmRh...k9eS",
        "user_code": "WDJB-MJHT",
        "verification_uri": "https://my-tenant.auth0.com/device",
        "verification_uri_complete": "https://my-tenant.auth0.com/device?user_code=WDJB-MJHT",
        "expires_in": 900,
        "interval": 5
      }
      ```
    </Section>

    <Section id={sections[3].id} title={sections[3].title} stepNumber="4">
      Après que votre application de l’appareil a reçu le `device_code` et le `user_code`, elle doit
      inviter l’utilisateur à se rendre à `verification_uri` et à saisir le `user_code`.

      <Frame>
        <img src="/docs/images/cdy7uua7fh8z/3Q9q41wocl6SojRoiGefXT/a98582a3c86740aaeb3d957f2dc4afe6/request-device-activation.png" />
      </Frame>

      <Info>
        Le `device_code` n’est pas destiné directement à l’utilisateur et ne doit pas être affiché pendant
        l’interaction afin d’éviter toute confusion.
      </Info>

      <Info>
        Lors de la création d’une CLI, vous pouvez ignorer cette étape et ouvrir immédiatement le navigateur avec
        `verification_uri_complete`.
      </Info>
    </Section>

    <Section id={sections[4].id} title={sections[4].title} stepNumber="5">
      Pendant que votre application attend que l’utilisateur l’active, elle doit appeler périodiquement le [point de terminaison POST /oauth/token de l’Authentication API](https://auth0.com/docs/api/authentication#-post-oauth-token-)
      et traiter la réponse comme il se doit.

      <Info>
        Assurez-vous que votre application attend pendant la durée de `interval` (en secondes) ou jusqu’à
        ce qu’elle reçoive une réponse positive, selon la période la plus longue, afin d’éviter les problèmes de latence réseau.
      </Info>

      ```bash
      curl --request POST \ 
        --url 'https://{yourDomain}/oauth/token' \
        --header 'content-type: application/x-www-form-urlencoded' \
        --data grant_type=urn:ietf:params:oauth:grant-type:device_code \
        --data device_code=AUTH0_SCOPES \
        --data 'client_id={yourClientId}'
      ```
    </Section>

    <Section id={sections[6].id} title={sections[6].title} stepNumber="7">
      Une fois que l’utilisateur a autorisé l’application de l’appareil, celle-ci reçoit une réponse HTTP 200 et le payload suivant :

      ```json lines
      {
        "access_token": "eyJz93a...k4laUWw",
        "refresh_token": "GEbRxBN...edjnXbL",
        "id_token": "eyJ0XAi...4faeEoQ",
        "token_type": "Bearer",
        "expires_in": 86400
      }
      ```

      <Info>
        Vous devriez valider vos jetons avant de les enregistrer. Pour savoir comment faire, consultez Valider les jetons d’accès et Valider les jetons d’identité.
      </Info>

      Les jetons d’accès servent à appeler l’Authentication API [point de terminaison Get User Info](https://auth0.com/docs/api/authentication#get-user-info) (si votre
      application sur l’appareil a demandé la portée `openid`) ou l’API indiquée par le paramètre `audience`.
      Si vous appelez votre propre API, votre application sur l’appareil doit vérifier le jeton d’accès avant de l’utiliser.

      Les jetons d’identité contiennent des renseignements sur l’utilisateur qu’il faut [décoder et extraire](/fr-CA/docs/secure/tokens/id-tokens#id-token-payload). L’Authentication API
      ne renvoie un `id_token` que si votre application sur l’appareil a demandé la portée `openid`.

      Les jetons d’actualisation servent à obtenir un nouveau jeton d’accès ou
      jeton d’identité après l’expiration du précédent. L’Authentication API ne renvoie un `refresh_token` que si
      l’option **Allow Offline Access** est activée pour l’API indiquée par le paramètre `audience`,
      et que votre application sur l’appareil a demandé la portée `offline_access`.
    </Section>

    <Section id={sections[7].id} title={sections[7].title} stepNumber="8">
      Pour appeler votre API, l’application sur votre appareil doit transmettre le jeton d’accès comme jeton Bearer dans l’en-tête
      `Authorization` de votre requête HTTP.

      ```bash lines
      curl --request GET \
        --url https://myapi.com/api \
        --header 'authorization: Bearer AUTH0_API_ACCESS_TOKEN' \
        --header 'content-type: application/json'
      ```
    </Section>

    <Section id={sections[8].id} title={sections[8].title} stepNumber="9">
      Pour obtenir un nouveau jeton d’accès pour un utilisateur, l’application de votre appareil peut appeler le point de terminaison [POST /oauth/token de l’Authentication API](https://auth0.com/docs/api/authentication#-post-oauth-token-) avec le
      paramètre `refresh_token`.

      ```bash lines
      curl --request POST \
        --url 'https://{yourDomain}/oauth/token' \
        --header 'content-type: application/x-www-form-urlencoded' \
        --data grant_type=refresh_token \
        --data 'client_id={yourClientId}' \
        --data 'client_secret={yourClientSecret}' \
        --data refresh_token=AUTH0_REFRESH_TOKEN
      ```

      Si la requête a réussi, l’application sur votre appareil reçoit une réponse HTTP 200 avec la charge utile suivante :

      ```json lines
      {
        "access_token": "eyJ...MoQ",
        "expires_in": 86400,
        "scope": "openid offline_access",
        "id_token": "eyJ...0NE",
        "token_type": "Bearer"
      }
      ```

      Pour en savoir davantage sur les jetons d’actualisation, consultez [Jetons d’actualisation](https://auth0.com/docs/secure/tokens/refresh-tokens).
    </Section>

    <Section id={sections[9].id} title={sections[9].title} stepNumber="10">
      Les journaux du tenant sont créés pour toute interaction qui a
      lieu et peuvent servir à résoudre les problèmes.

      ### Réponses du jeton

      Pendant que vous attendez que l’utilisateur autorise l’appareil, vous pouvez recevoir différentes réponses HTTP 4xx.

      #### Autorisation en attente

      Cette erreur s’affichera pendant que vous attendez que l’utilisateur agisse. Continuez le polling en utilisant la valeur
      `interval` suggérée, récupérée à l’étape précédente de ce tutoriel.

      ```json lines
      `HTTP 403`
      {
      "error": "authorization_pending",
      "error_description": "..."
      }
      ```

      #### Ralentissez

      Vous effectuez des requêtes de polling trop rapidement. Ralentissez et utilisez l’intervalle suggéré récupéré à l’étape précédente de ce
      tutoriel. Pour éviter de recevoir cette erreur en raison de la latence du réseau, vous devriez commencer à compter chaque intervalle après
      la réception de la réponse à la dernière requête de polling.

      ```json lines
      `HTTP 429`
      {
      "error": "slow_down",
      "error_description": "..."
      }
      ```

      #### Jeton expiré

      L’utilisateur n’a pas autorisé l’appareil assez rapidement; le `device_code` a donc expiré. Votre
      application doit informer l’utilisateur que le flux a expiré et lui demander de le relancer.

      <Info>
        L’erreur `expired_token` n’est renvoyée qu’une seule fois. Par la suite, le point de terminaison renvoie
        l’erreur `invalid_grant`.
      </Info>

      ```json lines
      `HTTP 403`
      {
      "error": "expired_token",
      "error_description": "..."
      }
      ```

      #### Accès refusé

      Si l’accès est refusé, le message suivant s’affiche :

      ```json lines
      `HTTP 403`
      {
      "error": "access_denied",
      "error_description": "..."
      }
      ```

      Cela peut se produire pour diverses raisons, notamment :

      * L’utilisateur a refusé d’autoriser l’appareil.
      * Le serveur d’autorisation a refusé la transaction.
      * Une Action configurée a refusé l’accès.
    </Section>

    <Section id={sections[11].id} title={sections[11].title} stepNumber="12">
      Pour utiliser le Device Authorization Flow, une application d’appareil doit :

      * Prendre en charge l’indication du nom du serveur (SNI)
      * Être une Application native
      * Avoir la [méthode d’authentification](/fr-CA/docs/secure/application-credentials#application-authentication-methods) définie sur **None**
      * Être [conforme à OIDC](/fr-CA/docs/get-started/applications/application-settings#oauth)
      * Ne pas avoir été créée par Dynamic Client Registration

      De plus, le Device Authorization Flow ne permet pas :

      * les Connexions sociales utilisant les clés de développeur Auth0, sauf si vous utilisez la nouvelle expérience Universal Login
      * l’accès aux paramètres de chaîne de requête à partir d’une page de connexion hébergée ou d’Actions.
    </Section>
