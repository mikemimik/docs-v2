<div id="saml-service-providers">
  ## Fournisseurs de services SAML
</div>

Les applications, en particulier les applications personnalisées, peuvent authentifier les utilisateurs auprès d’un <Tooltip tip="Fournisseur d’identité (IdP) : service qui stocke et gère les identités numériques." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=IdP">IdP</Tooltip> externe au moyen de protocoles comme <Tooltip tip="OpenID : norme ouverte d’authentification qui permet aux applications de vérifier l’identité des utilisateurs sans recueillir ni stocker les renseignements de connexion." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=OpenID">OpenID</Tooltip> Connect (OIDC) ou <Tooltip tip="OpenID : norme ouverte d’authentification qui permet aux applications de vérifier l’identité des utilisateurs sans recueillir ni stocker les renseignements de connexion." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=OAuth+2.0">OAuth 2.0</Tooltip>. Cependant, il se peut que vous souhaitiez utiliser un fournisseur <Tooltip tip="Security Assertion Markup Language (SAML) : protocole normalisé permettant à deux parties d’échanger des renseignements d’authentification sans mot de passe." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=SAML">SAML</Tooltip> d’entreprise pour l’authentification, même si votre application a été conçue pour utiliser l’un ou l’autre de ces protocoles.

<div id="saml-identity-providers">
  ## Fournisseurs d’identité SAML
</div>

Certaines applications (comme Salesforce, Box et Workday) permettent aux utilisateurs de s’authentifier auprès d’un IdP externe à l’aide du protocole SAML. Vous pouvez ensuite intégrer l’application à Auth0, qui agit comme l’IdP SAML de l’application. Les utilisateurs de l’application seront redirigés vers Auth0 pour ouvrir une session, et Auth0 peut les authentifier au moyen de n’importe quelle connexion d’authentification côté serveur, comme un répertoire LDAP, une base de données ou un autre IdP SAML ou fournisseur Social. Une fois l’utilisateur authentifié, Auth0 renvoie à l’application une assertion SAML l’indiquant.

<div id="auth0-as-service-provider">
  ## Auth0 comme fournisseur de services
</div>

Si Auth0 agit comme fournisseur de services dans une fédération SAML, Auth0 peut acheminer les requêtes d&#39;authentification vers un fournisseur d&#39;identité sans qu&#39;un compte ait déjà été créé pour un utilisateur donné. À l&#39;aide de l&#39;assertion renvoyée par le fournisseur d&#39;identité, Auth0 peut recueillir les renseignements nécessaires pour créer un profil utilisateur (ce processus est parfois appelé provisionnement juste-à-temps). Pour en savoir plus, consultez [Select from Multiple Connection Options](/docs/fr-CA/libraries/lock/selecting-from-multiple-connection-options).

Même si Auth0 n&#39;exige pas que des comptes utilisateur soient créés à l&#39;avance avant le processus d&#39;authentification, l&#39;application intégrée à Auth0 pourrait l&#39;exiger. Dans ce cas, plusieurs options s&#39;offrent à vous :

* Une fois l&#39;utilisateur créé par le fournisseur d&#39;identité, vous pouvez utiliser un processus hors bande pour créer l&#39;utilisateur correspondant dans l&#39;application (ou dans Auth0) et ajouter les attributs du profil utilisateur exigés par l&#39;application. Si, après l&#39;authentification, certains attributs sont absents du profil, l&#39;application peut les récupérer à partir de la source appropriée et les stocker dans le profil utilisateur Auth0. Ces attributs supplémentaires seront ensuite transmis à l&#39;application (en plus de ceux ajoutés par le fournisseur d&#39;identité) à la prochaine ouverture de session de l&#39;utilisateur.
* Vous pouvez utiliser une règle Auth0 pour appeler une API afin de récupérer toute information manquante et l&#39;ajouter dynamiquement au profil Auth0 (qui est ensuite renvoyé à l&#39;application). Les règles s&#39;exécutent après une authentification réussie, et votre application peut récupérer les attributs du profil chaque fois, ou vous pouvez enregistrer ces attributs dans le profil Auth0.
* Auth0 peut transmettre à l&#39;application les renseignements de base du profil provenant du fournisseur d&#39;identité, après quoi l&#39;application récupère toute information manquante à partir d&#39;une autre source. À partir de ces deux ensembles de renseignements, l&#39;application crée un profil utilisateur local.

Vous pouvez spécifier des domaines de courriel dans la configuration de la connexion SAMLP Auth0 afin de contrôler l&#39;IdP qui prendra en charge un groupe précis d&#39;utilisateurs. Par exemple, si vous ajoutez le domaine de courriel `example.com` à la configuration de la connexion SAMLP Auth0 pour l&#39;entreprise X, tous les utilisateurs dont l&#39;adresse courriel appartient au domaine `example.com` seront pris en charge par l&#39;IdP propre à l&#39;entreprise X.

<div id="auth0-as-identity-provider">
  ## Auth0 comme fournisseur d’identité
</div>

Si Auth0 sert de fournisseur d’identité dans une fédération SAML, les comptes utilisateur peuvent être créés de plusieurs façons :

* Au moyen d’un système d’authentification backend, comme un répertoire LDAP, une base de données ou un autre fournisseur d’identité SAML.
* En utilisant le <Tooltip tip="Auth0 Dashboard : le principal produit d’Auth0 pour configurer vos services." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip>.
* En appelant l’<Tooltip tip="Management API : un produit qui permet aux clients d’effectuer des tâches administratives." cta="Voir le glossaire" href="/docs/fr-CA/glossary?term=Management+API">Management API</Tooltip> d’Auth0.
* En mettant en place l’inscription libre-service des utilisateurs.

Si votre application est conçue pour récupérer les données du profil utilisateur à partir d’un stockage local, vous devrez créer le profil local une fois les comptes créés dans Auth0. Voici quelques façons de le faire :

* Un processus hors bande qui crée des profils utilisateur dans l’application;
* Une règle Auth0 qui s’exécute à la première connexion et appelle une API de l’application pour y créer le profil utilisateur;
* Modifier l’application pour créer des profils utilisateur de façon dynamique, selon les informations contenues dans l’assertion SAML.
