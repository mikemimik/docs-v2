---
description: Comment configurer des courriels non uniques et les identifiants qui y sont associés
title: Courriels non uniques
---

Les courriels non uniques permettent à plusieurs comptes d’utilisateur partageant une connexion de base de données d’utiliser une même adresse courriel, tandis qu’un autre attribut (comme le nom d’utilisateur ou le numéro de téléphone) sert d’identifiant principal. Que vous soyez un parent qui gère plusieurs comptes d’enfants avec une seule adresse courriel ou une petite entreprise qui utilise une adresse courriel par établissement, les courriels non uniques vous permettent de maintenir une expérience de connexion et de réinitialisation du mot de passe sécurisée et propre à chaque compte.

<div id="considerations">
  ### Considérations
</div>

Passez en revue les points suivants pour vérifier si le courriel non unique convient à votre cas d’usage.

<div id="primary-identifier-requirements">
  #### Exigences relatives à l’identifiant principal
</div>

L’adresse courriel ne peut pas servir d’identifiant principal lorsque vous utilisez le courriel non unique. Vous devez configurer un autre attribut comme identifiant principal; il sera utilisé pour l’authentification, les réinitialisations de mot de passe et la gestion du compte.

Pour en savoir plus sur les identifiants et les attributs, consultez [identifiant flexible](/fr-CA/docs/authenticate/database-connections/flexible-identifiers-and-attributes).

<div id="password-resets">
  #### Réinitialisations de mot de passe
</div>

Les utilisateurs finaux doivent fournir leur nom d’utilisateur, leur numéro de téléphone ou l’attribut que leur administrateur a défini comme attribut principal lorsqu’ils réinitialisent leur mot de passe. Auth0 utilise cet identifiant principal pour repérer le compte associé à une adresse courriel partagée et en réinitialiser le mot de passe.

<div id="irreversible-settings">
  #### Paramètres irréversibles
</div>

Une fois que l’attribut de courriel d’une connexion est défini comme non unique, il ne peut plus redevenir unique. De plus, seules de nouvelles connexions de base de données peuvent être créées avec la prise en charge des courriels non uniques; vous ne pouvez pas modifier une connexion existante et vous devez mettre à jour votre application pour utiliser l’identifiant principal choisi.

<div id="flexible-identifiers">
  #### Identifiants flexibles
</div>

Les Identifiants flexibles doivent être activés dans la connexion de base de données pour pouvoir utiliser les courriels non uniques, et ils ne peuvent pas être désactivés une fois la connexion créée. Lorsque vous activez les courriels non uniques à l’aide de l’<Tooltip tip="" cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=Management+API">Management API</Tooltip>, les Identifiants flexibles sont automatiquement configurés pour vous.

<div id="api-behavior-changes">
  ### Changements de comportement de l’API
</div>

`GET /api/v2/users-by-email` retourne tous les utilisateurs qui partagent la même adresse courriel.

`DELETE /api/v2/connections/{id}/users` n’est pas compatible avec les connexions à courriel non unique.

`POST /dbconnections/change_password` ne fonctionne pas avec les connexions à courriel non unique, car il nécessite une adresse courriel unique pour repérer le compte utilisateur. Les utilisateurs doivent utiliser des flux qui s’appuient sur l’identifiant principal pour réinitialiser leur mot de passe.

<div id="enable-non-unique-emails-in-the-auth0-dashboard">
  ### Activer le courriel non unique dans l’Auth0 Dashboard
</div>

1. Allez à **Authentication &gt; Database** et créez une nouvelle connexion.
2. Accédez à la section **Choisissez un ou plusieurs attributs comme identifiants d’utilisateur** et mettez **Adresse courriel** à **On,** puis activez le bouton **Autoriser les adresses courriel non uniques** qui s’affiche.
3. Activez également **username** ou le **numéro de téléphone** en le mettant à **On** afin qu’il serve d’identifiant principal pour les flux de login et de Password Reset.
4. Après avoir confirmé que le courriel ne sera pas utilisé comme identifiant, choisissez **Create** pour enregistrer la connexion.

<div id="enable-non-unique-emails-via-the-management-api">
  ### Activer les courriels non uniques via le Management API
</div>

Utilisez le point de terminaison `POST /api/v2/connections` du Management API pour créer une connexion de base de données qui prend en charge les courriels non uniques.

Lors de la création de la connexion :

* Définissez **unique: false** dans l’objet `options.attributes.email` pour permettre à plusieurs comptes d’utiliser la même adresse courriel. Définissez **identifier.active: false** pour vous assurer que le courriel ne peut pas servir d’identifiant principal lorsqu’il n’est pas unique.
* Choisissez un autre attribut comme identifiant principal et définissez **identifier.active: true** pour l’attribut choisi.

Sans identifiant principal autre que le courriel, les flux d’authentification et de réinitialisation du mot de passe ne fonctionneront pas correctement. Assurez-vous qu’au moins un attribut est défini comme identifiant actif.

<div id="example-request">
  #### Exemple de requête
</div>

Vous trouverez ci-dessous un exemple de corps de requête pour créer une connexion de base de données qui utilise un nom d&#39;utilisateur comme identifiant principal et prend en charge les e-mails non uniques :

```json lines expandable
{
 "name": "new-non-unique",
 "strategy": "authO",
 "options": {
  "attributes": {
   "email": {
     "unique": false,
     "signup" : {
       "status": "required"
     },
     "identifier": {
      "active": false
     },
     "profile_required": true
   ｝，
   "username": {
    "signup": {
     "status": "required"
   },
   "identifier": {
    "active": true
   ｝，
   "profile_required": true
   }
  }
 }
}
```

<div id="shared-email-risk-disclaimer">
  ### Avis de non-responsabilité concernant les risques liés aux adresses courriel partagées
</div>

Bien que la fonctionnalité **courriel non unique** comprenne des mesures de protection, comme l’interdiction d’utiliser le courriel comme identifiant principal et l’obligation d’effectuer les réinitialisations de mot de passe à l’aide d’un nom d’utilisateur ou d’un numéro de téléphone**,** il existe tout de même un risque inhérent lorsque plusieurs comptes d’utilisateur partagent la même adresse courriel. Par exemple :

* Toutes les communications par courriel (p. ex., les liens de réinitialisation de mot de passe, les notifications) seront livrées dans la même boîte de réception, peu importe l’utilisateur qui a lancé l’action.
* Cela pourrait entraîner de la confusion chez les utilisateurs ou un accès involontaire à des liens envoyés par courriel si la boîte de réception est partagée.

En activant cette fonctionnalité, vous reconnaissez et acceptez la responsabilité de veiller à ce que :

* Les scénarios de partage de courriel conviennent à votre cas d’utilisation.
* Les utilisateurs finaux soient informés et formés en conséquence.
* La conception de votre application tienne compte des chevauchements possibles dans les flux de travail fondés sur le courriel.

Ce compromis offre de la souplesse, mais exige une mise en œuvre réfléchie et une communication claire avec vos utilisateurs.
