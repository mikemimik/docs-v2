<ReleaseStageNotice feature="Clés d’accès natives" stage="ea" contact="soutien Auth0" />

Les clés d’accès sont une solution de rechange aux formes traditionnelles d’authentification (comme le nom d’utilisateur et le mot de passe) qui résiste à l’hameçonnage et offre une expérience utilisateur plus simple et plus sécuritaire. Pour obtenir tous les détails d’implémentation, consultez [Native Passkeys for Mobile Applications](/fr-CA/docs/authenticate/database-connections/passkeys/native-passkeys-for-mobile-applications).

Les clés d’accès natives utilisent une combinaison d’Auth0 et d’API iOS ou Android natives pour intégrer directement les flux de vérification à votre application mobile. Les points de terminaison ci-dessous constituent un sous-ensemble de l’Auth0 Authentication API. Pour en savoir plus sur l’utilisation de cette API, consultez l’[introduction à l’Authentication API](https://auth0.com/docs/api/authentication#introduction).

Les clés d’accès comportent trois flux connexes :

* **inscription** : Crée un nouveau compte utilisateur avec une clé d’accès comme principale méthode d’authentification.
* **enrôlement** : Ajoute une clé d’accès comme méthode d’authentification au compte d’un utilisateur existant.
* **connexion** : Demande à un utilisateur existant de s’authentifier à l’aide d’une clé d’accès associée à son compte.

<div id="signup-flow">
  ## Flux d’inscription
</div>

<div id="request-signup-challenge">
  ### Request Signup Challenge
</div>

`POST /passkey/register`

Démarre le flux d’inscription par clé d’accès pour un nouvel utilisateur.

En réponse, Auth0 renvoie [PublicKeyCredentialCreationOptions](https://www.w3.org/TR/webauthn-3/#dictdef-publickeycredentialcreationoptions) et un ID de session. Consultez `timeout` sous `authn_params_public_key` dans la réponse pour connaître le délai d’expiration de la session.

Le flux d’inscription par clé d’accès prend en charge Organizations au moyen du paramètre `organization`, selon le comportement décrit dans [connexion Flows for Organizations](/fr-CA/docs/manage-users/organizations/login-flows-for-organizations). Si votre application est configurée pour **Business Users**, vous devez fournir le paramètre `organization` ainsi qu’un nom d’Organization valide ou une valeur d’identifiant valide. Une fois que l’utilisateur a enregistré une clé d’accès, Auth0 l’inscrit dans l’Organization fournie.

<div id="request-parameters">
  #### Paramètres de la requête
</div>

| Paramètre      | Description                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                   |
| -------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `client_id`    | **Obligatoire**. Le `client_id` de votre application.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                         |
| `realm`        | **Facultatif**. Le nom de la connexion à associer à cet utilisateur.<br /><br />Si aucune connexion n&#39;est précisée, le répertoire par défaut de votre tenant est utilisé.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                               |
| `user_profile` | **Obligatoire**. Un objet contenant les renseignements d&#39;identification de l&#39;utilisateur. Par défaut, il comprend une adresse `email` valide et un `name` d&#39;affichage facultatif.<br /><br />Si vous avez activé [Flexible Identifiers](/fr-CA/docs/authenticate/database-connections/activate-and-configure-attributes-for-flexible-identifiers) pour votre connexion de base de données, vous pouvez utiliser une combinaison de `email`, `phone_number` ou `username` comme identifiants. Ces options peuvent être obligatoires ou facultatives et doivent correspondre à votre configuration Flexible Identifiers.<br /><br />Si l&#39;identifiant transmis (comme `email`) existe déjà dans le répertoire, l&#39;utilisateur devrait plutôt être invité à terminer le flux connexion. |

<div id="remarks">
  #### Remarques
</div>

* Une fois la demande de défi terminée, votre application peut poursuivre le processus d’inscription de l’utilisateur à l’aide des API natives [Android](https://developer.android.com/identity/passkeys/create-passkeys) ou [iOS](https://developer.apple.com/documentation/authenticationservices/supporting-passkeys#Register-a-new-account-on-a-service).
* Vous devez ensuite [authentifier le nouvel utilisateur](#authenticate-new-user) à l’aide des informations obtenues par les API natives pour terminer le flux.

<div id="authenticate-new-user">
  ### Authentifier un nouvel utilisateur
</div>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  L’inscription d’une Passkey Native n’est pas prise en charge pour le moment lorsqu’une vérification par SMS/Email OTP est requise sur la même connexion au moment de l’inscription.
</Callout>

`POST /oauth/token`

Utilise le <Tooltip tip="Token endpoint : point de terminaison du serveur d’autorisation utilisé pour demander des jetons par programmation." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=Token+endpoint">Token endpoint</Tooltip> pour authentifier l’utilisateur avec les informations d’identification fournies, créer son compte et renvoyer les jetons demandés.

Le paramètre `authn_response` est basé sur la [spécification de la Web Authentication API](https://www.w3.org/TR/webauthn-3/). Dans le flux de Passkey Native, les renseignements transmis à ce point de terminaison peuvent être récupérés au moyen des API natives de votre application mobile :

* [documentation d’inscription Android](https://developer.android.com/identity/passkeys/create-passkeys)
* [documentation d’inscription iOS](https://developer.apple.com/documentation/authenticationservices/supporting-passkeys#Register-a-new-account-on-a-service)

<div id="enrollment-flow">
  ## Processus d’enrôlement
</div>

L’enrôlement d’une nouvelle clé d’accès pour un utilisateur se fait en deux étapes à l’aide de la My Account API. Ce processus garantit que l’enrôlement de la clé d’accès est amorcée de façon sécuritaire, puis vérifiée.

Avant d’amorcer le processus d’enrôlement, assurez-vous d’avoir un <Tooltip tip="Jeton d’accès : justificatif d’autorisation, sous la forme d’une chaîne opaque ou d’un JWT, utilisé pour accéder à une API." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=access+token">jeton d’accès</Tooltip> avec la portée `create:me:authentication_methods` pour le point de terminaison `/me`.

<div id="initiate-passkey-enrollment">
  ### Amorcer l’enrôlement de la clé d’accès
</div>

`POST /me/v1/authentication-methods`

La première étape consiste à amorcer le processus d’enrôlement. Pour ce faire, envoyez une requête POST au point de terminaison `/me/v1/authentication-methods`.

<div id="request-parameters">
  #### Paramètres de la requête
</div>

| Paramètre    | Description                                                                   |
| ------------ | ----------------------------------------------------------------------------- |
| `type`       | **Obligatoire**. Indiquez la valeur : `public-key`.                           |
| `connection` | **Facultatif**. Le nom de la connexion dans laquelle créer la clé d’accès.    |
| `identity`   | **Facultatif**. L’identité de l’utilisateur. S’utilise avec les comptes liés. |

<div id="remarks">
  #### Remarques
</div>

* La propriété `auth_session` dans le corps de la réponse est l’identifiant de la session d’authentification en cours. Elle doit être transmise à l’endpoint `/verify`.
* Une fois la demande de défi terminée, votre application peut poursuivre le processus d’enrôlement de l’utilisateur à l’aide des API natives [Android](https://developer.android.com/identity/passkeys/create-passkeys) ou [iOS](https://developer.apple.com/documentation/authenticationservices/supporting-passkeys#Register-a-new-account-on-a-service). L’utilisateur sera alors invité à créer une clé d’accès à l’aide de son authentificateur (comme un lecteur d’empreintes digitales, une security key ou un téléphone).

<div id="verify-passkey-enrollment">
  ### Vérifier l’enrôlement d’une clé d’accès
</div>

`POST /me/v1/authentication-methods/passkey|new/verify`

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  L’ID dans le chemin est toujours `passkey|new` pour les nouveaux enrôlements.
</Callout>

Une fois que l’utilisateur a créé la clé d’accès avec son authentificateur, l’application cliente reçoit un `AuthenticatorAttestationResponse` de l’API WebAuthn. Cette réponse doit être renvoyée au service Auth0 pour terminer et valider l’enrôlement.

* [Documentation sur l’inscription Android](https://developer.android.com/identity/passkeys/create-passkeys)
* [Documentation sur l’inscription iOS](https://developer.apple.com/documentation/authenticationservices/supporting-passkeys#Register-a-new-account-on-a-service)

<div id="login-flow">
  ## Flux de connexion
</div>

<div id="request-login-challenge">
  ### Request connexion Challenge
</div>

`POST /passkey/challenge`

Lance le flow de connexion par clé d’accès pour un utilisateur existant qui a enregistré une clé d’accès dans son compte lors de son inscription initiale.

En réponse, Auth0 renvoie [PublicKeyCredentialRequestOptions](https://www.w3.org/TR/webauthn-3/#dictdef-publickeycredentialrequestoptions), un ID de session et un identifiant de Relying Party `rpId`. Vérifiez la valeur `timeout` sous `authn_params_public_key` dans la réponse pour connaître le délai d’expiration de la session. Le `rpId` renvoyé dans la réponse est l’identifiant que l’appareil natif (iOS/Android) utilise pour retrouver les credentials enregistrées associées à un domaine. Pour qu’une clé d’accès créée dans une application web (p. ex., `example.com`) soit accessible dans le flow natif, le `rpId` renvoyé doit correspondre au `rpId` du web. Pour savoir comment personnaliser l’ID RP pour votre tenant, consultez [Configure clé d’accès Policy](/fr-CA/docs/authenticate/database-connections/passkeys/configure-passkey-policy#configure-relying-party-id-rp-id).

Le flow de connexion par clé d’accès prend en charge Organizations au moyen du paramètre `organization`, conformément au comportement décrit dans [connexion Flows for Organizations](/fr-CA/docs/manage-users/organizations/login-flows-for-organizations). Si votre application est configurée pour **Business Users**, vous devez fournir le paramètre `organization` ainsi qu’un nom ou une valeur d’identifiant Organization valide.

Tous les tokens émis sont dans le contexte de l’Organization fournie. Si vous avez activé [Auto-Membership](/fr-CA/docs/manage-users/organizations/login-flows-for-organizations#auto-membership) pour votre Organization, l’utilisateur est automatiquement inscrit dans l’Organization après s’être authentifié avec succès.

#### Paramètres de requête

| Paramètre   | Description                                                                                                                                                              |
| ----------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `client_id` | **Obligatoire**. Le `client_id` de votre application.                                                                                                                    |
| `realm`     | **Facultatif**. Le nom de la connexion à associer à l’utilisateur.<br /><br />Si aucune connexion n’est spécifiée, le répertoire par défaut de votre tenant est utilisé. |

<div id="authenticate-existing-user">
  ### Authentifier un utilisateur existant
</div>

`POST /oauth/token`

Utilise le Token endpoint pour authentifier l’utilisateur à l’aide des informations d’identification fournies et renvoyer les tokens demandés.

Le paramètre `authn_response` repose sur la [spécification de la Web Authentication API](https://www.w3.org/TR/webauthn-3/). Dans le flow de passkey natif, les informations transmises à ce endpoint peuvent être récupérées au moyen des API natives de votre application mobile :

* [documentation de connexion Android](https://developer.android.com/identity/passkeys/sign-in-with-passkeys)
* [documentation de connexion iOS](https://developer.apple.com/documentation/authenticationservices/supporting-passkeys#Connect-to-a-service-with-an-existing-account)

ID d’information d’identification Base64URL.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| `authn_response.type`                    | **Obligatoire**. Incluez la valeur : `public-key`                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| `authn_response.authenticatorAttachment` | **Facultatif**. Incluez les valeurs :<ul><li>`platform`</li><li>`cross-platform`</li></ul>                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                        |
| `authn_response.response`                | **Obligatoire**. Un objet contenant les éléments suivants : <ul><li>`authenticatorData` : Contient les [données de l’authentificateur](https://www.w3.org/TR/webauthn-3/#authenticator-data) renvoyées par l’authentificateur.</li><li>`clientDataJSON` : Contient une sérialisation des données client compatible JSON; héritée de [AuthenticatorResponse](https://www.w3.org/TR/webauthn-3/#authenticatorresponse).</li><li>`signature` : Signature Base64URL renvoyée par l’authentificateur.</li><li>`userHandle` : Identifiant Base64URL du compte utilisateur, renvoyé sous la forme de `user.id` à l’étape d’inscription.</li></ul> |
| `authn_response.clientExtensionResults`  | **Facultatif**. Contient les résultats du traitement des extensions client demandées par la partie de confiance.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                  |
