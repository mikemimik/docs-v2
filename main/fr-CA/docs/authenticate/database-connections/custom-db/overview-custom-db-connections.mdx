<Card title="La disponibilité varie selon le plan Auth0">
  Votre plan Auth0 ou votre entente personnalisée détermine si cette fonctionnalité est offerte. Pour en savoir plus, consultez [Pricing](https://auth0.com/pricing).
</Card>

Utilisez une connexion de base de données personnalisée lorsque vous souhaitez donner accès à votre propre référentiel indépendant de données d’identité (hérité) pour les usages suivants :

* **Authentification** : utilisez votre base de données comme <Tooltip tip="Fournisseur d’identité (IdP) : service qui stocke et gère les identités numériques." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=identity+provider">fournisseur d’identité</Tooltip> dans Auth0 pour authentifier les utilisateurs. (On parle alors d’authentification héritée.)
* **Importation d’utilisateurs** : utilisez la migration automatique (migration progressive ou différée)
* **Accès par proxy à un tenant Auth0** : utilisez l’architecture multi-tenant d’Auth0.

Vous pouvez créer et configurer une connexion de base de données personnalisée en effectuant l’une des tâches suivantes :

1. Utilisez le point de terminaison [Créer des connexions](https://auth0.com/docs/api/management/v2#!/Connections/post_connections) avec la stratégie `auth0`.
2. Accédez à [Auth0 Dashboard &gt; Authentication &gt; Database](https://manage.auth0.com/#/connections/database), créez la connexion et activez l’option **Use my own database** pour permettre la modification des scripts d’action de base de données.

   <Frame>![Activer l’option Use my own database dans Custom Database](/docs/images/cdy7uua7fh8z/3kgHDpBFdVWNq9XOfhsTXI/2efa43e44793b8dd69b9f9e8b54bf752/2025-02-25_10-20-20.png)</Frame>

<div id="how-it-works">
  ## Fonctionnement
</div>

Comme l’illustre le schéma ci-dessous, vous utilisez des connexions de base de données personnalisées dans le cadre du flux <Tooltip tip="Universal Login : votre application redirige vers Universal Login, hébergé sur l’Authorization Server d’Auth0, afin de vérifier l’identité d’un utilisateur." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=Universal+Login">Universal Login</Tooltip> pour obtenir des renseignements sur l’identité des utilisateurs à partir de votre propre référentiel d’identités existant.

<Frame>![Anatomie des Custom Database Connections](/docs/images/cdy7uua7fh8z/2lHqvZKFiEbAXURU2gmchc/626cac94211c266f2135a41456b2e49d/custom-database-connections.png)</Frame>

En plus des éléments communs à tous les types de [connexion de base de données](/fr-CA/docs/authenticate/database-connections), une connexion de base de données personnalisée vous permet de configurer des scripts d’action : du code personnalisé utilisé pour interagir avec votre référentiel d’identités existant. Auth0 fournit des [modèles de scripts d’action de base de données personnalisée](/fr-CA/docs/authenticate/database-connections/custom-db/templates) pour la configuration, et ceux que vous utilisez dépendent du fait que vous créez une connexion de base de données personnalisée pour l’authentification héritée ou pour la migration automatique.

<Card title="Bonne pratique">
  Les scripts d’action peuvent être implémentés sous forme de fonctions anonymes. Toutefois, les fonctions anonymes compliquent le débogage lorsqu’il faut interpréter la pile d’appels générée à la suite d’une erreur. Par souci de simplicité, nous recommandons de donner un nom de fonction à chaque script d’action.
</Card>

<div id="legacy-authentication-scenario">
  ### Scénario d’authentification héritée
</div>

Dans un scénario d’authentification héritée, un nouvel enregistrement d’utilisateur est créé dans Auth0 lors de la première connexion de l’utilisateur, mais Auth0 ne conserve pas le hachage du mot de passe de l’utilisateur. Auth0 utilisera toujours le magasin d’identités hérité et l’identité qu’il contient pour authentifier l’utilisateur.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Les connexions à une base de données personnalisée sont aussi utilisées en dehors du flux de travail Universal Login. Par exemple, le script d’action `changePassword` d’une connexion est appelé lorsqu’un changement de mot de passe est effectué pour un utilisateur qui se trouve dans un magasin d’identités hérité.
</Callout>

<div id="automatic-migration-scenario">
  ### Scénario de migration automatique
</div>

Lors d’une migration automatique ou progressive, Auth0 crée un nouvel utilisateur dans un magasin d’identités (base de données) géré par Auth0. À partir de ce moment, Auth0 utilise toujours l’identité du magasin d’identités géré par Auth0 pour authentifier l’utilisateur. Pour que cela se produise, Auth0 doit d’abord authentifier l’utilisateur auprès de l’ancien magasin d’identités, et le nouvel utilisateur n’est créé que si cette étape réussit. Le nouvel utilisateur sera créé à l’aide du même identifiant et du même mot de passe fournis lors de l’authentification.

<Card title="Bonne pratique">
  La création d’un utilisateur dans un scénario de migration automatique se produit généralement après l’exécution du script d’action `login`. Nous vous recommandons donc de ne pas tenter de supprimer un utilisateur d’un ancien magasin d’identités au moyen d’une opération en ligne (c.-à-d. dans le script `login`), mais plutôt d’effectuer la suppression dans le cadre d’un processus distinct. Cela évitera la suppression accidentelle d’un utilisateur si une erreur survient pendant le processus de migration.
</Card>

Dans un scénario de migration automatique, les utilisateurs restent dans l’ancien magasin d’identités et peuvent être supprimés ou archivés au besoin. Un effet secondaire peut survenir si un utilisateur est supprimé d’Auth0, mais est toujours présent dans l’ancien magasin d’identités. Dans ce cas, une tentative de connexion par l’utilisateur supprimé pourrait entraîner l’exécution du script `login` et/ou `getUser`, puis la migration de l’utilisateur à partir de l’ancien magasin d’identités une fois de plus.

<Card title="Bonne pratique">
  Nous recommandons de marquer toute identité d’utilisateur existante comme ayant été migrée avant la fin de `login` ou de `getUser`, et avant toute suppression éventuelle de l’ancien magasin d’identités.
</Card>

<div id="size">
  ## Taille
</div>

La taille totale du code d’implémentation d’un action script ne doit pas dépasser 100 kB. Plus cette taille est grande, plus la latence causée par le processus de mise en package et de transport utilisé par la plateforme Webtask sans serveur d’Auth0 augmente, ce qui aura une incidence sur les performances de votre système.

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  La limite de 100 kB n’inclut pas les modules `npm` qui peuvent être référencés dans des instructions require.
</Callout>

<div id="environment">
  ## Environnement
</div>

Les scripts d’action s’exécutent comme une série de fonctions JavaScript appelées dans une instance de conteneur Webtask sans serveur. Dans ce cadre, un environnement précis est fourni, ainsi qu’un certain nombre d’artefacts provenant à la fois du conteneur Webtask et du serveur <Tooltip tip="Serveur d’authentification : serveur qui confirme ou refuse l’identité d’un utilisateur." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=authentication+server">d’authentification</Tooltip> Auth0 lui-même (votre tenant Auth0).

<div id="npm-modules">
  ### modules npm
</div>

Les conteneurs Webtask serverless d’Auth0 peuvent utiliser une [vaste gamme de modules npm](https://auth0-extensions.github.io/canirequire/); les modules `npm` réduisent non seulement la taille globale du code de l’action script, mais donnent aussi accès à un large éventail de fonctionnalités prêtes à l’emploi.

De nombreux modules `npm` accessibles au public sont pris en charge d’emblée. La liste a été compilée et vérifiée afin de relever tout risque potentiel en matière de sécurité. Si vous avez besoin d’un module `npm` qui n’est pas pris en charge d’emblée, vous pouvez en faire la demande par l’intermédiaire du [portail de soutien Auth0](https://support.auth0.com) ou auprès de votre représentant Auth0. Auth0 évaluera votre demande afin de déterminer si elle convient. Auth0 ne prend actuellement pas en charge l’utilisation de modules `npm` provenant de dépôts privés.

<div id="variables">
  ### Variables
</div>

Les scripts d’action Auth0 prennent en charge les variables d’environnement, auxquelles on accède au moyen de l’objet `configuration`, accessible globalement. Pour en savoir plus, consultez la section Ajouter des paramètres de configuration dans [Créer des connexions de base de données personnalisées](/fr-CA/docs/authenticate/database-connections/custom-db/create-db-connection).

<Card title="Bonne pratique">
  L’objet `configuration` doit être considéré comme étant en lecture seule et servir à stocker des renseignements sensibles, comme des identifiants ou des clés d’API, utilisés pour accéder à des répertoires d’identités externes. Cela évite d’intégrer en dur des valeurs sensibles dans un script d’action.
</Card>

L’objet de configuration peut aussi servir à prendre en charge les stratégies du [cycle de vie du développement logiciel (SDLC)](/fr-CA/docs/get-started/auth0-overview/create-tenants/set-up-multiple-environments) que vous utilisez, en vous permettant de définir des variables ayant des valeurs propres au tenant. Cela évite d’intégrer en dur dans un script d’action des valeurs qui peuvent changer selon le tenant qui l’exécute.

<div id="global-object">
  ### objet global
</div>

Les conteneurs Webtask serverless d’Auth0 sont provisionnés à partir d’un groupe associé à chaque tenant Auth0. Chaque instance de conteneur met l’objet `global` à disposition; on peut y accéder à partir de tous les scripts d’action qui s’exécutent dans cette instance. L’objet `global` agit comme une variable globale propre au conteneur et peut servir à définir de l’information ou des fonctions utilisées par tous les scripts d’action exécutés dans l’instance du conteneur.

Cela signifie que l’objet `global` peut servir de cache pour des ressources coûteuses, tant que ces ressources ne sont pas propres à l’utilisateur. Par exemple, vous pourriez l’utiliser pour stocker un <Tooltip tip="Jeton d’accès : information d’autorisation, sous la forme d’une chaîne opaque ou d’un JWT, utilisée pour accéder à une API." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=Access+Token">jeton d’accès</Tooltip> pour une API de journalisation tierce qui fournit des fonctionnalités non propres à l’utilisateur. Vous pourriez aussi y stocker un jeton d’accès pour votre propre API non propre à l’utilisateur définie dans Auth0 et obtenu au moyen du [flux Client Credentials](/fr-CA/docs/get-started/authentication-and-authorization-flow/client-credentials-flow).

<Warning>
  Un script d’action peut s’exécuter dans n’importe quelle instance de conteneur déjà en cours d’exécution, ou dans une nouvelle instance de conteneur (qui pourrait ensuite être ajoutée au groupe). Dans Auth0, il n’existe aucune affinité entre un script d’action et un conteneur donné. Vous devriez donc éviter de stocker dans l’objet `global` toute information propre à l’utilisateur, et toujours vous assurer que toute déclaration faite dans l’objet `global` prévoit également une initialisation.
</Warning>

Chaque fois qu’un conteneur Webtask est recyclé, ou chaque fois qu’un nouveau conteneur Webtask est instancié, l’objet `global` qu’il définit est réinitialisé. Ainsi, toute affectation dans l’objet `global` associé à un conteneur devrait également prévoir une initialisation.

<Card title="Bonne pratique">
  Pour offrir une certaine souplesse sur le plan des performances, les conteneurs Webtask serverless sont provisionnés dans Auth0 de façon ponctuelle et sont aussi soumis à diverses politiques de recyclage. En règle générale, nous recommandons de ne pas supposer qu’un objet `global` a une durée de vie de plus de 20 minutes.
</Card>

<div id="security">
  ## Sécurité
</div>

<div id="access-legacy-identity-storage-via-custom-api">
  ### Accéder au stockage des identités existant au moyen d’une API personnalisée
</div>

Protéger le stockage des identités existant contre tout accès généralisé est une bonne pratique recommandée. Par exemple, exposer directement une base de données à Internet peut être extrêmement problématique : les interfaces de base de données pour SQL et autres technologies du genre offrent un éventail de fonctionnalités très large, ce qui contrevient au principe du moindre privilège en matière de sécurité.

<Card title="Bonne pratique">
  Nous vous recommandons de mettre en place une API pour appliquer le principe du moindre privilège à votre magasin d’identités existant (base de données), plutôt que d’ouvrir simplement un accès général par Internet.
</Card>

L’autre approche consiste à créer une API simple (personnalisée), protégée à l’aide d’un jeton d’accès, que chaque script d’action peut appeler. Elle servirait alors d’interface vers le magasin d’identités existant. Le flux Client Credentials Grant Flow peut ensuite être utilisé pour obtenir un jeton d’accès à partir d’un script, puis ce jeton peut être mis en cache dans l’objet `global` afin d’être réutilisé et d’améliorer les performances. L’API peut alors fournir un nombre restreint de points de terminaison protégés qui n’exécutent que les fonctions de gestion existantes nécessaires (par exemple, `read user`, `change password`).

<Card title="Bonne pratique">
  Par défaut, Auth0 vous fournira un jeton pour n’importe quelle API si vous vous authentifiez avec succès et incluez l’audience appropriée. Restreindre l’accès à l’API du magasin d’identités existant en limitant l’attribution des jetons d’accès au moyen d’une Rule empêchera toute utilisation non autorisée et atténuera plusieurs scénarios de vecteurs d’attaque, par exemple lorsqu’une redirection vers `/authorize` est interceptée et que l’audience de l’API est ajoutée.
</Card>

<div id="allowlist-access-to-legacy-identity-storage">
  ### Mettre sur liste d’autorisation l’accès au stockage d’identités hérité
</div>

Que vous gériez l’accès à un répertoire d’identités hérité au moyen d’une API personnalisée ou que vous utilisiez l’interface native fournie, limitez l’accès à la liste des adresses IP associées à votre tenant Auth0. La mise sur liste d’autorisation restreint l’accès au répertoire d’identités hérité, de sorte que seuls les scripts d’action de base de données personnalisée définis dans Auth0 soient autorisés.

<Warning>
  La liste d’autorisation d’adresses IP d’Auth0 est partagée entre tous les tenants Auth0 définis pour une région. N’utilisez jamais la liste d’autorisation comme seul moyen de sécuriser l’accès à votre répertoire d’identités hérité; cela pourrait créer des vulnérabilités de sécurité et permettre un accès non autorisé à vos utilisateurs.
</Warning>
