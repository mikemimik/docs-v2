<div id="access-legacy-identity-storage-via-custom-api">
  ## Accéder au stockage d’identités hérité au moyen d’une API personnalisée
</div>

Nous recommandons de mettre en place une API afin d’appliquer le principe du moindre privilège à votre stockage d’identités hérité, plutôt que de simplement ouvrir un accès général sur Internet.

Protéger le stockage d’identités hérité contre tout accès général est une pratique exemplaire recommandée. Exposer directement à Internet une base de données (d’identités héritée), par exemple, peut être extrêmement problématique : les interfaces de base de données SQL et autres offrent des fonctionnalités très étendues, ce qui va à l’encontre du principe du moindre privilège en matière de sécurité.

L’autre solution consiste à créer une API (personnalisée) simple — protégée au moyen d’un <Tooltip tip="Access Token : information d’autorisation, sous la forme d’une chaîne opaque ou d’un JWT, utilisée pour accéder à une API." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=access+token">access token</Tooltip> — que chaque action script peut appeler. Celle-ci servirait d’interface vers le référentiel d’identités hérité. Le Client Credentials Grant Flow peut ensuite être utilisé pour obtenir un access token à partir d’un script, puis le mettre en cache pour le réutiliser dans l’objet `global` afin d’améliorer les performances. L’API peut alors fournir un nombre restreint de endpoints protégés qui n’exécutent que les fonctions de gestion (des identités héritées) nécessaires (p. ex., lire un utilisateur, change password).

Par défaut, Auth0 vous fournira un token pour n’importe quelle API si vous vous authentifiez avec succès et incluez l’<Tooltip tip="Audience : identifiant unique de l’audience d’un token émis. Nommée aud dans un token, sa valeur contient l’ID soit d’une application (Client ID) pour un ID Token, soit d’une API (API Identifier) pour un Access Token." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=audience">audience</Tooltip> appropriée. Restreindre l’accès à l’API du stockage d’identités hérité en limitant l’allocation des access tokens au moyen d’une rule empêchera toute utilisation non autorisée et atténuera plusieurs scénarios de vecteurs d’attaque, par exemple lorsqu’une redirection vers `/authorize` est interceptée et que l’audience de l’API y est ajoutée.

<Warning>
  Restreindre l’accès à l’API au moyen d’une Rule atténuera les scénarios de vecteurs d’attaque, par exemple lorsqu’une redirection vers `/authorize` est interceptée et que l’audience de l’API y est ajoutée, et garantira que seul l’accès au moyen de client credentials précis est accordé.
</Warning>

<div id="access-to-legacy-identity-storage">
  ## Accès au stockage d’identités hérité
</div>

Que vous gériez l’accès au stockage d’identités hérité au moyen d’une API personnalisée ou à l’aide de l’interface native fournie, vous devriez restreindre l’accès à la liste des adresses IP associées à votre tenant Auth0. Pour consulter la liste des adresses IP, lisez [Adresses IP d’Auth0 pour les listes d’autorisation](/fr-CA/docs/secure/security-guidance/data-security/allowlist). L’ajout d’URL à la liste d’autorisation limite l’accès au référentiel d’identités hérité et garantit que seuls les scripts d’action de base de données personnalisée définis dans Auth0 sont autorisés.

<Warning>
  La liste d’autorisation des adresses IP d’Auth0 est partagée entre tous les tenants Auth0 définis dans une région. N’utilisez jamais la liste d’autorisation comme seule méthode pour sécuriser l’accès à votre référentiel d’identités hérité; cela pourrait créer des vulnérabilités de sécurité et permettre un accès non autorisé à vos utilisateurs.
</Warning>
