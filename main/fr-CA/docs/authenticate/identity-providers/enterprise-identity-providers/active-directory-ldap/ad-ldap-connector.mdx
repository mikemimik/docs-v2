Auth0 s’intègre à Active Directory (AD) au moyen du Lightweight Directory Access Protocol (LDAP), par l’intermédiaire d’un connecteur Active Directory/LDAP que vous installez sur votre réseau.

<Warning>
  Le connecteur AD/LDAP est conçu pour les scénarios où votre entreprise contrôle le serveur AD/LDAP. Il ne doit **pas** être installé sur les serveurs de votre client.

  Dans les scénarios B2B où vous voulez permettre aux utilisateurs de votre client d’accéder à vos applications avec leurs identifiants d’entreprise, connectez-vous au service de fédération de votre client (par exemple, son propre service Auth0, ADFS ou tout fournisseur d’identité SAML) à l’aide de l’une des connexions d’entreprise disponibles.

  Si vous installez un connecteur AD/LDAP sur les serveurs de votre client et qu’il est connecté directement à votre domaine Auth0, vous devrez gérer directement les mots de passe des utilisateurs de votre client. **Auth0 recommande fortement d’éviter ce type de déploiement et n’offre aucun soutien à cet égard.**
</Warning>

Le connecteur AD/LDAP (1) sert de pont entre votre Active Directory/LDAP (2) et le service Auth0 (3). Ce pont est nécessaire, car AD/LDAP est généralement limité à votre réseau interne, tandis qu’Auth0 est un service infonuagique qui s’exécute dans un contexte complètement différent.

<Frame>![Connecteur AD/LDAP - intégrer AD/LDAP à Auth0 - diagramme](/docs/images/cdy7uua7fh8z/1xMpdHrtor0TR7F1gZ96zL/130e13f59b728fe14e00f1815b90103f/ldap-connect.png)</Frame>

Lorsqu’un utilisateur s’authentifie avec Auth0, il est redirigé vers le connecteur AD/LDAP, qui valide l’utilisateur auprès de votre service AD, puis renvoie les résultats de la validation à Auth0. Le connecteur AD/LDAP prend en charge l’authentification LDAP, Kerberos et par certificats client.

Pour en savoir plus, consultez [Configurer l’authentification du connecteur AD/LDAP avec Kerberos](/docs/fr-CA/authenticate/identity-providers/enterprise-identity-providers/active-directory-ldap/ad-ldap-connector/configure-ad-ldap-connector-with-kerberos) et [Configurer l’authentification du connecteur AD/LDAP avec des certificats client](/docs/fr-CA/authenticate/identity-providers/enterprise-identity-providers/active-directory-ldap/ad-ldap-connector/configure-ad-ldap-connector-client-certificates).

<div id="caches">
  ## Caches
</div>

Le connecteur AD/LDAP met en cache les profils d’utilisateur et les données d’authentification (Auth0 stocke un hachage du mot de passe de l’utilisateur) afin d’assurer une disponibilité et des performances optimales, et met à jour ces données chaque fois qu’un utilisateur se connecte. Le cache n’est utilisé que lorsque le Connector est hors service ou inaccessible. Les données mises en cache sont toujours conservées, sauf si vous désactivez la mise en cache des données d’authentification dans le Dashboard. Les valeurs du cache sont sensibles à la casse, ce qui signifie que les tentatives de connexion ne réussiront que si les utilisateurs fournissent exactement le nom d’utilisateur qui a été mis en cache.

<div id="high-availability-and-load-balancing">
  ## Haute disponibilité et répartition de charge
</div>

Pour assurer la haute disponibilité et la répartition de charge, vous pouvez installer plusieurs instances du connecteur. Toutes les connexions sont sortantes, du connecteur vers le serveur Auth0; il n’est donc généralement pas nécessaire de modifier votre pare-feu.

Chaque instance du cluster haute disponibilité sera en permanence active et connectée à Auth0. Auth0 enverra les transactions de connexion et les autres requêtes à n’importe lequel des connecteurs disponibles. Si l’une des instances tombe en panne en raison d’un problème de réseau ou de matériel, Auth0 redirigera les transactions de connexion vers l’autre connecteur. Un déploiement hautement disponible vous permet également de mettre à jour le connecteur sans temps d’arrêt.

<div id="openldap-directories">
  ## Annuaires OpenLDAP
</div>

Par défaut, le Connector est fortement optimisé pour Active Directory. Pour le configurer pour tout autre annuaire LDAP (comme OpenLDAP), vous devez personnaliser ces paramètres dans le fichier `config.json` :

```json config.json lines
"LDAP_USER_BY_NAME": "(cn={0})",
"LDAP_SEARCH_QUERY": "(&(objectClass=person)(cn={0}))",
"LDAP_SEARCH_ALL_QUERY": "(objectClass=person)",
```

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Dans certains cas, l’attribut `uid` peut être plus approprié que `cn`.
</Callout>

<div id="opendj-example">
  ### Exemple OpenDJ
</div>

Avec le **panneau de configuration OpenDJ**, vous pouvez obtenir la liste des attributs de chaque utilisateur. Cette liste peut vous aider à déterminer quel attribut sera utilisé comme nom d’utilisateur pour l’authentification avec Auth0.

<Frame>![Diagramme d’exemple OpenDJ du connecteur AD/LDAP](/docs/images/cdy7uua7fh8z/14fTMlPqtzUicW2VTZPLPG/3ffa90a44eb864455ad3b77406f8ad32/opendj-attributes.png)</Frame>

Dans cet exemple, la valeur de `cn` pour John est **johndoe** et le champ `mail` est défini sur **johndoe@contoso.com**. Si votre organisation souhaite que les utilisateurs s’authentifient à l’aide de leur nom d’utilisateur (`cn`), vous pouvez définir le paramètre `LDAP_USER_BY_NAME` sur `(cn={0})`. En revanche, si les utilisateurs doivent s’authentifier à l’aide de leur adresse courriel, vous devez le définir sur `(mail={0})`.
