<Card title="Aperçu">
  Concepts clés

  * Consultez les limites des clés de développeur d&#39;Auth0.
</Card>

Lorsque vous utilisez l&#39;un des [fournisseurs d&#39;identité sociaux](/fr-CA/docs/authenticate/identity-providers/social-identity-providers) disponibles, vous devez enregistrer votre application auprès du <Tooltip tip="Fournisseur d'identité (IdP) : service qui stocke et gère les identités numériques." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=identity+provider">fournisseur d&#39;identité</Tooltip> concerné afin d&#39;obtenir un <Tooltip tip="Client ID : identifiant unique attribué à votre application lorsque vous l'enregistrez auprès d'un fournisseur d'identité." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=Client+ID">Client ID</Tooltip> et un <Tooltip tip="Client Secret : valeur confidentielle utilisée avec le Client ID pour authentifier votre application auprès d'un fournisseur d'identité." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=Client+Secret">Client Secret</Tooltip>.

<Warning>
  Auth0 vous permet de tester un fournisseur d&#39;identité social sans indiquer votre propre Client ID ni votre propre Client Secret en utilisant les clés de développeur Auth0. Vous pouvez ainsi activer et tester rapidement un fournisseur d&#39;identité social en particulier, mais cela ne doit pas être utilisé en production.
</Warning>

Les clés de développeur Auth0 ne sont pas disponibles dans les [déploiements Private Cloud](/fr-CA/docs/deploy-monitor).

Pour les environnements de production, assurez-vous de [suivre les étapes pour le fournisseur de votre choix](/fr-CA/docs/authenticate/identity-providers) afin d&#39;obtenir le Client ID et le Client Secret auprès du fournisseur et d&#39;éviter les [limites](#limitations-of-developer-keys) liées à l&#39;utilisation des clés de développeur. Pour savoir comment convertir des clés de développeur Google en clés de production, [consultez cet Auth0 Developer Lab.](https://developer.auth0.com/resources/labs/authentication/google-social-connection-to-login#set-up-google-production-keys)

<Card title="Clés de développeur personnalisées">
  Une ou plusieurs connexions utilisent des clés de développeur Auth0, qui sont uniquement destinées au développement et aux tests. Les connexions doivent être configurées avec vos propres clés de développeur pour que la page de consentement affiche votre logo au lieu de celui d&#39;Auth0 et pour configurer le Single Sign-On (SSO) pour ces connexions. Les clés de développeur Auth0 ne sont pas recommandées pour les environnements de production.
</Card>

<Card title="Client ID et Client Secret">
  La terminologie exacte de Client ID / Client Secret peut varier d&#39;un fournisseur d&#39;identité à l&#39;autre. Par exemple, X les appelle Consumer Key / Consumer Secret et LinkedIn les appelle API Key / Secret Key.
</Card>

<div id="limitations-of-developer-keys">
  ## Limites des clés de développeur
</div>

Les clés de développeur d’Auth0 sont strictement destinées aux tests. Les utiliser à la place de votre propre Client ID et Client Secret peut entraîner des comportements inattendus, des fonctionnalités limitées ou des erreurs dans l’application.

Lorsque vous utilisez les clés de développeur d’Auth0, le flux d’authentification de différents fournisseurs d’identité peut afficher aux utilisateurs le nom, le logo et les renseignements d’Auth0. En enregistrant votre propre application, vous pouvez plutôt utiliser votre propre logo ainsi que d’autres renseignements sur l’application.

<div id="limitations-of-developer-keys-when-using-universal-login">
  ## Limites des clés de développeur lors de l’utilisation d’Universal Login
</div>

Si vous utilisez l’[expérience Classic Login](/fr-CA/docs/authenticate/login/auth0-universal-login/universal-login-vs-classic-login/classic-experience) ou l’[expérience Universal Login](/fr-CA/docs/authenticate/login/auth0-universal-login/universal-login-vs-classic-login/universal-experience), les limitations suivantes s’appliquent :

1. Vous ne pouvez pas utiliser de clés de développeur avec des [domaines personnalisés](/fr-CA/docs/customize/custom-domains).
2. Le [Single Sign-On](/fr-CA/docs/authenticate/single-sign-on) ne fonctionnera pas correctement si vous utilisez les clés de développeur d’Auth0. Les applications de développeur d’Auth0 sont configurées pour rediriger vers l’URL `https://login.auth0.com/login/callback` au lieu de l’**URL de rappel** de votre propre tenant, par exemple `https://YOUR_AUTH0_DOMAIN/login/callback`.
   Par conséquent, le cookie SSO n’est pas défini sur le domaine de votre propre tenant. Ainsi, la prochaine fois qu’un utilisateur s’authentifie, aucun cookie SSO ne sera détecté, même si vous avez configuré votre application pour **utiliser Auth0 au lieu du fournisseur d’identité pour effectuer le Single Sign-On** (tenants Legacy uniquement).
3. [Rediriger les utilisateurs avec Actions](/fr-CA/docs/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger/redirect-with-actions) ne fonctionnera pas correctement. Les Redirect Actions reprennent sur le endpoint `https://YOUR_AUTH0_DOMAIN/continue`. Lorsque vous utilisez les clés de développeur d’Auth0, la session est établie sur un endpoint spécial, générique et indépendant du tenant, et l’appel à `/continue` ne retrouvera pas votre session précédente, ce qui entraînera une erreur.
4. Le [Federated Logout](/fr-CA/docs/authenticate/login/logout) ne fonctionne pas. Lorsque vous utilisez les clés de développeur d’Auth0, l’appel à `/v2/logout?federated` déconnecte l’utilisateur d’Auth0, mais pas du fournisseur d’identité social.
5. `prompt=none` ne fonctionnera pas sur le endpoint [/authorize](https://auth0.com/docs/api/authentication/reference#social). La méthode checkSession() d’[Auth0.js](/fr-CA/docs/libraries/auth0js) utilise `prompt=none` en interne, donc elle ne fonctionnera pas non plus.
6. Si Auth0 agit comme fournisseur d’identité SAML et que vous utilisez une connexion sociale avec les clés de développeur d’Auth0, la réponse SAML générée comportera certaines erreurs, comme un attribut `InResponseTo` manquant ou un élément `AudienceRestriction` vide.
7. L’[authentification multifacteur](/fr-CA/docs/secure/multi-factor-authentication) ne fonctionnera pas correctement. Lorsque l’authentification MFA réussit, une requête POST est envoyée à `https://YOUR_AUTH0_DOMAIN/mf`. Lorsque vous utilisez les clés de développeur d’Auth0, la session est établie sur un endpoint spécial, générique et indépendant du tenant, et l’appel à `/mf` ne retrouvera pas votre session précédente, ce qui entraînera une erreur.
