Les [fournisseurs d&#39;identité (IdP)](/fr-CA/docs/authenticate/identity-providers) les plus courants sont offerts dans le [Auth0 Dashboard](https://manage.auth0.com/#) et dans le [Auth0 Marketplace](https://marketplace.auth0.com/features/social-connections). Vous pouvez toutefois ajouter n&#39;importe quel fournisseur <Tooltip tip="OAuth 2.0 : Cadre d'autorisation qui définit les protocoles et les flux d'autorisation." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=OAuth+2.0">OAuth 2.0</Tooltip> en tant que **connexion sociale personnalisée** dans le <Tooltip tip="Auth0 Dashboard : Principal produit d'Auth0 pour configurer vos services." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip>.

1. Dans le Dashboard, accédez à [Authentication &gt; Social](https://manage.auth0.com/#/connections/social).
2. Sélectionnez **Create Connection**, allez au bas de la liste, puis sélectionnez **Create Custom**.

Le formulaire qui s&#39;affiche contient plusieurs champs que vous devez utiliser pour configurer la connexion personnalisée :

* **Connection Name** : Identifiant logique de la connexion que vous créez. Ce nom ne peut pas être modifié, doit commencer et se terminer par un caractère alphanumérique, et ne peut contenir que des caractères alphanumériques et des traits d&#39;union.
* **Authorization URL** : URL vers laquelle les utilisateurs sont redirigés pour se connecter.

  <Callout icon="file-lines" color="#0EA5E9" iconType="regular">
    N&#39;essayez pas de définir le paramètre OAuth 2.0 `response_mode` dans l&#39;URL d&#39;autorisation. Cette connexion prend uniquement en charge le paramètre `response_mode` par défaut (`query`).
  </Callout>
* **Token URL** : URL utilisée pour échanger le code d&#39;autorisation reçu contre des <Tooltip tip="Jeton d'accès : Informations d'identification d'autorisation, sous la forme d'une chaîne opaque ou d'un JWT, utilisées pour accéder à une API." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=access+tokens">jetons d&#39;accès</Tooltip> et, s&#39;ils sont demandés, des <Tooltip tip="ID Token : Informations d'identification destinées au client lui-même, plutôt qu'à l'accès à une ressource." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=ID+tokens">jetons d&#39;identité</Tooltip>.
* **Scope** : Paramètres `scope` à envoyer avec la demande d&#39;autorisation. Séparez plusieurs scopes par des espaces.
* **Separate scopes using a space** : Bascule qui détermine comment les scopes sont délimités si le paramètre `connection_scope` est inclus lors de [l&#39;appel de l&#39;API du fournisseur d&#39;identité](/fr-CA/docs/authenticate/identity-providers/calling-an-external-idp-api). Par défaut, les scopes sont délimités par une virgule. Si la bascule est activée, les scopes sont délimités par un espace. Pour en savoir plus, consultez [Add Scopes/Permissions to Call Identity Provider APIs](/fr-CA/docs/authenticate/identity-providers/adding-scopes-for-an-external-idp).
* **<Tooltip tip="Client ID : Valeur d'identification attribuée à votre ressource enregistrée par Auth0." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=Client+ID">Client ID</Tooltip>** : Client ID d&#39;Auth0 en tant qu&#39;application utilisée pour demander l&#39;autorisation et échanger le code d&#39;autorisation. Pour obtenir un Client ID, vous devrez enregistrer votre application auprès du <Tooltip tip="Fournisseur d'identité (IdP) : Service qui stocke et gère les identités numériques." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=identity+provider">fournisseur d&#39;identité</Tooltip>.
* **<Tooltip tip="Client Secret : Secret utilisé par un client (application) pour s'authentifier auprès du Authorization Server; il ne doit être connu que du client et du Authorization Server et doit être suffisamment aléatoire pour ne pas pouvoir être deviné." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=Client+Secret">Client Secret</Tooltip>** : Client Secret d&#39;Auth0 en tant qu&#39;application utilisée pour échanger le code d&#39;autorisation. Pour obtenir un Client Secret, vous devrez enregistrer votre application auprès du fournisseur d&#39;identité.
* **Fetch User Profile Script** : Script Node.js utilisé pour appeler une URL userinfo avec le jeton d&#39;accès fourni. Pour en savoir plus sur ce script, consultez [Fetch User Profile Script](#fetch-user-profile-script).
* **Purpose** : Active la connexion sociale pour l&#39;authentification, Connected Accounts for Token Vault, ou les deux. Pour en savoir plus, consultez [User authentication vs Connected Accounts](/fr-CA/docs/secure/tokens/token-vault/connected-accounts-for-token-vault#user-authentication-vs-connected-accounts).

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Lorsque vous configurez le fournisseur d’identité personnalisé, utilisez l’URL de rappel `https://{yourDomain}/login/callback`.
</Callout>

Une fois la connexion personnalisée créée, vous verrez la vue **Applications** et votre connexion sera assujettie à la [Politique de limitation du débit](/fr-CA/docs/troubleshoot/customer-support/operational-policies/rate-limit-policy) d’Auth0. Vous pouvez y activer ou désactiver les applications pour lesquelles vous souhaitez que la connexion s’affiche.

<div id="update-authentication-flow">
  ## Mettre à jour le flux d’authentification
</div>

Lorsque vous créez une connexion, le type d’octroi OAuth 2.0 attribué par défaut à la connexion est le flux de code d’autorisation. Si vous avez une application publique qui ne peut pas stocker de Client Secret, comme une application monopage ou une application native, vous pouvez utiliser la <Tooltip tip="Management API : un produit permettant aux clients d’effectuer des tâches administratives." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=Management+API">Management API</Tooltip> pour mettre à jour la connexion afin qu’elle utilise le [flux de code d’autorisation + PKCE](/fr-CA/docs/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce). Pour en savoir plus sur les <Tooltip tip="Flux d’autorisation : attribution d’autorisation (ou workflow) spécifiée dans le cadre OAuth 2.0." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=authorization+flows">flux d’autorisation</Tooltip>, consultez [Quel flux OAuth 2.0 devrais-je utiliser ?](/fr-CA/docs/get-started/authentication-and-authorization-flow/which-oauth-2-0-flow-should-i-use)

1. Effectuez une requête `GET` vers le point de terminaison [`/get-connections-by-id`](https://auth0.com/docs/api/management/v2/connections/get-connections-by-id). La réponse ressemblera à :

   ```json lines
   {
     "id": "[connectionID]",
     "options": {
       "email": true,
       "scope": [
         "email",
         "profile"
       ],
       "profile": true
     },
     "strategy": "google-oauth2",
     "name": "google-oauth2",
     "is_domain_connection": false,
     "realms": [
       "google-oauth2"
     ]
   }
   ```

2. Copiez l’objet `options` au complet.

3. Effectuez une requête [`PATCH`](https://auth0.com/docs/api/management/v2/connections/patch-connections-by-id) avec l’objet `options` et ajoutez `"pkce_enabled":` `true`.

<Warning>
  Si vous n’incluez pas l’objet `options` au complet, des informations seront perdues et la connexion cessera de fonctionner.
</Warning>

<div id="fetch-user-profile-script">
  ## Fetch User Profile Script
</div>

Le Fetch User Profile Script est appelé une fois que l’utilisateur s’est connecté avec le fournisseur OAuth2. Auth0 exécute ce script pour appeler l’API du fournisseur OAuth2 et obtenir le profil d’utilisateur :

```javascript lines expandable
function fetchUserProfile(accessToken, context, callback) {
  request.get(
    {
      url: 'https://auth.example.com/userinfo',
      headers: {
        'Authorization': 'Bearer ' + accessToken,
      }
    },
    (err, resp, body) => {
      if (err) {
        return callback(err);
      }
      if (resp.statusCode !== 200) {
        return callback(new Error(body));
      }
      let bodyParsed;
      try {
        bodyParsed = JSON.parse(body);
      } catch (jsonError) {
        return callback(new Error(body));
      }
      const profile = {
        user_id: bodyParsed.account.uuid,
        email: bodyParsed.account.email
      };
      callback(null, profile);
    }
  );
}
```

La propriété `user_id` dans le profil renvoyé est obligatoire, et la propriété `email` est facultative, mais fortement recommandée. Pour en savoir plus sur les attributs qui peuvent être renvoyés, consultez [Attributs racine du profil utilisateur](/fr-CA/docs/manage-users/user-accounts/user-profiles/root-attributes/update-root-attributes-for-users).

Vous pouvez filtrer, ajouter ou supprimer n’importe quel élément du profil renvoyé par le fournisseur. Cependant, il est recommandé de garder ce script aussi simple que possible. Une manipulation plus avancée des renseignements sur l’utilisateur peut être effectuée à l’aide de [Rules](/fr-CA/docs/customize/rules). L’un des avantages des Rules, c’est qu’elles s’appliquent à toute connexion.

<div id="pass-static-parameters">
  ### Transmettre des paramètres statiques

<div id="pass-dynamic-parameters">
  ### Transmettre des paramètres dynamiques
</div>

Dans certains cas, vous pouvez vouloir transmettre une valeur dynamique à un fournisseur d’identité OAuth 2.0. Dans ce cas, vous pouvez utiliser l’élément `authParamsMap` de `options` pour définir une correspondance entre l’un des paramètres supplémentaires existants acceptés par le point de terminaison [Auth0 `/authorize`](https://auth0.com/docs/api/authentication#social) et le paramètre accepté par le fournisseur d’identité.

En reprenant le même exemple ci-dessus, supposons que vous vouliez transmettre le paramètre `custom_param` au point de terminaison d’autorisation, mais que vous souhaitiez préciser la valeur réelle du paramètre au moment d’appeler le point de terminaison Auth0 `/authorize`.

Dans ce cas, vous pouvez utiliser l’un des paramètres supplémentaires existants acceptés par le point de terminaison `/authorize`, comme `access_type`, et l’associer au paramètre `custom_param` :

Désormais, lors de l’appel à l’point de terminaison `/authorize`, vous pouvez transmettre le type d’accès dans le paramètre `access_type`, et cette valeur sera ensuite transmise à l’point de terminaison d’autorisation dans le paramètre `custom_param`.

<div id="pass-extra-headers">
  ## Ajouter des en-têtes supplémentaires
</div>

Dans certains cas, vous devrez ajouter des en-têtes supplémentaires au <Tooltip tip="Point de terminaison des jetons : point de terminaison du serveur d’autorisation utilisé pour demander des jetons par programmation." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=Token+endpoint">point de terminaison des jetons</Tooltip> d’un fournisseur OAuth 2.0. Pour configurer des en-têtes supplémentaires, ouvrez les Paramètres de la connexion et, dans le champ **En-têtes personnalisés**, indiquez un objet JSON contenant les en-têtes personnalisés sous forme de paires clé-valeur :

```json lines
{
    "Header1" : "Value",
    "Header2" : "Value"
}
```

Prenons l’exemple d’un cas où un fournisseur d’identité peut vous demander de transmettre un en-tête `Authorization` avec des informations d’identification d’[authentification Basic](https://en.wikipedia.org/wiki/Basic_access_authentication). Dans ce scénario, vous pouvez préciser l’objet JSON suivant dans le champ **En-têtes personnalisés** :

```json lines
{
  "Authorization": "Basic [your credentials]"
}
```

Où `[your credentials]` correspond aux informations d’identification réelles à envoyer au fournisseur d’identité.
