<Card title="Avant de commencer">
  Pour utiliser le Back-Channel Logout, votre application doit répondre aux exigences suivantes :

  * Le point de terminaison URI de votre OIDC Back-Channel Logout doit être accessible sur Internet pour que votre tenant Auth0 puisse y accéder.
  * Les applications prêtes pour la Production doivent utiliser TLS. Pour en savoir plus, consultez [TLS (SSL) Versions and Ciphers](/fr-CA/docs/customize/custom-domains/self-managed-certificates/tls-ssl).
  * L’application doit pouvoir stocker l’identifiant de session (`sid`) fourni et l’associer aux sessions utilisateur qu’elle crée. Nous recommandons aux applications de production d’utiliser un stockage de session persistant.
  * L’application doit pouvoir récupérer une session existante à l’aide du `sid` pendant le processus de logout sans utiliser de cookies côté client. Les cookies se trouvent dans le navigateur et sont inaccessibles au callback point de terminaison de logout.
</Card>

<div id="availability">
  ## Disponibilité
</div>

OIDC Back-Channel Logout est offert à tous les clients ayant un forfait Enterprise. Vous devriez vérifier le point de terminaison de métadonnées OIDC standard `/.well-known/*` pour déterminer si votre application répond aux exigences.

```json lines
GET https://acme.eu.auth0.com/.well-known/openid-configuration

HTTP 200
{ ..., "backchannel_logout_supported": true, 
  "backchannel_logout_session_supported": true }
```

<div id="back-channel-logout-restrictions">
  ## Restrictions relatives à Back-Channel Logout
</div>

Les URL de Back-Channel Logout sont appelées sur des points de terminaison exposés publiquement et doivent respecter certaines restrictions :

1. Vous devez utiliser le protocole HTTPS. Le protocole HTTP non chiffré ou tout autre protocole n’est pas autorisé.
2. Vous ne devez pas utiliser de sous-domaines Auth0. Voici quelques sous-domaines Auth0 :

   * auth0.com
   * auth0app.com
   * webtask.io
   * webtask.run
3. Nous ne recommandons pas d’utiliser des adresses IP sans nom de domaine. Les adresses IP doivent être publiques pour pouvoir utiliser Back-Channel Logout. Les adresses IP provenant de plages internes, réservées ou de bouclage ne sont pas autorisées.

<div id="configure-auth0">
  ## Configurer Auth0
</div>

Enregistrez votre application pour recevoir des Logout Tokens via <Tooltip tip="Auth0 Dashboard : principal produit d’Auth0 pour configurer vos services." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip> ou <Tooltip tip="Auth0 Dashboard : principal produit d’Auth0 pour configurer vos services." cta="Voir le glossaire" href="/fr-CA/docs/glossary?term=Management+API">Management API</Tooltip>.

<Tabs>
  <Tab title="Auth0 Dashboard">
    #### Inscrire des applications

    1. Accédez à [**Auth0 Dashboard &gt; Applications**](https://manage.auth0.com/#/applications).
    2. Choisissez l’application que vous voulez enregistrer.
    3. Sélectionnez l’onglet **Settings**.
    4. Sous **OpenID Connect Back-Channel Logout &gt; Back-Channel Logout URI**, ajoutez l’URI de déconnexion de l’application qui recevra les logout&#95;tokens
    5. Une fois terminé, sélectionnez **Save Changes**.

       <Frame>![Auth0 Dashboard > Applications > Settings](/docs/images/cdy7uua7fh8z/19uQRkVswdmaO8gHTVTXCc/0564bfeda05bc7df90af9e8ee852047f/Screenshot_2024-11-05_at_13.31.54.png)</Frame>

    #### Désinscrire des applications

    La désinscription de votre application empêche le service de suivre les nouvelles ouvertures de session et d’envoyer des événements de déconnexion. Le service ignore les événements de déconnexion en attente une fois votre application désinscrite.

    Pour désinscrire votre application, supprimez l’URL de Back-Channel Logout.

    1. Accédez à [Auth0 Dashboard &gt; Applications](http://manage.auth0.com/#/applications).
    2. Choisissez l’application que vous voulez enregistrer.
    3. Sélectionnez l’onglet **Settings**.
    4. Sous **OpenID Connect Back-Channel Logout &gt; Back-Channel Logout URI**
    5. Supprimez l’URL back-channel.
    6. Une fois terminé, sélectionnez **Save Changes**.

       <Frame>![Auth0 Dashboard > Applications > Settings](/docs/images/cdy7uua7fh8z/5SlrBhwrNbJx0Wdw2kQ19t/9b69eb926e44c35b426daa76a851a5db/Screenshot_2024-11-05_at_13.31.54.png)</Frame>

    À des fins d’audit, le service consigne toujours l’inscription ou la désinscription des URL de Back-Channel Logout comme `API Operation Event` dans les journaux du tenant Auth0. Pour en savoir plus, consultez [Logs](/fr-CA/docs/deploy-monitor/logs).
  </Tab>

  <Tab title="Management API">
    #### Inscrire une application

    Ajoutez la propriété `oidc_logout` pour effectuer un `PATCH` sur un client au moyen de la Management API :

    ```bash cURL lines
    PATCH /api/v2/clients/{id}
    "oidc_logout": {
      "backchannel_logout_urls": ["http://acme.eu.auth0.com/events"]
    }
    ```

    <Callout icon="file-lines" color="#0EA5E9" iconType="regular">
      La requête peut inclure d’autres paramètres de l’application.
    </Callout>

    #### Désinscrire une application

    Supprimez le paramètre `oidc_logout` :

    ```bash cURL lines
    PATCH https://acme.eu.auth0.com/api/v2/clients/{id}
    {
      "oidc_logout": {
        "backchannel_logout_urls": []
      }
    }
    ```
  </Tab>

<div id="configure-your-application">
  ## Configurez votre application
</div>

Une fois que vous avez configuré Back-Channel Logout dans l’Auth0 Dashboard ou au moyen de la Management API, configurez votre application pour utiliser le service selon la technologie ou le framework.

1. Implémentez l’authentification de l’utilisateur final selon le type de votre application.

   1. Les utilisateurs finaux doivent pouvoir se connecter à l’application, et une session doit être créée.
   2. Un ID token doit être émis par Auth0 et être accessible dans le backend de l’application pour un traitement ultérieur.
2. Étendez le processus de connexion pour enregistrer le `sid` et, au besoin, les claims `sub` après la validation de l’ID token.

   1. Ces claims doivent être enregistrées dans la session d’application en cours.
   2. Les fonctions de gestion de session doivent pouvoir récupérer une session précise à partir de la valeur `sid`.
3. Configurez le point de terminaison Back-Channel Logout :

   1. Le point de terminaison doit traiter uniquement les requests `HTTP POST`.
   2. Extrayez le paramètre `logout_token` et validez-le comme un JWT ordinaire conformément à la spécification.
   3. Vérifiez que le token contient une claim `events` avec une valeur d’objet JSON et un membre nommé `http://schemas.openid.net/event/backchannel-logout`.
   4. Vérifiez que le token contient les claims `sid` et/ou `sub`.
   5. Vérifiez que le token ne contient PAS la claim `nonce`. Cela est nécessaire pour prévenir les abus en distinguant le Logout Token de l’ID token.
   6. Une fois le token validé, récupérez la session correspondant à la valeur `sid` et/ou `sub` reçue, puis terminez-la. Le processus exact de fin de session de l’application dépend des détails de l’implémentation. Par exemple, il peut être nécessaire de transmettre cet événement au front-end.

<div id="expected-responses">
  ## Réponses attendues
</div>

* **HTTP 200**:  Confirme la déconnexion de l’utilisateur de l’application concernée.
* **HTTP 400**: Indique une requête incorrecte. La requête n’est pas comprise ou la validation du jeton a échoué. Auth0 consigne le problème dans les journaux du tenant, mais ne tente pas d’autres requêtes pour cette session en particulier.

<div id="troubleshoot-issues">
  ## Dépannage
</div>

<div id="application-did-not-receive-the-logout-events">
  ### L’application n’a pas reçu les événement de déconnexion
</div>

Si votre application n’a pas reçu de requête de logout après un événement de déconnexion.

1. Assurez-vous qu’une URL de Back-Channel Logout est enregistrée pour votre application dans Auth0 Dashboard.
2. Assurez-vous que l’URL de Back-Channel Logout est accessible depuis le tenant Auth0.
3. Assurez-vous qu’une session valide est établie. L’utilisateur final doit se connecter à cette application précise par l’entremise d’Auth0.

   <Callout icon="file-lines" color="#0EA5E9" iconType="regular">
     Les applications reçoivent des événement de déconnexion uniquement lorsque les utilisateurs finaux se connectent à cette application précise avec Auth0. Si l’utilisateur final se connecte à d’autres applications, les événement de déconnexion ne sont pas déclenchés.
   </Callout>
4. Vérifiez les logs du tenant Auth0 pour repérer les messages indiquant un échec de livraison du message de logout.
5. Assurez-vous que le logout est déclenché au moyen du point de terminaison de logout standard. Les autres événements ne déclenchent pas les événement de déconnexion.
6. Si possible, vérifiez s’il y a des requêtes bloquées dans les logs du serveur Web et/ou du pare-feu de l’application.

<div id="i-cant-find-oidc-back-channel-logout-tenant-logs">
  #### Je ne trouve pas les journaux du tenant pour OIDC Back-Channel Logout
</div>

Cette fonctionnalité sera déployée progressivement pour tous les tenants à compter du 3 octobre 2023. Il se peut que vous receviez quand même des [codes d’événement des journaux du tenant](/fr-CA/docs/deploy-monitor/logs/log-event-type-codes) `sslo` pour `oidc_backchannel_logout_succeeded` ou `fslo` pour `oidc_backchannel_logout_failed`.

<div id="client-app-cannot-verify-the-received-logout-token">
  ### L’application cliente ne peut pas vérifier le Logout Token reçu
</div>

Si la transaction échoue toujours avec une erreur 400 :

1. Vérifiez si le token est un JWT standard encodé en base64. Certains serveurs web peuvent tronquer les paramètres longs. Pour en savoir plus, consultez [Signing Algorithms](/fr-CA/docs/get-started/applications/signing-algorithms).
2. Si possible, capturez un token et vérifiez qu’il s’agit bien d’un JWT. Utilisez une source fiable, comme [JWT.IO](http://jwt.io).
3. Assurez-vous que la fonction de vérification récupère dynamiquement la clé de signature du tenant via [JSON Web Key Sets](/fr-CA/docs/secure/tokens/json-web-tokens/json-web-key-sets) (JWKS).

   <Callout icon="file-lines" color="#0EA5E9" iconType="regular">
     Nous ne recommandons pas d’intégrer une clé statique en dur. Si votre configuration exige qu’une clé statique soit codée en dur, assurez-vous qu’elle contient la clé publique la plus récente du tenant Auth0.
   </Callout>

<div id="response-timeout-errors">
  ### Erreurs de délai de réponse
</div>

Auth0 attendra cinq secondes que l’URL de OIDC Back-Channel Logout de l’application réponde. Passé ce délai, il consignera « Failed OIDC Back-Channel Logout request » dans les journaux du tenant, avec une description de réponse vide.
