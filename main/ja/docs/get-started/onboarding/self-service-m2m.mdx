ユースケースの対象が、API やバックエンドサーバーなどの非対話型の<Tooltip tip="Audience: 発行されたトークンの audience を一意に識別する値です。トークン内では aud という名前で表され、その値には、ID トークンの場合はアプリケーション（Client ID）の ID、Access Token の場合は API（API 識別子）の ID が含まれます。" cta="用語集を見る" href="/ja/docs/glossary?term=audience">audience</Tooltip>である場合は、machine-to-machine (M2M) 構成でオンボーディングします。

<div id="use-cases">
  ## ユースケース
</div>

次のいずれかに当てはまる場合は、M2M オンボーディング パスを使用してください。

* サービス間通信をサポートしている
* 保護されたリソースや API にアクセスする必要がある、サーバー上で実行されるスケジュール済みのジョブや cron タスクがある
* IoT デバイスがバックエンドサービスや API と通信できるようにしている
* ユーザーの関与なしに、またはユーザートークンの有効期限が切れた後に、他の API レイヤーと通信する必要がある API レイヤーがある
* ユーザーが認証される前に呼び出す必要がある特権 API がある (つまり、Auth0 テナント内の Action やカスタムデータベーススクリプトから)
* バックエンドサービスの管理に API Gateway を使用している
* デーモンやバックエンドサービスなど、人による操作を伴わない非対話型のアプリケーションやその他のツールを使用またはサポートしている

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  これらのサービスでも、認証には引き続き M2M アクセストークンが必要です。
</Callout>

<div id="how-to-use-this-guide">
  ## このガイドの使い方
</div>

このガイドでは、Auth0 で M2M 実装を構築するための進め方を紹介します。確認しておくべき考慮事項、ベストプラクティス、概念をまとめています。

* Architecture では、ソフトウェア開発ライフサイクルと既存のインフラストラクチャを支えられるように Auth0 を設定することを推奨します。
* Create an account では、Auth0 で API インスタンスを作成し、machine-to-machine authentication に必要な authentication flow (またはグラント) に対応するアプリケーションを作成する手順を説明します。
* Authentication では、authentication に必要なグラントに加えて、設定できる <Tooltip tip="アクセストークン: API へのアクセスに使用される、opaque な文字列または JWT 形式の認可クレデンシャル。" cta="用語集を見る" href="/ja/docs/glossary?term=access+tokens">アクセストークン</Tooltip> や permissions (またはスコープ) について説明します。
* Branding では、証明書の管理方法に応じて、<Tooltip tip="カスタムドメイン: 特別な名前、または独自名を持つサードパーティのドメイン。" cta="用語集を見る" href="/ja/docs/glossary?term=Custom+Domains">カスタムドメイン</Tooltip> の設定方法に関する情報の参照先を案内します。
* Deployment Automations では、デプロイを支援する当社のツールについて確認できます。
* Quality Assurance では、単体テストや、<Tooltip tip="Auth0 Dashboard: サービスを設定するための Auth0 の主要製品です。" cta="用語集を見る" href="/ja/docs/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip> で提供している readiness checks について詳しく学べます。

<div id="architecture">
  ## アーキテクチャ
</div>

Auth0 のアカウントやテナント、あるいは Auth0 サービスのグループや構造を設定する前に、既存のインフラストラクチャの全体像を整理しておくと、現在の環境で Auth0 の機能を最大限に活用できます。

一般的なシナリオで述べたように、Auth0 を設定する前に、アプリケーションドメイン、ネットワークドメイン、または M2M デバイスドメインに存在する、他の非対話型テクノロジーについても考慮する必要があるかもしれません。M2M シナリオの例を確認するには、[Server + API](/ja/docs/get-started/architecture-scenarios/server-application-api/part-1) をお読みください。Node を使って API デプロイをテストする実践的なラボを試すには、こちらの [GitHub repository](https://www.google.com/url?q=https://github.com/auth0-training/labs-node-working-with-apis\&sa=D\&source=docs\&ust=1723578190947730\&usg=AOvVaw3IXRQG9ogH4pW6LEY047T6) をご覧ください。

現在の技術スタックを可視化するとともに、現在の Software Development Lifecycle (SDLC) に Auth0 をどのように組み込むかを計画しておくとよいでしょう。これは、必要なテナント数を判断するのに役立ちます。

<div id="considerations">
  ### 考慮事項
</div>

新しいアカウントを作成したり、最初のテナントを設定したりする前に、次の点を検討してください。

* 特定のエンドポイントを呼び出すために、API をどのように分割またはグループ化するか。

  * これにより、アクセストークンの audience やその他のクレームが決まる場合があります。
* リソースを利用するサードパーティのコンシューマーは、呼び出しのたびにアクセストークンを要求する可能性があります。呼び出しが過剰になると、[レート制限](/ja/docs/troubleshoot/customer-support/operational-policies/rate-limit-policy/rate-limit-configurations/essentials-professional-b2b) に影響するおそれがあります。

  * API Gateway を使用すると、サードパーティが要求できるアクセストークンの数を制限できます。API Gateway と Auth0 の詳細については、[Access Gateway で IDプロバイダーを構成する](/ja/docs/get-started/auth0-overview/create-applications/configure-an-identity-provider-in-access-gateway) を参照してください。

<div id="create-an-account">
  ## アカウントを作成する
</div>

アーキテクチャの計画ができたら、次は Auth0 アカウントとテナントを設定します。Auth0 のサービスにサインアップすると、最初の[テナント](/ja/docs/get-started/auth0-overview/create-tenants)を作成します。ここで Auth0 のアセット、サービス、リソースを設定します。まずは[サインアップ](https://auth0.com/signup)してください。

<Card title="開始前に">
  Auth0 Dashboard または Auth0 Management API で、次のものを作成します。

  * ご利用の API を表す API
  * Client Credential Flow を使用するための M2M アプリケーション

  アカウントを作成する前に、いくつかの設定項目をあらかじめ検討しておくとよいでしょう。

  * テナント名は Auth0 ドメインの一部になります。名前を決める前に、[テナントの特性](/ja/docs/get-started/auth0-overview/create-tenants#create-a-tenant-and-domain)を確認してください。
  * ユースケースに必要な Auth0 の機能。一部の機能は Professional プランと Enterprise プランでのみ利用できます。
  * 開発、ステージング、本番環境など、複数の環境をサポートする必要があるかどうかを検討してください。詳しくは、[Set Up Multiple Environments](/ja/docs/get-started/auth0-overview/create-tenants/set-up-multiple-environments)を参照してください。
  * テナントに登録したいサードパーティ製アプリケーションに関するユースケースがある場合は、[OIDC Client Registration](https://openid.net/specs/openid-connect-registration-1_0.html) 仕様に基づく [Dynamic Application Registration](/ja/docs/get-started/applications/dynamic-client-registration) を使用できます。
</Card>

<div id="provision-a-tenant">
  ### テナントを設定する
</div>

アーキテクチャの計画が固まったら、Auth0 アカウントと<Tooltip tip="テナント: Auth0 では、単一のソフトウェアインスタンスに対して、特定の権限を持つ共通のアクセスを共有する、論理的に分離されたユーザーグループを指します。複数のテナントが同じマシン上で稼働していても、どのテナントも他のテナントのデータにアクセスすることはできません。" cta="用語集を見る" href="/ja/docs/glossary?term=Tenant">テナント</Tooltip>を設定します。

<div id="authentication">
  ## 認証
</div>

ある API から別の API を呼び出す場合や、認証済みユーザーのコンテキストがない状況では、ユーザーではなくアプリケーションを認可する仕組みが必要です。これは 1 ステップのプロセスで、アプリケーションは (`client_id` と `client_secret` を使用して) 認証され、その後 1 回の呼び出しで認可されます。

非対話型のアプリケーションや、認証を行うサービスでは、クライアントグラントまたは認証フローを選択する必要があります。<Tooltip tip="OAuth 2.0: 認可プロトコルとワークフローを定義する認可フレームワーク。" cta="用語集を見る" href="/ja/docs/glossary?term=OAuth+2.0">OAuth 2.0</Tooltip> の [クライアント認証情報フロー](https://tools.ietf.org/html/rfc6749#section-4.4) では人による操作は不要で、M2M アプリケーションに最適です。

<Card title="始める前に">
  Auth0 Dashboard または Management API では、次のことを行います。

  * アプリケーションでクライアント認証情報フローを使用するよう設定する
  * M2M アクセストークンのスコープを更新する

  認証方法を設定する前に、次のことを確認してください。

  * マシン間認証のための [クライアント認証情報フロー](/ja/docs/get-started/authentication-and-authorization-flow/client-credentials-flow) を確認します。これは、非対話型の認証および認可のワークフローです。
  * API のアクセスレベルを判断します。これにより、API の作成時に設定する [スコープ](/ja/docs/get-started/apis/scopes/api-scopes) や permissions を決めやすくなります。
</Card>

<div id="configure-the-client-credential-flow">
  ### クライアントクレデンシャルフローを設定する
</div>

Auth0 Dashboard または <Tooltip tip="Management API: お客様が管理タスクを実行できるようにする製品です。" cta="用語集を見る" href="/ja/docs/glossary?term=Management+API">Management API</Tooltip> を使用して、クライアントクレデンシャルを提示してアクセストークンを取得できるように認証フローを設定できます。

Auth0 Dashboard または Management API を使用するには、[グラントタイプを更新する](/ja/docs/get-started/applications/update-grant-types) の手順に従ってください。

<div id="m2m-access-tokens">
  ### M2M アクセストークン
</div>

トークンベースの認証では、非対話型クライアントは [Authentication API の token endpoint](/ja/docs/customize/actions/explore-triggers/machine-to-machine-trigger) への呼び出しで `client_id` と `client_secret` を指定し、[アクセストークン](/ja/docs/secure/tokens/access-tokens) を取得します。このアクセストークンにより、保護された API へのアクセスが可能になります。

デフォルトのプロファイル、つまり形式は、2 つあるトークンプロファイルのうち Auth0 トークンプロファイルです。トークンプロファイルは RFC 9068 に変更することもできます。詳しくは、[Access Token Profiles](/ja/docs/secure/tokens/access-tokens/access-token-profiles) をご覧ください。トークンが有効かどうかを検証するために、API は [Signing Algorithms](/ja/docs/get-started/applications/signing-algorithms) を確認します。デフォルトの <Tooltip tip="Signing Algorithm: トークンが改ざんされていないことを保証するために、トークンにデジタル署名する際に使用されるアルゴリズム。" cta="用語集を見る" href="/ja/docs/glossary?term=signing+algorithm">署名アルゴリズム</Tooltip> は RSA256 で、キー ベースのアルゴリズムです。

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Auth0 では、資格情報として Client ID と Client Secret を指定する方法以外のクライアント認証方法もサポートしています。これらの方法は、M2M 構成向けの推奨方法である [Private Key JWT](/ja/docs/get-started/authentication-and-authorization-flow/authenticate-with-private-key-jwt) を含め、Enterprise plan で利用できます。詳しくは、[Application Credentials](/ja/docs/secure/application-credentials) をご覧ください。
</Callout>

<div id="example">
  #### 例
</div>

`/oauth/token` エンドポイントへのリクエストは、以下のサンプルのようになります。

<AuthCodeGroup>
  ```bash cURL
  curl --request POST \
    --url 'https://{yourDomain}/oauth/token' \
    --header 'content-type: application/x-www-form-urlencoded' \
    --data grant_type=client_credentials \
    --data client_id={yourClientId} \
    --data client_secret={yourClientSecret} \
    --data audience=YOUR_API_IDENTIFIER
  ```

  ```csharp C#
  var client = new RestClient("https://{yourDomain}/oauth/token");
  var request = new RestRequest(Method.POST);
  request.AddHeader("content-type", "application/x-www-form-urlencoded");
  request.AddParameter("application/x-www-form-urlencoded", "grant_type=client_credentials&client_id={yourClientId}&client_secret={yourClientSecret}&audience=YOUR_API_IDENTIFIER", ParameterType.RequestBody);
  IRestResponse response = client.Execute(request);
  ```

  ```go Go
  package main

  import (
  	"fmt"
  	"strings"
  	"net/http"
  	"io/ioutil"
  )

  func main() {

  	url := "https://{yourDomain}/oauth/token"

  	payload := strings.NewReader("grant_type=client_credentials&client_id={yourClientId}&client_secret={yourClientSecret}&audience=YOUR_API_IDENTIFIER")

  	req, _ := http.NewRequest("POST", url, payload)

  	req.Header.Add("content-type", "application/x-www-form-urlencoded")

  	res, _ := http.DefaultClient.Do(req)

  	defer res.Body.Close()
  	body, _ := ioutil.ReadAll(res.Body)

  	fmt.Println(res)
  	fmt.Println(string(body))

  }
  ```

  ```php PHP
  curl_close($curl);

  if ($err) {
    echo "cURL Error #:" . $err;
  } else {
    echo $response;
  }
  ```

  ```python Python
  import http.client

  conn = http.client.HTTPSConnection("")

  payload = "grant_type=client_credentials&client_id={yourClientId}&client_secret={yourClientSecret}&audience=YOUR_API_IDENTIFIER"

  headers = { 'content-type': "application/x-www-form-urlencoded" }

  conn.request("POST", "/{yourDomain}/oauth/token", payload, headers)

  res = conn.getresponse()
  data = res.read()

  print(data.decode("utf-8"))
  ```

  ```ruby Ruby
  require 'uri'
  require 'net/http'
  require 'openssl'

  url = URI("https://{yourDomain}/oauth/token")

  http = Net::HTTP.new(url.host, url.port)
  http.use_ssl = true
  http.verify_mode = OpenSSL::SSL::VERIFY_NONE

  request = Net::HTTP::Post.new(url)
  request["content-type"] = 'application/x-www-form-urlencoded'
  request.body = "grant_type=client_credentials&client_id={yourClientId}&client_secret={yourClientSecret}&audience=YOUR_API_IDENTIFIER"

  response = http.request(request)
  puts response.read_body
  ```
</AuthCodeGroup>

レスポンスは、以下のサンプルのようになります。

```json lines
HTTP/1.1 200 OK
Content-Type: application/json
{
  "access_token":"eyJz93a...k4laUWw",
  "token_type":"Bearer",
  "expires_in":86400
}
```

<div id="token-expiration">
  #### トークンの有効期限
</div>

アクセストークンには、有効期間の上限があります。通信はバックチャネルで行われるため、セッションを延長する目的で<Tooltip tip="リフレッシュトークン: ユーザーに再度ログインを求めることなく、新しいアクセストークンを取得するために使用されるトークン。" cta="用語集を見る" href="/ja/docs/glossary?term=refresh+tokens">リフレッシュトークン</Tooltip>を使用することはできません。そのため、アクセストークンの有効期限は1時間に設定することを検討してください。環境に応じて、セキュリティとパフォーマンスのバランスを調整する必要がある場合があります。詳しくは、[アクセストークンの有効期間を更新する](/ja/docs/secure/tokens/access-tokens/update-access-token-lifetime)をご覧ください。

<div id="scopes">
  ### スコープ
</div>

非対話型のクライアントやサービスが API を呼び出す前に、その API で許可する権限 (スコープ) を定義する必要があります。Authentication API への認証リクエストに含めるスコープは、Auth0 Dashboard で設定できます。API スコープの使用方法についてさらに多くの例を確認するには、[API スコープ](/ja/docs/get-started/apis/scopes/api-scopes)を参照してください。

スコープを設定するには、Auth0 Dashboard を使用する場合は [API の権限を追加する](/ja/docs/get-started/apis/add-api-permissions) の手順に従うか、Management API 用に提供されているサンプルを使用してください。

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  access token にカスタムクレームを追加するには、Actions の Machine to Machine Flow を使用できます。詳しくは、Machine to Machine Flow を参照してください。
</Callout>

<div id="branding">
  ## ブランディング
</div>

対話を伴わないクライアントやバックチャネルで動作するサービス向けであっても、既存のブランドのルック＆フィールに合わせて体験をカスタマイズできます。

<div id="custom-domains">
  ### カスタムドメイン
</div>

Auth0 では、アクセストークンをリクエストするために `/authorize` エンドポイントを呼び出す際、カスタムドメインを使用できます。

<Card title="M2M オンボーディング - カスタムドメイン">
  Auth0 Dashboard では、次の作業が必要です。

  * Auth0 のサービスで使用する前に、ドメインを登録して検証する。
  * 証明書を自分で管理するか、Auth0 管理の証明書を使用するかを決定する。証明書の詳細については、[Certificate management options](/ja/docs/customize/custom-domains#certificate-management-options) を参照してください。
  * 自己管理証明書で使用する TLS (SSL) のバージョンと暗号スイートが Auth0 でサポート対象であることを確認する。詳細については、[TLS (SSL) Versions and Ciphers](/ja/docs/customize/custom-domains/self-managed-certificates/tls-ssl) を参照してください。
</Card>

1. Auth0 管理の証明書を使用してカスタムドメインを設定するには、[Configure Custom Domains with Auth0-Managed Certificates](/ja/docs/customize/custom-domains/auth0-managed-certificates) の手順に従ってください。

   1. 証明書を自分で管理する場合は、[Configure Custom Domains with Self-Managed Certificates](/ja/docs/customize/custom-domains/self-managed-certificates) の手順に従ってください。

      <Callout icon="file-lines" color="#0EA5E9" iconType="regular">
        カスタムドメインで証明書を管理するには、Enterprise サブスクリプションが必要です。詳細については、[Auth0 Pricing](https://auth0.com/pricing/) と [Login](/ja/docs/authenticate/login) を参照してください。
      </Callout>
2. [API configuration with custom domains](/ja/docs/customize/custom-domains/configure-features-to-use-custom-domains#apis) を確認してください。カスタムドメインに対応させるため、API の設定調整が必要になる場合があります。

<div id="deployment-automation">
  ## デプロイ自動化
</div>

Auth0 では、デプロイ自動化の方法としていくつかの選択肢をサポートしており、これらは組み合わせて使うこともできます。

<Card title="ベストプラクティス">
  どの方法でデプロイ自動化を構成する場合でも、デプロイ前にカスタムコードと Actions の単体テストを行い、デプロイ後にはテナントに対する統合テストも実施することをお勧めします。
</Card>

<div id="deploy-cli-tool">
  ### Deploy CLI ツール
</div>

Architecture セクションで推奨されているように、開発、テスト、本番用に Auth0 テナントを用意する必要があります。これらのテナントは、品質確認やテストのために同一の設定にしておくべきですが、環境間で設定に不一致があるとエラーが発生することがあります。たとえば、各環境では <Tooltip tip="Client ID: Auth0 から登録済みリソースに付与される識別値。" cta="用語集を見る" href="/ja/docs/glossary?term=Client+IDs">クライアント ID</Tooltip> や <Tooltip tip="Client ID: Auth0 から登録済みリソースに付与される識別値。" cta="用語集を見る" href="/ja/docs/glossary?term=Client+Secrets">クライアント シークレット</Tooltip> がそれぞれ異なります。

こうした不一致によるエラーを軽減するには、Deploy CLI Tool を使用して既存の CI/CD パイプラインに Auth0 インスタンスを統合できます。  動的なキーワード置換を使うと、類似した設定を共有するテナント間で環境変数を置き換えることができます。詳しくは、[Deploy CLI Tool](/ja/docs/deploy-monitor/deploy-cli-tool) と [Keyword Replacement](/ja/docs/deploy-monitor/deploy-cli-tool/keyword-replacement) を参照してください。

<div id="actions-real-time-logs">
  ### Actions リアルタイムログ
</div>

Actions リアルタイムログには、`console.log` の出力やその他の例外を含む、カスタムコードのすべてのログがリアルタイムで表示されます。Auth0 Actions やその他のカスタムロジックを使用している場合は、この拡張機能を使ってデバッグやトラブルシューティングを行えます。インストール方法と設定の詳細については、[Actions リアルタイムログ](/ja/docs/customize/actions/actions-real-time-logs)をお読みください。

<div id="quality-assurance">
  ## 品質保証
</div>

本番稼働前に問題を発見するうえで、品質保証は重要です。プロジェクトの性質によっては、Auth0 との連携の一環として検討すべき品質保証テストがいくつかあります。

* 想定外の本番環境の負荷がかかった場合、API はどのように動作するでしょうか。
* サードパーティアプリケーションによって、レート制限はどのような影響を受けるでしょうか。

Auth0 ウィジェットや <Tooltip tip="Universal Login: アプリケーションは、ユーザーのアイデンティティを確認するために、Auth0 の認可サーバーでホストされる Universal Login にリダイレクトされます。" cta="用語集を表示" href="/ja/docs/glossary?term=Universal+Login">Universal Login</Tooltip> のような機能を使用していない場合、多くのブラウザーやデバイスにおいて、ユーザビリティやアクセシビリティに関する組み込みのベストプラクティスをそのまま利用することはできません。機能要件が満たされ、想定外の事象にも正しく対処できるようにするため、アプリケーションと Auth0 の連携テストや、Auth0 Actions などの個々の拡張モジュールに対する単体テストについてのガイダンスを提供しています。また、Auth0 の[ペネトレーションテストポリシー](/ja/docs/troubleshoot/customer-support/operational-policies/penetration-testing-policy)を確認し、[負荷テストポリシー](/ja/docs/troubleshoot/customer-support/operational-policies/load-testing-policy)とあわせて活用できるモックテストを実施することをお勧めします。これにより、想定外の負荷がかかった場合でもアプリケーションが適切に動作することを確認できます。

<div id="unit-testing">
  ### 単体テスト
</div>

単体テストとは、Auth0 Actions のような拡張の各単位を検証することです。独自のコードを使用している場合は、デプロイ前にテストフレームワーク ([Mocha](https://mochajs.org/) など) を使って、その追加コードをテストすることをお勧めします。

<div id="mock-testing">
  ### モックテスト
</div>

Auth0 の [負荷テストポリシー](/ja/docs/troubleshoot/customer-support/operational-policies/load-testing-policy)と負荷テストを実施したいというニーズの両立策として、Auth0 のエンドポイントをモック化してテストするのは一般的な方法です。これは、テストを制限することなく、想定しているインターフェースで構成が正しく機能することを確認するうえで有効です。また、[MockServer](http://www.mock-server.com/)、[JSON Server](https://github.com/typicode/json-server)、あるいは [Postman](https://learning.getpostman.com/docs/postman/mock_servers/setting_up_mock/) などのツールも活用できます。

<div id="deployment">
  ## デプロイ
</div>

[デプロイと監視](/ja/docs/deploy-monitor)セクションでは、デプロイのベストプラクティスに関するガイダンスを提供しています。[デプロイ前](/ja/docs/deploy-monitor/pre-deployment-checks)[チェック](/ja/docs/deploy-monitor/pre-deployment-checks)、特に組み込みの[Auth0 Dashboard Readiness Checks](/ja/docs/deploy-monitor/pre-deployment-checks/how-to-run-production-checks)を確認することをお勧めします。

Readiness Check を確認するには、[Auth0 Dashboard &gt; Run Readiness Checks](https://manage.auth0.com/#/production-checks)で、テナント名と環境タグの下にあるドロップダウンメニューを選択します。

<Frame>![Auth0 Dashboard > Readiness Checklist](/docs/images/cdy7uua7fh8z/7iu1CzH0NgaXJbLBNWXZli/b598b8cdaad0c676e83a0fb0595d4603/Readiness_Checks_-_English.png)</Frame>

フィルターを使用して、選択したアプリケーションに Readiness Checks を適用できます。**これらのチェックは、設定済みの API には適用されません**。

特定の設定に当てはまらないチェックについては、**Dismiss** を選択して最終結果から除外できます。

本番公開前の最終確認として、[デプロイのベストプラクティス](/ja/docs/deploy-monitor/deployment-best-practices)を確認し、[ログ](/ja/docs/deploy-monitor/logs)を活用してサービスを監視することをお勧めします。
