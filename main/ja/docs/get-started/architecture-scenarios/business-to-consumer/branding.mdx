validatedOn: 2026-02-17

Auth0 は、組織のブランド要件やユーザーの期待に沿ったルック＆フィールにカスタマイズできます。Auth0 の各種要素をブランディングすることで、顧客に一貫したユーザー体験を提供できるだけでなく、信頼できる安全なプロバイダーの製品を利用しているという安心感も与えられます。

Auth0 は、[国際化 (I18N)](/ja/docs/customize/internationalization-and-localization) と <Tooltip tip="ローカライゼーション: 特定の地域や言語に合わせてコンテンツを適応させること。" cta="用語集を見る" href="/ja/docs/glossary?term=Localization">ローカライゼーション</Tooltip> (L10N) の両方をサポートしており、国際的な顧客向けにブランディングを行う場合、どちらも重要です。Auth0 Lock UI ウィジェットのような標準搭載の要素は、複数言語のサポートがあらかじめ有効になっており、必要なものが[まだ用意されていない](/ja/docs/customize/internationalization-and-localization/lock-internationalization)場合は、さらに言語を追加できる拡張性も備えています。

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  ほぼすべてのアプリケーションでは、何らかの形で国際化やローカライゼーションが必要になります。Auth0 を使えば簡単に追加できますが、最初から考慮しておく必要があります。たとえば、後からローカライゼーションを追加対応しようとすると、手遅れになってからでは大変な作業になることがあります。
</Callout>

どの項目をブランド化したいか、またそれらをどのようにブランド化するのが最適かを検討する際には、確認しておきたい点がいくつかあります。

* ログインページをブランド化する必要がありますか？
* ログインページをローカライズする必要がありますか？
* 単にブランド化するだけでなく、ユーザーの好みに応じて内容が変わるようにメールをカスタマイズするにはどうすればよいですか？
* ユーザーがログインページを見たときに、引き続き自社ドメイン上にいることをどのように認識できますか？
* 追加のブラウザーセキュリティを提供するには何が必要ですか (例: Extended Validation の実装) ？
* エラーが発生した場合、ユーザーをどこに案内したいですか？

Auth0 では、[Universal Login](#universal-login-and-login-pages) や [Password Reset](#password-reset-page-customization) といった Auth0 ページのカスタマイズや設定に関して、非常に高い柔軟性が提供されています。そのため、必要な UX のルック＆フィールをほぼ自由に構築できます。多くの場合は、標準のまま、あるいは少し手を加えるだけで十分です。一方で、ブランド価値や認知度の観点から、より大幅なカスタマイズが必要になる場合もあります。この柔軟性は Auth0 のページだけでなく、拡張性を通じて [メールテンプレート](#email-template-customization) にも適用できます。さらに、Auth0 の [カスタムドメイン](#custom-domain-naming) 機能により、ユーザーは安全性やセキュリティの面でいっそう信頼感と安心感を持てるようになり、ブランド認知の向上にもつながります。

Auth0 ではエラー時に既定の情報が提供されますが、標準の情報だけでは、あなたしか提供できない文脈が欠けているため、ややわかりにくい場合があります。ただし、Auth0 の [エラーページのカスタマイズ](#error-page-customization) に関するガイダンスを利用すれば、自社のサポート組織を通じて、より状況に即した情報を提供できるため、その問題を軽減できます。

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  問題が発生したユーザーに役立つリソースを提供するために、わかりやすい名前とロゴを設定し、あわせて組織のサポート用メールアドレスと URL も提供してください。方法については、[Dashboard Tenant Settings](/ja/docs/get-started/tenant-settings#settings) を参照してください。
</Callout>

<div id="universal-login-and-login-pages">
  ## Universal Login とログインページ
</div>

[Universal Login](/ja/docs/authenticate/login/auth0-universal-login) は、ユーザー認証に推奨される方法であり、ログインページの使用を中心としています。ログインページは、組織のブランディング要件に合わせてカスタマイズできます。

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Universal Login ページのスクリプトをカスタマイズする場合は、バージョン管理を利用することを強くお勧めします。そのためには、[deployment automation](/ja/docs/get-started/architecture-scenarios/business-to-consumer/deployment) または [alternative strategies](/ja/docs/customize/login-pages/classic-login/version-control) のいずれかを使用して、スクリプトを Auth0 テナントにデプロイしてください。
</Callout>

<div id="custom-domain-naming">
  ## カスタムドメインの命名
</div>

デフォルトでは、テナントに関連付けられた URL には、その名前と、場合によってはリージョン固有の識別子が含まれます。たとえば、米国を拠点とするテナントの URL は `https://example.auth0.com` のようになります。一方、ヨーロッパを拠点とするテナントの URL は `https://example.eu.auth0.com` のような形式になります。[カスタムドメイン](/ja/docs/customize/custom-domains) を使用すると、組織のブランドに沿った一貫性のある名前を使って、ユーザーに統一された体験を提供できます。

<Warning>
  Auth0 テナント ごとに適用できるカスタムドメイン名は 1 つだけです。そのため、独立したドメイン名でのブランディングがどうしても必要な場合は、複数の Auth0 テナント を本番環境にデプロイする [アーキテクチャ](/ja/docs/get-started/architecture-scenarios/business-to-consumer/architecture) が必要になります。
</Warning>

さらに、<Tooltip tip="Custom Domain: 特別な名前、またはバニティ名を持つサードパーティーのドメイン。" cta="用語集を見る" href="/ja/docs/glossary?term=Custom+Domain">カスタムドメイン</Tooltip> 機能を使うと、証明書管理プロセスを完全に制御できます。デフォルトでは Auth0 が標準の SSL 証明書を提供しますが、カスタムドメインを設定すれば、Extended Validation (EV) SSL 証明書などを使用して、ブラウザー上で視覚的にわかりやすい安心材料を示すことができます。

一般に、認証に一元化されたドメインを使用しているお客様が最も成果を上げる傾向にあります。これは、企業が複数の製品やサービスブランドを提供している場合に特に当てはまります。一元化されたドメインを使用することで、エンドユーザーに一貫したユーザー体験を提供できるだけでなく、Auth0 で複数の本番テナントを維持する必要性も最小限に抑えられます。

<div id="email-template-customization">
  ## メールテンプレートのカスタマイズ
</div>

Auth0 では、ユーザーへの通知だけでなく、安全な ID 管理に必要な機能 (たとえば、メールアドレスの確認、アカウントの復旧、ブルートフォース攻撃対策) にもメールを幅広く利用しており、そのためのテンプレートを複数提供しています。

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  メールテンプレートをカスタマイズする前に、[Email Provider](/ja/docs/get-started/architecture-scenarios/business-to-consumer/operations#email-provider-setup) を設定してください。
</Callout>

標準では、使用されるメールテンプレートには一般的な文言と Auth0 のブランディングが含まれています。ただし、これらのテンプレートはほぼあらゆる部分を設定できるため、希望する文言やユーザー体験を反映したり、優先言語やアクセシビリティ オプションなどを変更したりできます。

メールテンプレートは、[Liquid syntax](/ja/docs/customize/email/email-templates/use-liquid-syntax-in-email-templates) を使用してカスタマイズします。ユーザーの設定に基づいてテンプレートをカスタマイズしたい場合は、ユーザープロフィール内の [metadata](/ja/docs/manage-users/user-accounts/metadata) や、アプリケーション固有のメタデータにもアクセスできます。

<div id="password-reset-page-customization">
  ## パスワードリセットページのカスタマイズ
</div>

[Password Reset](/ja/docs/customize/login-pages/classic-login/customize-password-reset-page)ページは、ユーザーがパスワード変更機能を利用する際に使用されます。ログインページと同様に、組織固有のブランディング要件を反映するようカスタマイズできます。

<div id="error-page-customization">
  ## エラーページのカスタマイズ
</div>

ユーザーの対話型ワークフロー (ユーザー登録やログインなど) の途中で問題が発生した場合、Auth0 は内部的に何が原因かを示すエラーメッセージを表示します。ただし、デフォルトのメッセージは、特にエンドユーザーにとっては少し分かりにくいことがあります。というのも、多くの場合、必要な文脈情報が不足しており、その情報を補えるのはあなたしかいないからです。そのため、ユーザーに不足している文脈に応じた情報を直接提供できるよう、[エラーページをカスタマイズする](/ja/docs/customize/login-pages/custom-error-pages)ことをお勧めします。さらに、エラーページをカスタマイズすれば、Auth0 ではなく自社のブランドを表示できるほか、次に何をすべきかについて役立つ情報もユーザーに提供できます。たとえば、FAQ へのリンクや、自社のサポートチームまたはヘルプデスクへの問い合わせ方法などを含めることができます。

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Auth0 が提供するエラーページをカスタマイズするためのユーザーインターフェースは標準では用意されていませんが、設定には [Management API の Tenant Settings エンドポイント](https://auth0.com/docs/api/management/v2#!/Tenants/patch_settings) を使用できます。あるいは、独自のエラーページを作成してホストできる場合は、Auth0 ホストのページを使う代わりに、Auth0 からそのページへユーザーをリダイレクトさせることもできます。
</Callout>

<div id="guardian-multi-factor-page-customization">
  ## Guardian 多要素認証ページのカスタマイズ
</div>

多要素認証ページは、[Universal Login Settings](https://manage.auth0.com/#/login_settings) セクションで Universal Login のブランディングオプションを調整することでカスタマイズできます。

さらに細かくカスタマイズする必要がある場合は、組織固有の UX 要件を反映するために、[HTML コンテンツ全体](/ja/docs/secure/multi-factor-authentication/customize-mfa/customize-mfa-classic-login)をカスタマイズすることもできます。
