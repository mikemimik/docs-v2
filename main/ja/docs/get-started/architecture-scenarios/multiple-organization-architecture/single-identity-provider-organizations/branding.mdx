組織に関連付けられたブランド資産は非常に重要です。これらを活用することで、ユーザーにとって見慣れた信頼できる環境を提供できます。認知されたブランド資産を使用することで、ユーザーが提供する情報 (たとえば、資格情報) が安全かつ確実に取り扱われるという安心感も高まります。アーキテクチャシナリオでは、[B2B Branding](/docs/ja/get-started/architecture-scenarios/business-to-business/branding) に関するさまざまなトピックについて一般的なガイダンスを提供しています。ここで示すガイダンスとあわせて確認することをお勧めします。

<div id="universal-login">
  ## Universal Login
</div>

マルチ組織環境でブランディングを行うのは、1 セットのブランド資産だけを扱えばよい標準的な環境に比べて複雑になることがあります。しかし、[Auth0 Organizations](/docs/ja/manage-users/organizations/organizations-overview) では [Universal Login エクスペリエンス](/docs/ja/authenticate/login/auth0-universal-login/universal-login-vs-classic-login/universal-experience) を活用しており、ブランディングのカスタマイズを簡単にするための複数の[ページテンプレート](/docs/ja/customize/login-pages/universal-login/customize-templates)が用意されています。また、Auth0 Organization の設定自体でも、[基本的なカラースキームのブランディングを選択](/docs/ja/manage-users/organizations/configure-organizations/create-organizations)できます。

<div id="login-page">
  ### ログインページ
</div>

[データベース接続またはカスタムデータベース接続](/docs/ja/authenticate/database-connections)を使用する場合、ログインページはユーザーの第1認証要素の資格情報を取得するために使用されます。ログインページは、組織ごとのカスタマイズに役立つ[テンプレート変数](/docs/ja/manage-users/organizations/create-first-organization)一式を使って設定できます。

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  ログインページは、第1認証要素による認証のために Enterprise 接続や Social 接続を表示するよう設定できるため、このような場合にもログインページのブランディングを適用できます。
</Callout>

<div id="password-reset-page">
  ### Password Reset ページ
</div>

[ログインページ](#login-page)と同様に、Password Reset ページは、データベース接続またはカスタムデータベース接続を使用する場合に設定できます。エンタープライズ接続またはソーシャル接続を使用する場合、パスワードの変更は上流の<Tooltip tip="IDプロバイダー（IdP）: デジタルアイデンティティを保存・管理するサービス。" cta="用語集を表示" href="/docs/ja/glossary?term=Identity+Provider">IDプロバイダー</Tooltip> (IdP) 側で Auth0 の外部で処理されるため、Auth0 の Password Reset ページが表示されることはありません。

<div id="multi-factor-authentication-pages">
  ### 多要素認証ページ
</div>

<Tooltip tip="多要素認証（MFA）: SMS で送信されるコードなど、ユーザー名とパスワードに加えて認証要素を使用するユーザー認証プロセスです。" cta="用語集を見る" href="/docs/ja/glossary?term=Multi-Factor+Authentication">多要素認証</Tooltip> (MFA) は Auth0 では、ユーザーの第 1 認証要素の資格情報を取得するためのあらゆる方法 (つまり、データベース接続またはカスタムデータベース接続、エンタープライズ接続、またはソーシャル接続) でサポートされています。MFA を使用している場合は、MFA に関連する <Tooltip tip="Universal Login: ユーザーのアイデンティティを確認するために、アプリケーションは Auth0 の認可サーバーでホストされている Universal Login にリダイレクトされます。" cta="用語集を見る" href="/docs/ja/glossary?term=Universal+Login">Universal Login</Tooltip> のエクスペリエンスページにもブランディングを適用したい場合が多いでしょう。これは、[ログインページ](#login-page) のブランディングと同様の方法で行えます。

<div id="email-templates">
  ## メールテンプレート
</div>

Universal Login のページテンプレートと同様に、組織 ごとのメールのカスタマイズに役立つ [メールテンプレート変数](/docs/ja/customize/email/email-templates/supported-liquid-syntax#general-variables) を利用できます。カスタマイズが必要なメールテンプレートは、ユーザーの認証方法 (つまり、データベース接続、カスタムデータベース接続、エンタープライズ接続、または Social Connection のいずれを使用するか) によって異なります。Auth0 Organizations 機能を使用する場合のメールテンプレートのカスタマイズについて詳しくは、[Create Your First Organization: Email Templates](/docs/ja/manage-users/organizations/create-first-organization) を参照してください。

<div id="custom-domains">
  ## カスタムドメイン
</div>

Auth0 Organizations 機能では、Auth0 テナント レベルで定義される [カスタムドメイン](/docs/ja/customize/custom-domains) を 1 つしかサポートしていません。そのため、組織 に依存しない <Tooltip tip="カスタムドメイン: 特殊な、または独自の名前を持つサードパーティのドメイン。" cta="用語集を表示" href="/docs/ja/glossary?term=Custom+Domain">カスタムドメイン</Tooltip>名を使用することをお勧めします。どうしても 組織 ごとにドメイン名を設定する必要がある場合は、複数の Auth0 テナント を本番環境にデプロイする必要があります。複数の tenant を使用すると、<Tooltip tip="シングルサインオン (SSO): ユーザーが 1 つの applicaton にログインすると、そのユーザーは他のアプリケーションにも自動的にログインされるサービス。" cta="用語集を表示" href="/docs/ja/glossary?term=Single+Sign-On">シングルサインオン</Tooltip> (SSO) やユーザープロファイル管理などに影響が及ぶため、この方法を選ぶ前に慎重に検討してください。
