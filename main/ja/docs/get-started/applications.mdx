title: Auth0 のアプリケーション
---

Auth0 における **アプリケーション** または **アプリ** という用語は、特定の実装形態を意味するものではありません。たとえば、モバイルデバイス上で動作するネイティブアプリ、ブラウザー上で動作するシングルページアプリケーション、またはサーバー上で動作する従来型Webアプリケーションを指す場合があります。

Auth0 では、これらの特性に基づいてアプリを分類します。

* **アプリケーションの種類**: アプリケーションに認証を追加するには、<Tooltip tip="Auth0 Dashboard: サービスを設定するための Auth0 の主要製品です。" cta="用語集を見る" href="/ja/docs/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip> にアプリケーションを登録し、次のいずれかの種類を選択する必要があります。

  * **従来型Webアプリケーション**: アプリケーションロジックの大半をサーバー上で実行する従来型の Web アプリです (Express.js や ASP.NET など) 。従来型Webアプリケーションの設定方法については、[従来型Webアプリケーションを登録する](/ja/docs/get-started/auth0-overview/create-applications/regular-web-apps) を参照してください。
  * **シングルページWebアプリケーション (SPA)**: ユーザーインターフェースロジックの大半を Web ブラウザー内で実行し、主に API を使用して Web サーバーと通信する JavaScript アプリです (AngularJS + Node.js や React など) 。シングルページWebアプリケーションの設定方法については、[シングルページWebアプリケーションを登録する](/ja/docs/get-started/auth0-overview/create-applications/single-page-web-apps) を参照してください。
  * **ネイティブアプリケーション**: デバイス上でネイティブに動作するモバイルまたはデスクトップのアプリケーションです (iOS や Android など) 。ネイティブアプリケーションの設定方法については、[ネイティブアプリケーションを登録する](/ja/docs/get-started/auth0-overview/create-applications/native-apps) を参照してください。
  * **マシンツーマシン (M2M) アプリケーション**: コマンドラインツール、デーモン、IoT デバイス、またはバックエンドで実行されるサービスのような、ユーザー操作を伴わないアプリケーションです。通常、このオプションは API へのアクセスを必要とするサービスがある場合に使用します。Machine-to-Machine アプリケーションの設定方法については、[Register Machine-to-Machine Applications](/ja/docs/get-started/auth0-overview/create-applications/machine-to-machine-apps) を参照してください。
* **資格情報の安全性**: [OAuth 2.0 の仕様](https://tools.ietf.org/html/rfc6749#section-2.1) によると、アプリはパブリックまたは confidential のいずれかに分類されます。confidential アプリは資格情報を安全に保持できますが、パブリックアプリは保持できません。詳しくは、[コンフィデンシャルアプリケーションとパブリックアプリケーション](/ja/docs/get-started/applications/confidential-and-public-applications) を参照してください。
* **所有**: アプリがファーストパーティかサードパーティかは、そのアプリの所有者と管理主体によって決まります。ファーストパーティアプリは、Auth0 ドメインを所有する同じ organization または個人によって管理されます。サードパーティアプリケーションは、外部の当事者やパートナーが API の背後にある保護されたリソースへ安全にアクセスできるようにします。詳しくは、[ファーストパーティアプリケーションとサードパーティアプリケーション](/ja/docs/get-started/applications/first-party-and-third-party-applications) を参照してください。

<div id="manage-applications-settings">
  ## アプリケーション設定の管理
</div>

アプリケーションは、[Auth0 Dashboard &gt; アプリケーション &gt; アプリケーション](https://manage.auth0.com/#/applications/\{yourClientId}/settings) で登録します。Auth0 Dashboard でアプリケーションを設定するだけでなく、[OpenID Connect (OIDC) Dynamic Client Registration 1.0](https://openid.net/specs/openid-connect-registration-1_0.html) 仕様で説明されているように、プログラムから設定することもできます。

アプリごとに異なる方法でユーザーがログインできるような、より複雑な構成にすることもできます。詳しくは、[Multi-Tenant Application Best Practices](/ja/docs/get-started/auth0-overview/create-tenants/multi-tenant-apps-best-practices) と [Create Multiple Tenants](/ja/docs/get-started/auth0-overview/create-tenants/create-multiple-tenants) を参照してください。

デフォルトでは、Auth0 は新しいアプリケーションを作成するときに、テナントに関連付けられているすべての接続を有効にします。これを変更するには、Auth0 Dashboard の **アプリケーション設定** で [アプリケーションの接続を更新](/ja/docs/get-started/applications/update-application-connections) します。

<div id="monitor-applications">
  ## アプリケーションを監視する
</div>

[アプリケーション](/ja/docs/deploy-monitor/monitor/monitor-applications)を監視し、独自のテストを使用してエンドツーエンドテストを実行できます。Auth0 では、Auth0 Dashboard 管理者の操作、ユーザー認証の成功と失敗、パスワード変更リクエストなどを含む[ログデータ](/ja/docs/deploy-monitor/logs)が保存されます。[Auth0 Marketplace](https://marketplace.auth0.com/features/log-streaming) のログストリーミングを使用すると、ログデータをエクスポートし、Sumo Logic、Splunk、Mixpanel などのツールでログデータを分析・保存できます。

<div id="remove-applications">
  ## アプリケーションを削除する
</div>

Auth0 Dashboard または <Tooltip tip="Management API: お客様が管理タスクを実行できるようにするプロダクト。" cta="用語集を見る" href="/ja/docs/glossary?term=Management+API">Management API</Tooltip> を使用して、[アプリケーションを削除](/ja/docs/get-started/applications/remove-applications)できます。

<div id="manage-client-secrets">
  ## クライアントシークレットを管理する
</div>

<Tooltip tip="Authorization Server: ユーザーのアクセス範囲の境界を定義する役割を担う中央集約型サーバー。たとえば、認可サーバーは、ユーザーが利用できるデータ、タスク、機能を制御できます。" cta="用語集を見る" href="/ja/docs/glossary?term=client+secret">クライアントシークレット</Tooltip>は、アプリケーションと<Tooltip tip="Client Secret: クライアント（アプリケーション）が認可サーバーに対して認証を行うために使用するシークレット。クライアントと認可サーバーだけが知っているべきであり、推測されないよう十分にランダムでなければなりません。" cta="用語集を見る" href="/ja/docs/glossary?term=authorization+server">認可サーバー</Tooltip>だけが知っているシークレットです。これにより、認可されたリクエスト送信元にのみ[トークン](/ja/docs/secure/tokens)を付与し、リソースを保護します。

クライアントシークレットは厳重に保護し、モバイルアプリや**ブラウザベース**のアプリには**決して**含めないでください。クライアントシークレットが万一漏えいした場合は、[新しいものにローテーション](/ja/docs/get-started/applications/rotate-client-secret)し、新しいクライアントシークレットで認可済みのすべてのアプリケーションを更新してください。

<div id="grant-types">
  ## グラントタイプ
</div>

Auth0 では、さまざまな認証および認可のグラントタイプやフローが提供されており、Auth0 に登録したアプリケーションの `grant_types` プロパティに基づいて、適切なグラントタイプを指定できます。詳しくは、[アプリケーションのグラントタイプ](/ja/docs/get-started/applications/application-grant-types) を参照してください。
