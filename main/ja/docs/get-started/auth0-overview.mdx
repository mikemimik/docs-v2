title: Auth0 の概要
---

Auth0 は、アプリケーションに認証および認可サービスを追加するための、柔軟ですぐに導入できるソリューションです。チームや組織は、ユーザーの認証と認可のために独自のソリューションを構築することで生じるコスト、時間、リスクを回避できます。

Auth0 のユースケースをいくつか見てみましょう。

* 優れたアプリを構築し、ユーザー認証と認可を追加したいと考えています。ユーザーは、識別子 (ユーザー名、メールアドレス、または電話番号) とパスワード、あるいはソーシャルアカウント (Facebook や X など) のいずれかでログインできる必要があります。さらに、ログイン後にユーザーのプロファイルを取得して UI をカスタマイズし、認可ポリシーを適用したいと考えています。
* API を構築し、[OAuth 2.0](/ja/docs/authenticate/protocols/oauth) で保護したいと考えています。
* 複数のアプリがあり、[シングルサインオン (SSO)](/ja/docs/authenticate/single-sign-on) を実装したいと考えています。
* JavaScript のフロントエンドアプリとモバイルアプリを構築し、その両方から API に安全にアクセスさせたいと考えています。
* [Security Assertion Markup Language (SAML)](/ja/docs/authenticate/protocols/saml/saml-configuration) を使用してユーザーを認証する必要がある web app を構築しています。
* パスワードには限界があると考えており、メールまたは SMS で配信されるワンタイムコードでユーザーがログインできるようにしたいと考えています。
* ユーザーのメールアドレスの 1 つが、どこかのサイトで発生した公開情報漏えいに含まれていた場合に通知を受け取り、さらにそのユーザーに通知したり、パスワードをリセットするまでアプリへのログインをブロックしたりしたいと考えています。
* DDoS 攻撃を防ぐため、ログイン失敗が連続して発生した場合に、不審な IP アドレスを先回りしてブロックしたいと考えています。
* 大規模な組織の一員として、既存のエンタープライズディレクトリサービスをフェデレーションし、従業員が既存のエンタープライズ資格情報を使ってさまざまな社内アプリケーションやサードパーティアプリケーションにログインできるようにしたいと考えています。
* 独自のユーザー管理ソリューションを実装したくない (または実装方法が分からない) と考えています。パスワードリセット、ユーザーの作成、プロビジョニング、ブロック、削除、さらにそれらすべてを管理するための UI まで必要です。アプリそのものに集中したいだけです。
* ユーザーが機密データにアクセスしようとするときに、[多要素認証 (MFA)](/ja/docs/secure/multi-factor-authentication) を必須にしたいと考えています。
* SOC2、GDPR、PCI DSS、HIPAA など、増え続けるコンプライアンス要件への対応に役立つアイデンティティソリューションを探しています。
* サイトやアプリケーション上のユーザー行動を把握したいと考えています。このデータを使ってファネルを作成し、ユーザー維持率を測定し、サインアップフローを改善する予定です。
* リソースに対する[関係性](https://docs.fga.dev/authorization-concepts)や、組織における[ロール](/ja/docs/manage-users/access-control/rbac)に基づいて、ユーザーにリソースへのアクセスを許可できる堅牢な認可ポリシーを求めています。
