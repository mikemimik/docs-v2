ほとんどの場合、Auth0 を介してユーザーを認証するにはインターネット接続が必要です。ただし、Auth0 を使用するアプリケーションはローカル環境でも開発およびテストできます。場合によっては、インターネット接続がなくても問題ありません。

Auth0 を使用する際に開発環境、テスト環境、本番環境をどのように構成するかについては、[複数の環境を設定する](/ja/docs/get-started/auth0-overview/create-tenants/set-up-multiple-environments)を参照してください。

<div id="use-json-web-tokens-with-client-side-applications">
  ## クライアントサイドのアプリケーションで JSON Web トークンを使用する
</div>

[JSON Web Tokens (JWTs)](/ja/docs/secure/tokens/json-web-tokens) はステートレスです (つまり、それを利用するアプリは内容だけを見て判断し、それ以前の状態は考慮しません) 。そのため、これはローカルで最も簡単にテストできるシナリオの 1 つです。

テスト用の <Tooltip tip="JSON Web Token (JWT): 2者間でクレームを安全に表現するために使用される、標準的な ID トークン形式（多くの場合 access token の形式でもあります）。" cta="用語集を見る" href="/ja/docs/glossary?term=JWTs">JWTs</Tooltip> は、次のいずれかの方法で取得できます。

1. [データベース接続](/ja/docs/get-started/applications/set-up-database-connections) のテストユーザーを作成し、プログラムでそのユーザーをログインさせます。つまり、[Resource Owner Password Flow](/ja/docs/get-started/authentication-and-authorization-flow/resource-owner-password-flow) を使用することになります。実装手順の詳細については、[Call Your API Using the Resource Owner Password Flow](/ja/docs/get-started/authentication-and-authorization-flow/resource-owner-password-flow/call-your-api-using-resource-owner-password-flow) を参照してください。
2. ブラウザボットでユーザーとしてログインし、JWT を取得します。

<div id="use-sessions-with-server-side-applications">
  ## サーバーサイドアプリケーションでセッションを使用する
</div>

サーバーサイドアプリケーションで、テスト用の模擬セッションを生成できない場合は、Auth0 を介して手動でログインする方法が必要になります。

その方法の概要については、[認可コードフロー](/ja/docs/get-started/authentication-and-authorization-flow/authorization-code-flow)を参照してください。実装手順の詳細については、チュートリアル「[認可コードフローを使用してログインを追加する](/ja/docs/get-started/authentication-and-authorization-flow/authorization-code-flow/add-login-auth-code-flow)」を参照してください。

<div id="use-local-domains-with-auth0">
  ## Auth0 でローカルドメインを使用する
</div>

アプリケーションをローカルで開発している場合は、`localhost` や Auth0 からアクセスできないその他のドメイン (イントラネット上のものなど) を[コールバック URL](/ja/docs/authenticate/login/redirect-users-after-login)として使用できます。たとえば、開発中であればコールバック URL として `http://localhost:3000/callback` を使用できます。

1. [Auth0 Dashboard &gt; アプリケーション &gt; アプリケーション](https://manage.auth0.com/#/applications/\{yourClientId}/settings) に移動し、該当するアプリケーションをクリックします。
2. URL を **Allowed Callback URLs** のリストに追加します。

Auth0 の主要な認証プロトコルは [OpenID Connect (OIDC)](/ja/docs/authenticate/protocols/openid-connect-protocol) であるため、Auth0 がアプリケーションのサーバーに直接アクセスする必要はありません。代わりに Auth0 は、必要な情報をクエリ文字列またはハッシュフラグメントに含めて、ユーザーをアプリケーションのエンドポイントにリダイレクトします。

<div id="divert-emails-for-testing">
  ## テスト用にメールを振り分ける
</div>

ローカルのアプリケーションをテストする際に、アプリケーションで作成または検証するユーザーの実際のメールアドレスへメール (アカウント作成、検証など) を送信したくない場合は、Auth0 ではカスタムメールプロバイダーの使用を推奨しています。たとえば、[Mailtrap](https://mailtrap.io/signin) のようなサービスや独自に実装したカスタム SMTP サーバーを使えば、メールを捕捉するために必要な任意のロジックを適用できます。これにより、ユーザーにメールを届けることなく、検証やトラブルシューティングのためにメールの内容を確認できます。詳しくは、[Configure Test SMTP Email Server](/ja/docs/customize/email/configure-test-smtp-email-servers) をご覧ください。

Auth0 では、テスト時によく使用される一部の「偽の」ドメインは使用できません。中断やエラーを避けるため、実在するメールアドレスを使用してください。
