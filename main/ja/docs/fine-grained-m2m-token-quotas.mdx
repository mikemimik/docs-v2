[クライアント認証情報フロー](/ja/docs/get-started/authentication-and-authorization-flow/client-credentials-flow) を使用すると、CLI、デーモン、バックエンドサービスなどの Machine-to-Machine (M2M) アプリケーションは、ユーザーの操作なしに、自身のために <Tooltip tip="アクセストークン: API へのアクセスに使用される、不透明な文字列または JWT 形式の認可資格情報。" cta="用語集を見る" href="/ja/docs/glossary?term=access+tokens">アクセストークン</Tooltip> を取得できます。

アクセストークンのキャッシュに関するベストプラクティスを実装することで、Auth0 との往復通信を最小限に抑え、トークン発行量を制御できます。ただし、サードパーティのシステムや、大規模で複雑なデプロイ内のアプリケーションなど、一部のアプリケーションでは適切なキャッシュが実装されておらず、突発的にリクエストが集中する挙動を示すことがあります。その結果、Auth0 が短時間に大量の M2M トークンを生成し、テナント全体の M2M トークンクォータに影響を与え、予期しないサービス動作を引き起こす可能性があります。

Auth0 のきめ細かな M2M トークンクォータを使用すると、アプリケーションおよび Auth0 Organizations に対して、1 時間ごとおよび 1 日ごとの M2M アクセストークン上限を設定できます。これにより、トークン発行をきめ細かく運用管理し、過剰なトークン消費を防止できます。

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  きめ細かな M2M トークンクォータは、テナント全体に適用される月次の M2M トークン割り当てとは別のものです。きめ細かなトークンクォータは、トークンリクエストの急増に対する保護策として使用できます。
</Callout>

<div id="use-cases">
  ## ユースケース
</div>

* **暴走するアプリケーションを抑制**: アプリケーション単位のクォータを適用することで、M2Mアプリケーションがトークンを過剰な頻度でリクエストすること (バグやキャッシュ未使用などが原因) を制限しつつ、テナント全体の M2M トークンクォータには影響を与えないようにします。
* **マルチテナントでの公平な利用を確保**: 顧客ごと (Auth0 Organizations) に組織単位のクォータを割り当てることで、1つの顧客によって他の顧客のクォータが圧迫されるのを防ぎます。
* **テナント全体の保護策を設定**: アプリケーションや組織からの過剰な M2M トークンリクエストに対する基準値として、テナント単位のデフォルトクォータを使用します。
* **強制適用前に監視**: `enforce: false` でクォータを導入し、[消費量警告ログ](#consumption-warning-logs) と [Auth0 クォータヘッダー](#auth0-quota-headers) でトークン消費を観察します。その後、実際のブロックを有効にする前に、根拠に基づいた制限値を設定します。

<div id="how-it-works">
  ## 仕組み
</div>

次のエンティティに、きめ細かなM2Mトークンクォータを適用できます。

* **Application**: クライアントに対して発行されるトークンを制限し、個々のアプリケーションの動作を制御します。
* **Organization**: organization を指定して行われるリクエストのトークンを制限し、`org_id` に関連付けられたトークンを利用する複数のアプリケーションにまたがって、SaaS顧客または事業部門のM2Mアクティビティを管理しやすくします。

Auth0 では、2種類のきめ細かなM2Mトークンクォータをサポートしています。

* **Hourly:** 1時間内に取得できるトークンの最大数です。クォータはUTCの各時間の開始時にリセットされます。
* **Daily:** 24時間内に取得できるトークンの最大数です。クォータはUTCの各日の開始時にリセットされます。

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  Auth0 のトークンカウントの仕組みは結果整合性に基づいているため、その後のリクエストで制限が適用されるまでの間、一時的にわずかな超過が発生する可能性があります。
</Callout>

<div id="quota-evaluation-and-enforcement-rules">
  ### クォータの評価と適用ルール
</div>

Auth0 は、テナントに関連付けられたアプリケーションと組織に対するきめ細かな M2M トークンクォータを判定し、適用します。

1. **クライアントレベルのクォータ:**

   * `client_id` に **Application-Specific Quota** を設定している場合、Auth0 はそのクライアントにクォータを適用します。
   * それ以外の場合、Auth0 はそのクライアントに **Tenant-Level Default Quota for Clients** を適用します。クライアント向けのテナントレベルのデフォルトが設定されていない場合、Auth0 はそのクライアントにきめ細かなクォータを適用しません。
2. **組織レベルのクォータ:**

   * トークンリクエストに関連付けられた `org_id` に **Organization-Specific Quota** を設定している場合、Auth0 はその組織にクォータを適用します。
   * それ以外の場合、Auth0 はトークンリクエストに関連付けられた `org_id` に **Tenant-Level Default Quota for Organizations** を適用します。Organizations 向けのテナントレベルのデフォルトが設定されていない場合、Auth0 はトークンリクエストに関連付けられた `org_id` にきめ細かなクォータを適用しません。
3. トークンリクエストに関連付けられた組織の有無に応じて、Auth0 は適用可能なクォータを確認します。

   * トークンリクエストに、明示的に定義された組織、またはデフォルトの組織設定による関連付けがある場合、Auth0 はクライアントレベルと組織レベルのクォータを同時に確認します。
   * トークンリクエストに関連付けられた組織がない場合、Auth0 はクライアントレベルのクォータのみを確認します。

Auth0 は transaction に適用可能なすべてのクォータを同時に評価し、クォータ設定で `enforce` フラグが `true` に設定されているものを適用します。

* **クォータが適用される場合 (**`enforce: true`**):** 適用対象のクォータを超過すると、トークンリクエストは HTTP 429 Too Many Requests エラーで**拒否**されます。詳しくは、[クォータ超過時のエラーレスポンス](#error-responses-for-exceeded-quotas)を参照してください。
* **クォータが適用されない場合 (**`enforce: false`**):** 制限を超過していても、このクォータによってトークンリクエストが**拒否**されることはありません。Auth0 は引き続きトークン数をカウントし、しきい値に達すると [消費量警告ログ](#consumption-warning-logs) を生成するため、適用を有効にする前にモニタリングできます。

<div id="observability">
  ### 可観測性
</div>

クォータ消費の状況を把握できるようにするため、Auth0 は [消費量警告ログ](#consumption-warning-logs) を生成します。これらのログは、クォータが上限の 60%、80%、100% に達すると出力されます。これらのログを使うことで、トークン使用量を事前に監視し、潜在的な問題を特定できます。

また、アプリケーションは [Auth0 クォータヘッダー](#auth0-quota-headers) を使用して、残りのクォータをプログラムから確認することもできます。これらの HTTP ヘッダーは、クライアント認証情報フローの request に対する成功レスポンスとエラーレスポンスの両方に含まれており、クォータ使用量に関するリアルタイムの情報を提供します。

<div id="configure-m2m-token-quotas">
  ## M2M トークン のクォータを設定する
</div>

[Management API](https://auth0.com/docs/api/management/v2) を使用して、きめ細かな M2M トークン クォータを設定できます。

<div id="tenant-level-default-quotas">
  ### テナントレベルのデフォルトクォータ
</div>

テナントレベルのクォータは、Auth0 テナント内のすべてのアプリケーションと組織に対するデフォルト設定として機能します。テナントレベルのクォータを使用すると、M2M トークン使用量の基準となる上限を設定できます。これらのクォータは、アプリケーション固有のクォータおよび組織固有のクォータによって上書きされます。

テナントに関連付けられたアプリケーションや組織に対して、デフォルトのトークンクォータを指定できます。テナントレベルのクォータを設定するには、[Update a tenant](https://auth0.com/docs/api/management/v2/tenants/patch-settings) エンドポイントを使用します。

```bash lines
curl --request PATCH 'https://YOUR_DOMAIN/api/v2/tenants/settings' \
  --header 'Authorization: Bearer YOUR_MANAGEMENT_API_TOKEN' \
  --header 'Content-Type: application/json' \
  --data '{
    "default_token_quota": {
      "clients": {
        "client_credentials": {
          "per_hour": 100,
          "per_day": 1000,
          "enforce": true
        }
      },
      "organizations": {
        "client_credentials": {
          "per_hour": 500,
          "per_day": 5000,
          "enforce": false
        }
      }
    }
  }'
```

`default_token_quota` オブジェクトには、次のアプリケーションレベルおよび組織レベルのトークンクォータ設定が含まれます。

* `clients.client_credentials`:  (任意) 個々のアプリケーションに対する M2M トークンクォータを定義します。
* `organizations.client_credentials`:  (任意) 組織に対する M2M トークンクォータを定義します。
* `per_hour`:  (任意) 1 時間あたりのトークンクォータを設定します。
* `per_day`:  (任意) 1 日あたりのトークンクォータを設定します。
* `enforce`:  (任意) クォータを超えたときにトークンリクエストを拒否するかどうかを決定します。既定値は true です。

この例では、テナント内のすべてのアプリケーションは、既定で 1 時間あたり 100 個、1 日あたり 1000 個の M2M トークンに制限されます。Organizations では、既定で 1 時間あたり 500 個、1 日あたり 5000 個のトークンですが、制限の適用は現在無効になっています (`enforce: false`) 。これにより、制限を適用する前に、組織のトークン使用状況を監視できます。

<div id="application-specific-quotas">
  ### アプリケーション固有のクォータ
</div>

特定のアプリケーションに対してクォータを設定し、テナントレベルの既定値を上書きできます。これは、各アプリケーションの動作をより細かく制御するのに役立ちます。

アプリケーション固有のクォータは、アプリケーションの作成時または更新時に設定できます。

トークンクォータを設定したアプリケーションを作成するには、[クライアントを作成する](https://auth0.com/docs/api/management/v2/clients/post-clients)エンドポイントを使用します。

```bash lines
curl --location 'https://YOUR_DOMAIN/api/v2/clients' \
--header 'Authorization: Bearer YOUR_MANAGEMENT_API_TOKEN' \
--header 'Content-Type: application/json' \
--data '{
    "name": "APP_NAME",
    "app_type": "non_interactive",
    "token_quota": {
      "client_credentials": {
        "per_hour": 10,
        "per_day": 50,
        "enforce": true
      }
    }
}'
```

アプリケーションのトークンクォータを更新するには、[クライアントを更新](https://auth0.com/docs/api/management/v2/clients/patch-clients-by-id)エンドポイントを使用します。

```bash lines
curl --location --request PATCH 'https://YOUR_DOMAIN/api/v2/clients/a01FUMJHEtb0q8jcXm7y2k9EAGe5fcxZ' \
--header 'Content-Type: application/json' \
--header 'Authorization: Bearer YOUR_MANAGEMENT_API_TOKEN' \
--data '{
    "token_quota": {
        "client_credentials": {
            "per_hour": 5,
            "enforce": true
        }
    }
}'
```

`token_quota` オブジェクトには、アプリケーションのトークンクォータ設定が含まれます。

* `client_credentials`: M2M トークンのクォータを定義します。
* `per_hour`: 1 時間あたりのトークンクォータを設定します。
* `per_day`: 1 日あたりのトークンクォータを設定します。
* `enforce`: クォータを超過した場合にトークンリクエストを拒否するかどうかを指定します。既定値は `true` です。

この例では、`{yourClientId}` を持つアプリケーションは、M2M トークンが 1 時間あたり 10 個、1 日あたり 50 個に制限され、テナントレベルの設定より優先されます。

アプリケーションのトークンクォータ設定は、[Get a client by ID](https://auth0.com/docs/api/management/v2/clients/get-clients-by-id) エンドポイントを使用して取得できます。

<div id="organization-specific-quotas">
  ### 組織固有のクォータ
</div>

特定の組織に対してクォータを設定し、テナントレベルのデフォルト設定を上書きすることもできます。これにより、個々の SaaS 顧客やパートナーの M2M トークン使用量を管理できます。

組織固有のクォータは、organization の作成時または更新時に設定できます。

トークンクォータを設定した organization を作成するには、[Create an organization](https://auth0.com/docs/api/management/v2/organizations/post-organizations) エンドポイントを使用します。

```bash lines
curl --location 'https://YOUR_DOMAIN/api/v2/organizations' \
--header 'Authorization: Bearer YOUR_MANAGEMENT_API_TOKEN' \
--header 'Content-Type: application/json' \
--data '{
    "name": "acme",
    "display_name": "Acme", 
    "token_quota": {
      "client_credentials": {
        "per_hour": 50,
        "per_day": 250,
        "enforce": true
      }
    }
}'
```

organization のトークンクォータを更新するには、[organization を更新する](https://auth0.com/docs/api/management/v2/organizations/patch-organizations-by-id)エンドポイントを使用します：

```bash lines
curl --request PATCH 'https://YOUR_DOMAIN/api/v2/organizations/YOUR_ORG_ID' \
--header 'Authorization: Bearer YOUR_MANAGEMENT_API_TOKEN' \
--header 'Content-Type: application/json' \
--data '{
    "token_quota": {
        "client_credentials": {
            "per_hour": 50,
            "per_day": 250,
            "enforce": false
        }
    }
}'
```

`token_quota` object には、organization のトークンクォータ設定が含まれます。

* `client_credentials`: M2M トークンのクォータを定義します。
* `per_hour`: 1 時間あたりのトークンクォータを設定します。
* `per_day`: 1 日あたりのトークンクォータを設定します。
* `enforce`: クォータを超過した際にトークンリクエストを拒否するかどうかを決定します。デフォルトは `true` です。

この例では、`YOUR_ORG_ID` を持つ organization は、M2M トークンが 1 時間あたり 50、1 日あたり 250 に制限されています。

organization のトークンクォータ設定は、[Get an organization](https://auth0.com/docs/api/management/v2/organizations/get-organizations-by-id) エンドポイントを使用して取得できます。

<div id="auth0-quota-headers">
  ### Auth0 のクォータヘッダー
</div>

Auth0 は、クォータ消費量に関するリアルタイムの情報を提供するため、クライアント認証情報フローのレスポンス (成功時のレスポンスと 429 エラーレスポンスの両方) に HTTP ヘッダーを含めます。

* `Auth0-Client-Quota-Limit`: アプリケーションのクォータ情報を提供します。
* `Auth0-Organization-Quota-Limit`: organization のクォータ情報を提供します。

Auth0 は、レスポンスに、トークンリクエストのクォータに対応するヘッダーのみを含めます。

次のコードサンプルは、Auth0 のクォータヘッダーの例です。

```http lines
Auth0-Client-Quota-Limit: b=per_hour;q=10;r=7;t=3540,b=per_day;q=50;r=47;t=43200
Auth0-Organization-Quota-Limit: b=per_hour;q=50;r=47;t=3540,b=per_day;q=250;r=247;t=43200
```

ヘッダー値は、カンマ区切りのクォータバケットの一覧です。各バケットは、セミコロン区切りのキーと値のペアの一覧で表されます。

* `b` (bucket&#95;name): クォータバケットの名前 (`per_hour` または `per_day`) 。
* `q` (quota): そのバケットに設定されたクォータ上限。
* r (remaining): そのバケットに残っているトークン数。
* `t` (time): そのバケットがリセットされるまでの秒数。

`Auth0-Client-Quota-Limit` の例では、次のようになります。

* アプリケーションには、1 時間あたり 10 トークンのクォータ (`b=per_hour`、`q=10`) が設定されています。残りは 7 トークン (`r=7`) で、クォータは 3540 秒後にリセットされます (`t=3540`) 。
* アプリケーションには、1 日あたり 50 トークンのクォータ (`b=per_day`、`q=50`) も設定されています。残りは 47 トークン (`r=47`) で、クォータは 43200 秒後にリセットされます (`t=43200`) 。

Auth0 SDK には、クォータヘッダーの読み取りと解析に使える組み込みユーティリティが用意されており、アプリでリアルタイムのクォータ情報を活用して、デバッグやリクエスト頻度を抑える戦略の実装などを行えます。以下は、Node.js と [node-auth0](https://github.com/auth0/node-auth0) SDK を使用した例です。

export const codeExample1 = `const {
    AuthenticationClient,
    HttpResponseHeadersUtils,
  } = require("auth0");

const auth0 = new AuthenticationClient({
    domain: '{{yourDomain}}',
    clientId: '{{yourClientId}}',
    clientSecret: '{{yourSecret}}'
});

async function getAccessToken() {
    try {
        // トークンを取得する
        const response = await auth0.oauth.clientCredentialsGrant({
            audience: '{{yourAudience}}'
        });

        // 監視に利用できるよう、M2M トークンのクォータ情報をアプリケーションで参照できるようにします
        const clientQuota = HttpResponseHeadersUtils.getClientQuotaLimit(response.headers);
        console.log("clientQuota ", clientQuota);
        console.log("clientQuota 1日あたり - クォータ:", clientQuota?.perDay?.quota);
        console.log("clientQuota 1日あたり - 残り:", clientQuota?.perDay?.remaining);
        console.log("clientQuota 1日あたり - リセットまで:", clientQuota?.perDay?.resetAfter);
        console.log("clientQuota 1時間あたり - クォータ:", clientQuota?.perHour?.quota);
        console.log("clientQuota 1時間あたり - 残り:", clientQuota?.perHour?.remaining);
        console.log("clientQuota 1時間あたり - リセットまで:", clientQuota?.perHour?.resetAfter);

        return response.data.access_token;

    } catch (error) {

        // ここでエラーを処理します

    }
}

getAccessToken();`;

<div id="error-responses-for-exceeded-quotas">
  ### クォータ超過時のエラーレスポンス
</div>

適用されているクォータを超過すると、Auth0 Authentication API は `HTTP 429 Too Many Requests` エラーを返します。レスポンス本文では、Auth0 はエラーコードと、より詳細な説明を返します。Auth0 は、`feccft` タイプのイベントログ (クライアントクレデンシャルグラントでのアクセストークン交換の失敗) も出力します。

次のコードサンプルは、クォータ超過時のエラーレスポンスの例です。

```json lines
{
  "error": "too_many_requests",
  "error_description": "クライアントのクォータを超過しました"
}
```

レスポンス本文に加えて、Auth0 は次のヘッダーを返します。

* `Auth0-Client-Quota-Limit` または `Auth0-Organization-Quota-Limit`: アプリケーションまたは organization で消費されたクォータに対応する Auth0 クォータヘッダーです。ヘッダー形式の詳細については、[Auth0 クォータヘッダー](#auth0-quota-headers) を参照してください。
* `X-RateLimit-Limit`: 消費されたクォータに設定されている上限です。
* `X-RateLimit-Remaining`: クォータが完全に消費されたことを示すため、0 に設定されます。
* `X-RateLimit-Reset`: クォータがリセットされ、以降のリクエストが再び許可される予定時刻を表す UNIX タイムスタンプ (秒単位) です。
* `Retry-After`: クォータがリセットされ、その後のリクエストが成功するまでの残り秒数です。

レート制限ヘッダーの詳細については、[Predict when requests to a tenant will be rate-limited](/ja/docs/troubleshoot/customer-support/operational-policies/rate-limit-policy/rate-limit-use-cases#predict-when-requests-to-a-tenant-will-be-rate-limited-) を参照してください。

`Retry-After` ヘッダーを使用して、新しい M2M token を取得するまでの待機時間を判断し、再試行するかエラーをスローするかをアプリケーションで決定できるようにします。これは、テナントのレート制限を消費しすぎないようにするうえでも重要です。

次のコードサンプルは、新しい M2M token の取得を試み、429 エラーを受け取った場合に、待機時間が 60 秒未満であれば再試行する Node.js の例です。それ以外の場合はエラーをスローします。

export const codeExample2 = `const {
    AuthenticationClient,
    HttpResponseHeadersUtils,
  } = require("auth0");

const auth0 = new AuthenticationClient({
    domain: '{{yourDomain}}',
    clientId: '{{yourClientId}}',
    clientSecret: '{{yourSecret}}'
});

const MAX_RETRY_AFTER = 60;

async function sleep(seconds) {
    return new Promise(resolve => setTimeout(resolve, seconds * 1000));
}

async function getAccessToken(retry = true) {
    try {
        // アクセストークンを取得
        const response = await auth0.oauth.clientCredentialsGrant({
            audience: '{{yourAudience}}'
        });

        return response.data.access_token;

    } catch (error) {

        if (error.statusCode === 429 && retry) {
            // 429 エラーが発生した場合は、新しいトークンを取得するまでに必要な待機時間を計算する
            const retryAfter = parseInt(error.headers.get('retry-after'), 10);
            if (retryAfter < MAX_RETRY_AFTER) {
                // 再試行までの待機時間が最大秒数未満の場合
                console.warn(\`レート制限に達しました。${retryAfter} 秒後に再試行します...\`);
                await sleep(retryAfter);
                return getAccessToken(false); // 再試行は 1 回のみ
            }
        }

        console.error('アクセストークンの取得時にエラーが発生しました:', error.response?.data || error.message);
    }
}

getAccessToken();`;

消費量のwarning Logの例には、次のフィールドが含まれます。

* **type**: `token_quota_consumption_warning`。しきい値に近づいている M2M トークンのクォータ消費に対する warning を示します。
* **description**: (String) `<percent>% <entity_type> <bucket>` quota consumed 形式の、人が読んで理解しやすい概要です。たとえば、&quot;60% of client per day quota consumed.&quot; です。

<div id="best-practices">
  ## ベストプラクティス
</div>

* **トークンのキャッシュを優先する**: M2Mアプリケーションでは、access token を適切にキャッシュすることを重視してください。クォータは保護手段であり、効率的なトークン管理の代わりにはなりません。
* **適切なクォータレベル**: 実際のトラフィックパターンと妥当なバースト容量に基づいてクォータを設定してください。
* **適用前に監視する**: まずは `enforce: false` で開始し、ログとヘッダーで使用状況を確認してから、`enforce: true` を有効にする前に適切な制限値を設定してください。
* **クライアント側での処理**: アプリケーションは、バックオフ戦略として `Retry-After` を使用し、429 エラーを適切に処理する必要があります。
* **クォータヘッダーを事前に確認する**: トークンの使用状況を監視して、潜在的な問題を特定してください。
* **ログを定期的に確認する**: `token_quota_consumption_warning` ログを監視し、クォータを事前に調整してください。

<div id="limitations">
  ## 制限事項
</div>

* **API 呼び出し数の上限ではありません**: Fine-Grained M2M Token Quotas は、クライアント認証情報フローで取得するアクセストークンの数に適用されるものであり、それらのトークンを使った API 呼び出しには適用されません。
* **結果整合性:** トークンのカウントは結果整合性で処理されるため、後続のリクエストで制限が適用される前に、わずかに短時間超過する場合があります。
* `429` 応答となったリクエストは、テナントの Auth0 Authentication API のレート制限にカウントされます。詳しくは、[Rate Limit Policy](/ja/docs/troubleshoot/customer-support/operational-policies/rate-limit-policy) を参照してください。

<div id="troubleshoot">
  ## トラブルシューティング
</div>

* **正当なトラフィックがブロックされる場合:**

  * `enforce: false` の設定を検討してください。
  * `token_quota_consumption_warning` と、失敗した交換イベント (`feccft`) について Auth0 テナントログを確認してください。
  * レスポンス内の `Auth0-...-Quota-Limit` ヘッダーを確認してください。
  * アプリケーションのトークンリクエストのパターンとキャッシュを見直してください。
  * クォータ上限の調整が必要かどうかを検討してください。
* **429 エラーの見方**: `X-RateLimit-*`、`Retry-After`、および該当する `Auth0-...-Quota-Limit` ヘッダーを使用して、どの上限が消費されたかとリセット時刻を特定します。
