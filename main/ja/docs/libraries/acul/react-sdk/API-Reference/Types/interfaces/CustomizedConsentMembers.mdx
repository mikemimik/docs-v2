````ts Example
export interface CustomizedConsentMembers extends BaseMembers {
  /**
   * Customized Consent画面の特定のプロパティとデータへのアクセスを提供します。
   * リクエストされている`scopes`と`authorizationDetails`のリストを含みます。
   * @type {ScreenMembersOnCustomizedConsent}
   */
  screen: ScreenMembersOnCustomizedConsent;

  /**
   * リクエストされた権限とauthorization detailsを承認（グラント）するユーザーの決定を送信します。
   * このactionは`action: "accept"`を指定して`/u/customized-consent` エンドポイントにPOSTします。
   * transaction stateは自動的に含まれます。
   *
   * @param {CustomOptions} [payload] - 省略可能。リクエストと共に送信するカスタムキーと値のペアを格納するオブジェクト。
   *                                    これらのパラメーターはサーバーに送信されるフォームデータに含まれます。
   * @returns {Promise<void>} acceptのactionが正常に送信されたときに解決されるPromise。
   *                          成功した場合、Auth0は通常ユーザーをアプリケーションに戻すか、
   *                          認証フローの次のステップにリダイレクトします。
   * @throws {Error} フォームの送信に失敗した場合（例：ネットワークエラー、無効なstate）にエラーをスローします。
   *                 サーバーサイドのエラー（"invalid_request"など）は、JavaScriptエラーとしてスローされるのではなく、
   *                 操作後に`this.transaction.errors`に反映されます。
   *
   * @example
   * ```typescript
   * // 'customizedConsentManager'はCustomizedConsent SDKクラスのインスタンスと仮定します
   * try {
   *   await customizedConsentManager.accept();
   *   // 成功した場合、ページは通常リダイレクトされます。
   * } catch (error) {
   *   // 送信処理中の予期しないエラーを処理します。
   *   console.error("Failed to submit consent acceptance:", error);
   * }
   * // awaitの後、サーバーサイドのバリデーションの問題についてcustomizedConsentManager.transaction.errorsを確認してください。
   * ```
   */
  accept(payload?: CustomOptions): Promise<void>;

  /**
   * リクエストされた権限とauthorization detailsを拒否（却下）するユーザーの決定を送信します。
   * このactionは`action: "deny"`を指定して`/u/customized-consent` エンドポイントにPOSTします。
   * transaction stateは自動的に含まれます。
   *
   * @param {CustomOptions} [payload] - 省略可能。リクエストと共に送信するカスタムキーと値のペアを格納するオブジェクト。
   *                                    これらのパラメーターはサーバーに送信されるフォームデータに含まれます。

   *                          成功した場合、Auth0は通常ユーザーをリダイレクトし、アクセス拒否メッセージを表示するか、
   *                          アプリケーションにエラーを返す場合があります。
   * @throws {Error} フォームの送信に失敗した場合（例：ネットワークエラー、無効なstate）にエラーをスローします。
   *                 サーバーサイドのエラーは`this.transaction.errors`に反映されます。
   *
   * @example
   * ```typescript
   * // 'customizedConsentManager'はCustomizedConsent SDKクラスのインスタンスと仮定します

   *   await customizedConsentManager.deny({ reason_code: "user_rejected_details" }); // カスタムオプションの例
   *   // 成功した場合、ページは通常リダイレクトされます。
   * } catch (error) {
   *   console.error("Failed to submit consent denial:", error);
   * }
   * // awaitの後、サーバーサイドのバリデーションの問題についてcustomizedConsentManager.transaction.errorsを確認してください。
   * ```
   */
  deny(payload?: CustomOptions): Promise<void>;
}

<ParamField body="accept" type="Promise<void>">
  要求された権限と authorization details を受け入れる (グラントする) というユーザーの決定を送信します。
  このアクションは、`action: "accept"` を指定して `/u/customized-consent` エンドポイントに POST します。
  transaction state は自動的に含まれます。

  accept アクションが正常に送信されると解決される Promise です。
  成功すると、Auth0 は通常、ユーザーをアプリケーションへリダイレクトするか、
  認証フローの次のステップへ進めます。

  #### Throws

  フォームの送信に失敗した場合 (例: ネットワークの問題、無効な state) にエラーをスローします。
  サーバー側のエラー (&quot;invalid&#95;request&quot; など) は、JavaScript エラーとしてスローされるのではなく、
  操作後に `this.transaction.errors` に反映されます。

  ```typescript Example
  // 'customizedConsentManager' が CustomizedConsent SDK クラスのインスタンスであるとします
  try {
    await customizedConsentManager.accept();
    // 成功した場合、通常はページがリダイレクトされます。
  } catch (error) {
    // 送信処理自体で発生した予期しないエラーを処理します。
    console.error("Failed to submit consent acceptance:", error);
  }
  // await の後で、サーバー側のバリデーションの問題について customizedConsentManager.transaction.errors を確認します。
  ```
