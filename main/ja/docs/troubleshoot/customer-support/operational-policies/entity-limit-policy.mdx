description: 契約者向けに、Auth0 のテナントにおけるエンティティ上限ポリシーについて説明します。
title: エンティティ上限ポリシー
---

Auth0 におけるエンティティとは、アプリケーション、接続、Organizations、API <Tooltip tip="リソースサーバー: 保護されたリソースをホストするサーバー。リソースサーバーは保護されたリソースへのリクエストを受け取り、応答します。" cta="用語集を見る" href="/ja/docs/glossary?term=resource+servers">リソースサーバー</Tooltip>、Permissions、Roles、Actions などのテナント設定要素を指します。

<Warning>
  エンティティ上限は、当社プラットフォームの稼働状況を維持するために適用されるエンティティのハードリミットです。料金体系の一部ではありませんが、料金プランのグループごとに異なる場合があります。個別の料金プランには別途制限があり、これは特定のプランの契約時に付与される MAU やエンタープライズ接続などの利用枠を指します。 (料金プランの制限については、[Auth0 Pricing](https://auth0.com/pricing/) を参照してください。)
</Warning>

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  「無制限」という表現はハードリミットがないことを意味しますが、Okta はしきい値にシステム上の制限を設けています。これらの制限についてご不明な点があれば、サポートまでお問い合わせください。
</Callout>

<div id="global-subscription-plan-limits">
  ## すべてのサブスクリプションプランに共通する制限
</div>

このセクションでは、すべてのサブスクリプションプランに適用される制限を一覧で示します。

<div id="organizations">
  ### Organizations
</div>

| Entity                                         | Limit   |
| ---------------------------------------------- | ------- |
| テナントあたりの組織数                         | 100,000 |
| 組織あたりのメンバー数                          | 100,000 |
| 組織あたりの接続数                            | 10      |
| 組織メンバーあたりのロール割り当て数                   | 50      |
| Enterprise Groupあたりのロール割り当て数                   | 10      |
| Enterprise Groupsに割り当て可能な一意のロール数               | 100     |
| ロールを割り当て可能なEnterprise Groups数                  | 10,000  |
| 組織あたりのM2Mクライアントグラント数 (Public Cloud)  | 100     |
| 組織あたりのM2Mクライアントグラント数 (Private Cloud) | 1000    |
| カスタムトークン交換プロファイル                               | 100     |

Enterpriseプランをご利用のお客様は、[サポートに連絡する](https://support.auth0.com/)ことで、テナントあたりの組織数と組織あたりのメンバー数の上限引き上げをリクエストできます。

これらの上限は、Public Cloudではテナントあたり2,000,000 組織、組織あたり2,000,000メンバーまで引き上げ可能です。Private Cloudインスタンスでは、これらのエンティティに上限はありません。

<div id="authorization-core-role-based-access-control-rbac">
  ### Authorization Core ロールベースのアクセス制御 (RBAC)
</div>

| Entity                     | Limit  |
| -------------------------- | ------ |
| テナントあたりのロール                | 1000   |
| API (リソースサーバー) あたりのスコープ    | 1000   |
| ユーザーあたりのロール                | 50     |
| Enterprise Groupあたりのロール        | 10     |
| Enterprise Groupに割り当て可能な一意のロール | 100    |
| ロールを割り当て可能なEnterprise Group    | 10,000 |
| ユーザーあたりの権限                 | 1000   |
| ロールあたりの権限                  | 1000   |

上記のEnterprise Groupおよびロールの上限は、Organizations 機能を使用せずに RBAC を利用する場合に適用されます。Organizations 機能を使用すると、これらの上限はテナント単位ではなく組織単位で適用されます。

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  ユーザーあたりの権限数の上限は、ユーザーに直接割り当てられた権限にのみ適用される点に注意してください。

  技術的には、権限を複数の異なるロールに割り当ててから、そのロールをユーザーに割り当てれば、ここに記載された数を超える権限をユーザーが持つことも可能です。
</Callout>

| Entity                       | Limit |
| ---------------------------- | ----- |
| ユーザーごと・アプリケーションごとのリフレッシュトークン | 200   |

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  <Tooltip tip="リフレッシュトークン: ユーザーに再度ログインを求めることなく、新しいアクセストークンを取得するために使用されるトークンです。" cta="用語集を見る" href="/ja/docs/glossary?term=Refresh+tokens">リフレッシュトークン</Tooltip> の上限に達した状態で新しいリフレッシュトークンを作成すると、
  システムはその
  アプリケーションにおける、そのユーザーの最も古いトークンを取り消すか削除します。

  取り消されたトークンおよび期限切れのトークンは、上限数にはカウントされません。
</Callout>

<div id="actions">
  ### Actions
</div>

| Entity                              | Limit |
| ----------------------------------- | ----- |
| テナントあたりのActions数                    | 100   |
| テナントあたりのトリガーごとのActions数             | 20    |
| テナントあたりのActionsモジュール数               | 100   |
| ActionあたりのActionsモジュール数             | 3     |
| Actionあたりのバージョン数                    | 50    |
| Actionモジュールあたりのバージョン数               | 50    |
| Actionあたりのシークレット数                   | 30    |
| Actionモジュールあたりのシークレット数              | 30    |
| Actionあたりの依存関係 (NPM Modules) 数      | 10    |
| Actionモジュールあたりの依存関係 (NPM Modules) 数 | 10    |

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  ActionsとActionsモジュールの上限には、デプロイ済みと未デプロイの両方のエンティティが含まれます。削除すると、その上限のカウント対象から外れます。

  Actionのバージョン数とActionモジュールのバージョン数の上限には、ドラフトとアクティブの両方が含まれます。上限に達すると、新しいバージョンを追加するたびに、Auth0が最も古い未使用のバージョンを削除します。
</Callout>
