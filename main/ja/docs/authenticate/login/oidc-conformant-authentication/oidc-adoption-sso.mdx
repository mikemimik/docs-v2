description: OIDC 準拠パイプラインが Single Sign-On（SSO）の利用にどのような影響を与えるかを学びます。
title: OIDC での Single Sign-On
---

OIDC 準拠の認証パイプラインでは、シングルサインオン (<Tooltip tip="Single Sign-On (SSO): ユーザーが 1 つのアプリケーションにログインすると、そのユーザーを他のアプリケーションにも自動的にログインさせるサービスです。" cta="用語集を見る" href="/docs/ja/glossary?term=SSO">SSO</Tooltip>) はアプリケーション側ではなく <Tooltip tip="Single Sign-On (SSO): ユーザーが 1 つのアプリケーションにログインすると、そのユーザーを他のアプリケーションにも自動的にログインさせるサービスです。" cta="用語集を見る" href="/docs/ja/glossary?term=authorization+server">認可サーバー</Tooltip> (つまり Auth0) で行う必要があります。つまり、<Tooltip tip="Authorization Server: ユーザーのアクセス範囲を定義するうえで役立つ一元化されたサーバーです。たとえば、認可サーバーはユーザーが利用できるデータ、タスク、機能を制御できます。" cta="用語集を見る" href="/docs/ja/glossary?term=Universal+Login">Universal Login</Tooltip> を使用し、ユーザーをログインページにリダイレクトする必要があります。詳しくは、[Universal Login](/docs/ja/authenticate/login/auth0-universal-login) と [Single Sign-On](/docs/ja/authenticate/single-sign-on) を参照してください。

一般的に、SSO を行う場合は次のようになります。

1. ユーザーがローカルでログインしていない場合は、アプリケーションの種類に応じて、[認可コードフロー](/docs/ja/get-started/authentication-and-authorization-flow/authorization-code-flow) や [Implicit Flow](/docs/ja/get-started/authentication-and-authorization-flow/implicit-flow-with-form-post) などのリダイレクトベースのフローを使用し、認証のために Auth0 のログインページ (`/authorize`) へリダイレクトする必要があります。
2. ユーザーがすでに SSO でログインしている場合、Auth0 は認証情報を再入力させることなく、すぐにそのユーザーを認証します。

ユーザーが SSO でログインしているかどうかを確認するには、サイレント認証を使用します。これにより、ユーザーがすでにログインしていれば再認証され、認証が必要な場合はエラーが返されます。legacy authentication パイプラインでは、これは `/ssodata` endpoint を使用して実現できましたが、OIDC 準拠パイプラインでは deprecated です。サイレント認証の設定方法について詳しくは、[Configure Silent Authentication](/docs/ja/authenticate/login/configure-silent-authentication) を参照してください。
