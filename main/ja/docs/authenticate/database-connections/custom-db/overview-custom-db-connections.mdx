<Card title="利用可否は Auth0 のプランによって異なります">
  この機能を利用できるかどうかは、Auth0 のプランまたはカスタム契約によって異なります。詳しくは、[Pricing](https://auth0.com/pricing)をご覧ください。
</Card>

以下の目的で、独自の分離された (レガシー) アイデンティティデータストアへのアクセスを提供したい場合は、カスタムデータベース接続を使用します。

* **認証**: Auth0 でデータベースを <Tooltip tip="アイデンティティプロバイダー (IdP): デジタルアイデンティティを保存および管理するサービス。" cta="用語集を見る" href="/ja/docs/glossary?term=identity+provider">アイデンティティプロバイダー</Tooltip> として使用し、ユーザーを認証します。 (これはレガシー認証と呼ばれます。)
* **ユーザーのインポート**: 自動移行 (段階的移行または遅延移行) を使用します
* **Auth0 テナントへのプロキシアクセス**: Auth0 のマルチテナントアーキテクチャを使用します。

カスタムデータベース接続は、次のいずれかの方法で作成および設定できます。

1. `auth0` 戦略を指定して、[Create connections](https://auth0.com/docs/api/management/v2#!/Connections/post_connections) エンドポイントを使用します。
2. [Auth0 Dashboard &gt; Authentication &gt; Database](https://manage.auth0.com/#/connections/database) に移動し、接続を作成してから、**Use my own database** オプションを有効にしてデータベースアクションスクリプトを編集できるようにします。

   <Frame>![Enable Custom Database Use My Own Database Option](/docs/images/cdy7uua7fh8z/3kgHDpBFdVWNq9XOfhsTXI/2efa43e44793b8dd69b9f9e8b54bf752/2025-02-25_10-20-20.png)</Frame>

<div id="how-it-works">
  ## 仕組み
</div>

以下の図に示すように、独自のレガシー identity store からユーザーのID情報を取得するために、<Tooltip tip="Universal Login: アプリケーションは、ユーザーのIDを確認するために、Auth0 の Authorization Server でホストされている Universal Login にリダイレクトされます。" cta="用語集を見る" href="/ja/docs/glossary?term=Universal+Login">Universal Login</Tooltip>ワークフローの一部として、カスタムデータベース接続を使用します。

<Frame>![カスタムデータベース接続の構成](/docs/images/cdy7uua7fh8z/2lHqvZKFiEbAXURU2gmchc/626cac94211c266f2135a41456b2e49d/custom-database-connections.png)</Frame>

すべての[データベース接続](/ja/docs/authenticate/database-connections)タイプに共通する要素に加えて、カスタムデータベース接続では action scripts を設定できます。これは、レガシー identity store と連携する際に使用するカスタムコードです。Auth0 では設定用に[カスタムデータベース action script テンプレート](/ja/docs/authenticate/database-connections/custom-db/templates)を提供しており、使用するテンプレートは、レガシー authentication 用のカスタムデータベース接続を作成するか、自動移行用に作成するかによって異なります。

<Card title="ベストプラクティス">
  action scripts は無名関数として実装することもできますが、無名関数だと、例外エラー発生時に生成されるコールスタックの解析がデバッグ時に難しくなります。そのため、各 action script には関数名を付けることを推奨します。
</Card>

<div id="legacy-authentication-scenario">
  ### レガシー認証のシナリオ
</div>

レガシー認証のシナリオでは、ユーザーの初回ログイン時に Auth0 内に新しいユーザーレコードが作成されますが、Auth0 にユーザーのパスワードハッシュは保存されません。ユーザーの認証時、Auth0 は常に従来の identity store と、そこに含まれる ID 情報を使用します。

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  カスタムデータベース接続は、Universal Login のワークフロー以外でも使用されます。たとえば、従来の identity store に存在するユーザーに対してパスワード変更が行われると、接続の `changePassword` アクションスクリプトが呼び出されます。
</Callout>

<div id="automatic-migration-scenario">
  ### 自動移行シナリオ
</div>

自動移行または段階的移行では、Auth0 によって管理される identity store (データベース) に、Auth0 が新しいユーザーを作成します。それ以降、Auth0 はユーザーの認証に常に Auth0 管理の identity store 内の ID を使用します。これが行われるには、まず Auth0 がレガシー identity store に対してユーザーを認証する必要があり、それに成功した場合にのみ新しいユーザーが作成されます。新しいユーザーは、認証時に指定されたものと同じ id とパスワードを使用して作成されます。

<Card title="ベストプラクティス">
  自動移行シナリオでのユーザー作成は、通常 `login` アクションスクリプトの完了後に行われます。そのため、レガシー identity store からのユーザー削除をインライン操作 (つまり `login` スクリプト内) として行おうとせず、削除は独立したプロセスとして実行することをお勧めします。これにより、移行プロセス中にエラーが発生した場合でも、ユーザーを誤って削除してしまうのを防ぐことができます。
</Card>

自動移行シナリオでは、ユーザーはレガシー identity store に残るため、必要に応じて削除またはアーカイブできます。このことによる副作用の 1 つとして、ユーザーが Auth0 から削除されても、レガシー identity store には引き続き存在している場合があります。この場合、削除されたユーザーがログインすると、`login` および/または `getUser` スクリプトが実行され、ユーザーが再度レガシー identity store から移行される可能性があります。

<Card title="ベストプラクティス">
  `login` または `getUser` が完了する前、かつ最終的にレガシーストアを削除する前に、各レガシーユーザー ID に移行済みであることを示す印を付けておくことをお勧めします。
</Card>

<div id="size">
  ## サイズ
</div>

任意のアクションスクリプトの実装全体のサイズは、100 kB を超えないようにしてください。サイズが大きくなるほど、Auth0 のサーバーレス Webtask プラットフォームで使われるパッケージ化と転送の処理によりレイテンシが増加し、システムのパフォーマンスに影響します。

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">
  100 kB の上限には、`require` 文で参照される `npm` モジュールは含まれません。
</Callout>

<div id="environment">
  ## 環境
</div>

アクションスクリプトは、サーバーレスの Webtask コンテナーのインスタンス内で、呼び出された一連の JavaScript 関数として実行されます。その一環として、特定の実行環境に加え、Webtask コンテナーと Auth0 <Tooltip tip="認証サーバー: ユーザーの本人確認を行い、認証または拒否するサーバー。" cta="用語集を見る" href="/ja/docs/glossary?term=authentication+server">認証サーバー</Tooltip> (Auth0 テナント自体) の両方から提供されるさまざまなアーティファクトも用意されます。

<div id="npm-modules">
  ### npm モジュール
</div>

Auth0 のサーバーレス Webtask コンテナでは、[幅広い npm モジュール](https://auth0-extensions.github.io/canirequire/)を利用できます。`npm` モジュールは、action スクリプトの実装コード全体のサイズを削減できるだけでなく、あらかじめ用意された幅広い機能も利用できるようにします。

一般公開されている多くの `npm` モジュールは、追加設定なしですぐに利用できます。この一覧は、潜在的なセキュリティ上の懸念がないかを確認したうえで作成されています。すぐに利用できる対象に含まれていない `npm` モジュールが必要な場合は、[Auth0 Supportポータル](https://support.auth0.com)または Auth0 の担当者を通じてリクエストできます。Auth0 は、そのリクエストが適切かどうかを評価します。現在、Auth0 はプライベートリポジトリの `npm` モジュールの利用をサポートしていません。

<div id="variables">
  ### 変数
</div>

Auth0 のアクションスクリプトでは、グローバルに利用できる `configuration` オブジェクトを通じて環境変数にアクセスできます。詳しくは、[カスタムデータベース接続を作成する](/ja/docs/authenticate/database-connections/custom-db/create-db-connection) の「設定パラメーターを追加する」セクションを参照してください。

<Card title="ベストプラクティス">
  `configuration` オブジェクトは読み取り専用として扱い、外部のアイデンティティストアにアクセスするための認証情報や API キーなどの機密情報の保存に使用してください。これにより、セキュリティ上重要な値をアクションスクリプトにハードコードせずに済みます。
</Card>

`configuration` オブジェクトは、テナントごとの値を持つ変数を定義できるため、採用している [ソフトウェア開発ライフサイクル (SDLC) ](/ja/docs/get-started/auth0-overview/create-tenants/set-up-multiple-environments) 戦略をサポートする目的でも使用できます。これにより、実行しているテナントに応じて変わる可能性のある値をアクションスクリプトにハードコードせずに済みます。

<div id="global-object">
  ### global object
</div>

Auth0 のサーバーレス Webtask コンテナーは、各 Auth0 テナントに関連付けられたプールからプロビジョニングされます。各コンテナーインスタンスでは `global` オブジェクトが利用可能で、このコンテナーインスタンス内で実行されるすべてのアクションスクリプトからアクセスできます。`global` オブジェクトは、そのコンテナー固有のグローバル変数として機能し、コンテナーインスタンス内で実行されるすべてのアクションスクリプトで利用する情報や関数を定義するために使用できます。

つまり、`global` オブジェクトは、それらのリソースがユーザー固有でない限り、コストの高いリソースをキャッシュするために使用できます。たとえば、ユーザー固有ではない機能を提供するサードパーティの (ロギング) API 用の <Tooltip tip="アクセストークン: API へのアクセスに使用される、不透明な文字列または JWT 形式の認可資格情報。" cta="用語集を見る" href="/ja/docs/glossary?term=Access+Token">アクセストークン</Tooltip> を保存できます。あるいは、Auth0 で定義され、[Client Credentials Flow](/ja/docs/get-started/authentication-and-authorization-flow/client-credentials-flow) を通じて取得した、自身のユーザー固有ではない API のアクセストークンを保存することもできます。

<Warning>
  アクションスクリプトは、すでに実行中のいずれかのコンテナーインスタンス、または新しく作成されたコンテナーインスタンス (その後プールに追加されることがあります) で実行される可能性があります。Auth0 では、アクションスクリプトの実行にコンテナーアフィニティはありません。つまり、`global` オブジェクトにユーザー固有の情報を保存することは避け、`global` オブジェクト内で行う宣言には必ず初期化の処理も含めるようにしてください。
</Warning>

Webtask コンテナーが再利用されるたび、または新しい Webtask コンテナーがインスタンス化されるたびに、その `global` オブジェクトはリセットされます。したがって、コンテナーに関連付けられた `global` オブジェクト内で代入を宣言する場合は、初期化の処理も含める必要があります。

<Card title="ベストプラクティス">
  パフォーマンスの柔軟性を確保するため、サーバーレス Webtask コンテナーは Auth0 で必要に応じてプロビジョニングされ、さらにさまざまな再利用ポリシーの対象にもなります。一般的には、`global` オブジェクトの寿命は 20 分を超えるものとは考えないことを推奨します。
</Card>

<div id="security">
  ## セキュリティ
</div>

<div id="access-legacy-identity-storage-via-custom-api">
  ### カスタムAPIを介してレガシーIDストレージにアクセスする
</div>

レガシーIDストレージを一般アクセスから保護することは、推奨されるベストプラクティスです。たとえば、データベースをインターネットに直接公開することは、非常に大きな問題を招くおそれがあります。SQL などのデータベースインターフェースは機能面で非常に広範な操作を許してしまうため、セキュリティにおける最小権限の原則に反します。

<Card title="ベストプラクティス">
  インターネット経由で一般アクセスをそのまま許可するのではなく、レガシーIDストア (データベース) に対して最小権限でアクセスできる API を実装することをお勧めします。
</Card>

代替策としては、各 action スクリプトから呼び出せるシンプルな (カスタム) API を作成し、アクセストークンで保護する方法があります。これにより、その API がレガシーIDストアへのインターフェースとして機能します。さらに、Client Credentials Grant フローを使用すれば、スクリプト内からアクセストークンを取得できます。取得したトークンは、パフォーマンス向上のために `global` オブジェクトにキャッシュして再利用できます。API では、必要なレガシー管理機能だけを実行する、保護された限られた数のエンドポイント (たとえば `read user`、`change password`) を提供できます。

<Card title="ベストプラクティス">
  デフォルトでは、適切な audience を指定して認証に成功すると、Auth0 は任意の API に対するトークンを発行します。Rule を使用してアクセストークンの払い出しを制限し、レガシーIDストア API へのアクセスを制御することで、不正利用を防ぎ、`/authorize` へのリダイレクトが傍受されて API の audience が追加されるようなケースをはじめ、さまざまな攻撃ベクトルのリスクを軽減できます。
</Card>

<div id="allowlist-access-to-legacy-identity-storage">
  ### レガシーなアイデンティティストアへのアクセスを許可リストで制限する
</div>

カスタム API を介してレガシーなアイデンティティストアへのアクセスを管理する場合でも、提供されているネイティブインターフェースを使用する場合でも、アクセスは Auth0 テナントに関連付けられた IP アドレスのリストのみに制限してください。許可リストを使用するとレガシーなアイデンティティストアへのアクセスが制限され、Auth0 で定義されたカスタムデータベースのアクションスクリプトだけが許可されます。

<Warning>
  Auth0 の IP アドレス許可リストは、同じリージョン内のすべての Auth0 テナントで共有されます。レガシーなアイデンティティストアへのアクセスを保護する唯一の手段として、許可リストだけに頼らないでください。そうすると、権限のないユーザーによるアクセスを許し、潜在的なセキュリティ脆弱性につながるおそれがあります。
</Warning>
