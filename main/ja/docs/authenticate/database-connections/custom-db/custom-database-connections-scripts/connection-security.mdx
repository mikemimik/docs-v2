<div id="access-legacy-identity-storage-via-custom-api">
  ## カスタム API 経由でレガシーアイデンティティストアにアクセスする
</div>

インターネット経由で広くアクセスできるようにするのではなく、レガシーアイデンティティストアに対して最小権限でアクセスできる API を実装することをお勧めします。

レガシーアイデンティティストアを広範なアクセスから保護することは、推奨されるベストプラクティスです。たとえば、 (レガシーアイデンティティの) データベースをインターネットに直接公開すると、非常に大きな問題を招くおそれがあります。SQL などのデータベースインターフェースは、機能面で非常に広範な操作を許してしまうため、セキュリティにおける最小権限の原則に反します。

代わりに、各アクションスクリプトから呼び出せる、<Tooltip tip="アクセストークン: API へのアクセスに使用される認可の資格情報で、オペーク文字列または JWT の形式を取ります。" cta="用語集を見る" href="/ja/docs/glossary?term=access+token">アクセストークン</Tooltip>で保護されたシンプルな (カスタム) API を作成する方法があります。これがレガシーアイデンティティストアへのインターフェースとして機能します。さらに、スクリプト内でアクセストークンを取得するためにクライアントクレデンシャルズグラントフローを使用でき、その後はパフォーマンス向上のために `global` オブジェクトにキャッシュして再利用できます。API はそのうえで、必要なレガシー (アイデンティティ) 管理機能だけを実行する、限定された数の保護されたエンドポイント (例: ユーザーの読み取り、パスワード変更) を提供できます。

デフォルトでは、認証に成功し、適切な <Tooltip tip="Audience: 発行されたトークンの audience を一意に識別する値です。トークン内では aud という名前で、ID トークンの場合はアプリケーション（Client ID）の ID、アクセストークンの場合は API（API 識別子）の ID がその値に含まれます。" cta="用語集を見る" href="/ja/docs/glossary?term=audience">audience</Tooltip> を含めれば、Auth0 はどの API に対してもトークンを発行します。Rule を使用してアクセストークンの割り当てを制限し、レガシーアイデンティティストア API へのアクセスを制限することで、不正利用を防止でき、`/authorize` へのリダイレクトが傍受されて API への audience が追加されるような、さまざまな攻撃ベクトルのシナリオを軽減できます。

<Warning>
  Rule によって API へのアクセスを制限すると、`/authorize` へのリダイレクトが傍受されて API への audience が追加される場合などの攻撃ベクトルのシナリオを軽減でき、さらに、特定のクライアント資格情報を使用したアクセスのみが許可されることを保証できます。
</Warning>

<div id="access-to-legacy-identity-storage">
  ## レガシーアイデンティティストアへのアクセス
</div>

カスタム API を介してレガシーアイデンティティストアへのアクセスを管理する場合も、提供されている標準インターフェースを使用する場合も、Auth0 テナントに関連付けられた IP アドレス一覧のみにアクセスを制限する必要があります。IP アドレスの一覧を確認するには、[AllowLists 向け Auth0 IP アドレス](/ja/docs/secure/security-guidance/data-security/allowlist) を参照してください。AllowList に URL を追加すると、レガシーアイデンティティストアへのアクセスが制限され、Auth0 で定義されたカスタムデータベースアクションスクリプトのみが許可されます。

<Warning>
  Auth0 の IP アドレス AllowList は、同一リージョン内のすべての Auth0 テナントで共有されています。レガシーアイデンティティストアへのアクセス保護を allow list だけに頼ってはいけません。そうすると、未承認のアクセスを許してしまう可能性があり、セキュリティ上の脆弱性につながるおそれがあります。
</Warning>
