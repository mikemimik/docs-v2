  <Content>
    このチュートリアルでは、カスタム `Auth0Client` クラス内で [jwt](https://github.com/jwt/ruby-jwt) Gem を使用してアクセストークンのバリデーションを行います。`Secured` という Concern を使用して、受信したアクセストークンによる認証が必要なエンドポイントを認可します。

    Auth0 ダッシュボードでまだAPIを作成していない場合は、インタラクティブセレクターを使用して新しいAuth0 APIを作成するか、プロジェクト用の既存のAPIを選択してください。

    Auth0 Dashboardで最初のAPIを設定する方法については、[入門ガイド](https://auth0.com/docs/get-started/auth0-overview/set-up-apis)をご覧ください。

    各 Auth0 API は API 識別子を使用しており、アプリケーションがアクセストークンを検証する際に必要となります。

    <Info>
      **Auth0は初めてですか？** [Auth0 の仕組み](https://auth0.com/docs/overview)
      を確認し、OAuth 2.0 フレームワークを使用した [API の認証
      と認可の実装](https://auth0.com/docs/api-auth) についてご覧ください。
    </Info>

    <Section id={sections[0].id} title={sections[0].title} stepNumber="1">
      Permissions を使用すると、指定された access token でユーザーに代わってリソースにどのようにアクセスできるかを定義できます。た
      とえば、ユーザーのアクセスレベルが manager の場合は `messages` リソースへの読み取りアクセスを許可し、
      administrator のアクセスレベルの場合はそのリソースへの書き込みアクセスを許可するように設定できます。

      許可する permissions は、Auth0 Dashboard の [APIs](https://manage.auth0.com/#/apis) セクションにある **Permissions** view で定義できます。

    <Section id={sections[1].id} title={sections[1].title} stepNumber="2">
      &#42;&#42;jwt &#42;&#42;gemをインストールします。

      ```bash lines
      gem 'jwt'
      bundle install
      ```

    <Section id={sections[2].id} title={sections[2].title} stepNumber="3">
      `Auth0Client` というクラスを作成します。このクラスは、リクエストの `Authorization` ヘッダーから
      取得した受信アクセストークンをデコードして検証します。

      `Auth0Client` クラスは Auth0 テナントの公開キーを取得し、それを使って
      アクセストークンの署名を検証します。`Token` 構造体では `validate_permissions` メソッドを定義し、
      必要なスコープの配列を指定してアクセストークン内の特定の `scope` を探し、
      それらがトークンのペイロードに含まれているかどうかを確認します。
    </Section>

    <Section id={sections[3].id} title={sections[3].title} stepNumber="4">
      `Authorization` ヘッダー内のアクセストークンを受信したリクエストから探す、`Secured` という Concern を作成します。

      トークンが存在する場合、`Auth0Client.validate_token` は `jwt` Gem を使用して
      トークンの署名を検証し、トークンのクレームを検証します。

      アクセストークンが有効であることの検証に加えて、この Concern には、
      要求されたリソースにアクセスするのに十分な **scope** をトークンが持っていることを確認する仕組みも含まれています。この例では、
      ブロックを受け取り、`Auth0Client` クラスの `Token.validate_permissions` メソッドを呼び出して
      権限を確認する `validate_permissions` メソッドを定義します。

      `/private-scoped` ルートでは、定義されたスコープと
      ペイロードに含まれるスコープの積集合を取り、もう一方の配列に含まれる項目を 1 つ以上含んでいるかどうかを判定します。
    </Section>

    <Section id={sections[4].id} title={sections[4].title} stepNumber="5">
      `Secure` concern をアプリケーションコントローラーに追加すると、認可が必要なコントローラーでは
      `before_action` フィルターを使うだけで済みます。
    </Section>

    <Section id={sections[5].id} title={sections[5].title} stepNumber="6">
      `/api/public` の公開 endpoint を処理するコントローラーを作成します。

      `/public` endpoint では認可は不要なため、`before_action` は必要ありません。
    </Section>

    <Section id={sections[6].id} title={sections[6].title} stepNumber="7">
      プライベートエンドポイント (`/api/private` および `/api/private-scoped`) を処理するコントローラーを作成します。

      `/api/private` は、追加のスコープを持たないアクセストークンを含む認証済みリクエストで利用できます。

      `/api/private-scoped` は、`read:messages` スコープが付与されたアクセストークンを含む認証済みリクエストで利用できます

      保護されたエンドポイントでは、`Secured` concernの`authorize`メソッドを呼び出す必要があります。そのために`before_action :authorize`を使用します。これにより、`PrivateController`内のすべてのアクションの前に`Secured.authorize`メソッドが呼び出されます。

      ### APIを呼び出す

      APIを呼び出すには、アクセストークンが必要です。テスト用のアクセストークンは、[API設定](https://manage.auth0.com/#/apis)の**テスト**ビューから取得できます。

        ##### チェックポイント

        アプリケーションの設定が完了したら、アプリケーションを実行して次の点を確認してください。

        * `GET /api/public` は、未認証のリクエストでも利用できます。
        * `GET /api/private` は、認証済みのリクエストで利用できます。
        * `GET /api/private-scoped` は、`read:messages` スコープを含むアクセストークンを持つ認証済みリクエストで利用できます。
