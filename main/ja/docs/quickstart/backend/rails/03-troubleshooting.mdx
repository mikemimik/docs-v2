このドキュメントは、APIから想定外のレスポンスが返された場合に、設定の問題を切り分けるのに役立ちます。アカウント用に設定されたサンプルを使ってこのQuickstartを進められるよう、ログインしておくことをお勧めします。

JWTのバリデーションを正しく設定していれば、リクエストを送信したときにAPIから適切なレスポンスを受け取れます。ただし、APIから401 (Unauthorized) レスポンスが返ってくる場合は、JWTミドルウェアの設定が渡されたJWTと一致していないことが原因です。

このドキュメントは、JWTミドルウェアの設定に関する問題の切り分けに役立ちます。

<div id="how-does-a-token-get-validated">
  ## トークンはどのように検証されますか？
</div>

JWT を検証する際には、いくつか考慮すべき点があります。

1. **トークンの形式は正しいですか？** つまり、このトークンは JSON Web トークン (JWT) の構造に準拠していますか？ JWT の構造について詳しくは、[JWT の構造に関するこちらのセクション](/ja/docs/secure/tokens/json-web-tokens#what-is-the-json-web-token-structure-)を参照してください。
2. **トークンは改ざんされていませんか？** JWT の最後の部分は署名です。この署名は、トークンが実際に送信者によって署名されており、途中で一切変更されていないことを確認するために使用されます。
3. **トークンは有効期間内に受信されましたか？** JWT は指定された期間内でのみ有効です (`exp` クレーム で示されます) 。
4. **トークンは想定された認可主体から発行されたものですか？** これには 2 つの要素があります

   * **署名の検証**: 発行主体が発行したキーを使用して、JWT が正しく署名されていることを確認できますか？
   * **発行者の値**: 発行者は `iss` クレーム で定義されます。ここでも、この クレーム はアプリケーションが想定している値と一致していますか？
5. **トークンは現在のアプリケーション向けですか？** つまり、JWT の `aud` クレーム はアプリケーションが想定している値と一致していますか？

<div id="inspecting-a-token">
  ## トークンを調べる
</div>

JWT を手早く調べるには、[JWT.io](https://jwt.io/) の Web サイトを使う方法があります。便利なデバッガーが用意されているので、JWT の形式が正しいかをすばやく確認できるほか、さまざまなクレームの値を確認することもできます。

<Frame>![JWT.io で JWT をデバッグする](https://cdn2.auth0.com/docs/1.14550.0/media/articles/server-apis/aspnet-core-webapi/jwt-io-debugger-rs256.png)</Frame>

上のスクリーンショットを見ると、このトークンが **RS256** アルゴリズムで署名されていることがわかります。トークンの **発行者** は **https://jerrie.auth0.com/** で、**audience** は **https://rs256.test.api** です。

つまり、JWT バリデーション構成の登録にあるこれらの値は、**完全に** 一致している必要があります。たとえば、発行者の末尾のスラッシュも含めて

```
audience = "https://rs256.test.api",
domain = "https://jerrie.auth0.com/"
};
```

HS256 で署名された token の場合、デバッガー表示は少し異なります。

<Frame>![JWT.io で JWT をデバッグする](https://cdn2.auth0.com/docs/1.14550.0/media/articles/server-apis/aspnet-core-webapi/jwt-io-debugger-hs256.png)</Frame>

上のスクリーンショットから、この token が **HS256** アルゴリズムで署名されていることがわかります。token の **発行者** は **https://jerrie.auth0.com/**、**audience** は **https://hs256.test.api** です。
