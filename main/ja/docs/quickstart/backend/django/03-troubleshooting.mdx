このドキュメントは、APIから予期しないレスポンスが返された場合に、設定のトラブルシューティングを行う際に役立ちます。アカウント用に設定された例を使ってこのQuickstartを進められるよう、あらかじめログインしておくことをお勧めします。

JWTのバリデーションが正しく設定されていれば、リクエストを送信した際にAPIから適切なレスポンスを受け取れます。ただし、APIから401 (Unauthorized) レスポンスが返ってくる場合は、JWTミドルウェアの設定が渡されたJWTと一致していないことが原因です。

このドキュメントは、JWTミドルウェアの設定をトラブルシューティングする際に役立ちます。

<div id="how-does-a-token-get-validated">
  ## トークンはどのように検証されますか？
</div>

JWT を検証する際には、いくつか確認すべき点があります。

1. **トークンは適切な形式になっていますか？** つまり、このトークンは JSON Web トークン (JWT) の構造に準拠していますか？ JWT の構造について詳しくは、[JWT の構造に関するこちらのセクション](/ja/docs/secure/tokens/json-web-tokens#what-is-the-json-web-token-structure-)を参照してください。
2. **トークンは改ざんされていませんか？** JWT の最後の部分は署名です。この署名は、そのトークンが実際に送信者によって署名されており、途中で一切変更されていないことを確認するために使われます。
3. **トークンは有効期間内に受信されていますか？** JWT は、指定された一定期間のみ有効です (`exp` クレームで示されます) 。
4. **トークンは想定された認可機関から送られてきたものですか？** これには 2 つの要素があります。

   * **署名の検証**: 発行元の認可機関が発行したキーを使って、JWT が正しく署名されていることを確認できますか？
   * **発行者の値**: 発行者は `iss` クレームで定義されます。このクレームは、アプリケーションが想定している値と一致していますか？
5. **トークンは現在のアプリケーション向けのものですか？** つまり、JWT の `aud` クレームは、アプリケーションが想定している値と一致していますか？

<div id="inspecting-a-token">
  ## トークンの確認
</div>

JWT を手早く確認するには、[JWT.io](https://jwt.io/) のWebサイトを使う方法があります。ここには便利なデバッガーがあり、JWT の形式が正しいかをすばやく確認できるほか、さまざまなクレームの値も確認できます。

<Frame>![JWT.io で JWT をデバッグする](https://cdn2.auth0.com/docs/1.14550.0/media/articles/server-apis/aspnet-core-webapi/jwt-io-debugger-rs256.png)</Frame>

上のスクリーンショットでは、このトークンが **RS256** アルゴリズムで署名されていることがわかります。トークンの **発行者** は **https://jerrie.auth0.com/**、**Audience** は **https://rs256.test.api** です。

つまり、JWT バリデーションの構成で設定するこれらの値は、発行者の末尾のスラッシュも含めて、**完全に** 一致している必要があります。たとえば次のようにします。

```
audience = "https://rs256.test.api",
domain = "https://jerrie.auth0.com/"
};
```

HS256 で署名された トークン の場合、デバッガービューは少し異なって見えます。
