    <Section id={sections[0].id} title={sections[0].title} stepNumber="1" isSingleColumn>
      Auth0 のサービスを利用するには、Auth0 Dashboard でアプリケーションを設定しておく必要があります。Auth0 アプリケーションでは、
      開発中のプロジェクトで認証をどのように機能させるかを設定します。

      ### アプリケーションを設定する

      対話型セレクターを使って新しい Auth0 アプリケーションを作成するか、統合したいプロジェクトに対応する既存のアプリケーションを選択します。Auth0 のすべてのアプリケーションには、
      一意の英数字の client ID が割り当てられており、アプリケーションのコードはこれを使って SDK 経由で Auth0 API を呼び出します。

      このクイックスタートで行った設定は、[Auth0 Dashboard](https://manage.auth0.com/#/) 内のアプリケーションに自動的に反映されます。今後はここで
      アプリケーションを管理できます。

      設定例をひととおり確認したい場合は、代わりにサンプルアプリケーションを参照することもできます。

      ### コールバック URL を設定する

      コールバック URL とは、ユーザーの認証完了後に Auth0 がリダイレクトする、アプリケーション内の URL です。設定しない場合、
      ユーザーはログイン後にアプリケーションへ戻れません。

      <Info>
        サンプルプロジェクトに沿って進めている場合は、これを
        `http://localhost:3000/auth/auth0/callback` に設定してください。
      </Info>

      ### Logout URL を設定する

      ログアウト URL とは、ユーザーのログアウト後に Auth0 がリダイレクトする、アプリケーション内の URL です。設定しない場合、
      ユーザーはアプリケーションからログアウトできず、エラーが表示されます。

      <Info>
        サンプルプロジェクトに沿って進めている場合は、これを `http://localhost:3000` に設定してください。
      </Info>

      ### 許可する Web Origin を設定する

      許可する Web Origin とは、認証フローへのアクセスを許可する URL です。ここには
      プロジェクトの URL を含める必要があります。正しく設定されていない場合、プロジェクトは認証トークンをサイレントに更新できなくなり、
      その結果、ユーザーは次回アプリケーションにアクセスしたときやページを更新したときにログアウトされます。

      <Info>
        サンプルプロジェクトに沿って進めている場合は、これを `http://localhost:3000` に設定してください。
      </Info>

      認証フローを処理するには、カスタムの[OmniAuthストラテジー](https://github.com/intridea/omniauth#omniauth-standardized-multi-provider-authentication)である`omniauth-auth0`を使用します。

      以下の依存関係を`Gemfile`に追加します。

      ```gem lines
      gem 'omniauth-auth0', '~> 3.0'
      gem 'omniauth-rails_csrf_protection', '~> 1.0' # 偽造された認証リクエストを防ぐ
      ```

      gem を追加したら、`bundle install` を実行してインストールします。

      認証のコールバック、`logout` アクション、および
      ログアウト URL を構築するメソッドを処理する Auth0 コントローラーを作成します。

      次のコマンドを実行します。
      `rails generate controller auth0 callback failure logout --skip-assets --skip-helper --skip-routes --skip-template-engine`.

      `request.env['omniauth.auth']` として返されるユーザー情報のハッシュを、現在のセッションに割り当てます。

      メソッドを呼び出し、セッション内に保存されているすべてのオブジェクトを消去します。
      その後、Auth0 のログアウト エンドポイントにリダイレクトします。`reset_session` の詳細については、
      [Ruby on Rails ActionController documentation](http://api.rubyonrails.org/classes/ActionController/Base.html#M000668) を参照してください。

    <Section id={sections[8].id} title={sections[8].title} stepNumber="9" isSingleColumn>
      ユーザーのプロファイルを表示するには、アプリケーションに保護されたルートを用意する必要があります。複数の
      コントローラー間で共有できるルートへのアクセスを制御するには、[Concern](https://guides.rubyonrails.org/getting_started.html#using-concerns) を使用できます。ユーザーが未認証の場合、Concern は自動的に Auth0 にリダイレクトし、認証済みの場合は現在のユーザープロファイルを返すようにします。

      Concern を用意したら、ログイン済みユーザーを必要とするコントローラーにそれを include します。すると、次の例のように
      セッション `session[:userinfo]` からユーザー情報にアクセスできます。

      ```ruby lines
      class DashboardController < ApplicationController
      include Secured
      def show
      @user = session[:userinfo]

      end
      end
      ```

      セッションからユーザー情報を読み込んだら、それを使ってフロントエンドに表示します。

      ```html lines
      <div>
      <p>Normalized User Profile:<%= JSON.pretty_generate(@user[:info])%></p>
      <p>Full User Profile:<%= JSON.pretty_generate(@user[:extra][:raw_info])%></p>
      </div>
      ```

      <Note>
        ##### チェックポイント

        アプリに `Secured` concern を追加し、認証済みユーザーのみがアクセスできる
        コントローラーに組み込みます。認証済みユーザーがその
        コントローラー内のアクションにアクセスできること、
        また未認証ユーザーが認証のために Auth0 にリダイレクトされることを確認します。
      </Note>
