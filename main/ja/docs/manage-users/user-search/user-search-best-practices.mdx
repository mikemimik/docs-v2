title: ユーザー検索のベストプラクティス
---

ユーザー検索に関するベストプラクティスをいくつか紹介します。

* <Tooltip tip="Management API: お客様が管理タスクを実行できるようにする製品です。" cta="用語集を見る" href="/ja/docs/glossary?term=Management+API">Management API</Tooltip> へのリクエストを行うには、token が必要です。詳しくは、[Management API Access Tokens](/ja/docs/secure/tokens/access-tokens/management-api-access-tokens) を参照してください。
* ユーザー検索の request を実行するには、`read:users` scope が必要です。
* 非決定的検索を活用してください。アプリケーションで検索結果の順序が常に一定である必要がなく、queries も短くてページネーションに依存しない場合は、`primary_order=false` を設定するとクエリのパフォーマンスを改善できます。
* 最新の検索結果を取得するには、認証プロセス中に [Get Users by ID](/ja/docs/manage-users/user-search/retrieve-users-with-get-users-by-id-endpoint) や [Get Users by Email](/ja/docs/manage-users/user-search/retrieve-users-with-get-users-by-email-endpoint) などの **即時整合性のある** endpoint を使用してください。これらの endpoints を使った検索には、request の直前に行われたものを含め、成功したすべての書き込み操作の結果が反映されます。
* メタデータには、一般的な schema を使用してください。

  * プロパティには一貫したデータ型を使用してください。
  * 動的なプロパティ名は避けてください。
  * schema のサイズが大きくなりすぎたり、構造が深くなりすぎたりしないようにしてください。
  * 認証および認可に不要なデータは保存しないでください。
* 検索 queries は、2 秒以内に完了しないとタイムアウトします (HTTP status code 503) 。それより長くかかる queries は、負荷の高いクエリであるか、すぐに完了しない原因となるエラーがクエリにあることを示しています。
* `app_metadata` と `user_metadata` のユーザー定義 attributes を含む検索 queries では、タイムアウトが発生することがあります。重要な処理に [List or search users](https://auth0.com/docs/api/management/v2/users/get-users) を使用しないことをお勧めします。
* 大規模なデータセット (1000 件を超える結果) を返す検索条件は使用しないでください。
* 存在クエリ (たとえば「値に関係なく、あるプロパティを持つすべてのユーザーを返す」) は使用しないでください。
* 検索 API をポーリングしないでください。
* ルール や `post-login` Actions など、ログインフローの extension ポイント内でユーザー検索の request を実行しないでください。
* 大きな メタデータ フィールドは使用しないでください (メタデータ フィールドは 2 KB 以下に抑えるようにしてください) 。
* 検索で wildcard を使用すると、パフォーマンスに影響する場合があります。場合によっては、大規模なデータセットに対する wildcard 検索でタイムアウトエラーが発生することがあります。また、検索タームの先頭に付ける wildcards は避け、末尾に付けた方がパフォーマンスは向上します。
* パフォーマンス向上のため、空白文字はエスケープしてください (例: `q=name:John Doe` は `q=name:John\ Doe` と記述してください) 。
