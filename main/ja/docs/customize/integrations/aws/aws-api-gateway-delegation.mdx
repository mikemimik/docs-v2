<Warning>
  この機能は委譲を使用します。既定では、2017 年 6 月 8 日時点で使用中のアドオンがないテナントでは、委譲は無効になっています。現在、委譲を必要とするアドオンを使用しているレガシーテナントは、この機能を引き続き利用できます。今後、委譲機能が変更されたり、サービスから削除されたりする場合は、現在利用しているお客様に事前に通知し、移行のための十分な時間が提供されます。さらに、委譲ではカスタムドメインの使用はサポートされていないため、それに依存する機能はカスタムドメインと併用すると完全には動作しない可能性がある点に注意してください。
</Warning>

AWS を使用すると、AWS Lambda、Amazon API Gateway、JavaScript アプリケーションを通じて、強力でサーバーレスかつ高い拡張性を備えた API やアプリケーションを作成できます。

サーバーレスアプリケーションでは、サービスをホストする実行環境を維持しなくても、コンピューティングサービス上でカスタムコードを実行できます。代わりに、[AWS Lambda](https://aws.amazon.com/lambda/) のようなサービスがコードを実行します。

Amazon API Gateway は、Lambda 関数の前段にサービスレイヤーを追加することで AWS Lambda の機能を拡張し、セキュリティの強化、入出力メッセージ変換の管理、throttling や監査などの機能を提供します。サーバーレスアプローチを採用すると、スケールアウトや耐障害性といった課題はコードを実行するコンピューティングサービス側が担うため、運用上の負担を軽減できます。

ただし、多くの場合、Twitter や Facebook などのソーシャルプロバイダーの既存ユーザーや、Active Directory や顧客データベース内の自社ユーザーと API を結び付けたいと考えるでしょう。このチュートリアルでは、AWS 向け Auth0 委譲と AWS Identity and Access Management (IAM) との連携を使用して、既存ユーザーによる Amazon API Gateway メソッドへのアクセスを認可する方法を説明します。

続いてこのチュートリアルでは、AWS Lambda 関数を使用した Amazon API Gateway の設定、それらの関数を AWS IAM role で保護する方法、さらに AWS IAM role 用のトークンを取得するために Auth0 委譲を使用する方法を順を追って説明します。また、内部データベースユーザーやソーシャルユーザーなど、さまざまな種類のユーザーに異なる権限を割り当てる方法と、<Tooltip tip="JSON Web トークン (JWT): 2 者間でクレームを安全に表現するために使用される標準的な ID トークン形式（多くの場合 Access Token 形式でもあります）。" cta="用語集を見る" href="/ja/docs/glossary?term=JSON+Web+Token">JSON Web トークン</Tooltip> (JWT) を使用して ID 情報を受け渡す方法も紹介します。
