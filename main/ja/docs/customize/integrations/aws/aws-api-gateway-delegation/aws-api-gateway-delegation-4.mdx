<div id="step-4-use-multiple-roles-with-amazon-api-gateway">
  ## ステップ 4 - Amazon API Gateway で複数のロールを使用する
</div>

このステップでは、認証情報に基づいて、ユーザーに異なる AWS IAM ロールを割り当てます。

* Social Connections で認証されたユーザーは購入者として扱われます。
* データベース接続 で認証されたユーザーは管理者として扱われます。

このロール割り当てロジックは、次の 2 つの方法で実装します。

* JavaScript
* Auth0 ルール

多くの Auth0 アプリケーションでは、ユーザーごとに異なるレベルのアクセス権を設定したり、特定の ID に関する追加情報をサービスのロジックで利用したりしたい場合があります。API レベルでアクセスを制限できれば十分な場合は、異なる AWS IAM ロールを使用できます (たとえば、管理者は update 関数を使ってペットを追加・削除できますが、ソーシャルユーザーはペットを購入することしかできません) 。

次の図は、2 種類のユーザークラスに対する AWS IAM ロールの割り当てを示しています。1 つは Social Connections 経由で認証されたユーザー、もう 1 つは データベース接続 経由で認証されたユーザーです。また、AWS IAM ロールは AWS Lamdba 関数のようなほかのエンティティにも割り当てることができ、アカウント内でそれらのエンティティに付与される権限を制御できることも示しています。つまり、IAM ロールとは、1 つ以上のポリシーで定義され、エンティティに割り当てられる、AWS の機能に対する権限の集合です。

コード内で判断を行いたい場合 (たとえば、ペットを購入するユーザーの与信チェックを行いたい場合) には、ID 情報もフローさせる必要があります。これについては、以下の [Step 5 - Using Identity Tokens to Flow Identity](/ja/docs/customize/integrations/aws/aws-api-gateway-delegation/aws-api-gateway-delegation-5) で説明します。

<div id="step-1-create-the-petpurchase-api-resource">
  ### ステップ 1. PetPurchase API リソースを作成する
</div>

[Amazon API Gateway Console](https://console.aws.amazon.com/apigateway) で Pets API を選択します。Resources ページが表示されます。

**Actions** をクリックし、**Create Resource** を選択します。新しい子リソースの名前を `Purchase` にします。**Create Resource** をクリックします。

`purchase` リソースに OPTIONS メソッドを追加します。追加方法は、[ステップ 2 - Amazon API Gateway の保護とデプロイの「Set Up Cors and Deploy the API」セクション](/ja/docs/customize/integrations/aws/aws-api-gateway-delegation/aws-api-gateway-delegation-2#set-up-cors-and-deploy-the-api)で前述した `pets` の場合と同様です。

`PetPurchase` という名前の、ペット購入用の新しい AWS Lambda 関数を作成します。この関数は、次のようにペットに `isSold` 属性と `soldTo` 属性を追加します。

```javascript lines expandable
var AWS = require('aws-sdk');
var DOC = require('dynamodb-doc');
var dynamo = new DOC.DynamoDB();

exports.handler = function(event, context) {
   var petId = event.petId;
   var user = event.userName;
   var pets = {};
   console.log('PetsPurchase開始, petId', petId, ' userName', user);

   var writecb = function(err, data) {
      if(!err) {
          context.done(null, pets);
      } else {
          console.log('GetPetsInfoでエラー: ',err);
          context.done('failed on update', null);
      }
   };

   var readcb = function(err, data) {
      if(err) {
          console.log('GetPetsInfoでエラー: ',err);
          context.done('failed to retrieve pet information', null);
      } else {
          // ペットデータの存在を確認する
          if(data.Item && data.Item.pets) {
              pets = data.Item.pets;
              var found = false;

              for(var i = 0; i < pets.length && !found; i++) {
                  if(pets[i].id === petId) {
                     if(!pets[i].isSold) {
                        pets[i].isSold = true;
                        pets[i].soldTo = user;
                        var item = { username:"default",pets: pets};
                        dynamo.putItem({TableName:"Pets", Item:item}, writecb);
                        found = true;
                     }
                  }
               }
               if(!found) {
                 console.log('ペットが見つかりません');
                 context.done('That pet is not available.', null);
               }
           } else {
              console.log('ペットはすでに売却済みです');
              context.done('That pet is not available.', null);
           }
       }
   };

   dynamo.getItem({TableName:"Pets", Key:{username:"default"}}, readcb);
};
```

Lambda 関数を定義したら、`PetPurchase` Lambda を呼び出す [POST メソッドを `purchase` リソースに追加します](/ja/docs/customize/integrations/aws/aws-api-gateway-delegation/aws-api-gateway-delegation-1#method-post-pet-information)。また、[Step 2 - Amazon API Gateway の保護とデプロイの Set Up Cors and Deploy the API セクション](/ja/docs/customize/integrations/aws/aws-api-gateway-delegation/aws-api-gateway-delegation-2#set-up-cors-and-deploy-the-api) にある method response / integration response の設定を使用して、値 `*` の `Access-Control-Allow-Origin` ヘッダーも POST メソッドに追加してください。

次の内容を入力メッセージとして指定して、API Gateway メソッドをテストします。

```json lines
{
    "petId": 1,
    "userName": "fred flintstone"
 }
```

テストレスポンスには、ID が 1 のペットが Fred Flintstone に販売済みになっていることが表示されるはずです。

```json lines
[
  {
    "id": 1,
    "price": 249.99,
    "type": "dog",
    "isSold": true,
    "soldTo": "fred flintstone"
  },

  ...
```

<div id="step-2-use-iam-to-secure-the-purchasepet-api">
  ### ステップ 2. IAM で PurchasePet API を保護する
</div>

<div id="update-iam">
  #### IAM を更新する
</div>

API を保護するには、このチュートリアルの[パート 2](/ja/docs/customize/integrations/aws/aws-api-gateway-delegation/aws-api-gateway-delegation-2)で行ったのと同じ手順で、新しいロールを追加します。新しいロール名は `auth0-api-social-role` にします。

IAM ポリシーで保護するメソッドの ARN は、次のようになります。

```bash wrap lines
arn:aws:execute-api:us-east-1:your-accountid:your-api-id/*/pets/purchase
```

信頼ポリシーも忘れずに更新してください。

[Amazon API Gateway Console](https://console.aws.amazon.com/apigateway) に移動し、`/pets/purchase` リソースの POST メソッドを選択します。**メソッド要求** を選択し、**Authorization Type** を AWS&#95;IAM に変更します。チェックマークをクリックして設定を保存します。

これで、API Gateway で使用できる2つのロールを定義できました。

* `auth0-api-role`: ペットの更新を許可します
* `auth0-api-social-role`: ペットの購入を許可します

<div id="configure-login-with-amazon-and-update-auth0">
  #### Login with Amazon を設定して Auth0 を更新する
</div>

Login with Amazon (LWA) を使用してソーシャルロールを作成できます。

このチュートリアルでは Login with Amazon を使用する手順を説明していますが、ほかのソーシャルプロバイダーを使用することもできます。

1. [Auth0 Dashboard &gt; Authentication &gt; Social](https://manage.auth0.com/#/connections/social) に移動し、**Create Connection** を選択します。
2. 設定したい接続を選択し、同意します。
3. ソーシャルアイデンティティプロバイダーから `Client ID` と `Client Secret` をコピー＆ペーストし、**Attributes** (該当する場合は **Permissions** も) を選択して、**Save** をクリックします。
4. **Applications** ビューを選択し、この接続を使用できるようにする各 Auth0 アプリケーションのスイッチを有効にして、**Save** を選択します。

必要な情報を入力したら、**Try Connection** を選択して、すべてが正しく設定されていることを確認します。

Amazon コンソールを使用して LWA を設定する際は、Allowed Return URLs に Auth0 アプリケーションのコールバックURLを必ず入力してください。これは `https://johndoe.auth0.com/login/callback` のような形式になります。Auth0 のヘルプページには、具体的に何を入力すべきかが示されています。

[Auth0 Dashboard &gt; Applications &gt; Applications](https://manage.auth0.com/#/applications) に移動し、アプリケーションを選択してその設定を表示します。**Connections** ビューを選択し、**Social** セクションで **Amazon** が有効になっていることを確認します。

<div id="deploy-the-api">
  ##### API をデプロイする
</div>

[Amazon API Gateway Console](https://console.aws.amazon.com/apigateway) を使用して、あらためて [API をデプロイし、新しい JavaScript SDK を生成します](/ja/docs/customize/integrations/aws/aws-api-gateway-delegation/aws-api-gateway-delegation-2#set-up-cors-and-deploy-the-api)。

この時点で、ペットの購入を有効にするために必要な設定変更は完了しています。これを本番環境に反映するには、新しくダウンロードした SDK で、`pets` フォルダー内の以前の SDK と Amazon S3 バケット内の以前の SDK を上書きしてください。

<div id="update-the-login-controller-logic-to-choose-different-roles-for-different-types-of-users">
  ##### ユーザーの種類に応じて異なるロールを選択するように Login コントローラーのロジックを更新する
</div>

ログインコントローラーのロジックでは、`getOptionsForRole` を使ってユーザーごとに異なるロールを選択します。委譲トークンを取得する際に、どのロールを使うかを Auth0 に指定できます (つまり、そのユーザーが管理者かどうかを指定できます) 。

`pets/login/login.js` ファイルで、先ほど作成したソーシャルユーザー用の AWS IAM ロール に合わせて、管理者ではないユーザーの `role` と `principal` の値を変更します。

この時点で、Amazon の資格情報 **または** 以前に作成したデータベースユーザーでログインできるはずです。UI では、ソーシャルユーザーはペットを購入できる一方、管理者ユーザーはペットの追加と削除ができます。

この機能をテストするには、`/pets/home/home.html` で `ng-show="isAdmin"` を削除し、UI の remove ボタンを一時的に非表示にできます:

```html wrap lines
<button ng-show="isAdmin" class="btn delete-btn" ng-click="removePet(pet.id)">remove</button>
```

変更をS3バケットにコピーしたら、ソーシャルユーザーとしてログインした状態で、ペットを削除できるか試してください。

コードを S3 バケットにコピーし、ログアウトしたら、Lock のログインダイアログで Amazon アイコンをクリックしてソーシャルユーザーとして再度ログインします。前回のログイン状態が Lock ペインに残っている場合は、**SHOW ALL** をクリックする必要があるかもしれません。

Amazon ユーザーではペットを購入できますが、ペットの追加や削除はできない点に注意してください。一方、データベース接続に関連付けられたユーザーでログインすると、ペットの追加と削除はできますが、購入はできません。

<div id="enforce-role-assignment-with-auth0-rules">
  ### Auth0 ルールでロールの割り当てを強制する
</div>

場合によっては、適切なロールをアプリケーション (ここで示すように) に基づいて判断することもできますが、セキュリティ上の理由から (たとえば、ユーザーが必要以上に高い権限を持つロールを引き受けるのを防ぐために) 、サーバー側でユーザー権限を判断したいこともあります。

Auth0 では、これはルールによって実現します。ルールとは、Auth0 の認証プロセス中に実行される、ユーザーが定義するサービスロジックです。たとえば、次のようなルールを作成できます。

* ブラウザーからアプリケーションへロール情報を渡さないようにする。
* 認証元に基づいて、委譲リクエストにロール情報を追加する。

<div id="enforce-role-assignment">
  #### ロール割り当てを強制する
</div>

Social またはデータベース接続との関連付けに応じて、ユーザーが要求したロールが許可されているかどうかを確認するルールを追加します。

1. [Auth0 Dashboard &gt; Auth Pipeline &gt; Rules](https://manage.auth0.com/#/rules) に移動し、**Create Rule** を選択します。
2. **Empty rule** テンプレートを選択します
3. ルール名を **AWS Pets** (またはそれに類する名前) にして、ルールの本文に次の JavaScript コードを入力します。

export const codeExample = `   function (user, context, callback) {
  if(context.clientID === '{yourClientId}') {
    var socialRoleInfo = {
      role:"arn:aws:iam::<your account>:role/auth0-api-social-role",
      principal: "arn:aws:iam::your account>:saml-provider/auth0"
    };

    var adminRoleInfo = {
      role:"arn:aws:iam::<your account>:role/auth0-api-role",
      principal: "arn:aws:iam::<your account>:saml-provider/auth0"
    };

    var requestRole = context.request.body.role;
    var requestPrincipal = context.request.body.principal;
    var allowedRole = null;

    if(user.identities[0].isSocial === false) {
      allowedRole = adminRoleInfo;
    } else {
      allowedRole = socialRoleInfo;
    }

    if((requestRole && requestRole !== allowedRole.role) ||
      (requestPrincipal && requestPrincipal !== allowedRole.principal)) {
        console.log('要求されたロールが一致しません:',requestRole, ':', requestPrincipal);
        console.log('上書きします');
    } else {
      console.log('委譲で要求されたロールは有効か、ロールは要求されていません');
    }

    context.addonConfiguration = context.addonConfiguration || {};
    context.addonConfiguration.aws = context.addonConfiguration.aws || {};
    context.addonConfiguration.aws.role = allowedRole.role;
    context.addonConfiguration.aws.principal = allowedRole.principal;
    callback(null, user, context);

  } else {
    callback(null, user, context);
  }
}
`;

上記のコードは、連携に応じて正しい値に調整してください。該当するフィールドは **Princial ARN**、**Role ARN**、**Client Secret** です。
4. 変更内容を **Save** します。

<div id="caveats">
  #### 注意事項
</div>

* ルールは、すべての認証でグローバルに実行されます。ロジックは、特定のアプリケーションに関連付けられた認証リクエストに対してのみ実行するようにしてください (そのため、このスクリプトでは clientID を要求します) 。この情報がないと、ロジックは Auth0 アカウントに関連付けられたすべての認証リクエストで実行されます。
* 情報は、context と user を通じてルールに渡されます。
* ルールに渡されるオブジェクトは拡張できます。上記のコードでは、ルールはロール情報を確認するためにリクエストボディをチェックします。ロールは、許可されたロールの context addonConfiguration に設定され、これは常にリクエストボディ内の設定より優先されます。
