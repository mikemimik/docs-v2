<Card title="始める前に">
  [ボット検出](/ja/docs/secure/attack-protection/bot-detection)を有効にし、[CAPTCHAプロバイダー](/ja/docs/secure/attack-protection/bot-detection/configure-captcha)を設定する必要があります。
</Card>

Auth0 の[ボット検出監視機能](/ja/docs/secure/security-center#bot-detection)は、ボットネットや攻撃を早期に検知するためのシステムです。以下では、テナントへのログインを試みるボットを特定するためのガイダンスを紹介します。

<div id="find-log-events-of-interest">
  #### 注目すべきログイベントを見つける
</div>

対応を検討する際は、まず攻撃の可能性があるアクティビティに関連するログメッセージを精査してください。より高度な分析を行うには、[ログストリーミング](/ja/docs/customize/log-streams)を有効にして、任意の外部ツールに接続します。

ボットアクティビティの増加を調査する際は、次の[ログイベントタイプ](/ja/docs/deploy-monitor/logs/log-event-type-codes)が参考になります。

| ログイベントタイプ   | 説明                                                                                                |
| ----------- | ------------------------------------------------------------------------------------------------- |
| `pla`       | ログイン前に生成されるイベントで、ボット検出がボットの識別に CAPTCHA を使用していない場合や、監視モードでのみ動作している場合でも記録されます。                      |
| `fu`        | 無効なユーザー名によるログイン失敗イベントで、ユーザー名列挙やアカウント乗っ取りの試行を示している可能性があります。                                        |
| `fp`        | 無効なパスワードによるログイン失敗イベントで、クレデンシャルスタッフィング攻撃の試行を示している可能性があります。                                         |
| `pwd_leak`  | 漏えいしたパスワードを使ったログイン試行イベントで、クレデンシャルスタッフィング攻撃の試行を示している可能性があります。                                      |
| `limit_wc`  | 単一アカウントに対して 10 回を超えるログイン失敗試行があった場合の IP ブロックイベントで、その IP アドレスがボットによるものである可能性が高いことを示します。             |
| `limit_sul` | 同じ IP アドレスから 1 分間に 20 回を超えるログイン試行があった場合のユーザーブロックイベントで、ボットアクティビティの可能性が高いことを示します。                   |
| `limit_mu`  | 同じ IP アドレスから 100 回を超えるログイン失敗試行、または 50 回を超えるサインアップ試行があった場合の IP ブロックイベントで、ボットアクティビティの可能性が高いことを示します。 |
| `fcoa`      | クロスオリジン認証の失敗イベントで、攻撃者が自動化を使ってアカウント乗っ取りを試みていることを示します。                                              |
| `scoa`      | クロスオリジン認証の成功イベントで、少数の IP アドレスから複数のユーザーに対して発生している場合は、攻撃者が自動化を使ってアカウント乗っ取りを試みていることを示します。            |

<div id="attack-response">
  #### 攻撃への対応
</div>

<Tooltip tip="ボット検出: Auth0 がログインプロセス中に CAPTCHA を有効にして、疑わしいボットトラフィックをブロックする攻撃対策の一形態。" cta="用語集を見る" href="/ja/docs/glossary?term=Bot+Detection">ボット検出</Tooltip> レベルを High に設定すれば攻撃を直ちに緩和できますが、包括的な戦略では、ビジネスのリスク許容度や技術的能力と、ユーザーのサインイン時の体験とのバランスを取ることが重要です。対応にあたっては、主に次の 2 つの要素を考慮してください。

* **ユーザーの負担**: 緩和策 (例: CAPTCHA の表示頻度) がユーザー体験に与える影響を評価します。
* **技術的能力**: IP ブロック、WAF ルール、<Tooltip tip="多要素認証 (MFA): SMS で送信されるコードなど、ユーザー名とパスワードに加えて認証要素を使用するユーザー認証プロセス。" cta="用語集を見る" href="/ja/docs/glossary?term=MFA">MFA</Tooltip> の適用を実装できるかを評価します。

Auth0 では、最適な保護のために、複数の緩和手法を組み合わせた多層防御のアプローチを推奨しています。

<div id="mitigation-strategies">
  #### 緩和策
</div>

攻撃から効果的に保護するために、次の対策を検討してください。

* 1 つ以上のフローで CAPTCHA を有効化し、必要に応じて CAPTCHA の実行頻度を上げてください。ただし、CAPTCHA は抑止策であって、根本的な解決策ではない点に注意してください。
* 攻撃者に現在の CAPTCHA を回避されている場合は、CAPTCHAプロバイダー を変更するか、Auth0 の Auth Challenge または別の[サポート対象の provider](/ja/docs/secure/attack-protection/bot-detection/configure-captcha)への移行を検討してください。
* 不正な signup キャンペーンが疑われる場合は、公開された未認証のエンドポイントからアプリケーションへの新規ユーザー signup を一時的に停止してください。
* エッジプロバイダーで Web アプリケーションファイアウォールのルールを変更するか、[テナントアクセス制御リスト](/ja/docs/secure/tenant-access-control-list)を使用して、不正な IP、自律システム番号、地理的な場所、[TLS clients](/ja/docs/customize/custom-domains/self-managed-certificates/tls-ssl)、`user-agent` 文字列のような HTTP header 要素をブロックし、[reverse proxy.](/ja/docs/customize/custom-domains/self-managed-certificates#configure-reverse-proxy) の導入も検討してください。
* 許可される接続数の上限を引き下げてブルートフォース攻撃を緩和するために、[ブルートフォース](/ja/docs/secure/attack-protection/brute-force-protection) と [Suspicious IP](/ja/docs/secure/attack-protection/suspicious-ip-throttling) のしきい値を厳しくしてください。ブルートフォース攻撃の詳細については、[Brute Force playbook](/ja/docs/secure/attack-protection/playbooks/brute-force-protection-playbook)を参照してください。
* [Cross-Origin Authentication](/ja/docs/authenticate/login/cross-origin-authentication) の設定を変更して、使用していないエンドポイントを無効化してください。漏えいしたパスワードを使った攻撃が疑われる場合は、[Breached Password プレイブック](/ja/docs/secure/attack-protection/playbooks/breached-password-playbook)を参照してください。
* 侵害されたアカウントには、危険性が高いと考えられるアカウントに MFA を必須化することも含めて、[step-up MFA](/ja/docs/secure/multi-factor-authentication/step-up-authentication) を適用してください。
* SMS ポンピングや通話料金詐欺攻撃を緩和するため、SMS または音声ベースの MFA を OTP または Webauthn に置き換え、より強力な MFA オプションに移行してください。
